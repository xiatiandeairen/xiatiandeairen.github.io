# Blueprint: tutorial-intro

教程/介绍类文章骨架。适用于"介绍一个工具/能力集合，让读者读完知道什么时候用、怎么用"。

## 何时选此 blueprint

- 目标是**扩读者工具箱**，而非"讲我踩的坑"
- 主题是一类能力集合（如"GitHub 高级技巧"、"vim 编辑模式"），不是一个单点经验
- 读者预期：读完能上手，不是读完有共鸣

若主题是"我做过 X 的经验教训"→ 用 `blueprint.md`（skill-experience）。

## 五节骨架

```
# {title}                          # 一行标题：具体工具名 + 能力范围（如"GitHub 高级技巧：让日常工作流更省力的 8 个用法"）

## 这是什么
- 1-2 句定义：{工具名}是什么，跟什么东西并列/区别
- 一句话类比（降低认知门槛）
- 100-200 字

## 什么时候用
- 2-3 个典型触发场景（"你在 X 的时候会想起它"）
- 每个场景 1 段，含**具体痛点**（不是"提高效率"，写"每次改完都手动跑 5 个检查"）
- 200-400 字

## 核心能力
- 枚举 N 项（推荐 3-8 项），每项独立小节，形如 `### 1. {能力名}`
- 每项结构：1 句说明 + 1 个最小命令/示例 + 1 句"适用场景"
- 宁可少而精，不要凑数
- 500-1200 字（视 N 而定）

## 典型用法
- 2-3 个端到端场景，示范核心能力如何组合
- 每个场景给"输入 → 命令链 → 输出"，可带 diff / 截图描述
- 200-500 字

## 进阶建议 / 别踩的坑
- 1-2 条进阶方向（读者学会基础后下一步往哪走）
- 1-2 条"看起来该这样做，其实别"（反共识边界）
- 100-300 字
```

## AI 扩写 Prompt（draft 时使用）

```
你正在为 article skill 生成一篇"教程介绍类"文章草稿。

输入：
- topic: {用户给的主题，通常是一个工具或能力集合}
- core opinion: {用户对此工具的独特视角，如"大多数人只用到它 20% 的能力"}
- 素材: {inbox/ 中筛出的相关 .md 内容}

要求：
1. 严格按本 blueprint 五节结构填充。每节标题不得改写。
2. **具体化**：每个"核心能力"必须配 1 个可复制粘贴的命令/代码片段。没有命令就删这项。
3. **不编造**：具体命令、输出、flag 名必须真实存在于对应工具；不确定的写 `<!-- TODO: 核对命令语法 -->` 占位。
4. **典型用法节必须是组合示范**：不是重复核心能力，是把 2 项以上核心能力串成一个工作流。
5. **去 AI 腔**：禁用 "在当今快速发展的" / "综上所述" / "让我们深入探讨" / "无论…还是…" 长排比 / "极大地提升了"。
6. **长度控制**：总字数 1500-3500 中文字符。每节按骨架内字数范围。
7. **frontmatter 必填**：article-ctl.sh 模板生成 frontmatter 骨架后，你必须用 Edit 工具把以下字段改为实值：

| 字段 | 规则 |
|------|------|
| `title` | 实际标题，含工具名 + 范围限定 |
| `question.type` | 推荐 `"工具使用"` 或 `"工程实践"` |
| `question.subType` | 具体工具名，如 `"git"` / `"GitHub Actions"` |
| `quality.overall/coverage/depth/specificity` | 1-10 实话自评 |
| `quality.reviewer` | `"ai"` |
| `analysis.objectivity.*` | 三数之和 = 1.0；教程类典型 `0.7 / 0.15 / 0.15`（事实高、观点少） |
| `tags` | ≥1 条，对象数组；必含主工具名 tag |
| `topics` | ≥1 条；教程类常见 `"开发工具"` / `"工程实践"` |
| `gate_passed` | `false`（gate 通过后改 `true`）|

输出：完整的草稿 markdown，含五节内容（不要把这条 prompt 写进去）。
```

## 风格参考

- 第一段可以不用"我" — 教程类可以更客观
- 核心能力小节用 h3 编号（### 1. xxx），便于 TOC
- 每条命令必须有上下文说明"在哪个目录跑、会输出什么"
- 进阶建议节保留 1 个反共识结论，即使文章整体是中性介绍
