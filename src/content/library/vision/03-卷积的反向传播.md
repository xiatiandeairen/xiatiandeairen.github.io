直接说结论：你不需要为卷积单独发明一套反向传播。第 01 章已经把前向卷积写成了 `cols @ weightᵀ`——im2col（把每个滑窗位置的输入小块摊成矩阵的一行）展开后接一次矩阵乘。matmul 的反向你在任何 tiny 自动微分库里都见过：`Y = A @ B` 时 `dA = dY @ Bᵀ`、`dB = Aᵀ @ dY`。卷积反向 90% 就是把这条公式套到 `cols @ weightᵀ` 上，唯一的卷积特有步骤是最后一步：怎么把 `dCols`（对展开矩阵的梯度）散回原图。这一步叫 col2im，它是整章唯一会咬人的地方——而且咬的方式很隐蔽，普通单测抓不到。

本章用配套仓里的 `examples/vision-from-scratch/src/stage03-conv-backward.ts` 把两条梯度都手写出来、用中心差分（central finite difference，左右各扰动一点测斜率）逐元素验证，再故意把 col2im 写错给你看它怎么静默通过测试、又怎么在换个 stride 后瞬间暴露。读完你能回答一个面试常考但很多人背不出 why 的问题：为什么卷积对输入的反向"也是一个卷积"，但工程上几乎没人那么写。

## 把卷积反向拆成两条 matmul 梯度

先把前向的形状钉死，后面所有下标都对得上。配套仓 `stage03-conv-backward.ts` 的 `convForward`（第 52-75 行）做的事和第 01 章的核心引擎一致：

```ts
const cols = im2col(input, kH, kW, p); // (N*outH*outW, inC*kH*kW)
const patch = inC * kH * kW;
const rows = N * outH * outW;
// out (pixel-major): row r, channel oc = sum_p cols[r,p] * weight[oc,p]
const out = new Float64Array(rows * outC);
for (let r = 0; r < rows; r++) {
  for (let oc = 0; oc < outC; oc++) {
    let acc = 0;
    for (let pp = 0; pp < patch; pp++) acc += cols.data[r * patch + pp] * weight.data[oc * patch + pp];
    out[r * outC + oc] = acc;
  }
}
```

把它读成矩阵：`cols` 是 `(rows, patch)`，`weight` 摊平成 `(outC, patch)`，输出 `out` 是 `(rows, outC)`。即 `out = cols @ weightᵀ`。`rows = N·outH·outW` 是「一共有多少个输出像素位置」，`patch = inC·kH·kW` 是「一个滑窗看到多少个数」。这两个数后面反复出现，记住它们就是 matmul 的两个内积维度。

设损失 `L`，反向的入口是 `dOut`（`L` 对每个输出元素的梯度）。本章为了让公式干净，loss 取 `sum(out)`，于是 `dOut` 全是 1——这不是偷懒，是刻意把 matmul 反向的两个方向单独拎出来看清楚，不让 loss 的复杂度混进梯度推导。套 matmul 反向：

- **对核**：`dWeight = dOutᵀ @ cols`，形状 `(outC, patch)`。`dWeight[oc,pp] = Σ_r dOut[r,oc]·cols[r,pp]`。
- **对展开矩阵**：`dCols = dOut @ weight`，形状 `(rows, patch)`。`dCols[r,pp] = Σ_oc dOut[r,oc]·weight[oc,pp]`。

`stage03-conv-backward.ts` 的 `convBackwardSumLoss`（第 89-113 行）就是逐字翻译这两条：

```ts
// dWeight[oc, pp] = sum_r dOut[r,oc] * cols[r,pp]; dOut is all-ones (loss = sum).
for (let oc = 0; oc < outC; oc++) {
  for (let pp = 0; pp < patch; pp++) {
    let acc = 0;
    for (let r = 0; r < rows; r++) acc += cols.data[r * patch + pp]; // dOut[r,oc] == 1
    dWeight[oc * patch + pp] = acc;
  }
}
```

到这里没有任何"卷积特有"的东西。`dWeight` 是一次普普通通的矩阵乘，`dCols` 也是。**真正的卷积只剩最后一公里**：`dCols` 的形状是 `(rows, patch)`——它说的是"每个滑窗位置、每个核内偏移处的梯度"，但你要的 `dInput` 是 `(N, inC, H, W)`——原图每个像素的梯度。从前者到后者的映射，就是 im2col 那张展开索引表的逆向，col2im。

## col2im：为什么必须累加，不能覆盖

im2col 前向干的事，是按一张固定的索引表把原图像素**复制**到 `cols` 的各个格子里。关键词是"复制"——同一个输入像素，会被多个相邻滑窗读到，于是它在 `cols` 里出现了好多次。反向时，链式法则的铁律是：**一个变量参与了多个下游计算，它的梯度等于各路贡献之和**。所以 col2im 把 `dCols` 散回原图时，落到同一像素的多份梯度必须 `+=` 累加，而不是 `=` 覆盖。

这就是本章的专家钩子（✦）。看 `convBackwardSumLoss` 里那个用 `mode` 控制的散射循环（第 124-149 行），整章的胜负就压在那一行上：

```ts
// col2im: walk the SAME index map im2col used in forward, scatter dCols to source pixels.
const dInput = new Float64Array(N * inC * H * W);
// ... 四层循环走 (n, oy, ox) 滑窗位置 × (c, ky, kx) 核内偏移 ...
const dst = ((n * inC + c) * H + iy) * W + ix;
const g = colGradPerPatch[col];
// THE ONE LINE THAT MATTERS. += accumulates contributions from every window
// that reads this pixel; = throws away all but the last writer.
if (mode === "sum") dInput[dst] += g;
else dInput[dst] = g;
```

注意它走的是和 im2col 前向**完全相同**的索引映射（同样的 `iy = oy*stride + ky - padding`、同样的边界判断）——col2im 不是另一套逻辑，它就是把前向"读"的方向反过来变成"写"。配套仓核心引擎 `core/autograd.ts::im2col`（第 489-511 行）里真正给整个网络用的那份 col2im，写的也是同一行 `+=`：

```ts
if (iy >= 0 && iy < H && ix >= 0 && ix < W) {
  input.grad[((n * C + c) * H + iy) * W + ix] += t.grad[row * patch + col];
}
```

stage03 之所以不直接调用引擎那份（它已经对了），而是把 backward 在本文件里手写一遍，就是为了让 `+=` 和 `=` 变成一个可切换的旋钮，能当场把错的那版跑给你看（见该文件头部第 29-32 行的设计说明）。这是反 survey 稿的做法：不告诉你"要累加哦"，而是把覆盖版真的跑崩。

### 重叠到底有多严重：贡献计数图

光说"一个像素被多个窗口读到"太抽象。stage03 的 `overlapCountMap`（第 207-223 行）直接数：3×3 核、stride 1、pad 1 滑过 5×5 输入时，每个输入像素被多少个窗口读到。这是真实运行输出，不是估算：

```
[PART 2] overlap-count map (windows reading each input pixel, stride 1 pad 1):
      4  6  6  6  4
      6  9  9  9  6
      6  9  9  9  6
      6  9  9  9  6
      4  6  6  6  4
  per-pixel reads: min=4 max=9 mean=6.76
  overwrite (=) would force every one of these counts down to 1 (last writer wins).
```

中心像素被 9 个窗口读到，意味着它的梯度是 9 路贡献之和。如果你写成覆盖，等于把这 9 路里的 8 路直接丢掉，只留最后一个写入者——梯度幅度被压到原来的 1/9 到 1/4 不等。这不是小数点后的误差，是数量级错误，会让训练朝错误方向走或干脆不收敛。

## 用数值梯度检查钉死它

手推公式可能推错，代码可能敲错。判定卷积反向对不对，唯一靠谱的办法是**数值梯度检查**：对每个参数扰动一个极小量 `eps`，看 loss 变化除以 `2·eps` 得到的斜率（中心差分），和你解析算出来的梯度比。两者一致到机器精度，才算对。

stage03 的 `numericalGrad`（第 167-183 行）是标准实现，有一个容易写错、写错就静默污染后续所有元素的细节：

```ts
const orig = target.data[i];
target.data[i] = orig + eps;
const lp = loss();
target.data[i] = orig - eps;
const lm = loss();
target.data[i] = orig; // restore; a leaked perturbation corrupts later elements
g[i] = (lp - lm) / (2 * eps);
```

那行 `restore` 不是装饰——少了它，第 `i` 个元素的扰动会泄漏到测第 `i+1` 个元素时的基准上，让后面每个数值梯度都偏。这是中心差分实现里最常见的隐藏 bug。

比较用的是逐元素最大相对误差（`maxRelError`，第 187-195 行），分母用 `max(1e-8, |analytic|+|numeric|)` 避免梯度接近 0 时除爆。`stage03` PART 1 在 stride 1、pad 1（窗口重叠最狠）下跑正确的 `+=` 版本，真实输出：

```
[PART 1] correct col2im (+=), stride 1 pad 1 (overlapping windows)
  dInput  max rel error = 3.90e-10  PASS
  dWeight max rel error = 1.95e-10  PASS
```

`3.90e-10` 和 `1.95e-10` 都远在 `1e-5` 容差内。诚实说明：这是 1×2×5×5 输入、3×2×3×3 核的 toy 规模、双精度浮点（f64）跑出来的，绝对量级（1e-10）是"f64 + 中心差分"这套组合下正确算子的典型落点，不代表真实框架在大张量上的数字。可迁移的不是这个绝对值，而是后面那个**对比**：正确版 ~1e-10，写错版能到 O(1)。

## 失败模式：覆盖 bug 为什么会被 stride==kernel 静默放过

现在把旋钮拨到 `overwrite`，这是本章最有教学价值的部分。直觉会说"覆盖肯定立刻测出来啊"——错。它是不是错，取决于 stride。stage03 PART 3（第 295-338 行）用同一个错误的覆盖版 backward 跑两种 stride，真实输出：

```
[PART 3] FAILURE MODE — overwrite (=) col2im, stride-gated bug
  stride 1 (overlap)     dInput max rel error = 1.00e+0  FAIL  (expected FAIL: windows overlap)
  stride 3 == kernel     dInput max rel error = 7.40e-10  PASS  (expected PASS: max per-pixel reads = 1)
    (correct += backward here: 7.40e-10 — identical, because no pixel is read twice)
  VERDICT:
    overwrite under overlap (stride 1): rel err 1.00e+0  -> bug EXPOSED
    overwrite under no-overlap (stride 3): rel err 7.40e-10  -> bug HIDDEN
    A test suite using only stride==kernel would ship this bug silently.
```

读懂这三行就读懂了整章：

- **stride 1**：窗口重叠，像素被读多次（回看那张计数图，最高 9 次）。覆盖只留最后一个写入者，相对误差直接顶到 `1.00e+0`——100% 偏差，bug 当场暴露。
- **stride 3 == kernel**：3×3 核每步跳 3 格，窗口恰好平铺不重叠，每个像素只被读一次。此时"累加一份"和"覆盖一份"结果**完全相同**——覆盖版的相对误差是 `7.40e-10`，和正确版逐位一致。bug 被完美隐藏。

这就是危险所在：stride==kernel 是个极常用的默认配置——所有"用一个 stride=2、kernel=2 的卷积做降采样"的写法都落在这区域。一套**只测了 stride==kernel 的单测**会给这个覆盖 bug 发一张干净的通过证书，然后它跟着代码进生产；直到某天有人把某层换成 stride 1 的 same 卷积，训练莫名其妙不收敛,根因藏在几个月前那行 `=` 里。这是诊断起来最折磨的一类 bug——"它明明有测试且全绿"。

教训不是"记得用 +="，而是**梯度检查的输入配置本身是有覆盖度的**：测卷积反向必须包含 stride < kernel 的重叠情形，否则你测的是 col2im 里最不容易出错的那个退化子集。注意 PART 3 还顺带验证了 `dWeight` 在两种 mode 下完全相同（它不经过 col2im，是纯 matmul），所以这个 bug 是**只伤输入梯度、且被 stride 闸住的**——伤害面窄、触发条件刁钻，恰好是最难靠肉眼 code review 抓到的形状。

## ⚡ "卷积的反向也是一个卷积"——为什么没人这么实现

教科书会告诉你一个漂亮结论：卷积对输入的反向传播，本身就是一个卷积——具体说是用**空间翻转 180° 的核**对 `dOut`（补过零的）做卷积。这在数学上严格成立，是信号处理里相关与卷积对偶性的直接推论，也是"为什么 CNN 反向能高效"的经典解释。

但工程现实是：主流框架（PyTorch、cuDNN 那一层往下）的通用卷积反向，**几乎都不走"翻转核再卷一次"这条路**，而是走本章这条 im2col / col2im 复用 GEMM（通用矩阵乘）的路线。原因很实在——`dWeight` 那一步无论如何都是个矩阵乘（`dOutᵀ @ cols`），既然 GEMM 已经是被几十年优化榨干的最快算子，把 `dInput` 也表达成同一套 unfold + GEMM + col2im，整条反向就只依赖一个高度优化的 kernel，而不是再单独维护一个"翻转核卷积"的代码路径和它的边界处理。两种实现数学等价，工程成本天差地别。

为什么要同时知道这两种等价实现？因为它是看懂**深度可分离卷积**（depthwise separable convolution，把"逐通道空间卷积"和"逐点 1×1 通道混合"拆成两步，MobileNet 那套省算力的核心）反向为何那么便宜的钥匙。depthwise 部分每个通道独立、核极小，用"翻转核卷积"视角去想它的反向，会发现计算量和通道数解耦、几乎没有跨通道的大矩阵乘；只盯着 im2col/GEMM 视角反而看不清省在哪。

开放的部分在这里（⚡ **目前无通用最优解，仍在工程演进**）：到底哪种卷积反向实现最快，**没有一个跨所有形状的通用赢家**。im2col 把输入显式展开成 `cols` 矩阵，对小核、大通道很划算，但 `cols` 的显存占用是输入的 `kH·kW` 倍，大核或高分辨率下会爆显存；于是有 implicit GEMM（不显式物化 `cols`，在矩阵乘的取数阶段即时算索引）、Winograd（用类似快速傅里叶的代数变换把 3×3 卷积的乘法次数砍下来，但数值稳定性变差、只对特定核尺寸有效）、FFT 卷积（大核才划算）等多条路线并存。框架在运行时按"核尺寸 / 通道数 / 输入分辨率 / 硬件"去启发式地选（cuDNN 的 algorithm selection 就是干这个），而不是有一个永远最优的实现——这块至今仍是编译器和算子库在持续打磨的活地方。本章给你的 im2col/col2im 是**理解地基**，不是性能终点。

## 这一章交付了什么

到此，第 01-03 章拼出的东西第一次完整：第 01 章把卷积前向写成 im2col + matmul，第 02 章给了池化，本章给了卷积的反向——`dWeight` 是一次 matmul，`dInput` 是 col2im（重叠处**累加**），两条都用数值梯度检查钉到了 1e-10。这意味着配套仓 `core/autograd.ts` 里那个卷积层现在是一个**完整可训练的层**：前向能算、反向能回传、梯度可验证。

第 07-08 章要把这些层叠成一个真能跑的小 CNN 去训练——那时候 `loss.backward()` 一行就会把梯度顺着这条 col2im 链路全程倒灌回来。如果本章这条链路上藏着那个 stride 闸住的覆盖 bug，到第 07 章你看到的就只是"loss 不降"这个毫无指向性的症状。所以本章真正的交付不是公式，是那条铁律：**测卷积反向，输入配置必须包含 stride < kernel 的重叠情形**——否则你的梯度检查测的是 col2im 里最不会出错的退化角落。
