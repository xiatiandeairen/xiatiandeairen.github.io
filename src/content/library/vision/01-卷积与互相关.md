先看一个数字：把一个 8×8、左暗右亮、竖边卡在第 3|4 列之间的图，喂给一个 3×3 的 Sobel 竖边核（一种检测竖直边缘的固定权重矩阵），padding=1 保持 8×8，输出特征图第 4 列的值是 `2.95, 4.03, 4.04, ...`，而第 0、1、2 列全是 `±0.0x` 的噪声。边缘列被点亮了，平坦区归零。这就是卷积的全部秘密——一个 3×3 的小核，9 个数，在整张图上滑过去做点积，**核里的权重决定了它检测什么**。换个核（box-blur，9 个 1/9）跑同一张图，竖边被抹平，对比度从 1.056 掉到 1.011。conv2d 这个算子从头到尾没变，变的只有那 9 个数。

这一章我们把这个算子做出来，配套代码在 `examples/vision-from-scratch/src/stage01-convolution.ts`，核心算子在 `examples/vision-from-scratch/src/core/autograd.ts`。但在写第一行之前，得先拆掉一个几乎所有中文教程都埋着的雷。

## 你叫它卷积，但它是互相关

数学课本里的卷积有一步翻转：把核上下左右翻过来再做点积。深度学习框架里的 `Conv2d` **不翻转**——它做的是互相关（cross-correlation，核不翻转直接滑窗点积）。PyTorch、TensorFlow、cuDNN 全部如此，只是沿用了"卷积"这个名字。我们的 `conv2d` 也走这个约定，stage01 启动时第一行就打出来：

```
提示: 深度学习里的 'convolution' 实为 cross-correlation (核不翻转); conv2d 走此约定.
```

为什么要在第一段就钉死这件事？因为它不是学术洁癖。**翻转与否决定了第 03 章手推反向传播时梯度往哪个方向流**。真正的数学卷积，前向翻核、反向也翻核（翻转是它自己的逆操作的一部分）；互相关前向不翻，反向就是一个干净的转置（transpose）。如果你脑子里装着"卷积要翻转"，到第 03 章推 col2im（im2col 的反向）时会凭空多翻一次核，梯度全错，而且错得很隐蔽——loss 还在降，只是降得不对。所以现在就记死：**本书的卷积 = 互相关 = 不翻核**。

为什么框架敢偷换这个名字而不出问题？因为核是**学出来的**。翻转的核和不翻转的核只差一个镜像，反向传播会把权重学到正确的镜像位置，结果等价。翻转那一步纯属冗余计算，框架直接省了。但"训练时等价"不代表"实现时可以混"——你手写反向时必须前后一致，要么全翻要么全不翻，不能前向不翻反向翻。

## 一个核决定检测什么

先看竖边核的真实输出。输入是这张 8×8 图（左暗右亮，竖边在第 3|4 列之间，带一点 seeded 抖动让它不是完美阶跃，因为真实图像从不是）：

```
@@@@@@@@
@@@@@@@@
... (8 行) range [-0.029, 1.027]
```

用 3×3 Sobel 竖边核（列和为 0：`-1 0 1 / -2 0 2 / -1 0 1`），padding=1，输出：

```
   0.01    0.07   -0.01    2.95    3.00   -0.03   -0.03   -2.94
   0.00   -0.02   -0.05    4.03    4.07   -0.08   -0.05   -3.94
  -0.01   -0.07   -0.04    4.04    4.10   -0.06   -0.07   -3.95
   ... (8 行)  range [-4.054, 4.100]
```

代码自己验证了机制正确——它不是断言"答案应该是这样"，而是从实际输出里找出响应最强的列，确认它落在已知边界上：

```
响应最强的列 = 第 4 列 (输入竖边在 3|4 之间; 边缘核在过渡处响应最大 -> 机制正确)
```

注意三件事。第一，平坦区（第 0/1/2 列、第 5/6 列）响应都是 `±0.0x`，因为核的列和为 0，常数区域点积归零——这正是"边缘检测"的定义：只对变化敏感。第二，第 7 列是 `-2.94 ~ -3.95`，**负的**，因为右边界处亮度从高跌回（padding 补的 0），方向相反所以符号相反。第三，第 3 列和第 4 列都有强响应（`2.95` 和 `3.00`），因为 3×3 核跨在边界两侧时，左右两个位置都"看得见"这条边。**边不是一条数学上无宽度的线，它在特征图上有 1~2 列的厚度**，这个厚度等于核宽。

换成 box-blur 核（9 个 1/9，邻域取平均）跑同一张图：

```
对比度 (max-min): 原图 1.056 -> 模糊后 1.011 (模糊核把锐边抹平, 对比度下降)
```

同一个 `conv2d` 函数、同一张图，只换了 9 个权重，行为从"挑出边"变成"抹平边"。这就是本章想让你形成的肌肉记忆：**conv2d 是引擎，核是程序**。整个 CNN 的学习过程，就是用反向传播去搜索这些核里该填什么数。第 00 章说卷积靠权重共享逃出维度灾难——这里你看到了"共享"的物理含义：这 9 个数在 64 个像素位置上被复用了 64 次，参数量与图像大小解耦。

### ✦ 没人现场算卷积：它被转成矩阵乘

上面那张特征图，你脑子里大概想象的是"七层嵌套循环"：遍历每个输出像素、每个通道、每个核位置，累加点积。这个写法对，但**没有任何生产框架真这么算**。PyTorch / cuDNN 把卷积转成 **im2col + GEMM**（把滑窗摊平成大矩阵，再做通用矩阵乘）或 Winograd（一种用更少乘法换更多加法的快速变换）。原因很现实：矩阵乘（matmul / GEMM）是被工业界优化了几十年的内核，有 BLAS、有 cuBLAS、有专门的 tensor core 硬件单元，吞吐量比手写嵌套循环高几个数量级。**"卷积是矩阵乘的特例"不是学术比喻，是工程现实。**

我们这本书也走这条路——不是为了性能（toy 规模谈性能没意义），而是为了**反向传播**。看 core 里 conv2d 的实现（`examples/vision-from-scratch/src/core/autograd.ts`，第 525 行起）：

```ts
export function conv2d(input: Tensor, weight: Tensor, bias: Tensor | null, p: Conv2dParams): Tensor {
  const [outC, inC, kH, kW] = weight.shape;          // 权重布局 OIHW
  const [N, C, H, W] = input.shape;                  // 图像布局 NCHW
  if (C !== inC) throw new Error(`conv2d: input channels ${C} != weight inCh ${inC}`);
  // cols: (N*outH*outW, inC*kH*kW). wMat: (inC*kH*kW, outC) = weight reshaped+transposed.
  const cols = im2col(input, kH, kW, p);
  const wMat = weight.reshape([outC, inC * kH * kW]).transpose(); // (patch, outC)
  let outMat = cols.matmul(wMat);                    // ← 卷积在这一行变成了一次矩阵乘
  // ... bias add + permute 回 NCHW
}
```

整个前向核心就一行：`cols.matmul(wMat)`。`im2col` 把每个输出像素要"看"的那一小块（inC×kH×kW 个数）摊平成大矩阵的一行，权重摊平成另一个矩阵，一次矩阵乘搞定所有输出位置。

为什么这对反向传播是命根子？因为 `matmul` 的反向我们在 core 里已经写过、已经被 gradCheck（数值梯度校验）验证正确。conv2d 的反向于是**不用从头推**——它由 `im2col` 的反向 + `matmul` 的反向 + 一个 permute 的反向**自动组合**出来。看 autograd.ts 第 416 行的设计注释说得很直白：

```
WHY im2col at all: a direct 7-loop conv is correct but its backward must be hand-derived
  over 4 dims. Instead we UNFOLD: ... Then conv = (cols matrix) @ (weights reshaped to a
  matrix). The matmul adjoint is already tested, so the ONLY new gradient is col2im = the
  transpose of the unfold, which is small and local.
```

翻译：直接七层循环写前向，反向得在 4 个维度上手推梯度，极易错。改成 im2col 后，唯一需要新写的反向只剩 col2im（im2col 的转置），又小又局部。这就是第 03 章能轻松推卷积反向的全部铺垫——**摊平这一步把卷积的反向降维成了矩阵乘的反向**。

### NCHW 与 OIHW：布局不是细节

代码里两个形状约定，记死：图像是 **NCHW**（batch、channel、height、width），权重是 **OIHW**（输出通道、输入通道、核高、核宽）。`conv2d` 第一件事就是检查 `C !== inC` 直接抛错：

```ts
if (C !== inC) throw new Error(`conv2d: input channels ${C} != weight inCh ${inC}`);
```

为什么宁可抛错也不"自动适配"？因为通道数对不齐通常是上游建网时的笔误（比如忘了 RGB 是 3 通道、或上一层输出通道数记错），**静默截断或广播会让网络跑起来但学不出东西**，比直接崩溃难查一百倍。这是贯穿全书的一条原则：形状契约用硬错误守住，不给"看起来能跑"留后门。多通道在 im2col 里是怎么摊平的？看第 459 行，`patch = C * kH * kW`——每个输出像素的那一行，把所有输入通道的核覆盖区**首尾相接拼成一长条**，所以多通道卷积和单通道卷积在矩阵乘层面是同一个操作，只是行变长了。这是单通道→多通道几乎零成本扩展的原因。

## 输出尺寸公式：一格都不能错

卷积层的输出有多大？公式是 `floor((H + 2p - k) / s) + 1`，其中 H 是输入边长，p 是 padding，k 是核大小，s 是 stride（步长）。stage01 Part 2 把这个公式逐项核对——三条独立路径必须一致：手算公式、引擎的 `convOutSize` 辅助函数、以及真跑一次 conv2d 读输出形状：

```
stride pad | formula | engine | match
-------------------------------------
  1     0  |  6x6   | 6x6    | PASS
  1     1  |  8x8   | 8x8    | PASS
  2     0  |  3x3   | 3x3    | PASS
  2     1  |  4x4   | 4x4    | PASS
  3     0  |  2x2   | 2x2    | PASS
全部一致: 手算公式 == convOutSize == 实跑输出形状
```

`convOutSize` 就是公式的直接翻译（autograd.ts 第 439 行），它是 conv2d 内部计算输出大小的**唯一真源**——公式和实际跑出来的形状用的是同一段代码，所以"公式即实现"不是口号：

```ts
export function convOutSize(inSize: number, kernel: number, stride: number, padding: number): number {
  return Math.floor((inSize + 2 * padding - kernel) / stride) + 1;
}
```

最该盯的是 stride=3 那一行。`(8 - 3) / 3 = 1.67`，floor 取整成 1，+1 = 2，输出 2 列：

```
注意 stride=3 行: (8-3)/3 = 1.67, floor -> 1, +1 = 2 列; 最后那个不满一格的窗口被丢弃,
不是补零凑整 (这是 'valid' 语义).
```

**那个 floor 是有代价的**：当 stride 除不尽时，输入末尾凑不满一个完整窗口的部分被**直接丢弃**，不是补零凑整。8 列里第 6、7 列在 stride=3 时根本没参与任何一次卷积。这叫 "valid" 语义——只算能放下完整核的位置。很多人看到输出尺寸不对，第一反应是"框架 bug"，其实是 floor 在悄悄丢边角。

### ⚡ 一格错位，整张图歪掉：尺寸公式没有通用自动解

这条公式看着像小学算术，却是 U-Net、目标检测器里 shape mismatch 报错的**头号原因**。原因在于现代网络要把多个尺度的特征图对齐相加或拼接（U-Net 的跳连、FPN 的特征金字塔、ResNet 的残差加法）。一旦某层 padding 少补一格、或 stride 多走一步，特征图就差一两个像素，到拼接那一步整张图错位——而且 loss 通常还能降，只是降得不对，**最难查的那种 bug**。

框架给的"通用解"是 `'same'` padding——你声明输出要和输入一样大，框架反推该补几格。但这只是把问题挪了位置：当 stride > 1、或核是偶数、或图边长是奇数时，`'same'` 需要**非对称补零**（左补 1 右补 2 这种），不同框架（TensorFlow vs PyTorch）补法不一致，迁移模型时是经典踩坑点。**所以"输出尺寸自动对齐"至今没有跨框架的通用解**——`'same'` 是各家自己定义的约定而非数学定理，自己实现时必须老老实实算 `floor` 公式，一格都不能含糊。这就是为什么我们在 stage01 里不嫌烦地三路核对：公式、helper、实跑形状，三者对上才敢往下走。

## 失败模式：漏掉 padding，下游才爆炸

最后是初学者最常摔的坑，也是上面那条 ⚡ 的具体犯罪现场。stage01 Part 3 演示：8×8 图用 3×3 核但 **padding=0**，输出缩成 6×6。

```
输入 8x8, 用 3x3 核但 padding=0 -> 输出 6x6 (每边各丢 (k-1)/2 = 1 行/列, 因为边界像素凑不齐完整窗口).
边界信息丢失: 第 0 行/列和最后一行/列从不作为窗口中心, 它们的边缘响应永远算不到.
```

为什么缩水？因为没补零时，图的最外圈像素无法当窗口中心——3×3 核要中心的话，中心至少得离边 1 格。所以每边丢 `(k-1)/2 = 1` 行/列，8 变 6。**附带损失**：边界像素的信息永久丢失，它们从不作为任何窗口的中心。这在浅网络里无所谓，但堆 20 层、每层丢一圈，深层特征图会"缩水到看不见原图边缘"——这也是为什么深网络默认到处用 padding。

致命的不是缩水本身，是它**何时爆炸**。下游有段天真代码想做残差连接（residual，`输出 = 输入 + conv(输入)`，ResNet 的核心结构）：

```
下游代码 (天真地以为尺寸不变, 想做 residual: input + conv(input)):
  抛错: elementwise(+): cannot broadcast shape 1,1,6,6 into 1,1,8,8
  根因不在这一行 — 真正的错在上游 conv 漏了 padding=1. 报错点离病灶很远,
  这正是 '尺寸缩水' bug 难查的原因.
```

报错在 `add` 这一行，但病根在上游 conv 漏了 `padding=1`。**报错点和病灶隔着十万八千里**——这正是尺寸缩水类 bug 难查的本质：错误不在出错的地方暴露，而在下一个假设"尺寸不变"的算子那里才崩。修复也简单，padding=1 让 conv 输出回到 8×8，残差加法成功：

```
修复: padding=1 -> conv 输出 8x8, residual add 成功, 结果形状 8x8.
```

注意这里 core 的 `Tensor.add` 选择了**抛错而非自动广播 6×6 到 8×8**。这是刻意的：自动广播会让形状错误静默通过，残差加法变成"加了张错位的图"，网络照样训练但学的是垃圾。**让形状契约在第一时间崩溃，是把"难查的语义 bug"换成"好查的崩溃"** ——本书所有算子都遵守这条。这也回扣本章开头那个判断：在卷积这层，宁可早崩，不要静默错。

## 小结与下一章

这一章你拿到了三样东西：(1) 一个判断——深度学习的卷积是互相关，核不翻，记死它第 03 章才不翻错；(2) 一个引擎视角——conv2d 通过 im2col 变成矩阵乘，前向就一行 `cols.matmul(wMat)`，这是 PyTorch/cuDNN 的真实做法也是本书反向能复用 matmul 的关键；(3) 一条不能错的公式——`floor((H+2p-k)/s)+1`，一格错位整张图歪，至今无跨框架通用解。

第 02 章讲 pooling（池化，下采样的另一种手段），看它和 stride 卷积在降分辨率上的分工与取舍。第 03 章兑现本章埋的两张支票：手推 conv2d 的反向，你会看到 im2col 的"摊平"如何让卷积反向坍缩成 matmul 反向 + 一个 col2im 转置——以及为什么互相关约定让这个转置干净到不需要翻核。
