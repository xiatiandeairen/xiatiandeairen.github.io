先抛一个判断:在我带过的工程师里，能把一个含糊的业务需求干净地"建成图"的人，和不能的人，产出质量差一个量级。不是因为图算法多难——Dijkstra 二十行——而是因为"看出这是图问题"这一步本身就是离散数学训练出来的抽象肌肉。依赖解析是图，服务路由是图，任务调度是图，社交关系是图，编译器的寄存器分配是图，连"哪些数据库迁移能并行跑"也是图。你一旦学会把现实拍扁成"节点 + 边"，一大类问题立刻有了现成的多项式算法可用；学不会，就会在 for 循环里手搓一个最终被证明是错的暴力。

这章不讲教科书顺序。我按工程师真正会撞上的四件事组织:**怎么把问题建成图并用对算法**、**怎么数清楚有多少种情况（以及为什么数不清的时候系统就崩了）**、**怎么用代码把"对所有 n 成立"这种数学声明逼成可证伪的东西**、以及贯穿全章的那条铁律——**复杂度的增长形状决定一个算法是工具还是玩具**。配套代码全在 `examples/math-for-builders-from-scratch/src/stage05-discrete.ts`，下面引用的每个数字都是它在我本机真跑出来的，不是我编的。

## 把问题建成图:工程师最高频的抽象动作 ✦

图就是"一堆东西 + 它们之间的连接"。`stage05` 里我没用矩阵存图，用的是邻接表（adjacency list，每个节点挂一个出边列表）:

```ts
interface Edge {
  to: number;
  weight: number;
}
type Graph = Edge[][]; // graph[u] = outgoing edges of node u
```

为什么是邻接表不是邻接矩阵?这是第一个有取舍的工程决策，不是审美。矩阵是 `n×n`，无论图多稀疏都吃 `O(n²)` 内存；现实里的依赖图、路由图绝大多数是稀疏的（每个节点连几条边，不是连所有节点）。Dijkstra 在邻接表上是 `O(E log V)`，在矩阵上扫每个节点的邻居要 `O(V²)`。当你的服务有一万个节点、每个平均连五条边，矩阵是一亿格子，邻接表是五万条边——差三个数量级。代码注释里写得很直白:用列表是因为"which edges does a node have"这个问题对读者一目了然，而且 Dijkstra 在列表上的代价是 `O(E log V)`。

### Dijkstra:贪心为什么能给出全局最优

最短路是图问题的 hello world。`stage05` 在一张 8 节点的种子图（用第 3 章那套 `mulberry32` 伪随机数生成器播种，保证可复现）上跑 Dijkstra，实测:

```
0 → 7 最短距离  = 10
最短路径          = 0 → 4 → 7
暴力枚举对拍      = 10 (一致 ✓)
```

注意那行"暴力枚举对拍"。我没有只信 Dijkstra 的输出，而是另写了一个 `bruteForceShortest`，DFS 穷举所有从源到目标的简单路径取最小值，拿它当 ground truth（标准答案）和 Dijkstra 对账。两者都是 10，才敢说算法对了。这是本书的方法论:**闭式/聪明算法 vs 暴力穷举对拍**，闭式可能写错，暴力慢但笨得不会错（在小规模上）。

Dijkstra 是贪心算法——每步选当前距离最小的未访问节点，宣布它"定终"。凭什么这一步贪心能凑出全局最优?核心不变量在代码注释里:

```ts
/* CORE INVARIANT (the thing that breaks on negative edges): once a node is popped as the
 * current minimum among unvisited, its distance is FINAL — because every remaining path to
 * it must go through some unvisited node whose tentative distance is already ≥ this one, and
 * adding non-negative edges can only increase it. Negative edges violate "can only increase". */
```

翻译成大白话:一个节点一旦被选为"当前最小"，它的距离就锁死了——因为任何还没走的路径都得先经过某个未访问节点，而那个节点的暂定距离已经 ≥ 它了，再加非负的边只会更大、不可能更小。**这条证明的命门是"加边只增不减"，它依赖所有边权非负。** 记住这句话，下一节它会以最阴险的方式塌掉。

### 二分匹配:同一个增广路径思想的另一张脸

把工人分配给任务、把订单派给骑手、把 GPU 卡分给训练任务——这些是同一个问题:二分匹配（bipartite matching，两边节点配对，每边各用一次）。`stage05` 跑了 4 工人 × 4 任务的种子可行表:

```
工人0 可做任务 = {0,1}
工人1 可做任务 = {1,2,3}
工人2 可做任务 = {1,2}
工人3 可做任务 = {0,3}
最大匹配 = 4 对：工人3→任务0, 工人0→任务1, 工人2→任务2, 工人1→任务3
```

这里有个反直觉的点值得停一下。工人0 只能做 {0,1}，工人3 只能做 {0,3}——它俩都想要任务0。贪心地按工人顺序分配会卡住:工人0 先抢了任务0，工人3 来的时候任务0 没了，但工人3 只剩任务3 可选——如果任务3 又被工人1 占了呢?这就是为什么不能简单贪心。`maxBipartiteMatching` 用的是**增广路径**（augmenting path）:

```ts
const tryAugment = (l: number, seen: boolean[]): boolean => {
  for (const r of adj[l]) {
    if (seen[r]) continue;
    seen[r] = true;
    // Right node r is free, OR its current owner can be re-routed elsewhere.
    if (matchR[r] === -1 || tryAugment(matchR[r], seen)) {
      matchR[r] = l;
      return true;
    }
  }
  return false;
};
```

关键在 `tryAugment(matchR[r], seen)` 这个递归:如果工人 l 想要的任务 r 已被别人占了，就**反过来问那个占用者能不能改去别处**。这是 Kuhn 算法（匈牙利算法的核），背后是 Berge 定理——一个匹配是最大的，当且仅当不存在增广路径。它和最大流在单位容量图上是同一个东西。所以最终能凑出满匹配 4 对，靠的就是这种"挤一挤、腾个位"的重路由。

代码里有个一行注释标了个真实陷阱:`seen` 数组必须每个左节点重置一次。

```ts
const seen = new Array<boolean>(numRight).fill(false); // fresh per search (see INVARIANT)
```

为什么?`seen` 的作用是防止一次增广搜索里无限重路由（A 让 B 腾位、B 又让 A 腾位、死循环）。但如果跨节点不重置，上一个工人的 `seen` 会挡住下一个工人合法的重路由。这是写匹配算法最常见的 bug——共享了本该隔离的 visited 状态，结果匹配数莫名其妙偏小，还不报错。又是"沉默的错"，下面要专门讲。

## 组合计数:能数清楚，才知道暴力会不会炸

第二根支柱是计数。听起来小学水平，但它直接决定一个朴素算法是 `O(n²)` 还是 `O(n!)`——而后者会要你的命（下一节用实测数字证明）。

`stage05` 把闭式公式和暴力穷举对拍:

```
P(6,3) = 120   C(6,3) = 20
暴力数 6 元集的 3-子集 = 20 (= C(n,r) ✓)
```

排列 `P(6,3)=120`（有序，关心顺序），组合 `C(6,3)=20`（无序，只关心选了谁）。但光打印两个公式的结果证明不了什么——你怎么知道 `n!/(r!(n-r)!)` 真的在数"3 元子集"这个东西，而不是凑巧算出个 20?所以 `bruteForceCountSubsets` 老老实实枚举全部 2⁶=64 个 bitmask，数恰好有 3 个 bit 的:

```ts
function bruteForceCountSubsets(n: number, r: number): number {
  let count = 0;
  for (let mask = 0; mask < 1 << n; mask++) {
    let bits = 0;
    for (let b = 0; b < n; b++) bits += (mask >> b) & 1;
    if (bits === r) count++;
  }
  return count;
}
```

暴力数出来 20，跟 `C(6,3)` 一致。代码注释说得很重:**一个你没法从第一性原理复现的公式就是民间传说（folklore）。** 这正是本书要做的反面——AI 批量生成的 survey 稿最爱直接甩一个公式让你背，从不告诉你这公式数的到底是不是你以为的那个对象。

### 容斥:为什么"各自加起来"是错的

`stage05` 算了 `[1,1000]` 里能被 2、3 或 5 整除的数:

```
容斥：[1,1000] 中能被 2/3/5 整除的数 = 734 (暴力 734, 一致 ✓)
```

天真做法是 `⌊1000/2⌋ + ⌊1000/3⌋ + ⌊1000/5⌋ = 500+333+200 = 1033`——但 6 的倍数被 2 和 3 各数了一次，重复了。容斥原理（inclusion-exclusion）是系统性的修正:加单个集合、减两两交集、加三者交集。

```ts
function inclusionExclusionDivisible(limit: number): number {
  const div = (k: number) => Math.floor(limit / k);
  // +singles − pairs(lcm) + triple(lcm). All inputs coprime-ish so lcm = product here.
  return div(2) + div(3) + div(5) - div(6) - div(10) - div(15) + div(30);
}
```

`500+333+200 - 166-100-66 + 33 = 734`，和暴力扫一遍 `[1,1000]` 数出来的 734 完全对上。为什么工程师该认得它?因为它是概率并集 `P(A∪B∪C)`、错排（derangement）、以及埃拉托斯特尼筛法去重的同一根骨架。你在做"统计有多少用户命中了这些 feature flag 任意一个"时，如果直接把各 flag 命中数加起来报给老板，就是犯了那个 1033 的错。

这里埋一个 ⚡ 的引子:`stage05` 的容斥能写成闭式，是因为 2、3、5 两两互质（coprime），所以交集的大小恰好等于 `⌊limit / 乘积⌋`。一旦集合之间的交叠结构是任意的、约束之间相互纠缠，容斥的项数会随集合数指数膨胀（n 个集合有 2ⁿ−1 项），**一般情形下的精确计数本身就是 #P-hard（计数复杂度类，比 NP 还硬）**——这是第 9 章计算理论要展开的地基。我在这里只标一句:计数问题的"难"和判定问题的"难"是两套理论，目前没有把任意计数问题变高效的通用解。

## 归纳证明:把"对所有 n 成立"逼成可证伪

数学家说 `∑(1..n) = n(n+1)/2 对所有 n 成立`，靠的是归纳法——证基础情形 `n=1`，再证"`n` 成立则 `n+1` 成立"。工程师没法在代码里跑"对所有 n"（n 是无穷的），但能做一件有力的事:**把闭式公式和逐步累加在前 N 个 n 上逐一对账，找第一个不一致的 n**。

```ts
for (let n = 1; n <= maxN; n++) {
  runningSum += n;
  runningSq += n * n;
  const closedSum = (n * (n + 1)) / 2;
  const closedSq = (n * (n + 1) * (2 * n + 1)) / 6;
  if (firstFailSum === -1 && runningSum !== closedSum) firstFailSum = n;
  if (firstFailSq === -1 && runningSq !== closedSq) firstFailSq = n;
}
```

`runningSum` 是一步步真加出来的（不可能算错，它就是定义），`closedSum` 是公式。N=10000 的实测:

```
∑(1..n)=n(n+1)/2        : 前 10000 个 n 全部成立 ✓
∑(i²)=n(n+1)(2n+1)/6    : 前 10000 个 n 全部成立 ✓
```

说清楚这是什么、不是什么。**这不是证明**——通过 10000 个 case 不等于对所有 n 成立，数学归纳法才是。但它是极强的可证伪性:如果你的闭式公式抄错了（比如把 `2n+1` 写成 `2n−1`），这段代码会立刻吐出第一个失败的 n，而不是让你在生产里用着错公式算分页/容量直到某天爆掉。逐步累加在这里扮演的角色，和上一节暴力穷举对拍闭式是同一个角色:**用笨办法守住聪明办法**。`firstFailSum === -1`（从未失败）就是经验证据，不是定理证书，这个区别我必须给你点破——它正是机械验证（test）和形式证明之间那道永远填不平的缝。

为什么挑这两个和式?因为它们是复杂度分析的日常货:`∑(1..n)` 是单层累加循环的总操作数，`∑(i²)`（更准确说是 `∑i`）是嵌套循环 `for i { for j<i }` 的三角形操作数。你算一段代码"到底跑多少次"时，用的就是这俩闭式。

## 复杂度:增长形状决定工具还是玩具,组合爆炸如何瞬间不可行 ⚡

这是全章的高潮，也是配套仓里我最想让你盯着看的数字。`stage05` 给每个复杂度类挑一个代表算法，在增长的 n 上实测 wall-clock（墙钟时间），取多次中位数:

```
n      O(n)      O(n log n)   O(n²)       O(2ⁿ)
n=4     0.0021    0.6560      0.0514     0.0013
n=8     0.0041    1.5012      0.2084     0.0230
n=12    0.0040    2.2907      0.4592     0.1119
n=16    0.0057    3.1646      0.9605     2.1925
n=20    0.0078    4.1373      1.2658    38.3030
(校验和 633513060, 防 JIT 把循环当死代码删掉)
```

先讲怎么诚实地读这张表，再讲它说了什么。

**诚实第一条:别横向比绝对值。** 注意代码里每列的 n 量纲不同——`O(n)` 喂的是 `n×1000`，`O(n²)` 喂 `n×100`，`O(2ⁿ)` 喂 `n` 本身。所以你不能说"`n=4` 时 `O(n log n)` 比 `O(2ⁿ)` 慢"就得出 nlogn 更差的结论，那是拿不同输入规模比。**要看的是每一列随 n 的增长形状**，列内纵向比。这是测复杂度时最常见的误读，我自己也踩过。

**诚实第二条:那个校验和 633513060 不是装饰。** 每个 work 函数都返回一个真实结果累加进 sink:

```ts
function timeMedianMs(fn: () => number, repeats: number): { ms: number; sink: number } {
  const samples: number[] = [];
  let sink = 0;
  for (let r = 0; r < repeats; r++) {
    const t0 = performance.now();
    sink += fn(); // accumulate result so the work is observable (anti dead-code-elimination)
    samples.push(performance.now() - t0);
  }
  ...
}
```

如果不消费返回值，V8 的 JIT 编译器会发现"这个循环的结果没人用"，直接把整个循环优化删掉——你测出来的就是 0ms，得出"指数算法很快"的灾难性错误结论。这是第 12 章硬件那章会专门展开的诚实陷阱:**你以为你在测算法，其实在测编译器有多聪明地骗了你。** 取中位数也是同理:亚毫秒级的单次测量被定时器精度和 JIT 预热噪声主导，单跑一次的数字是废的。

现在看形状。`O(n)` 那列从 0.0021 到 0.0078，n 翻了 5 倍时间也大致线性微增——这是你想要的算法。`O(n²)` 从 0.0514 到 1.2658。重点是 `O(2ⁿ)` 那列:

```
--- 失败模式 A：指数级问题上暴力 ---
   O(2ⁿ) 从 n=4 到 n=20：实测 0.0013ms → 38.3030ms
   n 每 +1 工作量翻倍：2²⁰≈1e6 子集尚可，2⁶⁰≈1.15e18 子集。
   按本机 n=20 单子集耗时外推 (est.)：n=60 需约 1.34e+3 年 —— 必须换多项式算法或剪枝。
```

`exponentialWork` 枚举 n 元集的全部 2ⁿ 个子集:

```ts
function exponentialWork(n: number): number {
  let evenSubsets = 0;
  for (let mask = 0; mask < 1 << n; mask++) {
    let sum = 0;
    for (let b = 0; b < n; b++) if ((mask >> b) & 1) sum += b;
    if ((sum & 1) === 0) evenSubsets++;
  }
  return evenSubsets;
}
```

`n` 每 +1，子集数翻倍。`n=20` 是 ~100 万个子集，本机 38ms，毫无压力。但 `n=60` 是 ~1.15×10¹⁸ 个子集。**按本机 n=20 的单子集耗时外推（这是估算，标注 est.，不是实测），n=60 要约 1.34×10³ 年。** 这就是组合爆炸:一个看起来人畜无害、只比 n=20 大了 3 倍的问题，暴力枚举从 38 毫秒变成一千多年。stage 里还画了条 ASCII 曲线，前面四个点几乎贴在 0 上，最后一个点孤零零冲到 38.3——指数曲线的典型样子,前期骗你"挺快的嘛",到墙根前才暴起:

```
38.3 │    ●
0.00133 │●●●●
```

这条曲线就是为什么需要"更聪明的算法"的全部理由——动态规划、剪枝、近似、贪心、把问题映射到已知有多项式解的图问题（比如本章开头的最短路/匹配）。**当你发现自己在 for 循环里枚举所有子集/所有排列/所有组合，停下来，这个 n 一旦超过 30 你就完了。**

这里给出本章的 ⚡ 开放问题，必须诚实标注:那些被怀疑没有多项式算法的问题（旅行商、子集和、布尔可满足性 SAT 等 NP-hard 问题），**到底是不是真的没有高效解,即 P 是否等于 NP,是计算机科学最大的未解之谜，目前无通用解、悬赏一百万美元、绝大多数人相信 P≠NP 但没人证出来**。这意味着:面对一个 NP-hard 问题，你不是"还没想到聪明算法"，而是很可能根本不存在(但谁也没法保证)。务实的工程态度是认出它属于这类、然后用近似/启发式/限制问题规模绕过——而不是继续优化你那个注定要跑一千年的暴力循环。这道墙的精确形状，第 9 章计算理论会从可计算性和复杂度类的角度正式刻画。

## 失败模式:Dijkstra 遇负权,沉默地给你错答案

前面三节我反复说"暴力守住聪明算法"。这一节告诉你不守会怎样——而且是最危险的那种怎样:不崩溃、不报错、安静地返回一个错的数。

回到 Dijkstra 那条命门:"加边只增不减"依赖边权非负。`stage05` 精心构造了一张带一条负边的图来引爆它:

```ts
// edges: 0→1 (1), 0→2 (2), 1→3 (1), 2→1 (−5);  target = 3.
const negGraph: Graph = [
  [{ to: 1, weight: 1 }, { to: 2, weight: 2 }],
  [{ to: 3, weight: 1 }],
  [{ to: 1, weight: -5 }],
  [],
];
```

实测:

```
--- 失败模式 B：Dijkstra 遇负权悄悄给错答案 ---
   Dijkstra 给出 0→3 = 2
   暴力(正确) 0→3   = -2
   相差 4 (偏大)
```

事故的具体时序（代码注释和输出都说清了）:节点1 被早早"定终"在 dist=1，并立刻据此把节点3 松弛到 2。**之后**负边 `2→1`（权重 −5）才把节点1 的真实距离压到 −3——但此时节点3 早已被标记完成、不再回头更新。于是 Dijkstra 报 `0→3 = 2`，而真实最短是 `−3+1 = −2`，偏大了 4。`bruteForceShortest` 因为穷举所有路径、不依赖任何"加边只增"的假设，所以无论边权正负都给出正确的 −2。

这就是我要你刻进骨子里的那条:

```
Dijkstra 不报错、不崩溃，直接返回偏大值 —— 这种"沉默的错"最危险，负权要用 Bellman-Ford。
```

崩溃是仁慈的——它逼你立刻去修。沉默的错是恶毒的:你的路由系统、你的成本优化、你的调度器会带着这个偏大的距离照常运行，财报、SLA、用户体验里悄悄渗进系统性偏差，等你发现时已经污染了一片下游数据。**算法的前置条件（precondition）不是教科书的装饰，是合同。** 边权可能为负（退款、返利、套利、能量回收），就必须换 Bellman-Ford（能处理负权，代价是 `O(VE)` 比 Dijkstra 慢）或 SPFA。这和第 2 章统计里"用错了分布假设、估计照样给你一个数"是同一类病:**模型/算法不知道自己的假设被违反了，它只会继续算。** 把假设写进 `assert`、写进对拍测试、写进 code review 的检查项,是你唯一的防线。

## 这章给你留下什么

四根支柱，一条铁律。图让你把现实问题塞进现成的多项式算法（最短路、匹配都是 `O(多项式)`，因为前人把结构嚼透了）；计数让你在动手前就预判暴力会不会炸；归纳/对拍让你把"对所有情况成立"从信仰变成可证伪的代码;复杂度的增长形状是判断一个算法是工具还是玩具的唯一标准——`n=60` 的指数算法要跑一千年，这不是优化能救的，是必须换算法或承认问题本身 NP-hard。

而那条贯穿全章的铁律是:**笨办法（暴力穷举、逐步累加）的存在不是为了用在生产，是为了在小规模上守住聪明办法的正确性。** stage05 里每一个聪明算法旁边都站着一个暴力对拍——这正是本书对抗"AI 生成的、看着对其实没验证过的 survey 稿"的方式。Dijkstra 的负权事故告诉你不守会怎样:沉默地错下去。
