你训练一个分类器,损失函数那一行写的是 `cross_entropy(logits, labels)`,你大概率没想过:这个数字的单位是 bit,它字面意思就是"你的模型每个样本比真分布多浪费了几个二进制位去描述答案"。同样,你 `gzip` 一个日志文件压到原来的 1/5,这个 5 也不是随便能再提的——它被这个文件的熵卡死了,再聪明的算法也越不过去。压缩、编码、ML 损失,工程师平时当三件事,信息论里是一个量。这一章用一份 200 行的 TypeScript(代码在 `examples/math-for-builders-from-scratch/src/stage07-infotheory.ts`,所有数字都从真实计数跑出来,不是手算估的)把这三者对上号。

## 熵:一个量纲是 bit 的下界

熵(entropy,一句话:平均每个符号至少要花几个 bit)是这一章所有东西的地基。香农给的定义是 `H = -Σ p·log2(p)`,以 2 为底所以单位是 bit。直觉上它回答的是:"我要把一个来自分布 p 的符号告诉你,平均最少问几个是/否问题?"

stage 里这个函数刻意不做归一化:

```typescript
function entropyBits(probs: readonly number[]): number {
  let h = 0;
  for (const p of probs) if (p > 0) h -= p * Math.log2(p);
  return h;
}
```

两个设计点值得停一下。第一,`if (p > 0)` 不是防御性 hack,是数学约定:`0·log0` 的极限是 0,但 IEEE 浮点里 `log2(0) = -Infinity`,真去算它会把整个 sum 污染成 NaN。跳过 p=0 项是对的。第二,它**不**重新归一化。如果调用方传进来的概率和不是 1,它就算出一个错的数,而不是悄悄"帮你修好"。这是故意的——一个 bug 的分布应该以错数字的形式暴露,而不是被静默掩盖。这跟上一章数值稳定性的态度一脉相承:宁可炸出来,不要假装正常。

跑真实文本(代码里嵌了一段 182 字符的英文):

- 文本长度 = 182 chars,去重符号数 = 30
- 经验熵 H = **4.2973 bits/char**
- 定宽编码需要 = **5 bits/char**(`ceil(log2 30)`)
- 熵比定宽省 **0.70 bits/char**

读法是这样:30 个不同字符,最笨的办法是每个字符固定分 5 bit(因为 `2^5=32 ≥ 30`)。但这 30 个字符出现频率不均(空格和 e 多,标点少),熵告诉你信息量其实只有 4.30 bit/char。那 0.70 bit/char 的差,就是"利用频率不均"能省下来的理论上限。注意"理论上限"——熵不告诉你怎么压,只告诉你最多能压到哪。怎么压是下一节的事。

### 失败模式:把 nats 当 bits 读

这是信息论代码里最高频的单位 bug,stage 专门留了个对照函数 `entropyNats`(用 `Math.log` 自然对数而非 `Math.log2`)来演示。同一个分布:

- H 用 log2 = **1.7427 bits**
- H 用 ln = **1.2080 nats**
- 比值 = 1.4427 = 1/ln2

陷阱在哪:很多深度学习框架的 loss 默认用自然对数(nats),你看到 loss 打印 `1.21` 觉得比 `1.74` "更好"。其实它俩是**同一个量、不同单位**。把 1.21 nats 当成 1.21 bits 读,你会低估熵 31%。如果你在两个用不同 log 底的框架之间比 loss,或者把论文里的 nats 数字跟自己的 bits 数字对照,这个 bug 让你得出完全错误的结论。记住:loss 没标单位时,先确认它是 `log2` 还是 `ln`。

## 霍夫曼编码:逼近熵下界的具体算法

熵是下界,霍夫曼编码(Huffman coding,一句话:给高频符号短码、低频符号长码)是逼近这个下界的经典构造。算法贪心:反复合并当前权重最小的两个节点。

```typescript
function buildHuffmanTree(counts: Map<string, number>): HuffNode {
  if (counts.size === 0) throw new Error("[stage07] huffman: empty alphabet");
  const nodes: HuffNode[] = [...counts].map(([symbol, weight]) => ({ symbol, weight }));
  while (nodes.length > 1) {
    nodes.sort((a, b) => a.weight - b.weight);
    const left = nodes.shift()!;
    const right = nodes.shift()!;
    nodes.push({ weight: left.weight + right.weight, left, right });
  }
  return nodes[0];
}
```

频率越高的符号合并得越晚、离根越近,码就越短。跑出来的高频符号码长(节选):

```
" " │██████████████████······│ 3.000
"o" │██████████████████······│ 3.000
"t" │████████████████████████│ 4.000
"e" │████████████████████████│ 4.000
```

空格 3 bit、t 和 e 4 bit——全都比定宽的 5 bit 短。这就是压缩的来源。

结果对账:

- Huffman 平均码长 = **4.3297 bits/char**
- Shannon 下界 H = **4.2973 bits/char**
- 冗余(码长 − H) = **0.0324 bits/char**
- 压缩比 vs 定宽 5bit = **1.155×**(910 → 788 bits)
- 压缩比 vs ASCII 8bit = **1.848×**(1456 → 788 bits)

两个关键事实。其一,平均码长 4.33 **严格大于等于**熵 4.30——这是香农信源编码定理的下半句,码长不可能低于熵。stage 里有一行断言专门检查这个不变量,如果 `avgCodeLen < H` 就报"违反!不可能"。这不是装饰,是 sanity check:任何声称"压到熵以下"的无损算法都是算错了或者偷偷丢了信息。其二,冗余 0.0324 bit/char,定理保证它**严格小于 1**——霍夫曼的代价是它一次编一个符号,凑不出小数 bit,所以每符号最多浪费不到 1 bit。想吃掉这最后一点冗余,就得上算术编码(arithmetic coding)按整段编而非逐符号,但那是另一套机器了。

这里也暴露霍夫曼在工程上的真实局限:1.848× 的压缩比对真实文件其实不算高。原因是霍夫曼只利用**单字符频率**,完全无视上下文。真实文本里 "th" 后面大概率是 "e","qu" 几乎必然连着出现——这些是字符**之间**的相关性,霍夫曼看不见。gzip 之类之所以能压到 5×、10×,是因为前面挂了个 LZ 字典(把重复的子串替换成短引用),霍夫曼只是它最后一道熵编码工序。所以"为什么我的压缩比上不去"的答案往往不是熵编码不够好,而是你没先消掉序列冗余。

`buildCodes` 里还藏了一个边界 case 值得一提:如果整个文本只有一种字符,霍夫曼树退化成单个叶子,自然走出来的码是空串 `""`,而空串是没法编码的(前缀码里你无法用 0 个 bit 表示一个符号)。代码把它兜底成至少 1 bit 的 `"0"`:

```typescript
codes.set(node.symbol, prefix === "" ? "0" : prefix);
```

这种"看起来永远不会发生"的分支恰恰是真实系统崩在边界上的地方——单符号输入并不罕见(全 0 的初始化文件、单字符日志)。

## ✦ 交叉熵损失:不是随便选的,它就是最优编码代价

现在把上面两节接到 ML 上。这是这一章的核心钩子:**交叉熵损失不是某个数学家拍脑袋挑的损失函数,它字面就是"你用错误的编码方案,平均每个样本多花几个 bit"**。

交叉熵 `H(p, q) = -Σ p·log2(q)`:数据真实服从分布 p,但你用了一套为 q 优化的编码。

```typescript
function crossEntropyBits(p: readonly number[], q: readonly number[]): number {
  let ce = 0;
  for (let i = 0; i < p.length; i++) {
    if (p[i] > 0) {
      if (q[i] <= 0) return Infinity; // q 说不可能的事却发生了
      ce -= p[i] * Math.log2(q[i]);
    }
  }
  return ce;
}
```

训练分类器时,p 是数据的真实标签分布(one-hot 或软标签),q 是你模型 softmax 出来的预测。最小化交叉熵,就是在让你的编码方案(模型预测)逼近最优编码方案(真分布)。stage 跑了个 4 符号的玩具例子:

- H(p) = **1.7427 bits**(真分布的最优码长,也就是无论如何省不掉的下限)
- H(p, q) = **2.0000 bits**(用 q 的码编码 p 的数据)
- D(p‖q) = **0.2573 bits**(= H(p,q) − H(p),纯浪费掉的那部分)

那个差额 0.2573 就是 KL 散度(KL divergence,一句话:用错分布编码多花的 bit)。这给了"为什么训练目标是交叉熵"一个一句话的答案:`交叉熵 = 真分布的熵 + KL散度`。前一项 H(p) 是常数(数据固定),你优化交叉熵其实就是在优化 KL,即在把模型预测往真分布上推。当模型完美,q=p,KL=0,交叉熵降到熵 H(p) 这个地板就不动了。**你训练 loss 的下限不是 0,是数据本身的熵**——这解释了为什么有内在噪声的任务 loss 永远降不到 0,不是模型不够大,是地板在那。

`q[i] <= 0` 那行 `return Infinity` 也是个真实失败模式:模型给某个真实出现过的类别预测了概率 0,交叉熵就是正无穷。这就是为什么所有实现都要 label smoothing 或 log 加 epsilon——你不能对发生过的事赋零概率,否则"描述它"需要无限长的码。代码诚实地返回 Infinity 而不是偷偷护住,把责任推回给调用方:你得做平滑。

### KL 散度不是距离

工程师常把 KL 当"两个分布的距离"用,这会出微妙的 bug,因为**KL 不对称**。stage 实测:

- D(p‖q) = **0.2573 bits**
- D(q‖p) = **0.2647 bits**
- D(p‖p) = **0.000000 bits**

`D(p‖q) ≠ D(q‖p)`,所以它不是距离(距离必须对称)。这有实际后果:在变分推断、知识蒸馏里,你优化 `KL(学生‖老师)` 和 `KL(老师‖学生)` 得到的解不一样——前者倾向于"模式寻求"(钻进真分布的一个峰),后者倾向于"模式覆盖"(摊平去盖住所有峰)。选错方向,你的生成模型要么输出单调要么糊成一团。另外两个性质代码也验证了:KL 永远 ≥ 0(Gibbs 不等式),且当且仅当两分布完全相同时 = 0。

stage 还跑了个 N=20000 的种子采样,从 p 抽样后用 q 打分,经验交叉熵 **2.0000 bits ≈ 理论 2.0000**。但代码自己标注了一句诚实话:"合成绝对值偏乐观;可迁移的是 p̂→p 的收敛与 KL 的符号性质。" 意思是这个玩具数字别太当真——它的价值在于让你眼见经验分布随样本量收敛到真分布、KL 的符号性质成立,而不是这个 2.0000 本身有什么普适意义。

## ⚡ 没有压缩器能压缩所有输入:鸽巢原理

每隔几年就有人声称做出了"能压缩任意数据"的算法,甚至能"反复压缩越压越小"。这类东西**一定是骗局或算错了**,而且这不是"目前没人做到",是数学上**永远不可能**——这是少数能用一行小学算术证死的工程铁律。

证明是鸽巢原理(pigeonhole principle:n+1 只鸽子塞进 n 个笼子,必有一笼挤俩)。stage 直接枚举:

```typescript
function pigeonholeCompression(n: number): { inputs: number; shorterSlots: number } {
  const inputs = 2 ** n; // 所有长度 n 的串
  let shorterSlots = 0;
  for (let len = 0; len < n; len++) shorterSlots += 2 ** len; // 所有更短的串
  return { inputs, shorterSlots };
}
```

n=8 时:

- 长度 8 的输入共 = **256** 个(2^8)
- 更短的串总共只有 = **255** 个(长度 0..7 之和 = 2^8 − 1)

256 个不同输入,要塞进 255 个"更短"的格子。无损压缩必须是**单射**(不同输入压成不同输出,否则解压时分不清),256 个东西一对一塞进 255 个坑,鸽巢原理说至少有一个塞不进——也就是至少一个长度-8 的输入压完不会变短(甚至变长)。

推论很硬:**任何对某些输入变短的无损压缩,必然对另一些输入变长**。免费午餐不存在。gzip 压你的文本压得好,是因为它赌你的输入是"低熵的、有结构的"那一类,代价是它压随机数据会变大(加了头部反而更长)。所以这跟熵下界是同一枚硬币的两面:压缩器只能把概率质量从"常见输入"挪到"罕见输入",总的 bit 预算守恒。

那为什么这条标⚡?证明本身一百年前就盖棺了,**开放的是另一头**:给定一份具体数据,它的真实熵(也就是最优压缩下界)到底是多少?这等价于柯尔莫哥洛夫复杂度(Kolmogorov complexity,一句话:能生成这串数据的最短程序长度),而它被证明是**不可计算的**——不存在通用算法能算出任意数据的最短描述。所以实践里我们永远只有上界(某个压缩器实际压到的大小),拿不到那个真正的下界。"这份数据还能不能再压"在一般情况下没有通用解,正在被逼近(各种压缩基准、神经压缩),但理论的天花板就钉在不可计算性上。下一章讲密码学时会再碰到这个熵——一个密钥"安不安全"问的就是它对攻击者有多少 bit 的不可预测性,跟这里是同一把尺子。

## 一句话收尾

熵是 bit 的下界,霍夫曼是逼近它的算法,交叉熵是你训练时为偏离真分布付的 bit 账,鸽巢原理是这一切守恒的根。下次写 `cross_entropy(...)` 那一行,或者纠结日志为什么压不动,你知道屏幕背后是同一个量在说话。
