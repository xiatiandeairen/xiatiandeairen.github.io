你训练一个推荐模型,embedding(把东西编码成一串数字的向量)是 768 维。你想画出来看看分布——但屏幕只有 2 维。你想存 10 万张缩略图——但磁盘只有那么大。你想从一百万篇文档里找出和查询最像的 5 篇——但逐对比较算不动。这三件事看起来八竿子打不着,在线性代数里是同一个操作的三个变体:**把高维数据投影到一个「最值钱」的低维子空间**。

本章不教你解方程组,也不证 Cayley-Hamilton 定理。资深工程师需要的是另一套东西:看到一个矩阵,知道它的秩告诉你信息有几维;看到一组向量,知道余弦比欧氏距离更适合做语义检索;看到 PCA 跑出来的主成分指向数据中心而不是数据形状,立刻知道是忘了中心化。几何直觉优先,代数推导能省则省。

配套代码在 `examples/math-for-builders-from-scratch/src/stage03-linalg.ts`,整章不调任何 BLAS/LAPACK(线性代数底层库),特征值靠幂迭代手搓——这不是炫技,是为了让一个真实的失败模式浮出水面,后面会讲。本章数字全部来自这份代码的真实运行输出,凡是 toy 数据的乐观数字我都会标出来。

## 向量、矩阵、秩:三个你必须有手感的概念

向量就是一串数字,既是一个点也是一个方向。矩阵是一沓向量摞在一起,既是一张数据表(每行一个样本),也是一个「把向量搬到别处」的变换。这两种身份要随时切换:做 PCA 时矩阵是数据表,做图像压缩时同一个矩阵是一张图、是一个变换。

真正需要建立手感的是**秩**(rank,矩阵里真正独立的信息有几维)。一张 32×32 的图有 1024 个格子,但如果它只是一个从左到右的渐变,那它本质上是「一行模式」乘「一列权重」——秩为 1,1024 个数里只有约 64 个是独立的,剩下全是冗余。**秩是数据的内在维度,矩阵的尺寸只是它的外包装。** 压缩、降维、去噪,全是在赌「真实秩 ≪ 表面尺寸」这件事——而它对真实数据几乎总是成立,因为现实数据有结构(像素相邻相关、用户口味聚类、词义共现)。

`stage03` 的 SVD 演示里那张合成图就是故意造成近似低秩的:两个正弦渐变加一条斜坡再加一点噪声(代码 `demoSvd` 里的 `g1/g2/g3`)。所以 k=3 就几乎重构原图——下面会看到数字。**这是 toy 设定**,真实照片秩高得多。但「前几个奇异值占大头、误差随秩单调降」这个趋势是真的,这才是要带走的东西。

## PCA:沿数据最胖的方向看过去

PCA(主成分分析,找数据变化最大的方向)的几何故事一句话讲清:数据是空间里一团椭球形的点云,PCA 找的就是椭球最长的那根轴。沿着这根轴看,数据摊得最开、信息最多;垂直方向数据挤成一条,丢了也不心疼。把每个点投影到最长轴上,你就用 1 个数代替了 2 个数(或用 k 个数代替 768 个),这就是降维。

代数上,「最长轴」就是**协方差矩阵的最大特征向量**。协方差矩阵描述数据在各方向上怎么散开、哪些维度一起动;它的特征向量是那些「变换只拉伸不旋转」的特殊方向,对应特征值就是该方向上的方差(摊开程度)。`stage03` 的 `covariance` 把中心化后的数据算成 `Xᶜᵀ Xᶜ / (n-1)`:

```ts
function covariance(centered: readonly number[][]): number[][] {
  const n = centered.length;
  const cov = matmul(transpose(centered), centered);
  return cov.map((row) => scale(row, 1 / (n - 1)));
}
```

跑一个 400 点、真实主轴沿 `[0.894, 0.447]` 的点云,输出是:

```
样本数 n=400，真实主轴方向 ≈ [0.894, 0.447]
PC1: 方向 [0.8981, 0.4397]  特征值 15.6176  解释方差 97.99%
PC2: 方向 [0.4397, -0.8981]  特征值 0.3199  解释方差 2.01%
降到 1 维（投影到 PC1）后保留方差 = 97.99% → 用 1 个数代替 2 个数，几乎不丢信息
PC1 与真实主轴夹角 = 0.48°（≈0 说明 PCA 找对了）
```

两件事值得停下看。第一,PC1 找到的方向 `[0.8981, 0.4397]` 和真实主轴只差 **0.48°**,几乎完全对上——协方差的特征向量确实抓到了数据的形状。第二,PC1 一个方向就解释了 **97.99%** 的方差,意味着把 2 维数据压成 1 维,只丢 2.01% 的信息。**这就是降维的全部道理:不是均匀砍维度,是按方差排序、留头部、扔尾巴。** 真实数据里这个比例没这么极端(toy 数据是我故意造的近似 1 维),但「方差集中在前几个主成分」这个分布形状是真实数据的普遍规律,也是 PCA 能压缩的前提。

### ⚡ 失败模式:不中心化的 PCA 追的是位置不是形状

这是 PCA 最常见、最隐蔽的坑,因为它不报错、还给你一个看起来很合理的数字。中心化(centering,把每列减去均值,让数据团挪到原点)如果忘了,协方差公式 `XᵀX` 算出来的不是「数据怎么散开」,而是「数据离原点多远」——它的最大特征向量会被拽向**数据中心的方向**,而不是数据的延展方向。

`stage03` 故意把点云中心放在远离原点的 `[10, 6]`,然后跳过中心化直接做 PCA:

```
失败模式：忘了中心化（直接对原始数据做 PCA）
未中心化 PC1 方向 [0.8616, 0.5076]
它与真实主轴差 3.94°，却与「指向均值的方向」只差 0.49° → 没中心化时主成分追的是位置不是形状
```

读这三个数字:未中心化的「主成分」和真实形状轴差了 **3.94°**(偏了),却和「原点指向数据均值的方向」只差 **0.49°**(几乎重合)。换句话说,**它把「平均值在哪」当成了「主成分」**。3.94° 在 toy 数据上看着不大,但中心离原点越远、它就偏得越狠——你在真实 768 维 embedding 上忘了中心化,主成分会彻底变成「平均向量」,降维结果全错,而你拿到的还是个不报错的漂亮矩阵。

为什么代码要把这个设计成显式对比而不是直接帮你中心化?因为 `covariance` 函数在文档里明确写了不变量:**调用方必须传中心化数据**。这是「functional core」的取舍——函数不偷偷帮你做正确的事,而是把前置条件钉死、把违反后果摆出来。库帮你兜底一时爽,但你永远学不会为什么。生产里 scikit-learn 的 `PCA` 默认帮你中心化,正因为这个坑太常踩;但你得知道它在背后做了这一步,否则换个不自动中心化的实现(比如自己手搓、或某些 GPU kernel)就会栽。

参照真实系统:`numpy.cov` 默认假设你传的是观测数据并自己减均值,但 `numpy.linalg.eig` 作用在你给的任意矩阵上——这两个 API 的边界差异,正是这个坑的源头。

## ✦ SVD:线代的瑞士军刀

如果只能记住本章一个工具,记 SVD(奇异值分解)。它把任意矩阵 A 拆成三块:`A = U Σ Vᵀ`。Σ 是一个对角矩阵,对角线上是从大到小排的**奇异值**(singular value,衡量每个方向上「能量」多大);U 和 V 是两组正交方向。直觉上,SVD 说的是:**任何线性变换,都可以分解成「转一下、按各方向不同比例拉伸、再转一下」**。奇异值就是那些拉伸比例,大的重要、小的可丢。

为什么它是瑞士军刀?因为下面这些看起来不相干的任务,底层都是同一个 SVD:

- **降维**:PCA 本质就是对中心化数据做 SVD(协方差的特征分解 = 数据矩阵的 SVD)。
- **压缩**:只留前 k 个奇异值,得到最优的秩-k 近似(下面演示)。
- **推荐**:用户-物品评分矩阵做 SVD,U 和 V 就是用户隐向量和物品隐向量,Netflix Prize 时代的主力。
- **伪逆**:矩阵不可逆(超定/欠定方程)时,用 SVD 求 Moore-Penrose 伪逆,这是最小二乘的数值稳定解法。
- **去噪**:小奇异值往往对应噪声,砍掉它们等于去噪。

一个分解打通这么多场景,这就是「值得深学」的标志。

`stage03` 没有 LAPACK,所以 SVD 是借**对称矩阵的特征分解**绕出来的:`AᵀA = V Σ² Vᵀ`,对 `AᵀA` 求特征对就拿到 V 和奇异值(开方),再 `uᵢ = A vᵢ / σᵢ` 拿 U:

```ts
function svd(A: readonly number[][], rank: number, rng: Rng): Svd {
  const At = transpose(A);
  const gram = matmul(At, A); // n × n, symmetric PSD
  const eig = topEigenpairs(gram, rank, rng);
  // ... σ = sqrt(eigenvalue), uᵢ = A vᵢ / σᵢ
}
```

### 低秩近似:用秩换存储

秩-k 近似就是只留前 k 个奇异值重构 `Aₖ = Σ σᵢ uᵢ vᵢᵀ`(前 k 个外积之和)。一张 32×32 = 1024 个数的图,存秩-k 因子只需 `k·(m+n+1)` 个数。跑出来:

```
原始矩阵 32×32 = 1024 个数，‖A‖_F = 23.1188
秩 k | 存储数 | 压缩比 | 重构误差(Frobenius) | 相对误差
   1 |     65 | 15.75× |            2.05269 | 8.879%
   2 |    130 | 7.88× |            0.99993 | 4.325%
   3 |    195 | 5.25× |            0.58841 | 2.545%
   4 |    260 | 3.94× |            0.54837 | 2.372%
   8 |    520 | 1.97× |            0.41686 | 1.803%
```

(Frobenius 范数 = 把矩阵所有元素平方求和再开方,衡量整体「大小」/重构差距。)

读这张表的方式:**第一个奇异值就扛下大部分**。k=1 时 15.75× 压缩,相对误差已经只有 8.879%;k=3 时误差掉到 2.545%,几乎看不出差别。误差随 k 单调下降——这不是巧合,是 SVD 的数学保证(Eckart-Young 定理:秩-k 截断 SVD 是所有秩-k 矩阵里误差最小的那个,没有别的秩-k 矩阵能比它更接近原图)。

但要诚实:**这张 toy 图本身就近似秩-3**(我用两个正弦加一条斜坡造的),所以 k=3 误差就极小。真实照片信息维度高得多,要 k 几十上百才看得过去。代码注释里明说了这点:

```
注：toy 图本身近似低秩，所以 k=3 误差已极小；真实照片需要大得多的 k，
但「误差随 k 单调降、前几个奇异值占大头」这个趋势是真的。
```

会被带走的不是「k=3 就够」这个数字,是**误差-秩曲线的形状**:陡降然后拉平。那个「拐点」(elbow)就是你该截断的地方——保留多少 k,是在存储和保真之间的取舍,曲线告诉你性价比临界点在哪。

### 失败模式:幂迭代需要谱间隙

为什么 `stage03` 的 SVD 注释里反复强调「只适用于秩 ≤ 8」?因为它的特征分解是**幂迭代 + 收缩**(power iteration + deflation):反复用矩阵乘一个随机向量,它会收敛到最大特征向量;减掉 `λvvᵀ` 暴露次大的,如此往复。

```ts
function powerIteration(S: readonly number[][], rng: Rng, iters = 200): EigenPair {
  let v = normalize(Array.from({ length: n }, () => sampleNormal(rng)));
  for (let i = 0; i < iters; i++) {
    const Sv = matVec(S, v);
    const next = normalize(Sv);
    if (norm(next) === 0) break;
    v = next;
  }
  const value = dot(v, matVec(S, v)); // Rayleigh quotient
  return { value, vector: v };
}
```

它怎么坏:幂迭代的收敛速度取决于**谱间隙**(spectral gap,最大与次大特征值的差距)。当前两个特征值接近相等时,这两个方向「一样重要」,主方向在数学上就是病态的(ill-defined),迭代会在这个退化子空间里漂移,**主轴每次运行可能翻向不同方向**。代码注释 19-21 行专门点了这件事:加迭代次数没用,你只是在退化空间里更精确地漂移。其次,收缩会累积误差,每减一次 λvvᵀ 都把数值噪声叠进残差,所以 k 大了不可信——这就是「教学 SVD,k ≤ 8」的真实原因,不是偷懒。

生产里没人用幂迭代做完整 SVD,LAPACK 用的是 Golub-Kahan 双对角化加隐式 QR;要前 k 个大奇异值才用迭代法(Lanczos / 随机化 SVD)。但幂迭代是理解「为什么特征分解可能不稳定」最透明的窗口——你亲手看到谱间隙小时它漂移,以后看到 SVD 结果 run-to-run 不一致,第一反应就是查谱间隙,而不是怀疑自己代码写错。

## 余弦相似度:检索为什么不用点积

最后一块:在向量空间里「找最像的」。直觉答案是点积或欧氏距离,但生产检索默认用**余弦相似度**(cosine similarity,两向量夹角的余弦),原因值得讲透。

语义检索关心的是**方向**(主题),不是**模长**(文档长度/词频量级)。一篇讲同一主题的文章,不该因为它写得长、词频高就排得更前。余弦 = `dot(a,b) / (‖a‖·‖b‖)`,把模长除掉了,只比方向:

```ts
function cosine(a: readonly number[], b: readonly number[]): number {
  const na = norm(a);
  const nb = norm(b);
  if (na === 0 || nb === 0) return 0; // 零向量无方向 → 当作零相似度，不是 NaN
  return dot(a, b) / (na * nb);
}
```

`stage03` 造了一个故意暴露分歧的语料:一篇 B 主题文档短而精准(模长 2.50),一篇 A 主题文档跑题但超长(模长 9.07),查询是 B 主题。两种排序对比:

```
查询属于主题 B。余弦 top-3（应全是 B，无视模长）：
  doc#2  主题=B  ‖doc‖=2.50  余弦=0.9998
  doc#0  主题=B  ‖doc‖=0.96  余弦=0.9988
  doc#4  主题=B  ‖doc‖=0.81  余弦=0.9970
  余弦第一名是 B 类（模长仅 2.50）—— 正确，方向对就排前

对照：改用原始点积排序（对模长敏感）：
  doc#1  主题=A  ‖doc‖=9.07  点积=7.4091
  doc#2  主题=B  ‖doc‖=2.50  点积=2.5037
  doc#5  主题=A  ‖doc‖=1.21  点积=0.9688
  点积第一名是 A 类（模长 9.07） —— 被超长的跑题文档劫持了！
```

这就是核心教训:**点积让那篇 9 倍长的跑题 A 文档冒到第一名**,纯靠模长大;余弦把短而精准的 B 文档排第一。这里的关键是 A 和 B 主题**不正交**——它们共享一个维度(代码里 `topicA = [1,1,1,0,0]`、`topicB = [1,1,0,0,0]` 共享前两维),所以超长的 A 向量能往 B 查询上漏一大坨点积分数。如果主题完全正交,点积也不会出这种事;但真实 embedding 几乎从不正交(词义本来就交叠),所以这个劫持是常态不是特例。

注意 `cosine` 对零向量返回 0 而不是 NaN——这是个刻意的边界设计:零向量没有方向,「无信息」的诚实答案是 0,不是让 NaN 污染整个排序后崩溃。检索系统里一个空 embedding 不该让整次查询挂掉,这种边界处理在生产里比算法本身更常出 bug。

一个实务提示:很多向量库(FAISS 等)内部其实只算点积,做法是**先把所有向量归一化**(normalize 成单位长度),归一化后的点积就等于余弦。所以你看到「用内积索引」时别以为它在用裸点积——大概率前面有一步归一化,效果就是余弦。

## ⚡ 一个仍然开放的问题:高维空间里「距离」本身在退化

本章所有方法——PCA 找方向、SVD 压秩、余弦比相似——都默认「距离/相似度是有意义的」。但在很高维度下,这个前提开始动摇,这是**维度灾难**(curse of dimensionality)里至今没有通用解的一面。

具体现象:当维度涨到几百上千,一组随机点之间「最近距离」和「最远距离」会趋于相等——也就是说**所有点都差不多远,「最近邻」这个概念在统计上变得没区分度**(这是 Beyer 等人 1999 年「When Is Nearest Neighbor Meaningful?」的结论)。embedding 通常 768 到几千维,正落在这个尴尬区。为什么实践里检索还能用?因为真实数据不是均匀随机填满高维空间的,它躺在一个低得多的「流形」(manifold,嵌在高维里的低维曲面)上,真实内在维度远低于名义维度——这正是 PCA/SVD 能压缩的同一个原因。

但「真实内在维度到底是多少、怎么可靠估计、在什么条件下高维距离仍可信」,目前**没有通用解**,是表示学习和度量学习的活跃研究方向。实务上你能做的是经验性的:对 embedding 先 PCA/SVD 降到信息集中的几十到一两百维再检索,常常**又快又准**(去掉的尾部维度多是噪声);用近似最近邻(ANN)算法回避精确距离的脆弱性。但这些都是绕行,不是把问题解决了——当有人跟你说「直接拿 4096 维原始 embedding 算余弦就行」,你应该知道这背后压着一个没关掉的理论问题。
