线上跑了个 A/B 测试,新文案版本转化率高了一截,p 值 0.0007,你截图发群里"统计显著,上线"。三周后全量上线,转化率没动。问题不在你的实验跑错了,在于你把 p=0.0007 读成了"新版本更好的概率是 99.93%"——它根本不是这个意思。这一章不教你背 t 检验公式(配套代码里那套连续分数逼近 t 分布 CDF 的活儿,真实统计库都替你干了),教的是:当一个数字摆在面前,它**到底在说什么、什么时候在骗你、为什么会骗**。

配套代码全在 `examples/math-for-builders-from-scratch/src/stage02-stats.ts`,一个固定随机种子(`mulberry32(20260622)`)驱动所有随机实验,你本地重跑会得到逐位相同的输出。下面引用的所有数字都来自这次真实运行,不是我编的;凡是估算或玩具数据我会标出来。

## 一、A/B 测试:t 检验在算什么

先把场景钉死。你有对照组(老文案)和实验组(新文案),各 200 个用户,每人产生一个数值指标(比如下单金额)。对照组真实均值 10.0,实验组真实均值 10.5——也就是真有 +0.5 的效应,但你**不知道**,你只有两组抽样数据。问题:从这两组样本,能不能判断"实验组确实更高",还是说看到的差异只是抽样噪声?

这就是双样本 t 检验(two-sample t-test,从两组样本判断均值是否真有差异)干的事。代码里用的是 Welch 版本(不假设两组方差相等的稳健默认版),核心在 `welchTTest`:

```typescript
function welchTTest(a: readonly number[], b: readonly number[]): TTestResult {
  const ma = mean(a);
  const mb = mean(b);
  const va = variance(a); // sample variance (n-1)
  const vb = variance(b);
  const na = a.length;
  const nb = b.length;
  const se = Math.sqrt(va / na + vb / nb); // standard error of the difference
  const meanDiff = ma - mb;
  const t = meanDiff / se;
  // Welch–Satterthwaite degrees of freedom.
  const df =
    (va / na + vb / nb) ** 2 /
    ((va / na) ** 2 / (na - 1) + (vb / nb) ** 2 / (nb - 1));
  const pValue = tTestPValueTwoSided(t, df);
  const half = 1.96 * se; // ~95% normal critical value
  return { meanDiff, t, df, pValue, ci95: [meanDiff - half, meanDiff + half] };
}
```

读懂三个量就够了:

- **`se`(标准误,sample 均值差的波动幅度)**:两组方差除以各自样本量再开方。样本越多,`se` 越小,均值差越稳。
- **`t`(t 统计量)**:观测到的均值差,以 `se` 为单位。`t=3` 意思是"我看到的差异是噪声尺度的 3 倍"。
- **`df`(自由度)**:Welch 版本算出来是非整数,这是正常的,不是 bug。

跑出来:

```
实验组均值 - 对照组均值 = 0.6158 (真实效应设为 +0.5)
t 统计量               = 3.432, df ≈ 397.0
p 值                   = 6.624e-4
95% 置信区间 (差值)    = [0.2641, 0.9674]
判定 (α=0.05)          = 拒绝原假设：检测到效应
```

注意第一行:真实效应是 +0.5,但抽样出来观测到的是 0.6158。**抽样本身就有误差**,这正是为什么需要统计推断,而不是直接拿差值拍板。p 值 6.6e-4 远小于阈值 0.05(这个 0.05 就是 α,你预先接受的假阳性率),所以判定"拒绝原假设"——原假设(null hypothesis)是"两组没差异"。

### 一个工程细节:为什么 t 分布的 CDF 要手写

把 t 统计量变成 p 值,需要算 Student-t 分布的尾部概率 `P(|T| > |t|)`,而这玩意没有闭式解,本质是个不完全 beta 积分。代码用了 Numerical Recipes 那套 Lentz 连分数(`betacf`),和真实统计库调的是同一个算法。它**不是糊弄人的近似**,对我们用的自由度能收敛到机器精度。但它有失败模式:

```typescript
// FAILURE MODE: the continued fraction loses accuracy in the far tail (x near 0 or 1) and for
//   tiny df; we mitigate with the symmetry betacf(x) = 1 - betacf(1-x) and cap iterations.
```

极远尾部(x 接近 0 或 1)和极小自由度时连分数会掉精度,代码用对称性把 x 路由到收敛快的那一侧来缓解。这里的设计取舍值得记住:**核心库 `core/stats.ts` 只放描述性统计(均值/方差/分位数),推断性的机器(t 分布 CDF)放在用它的这一章旁边**。原因是不完全 beta 是这一章特有的需求,不该污染通用核。而且代码没有"断言"这个 CDF 准——它用下面第二节那张均匀直方图**实证**它准。这是个好习惯:能跑出来证明的,别用注释吹。

## 二、p 值到底是什么:把误解跑成直方图 ✦

现在到这一章最关键的部分。**p 值不是"结论为真的概率",置信区间也不是"真实参数落在这个区间里的概率"。** 这两个是统计里最常见的误读,而且光靠文字解释很难让人信服。代码的办法是:把它们的**真实定义**直接跑出来给你看。

做法很简单:让原假设**真的为真**——两组都从同一个分布(均值 10、标准差 2)抽样,**真实差异就是 0**——然后把同一个 t 检验跑 10000 次,每次记下 p 值和置信区间:

```typescript
function demoNullDistribution(rng: Rng): void {
  const nReps = 10000;
  const pValues: number[] = [];
  let ciCovers = 0; // how often the 95% CI contains the true diff (= 0)
  for (let i = 0; i < nReps; i++) {
    // Same mean (10), same std (2): there is NO real difference. n=40 each.
    const g1 = drawNormal(rng, 40, 10, 2);
    const g2 = drawNormal(rng, 40, 10, 2);
    const r = welchTTest(g1, g2);
    pValues.push(r.pValue);
    if (r.ci95[0] <= 0 && 0 <= r.ci95[1]) ciCovers++;
  }
  // ...
}
```

十个桶基本齐平,每桶约 1000 条。**当原假设为真时,p 值在 [0,1] 上服从均匀分布。** 这条性质你得盯着看一会儿,因为它直接拆穿了那个误读:

如果 p 值是"原假设为真的概率",那原假设真为真的时候,p 值应该总是接近 1 才对。但它不是——它均匀分布,意味着**就算两组毫无差异,你也有整整 10% 的概率拿到 p < 0.1,有 5% 的概率拿到 p < 0.05**。p 值的真实定义是:**假设原假设为真,看到当前数据或更极端数据的概率**。它衡量的是"数据和'没差异'这个假设有多不相容",而**不是**"假设本身有多可能为真"。

再看置信区间和假阳性率:

```
95% CI 覆盖真值(0) 的比例 = 94.94%  (理论 ≈95%)
p<0.05 的比例 (假阳性率)  = 4.68%  (理论 ≈5% —— 这就是 α)
```

95% 置信区间(confidence interval,从样本算出的一个区间)的真实含义就藏在这 94.94% 里:**它不是"真值落在这个区间的概率是 95%"**。真值(这里是 0)是个固定的数,它要么在某个具体区间里、要么不在,没有"概率"可言。会变的是**区间本身**——你每跑一次实验,数据不同,算出的区间就不同。"95% 置信"说的是:**如果你把这个实验重复无数次,每次算出的区间里,约 95% 会盖住真值。** 代码跑了 10000 次,94.94% 的区间盖住了 0,正好兑现了这个定义。

最后那行 4.68% 就是 α 的本质:原假设为真时,你仍会"误报发现"的比例,理论值 5%。你设 α=0.05,就是主动接受"长期看 20 次里有 1 次假阳性"。这不是 bug,是你和不确定性签的合同条款。

**这就是为什么代码要同时返回 p 值和置信区间。** 注释写得很直白:

```typescript
// WHY a CI alongside the p-value: the p-value answers "could the null produce this?" while
//   the CI answers "how big is the effect, with what uncertainty?" — the second is what a
//   builder actually needs to decide whether an effect is worth shipping.
```

p 值回答"原假设能不能产生这个数据",置信区间回答"效应有多大、不确定性多少"。**后者才是你拍板上不上线真正需要的东西**——这一点第五节会狠狠打脸只看 p 值的人。

## 三、多重比较:数得够多,就能数出假发现 ⚡

上一节说了,原假设为真时,单次检验也有 5% 概率假阳性。那如果你一口气跑很多次呢?

工程现场天天发生:监控面板上挂着 100 个指标,你发版后扫一遍,"找那个动了的"。或者你跑了一个实验,主指标没动,于是切了 20 个细分人群("是不是对新用户有效?对 iOS 有效?对高客单价有效?")挨个看。这就是多重比较(multiple comparisons,一次跑很多个检验)。

代码造了个极端干净的演示:100 个检验,**每个都是同分布对同分布,真实效应全是 0**,然后数有几个"显著":

```typescript
function demoMultipleComparisons(rng: Rng): void {
  const nTests = 100;
  let falsePositives = 0;
  let smallestP = 1;
  for (let i = 0; i < nTests; i++) {
    const g1 = drawNormal(rng, 50, 0, 1);
    const g2 = drawNormal(rng, 50, 0, 1); // identical distribution: every effect is fake
    const p = welchTTest(g1, g2).pValue;
    if (p < 0.05) falsePositives++;
    if (p < smallestP) smallestP = p;
  }
  // ...
}
```

结果:

```
'显著' (p<0.05) 的检验数 = 7 / 100  (期望 ≈5)
最小 p 值               = 8.852e-3  (挑这个汇报 = p-hacking)
```

100 个根本没有任何效应的检验,凭空数出来 7 个"显著"(期望约 5,这次抽到 7)。而最小的那个 p 值是 0.0089——**如果你只把这一个截图发出去,谁看了都会觉得这是个扎实的发现**。这就是 p-hacking(p 值操纵):跑很多个,只报最好看的那个。它通常不是恶意造假,而是"我多看了几个角度,挑了最显著的讲"这种看起来无害的行为。

数学很无情:单个检验假阳性率 5%,跑 100 个独立检验,**至少出一个假阳性**的概率是 `1 - 0.95^100 ≈ 99.4%`(我用 α 和检验数现算的,非代码输出)。也就是说,你扫 100 个指标,几乎**注定**会看到"显著"的东西,哪怕什么都没发生。

修法是校正阈值。代码注释点了名:

```typescript
console.log("没有任何真实效应，却凭'数得够多'制造出发现 —— 必须用 Bonferroni/FDR 校正。");
```

- **Bonferroni 校正**:把阈值除以检验数(100 个检验就用 0.05/100 = 0.0005)。简单粗暴,但当检验很多时极其保守,容易把真效应也一起毙掉。
- **FDR(False Discovery Rate,假发现率)/ Benjamini-Hochberg**:控制的是"被你判为显著的那批里,假阳性占的比例",比 Bonferroni 宽松,是现代多重比较的主流默认。

> ⚡ **开放问题**:多重比较校正在"检验数固定且事先知道"时是干净的,但真实数据分析是**自适应的**——你看了第一个结果再决定切哪个人群、加哪个协变量,检验数根本数不清,甚至连"我做了多少次隐性比较"自己都不知道(这就是所谓的 garden of forking paths,分叉路径花园)。如何对**整个自适应分析流程**做有效的多重性校正,目前**没有通用解**。预注册(preregistration,实验前公开声明要测什么)是社会层面的缓解手段而非数学解;differential privacy 派生的 adaptive data analysis 理论给了部分场景的形式化保证,但远没到能丢进日常 BI 面板的程度。这是统计学和机器学习交叉处仍在活跃研究的方向。

## 四、辛普森悖论:每一步算术都对,结论却反了 ⚡

前面三节的"骗"都跟随机性有关。这一节的更阴险:**没有任何随机性,每一步算术都对,合计的结论却和每个分组的结论完全相反。**

代码用的是真实数据——Charig 1986 年的肾结石治疗研究。两种疗法 A 和 B,按病情轻重分组:

```typescript
const data = {
  A: { mild: { ok: 81, n: 87 }, severe: { ok: 192, n: 263 } },
  B: { mild: { ok: 234, n: 270 }, severe: { ok: 55, n: 80 } },
};
```

跑出来:

```
分组成功率 (每个子组里 A 都更高):
  病情      A 成功率        B 成功率
  轻症    93.1% (81/87)      86.7% (234/270)
  重症    73.0% (192/263)      68.8% (55/80)
合计成功率 (趋势反转，B 反而更高):
  方案    合计成功率
  A     78.0% (273/350)
  B     82.6% (289/350)
```

盯着看:**轻症里 A 更好(93.1% > 86.7%),重症里 A 也更好(73.0% > 68.8%)**。A 在每一个分组里都赢。但**合计起来 B 反而更好(82.6% > 78.0%)**。

每个数字都没错,你可以拿计算器逐个验。陷阱在于**两组接收的病情构成完全不同**:

- A 主要被用在重症(263 个重症 vs 87 个轻症),而重症本身成功率就低。
- B 主要被用在轻症(270 个轻症 vs 80 个重症),而轻症本身成功率就高。

合计成功率被"病情构成"这个**潜伏变量(confounder,混淆因素)** 主导了,而不是被疗效主导。A 看起来差,是因为它被派去打硬仗。代码注释把机制说透了:

```typescript
// WHY it happens: the groups receive very different MIXES of an easy vs hard segment, and
//   the aggregate rate is dominated by whichever group sits mostly in the easy segment.
```

这就是辛普森悖论(Simpson's paradox,分组趋势在合计后反转)。它在工程里的杀伤力极大:你看合计转化率"B 版本更高"就上了 B,但实际上 A 在每个细分人群里都更好,只是 A 恰好被更多地展示给了难转化的人群(比如冷流量)。**这正是相关不等于因果(correlation ≠ causation)最具体的一个反面教材**:合计层面 B 和高转化"相关",但把 B 强加给所有人并不会**导致**更高转化——那个相关是病情/人群构成制造出来的假象。

> ⚡ **开放问题**:辛普森悖论暴露了一个更深的麻烦——**到底该不该分组(该不该控制某个变量),没有纯统计的答案。** 这个肾结石例子里,"病情严重程度"显然该控制,所以应该信分组结论(A 更好)。但换个场景,如果你控制的那个变量是处在因果链**中途**的(比如"是否点击了推荐"夹在"算法版本"和"最终下单"之间),控制它反而会引入偏差,得到错误结论。判断"该控制哪些变量"需要的是**因果图(causal graph)** 这种因果模型,而因果模型来自领域知识和假设,**无法仅从数据本身推出来**。Judea Pearl 的 do-calculus 给了形式化框架,但"哪个变量是 confounder、哪个是 mediator"这一步永远需要人带着领域假设进来。**纯靠数据无法区分二者**,这是观测数据做因果推断的根本限制,至今没有绕过去的通法。

## 五、失败模式:p 小不代表效应大

最后一个,也是生产里最常见的一个误读:**p < 0.05 就宣布发现,完全无视效应量。**

代码构造了一个最阴的情况:**效应是真的存在,但小到可以忽略**——真实差异只有 +0.02,但样本量拉到每组 50000:

```typescript
function demoFailureModePChasing(rng: Rng): void {
  // Real effect of +0.02 on a metric with std 1, but n=50000 per group makes even a
  // microscopic difference "significant".
  const a = drawNormal(rng, 50000, 0.02, 1);
  const b = drawNormal(rng, 50000, 0.0, 1);
  const r = welchTTest(a, b);
  // ...
}
```

跑出来:

```
样本量                 = 50000 / 组
观测均值差             = 0.0223 (真实效应仅 +0.02)
p 值                   = 4.357e-4  → 形式上"显著"
95% 置信区间 (差值)    = [0.0099, 0.0347]
```

p 值 0.0004,比第一节那个 +0.5 的真实大效应还"显著"。如果你只看 p 值,你会兴奋地上线。但看置信区间:差异在 [0.0099, 0.0347] 之间,**满打满算也就 0.02 上下一个单位**——业务上几乎为零。

这里的机制是:**p 值同时取决于效应大小和样本量**。`t = meanDiff / se`,而 `se` 随样本量增大而变小。样本量足够大时,**任何**非零的微小差异都能被推过显著线。所以在大数据场景里,"统计显著"几乎是免费的,它根本不告诉你这个效应**值不值得做**。

这就闭合了第二节埋下的伏笔:p 值回答"差异是不是真的存在",置信区间回答"差异有多大"。**上不上线该看后者。** 一个干净的判断流程是:

1. 先看置信区间能不能排除掉"业务上无意义"的小效应区间(预先定一个最小有意义效应,minimum detectable effect)。
2. 区间整体落在有意义范围之外,才谈得上"值得做"。
3. p 值只是个准入门槛,不是结论。

第一节那个例子,置信区间是 [0.2641, 0.9674],整段都是有意义的正效应,那才是真该上线的。这一节这个,区间贴着 0,p 再小也是个假发现。

## 给工程师的落点

这一章没有一个公式需要你背——t 分布 CDF、Welch 自由度,真实统计库(scipy.stats、R 的 `t.test`)都替你算好了。需要刻进直觉的是四条**读数纪律**:

1. **p 值不是"假设为真的概率"**,是"假设为真时看到这数据的概率"。原假设为真时它均匀分布,5% 概率自己跨过 0.05 线。
2. **置信区间不是"参数落在区间的概率"**,是"重复实验时区间盖住参数的长期比例"。会变的是区间不是参数。
3. **多看几个指标/人群 = 制造假发现**。扫 100 个无效应指标几乎注定数出 7 个"显著"。要校正(Bonferroni/FDR),而自适应分析的校正目前无通用解。
4. **p 小 ≠ 效应大**。大样本下任何微小差异都显著。拍板看置信区间和效应量,p 值只是门槛。

辛普森悖论是这四条之外的一记重锤:**每步算术都对的数字,照样能给出相反结论**,因为合计抹掉了潜伏变量。它把"相关不是因果"从口号变成了你能在终端里亲手跑出来的反例。

想自己玩,把 `examples/math-for-builders-from-scratch/src/stage02-stats.ts` 跑起来,改改 `demoFailureModePChasing` 里的样本量和真实效应,看 p 值和置信区间怎么各自动——你会很快建立起"什么时候该信这个数字"的肌肉记忆。下一章 [03 线性代数] 会把这一章的"一维均值"推广到高维:当你的指标不再是一个数、而是一个向量(用户的 embedding、特征矩阵),"差异""方差""相关"都会变成几何对象,而那套几何直觉是后面优化和机器学习章节的地基。
