> 这一章是全书唯一一章在教你**什么时候应该放弃**。前八章默认"只要够聪明就能算出来",这章告诉你两个工程上必须能一眼识别的边界:一类问题被**证明**永远没有通用算法(不可判定),另一类有算法但代价随规模指数膨胀(NP 难,目前无多项式解)。把这两条边界刻进直觉,你就不会再对着一个停机检测器、一个"自动验证任意配置文件不会死循环"的需求空耗,也不会在 n=20 能跑就拍胸脯说 n=200 也行。

配套代码:`examples/math-for-builders-from-scratch/src/stage09-computability.ts`,一个文件五段,本章所有数字都来自它在本机的真实运行,不是我narration出来的。它故意把"可计算性"和"复杂度"塞进同一个 stage,因为对工程师而言这是一个连续的故事:机器**能**判定什么(DFA / 图灵机)、**没有机器能**判定什么(停机)、**能判定但跑不动**什么(NP)。

## 一、有限内存能判定什么:DFA 与那堵"数不过来"的墙

先从最弱的机器开始,因为它的弱恰好暴露了"内存"是计算能力的硬通货。

DFA(deterministic finite automaton,确定性有限自动机,大白话:一个只有固定几个状态、读一个字符跳一次的小机器)是"不带回溯的正则表达式"的本体——一个**没有反向引用的正则,本质就是一个 DFA**。stage 里手搓了一个识别"二进制串里 1 的个数为偶数"的 DFA:

```typescript
const EVEN_ONES_DFA: Dfa = {
  states: ["even", "odd"],
  start: "even", // zero 1s is even
  accept: new Set(["even"]),
  delta: {
    even: { "0": "even", "1": "odd" }, // a 1 flips parity
    odd: { "0": "odd", "1": "even" },
  },
};
```

注意它只有**两个状态**。这不是省事,是定理:识别"奇偶"只需要记住"到目前为止 1 是奇是偶",一个 bit。运行输出:

```
"    " (0 个 1) → 接受  (0 次转移)
"11  " (2 个 1) → 接受  (2 次转移)
"101 " (2 个 1) → 接受  (3 次转移)
"1010" (2 个 1) → 接受  (4 次转移)
"111 " (3 个 1) → 拒绝  (3 次转移)
```

"转移次数 = 输入长度"——线性、单遍、不回头。这就是正则识别便宜的全部理由。

### 失败模式:缺转移不等于"拒绝"

`runDfa` 在遇到字母表外符号时**抛异常**,而不是默默当作拒绝:

```
失败模式:喂入字母表外符号 '2' → [dfa] no transition for state=odd symbol=2
```

为什么这么设计?因为 DFA 的转移函数 δ 在定义上**必须是全函数**(对字母表每个符号、每个状态都有定义)。工程上的对应坑:很多人写状态机时,对"没预料到的输入"既不报错也不显式拒绝,而是悄悄落到某个默认分支。这会把"我没建模这个输入"和"我建模了并拒绝它"混为一谈——前者是 bug,后者是行为。stage 选择 fail loud,正是 `arch-runtime.md` 里"silent catch 是坏味"的同一条原则在自动机上的体现。

### ✦ 为什么 DFA 数不过来 n 个 a 接 n 个 b

DFA 的能力天花板是写死的:它只有有限个状态,**看不了回头路**,所以记不住任意大的计数。语言 `aⁿbⁿ`(n 个 a 后面跟同样多的 b)就超出它的能力——你需要"记住到底来了几个 a",而 n 无上界,有限状态装不下。这是**抽水引理(pumping lemma)**保证的不可能,不是"还没想到聪明的正则"。

这条边界有直接现金价值:**不要用正则去解析嵌套括号 / HTML / JSON**。嵌套是无界计数,正则(=DFA)在原理上做不到。你看到的"能跑"的正则匹配嵌套,要么是限定了深度(其实是更大的 DFA),要么是用了非正则扩展(回溯引擎,已经不是 DFA 了,而那正是 ReDoS 灾难性回溯的来源)。识别"这是个无界计数问题"→ 立刻换下推自动机 / 真正的 parser,这是 DFA 这一节的工程产出。

## 二、加一条无界纸带:图灵机与"停机 ≠ 没预算"

DFA 缺的那样东西——可读写的无界存储——正是图灵机补上的,而这一补就补成了**图灵完备**(Turing-complete,大白话:能算的它都能算)。stage 里的图灵机跑一个最朴素的程序:一元加一,把 `111`(=3)变成 `1111`(=4)。

```typescript
const INCREMENT_TM: TuringMachine = {
  blank: "_",
  start: "scan",
  halt: "done",
  rules: {
    scan: {
      "1": { write: "1", move: "R", next: "scan" }, // keep walking over the number
      _: { write: "1", move: "N", next: "done" }, // hit the end -> append a 1, halt
    },
  },
};
```

运行:

```
输入 "111" (=3) → 输出 "1111" (=4)
停机=true,走了 4 步(扫到末尾的空格再写一个 1)
```

加法本身无聊,重点是它**用到了 DFA 没有的三件事**:读写纸带、移动读写头、靠进入终态来停机。`runTm` 的纸带用 `Map<number, string>` 实现"按需生长"——读到任何没写过的位置就物化出一个空格,这就是"无界"在代码里的样子。

### 失败模式:预算耗尽是资源问题,不是停机问题

`runTm` 接受一个 `maxSteps` 预算。stage 故意把预算卡到 3 步去跑一个本来要更多步的输入:

```
失败模式:预算只给 3 步 → 停机=false,走了 3 步、纸带 "11111111"
```

这里有一句注释是整个 stage 最值钱的设计决策(`stage09-computability.ts` 第 112-115 行):

> The budget is NOT a halting decider — it is a resource bound … "budget exhausted" must never be reported as "this machine does not halt".

为什么死磕这个区分?因为它直通下一节。`halted=false` 在这里的意思是"**3 步没跑完**",**不是**"这程序永远不停"。把"我等不及了"误报成"它不会停",是工程师面对超时任务最常犯的认知错误——你 kill 掉一个跑了 10 分钟的 job,你知道的只是"10 分钟没出结果",你**不**知道"再给它一年会不会出"。下一节会证明:那个"再给多久会不会出"的问题,**没有任何程序能替你回答**。

## 三、停机问题:有些答案不是难,是被证明不存在

这是本章的 ✦ 核心钩子,也是每个工程师都该能在白板上手画的一个矛盾。

需求长这样:写一个函数 `halts(program, input)`,对任意程序+输入,返回它会停机还是死循环。听起来像个高级 linter,很多人第一反应是"难写但应该能写"。**它被证明不可能写出来**——不是"现在没人会",是"逻辑上存在即矛盾"。

stage 把图灵 1936 年的**对角线论证(diagonalization)**直接做成了可执行的反例。我们写不出真的 `halts`(它不存在),所以 mock 一个"判定器",再造一个**故意唱反调**的程序 `spite` 去击穿它:

```typescript
function runSpiteAgainst(oracle: HaltingOracle, selfSrc: string): "halts" | "loops" {
  const predictionHalts = oracle(selfSrc, selfSrc); // oracle predicts spite on itself
  // Do the OPPOSITE of the prediction.
  return predictionHalts ? "loops" : "halts";
}
```

逻辑很短:`spite` 先问判定器"我跑我自己会怎样",然后**做相反的事**。判定器说停机,它就死循环;说死循环,它就停机。于是判定器对 `spite` 的预测**按构造必然错**。stage 喂了三种不同策略的判定器,无一幸免:

```
判定器=总是猜『停机』:预测 spite→停机,实际 spite→不停机  [判错 ✓矛盾]
判定器=总是猜『不停机』:预测 spite→不停机,实际 spite→停机  [判错 ✓矛盾]
判定器=按源码长度猜:预测 spite→不停机,实际 spite→停机  [判错 ✓矛盾]
→ 对任意判定器都能造出『故意唱反调』的程序使其判错 ⇒ 通用停机判定器不存在。
```

三个判定器都判错不是巧合凑出来的——这正是对角线论证的精髓:**对你能写出的任意判定器,我都能机械地造出一个让它出错的程序**(把它的预测取反即可)。所以"通用停机判定器"作为一个对象就不可能存在。第二节那个"预算 vs 停机"的区分到这里收口:你 kill 一个超时进程后那个悬而未决的"它到底会不会停",在一般情况下**没有算法能回答**。

工程上怎么用这条定理?**学会识别"这个需求其实是停机问题的伪装"**,然后停止空耗:

- "写个工具静态判定任意用户脚本不会死循环" → 停机问题,不可判定。退而求其次:加超时 / 资源配额 / 限制语言不带无界循环(很多配置 DSL 故意不图灵完备就是为了这个)。
- "自动判定两段代码是否等价" → 由莱斯定理(Rice's theorem,推广版:程序行为的任何非平凡性质都不可判定)同样不可判定。
- "证明这个正则一定能在多项式时间内匹配完" → 一般不可判定。

不可判定 ≠ "完全没办法"。它是说**没有对所有输入都正确的通用算法**;你永远可以做"对大部分实际输入有用、对刁钻输入认怂(超时/保守拒绝)"的近似器。区分这两者,是这一节给你的判断力。

## 四、归约:解一个 = 解一片(NP 完全的传染机制)

跨过"不可判定"这条线,剩下的是**可判定但可能跑不动**的问题。这里的主角是 NP,和把一堆问题绑成命运共同体的工具:**归约(reduction)**。

NP(大白话:答案猜出来后能快速验证的问题)里有一类"最难的",叫 NP 完全。它们最惊人的性质是:**你解开任何一个,就等于解开了全部**。串起它们的就是归约——把问题 A 的实例机械地翻译成问题 B 的实例,使得"A 是 yes ⟺ B 是 yes"。stage 实现了教科书级的经典归约:**SAT → 子集和**。

SAT(布尔可满足性,大白话:能不能给一堆变量赋真假让一个逻辑公式成立)是第一个被证明 NP 完全的问题。子集和(给一堆数,能不能挑出一个子集恰好加到目标值)看起来是个纯算术问题,八竿子打不着。归约的魔法在于用**进制位**把逻辑结构编码进数字:每个变量、每个子句占一个十进制列,"挑哪个数"就对应"变量取真还是假",数字相加到目标值当且仅当存在一个满足所有子句的赋值。核心编码在 `reduceSatToSubsetSum`:

```typescript
for (let v = 1; v <= numVars; v++) {
  for (const polarity of [true, false] as const) {
    let value = colValue(v - 1); // variable column: 强制每个变量恰好选一种极性
    cnf.forEach((clause, ci) => {
      const satisfies = clause.some((lit) =>
        polarity ? lit === v : lit === -v,
      );
      if (satisfies) value += colValue(numVars + ci); // 命中的子句列 +1
    });
    numbers.push(value);
  }
}
```

光写出归约不算数,stage 用**两边都暴力解一遍**来证明归约没说谎——对 40 个随机公式,分别用 `bruteForceSat` 和 `bruteForceSubsetSum` 求解,看答案是否永远一致:

```
随机跑 40 个公式:可满足 39 个 / 不可满足 1 个
归约答案一致 (SAT ⇔ 子集和) = 40/40  ✓ 归约保持答案
示例公式 (x1∨x2∨¬x3)∧(¬x1∨x2∨x3)∧(x1∨¬x2∨x3): SAT=true
  → 子集和实例:12 个数,目标 111333,存在子集=true
```

40/40 一致(而且故意包含了 1 个不可满足的公式,让归约在 yes 和 no 两个方向都被验证)就是"SAT ⇔ 子集和"在玩具规模上的实证。这正是钩子里说的"**解一个 = 解一片**":你要是真给子集和找到了多项式算法,这套归约会立刻把它变成 SAT 的多项式算法,进而(经由 SAT 是 NP 完全)变成**所有 NP 问题**的多项式算法。这种"牵一发动全身"就是 NP 完全这个概念的全部分量。

> 实现细节里藏着一个 honest-number 的诚实:`bruteForceSubsetSum` 在数字超过 24 个时直接抛错(`n > 24` → throw),因为 `1 << n` 在 31 位以上会让 32 位整数静默溢出、搜错空间、报出假的"无解"。stage 宁可 fail loud 也不验证一个坏掉的搜索。所以它把测试规模钉死在 3 变量——这不是偷懒,是为了让"验证器本身"可信。

### ⚡ 开放问题:P vs NP——至今无人能解,也无人能证伪

到这里必须把那块"目前无通用解"的牌摆上桌。

SAT、子集和、旅行商、图着色……这些 NP 完全问题至今**没有人找到多项式时间算法**,但也**没有人证明这种算法不存在**。这就是 **P vs NP**:P(能在多项式时间内解)是否等于 NP(能在多项式时间内验证)?这是 Clay 研究所七个千禧难题之一,悬赏一百万美元,自 1971 年 Cook 提出至今**仍然完全开放**——这是货真价实的"目前无通用解、正在研究"。

工程上为什么你该在意一个纯数学悬案?因为现实里压着一个**赌注**:整个现代密码学(下一组数学里的 RSA、椭圆曲线)建立在"某些 NP 类问题事实上算不动"的假设上。如果哪天 P=NP 被证明且给出了高效构造,这些加密一夜失效。绝大多数研究者**赌 P≠NP**(否则世界早该乱套了),但请记清楚:这是**信念,不是定理**。所以当你看到一个 NP 完全问题,正确的工程姿态不是"我再想想聪明算法",而是"通用快解大概率不存在,我该上近似算法 / 启发式 / SAT 求解器 / 限定特殊结构"。

## 五、指数墙:n=20 一眨眼,n=60 等到宇宙凉

理论说 NP 难"跑不动"是抽象的,stage 第五段把它**实测成秒数**让你疼一次。它跑最坏情况的暴力 SAT(用一个矛盾公式 `[[1],[-1]]` 逼暴力枚举遍历全部 2ⁿ 个赋值),测真实墙钟时间:

```
n=16: 暴力枚举 65,536 个赋值,实测 11.8 ms
n=20: 暴力枚举 1,048,576 个赋值,实测 167.5 ms
n=22: 暴力枚举 4,194,304 个赋值,实测 668.5 ms
```

到 n=22 还只是 0.67 秒,温水煮青蛙。然后按实测的 **159.4 ns/赋值** 外推(stage 明确标 `est.`,因为它**拒绝真去跑 2⁶⁰**):

```
按实测 159.4 ns/赋值 外推 (est.):
  n=40 → 2.0 天   n=50 → 5.69e+0 年   n=60 → 5.83e+3 年
```

n=40 两天,n=60 五千八百年。每 +10 个变量,代价 ×1024。stage 还画了 log10(秒数) 对 n 的 ASCII 图——之所以画对数而不是原始 2ⁿ(原始值会让最大那根柱子把其余全压成空白),是为了让那条**笔直上升的斜线**显形:

```
n=20 │················································│ 0.000
n=30 │██████████······································│ 2.233
n=40 │██████████████████████··························│ 5.244
n=50 │███████████████████████████████████·············│ 8.254
n=60 │████████████████████████████████████████████████│ 11.26
```

**那条直线就是指数代价的指纹。** 对数尺度下指数函数变成直线,你一看斜率就知道"每加一点输入,墙钟乘一个固定倍数"。这正是第 5 章组合爆炸的延续——只不过那里讲"组合数有多少",这里把它兑成了"等多少秒"。

### 失败模式:"先用暴力跑通,大不了以后优化"

这是 stage 第五段要打掉的工程幻觉。n=20 能跑(0.17 秒)给你一种"算法没问题,只是数据大点"的错觉,于是你拍胸脯说生产上 n=200 也行。**指数增长会让你的直觉系统性失灵**:你的直觉是线性的(数据大 10 倍,慢 10 倍),而 2ⁿ 是数据多 1 个变量就慢 1 倍。n=20 到 n=60 不是"大了 3 倍",是慢了 2⁴⁰ ≈ 一万亿倍。

诚实地说,stage 也标注了这外推的边界:绝对秒数偏**乐观**——真实 SAT 求解器(MiniSat、Z3 这类)有剪枝和启发式,实际能解的工业实例远比 2ⁿ 暗示的大。**可迁移的不是绝对秒数,是那条直线的斜率本身**:指数就是指数,启发式能把墙往后推,但推不平。所以正确的工程动作是:

1. 识别问题是不是 NP 难(查表 / 看能不能从已知 NP 完全问题归约过来)。
2. 是 → 别写朴素暴力上生产,接受"通用最优解算不出"。
3. 转而上**近似算法**(要个差不多的解)、**启发式 / 元启发**(模拟退火、遗传)、**现成 SAT/ILP 求解器**(让别人多年的剪枝替你干活),或**限定输入的特殊结构**(很多 NP 难问题在树状 / 平面 / 小参数等子情形下有多项式算法)。

## 六、把这章压成一张工程检查表

这一章给你的不是算法,是**判断力**——一套"动手前先问的话":

| 信号 | 这是什么边界 | 工程动作 |
|---|---|---|
| 要解析嵌套结构(括号/HTML/JSON) | 超出正则(DFA)的有限内存 | 用 parser / 下推自动机,别堆正则 |
| 把超时误读成"它不会停" | 资源耗尽 ≠ 不停机 | 报"超时",别报"死循环";加配额 |
| "静态判定任意程序会不会死循环/等价" | 停机问题 / 莱斯定理,不可判定 | 放弃通用解,做带超时的保守近似 |
| 问题能从 SAT 等 NP 完全问题归约过来 | NP 难,通用快解大概率不存在 | 上近似/启发式/求解器,别等灵感 |
| n 小能跑就推断 n 大也行 | 指数墙,直觉系统性失灵 | 外推到生产规模再下结论 |

下一章(第 10 章 Fourier)我们回到**能算且算得动**的世界——信号、卷积、FFT 把 O(n²) 压成 O(n log n)。但记住这一章的两条线:有些墙(指数)是用更聪明的算法推后的,有些墙(不可判定)是任何算法都翻不过去的。分清你撞的是哪一种,是工程师比"会调库"更稀缺的能力。
