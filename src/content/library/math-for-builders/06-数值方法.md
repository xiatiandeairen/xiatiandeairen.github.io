`0.1 + 0.2 === 0.3` 在几乎每种语言里都返回 `false`。这不是 JavaScript 的笑话,是 IEEE-754 的事实——`0.1`、`0.2`、`0.3` 三个十进制小数都不是「二进制小数」(分母是 2 的幂的分数),在 64 位浮点里全都只能存近似值。本章对应代码 `examples/math-for-builders-from-scratch/src/stage06-numerics.ts` 实测:

```
0.1 + 0.2          = 3.00000000000000044e-1
0.3                = 2.99999999999999989e-1
0.1 + 0.2 === 0.3  ? false   （差 = 5.551e-17）
```

这个 `5.551e-17` 的差不是 bug,是地板。我见过太多代码——金额对账、几何相交判定、迭代收敛检测——栽在「我假设它跑在 ℝ 上,实际它跑在一个有限集合上」。这一章把这个有限集合的五个高频地雷逐个引爆,每个都对着一个高精度参照量出一个**真实测得的误差数字**,不讲传说。

## 浮点是什么:有限的实数近似

f64 (64 位双精度浮点) 把一个数存成 `符号 × 1.尾数 × 2^指数`:1 个符号位、11 个指数位、52 个尾数位。可表示的数是「二进制小数」(dyadic rational,分母为 2 的幂的有理数) 的一个有限子集。ℝ 是连续无限的,f64 是离散有限的——这两者之间隔着一道永远填不平的缝。

衡量这道缝的标准量是**机器精度** (machine epsilon,能让 `1` 发生变化的最小增量)。实测:

```
机器精度 EPSILON = 2.2204e-16（1 + EPS 才 !== 1）
```

也就是说,在 `1.0` 附近,两个相邻可表示浮点数的间距约 `2.2e-16`。比这更小的差,加上去就被舍入吃掉,`1 + x === 1`。注意这个间距是**相对的**:在 `1e9` 附近,相邻浮点的间距约 `1e9 × 2.2e-16 ≈ 2.4e-7`——数越大,分辨率越粗。这个「大数附近精度变粗」的性质,是下一节灾难性相消的根。

> ✦ **浮点是有限的实数近似,等号几乎从不成立。** 任何对计算结果做 `==` 的代码都是在赌「这两个数恰好落在同一个可表示格点上」,而一旦中间经过加减乘除,这个赌注几乎必输。正确做法是**容差比较**。

### 处方:带绝对 + 相对容差的 `approxEqual`

`stage06-numerics.ts` 里的容差比较函数:

```ts
function approxEqual(a: number, b: number, rtol = 1e-9, atol = 1e-12): boolean {
  return Math.abs(a - b) <= atol + rtol * Math.max(Math.abs(a), Math.abs(b));
}
```

为什么是绝对容差 `atol` + 相对容差 `rtol` 混合,而不是只取一种?

- **只用相对容差** `|a-b| <= rtol·max(|a|,|b|)`:当 `a`、`b` 都趋近 0 时,右边也趋近 0,容差塌缩成「必须严格相等」,在零点附近完全失效。
- **只用绝对容差** `|a-b| <= atol`:在大数量级上失效——`1e9` 和 `1e9+1` 差 1,但在 f64 精度下它们本就「相等」(间距 2.4e-7 > 1?不,1 > 2.4e-7,这里差 1 其实可分辨;但 `1e16` 和 `1e16+1` 就真的不可分辨了)。固定的绝对容差无法跟着量级缩放。

混合形态是工业标准——`numpy.isclose`、Catch2 的 `Approx`、Google Test 的 `EXPECT_NEAR` 都是这个形状。实测它在 `0.1+0.2` vs `0.3` 上给出正确判断:

```
approxEqual(...)   ? true   （0.1/0.2/0.3 都不是 f64 可精确表示的二进制小数）
```

**失败模式(容差自身的坑)**:容差不是越大越好。`rtol` 设太松会把真不相等的数判成相等(假阴性地放过 bug);设太紧又回到 `==` 的脆弱。`rtol=1e-9` 大约留了 7 位十进制的安全余量(f64 满精度约 15-16 位),是「经过几十次浮点运算后仍稳妥」的经验值。真正需要逐位精确的场景(比如下一章信息论里的整数码长、第 8 章密码学的模运算),**根本不该用浮点**——用整数 / 大整数 / 有理数,容差比较救不了你。

## 灾难性相消:有效位瞬间蒸发

这是数值代码里最阴险的 bug,因为公式**代数上完全正确**,代码也**看着对**,结果却**全错**。

> ⚡ **灾难性相消 (catastrophic cancellation):两个相近的大数相减,有效位瞬间蒸发。** 设 `a ≈ 1.234567890123456e9`,`b ≈ 1.234567890000000e9`,二者前 9 位有效数字相同。相减 `a - b ≈ 1.23456e2`——结果只剩约 6 位有效数字,而且这 6 位里有几位本来就是 `a`、`b` 各自的舍入噪声。**前面相同的高位全部抵消,只把藏在低位的舍入误差顶到了台前。** 这是数值不稳定的头号来源。

### 真 bug 演示:同一个方差,两个公式两种命运

方差有两个代数等价的算法:

- **两遍法**(稳定):先扫一遍算均值 `x̄`,再扫一遍累加 `(x - x̄)²`。`core/stats.ts` 的 `variance` 用的就是这个,作为本章的「正确参照」。
- **一遍法**(不稳定):`Σx² - n·x̄²`,只扫一遍。教科书和无数博客都推荐它「省一遍循环」。

`stage06-numerics.ts` 里的一遍法实现:

```ts
function naiveVarianceOnePass(xs: readonly number[]): number {
  const n = xs.length;
  let sum = 0;
  let sumSq = 0;
  for (const x of xs) {
    sum += x;
    sumSq += x * x;
  }
  const meanX = sum / n;
  return (sumSq - n * meanX * meanX) / (n - 1); // (Σx² - n·x̄²)/(n-1)
}
```

喂给它一组**均值很大、方差很小**的数 `[1e9+1, 1e9+2, 1e9+3, 1e9+4]`(真方差就是 `[1,2,3,4]` 的方差 = 5/3,因为平移不改方差)。实测:

```
方差真值 (整数算)      = 1.666666666667
两遍法 (core, 稳定)    = 1.666666666667  误差 0.00e+0
一遍法 (Σx²-n·x̄², 不稳) = 0.000000000000  误差 1.67e+0
```

一遍法把方差算成了 **0**——100% 相对误差,数学上甚至可能算出负方差(不可能的值)。为什么?`Σx²` 和 `n·x̄²` 都是 `1e18` 量级的巨数,二者前十几位有效数字完全相同,相减后真实信号(5/3 量级)早就掉到 f64 在 `1e18` 处的分辨率(约 100)之下了。**两个 18 位的数相减,想得到一个 1 位的结果,而这两个数本身只有约 16 位有效——结果的有效位是负的,纯噪声。**

为什么两遍法稳?它先减去均值,后续累加的全是 `(x - x̄)` 这种**小量**(约 1.5 量级),没有大数相减,相对误差始终被压在 EPS 附近。代价是多扫一遍数据——这就是稳定性的价钱,几乎总值得付。

### 真 bug 演示之二:二次方程的小根整个被吃掉

`x² + 1e8·x + 1 = 0`,两根约为 `-1e8`(大根)和 `-1e-8`(小根)。教科书公式 `(-b ± √(b²-4ac)) / 2a` 对**大根**没问题,但对小根是相消重灾区:当 `b` 很大、`4ac` 很小时,`√(b²-4ac) ≈ |b|`,于是 `-b + √D` 是两个几乎相等的大数相减。`stage06-numerics.ts` 里把稳定式和教科书式并排:

```ts
function quadraticRootsStable(a: number, b: number, c: number): { stable: number; naive: number } {
  const disc = Math.sqrt(b * b - 4 * a * c);
  // q picks the sign so we ADD magnitudes (never subtract near-equals).
  const q = -0.5 * (b + Math.sign(b) * disc);
  const stableRoot = c / q; // the root that the naive formula computes by cancellation
  const naiveRoot = (-b + disc) / (2 * a); // textbook; cancels when b>0 and 4ac«b²
  return { stable: stableRoot, naive: naiveRoot };
}
```

稳定式的技巧:先用**加法**(`b + sign(b)·√D`,符号选成同号相加,绝不相减) 算出条件良好的那个量 `q`,再用韦达定理 `x₁·x₂ = c/a` 反推小根 `c/q`——全程没有相近大数相减。实测:

```
二次方程 x²+1e8·x+1=0 的小根：
真值 (韦达 -c/b)       = -1.000000000000e-8
稳定式 (c/q)           = -1.000000000000e-8  误差 0.00e+0
textbook ((-b+√D)/2a)  = -7.450580596924e-9  误差 2.55e-9
  textbook 式把小根算成了 0：√D 和 b 几乎相等，相减后只剩舍入噪声。
```

教科书式把 `-1e-8` 算成了 `-7.45e-9`——错了约 25%。注意 `7.45e-9` 这个数本身就是 `1e8` 量级浮点的分辨率残渣(`1e8 × EPS / 2 ≈ 1.1e-8`),它是被相消顶上来的纯舍入噪声,跟真根没关系。

**工程经验法则**:任何时候你写 `a - b` 且 `a ≈ b`,停下来问——这两个数是怎么来的?如果它们各自经过浮点运算(不是精确输入),相减就在放大它们的舍入误差。重排公式让相减变成相加,或者用代数恒等式绕开相减(像上面的 `c/q`),是数值稳定的核心手艺。

## Kahan 补偿求和:把丢掉的低位 bit 找回来

累加大量数时,即使没有相消,误差也会**漂移**:每次 `sum += x`,`x` 里太小、塞不进 `sum` 尾数的那几位被直接丢掉;累加几百万次,这些被丢的「灰尘」攒成可见误差。

Kahan 补偿求和的思路:用一个补偿变量 `c` 记住**上一步被丢掉的低位**,下一步加回去。`stage06-numerics.ts` 的实现:

```ts
function kahanSum(xs: readonly number[]): number {
  let sum = 0;
  let c = 0; // running compensation for lost low-order bits
  for (const x of xs) {
    const y = x - c; // bring in the previously-lost dust
    const t = sum + y; // ...which rounds; (t - sum) recovers the high part of y
    c = t - sum - y; // ...and c captures the NEW low part that just got dropped
    sum = t;
  }
  return sum;
}
```

实测累加 1000 万个 `0.1`(真值 100 万,是整数,f64 可精确表示):

```
累加 10,000,000 个 0.1，真值 = 1,000,000
朴素求和   = 999999.99983898  误差 1.610e-4
Kahan 求和 = 1000000.00000000  误差 0.000e+0
Kahan 误差降到 0（真值是整数，补偿把丢掉的低位 bit 全找回了）。
  （纯函数自检：kahanSum 100k×0.1 = 10000.00000000, naiveSum = 10000.00000002）
```

朴素求和漂了 `1.6e-4`——对一笔金额或一个物理量,这可能已经超出可接受误差。Kahan 把它压到 0。注意末行的「纯函数自检」:同样的算法在 10 万个 `0.1` 的独立数组上跑一遍(`naiveSum` 误差 `2e-8`),证明这不是内联循环里的巧合,纯函数版本一致——这是「别用 N=1 的巧合冒充可靠结论」的习惯。

**失败模式(编译器会删掉你的补偿)**:`c = t - sum - y` 在**实数代数**上恒等于 0(因为 `t = sum + y`)。一个假设浮点满足结合律 / 分配律的优化编译器——典型是 C/C++ 开了 `-ffast-math`——会把整个补偿当成无操作**删掉**,Kahan 退化成朴素求和。JS 引擎不做这种重排,所以这里的技巧成立;但在 C 里你需要 `-fno-fast-math` 或把变量标 `volatile`。这条「编译器优化可能破坏数值技巧」的坑,代码注释里专门留了 `NOTE` 提醒未来读者。

**何时不用 Kahan**:它每次迭代多 3 次浮点运算,约 4 倍开销。如果只加几百个数,误差本就在 EPS 量级,不值得。它的主场是**长序列**(百万级以上) 或**量级差异大**(把 `1e-10` 加进 `1e10`)。中间档可以用「成对求和」(pairwise summation,分治两两相加,误差从 O(n)·EPS 降到 O(log n)·EPS),numpy 的 `sum` 默认就是它——比朴素好很多,比 Kahan 便宜。

## 条件数:病态问题放大扰动

前面讲的是**算法**不稳定(同一问题,坏公式)。条件数讲的是**问题本身**有多敏感——再完美的算法也救不了一个病态问题。

**条件数** (condition number,记作 κ) 衡量「输入相对扰动」被放大成「输出相对变化」的倍数:`输出相对误差 ≲ κ × 输入相对误差`。κ 接近 1 = 良态;κ 很大 = 病态,输入抖一下解就崩。对解线性方程组 `Mx = b`,κ 就是矩阵 `M` 的条件数(最大奇异值 / 最小奇异值)。这直接接上第 3 章线性代数:一个行几乎线性相关的矩阵,行列式接近 0,逆矩阵元素巨大,就是病态的几何根源。

`stage06-numerics.ts` 用一个对称 2×2 的近奇异矩阵 `[[1,1],[1,1+1e-10]]`(两行几乎相同) 演示。它用闭式特征值算真实条件数 `κ = |λmax/λmin|`,然后给右端项 `b` 一个 `1e-8` 量级的确定性扰动(`core/rng.ts` 的种子随机数保证可复现),看解漂多远:

```
矩阵 [[1,1],[1,1+1e-10]] 的条件数 κ = 4.000e+10
输入(RHS) 相对扰动 = 1.168e-8
输出(解) 相对变化  = 3.841e+2
放大倍数 ≈ 3.288e+10（理论上界就是 κ；κ 大 = 输入一抖解就崩）
```

输入抖了亿分之一(`1.2e-8`),解变了 **384 倍**(相对变化 `3.8e2`)。实测放大倍数 `3.3e10`,紧贴理论上界 κ = `4e10`。这就是病态的含义:**问题对输入的敏感度是固有的,与你用什么算法无关。**

**工程含义**:遇到病态问题,你能做的不是「换个更精确的求解器」(没用,问题本身放大了误差),而是:(1) 重新建模让矩阵良态(归一化、换基、加正则项 / Tikhonov 正则);(2) 用对病态更鲁棒的分解(QR、SVD)而不是直接求逆;(3) 至少**先算 κ**,知道自己站在悬崖边上。在第 4 章优化里,海森矩阵 (Hessian) 的条件数大正是牛顿法 / 梯度下降收敛慢的根因,那里和这里是同一个 κ。

## 牛顿法:正确位数每步翻倍

换个角度看数值方法的**好**的一面。牛顿法解 `f(x) = 0` 的迭代是 `x ← x - f(x)/f'(x)`——几何上就是「在当前点画切线,取切线与 x 轴的交点作下一个猜测」。它在单根附近**二次收敛**:正确的有效位数大约每步翻一倍。这是它值钱的地方,也接上第 4 章——优化里求梯度为零的点用的就是牛顿法(对一维是求导数的根)。

`stage06-numerics.ts` 的实现刻意返回**整个误差历史**,因为「位数翻倍」这个卖点只有逐步打印误差才看得见:

```ts
function newtonRoot(
  f: (x: number) => number,
  df: (x: number) => number,
  x0: number,
  steps: number,
): number[] {
  const history: number[] = [x0];
  let x = x0;
  for (let i = 0; i < steps; i++) {
    const slope = df(x);
    if (slope === 0) break; // flat tangent -> step undefined; bail loudly via short history
    x = x - f(x) / slope;
    history.push(x);
  }
  return history;
}
```

用它解 `x² - 2 = 0`(即求 √2),从 `x0 = 1` 起步,实测:

```
  step 0: x = 1.0000000000000000  误差 4.14e-1  正确位数≈0.4
  step 1: x = 1.5000000000000000  误差 8.58e-2  正确位数≈1.1
  step 2: x = 1.4166666666666667  误差 2.45e-3  正确位数≈2.6
  step 3: x = 1.4142156862745099  误差 2.12e-6  正确位数≈5.7
  step 4: x = 1.4142135623746899  误差 1.59e-12  正确位数≈11.8
  step 5: x = 1.4142135623730951  误差 0.00e+0  正确位数≈全部
  误差指数每步约翻倍 → 二次收敛，这是牛顿法值钱的地方。
```

看误差列:`4e-1 → 9e-2 → 2e-3 → 2e-6 → 2e-12 → 0`。每一步的误差指数大约是上一步的两倍(`-1 → -3 → -6 → -12`),5 步就从 0.4 位有效数字打到满精度。日志里的 ASCII 收敛曲线(用 `core/plot.ts` 的 `asciiLine` 画 log₁₀ 误差) 是一条急速下坠的直线,直观看出超线性收敛:

```
误差(log10)收敛曲线：
-0.383 │●●
       │  ●
       │   ●
       │    ●
 -16.0 │     ●
```

## 失败模式:数值方法怎么坏,为什么

把上面四个机制的失败模式集中演示——这些不是边角料,是生产里真会咬人的地方。

### 病态矩阵直接求逆 = 把舍入噪声放大成 O(1) 误差

「解线性方程组别求逆矩阵」是数值课的老生常谈,这里量化它为什么。`stage06-numerics.ts` 把矩阵收紧到**数值奇异**——`[[1,1],[1,1+EPS]]`,`1+EPS` 是「能让 1 变化的最小量」,再小就真奇异了——然后故意用显式求逆去解:

```ts
function solve2x2ViaInverse(m: number[][], rhs: number[]): number[] {
  const [[a, b], [c, d]] = m;
  const det = a * d - b * c;
  if (det === 0) throw new Error("[stage06] solve2x2: singular matrix, inverse undefined");
  const inv = [
    [d / det, -b / det],
    [-c / det, a / det],
  ];
  return matVec(inv, rhs); // 复用 core/linalg.ts 的矩阵×向量
}
```

构造右端项让真解恰为 `[1, 1]`,实测:

```
病态矩阵 det = 2.220e-16（≈EPS）
真解 = [1, 1]，直接求逆解 = [2.0000, 0.0000]，解的误差 = 1.41e+0
  RHS 里一个 EPS 量级的舍入被 1/det≈1/EPS 放大成 O(1) 的解误差 → 病态问题别求逆，用 QR/SVD/带 pivot 的消元。
```

求逆要算 `1/det`,而 `det ≈ EPS ≈ 2.2e-16`,所以 `1/det ≈ 4.5e15`。构造右端项 `[2, 2+EPS]` 时,`2+EPS` 已经发生了一次 EPS 量级的舍入;这点舍入被 `1/det` 放大 `1/EPS` 倍,正好顶成 O(1) 的解误差——真解 `[1,1]` 被算成 `[2,0]`,误差 1.41(相当于解完全错)。处方:别求逆,用带部分主元的高斯消元 (partial pivoting) 或 QR / SVD 分解,它们不显式计算 `1/det`,对病态鲁棒得多。

### 牛顿法在驻点 / 坏起点处除零或发散

牛顿法快,但**不是全局安全**的。它的步长是 `f(x)/f'(x)`,只要 `f'(x) ≈ 0`(切线水平),步长就爆炸或除零。`stage06-numerics.ts` 从 `x0 = 0` 起步解 `x² - 2 = 0`:`f'(0) = 2·0 = 0`,切线水平,第一步就无定义:

```
牛顿法从 x0=0 解 x²-2=0：迭代历史长度 = 1（f'(0)=0，切线水平，第一步就无定义直接中止）
  牛顿法快但不安全：起点差 / 驻点处 f'≈0 会发散或除零，生产里要配 bisection 兜底或带阻尼。
```

注意实现里 `if (slope === 0) break` 是**显式响亮地中止**(返回长度为 1 的短历史),而不是让 `f(x)/0 = Infinity` 静默污染后续——「失败要吵闹,不要假装成功」是数值代码的基本卫生。生产处方:给牛顿法配兜底——用二分法 (bisection,一定收敛但慢) 框住根的区间,牛顿步跳出区间或 `f'` 太小时退回二分;或用阻尼牛顿 (damped Newton,步长乘一个 <1 的因子) 防止跨过头。这就是为什么真实的求解器(SciPy 的 `brentq`、各种 line search)都是混合算法,从不裸用牛顿。

## ⚡ 前沿:浮点累加误差没有通用最优解

回到 Kahan 那一节的核心问题:**给定一串浮点数,如何求和才能误差最小?** 这个看似初等的问题,至今**没有「又快又总是最优」的通用解**,仍在研究。

- 朴素求和:误差界 `O(n)·EPS`,最快,最差。
- 成对求和:误差界 `O(log n)·EPS`,几乎不增开销,numpy 默认。
- Kahan / Neumaier / Klein 等补偿求和:误差界接近 `O(1)·EPS`,但每个数多 3-7 次运算,且**依赖编译器不重排**(`-ffast-math` 会废掉它)。
- 完全精确求和(如 Shewchuk 的无误差变换、Rump 的 `AccSum`、`xsum` 库):能给出**正确舍入**的结果(等价于先无限精度相加再舍入一次),但开销显著,且实现复杂。

没有一个方案同时拿下「最快 + 最准 + 对编译器优化鲁棒 + 不依赖输入分布」。误差大小还取决于数据的量级分布和加法顺序——而最优加法顺序本身是个难题。再往上,GPU / SIMD 上的并行归约 (parallel reduction) 因为加法**不满足结合律**,结果会随线程数、归约树形状变化,导致「同一份代码同一份数据,换块卡结果就不同」——这是机器学习训练里**实验不可复现**的隐形来源之一,确定性归约 (deterministic reduction) 至今是性能与可复现性之间的活跃权衡,无免费午餐。

务实的结论:知道你的精度需求和数据规模,选对应档位(几百个数→朴素;百万级→成对;要钱要命的金额/科学计算→补偿或精确求和;要逐位精确→干脆别用浮点,上整数 / 定点 / 有理数)。把「求和怎么求都一样」这个错觉从脑子里删掉。
