> 这章只做一件事:把"one-hot 是天真的第一步"这句正确但廉价的话,变成你能在代码里重新跑出来、能对着数字下判断的结论。00 章给了路线图——从 one-hot 一路走到对比学习的 embedding。这一站我们要钉死起点为什么不能停在 one-hot:它的失败不是调参能救的,是结构性的。理解这个结构性失败,后面每一章(02 的共现、03 的 skip-gram、05 的 InfoNCE)才知道自己到底在修什么病。

先把结论摆在最前面,免得你读完一堆推导才发现重点:**one-hot 编码下,"猫"和"狗"的余弦相似度严格等于 0,而且"猫"和"鱼"的相似度也是 0——一模一样的 0。** 不是"差不多 0",是数学上精确的 0。这意味着无论你下游接多强的模型,它从输入的几何结构里读不到"猫和狗都是动物"这一个 bit 的信息,因为这个信息根本没被编码进去。这是病根。本章其余部分都是这个病根派生的症状。

配套代码在 `examples/embeddings-from-scratch/src/stage01-onehot.ts`,所有数字都来自它的真实运行,可以 bit-for-bit 复现(随机数走 seeded RNG,不是 `Math.random`)。建议你边读边把它跑一遍。

## 一、one-hot 是什么,为什么它等于"用标准基"

one-hot 编码(独热编码,一个词对应一个只有单一位置为 1 的向量)的定义朴素到不需要解释:词表有 `V` 个词,第 `i` 个词就编码成一个长度 `V`、第 `i` 位是 1、其余全 0 的向量。

但有一个视角值得你立刻内化,因为它直接决定了后面所有的坏消息。把整个词表的 one-hot 向量按行堆起来,你得到的是一个 `V×V` 的矩阵——而它**就是单位矩阵**(identity matrix)。代码里我故意把它显式造出来,而不是用某个库的稀疏表示,就是要你看见这一点:

```typescript
// stage01-onehot.ts:46
function buildOneHotMatrix(vocabSize: number): number[][] {
  const rows: number[][] = [];
  for (let i = 0; i < vocabSize; i++) {
    const row = new Array(vocabSize).fill(0);
    row[i] = 1;
    rows.push(row);
  }
  return rows;
}
```

"对一个词表做 one-hot"在数学上等价于"用标准正交基的基向量给每个词编号"。一旦你接受这个等价,后面的灾难就是必然的:**标准基的任意两个不同基向量,内积恒为 0,夹角恒为 90°。** 几何上,每个词都被钉在一个互相垂直的坐标轴上,谁也不比谁更靠近谁。

## 二、✦ 正交性:相似度恒为 0 才是病根

把上一节的几何事实落成测量。stage01 选了几对人类一眼能判断亲疏的词,算它们的 one-hot 余弦相似度:

```
(1) one-hot orthogonality  (vocab size V = 43)
  cos(king  , queen ) = 0.0000  (distinct → exactly 0)
  cos(dog   , cat   ) = 0.0000  (distinct → exactly 0)
  cos(king  , fish  ) = 0.0000  (distinct → exactly 0)
  cos(king  , king  ) = 1.0000  (same word → 1)
  measured max |cos| over all 903 distinct pairs = 0.0000
```

注意最后一行:这不是抽样,是**扫遍全部 903 对不同词**(`43×42/2`)测出来的最大值,它是 0。也就是说在这个 43 词的玩具词表里,**不存在任何一对不同词的相似度大于 0**。`king` 离 `queen` 的距离,和它离 `fish` 的距离,完全相等。

这一点为什么是病根,而不只是众多缺点之一?因为它**封死了几何里"更相似"的表达空间**。相似度只有两个取值:同一个词 = 1,不同词 = 0。中间那一整段——"有点像""挺像""几乎一样"——在 one-hot 里**无处安放**。下游模型想学"猫和狗相关",输入里压根没有可供它依附的信号。你给它再多数据、再深的网络,也变不出输入里不存在的那个维度。

对比一下后面章节要追求的东西:第 03 章的 skip-gram 学出来的向量里,`king` 和 `queen` 的余弦会落在 0.7 量级而 `king` 和 `fish` 接近 0——**那个"0.7 vs 0.1"的差,就是 one-hot 永远给不出的东西。** 这一章其实是在论证那个差为什么必须靠"学",而不能靠"编码方式选得更聪明"。

### 为什么不能靠"把 one-hot 拼起来"绕过

一个很自然的反驳:单个词正交我认,但一篇文档是很多词,把词的 one-hot 加起来(词袋,bag-of-words,把文档表示成各词计数的向量)不就有重叠了吗?重叠确实有,但只在**字面相同的词**上有。两篇文档要想相似度非 0,必须共享至少一个**literal token**(字面相同的词)。意思相近但用词不同的两篇——比如一篇讲 `king/queen`,一篇讲 `prince/princess`——依然正交,依然 0。第四节会用真实检索数字把这个漏洞演给你看。

## 三、⚡ 维度灾难:高维里"最近邻"会失去意义

正交性是 one-hot 的第一宗罪。第二宗罪更隐蔽,也更深,而且**目前没有通用解法**(后面细说):维度灾难(curse of dimensionality)。

通俗版本人人会背:维度越高越慢、越占内存。但这只是表层。真正致命的是一个反直觉的几何事实——**在足够高的维度里,一堆随机点会变得两两近似正交,所有点对的距离趋于相同。** 也就是说"最近的点"和"最远的点"几乎一样远,"最近邻"这个概念本身退化成噪声。

不要信我嘴说,看 stage01 的测量。它在 `d = 2/10/100/1000` 四个维度里各撒 200 个标准正态(Gaussian)随机点,测最近点对距离和最远点对距离的比值。这个比值越接近 1,说明距离对比度越差——"近"和"远"越没区别:

```
(2) curse of dimensionality: nearest/farthest distance ratio → 1
  N = 200 seeded Gaussian points per dimension
  d      minDist   maxDist   min/max ratio
  ----   -------   -------   -------------
  2        0.016     5.993   0.0027
  10       1.366     8.766   0.1558
  100     10.931    18.375   0.5949
  1000    40.587    48.325   0.8399

d=2    │░░░░░░░░░░░░░░░░░░░░░░░░░░░░░░│ 0.003
d=10   │██████░░░░░░░░░░░░░░░░░░░░░░░░│ 0.156
d=100  │█████████████████████░░░░░░░░░│ 0.595
d=1000 │██████████████████████████████│ 0.840
```

读这张表的方式:在 `d=2`,最近点对的距离只有最远点对的 **0.3%**——近和远天差地别,"最近邻"信息量饱满。到 `d=1000`,这个比值飙到 **0.84**——最近的那个点已经差不多和最远的点一样远了。在这个维度上做"找最相似",你找回来的东西和随机抓一个的区别,小得可怜。

这里用 Gaussian 随机点而不是均匀立方体里的点,是有讲究的:Beyer 等人 1999 年关于距离集中(distance concentration)的经典结果,对各坐标独立同分布的情形最干净,而 Gaussian 也正好是本书后面所有向量初始化用的分布。换句话说,这不是为了好看挑的分布,是和后续章节一脉相承的。

**诚实声明(toy 标注)**:这四个绝对数字是玩具规模(每维 200 点、固定 seed),换个 seed、换个点数,具体小数会变。能转移到真实场景的是**趋势的方向**——比值随维度单调爬升、逼近 1——而不是 `0.84` 这个具体值。这是本书贯穿的纪律:相信趋势,不迷信玩具的小数点。

现在把这个事实接回 one-hot:**one-hot 活在 `d = V` 的空间里。** 真实词表 `V` 是 10 万量级,远在上表 `d=1000` 这一行的右边,早就泡在距离坍缩的区域里了。所以"在 one-hot 上做最近邻检索"这件事,即便抛开正交性不谈,光维度这一条就已经让它接近无意义。

### ⚡ 为什么说"目前无通用解"

需要把话说准,避免读者以为"换个度量/降个维就解决了":

- 距离集中是高维 iid 随机向量的**固有几何性质**,不是某个算法的 bug。你不能通过"选个更好的距离函数"把它消掉——换成余弦、换成曼哈顿,集中现象依然在,只是程度不同。
- 真正的出路不是"消除维度灾难",而是**让数据不要均匀填满高维空间**。真实语义数据有低维结构(所谓 manifold hypothesis,流形假设:高维数据其实集中在一个低维曲面附近),稠密 embedding 之所以能用,正是因为它把数据塞进了一个**有结构的低维空间**,而不是 one-hot 那种无结构的高维空间。
- 但"高维数据的内在维度到底多少、怎么可靠估计、流形假设在什么数据上成立到什么程度",这些**至今没有通用答案,仍是活跃研究领域**。我们后面的稠密向量是工程上"够用"的近似,不是这个理论问题的解。这个⚡你要一直记着:本书教你绕开维度灾难的工程手段,但维度灾难本身没被任何人"解决"。

## 四、把两宗罪合起来看:检索任务上的实测对比

前两节是机制。这一节让机制在一个具体任务上撞墙,并预览解药长什么样。任务很简单:6 篇文档,3 个主题(royalty / animal / food)各 2 篇,给定一篇,找最相似的另一篇,正确答案永远是"同主题的那一篇"。

这里有个**为了诚实而刻意设的约束**:同主题的两篇文档**不共享任何字面词**。royalty 的两篇分别是 `{king,queen,rules}` 和 `{prince,princess,kingdom}`。如果它们共享一个词,词袋 one-hot 会靠字面重叠"蒙对",本章的论点就被自己的数字偷偷打脸了。所以代码里把每个主题的两篇拆成了不相交的词集:

```typescript
// stage01-onehot.ts:206
const DOCS: LabeledDoc[] = [
  // royalty — disjoint: {king,queen,rules} vs {prince,princess,kingdom}
  { label: "royalty-A", words: ["king", "queen", "rules"] },
  { label: "royalty-B", words: ["prince", "princess", "kingdom"] },
  // animals — disjoint: {dog,cat,run} vs {mouse,bird}
  { label: "animals-A", words: ["dog", "cat", "run"] },
  { label: "animals-B", words: ["mouse", "bird"] },
  // food — disjoint: {bread,cheese} vs {rice,fish}
  { label: "food-A", words: ["bread", "cheese"] },
  { label: "food-B", words: ["rice", "fish"] },
];
```

跑两种编码——词袋 one-hot(43 维)对手工搭的稠密主题码(3 维):

```
(3) topical retrieval: bag-of-words one-hot vs hand-built dense
  [A] bag-of-words over one-hot  (dim = V = 43)
    royalty-A  → royalty-B  (cos 0.000)  ✓ same topic
    royalty-B  → royalty-A  (cos 0.000)  ✓ same topic
    animals-A  → royalty-A  (cos 0.000)  ✗ wrong topic
    animals-B  → royalty-A  (cos 0.000)  ✗ wrong topic
    food-A     → royalty-A  (cos 0.000)  ✗ wrong topic
    food-B     → royalty-A  (cos 0.000)  ✗ wrong topic
    hits: 2/6

  [B] hand-built dense topic code  (dim = 3: royalty/animal/food)
    ... 全部命中 ...
    hits: 6/6
  verdict: dense 6/6 vs one-hot 2/6 in 3 dims vs 43.
```

几个点必须看清楚:

1. **one-hot 那 2/6 是怎么蒙对的。** 看 royalty-A→royalty-B 那行,余弦是 `0.000`,可它被判成命中了。这恰恰暴露了 one-hot 检索的荒诞:**所有候选的相似度都是 0,全平局,谁排第一全靠 tie-break(取最小下标)。** royalty 那两篇凑巧被下标顺序排到了一起,纯属运气。其余四篇全部错指向 `royalty-A`(下标最小的那个),因为它们离谁都"一样远(0)",平局后默认抓下标最小的。这不是检索,是抽签。

2. **3 维稠密码用 1/14 的维度做到了 6/6。** 因为它把 `king` 和 `prince` 放到了**同一根 royalty 轴**上,字面不同但几何上靠近,余弦非 0。这正是 one-hot 结构上做不到的事——graded similarity(分级相似度)。

诚实声明:这里的稠密码是**人手摆的**(`king:[1,0,0]`,`dog:[0,1,0]`...),不是学出来的。它的作用是证明"稠密空间有能力表达分级相似",**不是**证明"我们已经会自动得到它"。把这些轴从数据里自动学出来,是 02(共现统计)和 03(可学习向量)的任务。这一章只负责把"为什么需要它"钉死。

## 五、失败模式:在 one-hot 上做欧氏最近邻 ≈ 抛硬币

新手最容易掉进的坑,值得单独拎出来量化:"既然有了向量,我直接对词袋向量做欧氏距离(Euclidean,直线距离)最近邻不就行了?" stage01 专门演示了这条路烂到什么程度:

```
FAILURE MODE: Euclidean NN on one-hot bag-of-words is ~random
  query = royalty-A (words: king,queen,rules)
  Euclidean distance to every other doc:
    animals-B  dist=2.236  (different topic)
    food-A     dist=2.236  (different topic)
    food-B     dist=2.236  (different topic)
    royalty-B  dist=2.449  (SAME topic — should be nearest)
    animals-A  dist=2.449  (different topic)
  Euclidean top-1 topical accuracy: 1/6 = 16.7%
  random-guess baseline (1 of 5): 20.0%
```

看 `royalty-B` 那行:它是**应该排第一**的同主题文档,结果距离 `2.449`,反而比三篇不同主题的文档(`2.236`)**更远**。为什么?因为词袋的欧氏距离平方等于 `|A| + |B| - 2k`,`k` 是两篇共享的词数。同主题但不共享词(`k=0`)的文档,距离完全由各自词数决定,和"主题"半点关系没有。`royalty-B` 有 3 个词、那几篇竞争者词数更少,于是同主题的反而被推远了。

最后两行是判决:欧氏最近邻的主题命中率 **16.7%**,而**纯随机基线是 20%**——它比瞎猜还差一点点。(toy 标注:6 篇文档让百分比很粗糙,16.7% 和 20% 这种细微高低别当真;真正稳的是机制——距离只认字面词重叠、不认意思——这个在任何词表规模下都成立。)

这条失败模式的教训值千金:**度量本身没有错,错的是把它喂给了一个不编码语义的表示。** 你后面会无数次遇到这个分工——余弦/欧氏只是尺子,尺子量出什么取决于向量里被编码了什么。one-hot 的尺子量的是"字面重不重叠",而我们想要的是"意思近不近"。换尺子救不了,得换向量。

## 六、所以下一站要什么

把这一章的账结一下,three counts of failure:

1. **正交(✦)**:任意两个不同词余弦恒为 0,几何里没有"更相似"的容身之处——病根。
2. **维度灾难(⚡)**:one-hot 活在 `d=V` 的高维空间,距离对比度坍缩(实测比值 `d=2` 的 0.003 爬到 `d=1000` 的 0.840),"最近邻"退化为噪声;而这背后的高维几何问题至今无通用解。
3. **零泛化**:对字面没见过的词组合,相似度一律 0,检索退化成抛硬币(实测 16.7% ≈ 随机 20%)。

而第四节那个 3 维稠密码 6/6 的反例,已经把出路指明了:**我们要的是低维、稠密、且把语义相近的东西放在几何相近位置的向量。** "低维"治维度灾难,"稠密 + 语义对齐"治正交和零泛化。

唯一的问题是:那些轴不能再靠人手摆。**怎么从数据里自动学出"king 和 prince 该在同一根轴上"?** 最古典的答案是统计学的——下一章(02)从一个朴素观察出发:意思相近的词,倾向于出现在相似的上下文里("you shall know a word by the company it keeps")。我们会用共现矩阵 + PMI(逐点互信息)把这个直觉变成第一批不靠手摆、而是数出来的稠密向量,然后看它能不能复现本章那个 6/6。再往后的 03 章,才把"数出来"升级成"学出来"。
