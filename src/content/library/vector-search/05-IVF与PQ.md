HNSW（第 4 章那个图索引）有个它自己解决不了的问题：它只让你**少比**，没让你**比得更便宜**。每个向量在内存里仍然是 `dim` 个 float，128 维就是 512 字节（单精度）或 1KB（双精度）。十亿向量 × 512 字节 = 512GB 纯 payload，还没算图的边。这台机器不存在，或者很贵。HNSW 的图结构本身又是另一笔内存（每个节点几十条边的指针）。所以「召回优先、库装得下内存」的场景 HNSW 赢；「库装不下内存」的场景，你需要的是另一套东西——压缩。

IVFPQ 就是这套东西。它由两个**正交**的机制拼成，各自砍一笔成本：

- **IVF**（inverted file，倒排文件）—— 聚类分簇。把向量先 k-means 分到 `nlist` 个簇，查询时只查离查询最近的 `nprobe` 个簇，跳过其余。少比向量。
- **PQ**（product quantization，乘积量化）—— 子空间量化。把每个向量切成 `nsub` 段，每段独立用 256 个质心量化成 1 个字节。比得更便宜，存得更省。

关键判断先给你：**这两个机制各自独立损失召回，而且损失会叠加**。IVFPQ 的召回低于单跑 IVF、也低于单跑 PQ。理解这点是用好它的前提——你有两个旋钮（`nprobe` 和 `nsub`），调错任何一个都会塌，而且塌的原因不一样。下面用配套仓的 stage 05 把这两笔损失拆开量化给你看。

代码在 `examples/vector-search-from-scratch/src/stage05-ivfpq.ts`。数据集是 5000 个 16 维向量、16 个真实簇、100 条查询、k=10。IVF 切 64 个 cell（`nlist=64`，大致是 √5000）。下面所有数字都是这个 stage 跑出来的实测，内存是 payload 下限估算（标注清楚，真实 V8 堆要乘 1.5–3 倍）。

## PQ：为什么 32 倍压缩不是魔法

先看 PQ 单独干了什么。一个 16 维双精度向量是 128 字节。PQ 切成 `nsub=4` 段（每段 4 维），每段查一个 256 质心的码本（codebook），存它最近质心的 id（0–255，正好 1 字节）。于是整个向量从 128 字节变成 4 字节。stage 实测：

```
full float vectors : 625.0 KB
PQ codes (nsub=4) : 19.5 KB  (+32.0 KB fixed codebook)
compression ratio  : 32.0x (codes only; 12.1x incl. codebook)
```

32 倍。这是「能不能装进内存」和「不能」的差别。但注意那个括号里的 12.1×——码本是**固定开销**，跟向量数无关。5000 向量时码本（32KB）比 codes（19.5KB）还大，压缩比被拖到 12 倍；十亿向量时码本可以忽略，压缩比就回到 32 倍。这是 PQ 只在大规模才划算的原因之一：小库别用 PQ，码本开销吃掉收益还赔上召回。

为什么子空间切分能做到这个压缩率？关键在码本是**乘积**的。`nsub=4` 个码本、每个 256 质心，组合起来能表达 256⁴ ≈ 43 亿个不同的重构点，而存储只要 4 个码本（`examples/vector-search-from-scratch/src/stage05-ivfpq.ts` 第 178–180 行的注释把这个算清楚了）。要用单个扁平码本达到同样表达力，你得存 43 亿个质心——那比原始数据还大。乘积量化的「乘积」就是指这个笛卡尔积式的表达力放大。

切分用连续块而不是跨步取（`subSlice`，第 191–196 行）也是有讲究的：真实 embedding 里相邻维度往往相关，把它们放进同一个子空间一起量化，比把相关维度拆散到不同子空间误差更小。

### 失败模式：子段太少，量化误差吞掉一切

`nsub` 是 PQ 的内存/召回旋钮。调小它内存更省但量化更粗。stage 把这条曲线整个扫出来（`nprobe=8` 固定）：

```
nsub  bytes/vec  recall@10   compression
1            1    0.3310   128.0x
2            2    0.5210   64.0x
4            4    0.6830   32.0x
8            8    0.8980   16.0x
```

读这张表读出两件事。第一，召回随 `nsub` 单调上升，这符合直觉——切得越细，每个子空间维度越少，256 个质心覆盖得越密。第二，也是更重要的：`nsub=1` 时召回崩到 0.33。一个字节描述整个 16 维向量，意味着全库所有向量被强行归到 256 个重构点上，量化误差大到 ADC 距离（下面讲）排序基本失效。这是 **under-coded 失败模式**——压缩拉满，召回崩盘。

实战含义：`nsub` 不是越小越好，存在一个对你的数据「够用」的下限。FAISS 里你会看到 `PQ8`、`PQ16`、`PQ32` 这些配置，数字就是 `nsub`。选多大没有公式，得在你自己的 embedding 上扫这条曲线——合成的高斯簇（本 stage 的数据）比真实 embedding 友好，所以这里的绝对召回偏乐观，但**趋势**是可迁移的（这点第 7 章评测会展开讲为什么合成数据偏乐观）。

## ADC：查询时怎么用 1 字节的码算距离

存成 1 字节的码，怎么算距离？不可能把码解回向量再算——那等于没压缩。PQ 的查询时技巧叫 **ADC**（asymmetric distance computation，非对称距离计算）：查询保持全精度，库这边是量化的，所以叫「非对称」。

机制是查表（`buildAdcTable` / `adcDistance`，第 250–266 行）。对一条查询，预先算出「查询的每个子向量」到「该子空间码本里每个质心」的距离，存成一张 `nsub × 256` 的表。之后每个候选的距离 = 把它 `nsub` 个码对应的表项加起来。`nsub` 次加法，零次乘法，没有逐候选的向量运算：

```ts
function adcDistance(table: number[][], code: Uint8Array): number {
  let sum = 0;
  for (let m = 0; m < table.length; m++) sum += table[m][code[m]];
  return sum; // squared distance estimate; monotonic, fine for top-k ranking
}
```

这张表**每条查询只建一次**，然后摊到所有候选上。这就是 PQ 在查询时真正省的地方——不是省存储，是把候选打分从「一次全维度距离」降成「`nsub` 次查表加法」。候选越多，摊得越值。

## IVF：少查几个簇，但会漏

PQ 让每次比较便宜，IVF 让你少比。把全库 k-means 成 `nlist` 个 cell，查询时按「查询到各 cell 质心的距离」排序，只打开最近的 `nprobe` 个 cell，对它们的成员做 ADC 排序。`nprobe` 是 IVF 的召回/速度旋钮。stage 扫出来：

```
nprobe  recall@10   QPS      cells scanned
1         0.4180    23538   1/64
2         0.5470    17548   2/64
4         0.6590    10146   4/64
8         0.6830     5606   8/64
16        0.6850     2809   16/64
64*       0.6850      538   64/64 (exhaustive IVF)
```

这张表是本章的核心。逐行读：

- `nprobe=1` 召回只有 0.42，QPS 高达 23538（对照暴力扫描的 826 QPS，快 28 倍）。
- `nprobe` 往上加，召回涨、QPS 跌——典型的 speed/recall trade-off。
- 到 `nprobe=16` 召回就 0.685 封顶了，再往上加到 64（即查全部 cell，IVF 完全不剪枝）召回**纹丝不动**还是 0.685，但 QPS 从 2809 暴跌到 538。

最后这点最值得停下来想。`nprobe=64` 时 IVF 已经不剪枝了——所有 cell 都查——可召回还是上不去。**这说明 0.685 这个天花板不是 IVF 造成的,是 PQ 的量化损失。** 当你把 `nprobe` 调到 `nlist` 还提不动召回，再加 `nprobe` 纯属浪费 QPS，该去调 `nsub` 了。两个旋钮治不同的病:`nprobe` 治剪枝漏召回,`nsub` 治量化损失。分不清就会在错误的旋钮上空转。

### 失败模式：nprobe=1 时,真正的近邻在没打开的簇里

为什么 `nprobe` 小会漏？因为查询可能落在簇的边界上,它真正的最近邻在隔壁 cell——那个 cell 的质心是第 2 近或第 3 近,`nprobe=1` 根本不打开它。stage 把这个量化得很直接(第 472–497 行):

```
--- failure mode B: nprobe=1 cell-miss on true #1 neighbor ---
  43/100 queries have their true #1 neighbor in an UNPROBED cell.
```

100 条查询里 **43 条**的 #1 近邻落在 `nprobe=1` 没打开的 cell 里。这 43 条在 `nprobe=1` 下**无论 PQ 多好都救不回来**——近邻根本不在候选集里,ADC 算得再准也排不到一个不存在的东西。唯一的解药是加 `nprobe`(回去看上面那张扫描表,召回随 `nprobe` 涨就是在捞回这些边界查询),代价是 QPS。

这是 IVF 这类「分簇剪枝」索引的结构性弱点:它赌「最近邻在最近的几个簇里」,而簇边界上的查询会让这个赌注输掉。HNSW 没有这个特定失败模式(它是图遍历不是硬分簇),但代价是 HNSW 存不下大库。没有免费的午餐。

## 最隐蔽的 bug:PQ 必须在残差上训练

到这里 IVF 和 PQ 都讲完了,但把它们拼起来有个**不拼不知道、拼了就踩**的坑:PQ 不能在原始向量上训练,必须在**残差**(residual,向量减去它所属 cell 的质心)上训练。

为什么?原始向量散布在整个数据空间(本 stage 是 10 宽的盒子),码本得把分辨率浪费在表达「簇与簇之间的偏移」上——而这个偏移其实 IVF 的粗质心已经免费告诉你了。残差是「向量相对它自己簇心的位置」,它们都聚在原点附近、跨所有 cell 尺度相近,一个共享码本能把它们量化得紧得多。代码里就是 `buildIvfpq` 第 298–302 行那个减法:

```ts
const trainVecs = dataset.map((v, i) => {
  if (!useResiduals) return v;
  const centroid = coarse.centroids[coarse.assignment[i]];
  return v.map((x, d) => x - centroid[d]);
});
```

stage 把残差训练和原始训练并排跑,gap 直接摆出来(第 456–470 行,`nsub=4, nprobe=8`):

```
  residual-trained PQ : recall@10 = 0.6830
  raw-trained PQ      : recall@10 = 0.4730
  gap                 : 0.2100
```

**0.21 的召回,白捡的。** 同样的内存、同样的 `nprobe`、同样的 `nsub`,只是训练时减没减簇心,召回差 21 个点。这是经典的 IVFPQ 实现 bug——很多人自己写一版,PQ 直接喂原始向量,然后纳闷为什么召回比 FAISS 差一大截。残差这步在论文里是一句话,在代码里是一个减法,漏了不报错、只是默默掉召回。

还有个配套的细节容易错(第 350–352 行):查询时,既然库里存的是残差码,查询也必须**相对当前 cell 的质心**表达成残差再去查表,否则几何对不上。用原始查询去查残差码是个更隐蔽的正确性 bug——它不崩,只是结果错。

## 何时选 IVFPQ,何时选 HNSW

把第 4 章和这章放一起,决策其实清楚:

- **内存受限、库装不下全精度** → IVFPQ。32 倍压缩让十亿向量进单机内存,这是它存在的全部理由。代价是召回有天花板(两笔损失叠加),且对数据分布敏感(簇要分得开)。DiskANN、Milvus 的大规模档位、FAISS 的 `IVF*,PQ*` 工厂串都是这条路。
- **召回优先、库装得下内存** → HNSW(第 4 章)。它召回更高、没有 `nprobe` 那种边界漏召回,代价是每个向量全精度存 + 图的边开销,撑不住十亿规模。hnswlib、pgvector 默认走这条。

中间地带有 **IVFPQ + HNSW 粗量化器**(用 HNSW 替代 IVF 的暴力质心搜索,FAISS 的 `IVF_HNSW`)、以及 **DiskANN** 把图压到 SSD 上、内存只放压缩码。但核心权衡不变:你在「内存」和「召回」之间选位置,IVFPQ 给你最省内存的那一端。

一句话记法,也是 stage 的结语:**`nprobe` 调召回(到天花板为止),`nsub` 调内存/召回平衡点,PQ 永远在残差上训练。**

## ⚡ 开放问题:更优的量化,目前无通用解

PQ 把每个子空间独立量化,这隐含一个假设:子空间之间不相关。真实 embedding 不满足——维度之间有相关性,独立量化会浪费比特。这个方向有一串改进,但**没有一个是「换上去就普遍更好」的通用解**:

- **OPQ**(optimized PQ)—— 量化前先学一个旋转矩阵,把数据旋到「子空间尽量不相关」的坐标系再切。几乎总能涨几个点,代价是多一步训练和一次矩阵乘。但旋转矩阵本身也是过拟合训练集的风险。
- **RaBitQ** 这类近两年的二值/低比特量化—— 用随机化 + 严格误差界把向量压到极低比特(甚至每维 1 比特),并给出可证明的距离估计误差界。在某些 benchmark 上以更低内存打平 PQ,但对数据分布和维度的假设、以及在真实 embedding 上的稳健性仍在被各家独立复现和质疑。
- **学习型索引**(learned index)—— 用神经网络直接学「查询 → 候选桶」的映射,绕开 k-means 分簇。论文里有亮眼结果,但泛化性、训练成本、分布漂移时的退化都没有定论,生产系统几乎没人敢把它当主力。

为什么没有通用解?因为「最优量化」本质是在「你的具体 embedding 分布」上找最省比特的编码,而 embedding 分布随模型、随领域、随时间漂移。任何在一个分布上调优的量化方案,换个分布都可能退化。这也是为什么 FAISS 到今天还把 PQ/OPQ 作为稳健默认,而不是直接上最新论文的方案——**在你不了解的数据上,可证明的稳健 baseline 胜过 benchmark 上更高但脆弱的数字**。这个领域正在动,谁要是告诉你某个量化方案「普遍最优」,值得怀疑。
