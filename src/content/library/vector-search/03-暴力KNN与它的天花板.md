先把结论钉死，省得你读完还在纠结：**暴力 KNN 不是一个"备选索引"，它是你这整本书里唯一不会骗你的那把尺子。** 它的 recall@k（召回率，找回的真邻居占应找回的比例）永远是 1.0，不是因为它聪明，而是因为它笨到把每个向量都算了一遍——真邻居无处可藏。代价是它的 QPS（每秒查询数）随数据量 N 线性崩塌。后面 IVF、PQ、HNSW 那些花活，本质都是在拿这把尺子量出来的 recall 损失去换速度。你要是连这把尺子都没立稳，后面任何"recall 0.94"的宣称都是悬空的——0.94 是相对谁的 0.94?

这章只干三件事：(1) 证明暴力法为什么是 recall 上限同时是速度下限；(2) 用实测曲线让你亲眼看 QPS 怎么随 N 崩、崩多快、外推到生产规模有多绝望；(3) 戳破"我数据不大，暴力够用"这个最常见的自我安慰,顺便揭一个静默错误模式。配套代码全在 `examples/vector-search-from-scratch/src/stage03-bruteforce.ts`,跑得动,数字都是真的。

## 暴力 KNN 是召回上限同时是速度下限

### 机制：它没有"索引"这个概念

暴力 KNN 的算法只有一句话:对每个查询,把它跟数据集里每一个向量都算一次距离,排序,取最近的 k 个。没有预处理、没有剪枝、没有数据结构。看 `stage03-bruteforce.ts` 里的核心函数:

```typescript
function bruteForceTopK(
  dataset: number[][],
  query: number[],
  k: number,
): number[] {
  const scored = dataset.map((vec, idx) => ({ idx, dist: l2dist(query, vec) }));
  scored.sort((a, b) => a.dist - b.dist || a.idx - b.idx);
  return scored.slice(0, k).map((e) => e.idx);
}
```

这就是全部。`l2dist` 是第 2 章那四个原语之一(`src/core/vec.ts`),在最内层每个向量调一次。注意 `a.dist - b.dist || a.idx - b.idx` 这个 tie-break(平局判定):距离相等时小 index 优先。这不是随手写的——它跟 ground truth 生成器 `computeGroundTruth`(`src/core/dataset.ts`)用的**完全同一个**平局规则:

```typescript
scored.sort((x, y) => (y.s - x.s) || (x.idx - y.idx));
```

为什么这一行值得专门讲?因为 ground truth(真值,正确答案)本身也是一次暴力扫描——本书的"正确"就是暴力扫描的定义。如果暴力查询和真值用了不同的平局规则,recall 会在浮点距离碰撞处莫名其妙地掉到 1.0 以下。那个掉,**不是真的漏召回,是测量假象**。而本章整个存在的意义就是给你一个 recall 恒等于 1.0 的基线。平局规则一旦不一致,基线就不再恒为 1.0,你后面拿它当尺子量别人,量的全是噪声。所以这一行的注释在代码里写得很重:tie-break 跟真值对齐是不变量,不是风格选择。

### 为什么 recall 必为 1.0:这是定义不是性质

暴力法扫了全量 N 个向量。top-k 的真邻居,按定义就是距离最近的 k 个,它们一定在这 N 个里,一定被算到了,一定不会被任何剪枝丢掉——因为根本没有剪枝。所以 recall@k 恒等于 1.0。这是 stage03 打印的第一组数(实测,dim=16, clusters=8, queries=50, k=10, metric=l2):

```
N	recall@10	batch(ms)	QPS		内存(est.)
1000	1.0000		10.1		4942		125.0 KB
5000	1.0000		59.6		839		625.0 KB
20000	1.0000		297.1		168		2.44 MB
50000	1.0000		843.4		59		6.10 MB
```

看 recall 那一列,1.0000、1.0000、1.0000、1.0000——N 涨了 50 倍,纹丝不动。这就是"召回上限"的含义:**没有任何索引能在同一度量下做得比 1.0 更好**,因为 1.0 已经是满分。近似索引能做的只有逼近它,逼近的代价是速度换来的——这正是后面几章的全部内容。

同时它是速度下限:看 QPS 那一列,4942 → 839 → 168 → 59,一路往下掉。没有比"全部算一遍"更慢的诚实做法了(你可以算两遍,但那是 bug 不是策略)。所以暴力法把可能的工程空间框死了——recall 顶在 1.0,QPS 蹲在地板。后续每个索引都是这个矩形里的一个点:牺牲一点 recall,往上爬一截 QPS。**没有基线,你不知道自己爬了多高,也不知道掉了多少。**

> 顺带交叉引用:第 4 章 HNSW、第 5 章 IVF-PQ 跑出来的每一个 recall 数字,recall 的分母(真值)都来自这里的 `computeGroundTruth`。它们是"被暴力法打的分"。这也是为什么 eval 阶段(stage07)即使慢得离谱也照样用暴力法算真值——尺子的正确性不可妥协,慢可以忍。

## 为什么生产撑不住:成本是 O(N·d),崩在钱不在召回

### 失败模式 (a):成本爆炸,而且崩得比线性还快

暴力法的失败模式很反直觉:**它从不漏召回,它的失败是付不起账。** 每个查询是 O(N·d) 次距离计算(N 个向量,每个 d 维全乘一遍)。N 翻倍,查询时间翻倍。stage03 不让你听我空口说"线性",它打印证据。如果真是 O(N),那么 QPS×N 应该近似常数:

```
--- 线性性验证: 若 O(N), 则 QPS×N 应近似常数 ---
N	QPS×N		相对 N=1000 (理想=1.00)
1000	4.94M		1.00
5000	4.20M		0.85
20000	3.37M		0.68
50000	2.96M		0.60
注: QPS 从 N=1000 到 N=50000 (50×) 的下降倍数 = 83.4× (理想线性应≈50×)
```

注意这里有个诚实的细节,别错过:QPS×N **没有**保持常数,而是从 1.00 滑到 0.60。N 涨 50 倍,QPS 却掉了 83.4 倍——**比线性还差**。为什么? 因为我这版实现为了让"一次距离、再取最小"这个机制可见,用的是全排序 O(N log N) 而不是 O(N log k) 的部分选择(quickselect / 有界堆),代码注释里点明了这是故意留在桌上的。再叠加大数组的 cache 不友好和 GC 抖动,实际曲线比纯线性更陡。

这恰恰是要你记住的第一课:**"它只是线性"已经是乐观说法了。** 真实系统里,常数因子和缓存行为会让"只是线性"变成"比线性更糟"。一个真正的引擎(FAISS 的 `IndexFlatL2`、hnswlib 的暴力 baseline)会用 SIMD、Float32、blocked 矩阵乘把常数因子压到极致,把这条曲线拉回贴近线性——但**斜率改不了**,它仍然是 O(N)。优化常数因子能让你多撑一个数量级,改不了"翻倍数据翻倍延迟"的命。

### 外推到生产:数秒一次查询

dim=16、N=50000 的玩具数据已经是每查询 16.867 ms、单线程 QPS≈59。把它外推到真实嵌入的规模(估算,线性外推每向量成本,代码里明确标了"估算"):

```
--- 失败模式 (a): 成本爆炸, 不是召回 ---
N=50000 时每查询 16.867 ms ⇒ 单线程 QPS≈59
外推 (估算, 线性): N=10M, dim=768 ⇒ 每查询 ≈ 161923 ms (≈161.9 s/查询), QPS≈0.01
  ⇒ 暴力法在生产规模下单查询要数秒。这就是 IVF/PQ/HNSW 存在的理由 (后续 stage)。
```

每查询 161.9 秒。这数字得拆开看,别被它吓到也别被它骗到。它是**纯 JS、单线程、`number[]` 不是 `Float32Array`** 的乐观曲线线性外推,绝对值不能信——FAISS brute-force 在一块好 CPU 上 10M×768 大概是几十到几百毫秒级,不是秒级。但**量级关系是对的**:从"毫秒"挪到"百毫秒到秒",单台机器单线程伺候不了任何在线 QPS 需求。这就是为什么没人在生产里裸跑暴力法做在线检索——pgvector 不加索引就是这副德行(它的 `<->` 顺序扫描本质是暴力 KNN),Lucene 的 `KnnFloatVectorQuery` 也提供 brute-force 的 `exact` 路径但只推荐在小候选集上用。

**这里就是要拆穿"数据不大可以暴力"的地方。** 这句话错在两点。第一,它把"数据不大"和"今天不大"混为一谈——N=1000 时 QPS 4942 很爽,等于在 N=5000 时已经跌到 839,你的"不大"撑不过一次业务增长。第二,它忽略了 d。生产嵌入是 768 / 1024 / 1536 维,不是这里的 16。距离成本正比于 d,你在 dim=16 上测出来的"够快",换到 dim=1536 直接 ×96。这两个乘数叠起来,"暴力够用"的安全区比你以为的小一两个数量级。**真正适合暴力法的,是 N×d 小到单查询稳定亚毫秒、且并发极低的场景**:几千条候选的重排序、一个用户的私有小知识库、或者就是拿来当你 ANN 索引的对照基线。除此之外别裸用。

## 失败模式 (b):扫了全量也能静默给错答案

暴力法 recall=1.0 这个保证有个致命的小字条款:**它只对"扫描时用的那个度量"成立。** 度量换了,索引不重建,recall 会暴跌,而且不报错——静默返回错答案。stage03 专门演示了这个:

```
--- 失败模式 (b): 度量不匹配 ⇒ 扫全量也错 (静默!) ---
同一份 l2 暴力结果: vs l2 真值 recall=1.0000 (匹配, 满分) | vs cosine 真值 recall=0.7480 (度量不匹配, 暴跌)
  ⇒ 暴力的 recall=1.0 只对它扫描所用度量成立; 换度量不重建索引会静默返回错答案。
```

同一份用 L2 距离暴力扫出来的结果,拿 L2 真值量是满分 1.0000,拿 cosine(余弦,看方向不看长度)真值量直接掉到 0.7480。代码做的事很简单:用 L2 扫,但用 cosine 真值对照(`computeGroundTruth(smallDs, smallQ, K, 'cosine')`)。这不是 contrived 的演示——这是真实团队的 bug:嵌入模型换了,推荐用 cosine,但检索代码还在用 L2;或者向量入库时没归一化,以为 dot product 等价于 cosine 结果两者发散。

为什么这个失败比"漏召回"更危险?因为漏召回至少 recall 数字会掉,你的监控能抓到。**度量不匹配是静默的**:没有异常、没有报错,top-k 照样返回 k 个结果,看起来一切正常,只是答案悄悄错了。第 2 章我们花整章把 L2、cosine、dot 的关系讲清楚,根因就在这——选错度量,后面所有索引(暴力的、近似的)都在错误的几何空间里找邻居,而且不会有任何东西提醒你。**connect 到第 2 章:度量是几何空间的定义,定义错了,"最近"这个词本身就错了。** 暴力法的诚实在这里反而成了陷阱:它太诚实地服从你给的度量,你给错了它就忠实地错下去。

## ⚡ 开放问题:精确 ANN 没有又快又省的下界

把暴力法的天花板讲完,自然冒出一个问题:能不能既保持 recall=1.0(精确),又把查询做到 sub-linear(亚线性,比 O(N) 快)? 低维可以——k-d 树、ball tree 在 d≤20 左右能做到接近 O(log N) 的精确查询。但**到了生产的高维(d 几百上千),没有通用解**。这是"维度灾难"(curse of dimensionality)的硬后果:高维空间里所有点的距离趋于相等,任何基于划分的精确数据结构,要么退化到几乎遍历全部节点(等于暴力),要么内存/预处理代价爆炸到不可接受。理论上,精确高维最近邻在最坏情况下的查询下界至今没有令人满意的亚线性算法——这是计算几何里**仍在研究、没有通用解**的开放问题。

所以实践里没人硬刚"精确 + 高维 + 快"这个不可能三角,大家直接放弃"精确",转向 ANN(approximate nearest neighbor,近似最近邻):接受 recall < 1.0,换 sub-linear 查询。这就是第 4 章 HNSW、第 5 章 IVF-PQ 的全部出发点。DiskANN、Milvus 这些生产系统也都站在这条线上——它们竞争的不是"谁精确",而是"同样的 recall 下谁更快更省、同样的资源下谁 recall 更高"。**记住这个 framing:从下一章起,我们讨论的不再是"对不对",而是"在 recall-速度-内存这个三角里,你愿意放弃哪个角"。** 暴力法是这个三角的一个顶点(recall 满、另外俩塌),后面每个索引是三角内部的一个可调点。

> 与隔壁书的分工:RAG 工程师那条线讲的是检索管道——怎么切块、怎么 rerank、怎么塞进 prompt;这本讲的是管道里那个"检索引擎"内核本身怎么造。agent 书第 7 章 agentic retrieval 讲的是"agent 何时该去检索";这章和后续讲的是"那台被调用的引擎里面到底装了什么"。三条线不重叠:何时检索、检索什么管道、引擎内核怎么实现。

## 小结:基线立住了

```
=== 基线就绪: recall 上限=1.0, 速度随 N 线性下降, 后续近似法以此为对照 ===
```

带走三句话:(1) 暴力 KNN 的 recall=1.0 是定义不是优化,它同时框死了 recall 上限和速度下限,是后续一切近似法唯一可信的对照尺;(2) 它的失败模式是成本不是召回——而且实测比线性更陡,"数据不大可以暴力"在 N 增长 + 高维这两个乘数下比你以为的危险得多;(3) recall=1.0 只对扫描所用度量成立,换度量不重建索引会静默给错答案,根因在第 2 章的度量选择。下一章我们造第一个真正的近似索引 HNSW,你会第一次看到 recall 从 1.0 掉下来——而正因为有这章这把尺子,那个"掉"才是可量化、可接受、可决策的。
