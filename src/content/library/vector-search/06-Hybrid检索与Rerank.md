先看一组数字，本章配套代码 `examples/vector-search-from-scratch/src/stage06-hybrid.ts` 在 2000 向量、50 查询、recall@10 下的真实输出：

```
BM25 单路 (关键词)        recall@10 = 50.2%
向量 单路 (余弦语义)      recall@10 = 10.8%
Hybrid RRF (倒数排名融合) recall@10 = 50.2%
Hybrid 加权 (归一化求和)  recall@10 = 67.8%
Hybrid RRF + Rerank       recall@10 = 95.0%
```

两路单独都很弱——最好的 BM25 也才 50.2%，向量路只有 10.8%。但把它们拼起来再精排，recall 干到 95.0%。这不是某个魔法参数调出来的，是 hybrid 检索的结构性收益：两路漏的东西不一样，融合把对方漏的捞回来。这一章讲清楚为什么会这样、什么时候不这样、以及你第一次做 hybrid 几乎必然会撞上的那堵墙。

> 一句话立场：生产检索系统几乎没有只信一路的。单路 ANN（approximate nearest neighbor，近似最近邻）召回看着够用，是因为你还没遇到它系统性漏召的那类 query。FAISS / hnswlib 给你的是单路引擎，pgvector / Milvus / Lucene 上层叠 hybrid 不是锦上添花，是补单路的结构性盲区。

## 为什么单路必然有盲区

先说清楚这章的 recall 是对着什么标尺量的——这决定了所有数字的可信度。

向量检索（余弦相似度，衡量两个向量方向有多接近）擅长语义：query 说"狗"，文档写"犬""puppy"，向量空间里它们挨得近。但它对训练时没见过的稀有专有名词、ID、数字、错别字会漂——那些 token 没有可靠的 embedding，方向乱指。

BM25（关键词检索，按词频和稀有度打分）正相反：精确 term 命中是它的主场，但它对同义和改写完全瞎——"狗"和"犬"在 BM25 眼里是两个毫不相干的 token，一个字符不差才算匹配。

所以**相关性本身就是个混合量**：用户想要的结果，既要语义近、又要关键词在点上。本章的 ground truth 就直接把这个前提写进标尺——`computeHybridTruth` 用 `0.6·余弦 + 0.4·词法` 的混合分作真值，而不是纯几何 L2 距离：

```ts
// stage06-hybrid.ts: computeHybridTruth
const wCos = 0.6;
const wLex = 0.4;
// ...
const cos = Math.max(0, cosineSim(qVec, dVec));
const lex = lexicalOverlap(qTokens, docs[id]);
return { id, s: wCos * cos + wLex * lex };
```

这个选择是诚实性的关键，得展开说一句。如果拿纯几何距离当真值，向量路天然全赢，"hybrid 有用"就成了自欺。混合真值意味着**每条单路天生只覆盖真值的一个轴**：BM25 漏掉语义轴，向量漏掉词法轴。所以你看到向量单路只有 10.8% 不要慌——不是向量检索废,是这套真值的词法权重（0.4）足够重,纯语义覆盖不全;换个偏语义的真值数字会翻上去。重点不是某条单路的绝对值,是**两路的盲区不重叠**,这才是融合能赚钱的前提。

⚙ 数据是怎么来的也得标清楚：核心数据集只有几何高斯聚类，没有文本可以跑 BM25。所以代码用 `makeDoc` 把每个向量的坐标量化成一袋 token（词表 40 词、每篇约 8 词），让词法信号成为几何信号的有损投影——同簇向量通常共享 token（词法和几何一致），但量化边界切开簇，部分近邻落到不同 bin（词法和几何分歧）。这个分歧就是现实里同义词/错别字缺口的合成版本。**这是合成文本层，不是真语料**；真实系统里两路的分歧来自语言本身，但结构是一样的。

## RRF：扔掉分数，只信排名

融合的第一个流派叫 RRF（Reciprocal Rank Fusion，倒数排名融合）。它的核心动作是**把两路的原始分数全扔掉，只用排名**：

```ts
// stage06-hybrid.ts: fuseRrf
for (const list of lists) {
  for (let rank = 0; rank < list.length; rank++) {
    const id = list[rank].id;
    fused.set(id, (fused.get(id) ?? 0) + 1 / (rrfK + rank + 1));
  }
}
```

每个文档的 RRF 分 = 各路 `1/(rrfK + 排名)` 之和。为什么这是个聪明的默认值：BM25 的分数能到 9.8，余弦最多 1.0，两者数量级差一个十倍——但**排名永远是 1、2、3…，天然同尺度**。谁都没法靠"我的数字大"压过对方。`rrfK`（默认 60，Cormack 等人的原值）的作用是压低头部排名的统治力，让单一一路无法独断冠军。

代价是 RRF **把分数信息全丢了**。看本章数字：

```
BM25 单路        recall@10 = 50.2%
Hybrid RRF       recall@10 = 50.2%
```

RRF 在这个数据上恰好等于 BM25 单路，一点没涨。这不是 bug——当一路（BM25）的排名质量明显碾压另一路（向量 10.8%），RRF 只看排名、无法表达"BM25 这一票该比向量重得多"，于是弱路的低质排名稀释了强路。**RRF 的稳健是有代价的：两路质量悬殊时，它的不偏不倚反而拖后腿。** 这就是 RRF 的失败模式——它假设两路大致同等可信，这个假设一破，它就退化。

那为什么生产里 RRF 还是最常见的默认（Elasticsearch、Weaviate 的 hybrid 默认走它）？因为它**不用调参、不用对齐尺度、换模型不用重标定**。当你两路质量接近、又懒得维护一套归一化标定时，RRF 是无脑能用的保底。它不是最优，是最省心。

## 归一化加权：把分数对齐再相加

第二个流派信任原始分数，但先把两路压到同一尺度再加权。关键是 min-max 归一化（把一组分数线性压到 [0,1]）：

```ts
// stage06-hybrid.ts: minMaxNormalize
const lo = Math.min(...scores);
const hi = Math.max(...scores);
const range = hi - lo;
// range===0 (全相等) ⇒ 映射到 0，不是 NaN：平坦列表不携带信号
out.set(e.id, range === 0 ? 0 : (e.score - lo) / range);
```

归一化之后，两路都在 [0,1]，再按权重相加（本章用 `0.4·BM25 + 0.6·向量`，对齐真值的语义倾向）。效果：

```
Hybrid RRF (倒数排名融合) recall@10 = 50.2%
Hybrid 加权 (归一化求和)  recall@10 = 67.8%
```

加权融合 67.8%，比 RRF 高 17.6 个百分点。因为它保留了"这个文档余弦 0.9 比那个 0.5 强多少"的连续信息，RRF 把这些全压成排名序号丢了。**当你信任两路的分数标定、且愿意维护一套归一化逻辑，加权胜过 RRF。** 但它的脆弱性也正在这里——归一化是承重墙，抽掉就塌，下一节就是那个塌法。

注意一个实现细节：min-max 是在**实际返回的候选池**上做的，不是全库。因为你要决定顺序的就是这批候选，拿全库的极值归一化会让候选池内部的分差被压扁。

## 失败模式：两路尺度不一致直接相加

这是 hybrid 检索最经典、也最隐蔽的坑——隐蔽在它"看起来在工作"。你写了融合，跑通了，recall 也没崩，于是以为没问题。实际上一路已经把另一路彻底吃掉，融合形同虚设。

代码里的失败路径就是把归一化关掉、两路裸分直接相加：

```ts
// stage06-hybrid.ts: 失败模式路径
const naive = fuseWeighted(bmList, vecList, 1.0, 1.0, false);
```

本章输出把病根量化得很清楚：

```
原始分数量级 (query#0): BM25 max=9.867  cosine max=0.821
→ BM25 量级约为 cosine 的 12.0x; 直接相加, cosine 被淹没
裸相加 (未归一化) recall@10 = 53.8%
归一化加权        recall@10 = 67.8%
裸相加 top-10 与 BM25 单路重合度 = 89.2%
```

BM25 最大分 9.867，余弦最大分 0.821，量级差 12 倍。`total = bm25 + cosine` 这个式子里，余弦那 0.8 的浮动相对 BM25 的 9.8 跳动几乎是噪声——**融合结果就是 BM25 戴了顶帽子**。那条 89.2% 的重合度是诊断这个病的关键指标：裸相加的 top-10 有 89.2% 和 BM25 单路一模一样，说明余弦信号被吃得只剩 11%。recall 也从归一化的 67.8% 掉到 53.8%。

为什么这个坑这么常见？因为它**不报错**。代码跑通、有结果、recall 看着也还行（53.8% 比向量单路的 10.8% 高多了），review 时很容易过。你得专门去量"融合结果和强路的重合度"才能发现弱路被吞了。这是个语义 bug 不是语法 bug——类型系统、单测都抓不到，因为每一步计算都"正确"，错的是两个不同量纲的数被当同类相加这个建模决策。

排查清单（撞上"hybrid 没涨/约等于强路单路"时）：

1. **量级体检**——打印两路的 `max` 分，差 ≥ 一个数量级就是嫌疑。
2. **重合度体检**——算融合 top-K 和各单路 top-K 的重合度，某一路逼近 100% = 它把融合独占了。
3. 确诊后二选一：要么归一化再加权，要么直接换 RRF（RRF 只看排名，天生免疫尺度问题——这也是它存在的理由）。

## Rerank：在短名单上重算真相

融合解决"候选齐不齐"，rerank 解决"顺序对不对"。最后那跳——95.0%——是它干的：

```
Hybrid RRF (倒数排名融合) recall@10 = 50.2%
Hybrid RRF + Rerank       recall@10 = 95.0%
```

注意 rerank 的输入是 RRF 的 top-N（代码里 `FUSE_TOP_N = 50`）。RRF 自己只有 50.2%，但它在 top-50 里**把对的候选都捞进来了**——只是排序粗。rerank 在这 50 个候选上重算精确的混合相关性，把顺序修对，于是 top-10 的 recall 飙到 95.0%。这就是 rerank 的分工：**融合负责招够人，rerank 负责排对座次。**

> 关键认知：rerank 救不了召回阶段就漏掉的文档。它只能在候选名单里重排——名单里没有的，它变不出来。所以 hybrid 召回的覆盖率是 rerank 收益的天花板。本章 RRF 的 top-50 覆盖率足够高，rerank 才能榨到 95%；如果召回阶段就把对的文档漏在 50 名之外，rerank 再强也是 0。

真实系统里 rerank 用的是 cross-encoder（交叉编码器，把 query 和文档拼在一起喂进 transformer 打一个相关分）。它比双塔 embedding 准，因为能看 query 和文档的逐词交互；但慢得多，所以只能在融合后的短名单上跑，绝不上全库——这是 O(N) 级的昂贵操作。

**本章的 rerank 是 toy，必须标清楚。** 代码里它不是模型、不调 API，是一个确定性打分函数，复用了和真值同款的 0.6/0.4 混合：

```ts
// stage06-hybrid.ts: rerank — 玩具 cross-encoder，模拟角色非精度
const cos = Math.max(0, cosineSim(queryUnit, datasetUnits[id]));
const lex = lexicalOverlap(queryTokens, docs[id]);
return { id, score: 0.6 * cos + 0.4 * lex };
```

它模拟的是 cross-encoder 的**角色**（在短候选上重算完整相关性），不是它的精度。这里 95% 这么漂亮，部分原因是 toy reranker 用了和真值完全相同的混合公式——等于让 reranker 偷看了答案。真实 cross-encoder 是学出来的近似，不会这么贴。**所以请把 95% 读作"rerank 这个阶段能修多少顺序"的上界示意，不是某个真实模型的实测精度。** 这个数字诚实，但它诚实地标着"我是上界"。

cross-encoder 自己的失败模式也得记一笔：它对长文档会因截断而漏关键段（输入长度有限），对训练分布外的领域会高估相关性。生产里常见的折中是 reranker 只看每篇的代表段，而不是全文——这又把"选哪段给 reranker 看"变成新的工程问题。

## 一个没有通用解的问题 ⚡

到这里你能搭出一条像样的 hybrid 管线了：两路召回 → 融合（RRF 或归一化加权）→ rerank。但每一环都有手调的旋钮——融合权重 0.6/0.4、`rrfK=60`、候选数 `FUSE_TOP_N=50`、reranker 的截断策略。本章为了隔离变量，把权重直接设成了已知真值的混合比；真实系统里**这些参数互相耦合，没有解析最优解，只能在标注集上调**。

更前沿的开放问题是：**能不能把召回、融合、rerank 端到端联合训练，让一个模型学会这套配合，而不是人肉拼三个独立阶段、各调各的旋钮？** 学习排序（learning-to-rank，用标注的相关性数据训练排序模型）在工业界（搜索引擎）成熟很久了，但那是在**给定特征**上学权重。把检索器、融合器、重排器**联合**训练——让召回阶段知道下游 rerank 的偏好、让融合权重随 query 自适应——目前**没有通用解，仍在活跃研究**。难点是几个：召回的 top-K 截断不可导（梯度传不回去）；标注成本高（联合训练要的相关性标签量级远超单阶段）；线上分布漂移让离线学的权重失效。今天能上生产的端到端方案（如学习式融合权重、可微检索 ColBERT 系）都还是窄场景特调，没有一个能像"BM25 + RRF + cross-encoder"这条手搭管线那样跨领域无脑能用。**所以现状是：手搭三段式仍是稳健默认，端到端是研究热点而非生产标配。**
