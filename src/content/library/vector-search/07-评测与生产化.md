前六章每一章结尾都给了你一个 recall 数字：暴力扫描 recall@10 = 1.0，HNSW 调到 efSearch=128 是 0.98，IVF-PQ 压到 1/16 内存掉到 0.91。这些数字都是在一个冻结的、单机的、无过滤的数据集上跑出来的。问题是：**冻结数据集上的单个 recall 数字，是这世界上最容易骗自己的数字。** 生产环境会把让这个数字有意义的每一条假设逐个打破——语料大到单机放不下（你得分片），用户要的是"30 块以下还有货的红鞋"而不是"最近的向量"（你得过滤），语料天天在变（你得增删)。这三件事每一件都有一个所有人都会先写错的版本，而错的版本不会报错，只会悄悄少召回。

这章对应配套仓的 `examples/vector-search-from-scratch/src/stage07-eval.ts`。它故意用一个**精确（brute-force，暴力扫描）索引**而不是前几章的近似索引——因为分片合并、过滤、tombstone 三件事的正确性，和 ANN（近似最近邻）的近似误差是**正交**的两回事。用精确索引意味着这章里任何 recall < 1.0 都只可能是分片/过滤/更新逻辑写错了，不是 ANN 剪枝丢的。把这两个误差源隔离开，是这章实验设计的全部用意。固定 fixture：2000 条 16 维向量、8 个高斯簇、50 条查询、L2 度量、k=10，种子写死（data=7, query=99），所以同一份代码在任何机器上跑出的数字都一样。

## 评测：recall@k 和 nDCG@k，以及为什么 recall 不等于好

先把度量算清楚。**recall@k（召回率）**：你返回的 top-k 里，有几个命中了"真正的 top-k"。配套仓 `core/metrics.ts::recallAtK` 的核心就三行：

```ts
const truth = truthIdsPerQuery[q].slice(0, k);
const approx = new Set(approxIdsPerQuery[q].slice(0, k));
let hit = 0;
for (const t of truth) if (approx.has(t)) hit++;
totalRecall += hit / truth.length; // per-query denom = its own truth size
```

注意它是**集合命中**：把真值前 k 个塞进 Set，数你返回的前 k 个里有几个在 Set 里，除以 k，再对所有查询取平均。真值哪来的？`core/dataset.ts::computeGroundTruth` 用暴力扫描算出每条查询的精确最近邻——这是这套评测能成立的地基，下面会讲为什么这地基本身必须先校准。

recall@k 有个致命盲区：**它把 k 个真邻居当成等价的，完全不管顺序。** 你把真正的第 1 名排到第 10 位，和把它排在第 1 位，recall@10 一模一样，都是命中了。但对一个不做重排（reranking）的检索系统，索引返回的顺序**就是用户看到的顺序**——第 1 名沉到第 10 位是实打实的体验损失，而 recall 这把尺子根本量不出来。

这就是为什么要 **nDCG@k（归一化折损累计增益，normalized discounted cumulative gain）**。它问的是更尖的问题："最好的那几个邻居，有没有出现在**靠前**的位置？"`stage07-eval.ts` 里 `ndcgAtK` 自己实现了一版（`core/metrics` 不提供），关键是两点：

```ts
// 按真值里的排名给增益：rank 0（真正最近）增益最高
const gain = (rank) => rank === undefined ? 0 : truthLen - rank;
// log2(i+2)：标准 DCG 折损，位置 0 的折损是 log2(2)=1
dcg += gain(truthRank.get(topk[i])) / Math.log2(i + 2);
```

增益按真值排名递减（真·第 1 名值 truthLen 分，非邻居 0 分），再除以一个随位置对数增长的折损因子——排得越靠后，同样的增益打折越狠。除以 IDCG（理想排序的 DCG）归一化到 [0,1]，1.0 意味着"按真值的精确顺序返回了真值 top-k"。

**那为什么实测两个都是 1.0？** stage07 实验 [0] 的输出：

```
recall@10 = 1.0000 | nDCG@10 = 1.0000 | QPS = 2267 (22.1 ms / 50 q)
dataset memory = 250.0 KB (est. payload floor)
check: exact index recall == 1.0000 -> harness is calibrated
```

因为这里跑的是**精确索引**，它当然把真 top-k 按真顺序全返回了，两个度量都满分。这不是用来夸索引好的——这是**校准**。代码里有一行硬断言：精确索引 recall 必须等于 1.0，否则 `throw new Error('harness bug')`。逻辑是：如果连暴力扫描这种不可能错的索引都被你的评测算出 recall < 1.0，那是**你的评测自己写错了**——真值生成、k 截断、还是命中判断有 bug——这种情况下下游所有 ANN 的 recall 数字全是垃圾。**先校准评测本身，再用它去量近似索引**，顺序不能反。这是 [《向量检索的物理学》](/library/vector-search/02) 讲度量、第三到六章报 recall 时都默默依赖、却很少有人显式做的一步。

`dataset memory = 250.0 KB` 标了 `(est. payload floor)`——这是估算，不是实测 RSS。`core/metrics.ts::estimateBytes` 注释说得很直白：JS 对象开销让真实堆占用没有 profiler 就不可知，它只算 payload 地板（每个 number 8 字节 IEEE-754 double，n·dim·8），真实 V8 堆通常是这数的 1.5–3 倍。**凡是估算就标估算**，别让读者把它当成测出来的内存。QPS = 2267 同理：单机单线程墙钟时间，且跑前先做了一次 warmup pass 让 V8 JIT 预热——不预热的话第一次跑会被 JIT/GC 拖慢，QPS 虚低。

> **工程推论（可当面试题）**：你接手一个向量检索服务，前任留下 "recall@10 = 0.95" 的验收报告。你第一件该核的事不是这个 0.95 高不高，而是"真值怎么算的、评测在精确索引上是不是 1.0"。真值若是用另一个近似索引当 ground truth 算的，那 0.95 是两个近似误差的卷积，毫无意义。

## 生产关切一：分片 + 合并，错的合并会悄悄丢掉全局 top-k

语料大到单机放不下，就分片（sharding）：切成 S 份，每份一个独立索引，查询打到所有分片，再把结果**合并**。合并这一步是头号坑。

正确的合并：每个分片返回它**本地的 top-k**，把这 S×k 个候选**按真距离重排**，取全局 top-k。stage07 实验 [1] 用 4 个分片、轮询（round-robin）分配（不是按簇连续切，故意让每个分片混着各簇的数据，更接近真实），正确合并的代码：

```ts
const mergedCorrect = queries.map((q) => {
  const candidateIds = shards.flatMap((sh) => sh.search(q, K)); // 每片要满 K
  return rerankByDistance(q, candidateIds, K);                  // 合并后重排取全局 K
});
```

诱人但错的捷径：为了"省点活"，让每个分片只返回 top-(k/S) = ceil(10/4) = 3 个，"反正 4 片凑起来也有 12 个候选"。实测结果：

```
correct merge (each shard top-10)  recall@10 = 1.0000
WRONG  merge (each shard top-3)   recall@10 = 0.4520  <- failure mode: k/S under-provisioning drops global top-k
```

**recall 直接腰斩到 0.45。** 为什么？因为这个 k/S 的算术假设了"每条查询的真邻居在 S 个分片间均匀分布"——每片摊 1/S。但数据不是均匀的：一条查询离簇 c 很近，它的真邻居**几乎全在拥有簇 c 的那个分片里**。那个分片的本地 top-3 远不够装下这条查询的全部真邻居，多出来的就被你提前砍掉了，合并时再怎么重排也救不回来——它压根没进候选集。这是分片合并的**失败模式 #1：每片配额不足，悄悄丢掉全局 top-k**。FAISS 的 `IndexShards`、Milvus 的 segment 合并、Elasticsearch 跨 shard 的 `top_hits` 都踩过这个形状的坑，ES 文档里专门有一段讲分布式打分的近似性。

那正确合并要每片都返回满 K，岂不是浪费?对，候选集是 S×K 而不是 K,这是分片必然付出的代价——**分片省的是单机内存和单片扫描量，不是合并候选量。** 想进一步省，得靠"提前判断哪些分片不可能有结果"（比如分片带元数据范围、或 IVF 那种粗量化先定位），而不是靠砍每片配额。

代码里还有一条比 recall 更强的检查——**一致性不变量**：

```
consistency: merged top-10 vs single-index top-10 -> IDENTICAL on all 50 queries
```

正确的"分片+合并"作用在精确索引上，必须和"根本不分片"的单索引结果**逐查询、逐 id 字节级完全一致**。不是"recall 高就行"，是"和不分片一模一样"。代码里若发现任何一条查询发散就 `throw`。这是分片正确性的金标准：分片是个纯粹的工程切分手段，它**不应该改变答案**，只应该改变答案是怎么算出来的。任何"分片后 recall 掉了一点点但还行"的说法,都是 bug 在伪装成噪声。

## 生产关切二：带过滤的检索,pre-filter 和 post-filter 顺序搞反就召回不足

用户要的从来不是"最近的向量",是"最近的、**且**还有货、**且**30 块以下的向量"。这就是 **filtered-ANN（带属性过滤的近似检索）**:向量相似度 + 结构化谓词,两个条件一起满足。配套仓给每条向量挂了一份平行的 `meta`(price / inStock),谓词是 `inStock && price < 30`。

实验 [2] 先报了谓词的选择性:

```
predicate "inStock && price<30" matches 480/2000 items (24.0% — selective)
before filter: avg 2.12/10 of plain top-10 happen to match
```

只有 24% 的数据满足谓词——**选择性强**。而且不带过滤的纯向量 top-10 里,平均只有 2.12 个碰巧满足谓词。记住 2.12 这个数,它就是下面那个坑的坑底。

两种朴素策略:

**pre-filter(先过滤)**:扫描时就把谓词算进去,只对幸存者排距离。stage07 的 `FlatIndex.search` 在内循环里干这事:

```ts
if (opts.filter && opts.meta && !opts.filter(opts.meta[id])) continue;
scored.push({ id, dist: l2dist(query, this.vectors[row]) });
```

**post-filter(后过滤)**:先按向量取 top-k,**再**把不满足谓词的扔掉。实测对比:

```
pre-filter  : avg 10.00/10 returned, all match predicate = true
post-filter : avg 2.12/10 returned <- failure mode: 50/50 queries under-fill (got < 10)
```

pre-filter 稳稳返回 10 个、全部满足谓词。post-filter 平均只返回 **2.12** 个——和上面"纯 top-10 里碰巧满足的数量"完全吻合,因为它就是先拿那 10 个再筛。**50 条查询全部欠填(返回数 < 10)**。这是失败模式 #2:**用户要 10 双红鞋,系统给了 2 双。** 谓词越选择性强,后过滤欠填越狠——极端情况下纯 top-k 里一个都不满足,返回空。

那 pre-filter 是不是就赢了?在这个**精确索引**上是。但这正是这个开放问题的引子——

### ⚡ filtered-ANN 是开放难题:目前无通用最优解

把上面的结论搬到**真正的 ANN 索引**(比如第四章的 HNSW)上,pre-filter 的代价就暴露了:**ANN 索引的剪枝建立在"邻居的邻居也是邻居"的图结构/簇结构上,而过滤会把图打出窟窿。** 你在 HNSW 上做 pre-filter,等于一边走图一边扔掉不满足谓词的节点——但被扔的节点可能是通往某个满足谓词的远处幸存者的**唯一桥梁**,把它跳过,那片满足谓词的区域就再也走不到了,recall 崩塌。IVF 里更直接:你不敢跳过任何一个簇,因为不知道哪个簇里藏着幸存者,于是退化成近乎全扫。stage07 用精确索引,没有图也没有簇可破坏,所以 pre-filter 在这里零代价——**这个"零代价"是精确索引的特权,不要外推到 ANN。**

这就是为什么 filtered-ANN **至今没有通用最优解**,是工业界和学界都在啃的活课题:

- **pre-filter** 在低选择性(谓词命中率高)时好,高选择性时把 ANN 拖成全扫;
- **post-filter** 在高选择性时欠填,得靠"超采"(先取 top-10k 再筛)硬怼,但超采多少没有先验;
- **inline/混合**:把谓词编进索引结构。Milvus 的分区(partition)、Weaviate 的 filtered HNSW、pgvector 配 B-tree 的迭代扫描、Qdrant 的 filterable HNSW(在图里维护额外的"满足谓词子图"连通性)各有各的打法,**没有一个在所有选择性区间都赢**。ACORN、Filtered-DiskANN 这类 2023–2024 的工作就是在攻"任意谓词 + 任意选择性"下还能保持 ANN 复杂度,目前仍是 trade-off 而非银弹。

> **工程推论(可当面试题)**:线上反馈"加了价格筛选后搜索经常返回不满 10 条"。你脑子里第一个假设应该是 post-filter + 选择性强,而不是"召回模型不行"。验证:把谓词的命中率打印出来,再看是先筛还是后筛。命中率低于 ~20% 且用后过滤,基本实锤。

## 生产关切三:增量更新,忘了查 tombstone 会让删掉的文档诈尸

语料天天变,加文档、删文档,都不能每次重建整个索引。stage07 的 `FlatIndex` 用最朴素的两招:add 直接 append,delete 不真删而是打 **tombstone(墓碑标记)**——把 id 丢进一个 `deleted` 集合,查询时跳过。为什么不真删?注释讲得清楚:真删要压缩数组,会让每个 id→row 的映射全部失效,且是 O(n);tombstone 删除是 O(1),代价推迟到查询时跳过那一下。HNSW / IVF / Lucene 删除全是这个套路——**删除即标记,查询即跳过,空间靠后台 compaction 回收。**

实验 [3] 先验证 add/delete 的正确性:

```
before add: q0 top-10 = [145, 1569, 1665, 33, 1713, ...]
after add (id 2000): appears at rank 0 (correct: it is the nearest)
after delete (id 2000): present = false (correct: tombstone skipped it)
```

加一个特意构造成 q0 最近邻的向量(q0 自己加 1e-6 扰动,距离≈0),它必须排第 0——没重建,纯 append 就生效了。删掉它,它必须消失。到这都对。然后是这章最该刻进肌肉记忆的失败模式 #3:

```
failure mode (tombstone check disabled): deleted id 2000 RESURRECTS at rank 0 <- this is why every search path MUST honor the tombstone set
post-delete top-10 == original top-10: true (delete left the rest of the index intact)
```

`FlatIndex.search` 有个 `respectTombstones` 开关,默认 true,代码里只有为了演示这个坑才传 false:

```ts
const respectTombstones = opts.respectTombstones ?? true;
for (let row = 0; row < this.vectors.length; row++) {
  const id = this.ids[row];
  if (respectTombstones && this.deleted.has(id)) continue; // 漏了这行 = 诈尸
  ...
}
```

漏掉这一句 `continue`,删掉的 id 2000 就在 top-0 **诈尸**了。这是增量索引最常见的 bug,而且**特别阴险**:它不会在"删除"那条代码路径上出错——删除明明成功了,`deleted` 集合里有它。它会在**某一条忘了查 tombstone 的查询路径**上出错。生产里你往往有不止一条检索路径(主搜、推荐、相似召回、批量导出),只要有**一条**忘了 consult tombstone 集合,删掉的文档就从那条路径漏出来。所以注释那句话是规范级的:**every search path MUST honor the tombstone set**——不是"搜索函数要查 tombstone",是"每一条会返回结果的路径都要查"。

最后一行 `post-delete top-10 == original top-10: true` 是个收尾不变量:删掉那个诈尸者之后,剩下的 top-10 必须和加它之前**完全一致**——证明这次增删只动了目标 id,没有顺带破坏索引其他部分。增量操作的正确性不只是"目标对了",还包括"没误伤别人"。

> **工程推论(可当面试题)**:为什么向量库的删除大多是"软删 + 后台 compaction"而不是立即物理删?答:物理删要重排 id→向量的映射,O(n) 且会让所有持有旧 id 的引用(包括正在跑的查询)失效;tombstone 是 O(1),把代价摊到查询时一次集合查找,再由低峰期的 compaction 批量回收空间。代价是:tombstone 堆积会让查询扫到越来越多"已删但没回收"的死节点,recall 不变但 QPS 下降,所以 compaction 触发阈值(死节点占比)是个要调的运维参数。

## ⚡ 第二个开放问题:十亿级磁盘索引(DiskANN)

这章的 fixture 是 2000 条 × 16 维,250 KB,整个躺在内存里。把规模拉到**十亿条 × 768 维**,光原始向量就是 ~3 TB,HNSW 那种"图全放内存、随机访问邻居"的假设直接破产——没有哪台单机有 3 TB 内存,而把图放 SSD 上,HNSW 每跳一次邻居就是一次随机磁盘 IO,十亿规模下一条查询要几百跳,延迟爆炸。

**DiskANN(微软的磁盘原生 ANN 索引)** 是目前这个方向最有代表性的答案,核心思路是为"内存装不下、随机磁盘 IO 是瓶颈"这个新约束**重新设计图结构**:用一种长边更多、直径更短的图(Vamana),让查询跳数尽量少;向量本体放 SSD,内存里只留 PQ(乘积量化,见 [《IVF 与乘积量化》](/library/vector-search/05))压缩后的码用来粗筛,真正要算精确距离时才按 id 去 SSD 取那一条完整向量。这样把磁盘 IO 从"每跳一次"降到"每条查询几次"。

但这仍然是**开放问题、正在研究**,不是已解决:

- **增删 + 磁盘图**:本章讲的 tombstone 增量更新,放到磁盘图上更难——图的边要改,而边在 SSD 上,原地改图就是随机写,SSD 最怕随机写。FreshDiskANN 这类工作就是在攻"磁盘图怎么支持高频增删而不重建",目前是分层(内存层吸收新写、定期 merge 进磁盘层)的折中,merge 的时机和代价仍无定论。
- **filtered-ANN × DiskANN**:上面那个开放问题和这个开放问题**叠加**——带过滤的十亿级磁盘检索,是两个未解难题的乘积,Filtered-DiskANN 是起步,远未通用。
- **SSD 寿命与成本**:压缩多狠(PQ 码多短)直接决定内存够不够装下粗筛层,压太狠粗筛不准、回 SSD 的次数暴涨;这条 trade-off 曲线随硬件(QLC SSD、CXL 内存)一直在变,没有稳定最优点。

记住这章开头那句锚点:**向量检索之于 RAG = 索引之于数据库**。数据库的索引也经历过"内存放得下→放不下要上磁盘 B+树→十亿行要分布式"的同一条路,每一级新约束都逼出新结构。向量索引现在正走到"放不下要上磁盘"这一级,DiskANN 就是这一级的 B+树——但它远没到 B+树那样几十年沉淀下来、闭着眼用的成熟度。

## 收束:这本书到底教了你什么

七章走下来,这本《向量检索引擎》的内核就讲完了。把它和书库另外两条线划清楚:RAG 工程师那条线讲的是**检索管道**(chunk、rerank、把检索结果喂给 LLM),你可以从 [ai-app-engineering 的 RAG 流水线](/library/ai-app-engineering/08) 进那条线;agent 书第 7 章讲的是 **agentic retrieval——"何时该去检索"**;而你刚读完的这本,讲的是这一切底下那个被当成黑箱的东西——**引擎本身怎么造**。

退一步看全书的主线:从 [BM25 的关键词检索](/library/vector-search/01) 起步,到向量相似度的物理学,到暴力扫描的精确基线,到 HNSW 用图换速度、IVF-PQ 用量化换内存、混合检索用 RRF 融合两条召回——每一章都在同一个四角拉扯里做取舍:**精度 vs 速度 vs 内存 vs 构建成本,没有银弹。** 而这最后一章告诉你:就算你把这四角调得再漂亮,只要你不会**诚实地评测**它(先校准评测、recall 之外还看 nDCG),不会处理**生产的三个真相**(分片要满 K 合并、过滤要选对 pre/post、删除要每条路径查 tombstone),它在 demo 之外就活不下来。

而且这章特意留了两个**没有通用解**的口子——filtered-ANN 和十亿级磁盘索引——是想让你带着这个认知离开:向量检索不是一个"调个库就完事"的已解决问题,它的内核里还有正在被研究、值得你下场的硬骨头。这正是这本书想做的——AI 批量生成的 survey 稿会告诉你"有 HNSW、IVF、DiskANN 这些技术",但不会告诉你它们各自**怎么坏、何时坏、代价是什么、哪里还没解**。后者才是工程师吃饭的本事。

### 几道收尾面试题(答案都在前文)

1. 拿到一份 "recall@10 = 0.95" 的验收报告,你第一件该核的事是什么?(评测校准:真值怎么算、精确索引上是不是 1.0)
2. recall 已经 0.98 了,nDCG 才 0.6,系统出了什么问题?(找到了但排在后面;不做重排时这是真损失)
3. 分片后 recall 掉了一点,同事说"近似嘛正常",对吗?(不对,精确索引上分片必须和单索引字节级一致,掉了就是合并 bug)
4. "加了筛选后经常返回不满 k 条",最可能的根因?(post-filter + 谓词选择性强欠填)
5. 为什么向量库删除多用 tombstone 而非物理删?物理删的代价是什么?(O(1) vs O(n) 重排映射;代价是死节点堆积拖慢 QPS,靠 compaction 回收)
6. filtered-ANN 为什么在真 ANN 索引上比在精确索引上难?(过滤会破坏图/簇的剪枝前提,pre-filter 在高选择性下退化成全扫)
