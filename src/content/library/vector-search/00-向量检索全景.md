先说一个会让很多人不舒服的结论：你部署的那个向量数据库（pgvector / Milvus / Qdrak 之类），默认配置下大概率**不返回真正的最近邻**。它返回的是"很可能是最近邻"的一批候选，recall（召回率，真最近邻被找回的比例）通常在 0.9 到 0.99 之间，剩下那几个百分点是被索引悄悄剪掉的。这不是 bug，是设计。维度一上去，"既精确、又快、又省内存"在数学上就不可兼得——这句话不是工程经验，是有理论根源的，整本书就是围着这个取舍转。

这本书的做法是从零手写一个检索引擎，不调库。配套代码仓在 `examples/vector-search-from-scratch/`，纯 TypeScript、零运行时依赖、种子 PRNG 合成数据，同一台机器多次运行 bit-for-bit 可复现。每一章对应一个 stage 文件，可以 `npx tsx` 单独跑，recall / QPS（每秒查询数）/ 内存全是代码真算真打印出来的，不是我替你想象的。这一开篇章本身没有独立 stage，它的职责是把后面每个 stage 摆进同一张地图，并讲清楚这张地图为什么长这样。

## 为什么检索难：维度灾难

"找最相似的 k 个"听起来是个排序问题：算出查询向量到每个库向量的距离，取最小的 k 个。在二维、三维这是对的，你的空间直觉也对。问题是 embedding（把文本/图像映射成的稠密向量）动辄 768、1536、3072 维，而高维空间不遵守你的直觉。

最反直觉的一条叫**距离集中**（distance concentration）：在高维里，一个随机点到一堆随机点的"最远距离"和"最近距离"会趋于相等。换句话说，"最近邻"和"随便一个点"几乎一样近。如果最近邻并不显著地比别的点近，那"找最近邻"这件事本身就开始失去意义——后面所有索引剪枝的前提（近的归近、远的可以放心跳过）也跟着松动。

这不是空谈，stage 02 直接测了它。`examples/vector-search-from-scratch/src/stage02-similarity.ts` 的 `demoCurseOfDimensionality()` 对 `dims = [2, 16, 128, 1024]` 各跑 20 个独立查询（不是 N=1 撞运气），测每个查询的 `farthest / nearest` 比值：

```ts
// stage02-similarity.ts
const dims = [2, 16, 128, 1024];
const n = 1000;       // points per trial
const queries = 20;   // independent queries averaged ⇒ ratio is signal, not 1 lucky draw
// ...
for (const p of points) {
  const d = l2dist(query, p);
  if (d < nearest) nearest = d;
  if (d > farthest) farthest = d;
}
ratioSum += farthest / nearest;
```

在**均匀随机**数据上，这个比值随维度升高单调塌向 1——到 1024 维时，最远点几乎只比最近点远一点点。**这就是维度灾难下"最近邻"退化的样子。** 你跑一下能看到具体数字曲线，这里不引用绝对值（绝对值跟随机种子和机器相关），可迁移的是趋势：维度↑ ⇒ 对比度↓。

那为什么真实系统里向量检索还能用？因为同一个 stage 还测了第二组数据：**聚类**数据（8 个中心 + 小噪声）。聚类数据的 far/near 比值在 1024 维依然显著大于 1，对比度被结构保住了。代码里这句注释把因果讲死了：

```ts
// Why the book works anyway: ANN search exploits exactly this gap. On uniform
// noise there is no neighborhood to find and every index degrades to brute
// force; on clustered (real) data the contrast survives, so pruning is safe.
```

这是整本书第一个、也是最重要的隐含前提：**真实 embedding 不是均匀噪声，它有拓扑结构**（相似的东西扎堆）。所有近似索引——IVF 的分桶、HNSW 的图、PQ 的码本——本质上都在赌这个结构存在并去利用它。一旦你的数据真的接近均匀随机（比如喂进去一堆没训好的、各向同性的 embedding），所有近似方法的 recall 都会崩，退化成跟暴力扫描一样慢、却还不准。**调不动 recall 时，先怀疑数据有没有结构，而不是先去调索引参数。** 这是后面每一章排错的第一性问题。

> 说明：stage 的合成数据是球状、良分离的聚类高斯，比真实 embedding **更容易**。书里的 recall 绝对值偏乐观；可迁移的是相对趋势，不是绝对数。这一点贯穿全书，我不会拿乐观的绝对值唬你。

## 精确 vs 近似：你只能要两个

距离度量本身不难。stage 02 的 `core/vec.ts` 给了 `dot` / `cosineSim` / `l2dist`，还顺手证明了一个全书复用的恒等式：**对单位向量，`dot(a,b) === cosine(a,b)`**。所以每个索引都在插入时归一化一次，查询时用裸点积回答 cosine 查询，省掉每次比较的两个 `sqrt`。`stage02-similarity.ts` 的 `demoNormalizedDotEqualsCosine()` 用 1000 对随机向量验证两者误差在 1e-15 量级（浮点舍入，不是 bug）。这是免费的优化，没有取舍。

**真正的取舍在索引层。** 精确最近邻（exact NN）只有一个老实做法：暴力扫描，算 N 次距离，取最小 k 个。stage 03 的 `bruteForceTopK` 就是它，`examples/vector-search-from-scratch/src/stage03-bruteforce.ts`：

```ts
//   1. Brute force is the *definition* of correct: it scans all N vectors, so its
//      recall@k against the same-metric ground truth is exactly 1.0 — there is
//      nowhere for a true neighbor to hide. This makes it the recall *ceiling*
//      and the speed *floor* simultaneously.
//   2. Its cost is O(N · dim) distance computations *per query*. ... QPS falls
//      ~linearly as N grows.
```

暴力扫描同时是**召回天花板**（recall 恒等于 1.0，真邻居无处可藏）和**速度地板**（每查询 O(N·dim)，N=1k 还行，N=10M 不可接受）。stage 03 对 N = 1k / 5k / 20k / 50k 各测一遍，打印 QPS ∝ 1/N 的曲线——这就是为什么需要近似。

近似最近邻（ANN, approximate nearest neighbor）的全部花招，都是用 recall 去换速度或内存。这里有个**三角约束**，定义在 `core/metrics.ts` 的注释里，是全书的判据：

```ts
// An ANN index is a deliberate trade: it gives up exactness to win
// speed and/or memory. You cannot reason about that trade without quantifying all
// three axes, so every stage prints recall@k, QPS, and an (estimated) byte cost.
```

三个轴：**recall（准）、QPS（快）、内存（省）。你只能同时要两个，第三个必然让步。** 这不是悲观，是高维几何 + 信息论的硬约束。三个具体的角落，每个对应本书一章：

- **要准 + 要快，牺牲内存** → HNSW（图索引）。把每个点连成一张可导航的图，查询沿图跳几步就到，对数级。代价是图本身要存边，内存比裸向量更高。`stage04-hnsw.ts`。
- **要准 + 要省，牺牲速度** → IVF（倒排分桶）。把向量聚类分桶，查询只扫几个最近的桶（nprobe 个）。nprobe 调大 recall 升、QPS 降，是一条连续的权衡曲线。`stage05-ivfpq.ts` 里 `nprobe ∈ [1, 2, 4, 8, 16, NLIST]` 就是在画这条曲线。
- **要快 + 要省，牺牲准** → PQ（乘积量化，把向量切段后用码本压成几个字节）。stage 05 用 `KSUB = 256`（每个子码恰好 1 字节）、`NSUB = 4`，把一个 16 维 float64 向量从 `core/metrics.ts::estimateBytes` 估的 128 字节（16×8）压到 4 字节——**约 32× 压缩**（这是按 IEEE-754 double payload 算的下界估算，标 `(est.)`，不是测出来的 RSS）。代价是量化误差，recall 下降。

关键洞察是**这些机制正交、可组合**。stage 05 把 IVF 和 PQ 叠在一起（IVFPQ，FAISS 时代的大规模检索主力），但有个反直觉的坑：**两种近似各自独立掉 recall，损失会叠加**，IVFPQ 的 recall 低于单独的 IVF 也低于单独的 PQ。stage 05 还 demo 了一个经典实现 bug——PQ 应该量化**残差**（向量减去所属桶的质心）而不是原始向量，跳过这步 recall 会明显更差，代码里两种都跑给你看差距，不是嘴上说。

这就是反 survey 的态度：每个机制不只讲怎么用，先讲**怎么坏、为什么这么设计**。BM25 的 `b=0` 会让长文档靠堆词碾压短的对口文档（`stage01-bm25.ts` 的 FAILURE MODE 段）；HNSW 的 M（每节点最大边数）太小，图会碎成连不通的孤岛、recall 灾难性塌方（`stage04-hnsw.ts` 显式 demo M=2、M=1 的塌方）；分片检索时图省事让每片只返回 k/S 个结果，会悄悄漏掉全局 top-k（`stage07-eval.ts` 的 sharding 实验）。这些都是真实系统翻过的车。

## 本书地图：六个 stage + 一个评测台

每章一个 stage，每个 stage 在三角上钉一个点，前后呼应：

| 章 | stage 文件 | 在三角上的位置 | 必看的失败模式 |
|---|---|---|---|
| BM25 基线 | `stage01-bm25.ts` | 词法检索，对同义词零召回 | `b=0` 长文档堆词碾压；纯词法跨不过词表鸿沟 |
| 相似度与维度灾难 | `stage02-similarity.ts` | 距离度量 + 维度灾难根源 | 零向量产生 NaN 毒化整个 top-k |
| 暴力扫描 | `stage03-bruteforce.ts` | recall 天花板 / 速度地板 | QPS ∝ 1/N，规模一大就崩 |
| HNSW 图 | `stage04-hnsw.ts` | 准+快，牺牲内存 | M 太小图碎成孤岛，recall 塌方 |
| IVF-PQ | `stage05-ivfpq.ts` | 准+省 / 快+省 的两条边 | 两种近似 recall 损失叠加；PQ 量化原始向量而非残差 |
| Hybrid 混合 | `stage06-hybrid.ts` | 词法+语义融合 | 两个不同量纲的分数直接相加，一路淹没另一路 |
| 评测台 | `stage07-eval.ts` | 怎么诚实地量 | 分片漏召回 / 过滤后 under-fill / 删除后墓碑复活 |

读法上有两条建议。第一，**先把 stage 03 暴力扫描的 recall=1.0 当作锚**。后面每个近似索引的 recall 都是相对这个锚在掉，掉多少换来多少 QPS/内存，这才是你要建立的体感。脱离暴力基线谈 recall 是没有意义的——stage 07 的 `runBaseline()` 第一件事就是断言精确索引 recall 必须等于 1.0，否则整个评测台是坏的、下游所有数字无意义。第二，**recall 绝对值别背，记趋势和参数怎么动**：nprobe↑ ⇒ recall↑/QPS↓，PQ 用 recall 换内存，HNSW 的 ef（搜索时的候选束宽）↑ ⇒ recall↑/QPS↓。这些趋势在真实数据上成立，绝对值不一定。

## 这本书和书库别的线、和 agent 书怎么分工

书库里已经有一条 **RAG 工程师路线**，讲的是 RAG（检索增强生成）**管道**：怎么切块、怎么选 embedding 模型、检索回来怎么塞进 prompt、怎么做 reranking 和评估。那条线把向量检索当成一个能用的黑盒组件。

**这本书撬开那个黑盒，讲引擎内核——索引算法本身怎么造。** 你在那条线学到"用 hybrid search 提升召回"，到这本书你会亲手把 BM25 和向量分数融合，并撞上"两个量纲不同的分数直接相加会互相淹没"这个真实的坑（`stage06-hybrid.ts`）。两条线互补：一条教你**怎么用好**检索，一条教你**检索为什么这么转、什么时候会坏**。

和 **agent 书第 7 章 agentic retrieval** 的分工是另一个维度：那一章讲 agent **何时该去检索**（检索是 agent 的一个动作，怎么决策触发、怎么用结果）；这本书讲**引擎怎么造**（检索这个动作背后的数据结构和算法）。一个在决策层，一个在系统层，不重叠。

## ⚡ 开放问题：高维精确最近邻无法既快又省

这本书存在的根本理由，是一个目前**没有通用解**的理论事实：**在高维空间，精确最近邻查询无法同时做到亚线性查询时间和接近线性的空间**。

把它说具体些。低维（比如 2D/3D）有 kd-tree、R-tree 这类空间划分结构，能做到精确 + 亚线性查询 + 线性空间，三角不成立、三个都能要。但随维度升高，这些树的剪枝能力指数级退化——到几十维，kd-tree 查询就退化成几乎全扫，跟暴力没区别。这背后是一族被研究了二十多年的下界结果（locality-sensitive hashing 及其理论分析、cell-probe 模型下的时空权衡下界等），它们的共同含义是：想要精确，要么放弃亚线性时间（退回暴力），要么接受空间随维度爆炸；想两者兼得，**只能放弃精确**——这正是 ANN 出现的理由，也是本书所有近似机制存在的根。

要强调的是，这是**正在研究、没有通用最优解**的领域，而不是已经盖棺的定论。学术上 worst-case 下界很硬，但实践中大家靠的是"真实数据有结构、不是 worst case"这个前提（回到本章开头的聚类 vs 均匀对比）。具体到"给定数据分布、给定 recall 目标，什么索引参数在时空上最优"——没有闭式解，FAISS / hnswlib / DiskANN / pgvector 各有各的经验配方和 autotune 启发式，仍是活跃工程与研究方向。**所以你不会在本书结尾拿到"用这个索引就对了"的银弹；你会拿到一把尺子——recall/QPS/内存三角——和一组关于每种机制何时坏、为什么坏的判断力。** 在一个没有通用解的领域，尺子和判断力比结论更值钱。
