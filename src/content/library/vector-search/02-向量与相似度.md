写检索引擎,绝大多数人第一行就写错了:他们随手抓了一个 `cosineSimilarity`,以为「相似度」是一个客观存在的量,像两点间的距离一样唯一。不是。对同一个查询、同一组候选,换一个度量函数,top-1 的赢家会换人。这不是边界 case,是设计的常态。下面这张表是配套仓 `examples/vector-search-from-scratch/src/stage02-similarity.ts` 跑出来的真实输出,三个度量在 `query = [1,1]` 上各自加冕了不同的向量:

```
name              dot      cosine    l2dist
a_huge_off_dir    10.000   0.7809    8.0000
b_same_dir         6.000   1.0000    2.8284
c_spatially_near   2.700   0.9939    0.5385
dot   picks: a_huge_off_dir   (rewards raw magnitude)
cosine picks: b_same_dir   (rewards direction, ignores length)
l2    picks: c_spatially_near   (rewards spatial closeness)
```

`dot` (点积,逐维相乘再求和) 把冠军给了 `a_huge_off_dir`——一个方向偏掉、但长度巨大的向量;`cosine` (余弦相似度,只看夹角) 给了方向完全对齐的 `b_same_dir`;`l2` (欧氏距离,空间直线距离) 给了离查询点物理上最近的 `c_spatially_near`。三个度量,三个赢家,零重叠。所以「我接了向量库、用了相似度检索」这句话信息量约等于零——没说度量,就没说清你到底在找什么。这一章要把这件事讲到你写下任何一行检索代码前,先能回答「我为什么用这个度量」。

## 三种度量,三种「相似」的定义

度量不是实现细节,是建模决策。先看三个函数到底在算什么。配套仓 `examples/vector-search-from-scratch/src/core/vec.ts` 里,它们是整个引擎仅有的三个数值原语:

```ts
// dot: 逐维相乘求和。整本书最重要的操作——cosine 和(归一化后的)最近邻都归约到它。
export function dot(a: number[], b: number[]): number {
  let sum = 0;
  for (let i = 0; i < a.length; i++) sum += a[i] * b[i];
  return sum;
}

// cosine: dot / (||a||·||b||),范围 [-1,1]。1=同向,0=正交,-1=反向。
export function cosineSim(a: number[], b: number[]): number {
  const denom = Math.sqrt(normSq(a)) * Math.sqrt(normSq(b));
  if (denom === 0) return 0;
  return dot(a, b) / denom;
}

// l2: 欧氏距离。越小越近(和 cosine 的排序方向相反)。
export function l2dist(a: number[], b: number[]): number {
  let s = 0;
  for (let i = 0; i < a.length; i++) {
    const d = a[i] - b[i];
    s += d * d;
  }
  return Math.sqrt(s);
}
```

三者的语义差别,一句话各自概括:

- **dot 奖励「长度 × 对齐度」**。两个向量越长、越同向,点积越大。它对模长 (magnitude,向量长度) 敏感。
- **cosine 只奖励「对齐度」**,把模长除掉了。它问的是「方向多接近」,完全无视谁长谁短。
- **l2 奖励「空间近」**,问的是「两点直线距离多短」,长度和方向都揉进了这一个标量里。

### 什么时候用哪个

这不是品味问题,取决于你的 embedding (向量化,把文本/图片压成定长数字串) 怎么训出来的——更准确说,取决于**模长在你的向量里编码了什么信息**。

**模长无意义 → 用 cosine(或归一化后的 dot)**。绝大多数文本 embedding (OpenAI `text-embedding-3`、BGE、E5 系列) 训练时就把语义编码进了**方向**,模长基本是噪声(受文本长度、token 数等无关因素影响)。这时用 dot 会出上面 `a_huge_off_dir` 那种灾难:一篇又臭又长、主题不沾边的文档,只因为向量长,就被顶到 top-1。真实系统里这点是默认共识——FAISS 的 `IndexFlatIP` (内积索引) 配文档 embedding,前提就是你已经归一化过;pgvector 直接提供 `<=>` 余弦距离算子,就是因为大多数人要的是这个。

**模长承载信息 → 用 dot**。推荐系统的双塔模型 (two-tower,用户塔和物品塔各出一个向量,内积当评分) 是典型反例:模长被训练去编码「热度 / 置信度 / 点击率先验」。这时你**故意**要 dot 那种「长向量赢」的行为——一个又对齐、又高置信的物品就该排前面。把它换成 cosine,等于把模型辛苦学到的一维信息直接抹掉。

**坐标本身有物理意义 → 用 l2**。地理坐标、图像像素空间、某些没归一化的特征空间,「直线距离近」就是你真正要的语义。但要注意 l2 和 cosine 在归一化向量上其实高度相关(下一节会算),纯文本检索里两者排序往往一致,差别主要在未归一化的场景。

判断口诀:**先问「我的向量,长度代表什么?」**。代表噪声就除掉它(cosine);代表信号就留着它(dot);代表空间位置就量距离(l2)。答不出这个问题,说明你对自己用的 embedding 还不够了解,这比选错度量更危险。

## 归一化:让 dot 一个函数干完两件事

cosine 每次调用要算两个模长,即两次 `Math.sqrt`。在百万级向量的扫描里,这是纯浪费——因为有一个恒等式可以让你一次都不付:

> 对**单位向量**(模长为 1)而言,`cosine(a,b) === dot(a,b)`。

推导一行就够:`cosine(a,b) = dot(a,b) / (||a||·||b||)`,当 `||a||=||b||=1`,分母是 1,于是 `cosine(a,b) = dot(â,b̂)`。配套仓 Part 2 用 1000 对随机向量验证了这个恒等式,报告的是最坏情况的浮点误差:

```
trials: 1000 random pairs, dim 8
max |cosineSim(a,b) - dot(normalize(a),normalize(b))| = 3.331e-16
OK: equal to double precision (the ~1e-16..1e-15 residue is float rounding, not a bug).
```

`3.331e-16` 是 double 精度的舍入残差,不是 bug——它恰恰证明这个等式在数值上成立到机器精度。注意这里的方法学:验证用了 1000 对随机向量而非挑一两个好看的例子,因为 N=1 的「相等」可能是运气,只有大量样本里最坏误差仍贴着 `1e-16`,才说明等式是结构性的而非偶然。

工程上的现金回报很直接:**在写入时归一化一次,之后所有 cosine 查询用裸 dot 回答,每次比较省下 2 个 sqrt**。这就是为什么所有正经索引——FAISS、hnswlib、Milvus——的标准用法都是「存之前先 normalize,索引内部只跑内积」。归一化是 O(1) 摊销的一次性成本,换来查询热路径里每次比较都更省。第 7 章评测会量这个吞吐差;这一章只确立你**有代数许可**去省那两个 sqrt。

### 失败模式:零向量会无声毁掉你的 top-k

「归一化一次然后只跑 dot」有个埋伏:零向量没有方向。天真的 `normalize` 实现会除以 0,而 IEEE 浮点里 `0/0 = NaN`,然后 NaN 会顺着求和、排序一路扩散。配套仓 Part 2b 把危险和防御都演了出来:

```
naive normalize(zero) = [NaN,NaN,NaN]  ⇒ dot = NaN
cosineSim(zero, real) = 0   (guarded: zero treated as orthogonal)
```

为什么这个 bug 特别阴险:一个 NaN 分数会让「NaN 和任何数比较」都返回 false,于是这个坏向量**既不会排高也不会排低**——它不报错、不崩溃,只是悄悄把一个本该进 top-k 的真邻居挤出去。你的召回率掉了几个点,日志干干净净,查一周查不出来。

`core/vec.ts` 的处理是边界守卫(`denom === 0` 直接返回 0,把零向量当作和所有人正交),`normalize` 则返回一份零拷贝而非 NaN,维持「输出模长非 0 即 1,绝不是 NaN」的不变量。注意这里的判断:这不是「希望线上没有零向量」,而是**假设它一定会出现**(空文本、全 mask 的 token、上游 embedding 服务超时返回空)然后用最无聊的守卫钉死它。检索引擎里,无声的错误比崩溃危险得多——崩溃你至少看得见。

## 维度灾难:为什么「最近」会失去意义

前面默认了一件事:存在一个「最近邻」可找。高维空间里,这个前提会动摇。这就是**维度灾难**(curse of dimensionality):维度一高,一组点里最远和最近的距离会越来越接近,「最近」和「随便一个点」几乎一样近。

这话常被当口号背,但很少有人量过。配套仓 Part 3 直接测了:对每个维度,取一个查询点,看它到 1000 个随机点的最近距离和最远距离,算比值 `farthest/near`(对 20 个独立查询求平均,排除单次抽样的运气)。先看**均匀随机**数据(无任何结构,灾难全开):

```
UNIFORM random data (no structure — the curse in full force):
dim    nearest   farthest   ratio(far/near)
   2    0.0162     1.0534    100.9390
  16    0.8535     2.2691      2.6703
 128    3.8228     5.3166      1.3919
1024   12.3573    13.7631      1.1138
```

读这条曲线:dim=2 时,最远点比最近点远 **100 倍**——「最近邻」鹤立鸡群,一眼可辨。到 dim=1024,比值塌到 **1.11**——最远的点只比最近的点远 11%。换句话说,在 1024 维的均匀噪声里做最近邻查询,你捞回来的「最近邻」只比一个随机点近一丁点。「最近」这个概念在这里近乎失效。

这给出两个直接推论,都不是空谈:

1. **为什么 embedding 维度选择有讲究**。维度不是越高越好。维度越高,对比度(contrast,最远最近的区分度)越被稀释,而且每个向量的存储和每次距离计算的成本线性上涨(第 5 章 PQ 压缩、第 4 章 HNSW 的内存账都和维度直接挂钩)。这就是为什么 embedding 模型常提供 256/512/768/1024 多档维度,以及 Matryoshka(套娃式 embedding,前 k 维即可独立使用)这类技术存在的意义——你常常可以砍维度换速度和内存,而召回掉得没你怕的多。
2. **为什么必须用近似而非精确**。如果在你的数据上对比度已经塌到接近 1,那么精确最近邻和近似最近邻给出的「邻居」质量差不多——精确算法多付的那些距离计算,买回来的区分度微乎其微。这是第 4 章敢用 HNSW、第 5 章敢用 IVF-PQ 去**牺牲精确性换速度**的底层许可证。

### 为什么真实数据救了这本书

如果到此为止,结论很灰暗:高维下检索没意义,这本书可以合上了。但救赎在于**真实 embedding 不是均匀噪声,它们聚类**(同主题的文本扎堆)。配套仓对**带结构的聚类数据**(8 个中心)跑了同样的测量:

```
CLUSTERED data (8 centers — the structure real embeddings have):
dim    nearest   farthest   ratio(far/near)
   2    0.0053     1.1409    364.0486
  16    0.1012     1.8969     18.9159
 128    0.4012     5.0166     12.5176
1024    1.2516    13.5858     10.8558
```

关键对比在最后一行:dim=1024 时,均匀数据的对比度是 **1.11**,聚类数据是 **10.86**——差了近 10 倍。结构守住了对比度。这正是 ANN(approximate nearest neighbor,近似最近邻)搜索能成立的根本原因:在均匀噪声里没有「邻域」可找,任何索引都退化成暴力扫描(第 3 章会从另一头印证这点);而在聚类的真实数据上,对比度活了下来,于是**剪枝是安全的**——HNSW 跳过远处的图节点、IVF 跳过不相关的 cell,都不会误杀真邻居,因为真邻居和噪声之间还有 10 倍的距离差撑着。

所以这本书后面所有索引技巧,本质都在吃这个「对比度缺口」。你的数据聚类得越好,缺口越大,索引能剪得越狠、跑得越快。反过来,如果你发现某个索引在你的数据上召回怎么调都上不去,先别怪索引——量一下你数据的对比度,很可能是 embedding 本身没把语义聚拢,缺口太小,巧妇难为无米之炊。这件事第 7 章评测会给出可操作的诊断方法。

## ⚡ 开放问题:对比度本身能不能被「学」出来

上面把对比度当成数据的固有属性——给定 embedding,缺口就那么大,你只能接受。但有没有可能**主动把缺口撑大**?这是度量学习(metric learning,学一个让相关样本更近、无关样本更远的距离函数)和各向异性 embedding(anisotropic embedding,向量在空间里不是各向均匀分布,而是挤在某些方向上)正在研究的方向,目前没有通用解。

两个具体的活跃前沿:

- **各向异性塌缩**。实测发现,很多预训练语言模型出来的 embedding 是「锥形」的——所有向量挤在高维空间的一个窄锥里,余弦相似度普遍偏高(任意两段文本都「有点像」),对比度被人为压低。whitening(白化,把向量分布拉成各向同性)、对比学习后处理等手段能部分缓解,但「最优的各向同性变换是什么、会不会损失下游任务信息」没有定论。这直接关系到你这一章选的度量:如果底层向量是各向异性的,cosine 的区分力会比理论值差,你可能需要先校正分布。
- **任务相关的度量**。一个固定的 cosine/l2 对所有查询一视同仁,但「相关」在不同任务里含义不同(问答里的相关 ≠ 推荐里的相关)。学一个查询条件化(query-conditioned)的距离函数理论上能把对比度按任务撑大,但训练数据、泛化、以及能不能塞进 ANN 索引的几何假设里(HNSW/IVF 都默认度量满足三角不等式),都还是开放的——这也是为什么生产系统至今几乎全用固定度量 + rerank(用一个慢但准的模型对 top-k 重排)这种妥协方案,而不是端到端学度量。

换句话说:这一章教你在**给定**度量和 embedding 下做对的工程判断;而度量和 embedding 本身怎么共同优化以最大化对比度,还是一片没有标准答案的地。下一章我们先假定度量已定,直面最朴素的实现——暴力扫描,看它在真实数据上到底有多慢、慢在哪,从而理解后面所有索引到底在优化什么。
