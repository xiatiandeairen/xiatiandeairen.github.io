> 上一章（暴力扫描）给了我们一根标尺：在 5000 向量 × 32 维、200 条查询、k=10 的合成数据上，全量比对每次都对（recall@10=1.0，定义上的精确），跑出 **807 QPS**，索引内存约 1.22 MB。问题不在它错，在它扫。query 数据量一上去，QPS 跟着 n 线性掉。HNSW 的全部价值，就是在「几乎不丢召回」的前提下，把每次查询从「扫 n 个向量」变成「在图上走 log n 跳」。这章我们把它造出来，然后专门去找它坏掉的地方。

跑这一章的代码（`examples/vector-search-from-scratch/src/stage04-hnsw.ts`），同一份数据、同一台机器，得到的是这样一条曲线：

```
   ef  | recall@10 |    QPS   | vs brute
  -----+-----------+----------+---------
     1 |  0.6500  |    25210 | 31.3x
    16 |  0.7485  |    18464 | 22.9x
    64 |  0.9455  |     6353 |  7.9x
   128 |  0.9745  |     3780 |  4.7x
   256 |  0.9990  |     2118 |  2.6x
```

读这张表的方式不是「ef 越大越好」，而是：**recall 0.945 到 0.999 这最后五个百分点，QPS 从 6353 砍到 2118，掉了三分之二。** 这条「越往高召回越贵」的边际成本曲线，就是 ANN（approximate nearest neighbor，近似最近邻——只保证「大概率找到真正最近的」而非「一定找到」）这门生意的全部经济学。你交付的不是「快」，是「在某个 recall 点上的 QPS」，而这个点是你自己选的。

> 和书库分工说一句：RAG 工程师那条线讲的是「检索管道怎么搭、chunk 怎么切、上下文怎么拼」；这本书讲的是管道最底下那个黑盒——检索引擎内核本身怎么造。Agent 书第 7 章讲 agentic retrieval「何时该去检索」；这章讲的是「检索引擎内部怎么走图」。三条线是正交的，别混。

## 为什么是图，不是树

先说清楚我们在逃离什么。低维（2~3 维）做最近邻，经典答案是 KD-tree、ball-tree 这类空间划分树，能做到真 O(log n)。但向量检索是 32 维、768 维、1536 维。维度一高，空间划分树就被「维度灾难」（curse of dimensionality——高维下点之间的距离趋于相等，划分失去意义）打垮：要剪掉的子树几乎剪不掉，最后退化成全扫，还不如暴力。**这是高维近邻问题的硬骨头：树结构在这里基本死了。** 这也是为什么 FAISS、hnswlib、Milvus、pgvector 这些真实系统的主力索引要么是图（HNSW），要么是量化+倒排（IVF-PQ，下一章），没人在生产里对 768 维上 KD-tree。

HNSW 选的是图。直觉一句话：**把所有向量连成一张「小世界网络」，查询就是从一个入口点出发，每一步贪心地跳到离目标更近的邻居，直到跳不动。** 「小世界」（small world——任意两点间只需很少几跳就能到达，社交网络的「六度分隔」就是这个）是关键词：图里既要有大量短程边（保证局部精度），又要有少量长程边（保证你能从图的一头几跳到另一头，不至于沿着邻居一格一格挪 n 步）。

光一层图（这叫 NSW，HNSW 去掉 H）也能用，但导航慢——你总得从某个固定入口爬过去。HNSW 的 H（hierarchical，分层）就是在这上面加了「高速公路」。

## 分层：高速公路 + 地方道路

HNSW 的结构是一摞图，从上到下越来越密：

- **顶层**：极稀疏，只有少数几个节点，边是长程的——这是高速公路，几跳就能跨越整个数据集。
- **中间层**：逐层变密。
- **第 0 层（layer 0）**：包含每一个节点，最密——这是地方道路，做最后的精细定位。

查询时的走法（看代码里的 `query`）：

```typescript
query(q: number[], k: number, ef: number): number[] {
  if (this.entryPoint === -1) return [];
  let ep = this.entryPoint;
  // 上层只用贪心爬山（相当于 ef=1），目的只是把入口点「传送」到 q 附近
  for (let l = this.topLayer; l > 0; l--) ep = this.greedyDescend(q, ep, l);
  // 到了 layer 0 才用宽度为 ef 的束搜索做精细定位
  const found = this.searchLayer(q, [ep], 0, Math.max(ef, k));
  return found.slice(0, k).map((e) => e.id);
}
```

先在上层用最便宜的贪心爬山（`greedyDescend`，每步只跳到当前最近的一个邻居，跳不动就停）一路传送到目标附近，最后只在 layer 0 做一次真正的束搜索。本章的实测里这张图建出来是 **4 层**，layer0 平均度 25.1，5000 个节点全部可达（`可达节点=5000/5000`）。

每个节点该出现在多少层？这是 HNSW 最精巧也最容易被实现者写错的一步。看 `randomLevel`：

```typescript
private randomLevel(): number {
  if (this.mL === 0) return 0; // M<=1：故意压成平图（失败模式 2 会用到）
  let u = this.rng();
  while (u === 0) u = this.rng(); // ln(0) = -Infinity，重采那个测度为零的 0
  return Math.floor(-Math.log(u) * this.mL);
}
```

层级是从一个指数衰减分布里抽的：绝大多数节点只在 layer 0，约 1/M 的节点上到 layer 1，再 1/M 上到 layer 2……`mL = 1/ln(M)` 这个归一化因子（代码里 `this.mL = cfg.M > 1 ? 1 / Math.log(cfg.M) : 0`）让层数期望是 log_M(n) 量级。**这就是「近 O(log n)」的来源：每往上一层节点数几何级变少，高速公路天然成形。** 注意那行 `while (u === 0)`——`ln(0)` 是负无穷，会把节点抛到无穷高层，这是一个真实存在的边界 bug，不是装饰。

## 两个旋钮：M 和 ef

整个 HNSW 你只需要拧两个旋钮，但它俩的性质完全不同，分清楚是用好它的前提：

| 旋钮 | 含义（大白话） | 何时生效 | 牵制什么 |
|------|------|------|------|
| **M** | 每个节点最多连几个邻居（图有多密） | 建图期，定死 | 图的连通性下限、内存、建图时间 |
| **ef** | 搜索时同时存活的候选数（束有多宽） | 查询期，可随时调 | 召回 ↔ QPS |

**ef 是运行时旋钮**——同一张图，你这次查 ef=16 图便宜召回低，下次查 ef=256 慢但准，不用重建。**M 是建图期旋钮**——它决定了图的物理连通性，一旦建错，查询期任何 ef 都救不回来。这个非对称性是本章最重要的判断：**ef 拧错是运营问题，M 拧错是事故。**

### ef：束搜索的宽度

束搜索（beam search——同时保留一批最有希望的候选一起往前探，而不是只认一条路）的心脏在 `searchLayer`。它维护两个堆：`candidates` 是前沿（还没展开的、按最近优先的 min-heap），`results` 是当前最好的 ef 个（按最远优先的 max-heap，堆顶是「最好里最差的那个」）。停止条件是这一行：

```typescript
// 剩下最近的待探节点已经比我们保留的最差结果还远，且 results 已装满 ef 个
// → 没有任何可达路径能改进 top-ef 了，停。
if (results.size >= ef && nearest.dist > results.peekDist()) break;
```

ef 越大，`results` 越晚装满，停止条件越晚触发，搜索探得越深，召回越高——代价是每次查询多走很多跳，QPS 掉。这就是开头那条曲线的机制。**你在生产里调的就是这个数：先定一个能接受的召回（比如 0.95），二分找到刚好达标的最小 ef，那个 ef 就是你的最优工作点。** hnswlib 暴露的 `ef`、FAISS HNSW 的 `efSearch`、pgvector 的 `hnsw.ef_search`，全是同一个旋钮。

### M：图的密度与连通性

M 控制每个节点保留多少边。layer 0 特殊照顾，用 `2*M`（代码里 `this.Mmax0 = cfg.M * 2`），因为底层扛着真正的召回负担，密一点值得。本章 M=16，建出来 layer0 平均度 25.1（双向连边、加上溢出裁剪后的实测值），索引内存约 **2.22 MB**——比暴力的 1.22 MB 多出来的就是图的边（每条有向边在 JS 里占一个 8 字节 double 槽）。

但 M 真正的作用不是密度，是**连通性下限**。这就引出本章最该让你记住的两个失败模式。

## 失败模式一：ef 太小，召回有天花板

最常见的生产事故，不是图建坏了，是**有人把 ef 留在了那个很小的构建默认值上，然后发现召回只有 0.65 还不知道为什么。** 看实测：

```
   ef=1: recall@10=0.6500 (被 query() 内的 max(ef,k) 钳到 ef=10; 仍远低于 ef=256 的 0.999)
   ef=4: recall@10=0.6500 (被 query() 内的 max(ef,k) 钳到 ef=10; 仍远低于 ef=256 的 0.999)
```

为什么 ef=1、ef=4 召回一模一样、都是 0.65？因为 `query` 里有一行 `Math.max(ef, k)`——k=10，所以任何小于 10 的 ef 都被钳到 10。也就是说曲线表里 ef=1 到 ef=8 那四行召回全是 0.65，不是巧合，是它们的有效 ef 都被钳成了 k=10。

**为什么 ef 小召回就崩？** 束太窄时，`results` 几乎瞬间装满，停止条件在走了几跳之后就触发，搜索一头扎进第一个局部不错的区域就再也不回头——而真正的最近邻可能就在隔壁一个簇里，永远没被探到。这是束搜索的本质局限：**ef 就是你愿意为「回头看看」付的预算，预算给太少，就在第一个像样的坑里躺平了。**

判断：ef 太小是「软失败」——索引照常返回 k 个结果，看起来一切正常，只是其中混进了一堆不那么近的。没有报错，没有崩溃，只有线上召回莫名其妙低。**这就是为什么离线 eval（第 7 章会专门讲）不是可选项**：不测召回，你根本看不见这个洞。

## 失败模式二：M 太小，图不连通，大 ef 也救不回

这个更阴险。ef 错了至少调 ef 能补；M 错了图已经物理断裂，查询期束开多宽都没用。实测：

```
   M=2: recall@10=0.3060 @ef=200 | 层数=14 | layer0 平均度=2.9 | 可达=1550/5000 (稀疏图, 长程链路不足)
   M=1: recall@10=0.0105 @ef=200 | 层数=1  | layer0 平均度=1.2 | 可达=22/5000  (M=1 ⇒ mL=0 ⇒ 单层平图, 无上层高速路, 导航退化)
```

注意这两行都开了 **ef=200**（很慷慨的束宽），召回照样是 0.306 和 0.011。关键看最后一列 `可达`：

- **M=2**：从入口点出发，layer 0 上只有 1550/5000 个节点可达。**剩下 3450 个节点，无论你 ef 开多大，都永远不可能被返回**——它们根本不在搜索能走到的图里。recall 卡在 0.306 是图的天花板，不是束的。
- **M=1**：更极端。`mL = 1/ln(1) = 1/0`，代码用 `cfg.M > 1 ? ... : 0` 把它兜成 0，`randomLevel` 直接 `return 0`，于是所有节点都在 layer 0，没有上层高速公路，图退化成一张平的、平均度只有 1.2 的稀疏图。可达只剩 22/5000，导航彻底失效，recall 0.011 基本等于瞎猜。

**为什么这么设计、为什么这么坏？** HNSW 靠的是「小世界」假设——既要短程边保精度，又要长程边保可达。M 太小，每个节点的边预算不够，diversity 启发式（`selectNeighbors`，倾向于保留指向不同方向的边而非全挤进同一个稠密簇）也没法在 1~2 条边里腾出长程链路，图就碎成一堆互不连通的孤岛。

代码里有一处专门防这个的细节，值得拎出来看——插入时为「下一层的入口点」选谁：

```typescript
// 下一层的入口必须是已经在图里、且在该层有边的节点——也就是这次束搜索找到的最近候选，
// 绝不能是正在插入的 id。id 在下层还没有任何边，拿它当入口会从一个死胡同开始束搜索，
// 新节点最后只连进它自己落地的一小块区域 → 图碎掉（reachable ≪ n），每个 ef 召回全趴。
if (candidates.length > 0) ep = candidates[0].id;
```

这一行如果写错（拿正在插入的 `id` 当下一层入口），症状和 M 太小一模一样：图碎、可达远小于 n、所有 ef 召回全趴。**「图碎掉」是 HNSW 实现 bug 和参数误配的共同终点**，而 `countReachableFromEntry()` 这个内省函数（数从入口能走到几个节点）就是你的诊断探针。生产里的 hnswlib/FAISS 不会给你这个数，所以排查图连通性是出了名地难——这也是为什么很多团队宁可重建也不敢动 M。

## 何时别用 HNSW

讲完它怎么坏，得讲它根本不该上场的地方。HNSW 的两个软肋决定了它的适用边界：

**1. 内存。** 本章 2.22 MB 看着没什么，但记住它的构成：**全量原始向量 + 全部图边，全在 RAM 里。** 线性外推一下（这是估算，标注为估算）：10 亿条 768 维 float32 向量，光向量就是 10⁹ × 768 × 4 B ≈ **3 TB**，再加每节点几十条边的图，单机内存直接爆。FAISS、hnswlib 的 HNSW 都是内存索引，这就是它在十亿级别撞墙的根本原因。第 5 章的 IVF-PQ 用乘积量化（product quantization）把每条向量压到几十字节，正是冲着这个内存账去的。而 **DiskANN**（微软的磁盘 ANN 方案）干脆把图主体放 SSD、内存只留压缩后的导航摘要——它存在的全部动机，就是「HNSW 的图装不进十亿级的内存」。

**2. 删除与动态更新。** 这是 ⚡ 本章的前沿/开放问题。HNSW 论文给的是**增量插入**算法，但**没给删除**。真删一个节点意味着要修复所有指向它的边、可能重选邻居、甚至影响层级结构——代价高到没人在线上真做。业界的实际做法是「软删除」：打个 tombstone（墓碑标记），查询时过滤掉，节点和边还赖在图里。后果是：

- 删多了图里全是死节点，搜索路径被它们污染，召回和速度都退化。
- 高频增删的工作负载下，图会逐渐「退化」——长程链路被破坏、局部越来越乱，最终只能**周期性全量重建**。

**到今天为止，「支持高效原地删除、且长期增删后图不退化」的 HNSW 变体仍没有公认通用解，是活跃研究方向。** 各家系统的妥协各不相同：有的定期 compaction（压实重建），有的限制删除比例，有的（如部分 DiskANN 变体 FreshDiskANN）专门为「fresh / streaming」更新设计了重建较少的更新协议，但都还没到「装上就不用管」的程度。**所以一个判断：如果你的数据是高频增删的（比如实时去重、会话级临时索引），HNSW 不是默认答案，先想清楚重建策略再上。**

## 小结：两个旋钮，一条曲线，两种死法

HNSW 做的事，一句话：**用「分层高速公路 + ef 受限的束搜索」，把 O(n) 的暴力扫降到近 O(log n) 的图上导航。** 实测它在本章数据上 ef=256 拿到 recall 0.999 / 2118 QPS，ef=64 拿到 0.945 / 6353 QPS（暴力的 7.9 倍），代价是内存从 1.22 MB 涨到 2.22 MB。

但你真正要带走的不是这几个数，是这三条判断：

1. **ef 是运行时召回↔速度旋钮**，按目标召回二分调最小 ef；调小了召回有软天花板，且不会报错——所以离线 eval 是必需品（第 7 章）。
2. **M 是建图期连通性下限**，调小了图会物理碎裂，`可达 ≪ n`，任何 ef 都救不回；M 错=事故，ef 错=运营。
3. **HNSW 是内存索引、删除是开放难题**——十亿级别和高频增删这两种场景，是它该让位给第 5 章 IVF-PQ（省内存）和 DiskANN（上磁盘）的地方。

下一章（IVF-PQ）我们就去解决「向量本身太占内存」这件事：先用倒排把搜索范围切小，再用乘积量化把每条向量压成几十字节——你会看到召回又掉一截，换来的是内存数量级的下降，以及一条和本章 ef 曲线形状不同、但同样要你自己选工作点的权衡曲线。
