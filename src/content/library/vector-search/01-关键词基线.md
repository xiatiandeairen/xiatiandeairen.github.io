> 这是一本讲检索引擎内核的书：HNSW 怎么造、IVF-PQ 怎么压、Hybrid 怎么融、recall 怎么测。和书库里那条 RAG 工程师路线分工明确——那条线讲整条 RAG 管道（chunking / rerank / 文档智能 / 评测），这本只钻一件事：**索引算法本身**。和 agent 书第 7 章 agentic retrieval 也互补——那讲"agent 何时该去检索"，这讲"被检索的那个引擎怎么造出来"。

第一章不写向量。先写 BM25，理由很务实：**如果你不知道 BM25 在什么 query 上赢、在什么 query 上彻底失灵，你就没资格判断 embedding 到底买来了什么。** 我见过太多"上来就 all-in 向量"的系统，在精确关键词查询（错误码、函数名、SKU、人名）上召回反而比一行 `WHERE text MATCH ?` 还差，然后花三个月调 embedding model，最后发现答案是"把 BM25 加回来"。这章就是为了让你不走那三个月弯路。

本章覆盖：倒排索引（让检索 sublinear 的数据结构）、BM25 打分的三个零件（idf / tf 饱和 k1 / 长度归一化 b）、一个**一个旋钮就能复现**的失败模式（关掉长度归一化，长文档霸榜），以及 BM25 的根本死穴——词表不重叠时的零召回，这正是下一章 embedding 存在的全部理由。所有数字都来自配套仓 `stage01-bm25.ts` 的真实运行输出，代码可跑、可改旋钮、可复现。

## 一、倒排索引：为什么检索能做到 sublinear

先说数据结构，因为打分是建立在它之上的。

朴素做法是"扫全库"：query 来了，遍历每篇文档算一次相关性。N 篇文档就是 O(N)。这在 30 篇玩具语料上无所谓，在一亿篇上是灾难。

**倒排索引（inverted index）**——一句大白话：一张"词 → 哪些文档里有这个词"的反查表——把这件事翻转过来。正排是"文档 → 它包含的词"，倒排是"词 → 包含它的文档列表（postings list，倒排表）"。每个 postings 条目还存一个 `tf`（term frequency，该词在该文档里出现几次），这样打分时**不用回头再读文档原文**。

配套代码里这个结构就是一个两层 Map（`examples/vector-search-from-scratch/src/stage01-bm25.ts`）：

```typescript
interface InvertedIndex {
  postings: Map<string, Map<number, number>>; // term -> (docId -> tf)
  docLengths: number[]; // 每篇文档的 token 数，给长度归一化用
  avgDocLength: number; // 全库平均长度，b 归一化的基准
  docCount: number;
}
```

注意 `postings` 的 value 是 `Map<docId, tf>` 而不是单纯的 `Set<docId>`。**把 tf 存进 postings，是倒排索引能在查询时直接打分、不回读原文的关键**——这是它和"只记录文档是否包含某词"的布尔索引的本质区别。

跑一遍 stage01，索引建出来长这样：

```
[index] 26 docs, 320 unique terms
[index] avg doc length = 18.5 tokens
[postings] "vector" -> df=4, idf=1.7918
[postings] doc0:tf=2  doc12:tf=1  doc18:tf=1  doc25:tf=1
```

`"vector"` 的 postings 列表只有 4 个文档。**这就是 sublinear 的来源**：查询 `vector` 时，引擎只碰这 4 篇，那些讲煮意大利面、骑车、养绿植的文档（语料里故意掺的噪声）连看都不会看一眼。检索代码里这一步叫候选收集（candidate gathering）——取所有 query term 的 postings 的并集：

```typescript
const candidates = new Set<number>();
for (const term of queryTerms) {
  const postingList = index.postings.get(term);
  if (postingList === undefined) continue;
  for (const docId of postingList.keys()) candidates.add(docId);
}
```

代价随匹配文档数走，不随全库规模走。这就是为什么 Lucene / Elasticsearch 能在十亿级文档上毫秒响应——它们碰的从来不是全库，只是 postings。生产级实现还会在 postings 上叠 skip list、block-max WAND 之类的剪枝，让"取并集再打分"进一步只触碰可能进 top-k 的文档，但内核思想就是这张反查表。

### tokenize：一个不起眼但能毁掉召回的不变量

建索引前要把文本切成词（tokenize，分词）。stage01 用的是最朴素的管线：小写化、按非字母切分、去掉一个极小的停用词表。它**故意不做 stemming（词干提取，把 running/ran 归一成 run）**——stemming 是和 BM25 正交的质量杠杆，混进来会污染演示。

这里有个新手必踩的坑，代码注释专门标了：

```typescript
// query-time 和 index-time 的 tokenize 必须完全一致，
// 否则某个 query term 永远匹配不上它在索引里的形态。
function tokenize(text: string): string[] { ... }
```

**索引时和查询时必须用同一个 tokenize 函数**。如果索引时小写化了、查询时没有，那 `Vector` 这个 query 永远命中不了索引里的 `vector`——召回直接掉一截，而且静默无报错。这是个不变量（invariant），不是优化建议。真实系统里 analyzer 配置不一致（索引用 standard analyzer，查询用 keyword analyzer）是 Elasticsearch 召回莫名其妙偏低的头号原因之一。

## 二、BM25：把"相关"量化成一个能排序的分数

有了候选集，得给每篇打个分排序。BM25（Best Matching 25，一个 1990 年代的概率检索打分函数，至今是 Lucene/ES 的默认）的完整公式，对每个 query term 求和：

```
score(doc) = Σ_t  idf(t) · ( tf · (k1 + 1) ) / ( tf + k1 · (1 - b + b · dl/avgdl) )
```

看着唬人，拆成三个独立零件就清楚了。每个零件都在修正一个朴素 tf·idf 的具体毛病。

### 零件一：idf —— 稀有词才有信息量

**idf（inverse document frequency，逆文档频率）**——大白话：一个词越少见，命中它越值钱。出现在每篇文档里的词（the、of）等于没说，出现在 1 篇里的词是强信号。

stage01 用的是 Lucene 那版带平滑的概率型 idf：

```typescript
function idf(term: string, index: InvertedIndex): number {
  const df = index.postings.get(term)?.size ?? 0;
  if (df === 0) return 0; // 词不在库里 -> 不贡献
  const n = index.docCount;
  return Math.log((n - df + 0.5) / (df + 0.5) + 1);
}
```

`df`（document frequency）就是 postings 列表的长度。为什么不用教科书的 `ln(N/df)`？因为那个公式在某个词出现在**全部** N 篇时会算出负数（`ln(0.5/(N+0.5)) < 0`），意味着常见词会**倒扣**分数——荒谬。`+0.5` 平滑 + 外层 `+1` 保证 idf 永远 ≥ 0。这不是数学洁癖，是踩过坑的设计。

真实跑出来的 idf 分布，rare term 确实拿到高权重：

```
[idf] specificity weights (rare term = higher idf):
  vector        df=4  idf=1.7918
  search        df=8  idf=1.1558
  index         df=7  idf=1.2809
  quantization  df=1  idf=2.8904
  cosine        df=1  idf=2.8904
```

`quantization` 只在 1 篇出现，idf=2.89；`search` 在 8 篇出现，idf 只有 1.16。同样命中一次，命中 `quantization` 贡献的分数是命中 `search` 的 2.5 倍。**这就是 idf 在替你做的事：自动判断哪个 query term 更值得信。** 注意这套权重是从 26 篇玩具语料算的，df 都是个位数所以 idf 颗粒很粗；在亿级语料上同样的公式给出的是平滑连续的权重谱,相对关系完全一致。

### 零件二：k1 —— tf 饱和，堵死关键词堆砌

朴素想法：一个词在文档里出现越多次越相关。但**线性叠加会被关键词堆砌（keyword stuffing）攻破**——把 `vector` 复制 100 遍的垃圾页会碾压一篇正经讲 vector 的文章。

BM25 用 `k1` 让 tf **饱和（saturate）**：看公式里的分式 `tf·(k1+1) / (tf + k1·...)`，当 tf → ∞ 时它趋近上限 `(k1+1)`。也就是说，**一个词无论重复多少次，单词贡献有天花板** `idf·(k1+1)`。第 1 次出现加很多分，第 10 次几乎不加分。`k1` 控制饱和速度，Lucene 默认 1.5。

### 零件三：b —— 长度归一化，这是本章的重头戏

长文档天然包含更多词，raw tf 被它的体量虚高了。一篇 5000 字的综述提到 `vector` 三次，和一篇 50 字的精准定义提到 `vector` 三次，含金量天差地别。

`b` 就是长度归一化（length normalization）的强度旋钮，取值 [0, 1]。看公式分母里的 `(1 - b + b·dl/avgdl)`：
- `b = 1`：完全按长度惩罚。一篇是平均长度两倍的文档，它的 tf 被打对折。
- `b = 0`：长度因子整项消失，**完全无视文档长度**。
- Lucene 默认 `b = 0.75`：惩罚但不极端。

### 三个零件合起来：先看正常工作的样子

query `vector index search`，默认参数（k1=1.5, b=0.75）：

```
--- Query: "vector index search" (k1=1.5, b=0.75) ---
Top 5 (BM25 default):
  #1  score=6.0030  [doc 0] "What is a vector index"
  #2  score=3.5522  [doc 18] "Hybrid search"
  #3  score=3.2602  [doc 1] "Inverted index basics"
  #4  score=2.7340  [doc 15] "Sharding a search engine"
  #5  score=2.4051  [doc 10] "HNSW graph index"
```

排第一的是标题就叫 "What is a vector index" 的短文档——它三个 query term 全命中、且短，长度归一化没惩罚它。这正是我们想要的。语料里那篇 83 token 的 padding 长文档（标题 "PADDING long survey..."，把 query 词夹在一堆无关计算机话题里）**没进 top 5**。

saturation + 长度归一化联手压制 padding 文档的效果，代码直接对比两篇打分：

```
[saturation] same query terms, two docs, default params:
  padding doc (len 83): score=1.6433
  focused doc (len 17): score=6.0030
```

83 token 的 padding 文档只拿到 1.64，17 token 的精准文档拿到 6.00——差 3.6 倍。**注意 padding 文档其实命中了 query 词，但 b=0.75 的长度惩罚 + tf 饱和让它的"靠体量蹭词"策略失效了。** 这是 BM25 设计精妙的地方：它不只看命中没命中，它看"命中的密度"。

## 三、失败模式：关掉长度归一化，长文档霸榜（一个旋钮复现）

现在把 `b` 从 0.75 调到 0，其他不动。这是最值得刻进直觉的 BM25 失败模式，因为它**只需要拨一个旋钮就能从"正常"翻转到"崩坏"**：

```
--- FAILURE MODE: same query, b=0 (length normalization OFF) ---
Top 5 (b=0, BROKEN):
  #1  score=5.8503  [doc 0] "What is a vector index"
  #2  score=4.2285  [doc 25] "PADDING long survey of many unrelated computing systems"  <-- long padding doc
  #3  score=3.4429  [doc 18] "Hybrid search"
  #4  score=3.2907  [doc 1] "Inverted index basics"
  #5  score=2.4367  [doc 8] "IVF inverted file index"
```

padding 长文档冲到了 **#2**。代码把损伤量化得很清楚：

```
[failure-mode summary]
  padding doc rank with b=0.75 (correct): not in top 5
  padding doc rank with b=0    (broken):  #2
  padding doc score b=0.75 -> b=0: 1.6433 -> 4.2285 (inflated 2.57x by removing the length penalty)
```

同一篇 padding 文档，分数从 1.64 暴涨到 4.23，**虚高 2.57 倍**——纯粹因为拿掉了长度惩罚。它没变得更相关，它只是"长"这件事不再被扣分了。

**为什么这个失败模式重要，而不只是个调参问题？** 因为它揭示了 BM25 相关性的本质：相关 ≠ 命中了词，相关 = 在合适的篇幅里以合适的密度命中了词。`b` 是这个引擎里**把"啰嗦"和"相关"区分开的那个零件**。关掉它，啰嗦就被当成了相关。这在生产里的真实对应是：有人觉得"归一化让我的长文档排不上去"，手贱把 b 调到 0 或很小,结果就是 SEO 垃圾页、自动生成的长 FAQ、把关键词塞满的 landing page 集体上位。**当你的 BM25 召回突然被一堆又长又水的文档占领，第一件事就是去查 b。**

> 这个失败模式之所以能干净地复现，是因为 stage01 的 padding 文档被刻意调过：每个 query 词只出现约 3 次、其余全是无关填充。重复再多一点，saturation+idf 会让它在 b=0.75 下也能赢，演示就变成"怎么调都坏"而不是"b 控制的翻转"。这也顺带说明了 BM25 三个零件是**协同**的——单独看 k1 或单独看 b 都不完整。

## 四、根本死穴：词表不重叠 = 零召回（这就是下一章的全部理由）

前面的失败模式还是"参数调错"，能修。下面这个是 BM25 **修不了**的——它是这套范式的天花板。

BM25 的全部信号来自"query 词和文档词的字面重叠"。**query 里出现的词，如果文档里一个都没有，分数恒为 0，文档根本进不了候选集。** 它不懂同义、不懂改写、不懂语义。

stage01 用两个 query 演示这道裂缝。第一个是语义意图明确、但精确关键词命中很少的 query：

```
--- Query: "find similar items quickly" (semantic intent, few exact keyword hits) ---
Top 5 (BM25 default):
  #1  score=6.3134  [doc 12] "Embeddings overview"
  #2  score=2.8529  [doc 10] "HNSW graph index"
```

它居然蒙对了——但靠的是 query 里 `similar` / `items` 之类的词碰巧和文档字面重叠，运气成分大。换一个纯粹词表不重叠的 query，立刻原形毕露：

```
--- Query: "automobile photograph" (pure vocabulary mismatch) ---
  matched 0 docs. BM25 cannot bridge "automobile" -> "vector";
  this zero-recall-on-synonyms gap is what stage 02+ embeddings fix.
```

**零文档命中。** 用户想找的可能就是讲图像向量、讲 embedding 的那几篇，但因为他用了 "automobile" / "photograph" 而文档里写的是 "vector" / "image"，BM25 两手一摊。这不是排序排差了，是**召回直接为 0**——再好的 reranker 也救不回一个空的候选集。

这道"vocabulary mismatch（词表不匹配）→ 零召回"的裂缝，就是 dense vector（稠密向量）检索存在的**全部理由**。embedding 把文本映射进一个连续语义空间——大白话：意思相近的句子，向量距离也近——于是 "automobile" 和 "car" 即使一个字符都不重叠，向量也挨得近。第 02 章会从相似度度量（cosine / 点积）讲起，正式把这道裂缝补上。

> ⚡ **开放问题：lexical 和 semantic 谁也替代不了谁，最优融合至今无通用解。** 你可能想"那向量这么强，直接全用向量不就行了"——错。向量检索在反方向同样有死穴：精确 token（错误码 `E1042`、函数名 `__init__`、SKU、罕见专名）embedding 经常糊成一团，BM25 反而一击命中。所以工业界（pgvector、Milvus、Elasticsearch 的 hybrid、Vespa）的答案是 **Hybrid（混合检索）**：BM25 和向量各跑一路、分数融合（RRF / 加权 / learned fusion）。但"两路分数怎么融最优"——固定权重、RRF、还是针对 query 类型动态路由——**目前没有跨数据集普适的最优解，仍是活跃研究方向**。本书第 06 章会自己造一个 Hybrid 引擎,亲手把这两半缝回去,并直面"融合权重怎么定"这个没有标准答案的问题。BM25 不是被向量取代了,它是 Hybrid 的另一半。

## 五、一个容易被忽略的工程属性：BM25 是确定性的

最后一点,在你后面对比向量检索时会反复用到:**BM25 的排序是确定性的**。它对每个 term 的贡献求和(加法可交换),最终排序是 `(score, docId)` 上的全序。query 词换个顺序,结果逐字节相同。stage01 用种子化 PRNG 做 Fisher-Yates 洗牌打乱 query 词序后重跑,验证了这点:

```
[determinism] re-ranking with shuffled query-term order identical to original: YES
```

为什么强调它?因为后面 HNSW(第 04 章)是 **近似(approximate)** 算法——同一 query 多跑几次、换个构建顺序,top-k 可能不同。届时你需要一个确定性的基线来量度"近似到底损失了多少 recall"。**BM25 / 暴力精确检索(第 03 章)就是那把不动的尺子。** 这也是为什么本书要先建立一个老实、可复现、可解释的基线,再去碰那些用确定性换速度的花哨索引。

## 本章 takeaways

直接引 stage01 的三条结论,它们也是这本书后续每一章的出发点:

1. **倒排索引 = 只碰共享 query 词的文档(sublinear)。** 检索能 scale 的根本不是算得快,是碰得少。
2. **k1 让 tf 饱和,b 归一化长度。b=0 时长文档作弊。** 相关 ≠ 命中词,相关 = 合适篇幅里合适密度地命中词。
3. **BM25 在词表不重叠时召回为 0 → 这就是 embedding 存在的理由。** 下一章(第 02 章)从相似度度量讲起,把这道裂缝补上;第 06 章再用 Hybrid 把 lexical 和 semantic 缝回一个引擎。

要亲手验证本章每个数字,跑配套仓的 `examples/vector-search-from-scratch/src/stage01-bm25.ts`——改 `b`、改 `k1`、换 query,看排序怎么翻转。**这本书的每一章都配一个能跑、能改旋钮、能复现失败模式的 stage,这是它和 AI 批量生成的 survey 稿的根本区别。**
