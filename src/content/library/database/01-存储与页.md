先看一个数字:一个 4096 字节的页,塞进 102 行长度在 4 到 60 字节之间的变长行后,有用 payload 只占了 76.9%。剩下的 23% 去哪了?7 字节页头、408 字节 slot 目录(102 × 4)、23 字节填不下下一行的空隙,再加上每行 payload 之外的 int 列和 varint 长度前缀。这 23% 不是浪费,是 slotted page(带 slot 目录的页)为了让变长行能原地变长、能删除、能保号而付的税。这一章就是把这笔税算清楚——为什么必须交、交多少、什么时候交不起。

如果你来自应用层,习惯了 `INSERT` 一行就有一行,这一章会让你看到那行字在物理上经历了什么:它被编码成字节,从页的尾端往上找一块空地写进去,然后页头里多了一个 4 字节的指针指向它。这个指针——我们叫 slot——是上层 B+树索引的命脉。本章的核心论点只有一句:**行可以挪,slot 不能动**。把这句话理解透,后面三章的设计取舍就都有了根。

## 为什么要先假装有一块磁盘

整本书讲的是一件事:怎么少做 IO(磁盘读写)。如果我们把行存进一个 JavaScript 数组,那么缓冲池命中率、B+树扇出(fanout,一个节点能指向多少子节点)对树高的影响、WAL 的 fsync 代价——这些全书最有意思的数字会全部蒸发,因为数组没有"页"这个概念,也没有"读一页要花一次 IO"这个约束。

所以我们模拟磁盘强加给我们的唯一一件事:数据按固定大小的页存放,你用页号寻址,读写的最小单位是整页。`core/disk.ts` 就是这块假磁盘,用一整块 `ArrayBuffer`(JS 里的连续二进制内存块)当后备存储:

```ts
// examples/database-from-scratch/src/core/disk.ts
export const PAGE_SIZE = 4096;

export class Disk {
  private buf: ArrayBuffer;
  private bytes: Uint8Array;
  // ...
  readPage(pageId: number): Uint8Array {
    this.checkId(pageId);
    this.reads++;
    return this.bytes.slice(pageId * PAGE_SIZE, (pageId + 1) * PAGE_SIZE);
  }
  writePage(pageId: number, buf: Uint8Array): void {
    this.checkId(pageId);
    if (buf.length !== PAGE_SIZE) {
      throw new Error(`writePage: expected ${PAGE_SIZE} bytes, got ${buf.length}`);
    }
    this.writes++;
    this.bytes.set(buf, pageId * PAGE_SIZE);
  }
}
```

注意三个故意的设计,每一个都对应真实磁盘的一条物理约束:

**`readPage` 返回的是拷贝(`.slice()`),不是视图。** 真磁盘读出来的是一份独立的数据,你改它不会改到盘上。如果这里返回 `subarray`(共享底层内存的视图),调用方一改,假磁盘的内容就被偷偷改了,而 `writes` 计数器还是 0——这是一种"幽灵写入",排查起来极其痛苦。第 02 章的缓冲池整个建立在"`readPage` 给我一份我能随便改的副本"这个假设上。

**`writePage` 把字节拷贝进去(`bytes.set`),且强制必须是整页。** 短写(只写半页)在真硬件上叫 torn page(撕裂页),是断电时最经典的损坏。这里直接 throw,拒绝静默接受半页。注意每次 `readPage`/`writePage` 都 `this.reads++`/`this.writes++`——**这些计数器是全书的主要度量**,后面所有"快了多少"都是用 IO 次数说话,不是用墙钟时间(墙钟时间机器一换就变,IO 次数是可迁移的)。

为什么用一整块 `ArrayBuffer` 而不是"一个页一个 buffer 的数组"?因为真磁盘就是一条线性的地址空间。把它建模成一整块连续内存,"页号就是偏移量"这个心智模型才诚实:页 0 在字节 `[0, 4096)`,页 5 在 `[20480, 24576)`。扩容时也只是一次 memcpy(`grow()` 里把旧 buffer 拷进两倍大的新 buffer),而不是往数组里 push。

⚡ **这里有个本章不碰、但值得标注的开放设计点:`PAGE_SIZE = 4096` 是硬编码的常数。** 为什么是 4096?因为它对齐了绝大多数操作系统的内存页和磁盘扇区,一次 IO 正好搬一个 OS 页,不浪费。但"页该多大"在分析型数据库里至今没有通用解:OLTP(在线事务,小读小写)偏好 4-8KB 的小页降低读放大,OLAP(分析,大范围扫描)偏好 256KB 甚至 1MB 的大页摊薄寻址开销,列存引擎里"页"的概念甚至被 row group(几万行一组)取代。把 PAGE_SIZE 写成全局常数是 toy 简化;真实引擎里它是个需要按负载调的参数,而"为混合负载选一个页大小"——目前无通用解,仍是研究和工程权衡的活题。我们在本章末尾的前沿钩子会再回到这个点上。

## slotted page:页内的三段地形

拿到一块 4096 字节的空白页,我们要在里面塞变长行。最朴素的做法是行紧挨着行从头排:第一行写在偏移 7(页头之后),第二行接着第一行尾巴写,以此类推。这种"紧凑数组"布局对定长行没问题,但对变长行——更重要的是对**删除和变长**——是灾难。我们稍后会论证为什么。先看 slotted page 怎么排:

```
[ header(7B) | slot 目录从这往下长 -> | <- 自由空隙 -> | <- 行堆从页尾往上长 ]
  偏移 0       偏移 7                                                偏移 4096
```

页头只有 7 字节(`SLOTTED_HEADER.HEADER_SIZE = 7`),定义在 `core/page.ts`:

```ts
// examples/database-from-scratch/src/core/page.ts
export const SLOTTED_HEADER = {
  TYPE: 0,        // u8  PageType (META/LEAF/INTERNAL/OVERFLOW)
  SLOT_COUNT: 1,  // u16 目录里有几个 slot
  FREE_START: 3,  // u16 下一个 slot 条目该写的偏移(向下长)
  FREE_END: 5,    // u16 已写入的最低一个行字节的偏移(向上长)
  HEADER_SIZE: 7, // slot 目录可用的第一个字节
} as const;

export const SLOT_ENTRY_SIZE = 4; // 每个 slot 条目 = (offset:u16, length:u16)
```

关键在于**两端对生长**:slot 目录从页头往下长(`FREE_START` 递增),行堆从页尾往上长(`FREE_END` 递减),它们在中间相遇,中间那段就是自由空隙。自由空间 = `FREE_END - FREE_START`,这就是两端争夺的那块地:

```ts
// examples/database-from-scratch/src/stage01-slotted-page.ts
function freeSpace(r: PageReader): number {
  return r.readU16(SLOTTED_HEADER.FREE_END) - r.readU16(SLOTTED_HEADER.FREE_START);
}
```

一个 slot 是 `(offset:u16, length:u16)`——一行在页内的偏移和长度。`length === 0` 是墓碑(tombstone,删除标记),稍后讲删除时细说。

### ✦ 为什么是"slot 目录 + 间接寻址",不是行紧挨着存

这是本章最该想透的一个设计决策,也是 slotted page 存在的全部理由。直接说结论:**让行能原地变长、能删除而不挪动它的 tid(tuple id,行标识),因为 tid 稳定是上层索引的命脉。**

展开讲。假设我们不要 slot 目录,行紧挨着从头排。现在 B+树的某个叶子节点里存了一条索引:"key=Alice 对应的行在页 5 的偏移 312"。一切正常,直到:

1. **有人删了页 5 里偏移 312 之前的某一行。** 紧凑数组要回收空间,就得把后面所有行往前挪。Alice 的行从偏移 312 挪到了 280。但 B+树里那条索引还指着 312——现在它要么指向半行垃圾,要么指向另一行。**所有指向这一页的索引全部失效**,因为行一动,偏移就变了。

2. **有人把 Alice 的 payload 从 20 字节改成 40 字节。** 原地放不下,得把它挪到页里有空地的地方。同样,偏移变了,索引失效。

slot 目录用一层间接寻址解决了这个:索引不再记物理偏移,而是记 **slot 号**(逻辑标识,即 tid)。"key=Alice 对应页 5 的 slot 7"。slot 7 是目录里一个固定位置的 4 字节条目,里面记着 Alice 的行当前在哪。行在页里挪到天涯海角,我们只改 slot 7 的 offset 字段;slot 号本身永远是 7。**物理位置可变,逻辑标识不变**——这就是间接寻址买来的东西。

代价是每行多 4 字节(一个 slot 条目)和一次额外的指针跳转(读行要先读 slot 拿到偏移,再去读行)。在我们的实测里,这 4 字节 × 102 行 = 408 字节,占了页的 10%。值不值?值,因为没有它,上层索引根本没法稳定地指向一行。这笔税是 OLTP 数据库愿意一直交的——直到列存场景下它变得难以承受,我们在前沿钩子里会看到。

(顺带:不只本章的页用这套布局。第 03 章的 B+树叶子节点存行的方式和这里一模一样——同一套 slot 目录代码会被复用。slotted page 不是一个孤立的 demo,是贯穿存储层的统一原语。)

## 插入:两端各让一步

插入一行的成本不是行本身的字节数,而是**行 + 一个新 slot 条目(4 字节)**。忘掉 slot 这 4 字节是 slotted page 最经典的 off-by-four(差 4 字节)bug:你以为行能塞下,结果它的目录条目把空隙撑爆了。`insertRow` 把这个算清楚:

```ts
// examples/database-from-scratch/src/stage01-slotted-page.ts
function insertRow(bytes: Uint8Array, row: Uint8Array): InsertResult {
  const r = new PageReader(bytes);
  const need = row.length + SLOT_ENTRY_SIZE; // 行 + slot 条目
  const have = freeSpace(r);
  if (need > have) {
    // 页满:不是 bug,是上层去申请新页的信号。在改任何东西之前返回,
    // 保证页处于一致状态可供重试。
    return { ok: false, reason: "page-full", needBytes: need, haveBytes: have };
  }

  const w = new PageWriter(bytes);
  const freeStart = r.readU16(SLOTTED_HEADER.FREE_START);
  const freeEnd = r.readU16(SLOTTED_HEADER.FREE_END);
  const rowOffset = freeEnd - row.length; // 行堆向下长:新行落在当前前沿下方
  w.writeBytes(rowOffset, row);

  const slotIndex = slotCount(r);
  const slotBase = SLOTTED_HEADER.HEADER_SIZE + slotIndex * SLOT_ENTRY_SIZE;
  w.writeU16(slotBase, rowOffset);     // slot.offset
  w.writeU16(slotBase + 2, row.length); // slot.length

  w.writeU16(SLOTTED_HEADER.SLOT_COUNT, slotIndex + 1);
  w.writeU16(SLOTTED_HEADER.FREE_START, freeStart + SLOT_ENTRY_SIZE); // 目录又占一格
  w.writeU16(SLOTTED_HEADER.FREE_END, rowOffset);                     // 行堆又下探
  return { ok: true, slot: slotIndex };
}
```

行堆从 `FREE_END` 往下长:新行的家是当前前沿下方 `row.length` 个字节。写行,写 slot,然后两端各自推进——`FREE_START` 加 4(目录又占了一格),`FREE_END` 减 `row.length`(行堆又下探了)。

### 失败模式:页满不是异常,是返回值

这里有个看似小、实则定调全书的决策:**页满(page-full)用返回值表达,不用抛异常。**

直觉上你可能想:写到放不下时,让 `PageWriter` 的越界检查抛个异常,catch 住当作"满了"不就行了?不行,有两个致命问题:

1. **它混淆了"满"和"真 bug"。** `PageWriter.bounds()` 那个 throw 是最后一道防腐栏——它该在"代码算错了偏移、要写穿页边界"时炸,那是真 bug。如果你也用它表达"满了",那么一个越界 bug 就会被 catch 当成"页满,去开新页",bug 被静默吞掉,直到很久以后以"数据莫名其妙少了几行"的形式浮现。

2. **它会留下半改的页。** 如果你先写了行字节、推进了 `FREE_END`,然后在写 slot 时才发现越界抛异常——这时页头的 `FREE_END` 已经被改了,但 slot 没写成功,`SLOT_COUNT` 也没加。页处于一个不一致的中间状态(mutate-then-throw,改了再抛)。下一次操作就踩在这个烂摊子上。

正确做法是 `insertRow` **先算够不够(`need > have`),不够就在改动任何字节之前返回 `page-full`**,页原封不动。我们用实测证明这条"拒绝路径不改页"的不变量成立:故意插一个超大行被拒后,一个正常小行仍能插进去:

```
=== 失败模式: 大行无法存入单页 (overflow 页未实现) ===
尝试插入 payload=4085 字节的行 (编码后 4091 字节, 含 int 列 + varint 长度前缀)
空页可用空隙最多 4089 字节, 需要 4095 字节 (行 4091 + slot 4) => 返回 page-full, 页保持不变 (未损坏)。
拒绝大行后, 正常小行仍可插入 ✓ (拒绝路径不修改页, 不是 mutate-then-throw)。
```

读这几个数字:空页的最大空隙是 `4096 - 7 = 4089` 字节。一个 payload 4085 字节的行,编码后是 4091 字节(多出的 6 字节 = int 列 4 字节 + varint 长度前缀 2 字节),再加 4 字节 slot 条目 = 4095 字节,超过 4089。于是返回 page-full,页原封不动。

这暴露了 slotted page 的**硬边界:任何一行必须 ≤ 页大小 − 开销**。一行装不下一页怎么办?真实引擎用 **overflow 页链**:行头留在主页,超出部分溢出到专门的 OVERFLOW 页,用页内指针串成链。`core/page.ts` 里 `PageType.OVERFLOW = 3` 已经占了位,但本章故意不实现它。

这不是偷懒,是诚实。这个失败模式的价值在于:**它揭示了为什么 B+树必须支持 overflow / 大值外置(large value off-page storage)**——因为单页定长是存储的原子,而键值可以任意大。这个根因留给第 03 章解决。一本反 survey 的书,价值恰恰在于把"这里故意没做、为什么、谁来收尾"讲清楚,而不是假装一切都已完美。

## 删除:O(1) 打墓碑,空间先欠着

删一行,最诱人的做法是立刻把它的字节抹掉、把后面的行往前挪、回收空间。**别这么做。** 原因还是上面那条命根子:挪行就改偏移,改偏移就动 tid,动 tid 就废索引。

正确做法是**打墓碑**:把 slot 的 length 字段设成 0,标记"这行死了",但**一个字节都不挪**:

```ts
// examples/database-from-scratch/src/stage01-slotted-page.ts
function deleteRow(bytes: Uint8Array, slot: number): boolean {
  const r = new PageReader(bytes);
  if (slot < 0 || slot >= slotCount(r)) return false;
  const { length } = readSlot(r, slot);
  if (length === 0) return false; // 已经是墓碑,幂等
  const w = new PageWriter(bytes);
  const slotBase = SLOTTED_HEADER.HEADER_SIZE + slot * SLOT_ENTRY_SIZE;
  w.writeU16(slotBase + 2, 0); // length = 0 即墓碑标记
  return true;
}
```

这是 O(1) 操作——只改 2 个字节,不碰行堆,不动 `FREE_END`。空间是"逻辑上空了,物理上还占着"。读一个被墓碑标记的 slot 返回 null:

```ts
function getRow(bytes: Uint8Array, slot: number): Uint8Array | null {
  const r = new PageReader(bytes);
  if (slot < 0 || slot >= slotCount(r)) return null;
  const { offset, length } = readSlot(r, slot);
  if (length === 0) return null; // 墓碑
  return r.readBytes(offset, length).slice(); // 拷出一份(防 compaction 移动底层字节)
}
```

注意 `getRow` 返回的是拷贝(`.slice()`)。调用方可能在拿到行之后、用到它之前,触发了一次 compaction,把底层字节搬走了——如果返回视图,这份数据就成了悬空引用。和 `Disk.readPage` 返回拷贝是同一个道理:跨越生命周期的数据必须独立。

**为什么 delete 要 O(1) 而不顺手回收空间?** 因为删除在 OLTP 里是高频操作,如果每次删都要挪一堆行(O(n)),删一行的成本就和页里的行数挂钩了。把回收推迟到批量做(compaction),让删除保持廉价,这是典型的时间换时间的摊销:删快了,代价是空间暂时浪费、需要后续整理。

## compaction:回收墓碑,但 slot 号雷打不动

墓碑攒多了,页里全是死行占着的空洞。compaction(压实)就是把活行重新紧密排列、把死行的空间还给自由空隙。

这里有个实现上的细节,正好印证了全局规则里反复强调的一条——**带语义负载的数字,实现前要先问用户读到它会理解成什么**。compaction 要报告"回收了多少字节",但 delete 把 slot 的 length 设成 0 之后,页**自己已经不知道那行原来多长了**(length 被 0 覆盖了)。这是 O(1) delete 的代价。所以 `compact` 必须由调用方传入一份删除前的长度快照:

```ts
// examples/database-from-scratch/src/stage01-slotted-page.ts
function compact(bytes: Uint8Array, originalLengths: number[]): number {
  const r = new PageReader(bytes);
  const n = slotCount(r);

  // 1. 收集活行(slot 号 + 字节),保持 slot 顺序。先拷出来,因为马上要原地重写行堆。
  const live: { slot: number; row: Uint8Array }[] = [];
  let reclaimed = 0;
  for (let i = 0; i < n; i++) {
    const { offset, length } = readSlot(r, i);
    if (length === 0) {
      reclaimed += originalLengths[i]; // 墓碑:它原来的字节就是回收量
      continue;
    }
    live.push({ slot: i, row: r.readBytes(offset, length).slice() });
  }

  // 2. 从页尾往下重建行堆,就地改写每个存活 slot 的 offset。
  //    slot 号不变,只有它的 offset 变。这是外部行指针赖以生存的不变量。
  const w = new PageWriter(bytes);
  let frontier = PAGE_SIZE;
  for (const { slot, row } of live) {
    const newOffset = frontier - row.length;
    w.writeBytes(newOffset, row);
    const slotBase = SLOTTED_HEADER.HEADER_SIZE + slot * SLOT_ENTRY_SIZE;
    w.writeU16(slotBase, newOffset); // 只改 offset,length 不变
    frontier = newOffset;
  }
  w.writeU16(SLOTTED_HEADER.FREE_END, frontier);
  return reclaimed;
}
```

把整个删除 + compaction 的循环跑一遍——删掉每隔一个的 slot(0, 2, 4, ...),共 51 行,然后 compact——实测如下:

```
rows deleted                   51  every other slot
gap before compact (bytes)     23  O(1) delete, unchanged
bytes reclaimed              1810  by compaction
gap after compact (bytes)    1833  = before + reclaimed
compaction 回收 1810 字节, 空隙从 23 增长到 1833; 存活行 slot id 不变 (墓碑保号), 已删 slot 仍读为 null ✓
```

逐行读这四个数,每个都在验证一条不变量:

- **删除后空隙仍是 23,没变。** 这证明 delete 是 O(1)、没挪字节——删了 51 行,但那些行的物理空间还堵在原地,空隙纹丝不动。
- **回收 1810 字节,空隙从 23 涨到 1833**(`23 + 1810 = 1833`,精确对得上)。compaction 回收的字节数必须**恰好**等于被删行的总字节数,空隙必须**恰好**增长这么多。如果对不上,说明 compaction 漏算或重复计算了某些字节——代码里用 `assertEq` 把这两条钉死。
- **存活行 slot id 不变,已删 slot 仍读为 null。** 这是整章的命根子。compaction 把活行的字节全搬了家(offset 全变了),但**slot 号一个没动**。一个手里攥着"我的行是 slot 7"的上层索引,在 compaction 之后再去读 slot 7,读到的还是它那行——只是 slot 7 的 offset 字段被悄悄更新了。死掉的偶数号 slot 依然是墓碑,读出来还是 null,slot 号也保留着(墓碑保号),不会被复用——因为某个上层可能还攥着这个号,我们不能把它的指针从底下抽走。

这就是 ✦ 那条论点的完整闭环:**行可以挪,slot 不能动**。插入、删除、compaction 三个操作,没有一个会改变一个已分配 slot 的编号。这是 slotted page 用一层间接寻址向上层做出的契约,也是第 03 章 B+树敢于把 tid 当稳定指针存进索引的全部底气。

## 把字节算到最后一位:利用率的真相

slotted page 的空间到底花在哪?把一页填满后做一次字节核账,实测:

```
page size (bytes)         4096  fixed
rows accepted              102  of 200 attempted
header overhead (bytes)      7  7B fixed
slot dir (bytes)           408  102 x 4B
row heap (bytes)          3658  encoded rows
leftover gap (bytes)        23  < next row, wasted
row-heap utilization %    89.3  encoded/page
payload utilization %     76.9  useful/page
字节核账: header 7 + slot dir 408 + 行堆 3658 + 空隙 23 = 4096 (= 4096) ✓
```

这八行数据值得逐项咀嚼,因为它们就是 slotted page 的"成本结构":

- **尝试插 200 行,只收下 102 行**——页在 102 行处满了。这个数字是确定性的(用固定种子 seed=42 的伪随机数决定每行长度),换机器重跑也是 102,因为它纯粹由布局算术决定。
- **header 7 + slot dir 408 + 行堆 3658 + 空隙 23 = 4096**,一个字节不多不少。这个等式是用 `invariant` 强制的:核账对不上,说明布局算术漂了,立刻报警。这种"每个字节都有去处"的核账,正是把存储层当真而不是当玩具的标志。
- **两个利用率,差在哪。** 行堆利用率 89.3%(编码后的行字节 / 页),payload 利用率 76.9%(纯有用数据 / 页)。这 12.4 个百分点的差距,就是编码开销:每个 string 列前面的 varint 长度前缀、每行的 int 列。slotted page 本身的开销则是 408 字节 slot 目录(10%)+ 7 字节页头 + 23 字节填不下下一行的边角料。
- **23 字节空隙是"碎片税"。** 它太小,装不下下一行(下一行至少要几十字节 + 4 字节 slot),所以永远空着。变长行打包必然产生这种边角料——这是变长存储相对定长存储的固有代价,定长存储能把页填到几乎 100%,但代价是定长本身的浪费(短数据也得占满字段宽度)。

⚠ 一句必须标注的话:**这些字节数是确定性的、可信的;但下面要讲的吞吐量数字是估算性质的,绝对值偏乐观。** 看吞吐:

```
op                 ns/op   ops/sec    iters
-----------------  -----  --------  -------
insert (measured)   93.7  10668184  1000000
get (measured)     106.4   9401839  1000000
注: ops/sec 为真实墙钟实测 (machine-dependent), 跑跑会变; 这是单页纯内存操作, 绝对值偏乐观 (无磁盘 IO/无并发), 可迁移的是相对量级与 insert vs get 的比值。
```

insert 约 94 纳秒/次,get 约 106 纳秒/次。**这是墙钟实测,机器一换、JIT 一热身就变,绝对值别当真。** 而且这是单页纯内存操作——没有磁盘 IO、没有并发、没有锁,所以快得不真实。真实引擎里一次 get 可能要先去缓冲池查页(第 02 章)、可能要走 B+树查 slot(第 03 章)、可能要等锁(第 06 章)。

但有一个**可迁移的信号**藏在里面:**get 比 insert 略慢(106 vs 94)。** 这反直觉——读不是该比写快吗?在这里不是,因为 get 要 `.slice()` 把行字节拷贝出来(给调用方一份稳定副本),而 insert 只是往空地写、不拷贝。这个"读要付拷贝税"的相对关系,换机器也成立,因为它是算法决定的,不是硬件决定的。**读 benchmark 时永远先看相对量级和比值,绝对值留给 profiler 去较真。**

## ⚡ 前沿:slotted row-page 在分析负载下的缓存噩梦,以及 PAX / zone map

到这里我们造出的是一个 **row-page**(行式页):一行的所有列连续存在一起。这对 OLTP 完美——你要操作整行(读一个订单、改一个用户),整行在一起,一次读全拿到。

但把它丢进分析型查询,缓存友好度会塌方。考虑一个典型 OLAP 查询:`SELECT AVG(price) FROM orders`。它只要 price 一列,但在 row-page 里,price 和 order_id、customer、address、timestamp 全挤在一行里。CPU 缓存行(cache line,一次从内存搬进缓存的最小单位,通常 64 字节)一次搬进来 64 字节,里面可能就一个 price 是有用的,其余几十字节是你这次根本不看的列。结果:你为读 1 列的数据,污染了整个缓存,有效带宽利用率可能低到个位数百分比。row-page 对"只扫几列、扫很多行"的分析负载是结构性地不友好。

业界有两条应对路线,都值得知道:

**第一条:列存(columnar storage)。** 干脆把同一列的所有值连续存在一起——price 一个块、customer 一个块。扫 price 时一个缓存行装满 price,带宽利用率拉满,还顺便对 SIMD(单指令多数据,CPU 一条指令处理一批数据)和压缩极其友好(同列数据相似,压得狠)。代价是要重组一整行就得跨好几个列块去拼,OLTP 的点查变贵。这是 row vs column 的根本张力。

**第二条:PAX(Partition Attributes Across,混合布局)。** 这是个更精巧的折中——在**一个页内部**按列分区。页还是页(对上层、对缓冲池透明,仍是 4096 或更大的一块),但页里不是行挨行,而是先放第一列的所有值,再放第二列的所有值。这样跨页的 IO 局部性保留了 row-store 的好处(一个页里有完整的几行),页内的扫描又有了 column-store 的缓存友好(扫一列时该列在页内连续)。PAX 出自一篇 2001 年的经典论文(Ailamaki 等),它的思想至今活在很多分析引擎的页布局里。

**第三条(正交的加速):zone map(区域图,又叫 min/max index)。** 为每个页(或 row group)记一组轻量元数据:这个页里 price 列的最小值和最大值。查询 `WHERE price > 1000` 时,先看 zone map——如果某个页的 price 最大值才 800,**整页跳过,连读都不用读**。这是分析引擎少做 IO 的看家手段之一,代价极小(每页几个数字),收益在数据有局部性(比如按时间排序、按地域聚集)时巨大。

把这些放回本章的坐标系:我们做的 slotted row-page 是 OLTP 的正确选择,也是后面 B+树的物理基础,但它**故意没有为分析负载优化**。**"如何在一个存储引擎里同时把 OLTP 和 OLAP 服务好"——HTAP(混合事务/分析处理)——至今没有通用解。** 列存做不好点查更新,行存做不好大范围扫描,PAX 是折中但不是免费午餐(更新一行要碰页内多个列区),很多系统干脆用两套存储 + 后台同步(把数据从行存复制到列存)来回避这个矛盾,而那又引入了同步延迟和一致性问题。这是个仍在活跃研究、且工程上每家都在用不同 trade-off 凑合的开放问题。本章把行式页的边界划清楚,正是为了让你看见这条边界之外那片没有标准答案的地。

## 这一章给后面铺了什么

回头看,我们用 `core/disk.ts` 的 `ArrayBuffer` 造了一块只能整页读写、每次 IO 都计数的假磁盘;用 `core/page.ts` 的 slotted-page 布局,在单页里实现了变长行的插入、读取、O(1) 删除、和 compaction;用实测把每个字节、每条不变量都钉死了。

最该带走的两句话:

1. **slot 目录这层间接寻址,是为了让行的物理位置可变、而逻辑标识(tid)不变。** 这是上层索引敢把 tid 当稳定指针的全部基础。插入、删除、compaction 三个操作,没有一个动过已分配 slot 的编号。
2. **单页是存储的原子,它有一条硬边界:行 ≤ 页 − 开销。** 我们故意让大行撞上这条边界(返回 page-full,不实现 overflow),因为这条边界正是后面 B+树必须处理 overflow 和大值外置的根因。

第 02 章会把"页"包进**缓冲池**:磁盘上有成千上万页,内存只放得下一小部分,缓冲池决定哪些页留在内存、哪些被换出去,而本章那个被反复强调的 IO 计数器,正是衡量缓冲池好坏的尺子。第 03 章再把"页"当作 **B+树节点**的物理载体,你会看到本章的 slot 目录原封不动地用在叶子节点上,而 tid 稳定这条不变量,会成为整棵树能正确工作的前提。
