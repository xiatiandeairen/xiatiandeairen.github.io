崩溃恢复的本质难题是这个:断电那一刻,内存里的事务状态(谁提交了、哪页是脏的、回滚段在哪)**全部蒸发**,留给你的只有两样东西——一个被 fsync 截断在某个时刻的数据文件,和一条 append-only 的日志。你要从这两样残骸里,重建出一个谁都同意的「正确状态」:所有已提交事务的修改一个不少,所有未提交事务的痕迹一个不留(原子性 + 持久性)。

第 05 章我们立了规矩:WAL(write-ahead log,预写日志)+「先把日志 fsync 到盘,才允许数据页落盘」。但一条日志只有在断电后**真能用来重建真相**时,它那笔 fsync 开销才花得值。这一章就是把那张空头支票兑现。

参照系是 ARIES——IBM 1992 年那篇,几乎所有商业数据库(DB2、SQL Server、Postgres 的 WAL replay)的恢复都是它的变体。我们实现它的三阶段骨架 Analysis → Redo → Undo,配套代码在 `examples/database-from-scratch/src/stage07-recovery.ts`,可以直接 `npm run stage07` 跑。本章所有数字来自 `seed=7` 那次真实运行,不是我编的。

## 一、崩溃现场:你拿到的到底是什么

先把「恢复的输入」摆清楚,否则后面三阶段全是空中楼阁。

第 05 章的 `Disk` 用 `crashAt: N` 模拟断电:第 N 次 fsync **会先把前面的写落盘,然后抛 `CrashError`**——精确模拟「这次 fsync 没完成,机器死了」。看 `core/disk.ts` 里 fsync 怎么定义这条持久性边界:

```ts
fsync(): void {
  this.fsyncs++;
  if (this.crashAt !== undefined && this.fsyncs === this.crashAt) {
    // Commit what we had, then "lose power".
    throw new CrashError(this.fsyncs);
  }
  this.durable.set(this.bytes.subarray(0, this.pageCount * PAGE_SIZE));
}
```

关键在 `durable` 这个快照:只有成功完成的 fsync 才会把活跃字节复制进去。崩溃后,stage07 调 `reopenFromDurable()` 重开文件,看到的只有**最后一次成功 fsync 落下的字节**——之后写进易失内存的全没了。这是恢复算法的悲观假设:别指望任何「崩溃前那一瞬的写」侥幸活下来。

stage07 故意把这个假设推到极端。`reopenAfterCrash()` 的做法是:先写一份 baseline(4 个账户全 = 100)并 fsync(这是「崩溃前最后一次好 fsync」),然后整个 workload 的写**全部发生在那次 fsync 之后**。于是重开数据文件时:

```
重开数据文件, 起点计数: [100, 100, 100, 100] (仅 baseline 持久, workload 写入全丢)
```

为什么要这么狠?因为这是 Redo 最强的测试。真实的「torn page(撕裂页)」崩溃会侥幸保住一部分写,那只会让 Redo 的活更轻。我们让数据页**一点 workload 都没存下来**,逼 Redo 从零把所有已提交的写重新落盘——它要是能扛住这种全丢,真实场景就更没问题。这是个 toy 规模(5 事务、1 数据页)的刻意选择,为的是 trace 可读;能迁移到真引擎的是**形状**,不是绝对数字。

那日志里有什么?seed=7 跑出来的持久化 WAL 是恢复**唯一**的输入:

```
lsn  txn  type    account  before  after
  1  T0   update  a0       100     102
  2  T4   update  a3       100     199
  3  T1   update  a1       100     107
  4  T2   update  a2       100     103
  5  T3   update  a1       107     157
  6  T3   update  a2       103     143
  7  T0   update  a0       102     107
  8  T0   commit
  9  T1   commit
 10  T2   commit
```

10 条记录,T0/T1/T2 有 commit 记录(winners,已提交),T3/T4 没有(losers,崩溃时进行中)。注意每条 update 都同时带 `before` 和 `after` 两个镜像——这是本实现相对真 ARIES 最大的简化,后面专门讲它为什么让恢复变得几乎 trivial。

### 为什么日志记 before + after 两个镜像

LogRecord 的结构(`src/stage07-recovery.ts`):

```ts
interface LogRecord {
  readonly lsn: number;      // Log Sequence Number, 严格递增的日志位置
  readonly txnId: number;
  readonly type: "update" | "commit" | "abort";
  readonly pageId?: number;
  readonly account?: number;
  readonly before?: number;  // 撤销要它
  readonly after?: number;   // 重做要它
}
```

这叫 **physiological(物理逻辑)日志**——大白话:对页是物理的(精确到字节偏移),对页内是逻辑的。它同时带 before-image(改前的值)和 after-image(改后的值),于是:

- **Redo = 盲写 after-image**。不管页现在是什么样,直接覆盖成 after。这让 Redo 天然**幂等**(idempotent,重复执行结果一样)——一条已经生效过的记录再重放一遍,无害。
- **Undo = 盲写 before-image**。把那个 slot 改回改前的值。

真 ARIES 没这么奢侈。它的日志记录通常只存「差异」或逻辑操作(比如「这个 B+ 树节点插了一个 key」),Redo 要重新执行那个逻辑操作,Undo 要从一串部分记录里**重构**出 before-image,还要靠每页的 `pageLSN`(页上记的最后一条作用于它的日志序号)来判断「这条记录是否已经生效过、要不要跳过」。我们把这套全省了,因为本章要讲清楚的是**三阶段的骨架和它为什么正确**,不是日志压缩。代码注释里明确把这点 call out 了,不藏着——这是教材该有的诚实。

> **失败模式**:如果 before 和 after 在内存里**别名**到同一段字节(aliasing),会发生什么?那么 update 之后 before-image 会被悄悄改成等于 after,Undo 就成了空操作——回滚个寂寞,这正是「丢失回滚」这类最阴险的恢复 bug。stage07 的 `withCounter()` 用 `page.slice()` 拷贝整页再改,就是为了根绝这个别名。注释原话:别名会让 before==after,Undo 变 no-op。

## 二、Phase 1 — Analysis:只读日志,不碰数据页

恢复第一步**绝不**碰数据文件。它只扫日志,在内存里重建两张表:

- **ATT(Active Transaction Table,活跃事务表)**:扫到 update 就把该事务加进来,扫到它的 commit 就移出去。扫完还留在表里的——就是 losers,待回滚。
- **DPT(Dirty Page Table,脏页表)**:哪些页被 update 碰过,Redo 需要重新检查它们。

核心循环(`analyze()`):

```ts
for (const rec of log) {
  if (rec.type === "update") {
    // 任何 update 都让其事务进 active, 直到被证明已提交; 并标记该页脏。
    // 故意 over-include: Analysis 绝不能漏掉一个 loser; Redo/Undo 容忍集合偏大(幂等)。
    activeTxns.set(rec.txnId, rec.lsn);
    if (!dirtyPages.has(rec.pageId!)) dirtyPages.set(rec.pageId!, rec.lsn);
  } else if (rec.type === "commit") {
    // commit 记录是持久性的见证。见到它, 事务成为 winner, 离开 ATT——它的效果必须存活。
    committed.add(rec.txnId);
    activeTxns.delete(rec.txnId);
  }
}
```

seed=7 跑出来:

```
扫描记录数: 10
committed (winners): {T0, T1, T2}
ATT 活跃事务表 (losers): {T4, T3} -> 待回滚
DPT 脏页表: {page0}
```

注意 ATT 输出是 `{T4, T3}` 而不是 `{T3, T4}`——这是 Map 的插入顺序:lsn=2 时 T4 先被 update,lsn=5 时 T3 才出现。顺序无所谓,集合相等就行。

stage07 在这里加了断言,防止「Analysis 出了 bug 却伪装成后面恢复成功」:

```ts
assertEq(analysis.committed.size, wl.committedTxns.length, "analysis found all winners");
assertEq(analysis.activeTxns.size, wl.abortedTxns.length, "analysis found all losers");
```

为什么 Analysis 要**宁可多算不可少算**?因为 Redo 和 Undo 都是幂等的,集合偏大顶多多做点无用功;但 Analysis 一旦漏掉一个 loser,这个 loser 的脏数据就永远留在库里——这是不可恢复的数据污染。「over-include on purpose」是这一阶段的设计哲学。

> ⚠ 我们这里从日志**第一条**开始扫。真 ARIES 不会——日志可能有几百 GB,从头扫一遍开机要几小时。它从**最近一个 checkpoint(检查点)**开始扫,checkpoint 里存了崩溃前那一刻的 ATT 和 DPT 快照,Analysis 只需补扫 checkpoint 之后的尾巴。我们没有 checkpoint(留作练习),所以从头扫。这是「正确但不实用」到「实用」之间隔着的一层工程,真实系统离不开它。

## 三、Phase 2 — Redo:无脑重放,包括将被回滚的事务

这是整个 ARIES 最反直觉的一步,也是本章的 ✦ 专家钩子。

Redo 的规则极其粗暴:**按日志顺序,重放每一条 update 的 after-image——包括那些待会儿要被 Undo 回滚的 loser 的修改**。代码(`redo()`):

```ts
function redo(disk: Disk, log: readonly LogRecord[]): RedoResult {
  let redone = 0;
  const touched = new Set<number>();
  for (const rec of log) {
    if (rec.type !== "update") continue;
    // 幂等: 应用 after-image 是盲覆盖, 所以重放一条已生效的记录无害——
    // 这正是 before/after physiological 日志对恢复友好的原因。
    const page = disk.readPage(rec.pageId!);
    disk.writePage(rec.pageId!, withCounter(page, rec.account!, rec.after!));
    redone++;
    touched.add(rec.pageId!);
  }
  return { redone, pagesWritten: touched.size };
}
```

seed=7 的结果:

```
Phase 2 REDO:  重放 7 条 update (含将被回滚的 loser), 写 1 页
  redo 后计数: [107, 157, 143, 199] (此刻含 loser 的脏数据, 符合 'repeat history')
```

看这个中间状态 `[107, 157, 143, 199]`——它**故意是错的**!a1=157 包含了 loser T3 的 +50,a3=199 是 loser T4 的全部,a2=143 含 T3 的 +40。Redo 把库带到了「崩溃那一瞬间的样子」,脏数据全在。

**这就是 ✦ 钩子要回答的问题:为什么 Redo 要重做包括未提交事务的所有修改,再用 Undo 回滚?直接只重做已提交的不就完了吗?**

ARIES 给的答案叫 **repeating history(重复历史)**:先无脑把数据库精确追平到崩溃发生的那一瞬间(连未提交的脏写都复现),**然后**再从那个确定的状态精确回退 loser。两段式。

为什么不挑拣式恢复(只 redo winner、跳过 loser)?三个理由:

1. **崩溃时你不知道哪些 loser 的写已经落盘了**。真实场景里数据页是惰性刷盘的,崩溃那一刻,page0 上可能已经有 T3 的脏写(已经 fsync 过),也可能没有。如果你跳过 redo loser,但盘上其实**已经有**它的脏写,那这笔脏写就赖在库里没人回滚了(因为 Undo 倒序回滚时,它假设的起点是「崩溃瞬间的完整状态」,而不是你挑拣过的残缺状态)。

2. **repeating history 把恢复变成两个互相独立、各自可证正确的阶段**。Redo 的不变量:「重放后,库 == 崩溃瞬间的库」。Undo 的不变量:「从崩溃瞬间的库出发,倒序撤销所有 loser,得到 == 只有 winner 生效的库」。两个不变量串起来就是最终正确性。挑拣式恢复把这两步耦合在一起,得同时推理「哪些该 redo、哪些已落盘、哪些该跳」,组合爆炸,极难证明无 bug。

3. **简单且统一**。Redo 不需要查事务状态、不需要查 pageLSN(在我们的简化版里)、不需要分支判断,就是一个 for 循环盲写。代码越笨,bug 越少。恢复路径是数据库里最不容出错的代码——它在你最脆弱(刚崩溃)的时候运行,它再崩一次就是灾难。用「笨但可证正确」换「聪明但易错」,在这里是绝对正确的取舍。

下一阶段会消掉这个中间态的脏数据。但 Redo 这一步先把库带到一个**所有人都同意的确定起点**,这是它的全部价值。

## 四、Phase 3 — Undo:倒序回滚 loser

Redo 之后,库追平到了崩溃瞬间(含脏数据)。Undo 负责把 loser 的痕迹精确擦掉。规则:**倒序遍历日志,对每条 loser 的 update,盲写它的 before-image**。

```ts
function undo(disk: Disk, log: readonly LogRecord[], losers: ReadonlySet<number>): UndoResult {
  let undone = 0;
  for (let i = log.length - 1; i >= 0; i--) {
    const rec = log[i];
    if (rec.type !== "update") continue;
    if (!losers.has(rec.txnId)) continue;
    const page = disk.readPage(rec.pageId!);
    disk.writePage(rec.pageId!, withCounter(page, rec.account!, rec.before!));
    undone++;
  }
  return { undone };
}
```

seed=7:

```
Phase 3 UNDO:  回滚 3 条 loser update (倒序应用 before-image)
  undo 后计数: [107, 107, 103, 100]
```

回滚了 3 条(T3 两条 + T4 一条),库回到 `[107, 107, 103, 100]`。和独立计算的 ground truth 一致:

```
期望计数 (ground truth, 独立计算): [107, 107, 103, 100]
恢复后计数:                        [107, 107, 103, 100]
✓ 断言通过: 已提交事务全部生效, 未提交事务全部消失
```

a0=107(T0 的 +5+2 全在)、a1=107(T1 的 +7 在,T3 的 +50 被撤)、a2=103(T2 的 +3 在,T3 的 +40 被撤)、a3=100(T4 的 +99 整个被撤,回到 baseline)。原子性和持久性同时成立。

**为什么 Undo 必须倒序?** 因为同一个 slot 可能被同一事务写过多次,只有**逆序**应用 before-image 才能回到事务开始前的值。假设某 loser 对 a1 写了两次:先 100→120(before=100),再 120→135(before=120)。正确的回滚是:先撤第二次(写回 120),再撤第一次(写回 100),最终 100。如果你**正向**撤,会先写回 100、再写回 120,留下 120——第一次写的 after-image 赖着不走,这是一种很隐蔽的污染。倒序天然规避。代码注释原话:forward undo 会把第一次写的 after-image 留在原地。

(seed=7 里 loser T3/T4 各 slot 只写一次,所以这里没触发多写场景。但 winner T0 对 a0 写了两次(lsn 1 和 7),这正是用来验证倒序逻辑对「同一 slot 多写」健壮的——只不过 T0 是 winner 不被 undo,它的两次写由 Redo 顺序重放正确累加到 107。)

> ⚡ **前沿钩子:崩溃中的崩溃,以及边恢复边服务。** 现在问一个要命的问题:Undo 进行到一半,机器**又**断电了,怎么办?重启后第二次恢复,Analysis 会把那些 loser 又当成待回滚的——它们的 update 记录还在日志里。但已经被第一次 Undo 撤掉一半的写,会被**重复撤销**吗?在我们这个盲写 before-image 的简化版里,因为 Undo 是幂等的(写回 before 写几次都一样),侥幸没事。但真 ARIES 的 Undo **不**幂等(它执行的是逻辑逆操作,比如「从 B+ 树删一个 key」,删两次会出错)。它的解法叫 **CLR(Compensation Log Record,补偿日志记录)**:每撤销一步,就**写一条 CLR 日志**记录「我已经撤到这了」,CLR 带一个 `UndoNxtLSN` 指针指向下一条该撤的记录。第二次恢复时,Analysis 读到 CLR 就知道这步已撤过,顺着指针跳过,**绝不重撤**。CLR 是 only-redo 的(它自己只会被重做不会被撤销),从而让「Undo 自身也可恢复」。这是 ARIES 真正精巧的地方,我们简化版用幂等盲写绕过了它,但工业级实现绕不过。
>
> 更前沿的是 **instant recovery(即时恢复)**:传统数据库必须把 Redo+Undo 全跑完才对外开门,大库可能要几十分钟。研究方向(Goetz Graefe 等 2010s 的工作,以及部分云数据库的实践)是**边恢复边提供服务**——把恢复做成 on-demand:某个页被查询访问到时才即时 redo 它,某个 loser 被新事务挡住时才即时 undo 它。这让「可对外服务」的时间从「恢复总时长」降到接近零。但要把 on-demand 恢复做对、做到不和正常事务相互踩踏,**目前没有放之四海皆准的通用方案**,各家实现差异很大、仍在活跃研究。这是个开放问题。

## 五、失败实验:删掉 Redo,看持久性当场崩掉

教材最该做但最少做的事:**把机制拆掉,亲眼看它怎么坏**。stage07 跑了第二遍恢复,这次故意跳过 Redo,直接 Analysis → Undo:

```
--- 失败模式: 跳过 REDO, 直接 UNDO (证明三阶段顺序不可省) ---
重开数据文件, 起点计数: [100, 100, 100, 100]
(跳过 REDO: 不重放任何 after-image)
Phase 3 UNDO:  回滚 3 条 loser update
  结果计数:   [100, 107, 103, 100]
期望计数:     [107, 107, 103, 100]

✗ Durability 被破坏: 1/4 个账户丢失了已提交的修改
   account 0: 期望 107, 实得 100 (差 7 = 已提交但从未重放的增量)
```

a0 丢了!ground truth 是 107(T0 已提交的 +5+2),实得 100。为什么单单 a0 出问题?

因为 a1、a2 凑巧被 loser 碰过——Undo 把它们的 before-image 写回去时,顺手把值「拉」到了一个看似对的数(纯属巧合:T3 对 a1 的 before 恰好是 T1 commit 后的 107)。但 a0 **只有 winner T0 碰过、没有任何 loser 碰它**,于是跳过 Redo 后,没有任何力量把 T0 的已提交写放回 a0——它永远停在 baseline 100。

这就是 ✦ 钩子的反面证明:**Redo 的职责是把所有 update(包括崩溃前没来得及落盘的已提交写)重新落盘。没有 Redo,Undo 只能从一个残缺起点回滚 loser,已提交的写永远回不来。** 持久性(committed 的必须存活)被直接违反——这是数据库最不可接受的故障,用户明明收到了「提交成功」,数据却没了。

stage07 把这个失败做成**可执行的断言**,防止 demo 退化成「碰巧能跑」:

```ts
invariant(lost.length > 0, "skipping Redo MUST lose committed work, else the demo is vacuous");
```

结论钉死:三阶段顺序 Analysis → Redo → Undo,缺一不可。

## 六、恢复的代价:数一数 IO

恢复不是免费的。stage07 用 `resetStats()` 把崩溃前的 IO 清零,只量恢复本身:

```
恢复 IO (实测计数): reads=14, writes=10
```

拆开看:Redo 7 条 update,每条先 readPage 再 writePage = 7 读 7 写;Undo 3 条 loser,3 读 3 写;另有读取计数器做断言验证的若干次读(`counters()` 每次调 readPage)。合计 reads=14、writes=10。

这是 toy 数字(1 个数据页),但**形状**是真的:恢复成本由**日志长度**和**页 IO 次数**主导,两者我们都精确计数了。注意 Redo 对同一个 page0 反复 read-modify-write 7 次——真实引擎绝不会这么干,它会把页钉在 buffer pool(缓冲池)里、所有 redo 在内存累积、最后刷一次。我们这里每次都走 `disk.readPage`/`writePage` 是为了让 IO 计数器**诚实**(每次逻辑 IO 都被数到),不优化掉、不藏数字。代码注释明说:这是刻意的可读性选择,不是真引擎的写法。

为什么不报「恢复耗时 N 毫秒」?因为在 1 页的 toy disk 上,墙钟时间纯粹是噪音,报一个 ms 数字是自欺。恢复的真实成本就是 IO 次数和日志长度,这两个我们数得清清楚楚——其余的别编。

## 把这章接回主线

往回看:第 05 章的 WAL 规则(先日志后数据)+ `crashAt` 注入的崩溃现场,在这一章被**消费**了——没有那条 fsync 过的日志,这章无米下锅。第 06 章的 MVCC 给了你「并发下谁看到什么」,而这一章给的是「崩溃后大家最终同意什么」——前者是运行时的隔离性,后者是故障后的原子性 + 持久性,合起来才是 ACID 里的 A、I、D(C 一致性是它们的产物)。

往前看:Analysis 从头扫日志这件事,真实系统靠 **checkpoint** 把它压到可接受;Undo 的「崩溃中崩溃」靠 **CLR** 兜底;「恢复期间不可用」靠 **instant recovery** 这类仍在研究的方向去消除。这一章给你的是骨架和它**为什么正确**的证明;工业级的血肉,是在这副骨架上一层层补上的工程。

记住一句话就够本章:**先无脑追平到崩溃瞬间(repeating history),再精确回退——这比挑拣式恢复更简单,且可证正确。** 恢复路径是数据库里你最不愿意它聪明的地方。
