前七章你写的全是"函数式"数据库:想读一行,你得自己 `tree.search(key)`;想范围扫,自己 `disk.readPage(id)` 一页页翻。没有 SQL,没有"声明你要什么、由引擎决定怎么拿"的那层。这一章补上的就是这层——它是全书唯一一个把存储、索引、编解码全部串起来跑的子系统。一条字符串进去,五个算子流水作业,结果出来。

我先把这条查询贴出来,本章所有数字都是它跑出来的(配套代码 `examples/database-from-scratch/src/stage08-query-executor.ts`,确定性种子 `0x5108`,两张各 1 万行的表):

```sql
SELECT u.country, COUNT(o.id) AS order_count, SUM(o.amount) AS revenue
FROM users u
JOIN orders o ON o.user_id = u.id
WHERE u.id < 2000
GROUP BY u.country
```

跑出来的物理计划是这棵树,以及它的代价:

```
HashAggregate(group=u.country, aggs=count(o.id),sum(o.amount))
  └─ HashJoin(o.user_id = u.id)
    └─ SeqScan(orders o, pages=40)
    └─ IndexScan(u.id in [0,1999])
```

```
next() 调用总数      14080
页触达 reads        53
HashJoin build 行数  2000
HashJoin probe 行数  10000
HashJoin 输出行数    2071
最终结果行数         5
wall time (measured) 6.966 ms
```

记住 `14080` 这个数。它是本章后半段所有性能论证的靶子——火山模型优雅,但优雅有代价,这个数就是代价的标价牌。

## 为什么是三棵树,不是一棵

新手写查询引擎最常见的错误:把 SQL 直接解析成一棵能执行的树,一步到位。看起来省事,实际上你把三件本质不同的事搅在了一起,之后任何优化都做不进去。

正确的分层是三棵独立的树,每棵回答一个不同的问题:

| 阶段 | 数据结构 | 回答的问题 | 代码位置 |
|------|----------|-----------|----------|
| **AST**(抽象语法树) | `SelectQuery` | 这串字符**语法上**是什么 | `Parser.parse()` |
| **逻辑计划** | `LogicalNode` | 用户**想要什么关系结果** | `planLogical()` |
| **物理计划** | `PhysicalOp` 树 | **怎么**算出这个结果 | `planPhysical()` |

为什么必须分开?因为同一个逻辑计划可以有多个物理实现。`WHERE u.id < 2000` 这个过滤,逻辑上就是"筛掉一部分 users";物理上既能走 `IndexScan`(用第 03 章的 B+树跳过不匹配的页),也能走 `SeqScan + Filter`(全表扫再逐行判断)。如果你把"过滤"这个意图和"用索引"这个实现焊死在一棵树上,你就永远没法在二者之间做选择——而做选择正是查询优化器的全部工作。

代码里这三层是物理隔离的。`LogicalNode` 是个 union type,故意不带任何"怎么执行"的信息:

```typescript
type LogicalNode =
  | { op: "scan"; table: string; alias: string }
  | { op: "filter"; pred: Predicate; input: LogicalNode }
  | { op: "join"; on: { left: ColRef; right: ColRef }; left: LogicalNode; right: LogicalNode }
  | { op: "aggregate"; groupBy: ColRef | undefined; aggs: SelectItem[]; input: LogicalNode }
  | { op: "project"; items: SelectItem[]; input: LogicalNode };
```

注意这里没有 `IndexScan`、没有 `HashJoin`。逻辑层只说"scan"和"join",不说"用什么 scan、用什么 join 算法"。物理层才把 `scan` 翻译成 `SeqScan` 或 `IndexScan`,把 `join` 翻译成 `HashJoin`。这个区分听起来是教科书废话,但它是"这个优化器能不能扩展"的分水岭——商业数据库(PostgreSQL、SQL Server)的优化器之所以能在几十种 join 算法间选,就是因为逻辑/物理这条线划得干净。

### 失败模式:解析器静默吞 token

parser 这层最阴险的 bug 不是崩溃,而是**不崩溃**。一个 SQL 子集解析器,如果遇到看不懂的 token 就跳过、继续解析剩下的,你会得到一个"查询跑了,但算的是另一回事"的灾难——而且没有任何报错。

所以本章的 parser 对任何不在文法内的 token 一律抛错,且报出**期望什么 vs 见到什么**:

```typescript
private expectKw(kw: string): void {
  const t = this.next();
  if (t.kind !== "kw" || t.text.toLowerCase() !== kw) {
    throw new Error(`parse: expected keyword '${kw}', got '${t.text}'`);
  }
}
```

更关键的是解析结束后的这一句:

```typescript
if (this.peek()) throw new Error(`parse: trailing tokens starting at '${this.peek()!.text}'`);
```

如果文法走完了还剩 token,说明你的 SQL 有引擎不理解的部分——宁可拒绝整条查询,也绝不"尽力而为"地执行一半。这是数据库正确性的底线:解析器的契约是"要么完全理解,要么明确拒绝",没有中间地带。上面那条 47 token 的查询能跑通,正是因为它的每个 token 都落在文法内。

## 火山模型:把整个引擎压缩成一个 next()

物理计划是一棵算子树,每个算子实现同一个接口——这就是 **Volcano 模型**(火山模型,又叫迭代器模型,1994 年 Goetz Graefe 提出),四十年来几乎所有教科书执行器的起点:

```typescript
interface PhysicalOp {
  open(): void;
  next(): Tuple | null;   // 返回下一行,或 null 表示流结束
  close(): void;
  outputColumns(): string[];
}
```

整个执行引擎的控制流就这么三句话——`open` 准备,`next` 拉一行,`close` 收尾。算子之间的组合关系简单到只有一句:**"我的输入是另一个迭代器"**。`Filter` 的输入是个 `PhysicalOp`,`HashJoin` 的两个输入也是 `PhysicalOp`。这种"算子套算子"的统一接口,让你能用积木拼出任意复杂的查询树。

驱动整棵树的"引擎"本身也短到不像引擎——就是反复 `next()` 直到拿到 null:

```typescript
function runPlan(root: PhysicalOp): Tuple[] {
  root.open();
  const out: Tuple[] = [];
  for (;;) {
    const row = root.next();
    if (row === null) break;
    out.push(row);
  }
  root.close();
  return out;
}
```

这就是火山模型的全部魅力:每个算子是一个小状态机,引擎是个 5 行循环。它叫"pull 模型"(拉取模型)——根算子向下要一行,这个需求像吸管一样一路传到叶子,叶子从页里抠出一行往上吐。数据是被"拉"上来的,不是被"推"上去的。

### 流式与阻塞:不是所有算子都能边读边吐

火山模型最值钱的性质是**流式**(streaming):理想情况下,一行从叶子流到根、产出,内存里始终只压着一行(加一页缓冲)。`SeqScan` 就是教科书式的流式算子——它读一页、缓存这页的行、逐行吐出,吐完再读下一页:

```typescript
next(): Tuple | null {
  this.stats.nextCalls++;
  for (;;) {
    if (this.bufIdx < this.buffer.length) {
      this.stats.rowsEmitted++;
      return this.buffer[this.bufIdx++];
    }
    if (this.pageCursor >= this.table.pageCount) return null; // EOS, 稳定返回 null
    const pageId = this.table.firstPage + this.pageCursor++;
    const page = this.disk.readPage(pageId);  // 被计数的数据页 IO
    this.buffer = readPageRows(page, this.table.types);
    this.bufIdx = 0;
  }
}
```

无论表有 40 页还是 4000 万页,`SeqScan` 的内存占用都是一页——这正是火山模型存在的理由:它让你在内存装不下的数据上跑查询。

但**不是所有算子都能流式**。两类算子是"阻塞"(blocking)的:它们必须先把输入吃完,才能吐出第一行。

第一类是 `HashJoin`。它的 build 阶段(构建哈希表)必须把构建侧**整个**读进内存:

```typescript
open(): void {
  // ... 重置计数器、解析 join key 位置 ...
  // build 阶段:物化整个构建侧。这是唯一一个不完全流式的算子。
  for (;;) {
    const r = this.right.next();
    if (r === null) break;
    this.buildRows++;
    const key = r[this.rightKeyPos];
    const bucket = this.table.get(key);
    if (bucket) bucket.push(r);
    else this.table.set(key, [r]);
  }
}
```

这就是为什么计划里 `WHERE u.id < 2000` 过滤后的 users(2000 行)被放在**构建侧**,而全部 10000 行 orders 放在**探测侧**。哈希连接要把构建侧整个塞进内存哈希表,谁小谁当构建侧——构建 2000 行的哈希表,比构建 10000 行的省 5 倍内存。这个选择在 `planLogical()` 里用一条规则做掉:

```typescript
// 携带 WHERE 过滤的那一侧,过滤后预期更小,让它当 join 的构建侧(right)。
const fromIsFiltered = !!(q.where && q.where.left.table === q.from.alias);
if (fromIsFiltered) {
  root = { op: "join", on: { left: q.join.left, right: q.join.right }, left: joinScan, right: root };
}
```

我得诚实标注:这是个**启发式规则**,不是真正的代价估算。真正的优化器会用统计信息(直方图、行数估计)算出每一侧过滤后的基数,再决定谁当构建侧。这里我用"带过滤的那侧更小"这条经验法则替代了统计——对这条查询恰好对(2000 < 10000),但它会在某些查询上选错。这是 toy 实现和生产优化器最大的鸿沟,后面 ⚡ 会再提。

第二类阻塞算子是 `HashAggregate`。聚合天然阻塞——你不看完一个组的所有行,就报不出这个组的 `COUNT`/`SUM`。所以它在 `open()` 里把输入抽干,边读边折叠进每组的累加器:

```typescript
for (;;) {
  const row = this.input.next();
  if (row === null) break;
  const key: Value = this.groupBy ? row[this.groupPos] : "__all__";
  let acc = this.groups.get(key);
  // ... 给每个聚合槽累加 count / sum ...
}
```

阻塞算子是流式模型里的"水坝":数据流到这里要先蓄起来。一个查询的内存峰值,基本就由它最大的阻塞算子的物化量决定。

## 索引扫描 vs 全表扫描:省下来的是页 IO

`IndexScan` 是这章把第 03 章 B+树真正用起来的地方。它不读全表,而是先用 B+树定位匹配的 row,再只读这些 row 所在的堆页:

```typescript
open(): void {
  // 探测索引(计入"内部页 + 命中叶页"的页触达),再逐个从堆页取行
  // (只计入持有命中行的页,去重避免一页多次命中被重复计数)。
  const rowIndexes = probeIndexRange(this.disk, this.table.indexRoot, this.lo, this.hi);
  const wantedPages = new Map<number, Tuple[]>();
  for (const ri of rowIndexes) {
    const pageId = this.rowToPage[ri];
    if (!wantedPages.has(pageId)) {
      const page = this.disk.readPage(pageId);  // 被计数:堆页触达
      wantedPages.set(pageId, readPageRows(page, this.table.types));
    }
  }
  // ...
}
```

数字最能说明问题。同一条查询,两条物理计划的对照实测:

```
plan                 page_reads  next_calls  wall_ms
-------------------  ----------  ----------  -------
IndexScan(users.id)          53       14080    6.966
SeqScan(users)               77       24081   10.248
```

`IndexScan` 比 `SeqScan` 少触达 24 页(77 → 53,约 1.5 倍更少页 IO)。这里我必须给数字打个诚实标签:**数据是 toy(用 RAM 模拟 disk),墙钟绝对值偏乐观**——真实磁盘上一次随机 IO 是毫秒级,这里是纳秒级。可迁移的不是 `6.966 ms` 这个绝对值,而是**页 IO 的比例趋势**:索引让你跳过大量页,这个比例在真实磁盘上会被放大成数量级的差距,因为真实系统里页 IO 才是瓶颈,而不是 CPU。

为什么 B+树能做到只触达命中页?因为它是有序的。`WHERE u.id < 2000` 翻译成 key 范围 `[0, 1999]`,B+树从根降到内部目录页,定位到第一个可能含 key 0 的叶页,然后**顺序扫叶页直到 firstKey 超过 1999 就停**:

```typescript
const fk = ir.readBytes(entryBase, 4);
if (compareKeys(fk, hiKey) > 0) break;  // 叶子有序,后面不可能再匹配,停
```

这个"早停"是索引省 IO 的核心机制——它把扫描量从"全表"压缩到"匹配范围"。代码里还藏了一个第 03 章传下来的关键技巧:key 用大端序(big-endian)编码,使得**字节序等于整数序**,所以 `compareKeys` 用 memcmp 风格的字节比较就是正确的整数比较。这个不起眼的编码选择,是 B+树能用统一字节比较跑通的地基。

### 失败模式:对没索引的列硬上索引

现在讲这章最有教学价值的失败模式。优化器不能凭空变出索引——你 `WHERE` 一个没建索引的列,无论你多想用索引,引擎都只能全表扫。我用第二条查询逼出这个回退:

```sql
SELECT o.user_id, COUNT(o.id) AS cnt FROM orders o WHERE o.amount > 990 GROUP BY o.user_id
```

`orders` 表故意没建任何索引。代码里我强制规划器在 `o.amount` 上走 `IndexScan`,看它怎么处理。规划器的判定逻辑会先**校验索引真的存在**,不存在就记下原因并回退:

```typescript
if (forced && !indexUsable) {
  ctx.fallbackNote =
    `requested IndexScan on ${node.pred.left.table}.${node.pred.left.col}, ` +
    `but '${table.name}' has no index on '${node.pred.left.col}' ` +
    `(only on '${table.indexedCol >= 0 ? table.columns[table.indexedCol] : "(none)"}'); ` +
    `planner fell back to full SeqScan + Filter`;
}
return new Filter(new SeqScan(ctx.disk, table, alias, ctx.stats), node.pred, ctx.stats);
```

跑出来的回退判定原文:

```
requested IndexScan on o.amount, but 'orders' has no index on 'amount'
(only on '(none)'); planner fell back to full SeqScan + Filter
```

代价是实打实的:

```
orders 数据页总数         40       全表
实际页触达 (SeqScan 回退)  40       被迫扫全表
next() 调用总数          10194     每行都过 Filter
命中行数 (amount>990)     96       通过谓词的 orders 行
若有 amount 索引 (est.)  ~2 页     估算: 高选择性谓词只需触达命中页
```

注意这个对比有多刺眼:谓词 `amount > 990` 是**高选择性**的——1 万行里只有 96 行命中(0.96%)。一个 `amount` 上的 B+树本可以只触达约 2 页(1 内部页 + 约 1 堆/叶页)就拿到这 96 行。但因为没索引,引擎被迫扫满全部 40 页。估算加速比约 20 倍(40 页 → ~2 页)。这里 `~2 页` 我明确标注是**估算**(est.),因为我没真建这个索引,不能伪造一个测出来的数字——诚实地说"如果有索引会怎样",而不是编一个数。

结论一句话钉死:**缺索引 ⇒ 全表扫描,触页数与表大小线性增长,与谓词选择性无关。** 这就是为什么"这条查询慢"九成是"少建了个索引"——选择性再高也救不了你,因为引擎根本没法利用选择性。这条经验,是 DBA 看慢查询日志时的第一反射。

## ✦ 优雅的代价:14080 次 next() 是怎么来的

现在回到那个一直在等的数字。火山模型每个算子的 `next()` 第一行都是 `this.stats.nextCalls++`——我把每一次虚调用都数出来了。这条 5 算子的查询,总共调了 **14080 次 next()**。

这个数怎么来的?粗略拆解:`SeqScan(orders)` 吐 1 万行,每行一次 `next()`;`IndexScan` 吐 2000 行;`HashJoin` 探测 1 万行、输出 2071 行;`HashAggregate` 在 open 里把 join 的输出全抽干……每一行**穿过每一个算子边界**,都是一次独立的函数调用。行数 × 算子层数,就是这个量级。

问题来了:**为什么这是个性能问题?** 在分析型负载(扫几百万、几十亿行做聚合)上,火山模型会比向量化执行慢 5 到 10 倍。慢在哪?不在算法,算法是一样的;慢在 `next()` 这个调用本身的**固定开销**:

1. **虚函数调用**(virtual call):`op.next()` 是接口方法,CPU 不能在编译期确定跳到哪段代码,要查虚函数表。每行每算子一次,分支预测器命中率低,流水线频繁打嗝。
2. **一次只搬一行**:`next()` 返回一个 `Tuple`(一行)。CPU 的 SIMD(单指令多数据)指令一条能同时算 8 个、16 个数,但火山模型一次只喂一行,SIMD 闲着。
3. **解释开销摊不薄**:每行都要重新判断"这是什么算子、走哪个分支",这些控制开销没法在多行之间分摊。

换句话说:火山模型把"处理一行数据"的**真实工作**(比较一个 int、累加一个 sum)淹没在了"调一次函数"的**仪式开销**里。当真实工作很重(复杂表达式)时这没关系;但分析负载的真实工作往往就是"加个数"——这时仪式开销反而是大头。一个 `next()` 调用的真实计算可能就几纳秒,而虚调用 + 边界开销也是几纳秒,等于一半时间花在搬运上。

这就是 ✦ 专家钩子的全部:**火山模型的优雅(统一 next() 接口、积木式组合)和它的慢是同一件事**。正是"一行一个 next() 调用"这个让代码如此干净的设计,在大数据量上变成了性能毒药。你没法既要这种优雅,又要分析负载的吞吐——除非换执行模型。

## ⚡ 前沿:火山模型之后,执行引擎往哪走

火山模型的 `14080` 次调用是靶子,过去十几年的执行引擎研究基本都在打这个靶子。三条主线,各有取舍,且**没有一条是公认的通用最优解**——这是目前仍在活跃研究、各大数据库选型分歧很大的领域。

**第一条:向量化执行(vectorized execution)。** 不改 pull 模型的骨架,但把 `next()` 的返回值从"一行"换成"一批"——典型 1024 行一个 batch。`next()` 调用次数直接除以 1024,虚调用开销摊薄到几乎为零,而且一批行存成列式数组后,编译器能自动用 SIMD 一次算 16 个。代表是 DuckDB 和 MonetDB/X100(2005 年那篇 "MonetDB/X100: Hyper-Pipelining Query Execution" 是这条路线的奠基论文)。它的好处是**改动局部**——算子接口从 `Tuple next()` 变成 `Batch next()`,整体结构不动,工程上最容易落地。

**第二条:编译执行(compilation / codegen)。** 更激进——干脆别"解释"算子树了,把整条查询**编译成机器码**。火山模型每行都在问"我是什么算子",编译执行直接把这条查询特化成一段没有虚调用、没有分支判断的紧凑循环,用 LLVM 生成原生代码。代表是 HyPer(2011 年 Thomas Neumann 的 "Efficiently Compiling Efficient Query Plans for Modern Hardware",这条路线最重要的论文)。代价是编译本身要时间——对跑一次的小查询,编译开销可能比执行还久,所以通常对热查询或大查询才值得。

**第三条:push 模型 + morsel-driven 并行。** 这条最颠覆——它把数据流的方向**反过来**。火山模型是根算子向下"拉"(pull),push 模型是叶子算子向上"推"(push):叶子产出一行(或一批)就主动调用父算子的"消费"函数,数据被推着往上走。为什么要反向?因为 push 模型对**并行**友好得多——HyPer 的 morsel-driven 调度(2014 年 "Morsel-Driven Parallelism" 那篇)把数据切成小块(morsel),用工作窃取(work-stealing)在多核间动态分配,天然适配 NUMA 架构。push 模型还能让多个算子的代码"融合"(fusion)进一个循环,进一步消灭边界开销。

取舍在哪?**这三条不是互斥的,真实系统在混搭,而且没有定论。** DuckDB 选了向量化(因为编译的预热开销对交互式分析不友好);Spark SQL 的 Tungsten 选了 whole-stage codegen(编译);ClickHouse 是向量化 + 部分编译。push 还是 pull、向量化还是编译,在不同负载(短查询 vs 大扫描、OLTP vs OLAP、单机 vs 分布式)上的最优解不同——**目前没有一个执行模型在所有负载上通吃**。这正是执行引擎仍是数据库研究热点的原因:它是"硬件越来越宽(SIMD、多核、NUMA),怎么让查询引擎吃满硬件"这个开放问题的主战场。

本章的火山模型是这一切的**起点和基线**——你得先理解为什么 `next()` 优雅、为什么它慢,才能理解后面所有花活到底在优化什么。把 `14080` 这个数记牢:向量化要把它除以 1024,编译要把它变成 0(没有 next() 调用了,只有一个循环),push 要把它的方向调转。它们消灭的,都是同一个东西。

## 一致性 assert:优化器的契约是"不改变语义"

最后一个容易被忽略但至关重要的点。`IndexScan` 和 `SeqScan` 是两条完全不同的执行路径,但它们**必须产出逐字节相同的结果**——否则"用索引优化"就是在改变查询的含义,那不叫优化,叫 bug。

代码里这条契约是**可执行的**,不是口头承诺:

```typescript
invariant(
  JSON.stringify(normalize(resultIdx)) === JSON.stringify(normalize(resultSeq)),
  "IndexScan and SeqScan must produce identical results",
);
```

跑出来两条路径的结果完全一致(都是按 country 分组的 5 行):

```
country  order_count  revenue
-------  -----------  -------
BR               457   223383
CN               425   206327
DE               381   190542
JP               395   191334
US               413   204733
```

这个 assert 把"优化器保持语义"从一个假设变成了一个**会爆炸的检查**。这是整个查询优化领域的地基:优化器可以自由地选 IndexScan 还是 SeqScan、选 HashJoin 的哪一侧当构建侧,但它**绝不能改变结果**。一旦改了,上面这行 `invariant` 立刻 throw——这正是第 07 章崩溃恢复里那种"用断言把不变量钉成代码"的思路,在查询层的复刻。前七章你为存储建立的所有正确性保证,到这一章必须在查询路径上重新被验证一遍:存储对了不代表查询对了,得有自己的对拍。

这一章是全书的汇合点:tokenizer 吐 token,parser 拼 AST,planner 分两步把 AST 变成算子树,火山引擎一路 `next()` 把第 03 章 B+树里的字节抠成你要的聚合结果。你现在有了一个能跑 `SELECT...WHERE...JOIN...GROUP BY` 子集的真引擎——它小,但每个环节都是真的:真解析、真索引、真页 IO、真哈希连接。而那个 `14080`,是你接下来理解任何现代执行引擎(向量化、编译、push)的钥匙:它们存在的全部理由,就是让这个数字小下去。
