`SELECT name FROM users WHERE id = 42` 这一行,在 PostgreSQL 里要穿过解析器、规划器、执行器、访问方法、缓冲管理器、存储管理器,最后落到一次 `pread()` 系统调用读上来一个 4KB 的块。这本书不教你用数据库——教你把这条路径上的每个零件亲手用 TypeScript 写一遍,而且让它**真能跑、真能崩、真能恢复**。配套代码在 `examples/database-from-scratch/`,每章一个可独立运行的 `stage`,结尾打印代码真算出来的数字:真实 IO 次数、缓冲池命中率、B+树高度、WAL 的 fsync 成本、崩溃恢复后的一致性断言。

这是一本反「AI 批量生成的 survey 稿」的书。Survey 稿会告诉你「数据库由存储引擎和查询引擎组成,存储引擎负责持久化」——正确、完整、且毫无用处。本书要回答的是另一类问题:**为什么几乎所有引擎都按 4KB/8KB 的页存盘,而不是一行一行写?为什么 fsync 是耐久性逃不掉的账单?为什么 MVCC 能让读不阻塞写、写不阻塞读,代价是什么?** 这些问题在后端 / 基础设施岗的面试里反复出现,不是因为面试官想为难你,而是因为这条「字节 → 结果」的链路是工程权衡(trade-off)最密集的地方——每一段都有人为它写过论文、踩过生产事故。

本章不配 stage,它是全书的地图。读完你应该能在脑子里画出:一条 SQL 进来,数据在 7 个子系统之间怎么流;每个子系统依赖谁、被谁依赖;以及为什么是这个顺序。后面每章深入一个零件时,你都能回到这张图上找到它的位置。

## 一条 SELECT 的完整路径:数据在哪里流动

先把全景拉出来。一条点查(point lookup,按主键查单行)从发出到拿到结果,数据流是这样的:

```
SQL 文本
  │  ① 解析 + 规划(本书最后才碰,因为它依赖下面全部)
  ▼
查询执行器(stage08:火山模型算子)
  │  "我要 id=42 这一行" → 问索引
  ▼
B+树索引(stage03:有序索引,把 key 映射到 tid)
  │  返回 tid = (pageId, slotId)
  ▼
缓冲池(stage02:LRU 缓存页)
  │  pageId 在内存吗?在 → 命中,不在 → miss,触发一次磁盘读
  ▼
磁盘 / 页(stage01 + core:4KB 固定页,slotted layout)
  │  pread 一个 4096 字节的页,按 slotId 在页内定位 tuple
  ▼
解码出 tuple(core/codec:字节 → (42, "Alice"))
```

读路径看着是单向的。但**写路径**(`INSERT` / `UPDATE`)会把另外三个子系统拽进来,而且顺序变得微妙:

```
写一行
  ▼
WAL(stage05:预写日志)──── 必须先持久化日志,再改数据页 ────┐
  ▼                                                          │
MVCC 版本链(stage06:写不覆盖旧值,追加新版本)               │ 崩溃后
  ▼                                                          │ 用日志
缓冲池(脏页可以晚点才刷盘)                                  │ replay
  ▼                                                          ▼
崩溃恢复(stage07:掉电后 replay WAL,断言 DB == 崩溃前)──────┘
```

「先写日志再改数据」这五个字,就是 WAL(Write-Ahead Logging,预写日志)的全部哲学,也是整个数据库可靠性的基石。为什么不能反过来?后面崩溃恢复一节会拆。这里先记住数据流的形状:**读路径求快(少 IO),写路径求稳(可恢复),两者在缓冲池这个十字路口交汇。**

## 七个子系统,以及它们为什么是这个依赖顺序

本书的章节顺序不是随便排的,是依赖关系**逼出来的**——上层零件没法在下层零件存在之前测试。

| stage | 子系统 | 一句话职责 | 依赖谁 |
|---|---|---|---|
| core | 字节底座 | 页读写原语 + 行/键编码 + 逻辑时钟 | 无(全书地基) |
| 01 | 存储页 | 变长行在一页内怎么摆(slotted page) | core |
| 02 | 缓冲池 | 把热页缓存在内存,决定真实磁盘 IO | 01 |
| 03 | B+树 | 有序索引,把 key 高效映射到行位置 | 01, 02 |
| 04 | LSM 树 | 写优化的另一条路线,换写放大省随机写 | 01 |
| 05 | WAL | 预写日志,把"改了什么"先记下来 | 01 |
| 06 | MVCC | 多版本并发,读写互不阻塞 | 05 |
| 07 | 崩溃恢复 | 掉电后 replay 日志,恢复到一致状态 | 05, 06 |
| 08 | 查询执行 | 火山模型算子:扫描 / 过滤 / 连接 | 03, 02 |

读这张表的正确方式是从**底**往上看。`core/` 是所有人的地基:它定义了「一个页是 4096 个字节」「一行怎么变成字节」「两个 key 怎么比大小」。这三件事一旦定死,上面所有子系统都得遵守——这正是为什么本书把它们抽到 `core/` 而不是让每章各写一份:

> `core/page.ts` 头注释里写得很直白:「a page is just 4096 raw bytes. Everything above (slotted rows, B+tree nodes, WAL records) is a discipline imposed on those bytes. If each chapter invented its own read/write helpers, off-by-one byte bugs would be unattributable」。

字节层的纪律一旦各写各的,一个差一字节的 bug 就会变得无法归因——你不知道是页布局写错了,还是 B+树节点写错了。所以全书的二进制读写都走 `PageReader` / `PageWriter`(`examples/database-from-scratch/src/core/page.ts`),薄薄地包在 `DataView` 外面,不藏分配、不藏字节序意外。

注意 04(LSM)和 03(B+树)是**并列的两条路线**,不是上下游。B+树是读优化的就地更新结构,LSM(Log-Structured Merge-tree,日志结构合并树)是写优化的追加结构。一个引擎选哪条,取决于负载是读多还是写多——PostgreSQL / MySQL InnoDB 走 B+树,RocksDB / Cassandra / LevelDB 走 LSM。本书两条都写,因为它们的取舍对比本身就是最值钱的工程判断之一。第 04 章会用代码实测同一组写入下,B+树的就地更新和 LSM 的写放大(write amplification,一次逻辑写引发多少次物理写)各是多少。

## 全书共享词汇:六个一旦说错就全乱的词

数据库圈黑话密集,且很多词在不同语境下指不同东西。本书把六个核心词钉死,后面每章都按这个定义用。术语首现给一句大白话,这里集中列清:

- **page(页)**:磁盘和内存交换数据的最小单位,本书固定 **4096 字节**。`core/disk.ts` 里写死 `export const PAGE_SIZE = 4096`,且注释标明「PAGE_SIZE is fixed at 4096 forever. Pages are the atom」。所有 IO 都是整页的——读半页这种事不存在。
- **tuple(元组)**:数据库行话里的「一行」。一个 tuple 是一行编码成的字节串,住在某个页里。本书用 `core/codec.ts` 的 `encodeRow` / `decodeRow` 在「值」和「字节」之间转换。
- **tid(tuple id,行标识)**:一个 tuple 的物理地址,= `(pageId, slotId)`。「在第几页的第几个槽位」。索引里存的不是行内容,是 tid;查到 tid 再去缓冲池捞那个页、按 slotId 定位行。这个二元组是第 01 章页布局和第 03 章索引之间的契约。
- **lsn(log sequence number,日志序号)**:WAL 里每条日志记录的严格递增编号。`stage05-wal.ts` 里 `private nextLsn = 1`,且注释标明「LSN 0 reserved as 'no log record' sentinel」——0 是哨兵值,表示「还没有任何日志」。崩溃恢复靠 lsn 的顺序决定 replay 哪些记录。
- **txid(transaction id,事务号)**:一个事务的唯一标识,本书用逻辑时钟(logical clock,不看墙上时间、只看先后顺序的计数器)发号。`core/clock.ts` 的 `LamportClock.tick()` 严格递增、不重复、不复用。为什么不用 `Date.now()`?注释说透了:「wall-clock time is neither monotonic nor reproducible」——墙钟既不单调也不可复现,而 MVCC 的可见性判断纯粹靠顺序,顺序乱了就会出现「看起来正确的丢更新」。
- **version(版本)**:MVCC 下,一个 key 不是一个值,而是一条**版本链**——每次写追加一个新版本,旧版本留着。`stage06-mvcc.ts` 的 `interface Version` 每个版本带 `commitTs`(提交时间戳);读取时按读者的快照时间戳沿链找「提交时间 ≤ 我的快照」的最新版本。这就是为什么读不阻塞写:读者看的是某个时间点的快照,写者尽管去追加新版本。

记住 **tid = (pageId, slotId)** 和 **lsn 严格递增**这两条,第 01 章的页布局和第 07 章的崩溃恢复就有了挂钩的钉子。

## ✦ 专家钩子:为什么几乎所有引擎都选 4KB/8KB 页,而不是按行存盘?

这是面试高频题,也是新手最容易给出 survey 式空答案的地方(「为了提高 IO 效率」——对,但没说出任何东西)。真实答案是**三股力量的合谋**,缺一条都不足以解释为什么是「页」而且是「4KB 那个量级」。

**第一股:对齐操作系统的页。** OS 管理内存和文件 IO 的最小单位就是页(x86-64 上通常 4KB)。`pread`/`pwrite`、mmap、page cache、磁盘扇区——整条 IO 栈都按页对齐。如果数据库按任意长度的行去读写,每次 IO 都会跨页边界,触发额外的页错误和读放大。把数据库页对齐到 OS 页(4KB)或其整数倍(8KB),IO 栈每一层都顺滑。这是为什么 SQLite 默认 4KB、PostgreSQL 默认 8KB、InnoDB 默认 16KB——全是 4KB 的整数倍,不是巧合。

**第二股:摊薄寻道 / 单次 IO 的固定开销。** 一次磁盘 IO(哪怕是 SSD)有固定开销:系统调用、寻址、控制器往返。读 1 字节和读 4096 字节的开销差不太多。所以你要让每次 IO 搬回来尽量多的、**很可能马上要用**的数据。一行 50 字节,一页 4096 字节能装大几十行——查一行时把它周围的行也捎回来,正好命中「访问有局部性」这个几乎所有真实负载都成立的假设。本书 `core/disk.ts` 把这件事建模成「唯一的 IO 形式就是整页读写」,正是因为只有这样,缓冲池命中率、B+树页读次数这些数字才有意义——「if we stored rows in a JS array, every interesting number would evaporate」(注释原话)。

**第三股,也是最容易被漏掉的:B+树的扇出(fanout)。** B+树一个节点就是一个页。节点越大,一个节点能放下的 key 越多,即扇出越大;扇出越大,树越矮;树越矮,点查要读的页越少(每读一页 = 一次潜在磁盘 IO)。一个 4KB 的内部节点能放几百个 key,于是几百万行的表,B+树高度也就 3~4 层——点查最多 3~4 次页读。如果页太小(比如 64 字节),扇出锐减,树长得又高又瘦,点查的 IO 次数暴涨;如果页太大(比如 1MB),每次 IO 搬回一大堆用不上的数据,缓冲池里装不下几个页,命中率崩。**4KB/8KB 恰好坐在这三股力量的交点上**:对齐 OS、摊薄 IO 固定开销、给 B+树足够扇出又不浪费缓存。第 03 章会用代码实测扇出和树高的关系——你会亲眼看到改页大小如何改变点查的页读次数。

这就是反 survey 的回答方式:不是「页提高效率」,而是「三个独立的约束(OS 页、IO 固定开销、B+树扇出)各自把页大小往一个方向推,4KB/8KB 是它们共同的甜点」。

## 写路径的脊梁:WAL 与「先日志后数据」

读路径求快,写路径求稳,稳的核心是 WAL。规则一句话:**任何对数据页的修改,必须先把对应的日志记录持久化(fsync 落盘),才能让数据页的修改对外可见 / 才能把脏页刷盘。**

为什么必须这个顺序?设想反过来:先改数据页、还没记日志就掉电。重启后你看到一个被改了一半的页(torn page,撕裂页),却没有任何记录告诉你「本来想干什么」——无法恢复,数据损坏。而 WAL 顺序保证:崩溃时,要么日志里有这条记录(于是 replay 能把数据页补成正确状态),要么连日志都没有(于是这次修改当作从没发生)。两种情况都是一致的。

这里藏着耐久性逃不掉的账单:**fsync**。把字节写进文件(`write`)不等于落盘——数据可能还躺在 OS page cache 里,掉电就没。`fsync` 才强制刷到持久介质,而 fsync 是**慢的**(机械盘几毫秒,SSD 几十到几百微秒)。本书 `core/disk.ts` 把 fsync 建模成显式的耐久性边界,并能注入掉电:

> `core/disk.ts` 的 `fsync()` 在正常情况下把活跃字节复制进 `durable` 快照;但若构造时设了 `crashAt: N`,第 N 次 fsync 会**先提交此前的写、再抛 `CrashError`**——精确建模「最后一次 fsync 可能没完成」的真实保证。

`CrashError` 是个独立的异常类型(`core/disk.ts` 里 `class CrashError extends Error`),注释说明用意:「Distinct type so recovery code can catch *only* crashes, not real bugs」——恢复代码只该捕获模拟掉电,不该顺手吞掉真 bug。第 05 章实测 fsync 的真实成本,第 07 章用 `crashAt` 注入掉电、replay WAL,然后断言「恢复后的 DB 字节级等于崩溃前已提交的状态」。这个断言不是嘴上说说,是代码跑出来的——这正是本书和 survey 稿的根本区别。

## 并发的代价:MVCC 为什么用版本链

单用户的数据库简单。一旦多个事务并发读写同一行,就要回答:事务 A 正在改 id=42,事务 B 此刻读 id=42,B 该看到改前还是改后的值?

最笨的答案是加锁:B 等 A 写完。但这样读会被写阻塞,吞吐崩。MVCC(Multi-Version Concurrency Control,多版本并发控制)的答案是:**别覆盖,追加。** A 写 id=42 时不抹掉旧值,而是在版本链上挂一个新版本,带上自己的提交时间戳;B 读时拿一个快照时间戳,沿链找「提交时间 ≤ 我的快照」的最新版本——于是 B 稳稳看到一个一致的旧快照,完全不必等 A。**读不阻塞写,写不阻塞读。**

代价有两个,都不小。其一,旧版本会堆积,需要后台回收(PostgreSQL 的 `VACUUM`、InnoDB 的 purge 线程就干这个)——本书 `stage06-mvcc.ts` 用 `versionCount` 展示版本链如何变长。其二,快照隔离(snapshot isolation)并非完全没有异常:它防住了脏读,却挡不住**写偏序(write skew)**——两个事务各自基于一致快照做决策、各自合法,合起来却破坏了一个跨行约束。`stage06` 用确定性调度器(`core/scheduler.ts`,generator 协程 + 种子 PRNG 交错)精确复现这个异常,而不是靠运气撞出来。这是本书又一个反 survey 的地方:不光说「快照隔离有写偏序问题」,而是用可复现的代码把它演出来给你看。

## ⚡ 前沿钩子:磁盘消失了,「页=磁盘块」的假设还成立吗?

本书把页建模成 4KB,把 fsync 建模成昂贵的耐久性边界——这套世界观建立在一个隐含假设上:**存储是块设备,随机访问比顺序访问慢得多,落盘是一笔要省着花的开销。** 这个假设正在被硬件松动。

`io_uring`(Linux 的异步 IO 接口,2019 年进主线)让一次系统调用批量提交、批量收割成百上千个 IO,把过去「一次 IO 一次系统调用」的固定开销摊到接近零;NVMe SSD 的随机读写延迟低到几十微秒,且能并行几十万次 IOPS,「随机 vs 顺序」的鸿沟大幅收窄;更激进的是**持久内存(PMEM,Persistent Memory,如曾经的 Intel Optane)**——它字节寻址(byte-addressable)、断电不丢、延迟接近 DRAM。在 PMEM 上,「读必须整页搬进缓冲池、改完再整页刷回、还得 fsync」这套流程的每一个前提都松动了:你可以直接按字节读写持久存储,缓冲池和 fsync 仿佛多余。学术界为此造了一批新结构(如 PMEM 上的 B+树变体 BzTree、FPTree),试图绕开页和缓冲池。

那为什么本书还在教页?为什么 PostgreSQL、RocksDB 这些主力引擎到今天还是页 + 缓冲池 + WAL?**因为页抽象的价值远不止「适配块设备」。** 页同时还是:崩溃恢复的原子单位(WAL 按页记 before/after image)、并发控制的锁粒度、缓冲池的换入换出单位、B+树节点的容器。就算底层介质支持字节寻址,你依然需要一个「成批管理、原子恢复、统一缓存」的单位——而那个单位的形状,恰好还是页。Optune 产品线已停产、PMEM 的生态没起来,更说明硬件替代页抽象的路远比想象艰难。**目前没有通用解:在 PMEM / 极速 NVMe 上彻底抛弃页抽象的引擎设计仍是开放研究问题**,业界的现状是「页抽象松动但没死,缓冲池在变薄(direct IO、用户态缓存),WAL 在变形(group commit、并行 replay),但核心骨架还在」。你在这本书里手写的页 + 缓冲池 + WAL,既是过去四十年的工程结晶,也是理解「为什么它还没被取代」的唯一办法。

## 怎么读这本书

这本书的承诺是:**每个机制都有可跑的代码、可测的数字、想清楚的取舍、以及一个老实标注「还没有通用解」的开放问题。** 配套仓库 `examples/database-from-scratch/` 零运行时依赖、离线可跑、确定性可复现(`npm install` 后逐章 `npm run stage0N`)。

往下走的路径:第 **01 章**从最底层开始——一个 4KB 的页里,变长的行怎么摆(slotted page),实测一页能塞多少行、页满了怎么失败。它会把本章定义的 `tid = (pageId, slotId)` 落成真实的字节布局。第 **02 章**在页之上盖缓冲池,实测命中率如何直接决定真实磁盘 IO 次数。一路盖到第 **07 章**崩溃恢复——那一章会回收本章埋下的所有伏笔:用 `crashAt` 注入掉电,靠 lsn 顺序 replay WAL,最后断言恢复后的数据库字节级等于崩溃前。

读的时候别只读,跟着跑。本书最核心的那句话和姊妹书库一致:**不跑代码 = 没学。** 每章结尾打印的数字是代码真算出来的,不是我编的——你在自己机器上跑出的相对趋势(命中率随缓存大小怎么变、树高随数据量怎么涨)会和书里一致,这正是「机制是真的」的证据。
