> 用第 01 章的槽页 (slotted page) 当节点、第 02 章的缓冲池 (buffer pool) 做缓存,我们手写一棵真在磁盘上分页的 B+树:内部节点二分查找、叶子链表、插入分裂与根上提、范围扫描。本章给出索引的核心直觉——扇出决定高度、高度决定 IO 次数——它是第 04 章 LSM 树的对照组,也是第 06 章查询执行器选 index scan 还是 full scan 的依据。

先把结论摆在桌上:一棵存了一亿行的 B+树,点查 (point lookup,按单个键找一行) 只要读 3 个页。不是"大约 3 个",是恰好 3 个——我们这章会用计数器把它逼出来,触页数 (touched pages) 必须严格等于树高度,一个不多一个不少。这个"3"不是经验值,是 `PAGE_SIZE`、键宽、扇出三者算出来的算术。理解这章,本质就是理解这条算术为什么成立、以及它在什么情况下崩掉。

如果你只记一句话:**B+树的全部价值,是把"高度"这个直接等于磁盘 IO 次数的量,压到 3 或 4,而且让它对行数几乎不敏感。** 一亿行高度 3,十亿行还是高度 4。下面拆开讲为什么。

## 为什么是 B+树,不是平衡二叉树或红黑树

这是面试和真实选型里都绕不过的问题,也是这章的 ✦ 专家钩子。答案不在"哪种树更平衡",而在**磁盘的物理读取单位**。

内存里,一次内存访问读一个 cache line (大约 64 字节),所以红黑树、AVL 这类二叉树是合理的——每下降一层付一次访问,树高 `log₂(n)`,一百万个键 20 层,每层一次内存访问,可以接受。

但磁盘 (以及 SSD) 的最小读取单位是**一整页**,我们这本书定为 4096 字节 (见第 01 章 `core/disk.ts`)。这里有个残酷的不对称:读 4 字节和读 4096 字节,磁盘付出的代价几乎一样——主要成本是寻道/通道往返,不是搬运字节。

那么问题来了:既然一次 IO 无论如何都把整页 4096 字节搬上来了,你用二叉树,这一页里只放了一个键加两个子指针,等于花了一次 IO 只换来 `log₂` 里的"1 bit 决策"。这是巨大的浪费。B+树的设计就是把这次 IO 用满:**一页里塞几百个键和几百个子指针**,一次 IO 带来的不是二分,而是几百分。树高于是从 `log₂(n)` 变成 `log_fanout(n)`,底数从 2 变成几百。

把数字代进去就一目了然。一亿行:

- 二叉树 (底数 2):约 27 层 → 27 次 IO
- B+树 (底数 340,下面会算出这个 340):3 层 → 3 次 IO

9 倍的 IO 差距,且行数越大差距越夸张。这就是为什么所有面向磁盘的数据库——PostgreSQL、MySQL/InnoDB、SQLite——主索引都是 B+树或其变种,而没人在磁盘上用红黑树。**红黑树优化的是比较次数,B+树优化的是 IO 次数,后者在磁盘上贵几个数量级。**

顺带说清 B+树和 B-树 (注意没有加号) 的区别,因为代码里有处关键决策依赖它:B-树的内部节点也存数据 (值),B+树的内部节点**只存键当路标,所有值都在叶子层**,而且叶子之间用指针串成链表。这两点带来两个直接好处:内部节点不存值 → 同样一页能塞更多路标 → 扇出更大 → 树更矮;叶子成链 → 范围扫描可以"下降一次然后顺着链走",不用为每个键重新从根找。后面范围扫描那节会兑现这第二点。

## 容量算术:340 这个数从哪来

不要把扇出当魔法常数背下来,它是页布局直接算出来的。我们的代码把这步做成了可调用的方法,而不是写死的注释——见 `examples/database-from-scratch/src/stage03-bplus-tree.ts` 里的 `leafCapacity()` 和 `maxInternalChildren()`:

```ts
/** Max (key,value) entries a leaf holds = usable bytes / (record + slot entry). */
leafCapacity(): number {
  const usable = PAGE_SIZE - NODE_HEADER_SIZE;
  return Math.floor(usable / (LEAF_RECORD_SIZE + SLOT_ENTRY_SIZE));
}

/** Max children an internal node holds. Each child after the first carries a
 *  separator; the first does not. Solve for n where:
 *    n*(child+slot) + (n-1)*sep <= usable.  */
maxInternalChildren(): number {
  const usable = PAGE_SIZE - NODE_HEADER_SIZE;
  const perChild = INTERNAL_CHILD_SIZE + SLOT_ENTRY_SIZE + INTERNAL_SEP_SIZE;
  return Math.floor((usable + INTERNAL_SEP_SIZE) / perChild);
}
```

代入本章常数 (页 4096B,键 4B 的 u32,值 4B 的行指针,槽目录每项 4B),运行输出是:

| 量 | 值 | 推导 |
|---|---|---|
| leaf 每页 entry 上限 | 340 | `floor((4096-11)/(8+4))` |
| internal 扇出 (子节点上限) | 340 | `floor((usable+sep)/(child+slot+sep))` |

这里有几处值得停下来看的设计决策。

**为什么键是定长 u32 而不是变长字符串。** 这是为教学刻意做的简化,但它不是偷懒,而是为了让扇出成为一个读者能手算验证的**诚实常数**。变长键 + 前缀压缩会让扇出变成一个模糊的平均值,你没法说"扇出就是 340",只能说"大约几百"。把它定死,这章唯一的那条算术 (扇出 → 高度) 才看得清。变长键、前缀压缩是后面的事——代码注释里也明确把它标成"explicitly a later concern",在这里硬上只会糊掉唯一的那课。

**为什么内部节点是 N 个孩子配 N-1 个分隔键 (separator)。** 分隔键就是路标:它告诉你"键小于我往左走,大于等于我往右走"。N 个孩子之间只有 N-1 个间隙,所以最左孩子不带分隔键。代码里 `encodeInternalRecord` 对 slot 0 传 `null` 就是这个意思——这不是省一点空间,而是 B+树语义本身:最左子树是"比第一个路标还小"的兜底。

**为什么叶和内部的扇出碰巧都是 340。** 巧合,别赋予它意义。叶记录是 `键4 + 值4 = 8` 字节加 4 字节槽;内部记录是 `孩子指针4 + 分隔键4 = 8` 字节加 4 字节槽。两边每项都是 12 字节,所以容量一样。换个键宽或值宽,两者立刻分开。

把 340 这个扇出乘起来,就得到本章标题那句话的硬证据:

| 高度 | 最大行数 | 点查 IO (页) |
|---|---|---|
| 1 | 340 | 1 |
| 2 | 115,600 | 2 |
| 3 | 39,304,000 | 3 |
| 4 | 13,363,360,000 | 4 |

高度 3 就能装约 3930 万行,高度 4 装 133 亿行——远超十亿。**这就是"为什么三层"的全部秘密:每加一层,容量乘以扇出 (×340),而 IO 只加一 (+1)。** 指数增长的容量,线性增长的代价。十亿行落在高度 4,但实际系统里因为节点装不满、加上键更宽,真实电商主表常见高度就是 3~4,和这张表完全对得上。

⚡ 顺带钉一个边界:上表是**理论上限** (每个节点都装满)。真实树永远装不满 (后面利用率那节会量到只有 66.7%),所以同样高度装的行数比表里少。但这不改变结论——利用率打个对折,高度也最多多一层。容量对高度是指数级冗余的,这正是 B+树健壮的来源。

## 节点长什么样:复用第 01 章的槽页

B+树的节点不是新发明的数据结构,就是第 01 章那个槽页 (slotted page,页内用一个槽目录指向变长记录)。这是这本书刻意的连续性:行存页、索引节点页,用同一套页布局纪律。代码里节点头在共享槽页头之上,只多叠了一个字:

```ts
const NEXT_LEAF_OFF = SLOTTED_HEADER.HEADER_SIZE; // u32
const NODE_HEADER_SIZE = NEXT_LEAF_OFF + 4;
```

`NEXT_LEAF` 是叶子的右兄弟页号,这是 B+树 (相对 B-树) 的命脉——叶子靠它串成链。内部节点这个字段不用。这里有个微妙的哨兵 (sentinel) 设计值得学:

```ts
/** No-sibling sentinel for the leaf chain. Page 0 is always the meta page, so it
 *  can never legitimately be a leaf's right sibling — 0 unambiguously means "end".*/
const NO_SIBLING = 0;
```

用 0 表示"没有下一个叶子",前提是 0 号页永远是元数据页 (meta page),绝不可能是真叶子。这是一个典型的"用业务不变量换掉一个额外标志位"——不用单独存一个 bool 说"我是不是最后一个",直接让 0 这个本来就非法的值承担"结束"语义。建表时代码确实先 `disk.allocPage()` 占掉 0 号页保证这点。

## 缓冲池:让 IO 计数说真话

这章所有"读了几个页"的数字,不是代码里手动 `count++` 出来的,是缓冲池的**缓存未命中 (cache miss)** 计数器自然吐出来的。这点很重要:如果手动数,容易数成"逻辑上应该读几次";让缓存 miss 来数,数的是"真从磁盘搬了几次"。代码里 `BufferPool` 在 `fetch` 里区分命中和未命中:

```ts
private fetch(pageId: number): Uint8Array {
  const cached = this.frames.get(pageId);
  if (cached !== undefined) {
    this.touch(pageId);
    return cached;
  }
  // Cache miss => a real Disk read. This is the IO we measure.
  const bytes = this.disk.readPage(pageId);
  this.admit(pageId, bytes);
  return bytes;
}
```

这就是为什么"点查读几个页"必须先把缓存清空 (cold cache,冷缓存) 再量——否则根节点和上层内部节点都还热在池里,你量到的是 0 次读,那是稳态而非下降成本。两个数都得说,才算诚实:冷缓存量"一次完整下降的 IO",热缓存才是生产稳态。

池的容量是故意调小的 (256 个帧,约 1MB),小到 10 万键的构建过程真的会发生淘汰 (eviction),否则"缓冲池"就是个摆设;但又大到根和上层内部节点能常驻。淘汰用写回 (write-back):脏页淘汰前必须落盘,否则修改就丢了——这是写回池经典的丢写 bug,代码 `evictOne` 里特意处理:

```ts
if (this.dirty.has(victim)) {
  // Write-back: a dirty page must hit disk before its frame is reclaimed, or
  // the mutation vanishes — the classic lost-write bug a write-back pool risks.
  this.disk.writePage(victim, this.frames.get(victim)!);
  this.dirty.delete(victim);
}
```

## 插入:分裂如何自底向上、根如何上提

插入的难点不在"找到叶子塞进去",而在塞不下时怎么办。B+树的回答是**分裂 (split)**:满了的节点对半切成两个,把中间的分隔键交给父亲;如果父亲也满了,父亲继续分裂、继续往上交;一路交到根,根也满了,就**新建一个根**,树长高一层。

这是全书唯一让树高度增加的路径,代码 `insert` 里写得很直白:

```ts
if (result.promoted) {
  // Root split: grow the tree by one level. This is the ONLY way height
  // increases, and it happens rarely (every ~fanout^height inserts), which is
  // exactly why height stays tiny.
  const newRoot = this.pool.alloc();
  initNode(this.pool.getMut(newRoot), PageType.INTERNAL);
  const rootPage = this.pool.getMut(newRoot);
  appendRecord(rootPage, encodeInternalRecord(this.rootId, null)); // old root = left child
  appendRecord(rootPage, encodeInternalRecord(result.promoted.rightChild, result.promoted.sep));
  this.rootId = newRoot;
}
```

注意那句注释:根分裂"每约 `fanout^height` 次插入才发生一次"。这就是高度稳定的根本原因——长高一层需要先把当前这层全部填满,而填满需要指数级的插入量。十万次插入里,根分裂只发生区区两三次。

叶子分裂还有一个 B+树相对 B-树的关键区别,藏在 `splitLeaf` 的最后一行:

```ts
// Separator = first key of the right node. In a B+tree, leaf keys are NOT
// removed on split (unlike a B-tree); the separator is a COPY pushed up.
const sep = rightEntries[0].key;
return { promoted: { sep: copyKey(sep), rightChild: rightId } };
```

分隔键是右节点首键的**拷贝**上推,原键仍留在叶子里。B-树会把这个键从叶子搬走、只留在内部——这样内部节点就承担了存数据的职责,代价是范围扫描必须遍历内部节点。B+树宁可让这个键在内部"重复"一份当路标,也要保证**所有真实数据都齐整地躺在叶子链上**,范围扫描才能只走叶子。这处 `copyKey` 还防一个具体 bug:分隔键如果直接引用页内的视图 (subarray),那块帧一旦被缓冲池淘汰复用,这个键就被悄悄改写了——这是 buffer pool 之上做树最容易踩的别名 (aliasing) 坑。

内部节点分裂的规则又有一处不同——**中间分隔键是被"挤上去"而非"复制上去"的**:

```ts
// Internal split PUSHES the middle separator up (it does NOT stay in either
// child — that's the B-tree/B+tree internal-node rule). ...
const mid = Math.floor(children.length / 2);
const promotedSep = children[mid].sep!;
```

记住这个对比就抓住了 B+树的精髓:**叶子分裂复制键上推 (键留下),内部分裂挤压键上推 (键不留)**。前者保证数据完整在叶层,后者保证路标不冗余浪费内部空间。

## 实测:在 10 万随机键上把不变量逼出来

光讲机制是 survey 写法。这章的态度是:跑一遍,让数字证明。代码用固定种子 (`seed=0x5eed`) 生成 10 万个去重随机键插进去,然后**读真页**量出扇出、高度、利用率,而不是断言:

| 指标 | 值 | 说明 |
|---|---|---|
| 实测扇出 (叶上一层均值) | 220.5 | 2 个内部节点真从页读出,对比容量 340 |
| root 子节点数 | 2 | root 通常远不满,故扇出看叶上一层而非 root |
| 树高度 | 3 | root→leaf 实走层数 |
| 叶子页数 | 441 | leaf 链长度 |
| 叶利用率 | 66.7% | 已存 entry / 容量 |
| 构建总磁盘写 | 9166 | buffer pool 回写次数 (measured) |

两个数字需要解释,因为它们暴露了真实树和理论上限的差距。

**为什么实测扇出是 220.5 而不是容量 340。** 节点平时不会装满——它在被填到满之前就可能因为别处分裂而停在某个中间状态。220.5 / 340 ≈ 65%,和叶利用率 66.7% 同源:随机插入下,节点平均填到约 2/3 满。代码 `measuredFanout()` 特意量"叶子上面那一层"而不是根,因为根几乎总是远不满 (这里根只有 2 个孩子)——拿根算扇出会把容量结论说成假的。这是个诚实性细节:报哪个数,取决于哪个数支撑你的论点而不歪曲它。

**为什么高度是 3。** 10 万行,查上面的容量表落在高度 3 的区间 (11.5 万行),完全对得上。10 万行 3 次 IO。

接着是这章的核心不变量验证,也是它区别于纸上谈兵的地方——**每次点查触的页数,严格等于树高度**:

| 查询键 | 命中值 | 磁盘读 (页) | ==高度 |
|---|---|---|---|
| 7803 | 7804 | 3 | yes |
| 49474 | 49475 | 3 | yes |
| 71003 | 71004 | 3 | yes |
| 7 | 8 | 3 | yes |
| 42 | 43 | 3 | yes |

代码在每次探测前 `evictAll()` 强制冷缓存、`resetStats()` 归零计数,然后断言读页数恰好等于高度,不等就直接 `assertEq` 报错:

```ts
const reads = randDisk.stats().reads;
invariant(v === k + 1, `search(${k}) must return ${k + 1}, got ${v}`);
assertEq(reads, height, `point lookup must read exactly ${height} pages (= height)`);
```

更值得玩味的是**未命中的键也读恰好 3 页**:查一个根本不存在的键,照样老老实实从根下降到叶,在叶里二分发现没有,返回 null。运行输出原话:"未命中键也读 3 页 (=高度):search 成本与命中与否无关,只看高度。" 这是 B+树成本模型干净的体现——点查的代价由树高度单独决定,和数据在不在、键的具体值都无关。这条直觉直接服务第 06 章:查询执行器估算 index scan 成本时,可以拿"高度"当一个稳定的常数,不用管选择率 (selectivity)。

## 范围扫描:下降一次,然后顺着叶链走

点查是 B+树的基本功,范围扫描才是它相对哈希索引 (hash index) 的杀手锏。哈希索引点查 O(1) 很爽,但"找出所有 50 < key < 5000 的行"它彻底抓瞎——哈希把键打散了,范围在它眼里毫无结构。B+树的叶链让范围扫描变成:下降一次到起点叶,然后**顺着 `NEXT_LEAF` 指针走,一个叶接一个叶,完全不回头找根**:

```ts
rangeScan(lo: number, hi: number): number[] {
  // ...
  let pageId = this.descendToLeaf(loKey);
  for (;;) {
    const page = this.pool.get(pageId);
    // ... 收集 [lo, hi) 区间内的值 ...
    const sib = nextLeaf(page);
    if (sib === NO_SIBLING) return out;
    pageId = sib;
  }
}
```

跑 1000 个键的范围扫描,实测吞吐:

| 指标 | 值 | 类型 |
|---|---|---|
| 每次扫描返回键数 | 1000 | deterministic |
| 扫描 ops/sec | 4,065 | MEASURED |
| 键吞吐 keys/sec | 4,065,261 | MEASURED |
| ns/扫描 | 245,987 | MEASURED |

这里必须把话说死,免得有人拿 400 万 keys/sec 去吹:**这个绝对值偏乐观,不可迁移**。它是内存模拟磁盘 + 全驻留缓冲池下的墙钟,真实磁盘上会慢几个数量级。运行输出自己就标注了:"可迁移的是「单次下降一次、之后顺链」这一相对结构,不是 keys/sec 数字本身。" 你该带走的不是这个数,是这个**结构**——范围扫描的代价 ≈ 一次下降 (高度次 IO) + 区间跨越的叶子页数,而不是每个键各下降一次。这正是第 06 章里"范围查询为什么偏爱 index scan、什么时候反而该退回 full scan"的判断依据:如果区间宽到要扫过半数叶子,顺链的好处被磁盘随机 IO 吃掉,full scan 的顺序读反而赢。

## 失败模式:自增主键会让索引膨胀近一倍

到这里 B+树看着很完美。现在讲它怎么坏——而且是以最常见的方式坏:**单调递增键插入**,比如自增主键 (auto-increment primary key)。这是几乎每张表都有的模式,也是 half-split 这个教科书规则的滑铁卢。

机制是这样的:half-split 把满叶对半切。随机插入时,左右两半将来都还会有键插进来,慢慢都会填满,所以平均利用率约 2/3,没问题。但**顺序插入时,所有新键永远落在最右边**——一旦某个叶分裂,它的左半永远不会再被插入,因为后来的键全都更大、全都去了右边。左半就这样永久卡在 50% 满。每个分裂出来的左叶都浪费一半空间。量出来的对比:

| 插入模式 | 叶利用率 | 叶子页数 | 磁盘写 |
|---|---|---|---|
| 随机键 + half 分裂 | 66.7% | 441 | 9166 |
| 顺序键 + half 分裂 (退化) | 50.4% | 584 | 588 |
| 顺序键 + 右侧特判 (修复) | 99.7% | 295 | 296 |

顺序插入把利用率从 66.7% 砸到 50.4%,叶子页数从 441 涨到 584——**同样 10 万行,索引多占了约一倍空间**,也多占一倍缓存。对一个"最常见的插入模式"来说,这是不可接受的退化。

修复办法是**右侧特判 (rightmost fast-path)**:当新键是当前最大、且落在最右叶 (没有右兄弟) 时,别对半切,只把这一个新键切到右节点,让左叶保持 100% 满。代码里这就是 `splitLeaf` 的策略分叉:

```ts
let splitAt: number;
if (this.policy === "rightmost-aware" && oldNext === NO_SIBLING && this.isAppend(entries)) {
  splitAt = entries.length - 1; // right node gets only the last (new) key
} else {
  splitAt = Math.ceil(entries.length / 2);
}
```

加了这一处特判,利用率回到 99.7%,叶子页数降到 295——比随机插入的 441 还少 (因为几乎不浪费)。这正是**批量装载 (bulk load) 的雏形**。真实引擎都做这件事:PostgreSQL 的 B+树对最右页有 fast-path,SQLite 顺序插入也有类似优化,InnoDB 对自增主键专门走右侧插入路径。**不做这个,自增主键这个最常见模式会让你的索引膨胀近一倍**——这不是教学玩具的边角,是生产系统必须处理的真实成本。

最后还有一个反直觉的观察,藏在"磁盘写"那一列:随机键构建写了 9166 次,顺序键只写了 588 次,差了 15 倍。利用率明明随机更好,为什么写得更多?这是**缓冲池效应**,和利用率是两笔独立的账:

> 随机访问颠簸 LRU 反复回写脏页,顺序访问对池友好。

随机插入命中的叶子页在缓冲池里跳来跳去,刚被改脏的页很快被淘汰、被迫写回,过会儿又被读回来再改——反复颠簸 (thrashing)。顺序插入永远只动最右那几个页,它们稳稳热在池里,改很多次才落盘一次。**这说明索引的真实成本不只看空间利用率,还看访问局部性 (access locality)。** 同一棵树,换个插入顺序,空间和 IO 两个维度往相反方向动——这也是第 04 章 LSM 树要正面解决的问题:LSM 干脆放弃就地更新,把所有写变成顺序追加,用结构换掉这种随机写颠簸。本章的 B+树就是它的对照组。

## ⚡ 前沿:当"树"被回归函数部分替代

B+树几十年来稳如磐石,但近年有两条研究线在啃它的根基,都还没有通用解,值得知道边界在哪。

**第一条:无锁 / append-only 的 B+树变种,代表是 Bw-tree。** 传统 B+树的痛点是高并发下的页内原地修改 (in-place update)——多个线程同时改一个页需要加锁,锁是并发杀手。Bw-tree (微软为内存数据库 Hekaton 设计) 的思路是:不原地改页,而是把每次修改记成一个"delta 记录",用一个映射表把逻辑页号指到一串 delta 链,改动靠无锁的 CAS (compare-and-swap,比较并交换原子指令) 追加到链头。读的时候顺着 delta 链叠加。这把原地更新换成了追加,绕开了页级锁——你会发现这和第 04 章 LSM 树、以及上一节"顺序写对池友好"的直觉是同一脉络:**用追加换掉原地随机写**。代价是 delta 链会变长、需要后台合并 (consolidation),工程复杂度陡增,至今没成为主流默认结构,只在特定内存引擎里用。

**第二条更激进:学习型索引 (learned index),代表是 RMI (Recursive Model Index,递归模型索引)。** 它的核心质疑很大胆:B+树本质上是把"键 → 它在排序数组里的位置"这个映射,用一棵树硬编码下来。但如果数据分布是有规律的 (比如键近似均匀),这个映射其实是一条可以用模型拟合的曲线——那为什么不直接训练一个回归模型来**预测**位置,而要存一整棵树?RMI 用一层层小模型 (顶层模型预测该交给哪个下层模型,逐级细化) 替代内部节点,预测出大致位置后,在一个小误差窗口内做局部搜索修正。在静态、分布友好的只读数据上,它能比 B+树更快更省内存——因为模型比一堆指针小得多。

但它的边界很硬,这正是"还没有通用解"的地方:一是**写**——B+树天然支持插入,学习型索引一旦数据变了模型就得重训或留出缓冲,在线更新至今是开放难题;二是**最坏情况保证**——B+树的高度是有上界的、可证明的,而模型预测错了要兜底,对抗性 (adversarial) 或分布突变的数据上,误差窗口可能退化到全表扫描,失去了 B+树那种"无论数据长什么样都是 log_fanout"的稳定保证。所以你今天能看到学习型索引出现在论文和一些只读分析场景的原型里,但没有任何生产 OLTP 数据库敢把主索引换成它——**它能赢的前提 (静态、分布可学、读多写少) 恰好是 B+树最不在乎的那些假设**。

把这两条放一起看,本章那条核心算术 (扇出 → 高度 → IO) 之所以还没被推翻,正是因为它不假设任何数据分布、不需要训练、写和最坏情况都有硬保证。前沿在试图用"知道数据长什么样"换性能,而 B+树的护城河恰恰是"我不需要知道数据长什么样"。这个 trade-off 边界,目前没有谁通吃。
