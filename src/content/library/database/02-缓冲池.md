> 第 01 章里,我们把变长行塞进一个 4096 字节的页,读出来字节级一致。但那一章的每一次 `getPage` 都是一次 `Disk.readPage`——一次真实 IO。一个查询要遍历 B+树三四层、扫几百个叶子页,如果每个页都打一次盘,你的数据库就是个"会发 SQL 的文件读取器",慢到没法用。本章只解决一件事:**在磁盘和上层之间塞一层定长内存,让大部分页访问命中内存、不碰盘**。这层内存就是缓冲池(buffer pool,数据库自己管的页缓存)。

缓冲池听起来就是个 cache,但它和你写业务时随手 `new Map()` 当缓存有三个本质区别,每一个都是这章的主线:

1. **它管的是有限的帧(frame,一块固定大小的内存槽,正好放一个页)**。帧数在启动时定死,装满了就必须踢人——于是有了"踢谁"的替换策略(replacement policy)。
2. **它的 cache entry 可能正在被别人读着字节**。上层拿到的是帧里的活内存,不是副本。这一刻把帧偷走换成别的页,等于在读者眼皮底下掉包数据。于是有了 **pin(钉住,引用计数 +1,禁止被踢)**。
3. **它的 cache entry 可能比磁盘上的版本新**。写操作只改内存帧、不立刻落盘,这个帧就是"脏的"。踢一个脏页前必须先刷盘,否则数据丢了。于是有了 **dirty 标记**和"何时刷"的决策——这个决策直接连到第 05 章 WAL 的 force/no-force。

配套代码全在 `examples/database-from-scratch/src/stage02-buffer-pool.ts`,一个文件、两种替换策略(LRU 与 CLOCK)、两种对抗性负载(Zipf 热点 vs 全表扫描)。所有命中率、磁盘读次数都是**精确确定性计数**(seeded PRNG,种子 42),不是估算。

## 一、page table:从"页号"到"内存帧"的那张表

缓冲池的核心数据结构其实只有两个:一个定长的帧数组,和一张"页号 → 帧下标"的哈希表。后者就是 page table。看代码里的 `BufferPool` 字段:

```ts
private frames: Frame[] = []; // dense; index is the frame slot
private byPage = new Map<number, number>(); // pageId -> frame index, O(1) lookup
```

`byPage` 就是 page table。每次有人要页 `pageId`,先查这张表:在表里 = **命中(hit)**,直接返回帧里的字节,零 IO;不在 = **未命中(miss)**,得找个空帧(或踢一个)、发一次 `disk.readPage`、把页装进去。`getPage` 的开头三行把这个分叉讲得很干净:

```ts
getPage(pageId: number): Uint8Array {
  const existing = this.byPage.get(pageId);
  if (existing !== undefined) {
    this.stats.hits++;
    const frame = this.frames[existing];
    frame.pinCount++;
    this.touch(existing);
    return frame.bytes;
  }
  // ... miss 路径:acquireSlot() 可能踢人,然后 disk.readPage
```

注意 miss 路径里那行注释把全章的目的浓缩成一句话:`disk.readPage` 是"the ONE unavoidable IO of a miss. Everything in this file exists to make this call rarer"。整章所有机制——替换策略、脏页延迟刷盘——都是为了让这一行少执行几次。

**为什么是哈希表而不是直接拿页号当数组下标?** 因为页号空间是稀疏的:一张表可能有几百万个页,缓冲池只有几千个帧。你不可能开一个几百万长的数组让大部分槽空着。哈希表把"几百万种可能的页号"映射到"几千个真实帧",这是 page table 存在的全部理由。这也埋下一个伏笔:每次访问都要过一次哈希查找,这个固定开销在第六节的前沿话题里会被 pointer swizzling 直接绕掉。

## 二、pin / unpin:cache entry 正在被读,不能偷

普通 cache 的 entry 是不可变副本,你随便淘汰,读者顶多拿到旧值。缓冲池不行——`getPage` 返回的是**帧的活字节**(`frame.bytes`),上层可能正拿着这个 `Uint8Array` 在解析行、在沿 B+树指针往下跳。这一刻你把帧淘汰、装进另一个页,读者手里的引用还指着同一块内存,内容已经变成别的页了。这是典型的 use-after-free 级别的数据腐坏,而且静默无报错。

防线就是引用计数:`getPage` 把 `pinCount++`(钉住),用完必须 `unpin` 把它减回去。替换策略选 victim(牺牲品,要被踢的帧)时,**跳过所有 `pinCount > 0` 的帧**:

```ts
private chooseLruVictim(): number {
  for (const slot of this.lruOrder) {
    if (this.frames[slot].pinCount === 0) return slot;
  }
  this.stats.evictFailures++;
  throw new BufferFull(this.frameCount);
}
```

这里有个值得停下来看的设计决策:**脏标记不在 `getPage` 时打,而在 `unpin` 时由调用方声明**:

```ts
unpin(pageId: number, markDirty: boolean): void {
  // ...
  frame.pinCount--;
  if (markDirty) frame.dirty = true;
}
```

为什么?因为 `getPage` 区分不出调用方是要读还是要写——它只是"把页钉住给你"。读者 pin 完只读、`unpin(id, false)`;写者 pin 完改了字节、`unpin(id, true)` 声明"我弄脏了"。把脏标记的责任放在 unpin、让调用方显式声明,既避免了"读一下也算脏、白白触发刷盘"的浪费,也让"谁弄脏的"这件事在代码里可追溯。这是命名层面的副作用契约:`markDirty` 这个布尔参数本身就是 API 在说"你得告诉我你是不是写者"。

### ✦ pin 用完不还,缓冲池会死锁:BufferFull

pin 的代价是:**调用方有义务及时 unpin,否则缓冲池会卡死**。代码末尾的 `demoBufferFull()` 故意演示这个失败模式——把所有 4 个帧全 pin 住、一个都不还,然后请求第 5 个页:

```ts
const pool = new BufferPool(disk, frames, "LRU"); // frames = 4
for (let pageId = 0; pageId < frames; pageId++) pool.getPage(pageId); // 全 pin,不 unpin
pool.getPage(frames); // 页 4 既不在池里,又没有可踢的 victim
```

实测输出:

```
pinned all 4 frames (pages 0..3), none unpinned
caught as expected: buffer pool full: all 4 frames are pinned, no victim to evict
pool recorded evictFailures=1 (each = a scan that found every frame pinned)
after unpinning page 0, the same getPage(4) succeeds — liveness restored by prompt unpin
```

我们的实现选择**抛 `BufferFull` 异常**(fail-fast),而真实数据库通常选择**阻塞**:请求者在这里挂起,等别人 unpin 出一个帧再继续。两种都对,但都暴露同一个深坑——**如果挂起的请求者自己手里也攥着一个 pin,而别的等待者正需要那个 pin,就是经典死锁**。这就是注释里那句 "deadlock precursor(死锁前兆)"的含义。

注意失败后的恢复:`unpin(0)` 还回一个帧,**同一个 `getPage(4)` 立刻成功**。这证明缓冲池没坏,它只是在正确地拒绝违反 pin 不变量——宁可报错,不偷正在被读的帧。这是本书反复强调的态度:与其静默地把数据搞坏,不如响亮地拒绝。

为什么我们这层选 fail-fast 而非阻塞?因为这是单线程教学代码,没有"另一个线程会来 unpin"这回事——阻塞就是真死锁、永远不返回。fail-fast 让失败立刻可见、可断言。生产系统是多线程,阻塞才有意义(别人会还),但代价是你得另上一套死锁检测。**这是个典型的"取决于并发模型"的设计选择,没有绝对对错**。

## 三、脏页:写不落盘,踢出时才刷(write-back)

缓冲池对写的处理有两条路线:**write-through(写穿透)**——每次写内存帧的同时立刻写盘;和 **write-back(写回)**——只改内存、打脏标记,等这个帧被淘汰时才刷盘。我们选 write-back,看 `evict`:

```ts
private evict(slot: number): void {
  const frame = this.frames[slot];
  if (frame.dirty) {
    this.disk.writePage(frame.pageId, frame.bytes);
    this.stats.writebacks++;
  }
  this.byPage.delete(frame.pageId);
  // ...
}
```

注释把好处说透了:"Writeback-on-evict (not write-through) is why a hot dirty page costs zero IO until it finally leaves the cache"。一个被反复改写的热点页(比如计数器、热点账户余额),write-through 下每次写都打一次盘;write-back 下,它在内存里被改一百次也只在最后离开缓存时刷一次盘。**写 IO 从一百次压到一次**,这就是 write-back 的全部价值。

代价同样直白:**机器在脏页刷盘前崩溃,内存里那些"改了但没落盘"的更新全丢**。你以为提交成功的事务,重启后凭空消失。光靠缓冲池,write-back 是不安全的。

### ✦ no-force 的伏笔:这个坑由第 05 章 WAL 来补

write-back 在数据库术语里叫 **no-force**——提交时不强制把脏页刷盘。它和上面说的崩溃丢数据是同一件事的两个名字。解法不是放弃 write-back(那会把写性能打回原形),而是**先写一条 WAL(write-ahead log,预写日志)记录这次改动、再改内存页**:只要日志落了盘,崩溃后就能 replay 日志把丢失的内存更新重建出来。于是缓冲池可以放心地攒脏页、晚点刷,持久性由 WAL 兜底。

所以本章的 `evict` 里那句"脏了才 `writePage`"不是孤立优化,它是第 05 章 WAL force/no-force 策略的下半场。这里先把"脏页可以攒着"这个能力建出来,第 05 章再补上"攒着也不会丢"的保证。两章合起来才是完整的故事:**缓冲池负责快,WAL 负责不丢**。

## 四、替换策略一:LRU,以及它为什么会被一次扫描冲垮

帧装满了、还要装新页,就得踢一个。最直觉的策略是 **LRU(least recently used,踢最久没被碰过的)**:假设"最近用过的,接下来还会用"。代码用一个数组维护访问顺序,`touch` 在每次访问时把帧挪到队尾(最新),`chooseLruVictim` 从队头(最旧)找第一个没被 pin 的:

```ts
private touch(slot: number): void {
  if (this.policy === "LRU") {
    this.lruOrder = this.lruOrder.filter((s) => s !== slot); // 从旧位置摘掉
    this.lruOrder.push(slot);                                 // 接到队尾 = 最新
  } else {
    this.frames[slot].refBit = true; // CLOCK 走另一条路,见第五节
  }
}
```

(这里的 `filter` 实现是 O(n) 的,只为教学可读;真实系统用侵入式双向链表 + 哈希做到 O(1),但那是工程细节,LRU 的语义不变。)

LRU 在**有热点的负载**下表现很好。看实测——负载 `zipf-hot` 是 Zipf 分布(少数热点页吃掉大部分访问,典型的 OLTP 形状),64 页的表、8 个帧的池(只缓存 13%)、2000 次访问:

```
workload   policy  accesses  hits  misses  hitRate%  diskReads  evictions
---------  ------  --------  ----  ------  --------  ---------  ---------
zipf-hot   LRU         2000   808    1192      40.4       1192       1184
zipf-hot   CLOCK       2000   753    1247      37.6       1247       1239
```

只用 13% 的内存就接住 40.4% 的访问——热点页一旦进了缓存就一直命中,LRU 把它们牢牢留住。这正是缓冲池存在的意义。

> ⚠ 数字诚实声明:这里 40.4% 是 64 页 toy 表的精确确定性计数,不是生产 OLTP 缓冲池的典型值(后者通常 95%+,因为表大、池占比也常更高)。**能跨场景迁移的不是这个绝对数,而是 LRU vs CLOCK 的相对关系和扫描下的崩溃模式**。

### ✦ 全表扫描:LRU 的最坏情况,命中率跌到 0

现在换一个负载:全表顺序扫描,`0,1,2,...,63,0,1,...` 循环。这是 LRU 的天敌:

```
full-scan  LRU         2000     0    2000         0       2000       1992
full-scan  CLOCK       2000     0    2000         0       2000       1992
```

**命中率 0%。2000 次访问,2000 次打盘,一次都没命中。** 为什么?池只有 8 个帧,扫描有 64 个页。当扫描转完一圈、回头要页 0 时,LRU 早就把页 0 当作"最久没用的"踢掉去腾地方给页 8、9... 了。等你绕回来,它已经不在缓存里——每个页都是在"下一次会用到它之前刚好被踢掉"。这就是教科书里的 **sequential flooding(顺序冲刷)**。

危害远不止"这次扫描慢"。真正可怕的是 **缓存污染(cache pollution)**:一次顺序大扫描(比如一个不带索引的 `SELECT COUNT(*)`、一次全表统计)会把整个缓冲池里辛苦攒了几个小时的热集**全部冲走**,换成一堆扫完就再也不碰的冷页。扫描结束后,你的 OLTP 热点查询全部 miss、集体打盘,系统延迟瞬间飙升——一个后台报表把前台业务拖垮了。**这就是为什么"纯 LRU 抗不住扫描"是个生产事故级别的问题,而不是学术细节。**

LRU 的根本缺陷在这里暴露:它的"最近用过"假设对扫描完全失效——扫描里每个页都"最近用过",但**没有一个会再用第二次**。LRU 无法区分"热"和"刚被扫过",于是被扫描牵着鼻子走。

## 五、替换策略二:CLOCK,用近似换抗扫描和低开销

CLOCK(时钟算法)是 LRU 的近似。它不维护精确的访问顺序(那个 O(n) 链表操作),只给每个帧一个 **ref bit(引用位,1 表示最近碰过)**。一只"指针"(clock hand,时钟指针)绕着帧环扫:碰到 ref=1 的,清成 0、给它一次"第二次机会"放过;碰到 ref=0 且没 pin 的,就是它了,踢掉。代码:

```ts
private chooseClockVictim(): number {
  const maxSteps = this.frameCount * 2;
  for (let step = 0; step < maxSteps; step++) {
    const slot = this.clockHand;
    this.clockHand = (this.clockHand + 1) % this.frameCount;
    const frame = this.frames[slot];
    if (frame.pinCount > 0) continue;      // pin 的永不踢
    if (frame.refBit) {
      frame.refBit = false;                // 第二次机会:这圈活,下圈还没碰就死
      continue;
    }
    return slot;
  }
  this.stats.evictFailures++;
  throw new BufferFull(this.frameCount);
}
```

而访问时的 `touch` 对 CLOCK 只是 `refBit = true` 一行——**比 LRU 的"摘下来重新挂到队尾"便宜得多**。这是 CLOCK 的第一个好处:近似 LRU 的命中率,但每次访问的记账成本(bookkeeping)低一个数量级。回头看 Zipf 实测,CLOCK 37.6% vs LRU 40.4%——命中率只差 2.8 个百分点,却省掉了维护精确顺序的全部开销。生产系统里这是非常划算的交易,所以 PostgreSQL、许多存储引擎用的都是 CLOCK 系(及其变体),而非纯 LRU。

**为什么 CLOCK 更抗扫描?** 关键在那个"第二次机会":一个真热点页会被反复访问、ref bit 一次次被重新置 1,指针扫过它时总看到 ref=1、总放它一马;而扫描带进来的冷页 ref bit 置 1 后再没人碰,指针下一圈扫到它就 ref=0、直接踢。于是**热页有韧性、冷页快速流出**,扫描不容易一次冲垮热集。这是近似换来的抗污染能力。

但要诚实:在我们这个 toy 全表扫描里,CLOCK 也是 0% 命中(8 帧装不下 64 页的工作集,第二次机会也救不了——绕回来时它早被踢了)。**当扫描的工作集本身就大于整个池,任何替换策略都无解**,这是物理限制不是策略缺陷。CLOCK 的优势体现在"扫描 + 热点混合"的真实负载里:它能在被扫描冲刷的同时,靠 ref bit 保住一部分真热点;纯 LRU 则会被冲得片甲不留。我们的 toy 负载是纯扫描,放大的是"工作集 > 池"这个共同下限,而 CLOCK 抗污染的增量得在混合负载下才看得出——这是这个 toy benchmark 的边界,得说清楚。

> ⚡ **前沿/开放问题:替换策略还没有"通用最优解"**。CLOCK 的进阶 **CLOCK-Pro / LIRS** 用"重用距离"区分"真热"和"刚扫过",显著改善抗扫描,但参数调优依负载而定。更激进的方向是 **LeanStore 的 pointer swizzling(指针原地变形)**:它干脆**不维护 page table 哈希表**——页在内存时,指向它的"指针"就是直接的内存地址(swizzled,已变形);页被换出磁盘时,指针原地改回页号(unswizzled)。访问时直接解引用指针,命中内存的路径**零哈希查找**,把第一节那个"每次访问过一遍哈希"的固定开销彻底绕掉。代价是替换策略和指针状态深度耦合,实现复杂度陡增。另一条线是 **机器学习驱动的替换**(用模型预测每个页的下次访问时间,趋近理论最优的 Belady 算法)——目前仍在研究,**没有一个能在所有负载上稳定打赢 CLOCK-Pro 的通用方案**,模型推理本身的开销和负载漂移下的失准是两个尚未解决的拦路虎。换句话说:"踢谁"这个问题,四十年了,**仍然没有标准答案**。

## 六、把这层验证锁死:disk reads 必须等于 misses

本书的纪律是"打印的每个数字都得是真的"。缓冲池最容易撒的谎是"我命中率很高",但偷偷在背后多发了几次 IO 没计入。代码用一条断言把这个谎堵死:

```ts
assertEq(diskReads, pool.misses, `disk reads must equal misses for ${workload}/${policy}`);
```

逻辑很硬:缓冲池能省下 IO 的唯一途径是命中,所以**磁盘读次数必须精确等于未命中次数**。如果这条断言哪天炸了,说明池在某处漏了一次没计数的读——那命中率就是假的,正是本书拒绝打印的东西。回看实测表里 `misses` 列和 `diskReads` 列:1192=1192、1247=1247、2000=2000,逐行相等。这不是巧合,是断言每次运行都在守着。

这也是把"缓冲池"和"随手写的 Map 缓存"区分开的最后一点:**它的有效性是被测量和断言守护的,不是被相信的**。

## 小结:这一层给了上层什么,欠了什么

建完这层,第 01 章那个"每次都打盘"的世界变了:

- **page table + 帧数组**把页号映射到有限内存,热点页命中后零 IO(Zipf 实测 13% 内存接住 40.4% 访问)。
- **pin/unpin** 保证正在被读的帧不被偷换,代价是调用方必须及时 unpin,否则 `BufferFull`(实测可触发、可恢复)。
- **脏页 + write-back** 让热点写从 N 次盘压到 1 次,代价是崩溃丢数据——这个洞留给第 05 章 WAL 补。
- **替换策略**决定踢谁:LRU 在热点下好但被扫描冲垮(实测 hit rate 0),CLOCK 用 ref bit 近似 LRU、记账更便宜、更抗污染。

往后看:第 03 章 B+树遍历会疯狂调用 `getPage`——一次查找走 root → internal → leaf 三四层,每层一个页,正是缓冲池命中率决定查询快慢的地方;内部节点(总在被访问)几乎永驻缓存,叶子页则随查询冷热进出,这正是替换策略发挥作用的舞台。第 08 章查询执行里,全表扫描算子会直接撞上本章第四节那个"sequential flooding"问题——到那时你就明白为什么真实数据库要给扫描单独开一条"绕过缓冲池 / 用小环形 buffer"的快速通道,不让它污染主缓存。
