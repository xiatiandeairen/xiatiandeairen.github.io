到这一章，奖励的来源变了。前四章里奖励是环境白送的——bandit 的真实期望、gridworld 的格子收益、PPO 优化的 `V*`，你总有一个可信的标量函数告诉你"这一步多好"。RLHF 没有这个东西。你手里只有一堆人类标注的成对比较：「回答 A 比回答 B 好」。没有分数，没有刻度，只有序。奖励模型（reward model，简称 RM，一个把回答打成标量分的小模型）的全部工作，就是从这种**只有相对序、没有绝对值**的标注里，反推出一个能给任意回答打分的函数，好让第 6 章的 PPO 把它当奖励去最大化。

这一步是整本书后半段的枢纽，也是后面所有病理（reward hacking、over-optimization）的源头。理由很直白：RM 是人类偏好的**代理（proxy）**，是从有限、有噪、有偏的标注里拟合出来的。它不可能等于真实偏好。它和真实偏好的那道缝，就是优化器将来要钻的洞。本章我们能把这道缝**量出来**——因为合成世界里 `trueRewardFn` 是可计算的——这是你在真实 RLHF 里永远没有的奢侈品。

## RM 看不见真奖励，只能从对子里猜

先看配套代码里 RM 到底"看见"什么。`examples/rl-posttrain-from-scratch/src/stage05-reward-model.ts` 的特征映射是这样的：

```ts
const GOLDEN = 12;
function feat(r: Response): number[] {
  const d = (r.length - GOLDEN) / 10; // signed, scaled length error
  return [
    -d * d, // captures the length PEAK (max near golden) — the load-bearing basis
    r.keywordHits / 5, // scaled keyword signal
    1, // bias
  ];
}
```

注意 RM 拿到的输入是 `Response` 的特征（长度、关键词命中数），**不是** `trueRewardFn`。真实奖励藏在 `examples/rl-posttrain-from-scratch/src/core/preference.ts` 里，RM 全程看不到：

```ts
const trueRewardFn = (prompt: Prompt, response: Response): number => {
  const lengthErr = response.length - goldenLength;
  const lengthTerm = -lambda * lengthErr * lengthErr; // 长度是个二次惩罚：有峰
  const usefulHits = Math.min(response.keywordHits, prompt.requiredKeywords);
  const keywordTerm = Math.sqrt(usefulHits); // 关键词是凹的：边际递减
  return keywordTerm + lengthTerm;
};
```

真实偏好里长度是个**峰**——太短和太长都被罚，最优在 golden（12）。这个形状是刻意设计的：它让"越长越好"这种最容易过拟合的单调规律在过峰之后**可证地错**。这正是 reward hacking 的钩子。

### 为什么特征要做缩放

`feat()` 里每个特征都除了一个典型尺度（length 除 10、keywordHits 除 5）。这不是装饰。未缩放时 `length`（约 2..40）的原始量级会压垮 `keywordHits`（约 0..5），梯度被 length 维度独占，RM 永远学不到关键词轴。这是真实 RM 训练里常见且**沉默**的失败——尺度不匹配让某条特征事实上死掉，loss 还在降，你却以为它学全了。代码里把它修掉并显式命名，就是不想让它沉默。

## Bradley–Terry：把"A 比 B 好"变成可优化的 loss

RM 的训练目标是 Bradley–Terry 模型（一种把成对比较转成概率的经典模型，1952 年提出）。它说：A 被偏好的概率是 `σ(s_A − s_B)`，其中 `s` 是 RM 打的分，`σ` 是 sigmoid。于是每个偏好对的损失就是 `−log σ(s_chosen − s_rejected)`——逼着 RM 把 chosen 的分顶上去、rejected 的分压下来。一个 epoch 的 SGD 长这样：

```ts
function trainEpoch(w: number[], pairs: PreferencePair[], lr: number, featFn: FeatFn): void {
  for (const { chosen, rejected } of pairs) {
    const fc = featFn(chosen);
    const fr = featFn(rejected);
    const margin = dot(w, fc) - dot(w, fr);
    const p = 1 / (1 + Math.exp(-margin)); // σ(margin) = P(模型同意这条标注)
    const g = 1 - p; // 梯度幅度：模型越自信地错，推得越狠
    for (let k = 0; k < w.length; k++) w[k] += lr * g * (fc[k] - fr[k]);
  }
}
```

这就是真实 RLHF 的 RM 目标函数，一字不差，唯一的"toy"是特征映射（真实里是 transformer 的最后一层 hidden）。梯度项 `(1 − p)` 是关键：模型对一个对子越自信地判错，`p` 越接近 0，`(1 − p)` 越接近 1，推得越用力。这也埋下了过拟合噪声的隐患——一条**被翻转的错标**会让模型"自信地错"，于是被狠狠拉向错误方向。

### ✦ RM 的输出尺度无意义，只有序有意义

这是本章最该带走的一条专家级判断。看 BT loss：它只依赖 `s_chosen − s_rejected` 这个**差**。把所有分数同时加一个常数、或同乘一个正数，loss 不变。换句话说 RM 学到的标量是**仿射不确定的**——绝对刻度、零点都没有意义，唯一有意义的是序（哪个比哪个高）和相对间距的符号。

这条性质不是哲学清谈，它直接决定了第 6 章 RLHF 里为什么必须做 reward 归一化 / whitening（把一批奖励减均值除标准差）。因果链是这样的：RM 的绝对值没意义 → 同一批 rollout 的 RM 原始分可能整体偏移到 +50 或 −30，纯粹取决于训练时的随机零点 → 第 4 章我们已经看到 PPO 的步长隐式被 `|advantage|` 缩放（见 stage04-ppo.ts 的注释：未归一化的 advantage 会把步长和奖励尺度耦合在一起）→ 若直接拿 RM 原始分当奖励，advantage 的尺度就跟着 RM 那个无意义的零点漂，KL 控制和学习率全失准。所以 RLHF 实现里对 RM 输出做 whitening 不是调参 trick，是在补偿"RM 尺度无意义"这条结构性事实。能把这条因果链一口气讲通，你对 RLHF 的理解就比"按公式 whitening"深一层。

代码里 `correlateWithTruth` 同时算 Pearson（看绝对刻度对不对）和 Spearman（看序对不对），就是为了把这件事量化：

```ts
function correlateWithTruth(...): { pearson: number; spearman: number } {
  const rmScores = responses.map((r) => dot(w, featFn(r)));
  const trueScores = responses.map((r) => world.trueRewardFn(prompt, r));
  return { pearson: pearson(rmScores, trueScores), spearman: spearman(rmScores, trueScores) };
}
```

## 怎么测 RM 好不好：留出 pairwise 准确率

RM 的标准评测是**留出 pairwise 准确率**：拿一批没训过的对子，看 RM 给 chosen 的分是否真的高于 rejected 的分。代码里有个关键的诚实设计——**留出测试集恒为干净标签**（用 `trueRewardFn` 评分，flipProb=0），即便训练集注入了翻转噪声：

```ts
const train = world.generatePairs(120, rng, flipProb);
const test = world.generatePairs(400, rng, 0); // clean held-out
```

为什么？如果测试集也带噪，"准确率"就变成在测量噪声本身，毫无意义。要测 RM 学到真偏好没有，标尺必须干净。

跑出来的训练曲线 + 噪声扫描（数字均来自 stage05 真实运行输出，sparkline 自归一化只看形状）：

| 翻转率 | 留出准确率曲线 (epoch 0..60) | 最佳→终值 | Spearman | Pearson |
|---|---|---|---|---|
| 0.00 | `▁▃▃██████████` | 0.777→0.777 | 0.933 | 0.971 |
| 0.10 | `█▂▁▁▂▁▁▁▁▁▁▁▁` | 0.775→0.760 | 0.972 | 0.984 |
| 0.30 | `▅█▇▁▁▁▁▁▁▁▁▁▁` | 0.775→0.755 | 0.952 | 0.968 |

四个读数，每个都对应一条真实工程教训：

**干净标签：单调爬升后走平**（0.777→0.777）。学到了，没崩。这是健康曲线该有的样子。

**翻转标签：早期见顶后回落**（最佳 0.775 > 终值 0.760 / 0.755）。这是过拟合噪声的指纹——多训反而把翻转的错标记住了。它直接给出了 early-stop 的**定量理由**：必须在干净验证集上盯着准确率，一旦回落就停，而不是训满 60 epoch。注意 early-stop 的标尺必须是干净集，拿带噪集 early-stop 等于让噪声决定何时停。

**翻转率↑ ⇒ 终值准确率↓**（0.78→0.76→0.75）。本合成集信号强，所以降幅温和，但方向是对的，而且这是天花板：RM 准确率被标注质量上界卡死，没有任何优化器能突破。降幅温和这点要诚实标注——真实数据上标注噪声咬得更狠，这里偏乐观。

**Spearman ≥ Pearson**（如 0.972 vs 0.984 这行 Pearson 略高是合成集信号强的特例，但整体趋势是 RM 序对、绝对刻度尤其 length 峰附近常不对）。这呼应了前面 ✦ 那条：序比刻度可靠。所以下游用 RM 时优先用排序信息（ranking / 相对比较），别太信它的绝对数值。

## 失败模式一：归纳偏置错了，分布外直接反号

现在到本章的核心警告。一个常见误解是"RM 在分布外（OOD，训练没见过的输入区域）总会崩"。**不对**。会不会崩，取决于 RM 的归纳偏置（inductive bias，模型对函数形状的先验假设）对不对。代码用两个 RM 对照演示：一个有平方基（能表达长度的峰），一个只有单调基（只能表达"越长越好/越差"）:

```ts
function featMonotone(r: Response): number[] {
  return [r.length / 10, r.keywordHits / 5, 1]; // 无平方项：结构上无法表达峰
}
```

OOD 探针把长度推到约 3.5 倍 golden（训练分布从没产生过这么长的回答，模拟一个学会"越长越好"的策略会漂到的区域）：

```ts
return { length: padded.length + 30, keywordHits: padded.keywordHits };
```

对照结果（stage05 真实输出）：

| RM | 分布内 Spearman | 分布外 Spearman (length≈3.5×golden) |
|---|---|---|
| 平方基 RM（对的偏置） | 0.998 | 1.000 |
| 单调基 RM（错的偏置） | 0.327 | **−0.660** |

读这张表要读出三层：

1. **对的偏置（平方基）分布内外都≈1**。函数形式对了，外推也对。OOD 不是诅咒。
2. **错的偏置（单调基）分布内就已经偏弱**（0.327）——但 0.327 是正的，光看分布内准确率你**分辨不出**它有结构性缺陷。这是最阴险的地方：坏 RM 在见过的数据上看起来还行。
3. **错的偏置分布外掉到 −0.660**，与真值**反相关**。这意味着：最大化这个 RM，就是在最小化真奖励。

第三条就是 reward hacking 的机理。优化器不傻，它会把策略一路推到 RM 给高分的地方——而对单调 RM 来说，"高分"在过峰之后恰好是真奖励的反号盲区。第 6 章会直接拿这个机制量出真奖励崩盘的那条曲线：proxy reward 一路爬，true reward 见顶后跳水。stage06 的开头注释把这件事说得很清楚——「the optimum of the proxy is not the optimum of the truth」，KL 缰绳只限制策略追得多远，不修 RM 本身的盲区。

### ⚡ 前沿：reward 的校准与过度优化，目前无通用解

把上面这件事放大到真实尺度，就是 2023–2025 对齐研究最热的开放问题之一：**reward model over-optimization（奖励过度优化）**。Gao et al.（2023, "Scaling Laws for Reward Model Overoptimization"）给出了一条经验缩放律——随着你对 RM 优化得越狠（用 KL 距离度量），按**真**奖励衡量的策略质量先升后降，且这条 proxy-vs-gold 的背离曲线随 RM 规模、数据量呈可预测的函数形式。本章表格里"翻转率↑→终值准确率↓"和"错偏置在 OOD 反号"，就是这条缩放律的可跑缩影：RM 越不准、越早达到那个 true-reward 见顶的拐点。

为什么说**无通用解**：你没法事先知道 RM 在策略将要漂进的那片 OOD 区域里是不是反号的——因为按定义那片区域没有标注数据，你测不了它的准确率（回看上面第 2 条，分布内准确率根本看不出问题）。当前的缓解手段都是**部分**的：KL 惩罚（限制漂移距离，第 6 章）、RM ensemble 取保守估计、训练中途用新策略的 rollout 重新标注刷新 RM（迭代式 RLHF）、用更校准的不确定性估计。这些都在压低过度优化的代价，但没有一个能在不知道真奖励的前提下**保证**找到真最优。这是仍在研究、没有通用解的方向。

## 失败模式二：偏好对全部同向 = 没有对比信号

最后一个失败模式更上游，也更绝望——问题不在优化器、不在归纳偏置，在**数据本身**。代码构造一个极端集：每个对子都标"更长的赢"，不管真奖励：

```ts
function generateCollapsedPairs(world, n, rng): PreferencePair[] {
  // ...
  const aLonger = a.length > b.length;
  pairs.push({ prompt, chosen: aLonger ? a : b, rejected: aLonger ? b : a, flipped: false });
}
```

这种数据沿任何真实偏好轴都**零信息**，只有一条虚假的 length 信号。结果（stage05 真实输出）：

- 注入的「越长越好」标签与真值一致的比例 = **0.417**（小于 0.5，意味着"越长越好"其实多数时候**与真值相反**）
- 全同向数据训出的 RM 干净留出准确率 = **0.280**（对照：正常多样标签 = 0.777）

0.280 这个数字要盯死：它**跌破了随机线 0.5**。RM 没有学成"我不知道"（那会停在 0.5 附近），而是**自信地学反了**——它把虚假的 length 相关当成真理，于是在干净测试集上系统性判错。garbage-in / garbage-out 在这里是字面意义的：标注的对比结构就是天花板，没有任何优化器能从全同向的标签里救出真实偏好。

这条对实践的告诫很硬：RLHF 翻车时，先别急着调 RM 架构或 PPO 超参，先去看**偏好数据的对比结构**。如果你的标注里 chosen 总是更长 / 总是来自某个模型 / 总是排在前面，你的 RM 学的就是这个假相关，而不是质量。length bias 是最常见的一种，恰好就是第 6 章 reward hacking 演示里那个被指定的 length-biased proxy 的现实来源。

## 小结与诚实声明

把这一章钉死成几条可迁移的判断：

- RM 用 Bradley–Terry loss 从成对偏好学一个标量，**输出尺度无意义、只有序有意义**——这条直接决定了 RLHF 要做 reward whitening。
- 评 RM 看**干净留出 pairwise 准确率**，噪声会让曲线早期见顶后回落，所以要在干净集上 early-stop。
- RM 在分布内看着好不代表分布外好；**归纳偏置错的 RM 会在 OOD 反号**，这就是 reward hacking 的入口，第 6 章直接量。
- 数据没有对比结构（全同向标注）时，RM 会自信地学反，**准确率跌破随机线**——这是优化器救不回来的上游缺陷。

诚实声明（与代码里 `[诚实声明]` 一致）：这是合成偏好世界，`trueRewardFn` 可计算，绝对数值偏乐观。可迁移的是**相对趋势**：噪声↑→终值准确率↓且需 early-stop、Spearman≥Pearson、错偏置在 OOD 反号、无对比信号→准确率跌破随机。真实 RLHF 里你拿不到 `trueRewardFn`，所以这些故障你只能**间接**察觉——而这正是它们危险的原因。下一章我们就拿这个 RM 当奖励喂给 PPO，直接看着真奖励在 proxy 还在涨的时候崩给你看。
