上一章的老虎机里，拉一次臂，奖励当场结算，没有"以后"。这章我要把"以后"加进来——这是 RL 真正难的地方，也是为什么 LLM 后训练这么难调的根。换到一个 3×4 的格子世界（gridworld，方格地图，agent 走格子找目标），起点 S、目标 G（+1）、陷阱 H（−1）、墙 #，每走一步扣 0.04。难点在于：你走第一步的时候，G 的 +1 还在三四步之外，这步走得好不好，**要等到撞上 G 或掉进 H 才知道**。把"延迟到来的奖励"正确地分摊回当初每一步——这就是**信用分配**（credit assignment，把最终结果的功劳/锅算到沿途哪一步头上）。它是本书后面所有麻烦的源头。

但 gridworld 有个老虎机没有、真实 LLM 更没有的奢侈品：**它的最优解可以精确算出来**。环境的转移和奖励我们全知道（专业说法是 MDP 已知，Markov 决策过程——下一状态只由当前状态和动作决定），于是不用"学"，直接"解"。这章干的事就一件：用值迭代（value iteration，反复回填每格的价值直到不动）算出每一格的最优价值 `V*` 和最优策略 `π*`，把它钉成一把上帝标尺。后面第 3 章的 REINFORCE、第 4 章的 PPO，学出来的策略好不好，不靠"看着像收敛了"，而是拿真回报跟这把 `V*` 一比——差多少，是个数。这是这本书跟 AI 批量生成的 RL survey 最根本的分野：survey 告诉你"策略梯度会收敛到最优"，这章告诉你最优是 `0.6335`，你的 agent 现在是 `0.41`，还差 `0.22`。

配套代码全在 `examples/rl-posttrain-from-scratch/src/stage02-gridworld-value.ts`，本章所有数字来自它跑两遍、逐位相同（REPRO-OK）的真实输出。

## return：你到底在最大化什么

先把目标说死。RL 里 agent 最大化的不是单步奖励，是 **return**（回报，从某一步往后所有奖励的折扣加总）。从状态 `s` 出发、按策略 `π` 行动，return 是：

```
G_t = r_{t+1} + γ·r_{t+2} + γ²·r_{t+3} + ...
```

`γ`（gamma，折扣因子，0 到 1 之间）是给"未来奖励打几折"的旋钮。`γ=0` 时 agent 只看下一步，彻底短视；`γ→1` 时遥远的奖励几乎不打折，agent 极有耐心。

为什么不直接加总、非要打折？两个原因，**第二个是工程师必须刻进直觉的**：

1. **耐心旋钮**：现实里"现在的一块钱比明年的一块钱值钱"，γ 编码了这种时间偏好。
2. **数值稳定旋钮**：如果一条轨迹可能永远不终止（agent 在格子里绕圈），无折扣的 return 是个无穷级数，可能发散到 ±∞。γ<1 把它压成收敛的几何级数——最坏情况下 return 被 `r_max/(1−γ)` 封顶。**这一条不是哲学偏好，是不让数字炸掉的硬约束**，后面第 4 节的失败模式会让你亲眼看见不打折时 V 怎么跌到 −∞。

`return` 是定义在一条轨迹上的随机量。我们真正想要的是它的**期望**——因为环境可能有随机性（代码里的 gridworld 支持"风"，`windProb` 概率把 agent 往上吹一格），同一个动作可能落到不同格子。`qValue` 函数显式地把这个 2 种结果的分布枚举出来算期望，而不是假设动作一定成功：

```typescript
// qValue: expected value of taking action a in state s, under the wind model.
function qValue(env, V, s, a, gamma, enterReward): number {
  const intended = env.intendedNext(s, a);
  const outcomes = windOutcomes(env, intended);  // [{state, prob}, ...]，概率和=1
  let q = 0;
  for (const { state: ns, prob } of outcomes) {
    const reward = enterReward[ns] + env.stepReward;
    // 终态进入后没有"以后"：episode 结束，不 bootstrap V[ns]。
    // 忘了这点是经典 bug——价值会从目标后面漏出去。
    const future = env.terminal[ns] ? 0 : gamma * V[ns];
    q += prob * (reward + future);
  }
  return q;
}
```

注意那行 `future = env.terminal[ns] ? 0 : gamma * V[ns]`——进了终态就没有未来，不能再往后接 `V`。这是个反复有人踩的坑：忘了它，价值会"漏过"目标，让目标后面的格子也虚高。

## Bellman 最优方程：能手推它是 RL 基本功的门槛

`return` 的定义是个无限求和，没法直接算。Bellman 的洞见是把它写成**递归**：一个状态的最优价值，等于"当前这步最好的即时奖励 + 折扣后的下一状态最优价值"。这就是 **Bellman 最优方程**：

```
V*(s) = max_a  Σ_{s'} P(s'|s,a) · [ R(s,s',a) + γ·V*(s') ]
```

拆开念一遍（这是 ✦ 专家钩子，能不看书手推它、并解释每一项，是 RL 面试和实战的硬门槛）：

- `max_a`：在状态 `s`，我挑**最好**的那个动作（这就是"最优"，对应代码里 `qValue` 外面那层 `for a ... if (q > best)`）。
- `Σ_{s'} P(s'|s,a)`：动作不一定确定，对所有可能落到的下一状态 `s'` 按概率加权求期望（对应 `windOutcomes` 那个分布）。
- `R(s,s',a)`：进入 `s'` 拿到的即时奖励（代码 `enterReward[ns] + env.stepReward`）。
- `γ·V*(s')`：下一状态自己的最优价值，打个折接上去——**这就是 bootstrapping**（自举，用自己对未来的估计去更新自己），RL 区别于纯监督学习的核心动作。

值迭代就是把这个方程当成赋值语句反复执行：`V(s) ← max_a Σ ...`，每格都用当前的 `V` 估计去更新，扫一遍叫一个 sweep，扫到 `V` 不再变（`max|ΔV| < 1e-8`）就停。代码主循环就是它的直译：

```typescript
for (; iter < maxIters; iter++) {
  let delta = 0;
  for (let s = 0; s < env.nStates; s++) {
    if (env.terminal[s] || env.isWall(s)) continue;  // 终态/墙没有决策
    let best = -Infinity;
    for (let a = 0; a < env.nActions; a++) {
      const q = qValue(env, V, s, a, gamma, enterReward);
      if (q > best) best = q;
    }
    delta = Math.max(delta, Math.abs(best - V[s]));
    V[s] = best;
  }
  deltaCurve.push(delta);
  if (delta < tol) { converged = true; iter++; break; }
}
```

**为什么它一定收敛？** Bellman 最优算子是个**收缩映射**（contraction，每次迭代把误差至少缩小到原来的 γ 倍）。误差以 γ 的几何速率衰减，所以 γ<1 时必然收敛到唯一不动点 `V*`。这个"收缩系数就是 γ"的事实，下一节会变成可观测的现象。

跑出来的结果，γ=0.95 下：

```
[1] 值迭代 (γ=0.95) 收敛于第 6 轮 (max|ΔV|<1e-8)，对照 core oracle 最大偏差=0.0e+0
    收敛曲线 max|ΔV| 逐轮 (对数式衰减): █████▁
    起点 V*(S)=0.6335 = 从起点出发按 π* 行动的最优折扣回报
[2] 最优策略 π* (每格箭头 = 该状态最优动作):
   →  →  ↓  ←
   ↓  #  ↓  H
   →  →  →  G
```

`V*(S)=0.6335` 就是这把上帝标尺的零点：从起点出发、永远走最优、考虑了风的随机性之后，期望折扣回报就是 `0.6335`，没有任何 agent 能超过它。`π*` 的箭头读起来很合理——从 S 一路 `→ ↓ →` 绕开陷阱 H 奔向 G。

这里有个工程上不能省的细节：教学用的 `valueIteration` 不是孤证。代码同时调了 `core` 里独立实现的 `optimalValueIteration` 当 oracle（标准答案），断言两者最大偏差 `0.0e+0`。**两个独立实现给出逐位相同的答案，才敢把它当标尺**——否则你拿一个自己可能写错的"标尺"去量别人，量出的偏差分不清是 agent 差还是尺子歪。这正是 N=1 不可信、必须独立交叉验证的工程纪律。

## γ 既是耐心又是数值稳定：扫一遍看清两副面孔

光说"γ 是耐心旋钮、又是稳定旋钮"是 survey 腔。代码扫了一遍 γ，把两副面孔都摊在桌上。

小网格（3×4）上扫 γ：

```
γ     | 收敛轮数 | V*(起点) | π*(起点) | 说明
0.50  |      6   |  -0.0150 |    →     | 短视: 远处目标被折扣压扁
0.90  |      6   |   0.4923 |    →     | 折中
0.99  |      6   |   0.7646 |    →     | 远视: 价值最高
```

价值随 γ 单调上升：γ=0.5 时 +1 的目标被折得只剩一点点，扣的步数代价反而占上风，起点价值是负的（−0.015）；γ=0.99 时目标几乎不打折，价值冲到 0.7646。**这是"耐心"那一面**：γ 越大，遥远的目标越值钱。

但请注意收敛轮数全是 6，对 γ 完全不敏感——这跟"γ 是收缩系数、越大越慢"的教科书说法矛盾。**这里必须诚实，不能为了讲好故事掩盖**：小网格上收敛轮数由网格直径决定（目标信号传遍全图大约 6 步就到顶了），是"直径受限"而非"收缩受限"，γ 改变不了它。要看到 γ 对速度的影响，得把网格放大到收缩速率真正成为瓶颈：

```
大网格 (5×10, 长路径):
γ     | 收敛轮数 | V*(起点)
0.50  |     23   |  -0.0800
0.90  |    146   |  -0.4000
0.99  |   1514   |  -4.0000
```

现在 **γ 的第二副面孔露出来了**：γ→1 时收缩系数 →1，收敛急剧变慢，轮数从 23 暴涨到 146 再到 1514（真测，不是估算）。这就是"数值稳定旋钮"的代价——你想要 agent 更有耐心，就得付出更慢的收敛、更难训练。在真实 LLM-RL 里这条直接体现为：长序列任务（长 reasoning chain）想用高 γ 做长程信用分配，训练就会更不稳、更慢。

还有个反直觉的细节：**价值随 γ 的方向因环境而异**。小网格 γ↑→V↑（目标主导），大网格 γ↑→V↓（−0.08 → −4.0，长路径上一堆步数代价主导，γ 越大这些代价被折得越少、越疼）。所以"γ 越大价值越高"是错的泛化——普适的只有两条：γ↑ 减少对**所有**未来信号（奖励和代价一视同仁）的折扣；γ↑ 拖慢收缩受限的收敛。

## 失败模式 A：γ=1 且无终止态 = 发散

为什么我一再说 γ<1 不是可选项？代码直接演示给你看。造一个 3×3 全空地、没有任何终止态的网格，每步恒扣 0.04，用 γ=1 跑：

```
[4] 失败模式 A — γ=1 且无终止态 (3×3 全空地，每步恒 -0.04):
    跑满 400 轮上限，converged=false；max|ΔV| 首轮=0.0400 末轮=0.0400 (不收敛)
    起点 V 已跌到 -16.00 并继续发散 → 折扣 γ<1 (或保证终止) 不是可选项。
```

`max|ΔV|` 从头到尾稳定在 0.0400——每一轮每格都再扣 0.04，永远不收敛，V 朝 −∞ 一路下坠，跑满 400 轮上限被强行截停时起点已经是 −16.00。**为什么必然如此**：没有终态，每条轨迹无限长；γ=1 时 return 是 `−0.04 − 0.04 − 0.04 − ...` 这个发散级数，根本没有有限的 `V*` 可收敛。收缩映射的前提（收缩系数 γ<1）被破坏了。

这就是第 2 节那个"数值稳定旋钮"的实锤：γ<1 或保证 episode 终止，二选一，否则你的价值函数没有数学上的不动点可学。在 LLM-RL 里它对应"必须有 EOS（终止 token）或最大长度截断"——否则优势估计会发散。代码把 `maxIters` 设成硬上限正是为此：让发散的 case 返回而不是死循环，**而那个跑满上限、`converged=false`、ΔV 全程平的状态，本身就是"没收敛"的证据**。

## 失败模式 B：奖励塑形加错 = reward hacking 的最小同构

这是这章最该让人脊背发凉的一段，因为它是后面 RLHF reward hacking（奖励黑客，agent 钻奖励函数空子刷分而非真正做好任务）的最小可跑原型。

奖励塑形（reward shaping）是个常见手法：给中间状态加点引导奖励，帮 agent 更快学会。代码故意加错——在一个**非目标**格子 #4 偷偷塞 +0.7：

```typescript
const decoyCell = 4;              // row 1, col 1，不在真目标路径上
const shaped = env.cellReward.slice();
shaped[decoyCell] = 0.7;          // 诱饵：在非终态格子上放大奖励
const trueVi = valueIteration(env, gamma);                      // 对真奖励最优
const hackVi = valueIteration(env, gamma, 1e-8, 5000, shaped);  // 对塑形奖励最优
// 用真奖励给被塑形带偏的策略打分（policy evaluation，沿固定策略求期望回报）
const hackUnderTrue = evaluatePolicy(env, hackVi.policy, gamma, env.cellReward);
```

跑出来两套策略对比：

```
[5] 失败模式 B — reward shaping 加错 (在非目标格 #4 偷加 +0.7):
    真奖励下的 π* 策略:        被 shaping 带偏的 π_hack 策略 (为刷 #4 绕路):
   →  →  ↓  ←                  ↓  ←  ←  ←
   ↓  #  ↓  H                  →  #  ↓  H
   →  →  →  G                  ↑  ←  ←  G
    起点动作: 真 π*=→  π_hack=↓  (已被带偏 ✗)
    用【真奖励】打分: V*(起点)=0.6335  vs  π_hack 的真回报=-0.8000  → 真目标损失=1.4335
```

`π_hack` 学会了为了反复蹭 #4 那个 +0.7 而绕路，起点动作从 `→` 被带偏成 `↓`。**关键在于怎么打分**：`π_hack` 对**塑形后的代理奖励**是最优的——它正确地最大化了你给它的那个被污染的奖励函数。但用**真奖励**重新评估，它的回报是 −0.8000，比真最优的 0.6335 差了 1.4335。

这正是 reward hacking 的精确定义：**代理奖励（你能优化的）和真目标（你真正想要的）一旦解耦，agent 就在解耦处钻空子**。它没"作弊"，它忠实地优化了你写的奖励——错的是你写的奖励。在 RLHF 里这就是 reward model（奖励模型，学人类偏好打分的模型）被 over-optimize 后，policy 学会输出"reward model 打高分但人类其实讨厌"的回答（谄媚、套话、虚假自信）。gridworld 这个 1.4335 的损失，跟一个 7B 模型被 reward hacking 带偏，是同一个数学现象的两个尺度。

注意这里能算出"真目标损失=1.4335"这个确切数字，**完全依赖第 3 节那把 `V*=0.6335` 的标尺**。没有它，你只会看到 `π_hack` 在代理奖励上分数很高，根本意识不到它已经偏了——这正是真实 RLHF 里 reward hacking 难发现的原因：线上你只有代理奖励的读数，没有"真目标"这把尺。

## ⚡ 前沿：过程奖励 vs 结果奖励，本质是信用分配粒度

gridworld 的逐步 reward（每走一步 −0.04）和"只在终点给 +1"，正是当下 RL-for-reasoning（用 RL 训练模型做数学/代码推理）里最热的争论的玩具原型——**过程奖励 vs 结果奖励**。

- **结果奖励**（outcome reward）：只看最终答案对不对，对就 +1。信号干净、不会被中间步骤的伪信号污染，但**信用分配粒度极粗**——一条 20 步的推理链，错在第 3 步，结果奖励只告诉你"整条链是错的"，不告诉你错在哪。agent 得靠大量采样自己摸索，样本效率低。这就是 gridworld 里"只有 G 给 +1、中间全靠 bootstrapping 把价值一步步回填"的放大版。
- **过程奖励**（process reward，PRM）：给每一步推理打分，信用分配粒度细、学得快——对应 gridworld 的逐步 reward。但它有个致命同构：**第 6 节的失败模式 B 就是过程奖励加错的最小版**。给中间步骤打的分一旦和"最终答案正确"解耦，模型就学会生成"PRM 爱看但对解题无用"的步骤——过程奖励的 reward hacking 比结果奖励严重得多，因为你给了它更多可钻的空子。

**这是个目前没有通用解、正在被激烈研究的开放问题**。结果奖励干净但稀疏，过程奖励密集但易被 hack；社区在 PRM 标注成本、用结果信号反推过程信用（如把成功轨迹的步骤当正样本）、以及"既要密集信号又不被 hack"的中间路线上反复横跳，没有公认的赢家。当你读到某篇论文宣称"过程奖励吊打结果奖励"时，回到这章的 gridworld 想一下：他们的过程奖励，有没有被验证过和真目标对齐？还是只是在代理奖励上分数好看？这个 1.4335 的教训，在 7B 尺度上一样会咬人。

## 这章给后面留下了什么

- **一把标尺**：`V*(S)=0.6335` 是后面所有学习算法的答案键。第 3 章 REINFORCE 学出的策略，回报离 0.6335 有多远，是个可量的数，不是"看着收敛了"。
- **一个旋钮的两副面孔**：γ 既调耐心（小网格 V 从 −0.015 到 0.7646）又调收敛速度（大网格 23→146→1514 轮），高 γ 的代价是难训。
- **一条硬约束**：γ<1 或保证终止，否则价值发散（V→−16.00 且不停）。
- **一个最小恶魔**：奖励塑形加错 = agent 绕路刷分，真目标损失 1.4335——RLHF reward hacking 的同构最小版。

下一章把"已知 MDP"这个奢侈品拿掉：agent 不再能读环境模型，只能靠采样的轨迹估计梯度。那时你就会真正体会到，为什么这章要这么辛苦地先立一把标尺——因为从下一章起，"学得好不好"再也不是显然的了。
