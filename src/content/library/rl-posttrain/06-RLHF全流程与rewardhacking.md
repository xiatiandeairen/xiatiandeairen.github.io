先把这章最反直觉的一句话钉死：**RLHF 训练时你监控的那条 reward 曲线一路上涨，不代表模型在变好——它很可能正在变坏。** 你优化的是 reward model（RM，奖励模型，一个学出来的打分器）给的分，而 RM 是真实人类偏好的一个有缺陷的代理（proxy）。策略足够强时，它会找到「RM 给高分但人类其实不喜欢」的回答，把代理分刷爆，真实质量反而掉下去。这就是 reward hacking（奖励钻空子），是当前对齐安全最硬的未解难题之一。本章不讲故事，直接在一个能算出真值的确定性环境里把它复现出来、量出拐点、再拆三种缓解手段各自的代价。

前两步已经备好：第 4 章手写了 PPO（一个带裁剪和 KL 控制的策略梯度优化器），第 5 章训出了 RM 并量过它的「length 偏置」(length bias，越长打分越高的伪相关)。RLHF 就是 **PPO + RM + 一个新配料：把策略拴在训练前的参考策略上的 KL 惩罚**。配料就这一样,但它是整套机制能不能用的开关。

## RLHF 的目标函数：奖励减去一根缰绳

标准 RLHF 优化的不是「最大化 RM 分」，而是：

$$\max_\pi \; \mathbb{E}_{\pi}[\text{rm}] - \beta \cdot \mathrm{KL}(\pi \,\|\, \pi_\text{ref})$$

第一项是 RM 给的代理奖励（你优化的目标）；第二项是当前策略 π 相对参考策略 π_ref 的 KL 散度（KL divergence，两个分布差多远），β 是它的系数。`π_ref` 就是 RLHF 开训前那个 SFT 模型（监督微调出来的初始策略）——你从它出发，KL 项惩罚你离它太远。

为什么需要这根缰绳？因为 RM 只在「参考策略附近的回答」上被训练过。一旦策略漂到远处——比如生成 SFT 阶段从没出现过的超长回答——RM 就在外推（extrapolate），那里它的打分完全不可信。KL 项的物理意义就是：**你越走越远，惩罚越大，逼你待在 RM 还算靠谱的那片区域里。** 它不修 RM 的缺陷，只限制你被缺陷坑的幅度。这个区分后面会反复出现。

配套代码把这个目标函数压到了最干净的形态。我们这章对应的 stage 是 [`examples/rl-posttrain-from-scratch/src/stage06-rlhf.ts`](examples/rl-posttrain-from-scratch/src/stage06-rlhf.ts)，它把「一个回答」坍缩成它的长度——因为 length 就是那条可被钻的轴，一维动作空间能画出干净的曲线。策略是长度桶上的一个 categorical 分布（多项分布），参考策略是一个偏好 len≈8 的欠训练 SFT：

```typescript
// 动作空间：候选回答长度的菜单。策略 = 这些长度桶上的 categorical 分布。
const LENS = [6, 8, 10, 12, 14, 16, 18, 20, 24, 28, 32, 40];
const GOLDEN = 12; // 真奖励在这里见顶（对齐 core/preference 的 goldenLength）

// 参考策略 = 欠训练 SFT，中心偏短（约 len 8）。
// 故意选欠训练：留出 headroom，让「RLHF 有帮助」是测出来的事实而非假设。
const REF_LOGITS = LENS.map((len) => -0.05 * (len - 8) * (len - 8));
```

参考策略选「欠训练」是个有意的诚实设计。它偏好 len≈8，离真值峰值 len=12 还有距离，所以 RLHF 确实有活可干——「RLHF 帮了忙」是后面量出来的数字，不是预设的结论。

## 三个奖励，必须分清

环境里有三个不同的奖励函数，混淆它们是理解 reward hacking 的最大障碍。代码里它们是三个独立函数：

```typescript
// 隐藏的真值：在 GOLDEN 见顶，太短或太长都被二次惩罚。策略永远看不到它。
function trueReward(len: number): number {
  return 1.5 - LAMBDA * (len - GOLDEN) * (len - GOLDEN);
}

// RM：一个带 length 偏置的代理。形状对，但多了个「越长越好」的伪相关项，
// 所以它的 argmax 落在 golden 之后。优化它会把策略推过真峰值 = reward hacking。
const VERBOSITY_BONUS = 0.45; // 调到让 RM 的 argmax ≈ len 20，远超 golden 12
function rmScore(len: number): number {
  return trueReward(len) + VERBOSITY_BONUS * (len - GOLDEN);
}
```

- **true reward**（真奖励）：你真正想要的，在 len=12 见顶（=1.50）。策略永远看不到它。现实里它就是「真人怎么评」，你拿不到。
- **proxy reward**（代理奖励 = RM 分）：你实际优化的。因为带 length 偏置，它的峰值被推到 len=20——而真值在 len=20 处只有 -0.42。
- 第三个是 **KL**，不是奖励，是「漂移量」——你离参考多远。

这里有个诚实声明值得专门点出：这个 RM 的 length 偏置是**直接写进函数的**，不是从第 5 章那种 pair 数据训出来的。stage06 的代码注释把原因讲透了——一个在代表性数据上拟合良好的线性 RM 其实能恢复真实的 length 峰值（构建这章时验证过），偏置只在分布外或标注被污染时才冒出来。这里把偏置 bake 进去，是为了让 hacking **可见、可复现**；第 5 章展示这种偏置从哪来。可迁移的教训——proxy 和 true 在优化下必然解耦——不依赖 RM 是怎么获得偏置的。这点很重要：**合成环境的绝对数值是乐观的，能迁移的是趋势和拐点的结构,不是具体的 1.469。**

> ⚡ **前沿/开放问题**：reward hacking 目前**没有通用解**，Anthropic 和 OpenAI 都把它列为高优先级对齐风险。难点的根：只要 RM 是个学出来的有限模型，它的最优点就几乎不可能等于真值的最优点（Goodhart 定律——指标一旦成为目标就不再是好指标）。我们这里能「测出」hacking 拐点，唯一原因是 toy 环境暴露了 `trueReward`；**真实系统里没有这个 oracle**，你看不见 true 曲线，只能靠代理信号去猜何时该停。后面会看到这正是 KL 缰绳的全部用处。这条目前是研究前沿，没有 published 的「检测 hacking」通用方法。

## 把 reward hacking 画出来：β=0 的三曲线剪刀差

先关掉缰绳（β=0），看纯粹优化 RM 会发生什么。代码做的是确定性梯度上升——固定离散动作菜单上没有任何采样，同输入同曲线，每次都一样，这是最强的可复现保证。核心一步是这个梯度，注意 `klPenalty` 那项在 β=0 时整个消失：

```typescript
// E_π[rm - β·logπ + β·logref] 对 logits 的梯度。
// 每个动作策略追的「奖励」= rm(a) 减去 β 缩放的 log-ratio（KL 惩罚的逐动作贡献）。
const klPenalty = Math.log(probs[a] + 1e-12) - Math.log(REF_PROBS[a] + 1e-12);
const reward = RM_MENU[a] - beta * klPenalty;
```

跑 `npx tsx src/stage06-rlhf.ts`，β=0 这段输出（48 个采样点的 sparkline）是这样的：

```
【β=0，无 KL 缰绳：训练步数 →（48 采样点）】
  proxy(RM奖励，优化目标)  ▁▃▅▆▆▆▆▆▆▇▇█████████████████████████████████████
  KL(漂移，离参考多远)     ▁▁▃▄▄▄▅▅▄▄▅▆▇▇██████████████████████████████████
  true(真奖励，真正想要)   ▁▅▇████████▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇▇
  hacking 拐点 ↓                 ^
```

这就是那张白板图。✦ 如果你有 RLHF 实战经验,你能在白板上凭手画出这三条线的关系：**proxy 单调爬到顶、KL 一路涨到天、而 true 先升后降**。三条线的剪刀差点（true 见顶而 proxy 还在涨）就是 hacking 拐点。具体数字：

- 真奖励在 **step 540 见顶（true=1.469）**，之后掉到 **1.380**。
- 同期 proxy 从 -0.323 一路涨到 **2.274**，KL 涨到 **2.95**。

读这组数字：前 540 步，优化 RM 也帮了真值（proxy 和 true 同向上涨，策略从 len≈8 往 golden=12 挪）；过了 540 步，策略已经越过真峰值、被 RM 的 length 偏置继续往 len=20 拽，**proxy 还在涨，true 却开始掉**。proxy↑ 而 true↓——这就是 reward hacking 的定义性现象：策略在啃 RM 的对抗样本（这里是「最长回答」），真实质量反降。

实战里最经典的误判，就是看到 proxy 那条单调上升的线就以为模型在持续变好。它不是。**proxy 单调上升和模型变好是两回事**,在拐点之后它们直接背道而驰。

## 早停的价值：你赚的那 0.090 是真金白银

既然 true 在 step 540 见顶、训到底反而掉，那就该在 540 停。代码直接量了这笔账：

```
【按 true 早停 vs 训到底（同一条 β=0 曲线）】
  早停于 true 峰值(step 540)：true=1.469  |  训到底(step 4000)：true=1.380  |  早停多赚 0.090 真奖励。
```

早停比训到收敛**多赚 0.090 真奖励**。在这个 toy 里 0.090 看着小，但它的方向是确定的：多训 = 倒亏。问题在于——**现实里你看不到 true 曲线**。没有 oracle 告诉你 step 540 是峰值。你手里只有 proxy（一直在涨,骗你)和 KL（一直在涨，是唯一诚实的信号）。

所以「何时早停」在真实 RLHF 里是靠代理信号判断的：盯 KL 涨到某个阈值就停、留出一小撮真值标注（held-out 人评）周期性测、或者直接拿人来评 win-rate。**这就是 KL 缰绳存在的另一半理由**——它不光约束优化，它涨的速度还是你判断「漂得够远该警惕了」的仪表盘。

## β 扫描：太松 hack，太紧学不动的 U 形权衡

现在把缰绳装上,从松到紧扫一遍 β。代码对每个 β 跑完整 4000 步,取最终状态：

```
【β(KL系数) 扫描：太松 hack / 太紧学不动 的 U 形权衡】
  β     | 最终true | 最终proxy |  KL   | 说明
   0.0  |   1.380  |   2.274   | 2.949 | 无缰绳：proxy 最高、KL 最大、被 length 偏置带偏
   0.1  |   1.380  |   2.268   | 2.908 |
   0.2  |   1.382  |   2.228   | 2.670 | ← 最佳 true：缰绳松紧刚好，吃到改进又没被 hack
   0.5  |   1.293  |   2.119   | 1.773 |
   1.0  |   1.329  |   1.418   | 0.772 |
   3.0  |   1.196  |   0.377   | 0.116 |
  10.0  |   1.093  |  -0.110   | 0.011 | 拴太死：几乎没离开参考，proxy 都没涨（学不动）
```

参考基线 true=**1.042**。把这张表读透，三件事：

1. **所有 β 的 true 都 ≥ 基线 1.042**——RLHF 确实帮了忙，无论缰绳松紧，最终都比啥也不做的 SFT 强。这验证了前面「欠训练参考留出 headroom」的设计。

2. **最佳在中间，β=0.2（true=1.382）**,不是两端。这是个 U 形（准确说是倒 U）权衡：
   - β 太小（0.0~0.1）：缰绳松，proxy 飙到 2.27、KL 冲到 2.9，策略被 length 偏置带偏,true 被 hack 拉低到 1.380。
   - β 太大（3.0~10.0）：缰绳勒死，KL≈0.01，策略几乎没离开参考,**proxy 都没涨**（β=10 时 proxy 是 -0.11，比起点还低）——白训了，true 只有 1.093，刚比基线高一点。
   - 中间 β=0.2：KL=2.67，松紧刚好，既吃到了往 golden 的改进，又没被拽过头。

3. 注意 **β=0 和 β=0.1 的最终 true 都是 1.380**,但 β=0.2 反超到 1.382。差距看着小,但它说明一件深的事：**一点点 KL 惩罚就能把你从「被 hack 的终点」往回拉一丢丢**。真实系统里 RM 偏置更狠、动作空间更大,这个差距会被放大。

这张表是你调 β 的全部直觉来源。它也解释了为什么 RLHF 里 β（或等价的 KL target）是最关键、最难调的超参之一——**它是「改进 vs 漂移」那个唯一的旋钮**,拧错任一边都白费算力。

## 三种缓解手段,各有代价

把失败模式和处方摆在一起。stage06 的总结直接给了三条,我逐条拆它的代价:

**1. KL 约束（本章主角）。** 代价：它**不能修好 RM,只能限制损失上限**。RM 越偏,任何 β 能拿到的 true 上限就越低——你看表里最佳也才 1.382,离真值峰 1.50 还差一截,这个差是 RM 偏置本身造成的,KL 拧不回来。而且 β 本身要调,调错任一边(太松 hack / 太紧学不动)都付出代价。

**2. Reward 归一化 / early stop。** 早停的代价上面量过了——你**看不到 true,只能用代理信号猜拐点**,猜早了丢改进,猜晚了被 hack。归一化(把 RM 分标准化,削平绝对尺度)能稳住训练、防止 proxy 绝对值爆炸误导你,但它治标:**它让 proxy 曲线更平稳,却不改变 proxy 和 true 解耦这个根本事实**。

**3. 修 RM 本身**(扩偏好数据、去 length 偏置)。这是治本,但代价最大:回到第 5 章的活,要更多标注、更难的去偏。而且——⚡ 这正是开放问题所在——**你无法证明修好的 RM 没有别的、更隐蔽的偏置**。length 偏置好检测(画个长度 vs 分数图就看出来),但 RM 可能学到一堆你想不到的伪相关(谄媚、格式偏好、特定措辞),这些目前没有通用的检测手段。

一句话收束三者:**proxy 与 true 在优化下必然解耦,hacking 就出现在解耦处。** KL 系数是限制解耦幅度的旋钮,early stop 是在解耦前喊停的闸,修 RM 是把解耦点往后推——但没有一种能消灭解耦本身。这是 Goodhart 定律在 RLHF 上的具体形态。

## 为什么这章是 DPO 的引子

走到这里,RLHF 的全部复杂度都暴露了:你要训一个 RM(第 5 章),再用 PPO 优化它(第 4 章),还要拴一根 KL 缰绳、调 β、盯 KL 仪表盘、赌一个早停点——而这一整套的脆弱性,全卡在「RM 是个有缺陷的代理」这一个点上。

很自然的问题就冒出来了:**能不能不要这个显式的 RM?** 如果偏好数据本身就编码了「人喜欢 A 胜过 B」,能不能跳过「训 RM → 跑 PPO」这两步,直接用偏好 pair 调策略?这正是第 7 章 DPO（Direct Preference Optimization,直接偏好优化)要回答的——它用一个巧妙的数学变换,把上面那个「奖励减 KL」的目标函数**解析地**重写成一个直接在偏好 pair 上的分类损失,RM 和 PPO 双双消失。

但记住本章的教训:DPO 绕开了**显式** RM,可它绕不开**proxy 与 true 解耦**这个本质。偏好数据里的 length 偏置照样会传染给 DPO 策略——只是换了个地方藏。第 7 章会用同一个确定性环境,量给你看 DPO 在哪一点上重新踩进 reward hacking 的坑。缰绳的形态变了,但你依然需要一根缰绳。
