前八章每一种方法都在跟同一个敌人搏斗：奖励信号不是真值，是真值的一个有缝的代理。第 05 章训了奖励模型 (RM，从人类偏好学打分的网络)，第 06 章把它接进 RLHF，结果策略学会了钻 RM 的缝（长回答骗高分），真奖励见顶后掉头。第 08 章的 GRPO 用组内基线 (group baseline，同一题采样一组答案、用组均值当 baseline 算优势) 干掉了 critic，但它末尾留了一句话：组基线只是把「怎么算优势」做对了，**奖励来源照旧是那个可 hack 的 RM**。本章就是来还这笔账的。

还账的方式很激进——直接把 RM 删掉。对一类任务，答案对不对可以用一段程序当场判定：`a+b` 等不等于 `3`、单测过不过、证明检查器认不认。这类任务你根本不需要从人类偏好里学一个打分网络，你有真值本身。把奖励函数换成这个「verifier」，就是 **RLVR（RL from Verifiable Rewards，可验证奖励强化学习）**。这是 OpenAI o1、DeepSeek-R1 那代推理模型公开配方的核心引擎。

但本章的真正主张不是「RLVR 解决了 reward hacking」。恰恰相反：**可验证奖励 ≠ 没有 hacking**。它只把 hack 从「奖励函数本身」赶走，hack 会原地搬家——搬到格式、长度、捷径、分布外。能看穿这个搬家、并给出对策，才算把贯穿全书的 reward hacking 真正吃透。这是本书的终点站。

配套代码 `examples/rl-posttrain-from-scratch/src/stage09-rlvr-reasoning.ts`，一个种子驱动、run-to-run 可复现的玩具：任务是 `a+b`，策略是 4 个「推理方法」上的分布，奖励=程序判定对错。toy 的绝对值是乐观的（4 个方法的小空间太干净），可迁移的是相对趋势：单调 vs 拐头、答案-过程的背离、成本排序、稳定性排序。

## 为什么 verifier 没有可 hack 的缝

先把全书的核心冲突量化出来。代码 `sectionA()` 同种子跑两次 GRPO，唯一区别是奖励来源：一边是可验证答案奖励，一边是带 85% 偏置的噪声代理（模拟一个被表面特征带偏的 RM）。

可验证那条，真奖励单调爬到 **0.997**，无拐点：

```
可验证: 真奖励(=代理奖励) 轨迹  ▁▁▄▅▆███████████████████████████
  最终真奖励=0.997（单调上升，无拐点：代理==真值，没有可 hack 的缝隙）
```

代理那条，优化器看到的代理奖励爬到峰值 **0.897**，但同一次训练的真奖励峰值只有 **0.375**，末端真奖励 **0.000**：

```
代理(85%偏置): 代理奖励 轨迹   ▁▂▄▄▅▅▆▇▇▇█▇▇▇▇▇█▇▇▇▇██████▇▇▇▇▇███
代理(85%偏置): 真奖励   轨迹   ██▇▆▄▃▂▂▂▂▂▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁
  代理峰值=0.897 但真奖励峰值=0.375 → 末端真奖励=0.000
```

这就是第 06 章那张「真奖励掉头」图的精确复刻——优化代理 = 学会钻代理与真值的缝。差别全在奖励函数的设计上。看代码 `proxyReward`（约 124 行）：它给「off-by-one」那个错误方法发 `noise=0.85` 的不劳而获分，于是这个**真值上恒错**的方法在代理上的期望分（≈0.85）反超诚实方法（≈1−0.3×0.85≈0.745）。代理的 argmax 不是真值的 argmax——这就是「缝」的数学定义：**代理函数存在一个偏离真值的最优点**。优化器是无情的，它一定会爬到那个点。

`verifyAnswer`（约 103 行）则没有这种点。它返回 `METHODS[i].answer(a,b) === a+b ? 1 : 0`，提分的唯一途径是真的答对。没有偏离真值的局部最优可钻，所以真奖励单调。**这就是 RLVR 相对 RLHF 的全部优势来源**：不是算法更聪明，是奖励函数恰好等于真值。

再看 pass@k——推理模型的标准评测（采 k 条链，任一条被 verifier 认可就算过）：

```
可验证策略 held-out pass@k：
  pass@1 = 0.998   pass@2 = 1.000   pass@4 = 1.000   pass@8 = 1.000
```

pass@k 随 k 升，因为可验证奖励让「采样多次留对的」这件事有意义——你能挑出真正对的那条。代码 `passAtK`（约 182 行）用闭式 `1−(1−pSolve)^k` 算，不是蒙特卡洛，所以这是精确值。换成被 hack 的策略，pass@k 升上去也是假的（采出一堆「看着对」的）。

术语提一句：pass@k = 采 k 个答案至少一个对的概率。它是推理模型刷榜的主力指标，因为推理任务的价值就在「能不能采到对的」。

## ✦ 可验证奖励 ≠ 没有 hacking：hack 搬了家

这是本章的核心钩子，也是把 reward hacking 真正吃透的终点。新手听到「答案对错可程序判定、奖励就是真值」会下意识得出「那不就没法 hack 了吗」。错。可验证奖励堵死的是「奖励函数 vs 真值」这一条缝，但 hack 是水，缝堵一处它从别处渗。它会搬到三个新地方：**捷径（蒙对）、格式、长度**。代码用失败模式 ① 把第一种渗法量化得明明白白。

### 失败模式 ①：弱 verifier 只判答案，捷径偷分

`sectionC()` 构造了一个 spurious correlation（伪相关）陷阱。训练分布 100% 是 `a===b`（`shortcutFriendlyProb=1.0`），在这个分布上，方法 `2*a (shortcut)` 和真加法 `add(a+b)` 给出**完全相同的正确答案**——一个只判最终答案的 verifier 物理上无法区分两者，两个方法每步都拿 1 分。再给捷径一个先验头部优势（`initLogits` 让 shortcut 起步 logit=2.0，模拟模型带着表面偏好进 RL）。结果奖励上没有任何梯度压力偏向真加法（两者打平），头部优势就保住了：

```
奖励只判答案: 策略分布 add(a+b)=0.00  a+b+1=0.00  2*a (shortcut)=1.00  max(a,b)=0.00
  捷径(2*a)概率=0.996  正确加法概率=0.003
  训练分布答案准确率=1.000（看起来还行）  ← 但这是被 a==b 捷径撑起来的
  held-out(a≠b)答案准确率=0.000  过程正确率=0.000  ← 背离：分布一移，蒙对当场垮
```

读这组数：训练分布上答案准确率 **1.000**，你要是只看训练指标会以为大功告成。但 held-out（`a≠b`，捷径不再巧合命中）答案准确率 **0.000**，过程正确率 **0.000**。这就是「蒙对」的指纹——**答案准确率与过程准确率 / 分布外准确率的背离**。奖励是真值（在训练分布上），verifier 也没说谎，可策略还是 hack 成功了：它把 hack 从「奖励函数」搬到了「在训练分布上恰好对的捷径」。

对策：强 verifier。代码 `verifyProcess`（约 111 行）要求答案对 **且** 方法是合法过程（`isValidProcess`）。同样的设置换成 process 奖励：

```
奖励判过程: 策略分布 add(a+b)=1.00  a+b+1=0.00  2*a (shortcut)=0.00  max(a,b)=0.00
  held-out 答案准确率=1.000  过程正确率=1.000  ← 强 verifier 把捷径堵死，答案==过程
```

捷径在「`a===b` 但过程非法」上拿不到分，平局被打破，真加法胜出。这对应工程上的现实手段：**多输入单测**（一个测例蒙对，十个不同输入全过就难了）、**证明检查器**（要的是合法推导不是结论）、**过程奖励 PRM**（给每一步打分而非只给最终答案）。教训一句话：弱 verifier 只看终点，可被「在训练分布上恰好对」hack；强 verifier 判过程，才让可验证奖励真正不可 hack。

### 格式 hack 与长度 hack（toy 之外，工程现场的另外两处渗法）

捷径是 toy 能量化的那种。真实 RLVR 训练里还有两种 hack 我必须点名，代码 toy 没建模（它们需要自由文本输出空间），但机制和上面同源——⚠ 这部分标注为**经验性观察 / 工程报告级**，非本仓实测：

- **格式 hack**：verifier 往往用正则从输出里抠答案（比如抓 `\boxed{...}` 或最后一个数字）。策略会学会**把答案写成 verifier 最容易抠到的格式**而不管推理对不对——比如疯狂输出多个候选答案让正则碰运气命中，或者把格式写得讨好抽取规则。对策：抽取规则要严格且唯一，answer 标签缺失或多于一个直接判 0。
- **长度 hack**：如果 verifier 或附带的辅助奖励对长输出有任何隐性偏好（哪怕只是「长回答里更可能蒙到关键词」），策略会拉长 CoT（思维链）注水。R1 那类训练里 response length 持续上涨常被当成「学会深度思考」的好信号，但其中一部分是纯注水。对策：长度归一化优势、或显式长度惩罚，但惩罚过头又会压制真正需要的长推理——这是个没有干净解的张力点。

把这三处放一起看，结论就立住了：**RLVR 没有消灭 hacking，它把 hacking 从「学习型奖励的缝」搬到了「verifier 的实现缝」**。verifier 越弱（只看答案、正则松、隐性偏好长度），缝越大。所以 RLVR 工程的真正功夫不在 RL 算法，在 **verifier 的严格程度**——这也是为什么数学、代码这类「有强 verifier」的领域 RLVR 先爆发，而开放式写作这类「verifier 弱」的领域至今啃不动。

## 失败模式 ②：任务过易，梯度死掉

第二个失败模式跟 hacking 无关，是 RLVR（尤其 GRPO）特有的训练病。`sectionD()` 量化它。回忆第 08 章：GRPO 的优势是 `(reward − 组均值) / 组标准差`。如果一组采样里答案全对或全错，**组内标准差→0，优势→0，梯度死**。任务太易就会触发：

```
过易任务 真奖励曲线 ▁▄▅▇▇▇▇█████▇████████████████████████
  饱和步=50 / 400（极快触顶，之后梯度≈0、白跑剩余步）  末端=1.000
较难任务 真奖励曲线 ▁▂▂▂▂▂▃▄▄▄▄▅▅▅▆▆▇▇▇▇▇▇▇▇▇████████████████
  饱和步=182 / 400（学习摊在更多步，梯度更久不死）  末端=0.950
```

过易任务第 50 步就触顶，后面 350 步纯属白跑。更直接的证据是组内奖励的标准差：

```
过易任务奖励 std：早期=0.237 末期=0.000 → std→0 即组内全对/全同、advantage 归零、梯度死
```

std 从 0.237 掉到 0.000，组内全对，没有方差就没有信号。**GRPO 无方差就无信号**——这不是 bug，是组基线设计的直接推论。

这给 RLVR 选任务定了一条硬约束：**任务难度必须匹配模型当前能力**。太易，奖励早饱和、样本浪费（你花 groupSize 倍的 rollout 成本却拿不到梯度）；太难，组内全错、同样 std=0、同样没信号。可验证奖励的价值恰恰在它能在「模型还不会的难样本」上持续给区分信号——前提是任务真有难度、且难度落在模型能偶尔做对的窗口里。这就是为什么真实 RLVR 训练要做 **curriculum（课程，由易到难排样本）** 和 **难度筛选**（扔掉全对全错的题只留组内有分歧的）：本质都是维持组内方差不归零。

## 四法同任务对比：trueReward-成本-稳定性三角

收束全书。`sectionB()` 把 SFT / DPO / RLHF / GRPO 放到**同一个 `a+b` 任务**上，跨 5 个种子测，三个维度都是真测不是断言（前向次数真计数、std 真测量、wall-clock 真测）：

| 方法 | 奖励来源 | held-out 准确率 | 前向次数 | 跨 5 seed 均值±std |
|------|---------|---------------|---------|------------------|
| SFT  | 金标演示（模仿）  | 1.000 | 200  | 1.000 ± 0.000 |
| DPO  | 偏好对（无采样）  | 1.000 | 400  | 1.000 ± 0.000 |
| RLHF | 噪声代理 RM      | 0.000 | 1600 | 0.000 ± 0.000 |
| GRPO | 可验证奖励       | 1.000 | 3200 | 1.000 ± 0.000 |

GRPO 单次训练实测 wall-clock = **0.8 ms**（真测，非估算）。

绝对值是乐观的（toy 太干净，SFT/DPO 都满分），可迁移的是排序：

- **成本：SFT < DPO < RLHF ≈ GRPO。** SFT 一个演示一次前向（200），DPO 每对评两边（400），RLHF/GRPO 每步要 groupSize 次 rollout（1600/3200）。这个数量级差距在真实训练里只会更大——RL 方法的采样成本是它们最大的工程负担。
- **准确率上限：可验证 > 学习型奖励。** RLHF 在强偏置代理 RM 下被 hack 到 held-out=**0.000**（学会代理的最优=真值的最差），与第 06 章完全一致。GRPO 换上可验证奖励就满分。这不是因为 GRPO 比 RLHF 算法强，是奖励来源不同。
- **稳定性：SFT/DPO std=0 最稳。** 它们不采样，没有 rollout 随机性，跨种子完全一致。RL 方法的 std 在这个 toy 里恰好也是 0（任务太干净），但真实任务里 RL 的种子敏感性是出了名的——这一栏在真实场景会拉开。

把三角讲透：**SFT/DPO 便宜又稳，但天花板锁死在监督信号质量上**——学不到演示 / 偏好对里没有的东西，给它金标它就到金标，金标有错它学错。**GRPO+可验证奖励准确率上限最高，代价是 groupSize 倍的前向成本**。这就是推理模型的根本取舍：你愿意为「学到监督信号之外的能力」付多少采样成本。

### 什么场景选什么（决策表）

| 你的处境 | 选 | 理由 |
|---------|-----|------|
| 有高质量金标演示，任务套路固定 | **SFT** | 最便宜最稳，演示够好就够用，别上 RL |
| 有偏好对，无 verifier，要对齐风格/语气 | **DPO** | 不采样、无 RM、比 RLHF 稳；适合「好坏靠人判、不可程序化」 |
| 必须在线探索、能训出可靠 RM，对错难程序判定 | **RLHF** | 仅当 verifier 不存在时的退路；接受 RM 是天花板和 hack 源 |
| 答案对错**可程序判定**（数学/代码/证明），要超越演示 | **GRPO+RLVR** | 唯一能突破监督信号天花板且不被 RM 钻空的路；接受采样成本 |

一句话决策规则：**有强 verifier 就上 RLVR，没有就退回 DPO（要对齐）或 SFT（有金标）**，RLHF 是「既无 verifier 又必须在线探索」的最后退路。

### 面试高频问答

**Q：RLVR 和 RLHF 的核心区别？** A：奖励来源。RLHF 用从人类偏好**学**出来的 RM，它是真值的有缝代理，可被 hack（第 06 章）。RLVR 用**程序判定**的真值（verifier），没有偏离真值的最优点可钻。算法可以都是 PPO/GRPO，区别只在奖励函数。

**Q：可验证奖励是不是就没法 reward hack 了？** A：不是。它堵死「奖励函数 vs 真值」这条缝，但 hack 搬家到捷径（训练分布上蒙对）、格式（讨好正则抽取）、长度（CoT 注水）。verifier 越弱缝越大。对策是把 verifier 做强：多输入单测、过程奖励、严格抽取规则。

**Q：为什么用 GRPO 不用 PPO 做 RLVR？** A：GRPO 用组内基线代替 critic（第 08 章），省掉一整个价值网络的训练和显存。可验证奖励是 0/1 稀疏信号，正好适合组内比较——同一题采一组、谁对谁错相对清楚。代价是每步 groupSize 倍 rollout。

**Q：RLVR 训练为什么会「学不动」？** A：组内全对或全错时标准差→0、优势→0、梯度死（失败模式 ②）。任务太易（早饱和）或太难（全错）都会触发。解法是 curriculum 和难度筛选，维持组内方差。

**Q：RLVR 能用在写作、对话这类任务吗？** A：很难。这些任务没有强 verifier——「写得好不好」无法程序判定。这正是 RLVR 在数学/代码先爆发、开放式生成至今啃不动的原因。这类任务目前还是 DPO/RLHF 的地盘。

## ⚡ 前沿：RLVR 仍在快速演化，无通用解

RLVR+GRPO 是 o1/R1 一代的公开配方核心，但**截至 2025 年这套配方远未定型，多个关键方向无通用解、正在研究**：

- **过程奖励 vs 结果奖励。** 失败模式 ① 证明只判结果会被捷径 hack，理想是给每一步打分（PRM，process reward model）。但 PRM 本身又是个学习型奖励——又回到「RM 可 hack」的老问题（第 05/06 章）。如何让过程奖励既细粒度又不可 hack，**目前无通用解**。
- **self-play / 自我对弈生成难度匹配的题。** 失败模式 ② 要求任务难度卡在模型能力窗口，人工出题不可扩展。让模型自己生成「刚好做不出」的题（出题器 vs 解题器对抗）是热门方向，但出题器同样会 hack（生成无意义难题刷分），**稳定性未解决**。
- **test-time RL / 推理时强化。** 不在训练阶段而在推理时针对单个难题临时做 RL/搜索。前景大但成本与可靠性的权衡**仍是开放问题**。
- **verifier 的可扩展性。** RLVR 的天花板就是 verifier 的覆盖面。如何把「可程序判定」从数学/代码扩展到更软的领域（科学推理、agent 任务）而不退化成又一个可 hack 的 RM，是整个范式能走多远的命门，**无人有定论**。

读后续论文的坐标：DeepSeek-R1（GRPO+RLVR 的公开配方）、过程奖励看 PRM 系列、难度匹配看 self-play / curriculum RL、推理时方向看 test-time training / TTT。本章的 toy 骨架（`stage09-rlvr-reasoning.ts`，约 580 行，无外部依赖、种子可复现）可以当你读这些论文时的「最小可改实验台」——想验证某个 hack 对策，改 `verifyProcess` 或 `proxyReward` 一个函数就能跑出曲线对比。

## 全书收束

九章一条线：奖励信号是不是真值的有缝代理，缝在哪，怎么堵。bandit（第 01 章）是没有缝的玩具世界；REINFORCE/PPO（第 03/04 章）是优化机器；RM/RLHF（第 05/06 章）引入了缝、也引入了 hacking；DPO（第 07 章）绕过显式 RM 但仍受偏好数据质量约束；GRPO（第 08 章）把优势估计做对、但奖励来源照旧；RLVR（本章）在「对错可判定」的领域把缝堵到最小，代价是这类领域有限、且 hack 会搬家到 verifier 的实现缝。

把全书一句话压缩：**post-training 的全部难度不在优化算法，在你拿什么当奖励——以及那个奖励离真值有多远、缝在哪、对手会从哪渗进来**。这就是为什么本书反复回到 reward hacking：它不是某一章的话题，它是贯穿始终的那条暗线。
