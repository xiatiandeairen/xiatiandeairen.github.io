先说一个会在生产训练里坑你的具体场景:你照着论文搭了一个 8 专家 MoE,前几章的稀疏路由(第 02 章)、不可导门控的梯度回传(第 03 章)、容量丢弃(第 05 章)都验过了,单测全绿。开训。loss 曲线漂亮地往下走,你去喝杯咖啡。回来一看,loss 还在降——但你顺手 dump 了一下每个专家被路由到多少 token,发现一个专家吃了几乎全部 token,剩下 7 个专家里大半**整个训练过程一次都没被激活过**。那些专家的几千万参数,是死的。你花了 8 倍的参数预算,实际只训出了一个略胖的稠密 FFN。

这就是**专家坍塌(expert collapse,门控退化到只挑少数几个专家)**。它是 MoE 的头号病症,最阴险的地方在于:**它不在 loss 上报警**。前面五章每一章都埋了一个会助推坍塌的因素;这一章把它们同时打开,看病症怎么爆发,再用前几章的解药做对照消融,看清每味药到底治哪个漏洞、哪味药其实是安慰剂。

本章引用的所有数字都来自配套仓 `examples/moe-from-scratch/src/stage06-expert-collapse.ts` 的真实运行(`npx tsx src/stage06-expert-collapse.ts`,seed=7,确定性可复现),不是示意值。度量与监控面板来自 `src/core/metrics.ts` 和 `src/core/viz.ts`——它们就是全书量化诚实的仪器,坍塌的每一个声明都是其中一个数字在动。

> 诚实声明:stage06 用的是 toy 合成数据(`makeTokenStream`,token id % 4 == topic),**绝对值无意义,本章从不报准确率**。可迁移的是**熵/CV 的相对趋势和曲线形状**,不是某个具体数能搬到千卡集群上。这与配套仓 README 的「诚实数字约定」一致。

## 这一章只动 router,不动专家

先讲清楚 stage06 的实验设计,否则后面的数字会被误读。`runTrial` 里**只有门控(gate)在训练,专家和 embedding 全部冻结**:

```typescript
// src/stage06-expert-collapse.ts
const embedding = new Embedding(VOCAB, EMBED_DIM, initRng);
const gate = new Linear(EMBED_DIM, NUM_EXPERTS, initRng);
// Only the gate trains. Embeddings are frozen on purpose (see header).
const opt = new Adam(gate.parameters(), AGGRESSIVE_LR);
```

为什么要冻结专家?因为专家质量是另一根轴(第 01、07 章),如果让专家一起训,熵曲线塌下去就分不清是「门控自己锁死了」还是「某个专家真的变强了所以门控该选它」。**冻结专家把混杂因子去掉,熵曲线就纯粹反映路由动力学**——这是隔离变量的标准做法,也是为什么这章的坍塌能干净地确定性复现。

还有一个关键的实验纪律:每个变体(baseline、+aux、+noise……)都重建同一套 `rng(SEED)`,所以 embedding、token batch、noise 抽样逐位相同,**变体之间唯一的差别就是那味药**。任何曲线差异都能归因到药,不是采样运气。

## 坍塌不是退化,是相变

第一个反直觉的点,也是本章的核心专家判断:**坍塌不是缓慢退化,是正反馈失控导致的相变(phase transition,系统状态突然跳变),常表现为训练初期某一刻突然「锁定」。**

机制的根子在 stage06 故意挑的那个**坍塌驱动 loss**:每个 token 的 task loss 是 `-log(prob[chosen])`,即门控对它**自己当前那一票**的交叉熵——「对你已经偏好的那个专家更自信一点」。

```typescript
// src/stage06-expert-collapse.ts — runTrial 内
const chosen = topExpert(probs);
// Collapse driver: target 是门控自己当前的 pick,梯度只会强化赢家。
taskLoss = taskLoss.add(crossEntropy(logits, chosen));
```

这个目标的**唯一全局最优解就是「把 100% 的 token 路由到单个专家、概率 1」**——也就是彻底坍塌、熵→0、其余全死。没有任何一项奖励分散。于是循环是这样转的:

1. 初始化时某个专家因随机性碰巧 logit 略高,赢下一批 token;
2. 赢家是 top-1,所以只有它被 dispatch,**只有它的梯度路径被走到**;
3. loss 推门控对它更自信 → 下一步它赢更多;
4. 回到第 2 步,差距拉大。

这是**富者愈富**,而且没有负反馈刹车——除非你显式加一个(那正是第 04 章 aux loss 干的事)。正反馈系统的典型行为不是线性下滑,而是越过临界点后**雪崩式锁定**。stage06 的 baseline 跑出来正是这个形状,熵曲线(`plotLoss` 渲染,21 个采样点)肉眼可见地先快后稳地砸向 0:

```
路由熵 (nats, 越低越坍塌)（按步采样）：
  █▇▅▄▃▂▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁  [0.0517 .. 2.0780], n=21
  step 0 -> 299: 2.0780 -> 0.0517 nats
最大熵参考 = ln(8) = 2.0794 nats（完全均衡上界）
```

注意起点 2.0780 几乎顶到 `ln(8)=2.0794` 的理论上界——开训那一刻门控完全均匀、毫无偏好。然后塌到 0.0517,离 0 一步之遥。死掉的专家因为再也收不到 token,梯度恒为零,**永久死亡**——这不是「训久了会好」,它出局了。

为什么早期最危险?因为早期专家之间几乎没差异,门控选择近乎随机,任何一点初始不对称都会被循环放大。这就是为什么大规模训练里 **warmup 和初始化**对坍塌如此敏感(后面 ⚡ 节展开)——它们决定了相变前你有多少缓冲。

### 最隐蔽的一种:部分坍塌

真正会让你在复盘会上下不来台的不是全坍塌,而是**部分坍塌**:8 个专家里活了几个、死了几个,整体 loss 看着完全正常,甚至比某些健康配置还低一点(因为活着的专家更专精)。你盯着 loss 曲线,一切岁月静好。

✦ **专家钩子:监控路由指标比监控 loss 更早、更准地预警坍塌。** loss 是被许多因素平均掉的标量,死专家的损失被活专家的优秀表现掩盖。但路由指标直接看分布:利用率向量里有几个 0、熵掉了多少,一目了然。**所以坍塌的一等观测面是路由日志,不是 loss。** 这条原则在 toy 和千卡集群上同样成立,区别只是面板的规模。

## 触发条件:把五章的雷同时踩响

要稳定复现坍塌,得让正反馈循环跑赢一切刹车。stage06 的配置(`E=8`、`top-1` 硬路由、`lr=0.05` 偏大、无 aux、无 noise)就是这么调的,对应前面各章「不这么做就会出事」的伏笔:

| 触发条件 | 机制 | 对应章节伏笔 |
|---|---|---|
| **无负载均衡** | 没有任何力量惩罚「都挑同一个」,正反馈无刹车 | 第 04 章:aux loss 就是那个刹车 |
| **学习率过高(0.05)** | 门控参数更新步子大,临界点更早到、锁定更快 | 第 03 章:router 梯度本就经 gate 权重缩放,再叠高 lr 易过冲 |
| **对称破缺不足** | 专家初始化太接近,门控只能随机抢跑,但一旦抢跑就赢家通吃 | 第 01 章:专家是 `nn.ts` 里同构的层 |
| **容量过紧** | 容量上限太低,被冷落的专家偶尔抢到的 token 也被挤掉,雪上加霜 | 第 05 章:capacity drop 在均衡缺失时会**加速**而非缓解坍塌 |

第四条值得多说一句,因为它推翻一个常见误解。很多人以为容量上限是用来**防止**某个专家过载的——它确实能防过载。但在**没有负载均衡**的前提下,容量过紧反而**助推**坍塌:当热门专家满了、token 被丢,被丢的 token 本可以是冷专家复活的机会,结果连这点机会也被容量机制抹掉了。第 05 章里 capacity drop 是救火工具;在本章的病态配置里,它成了帮凶。这就是为什么单独验过每个机制不等于组合起来安全——**坍塌是组合病,不是单点 bug。**

## 三个诊断信号

坍塌没有单一的「报错」,你得靠三个互相印证的信号。它们全部由 `src/core/metrics.ts` 真算,不是肉眼看曲线。

### 信号一:路由熵塌缩

`routingEntropy` 算的是门控输出概率行的香农熵(单位 nats):

```typescript
// src/core/metrics.ts
export function routingEntropy(probs: number[]): number {
  let h = 0;
  for (const p of probs) if (p > 0) h -= p * Math.log(p);
  return h;
}
```

健康时门控对一个 token 的概率分布接近均匀,熵接近上界 `ln(E)`(8 专家约 2.08 nats);坍塌时门控变成近似 one-hot,熵塌向 0。stage06 baseline 从 2.0780 跌到 0.0517,**就是坍塌的指纹**。注意 `if (p > 0)` 这个守卫——它不是装饰,是防止 `log(0)` 在死专家(概率恰为 0)上炸出 `NaN`,而坍塌时恰恰会大量出现 0 概率项。仪器本身得能在病态输入下活着。

### 信号二:利用率 CV 飙升

熵看的是单 token 的门控犹豫程度;利用率看的是**全局**每个专家实际吃了多少 token。`expertUtilization` 返回长度 E、和为 1 的向量,再用 `coefficientOfVariation` 压成一个标量:

```typescript
// src/core/metrics.ts
export function expertUtilization(assignments: number[], E: number): number[] {
  const counts = new Array(E).fill(0);
  for (const a of assignments) {
    // INVARIANT: 越界 id 直接 throw —— silent drop 会把路由 bug 伪装成正常分布。
    if (a < 0 || a >= E) throw new Error(`expertUtilization: assignment ${a} outside [0,${E})`);
    counts[a]++;
  }
  const n = assignments.length || 1;
  return counts.map((c) => c / n);
}
```

选 CV(变异系数 = 标准差 / 均值)而不是裸标准差,是因为它**无量纲、跨 E 和跨 token 数可比**。完美均衡时所有专家都是 `1/E`,CV=0;一个专家通吃时 CV 冲向 `sqrt(E-1)`(8 专家约 2.65)。stage06 baseline 的 CV 从 1.6539 升到 2.6458——几乎顶到那个上界,说明**真的是单个专家在吃**:

```
利用率 CV (越高越坍塌)（按步采样）：
  ▁▁▁▃█████████████████  [1.6539 .. 2.6458], n=21
  step 0 -> 299: 1.6539 -> 2.6458
```

这里有个值得当场推回的工程取舍:**熵和 CV 不是冗余,它们抓的是正交维度。** 熵看的是门控的「软概率信念」,CV 看的是 argmax 之后的「硬利用率」。下一节你会看到 noisy gating 能把熵抬回 1.27 却完全压不动 CV(仍是 2.646)——这就是两者解耦的铁证。部分坍塌尤其爱躲在「熵尚可、CV 已高」的缝里,**只盯一个指标会漏报**。注意上面 `expertUtilization` 对越界 id 直接 `throw` 而非静默丢弃——静默丢一个 token 会把路由 bug 伪装成正常分布,宁可炸也不要假阳性的健康。

### 信号三:共现热图退化成单列

前两个是标量,第三个是结构。stage06 的合成数据带 ground-truth 话题(`makeTokenStream` 暴露 `topic`,token id % 4 == topic),统计「专家 e × 话题 t」的共现次数,过 `viz.heatmap` 渲染成 ASCII(`RAMP = " ░▒▓█"`)。健康分工时热图应该多行点亮(不同话题去不同专家);坍塌时退化成**一整行全亮、其余空白**——所有话题不管语义如何,全被同一个专家接走。stage06 baseline 的最终利用率和热图把全坍塌画得明明白白:

```

E1 │······························│ 0.0000
E2 │██████████████████████████████│ 1.0000   ← 通吃
E3 │······························│ 0.0000
E4 │······························│ 0.0000
E5 │······························│ 0.0000
E6 │······························│ 0.0000
E7 │······························│ 0.0000
死专家数（从未被任何 token 选中）= 7 / 8

专家×话题 共现热图（坍塌后退化成几乎只有 1 行亮）：
    t0t1t2t3
 E2 ▓▓██▓▓▓▓     ← 一行通吃 = 无分工
```

E2 一柱独大、占比 `1.0000`,7/8 死专家,热图只有 E2 那一行亮——四个话题全涌向它。这是最直观的「专家没学到分工」的视觉证据,也是为什么本书坚持用能在 stdout 里复现的 ASCII 图而非 PNG:你能自己跑出来对拍。

下面是把三个信号拼到一起的**最小监控面板**(取自 stage06 baseline 终态):

```

```

✦ 回到那个钩子:**这块面板,而不是 loss 曲线,才是坍塌的一等观测面。** loss 此刻可能仍在平稳下降——baseline 训练时 task loss 一路在降,因为「对单个专家越来越自信」本身就是在最小化那个坍塌驱动 loss。**loss 漂亮和路由健康是两回事。**

## 逐一施药:消融每味药的边际贡献

诊断完,做对照实验。stage06 从全坍塌的 baseline 出发,一次只加一味药,跑 4 个变体,打印每味药相对 baseline 的边际贡献(熵增益 / 复活专家数 / CV 变化)。这是本章作为「反面对照」的核心价值——不是「加上这些就好了」,而是看清**每味药各自堵哪个漏洞,以及哪味药其实没用**。先看全表:

```
变体                 final熵(nats)  final-CV   死专家数
  baseline          0.0517        2.646      7/8
  +aux(0.01 弱)      0.0540        2.646      7/8
  +aux(1 调优)       1.7167        0.242      0/8
  +noise(σ=8)        1.2691        2.646      7/8
  +aux(1)+noise      2.0071        1.847      6/8
```

这张表有三个会颠覆直觉的结论,逐个拆。

### 药一(失败的默认值):弱 aux(0.01) 几乎无效

第 04 章的 `loadBalanceLoss` 是负载均衡的主药,但**系数(coefficient)是个必须按坍塌压力标定的旋钮,不是魔法常数**。很多教程随手抄一个 `0.01` 的默认值——stage06 专门跑了这个默认值,结果是:

```
弱aux(0.01)  : 熵 +0.0023 nats, 复活 0 个, CV 2.65->2.65  （太弱，几乎无效）
```

熵只动了 0.0023,死专家还是 7/8,CV 纹丝不动。**这是个诚实的负面结果**:在本章这么强的坍塌驱动 loss 下,`0.01` 的均衡力被 task loss 彻底压过,加了等于没加。如果你照搬默认值就以为「我加了 aux loss 所以安全」,这正是会坑你的地方。

### 药二(真解药):调优 aux(1.0) 让死专家全部复活

把系数提到 `1.0`(按这个坍塌压力标定),aux loss 才真正成为正反馈循环唯一缺的**负反馈**:

```typescript
// src/core/metrics.ts — Switch/GShard 风格,对 gate probs 可导
//   aux = E * sum_e ( f_e * P_e )
export function loadBalanceLoss(gateProbsPerToken: Value[], assignments: number[], E: number): Value {
  // f_e: 硬计数比例,CONSTANT(无梯度,来自 argmax)
  const f = expertUtilization(assignments, E); // numbers
  // P_e: 门控平均概率质量,Value(带梯度)
  // ... 累加每 token gate probs 求均值得 P_e,再 dot(f, meanP) 乘 E ...
}
```

读懂公式才懂它为什么能刹车:`f_e` 是实际分到专家 e 的硬比例(**常数**,argmax 出来的,不可导),`P_e` 是门控分给 e 的平均概率质量(**带梯度**)。某个专家被过度选中时,它的 `f_e` 和 `P_e` 同时变大,乘积爆涨,aux loss 把梯度推回去**压低**门控对它的概率——正好抵消富者愈富。`E` 因子让完美均衡时 aux=1.0,打印值直接读成「偏离均匀几倍」。

stage06 跑出来这是把曲线真正扳回来的药:

```
调优aux(1): 熵 +1.6650 nats, 复活 7 个, CV 2.65->0.24  （真解药：死专家归零、CV 崩塌）

调优aux 最终利用率（回到多专家分担）：
E0 │██████████████················│ 0.0742
E1 │███████████████████···········│ 0.0977
E2 │████████████████████████████··│ 0.1445
E3 │██████████████████████████████│ 0.1563
E4 │███████████████████████████···│ 0.1406
E5 │█████████████████·············│ 0.0898
E6 │█████████████████████████████·│ 0.1523
E7 │████████████████████████████··│ 0.1445
```

死专家 7→0,CV 2.646→0.242,8 个专家全部回到 0.07~0.16 的分担区间。对比 baseline 那个 E2 占 `1.0000` 的单柱,这就是「均衡」长什么样。

这里还藏着一个**经典实现 bug**,本书的函数签名故意防住它:`f_e` 必须是普通 `number`(detach),不能是 `Value`。如果你手滑把 `f_e` 也做成带梯度的 `Value`,梯度会经那个不可导的硬计数乱流,均衡力失效——而且不报错,你只会发现 aux 加了等于没加(症状和上面「弱 aux」一模一样,但根因完全不同)。`metrics.ts` 注释里专门写了「Getting this wrong (making f_e a Value) is a classic bug that this signature prevents」。**这就是本书「绑定可跑代码」的意义:这种 bug 在 survey 稿的公式里看不见,只有在能跑的类型签名里才暴露。**

### 药三(治标不治本):noisy 路由只抬高信念熵,压不动硬利用率

第 03 章为让 argmax 路由可学,在 gate logits 上加噪声(noisy gating,Shazeer 2017)。直觉上它该破坏早期锁定、缓解坍塌。stage06 用 `σ=8` 单独跑了它,结果是本章最反直觉的一条:

```
noise(σ=8)  : 熵 +1.2174 nats, 复活 0 个, CV 2.65->2.65  （只抬高门控信念熵，硬利用率没回落）
```

熵从 0.05 抬到 1.27,看着像在恢复——但**复活 0 个专家,CV 还是 2.646**。为什么?因为 noise 只在训练期加到 logits 上,让门控的**软概率**显得更犹豫(信念熵升高);但评估期读的是干净 logits 的 **argmax**,硬利用率该谁通吃还是谁通吃。

```typescript
// src/stage06-expert-collapse.ts — gateForward
if (noiseStd > 0) {
  // noisy gating:训练期给 logits 加高斯噪声,让 runner-up 偶尔翻盘。
  // INVARIANT: 噪声只在训练期加;eval 读干净 logits,报的是门控真实信念。
}
```

**结论:noisy gating 是「探索」不是「均衡」,两者正交。** 它让抢跑变难、推迟相变,但没有任何力量主动把 token 推回冷专家。这就是为什么 Shazeer 的原始方案里 noisy gating 和 load-balance loss 是**一起上**的——单独的噪声治标。

### 反作用:aux + noise 不能机械相加

最后一个坑。既然调优 aux 把 CV 压到 0.24、noise 又能抬高探索,叠一起应该更好?stage06 实测打脸:

```
aux+noise 交互（看硬利用率 CV）：调优aux 单用 CV=0.24，叠加 noise 后 CV=1.85。
  => 反作用：noise 的训练期抖动干扰了 aux 已达成的均衡，叠加反而更不平。两味不可机械相加。
```

调优 aux 单用 CV=0.242、0 死专家;叠上 noise 后 CV 反弹到 1.847、又死了 1 个专家。`σ=8` 的噪声足够大,把 aux 已经摆平的利用率重新搅乱了。**两味药作用在不同维度(aux 管硬均衡、noise 管软探索),机械叠加会互相干扰**——剂量和组合得调,不是「好药越多越好」。

把三味药并排看,本章的取舍图就完整了:

| 药 | 作用维度 | 单独效果(vs baseline) | 定位 |
|---|---|---|---|
| 弱 aux(0.01) | 软+硬均衡 | 熵 +0.002、复活 0、CV 2.65→2.65 | 剂量不足,无效 |
| 调优 aux(1.0) | 软+硬均衡 | 熵 +1.67、复活 7、CV 2.65→0.24 | **主药** |
| noise(σ=8) | 软探索 | 熵 +1.22、复活 0、CV 2.65→2.65 | 治标,正交于均衡 |
| 容量(第 05) | 善后 | (本章未单跑) | 辅药,保住复活通道 |

容量为什么是辅药:只加容量不加 aux,等于给没刹车的车换更大油箱,坍塌照样发生,只是热门专家能吃更多。**缺主药(标定好的 aux),另两味都白搭。**

## ⚡ 前沿:大规模下坍塌还和什么纠缠,目前无通用解

上面的消融在 toy 上能把曲线扳回来,但**坍塌在大规模训练里仍是没有通用解的开放问题**——它和几个因素深度纠缠,每个都还在研究中:

- **Router z-loss(ST-MoE,2022)。** 大规模训练里 gate logits 容易变得很大,softmax 进入饱和区,数值不稳还放大坍塌。z-loss 惩罚 logits 的 log-sum-exp(本质压住 logits 的绝对幅度),和 aux loss 是两套正交的力:aux loss 管「分布平不平」,z-loss 管「logits 大不大、稳不稳」。本章 toy 规模碰不到这个数值墙,所以没演示,但真上规模必须叠上。
- **专家初始化与对称破缺。** 初始化分布的方差直接决定早期相变前你有多少缓冲。配套仓 `prng.ts` 给了确定性的初始化基准,但「什么初始化最抗坍塌」在大规模上没有定论。
- **Warmup。** 学习率 warmup 不只是优化技巧——它在早期(最危险的相变窗口)压低 router 的更新步长,等专家分化出来一点再放开。本章 baseline 的 `lr=0.05` 全程不变正是为了**故意**不给这个缓冲。warmup 长度、aux loss 权重、容量因子三者怎么联合调,目前主要靠经验和扫参,**没有从第一性原理推出的配方。**

⚡ 把这些串起来的工程共识是:**前沿训练把「路由日志 + 专家激活直方图」当成一等观测面**,像盯 loss 一样盯它们,甚至更早。本章那块 ASCII 监控面板,就是这套观测面在 toy 规模上的最小版——`metrics.ts` 的三个度量(`routingEntropy` / `expertUtilization` / `coefficientOfVariation`)+ `viz.ts` 的 `plotLoss` / `bar` / `heatmap`,放大到千卡集群就是 TensorBoard 上那几张永远开着的路由健康图。**坍塌没有「设好就忘」的解,只有「持续观测 + 多味药按剂量协同」的工程纪律。**
