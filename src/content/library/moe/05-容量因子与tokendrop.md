先把第 04 章没说完的话说完。上一章我们用 Switch/GShard 风格的辅助损失把专家利用率(每个专家分到的 token 比例)压平了——`expertUtilization` 从一头沉变成接近均匀的 1/E。但那是个**软**手段:它往损失里加一项,让优化器**倾向于**均衡,可它管不住任何一个具体 batch 里、任何一个具体专家**实际收到了多少 token**。辅助损失收敛只是说"长期平均下来均匀",它不阻止某个 batch 里第 3 号专家临时收到比平均多一截的 token。

而本章要装的就是这个临时溢出。因为在真实 MoE 训练里,专家不是一个能随收随算的函数——它是一块**预先分配好、固定大小的内存 buffer**。token 来多了,buffer 满了,后来的没地方放,就被**丢掉**(drop)。本章的核心论点一句话:**token drop 不是路由算法的失误,是工程把『专家容量』定死成常数后,数学上必然漏出来的那部分**。理解了这句,你才会明白 capacity_factor 这个旋钮为什么存在,以及它两头各通向什么深渊。

> 诚实声明:本章对应配套仓的 `examples/moe-from-scratch/src/stage05-capacity-drop.ts`,所有引用的 drop 率、accuracy、激活 MAC **都是 `npm run stage05` 的真实输出**,不是估算。但玩具数据是 6 类近可分高斯簇(cluster==class==理想专家),绝对 accuracy 偏乐观;**可迁移的是趋势和形状**——drop 率随容量单调降、aux loss 显著降 drop、紧容量+不均衡=系统性丢信息——不是绝对精度。这与配套仓 `core/metrics.ts` 顶部的诚实约定一致。

## 为什么 buffer 必须定长:从 all-to-all 说起

如果 MoE 层跑在单卡上,根本不需要"容量"这个概念。你拿到一个 batch 的 token,算出每个 token 的 top-k 专家,然后**按需**喂给对应专家——收 5 个就算 5 个,收 500 个就算 500 个,变长循环而已,没有任何东西需要丢。

那为什么真实系统要引入容量、要丢 token?**因为专家不在同一张卡上。**

✦ **这正是本章的专家钩子要钉的根:容量因子是『内存/算力 vs 信息丢失』的旋钮,而 token drop 的真因不是算法想丢,是工程要定长。** 在分布式 MoE 里,E 个专家分散在不同设备(GPU/TPU)上。token 路由完之后,必须通过一次 **all-to-all**(集合通信原语,每张卡把"该发给别人的 token"打包发出、把"别人发给我的"收进来)。all-to-all 要求**每张卡发往每张卡的数据块大小在编译/启动时就确定**:你不能在运行时才告诉 NCCL"这次第 0 卡要给第 3 卡发 137 个 token,下个 batch 发 89 个"。变长的通信缓冲在静态图、kernel 预编译、内存预分配的世界里是灾难。

于是工程上的解法就一个字:**定长**。给每个专家一个固定大小的桶。这个容量怎么算,配套仓 stage05 里就是一行:

```ts
// examples/moe-from-scratch/src/stage05-capacity-drop.ts
// Capacity per expert. ceil so factor=1.0 on perfectly balanced load drops nothing.
const capacity = Math.max(1, Math.ceil((factor * T * K) / E));
```

`容量 = capacity_factor × tokens × k / E`。式子的来历很直白:`T × K`(代码里 `T` 是 token 数、`K`=2)是这个 batch 所有 token 发出的"专家请求"总数(每个 token 发 k 个),除以 E 是"如果完美均匀,每个专家该收的平均数",再乘 `capacity_factor`(下简称 cf)留一点冗余裕量。token 路由到一个已经装满的专家时,它**没地方坐**,被丢。

那个 `ceil` 是刻意的:它保证 cf=1.0 时,在**绝对完美均匀**的负载下一个不丢;只要分布有任何抖动,某些专家就会超额。所以 drop 的根因不是"算法判断这个 token 不重要",而是"这个 token 选中的专家这一桶满了,而桶是为了通信定长才存在的"。把因果搞反——以为是某种重要性筛选——你就会去优化错误的东西。

## 容量、溢出、丢弃:把机制写进代码

stage05 在第 02 章 top-k 路由、第 04 章辅助损失的基础上,加三件事:算容量、按桶接收、溢出丢弃。先看怎么**填桶**——这里有个关键设计选择:

```ts
// examples/moe-from-scratch/src/stage05-capacity-drop.ts
// 每个专家把申请者按"它给该专家的 gate 概率"排序,留前 capacity 个,其余丢。
const accepted: Set<number>[] = applicants.map((list) => {
  list.sort((a, b) => b.prob - a.prob);
  return new Set(list.slice(0, capacity).map((a) => a.token));
});
```

注释里写明了为什么按 gate 置信度排序而不是按到达顺序:**丢掉最不自信的那批 dispatch,是标准的、伤害最小的策略**(Switch Transformer 风格)。如果你改成随机丢或按序列位置丢,会引入位置偏置——序列尾部的 token 永远先死。谁先有座、谁被丢,从来不是无所谓的细节。

第二个值得讲透的设计藏在"被丢的 token 怎么继续往下走"。一个被丢的 token 不能凭空消失,它在 Transformer 里还要进下一层。stage05 的做法是让它**走残差直通**(residual passthrough):

```ts
// examples/moe-from-scratch/src/stage05-capacity-drop.ts
// 全丢的 token 只剩残差路径——字面意义的"专家没碰我,skip connection 把我抬过去了"。
if (combined === null) {
  logits = res; // residual-only fallback
} else {
  logits = combined.mul(Value.scalar(1 / weightSum)).add(res);
}
```

注意配套仓在这里下了一个值得拎出来讲的赌注:**残差路径是被冻结的、不训练的**。代码里 `MoELayer.parameters()` 故意把 `residual` 排除在外。作者的注释解释得很到位——如果让残差也训练,在近可分的玩具数据上它会**偷偷长成一个完整的分类器**,那样"丢一个 token 几乎不损失什么",本章要演示的失败模式就被掩盖了。冻结残差,才让"专家干了真活,丢掉它们要付准确率代价"这件事在数字上真实可测。这是反 survey 的一个缩影:survey 会写"丢的 token 走残差",但不会告诉你残差该不该训、训了会怎样掩盖问题。

> 一个早期常见 bug:把丢的 token 直接**置零**(而不是走残差),那等于在序列里凿了个洞,梯度和表示都被打断,loss 会莫名其妙地差。丢 token 必须配残差,这是第一条工程纪律。

## 旋钮的两头:cf 调大,少丢 token,多花算力

现在来量这个权衡——这是本章最该被记住的曲线形状。`npm run stage05` 在均衡路由器(带 aux loss,负载 CV=0.494)上扫 cf,T=240 个 eval token、k=2、E=6,实测输出如下:

| cf | 每专家容量 | drop 率(派发口径) | 全丢 token | accuracy | 激活 MAC |
|----|-----------|------------------|-----------|----------|---------|
| 0.5 | 40 | 53.8% | 13.3% | 90.4% | 97,632 |
| 1.0 | 80 | 17.3% | 0.0% | 100.0% | 156,432 |
| 1.5 | 120 | 0.8% | 0.0% | 100.0% | 182,976 |
| 2.0 | 160 | 0.0% | 0.0% | 100.0% | 184,320 |

(容量列由 `ceil(cf × 240 × 2 / 6)` 算出;其余三列是真实运行计数。)

读这张表有两个要点。

**容量这一侧:** cf=1.0 时每专家容量恰好 80,而完美均匀时每专家该收 `240×2/6 = 80` 个——总桶位恰好等于总请求。可即便路由器已经被 aux loss 压到 CV=0.494,cf=1.0 仍丢了 17.3% 的派发。这就是为什么实践中几乎没人用 cf=1.0:总桶位刚好够用 = 任何抖动都溢出。Switch Transformer 默认用 cf=1.25,GShard 常用到 2.0。cf 提供的就是吸收"分布抖动"的缓冲垫。注意 drop 率 53.8% → 17.3% → 0.8% → 0.0% **单调下降**(stage05 把这条单调性写成了硬断言,回归会当场报错)。

**这里有个直接的因果链回指第 04 章。** 下一节的实测会量化它:aux loss 把利用率压平 → 各专家收到的 token 更接近平均 80 → 同样 cf 下溢出更少 → drop 率下降。第 04 章的软均衡和本章的硬容量是**配合**关系:aux loss 让你能用更小的 cf(更省算力)达到同样低的 drop。坍塌时则反过来——那是第 06 章的剧情。

**算力这一侧,代价是真金白银。** 看激活 MAC 那一列:cf 0.5→2.0,激活 MAC 从 97,632 涨到 184,320,**涨了 89%**。这违反很多人的直觉——"容量是上限,调大它怎么反而更费算力?"。根子在 stage05 怎么算激活 FLOPs:

```ts
// examples/moe-from-scratch/src/stage05-capacity-drop.ts
// 真实激活 FLOPs:每 token 的 gate + 真正"开火"的专家运行 + 每 token 的残差投影。
// 这就是为什么容量越大算力越多:丢得越少 => 更多专家真的执行了。
const activatedMac = gateMac + firedExpertRuns * oneExpertMac + residualMac;
```

关键在 `firedExpertRuns`——**只有没被丢、真正跑了的 dispatch 才计入算力**。cf 小 → 丢得多 → 真开火的专家次数少 → 激活 MAC 低。所以从算力账本看,**容量是省算力的反向旋钮:丢 token 其实是在省 FLOPs**(代价是丢信息)。cf=1.5(MAC 182,976)到 cf=2.0(184,320)只涨了 0.7%,而 drop 早在 cf=1.5 就降到 0.8%——继续加 cf,就是纯纯为空桶位烧算力和显存,边际收益归零。

所以这个旋钮的调法是:**把 cf 调到 drop 率落到可接受阈值(经验上 < 1%)的最小值**,再大就是浪费。它不是"越大越好",是"够用就停"。stage05 里 cf=1.5 已经把 drop 压到 0.8% 且 accuracy 满格,就是这个"够用点"。

## 实测:aux loss 在固定容量下直接降 drop

第 04 章那条因果链不能光嘴说,得拿数字钉死。stage05 在 **cf=1.0 固定**下,跑了 aux loss 开/关两组对照(两组用不同种子训练,只差 aux 这一项):

| 配置 | 负载 CV | drop 率 | accuracy |
|------|---------|--------|----------|
| 无 aux loss | 0.717 | 22.3% | 99.2% |
| 有 aux loss | 0.494 | 17.3% | 100.0% |

aux loss 把负载 CV 从 0.717 压到 0.494,drop 率相应从 22.3% 降到 17.3%(降了 5 个百分点)。机制很干净:**均衡 → 没有专家被挤爆 → 少 drop**。这就是第 04 章的软均衡如何"间接"帮本章的硬容量省钱——它不直接管容量,但它把负载摊平,让同样大小的桶溢出得更少。

> 注意一个细节:无 aux 那组 accuracy(99.2%)只比有 aux(100.0%)差一点点,但 drop 率差了 5 个百分点。在这个近可分玩具上,丢掉的多半是冗余 token,准确率没怎么掉。**别被这个乐观数字骗了**——下一节把容量收紧到 0.5、再叠加不均衡,accuracy 就会崩,而且崩得很不均匀。

## 一个隐蔽的失败模式:紧容量 + 不均衡 = 系统性丢信息

讲一个最该栽心里的坑,因为它把"容量是静态约束"的后果暴露得最彻底。前面 cf=0.5 那行 accuracy 还有 90.4%,看着不算太糟——但那是在 aux loss 把分布压平的前提下。现在把两个坏因素**同时打开**:cf=0.5(容量紧到只有 40)**且**无 aux loss(路由不均衡)。stage05 实测:

- **失败配置(无 aux + cf=0.5):** drop 率 51.5%,全丢 token 17.1%(这些 token 只剩残差),整体 accuracy 跌到 **80.4%**。
- **对照安全配置(有 aux + cf=2.0):** drop 0.0%,accuracy 100.0%。

整体 accuracy 从 80.4% 看,像是"均匀地差了一截"。但这正是最骗人的地方。按类别拆开看,真相完全不同:

| 类别 | 失败(无 aux, cf=0.5) | 安全(aux, cf=2.0) |
|------|---------------------|-------------------|
| c0 | 100.0% | 100.0% |
| c1 | 81.6% | 100.0% |
| c2 | 100.0% | 100.0% |
| c3 | 95.5% | 100.0% |
| c4 | 64.1% | 100.0% |
| c5 | 50.0% | 100.0% |

失败配置的类别准确率 CV = 0.232(高 = 不均匀),最差类别 c5 只有 50.0%;安全配置 CV = 0.000(全部满格)。**信息丢失是系统性的,不是均匀噪声。** c0、c2 完好无损,c5 直接腰斩。原因:不均衡的路由器把某几个专家挤爆,这些专家服务的类别(c4/c5)的 token 被成批丢、只能走弱残差,这些类别的准确率崩塌;而被路由器偏爱的类别(c0/c2)的专家没满,看起来毫发无伤。

stage05 还把失败路由器的 expert×class 共现热图打了出来,一眼能看到病灶:

```
    c0c1c2c3c4c5
 E0
 E1   ██
 E2         ▓▓▓▓
 E3 ░░    ▓▓
 E4 ░░
 E5     ▒▒
```

E2 这一行横扫 c3/c4/c5(过热,抢光),而 E0 整行是空的(饿死,死参数)。一个专家既要服务 c3 又要服务 c4、c5,容量只有 40,必然爆——爆出来的就是 c4/c5 那些被丢成腰斩的 token。这张图把"为什么是这几个类别崩"从猜测变成了可视的证据。

这个失败模式在生产里最阴险的变体是 **train/eval 分布漂移**:训练时 batch 大、token 分布被采样打散,某专家溢出概率被平均掉;推理时可能是单条短序列、或领域高度集中的输入(整段都在讲代码),token 高度扎堆到少数"代码专家",这些专家瞬间爆容量,drop 率飙升——而你的容量是按训练时的均匀假设定的。**模型在验证集上好好的,上线遇到分布集中的真实输入就掉链子**,根因常常就是这个:容量是静态预算,真实负载却是动态、有结构、会扎堆的。

## ⚡ 前沿:能不能从根上不丢 token?

token drop 是"专家容量定长"逼出来的,那把"定长"这个前提拆掉,drop 就不存在了。目前有两条都已落地、各有代价的路线,攻击的是同一个根因的两端——**这块仍是活跃的工程/研究权衡区,没有一个方案在所有场景上通吃**。

⚡ **路线一:expert-choice routing(专家选 token,而非 token 选专家)。** 翻转选择方向:不再是每个 token 挑 k 个专家(导致专家收到的数量不可控、要靠容量截断),而是**每个专家恰好挑固定数量个最想要它的 token**。这样每个专家的负载**精确等于那个固定数**——天然定长、天然均衡,drop 和不均衡**同时**从定义上消失了。代价也实在:一个 token 可能被 0 个专家选中(没人要它,等于被"软丢弃",只是换了形式),也可能被很多专家抢(算力超预算);而且"专家选 token"需要看到全 batch 才能排序,**破坏了自回归解码的因果性**,在 decoder 推理时不能直接用。它把"丢哪个 token"转成了"哪个 token 没被任何专家选中"——更可控,但没免费。

⚡ **路线二:dropless MoE,代表是 MegaBlocks 的块稀疏 matmul。** 它的态度是:既然 drop 来自"必须定长才能高效算 matmul",那就**造一种能高效算变长的 matmul**。MegaBlocks 把"E 个专家各处理不定数量 token"重新表达成一个**块稀疏(block-sparse)矩阵乘法**——用块稀疏 kernel 直接吃变长的、每专家数量不等的 token,既不 padding 也不截断,容量上限**整个取消**,一个 token 都不丢。代价是它需要专门的块稀疏 GPU kernel(不是现成 dense matmul 能干的),实现和移植成本高,收益依赖你的硬件/kernel 优化到位与否。它本质是用"工程把变长算高效了"来消解"为了高效才定长"的原始妥协。

把三者并排,本章的取舍图就完整了:

| 方案 | drop | 负载均衡 | 代价 / 适用边界 |
|------|------|---------|----------------|
| 有容量(本章) | 会丢(cf 调溢出量) | 软靠 aux loss(第 04 章) | 简单、all-to-all 友好;异构负载下要在 drop 与算力间手调 cf |
| expert-choice | 无显式 drop(但 token 可能没人选) | 定义上精确均衡 | 破坏因果、decoder 推理难用 |
| dropless / MegaBlocks | 彻底无 drop | 不靠截断,靠变长计算 | 需块稀疏 kernel,工程门槛高 |

没有哪一行是免费午餐:容量法把代价放在"丢信息 + 调旋钮",expert-choice 放在"因果性",dropless 放在"kernel 工程"。**选哪个取决于你的瓶颈在哪**——训练吞吐受限就值得上 dropless 的 kernel 投入;encoder-only 或非自回归场景 expert-choice 很香;要简单稳妥、跟现有 all-to-all 栈对齐,容量法至今仍是最常见的默认。

## 小结与往下读

本章把第 04 章的软均衡和一个硬约束接上了:辅助损失让分布更平,但每个专家在一个 batch 里仍只有 `capacity_factor × tokens × k / E` 个定长座位,因为分布式 all-to-all 通信要求 buffer 静态定长——这才是 token drop 的真根因,不是算法想丢。capacity_factor 是"信息丢失 vs 算力/内存"的旋钮:实测 cf 0.5→2.0,drop 率从 53.8% 降到 0,但激活 MAC 涨了 89%(丢 token 其实在省 FLOPs);aux loss 在固定 cf=1.0 下把 drop 从 22.3% 降到 17.3%,印证第 04 章软均衡间接降 drop;紧容量+不均衡(cf=0.5 无 aux)则把 accuracy 砸到 80.4%,且 c5 类腰斩到 50.0%——信息丢失是系统性的。前沿的 expert-choice 和 dropless MoE 各从一端拆掉"定长"前提来消除 drop,但分别把代价转嫁到因果性和 kernel 工程上,无通用最优解。

接着读**第 06 章:专家坍塌**。本章假设了分布"有抖动但大体均衡";一旦路由退化成几乎只选少数几个专家(熵塌缩、CV 飙升),那少数专家会**瞬间爆容量**,海量 token 被丢、走残差直通,模型实质退化成稀疏到病态的网络——损失曲线随之崩。你会看到本章的 drop 机制在坍塌下从"偶尔丢几个"变成"丢掉大半",以及为什么这是 MoE 训练最经典的翻车现场。
