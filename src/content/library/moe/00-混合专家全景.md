先把一个被反复说错的事实摆正:**稀疏激活不是剪枝,也不是蒸馏。** 剪枝是训练完之后永久删掉一批权重,模型从此变小;蒸馏是用大模型当老师训一个结构不同的小模型。这两件事都在"减少模型有多少参数"上做文章。混合专家(Mixture of Experts,简称 MoE——一层里放多个并列的子网络,每个 token 只走其中几个)干的是完全不同的事:**参数一个都不删,只是在同一次前向里,根据当前 token 的内容,只点亮其中一小部分。** 下一个 token 来了,可能点亮另一部分。被点暗的专家,权重还在显存里,只是这一步没参与计算。

为什么这件事值得单独写一本书?因为它捅破了一个被 dense(稠密,所有参数对每个输入都参与计算)架构默认到没人质疑的假设:**模型有多少参数,和每处理一个 token 要算多少次乘法,是同一个旋钮。** 你想让模型更聪明(更多参数),就必须接受它更慢更贵(更多算力)——这两件事在 dense Transformer 里是焊死的。MoE 的全部价值,就是把这个焊点切开,让你能单独拧"参数容量"这个旋钮,而几乎不动"每 token 计算预算"那个旋钮。

这本书的做法是:用纯 TypeScript 从零手写一个 MoE,**纯算法、离线、零运行时依赖**,每章对应配套仓 `examples/moe-from-scratch/` 里一个能 `npm run stageXX` 直接跑的 stage。本章不绑某个 stage——它是地图本身。先讲清楚这本书测什么、不测什么,再带你走一遍从第 01 章到第 07 章每一站要解决的具体难题。

## 焊点在哪:dense FFN 把容量和算力绑死了

要理解 MoE 切的是哪个焊点,得先看清 dense 是怎么焊的。假设你已经知道 Transformer 里每个 block 有一个前馈网络(FFN,Feed-Forward Network——两层全连接,中间一个宽隐层)。这块 FFN 是 Transformer 里参数的大头(通常占一半以上),也是算力的大头。

它的算力账非常直白:一个 FFN 把 `inDim` 维输入升到 `hidden` 维、再降回来,处理一个 token 的乘加(MAC,Multiply-ACcumulate——一次乘法加一次累加,深度学习里数算力的基本单位)就是 `inDim × hidden + hidden × inDim`。配套仓把这笔账钉成了代码,`examples/moe-from-scratch/src/core/metrics.ts` 里的 `denseFLOPs`:

```typescript
// dense FLOPs (MACs): the SAME total parameters as the MoE but every token runs through
// ALL E experts (equivalently one big FFN of width E*hidden). This is the baseline the
// MoE claims to beat: the ratio activated/dense ~ k/E is the sparsity win that transfers.
export function denseFLOPs(tokens: number, inDim: number, hidden: number, outDim: number, E: number): number {
  const allExperts = E * (inDim * hidden + hidden * outDim);
  return tokens * allExperts;
}
```

注意这里的 `E *`(专家数)。把它读成大白话:dense 等价于"一个宽度是 `E × hidden` 的巨型 FFN",每个 token 都得从头到尾走完整个巨型 FFN。**你想让 FFN 容量翻 8 倍(`E=8`),它的每 token 算力也精确翻 8 倍。** 这就是焊点:`denseFLOPs` 里参数量和算力共用同一个 `E`,改一个就改了另一个,物理上无法分开。

这不是 dense 设计得差,是它根本没打算分开。dense FFN 的前提就是"所有参数对所有输入都有用",既然都有用,自然都要算。问题出在我们把模型做到几百 B、上 T 参数之后:大量参数其实只对某一类输入有用(处理代码的子网络对处理诗歌的 token 是浪费),但 dense 架构没有"让某些参数对某些 token 闲置"的机制,只能让所有 token 付全额算力。**焊点的代价,是在你想把模型做大的那一刻才暴露的——而那正是现在。**

## 切焊点:条件计算让两个旋钮独立

MoE 的切法是引入"条件计算"(conditional computation——根据输入内容,决定哪部分网络参与)。具体到 FFN:不再放一个宽度 `E × hidden` 的巨型 FFN,而是放 `E` 个各自宽度 `hidden` 的小 FFN(每个叫一个**专家**),外加一个小小的路由器(router/gate——一个线性层,看一眼 token 决定派给谁)。每个 token 来了,路由器只挑 `k` 个专家(通常 `k=1` 或 `2`),token 只走这 `k` 个,其余 `E−k` 个这一步完全不算。

配套仓里专家就是一个普通两层 MLP,`examples/moe-from-scratch/src/core/nn.ts` 里的 `Expert`:

```typescript
export class Expert implements Module {
  fc1: Linear;
  fc2: Linear;
  // ...
  forward(x: Value): Value {
    const h = this.fc1.forward(x);
    const a = this._activate(h);
    return this.fc2.forward(a);
  }
}
```

没有任何花活——专家就是一块标准 FFN。MoE 的魔法不在专家里,在"只点亮 `k` 个"这个动作上。这个动作的算力账,是 `denseFLOPs` 旁边的 `activatedFLOPs`:

```typescript
// activated FLOPs (MACs) for a top-k MoE layer: each token runs through k experts, each
// expert being two matmuls (in×hidden and hidden×out), plus the gate (in×E).
export function activatedFLOPs(
  tokens: number, inDim: number, hidden: number, outDim: number, E: number, k: number,
): number {
  const gate = inDim * E; // gate logits per token
  const oneExpert = inDim * hidden + hidden * outDim;
  return tokens * (gate + k * oneExpert);
}
```

关键差别就一个字:dense 是 `E × oneExpert`,MoE 是 `k × oneExpert`。把这两个函数的比值算出来,就是 MoE 这本书的中心论点:

> 激活算力 / dense 算力 ≈ `k / E`(那个小小的 gate 项 `inDim × E` 相对专家矩阵可忽略)

拿一组具体数字感受一下(下面是按上面两个公式手算的**示意算术**,标 `(est.)`,不是某个 stage 的实测输出——本章无独立 stage)。设 `inDim = hidden = outDim = 256`,`E = 8`,`k = 2`,处理 1 个 token:

- 一个专家 `oneExpert = 256×256 + 256×256 = 131,072` MAC
- dense:`8 × 131,072 ≈ 1.05M` MAC `(est.)`
- MoE 激活:`gate(256×8=2,048) + 2 × 131,072 ≈ 264K` MAC `(est.)`
- 比值 ≈ `264K / 1.05M ≈ 0.25`,正好 `k/E = 2/8`

**两个旋钮分开了:** 想让模型更聪明,你把 `E` 从 8 拧到 64,总参数(所有专家权重之和)涨 8 倍;但只要 `k` 不变(还是 2),每 token 激活算力几乎纹丝不动,只多了 gate 那一点点。这就是开头那句"参数涨 8 倍而激活算力几乎不变"的来源。dense 做不到这件事,因为它的 `E` 同时管着两个量。

⚡ 这里得诚实标一个本书绕不开的边界:**`k/E` 算的是算法工作量(乘加次数),不是吞吐(tok/s)。** 配套仓 `metrics.ts` 的注释把这点讲得很白——FLOP 是真按 matmul 维度数出来的 MAC,量化的是"稀疏 vs dense 的算力比",这个比例可迁移;但 wall-clock(真实墙上时间)受硬件、kernel、专家分布到多少张卡上影响极大。真实 MoE 训练里,把 token 按路由结果分发到不同专家(还可能跨设备)的通信开销,经常吃掉一部分理论节省。**"省下的算力比"和"省下的时间比"不是一回事,这是 MoE 工程里最容易被 PR 稿糊弄过去的地方。** 本章往后每次给绝对数字,都会区分"机制/比例可迁移"和"绝对值偏乐观(toy)"。

## 焊点切开之后,六个新麻烦冒出来——这就是本书路线

切焊点不是免费的。dense 的好处恰恰是"没有选择"——所有 token 走同一条路,行为可预测、梯度顺畅、负载天然均匀。MoE 一旦引入"选择",就同时引入了一堆 dense 从来不存在的失败模式。本书的章节顺序,就是这些麻烦冒出来的顺序。这也是这本书区别于一篇"MoE 综述"的地方:综述告诉你有哪些机制,这本书告诉你**每个机制是为了堵哪个具体的窟窿,以及它自己又会怎么坏。**

### 第 01 章:MoE 层结构——把 dense FFN 换成"门 + 专家"

第一站就是把上面这套话变成能跑的代码:一个最小 MoE 层,gate 算出每个专家的权重,专家各自前向,加权求和。这章并排跑稀疏版和 dense 版,用 `activatedFLOPs` / `denseFLOPs` 实测那个 `k/E` 比例,让你亲眼看到"同样的参数量,激活算力少了一截"。**这里第一个失败模式登场:** 如果 gate 给所有专家几乎相同的权重,MoE 就退化成一个被平均过的 dense FFN——花了路由的钱,没买到稀疏的好处。

### 第 02 章:top-k 路由——每个 token 只激活 k 个专家

第 01 章的 gate 还是"软"的(每个专家都给点权重)。要真省算力,必须"硬"选:只留权重最高的 `k` 个,其余直接不算。这章实现 top-k 选择,并讲清一个反直觉的取舍:`k=1`(Switch Transformer 的选择)最省算力但路由一错就全错;`k=2`(GShard、Mixtral 的选择)给了 token 一个"备胎专家",更稳但算力翻倍。**为什么大多数前沿模型选 `k=2` 而不是 `k=1`,这章用路由稳定性说清。**

### 第 03 章:不可微难题——argmax 挡住了梯度

这是全书技术上最硬的一章,也是 MoE 真正反直觉的地方。top-k 里那个"选最高的几个"动作是 `argmax`(取最大值的下标),它**不可微**——下标是个整数,你没法对"选了 3 号专家"这件事求导,梯度到这里就断了,路由器学不到东西。dense 没这个问题,因为 dense 没有"选择"。这章讲清主流解法:梯度不走那个硬选择,而是走 gate 输出的连续权重(softmax 概率)绕回路由器。**机制清楚之后还要讲它怎么坏:** 梯度走权重这条路,意味着路由器只在"被选中的专家"上拿到学习信号,没被选中的专家长期收不到梯度,这就埋下了第 06 章坍塌的种子。

### 第 04 章:负载均衡——不然所有 token 挤向同几个专家

路由器是学出来的,而学习有个恶性正反馈:某个专家early 稍微好一点 → 被路由更多 token → 拿到更多梯度 → 变得更好 → 被路由更多……几步之内,少数专家吃掉绝大多数 token,其余专家饿死(永远收不到 token,永远学不会)。这章实现 Switch/GShard 风格的辅助损失(auxiliary loss——一个额外的惩罚项,逼路由把 token 摊平)。配套仓 `metrics.ts` 里的 `loadBalanceLoss` 已经把这个机制连同它最经典的实现 bug(那个硬计数项必须 detach、不能带梯度)都钉死在签名里了。**这章用 `expertUtilization`(每个专家分到的 token 比例)和 `coefficientOfVariation`(变异系数,衡量分布有多不均)实测 aux loss 上线前后利用率被压平的过程。**

### 第 05 章:容量丢弃——专家也会"满"

辅助损失是软约束,压不住极端情况。真实系统还要一道硬闸:给每个专家设一个容量上限(capacity——这一批最多接收多少 token),超过的 token 直接被丢弃(它的 FFN 输出当 0,只靠残差连接活着)。这是个赤裸裸的取舍:容量设小了省显存但丢 token 掉质量,设大了不丢但浪费算力和显存。**这章实测不同容量系数下的 drop rate(丢弃率),让你看到"省显存"和"丢信息"之间那条没有免费午餐的曲线。**

### 第 06 章:专家坍塌——路由退化成常数

这是 MoE 最著名的失败模式,也是本书把"怎么坏"讲透的核心案例。前面所有机制(可微的绕路、负载均衡、容量)都没拦住的时候,路由器会彻底躺平:不管输入是什么,都路由到同一个(或同几个)专家,其余专家变成死代码。这章用 `routingEntropy`(路由熵——路由器有多"犹豫",高熵=在均匀分散,熵塌向 0=退化成常数)的塌缩曲线和 `coefficientOfVariation` 的飙升,把坍塌的指纹画出来,再演示几种缓解手段为什么有效、什么时候仍然救不回来。

### 第 07 章:capstone——训出会自发语义分工的 toy MoE

最后一章把前六章的机制全装到一个能训练的 MoE 上,在 `examples/moe-from-scratch/src/core/data.ts` 提供的**带 ground-truth 分组**的合成数据上训练。这套数据的设计目的就是验证 MoE 的核心承诺——`data.ts` 注释里说得很清楚:"different experts SHOULD specialize on different sub-distributions"(不同专家应该各自专精不同子分布)。因为我们提前知道每个样本属于哪个 latent 分组,就能用 `viz.heatmap` 画出"专家 × 分组"的对照表,直接看路由有没有真的学到语义分工——而不是嘴上说说。**这是反 survey 的终极姿态:不是断言 MoE 能分工,是把分工长出来给你看。**

## 这本书测什么,不测什么

工程师读到这该警觉:toy 实现的数字能信吗?直说——**绝对值不能信,机制和曲线形状能信。** 配套仓 `README.md` 的"诚实数字约定"和每个 core 模块顶部的 `HONESTY:` 注释,都在反复强调同一件事:

- **可迁移的**:稀疏省算力的 `k/E` 比例、aux loss 压平利用率 CV 的过程、坍塌时路由熵塌缩的形状。这些是机制层面的真相,toy 和真实系统共享。
- **不可迁移的**:具体的收敛精度、绝对 tok/s、`drop rate` 的精确值。toy 数据线性可分、规模极小,这些绝对值偏乐观。

所有随机性走种子 PRNG(`examples/moe-from-scratch/src/core/prng.ts`,禁用 `Math.random`),所以每个 stage 同种子跑出来逐位一致,你能复现书里的每条曲线。这点和姊妹书《LLM 推理从零做透》同源——那本书测的是推理引擎的等价性,这本书测的是路由机制的行为,共用同一套"toy 但诚实、合成但可复现"的底座。

需要前置说清:**本书假设你已经懂 Transformer 和 FFN。** 我们不重讲 attention、不重讲反向传播,只把 Transformer block 里的那块 dense FFN 抠出来,换成稀疏 MoE。如果你对"FFN 在 Transformer 里长什么样、为什么是参数和算力大头"还没把握,先去看《LLM 推理从零做透》第 01 章的 TinyTransformer 前向,再回来。

## ⚡ 前沿:专家越来越多、越来越小、激活比越来越低

最后给一条 MoE 当前正在走、但远未走完的路线,也是本书 toy 实现对标的方向。

从 2020 年的 GShard(Google,第一个把 MoE 做到 600B 参数、证明稀疏能 scale 的工作)开始,到 2023 年的 Mixtral 8x7B(8 个专家、每 token 选 2 个,第一个真正好用的开源 MoE),再到 2024 年的 DeepSeek-V3(256 个细粒度专家 + 几个所有 token 都走的"共享专家",每 token 激活比低到个位数百分比)和 Qwen 系列的 MoE 变体,有一条清晰的趋势:**专家越来越多、单个越来越小、激活比(`k/E`)越来越低。** 直觉上这合理——专家越细,语义分工越精细,每个 token 越能只调动真正相关的那一小撮参数;但它把第 04 章的负载均衡和第 06 章的坍塌问题放大到了极致:256 个专家要全部学到东西、不饿死、不坍塌,比 8 个难得多,DeepSeek-V3 为此专门设计了不依赖辅助损失的均衡策略。

哪些是已解决的工程、哪些还在研究?**坦白说,后者更多。** 最优专家数与激活比没有理论公式,全靠经验和算力预算试;细粒度专家下负载均衡和坍塌的稳健解法仍在迭代(辅助损失会干扰主任务、无辅助损失方案又难调);专家该按什么粒度分工(按 token?按领域?按位置?)更是开放问题——没人能事先指定"3 号专家负责代码",分工是训练中自发涌现的,可控性很差。**这些都是 2024-2025 还在活跃研究、没有通用解的方向。** 本书的 toy 实现,正是这条"多专家 + 细粒度 + 低激活比"路线的最小可跑骨架:第 07 章那个会自发语义分工、也会坍塌的 toy MoE,和 DeepSeek-V3 面对的是同一类问题,只是规模差了六个数量级。把最小骨架的每个机制和失败模式吃透,你再去读那些大模型的技术报告,看到的就不是一串陌生的术语,而是同一套你亲手训坏过、又救回来过的旋钮。

下一章见——第 01 章,我们动手把第一块 dense FFN 拆成"门 + 专家"。
