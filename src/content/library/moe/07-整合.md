到这一章,你手里应该攥着所有零件了:第 01 章的稀疏激活骨架、第 02 章的 top-k 路由、第 03 章解决的「argmax 不可导怎么把梯度引回 router」、第 04 章的负载均衡辅助损失、第 05 章的容量丢弃、第 06 章诊断的专家坍塌。每个零件单独看都成立。但单独成立不等于拼起来成立——这正是 MoE 工程里最容易翻车的地方:aux loss(辅助损失,逼路由别偏心的一个附加目标)调大一点专家就被压成均匀噪声、退回 dense,调小一点又压不住坍塌;容量上限设低了被丢 token 比例飙升、训练发散,设高了又失去稀疏意义。

所以本章只做一件事,但做透:把零件拼成一个能从头训到收敛的 MoE,然后用它回答全书唯一真正重要的问题——**专家的「分工」是我们硬塞进去的,还是它自己长出来的?** 这个问题的答案决定了 MoE 是个工程 trick,还是一个有结构性意义的架构。剧透:是涌现的,而且我会给你一个比 loss、比任何标量指标都硬的证据来看它。

本章的全部数字来自 `examples/moe-from-scratch/src/stage07-moe-capstone.ts` 的真实运行(`seed=7`,确定性,重跑逐位一致)。这个文件不引入任何新魔法:它只 import `core/` 里前六章共用的张量引擎、层原语、度量和数据生成,然后把它们接成一个 MoE FFN 块加一个分类头。组装 MoE 不需要新代码,只需要把已有零件接对。

## 一个 MoE 块就是已有零件的拼装

`stage07-moe-capstone.ts:73` 的 `buildModel` 把整个 MoE 块摊开,你能数清它有几个零件:

```typescript
function buildModel(rng: Rng): MoeModel {
  const proj = new Linear(DIM, HIDDEN, rng);        // 路由前的共享主干
  const gate = new Linear(HIDDEN, E, rng);          // router: HIDDEN -> E 个 gate logit
  const experts: Expert[] = [];
  for (let e = 0; e < E; e++) experts.push(new Expert(HIDDEN, HIDDEN, HIDDEN, rng));
  const head = new Linear(HIDDEN, NUM_CLASSES, rng); // 合并后的专家输出 -> 类别 logit
  ...
}
```

四样东西:一个共享主干 `proj`(把输入投到 model 维度)、一个门控 `gate`(就是 `Linear(HIDDEN, E)` 加 softmax,`E` 是专家数,输出每个 token 走哪几个专家的打分)、`E` 个独立的 `Expert`(每个是两层 MLP,这就是第 01 章定义的「被路由的原子」)、一个分类头 `head`。超参在文件顶端 `stage07-moe-capstone.ts:50` 一处定义:`E=8` 个专家、`K=2`(top-2 路由,每个 token 只过 2 个专家)、`MOD=8` 个余数类、`NUM_CLASSES=4`、`CAPACITY_FACTOR=1.25`。`E` 故意等于 `MOD`,这样「完美路由」就是「一个专家对一个余数」,后面验证分工时矩阵该长什么样一目了然。

路由那一步——对 gate 概率取 top-k——是不可导的,第 03 章已经讲透:`argmax` 没有梯度,trick 是让梯度只走可导的 gate 概率那条路。`forwardToken`(`stage07-moe-capstone.ts:132`)把这件事和容量丢弃揉在了一起,值得逐行看:

```typescript
for (let r = 0; r < route.experts.length; r++) {
  const e = route.experts[r];
  if (useCapacity && counts[e] >= capacity) continue; // 这个专家本批已满 -> 丢弃
  counts[e]++;
  const w = probsRow.gather(e);                  // 可导的 gate 权重 -> 梯度回流 router
  const expertOut = model.experts[e].forward(trunk);
  const weighted = expertOut.mul(w);
  combined = combined === null ? weighted : combined.add(weighted);
}
if (combined === null) combined = trunk;          // 全满才回退主干, 保住一个带梯度的 logit
```

注意 `probsRow.gather(e)`:被选中专家的输出乘的是它的 gate 概率,梯度顺着这条乘法链回到 router,这就是第 03 章那个「绕开不可导 argmax」的具体落地。而 `counts[e] >= capacity` 的 `continue` 就是第 05 章的容量丢弃:每个专家每批最多收 `capacity` 个 token,满了的直接跳过(`capacity = ceil(1.25 × 2 × 32 / 8)`,见 `stage07-moe-capstone.ts:185`)。

为什么要丢而不是让它溢出?因为容量正是真实 MoE 里**约束激活 FLOPs 的那道闸**——没有它,一个坍塌的 router 会把所有 token 全送给一个专家,稀疏的算力保证瞬间蒸发。丢弃带来的「损失一个专家的 slice」的压力,配合 aux loss,才是把 token 摊到各专家的力。这一章会用真实数字证明:把这道闸和 aux 一起关掉,系统会怎么坏。

## 组装后第一个失败模式:零件都对,拼起来照样坍塌

把零件接到 `core/data.ts` 的 `makeModularTask` 上训。这个数据集是专门为「验证分工」造的——它的子结构是**已知的**:每个样本带一个 `clusterId`,存的就是这个 token 的余数类(residue),也就是「它天然该归哪个专家管」的 ground truth。关键在于 `NUM_CLASSES=4 < MOD=8`,多个余数共享同一个标签,所以「一个专家背一个余数」并不足以解题——逼专家学的是真的子结构,不是查表。这个 `clusterId` 从不进训练 loss,只在事后画图对照用。

**但你不能指望接起来就自发分工。** 第 06 章诊断过坍塌的指纹:路由退化成常数、熵塌缩、专家成片死亡。本章用一个 A/B 把它钉死——同样的数据、同样的初始化(`trainMoe` 里 `rng(SEED)` 让两次跑从**完全相同**的权重出发,`stage07-moe-capstone.ts:182`)、同样的学习率,唯一的区别是 `useProtection` 这一个开关同时控制 aux loss 和容量。

先看健康版(aux + capacity 全开)的真实输出:

```
专家利用率 (top-1 占比):
  E0 │······························│ 0.0039
  E1 │███████████···················│ 0.0938
  E2 │████████████████████████████··│ 0.2266
  ...
  E5 │██████████████████████████████│ 0.2461
  ...
死专家 (从不被选中) = 0/8 | CV = 0.615 | mean routing entropy = 1.875 nats (max=ln8=2.079)
task loss 1.313 -> 0.007
训练期容量丢弃 expert-slot 总数 = 2438
```

8 个专家全活着(死专家 0/8),利用率有高有低但没有谁归零,routing entropy 1.875 nats 接近理论上限 ln8=2.079,task loss 从 1.313 干净降到 0.007。注意那个 2438:这是训练期间因为容量满而被丢弃的 expert-slot 总数——它不是 bug,它是**压力的读数**。正是这股「想挤进满专家会被丢」的压力,配合 aux loss,把 token 逼着摊向冷专家。

CV(变异系数,标准差除以均值,0=完美均衡)是 0.615——不是 0。**这是一个值得停下来的点**:健康的 MoE 不该追求 CV=0。完美均衡意味着每个专家收等量 token,但 `makeModularTask` 的各余数样本数本就不必等量,而且强行压到 CV=0 等于强行让专家不分化、退回 dense。健康的目标是「没有死专家 + 熵不崩」,而不是「利用率拍平」。第 04 章讲 aux loss 怎么压平,第 06 章讲坍塌怎么来——本章的整合视角是:**这俩是同一枚硬币**。你调 aux 系数,本质是在「允许专精(低权重)」和「防止退化坍塌(高权重)」之间找平衡点,没有万能系数,得对着 CV 和熵这两条曲线调。

反例版(`useProtection=false`,aux 和容量全关)的真实输出,对照着看:

```
死专家 (从不被选中) = 5/8 | CV = 1.431 | mean routing entropy = 0.904 nats (max=ln8=2.079)
专家利用率: E1=0.5117, E2=0.2422, E7=0.2461, 其余 5 个全 0.0000
```

8 个专家死了 5 个。剩下 3 个里 E1 一家独吞 51%。熵从 1.875 崩到 0.904,CV 从 0.615 飙到 1.431。这就是组装后第一个必须盯的失败模式:**不开保护,router 会发现「把 token 反复送给同一批专家」是个局部最优**——那几个专家被反复训练越来越强,反过来让 router 更想送它们,正反馈直到坍塌。aux loss 就是来打断这个正反馈的,它给「偏心」一个持续的梯度惩罚。

## ✦ 判断 MoE 学没学到分工:别看 loss,看共现矩阵接不接近置换矩阵

现在到本章、也是本书的结论性证据。一个新手会这么验证「MoE 训好了」:看 loss 降了、accuracy 上来了。**这是错的方法。** 看上面两组数字里有一行我故意还没提:

```
各残基 accuracy: 100.0% 100.0% 100.0% 100.0% 100.0% 100.0% 100.0% 100.0%   <- 健康版
各残基 accuracy: 100.0% 100.0% 100.0% 100.0% 100.0% 100.0% 100.0% 100.0%   <- 坍塌版
```

**坍塌版死了 5 个专家,accuracy 跟健康版一样全是 100%。** 如果你只看 loss 和 accuracy,你会判定这两个模型一样好。它们不一样好——一个是健康分工,一个是一半参数变死重。loss 和 accuracy 完全没能力区分。loss 降下去只说明模型整体能拟合任务(dense 也能),它根本不告诉你 MoE 这个结构有没有在起作用。

正确的验证方法是看**专家-子任务共现矩阵**:行是专家,列是 ground-truth 子任务(`clusterId`),格子里是「子任务 c 的 token 被路由到专家 e 的比例」。`stage07-moe-capstone.ts:263` 按列归一(「余数 r 的 token,有多大比例流向专家 e」),然后问一个几何问题:**这个矩阵接不接近一个置换矩阵(permutation matrix,每行每列只有一个大值的方阵)?**

健康版的真实热图(`core/viz.ts` 的 `heatmap`,纯 ASCII,终端就是显示器):

```
    r0r1r2r3r4r5r6r7
 E0
 E1     ▒▒
 E2   ██      ██
 E3         ▓▓
 E4     ░░
 E5       ██      ██
 E6             ▓▓
 E7 ▓▓
对角占比 (路由一致性) = 0.939 | 被专家认领的残基 = 8/8
```

每一列(每个余数)的质量几乎全压在某一个专家上:r0→E7、r1→E2/E4、r2→E5、r3→E3……8 个余数全部被某个专家「认领」(占该余数 token 过半)。这就是「专家自发对齐到任务子结构」的视觉指纹。没有人告诉 router「r0 应该去 E7」——`clusterId` 只用在这里画图对照,从不进 loss。专家的分工是从「稀疏激活 + 负载均衡」这两个约束里**涌现**出来的,直接呼应第 01 章立的论点和第 06 章对路由动力学的分析。

为什么共现矩阵比 loss 强这么多?因为它直接测的是 MoE 唯一的存在理由。MoE 不为降 loss(dense 也能),它为的是**用稀疏激活换得「不同专家专精不同子分布」的容量**。共现矩阵就是这件事成没成的直接读数。一个标量只能告诉你一个侧面。

而且有个陷阱:**别拿「对角占比」这个标量当分工的判据。** `stage07-moe-capstone.ts:299` 的 `permutationCloseness` 算的是「每个余数最大单专家占比的均值」——它测的是路由**一致性**,不是**均衡**。看坍塌版:

```
对角占比 (路由一致性) = 1.000 | 被专家认领的残基 = 8/8
```

坍塌版对角占比 **1.000**,比健康版 0.939 还高!因为残基仍然被**一致地**送到某个固定专家(只是这些专家就那 3 个,大部分专家空着)。一致的路由 ≠ 健康的分工。所以必须把对角占比和**死专家数、CV、熵**一起读,绝不能单看。这正是为什么 `stage07-moe-capstone.ts:296` 的函数注释明写「Read it WITH dead-expert count and CV, never alone」。判断健康度的真正信号是 ASCII 热图里**有没有空行**(死专家)——健康版 E0 那行是空的但只 1 行,坍塌版 E0/E3/E4/E5/E6 五行全空。**热图近置换且无空行 = 真分工;对角占比高但成片空行 = 坍塌伪装成一致。**

> 一个诚实的边界:`makeModularTask` 是 toy、近线性可分的小数据,所以这张对角矩阵漂亮得有点理想。`data.ts` 自己就标了:「可迁移的是故事的**形状**——分工会涌现、aux 压平利用率、坍塌时熵塌缩——而不是绝对数字。」真实 MoE(Mixtral、DeepSeek)的共现矩阵远没这么干净:专家分工更模糊、更多是「软专精」(偏好某类 token 但不独占),对齐的也往往不是人类可命名的语义而是某种统计子分布。但「看共现而非看 loss、且热图配死专家数一起读」这套方法论是直接迁移的。

## 稀疏到底省了多少:把 FLOPs 算到 MAC 级

现在兑现第 01 章的承诺:稀疏激活省算力,省多少,精确算。`core/metrics.ts` 的 `activatedFLOPs` 和 `denseFLOPs` 按 matmul 维度真实计数 MAC(乘加,一次 MAC = 一乘一加),量化的是**算法工作量**。`stage07-moe-capstone.ts:370` 调它们,真实输出(评估 256 token,`E=8`、`K=2`、`HIDDEN=12`):

```
① 本 MoE 激活 FLOPs (只 top-2/8 专家 + gate) = 172,032 MAC
② dense FFN 激活 FLOPs (全 8 专家, 无 router)   = 589,824 MAC
③ 总参数 FLOPs ("若每 token 付全部参数")        = 614,400 MAC
激活/dense 比 = 0.292 (≈ k/E = 0.250; gate 开销让它略高于 k/E)
省下 442,368 MAC/批 (72.0%)
```

整理成对账表:

| 配置 | 总 MAC (T=256) | 相对 dense | 含义 |
|---|---|---|---|
| ③ 总参数(全 8 专家 + gate) | 614,400 | 1.042 | 你**存**了这么多参数 |
| ② dense FFN(全 8 专家激活) | 589,824 | 1.000 | 同参数量的 dense baseline |
| ① 本 MoE(top-2/8 + gate) | 172,032 | 0.292 | 每 token 实际**算**了这么多 |
| 理论下限 k/E | — | 0.250 | 纯维度算术 |

三个数讲一个故事:你**存**了 ③ 那么多参数(8 个专家全在显存里),但每 token 只**算**了 ① 那么多(只过 2 个专家加 gate),省下 72%。

两个细节值得停一下。其一,dense baseline(②)故意**不算 gate**——dense 模型没有路由器,为了苹果对苹果,不该给它硬加一个不存在的开销;反过来 MoE 的 gate 开销是真实的,必须计入。这正是为什么实测比例 0.292 **略高于**理论 k/E=0.250:那 0.042 就是 gate(`HIDDEN × E = 12 × 8 = 96 MAC/token`)的额外开销。其二,③和②的差(614,400 vs 589,824)就是 gate 这点参数——它说明 router 本身在参数账上几乎可以忽略,MoE 的成本全在专家上。

「激活/dense ≈ k/E」是**可迁移的那个数**:从这个 toy,到 Mixtral(8 选 2),到 DeepSeek-V3(256 个路由专家选 8),都成立,因为它纯粹来自维度算术、跟硬件无关。

### 但这张表的可迁移边界要诚实划清

`stage07-moe-capstone.ts:390` 在打印 FLOP 表后紧跟一段警告,我必须原样转述,否则这章就成了我最该反对的那种 survey 稿:

```
⚠ 诚实标注: 此比值 (k/E 量级) 可迁移到真实 MoE; 但绝对 MAC 数与真实模型吞吐无关——
   toy 规模、无 kernel/硬件, 这里不测 wall-clock, 只数算法乘加。真实加速还受 router/通信开销影响。
```

翻译成大白话:**这张表说的是「该算多少」,不是「实际跑多快」。** 算法上你省了 72% 的乘加,但你实际能不能拿到 72% 的加速,取决于一件这本书的代码里**完全没有、也不可能有**的东西——通信和 kernel 效率。toy 规模这些全被掩盖了:8 个专家全在一个进程的内存里,top-2 路由就是几个数组索引,零通信成本。绝对 MAC 数(172,032)对真实模型吞吐毫无意义。这恰好是下一节、也是交给读者的下一程。

顺带把坍塌的代价也算进算力账:坍塌版 5/8 个专家从不参与计算,等于 **62.5% 的专家参数被存下却空占显存、不贡献任何容量**。MoE 「总参数大、激活算力小」的红利,前提是**存的参数都被路由用上**;坍塌恰恰打破这个前提,让一半参数变成死重。这就是为什么 aux + capacity 不是装饰——它们是这张 FLOP 对账表能成立的先决条件。

## ⚡ 最后一公里:真实系统的瓶颈根本不在算法,在通信

到这里你已经手写出一个会自发分工、FLOPs 账目清清楚楚的 MoE。但如果你以为「把这个 scale 上去就是 Mixtral 了」,那是这本书最该帮你戳破的幻觉。**本书的 toy 到真实生产系统之间,隔着的最后一公里几乎全是算法之外的东西,而且目前没有通用解。**

核心矛盾:真实 MoE 里,专家分布在多张卡上(几十到上百个专家,单卡放不下)。一个 token 的 top-k 路由结果是「我要去 3 号卡的 E17 和 9 号卡的 E58」——这意味着每一层 MoE,所有 token 都要按路由结果**跨卡重新分发**,算完再收回来。这个操作叫 all-to-all(每张卡都要和其余每张卡交换数据的集合通信原语),它是 MoE 训练和推理里最贵的一步,常常比专家的矩阵乘本身还贵。

这就推翻了一个直觉:你省下的那 72% FLOPs,很可能被 all-to-all 的通信开销吃掉一大半甚至全部。**算法上的稀疏胜利,在系统层面会被通信成本反噬。** 所以 Mixtral、DeepSeek 这些系统真正的工程重心,根本不是路由算法(那部分就是本书这点东西)——而是:

- **通信优化**:怎么把 all-to-all 和计算重叠、怎么用 expert parallelism 的拓扑减少跨节点流量、怎么处理某张卡负载过重卡住所有卡的掉队者问题。你在本章看到的 CV=0.615 那点利用率不均,在单进程里无所谓;在多卡里,最忙的那张卡决定整层的时间,均衡直接等于吞吐。
- **kernel**:稀疏激活在 GPU 上天然不友好——不同 token 走不同专家,batch 不规整,标准 dense matmul kernel 跑不满。MegaBlocks 这类工作把 MoE 重写成**块稀疏(block-sparse)矩阵乘**,用专门的 kernel 把不规整的稀疏计算重新打包成 GPU 能吃满的形状。这部分的工程量和收益,远超路由算法本身。

换句话说:**算法只是冰山一角,水面下是通信和 kernel。** 这也是为什么本章的 FLOP 表明确标成「算法工作量,不是吞吐」——我们诚实地只做了水面上那部分,而且做透了。这最后一公里目前没有「装一个库就解决」的通用方案:不同硬件拓扑、不同专家数、不同 batch 模式,最优的通信策略和 kernel 都不一样,expert parallelism 的最优切分、容量因子的动态调度、通信-计算重叠的编排都仍在演进——这是活跃的工程和研究前沿,**没有通用解**。

## 收尾:你现在站在哪,下一步去哪

回头看这本书做了什么:从一个张量引擎(`core/tensor.ts`)和几个层原语(`core/nn.ts`)出发,把 MoE 拆成可导路由、负载均衡、容量丢弃、坍塌诊断四个机制,逐个讲透怎么坏、为什么这么设计,最后在这一章用 `stage07-moe-capstone.ts` 组装起来,亲眼看见:

- **分工是涌现的**——健康版共现热图近置换、8/8 余数被认领、0 死专家,这是分工涌现的硬证据;而判断它健康靠的是「热图无空行」,不是 loss、不是 accuracy(两个版本 accuracy 都 100%)、甚至不是对角占比(坍塌版反而 1.000)。
- **稀疏真省算力**——FLOP 对账给出 0.292 的激活比例、72% 的节省,兑现了第 01 章的承诺,同时诚实标注它只是算法账、不是吞吐账。
- **保护不是装饰**——关掉 aux + capacity,同任务同初始化重训,5/8 专家死亡、熵从 1.875 崩到 0.904、62.5% 参数变死重。红利依赖这些保护把 token 摊到各专家。

留给读者的下一程很明确,建议按这个顺序:

1. **先把「看共现 + 数死专家」变成肌肉记忆**——以后看任何 MoE(包括你 fine-tune 别人的),第一件事是画专家-子任务共现热图、数有没有空行,而不是看 loss。
2. **再去碰真实系统的通信层**:读 MegaBlocks 的块稀疏 kernel、读任一个 expert parallelism 的 all-to-all 实现,你会立刻明白为什么本章的 toy 跑得飞快、而真实系统在为通信流血。
3. 本书的 `core/` 是确定性、零依赖、CPU 可跑的——它存在的意义是让你把**算法**这一层彻底搞懂,这样当你面对真实系统那 90% 的通信和 kernel 复杂度时,剩下的 10% 算法部分对你是透明的、没有黑箱。

算法这一层,你现在做透了。剩下的,是工程。
