第 02 章把稀疏激活跑通了,但精确地停在一道墙前:门控网络吐出 E 个 logits(未归一化的原始分数),你对它们做 top-k,取前 k 个。`argmax` / top-k 是阶梯函数——把某个 logit 推高一丝,被选中的专家不变(导数为 0),直到它越过邻居,选择"啪"地翻转(导数无定义)。这条路上没有可导的地方。可门控网络偏偏是一层 `Linear`(线性层),你想训它的权重 `W`,梯度从哪来?

这章给出答案,并且给的是工业界真实在用的那个答案,不是教科书第一反应。**很多人以为 MoE 路由要上 Gumbel-softmax(可微近似离散采样)或者 REINFORCE(策略梯度)这类重武器去"绕过"不可微。实际上主流框架(Switch Transformer、GShard、Mixtral)用的是一个朴素到有点反高潮的近似:梯度根本不流过"选了谁",只流过被选中专家的连续门控权重。** 这是本章第一个要钉死的判断,理解它能让你省掉一大堆过度工程。

## 一、墙在哪:把 argmax 和门控概率分开看

先把两件事分清,它们在代码里长得像,作用完全不同。

第一件是**选择**:决定这个 token 走哪 k 个专家。这步在配套代码 `examples/moe-from-scratch/src/stage03-nondifferentiable-routing.ts` 里就是一个纯 JS 排序,返回普通数字下标:

```ts
// examples/moe-from-scratch/src/stage03-nondifferentiable-routing.ts
function topKIndices(row: Value, k: number): number[] {
  const pairs = Array.from(row.data, (v, j) => ({ v, j }));
  pairs.sort((a, b) => b.v - a.v);
  return pairs.slice(0, k).map((p) => p.j);
}
```

注意它读的是 `row.data`(裸 `Float64Array`),返回的是 `number[]`。这些下标**脱离了计算图**(detached,不带反向传播信息)——它们是从图里"读出来"的标量,不是图里的节点。这就是不可微那道墙:墙本身是真的,你没法对"选了 2 和 3 还是选了 0 和 1"求导。

第二件是**权重**:被选中的专家输出,各自乘以多少。这才是连续的、可导的。门控 logits 过 `softmaxRow()`(把一行分数压成和为 1 的概率)得到 E 维概率,我们只把被选中专家对应的那几个概率取出来当乘子:

```ts
// examples/moe-from-scratch/src/stage03-nondifferentiable-routing.ts (forwardToken, routeMode="prob")
const gateProbs = noisy.softmaxRow(); // [1,E], 可导
const selected = topKIndices(noisy, TOP_K); // detached 下标 — 不可微那一步
// ...
weight = gateProbs.gather(e); // 标量, 仍在图里
out = out.add(expertOut.mul(weight));
```

关键就在 `gather(e)`——按下标从概率行里取一个标量。`argmax` 只决定**走哪条路**(取哪个 `e`);路上的乘子是个可导的 softmax 概率值,门控 `Linear` 学到的全部信号都来自这个乘子。换句话说:**离散选择负责路由,连续概率负责承梯度**。墙绕不过去,但你根本不需要绕——梯度从来就不打算穿墙,它走的是墙边那条连续小路。

### 为什么 `gather` 能把梯度送回门控

光说"概率承梯度"还不够,得看反向传播到底把梯度送去哪。看核心张量库里 `gather` 的反向定义 `examples/moe-from-scratch/src/core/tensor.ts:321`:

```ts
// examples/moe-from-scratch/src/core/tensor.ts
gather(j: number): Value {
  // ...
  const out = new Value([this.data[j]], [1, 1], [this], `gather[${j}]`);
  out._backward = () => {
    this.grad[j] += out.grad[0]; // 只把梯度累加到被选中的那一列 j
  };
  return out;
}
```

`out.grad[0]` 是 loss 对这个被选概率的导数,反向时只累加到 `this.grad[j]`——也就是被选专家那一列。其余列的 `grad` 一动不动。这等价于"one-hot 向量点乘概率行",但不用真去物化那个 one-hot 向量。门控 `W` 的梯度,就是顺着 `softmaxRow → gather[j]` 这条链路回流的:被选中专家对应的概率列收到信号,门控据此学"下次该不该更看好这个专家";未选专家这一步压根没进图,自然没有信号。

## 二、用数值证给自己看:梯度只走被选专家

上面是讲道理。资深工程师对"应该是这样"保持怀疑,所以这章不 assert,直接读 `.grad` 缓冲区证。代码 `[C]` 段(`microGradientPathExperiment`)固定一个输入、跑一次前向反向,然后逐个打印每个专家 `fc1.W` 的梯度 L2 范数。真实输出(seed=7,toy 规模):

```
固定输入下被选中的 top-2 专家: [2, 3]
expert 0 fc1.W grad L2 = 0.000e+0  [✓ 未选 → grad 恒为 0]
expert 1 fc1.W grad L2 = 0.000e+0  [✓ 未选 → grad 恒为 0]
expert 2 fc1.W grad L2 = 6.949e-2  [✓ 被选 → grad ≠ 0]
expert 3 fc1.W grad L2 = 7.993e-3  [✓ 被选 → grad ≠ 0]
gate.W grad L2 = 1.662e-2  [门控自身有梯度: 通过被选概率的 gather]
```

未选的 expert 0 / 1 梯度**精确为 0**——不是"很小",是 0。原因在 `forwardToken` 的循环结构:只有被选中的专家才调 `moe.experts[e].forward(x)` 把自己的输出加进 `out`。未选专家的前向从未进入这一步的计算图,`backward()` 自然永远不会访问它们的 `_backward`。这是图的拓扑性质,不是数值大小问题——**所以它在任何规模下逐字成立**,不像 loss / 熵那些绝对数会随规模漂移。这一点请记牢:第 04、06 章那个"没被选的专家收不到梯度→训不动→更不被选"的坍塌动力学,根就在这里。

顺带:门控自己有梯度(`gate.W grad L2 = 1.662e-2`),正是通过被选概率的 `gather` 回流的。这条线活着,门控才学得动。

## 三、失败模式:把乘子建在离散下标上,门控直接冻死

讲完正确做法,得讲它怎么坏、为什么必须这么设计——否则你不知道自己躲过了什么。代码 `[D]` 段(`failureModeArgmaxBackprop`)故意走错路:`routeMode="index"`,乘子不取门控概率,改成用 argmax 排名算一个标量(`1/(rank+1)`):

```ts
// 失败模式: 乘子来自离散排名, 与门控图断开
weight = Value.scalar(1 / (rank + 1));
```

`1/(rank+1)` 是个普通数字,`Value.scalar` 造的是一个**没有反向边的叶子**——它和 `gate.W` 之间没有任何可导路径。后果是门控网络收不到一丁点梯度。真实输出:

```
门控权重梯度范数: 平均=0.000e+0  最大=0.000e+0
→ 梯度全程 ≈0: argmax 索引是 detached 标量,损失到 gate.W 之间没有可导路径
训练前后门控权重最大变化量 = 0.000e+0  (≈0 = 门控被冻结)
专家利用率冻结在初始化偏好上:
init E0 │██████████████████████████████│ 0.3250
     E1 │█████████████████████████·····│ 0.2750
aftr E0 │██████████████████████████████│ 0.3250
     E1 │█████████████████████████·····│ 0.2750
利用率是否原地不动: ✓ 完全冻结 (利用率向量逐元素相等)
```

训练 120 步,门控权重最大变化量 `0.000e+0`,利用率向量训练前后逐元素相等。门控被彻底冻在随机初始化偏好上,等于没训。**教训:argmax 只决定走哪条路,不可导;必须让连续的门控概率(softmax 后 gather 被选列)充当乘子来承梯度。把乘子建在离散索引上 = 门控网络收不到任何学习信号。** 这就是为什么所有真实 MoE 实现都把 gate prob 乘在专家输出上——不是为了"加权平均好看",是这个乘法是门控唯一的命根子。

## 四、noisy top-k gating:给 logits 加噪声,到底买到了什么

直通式近似(直接让概率承梯度)够用,但有个真实痛点:`argmax` 的硬边界让选择"非黑即白"。两个专家分数咬得很近时,梯度一旦让其中一个反超,选择瞬间翻转,门控梯度会出现尖刺。Shazeer 2017(那篇把 MoE 做进 LSTM、第一次大规模跑通稀疏门控的工作)提出 **noisy top-k gating**:在 top-k **之前**给每个 logit 加一个高斯噪声,再做选择和 softmax。

代码里噪声作为一个常量叶子加进去——它扰动 softmax 看到的 logits 和 argmax 的选择,但自身没有 `_backward`,所以不承梯度:

```ts
// examples/moe-from-scratch/src/stage03-nondifferentiable-routing.ts
let noisy = logits;
if (opts.noiseStd > 0) {
  const noise = new Float64Array(logits.cols);
  for (let j = 0; j < logits.cols; j++) noise[j] = r.normal() * opts.noiseStd;
  noisy = logits.add(Value.from(Array.from(noise), [1, logits.cols]));
}
```

教科书对噪声的宣传通常是两条:(1) 平滑决策边界、稳定梯度;(2) 增加探索、防止过早锁死少数专家。这章 `[A]` 段做了硬路由 vs noisy 的 A/B,把两条都拿真数字检验——**而且诚实报告了哪条没成立**。

### A/B 真实结果

门控 `Linear` 权重梯度范数的抖动(整条曲线的标准差,越小越平滑):

```
抖动(std of grad-norm): hard=0.4977  noisy=0.2953  → noisy 更平滑(更小)
```

**梯度更平滑这条成立**(noisy std 0.2953 < hard 0.4977)。这是噪声在本任务上唯一稳健、可迁移的收益:它把选择翻转抹匀了,梯度曲线少了 hard 路由那些 snap 尖刺。

但第二条,探索更广,在这个任务上**不成立**:

```
探索峰值熵(max over steps): hard=1.3442  noisy=1.2972  → noisy 峰值未更高
收敛 loss: hard=0.0061  noisy=0.0044
最终专家利用率(top-1): hard 与 noisy 完全一致 (E0=0.4625 E1=0.2750 E2=0.0000 E3=0.2625)
```

路由熵(跨专家分配分布的熵,越高代表 token 摊得越散)noisy 峰值 1.2972 反而略低于 hard 的 1.3442;终态利用率两者**逐元素完全一致**,连 E2 都同样被晾在 0.0000。为什么?这批 toy 数据是干净可分的 clusters(簇),没有局部最优陷阱——确定性门控本来就能快速找对专家,**噪声无处可探**;而且收敛后门控足够自信,微小噪声不足以翻转 top-2 的选择,所以终态一模一样。

我特意不去调任务把"噪声促进探索"的故事造出来——那是 survey 式的不诚实。**噪声的探索收益是真的,但它只在有坍塌陷阱的地形上才显现**:当确定性门控会一头扎进"只用 2 个专家"的坏局部解时,噪声能把 token 偶尔甩给冷门专家、给它们一次被训练的机会。那个场景留给第 06 章专门构造。这里要带走的可迁移结论只有一条:**noisy gating 稳的是梯度,不必然促进探索;探索收益依赖地形**。

> 诚实标注:以上 loss / 熵的绝对值偏乐观(toy 规模、单 seed=7、tiny net),可迁移的是相对趋势(noisy 梯度抖动更小、终态在易任务上与 hard 一致),不是绝对数字本身。

## 五、前沿:绕开不可微的另一条路——让专家挑 token

⚡ **token-choice 与 expert-choice 是两种范式,后者从机制上绕开不可微 + 天然均衡,但两者目前都无通用最优解,仍在研究。**

到这里我们用的都是 **token-choice routing**:每个 token 自己挑 top-k 个专家。它有个结构性麻烦——既不可微(本章主题),又不保证均衡(token 们可以集体涌向同一个专家,把它挤爆,这是第 05 章容量问题、第 06 章坍塌问题的根源)。这两个毛病其实同源:让 token 单方面选,选择就是离散的、且无人协调全局负载。

近年有两类方向反过来想:

- **Expert-choice routing**(Google 2022):翻转主语,**让每个专家挑固定数量的 token**,而非 token 挑专家。每个专家有固定容量,从所有 token 里按门控分数选出 top-c 个。好处是双重的——负载**天然均衡**(每个专家就吃固定份额,不可能被晾成 0,也不可能被挤爆),而且因为每个专家选的是固定个数,可以用更可控的方式逼近这个选择。代价是一个 token 可能被 0 个或多个专家选中(分配不再对称),且需要看到整个 batch 才能做选择,在自回归推理(逐 token 生成)时不自然。
- **把路由建模成最优传输**(optimal transport,在两组对象间找总代价最小的分配方案):BASE layers(Lewis 2021)、Sinkhorn 路由等,把"token↔专家"的分配看成一个带均衡约束的指派问题,用 Sinkhorn 迭代或匈牙利算法求一个近似均衡的软分配。这等于把"不可微 + 不均衡"两个问题打包成一个有成熟数值解法的优化问题来解。代价是每步多一轮迭代求解,且约束严格时灵活性下降。

哪种范式更好,没有定论:token-choice 简单、推理友好、生态成熟(Mixtral 这类主力开源模型都用它),但要靠第 04 章的辅助均衡损失硬掰;expert-choice 和最优传输路由结构上更优雅、自带均衡,但有 batch 依赖、推理不自然、分配不对称等新麻烦。**这是一个开放问题,目前没有在所有场景下通吃的方案**,工业主力仍是 token-choice + 直通近似 + 辅助损失这套朴素组合。

知道这条朴素组合"有偏但够用",并且知道它偏在哪(不均衡、要靠下一章的损失硬掰),就能避开两个极端:既不会因为"它不优雅"而过度上 Gumbel / 最优传输,也不会以为加了 noisy gating 就万事大吉。

---

下一章直面这套朴素组合最大的债:第 04 章会看到,即便有 noisy gating 帮着平滑梯度,token-choice 路由的正反馈仍会把负载推向少数专家——本章 `[A]` 里那个始终是 0.0000 的 E2 就是预兆。光靠噪声压不住,得上一个显式的负载均衡损失。
