先把这章最容易被误会的一句话钉死:**一个 MoE 层里的「专家」不是按领域人工分工的子网络,它们结构完全相同、初始化也只差随机种子,谁负责什么是训练自发涌现出来的,不是你设计出来的。** 你不会写一个「负责代码的专家」和一个「负责诗歌的专家」,你写的是 E 份一模一样的 MLP(多层感知机,几个线性层夹非线性),然后让门控(gating,一个小线性层,决定每个 token 走哪个专家)在训练里慢慢把它们推开。第 00 章论证了 MoE 为什么值得整本书拆透——它是当前把参数量和算力解耦的主流手段。这一章从最朴素的一步起步:**先不省任何算力**,把一个普通 FFN(前馈网络)拆成 E 个并行专家、每个 token 跑遍所有专家。听起来蠢(确实蠢,生产绝不这么干),但它是唯一干净的对照组——只有先看清「全选」花多少算力、买到多少容量,你才能在第 02 章量出「只选 k 个」省下了多少。

本章对应的可跑代码是 `examples/moe-from-scratch/src/stage01-moe-layer.ts`,跑法 `npx tsx src/stage01-moe-layer.ts`。它在共享底座 `src/core/`(纯 TypeScript、零运行时依赖、固定种子 PRNG、禁用 `Math.random`)之上,做四件事:① gradCheck 证明引擎、② dense 单专家 vs 4 专家 MoE 的对照训练、③ 两条 loss 曲线、④ 一个对称性破缺的失败模式 demo。下面引用的所有数字都来自这个脚本 `seed=7` 的真实输出,逐字节可复现;凡是 toy/合成数据上的实测值我会明确标注「绝对值偏乐观,可迁移的是机制和比例」。

## 先有引擎,才谈专家:为什么第一件事是 gradCheck

在拆 FFN 之前,得先回答一个被大多数教程跳过的问题:**你凭什么相信你的反向传播是对的?** MoE 比普通网络更容易藏反向 bug——它有门控权重把梯度从专家引回路由器、有跨 token 的参数复用(同一个专家被一个 batch 里很多 token 反复调用),这些都是梯度累加(grad accumulation——多条路径的梯度相加)出错的高发区。如果引擎的某个算子反向写错了,你照样能看到 loss 下降,只是下降得「不对」,然后你会花三章去 debug 一个其实在第一章就埋下的 adjoint(伴随,反向传播里每个算子对应的梯度公式)错误。

所以底座 `src/core/tensor.ts` 是一个极小的 reverse-mode autograd(反向模式自动微分,micrograd 风格但支持矩阵),它有一条贯穿全书的核心不变量:梯度必须 `+=` 累加,绝不 `=` 覆盖。为什么对 MoE 尤其致命:一个专家在一个 batch 里被多个 token 命中,它的参数梯度必须是**所有命中 token 的梯度之和**;如果反向用了 `=`,只有最后一个 token 的梯度被留下,专家就训不动。普通网络里这个 bug 也存在,但 MoE 的「一个专家服务多个 token」结构让它更隐蔽、更普遍。

stage01 第一件事就是验证引擎,而且它验的不是抽象的算子,是**本章真要用的那段计算图**——一个 `Expert` 喂进 `crossEntropy`。看 `src/stage01-moe-layer.ts::proveEngine`:

```ts
// 对本 stage 真实用到的计算图切片做 gradCheck:单个 Expert 接 cross-entropy。
function proveEngine(): number {
  const r = rng(SEED);
  const expert = new Expert(DIM, HIDDEN, NUM_CLASSES, r);
  const x0 = rowFrom([0.4, -0.7, 0.2, 1.1, -0.3, 0.6]);
  const target = 1;
  const f = (x: Value) => crossEntropy(expert.forward(x), target);
  return gradCheck(f, x0);  // 数值中心差分 vs autograd 解析梯度,返回最大相对误差
}
```

`gradCheck` 对输入逐元素做中心差分(用 `(f(x+ε)-f(x-ε))/2ε` 数值估梯度)和解析梯度对拍。stage01 真实输出:

```text
① 引擎自检 (gradCheck: Expert→交叉熵, 数值梯度 vs autograd)
   max 相对误差 = 1.312e-10  (阈值 <1e-4: PASS)
   含义: 下面所有训练数字建立在『梯度算对了』之上, 否则一切归零。
```

`1.312e-10` 远低于通常 `1e-4` 的通过阈值,且不只是个 PASS——脚本结尾有一行硬断言 `if (!enginePass) throw`,引擎一旦回归就是响亮的崩溃而不是悄悄漂移。**这不是断言引擎对,是证明引擎对。** 注意被测的 `Expert` 是带 GELU(高斯误差线性单元,Transformer 标准激活)的 2 层 MLP,它的反向是 autograd 自动求的,没人手写 adjoint——这正是 `src/core/nn.ts` 顶部注释解释的设计:所有层都只是 tensor 算子的薄组合,「我们从不在这一层写自定义反向,所以 adjoint bug 没有第二个藏身处」。翻译成大白话:**专家不写自己的反向,bug 只可能藏在引擎一个地方,gradCheck 一次兜住全部。** 这是从零做 MoE 和「调 PyTorch」最大的认知差异——你得自己保证微分正确,而保证的方式是数值对拍,不是相信。

## 一个专家长什么样:就是个 2 层 MLP

确认引擎可信后看「专家」本体。在 `src/core/nn.ts` 里,`Expert` 就是一个两层 MLP:

```ts
export class Expert implements Module {
  fc1: Linear;
  fc2: Linear;
  act: Activation;
  constructor(inDim: number, hidden: number, outDim: number, rng: Rng, act: Activation = "gelu") {
    this.fc1 = new Linear(inDim, hidden, rng);
    this.fc2 = new Linear(hidden, outDim, rng);
    this.act = act;
  }
  forward(x: Value): Value {
    const h = this.fc1.forward(x);
    const a = this._activate(h);
    return this.fc2.forward(a);
  }
  parameters(): Value[] {
    return [...this.fc1.parameters(), ...this.fc2.parameters()];
  }
}
```

这就是 Transformer 里那个再普通不过的 FFN 子层——`in → hidden → out`,中间一个 GELU。**一个专家 = 一个 FFN,没有任何特殊结构。** 这点必须先建立,否则后面会一直误以为专家是某种「带领域知识的特化模块」。

这里有一个朴素但关键的失败模式,`nn.ts` 用一条不变量防住:`parameters()` 必须返回前向里**那同一批** `Value` 对象(不是副本)。为什么 MoE 里这个坑特别大:MoE 层是一个**容器**,里面装 E 个专家。如果你在容器的 `parameters()` 里漏登记某个专家的参数,优化器就拿不到它的梯度,那个专家**永远停在初始化值**——而且因为别的专家在训练,loss 照样下降,你完全不会从 loss 曲线上发现有个专家是死的。stage01 的 `DenseMoELayer` 用 `flatMap` 把每个专家的参数显式拼起来,就是为了让「漏登记」很难发生:

```ts
class DenseMoELayer {
  experts: Expert[];
  parameters(): Value[] {
    return this.experts.flatMap((e) => e.parameters());  // 每个专家的参数都进优化器
  }
}
```

注释里把这条钉成了不变量:「parameters() 返回每个专家的参数(展平)。在这里漏掉一个专家会悄悄冻结它。」这是设计层的防御,不是实现细节。

## 把 FFN 拆成 E 个专家:全选(无权重均值)对照组

现在搭本章的对照组。**MoE 层 = 一组结构相同、参数独立的专家 + 一个门控**(这是本章第一个 ✦ 专家钩子)。但 stage01 故意把门控**砍掉**,用最笨的合并方式:**每个 token 跑遍全部 E 个专家,输出取无权重均值**。看 `src/stage01-moe-layer.ts::DenseMoELayer.forward`:

```ts
class DenseMoELayer {
  private invE: Value;  // 预存 1/E,均值就是一次标量广播乘
  constructor(experts: Expert[]) {
    this.experts = experts;
    this.invE = Value.scalar(1 / experts.length);
  }
  // forward: mean_e expert_e(x)。每个专家都看每个 token(本章无路由)。
  forward(x: Value): Value {
    let acc = this.experts[0].forward(x);
    for (let e = 1; e < this.experts.length; e++) acc = acc.add(this.experts[e].forward(x));
    return acc.mul(this.invE);
  }
}
```

为什么连门控的加权都不要、直接均值?这是刻意的隔离纪律。第 02 章会上 softmax 门控,但门控分布是连续可导的、不引入新麻烦;真正的雷在第 03 章——稀疏路由的「选哪 k 个」是离散 argmax,**不可导**,梯度传不回门控。本章把这两件事全部隔离掉:**专家结构 + 容量这件事单独成立,不掺门控、不掺稀疏。** 用最朴素的均值,你能确认「E 个独立专家能不能比 1 个专家拟合得更好」这一个问题,不被任何路由逻辑干扰。

代价说清楚:这个东西**没有省任何算力**——它比单个 FFN 慢 E 倍,因为把 E 个 FFN 全跑了一遍。它存在的唯一价值是当干净基线。这恰恰是 MoE 要解决的矛盾:**我想要 E 倍参数,但不想付 E 倍算力。** 本章先把「E 倍参数到底买到什么」量清楚,第 02 章再来省算力。

## 实测:4 倍参数换来多少拟合能力

stage01 在同一份 toy 数据上训两个模型,唯一变量是专家数量:dense 是 1 个 `Expert`(就是一块普通 FFN),MoE 是 4 个 `Expert` 的 `DenseMoELayer`。两者**每专家宽度相同**(`hidden=4`,故意做小让单专家欠拟合)、同数据、同 300 步、同优化器。数据是 6 个高斯簇(`spread=2.2` 故意重叠,所以不是线性可分,小模型真的容量受限)。真实输出:

```text
② 两模型对比 (同数据 / 同每专家宽度 hidden=4 / 同 300 步)
   模型           参数量      ×单专家   终态 train loss   终态 train acc
   dense (1 专家)     58      1.00x      0.2733           87.7%
   MoE (4 专家)      232      4.00x      0.0793           97.3%
   观察: 参数 ≈4× (4.00x), MoE 终态 loss 更低 (0.1941 的差)。
```

读这组数:把 1 个专家变成 4 个,参数从 58 涨到 232,精确 `4.00x`(因为每个专家结构相同、参数独立,E 份就是 E 倍);终态 train loss 从 `0.2733` 降到 `0.0793`,准确率从 `87.7%` 升到 `97.3%`。**容量是真涨的,拟合能力也是真涨的。** 两条 loss 曲线也佐证了「都在下降」:

```text
③ loss 曲线 (左高右低 = 在下降)
   dense (1 专家):
   █▃▁▁▁... [0.2742 .. 10.9993], n=300
   MoE (4 专家):
   █▂▁▁▁... [0.0798 .. 3.5814], n=300
```

但**必须**就着诚实标注读这组数,否则会被误导:这是 6 类高斯簇的 toy 任务,绝对 loss/acc 偏乐观(脚本自己也标了「绝对 loss/acc 偏乐观」)。**可迁移的不是 0.0793 这个数,是「同样的每专家宽度下,E 份独立参数能拟合得更好」这个相对趋势。** 还有一个更重要的限定:这里 4 倍参数也付了 4 倍算力(全选嘛),所以本章的 MoE 在算力账上**毫无优势**——它只证明了容量这件事,没证明 MoE 的价值。MoE 的价值要等第 02 章把算力从 `4.00x` 砍回 `k/4` 才兑现。本章你只需要确立分子(容量)真实存在;分母(算力)怎么省是下一章的事。

## 失败模式:不破缺对称性,4 个专家等于 1 个

「4 个独立专家比 1 个强」有个**前提**,而这个前提一旦不满足,多出来的参数就是纯浪费。stage01 的第 ④ 段专门把这个失败模式做成可复现的 demo,这是本章真正的工程教训。

问题是这样:E 个专家结构完全相同,**唯一让它们不同的是初始化的随机种子**。如果你手滑,给每个专家喂了**同一个 seed**,它们就会被初始化成逐字节相同的权重,前向输出也完全相同,那么「4 个专家取均值」在数学上**等价于 1 个专家**——你白白多算了 3 份,参数还学不出任何差异。看 `src/stage01-moe-layer.ts::buildExperts` 怎么对比这两种初始化:

```ts
function buildExperts(broken: boolean, seed: number): Expert[] {
  const experts: Expert[] = [];
  if (broken) {
    // 坏:每个专家用同一个 seed -> 同样的初始化 -> 没有对称性破缺
    for (let e = 0; e < E; e++) experts.push(new Expert(DIM, HIDDEN, NUM_CLASSES, rng(seed)));
  } else {
    // 健康:一条 rng 流喂所有专家,每个消费到不同的随机数
    const r = rng(seed);
    for (let e = 0; e < E; e++) experts.push(new Expert(DIM, HIDDEN, NUM_CLASSES, r));
  }
  return experts;
}
```

差别只在一行:健康版本**共用同一条 rng 流**(每个专家顺次消费不同的随机数,所以各不相同),坏版本**每个专家各自 `rng(seed)` 重开同一个种子**(于是个个一样)。stage01 用一个探针输入量两种初始化下「E 个专家输出的逐元素最大差」:

```text
④ 失败模式: 不破缺对称性 (E 个专家用同一 seed 初始化)
   健康初始化 (各专家独立随机): 专家输出逐元素最大差 = 4.149e+0
   坏初始化   (各专家同一 seed): 专家输出逐元素最大差 = 0.000e+0
   坏初始化 MoE 终态 loss = 0.2733 vs dense(单专家) = 0.2733 (差 0.0000)
   结论: 同 seed 初始化时 4 个专家输出几乎相同 (差≈0), mean 等价于单专家,
         多出的 174 个参数白涨 — 对称性破缺 (独立随机初始化) 是专家分化的前提。
```

把这几个数串起来读,失败模式就一目了然:健康初始化下 4 个专家的输出最大差是 `4.149`(它们计算的是真正不同的函数),坏初始化下是**精确的 `0.000`**(完全一样)。更狠的是把坏版本也训完 300 步:它的终态 loss 是 `0.2733`,和 dense 单专家的 `0.2733` **差 0.0000**——多出来的 `232 - 58 = 174` 个参数,一分钱拟合能力都没买到。脚本结尾还有断言 `if (!(brokenSpread < healthySpread)) throw`,保证这个对比永远成立。

为什么会这样?这是深度学习里的**对称性破缺**(symmetry breaking)问题:几个结构相同的单元如果起点完全一样,梯度也会完全一样,它们会**同步更新、永远保持相同**,训练再久也分不开。打破对称性的唯一办法就是让起点不同——对专家来说,就是**独立随机初始化**。这条在普通网络里(比如同一层的多个神经元)就成立,但 MoE 把它放大了:MoE 的全部价值建立在「专家会分化」之上,而分化的**先决条件**就是这里的对称性破缺。一个种子写错,整个 MoE 退化成 dense,而且 loss 曲线看起来一切正常——这正是它危险的地方。

## 分工是涌现的,不是设计的(✦)

现在能把开篇那句话讲透了。对称性破缺只是**让专家有机会不同**,它没规定专家**该怎么不同**。专家之间的分工——谁擅长处理哪类 token——不是你设计的,是训练自发涌现的。

机制是个正反馈:专家初始化只差随机种子(微小差异),一旦后续章节引入门控,门控的竞争(softmax 让某个专家在某类 token 上略占优 → 拿到更多这类 token → 在这类 token 上训得更好 → 下次更占优)会把这点微小差异放大成稳定的特化。你不告诉门控「topic 0 的 token 该去专家 3」,你只给任务 loss,然后训练结束去看专家和数据类别之间是不是形成了近对角的对应关系——是,就说明分工**涌现**了。

初学者最常见的误解恰恰反过来:以为要手动给专家分配领域。事实是,**你给专家硬分领域反而会限制它**——训练找到的最优分工往往不是人类直觉的「代码 vs 诗歌」,而是某种更细、更难命名的特征聚类。这个涌现现象第 07 章会用实测的专家×簇共现热图坐实(本章只埋钩子,不提前编数)。本章你只需建立这个心智模型:**专家是同质容器,分工是训练的产物——而它的物理起点,就是上一节那个 `4.149` vs `0.000` 的对称性破缺。**

这个正反馈也是把双刃剑:失控就是**专家坍塌**(expert collapse——所有 token 涌向少数专家,其余饿死),那是第 06 章的主题;为了压住它要引入负载均衡损失,那是第 04 章。换句话说,本章建立的「涌现分工」机制,既是 MoE 能 work 的根,也是后面三章要驯服的病根。先理解涌现为什么是好事,才能理解它失控为什么是灾难。

## ⚡ 前沿:纯稀疏不是终点,DeepSeek 的共享专家

本章把专家讲成「同质、独立、靠路由差异化」。但有一个重要的现实修正值得现在就埋下:**纯稀疏路由有个浪费——通用知识被迫在每个专家里重复存一份。** 想象「语法、常识」这类几乎每个 token 都要用的能力,如果每个稀疏专家都得自己学一遍,E 份专家里就有一大块参数是冗余的。

DeepSeek 系列(DeepSeekMoE / DeepSeek-V2/V3)提出的修正是**共享专家**(shared expert):把一小部分专家设为**恒激活**——不参与路由,每个 token 都过它们,专门承载通用知识;剩下的「路由专家」才走稀疏 top-k,专心做差异化。这样通用部分只存一份,稀疏专家腾出来全用于特化,参数利用率更高。落到本章的代码框架,这只是配置上加一行(概念预告,真正实现在后续章):

```ts
const cfg = { E: 8, k: 2, sharedExperts: 1 };  // 8 个路由专家 top-2 + 1 个恒激活共享专家
```

`sharedExperts: 1` 意味着每个 token 实际激活 = 1(共享,必过) + 2(路由,top-2) = 3 个专家,但共享那个不参与「选哪个」的离散决策,梯度更稳。

为什么标 ⚡:**共享专家的最优数量、共享 vs 路由的参数预算如何分配、它和负载均衡的相互作用,目前没有通用解、仍在研究。** DeepSeek 给的是一组经验配比(在它们的规模和数据上有效),但「多少通用知识该共享、多少该让路由专家各自涌现」本质是个开放权衡——换个数据分布、换个模型尺寸,最优点会变。这正是本书反复强调的:MoE 的每个机制都是**选择**,带可量化的代价,没有放之四海皆准的配方。

## 小结与下一章

这一章用一个能跑的 stage01 做了四件事,每件都为后面铺路:

1. **先证引擎再谈专家**:`gradCheck` 实测最大相对误差 `1.312e-10`,把反向 bug 在第一章就兜死,因为 MoE 的「一专家多 token」结构让梯度累加错误尤其隐蔽。
2. **拆 FFN 成 E 个同质专家、用无权重均值合并(全选)**:确立「多专家 = 容量变大」——实测 4 专家参数 `4.00x`、终态 loss 从 `0.2733` 降到 `0.0793`,但故意不省算力,作为干净的算力基线。
3. **失败模式:对称性破缺是分化的前提**——同 seed 初始化让 4 个专家输出逐元素差 `0.000`,MoE 退化成单专家(loss 差 `0.0000`),174 个参数白涨。独立随机初始化是 MoE 能 work 的物理起点。
4. **分工是涌现的(✦),不是设计的**:专家是同质容器,谁负责什么靠训练正反馈自发形成,第 07 章实测坐实。

核心心智模型:**专家是同质容器,分工靠训练涌现(✦);全选只是对照组,真正的 MoE 要把 `4.00x` 的激活算力砍到 `k/E`。**

第 02 章接手最难的那块:把无权重均值换成 softmax 门控 + top-k 稀疏路由。届时门控不再让每个专家都跑,而是离散地「选 k 个」——`argmax` 不可导,梯度怎么回到门控?这是稀疏 MoE 能不能训起来的命门,也是本章故意隔离掉、留给下一章单独攻克的问题。
