> 第 03 章你刚把不可导的 argmax 路由"骗"过梯度——门控权重 `g_e` 乘在专家输出上,梯度从 loss 沿着这个标量权重回流到 router。通了。但这条通路本身埋着一颗定时炸弹:它是一个**正反馈回路**(positive feedback loop,自我放大的循环)。本章就是来拆这颗炸弹的,而且这一次我们有真实运行的数字撑腰。

## 先看炸弹怎么响:门控的正反馈

把第 03 章那条梯度通路在脑子里跑两步。

某个 batch 里,专家 3 碰巧权重略高、被多选了几次。被选中意味着它的 FFN(前馈网络)参数收到梯度、被更新、于是对这批 token 拟合得更好。下一步,router 看到专家 3 在这些 token 上 loss 更低,于是给它更高的 gate logit(门控打分)。logit 更高 → 被选更多 → 训练更多 → 拟合更好 → logit 更高。

这是个标准的**富者愈富**(rich-get-richer)动力学。没有任何外力时,它的不动点(fixed point,系统稳定下来的状态)不是"每个专家分到 1/E"的均匀态,而是少数专家吃掉绝大部分 token、其余专家几乎不被激活的退化态。这不是抽象担忧——配套仓 `stage04` 不加任何均衡、E=8 的 toy 跑出来就是这个样子:

```
[no-balance]  CV=0.714
E2 │······························│ 0.0000   ← 死专家
E5 │······························│ 0.0000   ← 死专家
E7 │██████████████████████████████│ 0.2667   ← 通吃
```

八个专家里有两个利用率精确为 0——它们一个 token 都没分到,从训练第一步到最后一步都在睡觉。退化到极致就是**专家坍塌**(expert collapse)——router 退化成一个近乎常数的函数,这是第 06 章的主题。本章先处理它的温和版本:**分布不均**。

为什么这是个严重问题,而不是"看起来不公平而已"?这正是本章第一个专家钩子要钉死的判断。

✦ **专家钩子:负载均衡不是为了公平好看,是为了不浪费参数容量。** MoE 的整个卖点(第 01 章)是"参数量大但每 token 只激活 k/E"——你为了 E 个专家付了 E 份参数的显存和训练算力。上面那两个利用率为 0 的专家就是**死参数**(dead parameters):占着显存、白白参与了初始化,却从不贡献前向计算,也几乎收不到梯度。你花 8 个专家的钱,实际只跑了 6 个。坍塌的专家 = 烧掉的预算。均衡损失的 KPI 不是基尼系数好看,是"每一份你付过钱的容量都在干活"。

## 怎么量"不均":利用率与变异系数

讲机制前先把度量钉死,否则"压平了"就是一句没有数字支撑的 survey 话。本书把所有 MoE 度量集中在配套仓的 `examples/moe-from-scratch/src/core/metrics.ts`,这一章用到三个。

第一个是**专家利用率**(expert utilization)——每个专家分到了多少比例的 token:

```ts
// examples/moe-from-scratch/src/core/metrics.ts
export function expertUtilization(assignments: number[], E: number): number[] {
  const counts = new Array(E).fill(0);
  for (const a of assignments) {
    if (a < 0 || a >= E) throw new Error(`expertUtilization: assignment ${a} outside [0,${E})`);
    counts[a]++;
  }
  const n = assignments.length || 1;
  return counts.map((c) => c / n);
}
```

`assignments[i]` 是第 i 个 token 被路由到的专家 id(top-1)。返回一个长度 E、和为 1 的数组。健康态是每项约等于 `1/E`(E=8 时是 0.125),某项逼近 1.0 就是坍塌。注意那个 `throw`——越界的专家 id 直接抛错,而不是静默丢弃。这是刻意的:静默丢一个越界 id 会把一个路由 bug 藏起来,让利用率看起来正常,实际上你的 dispatch 逻辑已经坏了。**度量代码宁可炸,不可撒谎**,这是本书所有 instrument 的共同原则。

利用率是个向量,不方便当训练监控的单一标量。压成一个数就是**变异系数**(coefficient of variation, CV,标准差除以均值):

```ts
// examples/moe-from-scratch/src/core/metrics.ts
export function coefficientOfVariation(util: number[]): number {
  // ... mean, then std, then:
  return std / mean;
}
```

为什么用 CV 而不用裸标准差?因为 CV 是**无量纲的**(scale-free):它在不同 E、不同 token 数之间可比。完美均衡时 CV = 0;少数专家通吃时 CV 飙高。本章你会看到它被均衡损失压下来——从 0.714 一路到 0.067。这也是第 06 章的头号指标。

> 一句话区分两个常被混淆的概念:**利用率分布不均**(本章)是软问题——大家都在跑,只是有的跑得多;**单专家被挤爆**(第 05 章的容量丢弃)是硬问题——某专家收到的 token 超过它的容量上限,溢出的 token 被直接丢弃,梯度都没了。前者用 aux loss 这种软约束治,后者必须靠 capacity factor + drop 这种硬机制治。两者正交:你可以分布很均匀但仍然在尖峰 batch 上挤爆容量,也可以分布很偏但每个专家都没到上限。下一章会专门拆这个区别。

## 核心机制:Switch/GShard 的 auxiliary loss

现在拆炸弹。思路:既然正反馈会把分布推向极端,就在主 loss(语言模型的交叉熵)旁边加一项**辅助损失**(auxiliary loss),它在"分布不均"时变大,从而给 router 一个反向的梯度推力,把分布往均匀拽。

最被广泛采用的形式来自 GShard 和 Switch Transformer(Google, 2020/2021)。配套仓的实现:

```ts
// examples/moe-from-scratch/src/core/metrics.ts
//   aux = E * sum_e ( f_e * P_e )
export function loadBalanceLoss(gateProbsPerToken: Value[], assignments: number[], E: number): Value {
  const T = gateProbsPerToken.length;
  if (T === 0) throw new Error("loadBalanceLoss: no tokens");
  // f_e: hard fraction dispatched to each expert — a CONSTANT (no grad).
  const f = expertUtilization(assignments, E); // numbers
  // P_e: mean gate probability on expert e — a Value (carries grad).
  let acc: Value = Value.zeros(1, E);
  for (const g of gateProbsPerToken) acc = acc.add(g);
  const meanP = acc.mul(Value.scalar(1 / T)); // [1,E], P_e
  const fRow = Value.from(f, [1, E]); // constant leaf; we never backprop into it (it's f_e)
  const weighted = meanP.mul(fRow);   // [1,E], grad flows only into meanP path
  return weighted.sum().mul(Value.scalar(E));
}
```

公式就一行:`aux = E · Σ_e (f_e · P_e)`。拆开两个量:

- **f_e**:专家 e 实际分到的 token 比例,就是上面 `expertUtilization` 的输出。它是一个**硬计数**(hard count)。
- **P_e**:router 给专家 e 的**平均门控概率**(对 batch 里所有 token 的 softmax 概率取均值)。这是个软的、连续的量。

为什么是这两个量的点积(dot product),而不是别的?直觉是这样:`f_e · P_e` 这一项,只有当专家 e **既被频繁选中**(f_e 大)**又被分配了高概率质量**(P_e 大)时才会很大。最小化它,等于逼 router 不要把"高频选中"和"高概率"同时集中在少数专家身上——也就是把概率质量摊开。前面那个 `E` 系数是个归一化技巧:完美均衡时 f_e = P_e = 1/E,代入得 `aux = E · E · (1/E · 1/E) = 1.0`。所以打印出来的 aux 值可以直接读成"偏离均匀的倍数",1.0 是地板。`stage04` 把这个 aux 值逐步打出来,你能直接看到它从 1.425 往 1.0 收:

```
[aux loss (1.00)]  aux 1.425 -> 1.004
  █▂▁▁▁▁▁▂▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁▁
```

起点 1.425(不均),几步内压到 ~1.0(地板)并钉死在那儿。弱权重(0.01)只把它从 1.425 拉到 1.195——压了,但没压到底,这跟下面 CV 的压制力度完全对得上。

### 为什么 f_e 必须断开梯度——一个经典 bug

这是整个机制最容易写错、也最值得讲透的一步。看代码里这个细节:`f` 是普通 number 数组,`fRow = Value.from(f, ...)` 是一个**常量 leaf**(不参与反向传播的叶子节点);只有 `meanP` 是带梯度的 `Value`。梯度**只**从 P_e 这条路回流,f_e 那条路是断的。

为什么必须这样?因为 **f_e 本质不可导**。它来自 `assignments`,而 assignments 是 argmax 的结果(第 03 章讲过 argmax 不可导)。如果你头脑发热把 f_e 也做成带梯度的 `Value`,autograd 会试图对一个"计数"求导——这在数学上没意义,在实现上会让梯度沿着一条假通路乱流,训练直接发散或者得到无意义的解。

GShard 的精髓正是这个"一软一硬"的设计:**用可导的 P_e 当梯度的载体,用不可导的 f_e 当不带梯度的权重**。两者相乘,梯度只走软的那条。配套仓特意把这一点编码进了函数签名——`gateProbsPerToken: Value[]`(软,带梯度)和 `assignments: number[]`(硬,纯数字)分两个参数传,类型系统就帮你挡住了"把 f_e 做成 Value"这个 bug。代码注释里原话:"Getting this wrong (making f_e a Value) is a classic bug that this signature prevents."

> 这是本书反 survey 的一个缩影:survey 稿会写"aux loss = E·Σf·P,加到主 loss 上即可";它不会告诉你 f 和 P 哪个带梯度、写反了会怎样、为什么要从类型上防住。而这恰恰是你真上手时唯一会卡住的地方。

### 加进训练:一行加法

辅助损失算出来之后,加到主 loss 上,乘一个权重系数 `α`。`stage04` 的训练循环里就是这一行:

```ts
// examples/moe-from-scratch/src/stage04-load-balance-loss.ts (trainCondition 内)
if (balance.kind === "aux") {
  const aux = loadBalanceLoss(gateProbsPerToken, assignments, E); // scalar Value, grad into probs
  auxValue = aux.data[0];
  total = total.add(aux.mul(Value.scalar(balance.weight)));        // total = ce + α·aux
}
```

因为 `aux` 是个带梯度的 `Value`,它自然会通过 `meanP` → 各 token 的 `softmaxRow` 输出 → router 的 gate logit → router 权重,把"该均衡"的信号一路反传回去。你不需要手写任何额外的梯度逻辑,autograd 全包了——这正是第 02/03 章把门控 softmax 接进 autograd 图的回报。

## 实测:同一个 router,五种均衡方式

讲到这儿可以兑现"实测"了。`stage04` 的实验设计很克制,但克制得有道理:它把**同一份数据(DATA_SEED=7)、同一份初始化(INIT_SEED=101)** 的 router 重建五遍,每遍只换均衡机制,别的全锁死。这样五个条件之间的任何差异,都只能归因于均衡机制本身,不是运气。

数据是 240 个点、6 维、8 个可分的高斯簇(`makeClusters`),模型是 E=8、top-2、每个专家直接吐 8 类 logit,训 400 步。先看利用率 CV 这一刀切下去的五个结果:

| 条件 | CV(实测) | 一句话 |
|------|-----------|--------|
| no-balance | **0.714** | 2 个死专家,1 个吃 27% |
| aux loss (0.01) | **0.518** | 弱权重,死专家被激活,但只压了一截 |
| bias-based | **0.284** | 不进梯度的旁路 bias,压得比弱 aux 还狠 |
| aux loss (1.00) | **0.067** | 强权重,几乎压平到完美均衡 |
| entropy-pen 0.3 | 0.807 | 这条是反派,下面单讲 |

前四行就是均衡机制的全部价值:CV 单调被压下来,而且——这是关键——**几乎不花性能的钱**。`stage04` 同时测了主任务 accuracy:

```
no-balance       acc=100.0%   final task loss=0.0043
aux loss (0.01)  acc=100.0%   final task loss=0.0044
bias-based       acc=100.0%   final task loss=0.0042
aux loss (1.00)  acc=100.0%   final task loss=0.0067
```

CV 从 0.714 砍到 0.067、把死专家全激活,accuracy 纹丝不动(100% → 100%),任务 loss 只从 0.0043 微涨到 0.0067。**健康的均衡近乎免费。** 这是 demo 想让你建立的第一个直觉。

> 诚实声明(也是配套仓 README 的硬约定):这是 toy 合成数据,8 个干净可分的高斯簇,accuracy 绝对值偏乐观。**能迁移的不是"100% 精度",是各条件之间的相对关系**——CV 排序、均衡近乎免费、强弱 aux 的压制力度差异。绝对数字别外推到真实 LLM,机制和趋势可以。

## ⚡ 前沿:auxiliary-loss-free 均衡(DeepSeek-V3)

注意上表第三行 `bias-based`:CV 0.284,压得比弱 aux(0.518)还狠,而它**没往 loss 里加任何项**。这就是 2024 年的那个重要简化。

aux loss 有个挥之不去的副作用:它**往主 loss 的梯度里掺了私货**。router 收到的梯度是"既要拟合任务、又要均衡"两个目标的混合,这两个目标并不总是同向——均衡那一项会污染、稀释任务那一项的梯度。

⚡ **DeepSeek-V3(2024)提出了 auxiliary-loss-free load balancing——干脆不要这个辅助损失项。** 做法是给每个专家维护一个可动态调整的**偏置项**(bias term)`b_e`:路由选 top-k 时,用 `logit_e + b_e` 来选,但 gate 权重仍然只用原始的 `softmax(logit_e)`。训练中观察各专家的负载:某专家被选太多,就调低它的 b_e(让它在 top-k 竞争里吃亏);被选太少,就调高 b_e。

`stage04` 把这个对照做成了真实可跑的实现,不是伪代码。先看模型里 bias 怎么参与路由——核心是**选谁用带 bias 的 logit,但 combine 权重用不带 bias 的原始 softmax**:

```ts
// examples/moe-from-scratch/src/stage04-load-balance-loss.ts (MoEClassifier.forward 内)
// Top-K selection on BIASED logits (the load-steering hint):
for (let e = 0; e < E; e++) scores.push({ e, s: gateLogits.data[e] + this.gateBias[e] });
scores.sort((a, b) => b.s - a.s);
const chosen = scores.slice(0, K).map((c) => c.e);
// ...but combine weights use the UNBIASED gate softmax restricted to chosen experts:
const w = gateProbs.gather(e).mul(Value.scalar(1 / denom));
```

这一步的不变量值得钉死(代码注释也明说了):**bias 是个负载导向的提示,不是价值判断**。让它进 combine 权重就会污染模型输出,等于白用这个 trick。所以 bias 只决定"选谁",权重永远走原始门控的真实信念。

再看 bias 怎么更新——纯规则纠偏,优化器的梯度根本看不见它:

```ts
// examples/moe-from-scratch/src/stage04-load-balance-loss.ts (trainCondition 内)
if (balance.kind === "bias") {
  const util = expertUtilization(assignments, E);
  const target = 1 / E;
  for (let e = 0; e < E; e++) model.gateBias[e] += balance.rate * (target - util[e]);
}
```

某专家利用率高于 `1/E`(超载)→ `(target - util[e])` 为负 → 调低它的 bias → 下一步它在 top-k 竞争里吃亏。这就是个旁路控制器(像个 PID 调节器)。`gateBias` 在代码里刻意定义成 `Float64Array` 而不是 `Value`——它从设计上就不在 autograd 图里。

关键在于:**b_e 只影响"选谁",不进梯度、不进主 loss**。主 loss 的梯度因此保持干净——router 收到的梯度只关心任务,均衡完全由 bias 这条独立通路负责。这就绕开了 α 调参和梯度污染两个麻烦。代价是引入了 bias 的更新速率(`rate`,实测用 0.1)这个新旋钮,但它直接作用在"选频率"上,不经过梯度的二阶传导,行为比 aux loss 的 α 更可预测。

这个方向是**正在被验证的现役方案,不是已经盖棺定论的通用解**——它在 DeepSeek-V3 这种超大规模、特定专家数和路由配置下验证过,但"对所有 MoE 规模和任务都优于 aux loss"还没有定论;aux loss 仍是最广泛部署、最被理解的 baseline。两条路线目前并存。把 `stage04` 里 aux 版和 bias 版并排读,你就摸到了 2020(GShard)到 2024(DeepSeek-V3)这条线上,工程界对"如何不污染主 loss 还能均衡"的认识是怎么演进的:**aux 版把均衡压力编码进梯度(P_e 带梯度),bias 版把均衡压力踢出梯度、放进旁路规则(bias 是纯 number)。** 同一个目标,两个完全不同的施力点。

## aux loss 权重 α:微妙超参,但要小心一个流行误解

✦ **专家钩子(续):α 是个微妙超参——太小压不住,太大……到底会怎样?** 这是 aux loss 最值得资深工程师警惕、也最容易被 survey 稿讲错的地方。

`α 太小`的那一半是对的,实测也支持:α=0.01 时 aux 值只从 1.425 压到 1.195,CV 只从 0.714 降到 0.518——压了,但远没压到底。辅助损失的梯度推力不够,正反馈还在拉扯。

但流行的另一半说法——"`α 太大`会逼 router 学出**均匀但与语义无关**的随机路由、抹平专家分工"——`stage04` 把 α 拉到 1.0 实测,结果**不支持这个直觉**,而这正是 demo 想纠的一个误解。看路由熵(routing entropy,衡量路由"犹豫"程度,越高越接近随机)和专家分工纯度(purity,每个专家的 token 有多集中在单一簇上):

```
                 entropy(占满分 ln8 的比例)    purity
aux loss (0.01)  0.573 (28%)                  89.3%
aux loss (1.00)  0.775 (37%)                  85.9%
```

α 从 0.01 拉到 1.0(放大 100 倍),熵只从 0.573 微升到 0.775(离满分 ln8 ≈ 2.08 还远),purity 几乎没掉,accuracy 仍 100%。**强 aux 没有把路由变随机。**

为什么?这是本章最该被记住的一句机制:**Switch/GShard 的 aux loss 只约束「平均负载」,不约束「单 token 路由」。** 它要的是"每个专家平均下来分到 1/E",而这个目标完全可以由"每个 token 仍然去对的专家、只是不同 token 去不同专家、平均起来均匀"来满足。它根本没有逼任何单个 token 走随机路由的能力。所以把它拉到 1.0,它能压平 CV,却不会randomize 路由、也不掉 accuracy。"α 太大 = 路由变随机"是个把 aux loss 的作用对象搞错了的误解。

### 那"过度均衡是一种病"还成立吗?成立,但得用另一把刀逼出来

`stage04` 里那条反派 `entropy-pen 0.3` 就是专门来演真正的过度均衡的。它不用 aux loss,而是**显式地往 loss 里加一项最大化逐 token 熵**的惩罚——这才是有能力直接 randomize 单个 token 路由的东西:

```ts
// examples/moe-from-scratch/src/stage04-load-balance-loss.ts
// ADD -weight * mean entropy: gradient descent then MAXIMIZES entropy (uniform routing).
const meanEnt = (entropyAcc as Value).mul(Value.scalar(1 / order.length));
total = total.add(meanEnt.mul(Value.scalar(-balance.weight)));
```

结果就是教科书里那个"过度均衡":

```
entropy-pen 0.3   entropy=1.900 (91% of max)   purity=74.4%   acc=99.6%
```

熵冲到 ln8 的 91%(确实逼近随机路由),purity 从 no-balance 的 84.2% 掉到 74.4%(专家不再按簇分工)。**这才是真正的过度均衡的指纹:不是 CV 低,是熵爆高 + 分工纯度塌。**(注意它的 CV 反而是 0.807,比 no-balance 还高——证明 CV 一个指标根本抓不住这种病,看 CV 全绿你都不知道路由已经废了。)

但 demo 在这里又诚实地踩了一脚刹车:即便路由近乎随机,这个 toy 的 accuracy 仍有 99.6%——几乎没掉。**为什么?** 因为这 8 个簇太干净可分了,每个专家被迫看遍所有簇之后,各自学成了一个能分全部 8 类的通才,拼起来照样对。教科书里"过度均衡 → accuracy 暴跌"那一刀,需要一个**负载均匀和正确路由相互冲突**的任务才砍得出来——真实 LLM 的 token 异质、强依赖专家分工,那里逼出随机路由就是直接掉点;干净高斯簇没有这种张力。**这点 demo 不假装有,你也别把 toy 的 accuracy 当真实模型的结论。**

所以本章必须诚实交代的判断是:过度均衡确实是一种病,但**它不是"aux loss 权重调大"导致的**——aux loss 由于只管平均负载,天然压不出随机路由。真正逼出随机路由的是直接惩罚决断性(熵惩罚那一类)的机制。把这两者混为一谈("α 调大就会过度均衡")是个流行但错误的因果。第 06 章会专门展开:坍塌(熵塌、CV 高)和过度均衡(熵爆、purity 塌)是天平两端,而 CV 一个指标只能看见前者——判断路由有没有学到真分工,必须 CV + 路由熵 + ground-truth 分组三个一起看。

> 调 α 的实操建议:从一个小值起步(真实系统里 Switch Transformer 论文用的量级就是 1e-2),盯着 CV 和主任务 loss 一起看——CV 在降说明压住了,任务 loss 没崩说明没压过头。但别因为"怕过度均衡"就不敢调大 α:实测告诉你 aux loss 本身把权重拉到 1.0 也不会 randomize 路由。真正要防的是你别另外引入显式的熵最大化项还不自知。

## 小结与下一章

本章把第 02/03 章埋下的正反馈炸弹拆了,而且这次每个判断都有 `stage04` 的真实数字撑着:

- 门控的正反馈(谁强谁被选更多)会把利用率推向坍塌——实测 no-balance 下 8 个专家 2 个利用率精确为 0,死参数 = 烧掉的预算。
- Switch/GShard 的 aux loss = `E·Σ f_e·P_e`:f_e 硬计数不带梯度、P_e 软概率带梯度,相乘后梯度只走软路——这是机制的命门,写反就发散,配套仓用函数签名把它防死。
- 实测 CV:no-balance 0.714 → 弱 aux(0.01) 0.518 → 强 aux(1.0) 0.067,accuracy 全程 100%。**健康的均衡近乎免费**(toy 数字偏乐观,迁移的是相对关系)。
- ⚡ DeepSeek-V3 的 auxiliary-loss-free 方案用旁路 bias 纠偏(实测 CV 0.284),不污染主 loss 梯度,是 2024 年的重要简化、现役但未盖棺;`stage04` 附了真实可跑的对照实现。
- 纠一个流行误解:**aux loss 权重调大不会逼出随机路由**——它只约束平均负载,不约束单 token 路由。真正的过度均衡(熵爆 91% of max、purity 从 84% 塌到 74%)要靠显式熵惩罚才逼得出,留给第 06 章。

但本章只解决了"分布不均"这个**软问题**。还有一个**硬问题**没碰:即便分布整体均匀,某个尖峰 batch 仍可能让单个专家收到的 token 超过它的物理容量上限,溢出的 token 会被**直接丢弃**——连参与计算的机会都没有。这是分布式 MoE 绕不开的工程现实。第 05 章《容量丢弃》会拆这个机制:capacity factor 怎么设、drop rate 怎么实测、丢掉的 token 对训练意味着什么。记住本章末尾那句区分:**aux loss 治分布形状,capacity 治单专家上限,两者正交,别用一个去顶另一个。**
