面试里问"你们为什么不直接全量微调",一半人答得出"省显存",但答不出量级——省多少、省在哪一项、代价是什么。这章的目的就是把"代价"变成你能脱口而出的数字:全量微调 = 100% 参数可训练,Adam 把梯度加优化器状态推到训练显存的 60%,而且学会新任务 B 的同时把旧任务 A 从 100% 准确率砸到 12%。后面 LoRA / QLoRA / Adapter 每一章都在拆这两堵墙,不先把墙的高度量出来,后面的"节省"就是空话。

第 01 章我们把一个 toy Transformer 在任务 A(copy,原样复制输入序列)上训到 100% 准确率,并把它快照成共享基座(checkpoint)。本章直接拿这个基座,用教科书最朴素的做法去解决一个**冲突任务** B,看它要付多少代价。代码全在 `examples/peft-from-scratch/src/stage02-fullft.ts`,数字都可以用 `npx tsx src/stage02-fullft.ts` 复现。

## 先把任务设计讲清楚:为什么 B 是 vocab 重映射

任务 B 不是随便选的。它是"逐 token 的固定词表置换"(vocab relabeling,大白话:把每个 token 按一张固定查找表换成另一个 token),`targetB[i] = VOCAB_PERM[input[i]]`,本次运行 `perm=[1,2,3,7,6,5,4,0]`。

为什么不用"反转""排序"这类更直观的任务?因为本书的核(`core/nn.ts`)**没有位置编码**(positional encoding,让模型知道 token 在第几位的信号)。反转/排序是纯位置变换,toy 模型物理上学不会,准确率会卡在随机基线附近——那样"全量微调学会了 B"就成了谎话。代码注释把这点说死了:

```typescript
// WHY task B is a vocab RELABELING of copy, not the data module's default copy->reverse:
//   reverse/sort are PURELY POSITIONAL transforms, but the core has no positional encoding
//   (see nn.ts), so a toy model literally cannot learn them past ~chance — which would make
//   "full FT masters task B" a lie. A fixed permutation of the vocab (target[i]=perm[in[i]])
//   is TOKEN-LOCAL (learnable to ~100%) yet maximally CONFLICTS with copy (every token
//   remaps), so it is the honest substrate for "learns B / forgets A".
```

关键词是 **token-local 且最大冲突**。每个 token 都被改写(`perm` 是全打乱,没有一个 token 映射到自己),所以学会 B 的映射**必然**覆盖掉 copy 的"原样输出"映射。这是本章想要的诚实最坏情形:遗忘不是偶然漏掉,是结构上无法避免——A 和 B 对同一组权重提出了互相矛盾的要求。这点很重要,后面讲 PEFT 缓解遗忘时,缓解的正是这种"覆盖",而不是某种巧合。

词表 vocab=8、序列长 seqLen=6,随机瞎猜的基线准确率 = 1/8 = 13%(代码里 `chance = 1 / VOCAB`)。后面所有"崩坏到随机基线"的判断都以这个 13% 为锚。

## 第一堵墙:100% 可训练,以及它在显存上意味着什么

全量微调在代码里就是一个开关。微调函数 `fullFinetune()` 调 `loadBase(model, baseCkpt, false)`,第三个参数 `freeze=false` 就是"全量"的全部含义——把基座权重拷进来,但所有参数保持可训练:

```typescript
// freeze=false => FULL fine-tuning: copy base weights but keep them all trainable.
loadBase(model, baseCkpt, false);
const opt = new Adam(model.trainable(), lr);
```

`model.trainable()` 来自 `core/nn.ts`,它的实现就一行,且它是"可训练比例"这个诚实指标的唯一真源:

```typescript
/** Only the leaves the optimizer is allowed to move. */
trainable(): Tensor[] {
  return this.parameters().filter((p) => p.requires_grad);
}
```

冻结一个参数的机制是把它的 `Tensor.requires_grad` 翻成 `false`——没有单独的"冻结清单"会和实际状态漂移,真源就是 tensor 上那个标志位。全量微调时没人翻它,于是:

```
可训练参数 = 2488 / 2488 = 100.0%  ← 全量微调: 100% 参数都在动 (成本墙的顶点)
```

100% 这个比例就是成本墙的顶点。为什么它直接决定显存?看 `core/mem.ts` 的训练显存公式(纯算术,不分配任何真实内存):

```typescript
const weightsBytes = frozen * baseBpp + cfg.trainableParams * bpp;
const gradBytes = cfg.trainableParams * bpp;          // 只有可训练参数有梯度
const optimStateBytes = cfg.trainableParams * bpp * optMul;  // Adam: optMul=2 (m 和 v)
```

三项里有**两项**(梯度、优化器状态)只跟 `trainableParams` 走。Adam(自适应学习率优化器)给每个可训练参数都存两份状态:一阶动量 `m` 和二阶动量 `v`。`core/optim.ts` 里能看到它确实是按需为每个可训练参数 lazy 分配两条 `Float64Array`:

```typescript
if (!p.requires_grad) continue; // GUARANTEE 1
let mt = this.m.get(p);
let vt = this.v.get(p);
if (!mt || !vt) {
  mt = new Float64Array(p.size);   // 一阶动量 m
  vt = new Float64Array(p.size);   // 二阶动量 v
}
```

注意第一行 `if (!p.requires_grad) continue`:冻结参数连状态都不分配。这一行就是整个 PEFT 显存故事的引擎——后面 LoRA 把 99% 的参数冻住,这个 `continue` 就让 99% 的 `m/v` 分配根本不发生。

把公式套到本章 stage 的实测拆解上(est. = 估算,纯公式,不是实测 VRAM):

```
full-FT 拆解 (bytes, est.): weights=9952 grad=9952 adam(m,v)=19904 act=9952
full-FT (100% trainable) │████████████████████████████████████████│ 0.047  MB (est.)
PEFT-shaped (~0.5%)      │████████████████························│ 0.019  MB (est., 对照: 后续章节实现)
成本墙在哪: full-FT 的 grad+Adam state = 60% 的训练显存, 且随可训练参数线性增长。
```

读这几个数:权重本身只占 9952 字节,但梯度又一份 9952、Adam 的 m+v 再来 19904——光优化器侧就是权重的 3 倍。**梯度加优化器状态 = 60% 的训练显存**。这就是"全量微调贵"的精确含义:你不是为模型本身付钱,你主要在为"让全部参数都可学"配套的梯度和优化器状态付钱,而这部分随可训练参数**线性增长**。

> ⚠ toy-scale 声明:绝对值(0.047 MB)毫无意义,且偏乐观——真实场景里激活值(activation)项会因为 batch/序列长度膨胀得多,本公式里 `actFactor` 默认只取 1。可迁移的不是 MB 数,而是三件事:(1)100% 可训练这个比例;(2)梯度+优化器状态 ≈ 权重的 3 倍这个结构;(3)它随可训练参数线性缩放。换算成真实的 7B 模型,fp32 权重 28GB、梯度 28GB、Adam 状态 56GB,光训练侧就 112GB——这正是单卡放不下 7B 全量微调的来源。

## 第二堵墙:灾难性遗忘,B 上去了 A 塌了

显存是钱的问题,遗忘是能不能用的问题,后者更致命。全量微调把任务 B 学得很好:

```
任务 B 准确率: 14.6% (起点) -> 100.0% (学会 B)
```

起点 14.6% 几乎就是随机基线(基座没见过 B),120 步后到 100%——全量微调确实有能力把新任务学到满。问题在同一时刻回测旧任务 A:

```
任务 A 准确率: 100.0% (起点) -> 11.7% (微调后回测 → 灾难性遗忘量化)
遗忘量 (A 准确率下降) = 88.3 个百分点
```

A 从 100% 崩到 11.7%,比 13% 的随机基线还低——**任务 A 被彻底抹掉了**。这就是灾难性遗忘(catastrophic forgetting,大白话:学新的把旧的忘光)。本章把它做成一张随步数变化的交叉表,让你看到两条线一升一降地交叉:

```
A/B 准确率随步数 (moderate lr) — A 下滑 / B 上升
step │ taskA(old)  taskB(new)  bars (A=▒  B=█)
   0 │   100%        15%     A:▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒▒
     │                        B:███
  10 │    12%       100%     A:▒▒
     │                        B:████████████████████
  ... (每 10 步一行, 全程 A≈12% / B=100%) ...
 120 │    12%       100%     A:▒▒
     │                        B:████████████████████
```

注意崩塌发生得多快:**第 10 步**,B 已经 100%、A 已经 12%。不是渐变,是在头几个 epoch 内骤崩,之后稳定在"A 死透、B 满分"。

### 为什么必然遗忘——这不是 bug

回到任务设计。`VOCAB_PERM` 是全打乱,每个 token 都要换成不同的 token。任务 A 要求 token `3` 输出 `3`,任务 B 要求 token `3` 输出 `7`(perm 第 4 位)。同一组权重不可能同时满足"输出 3"和"输出 7"。代码注释把这条因果链写死了:

```typescript
//   WHY a full shuffle (every token remaps): maximizes the conflict with copy, so learning B
//   NECESSARILY overwrites the copy mapping => forgetting is total, not incidental. That is
//   the honest worst case the chapter wants: even a perfectly-learned B costs you all of A.
```

关键词 **NECESSARILY overwrites**。这不是优化器没调好,也不是步数不够——是任务在权重空间里直接冲突,全量微调允许每个权重自由移动去满足 B,移动的代价就是 A 编码的映射被覆写。这正解释了为什么"准确度去到比随机还低":权重不是回到随机初始化,而是被 B 主动推到了与 A 相反的位置。

这条因果链直接给出 PEFT 缓解遗忘的机制预告:**如果基座冻结**(`requires_grad=false`),A 编码在基座里的权重一个字节都不能动,B 只能挤进一个独立的小增量(LoRA 的低秩矩阵 / Adapter 的旁路层)。基座原样保留 = A 原样保留。这不是 PEFT 顺带的好处,是它"只动增量、不动基座"这个结构的直接推论。第 03 章会让你看到:同样冻结这份基座,LoRA 在新任务上学到东西的同时,关掉 adapter 回测任务 A 仍稳在 100%——遗忘几乎为零,对照本章这 88.3 个百分点,就是对照基线存在的意义。

> ⚡ 前沿钩子:灾难性遗忘是 continual learning(持续学习,大白话:让模型边用边学新任务还不忘旧的)和多任务部署的**核心未解难题,目前没有通用解**。学术界有 EWC(给重要权重加正则别让它动太多)、replay(混入旧数据一起训)、参数隔离(给每个任务划独立参数)等路线,但每条都有明确代价:EWC 需要估计每个权重的"重要性"且对任务序列敏感、replay 要存旧数据(隐私/存储成本)、参数隔离随任务数线性膨胀。没有哪个方案能在"不存旧数据、不预知任务序列、参数不膨胀"三个约束下同时成立。PEFT 之所以流行,一半原因正是它**天然侧写式地缓解了遗忘**——冻结基座等于把旧能力锁进只读区,代价只是新任务的表达能力受限于增量的容量。本章的 88.3 个百分点遗忘量,就是衡量这个问题严重性的可观测证据,也是评估任何 PEFT 方法"保留旧能力"时的对照值。

## 第三堵墙(隐藏的):全量微调对学习率更脆弱

前面用的是中等学习率(lr=0.005)。有人会想:既然要学 B,把学习率调大点学得更快?stage 里专门跑了一个 6 倍学习率(lr=0.03)的失败模式,而且这两次运行**只有学习率一个变量不同**——同一个冻结基座、同一个 PRNG 种子重新载入,代码注释强调了这个隔离:

```typescript
// WHY re-load + re-seed inside: each run must start from the IDENTICAL frozen base and the
//   identical PRNG state, so lr is the only difference between the moderate and large runs.
```

结果不是"学得更快",是两个任务一起输:

```
失败模式: lr 偏大 (lr=0.03, 6x)
任务 B 准确率: 14.6% -> 11.2%  (对比 moderate lr 的 100%)
任务 A 准确率: 100.0% -> 16.7%  (随机基线=13%)
large-lr 下 A 的最低准确率 = 11.7% (随机基线=13%, 即 A 已彻底崩坏)
诊断: lr 偏大 = 训练失稳, B 没学会 (11%) 且 A 仍崩坏 — 两个任务一起输, 比 moderate 更糟。
```

中等 lr 至少是"学会 B、丢掉 A"的明确权衡;大 lr 是训练失稳,B 卡在 11%(随机基线附近)根本没学会,A 也照样崩。交叉表里两条线都趴在随机基线附近:

```
A/B 准确率随步数 (large lr) — 训练失稳, 两条线都趴在随机基线附近
step │ taskA(old)  taskB(new)
   0 │   100%        15%
  20 │    17%        11%
 ... (其余步全程 A≈17% / B≈11%, 均在随机基线附近) ...
 120 │    17%        11%
```

注意代码里这个"诊断"不是硬编码的结论,而是从实际数字推出来的——`lrgLearnedB = lrgBFinal >= 0.8` 判断这次大 lr 到底学没学会 B,再据此选打印哪句话。这是本书的诚实原则:不预设"大 lr 一定更糟",而是让数字自己说。本次运行的事实是它更糟。

为什么全量微调对 lr 更敏感?因为 100% 参数都在动,2488 个参数在一个高维空间里同时被一个偏大的步长推动,任何一步的过冲都会被后续步放大,整个网络的表征在崩。这给后面又埋了一个伏笔:PEFT 只训练极少量参数,优化的是一个低维得多的子空间,对学习率的鲁棒性通常更好——这同样是后续章节可以拿本章数字做对照的维度。

## 这章建立了什么

把三堵墙钉成对照基线:

- **可训练比例 100%**——成本墙的顶点。所有 PEFT 方法的第一个 KPI 就是"我把这个降到了多少 %",第 03 章 LoRA 的目标是 <1%。
- **梯度 + 优化器状态 = 60% 训练显存,随可训练参数线性增长**——省显存省的就是这一项,机制是 `core/optim.ts` 里那行 `if (!p.requires_grad) continue`。
- **灾难性遗忘 88.3 个百分点**——学会冲突任务 B 必然抹掉 A,因为全量微调允许覆写基座。PEFT 冻结基座 = 锁住旧能力,这是它缓解遗忘的结构性原因,不是巧合。
- **附带:全量微调对 lr 更脆弱**——大 lr 下两个任务一起输。

下一章(第 03 章 LoRA)用的是**同一份冻结基座、同样的 toy 设定(vocab=8、seqLen=6)**,但把任务 B 换成了 reverse(倒序)——一个同样需要在 copy 基座上做局部修正、却比 vocab 重映射更适合演示"低秩增量补一个小能力"的任务。真正可比的不是 B 的绝对准确率(任务不同),而是**遗忘维度**:本章全量微调把 A 从 100% 砸到 11.7%(掉 88.3 个百分点);第 03 章冻结基座、只塞一对低秩矩阵后,关掉 adapter 回测 A 还能不能稳在 100%?本章这个 88.3 个百分点,就是衡量"PEFT 到底缓解了多少遗忘"的标尺。
