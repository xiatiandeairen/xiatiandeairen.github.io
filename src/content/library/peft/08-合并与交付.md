先把这一章要证的话摆在桌面上:**"LoRA 能合并回权重,所以推理零开销"**——这句你在每篇综述里都见过,它是 LoRA 相对 Adapter(第 05 章那个串在 FFN 后的小瓶颈层)和 Prefix(第 06 章那串塞进 KV 的虚拟 token)的核心卖点。Adapter 永远在前向路径上多挂一层算子,Prefix 永远在每层注意力里多算一段;它们的额外推理开销没法消掉,只能优化。LoRA 不一样:它的增量是个纯线性的低秩乘积 `(α/r)·BA`,跟基座权重 `W` 同形状、同位置,训练完可以直接加进去,得到一份**和原模型字节级同构**的新权重——下游谁都看不出它被微调过,推理引擎、量化工具、KV cache 一行都不用改。

问题是,综述把这句话当公理写,从不证。本章不这么干。我们手写 `merge`,然后断言:合并前 `base + LoRA` 两条路径的前向输出,和合并后单条 `W'` 路径的前向输出,**逐元素之差落在 f64 浮点 round-off(舍入误差)地板内**。这是个可以被代码 `throw` 掉的硬断言,不是口头声称。配套代码在 `examples/peft-from-scratch/src/stage08-merge.ts`,这一章所有数字都来自它的真实运行,标 `⚠ toy-scale` 的地方是玩具规模乐观值,我会逐个点出哪些数能迁移、哪些不能。

## merge:把两块瘦矩阵折回一块方阵

回顾第 03 章的 LoRA 前向。注入点是一个 `Linear`,基座算 `y = x·Wᵀ + b`,LoRA 在旁边并联一条 `delta(x) = (α/r)·((x·Aᵀ)·Bᵀ)`,其中 `A` 是 `(r, d)`、`B` 是 `(d, r)`,`r` 是秩(low-rank 的那个"低")。两条相加:

```ts
const projBase = model.proj.forward(x);        // x·Wᵀ + b
const projFull = projBase.add(lora.delta(x, train)); // + (α/r)·((x·Aᵀ)·Bᵀ)
```

关键观察:`delta(x)` 对 `x` 是**纯线性**的(eval 态下,先按下不表的 dropout 关掉)。线性变换可以提取出一个等价的权重矩阵。把 `delta` 展开:

```
delta(x) = (α/r)·((x·Aᵀ)·Bᵀ) = x · (α/r)·(B·A)ᵀ
```

于是 `projFull = x·Wᵀ + b + x·((α/r)·BA)ᵀ = x·(W + (α/r)·BA)ᵀ + b`。也就是说,只要令 `W' = W + (α/r)·(B@A)`,合并后单条线性的输出就和 `base + LoRA` 两条路径**数学恒等**。`stage08-merge.ts` 里的 `mergeInto` 就这一行算术:

```ts
function mergeInto(proj: Linear, lora: LoraAdapter): void {
  const { data } = lora.weightDelta();   // (d,d) = (α/r)·(B@A),和 W 同形状
  for (let i = 0; i < proj.W.size; i++) proj.W.data[i] += data[i];
}
```

`weightDelta()` 算的是 `B@A`(注意是 `B@A` 不是 `A@B`):

```ts
weightDelta(): { data: Float64Array; shape: [number, number] } {
  const ba = this.B.matmul(this.A); // (d,r)@(r,d) -> (d,d)
  const out = new Float64Array(ba.size);
  for (let i = 0; i < ba.size; i++) out[i] = ba.data[i] * this.scale;
  return { data: out, shape: [ba.shape[0], ba.shape[1]] };
}
```

### 为什么是 B@A 不是 A@B——也是最经典的 merge bug

前向里算子的顺序是"先 A 后 B":`(x·Aᵀ)·Bᵀ`。把它折成单矩阵时,等价权重是 `(α/r)·(B·A)`,维度 `(d,r)@(r,d) = (d,d)`,正好和 `W` 同形。如果你顺手写成 `A@B`,维度 `(r,d)@(d,r) = (r,r)`,形状都不对(`r=2` 时是个 2×2),根本加不进 `W`;就算你硬 reshape 凑上去,数值也完全错。还有人忘了乘 `scale = α/r`,或者把某一边 transpose 错——这些 bug 的共同特征是:**它们不会让程序崩,只会让合并后的模型输出悄悄偏掉**。

这就是为什么本章的等价断言是"证明"而不是"装饰"。`stage08-merge.ts` 跑出来:

```
数学等价: max|merged_out - (base+LoRA)_out| = 9.44e-16  (阈值 < 1e-6)
✓ 逐元素差在 f64 round-off 地板内,合并前向与 base+LoRA 前向等价。
```

`9.44e-16` 是什么量级?是 f64 机器精度(约 `2.2e-16`)的几倍——这是**两条路径用不同运算顺序累加,浮点求和不可结合**留下的舍入残差,不是误差,是地板。一旦 `B@A` 写成 `A@B`、scale 漏乘、transpose 错位,这个差会瞬间跳到 `1e0` 甚至更高,直接撞穿 `1e-6` 阈值被 `throw`:

```ts
if (equivDiff >= 1e-6) throw new Error(`merge equivalence FAILED: diff ${equivDiff} >= 1e-6`);
```

`1e-6` 不是 `0`:要求 `0` 等于不懂浮点,会把正确实现也判挂。挑 `1e-6` 是因为它比 round-off 地板(`~1e-15`)宽九个数量级、却比任何真实 merge bug(`~1e0`)严十几个数量级——它把"对"和"错"清清楚楚地分在两侧。这个阈值哲学是**可迁移的**:你在 PyTorch / HuggingFace PEFT 里写 merge 验证,该用的也是 `torch.allclose(a, b, atol=1e-5)` 这一类相对宽松的容差,而不是 `==`。

## 零开销不是修辞,是数 matmul 数出来的

"零额外推理开销"这句话也常被当口号。`stage08-merge.ts` 把全局 `Tensor.matmul` 包了一层计数器,真的去数一次前向触发了几次矩阵乘:

```ts
let matmulCalls = 0;
const _rawMatmul = Tensor.prototype.matmul;
Tensor.prototype.matmul = function (other: Tensor): Tensor {
  matmulCalls += 1;
  return _rawMatmul.call(this, other);
};
```

合并前数 `base+LoRA` 路径、合并后数纯基座路径,结果:

```
推理 matmul 次数: base+LoRA = 4  ->  合并后 = 2  (回到基座水平,省下 2 次)
```

base 路径本身 2 次(proj 一次、head 一次),LoRA 分支多出 2 次(`x·Aᵀ` 一次、`·Bᵀ` 一次)。合并把这两次彻底吃掉,前向回到和**从没微调过的基座完全一样**的算子数。这正是 LoRA 区别于 Adapter/Prefix 的物理根源:Adapter 的瓶颈层、Prefix 的虚拟 KV 是**结构性的额外算子**,无论怎么优化都消不掉;LoRA 的增量是**可被吸收进既有算子的线性项**,合并后字面意义上不存在了。

交付侧还有一笔账:

```
合并后交付模型可训练参数 = 0  (基座全冻结,适配器已折进权重,无额外可训练量)
(注: 适配器对象本身仍有 2 个张量,但它们不再参与交付前向)
精度自洽: 合并前 acc=13.9%  合并后 acc=13.9%  (应相同)
```

合并后你交付的就是 `model` 本身——一份普通权重,可训练参数 0,适配器的 `A/B` 对象虽然还在内存里,但已经不在前向路径上了。精度合并前后都是 `13.9%`(toy reverse 任务的 token 准确率),这是输出逐元素等价的必然推论:输出一样,argmax 就一样,准确率就一样。这个 `13.9%` 绝对值是玩具规模、别当真;但"合并前后精度严格相等"这条结论与规模无关,是数学等价的直接后果。

## unmerge:merge 必须是可逆的,否则没法热切换

merge 把适配器永久焊进权重了——那多任务怎么办?你不可能为每个任务复制一份 7B 基座。答案是 unmerge:把刚才加进去的 `(α/r)·BA` 再精确减回来,基座复原:

```ts
function unmergeFrom(proj: Linear, lora: LoraAdapter): void {
  const { data } = lora.weightDelta();   // 用当前 A/B 重新算 delta
  for (let i = 0; i < proj.W.size; i++) proj.W.data[i] -= data[i];
}
```

注意它用**当前适配器权重重新计算** `weightDelta()`,而不是缓存 merge 时减过的值。因为适配器训完就冻住了,`B@A` 是确定的,重算和缓存等价;但重算让 `unmerge` 在代码意义上是 `merge` 的精确逆运算,不依赖任何外部快照。`stage08-merge.ts` 验证还原精度:

```
[2b] unmerge 还原基座: max|W_restored - W_base| = 4.44e-16 (unmerge 是 merge 的精确逆)
```

`4.44e-16`——又是 round-off 地板。`merge` 加了 `Δ`、`unmerge` 减了同一个 `Δ`,浮点加减不完全可逆(`(W+Δ)-Δ ≠ W` 严格成立的概率极低),残留的就是这点舍入。它远低于任何阈值,意味着你可以 `merge → 用 → unmerge → 再 merge 别的`,反复切换而基座**不漂移**。如果 unmerge 不精确,切换几十次后基座会被舍入误差啃出一个累积偏移——这就是为什么"可逆"必须验证到地板,而不是"看起来还原了"。

### 多适配器热切换:S-LoRA 的最小原型

有了精确的 merge/unmerge,多任务部署形态就出来了:**一份冻结基座 + N 个躺在硬盘上的小 LoRA**,请求来了按需 merge 上对应适配器、回测、unmerge 卸下,再 merge 下一个。`stage08-merge.ts` 训了第二个适配器(sort 任务)演示这个循环:

```ts
mergeInto(model.proj, loraB1);
const accB1Merged = tokenAccuracy((ids) => model.forwardIds(ids), dsA.finetune);
unmergeFrom(model.proj, loraB1);

mergeInto(model.proj, loraB2);
const accB2Merged = tokenAccuracy((ids) => model.forwardIds(ids), dsB2.finetune);
unmergeFrom(model.proj, loraB2);
```

跑出来:

```
[3] 第二个 LoRA 适配器 task=sort: loss -> 2.4972, token acc=22.6%
    切换演示 (同一冻结基座, merge->unmerge->merge):
reverse via merged adapter │██████████████████············│ 13.9  %
sort via merged adapter    │██████████████████████████████│ 22.6  %
    每个任务装上对应适配器后各自回测,基座始终是同一份 (切换无需复制基座)。
```

两个任务各自装上对应适配器都跑出了它该有的准确率(reverse `13.9%`、sort `22.6%`),而**基座自始至终是同一份内存**。这就是把"一个 7B 底座服务上千个客户/任务,每个客户一份几 MB 的 LoRA"这件事讲到了骨头上——你存的不是上千份模型,是一份模型加上千份增量。

> ⚡ **前沿钩子(目前无通用解 / 正在研究):** 本章演示的是**串行**切换——一次只 merge 一个,服务单一任务流。真实的多 LoRA 服务系统(代表作 S-LoRA,以及它催生的一类"一个基座同时挂上千 LoRA、请求级路由"的推理框架)要解的是更难的问题:**一个 batch 里不同请求要用不同适配器**,你没法把整份权重 merge 成某一个适配器——那会让同 batch 的其他请求用错增量。它们的做法是**保持 LoRA 不合并**,在 batched matmul 里为每个请求动态选取它自己的 `A/B`(所谓 heterogeneous batching),配合统一内存池管理上千份适配器的换入换出。这里的工程权衡至今没有定论:不合并就放弃了本章证明的"零额外算子"红利、回到 base+LoRA 的多 matmul,但换来请求级混合服务的能力;合并则零开销却只能服务单一适配器。**怎么在"按请求合并"(零开销但不能混 batch)和"始终不合并"(能混 batch 但有开销)之间取一个对工作负载最优的动态策略,是仍在研究的开放问题**——它高度依赖你的请求分布(适配器集中度、QPS、batch 组成),没有一刀切的答案。本章的串行切换演示,正是这整套系统最底层、被反复调用的那个 merge/unmerge 原语。

## 失败模式:dropout 开着的时候 merge 会爆

到这里 merge 看起来太顺了。它有个必须当心的前提:**只能在确定性的 eval 态下合并**。LoRA 训练时通常在输入侧挂 dropout(随机置零一部分激活,防过拟合),`stage08-merge.ts` 的 `delta` 也支持:

```ts
delta(x: Tensor, train: boolean): Tensor {
  const xd = this.dropoutP > 0 ? x.dropout(this.dropoutP, train) : x;
  return xd.matmul(this.A.transpose()).matmul(this.B.transpose()).scale(this.scale);
}
```

`train=true` 时 dropout 用随机 mask 把一部分激活置零,`delta(x)` **不再是 `x` 的确定性线性函数**——同一个 `x` 跑两次结果不同。而 merge 折进 `W'` 的是个**静态的、确定的** `(α/r)·BA`,它没有任何随机性可言。所以拿训练态(dropout 开)的输出去对比合并权重的输出,必然对不上。`stage08-merge.ts` 故意触发这个错配:

```
[4] 失败模式 — merge 必须在确定性 eval 态 (dropout 关):
    eval 态 (dropout off): max|base+LoRA - merged| = 4.44e-16  -> < 1e-6, 合法合并
    train 态 (dropout on): max|base+LoRA - merged| = 6.60e+0  -> 远超 1e-6
    ✗ 训练态的随机 dropout 无法被静态合并权重复现 -> 合并前必须切 eval(关 dropout)。
    根因: merge 把 LoRA 当成确定性线性增量折进 W;dropout 让 LoRA 分支变成随机函数,二者不再等价。
```

看这两行的对比:eval 态差 `4.44e-16`(地板),train 态差 `6.60e+0`——**相差十六个数量级**。`6.6` 这种量级一眼就知道是结构性错配,不是数值问题。根因一句话:merge 的全部合法性建立在"LoRA 是确定性线性增量"这个假设上;dropout 把 LoRA 分支变成了随机函数,假设崩了,等价自然崩。

这不是玩具特有的坑,是真实工程里 merge 翻车的头号原因。HuggingFace PEFT 的 `merge_and_unload()` 之所以要你先 `model.eval()`,正是这个道理。你会在生产里见到的症状是:训练 loss 漂亮、合并后线上指标却掉一大截——十有八九是 merge 时模型还在 train 态,dropout / BN 之类的随机或 running-stats 模块污染了合并权重。本章把它从"线上事故"前置成了一个 `1e-6` 断言能当场抓住的本地检查。注意 `stage08-merge.ts` 里还留了个诚实的旁注:dropout 的随机性意味着**偶尔某次 mask 恰好没丢关键单元**,这个 train 态 diff 也可能凑巧很小——所以代码不把"diff 大"当通过条件,只把它当"为什么不能在 train 态 merge"的演示。这正是第 00 章反复强调的:**单次 N=1 的命中是噪声,不是信号**。

## 回收第 00 章的预测表:全书在这里收口

第 00 章开篇给过一张"预测表",当时是空的占位——它许诺全书跑完会用实测数据填满,让你亲眼看到 PEFT 到底省了什么、代价是什么。`stage08-merge.ts` 现在把它填上了,而且为了让"LoRA 逼近全量微调"是**实测对比而非断言**,它特地从同一基座解冻全部参数、在同任务同步数上真训了一个 full-FT 基线:

```
[5] 第 00 章预测表 · 实测填充版 (toy-scale):
    ┌─────────────────────┬───────────────┬──────────────┬───────────────┬──────────────┐
    │ 方法                │ 可训练参数    │ 占总参数比   │ 训练显存(est.)│ task acc     │
    ├─────────────────────┼───────────────┼──────────────┼───────────────┼──────────────┤
    │ 全量微调 (full-FT)  │           602 │       100.0% │      11.76 KB │        16.3% │
    │ LoRA (r=2, 冻结)   │            64 │       10.63% │       5.45 KB │        13.9% │
    │ LoRA 合并后交付     │             0 │        0.00% │        = 基座 │        13.9% │
    └─────────────────────┴───────────────┴──────────────┴───────────────┴──────────────┘
    (full-FT 末轮 loss=2.2246, LoRA 末轮 loss=2.2870; 同任务/步数对比)
```

逐列读这张表,每一列都对应前面某一章证过的机制:

- **可训练参数 602 → 64 → 0**。full-FT 动全部 602 个参数(第 02 章);LoRA 只动 `A(2×16) + B(16×2) = 64` 个(第 03 章);合并后交付 0——增量焊进权重,没有任何可训练量留下。
- **占比 100% → 10.63% → 0%**。这是 PEFT 的字面定义:Parameter-Efficient,只动一小撮。`10.63%` 是玩具规模偏高的数字(模型太小,`64/602` 摊不开);真实 7B 模型上 LoRA 的可训练占比通常在 **0.1% 量级甚至更低**——⚠ 这个比例随规模剧烈变化,玩具值别外推,可外推的是"远小于 1"这个量级结论。
- **训练显存 11.76 KB → 5.45 KB → = 基座**。这是 `estMem` 用第 07 章同款公式估的(Adam 的两份 optimizer state + 梯度都只为可训练参数分配)。full-FT 要为 602 个参数全套 state,LoRA 只为 64 个——这才是 LoRA 省显存的真正大头,**不是省在权重本身(权重照样全程在显存里),而是省在梯度和 optimizer state**。这点第 02、03 章已经讲透。注意标 `(est.)`:这是按公式估算,不是实测峰值。
- **task acc 16.3% → 13.9% → 13.9%**。full-FT `16.3%`,LoRA `13.9%`,合并后仍 `13.9%`。差距 `2.4` 个百分点。

收束那句:

```
一句话收束: LoRA 用 10.63% 的可训练参数,acc 与全量微调相差 2.4 个百分点
(toy 任务,差距随规模通常收窄);合并后推理与基座同开销、同参数量,可像普通权重一样交付与热切换。
```

`2.4` 个点的 gap 是玩具任务的乐观值——⚠ 在真实大模型上,LoRA 与 full-FT 的差距通常**比这更小**(原 LoRA 论文在多个 NLU 基准上报告过几乎打平甚至反超),因为大模型的有效更新本就是低秩的(这正是第 04 章讨论秩时的核心假设)。但请别把 `2.4` 当成可迁移的定论:它依赖任务、秩、数据量。这张表里**真正与规模无关、可以拍胸脯带走的**只有最后一行:**合并后交付 = 0 可训练参数、与基座同算子开销、同参数量、可像普通权重一样交付与热切换**——这是数学等价保证的,不是拟合出来的。

```
⚠ toy-scale: 绝对值偏乐观;可迁移的是 <1e-6 数学等价、合并后 matmul 回落、dropout 失败模式。
```

## 全书图景:一个冻结基座,一堆可插拔的小增量

走到这里,PEFT 的统一图景已经完整。第 01、02 章建了基座和全量微调的对照;第 03 章把 LoRA 的低秩并联讲透;第 04 章拆了秩这个旋钮的取舍;第 05、06 章用 Adapter 和 Prefix 当对照系,让你看清 LoRA 的结构优势从何而来;第 07 章用 QLoRA 把基座压到 4-bit、把"省显存"推到极致。本章是收口:**所有这些方法共享同一个底层范式——冻住一个昂贵的通用基座,只训练一小撮可插拔的增量**。而 LoRA 在这个范式里占了最甜的位置,因为只有它的增量能 merge 回权重,做到训练时省显存、推理时零开销、交付时和原模型字节同构、部署时按需热切换。

这就是为什么综述里那句"LoRA 推理零开销"值得被认真对待——它不是营销话术,是一个能被 `9.44e-16` 这种数字证死的数学事实。但也正因为它是数学事实,它有边界:dropout 一开就崩(本章 `6.60e+0` 的反例),batch 里要混适配器就 merge 不了(那个仍在研究的开放问题)。把卖点和它的边界一起讲清楚,才是这本书想区别于 AI 批量综述的地方——**讲透为什么有效,也讲透什么时候失效、代价在哪。** 现在你不只是"会跑 LoRA",你能从第一性原理判断:这个 case 该不该 merge、能不能合、合了之后省了什么、又放弃了什么。
