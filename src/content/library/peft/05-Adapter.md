LoRA 和 Adapter 经常被人混着说,好像都是"加个小模块微调"。它们确实都属于 PEFT(参数高效微调,只训一小撮参数、冻结主干),可训练占比也都在百分之几。但它们的本质分野在两条轴上,而且这两条轴决定了你部署时会不会被坑:**增量加在哪**(改已有权重的数值 vs 在前向图里塞一个新子层),以及**能不能合并**(推理时增量能否折回原权重、把额外开销降到零)。LoRA 两条都站在"轻量"一侧,Adapter 在第二条上恰好相反——它永远多付一笔 matmul 税。这一章我们手写一个 Adapter,把这两条轴用真实代码和真实运行数字钉死,而不是停留在"它俩差不多"的模糊印象。

代码在 `examples/peft-from-scratch/src/stage05-adapter.ts`,复用前几章那套手写引擎(`core/tensor.js` / `core/nn.js` / `core/optim.js`),所以你能逐行看到瓶颈模块怎么挂上一个冻结的 base、梯度怎么只流进 adapter。

## Adapter 到底是什么:一个 d→m→d 的瓶颈子层

Adapter 的核心就一个 `forward`,八行不到:

```typescript
class Adapter extends Module {
  down: Linear;
  up: Linear;
  residual: boolean;
  constructor(d: number, bottleneck: number, residual = true) {
    super();
    this.down = new Linear(d, bottleneck);
    this.up = new Linear(bottleneck, d);
    // Near-zero up-proj: overwrite xavier init so adapter(x) ≈ 0 at step 0.
    this.up.W.data.fill(0);
    this.up.b.data.fill(0);
    this.residual = residual;
  }
  override forward(x: Tensor): Tensor {
    const delta = this.up.forward(this.down.forward(x).gelu());
    return this.residual ? x.add(delta) : delta;
  }
}
```

读法:输入是 d 维(本 stage `d=32`)的隐藏向量,先用 `down` 投影到一个很窄的瓶颈维 m(bottleneck,这里 `m=8`),过一个 GELU(一种平滑的非线性激活)制造表达力,再用 `up` 投回 d 维,最后**残差直连**加回原输入。降维→非线性→升维这个"沙漏"形状就是 bottleneck(瓶颈)一词的来源:中间窄,两头宽,可训练参数被 m 卡住,因此很省。

它和 LoRA 形状上很像——LoRA 的 ΔW=BA 也是 d→r→d 的低秩分解。区别在**作用对象**:LoRA 的 A、B 是某个已有权重 W 的增量,前向时算的是 `(W + BA)·x`;Adapter 的 down、up 是一个**独立子层**,前向时算的是 `x + up(gelu(down(x)))`,它不依附任何已有权重,而是被插在两个已有子层中间。一个改已有权重的数值,一个在图里新增一层——这是第一条轴。

`down`/`up` 都是普通 `Linear`,所以可训练参数 = down 的 (d·m + m) + up 的 (m·d + d)。冻结的是整个主干,训练的就这 4 个张量(down.W/b、up.W/b)。stage 输出里那行 `仅训练 adapter (4 个张量)` 就是这个意思。

### 为什么 up 投影要初始化成近零

注意构造函数里那两行 `this.up.W.data.fill(0)` / `this.up.b.data.fill(0)`——把 `Linear` 默认的 xavier 随机初始化**覆盖成零**。这不是可有可无的细节,是 Adapter 能安全挂到一个已经训好的模型上的全部理由。

up 是零,意味着 `delta = up(gelu(down(x))) ≈ 0`,于是带残差时 `forward(x) = x + 0 = x`——adapter 在第 0 步是一个**精确的恒等变换(no-op)**。你把它焊到一个预训练好的 base 上,第一次前向跑出来的结果和没插它时**一模一样**,然后训练才让它从恒等慢慢偏离。代码注释里把这条标成 INVARIANT(不变量):"the first forward pass reproduces the base almost exactly"。

stage 怎么验证这条不是嘴上说说?它在训练**前**测了一次 loss:

```
step-0 loss=13.5777 == 基座在 B 的 loss=13.5777  ⇒ 残差 adapter 初始 == 基座 (近似 no-op, 安全挂载)
```

挂了 adapter 的模型在任务 B 上的 step-0 loss,和**冻结基座自己**在 B 上的 loss 一字不差(13.5777)。这是 near-zero init 的直接后果,也是一个能被数值证明的设计断言,不是信仰。

## 实验设置:为什么用 copy→reverse 这对任务

和前几章一致,stage 用一个极小的 Transformer(单 block,`d=32`,2 头,`vocab=12`)。先在**任务 A=copy**(输入序列原样输出)上预训练当 base,再用 adapter 适配到**任务 B=reverse**(输出倒序)。选这对任务有讲究:reverse 复用了 copy 学到的"记住这些 token"的能力,只是把顺序翻过来——一个 base 大体会、但需要局部修正的改动。这正是 adapter 该擅长的场景:不是从零学新技能,而是在冻结知识上做小幅扭转。

预训练后的状态:

```
预训练后: 基座在 A 上 acc=100.0%  直接拿去做 B 的 acc=8.3% (未适配, 应较差 → 这就是 PEFT 要补的 gap)
```

base 在 copy 上学满了(100%),但直接拿去做 reverse 只有 8.3%——`8.3% ≈ 1/vocab = 1/12`,即随机瞎猜的水平。这个 gap 就是 PEFT 要补的东西。挂上残差 adapter 训练后:

```
适配前 acc=8.3%  → 适配后 acc=18.4%
```

从 8.3% 抬到 18.4%。注意这个绝对值很低——别被它误导,stage 末尾专门标了 `⚠ toy-scale`:单 block + reverse 对一个宽度 m=8 的瓶颈本来就难,最终准确率含噪、不漂亮。**可迁移的信号不是最终 acc,而是机制**:near-zero 安全挂载、残差 vs 串联的初期分化、不可合并的额外开销。这是这本书的态度——toy 数字标清楚,讲透机制,不拿玩具规模的乐观数字冒充结论。

## 失败模式:去掉残差,adapter 就把基座信号抹平了

Adapter 的残差直连不是"顺手加的好习惯",它是保命的。把它去掉(`residual=false`),你会看到一个干净的灾难。

串联版本(纯 `delta`,不加 `x`)在第 0 步的前向是 `forward(x) = up(gelu(down(x))) ≈ 0`。注意:up 还是零初始化,所以输出**整个是约零向量**。这个 ≈0 把冻结 base 辛辛苦苦算出来的全部隐藏状态**一笔抹平**,送进输出头的是一片零,头只能吐出近似均匀分布。stage 测到的 step-0 loss:

```
step-0 loss=2.4860 ≈ ln(vocab)=2.4849  ⇒ ≈0 输出把基座信号抹平成均匀分布 (退化到随机猜)
适配前 acc=8.1% (≈ 1/vocab=8.3%, 随机水平)  → 适配后 acc=21.6%
```

`ln(vocab) = ln(12) = 2.4849` 是均匀分布下交叉熵的理论值——预测完全均匀(每个 token 概率都是 1/12)时的 loss。串联 adapter 的 step-0 loss 精确落在这个"死底"上。这意味着不管 base 预训练得多好,串联结构在 init 时把它**清零**了,模型退化到必须从随机分布重新学起。

为什么这是结构性的而不是调参能救的?对比两个版本前 40 步的训练曲线:

```

  11.871 │    • •

串联 Adapter (起点 ≈ ln(vocab)=2.48 的均匀分布, 退化)

```

两条曲线起点完全不同:残差版从基座的高 loss(~14,因为 base 没学过 reverse)开始往下走,**整条信息流是保住的**,它只需要微调;串联版从 ~2.48 那个均匀分布的"死底"贴地爬行,它得先把 base 早就算好的信号**重新从一个 m=8 的瓶颈里推导出来**,再谈适配。换句话说,残差版在"修",串联版在"重建一座被自己拆掉的桥"。

这就是为什么 Houlsby/Pfeiffer 那条经典 Adapter 配方一定带残差、一定 near-zero init:两者配合才能保证"挂上去等于没挂,训练才慢慢偏离"。把任一个拿掉都会在 init 时破坏 base 的信息流。

> 诚实标注:toy 规模下两个版本最终 acc 都低且含噪(残差 18.4% vs 串联 21.6%,串联甚至偶尔更高),**最终名次不是可靠信号**。可靠的是 step-0 行为——精确 no-op vs 精确 ln(vocab) 坍缩。这是 stage 注释反复强调的:"the robust, transferable signal is the STEP-0 behavior, not the final-acc horse race."别去过度解读那两个收敛后的数字。

## 成本账:可训练参数省,但前向多一笔税

PEFT 的卖点是省可训练参数,这点 Adapter 和 LoRA 一样达标。stage 把三者放一起比(都相对同一个 base):

```
full-FT (全参)   │████████████████████████████████████████│ 9,876  100%
Adapter m=8    │██······································│ 552  5.59% of base
LoRA r=8 (q,v) │████····································│ 1,024  10.37% of base
```

Adapter(m=8)552 个可训练参数,占基座 5.59%;同秩 LoRA(r=8,加在 q、v 两个权重上)1024 个,10.37%。两者同量级,都远小于 full-FT 的 9876。**这是 PEFT 共有的训练成本优势**,Adapter 不输。

但训练省不等于推理省。Adapter 的代价藏在前向计算量里:

```
基座 matmul 次数 = 11
+ adapter (down+up) = 2  ⇒  额外 +18.2%
```

每次前向,adapter 的 down 和 up 各是一次矩阵乘,额外 +2 次。相对基座 11 次 matmul 就是 **+18.2%** 的前向开销。toy 规模下这个百分比偏大(真实模型 base 的 matmul 多得多,占比会小),但**关键不是百分比大小,是这笔开销永远存在、无法消除**。

为什么 LoRA 没有这笔税?因为 LoRA 的增量 ΔW=BA 和原权重 W 是**同一个 d×d 形状的可加项**,推理前可以一次性折回:`W' = W + ΔW`,之后前向就是普通的 `W'·x`,**零额外 matmul**。第 08 章会专门手写这个 merge 过程。而 Adapter 是前向图里一个**独立子层**,它的 down→gelu→up 中间夹着非线性,数学上没法被吸收进任何相邻的线性层——你永远得显式跑那两次 matmul。stage 的总结一针见血:

```
LoRA 的 ΔW=BA 可在推理前折回 W (W' = W + ΔW), 推理 0 额外开销;
adapter 是图里独立子层, 永远付这笔 matmul 税 —— 模块化 vs 推理延迟的权衡。
```

这就是第二条轴——**能否合并**——的实质。它不是实现细节,是部署时的硬权衡:如果你追求推理延迟极致、且一次只服务一个任务,LoRA 合并后零开销更划算;如果你要**热插拔多个任务、运行时切换**(下面会讲),Adapter 那个"独立子层"反而成了优点,因为它本来就是个可装可卸的模块。LoRA 一旦 merge 进 W 就和那个任务焊死了,要换任务得 unmerge 再 merge 另一个。

## ⚡ 前沿:多 adapter 组合还没有通用解

你刚手写的这个瓶颈模块,正好是参数高效**多任务**学习一整条活跃研究路线的基本单元。思路很自然:既然 adapter 是个独立可插拔的子层,那一个冻结 base 上能不能同时挂好几个 adapter,每个学一个任务,推理时按需组合甚至融合?

AdapterFusion 是这方向的代表:先为每个任务单独训一个 adapter,再训一个轻量的注意力式融合层,让模型在推理时**学着加权组合**多个 adapter 的输出,而不是简单相加。理想情况下你能得到任务间的知识迁移,又不互相干扰(full-FT 多任务时常见的负迁移)。LoRA 那边也有平行的探索——多个 LoRA 加权合并、按输入路由到不同 LoRA。

但要清醒:**多 adapter 怎么组合才稳,目前没有通用解,仍在研究中**。已知的麻烦至少有三类。一是**组合顺序与位置**:adapter 插在 FFN 后还是 attention 后、多个 adapter 串行还是并行,结果对架构选择很敏感,没有一个"照着做就对"的配方。二是**干扰与容量**:adapter 越加越多,瓶颈维 m 又很窄,任务间会争抢这点可怜的容量,加到一定数量收益就掉头向下,但拐点在哪没有理论刻画。三是**融合的训练成本**:AdapterFusion 那个融合层本身要在多任务数据上训,把"参数高效"省下的一部分又吃回去了,什么场景下这笔投入划算缺乏明确判据。

所以这是个真实的开放问题:基本单元(本章这个 bottleneck)是清楚的、可手写的,但**怎么把多个单元组合成一个稳定、可扩展、不互相打架的多任务系统**,还在被活跃研究,没有可照搬的标准答案。你手里这八行 `Adapter` 就是这条路的起点。

## 小结:把两条轴记牢

回到开头那两条轴,现在有代码和数字撑着了:

- **增量加在哪**——LoRA 改已有权重的数值(`W+BA`),Adapter 在前向图里塞一个新子层(`x + up(gelu(down(x)))`)。
- **能不能合并**——LoRA 能折回 `W'=W+ΔW`、推理零开销;Adapter 因为中间夹非线性、是独立子层,**永远多付 +18.2% 那两次 matmul**,但换来天然的模块化/可热插拔。

这两条轴是理解整个 PEFT 设计空间的坐标。第 03/04 章的 LoRA 占了"轻量合并"那一角;本章的 Adapter 占了"模块化但有推理税"那一角。还有一角我们还没碰:增量不加在权重上、也不加在层间,而是彻底移到**输入端**——给序列前面拼一段可训练的"虚拟 token"。那就是第 06 章 prefix/prompt tuning 要走的路,把"增量加在哪"这条轴推到最极端的位置。
