先说一个面试现场会被问、上线现场会被坑的事实:你手上有一个 7B 基座,要服务 30 个客户、每个客户一套定制行为。全量微调(full fine-tuning,把所有权重都拿去训)的代价是——你要存 30 份各 14GB 的权重拷贝,训练时每份还要再叠上梯度和优化器状态。一台 A100 训一个都吃力,30 个任务的部署和切换是另一个噩梦。PEFT(parameter-efficient fine-tuning,只动一小撮参数的微调)不是"省钱的小技巧",它是让"一个基座服务 N 个任务"这件事在工程上成立的前提。

这本书的做法跟市面上 AI 批量生成的 PEFT survey 稿相反:那些稿子会给你一张"LoRA vs Adapter vs Prefix"对比表,术语齐全、判断为零。我们不背对比表——我们把 LoRA、Adapter、Prefix、QLoRA 四种全部用 TypeScript 从零手写一遍,在玩具任务上真跑梯度下降,让"可训练参数占比""ΔW 低秩性""合并前后等价误差"这些数字是代码算出来的,不是从论文抄的。配套代码在 `examples/peft-from-scratch/`,纯算法、离线 CPU 可跑、种子确定可复现,不需要任何 LLM 或 API key。

## 全量微调的三堵成本墙

全量微调的直觉很顺:预训练给了一个通用基座,下游任务再训一遍就专精了。问题全在"再训一遍"这四个字的代价上,而且三个代价是叠加的。

### 墙一:每任务一份全权重拷贝

全量微调改的是所有权重 W。任务 A 训完得到 W_A,任务 B 训完得到 W_B,两者跟原基座 W 一样大、彼此独立。N 个任务就是 N 份完整权重。对 7B 模型,每份 fp16 约 14GB——存储、分发、加载、热切换全部按这个量级走。

PEFT 的第一个杠杆就在这里:基座 W 永远不动(冻结),每个任务只额外存一小撮增量参数。第 03 章的 LoRA 增量在玩具规模下也才几百到几千个浮点数;真实规模下一个 LoRA adapter 通常是几 MB 到几十 MB,对比 14GB 的基座,你可以在一份基座旁边挂几十个任务的 adapter,这正是第 08 章"一个基座 + 多 adapter 热切换"的部署故事。

### 墙二:显存按可训练参数翻几倍

训练时显存不只是权重本身。配套仓 `src/core/mem.ts` 把真实训练显存的会计公式照搬过来(纯算术,不真分配显存),拆成四项:

```typescript
// examples/peft-from-scratch/src/core/mem.ts
const weightsBytes = frozen * baseBpp + cfg.trainableParams * bpp;
const gradBytes = cfg.trainableParams * bpp;            // 只有可训练参数有梯度
const optimStateBytes = cfg.trainableParams * bpp * optMul; // Adam: optMul=2 (m 和 v)
const activationBytes = cfg.totalParams * actFactor * bpp;  // 前向要全基座的激活
```

看清楚这里的不对称:**梯度和优化器状态只跟"可训练参数"成正比,跟基座总量无关。** 全量微调下可训练参数 = 全部参数,所以用 Adam 训练时,光优化器的 m、v 两个状态就是权重本身的 2 倍,加上一份梯度,等于权重的 3 倍显存——这还没算激活。这就是为什么一个能塞进显存做推理的模型,做全量微调时会 OOM(显存溢出)。

PEFT 砍掉的正是这两项。把可训练参数从 100% 压到个位数百分比,梯度和优化器状态就同比例缩水。`mem.ts` 顶部的注释把这点说死了:PEFT 的省钱发生在 `gradBytes + optimStateBytes`,**不在** `weightsBytes`——冻结的基座照样得驻留在显存里。QLoRA(第 07 章)是唯一额外动 `weightsBytes` 的方法:它把驻留的冻结基座按 4-bit 存(`bytesPerBaseParam` 设成 0.5),连基座那一项也压下去。

> ⚠ toy-scale 免责:`mem.ts` 算的是估算,每个输出都标 `(est.)`。玩具规模下绝对 MB 没意义,可迁移的是公式结构和"全量微调的柱子高高压过 PEFT"这个形状,不是具体数字。

### 墙三:灾难性遗忘

第三堵墙最隐蔽:全量微调动了所有权重,模型学会任务 B 的同时会忘掉任务 A,这叫灾难性遗忘(catastrophic forgetting,学新任务把旧能力冲掉)。第 02 章会把这件事画成两条交叉曲线——任务 B 准确率往上爬的同时,任务 A 准确率往下掉;激进 learning rate 下任务 A 直接崩到随机水平。"它学会了新任务"这句话,藏着"它忘了旧任务"。

PEFT 天然缓解这点,原理在下一节的统一心智模型里会变得显然:基座权重根本没被碰过。

## 统一心智模型:冻结基座 + 小可训练增量

把整本书压缩成一句话:**PEFT = 冻结一个预训练基座,只学一个小的可训练增量,让增量去承载新任务。** 四种方法的差别,只是"增量插在哪、长什么样"。

这个心智模型在代码里是一个硬不变量。配套仓 `src/core/checkpoint.ts` 的 `loadBase` 默认就把加载进来的权重冻结:

```typescript
// examples/peft-from-scratch/src/core/checkpoint.ts
params.forEach((p, i) => {
  // ... 形状校验 ...
  p.data.set(ckpt.buffers[i]);
  p.requires_grad = !freeze; // freeze => requires_grad=false (基座只读)
});
```

"冻结"在引擎里的精确含义,`src/core` 的契约写得很干净:冻结的叶子参数(`requires_grad=false`)**照样参与前向、照样把梯度传给它的输入,但它自己的 `.grad` 永远是 0,优化器永远不移动它。** 优化器 `src/core/optim.ts` 只更新 `requires_grad=true` 的参数。这条不对称——前向用它、反向跳过它——就是 PEFT 便宜的全部原因,也是它不遗忘的全部原因(基座没动,旧能力还在)。

有了这个框架,四种方法就是四种"增量长什么样":

| 方法 | 增量插在哪 | 增量是什么 | 可训练参数量级 | 章节 |
|------|-----------|-----------|--------------|------|
| LoRA | 已有权重矩阵 W 旁边,加性 | ΔW = (α/r)·B·A,低秩 | r·(out+in) | 03 / 04 |
| Adapter | 层与层之间,串入新子层 | 瓶颈 down→激活→up + 残差 | 2·d·m | 05 |
| Prefix / Prompt | 输入序列最前面 | p 个可训练虚拟 token 向量 | p·d | 06 |
| QLoRA | = LoRA,但基座量化存储 | LoRA 增量 + 4-bit 冻结基座 | 同 LoRA | 07 |

✦ **面试高频题:"LoRA、Adapter、Prefix tuning 的区别是什么,各自动哪部分参数?"** 看上面这张表你能背出来,但背答案和讲透是两回事。读完这本书你不是背过这张表,而是三种都亲手写过 forward、跑过参数占比、踩过各自的失败模式——LoRA 的 B 不归零会有冷启动尖峰、Adapter 去掉残差会在初始化时湮灭基座信号、Prefix 的软 token 初始化偏离基座流形就训不动。这些是表里没有的"何时坏、为什么这么设计",也是面试官追问第二层时真正想听的。

### 四种方法的关键取舍,不是"哪个最好"

资深工程师选 PEFT 方法不看"谁参数最少",看场景的取舍:

- **LoRA / QLoRA 能合并回基座**(第 08 章证明合并前后输出等价,误差在 1e-6 浮点舍入级别):推理时把 ΔW 折回 W,零额外延迟、零额外参数。这是它今天最主流的原因。
- **Adapter 不能合并**:它是前向图里一个独立子层,推理永远多付两次矩阵乘的延迟税。换来的是模块化、可组合。
- **Prefix / Prompt tuning 参数最省**(只有 p·d,跟基座大小完全解耦),但对初始化敏感、容量上限低。
- **QLoRA 是唯一压缩驻留基座显存的**,代价是量化重构误差——第 07 章会演示 2-bit + 单一全局缩放下基座技能直接崩掉。

没有"最好",只有"这个约束下选哪个"。这正是 survey 稿给不了的判断。

## 一切的前提:基座得先被稳定训出来

PEFT 把基座当成可信的地基,但地基本身得是好的。如果你冻结的是一组没被稳定训练过的垃圾权重,在上面叠多精巧的 LoRA 都是空中楼阁。所以第 01 章的任务很窄也很关键:训出一个**确实学到了东西**的 toy 基座,并证明"它能被稳定训练"这件事依赖一个后面所有章节都默认存在的部件——LayerNorm(层归一化,把残差流的尺度钉住)。

第 01 章的 `stage01-base.ts` 用一个极小配置(`dModel=32`、`heads=2`、`dFF=64`、`vocab=12`、`seqLen=8`)训一个 toy Transformer 做 copy 任务,CPU 秒级跑完。它做的不只是 happy path——它故意去掉 LayerNorm,在同种子、同数据、同优化器、唯一变量是"有没有归一化"的对照下,把 learning rate 推到激进值,演示无归一化的模型残差尺度失控、loss 震荡爆到 NaN。这个失败模式的教训直接转移到微调:

> 后续微调冻结的是 base 权重——base 没被稳定训过,LoRA 就建在垃圾权重上,所以本书每章都要求 base 先收敛再 snapshot。

这就是为什么全书从"训基座"而不是"写 LoRA"开始:没有一个 100% 可训练、确实收敛的基座做参照,"LoRA 只用 0.5% 参数就追平"这句话连分母都没有。

## ⚡ 前沿:2024+ 的 LoRA 变体,全是 ΔW=BA 的小改

把 `ΔW=BA` 这个核心机制学透,你会发现近两年刷屏的 LoRA 变体不过是在这个骨架上动一两个零件:

- **DoRA**(weight-decomposed LoRA):把权重分解成"幅度 + 方向"两部分,方向部分用 LoRA 更新,幅度单独学。本质是给 `ΔW=BA` 加了一层幅度/方向的解耦。
- **PiSSA**:不再把 B 初始化成零,而是用基座权重 W 的 SVD(奇异值分解)主成分来初始化 A、B,让低秩增量从"最重要的子空间"出发。改的只是第 03 章会反复强调的"B 怎么初始化"。
- **LoRA+**:给 A 和 B 设不同的 learning rate(B 的 lr 调大),修正标准 LoRA 里两个矩阵学习速率失衡的问题。改的只是优化器的 paramGroups——配套仓 `optim.ts` 已经支持 paramGroups,你写完第 03 章就能自己加。

这里有个诚实的边界要标清楚:**这些变体哪个在你的任务上更好,目前没有通用结论,仍在活跃研究中。** 不同论文在不同 benchmark 上给出互相矛盾的排名,数据集、任务类型、rank 取值都会翻转结果。这本书不替你选——它给你的是:当你读到下一篇"我们提出 XxxLoRA"时,你能一眼看穿它改的是 BA 分解的哪个零件、那个改动想解决标准 LoRA 的什么毛病、代价又落在哪。把机制吃透,新变体对你就不是新东西,是排列组合。
