先把最容易被搞混的一点钉死：**本章说的「软提示」（soft prompt）跟你在 ChatGPT 输入框里打的那段文字 prompt 是两回事**。文字 prompt 是离散 token——先查词表映射成整数 id，再过 embedding 表变成向量，这条路对输入向量不可微，你只能换词、加例子、调措辞。软提示是直接在 embedding 空间里凭空放几个**可训练的连续向量**，绕过词表，没有任何离散 token 与之对应，能直接吃梯度被 Adam 推着走。换句话说，你不是在「写更好的提示词」，你是在**梯度下降里搜一段没法用人类语言写出来的提示**。这一章我们手写这个拼接，看清它到底是什么、为什么这么省参、又为什么这么难训。

配套代码 `examples/peft-from-scratch/src/stage06-prefix.ts`，编译后纯离线跑、固定种子（`seed(1234)`）、同机逐字节可复现。下面引用的所有数字都来自它的真实输出。

## 增量的第三个位置：从「改权重」到「改输入」

回顾前面几章的增量都塞在哪：第 02 章全量微调直接动所有权重；第 03/04 章 LoRA 把增量塞进权重矩阵的低秩旁路（`W + BA`）；第 05 章 Adapter 在 block 里插了个小瓶颈模块——它们的共同点是**都在动「函数本身」**，要么改 W，要么往计算图里加层。

软提示走的是完全正交的一条路：**函数一个字节都不改，只改喂给它的输入**。把 frozen 基座当一个固定的黑盒 `f`，你不去碰 `f`，而是在它的输入序列前面拼 `p` 个可训练向量，让 `f` 把这段前缀当成额外的「上下文 token」去 attend。训练时梯度穿过整个 frozen 网络回流到这 `p` 个向量上，只更新它们。

这个区别在代码里是物理可见的。看本章 frozen 基座的前向（`stage06-prefix.ts` 的 `FrozenEncoder.forward`）：

```ts
forward(ids: number[], prefix: Tensor | null): Tensor {
  const tokEmb = this.embed.lookup(ids); // (seq, d)
  const x = prefix ? concatRows(prefix, tokEmb) : tokEmb; // (p+seq, d)
  // pre-LN attention block with two residuals
  const a = this.attn.forward(this.ln1.forward(x)).add(x);
  const f = this.ff2.forward(this.ff1.forward(this.ln2.forward(a)).gelu()).add(a);
  const pooled = meanRows(f); // (1, d)
  return this.head.forward(pooled); // (1, C)
}
```

注意 `prefix` 是在 **embedding 查表之后、attention 之前**沿序列轴拼进去的（`concatRows(prefix, tokEmb)`）——这正是 prompt tuning 的注入点。`prefix` 是一个 `(p, d_model)` 的张量，`requires_grad = true`，而 `this.embed / this.attn / ...` 全部冻结。整个前向里唯一带梯度的叶子节点就是 `prefix`。

软提示要能学到东西，梯度必须能穿过这个 concat 流回前缀。这就是为什么本章不能直接复用 `TransformerBlock`，而要手写带反向的 `concatRows`：

```ts
function concatRows(top: Tensor, bottom: Tensor): Tensor {
  // ... 前向把 top/bottom 的数据按行拼起来
  out._backward = () => {
    if (top.requires_grad) for (let i = 0; i < r1 * d; i++) top.grad[i] += out.grad[i];
    if (bottom.requires_grad) for (let i = 0; i < r2 * d; i++) bottom.grad[i] += out.grad[r1 * d + i];
  };
  return out;
}
```

`out.grad` 的前 `r1*d` 个元素回流给前缀，剩下的回流给真实 token embedding（但后者 frozen，不更新）。`meanRows` 同理把池化的梯度按 `1/rows` 摊回每一行，包括前缀那几行——这就是软提示影响输出的全部通道：它通过池化直接挪动那个被分类头读取的 pooled 向量。**没有这条反向路径，软提示就是死的常数。**

### prompt tuning vs prefix tuning：注入到哪一层

本章手写的是 **prompt tuning**（Lester et al., 2021）——只在最底层的输入 embedding 前拼一段前缀。它的近亲 **prefix tuning**（Li & Liang, 2021）更激进：不止输入端，而是给**每一层 attention 的 K 和 V 都注入一段可训练前缀**，相当于让前缀在每一层都参与注意力计算。

两者的取舍很直接：prefix tuning 表达力更强（每层都能注入），但可训练参数翻成 `层数 × 2 × p × d_model`（K 和 V 各一份）；prompt tuning 只有 `p × d_model`，更省，但所有调控都得挤过最底层那一次注入、再靠网络自己往上传。原论文的经验是模型越大、prompt tuning 越能追平 prefix tuning——这也是为什么大模型时代「软提示」基本默认指更省的 prompt tuning。本章 toy 是单层 block，两者在结构上退化成同一件事，所以我们手写最省的那版，把「参数与基座解耦」这条结论顶到极限来观察。

## 实测：256 个参数，把 2.2% 提到 86.7%

本章的实验设计是为了让「软提示在补什么」一眼可见。先在任务 A（label = 序列 token 均值分桶到 3 类）上预训练基座并整体冻结：

```
base loss: 1.2041 -> 0.0001
base 任务 A 准确率：训练集 100.0%  验证集 91.1%（真泛化，非死记）
冻结后 base 可训练参数 = 0 个张量（期望 0）
```

验证集 91.1% 是个关键设计——它证明基座**真学会了「按 token 均值分桶」这个能力并能泛化**，不是死记训练集。为什么非要它泛化？因为任务 B 是把任务 A 的标签循环 +1（`(labelA + 1) % 3`），**特征完全相同、只是重新映射标签**。这样基座抽取的「均值桶」特征对 B 仍然有效且能泛化，软提示要做的只是把这个已有能力重新路由到新标签上。这隔离出了本章最重要的判断：

> 软提示是**操纵基座已有的能力**，不是注入基座没有的能力。

证据是基座直接评测任务 B 的成绩：

```
无软提示时任务 B 准确率 = 2.2%  ← 低于随机（A 的 head 系统性答错）
```

低于随机（3 类随机是 33%）不是 bug，是信号：A 训出来的分类头对 B 是**系统性答错**——它准确地把「应该是类 1」的样本判成类 0，因为标签被循环移位了。这 2.2% 就是软提示要填的坑。然后拼 `p=8` 个软 token、基座全冻、训 200 步：

```
任务 B 准确率：无提示 2.2% -> 软提示 86.7%
```

只动 8 个向量，从 2.2% 拉到 86.7%。再看参数账（诚实计数，非论文引用）：

```
软提示参数 = p×d_model = 8×32 = 256
冻结基座参数 = 9027
可训练占比 = 2.836%  ← 与 base 大小解耦
```

**2.836% 是全书最低的一档**。更重要的是分子的构成：`p × d_model = 8 × 32`。这里**没有任何一项是基座大小的函数**。第 03 章 LoRA 的参数是 `r × (d_in + d_out)`，跟权重矩阵维度挂钩；第 05 章 Adapter 是 `2 × d × bottleneck × 层数`，跟层数挂钩。软提示只跟你**自己选的前缀长度 `p`** 和 `d_model` 有关——基座从 70 亿涨到 700 亿，只要 `d_model` 不变（它一般会变，但远慢于参数总量），软提示参数几乎不动。这是「参数与基座解耦」最干净的体现，也是为什么在超大模型上软提示的参数占比能压到千分之一以下。

⚠ toy-scale 警告：`d_model=32`、单层 block、约百级样本。**绝对数字（86.7%、2.836%、200 步）偏乐观，别当真实规模的预期**。可迁移的是两条相对趋势——参数占比极低、且与基座解耦。

## 前缀越长越好？accuracy(p) 的形状

软提示唯一的容量旋钮是 `p`。本章扫了 `p ∈ {1,2,4,8,16}`，每个 `p` 用同样的稳定初始化重训：

```
p= 1  可训练= 32  任务B准确率=6.7%
p= 2  可训练= 64  任务B准确率=50.0%
p= 4  可训练=128  任务B准确率=82.2%
p= 8  可训练=256  任务B准确率=90.0%
p=16  可训练=512  任务B准确率=93.3%
accuracy(p) 形状  ▄▇██  (左=p小 右=p大)
```

形状是典型的**先陡后平**（concave，边际递减）。`p=1` 几乎没用（6.7%，比 2.2% 强不了多少——单个向量挪不动池化结果），`p=1→4` 是断崖式上升（6.7%→82.2%），`p=8→16` 只多了 3 个点（90%→93.3%）却把参数翻倍。

这条曲线的实践含义：**`p` 不是越大越好，存在一个甜点**。盲目加长前缀，参数线性涨、收益却快速饱和，而且——下一节会看到——更长的前缀对初始化更敏感、更难训。原论文里这条曲线在大模型上甚至更平：`p` 超过 20 左右后基本不动。所以工程上的默认做法是从 `p=8~20` 起步，而不是上来就拉到 100。

（注意 `p=8` 这里报 90.0%，跟前一节的 86.7% 不同——同一配置两次跑数字有差，因为扫描循环里的 PRNG 状态跟单独那次不同。这恰好是个诚实提醒：**toy 规模下单点数字有 run-to-run 噪声，可信的是趋势不是小数点**。)

## 失败模式：软提示对初始化的敏感是「物理性」的

这是软提示最反直觉、也最该讲透的地方。LoRA 你随便初始化（B 置零、A 高斯）基本都能训起来；软提示**换个初始化能直接让它训不动**。本章用一个对照实验把原因量化出来。

两种初始化：

- **稳定**（`initPromptFromEmbedMean`）：每个软 token = embedding 表的均值 + 极小抖动（`normal(0, 0.02)`）。它落在真实 token embedding 所在的那片流形上。
- **大方差随机**（`initPromptLargeRandom`）：`normal(0, 5.0)`，std 是 embedding 流形 std（≈0.18）的约 28 倍。

为什么 std 差这么多就出事？因为前缀拼进去后，第一件事是过一个 **frozen 的 LayerNorm**。LayerNorm 在预训练时只见过 std≈0.18 量级的输入，它的 `gamma/beta` 是针对那个量级标定死的。一段 std=5 的前缀把它推进一个它从没见过的输入区域——在那里 LayerNorm 的归一化把这段巨大的输入压扁、梯度也跟着被压扁。代码里直接测了首步梯度范数（训练前、不走优化器）：

```
首步梯度范数：大随机=4.61e-1  稳定=2.48e+1  (相差约 54×，越小越学不动)
```

**稳定初始化的首步梯度比大随机强约 54 倍。**梯度小 54 倍，意味着同样的学习率下每步只能挪 1/54 的距离。这不是「学得慢一点」，这是学习信号在源头就被 frozen LayerNorm 掐掉了大半。同样 60 步预算下的对照：

```
同 60 步预算下任务 B 准确率：大随机 58.9%  vs 稳定 84.4%
同 60 步 loss 末值：大随机 1.6983  vs 稳定 0.7955
```

大随机的 loss 曲线（同 60 步）降得明显慢、停在高位；稳定的早早压到 0.79。

**为什么这么设计——以及诚实的边界**。本章特意用「**匹配的、短的**」预算（60 步）做苹果对苹果的对比，而不是声称大随机「永久卡死」。因为这个 toy 的软提示只有 `p·d = 256` 维，Adam 给足步数是能爬回来的——代码注释和结论都明确标了这点：

```
诚实补充：本 toy 上大随机并非永久卡死（256 维软提示，Adam 给足步数能爬回），
可迁移的是"收敛代价"而非"一定不收敛"
```

可迁移的教训是**收敛代价**（convergence tax，初始化不对要多花的训练成本）：同一固定预算下，坏初始化系统性落后。在真实规模上，这个「多花的步数」就是真金白银的算力，也是为什么软提示的标准做法是**从真实 token 的 embedding 初始化**（比如拿任务相关词的 embedding，或词表均值），而不是随机高斯。这一点又跟 LoRA/Adapter 形成对比：那两者的初始化容错度高得多，软提示是 PEFT 里把「初始化敏感」放大到最显眼的一档。

把这一节和上一节连起来看就更完整：更长的 `p` 不仅边际收益递减，还意味着更多向量要落进那片狭窄的「in-distribution」流形里，初始化不当时被 LayerNorm 压扁的总量更大——**长度敏感和初始化敏感是同一个物理现象的两面**。

## ⚡ 前沿：软提示、prompt 工程、可学习 system prompt 是同一件事的三种形态

软提示的本质是「**把一个任务编码进极少量连续向量**」。一旦这样看，它跟当下两个很热的方向就是同源的：

- **离散 prompt 工程**——你手调措辞、加示例、改 system prompt，是在**离散 token 空间**里搜一段让模型表现更好的前缀。
- **软提示 / prefix tuning**——在**连续 embedding 空间**里用梯度搜同样的东西，搜索空间更大、能找到人类写不出的解，代价是要能访问梯度、且训练敏感。
- **可学习 / 优化的 system prompt**——介于两者之间的活跃研究：有的把连续软提示蒸馏回离散 token（让它可读、可跨模型迁移），有的用黑盒优化（无梯度，只能调 API 的场景）在离散空间搜，有的研究软提示能否跨任务/跨模型迁移、能否压缩进更少的向量。

这条线上的开放问题，明确**目前没有通用解、仍在研究**：

> ⚡ **软提示能逼近多大比例的「全量微调能达到的能力」，以及这个比例如何随模型规模、任务类型、前缀长度变化——没有统一的理论或可靠预测公式。** 经验上「模型越大、软提示越接近全量微调」有大量观察支持，但为什么、在什么任务上会失效、`p` 的甜点如何先验确定，都还是经验调参。更深一层：软提示找到的那段连续向量**几乎不可解释**——它不对应任何 token，你没法读懂模型到底被「提示」了什么，这让软提示在需要可审计的场景（比如安全对齐的 system prompt）里难以信任。「把任务压进少量向量」既是它最大的优势（极致省参），也是它最大的未解难题（不可解释 + 不可预测）。

## 小结：什么时候选软提示

放进全书的坐标系里收一下尾：

- **参数占比**：全书最低（本章 2.836%，真实规模可低于千分之一），且**唯一与基座大小解耦**的方法——基座越大，软提示的相对优势越明显。
- **代价**：最难训。初始化敏感（首步梯度可差 50× 以上）、长度敏感、收敛慢，且找到的向量不可解释。
- **何时用**：基座极大、显存/参数预算被卡死、且你能拿到梯度（不是纯黑盒 API）时，软提示是把「每任务额外存储」压到最低的选择——一个任务只需存 `p × d_model` 个数，可以为成千上万个任务各存一小段前缀、共享同一个 frozen 基座。
- **何时别用**：训练预算紧、要快速调通、或需要可解释/可审计的增量时——这时第 03 章的 LoRA 或第 05 章的 Adapter 更稳。

第 05 章我们在「改架构」这条路上插了 Adapter 模块；本章补完了增量位置的最后一块拼图——**输入端 / 注意力端**，根本不碰权重。下一章（第 07 章）换个维度做取舍：不再纠结增量放哪，而是把第 03 章的 LoRA 直接叠到一个**量化（quantized，把权重压成低比特存）后的基座**上做 QLoRA，看「省显存」和「省参数」是两个正交的省法、能不能叠加。
