先把 QLoRA 这个名字里最容易被误读的东西说清楚：它不是一个新的微调算法。它是两条你在前两章已经验证过的事实做了一次乘法。第一条来自第 03 章——可训练的增量只要很小一块（LoRA 的 `B@A`），就够把基座掰到新任务上，所以梯度和优化器状态可以只为这一小块分配。第二条是这章要现场量给你看的——**冻结的基座可以用很低的精度存，原任务的能力几乎不掉**。把这两条乘起来：grad/optim 跟着可训练参数走（LoRA 已经砍掉了），weights 跟着基座精度走（这章再砍一刀）。QLoRA 就是这两刀同时落下。

所以这章不会有新的反向传播推导，也不会有新的注入点设计——LoRA 部分原封不动用第 03 章那套 `r=4, α=8` 注在 q/v 上。新东西只有一个：一个 toy 量化器，外加一组诚实的实验，回答"基座到底能压多低、压崩了 LoRA 能不能补"。

> 本章配套代码 `examples/peft-from-scratch/src/stage07-qlora.ts`，`seed(1234)`，deterministic（重跑数字完全一致）。下文所有数字来自这份代码的真实运行输出。**caveat 先说**：这是 ~5800 参数的 toy 模型在合成 copy→reverse 数据上的 toy 量化，位宽、误差、准确率、显存的**绝对值都偏乐观**。可迁移的是**结构**（冻结=低精度 OK 到某个位宽下限、增量=需高精度）和**相对趋势**（per-channel > global；8/4/3-bit 几乎无损、2-bit 打崩基座；LoRA 救不回被量化抹平的基座）。

## ✦ 这一章的专家钩子：QLoRA 是两条已有事实的乘法

我把这一点单独拎出来，因为它决定了你读后面所有数字时该带的视角。很多人把 QLoRA 当成"4-bit 的 LoRA"，好像 4-bit 是它的核心创新。不是。4-bit 只是个实现选择，论文里真正的工程内核是 NF4、double quantization、paged optimizer 那一套——这章的 per-channel 对称量化是它们的最小公倍数。

真正值得记住的是这个判断：**模型里的参数不是同一种东西，不该用同一种精度对待。** 冻结的基座是只读的、巨大的、误差会被后续层平滑掉一部分的；可训练的增量是要被梯度反复更新的、很小的、误差会被优化器累积放大的。前者你随便压，后者你一压它就学不动。QLoRA 的全部聪明之处就是看穿了这个不对称，并且把它落到存储布局上：基座存 4-bit，LoRA 存 fp32。

这章第 [7] 步的显存柱状图就是这层乘法的可视化。我先把数字摆出来，下文再逐项拆。

```
full-FT total │████████████████████████████████████████│ 0.111 MB (est.)
LoRA    total │██████████████████······················│ 0.049 MB (est.)
QLoRA   total │███████████·····························│ 0.031 MB (est.)
```

`full-FT → LoRA` 这一跌（0.111 → 0.049）是第 03 章的功劳——砍掉 grad+optim。`LoRA → QLoRA` 这一跌（0.049 → 0.031）才是这章的功劳——砍 weights。两刀作用在不同的项上，所以能叠加，这就是"乘法"的含义。

## 量化在做什么：把连续权重钉到几个台阶上

机制先讲透，再讲它怎么坏。

量化的核心是一条线性映射，把一段浮点区间均匀铺到整数网格上。配套用的是 symmetric quantization（对称量化，零点固定在 0）——权重大致以 0 为中心，对称方案让 0 精确可表示，反量化也最简单。看配套的 `quantDequant`（`examples/peft-from-scratch/src/core/` 之外，就在 `stage07-qlora.ts` 里）：

```ts
// k-bit 对称量化的最大可表示整数幅值，4-bit -> 7
function intMax(bits: number): number {
  return Math.pow(2, bits - 1) - 1;
}

// per-channel：每个输出行一个 scale
for (let r = 0; r < out; r++) {
  let maxAbs = 0;
  for (let c = 0; c < inDim; c++) maxAbs = Math.max(maxAbs, Math.abs(W[r * inDim + c]));
  const scale = Math.max(maxAbs / qmax, 1e-12); // floor 住 scale, 死通道反量化到 0 而非 NaN
  for (let c = 0; c < inDim; c++) {
    const idx = r * inDim + c;
    const q = Math.max(-qmax, Math.min(qmax, Math.round(W[idx] / scale)));
    recon[idx] = q * scale; // 立即反量化, 返回前向真正读到的重建值
  }
}
```

三步：`scale = absmax / qmax` → `q = round(w / scale)` → `w_hat = q * scale`。4-bit 的 `qmax=7`，所以整数网格是 `-7..7`，共 15 个台阶（`numLevels(4)`）。配套这个函数有意把量化和反量化合在一个函数里返回重建值——因为这本书从不需要打包后的整数，只需要前向真正读到的那个反量化结果，以及它跟原值的误差。

> NOTE：这里有个不变量值得记住——`scale` 被 floor 在 `1e-12`。一个全零或近零的行（死通道）如果不 floor，`absmax/qmax` 会是 0，`w/0` 直接 NaN。floor 之后死通道老老实实重建成 0。真实量化库也都有这个守卫，只是藏得深。

看配套第 [2] 步对一个真实基座矩阵 `Wq`（attention 的 q 投影，shape 24×24）量化前后的分布：

```
Wq 原始权重分布 (fp32):          Wq 4-bit per-channel 反量化分布:
-0.18 │███████████████████████████████ 121   -0.18 │████████████████████████████████ 128
-0.05 │████████████████████████████████ 124   -0.05 │███████████████████████████ 107
 0.09 │████████████████████████ 94            0.09 │██████████████████████████ 102
反量化误差 |W - dequant(W)|:  max=6.290e-2  mean=1.993e-2
```

分布的形状基本保住了，只是被"梳"到了 15 个离散台阶上，所以柱子高度略有挪动。平均每个权重错 `1.993e-2`，最坏错 `6.29e-2`。**注意这个误差是受控的，而且 per-channel scale 是受控的关键**——下文 [6] 会让你看到换成全局单一 scale 时这个数字怎么暴涨 9 倍。

## ✦ 前沿钩子拆解：per-channel scale 为什么是 NF4 那套的最小内核

⚡ 这里标一个前沿/开放问题：**量化基座的位宽下限、以及怎么把更多信息塞进 4-bit，目前没有通用最优解，仍在活跃研究。** QLoRA 论文用 NF4（normal-float 4-bit，按权重的正态分布裁剪量化格点，让格点密度匹配权重密度）而不是这章的均匀对称量化；又叠了 double quantization（把 scale 本身再量化一次省那点元数据）；再配 paged optimizer 防 OOM 峰值。这些都是工程细节，而且每隔几个月就有新方案（AWQ、GPTQ、各种 mixed-precision 的离群值保留），没有一个被公认为"终局"。

但你不需要读懂 NF4 才能理解 QLoRA 为什么可行。理解 per-channel scale 就够了，因为它是所有这些方案共享的那个内核思想：**量化的敌人是动态范围不均，解药是把 scale 切细到误差能被隔离的粒度。**

配套第 [3] 步把这件事量成了一张扫描表——对全部 7 个线性层量化、不训练，看冻结基座还能不能解原任务 A（copy）：

```
bit  per-channel   global      (clean fp32 = 100.0%)
8-bit     100.0%      100.0%
4-bit     100.0%      100.0%
3-bit     100.0%       98.6%
2-bit      82.0%       35.5%
```

三个结论，每个都对应一句可迁移的判断：

1. **8/4/3-bit 几乎无损**——这就是"冻结基座可低精度"的实证。原任务 A 在量化后仍 100%（3-bit per-channel 也 100%）。冻结的基座对量化误差有惊人的容忍度，因为这些误差不会被梯度累积，只是一次性地、被后续层部分平滑掉。
2. **per-channel 全面优于 global**——3-bit 时 global 已经开始掉（98.6%），2-bit 时 global 崩到 35.5% 而 per-channel 还有 82%。差距随位宽变窄而拉大。
3. **位宽有下限**——2-bit 连 per-channel 都开始崩（82%）。"冻结可低精度"不等于"可任意低"。这是个硬边界，不是渐变。

第 2 条是 per-channel 价值的全部证据，也是 NF4 那套的逻辑起点：scale 越粗（global = 整个矩阵一个 scale），一个大幅值的行就能绑架所有其他行的分辨率；scale 越细（per-channel = 每行一个），大幅值行被隔离在自己那行里，不污染别人。NF4 是在这个基础上再把"格点该放哪"也按分布优化了一层，但隔离的思想是同一个。

## 跑 QLoRA：同一份 LoRA，只换基座的存储精度

现在把两边接起来。配套 [4] 和 [5] 做的是严格对照实验：同一个预训练基座、同一个 seed、同样 40 epoch、同样的 LoRA（`r=4, α=8` on q/v）——**唯一的变量是基座存 fp32 还是存 4-bit per-channel**。

LoRA 的封装代码跟第 03 章一脉相承，关键是它读的是基座自己的 `W`，而那个 `W` 在 QLoRA 路径里已经被原地量化过了：

```ts
class LoraLinear extends Module {
  base: Linear; // 冻结; QLoRA 路径下 W 已被反量化覆盖
  loraA: Tensor; // (r, in) 可训练, normal-init
  loraB: Tensor; // (out, r) 可训练, ZERO-init -> delta 从 0 起步
  override forward(x: Tensor): Tensor {
    const baseOut = x.matmul(this.base.W.transpose()).add(this.base.b);
    const lowRank = x.matmul(this.loraA.transpose()).matmul(this.loraB.transpose()).scale(this.scale);
    return baseOut.add(lowRank); // 高精度增量 ADD 到(低精度)基座输出上
  }
}
```

量化发生在 LoRA 包上去之前，就一句 `quantizeBaseInPlace(baseQ, 4, "per-channel")`，它把每个基座 Linear 的 `W.data` 直接覆盖成反量化重建值。**这正是真实 QLoRA 前向的简化模型**：基座以 4-bit 常驻，读的瞬间反量化成高精度参与 matmul，LoRA 的低秩增量始终是 fp32，最后 add 在一起。`base.W.requires_grad = false`——量化误差因此永远不进梯度，这是它能容忍的根本原因。

> NOTE：配套用 monkey-patch 把 `attn.q.forward` / `attn.v.forward` 换成 LoRA 包的 forward，而不是去改 `core/nn.ts`。这么做是为了让注入点局部、显式，不污染共享的核心模块——是教学取舍，真实库会用更正规的 hook 机制。

对照结果（第 [4][5] 步）：

```
                  task-A 保持   task-B (LoRA=0 起点 → 训练后)
全精度 LoRA        100.0%       4.2% → 9.1%
QLoRA(4-bit pc)    100.0%       4.2% → 12.2%
```

可训练参数 `384 / 5844`（6.57%），两条路径完全相同。两个要看的点：

- **基座 task-A 保持 100%**——4-bit 量化没碰掉基座原本会的技能。这是 QLoRA 的前提条件，配套每次都先验证它，因为如果这条破了，后面 task-B 学得再好也没意义（你交付的是个忘了原本能力的模型）。
- **task-B 上 QLoRA 12.2% vs 全精度 9.1%，差 -3.1pp**——方向看着反了（量化的反而更高），但别过度解读。reverse 任务对 q/v-only 的 LoRA 本就难，两者都偏低、都在噪声带里，3pp 是预期内的 run-to-run 波动而非"量化更好"。**诚实的结论是：4-bit 基座没让 task-B 的学习能力可观地变差。** 这才是 QLoRA 要证的——省了显存，微调效果不塌。

task-B 的训练 loss 曲线（第 [5] 步）单调下行，跟全精度 LoRA 是一个形状：

## 失败模式：2-bit + 全局 scale 把基座打崩，LoRA 救不回

每个机制都得讲它怎么坏，不然你不知道边界在哪。这章的失败模式是同时踩两个坑：位宽压到 2-bit，**并且**用全局单一 scale（而不是 per-channel）。这两个都是真实场景里有人会图省事去做的——"反正都是量化，2-bit 更省，全局 scale 元数据更少"。

配套第 [6] 步先量误差：

```
Wq 2-bit global 反量化误差:  max=4.430e-1  mean=1.822e-1  (对比 4-bit pc mean 1.993e-2, 放大 9.1×)
Wq 2-bit global 反量化分布 (台阶塌缩):
-0.89 │█ 24
-0.00 │████████████████████████████████ 531
 0.74 │█ 21
```

看这个分布——这是这章最重要的一张图。15 个台阶塌成了 3 个：几乎所有权重（531 个）被钉到 0，只有两端各一小撮幸存。**机理**：全局 scale 由整个矩阵的 absmax 决定，少数大幅值权重把 scale 撑大，于是 2-bit 那点可怜的分辨率被这几个离群值吃光，剩下绝大多数中小权重全部 round 到最近的格点——也就是 0。这是 global scale 的死法，per-channel 之所以能扛是因为它把每一行的 absmax 隔离开了。

误差暴涨的后果（第 [6] 步）：

```
基座 task-A 保持率: 4-bit pc 100.0%  →  2-bit global 35.5%  (基座能力被抹平)
训练 LoRA(任务 B) 后再测 task-A = 12.1%  — LoRA 救不回被量化打崩的基座能力
```

这里有个关键实验，比"准确率掉了"深一层：基座被 2-bit global 打崩后（task-A 只剩 35.5%），我们照常在它上面训 LoRA 学 task-B，训完**再回头测 task-A**——只剩 12.1%。LoRA 没把基座的原能力救回来，反而因为它是为 B 优化的，把残存的一点 A 也挤掉了。

为什么救不回？因为 **LoRA 只能在 q/v 上 ADD 一个低秩增量，它没有能力重建一个被量化抹平了内部表征的基座。** LoRA 的数学本质是"在原基座旁边加一个小修正项"，前提是原基座是好的、只需要微调。当基座本身的表征被 2-bit 量化压成了一团（531 个权重都是 0），那不是"需要微调"，那是"信息已经丢了"，加多少低秩项都补不回丢掉的信息——尤其这个低秩项还只动 q/v 两个矩阵、还是朝着另一个任务优化的。

> WARNING：这条边界是 QLoRA 实践里最容易踩的雷。它给人的错觉是"量化是免费的、压得越低越省"，于是有人一路压到 2-bit 还纳闷为什么微调效果崩了。崩的不是微调，是基座，而且在你开始训 LoRA 之前就已经崩了。**先验证量化后基座的原任务保持率，再训 LoRA**——配套每条路径都先做这步，不是多余的。

教训一句话：per-channel scale 隔离大幅度行，位宽有下限；冻结基座可低精度，但不是任意低。

## 把显存账算清楚：三项里 QLoRA 动的是哪一项

回到开头那张柱状图，现在逐项拆。配套 `core/mem.ts` 的 `estBytes` 把训练显存拆成四项，其中三项是这本书反复算的主线：

```
weights         = frozen * bytesPerBaseParam + trainable * bytesPerParam
gradients       = trainable * bytesPerParam            (只有可训练参数有梯度)
optimizer state = optMultiplier * trainable * bytesPerParam   (Adam=2: m 和 v)
```

> NOTE：这是纯算术估算，不分配任何显存——真实 PEFT 的显存故事是十亿参数模型训练时的 GPU VRAM，笔记本上既不能也不该真去分配。所以配套算的是真实场景同样遵守的那个公式，套在 toy 参数量上。柱子的**形状**（full-FT 高耸、PEFT 矮）是可迁移的真相，toy scale 的绝对 MB 没意义，标了 (est.)。

三种方案的差别全在公式里两个参数的取值：

| 方案 | weights | grad + optim | 公式里改了什么 |
|------|---------|--------------|----------------|
| full-FT | 0.022 MB | 0.067 MB | `trainable = total`，全员有梯度+优化器 |
| LoRA | 0.022 MB | 0.004 MB | `trainable` 砍到 6.57%，grad+optim 跟着塌 |
| QLoRA | 0.004 MB | 0.004 MB | 在 LoRA 基础上 `bytesPerBaseParam = 0.5`（4-bit）|

权重项的细分（第 [7] 步）：

```
full-FT weights │████████████████████████████████████████│ 0.022 MB
LoRA    weights │████████████████████████████████████████│ 0.022 MB (fp32 base)
QLoRA   weights │███████·································│ 0.004 MB (4-bit base)
```

这张图把"乘法"讲死了：

- **LoRA 和 full-FT 的 weights 一样高（都 0.022 MB）**——因为 LoRA 没动权重，冻结的基座照样满精度常驻。LoRA 砍的是 grad+optim（0.067 → 0.004），不是 weights。
- **QLoRA 的 weights 塌到 0.004 MB**——4-bit = 0.5 字节/参数，冻结基座的存储除以 8。这是 LoRA 碰都没碰的那一项。
- **QLoRA 的 grad+optim 和 LoRA 完全相同（都 0.004 MB）**——因为可训练参数没变，还是那 384 个 LoRA 参数。QLoRA 不改 LoRA 的训练侧，只改基座的存储侧。

所以两刀作用在正交的项上：LoRA 砍 grad+optim（训练侧），QLoRA 在此之上再砍 weights（存储侧）。这就是为什么工业界把它们组合用——它们不抢同一块收益，是真正叠加的。`0.111 → 0.049 → 0.031` 这条下降链，每一段都对应公式里一个不同的项被压下去。

## 小结与接下来

这章其实没教新算法，教的是一个判断：**参数不是同一种东西，冻结的可以低精度、可训练的要高精度。** QLoRA 把这个判断落到存储布局上，于是省显存的两刀（LoRA 砍训练侧、量化砍存储侧）能正交叠加。配套的 toy 量化是 NF4 那一整套工程的最小内核——理解了 per-channel scale 怎么隔离离群值、为什么 2-bit global 会让台阶塌缩，你就有了读懂任何量化 PEFT 论文的地基。

留两个边界在脑子里：冻结基座可低精度但有位宽下限（2-bit 开始崩），以及 LoRA 救不回被量化打崩的基座（信息丢了就是丢了，低秩增量补不回）。

⚡ 还有那个开放问题：量化的最优位宽、最优格点分布、离群值该怎么保留——目前没有通用最优解，NF4 / AWQ / GPTQ 各有取舍，每隔几个月就有新方案，这是个活跃的研究前沿，不是已经盖棺的工程问题。

下一章（第 08 章）收尾整个 PEFT 链路：现在你手里有一个 4-bit 基座 + 一组 fp32 LoRA 增量，训练时这么分开存很省，但**交付时**呢？推理服务不想每次前向都反量化基座、再额外算一遍 LoRA 的低秩乘法。第 08 章讲怎么把 LoRA 合并（merge）回一份满精度权重——把 `W + (α/r)·B@A` 算实、固化成一个普通的 Linear，让部署侧拿到的就是一个干净的、没有任何 PEFT 痕迹的模型。承接这章留下的"训练时分开、交付时合并"这个收尾动作。
