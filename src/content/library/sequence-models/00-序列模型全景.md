图像有一个序列没有的奢侈:它的全部信息一次性摆在你面前。一张 224×224 的图,卷积核第一层就能同时看到所有像素,空间上哪两个点离得近是固定的、可以静态布线的。序列不是。一句话、一段 DNA、一串传感器读数,是**一个一个**到达的,而且「哪个早先的元素跟当前这个相关」事先不知道——主语和它的谓语可能隔 3 个词,也可能隔 30 个词。所以序列模型从第一天起就要回答一个图像模型可以回避的问题:**当第 t 个元素到来时,前面 t-1 个元素里哪些还需要记着,记在哪?**

记在哪——这就是全书的轴心。你不可能把「前面所有元素」原样存下来再处理,那样状态会随序列长度无限膨胀。所以序列模型的本质动作是**压缩**:把任意长的历史压进一个**固定大小**的东西里,带到下一步去。这个固定大小的东西,RNN 叫隐状态(hidden state),LSTM 叫 cell state,SSM 叫 state,Transformer 干脆赌气说「我不压,我全留着」——而它为这个倔强付的代价,正是本书后半段的主线。

**一句话定调:序列建模 30 年(1990s 的 RNN 到 2023 的 Mamba)的全部争论,是「怎么把任意长历史压进固定状态」这一个问题的不同答案。** 这本书不带你看 survey 式的「模型动物园」,它带你沿这一条矛盾,从零用纯 TypeScript 把每种答卷手写出来、跑出来、看它在哪儿坏。配套代码在 `examples/sequence-models-from-scratch/`,每章一个能独立跑、几秒到几十秒出结果的 stage,结尾打印代码真算出来的数字。本章不配 stage,它是全书的地图和坐标系。

## 三条坐标轴:后续每章都在这上面定位

把「压缩历史」这个动作拆开,会发现它在三个维度上各有一笔账。这三笔账互相拉扯,没有哪个模型能三个都赢——这正是范式不停转移的原因。先把三轴说死,后面每章你都用它们给新模型定位。

### 轴一:记忆机制——隐状态怎么携带历史

「隐状态」(hidden state,就是模型一路带着走的那点记忆)是怎么更新的,决定了它能记多久、记什么。这一轴上有三种质地不同的更新方式:

- **乘性覆盖**(RNN):新状态 = 一个矩阵乘旧状态,再混入新输入,过一个非线性。旧信息每步都被一个矩阵「重写」一遍。
- **加性保留 + 门控**(LSTM / GRU):新状态 = 旧状态**加上**一点增量,加多少由「门」(gate,一个 0~1 的闸)按内容决定。旧信息默认留着,除非门主动擦掉。
- **直接寻址**(Attention):不压缩,把全部历史摆着,当前元素按「内容相似度」去里面查需要的那条。没有「状态」这个瓶颈。

这三种从「每步全量重写」到「按需保留」到「干脆不压」,记忆能力递增——但你马上会看到,记忆能力的每一分提升,都在另外两轴上记了账。

### 轴二:训练可行性——梯度能不能穿越时间

模型要学,得让损失的梯度从序列末端一路传回开头,告诉第 1 步的参数「你当时该怎么做」。这个「传回开头」在 RNN 里要穿过几十个时间步,而**每穿一步就乘一次同一个矩阵**。乘几十次同一个东西,数学上只有两种结局:要么趋近 0(梯度消失,vanishing),远端的参数收不到信号,学不会长依赖;要么炸到无穷(梯度爆炸,exploding),直接 NaN。这是第 02 章整章在解剖的事。

这一轴解释了一个反直觉的历史事实:**RNN 在 1990 年就有了,但「能训得动」要等到 1997 年的 LSTM。** 不是想不到要记长历史,是梯度穿不过去,记的能力训不出来。

### 轴三:计算复杂度——每步代价随序列长 n 怎么涨

第三笔账是钱(算力)。处理一条长度 n 的序列,总计算量随 n 怎么长?

- **RNN / LSTM / SSM**:每步只跟固定大小的状态打交道,处理整条序列是 **O(n)**——线性,理想。但有个隐藏代价(轴二、轴一会还):状态是顺序依赖的,第 t 步必须等第 t-1 步算完,**无法在时间维度并行**。
- **Attention**:每个位置要跟其它所有位置算相似度,n 个位置两两配对是 **O(n²)**——平方,序列翻倍计算量翻四倍。但它**可以并行**:所有位置的注意力同时算,没有顺序依赖。

看清这张账没有?**O(n) 但串行 vs O(n²) 但并行。** 这就是过去十年序列建模的核心张力,也是本章末尾⚡要讲的、为什么 2023 年之后大家又绕回来的根。

| 模型 | 轴一·记忆机制 | 轴二·梯度穿越时间 | 轴三·复杂度 | 轴三·能否并行训练 |
|------|--------------|------------------|------------|------------------|
| **RNN** (ch.01-02) | 乘性覆盖,每步重写 | 消失/爆炸,长依赖学不会 | O(n) | 否(状态串行) |
| **LSTM/GRU** (ch.03) | 加性保留 + 门控 | 加性路径=梯度高速路,缓解消失 | O(n) | 否(状态串行) |
| **Attention** (ch.05) | 直接寻址,不压缩 | 无递归,梯度一跳到位 | **O(n²)** | **是** |
| **SSM** (ch.06) | 线性递归状态 | 线性=可展开成卷积绕过 | O(n) | **是**(扫描可并行) |
| **Mamba** (ch.07) | 选择性递归(门控+递归) | 同上 | O(n) | 是(硬件感知扫描) |

读这张表的正确姿势:**横着是模型,竖着是三笔账,而每一次范式转移都是为了买下某一格、付出另一格。** LSTM 买下了轴二(梯度能传了),没动轴三(还是串行 O(n))。Attention 买下了轴二和「并行训练」,代价是轴三炸成 O(n²)。SSM / Mamba 想把轴三买回 O(n)、又不丢并行——这是最难的一格,也是全书的终点。

> **诚实声明(贯穿全书)**:本书在 toy 任务(copyTask 复制任务、加法问题、字符级语言模型等,见 `src/core/data.ts`)上把机制讲到底。toy 模型只有几万参数、几百到几千步训练,所以**绝对的 loss / 准确率数字偏乐观**。能迁移到真实大模型的是**机制与曲线形状**:梯度 norm 随距离的衰减斜率、耗时随 n 的标度斜率、长依赖学得会还是学不会的相变点。代码里能 wall-clock 的真测(`performance.now()`),解析估算的标注,断言的阈值都是代码里写死、运行时会真触发的。所有随机走 `src/core/prng.ts` 的种子 PRNG,同种子同机器 bit 级一致。

## ✦ 最容易混的两件事:参数共享 ≠ 状态递归

讲 RNN 时,从业者最常把两个独立的性质搅成一团,导致后面理解 Mamba 时整个拧巴。这里钉死。

**参数共享(parameter sharing):同一组权重,在每个时间步复用。** RNN 不是「序列多长就有多少层不同的权重」,而是**一个 cell,反复调用**。看配套仓 `src/stage01-rnn.ts` 里的 `RNNCell`:

```typescript
// src/stage01-rnn.ts —— h_t = tanh( x_t @ W_xh + h_{t-1} @ W_hh + b_h )
export class RNNCell extends Module {
  private inToHidden: Linear;     // W_xh : [inDim, hiddenDim]
  private hiddenToHidden: Linear; // W_hh : [hiddenDim, hiddenDim]
  private bias: Tensor;           // b_h

  /** One recurrent step. x:[B,inDim], hPrev:[B,hiddenDim] -> hNext:[B,hiddenDim]. */
  step(x: Tensor, hPrev: Tensor): Tensor {
    const fromInput = this.inToHidden.forward(x);
    const fromState = this.hiddenToHidden.forward(hPrev);
    return fromInput.add(fromState).add(this.bias).tanh();
  }
}
```

整条序列的处理,就是在一个普通 JS 循环里对**同一个 cell 实例**调 `step()` 几十次。`stage01-rnn.ts` 的源码注释把这点说得很重:「the SAME cell instance is called at every timestep。Re-instantiating the cell per step would silently give each step its own untied weights — no longer an RNN.」——每步换一组新权重,它就不是 RNN 了。

参数共享带来的直接后果:**参数量 O(1),不随序列长度 n 涨。** 处理长度 10 和长度 10000 的序列,RNN 的参数一样多。这是 RNN 相对「为每个位置配一套权重」的朴素做法的巨大胜利,也是它能处理变长输入的根。

**状态递归(state recurrence):隐状态 h_t 依赖 h_{t-1},顺序产生,无法并行。** 这是另一件事。即便权重共享、即便每步计算很便宜,你也**必须**先算出 h_1 才能算 h_2,先算 h_2 才能算 h_3。第 t 步的输入是第 t-1 步的输出。

把这两件事掰开,RNN「训练慢、又难并行」的根因就清清楚楚:

- **参数共享(轴一,好事)** → 参数省、能处理变长。
- **状态递归(轴二+轴三,坏事)** → 反向传播时,同一个 `W_hh` 在所有时间步被乘了一遍又一遍(这是梯度消失的算术根源,第 02 章详解);前向时,时间步之间有数据依赖,**喂不满 GPU**(这是训练慢的根源)。

为什么这条要在全景章就钉死?因为**理解 Mamba 全靠它**。后面你会看到,Transformer 砍掉了「状态递归」(所以能并行训练),但代价是把记忆从「固定状态」变成「全部历史」(所以 O(n²));而 Mamba 想把「状态递归」请回来拿 O(n),又得想办法绕开它「不能并行」的诅咒——它绕开的方式,恰恰是利用了 RNN 没有、SSM 才有的一个性质(线性递归可展开成卷积/扫描)。**不把「参数共享」和「状态递归」分清,你会以为 Mamba 只是「更快的 RNN」,而错过它真正的创新在哪一轴。**

## 30 年三次范式转移:每一次买了哪格、付了哪格

把三轴的账连起来读,序列建模的历史就是一条清晰的线,而不是一堆并列的模型名。

### 第一次转移:RNN → LSTM(1997)——买下轴二

vanilla RNN 的隐状态每步被 `W_hh` 乘性重写一遍。反向传播穿过 L 个时间步,梯度就是 L 个雅可比矩阵连乘,由 `W_hh` 的谱半径 ρ(spectral radius,矩阵最大特征值的模,衡量它连乘 L 次后是放大还是缩小)主宰:ρ<1 梯度按 ρ^距离 指数衰减→消失,ρ>1 指数增长→爆炸。配套 `src/stage02-vanishing-grad.ts` 在**一次真实的前向+反向**里逐时间步量出 `||dL/dh_t||` 画成曲线,把这件事从断言变成数字。

LSTM 的招不是调参,是**改递归的代数**。它把状态更新从「乘性覆盖」改成「加性保留 + 门控」:cell state 默认**加**一个增量带到下一步,forget gate(遗忘门)接近 1 时旧信息几乎原样保留。于是 c_t 到 c_0 的路径是一串接近 1 的门值连乘,而不是一串接近 0 的 tanh 导数连乘——这条加性路径就是著名的「constant error carousel」(恒定误差传送带),梯度的高速公路。`stage03-lstm-gru.ts` 把 RNN 和 LSTM 的状态梯度衰减曲线画在一起,LSTM 的衰减「orders of magnitude gentler」(温和好几个数量级)。

代价?**轴三纹丝没动。** LSTM 还是串行的 O(n),还是喂不满 GPU。而且——这点很多人忽略——`stage03` 的源码注释明确做了第三个实验证明:门控修的是**梯度消失**,**不**给无限记忆。把序列拉到超过固定宽度状态能缓冲的长度,LSTM 准确率照样掉。固定宽度的状态向量,本身就是个瓶颈。**这个瓶颈,正是 attention 要拆的东西。**

### 第二次转移:LSTM → Attention(2017)——买下并行,炸掉轴三

Attention 的根本动作是**取消压缩**。RNN/LSTM 必须把「以后可能用到的一切」挤过一个固定宽度的隐状态,每步还覆盖一次;在复制任务上,第 0 步的关键 token 要熬过 `delay` 次覆盖才能到答案区。Attention 不玩这个:答案位置直接对**每一个**更早的位置算相似度,一跳读到匹配的内容。没有信息挤压,没有衰减路径。`src/stage05-attention.ts` 的注释一句话点透:「That is why one O(n^2) operation displaced the recurrent stack」——一个 O(n²) 操作干掉了整个递归栈。

这一刀同时买下了两格:**轴二**(梯度一跳到位,不用穿几十步)和**并行训练**(所有位置的注意力同时算,没有状态递归的顺序依赖,GPU 喂得满)。这是 Transformer 统治 2017–2023 的真正原因——不只是「效果好」,是「效果好**且**能在大集群上高效训」。

但账记在轴三。`stage05` 同时演示了那堵「成本墙」:序列长度翻倍,attention 的前向 MAC(乘加次数)和墙钟时间约**翻四倍**。代码用 `src/core/metrics.ts` 的 `countMACs.attention` 解析算出 MAC(`2·n²·d`),又真测墙钟,对二者各拟合一条 log-log 斜率——**都落在 2.0 附近(O(n²))**,而一个递归扫描会落在 1.0 附近。这条「斜率 2 vs 斜率 1」的对比,就是下一次转移要解决的具体痛点。

> 注意这笔账的性质:O(n²) 在短序列(几百 token)上无所谓,在长序列(几万 token 的文档、基因组、音频)上是致命的。这正是为什么「长上下文」一直是 Transformer 的老大难,也是 SSM 复兴的市场窗口。

### 第三次转移:Attention → SSM / Mamba(2021–2023)——把轴三买回 O(n),不丢并行

SSM(state-space model,状态空间模型)表面看就是回到了线性递归:`h_t = A·h_{t-1} + B·u_t`,`y_t = C·h_t`。这不就是 RNN 吗?**关键区别在「线性」**——步与步之间**没有 tanh**。`src/stage06-ssm.ts` 把这件事的威力讲透:因为递归是线性的,同一个计算可以**展开成一个卷积**,kernel `K[k] = C·A^k·B`。递归形式是 O(n) 内存、串行;卷积形式是 O(n) 计算、**在时间维度并行**。**它俩是同一个函数**——而 `stage06` 不是嘴上说,是数值验证二者输出一致到 `err < 1e-8`(机器精度)。

看懂这个「递归↔卷积」对偶没有?**它一举解决了 RNN 当年解决不了的事:既要 O(n) 的线性复杂度(像 RNN),又要能并行训练(像 Attention)。** 这是 RNN 做不到的(tanh 非线性让你没法展开成卷积),也是 Attention 用 O(n²) 才换到的。SSM 用「线性递归」这个看似倒退的设计,反而拿到了两全。

但纯线性 SSM 有个致命短板,Mamba 来补。线性时不变(LTI,所有时间步用同一组 A/B/C)意味着**它没法根据内容决定记什么**——每个 token 用同样的增益写进状态,该忽略的噪声和该记住的关键 token 一视同仁。`src/stage07-mamba.ts` 专门构造了一个**没有内容选择就解不了**的任务:一个关键 token 藏在一堆同词表的干扰 token 里。Mamba 的贡献是把 Δ/B/C 变成**输入的函数**(input-dependent,即「选择性」selective),于是模型能在遇到关键 token 时开闸写入、遇到干扰时关闸跳过。`stage07` 训三个模型对比——选择性 SSM 能解、固定 SSM 不能解(噪声覆盖状态)、LSTM 也吃力——并做消融(把选择 Linear 冻成常数)证明**是选择性这个变量在起作用**,不是「SSM 更大」。

> **诚实边界**:`stage07` 的注释自己声明,它的扫描是「sequential Python-style loop over timesteps,NOT the hardware parallel scan」——刻意用可读的逐步递归,好把门 Δ_t 随时间打印成热力图给你看清机制。真实 Mamba 的头条性能需要 GPU + 长序列 + 硬件感知的并行扫描,本书**不**实现那一层。能迁移的是机制(输入相关门控)和 O(n) 标度,不是 SOTA 速度数字。

## ⚡ 开放问题:这一圈绕回来,到底证明了什么?

把三次转移连起来看,序列建模在「并行训练 vs 线性推理」之间**绕了一整圈又回到原点附近**:RNN(O(n) 但串行)→ Attention(并行但 O(n²))→ SSM/Mamba(O(n) 又并行)。乍看像复古,其实是综合——Mamba 拿的是 RNN 的复杂度形状 + Attention 的并行性 + 一个两者都没有的「内容选择性门控」。**理解 RNN 为什么慢、为什么训不动,你才能理解 Mamba 的每一个设计都是在精准地拆掉 RNN 的某一个具体诅咒。这就是为什么本书从 RNN 讲起,而不是直接上 Mamba。**

但故事**远没有收尾**——这是本书唯一一个我明确标注「目前无通用解、正在研究」的问题:

**SSM/Mamba 真能取代 Transformer,还是只在某些任务上更优?目前没有定论。** 已知的张力是:Transformer 的 attention 是**精确**的全局召回(任何位置都能一跳读到任何历史位置的精确内容),而 SSM 的固定状态本质上是**有损压缩**——它在「需要精确逐字召回长上下文」的任务(比如从一篇长文里抄出某个具体 ID、in-context learning 里精确复制示例)上,结构性地弱于 attention。这正是本书 `copyTask`(精确复制)能把不同模型的记忆能力区分开的原因:**精确复制是压缩式记忆的天敌。** 2024 年起的主流工程答案是「不要二选一」——做**混合架构**(hybrid),大部分层用便宜的 Mamba/SSM 拿 O(n),少数几层插 attention 补精确召回。但「该插几层 attention、插在哪、什么任务下纯 SSM 就够」**没有通用答案**,仍在各家实验室用经验摸索。

换句话说:三轴的账到今天也没有谁能三格全赢。Transformer 在「精确召回」那一格还没被买下,SSM 在「O(n) 长上下文」那一格暂时领先。**下一次范式转移会发生在哪一格,正是这个领域当下最值钱的开放问题。** 读完这本书,你不会拿到这个问题的答案——没人有——但你会拥有判断任何新架构「它在三轴上买了哪格、付了哪格」的坐标系。这比记住任何一个模型名都耐用。
