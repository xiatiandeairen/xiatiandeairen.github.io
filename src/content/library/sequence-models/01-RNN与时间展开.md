先把一个常见误解摆上台面:很多人以为 RNN(循环神经网络)"天然就能记住很长的历史",因为它"每一步都把过去带着走"。本章用一个能跑的实验直接证伪——同一个 64 维隐状态的 vanilla RNN,在依赖跨度 9 步的任务上留出测试准确率 100.0%,把任务跨度拉到 39 步,不改模型、不改训练步数,准确率塌到 30.9%(随机基线 16.7%),loss 卡在平台上从未逼近 0。"带着过去走"是真的,但隐状态容量是固定的,而且每步都被 tanh 压一遍,有效记忆窗口其实很短。这是本章要讲透的第一件事。

第二件事是关于训练代价的:RNN 慢不是因为实现烂,是结构性的。隐状态 `h_t` 依赖 `h_{t-1}`,反向传播必须按时间倒序一步步串行走完整条链,没法在时间维度并行。同样 600 步,短依赖(T=12)跑 3.045 秒,长依赖(T=42)跑 10.645 秒——序列变长 3.5 倍,墙钟时间也涨 3.5 倍,线性正比于时间步数,因为那条反传链只能一个一个 timestep 倒着走。这条串行链就是 BPTT(Backpropagation Through Time,随时间反向传播),也是后面所有架构想方设法要绕开的东西。

本章代码全部对应 `examples/sequence-models-from-scratch/src/stage01-rnn.ts`,数字都是这个文件 `main()` 真实跑出来的(`performance.now()` 计时、留出测试集评估)。下面边讲机制边对照代码。

## 一组权重,reapply 在每一步

RNN 的全部秘密就一行递推式:

```
h_t = tanh( W_xh · x_t + W_hh · h_{t-1} + b_h )
```

`h_t` 是 t 时刻的隐状态(hidden state,可以理解成"模型到目前为止记住的东西"),`x_t` 是 t 时刻的输入。关键在于 `W_xh / W_hh / b_h` 这三组权重**不随 t 变**——同一组权重在每个时间步被重新应用一次。这个"权重在时间上共享"才是循环网络的本质,不是"有个 for 循环"。

代码里 `RNNCell` 就是这一行,用两个无 bias 的 `Linear`(线性层,即矩阵乘)加一个共享 bias 拼出来:

```typescript
// stage01-rnn.ts:81-85
step(x: Tensor, hPrev: Tensor): Tensor {
  const fromInput = this.inToHidden.forward(x);      // W_xh · x_t
  const fromState = this.hiddenToHidden.forward(hPrev); // W_hh · h_{t-1}
  return fromInput.add(fromState).add(this.bias).tanh();
}
```

注意 `W_hh` 用的是 orthogonal(正交)初始化,`W_xh` 用 xavier。为什么这里要特意挑正交矩阵?因为正交矩阵的所有奇异值都等于 1,反复乘它既不放大也不缩小隐状态的范数(norm,可理解为向量长度)——这是反传链最温和的起点。本章故意选这个最有利的起点,正是为了让第 02 章的结论更有说服力:即便起点这么干净,tanh 的饱和还是会把梯度推向消失。这是一个"我先把对手的条件拉满,它还是会坏"的论证结构。

### 为什么同一个 cell 实例必须在每步复用

代码里有一条容易被忽略但绝对不能违反的不变量:**整条序列从头到尾调用的是同一个 `RNNCell` 实例**。

```typescript
// stage01-rnn.ts:125-135 forwardUnroll
let h = this.cell.initialState(batch);     // h_0 = zeros
for (let t = 0; t < T; t++) {
  const x = oneHotRows(batchTokens[t], this.vocab);
  h = this.cell.step(x, h);                // 同一个 this.cell,每步复用
  logitsByStep[t] = this.readout.forward(h);
}
```

如果你图省事在循环里每步 `new RNNCell(...)`,那每一步就有了各自独立的、互不绑定的权重——那已经不是 RNN 了,而是一个深度等于序列长度的前馈网络,参数量随序列长度爆炸,也学不到"跨时间复用的规律"。权重共享是 RNN 用固定参数量处理任意长度序列的根本前提:序列长 12 还是长 42,参数量都是 5192(隐藏维度 64),这正是代码实测的参数量。

输入这里用 one-hot 向量(独热编码,每个 token 对应一个只有一位为 1 的向量)而不是可学习的 embedding,是刻意的:本章唯一能"记忆"的可学习参数只在循环权重里,这样失败模式才能干净地归因到 recurrence 本身,而不是某个 embedding 偷偷替模型干了活。

## 时间展开:那个 for 循环就是 BPTT 的全部

这是本章最值得停下来想清楚的一点。`forwardUnroll` 里那个朴素的 JS for 循环,**每迭代一次就给自动微分(autograd)的计算图接上一节**。循环跑完,这张图是一条长度为 T 的链:`h_0 → h_1 → ... → h_T`,中间每一节都挂着同一组共享权重。

当你对 loss 调 `backward()`,autograd 沿这张图反着走回去——这就是 BPTT。它不是一个需要你单独实现的算法,而是"在展开的计算图上跑普通反向传播"的别名。本仓的 `Tensor` 自己实现了 autograd,所以你写 forward 循环时 BPTT 是"免费"得到的。

真正需要理解的是梯度怎么累加。共享权重 `W_hh` 在 T 个时间步里被用了 T 次,所以反传时 T 个时间步各自产生一份对 `W_hh` 的梯度贡献。autograd 的规则是 accumulate-not-overwrite(累加而非覆盖),它把这 T 份梯度**加**进同一个叶子张量。这个"沿时间求和"就是 BPTT 的数学核心。代码里对应一条纪律——每次 backward 前必须清零:

```typescript
// stage01-rnn.ts:224 训练循环内
opt.zeroGrad(); // grads accumulate; must clear before each backward
```

忘了 `zeroGrad`,上一步的梯度会和这一步的累加在一起,训练直接发散。这不是花哨 bug,是自动微分累加语义的直接后果。

### ✦ 串行反传是 RNN 训练慢的结构性原因

现在把"慢"这件事讲死。前向时,`h_t = f(h_{t-1}, x_t)`,要算 `h_5` 必须先有 `h_4`,先有 `h_3`……前向就是串行的。反向更糟:`∂loss/∂h_t` 依赖 `∂loss/∂h_{t+1}`,梯度必须从最后一个时间步往前一步步倒着传,**时间维度上无法并行**。

实测把这个结构性代价摆出来了:

| 任务 | 序列长度 T | 依赖跨度 | 训练 600 步墙钟 |
|---|---|---|---|
| 短依赖 | 12 | 9 步 | 3.045 s |
| 长依赖 | 42 | 39 步 | 10.645 s |

T 涨 3.5 倍,时间涨约 3.5 倍——墙钟时间线性正比于序列长度,因为那条反传链的长度就是 T,而它只能一节一节走。换句话说,RNN 的训练吞吐被序列长度卡死,你有再多 GPU 核心也没用,因为时间步之间存在真实的数据依赖,不是"代码没写好"。

这是一个贯穿全书的张力:**记忆靠递归(每步依赖上一步),而递归就意味着串行**。Transformer(第 05 章)用注意力把序列内所有位置一次性铺开做并行,代价是显存和计算量随长度平方增长;而 RNN 用线性的计算量换来了串行的训练。两条路各有各的死穴。

### ⚡ 前沿:能不能让"递归"也并行训练?

这正是当前序列建模最活跃的开放方向之一。直觉上递归必须串行,但如果递推式是**线性**的(去掉 tanh 这种逐元素非线性),`h_t = A·h_{t-1} + B·x_t` 这种形式可以用 parallel scan(并行扫描,一种把"前缀累积"在 log T 深度内算完的并行算法)在时间维度上并行展开——这就是 Mamba / 现代 SSM(状态空间模型)绕开 BPTT 串行链的核心招式。

但代价不是没有:把非线性递归改成线性递归,会不会损失 vanilla RNN 那种逐步非线性混合带来的表达力?在哪些任务上等价、哪些任务上有差距?**目前没有通用结论,仍在活跃研究中**——这也是为什么 Transformer 至今没被线性递归模型完全取代。本章你亲手写的 `forwardUnroll` 串行链,到第 07 章你会看到它被改写成可并行的 scan,届时这个 trade-off 会变得很具体。先记住:本章的串行链不是历史包袱,它是后面所有"如何绕开它"工作的参照物。

## 实验:同一个模型,改一个旋钮就从能学滑向学不动

要证明"有效记忆窗口很短",最干净的做法是只动一个变量。我们用 copyTask:模型看到 k 个随机 token,然后 `delay` 个空白,再一个分隔符,最后必须把开头那 k 个 token 原样复述出来(代码见 `core/data.ts` 的 `copyTask`)。依赖跨度由单个旋钮 `delay` 控制,模型、隐藏维度、训练步数全部固定,**只有 delay 在变**。

```typescript
// stage01-rnn.ts:287-295 两次 run 共享的预算,只有 delay 不同
const base = {
  k: 3, symbols: 6, count: 256,
  hiddenDim: 64, steps: 600, batchSize: 32, lr: 5e-3,
};
// SHORT_DELAY = 5 (跨度 9 步) vs LONG_DELAY = 35 (跨度 39 步)
```

symbols=6 意味着答案区每个 token 的随机基线是 1/6 ≈ 16.7%——任何高于这条线的准确率才说明模型真的记住了东西。

### 短依赖:它学得动,而且会泛化

跨度 9 步,训练 600 步:

- 训练 loss:2.117 → 0.000(sparkline 一路压到地板)
- 留出测试集答案区 token 准确率:**100.0%**,是随机的 6.0 倍

这里有个方法论细节值得点明:测的是**留出测试集**(held-out,训练时没见过的全新序列),不是训练集。为什么必须这样?256 条训练序列,一个 64 维 RNN 完全有能力把它们死记硬背下来——如果只看训练集准确率,"背下来"和"学会复制机制"长得一模一样。只有在没见过的序列上还能 100%,才能说它学到的是 copy 这个机制本身,而不是记住了具体哪 256 条。代码用独立 seed 生成 train / test 两份不相交数据来保证这一点(`stage01-rnn.ts:193-198`)。

### 失败模式:跨度拉到 39 步,塌回随机

同一个模型,同样 600 步,只把 delay 从 5 改到 35(跨度 39 步):

- 训练 loss:起始 2.157,**全程最低仅 1.750**,在平台上振荡,从未逼近 0
- 留出测试准确率:**30.9%**,只有随机的 1.9 倍

注意 loss 曲线的形状区别——短依赖是 `█▂▁▁▁...` 一压到底,长依赖是 `▇▂▁▁▂▁▂...` 降到一个平台后就上下抖,最后甚至回弹。这不是"还没训够",代码里特意验证过:多训 50% 步数也救不回来。模型撞到了它的记忆地平线(memory horizon),靠加训练步数翻不过去。

为什么报告的是全程**最低** loss 而不是最后一步?因为长依赖 loss 在平台上振荡,单看末步可能正好落在一个尖峰上,显得比实际更差。取全程最小值是对"它到底学到了多远"最公平的度量(代码注释 `stage01-rnn.ts:344-346` 解释了这个选择)。诚实地报告失败,比把失败夸大同样重要。

### 关于这道 cliff 的两点诚实校准

第一,cliff 的位置比直觉靠后。用 Adam 优化器时(它对每个坐标做梯度归一化,部分救回了过小的梯度),delay=25 这个 RNN 居然还能学到约 99%。真正的塌陷出现在 delay≈35。所以"RNN 记不住长依赖"不是说它记不住几步,而是有一道相当具体、可复现的悬崖——选 delay=35 是实测校准出来的,不是拍脑袋。

第二,copyTask 是干净的玩具任务,小词表、零噪声、无分布漂移,绝对准确率偏乐观。**能迁移到真实场景的不是这些数字本身,而是相对趋势**:同一架构,短跨度学得动、长跨度塌陷。真实语料里依赖跨度是混合的、被噪声掩盖的,这道 cliff 不会这么干脆,但"超过有效记忆窗口就学不动"的机制是一样的。把玩具任务的绝对数当真,是读这类实验最常见的错误。

## 病在哪里:tanh 反复压缩 + 固定容量

为什么跨度一长就塌?本章先给出机制直觉,严格推导留给第 02 章。

第一个病根是 **tanh 的反复压缩**。tanh 把任何输入压进 (-1, 1),它的导数是 `1 - tanh²`(代码 `tensor.ts:350`:`1 - o * o`),这个值最大才 1,只在输入接近 0 时取到,输入一大就趋近 0。反传时梯度要穿过 T 个 tanh,每穿一个就乘一个 ≤1 的数。穿过 9 个还行,穿过 39 个,这些 ≤1 的因子连乘起来迅速趋近 0——开头那 k 个 token 对最终 loss 的梯度信号,在传回去的路上被压没了。模型"看得到"开头有东西,但梯度告诉不了它"开头那东西很重要",于是学不动。

第二个病根是**固定容量**。隐状态就 64 维,无论序列多长,所有要记的东西都得塞进这 64 个数。短序列里要记的少,塞得下;长序列里要记的多,新信息不断覆盖旧信息,等分隔符到来时,开头的 payload 早被冲掉了。这就是"隐状态容量固定"这句话的实际后果——它不是抽象的理论上界,是 30.9% 这个具体数字。

这两个病根第 02 章会用梯度范数(grad norm)的实测曲线讲透——代码已经在每个 checkpoint 记录了 pre-clip 的全局梯度范数(`stage01-rnn.ts:239` 的 `clipGradNorm` 返回裁剪前的范数),第 02 章会让你看着这个数随跨度变化。第 03 章的 LSTM 则会用门控机制开一条"梯度高速公路"(让某些信息近似 `×1` 地传过去,绕开 tanh 连乘),把这道 cliff 往后推。

## 这一章给后面铺了什么

落到第 00 章的三轴框架上,本章只交代了记忆机制最基础的一档:**固定大小隐状态 + tanh 非线性**,信息混合靠逐步递推,训练并行度为零(纯串行 BPTT)。后面每一章都是在这三轴上的某一轴动手:

- **第 02 章**会拿本章这个一塌陷的 RNN 当病人,用梯度范数实测解释 tanh 连乘为什么让长跨度梯度消失——你已经看到的 30.9% 在那里会有数学解释。
- **第 03 章**的 LSTM / GRU 用门控修记忆机制这一轴,把 cliff 往后推,但训练仍然串行。
- **第 07 章**的 Mamba 用 parallel scan 修训练并行度这一轴,把本章你亲手写的串行反传链改写成可并行的扫描——回到本章的 ⚡ 那个开放问题:线性递归换来了并行,代价是什么,至今没有通用答案。

读完本章你手上应该有一个能跑的 `RNNCell` 和一组诚实的数字:短依赖 100.0%、长依赖 30.9%、训练时间随长度线性增长。带着"它为什么在 39 步塌掉"这个问题进第 02 章。
