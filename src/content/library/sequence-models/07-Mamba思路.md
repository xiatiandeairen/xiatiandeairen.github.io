我先给你一个第 06 章那个 SSM(state-space model,状态空间模型——一句话:用一个隐状态把过去压缩成定长记忆)解不了的任务,你立刻就懂 Mamba 为什么必须存在。

一条序列 24 步,词表只有 6 个 token。其中**恰好一步**带一个 flag=1,那一步的 token 是答案;其余 23 步是干扰项,flag=0,token 从**同一个词表**里随机抽。注意这个设计的狠处:key 和干扰项长得一模一样,模型没法靠"这个 token 长得特别"找到它——唯一的区分信号是 flag,也就是**内容**。随机猜的准确率是 1/6 = 16.7%。

第 06 章那个时不变(time-invariant,意思是每一步用的参数都一样)的 SSM,在这个任务上的测试准确率是 **15.6%**——比随机还低一丢丢,等于完全没学会。这不是调参问题,是结构性的死。这一章就讲它为什么必死,以及 Mamba 改了哪一行就让它活了。

配套代码在 `examples/sequence-models-from-scratch/src/stage07-mamba.ts`,跑 `npx tsx src/stage07-mamba.ts`。下面所有准确率、Δ 数值、wall-clock 耗时都是这个文件真跑出来的,不是我编的;凡是 toy 规模导致偏乐观的地方我都会标出来。

## 时不变 SSM 为什么解不了选择性回忆

回顾第 06 章的离散 SSM 递推,每一步对每个通道做的是同一件事:

```
h_t = Ā · h_{t-1} + B̄ · x_t
y_t = C · h_t
```

`Ā` 是衰减(把旧记忆按比例往下压),`B̄` 是写入门(把当前输入写进状态),`C` 是读出。关键是:**A、B、C 在整条序列上是同一组常数**。这是 SSM 效率的来源——参数不随时间变,整个递推就是一个固定核的卷积,可以用 FFT 一把并行算完(第 06 章那个 `attn/scan` 的标度优势就是这么来的)。

但"固定核"恰好是它在这个任务上的死因。我们的 selective-recall 任务里,key 步和干扰步的 token 分布**完全相同**,模型唯一能用的信息是 flag 这一维。可时不变 SSM 的 `B̄`(写入门)是个常数,它对 key 步和干扰步**用同样的写入强度**。结果是:模型先把 key 写进状态,然后后面 23 步干扰项用同样的力度一遍遍往里写,key 的信息被冲刷殆尽。读出时拿到的基本是最后几个干扰项的混合——所以准确率 ≈ 随机。

代码里我把这个"固定核"版本直接做成了对照组 `fixed`,它和 selective 版**结构完全一样**,唯一差别是把生成 Δ/B/C 的那个 Linear(线性层)的权重冻成零、移出可训练集,让门控值只剩下一个常数 bias:

```ts
// stage07-mamba.ts — SelectiveSSM 构造函数,ablate 分支
if (this.ablate) {
  // Freeze the selection projection's input dependence: zero its weight matrix so the
  // gate values come only from the (trainable) bias => same gate for every token.
  this.selProj.W.data.fill(0);
}
```

跑出来:

```
selective SSM | ████████████████████████████████████████ 1
fixed SSM     | ██████ 0.156
LSTM          | █████████████████████████████████████ 0.914
chance        | ███████ 0.167
  selective: train=100.0% test=100.0%  (params=630)
  fixed    : train=19.5%  test=15.6%   (params=294)
  lstm     : train=92.6%  test=91.4%   (params=1638)
```

`fixed` 的 train 准确率只有 19.5%——它在训练集上都学不会,不是过拟合,是这套结构根本表达不出"按内容选择写入"这件事。这就是第 06 章 SSM 的硬伤被钉死的样子。

## 选择性:让 Δ/B/C 成为输入的函数

Mamba 的核心改动,说穿了就一句:**把 Δ、B、C 从常数变成当前输入算出来的值**。这一步叫 selective(选择性)。代码里 `selProj` 这个 Linear 把每一步的输入特征映射成 `[Δraw, B, C]` 三段:

```ts
// stage07-mamba.ts — SelectiveSSM.forward 的单步
const sel = this.selProj.forward(xt);   // [1, 3D] 由当前输入算出的 Δ,B,C
const dRaw = sel.slice(1, 0, D);
const B = sel.slice(1, D, 2 * D);
const C = sel.slice(1, 2 * D, 3 * D);
const delta = softplus(dRaw);           // [1, D] 正的步长 = 选择性门
// 状态更新: h = exp(-Δ⊙decay) ⊙ h + (Δ⊙B) ⊙ xv
const decayT = delta.mul(decay).scale(-1).exp();  // exp(-Δ*decay) 逐元素
const gain = delta.mul(B).mul(xv);                // 输入相关的写入量
h = decayT.mul(h).add(gain);
lastY = C.mul(h);
```

读懂这三行,你就懂了整个 Mamba 的机制。关键在 `Δ_t`(step size,离散化步长)——它现在是当前 token 算出来的,而且控制着两件事:

- **衰减** `exp(-Δ·decay)`:Δ→0 时这一项→1,旧状态原样保留;Δ 大时衰减强,旧记忆被压掉。
- **写入** `Δ·B·x`:Δ→0 时写入量→0,这一步等于**什么都不写**;Δ 大时把当前 token 大力写进去。

所以 **Δ 是一个输入相关的"写入/遗忘"门**:看到干扰项就把 Δ 压到接近 0(既不写也不忘,状态原地不动,跳过这一步),看到 key 就把 Δ 开大(把 key 写进去)。

> 如果你读过第 03 章,这味道应该很熟。LSTM 的输入门 `i_t = σ(W·x_t + ...)` 就是一个输入相关的写入门;forget 门是输入相关的遗忘门。Mamba 没发明新东西——它是把 LSTM 那套"门控由输入决定"的思想,搬回到 SSM 这个**线性递推**的骨架上。区别在于 LSTM 的门是非线性纠缠在一起的(所以没法并行,见第 03 章那个逐步 BPTT 的代价),而 Mamba 的递推保持线性,门只调节系数——这给后面的 parallel scan 留了活路。这就是为什么我说选择性是"LSTM 输入门思想在 SSM 上的回归"。

### 数值证据:Δ 真的在 key 步开了门

光说不算,代码会挑一条测试样本,把训练好的 selective 模型每一步的 Δ(对通道取均值)画出来,和 flag 对齐看:

```
--- 选择性门控 Δ_t 随时间 (selective SSM, 1 条测试样本) ---
  key flag 在第 12 步 (token id=2)
  row0 = flag, row1 = Δ_t (越亮门越开):
            @
++++++++++++@+++++++++++
  Δ 在 key 步 = 0.9961, 其余步均值 = 0.5538, 比值 = 1.80x  (>1 => 模型在 key 处开门)
  Δ 最大值出现在第 12 步 (期望=12) ✓ 对齐, Δmax=0.9961
```

Δ 在 key 步(第 12 步)冲到 0.9961,其余步均值 0.5538,比值 1.80x,而且**整条序列的 Δ 最大值恰好落在 key 步**。模型自己学会了"在带 flag 的那一步把门开到最大"。没人告诉它 flag 在第 12 步——它从 flag 这一维内容里读出了"该记住了"。这就是选择性。

注意一个诚实的细节:干扰步的 Δ 均值是 0.55,不是 0。理想情况下干扰步 Δ 应该接近 0(完全跳过),这里没压到那么低是 toy 规模 + 短训练的产物。但**比值 1.80x 和 argmax 对齐**这两个信号足够说明机制在工作了——真实 Mamba 在长序列上这个对比会拉得更开。

### 失败模式归因:选择性是因,不是相关

我必须证明 selective 赢不是因为它"碰巧参数多"或"碰巧初始化好",而是因为选择性本身。代码做了消融(ablation,关掉某个机制看效果)实验:fixed 版和 selective 版**同结构、同初始化种子**,唯一差别就是 Δ/B/C 是否由输入生成。

```
--- 失败模式: 关掉选择性 => 准确率掉回随机 ---
  selective 相对随机提升 = +83.3pp; fixed (选择性关闭) 提升 = -1.0pp (≈ 随机水平)。
```

selective 比随机高 83.3 个百分点,fixed 关掉选择性后是 -1.0pp,基本就是随机线。这里有个反直觉点要讲清楚:fixed 的参数更少(294 vs 630),你可能怀疑是参数少导致它输。但代码注释里点破了——**参数不是失败主因**:

```
注: 关闭选择性后 selProj 的输入权重被冻结并移出可训练集, 故参数更少 (294 vs 630) ——
但即便参数更少不是失败主因: selective 在更少 epoch 内就解题, fixed 给再多参数也学不到
"按内容选择" 这一步。
```

道理:fixed 缺的不是容量,是**表达能力**。它的写入门是常数,无论给多少参数、训多少轮,它都无法表达"这一步该写、那一步该跳"这个依赖于输入内容的判断。这是结构上的天花板,不是规模问题。这也呼应了本书库里反复出现的一条判断:能力缺口分两种,一种加规模能补,一种是结构性的、加规模补不了——选择性属于后者。

顺带看 LSTM 基线:它拿到了 91.4%,确实靠门控解得不错,但用了 1638 个参数(selective 的 2.6 倍),而且 train 只到 92.6%。LSTM 能解这题,但它的代价是没法并行(第 03 章讲过),序列一长就被 O(n) 的串行依赖拖死。Mamba 想要的是:**LSTM 的选择性 + SSM 的可并行**。

## ✦ 代价:卷积并行没了,改用 parallel scan

这是这一章最该让你记住的工程判断,也是 Mamba 论文一半篇幅在干的事。

第 06 章时不变 SSM 之所以快,是因为参数固定 → 递推等价于一个固定核的卷积 → FFT 并行。现在 Δ/B/C **随每一步输入变化**,核也跟着每步变,**卷积形式直接失效了**。你没法再把它写成一个卷积一把算完。

最朴素的退路是回到逐步递推——本章的 toy 实现就是这么做的,一个 `for (let t = 0; t < T; t++)` 顺序扫描:

```ts
// stage07-mamba.ts — forward,逐步顺序扫描(toy 版,非并行)
for (let t = 0; t < T; t++) {
  const xt = feats.slice(0, t, t + 1);
  // ... 算 Δ/B/C ...
  h = decayT.mul(h).add(gain);   // 严格依赖 h_{t-1},顺序执行
}
```

这能跑,而且仍然是 O(n)——每步的工作量和序列长 n 无关,一共 n 步。代码实测了 wall-clock 验证这个标度:

```
  实测前向耗时 vs n (selective SSM, 真实 wall-clock):
    n= 16: 0.1909 ms/fwd  (基准)
    n= 32: 0.3565 ms/fwd  (n 翻 2x => 时间 1.87x; O(n) 预期 ≈ 2x)
    n= 64: 0.6921 ms/fwd  (n 翻 2x => 时间 1.94x; O(n) 预期 ≈ 2x)
    n=128: 1.4028 ms/fwd  (n 翻 2x => 时间 2.03x; O(n) 预期 ≈ 2x)
```

n 每翻倍,时间就涨约 2x,干净的线性。对照 attention 的 O(n²)(代码里的 MAC 估算:n=128 时 attn/scan = 256x),长序列上 SSM 的标度优势是实打实的。

但 toy 这个**顺序** for 循环在 GPU 上是灾难——GPU 靠成千上万核并行吃饭,你给它一个严格串行、每步等上一步的循环,几千个核全在干等。**Mamba 真正的工程贡献在这里:它用 parallel scan(并行扫描,又叫前缀和扫描)在 GPU 上重新拿回了并行性。**

直觉是这样:`h_t = a_t · h_{t-1} + b_t` 这种递推,虽然看着每步依赖前一步,但它满足结合律——和前缀和(prefix sum)同构。前缀和有经典的树形并行算法(Blelloch scan):n 个元素用 log(n) 层、总共 O(n) 工作量并行算完。Mamba 把选择性递推套进这个 scan 框架,于是即便核随输入变,也能在 GPU 上并行,只是并行方式从"卷积"换成了"扫描"。

这就是 ✦ 那个专家钩子:**Mamba 放弃卷积换 parallel scan,导致算法和系统在这里深度耦合**。Mamba 论文里大量篇幅在讲一个 hardware-aware(硬件感知)的 selective scan CUDA kernel——它要把状态留在 GPU 的 SRAM(片上高速缓存)里做扫描,避免反复读写 HBM(显存),否则光是 IO 就把 O(n) 的理论优势吃光。换句话说,Mamba 不是一个纯算法,它是算法 + kernel 工程的合体。这正是它**工程门槛高**的原因:你照着公式实现一个慢 100 倍的版本很容易(本章就是),但要拿到论文里那个速度,你得会写融合了扫描逻辑的 CUDA kernel。

本章**明确没实现**这个并行 kernel——toy 版是顺序扫描。我能诚实交付的是机制(选择性解题、固定版崩塌)和 O(n) 标度;真实 Mamba 的量级优势需要 GPU + 长序列 + 硬件感知并行扫描,这超出 toy 的范围。这个边界我必须讲清楚,否则就成了我在本书库反对的那种"survey 稿"——拿玩具数字冒充生产结论。

## ⚡ 前沿:SSM 取代 Transformer,还是互补?

到这里你手里有了判断混合架构所需的全部机制理解,正好用来看一个**目前还没有通用答案、业界正在押注**的开放问题。

2023 年 Mamba 出来时,叙事是"线性复杂度的 Transformer 杀手"。但 2024-2025 年的实际走向把这个叙事修正了。两个标志性动向:

- **Mamba-2** 重新整理了选择性 SSM 和 attention 的数学关系(structured state-space duality,SSD,把两者放进同一个框架),并优化了 scan 的硬件效率。它没有宣布取代谁,反而让"SSM 和 attention 是一枚硬币两面"这件事更清楚了。
- **混合架构成为主流押注**,代表是 Jamba(把 Mamba 层和 attention 层交替堆叠,中间插 MoE)。业界共识从"SSM 取代 Transformer"转向了"两者互补"。

为什么是互补而不是替代?从机制层面你现在能自己推出来:

- **选择性 SSM 擅长长、稀疏的依赖**。它的状态是**定长**的——把整段历史压进一个固定大小的 `h` 里。这对超长序列省内存(状态不随 n 增长),对"大部分内容是噪声、偶尔有关键信号"的场景(就是本章那个 selective-recall 任务)很合适。但定长状态是把双刃剑:历史被有损压缩了。

- **attention 擅长精确检索**。它把每个 token 都原样存进 KV cache,要用时按相关度精确召回任意一个历史 token——这是无损的随机访问。代价就是第 05 章讲的 O(n²) 和随 n 线性增长的 KV cache。需要"逐字回忆很久以前某个具体 token"(比如长文档里精确引用、in-context 学习里复现示例)时,定长状态的 SSM 会丢细节,attention 不会。

所以混合是务实答案:**用 SSM 层廉价地处理长程的、压缩式的上下文,用少数几层 attention 做精确检索**。这不是妥协,是各取所长。

但我必须标注这是 ⚡ **开放问题,目前无通用解**:SSM 层和 attention 层该按什么比例混、attention 该放在哪几层、长序列上定长状态的有损压缩到底在什么任务上会致命——这些**还没有跨任务跨规模都成立的定论**,各家在不同 benchmark 上结论不一,本质上还在拿真实训练成本做实验探索。谁敢现在给你一个"最优混合比例",要么是过拟合到某个 benchmark,要么是在卖货。这一题正在被研究,不是已经解决。

## 收尾:三轴闭环

这是全书终点,回头看这条线索就清楚了。我们一路在追三个东西,而它们一度互相打架:

| 轴 | 谁解决的 | 代价 |
|---|---|---|
| **O(n) 复杂度** | 第 06 章 SSM(对照 attention 的 O(n²)) | 时不变 → 无选择性 |
| **可控的长依赖梯度** | 第 03 章 LSTM 门控 + SSM 的稳定线性递推 | LSTM 非线性门 → 无法并行 |
| **内容选择性记忆** | 本章 Mamba 的输入相关 Δ/B/C | 卷积失效 → 必须 parallel scan + CUDA kernel |

第 01-02 章的 RNN 给了 O(n) 但梯度炸/消失;第 03 章 LSTM 用门控稳住了梯度但锁死了并行;第 05 章 attention 拿到了并行和精确检索但复杂度 O(n²);第 06 章 SSM 用线性递推重新拿回 O(n) 和可并行,但丢了选择性。Mamba 在 SSM 的线性骨架上把 LSTM 的选择性门控装了回去,代价转嫁到了系统层——用 parallel scan 和硬件感知 kernel 把因选择性而失效的并行重新挣回来。

三轴第一次在一个架构里同时成立,代价是算法和系统深度耦合。这既是 Mamba 的成就,也是它工程门槛高、且最终走向"与 attention 混合而非取代"的根本原因。
