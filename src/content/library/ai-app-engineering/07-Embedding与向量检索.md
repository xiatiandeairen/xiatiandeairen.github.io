锚点类比贯穿全章：**向量检索之于 RAG = 索引之于数据库**。你过去十年知道"全表扫描慢、加索引快、但索引要占空间、要维护、选错索引字段等于没加"——这套直觉一条都没浪费。ANN 索引就是"为高维向量的相似度查询造的索引"，所有 trade-off 的形状都一样：**精度 vs 速度 vs 内存 vs 构建成本**，四角拉扯，没有银弹。

## 一、语义向量：把"意思"压成一串数字

### 1. 背景：为什么需要 embedding

回到第 1 章的物理事实：LLM 是无状态的、有上下文窗口上限的。你有 10 万篇文档、几个 G 的内部 wiki，不可能全塞进 prompt（撑爆窗口、贵到飞起、还稀释注意力）。RAG 的核心动作是：**用户来一个问题，先从海量文档里"检索"出最相关的几段，只把这几段塞进 prompt**（见第 8 章完整流水线）。

问题来了：**怎么"检索最相关的"？**

最朴素的办法是**关键词匹配**（全文检索、倒排索引、BM25）——用户搜"如何重置密码"，去文档里找包含"重置""密码"这些词的段落。这套在搜索引擎时代统治了几十年，**至今依然极其有用**（本章后面 hybrid search 会专门讲），但它有个硬伤：**它匹配的是字面，不是意思**。

- 用户问"我登录不上了"，文档里写的是"账号无法访问的排查步骤"——**一个关键词都不重合**，BM25 召回为零。
- 用户问"怎么把钱退回来"，文档标题是"退款流程"——"钱"和"款"、"退回来"和"退款"，字面不同、意思相同。

人能看懂这是一回事，因为人理解**语义**。要让机器也能按"意思"检索，就需要把文本变成一种**能比较语义相似度**的表示。这个表示就是 **embedding（向量嵌入）**：一个高维实数向量，比如一段文本 → `[0.013, -0.21, 0.88, ..., 0.04]`（768 维、1024 维、3072 维不等）。

核心思想一句话：**把"意思"编码成空间里的一个点，意思越近的文本，点的距离越近。**"登录不上"和"账号无法访问"会被映射到向量空间里几乎挨着的两个点；"退款流程"和"如何申请退钱"也挨着；而"退款流程"和"今天午饭吃什么"则隔得很远。

> 新手第一误区：以为 embedding 是"把词翻译成数字编号"。不是。编号（如 one-hot：第 1024 个词就是第 1024 位为 1）是**离散、无意义距离**的——"猫"是 5、"狗"是 6、"汽车"是 7，但 5 和 6 的距离不代表猫狗比猫车更近。embedding 的全部价值在于**距离有语义意义**：它是一个**稠密（dense）、连续、低维**的表示，几何距离 ≈ 语义距离。这是从"符号"到"几何"的范式跃迁。

### 2. 演进：从 word2vec 到上下文 embedding

这是一条值得记死的演进线，面试高频考"你了解 embedding 的发展吗"，本质是在考你懂不懂**静态 vs 上下文**这个分水岭。

**第一代：静态词向量（word2vec / GloVe，2013 起）**

word2vec 的洞察极其优雅：**一个词的意思，由它周围经常出现的词决定**（"You shall know a word by the company it keeps"）。训练时让模型预测"中心词周围会出现哪些词"（或反过来），训练完，每个词就得到一个固定向量。

它带来了那个著名的"向量代数"现象：`vec(国王) - vec(男人) + vec(女人) ≈ vec(女王)`。语义关系变成了向量的加减——这第一次让人直观感受到"意思真的被编码进几何了"。

**致命局限：一词一向量，无视上下文。** "苹果"在 word2vec 里只有一个向量，但"我买了个苹果手机"和"我吃了个苹果"里的苹果意思完全不同。静态词向量做不到——它给的是这个词所有用法的"平均值"，遇到一词多义（polysemy）就糊了。

**第二代：上下文 embedding（ELMo → BERT，2018 起）**

Transformer（见第 1 章）来了。BERT 这类模型的关键变化：**同一个词，在不同句子里产生不同的向量**。"苹果手机"里的苹果向量偏向"科技公司/电子产品"，"吃苹果"里的偏向"水果"。向量是**根据整个上下文动态算出来的**，一词多义自然化解。

但 BERT 原生输出的是**每个 token 一个向量**，要得到"一整句话/一整段的向量"还需要池化（pooling，常见做法是取 `[CLS]` token、或对所有 token 向量做 mean pooling）。而且原生 BERT 的句向量**质量并不好**——直接拿来算句子相似度效果很差。

**第三代：专门的句子/文本 embedding 模型（Sentence-BERT → 现代 embedding model，2019 起至今）**

这是你今天实际会用的东西。核心是用**对比学习（contrastive learning）** 专门训练模型去优化"句子级"的语义表示：给一批"语义相近的句子对"（正样本）和"不相近的"（负样本），训练目标就是**让正样本对的向量靠近、负样本对的远离**。

> **坑点提示（这条决定你模型选对没选对）**：训练目标决定了 embedding 的"好坏方向"。一个为"语义相似度"训练的模型（如 bge、sentence-transformers 系），和一个为"问答检索"（query 和 passage 不对称）训练的模型，在不同任务上表现差很多。检索（retrieval）是**非对称**任务——查询短、文档长，二者风格不同。好的检索 embedding 模型会用**双塔/非对称结构**专门优化"短 query 找长 doc"。这就是为什么很多模型要求你给 query 和 document **加不同的前缀**（见本节工程推论 3）。

### 3. embedding 怎么用：调一个 API / 跑一个模型

工程上，你要么调云端 embedding API，要么本地起一个 embedding 模型，输入文本、输出向量。形态和调 LLM 很像，但有几个关键区别：

```python
# 概念示意（具体 SDK 因 provider 而异，按 2026 初，名单会变）
vec = embed(model="某个embedding模型", input="如何重置密码")
# vec 是一个定长 float 数组，比如 1024 维
```

**关键工程事实（物理性的，记死）：**

1. **同一个模型，输出维度固定。** 选了 1024 维的模型，每段文本都是 1024 维。维度是模型的属性，不是你随便设的（虽然有些新模型支持"可裁剪维度"——见本节第 4 部分"embedding 模型选型"末尾的 Matryoshka）。
2. **query 和 document 必须用同一个模型 embed。** 你不能用模型 A 编码文档、模型 B 编码查询然后比距离——两个模型的向量空间根本不是一个坐标系，距离毫无意义。**升级 embedding 模型 = 全量文档重新 embed 重建索引**，这是个有成本的运维动作，选型时就要想到。
3. **embedding 是确定性的（基本上）。** 同一个模型、同一段文本，输出向量基本一致（不像 LLM 生成有采样随机性）。这让它可缓存——相同文本不必重复 embed。

> **关于 Anthropic 与 embedding 的诚实说明（按 2026 初，以官方为准）**：Claude 系列是**生成模型**，Anthropic 官方**不提供自家的文本 embedding 端点**——`/v1/messages` 是生成接口，不产出向量。做 RAG 时，embedding 这一步通常交给**专门的 embedding 提供方**（OpenAI text-embedding 系、Cohere、Voyage、Jina，或开源的 bge / Qwen-embedding 自托管），检索到的文本再喂给 Claude 去生成答案。**这是一个健康的架构解耦：embedding 层和生成层可以独立选型、独立替换。** 别想当然以为"用 Claude 就用 Claude 的 embedding"——那个东西不存在，硬找会浪费时间（这正是 know.md 里"平台不会做 X 别想当然"的反面：这里是平台确实没做这一层，你得外接）。

### 4. embedding 模型选型：维度 / 多语言 / 中文表现 / 成本怎么权衡

> **版图警告（按 2026 初写，名单按季度过期，重点学选型维度而非记产品）**：embedding 模型迭代极快，下面的产品定位一年内就会变。**真正不过时的是"用哪几个维度去评估一个 embedding 模型"** ——和上一章你评估一个数据库驱动一样，先有评估框架，产品换了你照样会选。

embedding 是 RAG 漏斗最上游的闸门（推论一会讲），所以这是**最不该拍脑袋、也最不该图新鲜的一环**。先建立评估框架：

**评估一个 embedding 模型的六个维度：**

1. **维度（dimension）**：768 / 1024 / 1536 / 3072 不等。**维度不是越高越好**——高维召回略好但**存储、内存、检索算力全线上涨**（回想第六节的成本账：每维 4 字节，亿级向量下 1024 维和 3072 维差三倍存储）。维度是你为"召回质量 vs 规模成本"做的第一个权衡。
2. **多语言 / 中文表现**：模型是"英文为主"还是"多语言原生训练"？中文、中英混合检索必须选**中文语料充分**的模型，否则中文召回会明显塌。这一条**没有放之四海的答案，必须在你自己的数据上实测**（见下方坑点）。
3. **最大输入长度（max sequence length）**：512 token 还是 8K token？这直接约束你第 8 章的 chunk 能切多大——模型只能吃 512 token，你切 2000 token 的块就会被**静默截断**，超出部分根本没进向量。
4. **对称 vs 非对称（检索专用）**：是为"句子相似度"训练的，还是为"短 query 找长 doc"的检索任务训练的？检索场景要选后者，且注意它的 query/document 前缀要求（推论三）。
5. **托管 vs 自部署**：调云端 API（按量付费、零运维、但数据出境 + 有调用成本）还是开源自托管（要 GPU、要运维，但数据不出门、可离线、长期成本可控）？**企业内部知识库常因数据合规必须自托管**——这一条经常是硬约束，不是偏好。
6. **License 与商用**：开源权重的 license 允许商用吗？闭源 API 的数据条款能接受吗？企业落地这条要先过法务。

**主流 embedding 模型横向定位（按 2026 初，以官方与你自己的实测为准）：**

| 模型 / 系列 | 出身 | 形态 | 语言定位 | 选型场景 |
|---|---|---|---|---|
| **bge 系列**（如 bge-m3） | 智源 BAAI，开源 | 自托管 | **中英俱强**，m3 多语言 + 长文本 + 稠密/稀疏混合 | **中文/中英混合 RAG 的开源首选**；要自托管、要数据不出门 |
| **Qwen-embedding** | 阿里，开源 | 自托管 | 多语言、中文强 | 已用 Qwen 生态、要中文 + 自托管的另一主力 |
| **OpenAI text-embedding**（3-small / 3-large） | OpenAI，闭源 API | 托管 | 多语言、英文最稳，支持 **Matryoshka 可裁剪维度** | 英文/多语言为主、要省运维、能接受云 API 和数据出境 |
| **Cohere embed** | Cohere，闭源 API | 托管 | 多语言强，有检索专用版 | 多语言检索、要托管、看重检索专项优化 |
| **jina embeddings** | Jina AI，开源 + API | 两者皆可 | 多语言、**长文本上下文**见长 | 长文档检索、要灵活在托管/自托管间选 |

> **坑点提示（这条直接对应 P2 项目验收）**：**embedding 模型的中文 benchmark 数字，别背榜单、更别信营销页**。一是榜单（如 MTEB 中文榜）和你的**真实业务语料分布往往不一致**——榜单在通用语料上第一的模型，到你的法律/医疗/客服垂域可能输给第二名；二是名单按季度刷新，今天的第一下季度就被超。**唯一靠谱的做法**：拿你自己的 ≥20 条标注 query（你知道正确答案在哪些文档里），在候选模型上实测 **recall@10**，用你的数字做决定（这正是 know.md 里"≥3 样本验证可行性、不背名单学方法论"的落地）。一句话：**别人的 benchmark 是参考，你的标注集才是裁判。**

**Matryoshka / MRL（可裁剪维度）—— 为什么有些模型能"一个向量当多个维度用"：**

普通 embedding 模型的维度是死的：1024 维就是 1024 维，想要 256 维得换模型重训。**Matryoshka Representation Learning（MRL，套娃表示学习）** 改了训练目标，让向量的**前 k 维就是一个可用的低维表示**——取前 256 维当 256 维向量用、取前 512 维当 512 维用，**精度随维度平滑下降，但不用重训、不用换模型**（OpenAI text-embedding-3 系、部分 bge/jina 版本支持，按 2026 初以官方为准）。

- **工程价值**：同一份 embedding，**先用裁剪后的低维做粗筛**（省存储省算力，亿级向量下省得很狠），**再用完整高维做精排**——又一个"精度换资源"的旋钮，且这次免费送（一次 embed，多档维度可用）。
- **怎么用**：存索引时存完整维度，检索时按 SLA 选裁到多少维。内存吃紧就裁狠点，召回不够就放宽。它和量化（第六节）是两条正交的压缩路：MRL 砍维度数，量化砍每维比特数，**可以叠加**。

> 心法：**embedding 选型的第一性原理和向量库一样——不是"哪个榜单第一"，是"哪个在你的语料、你的语言、你的合规约束、你的规模下最合适"。维度、多语言、长度、托管方式四个硬维度先过一遍，再用自己的标注集让 recall 数字拍板。**

### 5. 工程推论（本节地基，逐条都是面试题）

**推论一：embedding 是有损压缩，召回上限由它锁死。**

把一整段文本压成 1024 个浮点数，必然丢信息。两段意思有微妙差异的文本，可能被压到几乎同一个点——这叫**语义坍缩**。后果是：**如果对的文档在 embedding 空间里就没被映射到 query 附近，那么后面无论 rerank 多强、prompt 多好，都救不回来——它根本不在召回候选里。** 召回率的天花板，在 embedding 这一步就定死了。这是为什么 embedding 选型是 RAG 里**最不该省钱的一环**：它是漏斗的最上游，上游漏了的，下游全是无米之炊。

**推论二：向量检索是"模糊匹配"，天然会召回"像但不对"的东西。**

关键词匹配是精确的——含"密码"就是含。向量匹配是连续的——它返回的是"最相似的 top-k"，哪怕第 k 个其实并不相关，它也会返回（因为它只会排序，不会说"以下都不够格"）。所以**向量检索几乎总会返回 k 个结果，包括噪声**。这意味着：(a) 你需要**相似度阈值**来过滤明显不相关的（"分数低于 0.7 的就当没召回到"）；(b) 你需要第 9 章的 **rerank** 来对 top-k 做精排，把"像但不对"的踢下去。**新手最容易栽**：以为向量库返回的 top-5 就是"对的 5 个"，直接塞进 prompt，结果模型被噪声带偏。

**推论三：query/document 前缀和归一化是"隐形正确性"，错了不报错只是变差。**

很多 embedding 模型（尤其检索专用的）要求：编码查询时加 `query:` 前缀、编码文档时加 `passage:`/`document:` 前缀（具体前缀看模型卡片，按 2026 初，模型不同要求不同）。漏加前缀**不会报错**，但召回质量明显下降——因为模型是带着前缀训练的，你不给它就用错了分布。同理，很多相似度度量要求向量**先归一化**（见第二节）。**这类 bug 最阴险**：代码能跑、有结果、只是"莫名其妙差一点"，排查时根本不会怀疑到这。**落地动作**：选模型先读它的 model card，把"前缀怎么加、要不要归一化、用哪种相似度"抄进你的 ingestion 代码注释里。

**推论四：embedding 模型一换，整个索引作废，所以选型要"想三年"。**

升级 embedding 模型不是改一行配置——它要求**把全量文档重新 embed、重建整个向量索引**。10 万文档还好，10 亿文档就是一笔可观的算力和时间账。所以 embedding 选型有"路径依赖"：**今天选的模型，你大概率要用很久**。这跟 know.md 里"未验证的远期细节别写死"是一个道理的反面——embedding 是地基，地基要按长期选，不能图一时新鲜频繁换。

> 心法：**embedding 是 RAG 漏斗的最上游闸门。上游能放进来的水，决定了下游最多能用多少。所有省钱、偷懒、想当然，在这一层的代价都是"召回率永久封顶"。**

## 二、相似度度量：cosine、dot、欧氏，到底用哪个

向量有了，怎么算"两段文本有多像"？这是一道纯物理题，**面试高频且很多人答不利索**。

### 1. 三种度量及其几何含义

| 度量 | 公式（直觉） | 几何含义 | 范围 | 用在哪 |
|---|---|---|---|---|
| **Cosine similarity（余弦相似度）** | 两向量夹角的余弦 | 只看**方向**，不看长度 | [-1, 1]，越大越像 | 文本语义检索的**默认选择** |
| **Dot product（点积/内积）** | 对应位相乘求和 | 同时受**方向和长度**影响 | (-∞, +∞) | 向量已归一化时，等价于 cosine；某些模型专门为内积训练 |
| **Euclidean / L2（欧氏距离）** | 两点直线距离 | 空间里的**绝对距离** | [0, +∞)，越小越像 | 图像、某些聚类场景；文本检索较少直接用 |

**为什么文本检索默认用 cosine？** 因为我们关心的是"意思的方向"，不关心"向量的长度"。一段长文档和一段短摘要，意思相同但向量模长可能差很多——cosine 把长度因素除掉，只比方向，正好契合"语义相似"的需求。

**cosine 和 dot 的关键关系（这是考点）：** 如果向量都**归一化**成单位长度（模长=1），那么 **dot product 在数值上等于 cosine similarity**。这就是为什么很多向量库/模型推荐你"先归一化向量，然后用内积检索"——既拿到了 cosine 的语义正确性，又拿到了内积的**计算更快**（少一步除以模长的运算，在亿级向量上这点差异会放大）。

### 2. 归一化：那个最容易被忽略的步骤

**归一化（normalization / L2 normalize）**：把向量除以它自己的模长，变成单位向量。归一化之后，所有向量都落在单位超球面上，此时"夹角"和"距离"是单调对应的——cosine 大 ⟺ L2 距离小，三种度量趋于一致。

> **坑点提示（生产真实事故来源）**：度量选错或忘了归一化，是向量检索"结果莫名其妙不对"的头号原因，而且**不报错**。两个典型翻车：
> - 模型输出的是**未归一化**向量，你却在向量库里配了"内积"度量 → 内积被向量长度污染，长文档因为模长大被无脑排前面，召回偏向"长的"而非"相关的"。
> - 模型卡片明确说"用 cosine"，你建库时默认配了"欧氏" → 能跑、有结果、就是质量差，排查半天找不到原因。
>
> **铁律**：建库前确认三件事并写进代码——(1) 模型输出**是否已归一化**（很多模型已经帮你归一化了，再归一化无害；有些没有）；(2) 向量库索引配的度量（cosine / dot / L2）和模型推荐的**一致**；(3) query 和 document 走的是**同一套**度量和归一化。

### 3. 工程推论

1. **文本检索默认 cosine（或"归一化 + 内积"），别用欧氏除非模型卡片明确要求。** 被问"为什么用 cosine 不用欧氏"，答"文本语义关心方向不关心模长，cosine 把长度因素剔除；归一化后内积≡cosine 但更快"——这就把考点全踩到了。
2. **归一化是正确性的一部分，不是优化项。** 它决定了你算的到底是不是 cosine。漏了不报错，只是悄悄变差。
3. **度量必须模型、库、query/doc 三方一致。** 任何一方不一致，距离就失去语义，召回质量崩塌且难排查。

> 心法：**相似度度量不是"调参选项"，是"你到底在比什么"的定义。选错了，你比的根本不是语义，只是数字凑巧。**

## 三、ANN：为什么不能精确检索，以及 HNSW/IVF 怎么工作

这是本章**最硬核的物理学**，也是面试区分"用过向量库"和"懂向量库"的核心分水岭。讲透。

### 1. 背景：精确检索为什么不可行

最朴素的检索叫 **暴力检索 / 精确 KNN（brute-force / exact k-Nearest-Neighbors）**：query 来了，跟库里**每一个**向量都算一遍相似度，排序取 top-k。

- **优点**：100% 准确，召回的就是真正最近的 k 个。
- **致命缺点**：复杂度 O(N×d)。N=10 亿、d=1024，每次查询要做 10 亿次 1024 维的相似度计算。**单次查询几秒甚至几十秒**，QPS 惨不忍睹。

这就是数据库的老问题换了层皮：**全表扫描精确但慢**。解法也一样——**建索引，用一点点精度换巨大的速度**。

在向量世界，这个索引叫 **ANN（Approximate Nearest Neighbor，近似最近邻）**。关键词是 **Approximate**：它**不保证**返回的就是真正最近的 k 个，但保证**绝大多数情况下八九不离十**，换来的是**几个数量级的速度提升**。

> **核心认知（记死）**：ANN 是一桩**明牌的精度换速度交易**。你接受"召回的不一定是 100% 最近邻"（可能漏掉真正第 3 近的、混进真正第 8 近的），来换"查询从秒级到毫秒级"。衡量这桩交易划不划算的指标叫 **recall@k**——"ANN 返回的 top-k 里，有几个是真正精确 top-k 里的"。recall=0.95 意思是 ANN 的结果和精确结果重合 95%。**生产里 0.95~0.99 的 recall 通常完全够用**，因为 RAG 后面还有 rerank 兜底（见第 9 章），上游近似带来的微小损失，下游能补。

### 2. HNSW：图索引，当前最主流

**HNSW（Hierarchical Navigable Small World，分层可导航小世界图）** 是目前生产中最常用的 ANN 索引，几乎所有现代向量库都支持。理解它的直觉**用得上你的分布式直觉**——它本质是个**跳表（skip list）的高维推广**。

**直觉：高速公路网。**

想象你要从一个城市开车到目标城市最近的加油站。HNSW 把向量组织成**多层图**：

- **最顶层**：节点极少，但每个节点连着很远的节点——像**洲际高速**，一跳跨越半个国家。
- **中间层**：节点变多，连接距离变中等——像**省道**。
- **最底层**：包含**所有**节点，连接都是近距离的——像**市区小路**。

查询时**从顶层进入**：在洲际高速上快速朝目标大方向飞奔，到了大致区域**下沉一层**，在省道上更精细地接近，再下沉，最后在市区小路上找到那几个最近的点。**每一层都把搜索范围指数级缩小**——这和跳表"先在稀疏层粗定位，再到密集层精定位"是同一个思想，也和你熟悉的"先粗排索引再精确定位"一脉相承。

**关键参数（生产必须会调，面试高频问）：**

| 参数 | 作用 | 调大的后果 | 调小的后果 |
|---|---|---|---|
| **M** | 每个节点的最大连接数（图的"度"） | 召回↑、内存占用↑、构建慢↑ | 召回↓、内存省、构建快 |
| **efConstruction** | 建索引时每步探索的候选数 | 索引质量↑、**构建时间↑** | 构建快、索引质量略降 |
| **efSearch**（也叫 ef） | 查询时每步探索的候选数 | **召回↑、查询变慢↑** | 查询快、召回↓ |

**怎么记**：`M` 和 `efConstruction` 是**建库时**定的（影响索引结构和内存），改了要重建；`efSearch` 是**查询时**定的（影响单次查询的精度/速度），可以**每次查询动态调**——这是它最实用的地方。

**生产调参心法（可执行判断）：**
- **M**：常见 16~64。中英文文本检索从 **32** 起步通常稳妥。内存吃紧就降到 16，召回不够就升到 48/64。
- **efConstruction**：常见 100~500。**构建是一次性成本**，舍得花时间就调高（如 200~400）换更好的索引质量，反正只建一次。
- **efSearch**：**这是你的运行时旋钮**。从 **64~128** 起步，召回不够就往上加（128→256），延迟敏感就往下压。它直接体现 recall vs latency 的 trade-off，**应该按线上 SLA 实测调**，不是拍脑袋。

> **坑点提示**：HNSW 是**内存索引**——整个图常驻内存才快。N 越大、M 越大，内存占用越凶（粗估：N × d × 4 字节的原始向量 + 图结构的额外开销）。**亿级向量纯 HNSW 会吃掉巨量 RAM**，这时要么上量化（见第四节）把向量压小，要么换 IVF 这类磁盘友好的索引。**面试加分点**：能说出"HNSW 召回好、查询快，但内存贵、且不擅长频繁删除（删节点会破坏图结构）"——这就显出你真用过而不是只看过名字。

### 3. IVF：倒排+聚类，省内存的另一条路

**IVF（Inverted File，倒排文件索引）** 是另一大类思路，**用聚类把搜索空间切块**。

**直觉：图书馆分区。**

- **建库**：先把所有向量**聚类**成比如 1024 个簇（用 k-means），每个簇有一个中心点（centroid）。每个向量归到离它最近的簇——像把书按主题分到 1024 个书架。
- **查询**：query 来了，先看它离哪几个**簇中心**最近，**只在最近的 nprobe 个簇里搜**，其它簇直接跳过。像找书时只去最相关的几个书架，不扫整个图书馆。

**关键参数：**

| 参数 | 作用 | trade-off |
|---|---|---|
| **nlist** | 簇的总数（建库时定） | 多→每簇更小搜得快但要搜更多簇才覆盖；少→每簇大 |
| **nprobe** | 查询时搜几个最近的簇（查询时定） | 大→召回↑、慢↑；小→快↑、召回↓（可能漏掉"恰好在边界簇里的近邻"） |

`nprobe` 是 IVF 的运行时旋钮，地位类似 HNSW 的 `efSearch`。

**IVF 的核心 trade-off 和软肋：** 它的命门是**边界问题**——如果一个真正的最近邻**恰好落在你没搜的那个簇里**（在簇边界附近），就漏了。调大 nprobe 能缓解（搜更多簇），但搜的簇多了就慢，退化回接近暴力。

**HNSW vs IVF 怎么选（生产判断）：**

| 维度 | HNSW | IVF（常配 PQ 量化，见第四节） |
|---|---|---|
| 召回质量 | 通常更高 | 中高（依赖 nprobe） |
| 查询速度 | 快 | 快 |
| 内存占用 | **高**（图常驻内存） | **可控**（配 PQ 后大幅压缩，能上磁盘） |
| 构建速度 | 较慢 | 较快 |
| 增删友好度 | 删除不友好 | 较友好（重训聚类即可） |
| 适用规模 | 百万~千万级，内存够就上 | **亿级以上**、内存受限时的主力 |

**一句话决策**：**中小规模（百万~千万）、内存够、要最好召回 → HNSW；超大规模（亿级+）、内存受限、能接受略低召回 → IVF + PQ。** 很多向量库默认给 HNSW，因为大多数业务数据量到不了非上 IVF 不可的程度。

### 4. 工程推论

1. **ANN 是明牌的精度换速度，用 recall@k 量化这桩交易。** 被问"向量检索是精确的吗"，答"不是，生产用的是 ANN 近似检索，拿 5% 以内的召回损失换几个数量级的速度，损失靠下游 rerank 兜底"。
2. **HNSW 的 efSearch / IVF 的 nprobe 是运行时旋钮，按 SLA 实测调。** 这是 recall vs latency 的直接控制点，面试能点出"哪些参数建库定、哪些查询定"是硬功夫。
3. **索引选型先看内存预算。** 内存够上 HNSW 拿召回；内存不够上 IVF+PQ 拿规模。这跟"选数据库索引先看读写比和存储成本"是一个判断逻辑。
4. **别迷信"全精确"。** 在 RAG 里追求 recall=1.0 是过度优化——下游有 rerank，上游 0.97 和 1.0 的差异常常在最终答案里看不出来，却要付出成倍的延迟/内存代价。

> 心法：**ANN 索引和数据库索引是同一种生物：都是花空间和构建成本，买查询速度，且都得为你的数据规模和访问模式量身选。没有"最好的索引"，只有"最适合你这批数据和这个 SLA 的索引"。**

## 四、向量库选型：pgvector / Milvus / Qdrant / Weaviate / Elasticsearch

> **版图警告（按 2026 初写，产品迭代快，名单会变，重点学选型维度而非记产品）**：下面的产品对比一年内就会有版本变化、功能补齐。**真正不过时的是"用哪几个维度去评估一个向量库"** ——记住维度，产品换了你也会选。

### 1. 评估一个向量库的六个维度

这是方法论核心，先建立评估框架再看产品：

1. **规模上限**：百万、千万、亿、十亿？数据量决定了能不能用轻量方案。
2. **运维复杂度**：是嵌进现有数据库（零新增运维），还是要独立部署、调优、扩容的分布式系统？
3. **混合查询能力**：能不能"向量相似度 + 结构化过滤（如 `where tenant_id=X and date>Y`）"一起做？这是企业 RAG 的刚需（多租户、权限、时间范围）。
4. **索引类型与可调性**：支持 HNSW/IVF/PQ？参数能调到多细？
5. **一致性与事务**：写入后多久可查？要不要强一致？（影响"文档更新后立刻能检索到"的体验）
6. **生态与上手成本**：团队会不会、文档全不全、和现有栈（Postgres / ES / 云厂商）融不融。

### 2. 主流产品横向对比（按 2026 初，以官方为准）

| 产品 | 本质 | 规模甜区 | 混合查询 | 运维 | 适合谁 |
|---|---|---|---|---|---|
| **pgvector** | Postgres 扩展 | 百万~千万 | **强**（就是 SQL，`WHERE` 随便加） | **极低**（已有 PG 就零新增） | **绝大多数从 0 起步的项目**；数据本就在 PG |
| **Qdrant** | 专用向量库（Rust） | 千万~亿 | 强（payload 过滤一等公民） | 中（独立部署但轻量） | 要专用向量库、重视过滤和性能、不想要 Milvus 那么重 |
| **Milvus** | 专用向量库（分布式） | **亿~十亿+** | 中 | **高**（分布式组件多、运维重） | **超大规模**、有专门团队扛运维 |
| **Weaviate** | 向量库 + 内置模块化 | 千万~亿 | 强 | 中 | 想要开箱即用（自带 embedding 模块、GraphQL） |
| **Elasticsearch** | 搜索引擎 + 向量能力 | 千万~亿 | **极强**（BM25 和向量同库，hybrid 天生顺手） | 中高（ES 集群本身有运维成本） | **已经在用 ES 做全文检索**、想原地加向量、要 hybrid |

### 3. 选型决策树（可执行判断）

把上面压成几条能直接照着做的判断：

- **从 0 起步 / 数据量百万级 / 团队已用 Postgres** → **闭眼上 pgvector**。零新增运维、SQL 混合查询无缝、足够用很久。**这是大多数项目的正确起点**，别一上来就 Milvus（典型过度工程，见 know.md "躲进机械建设"的反面——别用部署复杂度假装在做大事）。
- **数据量千万~亿 / 要专用向量库的性能和过滤** → **Qdrant**（轻量专用）或 **Weaviate**（要开箱模块）。
- **数据量亿级以上 / 有运维团队** → **Milvus**（为超大规模而生，但别低估它的运维成本）。
- **已经重度使用 Elasticsearch 做全文检索** → **直接用 ES 的向量能力**，省一套系统，还天然支持 hybrid（下一节）。
- **云上托管、不想自己运维** → 各云厂商和上述产品大多有托管版（按 2026 初，名单会变），用托管换运维省心，代价是成本和锁定。

> **面试加分点**：被问"你怎么选向量库"，**先反问规模、团队、是否已有 PG/ES、是否需要混合查询**，再给带 trade-off 的结论。能说出"百万级别先 pgvector，别过早上 Milvus，分布式向量库的运维成本经常被低估，等真撞到 pgvector 的规模天花板再迁移也不迟"——这就是 P7 级别的判断力：**敢说"这个现在不该上"并讲清为什么**。
>
> **新手第一误区**：以为"向量库越专业越高级越该用"。真相相反——**绝大多数 RAG 项目的数据量，pgvector 完全扛得住**，过早上分布式向量库是拿运维复杂度换不存在的规模需求，纯亏。

### 4. 工程推论

1. **选向量库先问规模和"数据现在在哪"。** 数据已在 PG → pgvector；已在 ES → ES 向量；都不是且超大 → 专用库。让基础设施现状决定，而不是让"听起来酷"决定。
2. **混合查询能力是企业 RAG 的硬指标。** 多租户隔离、权限过滤、时间范围——这些都要"向量检索 + 结构化过滤"。选型必须确认这条支持得好不好。
3. **运维成本要算进总账。** 分布式向量库（Milvus）的吞吐很猛，但要专人运维、扩容、调优。小团队用它常常是"买了把屠龙刀切菜"。
4. **能晚一点上重型方案就晚一点。** pgvector → Qdrant → Milvus 是一条随规模递进的路，没必要一步到位。**过早优化在向量库选型上同样是万恶之源。**

> 心法：**向量库选型的第一性原理不是"哪个最强"，是"哪个最匹配你的数据规模、现有基础设施和团队能力"。九成项目的正确答案，比你以为的更朴素。**

## 五、Hybrid Search：BM25 + 向量，以及 RRF 融合

纯向量检索很强，但**不是万能**。这一节讲的 hybrid 思路，是召回质量从"能用"到"好用"的关键一跃，也直接为第 9 章的高级 RAG / rerank 铺路。

### 1. 背景：向量检索的盲区

向量检索擅长**语义模糊匹配**，但有两个它干不过关键词的硬盲区：

- **精确字面匹配**：错误码 `ERR_4012`、产品型号 `X-2000`、人名、专有名词、缩写。这些东西**没有语义可言**，向量模型可能把 `ERR_4012` 和 `ERR_4013` 编码到几乎同一个点（数字对它"看不见"，回想第 1 章 token 化的"数不对字母"），但用户要的就是**精确那一个**。
- **关键词刚性需求**：用户明确搜 "GDPR 第 17 条"，他要的是字面包含这几个词的段落，语义近似的"数据删除权相关条款"反而不是他想要的精确目标。

这正是 **BM25（关键词检索）** 的主场——它是几十年成熟的倒排索引算法，**精确、可解释、对专有名词/罕见词敏感**（罕见词权重高，正好补向量的短板）。

**核心洞察**：向量检索和 BM25 **优劣互补**——
- 向量强在语义、弱在精确字面；
- BM25 强在精确字面、弱在语义换词。

把两者**结合**，就是 **hybrid search（混合检索）**：**两路并行检索，各出一批候选，再融合成一个排序。** 既要语义的召回广度，又要关键词的精确命中。这是现代生产级 RAG 的**标配**，不是可选项。

### 2. 怎么融合：RRF（Reciprocal Rank Fusion，倒数排名融合）

两路检索各返回一个排序列表，分数体系还不一样（cosine 在 [-1,1]、BM25 分数是另一个量纲），**不能直接把分数相加**。怎么融合？

最朴素的想法是"分数归一化后加权求和"，但归一化很脆弱（分数分布一变就失灵）。生产中更稳健、更常用的是 **RRF**，它的妙处是**只用排名（rank），不用原始分数**，从而绕开了量纲不可比的问题。

**RRF 公式（直觉版）**：一个文档的最终分数 = 各路检索中 `1/(k + 该文档在这一路的排名)` 之和。`k` 是个平滑常数（常取 60，按 2026 初的经验值）。

```
RRF_score(doc) = Σ_over_each_retriever  1 / (k + rank_of_doc_in_that_retriever)
```

**为什么 RRF 好用（这是考点）：**
- **只看排名不看分数**——彻底回避了 cosine 和 BM25 分数量纲不可比的难题。一个文档在某路排第 1，贡献 `1/(60+1)`；排第 100，贡献 `1/(60+100)`，越靠前贡献越大，且贡献随排名递减得很平滑。
- **被多路同时召回的文档会"叠加得分"自然上浮**——一个文档既被向量召回（排第 3）、又被 BM25 召回（排第 5），两份倒数相加，总分高于只被一路召回的，**它大概率就是真正相关的**。这个"多路共识"的信号非常强。
- **无参数、无需训练、实现极简**——这是它能成为事实标准的关键。鲁棒、便宜、好懂。

> **工程含义**：RRF 是"穷人的融合"，但效果常常打败精心调过权重的加权求和，因为它对分数分布不敏感、不需要per-query 调参。**很多向量库（按 2026 初，如 Qdrant、Weaviate、ES 等）已内置 hybrid + RRF**，你常常只要开个开关、给两路 query，不必自己实现融合逻辑。但**懂原理你才知道它什么时候会失灵**——比如某一路检索整体质量极差时，它的排名照样参与融合会拉低结果，这时你得给两路加权（RRF 有带权变体）或干脆砍掉烂的那一路。

### 3. hybrid 在 RAG 链路里的位置（承上启下）

把这章和后面串起来，hybrid 在完整 RAG 漏斗里是这样的（详见第 8、9 章）：

```
用户 query
   ├──→ 向量检索（语义，top-N）  ┐
   │                            ├──→ RRF 融合 ──→ 融合后 top-K ──→ rerank 精排（第 9 章）──→ top-k 喂给 Claude 生成
   └──→ BM25 检索（关键词，top-N）┘
```

注意这条链的层次：**hybrid 是"粗召回"阶段的提质**（把对的文档尽量捞进候选池），**rerank 是"精排"阶段的提质**（在候选池里把最相关的顶到最前，第 9 章详讲）。两者解决不同问题、在不同位置，**不是二选一，是叠加用**。理解这个分层，你就懂了为什么生产 RAG 既要 hybrid 又要 rerank。

### 4. 工程推论

1. **生产 RAG 默认上 hybrid，不要只用纯向量。** 专有名词、错误码、型号这类精确需求，纯向量会漏，BM25 补得上。被问"纯向量检索够吗"，答"语义够、精确字面不够，专有名词/罕见词要靠 BM25，所以生产用 hybrid"。
2. **融合优先用 RRF，因为它只看排名、鲁棒、零调参。** 能说清"为什么不直接加权求和分数"（量纲不可比、归一化脆弱）是关键考点。
3. **hybrid 和 rerank 是漏斗的两个不同阶段，叠加而非替代。** hybrid 管"召回广"，rerank 管"排序准"（见第 9 章）。
4. **优先用向量库内置的 hybrid，但要懂原理以便兜底。** 内置开关方便，但烂的那一路会拖累融合，这时要懂得加权或剔除。

> 心法：**没有一种检索方式能同时擅长"懂意思"和"抠字面"。hybrid 的智慧不是找一个全能选手，是让擅长不同事的两个选手各干各的，再用 RRF 公平地把两份意见合成一份。**

## 六、量化与规模化：PQ、SQ 与亿级向量的成本账

最后一节回到工程现实：当向量从百万涨到亿、十亿，**内存和成本会先于召回率把你逼到墙角**。量化（quantization）就是这堵墙前的主要武器。这部分偏进阶，做超大规模才用得上，但**面试问规模化必考**。

### 1. 背景：向量存储的成本爆炸

算笔账：1 个 1024 维向量，每维 float32 = 4 字节，单条 = 4KB。

- 100 万条 = 4 GB——小意思。
- 1 亿条 = 400 GB——开始肉疼，HNSW 还要全进内存。
- 10 亿条 = 4 TB——纯靠内存存原始向量，**机器买不起也塞不下**。

问题清晰了：**高维 float32 向量太占空间**。量化的思路一句话：**用更少的比特表示每个向量，牺牲一点精度，换巨大的存储/内存压缩。** 这又是那桩熟悉的"精度换资源"交易，只不过这次换的是**内存/存储**。

### 2. 两种主流量化

**SQ（Scalar Quantization，标量量化）—— 简单粗暴的降精度**

把每一维从 float32（4 字节）压成 int8（1 字节），**直接 4 倍压缩**。原理：找出每维数值的范围，映射到 256 个离散档位（int8）。

- **优点**：实现简单、压缩稳定（4×）、精度损失通常很小（int8 对大多数检索够用）。
- **定位**：**性价比最高的"第一档"量化**。内存有点紧但没到极限时，SQ 是低风险首选——4 倍压缩，召回几乎不掉。

**PQ（Product Quantization，乘积量化）—— 极致压缩**

更激进。把高维向量**切成若干段子向量**，对每一段子空间**各自做聚类**（如每段聚成 256 个中心），然后**每段只存"它属于哪个中心"的编号**（1 字节）。比如 1024 维切成 8 段，每段一个字节，**整个向量从 4096 字节压到 8 字节——512 倍压缩**。

- **优点**：压缩率惊人（可达几百倍），亿级、十亿级向量的**唯一现实选择**。
- **代价**：精度损失明显大于 SQ（毕竟用"最近的中心编号"近似了真实子向量），召回会掉。**所以 PQ 几乎总是配合 rerank**——PQ 粗召回（快、省内存，召回略糙），再用原始向量或 rerank 模型精排（见第 9 章），把 PQ 损失补回来。
- **经典组合**：**IVF + PQ**——IVF 先聚类缩小搜索范围，PQ 把每个向量压到几字节。这是 Faiss 等库做十亿级检索的看家组合（按 2026 初，以官方为准）。

**对照表（精度换内存的三档）：**

| 方案 | 压缩率（粗估） | 精度损失 | 何时用 |
|---|---|---|---|
| **不量化（float32）** | 1× | 无 | 百万~千万级，内存够，要最高召回 |
| **SQ（int8）** | ~4× | 很小 | 内存有点紧的"第一档"优化，低风险 |
| **PQ** | 数十~数百× | 明显（需 rerank 补） | **亿级以上**、内存是硬约束时的主力 |

### 3. 规模化的完整成本视角

量化只是规模化的一个杠杆。做亿级 RAG，你要在脑子里有这张总账（这正是 P5 平台设计文档要算的容量估算）：

- **存储/内存账**：原始向量多大？量化后多大？HNSW 全进内存还是 IVF+PQ 能上磁盘？
- **embedding 账**：全量首次 embed 的算力/时间/费用？增量更新的速率？（10 亿文档首次 embed 是一笔可观的成本）
- **检索延迟账**：目标 P99 多少毫秒？这反过来约束你 efSearch/nprobe 能开多大、量化能压多狠。
- **构建账**：索引重建一次要多久？（embedding 模型升级会触发全量重建——回想推论四）
- **更新账**：文档增删改的频率？HNSW 删除不友好，高频更新场景要特殊设计或换索引。

> **面试加分点（系统设计题常考）**：被问"10 亿文档的 RAG 检索怎么设计"，能拉出这张账并给量级估算——"10 亿×1024×4B≈4TB 原始向量，纯内存不现实，上 IVF+PQ 压到几十 GB 量级能上磁盘，nprobe 调到召回够用，PQ 损失靠 rerank 兜底，P99 控制在百毫秒"——这就从"会用 API"跨到了"会做系统"。

### 4. 与生成层的衔接：检索结果进 prompt 后的缓存账

检索做完，文档要拼进 prompt 喂给生成模型（Claude，见第 8 章）。这里有一个**与本章衔接、且生产省钱的关键点**，用得上第 2 章的 prompt caching：

RAG 的 prompt 通常结构是 `稳定的系统指令 + 检索到的文档 + 用户问题`。**prompt caching 是前缀匹配**（见第 2 章）——只有"从开头到某个断点完全字节一致"的前缀才能命中缓存。这对 RAG 的直接含义：

- **检索到的文档块每次都不同**（不同 query 召回不同文档），所以**文档部分通常无法跨请求缓存**——这是 RAG 的固有特性，别指望靠缓存省检索拼进来的那部分。
- 但**系统指令是稳定的**，应该放在 prompt 最前面并设缓存断点，让这部分跨请求命中。
- **反面坑**：如果你把"当前时间戳"或检索到的文档**放在系统指令之前**，会把后面整个前缀的缓存击穿（第 2 章讲过：前缀任何字节变化使其后全部失效）。**渲染顺序要按"越稳定越靠前"排**：稳定系统指令 → （断点）→ 检索文档 → 用户问题。

> **工程含义**：embedding/检索这一层产出的"动态文档"，和生成层的"静态指令前缀"在缓存上是两种命运。设计 RAG 的 prompt 拼装时，**把稳定的放前面吃缓存、动态的放后面**——这是检索层和生成层在工程上的接缝，做对了能实打实省钱。（这条会在第 8 章拼 prompt 时再细化。）

### 5. 工程推论

1. **量化是"精度换内存"，分档上：SQ 先行、PQ 极致。** 被问"亿级向量内存放不下怎么办"，答"量化压缩——SQ 4× 低损失打头阵，再不够上 PQ 几百倍压缩配 rerank 补精度，常配 IVF 成 IVF+PQ"。
2. **PQ 必配 rerank。** PQ 的召回损失明显，靠下游精排补回来，二者是套餐。能说清"PQ 为什么要配 rerank"是懂规模化的标志。
3. **规模化是一张多维成本账，不只是召回率。** 存储、embedding、延迟、构建、更新五本账一起算——这是平台级（P5）设计的思维方式。
4. **检索层的动态文档无法靠 prompt caching 省，但稳定前缀能省。** 设计 RAG prompt 时按"稳定在前、动态在后"排，吃住生成层的缓存（接第 2、8 章）。

> 心法：**小数据量的 RAG，怎么搞都行；亿级 RAG，赢家不是召回率最高的，是在召回、内存、延迟、成本这张多维账上做出最优平衡的。量化、IVF、rerank 不是炫技，是被成本逼出来的工程必然。**

> **embedding 与向量检索是 RAG 这座楼的地基和承重墙。** 这一章你建立了四块能下判断的肌肉：**embedding 选型锁死召回上限**（漏斗最上游，最不该省）、**相似度度量定义你到底在比什么**（cosine 默认、归一化是正确性）、**ANN 是明牌的精度换速度**（HNSW 吃内存换召回、IVF+PQ 换规模，efSearch/nprobe 是运行时旋钮）、**向量库选型按数据规模和现有基建走**（九成项目 pgvector 起步就对、别过早上 Milvus）。再加两件武器：**hybrid+RRF 让语义和字面互补**（承接第 9 章 rerank），**量化让亿级向量在成本墙前活下来**。

## 章末五件套

### 一、高频面试题（11 道，附答题框架）

**Q1：word2vec 这类静态词向量和 BERT 这类上下文 embedding 的本质区别是什么？**
- 框架：静态——一词一向量，无视上下文，处理不了一词多义（"苹果"只有一个平均向量）；上下文——同一个词在不同句子里向量不同，由整句上下文动态算出，化解多义。
- 陷阱：只说"BERT 更先进"是空话。**要点在"一词一向量 vs 一词多向量"这个分水岭**，并能举一词多义的例子。加分：现代检索用的是第三代——专门用对比学习训练的句子级 embedding 模型，且检索是非对称任务（短 query 找长 doc），常需 query/doc 不同前缀。

**Q2：文本语义检索为什么默认用 cosine 而不是欧氏距离？归一化在其中起什么作用？**
- 框架：cosine 只看方向不看模长，契合"语义相似关心意思方向、不关心向量长度（长文档模长大但意思可能相同）"；归一化后所有向量到单位球面，此时 dot product ≡ cosine，且内积算得更快。
- 陷阱：答不出"为什么不看模长"是没理解。**关键句**："归一化后内积等于 cosine 但更快，所以生产常用'归一化+内积'"。能点出"忘归一化不报错只是变差"是实战分。

**Q3：什么是 ANN？为什么生产用 ANN 而不是精确 KNN？用什么指标衡量？**
- 框架：精确 KNN 跟每个向量都算一遍，O(N×d)，亿级数据单查秒级，不可用；ANN 近似最近邻，用一点召回损失换几个数量级速度；用 **recall@k**（ANN 的 top-k 与精确 top-k 的重合比例）量化，生产 0.95~0.99 通常够，因为下游有 rerank 兜底。
- 陷阱：把 ANN 说成"另一种精确算法"就错了。核心是 **Approximate** 和"精度换速度"这桩明牌交易。

**Q4：讲讲 HNSW 的工作原理和关键参数，哪些建库时定、哪些查询时定？**
- 框架：分层小世界图，像高速公路网/跳表的高维推广——顶层稀疏连远点快速跨区、逐层下沉精细逼近。参数：**M（每节点连接数）和 efConstruction（建索引探索宽度）建库时定、改了要重建**；**efSearch（查询探索宽度）查询时动态调，是 recall vs latency 的运行时旋钮**。
- 陷阱：分不清哪个参数建库定哪个查询定，是没真用过。加分：点出 HNSW 内存贵、删除不友好。

**Q5：HNSW 和 IVF 怎么选？**
- 框架：HNSW 召回高、查询快、但**内存贵**（图常驻内存）、删除不友好，甜区百万~千万级内存够的场景；IVF（聚类分块，nprobe 控制搜几个簇）配 PQ 后**内存可控、能上磁盘**、增删友好，甜区亿级以上内存受限。决策：中小规模要召回上 HNSW，超大规模内存紧上 IVF+PQ。
- 陷阱：只背名字不说 trade-off。要落到"内存预算"这个决定性维度。

**Q6：你怎么给一个企业 RAG 项目选向量库？**
- 框架：**先反问规模、团队、数据现在在哪（PG?ES?）、要不要混合查询**，再按六维度（规模/运维/混合查询/索引可调/一致性/生态）评估。结论：百万级+已用 PG → pgvector 起步；千万~亿 → Qdrant/Weaviate；亿级+有运维团队 → Milvus；已重度用 ES → ES 向量+hybrid。
- 陷阱：一上来就推 Milvus 是新手。**P7 答案是"别过早上分布式向量库，pgvector 能扛很久，撞墙再迁不迟"**——敢说"现在不该上"。

**Q7：什么是 hybrid search？为什么需要它？RRF 又是什么？**
- 框架：纯向量擅长语义、弱在精确字面（错误码、型号、专有名词、罕见词向量"看不见"）；BM25 擅长字面、弱在语义换词；hybrid 两路并行召回再融合，取长补短。RRF（倒数排名融合）**只用排名不用原始分数**，绕开 cosine 和 BM25 分数量纲不可比的问题，被多路共同召回的文档得分叠加自然上浮，鲁棒、零调参。
- 陷阱：不知道"为什么不直接加权求和分数"（量纲不可比、归一化脆弱）就没答到点。加分：hybrid 管召回广、rerank 管排序准，是漏斗两个阶段，叠加不是二选一。

**Q8：embedding 模型升级，工程上意味着什么？这对选型有什么启示？**
- 框架：query 和 doc 必须同模型 embed，换模型 = 两个向量空间坐标系不同，旧向量全作废 → **必须全量文档重新 embed、重建整个索引**，亿级数据是可观的算力/时间/钱。启示：embedding 选型有路径依赖，要"按长期选"，不能图新鲜频繁换。
- 陷阱：以为换模型是"改个配置"。核心是"全量重建"这个重成本动作。

**Q9：向量从百万涨到十亿，内存放不下，怎么办？**
- 框架：量化压缩，精度换内存。**SQ（标量量化）** float32→int8 直接 4×、损失很小，低风险第一档；**PQ（乘积量化）** 切段聚类只存中心编号，数十~数百倍压缩，亿级唯一现实选择，但精度损失明显**必须配 rerank** 补回来；经典组合 IVF+PQ。还要算存储/embedding/延迟/构建/更新五本账。
- 陷阱：只说"用 PQ"不提"配 rerank"是不完整。加分：给量级估算（10 亿×1024×4B≈4TB）。

**Q10：RAG 里检索到的文档拼进 prompt，能用 prompt caching 省钱吗？**
- 框架：prompt caching 是前缀匹配（见第 2 章），任何字节变化使其后缓存失效。RAG 里**检索文档每次不同，无法跨请求缓存这部分**（固有特性）；但**稳定的系统指令放最前面+缓存断点能跨请求命中**。渲染顺序必须"稳定在前、动态在后"，且**绝不能把时间戳/动态内容放系统指令之前**（会击穿后面全部缓存）。
- 陷阱：以为"开了缓存就全省了"。要点是动态文档省不了、稳定前缀能省，以及顺序坑。

**Q11：给一个中文为主的企业知识库选 embedding 模型，你按什么维度选？为什么不直接看 benchmark 榜单挑第一？**
- 框架：六维度过一遍——**维度**（召回 vs 存储/算力成本）、**多语言/中文表现**（中文语料是否充分）、**最大输入长度**（约束 chunk 能多大，超了静默截断）、**对称 vs 非对称**（检索要非对称 + 注意 query/doc 前缀）、**托管 vs 自托管**（企业数据合规常逼你自托管）、**license/商用**。中文场景开源主力是 bge 系（如 bge-m3）/Qwen-embedding，托管选 OpenAI/Cohere。
- 陷阱：直接背榜单挑第一就错了。**两个原因**：(1) 通用榜单（MTEB 中文榜）和你的垂域语料分布不一致，榜一到你的业务可能输给榜二；(2) 名单按季度刷新，背了就过期。**正确做法**：用自己的 ≥20 条标注 query 实测 recall@10，用自己的数字拍板——"别人的 benchmark 是参考，你的标注集才是裁判"。加分：能点出 Matryoshka（MRL）可裁剪维度——一次 embed 多档维度可用，低维粗筛 + 高维精排，且和量化正交可叠加。

**暗坑提示汇总：**
- **暗坑 A（归一化 axis）**：`l2_normalize` 必须按行（每个向量）归一化，`axis=1` + `keepdims=True`，写错会把整个矩阵当一个向量、或触发广播错误。
- **暗坑 B（argsort 方向）**：cosine 越大越像，但 `np.argsort` 默认升序，取 top-k 要 `[::-1]` 反转或取尾部——方向搞反会返回**最不相似**的 k 个，且不报错。
- **暗坑 C（RRF 别用分数）**：RRF 的全部价值在"只用排名绕开量纲不可比"。测试 4 就是来抓"你是不是偷偷把 cosine 分数也加进融合了"——加了就违背了 RRF 的设计，且让它对分数分布敏感、退化成脆弱的加权求和。
- **暗坑 D（前缀/归一化的真实坑）**：本题 `embed` 是占位的不涉及前缀；但**接真实模型时**，记得按 model card 给 query 和 document 加正确前缀、确认模型输出是否已归一化——这俩错了**不报错只是召回变差**（对应正文推论三），是真实项目里最难排查的 bug。
