> 这章排在阶段四 LLMOps 的中段，紧跟评测（第 17 章）之后、安全（第 21 章）之前，因为它是把你前面所有单点能力**收口成一条可治理的统一出口**的那一层。第 17 章给了你"这版好不好"的尺子；这一章给你"线上这一刻还活不活、贵不贵、能不能换"的开关。**没有网关，你的 RAG、agent、工具调用全都是一根根直连 provider 的裸线**——provider 抖一下你整条服务跟着抖，想换模型要改代码重新发版，想知道这个月烧了多少钱只能去翻三家账单。网关就是把这些裸线全部插进一块**配电盘**：统一进、统一出、统一计量、统一熔断。
>
> 替你解决的问题：把"模型 provider"从一个**硬编码在业务代码里的依赖**，降级成一个**配置文件里的可替换条目**。建立一个判断——**LLM Gateway 之于你的 AI 系统 = API Gateway / Service Mesh 之于你的微服务**：业务不该知道下游是谁、在哪、怎么认证、挂了怎么办，这些全是基础设施层的事。你做后端时早就接受"业务代码不该写死数据库 IP"，这一章就是把同一条工程纪律搬到 LLM provider 上。
>
> 读法提示：这章每个机制后面紧跟**工程推论**（编号列出，每条都能独立成一道面试题）。第一遍至少把"网关六大职责表""路由策略三档对照表""fallback 触发分类表""自建 vs LiteLLM 对照表""语义缓存命中/击穿表"这五张表啃下来；第二遍再吃"**流式 fallback 阶段表**"（流式怎么把 fallback 的透明性撕开一道口子，第四节四.4）这张进阶表——它是区分"画过架构图"和"真扛过线上流式链路"的分水岭。代码题对应主线项目 **P4 增量**——在 P2/P3 的 RAG / agent 前面插一层网关，做 provider 抽象 + fallback，让"换模型"和"主模型挂了"都不再是发版事件。
>
> 时效标注：LiteLLM / Portkey / Kong AI Gateway 等具体方案的功能边界、各家模型名单与价格按 **2026 初**写，名单和接口按季度过期，**重点学方法论**；而"网关是统一收口层""provider 进配置不进代码""fallback 要按错误类型分级""语义缓存的命中是概率事件"这些是**平台工程规律**，讲透到骨子里，不会变。
>
> 交叉引用：网关里最肥的一块成本杠杆是 prompt caching 和语义缓存，缓存的物理机制在**第 2 章**讲透（前缀匹配、命中即 0.1×），本章只讲"缓存放在网关层怎么做、跨 provider 怎么处理"；网关产出的 per-request 成本/延迟/token 怎么接进 tracing、怎么做全链路下钻在**第 18 章**可观测性展开；网关作为"统一入口"天然是 prompt injection 防护、PII 脱敏、输出审查的**最佳挂载点**，这部分安全逻辑在**第 21 章**展开，本章只指出"为什么安全要集中在网关而不是散在每个 agent 里"。

## 一、为什么需要网关：六个理由，每一个都来自一次线上事故

### 1. 背景：没有网关的世界长什么样

你 P0~P3 一路写下来，调 LLM 的代码大概长这样：业务函数里 `import anthropic`，`client = anthropic.Anthropic()`，硬编码 `model="claude-opus-4-8"`，直连。能跑。demo 阶段完全没问题。

问题在你**上线之后第一周**集中爆发。我们不抽象地讲"网关有什么好处"，而是把六个好处倒过来，从**六次真实会发生的事故**讲——每一次事故，都是一条你迟早要补的网关职责。

> 新手第一误区：以为网关是"等量大了再加的优化"。错。网关是**第一次真上线就该有的基础设施**，跟你不会让业务代码直连数据库一个道理。等量大了再补，意味着你要在一堆散落各处的直连点上做手术，比一开始就插一层贵十倍。

### 2. 六次事故 → 六大职责

#### 事故一：provider 挂了，你整条服务跟着挂 → fallback

某个周二下午，Anthropic 返回一片 `529 overloaded_error`（这是真实存在的错误码，意思是"服务暂时过载"，可重试）。你的直连代码没有任何兜底，异常一路冒泡到用户。**你的可用性，此刻完全等于单一 provider 的可用性。**

网关职责：**fallback / failover**——主模型不可用时自动切到备用模型/备用 provider，对业务透明。

#### 事故二：要换模型，得改代码发版 → 模型抽象层

老板说"Opus 太贵，把摘要这条链换成 Sonnet 试试 A/B"。你发现 `model="claude-opus-4-8"` 这个字符串散落在十七个文件里。改完、过 CR、重新发版、灰度——一个本该是**配置变更**的事，被你做成了**代码变更**。

网关职责：**模型抽象**——模型名进配置不进代码，换模型是改一行 config + 热加载，不是发版。

#### 事故三：月底三张账单，对不上账 → 成本可观测

财务问"这个月 LLM 花了多少、哪个业务线烧的"。你打开 Anthropic 账单、OpenAI 账单、火山引擎账单，三个格式、三个口径，且**没有任何一张能告诉你"是搜索业务还是客服业务花的"**——因为成本数据在 provider 那边，按 API key 聚合，不按你的业务维度。

网关职责：**统一计量与成本归因**——每个请求打上 `tenant` / `business_line` / `feature` 标签，成本在网关侧按你的维度聚合。

#### 事故四：一个租户写死循环，把全公司配额烧穿 → 多租户配额与限流

你的 agent（第 12-16 章）有个不收敛的 bug，某个大客户触发后疯狂调用，几分钟把你整个 org 的 TPM（tokens per minute）配额打满，**所有其他租户一起 503**。

网关职责：**多租户配额与限流**——每个租户独立配额桶，一个租户烧穿不影响其他人，网关侧就把超额请求挡掉。

#### 事故五：safety / injection 防护散在每个 agent 里，漏了一个 → 安全集中

你有 RAG、有 research agent、有客服 bot，三套代码各自接 LLM。prompt injection 防护你在 RAG 里写了，agent 里忘了。**安全策略只要有一个入口没覆盖，整条防线就是漏的。**

网关职责：**安全集中**——把 injection 检测、PII 脱敏、输出审查挂在网关这唯一出口上，任何业务接 LLM 都必须穿过它（详见第 21 章）。

#### 事故六：想知道"哪个 prompt 慢、哪次调用贵"，无从下钻 → 可观测

P99 延迟报警，你想知道是哪条链路、哪个 prompt、prefill 还是 decode 阶段慢。直连代码什么都没记，你只有一个干巴巴的"慢"。

网关职责：**可观测**——每个请求的 model / tokens / 延迟（拆 TTFT 和 decode）/ 成本 / 缓存命中，统一埋点，可下钻（详见第 18 章）。

### 3. 网关六大职责对照表（本章地图，背下来）

| 职责 | 解决的事故 | 没有它会怎样 | 类比你的后端直觉 |
|---|---|---|---|
| **多 provider 统一接入** | 接口不一致 | 每接一家写一套适配 | 多数据源统一 DAO 层 |
| **fallback / 优雅降级** | provider 挂 | 可用性 = 单 provider 可用性 | 主从切换 / 熔断降级 |
| **模型抽象（名进配置）** | 换模型要发版 | 配置变更被做成代码变更 | DB 连接串进配置 |
| **统一计量与成本归因** | 账单对不上 | 不知道钱花哪了 | 按业务维度的资源计费 |
| **多租户配额与限流** | 单租户烧穿 | 一个租户拖垮所有人 | 租户级 rate limit / quota |
| **统一可观测** | 无法下钻 | 只有"慢"没有"为什么慢" | APM / tracing |

> 心法：**网关的本质不是"加功能"，是"加约束"——它强制所有 LLM 流量走同一个收口，于是治理才有了唯一的挂载点。** 治理能力（限流、成本、安全、观测）的前提，永远是"流量先收口"。

> 工程推论：
> 1. **网关是上线第一天的基础设施，不是规模化后的优化**——它要解决的六件事，量小的时候只是不痛，不是不存在。
> 2. **"统一收口"本身就是最大的价值**，具体功能（fallback、缓存）都是收口之后才挂得上去的二阶能力。
> 3. **判断一个团队 LLMOps 成不成熟，看它有没有唯一收口**——还在业务代码里直连 provider 的，治理一定是散的、漏的。

## 二、模型抽象层：把 provider 从代码里赶进配置

### 1. 背景：硬编码 provider 的三宗罪

直连 provider，等于把三个"易变量"焊死进了"稳定层"（业务代码）：

- **provider SDK**：`anthropic.Anthropic()` vs `openai.OpenAI()`，初始化、参数、返回结构全不一样。
- **模型 ID 字符串**：`claude-opus-4-8`（注意：精确 ID，**不加日期后缀**，加了会 404）散落各处。
- **认证方式**：API key、base_url、region，每家一套。

这三个东西**每一个都会变**，而业务逻辑**不该因为它们变而改**。这正是依赖倒置原则（DIP）的教科书场景——业务依赖一个稳定的抽象接口，provider 是这个接口的可替换实现。

### 2. 演进：从直连，到适配器，到声明式路由

- **阶段零·直连**：业务里 `import anthropic`。最快，最脆。
- **阶段一·自己包一层 client wrapper**：定义一个 `LLMClient` 接口，`chat(model, messages) -> response`，内部按 provider 分发。解决了"SDK 不一致"，但模型映射还在代码里。
- **阶段二·声明式模型注册表**：模型名是一个**逻辑别名**，映射关系进配置：

```yaml
# 逻辑名 → 物理 provider + model + 认证
models:
  fast-summarizer:                 # 业务只认这个逻辑名
    primary:   {provider: anthropic, model: claude-haiku-4-5}
    fallback:  {provider: openai,    model: gpt-5-mini}
  smart-agent:
    primary:   {provider: anthropic, model: claude-opus-4-8}
    fallback:  {provider: anthropic, model: claude-sonnet-4-6}
```

业务代码只写 `gateway.chat("fast-summarizer", messages)`。**换模型、加 fallback、调 region，全在 YAML 里改，热加载生效，零发版。**

### 3. 关键设计判断：抽象到哪一层？

这是网关设计里第一个真正的 trade-off，也是面试高频追问点。

| 抽象层次 | 业务看到的 | 优点 | 代价 |
|---|---|---|---|
| **薄抽象**（只统一 transport） | 各家原生参数 | 不丢任何 provider 特性 | 业务仍要知道 provider 差异 |
| **厚抽象**（统一一套参数 schema） | 一套标准参数 | 业务完全 provider-agnostic | **难处理各家专有能力** |

厚抽象的坑非常具体，必须讲透：**各家的"行为控制通道"已经不一样了**。按 2026 初，Anthropic 的 Opus 4.7/4.8 与 Fable 5 已经**移除了 `temperature` / `top_p` / `top_k`**（传了直接报 400），思考深度改用 `thinking:{type:"adaptive"}` + `effort` 档位（low/medium/high/xhigh/max）控制；而 OpenAI 系仍在用 temperature。

> 坑点提示：如果你的厚抽象 schema 里硬塞了一个 `temperature` 字段并"透传给所有 provider"，那么同一份配置打到 Opus 4.8 上**会直接 400**。厚抽象不是"取所有 provider 参数的并集硬塞"，而是**取一个稳定的公共子集 + 给专有参数留一个 `provider_params` 逃生口**。这就是"行为控制主通道正从采样超参迁移到自然语言 + 思考预算"这个趋势在网关层的直接后果——你的抽象层要能容纳这种迁移，不能假设所有 provider 共享同一套旋钮。

> 面试加分点：被问"你怎么设计模型抽象层"，平庸答案是"包一层统一接口"。高手答案要点出**抽象粒度的 trade-off**，并能举出"Opus 4.8 没有 temperature 而 GPT 有"这种**真实的能力不对称**，说明"为什么厚抽象必须留逃生口"——这一句直接区分"读过文档"和"踩过坑"。

> 工程推论：
> 1. **模型名是逻辑别名，不是物理 ID**——业务认 `smart-agent`，物理映射到 `claude-opus-4-8` 还是别的，是配置层的事。
> 2. **抽象层要为"参数 schema 的代际迁移"留口子**——采样超参正在被思考预算替代，写死任何一家的参数模型都会在下一代模型上崩。
> 3. **逃生口（`provider_params` 透传）是厚抽象的必备件**，没有它，厚抽象会在第一个专有能力上破功。

## 三、路由策略：同一个逻辑名，怎么选物理实现

收口之后，网关手上握着"这个请求该打给谁"的决策权。这就是**路由（routing）**——网关从基础设施升级为"智能调度层"的地方。

### 1. 三种路由维度

| 路由维度 | 决策依据 | 典型场景 | 主要风险 |
|---|---|---|---|
| **按能力路由** | 任务难度 / 类型 | 简单分类走 Haiku，复杂推理走 Opus | 难度判断本身要花成本/可能判错 |
| **按成本路由** | 预算 / 单租户档位 | 免费用户走 Fast 档，付费走 Frontier | 省钱可能牺牲质量，要有质量底线 |
| **按负载路由** | provider 健康度 / 配额余量 | 主 provider 限流时分流到备用 | 备用质量不一致，要标注降级 |

### 2. 按能力路由：省钱的最大杠杆，也是最容易翻车的地方

核心思路：**不是所有请求都值得用 Frontier 模型**。一个"把这句话分类成正面/负面"的请求，用 Haiku 4.5（输入约 $1 / 输出约 $5 每百万 token）就够了，用 Opus 4.8（$5 / $25）是**五倍浪费**。记住第 1 章的价格不对称：**输出价≈输入价的 5 倍**，所以"输出长的任务用错模型"罚得最狠。

但"按难度选模型"有个鸡生蛋问题：**判断难度本身要不要花一次 LLM 调用？** 如果用一次 Opus 调用去判断"这题该不该用 Opus"，你已经付了 Opus 的钱。生产里的解法分三档：

1. **规则路由**（最便宜）：按 query 长度、是否命中关键词、来自哪个 endpoint，硬规则分流。零额外成本，但粗。
2. **小模型分诊**（router model）：用一个 Haiku 级别的便宜模型做 query 分类，输出"难度档位"，再据此路由。加一次便宜调用，换可能省下的一次贵调用。
3. **级联（cascade）**：先用便宜模型答，对答案做一个置信度/质量检查，**不达标再升级**到贵模型重答。最省，但有"答两次"的延迟风险和"质量检查器本身可不可信"的问题（呼应第 1 章推论四：模型不能可靠自评，质量检查要外置）。

> 坑点提示：级联路由最大的隐藏成本是**最坏情况下你付了两次钱**——便宜模型答一遍、判定不行、贵模型再答一遍。如果你的 query 分布里"难"的占多数，级联反而比"一律走贵模型"更贵更慢。**路由策略必须用真实流量分布回测，不能拍脑袋。** 这正是 know 里那条"N=1 不算 feasibility"的应用——路由策略要在 ≥数千条真实 query 上看命中分布，才能区分"省钱信号"和"采样噪声"。

### 3. 按负载路由：和 fallback 的边界

按负载路由（主 provider 接近限流就分流）和 fallback（主 provider 挂了就切换）**经常被混为一谈，但触发条件和语义不同**：

- **负载路由**是**主动**的、**预防性**的——还没出错，只是配额快满了，提前分流摊负载。
- **fallback**是**被动**的、**故障性**的——已经出错了（429/529/超时），切到备用救场。

实现上两者可以共用一套 provider 健康度探测，但决策点不同：负载路由看"余量百分比"，fallback 看"是否已失败"。

> 工程推论：
> 1. **路由是网关从"管道"变"调度器"的分水岭**——能按能力/成本/负载选下游，才叫智能网关。
> 2. **按能力路由的省钱潜力巨大，但难度判断自带成本**，规则/分诊/级联三档各有 trade-off，按 query 分布选。
> 3. **负载路由（预防）和 fallback（故障）是两件事**，混在一起讲会答错面试题——前者看余量，后者看失败。

## 四、Fallback 与优雅降级：可用性是你在网关层造出来的

这是网关最硬核、面试最爱问、也最容易写错的一块。直接呼应第 1 章那句锚点：**LLM 之于你的应用 = 不可靠 IP 包之于 TCP**。provider 不保证可用，**可用性是你在网关这一层用 fallback 造出来的**。

### 1. 第一刀：不是所有错误都该 fallback

新手写 fallback 的典型错误：`try: 主模型 except: 备用模型`，**把所有异常一视同仁**。这是错的，而且危险。错误必须分类——这张表是本节的灵魂（错误码以 Anthropic 为锚，各家语义类似）：

| HTTP | 错误类型 | 该重试/fallback 吗 | 为什么 |
|---|---|---|---|
| 400 | invalid_request_error | **否** | 你的请求本身错了，换 provider 还是错（比如传了 Opus 不支持的 `temperature`） |
| 401 | authentication_error | **否** | key 错了，重试无意义，会无限循环 |
| 403 | permission_error | **否** | 没权限，换不出来 |
| 404 | not_found_error | **否** | 模型 ID 拼错了（如加了日期后缀），fallback 救不了 bug |
| 413 | request_too_large | **否** | 请求太大，得截断/分块，不是换 provider 的事 |
| 429 | rate_limit_error | **是** | 限流，退避重试或切备用 provider |
| 500 | api_error | **是** | provider 内部错误，可重试 |
| 529 | overloaded_error | **是** | provider 过载，退避重试或切备用 |

> 面试官想区分什么：你是否理解"**4xx（除 429）是你的错，5xx 和 429 是 provider 的错**"。把 400/401 也拿去 fallback，是在用"换 provider"治"代码 bug"——备用 provider 同样会 400，你只是把一个明确报错变成了**两次报错 + 一次无谓的钱**。能说出"fallback 只对 retryable 错误生效，non-retryable 要快速失败并暴露给上游"，直接甩开一片人。

### 2. 第二刀：fallback 的两个降级方向

provider 挂了，往哪降？两个正交的方向，生产里常组合用：

- **横向降级（换 provider，同档位）**：Anthropic Opus 挂 → 切 OpenAI GPT-5。质量基本持平，但要求你**两家都接了、prompt 都适配了**。这是国内"一主一备"的核心（见第七节）。
- **纵向降级（降档，同 provider 或跨 provider）**：Opus 挂 → 退到 Sonnet 甚至 Haiku。质量下降，但**至少有响应**。关键：**降级响应必须被标注**，让上游知道"这是降级结果"，避免把 Haiku 的答案当 Opus 的质量用。

更彻底的降级方向（当所有 LLM 都不可用时）：

- **退到缓存**：返回语义相近的历史答案（见第五节）。
- **退到模板/规则**：返回一个"系统繁忙，请稍后"或规则兜底答案，**绝不能让用户看到 500 堆栈**。

### 3. 第三刀：SDK 自带重试 vs 网关 fallback，别重复造轮子也别漏

这是个非常实际、面试也爱挖的细节。按 2026 初，Anthropic 官方 SDK **自带自动重试**：默认对 408 / 409 / 429 / ≥500 做指数退避，默认 `max_retries=2`，并且会读 `retry-after` 响应头决定等多久。

所以网关的 fallback 层要想清楚分工：

- **SDK 层（provider 内重试）**：处理"同一个 provider 的瞬时抖动"——429/5xx 退避重试几次。这是**同 provider 内的韧性**。
- **网关层（跨 provider fallback）**：处理"这个 provider 持续不可用"——SDK 重试都耗尽了还失败，才切到**另一个 provider**。这是**跨 provider 的韧性**。

> 坑点提示：两层都设重试时小心**重试风暴**。如果 SDK 内重试 2 次、网关又对 2 个备用各重试 2 次，一个请求最坏会放大成 6+ 次后端调用——provider 过载时（529），你的重试反而在**给火上浇油**。生产做法：网关层 fallback 切换要**带熔断（circuit breaker）**，某 provider 连续失败 N 次就短路一段时间，别傻乎乎一直重试一个已经躺平的下游。这跟你做微服务时的熔断是同一套思路。

> 面试加分点：被问"你的 fallback 怎么设计"，能区分"SDK 内重试（同 provider 瞬时抖动）"和"网关跨 provider fallback（provider 持续不可用）"两层，并主动提到"两层叠加要防重试风暴、要上熔断"——这说明你既懂 SDK 又懂分布式韧性，是 P7+ 的信号。

### 4. 第四刀：流式（streaming）把 fallback 的"透明"撕开一道口子

前面所有 fallback 的讨论，都默认了一个隐藏前提：**请求是"发出去—等完整响应—返回"的一次性事务**，所以失败了可以静默切备用，业务无感。但聊天类场景**默认是流式的**——首 token（TTFT）尽快吐给用户，后续 token 逐个 SSE 推送（这正是 Anthropic SDK 对长输出/高 `max_tokens` 默认推荐 streaming 的原因：不流式容易撞请求超时）。流式一旦进来，**"fallback 对业务透明"这句话就不再无条件成立**，这是面试区分"画过架构图"和"真扛过线上"的一刀。

把一次流式请求劈成两段看，fallback 的可行性完全不同：

| 阶段 | 时间点 | provider 此刻挂了能 fallback 吗 | 为什么 |
|---|---|---|---|
| **首 token 之前**（prefill 阶段失败） | 还没吐任何字节给用户 | **能，且对用户透明** | 用户屏幕上什么都还没有，网关静默切备用重发，用户只感觉"慢了一下" |
| **首 token 之后**（流到一半断了） | 已经吐了半句话给用户 | **不能无缝切** | 你已经把 A 模型的半句话送到用户眼前了，切 B 重发会**从头再吐一遍**，用户看到"重复 + 风格突变"，比直接报错还诡异 |

> 坑点提示：**流式 fallback 的黄金窗口只在首 token 之前。** 实现上，网关要做的是"**延迟提交**"——拿到上游的流之后，**先在网关侧 hold 住第一个 chunk**，确认流真的起来了（拿到 TTFT、没在 prefill 阶段报错）再开始往客户端转发；一旦开始转发，就**放弃这一请求的跨 provider fallback**，转而走"流中断后的降级话术"（比如尾部补一句"（生成中断，请重试）"，或让前端拿着已收到的部分内容发起一次新请求续写）。把切换窗口卡在"已经确认流健康、但还没发给用户"这个缝里，是流式网关最关键的工程判断。

> 面试加分点：被问"fallback 在流式下还成立吗"，平庸答案没想到流式会破坏透明性。高手要点出"**fallback 的透明性只在首 token 之前成立，流到一半挂了无法无缝切**"，并给出"网关延迟提交首 chunk 把切换窗口卡在 TTFT 之前"的实现，再补一句"流到一半断了只能走降级话术或前端续写，不能假装无事发生"——这说明你真在流式链路上做过 fallback，不是纸上谈兵。

流式还顺手破坏了另外两件网关核心能力，必须同时处理：

- **成本计量被推后**：非流式响应里 `usage` 一次性给全（input/output token 都有）；**流式下输出 token 数要等流结束（终态事件）才拿得到**。所以网关的成本归因不能在"请求发出"那一刻记账，必须挂在**流正常或异常结束的回调**上累加——这也是为什么前面限流要分"预扣 + 对账"两步（见第六节），流式让"对账"这一步从可选变成必须。
- **限流的实时性变差**：一个流式大请求可能吐 1 分钟，期间它占的 OTPM 配额是**逐渐**消耗的，不是发出时一次性占满。精细的网关要在流进行中**动态累加已消耗 token**，而不是等结束才扣——否则一个慢慢吐的大请求会让你的配额账本在它结束前一直失真。

> 工程推论：
> 1. **流式让"fallback 对业务透明"只在首 token 之前成立**——首 token 一出，跨 provider 无缝切换的窗口就关了，剩下只能降级话术或前端续写。
> 2. **流式网关要"延迟提交"首 chunk**：确认流健康再开始转发，把唯一的 fallback 窗口卡在 TTFT 之前。
> 3. **流式把成本计量和限流对账从"请求发出时"推到"流结束时"**——网关记账必须挂在流的终态回调上，否则成本和配额账本都会失真。

### 5. Fallback 触发分类表（背下来）

| 触发信号 | 归属层 | 动作 |
|---|---|---|
| 429 / 5xx 瞬时 | SDK 内重试 | 指数退避，重试 2 次 |
| SDK 重试耗尽仍失败 | 网关 fallback | 切备用 provider |
| 备用也连续失败 | 网关熔断 | 短路该 provider，走纵向降级 |
| 所有 LLM 不可用 | 网关终极降级 | 退缓存 / 退模板，标注降级 |
| 400/401/403/404/413 | 不 fallback | 快速失败，把错误暴露给上游 |

> 工程推论：
> 1. **可用性是网关造出来的，不是 provider 给的**——fallback 是把"单 provider 可用性"提升到"多 provider 联合可用性"的唯一手段。
> 2. **fallback 必须按错误类型分级**，对 non-retryable 错误 fallback 等于用换 provider 掩盖代码 bug。
> 3. **SDK 重试和网关 fallback 是两层不同粒度的韧性**，叠加时必须防重试风暴、上熔断，否则在 provider 过载时雪上加霜。

## 五、语义缓存：把"问过的问题"变成 0.1× 甚至 0× 成本

缓存是网关里最肥的成本杠杆。但要分清两种缓存，它们机制完全不同。

### 1. 两种缓存，别混

| 缓存类型 | 命中条件 | 命中后成本 | 谁实现 | 风险 |
|---|---|---|---|---|
| **prompt caching**（前缀缓存） | 前缀**逐字节相同** | 命中读 ≈ 0.1× 输入价 | provider 侧（你打标记） | 任何字节变化击穿（见第 2 章） |
| **语义缓存**（semantic cache） | 语义**相似**（向量近邻） | **0×**（不调 LLM） | 网关侧（你自己建） | 假命中：相似 ≠ 等价 |

**prompt caching** 是第 2 章讲透的物理机制——前缀匹配、命中即约 0.1× 输入价、写入约 1.25×（5 分钟 TTL）/2×（1 小时 TTL）、任何字节变化（system prompt 里嵌时间戳/UUID）静默击穿。网关层对它的责任是：**把稳定前缀（system prompt、工具定义）按命中率最优的方式打标记，并用 `usage.cache_read_input_tokens` 验证命中**——这是 provider 侧能力，本章不重复，重点看语义缓存。

### 2. 语义缓存：原理与最大的坑

语义缓存是**网关自己造**的一层：把每个 query 做 embedding（见第 7 章）存进向量库，新 query 来先查近邻，**相似度超阈值就直接返回历史答案，不调主 LLM**。命中后**不付主模型那次推理的钱、不等那次生成的时间**——比 prompt caching 的 0.1× 还狠，因为它压根不发那个昂贵请求。

> 新手第一误区：把语义缓存当成"0 成本 0 延迟"。**不对。** 每一次查缓存，都要先把进来的 query 做一次 **embedding 调用**（一次小模型 API 请求，有它自己的钱和延迟）再去向量库查近邻。所以准确的账是：**命中 = 省下主模型那次贵推理，但仍付了一次 embedding 的小钱 + 一次向量检索的延迟；未命中 = embedding 的钱白花了，主模型该调还得调。** 推论很硬：**语义缓存的经济性强依赖命中率。** 命中率高（比如 FAQ、高度重复的客服问题），embedding 那点开销摊薄到忽略不计，净省一大笔；命中率低（query 高度长尾、个性化），你在**每个请求**上都白加了一次 embedding 调用和一段延迟，**主模型一次没省**——这时语义缓存是净负收益。所以上语义缓存前必须先用真实流量估命中率（又是 know 那条"N=1 不算数，要真实分布"），别拍脑袋上。embedding 本身也可以缓存（同一 query 的向量算一次存下来），但跨 query 的首次 embedding 省不掉。

但它还有一个 prompt caching 没有的致命风险：**假命中（false hit）**。前缀缓存是逐字节比对，命中即"完全相同"，不会错；语义缓存是"相似度判断"，而**相似不等于等价**。

| 场景 | 向量相似度 | 该命中吗 | 后果 |
|---|---|---|---|
| "退款政策是什么" vs "怎么申请退款" | 高 | 也许 | 可能答非所问 |
| "北京今天天气" vs "上海今天天气" | **很高** | **绝对不该** | 城市错了，答案全错 |
| "1+1 等于几" vs "1+2 等于几" | 很高 | 绝对不该 | 数值错 |

> 坑点提示：语义缓存最危险的就是**"高相似度但关键实体不同"**——"北京天气"和"上海天气"在向量空间里离得很近，但答案完全不能复用。阈值定高了命中率低（省不了钱），定低了假命中暴增（答错坑用户）。**没有一个全局阈值能同时满足所有 query 类型。** 生产里的缓解：(1) 阈值按 query 类型分档；(2) 对带具体实体（城市、日期、数字、用户 ID）的 query **降低命中倾向甚至禁用语义缓存**；(3) 任何会因时间/用户/上下文变化的 query 不进语义缓存。

> 面试加分点：被问"语义缓存怎么做、有什么风险"，平庸答案只说"embedding + 向量近邻 + 阈值"。高手必须点出**假命中风险**，并给出"按实体敏感度区分能不能缓存"的工程缓解——这正是 know 里"带语义负载的字段，要先问用户读到这个会理解成什么"的同构问题：用户问"上海天气"读到"北京天气"的缓存答案，会以为系统在胡说。

### 3. 缓存的写入与失效

语义缓存还要解决**写什么、什么时候失效**：

- **写入时机**：LLM 答完且通过质量校验（别把一个幻觉答案缓存起来反复发）。
- **失效策略**：知识类答案要随知识库更新失效（呼应 RAG 的可更新真相源）；时效类答案设短 TTL；用户个性化答案带用户维度 key 或干脆不缓存。

> 工程推论：
> 1. **prompt caching（前缀、provider 侧、0.1×、逐字节）和语义缓存（相似、网关侧、0×、有假命中）是两套东西**，混为一谈会答错面试。
> 2. **语义缓存的命中是概率事件**，核心风险是假命中，缓解靠"按实体敏感度区分可缓存性"而非调一个全局阈值。
> 3. **缓存的失效策略和它的命中率同样重要**——缓存了过期/错误答案，比不缓存更糟。

## 六、多租户配额与限流：一个租户烧穿，不能拖垮所有人

网关是 SaaS 化 AI 服务的**计费与隔离边界**。这块直接对应你做后端时的"租户隔离"直觉。

### 1. 为什么 LLM 的限流比普通 API 难

普通 API 限流按 **QPS / RPS**（请求数）即可。LLM 不行，因为：

- **成本不在请求数，在 token 数**。一个请求可能 100 token，也可能 100K token，成本差千倍。所以 LLM 限流的主维度是 **TPM（tokens per minute）**，不是 RPM（requests per minute）。
- **provider 自己的配额也是 token 维度**（ITPM 输入、OTPM 输出分开算）。你的网关配额体系要和上游 provider 的配额维度对齐。
- **输入输出成本不对称**（输出 5×），精细的配额要分别核算输入/输出 token。

### 2. 多租户配额的三层结构

| 层级 | 限什么 | 防什么 |
|---|---|---|
| **全局** | 总 TPM / 总 $/天 | provider 配额被打穿、月度预算超支 |
| **租户** | 每租户 TPM / 每租户 $/月 | 单租户烧穿影响其他人（事故四） |
| **特性/链路** | 某条 agent 链路的步数/token 上限 | agent 不收敛烧钱（呼应第 12-16 章） |

实现上常用**令牌桶（token bucket）按 token 计量**：不是每请求扣一个令牌，而是**按 token 数扣**。但这里藏着一个"NL 翻译成 code 最容易漏的格"——**发请求之前你根本不知道输出有多长**，没法按真实 token 数扣，只能分两步走：

1. **预扣（reserve）**：请求发出**前**，先按"输入 token（可 count_tokens 精确算）+ 一个输出上限的预估值"从桶里**预扣**。预扣不够（桶空了）就当场挡掉，根本不发请求。这一步保证"先占坑再花钱"，防的是并发请求一起冲进来把配额冲爆。
2. **对账（reconcile）**：请求**回来后**，用响应里 `usage` 的**真实** input/output token，和当初的预扣值做差，**多退少补**回桶里。预估保守了就退还冗余配额，预估乐观了（输出比预想长）就再扣补足。

> 坑点提示：**只做预扣不对账，配额会系统性失真。** 你按"输出上限"预扣，但多数请求实际输出远小于上限——不对账退还，桶会被这些"虚占"的配额长期占着，明明还有余量却一直挡正常请求。反过来，**只看实际不预扣**（等响应回来才扣），高并发下一堆请求会在"还没扣"的窗口里一起冲进 provider，把上游配额打爆——这正是事故四"单租户烧穿"的技术成因。两步缺一不可。流式场景下"对账"这一步还从可选变成必须，因为输出 token 只有流结束才拿得到（见第四节）。

预估和对账都依赖准确的 token 数：预估用 count_tokens（注意：Anthropic 没有公开 tokenizer，要调 `count_tokens` API；用 tiktoken 估 Claude 会系统性偏差 15-20%，见第 1/2 章），对账用响应里的 `usage`。

### 3. 限流命中后怎么办

限流不是简单 reject。分级处理：

- **硬限流**（安全/防滥用）：直接 429，让客户端退避。
- **软限流 / 排队**：非实时任务（批量摘要）可以排队，甚至降级到 **Batch API**——按 2026 初，Anthropic Batch API 异步、**50% 价格**、单批最多 10 万请求/256MB、多数 1 小时内完成（上限 24 小时）。把"能等"的流量从实时通道挪到 Batch，既省一半钱又给实时通道让出配额。
- **降档**：付费档位高的优先用 Frontier，配额紧张时低档位降到 Fast 模型。

> 面试加分点：被问"LLM 服务怎么做限流"，只答"加个 rate limiter"是不及格的。要点出"**LLM 限流主维度是 TPM 不是 RPM，因为成本在 token 不在请求数**"，并能提到"输入输出不对称要分开算""能等的流量降级到 Batch API 省一半钱"——这三点直接体现你理解 LLM 成本结构，不是套用普通 API 经验。

> 工程推论：
> 1. **LLM 限流按 token（TPM）不按请求（RPM）**——成本与 token 强相关，与请求数弱相关。
> 2. **token 限流必须"预扣 + 对账"两步**：发请求前按输入+输出上限预扣防并发冲爆，响应后按真实 `usage` 多退少补防配额虚占。只做一步必失真。
> 3. **配额要分层（全局/租户/链路）**，对应"防 provider 打穿/防租户互相拖累/防 agent 烧钱"三类风险。
> 4. **限流命中不等于 reject**，能等的流量降级到 Batch API（50% 价、异步）是省钱和让配额的双赢手段。

## 七、自建 vs 开源方案：以及国内"一主一备 + 开源兜底"为何是刚需

### 1. 主流开源/商业网关（按 2026 初，名单会变，学方法论）

| 方案 | 形态 | 适合 |
|---|---|---|
| **LiteLLM** | 开源，Python，统一 100+ provider 为 OpenAI 格式 + Proxy Server | 想快速统一多 provider、自己掌控部署 |
| **Portkey** | 商业 + 开源 SDK，网关 + 可观测 + guardrails | 想要开箱即用的观测/治理 |
| **Kong AI Gateway** | 在成熟 API 网关上加 AI 插件 | 已用 Kong、想复用现有网关栈 |
| **云厂商 AI 网关** | 各云自带 | 已深度绑定某云 |

LiteLLM 是开源自建里的事实标准：它把各家 provider 调用**统一成 OpenAI 的 `chat/completions` 格式**，并提供一个 Proxy Server 让你集中配路由、fallback、预算、缓存。

### 2. 自建 vs 用开源：决策对照表

| 维度 | 用开源（LiteLLM 等） | 自建 |
|---|---|---|
| **上手速度** | 快，几天接好多 provider | 慢，从零造 |
| **provider 覆盖** | 广，社区维护 | 你接一家是一家 |
| **定制深度** | 受框架抽象限制 | 完全可控 |
| **专有/国内 provider** | 看社区支持度，可能滞后 | 你想接就接 |
| **治理/合规耦合** | 通用，未必贴合你的内部体系 | 可深度耦合内部计费/审计/合规 |
| **维护负担** | 跟社区升级 | 全在你身上 |

判断规则（可执行）：

- **早期 / 小团队 / 标准 provider**：用 LiteLLM，别自建。把精力花在业务，网关用成熟件。
- **强治理需求 / 深度耦合内部体系 / 大量专有 provider**：在开源件外**包一层自己的薄网关**，或自建。典型是大厂——计费、审计、合规、内部模型平台都要深度集成，纯开源件兜不住。
- **最务实的中间态**：**LiteLLM 做 provider 适配层（解决"接得通"），自己写薄薄一层业务网关做路由策略/配额/成本归因/安全挂载（解决"治得好"）。** 不重复造 provider 适配的轮子，又保住治理的控制权。

> 坑点提示：自建网关最容易低估的是**provider 适配的长尾维护**——每家 SDK 升级、新模型上线、参数变更（比如 Opus 4.7 突然移除 temperature），你都得跟。这正是 LiteLLM 这类件的核心价值：**把"跟 N 家 provider 变更"这件脏活社区化了**。除非你有专人盯，否则自己维护多 provider 适配是个无底洞。

### 3. 国内语境：为什么"一主一备 + 开源兜底"是刚需而非可选

这是国内做 AI 应用绕不开的现实约束，面试国内大厂尤其要答得出。三个理由叠加，把多 provider 从"高可用优化"变成了"业务连续性刚需"：

1. **可用性与限流现实**：单一闭源 provider（无论国内豆包/Kimi/GLM/混元/文心，还是境外）都会有限流、抖动、容量波动。单押一家 = 把业务连续性赌在别人的 SLA 上。
2. **合规与供应稳定性**：模型可用性、备案、出海/数据出境政策都可能变化。**"一主一备"是对供应侧不确定性的对冲**——主用 A，A 出问题（限流/价格/政策）能立刻切 B，不至于业务停摆。
3. **成本与议价**：手握可随时切换的多 provider，你对任何一家都有议价能力和退路；锁死一家，价格和条款由别人定。

而"**开源兜底**"（自部署 Qwen / DeepSeek 等开放权重模型）是这套架构的**最后一道防线**：当所有闭源 provider 都不可用（极端故障、配额耗尽、政策突变）时，自部署的开源模型是**完全由你掌控、不依赖任何外部 SLA 的底座**。质量可能降档，但**保证有响应**——这正是第四节"纵向降级到终极兜底"在国内语境下的具体落地。

> 面试加分点（国内大厂高频）：被问"为什么要多 provider / 国内为什么强调一主一备"，要把它讲成**三重对冲**——对冲可用性（限流抖动）、对冲供应不确定性（合规/政策/价格）、对冲议价权（不被单一供应商锁定）。再补一句"**开源自部署是不依赖任何外部 SLA 的最后防线**"。注意一个 know 校准点：**差异化和兜底要建立在"平台做不到"上，而不是"平台不会做"上**——多 provider 的价值在于"任何单一 provider 都给不了你跨家的连续性和议价权"，这是结构性的，不会因为某家 provider 变好而消失。

> 工程推论：
> 1. **网关选型先问"治理深度"**——标准需求用 LiteLLM，强治理/多专有 provider 才考虑自建或混合。
> 2. **最优解常是"开源做适配 + 自建薄层做治理"**，既不造 provider 适配的轮子，又保住路由/配额/成本/安全的控制权。
> 3. **国内"一主一备 + 开源兜底"是业务连续性刚需**，本质是对可用性、供应不确定性、议价权的三重对冲，开源自部署是不依赖外部 SLA 的最后防线。

## 八、本章小结与心法

把这一章压缩成一句话：**网关是把不可靠、易变、分散的 LLM provider，收口成一条可治理、可替换、可降级的统一出口的基础设施层。**

四句可记住的心法：

1. **收口先于治理**——所有限流、成本、安全、观测能力的前提，是流量先走同一个网关。
2. **provider 进配置不进代码**——换模型该是改一行 YAML 热加载，不是发版。
3. **可用性是你造的不是 provider 给的**——fallback 按错误类型分级，SDK 重试管瞬时抖动、网关 fallback 管 provider 不可用、熔断防重试风暴；但**流式下 fallback 的透明性只在首 token 之前成立**，吐了一半再挂只能降级话术。
4. **省钱靠分层缓存 + 按能力路由 + 能等就 Batch**，但每一招都有它的暗坑（语义缓存的假命中 + 每次查的 embedding 开销、级联付两次、降级要标注、限流要预扣再对账）。

> 全章心法：**LLM Gateway 之于 AI 系统 = API Gateway / Service Mesh 之于微服务。** 你做后端时坚持"业务不该知道下游是谁、在哪、挂了怎么办"，把同一条纪律搬到 LLM provider 上，你就拿到了这一章的全部。

## 章末五件套

### 一、高频面试题（10 道，附答题框架）

**Q1. 为什么需要 LLM Gateway？它和普通 API Gateway 有什么区别？**
- 框架：先答六大职责（统一接入/fallback/模型抽象/成本归因/配额限流/可观测），强调"收口先于治理"。区别要点出 LLM 特有的三点：(1) 限流主维度是 **TPM 不是 RPM**（成本在 token）；(2) 路由可**按能力/成本**选模型（普通网关不挑下游实现）；(3) **语义缓存**这种"相似即命中"的能力普通网关没有。
- 错误答案陷阱：只说"统一入口、负载均衡"——那是普通网关，没答出 LLM 特性。

**Q2. fallback 是不是所有错误都该触发？怎么设计？**
- 框架：**否**。按错误类型分级——**4xx（除 429）是你的错，不 fallback**（400/401/403/404/413，换 provider 还是错）；**429/5xx 是 provider 的错，可重试/fallback**。再分两层：SDK 内重试（同 provider 瞬时抖动）+ 网关跨 provider fallback（provider 持续不可用），叠加要防重试风暴、上熔断。
- **进阶追问（流式下还成立吗）**：fallback 的"对业务透明"**只在首 token 之前成立**。流到一半 provider 挂了，你已经把半句话吐给用户了，切备用会从头重吐（重复 + 风格突变），比报错还诡异。实现上网关要"延迟提交"——确认流健康（拿到 TTFT、没在 prefill 报错）再开始转发，把唯一的 fallback 窗口卡在首 token 之前；流中断只能走降级话术或前端续写。能答出这一层是 P7+ 信号。
- 陷阱：`try/except` 一把全 catch 去 fallback——把 400 拿去 fallback 等于用换 provider 掩盖代码 bug，备用也会 400；以及默认"fallback 永远透明"，没意识到流式会撕开这个前提。

**Q3. 模型抽象层抽象到什么粒度合适？厚抽象有什么坑？**
- 框架：薄抽象（只统一 transport，保留原生参数）vs 厚抽象（统一参数 schema）的 trade-off。厚抽象必须留**逃生口**（`provider_params` 透传），因为各家专有能力不一致——举例 **Opus 4.8 移除了 temperature，传了 400**，而 GPT 还有，说明"取并集硬塞"会崩。
- 陷阱：说"包一层统一接口就行"，答不出"参数 schema 代际迁移"（采样超参→思考预算）这个真实约束。

**Q4. 语义缓存怎么实现？最大的风险是什么？**
- 框架：embedding query → 向量近邻 → 相似度超阈值返回历史答案，命中**省下主模型那次贵推理**。最大风险是**假命中**——"相似 ≠ 等价"，"北京天气"和"上海天气"相似度很高但答案不能复用。缓解：按 query 类型分档阈值、带具体实体的 query 降低命中倾向/禁用、时效/个性化 query 不缓存。
- 陷阱：(1) 只说"embedding + 阈值"，没点假命中——这是面试官最想听的那一刀；(2) 说"命中就是 0 成本 0 延迟"——错，**每次查缓存都要先做一次 embedding 调用**，未命中时这笔 embedding 钱纯属白花，所以**语义缓存的经济性取决于命中率**，命中率低反而净亏。能主动点出这一点，说明你算过账不是背概念。

**Q5. prompt caching 和语义缓存有什么区别？**
- 框架：前者前缀**逐字节**匹配、provider 侧、命中约 **0.1×** 输入价、任何字节变化击穿（system prompt 嵌时间戳/UUID 静默击穿）；后者语义**相似**、网关侧自建、命中**省下主模型那次推理**（但每次查仍付一次 embedding）、有**假命中**风险。两套机制，别混。
- 陷阱：以为是一回事，或以为语义缓存也"逐字节"，或把语义缓存当"命中即 0 成本"（漏了每次查缓存的 embedding 开销）。

**Q6. LLM 服务的限流为什么不能照搬普通 API？**
- 框架：普通 API 按 RPS。LLM 成本在 **token 不在请求数**（一个请求 100 token 还是 100K token 成本差千倍），所以主维度是 **TPM**；且**输入输出成本不对称（输出 5×）**要分开算；还要和上游 provider 的 ITPM/OTPM 配额对齐。配额分层（全局/租户/链路）。
- **进阶追问（怎么扣）**：token 桶要"**预扣 + 对账**"两步——发请求**前**按输入+输出上限**预扣**（防并发一起冲爆 provider，这正是单租户烧穿的技术成因），响应**后**按真实 `usage` **多退少补**（防输出上限预扣造成的配额虚占）。只做预扣会虚占、只看实际会冲爆，缺一不可；流式下"对账"还是必须的，因为输出 token 要等流结束才有。
- 陷阱："加个 rate limiter 按 QPS 限"——没理解 LLM 成本结构；或只说"按 token 扣"答不出"发请求前根本不知道输出多长，所以要预扣再对账"。

**Q7. 按能力路由怎么做？"用哪个模型判断该用哪个模型"这个鸡生蛋问题怎么解？**
- 框架：三档——规则路由（零成本但粗）、小模型分诊（加一次便宜调用换可能省的贵调用）、级联（便宜模型先答+质量检查+不行再升级，最省但可能付两次）。强调必须用**真实 query 分布回测**，级联在"难题占多数"时反而更贵。
- 陷阱：只说"难的用大模型简单用小模型"，答不出"难度判断本身要花钱"和"级联的最坏情况"。

**Q8. SDK 自带重试了，网关还要 fallback 吗？两者怎么分工？**
- 框架：要。SDK 重试（Anthropic 默认 408/409/429/5xx 指数退避、max_retries=2、读 retry-after）管**同一 provider 的瞬时抖动**；网关 fallback 管**该 provider 持续不可用，切到另一家**。叠加要防**重试风暴**（最坏放大成 6+ 次），网关层切换要带**熔断**。
- 陷阱：以为有了 SDK 重试就不用网关 fallback——那只解决了同 provider 韧性，没解决跨 provider。

**Q9. 自建网关还是用 LiteLLM？怎么决策？**
- 框架：标准 provider/小团队用 LiteLLM（别造 provider 适配轮子）；强治理/多专有 provider/深度耦合内部计费审计合规才自建。**最优常是混合**——LiteLLM 做适配层 + 自建薄层做路由/配额/成本/安全。点出自建的长尾坑：**provider 适配的持续维护**（每次 SDK 升级、新模型、参数变更都要跟）。
- 陷阱：无脑"自建可控"或无脑"用开源省事"，不给 trade-off。

**Q10.（国内）为什么国内强调"一主一备 + 开源兜底"？**
- 框架：三重对冲——对冲**可用性**（限流抖动）、对冲**供应不确定性**（合规/政策/价格变化）、对冲**议价权**（不被单一供应商锁定）。开源自部署（Qwen/DeepSeek）是**不依赖任何外部 SLA 的最后防线**，质量降档但保证有响应。
- 陷阱：只说"高可用"，没答出"供应/合规/议价"的结构性对冲，也没提开源兜底这道终极防线。

### 二、实战项目增量（P4：在 P2/P3 前加网关层）

在你已有的 P2（RAG）/ P3（research agent）**前面**插入一层网关，让所有 LLM 调用穿过它。

**验收标准（缺一不算过）：**
1. **模型抽象**：P2/P3 的业务代码里**不再出现任何硬编码 provider/model 字符串**，全部改为调用 `gateway.chat("<逻辑名>", messages)`；逻辑名→物理 provider/model 的映射在配置文件里，**改配置热加载、不重启服务**即可切换主模型。
2. **fallback + 错误分级**：实现主模型失败时切备用。**必须按错误类型分级**——构造一个故意触发 400（如传非法参数）的请求，断言它**不触发 fallback** 而是快速失败；构造一个模拟 429/5xx 的场景，断言它**触发重试/fallback**。
3. **成本可观测**：每个请求记录 model / input_tokens / output_tokens / 估算成本 / 延迟，并能**按 tenant 或 feature 维度聚合**出"这条链路这段时间花了多少钱"。
4. **基础配额**：实现按 **token（TPM）** 的租户级限流（不是按 RPM），构造一个单租户高频调用场景，断言它被网关挡住而**不影响其他租户**。
5. **跑通对比**：分别用"直连"和"走网关"跑同一组 ≥20 条 query，证明走网关后**业务代码零 provider 知识**、**主模型可热切换**、**有成本报表**。

**面试话术目标**：做完能说出"我把 P2/P3 的 provider 依赖从代码降级成配置，主模型挂了能自动切备用且对 400 不做无谓 fallback，并能按业务线出成本报表"——带数字、带因果、带判断。

### 三、设计题（开放，考权衡）

> 你负责给公司搭一个内部 LLM 网关，要同时服务三类业务：(a) 面向 C 端、延迟敏感的客服 bot；(b) 后台批量文档摘要，不在意延迟；(c) 内部 research agent，质量优先、可慢。要求设计这套网关的**路由 + 缓存 + 限流 + fallback** 策略，并说明三类业务各自的配置差异与取舍。

要点（评分看是否做了**差异化**而非一刀切）：
- 客服 bot：实时通道，prompt caching 重度用，语义缓存**谨慎用**（带用户/订单实体易假命中），fallback 走横向降级保质量，限流硬。
- 批量摘要：**降级到 Batch API**（50% 价、异步）、不占实时配额，可接受纵向降级到 Haiku，语义缓存可激进。
- research agent：质量优先，按能力路由到 Frontier，要有**步数/token 预算上限**防不收敛，fallback 宁可慢不可错。
- 共性：成本归因到三条业务线、配额分层。

### 四、系统设计题（含量级估算）

> 设计一个支撑全公司的 LLM 网关，目标 **峰值 5000 QPS**，平均每请求 **输入 2K token、输出 500 token**。估算：(1) 峰值 TPM 需求；(2) 在不做缓存 vs 命中率 40% 的语义缓存下，按 Opus 4.8（输入 $5 / 输出 $25 每百万 token）估算每小时成本；(3) 给出网关本身的部署形态与单点故障预案。

参考量级（重在过程，数字按口径可浮动）：
- **峰值 token 吞吐**：5000 QPS ×(2000+500) token = 1.25×10⁷ token/s ≈ **7.5×10⁸ TPM**。这远超任何单一 provider 给单 org 的配额——**结论：必须多 provider 分摊 + 配额池管理**，这道题的"陷阱"就是逼你意识到单 provider 接不住。
- **每小时成本（无缓存）**：每秒输入 5000×2000=10⁷ token → 输入 $50/s；输出 5000×500=2.5×10⁶ token →$62.5/s；合计 ≈ **$112.5/s ≈ $40.5 万/小时**。（量级感知：Frontier 模型全量裸跑贵到离谱，**省钱不是优化是生存**。）
- **命中率 40% 语义缓存后**：约 60% 流量真打 LLM → 每小时 ≈ **$24.3 万**。再叠加按能力路由把简单请求降到 Haiku、把批量挪到 Batch，可再砍一大块。
- **部署与单点**：网关本身**无状态横向扩展 + LB**；配额/缓存状态外置（Redis/向量库）；网关挂了不能让 LLM 流量全断——要多副本、健康检查；provider 侧靠 fallback + 熔断；语义缓存/向量库也要有自己的降级（查不到就直接打 LLM，别因为缓存层挂了阻塞主流程）。

### 五、代码题（带 TODO 骨架 + 测试要求 + 暗坑提示）

实现一个最小 LLM 网关核心：**带错误分级的 fallback + 按 token 的租户限流 + 成本归因**。语言不限，下面给 Python 骨架。

```python
from dataclasses import dataclass, field
from enum import Enum

class ErrorClass(Enum):
    RETRYABLE = "retryable"          # 429 / 5xx / 529 —— 可重试 / 可 fallback
    NON_RETRYABLE = "non_retryable"  # 400 / 401 / 403 / 404 / 413 —— 快速失败

@dataclass
class ProviderCall:
    """对一个具体 provider+model 的一次调用封装。"""
    provider: str
    model: str
    # call(messages) -> (text, input_tokens, output_tokens)；可能抛带 http_status 的异常

@dataclass
class RouteConfig:
```

**暗坑提示：**
1. **429 是 RETRYABLE。** 它是 4xx 但语义是"限流，退避后能成功"，归错成 NON_RETRYABLE 会让你在该重试的时候放弃。
2. **限流扣 token 不扣请求。** LLM 成本在 token，按 RPM 写的限流挡不住"一个超大请求烧穿配额"。
3. **别用 tiktoken 估 Claude token。** 它是 OpenAI 的分词器，估 Claude 系统性偏差 15-20%，要调 provider 的 count_tokens。
4. **成本必须输入、输出分开按各自单价算。** 输出价 ≈ 输入价 5 倍（Opus 4.8 是 $5 vs $25），用一个混合单价乘总 token 会把成本算错一大截。
5. **400/401/403/404/413 绝不 fallback。** 这些是你的请求/配置错，换 provider 同样会错——fallback 只会把"一次明确报错"变成"N 次报错 + N 次无谓的钱"。fallback 只对 RETRYABLE 生效。
6. **成本归因不带 tenant 等于白记。** 网关存在的一大理由就是按业务维度归因成本，不带 tenant/feature 标签，你又回到了"翻 provider 账单也不知道哪条业务花的"的原点（事故三）。
