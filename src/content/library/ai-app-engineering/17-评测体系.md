> 这章排在阶段四 LLMOps 的开头，因为它是把"我感觉这版更好"变成"我有数据证明这版更好"的那把尺子——**没有 eval，你后面所有的 tracing、网关、安全治理都是在一个无法测量的系统上做工程，等于在沙子上盖楼**。第 11 章给 P2 那套 RAG 装了仪表盘（RAG 专用评测）；这一章把评测从 RAG 这个垂直场景**拔高成一套通用方法论**，覆盖你 P2/P3 乃至以后任何 LLM 系统：怎么像 TDD 一样让 eval 成为开发的第一公民、四类评测各自的边界、LLM-as-judge 怎么设计才不骗你、离线回归 + 在线 A/B + 影子流量怎么咬合、指标体系和失败分类怎么搭，以及最绕的元问题——**谁来评测评测者**。
>
> 替你解决的问题：把"改 prompt / 换模型 / 调 agent 后到底变好还是变坏"这个目前只能靠手测几个 case 拍脑袋的判断，变成一条**每次变更自动跑、出数字、能回归、能阻断上线**的流水线。建立一个判断——**eval 之于 AI 系统 = error budget 之于 SRE**：没有量化标准，你根本不知道这次改动是进步还是退步，只能靠感觉，而感觉在生产里等于裸奔。
>
> 读法提示：这章每个机制后面紧跟**工程推论**（编号列出，每条都能独立成一道面试题）。第一遍至少把"四类评测对照表""judge 五大偏见缓解表""离线/影子/A-B 三档对照表""指标分层表""失败分类法"这五张表啃下来。代码题对应主线项目 **P4 起步**——为 P2/P3 建评测流水线，每次变更跑回归。
>
> 时效标注：judge 模型名 / Promptfoo、Braintrust 等框架的具体 API 按 **2026 初**写，名单和接口会过期，**重点学方法论**；而"eval 是第一公民""judge 必须异源 + 校准""离线兜底 + 在线发现盲区"这些是**评测规律**，讲透到骨子里，不会变。
>
> 交叉引用：RAG 这一垂直场景的检索/生成分层评测在**第 11 章**展开（recall、faithfulness、context relevance 那套），本章不重复，只讲通用方法论；judge 偏爱长答案、自信语气的根子，连着**第 1 章**讲的"模型是概率延续器、置信度不可信"——judge 也是个 LLM，它会犯同一类病（第 1 章谄媚）；评测产生的数据怎么接进 tracing、怎么在线上持续观测，在**第 18 章**可观测性展开。

## 一、背景：为什么 eval 是 LLM 工程的第一公民

### 1. 问题的根源：LLM 系统没有"编译器"

后端工程你有一整套确定性的反馈回路：类型系统在编译期拦你、单测断言 `assertEqual(f(2), 4)`、CI 红了就不许合。这套回路的前提是——**同样的输入永远给同样的输出**。

LLM 把这个前提砸了。它是概率采样（第 1 章），同一个 prompt 跑两遍，措辞都不一样；你改了一行 system prompt，重跑发现答案变了——这是改动带来的**真实改善**，还是采样噪声（noise）？你**肉眼分不出来**。更要命的是没有"正确答案"的硬定义：摘要"好不好"、回答"够不够 helpful"、agent"该不该在这步调工具"，这些都不是 `==` 能判的。

于是绝大多数团队的 LLM 开发长这样：写个 prompt，手动测三五个 case，"嗯感觉不错"，上线。然后线上炸了某类 case，回去改 prompt，再手测那几个，"嗯这次对了"，再上线——结果把之前对的 case 改坏了（regression），自己不知道。**这就是没有 eval 的 LLM 开发：一个没有回归测试的、靠手感的、在生产环境碰运气的过程。**

> 新手第一误区：把"我手动测了几个 case 感觉挺好"当成验证。手测有三个致命伤——**样本太小**（你测的 5 个恰好是模型擅长的）、**不可复现**（"感觉变好了"传不给同事、下次改动无法对比）、**会回归而不自知**（修了 A 坏了 B，你只盯着 A）。这三条任意一条，都让你的"验证"在统计意义上等于零。

### 2. 演进：从"人肉看"到 eval-driven development

- **阶段零：人肉 demo。** 跑几个问题人眼看好坏。只适合最早期确认"能跑通"，样本小、不可复现、归因靠猜。
- **阶段一：攒一个固定测试集，手动跑。** 把 case 固定下来，每次改完都过一遍。比人肉强在"可复现 + 能回归"，但人工判分慢、不一致、跟不上迭代。
- **阶段二：自动化评测集 + 自动判分。** 测试集 + 自动评分器（规则 / LLM-as-judge），一条命令跑出准确率/失败率报表。这是当前主流。
- **阶段三：eval-driven development（EDD）。** 把 eval 提到和代码同级的**第一公民**地位——**先有 eval，再改 prompt**。改任何东西之前先问"这个改动该用哪条 eval 衡量？现在的 baseline 分数是多少？"。改完自动跑回归，分数没涨（或某个子集掉了）就不许合。这是 TDD 在 LLM 工程里的对应物。
- **阶段四（当前前沿）：离线 + 在线闭环。** 离线 eval 集兜住已知质量，线上真实流量的失败回流持续扩充 eval 集、发现新失效模式。离线防回归，在线找盲区，两者咬合成飞轮。

> 锚点类比（给后端老手）：**eval 之于 LLM 开发 = TDD 之于普通开发**。TDD 的精髓不是"测试"，是"测试先行倒逼你想清楚『正确』的定义"。EDD 同理——逼你在动手改 prompt 之前，先把"什么叫好"用一组可打分的 case 钉死。**写不出 eval，说明你根本没想清楚这个功能的成功标准是什么**，那你改 prompt 就是在瞎调。

### 3. 三个工程推论（本章地基，逐条都是面试题）

#### 推论一：没有 eval 的"优化"是赌博，eval 是 LLM 工程的前置条件不是收尾

绝大多数人把 eval 当成"功能做完了，补一套测试"。错。**eval 是动手优化的前提，不是收尾。** 没有度量，优化就没有方向（你不知道往哪改）、没有反馈（你不知道改对没）、没有刹车（你不知道有没有改坏别的）。先建尺子，再谈手术。

> 面试加分点：被问"你怎么知道你的 prompt 改动是有效的"，初级答案是"我测了几个 case 感觉好了"。高级答案是"我有一个带标注的 eval 集和一个 baseline 分数，改完自动跑回归，看目标指标是否显著上升**且**其它子集没回退；区分『真改善』和『采样噪声』靠的是足够样本量 + 固定随机种子/多次取均值"。能点出"区分改善 vs 噪声"这一层，说明你真做过。

#### 推论二：eval 集是 LLM 项目最值钱、最该早攒的资产

模型会换、prompt 会换、框架会换、agent 编排会换——但"这些输入的正确行为/正确答案是什么"这份标注是**稳定的**。它是你所有迭代的回归基线，是新人接手时唯一能信的"系统该做什么"的事实来源，是你换模型时唯一能证明"新模型没把老能力搞砸"的证据。**代码会过时，好的 eval 集会增值。** 把它当代码一样进版本控制、像测试用例一样维护。

#### 推论三：评测本身也是一个会出错的系统，必须被评测

这是本章的元主题，也是新手最想不到的一层。你拿 LLM-as-judge 给系统打分——但 judge 也是个 LLM，它会谄媚、有偏见、不稳定（见第四节）。你的 eval 集可能漏标、可能全是简单题、可能有歧义。**如果你不评测你的评测器，你的"客观指标"可能从一开始就是假的，而你还在拿它做上线决策。** 谁来评评测者？答案在第六节。

> 心法：**先有尺子，再做手术；尺子本身也要先被校准。** 一个没校准过的 judge 打出来的 0.91，和你拍脑袋猜的 0.91，可信度一样——都是零。

## 二、四类评测：参考答案 / 规则 / LLM-as-judge / 人工

评测的核心动作就一件事：**给定一个（输入，系统输出），判定它好不好。** 区别只在于"谁来判、怎么判"。生产里这四类各有适用边界，**绝不是越高级越好**，而是按"能不能用规则算"分层——能用便宜手段判的，绝不上贵的。

| 评测类型 | 怎么判 | 成本 | 客观性 | 适用场景 | 致命局限 |
|---|---|---|---|---|---|
| **基于参考答案（reference-based）** | 拿输出和"标准答案"比：精确匹配 / 包含 / 语义相似度（embedding）/ F1 | 极低 | 高 | 有唯一/有限正确答案：分类、抽取、闭式问答、SQL 生成 | 开放生成无"标准答案"；字面比对惩罚同义改写 |
| **基于规则（rule-based / assertion）** | 写断言：格式合法（JSON parse 过）、含/不含某关键词、长度区间、正则、调了哪个工具 | 极低 | 高 | 可形式化的硬约束：结构化输出、安全词过滤、引用格式、工具选择正确性 | 只能判"形式"，判不了"语义好不好" |
| **LLM-as-judge** | 让另一个强 LLM 按 rubric 给输出打分/判对错 | 中 | 中（有偏见，需校准） | 语义判断：helpfulness、忠实度、语气、相关性、两个答案谁更好 | judge 自带 LLM 全部毛病（谄媚/偏见/不稳定），未校准则分数不可信 |
| **人工标注（human eval）** | 人来打分/排序 | 极高、慢 | 最高（金标准） | 校准其它三类的基准、最终验收、judge 标准的源头 | 贵、慢、跟不上迭代；标注者之间也会不一致 |

### 1. 基于参考答案：能用就用，最便宜最可靠

如果你的任务有"标准答案"——情感分类该输出 positive、从发票抽取金额该是 ¥1280、文本转 SQL 该等价于某条查询——那就别上 judge，直接比对。

但有个坑：**字面精确匹配（exact match）对生成任务往往过严**。"巴黎是法国首都"和"法国的首都是巴黎"语义一样，exact match 判错。所以参考答案类内部还分几档，按任务松紧选：

- **exact match**：分类标签、枚举值、结构化字段——要求一字不差。
- **包含 / 关键点命中**：标准答案的几个 key point 是否都出现。适合简答。
- **语义相似度**：把输出和参考答案各自 embedding，算 cosine（第 7 章）。适合"意思对就行"的开放短答。坑：相似度阈值要校准，高了漏判对的、低了放过错的。
- **结构化等价**：SQL 生成不比字符串、比"执行结果是否一致"（execution accuracy）；JSON 不比字节、比"解析后字段是否相等"。

> 工程推论：
>
> **推论四：有参考答案的任务，优先用 reference-based + rule-based，把 LLM-as-judge 的预算省给真正需要语义判断的部分。** judge 又贵又慢又有偏见，能用 `==` 或正则解决的，绝不上 judge。这是 eval 成本控制的第一原则——**先问"这条能不能用规则算"，不能再上 judge**。一个成熟的 eval 套件里，规则类断言往往占大头、judge 只占需要语义判断的那一小撮。

### 2. 基于规则：判"形式"的硬约束，又快又确定

很多失败是**形式失败**而非语义失败，这类根本不需要语义判断，写个断言即可：

- 结构化输出该是合法 JSON 且符合 schema → `json.loads` 过不过 + schema 校验。
- 答案不该出现竞品名 / 脏词 / PII → 关键词与正则扫描。
- agent 该在这步调 `search` 工具 → 检查 tool_use 块的 name。
- 答案长度该在 50–200 字 → 长度区间断言。
- 引用格式该是 `[1]` 这种 → 正则。

规则类的价值被严重低估：**它免费、确定、零噪声、可秒级跑全量**，是 eval 套件的地基。先把所有能形式化的硬约束写成断言，剩下判不了的才交给 judge。

> 坑点提示：规则类只能判"形式对不对"，判不了"内容好不好"。一个 JSON 格式完美、字段齐全、但内容全是胡编的答案，规则类全绿。**别把"规则全过"当成"质量合格"**——规则是必要条件不是充分条件。

> 落地（Anthropic API，按 2026 初）：结构化输出别再用模型"自觉吐 JSON"然后祈祷它合法——用 **`output_config.format`** 传 `json_schema` 约束响应格式，工具参数用 **`strict: true`** 做 schema 校验；Python SDK 有 `messages.parse()` 自动按你的 Pydantic 模型校验、返回 `parsed_output`（旧的 `output_format` 顶层参数已废弃）。**把"格式合法"从一条 eval 断言下沉成一个 API 保证**——能在源头用 schema 约束掉的形式失败，就不必再花一条 eval 去抓它。这是"用平台能力消灭一整类失败"的典型，省下的 eval 预算去测真正难的语义。

### 3. 人工标注：贵但不可替代，它是其它三类的"地基真值"

人工最准也最贵，所以**不用它跑全量、跑每次回归**，而是用在三个刀刃上：

1. **校准 judge**：人标几十条金标准子集，拿 judge 在这个子集上的判断和人对齐，量化一致率（agreement）。judge 和人对不上，这个 judge 就不能用（详见第六节）。
2. **最终验收 / 对外汇报**：上线前、给老板演示前，抽样人审兜底。
3. **rubric 的源头**：judge 的评分标准（rubric）从哪来？从人类专家对"什么叫好答案"的判断里提炼出来。

> 工程推论：
>
> **推论五：人工标注的正确用法不是"评全量"，是"评一个金标准子集去校准便宜的自动评测"。** 把人力当"标定仪器的基准砝码"，而不是"日常称重的秤"。几十条人标 → 校准 judge → judge 跑全量，是成本和质量的最优解。谁拿人工去跑每次回归，谁就会因为太贵太慢而干脆不跑 eval。

## 三、LLM-as-judge：设计与陷阱（本章技术核心）

helpfulness、忠实度、"两个答案谁更好"这类语义判断，规则算不了、人工跟不上迭代速度——只能让另一个强 LLM 当裁判。这就是 **LLM-as-judge**，是开放生成评测的主力手段。但它**坑极多，用不好你的评测数字全是假的还浑然不觉**。

第 11 章讲过 judge 在 RAG faithfulness 上的具体用法和陷阱清单；这一节把它**抽象成一套通用的 judge 设计纪律**——无论你评的是 RAG、agent 轨迹、还是普通对话，这套纪律都适用。

### 1. 两种 judge 范式：打分 vs 成对比较

| 范式 | 怎么做 | 优点 | 缺点 | 用在哪 |
|---|---|---|---|---|
| **单点打分（pointwise / scoring）** | 给单个答案按 rubric 打分或判对错 | 能算绝对分、能设阈值、能跨版本比 | 绝对分校准难、judge 的"几分"标准会漂移 | 回归基线（要一个能跨时间比的绝对数） |
| **成对比较（pairwise / A vs B）** | 给 judge 两个答案，问"哪个更好" | 相对判断比绝对打分稳得多、人也更擅长 | 只给相对序、不给绝对质量、O(n²) 不可扩展 | A/B 选型、模型对决、prompt 版本 PK |

实践经验：**判断"A 和 B 哪个好"远比"给 A 打 7 分还是 8 分"稳定**——因为相对判断绕开了"绝对分校准"这个老大难。但成对比较有个臭名昭著的坑：位置偏好（见下）。回归测试需要一个能跨时间比较的绝对数，所以单点打分仍不可少；选型决策则优先用成对比较。

### 2. judge 的五大偏见与缓解（面试高频，必背）

**judge 是个 LLM，它有 LLM 的全部毛病。** 把它的打分当客观仪表读，你就是在自我欺骗。这张表是本节灵魂：

| 偏见 | 表现 | 危害 | 缓解手段 |
|---|---|---|---|
| **位置偏好（position bias）** | 成对比较时 judge 系统性偏向排在前面（或后面）的那个 | A/B 结论被"谁先呈现"污染，纯属噪声 | **两个顺序各跑一次取一致结果**（A在前、B在前都判A赢才算A赢）；不一致则记为平局/弃权 |
| **自我偏好（self-preference）** | judge 偏袒和自己同源的模型生成的答案 | 用模型 X 生成又用 X 当 judge → 分数虚高、毫无意义 | **judge 与被评模型不同源**（最硬的红线，下面单讲） |
| **长度偏好（verbosity/length bias）** | judge 倾向给更长、更详细的答案高分，哪怕没更对 | 奖励"又臭又长"，刷分但用户体验差 | rubric 显式写"简洁且正确 > 冗长"；必要时控制答案长度后再比 |
| **谄媚 / 自信偏好（sycophancy）** | judge 偏向语气自信、措辞肯定的答案（第 1 章谄媚同一个病根） | "自信地胡说"得分高于"诚实地说不确定" | rubric 强调"唯一依据是事实/材料，语气自信不加分" |
| **判断不稳定（inconsistency）** | 同一答案 judge 两次给不同分（采样随机） | 评测有噪声，小改动的真信号被淹没 | 用**离散判断**而非连续分；低 effort/多次取众数；judge 调用本身也要控随机性 |

> 补充第六类（高级面试加分）：**格式偏好（format/Markdown bias）**——judge 偏爱排版漂亮、带 bullet/加粗/分点的答案，哪怕内容更差。和长度偏好同源（都是"看起来更用功"），缓解也类似：rubric 显式声明"格式不计分，只看内容正确性"，或评分前把两个答案都剥成纯文本。为什么单列？因为它在"让 LLM 生成 Markdown 报告"这类任务里极隐蔽——你以为在比内容，judge 在比谁的标题层级多。

> 面试官想区分什么：你是否真用过 judge、踩过这些坑。能把 position / self-preference / verbosity 三个偏见说全，且对每个给出**可执行的缓解**（不是泛泛说"注意偏见"），就是高级水平。只会答"judge 可能有偏见"是背书。

#### 缓解的总开关：让 judge 先推理再裁决（CoT-judge / G-Eval）

上表每行的缓解是"对症下药"，但有一个**系统性提升 judge 质量的总开关**值得单独拎出来，因为它对几乎所有偏见都有效，也是面试问"你怎么让 judge 更准"时最该答的那一层：**强制 judge 先逐条对照 rubric 写出推理，再给最终裁决，而不是张口就给分。** 这就是 LLM-as-judge 里的 chain-of-thought judge（学术上的 G-Eval 是其代表）——让 judge 把"为什么"显式写出来，判断质量明显更稳、和人的一致率更高。落到 Claude 上就是 judge 调用开 **adaptive thinking**（见下文落地）。

> 但这里埋着一个**反直觉的深坑，是区分"读过论文"和"真趟过"的分水岭**：judge 输出的 `reason` 字段，不一定是它真实的判断依据，很可能是**先有结论、再补理由的 post-hoc rationalization（事后合理化）**——尤其当你让它"先给分、再解释"时，解释就是给分的装饰，毫无约束力（这正是第 1 章"模型是概率延续器"的延伸：它续写一段看起来合理的理由，不代表那理由真的驱动了结论）。两个工程后果：(1) **顺序很重要**——要让 reasoning 真正约束结论，必须 prompt 成"先逐条推理、最后一行才输出 verdict"，反过来（先 verdict 后 reason）拿到的理由是事后编的；(2) **别把 judge 的自然语言理由当成可信的失败归因**直接喂给下游——它是给人审计用的线索（理由明显前后矛盾 = 这条分数不可信，该弃用），不是 ground truth。

> 工程推论：
>
> **推论六之补：CoT-judge 能系统性提升判断质量，但"先推理后裁决"的顺序是硬约束——顺序反了，理由就退化成给结论贴的事后标签，既不提升质量也不可信。** 能点出"reason 可能是 post-hoc rationalization、所以 verdict 必须排在 reasoning 之后"，是 judge 设计里最容易被忽略、面试官最想听到的一层。

### 3. 头号红线：judge 不能和被评模型同源

逻辑很硬——生成答案的是模型 X，打分的也是 X，X 会系统性偏好"自己风格的输出"，分数毫无意义地虚高。生产做法：**用一个更强的、不同厂商的模型当 judge**（比如生成用某中杯模型，judge 用一个 frontier 级、且最好是不同家的模型），或至少用同厂更强/不同代的模型。**judge 应该比被评对象更权威，而不是它的同类。**

> 工程推论：
>
> **推论六：judge 必须比被评对象更权威，且不与之同源——否则自我偏好让分数系统性虚高。** 这是把 judge 当评测基建时的头号红线。"省钱用同一个便宜模型既当生成又当裁判"是最常见也最隐蔽的自欺。

### 4. judge 的工程纪律：rubric、离散化、结构化、可审计

让 judge 相对靠谱的几条工程要点（通用于任何 judge 任务）：

1. **给明确的 rubric，别笼统打分。** 别问"这答案好不好打 1-10 分"——太主观不稳定。拆成可判断的维度，每维有清晰判据。
2. **优先二元/离散判断，少用连续分。** "这条论断有依据吗（是/否）"比"忠实度打几分"稳得多。能转成"数命中点占比"就别让它直接给 0.87。
3. **结构化输出裁决。** 让 judge 返回 `{verdict, reason, evidence}` 这种可解析结构，而非一段散文——能批量统计、能审计 judge 的判断理由（理由不合理 = 这条分数不可信）。
4. **给足上下文。** 评忠实度，judge 必须同时看到（材料, 答案）；评相关性，必须看到（问题, 答案）。少喂一块 judge 就在瞎猜。
5. **判断类任务并发跑、走 Batch、缓存 rubric。** 评测离线非实时，没必要串行。

> 落地（judge 用 Claude，按 2026 初）：judge 调用开 **adaptive thinking**（`thinking:{type:"adaptive"}`）——逐条核对这种需要推理的判断，让模型先想再裁，质量明显更稳；用 **`output_config.format`** 把裁决约束成可解析 JSON。大批量离线评测走 **Batch API**：异步、**50% 价格**、单批最多 10 万请求/256MB、多数 1 小时内完成（上限 24 小时）——攒一批样本一次性提交。再把 **rubric / few-shot 示例这些固定前缀放在 prompt 最前并打 cache 断点**：渲染顺序 `tools→system→messages`，稳定内容必须排在易变内容（每条的"材料+答案"）之前，缓存读约 **0.1×**、写一次约 1.25×，用 `usage.cache_read_input_tokens` 验证命中。**Batch 的 50% 折扣和 prompt caching 的 0.1× 可以叠乘**，是评测降本的两层杠杆。关键纪律：固定前缀里**绝不能掺时间戳/UUID/每条变化的 id**——任何一个字节变化都会让其后缓存全部失效（前缀匹配原理），命中率瞬间归零。

> 工程推论：
>
> **推论七：judge 模型版本要钉死（pin），换 judge 等于换尺子，历史基线全部作废、必须重测。** 你的 helpfulness 基线是 0.88，半年后顺手把 judge 升级了——分数变 0.85，你根本分不清是系统退化了还是新 judge 标准变严了，因为你同时动了"被测对象"和"尺子"。纪律：(1) judge 模型 ID 显式钉死、记进评测报告，别用指向 latest 的别名；(2) 升级 judge 是一次**独立变更**——先在金标准子集上重新校准、对历史样本重测建新基线，再继续。一句话：**尺子和被测物，一次只能动一个。**

## 四、离线评测集与回归测试：eval-driven development 的主循环

这是 P4 你要亲手搭的东西，也是 EDD 落地的核心：**一个能跑全量、出报表、能阻断上线的离线评测流水线。**

### 1. 离线 eval 集怎么造（通用版，RAG 细节见第 11 章）

eval 集是一批 `(输入, 期望)` 对。"期望"按评测类型不同：参考答案、或一组该满足的断言、或交给 judge 的 rubric。三条铁律：

- **必须含难例和对抗例，不能只放系统答得出的题。** 只放简单题的 eval 集是在**给自己发奖**——demo 满分、上线翻车。好的 eval 集应该让你**当前版本有明显失分**，失分点就是下一步该优化的地方。一上来就 95 分，不是系统强，是你的 eval 集太软。
- **覆盖真实分布。** 最高价值来源是**线上真实日志**（系统已上线就扒真实输入按频次抽样）。冷启动没日志时用领域专家出题 + 从文档/场景反向生成（但反生成的要人工清洗，否则全是"照原文填空"的伪样本，系统性高估能力）。
- **当版本控制的资产维护。** 线上每发现一个 bad case、每出现一类新输入，就补进 eval 集。它随系统持续生长（这正是第五节在线闭环要喂的料）。

> 坑点提示（新手第一误区）：eval 集只放自己系统答得出的问题。这数字对预测线上表现**毫无价值**——你测的全是擅长的。再强调一遍：**好的 eval 集应该让当前版本失分**。

### 2. 回归测试：每次变更自动跑，分数没涨/有回退就阻断

EDD 的主循环，和 CI 长得一模一样，只是断言从 `==` 换成了"指标阈值 + 不回退"：

```
改 prompt / 换模型 / 调 agent
        │
        ▼
   跑全量 eval 集（规则类秒级 + judge 类走 Batch）
        │
        ▼
   出报表：总体准确率/失败率 + 按子集（难例/对抗例/各能力维度）拆分
        │
        ▼
   对比 baseline：目标指标是否显著上升？任何子集是否回退？
        │
        ├─ 达标且无回退 → 合并，把这次分数设为新 baseline
        └─ 有回退 → 阻断，定位是哪个子集掉的、为什么
```

两个关键设计，是回归测试和"跑个准确率"的本质区别：

- **必须按子集拆分看，不能只看总分。** 总体 0.85→0.86 看着涨了，可能是简单题涨了 2 分、对抗例掉了 5 分——总分被稀释掉了，而对抗例正是合规命门。**总分涨不代表没回退；分子集看才抓得住"修了 A 坏了 B"。**
- **必须区分"真改善"和"采样噪声"。** LLM 有随机性，0.85→0.86 可能纯是噪声。手段：足够样本量、judge 用离散判断降噪、必要时同一版本多跑几次看分数波动范围（方差），改善幅度要显著大于这个波动才算数。

> 工程推论：
>
> **推论八：回归测试的价值不在"算出一个准确率"，在"按子集拆分 + 对比 baseline 拦住回退"。** 一个只输出总分、不分子集、不和历史比的"评测脚本"，抓不住 LLM 最典型的回归——局部变好整体被稀释、或修了新 case 坏了老 case。**分子集 + 卡 baseline，是回归测试和『跑个分』的分水岭。**
>
> **推论九：eval 跑得越快越频繁，价值越大——所以要分层跑，别一把梭。** 规则类（秒级、免费）每次 commit 都跑；judge 类（慢、花钱）走 Batch、在 PR 合并前或每日定时跑。把"快而便宜"的挡在前面当冒烟测试，"慢而贵"的留给关键卡点。否则每改一行等十分钟、烧一笔钱，团队就会干脆不跑——**跑不动的 eval 等于没有 eval**。

## 五、在线评测：A/B、影子流量与离线的分工

离线 eval 集再用心造，也有个根本局限：**它是你"想象中"的输入，不是真实用户的输入。** 真实流量千奇百怪、会问你没料到的边角、会用你没见过的措辞。所以离线之外，还要有在线评测。三档手段，分工不同：

| 手段 | 是什么 | 在哪测 | 答什么问题 | 风险 | 用在哪 |
|---|---|---|---|---|---|
| **离线 eval（offline）** | 固定 eval 集上跑、不碰真实用户 | 上线前 | "改动相对 baseline 变好没""有没有回退" | 零（不影响线上） | 开发期回归、上线门禁 |
| **影子流量（shadow traffic）** | 复制真实请求喂给新版本，但**结果不返回给用户**，只记录对比 | 生产、对用户不可见 | "新版本在真实分布上表现如何""会不会崩/超时/成本飙" | 极低（用户看不到新版本输出） | 上线前最后一道真实流量验证 |
| **在线 A/B（A/B test）** | 真实流量按比例分给新旧版本，**新版本结果真的返回给一部分用户** | 生产、对部分用户可见 | "新版本在**真实业务指标**上是否更好"（留存、转化、点赞率、转人工率） | 中（部分用户暴露在新版本下） | 离线+影子都过后的最终裁决 |

### 1. 影子流量：上线前用真实流量"演习"

离线 eval 集覆盖不了真实分布的长尾。影子流量解决这个——**把线上真实请求复制一份喂给新版本，但新版本的输出丢弃不返回给用户**，只把它和线上版本的输出（或事后人审/judge 判分）对比、记日志、看新版本会不会超时/报错/成本异常。

它的妙处：拿到了真实分布的验证，又**零用户风险**（用户拿到的还是老版本的结果）。是"离线 eval 通过"和"敢放真实用户"之间的关键缓冲带。坑点：影子调用是**额外的真金白银**（每个真实请求你现在跑两遍），要采样跑、要算这笔成本；且**有副作用的操作绝不能在影子里真执行**（影子 agent 调了 `send_email` 真发出去就出事了）——影子模式下所有写操作必须 mock 或拦截。

### 2. 在线 A/B：唯一能测"对业务到底好不好"的手段

离线指标涨了（recall 高了、faithfulness 高了），不代表用户真的更满意——**离线指标是代理指标（proxy），业务指标才是终极目标**。在线 A/B 是唯一能把"我的 LLM 指标"和"用户留存/转化/满意度"挂钩的手段：真实流量分桶，新版本结果真返回给一部分用户，比两桶的**业务指标**有无统计显著差异。

要点：(1) 比的是**业务指标**（点赞率、复制率、追问率↓、转人工率↓、留存），不是离线那套技术指标；(2) 要做**统计显著性检验**，别看到 A 桶高 0.3% 就上线——可能是噪声，样本量不够时差异不可信；(3) 灰度放量，从 1%→5%→50% 逐步加，盯住兜底指标（错误率、延迟、成本）异常就回滚。

但只讲到这里是"知道有 A/B"的水平。下面三个概念把它拉到"在大厂跑过 A/B"的水平，也是这一节最该背下来的硬货：

**(a) OEC + 护栏指标（guardrail metrics）：一个主指标，一票否决一组兜底指标。** 大厂（Google/字节/Microsoft 的实验平台）不会让你看一屏几十个指标自己拍脑袋——而是先定一个 **OEC（Overall Evaluation Criterion，总体评价标准）**作为唯一的"赢/输"主判据（比如客服场景的"会话级问题解决率"），再挂一组**护栏指标**：哪怕主指标涨了，只要任一护栏指标显著恶化（延迟 P99↑、成本/请求↑、转人工率↑、投诉率↑），这个实验**直接否决，不许上**。为什么必须有护栏？因为 LLM 改动最典型的事故是"主指标涨、护栏崩"——把答案写长把点赞率刷上去了，但 token 成本翻倍、P99 飙到 8 秒。**OEC 定方向，护栏防你为了主指标把别的烧穿**，这就是 Goodhart 定律在 A/B 层的工程化防御（呼应第六节）。

**(b) peeking / 多重检验：偷看中间结果就上线，是 A/B 头号自欺。** 你设了 5% 显著性阈值，然后实验跑到第二天就去瞄一眼——"哎 p<0.05 了，上！"。**错得离谱。** 显著性检验的前提是"在预先确定的样本量到达后看一次"；你每天偷看一次、看到达标就停（optional stopping），等价于做了几十次检验，假阳性率会从 5% 飙到 30%+——你大概率是被噪声骗了。正确做法：**上线前用功效分析（power analysis）按"想检出的最小效应量 + 期望功效"算出所需样本量和时长，到点再看结论**；要中途监控就用序贯检验/always-valid p-value 这类专门为"反复看"设计的方法，而不是拿固定样本量的 p 值反复瞄。能说出"peeking 会抬高假阳性、要么固定样本量到点看要么用序贯检验"，是统计素养的直接体现。

**(c) novelty effect / 新奇效应：短期涨可能是"新鲜感"，不是真好。** 新版本（尤其改了 UI 或交互）上线初期指标常常虚高——用户只是因为"变了"而好奇地多点了几下，一两周后回落。反过来也有 primacy effect（老用户不适应新版，短期数据偏差）。所以**重大改动的 A/B 不能只看头两天**，要跑足够长看指标是否稳住，或者只在新用户群里测以剥离新奇效应。面试被问"A/B 第一天数据特别好能上吗"，答"小心 novelty effect，要看长期是否稳住"就是有经验。

### 3. 离线 vs 在线：怎么分工

**离线兜底质量、在线发现盲区。** 离线告诉你"已知 case 没退步、能不能上线"，是开发期的快速反馈和上线门禁；在线告诉你"真实用户到底买不买账、有哪些你没料到的失效"，是最终裁决和盲区雷达。**只有离线，你不知道真实表现；只有在线，你迭代太慢且拿生产当试验田。** 生产里两层都要：离线管日常迭代和回归，影子做上线前真实流量演习，A/B 做最终业务裁决。

线上发现的每个 bad case，**回流补进离线 eval 集**——这就闭环了：在线发现新失效 → 进离线集 → 下次回归就能拦住它再次发生。离线集越长越准，靠的就是这条回流管道。

> 工程推论：
>
> **推论十：离线指标是代理指标，在线业务指标才是终极目标——离线涨了不等于用户更满意。** 你可能把离线 faithfulness 从 0.88 调到 0.94，结果答案变得又臭又长，用户体验反而降、点赞率掉。**永远用在线业务指标做最终裁决，离线指标只是过程量。** 能讲清"代理指标 vs 终极指标"的区别，是评测体系成熟度的标志。
>
> **推论十一：影子流量是"零用户风险地拿真实分布验证"的关键缓冲，但写操作必须拦截、成本要算。** 它填的是"离线集覆盖不了真实长尾"和"不敢直接放用户"之间那道缝。面试被问"新版本上线前怎么用真实流量验证又不影响用户"，能答出"影子流量 + 写操作 mock + 采样控成本"就是有生产经验。
>
> **推论十一之补：A/B 的纪律不在"分桶比大小"，在 OEC 定主判据 + 护栏防牺牲 + 不 peeking 不被噪声骗 + 看够时长躲开新奇效应。** "看到新桶高就上线"四个坑全踩：没分主次（该看 OEC）、没看护栏（成本/延迟可能崩了）、偷看中间结果（假阳性）、只看头两天（novelty effect）。把 A/B 当统计实验而非"看个数对比"来跑，是离线评测工程师和在线实验工程师的分水岭。

## 六、指标体系、失败分类与元问题：谁评评测者

### 1. 指标体系：分层，别只盯一个总分

一个能用的 eval 体系，指标必须分层——单一总分会骗你（第四节推论八）。至少四层：

| 层级 | 指标例子 | 回答什么 | 谁看 |
|---|---|---|---|
| **业务层** | 留存、转化、点赞率、转人工率、任务完成率 | 对用户/业务到底有没有价值（终极目标） | 老板、PM、上线决策 |
| **质量层** | 准确率、忠实度、helpfulness、拒答正确率 | 系统输出质量好不好 | 算法、日常迭代 |
| **能力子集层** | 难例准确率、对抗例拒答率、各 intent 分桶分数 | 是哪类能力强/弱、哪里在回退 | 算法定位优化方向 |
| **系统层** | P99 延迟、单请求成本、错误率、token 消耗 | 跑得快不快、贵不贵、稳不稳 | SRE、成本治理（第 18 章） |

**关键纪律：质量和成本/延迟要一起看。** 一个准确率 95% 但单次要 $0.5、P99 要 30 秒的方案，未必比 90%、$0.05、3 秒的方案好——**质量、成本、延迟是个三角，eval 报表必须同时呈现这三者**，否则你会为了刷质量分把成本和延迟搞爆而不自知。

> 坑点提示：只优化一个指标，必然把别的搞坏（Goodhart's law：一个指标一旦成为优化目标，它就不再是好指标）。只盯 faithfulness 会鼓励"只敢复述原文不敢综合"；只盯长度会鼓励废话；只盯准确率会忽略成本爆炸。**永远多指标一起看，盯住被牺牲的那一个。**

### 2. 失败分类法（failure taxonomy）：把"答得不好"拆成可行动的类别

eval 给出一堆失败 case，但"失败了"本身不可行动——你得知道**失败在哪一类**才能修。建一套失败分类法，把每个 bad case 归类，是 eval 从"打分"升级到"指导优化"的关键。一个通用 LLM 应用的失败大类（按你的场景裁剪）：

- **检索失败**（RAG）：对的材料没捞上来 → 治检索（第 9/11 章）。
- **grounding 失败 / 幻觉**：有材料但没用、或无中生有 → 治 prompt、加引用核对。
- **指令遵循失败**：没按格式/约束/角色来 → 治 prompt、上结构化输出。
- **工具使用失败**（agent）：选错工具、参数错、该调没调/不该调乱调 → 治工具描述、tool schema。
- **拒答失败**：该拒答的编了、不该拒的乱拒 → 治 prompt 的拒答边界、对抗例。
- **安全失败**：被注入、吐了 PII、越权 → 治安全层（第 18 章及安全章）。
- **不收敛**（agent）：循环、烧钱、走错路不回头 → 治终止条件、step 预算。

> 工程推论：
>
> **推论十二：eval 的产出不该止于一个分数，要止于一张"失败分类分布表"——告诉你失败主要集中在哪类，那才是下一步该投的地方。** "准确率 82%"不可行动；"18% 的失败里，60% 是检索没召回、30% 是指令没遵循、10% 是幻觉"——立刻知道先修检索。**能把失败归因到类别并排序，是 eval 工程师和『跑分侠』的分水岭**，和第 11 章 RAG 的分层归因是同一种思维的推广。

### 3. 元问题：谁来评测评测者（quis custodiet）

这是本章最绕、也最体现深度的一层。你用 eval 给系统打分、用 judge 给输出打分——但**这套评测体系本身可信吗？** 如果不可信，你拿它做的所有上线决策都建在流沙上。三个必须回答的子问题：

**(1) judge 准不准？——拿人工金标准校准。**
人标几十条子集，算 judge 和人的一致率（agreement，可用 Cohen's kappa 这类指标）。一致率低 → 这个 judge/rubric 不能用，要么换更强的 judge、要么改 rubric 消歧义。**没校准过的 judge 分数约等于没有。**

**(2) eval 集本身够不够好？——三个体检维度。**
- **覆盖度**：含不含难例/对抗例？覆盖真实分布吗？还是全是简单题（虚高陷阱）？
- **区分度**：好系统和坏系统在这个 eval 集上拉得开差距吗？如果烂系统也能考 90 分，这个 eval 集没有区分度，是软尺子。
- **标注质量**：参考答案/标注本身对不对？漏标、错标会让"正确"答案被判错。

**(3) 指标和你真正在乎的东西对齐吗？——离线-在线相关性。**
你的离线 helpfulness 分，和线上点赞率正相关吗？如果离线分涨了、线上业务指标没动甚至跌了，说明**你的离线指标测的不是用户真正在乎的东西**——指标本身错了，得换。这是推论十的延伸，也是元评测的终极拷问。

> 工程推论：
>
> **推论十三：评测体系必须被评测——judge 用人工子集校准、eval 集查覆盖/区分/标注质量、离线指标查与在线的相关性。不做这三件事，你的"客观指标"可能从根上就是假的。** 面试被问"你怎么知道你的 eval 是可信的"，能答出这三层的人极少，但这正是区分"搭过 eval 流水线"和"思考过 eval 可靠性"的分水岭。
>
> **推论十四：评测器要 pin、要版控、要有自己的变更评审——它是基础设施，不是可以随手改的脚本。** 改 rubric、换 judge、改 eval 集口径，都会让历史分数失去可比性。把评测器当生产基建对待：版本化、变更走评审、改动前先在金标准上验证。

> 心法（本章收尾）：**先有尺子才能做手术，但尺子本身也是被造出来的、也会错——所以最高级的 eval 工程，是同时维护『被测系统』和『评测系统』两套东西，并定期问『我的尺子还准吗』。** 不会量化的工程师在生产里裸奔；量化了却从不校准尺子的工程师，是穿着一身假数据裸奔，更危险——因为他以为自己穿了衣服。

## 七、章末五件套

### 高频面试题（11 道，附答题框架）

1. **为什么说 eval 是 LLM 应用的"第一公民"？它和传统软件测试有什么本质区别？**
   - 框架：传统测试靠"确定性输出 + `==` 断言"；LLM 是概率采样、无标准答案，所以传统断言失效。eval 是把"什么叫好"量化、把"改动是进步还是退步"可测化的前置条件——**不是收尾，是动手优化的前提**（推论一）。本质区别：要处理随机性（区分真改善 vs 噪声）、要处理语义判断（没有 `==`）。
   - 陷阱答案：把 eval 等同于"写测试用例补在功能后面"。这丢了"eval 先行倒逼定义成功标准"的 EDD 精髓。

2. **四类评测（参考答案/规则/LLM-as-judge/人工）怎么选？给个决策原则。**
   - 框架：按"能不能用便宜确定手段判"分层——**有标准答案优先 reference-based；硬约束（格式/关键词/工具选择）用 rule-based；只有语义判断（helpfulness/忠实度）才上 judge；人工只用来校准 judge 和最终验收**（推论四、五）。核心原则：能用规则算的绝不上 judge，把 judge 预算省给真正需要语义判断的。
   - 加分：点出 reference-based 内部还分 exact/包含/语义相似度/结构化等价，按任务松紧选。

3. **LLM-as-judge 有哪些偏见？每个怎么缓解？**
   - 框架：position bias（两顺序各跑取一致）、self-preference（judge 与被评模型不同源——最硬红线）、verbosity bias（rubric 写明简洁优先）、sycophancy（rubric 写明自信不加分，连第 1 章谄媚）、inconsistency（离散判断+多次取众数）。
   - 陷阱：只笼统说"judge 有偏见要注意"。必须对每个偏见给**可执行**缓解，且能点出"self-preference 是头号红线"。
   - 拔高：能补出"系统性总开关是 CoT-judge（让 judge 先逐条对照 rubric 推理、最后一行才输出 verdict），但顺序必须是先推理后裁决——反了的话 reason 就是 post-hoc rationalization、既不提质量也不可信"，直接到 staff 水平。

4. **judge 该和被评模型同源吗？为什么？怎么选 judge 模型？**
   - 框架：绝不同源——X 生成又 X 打分会系统性偏好自己风格、分数虚高（推论六）。judge 应**比被评对象更权威**：用更强、最好不同厂商的模型；至少同厂更强/不同代。
   - 加分：点出"省钱用一个便宜模型既生成又裁判"是最隐蔽的自欺。

5. **离线 eval、影子流量、在线 A/B 各测什么？怎么分工？**
   - 框架：离线测"相对 baseline 变好没/有无回退"（零风险、开发期门禁）；影子测"新版本在真实分布上表现/会不会崩"（零用户风险的真实流量演习）；A/B 测"真实业务指标是否更好"（部分用户暴露、最终裁决）。分工：**离线兜底质量、影子做真实演习、A/B 做业务裁决**（推论十、十一）。
   - 陷阱：把离线指标当终极目标。必须点出"离线是代理指标，业务指标才是终极"。

5b. **跑 LLM 改动的在线 A/B，有哪些会让你得出错误结论的坑？**
   - 框架（四个，背全是加分）：(1) **没定 OEC + 护栏指标**——要先定一个主判据（OEC），再挂护栏（延迟/成本/转人工率），主指标涨但任一护栏崩就否决，防"刷主指标烧穿成本"；(2) **peeking / 多重检验**——每天偷看、达标就停会把假阳性率从 5% 抬到 30%+，要么按功效分析定样本量到点看一次，要么用序贯检验；(3) **novelty effect**——重大改动头两天虚高是新鲜感不是真好，要看长期是否稳住或只测新用户；(4) 灰度放量 + 盯兜底指标随时回滚。
   - 陷阱：只会说"分桶比业务指标看显著性"。这是 60 分答案，缺了 OEC/护栏/peeking/novelty 这四个"为什么显著也可能是错的"的层。能讲出 peeking 抬高假阳性，是统计素养的硬证据。

6. **回归测试只看总体准确率够吗？为什么要分子集？**
   - 框架：不够。LLM 最典型的回归是"局部变好整体被稀释"——简单题涨 2 分、对抗例掉 5 分，总分看着涨实则回退（推论八）。必须按子集（难例/对抗例/各能力维度）拆分看，且和 baseline 比、卡不回退。
   - 加分：补上"区分真改善 vs 采样噪声"——同版本多跑看方差，改善幅度要显著大于波动。

7. **怎么造一个有用的离线 eval 集？什么样的 eval 集是没用的？**
   - 框架：必须含难例和对抗例、覆盖真实分布（最高价值来源是线上日志）、当版控资产持续维护。**没用的 eval 集 = 只放系统答得出的简单题**（给自己发奖、demo 满分上线翻车）。好 eval 集应让当前版本明显失分。
   - 陷阱：用 LLM 从文档反生成问题不清洗，造出"照原文填空"的伪样本，系统性高估能力。

8. **怎么把 LLM-as-judge 评测的成本压下来？**
   - 框架：三层——(1) 结构上，能用规则算的不上 judge；(2) 批量上，离线评测走 **Batch API 拿 50% 折扣**；(3) 缓存上，judge 的 rubric/few-shot 是固定前缀，用 **prompt caching 降到约 0.1×**，且和 Batch 叠乘。
   - 加分：点出固定前缀里掺时间戳/UUID 会静默击穿缓存（前缀匹配），用 `cache_read_input_tokens` 验证命中。

9. **"谁来评测评测者"——你怎么保证你的 eval 体系本身可信？**
   - 框架：三层元评测——(1) judge 准不准？拿人工金标准子集校准、算一致率；(2) eval 集够不够好？查覆盖度/区分度/标注质量；(3) 指标对齐吗？查离线指标与在线业务指标的相关性（推论十三）。不做这三件，"客观指标"可能从根上是假的。
   - 这题答得出三层的人极少，是区分深浅的分水岭。

10. **换了一个更强的 judge 模型，发现 faithfulness 从 0.91 掉到 0.88，系统退化了吗？**
    - 框架：**判断不了**——你同时动了"被测对象"和"尺子"，分不清是系统退化还是新 judge 标准更严（推论七）。正确做法：换 judge 是独立变更，先在金标准子集重新校准、对历史样本重测建新基线，再继续；尺子和被测物一次只能动一个，judge 模型 ID 要 pin。
    - 陷阱：直接下结论"退化了"或"新 judge 更准所以没问题"——都犯了"同时改两个变量"的错。

### 实战项目增量（P4 起步：为 P2/P3 建评测流水线）

**目标**：给 P2（RAG）和 P3（agent）建一条离线评测流水线，每次改 prompt/模型/编排自动跑回归，出带数字、分子集的报表。

**验收标准（缺一不算过）**：
1. **eval 集 ≥50 条带标注**，按比例覆盖典型/难例/对抗例三类；对抗例专门测拒答（库外提问、诱导编造）。eval 集进 git 版控。
2. **多类评测器并存**：规则类断言（JSON 合法/格式/工具选择正确性）+ reference-based（有标准答案的）+ LLM-as-judge（语义判断的），并能说清每条 case 用哪类、为什么。
3. **judge 异源 + 已校准**：judge 模型与被评生成模型不同源、ID 钉死；有一个 ≥20 条的人工金标准子集，报告 judge 与人工的一致率。
4. **回归报表分子集 + 对比 baseline**：一条命令跑全量，输出总体 + 按子集（难例/对抗例/各能力维度）拆分的指标，并和上一版 baseline 对比、标出回退的子集。
5. **同时呈现质量 + 成本 + 延迟**：报表里有单请求 token/成本估算和延迟，不只有准确率。
6. **一次真实回归演练**：故意改坏 prompt（或换个更弱的模型），证明流水线能在某子集上抓到回退并阻断——**抓不到回退的 eval 流水线不算过**。

**加分项**：judge 走 Batch + rubric 用 prompt caching，报告省了多少成本；接一条"线上 bad case 回流进 eval 集"的手动通道。

### 设计题（考权衡判断）

> 你的团队要把客服问答系统的生成模型从"贵的 frontier 模型"换成"便宜的中杯模型"以降本 60%。请设计一套评测方案，回答"这个降级到底能不能上"。要求覆盖：用哪几类评测、judge 怎么选、离线/影子/在线怎么排布、用什么指标做最终裁决、什么情况下你会判定"不能换"。

考点：(1) 是否用分层评测（规则+reference+judge）而非单一手段；(2) judge 是否异源（不能用被评的中杯模型当 judge）；(3) 是否理解"离线过了不够，要影子验真实分布、A/B 验业务指标"；(4) 最终裁决是否落在**业务指标**（点赞率/转人工率）而非离线技术分；(5) A/B 是否设了清晰的 **OEC（如会话级问题解决率）+ 护栏指标**——这题的护栏恰恰是"降本"的反面：成本是你主动要砍的，但**质量/转人工率/延迟是护栏，任一显著恶化就否决**，别为了省 60% 成本把用户体验烧穿；(6) 是否敢下"不能换"的判断——比如离线质量掉超过阈值、或 A/B 显示转人工率显著上升（护栏破），即使省了 60% 成本也要否决。**敢说"这个不该做"并讲清为什么，是判断力的最高形态。**

### 系统题（含量级估算）

> 你要给一个日均 100 万次调用的 LLM 应用建在线评测。需求：对线上输出做持续的 LLM-as-judge 质量监控（不是全量，采样），并支持每次发版前用影子流量在真实分布上验证。请估算：(a) 如果对线上 1% 流量做 judge 监控，judge 每天调用多少次、用 Batch 大致花多少钱（自己设 judge 输入输出 token 量级和单价）；(b) 影子流量验证一次发版要复制多少请求、额外成本量级；(c) 这套监控的数据怎么和离线 eval 集闭环。

参考量级（自己的假设要写出来）：(a) 100 万 × 1% = 1 万次/天 judge 调用；设每次 judge 输入 ~2K token（材料+答案+rubric）、输出 ~200 token，Balanced 档输入 ~$3/输出 ~$15 每百万 token，单次 ≈ 2000×3/1e6 + 200×15/1e6 ≈ $0.006+$0.003 ≈ $0.009，1 万次 ≈ $90/天；走 **Batch 砍半 ≈ $45/天**；rubric 是固定前缀，**prompt caching 让那 ~2K 输入的大头降到 0.1×**，还能再砍一截。(b) 影子验证若复制 1 万条真实请求过新版本，成本 ≈ 这 1 万条的一次完整推理花费（按你主模型单价算），且**写操作必须拦截**。(c) judge 监控里判为低分的 case + 用户点踩的 case，回流补进离线 eval 集，下次回归就能拦——闭环。考点是能把采样率、token 量级、Batch/caching 折扣、回流闭环串成一笔明白账。

- 在测试注释里写明你的 judge 模型和被评生成模型**不同源**（暗坑5）。

**暗坑提示**：
1. **不适用 = 跳过，不是 fail。** `reference_evaluator` 遇到 `expected is None`、judge 遇到 `rubric is None`，是"这条不归我管"，要返回跳过语义，绝不能当 fail——否则分数被无关失败拉低。
2. **judge 输出必须容错解析。** judge 是 LLM，可能吐非法 JSON。解析失败的 case 标记无效、剔出分母，别让一条坏 judge 输出让整个评测崩或污染分数。
3. **多评测器聚合语义要想清楚。** 一个 case 既要 JSON 合法又要内容对，`passed` 取 AND；`score` 取平均还是别的，要和你的 rubric 语义一致——别无脑取一个就完事。
4. **回退判定必须在子集级。** 只看总分会漏掉"总分涨、对抗例掉"的回退——这是 LLM 最典型的回归，也是本题的核心考点。
5. **judge 异源 + pin。** 真实项目里 judge 模型要和被评模型不同源、ID 钉死写进报告；换 judge 等于换尺子，历史分作废。
6. **regression_threshold 不是 0。** LLM 有采样噪声，子集分波动几个百分点是正常的，阈值设 0 会把噪声误报成回退；设太大又漏真回退——这个值本身要靠"同版本多跑看方差"来定。
7. **"全跳过"的 case 必须显式定义语义，否则是隐藏 bug。** 暗坑1/2 说 skip 要剔出分母，但当一个 case 被**所有**评测器都 skip（没标准答案、没 rubric、规则也不适用）时会怎样？`passed = AND(空集)` 在 Python 里 `all([])` 返回 `True`——这条没人评的 case 会被**静默算成通过**，pass_rate 虚高；而 `score = mean(空集)` 又会 `ZeroDivisionError` 直接崩。两种结局都错。正确做法：把"全跳过"识别为独立的第三态，**既不进通过数也不进分母**，并单独计数——这类 case 一多，说明你的 eval 集和评测器集没对齐（有题没人判），是该告警的信号，不是该悄悄吞掉的边界。这正是 NL 描述（"跳过不算 fail"）翻译成 code 时最容易漏的一格：你处理了"单个评测器 skip"，没处理"全员 skip"的退化情形。
