> 心法先行：**AI 应用工程师的简历不是"学过什么"，是"上线过什么、坑过什么、判断过什么"。这份文档把课程拆成一条能直接长进简历的项目链。**

读者画像：后端老手、AI 从零。所以这条链不教你写代码,只教你把"会调 API"变成"能交付一个有 eval、有 tracing、有安全边界、扛得住老板问『为什么用 RAG 不用微调』的系统"。

记住一句话贯穿全程:**不写代码 = 没学。** 每章配一个项目增量,章末题做不出来就是没过。

---

## 一、主线项目链 P0 → P5

设计原则:每个 P 只新增**一个核心能力维度**,不贪多。上一个跑通才进下一个。技术栈给"建议",不是圣旨——但 P0 的"禁框架"是死规矩。

> 时效提示:下面提到的具体模型名 / SDK 按 2026 初写,**名单会变,请学方法论**;架构维度(无状态、工具协议、检索、可观测)是物理规律,讲透到骨子里。

### P0 · 裸 API 终端助手(禁框架)

- **目标**:不靠任何 agent 框架,手写一个能多轮对话的 CLI。把 LLM 当成一个"无状态、按 token 收费、会胡说、有上下文窗口上限"的远程函数彻底吃透。
- **技术栈**:Python 或你最顺的后端语言 + 官方 SDK(Anthropic / OpenAI 二选一)。**禁止** LangChain / LlamaIndex / 任何 agent 框架。存储用内存数组即可。
- **验收标准**(缺一不算过):
  1. 支持多轮对话,且你能说清"多轮"在 HTTP 层就是每次把全部历史重发——LLM 自己不记得上一句。
  2. 实现 streaming 输出,处理中途断流。
  3. 打印每轮的 input/output token 数和累计花费(自己按价目表算)。
  4. 上下文超窗时有降级策略(截断 or 摘要),不能直接报错崩。
- **对应章节**:第 1 章(LLM 本质 / 无状态 / token / 上下文窗口)、第 2 章(API 调用 / streaming / 参数)。
- **相比上一个新增**:这是起点。锚点类比——**LLM 之于你的应用 = 不可靠 IP 包之于 TCP**:底层无状态、会丢、会错,可靠性是你在上层造出来的。这一层不亲手造一遍,后面全是空中楼阁。

### P1 · 工具化助手(Function Calling)

- **目标**:让助手能调外部工具(查天气、算数、读文件),理解 tool use 的完整闭环:模型出意图 → 你执行 → 结果回喂 → 模型续写。
- **技术栈**:延续 P0 代码库,加 tool/function calling。工具先用本地函数,别急着接真实第三方 API。
- **验收标准**:
  1. 至少 3 个工具,模型能根据 query 自主选择调哪个、甚至连环调多个。
  2. 工具报错时模型能拿到 error 并自我修正或换路,而不是把异常吐给用户。
  3. 你能画出一次"模型→工具→模型"的完整消息序列图,说清每一步谁在说话。
- **对应章节**:第 3 章(Function Calling / Tool Use 协议 / ReAct 雏形)。
- **相比 P0 新增**:从"纯文本生成"到"**带副作用的执行**"。这是 agent 的原子能力。坑点提示:工具的 schema 描述写得烂,模型就选错工具——**prompt engineering 的一大半是工具描述工程**。

### P2 · 企业知识库 RAG

- **目标**:接入一批私有文档(产品手册 / 内部 wiki),做检索增强问答。彻底搞懂为什么大厂"先上 RAG 再考虑微调"。
- **技术栈**:向量库(pgvector / Chroma / Qdrant 任一)+ embedding 模型 + 你的 P1 代码库。chunking 和检索逻辑**自己写一遍**,别直接 `index.query()` 一把梭。
- **验收标准**:
  1. 完整 pipeline:文档切分 → embedding → 入库 → 检索 top-k → 拼 prompt → 生成。
  2. 答案能引用来源(citation),问到知识库外的东西会说"不知道"而不是编。
  3. 你能讲清 chunk size、overlap、top-k 三个参数怎么影响召回与幻觉,并给出你的选型理由。
  4. 做一组**有标注的问答评测**(哪怕 20 条),量化召回命中率。
- **对应章节**:第 4 章(Embedding / 向量检索)、第 5 章(RAG 架构 / chunking / 召回与重排)。
- **相比 P1 新增**:从"模型自带知识"到"**外接可更新的真相源**"。面试加分点:能用一句话说清 RAG vs 微调的 trade-off——**RAG 改的是知识(便宜、可更新、可溯源),微调改的是行为/风格(贵、固化、难追溯)**。

### P3 · 多工具 Research Agent

- **目标**:把 P1 的工具能力 + P2 的检索能力合体,做一个能自主规划、多步执行的 research agent:拆解问题 → 多轮搜索/检索 → 综合 → 产出带引用的报告。
- **技术栈**:此时**可以**引入框架(LangGraph 等)来管理 agent loop 和 state,但要能说清框架替你做了什么——因为 P0~P2 你都手写过。
- **验收标准**:
  1. 能处理需要 ≥3 步推理的复杂问题,有 planning 和 反思(reflection)环节。
  2. 有循环终止条件,不会无限调工具烧钱;有单次任务的 token/步数预算上限。
  3. 失败可恢复:某步工具挂了,agent 能重试或绕路,最终给出降级答案。
  4. 跑 ≥3 个不同领域的 query 验证稳定性(**N=1 命中可能是运气,≥3 个独立 sample 才能区分 noise 和 robust signal**)。
- **对应章节**:第 6 章(Agent 架构 / planning / reflection / memory)、第 7 章(multi-agent 与编排)。
- **相比 P2 新增**:从"单步问答"到"**自主多步决策闭环**"。这是"AI 应用"和"AI agent"的分水岭。坑点:agent 最大的生产风险不是不够聪明,是**不收敛**——会循环、会烧钱、会自信地走错路。

### P4 · 生产化外壳(eval / tracing / 网关 / 安全)

- **目标**:给 P3 套一层生产级外壳。这是后端老手最该出彩、也最能甩开"只会调 API 的人"的一章。**模型能力是别人的,工程可靠性是你的护城河。**
- **技术栈**:eval 框架(自建 or Promptfoo/Ragas)、tracing(LangSmith / Langfuse / OTel)、LLM 网关(LiteLLM 或自建,做多 provider 路由 + 限流 + 缓存)、安全层(prompt injection 防护、PII 脱敏、输出审查)。
- **验收标准**:
  1. **离线 eval 集**:≥50 条带标注用例,每次改 prompt/模型自动跑回归,出准确率/幻觉率报表。
  2. **全链路 tracing**:任一次请求可下钻到每一步的 prompt、token、延迟、成本。
  3. **网关**:支持模型热切换、失败 fallback 到备用模型、响应缓存、按租户限流。
  4. **安全**:能挡住基础 prompt injection,对输出做敏感信息过滤,有审计日志。
- **对应章节**:第 8 章(Evaluation 体系)、第 9 章(可观测性 / tracing / 成本治理)、第 10 章(LLM 安全 / 网关 / 生产部署)。
- **相比 P3 新增**:从"demo 能跑"到"**能上线、能 oncall、能被审计**"。锚点类比——**eval 之于 AI 系统 = error budget 之于 SRE**:没有量化标准,你根本不知道改 prompt 是变好了还是变差了,只能靠"感觉"——而感觉在生产里等于裸奔。

### P5 · Agent 平台架构设计文档

- **目标**:不写代码,**写一份能过架构评审的设计文档**:面向"支撑公司 N 个团队自助搭 agent"的内部平台。这是 P9 级别要求——从写系统到**设计让别人写系统的系统**。
- **技术栈**:不限。产出物是文档:架构图、组件拆分、技术选型对照表、容量/成本估算、灰度与回滚方案、failure mode 分析、演进路线。
- **验收标准**:
  1. 画出平台分层:接入网关 / agent runtime / 工具市场 / 知识库服务 / eval & 可观测 / 计费。
  2. 至少 3 个关键决策给出**带 trade-off 的方案对照表**(自建 vs 买、单体 vs 微服务、同步 vs 异步 agent…),并下判断、给理由。
  3. 有容量估算(QPS / token 吞吐 / 成本/月)和 SLO 定义。
  4. 有 failure mode 清单和对应预案(模型挂、向量库挂、agent 不收敛、成本飙升…)。
  5. 一句话回答"平台未来 1~2 年最大的不确定性是什么、你怎么对冲"。
- **对应章节**:第 11 章(平台化架构)、第 12 章(技术决策与未来演进)。
- **相比 P4 新增**:从"交付一个系统"到"**定义一类系统的设计空间**"。判断力大于代码力,这是高级/资深和专家的分水岭。

> 项目链心法:**P0→P5 是一条『可靠性逐层上移』的链**——从把不可靠的 LLM 调通,到把它围成能上线的系统,再到把这套能力产品化给别人用。每一层都在回答同一个问题:可靠性从哪来?答案永远是"你在上层造的"。

## 二、学习节奏建议

| 模式 | 周期 | 每周投入 | 节奏要点 |
|---|---|---|---|
| 全职冲刺 | ~3 个月 | 40h+ | 每章 3~5 天:1 天读+出题,2~3 天写项目增量,1 天回看+补盲。P3/P4 各留 1.5~2 周。 |
| 业余进阶 | ~6 个月 | 10~15h | 每章 1.5~2 周,周末成块写代码,工作日碎片读+出题。守住"每章必交项目增量"的底线。 |

铁律三条:
1. **每章配项目增量,不写代码不算学完。** 读完一章手上的 P 必须长出新东西。
2. **不堆进度。** P2 的 RAG 没跑出可量化召回率,就别碰 P3。基础不牢,agent 阶段会十倍奉还。
3. **留 buffer。** P3(agent 不收敛)和 P4(eval/tracing 工程量)是两个真实陷阱区,排期天然 ×1.5。

## 三、"出题力"训练法

每章开读前,**先自己出 3 道题再看章末题**——一道面试题(考概念边界)、一道设计题(考架构权衡)、一道 debug 题(考坑感)。然后和章末题对照。

为什么出题比答题狠:
- **答题是在别人画好的地图里找路,出题是先画地图——你必须知道边界在哪、坑在哪、哪里值得问。** 出不出来,说明你根本不知道这块知识的"考点"和"应用面",这才是真盲区。
- 答题暴露的是"这一题不会";出题暴露的是"**这整片区域我没建立结构**"。后者致命得多。
- 大厂面试官就是出题的人。你练出题,等于练面试官视角——**面试时你能预判他下一刀砍哪里**。
- 对照章末题时,重点不是"我答对没",是"**他出的角度我为什么没想到**"。那个 gap 就是你和资深之间的认知差。

落地动作:建一个 `questions.md`,每章三栏——我出的题 / 章末题 / 认知 gap 复盘。三个月后这份文件本身就是你的面试预案库。

## 四、如何把课程变战绩 · 对标 P9

P9(资深专家)= **知识 × 战绩 × 判断**,三者相乘,缺一项归零。

**把课程变战绩:**
- 每个 P 都要有**可展示的产出**:P0~P4 是能跑的 repo + README(讲清架构决策),P5 是一份能过评审的设计文档。简历上写的不是"学习了 RAG",是"**搭建知识库 RAG,20 条评测集召回命中率 X%,通过 chunk/top-k 调参把幻觉率从 A 降到 B**"——带数字、带因果、带判断。
- **坑要写进复盘**:每个 P 记 ≥1 个真实踩坑(agent 不收敛怎么治、injection 怎么挡住、成本怎么压)。面试时一个真坑的价值 > 十句正确的废话。
- **dogfood 它**:拿 P3/P4 解决你自己的真实问题。被自己用过的系统,你讲起来眼神都不一样。

**对标 P9 的三层:**
1. **知识**:不止"会用",要能讲清**为什么这么设计、不这么设计会怎样**。RAG vs 微调、单 agent vs multi-agent、同步 vs 异步——每个都要能下判断 + 给理由 + 说边界。
2. **战绩**:有上线/被用过的系统,有量化结果,有解决过的真实故障。**没战绩的知识在 P9 评审里一文不值。**
3. **判断**:面对"要不要上 agent / 自建还是买 / 这个需求该不该做"能给出**带 trade-off 的结论并扛住追问**。判断力的最高形态是**敢说"这个不该做"并讲清为什么**——这正是 P5 设计文档要逼出来的能力。

> 收尾心法:**这门课不是让你『懂 AI』,是让你成为团队里那个『AI 系统出事时大家第一个 at 的人』。** 懂的人很多,能扛事的人稀缺,能定义别人怎么扛事的人——那是 P9。
