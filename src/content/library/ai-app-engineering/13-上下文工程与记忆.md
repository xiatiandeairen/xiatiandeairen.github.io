> 你在第 1 章学过：LLM 是无状态的，每一轮都要把全部历史重新塞进去，成本随轮数呈 O(n²) 增长。第 11、12 章你把单次调用拼成了 Agent loop——但 loop 一旦跑长，那个 O(n²) 的窟窿就从理论问题变成了线上事故：context 爆了、缓存击穿了、模型开始"失忆"或"中毒"。这一章解决的就是这件事:**当一个 Agent 要跑几十上百轮、上百次工具调用时,如何管理它的"工作记忆"和"长期记忆"。**
>
> 这章排在 Agent 阶段的中段不是偶然。前面你让 Agent"能跑起来",这章让它"能跑得久、跑得稳、跨会话还记得事"。context engineering 是 2025-2026 把 demo 级 Agent 推向生产级 Agent 的那道工程门槛——不跨过去,你的 Agent 在第 30 轮就会原地失智。

---

## 一、背景:为什么 context engineering 会成为一门"新学科"

2023 年大家聊的是 prompt engineering——怎么把一句话问得更好。2025 年起,严肃做 Agent 的团队改口叫 context engineering,Anthropic、Cognition(Devin)、各家 Agent 框架都在用这个词。这不是换皮营销,是问题维度变了。

**prompt engineering 关心的是"一次调用里那段静态文本怎么写"。context engineering 关心的是"在一个长达数小时、几百次调用的 Agent 运行里,每一次调用时 context window 里该装什么、不该装什么、装的顺序是什么、旧的东西什么时候清掉"。**

一句话锚点(对齐你的后端直觉):

> prompt engineering : context engineering = 写一条 SQL : 设计整个数据库的缓存/淘汰/分页策略

前者是文案,后者是**资源调度**。一旦你把 LLM 的 context window 看成一种**带成本、有容量上限、会过期的稀缺资源**(像内存、像连接池、像 L1 cache),整套工程方法论就自然长出来了。

### 工程推论

1. **context window 是稀缺资源,不是免费的暂存区。** 它有硬上限(1M token,见后),有软上限(模型在长 context 上的注意力会衰减),有钱(输出价≈输入价的 5 倍,而长 context 会推高两者)。新手最大的误区是"反正窗口大,全塞进去"——这是在用最贵的方式喂垃圾。
2. **context 的内容是动态的、随时间演化的。** 第 1 轮的工具结果到第 50 轮可能已经是纯噪声。静态 prompt 思维管不了这个,你需要**淘汰策略**。
3. **context engineering 是 Agent 可靠性的主战场。** 模型本身的能力(reasoning、tool use)是 Anthropic 的事;但"喂给模型的上下文质量"是你的事。线上 Agent 翻车,九成不是模型笨,是你喂的 context 脏了。

> 心法:**模型是 CPU,context window 是它唯一的工作内存。你不是在写提示词,你是在做内存管理。**

## 二、上下文预算管理:把 token 当预算来花

### 2.1 为什么必须有"预算"这个概念

按 2026 初的版图,Frontier 模型(Claude Opus 4.8 / Fable 5、GPT-5 系、Gemini 3 级)普遍提供 **1M token 上下文窗口**(Claude Opus 4.8 / 4.7 / Sonnet 4.6 都是 1M,Haiku 4.5 是 200K)。听起来很大,但三件事会教你做人:

- **钱**。Frontier 输入约 $5-10 / 百万 token,输出约 $25-50 / 百万 token(关键不对称:**输出价≈输入价的 5 倍**)。一个跑满 1M 输入的 Agent,光读一次 context 就是 $5-10。一个 50 轮的 loop 如果每轮都把历史重发,累计输入 token 是 O(n²),账单会吓死你(缓解手段是 prompt caching,见第 2 章和本章 §6.4)。
- **延迟**。context 越长,首 token 越慢。1M token 的 prefill 不是瞬间的。
- **注意力衰减**(physics,讲透):即便窗口装得下,模型对中间段落的利用率也会下降——业界叫 "lost in the middle"。塞满 ≠ 用满。把关键指令埋在 800K token 的中间,模型大概率视而不见。

所以严肃的 Agent 会显式做 **token 预算管理**:为整个任务设一个 token 预算,实时监控消耗,接近上限时主动压缩或收尾。

### 2.1.1 一个被新手忽略的前提:你根本不知道一段文本是多少 token

"实时监控消耗"听着简单,但有个坑:**Claude 没有公开的 tokenizer**,你在本地算不准。

新手第一反应是抓 OpenAI 的 `tiktoken` 来估——错。`tiktoken` 是 OpenAI 的分词器,拿它估 Claude 的 token 数会**系统性偏 15-20%**(代码、非英文文本上偏得更多)。你照着这个偏差数去设预算阈值、去判断"还能不能再塞一轮",会在边界上反复误判。

正确做法是调 **`count_tokens` API**(`POST /v1/messages/count_tokens`),传你**真正要用的那个 model id**(token 数是按模型算的,不同模型不一样)。它是无状态的:要 diff 一段历史压缩前后省了多少 token,就分别 count 再相减。

```python
# 量一段 context 在 Opus 4.8 下是多少 token
resp = client.messages.count_tokens(
    model="claude-opus-4-8",
    messages=messages,
    system=system,
)
print(resp.input_tokens)   # 真实 token 数,不是 tiktoken 估的
```

> 时效坑(physics 级,埋一笔):**不同代模型 token 数会变**。Opus 4.7 和 4.6 对同一段文本算出来的 token 数不一样(4.7 偏高)——所以你迁模型时,所有按 token 数定的阈值(预算上限、compaction 触发点、客户端估算器)都得拿新模型重新 `count_tokens` 跑一遍重新标定,**别套一个固定倍率**。这就是为什么"预算阈值"不能写死成魔法数字,要跟着模型走。

### 2.2 大厂怎么做:从"硬上限"到"模型自知预算"

这里有一个 2026 的重要趋势,值得单独点出。传统做法是:你在代码层用 `max_tokens` 卡一个**硬上限**——但这是个"模型不知道的天花板",模型一头撞上去就被截断,输出戛然而止(`stop_reason: "max_tokens"`)。

新做法是让**模型自己知道还剩多少预算**。Anthropic 在 Opus 4.7/4.8、Fable 5 上引入了 **Task Budgets**(beta,header `task-budgets-2026-03-13`):

```python
client.beta.messages.create(
    betas=["task-budgets-2026-03-13"],
    model="claude-opus-4-8",
    max_tokens=64000,                       # 硬上限:模型不知道,撞上就截断
    thinking={"type": "adaptive"},
    output_config={
        "effort": "high",
        "task_budget": {"type": "tokens", "total": 128000},  # 软预算:模型看得到倒计时,会自我节制
    },
    messages=[...],
)
```

`task_budget` 给模型一个"整个 agentic loop 一共这么多 token"的概念,模型看得到倒计时,会优先做重要的事、临近耗尽时优雅收尾(最小值 20000 token)。这跟 `max_tokens` 是两个东西:**`max_tokens` 是强制的、模型无感知的单次响应上限;`task_budget` 是建议性的、模型有感知的全任务预算。**

### 工程推论

4. **`max_tokens` 和 `task_budget` 是两层闸,别混用。** `max_tokens` 是安全网(防失控烧钱),`task_budget` 是行为引导(让模型自己规划)。生产里两个都设:`task_budget` 给规划,`max_tokens` 兜底。
5. **"塞满窗口"和"用好窗口"是两件事。** 预算管理的目标不是省到极致,而是**把有限注意力投在高价值内容上**。该清的清,该留的留,关键指令放在模型注意力高的位置(开头的 system、最近的几轮)。
6. **这是一个 2026 的方向性信号:行为控制的主通道正从"采样超参"迁移到"自然语言 + 预算"。** Opus 4.7/4.8 和 Fable 5 已经**移除了 `temperature`/`top_p`/`top_k` 和 `budget_tokens`**(传了直接 400)。控制深度用 `effort`(low/medium/high/xhigh/max),控制花费用 `task_budget`。这是面试高频考点(见 §9)。

> 心法:**别再问"窗口够不够大",要问"这一轮我花的每个 token 值不值"。**

## 三、Compaction:窗口要满了,自动把历史摘要掉

### 3.1 背景与演进

最早的长对话方案,大家自己在客户端做:对话太长了,就拿一个 LLM 调用把前面 N 轮总结成一段摘要,用摘要替换原始历史。这叫 **client-side compaction**(客户端压缩)。能用,但有三个坑:

1. 你得自己判断"什么时候该压"——按 token 数?按轮数?阈值难调。
2. 压缩本身是一次额外的 LLM 调用,有成本、有延迟、可能压坏(把关键信息总结没了)。
3. 压缩后历史变了,**prompt cache 整个前缀失效**(见第 2 章:缓存是前缀匹配,任何字节变化击穿其后全部)。

### 3.2 现状:服务端自动 compaction

Anthropic 现在提供 **server-side compaction**(beta,header `compact-2026-01-12`,支持 Fable 5 / Opus 4.8 / 4.7 / 4.6 / Sonnet 4.6)。当对话接近触发阈值(默认 ~150K token)时,API **自动**把早期上下文摘要成一个 `compaction` 块,你几乎无感。

Python 最小实现:

```python
import anthropic

client = anthropic.Anthropic()
messages = []

def chat(user_message: str) -> str:
    messages.append({"role": "user", "content": user_message})

    response = client.beta.messages.create(
        betas=["compact-2026-01-12"],
        model="claude-opus-4-8",
        max_tokens=16000,
        messages=messages,
        context_management={
            "edits": [{"type": "compact_20260112"}]
        },
    )

    # 致命细节:必须 append 完整 content,不能只取 text
    messages.append({"role": "assistant", "content": response.content})

    return next(b.text for b in response.content if b.type == "text")

print(chat("帮我写一个 Python 爬虫"))
print(chat("加上 JS 渲染页面支持"))
print(chat("再加限流和错误处理"))     # context 长到一定程度自动触发 compaction
```

### 3.3 那个会让你线上静默出 bug 的坑

**这是本节最重要的一句话,也是面试加分点:**

> compaction 返回的 `compaction` 块,你必须原样回传。每一轮要 append `response.content`(完整的 content 列表),**而不是只提取 `.text` 字符串**。API 靠这个块在下一次请求时替换被压缩的历史。如果你只 append 了文本,compaction 状态会**静默丢失**——没有报错,但模型下一轮就"忘了"被压缩的那部分,行为退化。

新手第一误区:写惯了"从 response 里抠出文本拼进对话"的代码,迁到 compaction 上照抠不误,然后线上莫名其妙地丢上下文,还查不出报错。记住:**有了 compaction/thinking 这些结构化块之后,"只取 text" 是反模式,要 append 整个 content。**

### 工程推论

7. **server-side compaction 的隔离边界是"摘要 vs 原文"。** 它把旧历史**换成摘要**(有损),不是清空。你换来了窗口空间,代价是细节丢失。所以别把"绝对不能丢的事实"只放在会被压缩的对话历史里——放 memory(见 §5)。
8. **compaction 的成本要算进账。** 摘要是模型干的,要花 token。但相比"每轮重发完整历史"的 O(n²),它把长对话拉回近似线性,长跑任务通常是净赚。
9. **compaction 与 prompt cache 的关系要想清楚。** 压缩会改写历史前缀,触发那一刻缓存会重建。这是必要的代价,但意味着别把 compaction 阈值设得太敏感、太频繁——否则缓存反复重建,省的钱又赔进去了。

## 四、Context Editing:不摘要,直接裁掉过期的工具结果

### 4.1 compaction 之外的另一把刀

compaction 是"**总结**":把旧的换成摘要。context editing 是"**裁剪**":直接把过期的内容**删掉**,不留摘要。两者解决的问题不同:

| 维度 | Compaction(压缩) | Context Editing(裁剪) |
|---|---|---|
| 做什么 | 把早期历史**摘要**成一个块 | 把**过期的工具结果 / thinking 块**清除 |
| 留不留痕 | 留摘要(有损保留) | 不留(直接删除,保留对话结构) |
| 典型触发 | 接近窗口上限 | 工具结果变 stale、thinking 已完成 | 
| 适用场景 | 长对话整体要溢出 | 老的 tool_result 不再相关,想保持 transcript 精简 |
| Anthropic 形态 | `compact_20260112` | `clear_tool_uses_20250919` / `clear_thinking_20251015` 等 edit 类型 |

典型场景:一个 Agent 跑了 80 轮,前面读了 30 个文件、调了 50 次 grep。到第 80 轮,前面那些 `tool_result`(文件内容、搜索输出)绝大多数已经没用了——但它们还占着窗口、还在烧每轮的输入 token。context editing 就是**按可配置的阈值,把这些 stale 的 tool_result 和已经用完的 thinking 块清掉**,让 transcript 重新变瘦,而对话的逻辑结构(谁说了什么、调了什么工具)还在。

配置形态(Beta `context-management`,与 compaction 同属一个 `edits` 数组,可以叠加):

```python
context_management={
    "edits": [
        {"type": "compact_20260112"},          # 接近上限时摘要
        {"type": "clear_tool_uses_20250919"},  # 清理过期工具结果
        {"type": "clear_thinking_20251015"},   # 清理已完成的 thinking
    ]
}
```

### 4.2 怎么选:editing 还是 compaction?

实战里 long-running agent 经常**三个一起用**,各管一段:

- **context editing**:在会话内,把 stale 的工具输出和 thinking 剪掉——便宜、无损结构、不花模型 token。优先用。
- **compaction**:当真要逼近窗口上限、连剪都剪不出空间时,才上摘要——花模型 token,有损。
- **memory**(下一节):跨会话持久化,不在窗口里。

判断顺序很清晰:**能剪就剪(editing)→ 剪不动了再摘要(compaction)→ 要跨会话活下来就写 memory。**

### 工程推论

10. **editing 比 compaction "便宜且诚实"。** 它不调模型(零额外 token),也不会"总结出幻觉"。代价是被删的内容彻底没了。所以删的应该是**确定无用**的(老工具输出),不是"可能还要用"的。
11. **删 tool_result 要保证对话仍然自洽。** API 要求每个 `tool_use` 块有对应的 `tool_result`。裁剪要么由 API 托管的 edit 类型来做(它保证结构合法),要么你自己删时极其小心——别删出"有 tool_use 没 tool_result"的非法历史(直接 400)。这就是为什么优先用平台托管的 context editing,而不是手撸删除。
12. **editing/compaction 都会动缓存前缀。** 任何对历史的删改都在缓存前缀里制造字节变化,击穿其后缓存。把"剪枝/压缩"安排在每轮固定位置、用稳定的阈值,别让它在随机时机触发,否则缓存命中率会被你自己打穿。

> 心法:**剪枝(editing)是手术刀,压缩(compaction)是绞肉机。先动刀,实在不行才动绞肉机。**

## 五、短期记忆 vs 长期记忆:Agent 的两套记忆系统

### 5.1 一个清晰的二分

把人脑的类比拿过来,但要讲准(别停在概念):

| | 短期记忆(Short-term) | 长期记忆(Long-term) |
|---|---|---|
| 载体 | **context window 本身**(当前会话的 messages) | **会话外的存储**(文件 / DB / 向量库) |
| 生命周期 | 会话结束即灰飞烟灭 | 跨会话、跨进程持久化 |
| 容量 | 受窗口上限约束(稀缺) | 近乎无限(便宜) |
| 访问方式 | 模型直接"看见"(在 context 里) | 需要**主动检索**才能进 context |
| 工程手段 | compaction / context editing | memory tool / RAG |
| 类比 | RAM / L1 cache | 磁盘 / 数据库 |

**关键认知:短期记忆是"在场的",模型不用做任何事就能用;长期记忆是"离场的",必须先检索、再注入 context,模型才知道它存在。** 这个区别决定了所有工程设计。

### 5.2 为什么必须有长期记忆

因为短期记忆**会随会话死掉**。一个客服 Agent、一个编码 Agent,如果每次开新会话都是一张白纸——用户上次说的偏好、项目的约定、上一轮调试的结论,全没了。用户体验是"金鱼记忆"。长期记忆就是给 Agent 一块**能跨会话活下来的硬盘**。

### 工程推论

13. **"记不住"分两种,病因完全不同。** 同一会话里记不住 = 短期记忆问题(窗口溢出 / 被错误裁剪 / lost in the middle)→ 查 context 管理。跨会话记不住 = 长期记忆问题(根本没写 memory,或写了没检索)→ 查 memory/RAG 链路。面试和排障都先做这个二分,别混为一谈。
14. **长期记忆的成本主要在"检索",不在"存储"。** 存便宜。但每次都得决定"这次该取哪些记忆塞进 context"——取多了污染窗口、取少了等于没记。检索质量 = 长期记忆系统的天花板。
15. **别把该长期的东西只放短期。** "用户是 GAAP 会计准则""这个 repo 用 Go"——这类**会被反复需要**的事实,放会话历史里迟早被 compaction 摘没。它们应该一开始就写进 memory。

## 六、记忆写入策略:何时写、写什么、如何检索

这是整个长期记忆系统里最难、最考验工程判断的部分。存储和检索的 API 都是现成的,**难的是策略**。

### 6.1 何时写(When to write)

三种主流时机:

- **任务结束时写**(end-of-task):一个任务完成后,把"学到的结论 / 用户的新偏好"落盘。最常见,信号干净。
- **过程中持续写**(continuous):跑长任务时,边做边记 scratchpad。适合需要跨多轮维持状态的任务(Opus 4.7/4.8 在"给自己记笔记并在后续利用笔记"上有明显增强)。
- **compaction 时写**(at compaction):在历史被压缩之前,把"绝对不能丢"的事实抢救进 memory,再让 compaction 去压对话。

### 6.2 写什么(What to write)

> 新手第一误区:把对话原文一股脑全存进 memory。错。memory 是**提炼过的事实**,不是聊天记录的备份。

该写的是:**稳定的、会被未来复用的、高信息密度的**条目——用户偏好、项目约定、关键决策、踩过的坑、复用的参考资料。不该写的:一次性的中间结果、马上就过期的状态、能从别处重新拿到的东西。

判断标准一句话:**"这条信息下次还会被需要吗?如果会,它现在还能从别处廉价拿到吗?"** 会被需要、且拿不回来的,才写。

### 6.3 如何检索(How to retrieve)

存了不会自己进 context,得检索。检索策略从简到繁:

- **全量加载**(small memory):记忆很少时,开会话就把整个 memory 目录读进来。简单,但只适合小规模。
- **按路径 / 前缀取**:memory 按结构化路径组织(`/preferences/...`、`/projects/repo-x/...`),按当前任务取相关子树。
- **语义检索**(RAG 化,见 §7):memory 大了之后,靠 embedding 相似度召回最相关的 N 条。这就是"记忆即 RAG"。

### 6.4 一个绕不开的工程约束:检索不能打穿缓存

**面试加分点:** memory 检索注入的内容如果放错位置,会击穿 prompt cache。回顾第 2 章的铁律——缓存是前缀匹配,渲染顺序 `tools → system → messages`,前缀任何字节变化使其后全部失效。所以:

- **别把每轮变化的检索结果拼进 system prompt 开头**——那是缓存前缀的最前面,等于每轮击穿全部缓存。
- 正确做法:把检索到的 memory 作为 **messages 里靠后的内容**注入(一个 user 块,或在支持的模型上用 mid-conversation `role:"system"` 消息,header `mid-conversation-system-2026-04-07`),让稳定的 system 前缀保持字节不变,缓存照命中。

### 工程推论

16. **写入是"提炼",不是"备份"。** 存原文 = 用昂贵的检索成本换一堆噪声。每条 memory 都应该是被人(或模型)提炼过的、能独立读懂的事实。
17. **写入要做去重和冲突解决。** 用户上次说"用 2 空格缩进",这次说"用 tab"——你的 memory 系统得能更新而不是无脑追加两条矛盾记录。生产级 memory 需要 read-modify-write + 乐观锁(下面平台能力里会看到 `precondition` / `content_sha256`)。
18. **检索的位置 = 缓存命中率。** 把动态检索结果钉死在 system 开头是新手最常见的"省钱省成负优化"。动态的东西往后放,静态前缀保持冻结。
19. **memory 是安全敏感面。** 别往 memory 里写 API key、密码、token;PII 要看 GDPR/CCPA 合规;多用户系统必须按 user 隔离 memory 目录 + 鉴权。memory 是会持久化、会被检索回来的——写进去的敏感信息等于长期暴露。

> 心法:**写 memory 的那一刻问自己三句话:这条以后还要用吗?现在不存以后还拿得到吗?它干净到可以独立读懂吗?三个 yes 才写。**

## 七、记忆与 RAG 的关系:记忆即"RAG 化的历史"

### 7.1 一句话点破

**长期记忆,本质上就是把"Agent 自己产生的历史"做成了一个可检索的知识库。RAG 的检索对象是"外部文档",记忆的检索对象是"Agent 过去的经验"——两者的检索机制是同一套。**

你在 RAG 章节(Agent 阶段前置)学的那一套——切块、embedding、向量库、相似度召回、re-rank——原封不动适用于长期记忆。区别只在数据来源:

| | RAG | 长期记忆 |
|---|---|---|
| 检索对象 | 外部知识(文档、网页、知识库) | Agent 自身的历史经验、用户偏好、项目状态 |
| 数据来源 | 你预先灌进去的语料 | Agent 在运行中**自己写**的 |
| 写入时机 | 离线批量构建 | 在线持续追加 |
| 检索机制 | embedding 相似度 / 关键词 | **同一套**(embedding / 路径 / 前缀) |
| 时效性 | 相对静态 | 高度动态(每次会话都在长) |

### 7.2 为什么这个视角重要

因为它告诉你:**你不需要为"记忆"发明一套全新技术栈。** 记忆系统 = RAG 基础设施 + "Agent 自动写入"这一层。会做 RAG,就会做记忆的检索侧;难点全在写入侧的策略(§6)。

小到 memory tool 的 `/memories` 目录(全量或按路径读),大到给 memory 接一个向量库做语义召回——都是同一个谱系上的不同规模档位。memory 小用文件,大了用 RAG,平滑过渡。

### 工程推论

20. **记忆系统的检索侧 = 复用 RAG 基建。** 别重复造轮子。embedding、向量库、re-rank 直接拿过来。新增的只是"何时把哪些经验写进去"。
21. **写入侧才是记忆的独有难题。** RAG 的语料是人精心准备的;记忆的语料是 Agent 在线自动产生的,质量参差。所以记忆系统需要写入时的提炼、去重、冲突解决——这是 RAG 没有的环节。
22. **规模决定检索档位。** 几十条 memory:全量加载或路径过滤,够了,别上向量库。几千上万条:必须语义检索。过早上 RAG 是过度工程,过晚上是检索召回崩盘。按规模选档。

> 心法:**记忆不是新东西。它是 RAG 掉了个头——检索的不是别人的文档,是 Agent 自己的昨天。**

## 八、记忆工具:/memories 后端与平台托管两条路

落地长期记忆,2026 有两条成熟路径,对应两种 Agent 部署形态。

### 8.1 路径一:Memory Tool(客户端 `/memories` 后端)

Anthropic 的 **memory tool**(`memory_20250818`)是一个**客户端工具**:模型发出读写 `/memories` 目录的指令,但**存储后端由你实现**。它支持的命令:`view` / `create` / `str_replace` / `insert` / `delete` / `rename`——本质上是给模型一套对"记忆文件系统"的增删改查能力。

```python
response = client.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    messages=[{"role": "user", "content": "记住:我偏好用 Python。"}],
    tools=[{"type": "memory_20250818", "name": "memory"}],
)
```

SDK 提供了后端骨架(Python `BetaAbstractMemoryTool`,你子类化实现 `view/create/str_replace/insert/delete/rename`,可以落到本地文件系统、S3、数据库),配合 tool runner 自动跑工具循环。**关键:存储介质、访问控制、加密、多租户隔离全是你的事。** 平台只定义协议,不替你保管数据。

### 8.2 路径二:Memory Stores(Managed Agents 平台托管)

如果你用 Anthropic 的 **Managed Agents**(平台托管 Agent,见第 12 章 Agent 形态),有更省事的 **Memory Stores**:workspace 级、跨会话持久化、平台帮你存。它的设计有几个值得学的工程细节:

- **对象模型三层**:Memory Store(`memstore_`,workspace 级容器)→ Memory(`mem_`,单个文本文件,按 `path` 寻址,每个 ≤100KB,鼓励"多个小文件")→ Memory Version(`memver_`,每次改动的不可变快照,自带审计 + 回滚)。
- **Agent 怎么用**:store 以 **FUSE 文件系统**挂载进容器(`/mnt/memory/<store-name>/`),Agent 用普通文件工具(`read`/`write`/`edit`/`grep`)读写,**没有专用 memory 工具**。`read_only` 在文件系统层强制只读。一段 store 描述会自动注入 system prompt,Agent 不用你提醒就知道挂载在哪。
- **乐观并发**:`memories.update` 支持 `precondition`(`content_sha256`),read-modify-write 时防并发覆盖,冲突返回 409(`memory_precondition_failed_error`)。这就是 §6 工程推论 17 说的去重/冲突解决在平台层的样子。
- **审计与合规**:每次改动一个不可变 version,记录 `created_by`(actor 类型:session / api / user);支持 `redact` 抹掉历史 version 的内容但保留审计痕迹——专门给"泄露了密钥""用户要求删除 PII"用。

### 8.3 两条路怎么选

| | Memory Tool(`/memories`) | Memory Stores(Managed Agents) |
|---|---|---|
| 适用 | 你自己跑 Agent loop(Claude API + tool use) | 用平台托管的 Managed Agents |
| 存储 | **你实现**(文件/DB/S3) | 平台托管 |
| 控制力 | 最大(介质、加密、隔离全自己定) | 省心(版本/审计/并发平台给) |
| 合规 | 你自己扛 | 平台给 redact / version 审计 |
| 心智负担 | 高 | 低 |

判断:**自建/私有部署、要完全掌控数据落地 → memory tool;用平台托管 Agent、想要开箱即用的版本/审计/并发 → memory stores。** 这跟"自建 Redis vs 用云托管 Redis"是同一类决策。

### 8.4 国内厂商语境

按 2026 初版图,国内的 Agent/记忆能力大多还在"框架 + 自建存储"阶段——豆包(字节)、Qwen(阿里)、DeepSeek、Kimi(月之暗面)、GLM(智谱)、混元(腾讯)、文心(百度)各自的 Agent 平台都在补 memory 这一层,但"模型原生的 memory tool 协议 + 托管 memory store + FUSE 挂载 + version 审计"这套**端到端工程化**,Anthropic 走得比较靠前。对应用工程师的现实含义:**用国内模型时,长期记忆这一层大概率得你自己用 RAG 基建拼**(向量库 + 写入策略),而不是指望模型原生给你一个 memory 协议。这反而印证了 §7 的判断——会 RAG 就会做记忆。(名单和能力按季度变,学方法论别背名单。)

### 工程推论

23. **memory tool 是"协议",不是"存储"。** 平台只定义模型怎么读写记忆,数据放哪、怎么保管是你的活。别误以为开了 memory tool 数据就自动安全持久了。
24. **"多个小文件"优于"一个大文件"。** memory store 限制单文件 ≤100KB、鼓励拆分,是有道理的——细粒度文件让检索更精准、更新冲突更少、redact 更干净。一个巨型 memory 文件是反模式。
25. **生产记忆必须有版本和审计。** 不是为了好看。记忆会被模型自动改,出了问题(写错了、泄露了)你需要回滚和追溯。Memory Stores 的 `memver_` + `redact` 就是这个工程需求的标准答案;自建 memory tool 时你得自己补这一层。

> 心法:**记忆工具给的是一把锁和一扇门(协议),房子(存储)和保安(合规)还得你自己建。**

## 九、失效模式:上下文污染、中毒、无关信息干扰

context engineering 之所以是"可靠性主战场",是因为 context 一脏,模型就崩,而且崩得很隐蔽。讲透三种典型失效模式(physics 级,要会诊断):

### 9.1 上下文污染(Context Pollution)

**定义**:context 里堆了太多**低价值 / 过期 / 重复**的内容,挤占了模型的注意力预算,导致它抓不住真正重要的信息。

**典型成因**:老工具结果没清(该上 context editing 没上);把对话原文全存进 memory 又全检索回来;system prompt 里堆了一堆"以防万一"的条件分支。

**症状**:模型开始忽略你明确给的指令,或者答非所问,或者反复确认已经说过的事。

**药方**:context editing 剪枝(§4) + memory 写入做提炼而非备份(§6.2) + 关键指令放注意力高位(开头 system / 最近几轮)。

### 9.2 上下文中毒(Context Poisoning)

**定义**:context 里进了**错误的"事实"**,然后模型把它当真,后续推理全建立在这个错误之上,甚至把错误写进 memory,**污染扩散到未来所有会话**。

**典型成因**:工具返回了错误结果但没被识别;模型自己幻觉出一个"结论"被存进了 memory;prompt injection(用户输入里藏了伪造的指令/事实)。

**症状**:模型坚定地基于一个错误前提往下做,而且越来越自信——因为那个错误已经在 context 里被它反复引用了。中毒比污染更危险:**污染是噪声大,中毒是信号假。**

**药方**:(1)工具结果做校验,别无脑信任;(2)memory 写入前过滤,别把未经验证的模型"结论"落盘;(3)用 mid-conversation `role:"system"` 这种**不可伪造的 operator 通道**下达指令,而不是把指令塞进可被用户伪造的 user/tool 内容里(防 prompt injection);(4)memory 出问题用 `redact` 切断扩散。

### 9.3 无关信息干扰(Distraction / Irrelevant Context)

**定义**:context 里塞了**与当前任务无关**的信息(哪怕是正确的),把模型带偏。

**典型成因**:RAG / memory 检索召回了不相关的内容;一股脑塞了过多上下文"让模型自己挑"。

**症状**:模型纠结于无关细节,或者把无关信息硬扯进答案,任务焦点漂移。

**药方**:提高检索精度(re-rank、收紧 top-k);宁缺毋滥——§2 讲的"用好窗口"而非"塞满窗口"。

### 三种失效模式速查

| 失效模式 | 本质 | 一句话症状 | 主药方 |
|---|---|---|---|
| 污染 Pollution | 噪声多,挤占注意力 | 忽略指令 / 答非所问 / 重复确认 | context editing 剪枝 + memory 提炼 |
| 中毒 Poisoning | 信号假,错误被当真并扩散 | 基于错误前提越走越自信 | 工具结果校验 + memory 写入过滤 + operator 通道 + redact |
| 干扰 Distraction | 无关信息带偏焦点 | 纠结无关细节 / 焦点漂移 | 提高检索精度 + 宁缺毋滥 |

### 9.4 怎么"看见" context 出问题:三个 usage 字段就是你的仪表盘

上面三种失效是"行为级"症状。但等模型答非所问你才发现,已经晚了——而且很难复盘是哪一类。生产级 context 工程师靠 **`response.usage` 里的三个字段**把 context 状态变成可观测、可告警的数字。这是 §10.4 "context engineering 会和可观测性融合"的落地起点,也是面试里区分"读过文档"和"真上过线"的分水岭。

每次响应回来,这三个字段拼出"这一轮 context 到底花在哪":

| 字段 | 含义 | 你该盯它的什么 |
|---|---|---|
| `input_tokens` | **未命中缓存**、按全价处理的 token(注意:是缓存外的余量,不是总输入) | 突然飙高 = 缓存被击穿了(见 §6.4),或历史在疯长 |
| `cache_read_input_tokens` | 命中缓存、按约 0.1× 计费的 token | 在前缀相同的重复请求间**恒为 0** = 有静默击穿源(system 里塞了时间戳/UUID、JSON 没排序、tool 集在变) |
| `cache_creation_input_tokens` | 写入缓存、按 1.25×(5min)/2×(1h)计费的 token | 每轮都在写、几乎不读 = compaction/editing 触发太频繁,把缓存反复打穿 |

**一个反直觉但必考的点:`input_tokens` 不是总输入。** 真实 context 大小 = `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`。一个跑了几小时的 Agent,如果你只看 `input_tokens` 显示 4K 就以为"没花多少 token",那是被缓存读骗了——绝大多数 token 是从缓存喂进去的。**算总量要三个字段相加,别只看一个。**

这把仪表盘怎么接回三种失效的诊断:

- **三个字段之和持续逼近窗口上限** → 该上 compaction/editing 了(§3/§4),否则下一步就是 `model_context_window_exceeded` 截断。
- **`cache_read` 该命中却为 0** → 缓存被击穿,成本静默翻几倍,且往往是"检索结果被钉进了 system 开头"(§6.4)这个污染源——既烧钱又把动态噪声塞进了最稳定的前缀。
- **`input_tokens` 阶梯式上涨而非平台化** → 历史在线性甚至超线性膨胀,editing 没在剪、或剪枝阈值太松,污染正在累积。

> 埋坑点:把这三个字段连同"本轮 context 装了哪些块、为什么装"一起打进结构化日志/APM,context 调度才从"玄学调参"变成"可回归的工程"。能把这套讲出来的人,就是 §10.4 说的 3-5 年稀缺人才的雏形。

### 工程推论

26. **三种失效要会鉴别诊断。** 污染是"context 里垃圾太多",中毒是"context 里有假货",干扰是"context 里有跑题的真货"。线上 Agent 行为异常,先做这个三分类,再对症下药——这是 context 工程师的核心诊断能力,也是面试区分度极高的题。
27. **中毒最该防,因为它会跨会话扩散。** 污染和干扰是当次会话的问题,会话结束就没了。中毒一旦写进 memory,就**污染未来所有会话**。所以 memory 写入这道闸要把得最严——宁可漏存,不可错存。
28. **prompt injection 是中毒的人为版,要走"通道隔离"。** 别把 operator 指令和用户内容混在同一个 user 文本块里——用 `role:"system"`(mid-conversation,不可伪造)或工具通道把"系统的话"和"用户的话"在物理上分开。把信任边界做进通道,而不是指望模型自己分辨真假。

> 心法:**Agent 失智九成不是模型笨,是 context 脏了。脏分三种:垃圾多(污染)、有假货(中毒)、跑题(干扰)。先诊断是哪种,再下药。**

## 十、未来演进(3-5 年)与对应用工程师的含义

讲方向,不写未验证的细节(版图按季度过期,学趋势别背名单):

1. **窗口会更大,但"塞满"永远是反模式。** 1M 走向 10M 不是不可能,但注意力衰减、成本、延迟这三个 physics 约束不会消失。**context engineering 不会因为窗口变大而消亡,只会从"怕装不下"转向"怕用不好"。** 这门学科是长期的。
2. **行为控制持续往"自然语言 + 预算"迁移。** 采样超参(temperature 等)在 Frontier 模型上已经被移除,`effort` + `task_budget` 接管。趋势是:**你越来越像在"管理"一个 Agent(给目标、给预算、给约束),而越来越不像在"调参"一个模型。** 应用工程师的核心技能从调参转向"上下文与预算的调度"。
3. **记忆会越来越"原生"和"托管"。** memory tool 协议、托管 memory store、FUSE 挂载、version/审计——这些今天还是 Anthropic 领先的工程化,3-5 年内会成为各家 Agent 平台的标配。对应用工程师:**今天值得自己用 RAG 拼记忆来吃透原理,但要预期未来这层会被平台标准化,你的价值会从"造记忆系统"转向"设计记忆策略"。**
4. **context engineering 会和可观测性/评估深度融合。** "这一轮 context 里装了什么、为什么、命中缓存没、被污染没"会变成像 APM 一样的标准监控面。能把 context 调度做到可观测、可评估、可回归的工程师,是这个方向 3-5 年的稀缺人才。

> 心法(全章收束):**你不是在写提示词,你是在给一个无状态、健忘、注意力有限的智能体做内存管理 + 硬盘管理 + 预算管理。把它当成一个需要被精心调度的系统资源,而不是一个无所不知的聊天框——这就是 context engineering 的全部。**

## 章末五件套

### 高频面试题

**1. compaction 和 context editing 有什么区别?分别什么时候用?**

考点:能不能区分"摘要"和"裁剪"两种长 context 治理手段。
答题框架:compaction = 把早期历史**摘要**成一个块(有损保留),触发于接近窗口上限,要花模型 token;context editing = 直接**裁剪**过期的工具结果/thinking 块(不留摘要,保留对话结构),便宜、不调模型。选择顺序:能剪就剪(editing)→ 剪不动再摘要(compaction)→ 跨会话才写 memory。
错误答案陷阱:把两者说成一回事,或者认为 editing 也会"总结"。editing 是删除不是总结,这是关键区分。

**2. 用 compaction 时,为什么必须 append `response.content` 而不是只取 `.text`?**

考点:对结构化 content 块的理解 + 静默 bug 意识。
要点:compaction 的状态保存在 response 的 `compaction` 块里,API 靠它在下一轮替换被压缩的历史。只 append `.text` 会**静默丢失** compaction 块(无报错),模型下一轮丢上下文、行为退化。
加分点:能指出这是"只取 text"这个旧习惯在结构化块时代变成反模式——thinking 块、compaction 块都得 append 整个 content。

**3. 短期记忆和长期记忆的本质区别是什么?这个区别如何影响工程设计?**

考点:能不能从"载体"和"访问方式"上把二者讲透。
要点:短期记忆 = context window 本身,模型**直接看见**,会话结束即灭;长期记忆 = 会话外存储,**必须主动检索**才进 context,跨会话持久。影响:短期靠 compaction/editing 治理,长期靠 memory/RAG;"在场 vs 离场"决定了长期记忆的全部难点在检索。
陷阱:只答"一个短一个长"。要答到"在场/离场"和"是否需要主动检索"这层。

**4. "记忆即 RAG 化的历史"——解释这句话,以及它对技术选型的含义。**

考点:能不能看穿记忆和 RAG 是同一谱系。
要点:长期记忆 = 把"Agent 自己产生的经验"做成可检索知识库,检索机制和 RAG 完全同一套(embedding/向量库/re-rank);区别仅在数据来源(外部文档 vs Agent 自身历史)和写入时机(离线 vs 在线持续)。选型含义:记忆的检索侧直接复用 RAG 基建,不用重造;独有难题在写入侧策略。
加分点:能指出规模决定档位——少量用文件全量加载,大量才上向量库,过早上 RAG 是过度工程。

**5. 列举并区分上下文的三种失效模式,各给一个药方。**

考点:context 工程师的核心诊断能力。
要点:污染(噪声多挤占注意力 → context editing + memory 提炼)、中毒(假事实被当真并跨会话扩散 → 工具校验 + memory 写入过滤 + redact + operator 通道)、干扰(无关真信息带偏 → 提高检索精度)。
陷阱:把三者混为一谈,或答不出"中毒最危险因为会跨会话扩散"。能讲出"垃圾多/有假货/跑题"这个通俗三分,且点明中毒的扩散性,是高分。

**6. 记忆写入策略里,"何时写、写什么、如何检索"分别怎么做?**

考点:长期记忆里最考验工程判断的部分。
要点:何时写——任务结束 / 过程中持续 / compaction 前抢救;写什么——提炼过的稳定可复用事实,不是对话原文备份(判断:下次还要用且现在拿不回来才写);如何检索——按规模从全量加载→路径前缀→语义召回。
加分点:能提"写入要去重和冲突解决"(乐观锁/precondition),以及"检索结果别放 system 开头否则击穿缓存"。

**7. memory 检索如何与 prompt caching 共存?放错位置会怎样?**

考点:把 caching 的前缀匹配铁律和 memory 检索结合。
要点:缓存是前缀匹配,渲染顺序 tools→system→messages,前缀任何字节变化击穿其后全部。每轮变化的检索结果若拼进 system 开头 = 每轮击穿全部缓存。正确做法:检索结果作为 messages 靠后的内容注入(user 块或 mid-conversation `role:"system"`),保持 system 前缀字节冻结。
陷阱:不知道动态内容放 system 开头会打穿缓存——这是"省钱省成负优化"的经典坑。

**8. `max_tokens` 和 `task_budget` 有什么区别?为什么 Frontier 模型移除了采样超参?**

考点:2026 行为控制范式迁移的理解。
要点:`max_tokens` = 强制、模型无感知的单次响应硬上限(撞上截断);`task_budget` = 建议性、模型有感知的全任务预算(看得到倒计时会自我节制)。Opus 4.7/4.8、Fable 5 移除了 `temperature/top_p/top_k` 和 `budget_tokens`(传了 400),控制深度用 `effort`、花费用 `task_budget`——体现"行为控制主通道从采样超参迁移到自然语言+预算"的趋势。
加分点:能点出这是"从调参模型转向管理 Agent"的范式信号。

**9. 线上 Agent 跑了几小时,你怎么量它一轮到底花了多少 context token、又怎么判断缓存有没有被击穿?**

考点:从"概念会讲"到"上过线"的分水岭——token 测量 + usage 可观测性。
要点:(1)本地算不准——Claude 无公开 tokenizer,用 `tiktoken` 估会系统性偏 15-20%,要调 `count_tokens` API 并传真实 model id;(2)真实 context 大小 = `input_tokens + cache_creation_input_tokens + cache_read_input_tokens` 三者之和,`input_tokens` 只是**未命中缓存的余量**,只看它会被缓存读骗到"以为没花多少";(3)前缀相同的重复请求间 `cache_read_input_tokens` 恒为 0 = 有静默击穿源(system 里有时间戳/UUID、JSON 没排序、tool 集在变)。
陷阱:把 `input_tokens` 当总输入;不知道用哪个字段判断缓存命中;张口就用 `tiktoken` 估 Claude。
加分点:能把三个 usage 字段映射到三种失效的诊断(之和逼近上限→该压缩;`cache_read` 该命中却为 0→检索被钉进 system 开头的污染源),并指出迁模型后所有 token 阈值要重新 `count_tokens` 标定、不能套固定倍率。

### 系统题(含量级估算)

> 设计一个 Agent 平台,支撑 **1 万并发 Agent session**,每个 session 平均跑 **50 轮**、每轮平均 **20K input token**、长期记忆库每用户约 **5000 条** memory。估算:(a) 如果不做任何 context 治理(每轮重发完整历史),单 session 的累计输入 token 量级和成本(按 Frontier 输入 $5/百万);(b) 引入 prompt caching + compaction 后,成本大致能压到什么量级;(c) 长期记忆检索侧的 QPS 和向量库规模估算;(d) 给出你的架构分层(短期治理层 / 长期记忆层 / 缓存层 / 可观测层)。

参考量级(用于自检,数量级对即可):
- (a) 不治理,第 k 轮要发约 20K×k token,50 轮累计 ≈ 20K × (1+2+…+50) = 20K × 1275 ≈ **2550 万 input token / session**,单 session 输入成本 ≈ 2550 万 / 100 万 × $5 ≈ **$127.5**(只算输入,输出另计且单价 5 倍)。1 万并发 × 这个量级 = 百万美元级——这就是 O(n²) 的恐怖,也是为什么必须治理。
- (b) prompt caching 让重复前缀走缓存读(~0.1×),compaction 把历史拉回近似线性;两者叠加把单 session 输入成本压低一两个数量级到 **$1-10 量级**。
- (c) 检索 QPS ≈ 1 万 session × (每会话检索次数 / 会话时长),按每会话开场检索 + 若干轮检索估,峰值 QPS 在千级;向量库规模 = 用户数 × 5000 条 × embedding 维度,需要分片。
- (d) 分层:短期治理(compaction/editing)+ 长期记忆(memory store + 向量检索)+ 缓存层(prompt cache,缓存前缀冻结)+ 可观测(每轮 context 构成、缓存命中、token 消耗监控)。

考点:能不能算出"不治理 = O(n²) = 烧钱黑洞"这个核心结论,并把治理手段对应到成本量级的下降。
