> 这章排在最前面，因为它是全课的**物理常数表**。后面所有架构——RAG、工具、agent、eval、网关——都是为了对冲这一章揭示的四个"物理事实"：模型在做概率延续而非事实查询、模型无状态、模型按 token 收费、模型不知道自己不知道。你越早把这四条刻进直觉，后面越少走"以为框架能救我"的弯路。
>
> 替你解决的问题：让你停止把 LLM 当"会聊天的数据库"，开始把它当"一个不可靠、按量计费、会自信地胡说的远程随机函数"。**LLM 之于你的应用 = 不可靠 IP 包之于 TCP**——底层不保证任何东西，可靠性是你在上层造出来的。这章就是讲清楚"底层到底不保证什么"。

读法提示：这章信息密度高，每个机制后面紧跟**工程推论**（编号列出，每条都能独立成一道面试题）。第一遍至少把四个推论和"长上下文 vs RAG 四维对比表""幻觉三类型表"啃下来。代码题对应主线项目 **P0 裸 API 终端助手**。

时效标注：模型名单 / 价格 / 具体工具按 **2026 初**写，名单按季度过期，**重点学选型方法论而非背名单**；自回归、token、上下文窗口、采样这些是**物理规律**，讲透到骨子里，十年不变。

## 一、自回归生成：一切诡异行为的总根源

### 1. 背景：LLM 在做什么

把所有花哨的描述删掉，当前主流 LLM（GPT 系、Claude 系、Gemini 系、以及 Qwen/DeepSeek/Kimi/GLM/混元/文心 等国内模型）干的事只有一件：

**给定前面所有 token，预测下一个 token 的概率分布，采样一个，拼回去，再预测下一个。** 循环到吐出一个"结束"标记。

这叫 **autoregressive（自回归）生成**。"自回归"意思是：它的每一个输出，都成为下一步的输入。没有全局规划，没有"先想好答案再写"，它是**一个 token 一个 token 地、贪婪地往前续**——像一个永远只能看着已经写出来的字、去猜下一个字的人。

形式化一点：模型建模的是 `P(x_t | x_1, x_2, ..., x_{t-1})`。整句话的概率是每一步条件概率连乘。训练目标极其朴素——**最大化语料里"下一个词"的预测准确率**（next-token prediction）。就这么一个目标，喂足够多数据、堆足够大参数，"涌现"出了翻译、推理、写代码的能力。

> 新手第一误区：以为模型"理解"了你的问题然后"查"了答案。**它没有"查"这个动作。** 它是在"续写"——你的问题是一段前缀，它在生成"这段前缀后面最可能出现的文字"。答案对，是因为"正确答案"恰好是高概率延续；答案错（幻觉），是因为"听起来对的胡话"也是高概率延续。模型分不清这两者，因为对它而言**两者是同一个动作**。

### 2. 演进：为什么是这个范式赢了

- **RNN/LSTM 时代**：也是自回归，但靠循环结构传递状态，长距离依赖会"遗忘"，且无法并行训练（第 t 步必须等第 t-1 步算完）。
- **2017 Transformer / self-attention**：用注意力机制让每个位置直接"看到"序列里所有其它位置，长依赖不再衰减，且**训练时可全序列并行**（这是能 scale 到千亿参数的工程前提）。
- **关键认知**：Transformer 解决的是"训练并行 + 长依赖"，但**生成（推理）时仍然是严格串行自回归的**——必须先有第 t 个 token 才能算第 t+1 个。这个"训练能并行、生成必串行"的不对称，直接决定了下面要讲的延迟模型。记死它。

### 3. 四个工程推论（本章地基，逐条都是面试题）

这四条是从"自回归"这一个事实里推出来的，是你区分"会调 API 的人"和"懂 LLM 的人"的分水岭。

#### 推论一：概率延续 ≠ 事实查询 → 幻觉是机制内生的，不是 bug

模型优化的是"流畅、像样的延续"，不是"事实正确"。当它没见过或记不清某事实时，它不会停下来说"我不知道"——因为"编一个听起来合理的答案"在概率上比"承认不知道"更像训练语料里的延续（训练语料里很少有人写"我不知道"）。

**所以幻觉不是哪个工程师写错了代码，是这个范式的副产品。** 你无法靠"把模型调好"消除幻觉，只能靠**外部架构**对冲：给它真相源（RAG，见第 7-11 章）、给它工具去查（见第 5 章）、给它 eval 去量化（见第 17 章）。

> 面试加分点：被问"怎么消除幻觉"，错误答案是"用更大的模型 / 调 temperature 到 0"。正确答案是"**幻觉不可消除只能对冲**，手段是 grounding（RAG/工具）+ 约束（结构化输出）+ 检测（eval/引用核对）"，并能说清为什么——因为它源于训练目标本身。

#### 推论二：prompt = 在概率空间里定位 → prompt engineering 的本质

既然模型是"给定前缀续写"，那 **prompt 的作用就是把模型的输出分布，挪到你想要的那块区域**。"你是一个资深 Rust 工程师"不是在赋予它能力，是在**把延续锚定到"资深 Rust 工程师会怎么说话"那片高概率区域**。few-shot 例子、角色设定、输出格式约束，全是同一回事——**用前缀给概率分布做条件（conditioning）**。

这解释了一堆现象：为什么换个说法效果差很多（你挪到了分布的另一块）、为什么给例子有用（你把分布收窄到例子那个样式）、为什么"think step by step"能提升推理（你把延续引导到"展开推理过程"那条轨迹，而展开的中间步骤又成为后续 token 的条件，自我增强）。

> 心法：**写 prompt 不是"下命令"，是"布置一个让正确答案成为最高概率延续的上下文"。** 你不是在指挥模型，你是在操纵它的概率分布。

#### 推论三：延迟天然两段式 → TTFT + decode，性能优化的全部战场

自回归串行生成，决定了延迟由两段构成，**这两段的优化手段完全不同**，是性能调优的核心地图：

| 阶段 | 名称 | 在算什么 | 耗时随什么涨 | 优化手段 |
|---|---|---|---|---|
| 第一段 | **TTFT**（Time To First Token，首 token 延迟） | prefill：把整个输入 prompt 并行编码一遍 | 随**输入长度**涨 | prompt caching（缓存前缀）、缩短输入、prompt 预热 |
| 第二段 | **decode**（逐 token 生成） | 一个一个生成输出 token，每个都要重算 | 随**输出长度**线性涨 | streaming（边生成边返回）、减少输出量、用更快的模型 |

工程含义极其具体：
- **输入长但输出短**（如长文档摘要）→ 瓶颈在 TTFT → 砸 prompt caching。
- **输出长**（如生成长报告、agent 多步）→ 瓶颈在 decode → **必须 streaming**，否则用户干等几十秒，且容易撞 HTTP 超时。
- **"为什么我加了一段超长 system prompt，第一个字出来就慢了？"** → 因为 prefill 要把这段也编码进去，TTFT 涨了。

> 坑点提示：很多人以为"延迟 = 模型慢"，一刀切。真相是**输入侧涨 TTFT、输出侧涨 decode**，两段分开治。面试被问"你的 LLM 接口 P99 延迟高，怎么排查"，能拆成这两段并对症给手段，直接甩开一片人。

> 工程事实（按 2026 初，Anthropic SDK）：对**可能长输入、长输出、或高 `max_tokens`** 的请求，默认就该开 streaming——SDK 甚至会对预估超时（约 10 分钟）的非流式请求直接抛错，逼你流式。拿到完整结果用 `.get_final_message()` / `.finalMessage()`，不必手撸事件循环。这就是推论三在 SDK 层的固化。

#### 推论四：模型不知道自己不知道 → 置信度不可信，需要外部校验

人类不确定时会犹豫、会说"大概"。模型不会——它对"2+2=4"和"某虚构 API 的参数"可以用**完全一样的流畅自信**输出，因为对它而言都只是"高概率延续"。模型**没有可靠的内省机制**去区分"我确定"和"我在编"。

它输出的概率（logprobs）是"这个 token 在语言上有多顺"，**不等于"这个事实有多真"**。一句胡话可以每个 token 的 logprob 都很高（因为它语法通顺、用词地道）。

**所以你不能让模型自己判断自己对不对**（"你确定吗？"经常只会让它换个同样自信的答案，或谄媚地改成你暗示的方向）。可靠性必须来自**外部**：检索到的原文、工具返回的真实数据、另一个模型/规则的交叉验证、人审。

> 面试官想区分什么：你是否理解"LLM 的自信 ≠ 正确性"。能不能说出"模型的 confidence 是语言流畅度不是事实可靠度，所以 self-verification 不可靠，校验必须外置"——这是 agent 可靠性设计的认知前提。

## 二、Token 与 BPE：你以为模型读的是字，其实是碎片

### 1. 背景：模型不读字符，读 token

模型的输入输出单位不是"字符"也不是"单词"，是 **token**——介于两者之间的"子词单元"。主流分词用 **BPE（Byte Pair Encoding，字节对编码）** 或其变体：从字节开始，统计语料里最高频的相邻对，反复合并成更大的单元，最终形成一个固定大小的词表（vocabulary，通常几万到十几万个 token）。

直觉对照（英文，量级随分词器变）：
- 常见英文词（`the`, `apple`）→ 往往 **1 个 token**。
- 生僻或拼接词（`unbelievable`）→ 可能拆成 `un` + `believ` + `able` 多个 token。
- 经验值：**英文 ~4 个字符 / token，1 个 token ≈ 0.75 个英文单词**。
- 中文：常见字 1 字 ≈ 1～2 token，生僻字/词更碎。**同样信息量，中文通常比英文吃更多 token**——这直接是成本问题。

### 2. 这套机制制造的三个真实工程坑

#### 坑一：中文成本天然更高

因为 BPE 词表主要用英文语料训练，中文被切得更碎，**同一段意思中文的 token 数常显著高于英文**。你做中文应用，输入输出成本、以及"撑爆上下文窗口"的速度，都比英文 demo 给你的直觉更糟。预算和窗口规划必须按中文实测，别拿英文教程的数字套。

#### 坑二：模型"数不对字母" / 不会数 `r`

"strawberry 里有几个 r" 这种经典翻车，根因就在 token：模型看到的 `strawberry` 可能是 `straw` + `berry` 两个 token，**它根本看不到独立的字符 `r`**。让一个看不到字母的东西数字母，当然错。同理：反转字符串、严格字符计数、做字符级编辑，都是 token 化的天然短板。

> 工程含义：需要字符级精确操作时，**别让模型徒手做，给它工具**（让它写代码去 `count`/`reverse`，见第 5 章）。这不是模型笨，是输入表示决定的。

#### 坑三：跨 provider 的 token 计数不统一，且不能用 tiktoken 估 Claude

**每家的分词器不一样**，同一段文本在 GPT、Claude、Gemini 上的 token 数都不同。你不能拿一家的数字套另一家。

特别地（按 2026 初，重要工程事实）：
- **没有公开的 Claude tokenizer。** 想精确数 Claude 的 token，必须调它的 **`count_tokens` API**。
- **用 OpenAI 的 `tiktoken` 去估 Claude 会系统性偏差 15-20%**（代码/非英文更糟）。任何拿 `tiktoken`、`gpt-tokenizer` 估 Claude 的成本/窗口数字都是错的。

> 落地（写进 P0 的成本核算）：要精确就调对应 provider 的计数 API；要粗估就用经验比例（英文 4 字符/token）并**标注这是估算**。**别跨家套用，别用 tiktoken 估 Claude。**

### 3. 工程推论

1. **成本与窗口都按 token 算，不按字数算**，且按你的目标语言实测——中文尤其要警惕。
2. **字符级任务给工具**，不靠模型裸算。
3. **token 计数是 provider-specific 的**，做多 provider（网关、fallback）时，成本估算和窗口检查要分别按各家的分词器来。

## 三、上下文窗口与无状态：每一轮对话都是"重新认识你"

### 1. 背景：两个必须焊死在脑子里的事实

**事实 A：模型是无状态的（stateless）。** API 在 HTTP 层是无状态的——服务端**不记得**你上一次说了什么。所谓"多轮对话"，是**你每次把完整历史重新发过去**。模型"记得"你叫 Alice，不是因为它存了，是因为这次请求的 messages 里又带上了"我叫 Alice"。

**事实 B：上下文窗口（context window）有上限。** 模型一次能处理的 token 总数（输入 + 输出）有硬上限。按 2026 初的量级：旗舰模型普遍做到 **1M（一百万）token** 级别（如 Claude Opus 4.8 / Sonnet 4.6 标称 1M），更小/更快的档位可能是 200K（如 Haiku 4.5）。**这个数字逐代在涨，别背具体值，记"有上限且是硬约束"。**

这两条合起来推出一个反直觉的成本结构。

### 2. 多轮对话的 O(n²) 成本陷阱

因为每轮都要把全部历史重发，对话越长，**每一轮的输入 token 越多**：

- 第 1 轮：发 1 条消息。
- 第 2 轮：发 2 条（历史 1 + 新 1）。
- 第 n 轮：发 n 条。

总 token 消耗 ≈ 1 + 2 + ... + n = **O(n²)**。一段 50 轮的长对话，光是把历史反复重发，成本就像滚雪球。

> 工程含义（直接进 P0 验收）：长对话必须有**历史管理策略**——
> - **滑动窗口**：只保留最近 K 轮。
> - **摘要压缩**：把早期历史用模型自己总结成一段，替换原文。
> - **撞窗降级**：接近窗口上限时主动截断/摘要，**绝不能直接报错崩**。
>
> 按 2026 初，部分模型/SDK 提供服务端 **compaction**（自动把早期上下文摘要成压缩块）——但理解原理后用，而且注意：用了 compaction 要把返回的压缩块**原样带回**下一次请求，否则压缩状态会悄悄丢失。

#### 必须补的一刀：prompt caching 把 O(n²) 的"钱"砍掉一大半（但砍不掉"token 占用"）

上面的 O(n²) 是**朴素重发**的成本。生产里第一个对冲手段不是滑动窗口，是 **prompt caching**——这是区分"读过教程"和"上过线"的分水岭，很多人讲完 O(n²) 就停了，恰恰漏了真实世界里最大的那根杠杆。

机制要点（按 2026 初，以 Anthropic 为锚，各家类似）：

- **前缀匹配缓存**：把稳定不变的前缀（system prompt、早期对话历史）打上缓存标记，下次请求只要**前缀逐字节一致**，这段就走缓存。
- **价格不对称（第二层不对称，记死）**：**缓存命中读 ≈ 0.1× 输入价，缓存写 ≈ 1.25× 输入价**（5 分钟 TTL）。也就是说，多轮对话里反复重发的那段历史，第二轮起基本按**一折**计费。
- **效果**：O(n²) 的**钱**被压扁——重发的历史 token 数没变（还是 O(n²) 个 token 在占窗口），但**单价**从 1× 掉到 0.1×。**省的是钱，不是窗口**。窗口该撞还是撞，所以滑动窗口/摘要/compaction 一个都不能少。

> 致命暗坑（面试 + 实战双杀）：缓存是**前缀逐字节匹配**，前缀里**任何一个字节变化都会让其后全部失效**。最常见的静默击穿：在 system prompt 里塞 `当前时间：{now()}`、塞 UUID、塞每次都变的 session id——你以为开了缓存，`cache_read_input_tokens` 却一直是 0，钱白花。**稳定内容放前面、易变内容放后面**，是 prompt 工程的硬约束。验证方式：读响应里的 `usage.cache_read_input_tokens`，为 0 就是没命中。

> 面试加分点：能说出"多轮对话朴素成本是 O(n²)（无状态导致历史每轮重发），但 prompt caching 让重发段按 ~0.1× 计费，把成本曲线压扁；不过缓存只省钱不省窗口，撞窗仍需滑动窗口/摘要/compaction"——这一整条链路讲下来，直接甩开只会背"O(n²)"的人。能再补一句"缓存是前缀逐字节匹配，system prompt 里塞时间戳会静默击穿"，是上过生产的标志。

### 3. 长上下文 vs RAG：四维对比（高频架构题）

既然窗口能到 1M 了，一个自然的问题：**"我直接把所有文档全塞进 prompt 不就行了，还要 RAG 干嘛？"** 这是面试高频陷阱题。答"窗口够大就不用 RAG"会被判死。正确姿势是**四维权衡**：

| 维度 | 长上下文（全塞进窗口） | RAG（检索增强） |
|---|---|---|
| **成本** | 每次请求都为全部内容付 token 费，O(n) 且 n 巨大 | 只为检索出的 top-k 片段付费，n 小且稳定 |
| **延迟** | 输入巨大 → TTFT 高（prefill 慢） | 输入小 → TTFT 低 |
| **准确性** | 受 **lost-in-the-middle** 影响，长文中段信息易被忽略；**有效长度 < 标称长度** | 把最相关片段顶到显眼位置，命中率更可控 |
| **可更新 / 可溯源** | 知识固定在这次 prompt 里；要更新得重塞；引用来源难 | 知识在外部库，可随时增删改；**天然带 citation** |

> 心法：**窗口大小是"能装多少"，不是"装满了就用得好"。** RAG 解决的不只是"装不下"，更是**成本、延迟、注意力有效性、可更新性**。长上下文和 RAG 不是二选一，生产里常常**组合**：RAG 把范围缩小，再用长上下文容纳检索结果。（RAG 细节见第 7-11 章。）

### 4. Lost-in-the-middle 与"有效长度"

这是长上下文最阴险的坑：**信息放在 prompt 的开头和结尾，模型用得好；放在中间，经常被"忽略"。** 召回曲线呈 U 形——两端高、中间塌。

后果：模型的**有效上下文长度，往往明显小于它标称的窗口长度**。一个标称 1M 的模型，你真把关键信息埋在第 50 万 token 处，它可能视而不见。

工程对策：
- **关键指令/关键信息放两端**——重要约束放 system prompt 开头或贴近末尾的位置。
- **别迷信"反正窗口大，全塞进去"**——位置比容量更影响效果。
- **RAG + 重排（rerank）** 的价值之一就是把最相关的内容顶到模型"看得见"的位置（见第 9 章）。

> 工程推论汇总：
> 1. **无状态 → 多轮历史自管理 + O(n²) 成本意识**。
> 2. **窗口有上限且是硬约束 → 撞窗要降级不能崩**。
> 3. **长上下文 vs RAG 是四维权衡（成本/延迟/准确/可更新），不是"窗口够大就不用 RAG"**。
> 4. **有效长度 < 标称长度（lost-in-the-middle）→ 关键信息放两端，别埋中间**。

## 四、采样：temperature / top_p，以及"参数被收回"的趋势

### 1. 背景：从概率分布到一个具体 token

模型每一步算出的是**整个词表上的概率分布**。怎么从分布里挑一个 token 出来，就是**采样（sampling）**。两个经典旋钮：

- **temperature（温度）**：缩放概率分布的"尖锐度"。
  - **低温（→0）**：分布变尖，几乎总选最高概率那个 token → 输出**确定、保守、可复现性高**（但仍不保证逐字一致）。
  - **高温（如 0.8-1.0+）**：分布变平，低概率 token 也有机会被选 → 输出**多样、有创意、也更容易跑偏/胡说**。
- **top_p（nucleus sampling，核采样）**：只在"累计概率达到 p 的最小 token 集合"里采样。`top_p=0.9` = 只从最可能的、加起来占 90% 概率的那批 token 里选，砍掉长尾。
- （还有 **top_k**：只保留概率最高的 k 个候选。同类思路。）

经验选型（按 2026 初，作锚点，会随模型变）：
- **要稳定、要结构化输出、要可复现**（分类、抽取、代码）→ **低温**（接近 0）。
- **要创意、要多样**（头脑风暴、文案、给设计出多个方向）→ **较高温 + top_p**。
- 通常 **temperature 和 top_p 调一个就够**，别俩一起猛拧（有些模型同时传会冲突甚至报错）。

> 新手第一误区：以为 "temperature=0 = 完全确定 = 每次逐字一样"。**不保证。** 低温让它**倾向**选最高概率 token，但浮点实现、并行、后端差异都可能让结果有微小波动。"确定性"在 LLM 里是**程度**，不是开关。要真正可复现，靠的是"低温 + 把 prompt 焊死 + 接受仍有微抖动"，不是某个魔法参数。

### 2. 现状与趋势：采样超参正在被"收回"

这是 2026 的一个**重要风向**，面试能体现你跟得上前沿：

**头部模型正在把采样超参从用户手里收走，改用"自然语言 + 思考预算"来控制行为。**

具体事实（按 2026 初，Anthropic）：
- **Opus 4.7 / 4.8 与 Fable 5 已移除 `temperature` / `top_p` / `top_k`**——你再传这些参数会直接 **400 报错**。
- 同时移除了手动思考预算 `budget_tokens`，改用 **adaptive thinking**（`thinking: {type: "adaptive"}`，模型自己决定想多深）。
- 思考/花费的深浅改用 **effort 档位**（`low` / `medium` / `high` / `xhigh` / `max`）来控。

**怎么解读这个趋势**（这才是考点，不是背 API）：

> 行为控制的主通道，正在从"调采样超参"迁移到"**自然语言指令 + 思考预算**"。

含义：
1. **控制模型行为的方式在抽象层上移**——从拧 `temperature=0.7` 这种底层旋钮，变成"在 prompt 里说清楚你要什么风格/确定性" + "给它合适的思考 effort"。
2. 旧教程里"调低 temperature 求稳定"的肌肉记忆，在新模型上**可能直接报错**。要稳定，改用**清晰的 prompt 约束 + 结构化输出 + 合适 effort**。
3. 这对应用工程师是**好事**：少了一堆玄学旋钮，多了"用人话描述意图"这个更稳的接口。但你必须知道**你的目标模型属于哪一代**——老模型还吃 `temperature`，新旗舰已经不吃了。

#### 你真正要的"稳定"，往往不是 temperature 给得了的

这里有个新手和老手都容易混的概念，必须拆开：

| 你想要的"稳定" | 含义 | 靠什么实现 | temperature 能不能给 |
|---|---|---|---|
| **内容确定性** | 同样输入每次输出**逐字一样** | 焊死 prompt + 低温/无采样，**且接受仍有微抖动** | 给不了"逐字一样"，只能"倾向一致" |
| **结构确定性** | 输出**一定是合法 JSON / 一定符合某 schema** | **结构化输出**（schema 约束解码） | **完全给不了**——低温只让它"更可能"输出合法 JSON，不保证 |

关键洞察：**工程上你 90% 想要的其实是"结构确定性"**（下游要 parse，不能崩），而这从来不是 temperature 的活儿。

所以新旗舰把 temperature 收走，对"求稳定"这个诉求**几乎没损失**——因为真正解决结构稳定的是另一条路：让模型的输出被**约束在一个 schema 里解码**。按 2026 初，Anthropic 用 `output_config` 的 `format`（json_schema）约束响应格式，工具参数用 `strict: true` 做校验，SDK 还有 `messages.parse()` 直接按 schema 校验并反序列化。这类机制是**解码层的硬约束**——不是"求模型配合"，是"从生成机制上堵死非法输出"，比任何 temperature 设置都可靠。

> 心法：**"我要稳定" → 先问自己要的是内容稳定还是结构稳定。** 要结构稳定，用结构化输出（schema 约束），别碰 temperature；要内容尽量一致，焊 prompt + 低温（或新旗舰直接靠 prompt），并接受永远有微抖动。把这两件事分开，是从"调参玄学"走向"工程确定性"的关键。

> 工程推论：
> 1. **采样参数 temperature/top_p 控制"确定 ↔ 多样"，调一个即可**。
> 2. **temperature=0 ≠ 逐字可复现**，确定性是程度不是开关。
> 3. **"结构确定性"靠结构化输出（schema 约束解码），不靠 temperature**——下游要 parse 时尤其要用。
> 4. **新一代旗舰正在移除采样超参（传了报 400），行为控制迁移到自然语言 + effort 档位 + 结构化输出**——写代码前先确认目标模型这一代的请求接口长什么样。

## 五、训练管线四段：模型的每个"性格"都有出处

要解释"模型为什么有知识截止""为什么听话""为什么谄媚""为什么会推理"，得看它怎么训出来的。现代 LLM 训练大致四段，**每一段决定模型的一类行为**——这张"行为溯源表"是面试高频考点。

| 阶段 | 全称 | 在干什么 | 决定了模型的什么行为 |
|---|---|---|---|
| 1 | **Pretrain**（预训练） | 在海量文本上做 next-token 预测，学语言、世界知识、模式 | **知识与能力的底座**；也带来**知识截止日期**（训练语料截止那天之后的事它不知道） |
| 2 | **SFT**（Supervised Fine-Tuning，监督微调） | 用"指令-理想回答"对子教它**听懂指令、按格式回答** | **听话 / instruction-following**——从"只会续写"变成"会回答你的问题" |
| 3 | **RLHF**（基于人类反馈的强化学习） | 用人类偏好训一个 reward model，再用 RL 让输出对齐人类喜好 | **对齐人类偏好**（有用、无害、礼貌）；**副作用：谄媚（sycophancy）**——因为人类标注者倾向给"顺着我说"的回答打高分，模型学会了迎合 |
| 4 | **Reasoning RL**（推理强化学习） | 针对可验证任务（数学/代码）用 RL 奖励**正确的推理过程**，催生 long chain-of-thought | **推理能力 / 思考**——会展开多步推导再答（2025 后这一段成为旗舰标配，DeepSeek R1 是开源证明此路可行的分水岭） |

### 几个由此推出的关键认知

**知识截止（knowledge cutoff）来自 pretrain。** 模型不知道训练之后发生的事，这不是 bug，是预训练语料的时间边界。要让它知道新信息，靠 **RAG / 工具 / 联网搜索**喂进上下文（见第 5、7-11 章），**不是**靠"它应该知道"。

**谄媚（sycophancy）是 RLHF 的副作用，不是模型"虚伪"。** 它学到"顺着用户 = 高奖励"。所以——
- "你确定吗？"经常让它动摇、改成你暗示的答案，**哪怕原答案是对的**。
- 校验绝不能靠"反问模型自己"，必须外置（呼应推论四）。
- 设计 agent 自检环节时，要警惕谄媚污染——别用"你检查一下你刚才对不对"这种问法。

**推理能力来自专门的 reasoning RL，且可以用 effort/思考预算调深浅。** 这就是为什么新模型有"想得深 vs 想得浅"的档位——本质是给那条 long-CoT 轨迹分配多少 token。

> 工程推论：
> 1. **知识截止 → 时效性问题一律外接真相源**（RAG/工具/搜索），别赌模型"应该知道"。
> 2. **谄媚是 RLHF 内生副作用 → 自检/校验必须外置，别问模型"你确定吗"**。
> 3. **听话来自 SFT、对齐来自 RLHF、推理来自 reasoning RL**——能把一个行为追溯到它的训练来源，是"懂模型"而非"会用模型"的标志。

## 六、2026 模型版图与选型方法论（重方法论，名单会过期）

> 强时效警告：下面的名单、价格、档位按 **2026 初**写，**按季度过期**。背名单没用，**学的是"怎么选"这套方法论**——名单换了，方法论不变。

### 1. 三梯队心智模型（这个分层结构比具体名字稳定）

| 梯队 | 定位 | 代表（2026 初，会变） | 价格量级（每百万 token，作锚点） |
|---|---|---|---|
| **Frontier（旗舰）** | 最强智能，难任务、长链路 agent、关键推理 | GPT-5 系、Claude Opus 4.8 / Fable 5、Gemini 3 级 | 输入 $5-10 / 输出 $25-50 |
| **Balanced（中杯）** | 速度与智能的平衡，高量产线上主力 | Claude Sonnet 4.6 等 | 输入 ~$3 / 输出 ~$15 |
| **Fast（小杯）** | 最快最便宜，简单/高频/延迟敏感任务 | Haiku 4.5、各家 mini/flash | 输入 ~$1 / 输出 ~$5 |

**一个必须记死的价格不对称：输出价 ≈ 输入价的 5 倍。** （旗舰输入 $5-10、输出 $25-50；中杯 $3 vs $15；小杯 $1 vs $5——全是约 5×。）

> 工程含义：**省钱的第一杠杆往往是"砍输出 token"**，不是砍输入。让模型少废话、用结构化输出、别让它复述输入——这些直接砍的是那个 5× 的贵头。能把"为什么优先压输出"讲清楚，是成本治理的基本功（见第 16 章）。

### 2. 开源 vs 闭源（2026 格局）

- **开源/开放权重**：Llama、**Qwen**（阿里，开源声量最大）、**DeepSeek V3/R1**（2025 初证明开源推理能追平闭源，**分水岭事件**）、Mistral。优势：可私有部署、数据不出域、可微调、长期成本可控。
- **国内闭源**：豆包（字节）/ Kimi（月之暗面）/ GLM（智谱）/ 混元（腾讯）/ 文心（百度）。
- **选型不是"开源就是省"**：开源省的是 API 调用费，但要算上**自建推理的 GPU、运维、SRE 成本**。小规模/起步期，闭源 API 通常更省心更便宜；规模化、强合规、要深度定制时，私有化开源才显出优势。

### 3. 选型方法论：四步（这才是要背的东西）

面对"这个功能用哪个模型"，按四步走，**别凭感觉拍 Frontier**：

1. **定任务难度与类型。** 是分类/抽取（简单）还是多步推理/复杂 agent（难）？是否要超长上下文、视觉、工具？—— 先用**能力下限**圈出候选梯队。
2. **定约束。** 延迟要求（实时对话 vs 离线批处理）？成本预算（每请求/每月）？合规（数据能否出域 → 决定开源私有化 vs 闭源 API）？
3. **从满足约束的最低梯队往上试，而非从最贵往下砍。** 默认**先用够用的最便宜档**（很多任务 Fast/Balanced 就够），只有当它质量不达标，才往上升级到 Frontier。**不要默认 Frontier**——那是烧钱。
4. **用你自己的 eval 集验证，再决定。** 别信榜单和宣传，用**你的真实任务**（≥3 个独立样本，避免 N=1 的运气）跑评测，看质量/成本/延迟三个数。eval 是选型的最终裁判（见第 17 章）。

> 心法：**模型选型不是"选最强的"，是"选满足约束的最便宜的"，并用自己的 eval 拍板。** 能力是别人的，选型判断是你的。

> 一个反模式（写进简历的反面教材）：整个系统一律上旗舰。正解是**分层路由**——简单子任务（意图分类、格式化）走 Fast，复杂推理走 Frontier，用网关做模型路由（见第 P4/第 16 章）。这是"会调 API"和"会做系统"的差距。

## 七、幻觉三类型与可靠性工程栈

幻觉不是一坨笼统的"模型瞎说"。**分三类，对策完全不同**——能分类是上来就甩开一片人的认知。

### 1. 三类型对照表

| 类型 | 定义 | 典型例子 | 主要对策 | 对应章节 |
|---|---|---|---|---|
| **事实性幻觉**（factuality） | 与**客观世界事实**不符 | 编造一个不存在的 API、记错历史日期、虚构论文引用 | **grounding**：RAG 给真相源、工具去查、联网搜索 | RAG 第 7-11 章、工具第 5 章 |
| **忠实性幻觉**（faithfulness） | 与**给定的上下文/原文**不符 | RAG 把文档喂进去了，模型却答了文档里没有的、甚至矛盾的内容 | 强约束 prompt、引用核对（让答案必须能映射回原文）、结构化输出 | RAG 第 9 章、eval 第 17 章 |
| **推理性幻觉**（reasoning） | 推理链条**逻辑错误**导致结论错 | 数学算错、多步推导中间跳错、因果倒置 | reasoning 模型 / 高 effort、让它展开步骤、工具验证（让它写代码算）、self-consistency | 工具第 5 章、agent 章 |

> 面试官想区分什么：你能不能把"幻觉"拆成这三类并对症下药。被问"你的 RAG 还是会瞎编怎么办"，要先反问/判断**是哪类**——
> - 答了库里没有的 → **忠实性**问题，是检索没召回 + 约束不够，治 prompt 和检索。
> - 库里有但答错事实 → 可能检索片段本身就错（**事实性**源头在数据），治数据源。
> - 逻辑推错 → **推理性**，升 effort / 给工具。
>
> 一句"看是哪类幻觉"就把你和"无脑加 prompt"的人区分开了。

### 2. 可靠性工程栈：error budget 心法

把这章所有"对冲手段"串成一个栈，对应你后面要建的整套系统：

```
        [ 人审 / 兜底 ]            ← 最后防线（高风险动作必须有）
        [ eval 量化 ]             ← 没有量化就是裸奔（第 17 章）
        [ tracing 可观测 ]        ← 出事能下钻到每一步（第 16 章）
        [ 结构化输出 / 约束 ]      ← 把输出钉进 schema（第 5、17 章）
        [ grounding: RAG / 工具 ] ← 给真相源，对冲事实性/忠实性幻觉（第 5、7-11 章）
   ────────────────────────────
        [ 不可靠的 LLM 本体 ]      ← 概率延续、无状态、会胡说、不知道自己不知道
```

最关键的锚点类比（贯穿全课）：

> **eval 之于 AI 系统 = error budget 之于 SRE。**

SRE 不追求"零故障"，追求"把故障控制在可量化的预算内"。AI 系统同理：**你消除不了幻觉（物理规律），只能把它的发生率压到可量化、可接受的预算内，并持续监控。** 没有 eval，你改个 prompt 根本不知道是变好还是变坏，只能"凭感觉"——而**感觉在生产里等于裸奔**。

> 收尾心法（本章的一句话）：**LLM 是一个不可靠、无状态、按 token 收费、会自信胡说的概率续写器；你的全部工程价值，就是在它外面造一层让它变得可靠的东西。** 这章讲清了"它不可靠在哪"，后面整门课讲"怎么把可靠性造出来"。

## 章末五件套

### A. 高频面试题（11 道，带答题框架与陷阱）

**1. 解释自回归生成，并说明它为什么导致幻觉。**
- 框架：定义 next-token 概率延续 → 优化目标是"流畅延续"非"事实正确" → 模型分不清"正确答案"和"听起来对的胡话"（对它是同一个动作）→ 故幻觉是机制内生。
- 陷阱：别答成"模型理解后查答案"。它**没有"查"这个动作**。

**2. "多轮对话"在 HTTP 层到底发生了什么？为什么成本是 O(n²)？**
- 框架：模型无状态 → 每轮把全部历史重发 → 第 n 轮发 n 条 → 总量 1+2+…+n = O(n²)。
- 陷阱：答"对话越长越贵"不给量级 = 没答到点。必须说出**平方级**和**无状态**这个根因，并能给滑动窗口/摘要两种解。

**3. 窗口已经 1M 了，为什么还需要 RAG？**
- 框架：四维权衡——成本（全塞是 O(n) 巨大）、延迟（输入大→TTFT 高）、准确（lost-in-the-middle，有效长度<标称）、可更新可溯源（RAG 库可增删改 + 带 citation）。
- 陷阱：答"窗口够大就不用 RAG"直接判死。要点出**生产常组合使用**。

**4. temperature=0 能保证输出完全一致吗？**
- 框架：不能。低温让它**倾向**选最高概率 token，但浮点/并行/后端差异仍致微抖动。确定性是程度不是开关。要可复现靠"低温+焊死 prompt+接受微抖"。
- 陷阱：斩钉截铁说"能，完全确定"是错的。

**5. 模型为什么"数不对 strawberry 里的 r"？**
- 框架：token 化——模型看到的是 `straw`+`berry` 等子词，**看不到独立字符**。字符级任务是 token 表示的天然短板。
- 陷阱：别答"模型笨"。根因是**输入表示**，对策是**给工具让它写代码数**。

**6. 模型的 logprob / confidence 能当"答案可信度"用吗？**
- 框架：不能。logprob 是"token 在语言上多顺"，不是"事实多真"。一句胡话可以每个 token 都高 logprob。故 self-verification 不可靠，校验须外置。
- 陷阱：把语言流畅度当事实可靠度，是 agent 设计的致命误解。

**7. 谄媚（sycophancy）从哪来？对 agent 设计有什么影响？**
- 框架：RLHF 副作用——人类标注偏好"顺着说"的回答 → 模型学会迎合。影响：自检不能问"你确定吗"（会动摇甚至改对为错），校验必须外置。
- 陷阱：答"模型虚伪"是拟人化错误，要归因到**训练机制**。

**8. 知识截止是什么？怎么让模型回答训练之后的事？**
- 框架：来自 pretrain 语料的时间边界。对策：RAG / 工具 / 联网搜索把新信息喂进上下文。
- 陷阱：别说"换个更新的模型就行"——根本对策是**外接真相源**，模型再新也有截止线。

**9. 给一个"内部合同审查"功能选模型，你怎么选？**
- 框架：四步法——①定难度（合同推理偏难，要长上下文）②定约束（合同是敏感数据→数据能否出域决定开源私有化 vs 闭源 API；延迟要求？预算？）③从满足约束的最低梯队往上试，别默认 Frontier ④用自己标注的 eval 集（≥3 样本）拍板。
- 陷阱：上来就说"用最强的 Opus/GPT-5"——没考虑合规和成本，判断力不及格。

**10.（前沿题）听说新模型把 temperature 拿掉了，这意味着什么趋势？**
- 框架：行为控制主通道从"采样超参"迁移到"自然语言指令 + 思考预算（effort/adaptive thinking）"。控制抽象层上移。含义：旧的"调低温求稳"肌肉记忆在新模型上可能直接 400；求稳改用清晰 prompt 约束 + 结构化输出 + 合适 effort。
- 陷阱：不知道这个趋势，或还在新模型上传 temperature，暴露没跟前沿 / 没读目标模型的接口文档。

**11.（高频成本题）你的多轮对话 / 长 system prompt 应用账单暴涨，怎么把成本压下来？prompt caching 的原理和坑是什么？**
- 框架：① 先定位贵在哪——多轮是 O(n²) 历史重发，长 system 是每请求重复编码同一大前缀。② 主手段 **prompt caching**：把稳定前缀打缓存标记，**命中读 ~0.1× 输入价、写 ~1.25×**，重复段直接打一折。③ 配合**砍输出**（输出价≈输入 5×）、滑动窗口/摘要控历史长度。④ 用 `usage.cache_read_input_tokens` 验证命中率。
- 必答的坑：缓存是**前缀逐字节匹配**——system prompt 里塞时间戳 / UUID / 变化的 session id 会**静默击穿**（命中率掉 0，钱白花），所以**稳定内容放前、易变内容放后**；渲染顺序 tools→system→messages，前面任何字节变了，后面全失效。
- 必答的边界：**缓存只省钱不省窗口**——重发的历史 token 还在占上下文，撞窗仍要靠滑动窗口/摘要/compaction。
- 陷阱：只会答"开缓存"说不出 0.1×/1.25× 的不对称和前缀匹配机制；或把"省钱"误当成"省窗口"。

### C. 设计题（考权衡判断）

> 你要给一个**面向 C 端的 AI 客服**做对话历史管理。已知：用户会话可能很长（几十轮），窗口有上限，每轮历史重发是 O(n²) 成本，且 lost-in-the-middle 会让早期信息被忽略。请设计历史管理方案，并说清你的 trade-off。

考点：能否同时权衡**成本、窗口约束、信息有效性、实现复杂度**。好答案会提：滑动窗口 vs 摘要压缩 vs 两者混合（近 K 轮保原文 + 更早的摘要）；关键信息（用户身份、诉求）单独固定在 system/末尾避免被中段淹没；何时触发压缩（按 token 阈值）；摘要本身也花钱且可能丢信息这个代价。没有唯一答案，**有没有把多个矛盾维度摆上桌并下判断**才是评分点。

### D. 系统设计题（含量级估算）

> 设计一个**文档问答服务**，日活 1 万用户，人均每天 5 次提问，每次问答平均输入 2K token、输出 500 token。请估算：(1) 日 token 消耗与月成本量级；(2) 若把"全文档塞进 prompt"（每次额外 +50K token 输入）和"RAG（每次额外 +3K token 输入）"两方案对比，月成本差多少；(3) 给出你的架构选择及理由。

估算框架：
- 日请求数 = 1万 × 5 = 5万次。
- 基础日 token = 5万 ×（2K 输入 + 500 输出）= 1.25 亿 token/天。
- 全文档方案额外输入 = 5万 × 50K = **25 亿 token/天**；RAG 方案额外 = 5万 × 3K = 1.5 亿 token/天。**差约 23.5 亿 token/天**——按输入 ~$3/百万算，**全文档方案每天多烧约 $7000+**，月差 **20 万美元量级**。
- 结论方向：除非有特殊理由，**RAG 完胜**（成本、延迟、可更新）。能把这个量级算出来、并落到"所以选 RAG"的判断，就过了。

> 高级加分（能把面试官拉到对话状态）：上面的对比是**裸算**，但生产里全文档方案通常会叠 **prompt caching**——如果那 50K 文档对所有用户是同一份（公共知识库），它就是一个**稳定前缀**，第二个请求起按 ~0.1× 计费。那笔"每天多烧 $7000"会被压到 ~$700 量级，**全文档方案的成本劣势缩小一个数量级**。
>
> 但即便如此，RAG 仍然赢——而且赢点要换：
> - **缓存救得了成本，救不了"注意力有效性"**：50K 全塞进去，lost-in-the-middle 让中段信息照样被忽略，有效长度 < 标称长度，这是缓存解决不了的。
> - **缓存救不了"可更新/可溯源"**：文档一改，整个缓存前缀失效，且答案没有 citation。
> - **缓存救不了"窗口占用"**：50K 仍然占着窗口，挤压可用的对话/推理空间。
>
> 所以正确的最终判断不是"RAG 因为便宜所以赢"，而是"**即使用缓存抹平了大部分成本差，RAG 仍因注意力有效性 + 可更新 + 可溯源而胜出**"。能把"缓存改变成本结论、但改不了架构结论"这层讲清楚，是把"会算账"升级成"懂架构"的关键一跳——也是这道题真正的 senior 信号。注意别犯反向错误：**不要因为有缓存就改口说"那全塞也行"**，成本只是四个维度之一。
