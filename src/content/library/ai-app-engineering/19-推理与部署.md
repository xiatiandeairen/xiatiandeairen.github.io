>
> 替你解决的问题：把「自托管一个开源模型」这件听起来很硬核、很有技术含量的事，还原成一道**冷冰冰的单位经济学题**——每百万 token 自建成本 = GPU 小时单价 ÷ 该卡每小时能产出的 token 数，再和 API 价目表逐位比。同时把支撑这道题的底层引擎机制（continuous batching、PagedAttention、量化、speculative decoding、并行）讲透到你能判断「这台卡这个模型，吞吐到底卡在哪、还能不能榨」。建立一个贯穿全章的判断——**自建推理之于 API = 自建机房之于云**：绝大多数公司在绝大多数阶段，自建都是负 ROI 的伪需求；只有当规模、合规、定制三者之一真正把你逼到墙角，自建才从「秀肌肉」变成「省钱/保命」。
>

>
> 时效标注：vLLM / SGLang / TensorRT-LLM 的具体接口、GPU 型号和云上单价、各家模型的具体吞吐数字，按 **2026 初**写，**名单和价格一定会过期，重点学算账方法论和卡吞吐的物理瓶颈**；而 continuous batching 为什么能提吞吐、KV cache 为什么吃显存、量化为什么是精度换显存、prefill/decode 为什么一个算力受限一个带宽受限——这些是**推理的物理规律**，讲透到骨子里，不会变。
>
> 交叉引用：「自建 vs 买」的选型经济学，把第 1 章（LLM 物理学：为什么贵、为什么慢、输出比输入贵 5 倍的根因）的物理结论**落地成一张可执行的成本表**，本章是第 1 章那笔账的「生产兑现」；自建出来的推理服务怎么接进统一网关、怎么和 API 做混合路由与 fallback，在**第 20 章** LLM Gateway 展开；自建最硬的驱动力之一是数据合规（数据不能出境/不能过第三方），这条线连着**第 21 章**安全与合规——本章只讲「合规会逼你自建」这个判断，纵深防御在第 21 章。

## 一、背景：为什么「自建 vs API」是一道经济学题，不是技术题

### 1. 问题的根源：推理是 AI 系统真正的「电费」

训练一个大模型是一次性的资本开支（CAPEX），花完就完了。但**推理（inference）是持续的运营开支（OPEX）**——只要你的产品还在被人用，每一次对话、每一个 token，都在持续烧钱。一个跑了一年的 AI 产品，推理成本累计远超训练那一锤子，而且会随用户增长线性甚至超线性上涨。

所以「推理与部署」这一章的核心矛盾，不是「能不能跑起来」——开源模型 + vLLM，一个下午就能起一个服务——而是「**这笔持续的电费，走哪条路最省、最稳、最合规**」。摆在你面前就两条路：

- **买 API**：按 token 付费，OPEX 模式，零运维，弹性无限，但单价由别人定、数据要出门、深度定制受限。
- **自建**：买卡或租卡自己部署开源模型，CAPEX + 运维人力，单价理论上可以压到很低，但有规模门槛、有运维负担、有「卡买了却跑不满」的巨大浪费风险。

> 新手第一误区：把「自建推理」当成一个**技术挑战**去攻——研究怎么装 vLLM、怎么调 batch size、怎么上量化，然后部署成功了很有成就感。但**自建推理在 2026 年早就不是技术难题，它是一道商业决策题**。真正的高级工程师，第一反应不是「怎么部署」，而是「**按我们的量级，自建到底比 API 省钱吗？省下来的钱够不够付那两个 SRE 的工资 + 那批卡的折旧 + 跑不满的浪费？**」。能把这道题算清楚再决定动不动手，才是 P7 视角；上来就 `pip install vllm` 的，是把手段当成了目的。

### 2. 演进：从「只能自建」到「默认买 API」再到「按账分流」

- **阶段零（GPT-3 之前 / 早期开源）：想用就得自建。** 没有成熟的推理 API，要用模型只能自己部署，吞吐极低（早期没有 continuous batching，一张卡同时只能服务个位数请求），成本高得吓人。这个阶段「自建」是唯一选项，不是选择。
- **阶段一（API 商业化成熟）：默认买 API。** OpenAI / Anthropic 等把推理做成了水电一样的服务，按 token 计费、弹性无限、质量碾压自建小模型。绝大多数应用的正确默认值变成了「直接调 API，别自己碰 GPU」。这个阶段自建反而成了少数派、成了需要论证的选项。
- **阶段二（开源追平 + 推理引擎成熟）：自建重新进入视野。** 2025 年 DeepSeek V3/R1 证明开源推理能力可以追平闭源（分水岭事件，按 2026 初口径），同时 vLLM / SGLang / TensorRT-LLM 这批推理引擎把开源模型的吞吐拉高了一个数量级。自建的「质量天花板」和「成本地板」同时被改善，让「够大规模的玩家自建省钱」第一次成立。
- **阶段三（当前前沿）：混合路由 + 按账分流。** 成熟团队不再二选一，而是**按场景分流**：高频、简单、对延迟敏感、数据敏感的流量走自建小模型；低频、复杂、要顶级能力的走 Frontier API。一个统一网关（第 20 章）在前面做路由、做 fallback、做成本归集。这是把「自建 vs API」从单选题变成了**投资组合问题**。

> 锚点类比（给后端老手）：**自建推理之于 API = 自建机房（IDC）之于公有云**。你十年前就经历过这个决策：什么时候该上云、什么时候该自建机房？答案从来不是「云一定好」或「自建一定省」，而是**看规模和约束**——量级小、要弹性、没运维团队，上云；量级巨大且稳定、有合规要求、有专职团队，自建机房单位成本才打得过云。推理一模一样：API 就是云（弹性、省心、单价高），自建 GPU 就是 IDC（前期投入大、要养人、跑满了才划算）。**你过去判断「上云还是自建机房」的那套财务直觉，原封不动搬过来就能用。**

### 3. 三个工程推论（本章地基，逐条都是面试题）

#### 推论一：自建推理的成本拐点由「利用率」决定，不是由「单价」决定

很多人算账只算一笔：「API 每百万 token 5 块，自建理论上能压到 1 块，自建省 80%！」——**这个算法是错的，因为它假设了 100% 利用率**。自建的真实单位成本 = 卡的固定成本（折旧/租金） ÷ 实际产出的 token 数。如果你的卡只有 20% 时间在满负荷跑（白天高峰满、夜里空转、流量有波峰波谷），那分母缩水 5 倍，你那「1 块」立刻变成「5 块」，跟 API 打平甚至更贵——而且你还多背了运维和折旧风险。

> 面试加分点：被问「你们为什么不自建模型省钱」，初级答案是「自建太麻烦/我们量不够大」。高级答案是「**自建的盈亏平衡点不看单价看利用率**——我们算过，要让自建单位成本打过 API，那批 A100/H100 需要常年维持 60%+ 的利用率；但我们流量有明显波峰波谷，平摊下来利用率只有 25% 左右，自建单位成本反而高于 API，再加上两个 SRE 的人力和卡的折旧风险，ROI 是负的，所以继续买 API，把工程力投在网关和缓存上压成本」。能点出「利用率是分母、波峰波谷拉低有效利用率」这一层，说明你真算过账。

#### 推论二：推理优化的所有手段，本质都在回答一个问题——「怎么让一张卡的每秒 token 产出（throughput）更高 / 每个 token 的延迟（latency）更低」

continuous batching、PagedAttention、量化、speculative decoding、并行——这五大机制看起来各管各的，但**它们全都在优化同两个指标：吞吐（throughput，单位时间总产出，决定成本）和延迟（latency，单次响应快慢，决定体验）**。理解这一点，你就有了一把统一的尺子去看所有推理技术：任何一个新名词冒出来，先问「它是提吞吐还是降延迟？代价是什么（显存？精度？复杂度？）？」。

> 心法（贯穿本章）：**推理工程的全部，就是在『吞吐 ↔ 延迟 ↔ 成本 ↔ 质量』这个四角张力里找你这个业务的最优点。** 没有免费午餐——提吞吐通常牺牲单请求延迟，降延迟通常牺牲吞吐密度，省显存（量化）通常牺牲一点精度。**先想清楚你的业务对这四个角各自有多敏感，再选技术**，而不是看哪个技术 fancy 上哪个。

#### 推论三：决定「必须自建」的，往往不是成本，是合规和定制这两条「成本算不过来也得做」的硬约束

成本是「算得出来、可比较」的软约束——算下来自建省钱就自建，不省就买 API。但有两类约束是**「算不算得过来都不重要、不自建就出局」的硬约束**：①**数据合规**：数据不能出境、不能流经第三方（金融、医疗、政务、欧盟 GDPR 场景），这种情况下哪怕 API 更便宜你也不能用，只能私有化部署；②**深度定制**：你要改模型结构、要在推理层做特殊的 logits 干预、要跑一个 API 厂商根本不提供的微调/蒸馏后的私有权重——这些 API 给不了，只能自建。**遇到这两条，「自建 vs API」的成本对比直接失效，自建是唯一解**（合规细节见第 21 章）。

## 二、自建 vs API 的经济学：把账算到每百万 token

这是本章的判断核心。先把方法论钉死，再代入 2026 初的数字给你一个量级感。

### 1. 算账方法论：四步把「自建到底省不省」算出来

**第一步：算 API 这边的单位成本。** 直接查目标模型的价目表，注意输入价和输出价不对称（输出约为输入的 5 倍，根因见第 1 章：输出是自回归逐 token 生成、无法并行，输入是一次性 prefill）。按你的实际输入/输出比例加权，得到「每百万 token 综合单价」。

**第二步：算自建这边的「卡固定成本/小时」。** 自建你要么买卡（CAPEX，按 3 年折旧摊成每小时）要么租卡（云上 GPU 实例，直接有每小时单价）。把卡价 + 机房/电力/带宽 + 运维人力分摊，得到「**这张卡每小时的全成本**」（不是裸卡价，要含周边和人力）。

**第三步：测自建这张卡的「每小时 token 产出」。** 这是最容易被忽略、也最关键的一步——用 vLLM 把目标开源模型在目标卡上跑起来，**压测出在你能接受的延迟约束下，这张卡每小时能产出多少 token**（吞吐 × 3600）。注意：这个数字强依赖 batch 大小、序列长度、量化与否，必须实测，不能拍。

**第四步：相除得自建单位成本，再乘以利用率折扣。** 自建每百万 token 成本 = 卡每小时全成本 ÷ 每小时 token 产出 × 100 万。**然后除以真实利用率**（推论一）——如果卡只有 30% 时间满载，把这个数字除以 0.3（或等价地说，分母只剩 30%）。最后拿这个「打了利用率折扣的自建单位成本」去和第一步的 API 单价比。

> 工程推论：
>
> **推论四：自建省钱的前提是「高利用率 × 大规模」同时成立——任缺一个，API 都更优。** 高利用率（卡常年跑满）才能把固定成本摊薄；大规模（量足够大）才能让「省下的单价差 × 总量」覆盖运维人力和折旧风险的固定支出。小规模下，哪怕单价自建更低，省下的绝对金额都不够付一个 SRE 的工资，纯亏。**这就是为什么「多数公司够不到自建拐点」**——不是技术不行，是它们的量和利用率撑不起那笔固定成本。

### 2. 成本量级对照表（按 2026 初，**数字会过期，重点感受量级和结构**）

> ⚠️ 下面的具体单价是 2026 初的锚点量级，名单和价格按季度过期；**请学「怎么读这张表、各列之间什么关系」，别背具体数字**。GPU 单价随供需剧烈波动，模型价目表随竞争持续下探。

| 路线 | 成本结构 | 每百万 token 量级（综合，锚点） | 隐性成本 | 适合谁 |
|---|---|---|---|---|
| **Frontier API**（GPT-5 系 / Claude Opus 4.8 / Fable 5 / Gemini 3 级） | 纯 OPEX，按 token | 输入 \$5–10 / 输出 \$25–50 | 数据出门、单价受制于人、有速率上限 | 要顶级能力、量不大或波动大、无合规约束 |
| **Balanced API**（Claude Sonnet 4.6 等中杯） | 纯 OPEX，按 token | 输入 ~\$3 / 输出 ~\$15 | 同上 | 大多数生产工作负载的性价比首选 |
| **Fast API**（Haiku 4.5、各家 mini/flash） | 纯 OPEX，按 token | 输入 ~\$1 / 输出 ~\$5 | 同上，能力天花板较低 | 高频、简单、延迟敏感的任务 |
| **自建开源中等模型**（如 Qwen / Llama 中等档 + vLLM，单/双卡） | CAPEX（卡折旧）+ 运维人力 + 电力 | 满载理论 < \$1，**但乘利用率折扣后常回到 \$3–8** | 运维负担、跑不满的浪费、折旧风险、质量需自己保 | 量大且稳定、利用率高、有合规/定制约束、有专职团队 |
| **自建开源大模型**（如 DeepSeek V3 级 + 多卡 tensor parallel） | CAPEX（多卡集群）+ 重运维 + 电力 | 强依赖集群利用率，规模效应才显著 | 集群运维极重、对工程团队要求高 | 超大规模玩家、强合规、把推理当核心竞争力 |

读这张表的三个要点：

1. **API 三档的输出价≈输入价 5 倍**，这个不对称结构是物理决定的（第 1 章），自建也逃不掉——自建时 decode 阶段（出 token）也比 prefill（吃 token）更慢更贵。
2. **自建那两行的「满载理论 < \$1」是陷阱数字**——它假设了 100% 利用率。括号里的「乘利用率折扣后回到 \$3–8」才是真实世界的数字，**这一栏就是多数自建项目算账时被忽略、上线后被打脸的地方**。
3. **隐性成本那一列，自建明显更重**——运维、浪费、折旧、质量保障，这些都不进单价但实打实烧钱/烧人。算 ROI 必须把这一列折成钱加进去。

> 坑点提示：见过最多的错误，是拿「自建满载理论单价 \$0.8」直接对比「API \$5」，得出「自建省 84%」的结论去说服老板买卡。卡买回来，因为流量波峰波谷 + 模型选型不对 + batch 没调好，**实际利用率 25%、有效单价 \$3.2**，再加运维人力，总成本不降反升——但卡已经买了，沉没成本下只能硬用。**算账时不带利用率折扣 = 给自己挖坑。**

### 3. 一个能在白板上画的判断流程

被问「我们该自建吗」，按这个顺序问，**任何一步答「是」就直接出结论，不用往下走**：

1. **有没有「数据不能出门」的硬合规？** 有 → 必须私有化部署（自建或私有云），成本不在考虑范围（推论三）。
2. **有没有 API 给不了的深度定制（改结构 / 私有权重 / logits 干预）？** 有 → 必须自建。
3. **量级是否足够大、且利用率能否常年压到 60%+？** 两个都「否」 → 留在 API，把工程力投在缓存（第 1 章 prompt caching）、模型分级、batch 化（API 也有 Batch 接口，异步省 50%）上压成本。
4. **以上都过了（量大、利用率高、无合规无定制约束）** → 这才进入「自建 vs API」的真·成本对比，按第 1 节四步算账，算下来自建 ROI 为正才动手。

> 工程推论：
>
> **推论五：「该不该自建」的决策树里，成本对比排在最后一步——前面三步（合规、定制、规模利用率）任意一个就能终结讨论。** 这意味着大多数关于自建的争论，**根本不该停留在「单价谁低」上**——先问死合规和定制（硬约束），再问死规模和利用率（量级门槛），绝大多数公司在第 3 步就被刷掉了，根本走不到「精算单价」那一步。**把决策树的顺序记牢，你就能在 5 分钟内给出比「我研究研究」高得多的判断。**

## 三、vLLM 与 continuous batching：自建吞吐的第一引擎

从这一节到第七节，我们钻进推理引擎内部，搞懂「自建这张卡的吞吐到底由什么决定、能怎么榨」——这是你算账第三步（测每小时 token 产出）和后续调优的物理基础。

### 1. 背景：朴素推理为什么浪费到令人发指

你自己写一个最朴素的推理 loop（HuggingFace `model.generate()` 一把梭），会遇到两个致命浪费：

**浪费一：static batching（静态批处理）的「等齐」问题。** 为了提吞吐，你会把多个请求凑成一个 batch 一起算（GPU 喜欢大矩阵）。但朴素做法是 **static batching**——一个 batch 里的请求必须一起开始、**一起结束**。问题是不同请求生成长度天差地别：请求 A 要生成 10 个 token 就结束了，请求 B 要生成 1000 个。static batching 下，A 早早算完了却**必须在 batch 里干等 B 跑完 1000 步**，这期间 A 占的那块算力完全空转。一个 batch 的有效利用率被最长的那个请求拖死。

**浪费二：GPU 在 decode 阶段严重「吃不饱」。** 生成（decode）是逐 token 的，每步只算一个 token 的矩阵乘——这个计算量小到喂不饱 GPU 的算力，瓶颈卡在**显存带宽**（要反复把模型权重从显存搬进计算单元）而非算力。单请求 decode 时，GPU 算力大量闲置。

### 2. 现状：continuous batching 怎么把浪费吃掉

**continuous batching（连续批处理，也叫 in-flight batching / dynamic batching）** 是 vLLM 这类现代引擎的核心机制，它解决的正是「等齐」问题：

- **不再等齐。** batch 是**动态的、token 级别的**——每生成完一个 token（一个 decode step），引擎就检查：有没有请求已经结束（出了 EOS 或到长度上限）?结束的立刻**踢出 batch、释放它的资源**;同时把**排队等待的新请求立刻塞进刚空出来的槽位**。batch 的成员**逐 step 进进出出**，像一个流水线工位而不是一批齐步走的人。
- **效果：GPU 始终保持高负荷。** 因为短请求一结束就被换下、新请求马上补上，GPU 的每个槽位都尽量保持「在算有效 token」的状态，利用率从 static batching 的「被最长请求拖死」拉到接近满载。**实测吞吐通常能比 static batching 高数倍到一个数量级**（具体倍数依负载分布而定）。

### 3. static vs continuous batching 对照表（这是推理吞吐的物理分水岭，**讲透不会过时**）

| 维度 | static batching（朴素） | continuous batching（vLLM 等） |
|---|---|---|
| batch 边界 | 请求级，整批一起开始/结束 | token 级（step 级），成员动态进出 |
| 短请求的命运 | 算完了在 batch 里干等最长请求 | 算完立刻被踢出、释放资源 |
| 新请求入场 | 必须等整个 batch 结束才能开下一批 | 当前 step 一有空槽就立刻塞进来 |
| GPU 利用率 | 被最长请求拖死，常年很低 | 接近满载 |
| 吞吐 | 低 | 高数倍至一个数量级 |
| 实现复杂度 | 低（自己手写也行） | 高（引擎帮你做，别自己造） |

> 工程推论：
>
> **推论六：自建推理「别自己写 serving loop」——continuous batching、KV cache 管理这些是 vLLM / SGLang / TensorRT-LLM 已经做到极致的引擎能力，你手写的版本吞吐会差一个数量级。** 这跟「别自己写数据库」「别自己写负载均衡」是同一个道理：这些是有专门团队打磨多年的基础设施，你重造只会更慢更不稳。自建工程师的价值不在「实现 continuous batching」，而在「**选对引擎、把模型和卡配好、把 batch/显存/并行参数调到你这个业务的最优点**」。
>
> **推论七：吞吐（throughput）和延迟（latency）在 batching 上是直接冲突的——batch 越大，吞吐越高（成本越低），但单请求的排队和首 token 延迟越大。** 这是吞吐 vs 延迟权衡（第七节）的第一个具体体现。离线批处理（不在乎单条快慢、要总量便宜）就开大 batch；在线交互（用户等着首 token）就得限制 batch 大小、保住延迟。**同一个引擎、同一张卡，调 batch 参数就能在「省钱」和「快」之间滑动——选哪头取决于业务，不存在普适最优。**

### 4. 落地：vLLM 在生产里大厂怎么用（按 2026 初，**接口会变，重点是能力边界**）

- **vLLM** 是开源推理引擎里的事实标准之一，主打 continuous batching + PagedAttention（下一节），起服务通常一行命令（`vllm serve <model>`），且**直接暴露 OpenAI 兼容的 HTTP 接口**——这点极重要：意味着你自建的服务能**无缝接进任何按 OpenAI 协议写的客户端和网关**（第 20 章），切换「自建 vs API」对上层代码几乎透明。
- **SGLang** 在某些结构化生成、复杂 prompt 复用场景吞吐更优，是 vLLM 的有力竞品。
- **TensorRT-LLM**（NVIDIA）在 NVIDIA 卡上做极致延迟优化时常被选用，但和硬件绑定更深、易用性不如 vLLM。
- 国内大厂自建（豆包/字节、Qwen/阿里、GLM/智谱、混元/腾讯等的私有部署，以及它们对外的推理服务）普遍基于 vLLM/SGLang 这类引擎做二次优化，叠加自家的调度、量化、并行方案。

> 落地心法：**自建的第一步永远是「vLLM 起服务 + 压测吞吐」，而不是「研究怎么优化」。** 先用默认配置把目标模型在目标卡上跑起来、压出 baseline 吞吐和延迟，你才有第三节那道账的分母，也才知道值不值得继续往下调。**没有 baseline 数字就开始谈优化，是无的放矢。**

## 四、KV cache 管理与 PagedAttention：显存才是自建的真·瓶颈

### 1. 背景：KV cache 是什么、为什么它吃显存吃得这么狠

Transformer 自回归生成时，每生成一个新 token，都要让它和前面所有 token 做 attention。为了不每步都重算前面所有 token 的 Key/Value，引擎会把已经算过的每个 token 的 **K 和 V 向量缓存起来**——这就是 **KV cache**。它是推理提速的关键（避免重复计算），但代价是**吃显存，而且吃得随上下文长度线性增长**。

KV cache 的显存占用 ≈ `2（K 和 V）× 层数 × 注意力头维度 × 序列长度 × batch 内请求数 × 精度字节数`。关键结论：**上下文越长、并发请求越多，KV cache 越大**。在长上下文 + 高并发场景，**KV cache 占的显存常常超过模型权重本身**——也就是说，限制你一张卡能同时服务多少请求的，往往不是模型大小，而是**显存里塞不下那么多 KV cache**。

> 一个能让你彻底记住「显存才是瓶颈」的 back-of-envelope（数字按一个 ~32B 档开源模型估，量级感为主）：假设 64 层、KV 维度 ~1024、FP16（2 字节），那么每个 token 的 KV ≈ `2 × 64 × 1024 × 2 ≈ 256 KiB`。一个 8K 上下文的请求，光 KV 就吃 `256 KiB × 8192 ≈ 2.1 GB`。现在算并发：**50 个这样的 8K 请求 = 约 107 GB KV cache**。而这个 32B 模型的权重 FP16 才 ~64 GB（INT4 量化后只 ~16 GB）。**看清楚了——KV cache（107 GB）比模型权重（64 GB）还大。** 这就是为什么一张 80GB 的卡，装下模型权重后剩的显存才是你并发上限的真正约束：不是「卡跑不动这么多请求」，是「显存装不下这么多 KV cache」。把这个数字记牢，你就再也不会在面试里说「自建提并发就加卡算力」这种外行话。

> 锚点类比：KV cache 之于推理显存 = 连接状态（TCP control block）之于服务器内存。一台服务器能扛多少并发连接，不只看 CPU，更看「每个连接的状态占多少内存 × 连接数」会不会把内存撑爆。推理一样——一张卡能扛多少并发请求，KV cache 这个「每请求状态」的显存占用是硬约束。

### 2. 演进：朴素 KV cache 管理的「内存碎片」灾难

朴素做法是给每个请求**预分配一整块连续显存**来放它的 KV cache，按「可能的最大长度」预留。这有两个致命浪费（和操作系统早期连续内存分配的毛病一模一样）：

- **内部碎片（internal fragmentation）**：你按最大长度 2048 预留，但请求实际只生成了 100 个 token——剩下 1948 个 token 的显存槽**被预留了却用不上、又不能给别人**，纯浪费。
- **外部碎片（external fragmentation）**：请求有大有小，连续分配久了显存被切得七零八落，明明总空闲够，但凑不出一块连续的大空间给新请求——**显存利用率被碎片拖到很低**（朴素方案常浪费 60–80% 的 KV cache 显存）。

### 3. 现状：PagedAttention 怎么把显存碎片消灭（vLLM 的招牌发明）

**PagedAttention** 把操作系统**虚拟内存分页（paging）**的思想搬到了 KV cache 管理上——这是 vLLM 最核心、最漂亮的发明：

- **KV cache 切成固定大小的「块（block）」**，像内存页一样。一个请求的 KV cache 不再需要一整块连续显存，而是用**一串可以不连续的物理块**拼起来，靠一张「块表（block table）」做逻辑→物理的映射（完全类比 OS 的页表）。
- **按需分配**：请求生成到哪、就分配到哪个块，**用多少分多少**，内部碎片几乎消除。
- **块可共享**：多个请求如果有相同的前缀（比如同一个长 system prompt），它们的 KV cache 块**可以共享同一份物理块**（copy-on-write），这就是 prefix caching 的底层机制——和第 1 章讲的 prompt caching 异曲同工，都是「相同前缀只算一次」。

效果：**KV cache 的显存利用率从朴素方案的 20–40% 拉到 90%+**，等价于同一张卡能多塞 2–3 倍的并发请求——直接转化为吞吐提升和单位成本下降。

> 工程推论：
>
> **推论八：自建推理的并发上限，绝大多数时候卡在「显存装得下多少 KV cache」，而不是「算力够不够」。** 所以自建调优的核心抓手之一是**省 KV cache 显存**——PagedAttention（消碎片）、量化（第五节，KV cache 也能量化成低精度）、限制最大上下文长度、限制最大并发数，全都在围绕「KV cache 显存」做文章。**面试里被问「自建怎么提并发」，答得出「先看是不是 KV cache 显存打满了，用 PagedAttention 消碎片 + KV cache 量化 + 控上下文长度」，比泛泛说「加卡」高一个段位。**
>
> **推论九：长上下文是吞吐和成本的隐形杀手。** KV cache 随序列长度线性增长——一个塞满上下文的请求，吃的显存可能是短请求的几十倍，挤占的是本可以服务其他并发请求的空间。这就是为什么「把整个文档塞进上下文」虽然方便，但在自建场景下**会显著拉低单卡并发、抬高单位成本**——它和 RAG「只检索相关片段进上下文」（第 8 章）在成本上是此消彼长的关系。**别无脑塞长上下文，每多塞 1K token，你都在为 KV cache 显存买单。**

## 五、量化（Quantization）：用精度换显存和速度

### 1. 背景：为什么要量化

模型权重默认是 FP16/BF16（16 位浮点，每个参数 2 字节）。一个中等模型权重就要几十 GB 显存，大模型上百 GB——**单卡装不下，就得多卡（第六节），多卡又增加成本和复杂度**。量化（quantization）就是把权重（有时也包括激活值、KV cache）**从高精度压成低精度**（INT8、INT4、FP8 等），直接的好处是：

- **显存占用大幅下降**：FP16→INT4 直接砍到 1/4 显存，可能让「原本要 2 张卡」变成「1 张卡装得下」——省卡就是省钱（推论四的分母变小、利用率变高）。
- **推理可能更快**：decode 阶段瓶颈在显存带宽（第三节），权重精度低了、要搬运的字节数少了，带宽压力小，吞吐可能上升。

代价是**精度损失**——压得越狠，模型能力掉得越多。量化的全部艺术就在「**压到多低，能省最多显存/最多速度，而质量掉得还能接受**」。

### 2. 现状：主流量化方案对照（按 2026 初，**方案名会更新，重点是「权重量化 vs 激活量化」「训练后 vs 训练时」这两个分类轴**）

| 方案 | 精度 | 类型 | 典型显存收益 | 精度损失 | 适用场景 |
|---|---|---|---|---|---|
| **FP8** | 8 位浮点 | 训练后/原生 | ~1/2（相对 FP16） | 很小 | 新卡（H100 等）原生支持，质量损失小，生产首选之一 |
| **INT8（如 GPTQ/AWQ 的 8bit）** | 8 位整数 | 训练后量化（PTQ） | ~1/2 | 小 | 通用，质量和压缩比平衡好 |
| **INT4（如 GPTQ/AWQ 的 4bit）** | 4 位整数 | 训练后量化（PTQ） | ~1/4 | 中等（需校准/挑方法控制） | 显存吃紧、要单卡装大模型时，性价比之选 |
| **GPTQ** | 通常 4bit | 训练后，基于二阶信息逐层量化 | ~1/4 | 中，对校准数据敏感 | 经典 PTQ 方法，工具链成熟 |
| **AWQ**（Activation-aware Weight Quantization） | 通常 4bit | 训练后，按激活重要性保护关键权重 | ~1/4 | 中，常优于朴素 GPTQ | 4bit 下保质量的常用选择 |

两个分类轴帮你理解所有量化方案：

1. **量化什么**：只量化**权重**（weight-only，最常见，省显存为主）vs 同时量化**激活值**（weight + activation，省带宽、提速更多，但更难保质量）。GPTQ/AWQ 主要是 weight-only，FP8 常含激活。
2. **什么时候量化**：**训练后量化（PTQ，Post-Training Quantization）**——拿训练好的模型直接压，便宜快速，GPTQ/AWQ 都属此类，是自建场景的主流；vs **量化感知训练（QAT，Quantization-Aware Training）**——训练时就模拟量化，质量更好但要重训，成本高，自建很少自己做。

> 工程推论：
>
> **推论十：量化是「显存/速度 ↔ 质量」的旋钮，选档位前必须在你自己的 eval 集（第 17 章）上测质量损失——别信通用 benchmark 的「几乎无损」。** 不同任务对量化的敏感度天差地别：简单分类/抽取可能 INT4 都几乎无损，复杂推理/代码/长链路 agent 可能 INT8 就开始掉点。**「量化掉不掉点」是个必须在你的业务数据上实测的经验问题，不是查表能定的。** 这正好用上你第 17 章建的 eval 集——量化前后各跑一遍回归，看目标指标掉多少，掉到你能接受的线以下就回退一档。**没有 eval 集就敢上 INT4 = 在生产里盲赌质量。**
>
> **推论十一：FP8 在新卡上常是「免费午餐」级的选择——硬件原生支持、质量损失极小、显存和带宽都受益。** 如果你的卡是 H100 这类原生支持 FP8 的（按 2026 初），FP8 往往是性价比最高的起点：几乎不掉质量就省一半显存。**INT4 这种激进量化留给「单卡死活装不下、宁可掉一点质量也要省卡」的极端场景。** 选量化档位的顺序是：先 FP8/INT8 探质量底线，质量够且还想省更多再试 INT4，每一档都过 eval。

## 六、并行（Parallelism）：单卡装不下时怎么拆

当模型大到单卡显存装不下（即使量化后），就必须把模型**拆到多张卡**上协同推理。两种主要拆法，解决不同问题。

### 1. tensor parallelism（张量并行，TP）：把每一层「横切」到多卡

- **怎么拆**：把模型**每一层内部的大矩阵**切成几块，分到不同卡上，每张卡算一部分，然后**通过高速互联（如 NVLink）做 all-reduce 把结果合并**。一层算下来，所有参与的卡都参与了这一层的计算。
- **解决什么**：单层/单卡显存装不下时，靠 TP 把单层的参数和计算摊到多卡。**主要是为了「装得下 + 算得快」**。
- **代价**：每一层都要跨卡通信（all-reduce），**对卡间互联带宽极其敏感**——所以 TP 通常只在**单机内多卡**（有 NVLink 高速互联）做，跨机器做 TP 会被网络带宽拖死。

### 2. pipeline parallelism（流水线并行，PP）：把不同层「纵切」到多卡

- **怎么拆**：把模型的**不同层**分到不同卡上——卡 1 放第 1–10 层，卡 2 放第 11–20 层……一个请求像在流水线上流动，算完前几层的卡把中间结果传给放后几层的卡。
- **解决什么**：模型层数太多、总参数太大，单机多卡的 TP 也不够，就用 PP **跨机器**把模型按层切开。**主要是为了「跨节点也能装下超大模型」**。
- **代价**：有「**流水线气泡（pipeline bubble）**」——流水线刚启动和排空时，部分卡在空等，利用率有损失；通信量比 TP 小（只在层边界传一次中间结果），所以**对带宽没 TP 那么敏感，适合跨机器**。

### 3. 并行策略对照表（**原理不变，讲透**）

| 维度 | tensor parallelism（TP，横切） | pipeline parallelism（PP，纵切） |
|---|---|---|
| 切什么 | 每一层内部的矩阵 | 把不同的层分给不同卡 |
| 通信模式 | 每层都要跨卡 all-reduce | 只在层边界传中间结果 |
| 带宽敏感度 | 极高（要 NVLink 级互联） | 较低（可跨机器） |
| 典型部署范围 | 单机内多卡 | 跨机器/跨节点 |
| 主要损耗 | 通信开销 | 流水线气泡 |
| 何时用 | 模型单卡装不下，机内多卡协同 | 模型大到单机也装不下，要跨节点 |

实践中，超大模型常**TP + PP 混合**（机内用 TP、跨机用 PP），再叠加数据并行（多副本分摊请求）——这套组合就是大规模自建推理集群的骨架。

> 工程推论：
>
> **推论十二：并行不是「免费扩容」——每增加一层并行，都引入通信开销和复杂度，吞吐通常是亚线性增长（加倍卡数 ≠ 加倍吞吐）。** TP 的 all-reduce、PP 的气泡都在吃掉一部分本该有的加速。**所以「装得下」和「划算」是两回事**：你可能技术上能用 8 卡 TP 跑起一个大模型，但 8 卡的吞吐可能只有理想 8 倍的 5–6 倍，单位成本反而比用 API 调同级模型更高。**这又绕回推论四——并行让你『能自建大模型』，但不代表『自建大模型划算』，算账时要用并行后的真实吞吐，不是理想吞吐。**
>
> **推论十三：能不并行就不并行——优先用量化（第五节）把模型压进单卡，单卡推理永远比多卡简单、没通信开销、运维更省心。** 决策顺序是：能单卡（原生）→ 不行就量化进单卡 → 还不行才上 TP（机内）→ 再不行才上 PP（跨机）。**每往下一档，复杂度和运维负担都跳一级。** 很多团队一上来就想多卡集群，其实量化进单卡就解决了——别把简单问题复杂化。

## 七、吞吐 vs 延迟：自建调优永远在做的那个权衡

前面所有机制最终都汇聚到这个核心张力上。把它单独拎出来讲清，因为它是自建调优的**纲**。

### 1. 三个关键延迟指标（自建必须分清，**概念不会过时**）

- **TTFT（Time To First Token，首 token 延迟）**：从请求发出到吐出第一个 token 的时间。**决定用户「感觉响应快不快」**——聊天/流式场景，TTFT 是体验生命线。主要由 prefill（处理输入）耗时和排队等待决定。
- **TPOT（Time Per Output Token，每 token 生成间隔，也叫 ITL，inter-token latency）**：生成阶段每个 token 之间的间隔。**决定「打字机吐字快不快」**。主要由 decode 速度和 batch 大小决定。
- **Throughput（吞吐）**：整个系统单位时间产出的总 token 数（跨所有并发请求）。**决定成本**——吞吐越高，每 token 成本越低。

### 2. 核心冲突：batch 大小一调，吞吐和延迟反向走

这是整章最该记住的权衡机制：

- **batch 开大** → 一次算更多请求，GPU 更满载 → **吞吐↑（成本↓）**，但每个请求要等更大的批凑齐、要和更多请求抢算力 → **TTFT 和 TPOT↑（单请求变慢）**。
- **batch 开小**（甚至 batch=1） → 单请求独占资源、响应最快 → **延迟↓（体验好）**，但 GPU 严重吃不饱 → **吞吐↓（成本飙升）**。

**没有普适最优——选哪头完全取决于你的业务对「快」还是「省」更敏感。**

### 3. 吞吐 vs 延迟权衡表（按业务场景给可执行判断）

| 业务场景 | 首要指标 | batch 策略 | 配套手段 | 类比 |
|---|---|---|---|---|
| **在线聊天 / 流式交互** | TTFT + TPOT（延迟优先） | 限制 batch 大小，保延迟 | speculative decoding（降延迟）、流式输出、把高峰削给排队 | 像 OLTP——单笔要快 |
| **离线批处理**（批量摘要/打标/数据清洗） | Throughput（吞吐优先） | 开大 batch，吃满 GPU | 拉满并发、不在乎单条慢、可用 API 的 Batch 接口（异步省 50%） | 像 OLAP/批作业——总量便宜就行 |
| **混合负载** | 分流 | 在线和离线**用不同实例/不同 batch 配置**，别混在一个池子 | 网关按请求类型路由（第 20 章） | 像读写分离——别让批作业拖垮在线 |

> 工程推论：
>
> **推论十四：在线和离线负载必须物理隔离，不能共用一个推理池——否则离线大 batch 会把在线请求的延迟拖爆。** 这跟数据库「在线交易库和数据分析库要分开，别让一个大查询拖垮交易」是同一条铁律。一个推理实例的 batch 配置只能服务一种偏好（要么延迟优先要么吞吐优先），想同时伺候两种负载，正确做法是**起两组实例、配两套参数，网关按请求类型分流**，而不是在一个池子里试图调一个「既快又省」的万能 batch——那个万能点不存在。
>
> **推论十五：选 batch 大小本质是选「你愿意为延迟付多少吞吐税」——把这个决策绑到业务 SLA 上，不要拍脑袋。** 正确流程是先定 SLA（比如「P95 TTFT < 500ms」），然后在满足这个 SLA 的前提下**把 batch 开到最大**（吞吐最大化、成本最低）。**SLA 是约束，吞吐是目标函数**——在约束下求最优，而不是凭感觉调一个数。这套「先定 SLO 再在约束下优化」的思路，和你做后端容量规划是一模一样的。

### 4. speculative decoding：一个专门降延迟的巧招

**speculative decoding（投机解码 / 推测解码）** 是降低生成延迟的重要技术，原理很妙：

- 用一个**小而快的「草稿模型」**（draft model）先**一口气猜出未来几个 token**（猜得快，因为模型小）；
- 再用**大模型一次性并行验证**这几个猜测——验证（一次 forward 算多个位置）比逐个生成快得多；
- 猜对的部分直接采纳（一步顶好几步），猜错的从错的地方用大模型的结果纠正。

效果：**在不损失大模型输出质量的前提下，把生成速度提升约 2–3 倍**（取决于草稿模型命中率）。因为最终每个 token 都经过大模型验证，**输出分布和不用投机解码时一致**——这是它最珍贵的性质：**纯赚延迟，不赔质量**。

落地补充（**形态在演进，原理不变**）：你不需要从零实现它——**vLLM / SGLang 这类引擎已内建投机解码**，你的活是「选哪种草稿来源 + 调猜测长度」。主流草稿来源有几档：①**独立小模型**（拿同系列的 Fast 档当大模型的草稿，最经典）；②**自推测**（用大模型自己的浅层 / 额外预测头一次出多个 token，如 Medusa、EAGLE 这类，省掉单独养一个草稿模型）；③**n-gram / 提示词查找**（直接从已生成文本或 prompt 里捞重复片段当猜测，在代码补全、长文档改写这类「输出大量复制输入」的场景命中率极高、几乎零成本）。**决定收益的唯一硬指标是 acceptance rate（草稿被大模型接受的比例）**：命中率高，一次验证顶好几步、提速明显；命中率低，验证的冗余算力白烧、提速塌掉甚至变慢。所以草稿模型不是越大越准越好——**草稿和大模型分布要足够像（高命中）、又要足够快（低草稿开销），这两者的平衡点才是甜区**。面试被追问「为什么你们投机解码没提上去」，答得出「先测 acceptance rate，大概率是草稿模型和主模型分布不够贴或草稿太重」，就说明你真上手调过。

> 工程推论：
>
> **推论十六：speculative decoding 是「纯降延迟、不损质量」的优化，但它换的是「多花一点算力（草稿 + 验证的冗余计算）去省延迟」，且在高并发满载时收益缩水。** 它的甜区是**低/中并发、对单请求延迟敏感**的场景（草稿模型有空闲算力可用）；当 GPU 已经被大 batch 喂满时，再塞草稿模型的冗余计算反而抢资源，收益下降甚至变负。**所以它和「大 batch 提吞吐」某种程度上是互斥的两条路——延迟优先的在线服务用它，吞吐优先的离线批处理通常不用。** 又一次印证：所有推理优化都在吞吐 vs 延迟这根轴上各占一头。

## 八、何时必须自部署：把判断收口

绕了一圈技术，回到决策。**自部署的驱动力就三类，按「硬度」排序**：

### 1. 数据合规（最硬，算不算账都得做）

数据不能出境、不能流经第三方 API——金融、医疗、政务、欧盟 GDPR、强监管行业。这种情况下 API 再便宜也用不了，**私有化部署（自建或私有云）是唯一合法路径**，成本对比直接失效。这是「成本算不过来也得做」的第一类硬约束（合规的纵深细节见第 21 章）。

### 2. 规模经济（要算账，且门槛很高）

量足够大 + 利用率足够高（常年 60%+），自建单位成本才打得过 API 且省下的钱覆盖固定支出。**这是唯一一个「纯靠算账决定」的驱动力，也是多数公司够不到的那个拐点**（推论四、五）。一个判断标志：如果你的月 token 消耗大到 API 账单成了公司的显著成本项、且流量稳定可预测，才值得认真算自建账。

### 3. 深度定制（要算能力账）

API 给不了的：改模型结构、跑私有微调/蒸馏的权重、在推理层做特殊干预（logits 处理、特殊采样、嵌入私有逻辑）。**要做 API 厂商根本不开放的事情，只能自建。**

> 工程推论：
>
> **推论十七：「默认买 API，被合规/规模/定制三者之一逼到墙角才自建」——这是 2026 年绝大多数团队的正确默认姿势。** 把举证责任放在「自建」这一边：不是「为什么不自建」，而是「**有什么硬理由必须自建**」。答不出合规、答不出规模拐点、答不出非自建不可的定制需求，那就老老实实买 API，把工程力投到缓存、模型分级、网关、eval 上——这些对绝大多数团队的成本和可靠性收益，远大于折腾 GPU。**敢说『我们不该自建』并讲清三类驱动力都不成立，是比『我会部署 vLLM』更值钱的判断力。**

> 本章心法：**自托管一个模型是个下午就能完成的技术活，但「该不该自托管」是个能区分初级和资深的判断题。** 把推理还原成单位经济学（每百万 token 多少钱 × 利用率折扣），把所有引擎机制还原成吞吐/延迟这两个旋钮，把决策还原成「合规 / 规模 / 定制」三道硬门槛——你就既能在白板上算清那笔账，又能在 GPU 上把吞吐榨干。**能算账的人决定要不要做，能调优的人负责做得好——这一章要你两样都会，但先学会算账。**

## 九、未来演进：3-5 年，对应用工程师意味着什么

> 时效声明：这一节是**判断框架**，不是预言。具体技术名按 2026 初，**会过期；学的是「往哪个方向看」**。

1. **推理成本会继续断崖式下跌，进一步抬高自建门槛。** 硬件（更强的推理卡）、算法（更狠的量化、更好的投机解码、新的注意力机制如更省 KV cache 的变体）、API 厂商的规模化竞价，三股力量一起把 API 单价往下压。**含义**：今天算下来「自建勉强省钱」的场景，明年可能就被 API 降价抹平——**自建的拐点会越来越高、越来越少公司够得着**。除非你有合规/定制的硬约束，否则赌「自建省钱」的窗口在收窄。

2. **开源与闭源的能力差距持续波动，但「开源够用」的场景在扩大。** DeepSeek V3/R1 已证明开源能追平闭源推理（2026 初分水岭）。**含义**：越来越多「不需要绝对顶级能力」的任务，可以用自建开源模型以更低单价 + 数据自主搞定——这让「按账分流」（高频简单走自建开源、低频复杂走 Frontier API）这套混合架构越来越主流。**应用工程师要学的不是绑死某个开源模型，而是『评估一个开源模型够不够用我这个场景』的能力**（接回第 17 章 eval）。

3. **「推理即服务」的形态会越来越像 serverless，自建和买的边界变模糊。** 出现更多「按需起、按用付、帮你管 GPU」的中间形态（serverless GPU、托管 vLLM、按 token 计费的开源模型托管），让你既能用开源模型/私有权重，又不用自己养卡养人。**含义**：未来「自建 vs API」不再是非黑即白的二选一，而是一个连续光谱（裸 API → 托管开源 → serverless GPU → 租卡自部署 → 买卡自建），你要学会**在这个光谱上按合规/成本/控制力的需求选一个点**，而不是非此即彼。

4. **对应用工程师的最终含义**：你**几乎永远不需要亲手训练或从零优化推理引擎**——那是基础设施团队和模型厂的活。但你**必须有能力**：①算清楚任一条路线的单位经济学；②读懂吞吐/延迟/显存的瓶颈在哪、判断一个自建方案是否合理；③在合规/规模/定制三道门槛上做出「该不该自建」的判断并扛住追问。**这一章给你的不是「会部署」的技能，是「会决策部署」的判断力**——后者在职级评定里值钱得多。

## 章末五件套

### 一、高频面试题（10 道，附答题框架）

**Q1：我们现在用 API，每月 token 账单几十万了，老板问能不能自建省钱，你怎么分析？**
- 框架：**先走决策树再算账**。①先问硬约束：有没有数据合规、有没有 API 给不了的定制？有就直接私有化，不是省钱问题。②没有硬约束才算规模账：当前 API 综合单价 → 自建那张卡的每小时全成本（含运维人力、折旧、电力）÷ 该卡实测每小时 token 产出 = 自建理论单价 → **乘真实利用率折扣**（关键！流量波峰波谷拉低有效利用率）→ 比较。③还要把运维人力、跑不满浪费、质量保障这些隐性成本折钱加进去。
- 陷阱：只算「满载理论单价 \$0.8 vs API \$5」就喊「省 84%」——**没乘利用率折扣 = 错误结论**。能主动提「利用率是分母、波峰波谷是杀手」就过关。

**Q2：continuous batching 比 static batching 强在哪？为什么能提吞吐一个数量级？**
- 框架：static batching 整批一起开始/结束，**短请求算完了得在 batch 里干等最长请求**，GPU 被最长请求拖死、利用率极低。continuous batching 是 **token 级动态批**——每个 decode step 检查谁结束了就踢出、谁在排队就立刻补进空槽，GPU 始终接近满载。本质是把「批级等齐」改成「token 级流水」。
- 加分：点出「这是吞吐 vs 延迟的第一个权衡点——batch 越大吞吐越高但单请求延迟越大」。

**Q3：PagedAttention 解决什么问题？和操作系统什么概念同构？**
- 框架：解决 **KV cache 显存碎片**。朴素做法给每个请求预分配连续显存（按最大长度），导致内部碎片（预留用不上）+ 外部碎片（凑不出连续大块）,浪费 60–80% KV cache 显存。PagedAttention 把 KV cache 切成固定大小的块、用块表做逻辑→物理映射、按需分配、相同前缀的块可共享——**完全是操作系统虚拟内存分页（paging + 页表 + COW）搬到显存上**。效果：KV cache 利用率从 20–40% 拉到 90%+。
- 加分：点出「这也是 prefix caching / prompt caching 的底层机制——相同前缀块共享」，连回第 1 章。

**Q4：为什么说自建推理的并发瓶颈常在显存而非算力？**
- 框架：decode 阶段每步只算一个 token，算力喂不饱、瓶颈在显存带宽；而能同时服务多少请求，取决于**显存能装下多少 KV cache**。KV cache 随上下文长度 × 并发数线性增长，长上下文 + 高并发时常超过模型权重本身。所以并发上限是「显存装得下多少 KV cache」，不是算力。
- 加分：给出省 KV cache 的抓手——PagedAttention 消碎片、KV cache 量化、限上下文长度、限并发数。

**Q5：量化（GPTQ/AWQ/FP8/INT4）各是什么？怎么选档位？**
- 框架：量化 = 把权重（有时含激活、KV cache）从 FP16 压成低精度，**省显存（可能让多卡变单卡）+ 省带宽（decode 可能更快），代价是精度损失**。两个分类轴：①量化什么（weight-only 如 GPTQ/AWQ vs 含激活如 FP8）；②什么时候量化（训练后 PTQ，便宜，主流 vs 量化感知训练 QAT，贵，少用）。选档位：**FP8（新卡近乎免费）→ INT8 → INT4（最省但掉点风险大），每一档都在自己的 eval 集上跑回归看质量掉多少**。
- 陷阱：信通用 benchmark 的「几乎无损」就上 INT4。正解：**量化掉不掉点是业务数据上的实测经验问题**，必须用第 17 章的 eval 集验证，不同任务敏感度天差地别。

**Q6：tensor parallel 和 pipeline parallel 区别？分别什么时候用？**
- 框架：TP **横切**——把每层内部的矩阵切到多卡，每层都要跨卡 all-reduce，**对带宽极敏感，只在单机内（NVLink）做**，为了「单卡装不下时机内协同」。PP **纵切**——把不同层分到不同卡，只在层边界传中间结果，**带宽敏感度低、可跨机器**，为了「单机也装不下的超大模型跨节点切」，代价是流水线气泡。超大模型常 TP+PP 混合（机内 TP、跨机 PP）。
- 加分：点出决策顺序「能单卡就别多卡，能量化进单卡就别并行，TP（机内）优先于 PP（跨机），每往下一档复杂度跳一级」。

**Q7：TTFT、TPOT、throughput 分别是什么？batch 大小怎么影响它们？**
- 框架：TTFT（首 token 延迟，决定「感觉响应快不快」）、TPOT（每 token 间隔，决定「吐字快不快」）、throughput（总吞吐，决定成本）。**batch 开大 → 吞吐↑成本↓，但 TTFT/TPOT↑（单请求变慢）；batch 开小 → 延迟↓体验好，但吞吐↓成本飙升**。没有普适最优，看业务对快还是省更敏感。
- 加分：给落地——在线聊天限 batch 保延迟，离线批处理开大 batch 吃满吞吐，混合负载物理隔离 + 网关分流。

**Q8：speculative decoding 原理？什么场景用、什么场景不用？**
- 框架：小草稿模型快速猜未来几个 token → 大模型一次并行验证 → 猜对采纳（一步顶多步）、猜错纠正。**纯降延迟（约 2–3 倍）、不损质量（每 token 经大模型验证，分布一致）**，代价是多花冗余算力。**甜区：低/中并发、延迟敏感的在线服务（有空闲算力跑草稿）；不用：高并发满载的离线批处理（草稿冗余计算抢资源，收益变负）**。
- 加分：点出「它和大 batch 提吞吐某种程度互斥——延迟优先用它，吞吐优先不用」，呼应吞吐 vs 延迟主轴。

**Q9：什么情况下必须自部署，不能用 API？**
- 框架：三类驱动力按硬度排序：①**数据合规**（最硬，数据不能出境/过第三方，算不算账都得私有化）；②**规模经济**（要算账，量大+利用率高才划算，门槛高、多数公司够不到）；③**深度定制**（API 给不了的改结构/私有权重/logits 干预）。决策默认姿势：**默认买 API，被这三者之一逼到墙角才自建**，举证责任在「自建」一方。
- 加分：敢下「我们不该自建」的判断并讲清三类驱动力都不成立。

**Q10：你已经决定自建了，怎么把一张卡的吞吐榨到最高？**
- 框架：①选成熟引擎（vLLM/SGLang，别自己写 serving loop）拿 continuous batching；②PagedAttention 消 KV cache 碎片；③量化（FP8/INT8/INT4，过 eval）省显存换更大 batch；④在 SLA 约束下把 batch 开到最大；⑤先看瓶颈是显存还是算力——显存满了就省 KV cache，算力闲了就加 batch；⑥能单卡别多卡（并行有通信损耗、亚线性）。
- 加分：强调「先压出 baseline 再优化」「先定 SLA 再在约束下最大化吞吐」的工程纪律，而不是无目标乱调。

### 二、实战项目增量（P4 可选增量：自建 vs API 实测对比）

在 P4 生产化外壳的基础上，加一个**可选但极加分**的增量：**用 vLLM 自部署一个开源模型，和你 P2/P3 在用的 API 做三轴实测对比。**

**验收标准（缺一不算完整）：**
1. **部署跑通**：用 vLLM（或 SGLang）把一个开源模型（如 Qwen / Llama 中等档，按 2026 初可得型号）在一张可获取的 GPU（自有卡或云上租用实例）上起服务，暴露 OpenAI 兼容接口，**能被你 P2/P3 的现有客户端无改动或微改调用**。
2. **成本轴**：算清三个数——①API 这条路你实际跑那批请求花了多少（查账单/按价目表算）；②自建这条路：卡每小时全成本（租金/折旧 + 你估的运维分摊）÷ 实测每小时 token 产出 = 自建理论单价；③**给自建单价乘上你这个场景的真实利用率折扣**，得到「打折后自建单价」，和 API 单价同表对比。
3. **延迟轴**：在相同并发下，分别测 API 和自建的 **TTFT、TPOT、整体吞吐**，列表对比。至少跑「低并发（1–2）」和「高并发（拉满）」两档，观察自建的吞吐/延迟随 batch 怎么变。
4. **质量轴**：**复用你第 17 章建的 eval 集**，让自建模型和 API 模型各跑一遍回归，量化质量差距（自建开源模型在你这个任务上比 API 掉多少点）。可选：对自建模型做一次量化（FP8 或 INT8），eval 前后对比，量化质量损失。
5. **结论**：用一段话 + 一张三轴表给出判断——「**对我这个场景，自建 vs API 哪个更优、为什么、临界点在哪（量大到多少 / 利用率高到多少才反转）**」。带数字、带因果、带判断。

**简历话术示范**：不是「部署过 vLLM」，而是「**用 vLLM 自部署 Qwen-XX，与 Claude Sonnet 4.6 做成本/延迟/质量三轴实测：自建满载单价比 API 低 X%，但按本场景 25% 利用率折算后实际高 Y%；INT8 量化后质量在 eval 集上掉 Z 个点；据此判断本场景应继续用 API，给出自建反转的临界利用率 ≈ N%**」——这一条能在面试里讲 20 分钟。

> 坑点提示：很多人卡在「测每小时 token 产出」——记得这是**在你能接受的延迟约束下**的吞吐，不是不管延迟一味开大 batch 冲出来的虚高数字。延迟约束不同，吞吐不同，要和你的 SLA 对齐。

### 四、系统设计题（含量级估算）

**题目**：你要为一个日活 100 万的 AI 客服产品做推理容量规划与「自建 vs API」决策。已知：日活 100 万，人均每天 5 轮对话，单轮平均输入 500 token、输出 300 token。请估算：(a) 日/月 token 总量；(b) 走 API 的月成本量级（自选档位并说明）；(c) 若自建，需要多少张卡、利用率要多高才能打平 API；(d) 给出你的最终决策建议。

**参考解（数字按 2026 初锚点，重点是估算链路和量级感，不是精确值）：**

- **(a) token 总量**：日活 100 万 × 5 轮 = 500 万轮/天。单轮 input 500 + output 300 = 800 token，但 input/output 要分开算（价格不对称）。日 input = 500 万 × 500 = 25 亿 token；日 output = 500 万 × 300 = 15 亿 token。**月（×30）：input ≈ 750 亿，output ≈ 450 亿 token。**

- **(b) API 月成本**：客服场景简单高频，选 Fast 档（input ~\$1 / output ~\$5 每百万）合理。月成本 ≈ 750 亿/100 万 × \$1 + 450 亿/100 万 × \$5 = 7.5 万 × \$1 + 4.5 万 × \$5 = \$7.5 万 + \$22.5 万 = **约 \$30 万/月**（输出贡献了大头，印证输出贵 5 倍的结构性影响）。**注意：这里若叠加 prompt caching（system prompt 和常见话术前缀缓存），input 成本可大幅下降，实际可能远低于这个裸算——这正是「留在 API 的人该投的工程力」。**

- **(c) 自建打平估算**：日 output 15 亿 token 是主要负载（decode 是瓶颈，prefill 吃 input 的部分先忽略）。核心公式必须脱口而出：**所需卡数 = 日 token 量 ÷（单卡每秒吞吐 × 86400 × 利用率）**。把这条链算到底（**下面的吞吐和单价是占位锚点，必须用 vLLM 实测替换，但要敢代入数字把链路走完，别停在「占位」**）：

  - 假设一张 H100 级卡跑量化后的开源中等模型，在可接受延迟约束下 **decode 聚合吞吐 ≈ 2500 token/秒**（实测得到，强依赖 batch/量化/序列长度）。单卡日产 = 2500 × 86400 ≈ **2.16 亿 token/天**。
  - **满载（100% 利用率）所需卡数** = 15 亿 ÷ 2.16 亿 ≈ **7 张**。
  - **代入真实利用率**：客服有昼夜波峰波谷，利用率撑死 30% → 7 ÷ 0.3 ≈ **23 张**（推论一的分母效应，直接把卡数翻 3 倍多）。
  - **算这堆卡的月全成本**：按每卡全成本 \$3/小时（租金 + 运维分摊 + 电力）→ 23 张 × \$3 × 24 × 30 ≈ **\$5 万/月**。

  这里有个**专门埋的陷阱**：\$5 万自建 vs \$30 万 API，看上去自建省 83%，候选人很容易当场喊「该自建」——**但这恰恰是本章反复警告的错误**。两个反转因素都不在这个单价对比里：①(b) 的 \$30 万是 prompt caching 前的裸算，客服 prompt 高度重复，caching 后 API 实际成本可能腰斩到 \$10–15 万，自建优势立刻缩水；②自建的开源中等模型在你的客服 eval 集（第 17 章）上**能不能达到 API 的质量**还没验证——质量不达标，省下的钱毫无意义。**能把这条链算到「\$5 万」并立刻指出「这个数字不能直接拿去拍板」，才是高分答案。**

- **(d) 决策建议**：**优先留在 API + 重度 prompt caching + 模型分级**。理由：①客服场景 prompt 高度重复（话术模板、知识库前缀），prompt caching 能把 input 成本砍到很低，API 实际成本远低于裸算的 \$30 万；②昼夜波峰波谷让自建利用率撑不到 60%+，自建单位成本反而高；③省下的钱不足以覆盖 GPU 集群运维 + 折旧风险。**除非**：有数据合规要求（那必须私有化），或量级再涨一个数量级且能用离线/异步把利用率填平。**这道题的判断内核 = 推论四 + 推论五 + 第 1 章 prompt caching 的综合运用。**

> 估算心法：面试官不在乎你算的卡数精不精确，在乎你**有没有「日活→轮次→token→成本→卡数→利用率」这条完整估算链**，以及**有没有意识到「输出贵 5 倍」「利用率是分母」「prompt caching 改变 API 成本」这三个量级杠杆**。把链路和杠杆讲清，数字差一倍都不扣分。

**暗坑总结（这题专治三个常见错误）**：
- **坑一**：输入输出价格用简单平均而非按比例加权——忽略了「输出贵 5 倍 + 输入输出量不对称」。
- **坑二**：自建成本只算满载理论、不乘利用率折扣——这是现实里自建项目翻车的头号原因。
- **坑三**：算打平利用率时误把含折扣的真实单价代进去——要用满载理论单价解方程。
- **元坑**：程序算出「自建略便宜」就以为该自建——**忘了运维人力、折旧风险、质量保障这些不在单价里的隐性成本**。代码能告诉你单价对比，但「该不该自建」是单价 + 隐性成本 + 合规/定制约束的综合判断，**代码是输入，不是结论**（呼应推论五）。
