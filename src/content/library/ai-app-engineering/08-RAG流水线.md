> 这章排在阶段二的开篇，因为它是把第 1 章那四个"物理事实"第一次**系统性对冲**的地方。第 1 章告诉你模型会胡说（概率延续 ≠ 事实查询）、记不住私有知识（训练截止 + 无状态）、不知道自己不知道（置信度不可信）。RAG 就是针对这三条的工程答案：用检索把外部真相**塞进上下文**，让"正确答案"变成模型的高概率延续。前面 P0/P1 你已经能调 API、能让模型用工具；这一章你要造出第一个**有外接真相源、答案能溯源**的系统。
>
> 替你解决的问题：让你停止纠结"要不要微调来灌知识"，建立一个判断——**绝大多数"让模型懂我家业务"的需求，第一反应应该是 RAG 不是微调**。这章讲清楚 RAG 的完整流水线（八个环节，每个都有坑）、chunking 这个被严重低估的脏活、ingestion 这个被严重忽视的工程、以及 naive RAG 为什么在 demo 里很美、上线就拉胯。

读法提示：这章信息密度高，每个机制后面紧跟**工程推论**（编号列出，每条都能独立成一道面试题）。第一遍至少把"RAG vs 微调对照表""八环节流水线图""chunking 四策略对照表""naive RAG 失效模式清单"啃下来。代码题对应主线项目 **P2 企业知识库 RAG**——这一章结束你应该有一个端到端能跑、带引用的知识库问答。

时效标注：模型名 / embedding 维度 / 向量库具体产品按 **2026 初**写，名单会变，**重点学选型方法论**；而"检索-生成两段式""chunk 大小对检索与生成的拉扯""prefix 缓存被检索结果击穿"这些是**架构规律**，讲透到骨子里。

交叉引用：本章只把 naive RAG 的失效模式**列出来**并讲清成因；逐个治理（query 改写、混合检索、rerank 工程、上下文压缩）在**第 9 章**。文档解析（PDF/表格/扫描件这些脏数据怎么变成干净文本）单独放在**第 10 章**。如何量化"这套 RAG 到底好不好"（召回率、忠实度、答案相关性）放在**第 11 章**。本章默认你拿到的是"已经清洗好的纯文本"，把流水线骨架搭通。

## 一、背景：RAG 为什么出现

### 1. 三个一起出现的痛点

RAG（Retrieval-Augmented Generation，检索增强生成）不是某个人拍脑袋发明的优雅架构，是被三个**同时存在的硬约束**逼出来的：

**痛点一：grounding——抑制幻觉。** 第 1 章推论一说过，幻觉是机制内生的，不可消除只能对冲。对冲的核心手段就是 **grounding（接地）**：不让模型凭记忆作答，而是先检索出相关原文，把原文放进上下文，要求模型"基于以下材料回答"。模型这时做的还是概率延续，但延续的前缀里**已经有了正确答案的原文**，于是"照抄/转述原文"成了最高概率路径，胡编的概率被压下去。注意：RAG **不能消灭**幻觉（模型仍可能曲解原文、原文检索错了、或原文里压根没有答案而它硬答），它只是大幅**降低**幻觉率。把 RAG 当"幻觉杀手"是新手第一误区，它是"幻觉抑制器"。

**痛点二：注入私有/专有知识。** 模型的参数里只有训练语料见过的公开知识。你公司的产品手册、内部 wiki、客户合同、昨天的工单——它一概不知道。要让模型"懂你家业务"，两条路：把知识**写进参数**（微调/继续预训练）或**塞进上下文**（RAG）。后面会展开为什么大厂默认选后者。

**痛点三：时效与权限。** 模型有训练截止日期，它不知道上周发布的新版本、今早改的价格。而且参数里的知识是"全员可见"的——你没法让模型对 A 用户隐藏 B 部门的机密文档。RAG 把知识放在外部检索层，于是：知识**可更新**（改文档即生效，无需重训）、可**带时效**（检索时按时间过滤）、可**带权限**（检索时按用户的 ACL 过滤，从源头上不把无权文档喂给模型）。

> 心法：RAG 解决的三件事可以压成一句话——**让模型回答它本来不该知道、或不敢确定的东西，且能说清答案从哪来。** grounding 治幻觉、外接知识治"不懂业务"、外部检索层治时效与权限。三者缺一，你都得用别的架构补。

### 2. 演进：以前怎么做，为什么不够

- **纯 prompt 塞知识（stuffing）**：把整份文档直接拼进 system prompt。问题：文档一大就撑爆上下文窗口；就算窗口够，每次请求都重发整份文档，**token 成本爆炸**，且长上下文里"中间部分被忽略"（lost in the middle）会让效果打折。只适合知识极少（几页）的场景。
- **微调灌知识**：把企业知识做成训练样本去微调模型。后面单列对照表细说，这里先记结论：**微调改的是行为/风格，不是事实知识**——你很难靠微调让模型可靠记住"X 产品的保修期是 18 个月"这种具体事实，而且改一次知识要重训一次，贵且慢，还无法溯源。
- **关键词检索 + 拼接（ES/BM25 时代）**：先用 Elasticsearch 这类全文检索按关键词捞文档，再拼进 prompt。这其实已经是 RAG 的雏形，且 BM25 至今仍是混合检索的半壁江山（见第 9 章）。它的短板是**只能匹配字面**——用户问"怎么退货"，文档写"申请退款流程"，关键词对不上就召回失败。
- **2022 后 embedding + 向量检索成熟**：把文本编码成向量，用**语义相似度**检索。"退货"和"退款流程"在向量空间里挨得近，于是能召回。这是现代 RAG 的标志性拼图——**从字面匹配进化到语义匹配**。

> 工程推论（本节地基，逐条都是面试题）：
>
> **推论一：RAG 的第一性原理是"把正确答案变成高概率延续"。** 一切环节（切得准不准、检索得全不全、排得对不对、拼得好不好）都服务于一个目标——让模型在生成时，前缀里有它需要的原文。脱离这个目标谈 RAG 优化就是瞎调参。
>
> **推论二：RAG 把"知识正确性"从模型内部搬到了检索层。** 这是巨大的架构收益——知识对错现在是**可工程化、可测试、可审计**的（你能去查检索召回了什么），而不是埋在模型参数里的黑箱。这也是为什么后端工程师天然适合做 RAG：它把 AI 的不可控部分，转化成了你熟悉的"检索系统 + 数据管道"问题。

### 3. 现状：RAG vs 微调——必须能一句话下判断

这是 RAG 章节最高频的面试考点，没有之一。面试官想区分的是：**你是不是把"灌知识"和"调行为"分清楚了。**

| 维度 | RAG（检索增强） | 微调（Fine-tuning） |
|---|---|---|
| **改的是什么** | **知识 / 事实**（外接真相源） | **行为 / 风格 / 格式**（内化能力） |
| **更新成本** | 改文档即生效，秒级 | 重新训练，小时到天级 |
| **可溯源** | 能（答案指向具体来源 chunk） | 不能（知识糊进参数，黑箱） |
| **时效性** | 强（检索时拿最新文档） | 弱（知识冻结在训练时刻） |
| **权限控制** | 能（检索时按 ACL 过滤） | 不能（参数对所有人一致） |
| **幻觉** | 显著降低（有 grounding） | 不一定降，甚至可能"自信地"记错 |
| **前期投入** | 中（搭检索管道） | 高（数据标注 + 训练 + 评估） |
| **典型场景** | 问答、客服、文档检索、知识库 | 固定输出格式、特定语气、领域术语习惯、压缩 prompt |
| **能不能合用** | **能，而且常常该合用** | 微调出"风格/格式" + RAG 喂"事实" |

**一句话判断（背下来）**：**要让模型『知道得更多/更新』→ RAG；要让模型『表现得不一样』→ 微调。** 大厂默认"先上 RAG 再考虑微调"，因为 90% 的"让模型懂我家东西"需求本质是缺知识，而 RAG 在更新成本、可溯源、时效、权限上全面碾压微调。微调真正不可替代的场景是：你需要稳定的输出格式/语气、领域专属的措辞习惯、或者想把冗长的 system prompt 蒸馏进模型省 token。

> 面试加分点：被问"RAG 和微调怎么选"，初级答案是罗列优缺点。高级答案是先抛判据——"**先问这个需求是缺知识还是缺行为**"——再补一句"二者不互斥，生产里常是微调定风格、RAG 供事实"，最后给个反例证明你想清楚了边界："但如果是要模型严格输出某个 JSON schema，那是行为问题，RAG 帮不上，该用结构化输出或微调。"
>
> 坑点提示：有人觉得"微调能让模型记住知识，比 RAG 省事"。错。微调对**事实记忆**极不可靠——你喂 100 遍"保修 18 个月"，它可能还是答"12 个月"，因为微调优化的是分布而非记忆表。而且你没法让它溯源、没法快速改。把事实记忆寄托在微调上，是拿错工具干活。

## 二、完整流水线：从文档到带引用的答案

RAG 不是一个函数，是一条**两阶段、八环节**的流水线。两个阶段时间上是分离的，这是理解 RAG 工程的关键：

- **离线阶段（Ingestion / 索引构建）**：`load → chunk → embed → index`。在用户提问**之前**就把知识库处理好、入库。这是个数据管道，跑批、可重建、可增量。
- **在线阶段（Retrieval / 查询应答）**：`retrieve → rerank → generate → cite`。用户提问**时**实时执行。这是个低延迟请求路径。

```
┌─────────────────── 离线：Ingestion（建索引，提前跑）───────────────────┐
│                                                                        │
│  原始文档        切分           向量化            入库                   │
│  load   ──►   chunk   ──►   embed   ──►   index (向量库)               │
│ (PDF/MD/     (按结构/      (embedding   (HNSW/IVF                       │
│  HTML/库)     大小切块)     模型→向量)    索引 + 元数据)                 │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
                                                    │
            （知识库就绪，等待查询）                  ▼
┌─────────────────── 在线：Retrieval（用户提问时实时跑）─────────────────┐
│                                                                        │
│  用户 query                                                            │
│      │                                                                 │
│      ▼ embed(query)                                                    │
│  retrieve ──► rerank ──► generate ──► cite                            │
│ (向量库捞      (精排，    (拼 prompt:    (答案标注                       │
│  top-k 候选)   重排序     系统指令+      来源 chunk)                     │
│                top-n)     检索材料+query)                               │
│                                                                        │
└────────────────────────────────────────────────────────────────────────┘
```

逐个环节拆，每个都点出它在干什么、最容易踩的坑、以及它和下游的耦合。

### 1. load（加载）——把异构数据变成文本

从 PDF、Markdown、HTML、Confluence、数据库、Notion 等各种源把内容读出来，统一成纯文本 + 元数据。**这一步是整条流水线最脏、最被低估的活**，尤其 PDF——表格、多栏排版、扫描件、页眉页脚噪声，处理不好后面全是垃圾。

> 本章不展开解析细节，**第 10 章专门讲文档解析**。这里只记一条铁律：**garbage in, garbage out。** 你 chunking 和 embedding 再精妙，如果 load 出来的是把表格拍扁成乱序文字的垃圾，整个 RAG 就废了。很多人花一周调 top-k 和 prompt，问题其实在 load 阶段。

### 2. chunk（切分）——把长文档切成可检索的小块

文档太长不能整篇 embedding（一篇 50 页手册编成一个向量，语义被平均得毫无区分度，且超 embedding 模型输入上限）。必须切成**chunk**——几百到上千 token 的小块，每块单独 embedding、单独检索。

这是**被严重低估的环节**，单列一节细讲（见本章第三节）。先记一句：**chunk 切得好不好，直接决定 RAG 的天花板**——切碎了语义不完整，切大了检索不精准且喂太多噪声。

### 3. embed（向量化）——把文本变成语义坐标

用 **embedding 模型**把每个 chunk 编码成一个定长向量（几百到几千维，按 2026 初常见 768~3072 维）。语义相近的文本，向量在空间里挨得近。这是"语义检索"的物理基础。

几个必须知道的工程点：
- **embedding 模型 ≠ 生成模型**。它是专门训练来"把语义压成向量"的，输出的是向量不是文本。常见有 OpenAI 的 text-embedding 系列、各家开源的 BGE/GTE/E5 等（名单按 2026 初，会变，重点是选型方法）。
- **query 和 document 必须用同一个 embedding 模型**。否则两边向量不在同一空间，相似度无意义。这是新手常翻的车——离线用模型 A 建库，在线 query 用模型 B，召回全乱。
- **换 embedding 模型 = 整库重建**。向量空间变了，老向量全作废。所以模型选型要慎重，这是个有迁移成本的决策。
- **有的模型区分 query 和 passage 的编码方式**（非对称 embedding，给 query 和文档加不同前缀），用错前缀掉点。看模型文档。
- **成本**：embedding 也按 token 收费，但比生成便宜得多（量级上 embedding 输入价 ≈ 生成输入价的零头）。大库首次建索引是一次性大开销，之后增量很小。

> 关于 Claude：**Anthropic 自己不提供 embedding 模型**（按 2026 初），官方建议用第三方 embedding（如 Voyage AI、各家开源模型）做检索，再把检索结果喂给 Claude 生成。所以一个典型的"Claude RAG"是**混血架构**：第三方 embedding 负责检索，Claude 负责 grounding 生成。别去找"Claude embedding API"，没有。

### 4. index（建索引）——让海量向量能快速检索

几百万个向量，挨个算相似度（暴力 KNN）太慢。需要**向量索引**（HNSW、IVF 等近似最近邻 ANN 算法）做亚线性检索。落地用向量库——pgvector（Postgres 扩展，适合已有 PG 的团队）、Chroma（轻量、本地开发友好）、Qdrant/Milvus/Weaviate（专用向量库，规模大）等（名单按 2026 初）。

关键认知：**ANN 是用一点点召回精度换巨大速度提升**——它不保证返回的就是数学上最近的 k 个，而是"很可能是"。索引参数（如 HNSW 的 `ef_search`）就是在调"精度 vs 速度"这根杠杆。库小（几万条）可以暴力检索图省事，库大必须用 ANN。

**索引必须同时存元数据。** 每个 chunk 除了向量，还要存原文、来源文档 ID、页码/章节、时间戳、ACL 标签等。原因：检索回来要能拼回原文给模型、要能溯源给用户、要能按元数据过滤（时间/权限/文档类型）。**只存向量不存元数据是残废索引**——你召回了向量却不知道它对应哪段原文，没法用。

### 5. retrieve（检索）——把 query 转成向量去捞候选

在线阶段第一步：`embed(query)` 把用户问题编成向量，去索引里捞 **top-k** 个最相似的 chunk（k 常取 10~50 做粗召回）。

坑点集中区：
- **query 和 document 的不对称**。用户问"退货咋弄"，文档是规范的"退款政策第三条"。口语 query 和书面文档在向量空间可能离得有点远——这是 naive RAG 召回不全的主因之一。**治理手段（query 改写/扩展/HyDE）在第 9 章**。
- **top-k 的取舍**：k 太小漏召回（答案所在的 chunk 没捞上来，后面神仙难救）；k 太大引入噪声且费 token。粗召回阶段宁可 k 大一点（捞全），靠后面的 rerank 去精排（去噪）。
- **检索的是 chunk 不是文档**。返回的是切碎的小块，可能答案横跨两个 chunk 而你只召回了一个——这又指回 chunking 质量。

### 6. rerank（重排）——粗召回之后的精排

向量检索（top-k）是**粗召回**——快但不够准（ANN 近似 + 向量相似度本身的局限）。**rerank** 用一个更重、更准的模型（cross-encoder/reranker）对这 k 个候选重新打分，选出真正最相关的 **top-n**（n 常取 3~5）喂给生成。

为什么需要这一步：向量检索是"双塔"——query 和 document 各自独立编码再算距离，编码时彼此看不见，精度有上限。reranker 是"交互式"——把 query 和每个候选**拼在一起**送进模型打分，能捕捉细粒度相关性，准得多。代价是慢（每个候选一次模型调用），所以只能对粗召回的少量候选用，不能对全库用。

**这是 naive → 生产 RAG 最高性价比的一次升级**。很多 RAG 效果差，加一个 rerank 立竿见影。**rerank 的工程细节（选型、延迟权衡、和混合检索的配合）在第 9 章**，这里你只需在 P2 里留出这个环节的位置。

> 工程推论：**检索是『粗召回 + 精排』两段式，和延迟的『TTFT + decode』两段式异曲同工——分两段、各用不同手段。** 粗召回求全（向量检索，k 大），精排求准（rerank，n 小）。把这两段揉成一段（只做向量检索就直接喂模型）是 naive RAG 的典型形态，也是效果上不去的典型原因。

### 7. generate（生成）——拼 prompt，让模型基于材料作答

把 top-n 个 chunk 拼成上下文，连同系统指令和用户问题组装成最终 prompt，调生成模型。**这一步决定了检索到的材料能不能被模型正确利用**——检索做得再好，prompt 组装烂，照样白搭。单列一节细讲（见本章第五节）。

核心是 prompt 的三段结构：**系统指令（怎么用材料、不知道就说不知道）+ 检索材料（带来源标记）+ 用户问题**。

### 8. cite（引用溯源）——答案要能指回来源

生产级 RAG 的答案不能是"一段无出处的话"，必须**标注每个论断来自哪个 chunk/文档**。这是 RAG 相对微调的核心优势（可溯源），也是用户信任的来源——用户能点开看原文核对，发现错了能反馈。引用溯源的实现方式和 prompt 组装强耦合，也在第五节一起讲。

> 工程推论：**离线/在线分离是 RAG 工程的骨架认知。** 离线管道（load→chunk→embed→index）追求"建得全、建得对、可重建"，是批处理思维；在线路径（retrieve→rerank→generate→cite）追求"低延迟、可溯源、答得准"，是请求响应思维。把这两套混在一起想（比如以为"用户提问时才去切文档"）是根上的误解——切分和 embedding 是**提前**做好的，提问时只做检索和生成。

## 三、chunking 策略：被低估的脏活，决定 RAG 天花板

如果 RAG 只让你记一个工程细节，记 chunking。它朴素到不像技术活，却**单点决定了你的检索质量上限**。理由很简单：chunk 是检索的最小单位，**切坏了，后面所有环节都在抢救一份残缺的语义**。

### 1. 核心矛盾：chunk 大小的双向拉扯

chunk 大小不是个能"调到最优"的旋钮，是个**两头不讨好的 trade-off**：

| chunk 太小（如 100 token） | chunk 太大（如 2000 token） |
|---|---|
| 语义不完整，一个完整论点被切断 | 一个 chunk 混多个主题，语义被"平均"稀释 |
| 检索时答案横跨多块，召回不全 | embedding 向量区分度差（什么都沾一点） |
| 单块信息量不足，模型缺上下文 | 检索精度低（query 只匹配块里一小部分） |
| 块数多，索引大、检索候选多 | 喂给模型噪声多、费 token、易触发 lost-in-middle |

**关键洞察：chunk 大小同时拉扯『检索』和『生成』两端，且方向相反。**
- 站在**检索**角度：小 chunk 更精准（向量聚焦单一语义，query 容易精确命中）。
- 站在**生成**角度：大 chunk 上下文更完整（模型拿到的材料自洽，不会因为切断而缺前因后果）。

这就是 chunking 没有银弹的根因——**你为检索精度切小，就牺牲了生成的上下文完整性；为生成完整性切大，就牺牲了检索精度。** 经验起点是 **256~512 token**（按英文，中文按 token 实测会更紧），但"最优值"强依赖你的文档类型和查询模式，必须用第 11 章的评测去验证，别信任何"标准答案"。

> 坑点提示（token 计数是模型相关的，别跨模型套）：你用什么数 token，决定了 chunk 真实多大。**token 不是字符、也不是词，而且每个模型的 tokenizer 不一样。** 最常见的车祸是：embedding 用 A 家、生成用 Claude，却统一拿 OpenAI 的 `tiktoken` 来估 chunk 大小——而 **tiktoken 估 Claude 会系统性偏差 15~20%**（中文/代码偏得更多）。后果是你以为切了 512 token，Claude 侧实际可能是 600+，既可能撑爆 embedding 输入上限，也可能让你算的"上下文预算"全错。**规矩：chunk 大小要用你『实际生成所用模型』的 tokenizer 来量；Claude 没有公开 tokenizer，得调 `count_tokens` API，别拿 tiktoken 替代**（按 2026 初，以官方为准）。这条看着是小细节，但它直接污染你后面所有的"token 预算"计算。

> 高级解法预告：这个矛盾有个经典缓解——**small-to-big / 父子 chunk**：用**小 chunk 做检索**（精准命中），命中后**返回它所属的大 chunk/父文档**给模型（上下文完整）。检索和生成各取所需，绕开了二选一。这是第 9 章的进阶检索策略，这里先种下种子：**当你纠结 chunk 该切大还是切小时，正确答案往往是"检索用小的、喂给模型用大的"。**

### 2. overlap（重叠）——缝合切断的语义

切分时让相邻 chunk **重叠**一部分（如每块和上一块共享 50~100 token），目的是**防止一个完整语义恰好被切口切断**。比如一句话被切成两半分到两个 chunk，没有 overlap 的话两块都不完整；有 overlap，至少有一块包含完整的这句话。

overlap 的取舍：
- **太小**：缝合效果弱，边界语义仍可能丢。
- **太大**：冗余高，同一段内容反复入库——索引膨胀、检索时容易召回大量高度重复的块（挤占 top-n 名额）、成本上升。
- 经验值：**overlap 取 chunk 大小的 10~20%**。这是个缝合 vs 冗余的平衡点。

> 坑点提示：overlap 不是越大越保险。我见过有人设 50% overlap，结果 top-5 召回里三个块内容大半重复，等于浪费了生成上下文的名额，答案反而变差。overlap 是创可贴不是万能胶——**结构化切分（下面讲）比加大 overlap 更治本**。

### 3. 四种切分策略——从粗暴到智能

| 策略 | 怎么切 | 优点 | 缺点 | 适用 |
|---|---|---|---|---|
| **固定大小** | 按固定 token/字符数硬切，不管语义 | 实现最简单、可预测 | 经常从句子/单词中间切断，语义破碎 | 快速原型、文本结构均匀 |
| **递归切分** | 按分隔符优先级递归切（先按段落 `\n\n`，太大再按句号，再按词）| 尽量在自然边界切，兼顾大小 | 仍是启发式，不理解真实语义 | **通用首选**，大多数文档的默认 |
| **语义切分** | 用 embedding 算相邻句子相似度，在"话题转折点"切 | 切口落在语义边界，块内主题一致 | 慢、贵（要算大量句子 embedding）、实现复杂 | 主题混杂的长文档、对质量敏感 |
| **结构化切分** | 按文档固有结构切（Markdown 标题层级、HTML 标签、代码函数、表格行）| 完美贴合文档逻辑结构，保留层级 | 依赖文档有清晰结构、需按格式定制 | **有结构的文档**（技术文档、API doc、带 schema 的数据）|

**选型方法论（不是背名单，是建判据）**：
1. **默认从递归切分起步**——它是"够好且通用"的 80 分方案，绝大多数项目用它就够。
2. **文档有清晰结构（Markdown/HTML/代码/表格）→ 优先结构化切分。** 按标题层级切，每个 chunk 自带"它属于哪一节"的上下文，检索质量直接上一个台阶。这是性价比最高的升级。**结构化切分通常该和元数据一起做**——把"章节标题路径"塞进 chunk 元数据，检索时连标题一起匹配。
3. **递归 + 结构化都搞不定（长篇叙事、主题频繁切换、且无明显结构）→ 才上语义切分。** 它贵且慢，是最后手段不是默认选项。
4. **始终用评测验证（第 11 章）。** chunk 策略和大小的"最优解"是你的数据决定的，不是教程决定的。换个文档集，最优配置可能完全不同。

> 工程推论：**chunking 不是"切文本"，是"为检索单元设计语义粒度"。** 你切的每一块，本质是在定义"检索能命中的最小完整语义"。粒度设计错了（太碎/太糊/切断逻辑），整个 RAG 的召回质量被锁死在低位，下游再优化都是隔靴搔痒。这就是为什么"被低估"——它看着像字符串操作，实则是 RAG 架构里影响最大的设计决策之一。

> 面试加分点：被问"你的 RAG 召回率上不去怎么排查"，能把 chunking 列为首要嫌疑（而不是一上来调 top-k），并说清"先看 chunk 是不是切断了语义、是不是该用结构化切分、是不是该上父子 chunk"，说明你抓到了 RAG 的真正杠杆点。

## 四、ingestion 工程：被忽视的数据管道

demo 里 ingestion 就是个一次性脚本——读文档、切块、灌库，跑完拉倒。**生产里 ingestion 是一个需要长期维护的数据管道**，因为知识库不是静态的：文档天天在改、在加、在删。把 ingestion 当一次性脚本，是从 demo 到生产最大的认知断层之一。

### 1. 增量更新——知识库是活的

文档变了，向量库得跟着变。三种变化各有处理：
- **新增文档** → 切块、embedding、插入。简单。
- **删除文档** → 必须把它对应的**所有 chunk 向量**从库里删干净。漏删 = 模型会检索到已删除/已下线的内容,答出过时甚至错误信息。
- **修改文档** → 最容易做错。**不能只追加新版本**——否则库里新旧版本并存，检索可能召回旧版，答案自相矛盾。正确做法：**先删掉该文档的旧 chunk，再插入新 chunk**（按文档 ID 关联）。

> 坑点提示：增量更新做不好,会出现"幽灵知识"——文档早删了/改了,但旧向量还在库里被检索到,模型一本正经地引用一份不存在的文档。这类 bug 极难排查(代码没问题、检索没问题、就是数据脏了),**根因永远在 ingestion 管道没把删除/更新做干净**。设计 ingestion 时,"文档 ID → 它的所有 chunk"这个映射必须能快速反查,才能精准删改。

### 2. 去重——别让同一份知识占满 top-n

真实知识库里**重复内容无处不在**：同一份文档传了两遍、wiki 里复制粘贴的段落、不同文档引用了相同的免责声明。不去重的后果：检索 top-n 里塞满几乎一样的 chunk,**挤占了本该留给多样化信息的名额**,模型拿到的有效信息变少,答案变差。

去重发生在两个层面:
- **入库前去重**：基于内容 hash 或高相似度,同一段内容只入库一次。源头治理,最干净。
- **检索后去重(MMR 等)**：检索回来的候选里,用 Maximal Marginal Relevance 之类的算法,在"相关性"和"多样性"间平衡,避免 top-n 高度同质。这属于检索优化,偏第 9 章。

### 3. 元数据——RAG 的隐藏关节

每个 chunk 必须带丰富元数据,这是**生产 RAG 和玩具 RAG 的分水岭之一**。元数据至少应包含:
- **来源**：文档 ID、文档名、URL/路径——用于**引用溯源**(没有它没法 cite)。
- **位置**：页码、章节、标题路径——用于精确定位和给用户跳转。
- **时间**：创建/更新时间——用于**按时效过滤**(只检索近三个月的)和处理版本。
- **权限**：ACL 标签/部门/可见范围——用于**权限过滤**(检索时就排除用户无权看的 chunk,从源头防泄露)。
- **类型**：文档类别/语言/标签——用于**带过滤的检索**(metadata filtering),先按元数据缩小范围再做向量检索,又快又准。

**元数据过滤是被低估的提效手段。** "查 2026 年 Q1 的财报数据"——先按 `时间=2026Q1 AND 类型=财报` 过滤,再在这个子集里向量检索,比在全库做语义检索又快又准。很多检索精度问题,用元数据预过滤就解决了,根本不用动 embedding。

> 工程推论:**权限必须在检索层落地,不能指望 prompt 兜底。** 错误做法是把所有文档都检索出来,然后在 prompt 里写"别泄露 B 部门的内容"——这是把安全寄托在模型自觉上,第 1 章推论四早说了模型不可信。正确做法是**检索时就用 ACL 元数据过滤,无权的 chunk 根本不进候选集**。模型连看都看不到的东西,才是真安全。这是后端工程师的本能,但 AI 新手常忽略。

### 4. 版本——可重建、可回滚

embedding 模型会升级、chunking 策略会调整。每次这类变更都意味着**整库重建**。所以 ingestion 管道要支持:
- **可重建**：从原始文档(留底!别只存切完的 chunk)一键重建整个索引。
- **版本化**：新索引建好、验证通过,再切流;出问题能回滚到老索引。
- **embedding 版本标记**：chunk 元数据里记是哪个 embedding 模型/版本生成的,避免新旧向量混用。

> 心法:**把 ingestion 当成一个有 schema、有版本、能重建、能增量的 ETL 管道来设计,而不是一个 `for doc in docs: index(doc)` 的脚本。** 这是后端老手能立刻甩开纯 AI 背景的人的地方——你懂数据管道,而 RAG 的一半工程就是数据管道。P2 项目里,ingestion 哪怕简陋,也要把"删除/更新/元数据"这三件事的位置留出来。

## 五、生成阶段：prompt 组装与引用溯源

检索把"对的材料"找出来了,生成阶段负责"让模型正确地用这些材料,并说清出处"。这一步看着简单(不就是拼字符串调 API 吗),坑却不少。

### 1. prompt 三段结构

一个 RAG 生成 prompt 的标准骨架:

```
[系统指令]
你是一个基于知识库的问答助手。请严格根据下面提供的【参考材料】回答用户问题。
- 只用材料里的信息作答,不要用你自己的知识补充。
- 如果材料里没有答案,直接说"根据现有资料无法回答",不要编造。
- 在每个论断后用 [来源N] 标注它来自哪段材料。

[参考材料]
[来源1]（出处:产品手册 v3, 第12页）
{chunk_1 原文}

[来源2]（出处:FAQ, "退货政策"条目）
{chunk_2 原文}

[来源3]（出处:...）
{chunk_3 原文}

[用户问题]
{user_query}
```

几个关键设计:
- **明确指令"只基于材料、没有就说不知道"**。这是抑制幻觉的最后一道闸——告诉模型宁可不答也别编。第 1 章推论一说幻觉是高概率延续,这条指令是在**人为压低"编造"的概率、抬高"承认不知道"的概率**。但记住:这只是降低,模型仍可能违反(见下方失效模式),指令不是保证。
- **每段材料带来源标记**(`[来源N]` + 出处元数据)。这是**引用溯源的实现基础**——模型答题时引用 `[来源N]`,你在后处理时就能把它映射回真实文档链接。没有这个标记,模型答完你不知道每句话从哪来。
- **材料放中间还是问题放最后**:有讲究。检索材料长时,把用户问题**同时放在材料前和材料后**(或至少放最后)能缓解 lost-in-the-middle(模型对超长上下文中间部分注意力弱)。

### 2. 引用溯源——三种实现路线

让答案可溯源,从粗到精三种做法:

1. **prompt 内联标记(最常用)**:如上,让模型在生成时自己插 `[来源N]`,你后处理把标记替换成可点击链接。简单有效,但依赖模型听话(它可能标错或漏标)。
2. **结构化输出**:要求模型输出 JSON,每个论断带 `claim` 和 `source_ids` 字段。比内联标记更可控、可校验(能检查引用的 source_id 是否真存在)。对 Claude 用 `output_config.format` 配 json_schema 约束输出结构(见第 2 章),比让它自由发挥可靠。
3. **句子级归因(最严格)**:答案生成后,把每句话拿去和检索材料做相似度匹配/二次验证,确认每句确有出处支撑,没有的标红。最可靠也最重,用于高合规场景(法律、医疗)。

> 关于 Claude 的原生引用:Anthropic API 提供 **Citations 功能**(按 2026 初)——你把文档作为 document 块传入,模型生成时能直接返回结构化的引用(精确到原文 span),无需自己解析标记。这是比"让模型自插 `[来源N]`"更可靠的路线,因为引用是 API 层保证的结构化数据而非模型自由文本。不过有约束:Citations 和结构化输出(`output_config.format`)互斥,不能同时用——要引用就用 Citations,要强制 JSON 结构就用结构化输出,二选一。代价是要把材料按它的 document 格式组织。**如果你的 RAG 跑在 Claude 上,优先考虑原生 Citations,别急着手撸标记解析。**

### 3. 上下文窗口与材料预算

把检索材料拼进 prompt,要算账:
- top-n 材料的总 token + 系统指令 + query + 预留的输出空间(`max_tokens`),**总和不能超上下文窗口**。
- 材料太多会触发两个问题:**成本飙升**(输入 token 线性收费)和 **lost-in-the-middle**(塞太多反而稀释注意力,关键信息淹没在噪声里)。
- **不是塞越多材料越好**。这是反直觉但重要的点:n=20 不一定比 n=5 好,因为多出来的 15 个低相关 chunk 是**噪声**,会干扰模型抓住真正有用的那几个。**rerank 选出精准的 top-n(小而准)往往比 top-k 全塞(大而杂)效果好**——又回到"精排求准"。上下文压缩(把检索材料先精炼再喂)是第 9 章的治理手段。

> 工程推论(RAG 的成本模型,必须刻进脑子):**RAG 把成本结构从『输出主导』偏向了『输入主导』,但要命的是输出仍然按 ~5 倍单价收费。** 拆开看:检索材料注入 prompt → **输入 token 暴涨**(top-n 个 chunk 动辄几千 token);而答案通常不长 → 输出 token 不大。听起来"输入便宜、输出贵,那 RAG 成本可控"?错在哪——按 2026 初的价格量级,**输出单价≈输入单价的 5 倍**(各家普遍如此,以官方价目为准)。这条不对称推出两个反直觉的优化优先级:
> - **省钱的第一杠杆是『砍输入』而不是『砍输出』的直觉是错的——要分场景。** 输入便宜但量大,输出贵但量小,到底哪头是成本大头,取决于你的 n 和答案长度。**正确做法是先用 `count_tokens` 实测一次典型请求的输入/输出 token,按各自单价算出占比,再决定优化哪头**——别拍脑袋。这正是后端工程师该有的本能:先 profile 再优化。
> - **rerank 把 n 从 20 砍到 5,省的不只是效果(去噪),还有真金白银的输入 token**——一举两得。所以"加 rerank"在生产 RAG 里同时是质量优化和成本优化,这是它性价比高的隐藏原因。
>
> 把这条记牢:**谈 RAG 成本不能只说『token 收费』,要说清『输入量大但便宜、输出量小但 5 倍贵』这个不对称,优化前必须先实测两头的实际占比。**

> 工程推论:**检索材料注入 prompt 后,这部分内容是『每请求都变』的,会击穿 prefix 缓存。** 第 2 章讲过 prompt caching 是前缀匹配,任何字节变化使其后全部失效;渲染顺序是 tools→system→messages。所以 RAG prompt 的正确缓存布局是:**把固定不变的系统指令放最前面打 cache 断点(命中缓存),把每次都变的检索材料和 query 放在断点之后**。如果你手贱把检索材料拼进 system prompt 的中间,等于让每次请求的缓存从那里全部失效,白白多花钱。**稳定前缀在前、易变内容在后**,是 RAG 省钱的关键布局。
>
> 但这里有个**反直觉的结论必须想透**:在 naive RAG 里,prompt caching 能省的钱**远比你以为的少**。原因是——每次 query 检索回来的 chunk 几乎都不一样,而检索材料往往是 prompt 里最大的一块。能进缓存的只有那段固定的系统指令(几百 token),易变的检索材料(几千 token)永远进不了缓存、每次全价重算。所以"给 RAG 加 prompt caching"对单纯的问答场景收益有限,**真正吃到缓存红利的是这些形态**:(a) 同一份长文档被反复追问(把文档钉进稳定前缀,问题放后面——这正是『文档问答』而非『知识库检索』);(b) 系统指令本身很长(塞了大量 few-shot 示例、输出规范);(c) agentic RAG 多轮里前缀稳定的那部分历史。判据:**缓存省的是『稳定前缀 × 重复次数』,RAG 的检索材料天然不稳定,别指望它进缓存。** 缓存读约 0.1×、写约 1.25×(5 分钟 TTL),稳定前缀至少要重复用两次才回本——按 2026 初版图,以官方为准。

## 六、naive RAG 的典型失效模式清单

"naive RAG"指最朴素的实现:固定切分 → 向量检索 top-k → 直接拼 prompt 生成,没有 rerank、没有 query 改写、没有混合检索。它**demo 惊艳、上线拉胯**——因为 demo 的问题都是"软球",真实用户的问题千奇百怪。下面是 naive RAG 的失效模式全清单,**每一条的成因在本章已埋,逐个治理在第 9 章**。把这张表当成你 debug RAG 的检查清单。

| # | 失效模式 | 现象 | 根因(本章已讲) | 治理方向(第 9 章) |
|---|---|---|---|---|
| 1 | **检索没召回** | 答案明明在库里,却没被检索到 | query 与 document 表述不对称;chunk 切断了答案;纯向量检索漏字面匹配 | query 改写/扩展、混合检索(向量+BM25)、改进 chunking |
| 2 | **召回了但不相关** | top-k 里塞满低相关 chunk | 向量相似度精度有限;只做粗召回没精排 | 加 rerank(精排)、提高检索阈值 |
| 3 | **答案在多个 chunk 里凑** | 答案横跨多块,只召回片段,答不全 | chunk 切太小/切断逻辑;单块信息不完整 | 父子 chunk(small-to-big)、加大 chunk、overlap |
| 4 | **lost in the middle** | 塞了一堆材料,关键的那条被忽略 | 上下文太长,模型对中间注意力弱;材料太多噪声 | rerank 选精材料、上下文压缩、减少 n |
| 5 | **幻觉(没材料硬答)** | 材料里没有,模型仍编一个答案 | 第 1 章推论一:编造是高概率延续;指令约束不够强 | 强化"没有就说不知道"指令、答案归因校验、置信度过滤 |
| 6 | **幻觉(曲解材料)** | 材料里有,但模型理解错/张冠李戴 | 模型对材料的概率延续偏了;多源材料混淆 | 句子级归因校验、减少同时塞入的材料数、引用溯源 |
| 7 | **过时/幽灵知识** | 答出已删除或已改的旧内容 | ingestion 没做好删除/更新;旧向量残留 | 增量更新管道、按时间元数据过滤 |
| 8 | **权限泄露** | 答出用户无权看的内容 | 检索没按 ACL 过滤,全库召回 | 检索层元数据/ACL 过滤(本章已强调) |
| 9 | **多跳问题答不了** | 需要"先查A推出B再查C"的问题,一次检索搞不定 | 单轮检索拿不全多步推理所需材料 | 多步检索、agentic RAG(第 9 章) |
| 10 | **重复材料挤占** | top-n 全是几乎一样的内容 | 没去重;overlap 过大 | 入库去重、MMR 多样性重排 |

**怎么读这张表**:左边是你会观测到的现象,中间是病根,右边是药方。debug RAG 的正确姿势是**先定位是哪一类失效**(从现象倒推),而不是一上来乱调参。比如"答案明明在库里却答错",先分清是**没召回**(检索问题,看 1/3)还是**召回了但答错**(生成/曲解问题,看 5/6)——这两类的治理方向完全不同。

> 工程推论:**naive RAG 的失效,八成不是模型不行,是检索/数据/组装这三个工程环节没做好。** 现象表现在"模型答错",但根因在 chunking、检索粗糙、ingestion 脏、prompt 组装烂。这正是后端工程师的主场——这些全是你熟悉的检索系统、数据管道、缓存布局问题。**把 RAG 的锅甩给"模型不够聪明",是没找到真正的杠杆点。**

> 心法(本章收尾):**RAG 的核心不是『让模型更聪明』,是『把对的材料,以对的粒度、对的相关性、对的结构,在对的时候放进它的上下文』。** 八环节流水线里的每一环,都是在为这一件事服务。模型只是最后那个"基于材料做概率延续"的执行者——它答得好不好,九成取决于你喂给它的材料对不对。这一章你搭的是骨架(端到端跑通),第 9 章把每个环节从 naive 升级到生产级。**记住这条因果链:答案质量 ← 材料质量 ← 检索质量 ← chunking 质量 ← 数据质量。 从右往左,每一环都不能塌。**

## 章末五件套

### 一、高频面试题(附答题框架)

**Q1：RAG 和微调怎么选?能不能一起用?**
- 答题框架:先抛判据——**"先问需求是缺知识还是缺行为"**。缺知识(事实、私有文档、时效)→ RAG;缺行为(格式、语气、领域措辞)→ 微调。再补:**二者不互斥,生产常合用**(微调定风格 + RAG 供事实)。最后给边界证明想清楚了:"但要模型严格输出某 JSON schema,那是行为问题,RAG 帮不上。"
- 错误陷阱:只罗列优缺点不给判据(显得没结构);说"微调能灌知识所以更省事"(对事实记忆不可靠,且不可溯源、难更新)。

**Q2：chunk 大小怎么定?越大越好还是越小越好?**
- 答题框架:讲清**双向拉扯**——小 chunk 利检索(精准)但损生成(上下文断);大 chunk 利生成(完整)但损检索(区分度差)。给经验起点(256~512 token)但强调**最优值靠评测定,依赖文档和查询模式**。加分:抛出 **small-to-big(小块检索、大块喂模型)**绕开二选一。
- 错误陷阱:给一个固定"最佳值"(说明没理解这是 trade-off);只说大小不提 overlap 和结构化切分。

**Q3：你的 RAG 召回率上不去,怎么系统排查?**
- 答题框架:**先分层定位**——是没召回(检索问题)还是召回了答不对(生成问题)?没召回:查 chunking 是否切断语义、query 与文档是否不对称(上 query 改写)、是否该混合检索(向量漏字面)。召回了答不对:加 rerank 精排、查是否 lost-in-middle、查 prompt 指令。**强调先看 chunking 而非先调 top-k**。
- 错误陷阱:一上来"调大 top-k / 换更大模型"(没找杠杆点);分不清"没召回"和"曲解材料"是两类问题。

**Q4：RAG 怎么抑制幻觉?能彻底消除吗?**
- 答题框架:先纠正前提——**RAG 抑制不消除**(第 1 章:幻觉机制内生)。手段三层:grounding(检索原文进上下文,让正确答案成高概率延续)+ 约束(prompt 指令"没有就说不知道"、结构化输出)+ 检测(引用溯源、答案归因校验)。强调即便有 RAG,模型仍可能曲解材料或违反指令硬答。
- 错误陷阱:说"RAG 能消除幻觉"(过度承诺,naive RAG 失效模式 5/6 就是反例);只说"加 RAG"不展开三层手段。

**Q5：解释 RAG 的完整流水线,离线和在线分别做什么?**
- 答题框架:**两阶段八环节**。离线(Ingestion):load→chunk→embed→index,提前建好知识库,批处理思维。在线(Retrieval):retrieve→rerank→generate→cite,提问时实时跑,低延迟思维。强调**切分和 embedding 是提前做的,提问时只检索+生成**——这是常见误解点。
- 错误陷阱:以为"用户提问时才切文档"(混淆离线/在线);漏掉 rerank 和 cite(那正是 naive→生产的关键)。

**Q6：检索为什么要分"粗召回 + rerank"两步,不能一步到位吗?**
- 答题框架:向量检索是**双塔**(query/doc 独立编码,精度有上限)但快,适合粗召回求全(k 大);rerank 是**交互式**(query+候选拼一起打分)准但慢,适合精排求准(n 小)。一步到位(只向量检索直接喂)就是 naive RAG,效果上不去的典型。类比延迟的 TTFT+decode 两段式。
- 错误陷阱:说"向量检索够了不用 rerank"(naive RAG 主要短板之一);说不清双塔 vs 交互式的精度差异来源。

**Q7：知识库里文档更新了,RAG 怎么保证不答旧内容?**
- 答题框架:**ingestion 增量更新**是关键。修改文档要**先删旧 chunk 再插新 chunk**(按文档 ID 关联),不能只追加(否则新旧并存召回旧版)。删除要删干净所有 chunk。强调"文档ID→其所有chunk"的反查映射必须高效。点出"幽灵知识"bug 难排查,根因永远在 ingestion 没做干净。
- 错误陷阱:只想到"加新文档"忘了删除/更新;以为靠 prompt 提示模型"用最新的"能解决(数据脏了 prompt 救不回)。

**Q8(进阶)：怎么保证 RAG 不泄露用户无权访问的内容?**
- 答题框架:**权限必须在检索层落地**,不能靠 prompt 兜底(第 1 章推论四:模型不可信)。做法:chunk 元数据带 ACL 标签,检索时按用户权限过滤,**无权 chunk 根本不进候选集**——模型看都看不到才是真安全。对比错误做法(全库召回 + prompt 写"别泄露")。
- 错误陷阱:把安全寄托在模型自觉(prompt 说"别说机密");没意识到这是检索层(元数据过滤)而非生成层的问题。

**Q9(进阶,易被忽略)：一套 RAG 上线后成本主要花在哪?怎么砍?**
- 答题框架:先拆**一次性 vs 复发**——建库 embedding 是一次性的(几 GB token、单价低);**真正的长期大头是每次查询的生成成本**(日均 N 次 × 每次的输入+输出 token)。再点关键不对称:**检索材料注入让输入 token 暴涨,但输出单价≈输入的 5 倍**(按 2026 初量级),算账要分开两头、别按同一单价。砍法分三层:(a) **rerank 把 n 砍小**——同时去噪 + 省输入 token,一举两得;(b) **prompt caching 看场景**——naive 问答里检索材料每次都变进不了缓存,只有"同文档反复追问/长系统指令/多轮 agentic"才吃到红利;(c) **分流模型**——高频简单查询用更小更便宜的模型,难query 才上大模型。
- 错误陷阱:只说"token 收费"不拆输入/输出和一次性/复发;以为"给 RAG 套个 prompt caching 就省钱"(检索材料天然不稳定,缓存省不到它);优化前不先用 `count_tokens` 实测两头占比就拍脑袋砍。

### 二、实战项目增量(P2 v1:端到端知识库问答,带引用溯源)

### 三、设计题(考权衡判断)

**题目**:你要为一个**法律合同知识库**设计 RAG。文档特点:每份合同 20~50 页,有严格的条款编号结构(第一条、第二条...),用户查询多是"X 情况下违约责任怎么算""第几条关于保密的规定"。要求答案**必须精确溯源到具体条款**,且**绝不能编造**(法律场景容错率极低)。

### 四、系统设计题(含量级估算)

**题目**:为一个 **10 万篇文档、平均每篇 5000 字** 的企业知识库设计 RAG 系统,日均 **5 万次**问答查询,峰值 QPS 约 **10**。请估算并设计:

1. **向量规模估算**:按你的 chunk 策略,10 万篇文档大约切出多少 chunk?(给出你的 chunk size 假设和算法)总向量数量级?按 1024 维 float32,索引大约占多少存储?
2. **建索引成本(一次性)**:首次全量 embedding 的 token 量级和成本量级?(按 embedding 价目自己算,标注按 2026 初)
