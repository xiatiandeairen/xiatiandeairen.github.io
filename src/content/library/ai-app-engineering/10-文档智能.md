> 替你解决的问题：让你停止把"文档解析"当成 `pdf.extract_text()` 一行代码的体力活，开始把它当成**RAG 全链路的上游瓶颈和第一性约束**。一句话——**garbage in, garbage out 在 RAG 里不是格言，是物理定律**。这章给你三条解析路线的可执行选型判断、表格抽取为什么是地狱、版面如何反向决定 chunking、以及多模态 LLM 直读 PDF 这个 2026 的新变量。

## 一、背景：真实文档为什么这么脏

### 1. 一个残酷的认知前提：PDF 不是数据格式，是打印格式

这是整章的地基，也是新手第一误区的根源。**PDF（Portable Document Format）的设计目标是"在任何设备上像素级一致地呈现给人看"，不是"让机器读取结构化数据"。** 它本质上是一套**绘图指令集**——"在坐标 (72, 720) 放字符 'H'，字体 Helvetica，12pt；在 (79, 720) 放 'e'……"。

这意味着一个反直觉的事实：**PDF 文件里通常根本不存在"段落""表格""标题""阅读顺序"这些概念。** 它存的是一堆**散落在二维平面上的字符 + 坐标**。你看到的"一个表格"，在文件层面是几百个孤立的文本片段，恰好被摆放成了网格的样子——但文件里**没有任何地方写着"这是一个表格，它有 5 行 3 列"**。

> 新手第一误区：以为 `pdf.extract_text()` 拿到的是"文档内容"。**它拿到的是按某种顺序（往往是字符在文件里被写入的顺序，而非视觉阅读顺序）拼接起来的字符流。** 一个双栏 PDF，抽出来的文本极可能是"左栏第一行 + 右栏第一行 + 左栏第二行……"交错乱序；一个表格，抽出来可能是把所有单元格按某种诡异顺序拍扁成一行。文本"抽出来了"，但**语义结构全丢了**。

### 2. 脏的六种典型形态（每一种都单独能让 RAG 崩）

| 脏数据形态 | 为什么是问题 | 对 RAG 的杀伤 |
|---|---|---|
| **扫描件 / 图片型 PDF** | 整页就是一张图，**没有任何文本层**，`extract_text()` 返回空字符串 | 不上 OCR 直接 0 召回——库里这份文档等于不存在 |
| **多栏排版**（论文、报纸、杂志） | 字符坐标不携带"先读左栏再读右栏"的逻辑 | 抽出来左右栏交错，句子被腰斩，embedding 编的是乱码语义 |
| **复杂表格**（合并单元格、跨页、嵌套表头） | 二维行列关系在一维文本流里彻底丢失 | "2023 年 Q3 华东区营收"这种**需要行列交叉定位**的问题全错 |
| **页眉 / 页脚 / 水印 / 行号** | 每页重复的噪声，跟正文坐标混在一起 | 切 chunk 时混入"第 5 页 共 120 页 机密"这类垃圾，污染检索 |
| **公式 / 化学式 / 代码块** | 用特殊字体和精密定位拼出来，按普通文本抽会变成符号乱码 | 技术文档问答里公式相关的问题全废 |
| **图文混排 / 图表**（chart、流程图、架构图） | 信息**在图里**，不在文本层 | 纯文本管道直接丢失这部分信息，"图 3 显示了什么趋势"无法回答 |

> 心法：**真实业务文档的脏，不是"有一些噪声需要清洗"，而是"结构信息在文件层面就不存在，需要被重新推断出来"。** 清洗是减法（去噪），解析是**加法（重建结构）**。把解析当清洗，是低估了问题的本质。

### 3. 演进：解析方案是怎么一路被逼出来的

- **第一代：纯文本抽取库**（pdfminer、PyPDF2/pypdf、pdfplumber 等）。直接读 PDF 的绘图指令，把字符按坐标拼回文本。**对"数字原生、单栏、无表格"的 PDF 够用**（比如程序生成的简单报告）。一碰到双栏、表格、扫描件就崩。这是 90% 新手第一次写 RAG 时用的，也是 90% RAG demo 上线就拉胯的根因。
- **第二代：OCR + 规则**。扫描件没文本层，上 OCR（Tesseract 等）把图像识别成文字。但裸 OCR 只解决"图→字"，**不解决版面**——识别出一堆字，但哪些是标题、哪些是表格、阅读顺序如何，还得靠人写规则（按坐标聚类、按字号判标题）。规则脆弱、不通用、换个模板就失效。
- **第三代：版面理解模型**（layout understanding / document AI）。用专门训练的模型**同时理解视觉布局和文本**，输出结构化结果——这是标题、这是段落、这是一个 N 行 M 列的表格、阅读顺序是 1→2→3。代表是各类 document layout analysis 模型和商用 Document AI 服务。**质的飞跃**：第一次把"重建结构"这件事交给了学习出来的模型，而非手写规则。
- **第四代（2026 当下的新变量）：多模态 LLM 直读**。把 PDF 页面**当图片直接喂给多模态大模型**，让模型一步到位地"看懂"整页——文字、表格、图表、版面一起理解，直接回答问题或输出结构化文本。Claude / GPT / Gemini 这代模型原生支持 PDF 和图像输入（见本章第三节落地部分）。这条路线模糊了"解析"和"理解"的边界——**解析不再是独立的预处理步骤，可能直接融进了推理**。

> 别把这四代理解成"新的杀死旧的"——**四代在 2026 是并存的，且第三代和第四代正在收敛。** 一个工程上更重要的区分是"通用 frontier 多模态 LLM 直读"和"专门为文档/OCR 微调的视觉模型"：后者（2025-2026 出现了一批专做 OCR / 版面 / 表格的开放权重 VLM）本质上是把第三代的版面理解和第四代的视觉大模型**合并**成"输入一页图、输出结构化 Markdown/HTML"的专用模型，单位成本通常远低于调通用 frontier 模型。**选型方法论不变**：先看文档脏在哪、量多大、预算多少；新手第一误区是"看到多模态就以为传统库该淘汰了"——恰恰相反，海量简单文档永远是传统库的地盘，多模态只是新增了一个"贵但强"的选项。（工具名单按 2026 初会过期，**记的是这个收敛趋势和成本梯度，不是某个模型名**。）

> 工程推论（本节地基，逐条都是面试题）：
>
> **推论一：文档解析的第一性原理是"重建文件里不存在的逻辑结构"。** 不是"读取内容"，是"从字符坐标 + 视觉布局里，反推出人类一眼就懂、机器却看不见的结构层级"。理解了这一点，你才明白为什么这件事这么难、为什么没有银弹。
>
> **推论二：解析是 RAG 全链路里唯一"上游错误不可恢复"的环节。** chunking 切坏了可以换策略重切，检索召回差了可以加 rerank，prompt 不好可以迭代——这些都在"干净文本"之上,可以反复试。但**如果 load 出来的文本本身就是乱序、缺失、错乱的，下游再精妙的工程都是在垃圾上抛光**。这是为什么"花一周调 top-k，问题其实在解析"是 RAG 工程最常见的误诊。

## 二、现状：三条解析路线，以及怎么选

这是本章最高频的工程决策点。面试官问"你的 RAG 怎么处理 PDF"，初级答案是"用 pypdf"，高级答案是**先反问文档长什么样，再在三条路线里给出带 trade-off 的选型**。

### 1. 三条路线对照表（背下来）

| 维度 | 路线 A：传统文本库 | 路线 B：版面理解模型 | 路线 C：多模态 LLM 直读 |
|---|---|---|---|
| **代表** | pypdf、pdfplumber、pdfminer | Document Layout 模型、商用 Document AI、开源 document parser | Claude / GPT / Gemini 直接喂 PDF/图像 |
| **原理** | 读绘图指令，按坐标拼字符 | 学习模型识别版面元素 + 阅读顺序 | LLM 视觉理解整页 |
| **扫描件** | ❌ 无文本层直接废 | ✅ 内置 OCR | ✅ 原生看图 |
| **复杂表格** | ⚠️ pdfplumber 勉强，合并单元格易崩 | ✅ 专门的表格结构识别 | ✅ 视觉理解，跨页/合并较强 |
| **多栏阅读顺序** | ❌ 经常乱序 | ✅ 显式输出 reading order | ✅ 视觉上天然正确 |
| **图表/公式** | ❌ 丢失 | ⚠️ 识别为区域，内容未必懂 | ✅ 能"看懂"并描述 |
| **成本** | 极低（本地 CPU） | 中（GPU 推理 / API 调用费） | **高**（按 token，图像 token 不便宜，见下文） |
| **延迟** | 毫秒级 | 百毫秒~秒级 | 秒级（大文档更慢） |
| **可控性/可调试** | 高（纯确定性） | 中 | 低（黑盒，可能幻觉/漏读） |
| **批量大库 ingestion** | ✅ 便宜可扛百万页 | ⚠️ 要算 GPU 账 | ❌ 百万页成本爆炸 |
| **适用场景** | 数字原生简单文档、量大、预算紧 | 混合文档、有表格、要质量又要可控成本 | 高价值低频文档、复杂图表、demo/MVP 快速起步 |

### 2. 可执行的选型判断（大厂怎么决策）

**第一步：先分类你的文档源。** 不要对整个库用同一条路线——**混合管道（hybrid pipeline）才是生产常态**。典型做法：

```
拿到一份 PDF
   │
   ▼ 检测：有没有文本层？（pdf 文本字符数 / 页数 是否过低）
   ├─ 有文本层 + 结构简单 ──► 路线 A（pypdf/pdfplumber）  ← 便宜快，能用就用
   ├─ 有文本层 + 含表格/多栏 ──► 路线 B（版面模型）        ← 质量与成本的平衡点
   └─ 无文本层（扫描件） ──────► 路线 B（带 OCR）或 路线 C   ← 看预算和复杂度
                                      │
   表格特别复杂 / 图表问答需求 ──────────┘──► 路线 C（多模态直读）兜底
```

**第二步：按"文档价值密度 × 调用频率"决定要不要上 C。** 多模态直读质量最好但最贵。判据：
- **高价值、低频、复杂**（如：一份要反复问的关键合同、财报、含大量图表的尽调材料）→ 值得用 C，质量优先。
- **低价值、高频、海量**（如：百万页工单、日志、简单 FAQ 文档）→ 必须用 A/B，C 的 token 成本会让你破产。

**第三步：国内厂商语境。** 豆包 / Qwen-VL（阿里，开源多模态声量大）/ GLM-4V（智谱）/ 文心等都提供多模态文档理解能力，DeepSeek 等也在跟进 OCR/文档方向；很多国内云厂商还单独提供"文档智能 / OCR / 表格识别"的 Document AI 类 API（路线 B 的商用形态）。选型逻辑跟上面一致——**先看文档脏在哪、量多大、预算多少，再选路线，不要因为"多模态时髦"就无脑上 C**。

> 工程推论：
>
> **推论三：没有银弹，hybrid pipeline 是唯一成熟解。** 任何"我用 X 工具一把梭"的方案在真实异构文档库前都会破。生产级 RAG 的 ingestion 几乎都是**分类 → 路由 → 不同文档走不同解析器**的管道。面试时能说出"我会先做文档分类再路由到不同解析路线"，直接区别于只会调一个库的人。
>
> **推论四：解析路线的选择，本质是"质量、成本、可控性"的三角权衡，且这个权衡要按文档分桶做，不是全局做一次。** 把"对整个库用一种解析器"当默认，是规模化 RAG 的典型架构错误。

### 3. OCR 现状：解决了"图→字"，没解决"字→结构"

OCR（Optical Character Recognition）这些年（尤其是深度学习 + 多模态加持后）在**纯识别准确率**上已经相当能打——印刷体、清晰扫描件，字符识别率很高。但要破除两个幻觉：

1. **OCR ≠ 文档解析。** OCR 把图像变成"带坐标的字符 / 文本行"，**但版面结构（标题层级、表格行列、阅读顺序）仍然要靠 OCR 之上的版面分析来重建**。裸 OCR 给你一堆字，不给你结构。
2. **OCR 错误会沉默地向下游传导。** OCR 把 "0" 认成 "O"、"1" 认成 "l"、"，" 漏掉、表格串行——这些错误进了文本，**embedding 照编不误、检索照跑不误、模型照答不误，没有任何环节会报错**。最后用户拿到一个自信的错误答案。手写体、低质量扫描、复杂排版下，OCR 错误率会显著上升。

> 坑点提示：很多人觉得"现在 OCR 这么强，扫描件不是问题了"。错。**强的是字符识别，弱的依然是结构重建和错误的不可见传导。** 一个 OCR 把财务表的某一列串了行，下游全链路无人察觉——这种错误比"完全没识别出来"更危险，因为它**看起来成功了**。

## 三、深入：三个最难的硬骨头

### 1. 表格抽取：为什么是地狱级难题

表格是文档解析里**单独拎出来都能写一章**的难点。难在哪？

**难点一：二维结构 → 一维文本的信息坍缩。** 表格的语义**全在行列交叉关系里**——"第 3 行第 2 列 = 华东区 Q3 营收"。一旦拍扁成线性文本流，这个交叉关系就丢了。模型拿到 "华东 1200 华南 980 华北 1500" 这种压扁文本，根本不知道哪个数对应哪个区、哪个季度。

**难点二：合并单元格（merged cells）。** 表头经常是"2023 年"横跨 Q1/Q2/Q3/Q4 四列的合并单元格。文件层面这就是一个孤立的"2023 年"文本块，**它该归属于哪几列，没有任何显式信息**，全靠几何位置推断。推断错了，整个表的列对齐就崩。

**难点三：跨页表格（spanning tables）。** 一张大表跨了 3 页，每页顶部可能重复表头、底部混着页脚。解析器要识别"这是同一张表的延续"并把它们拼起来——纯文本库基本做不到，版面模型也未必稳。

**难点四：无边框 / 隐式表格。** 很多表格靠**空白对齐**而非画线来表达（尤其排版精良的报告）。没有可见的格线，连"这是个表格"都难判定,更别说切分行列。

**生产怎么应对：**
- 路线 A 里 **pdfplumber** 有专门的表格抽取能力，对**有清晰边框、规则的表格**还行，复杂表格（合并、跨页）会崩。
- 路线 B 的版面模型 / Document AI 通常有**专门的表格结构识别（table structure recognition）**模块，输出行列网格，是当前**可控成本下的最优解**。
- 路线 C 多模态直读对复杂表格**视觉理解能力最强**，但要警惕**幻觉**——模型可能"脑补"出一个看起来合理但数值错位的表。
- **关键工程动作：表格抽取出来后，转成 LLM 友好的格式再喂下游。** 实践中常把表格转成 **Markdown 表格或 HTML 表格**（保留行列结构的文本表示），而不是拍扁成纯文本。这样下游 LLM 还能"看懂"行列关系。

> 工程推论：
>
> **推论五：表格必须当成"结构化对象"而非"文本"来处理，且要保留到下游。** 把表格拍扁成纯文本塞进 chunk，是 RAG 在表格类问题上全面崩盘的头号原因。正确做法是抽成 Markdown/HTML 表格、或单独存为结构化数据、或作为独立 chunk 并附上下文说明。
>
> **推论六：表格类问答的准确率，是检验你 RAG 解析质量的试金石。** 如果你的 RAG 在"某年某地区某指标是多少"这种行列交叉问题上频繁出错，几乎可以断定是解析阶段把表格搞坏了，而不是检索或生成的问题。

### 2. 版面对 chunking 的决定性影响：上游崩则全链路崩

这是连接本章和第 8 章的关键，也是本章标题"文档智能"的核心论点。

**chunking 的前提是"你知道文档的结构边界在哪"**——按标题切、按段落切、按章节切（见第 8 章 chunking 策略）。而**结构边界正是解析阶段要重建的东西**。所以：

```
解析质量  ──决定──►  你能拿到的结构信号  ──决定──►  chunking 能做到多好  ──决定──►  检索/生成质量
（版面理解）          （标题/段落/表格边界）      （语义完整的 chunk）         （RAG 最终效果）
```

**举三个具体的崩塌链：**

- **崩塌链一（多栏乱序）：** 解析把双栏论文抽成左右栏交错的乱序文本 → chunking 按 token 数硬切，切出来的每个 chunk 都是**半句左栏 + 半句右栏的语义碎片** → embedding 编的是无意义的混合语义 → 检索召回一堆看似相关实则狗屁不通的 chunk → 模型基于碎片硬答 → 幻觉。**源头在解析，但症状在最后一步显现。**
- **崩塌链二（标题丢失）：** 解析没能识别出标题层级，所有文字都是同一级的"段落" → chunking 失去了"按章节切"的依据，只能按固定长度盲切 → 一个 chunk 横跨两个不相关小节，主题不纯 → 检索精度下降。
- **崩塌链三（表格拍扁）：** 解析把表格压成一行文本 → 这行超长文本要么被切碎（行列关系彻底没了），要么整块塞进一个 chunk（embedding 对一堆数字编不出有区分度的语义） → 表格问答全错。

> 心法：**chunking 的天花板，是解析质量。** 你在第 8 章学的所有 chunking 策略（按结构切、保留 overlap、附元数据）都**隐含假设"结构是已知的"**——而这个假设的成立与否，完全取决于这一章的解析。**先把解析做对，chunking 才有意义。** 这就是为什么我把这章放在 RAG 阶段的中游、且反复强调"上游"。

> 面试加分点：被问"你怎么做 chunking"，只答策略（固定长度/递归/语义）是中级。高级答案会先说"**chunking 策略的选择取决于上游解析能给我什么结构信号**——如果解析输出了清晰的标题层级和表格边界，我就按结构切；如果只能拿到扁平文本，我才退而求其次按长度切，并且我会知道这种情况下质量是有上限的"。把解析和 chunking 的**依赖关系**讲出来,直接拉开档次。

### 3. 多模态文档：图文混排与图表问答

真实文档里大量信息**不在文字里，在图里**——架构图、流程图、数据图表（chart）、截图、示意图。纯文本管道对这部分**直接致盲**。

**两种处理思路：**

1. **图像 → 文本描述（预处理阶段）：** ingestion 时用多模态模型把每张图 / 图表**生成一段文字描述**（"图3：折线图，显示2020-2023年营收逐年增长，2023年达到峰值1.5亿"），把描述当作文本塞进 RAG。优点：复用现有纯文本管道，检索和生成不用改。缺点：描述是有损压缩,生成时丢了细节;且要为每张图付一次多模态推理成本。
2. **图像作为一等公民进入上下文（检索/生成阶段）：** 检索到相关页面后，**把页面图像本身**连同问题一起喂给多模态 LLM，让模型在回答时直接"看"原图。优点：信息无损，图表细节模型能现场读。缺点：成本高、检索侧要支持"召回图像"、上下文 token 涨得快。

这部分的**模型侧能力**（多模态如何对齐视觉和文本、图表理解的边界、视觉幻觉）深入展开在**第 24 章**。本章只讲清楚一件事：**多模态文档的处理方式，要在 ingestion 阶段就决定**——是把图转成文字描述，还是保留图像走多模态生成,这是个架构决策,会反向约束你整条 RAG 管道的设计。

### 4. 落地：用 Claude 把 PDF / 图像接进管道（按 2026 初）

这一节给可执行的工程事实——多模态直读（路线 C）在 Anthropic API 上具体怎么落地，以及那些会让你账单爆炸/踩坑的细节。**API 字段和价格按 2026 初，以官方为准；但"图像按 token 计费、分辨率越高 token 越多"这个成本结构是稳定规律。**

**(1) 三种喂文档的方式。** Claude 这代模型原生支持 PDF 和图像输入（无需自己先 OCR）：
- **`document` block 直接喂 PDF**：base64、URL、或先用 Files API 上传拿 `file_id` 再引用（大文件、跨多次请求复用时用 Files API，避免重复上传)。
- **`image` block 喂页面截图**：base64 或 URL，`media_type` 指明 `image/png` 等。
- **关键：document block 可以开 `citations: {enabled: true}`**——让模型在回答时**标注答案出自原文哪一段**，这对 RAG 的可溯源性是直接的工程红利（呼应第 8 章"答案能溯源"）。

```python
# 直接把 PDF 喂给 Claude，开启引用——解析+理解一步到位
import anthropic
client = anthropic.Anthropic()

resp = client.beta.messages.create(
    model="claude-opus-4-8",
    max_tokens=16000,
    messages=[{
        "role": "user",
        "content": [
            {"type": "text", "text": "这份财报里 2023 年 Q3 华东区营收是多少？指出依据。"},
            {
                "type": "document",
                "source": {"type": "file", "file_id": uploaded.id},  # Files API 上传后的 id
                "citations": {"enabled": True},   # 答案标注来源,可溯源
            },
        ],
    }],
    betas=["files-api-2025-04-14"],
)
```

**(2) 成本结构：图像 token 是真金白银。** 这是路线 C 选型时最容易被低估的点。图像（包括 PDF 渲染成的页面图）**按 token 计费，且 token 数随图像分辨率/尺寸上涨**。一个工程后果直接的事实：**Opus 4.7 起支持高分辨率视觉（长边可到 2576px），精度更高，但全分辨率图像的 image token 可达每张约 4784 token，是早期约 1600 token 上限的约 3 倍。** 含义：
- 一份几十页、每页当高清图喂的 PDF，**光输入 token 就可能上万到几十万**，乘以 frontier 模型的输入价（按 2026 初约 $5-10/百万 token），单份文档解析成本远高于路线 A/B。
- **不需要那么高精度时，ingestion 前在客户端把图降采样**，能显著省钱。但要先用官方 `count_tokens` 在代表性图像上量一遍再决定，别凭感觉拍（无公开 Claude tokenizer，用 tiktoken 估会系统性偏差 15-20%，图像更不准）。

> 算一笔账（数量级，不是精算；按 2026 初价，以官方为准）：一份 **50 页** PDF 全部按高清整页喂 Opus 系列。每页约 **4784 image token**，输入侧 ≈ 50 × 4784 ≈ **24 万 token**。按 frontier 输入价约 **$5/百万** 算，**光这一份文档、光输入、问一次，就约 $1.2**。它的**杀伤力在乘法**：① 关键不对称——**输出价 ≈ 输入价的 5 倍**（约 $25/百万），所以让模型逐页"转写成结构化文本"时，输出 token 才是账单大头；② 若把原图**留在上下文里反复问**，每问一次都重付这 24 万输入 token；③ 放大到 ingestion，一个 **10 万页**的库一次性走路线 C，光输入就 ~**5 亿 token ≈ $2500 起**，再叠输出/重试/多模型，轻松到五位数美金。**同样这 10 万页走路线 A（数字原生本地抽取）几乎免费。** 这就是"海量场景禁止无脑上多模态"的数量级依据——记住量级和那个"输出 5×输入"的不对称，比记任何单价都重要。

> 省钱主杠杆：prompt caching。RAG 的典型形态是**"同一份大文档 / 同一页图，被反复问很多个问题"**——这正是 prompt caching 的主场。把稳定的大块上下文（system + 文档/图像）放在 prompt 前缀并打 cache 断点，**后续命中的请求只按约 0.1× 价计费**（首次写入约 1.25×）。注意三件事：① caching 是**前缀精确匹配**，前缀里任何一个字节变化（哪怕 system 里嵌了个时间戳/UUID）都会**静默击穿**整个缓存；② 渲染顺序是 tools → system → messages，把"稳定的文档/图像"放前面、"每次都变的问题"放最后；③ 用 `usage.cache_read_input_tokens` 验证是否真命中，命中为 0 就是被某个变量打穿了。对"一份文档问 N 个问题"的 RAG，这一招常常是把路线 C 成本从"破产"压到"可接受"的关键，**务必和 Files API 复用（同一份 PDF 传一次、跨请求引用 `file_id`）一起用**。

**(3) 页数 / 大小限制。** PDF 直读有**页数和文件大小上限**（具体数值按官方文档，会变——用到时查 pdf-support 文档，不要背数字）。超长 PDF 的工程对策是**先按页/章拆分再分批处理**，而不是一次性硬塞。

> 工程推论：
>
> **推论七：多模态直读把"解析成本"从一次性的离线 ingestion，部分转移成了按请求计费的在线 token 成本。** 路线 A/B 是"建库时付一次解析钱，之后检索几乎免费"；路线 C 如果在生成时把原图喂进上下文，则**每次问答都在为图像 token 付费**。这个成本模型的差异,决定了 C 适合"高价值低频"而非"海量高频",务必算清这笔账。
>
> **推论八：能拿到"带引用的解析"是路线 C 的隐藏价值。** 传统解析路线给你文本,溯源要自己在 chunk 上挂元数据;多模态直读 + citations 能让模型直接指明答案出自原文哪里。可溯源是 RAG 落地（尤其金融、法律、医疗等高合规场景）的硬需求,这一点值得单独纳入选型考量。

## 四、面试视角：高频考点与面试官想区分什么

文档解析这个话题，面试官用来区分**"跑过 RAG demo 的人"和"上线过生产 RAG 的人"**。demo 用的都是干净文本或简单 PDF，根本不会暴露解析问题；只有真正处理过脏文档的人，才会条件反射地把解析当成第一约束。

面试官想听到的核心信号：
1. 你知道 **PDF 没有逻辑结构**（不是数据格式），所以解析的本质是**重建结构**。
2. 你能在**三条路线间按 trade-off 选型**，且知道生产是 **hybrid pipeline**。
3. 你理解**解析质量是 RAG 的上游瓶颈**，能讲清"解析烂 → chunking 崩 → 全链路崩"的传导链。
4. 你知道**表格和图表是特殊难点**，有针对性处理意识。
5. 你能算 **多模态直读的成本账**，不盲目上多模态。

> 心法：**面试里只要你能主动说出"我会先看文档脏在哪、再决定解析路线，因为解析是 RAG 全链路里唯一上游错误不可恢复的环节"——这一句就把你和 80% 只会 `pdf.extract_text()` 的候选人区分开了。**

## 五、未来演进：3-5 年方向

- **多模态直读会持续吃掉传统解析的份额，但不会完全取代。** 随着多模态模型变强、变便宜、上下文变长，"PDF 当图直接喂"会越来越成为高价值场景的默认。但**海量、低价值、成本敏感**的批量 ingestion，传统路线 A/B 因成本优势仍会长期存在。趋势是**两端分化**：极简文档用极便宜的传统库，复杂高价值文档用多模态,中间地带被版面模型和"多模态 + 缓存"蚕食。
- **"解析"和"理解"的边界继续模糊。** 当模型能直接看懂整页 PDF 并回答问题，"先解析成文本再 RAG"这个两段式流程,在某些场景会被"文档直接进上下文 + 长上下文模型"压平。但**这不等于 RAG 消失**——大库、可溯源、成本控制、权限过滤这些需求,仍然需要检索层(见第 8 章对"长上下文 vs RAG"的讨论)。
- **对应用工程师的含义：** 解析不会变成"调一个 API 就完事"的 commodity,反而因为多模态引入了**新的成本/质量/延迟权衡维度**,选型判断变得更重要。你的核心竞争力是**理解每条路线的物理约束,并为具体文档场景做出可执行的工程取舍**——而不是记住某个库或某个模型的名字(那些一定会过时)。

> 全章心法：**RAG 的质量天花板,不在你最后调的那个 prompt,而在你最先解析的那份 PDF。文档智能是 RAG 的上游,上游决定下游的一切——把脏文档解析对,是整条链路里回报最高、却最常被跳过的一步。**

## 章末五件套

### 高频面试题（附答题框架）

**1. 为什么不能直接用 `pdf.extract_text()` 做 RAG 的文档加载？**
- 答题框架：先点破 **PDF 是打印格式不是数据格式**,文件里只有字符+坐标,没有段落/表格/阅读顺序等逻辑结构。`extract_text()` 拿到的是按文件写入顺序拼接的字符流,**多栏会乱序、表格会拍扁、扫描件直接返回空**。再补一句:它只对"数字原生、单栏、无表格"的简单 PDF 够用,真实业务文档会崩。
- 错误答案陷阱:只说"它效果不好"而说不出**为什么**(PDF 无逻辑结构)——这暴露你没理解本质。

**2. 文档解析有哪几条技术路线?各自适用什么场景?**
- 答题框架:三条——(A) 传统文本库(pypdf/pdfplumber),便宜快但只适合简单数字原生 PDF;(B) 版面理解模型/Document AI,质量与成本的平衡点,能处理表格多栏扫描件;(C) 多模态 LLM 直读,质量最好(尤其复杂表格图表)但 token 成本高。点出**生产是 hybrid pipeline**:先分类文档再路由到不同路线,按"价值密度×频率"决定要不要上 C。
- 错误答案陷阱:回答"用 X 库一把梭"——真实异构文档库不存在银弹,这个回答直接判定你没做过生产。

**3. 解析质量为什么是 RAG 的上游瓶颈?讲一条具体的崩塌链。**
- 答题框架:解析是全链路里**唯一上游错误不可恢复**的环节(chunking/检索/prompt 都可以反复试,但垃圾文本上抛光没用)。举崩塌链:多栏 PDF 解析乱序 → chunking 切出语义碎片 → embedding 编乱码语义 → 检索召回狗屁不通的 chunk → 模型基于碎片幻觉。强调**症状在最后显现,病根在解析**,这是 RAG 最常见的误诊。

**4. 表格抽取为什么难?生产怎么处理?**
- 答题框架:难在 **二维行列关系坍缩成一维文本就丢了**,加上合并单元格(归属无显式信息)、跨页表格、无边框隐式表格四大坑。处理:用专门的表格结构识别(版面模型),把表格抽成 **Markdown/HTML 表格保留行列结构** 再喂下游,而非拍扁成纯文本;多模态直读视觉理解最强但要防数值幻觉。
- 错误答案陷阱:说"把表格转成文本就行"——拍扁正是表格问答崩盘的头号原因。

**5. OCR 现在很强了,扫描件是不是不算问题了?**
- 答题框架:破两个幻觉。第一,**OCR ≠ 文档解析**:OCR 解决"图→字",**版面结构(标题/表格/阅读顺序)仍要靠 OCR 之上的版面分析重建**。第二,**OCR 错误会沉默传导**:认错字符(0/O、1/l)、表格串行,这些进了文本后 embedding/检索/生成全程无人报错,最后是个自信的错答——比"没识别出来"更危险。

**6. 什么时候该用多模态 LLM 直读 PDF,什么时候不该?**
- 答题框架:用判据"**价值密度 × 调用频率**"。高价值/低频/复杂(关键合同、财报、含大量图表)→ 值得上 C,质量优先,还能拿 citations 溯源。低价值/高频/海量(百万页工单)→ 必须 A/B,C 的 token 成本会破产。关键要会**算图像 token 账**:图像按 token 计费、分辨率越高 token 越多,高清整页 PDF 输入 token 轻松上万(高分辨率单页可达约 4784 token)。
- 加分点(把"会算账"讲到底):① 点出**输出价 ≈ 输入价的 5 倍**——让模型逐页转写结构化文本时,账单大头在输出不在输入;② 主动说**对"一份文档问 N 个问题"的 RAG,prompt caching 是省钱主杠杆**(命中按约 0.1× 计费),配合 Files API 复用同一份 PDF,常能把 C 从"破产"压到"可接受";③ 给得出**数量级**(如几十页高清 PDF 问一次约 $1 量级、百万页全走 C 是五位数美金量级),而不是只会说"很贵"。能把这三点说出来,直接和"只会说多模态贵"的候选人拉开。

**7. 解析路线的选择和 chunking 策略是什么关系?**
- 答题框架:**chunking 的天花板是解析质量**。chunking 按结构切(标题/段落/章节)的前提是"结构已知",而结构正是解析要重建的。所以选 chunking 策略前要先问"解析能给我什么结构信号"——有清晰标题层级就按结构切,只能拿扁平文本就退而求其次按长度切(且质量有上限)。把**依赖关系**讲出来是加分点。

### 实战项目(P2 增量)

**目标:** 在已有的 P2 企业知识库 RAG 上,接入**真实业务 PDF(必须包含:至少一份带复杂表格的文档 + 至少一份扫描件)**,用三条解析路线分别处理,量化对最终答案质量的影响。

**验收标准:**
1. **文档分类与路由**:实现一个简单的 PDF 分类器(至少能判断"有无文本层"),把文档路由到对应解析路线;能说明每份文档为什么走了哪条路线。

### 系统设计题(含量级估算)

设计一个**企业级文档 ingestion 流水线**,要求一次性导入**500 万页**历史文档(混合:60% 数字原生 PDF、25% 扫描件、15% 含复杂表格)进 RAG 知识库。

请估算并设计:
1. **成本量级**:若全部用多模态直读(路线 C),假设每页渲染成图约 1500-4800 image token(高分辨率单页可达约 4784)、输入价按约 $5/百万 token,**估算 500 万页的解析成本量级**(给出数量级即可,并说清这只是输入侧——若要逐页转写结构化文本,**输出价≈输入 5 倍**,输出才是大头)。再估算用 hybrid(数字原生走路线 A 免费、其余走 B/C)能省多少。这个对比要能驱动你的架构决策。
2. **吞吐与时间**:路线 A 毫秒级/页、C 秒级/页,估算两种极端下导完 500 万页各需多久(单机 vs 需要多少并发)。
3. **架构**:画出"分类 → 路由 → 多解析器并行 → chunk → embed → 入库"的离线批处理管道,指出哪里需要队列、哪里是瓶颈、扫描件的 OCR 算力怎么排布。**这是离线、非延迟敏感的场景,主动点出该用 Batch API 类异步批处理(约 50% 价格)而非逐条同步调用,是加分项。**
4. **失败处理**:500 万页里必然有解析失败/超时的,设计**重试 + 死信 + 人工复核**机制。

(考点:成本量级敏感度直接决定路线选择、输入/输出价不对称、离线批处理省钱手段(Batch API)、并发与瓶颈、大规模下的失败处理。核心是让你体会**为什么海量场景不能无脑上多模态**——并能把账算到数量级,而不是停在"很贵"。)

**测试要求:**
1. `detect_route`:用至少 4 个 fixture——纯文本单栏 PDF(→TEXT_LIB)、双栏论文(→LAYOUT)、含合并单元格表格的 PDF(→LAYOUT)、扫描件图片 PDF(→MULTIMODAL),断言路由正确。**特别加一个"双栏但能 extract 出文本"的 case,断言它走 LAYOUT 而非 TEXT_LIB**(防住最常见的误判)。
2. `route_and_parse`:断言表格 chunk 的 `text` 含 Markdown 表格语法(如 `|` 和 `---`),而非压扁的一行;断言每个 chunk 的 `source_ref` 非空。
3. **失败路径测试**:喂一个解析必然产出空结果的坏文件,断言**抛错或进死信**,而不是静默返回 `[]`。

**暗坑提示(综合):**
- 整个设计的灵魂是 **detect_route 里"能抽出字 ≠ 抽对了字"** 这个判断——双栏陷阱是 95% 的人会踩的。
- 表格保留结构、解析失败不静默,这两条是把"能跑的 demo"变成"敢上生产"的分水岭。
- 真实实现里 `detect_route` 的几何判断(双栏/表格检测)很难做准,生产中常用更鲁棒的版面模型直接出 route 信号——本题的坐标启发式是为了让你**体会判断的依据**,不是生产终态。
