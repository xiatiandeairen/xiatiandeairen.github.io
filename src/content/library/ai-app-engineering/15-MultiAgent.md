> 这章排在「单 Agent（第 12 章）」和「Workflow 编排（第 14 章）」之后，是阶段三 Agent 的收口。前两章教你"怎么让一个 agent 自己转"和"什么时候用确定性流程兜住它"。这章只解决一个判断题：**当一个 agent 不够用时，到底该不该拆成多个？** 替你解决的核心痛点是——别在没验证的情况下，因为"听起来更强"就把系统拆成 multi-agent，然后被 token 成本、协调开销和可靠性塌方反噬。这章给你一套**默认怀疑、要拆先证**的决策框架，而不是又一套框架教程。
>
> 先把结论甩在前面，整章都在论证它：**Multi-agent 大多数时候是过度设计。先别拆。**（这不是我个人偏见，是本课程主线项目 README 写死的本章主题。）

## 一、背景：为什么会出现 multi-agent，它到底解决什么问题

### 1.1 从一个真实诱惑切入

你做完第 12 章的单 agent 骨架，跑了几个真实任务，很快会撞到三类不舒服：

1. **角色冲突**：同一个 system prompt 里既要它"像严谨的 code reviewer 一样挑刺"，又要它"像产品经理一样判断这值不值得改"。两套人格揉在一个上下文里，互相稀释。
2. **上下文爆炸**：任务一长，历史里塞满了 reviewer 的细节、测试日志、文档摘录，关键信息被噪声淹没（第 13 章的老问题）。
3. **工具面太宽**：一个 agent 同时挂了 30 个工具——查日志的、重启服务的、发 Slack 的、查数据库的——模型选型时容易抓瞎，schema 占满上下文。

这时候一个非常自然、非常诱人的念头冒出来：**"那我拆成几个专职 agent 不就行了？一个专管 review，一个专管决策，一个专管执行，各自干净。"**

这就是 multi-agent 的起源动机。它本质上是**把『单进程里塞了太多职责』的问题，用『多进程 + 通信』来解**——这个直觉，对一个写过分布式系统的后端工程师来说，太熟悉了。

> **后端锚点**：单 agent → multi-agent，几乎就是**单体应用 → 微服务**的翻版。动机一样（职责解耦、独立扩展、隔离故障域），代价也一样（网络/通信开销、一致性难题、调试链路变长、整体可靠性是各环节可靠性的乘积）。**而你早就知道：微服务不是"更高级"，是一个用复杂度换边界的 trade-off，绝大多数小系统拆了就是自找麻烦。** Multi-agent 一模一样。把这个直觉焊死，这章你就赢了一半。

### 1.2 multi-agent 真正能买到什么

抛开炒作，multi-agent 在物理上只买得到三样东西，记牢，后面所有判断都围绕它们：

| 能买到 | 机理 | 单 agent 能不能近似做到 |
|---|---|---|
| **上下文隔离** | 每个 sub-agent 有独立的 context window，不互相污染 | 能——用 context editing / 子任务清空历史（第 13 章），成本低得多 |
| **并行 fan-out** | 多个 sub-agent 同时跑互不依赖的子任务，压缩 wall-clock 时间 | 不能真并行——单 agent loop 本质串行 |
| **角色/模型专用化** | 不同 sub-agent 用不同 system prompt、不同模型、不同工具集 | 部分能——但换模型会击穿 prompt cache（见第五节） |

**工程推论（每条都能独立成一道面试题）：**

1. **如果你拆 multi-agent 的理由是"上下文太乱"，那大概率拆错了**——上下文隔离是单 agent 用 context editing 就能买到的，不需要付协调开销。拆之前先问："这是真需要并行，还是只是想清理上下文？"
2. **multi-agent 唯一无法被单 agent 替代的硬价值是『真并行 fan-out』**——把这个当成"该不该拆"的第一性判据。任务里没有"多个互相独立、可同时进行的子工作"，就没有拆的物理必要。
3. **"角色专用化"是个伪刚需**——一个 Opus 级模型靠 prompt 切换人格的能力，强到你以为需要多个 agent 的场景，单 agent 换段 prompt 就够了。把"我想要一个 reviewer 和一个 PM"翻译成"我要不要两个独立上下文 + 两次模型调用"，诱惑立刻冷却一半。

## 二、演进：以前怎么做，为什么不够

### 2.1 第一代：人肉编排多个 LLM 调用（隐式 multi-agent）

最早大家不叫它 multi-agent，就是朴素的"链式调用"：第一次调用让模型出大纲，第二次调用让模型按大纲写，第三次让模型审。本质是**你（代码）在当协调者**，每个 LLM 调用是一个无状态的"工序"。

这其实是第 14 章的 Workflow，**不是真正的 multi-agent**——因为没有任何一个 LLM 调用拥有"自主决定下一步、自主调度别人"的权力。控制流在你手里。这一代足够好，覆盖了今天 80% 被错误标榜成"multi-agent"的系统。

### 2.2 第二代：Agent 当协调者，自主调度子 Agent

真正的 multi-agent 出现在：**让一个 agent（coordinator / orchestrator）拥有"把子任务派给其他 agent"的工具**。控制流从你的代码移交给了模型。Coordinator 自己决定要不要 fan-out、派给谁、怎么汇总。

这一步质变了——好处是处理开放式、难以预先编排的任务（"把这个设计文档变成一个 PR"）时，比死板的 Workflow 灵活；坏处是你**失去了对控制流的确定性掌控**，token 成本和失败模式都变得难以预测。第 14 章那条铁律在这里加倍生效：**自主性每上一档，可观测性和可控性就下一档。**

### 2.3 第三代：框架化 + 平台原生支持

2024–2025 涌现一批 multi-agent 框架（LangGraph / AutoGen / CrewAI 等），把"agent 之间怎么连、状态怎么传"做成了可复用的编程模型。再往后，平台开始**原生**支持 multi-agent——比如 Anthropic 的 Managed Agents（CMA）把"一个 coordinator 协调一组 sub-agent"做成了 API 一等公民（`multiagent: {type: "coordinator", agents: [...]}`，见第四节）。

> **知识时效（按 2026 初）**：框架名单和具体 API 形态**会过期，请学方法论别背名单**。但有一条物理规律不变——**框架越成熟，"拆 multi-agent"的工程门槛越低，于是『不该拆也拆了』的误用越泛滥**。工具越顺手，越要靠判断力踩刹车。这正是 README 把本章定性为"最不稳定领域、学判断该不该上"的原因。

## 三、现状：主流协作模式及各自优缺点

下面是当前业界在用的几种 multi-agent 拓扑。**重点不是记住名字，是记住每种的失败模式和"什么时候它退化成不该用"。**

### 3.1 五种模式对照

| 模式 | 结构 | 适用 | 主要失败模式 | 后端类比 |
|---|---|---|---|---|
| **Orchestrator-Worker（协调者-工人）** | 1 个 coordinator 派活给 N 个无状态 worker，自己汇总 | 可分解为独立子任务的工作（fan-out 研究、多文件处理） | coordinator 成为瓶颈和单点；汇总阶段上下文爆炸 | master-worker / MapReduce |
| **Handoff（交接）** | agent A 干完把整个对话所有权交给 agent B | 阶段分明的流水线（分诊 → 专科） | 交接时上下文丢失；"踢皮球"无限循环 | 工单在团队间转交 |
| **群聊 / Blackboard（黑板）** | 多个 agent 共享一块上下文，轮流发言 | 需要多视角碰撞的开放讨论 | token 成本随 agent 数**爆炸**；容易发散不收敛 | 共享内存 + 多线程抢锁 |
| **层级（Hierarchical）** | coordinator → sub-coordinator → worker，多层 | 超大型、可递归分解的任务 | 层数越深，指令在传递中失真越严重；调试地狱 | 多级组织架构 |
| **Coordinator + Subagent Threads（线程模型）** | 1 个 coordinator，每个 sub-agent 跑在独立 thread（独立上下文，共享文件系统） | 既要隔离上下文、又要共享工作产物的并行任务 | 协调者必须显式传递意图，否则 sub-agent 不知道全局 | 主线程 fork 出 worker 线程，共享磁盘不共享栈 |

### 3.2 重点讲两个：Orchestrator-Worker 和 Threads 模型

**Orchestrator-Worker** 是工业界最常用、最该优先考虑的模式，因为它**最接近你熟悉的 MapReduce**，失败模式最可控。Coordinator 负责"分 + 合"，worker 无状态、可并行、可重试。绝大多数"真的需要 multi-agent"的场景，用这个就够了——别上来就群聊或多层级。

**Coordinator + Subagent Threads** 是 Anthropic CMA 采用的模型，值得单独理解它的一个反直觉设计：

- 所有 sub-agent **共享同一个容器和文件系统**（写到磁盘的产物互相可见）；
- 但每个 sub-agent 跑在**独立的 thread**——独立的对话历史、独立的 model / system prompt / 工具 / 技能（来自它自己那份 agent 配置）；
- 关键陷阱：**线程间不共享对话历史**。coordinator 想让某个 sub-agent 基于某信息行动，**必须在派活的消息里说清楚，或者写到磁盘上**——不能假设 sub-agent"知道"coordinator 知道的东西。

**工程推论：**

4. **群聊（Blackboard）模式是 token 成本的黑洞，默认拉黑。** 它的成本随参与 agent 数**超线性**增长（每个 agent 每轮都要读全部历史，N 个 agent ≈ N² 级的 token 读取）。除非你明确需要"多视角辩论且能收敛"，否则不要选群聊。
5. **层级越深，失真越重——超过两层就要拉警报。** 这跟"组织层级越多、战略落地越走样"是同一个道理。CMA 干脆**只允许一层 delegation**（sub-agent 自己的 `multiagent` 配置不会级联生效），这是平台用硬约束帮你避免"多层失真地狱"。
6. **共享文件系统 ≠ 共享认知。** Threads 模型里 sub-agent 共享磁盘但不共享对话，这正是"共享内存的多线程"踩过的坑——你以为它们看得到彼此的状态，其实只看得到显式写出来的部分。**coordinator 的 prompt 里必须把"要传给 sub-agent 的上下文"显式写全**，这是 multi-agent prompt 工程的头号注意点。

## 四、落地：生产怎么选、大厂怎么做

### 4.1 框架对比（按 2026 初，名单会变，学定位）

| 框架 / 平台 | 心智模型 | 强项 | 适合 | 注意 |
|---|---|---|---|---|
| **LangGraph** | 把 agent 协作建模成**有向图**（节点 = agent/工具，边 = 控制流），状态显式 | 控制流可控、可回放、状态可检查 | 你想要"半自主"——结构卡死、局部放手 | 图的表达力强但学习曲线陡；框架迭代极快，别绑死 |
| **AutoGen（微软）** | **可对话的 agent**，靠消息互相通信，天然支持群聊 | 多 agent 对话、人在环、研究探索 | 快速搭多 agent 对话原型 | 群聊模式容易 token 爆炸 + 不收敛，生产要管住 |
| **CrewAI** | **角色扮演**（给每个 agent 派 role/goal/backstory），像组个团队 | 上手快、概念直观 | demo、内容流水线、非关键任务 | 角色抽象漂亮但对生产级控制流和成本约束较弱 |
| **OpenAI Agents SDK** | 轻量 **handoff + guardrails**，agent 间交接所有权 | 简洁、贴近 OpenAI 生态 | OpenAI 技术栈下的流水线式协作 | handoff 模型要小心上下文丢失 |
| **Claude Agent SDK**（自托管 loop） | 你用 Anthropic SDK 自己写 agent loop / tool runner，**编排和容器在你这边** | 完全可控：能插审批卡点、自定义日志、条件执行 | 你要自己掌控 loop、自带工具运行时 | 容器、重试、压缩、缓存策略全得你自己管 |
| **Claude Managed Agents（CMA）**（托管 loop） | coordinator + subagent threads，**Anthropic 托管 agent loop + 每会话容器** | 一等的 thread 隔离、共享文件系统、托管编排（缓存/压缩/思考内置） | 想让平台扛 agent loop + 容器编排，只写 coordinator 配置 + 消费事件流 | multi-agent 是 agent 上的 `multiagent` 顶层字段（`{type:"coordinator", agents:[...]}`），**不是工具项**；一层 delegation；roster ≤ 20、并发线程 ≤ 25 |

> **选型判断（可执行）**：
> - **想要确定性骨架 + 局部自主** → LangGraph（图把控制流焊死）。
> - **想要平台扛掉 agent loop 和容器** → CMA（你只写 coordinator 配置 + 消费事件流）。
> - **只是做个能跑的 demo / 内容流水线** → CrewAI 上手最快。
> - **任何关键路径** → 先别上框架的"全自主多 agent"，从 Orchestrator-Worker 起步，能用第 14 章的确定性 Workflow 兜住的就别给模型调度权。
>
> **比"选哪个框架"更本质的一根轴：agent loop 在谁手里。** 上面一堆框架名会过期，但这根轴不会——它决定了运维边界归谁：
> - **自托管 loop**（Claude Agent SDK / LangGraph / 自研 DAG）：循环跑在你的进程里，容器、重试、缓存策略、并发调度全是你的活。换来的是**能在 loop 里插审批卡点、自定义日志、条件执行**——关键路径要 HITL、要审计的，必须走这条。
> - **托管 loop**（CMA）：循环和每会话容器跑在 Anthropic 那边，你只发事件、消费事件流。换来的是**少写一大坨编排代码**，代价是控制权交出去、可观测性受平台 API 约束。
>
> 面试被问"你怎么搭 multi-agent"，先答这根轴（**我自己扛 loop 还是让平台扛**），再谈拓扑和框架——顺序错了就显得只会背框架名。
>
> **新手第一误区**：以为"选哪个框架"是 multi-agent 的核心问题。**不是。核心问题是"该不该拆"。** 框架是拆之后才需要操心的事——90% 的人在"该不该拆"上就该停下，根本走不到选框架。

### 4.2 国内厂商语境（按 2026 初）

- 国内闭源大模型——**豆包（字节）/ Qwen（阿里）/ DeepSeek / Kimi（月之暗面）/ GLM（智谱）/ 混元（腾讯）/ 文心（百度）**——在做 multi-agent 时，绝大多数走的是**框架层自建编排**（LangGraph 或自研 DAG 引擎）+ 调各家模型 API，而不是依赖某个平台的原生 multi-agent。**DeepSeek V3/R1**（2025 初证明开源推理可追平闭源的分水岭）让"coordinator 用强模型、worker 用便宜开源模型"的分层成本策略变得很现实。
- **国内落地的真实重心不在"多 agent 炫技"，而在成本**：用 Qwen / DeepSeek 这类开源或低价模型扛住 worker 层的高频调用，把贵模型省给 coordinator。这跟下面要讲的成本爆炸直接相关。
- 务实判断：**国内团队上 multi-agent，第一驱动力往往是"把贵模型调用拆到便宜模型上",而非能力**——这恰恰说明"角色专用化"在生产里常常是"成本专用化"的伪装。

### 4.3 大厂的真实取舍：Anthropic 多 agent 研究系统案例

Anthropic 公开过它们的多 agent 研究系统（用于深度调研类任务）的设计取舍，是少数有真实数据的工业案例，值得拆解：

**它为什么敢拆 multi-agent：** 研究/调研任务天然是 **fan-out 友好**的——"调研 X"可以分解成"查 X 的 A 面 / B 面 / C 面"，这些子调研**互相独立、可同时进行**。这命中了第一节那条硬判据：**有真并行价值**。它用的是 Orchestrator-Worker——lead agent 拆解问题、派给若干 subagent 并行检索、再汇总。

**它付出的代价（这才是重点）：**

- **token 消耗是单 agent 聊天的约 15 倍**。读清楚：**15×**。它们的原话大意是"multi-agent 系统烧 token 极凶，只有当任务价值足够高、足以justify这个开销时才值得"。
- 因此它们划了条明确的线：**multi-agent 只用在"任务价值 >> token 成本"的场景**（深度调研、值钱的开放问题）。日常问答、简单任务，**坚决用单 agent**。

**工程推论：**

7. **"要不要 multi-agent"本质是一道 ROI 题，不是技术题。** Anthropic 的判据可以直接抄作业：**任务价值 ÷ token 成本 >> 1 才拆。** 15× 的成本溢价，意味着只有高价值、强 fan-out 的任务才回得了本。把这句话变成你团队的准入门槛。
8. **能拿出"15×"这个数，本身就是判断力的体现。** 业余选手说"multi-agent 更强所以用"；专业选手说"multi-agent 贵 15 倍，所以只在 fan-out 强、价值高的任务上用，其余一律单 agent"。**带着量级谈成本，是面试和实战的分水岭。**
9. **Anthropic 自己都把 multi-agent 限定在极窄场景**——这是平台方用真金白银给你的背书：**默认单 agent，multi-agent 是例外，不是进步。**

## 五、深入：token 成本爆炸、可靠性衰减与缓存击穿

这一节是本章的"物理学"，讲透到骨子里——因为这三件事是 multi-agent **反对自己**的核心证据。

### 5.1 token 成本为什么会爆炸（不是线性，是更糟）

回忆第 12 章算过的单 agent 成本结构：**主成本驱动是"步数 × 累积上下文"，而非单次调用**，且上下文随步数**平方级膨胀**。Multi-agent 在这个基础上再叠两层放大：

1. **每个 sub-agent 自己是一个会膨胀的 agent loop**——N 个 sub-agent 就是 N 份平方级膨胀。
2. **coordinator 的汇总阶段**要把各 sub-agent 的产出都读进上下文——sub-agent 越多、产出越长，汇总这一步的输入 token 越恐怖。
3. **协调本身要花 token**——派活的消息、状态同步、结果回传，全是模型调用。

这就是 Anthropic 那个 **15×** 的来源。**输出价 ≈ 输入价 5 倍**（第 2 章的关键不对称）这条还在生效——多个 agent 各自输出冗长思考，输出端成本也跟着翻倍。

**工程推论：**

10. **multi-agent 的成本是"乘法"不是"加法"。** N 个 sub-agent 不是把成本变成 N 倍，是把"每个都平方膨胀 + 汇总爆炸 + 协调开销"叠起来，轻松上 10× 量级。做容量估算时，**默认按单 agent 的 10–15× 起算 multi-agent**，别用"N 个 agent ≈ N 倍"的天真模型。

### 5.2 可靠性随 agent 数下降（乘积律）

这条最该刻进 DNA，因为它来自你早就懂的可靠性数学：

> **串行依赖的系统，整体可靠性 = 各环节可靠性的乘积。**

假设单个 agent 完成它那一步的成功率是 95%（对 LLM 已经算乐观）。那么：

- 1 个 agent：成功率 95%
- 3 个 **串行依赖** 的 agent（handoff 链）：0.95³ ≈ **86%**
- 5 个串行依赖：0.95⁵ ≈ **77%**

**每加一个串行环节，整体可靠性就被乘掉一刀。** 而 multi-agent 还引入了单 agent 没有的新失败模式：交接时上下文丢失、sub-agent 误解 coordinator 意图、汇总时信息冲突、踢皮球死循环……这些都是**纯增的失败面**。

> **后端锚点**：这就是"微服务调用链越长、整体 SLA 越难保证"的翻版。一条 5 跳的同步调用链，每跳 99.9% 可用，整链也只剩 99.5%。Multi-agent 把这个问题搬到了一个**单环节本就只有 ~95% 可靠**的世界里——雪上加霜。

**工程推论：**

11. **agent 数和可靠性是反比，且是乘积级反比。** "多个 agent 更稳健"是彻头彻尾的错觉——除非它们是**并行独立**且**有汇总仲裁**（用冗余换可靠，像多副本投票），否则**串行 multi-agent 一定比单 agent 更不可靠**。区分"并行冗余（可能提升可靠性）"和"串行依赖（一定降低可靠性）"是这道题的胜负手。
12. **可靠性衰减的解药是"减少串行环节 + 加确定性兜底"，不是"加更多 agent"。** 看到一条长 agent 交接链，本能反应应该是"能不能压扁成单 agent + Workflow"，而不是"再加个 agent 来检查"。

### 5.3 prompt caching 在 multi-agent 里的击穿与对策

这是第 2 章 prompt caching 知识在 multi-agent 场景的硬核应用，也是个高频踩坑点。

回忆缓存铁律：**prompt caching 是前缀匹配，渲染顺序 `tools → system → messages`，前缀里任何一个字节变化，其后全部缓存失效。** Multi-agent 有三个天然的缓存杀手：

| 缓存杀手 | 为什么击穿 | 对策 |
|---|---|---|
| **不同 sub-agent 换模型** | 缓存是 model-scoped 的；coordinator 用 Opus、worker 换便宜模型，worker 这一路根本读不到 coordinator 的缓存 | 这是**故意的成本设计**（便宜模型本来就省），接受它——但别幻想跨模型复用缓存 |
| **不同 sub-agent 换 system / 工具集** | system 和 tools 在前缀最前面，一换全前缀失效 | 每个 sub-agent 的 system + 工具集**自身保持稳定**（跨调用 byte 一致），让它**自己这一路**能命中缓存 |
| **fork 出 sub-agent 时重建了前缀** | sub-agent 调用如果重新拼 system/tools 且和父级有任何差异，就完全错过父级缓存 | fork 时**逐字节复制父级的 system/tools/model**，只在末尾追加子任务内容 |

还有一个 multi-agent 特别容易碰的**正向技巧**：第 2 章讲过"换模型会击穿缓存"——而 multi-agent 的 **subagent 恰恰是『在主循环不换模型』的正确姿势**。主 loop 始终一个模型（缓存不断），需要用便宜模型干杂活时，**spawn 一个 subagent 去跑**，而不是在主 loop 里中途切模型。Claude Code 的 Explore subagent 用 Haiku 就是这个套路。

**工程推论：**

13. **multi-agent 里"换模型省钱"和"缓存命中省钱"会打架，要算总账。** worker 用便宜模型省了单价，但丢了 coordinator 的缓存；到底省不省，取决于"便宜模型的单价折扣"和"丢失缓存的重写成本"谁大。**别想当然——两个杠杆有时互相抵消。**
14. **缓存的 5 分钟 TTL 在 multi-agent 里更易失效。** sub-agent 并行跑、coordinator 等齐所有 sub-agent 才汇总——这中间的等待很容易超过默认 5 分钟 TTL，导致汇总前缀缓存失效要重写（1.25×）。长 multi-agent 任务要么上 1h TTL（写 2×），要么接受 cache miss，**把这块写进成本模型，别默认 0.1× 永远成立**（这条和第 12 章那个 agent TTL 陷阱是同一个道理，在 multi-agent 里更严重）。
15. **并发 fan-out 的缓存有个时序坑**：N 个 sub-agent 同时启动、共享同一前缀时，**谁都读不到别人正在写的缓存**（缓存要等第一个响应开始流式输出才可读）。正确姿势是**先发 1 个、等它开始返回、再放出剩下 N−1 个**，让它们读第一个刚写好的缓存。无脑同时 fan-out N 个，N 份全价写缓存，钱白烧。

## 六、未来演进：3–5 年方向，对应用工程师的含义

> **知识时效**：以下是趋势判断（按 2026 初），不是已验证事实，标注为方向。

1. **平台原生 multi-agent 会越来越成熟，门槛越来越低**——CMA 这类"coordinator + threads"会成为标配，你不用自己管 agent loop 和容器编排。**含义：拆 multi-agent 的工程成本会降，但"该不该拆"的判断成本不变。工具越顺手，判断力越是你的护城河。**
2. **成本会下降，但"乘法成本结构"不变**——单 token 会更便宜，可 multi-agent 相对单 agent 的"10–15× 溢价"是结构性的，不会因为模型降价就消失。**含义：ROI 判据（价值 ÷ 成本 >> 1）长期有效。**
3. **可靠性短板会被工程手段缓解，但不会消失**——会出现更好的"sub-agent 输出校验 / 自动重试 / 仲裁"机制。**含义：乘积律是数学，工程只能缓解不能违背；减少串行环节永远比加 agent 划算。**
4. **行为控制主通道继续从『采样超参』迁移到『自然语言 + 思考预算』**（第 2 章趋势）——multi-agent 的"协调"会越来越靠自然语言指令而非硬编码控制流。**含义：multi-agent prompt 工程（尤其"把上下文显式传给 sub-agent"）的重要性会上升。**
5. **对应用工程师最实际的一条**：未来三五年，**会写"该不该拆 multi-agent"的决策备忘录，比会搭 multi-agent 更值钱**。因为搭会被平台和框架抹平成几行配置，而判断"这个任务到底有没有 fan-out 价值、值不值 15× 成本"是抹不平的工程心智。

---

> **本章心法**：**Multi-agent 不是"更强的 agent"，是"用 10–15× 成本和乘积级可靠性衰减，换真并行 fan-out"的一笔昂贵交易。没有强 fan-out、没有高任务价值，这笔交易就是亏的。默认单 agent，要拆先证——证不出 ROI，就别拆。**

# 章末五件套

## 一、高频面试题（10 道，附答题框架）

**Q1：什么时候该用 multi-agent，什么时候坚决不该？**
- **要点**：该用的唯一硬判据是**任务有真并行 fan-out 价值**（多个互相独立、可同时进行的子任务）**且任务价值高到能 cover 成本溢价**。坚决不该的信号：任务本质串行、只是"想清理上下文"（用 context editing 就行）、只是"想要不同角色"（单 agent 换 prompt 就行）、关键路径要求高可靠（multi-agent 可靠性更差）。
- **陷阱答案**："multi-agent 更强大更智能所以该多用"——这是直接判负的答案，暴露了不懂成本和可靠性代价。

**Q2：multi-agent 的成本相对单 agent 大概是什么量级？为什么？**
- **要点**：Anthropic 公开数据约 **15×**。原因是三层叠加：每个 sub-agent 自身平方级上下文膨胀 × N、汇总阶段读入所有 sub-agent 产出、协调消息本身的开销。成本是**乘法不是加法**。
- **加分点**：点出"输出价≈输入价 5 倍"也在多 agent 输出冗长思考时被放大。
- **陷阱**：说"N 个 agent ≈ N 倍成本"——天真线性模型，错。

**Q3：为什么说"agent 越多越不可靠"？**
- **要点**：串行依赖系统的可靠性是各环节的**乘积**。单 agent 单步 ~95% 成功，3 个串行 ≈ 86%，5 个 ≈ 77%。外加交接丢上下文、误解意图、踢皮球死循环等纯增失败面。
- **关键区分**：**并行冗余 + 仲裁**可能提升可靠性（多副本投票），**串行依赖**一定降低。能区分这两者是高分线。
- **陷阱**："多个 agent 互相检查更稳健"——只有并行冗余成立，串行不成立。

**Q4：列举 multi-agent 的主流协作模式，并说哪个最该优先、哪个最该避免。**
- **要点**：Orchestrator-Worker（最该优先，像 MapReduce，失败模式最可控）、Handoff、群聊/Blackboard（最该避免，token 随 agent 数超线性爆炸）、层级（超两层就警报）、Coordinator+Threads。
- **加分点**：用 MapReduce / 微服务调用链 / 共享内存多线程做类比，体现"用分布式直觉理解 agent 拓扑"。

**Q5：Coordinator + Subagent Threads 模型里，最大的工程陷阱是什么？**
- **要点**：**线程间共享文件系统但不共享对话历史**。coordinator 不能假设 sub-agent"知道"它知道的——必须在派活消息里显式说清，或写到磁盘。等价于"共享内存的多线程不共享栈"。
- **陷阱**：以为 sub-agent 能自动看到 coordinator 的上下文。

**Q6：prompt caching 在 multi-agent 里有哪些击穿点？怎么应对？**
- **要点**：① sub-agent 换模型（缓存 model-scoped，跨不了）② 换 system/工具集（前缀变，全失效）③ fork 时重建前缀（错过父级缓存，要逐字节复制 system/tools/model）。正向技巧：用 subagent 跑便宜模型，避免在主 loop 中途切模型击穿主缓存。
- **加分点**：并发 fan-out 时"先发 1 个等它写好缓存、再放剩下 N−1 个"的时序技巧。

**Q7：你的团队想把一个单 agent 系统拆成 multi-agent，你作为 reviewer 会问哪三个问题？**
- **要点**：① 任务里有没有**真并行 fan-out**（没有就别拆）② 拆的理由是不是其实是"上下文乱"或"想要角色"（那用 context editing / 换 prompt 更便宜）③ 任务价值能不能 cover **10–15× 成本溢价**和**可靠性下降**。
- **这题考的是"踩刹车的判断力"**，能反问回去比急着设计加分。

**Q8：Batch API 能用来跑 multi-agent 吗？**
- **要点**：**不能跑交互式 agent loop**。Batch 是一发一收的异步批处理（≤24h、无事件流、无法中途回喂 tool_result / 协调消息），而 multi-agent 的协调本质是多步同步交互。正确拆法：把 agent 编排留在在线同步路径，只把"可离线、单次完成"的子任务（如对每个文档单独评分）塞 Batch。
- **陷阱**：直觉说"批量任务上 Batch 省 50%"——把交互式编排塞 Batch 是概念错误（这条和第 12 章同源）。

**Q9：LangGraph、CrewAI、CMA 各自的定位差异？**
- **要点**：LangGraph = 图建模控制流，要"确定性骨架 + 局部自主"选它；CrewAI = 角色扮演，上手快适合 demo / 内容流水线；CMA = 平台托管 coordinator+threads，想让平台扛 agent loop 和容器选它（注意 multiagent 是顶层字段非工具、一层 delegation、roster≤20）。
- **加分点**：强调"选框架是拆之后才操心的事，核心问题永远是该不该拆"。

**Q10（开放）：Anthropic 自己把 multi-agent 限定在什么场景？这对你的选型决策意味着什么？**
- **要点**：限定在**深度调研类、强 fan-out、高价值**任务，明确说日常问答用单 agent。意味着**连平台方都把 multi-agent 当窄场景例外**——这是最强的"默认单 agent"背书。
- **加分点**：把它转成可执行门槛：价值÷成本 >> 1 才拆。

## 二、实战项目增量（P3 决策：用评测判断要不要拆多 agent）

> **本章不写新功能，写一份决策。** 这是刻意的——multi-agent 这章的核心能力是"判断该不该拆"，而判断必须**用评测说话**，不是凭感觉。你在第 12 章已经有了 P3 的单 agent 骨架，本章的增量是给它做一次"要不要拆 multi-agent"的体检，并把结论落成文档。

**必做验收标准（缺一不算过）：**

1. **跑一组 ≥5 个真实任务**，用第 12 章的单 agent 骨架，记录每个任务的：是否完成、步数、累计 token、wall-clock 时间、失败原因（如有）。这是你的 **baseline**。
2. **对每个任务标注"是否含真并行 fan-out"**——明确写出"这个任务里有没有多个互相独立、可同时进行的子工作"。这一栏是后面所有判断的核心依据。
3. **算清楚拆 multi-agent 的预估代价**：基于 baseline 的 token 数，按 **10–15× 溢价**估算拆后成本；按**乘积律**（单步成功率 × 串行环节数）估算拆后可靠性。给出具体数字，不许"感觉会变贵"。
4. **写一份 ≤1 页的决策备忘录**，结论必须是三选一并给依据：**(a) 不拆**（默认，且要说明 baseline 已够用 / 没有 fan-out 价值）/ **(b) 部分拆**（只把某个有 fan-out 的子任务拆成 Orchestrator-Worker，其余保持单 agent）/ **(c) 全拆**（必须证明强 fan-out + 高价值 + ROI>1）。
5. **决策里必须显式回答"我拆的理由是不是其实能用更便宜的手段解决"**——如果理由是"上下文乱"，先试 context editing；如果是"想要角色",先试换 prompt。把这一步的排除过程写进备忘录。
6. **如果结论是 (a) 不拆，这就是满分答案**——前提是你拿数据证明了"不拆够用 / 没 fan-out / ROI 不划算"。**敢于用证据说"不该拆"，是这章的最高分。**

**做完要能回答**：拿着你的 5 个任务表，逐个说"这个为什么不拆 / 这个为什么值得拆"，并且每个判断都能指向表里的某一列数据（fan-out？成本？可靠性？）。**这份备忘录就是你面试讲"我怎么做架构决策"的素材。**

**坑点提示：**
- **别为了"做了点什么"就拆**——本章最容易翻车的就是"明明 baseline 够用，但为了交付一个 multi-agent demo 硬拆"。这正是本课程主线项目（decay）踩过的真实坑：连续两次冻结评测否定了"加更多结构层就更好"的假设，最终项目终止。**没验证就加复杂度 = 提前作废。**
- **fan-out 判断要诚实**——很多任务看起来能并行，其实子任务之间有隐藏依赖（B 需要 A 的结果）。有依赖就不是真并行，别自欺。
- **成本估算别只算单价**——要把"汇总阶段上下文爆炸 + 协调开销 + 缓存可能击穿"都算进去，这才是 multi-agent 真实成本。

## 三、设计题（考权衡判断）

> **题目**：你要为一个"竞品分析 Agent"做架构设计。它要做的事是：给定一个赛道，调研 5 家竞品，每家都要查官网、查最近融资、查用户评价、查技术栈，最后产出一份对比报告。请决定**用单 agent 还是 multi-agent**，给出拓扑和理由；如果用 multi-agent，说明怎么控成本、保可靠。

**考点**：能不能识别"这是个**强 fan-out** 任务"（5 家 × 4 个维度 = 20 个互相独立的子调研），并据此做出"值得拆"的判断，同时把成本和可靠性的代价处理掉——而不是无脑全自主多 agent。

**参考思路（不是唯一解，重点是判断链）：**
- **先判 fan-out**：5 家竞品互相独立、4 个维度互相独立 → **强并行 fan-out，命中拆 multi-agent 的硬判据**。这是少数真该拆的场景。
- **选拓扑**：**Orchestrator-Worker**（不是群聊、不是多层级）。1 个 coordinator 拆出 ~5 个 sub-agent（每家一个），每个 sub-agent 内部串行查 4 个维度（或再 fan-out，但注意别过度）。worker 无状态、可并行、可重试——最像 MapReduce，失败模式最可控。
- **控成本**：① worker 用便宜模型（Qwen/DeepSeek/Haiku 级），coordinator 用强模型；② 汇总阶段是成本黑洞，让每个 sub-agent **先自己压缩成结构化摘要**再回传，别把原始网页全塞给 coordinator；③ 缓存上，sub-agent 的 system+工具集保持稳定让各自命中缓存，fan-out 时先发 1 个等它写好缓存再放其余。
- **保可靠**：5 个 sub-agent 是**并行独立**（不是串行依赖），所以单个失败不拖垮整体——这正是"并行冗余优于串行"的体现。给 coordinator 加"某家查失败就标注缺失、不阻塞报告"的兜底（接第 12 章优雅降级）。
- **划边界**：查询是只读动作，可全自动；但"产出报告"如果要对外发布，加一个 HITL 卡点让人过目（接第 12 章 HITL）。

**好答案的标志**：**先用 fan-out 判据证明"这个该拆"，再用 Orchestrator-Worker + 便宜 worker + 摘要压缩 + 并行冗余把成本和可靠性兜住。** 反面教材是上来就群聊或多层级，或者不算成本直接全自主——那是不懂代价。**能说出"这是少数真该拆的场景，因为它强 fan-out"本身就是判断力。**

## 四、系统设计题（含量级估算）

> **题目**：承接第 12 章那道"代码评审 Agent"系统题（每天 2000 个 PR、每任务 8 步、单 agent 日成本约 \$1,100 无缓存 / \$400–500 有缓存）。现在产品经理说"评审质量不够，想拆成 multi-agent：一个 agent 看安全、一个看性能、一个看可读性、一个 coordinator 汇总"。请估算拆后的成本，并判断这个拆法值不值。

**估算框架（数字按 2026 初价目表锚点，重点是算法不是精确值）：**

- **拆后结构**：1 个 coordinator + 3 个专项 sub-agent。每个 sub-agent 自己也是个 agent loop（假设各自约 4–6 步看自己的维度）。
- **成本爆炸点 1——每个 sub-agent 自身膨胀**：单 agent 8 步累计输入 ≈ 124K token（第 12 章算过的平方级膨胀）。现在 3 个 sub-agent 各跑 ~5 步，每个累计输入估 ~50–70K token，**3 个加起来 ≈ 180K token 输入**，已经超过原来单 agent 的 124K。
- **成本爆炸点 2——汇总阶段**：coordinator 要读 3 份 sub-agent 的产出再综合，假设每份产出 ~4K，汇总输入 ~15K+，再加 coordinator 自己的几步。
- **成本爆炸点 3——协调开销**：派活 3 次 + 回收 3 次的消息往返。
- **粗略总账**：拆后单任务 token ≈ 单 agent 的 **3–4 倍**起（保守，没算缓存击穿；Anthropic 的真实数据是 15×，因为它们 fan-out 更宽）。**日成本从单 agent 的 \$1,100 量级抬到 \$3,500–4,500 量级（无缓存）。**
- **缓存进一步恶化**：3 个 sub-agent 若用不同 system/工具集，各自前缀不同，且 coordinator 等 3 个 sub-agent 跑完才汇总——**等待极易超 5 分钟 TTL，汇总前缀缓存失效**。缓存收益打折，实际可能比"单 agent 有缓存"贵 **8–10 倍**。

**要点（面试官想看到的判断）：**
1. **先问最关键的一句：代码评审任务有强 fan-out 吗？** ——**勉强算有**（安全/性能/可读性三个维度互相独立可并行），但 fan-out 宽度只有 3，远不如调研任务的 20。**fan-out 越窄，拆的 ROI 越差。**
2. **再问：质量问题真的是"单 agent 看不过来"导致的吗？** 很可能不是——更可能是 prompt 不好、上下文管理差。**拆 multi-agent 前，先试"给单 agent 更好的 review prompt + 分段聚焦"**，这几乎零成本，而拆要付 3–10× 成本。这是"理由是不是其实有更便宜解法"的排除。
3. **可靠性账**：3 个 sub-agent 是**并行独立**的（各看各的维度），所以可靠性不至于乘积式塌方——这点比串行链好。但 coordinator 汇总是新增的单点。
4. **结论判断**：**这个拆法 ROI 大概率不划算**——fan-out 太窄（只有 3）、质量问题更可能是 prompt 问题、成本却要涨 3–10×。**正确建议是：先优化单 agent 的 prompt 和上下文，把"拆 multi-agent"作为最后手段，且若要拆只拆成"3 个并行只读 sub-agent + 极薄的 coordinator 汇总"，并用便宜模型 + 摘要压缩控成本。**
5. **Batch 仍然跑不了这个**（同第 12 章陷阱）——交互式 agent 编排不能塞 Batch，但"对每个文件生成一段独立摘要"这种单次子任务可以离线塞 Batch。

**好答案标志**：**能算出"拆后成本涨 3–10×"的量级；能识别 fan-out 宽度只有 3 所以 ROI 差；敢说"先优化单 agent prompt，别急着拆"。** 比只会答"拆成多 agent 质量更好"高好几个段位——后者完全没算代价，正是本章批判的对象。

1. **验证真并行**：在 `run_worker` 里打一条带时间戳的日志（进入/离开）。跑 4 个城市，**确认它们的执行时间窗口是重叠的**（并行），而不是首尾相接（串行）。这是 multi-agent 唯一不可替代价值的实证。

**暗坑提示：**
- **`resp.content[0]` 不一定是 text**：adaptive thinking 下第一个块可能是 thinking 块，必须用 `next(b.text for b in resp.content if b.type == "text")` 过滤，别下标硬取（第 4、5 章反复踩过）。
- **`next(...)` 本身也会炸——这是 multi-agent 比单 agent 更险的地方**：worker 用了小 `max_tokens`（这里 400）+ adaptive thinking，模型有概率把预算**全花在 thinking 上、一个 text 块都不返回**（`stop_reason=max_tokens`）。此时上面那句裸 `next(...)` 抛 `StopIteration`，**单个 worker 崩了会把整个 fan-out 拖垮**。两个改法：给默认值 `next((b.text for b in resp.content if b.type == "text"), "")`，或先判 `if resp.stop_reason == "max_tokens": ...`。**这正是第五节"失败放大"的微缩版——单 agent 里这只是一次失败，multi-agent 里它会顺着汇总链传染。** 把 worker 调用包进 `try/except` 并对失败 worker 走"标注缺失、不阻塞汇总"的优雅降级（接第 12 章），是 Orchestrator-Worker 的标配，不是可选项。
- **worker 必须自己压缩**：如果 TODO 1 不限定"≤30 字摘要"，worker 会回传长篇大论，coordinator 汇总那一步的输入 token 直接爆炸——**这就是 multi-agent 汇总成本黑洞的微缩复现**。亲手让它爆一次，再加上压缩约束，体会"摘要压缩"为什么是 multi-agent 控成本的关键手段。
- **prompt cache 在这里基本帮不上忙**：4 个 worker 各调一次、各自前缀短，且这是一次性任务没有重复前缀——**别指望缓存省钱**。这恰恰说明 multi-agent 的成本省不了，只能靠"便宜模型 + 摘要压缩 + 减少 worker 数"硬控。
- **别给这个任务硬上更多层**：4 个城市的天气查询，fan-out 宽度刚好，**不需要再套一层 sub-coordinator**。能用一层 Orchestrator-Worker 解决的，多套一层只会增加失真和成本——这正是 CMA"只允许一层 delegation"想帮你避免的。
