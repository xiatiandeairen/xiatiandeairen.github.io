时效标注：rerank 模型 / embedding 模型 / 具体向量库产品 / GraphRAG 工具按 **2026 初**写，名单会变，**重点学选型方法论**；而"召回-精排两阶段""双塔 vs 交互式的精度-延迟权衡""小块检索大块喂""长上下文不能替代检索的成本-权限-可控三条硬约束"这些是**架构规律**，讲透到骨子里。

## 一、背景：naive RAG 到底差在哪，对症下药表

### 1. 先把问题归类：所有 RAG 失效，本质是三个环节出问题

第 8 章那张 10 条失效清单看着吓人，但**收敛一下只有三类病根**，记住这个分类，后面所有招数都能挂上去：

1. **query 侧问题（检索没召回 / 召回不全）**：用户的问法和文档的写法对不上，或者一个问题需要多次检索才够。表现：答案明明在库里却没捞出来。
2. **ranking 侧问题（召回了但不相关 / 关键的没排上来）**：粗召回把相关的捞回来了，但混在一堆噪声里，喂给模型的 top-n 不够精。表现：检索结果里有对的，但模型没抓住。
3. **chunk 结构问题（答案在多块里凑 / 块太小语义断）**：切分的粒度和检索-生成的需求错配。表现：召回了片段，答不全；或者召回的块缺上下文，模型看不懂。

外加两个**工程治理类**（不是检索算法问题，是数据/系统问题）：

4. **数据/隔离问题（幽灵知识 / 权限泄露 / 重复挤占）**：ingestion 和访问控制没做好。
5. **推理深度问题（多跳答不了）**：单轮检索的范式天花板，需要让检索变成多轮、可迭代、甚至 agent 自主驱动。

> 心法：**naive RAG 的全部升级，就是在这三类病根（query / ranking / chunk）上，各加一套对症的工程手段，再用评测确认每一步真有收益。** 别背招数，背这个"症状→病根→旋钮"的映射——面试和实战都靠它。

### 2. 演进：从 naive RAG 到 advanced RAG 到 modular RAG

业界对 RAG 的演进有个公认的三段划分，面试常考，先建立坐标：

| 阶段 | 形态 | 特征 | 局限 |
|---|---|---|---|
| **Naive RAG** | `检索 top-k → 拼 prompt → 生成` | 一条直线，无 query 处理、无精排 | demo 可用，生产拉胯（第 8 章那 10 条） |
| **Advanced RAG** | 在检索**前**加 query 优化、检索**后**加 rerank/压缩 | pre-retrieval + post-retrieval 两端增强 | 仍是固定流水线，路径写死 |
| **Modular RAG** | 把检索、改写、路由、记忆做成**可编排的模块**，由逻辑（甚至 agent）动态决定调用哪些、调几次 | 检索次数和路径**运行时决定** | 复杂度高，需要评测和可观测兜底 |

本章主体（第二到第六节）讲 **Advanced RAG**——在 naive 流水线的前后两端做增强，这是 80% 生产 RAG 该停的地方。第七节（多跳/迭代检索、GraphRAG、agentic RAG）和第八节（长上下文混合）开始进入 **Modular RAG** 的领地，那是当问题真的需要时才上的重武器。

> 工程推论（贯穿全章的判据，逐条都是面试题）：
>
> **推论一：每加一个高级 RAG 组件，都要回答『它治哪条失效模式、收益用什么指标证明』。** query 改写治"召不回"、rerank 治"召回了不准"、small-to-big 治"答不全"。如果你加了一个组件却说不清它治什么、也没有评测数字撑腰，那不是优化，是给系统**凭空加延迟、加成本、加故障面**。这是区分"跟风堆框架的人"和"懂 RAG 工程的人"的第一道分水岭。
>
> **推论二：高级 RAG 的每一招几乎都在拿延迟/成本换质量。** query 改写多调一次 LLM（+延迟+成本）、multi-query 把检索次数翻几倍、rerank 加一次模型推理、迭代检索把单轮变多轮。**没有免费的午餐**——所以"要不要上"永远是**收益（质量提升）vs 代价（延迟、成本、复杂度）**的权衡，而权衡的前提是你能测出两头。这把第 8 章"先 profile 再优化"的后端本能，直接搬到了 RAG 调优上。

## 二、query 侧：在检索之前，先把问题问对

naive RAG 直接拿用户原始 query 去检索，这是召回失败的头号来源。原因第 8 章讲过——**用户的问法和文档的写法天然不对称**（用户问"怎么退货"，文档写"申请退款的标准流程"）。query 侧的全部手段，都在解决一件事：**在检索之前，把 query 加工成更容易命中文档的形态。**

四招，从轻到重：

### 1. HyDE——用"假答案"去检索，而不是用"问题"

**HyDE（Hypothetical Document Embeddings，假设性文档嵌入）** 的洞察极其漂亮：**query 和 document 不在同一种"文体"里**——query 是疑问句、短、口语；document 是陈述句、长、书面。直接拿疑问句的向量去匹配陈述句的向量，天然有 gap。

HyDE 的做法：**先让 LLM 根据 query 生成一段"假想的答案文档"，再用这段假答案的向量去检索。** 假答案虽然内容可能是编的（甚至有幻觉），但它的**文体、用词、结构**和真实文档高度同构——于是向量匹配命中率大涨。

```
原始 query: "公司的年假有几天？"
        │
        ▼ LLM 生成假想答案（HyDE）
假想文档: "根据公司休假政策，正式员工每年享有 15 天带薪年假，
          入职满 3 年后增加至 20 天，年假可跨年度结转……"
        │
        ▼ embed(假想文档) 而非 embed(query)
        ▼ 拿假答案的向量去检索 → 命中真实的《休假政策》文档
```

> 工程推论：**HyDE 用『假答案的文体』对齐『真文档的文体』，绕开了 query-document 文体鸿沟——但代价是每次检索前多一次 LLM 调用（+延迟+成本），且假答案跑偏会把检索带偏。** 它在**问答型、文档是规范陈述文体**的场景收益最大；在 query 本身就是关键词、或者库里文体杂乱的场景收益有限。坑点：假答案里的具体数字（"15 天"）**不参与匹配判断**，参与的是整体语义文体，所以不用担心它编错数字——你要的是它"长得像文档"，不是它"答得对"。

### 2. multi-query——一个问题，多个问法，并集召回

**multi-query** 的思路：用户一种问法可能漏，那就让 LLM **把原 query 改写成 3~5 个语义等价但措辞不同的变体**，每个变体各检索一次，结果取**并集**再去重。

```
原始 query: "怎么退货？"
        │
        ▼ LLM 改写成多个变体
   ├─ "退货流程是什么"
   ├─ "如何申请退款"
   ├─ "商品退回的步骤"
   └─ "退货政策说明"
        │
        ▼ 每个变体各检索 top-k，结果并集去重
        ▼ 一起送进 rerank（见第三节）
```

它治的是**单一问法的召回盲区**——某个文档恰好用"退款"而非"退货"，原 query 漏了，但"如何申请退款"这个变体能命中。

> 工程推论：**multi-query 用『召回数量』换『召回覆盖』——检索次数翻 N 倍，召回率上去了，但候选集变脏、成本和延迟都涨。** 所以 multi-query **几乎必须配 rerank**：它负责"广撒网"求全，rerank 负责"精挑"求准，两者是黄金搭档。单用 multi-query 不加精排，等于把更多噪声塞给模型，可能**越改写越差**。这是新手第一误区：以为 multi-query 是免费的召回提升，其实它把"召回不准"的问题放大了，必须用下游精排接住。

### 3. query 分解——把复杂问题拆成可独立检索的子问题

有些 query 是**复合的**，一次检索答不全。比如"对比 A 产品和 B 产品的保修政策"——这需要分别检索 A 的保修、B 的保修，单次检索很难同时把两份都捞全。

**query 分解（decomposition）**：让 LLM 把复杂 query 拆成几个**原子子问题**，每个子问题独立检索，再把材料合起来生成。

```
原始 query: "对比 A 产品和 B 产品的保修政策，哪个更划算？"
        │
        ▼ LLM 分解
   ├─ 子问题1: "A 产品的保修政策是什么？"  → 检索 → 材料A
   └─ 子问题2: "B 产品的保修政策是什么？"  → 检索 → 材料B
        │
        ▼ 材料A + 材料B 一起喂模型 → 对比生成
```

注意它和 multi-query 的区别：**multi-query 是"同一个问题的不同问法"（语义等价），query 分解是"一个问题拆成不同的子问题"（语义互补）。** 前者求"问法覆盖"，后者求"子任务覆盖"。面试常把这两个搞混，能讲清区别是加分点。

> 工程推论：**query 分解是『多跳/复合问题』的轻量解法——但它假设子问题之间相互独立、能并行检索。** 一旦子问题有依赖（"先查 A 的供应商，再查这个供应商的资质"——子问题2 依赖子问题1 的结果），并行分解就不够了，得上**迭代检索/多跳**（见第七节开头）。判据：**子问题能并行 → query 分解就够；子问题有先后依赖 → 必须迭代检索。**

### 4. query 路由——先判断这个问题该去哪检索

生产系统往往**不止一个知识源**：产品文档库、API 文档库、工单库、还有一个能查实时数据的 SQL 工具、甚至一个该直接走闲聊不检索的分支。**query 路由（routing）** 就是在检索之前，先用一个分类器（通常是 LLM 或小模型）判断：**这个 query 该路由到哪个（或哪几个）数据源 / 检索策略。**

```
用户 query
   │
   ▼ 路由器（LLM 分类 / 语义匹配）
   ├─ "API 怎么调用" ────► 路由到 API 文档库
   ├─ "我的订单状态" ────► 路由到订单 SQL 工具（不是向量检索！）
   ├─ "你好" ───────────► 不检索，直接闲聊
   └─ "退货政策"  ───────► 路由到政策文档库
```

路由的价值：(a) **避免无关库的噪声**——查 API 的问题不该把政策文档捞进来；(b) **省成本/延迟**——闲聊不必检索；(c) **接异构后端**——有些问题该走结构化查询（SQL）而非语义检索。

> 工程推论：**query 路由是把『一个万能检索器』拆成『多个专用检索器 + 一个调度器』，本质是检索层的『微服务化』。** 对后端工程师这是熟悉的味道——这就是 API 网关的路由层。路由错了下游全错（路由到错误的库，再精的检索也白搭），所以路由器的准确率是**整个系统的天花板之一**。坑点：路由别做太细（类目太多分类器准确率掉），也别做太粗（退化成不路由）。生产常见做法是**先粗路由到 2~3 个大类，召回时让多个库都查、再用 rerank 融合**——这是"路由 + 融合"的稳健组合，比硬路由到单一库容错性好。

### 5. query 侧四招横向对照（背这张表）

| 招数 | 治哪条失效 | 核心机制 | 主要代价 | 何时上 |
|---|---|---|---|---|
| **HyDE** | 召不回（文体不对称） | LLM 造假答案，用假答案向量检索 | +1 次 LLM 调用；假答案可能带偏 | 问答型、文档是规范陈述文体 |
| **multi-query** | 召不全（单一问法盲区） | LLM 造多个等价问法，并集召回 | 检索次数翻 N 倍；候选变脏，**必须配 rerank** | query 表述多样、召回率是瓶颈 |
| **query 分解** | 复合问题答不全 | LLM 拆原子子问题，各自检索 | +1 次 LLM 调用；仅适合可并行子问题 | 对比类、多实体、复合问题 |
| **query 路由** | 噪声/接错源 | 分类器决定查哪个库/策略 | 路由错则全错；类目设计难 | 多知识源、异构后端 |

> 心法：**query 侧的一切，都在把"用户怎么问"翻译成"文档怎么写"。** 用户的口语疑问句和文档的书面陈述句之间有一道鸿沟，HyDE 用文体填、multi-query 用数量填、分解用拆解填、路由用分流填。这四招可以叠加（先路由、再 multi-query、每个变体可选 HyDE），但**每叠一层就多一份延迟和成本，别无脑全上——用第 11 章的评测决定哪几招对你的数据真有收益。**

## 三、rerank：召回求全，精排求准的两阶段架构

这是高级 RAG 里**性价比最高、几乎无脑该上**的一招。第 8 章已经埋了引子（rerank 环节），这节讲透它的原理、选型和工程。

### 1. 为什么必须两阶段：双塔的精度天花板

回顾第 8 章的关键认知：**向量检索是"双塔（bi-encoder）"结构**——query 和 document **各自独立**编码成向量，再算相似度。独立编码意味着：编码 document 时它**不知道** query 是什么，编码 query 时它也**不知道** document 是什么。两边各编各的，最后只比一个余弦距离。这让它**快**（document 向量可以离线预编码、建索引），但**精度有天花板**——它无法捕捉 query 和 document 之间细粒度的交互（比如"这个否定词让语义反转了"）。

**rerank 用的是"交互式（cross-encoder）"结构**——把 query 和**每个候选 document 拼在一起**，整体过一遍模型，直接输出一个相关性分数。因为 query 和 document **在同一次前向里互相看见**，它能捕捉细粒度交互，**精度高得多**。代价是**慢**——每个候选都要和 query 拼起来跑一次，无法预编码、无法建索引。

于是黄金架构是**两阶段召回-精排**：

```
                            ┌─────────────── 阶段一：粗召回（求全、快）──────────────┐
用户 query (可能已 query 侧处理过)                                                   │
        │                   │  双塔向量检索（+可选 BM25 混合）                         │
        ▼                   │  从百万级库里捞 top-k（k 大，如 50~100）                 │
   embed(query) ───────────►│  目标：把相关的尽量都捞进来，不漏 → 召回率优先          │
                            └────────────────────┬───────────────────────────────────┘
                                                 │ top-50 候选
                            ┌────────────────────▼──────────── 阶段二：精排（求准、慢）─┐
                            │  cross-encoder rerank：query+候选逐个拼一起打分          │
                            │  从 top-50 里精排出 top-n（n 小，如 3~5）                │
                            │  目标：把最相关的排到最前 → 精度优先                      │
                            └────────────────────┬───────────────────────────────────┘
                                                 │ top-5 精材料
                                                 ▼ 拼 prompt → 生成
```

**为什么不直接用 cross-encoder 检索全库？** 因为百万级文档每个都和 query 拼起来跑模型，延迟和成本会爆炸。**双塔负责"从百万里快速捞 50 个候选"（它擅长规模），cross-encoder 负责"从 50 个里精挑 5 个"（它擅长精度，50 个的量它跑得动）。** 这是用两种结构各自的长处，组合出"既快又准"。

> 工程推论：**召回-精排两阶段，是检索系统对抗『规模 vs 精度』矛盾的通用解，不是 RAG 专属。** 搜索引擎、推荐系统几十年都是这个架构——粗排从海量里筛候选（快、糙），精排从候选里定排序（慢、精）。这和第 1 章延迟的 **TTFT + decode 两段式**是同构的思维：**把一个矛盾的目标，拆成两个阶段、各用擅长的手段。** 面试能把 rerank 类比到搜索/推荐的粗排-精排，说明你理解的是架构规律而非 RAG 招式。

### 2. rerank 模型选型（按 2026 初，名单会变，学方法论）

rerank 模型市场上有几类，选型判据如下：

| 类型 | 代表（2026 初，会变） | 特点 | 适用 |
|---|---|---|---|
| **闭源 API rerank** | Cohere Rerank、各云厂商 rerank API | 开箱即用、效果稳、不用自己部署 | 中小团队、不想运维模型 |
| **开源 cross-encoder** | BGE-reranker、各开源系列 | 自部署、可微调、数据不出域 | 数据敏感、量大想控成本、有 ML 运维能力 |
| **late-interaction（晚期交互）** | ColBERT 系（多向量、token 级匹配） | 介于双塔和 cross-encoder 之间：doc 的 token 向量可**离线预编码建索引**（像双塔一样快），但匹配时做 token 级细粒度交互（比双塔准）。代价是存储爆炸（一个 chunk 存几十上百个 token 向量） | 想要"接近 cross-encoder 的精度 + 接近双塔的延迟"，且存储扛得住 |
| **LLM-as-reranker** | 直接用 LLM 给候选打相关性分（listwise，一次看一批排序） | 最灵活、可注入复杂判据（"优先最新""同等相关选权威来源"） | 候选少、判据复杂、对延迟不敏感 |

选型核心判据：**(a) 数据能不能出域**（敏感数据→自部署开源）；**(b) 量级与延迟预算**（高 QPS→专用 rerank 模型而非 LLM）；**(c) 团队运维能力**（没 ML 运维→闭源 API）。注意 rerank 模型和 embedding 模型一样**有语言适配问题**——中文场景要选中文支持好的 reranker，别拿纯英文 reranker 硬套。

> 工程推论（精排家族的速度-精度光谱，背这条）：**双塔（query/doc 各编一个向量）→ late-interaction（doc 编一堆 token 向量、查询时做 token 级 MaxSim 交互）→ cross-encoder（query+doc 拼一起整体过一遍）→ LLM-as-reranker（让大模型读着候选直接排），从左到右精度递增、延迟和成本递增，可预编码性递减。** 这不是"选一个最好的"，是按你的延迟预算在这条光谱上**选一个落点**：绝大多数生产 RAG 落在"双塔粗召回 + cross-encoder 精排"；对延迟极敏感又想要高精度的，late-interaction 是那个常被忽视的中间答案；判据极复杂、候选已经很少（<20）的，才值得上 LLM listwise rerank。面试能把"为什么不直接全用 cross-encoder / 为什么 ColBERT 既快又比双塔准"这条光谱讲清楚，是从"会调库"到"懂检索系统"的分水岭。

> 工程推论：**rerank 是高级 RAG 里『投入产出比最高』的一步——它同时是质量优化和成本优化。** 质量上：把精准的 top-n 排上来，直接干掉第 8 章失效模式 2（召回了不相关）和缓解 4（lost in the middle）。成本上（第 8 章推论）：rerank 把喂给模型的 n 从 50 砍到 5，**输入 token 大降**，而输出按 ~5 倍单价、不受影响——省的是真金白银。**所以"要不要加 rerank"在生产 RAG 里基本不是问题，是默认动作。** 唯一要权衡的是它那一次额外推理的延迟（几十到几百毫秒），对绝大多数问答场景完全可接受。

### 3. 工程细节：k 和 n 怎么定，rerank 放在哪

- **k（粗召回数量）**：取大些（50~100），因为粗召回的使命是"别漏"。k 太小，相关文档可能根本没进候选集，rerank 再精也救不回——**rerank 只能在候选集内部重排序，无法召回候选集外的文档**（这是死穴，必须记住）。
- **n（精排后喂模型的数量）**：取小（3~8），由两个约束卡：上下文窗口预算 + lost-in-the-middle（塞太多反而稀释注意力，第 8 章讲过）。
- **固定 n vs 分数阈值（进阶，常被忽略）**：直接"取 top-n"是**定数**策略——但不同 query 的难度天差地别。简单事实问答可能 top-1 就够，复杂综合问题可能 top-8 还嫌少。固定 n 的毛病是：简单问题塞了一堆勉强相关的（引入噪声、烧 token），难问题又砍掉了本该带上的。更精细的做法是**按 rerank 分数卡阈值**——只保留相关性分数 ≥ 某阈值的候选，让 n **随 query 自适应**。生产里常见组合：**先卡阈值过滤明显不相关的，再对剩下的取 top-n 兜底上限**（防止某些 query 阈值之上太多撑爆上下文）。阈值多少要用第 11 章评测集标定，不是拍脑袋。能讲"为什么固定 n 不够、什么时候该上分数阈值"，是精排环节的加分点。
- **rerank 在 multi-query / 路由融合之后**：如果你做了 multi-query（多个变体各召回）或路由融合（多个库各召回），所有候选**汇总去重后一起送进 rerank**——rerank 是把多路召回融合成一个统一排序的天然汇流点。**注意**：这里"汇总"只是把多路候选并成一个去重后的集合，**真正的统一排序是 rerank 做的**（cross-encoder 对每个候选独立打分，天然可比、不存在多路分数量纲不可比的问题）——所以有了 rerank 这一层，多路召回的融合非常干净，不必再纠结 RRF 那类"只靠排名融合"的技巧（那是第 7 章 hybrid 检索在**没有精排**时的融合手段）。

> 坑点提示：**rerank 救不了"召回阶段就没捞到"的文档。** 新手常见误区是"加了 rerank 还答不对，再换更强的 reranker"——但如果根因是 chunking 切断了答案、或 k 太小漏了，那是召回侧的病，换 reranker 治不了。**rerank 是精排，前提是粗召回把对的捞进来了。** debug 顺序永远是：先确认相关文档进没进候选集（召回问题），再谈排序（精排问题）。

## 四、chunk 结构：small-to-big、父子块、句子窗口

第 8 章讲了 chunk 大小的"双向拉扯"——**小块利检索（精准）但损生成（上下文断）；大块利生成（完整）但损检索（区分度差）**。当时埋了一句"small-to-big 绕开二选一"。这节把这个思路讲透——它治的是失效模式 3（**答案在多个 chunk 里凑、单块信息不完整**）。

### 1. 核心思路：检索用的块 ≠ 喂模型的块

二选一之所以是个伪问题，是因为它假设"**检索的单位**"和"**喂给模型的单位**"必须是同一个块。**解耦这两者，矛盾就化解了**：

> **用小块去检索（保证检索精准），但喂给模型时换成它所在的大块/上下文（保证信息完整）。**

这一族技术有三个常见变体，本质是同一个思想的不同实现：

| 变体 | 检索单位 | 喂模型单位 | 实现要点 |
|---|---|---|---|
| **small-to-big** | 小子块 | 子块所属的大父块 | 子块 embedding 入库，每个子块记录其父块 ID；命中子块后取父块全文喂模型 |
| **parent-child（父子块）** | 子块 | 父块 | 同上，是 small-to-big 的典型落地命名；父块是章节级，子块是段落/句子级 |
| **句子窗口（sentence-window）** | 单句 | 该句 + 前后 N 句 | 以句为检索粒度（最精准），命中后把它周围的窗口一起取出喂模型（补上下文） |

```
                  检索阶段（用小块，精准命中）          生成阶段（换大块/窗口，信息完整）
small-to-big:    [子块3] ←命中─ query                取 [父块A = 子块1+2+3+4] 全文喂模型
句子窗口:        [句子27] ←命中─ query               取 [句24~30 这个窗口] 喂模型
```

### 2. 为什么有效：检索精度和生成完整度各自达到最优

- **检索侧**：小块（甚至单句）embedding **语义集中、区分度高**，query 一来精准命中那一小段。如果用大块 embedding，语义被平均，区分度差，命中率低。
- **生成侧**：命中后**不只喂那一小段**，而是喂它所在的完整父块/窗口，模型拿到的是**有上下文的完整信息**，不会出现"召回了半句话，模型看不懂"或"答案横跨两个小块只召回一个"。

这一招直接干掉失效模式 3 和缓解 6（曲解材料——上下文不全是曲解的常见原因）。

> 工程推论：**small-to-big 的本质是『把"检索精度"和"生成上下文"两个目标解耦，各自优化到最好』，而不是在 chunk 大小上找一个妥协的中间值。** 这是个高级认知——naive 的做法是纠结"chunk 切 256 还是 512"，找一个两边都不太满意的折中；high-level 的做法是**意识到这俩根本不该是同一个块**，直接拆开。面试被问"chunk 大小怎么定"，能从"双向拉扯"进阶到"用 small-to-big 解耦，绕开二选一"，是明显的水平分层。

> 坑点提示：small-to-big 不是免费的——它要求 ingestion 阶段就**建立子块到父块的映射关系**（每个子块存它的父块 ID 或直接存父块文本），存储和索引复杂度上升。而且父块取多大要权衡：父块太大又退化成"喂太多噪声 + lost in the middle"。**句子窗口的窗口大小、父子块的父块粒度，都是要用评测调的旋钮，不是拍脑袋。**

## 五、元数据过滤与多租户权限隔离

这节治的是失效模式 7（幽灵知识）、8（**权限泄露**）、以及生产 RAG 绕不开的**多租户隔离**。这不是检索算法问题，是**数据治理 + 安全**问题——但它往往是企业 RAG 上线的**最高优先级**，因为权限泄露是事故，不是 bug。

### 1. 元数据过滤：检索不只是"语义相似"，还要"满足约束"

每个 chunk 入库时除了向量，还带一组**元数据（metadata）**：来源文档、部门、密级、时间、语言、tenant_id 等。检索时，**先按元数据硬过滤，再在过滤后的子集里做语义检索**。

```
query: "最新的报销标准是多少"  + 当前用户: {tenant: 'acme', dept: 'sales', clearance: 2}
        │
        ▼ 元数据过滤（硬条件，先筛）
   WHERE tenant_id = 'acme'           ← 多租户隔离：只看本租户数据
     AND clearance <= 2               ← 权限：只看够级别的
     AND updated_at > '2026-01-01'    ← 时效：只看新的
        │
        ▼ 在过滤后的子集里做向量检索
        ▼ rerank → 生成
```

元数据过滤同时解决三件事：**多租户隔离**（按 tenant_id）、**权限控制**（按密级/部门 ACL）、**时效过滤**（按时间，治幽灵知识）。

### 2. 多租户的两种架构：过滤式 vs 物理隔离

多租户（一套系统服务多个客户/组织，数据严格隔离）有两条路线，面试高频对比：

| 方案 | 做法 | 优点 | 缺点 | 适用 |
|---|---|---|---|---|
| **元数据过滤（共享索引）** | 所有租户数据在同一索引，靠 `tenant_id` 元数据过滤 | 资源利用率高、运维简单、扩容容易 | 隔离是"逻辑的"——一旦过滤条件写漏，**跨租户泄露**；存在噪声邻居 | 租户多、单租户数据量小、SaaS 标配 |
| **物理隔离（独立索引/命名空间）** | 每个租户一个独立索引或 namespace | 隔离是"物理的"，最安全；性能互不干扰 | 资源开销大、租户多时运维爆炸、冷租户浪费 | 强合规、大客户、数据极敏感 |

很多向量库支持 **namespace / collection** 这种中间方案——逻辑上共享一套部署，但每个租户有独立命名空间，兼顾隔离性和运维成本。**判据：合规要求 + 单租户数据量 + 租户数量**——金融/医疗大客户倾向物理隔离，通用 SaaS 倾向过滤式 + namespace。

### 3. 安全铁律：权限必须在检索层落地，不能靠 prompt 兜底

这是第 1 章推论四（**模型不可信**）在 RAG 安全上的直接应用，也是这节最重要的一条：

> **无权的 chunk，必须在检索阶段就被元数据过滤掉，根本不进候选集——让模型『看都看不到』，才是真安全。**

错误做法是：全库召回，然后在 prompt 里写"请不要泄露用户无权访问的内容"。这是把安全寄托在模型自觉上——**而模型不可信**（推论四）。只要无权材料进了上下文，就有泄露风险：模型可能违反指令、可能被 prompt 注入诱导、可能在引用里带出片段。**安全边界要前移到检索层，从源头上不把无权数据喂进去。**

> 工程推论：**RAG 的权限模型是『检索即授权』——能被检索到的，就是用户有权看到的；过滤发生在召回之前，而非生成之后。** 这条对后端工程师极其自然：这就是数据库的**行级安全（row-level security）**搬到向量检索上。坑点是**过滤和检索的执行顺序与性能**——理想是"先过滤再检索"（pre-filter，候选集干净但可能拖慢 ANN 索引），有些库是"先检索再过滤"（post-filter，可能过滤后候选不够 k 个）。**安全场景必须用 pre-filter 语义**（无权数据绝不进候选），哪怕牺牲一点性能。选向量库时务必确认它的过滤是 pre 还是 post，以及对 ANN 召回率的影响——这是个容易被忽略但很要命的工程点。

> 面试加分点：被问"RAG 怎么做多租户和权限"，初级答案是"加个 tenant_id 字段过滤"。高级答案要点出三层：**(1) 隔离架构选型**（过滤式 vs 物理隔离 vs namespace，按合规和规模权衡）；**(2) 安全边界前移**（检索层过滤而非 prompt 兜底，因为模型不可信）；**(3) pre-filter vs post-filter 的工程含义**（安全场景必须 pre-filter，且注意对 ANN 召回的影响）。能讲到第 3 层，说明你真在生产里踩过。

## 六、contextual retrieval：给每个 chunk 补回它丢失的上下文

这是 Anthropic 在 2024 年提出、2026 初已成生产常用手段的一招，专治一个被低估的失效根因：**chunk 被切下来后，丢失了它在原文里的上下文，导致检索时语义残缺。**

### 1. 问题：孤立的 chunk 是"失去语境的碎片"

设想一份财报被切成 chunk，其中一块是：

> "该季度营收增长了 3%，主要由订阅业务驱动。"

这个 chunk 单独看，**信息是残缺的**：哪个公司？哪个季度？和谁比增长 3%？检索时，用户问"Acme 公司 2025 年 Q3 营收情况"，这个 chunk 因为**没有"Acme""2025 Q3"这些词**，向量匹配可能命中不了——尽管它就是答案。**切分这个动作本身，剥离了 chunk 的上下文锚点。**

### 2. 解法：入库前，给每个 chunk 加一段上下文摘要

**contextual retrieval** 的做法：ingestion 阶段，对每个 chunk，**让 LLM 看着整篇文档，为这个 chunk 生成一小段"它在全文中的上下文说明"，拼在 chunk 前面再做 embedding（和 BM25 索引）。**

```
原始 chunk:
   "该季度营收增长了 3%，主要由订阅业务驱动。"
        │
        ▼ 喂给 LLM：整篇文档 + 这个 chunk，要求生成上下文前缀
LLM 生成的上下文:
   "本段出自 Acme 公司 2025 年 Q3 财报的营收分析章节，对比 2024 年同期。"
        │
        ▼ 拼接后入库（embedding + BM25 都用这个增强版）
增强 chunk:
   "本段出自 Acme 公司 2025 年 Q3 财报的营收分析章节，对比 2024 年同期。
    该季度营收增长了 3%，主要由订阅业务驱动。"
```

现在这个 chunk 带上了"Acme""2025 Q3"这些锚点，用户那个 query 就能命中了。它把"切分时丢失的上下文"**补了回来**。

### 3. 工程权衡：一次性的离线大开销，换长期检索质量

- **成本**：每个 chunk 都要调一次 LLM 生成上下文（外加要把整篇文档作为输入）。库大时这是**一笔可观的一次性离线开销**。
- **省钱关键**：把整篇文档作为生成上下文的输入时，**用 prompt caching 把"整篇文档"这段固定前缀缓存住**（第 2 章），同一文档的多个 chunk 复用同一份文档缓存——这能把 contextual retrieval 的成本砍掉一大截。这是 prompt caching 在 ingestion 阶段的经典应用：**同一份长文档被反复当输入处理时，缓存红利极大**（正好是第 8 章讲的"缓存真正吃到红利的形态"之一）。
- **收益**：显著降低失效模式 1（召不回）——尤其对**碎片化、强依赖上下文的文档**（财报、法条、技术手册）。Anthropic 的数据显示它能明显降低检索失败率，配合 rerank 效果更佳。

> 工程推论：**contextual retrieval 是用『离线算力（给每个 chunk 加上下文）』换『在线检索质量』——这是 RAG 工程里『把工作前移到离线』的典范。** 离线阶段不计较延迟、可批处理、可缓存，正适合干这种重活；把它做好，在线检索就轻松。这和"small-to-big 把生成上下文需求解耦到映射关系"是同一种工程哲学：**能在离线（ingestion）阶段解决的，绝不留到在线。** 坑点：contextual retrieval 依赖 load 出来的是**干净、完整的文档**（你得让 LLM 看到全文才能写对上下文）——如果文档解析（第 10 章）就拍扁了表格、丢了结构，生成的上下文也会错。又一次印证第 8 章那条因果链：**数据质量在最上游，塌了全塌。**

## 七、多跳检索、GraphRAG 与 agentic RAG：当单轮向量检索不够用时

前面六节都在优化"**单轮**向量检索 + 生成"这条主线。但有几类问题，**无论你把单轮检索优化到多好都解不了**，需要换范式：要么多查几轮（多跳/迭代检索），要么换数据结构（GraphRAG），要么把控制权整个交出去（agentic）。这节按"从轻到重"讲这三件武器，以及——更重要的——**什么时候不该上它们**。

### 1. 多跳 / 迭代检索：一轮不够就再查一轮

第二节的 query 分解能拆"可并行"的复合问题，但治不了**有先后依赖**的问题（失效模式 9）：

> "我们最大的供应商，它的母公司在哪个国家注册？"

这问题没法一次检索拿全——你得**先查出"最大的供应商是谁"，拿到答案（比如"X 公司"），再用 X 公司去查"X 的母公司注册地"**。第二步的 query **依赖第一步的检索结果**，没法提前并行拆出来。这就是**多跳（multi-hop）**：答案要顺着"检索→读结果→据此生成下一个 query→再检索"的链条走几步才能凑齐。

**关键认知：多跳不等于一定要上 agent。** 它有两种实现形态，复杂度差很多：

| 形态 | 谁决定"查几轮、查什么" | 适用 | 代价 |
|---|---|---|---|
| **固定循环迭代检索** | 代码写死的循环逻辑（如"最多迭代 N 轮，每轮把上一轮检索到的关键实体填进下一个 query 模板，直到某停止条件"） | 跳数**可预期、模式固定**的多跳（如"实体 A→A 的属性 B→B 的属性 C"这种链路结构已知） | 路径写死，跳出预设模式就抓瞎；但延迟/成本**可控可预测** |
| **agent 驱动迭代检索** | LLM 在循环里自主判断（见下面 §3） | 跳数**不确定、要动态决定**还查不查、查什么、够没够 | 最灵活，但延迟/成本不可控（见 §3） |

```
固定循环迭代检索（非 agent，路径可预期）：
  query "最大供应商的母公司注册地"
     │
     ▼ 第1轮检索 + 抽取 ──► "最大供应商 = X 公司"
     │
     ▼ 把 X 填进下一轮 query 模板："X 公司的母公司注册地"
     │
     ▼ 第2轮检索 ──► "X 的母公司在 Y 国注册"
     │
     ▼ 材料齐了 → 停止 → 生成
```

> 工程推论：**多跳检索的本质是『把单轮检索的"一次性拿全"放宽成"分几步逐步逼近"』——但先别急着上 agent。** 如果你的多跳问题**链路结构是已知固定的**（"先查实体、再查它的某个固定属性"），一个**写死的固定循环**就够了：它把单轮变多轮、解决了依赖问题，又保留了延迟和成本的可预测性，不引入 agent 那套不确定性。只有当"要不要再查一轮、下一个 query 是什么、什么时候算够"**本身需要模型动态判断**时，才升级到 agent 驱动（§3）。判据：**链路固定→固定循环迭代；链路要动态决策→agentic。** 面试能区分"固定循环迭代检索"和"agentic 检索"这两种多跳实现，并说清"不是所有多跳都得上 agent"，是明显的水平分层——很多人一听"多跳"就条件反射上 agent，白白引入不可控延迟。

### 2. GraphRAG：把知识连成图，回答"全局性/关系性"问题

**向量检索的盲区**：它擅长"找到和 query 语义最像的几个片段"，但不擅长**需要跨多个文档汇总、或追踪实体关系链**的问题。比如：

- "这份 200 页报告的核心主题有哪些？"（**全局综合**——答案不在某几个 chunk 里，而要俯瞰全局）
- "A 公司和 B 公司通过哪些人产生了关联？"（**关系链**——要顺着"人-公司-人"的关系图走）

这类问题向量检索天然抓瞎——top-k 个局部片段拼不出全局图景，也走不通关系链。

**GraphRAG** 的做法：ingestion 阶段，用 LLM 从文档里**抽取实体和关系，构建知识图谱**；再对图谱做**社区聚类**，为每个社区生成摘要。查询时：

- **全局问题** → 走社区摘要，自底向上综合出全局答案；
- **关系问题** → 在图上做多跳遍历，顺着实体关系链走。

```
文档集 ──LLM 抽取──► 实体+关系 ──► 知识图谱 ──社区聚类+摘要──► 分层图谱
                                                                    │
全局 query "核心主题" ──────────────────────────────────► 走社区摘要综合
关系 query "A 和 B 的关联" ─────────────────────────────► 图上多跳遍历
```

注意上图那两条查询路径**不是一回事**，它们对应 GraphRAG 的两种检索模式，机制不同，面试里能分清是加分点：

- **全局搜索（global search）**：用于"总结/俯瞰全局"类问题。**不进图谱遍历**，而是把预先生成好的**各社区摘要**当作检索单位——先让每个相关社区摘要各自产出一份局部答案（map），再把这些局部答案归并成一个全局答案（reduce）。它本质是"对社区摘要做一次 map-reduce 式综合"，解决的是"答案不在任何单个 chunk 里、要把全库的要点汇起来"。
- **局部搜索（local search）**：用于"具体实体 / 关系链"类问题。**从 query 里的实体入口进图**，向外做多跳遍历，把实体、它的邻居、相关的原文 chunk 一起捞出来喂模型。解决的是"顺着关系走"和"围绕某个实体把分散在多处的信息聚齐"。

> 工程推论：**GraphRAG 治的是向量检索的两个死穴——『全局综合』和『关系推理』，代价是 ingestion 极重（要抽实体、建图、聚类、给每个社区写摘要，是 naive RAG 建库成本的数倍，且文档更新时图谱要增量重建——这是常被低估的运维负担）。** 所以它**不是 naive RAG 的升级，是一个互补的、用于特定问题类型的重型方案**。判据：**你的问题是不是大量『总结全局』或『追踪关系链』的？** 如果用户问的 90% 是"X 是什么/怎么做"这种局部事实问答，GraphRAG 的重投入纯属浪费——向量 RAG + rerank 就够。**别看 GraphRAG 名字酷就上，先看你的问题类型配不配得上它的成本。** 生产里也常见**混合**：向量检索管局部事实问答，GraphRAG 管全局/关系问题，路由器（第二节）分流。

> 别一步跳到全套 GraphRAG——先看轻量替代够不够（这是成本意识）：很多被当成"得上 GraphRAG"的需求，其实有**更便宜的中间方案**：
>
> - "把所有合同里涉及竞业限制的列出来"这类**聚合检索**，往往**结构化元数据标注 + 过滤/聚合**就够了——ingestion 时让 LLM 给每个 chunk 打好结构化标签（条款类型、涉及方），查询变成一次 metadata 过滤 + 聚合，根本不用建图。
> - "围绕某实体把信息聚齐"有时**parent-doc 上卷**（按文档/章节聚合命中片段，small-to-big 的近亲）就能近似，不必抽全图。
> - 只有当问题**真的需要"沿任意关系链多跳"或"对全库做主题级综合"**、且这类问题占比高到值回 ingestion 重投入时，才上完整 GraphRAG。
>
> 决策顺序：**向量 RAG + rerank → 加结构化元数据聚合 → parent-doc 上卷 → 真不够再上 GraphRAG。** 从轻到重，每一步都先用第 11 章评测确认前一档真的不够再往上加——这正是本章推论一"每个组件都要证明它治了某条失效"在重型方案选型上的应用。（GraphRAG 具体实现/工具按 2026 初会变，重点记这条"由轻到重"的选型阶梯，以官方文档为准。）

### 3. agentic RAG：让 agent 自主决定"要不要查、查什么、查够没"

前面所有手段，检索流程都是**写死的**（固定几步、固定顺序）。**agentic RAG** 把检索的**控制权交给一个 agent**——由 LLM 在循环里自主决策：

- **要不要检索？**（这个问题需要外部知识吗，还是直接答？）
- **检索什么？**（生成 query，可能基于上一轮结果调整）
- **检索够了吗？**（评估已有材料够不够答，不够就再查一轮——这就是**多跳/迭代检索**的 agent 化）
- **该换工具吗？**（向量库不够，要不要查 SQL、调 API、搜网页？）

```
                  ┌──────────────── agent 循环 ───────────────┐
用户 query ──────►│  推理: 我需要什么信息？                     │
                  │     │                                       │
                  │     ▼ 决策: 检索 / 调工具 / 直接答          │
                  │  [检索一轮] → 看结果 → 够了吗？             │
                  │     │            ├─ 不够: 调整 query 再查   │◄─ 迭代/多跳
                  │     │            └─ 够了: 退出循环          │
                  │     ▼                                       │
                  └─────┴──────────────► 综合所有材料生成 ──────┘
```

这天然解决**多跳问题**（失效模式 9）——"先查 A 的供应商，再查这个供应商的资质"，agent 第一轮查出供应商，第二轮用这个结果去查资质，**迭代检索**。它是多跳处理的**最重一档**：当 §1 那种"固定循环迭代检索"也兜不住（跳数不定、下一步查什么要看上一步结果临场判断、可能中途要换工具）时，才把控制权整个交给 agent。把整条"有依赖多跳"的处理阶梯串起来就是：**query 分解（可并行，最轻）→ 固定循环迭代检索（链路固定的依赖多跳，中等）→ agentic 检索（链路要动态决策，最重）**——从轻到重，按问题真实需要往上爬，别一上来就用最重的那档。

> 工程推论：**agentic RAG 把『固定检索流水线』升级成『LLM 驱动的动态检索循环』——灵活性拉满，但代价是延迟不可控、成本不可控、可观测性变难、还可能死循环。** 这是典型的"能力换确定性"：naive RAG 路径固定、延迟可预测；agentic RAG 可能查 1 轮也可能查 8 轮，没法提前知道。所以它**只在问题真的需要多步推理/动态决策时才上**——简单事实问答用 agentic RAG 是杀鸡用牛刀，白白引入不确定性。**agent 的完整设计（工具循环怎么写、上下文怎么管、怎么防死循环、子 agent 怎么编排）是第 12 章的主题**，这里你只需建立判据：**问题需要『多步、依赖前序结果、动态选工具』→ agentic RAG；否则别碰。**

> 工程事实（按 2026 初，Anthropic SDK）：agentic RAG 在 Claude 上落地，本质是**工具循环**——把"检索"定义成一个 tool，模型自己决定何时调、调几次，`tool_use → 执行检索 → tool_result 回传 → 循环到 `stop_reason=end_turn``。SDK 提供 tool runner（beta）自动跑这个循环，你不必手撸。Claude 的 adaptive thinking（Opus 4.8 等用 `thinking:{type:"adaptive"}`）让模型自主决定"这一步要不要多想想再决定查什么"，配合 agentic 检索的多轮决策很契合。检索深不深、查几轮，用 effort 档位（low/medium/high/xhigh/max）调思考与花费的预算——这正是"行为控制从采样超参迁移到自然语言+思考预算"趋势在 RAG 上的体现。具体见第 12 章。

## 八、长上下文与 RAG：是替代还是互补？

随着模型上下文窗口冲到百万 token 级（按 2026 初，Frontier 模型普遍 1M 上下文，如 Claude Opus/Fable 系），冒出一个尖锐的问题，也是**当前 RAG 面试最高频的"显水平"题**：

> **"上下文窗口都 1M 了，还要 RAG 干嘛？直接把所有文档塞进去不就行了？"**

能不能漂亮回答这题，直接区分"跟着热点喊口号的人"和"懂工程权衡的人"。

### 1. 先承认长上下文确实改变了一些事

诚实地说：**长上下文让一部分原本必须 RAG 的场景，可以直接塞了。** 比如"针对这一份 80 页合同反复问答"——80 页几万 token，1M 窗口完全塞得下，直接塞进去，模型能看到全文、不会有 chunk 切断问题、不会检索漏召回。这种**单文档/小文档集、且要全局理解**的场景，长上下文（甚至比 RAG）更好——RAG 反而因为切块、检索可能漏信息。**这是长上下文的主场，别嘴硬说 RAG 永远更好。**

### 2. 但长上下文替代不了 RAG 的三条硬约束

然而，"把所有文档塞进去"在生产里撞三堵墙——这三条是回答这道题的**核心弹药**：

| 硬约束 | 为什么长上下文解不了 | RAG 怎么解 |
|---|---|---|
| **成本** | 1M token 输入**每次请求都全价重算**。你有 10GB 企业知识库，根本塞不进 1M；就算塞得进，每次问一句话都付百万 token 的输入费——**经济上不可行** | 只检索 top-n（几千 token）喂模型，成本低两三个数量级 |
| **规模** | 上下文窗口再大也有上限（1M），而企业知识库是 GB~TB 级，**物理上塞不下**；且窗口越满，"lost in the middle"越严重，有效信息被稀释 | 从海量库里精准捞相关片段，库多大都不怕 |
| **权限/时效/可控** | 全塞进去 = 模型看得到全部，**无法按用户做权限隔离**（第五节）；知识更新要重发全部；无法溯源（不知道答案来自哪份文档） | 检索层做 ACL 过滤、按时间过滤、返回来源 chunk 可溯源 |

> 工程推论：**长上下文不是 RAG 的替代品，是 RAG 的『最后一公里增强』——RAG 负责"从海量里精准捞少量相关材料"，长上下文负责"让捞出来的材料可以塞得更多更全"。** 两者是**互补的**：RAG 解决"规模、成本、权限、可控、溯源"，长上下文解决"召回的材料别因为窗口小而被迫切碎/砍少"。**最优架构是 RAG + 长上下文混合**——用 RAG/rerank 把候选缩小到几十个高相关片段，再借长上下文把这几十个**完整地、不切碎地**喂进去（甚至可以 small-to-big 取更大的父块），既享受 RAG 的精准与可控，又享受长上下文的"材料不必为了省窗口而残缺"。

### 3. 混合架构的实操判据

什么时候偏 RAG、什么时候偏长上下文、什么时候混合：

```
知识库规模？
  ├─ 单文档 / 小文档集（几万~十几万 token），且需全局理解
  │     └─► 偏长上下文：直接塞，配 prompt caching（文档钉前缀，复用缓存）
  │
  ├─ 海量库（GB~TB），或需权限/时效/溯源
  │     └─► 必须 RAG：检索缩小范围
  │           └─ 检索出的候选，借长上下文窗口"喂得更全更完整"（混合）
  │
  └─ 中等（一批相关文档，几十万~上百万 token）
        └─► 看成本和频率：偶尔问→可塞；高频问同一批→RAG+缓存更划算
```

> 面试加分点：被问"长上下文是不是要取代 RAG"，标准高级答案三段：**(1) 先承认长上下文确实抢了 RAG 一部分场景**（单文档/小集全局问答，长上下文甚至更好，别嘴硬）；**(2) 抛出三条 RAG 不可替代的硬约束**——**成本（百万 token 全价重算不可行）、规模（GB 库塞不下）、权限/时效/可控（无法隔离、无法溯源）**；**(3) 给出"二者互补、混合架构"的结论**——RAG 负责从海量精选，长上下文负责让精选的材料喂得更全。能讲完这三段，面试官基本认定你懂 RAG 工程而非只会喊概念。
>
> 坑点提示（呼应第 0 章 know）：千万别把这道题答成"长上下文永远取代不了 RAG"这种绝对论断——长上下文确实在持续蚕食 RAG 的场景边界，**未来这条边界还会继续移动**。正确姿势是讲清"在哪些约束下谁更优"，而不是站队。绝对化的论断在面试里是减分项，因为它暴露你没真正理解权衡。

> 心法（本章收尾）：**高级 RAG 不是『往 naive RAG 上堆框架』，是『针对每一类失效模式，精准地加一个对症旋钮，并用评测确认它真有收益』。** query 侧治召不回、rerank 治不精准、small-to-big 治答不全、元数据过滤治泄露、contextual retrieval 治上下文丢失、GraphRAG/agentic 治范式天花板、长上下文混合治"材料喂不全"。**每一个都对应第 8 章那张失效清单上的一条。** 别一上来全家桶——那只会堆出一个慢、贵、难维护的庞然大物。**正确的工程节奏是：先用第 11 章的评测量出 naive RAG 的失效分布，看哪类失效最严重，就先上对症的那一两招，测出收益，再决定下一步。 高级 RAG 的灵魂是『评测驱动的、按需的、可度量的增强』，不是『把能加的都加上』。**

## 章末五件套

### 一、高频面试题（附答题框架）

**Q1：上下文窗口都 1M 了，还要 RAG 吗？（全章最高频）**
- 答题框架：三段式。**(1) 先承认**长上下文抢了 RAG 一部分场景——单文档/小文档集的全局问答，长上下文甚至更好（别嘴硬）。**(2) 抛三条硬约束**：成本（百万 token 每次全价重算，GB 库经济不可行）、规模（库 GB~TB 塞不下，且越满 lost-in-middle 越重）、权限/时效/可控（无法按用户隔离、无法溯源、更新要重发）。**(3) 结论是互补**：RAG 从海量精选，长上下文让精选材料喂得更全，最优是混合架构。
- 错误陷阱：答"长上下文永远取代不了 RAG"（绝对化，暴露没理解权衡——边界一直在移动）；只说"窗口有限塞不下"一条（漏了成本和权限两条更要命的）。

**Q2：rerank 为什么必须有？双塔和 cross-encoder 的区别？**
- 答题框架：双塔（bi-encoder）query/doc **独立编码**算相似度——快（可预编码建索引），但精度有上限（编码时互相看不见）。cross-encoder query+候选**拼一起**过模型打分——精度高（互相看见、捕捉细粒度交互），但慢（不能预编码）。架构是**两阶段**：双塔从百万里粗召回求全（k 大），cross-encoder 从候选里精排求准（n 小）。类比搜索/推荐的粗排-精排、第 1 章延迟的 TTFT+decode 两段式。加分：点出 rerank 同时是质量优化（去噪）和成本优化（砍 n、省输入 token）。
- 错误陷阱：说"向量检索够了不用 rerank"（naive RAG 主要短板）；以为 rerank 能召回候选集外的文档（**rerank 只能在候选集内重排，召不回漏掉的，这是死穴**）。

**Q3：HyDE、multi-query、query 分解、路由，分别治什么？怎么区分？**
- 答题框架：四个都在 query 侧，但治不同病。**HyDE**：造假答案对齐文档文体，治"文体不对称召不回"。**multi-query**：造多个等价问法并集召回，治"单一问法盲区"，**必须配 rerank**。**query 分解**：拆原子子问题各自检索，治"复合问题答不全"。**路由**：分类器决定查哪个库，治"接错源/噪声"。关键区分：**multi-query 是同一问题的不同问法（语义等价），query 分解是拆成不同子问题（语义互补）**。
- 错误陷阱：把 multi-query 和 query 分解混为一谈（等价 vs 互补）；说 multi-query 是免费的召回提升（它放大了"召回不准"，必须下游 rerank 接住）。

**Q4：chunk 切大切小的矛盾，small-to-big 怎么解？**
- 答题框架：先讲双向拉扯——小块利检索（精准）损生成（断），大块反之。**small-to-big 的本质是解耦『检索单位』和『喂模型单位』**：用小块/单句检索（精准命中），命中后换成父块/句子窗口喂模型（信息完整）。三变体：small-to-big、父子块、句子窗口，同一思想。加分：这不是在 chunk 大小上找折中，是**意识到这俩根本不该是同一个块**。
- 错误陷阱：给一个"最佳 chunk 大小"固定值（没理解这是 trade-off，且 small-to-big 已绕开）；说不清检索块和生成块怎么映射（ingestion 要建子块→父块映射）。

**Q5：RAG 怎么做多租户和权限隔离？**
- 答题框架：三层。**(1) 隔离架构**：元数据过滤（共享索引、按 tenant_id，省资源但逻辑隔离有泄露风险）vs 物理隔离（独立索引，最安全但运维重）vs namespace（折中），按合规+规模+租户数选。**(2) 安全边界前移**：无权 chunk 必须在检索阶段过滤掉、不进候选集——靠 prompt 兜底不行，因为模型不可信（第 1 章推论四）。**(3) pre-filter vs post-filter**：安全场景必须 pre-filter（无权数据绝不进候选），注意对 ANN 召回的影响。
- 错误陷阱：只说"加 tenant_id 过滤"（没讲架构选型和安全原理）；把安全寄托在 prompt"别泄露"（无权材料进了上下文就有泄露风险）。

**Q6：contextual retrieval 解决什么问题？为什么有效？**
- 答题框架：问题是 **chunk 被切下来丢失了上下文锚点**（"该季度增长 3%"不知道哪个公司哪个季度，检索时缺关键词命中不了）。解法：ingestion 阶段让 LLM 看全文，为每个 chunk 生成一段上下文摘要拼在前面再 embedding。有效是因为补回了"切分时丢失的上下文"，召回率显著上升。加分：用 prompt caching 把"整篇文档"缓存住（同文档多 chunk 复用），大砍 ingestion 成本；本质是"离线算力换在线检索质量"。
- 错误陷阱：说不清它和 small-to-big 的区别（contextual 是给 chunk **加描述**改善检索，small-to-big 是检索后**换大块**改善生成）；忽略它依赖干净的全文输入（解析烂了上下文也错）。

**Q7：GraphRAG 和向量 RAG 各擅长什么？什么时候上 GraphRAG？**
- 答题框架：向量 RAG 擅长"找语义最像的局部片段"，**死穴是全局综合和关系推理**。GraphRAG 抽实体+关系建图、社区聚类+摘要，**全局问题**走社区摘要综合、**关系问题**走图上多跳遍历。判据：**你的问题是不是大量"总结全局"或"追踪关系链"**？是→GraphRAG（值它的重 ingestion 成本）；90% 是局部事实问答→向量 RAG+rerank 足够，别上。生产常混合+路由分流。
- 错误陷阱：说"GraphRAG 是 RAG 升级版，更好"（不是升级是互补，且 ingestion 成本数倍）；不问问题类型就推荐 GraphRAG。

**Q8：agentic RAG 比固定流水线 RAG 强在哪？代价是什么？**
- 答题框架：固定流水线检索路径写死；agentic RAG 把控制权交给 agent，**自主决定要不要查、查什么、查够没、换不换工具**，天然解决多跳/迭代检索（依赖前序结果的问题）。代价：延迟/成本不可控（可能查 1 轮也可能 8 轮）、可观测性差、可能死循环。判据：**问题需要多步推理/依赖前序/动态选工具→上；简单事实问答→别碰**（杀鸡用牛刀引入不确定性）。在 Claude 上是工具循环（检索做成 tool，tool runner 跑循环），完整 agent 设计见第 12 章。
- 错误陷阱：说"agentic RAG 更智能所以都该用"（无脑上 = 给简单场景引入不确定性、不可控延迟）；把多跳处理讲成"非分解即 agent"的二选一（漏了中间那档"固定循环迭代检索"——链路固定的依赖多跳用固定循环就够，不必上 agent）。

**Q9：一个"有先后依赖的多跳问题"（如"我最大供应商的母公司在哪国注册"），有哪几种解法？怎么选？**
- 答题框架：先点出它**不能并行拆**（第二步 query 依赖第一步检索结果），所以 query 分解治不了。给出**三档由轻到重**的解法：**(1) 固定循环迭代检索**——链路结构已知固定时，写死一个循环，每轮把上一轮抽到的实体填进下一个 query 模板，直到停止条件；延迟/成本可预测。**(2) agentic 检索**——链路不定、下一步查什么/查没查够要模型临场判断时，把控制权交给 agent 在循环里自主决策；最灵活但延迟/成本不可控。判据一句话：**链路固定→固定循环；链路要动态决策→agentic。** 加分：明确"不是所有多跳都得上 agent"，很多场景固定循环更稳更省。
- 错误陷阱：一听"多跳"就直接上 agent（漏了固定循环这档，白白引入不可控延迟和死循环风险）；把多跳和 GraphRAG 的关系球（多跳是"沿检索结果链多查几轮"，GraphRAG 的关系遍历是"在预建好的实体图上走"——一个查文本、一个走图，机制不同）。

### 二、实战项目（P2 增强：给知识库加 query 改写 + rerank + 权限过滤）

在第 8 章 P2（端到端知识库问答）的基础上做三处增强，**每一步都用第 11 章的评测集量化前后收益**（这是本项目的灵魂——不许"加了感觉变好了"，必须有数字）：

**增量 1：加 rerank（两阶段召回-精排）**
- 把原来"向量检索 top-5 直接喂"改成"向量检索 top-50 → rerank 精排 top-5 → 喂"。
- 接一个 rerank（闭源 API 或开源 cross-encoder 任选，注意中文场景选中文支持好的）。
- **验收标准**：(a) 召回阶段 k=50，精排后 n=5；(b) 在评测集上，rerank 前后的**答案相关性 / 忠实度指标有可量化提升**（用第 11 章指标）；(c) 实测并记录 rerank 增加的延迟，确认在预算内。

**增量 2：加 query 改写（multi-query 或 HyDE 二选一）**
- 实现 multi-query（LLM 把 query 改写成 3~4 个变体，并集召回去重）**或** HyDE（生成假答案再检索），跑评测对比哪个对你的数据更有效。
- **验收标准**：(a) 改写逻辑独立可测（能打印改写后的 query 变体）；(b) 评测集上**召回率有可量化提升**；(c) 记录额外的 LLM 调用成本和延迟；(d) **关键**：确认 multi-query 配了 rerank（否则候选变脏可能掉点）——做一组"multi-query 不加 rerank"的对照，验证推论二。

**增量 3：加权限/元数据过滤（多租户隔离）**
- 给每个 chunk 加元数据（至少 `tenant_id` + 一个密级/部门字段），检索时按当前用户**pre-filter**。
- **验收标准**：(a) 无权 chunk **不进候选集**（用一个跨租户测试用例验证：A 租户的 query 绝不召回 B 租户的文档）；(b) 实现一个"权限泄露测试"——构造无权用户问机密内容，确认检索阶段就过滤掉、模型根本看不到；(c) 确认用的是 pre-filter 而非 post-filter（post-filter 可能过滤后不够 n 个）。

> 项目心法：这三个增量合起来，就是把第 8 章那个"demo 级 naive RAG"升级成"生产级 advanced RAG"的最小集。**做完务必回看：每一步的评测数字证明了它的价值吗？** 如果某一步加了之后指标没动甚至变差，那是宝贵信号——要么实现有 bug，要么这一招不适合你的数据。**这正是推论一：每个组件都要用指标证明它治了某条失效，否则就该砍掉。**

### 三、设计题（开放式，考权衡判断）

**题目**：你为一家律所做合同审查 RAG。需求特点：(a) 文档是几千份长合同（每份几十~上百页），强依赖上下文（"本条款"指代前文）；(b) 既有"这份合同的违约责任是什么"的**局部问答**，也有"我们所有客户合同里，哪些涉及竞业限制条款"的**跨文档全局检索**；(c) 数据极度敏感，不同律师只能看自己经手的合同；(d) 答案必须**精确溯源到原文条款**（合规要求）。请设计这套 RAG 的架构，说明你在 chunking 策略、检索架构、权限隔离、上下文增强上的选择和权衡。

**考点**：能不能把本章多个旋钮**按需组合**，并对每个选择讲清权衡——
- chunking：合同强依赖上下文 → **small-to-big / 句子窗口**（小块检索精准，喂模型时带回条款上下文）+ **contextual retrieval**（给每个条款 chunk 加"出自哪份合同哪一章"的上下文，治指代和召回）；
- 检索架构：局部问答用**向量检索 + rerank**；"所有合同里哪些涉及竞业限制"是**全局/聚合检索**，向量检索抓瞎 → 考虑 **GraphRAG 或元数据标注 + 结构化检索**；用**路由**分流这两类问题；
- 权限：数据极敏感 + 按律师隔离 → **物理隔离或 namespace**（不能只靠共享索引过滤），且 **pre-filter** 铁律，无权合同绝不进候选；
- 溯源：合规要求精确到条款 → **句子级归因 / 原生 Citations**（第 8 章），别用模型自插标记的弱方案；
- 长上下文取舍：单份合同审查（几十页）可考虑**长上下文直接塞 + prompt caching**（同一份合同反复问，缓存红利大），但跨客户检索必须 RAG。
- 没有标准答案，看的是**判断的结构**——能不能识别"这里有局部和全局两类问题、得分治""这里敏感数据该物理隔离""这里强依赖上下文该 contextual + small-to-big"。

### 四、系统题（含量级估算）

**题目**：设计一个服务 1000 家企业客户（多租户）的 SaaS RAG 平台。假设：平均每家客户 50 万个 chunk（共 5 亿 chunk），embedding 1024 维（float32），峰值 100 QPS。请估算：(1) 向量存储量级；(2) 多租户隔离方案选型；(3) 在线检索的延迟预算分解；(4) 一个关键的成本权衡点。

**参考分析**（量级估算，数字按 2026 初，重在方法）：
- **存储**：5 亿 chunk × 1024 维 × 4 字节 ≈ **2 TB 裸向量**（还没算索引结构 HNSW 的额外开销，通常再 ×1.5~2，且元数据另算）。结论：单机塞不下，**必须分片**——按 tenant_id 分片天然合理（顺带做了隔离）。
- **隔离选型**：1000 个租户、单租户 50 万 chunk（不算小）→ **namespace/分片隔离**（不是物理独立部署 1000 套，那运维爆炸；也不是纯共享索引过滤，租户数据量够大值得隔开）。按 tenant_id 分片同时解决"塞不下"和"隔离"。
- **延迟预算分解**（在线路径，单请求）：query embedding（~10ms）+ 向量粗召回 top-50（ANN，~20~50ms）+ rerank top-50→5（cross-encoder，~50~200ms，是延迟大头）+ 生成（流式，首 token ~数百 ms，受输入材料量影响）。**rerank 是检索侧延迟大头，要权衡 k 大小**；生成走 streaming 让用户先看到首 token。
- **关键成本权衡**：(a) **要不要给所有客户上 contextual retrieval**——5 亿 chunk 每个调一次 LLM 建上下文是巨额一次性离线成本，但能用 prompt caching（同文档复用）大砍；权衡"检索质量提升"vs"建库成本"，可能只对召回率差的客户开。(b) **rerank 的 k**——k=50 比 k=20 召回好但 rerank 慢一倍、贵一倍，按延迟预算和质量评测定。(c) **100 QPS 下 rerank 用闭源 API 还是自部署**——高 QPS 下 API 调用费可能超过自部署，要算盈亏平衡点（呼应第 8 章"先 profile 再决定"）。

**暗坑汇总**：(a) **pre-filter 不是 post-filter**——安全场景过滤必须在召回前发生，召回后过滤既不安全（无权数据已进候选）又可能导致结果不够 n 个；(b) **rerank 只能在候选集内重排**——k 太小漏掉的文档 rerank 永远救不回，所以 k 要够大；(c) **纵深防御**——别假设 vector_store 的过滤一定对，安全相关的过滤要在应用层再断言一遍，过滤条件写错时能兜底；(d) **small-to-big 的父块去重**——多个子块共享父块时，不去重会让同一父块重复占满 n，挤掉其他材料。
