> 这章是**结课章**，排在最后不是因为它"最不重要"，恰恰相反——它是整门课的**收口与对账**。前 23 章你学的是"当前主流怎么做"，这一章要回答两个更狠的问题：(1) 这些做法里，哪些是**物理规律**（五年后还在），哪些是**当前版图**（一年后就过期）；(2) 当模型能力以每季度一个台阶往上跳时，**你这个"AI 应用工程师"的护城河到底在哪、往哪挪**。
>
> 替你解决的问题：让你停止两种极端的焦虑。一种是"模型这么强了，应用层是不是不用写了，我是不是要失业"；另一种是"模型再强也得有人调 API，我稳得很"。**两种都错。** 正确的姿势是拿一把尺子，去量"哪些胶水代码会被模型能力吃掉、哪些是真护城河"——这把尺子，就是这一章要交到你手里的东西。
>
> 锚点类比贯穿全章：**这一轮 AI 之于软件工程 = 当年云计算之于运维**。云没有消灭运维，它消灭的是"手动装机架线"这层；活下来并涨薪的，是把技能往上挪到"SRE / 平台 / 容量与成本治理"的那批人。被吃掉的从来不是"运维"这个职业，是运维里**那部分可被自动化的机械劳动**。这一章就是教你预判：你手上哪部分活是"装机架线"，哪部分是"SRE"。

读法提示：这章和前面章节的体感不同——前面讲"机制"，这章讲"判断"。但风格不变：每个趋势后面紧跟**工程推论**（编号列出，每条都能独立成一道面试/战略题）。第一遍至少把"test-time compute 经济学对照表""模型吃掉应用层的边界判断表""护城河四象限"和"职业技能迁移表"啃下来。

时效标注：这一整章的**版图部分**（test-time compute 的具体价格、agent 协议的标准之争谁赢、on-device 模型的具体参数）按 **2026 初**写，**这是全课时效性最强的一章，名单和结论会以季度为单位翻新，请务必学方法论和判断框架,而不是背我的结论**；但其中的**物理规律**（推理时算力是第二条 scaling 曲线、护城河在数据/评测/分发而非模型、能力边界的判断方法）讲透到骨子里，这些五年不变。

> 一句话先定调：**不要赌"模型会不会变强"——它一定会，这是确定性事件。要赌的是"当模型变强,价值会从哪里流走、又会在哪里淤积"。看懂资金和价值的流向,比看懂模型本身重要得多。**

## 一、背景：为什么现在必须谈未来 3-5 年

### 1.1 一个让所有"经验"贬值的事实

你做后端十年，很多经验是**复利**的——你五年前学的 TCP、数据库索引、分布式一致性，今天还在用，而且越用越值钱。这是因为底层技术栈**稳定**。

AI 应用这一层不是这样。这一层的底座（模型能力）每季度跳一级，**这意味着你今天写的某些代码,它的"保质期"可能只有 6-12 个月**。一个最具体的例子（见第 16 章 Code Agent）：

- **2023 年**，你要让 LLM 生成结构化 JSON，得手写一堆正则去解析它"大概是 JSON 但带了 markdown 包裹"的输出，还要写重试逻辑。这套代码是**真本事**，能写对的人不多。
- **2024 年**，厂商上了 `output_config.format`（json_schema，见第 4 章）+ `strict:true`，你那套正则解析代码**当场作废**——模型在解码层就保证了 schema 合规。
- 你那几百行精心调试的"JSON 修复器"，被一个 API 参数**吃掉了**。

> 新手第一误区：把"我现在写得很辛苦的代码"等同于"有护城河的代码"。**写起来辛苦 ≠ 有壁垒。** 很多今天辛苦的活，辛苦恰恰是因为模型还没补上这块能力——而它一定会补。**当前的痛点,往往就是下一版模型的 feature。** 你越是觉得"这块脏活只有我能搞定"，越要警惕：这可能正是下一个被吃掉的目标。

### 1.2 为什么是"3-5 年"这个尺度

- **短于 1 年**：纯版图，没法规划，今天的结论下周可能就被一个新模型推翻。谈了也白谈。
- **长于 5 年**：纯科幻，AGI、超级智能这些，对你"下个季度该把技能往哪练"没有指导意义。
- **3-5 年**：刚好是**一个职业决策的有效期**——你现在选择深耕的方向（是去做 agent infra？还是做领域 RAG？还是去管 LLMOps 平台？），3-5 年后兑现。这一章就是给这个尺度的决策提供判断框架。

> 工程推论 1（本章总纲）：**面对一个 AI 能力,先问"它是物理规律还是当前版图"。物理规律值得 all-in 学透,当前版图只值得"知道现状 + 持续跟踪"。** 把学习预算的 80% 砸在物理规律上,20% 用来跟版图。这个配比错了——比如花大量时间背各家模型的最新参数——你会很忙,但不增值。

## 二、Test-Time Compute：第二条 Scaling 曲线的经济学

这是过去两年 AI 领域最大的范式变化，也是你回收第 1 章那句"自回归"埋下的伏笔的地方。**理解它,是理解后面一切（为什么 agent 能跑通、为什么推理变贵、为什么 effort 档位取代了 temperature）的钥匙。**

### 2.1 背景：第一条 scaling 曲线撞墙了

2020-2023 年，AI 进步的主旋律是 **train-time scaling**——模型越大、训练数据越多、训练算力越多，能力越强（Scaling Laws）。整个行业的信仰是"把参数和数据堆上去就行"。

但到 2024 年，这条曲线开始显现**边际递减**：

- 高质量训练数据**接近枯竭**（互联网上的优质文本就那么多，已经基本喂完了）。
- 参数从百亿到千亿到万亿，每翻一倍带来的能力提升越来越小，而**成本翻倍**。
- 训练一个 frontier 模型的成本进入"单次上亿美元"量级，**没几家公司玩得起**。

### 2.2 演进：发现了第二个旋钮——在推理时"多想一会儿"

转折点是一个朴素到反直觉的发现（回收第 1 章自回归）：

> **既然模型是一个 token 一个 token 往前续的,那让它在给出最终答案前,先续写一大段"思考过程"(reasoning tokens),它的答案质量会显著提升。**

这就是 **test-time compute（推理时算力，也叫 inference-time scaling）**。它的革命性在于：**能力的提升,不再只能靠"重新训练一个更大的模型",而是可以靠"在回答时多花算力多想"来换取。** OpenAI 的 o 系列、DeepSeek R1（见下文，这是 2025 初的分水岭）、Claude 的 adaptive thinking（见第 2 章），全是这条路线。

为什么"多想"有用？回到第 1 章推论二：**reasoning tokens 是模型自己给自己布置的上下文。** 它写出"让我先分析一下边界条件……"这段中间步骤后，这段文字成为后续 token 的**条件（conditioning）**，把延续引导到"经过推理的、更可能正确"的轨迹上。本质上，**模型在用『生成中间步骤』给自己搭脚手架,把一个它一步答不对的难题,拆成它一步步能答对的小题。**

> 工程推论 2：**test-time compute 是"用钱（算力）换正确率"的旋钮,而且这个旋钮是连续可调的。** 这是它和 train-time 的关键区别——训练好的模型能力是固定的，但你可以**在每次请求时决定花多少推理算力**。这就是为什么 Anthropic 用 `effort` 档位（low/medium/high/xhigh/max，见第 2 章）取代了 `budget_tokens`——它把"花多少算力想"做成了一个**业务侧可以按请求动态决策**的参数。

### 2.3 现状：两条 scaling 曲线并存，经济学完全不同

现在的格局是**两条曲线叠加**。你作为应用工程师，必须搞懂它们的经济学差异，因为**这直接决定你的成本结构和架构选择**：

| 维度 | 第一条：Train-Time Scaling | 第二条：Test-Time Compute |
|---|---|---|
| 算力花在哪 | 训练阶段（一次性） | **推理阶段（每次请求都花）** |
| 谁付钱 | 模型厂商（已摊进 API 价格） | **你，按请求付，且随难度浮动** |
| 成本特性 | 固定成本，已沉没 | **可变成本,你能控制** |
| 提升能力的方式 | 等厂商发更大的模型 | **自己在请求时调高 effort** |
| 对你的含义 | 被动接受 | **主动权在你手里——也意味着你能把成本搞爆炸** |
| 延迟影响 | 无额外 | **巨大**——多想 = 多生成 reasoning token = TTFT 后 decode 段暴涨（见第 1 章推论三） |

这张表里藏着这门课最重要的一个**成本陷阱**：

> 坑点提示（生产事故级）：**reasoning tokens 是要计费的,而且按输出价计费（输出价 ≈ 输入价 5 倍,见第 1 章）。** 你把 effort 从 medium 调到 max，模型可能从想 500 token 变成想 8000 token，**这 8000 token 全按输出价烧钱,而且用户还要多等十几秒**。一个没经验的团队，把所有请求一刀切设成 max effort，**账单可能翻 5-10 倍而 95% 的请求根本不需要那么想**。

### 2.4 落地：大厂怎么用 test-time compute——分级思考是核心工程

成熟团队的做法不是"全开 max"，而是**按任务难度路由思考预算**，这是 test-time compute 时代的核心成本工程：

1. **简单任务（分类、抽取、格式化）→ low effort 或干脆关思考**。这类任务"多想"不仅不提升，还可能让模型**过度思考把简单问题想复杂**（这是真实现象，叫 overthinking）。
2. **中等任务（一般问答、常规代码）→ medium**。
3. **难任务（复杂推理、多步规划、疑难 debug）→ high / xhigh**，只在这里花大钱。
4. **极难且高价值（关键决策、数学证明级）→ max**，谨慎使用，最好配人审。

国内厂商也全面跟进了这条路线（按 2026 初版图，名单会变）：**DeepSeek R1** 是 2025 年初的标志性事件——它用开源模型证明了"推理能力可以追平闭源 frontier"，且把 reasoning 模型的训练方法（强化学习驱动的 reasoning）公开了，是开源推理的**分水岭**；**Qwen（阿里）**、**GLM（智谱）**、**Kimi（月之暗面）**、**豆包（字节）** 都推出了带 thinking 模式的版本。这条曲线已经是行业共识,不是某一家的独门绝技。

> 工程推论 3：**"分级思考路由"是 test-time compute 时代的核心成本工程,地位等同于 RAG 时代的 chunking 调优。** 面试时被问"你怎么控制推理成本"，能答出"按任务难度动态分配 effort 档位 + 监控 reasoning token 消耗 + 识别 overthinking"，直接甩开"我把 temperature 调低"那种答案。这套路由逻辑，本质上是把第 20 章 LLM Gateway 的"模型路由"下沉到了"思考深度路由"——同一个模型，不同请求给不同的思考预算。

但只答"分级 effort"是**初级答案**。`effort` 控的是"想多深",而推理成本的总账还有另外两个互相独立的旋钮——**一个砍输入侧的重复花费,一个砍非实时请求的单价**。三个旋钮作用在账单的不同位置，能叠加。一个 senior 会把这三个一起端出来：

| 成本旋钮 | 砍的是什么 | 量级（按 2026 初版图,以官方为准） | 适用场景 |
|---|---|---|---|
| **分级 effort 路由** | reasoning token 的**量**（想多深） | 简单任务关思考可省绝大部分思考开销 | 所有请求,按难度分档 |
| **prompt caching** | **输入侧**重复前缀的重复计费 | 缓存命中读约 0.1×、写约 1.25×（前缀匹配,见第 4 章） | system prompt / few-shot / 长上下文这类**跨请求不变的前缀** |
| **Batch API** | **单价**（异步换便宜） | 约 5 折,多数 1 小时内完成、上限 24 小时 | 离线评测、批量抽取这类**不要求实时**的请求 |

> 工程推论 3b（把成本账算全）：**"控制推理成本"不是单点是组合拳——`effort` 砍思考量、caching 砍重复输入、batch 砍非实时单价。** 三者作用在账单不同位置、可叠加。被问到时，能从"想多深 + 输入复用 + 实时性"三个维度同时下手，才是把成本当系统工程治理；只会调 effort 的人，漏掉了输入侧和单价这两块往往更大的水分。**注意一个反直觉陷阱：caching 是"前缀匹配"——前缀里任何一个字节变了（比如 system prompt 里塞了时间戳/UUID），缓存当场击穿且不报错，你以为在省钱其实全价重算**（见第 4 章的缓存命中验证）。

### 2.5 面试视角：test-time compute 高频考点

- **考点 1：你能不能讲清"两条 scaling 曲线"的本质区别。** 区分点：train-time 是固定成本被厂商摊进价格，test-time 是**可变成本由你按请求付且可控**。说不清这个，说明你只是"听过 o1/R1 这些名字"。
- **考点 2：为什么 reasoning 模型移除了 temperature/top_p（见第 2 章）。** 答题框架：因为行为控制的主通道从"采样超参"迁移到了"自然语言指令 + 思考预算"——你不再通过调采样分布来控制模型，而是通过给它更多思考算力 + 更清晰的指令。这代表一个**更深的趋势**：模型的"理解力"强到不需要你用底层旋钮微调，你直接用自然语言说清楚就行。
- **考点 3（拉开差距）：test-time compute 对 agent 意味着什么。** 答：它是 agent 能跑通的**经济学前提**。agent 一次任务要跑几十步（见第 12 章），每步都可能需要推理——test-time compute 让"每步多想一会儿"成为可买的商品，agent 的可靠性因此可以**用钱买**。但反过来,agent × test-time compute = 成本的乘法爆炸,这是 agent 落地最大的成本黑洞。

> 心法：**train-time scaling 是厂商的军备竞赛,你看戏；test-time compute 是你每天要做的成本决策,你下场。** 把"该让模型想多深"当成一个像"该缓存多久"一样的日常工程旋钮,你就入门了这个时代。

## 三、Agent-Native 架构与协议层之争

### 3.1 趋势：架构正在从"以模型为中心"转向"以 agent 为中心"

回收第 12-16 章的 agent 主线。过去两年发生的结构性变化是：**系统的设计原点,从"怎么调好一次模型"变成了"怎么编排好一群会自主行动的 agent"。**

- **旧范式（model-centric）**：一个请求进来，拼 prompt，调一次模型，拿结果返回。架构图的中心是"模型调用"。
- **新范式（agent-native）**：系统由一堆 agent 组成，每个 agent 有自己的角色、工具、记忆，它们之间会**互相调用、互相委托任务**。架构图的中心是"agent 之间的协作拓扑"（见第 15 章 multi-agent）。

这个转变带来一个全新的工程问题：**agent 怎么和 agent、agent 怎么和工具,标准化地对话?** 这就是协议层之争的由来。

### 3.2 协议层：两类协议，解决两个不同的问题

这是 2025-2026 年最热的基础设施战场。**新手最大的误区是把这两类协议搞混——它们解决的根本不是同一个问题**（回收第 5/6 章工具与协议）：

| 协议 | 全称 | 解决什么问题 | 类比（用你的后端直觉） | 成熟度（按 2026 初） |
|---|---|---|---|---|
| **MCP** | Model Context Protocol | **agent ↔ 工具/数据源** 怎么标准化连接 | 像 **ODBC/JDBC**——给"模型接外部能力"定一个统一插头 | 已是事实标准，生态最大（见第 6 章） |
| **A2A** | Agent-to-Agent | **agent ↔ agent** 怎么互相发现、委托、协作 | 像 **gRPC/服务发现**——给"agent 之间互相调用"定协议 | 早期，标准未定，多家在争 |

**MCP（见第 6 章）的核心价值**：它把"模型怎么接工具/资源/prompt"标准化了。在 MCP 之前，每接一个工具你都要写一套适配；有了 MCP，工具方实现一次 MCP server，所有支持 MCP 的 host（Claude、各种 IDE）都能直接用。Anthropic API 甚至支持 `mcp_servers` 参数**直连远程 MCP server**（见第 6 章）。**它解决的是"N 个模型 × M 个工具"的 N×M 适配爆炸,把它降成 N+M。** 这是 MCP 能快速成为事实标准的根本原因——它解决的是一个真实的、所有人都疼的工程痛点。

**A2A 解决的是另一个问题**：当你有一堆 agent（可能是不同公司、不同框架做的），它们怎么互相发现对方的能力、怎么把一个子任务委托出去、怎么传递中间状态。这是个**更难、更早期**的问题，因为它涉及跨信任域的协作（你的 agent 调别人公司的 agent，怎么鉴权、怎么计费、怎么防对方的 agent 是恶意的——这直接接上了第 21 章的安全模型）。

> 工程推论 4：**MCP 已经值得你现在就上手（它解决真问题、生态成熟）；A2A 现在只值得"知道它存在并跟踪",别赌任何一个具体标准会赢。** 判断一个协议该不该投入的标准很简单：**它解决的 N×M 适配痛是不是真的、现在就疼。** MCP 的痛是真的现在就疼（每个人都在重复写工具适配），所以它赢了；A2A 的痛在多数团队还没到那个规模（你还没多到需要跨公司 agent 协作），所以观望。

> 坑点提示：别一看到"协议"就觉得高大上要全上。**协议是用来消除重复适配的,不是用来炫技的。** 你团队就 3 个内部工具、全自己写的，硬上 MCP 可能是过度工程——直接函数调用更简单（见第 5 章）。**当你的工具开始需要被"团队外的人/系统"复用时,才是上 MCP 的信号。** 判断标准永远是"适配成本有没有真的爆炸"，不是"别人都在用"。

### 3.3 落地：agent-native 架构对你的系统意味着什么

如果你在设计 P5 那样的 agent 平台（结课项目，见下文），agent-native 意味着几个具体的架构决策：

1. **工具市场（tool marketplace）要按 MCP 标准建**——这样工具能被平台上所有 agent 复用，而不是每个 agent 团队各写一套（这正是 P5 验收标准里"工具市场"那一层的现代答案）。
2. **agent 之间的通信要预留协议层**——即使现在用内部 RPC，也要把"agent 间消息格式"抽象出来，为未来接 A2A 类标准留口子（呼应 MEMORY 里"design for evolution，pull 优于 push"）。
3. **可观测性要能跨 agent 追踪**（见第 18 章）——一个任务流经 5 个 agent，你得能把这条链路串起来 trace，否则出事根本没法定位。这是 agent-native 架构对第 18 章可观测性提出的新要求：trace 的单位从"一次模型调用"变成"一条跨 agent 的任务链"。

> 心法：**model-centric 时代你设计的是"调用";agent-native 时代你设计的是"协作拓扑 + 协议"。** 前者是写函数，后者是定接口、画组织架构图——这正是从高级工程师往架构师（P9）走的那一步：**你的产出从"代码"变成"让一群自主体能协作的规则"。**

## 四、On-Device、小模型与多模态原生

这两个趋势我放一起讲，因为它们共同回答一个问题：**模型会往哪些新的"形态"和"场景"扩散。**

### 4.1 On-Device 与小模型：不是所有任务都该上云端大模型

**趋势**：模型在**两端同时进化**——一端是 frontier 模型越来越强（往云端、往超大走），另一端是小模型（几 B 参数）越来越能打，能跑在手机、PC、边缘设备上。后一端常被忽略，但**对应用工程师的架构决策影响极大**。

为什么小模型/on-device 重要（每条都是一个真实的生产驱动力）：

| 驱动力 | 具体含义 | 典型场景 |
|---|---|---|
| **延迟** | 本地推理无网络往返，TTFT 极低 | 输入法补全、实时纠错、IDE 行内补全 |
| **隐私** | 数据不出设备，合规天然过关 | 医疗、金融、涉密数据处理 |
| **成本** | 本地推理边际成本≈0（不按 token 付费） | 高频、海量、低价值的请求 |
| **离线** | 无网络也能用 | 车机、工业现场、弱网环境 |

> 工程推论 5：**未来的生产架构大概率是"模型梯队 + 本地/云端混合",而不是"一个大模型包打天下"。** 这直接复用了第 20 章 LLM Gateway 的路由思想，但多了一个维度——**不只是"路由到哪个云端模型",还要路由"上不上云"**。简单高频的任务（分类、补全、敏感数据预处理）走本地小模型；复杂低频的任务（深度推理、复杂 agent）走云端 frontier。**会算"这个任务该用多大的模型、该在哪跑"这笔账,是这个时代成本工程的核心能力。**

落地判断框架（这是你该背的方法论，不是具体模型名）——一个任务该下沉到本地小模型，当且仅当：

1. **任务足够窄**（分类、抽取、补全这种，不需要广博知识和复杂推理）——窄任务小模型够用，宽任务必须大模型。
2. **量足够大 / 频次足够高**（值得为它优化，省下的 token 费可观）。
3. **对延迟或隐私有硬要求**（本地的核心优势就这两个，没有这两个需求就别折腾）。

反之，**宽任务、低频、可容忍延迟、数据不敏感的，老老实实上云端**——别为了"本地"而本地，那是新手为了技术炫技牺牲产品的典型错误。

> 国内语境（按 2026 初）：端侧小模型在国内是**热门战场**——手机厂商（把模型塞进系统做 AI 助手）、车厂（车机离线助手）都在推端侧部署。Qwen 系列有专门的小尺寸开源版本适配端侧。**但别神化端侧**——它能干的活窄得很，宽任务和复杂 agent 仍然必须云端大模型。端侧解决的是"特定窄场景的延迟/隐私/成本",不是"替代云端大模型"。

### 4.2 多模态原生：从"文本模型 + 外接 OCR/ASR"到"模型直接吃图/音/视频"

**演进**：早期处理多模态是**拼装**的——文本模型外面接一个 OCR 把图转文字、接一个 ASR 把语音转文字，再把文本喂给 LLM。这套 pipeline 脆、信息有损（见第 10 章文档智能里讲的，OCR 会丢版面信息）。

**现状**：frontier 模型越来越**多模态原生（multimodal-native）**——图像、音频直接作为 token 进模型，不经过"先转成文字"这一步。这带来质变：模型能理解图里的**版面、图表、空间关系**，而不只是图里的文字。一张财报截图，原生多模态模型能"看懂"表格结构和趋势线，而 OCR + 文本模型只能拿到一堆抽出来的数字、丢了结构。

> 工程推论 6：**多模态原生会吃掉一整类"预处理胶水"——你今天为了把图/音转成文本而写的 OCR/ASR pipeline,可能在 2-3 年内被"直接把原始模态喂给模型"取代。** 这是第 1.1 节"代码保质期"的又一个活例子。但注意：**被吃掉的是"格式转换"那层胶水,不是"领域理解"那层价值**——你对"这张财报该提取哪些指标、怎么判断异常"的领域知识，模型吃不掉（这一条直接引出第五节的护城河判断）。

> 面试加分点：被问"多模态原生和 OCR+LLM 有什么区别"，错误答案是"多模态就是能处理图片"。正确答案要点出**信息保真度**——原生多模态保留了版面/结构/空间信息，而"OCR→文本"是有损压缩，丢了正是文档智能最值钱的那部分（结构）。能把这层说清楚，说明你真懂第 10 章。

## 五、核心判断：模型会吃掉哪些应用层,哪些是真护城河

**这是全章、甚至全课的题眼。** 前面所有趋势汇聚到这一个问题：**当模型能力以季度为单位跃迁,你写的代码里,哪些会被吃掉,哪些是壁垒?** 回收第 16 章 Code Agent 那句弦外之音——它早就预告了这一节。

### 5.1 先建立判断的第一性原理

模型会吃掉的，是**"通用的、可被语言能力覆盖的"那部分逻辑**；模型吃不掉的，是**"私有的、需要外部真相/反馈/分发的"那部分**。一句话：

> **凡是"给定足够上下文,一个足够聪明的通用模型就能做对"的事,迟早被吃掉;凡是"再聪明的模型,没有你的私有资产（数据/评测/领域/渠道）也做不对"的事,就是护城河。**

### 5.2 会被吃掉的（别在这里建立长期身份认同）

按"被吃掉的概率"从高到低排，每一类都给你判断信号：

| 会被吃掉的层 | 为什么 | 已经在发生的信号 |
|---|---|---|
| **输出格式修复 / 解析胶水** | 结构化输出已在解码层解决（见第 4 章） | json_schema + strict 已让 JSON 修复器作废 |
| **简单 prompt 模板拼接** | 模型理解力够强，不需要精雕细琢的咒语 | reasoning 模型移除 temperature，"调参玄学"在退场（见第 2 章） |
| **多模态预处理 pipeline** | 多模态原生直接吃原始模态 | OCR/ASR 中间层在被原生能力侵蚀（见 4.2） |
| **简单 CRUD + 表单后台** | Code Agent 一句话生成+执行（见第 16 章） | Claude Code / Cursor 已能端到端生成这类应用 |
| **基础 agent 编排脚手架** | 框架和 SDK 把 agent loop 标准化了 | SDK 已提供 tool runner 自动跑工具循环（见第 5 章） |

> 工程推论 7：**判断一段代码会不会被吃掉,问一句话——"如果模型变得无限聪明、上下文无限长,这段代码还需要存在吗?"** 答"不需要"的（比如 JSON 修复、prompt 咒语调优），就是被吃掉的候选，别在上面建立你的长期职业身份。答"还需要"的（比如鉴权、计费、私有数据接入、合规审计），才是你该深耕的。**这一问,是你这辈子的"职业避坑器"——每隔半年拿它扫一遍你手上的活。**

### 5.3 真护城河：四类模型吃不掉的资产

这是你该 all-in 的地方。记住这四个词：**评测、数据、领域、分发。** 它们的共同点是——**都是"私有的、需要长期积累的、模型再强也替代不了的"外部资产。** 用一个四象限记它（横轴：偏技术 vs 偏业务；纵轴：靠积累 vs 靠卡位）：

| 护城河 | 为什么模型吃不掉 | 在本课对应 | 你怎么积累它 |
|---|---|---|---|
| **评测（Eval）** | 模型能力再强，**"在你的业务场景下到底好不好"只有你的私有评测集知道**。评测是你定义"什么叫对"的权力——模型不知道你的标准 | 第 11、17 章；P4 验收的离线 eval 集 | 沉淀带标注的领域评测集，这是越积越值钱的复利资产 |
| **数据** | 模型的通用知识是公开的，**你的私有数据（用户行为、领域文档、专有标注）是它没见过的** | 第 7-10 章 RAG；私有知识库 | 把业务数据资产化、可检索化、持续更新 |
| **领域（Domain）** | "这个医疗/金融/法律问题该怎么判断对错、有哪些坑"是**深领域知识**，需要和领域专家共建，不在公开语料里 | 全课贯穿；P2/P3 的领域适配 | 和业务/专家深度绑定，把领域 know-how 编码进系统 |
| **分发（Distribution）** | 模型是别人的，但**用户在谁手里、入口在谁手里**是你的。能力会平权，但渠道和用户关系不会自动平权 | 产品/业务层 | 占住用户入口、场景入口、决策入口 |

> 工程推论 8（最重要的一条职业判断）：**模型能力会"平权"——大家用的底座越来越接近同一批 frontier 模型;能拉开差距的,只剩这四类私有资产。** 这意味着：**你的简历价值,不该建立在"我会调最新的模型"(这个全行业都会、且明天就过时),而该建立在"我搭过/治理过 X 类业务的评测体系、数据管道、领域系统"(这个有积累、有壁垒)。** 这一条,直接决定你接下来三年该往哪使劲。

> 坑点提示（一个反直觉但关键的判断）：**护城河不在"模型层",甚至不在"prompt 层",而在"模型周围那圈工程和资产"。** 很多人把精力全砸在"调出更好的 prompt"上——这恰恰是最容易被吃掉的一层（reasoning 模型已经在弱化 prompt 调优的价值）。**真正的壁垒是 prompt 之外的东西:你喂给模型的私有数据质量、你验证模型输出的评测体系、你把它接进业务的领域逻辑、你触达用户的渠道。** 把这个想反了,你会在最容易贬值的地方拼命,在最有壁垒的地方偷懒。

### 5.4 一个完整的边界判断案例（把方法论跑一遍）

假设你在做一个"合同审查助手"。用 5.1 的第一性原理逐层扫描——**这就是你以后做任何 AI 产品都该跑一遍的"护城河审计"**：

- **"把合同 PDF 转成文本"这层** → 多模态原生会吃掉（见 4.2）。**别在这建壁垒。**
- **"让模型读合同并指出风险条款"这层** → 通用能力，模型会越来越强。**这层你提供不了差异化。**
- **"判断这条款在中国法 / 这个行业里到底有没有风险、风险多大"** → **领域护城河**，需要和法务专家共建知识库和判断标准。**深耕这里。**
- **"用 200 份标注过的真实合同评测助手准确率，每次模型升级跑回归"** → **评测护城河**，这是你证明"我的产品比裸调模型强"的唯一凭据。**这是你的命根子。**
- **"接进企业的合同管理系统、占住法务的工作入口"** → **分发护城河**。**这决定你能不能活下去。**

结论：**这个产品里,80% 的"AI 含量"(读文本、理解语义)是会被吃掉的通用能力,你真正的价值和壁垒,在领域知识库 + 评测集 + 业务入口这三块。** 把团队资源砸在这三块，而不是砸在"调 prompt 让它读得更准"上——后者是模型厂商的活，他们会替你越做越好。

> 心法：**模型吃掉的是"能力",留给你的是"判断力、私有资产和分发"。** 这一轮 AI 不是消灭工程师,是把工程师的价值,从"会实现"逼向"会判断该实现什么、能拿出别人没有的数据和评测、能把它送到用户面前"。**会实现的人,正在被模型追平;会判断 + 有资产 + 控分发的人,正在变得更稀缺。**

## 六、对 AI 应用工程师职业的含义 + 结课复盘

### 6.1 职业:你的技能要往哪挪

把第五节的判断落到你个人身上。用开篇"云计算之于运维"的类比收口——**技能迁移是有方向的，跟对方向涨薪，跟错方向被吃掉**：

| 正在贬值的技能（别当身份认同） | 正在增值的技能（往这挪） |
|---|---|
| 会调某个最新模型的 API | 会**为业务设计评测体系**、定义"什么叫对"（第 17 章） |
| 会写精巧的 prompt 咒语 | 会**做成本/能力/延迟的架构权衡与路由**（第 19、20 章） |
| 会拼 agent 编排脚手架 | 会**设计 agent 平台、定协议、画协作拓扑**（P5、第 15 章） |
| 会接某个向量库 | 会**治理数据资产、建可演进的知识管道**（第 7-10 章） |
| 知道"幻觉"这个词 | 会**做 grounding + 安全 + 可观测的可靠性工程**（第 1、18、21 章） |

> 工程推论 9（送给你的职业指南针）：**所有"会被吃掉的技能"都有一个共性——它们是"实现层"的;所有"会增值的技能"都有一个共性——它们是"判断层 + 资产层 + 治理层"的。** 这正好对应 P9 那个公式：**知识 × 战绩 × 判断（见第 00 章）。** 模型吃掉的是"知识"里可被通用化的部分，吃不掉的是"战绩"（你积累的私有资产）和"判断"（你的架构与取舍能力）。**你越往判断和治理走,越安全、越值钱;越在实现层卷,越容易被下一版模型抹平。**

> 一个必须破除的误区：**"模型这么强了,工程师要失业了"——错。失业的是"只会实现层、且实现的恰好是模型能吃掉的那部分"的人。** 历史上每一次能力平权（编译器吃掉汇编、云吃掉装机、框架吃掉样板代码），都不是消灭工程师，而是**把工程师从低层劳动里解放出来,逼他们去解决更高层的问题**。这一轮一样：**你被解放出来的精力,该投到"评测、数据、领域、分发"这四个模型够不着的地方。** 谁先完成这个迁移，谁就在下一轮拿到位置。

### 6.2 如何持续学习:学物理,不学版图

这门课从第 1 章到这里，反复在做一件事——**把"物理规律"和"当前版图"分开**。这就是你出师后该一直用的学习方法（回收第 00 章的出题力训练法）：

**学物理（值得 all-in,五年不过时）**：
- 自回归 / token / 上下文窗口 / 幻觉机制（第 1 章）——一切的地基。
- prompt 是概率空间定位、可靠性必须外置（第 1 章）——所有架构的认知前提。
- RAG 的本质是"外接可更新真相源"、检索-生成的张力（第 7-11 章）。
- agent 是"感知-决策-行动闭环 + 停机问题"、test-time compute 是第二条 scaling（第 12 章、本章）。
- 评测之于 AI = error budget 之于 SRE（第 17 章）、安全的信任边界（第 21 章）。
- **护城河在评测/数据/领域/分发,不在模型（本章）。**

**跟版图（知道现状即可,别背,会过期）**：
- 各家模型名单、参数、价格（按季度翻新）。
- 具体工具链（向量库、框架、网关、tracing 产品）选型。
- 协议标准之争谁赢（MCP/A2A 的最终格局）。
- on-device 模型的具体能力边界。

> 持续学习的操作方法（直接抄）：**建一个"物理 vs 版图"的二分笔记。** 每读到一个新东西，先问"这是物理还是版图"。是物理，深挖到第一性原理，建立直觉；是版图，记一句"现状是 X，我在跟踪"，定期刷新就行。**这个二分动作,就是你出师后不被信息洪流淹没、且不在过期知识上浪费时间的护身符。** 一年后你会发现：当别人还在追"最新模型又出了什么"，你已经在用不变的物理规律,快速判断每个新东西"值不值得学、该学多深"。

### 6.3 结课复盘:把 P0-P5 串成你的个人技术叙事

这是本课的最后一个项目动作，也是你把"学过"变成"战绩"的临门一脚（呼应第 00 章"不写代码=没学"和 MEMORY 里"用户成长方向：训练出题力 + 签收力"）。

**把 P0→P5 这条链,重述成一个能在面试里讲 20 分钟的技术叙事。** 它不是"我做了 6 个项目"的流水账，而是一条**有内在逻辑的能力进阶线**——这条线本身就是你区别于"调过 API 的人"的证据：

| 项目 | 一句话战绩（带数字、带判断，照这个模板写你自己的） | 它证明你掌握了什么"判断层"能力 |
|---|---|---|

### A. 高频面试题（附答题框架与陷阱）

**1. 什么是 test-time compute?它和 train-time scaling 的本质区别是什么?**
- 答题框架：test-time compute = 推理时让模型生成 reasoning tokens"多想一会儿"换正确率，是第二条 scaling 曲线。本质区别——train-time 是**固定成本被厂商摊进 API 价格**，test-time 是**可变成本由你按请求付、且可控**（用 effort 档位调）。
- 陷阱：只说"就是让模型多想"=没说到经济学。必须点出"**可变成本 + 你能控制 + 按输出价计费**"这层，否则只是听过名词。
- 加分：补一句"这就是为什么 reasoning 模型移除了 temperature/top_p——行为控制主通道从采样超参迁移到了自然语言 + 思考预算"（见第 2 章）。

**2. 你怎么控制一个生产 agent 系统的推理成本?**
- 答题框架（组合拳,别只答一个）：(1) **分级思考路由**——按任务难度动态分配 effort（简单任务 low/关思考，难任务才 high/max），识别并抑制 overthinking；(2) **prompt caching**——把不变前缀（system/few-shot/长上下文）缓存掉，砍输入侧重复计费（命中读约 0.1×）；(3) **Batch API**——非实时请求走异步换约 5 折单价；(4) 监控 reasoning token 消耗 + 注意 agent 场景 test-time × 步数的乘法爆炸。
- 陷阱：答"把 temperature 调低 / 用小模型"——前者在 reasoning 模型上根本没这个参数，后者是模型路由不是思考路由，没答到点。**只答"分级 effort"是初级答案**——漏了输入侧（caching）和单价（batch）这两块往往更大的水分。
- 加分：把三个旋钮按"作用在账单哪个位置"讲清楚——effort 砍思考量、caching 砍重复输入、batch 砍非实时单价，三者可叠加；再点一句 caching 是前缀匹配，system prompt 里嵌时间戳会静默击穿缓存。能把它和第 20 章 gateway 路由联系起来更好——"思考深度路由是模型路由的下沉，同一模型不同请求给不同思考预算"。

**3. MCP 和 A2A 分别解决什么问题?你会现在上哪个?**
- 答题框架：MCP 解决 **agent↔工具/数据源**的标准化连接（把 N×M 适配降成 N+M，类比 ODBC），已是事实标准、生态成熟、现在就该上；A2A 解决 **agent↔agent**的互相发现与委托（更早期、标准未定、跨信任域更难），现在只值得跟踪。
- 陷阱：把两者混为一谈（都说成"agent 协议"）=暴露没真懂。它们解决的根本不是一个问题。
- 加分：给判断标准——"协议该不该上，看它消除的 N×M 适配痛是不是真的现在就疼；MCP 的痛真且普遍所以赢，A2A 的痛多数团队还没到"。

**4. 哪些 AI 应用层代码会被模型能力"吃掉"?给判断方法。**
- 答题框架：判断方法一句话——"**如果模型无限聪明、上下文无限长,这段代码还需要存在吗?**" 答"不需要"的会被吃掉（JSON 修复胶水、prompt 咒语调优、多模态预处理 pipeline、简单 CRUD、基础 agent 脚手架）。
- 陷阱：答"模型再强也得有人写代码所以都吃不掉"=盲目乐观，没有判断力。或反过来"全会被吃掉"=盲目悲观。要能**分层判断**。
- 加分：举一个已发生的实例——结构化输出（json_schema + strict）已经把"JSON 修复器"吃掉了（见第 4 章）。

**5. AI 应用工程师真正的护城河是什么?**
- 答题框架：四个词——**评测、数据、领域、分发**。共性是"私有的、需长期积累的、模型再强也替代不了的外部资产"。模型能力会平权（大家用同一批 frontier），能拉开差距的只剩这四类。
- 陷阱：答"会调最新模型 / 会写好 prompt"——这恰恰是最容易被吃掉的层。护城河不在模型层、不在 prompt 层，在"模型周围那圈工程和资产"。
- 加分：点出反直觉判断——"很多人把精力全砸在调 prompt 上，这是壁垒最薄的一层；真正的壁垒是私有数据质量、评测体系、领域逻辑、用户分发"。

**6. on-device 小模型适合什么场景?不适合什么?**
- 答题框架：适合——任务窄（分类/抽取/补全）+ 量大频高 + 对延迟或隐私有硬要求（输入法、行内补全、敏感数据预处理）。不适合——宽任务、需广博知识、复杂推理、复杂 agent（这些必须云端 frontier）。
- 陷阱：神化端侧说"端侧会取代云端"——端侧解决的是特定窄场景，不是替代大模型。或反过来完全无视端侧。
- 加分：把它接到第 20 章路由——"未来架构是模型梯队 + 本地/云端混合，路由多了一个维度：不只'路由到哪个云端模型'，还要'上不上云'"。

**7. 多模态原生模型和"OCR/ASR + 文本模型"有什么本质区别?**
- 答题框架：核心是**信息保真度**。原生多模态把图/音直接作为 token 进模型，保留版面/结构/空间信息；"先转文本"是有损压缩，丢了文档智能里最值钱的结构信息（见第 10 章）。
- 陷阱：答"多模态就是能看图片"=没说到信息有损这层。
- 加分：点出这是"代码保质期"的活例子——你今天写的 OCR/ASR 预处理胶水，会被多模态原生吃掉那层"格式转换"，但你的"该提取什么、怎么判断异常"的领域逻辑吃不掉。

**8. 你怎么持续跟上 AI 这个领域?(考学习方法论)**
- 答题框架：核心方法是**"物理 vs 版图"二分**——物理规律（自回归、幻觉机制、test-time compute、护城河在数据/评测）深挖到第一性原理、值得 all-in；版图（模型名单、价格、具体工具、协议谁赢）只需"知道现状 + 持续跟踪"，别背、会过期。学习预算 80% 砸物理、20% 跟版图。
- 陷阱：答"我每天看 AI 新闻 / 追最新模型"——这是在版图上空转，很忙但不增值。
- 加分：给可操作动作——"建'物理 vs 版图'二分笔记，每读到新东西先归类，再决定学多深"。

### D. 系统设计题（含量级估算）

> 设计一个**企业内部的 AI 编程助手网关**，支撑公司 5000 名工程师。已知：人均每天触发 200 次"行内代码补全"(低价值、要求 <100ms 延迟）+ 20 次"复杂任务对话"(高价值、可容忍数秒、需要 reasoning)。请估算：(1) 两类请求的日量级；(2) 如果补全也走云端 frontier 模型会怎样；(3) 给出你的分层架构（哪些上端侧小模型、哪些上云端、reasoning 怎么分级）及成本量级判断。

估算框架：
- **补全请求**：5000 × 200 = **100 万次/天**；对话请求：5000 × 20 = **10 万次/天**。补全量是对话的 10 倍。
- **致命点（必须算出来）**：补全要求 <100ms，而云端 frontier 的 TTFT（见第 1 章）普遍在数百 ms~秒级，**物理上就满足不了 100ms 延迟要求**——这不是成本问题，是延迟硬约束直接否决了"补全上云端"。
- **成本侧**：补全若走云端，100 万次/天，每次哪怕只 500 token，就是 5 亿 token/天，且这是**最低价值、最高频**的请求——用 frontier 烧这个，是把最贵的资源用在最不值的地方。
- **正确架构方向**：补全 → **端侧/本地小模型**（满足 <100ms + 边际成本≈0 + 隐私，见 4.1，且补全是窄任务小模型够用）；复杂对话 → **云端 frontier + 分级 reasoning**（一般任务 medium、疑难 debug 才 high/max，见 2.4）；中间加 **LLM Gateway**（见第 20 章）做路由 + 缓存 + 限流 + fallback。
- 结论方向：**这道题的题眼是"延迟硬约束 + 价值/频次错配"逼出分层架构**——补全和对话是两个物理特性完全不同的负载，绝不能用同一个模型一刀切。能把"100ms 约束直接否决云端补全"这个判断点出来，并落到"端侧补全 + 云端分级 reasoning + 网关路由"的三层架构，就过了。

> 高级加分：补一句成本治理的动态视角——网关要监控每个工程师的 reasoning token 消耗，对异常高消耗（可能是某人把所有任务都设 max effort，见 2.3 的成本陷阱）告警和限额，否则一个误用就能烧穿预算。这把"会设计架构"升级成"会运营架构"。

**测试要求**：
1. 断言 `_route_effort(CLASSIFY)` 返回 low（或关思考），`_route_effort(CRITICAL)` 返回 max——验证分级路由没写反。
2. 把 `daily_reasoning_budget_tokens` 设很小，连续跑几次后断言 `_over_budget()` 为真，且后续高 effort 任务被**降级而非崩溃**（验证 2.3 成本陷阱的防御）。
3. 断言成本计算里 reasoning token 是按**输出价**算的（核对没把它当免费或当输入价）。
4. （进阶）mock 一个"CLASSIFY 任务被误设成 high effort"，断言 CostGuard 能纠正回 low（防 overthinking）。
5. （关键——这条最像真实事故）mock 掉 `client.messages.stream`，断言 `run()` **真的把路由出来的 effort 传进了 `output_config`**：跑一个 CLASSIFY 任务，断言透传给 SDK 的 `output_config["effort"] == "low"`。这一条专门抓"算了 effort 却没传"这个最隐蔽的 bug——分级路由的代码全在，但请求里没带 effort，线上表现是"成本怎么调都不降",查起来极费时间。

**暗坑清单**（踩中就当场翻车）：
- **算了 effort 却没透传**（最隐蔽、最该警惕）→ `_route_effort` 写得再漂亮，只要 `run()` 里漏了 `output_config={"effort": effort}`，所有请求都跑默认 high，分级路由形同虚设。这类 bug 不报错、不崩、测试若不验透传也照样绿——只有账单不降。这正是"代码意图（路由）和实际行为（没传）脱节"的典型，写完务必加测试要求 5 这种透传断言。
- 一刀切全设 max effort → 95% 请求过度思考，账单翻 5-10 倍 + 延迟暴涨（2.3 成本陷阱）。
- 给 CLASSIFY 这种窄任务高 effort → 触发 overthinking，把简单问题想复杂，又慢又可能更错。
- 手撸 `budget_tokens` 或传 `temperature/top_p` → 新旗舰 400 报错（用 `thinking:{type:"adaptive"}` + effort）。
- `max_tokens` 设太小（比如照抄某些 demo 的 1024）→ 长链推理被拦腰截断，`stop_reason=max_tokens`，答案残缺还白烧了思考 token。reasoning 模型要给 `max_tokens` 留足空间。
- 算成本时把 reasoning token 当免费或按输入价算 → 长 agent 跑下来数字对不上（reasoning 按输出价烧，可能是输出大头）。
- 超预算直接 `raise` → 把高频系统打崩；正确是**降级**（强制低 effort 或拒绝高 effort 任务）。

考点：能不能把"分级思考路由 + reasoning 预算守卫"做对——这是 test-time compute 时代的成本工程在代码层的硬落地，等同于 RAG 时代你得会写 chunking。
