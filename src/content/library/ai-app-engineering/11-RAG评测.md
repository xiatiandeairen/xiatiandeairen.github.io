## 一、背景：为什么没有 eval 的 RAG 是碰运气

### 1. 问题出现的根源：RAG 的质量是"多环节连乘"，肉眼归因不了

第 8 章说过，RAG 是两阶段八环节的流水线。一个用户问题答得烂，可能烂在任何一环：chunk 切碎了语义、embedding 召回不全、rerank 把对的排下去了、prompt 拼得让模型忽略了材料、模型曲解了原文……**这些环节是连乘关系**——任何一环掉链子，最终答案就崩，而你从最终那段文字里**根本看不出是哪一环崩的**。

更要命的是 LLM 的输出有随机性（第 1 章：概率采样）。同一个问题问两遍，措辞都不一样。你改了个参数，重跑一遍发现答案变了——这是改动带来的**真实改善**，还是采样噪声？没有量化指标和足够样本，你区分不了。

这就是为什么"我手动测了几个 case 感觉不错"是新手第一误区。手动看 case 有三个致命缺陷：

- **样本太小**：你测的 3~5 个问题，可能恰好是系统擅长的；线上千奇百怪的真实问题，你没覆盖。
- **不可复现**：你"感觉变好了"无法传递给同事、无法回归测试、下次改动无法对比。
- **归因靠猜**：答得不好，你凭直觉猜"大概是检索问题"，然后瞎调 chunk size，调完发现没用——因为根本是 prompt 的锅。

> 心法：**RAG 调优的前置条件不是"想清楚怎么优化"，是"先能测量"。** 没有度量，优化就没有方向，改动就没有反馈，"变好"就只是一种自我安慰。先建尺子，再谈手术。

### 2. 演进：评测手段是怎么一步步进化的

- **阶段零：人肉看 case。** 跑几个问题，人眼判断好坏。前面说了，样本小、不可复现、归因靠猜。只适合最早期 demo 阶段确认"能跑通"。
- **阶段一：传统 IR 指标上场。** 检索本质是信息检索（Information Retrieval），IR 领域几十年的成熟指标——recall、precision、MRR、NDCG——直接拿来量化"检索环节"好不好。这一步把**检索质量**变成了可计算的数字。但它只覆盖检索，管不了"生成环节有没有忠实于材料"。
- **阶段二：生成侧指标补齐。** 光检索对了不够——材料对了模型也可能曲解、可能答非所问、可能掺私货。于是有了 faithfulness（忠实度）、answer relevance（答案相关性）、context relevance（上下文相关性）这类**生成侧指标**。难点是这些东西没法用规则算，得靠"判断语义"——这就引出了 LLM-as-judge。
- **阶段三：LLM-as-judge + 框架化。** 用一个强模型当"裁判"去打分"这个答案忠不忠实于材料""答没答到点上"。RAGAS、TruLens、DeepEval 这类框架把上面这套指标体系封装好，几行代码跑出一张评测报告。这是 2023 年后 RAG 评测的主流形态。
- **阶段四（当前前沿）：线上反馈闭环。** 离线评测集再全，也覆盖不了真实用户的所有问法。把线上的用户行为信号（点了"有用/没用"、是否追问、是否复制了答案、是否转人工）回流，持续扩充评测集、发现新失效模式。离线兜底质量、在线发现盲区，两者咬合成飞轮。

> 工程推论（本节地基，逐条都是面试题）：
>
> **推论一：检索和生成必须分开评测，否则你永远在错误的地方调参。** 这是 RAG 评测的第一性原理。最终答案差，要么是"没找到对的材料"（检索问题），要么是"找到了但没用好"（生成问题），治法完全相反。混在一起测，你只知道"差"，不知道往哪修。**分层归因的能力，是 RAG 工程师和调参侠的分水岭。**
>
> **推论二：评测集（golden set）是 RAG 项目里最值钱、最该早攒的资产。** 模型会换、框架会换、chunk 策略会换，但"这些问题的正确答案/正确来源是什么"这份标注是稳定的。它是你所有迭代的**回归测试基线**。第 8 章 P2 验收第 7 条让你"攒 ≥20 条带标注问答"，就是为这一章铺路——现在该用上了。

## 二、Golden Set：评测集怎么造

评测集（业内叫 golden set / eval set / 测试集）是一批 `(问题, 标准答案, 标准来源)` 的三元组。它是整个评测体系的地基——**地基歪了，上面的指标再漂亮都是自欺**。

### 1. 三类必须覆盖的样本

只放"简单问题"的评测集是废的——它会让你的系统在 demo 里满分，上线就翻车。一个能用的 golden set，必须按比例覆盖三类：

| 样本类型 | 是什么 | 为什么必须有 | 举例（企业知识库场景） |
|---|---|---|---|
| **真实问题（typical）** | 用户实际会问的高频问法 | 代表主流场景，是基本盘 | "报销流程是怎样的" |
| **难例（hard）** | 答案藏得深、需要跨 chunk 综合、问法和原文措辞差异大 | 暴露检索召回和生成综合能力的短板 | "出差超过 3 天且跨境的住宿标准和报销凭证要求分别是什么"（答案分散在两个章节） |
| **对抗例（adversarial）** | 故意刁难：库里没答案的、问法有歧义的、诱导编造的、含错误前提的 | 测**拒答能力**和**抗幻觉**——这是合规和信任的命门 | "公司去年的离职率是多少"（库里压根没这数据，必须答"无法回答"而非编造一个） |

**比例建议（按 2026 初经验，非铁律）**：典型问题占 50~60%，难例 25~30%，对抗例 15~20%。对抗例占比看场景——法律、医疗、金融这类**容错率极低**的场景，对抗例（尤其是"诱导编造"和"库外提问"）要加码到 30% 以上，因为在这些场景里"编造一个看似合理的答案"比"答不出来"危害大得多。

> 坑点提示（新手第一误区）：评测集只放自己系统答得出的问题。这是在**给自己发奖**——你测的全是擅长的，自然分数高，但这数字对预测线上表现毫无价值。**好的评测集应该让你的系统当前版本有明显失分**，失分点就是你下一步该优化的地方。如果你的系统在 golden set 上一上来就 95 分，不是系统强，是你的 golden set 太软。

### 2. 怎么造：四个来源

1. **从真实日志挖（最高价值）**：如果系统已上线，扒线上真实 query，按频次和代表性抽样。这是最贴近真实分布的来源。冷启动没日志时用下面三个。
2. **领域专家出题**：让懂业务的人（客服主管、法务、医生）针对知识库出"用户真会问且重要"的问题，并给出标准答案和标准来源。难例和对抗例尤其需要专家——他们知道哪些是坑。
3. **从文档反向生成（规模化，但要清洗）**：用 LLM 读每个 chunk，反向生成"这段能回答什么问题"。能快速凑数量，但**必须人工审一遍**——LLM 生成的问题容易是"照着原文措辞填空"的伪问题（答案和问题字面高度重叠，检索零难度），跟真实用户的口语化、跨语义问法差很远。直接拿 LLM 生成的问题当 golden set，会系统性高估你的检索能力。
4. **标注"正确来源"，不只是"正确答案"**：这点最容易被偷懒省掉，但它是**分层归因的命根子**。每个问题不仅要标"答案应该是什么"，还要标"答案出自哪个/哪些 chunk（或文档+位置）"。没有标准来源，你就没法算 recall（不知道"对的 chunk"是哪个），分层归因里"检索这一层"就成了瞎子。

> 工程推论：
>
> **推论三：golden set 的标注成本，要花在"标准来源"上，不能只标"标准答案"。** 很多人只标了答案文本，结果只能做端到端评测（答案对不对），做不了组件级评测（检索召回率）。标准来源（哪些 chunk 是相关的）才是解锁检索指标的钥匙。这条直接决定你的评测能不能分层。
>
> **推论四：golden set 不是一次性交付物，是活的资产。** 线上每发现一个答错的 case、每出现一类新问法，就补进 golden set。它会随系统演进持续生长——这正是第六节"线上反馈闭环"要做的事。把 golden set 当代码一样进版本控制、像测试用例一样维护。

> 面试加分点：被问"怎么构造 RAG 评测集"，初级答案是"收集一些问题和答案"。高级答案要点出三个关键：(1) **必须含难例和对抗例**，且对抗例要专门测拒答（否则评测集会高估系统）；(2) **必须标注标准来源不只是标准答案**，否则做不了检索侧归因；(3) **警惕用 LLM 从文档反生成问题导致的"伪问题"偏差**，要人工清洗。能讲清这三点，说明你真做过、踩过坑。

## 三、分层归因：检索指标 vs 生成指标

这是本章的技术核心，也是 RAG 评测和"随便测测"的本质区别。**一个答案不好，先别急着改，先定位是哪一层的锅。**

### 1. 为什么必须分层

回到推论一：最终答案的质量 = 检索质量 × 生成质量（连乘）。这两层的**失效模式不同、治理手段不同**：

- **检索层失效**：对的材料根本没被捞上来（召回不全），或捞上来一堆不相关的（精度差）。→ 治法：调 chunk、换 embedding、加混合检索、加 rerank（第 9 章那套）。
- **生成层失效**：对的材料明明在上下文里，模型却曲解了、忽略了、或掺了私货。→ 治法：改 prompt、加约束、换生成模型、调上下文顺序。

如果你只看"最终答案对不对"，这两类问题会**混在一个数字里**，你无法知道该动检索还是动生成。分层归因就是把这个连乘**拆开单独量化**。

### 2. 检索侧指标（IR 指标，能精确计算）

检索的任务是：给定 query，从知识库里捞出"相关的 chunk"。有了 golden set 标注的"标准来源"，下面这些指标都能**用规则精确算出来**（不需要 LLM judge）：

| 指标 | 衡量什么 | 直觉解释 | 什么时候最该看它 |
|---|---|---|---|
| **Recall@k** | top-k 里捞回了多少比例的相关 chunk | "该找的，找回来了几成" | RAG 最关键指标——**没召回的材料，生成再强也救不了** |
| **Precision@k** | top-k 里有多少比例是真相关的 | "捞上来的，有几成是对的" | 噪声材料会干扰生成、挤占 context、增 token 成本 |
| **MRR**（Mean Reciprocal Rank） | 第一个相关结果排在第几位（取倒数） | "对的答案，是不是排在前面" | 只需一个正确答案的场景（如事实问答） |
| **NDCG@k** | 带"位置折扣"的相关性排序质量 | "相关的有没有排在前面，且越相关越靠前" | 多个相关材料、且相关度有层次时（rerank 效果的核心指标） |

**怎么理解这几个的分工**（这是高频考点）：

- **Recall@k 是 RAG 的命根子**。逻辑很硬：如果正确答案所在的 chunk 没进 top-k，那它根本不在模型的上下文里，模型不可能凭空答对（除非它瞎蒙撞对，那更糟）。所以**检索召回是生成的天花板**——recall@5 = 0.6，意味着 40% 的问题从一开始就没救。优化 RAG，第一个要盯死的就是 recall。
- **Precision 和 Recall 是一对拉扯**。把 k 调大，recall 升（捞得多自然漏得少）但 precision 降（混进更多不相关的）。噪声 chunk 不是无害的——它们挤占上下文、稀释信号、增加"lost in the middle"风险、还多烧 token。所以不是 k 越大越好。
- **MRR vs NDCG 的区别**：MRR 只关心"第一个对的排第几"，适合"找到一个正确答案就行"的场景。NDCG 关心"所有相关的有没有都排到前面、且越相关越靠前"，适合"需要综合多份材料"的场景，也是**衡量 rerank 价值**的核心指标（rerank 的本职就是把最相关的往前提，这正是 NDCG 在测的）。

> 工程推论：
>
> **推论五：检索召回（Recall@k）是 RAG 的硬天花板，优化 RAG 先盯 recall。** 这是最反直觉也最重要的一条。新手容易一上来就调 prompt、换大模型（动生成层），但如果 recall 低，这些全是白费——材料就没进来，生成端再怎么折腾也无米下炊。正确顺序是：**先把 recall 拉上去，再优化生成。**
>
> **推论六：检索指标能用规则精确算，是 RAG 评测里最便宜、最可靠的一块——优先把它建起来。** 不像生成指标要靠 LLM judge（贵、慢、有噪声），recall/precision/MRR/NDCG 只要有 golden set 的来源标注，纯计算就能出，确定性、零成本、可秒级回归。**这是性价比最高的评测基建，应该第一个落地。**

### 3. 生成侧指标（靠 LLM-as-judge 判断语义）

材料对了，还要看生成环节有没有用好。生成侧三个核心指标（RAGAS 把这套叫 RAG Triad，业内常用）：

| 指标 | 衡量什么 | 失效长什么样 | 这层出问题该怎么修 |
|---|---|---|---|
| **Faithfulness（忠实度）** | 答案的每个论断，是否都能在检索材料里找到依据 | 模型"基于材料"答，却掺了材料里没有的私货——这就是 grounding 之后**残余的幻觉** | 强化 prompt 约束、降生成温度、加"逐句核对材料"指令 |
| **Answer Relevance（答案相关性）** | 答案是否切题、有没有答到用户问的点上 | 答案虽忠实于材料，但答非所问、绕圈子、答了无关的 | 改 prompt 让模型聚焦问题、过滤无关材料 |
| **Context Relevance（上下文相关性）** | 检索出的材料和问题相不相关 | 捞回一堆不相关 chunk（其实这是**检索侧**的信号） | 提升检索精度（注意：这条其实在测检索，是分层归因的桥梁） |

这三个指标在 RAG 里如何**协同归因**（这是设计题级别的理解）：

- **Context Relevance 低** → 检索捞回的材料就不相关 → **检索层的锅**（去优化 recall/precision、rerank）。
- **Context Relevance 高，但 Faithfulness 低** → 材料是对的，模型却没忠实于它、掺了私货 → **生成层的锅**（去改 prompt、降温度、加约束）。
- **Faithfulness 高，但 Answer Relevance 低** → 模型忠实于材料、没瞎编，但没答到点上（可能材料相关但不是用户真正想问的那个点）→ 可能是**检索精度**问题（捞回相关但偏题的材料）或 **prompt** 没引导聚焦。

看出来了吗？**这三个生成侧指标 + 检索侧的 recall/precision，组合起来就是一棵完整的归因决策树。** 把它画出来：

```
        最终答案不好
              │
     ┌────────┴────────┐
     │  Recall@k 低？   │
     └────────┬────────┘
        是 │          │ 否（材料召回是全的）
           ▼          ▼
   ┌──────────────┐  ┌─────────────────────────┐
   │ 检索召回问题  │  │  Context Relevance 低？   │
   │ → 调chunk/    │  └────────────┬────────────┘
   │   embedding/  │      是 │           │ 否
   │   混合检索    │         ▼           ▼
   └──────────────┘  ┌──────────┐  ┌──────────────────┐
                     │检索精度差 │  │ Faithfulness 低？  │
                     │→加rerank/ │  └────────┬─────────┘
                     │  提precision│    是 │        │ 否
                     └──────────┘       ▼        ▼
                              ┌──────────┐  ┌──────────────┐
                              │生成不忠实 │  │Answer Rel 低？│
                              │→改prompt/ │  └──────┬───────┘
                              │ 降温/加约束│      是 │
                              └──────────┘         ▼
                                          ┌──────────────┐
                                          │答非所问       │
                                          │→prompt聚焦/   │
                                          │ 过滤偏题材料  │
                                          └──────────────┘
```

> 心法：**先问 recall，再问 context relevance，再问 faithfulness，最后问 answer relevance——这个顺序就是 RAG 的归因路径。** 从"材料有没有进来"一路问到"答案有没有答好"，每一步把锅精确定位到一层。能熟练走这棵树的人，调 RAG 又快又准；不会走的人，永远在错误的层瞎调。

## 四、RAGAS 类框架：别重复造轮子

上面的生成侧指标，自己用 LLM judge 实现一遍是可以的（代码题会让你写），但生产里通常用现成框架。**RAGAS**（RAG Assessment）是这个领域 2023 年后最知名的开源框架，同类还有 **TruLens**、**DeepEval**、**Arize Phoenix** 等。

### 1. 这类框架替你做了什么

- **封装了 RAG Triad 指标**：faithfulness、answer relevance、context relevance 开箱即用，内部用 LLM judge 实现，你不用自己写 judge prompt。
- **封装了检索指标**：有 golden set 的来源标注时，自动算 context recall / context precision。
- **标准化输入输出**：你喂 `(question, answer, contexts, ground_truth)`，它吐一张各指标的分数表。
- **对接评测数据集和可视化**：能批量跑、出报告、画趋势。

典型用法长这样（伪代码，具体 API 按 2026 初，以官方文档为准）：

```python
# 概念示意——展示评测的数据形态，不是某框架的精确 API
from ragas import evaluate
from ragas.metrics import faithfulness, answer_relevancy, context_recall, context_precision

# 每条样本：问题、系统给的答案、系统检索到的材料、标准答案
dataset = [
    {
        "question": "出差住宿标准是多少",
        "answer":系统生成的答案,
        "contexts": 系统检索到的_chunk_列表,   # 喂给 context relevance/precision
        "ground_truth": 标准答案,              # 喂给 context recall / answer 对比
    },
    # ... 你的整个 golden set
]

result = evaluate(dataset, metrics=[
    faithfulness, answer_relevancy, context_recall, context_precision,
])
# result → 每个指标一个聚合分数 + 每条样本的明细
```

### 2. 用框架的两个关键认知

**第一：框架解决的是"指标怎么算"，不解决"评测集怎么造"。** golden set 的质量（含不含难例、有没有标准来源、是不是伪问题）框架管不了，那是你的活。**框架是放大器——评测集好，它放大出可信的洞察；评测集烂，它放大出精确的垃圾。** 别因为用了高大上的框架就觉得评测靠谱了，地基永远是 golden set。

**第二：框架内部的 judge 也是 LLM，继承 LLM-as-judge 的全部陷阱（见下一节）。** RAGAS 的 faithfulness 分数是某个 judge 模型打的，这个分数有噪声、有偏见、会因 judge 模型版本变化而漂移。**别把框架吐的分数当物理测量值看**，它是"一个 LLM 的判断"，要校准、要交叉验证。

> 工程推论：
>
> **推论七：框架是工程加速器，不是质量保证。评测的可信度由 golden set 和 judge 校准决定，不由框架决定。** 大厂用 RAGAS 这类工具是为了省实现成本、统一指标口径，但绝不会盲信它的输出——会拿人工标注的"金标准子集"去校准框架分数（见第五节）。把框架分数当绝对真理，是新手对工具的迷信。

## 五、LLM-as-judge：在 RAG 里的应用与陷阱

faithfulness、answer relevance 这些"判断语义"的活，规则算不了，只能让另一个 LLM 来当裁判打分。这就是 **LLM-as-judge**——RAG 生成侧评测的主力手段，但**坑极多**，用不好你的评测数字全是假的。

### 1. 为什么要用，怎么用

**为什么用**：生成质量是语义判断（"这个答案忠不忠实于材料"），没有规则能算，人工标又贵又慢、跟不上迭代速度。用一个强 LLM 当 judge，是"语义评测"在成本和规模上唯一可行的方案。

**怎么用得相对靠谱**（这些是 RAG judge 的工程要点）：

1. **给 judge 明确的评分标准（rubric），不要笼统打分。** 别问"这答案好不好打 1-10 分"——太主观、不稳定。要拆成可判断的维度：faithfulness 就让它"逐条核对答案里的每个论断在材料里有没有依据，有几条没依据"，输出可量化的结果。
2. **优先二元/离散判断，少用连续分数。** "这条论断在材料里有依据吗（是/否）"比"忠实度打几分"稳定得多。连续分数在 LLM 这里噪声大、不同 judge 标准漂移。能转成"数有依据的论断占比"就别让它直接给 0.87 这种数。
3. **judge 要拿到它需要的全部上下文。** 评 faithfulness，judge 必须同时看到 `(检索材料, 答案)`；评 answer relevance，必须看到 `(问题, 答案)`。少喂任何一块，judge 就是在瞎猜。
4. **judge 输出结构化结果。** 用结构化输出约束 judge 返回 `{verdict, reason, unsupported_claims}` 这种可解析的结构，而不是一段散文。这样能批量统计、能审计 judge 的判断理由（理由不合理的，说明这条分数不可信）。
5. **判断类任务可以并发跑、可以走 Batch。** 评测是离线、非实时的，几百上千条 judge 调用没必要串行。Anthropic 的 Batch API 能以 50% 价格异步跑这种大批量评测，是离线 eval 的理想载体——攒一批样本一次性提交，省一半钱（按 2026 初，以官方为准）。
6. **judge 的 rubric 是固定前缀，用 prompt caching 把它"摊销"掉。** 这是被新手严重低估的成本杠杆。你跑 500 条 faithfulness judge，**每条的 rubric / 评分指令 / few-shot 示例是完全一样的**，只有"材料+答案"在变。如果不缓存，这段固定前缀每条都按全价重算一遍——而它往往占单条输入的一大半。把固定部分放在 prompt 最前面、加上 cache breakpoint，缓存命中后这段只按约 0.1× 计价（cache 写一次约 1.25×）。**对"同一把尺子量很多答案"的评测场景，缓存读省下的钱常常比 Batch 的 50% 还多，而且两者叠加。** 关键纪律：固定前缀里**绝不能掺时间戳/UUID/每条变化的 id**——任何一个字节变化都会让其后的缓存全部失效（前缀匹配原理），缓存命中率瞬间归零。用 `usage.cache_read_input_tokens` 验证是否真的命中。

> 落地细节（如果 judge 用 Claude）：judge 调用应开 **adaptive thinking**（`thinking: {type: "adaptive"}`）——判断忠实度这种需要逐条核对的任务，让模型先想再裁，质量明显更稳；并用**结构化输出**（`output_config: {format: {...}}`，旧的 `output_format` 已废弃）把裁决约束成可解析的 JSON。批量离线评测走 **Batch API**（异步、50% 价格、单批最多 10 万请求）。把 rubric/few-shot 这类**固定前缀放到 prompt 最前并打 cache breakpoint**，渲染顺序是 `tools→system→messages`、稳定内容必须排在易变内容之前（材料+答案放最后），用 `usage.cache_read_input_tokens` 验证命中——**Batch 50% 折扣 × prompt caching 缓存读 ≈0.1× 是可以叠乘的两层降本**。这些是判断准确性、可解析性、成本三件事的直接抓手（具体接口以官方文档为准）。

> 面试加分点：被问"你怎么把 LLM-as-judge 评测的成本压下来"，只会答"用便宜模型当 judge"是初级——便宜模型当 judge 会牺牲判断质量（见下文校准）。高级答案是三层：(1) **结构上**——检索指标用规则免费算，只把 judge 花在生成侧；(2) **批量上**——离线评测走 Batch 拿 50% 折扣；(3) **缓存上**——judge 的 rubric 是固定前缀，用 prompt caching 让它从全价降到约 0.1×，且和 Batch 叠加。能点出"固定前缀缓存"这一层，说明你真算过 judge 评测的账。

### 2. 陷阱清单：judge 会怎么骗你

这是本节灵魂，也是和第 1 章谄媚问题的连接点。**judge 是个 LLM，它有 LLM 的全部毛病**——会谄媚、有偏见、不稳定。把这些当物理仪表读，你的评测就是在自我欺骗。

| 陷阱 | 表现 | 危害 | 缓解 |
|---|---|---|---|
| **自我偏好（self-preference）** | judge 偏袒和自己同源的模型生成的答案 | 你用 模型 X 生成、又用 模型 X 当 judge → 分数虚高 | **judge 和被评模型不要同源**（关键！） |
| **冗长偏见（verbosity bias）** | judge 倾向给更长、更详细的答案打高分，哪怕没更对 | 鼓励系统输出又臭又长，刷分但用户体验差 | rubric 里明确"简洁且正确 > 冗长"；控制答案长度 |
| **位置偏见（position bias）** | 成对比较时，judge 偏向排在前面（或后面）的那个 | A/B 对比结果被呈现顺序污染 | 两个顺序各跑一次取平均；或随机化顺序 |
| **谄媚 / 自信偏见** | judge 偏向语气自信、措辞肯定的答案（连第 1 章 sycophancy 那套） | "自信地胡说"的答案可能比"诚实地说不确定"得分高 | rubric 强调"以材料为唯一依据，语气自信不加分" |
| **判断不稳定** | 同一答案judge 两次给不同分（采样随机性） | 评测数字有噪声，小改动的信号被淹没 | judge 低温度/多次取众数；用离散判断而非连续分 |
| **不会拒绝坏 rubric** | 你的评分标准本身有歧义，judge 会"配合"地给个分 | 你以为测的是 A，judge 其实在测 B | rubric 先人工试评、消歧义；抽查 judge 的判断理由 |

**最致命的是第一条：judge 不能和被评模型同源。** 逻辑很硬——如果生成答案的是 模型 X、打分的也是 模型 X，X 会系统性偏好"自己风格的输出"，分数毫无意义地虚高。生产里的做法是：**用一个更强的、不同家的模型当 judge**（比如生成用某中杯模型，judge 用一个 frontier 级的、不同厂商的模型），或者至少用同厂不同代/更强的模型。judge 应该比被评对象"更权威"，而不是它的同类。

> 工程推论：
>
> **推论八：LLM-as-judge 的输出是"一个有偏见的 LLM 的判断"，不是客观测量，必须校准后才能信。** 校准方法：人工标注一个"金标准子集"（几十条，人来打分），拿 judge 在这个子集上的判断和人工对齐——如果 judge 和人的一致率（agreement）低，这个 judge / rubric 就不能用。**没校准过的 judge 分数，约等于没有。**
>
> **推论九：judge 不能和被评模型同源，否则自我偏好让分数虚高——这是 LLM-as-judge 的头号红线。** 面试被问 "LLM-as-judge 有什么坑"，如果只答得出这一条，也比答不出强；能把 verbosity/position/self-preference 三个偏见说全，且点出"用更强的异源模型当 judge + 人工子集校准"，就是高级水平。
>
> **推论十：judge 偏爱长答案、自信语气——这和模型谄媚是同一个病根（第 1 章）。** judge 也是概率延续器，它对"看起来更努力、更自信"的文本有天然好感，跟它对用户谄媚是一回事。所以 rubric 必须显式对抗这种倾向（"简洁不扣分、自信不加分、唯一依据是材料"），否则你的评测会奖励"会表演的答案"而非"对的答案"。
>
> **推论十一：judge 模型版本要钉死（pin），换 judge 等于换尺子——历史基线全部作废，必须重测。** 这是把 judge 当生产基建时最容易踩的坑。你的 faithfulness 基线是 0.91，半年后厂商上了新一代模型、你顺手把 judge 升级了——分数变成 0.88，**你根本分不清是系统退化了还是新 judge 标准变严了**，因为你同时动了"被测对象"和"尺子"。模型名按季度过期（2026 初的 judge 模型名，过半年可能就换代了），所以纪律是：(1) **judge 模型 ID 显式钉死、记进评测报告**，别用会自动指向最新版的别名；(2) 升级 judge 是一次**独立变更**——先在金标准子集上重新校准、对历史样本重测建立新基线，再用新基线继续，绝不在升级 judge 的同时评估系统改动。一句话：**尺子和被测物，一次只能动一个。**

## 六、端到端 vs 组件级 + 线上反馈闭环

### 1. 端到端评测 vs 组件级评测

最后一块拼图：评测在**哪个粒度**上做。两种粒度都要有，分工不同。

| 维度 | 端到端评测（end-to-end） | 组件级评测（component-level） |
|---|---|---|
| **测什么** | 整条流水线：问题进 → 最终答案出，答案对不对 | 单个环节：检索 recall、rerank NDCG、生成 faithfulness 各自的质量 |
| **回答的问题** | "这套系统对用户到底好不好用" | "是哪个环节拖了后腿" |
| **优点** | 贴近真实用户体验，是终极目标 | 能精确归因、能独立优化某一环 |
| **缺点** | 答案差时不知道哪环的锅（归因黑箱） | 单环好不代表整体好（环节间有耦合） |
| **什么时候用** | 验收、对外汇报、A/B 决策 | 日常调优、定位瓶颈、回归测试 |

**两者的关系**（高频考点）：**端到端测"结果"，组件级测"原因"。** 端到端告诉你"系统从 72 分涨到了 80 分"（对用户的净收益），组件级告诉你"这 8 分是 recall 从 0.71 升到 0.83 贡献的，生成侧基本没动"（收益来自哪）。**只有端到端，你知道好坏但不知道为什么、改哪；只有组件级，你优化了单环却不知道对用户的真实净效果（单环涨了整体未必涨，因为环节间耦合）。** 生产里必须两层都建：组件级指导日常优化方向，端到端做最终验收和 A/B 决策。

> 坑点提示：**组件级指标全涨，不保证端到端涨。** 比如你把 recall@10 从 0.7 调到 0.9，但代价是塞进了更多噪声 chunk，结果生成端被噪声干扰、faithfulness 反而降了，端到端答案质量没变甚至变差。这就是"局部最优 ≠ 全局最优"。所以**改完任何单环，都要回看端到端指标确认净收益为正**——别拿组件级的局部胜利当最终战果。

### 2. 线上反馈闭环：离线评测的盲区补丁

离线 golden set 再用心造，也有个根本局限：**它是你"想象中"的用户问题，不是真实用户的问题。** 真实用户的问法千奇百怪、会问你没料到的边角、会用你没见过的措辞。线上反馈闭环就是把真实流量变成评测的养料：

**采什么信号**（从弱到强）：
- **隐式信号**：用户是否复制了答案、是否立刻追问（往往意味着没答好）、是否转人工、停留时长。这些不打扰用户，但噪声大。
- **显式信号**：答案旁的"有用/没用"按钮、点踩后的原因选择。信号强但收集率低（多数人不点）。
- **人工抽审**：定期抽线上真实 query + 答案，人工评判。最准但最贵，适合抽样兜底。

**闭环怎么转**：
1. 线上信号发现"这类问题答得差"（比如某类问题点踩率高）。
2. 把这类 case 捞出来、人工确认、**补进 golden set**（含标准答案和标准来源）。
3. 在扩充后的 golden set 上重测，量化这类问题当前有多差。
4. 针对性优化（先分层归因到检索还是生成），改完在 golden set 上验证收益。
5. 灰度上线，再看线上信号是否改善。回到第 1 步。

> 心法：**离线评测兜质量下限，线上反馈找质量盲区，两者咬合成飞轮。** 离线 golden set 保证"已知的问题不退化"（回归测试），线上反馈负责"发现未知的问题"（扩充覆盖）。只有离线，你会在自己的舒适区里反复优化、对真实盲区一无所知；只有线上，你没有稳定基线、无法回归、改动好坏靠运气。成熟的 RAG 团队，这两个轮子一起转。

> 工程推论：
>
> **推论十一：离线评测和线上反馈解决的是不同问题——前者防退化（已知问题的回归），后者找盲区（未知问题的发现），缺一不可。** 这是评测体系成熟度的标志。只有离线 eval 的团队，会陷入"在 golden set 上刷分但线上没变好"的虚假繁荣；只有线上 A/B 的团队，没有稳定回归基线，每次改动都在裸奔。
>
> **推论十二：线上反馈的最大价值不是"打分"，是"反哺 golden set"。** 用户的点踩本身噪声很大、不能直接当指标，但它是**发现新失效模式的雷达**——指向"这类问题需要被纳入评测"。把线上信号当"golden set 的进货渠道"，而不是直接当 KPI，才用对了它。

## 本章五件套

### 一、高频面试题（附答题框架）

**1. RAG 为什么必须把检索和生成分开评测？只看最终答案对不对不行吗？**
- 框架：先讲第一性原理——最终质量 = 检索质量 × 生成质量（连乘），任一环崩则结果崩。只看最终答案，两类失效（材料没召回 vs 召回了没用好）混在一个数字里，**无法归因、无法定向优化**。再讲两层失效模式与治法完全不同（检索→调 chunk/embedding/rerank；生成→改 prompt/降温/加约束）。收尾点出 recall 是天花板——材料没进来，生成端再强也无米下炊。
- 陷阱：只会说"分开测更细致"，说不出"连乘"和"治法不同"这两个硬逻辑，就是没真理解。

**2. Recall、Precision、MRR、NDCG 在 RAG 里分别什么时候看？为什么 recall 最关键？**
- 框架：Recall@k = 该找的找回几成（**RAG 命根子，是生成的天花板**，没召回的材料生成救不了）；Precision@k = 捞回的有几成对的（噪声挤占 context、增成本）；MRR = 第一个对的排第几（单答案场景）；NDCG = 带位置折扣的排序质量（多材料 + rerank 效果的核心指标）。为什么 recall 最关键：正确 chunk 没进 top-k 就不在模型上下文里，不可能答对。所以优化 RAG 先盯 recall。
- 加分：点出 precision-recall 拉扯（k 越大 recall 升 precision 降），k 不是越大越好。

**3. faithfulness、answer relevance、context relevance 这三个生成侧指标怎么协同做归因？**
- 框架：按归因链讲——Context Relevance 低 = 检索的锅；Context Relevance 高但 Faithfulness 低 = 生成不忠实（掺私货）；Faithfulness 高但 Answer Relevance 低 = 答非所问（检索精度或 prompt 问题）。强调这三个 + recall 组合成一棵归因决策树，顺序是 recall → context relevance → faithfulness → answer relevance。
- 陷阱：能背出三个指标定义但讲不出"它们怎么组合定位到某一层"，是只背了概念没理解用法。

**4. LLM-as-judge 有哪些坑？怎么让它的分数可信？**
- 框架：列偏见——**self-preference（头号红线：judge 不能和被评模型同源，否则虚高）**、verbosity bias（偏长答案）、position bias（成对比较偏前/后）、谄媚/自信偏见（连第 1 章 sycophancy）、判断不稳定（采样噪声）。怎么可信：(1) judge 用更强的异源模型；(2) 给明确 rubric、用离散判断而非连续分；(3) **人工金标准子集校准**（judge 与人工一致率低就不能用）；(4) 结构化输出 + 抽查判断理由；(5) **judge 模型版本钉死**——换 judge 等于换尺子，历史基线作废，升级 judge 必须独立重新校准+重测，绝不和系统改动同时动。
- 加分：把 judge 偏爱长答案/自信语气和"judge 也是概率延续器、也会谄媚"这个根子联系起来。

**4b. judge 评测很贵，你怎么把成本压下来？**
- 框架：三层叠加——(1) **结构上**：检索指标用规则免费算，judge 只花在生成侧（faithfulness 等）；(2) **批量上**：离线评测走 Batch API 拿 50% 折扣；(3) **缓存上**：judge 的 rubric/few-shot 是**所有样本共享的固定前缀**，用 prompt caching 让它从全价降到约 0.1×，且和 Batch 折扣叠乘。三层下来成本能降到原始的两三成。
- 陷阱：只答"换个便宜模型当 judge"——便宜 judge 牺牲判断质量，是用评测可信度换钱，方向错了。讲不出"rubric 固定前缀可缓存"，说明没真算过 judge 评测的账。
- 加分：点出缓存的纪律——固定前缀里掺时间戳/UUID 会让缓存静默击穿（前缀匹配，一字节变化全失效），用 `usage.cache_read_input_tokens` 验证命中；以及估 token 别用 tiktoken（估 Claude 系统性偏差 15~20%），要用 `count_tokens`。

**5. 怎么构造一个好的 RAG 评测集（golden set）？**
- 框架：三类样本必须齐——典型问题（基本盘）、难例（跨 chunk/措辞差异大）、对抗例（库外提问/诱导编造/歧义，专测拒答和抗幻觉）；高合规场景对抗例加码。三个关键：(1) **必须标注标准来源不只是标准答案**（否则做不了检索归因）；(2) **警惕 LLM 从文档反生成的"伪问题"偏差**（答案问题字面重叠、检索零难度，要人工清洗）；(3) golden set 是活资产，线上 case 持续补充。
- 陷阱：说"收集问题和答案"就完了，漏掉对抗例、漏掉标准来源、不知道伪问题偏差——说明没真造过。

**6. 组件级指标都涨了，端到端却没涨，可能是什么原因？**
- 框架：环节间有耦合，局部最优 ≠ 全局最优。典型：把 recall@k 调高但塞进更多噪声 chunk，生成端被干扰、faithfulness 反降，端到端没变甚至变差。结论：**改完任何单环必须回看端到端确认净收益**。端到端测结果、组件级测原因，两层都要。

**7. 离线评测和线上 A/B 反馈，各解决什么问题？为什么都要有？**
- 框架：离线 golden set 防退化（已知问题的回归测试，保下限）；线上反馈找盲区（发现 golden set 没覆盖的真实问法）。只有离线 → "刷分但线上没变好"的虚假繁荣；只有线上 → 没稳定基线、改动裸奔。线上信号（点踩等）噪声大，**最大价值是反哺 golden set 而非直接当 KPI**。两轮一起转才是成熟体系。

**8. 为什么 RAGAS 这类框架不能保证你的评测靠谱？**
- 框架：框架解决"指标怎么算"，不解决"评测集怎么造"（golden set 质量它管不了）和"judge 校准"（它内部 judge 也是 LLM、有偏见、会随版本漂移）。框架是放大器——评测集好放大出洞察，评测集烂放大出"精确的垃圾"。生产里要拿人工金标准子集校准框架分数，绝不盲信。

### 二、实战项目增量（P2 收尾：评测集 + 基线，量化每次改动的收益）

在第 8~10 章那套 P2 知识库 RAG 上，补齐评测体系，给系统装上仪表盘。**验收标准（缺一不算过）**：

1. **golden set 成形**：把第 8 章 P2 攒的 ≥20 条扩到 **≥30 条**，且**三类齐全**：典型问题、难例（≥5 条，需跨 chunk 或措辞差异大）、对抗例（≥5 条，含 ≥3 条库外提问/诱导编造）。每条**必须同时标注标准答案和标准来源（哪个 chunk/文档）**。
2. **检索指标自动化**：实现 **Recall@k 和 MRR 的自动计算**（基于标准来源标注，纯规则、不用 judge）。能对当前系统跑出一个数字。
3. **生成指标（LLM-as-judge）**：实现 **faithfulness 判断**——用一个**异源/更强的** judge 模型，给定 `(材料, 答案)`，逐条核对论断有无依据，输出结构化裁决。**judge 不能和你的生成模型同源**（验收会查）。
4. **拒答验证纳入指标**：对抗例里的库外问题，统计"正确拒答率"——答"无法回答"算对，编造算错。
5. **基线报告**：跑一遍当前系统，产出一份**基线数字**（recall@5、MRR、faithfulness、拒答率）。这是你后续所有改动的对比锚点。
6. **量化一次改动**：动一个参数（如 chunk size 或 top-k），**重测，用数字说明收益**（哪个指标涨了多少、有没有指标退化）。这是本章的核心能力——证明你能量化收益，而不是凭感觉。
7. **judge 校准（加分项）**：人工标 5~10 条 faithfulness 金标准，对比 judge 判断，报告一致率。一致率低就说明你的 judge/rubric 需要改。

> 交付物：评测脚本 + 一份基线报告（README 里贴指标数字 + 一次改动的前后对比表）。报告必须**带数字、带因果（这个指标涨是因为改了什么）、带判断（下一步该优化哪层）**。这份"能量化收益的评测报告"是简历上比"我做了个 RAG"含金量高一个档的素材。

参考量级（自检用，非标准答案）：
- **单次全量评测 judge 成本（无优化）**：500 条 × (2K 输入 + 300 输出)。输入 ≈ 100 万 token（$5~10），输出 ≈ 15 万 token（×5 倍单价，$25~50/百万 → 约 $3.75~7.5）。**单次量级在十美元上下（约 $9~18）**，输出虽只占 13% 的 token 量却贡献了约一半的钱——单价 5 倍不对称必须分开算。20 次/天 → **约 $180~360/天**（"几百美元/天"量级）。
- **加 prompt caching**：固定前缀（rubric+few-shot）占 1.2K、可变（材料+答案）占 0.8K。缓存命中后固定部分按 ~0.1× 计价：输入侧从 1M token 全价 ≈ $5~10 降到「1.2K×500 缓存读 ≈ 0.6M×0.1× + 0.8K×500 全价 ≈ 0.4M×1×」≈ 原来的 **~46%**。**对"同一把尺子量很多答案"的评测，缓存是单条成本的主导杠杆。**
- **再叠 Batch**：Batch 是在最终计费 token 上再打 5 折，**和缓存叠乘**（缓存改变 token 的计价系数，Batch 改变总价系数）。缓存后 ~46% 再砍半 → **约原始成本的 ~23%**。日成本从 ~$180~360 量级降到 **几十美元/天**量级。
- **关键认知**：(a) 检索指标是免费的（规则算），评测成本全在 judge 调用——**所以 faithfulness 这种 judge 评测要省着用、能缓存就缓存、能 Batch 就 Batch、能抽样就别每次全量**；(b) **judge 的 rubric 是固定前缀，prompt caching 是这类评测的头号降本手段，且和 Batch 的 50% 折扣叠乘**——不知道这一层的人会把 judge 成本估高一倍还没意识到；(c) CI 场景对反馈时延敏感，Batch 的"1 小时内"可能太慢，得权衡——**高频 smoke 用实时小子集（实时也能吃 caching）、低频全量用 Batch**；(d) golden set 线性增长会让评测成本失控，**分层评测（smoke/full）是必备的成本控制手段**；(e) 别忘了输入输出单价不对称，judge 的输出虽短但 5 倍贵，算成本时分开；(f) **别用 tiktoken 估 Claude 的 token 数**——那是 OpenAI 的 tokenizer，估 Claude 会系统性偏差约 15~20%（代码/非英文更大），把你的成本预算直接带偏；Claude 无公开 tokenizer，要估准得调 `count_tokens` API。

**测试要求**：五条全过。**第 3、4 条（鲁棒性）是核心**——评测器最容易死在"缺标注"和"judge 输出不合法"上，静默当 0 或直接崩都会让你的评测数字失真。**第 5 条**虽然是概念题，但"judge 异源"是 LLM-as-judge 的头号红线，必须想清楚。
**暗坑汇总**：(a) recall 分母为空要跳过、不能当 0 也不能当 1；(b) MRR 排名从 1 算；(c) judge prompt 必须同时给材料和答案、rubric 要对抗冗长/自信偏见、用离散判断；(d) judge 输出 JSON 解析要容错；(e) 聚合时跳过无效样本并报告 valid_count（别拿 None 当 0 拉低均值）。
**参考实现要点**：recall = `len(retrieved[:k] 与 relevant 的交集) / len(relevant)`，relevant 空则 `return None`；MRR 遍历 `retrieved_chunk_ids` 找第一个在 relevant 里的下标 `i`，返回 `1/(i+1)`，找不到返回 `0`；faithfulness 的 judge 输出用 `json.loads` 包 try/except，失败返回无效标记；`evaluate` 里用列表推导过滤掉 None/无效，再求均值，并把 `valid_count` 和 `total` 一起返回。

> 心法收尾：这道题的灵魂不是"会不会算 recall"，是**第 3、4 条鲁棒性 + 第 5 条异源 judge**——它逼你意识到评测器本身也是会骗人的：缺标注静默当 0 会让你误判系统变差、judge 同源会让你误判系统变好。**一个不可信的评测器，比没有评测器更危险——因为它会用精确的数字给你错误的信心。** 把这三条做对的人,才算真懂了"评测也要被评测"。
