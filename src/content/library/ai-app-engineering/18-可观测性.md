> 这章紧接第 17 章（评测）排在阶段四 LLMOps，因为 eval 和 observability 是一对孪生兄弟，分管两个时间维度：**eval 管"上线前这版好不好"（离线、批量、有标注），observability 管"上线后此刻正在发生什么、刚才那次为什么炸"（在线、单次、无标注）。** 你建好了 eval 流水线却没有 tracing，等于装了体检报告却没装监控摄像头——能定期体检，却看不见线上正在出血。这章替你把一个"线上炸了只能靠用户截图和你瞎猜"的系统，变成一个"任一次请求都能下钻到每一步的 prompt、token、延迟、成本、失败原因"的系统。
>
> 替你解决的问题：把"用户说 agent 答错了 / 卡了 / 这个月账单为什么翻倍"这三类目前只能靠复现和拍脑袋的问题，变成**点开一条 trace 就能看到根因**的工程动作。建立一个判断贯穿全章——**trace 之于 LLM 应用 = 分布式链路追踪之于微服务**：单体时代你看一行日志就懂，微服务时代一个请求穿十个服务、不串起来就抓瞎；LLM agent 一次任务穿 prompt→工具→检索→再 prompt→子 agent，调用链比微服务还长还不确定，**不做 tracing 你连"它到底走了哪条路"都不知道，谈何 debug**。
>
> 读法提示：这章每个机制后面紧跟**工程推论**（编号列出，每条都能独立成一道面试题）。第一遍至少把"LLM 难观测四宗罪表""三大平台对照表（Langfuse/LangSmith/Phoenix）""失败分类法七型表""成本归因三维表""SLO 指标分层表"这五张表啃下来。代码题对应主线项目 **P4 续做**——给 P3 的 agent 接上 tracing，建成本与失败看板。
>
> 时效标注：OpenTelemetry GenAI 语义约定的字段名、Langfuse/LangSmith/Phoenix 的具体 API、模型价格按 **2026 初**写，字段名和接口会演进、价格按季度过期，**重点学方法论**；而"调用链必须可追溯""trace 是 debug 的前提不是装饰""成本要按 token 归因到租户/功能""失败要分类不能只看错误率"这些是**可观测性规律**，讲透到骨子里，不会变。
>
> 交叉引用：评测产出的 bad case 怎么从 trace 里捞出来回流进 eval 集，连着**第 17 章**（离线-在线闭环）；多 provider 路由、限流、缓存命中这些 trace 里要打的字段，落在**第 20 章**网关；单次请求的成本怎么算、prompt caching 怎么压成本、Batch 怎么砍价，根子在**第 2 章**API 工程的 token 与计费模型。observability 不产生新的成本结论，它**让第 2 章的成本模型变得可观测、可归因**。

## 一、背景：LLM 应用为什么"天生难观测"

### 1. 问题的根源：传统 APM 的四个前提，LLM 全砸了

后端老手对可观测性不陌生——metrics（指标）、logging（日志）、tracing（链路），三大支柱你都熟。传统 APM（Application Performance Monitoring，应用性能监控）能 work，靠四个隐含前提：**请求是确定的、调用链是短而固定的、成本和请求数线性、失败会抛异常**。LLM 把这四个前提逐个砸碎，所以你不能拿监控一个 CRUD 服务的姿势去监控一个 agent。

| 传统服务的前提 | LLM 应用的现实 | 直接后果 |
|---|---|---|
| **确定性**：同输入同输出，bug 可复现 | 概率采样（第 1 章），同一个 prompt 跑两遍输出都不同 | "复现一下那个 bug"这个 debug 第一步**直接失效**——你重放请求得到的是另一个输出 |
| **短而固定的调用链**：A→B→C 写死在代码里 | agent 的调用链是模型**运行时决定**的：可能 prompt→search→检索→反思→再 search→子 agent→综合，每次路径都不同 | 不把整条链结构化串起来，你根本不知道"这次它走了哪条路、在哪步拐错的" |
| **成本 ∝ 请求数**：QPS 翻倍成本翻倍，可预测 | 成本 ∝ **token 数**，且 token 数随输入长度、输出长度、agent 步数、检索拼了多少上下文剧烈波动 | 同样 1 次请求，成本可以差 100 倍；"为什么这个月账单翻倍"不看 token 归因根本答不出 |
| **失败 = 抛异常**：5xx、超时、栈帧，监控能抓 | 失败常常是 **HTTP 200 但内容错**：幻觉、格式跑偏、答非所问、该调工具没调 | 你的 APM 仪表盘一片绿（没有 5xx），用户却在骂——**最危险的失败是静默的** |

> 新手第一误区：以为"我接了 Datadog/Prometheus，可观测性就有了"。传统 APM 抓的是**基础设施健康度**（CPU、内存、QPS、5xx 率、P99 延迟），这些 LLM 应用也要，但它们**对 LLM 特有的失败完全瞎**。一个幻觉率飙到 30%、但 HTTP 全 200、延迟正常的系统，在你的 Grafana 上是"健康"的。**LLM 可观测性 = 传统 APM（基础设施层）+ LLM 专属观测（语义层、成本层、调用链层）**，后者是新增的、是护城河、也是这章的全部内容。

### 2. 演进：从 `print(response)` 到结构化 trace

- **阶段零：裸 `print` / `logger.info`。** 把 prompt 和 response 打到日志里。单次调用够用，但 agent 多步一跑，日志是一团乱麻的文本流，你无法回答"第 3 步的 prompt 是什么""哪一步最慢""这次总共烧了多少 token"。**无结构 = 不可查询 = 等于没有。**
- **阶段一：手动埋点 + 自定义字段。** 每次 LLM 调用前后记 `{prompt, response, model, input_tokens, output_tokens, latency_ms, cost}` 一条结构化日志，灌进 ES/数仓。能查了，但 agent 的**父子关系**（哪个 LLM 调用属于哪次工具调用、哪次工具调用属于哪个 agent 步骤）你得自己用 `trace_id` 手工串，极易串错、漏埋。
- **阶段二：专用 LLM 可观测平台。** Langfuse / LangSmith / Phoenix 这类把"trace（一次完整任务）→ span（任务里的一步）→ generation（一次 LLM 调用）"的层级模型做成开箱即用，SDK 自动捕获父子关系、token、成本，给你时间线视图能逐步下钻。这是当前主流。
- **阶段三：OpenTelemetry 标准化。** 不绑定单一厂商——用 OTel 的 **GenAI 语义约定**（标准化的 span 属性名）埋点，数据能同时吐给 Langfuse、Phoenix、或你自己的 OTel collector。**埋点一次，后端随便换**。这是 2025-2026 正在成为事实标准的方向。
- **阶段四（当前前沿）：observability + eval 闭环。** 线上 trace 自动跑轻量 judge 给质量打分、自动按失败类型归类，低分 trace 回流进离线 eval 集（第 17 章），异常成本/失败模式触发告警。observability 不再只是"事后查",而是持续给系统体温、并反哺 eval 飞轮。

> 锚点类比（给后端老手）：**LLM 的 trace/span 之于 agent 调试 = OpenTracing 的 trace/span 之于微服务调试**，连模型都几乎一样（trace 包 span、span 可嵌套、每个 span 带 attributes 和 timing）。差别只在 LLM 多了三类专属载荷：**prompt/completion 文本、token/成本、模型行为元数据（用了哪个工具、thinking 花了多少、stop_reason 是什么）**。你已经懂分布式链路追踪，这章的认知成本比你想的低——**把 agent 当成一个"调用链由模型在运行时动态决定"的微服务系统**，剩下的全是熟悉的东西。

### 3. 四个工程推论（本章地基，逐条都是面试题）

#### 推论一：LLM 失败大多是"HTTP 200 的失败"，所以必须观测语义层而非只观测传输层

传统监控盯的是传输层信号（status code、latency、异常）。但 LLM 最高频、最伤业务的失败——幻觉、格式跑偏、答非所问、拒答——**全是 HTTP 200**。你的传输层监控对它们是色盲。所以 LLM 可观测性必须**额外建一层语义观测**：把输出送进轻量 judge / 规则检查，把"内容对不对"变成可监控的信号。

> 面试加分点：被问"LLM 应用监控和普通后端监控有什么不同"，初级答案是"也要监控延迟和错误率"。高级答案是"**普通服务的失败在传输层（5xx/超时），LLM 的失败主要在语义层（200 但内容错），所以除了基础设施监控，必须额外建语义层观测——线上采样跑 judge/规则、把幻觉率/格式合规率/拒答率做成指标**"。能点出"最危险的是静默失败"，说明你真在生产里被这类问题坑过。

#### 推论二：调用链是模型运行时决定的，所以 trace 不是 nice-to-have，是 debug 的唯一入口

普通服务的调用链写在代码里，你读代码就知道流程。agent 的调用链是模型**当场决定**的——它选了哪个工具、检索回了什么、反思后改了什么主意、要不要再来一轮，全是运行时的。**没有 trace，你连"它实际走了哪条路"这个最基本的事实都拿不到**，更别提定位"在哪一步拐错"。这和微服务里"不串 trace 就不知道请求穿了哪几个服务"是同一个道理，但 agent 的链更长、更不确定，所以 trace 的刚需程度比微服务**更高**。

> 心法：**普通服务的 debug 从读代码开始，agent 的 debug 从读 trace 开始。** 代码只告诉你"它可能怎么走"，trace 才告诉你"它这次实际怎么走的"。

#### 推论三：成本不透明源于 token 计费 + 调用链放大，必须按 token 归因到业务维度

第 2 章讲过成本按 token 算。在 agent 场景这事被放大到失控：一次用户请求，背后是 prefill + 多步 decode + 每步把工具结果/检索内容拼进上下文反复重发，token 像滚雪球。**单看"调了几次 API"完全无法预测成本**。所以成本可观测性的核心动作是：把每次 LLM 调用的 token 和成本，**打上业务标签（哪个租户、哪个功能、哪个 agent 步骤）**，做成可按维度切片的归因报表。否则"账单为什么涨"永远是个谜。

> 坑点提示：很多人统计成本只统计到"今天调了 N 次 API、花了 $X 总额"。这个总额**不可行动**——你不知道是哪个租户在烧、哪个功能在烧、是输入贵还是输出贵。account-level 的总成本和 per-tenant/per-feature 的成本归因，差别就像"这个月公司花了 100 万"和"市场部投放花了 60 万、其中信息流占 40 万"——后者才能做决策。

#### 推论四：可观测性数据是 eval 的最佳原料，观测和评测必须闭环而非两套孤岛

线上 trace 里藏着你 eval 集永远造不出来的东西——**真实用户的真实失败**。第 17 章说过 eval 集最高价值的来源是线上日志。observability 系统天然就在收集这些日志。所以正确架构不是"observability 一套、eval 一套、互不相通",而是：**trace 里被 judge 判低分的、被用户点踩的、触发了规则告警的，自动回流进离线 eval 集**——下次回归就能拦住这类失败。观测发现盲区，eval 固化防线，两者咬合成飞轮。

> 面试官想区分什么：你是否把 observability 和 eval 当成一个闭环系统。能说出"线上 trace 的 bad case 是离线 eval 集最值钱的增量来源，observability 不只是事后查问题、更是 eval 飞轮的进料口"——这是区分"接过 tracing 工具"和"设计过 LLMOps 闭环"的分水岭。

## 二、Tracing：trace / span / generation 三层模型

这是本章技术核心。把 agent 的执行过程结构化，靠的就是一套层级化的 trace 模型——和分布式追踪同构，但有 LLM 专属载荷。

### 1. 三层模型：一次任务怎么被拆成可下钻的树

| 层级 | 是什么 | 类比微服务 | LLM 专属字段 |
|---|---|---|---|
| **Trace** | 一次完整的端到端任务（一个用户请求 / 一次 agent 任务） | 一次分布式请求的完整链路 | 总 token、总成本、总延迟、最终 input/output、user_id/session_id/tenant |
| **Span** | 任务里的一个逻辑步骤（一次工具调用、一次检索、一次 agent 决策、一段子流程） | 一个服务内的一个操作 | 步骤类型（tool/retrieval/llm）、输入输出、耗时；可**嵌套**子 span |
| **Generation**（也叫 LLM span） | 一次具体的 LLM API 调用 | 一次 RPC 调用 | model、prompt（messages）、completion、input/output/cache token、stop_reason、thinking token、温度等参数 |

关键是**嵌套**：一个 trace 包多个 span，span 可以包子 span 和 generation。一次 RAG agent 的 trace 树可能长这样：

```
Trace: "用户问：对比 A 和 B 产品的退款政策"   [总 token 18.2k, 成本 $0.21, 延迟 6.3s]
├── Span: planning (agent 决策第一步)
│   └── Generation: 调 LLM 拆解问题            [in 1.2k / out 0.3k token, 0.8s]
├── Span: retrieve "A 退款政策"
│   ├── Generation: embedding query           [in 12 token]
│   └── Span: vector search top-5             [0.1s, 命中 5 条]
├── Span: retrieve "B 退款政策"
│   └── ... (同上)
├── Span: tool_call get_current_policy(B)     [外部 API, 0.4s, 返回 2.1k token]
└── Span: synthesize (综合生成最终答案)
    └── Generation: 调 LLM 拼上下文生成        [in 14.5k / out 0.9k token, 3.1s ← 最慢]
```

有了这棵树，前面那些"传统监控答不了"的问题瞬间可答：**哪一步最慢**？synthesize 的那次 generation（3.1s）。**token 花在哪**？最后一次 generation 输入 14.5k（检索拼了一堆上下文进去）。**它走了哪条路**？planning→双路检索→查实时政策→综合。**在哪步可能拐错**？点开任一 generation 看 prompt 和 completion 原文。

> 工程推论：
>
> **推论五：trace 的价值不在"记录了什么"，在"父子关系结构化了"——能下钻、能定位是哪一步出的问题，才是 trace 区别于一堆散日志的本质。** 同样的数据，平铺成日志你查不动，组织成 trace 树你一眼定位。**埋点时第一优先级是把父子关系（parent span id）串对，其次才是字段全**——关系串错了，再全的字段也拼不回那棵树。

### 2. OpenTelemetry GenAI 语义约定：为什么要用标准字段名

你可以自己定义字段名（`my_prompt`、`tokens_in`…），但有个隐藏代价：**绑死后端**。换可观测平台、接公司统一的 OTel collector，全得改埋点。OpenTelemetry 的 **GenAI 语义约定（semantic conventions）** 就是为了根治这个——它规定了一套**标准化的 span 属性名**，大家都用同一套 key，数据就能在任意兼容后端之间流转。

关键标准字段（按 2026 初的 GenAI semconv，字段名仍在演进，记规律别死背）：

| 标准属性名 | 含义 |
|---|---|
| `gen_ai.system` | 哪家模型供应商（anthropic / openai …） |
| `gen_ai.request.model` | 请求的模型 ID |
| `gen_ai.response.model` | 实际响应的模型 ID（可能因路由/别名不同） |
| `gen_ai.usage.input_tokens` / `output_tokens` | 输入/输出 token 数 |
| `gen_ai.request.temperature` / `max_tokens` | 请求参数 |
| `gen_ai.response.finish_reasons` | 停止原因（对应 Anthropic 的 stop_reason） |
| `gen_ai.operation.name` | 操作类型（chat / tool 等） |

> 落地（重要工程事实，按 2026 初 Anthropic API）：埋 token 字段时**别用 tiktoken 估 Claude 的 token**——会系统性偏差 15-20%（第 1 章、第 2 章）。要精确就读响应里 `usage` 返回的真实值（`usage.input_tokens` / `output_tokens` / `cache_read_input_tokens` / `cache_creation_input_tokens`），这是 API 给你的权威计数，直接打进 `gen_ai.usage.*`。**trace 里的 token 数应当来自响应的 usage 字段，不是你客户端估算的**——前者准、后者错，而成本归因全建在这个数上，错了全盘错。

> 一个常踩的演进坑（按 2026 初 GenAI semconv，仍在动）：**别把整段 prompt/completion 正文塞进 span 的 attribute**。早期约定确实有人这么干（`gen_ai.prompt`、`gen_ai.completion` 当 span attribute），但 semconv 后来把"对话内容"从 span attribute **挪到了独立的 event/log 通道**（span 上只留结构化元数据：模型、token、stop_reason、耗时；正文走单独的内容事件）。原因有两条工程硬约束：一是 **span attribute 不该装大 blob**——一段几 KB 的 prompt 塞进 attribute 会让 span 体积爆炸、拖垮 collector 和后端索引；二是**内容和元数据的隐私/留存策略天差地别**（元数据可全量长留，正文要脱敏+短留，见第四节），物理上拆成两条通道，才能给正文单独挂脱敏 processor、单独设短 TTL、单独做访问控制。**记规律**：span 装"这次调用的结构化事实"，内容正文走可独立治理的事件流——这条不只是字段名变更，是"元数据与敏感正文分离治理"的架构判断，比记住具体 key 名值钱。

> 工程推论：
>
> **推论六：用 OTel GenAI 语义约定埋点，是"埋点一次、后端可换"的解耦投资——不绑死可观测平台。** 自定义字段名省事于一时，绑死于长久。换厂商、对接公司统一 telemetry 管道时，标准字段是你唯一不用重写埋点的资本。**这和"用标准 HTTP 状态码而不是自定义错误码"是同一种工程品味——遵循公共契约换取可移植性。**

### 3. 异步、采样、PII：埋点的三个生产现实

埋点不是"加一行 log"那么天真，三个生产约束必须处理：

- **异步上报，绝不阻塞主链路。** trace 数据的写入（发往可观测后端）必须异步、批量、失败可丢弃。**可观测性是旁路，不能拖慢甚至拖垮业务请求**——你的监控挂了，业务还得正常跑。这条铁律和"日志系统挂了不能让服务挂"一模一样。
- **采样（sampling），别全量留 prompt。** 高流量下，把每次请求的完整 prompt/completion 全存下来，存储成本和隐私风险都炸。常见策略：**metrics（token/延迟/成本）全量采**（便宜、是聚合统计的基础），**完整 prompt/completion 文本按比例采样或只在失败时全量留**（贵、敏感）。这和 APM 里"指标全量、trace 详情采样"的分层一致。但"只在失败时全量留"这句话藏着一个**绕不过去的工程难题**，下面单独讲。
- **PII 脱敏，prompt 里全是用户隐私。** 见第四节，这是单独一大块。

#### head sampling vs tail sampling：为什么"失败时全量留"没那么简单

后端老手对采样不陌生，但 LLM tracing 里有个比传统 APM 更尖锐的版本。问题的核心：**采样决策在什么时刻做？**

- **head sampling（头部采样）**：请求一进来、trace 刚开始时就掷骰子决定"这条留不留"（比如随机留 10%）。**便宜、无状态、零缓冲**——决定丢弃的 trace，后续所有 span 根本不上报。但有个致命缺陷：**决策时你还不知道这条 trace 会不会失败、会不会慢、会不会烧爆 token**。等 agent 跑到第 5 步炸了，你早在第 0 步就把它骰没了。**"失败时全量留"用 head sampling 根本做不到**——失败是终点才知道的事，头部采样是起点做的决定。
- **tail sampling（尾部采样）**：等整条 trace 跑完、拿到全部 span（耗时、token、stop_reason、judge 分），再决定留不留。这才能实现"失败/慢/贵的全留，正常的按比例采"。代价是**必须把一条 trace 的所有 span 先缓冲住**（内存或临时存储里攒齐），直到 trace 结束才能裁决——有状态、要为每条在途 trace 维护缓冲、还要处理"trace 永远不结束"的超时兜底。

这个权衡和你熟悉的分布式追踪**完全同构**（OTel Collector 的 tail sampling processor 就是干这个的），但在 LLM 场景**收益被放大**：传统服务里慢请求/错请求本就稀少且同质，丢几个无所谓；LLM 里**最该留的恰恰是稀有的失败 trace**（幻觉、不收敛、误拒），它们是 eval 集的金矿（推论四），用 head sampling 会把金矿当尾矿丢掉。所以成熟做法往往是**混合**：metrics 全量；trace 详情用 tail sampling——`stop_reason ∈ {max_tokens, refusal}`、延迟/成本超阈值、judge 判低分、用户点踩的**强制全留**，其余正常 trace 按比例（如 5%）采样留作基线。

> 坑点提示：很多人上来就配个"随机留 10%"的 head sampling，自我感觉良好，直到线上炸了去翻 trace——**发现那条偏偏在被丢的 90% 里**。LLM 可观测的第一性原理是"留住该留的失败 trace"，而失败只有 tail 时刻才知道，所以 LLM tracing 天然偏向 tail sampling，哪怕它更重。能讲清"head 决策在起点、失败信息在终点、所以失败优先留必须 tail"，是这一节的题眼。

> 工程推论：
>
> **推论七：tracing 必须异步旁路 + 分层采样，且"失败优先全留"要求 tail sampling 而非 head sampling——失败是 trace 终点才知道的事，头部采样的骰子掷不出它。** 同步埋点会让监控变成业务的延迟来源和故障点；纯 head sampling 会把最该留的稀有失败 trace 随机丢掉。**把"监控本身的可靠性、成本、以及'能不能留住该留的那条'"都纳入设计**，是成熟工程师和"加个 SDK random 留 10% 就以为搞定了"的人的差别。

## 三、平台选型：Langfuse / LangSmith / Phoenix（按 2026 初）

> 时效提示：以下平台的定位、开源状态、API 按 **2026 初**写，会变。**重点学"按什么维度选 LLM 可观测平台"的方法论**，而不是背某家此刻的功能清单。

### 1. 三大平台对照

| 平台 | 定位 / 出身 | 开源 | 最适合 | 主要权衡 |
|---|---|---|---|---|
| **Langfuse** | 开源优先的 LLM 工程平台（tracing + eval + prompt 管理 + 成本） | 是（可自托管） | **数据要落自己手里**（合规/隐私强约束）、想要 tracing+eval 一体、不想绑厂商 | 自托管要自己运维；托管版功能跟开源版有差 |
| **LangSmith** | LangChain 官方配套的可观测/评测平台 | 否（SaaS 为主） | 已重度用 LangChain/LangGraph、想要最丝滑的官方集成 | 闭源、SaaS，数据出域；与 LangChain 生态深绑 |
| **Phoenix**（Arize） | 开源、偏 OpenTelemetry 原生 + 偏排障/评测分析 | 是 | **想走 OTel 标准路线**、重视 trace 分析与 evaluation、要本地起一个轻量实例 | 生态比前两者新；偏分析，prompt 管理等周边弱些 |

### 2. 怎么选：三个决策维度（这才是要学的）

不要问"哪个最好"，要按你的约束打分：

1. **数据要不要落自己手里？** 金融/医疗/政企、或 prompt 里有强 PII → **自托管能力是硬约束**，直接圈定 Langfuse / Phoenix 这类可开源自托管的，LangSmith 的 SaaS 出域模式可能过不了合规。这是第一个、也常常是决定性的筛子。
2. **要不要绑 OTel 标准？** 公司已有统一的 OpenTelemetry telemetry 管道、或你坚持"埋点一次后端可换"（推论六）→ 偏 OTel 原生的方案（Phoenix、或任何吃 OTel 的后端）。怕被某个 LLM 框架绑死，就别选和单一框架深绑的。
3. **你的 agent 跑在什么框架上？** 已重度 LangChain/LangGraph → LangSmith 的自动埋点最省事（一行 callback 全自动）。如果你是 P0~P3 一路手写过来的（本课的路线），你**不依赖任何框架的自动埋点**，反而更该走 OTel 标准 + 开源后端，保持中立。

> 面试加分点：被问"LLM 可观测平台怎么选型"，只会报名字（"我们用 Langfuse"）是 60 分。能说"**先看数据合规要求决定自托管 vs SaaS，再看是否要 OTel 标准化避免绑厂商，最后看现有 agent 框架决定集成成本**——这三个维度排序，合规通常是一票否决项"是 90 分。点出"合规是第一筛子、技术偏好其次",说明你在企业语境里做过选型。

> 国内语境补充（按 2026 初）：国内做 LLM 应用，模型侧常用豆包(字节)/Qwen(阿里)/DeepSeek/Kimi/GLM(智谱)/混元(腾讯)/文心(百度)，可观测侧上述开源方案（Langfuse/Phoenix）同样能用——它们观测的是"你发出去的请求和收到的响应"，**与底层换哪家模型解耦**。这正是可观测层做对了抽象的体现：它站在你的应用和模型 API 之间，不关心 API 后面是 Claude 还是 DeepSeek。换模型不该让你的监控重做。

### 3. 自建 vs 用平台

什么时候该自己搭、什么时候直接上平台？判断原则和所有"自建 vs 买"一样——**核心差异化能力自建，通用能力买**。tracing 的层级模型、时间线 UI、成本聚合，是高度通用的（没有任何公司的"trace 树视图"是差异化竞争力），**直接用现成平台**。只有当你有极特殊的合规要求（连开源自托管都不满足）、或要把 observability 深度嵌进自有的内部平台（如 P5 那种 agent 平台），才考虑在 OTel 标准之上自建采集和存储——但**展示层和分析层仍尽量复用开源**，别从零写 UI。

## 四、Prompt 与输出日志的隐私脱敏

这是最容易在 demo 阶段被忽略、在生产阶段变成事故的一块。**你记录的 prompt 和 completion 里，装满了用户隐私**——姓名、手机号、身份证、病历、对话内容。把它们原样灌进可观测后端，就是在你的监控系统里堆积一座 PII 火药库。

### 1. 为什么 LLM 可观测的隐私问题比传统日志更尖锐

传统服务的日志，你只记结构化字段（`user_id=123, action=purchase`），PII 容易识别和屏蔽。LLM 的 prompt 是**自由文本**——用户可能在任意位置写任意敏感信息（"我叫张伟，身份证 4101...，最近确诊了 XX 病，帮我查保险条款"）。这些信息**无结构、混在正文里、无法靠字段名屏蔽**。而你为了 debug 又特别想把完整 prompt 留下来。这个张力是 LLM 可观测隐私的核心难点。

### 2. 落地手段（分层处理）

- **脱敏（redaction）后再上报。** 在 trace 写入可观测后端**之前**，跑一遍 PII 检测，把检出的手机号/身份证/邮箱/姓名替换成占位符（`<PHONE>`、`<ID>`）。检测手段：正则（手机号、身份证、邮箱这类有固定模式的，便宜可靠）+ 实体识别（姓名、地址这类要模型/NER，贵）。**正则能盖的先用正则，剩下的才上 NER**——和第 17 章"能用规则算的不上 judge"是同一种成本分层思维。
- **分级存储。** metrics（token/延迟/成本，**不含文本**）可以全量长期留；完整 prompt/completion 文本短期保留 + 严格访问控制 + 到期删除。把"贵且敏感的文本"和"便宜安全的指标"分开对待。
- **访问控制与审计。** 谁能看到原始 prompt 必须有权限边界和审计日志（谁在什么时候看了哪条 trace 的原文）——讽刺的是，**你的审计日志系统本身也要被审计**。
- **合规对齐。** GDPR / 个人信息保护法下，用户数据的留存、被遗忘权（删除请求）必须能落地——意味着你的 trace 存储要支持按 user_id 检索并删除。设计存储 schema 时就要想到这点，事后补极痛。

> 坑点提示：最隐蔽的隐私泄漏不是"故意存了 PII"，是**把完整 prompt 打进第三方 SaaS 可观测平台**而没意识到数据已经出域。用 LangSmith 这类 SaaS 时，你的用户 prompt（含 PII）正在流向外部厂商的服务器——这在很多合规场景下是直接违规的。**这也是推论里"数据合规是平台选型第一筛子"的现实由来**：不是技术洁癖，是法务红线。

> 工程推论：
>
> **推论八：prompt 日志默认含 PII，脱敏必须在"写入可观测后端之前"完成，而不是"事后再清洗"。** 一旦原始 PII 进了第三方 SaaS 或长期存储，就已经泄漏了，事后删除只是亡羊补牢。脱敏要做在数据**离开你的可信边界之前**的那一刻。**和"密码不能进日志"是同一条铁律的推广——只是 LLM 里要脱敏的是自由文本里的任意 PII，难度更高。**

## 五、Token 与成本监控：per-tenant / per-feature 归因

第 2 章给了成本的"物理模型"（token 计费、输出≈输入 5 倍的不对称、Batch 半价、caching 降到 0.1×）。这一节讲怎么把那个模型**变得可观测、可归因到业务维度**——让"账单为什么涨"从玄学变成报表。

### 1. 为什么"账户总成本"没用，必须按维度归因

知道"这个月花了 $50k"对你做任何决策都没帮助。能行动的成本数据长这样：

- **按租户（per-tenant）**：哪个客户在烧钱？是不是某个大客户的用量该单独定价、或某个白嫖用户在滥用？这是 SaaS 计费和容量规划的基础。
- **按功能（per-feature）**：是"智能摘要"功能贵还是"对话"功能贵？哪个功能的单位经济性（unit economics）是负的、该优化或砍掉？
- **按模型 / 按 agent 步骤**：成本花在 frontier 模型还是中杯？agent 的哪一步最烧 token（通常是把一堆检索结果拼进上下文的那次 generation）？

把每次 LLM 调用的成本打上这三组标签，你就能切片分析。

### 2. 成本归因三维表

| 归因维度 | 怎么打标签 | 能回答的业务问题 | 对应动作 |
|---|---|---|---|
| **per-tenant** | trace 上带 `tenant_id` | 哪个客户在烧钱？谁该限流/涨价？ | 分层定价、按租户限流（见第 20 章网关）、识别滥用 |
| **per-feature** | trace 上带 `feature` 标签 | 哪个功能不赚钱？哪个该优化？ | 砍负毛利功能、优化高成本功能的 prompt/模型 |
| **per-model / per-step** | generation 上带 model、span 带 step 名 | 钱花在贵模型还是贵步骤？ | 降级模型（frontier→中杯）、压缩高成本步骤的上下文 |

### 3. 成本怎么算准：用 usage 而非估算

单次成本 = `input_tokens × 输入单价 + output_tokens × 输出单价`，再叠加 cache 的折扣项。三个必须做对的点：

- **token 数取响应的 `usage`**，别客户端估（推论六，tiktoken 估 Claude 偏 15-20%）。`usage` 里 `cache_read_input_tokens`（约 0.1× 价）和 `cache_creation_input_tokens`（约 1.25×/2× 价）要**分别按各自折扣计价**，不能和普通 `input_tokens` 混在一起算——算混了你会高估有缓存场景的成本、看不到 caching 省了多少。
- **输出比输入贵约 5 倍**（第 2 章的关键不对称）。所以盯成本要盯输出 token，一个让模型啰嗦输出的改动，比同样多的输入 token 贵得多。
- **把省下的钱也观测出来**：trace 里记录 cache 命中省了多少、Batch 走了没——否则你做了 prompt caching 优化，却无法证明它省了钱，下次没人敢继续投。

> 落地（Anthropic API，按 2026 初）：
> - 验证 caching 是否真命中，看 `usage.cache_read_input_tokens` 是不是 >0。**为 0 说明缓存被静默击穿**——最常见元凶是 system prompt 里掺了时间戳/UUID（前缀匹配，任何字节变化使其后全失效），或工具列表顺序不稳。这一条直接做成监控指标（cache 命中率），降了就告警。
> - 离线批量评测/归档类成本走 **Batch API 拿 50% 折扣**，且 Batch 里也能用 caching，两者**叠乘**。成本看板要能区分"实时调用成本"和"Batch 调用成本"，否则你看不出 Batch 优化的效果。

> 工程推论：
>
> **推论九：成本必须在 trace 层就带上业务标签（tenant/feature/step），归因报表才能驱动决策；只有账户总额的成本监控等于没有。** "总额涨了"不可行动，"X 租户的 Y 功能在 synthesize 步把上下文堆到 15k token、占了本月增量的 70%"才可行动。**能把成本归因到可优化的最小单元并排序，是 FinOps 思维在 LLM 上的落地，也是后端老手最容易出彩的地方——这就是你熟悉的"慢查询定位到具体 SQL"换了个领域。**

## 六、失败分类学：把"它答得不好"拆成可行动的类别

监控不能只有一个"成功率 95%"。那剩下 5% 是什么失败、各占多少、该先修哪个，**一个总数字什么都没告诉你**。和第 17 章 eval 的失败分类法（failure taxonomy）一脉相承——只是这里用在**线上实时观测**，不是离线 eval。

### 1. 七型失败分类表

| 失败类型 | 是什么 | 怎么自动检测（线上） | 对应根因 / 修法 |
|---|---|---|---|
| **幻觉（hallucination）** | 编造事实、引用不存在的来源 | 引用核对（答案里的事实能否在检索原文里找到）、judge 判忠实度 | grounding 不足 → 加强 RAG、强制引用（第 7-11 章） |
| **格式错（format error）** | 该出 JSON 却非法、schema 不符 | `json.loads` + schema 校验（规则类，零成本秒级） | 用 `output_config.format` / `strict` 从源头约束（见下） |
| **超时 / 截断（timeout / truncation）** | 没在时限内返回，或 `max_tokens` 截断 | 监控延迟超阈值、`stop_reason == "max_tokens"` | 调大 max_tokens、streaming、降级更快模型 |
| **工具失败（tool failure）** | 调工具报错、选错工具、参数错 | 监控 tool span 的 error、tool 选择正确性断言 | 改 tool schema 描述、加 error 重试/绕路（第 5 章） |
| **拒答（refusal）** | 模型拒绝回答（安全触发） | `stop_reason == "refusal"`、检测拒答话术 | 区分"该拒"（正确）vs"误拒"（过度保守，要调 prompt） |
| **答非所问 / 指令不遵循** | 答案跑题、没按要求做 | judge 判相关性/指令遵循 | prompt 不清、上下文太长冲淡指令 |
| **不收敛（agent 专属）** | 循环调工具、烧钱、走错不回头 | 监控单任务步数/token 超预算、循环检测 | 加终止条件、step/token 预算上限（第 6 章 agent） |

### 2. 检测分层：能用规则的零成本先抓，剩下的才上 judge

七型里有一半能用**零成本的规则/信号**实时抓：

- `stop_reason` 这一个字段就免费告诉你三件事——`max_tokens`（截断）、`refusal`（拒答）、`tool_use`（在调工具）。**把 stop_reason 做成监控维度，是性价比最高的失败观测**。
- 格式错：`json.loads` + schema 校验，零成本秒级。
- 超时/截断：延迟阈值 + stop_reason，零成本。
- 不收敛：步数/token 计数超预算，零成本。

剩下需要语义判断的（幻觉、答非所问），才上**采样 judge**——别全量跑 judge 监控（贵），按比例采样 + 失败优先采。

> 落地（Anthropic API，按 2026 初）：
> - **格式错这一整类，可以在源头消灭而不是事后监控**：用 `output_config.format` 传 `json_schema` 约束响应格式、工具参数用 `strict: true` 做 schema 校验、Python SDK 用 `messages.parse()` 自动校验返回 `parsed_output`（旧 `output_format` 顶层参数已废弃）。**能用 API 保证掐掉的失败类，就不必再花观测预算去抓它**——这和第 17 章"用平台能力消灭一整类失败"是同一招，省下的观测预算去盯真正难抓的幻觉。
> - **拒答有结构化信息可抓**：`stop_reason == "refusal"` 时，响应带结构化 `stop_details`（category、explanation），直接打进 trace，比你用关键词猜"这是不是拒答了"准得多。

> 工程推论：
>
> **推论十：失败监控的产出必须是"失败分类分布"而非单一成功率——告诉你失败集中在哪型，那才是下一步该投的地方。** "成功率 95%"不可行动；"5% 失败里，50% 是工具失败、30% 是幻觉、20% 是格式错"——立刻知道先修工具调用。**这和第 17 章离线 eval 的失败归因是同一种思维用在了线上实时层**：能把失败归因到类别并排序，是可观测工程师和"只会看一个大盘成功率"的人的分水岭。

## 七、告警与 SLO：从"被动救火"到"主动发现"

trace 和指标是"出事后能查"，SLO 和告警是"出事时/出事前能被叫醒"。这是可观测性的主动面，也是 oncall 体系的基础。

### 1. SLO 指标分层：别只盯一个大盘

SLO（Service Level Objective，服务等级目标）要分层定义，**不同层服务不同决策者**：

| 层级 | 指标举例 | 服务谁 / 用来干什么 |
|---|---|---|
| **基础设施层** | 可用性、5xx 率、P50/P99 延迟、TTFT | SRE / oncall——系统活着吗、快不快 |
| **LLM 质量层** | 幻觉率、格式合规率、拒答率、judge 平均分 | 应用工程师 / PM——答得对不对、好不好 |
| **成本层** | 单请求成本、日成本、per-tenant 成本、cache 命中率 | 工程 + 财务——烧钱可控吗、归因清楚吗 |
| **业务层** | 任务完成率、点赞率、转人工率、留存 | PM / 业务——用户真的被满足了吗 |

> 锚点类比（给后端老手）：这套分层和你熟悉的 SLI/SLO/error budget 完全同构——**LLM 质量层就是 LLM 应用新增的那一类 SLI**。"幻觉率 ≤ 2%"就是一个 SLO，破了就消耗 error budget，连续破就该冻结发布、回头修质量。**eval 之于 AI 系统 = error budget 之于 SRE**（第 17 章心法）在这里落到了具体指标上。

#### 延迟不是一个数：LLM 的延迟是三个 SLI

基础设施层那行写着"P99 延迟、TTFT"，但很多人把它们当成一回事，只在 trace 里记一个 `latency_ms`（整次调用从发出到收完的总时长）。**对流式（streaming）输出的 LLM 应用，这个单一数字会让你对最伤体验的指标瞎眼。** LLM 的延迟至少是三个独立的 SLI，各自对应不同的用户感受：

| 延迟指标 | 是什么 | 用户感受 / 为什么单独要 |
|---|---|---|
| **TTFT**（time to first token，首 token 延迟） | 从请求发出到**第一个 token** 吐出来 | 用户盯着光标转圈的"等待感"全在这——TTFT 800ms 和 3s 是"跟手"与"卡死"的差别。**这是聊天 UI 最该盯的延迟指标**，但它被总时长完全掩盖 |
| **inter-token latency / TPS**（token 间延迟 / 每秒吐字） | token 之间的平均间隔，或每秒吐多少 token | 决定回答"刷得顺不顺"。TTFT 很快但吐字一顿一顿，体验照样差 |
| **总生成时长**（total latency） | 从发出到最后一个 token | 决定一次完整回答要等多久，长输出时才显著。**但它和 TTFT 解耦**：一个 TTFT 200ms、吐了 4000 token 的长回答，总时长可能 20s，可用户从第 200ms 就在读了，体验是好的 |

关键工程后果：**Generation span 不能只记一个 `latency_ms`，要把 TTFT 单独打成字段**（streaming 下，首个 `content_block_delta`/首 token 到达的时刻减去请求发出时刻）。只记总时长，你会做出灾难性误判——看到"P99 总延迟 18s"以为系统慢爆了，慌忙降级模型，**其实 TTFT 一直是 300ms、用户体验毫无问题，慢的只是长输出的总时长**。反过来，TTFT 悄悄从 400ms 劣化到 2s（比如某次路由把请求打到了过载节点），总时长却因为输出变短而没怎么变，你的总延迟监控**一片绿，用户却在抱怨变卡了**。

> 锚点类比（给后端老手）：这和你区分"接口 P99"与"首字节时间（TTFB）"是同一种素养——**TTFT 就是 LLM 版的 TTFB**，是感知延迟的真正入口。后端做流式接口你早就懂"别只测端到端总时长"，搬到 LLM 上不过是把 TTFB 改叫 TTFT。

> 面试加分点：被问"LLM 应用的延迟怎么监控"，只答"测 P99 延迟"是 60 分。能说"**LLM 延迟至少拆三个 SLI——TTFT（感知等待，聊天 UI 第一指标）、token 间延迟（流畅度）、总生成时长（解耦于 TTFT，长输出才显著）；trace 的 generation span 必须单独记 TTFT，只记总时长会对最伤体验的等待感瞎眼**"，并点出"TTFT 是 LLM 版 TTFB"，说明你在生产里盯过流式体验。

### 2. LLM 特有的告警，以及怎么不被它淹没

除了传统告警（延迟、错误率飙升），LLM 应用要额外加几条：

- **成本异常**：日成本 / per-tenant 成本突增（某个 bug 让 agent 无限循环、或某租户滥用），**这是最该优先建的 LLM 告警**——成本失控比慢几百毫秒可怕得多，能一夜烧穿预算。
- **质量塌方**：采样 judge 分数 / 某类失败率突破阈值（换了 prompt 后幻觉率飙升）。
- **cache 命中率掉零**：前缀被静默击穿，成本即将翻几倍的前兆。
- **拒答率异常**：误拒突增（模型更新或 prompt 改动导致过度保守）。

> 坑点提示：LLM 系统的指标天生有**采样噪声**（概率输出，幻觉率天然抖动几个点）。告警阈值设太敏感 → 天天误报、被 oncall 拉黑、真出事时没人看（告警疲劳）；设太松 → 漏真问题。正确做法：**用一段时间窗的移动平均/分位数、而非单点触发告警**，并区分"突变"（值得叫醒人）和"在噪声带内波动"（不值得）。这和第 17 章"区分真改善 vs 采样噪声"是同一个统计素养——**LLM 工程里到处要和噪声打架**。

> 工程推论：
>
> **推论十一：LLM 告警里成本异常优先级最高，且所有阈值要按"时间窗聚合 + 突变检测"设，而非单点触发——否则被采样噪声制造的误报淹没。** 成本能一夜烧穿，质量塌方影响一片用户，这两类比延迟抖动值得优先叫醒人。而告警的设计哲学是**宁可漏报一次噪声内波动，不可天天误报训练 oncall 忽略你**——一个被狼来了喊废的告警系统，等于没有。

## 八、线上问题如何复现：概率系统的"复现"是什么

传统 debug 第一步"复现一下"，在 LLM 这里**直接失效**——概率采样，你重放同一个请求得到的是另一个输出（第 1 章）。所以 LLM 的"复现"不是"重跑出一模一样的结果",而是另一套方法。这是本章把前面所有工具串起来用的实战收尾。

### 1. 为什么不能简单"重跑"

`temperature` 即使设 0 也不保证逐 token 一致（不同 batch、不同后端状态都可能让输出漂移），何况 frontier 模型（按 2026 初，Opus 4.7/4.8、Fable 5 已移除 temperature/top_p/top_k，行为控制主通道迁到自然语言+思考预算）你根本调不了采样超参。**"复现"在概率系统里要重新定义**：不是"重现完全相同的输出",是"还原出当时的完整上下文，理解它为什么会产出那一类错误，并能稳定地让这类错误重现"。

### 2. 靠 trace 做"上下文级复现"

正确的复现流程，全靠你前面建的 trace：

1. **从告警/用户反馈定位到那条 trace。** 用户点踩、或告警触发，带着 trace_id（所以 trace_id 要回传到前端/返回给用户反馈通道）。
2. **下钻看完整执行树。** 那次它走了哪条路、每步 prompt 是什么、检索回了什么、哪步 generation 出的错——trace 树一目了然（第二节那棵树）。这一步往往**直接看出根因**：检索没召回对的文档、prompt 里上下文太长把指令冲淡了、工具返回了脏数据。
3. **抽取出最小复现 case。** 把那次的输入 + 关键上下文固化成一个 case，**多跑几次看这类错误的稳定性**——N=1 可能是采样运气，跑 ≥3 次才能区分"偶发噪声"和"稳定 bug"（这正是 know 里反复强调的：≥3 个独立 sample 才能区分 noise 和 signal）。
4. **回流进 eval 集。** 这个 case 进离线 eval 集（第 17 章），修完用它验证、且永久防回归。**至此 observability 和 eval 闭环了**（推论四）。

> 面试加分点：被问"线上 LLM 偶发答错，你怎么排查和复现"，初级答案是"我重跑一下看看"——立刻暴露不懂概率系统（重跑得到的是另一个输出）。高级答案是"**LLM 没有逐 token 复现，复现是『靠 trace 还原完整上下文 + 多次重跑判稳定性』**：从 trace 下钻看那次实际走的路和每步 prompt/检索，定位是哪步的根因，抽成最小 case 跑 ≥3 次区分噪声 vs 稳定 bug，再回流进 eval 集防回归"。能说出"复现的对象是上下文和失败模式、不是逐字输出"，是这一题的题眼。

> 工程推论：
>
> **推论十二：LLM 的"复现"不是重跑出相同输出，是用 trace 还原完整上下文 + 多次重跑判稳定性——没有 trace，概率系统的 debug 寸步难行。** 这把全章串起来了：trace 是复现的前提，复现产出最小 case，case 回流进 eval，eval 防回归。**observability 不是孤立的"事后查"工具，它是 debug 闭环和 eval 飞轮的起点。** 谁没建 trace，谁的 LLM 线上 debug 就只能靠用户截图 + 瞎猜。

> 心法（本章收尾）：**传统服务的 debug 从"复现 bug"开始，LLM 服务的 debug 从"读 trace 还原现场"开始——因为前者的 bug 是确定的、可重放的，后者的 bug 是概率的、只能从当时的上下文里理解。** 你为微服务装链路追踪，是因为请求穿了太多服务、不串起来看不懂；你为 LLM agent 装 tracing，是因为调用链是模型当场决定的、不记录下来你连"它干了什么"都不知道。**不会量化的工程师在生产里裸奔（第 17 章），不会观测的工程师在生产里盲奔——眼睛蒙着,还在烧着按 token 计费的钱。**

## 九、章末五件套

### 高频面试题（10 道，附答题框架）

1. **LLM 应用为什么比传统服务难观测？至少说三点。**
   - 框架：四宗罪任挑三点——(1) **非确定性**：概率采样，bug 不可重放复现；(2) **调用链运行时决定**：agent 的路径是模型当场选的，不串 trace 不知道它走了哪条路；(3) **成本不透明**：按 token 计费 + 调用链放大，同 1 次请求成本差百倍；(4) **失败静默**：最伤的失败（幻觉/格式错/答非所问）是 HTTP 200，传输层监控全瞎。
   - 陷阱答案：只说"也要监延迟和错误率"——这只覆盖了传统 APM 那层，没点出 LLM 特有的语义层/成本层/调用链层观测。必须说出"传统 APM 对 LLM 的静默失败是色盲"。

2. **trace / span / generation 三层模型分别是什么？为什么要分层而不是打平日志？**
   - 框架：trace = 一次完整任务、span = 任务里一步（可嵌套）、generation = 一次具体 LLM 调用（带 prompt/completion/token）。分层的本质价值是**父子关系结构化 → 能下钻定位是哪一步出问题**（推论五），打平的日志查不动、拼不回执行树。
   - 加分：能类比"trace/span 之于 agent = OpenTracing 之于微服务"，并点出 LLM 多了三类专属载荷（prompt 文本、token/成本、行为元数据如 stop_reason/用了哪个工具）。

3. **OpenTelemetry GenAI 语义约定是什么？为什么不自己定义字段名？**
   - 框架：OTel 规定的一套标准化 span 属性名（`gen_ai.request.model`、`gen_ai.usage.input_tokens` 等），让 telemetry 数据能在任意兼容后端间流转。不自定义的理由：**埋点一次、后端可换，不绑死可观测平台**（推论六）。
   - 加分：类比"用标准 HTTP 状态码而非自定义错误码"——遵循公共契约换可移植性。能补"token 字段要取响应 usage 而非 tiktoken 估算（Claude 偏 15-20%）"更佳。

4. **LLM 可观测平台（Langfuse/LangSmith/Phoenix）怎么选型？**
   - 框架：三个维度排序——(1) **数据合规**：要自托管/数据不出域 → Langfuse/Phoenix（开源），LangSmith 的 SaaS 可能过不了合规，这常是一票否决项；(2) **要不要 OTel 标准化**：避免绑厂商 → 偏 OTel 原生方案；(3) **现有 agent 框架**：重度 LangChain → LangSmith 集成最省事。
   - 陷阱：只报一个名字。必须给出"按什么维度选"且点出"合规是第一筛子"。加分：手写过 agent（无框架自动埋点）的项目更该走 OTel + 开源保持中立。

5. **LLM 的 prompt 日志有什么隐私风险？怎么脱敏？**
   - 框架：prompt 是自由文本，PII（手机/身份证/病历/姓名）无结构地混在正文里，无法靠字段名屏蔽。脱敏要点：**在写入可观测后端之前**做 redaction（正则盖手机/身份证/邮箱，NER 盖姓名/地址），分级存储（指标全量留、文本短期留+访问控制+到期删），合规支持按 user_id 删除（被遗忘权）。
   - 陷阱：说"事后清洗日志"——一旦 PII 进了第三方 SaaS 就已泄漏，必须做在数据离开可信边界**之前**（推论八）。最隐蔽的泄漏是把含 PII 的 prompt 打进 SaaS 平台没意识到数据出域。

6. **怎么把 LLM 成本监控做得"可行动"？只统计账户总成本有什么问题？**
   - 框架：账户总额不可行动（不知道谁在烧、哪个功能烧、输入贵还是输出贵）。要按三维归因——**per-tenant**（谁该限流/涨价）、**per-feature**（哪个功能不赚钱）、**per-model/step**（贵在哪个模型/哪步）。token 取响应 usage（别估），cache token 分折扣计价，输出比输入贵 5 倍要盯输出。
   - 加分：点出"把省的钱也观测出来"（cache 命中省了多少、Batch 走了没），否则做了优化无法证明效果。类比"FinOps / 慢查询定位到具体 SQL"。

7. **LLM 应用的失败分类有哪些？为什么不能只看一个成功率？**
   - 框架：七型——幻觉、格式错、超时/截断、工具失败、拒答、答非所问、不收敛（agent）。单一成功率不可行动，要的是**失败分类分布**告诉你失败集中在哪型、先修哪个（推论十）。检测分层：stop_reason/json 校验/步数预算等零成本规则先抓，幻觉/答非所问才上采样 judge。
   - 加分：点出"格式错可用 `output_config.format`/`strict` 从源头掐掉、不必事后监控"；"stop_reason 一个字段免费给你截断/拒答/工具调用三个信号，是性价比最高的失败观测"。

8. **给 LLM 应用定 SLO 和告警，和传统服务有什么不同？**
   - 框架：SLO 分四层——基础设施（可用性/延迟/TTFT）、LLM 质量（幻觉率/格式合规/拒答率/judge 分）、成本（单请求/日/per-tenant 成本、cache 命中率）、业务（完成率/点赞/转人工）。LLM 特有告警：**成本异常优先级最高**（能一夜烧穿）、质量塌方、cache 命中率掉零、误拒突增。
   - 陷阱：告警阈值设单点触发——LLM 指标有采样噪声，会天天误报导致告警疲劳。正确：时间窗聚合 + 突变检测，宁可漏报噪声内波动也别天天误报（推论十一）。能类比 SLI/SLO/error budget 且把"LLM 质量层"指认为新增 SLI 类别，是题眼。
   - 进阶加分：基础设施层那个"延迟"对流式应用要拆成三个 SLI——**TTFT（首 token，聊天 UI 第一感知指标，= LLM 版 TTFB）、token 间延迟（流畅度）、总生成时长（解耦于 TTFT，长输出才显著）**；只记一个 `latency_ms` 会对最伤体验的等待感瞎眼（七·1）。能主动拆这三个，立刻和"只会答 P99"的人拉开。

9. **线上 LLM 偶发答错，你怎么排查和复现？**
   - 框架：**LLM 没有逐 token 复现**（概率采样，重跑得到另一个输出，temperature=0 也不保证；frontier 模型连采样超参都移除了）。复现 = 靠 trace 还原完整上下文：定位 trace_id → 下钻看那次实际走的路和每步 prompt/检索/generation → 直接看出根因（检索没召回/上下文冲淡指令/工具脏数据）→ 抽最小 case 跑 ≥3 次区分噪声 vs 稳定 bug → 回流 eval 集防回归。
   - 陷阱：答"重跑一下看看"——暴露不懂概率系统。题眼：复现的对象是**上下文和失败模式**，不是逐字输出；且 N=1 命中可能是运气，≥3 sample 才能区分 noise 和 signal。

10. **observability 和 eval 是什么关系？该是两套系统还是一套？**
    - 框架：必须闭环、不是两套孤岛。observability 天然收集线上真实 trace，其中被 judge 判低分/被用户点踩/触发规则告警的 bad case，是离线 eval 集**最值钱的增量来源**（第 17 章 eval 集最高价值来自线上日志）。架构：trace 的 bad case 自动回流 eval 集，eval 固化防线、observability 发现盲区，咬合成飞轮（推论四、十二）。
    - 加分：点出 eval 管"上线前好不好"（离线/批量/有标注）、observability 管"上线后此刻在发生什么/刚才为什么炸"（在线/单次/无标注），两个时间维度互补。能说出这个"时间维度分工"，说明你把 LLMOps 当系统而非工具看。

### 实战项目增量（P4 续做：给 P3 的 agent 接 tracing + 建成本与失败看板）

**目标**：给 P3 的 research agent 接上全链路 tracing，建一个能下钻、能按维度归因成本、能看失败分类分布的看板。这是把"demo 能跑"推向"能上线、能 oncall、能被审计"的关键一步。

**验收标准（缺一不算过）**：
1. **全链路 trace**：agent 的一次任务能下钻成 trace→span→generation 树，每个 generation 带 model、prompt、completion、input/output/cache token、延迟、stop_reason。任一次请求可点开看"它实际走了哪条路、哪步最慢、token 花在哪"。
2. **token/成本取响应 usage 而非估算**：成本 = input×单价 + output×单价 + cache 折扣项，cache_read/cache_creation token 分别按 0.1×/1.25× 计价；**禁止用 tiktoken 估 Claude**。报表能证明 caching 省了多少（有则）。
3. **成本按维度归因**：trace 带 tenant/feature/step 标签，看板能按这三维切片，回答"哪个功能/哪步最烧 token"。
4. **失败分类看板**：把失败分成 ≥5 型（幻觉/格式错/超时/工具失败/拒答/不收敛），用零成本信号（stop_reason、json 校验、步数超预算）+ 采样 judge 检测，看板出**失败分类分布**而非单一成功率。
5. **SLO + 告警**：定义至少 3 条 SLO（含一条成本类如日成本上限、一条质量类如某失败率阈值），其中成本异常告警用时间窗聚合而非单点触发。
6. **PII 脱敏**：写入可观测后端前，对 prompt 里的手机号/身份证/邮箱做正则脱敏，证明原文 PII 不进存储/不出域。
7. **一次真实排障演练**：构造一个偶发失败（如某类 query 检索召不回），从 trace 下钻定位根因、抽成最小 case 跑 ≥3 次判稳定性、回流进 P4 的离线 eval 集——**走不通这条"trace→定位→case→回流"闭环的不算过**。

**加分项**：用 OTel GenAI 语义约定埋点（标准字段名，证明能换后端）；接一条"用户点踩 → 带 trace_id → 自动进 bad case 队列 → 回流 eval"的闭环通道；cache 命中率做成监控指标并设掉零告警。

### 设计题（考权衡判断）

> 你的 LLM 客服系统服务 50 个 B 端租户、日均 200 万次调用，含强 PII（用户对话里有姓名/手机/订单/有时病历）。法务要求用户数据不出域、可按用户删除（被遗忘权）。请设计这套系统的可观测性方案，覆盖：tracing 怎么埋（含父子关系、采样策略）、PII 怎么脱敏、成本怎么按租户归因、可观测平台怎么选、哪些数据能留多久。并回答：什么情况下你会拒绝某个看似方便的方案。

考点：(1) 平台选型是否把**数据合规当第一筛子**——强 PII + 不出域，直接排除 SaaS（LangSmith），圈定可自托管的开源方案（Langfuse/Phoenix）；(2) PII 脱敏是否做在**写入后端之前 / 离开可信边界之前**，而非事后清洗；(3) 采样策略是否分层（指标全量、文本采样或仅失败全留），平衡存储/隐私 vs debug 需要；(4) 成本是否带 tenant 标签做 per-tenant 归因（50 个租户的计费/限流/识别滥用全靠它）；(5) 存储 schema 是否支持**按 user_id 检索并删除**（被遗忘权要在设计期就埋进 schema，事后补极痛）；(6) 是否敢拒绝——比如"直接上 LangSmith SaaS 最省事"这个方案，要敢于因为"用户 PII 出域违反法务红线"而否决，即使它集成最快。**敢说"这个方便方案不能用"并讲清合规理由，是判断力的体现。**

### 系统题（含量级估算）

> 你要给一个日均 100 万次 agent 调用的系统建可观测性。每次 agent 任务平均 5 步、每步一次 LLM 调用，平均每次调用 input 3k / output 0.5k token。需求：(a) 估算每天产生多少条 trace、多少个 generation、总 token 量级；(b) 如果完整 prompt/completion 全量存储，估算日存储增量；改成"指标全量 + 文本 10% 采样 + 失败全留"后省多少；(c) 如果对线上 5% 流量做采样 judge 质量监控，judge 每天调多少次、用 Batch 大致多少钱（自设 judge token 量级和单价）；(d) 这套监控怎么和离线 eval 闭环。

参考量级（自己的假设要写出来）：(a) 100 万 trace/天；每 trace 5 个 generation → 500 万 generation/天；每 generation ~3.5k token（in 3k + out 0.5k）→ 总 ~175 亿 token/天。(b) 全量存文本：500 万 generation × 平均 ~3.5k token，按 ~4 字节/token 粗估 ≈ 500 万 × 14KB ≈ **~70GB/天纯文本**（还没算 trace 结构/索引开销），一个月 2TB+，存储和隐私都吃不消。改成"指标全量（每 generation 几十字节的数值，~500万×~100B ≈ 0.5GB/天）+ 文本 10% 采样 + 失败全留（设失败 5% 全留）"→ 文本量降到约 (10% + 5%) ≈ **原来的 ~15%，~10GB/天**，省约 85%。(c) 100 万 × 5% = 5 万次/天 judge；设每次 judge 输入 ~2k token（材料+答案+rubric）、输出 ~200 token，Balanced 档输入 ~$3/输出 ~$15 每百万 token，单次 ≈ 2000×3/1e6 + 200×15/1e6 ≈ $0.006+$0.003 = $0.009，5 万次 ≈ $450/天；走 **Batch 砍半 ≈ $225/天**；rubric 固定前缀用 **prompt caching 让那 ~2k 输入降到 0.1×**，再砍一截。(d) judge 判低分 + 用户点踩的 trace，带 trace_id 回流进离线 eval 集，下次回归就能拦——闭环。考点是把 trace 量、token 量、采样省的存储、judge 的 Batch/caching 折扣、回流闭环串成一笔明白账，且**意识到全量存文本不可行、必须分层采样**。

1. **三类输入 token 分别计价。** 普通 input、cache_read（~0.1×）、cache_creation（~1.25×）单价不同，混用一个单价会让有缓存的场景成本算错、彻底看不出 caching 的价值——而"证明 caching 省了钱"正是成本观测的关键产出。
2. **规则只抓零成本失败信号，别在这里调 judge。** stop_reason、span.error、json 校验是免费的实时信号；幻觉/答非所问要 judge，是另一条采样路径，别混进这个同步聚合器里（会变慢变贵）。报表要出**失败分布**不是成功率。
3. **脱敏在写入之前。** prompt 存进 Generation/上报后端**之前**就要 `redact()`，一旦原文 PII 进了存储/出了域就已泄漏，事后清洗是亡羊补牢（推论八）。
4. **token/成本聚合必须递归子 span。** 父子嵌套是三层模型的灵魂，只统计顶层 span 会漏掉深层工具调用/子 agent 的 token——而那往往正是烧钱大户。
5. **成本必须按业务维度归因，不能只给总额。** `cost_by_tenant`/`cost_by_feature` 才可行动（谁该限流、哪个功能不赚钱），一个总成本数字驱动不了任何决策（推论九）。
6. **token 数应来自响应 usage，不是 tiktoken 估算。** 本题用结构体直接传 token 数模拟，真实项目里这些数必须取 API 响应的 `usage` 字段——用 tiktoken 估 Claude 偏 15-20%，而成本全建在这个数上，错了全盘错。
