> 为什么这章排在这：你已经学完工具调用（第 5/6 章）——模型能出意图、你能执行、结果能回喂。但那还是**你在写循环**。这一章要把"循环的控制权"从你手里交给模型一部分，让系统能自己决定"下一步做什么、做几步、什么时候停"。这是整个阶段三的地基，也是"AI 应用"和"AI agent"的真正分水岭。学员到这一步要解决的核心问题是：**Agent 不是更聪明的 Chatbot，而是一个会自己烧你钱、会自己走错路、还自信满满的不收敛循环——你怎么把它焊成一个能上线的东西。**

阶段定位：阶段三 Agent · 第一章。对应主线项目 **P3 起步**：手写一个单 agent 骨架（ReAct loop + 工具 + 停机保护）。

一句话先把营销话术撕掉：**Agent = LLM + 工具 + 循环 + 停机条件。** 前三个你这一章会写出来，第四个（停机条件）才是把它和"玩具 demo"区分开的东西。新手第一误区就是盯着前三个看，觉得"不就是个 while 循环嘛"，然后上线被不收敛的 agent 烧掉三千刀 token。

## 一、背景：为什么需要 Agent，它到底解决什么问题

### 1.1 从一个具体场景切入

你的后端直觉里，处理一个请求的标准姿势是：**流程是你预先写死的**。"查用户 → 查订单 → 查库存 → 拼返回"，这条路径在代码里是确定的，分支也是你穷举好的。这套打法的前提是：**你在写代码时，就能把所有路径枚举完。**

现在来个需求：**"帮我把这份产品设计文档变成一个能跑的 PR。"**

你试着用老姿势枚举路径——你枚举不出来。因为：

- 这份文档里要改几个文件？不知道，得读了才知道。
- 改 A 文件会不会牵连 B 文件？不知道，得分析依赖。
- 写完跑测试挂了怎么办？得看报错决定是改代码还是改测试。

**这类任务的特征是：下一步做什么，取决于上一步的结果，而上一步的结果你事先不知道。** 路径不是预先确定的，是运行时一步步"长"出来的。

这就是 Agent 存在的理由：**当任务的执行路径无法预先枚举、必须根据中间结果动态决定时，你需要一个能在运行时自己规划路径的东西。** LLM 恰好能做"看到当前状态 → 决定下一步"这个判断。把这个判断塞进一个循环里，就是 Agent。

### 1.2 自主性谱系：不是"是不是 Agent"，是"自主到什么程度"

这是本章第一个、也是最重要的认知框架。"要不要用 Agent"是个**伪二元问题**。真实世界是一条连续的自主性谱系，从"你全控"到"模型全控"：

| 档位 | 谁决定控制流 | 形态 | 例子 | 你的后端类比 |
|---|---|---|---|---|
| L0 单次调用 | 你 | 一次 prompt → 一次响应 | 翻译、摘要、分类 | 一个无副作用的纯函数 |
| L1 链式 Chain | 你 | 固定的多步流水线，步骤写死 | "先抽取实体 → 再查库 → 再润色" | 一个写死的 pipeline |
| L2 Router 路由 | 你定分支，模型选 | 模型在你给的几条路里选一条 | 客服分流到"退款/查询/投诉" | switch-case，case 由模型选 |
| L3 Workflow 编排 | 你定图，模型填节点 | 你画好 DAG，模型执行每个节点 | 见第 14 章 | 编排引擎（你定义的 DAG） |
| **L4 单 Agent** | **模型决定控制流** | **模型在循环里自己决定下一步调什么工具、停不停** | research agent、coding agent | **一个你不写循环体的 while 循环** |
| L5 Multi-Agent | 多个模型互相决定 | 多 agent 协作分工 | 见第 15 章 | 多个自主服务互相调用 |

**这张表你要会反向背**：给你一个需求，你能说出它该落在哪一档，以及为什么不该再往上一档走。

工程推论（每条都能独立成一道面试题）：

1. **自主性每往上一档，能力上限抬高，但可控性、可预测性、可调试性同步下降。** 这是一条单调的 trade-off 曲线。L4 能干 L1 干不了的活，但 L1 永远不会循环烧钱、永远可复现，L4 两条都不保证。
2. **绝大多数生产需求的最优解在 L0–L3，不在 L4。** 工程师的成熟度，恰恰体现在"能压住手不上 Agent"。能上 L1 解决的，上了 L4 就是给自己埋雷。
3. **"我们做了个 Agent"在简历和 PR 里都不是加分项，"我们判断这个场景不需要 Agent、用 Workflow 更稳"才是。** 判断力 > 实现力。
4. **同一个产品里，不同功能可以落在不同档位。** 不是整个系统"是不是 Agent"，是每个功能各自选档。把"查订单状态"做成 L4 是脑子进水。

记住一句话：**自主性是成本，不是功能。** 你为更高的自主性付出的是可控性，只有当任务复杂到低档位真的覆盖不了时，这笔买卖才划算。

### 1.3 何时该用 Agent：四问决策框架（本章最高频考点）

在你动手写循环之前，先过这四问。**任何一问答"否"，就退回更简单的档位（L0–L3），不要上 Agent。** 这是 Anthropic、OpenAI 工程团队公开反复强调的判断框架，面试高频。

> **四问：复杂度 / 价值 / 可行性 / 纠错成本**

| 问 | 问什么 | 答"否"意味着 | 反例（答否，别上 Agent） |
|---|---|---|---|
| **① 复杂度** | 任务是不是多步、且无法事先把流程写死？ | 路径能枚举 → 用 Workflow（第 14 章） | "从 PDF 抽标题"——一步搞定，单次调用 |
| **② 价值** | 这个任务的产出，值不值更高的延迟和 token 成本？ | 不值 → 用便宜的单次调用 | "给评论打情感标签"——一次 Haiku 调用 0.001 刀，上 Agent 是杀鸡用牛刀还更慢 |
| **③ 可行性** | 当前模型在这类任务上，能力到位吗？ | 不到位 → 别硬上，等模型进步或换方案 | 让 Agent 操作一个没有任何 API、纯私有协议的老系统 |
| **④ 纠错成本** | Agent 出错时，错误能被发现并低成本恢复吗？（有测试 / 有 review / 能回滚） | 不能 → 风险不可控，别让它自主 | 让 Agent 直接对生产数据库执行 DELETE，错了无法回滚 |

工程推论：

5. **四问里，第①问筛掉"杀鸡用牛刀"，第④问筛掉"高危不可控"——这两问淘汰的需求最多。** 第①问让你别在简单任务上过度设计，第④问让你别在危险任务上失控。
6. **第④问（纠错成本）是后端老手最容易忽略、却最致命的一问。** 你习惯了确定性系统"错了能复现、能定位、能修"。Agent 是概率性的，同样的输入两次可能走不同的路。**没有 review / 测试 / 回滚兜底的 Agent，等于把一个会随机出错的实习生直接接到了生产数据库上，还不给他配 code review。**
7. **"可行性"这一问有时效性——它是版图，不是物理。** 半年前模型干不了的任务，今天可能就行了。所以这一问要定期重新评估（按 2026 初的模型能力，名单会变，请学评估方法而非记结论）。
8. **四问全过，也只意味着"可以用 Agent"，不意味着"必须用最复杂的 Agent"。** 过了四问，仍然优先选能完成任务的**最低自主性**形态——这正好接上第 1.2 节的谱系。

心法：**四问是一个『向下推』的过滤器——它的默认输出是"不要用 Agent"，只有四问全过才放行。** 抱着"我要做 Agent"的心态去套四问是本末倒置；正确姿势是抱着"凭什么这个场景配得上 Agent"去质问它。

## 二、演进：以前怎么做，为什么不够

### 2.1 第一代：纯 Chain（固定流水线）

最早把 LLM 串起来干活，就是写死的链：`输入 → LLM 步骤1 → LLM 步骤2 → 输出`。LangChain 早期的 `LLMChain`、`SequentialChain` 就是干这个的。

**为什么不够**：链是静态的。它没法处理"第 2 步的结果决定要不要有第 3 步、第 3 步该干嘛"。遇到需要动态分支、需要重试、需要根据中间结果改变后续动作的任务，写死的链就抓瞎。本质上它还停留在 L1，没有"运行时决策"。

### 2.2 第二代：ReAct（让模型边推理边行动）

2022 年那篇 ReAct 论文（Reasoning + Acting）是 Agent 范式的奠基。它的核心洞察简单到优雅：

**别让模型只"想"（纯 Chain-of-Thought 容易一本正经地编），也别让模型只"做"（不想就动手容易瞎搞）。让它『想一步、做一步、看一眼结果、再想下一步』，交替进行。**

一个 ReAct 循环的内心独白长这样：

```
Thought: 用户问北京今天比上海热多少。我需要分别查两个城市的气温。
Action: get_weather(city="北京")
Observation: 北京 12°C
Thought: 拿到北京了，再查上海。
Action: get_weather(city="上海")
Observation: 上海 18°C
Thought: 北京 12，上海 18，上海比北京热 6 度。可以回答了。
Answer: 上海比北京热 6°C，北京今天反而更凉。
```

注意这个结构里的三个角色：**Thought（推理）、Action（行动，即工具调用）、Observation（观察，即工具返回结果）**。这三者交替，直到模型认为可以给出最终答案。

**为什么 ReAct 是分水岭**：它第一次把"推理"和"行动"耦合进了一个反馈循环。模型不再是一锤子买卖，而是能**根据真实世界的反馈（Observation）修正自己的下一步**。这就有了 L4 的雏形——控制流真正交给了模型。

工程推论：

9. **ReAct 的"Thought"不是给用户看的，是给模型自己『把推理外显出来』用的。** 让模型先写出"我为什么要调这个工具"，能显著降低它瞎调工具的概率——这是把隐式决策变成显式 token，可观测、可调试。
10. **在现代 API 里，你几乎不用手写 `Thought:/Action:/Observation:` 这套文本协议了。** 原生 tool use（第 5 章）已经把 Action/Observation 结构化成 `tool_use` / `tool_result` 消息块，模型的 thinking（推理）也有原生支持。**ReAct 是思想，不是非得手撸的文本格式**——这是新手常踩的认知错位，以为 ReAct 就得自己解析 `Action:` 字符串。
11. **早期手撸 ReAct 用正则解析模型吐的 `Action: xxx` 字符串，极其脆弱**——模型多打个空格、换行、引号，解析就崩。这正是原生 tool use 要解决的问题：把"行动"从自由文本变成 schema 约束的结构化输出（呼应第 4 章结构化输出）。

### 2.3 第三代：原生 Agent Loop + 思考预算

到了 2025–2026，模型 API 把 Agent 需要的能力做成了原生一等公民，这是一个**行为控制主通道从『采样超参』迁移到『自然语言 + 思考预算』**的标志性转变。以 Anthropic 的 API 为例（按 2026 初，具体参数会变，学趋势）：

- **思考（thinking）从"手动给固定 token 预算"演进到"自适应思考（adaptive thinking）"**：你不再精确控制"想多少个 token"，而是用 `thinking: {type: "adaptive"}` 让模型自己决定每一步该想多深。新一代模型（Opus 4.7/4.8、Fable 5）甚至**移除了 `budget_tokens` 和 `temperature/top_p/top_k`**——传了直接报 400。
- **取而代之的是 `effort` 档位**（low / medium / high / xhigh / max）：用一个语义档位控制"思考深度 + 整体花费"，而不是抠具体数字。
- **Task Budgets（任务预算，beta）**：你可以告诉模型"这整个 agentic loop 你总共有 N 个 token"，模型能看到一个**倒计时**，自己学会"省着用、快收尾"。这是把"防失控预算"从外部硬控变成模型自我约束的内部信号。

这个演进的本质（重点理解，这是面试区分度极高的一点）：

> **行为控制的主通道，正在从"调采样超参（temperature 之类）"迁移到"自然语言指令 + 思考/预算控制"。**

你过去靠 `temperature=0` 求确定性、靠调 `top_p` 求多样性的那套，在新一代 agentic 模型上要么被移除、要么不再是主要抓手。**现在你主要靠提示词把意图说清楚、靠 effort 档位和 task budget 控制成本与深度。**

工程推论：

12. **`temperature=0` 从来就没保证过两次输出字节级一致**——它只是降低随机性。在新一代模型上它干脆被移除了。如果你的系统设计依赖"LLM 输出可复现"，那从一开始就是错的假设（呼应第 1 章 LLM 物理学：LLM 本质是概率的）。
13. **`max_tokens` 和 `task_budget` 是两个不同的东西，面试爱考。** `max_tokens` 是**单次响应**的硬上限，模型**看不到**它，撞上就被截断；`task_budget` 是**整个 agent loop** 的软预算，模型**看得到**倒计时，会自我调节。一个是"墙"，一个是"仪表盘"。
14. **adaptive thinking 默认不一定开，且思考内容默认可能不返回（"omitted"）。** 新一代模型上要显式开 `{type: "adaptive"}`，要看到思考摘要还得设 `display: "summarized"`——否则你的 UI 看起来就是"长时间卡住没反应"。这是个静默行为变化，踩坑高发区。

心法：**Agent 这十年的演进，是一条『把控制权和决策权，从硬编码的程序里，逐步让渡给模型』的路线。** 但每让渡一分，你就要在『停机条件』上补回一分——否则让渡出去的就不是决策权，是失控。

## 三、现状：当前主流方案及优缺点

### 3.1 单 Agent 的标准心智模型：感知-决策-行动-观察循环

把上面的 ReAct 抽象成一个通用的 **Agent Loop**，它有四个相位，循环往复：

```
        ┌─────────────────────────────────────────────┐
        │                                             │
        ▼                                             │
   ┌─────────┐   ┌─────────┐   ┌─────────┐   ┌───────────┐
   │ 感知    │──▶│ 决策    │──▶│ 行动    │──▶│ 观察      │──┐
   │Perceive │   │Decide   │   │Act      │   │Observe    │  │
   │(读上下文)│   │(LLM思考 │   │(执行工具)│   │(拿到结果)  │  │
   │         │   │ 选工具) │   │         │   │           │  │
   └─────────┘   └─────────┘   └─────────┘   └───────────┘  │
        ▲                                                   │
        │            把 Observation 加进上下文，回到感知       │
        └───────────────────────────────────────────────────┘
                              │
              每轮检查：停机条件满足了吗？满足 → 退出循环
```

- **感知（Perceive）**：把当前所有上下文（用户原始请求 + 历史动作 + 历史观察结果）组装成发给模型的 messages。
- **决策（Decide）**：模型读完上下文，输出"下一步动作"——要么调某个工具（`tool_use`），要么宣布任务完成（`end_turn`）。
- **行动（Act）**：你的代码执行模型选定的工具。
- **观察（Observe）**：拿到工具返回，作为 `tool_result` 喂回上下文。
- **回到感知**，进入下一轮——**除非停机条件已满足**。

这个循环和 tool use 的"工具循环"（第 5 章）是同一个东西，只是视角更高：tool use 关注"一次工具调用的消息往返"，Agent Loop 关注"整个循环的编排、状态累积、和何时停"。

工程推论：

15. **整个 Agent Loop 在 HTTP 层依然是无状态的**——每一轮你都要把"到目前为止的全部历史"重新发给模型（呼应第 1 章：LLM 不记得上一句）。"上下文越滚越长"是 Agent 的固有特性，也是成本和延迟的主要来源，更是上下文工程（第 13 章）要解决的核心矛盾。
16. **"决策"这一步是唯一花钱调模型的地方，"行动"是你本地执行的。** 一个 agent 跑了 20 步，就是 20+ 次模型调用，每次都带着越来越长的上下文。**Agent 的成本不是线性的，是『步数 × 累积上下文长度』，接近平方级膨胀。** 这是为什么"循环不收敛"在 Agent 里是头号成本杀手。
17. **prompt caching 在 Agent 里不是优化项，是必需品。** 因为每轮请求的"前缀"（system prompt + 工具定义 + 历史）高度重叠，缓存命中能把成本打到一折左右。但前缀任何一个字节变了缓存就失效——所以工具定义、system prompt 在一次 loop 里必须保持字节稳定，别中途加减工具、别往 system 里塞时间戳（这些是第 13 章的硬性约束，这里先建立意识）。

### 3.2 三大规划范式对比（设计题高频）

模型怎么"决定下一步"，有三种主流范式。**这是一道经典设计题：给定任务，你选哪种规划范式，为什么。**

| 范式 | 怎么工作 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|---|
| **ReAct**（边想边做） | 每步都重新决策：想一步→做一步→看结果→再想 | 灵活，能根据实时反馈纠偏；对动态/不确定环境鲁棒 | 没有全局视野，容易"只见树木"；步数多、token 贵；可能原地打转 | 探索性任务、环境会变、下一步强依赖上一步结果（如 debug、网页操作） |
| **Plan-and-Execute**（先规划后执行） | 先让模型一次性产出完整计划（步骤1..N），再逐步执行 | 有全局视野；计划可被人审查/批准；执行阶段可用更便宜的模型 | 计划是"拍脑袋拍在执行前"的，环境一变计划就废；重规划成本高 | 步骤相对确定、可预先拆解的任务（如"生成季度报告"的固定流程） |
| **Reflection**（反思/自我批判） | 产出结果后，让模型（或另一个模型）批判它，再据此修正，迭代多轮 | 显著提升输出质量；能自我发现错误 | 每轮反思都烧钱烧时间；可能陷入"反复修改但没真变好"的自嗨 | 对质量要求高、有明确好坏标准的任务（如写代码、写文案、解数学题） |

**怎么当场选（面试别只背表，要给一个判断流程）——用两根轴问两个问题：**

1. **轴一·环境确定性**："这个任务的步骤，我现在能不能先拆清楚？" 能拆清 → 偏 **Plan-and-Execute**（先出计划、可审查、执行阶段还能降级用便宜模型）；拆不清、下一步强依赖上一步实测结果（debug、网页操作）→ 偏 **ReAct**。
2. **轴二·质量门槛**："产出有没有明确的好坏标准、且值得为质量多烧 token？" 有 → 在上面选定的范式外**再叠一层 Reflection**（Reflection 不是和前两者并列三选一，而是一个『质量增益层』，谁都能叠）。

一句话决策规则：**ReAct / Plan-and-Execute 二选一定『主干怎么走』，Reflection 是按需叠加的『质量后处理』。** 能把"主干 + 增益层"这个结构讲清楚，比平铺三个定义高一个段位（这也正好接上推论 18 的组合用法）。

工程推论：

18. **这三个范式不是互斥的，生产里经常组合。** 典型组合：Plan-and-Execute 定大框架（全局视野），每个步骤内部用 ReAct 处理不确定性（局部灵活），关键产出再加一轮 Reflection（保质量）。面试时能说出"组合用法"比只会背三个定义高一个段位。
19. **Plan-and-Execute 的最大价值之一是『把计划暴露给人审查』**——这是 human-in-the-loop（见 3.4）的天然插入点。纯 ReAct 因为没有显式的完整计划，人想介入审查都没有抓手。
20. **Reflection 是"用 token 换质量"的典型，必须配预算上限。** "让模型自我批判直到满意"听起来很美，但"满意"是模型自己判断的——它可能第 2 轮就够好了还在改，也可能改了 5 轮越改越差。**Reflection 必须有硬性的最大迭代轮数，否则就是 Reflection 版的不收敛。** 现代 API 的 Outcome/rubric 机制（用一个独立的评分模型按明确标准判断"够不够好"）就是给 Reflection 装一个客观的停机裁判。
21. **"自我反思能纠错"有上限——模型批判自己时，用的还是同一套可能犯错的认知。** 让出错的模型审查自己，常常发现不了根本错误（它要是能发现，第一次就不会错）。所以高质量 Reflection 往往用**另一个模型或独立上下文**来当批判者，而不是让同一个会话自我审查。

### 3.3 停机条件与防失控（本章的"安全带"，绝对不能省）

这是把"玩具 Agent"和"能上线 Agent"区分开的**唯一一道工序**。新手写完 `while True: 调模型 → 执行工具` 就以为做完了，然后：

- 模型在两个工具之间反复横跳，永不停止 → **烧钱**
- 模型坚信某个错误前提，沿着错路一路走到黑 → **自信地交付错误结果**
- 模型卡在"调工具→失败→换个参数再调→又失败"的死循环 → **资源耗尽**

**Agent 最大的生产风险从来不是"不够聪明"，是"不收敛"。** 你必须给循环装多重独立的"刹车"。下面这套是生产标配，**每一条都要有**：

| 刹车 | 防的是什么 | 实现 |
|---|---|---|
| **① 最大步数（max iterations）** | 无限循环 | 循环计数器，到 N 步强制退出。这是最后一道物理底线，必须有 |
| **② 预算上限（budget cap）** | 烧钱 / 烧 token | 累计 token 或累计调用成本超阈值即停（可结合 task_budget 让模型自己也知道） |
| **③ 墙钟超时（wall-clock timeout）** | 单次任务卡死太久 | 整个任务级别的 deadline，超时中断 |
| **④ 循环检测（loop detection）** | 原地打转 | 检测"连续 K 次调用同一工具+同一参数"或"状态无变化"，命中则中断或强制换策略 |
| **⑤ 重复失败熔断** | 同一步反复失败 | 同一工具连续失败 M 次，停止重试，降级或上报 |
| **⑥ 无进展检测** | 看似在动实则没推进 | 若连续几步没有产生新信息/没接近目标，判定停滞 |

工程推论：

22. **这些刹车必须『多重且独立』，不能只靠一个。** 只靠 max iterations，模型可能在步数耗尽前就把预算烧光；只靠预算，可能在花光前已经卡死浪费墙钟时间。**它们防的是不同的失控模式，是纵深防御，不是冗余。** 这一点直接对标第 21 章安全的"纵深防御"思想。
23. **max iterations 是『底线刹车』，不是『正常停机方式』。** 健康的 agent 应该靠"任务完成"（模型返回 `end_turn`）自然停，撞到 max iterations 说明这次大概率失败了——它应该触发告警和降级，而不是被当成正常结束。**把"撞上限"和"正常完成"混为一谈，是 Agent 可观测性的头号 bug。**
24. **循环检测的难点是『定义"原地打转"』。** "调了同样的工具+同样的参数"是最容易检测的硬信号；但"换了参数其实在做无意义微调""语义上在重复"很难检测。生产里通常组合用：硬信号（精确重复）兜底 + 软信号（N 步无新信息/无状态变化）补充。
25. **停机条件触发后『怎么收尾』和『要不要停』同样重要。** 撞了上限不能直接抛异常崩给用户——要有降级答案（"我尽力了，这是目前的部分结论"）、要落审计日志、要能让上层决定是否重试或转人工。**没有优雅降级的停机，等于把不收敛的锅甩给了用户。**

心法：**写 Agent 循环时，先写停机条件，再写循环体。** 把"怎么停"当成一等设计目标——这是后端老手治理不可靠系统的肌肉记忆（限流、熔断、超时、降级）在 Agent 上的直接复用，**一行都没浪费**。

### 3.4 Human-in-the-Loop：在哪里给人留介入点

完全自主的 Agent 在高风险场景是不可接受的（回到四问的第④问：纠错成本）。**Human-in-the-loop（HITL）= 在 agent 的决策流里，给人类留出"批准 / 否决 / 修改 / 接管"的卡点。** 这是把"不可控的自主"降级成"可控的半自主"的关键设计。

四个标准介入点：

1. **计划审查点**：Plan-and-Execute 产出计划后、执行前，让人看一眼批准。适合改动大、不可逆的任务。
2. **危险动作确认点（最重要）**：在执行**不可逆/高风险**动作前暂停等人点头。发邮件、删数据、转账、提交 PR、调用花钱的外部 API——这些必须可门控。这正是第 5 章"该把哪些动作提升为专用工具"的核心动机之一：**`send_email` 这种专用工具能挂确认钩子，`bash -c "curl -X POST..."` 这种万能 bash 命令没法精确门控。**
3. **澄清点**：Agent 遇到歧义/信息不足时，主动停下来问人，而不是自己瞎猜往下走。
4. **结果验收点**：最终产出交付前，让人验收/修正（这本质是把人当成 Reflection 的批判者）。

工程推论：

26. **HITL 的介入点选择，本质是『按动作的可逆性和风险』分级，而不是一刀切全审或全不审。** 全自动（零介入）在高危场景不可接受；每步都问人则把 Agent 的效率优势全抹掉了。**正确做法是：低风险动作放行，高风险/不可逆动作才卡人——可逆性是最好用的分级标准。**
27. **"危险动作确认"要在工具层而非提示词层实现。** 靠 system prompt 写"执行删除前请先问我"是不可靠的——模型可能不听、可能被注入绕过（第 21 章）。可靠做法是在你的**执行代码里**对危险工具强制挂确认钩子：模型发出 `delete_data` 调用 → 你的 harness 拦截 → 弹给人确认 → 人点头才真执行。**控制权必须落在你的代码里，不能托付给模型的"自觉"。**
28. **现代 Agent API 把"暂停等人"做成了原生状态。** 比如工具可以配 `always_ask` 权限策略，模型一调它，session 就进入 idle 等你的 `tool_confirmation`（allow/deny，deny 还能附理由让模型换路）。你不用自己手撸"暂停-恢复"的状态机——但你要理解它在干什么，否则 session 卡在 idle 你都不知道是在等你。
29. **HITL 最阴的生产事故是『等人确认时连接断了 → 死锁』，这是必须在架构上防的。** 托管型 HITL（`always_ask`）靠一条长连接（SSE 事件流）通知你"该确认了"、你再回 `tool_confirmation`。**如果在『模型发出危险动作 → 等你点头』这个窗口里你的客户端断线重连，事件流不会重放（SSE 无 replay）——你重连后收不到那条待确认事件，而 session 还在 idle 死等你，永远不动。** 这是个真实的生产死锁：表面看 agent"卡住了"，实际是确认请求丢在了断开的连接里。**防法是每次（重）连都先拉一遍历史事件（list events）、按事件 ID 去重补齐，再消费实时流**——把"可能漏掉的待确认事件"从历史里捞回来。这条直接决定你的 HITL 是"能上线"还是"偶发卡死还查不出原因"。
30. **`idle` 不等于"任务完成"，是 HITL 可观测性的头号坑。** session 进 idle 可能是：真完成了、在等你确认危险动作、在等你回 custom tool 结果。**只凭"进了 idle"就当任务结束去清理 session，会把一个正等你点头的 agent 误杀。** 判停要看 idle 附带的 `stop_reason`——是"正常结束"还是"需要你介入"（requires_action）。这和 3.3 节"撞上限≠正常完成"是同一类错误：**把"停下来"和"干完了"混为一谈。**
31. **HITL 不是只在"人不信任 Agent"时才用，它也是『积累信任、收集训练信号』的手段。** 上线初期高频卡人审查，观察 Agent 哪里爱出错；信任建立后逐步放开卡点。人每次的 allow/deny 也是宝贵的标注数据。**HITL 是个可以随信任度动态调节的旋钮，不是固定开关。**

## 四、落地：生产怎么选、大厂怎么做

### 4.1 单 Agent 架构的标准组件图

一个能上线的单 Agent，拆开就是这几块：

```
┌──────────────────────────────────────────────────────────┐
│                      你的 Harness（编排代码）                  │
│  ┌────────────────────────────────────────────────────┐   │
│  │  Agent Loop（感知-决策-行动-观察 + 停机检查）            │   │
│  └────────────────────────────────────────────────────┘   │
│         │              │              │             │       │
│         ▼              ▼              ▼             ▼       │
│  ┌──────────┐  ┌──────────────┐ ┌──────────┐ ┌──────────┐  │
│  │ System   │  │ 工具面         │ │ 上下文/   │ │ 停机&安全 │  │
│  │ Prompt   │  │ Tool Surface  │ │ 记忆      │ │ 守卫      │  │
│  │ (角色/    │  │ (第5/6章)     │ │ (第13章)  │ │ (步数/预算 │  │
│  │  约束/    │  │ - 自定义工具   │ │           │ │  /超时/   │  │
│  │  停机指令) │  │ - bash/编辑   │ │           │ │  循环检测) │  │
│  └──────────┘  │ - HITL 钩子   │ └──────────┘ └──────────┘  │
│                └──────────────┘                            │
└──────────────────────────────────────────────────────────┘
         │
         ▼
   ┌───────────┐
   │  LLM API   │  ← 每轮"决策"打一次（带 thinking / effort / 工具定义）
   └───────────┘
```

注意一个关键概念：**Harness（外壳/编排器）**。**模型只负责"出意图"（emit tool calls），真正的执行、门控、渲染、调度全在你的 Harness 里。** 模型不知道你的安全边界、不知道你的审批策略、不知道你的 UI。所以：

> **工具调用的"形状"决定了你的 Harness 能做什么。** 给模型一个万能 `bash` 工具，Harness 拿到的只是一个不透明的命令字符串，每个动作长得一样，没法精确门控；把某个动作提升为专用工具（带类型化参数），Harness 就有了一个动作专属的拦截点，可以门控、可以渲染、可以审计、可以并行调度。

### 4.2 工具面（Tool Surface）设计：bash vs 专用工具（设计题富矿）

这是单 Agent 架构里最考工程判断的地方，直接接续第 5 章。核心权衡：

| 维度 | 万能 bash 工具 | 专用工具（typed） |
|---|---|---|
| 覆盖面 | 极广，几乎啥都能干 | 窄，只能干你定义的事 |
| Harness 可见性 | 低——只是个命令字符串，所有动作同形 | 高——动作专属、参数有类型，可拦截 |
| 安全门控 | 难——没法区分 `grep`（安全）和 `rm -rf`（危险） | 易——可对 `delete_x` 单独挂确认 |
| 并行调度 | 难——Harness 分不清哪些命令能并行 | 易——只读工具（glob/grep）可标记为并行安全 |
| 陈旧性检查 | 做不到 | 能——专用 `edit` 工具可拒绝"文件已被改动"的写入 |

**生产经验法则（记住这一条就够应付大部分设计题）：**

> **用 bash 起步求覆盖面；当你需要对某个动作『门控、渲染、审计、并行』时，把它提升为专用工具。**

工程推论：

32. **"需要门控的动作"是提升为专用工具的头号信号，而『可逆性』是最好的判据。** 难以撤销的动作（外部 API 调用、发消息、删数据）应该门控；`send_email` 容易门控，`bash -c "curl -X POST..."` 不行。这条直接呼应 3.4 的 HITL 和第 5 章的工具设计。
33. **工具数量不是越多越好，太多工具会让模型『选择困难』选错工具。** 工具面要聚焦。当工具库很大但每次只用得上几个时，用 **tool search**（模型按需检索工具，而非一次性把所有 schema 塞进上下文）——这还能保护 prompt cache（追加 schema 而非替换）。这是"工具多到撑爆上下文"时的标准解法。
34. **工具的 description 写得烂，模型就选错、乱调或该用不用。** 现代模型（尤其新一代 Opus）默认更"克制"调工具，**在 description 里明确写"什么时候该调这个工具"（而不只是"这个工具是干嘛的"）能显著提升正确调用率**。这是 prompt engineering 里 ROI 最高的活之一（呼应第 3 章：工具描述工程是提示工程的一大半）。
35. **只读工具应标记为"并行安全"，让 Harness 能并发执行。** 一个 agent 要读 5 个文件，串行调 5 次 vs 并行一把调，延迟差好几倍。但如果这 5 个动作走的是万能 bash，Harness 没法判断哪个是并行安全的 `grep`、哪个是并行不安全的 `git push`，只能全部串行。**专用工具是解锁并行的前提。**

### 4.3 大厂在用的两种部署形态：自托管 loop vs 托管 Agent

按 2026 初的主流 API 能力，单 Agent 落地有两种形态（学权衡，具体产品名会变）：

| 形态 | 谁跑 Agent Loop | 谁跑工具执行 | 你掌控什么 | 适合 |
|---|---|---|---|---|
| **自托管 loop**（API + tool use） | **你的代码** | 你的代码 | 全部：审批门、日志、条件执行、loop 怎么转 | 要细粒度控制、要自定义工具运行时、要把 agent 嵌进你的产品 |
| **托管 Agent**（Managed Agents 这类） | **平台方** | 平台方的沙箱容器 | 配置（模型/system/工具/技能），不管 loop 怎么转 | 要一个带工作区的有状态 agent，不想自己运维容器和 loop |

工程推论：

36. **P3 阶段，你必须先走"自托管 loop"路线——亲手把循环写一遍。** 托管 Agent 把 loop 藏起来了，对学习是灾难。**只有自己写过 `while + 停机条件 + 工具分发 + 上下文累积`，你才真懂 Agent 是什么、贵在哪、为什么会不收敛。** 这是 P0"禁框架"原则在 Agent 阶段的延续——先手撸，再谈托管。
37. **托管 Agent 的本质卖点是『平台帮你跑 loop + 提供工具执行的沙箱容器』。** 当你需要"一个 agent 有自己的工作区（能跑 bash、读写文件、执行代码），还不想自己运维这套沙箱和编排"时，托管方案有价值。判据：**你是想自己掌控 compute 和 loop（选自托管），还是想把这摊运维外包出去（选托管）。**
38. **无论哪种形态，"何时停"的预算/步数边界你都要懂**——自托管是你亲手写刹车，托管是你配 max_iterations/task_budget 这类参数。**Agent 的"停机条件"不会因为换成托管就消失，只是从『你写代码』变成『你填参数』。** 不理解它，托管 session 卡死或烧超你照样抓瞎。

### 4.4 国内厂商语境

按 2026 初，国内模型（豆包/字节、Qwen/阿里、DeepSeek、Kimi/月之暗面、GLM/智谱、混元/腾讯、文心/百度）大多支持 OpenAI 兼容的 function calling 接口，足以手撸 ReAct loop。但要注意几点工程现实（版图，会变）：

- **adaptive thinking / effort 档位 / task budget 这类『新一代 agentic 原生能力』在不同厂商成熟度不一**。很多还停留在"手动 function calling + 自己控温"的阶段，没有 task_budget 这种原生预算倒计时。
- **DeepSeek R1（2025 初）证明了开源推理能力可以追平闭源**，这对"自托管 agent 用什么模型"的选型影响很大——你可以在自己的 loop 里换开源推理模型，不被单一闭源 API 绑死。
- **选型方法论 > 记名单**：评估一个模型适不适合做你的 agent backbone，看四点——① function calling/tool use 是否稳定（schema 遵从度）、② 长上下文是否扛得住（agent 上下文会爆涨）、③ 是否有原生 thinking/推理、④ 有没有预算/思考控制旋钮。**按这四点去测当下任何一家，而不是背"谁家好"。**

心法：**模型能力是别人的，你的 Harness 才是你的护城河。** 停机条件、工具面设计、HITL 卡点、可观测——这些工程外壳决定了你的 Agent 能不能上线，而它们和你用哪家模型基本解耦。这正是后端老手该出彩的地方。

## 五、面试视角：高频题与考点

面试官在这一章想区分的是：**你是"会用 Agent 框架的人"，还是"理解 Agent 本质、能判断该不该用、能把它焊成可控系统的人"。** 前者一抓一大把，后者稀缺。

核心考点分布：

1. **谱系判断**：能不能把"要不要 Agent"从二元问题升级成谱系判断，并对具体需求选档 + 给理由。这是区分"被营销话术带跑"和"有架构判断力"的第一道题。
2. **四问框架**：能不能背出何时该/不该用 Agent，尤其能讲清第④问（纠错成本）为什么对后端老手最致命。
3. **停机与防失控**：能不能一口气说出多重独立刹车、并解释它们防的是不同失控模式（纠错成本）。**几乎必考——这是 Agent 工程化成熟度的试金石。**
4. **范式权衡**：ReAct vs Plan-and-Execute vs Reflection 的 trade-off 和组合用法。
5. **工具面设计**：bash vs 专用工具的取舍，及"可逆性"作为门控判据。
6. **成本直觉**：能不能算清 Agent 成本为什么是平方级膨胀、prompt caching 为什么是必需品。

（详细题目与答题框架见下方"章末五件套"。）

## 六、未来演进：3-5 年方向，对应用工程师的含义

按 2026 初判断（这是版图，学方向不背结论）：

1. **控制通道继续"自然语言化"**：从抠采样超参 → 自适应思考 + effort 档位 → 未来很可能是更高层的"目标/预算/约束"声明式接口。**对应用工程师的含义：你越来越像"给 Agent 下任务书和定预算"的产品经理，而不是"调参数"的工程师。** 把意图说清楚、把成功标准（rubric）定明确，会比调超参更重要。

2. **停机/收敛从"外部硬控"走向"模型自我约束"**：task budget 倒计时、Outcome/rubric 自动评判是早期信号。模型会越来越"知道自己有多少预算、什么算完成"。**含义：你的活从"写一堆刹车代码"部分转向"定义清晰的成功标准和预算"——但底线刹车（max iterations）作为物理兜底，5 年内不会消失。**

3. **单 Agent 能力上限持续抬高，挤压 Multi-Agent 的必要性**：很多今天要拆成多 agent 才能干的活，明天单个更强的 agent + 更好的上下文工程（第 13 章）就够了。**含义：别急着上 Multi-Agent（第 15 章），单 Agent 的天花板每半年都在涨——这正好呼应"自主性是成本"的心法。**

4. **托管 Agent 与自托管的分工固化**：控制平面（agent/环境配置）走托管 + 版本化 YAML，数据平面（具体 session、事件流）走 SDK 嵌进你的应用。**含义：你要同时懂"怎么手撸 loop"（理解本质）和"怎么用托管平台"（生产提效），两手都要硬。**

> **本章总心法：Agent 不是"更聪明的 AI"，是"你让渡了控制权的循环"。让渡多少自主，就要补回多少停机保护——否则让渡出去的不是决策权，是失控。判断『何时不让渡』（退回 Workflow、退回单次调用），比实现一个炫酷的 Agent 更值钱。**

# 章末五件套

## 一、高频面试题（10 道，附答题框架）

**Q1. 用一句话定义 Agent，再说清它和"普通 LLM 调用"的本质区别。**
- 框架：Agent = LLM + 工具 + 循环 + 停机条件。本质区别是**控制流的归属**——普通调用是"你写死流程，模型填空"；Agent 是"模型在循环里自己决定下一步做什么、做几步"。
- 陷阱：只答"Agent 能调工具"是不及格的——工具调用（第 5 章）本身不是 Agent，**带不带"模型自主决定的循环"才是分界线**。一次性 tool use 不是 Agent，把 tool use 塞进模型控制的循环才是。
- 加分：点出"循环 + 停机条件"里，停机条件才是工程难点，不是循环本身。

**Q2. 你怎么判断一个需求该不该用 Agent？**
- 框架：先讲自主性谱系（L0–L5），强调默认应落在低档位；再过四问（复杂度/价值/可行性/纠错成本），任一为否就退回更简单层。
- 陷阱：上来就讲"怎么实现 Agent"是掉进坑里了——**这题考的是"克制"，考你会不会判断"不该用"。** 能举一个"看起来该上 Agent 其实不该"的反例（比如简单分类任务）是强信号。
- 加分：强调第④问（纠错成本/可逆性）对生产最关键，并解释为什么。

**Q3. ReAct 是什么？现代 API 里你还需要手写 ReAct 的文本格式吗？**
- 框架：ReAct = Reasoning + Acting 交替，Thought/Action/Observation 三相循环，核心是"用真实反馈修正下一步"。现代 API 里**不需要手撸文本格式**——原生 tool use 把 Action/Observation 结构化成消息块，thinking 也有原生支持。**ReAct 是思想，不是非得手写的字符串协议。**
- 陷阱：以为 ReAct 就得自己正则解析 `Action:` 字符串——这是早期做法，脆弱且已被原生 tool use 取代。
- 加分：点出早期文本解析为什么脆弱（模型多打空格就崩），引出结构化输出（第 4 章）的必要性。

**Q4. 一个 Agent 在生产上最大的风险是什么？你怎么防？**
- 框架：最大风险是**不收敛**（无限循环/烧钱/原地打转/坚持错路），不是"不够聪明"。防护是多重独立刹车：max iterations + 预算上限 + 墙钟超时 + 循环检测 + 重复失败熔断 + 无进展检测。强调它们**防的是不同失控模式，是纵深防御**。
- 陷阱：只答"加个最大步数限制"——单一刹车不够，要能说出"为什么需要多重、它们各防什么"。
- 加分：点出"撞上限 ≠ 正常完成"，撞上限应触发告警和优雅降级，而不是当成功处理。

**Q5. 对比 ReAct、Plan-and-Execute、Reflection 三种规划范式。**
- 框架：用 trade-off 表答——ReAct 灵活但无全局视野、贵；Plan-and-Execute 有全局视野且计划可审查，但环境一变计划就废；Reflection 提质量但烧钱且可能自嗨。
- 陷阱：把它们讲成互斥三选一——**生产里经常组合**（Plan 定框架 + 步内 ReAct + 关键产出加 Reflection）。能讲组合用法是高一段位的信号。
- 加分：指出 Reflection 让"出错的模型审查自己"有上限，高质量做法是用独立模型/上下文当批判者。

**Q6. 什么是 Human-in-the-Loop？介入点该放在哪？为什么不能靠 system prompt 实现危险动作确认？**
- 框架：HITL = 在 agent 决策流里留出人类批准/否决/修改/接管的卡点。介入点按动作可逆性分级：计划审查、危险动作确认、澄清、结果验收。危险动作确认**必须在工具/代码层强制实现**，不能靠 system prompt——因为模型可能不听、可能被 prompt injection 绕过（第 21 章）。
- 陷阱：答"在 prompt 里写'删除前请先问我'"——这正是错误答案，控制权必须落在你的 Harness 代码里。
- 加分：把 HITL 描述成"可随信任度动态调节的旋钮"，而非固定开关。

**Q7. 为什么说 Agent 的成本是"平方级膨胀"？prompt caching 为什么是必需品？**
- 框架：Agent 无状态 + 每轮重发全部历史 → 成本 ≈ 步数 × 累积上下文长度，接近平方级。prompt caching 让高度重叠的前缀（system + 工具定义 + 历史）命中缓存，成本打到约一折。
- 陷阱：忽略"输出价 ≈ 输入价 5 倍"这个不对称——长 agent 的输出 token 累积也很贵。
- 加分：点出 caching 的前提是前缀字节稳定——所以一次 loop 内不能中途加减工具、不能往 system 塞时间戳/UUID（否则静默击穿缓存）。

**Q8. `max_tokens` 和 task budget（任务预算）有什么区别？**
- 框架：`max_tokens` 是**单次响应**的硬上限，模型**看不到**，撞上即截断；task budget 是**整个 agent loop** 的软预算，模型**看得到倒计时**，会自我调节"省着用、快收尾"。一个是墙，一个是仪表盘。
- 陷阱：以为两者是一回事或可互相替代——它们作用域和可见性都不同。
- 加分：把 task budget 归入"停机/收敛从外部硬控走向模型自我约束"的趋势。

**Q9.（坑感题）你的 Agent 上线后偶尔"长时间没反应然后突然出一大段"，可能是什么原因？**
- 框架：很可能是**新一代模型的 thinking 默认不返回内容（"omitted"）**——模型在思考（adaptive thinking）但思考摘要没流式出来，UI 看着就是卡住。解法：设 `display: "summarized"` 让思考摘要可见，给用户进度反馈。
- 陷阱：第一反应去查网络/超时——但这其实是一个静默的 API 行为变化。
- 加分：延伸到"adaptive thinking 在新模型上默认不一定开，要显式 `{type:"adaptive"}`"这个相关坑。

**Q10. 给模型一个万能 bash 工具 vs 给一堆专用工具，你怎么选？**
- 框架：bash 起步求覆盖面；当需要门控/渲染/审计/并行某个动作时，提升为专用工具。判据是**动作的可逆性**——难撤销的动作要专用工具以便挂确认钩子。
- 陷阱：一边倒说"专用工具一定更好"——bash 的覆盖面价值真实存在，工具太多还会让模型选错。要答"分场景 + 渐进提升"。
- 加分：点出专用工具是解锁"只读工具并行执行"的前提，bash 因为 Harness 看不透没法安全并行。

## 二、实战项目增量（P3 起步：单 Agent 骨架）

在 P1（工具化助手）的代码库上，构建一个**单 agent 骨架**。这是 P3 的第一块地基，本章只要求把"会自己转、能安全停"的循环跑通，复杂的 planning/reflection 留给后续。

**必做验收标准（缺一不算过）：**

1. **手写 Agent Loop**：实现感知-决策-行动-观察循环。**禁止用 agent 框架替你管 loop**——必须自己写 `while + 模型调用 + 工具分发 + 上下文累积`（延续 P0 禁框架精神）。
2. **接 ≥3 个工具**，模型能在循环里自主选择、连环调用（复用 P1 工具，但现在是"模型控制的多步循环"而非单次 tool use）。
3. **停机保护必须齐全（本章重点，硬性要求）**：至少实现 ① max iterations ② 累计 token/成本预算上限 ③ 循环检测（连续 N 次相同工具+相同参数即中断）。三者**独立**生效。
4. **撞上限要优雅降级**：触发任一停机条件时，不许直接抛异常崩——要返回"部分结论 + 说明为什么停了"，并打一条结构化日志（区分"正常完成"和"撞上限")。
5. **跑一个需要 ≥3 步的真实任务**（如"查 A 和 B 两地天气并算差值再判断穿衣建议"），打印每一步的 Thought（如果模型支持思考）/Action/Observation/累计成本，让整个决策链可观测。
6. **加一个 HITL 卡点**：挑一个"危险动作"工具（哪怕是模拟的 `delete_file`），在执行前强制暂停等你确认（y/n），确认逻辑必须在**你的代码里**，不能靠 prompt。

**做完要能回答**：把你的 loop 画成时序图，标出每一步谁在说话、钱花在哪一步、三道刹车分别在哪一行代码生效。**这张图就是你面试讲 40 分钟的素材。**

**坑点提示：**
- 别忘了每轮把完整历史重发（无状态）——新手常犯只发最后一条的错，导致模型失忆。
- 上下文会迅速涨大，先观察它涨多快，为第 13 章的上下文工程埋问题意识。
- 循环检测的"相同参数"判定要做规范化（trim、排序 key），否则模型多打个空格你就检测不到重复。

## 三、设计题（考权衡判断）

> **题目**：你要给一个"运维助手 Agent"做设计，它能查日志、查监控、重启服务、扩容、回滚发布。请设计它的**自主性档位**和**工具面 + HITL 策略**。要求：给出"哪些动作全自动、哪些必须人类确认、哪些干脆不让 Agent 碰"的分级，并说明判据。

**考点**：能不能用"可逆性 + 风险 + 纠错成本"对动作分级，而不是一刀切全自主或全人工。

**参考思路（不是唯一解，重点是判断链）：**
- **只读动作（查日志/查监控）→ 全自动**：可逆、零风险，正是 Agent 该自动化的部分，还可标记并行安全加速。
- **可逆的低风险写动作（重启单个无状态服务）→ 自动 + 事后告警**：影响可恢复，但留痕。
- **高风险/影响面大的动作（扩容/回滚发布）→ 强制 HITL 确认**：花钱、影响面大，但通常可逆——卡人确认点头再执行，确认逻辑在 Harness 代码里。
- **不可逆/灾难性动作（删数据库、缩容到 0）→ 干脆不给这个工具，或要求二人复核**：纠错成本无穷大，第④问直接判否，不让 Agent 单独碰。
- **整体档位**：这种场景适合 **Plan-and-Execute**（先出处置计划给人审查）而非纯 ReAct——因为运维动作不可逆性高，需要"执行前能看计划"的抓手。
- **停机/安全**：必须有预算和步数上限 + 重启类动作的失败熔断（连续重启失败 M 次停手转人工，别陷入"重启→挂→再重启"死循环）。

**好答案的标志**：把"可逆性"当成贯穿始终的分级主轴，且敢说"删库这种就不该给 Agent 工具"——**敢于划定"不让 Agent 碰"的边界，是判断力的最高形态。**

## 四、系统设计题（含量级估算）

> **题目**：设计一个面向公司内部、支撑"代码评审 Agent"的服务。每天处理 2000 个 PR，每个 PR 平均触发一次 Agent 任务，每个任务平均 8 步 agent loop。请做**成本与容量估算**，并指出主要成本驱动和优化手段。

**估算框架（数字按 2026 初价目表锚点，重点是算法不是精确值）：**

- **请求量**：2000 PR/天 × 8 步 = **16,000 次模型调用/天**。
- **上下文膨胀**：关键！每步要重发历史。假设首步上下文 5K token，每步增量约 3K（新的 tool_result + thinking），到第 8 步上下文约 5K + 7×3K ≈ 26K token。**8 步的累计输入 token ≈ 5+8+11+...+26 ≈ 124K token/任务**（这就是"平方级膨胀"的具体体现）。
- **输出 token**：假设每步输出（思考 + 工具调用 + 文本）约 1.5K，8 步 ≈ 12K output token/任务。
- **单任务成本（无缓存，用 Balanced 档锚点 输入 ~$3/百万、输出 ~$15/百万）**：输入 124K×$3/1M ≈ $0.37 + 输出 12K×$15/1M ≈ $0.18 ≈ **$0.55/任务**。
- **日成本（无缓存）**：2000 × $0.55 ≈ **$1,100/天** ≈ 3.3 万刀/月。
- **上 prompt caching 后**：每步前缀（system + 工具定义 + 历史）高度重叠，缓存读约 0.1×。乐观估计输入成本能降到一折附近，**单任务输入成本从 $0.37 → ~$0.05 量级**，日成本可压到 **$400–500/天** 量级。**caching 是这里最大的杠杆，不是可选优化。**

**要点（面试官想看到的判断）：**
1. **主成本驱动是"步数 × 累积上下文"，不是单次调用**——所以两个优化方向：**减步数**（更好的 planning，别瞎转）和**控上下文**（第 13 章：及时清理过期 tool_result）。
2. **输出价 ≈ 输入价 5 倍**——别让 agent 输出冗长的思考/解释。
3. **容量上**：16K 调用/天 ≈ 平均 0.2 QPS，但 PR 提交有明显峰谷（工作日白天集中），要按峰值（可能 5–10× 均值）配限流和并发，并对失败做 fallback 到备用模型（接第 20 章网关）。
4. **Batch API 能不能用——这是个埋雷的加分点，答错比不答更糟。** 直觉是"代码评审不要求实时，上 Batch 再省 50%"。但 **Batch API 是『一发一收』的异步批处理（最长 24h、无事件流、无法中途回喂 tool_result），它跑不了一个『8 步交替调工具』的 agent loop**——agent 的多步循环本质上是同步交互（调工具→执行→回喂→再调），而 Batch 的单个请求只能是一次性 request→response。**正确的拆法是：把 agent loop 留在同步在线路径上跑，只把「可以离线、单次完成」的子任务（如对每个文件生成一段总结、跑一次纯 LLM 评分）抽出来塞进 Batch。** 把整个交互式 agent 任务塞进 Batch 是概念性错误。
5. **caching 的 TTL 也要进成本模型，不是默认就省。** 默认缓存 5 分钟 TTL；agent 每步之间隔着工具执行，若某步工具慢（如跑测试几分钟）超过 TTL，下一步前缀缓存就失效要重写（1.25×）。长 agent 要么用 1h TTL（写 2× 但扛得住步间间隔），要么接受部分步骤 cache miss——别把"缓存读 0.1×"当成无条件成立。

**好答案标志**：能算出"平方级膨胀"的具体数字；把 prompt caching 当一等优化手段；并且**敢于点破"Batch 跑不了交互式 agent loop"这个反直觉陷阱**——能区分"可离线的单次子任务"和"必须在线的多步循环"，比只会背"Batch 省 50%"高一个段位。
