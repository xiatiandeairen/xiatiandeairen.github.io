先说一句会冒犯人的话：你在第 02 章画的那张漂亮注意力热力图，单独拿出来，证明不了任何东西。它告诉你某个头"在看哪里"，不告诉你那个头"被不被用到"。第 03 章的线性探针（linear probe，用一个线性分类器去解码激活里有没有某个信息）同理——它告诉你信息"在场"，不告诉你信息"被读"。这两件事差得很远，而绝大多数 AI 批量生成的可解释性 survey 稿恰恰在这里混为一谈：它们把"注意力高 + 探针能解码"当成"这个组件算出了答案"的证据，然后自洽地、流畅地、错误地讲完一个故事。

这章干的事是把前七章的工具按一个固定顺序串起来，跑在**同一个行为**上，最后逼出一张证据矩阵和一个判定。重点不是工具更多，而是工具之间能互相证伪。配套代码全在 `examples/interpretability-from-scratch/src/stage08-pipeline.ts`，它复用全书共享的那个 modAdd 玩具 checkpoint（modular addition，模加法：输入 `[a, b, =]`，在 `=` 位置预测 `(a+b) mod 7`）。

## 一个解释要可信，必须能被反事实证伪 ✦

先把这章的核心标准摆出来，后面所有代码都是为它服务的。

**「我认为 X 头负责 Y」的合格证据链是三件事同时成立：**

1. **ablate X 则 Y 消失**（必要性 / necessity——删掉它，行为崩）
2. **patch X 则 Y 恢复**（充分性 / sufficiency——把它的值搬进来，行为回来）
3. **SAE 特征对应 Y，且干预该特征能改变 Y**（特征级因果）

缺干预证据的解释只是好看的故事。这不是吹毛求疵，是机制可解释性区别于"看图说话"的唯一硬线：相关性（correlation）可以被混淆变量（confound，一个同时影响两边、制造伪相关的第三方）伪造，因果性（causation）只能被干预（intervention）建立。注意力、探针、logit lens、SAE 全是相关工具——它们读激活但不扰动激活。只有 patching 和 ablation 动手术。

代码里这条标准是写进**数据类型**的，不是写在注释里靠自觉。每个工具吐一行 `EvidenceRow`，带一个 `kind` 字段把相关证据和因果证据物理隔开（`src/stage08-pipeline.ts:96-104`）：

```ts
type EvidenceKind = "correlational" | "causal";
interface EvidenceRow {
  technique: string;
  kind: EvidenceKind;
  metric: number;     // 原始测量值（recovery / accuracy / drop ...）
  supports: boolean;  // 这一行支不支持"组件 X 算了答案"？
  strength: number;   // [0,1] 归一化置信
  note: string;
}
```

忠实度聚合函数 `faithfulness()` 只读 `kind === "causal"` 的行（`src/stage08-pipeline.ts:314-325`）。这样设计的目的很具体：后面那个"相关骗局"的失败演示，**在代码层面就不可能**把一行相关证据误算成证明——类型把作弊路径堵死了。

## 流水线 A：在真电路上串六把工具

研究对象是一个冻死的行为对，让 patch 有东西可搬：

```
clean   = [5,6,7]  答案=4
corrupt = [5,2,7]  答案=0   (只在 position 2 = "=" 处可判分)
```

`makeBehavior()` 强制要求 `cleanAnswer != corruptAnswer`（`src/stage08-pipeline.ts:72-85`）。为什么？因为 patch recovery（恢复率）= "复现的 clean→corrupt logit 落差的比例"。如果两个答案相同，落差为零，recovery 的分母为零，整个指标无定义。这是个容易被忽略的前置条件——很多人随手挑一对例子就开跑，结果 recovery 是个除零产物，还以为是信号。

**顺序很关键：先跑因果扫描，让它决定假设。** 不是先看注意力图猜一个头、再去验证它——那样你会被注意力的高 mass 牵着鼻子走，掉进确认偏误。代码先对所有组件做 patch 扫描（`patchScan`，`src/stage08-pipeline.ts:203-214`），取 recovery 最大的那个当假设：

```
[假设] 因果最强组件 = blocks.0.attn.head_z.0  (patch recovery=1.088)
       全组件 patch recovery 扫描 (因果证据):
 b0.h0 | ################################## 1.088
 b0.h1 |  0.012
 b0.h2 |  0.004
 b0.h3 | #### 0.113
 b0.ao | ############################### 0.998
b0.mlp | ##### 0.176
 b1.h0 | ## 0.053
 b1.h1 | ## 0.048
 b1.h2 |  0.016
 b1.h3 | # 0.023
 b1.ao | ############# 0.428
b1.mlp | ############## 0.461
```

`blocks.0.attn.head_z.0`（第 0 层第 0 号注意力头的输出）recovery=1.088——超过 1.0，意思是单独搬这一个头进去，比完整 clean 运行还多恢复了一点点（toy 模型上的过冲，正常）。这就是假设的来源：它是因果上最响的组件，不是看起来最像的组件。

接着六把工具依次给这个假设投票，凑成证据矩阵：

```
[证据矩阵] 假设: "blocks.0.attn.head_z.0 计算了答案"
  technique          kind           metric    support  strength
  -----------------  -------------  --------  -------  --------
  attention          correlational     0.955    yes       0.955   "="查询落在加数{0,1}上的注意力质量
  probe              correlational     1.000    yes       1.000   答案线性可解码 (baseline=0.371)
  logit-lens         correlational     9.660    yes       1.000   答案 logit 从 embed 到末层的净增长
  patching           causal            1.088    yes       1.000   clean<-corrupt 单组件恢复率 (充分性)
  sae                correlational     0.945    yes       0.936   最优特征对单一答案类的选择性 (chance≈0.14)
  ablation           causal            1.670    NO        0.189   mean-ablation 删该头后答案 logit 的下降 (必要性, /gap=8.83)
```

逐行讲它在测什么、怎么会坏：

### attention（相关，0.955）

读缓存里那个头在 `=` 位置（query）对两个加数位置 `{0,1}`（key）的注意力质量（`attentionToAddends`，`src/stage08-pipeline.ts:124-140`）。0.955 意思是这个头几乎把全部注意力都压在加数上——看起来"在读加数"。**怎么坏**：注意力高只说明信息流的拓扑允许这个头读到加数，不说明它读了之后干了什么有用的事。一个把加数拷过来又原样扔掉的头，注意力也可以是 1.0。这是为什么它只是 correlational。

### probe（相关，1.000，baseline=0.371）

在该头所在层的 `resid_post`（注意力写回残差流之后的残差）上，用很多样本训一个线性探针解码 `(a+b)%7`（`probeAnswerAt`，`src/stage08-pipeline.ts:151-166`）。准确率 1.000，对比 shuffle-label 基线 0.371（≈ 1/7 加上有限样本噪声）。**怎么坏**：探针有个经典陷阱——给足够维度，你能从纯噪声里探出"高于随机"的准确率。所以 `linearProbe` 强制带 shuffle 基线，这是第 03 章反复强调的纪律。但即便基线干净、准确率满分，它仍只证明"答案在这个残差流里线性可读"，不证明"这个头把它写进去的"。答案可能是别的组件写的，只是流经了这里。

### logit-lens（相关，净增 9.66）

把每个深度的残差流投过末层 LN + unembed，读 clean 答案 token 的 lens-logit，看它随深度怎么爬（`logitLensTrace`，`src/stage08-pipeline.ts:177-192`）：

```
[logit lens] 答案 token logit 随深度的轨迹 (相关证据):
  answer-logit  :=*@  [first=-0.4705 last=9.1891 min=-0.4705 max=9.1891]
```

从 embed 处的 -0.47 单调爬到末层的 9.19，净增 9.66。这定位了"计算在哪个深度落定"。**怎么坏**：早层的 lens-logit 是拿末层 LN 的统计量去归一化早层残差，分布外（off-distribution），绝对值不可信——第 04 章的告诫在这里仍然成立。**信形状（单调爬升），别信早层绝对值。**

### patching（因果，recovery 1.088，✅充分）

充分性证据。把 corrupt 运行里这个组件的值 splice 进 clean 运行，量恢复了多少落差（`src/stage08-pipeline.ts:430-437`）。1.088 > 0.5 阈值，支持成立。这是六行里第一条真正动了手术的。

### sae（相关，选择性 0.945，chance≈0.14）

在同一残差流上训一个过完备 SAE（sparse autoencoder，稀疏自编码器，把残差拆成稀疏的可解释特征），编码样本，找有没有某个特征专门为单一答案类点火（`saeAnswerFeature`，`src/stage08-pipeline.ts:226-267`）。选择性 0.945 远超随机 1/7≈0.14——存在一个近乎"答案单义"的特征。**怎么坏**：注意 strength 算法（`src/stage08-pipeline.ts:446`）把 chance baseline 减掉再归一，所以 0.945 的选择性归一后是 0.936 而非 0.945——别把原始选择性当置信度。更要紧的诚实点：toy 模型上 SAE 轻松找到干净原子，真模型上这是公认的难活（下面专门讲）。

### ablation（因果，drop 1.670，❌不支持，/gap=8.83）

必要性证据，也是这条流水线最有教育意义的一行。mean-ablation（用该组件激活的均值替换它，比直接清零更少分布外——清零一个归一化过的流会因为与电路无关的原因看起来灾难性地崩，第 05 章讲过）删掉这个头，量答案 logit 掉多少（`ablationDrop`，`src/stage08-pipeline.ts:277-280`）。

关键在归一化基准：必要性 = drop / **gap**，gap 是 clean→corrupt 的 logit 落差 8.83（`behaviorGap`，`src/stage08-pipeline.ts:285-290`），和 patch 用的是**同一把尺**。所以两个因果数能直接比，不是各活在各的坐标系里。这里 drop=1.670，必要性 = 1.670/8.83 ≈ 0.189，低于 0.5 阈值，**support=NO**。

这一行就是这条流水线最诚实的地方：因果最强的那个头，删掉它，答案只掉了不到两成。

## 关键判定：不要把充分性和必要性揉成一个数

到了聚合环节。一个图省事的做法是把充分性和必要性乘起来出一个"忠实度"标量。代码**故意拒绝**这么做（`src/stage08-pipeline.ts:292-325` 的长注释解释了原因），改成两问分开看：

```
[端到端忠实度] 因果两问分开看 (不合并成单一乘积):
  充分性 (patch recovery) = 1.000
  必要性 (ablation drop)  = 0.189
  因果支持 = 1/2   相关支持 = 4/4
  判定: ◐ 充分但非唯一必要 ⇒ 电路冗余 (有并联组件重算答案); 解释方向对, 但「该头独占答案」过强
```

为什么不能乘？因为 1.0 × 0.189 ≈ 0.19 这个"低忠实度"会**冤枉真电路**。这个头确实充分（搬它进去答案就回来），它只是不**唯一**必要——删掉它,有个并联组件把答案重算了一遍，所以单删它行为没崩。这是冗余电路（redundant circuit）的典型签名，不是解释错了。

诚实的读法是 `faithfulness()` 给出的：2-of-2 / 1-of-2 / 0-of-2 的门控，把冗余**报告出来**而不是当失败惩罚（`src/stage08-pipeline.ts:314-325`）。判定 `◐` 的意思是："解释方向对，但'该头独占答案'这个说法过强"。这比一个 0.19 的标量信息量大得多——标量只会让你以为解释失败，门控告诉你解释是对的、电路是冗余的。

把单一忠实度数字当**摘要**可以，当**证明**不行。这是从 toy 能带走的第一条方法论。

## 流水线 B：故意造一个"相关全绿、因果全红"的骗局

现在演示这章存在的全部理由。`runDecoyDemo`（`src/stage08-pipeline.ts:501-592`）在模型里搜一个"诱饵"组件（decoy）：相关证据全绿（注意力高 + 探针可解码），因果证据全红（patch / ablation 都≈0）。挑法是在因果无关的头里选注意力对加数最高的那个——"看起来最负责、实际做得最少"（`src/stage08-pipeline.ts:528`）。

跑在和流水线 A **同一个行为**上，所以对比纯粹是"哪把工具认同"，不掺"哪个例子"的变量：

```
=== 失败模式: 相关全绿 / 因果全红的 decoy ===
decoy 组件 = blocks.0.attn.head_z.1  (区别于真因果头 blocks.0.attn.head_z.0)
[证据矩阵] 假设: "blocks.0.attn.head_z.1 计算了答案" (只看相关证据会接受)
  technique          kind           metric    support  strength
  -----------------  -------------  --------  -------  --------
  attention          correlational     1.000    yes       1.000   注意力高 → 看起来"在读加数"
  probe              correlational     1.000    yes       1.000   答案可线性解码 → 看起来"含答案"
  patching           causal            0.012    NO        0.012   splice 进去几乎不动答案 → 不充分
  ablation           causal            0.064    NO        0.007   删掉它答案几乎不掉 → 不必要 (/gap=8.83)
相关证据: 2/2 条支持 (若在此停手 → 错误地接受该解释)
因果证据: 充分性=0.012 必要性=0.007 (0/2) → 否决
```

看清楚这个对比：`blocks.0.attn.head_z.1` 的注意力 **1.000**（比真头的 0.955 还高），探针 **1.000**。如果你是那种画完注意力图、跑完探针就收工的人，你会**百分百自信地命名这个头**，并且觉得证据扎实——两条相关证据全绿。

但 patch recovery 只有 0.012，ablation drop 折算必要性 0.007。把它 splice 进去答案纹丝不动，删掉它答案也不掉。**它跟答案共变，但它不算答案。**

为什么相关工具会被同时骗过?因为这个诱饵头在真电路的**下游或并联**位置：真电路已经把答案算出来了，答案信息自然流经/共变于这个头，于是注意力看着像在读加数、探针看着像含答案——它们被**同一个混淆变量**（真电路的输出）一起骗了。多个相关工具一致 ≠ 因果成立，因为它们可以被同一个 confound 同时点亮。只有干预能打破并列：把这个头的值换掉/删掉,答案不动,真相暴露。

**缺了干预环节的解释会"自洽地"骗过你。** 这就是为什么 AI 生成的可解释性 survey 读起来流畅却经不起推敲——它们停在相关层，永远不闭因果环，于是每个诱饵都被收编成"证据"。

## 诚实边界：toy 给你方法论，不给你结论 ⚡

到这里我必须把全书的诚实账一次结清，否则这章就成了它批判的那种东西。

**toy 数字偏乐观，且系统性偏乐观。** 这条流水线在 modAdd 上跑得这么干净——单个 position 可判分、常常单组件主导、recovery 接近甚至超过 1.0、SAE 一抓一个准——是因为任务小、模型小、电路浅。这些数字**不可迁移**。

**哪些可迁移、哪些不可，要分清楚：**

- **可迁移（机制 / 曲线形状 / 方法论）**：logit lens 随深度单调爬升的形状；patch 与 ablation 是充分性 vs 必要性两个正交问题；冗余电路表现为"充分但非唯一必要"；相关工具会被共同 confound 同时骗过——这些是结构性事实，到真模型仍成立。
- **不可迁移（绝对值）**：recovery≈1.0、探针 1.000、SAE 选择性 0.945 这些数,真模型上别指望。真模型上各工具**经常互相矛盾**——探针亮的地方 patch 找不到组件、ablation drop 对不上 patch recovery、电路是分布式的而非单头主导。

**机制可解释性当前没解决的硬骨头（明确：目前无通用解 / 正在研究）：**

- **叠加（superposition）未解**：模型把远多于神经元数的特征塞进同一组神经元，导致单个神经元多义（polysemantic）。SAE 是当前最被寄望的解法，但它本身的评估（特征是否真单义、字典够不够大、有没有特征分裂）仍是开放问题。
- **自动化电路发现规模化**：本章的 patch 扫描是暴力枚举所有组件——在 toy 上 19 个激活点能扫完，在前沿大模型上组合爆炸。ACDC、attribution patching 这类自动化方法在推进，但"如何在不暴力枚举下可靠定位电路"没有定论。
- **解释本身如何评估，是元问题**。这是最扎心的一条:我们用什么标准说一个解释"忠实"?causal scrubbing 这类方法尝试用"按解释重排激活后 loss 恢复多少"来打分,但它依赖你先写对解释的结构,而且 loss 恢复高不等于解释对(可能只是抓住了冗余)。解释的**可证伪性**、对**对抗样本**的鲁棒性,目前都没有公认指标。本章那个 2-of-2 门控只是 toy 上的玩具版,真问题远没解决。

**最后,把赌注说清楚。** 机制可解释性对 AI 安全/对齐的赌注是:能不能在部署**之前**,从模型内部查出它有没有欺骗性对齐(deceptive alignment——训练时装乖、部署后变脸)?如果能,我们就有了行为测试给不了的内部证据;如果不能,可解释性就停留在"事后理解小模型"的学术阶段。诚实的现状是:**我们还远不能**。在 toy 上把一个加法头的电路拆清楚,和在一个千亿参数模型里查出隐蔽的欺骗意图,中间隔着上面那几个未解的硬骨头。任何把 toy 成功外推成"可解释性已解决"的说法,都是这本书从第一章就要你警惕的那种 AI smell。

可迁移的不是数字,是纪律:**永远用干预闭环、报告分歧而不是挑一致的工具、把单一忠实度数字当摘要而非证明。** 这三条,从 modAdd 到 GPT,不变。

把这条 checklist 钉在墙上,你就从"画热力图讲故事的人"变成了"做反事实实验的人"。这是这本书想交给你的全部——剩下的硬骨头,在真实研究前沿等你。
