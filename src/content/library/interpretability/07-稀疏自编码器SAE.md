先把一个让很多人栽跟头的事实摆在前面：**你打开一个训练好的 transformer，挑一个 MLP 神经元想知道它代表什么，大概率得到的答案是「它什么都不代表，又什么都代表」**。同一个神经元会对「法语文本」「DNA 序列」「HTTP 状态码」三件毫不相关的事情同时激活。这不是模型训坏了，这是叠加（superposition，把多于维度数的特征压进同一个向量空间）的必然后果——一个 d 维残差流（residual stream，每层读写的共享向量）里，模型学会了塞进远多于 d 个的特征，代价就是没有任何一个坐标轴对应一个干净的概念。

前面几章我们一直按「组件」切模型：第 02 章看注意力头在看哪里，第 04 章用 logit lens 看每层往输出方向写了什么，第 05 章用 activation patching 定位哪个位置承载了因果。这些都是**组件级**解释——我能告诉你「第 1 层第 3 个头负责把 b 搬到 = 位置」。但组件级有个天花板：组件是模型的硬件单位，不是它的概念单位。真正的「概念」（modAdd 里就是「答案等于几」这件事）是分布在多个神经元上、又和别的概念挤在一起的。本章要换一把刀：不按硬件切，按**特征**切。

稀疏自编码器（SAE，sparse autoencoder，一种带稀疏约束的「压了又解」网络）就是这把刀。它的赌注是：虽然激活向量本身是叠加的、不可读的，但如果我用一个**过完备**（字典原子数 > 维度数）且**稀疏**（每个样本只点亮少数原子）的字典去重写它，那么单个字典原子有机会是**单义**的——只对应一件事，可以命名。本章配套代码在 `examples/interpretability-from-scratch/src/stage07-sae.ts`，研究对象还是第 01 章那个 modAdd(7) 的 toy transformer（同一个 checkpoint，`SEED=1234` 同种子，结论可比），采样点是 `blocks.1.resid_post`——最后一层 MLP 写完、unembedding 即将读取的位置，最「富含特征」的地方。

下面所有数字都来自这份代码的真实运行；我会把 toy 上「干净得不真实」的部分和能迁移到真模型的「形状」分开标。

## 为什么不能直接读神经元：叠加把账算给了你

先说清楚 SAE 要解决的到底是什么问题，否则它看起来只是个莫名其妙的自编码器。

假设模型内部真有 N 个互不相关的特征要表示，但残差流只有 d 维，且 N ≫ d。线性代数告诉你：d 维空间最多放 d 个正交向量。模型怎么办？它把特征放成**几乎正交但不完全正交**的方向，靠「同一时刻只有少数特征同时激活」这个稀疏假设来容忍干扰——这就是叠加。Anthropic 的《Toy Models of Superposition》(2022) 把这件事用玩具模型量化得很透：当特征稀疏时，模型能塞进的特征数远超维度数，神经元因此天然多义。

这对解释意味着：**神经元基（你能直接读的那组坐标轴）不是特征基**。你读神经元，读到的是好几个特征的线性叠加，像收音机串台。所以解释的前提是先做一次**基变换**——把激活从「神经元坐标」转回「特征坐标」。SAE 就是用数据学这个变换：它不知道真特征是什么，但它知道「真特征应该稀疏」，于是用稀疏性当唯一的归纳偏置去把基掰回去。

为什么用「过完备 + 稀疏」而不是普通 PCA（主成分分析，找方差最大的正交方向）？因为 PCA 强行要求正交、个数 ≤ d，正好和叠加的前提对着干——叠加的特征本来就不正交、个数 > d。SAE 反其道：字典开到 4×d（本章 dModel=32，nFeatures=128），允许非正交，只压稀疏。这是它能从叠加里捞出单义原子的根本原因。

## SAE 的最小实现：一层编码、一层解码、一个 L1

别被「自编码器」吓到，这里的 SAE 朴素到只有两个矩阵。看 `core/interp.ts::trainSAE` 的训练循环核心：

```ts
// encode: features = relu(X @ Wenc + bEnc) ; relu 强制非负码（原子只做加法）
const pre = X.matmul(Wenc).add(bEnc.broadcastRow(samples));
const feats = pre.relu();
// decode: recon = feats @ Wdec + bDec
const recon = feats.matmul(Wdec).add(bDec.broadcastRow(samples));
// recon loss = MSE(recon, X)
const diff = recon.sub(X);
const reconLoss = diff.mul(diff).mean();
// L1 稀疏惩罚（relu 后非负，mean abs == mean）。就是这一项逼每个输入只用少数原子，
// 从而让原子单义。
const l1Loss = feats.sum().mulScalar(l1 / samples);
const loss = reconLoss.add(l1Loss);
```

就两件事：`feats = relu(X·Wenc + bEnc)` 是编码（把激活打成 128 维稀疏码），`recon = feats·Wdec + bDec` 是解码（用字典原子线性重组回 32 维）。损失也就两项：重构 MSE（解码得像不像原激活）+ L1 惩罚（码要稀疏）。

三个设计决策值得停下来讲清楚「为什么」，因为每一个都对应一种坏掉的方式：

**为什么 relu？** relu 把码钳在非负。这不是随手选的——它让「原子只做加法」：一个特征要么不参与（码=0），要么往激活里加它那个方向，不能减。这对应「特征是存在性的」直觉（「这是法语」要么有要么没有，没有「负的法语」）。更实际的好处是 relu 天然产生**硬零**，L0（每样本平均激活的原子数）才有意义；如果用 tanh，码永远是小非零数，稀疏就成了空话。

**为什么 L1 而不是 L0？** 我们真正想要的是 L0 小（激活的原子个数少），但 L0 不可导，没法梯度下降。L1（码的绝对值和）是 L0 的凸松弛——它可导，且会把小码往 0 推。代价是 L1 有**收缩效应（shrinkage）**：它不光惩罚「激活了多少个」，还顺手把激活了的那些码值整体压小，导致重构系统性偏弱。这个收缩 bug 正是后面 Gated/JumpReLU SAE 要修的东西，待会儿前沿一节展开。

**为什么字典要过完备到 4×？** 因为叠加里真特征数 > 维度数。字典如果只有 d 个原子，等于又回到「正交基不够用」的死胡同。开到 4×d 给模型留出冗余，让它能把叠在一起的特征分到不同原子上。`nFeatures = model.cfg.dModel * 4` 这行就是这个意思。倍率不是越大越好——太大死特征会爆炸（capacity 用不满），这是个要调的超参。

## ✦ 真正的指标是 Pareto 前沿，不是重构 loss 一个数

这一节是本章的专家钩子，请认真看。

**新手评估 SAE 只看一个数：重构 loss 低不低。这是错的，而且错得很危险**——因为重构 loss 可以靠「退化成稠密多义」刷低。我把 L1 惩罚一调小，每个样本激活几十个原子，重构当然完美，但这时候 SAE 啥也没拆出来，每个原子还是多义的串台货。重构 loss 漂亮，解释价值为零。

正确的评估是同时看三个数，它们构成一条**重构-稀疏 Pareto 前沿**（一头换另一头、没有免费午餐的权衡曲线）。`stage07-sae.ts` 的第 [1] 步就是扫这条前沿：固定 nFeatures=128、各训 800 步，只动 L1，打印 FVU、L0、死特征三件套。真实运行输出：

```
[1] 重构-稀疏 权衡扫描 (nFeatures=128, 4x 过完备; 各 800 步训练):
    L1=0.001  | FVU=0.0000  L0= 18.76  死特征=  2.3%  存活=125/128
    L1=0.01   | FVU=0.0005  L0=  4.82  死特征= 43.8%  存活=72/128
    L1=0.05   | FVU=0.0175  L0=  2.71  死特征= 81.3%  存活=24/128
    L1=0.2    | FVU=0.6245  L0=  0.29  死特征= 98.4%  存活=2/128
```

逐个读懂这三个数：

- **FVU（Fraction of Variance Unexplained，未解释方差比例）**：重构质量的归一化版。0 = 完美重构，1 = 还不如直接输出均值。这里用 FVU 而不是裸 MSE，是因为 MSE 的绝对值依赖激活的尺度，没法横向比；FVU 把它压到 [0,1]，一眼知道好坏。代码里是 `fvu = sse / sst`（误差平方和 ÷ 围绕每维均值的总平方和）。
- **L0**：每个样本平均点亮几个原子。这是稀疏的真度量。L0=18.76 意味着「128 个原子里平均同时用 19 个」——这根本不稀疏，离「单义」远着呢。L0=2.71 才像样：平均 3 个原子拼出一个激活。
- **死特征比例**：整个数据集上从没激活过的原子占比。这是过稀疏的报警器。

现在看这四行讲的故事：**L1 从 0.001 涨到 0.2，FVU 从 0.0000 一路涨到 0.6245（重构越来越烂），L0 从 18.76 掉到 0.29（越来越稀疏），死特征从 2.3% 飙到 98.4%（字典越来越空）**。这就是 Pareto 前沿——你没法同时要「重构完美」和「极度稀疏」，L1 是那个滑块。

ASCII 条形图把 L0 随 L1 的塌缩画得很直观：

```
L1=0.001 | ################################ 18.755
 L1=0.01 | ######## 4.816
 L1=0.05 | ##### 2.714
  L1=0.2 |  0.286
```

关键判断：**L1=0.001 那行重构 FVU=0.0000 看着最香，但它是个陷阱**——L0=18.76 说明它退化回稠密了，每个样本用 19 个原子，原子之间还是叠加态，拆不出单义。L1=0.2 那行又过了头，死特征 98.4%，字典基本空了。能用的「knee（拐点）」在 L1=0.01 附近：FVU=0.0005 重构仍极好，L0=4.82 已经够稀疏，死特征 43.8% 虽不低但还留着 72 个活原子够命名。所以下一步代码选的就是 `chosenL1 = 0.01`。

这就是为什么我说**看一个重构 loss 会骗你**：L1=0.001 和 L1=0.01 的 FVU 差了 0.0005，肉眼几乎没差别，但 L0 差了 4 倍、可解释性天壤之别。判断 SAE 好坏的是这条前沿的形状和你站在哪个点，不是终点那个最低的 loss 数。死特征比例则是这条前沿上「你是不是已经走过头」的硬警报——后面失败模式一节会把它推到极端。

（toy 诚实标注：真模型上这条前沿的死特征比例在「健康」区间也常常高得多，且 knee 没这么干净。这里 L1=0.01 能同时拿到 FVU=0.0005 和 72 个活原子，是因为 toy 真特征少、叠加浅。能迁移的是**方法**：永远扫 L1、永远三个数一起看、永远盯死特征，而不是这几个具体数值。）

## 命名一个特征：看它的最大激活样本

拆出稀疏字典只是第一步，光有 128 个原子不算解释——你得能说出某个原子「是什么」。命名特征的标准动作是 Anthropic《Towards Monosemanticity》(2023) 立下的：**看哪些输入最强烈地激活它，找它们的共同点**。

modAdd 这个 toy 的好处是「共同点」有 ground truth：每个输入 (a,b) 的答案就是 (a+b)%7。所以代码的 `nameSharpestFeature` 不是随便挑一个原子画图，而是遍历所有活原子，对每个算它 top-7 激活样本的**答案纯度**（这些样本里答案最多的那一类占多大比例），挑纯度最高的那个原子来命名——纯度本身就是诚实性检查：纯度高才说明这原子真的只对应一件事，纯度低就是「我画了热图但这特征根本不单义」。

真实运行（选定 L1=0.01 的字典）：

```
[2] 选定 L1=0.01 的字典, 命名最尖锐的特征:
      SAE 重构 loss @                      [first=2.9465 last=0.0010 min=0.0010 max=2.9465]
    特征 #23: top-7 激活输入里 100% 的答案都是 (a+b)%7 = 4  ⇒ 命名为「答案=4 特征」
```

重构 loss 从 first=2.9465 收敛到 last=0.0010，训练是收敛的。然后特征 #23 的 top-7 激活样本里 **100%** 的答案都是 4——纯度满分，于是命名为「答案=4 特征」。代码把这 7 个样本画成 ASCII 热图，行是样本、列是 [a, b, ans]：

```
    特征 #23 的 top-7 激活输入 (每行一个样本):
    aba
s41 %@*
s10 :+*
s16 --*
 s4  **
s47 @%*
s28 * *
s22 +:*
  scale: ' .:-=+*#%@'  vmin=0.000 vmax=6.000
```

读这张图的诀窍在最后一列 `ans`：七行的 `ans` 列全是 `*`（同一个色阶值），说明这七个输入答案都一样——这就是「单义」的视觉证据。而 `a`、`b` 两列五花八门（`%@`、`:+`、`--`...），说明这个特征不在乎 a、b 各自是几，只在乎它们的**和模 7 等于 4**。一个真正抽象的「答案类」特征，正是 modAdd 这个任务里你希望模型学到的东西。

⚡ 但这里必须诚实划界，而且这是本章最重要的一句免责：**toy 上「100% 纯度、一眼可命名」是奢侈品，真模型上不是常态**。真模型的 SAE 字典动辄百万原子，其中**大量原子至今没人能命名**——它们激活的 top 样本看不出任何人类可懂的共同点。命名能做到的覆盖率本身就是个开放问题。更糟的是**特征分裂（feature splitting）**：同一个语义概念可能被字典拆成好几个高度相关的原子（「学术英语」裂成「学术英语-引言」「学术英语-方法」...），字典越大裂得越碎，到底算几个特征没有定论。所以你在论文里看到的漂亮单义特征是**精选**出来的，分母（不可解释 + 分裂的原子）很少被展示。这是 SAE 作为解释手段最诚实的软肋，下面前沿一节还会回到它。

## 验证因果：消融和放大同一个特征

命名了「答案=4 特征」还不够。命名只证明了**相关**——这个原子和答案 4 同时出现。但相关不是因果：也许它只是答案 4 的某个无辜伴随物，模型根本没用它做预测。要证明因果，得动手干预，看模型的预测会不会跟着动。这正是第 05 章 activation patching 的逻辑，只不过这次干预的不是某个组件，而是**特征空间里的一个坐标**。

`interveneOnFeature` 的做法：对一个真答案为 4 的输入 [0,4,7]，把 resid_post 编码成稀疏码，然后

- **消融**：把「答案=4 特征」那一维的码置 0，解码回 resid_post，splice 进模型重跑；
- **放大**：把那一维的码乘 4 加 1（`c[feature] * 4 + 1`）再解码重跑。

这里有个**至关重要的控制**，初学者最容易漏：基线不能用模型的真实 logit，必须用 **SAE 没编辑过的重构** 的 logit。看代码注释点破的理由：

```ts
const reconLogit = runWith(decodeRow(() => {})); // 未编辑的重构 = 基线
const ablatedLogit = runWith(decodeRow((c) => (c[feature] = 0))); // 杀掉这个特征
const boostedLogit = runWith(decodeRow((c) => (c[feature] = c[feature] * 4 + 1))); // 放大它
```

为什么基线要用「未编辑的重构」而不是「真模型」？因为 SAE 重构本身有误差（FVU=0.0005 不是 0）。如果拿真模型 logit 当基线，你测到的落差会**混入 SAE 自己的重构误差**，分不清是「编辑特征」造成的还是「SAE 重构不准」造成的。用未编辑重构当基线，则消融/放大相对它的偏移**只能归因于那一次编辑**——这是把因果和重构噪声切干净的唯一办法。

真实运行：

```
[3] 因果干预 (输入 [0,4,7], 答案 token=4):
clean(真模型) | ############################## 9.251
 SAE重构(基线) | ############################## 9.256
     消融该特征 | ############# 4.037
     放大该特征 | ############################## 9.311
    消融落差 = 5.219 (>0 ⇒ 该特征在把答案 logit 撑高)  放大增益 = 0.055
    判定: 因果成立 ✓ 消融压低、放大抬高同一答案 logit
```

读这四个数：clean=9.251 和 SAE 重构基线=9.256 几乎重合，说明 SAE 重构没破坏模型预测（基线可信）。消融该特征后答案 4 的 logit 从 9.256 掉到 **4.037**，落差 5.219——杀掉这一个原子，模型对正确答案的信心崩了一半多。放大后涨到 9.311（增益 0.055，小但为正，因为 logit 已经接近饱和，没多少头部空间）。

判定逻辑是 `causal = ablationDrop > 0.05 && boostGain > 0`——**双向都得对**才认。只消融下降可能是巧合（删任何东西都会扰动），只放大上升也一样；唯有「删它降、加它升」同时成立，才能说这个特征**因果地**在驱动「答案=4」这个预测。这下「答案=4 特征」从一个相关命名升级成了一个因果机制。

（toy 诚实标注：消融落差 5.219 这么大、放大也立刻见效，是因为 toy 里这个特征几乎一对一地承载了答案信息。真模型里特征更冗余、更分布式，删一个原子往往只让 logit 动一点点，因为别的原子会顶上——因果效应真实存在但量级小得多，需要更仔细的统计。能迁移的是**实验设计**：编码-编辑-解码-splice、用未编辑重构当基线、要求双向一致。)

## 失败模式：L1 拉到极端，字典塌缩

前面 Pareto 前沿已经预告了「L1 太大死特征会爆」。这一节把它推到崩溃，因为这是真实 SAE 训练里**最常见的翻车方式**，且它反直觉。

直觉上你可能觉得「稀疏是好事，L1 越大越好，逼出更干净的特征」。错。L1 是惩罚项，调到一定程度后，**让原子保持死亡（永不激活）比让它激活更省 loss**——因为激活就要交 L1 税，而这个原子贡献的重构改善还抵不过税钱。于是优化器干脆让一批原子直接死掉。L1 再大，死的越多，最后字典整个塌缩。代码第 [4] 步把 L1 拉到 5.0：

```
[4] 失败模式: L1 惩罚拉到极端 (l1=5.0) → 字典塌缩:
    死特征=100.0%  存活=0/128  L0=0.00  FVU=1.000
    后果: 全部特征死亡, nameSharpestFeature 抛错 — 没有特征可命名
    教训: L1 不是越大越好; 过稀疏让字典塌缩, 拆不出特征 (欠拟合而非过拟合)。
```

**死特征 100%，存活 0/128，L0=0.00，FVU=1.000**——字典彻底空了，重构退化成只输出 bias（FVU=1 等于「还不如均值」）。`nameSharpestFeature` 里那行 `if (!best) throw new Error("no live feature to name — dictionary fully collapsed")` 直接抛错：没有任何活原子可命名。

关键认知，也是很多人搞反的地方：**这是欠拟合，不是过拟合**。一般「正则化」给人的印象是防过拟合，多多益善。但 SAE 的 L1 过强是把模型逼到**连训练数据都重构不出来**——它牺牲的不是泛化，是表达能力本身。所以 L1 调参不是「越大越安全」，而是要卡在前沿的 knee 上：稀疏到能拆出单义，但还没稀疏到把字典饿死。死特征比例就是你的仪表盘——它从个位数往上爬到几十、逼近 100% 的过程，就是字典从健康走向塌缩的过程。

实操上，真模型训 SAE 时对付死特征还有专门的手段（resampling 重置死原子、ghost grads 给死原子一点梯度、调整 L1 warmup），因为死特征在真模型上即使 L1 不极端也会自然累积。toy 这里只演示了「极端 L1 → 全死」这个最干净的因果，但形状是真的：**死特征是 SAE 训练的头号慢性病**。

## ⚡ 前沿：从 Monosemanticity 到 Gated/JumpReLU/TopK，以及没解决的事

SAE 是 2023 年以来对齐圈（alignment，让 AI 行为可控可懂的研究）最热的解释手段，进展很快，但远没尘埃落定。给你一张当前地图，以及明确标注哪些是开放问题。

**演进主线，核心都在修 L1 收缩。** 前面讲过 L1 有收缩 bug：它在「该不该激活」之外还顺手把激活值压小，导致重构系统性偏弱、特征被迫分裂。三个后续方案都冲着这个去：

- **Gated SAE**（DeepMind 2024）：把「要不要激活」（门控）和「激活多少」（幅度）拆成两条路，L1 只罚门控、不罚幅度，从而去掉幅度上的收缩。
- **JumpReLU SAE**（DeepMind 2024）：用带阈值的跳变 relu，码要么是 0、要么直接跳到真实幅度，没有被 L1 压扁的中间地带。
- **TopK SAE**（OpenAI 2024）：最干脆——直接每个样本只保留最大的 K 个码、其余强制清零，把 L0 变成你直接设定的超参 K，根本不用 L1。代价是 K 是硬超参、不能自适应，且 TopK 也有自己的死特征问题。

本章配套代码留了一个 **TopK SAE 的最小实现作为对照**，让你能在同一个 toy 上对比「L1 软约束」和「TopK 硬约束」两条路在 L0、死特征、可命名性上的差异——把 L1 那套的「滑块调稀疏」换成「直接钉死激活个数」，亲手感受两种稀疏哲学。

**已经能做但仍粗糙的：特征引导（feature steering）。** 既然能因果地放大/消融一个特征（本章第 [3] 步就是 toy 版），那在真模型上放大「拒绝特征」「奉承特征」「某主题特征」就能在推理时**引导**模型行为——Anthropic 2024 的 "Golden Gate Claude"（把「金门大桥特征」夹高，模型逢话题就扯到金门大桥）就是最出圈的演示。这条线从解释跨到了控制，是 SAE 最有应用想象力的方向。

**明确没有通用解的开放问题（⚡ 正在研究）：**

1. **可命名性覆盖率。** 真模型百万级字典里，能被人类清楚命名的原子占多少、剩下那一大片不可解释的原子是「真特征但我们看不懂」还是「SAE 的伪影」——没有共识，也没有公认的自动评估指标。本章 toy 上 100% 纯度的奢侈在真模型上不存在。
2. **特征分裂没有停止判据。** 字典开多大才「对」？开大了语义碎成相关原子，开小了又欠分解，目前靠经验和下游任务凑，没有原理性的答案。
3. **SAE 学到的是模型的特征，还是数据的特征、抑或 SAE 自己造的特征？** 这是最根本的怀疑：一个用稀疏先验硬拟合出来的字典，凭什么相信它捞出的是模型「真在用」的计算单元，而不是恰好能稀疏重构激活的某组方便基？因果验证（本章第 [3] 步、真模型上的 patching）是目前最硬的证据，但它只能逐个特征验、没法保证整本字典都对。

所以 SAE 现在的位置是：**一个非常有用、但你必须带着怀疑用的工具**。它把组件级解释推进到了特征级，给了对齐研究第一个能规模化拆解叠加的抓手；但「这本字典真的就是模型的概念词典吗」这个问题，还没有人能拍胸脯回答。

---

回到全书的脉络：第 02 到 05 章我们用注意力、logit lens、patching 把模型当成由**组件**组成的硬件来读；这一章 SAE 把刀换成了**特征**，从硬件单位下沉到概念单位。两套视角是互补的——组件告诉你「计算在哪儿发生」，特征告诉你「计算的是什么概念」。可迁移的就这三条形状，记死它们比记本章任何一个 toy 数字都值钱：**过完备 + L1 → 更稀疏更可解释的基；L1 过大 → 字典塌缩（欠拟合）；真特征 → 因果而非相关的 logit 效应**。下一章（第 08 章）会把这些手法串成一条端到端的解释流水线。
