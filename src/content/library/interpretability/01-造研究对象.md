先说一个会让人不舒服的结论：可解释性研究里最容易被跳过、也最致命的一步，不是画 attention 热图，不是训探针，而是**确证你手上这个模型真的学会了那个算法**。如果它只是把训练集背成了一张查找表，你后面所有的电路（circuit，模型内部完成某个子计算的一小撮权重和激活的固定通路）分析都是在给噪声做考古——拆出来的"机制"是记忆碎片的投影，换个输入就散架，更别说迁移到真模型。本章不碰任何 attention，也不碰任何探针。它只干两件事，但这两件事不做扎实，第 02 章往后全是空中楼阁：**证明求梯度的引擎是对的**，**用一个能精确算出正确答案的任务证明模型是泛化而不是记忆**。

对应代码是配套仓的 `examples/interpretability-from-scratch/src/stage01-train-toy.ts`，`npm run stage01` 直接跑，CPU 几秒到二十秒出全部结果，零运行时依赖。下面的数字全部来自这次真实运行，估算和 toy 偏差我会明确标。

## 为什么是模运算，而不是随便挑个任务

任务选的是 `modAdd(23)`：给一个序列 `[a, b, =]`，在 `=` 这个位置预测 `(a+b) mod 23`。词表 24（0..22 是数字，23 是 `=` 分隔符），全部 pair 一共 23×23=529 个，随机猜中的概率 chance（基线命中率）= 1/23 ≈ 0.043。看着像玩具，但选它有一个不可替代的理由，这也是 ✦ 本章的专家钩子。

模加法**不是随意挑的**。它的最优解有一个干净的代数结构（algebraic structure，可以用数学公式精确写出来的内部组织）：把每个数字嵌到一个圆上（傅里叶基，Fourier basis，用一组正弦余弦把信号分解的坐标系），加法就变成转角度。这件事的分量在于——它给了"学会了"一个**可精确对照的地面真值**（ground truth，已知的标准答案），而不是靠 loss 低就拍脑袋说学会了。`core/tasks.ts` 的注释把这条设计意图写死了：

```ts
// WHY this task is a famous interp object: the optimal solution has a clean ALGEBRAIC
//   structure (the model learns to embed numbers on a circle and add angles). Stages can
//   check whether the learned embeddings show that circular structure — a mechanism that
//   is identical in kind at any scale, even though tiny p makes it trivially small.
```

这一点是全书的方法论种子：第 03 章的探针（probe，一个小分类器，用来检查某层激活里是否线性可读出某个量）会直接去模型的 embedding 里验这套圆环/傅里叶结构。如果探针真能从权重里读出周期性，那就是"模型确实学了角度加法"的硬证据，而不是我们一厢情愿的解读。**任务的代数结构 = 可解释性结论的可证伪性**。换个没有干净最优解的任务（比如随机映射表），你连"它该长什么样"都说不出，自然也无法判断拆出来的东西对不对。

再看 task 怎么定义 oracle（地面真值函数）——这是"精确判定"的物理基础：

```ts
export function modAdd(p: number): Task {
  const EQ = p; // delimiter token id
  const vocab = p + 1;
  return {
    name: `modAdd(${p})`, vocab, seqLen: 3,
    // ...
    oracle(input) {
      const [a, b] = input;
      // Targets at positions 0,1 are "don't care"; scorablePositions ignores them.
      return [EQ, EQ, (a + b) % p];
    },
    scorablePositions() { return [2]; },  // 只有 "=" 位置有确定答案
  };
}
```

注意 `scorablePositions` 只返回位置 2。前两个位置没有确定答案（看到 `a` 你无法预测 `b`），如果把它们也算进准确率，会用不可学的噪声稀释指标、**掩盖电路到底工作没有**。这是个反复出现的坑：评测位置选错，模型明明学会了核心结构，准确率却被噪声位置压低，你误判成"没学会"。只在有确定答案的位置打分，是让指标诚实的前提。

## 引擎不对，后面全是虚构

在训练之前，stage01 做的第一件事是全模型梯度检查（gradient check，用数值微分验证手写的解析梯度对不对）。这一步看着像形式主义，其实是整本书的拱顶石。理由很直接：第 04 章 logit lens、第 05 章 activation patching、第 06 章 induction head，每一个结论都建立在"梯度/激活流经这里"之上。如果你的 autograd 有一个符号错、一个忘了累加的 `+=`、一个广播没求和，那么"梯度说这条路重要"就是**虚构**——你会很自信地解释一个根本算错的数。

`core/autograd.ts` 的 `gradCheck` 用中心差分（central difference，两边各扰动一点再求斜率，误差 O(eps²)）对照解析梯度：

```ts
const numeric = (lp - lm) / (2 * eps);     // 中心差分
const analytic = p.grad[i];
const denom = Math.max(1e-8, Math.abs(numeric) + Math.abs(analytic));
const rel = Math.abs(numeric - analytic) / denom;
```

为什么是中心差分而不是前向差分？因为它的截断误差是 O(eps²)，f64 精度下一个**正确**的算子能给出 < 1e-6 的相对误差，而一个**有 bug** 的算子误差是 O(1)——两者差好几个数量级，不会模棱两可。这正是它能抓住"忘了广播求和/符号反了/漏了累加"的原因：这类 bug 不会让误差稍微变大，而是让它彻底爆掉。stage01 在 `gradCheckModel` 里复用了**训练器真正用的那条损失**（在 scorable 位置上的 crossEntropy），所以检查的是真实计算图，不是另写一个简化版自欺欺人。

这次运行的实测结果：

```
[1] 全模型梯度检查 (中心差分 vs 解析梯度):
    maxRelError = 6.119e-9  (checked 36 个参数元素)
    判定: 通过 ✓ 引擎可信
```

`6.119e-9` 远低于 1e-4 的判定阈值，引擎可信。**失败模式**长什么样：如果某个算子梯度写错，这里会跳到 0.1、0.5 甚至更大，判定直接打印"失败 ✗ 引擎有 bug"。一个诚实的坑是 `gradCheck` 在 relu 的拐点（kink，函数不可导的折点）处会和子梯度对不上——所以测试输入要避开恰好为 0 的拐点，这是 `core/autograd.ts` 注释里明确标注的 caveat。引擎不过这一关，后面别谈解释。

## train acc=1.0 不等于学会：grok 的相变曲线

这是本章的核心实验，也是 ⚡ 本章的前沿钩子。模运算正是 grokking（延迟泛化，训练准确率早早满分、测试准确率在大量步之后才突然跃升）现象的经典载体，2022 年后它是可解释性圈最热的实验台之一。stage01 要在这个 toy 上复现那条"先记忆后泛化"的相变（phase transition，指标突然跳变的临界过渡）曲线。

复现它有一个决定性的设计选择：**真正互不相交的 train/test 划分**。把全部 529 个 pair 用固定种子打乱，留出 30% 当 held-out 测试集，模型只在 train 上训练。这样测试准确率才是货真价实的泛化信号——模型不可能背下它从没见过的 pair。

```ts
function makeModAddSplit(task: Task, p: number, testFraction: number, rng: Rng): Split {
  const all: Example[] = [];
  for (let a = 0; a < p; a++)
    for (let b = 0; b < p; b++)
      all.push({ input: [a, b, p], answer: task.oracle([a, b, p])[2] });
  shuffle(all, rng);
  const nTest = Math.round(all.length * testFraction);
  return { test: all.slice(0, nTest), train: all.slice(nTest), p };  // 不变量：disjoint
}
```

这里有个很容易踩、stage01 注释专门解释了为什么不用 `core.trainToyModel` 的坑：那个通用训练器每步从全部 p² 个 pair 里均匀采样，它的"评测样本"和训练样本会重叠——那测的是**记忆**，不是泛化。grokking 只有在 train 和 test 严格不相交时才能诚实演示。所以 stage01 自己写了 split-aware 的训练循环，但其余原语（TinyTransformer / AdamW / crossEntropy / clipGradNorm / cosineWarmup）全部复用 core，唯一的差别就是数据采样。

模型是个几千参数的 `TinyTransformer`：`dModel=32, nHeads=4, nLayers=1, dHidden=64`。判定泛化的门槛设在 test acc ≥ 0.9，远高于 chance 的 0.043，几次蒙对不会误触发。

唯一的变量是 weight decay（权重衰减，每步把权重往 0 拉一点的正则化）。`core/optim.ts` 的 AdamW 用的是**解耦**衰减（decoupled，直接作用在参数上而非折进梯度，这是 AdamW 相对 Adam 对 transformer 真正要紧的改动）：

```ts
let update = (this.lr * mhat) / (Math.sqrt(vhat) + this.eps);
if (this.wd !== 0) update += this.lr * this.wd * p.data[j]; // decoupled
p.data[j] -= update;
```

**有正则（weightDecay=1.0）的实测结果**——这就是 grok：

```
[2] train=370 pairs   test=159 pairs (held-out, 训练永不可见)
[3] 有正则 (weightDecay=1.0):
    train@1.0 ~step 300   test@≥0.9 (相变) ~step 400
    => 记忆先于泛化: 训练集背完后又过了 ~100 步, 测试集才学会 (实测)
    最终: train acc=1.000  test acc=1.000  (实测 wall-clock 19718 ms)
```

读懂这条曲线：train 准确率约第 300 步就满了（背完了 370 个训练 pair），但 test 准确率直到约第 400 步才跨过 0.9 门槛。这中间约 100 步，模型 train 已经 100% 却还没泛化——它先记忆，后泛化。这正是 grokking 的形状。CPU 上约 19.7 秒跑完，几千参数，桌面就能复现。

## 失败模式：关掉正则，它永远停在查找表

把 weight decay 设成 0，架构、划分、步数（3000 步）全部不变，只动这一个旋钮：

```
[4] 失败模式 — 无正则 (weightDecay=0), 其余完全相同:
    最终: train acc=1.000  test acc=0.748  (相变@≥0.9: 从未发生)
    与有正则的 test 差距 = 0.252 (相同 train acc 下, 纯粹由 weight decay 造成)

[5] 对照 (train acc 都=1.0, test acc 分道扬镳):
    有正则 test | ############################## 1.000
    无正则 test | ###################### 0.748
    chance(1/23) | # 0.043
```

两个模型 train acc 都是 1.000，背训练集背得一样好。但 test acc 一个 1.000、一个卡在 0.748，再没跨过门槛。**0.252 的差距纯粹由 weight decay 造成**——同样的算力、同样的 3000 步，区别只在那一个正则项。

为什么 weight decay 是那个开关？记忆是把每个 pair 的答案塞进一组高范数权重里的"特解"，泛化要求模型找到 `(a+b) mod p` 那个低范数、平滑的"通解"。weight decay 持续给权重施加往 0 的压力，让背答案这条路越走代价越高，逼着优化器去找那个范数更小的代数解。没有这个压力，优化器在 train 满分后就没有任何动机继续走向泛化——它已经"做完作业"了。这也解释了 grok 的延迟：泛化解需要时间在正则压力下慢慢浮现。

**这对可解释性意味着什么**，才是本章把失败模式摆出来的真正目的。stage01 的结论说得很直白：

```
教训: train acc=1.0 不等于"学会"。无正则模型背下了全部 370 个训练 pair, 却没跨过泛化门槛。
对解释性的含义: 这种模型内部更像查找表, 拆出来的"电路"混着大量记忆碎片, 难分离出可迁移机制。
```

如果你拿无正则那个 0.748 的模型去做第 03-07 章的分析，探针会读到一堆记忆特解的碎片，activation patching 会定位到一些跟"模加法算法"无关的查找路径。你会很认真地解释一个根本不存在干净机制的模型。**所以这本书后面所有章节，统一在有正则、test acc=1.000 的那个 checkpoint 上做。** 全书的定位结果必须能在同一个模型上互相印证：第 03 章探针验出的傅里叶结构，要和第 05 章 patching 定位的因果路径对得上，和第 06 章的电路兜得拢。研究对象必须只有一个，且确证学会了。

## 诚实边界：什么能迁移，什么不能

stage01 自己打印了诚实边界，照抄不改：

```
诚实边界: 相变步号随种子抖动、且因 modAdd(23) 是 toy 而偏乐观, 绝对值不可迁移。
无正则模型的 test acc 并非恰好=随机(任务有代数结构, 记忆也会泄漏部分信号), 关键是它始终
跨不过泛化门槛、且明显低于有正则。可迁移的是形状 — 有正则时 train/test 先分离后重合
(记忆先于泛化), 无正则时持续分离(只记忆不泛化)。
```

拆开说三点。第一，"约第 300 步记忆、约第 400 步泛化"这两个**绝对步号不可迁移**——换种子会抖，modAdd(23) 太小本身就偏乐观，到 modAdd(97) 或真模型上数字完全不同。第二，无正则模型的 test acc 是 0.748 而不是恰好等于 chance 的 0.043——别误读成"完全没学到"。任务有代数结构，记忆也会顺带泄漏一部分泛化信号，所以它高于随机；关键不是它有多高，而是它**始终跨不过 0.9 门槛**且明显低于有正则的 1.000。第三，真正能迁移到任意规模的是**曲线的形状**：有正则是 train/test 先分离再重合（记忆先于泛化），无正则是持续分离（只记忆不泛化）。形状是结论，数字是 toy。

⚡ 这里还埋着一个**目前无通用解、正在研究**的开放问题：grokking 到底为什么发生、相变在哪一步触发，至今没有可预测的通用理论。我们能复现这条曲线、能用 weight decay 把开关拨到泛化那侧，但**无法事先算出**相变会在第几步发生，也无法保证在更大模型、更复杂任务上一定能 grok。"先记忆后泛化"的相变是 2022 年后机制可解释性的核心实验台之一，正是因为它把"泛化是怎么从记忆里长出来的"这个深问题压缩到了一个几千参数、桌面可跑的对象上——但把它讲透，仍是开放课题。本书的立场是务实的：我们不假装解决了 grokking，我们只是用它造出一个**确证学会了**的研究对象，然后从第 02 章开始，去读它的脑内激活。
