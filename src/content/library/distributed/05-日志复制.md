第 4 章我们选出了一个 leader。但选出 leader 本身什么都没解决——一个 leader 单独存在时，集群对任何事都还没达成一致。共识真正难的那一半是**复制**：把一条客户端命令放进多数派的日志里，放在**相同的 index、相同的 term**，使得即便 leader 随后崩溃，活下来的副本也永远不可能对"什么已经被提交"产生分歧。这个性质叫 State Machine Safety（状态机安全性，指所有副本在同一位置应用同一条命令），它是银行敢把账本跑在 Raft 上的全部理由。本章用 `examples/distributed-from-scratch/src/stage05-raft-log-replication.ts` 把它造出来，然后故意打破它，看清它防住的到底是什么。

为了让这一章聚焦在**日志**而不是**投票**上，我们做了一处简化：leader 不是选出来的，是**指派**的。我们钉死 n0 当 term 1 的 leader，跑到一半让它崩溃，再把预先指定的 n1 提升为 term 2 的 leader。复制机制和真实选举产生的 leader 完全一样——term 照样在换 leader 时递增，所以跨 term 的 log-matching 会被真实地触发——我们只是跳过了重跑 RequestVote。代码头部把这个 real vs simplified 的边界写得很清楚，别把指派当成 Raft 偷工减料。

## AppendEntries：复制不是"把日志发过去"

新手对复制最常见的误解是：leader 把自己的日志 push 给 follower，follower 存下来就行。如果真这么干，本章第二个实验里的灾难就会发生。

复制的载体是 AppendEntries RPC（远程过程调用）。它的 payload 长这样（`stage05-raft-log-replication.ts:62`）：

```typescript
interface AppendEntriesReq {
  term: number;
  leaderId: string;
  prevLogIndex: number; // index of entry immediately before `entries` (0 = none)
  prevLogTerm: number; // term of that entry (0 when prevLogIndex==0)
  entries: LogEntry[];
  leaderCommit: number; // leader's commitIndex, so followers can advance theirs
}
```

注意 `entries` 不是关键，`prevLogIndex / prevLogTerm` 这对字段才是整个算法的承重墙。它让 follower 在不传输整条日志的前提下，验证"我的日志和 leader 的日志，在新条目要挂上去的那个点之前，是逐字一致的"。leader 发"这批新条目接在 index=7 后面，而 index=7 那条是 term 1"，follower 检查自己 index=7 处是不是也是 term 1——不一致就拒绝。

为什么用 `(index, term)` 这一对就能判定"两条目是不是同一条"？看日志条目的定义（`stage05-raft-log-replication.ts:50`）：

```typescript
interface LogEntry {
  term: number;
  command: { client: string; seq: number };
}
```

关键在于：一个给定的 `(index, term)` 只可能由**单个 leader**产生，且永不复用。某 term 内只有一个合法 leader（这是第 4 章选举的产物），它在某个 index 写下的内容是唯一的。所以"index 相同且 term 相同"⟹"command 必然相同"。这条推理是后面 Log Matching 性质的地基。

leader 给每个 peer 维护一个 `nextIndex`（我下一条该发给它的 index），发送时按它框出 prevLog（`stage05-raft-log-replication.ts:228`）：

```typescript
private sendAppendEntriesTo(p: string): void {
  const ni = this.nextIndex.get(p) ?? this.lastLogIndex() + 1;
  const prevLogIndex = ni - 1;
  const entries = this.log.slice(prevLogIndex); // entries at index ni..end
  const req: AppendEntriesReq = {
    term: this.currentTerm,
    leaderId: this.id,
    prevLogIndex,
    prevLogTerm: this.termAt(prevLogIndex),
    entries,
    leaderCommit: this.commitIndex,
  };
  this.sendTo(p, "AppendEntries", req);
}
```

## Log Matching Property：用归纳法维持的强不变量

✦ 这是本章的专家钩子，也是 Raft 设计里最优雅的一处。Log Matching 性质有两条：

1. 若两份日志在某个 index 上的条目有相同的 term，则该条目存的 command 相同。
2. 若两份日志在某个 index 上的条目有相同的 term，则该 index **之前的所有条目**全部相同。

第 1 条上面已经论证（同 `(index, term)` ⟹ 同 leader ⟹ 同 command）。第 2 条才是真正强的——它说的是"一个点匹配，整段前缀都匹配"。这不是某次操作的结果，而是一个被**归纳法持续维持**的全局不变量。

归纳的 base case：日志初始为空，平凡成立。归纳步：每次 follower 接受 AppendEntries 前，那个 prevLog 一致性检查保证了"新条目挂上去的位置之前完全匹配"；而新条目本身由当前 leader 产生（同 term 同 command）。于是匹配的前缀只增不减，第 2 条永远成立。

**这就是为什么删掉一致性检查复制会静默分叉**：检查不是优化，不是为了"早点发现错误"，它是维持归纳不变量的那一步归纳。少了它，归纳链断裂，"一个点匹配 ⟹ 前缀全匹配"的保证当场失效，两份日志可以在某 index 看起来"对上了"，前面却早已不同。

follower 端这段就是整个协议正确性的所在（`stage05-raft-log-replication.ts:261`）：

```typescript
private onAppendEntries(req: AppendEntriesReq): void {
  if (req.term < this.currentTerm) {
    return this.replyAppend(req.leaderId, false);  // 旧 leader，拒绝让它下台
  }
  if (req.term > this.currentTerm) this.currentTerm = req.term;
  this.role = "follower";

  // THE CHECK：correct Raft 里 follower 拒绝 append，除非自己在 prevLogIndex
  // 处 term 与 leader 一致。这是阻止落后 follower 把新条目嫁接到陈旧前缀的关卡。
  if (!this.cfg.unsafeAppend) {
    const consistent =
      req.prevLogIndex === 0 || this.termAt(req.prevLogIndex) === req.prevLogTerm;
    if (!consistent) {
      return this.replyAppend(req.leaderId, false);  // 不匹配，告诉 leader 退回去重发
    }
  }
  // ... 通过检查后才 splice 进新条目，冲突处截断
}
```

`unsafeAppend` 这个 toggle（开关）就是本章的教学脊柱：关着是真 Raft，开着就跳过检查，让我们亲眼看见它防的灾难。

## 冲突修复：nextIndex 回退如何找到分歧点

一致性检查会拒绝不匹配的 append，但拒绝之后呢？落后的 follower 怎么追上来、分叉的 follower 怎么被纠正？答案在 leader 收到失败响应时的回退逻辑（`stage05-raft-log-replication.ts:323`）：

```typescript
private onAppendEntriesResp(resp: AppendEntriesResp): void {
  if (this.role !== "leader") return;
  if (resp.term > this.currentTerm) {
    this.currentTerm = resp.term;
    this.role = "follower";  // 有更高 term，说明被取代了，下台
    return;
  }
  if (resp.success) {
    this.matchIndex.set(resp.from, resp.lastLogIndex);
    this.nextIndex.set(resp.from, resp.lastLogIndex + 1);
    this.advanceCommitIndex();
  } else {
    // 日志不匹配：把 nextIndex 朝 peer 给的 hint 退一步重试。这个循环
    // 最终会找到分歧点，并用 leader 的条目覆盖 follower 的陈旧后缀。
    const hint = resp.lastLogIndex + 1;
    const cur = this.nextIndex.get(resp.from) ?? this.lastLogIndex() + 1;
    this.nextIndex.set(resp.from, Math.max(1, Math.min(cur - 1, hint)));
    this.sendAppendEntriesTo(resp.from);
  }
}
```

机制是：leader 对每个 peer 乐观地把 `nextIndex` 初始化为自己 `lastLogIndex + 1`（假设 follower 和我一样长），AppendEntries 失败时就退一格重发，逐步往回探，直到撞上两份日志真正匹配的那个 prevLog。一旦匹配，follower 在 `onAppendEntries` 里把冲突后缀截断（`this.log.length = idx - 1`），接受 leader 的条目——**leader 的日志永远是赢家**。响应里带的 `lastLogIndex` 是个 hint，让回退一次跳多格而不是一格一格挪，避免长日志退到天荒地老。

这里有个容易被忽略的设计：follower 之所以敢"截断自己已有的条目"，前提是这些被截断的条目**从未被提交**。Raft 保证已提交的条目一定在多数派上，不可能被某个新 leader 的回退覆盖掉——这由 leader 选举的限制（只有日志足够新的候选人能当选，第 4 章）和提交规则共同保证。

## 提交规则：多数派 + 当前 term 的两道闸

条目进了多数派日志还不算提交，提交是 leader 的单独决定（`stage05-raft-log-replication.ts:353`）：

```typescript
private advanceCommitIndex(): void {
  const majority = Math.floor(this.cfg.peers.length / 2) + 1;
  for (let n = this.lastLogIndex(); n > this.commitIndex; n--) {
    if (this.termAt(n) !== this.currentTerm) continue; // term safety rule
    let replicas = 1;  // 算自己
    for (const p of this.cfg.peers) {
      if (p === this.id) continue;
      if ((this.matchIndex.get(p) ?? 0) >= n) replicas++;
    }
    if (replicas >= majority) {
      this.commitIndex = n;
      this.applyCommitted();
      break;
    }
  }
}
```

两道闸：一条 index N 可提交，当且仅当 (1) 它存在于多数派（含 leader）上，**且** (2) `log[N].term == currentTerm`。

第二道闸是反直觉的微妙处。一条**上一个 term** 留下的条目，即使现在已经躺在多数派上，leader 也**不能**直接把它标记为已提交。这就是 Raft 论文里著名的 Figure 8 危险：如果允许新 leader 凭"这条旧条目现在在多数派上了"就提交它，可能出现一条已提交条目随后又被另一个 leader 覆盖的灾难。Raft 的处方是：旧 term 的条目只能被**间接**提交——leader 提交一条自己当前 term 的条目，那条之下的所有旧条目随之一并提交。代码里 `if (this.termAt(n) !== this.currentTerm) continue` 这一行就是这道闸，五个字符的判断，挡住的是整类安全性事故。

follower 这边只能跟着 leader 的 `leaderCommit` 推进 commitIndex，且不能超过自己实际持有的最后一条（`Math.min(req.leaderCommit, this.lastLogIndex())`）——你不能提交一条你手上还没有的条目。

## 持久 vs 易失：崩溃恢复的契约

复制状态机要扛崩溃，靠的是状态的持久/易失切分（`stage05-raft-log-replication.ts:98` 起）。`currentTerm` 和 `log` 进持久存储（这里用 Map 模拟磁盘），崩溃后还在；`commitIndex`、`lastApplied`、应用后的计数器状态、leader 的 `nextIndex/matchIndex` 都是易失字段，崩溃即丢，重启时重建（`stage05-raft-log-replication.ts:162`）：

```typescript
protected override onRestart(): void {
  this.role = "follower";
  this.commitIndex = 0;
  this.lastApplied = 0;
  this.state = new Map();
  // ... 其余易失字段清空
}
```

这个切分就是整个崩溃安全性的论证：一个节点可以忘记"我提交过什么"（易失），但绝不能忘记"我记录过什么"（持久）。所以重启后它能安全地从持久日志重新应用，重建出已提交状态。这与数据库的 WAL（Write-Ahead Log，预写日志）崩溃恢复是同一个思想——日志是真相之源，内存状态是可重建的派生物。

## 两个实验：同一种子，相反结局

代码用同一个故障脚本跑两遍，唯一变量是 `unsafeAppend`。故障时间线（虚拟毫秒，`stage05-raft-log-replication.ts:441` 注释）：

```
0     n0 当 leader (term 1)，开始复制客户端命令
300   分区：{n3,n4} 与 {n0,n1,n2} 切断，n3/n4 开始落后
600   leader n0 崩溃，多数派侧把 n1 提升为 leader (term 2)
900   愈合分区，n3/n4 带着陈旧日志回归，必须被纠正
1200  n0 重启为 follower，从日志重建易失状态
```

⚠ 这些绝对延迟数字是 toy-optimistic 的（种子化的内存仿真，链路延迟 8+抖动 6 ms，虚拟时间），代码头部和输出都明确标注了。能跨场景迁移的是**相对行为**：安全版收敛，不安全版分叉。

### 实验 A：开启检查（正确 Raft）

```
=== Scenario: raft-replication-safe (seed=5) ===
node               role      term  logLen  commitIdx  state(alice/bob/carol)
-----------------  --------  ----  ------  ---------  ----------------------
n0                 follower     2      49         49  15/17/17
n1                 leader       2      49         49  15/17/17
n2                 follower     2      49         49  15/17/17
n3                 follower     2      49         49  15/17/17
n4                 follower     2      49         49  15/17/17
ALL-CONVERGED      yes
commit-latency-ms  n=49                                p50=20 p99=303 max=303
[sim] finalTime=4009ms events=1051 safe=true
[msgs] sent=973 delivered=915 dropped=54 (loss=0 partition=54)
```

经历了分区、leader 崩溃、节点重启之后，五个副本全部收敛到 `logLen=49 commitIdx=49`，状态机 `15/17/17`（alice/bob/carol 三个客户端各自的计数）完全一致。注意 `dropped=54` 全部来自分区（`loss=0 partition=54`），说明一致性检查是在真实的丢包/落后条件下工作的，不是干净管道里的演示。两条安全不变量（state-machine-safety 和 log-matching）整个 run 全程成立。

### 实验 B：关闭检查（埋下的 bug）

同一种子，只把 `unsafeAppend` 打开，follower 不验证 prevLog，盲目 splice。结果：

```
[caught expected violation] invariant violated: "state-machine-safety"
  state: { "note": "committed entry mismatch between two live replicas",
    "logs": [
      { "id": "n1", "committed": [ index 1..13, 都是 term 1, index 8 = carol#8 ] },
      { "id": "n2", "committed": [ ...与 n1 完全相同, index 1..13... ] },
      { "id": "n4", "committed": [
          index 1..7 与多数派一致,
          index 8  term 2 cmd bob#16,   <-- 分叉：多数派此处是 carol#8 (term 1)
          index 9  term 2 cmd carol#17,
          ... index 10..13 同样是 term 2 的另一串命令 ] } ] }
```

承重的事实，逐字可复现：**安全版里五副本全收敛到 logLen=49 commitIdx=49 state 15/17/17；不安全版里 n4 在已提交 index 8 上记着 bob#16(term 2)，而 n1/n2 在 index 8 上记着 carol#8(term 1)——一个真实的分歧，仿真自己跑出来的，触发了 state-machine-safety**。

病理诊断很清楚：n4 在分区期间落后了，愈合后新 leader 给它发条目，没有 prevLog 检查，它就把新条目嫁接到一个**在 prevLogIndex 处根本不匹配**的陈旧前缀上。两个副本于是在同一个已提交 index 上应用了不同命令。这是分布式系统里最不可饶恕的 bug——已提交意味着"对客户端承诺过、不可撤回"，而两个副本对这条不可撤回的承诺给出了不同答案。

重新开启检查（实验 A）就让一切收敛、两条不变量全程成立。**同种子、相反结局——这个一致性检查是承重的，不是装饰。**

## minority 无法提交：给第 6 章 CAP 的引子

实验 A 里有个值得停下来看的细节：分区期间 `{n3, n4}` 这一侧（少数派）发生了什么？它们收不到多数派的提交，自己也凑不齐 3 个副本的多数派，所以它们的 commitIndex **卡住不前**——它们能继续接收客户端命令吗？能 append 到本地日志，但永远无法提交，因为提交需要多数派确认，而它们这一侧只有 2 个节点。

这不是 bug，是 Raft 的刻意设计：**少数派分区里没有进展，是用可用性换一致性**。一个落在少数派侧的客户端，宁可被拒绝/超时，也不会拿到一个可能与多数派冲突的"假提交"。这正是实验 B 反面教训的另一面——B 里我们看到放弃一致性的代价是分叉，A 里我们看到守住一致性的代价是少数派停摆。

这个"分区下 minority 无法提交"的实测结果，就是第 6 章对 CAP（Consistency 一致性 / Availability 可用性 / Partition tolerance 分区容忍）形式化讨论的直接入口。Raft 在 CAP 的坐标系里是一个 CP 系统：分区发生时它选择放弃少数派的可用性来保住一致性。第 6 章会把这个工程观察拔高成形式化的 trade-off。

## ⚡ 前沿：日志无限增长，与不投票的 learner

本章的复制状态机有个被刻意回避的现实问题：**日志只增不减**。49 条还好，跑一年就是几十亿条。每次重启都要从头重放整条日志，每个新副本加入都要从 index 1 开始追，这显然不可持续。

标准答案是**快照（snapshot）+ 日志压缩（log compaction）**：把状态机当前状态序列化成一个快照，丢掉快照点之前的所有日志条目，新副本直接装载快照再追后面的增量。Raft 论文给了基础方案，但它有个粗暴之处——快照是整个状态机的全量转储，状态大时一次快照就是一次巨大的 IO 和网络传输尖峰。

前沿系统在两个方向上还没有公认的通用解：

**Incremental snapshot（增量快照）**：理想情况下只快照自上次以来变化的部分，而不是每次全量。这在状态机内部结构已经天然分块时（例如底层是 LSM 树或 B+树）相对好做——可以复用存储层已有的 SST 文件/页作为快照单元。但对一个任意的内存状态机做通用的、对应用透明的增量快照，怎么界定"变化的部分"、怎么处理快照与持续写入的并发，目前没有一个能塞进 Raft 库就通用工作的方案，各家是按自己的状态机结构定制的。

**Learner 节点（也叫 non-voting member）**：给集群加新副本时有个尴尬——新节点日志是空的，如果立刻把它算进多数派计票，会拉低集群的提交门槛（本来 3/5，现在可能要等这个空节点），甚至在它追日志期间让集群短暂失去可用性。解法是先把新节点作为 **learner** 加入：它只追日志、**不参与投票、不计入多数派**，等它追上了再正式升格为 voter。这个机制在 etcd/Raft、TiKV 等生产系统里已经落地，但"何时判定 learner 追上了可以安全升格""升格瞬间的成员变更如何与正在进行的复制原子地协调"仍是工程上反复出 bug 的地方——成员变更（membership change）与日志复制的交互，是 Raft 实现里公认最难写对的部分，至今仍是活跃的研究与工程话题。

这两个问题都不在本章配套代码的范围内（我们的日志最多 49 条，五个副本固定不变），但它们是任何想从这台 toy 复制状态机走向生产的人，下一步必须面对的两堵墙。
