把一个 `counter++` 拆到两台机器上，它就不再是 `counter++` 了。单机上你写 `if (balance >= 100) balance -= 100`，CPU 保证这两步之间没人插队；换成「节点 A 读余额、节点 B 也读余额、两个都减」，钱就凭空多花了一份。这本书要讲的所有东西——逻辑时钟、故障检测、Raft、CRDT、CAP——本质上都是在回答同一个问题：**当你不能再假设「读了就是最新的」「发了就一定到」「它没回我就是它死了」，正确性要靠什么重新建立起来？**

这是一本反「AI 批量生成的 survey 稿」的书。Survey 稿会告诉你「分布式系统需要在一致性、可用性、分区容错之间权衡，这就是 CAP 定理」——正确、完整、且毫无用处。它不会告诉你：**为什么「分区容错」根本不是一个可选项**，为什么 CAP 真正逼你做的选择是「分区发生时，拒绝写还是接受分叉」，以及为什么一个号称 AP（可用性优先）的系统在你没注入网络分区时看起来和 CP（一致性优先）的一模一样——直到那一天。本书的每一章都配一段**真能跑、真能崩、真能重放**的 TypeScript 代码（在 `examples/distributed-from-scratch/`），结尾打印代码真算出来的数字：消息条数、选举轮数、收敛步数、不变量违反次数。

本章不配 stage，它是全书的地图。读完你应该能在脑子里装下两样东西：一是**八条反直觉断言**——单机程序员每一条都会本能地违反；二是**这些断言到具体章节的映射**——每条断言后面都有一章用代码把它演示成你能复现的失败。

## 分布式系统难在哪：三个公理，不是「机器多」

新手以为分布式难在「要协调很多台机器」。不是。难的根源是三件单机上不存在、分布式上无法回避的物理事实——本书把它们当公理：

1. **没有全局时钟。** 两台机器的物理时钟一定有偏差（drift，时钟漂移），而且谁也读不到「此刻的真实时间」。所以你无法可靠地说「事件 X 发生在事件 Y 之前」——除非 X 和 Y 之间有因果联系。
2. **消息会延迟、丢失、乱序、重复。** 网络不是函数调用。一条消息可能晚到 2 秒、可能永远不到、可能比后发的先到、可能到两次。
3. **节点会独立失败。** 单机程序要么整个跑、要么整个崩，状态一致。分布式里一个节点崩了，其他节点还在跑，而且**它们一开始并不知道那个节点崩了**——它们只看到「它不回我消息」。

这三条里，最毒的是它们的**交叉项**。第 1 条和第 2 条交叉，你就分不清「对方处理慢」和「消息还在路上」；第 2 条和第 3 条交叉，你就分不清「对方崩了」和「网络断了」。本书配套仓的网络层把这个交叉项做成了代码里一行注释（`examples/distributed-from-scratch/src/core/network.ts`）：

```ts
/** Split the cluster into two groups that cannot exchange messages. This is THE
 *  CAP fault. Messages across the cut are counted as dropped (the sender can't
 *  tell drop from partition — that indistinguishability is the heart of FLP).
 *  Nodes not listed stay in their previous group (or the default group 0). */
partition(groupA: string[], groupB: string[]): void { ... }
```

「发送方无法区分丢包和分区」——这句**indistinguishability（不可区分性）**是后面 FLP 不可能性定理的种子。先记住它的形状：在一个节点看来，「对方崩了」「网络断了」「对方只是慢」这三种情况发出的信号**完全一样**——都是「我没收到回复」。它没有任何办法在有限时间内分辨。这不是工程没做好，是信息论上做不到。

## 八条反直觉断言：单机直觉的尸检报告

下面八条，每一条都是单机程序员的本能假设，每一条在分布式里都是错的，而且每一条都对应本书后面用代码演示的一个失败。我把它们当「尸检报告」列出来——先看死因，后面章节再看现场。

### ① 你不能区分「慢」和「死」

单机里，函数没返回就是还在算，你可以一直等。分布式里，节点 B 没回复你，可能是它崩了（等到天荒地老也不会回），也可能是它在 GC 卡了 800ms 或者消息堵在网络里。**你必须在「再等等」和「当它死了」之间赌一把**，而这一赌本身就会出错：

- 等太短 → **误判（false positive）**：把活节点当死的，触发不必要的故障转移，可能选出第二个 leader（脑裂）。
- 等太长 → **漏判（slow detection）**：节点真死了你还在傻等，服务卡住。

这就是为什么所有故障检测都绕不开「超时」这个又脏又必要的旋钮，以及为什么 phi-accrual（一种把「死了多确定」从布尔值变成连续概率的检测器）会被发明出来。
→ **第 3 章 RPC 与故障检测**（`stage03-rpc-failure-detector.ts`）：用代码注入非对称延迟，实测同一套参数下误判率和漏判率怎么此消彼长。

### ② 你不能假设时钟同步

「给每个事件打个时间戳，按时间戳排序」——这个单机里天经地义的操作，分布式里会排出**因果倒错**的结果：B 收到 A 的消息后产生的事件，时间戳可能比 A 发消息还早，只因为 B 的时钟慢了 50ms。物理时间戳决定不了「谁先发生」。

能决定「谁先发生」的，只有**因果关系**（happens-before，A 的结果影响了 B 才叫 A 在 B 之前）。Lamport 把这件事形式化成逻辑时钟：不看墙上的钟，看消息的传递。
→ **第 1 章 Lamport 逻辑时钟**（`stage01-lamport-clock.ts`）：演示逻辑时钟能给出一个不违反因果的全序，但它有个致命短板——**它分不清「真的有因果」和「只是恰好编号有先后」**，也就是说它无法检测「并发」。
→ **第 2 章 向量时钟**（`stage02-vector-clock.ts`）：补上这个洞，能精确判断两个事件是因果还是并发，代价是每个节点要维护一个长度等于节点数的向量——元数据随节点数线性膨胀。这个 O(N) 代价不是实现笨，是信息量的下限。

### ③ 你不能假设消息只到一次

单机调函数，调一次就执行一次。网络消息不是：它可能丢（你以为发了其实没到），所以你要重试；而重试又意味着对方**可能收到两次**。于是「扣款」这种操作必须做成**幂等（idempotent，重复执行结果不变）**——否则一次网络抖动就让用户被扣两次钱。本书的网络层故意把这件事做成可注入的故障开关：

```ts
/** When on, each delivered message MAY be delivered a second time (coin per
 *  message). Forces protocols to be idempotent — non-idempotent handlers break
 *  visibly here, which is the teaching point. */
duplicate(on: boolean): void { this.duplicateOn = on; }
```

注意注释里的设计意图：打开重复投递，**非幂等的处理器会在这里肉眼可见地崩坏**——这正是教学点。把 bug 逼到台面上，比嘴上说「记得做幂等」有用一万倍。
→ **第 3 章**讲重试 + 幂等的配合，**第 7 章 CRDT**（`stage07-crdt-convergence.ts`）更进一步：设计一种数据结构，让消息乱序 + 重复**都不影响最终结果**，从根上免疫这个问题。

### ④ 「我没收到」不等于「它没发生」

你给协调者发了「我同意提交」，然后网络断了。提交了没有？你不知道。对方可能收到了并提交了，也可能没收到。**没有回执 ≠ 操作没生效**——这是两阶段提交（2PC）会卡死、以及为什么需要共识协议把决定写进多数派日志的根本原因。决定一旦被多数派记下，单个节点失联就不再能让整个系统忘记这个决定。
→ **第 5 章 Raft 日志复制**（`stage05-raft-log-replication.ts`）：用 `matchIndex` / `commitIndex` 演示「一条日志要复制到多数派才算 committed」，以及一个落后的 follower 重连后怎么被领导者修复到一致。

### ⑤ 多数派同意 ≠ 全体同意，但够了

直觉觉得「要么所有人同意，要么不算数」。分布式里**等所有人同意会被最慢/已死的那个拖死**。真正的设计是**多数派（quorum，超过半数）**：5 个节点里 3 个同意就生效。它的妙处是任意两个多数派必然有交集——所以新旧两次决定不可能各自凑齐多数派还互相矛盾。这是 Raft 选举安全和日志安全的数学地基。
→ **第 4 章 Raft 领导选举**（`stage04-raft-election.ts`）：演示 term（任期）+ 投票 + 「每任期至多一票」如何保证脑裂时不会选出两个 leader，以及网络分区下少数派那边为什么选不出 leader（凑不齐多数派）。

### ⑥ 「读到的」不一定是「最新写的」

单机里写完立刻读，读到的就是刚写的。多副本系统里，你读的副本可能还没收到最新的写。于是「一致性」从一个布尔值裂成一个谱系：线性一致（linearizable，表现得像只有一份数据）、最终一致（eventually consistent，停止写入后副本终将相同）、因果一致……强一致要花钱（每次读写都要协调），弱一致省钱但会读到旧值。选哪档不是技术问题，是**业务能容忍多旧**的问题。
→ **第 6 章 CAP 取舍**贯穿这条；**第 7 章 CRDT** 是「最终一致」这一档的极致：完全不协调，靠数据结构本身的数学性质（交换律 + 幂等 + 结合律）保证所有副本最终收敛到同一状态。

### ⑦ 分区不是「可不可能」，是「什么时候」

CAP 里的 P（分区容错）常被误读成「可选的第三个属性」。错。**网络一定会分区**——交换机重启、网线被挖断、跨机房专线抖动，这些不是 if 是 when。所以 P 不是你选不选的，是物理强加给你的。CAP 真正逼你选的是：**分区发生的那一刻**，少数派那边收到写请求，你是**拒绝它**（保一致性 CP，但这一刻不可用）还是**接受它**（保可用性 AP，但两边数据会分叉，愈合后要 merge）。
→ **第 6 章 CAP 取舍**（`stage06-cap-tradeoff.ts`）：同一份代码、同一个分区故障，跑 CP 模式（分区时少数派拒写）和 AP 模式（两边都收，愈合后分叉需要 reconcile），让你看到两种选择各自的代价——可用性 vs 一致性，没有免费的。

### ⑧ 「它一直好好的」证明不了「它是对的」

这条是给写代码的人的。一个分布式 bug 跑一万次可能都不出现，第一万零一次因为某个特定的消息交错（interleaving，事件在时间上的穿插顺序）就脑裂了。**「我本地跑过没问题」在分布式里几乎不构成证据**——因为真实网络的交错不可复现，你抓不住那个出问题的瞬间。这正是本书工程契约要解决的，下一节专讲。

## 全书工程契约：确定性仿真，让脑裂可以被重放

真实分布式系统最缺的能力，是**重放一次故障**。生产上「leader 昨晚抖了一下」基本无法复现——因为真实网络的延迟、丢包、交错每次都不同。survey 稿对此的回答是挥挥手「这种问题很难调试」。本书的回答是：**把不确定性全部收口到一个种子里**。

整本书跑在 `examples/distributed-from-scratch/src/core/` 的确定性内存仿真底座上，它做了三件事，每一件都为「可重放」服务：

**1. 时间是虚拟的离散事件时钟，不碰 `Date.now()`。** `core/clock.ts` 里 `SimClock` 把时间做成一个由事件队列驱动的整数，「等 150ms」只是把计数器往前拨。它的头注释把动机说得很直白：

> 如果时间走真实时钟，两个问题随之而来：(1) demo 要花真实的秒数；(2) 交错每次运行都不同，所以一次由特定超时竞争引发的脑裂永远无法复现。

更关键的是同一时刻的多个事件靠一个单调递增的 `seq` 打破平局——没有这个，两个排在同一毫秒的事件会以堆内部（实质随机）的顺序触发，**悄悄破坏确定性**。注释称这是「最恶心的失败模式，因为它只表现为『测试有时会失败』」。

**2. 随机性只有一个来源：种子 PRNG。** `core/prng.ts` 手写了一个 mulberry32，`Math.random()` 在全书被禁用：

```ts
// Why a hand-rolled, seeded PRNG instead of Math.random: every number this book
// prints (message counts, election rounds, convergence steps, partition
// outcomes) must be byte-identical across runs and machines. A distributed bug
// is only useful to a reader if it REPLAYS — "the leader flapped once on my
// machine" is the survey-grade hand-waving this book rejects.
```

所有的不确定性——延迟抖动、丢包硬币、选举超时、消息乱序——都从这一个种子派生。**同种子 ⇒ 每条消息的命运字节级一致。**「leader 在我机器上抖了一次」正是本书拒绝的 survey 级挥手。

**3. 网络是内存消息总线，故障是可注入、可脚本化的。** 上面已经看过 `partition` / `duplicate`，还有 `dropRate`（丢包率）、`reorder`（乱序）、`setLatency`（注入非对称延迟）。三者合起来，一个实验就被打包成 `core/scenario.ts` 里的 `Scenario` = **种子 + 故障时间线 + 不变量**：

> 每一章最终都在主张这样一件事：「在这些故障下，这个协议保持这个不变量，并付出这个代价」。Scenario 恰好打包了这件事……「它脑裂过一次」变成「种子 7 在第 142 个事件脑裂，这是当时的快照」。这就是本书和 survey 的区别。

而且不变量是**每个事件后都检查一次**，不是只在最后查——因为「至多一个 leader」这种安全性质可能瞬时违反又自愈，只在结尾查会**恰好错过这一章要讲的那个 bug**（`core/assert.ts` 的设计哲学）。

一句话总结契约：**本书里每一个数字都是代码真算出来的、可被同种子复现的；每一次脑裂都有案发现场快照。** 凡是折算成真实网络耗时的估算值，会显式标 `(est.)`；凡是用 `bench` 真测的 wall-clock 吞吐，会标注「机器相关」。仿真用 3~5 节点的 toy 拓扑，**绝对值偏乐观**，可迁移的是相对趋势（如向量时钟元数据随节点数线性增长、CP 与 AP 在分区下的可用性差异）。

## ✦ FLP 不可能性：为什么所有共识协议都在「绕过」而非「推翻」它

讲到这里必须把那块挡在所有共识协议前面的石头搬出来：**FLP 不可能性定理**（Fischer-Lynch-Paterson, 1985）。它说的是：

> 在一个**异步**网络模型里（消息延迟无上界，但终会送达），只要**允许哪怕一个**节点崩溃，就**不存在**一个确定性算法能保证在有限时间内达成共识。

注意它的精确边界，魔鬼全在定语里。它**不是**说「共识不可能」——现实里 Raft、Paxos 天天在达成共识。它说的是「**确定性 + 异步 + 容许一个崩溃**」这三个条件凑齐时，你无法同时保证 **safety（永不出错）** 和 **liveness（终会出结果）**。直觉上的原因正是断言①：在纯异步模型里，你永远分不清「一个节点崩了」和「它的消息只是还在路上」。总存在一个最坏的消息交错，让算法卡在「再等一下还是现在决定」的岔路口反复横跳，永远定不下来。

这块石头搬不走。所以后面每一个共识协议，做的都不是「推翻 FLP」，而是**精确地破坏它三个前提里的一个**，用现实里成立的假设把不可能性绕过去：

- **Raft / Paxos：放弃纯异步，引入超时。** 它们假设「部分同步」（partial synchrony）——网络大部分时候是好的，超时是个有效的故障信号。超时本质上是**往确定性算法里注入了一点随机性/时间假设**，让系统能从「反复横跳」里跳出来做决定。代价：超时设短了会误判（断言①），所以 Raft 的选举超时要加随机抖动，避免所有节点同时超时同时竞选——这正是 `stage04` 里能看到的设计。
- **理论上的另一条路：放弃确定性，引入真随机。** Ben-Or 等随机化共识算法靠抛硬币打破对称，以概率 1 终止——但实践里大家更接受超时这条路。

记住这个心智锚点：**当你在第 4-6 章看到「选举超时」「心跳」「随机退避」，你看到的就是工程师在 FLP 这块石头上凿出的绕行道。** 它们不优雅，但它们诚实——因为它们没假装那块石头不存在。

## ⚡ 前沿：能不能把「全局时钟」用钱买回来？

公理 1 说「没有全局时钟」。但 Google Spanner 做了一件近乎作弊的事：**它花钱把全局时钟买回来了一部分**。

Spanner 的 **TrueTime** 在每个数据中心部署**原子钟 + GPS 接收器**，让时钟同步的误差被压到一个**有界**的小区间。关键不是「时钟变准了」——再准也有误差——而是 TrueTime 把误差**显式量化**成一个区间 `[earliest, latest]`，并提供一个 `now()` 返回这个区间而非一个点。于是 Spanner 玩了个聪明的把戏叫 **commit-wait**：提交一个事务后，**故意等待**那个误差区间的长度（通常几毫秒）再对外可见，从而保证「任何后开始的事务读到的时间戳一定更大」。这就把断言②（不能假设时钟同步）从「绝对做不到」松动成「在一个花钱买来的有界误差内，可以做到」——Spanner 因此能提供跨洲的外部一致性（external consistency），这在纯软件世界里被认为是做不到的。AWS Aurora 走的是另一条偏软件的路（基于日志的复制 + quorum 读），但同样在用「把某种全局序近似买回来」的思路。

**为什么大多数系统买不起？** 因为这需要在每个机房部署、维护原子钟和 GPS 天线这类专用硬件，还要把误差边界做到可信——这是 Google 级别的基础设施投入，绝大多数团队既没有这种硬件，也没有把时钟误差当一等公民工程化的能力。所以对 99% 的系统，公理 1 依然成立，你依然得靠逻辑时钟（第 1-2 章）这种**不依赖物理时钟**的办法来建立顺序。

这里也有真正的开放问题：**TrueTime 把误差压小了，但没压到零**——commit-wait 等的那几毫秒是直接加在写延迟上的真实代价，误差区间越大等得越久。**如何在不依赖专用硬件、纯软件的前提下，把分布式时钟误差边界做到既小又可信，目前没有通用解，仍在研究**（NTP/PTP 软件时钟同步的误差边界在普通数据中心难以可信地收紧，是这个方向的核心难点）。换句话说：**「全局时钟」这个公理，现在是「能花大价钱买回来一部分」，但「便宜地、可信地买回来」仍是开放的。**

## 怎么读这本书

章节顺序是被依赖关系逼出来的，对应着上面八条断言的展开：

| 章 | 主题 | stage | 攻克的断言 |
|---|---|---|---|

| 03 | RPC 与故障检测 | `stage03-rpc-failure-detector.ts` | ①③（慢 vs 死、重试 + 幂等） |
