先把这章要打碎的错觉摆出来：**你给一个远程节点发了请求，迟迟没收到回复——这一刻，「它崩了」和「它只是慢」在你这边长得一模一样。** 两种情况你观察到的现象完全相同：什么都没回来。第 0 章把这件事当哲学结论讲过；这一章把它变成两段会跑的代码，并且用真实运行的数字告诉你，硬要区分这两者要付出什么代价。

代码在 `examples/distributed-from-scratch/src/stage03-rpc-failure-detector.ts`，跑在前几章同一套 `core/` 之上：`core/network.ts`（可注入丢包、重复、延迟尾、分区的假网络）、`core/node.ts`（带 inbox 和定时器的节点基类）、`core/scenario.ts`（种子化、确定性回放）。下面所有数字都来自 `seed=7` 的一次真实运行，墙钟时间是模拟时钟，不是真实 NIC（网卡），我会在该标注的地方标。

这一章你只需要刻住两件事。第一：「RPC 像本地函数调用」是新人最大的漏抽象（leaky abstraction，把底层复杂度藏不住的抽象），它在分布式下必然漏，漏出来的就是「恰好一次」语义的不可能。第二：故障检测不是「检测到了/没检测到」的布尔题，而是一个可调的误报预算，φ-accrual 把这个旋钮直接交到你手里。

## ✦ 「RPC 像本地调用」是个会漏的谎

远程过程调用（RPC，Remote Procedure Call，像调本地函数一样调远程服务）这个抽象的卖点就是：你写 `server.charge(100)`，感觉跟调本地函数一样。问题是本地函数调用有三个隐含保证，网络一个都给不了：一定会执行、最多执行一次、要么返回结果要么抛异常。网络上，请求可能丢、回复可能丢、服务可能在执行到一半时崩。

具体到「恰好一次」（exactly-once，每个请求的副作用不多不少正好生效一次）：**在分布式下它不可能直接实现。** 你只能在两个都不完美的语义里选：

- **at-most-once（至多一次）**：发一次，不重试。请求或回复一丢，这次操作就没了。副作用绝不重复，但可能根本没发生。
- **at-least-once（至少一次）**：没收到回复就重试，直到收到。操作一定会发生，但可能发生多次——因为「没收到回复」是模糊的，你不知道是请求丢了（服务没执行）还是回复丢了（服务已执行）。

代码选的是 at-least-once，因为「最终一定要落地」通常是业务的硬要求。客户端的重试逻辑很朴素——只要还有没被 ack（确认）的请求，就按固定间隔全部重发：

```ts
// stage03-rpc-failure-detector.ts，RpcClient.retryLoop
private retryLoop(): void {
  if (this.pending.size === 0) return; // all acked: nothing left to retry
  for (const [requestId, amount] of this.pending) {
    this.requestSends++;
    this.sendTo(this.serverId, "RpcRequest", { requestId, amount });
  }
  this.setTimer(RPC_RETRY_INTERVAL_MS, () => this.retryLoop());
}
```

这就引出了 at-least-once 必须配的另一半：**幂等**（idempotent，同一操作做多次和做一次结果相同）。没有它，重试就是在重复扣款、重复下单。

### 幂等怎么做：去重表把「至少一次投递」变成「恰好一次副作用」

关键 trick 是把「投递」和「副作用」分开。投递可以发生很多次（at-least-once），但副作用必须只生效一次。做法是给每个逻辑请求一个稳定的 `requestId`（重试时 id 不变），服务端用一张去重表记下「这个 id 我处理过、结果是什么」，第二次见到同一个 id 就直接重放上次的结果，不再执行副作用：

```ts
// stage03-rpc-failure-detector.ts，RpcServer.onMessage
const cached = this.processed.get(req.requestId);
if (cached !== undefined) {
  // 已处理过：这是重试或网络重复。用相同结果重新 ack，不重新执行副作用。
  this.duplicateArrivals++;
  this.sendTo(from, "RpcReply", { requestId: req.requestId, balance: cached });
  return;
}
// 第一次见这个 id：执行副作用、记账、ack。
this.balance += req.amount;
this.processed.set(req.requestId, this.balance);
```

这正是生产系统里的 idempotency key（幂等键）：Stripe 的 `Idempotency-Key` header、数据库的 `INSERT ... ON CONFLICT DO NOTHING`、消息队列消费端的 `event_id` 去重，全是这个模式。

### 实测：去重省下的那一大截，就是模糊性的成本

跑场景 A（25% 丢包 + 网络重复开启 + 第 3000ms 注入真崩溃），数字把成本量化得很清楚：

| 指标 | 值 |
|---|---|
| 逻辑请求数 | 20 |
| 客户端发送次数（含重试） | 31 |
| 服务端到达次数（含重复） | 32 |
| 其中重复（去重救下的） | 12 |
| 实际副作用（服务端 balance） | 210 |
| 若恰好一次的期望 balance | 210 |
| 被 ack 的逻辑请求数 | 20 |

读法：20 个逻辑请求，因为丢包重试和网络重复，服务端实际收到 32 次——多出来的 12 次全是同一批 id 的重复到达。但 balance 停在 210（即 1+2+…+20），跟「恰好一次」的期望完全一致，一分钱没多扣。**那 12 次重复就是网络模糊性的标价**：你为了「最终一定落地」付出了重复投递，幂等把这笔账兜住了。如果服务端是个 naive 实现（每次到达都 `balance += amount`），balance 会是 210 加上那 12 次重复对应的金额——精确地过扣。

代码里还挂了一条断言守这个不变量，崩溃注入也好、重复也好，整个 run `safe=true`：

```ts
ctx.watch(invariant(
  "side-effect-applied-at-most-once-per-request",
  () => server.balance <= expectedBalance(LOGICAL_REQUEST_COUNT),
  () => ({ balance: server.balance, max: expectedBalance(LOGICAL_REQUEST_COUNT) }),
));
```

## 故障检测：把「死/活」从布尔变成连续可疑度

RPC 这半解决了「请求落地」，但没解决「对方到底死没死」。最朴素的检测器是定时心跳 + 固定超时：每 100ms 收一次心跳，超过 X ms 没收到就判死。问题就出在 X 怎么选——这就是开头那句话的工程版本。

φ-accrual 检测器（Hayashibara et al. 2004，Cassandra 和 Akka 在用）的思路是不直接给布尔答案，而是输出一个连续的可疑度 φ。它把心跳到达的间隔时间建模成一个正态分布，然后算：「按历史规律，已经这么久没来心跳，这有多反常？」

$$\varphi = -\log_{10}\big(P(\text{下一个间隔} > \text{距上次心跳已过的时间})\big)$$

φ 的物理意义很直接：φ=1 大约对应「我现在怀疑它死了，有 10% 概率冤枉它」；φ=2 是 1%；φ=8 是一亿分之一。所以**给 φ 设一个阈值，本质就是直接给「误报率」设一个数**——这是它比裸超时强的全部理由。心跳越久不来，φ 平滑地往上爬，而不是到点啪一下跳成「死」。

核心计算是纯函数，不读时钟不做 IO，调用方喂它到达时间戳、问它某个「现在」的 φ，所以能确定性回放：

```ts
// stage03-rpc-failure-detector.ts，PhiAccrualDetector.phi
phi(nowMs: number): number {
  if (this.intervals.length < 2 || this.lastHeartbeatAtMs < 0) return 0;
  const mean = this.computeMean();
  const stdev = Math.max(PHI_MIN_STDEV_MS, this.computeStdev(mean));
  const elapsed = nowMs - this.lastHeartbeatAtMs;
  // 用论文 Eq.5 的 logistic 近似算正态尾概率——闭式、无 erf、确定性。
  const y = (elapsed - mean) / stdev;
  const e = Math.exp(-y * (1.5976 + 0.070566 * y * y));
  const pLater = elapsed > mean ? e / (1 + e) : 1 - 1 / (1 + e);
  const pSafe = Math.min(Math.max(pLater, 1e-12), 1);
  return -Math.log10(pSafe);
}
```

这里有两个看着多余、实则保命的设计，都值得停一下：

**`PHI_MIN_STDEV_MS = 5`（标准差下限）。** 如果心跳间隔历史上几乎一模一样，算出来的 stdev 会接近 0。stdev 一进分母趋零，心跳哪怕只晚 1ms，φ 就爆成无穷大——一个见谁杀谁的退化检测器。给 stdev 一个下限，是这个检测器的「谦虚」：它承认自己对未来抖动的估计不可能无限精确。

**warmup 期返回 0。** 样本不足 2 个时直接返回「无怀疑」（`return 0`）。你没法用一个间隔估方差，所以冷启动阶段绝不怀疑任何节点。这是新人写检测器最容易漏的边界——刚上线、还没攒够样本就开始杀节点。

### 实测：φ 平时趴着，真崩了才尖刺

还是场景 A，阈值设 φ≥8（远在延迟尾之上），第 3000ms 真的把服务端 `crash()` 掉：

| 指标 | 值 |
|---|---|
| 崩溃注入时刻 | 3000 ms |
| 崩溃那一刻的 φ（健康时） | 0.39 |
| 崩溃后 φ 峰值 | 12 |
| 怀疑阈值 | 8 |
| 检测延迟 | 320 ms |
| 误杀健康节点次数 | 0 |

φ 轨迹画出来是这样（崩溃前一路趴在地板上，崩溃后窜过阈值）：

```
phi trajectory (max per column, @=high):
          .             := .-%@@@@@@@@@@@@@@@@@@@@@@
                          ^   (^ = crash @ 3000ms, threshold φ≥8)
```

读法：健康时 φ=0.39，意思是「现在怀疑它就是在瞎怀疑」，检测器闭嘴。3000ms 崩溃后心跳断流，φ 一路爬到 12，越过阈值 8。**检测延迟 320ms**——大约 3 个多心跳间隔（心跳 100ms 一次）。这 320ms 不是 bug，是区分「死」和「慢」的不可避免的税：你必须等到「它只是慢」在统计上变得足够离谱，才敢宣判死亡。整个 run 0 次误杀，因为阈值压在延迟尾之上。

注意一个 φ-accrual 独有的好处：如果一个迟到的心跳最终还是来了，φ 会自己掉回阈值以下，检测器会**收回**之前的怀疑。裸超时做不到这个——它一旦判死就是死。代码里这段自我纠正：

```ts
} else if (!overThreshold && this.suspected) {
  // φ 掉回阈值以下（迟到的心跳终于到了）：收回怀疑。
  // accrual 能自我纠正一阵短暂的卡顿，裸超时不能。
  this.suspected = false;
}
```

## ⚠ 失败模式：阈值压不住延迟尾，就会谋杀健康节点

前面都是顺风局。现在看它怎么坏。场景 B 把服务端设成**全程健康、根本不崩**，但给心跳链路加一条肥尾延迟（jitter 窗口 300ms，叠在 100ms 心跳上，意味着一个活着的节点的心跳偶尔会晚到近 3 个间隔）。然后只动一个旋钮——阈值——其他全锁死（同 seed、同故障）。

| 指标 | 阈值 φ≥1 | 阈值 φ≥8 |
|---|---|---|
| 服务端崩了吗？ | 否（全程健康） | 否（全程健康） |
| 心跳到达 p50 | 97 ms | 97 ms |
| 心跳到达 p99 | 240 ms | 240 ms |
| 心跳到达 max | 240 ms | 240 ms |
| 健康节点的 φ 峰值 | 3.17 | 3.17 |
| **误杀健康节点次数** | **3** | **0** |

这张表是整章最该盯着看的东西。两次运行**完全相同**——同样的种子、同样的网络、同样的健康服务端、同样的 φ 峰值 3.17——唯一的区别是阈值。φ≥1 谋杀了健康节点 3 次，φ≥8 一次都没有。

为什么？因为 φ≥1 的意思是「我容忍 10% 的冤枉率」，但这条链路的 p99 到达间隔是 240ms，一个活节点的心跳骑上延迟尾时，φ 会临时窜到 3.17，远超 1——于是一个活得好好的、只是这一拍慢了的节点，被当成死的杀掉。把阈值抬到 8（压过延迟尾），φ 峰值 3.17 永远够不着，0 误杀。

**这就是故障检测的全部设计空间，它直接从 φ 的数学里掉出来：**

> 紧阈值 → 真崩溃检测快，但分不清「尾延迟的心跳」和「死」，于是谋杀健康节点。
> 松阈值 → 永不误杀，但真崩溃检测慢。

φ-accrual 最大的价值不是「更准」，而是**把这个旋钮直接暴露给你**：你选的那个数字，字面意义上就是你的误报预算。裸超时把这个旋钮藏在一个「30 秒」这样拍脑袋的常量里，你根本看不见自己在拿可用性换正确性。

诚实声明（代码注释里也写了）：这里的延迟是模拟的（种子化抖动，不是真网卡），所以绝对毫秒数偏乐观、偏干净。**能迁移到真实网络的是相对结构**：φ 在健康抖动下保持低位、在真崩溃后尖刺，以及「阈值压不住延迟尾就会误杀」这条因果。绝对毫秒数迁移不了，真实网络的尾比模拟脏得多。

这也直接接上第 4 章：Raft 的选主超时（election timeout）本质就是一个故障检测器——follower 多久没收到 leader 心跳就发起选举。Raft 里 election timeout 设太短会导致活着的 leader 被不断「误判死亡」、反复触发无谓选举（这一章的 false positive，在 Raft 里表现为 split vote 和吞吐崩塌）；设太长则 leader 真崩后恢复慢。你在这一章调的旋钮，第 4 章会再调一次，代价完全同构。

## ⚡ 前沿：超时正在从应用代码下沉到 sidecar，而 deadline 比单点超时更重要

这一章手写的超时、重试、幂等、熔断，工程上正在经历一次「位置迁移」：从散落在每个应用的业务代码里，下沉到服务网格（service mesh，专管服务间通信的基础设施层）的 sidecar（边车，和应用进程并排跑、代理其所有网络流量的进程）里。Istio、Linkerd 这类网格把超时、重试、熔断做成配置——你在 YAML 里写「这个调用超时 500ms、重试 2 次、连续失败 5 次就熔断」，应用代码一行都不用动。好处是策略统一、可观测性集中、跨语言一致（Python 服务和 Go 服务用同一套重试逻辑）。

但这里有一个被反复忽视、且**目前没有通用解**的难点：**deadline propagation（截止时间传播）比单点超时重要得多。** 单点超时是「我等你 500ms」。deadline 是「整个请求链必须在 t 时刻前完成，我把剩余预算往下传」。区别在哪？设想 A→B→C→D 的调用链，每段都配 500ms 超时。A 等了 480ms，B 才把请求传到 C，但 C 仍然天真地从头算自己的 500ms——结果总时延能堆到 2 秒，而 A 早在 500ms 就放弃返回错误了。C 和 D 还在为一个**没人等的结果**烧 CPU 和连接，这叫 work amplification（无效工作放大），在过载时会演变成雪崩。

正确做法是传 deadline 而非 timeout：A 算出绝对截止时刻 `now+500ms`，把它放进请求头一路传下去，B/C/D 各自用「截止时刻 − 当前时刻」算自己还剩多少预算，预算见底就立刻放弃、不再往下发。gRPC 原生支持这个（`grpc-timeout` header 自动传播），但**跨异构系统的 deadline 传播至今没有统一标准**：HTTP REST 没有约定俗成的 deadline header，消息队列里 deadline 的语义更模糊（消息可能在队列里躺很久），跨语言/跨框架的传播经常断链。再叠上重试——sidecar 自动重试时，重试预算（retry budget）必须从原始 deadline 里扣，否则「每段重试 2 次」会在多跳链路上指数级放大请求量，把本来想救的服务直接打死。如何在服务网格里把 deadline、重试预算、熔断三者一致地协同，且跨异构协议不断链，**仍是正在研究和工程化的开放问题**——这也是为什么哪怕你用了 Istio，理解这一章手写的超时语义依然不可省。
