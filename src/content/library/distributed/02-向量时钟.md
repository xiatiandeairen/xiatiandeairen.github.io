第 1 章我们用 Lamport 标量时钟(scalar clock，每个进程持一个单调递增的整数)证明了一件事：它能给所有事件排出一个**全序**(total order，任意两事件都分得出先后)。但那一章结尾留了个洞没补——标量时钟**测不出并发**。它对两个本来毫无因果关系的独立事件，照样吐出一个「a 在 b 之前」。这个顺序是它**编造**的。

把同一份事件历史(seed=7,同一套调度)喂给两种时钟,数字直接把洞露出来:

```
relation                  pair count
------------------------  ----------
a -> b / b -> a (causal)          57
a || b (concurrent)                9
a == b (equal)                     0

clock                        missed concurrency
---------------------------  ------------------
vector clock (ground truth)                   0
lamport scalar (ch.1)                         5
```

66 个事件对里,有 9 对是真正并发的。向量时钟一对没漏(missed concurrency=0)。Lamport 标量把其中 5 对**当成了因果先后**——它给 5 对独立事件发明了顺序。这一章就是把这 5 个谎言一个个揪出来,讲清楚向量时钟凭什么不撒这个谎,以及它要为此付出什么代价(剧透:O(N) 的空间,大规模下会爆)。

## 一、Lamport 的洞:全序 vs 偏序

先把第 1 章的结论压成一句话:**happens-before 关系(记作 →)本质是偏序,Lamport 标量却是全序。用全序去表示偏序,信息必然丢失。**

「偏序」(partial order)的大白话:有些事情分得出先后,有些事情根本就没有先后。你早上喝咖啡 → 你出门,这有先后(喝咖啡导致你晚出门 5 分钟)。但你在北京喝咖啡 / 你朋友同一秒在上海打喷嚏——这两件事**没有任何因果关系**,问「谁先谁后」是个伪问题,答案是「无所谓,它们并发」。

分布式系统里的并发就是这个意思:两个事件,谁都没给谁发过消息(直接或间接),它们就是独立的。Lamport 时钟的更新规则(本地事件 +1、收消息 max+1)能保证「a → b 则 L(a) < L(b)」,但**反过来不成立**:L(a) < L(b) 不代表 a → b,它们可能只是并发,标量却被迫给了个数值大小。

代码里这个「被迫」体现得很直白。看比较函数(`examples/distributed-from-scratch/src/stage02-vector-clock.ts`):

```typescript
function compareLamport(a: number, b: number): Relation {
  if (a === b) return "equal";
  return a < b ? "before" : "after";
}
```

一个 `number` 只有三种结局:相等、小于、大于。**结构上就没有「并发」这个返回值的位置**。这不是程序员偷懒,是标量这个数据结构的天花板——你拿一个一维的东西,没法表达「两个方向上各自领先」的二维事实。

代码里跑出来的具体谎言长这样:

```
n2#1 send [0,0,1] (L=1) vs n1#1 recv [1,1,0] (L=2):
  vector clock says CONCURRENT, lamport says before (L 1 vs 2) — a fabricated order
```

`n2#1 send`(n2 的第一次发送)和 `n1#1 recv`(n1 的第一次接收)。n2 发消息时只知道自己干了一件事(向量 `[0,0,1]`);n1 收消息时已经从 n0 那继承了历史(向量 `[1,1,0]`)。这两个事件**互相不知道对方存在**——n2 的发送没参与 n1 那次接收的因果链。但 Lamport 标量 L=1 < L=2,它咬定 n2 的发送在前。**纯属编造**。

### 失败模式:为什么不能「重试」或「加个 epsilon」修这个洞

工程师碰到「测不准」第一反应常是打补丁:能不能两个标量很接近(比如差 1)就当并发?不行。差值大小和因果关系**没有任何对应**——一条因果链可以让两个事件差 100,两个并发事件也可能恰好差 1。补丁修不了结构缺陷,你需要换数据结构。

## 二、向量时钟:用 N 个计数器装下偏序

修法很朴素:既然一个标量装不下偏序,就给**每个节点配一个计数器**。N 个节点,每个事件就带一个长度为 N 的向量,`vc[k]` 的含义是「在我这个事件之前,我所知道的节点 k 已经发生了多少个事件」。

数据结构和更新规则在 `VectorClockNode` 里:

```typescript
type VectorClock = number[];  // 一节点一个计数器,按 NODE_IDS 顺序索引

// 发送规则:发送是本地事件 -> 先给自己那一分量 +1,再把向量的副本随消息发出
sendEvent(to: string): void {
  this.vc[this.idx]++;
  this.lamport++;
  this.record("send");
  this.sendTo(to, "tick", { vc: [...this.vc], lamport: this.lamport });
}

// 接收规则:逐分量取 max 合并发送方的认知,再给自己那一分量 +1
override onMessage(_from: string, msg: Message): void {
  const payload = msg.payload as { vc: VectorClock; lamport: number };
  for (let k = 0; k < this.vc.length; k++) {
    this.vc[k] = Math.max(this.vc[k], payload.vc[k]);
  }
  this.vc[this.idx]++;
  this.lamport = Math.max(this.lamport, payload.lamport) + 1;
  this.record("recv");
}
```

两个细节值得停下来看,它们都是「写错了静默坏掉」的雷区:

**雷区 1:merge 必须在 bump 之前。** 接收时先逐分量 `max`(把发送方因果知道的一切吸收进来),**然后**才给自己 +1。顺序反了会怎样?如果先给自己 +1 再 merge,你给自己计的这次接收事件,会被算进「合并发送方认知」的范围里,导致因果关系的传递性(a→b、b→c 则 a→c)在边界上算错。merge-before-bump 才能保证「我继承了发送方因果知道的全部」。

**雷区 2:随消息发的必须是副本,不是引用。** 代码里写的是 `[...this.vc]`(展开成新数组)。如果发引用,发送方之后每次 `vc[idx]++` 都会顺着引用改掉「已经发出去的那条消息携带的时间戳」——接收方拿到的会是发送方**未来**的状态,而不是发送那一刻的状态。历史被改写,整个因果分析全错。`EventRecord.vc` 存进 history 时同样用 `[...this.vc]` 拍快照,理由一样:防止后续 mutation 倒灌污染已记录的历史。

### 偏序比较:domination 即 happens-before

有了向量,「谁在谁前」和「是否并发」就能精确判定:

```typescript
function compareVectorClocks(a: VectorClock, b: VectorClock): Relation {
  let aLessEq = true; // a <= b 在每一分量上是否都成立
  let bLessEq = true; // b <= a 在每一分量上是否都成立
  for (let k = 0; k < a.length; k++) {
    if (a[k] > b[k]) aLessEq = false;
    if (b[k] > a[k]) bLessEq = false;
  }
  if (aLessEq && bLessEq) return "equal";   // 向量完全相同
  if (aLessEq) return "before";             // a 被 b 支配
  if (bLessEq) return "after";              // b 被 a 支配
  return "concurrent";                      // 各有对方没有的分量 -> 独立
}
```

规则一句话:**a → b 当且仅当 a 的向量在每一分量上都 ≤ b,且不完全相等(即 a 被 b「支配」)。** 如果两者谁都不支配谁——a 在某分量领先、b 在另一分量领先——它们就是**并发**。

为什么这个判定是对的(而不只是凑巧)?因为 `vc[k]` 数的就是「节点 k 上因果地早于本事件的事件数」。a 在每个分量都 ≤ b,意思是「b 因果地知道 a 知道的一切,还可能更多」——这正是 happens-before 的传递闭包。而 `concurrent` 这个返回值的存在,就是标量结构缺失、向量结构补上的那一格。回头看上面那个例子:`[0,0,1]` vs `[1,1,0]`,前者第 3 分量领先(0,0,**1**),后者前两分量领先(**1,1**,0),互不支配 → 并发。向量时钟答得干脆,Lamport 答不出来。

### 失败模式:「只比第一个分量」的诱人捷径

向量比较要遍历全部 N 个分量。有人会想:能不能只看第一个分量图个快?代码专门把这个错误实现出来打脸:

```typescript
// BROKEN: 只比 component 0
function compareFirstComponentOnly(a: VectorClock, b: VectorClock): Relation {
  if (a[0] === b[0]) return "equal";
  return a[0] < b[0] ? "before" : "after"; // 退化成全序,并发被抹掉
}
```

跑出来的对比:

```
comparator                     missed concurrency  total mislabels
-----------------------------  ------------------  ---------------
first-component-only (BROKEN)                   9               31
full-component (FIXED)                          0                0

  broken e.g. n0#1 send [1,0,0] vs n2#1 send [0,0,1]:
    truly CONCURRENT, but first-component compare (1 vs 0) calls it after — concurrency erased
```

只比第一分量,等于把「n0 这一个节点的本地进度」当成全局顺序——**这就是 Lamport 的错误换了件向量的外衣**。9 对并发全被抹掉,66 对里 31 对直接判错。把比较换回全分量,missed 和 mislabel 同时归零。教训很硬:**偏序就是偏序,任何想把它压回一维的捷径都会重新引入你刚修掉的 bug。** 这不是优化,是退化。

(关于这套数字的可复现性:`[determinism] seed=7 fixed schedule => byte-identical output across runs`——固定 seed + 固定调度,每次跑字节级一致。所以上面所有计数都是真实运行产出,不是估算。需要注意的是**绝对**的并发对数(9)依赖这个 toy 拓扑和 seed,换个更密或更疏的消息调度数字会变;能跨场景迁移的是**相对**结论:Lamport missed > 0,向量时钟 missed = 0,捷径会重新引入错误。)

## 三、✦ 专家钩子:O(N) 是个会爆的代价

讲到这你可能觉得向量时钟完胜,该全面替换 Lamport。资深工程师在这里必须踩刹车,问一句:**这个精确度的账单是谁付的?**

答案:**每条消息都得带一个长度为 N 的向量。空间代价 O(N) per message,而且 N 是节点总数。** 小集群(本章 N=3)无所谓,三个整数而已。但把场景换成 Amazon Dynamo 那种成百上千节点的系统,每条内部消息都拖一个上千维的向量——带宽、序列化、存储,全线被这个向量吃掉。更糟的是,这个 N 还会**随集群扩容线性增长**,你节点越多,每条消息越胖。

这就是为什么真实系统对向量时钟做了一系列**工程妥协**,而不是教科书式照搬:

- **Dynamo 用的是 version vector(版本向量)**,只对「写过这个 key 的节点」记一个计数器,而不是全集群所有节点。一个 key 通常只被少数几个副本写过,向量就短得多。但它还有个更阴的问题:某个 client 长期写同一个 key,version vector 会无限膨胀。Dynamo 的对策是**截断**(truncation)——超过阈值就丢掉最老的条目。代价是:截断后两个版本可能被误判成因果可比(本来并发),也就是**重新引入了一点点 missed concurrency**,拿正确性换有界的空间。这是个清醒的取舍,不是 bug。
- **Dotted version vector(DVV,带点的版本向量)** 是对截断方案的修正。经典 version vector 在「同一 client 对同一 key 连续多次写、中间夹着并发写」的场景下会丢掉本该保留的并发分支(produce 假的因果或假的冲突)。DVV 给每个值额外附一个「dot」(单次写的精确标识),把「这个节点的连续写」和「这个节点的版本上界」分开记,从而在不无限膨胀的前提下精确保留并发分支。Riak 生产环境就是踩了 version vector 的坑之后换成 DVV 的。

一句话收口:**向量时钟是「正确性的基准」,但基准在大规模下买不起。真实系统的工程史,基本就是「在 O(N) 的精确和有界的空间之间找一个能接受的损失点」的历史。** 你设计系统选时钟时,第一个要问的不是「精不精确」,而是「N 有多大、向量带在哪、它会不会随扩容爆掉」。

## 四、⚡ 前沿:节点集合不是固定的怎么办

向量时钟(以及 version vector)有个被教科书悄悄藏起来的**硬假设:节点集合是固定且全局已知的。** 向量的每一个分量都对应一个确定的节点,索引和节点一一绑死(本章代码里就是 `NODE_IDS.indexOf(id)` 把节点 id 映射到固定下标)。

可现实里节点是动态的:云上自动扩缩容,机器随时加入退出;P2P 网络成员来去自由;边缘设备时连时断。**节点动态加入退出时,向量该长还是该短?新节点的分量从哪冒出来?退出的节点那一格能不能回收?谁来全局协调这个映射?** 经典向量时钟对这些问题统统没有答案——它根本没设计成能处理变动的成员集合。临时打补丁(比如每次成员变化全局重建向量)在大规模或高频变动下完全不现实。

目前最有意思的一个研究方向是 **Interval Tree Clock(ITC,区间树时钟)**。它的核心点子是:不再用「固定下标的向量」表示身份,而是用一棵二叉树把整个 id 空间切成区间,每个参与方持有其中一段区间。它定义了三个语义操作:

- **fork(分叉)**:一个参与方要派生出新参与方时,把自己持有的区间一分为二,各拿一半。**这就是「节点加入」**——新成员的身份是从老成员的区间里分出来的,不需要全局协调,也不需要事先知道总节点数。
- **join(合并)**:两个参与方退出/合并时,把它们的区间合回去。**这就是「节点退出」**——退出节点的区间被回收,不留空洞。
- **event(事件)**:在自己持有的区间上推进逻辑时间。

fork/join 让 ITC 的「向量」能随成员集合动态伸缩,从根上绕开了「节点集合固定」的硬假设。这个方向**目前还没有成为工业界默认方案**——多数生产系统还是用 version vector + 各种截断/DVV 的妥协扛着,ITC 更多停留在论文和少数实现里,动态成员的因果追踪在超大规模、高频变动场景下**仍是开放问题**,没有一个被广泛采纳的通用解。如果你做的系统成员高度动态,这是值得盯的一条线。

## 小结与下一章

把这章钉死成几条:

1. **happens-before 是偏序,标量时钟是全序;全序装不下偏序,所以 Lamport 必然对并发事件编造顺序**(实测:seed=7 下编造了 5 对)。
2. **向量时钟用 N 个计数器精确表示偏序**:发送给自己分量 +1,接收逐分量 max 再 +1(merge 必须先于 bump,消息携带的必须是副本)。
3. **判定规则**:a 的向量逐分量 ≤ b 且不全等 → a → b;互不支配 → 并发。实测把 missed concurrency 从 5 压到 0。
4. **任何「只看部分分量」的捷径都会退化回标量的错误**(实测:只比第一分量,9 对并发全丢、31 对判错)。
5. **代价是 O(N) per message,大规模会爆**——真实系统用 version vector + 截断 / DVV 做有界空间的妥协,拿一点正确性换可扩展。
6. **⚡ 节点集合固定是硬假设**;动态成员的因果追踪(ITC 的 fork/join 是一个方向)目前无通用解。

下一步去哪:向量时钟在本章只是「事后分析事件历史」的离线工具。把它**在线**用起来——让多个副本各自接收乱序更新、用向量时钟判断「这两个更新到底冲突还是可以安全合并」——就是 **第 7 章 CRDT(无冲突复制数据类型)** 的核心机制。CRDT 判冲突,本质就是这章的偏序比较:本章 `compareVectorClocks` 返回 `concurrent` 的那些更新对,正是需要 CRDT 的合并函数(merge)来调和的冲突。第 7 章的 G-Counter 把状态做成一个「逐副本贡献向量」、合并时逐分量取 max——那正是本章向量比较里「逐分量看谁支配谁」的同一套偏序运算,只是这次用它来**收敛状态**而非**事后判定关系**。你这章手写的偏序直觉,到第 7 章会换一身代码再出现。
