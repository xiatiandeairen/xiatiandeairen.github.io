先看一个数字。三个节点跑一轮消息交换,记下 18 个事件,系统里一共有 153 对不同的事件。其中 32 对在因果上是**并发**的——谁也没影响谁。但用 Lamport 时钟给它们盖的时间戳一比,23 对(71.9%)戳是不一样的。一个只看时间戳的程序会从这 23 对里读出"A 先于 B"的结论。**全错。** 它们之间根本没有先后,戳的差异是个假象。

这就是这一章要钉死的事:Lamport 逻辑时钟(logical clock,用整数计数器代替墙钟来给事件排序)是分布式系统里第一个真正能用的武器,但它有一条几乎所有人都会踩的边:**它只保证「因果在前 ⇒ 戳更小」,反过来不成立。** 你不能拿戳的大小反推因果。这个单向蕴含是无数分布式 bug 的根源,本章用配套代码仓里能跑的真实数据把它两半都证给你看。

## 接着第 0 章:没有全局时钟,那拿什么排序

第 0 章立的公理是:分布式系统里没有一个所有节点都同意的"现在几点"。每台机器有自己的石英晶振,有自己的时钟漂移(clock skew,两台机器读数的差),NTP(网络对时协议)校准也只能把误差压到几十毫秒、偶尔几百毫秒,还会因为闰秒、虚拟机时钟回拨而**倒流**。所以"按墙钟时间戳排序事件"这条路从一开始就堵死了——你拿 A 机的 `Date.now()` 和 B 机的 `Date.now()` 比大小,比出来的可能跟真实发生顺序相反。

Lamport 1978 年那篇 *Time, Clocks, and the Ordering of Events* 给出的洞见是:**分布式系统真正需要的不是"真实时刻",而是"谁导致了谁"。** 如果事件 A 是事件 B 的原因(比如 A 发了条消息,B 收到了它),那任何排序都必须把 A 排在 B 前面。但如果 A 和 B 八竿子打不着——两个节点各干各的,中间没有任何消息往来——那它们谁先谁后**根本无所谓**,甚至这个问题本身没有意义。

这个"谁导致了谁"的关系,Lamport 叫它 **happens-before**(发生在前,记作 `→`),严格定义只有三条:

1. 同一个进程内,先记录的事件 `→` 后记录的事件(程序顺序)。
2. 一条消息的"发送" `→` 它的"接收"(消息顺序)。
3. 传递性:`a → b` 且 `b → c` 则 `a → c`。

凡是不满足 `a → b` 也不满足 `b → a` 的两个事件,定义为**并发**(concurrent,记作 `a ∥ b`)。注意:并发不是"同时发生",而是"因果无关"——它们可能在墙钟上差了一小时,但只要中间没有因果链,就是并发。这是整章最反直觉、也最关键的一点。

## 时钟规则:三行代码买到一个一致的因果戳

Lamport 时钟的全部机制就是一个每个进程各持一份的整数计数器,加三条更新规则。配套仓里的实现 `examples/distributed-from-scratch/src/stage01-lamport-clock.ts` 把它压成一个继承自 `core/node.ts` 基类的进程类:

```ts
// examples/distributed-from-scratch/src/stage01-lamport-clock.ts
class LamportProcess extends Node {
  private counter = 0;

  private doLocalThenSend(to: string): void {
    // local 事件:counter += 1
    this.counter += 1;
    recordEvent(this.id, "local", this.counter, "internal work");

    // send 事件:counter += 1,并把戳贴到消息上随线发出
    this.counter += 1;
    const mid = midCounter++;
    recordEvent(this.id, "send", this.counter, `send -> ${to} (mid=${mid})`, mid);
    this.sendTo(to, "ping", { lamport: this.counter, mid });
  }

  onMessage(_from: string, msg: Message): void {
    const payload = msg.payload as { lamport: number; mid: number };
    // recv 事件:counter = max(本地, 消息戳) + 1
    this.counter = Math.max(this.counter, payload.lamport) + 1;
    recordEvent(this.id, "recv", this.counter, `recv from ${msg.from} (mid=${payload.mid})`, payload.mid);
  }
}
```

三条规则:

- **本地事件 / 发送**:`counter += 1`,再用新值给事件盖戳。
- **接收**:`counter = max(本地 counter, 消息里带的戳) + 1`。

第三条里的 `max(...) + 1` 是**承重墙**。想象一下没有它会怎样:节点 n2 的计数器一直很慢(它没怎么干活),突然收到 n0 发来的、戳是 13 的消息。如果 n2 只是 `counter += 1`,它给这个 recv 盖的戳可能是 3——比发送的 13 还小。那么"发送 → 接收"这条铁律就被违反了:因果在前的事件反而戳更大。`max` 把接收方的计数器强行拉到发送戳之上,`+1` 再保证严格大于。这一行就是 `send → recv ⇒ C(send) < C(recv)` 的全部保证来源。

为什么把整个进程状态压成"一个计数器"值得花一章讲?因为这正是 Lamport 的功力所在:**用最少的活动部件,买到一个全局一致的因果时间戳。** 没有协调,没有共识,没有额外消息——每个节点只在自己本来就要做的三类动作上顺手维护一个整数,搭车在已有消息上捎带一个戳。代价低到几乎免费,这是它至今活在每一个真实系统里的原因。

### 跑在确定性网络上,让并发"长"出来

这章不是用两个事件的玩具例子讲(`a` 发消息给 `b`,完事)。并发只有在真实的消息交错里才有意思,所以代码让三个节点跑在 `core/` 的确定性仿真网络上——这套网络(`core/network.ts` + `core/clock.ts`)用注入的种子化随机数(seeded PRNG)模拟真实网络的抖动(jitter)和乱序(reorder),同一个 seed 跑出来字节级可复现。stage 里开了 8ms 抖动 + 乱序,再用手工脚本让三个节点在大致相同的虚拟时刻(`atMs: 20`)各自干独立的活:

```ts
n0.scheduleScript([{ atMs: 20, to: "n1" }, { atMs: 200, to: "n2" }]);
n1.scheduleScript([{ atMs: 20, to: "n2" }, { atMs: 120, to: "n0" }]);
n2.scheduleScript([{ atMs: 20, to: "n0" }, { atMs: 300, to: "n1" }]);
ctx.net.reorder(true);
```

`atMs: 20` 三个节点撞在一起、彼此之间没有消息——这就是并发的产地。后面那些跨节点的消息(n0→n1→n2)则织出因果链,给我们提供一大批"真有先后"的对照组。整轮跑完:`[msgs] sent=6 delivered=6 dropped=0`,`finalTime=316ms`,18 个事件全部记下。

## 证明上半句:因果在前,戳一定更小(0 违反)

光说不算,代码要把保证**算**出来,不是断言出来。关键在于:仿真跑完后,stage 独立地、不看任何 Lamport 戳地,从两类边重建出真正的 happens-before 关系——程序顺序边(同节点相邻事件)+ 消息边(每个 send 配对它的 recv),再求传递闭包。这套重建是**ground truth**(地面真相),因为我们自己拥有完整事件日志和消息 DAG(有向无环图),比对是精确的,不是统计估计。

```ts
// computeHappensBefore:从 (程序顺序 ∪ 消息边) 求传递闭包,reaches[a][b] 表示 a -> b
// analyzePairs:遍历每一对事件,验证保证
if (aFirst || bFirst) {
  stats.causalPairs += 1;
  const earlier = aFirst ? events[i] : events[j];
  const later = aFirst ? events[j] : events[i];
  // 保证:a -> b ⇒ C(a) < C(b)。数违反,期望 0。
  if (!(earlier.lamport < later.lamport)) stats.causalStampViolations += 1;
}
```

跑出来:

```
causal pairs (a->b): 121  | stamp-order violations: 0 (guarantee: must be 0)
```

121 对有因果关系的事件,**违反次数 0**。这是 Lamport 时钟的承诺,实测兑现:只要 `a → b`,戳就一定 `C(a) < C(b)`。注意这是计算出来的 0,不是写死的断言——如果 `max(...) + 1` 这行被改坏,这个数立刻变非零。这就是"绑定可跑代码"和"survey 里贴个公式"的区别。

我们顺便用这套戳产出一个全局总序(把所有事件按戳排,平局用节点 id 再 gid 打破)。运行输出 (a) 段那张表就是它——L=1 到 L=12,18 个事件排成一条线。这个总序是合法的:它是 happens-before 偏序的一个**线性化**(把偏序补成全序的一种方式)。但请记住这句话:**它给并发事件也强行排了序**,而那个序是凭空发明的。这正是下半句要算的账。

## ✦ 证明下半句:戳更小 ≠ 因果在前(无数 bug 的根源)

现在反过来问:如果 `C(a) < C(b)`,能不能推出 `a → b`?**不能。** 这就是专家钩子里那条单向蕴含——`a → b ⇒ C(a) < C(b)` 成立,但它的逆 `C(a) < C(b) ⇒ a → b` **不成立**。这不是吹毛求疵的理论瑕疵,而是你写分布式代码时最容易掉进去的坑:看到时间戳 A 比 B 小,顺手就当成"A 先发生、A 是因",于是去做"用旧值覆盖新值"之类的判断——而它们其实是并发的,谁覆盖谁都是错。

代码把这笔账精确算出来。对那 32 对并发的事件(`computeHappensBefore` 里谁也到不了谁),数一数有多少对的 Lamport 戳是不同的:

```
concurrent pairs (a||b): 32  | same stamp: 9  different stamp: 23
=> of concurrent pairs, 71.9% carry DIFFERENT stamps, i.e. Lamport falsely implies an order for them
```

**71.9%。** 也就是说,本次运行里近四分之三的并发事件对,被 Lamport 时钟盖上了不同的戳——一个只读戳的程序会从这些对里凭空读出顺序。stage 还揪出一个具体的反例,白纸黑字证明它确实并发:

```
Two events with DIFFERENT Lamport stamps:
  A: [gid 0] n0 local L=1 (internal work)
  B: [gid 3] n1 send L=2 (send -> n2 (mid=1))
Stamps differ (1 vs 2), so a stamp-reader infers an order.
But in the happens-before DAG, A does NOT reach B and B does NOT reach A:
  => they are CONCURRENT; the stamp difference is an artifact, not causality.
```

A 是 n0 在 L=1 干的本地活,B 是 n1 在 L=2 发出的消息。戳 1 < 2,任何看戳的人都会说"A 在 B 前"。但在 happens-before DAG 里,A 到不了 B,B 也到不了 A——它们发生在不同节点,中间隔着零条消息。**它们是并发的,戳的差异纯属计数器各跑各的产生的噪声,不是因果。**

为什么会这样?根子在 happens-before 是个**偏序**(partial order,有些元素之间没有可比性),而 Lamport 戳是个**整数**——整数天生全序,任意两个整数都能比大小。你把一个偏序硬塞进全序,必然要给本来不可比的元素编造出可比性。**全序无法表达"不可比"这个信息。** 这是数学上的根本限制,不是实现没写好。

### 失败模式:这条坑在生产里长什么样

抽象的"不能反推"落到代码里,最经典的翻车是 **last-write-wins**(LWW,谁的时间戳大谁赢)冲突解决。两个客户端并发改同一个 key,服务端用 Lamport 戳(甚至墙钟戳)判定"戳大的是最新值,覆盖戳小的"。问题来了:并发写的两个戳**一定不相等**(同一计数器极少撞值,本例 32 对并发里也只有 9 对同戳),于是系统总会挑一个"赢家",**静默丢掉另一个写**。用户视角:我明明保存成功了,刷新一看没了。这种 bug 极难复现,因为它依赖特定的消息交错——换个 seed、换个网络延迟就不出现了。第 0 章那句"失败是不确定的"在这里具象化了。

更阴险的是它**不报错**。`stamp-order violations: 0` 告诉你保证从没被违反,系统看起来一切正常。bug 不在"戳排错了",而在"你**用错了**戳"——你把一个只承诺单向蕴含的工具,当成双向的来用。工具没坏,是用法越界。这也是为什么这类 bug 通常熬到生产环境、靠用户投诉数据丢失才暴露。

## 这缺陷怎么修:留给第 2 章的向量时钟

根因清楚了:单个整数信息量不够,它没法区分"真有先后"和"碰巧戳不同"。修法的方向也就清楚了——**别用一个数,用一组数。** 让每个节点维护一个长度为 N(节点数)的向量,第 i 位记"我所知道的节点 i 的进度"。这样两个事件的戳就能比出三种关系:逐位都 ≤(有先后)、逐位都 ≥(反向先后)、有的位大有的位小(**真并发**)。向量时钟(vector clock)能检测并发,Lamport 不能——本章输出最后那句 `Vector clocks, next chapter, can` 就是这个交接。

但向量时钟不是免费午餐:戳从 1 个整数变成 N 个,消息体积、存储都随集群规模线性膨胀。Lamport 用一个整数换"全序但不能测并发",向量时钟用 N 个整数换"能测并发但开销大"。这是分布式系统里反复出现的母题——**没有银弹,只有取舍**。第 2 章会把向量时钟也手写出来、跑同一套对照实验,你会看到那个 71.9% 的假象被消掉,代价写在消息字节数里。

## ⚡ 前沿:HLC——把物理时钟塞回逻辑时钟

逻辑时钟有个让运维抓狂的副作用:戳是个跟墙钟完全脱钩的整数。日志里一个事件标着 `L=10086`,你根本不知道它大概是什么时候发生的,没法跟监控、跟用户报障的时间对齐。而纯墙钟戳又会倒流、会因 skew 排错因果。能不能两个都要——既保住因果(`a → b ⇒ 戳更小`),又让戳**大致接近真实时间**、方便人读?

**混合逻辑时钟**(Hybrid Logical Clock,HLC)就是干这个的。它把戳拆成两段:一段记物理墙钟时间(`pt`),一段记逻辑计数器(`l`)。更新规则大意是:取本地物理时钟、收到消息的物理段、本地已有物理段三者的最大值当新的物理段;只有当物理段没往前走(因为时钟没动或在倒流)时,才靠逻辑计数器 `+1` 来保住单调和因果。这样戳永远单调递增、永远满足 happens-before 的单向蕴含,同时物理段始终咬住真实时间、漂移被逻辑段兜底,人一眼能读出"哦这大概是下午三点"。

HLC 不是论文里的玩具:**CockroachDB 用它做事务时间戳排序**,YugabyteDB、MongoDB 也用了类似机制。它是逻辑时钟思想在工业界最成功的落地之一。但要把话说清楚:**HLC 和 Lamport 一样,本质仍是个准全序戳,它依然不能可靠检测两个事件是否并发**——它修的是"可读性"和"对齐墙钟",不是 Lamport 那条"全序丢失并发信息"的根本缺陷。

而真正的开放问题在更上游:**如何把时钟不确定性的边界本身变成一等公民。** Google Spanner 的 TrueTime 走的是另一条路——不假装时钟精确,而是给每个时间戳附一个误差区间 `[t-ε, t+ε]`,靠数据中心里的 GPS 和原子钟把 ε 压到几毫秒,然后用"提交时主动等待 ε"(commit-wait)来换取跨地域的外部一致性。但 TrueTime 依赖专门的硬件基础设施,**普通集群拿不到那么紧的 ε**;而 HLC 完全软件实现、零特殊硬件,代价是它给的因果保证比 TrueTime 弱。**"在纯软件、无特殊硬件的前提下,既要紧的时间不确定性边界、又要可检测并发、还要低开销"——目前没有通用解,仍是活跃的研究方向。** 你能拿到的永远是这个三角里的某个折中,而选哪个折中,取决于你的系统到底怕错排序、还是怕丢并发、还是怕加硬件。
