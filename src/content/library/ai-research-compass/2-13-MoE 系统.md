> 这一章把你从"理解稠密 Transformer 怎么并行、怎么省显存"带到"能从零讲清混合专家(MoE)为什么能把参数堆大几十倍而单 token 算力几乎不变,能手推它的两次 all-to-all 通信量,能讲透负载均衡为什么是它的第一难题、capacity factor 和 token drop 各自在解决什么"。读完你应该能默写 top-1/top-2 routing 的前向、独立推导 all-to-all 在专家并行下的精确通信字节数、解释为什么辅助损失能压住"赢者通吃",并看懂 GShard / Switch Transformer / DeepSpeed-MoE 的工程取舍。

## 一、动机:参数和算力为什么必须解耦

到这一章为止,前面所有的并行手段——数据并行扩 batch、张量并行切单层、流水线并行切层、ZeRO 切存储——都在解决同一个隐含矛盾:**模型越大越聪明,但越大越放不下、越训不动**。它们全是在"稠密(dense)"假设下做文章:**每个参数,对每个 token,都参与计算**。

这个稠密假设藏着一个被默认却从未被质疑的等式。复习第 01 章的算力账本:一个有 N 个参数的 Transformer,处理一个 token 的前向计算量约 `2N` FLOPs(每个权重一次乘一次加),反向约两倍,合计约 `6N`。**注意这个 N 同时是"模型容量"和"单 token 算力"——它们被锁死在一个数上。** 想让模型更聪明(更大的 N),就必须线性地多烧算力。

但这个锁死真的有必要吗?来看人脑的直觉:你回答一道数学题时,负责视觉、运动、语言韵律的那些神经回路基本不激活。**容量是"总共会的东西",算力是"这次用到的东西",二者本不该相等。** 稠密 Transformer 之所以让它们相等,纯粹是因为 FFN(前馈层)这个结构——每个 token 都流过同一个巨大的 `[h, 4h] → [4h, h]` 矩阵——而 FFN 恰恰是 Transformer 里参数量和计算量的大头(约占 2/3,第 01 章算过)。

**混合专家(Mixture of Experts,MoE)的母题用一句话概括:**

> 把那个所有 token 共享的大 FFN,换成 `E` 个并列的小 FFN(称为"专家",expert),再加一个"路由器"(router / gate),让每个 token 只被送进其中 `k` 个专家(通常 k=1 或 2),其余专家对它完全不计算。

这一刀切下去,容量和算力就**解耦**了:

```
总参数量    ∝ E × (单专家参数)     ← 想多大有多大,E 可以是 8、64、256……
单token算力 ∝ k × (单专家算力)     ← 几乎不变,k 是个小常数(1 或 2)
```

举个具体数字感受这个杠杆:把一个 FFN 复制成 64 个专家,**总参数涨约 64 倍**,但若 top-1 路由(k=1),**单 token 流过的 FFN 算力和原来一个 FFN 完全一样**。这就是 MoE 的全部魅力——**用近乎恒定的算力买到几十倍的参数容量**。Switch Transformer 把模型推到 1.6 万亿参数,靠的正是这个;近年的开源 Mixtral 8×7B(8 专家、top-2)、DeepSeek-MoE、以及业界普遍相信的若干前沿大模型,都是这条路线。

**但天下没有免费的午餐,而 MoE 的午餐尤其贵。** 把这个思想真正落到几百上千张 GPU 上,会冒出三个稠密模型从不曾有的系统难题,它们构成本章的主线:

1. **专家放哪?** 64 个专家、每个都不小,一张卡放不下全部。于是引入**专家并行(expert parallelism)**:不同专家放不同卡。但 token 由 router 动态决定去哪个专家,而专家分散在各卡上——**token 必须被"路由"到它选中的那张卡去算,算完再收回来**。这就引出 MoE 系统的核心通信原语:**两次 all-to-all**。
2. **通信多贵?** 这两次 all-to-all 在每个 MoE 层的前向反向都要发生,我们要把它的通信量从头推到尾,看它和隐藏维、token 数、专家分布的关系。
3. **负载均不均?** router 是学出来的,它**天然倾向于把大多数 token 都送给少数几个"明星专家"**——这些专家所在的卡被打爆,其余卡闲置,整个系统被最慢的那张卡拖死。这是 MoE **最棘手的工程难题**,催生了辅助负载均衡损失、capacity factor、token drop/pad 这一整套机制。

**给系统老兵的迁移提示:** all-to-all 你可能在 MPI(`MPI_Alltoall`)、MapReduce 的 shuffle 阶段见过——**它就是一次全局的数据重分发,每个进程把自己手里的数据按目标分组,发给所有其他进程**。MoE 的 dispatch 就是一次 shuffle:按"token 去哪个专家"重新分组数据。你缺的不是通信直觉,而是"为什么路由会不均衡""capacity 这个上限怎么定"这两层 ML/系统耦合的常识。我们从机制开始。

## 二、稀疏激活的机制:router 与 top-k

### 2.1 一个 MoE 层到底在算什么

先把一个 MoE 层的前向写成数学。设输入是一批 token 的隐藏向量,展平后为 `X ∈ [T, h]`(T = batch × seq 的 token 总数,h 是隐藏维)。一个 MoE 层有:

- **`E` 个专家**,每个专家是一个独立的 FFN,记第 e 个专家为函数 `Eₑ(·)`,参数互不共享。
- **一个 router**,本质是一个小线性层 `W_g ∈ [h, E]`,把每个 token 映射成 E 个"亲和度分数"(logits)。

对单个 token 向量 `x ∈ ℝʰ`,前向分四步:

```
步骤1  路由打分:   s = softmax(x · W_g)          # s ∈ ℝᴱ,E 个专家的概率
步骤2  选 top-k:   I = TopK_indices(s, k)        # 选分数最高的 k 个专家下标
步骤3  各专家计算:  对每个 i ∈ I,算 yᵢ = Eᵢ(x)    # 只算选中的 k 个
步骤4  加权合并:   y = Σ_{i∈I} (sᵢ / Σ_{j∈I} sⱼ) · yᵢ   # 用归一化后的门控权重加权求和
```

**关键点 1:门控权重 sᵢ 必须乘进输出**,而不是只用来选专家。原因有二:其一,它让 router 可微——梯度能通过 sᵢ 回传到 W_g,router 才学得动(只做 argmax 选专家,选择本身不可导,梯度断了);其二,它表达"这个 token 有多依赖这个专家",是一个软权重。

**关键点 2:top-k 里的 k 直接决定算力。** top-1(Switch Transformer)单 token 流过 1 个专家;top-2(GShard、Mixtral)流过 2 个,算力翻倍但表达力更强(两个专家的组合)。**k 是"算力 vs 质量"的旋钮**,工业界主流是 k=1 或 k=2,极少更大——再大就退化回稠密、丢了 MoE 的意义。

### 2.2 为什么是 softmax 后选 top-k,而不是反过来

一个容易被一带而过、但值得想清楚的细节:上面是先对**全部 E 个**专家算 softmax,再选 top-k;另一种做法是先选 top-k logits,只对这 k 个算 softmax。二者数值不同。

主流(GShard、Switch)用**前者**:softmax over all E,选 top-k,再把选中的 k 个门控值重新归一化(步骤 4 的 `/Σⱼ` 就是重归一化)。这样做的好处是 softmax 的分母包含了所有专家,**未被选中专家的分数也参与了归一化,梯度能间接影响它们的相对大小**,有利于训练动态。具体哪种更优属于经验范畴,不同论文取舍不同(待核:Mixtral 等近期实现的确切归一化顺序)。本章按主流的"softmax-then-topk-then-renorm"展开,不影响系统层的所有结论。

### 2.3 单卡视角的伪代码(还没并行)

先不考虑多卡,把一个 MoE 层在单卡上的稠密-稀疏转换写清楚,这是理解后面 all-to-all 的基础:

```python
# 单卡 MoE 前向(top-1,便于聚焦机制),X: [T, h]
def moe_layer_single_device(X, W_g, experts):   # experts: E 个 nn.Module
    T, h = X.shape
    E = len(experts)

    logits = X @ W_g                  # [T, E]  router 打分
    probs  = softmax(logits, dim=-1)  # [T, E]
    gate, expert_id = probs.max(dim=-1)   # top-1: gate [T], expert_id [T]
    # gate 是被选中专家的门控权重;expert_id 是每个 token 去哪个专家

    Y = torch.zeros_like(X)           # [T, h] 输出
    for e in range(E):
        mask = (expert_id == e)       # 这一批里哪些 token 选了专家 e
        if mask.sum() == 0:
            continue
        x_e = X[mask]                 # 取出该专家的所有 token, [n_e, h]
        y_e = experts[e](x_e)         # 专家 FFN 计算, [n_e, h]
        Y[mask] = gate[mask].unsqueeze(-1) * y_e   # 门控加权写回
    return Y
```

**注意这段代码暴露了 MoE 的根本性"形状不规整"问题:** `mask.sum()`(每个专家分到几个 token,记作 `n_e`)是**运行时才知道、且各专家不同**的动态值。`Σₑ n_e = T`(top-1 下每个 token 恰好去一个专家),但单个 `n_e` 完全由 router 决定,可能是 0,也可能是 T(所有 token 都挤向一个专家)。**这个动态、不均匀的 `n_e`,正是后面所有麻烦(GPU 喜欢规整张量、负载不均、capacity 截断)的总根源。** 记住它。

### 2.4 复杂度与显存:解耦到底解耦了什么

把账算清楚。设单专家 FFN(`[h,4h]→[4h,h]`)的参数量 `P_ffn = 2 × h × 4h = 8h²`,稠密 FFN 处理 T 个 token 的前向算力 `≈ 2 × T × P_ffn = 16Th²`。

MoE 层(E 专家,top-k):

```
总参数量:     P_moe = E × 8h²  +  h × E (router,可忽略)  ≈ 8Eh²    → 涨 E 倍
单层前向算力:  C_moe = 2 × (kT) × 8h² = 16 k T h²                  → 仅涨 k 倍
              (因为总共只有 kT 个 "token-专家" 计算实例,k=1 时就和稠密 FFN 一样)
```

**这就是"参数 ×E、算力 ×k"的精确来源**:参数正比于专家数 E,算力正比于被激活的专家数 k(每 token)。E=64、k=1 时,参数 64 倍、算力 1 倍。

但**显存的账要单独算,而且这是 MoE 的隐藏成本**:总参数涨 E 倍,意味着**优化器状态、梯度也涨 E 倍**(第 01 章:Adam 混合精度每参数约 16 字节)。E=64 的 MoE,光参数+优化器状态就是稠密版的 64 倍——**单卡根本放不下,这从根本上逼出了专家并行**。算力省了,显存没省,反而暴涨——这个不对称是 MoE 系统设计的核心张力,下一节正面解决它。

## 三、专家并行与两次 All-to-All

### 3.1 专家放不下 → 拆到多卡

既然 E 个专家的总显存爆炸,自然的解法:**把 E 个专家分散到 `D` 张卡上,每卡放 E/D 个专家**(为简化推导,先设 E = D,即每卡恰好 1 个专家;一般情况每卡 E/D 个,结论同构)。这就是**专家并行(Expert Parallelism,EP)**。

但这立刻撞上一个数据并行的既有事实:在标准训练里,数据是按 DP 切分的——**每张卡手里拿着的是"自己那份 batch 的 token",而这些 token 经过 router 后,要去的专家可能在任意一张卡上**。于是出现一个根本性的"数据-计算错位":

```
卡 0 手里有 token {a, b, c, ...},router 算出:a→专家3, b→专家0, c→专家3, ...
卡 1 手里有 token {d, e, f, ...},router 算出:d→专家0, e→专家1, ...
...
专家 0 在卡 0,专家 1 在卡 1,专家 3 在卡 3 ...

问题:卡 0 上的 token a 要去专家3(在卡3),token b 的目标专家0就在本地。
     每张卡都既要"把自己的 token 发到别的卡",又要"接收别的卡发来的 token"。
```

**"每张卡把自己手里的数据按目标分组、发给所有其他卡,同时从所有卡接收发给自己的数据"——这正是 all-to-all 的定义。** MoE 前向因此需要两次 all-to-all,夹着专家计算:

```
[每卡本地 token]
   │  router 打分,决定每个 token 的目标专家(目标卡)
   ▼
   ① all-to-all DISPATCH:把每个 token 发往其目标专家所在的卡
   ▼
[每卡现在持有"发给本卡这个专家"的所有 token]
   │  本地专家 FFN 计算(纯本地,无通信)
   ▼
   ② all-to-all COMBINE:把计算结果发回每个 token 原来所在的卡
   ▼
[每卡收回自己 token 的输出,按门控权重合并]
```

**dispatch 和 combine 是一对互逆的 all-to-all**:dispatch 按"token→目标专家卡"重排,combine 按完全相反的映射把结果送回原位。这种"散出去算、收回来"的结构,和你在分布式 join/shuffle 里见过的 scatter-compute-gather 是同一个骨架。

### 3.2 为什么是 all-to-all,而不是别的原语

值得停一下,对照前几章的原语想清楚这个选择的必然性:

- **不能用 all-reduce**(第 02、03 章):all-reduce 是"所有卡的数据求和再广播",它假设每张卡的数据**语义相同**(都是同一份梯度/激活的一部分)。但 MoE 里每张卡的 token 是**不同的数据**,要去**不同的目的地**,没有"求和"语义。
- **不能用 broadcast/all-gather**:all-gather 是"每张卡都拿到所有卡的全部数据"。但一个 token 只需要去它选中的 1 张卡,不需要广播给所有卡——all-gather 会传输 D 倍的冗余数据。
- **必须 all-to-all**:它精确表达"点对点、每对卡之间交换各自相关的那一块"。**dispatch 的本质是一次全局数据置换(permutation by destination)**,这是 all-to-all 的核心语义。

**给系统老兵的对照:** all-to-all 在通信代价上比 all-reduce 更"重"也更"难调度"。Ring all-reduce 的通信量与卡数近似无关(第 02 章那个 `2(N-1)/N·M`),因为它能把数据切块在环上流水;**all-to-all 是 D 张卡两两之间都要建立 D-1 条数据流,总的链接数是 `O(D²)`,在大规模下对网络对分带宽(bisection bandwidth)压力极大**。这也是为什么 MoE 的扩展性强烈依赖高速互联拓扑(NVLink/NVSwitch 节点内、高质量 InfiniBand 跨节点)。

### 3.3 All-to-All 通信量的推导(本章核心量化)

现在从头推 dispatch 一次 all-to-all 的通信字节数。设:

- `D` = 专家并行度(卡数),为简化设每卡 1 个专家(E = D)。
- 每卡本地 token 数 `T_local`(全局 token `T = D × T_local`)。
- 隐藏维 `h`,激活精度 `b` 字节(FP16/BF16 则 b=2)。
- top-k 路由,每个 token 被复制 k 份发往 k 个专家(top-1 时 k=1)。

**理想均匀假设下的推导。** 先假设 router 完美均衡——每个专家恰好收到相同数量的 token。每卡有 `T_local` 个本地 token,top-k 后产生 `k × T_local` 个"token 副本"需要被发送。这些副本均匀去往 D 个专家,其中去往**本地专家**的那 1/D 不需要走网络,需要发出网络的是 `(D-1)/D` 那部分:

```
单卡发出字节(dispatch)
  = [发出的 token 副本数] × [每个 token 字节]
  = k · T_local · (D-1)/D · h · b
```

把 D-1 ≈ D(大 D 近似)简化,**单卡 dispatch 发送量 ≈ k · T_local · h · b**。直觉:每张卡基本上要把自己手里的(每 token k 份)激活几乎全部发出去(只有 1/D 留本地)。

**combine 是 dispatch 的逆**,通信量对称相等(把结果按相反映射送回):

```
单卡 combine 发送量 ≈ k · T_local · h · b
```

**单卡单 MoE 层、单次前向的总 all-to-all 发送量(dispatch + combine):**

```
C_a2a(单卡,前向) = 2 · k · T_local · (D-1)/D · h · b
                  ≈ 2 k T_local h b     (大 D)
```

如果你更习惯按"全局总通信量"记账(所有卡之和),乘以 D:

```
C_a2a(全局,前向) ≈ 2 k T h b           (因为 T = D · T_local)
```

**反向传播再来一遍**(梯度也要 dispatch/combine 一次,结构对称),所以**一个 MoE 层一次训练迭代的单卡 all-to-all 总量 ≈ 4 k T_local h b**。

**带数字感受量级。** 设 D=64、T_local=2048(每卡 2048 token)、h=4096、k=2、b=2 字节:

```
单卡前向 all-to-all 发送 ≈ 2 × 2 × 2048 × 4096 × 2 ≈ 6.7 × 10⁷ 字节 ≈ 64 MiB(约 67 MB)
```

每个 MoE 层、每次前向、每张卡约 67 MB,一个有几十个 MoE 层的模型,**每次迭代的 all-to-all 通信量是 GB 量级,且高频、在关键路径上**。这就是为什么 MoE 的吞吐对网络极度敏感,也是后面"通信-计算重叠"的动机。

**和张量并行通信量的对比(直觉锚点)。** 第 03 章 TP 通信量正比于 `Lyr · b · L · h`(正比 h¹)。MoE all-to-all 也正比于 `T · h`(正比 h¹)。**但二者性质不同:TP 是 all-reduce(对分带宽友好、可流水),MoE 是 all-to-all(`O(D²)` 链接、对分带宽压力大),且 MoE 的负载还不均匀(下一节),实际通信量可能远超理想均匀估计。**

### 3.4 Dispatch / Combine 的伪代码

把 top-1 + 专家并行的 dispatch/combine 写成可对照实现的伪代码。**核心技巧是:用 capacity 把不规整的 `n_e` 强行 pad 成规整张量,才能走规整的 all-to-all。**(capacity 下一节详解,这里先用)

```python
# 专家并行 MoE 前向(top-1),每卡 1 个专家,EP 组大小 = D
# X_local: [T_local, h]  本卡的 token;capacity C: 每个专家最多收多少 token
def moe_expert_parallel_forward(X_local, W_g, local_expert, D, C):
    T_local, h = X_local.shape

    # --- 路由:决定每个本地 token 去哪个专家(=哪张卡) ---
    logits = X_local @ W_g                       # [T_local, D]  (E=D)
    probs  = softmax(logits, dim=-1)
    gate, dst = probs.max(dim=-1)                # top-1: 目标专家 dst ∈ [0,D)

    # --- 关键:打包成规整的 dispatch 张量 [D, C, h] ---
    # 对每个目标专家 e,把去它的 token 填进 buffer[e] 的前若干槽位,超出 C 的丢弃(drop)
    dispatch_buf = torch.zeros(D, C, h)          # 规整! 每专家固定 C 个槽
    slot_counter = torch.zeros(D, dtype=int)     # 每个专家已占用的槽数
    combine_meta = []                            # 记录 (token_idx, e, slot, gate) 以便回写
    for t in range(T_local):
        e = dst[t]
        s = slot_counter[e]
        if s < C:                                # 还有空位
            dispatch_buf[e, s] = X_local[t]
            combine_meta.append((t, e, s, gate[t]))
            slot_counter[e] += 1
        # else: 这个 token 被 DROP(超出 capacity),输出置 0 或走残差

    # --- ① ALL-TO-ALL DISPATCH ---
    # dispatch_buf[e] 是"本卡发给专家 e 的 token";all_to_all 后
    # recv_buf[r] 变成"第 r 张卡发给本卡这个专家的 token"
    recv_buf = torch.empty_like(dispatch_buf)    # [D, C, h]
    dist.all_to_all_single(recv_buf, dispatch_buf)   # 沿第 0 维(卡维)交换

    # --- 本地专家计算(纯本地,无通信) ---
    # recv_buf 现在是 [D, C, h]: D 张卡各发来 C 个 token 给"本卡这一个专家"
    expert_in  = recv_buf.reshape(D * C, h)      # 拼成本专家的输入 batch
    expert_out = local_expert(expert_in)         # FFN 计算
    expert_out = expert_out.reshape(D, C, h)

    # --- ② ALL-TO-ALL COMBINE(逆向)---
    back_buf = torch.empty_like(expert_out)      # [D, C, h]
    dist.all_to_all_single(back_buf, expert_out) # 把结果发回 token 的原所在卡

    # --- 按 combine_meta 把结果乘门控权重写回原 token 位置 ---
    Y_local = torch.zeros_like(X_local)
    for (t, e, s, g) in combine_meta:
        Y_local[t] = g * back_buf[e, s]
    return Y_local
```

**这段伪代码的几个要害(对照工业实现时盯这几处):**

1. **`dispatch_buf` 是 `[D, C, h]` 的规整张量**——这是能用 `all_to_all_single`(要求各卡发送/接收块等大)的前提。代价是要预先定 C,并接受"填不满就 pad、超出就 drop"。
2. **真实实现里那两个 Python for 循环会被向量化**(用 cumsum 算槽位、用 scatter/one-hot dispatch mask 矩阵乘做置换),否则 T_local 上的 Python 循环会慢到不可用。这是 Megatron/DeepSpeed 实现的关键优化,**新手照着 for 循环写会得到一个正确但极慢的版本**。
3. **门控权重 `g` 在 combine 之后才乘**(也可在 dispatch 前乘,数学等价,但放 combine 后更省 dispatch 带宽?——实际两处带宽一样,放后侧主要是为反向求导方便,确切惯例待核)。

## 四、负载均衡:MoE 的第一难题

### 4.1 router 为什么天然不均衡:赢者通吃

把 3.3 节那个"完美均匀"的假设撕掉。**真实的 router 是学出来的,而学习动态会自发地走向极端不均衡**,机制如下:

router 用门控权重 sᵢ 加权专家输出。训练初期,某个专家 e 偶然被多送了一些 token,它就被多训练、变得更好;变好后,router 发现送 token 给 e 能降低更多损失,于是给 e 更高的分数、送更多 token;e 又被更多训练……**这是一个正反馈(rich-get-richer):热门专家越来越热,冷门专家收不到 token、得不到梯度、彻底"饿死"(dead expert)。** 最终少数几个专家处理了绝大多数 token,大部分专家形同虚设。

这在系统层是灾难:

- **计算层面**:热门专家所在的卡要算海量 token,冷门专家的卡空转。**整个 all-to-all + 专家计算的耗时由最热的那张卡决定**(木桶效应),其余卡的算力被浪费。
- **质量层面**:大部分参数(冷门专家)没被有效训练,MoE 涨参数的初衷落空——**名义上 64 个专家,实际只有几个在干活,等于白堆了参数**。

所以**负载均衡不是性能优化的"锦上添花",而是 MoE 能不能成立的"生死问题"**。三类机制协同解决它:辅助损失(从训练侧压住不均衡)、capacity factor(从系统侧设上限兜底)、token drop/pad(capacity 触发时的具体处置)。

### 4.2 辅助负载均衡损失:把"均衡"写进目标函数

既然 router 自发不均衡,就**显式地在损失里加一项惩罚不均衡**,逼 router 把 token 分散开。GShard / Switch 用的辅助损失(auxiliary load-balancing loss),对一个 batch 的 T 个 token、E 个专家,定义如下:

```
fₑ = (送到专家 e 的 token 数) / T         # 实际分配比例(硬计数,不可导)
Pₑ = (1/T) · Σ_token sₑ(token)            # router 给专家 e 的平均门控概率(可导)

L_aux = α · E · Σ_{e=1..E} fₑ · Pₑ        # 辅助损失,α 是权重系数(如 0.01)
```

**为什么这个式子能起作用?需要推导直觉。** 这是 fₑ 和 Pₑ 两个分布的内积(再乘 E 做尺度归一)。

```
- 完美均衡时:fₑ = 1/E,Pₑ = 1/E,代入 L_aux = α·E·Σ(1/E)(1/E) = α·E·E·(1/E²) = α  (最小值附近)
- 极度不均衡时:某个 e 同时有大 fₑ(收了很多 token)和大 Pₑ(被给了高概率),
  fₑ·Pₑ 这一项很大,L_aux 飙升 → 被惩罚
```

**精妙之处:fₑ 是硬计数(argmax 的结果,不可导),Pₑ 是软概率(可导)。** 损失里 fₑ 当常数系数(梯度不流过它),**梯度只通过 Pₑ 回传到 router 权重 W_g**:对那个又热(fₑ 大)又被给高概率(Pₑ 大)的专家,梯度会**压低它的门控 logits**,从而把 token 推向别的专家。**用可导的 Pₑ 当"梯度的把手",用不可导的 fₑ 当"该惩罚谁的权重"**——这是把一个组合优化(均衡分配)塞进梯度下降的经典技巧,值得记住。

**Switch Transformer 的简化版**就是上面这个式子(它 top-1,fₑ 是 token 计数比例);GShard 的形式略有差异(用了 mean gate 的变体)。两者本质相同:**让"被多送 token"和"被给高概率"同时发生的专家吃惩罚**。α 是超参,太小压不住不均衡,太大会干扰主损失、伤质量,典型 0.01 量级(待核确切默认值)。

### 4.3 Capacity Factor:从系统侧设硬上限

辅助损失是"软约束"——它**鼓励**均衡,但不**保证**均衡。任何一个 step,某专家仍可能临时收到超额 token。而 3.4 节我们已经看到,all-to-all 要求**规整张量 `[D, C, h]`**,这个 C(capacity,每个专家的容量上限)**必须是个编译期/运行前就固定的常数**,不能是运行时才知道的动态值。于是需要一个公式来定 C:

```
capacity C = capacity_factor · (T_total / E)
           = capacity_factor · (平均每专家应得的 token 数)
```

`T_total / E` 是"完美均衡下每个专家该收的 token 数"(平均负载)。**capacity_factor(容量因子,记 cf)是一个 ≥1 的冗余系数**,给不均衡留出缓冲:cf=1.0 表示"只够装平均负载,任何超额都丢";cf=1.25 表示"允许某专家比平均多收 25% 的 token";cf=2.0 则留一倍冗余。

**cf 是一个核心的"显存/通信 vs 质量"权衡旋钮:**

```
cf 越大 → C 越大 → dispatch_buf [D, C, h] 越大 → 通信量 ∝ C 线性增长、显存占用增长
         但 token 被丢的概率越低 → 模型质量越好
cf 越小 → 省通信省显存,但更多 token 被丢 → 质量下降
```

注意 3.3 节通信量公式里的 `T_local` 在用 capacity 后实际是被 `C` 取代的——**真实 all-to-all 传的是 padding 后的规整张量,其大小由 C 决定,与 cf 成正比**。所以 cf 直接是通信量和显存的乘数。工业界常用 cf ∈ [1.0, 2.0],训练和推理可设不同值(待核具体默认)。

### 4.4 Token Drop 与 Pad:capacity 触发时发生什么

C 是硬上限,真实负载又是动态的,于是每个 step、每个专家会遇到两种情况:

```
情形 A:某专家收到的 token 数 < C  →  PAD(补零)
  dispatch_buf 中该专家的剩余槽位填 0。专家会对这些 0 向量做无用计算(算力浪费),
  combine 时这些位置的输出被忽略。代价:浪费算力和通信带宽(传了 padding 的 0)。

情形 B:某专家收到的 token 数 > C  →  DROP(丢弃溢出 token)
  超过 C 的那些 token 进不了 dispatch_buf。它们这一层"没被任何专家处理"。
  通常做法:让这些被丢 token 直接走残差连接(residual),即 y = x(跳过 MoE 层),
  相当于这个 token 在这一层只保留了它自己、没获得专家加工。
```

**Drop 的影响要讲透——它不是 bug,是有意的工程取舍,但有真实代价:**

- **被丢的 token 在这一层学不到东西**(只剩残差),其表示质量受损。如果 drop 率高(比如 cf 太小或不均衡严重),大量 token 反复被丢,**模型质量明显下降**。
- **drop 主要伤"热门专家想要的 token"**:正是那些被很多 token 选中的专家会溢出,于是被丢的恰恰是"router 认为最该送给这个专家的 token"——伤的是模型最在意的那部分计算。
- **训练时通常容忍一定 drop**(配合辅助损失把整体不均衡压住,drop 率维持在低位),**推理时则倾向于用更大的 cf 或不丢**(推理对单个 token 质量更敏感,且推理 batch 的负载分布可能与训练不同)。Switch Transformer 论文报告训练在合适 cf 下 drop 率可控(待核具体数值)。

**这里触及一个值得想清的设计哲学:MoE 用"允许丢一部分 token"换"张量规整、通信可调度"。** 这和稠密模型"每个 token 都被完整计算"是根本不同的契约。**新手最容易踩的认知坑就是:默认 MoE 像稠密模型一样处理了每个 token——实际上在 capacity 截断下,一部分 token 在某些层是被跳过的。** 调试 MoE 质量问题时,**第一件事是看 drop 率和各专家负载分布**。

### 4.5 其他均衡手段(简述,够用即止)

工业实现里还有几类辅助技巧,知道它们解决什么即可:

- **Router z-loss(Switch / ST-MoE)**:惩罚 router logits 的数值过大(`L_z = (1/T) Σ (log Σₑ e^{logitₑ})²`),稳定训练、防止 logits 爆炸导致数值不稳。和负载均衡正交,是数值稳定性补丁(待核确切形式)。
- **专家容量的 token 选择反转(Expert Choice routing)**:反过来让**每个专家选它最想要的 top-C 个 token**(而非每个 token 选专家),天然保证每专家恰好 C 个 token、**完美均衡、零 drop**,代价是某些 token 可能被 0 个或多个专家选中(语义变了)。是较新的替代路由范式(待核论文细节)。
- **噪声(noisy top-k gating)**:早期 MoE(Shazeer 2017)在 router logits 上加可学习的高斯噪声,鼓励探索、防止过早收敛到少数专家。后期工作多用辅助损失替代。

## 五、通信与计算的重叠

3.3 节算出每个 MoE 层每次前向要发 GB 量级的 all-to-all,且**在关键路径上**——dispatch 不完成,专家算不了;专家算不完,combine 发不出。朴素实现下,通信和计算**串行**,GPU 在 all-to-all 期间纯等待,利用率(MFU)被拖垮。

**核心优化思想:把 all-to-all 拆块,让通信和计算流水重叠(overlap)。** 直觉:把 token 分成若干 chunk,当第 1 块的专家计算在跑时,第 2 块的 dispatch all-to-all 同时在网络上传输——用计算掩盖通信延迟。

```
朴素(串行):  [dispatch 全部]→[专家算全部]→[combine 全部]   总时间 = T_a2a + T_comp + T_a2a
重叠(流水):  dispatch块1 │ dispatch块2 │ dispatch块3
                          │ 专家算块1   │ 专家算块2   │ 专家算块3
                                        │ combine块1  │ combine块2 │ combine块3
              总时间 ≈ max(总通信, 总计算) + 一个块的启动开销   ← 二者重叠掉
```

**能重叠多少,取决于通信时间和计算时间的比值**:若计算时间 ≫ 通信时间(专家很大、token 很多),通信几乎被完全掩盖;若通信时间 ≫ 计算时间(网络慢、专家小),则受限于通信,重叠收益有限——**这又回到"MoE 必须配高速互联"的结论**。

实现上的难点:all-to-all 是集合通信,**拆块后要发起多个并发的 all-to-all、用 CUDA stream 和 NCCL 的异步接口让它们与计算 kernel 并发**,还要管理多份 buffer、处理块边界。DeepSpeed-MoE、Tutel(微软)等框架在这上面做了大量工程(分层 all-to-all、通信压缩、动态 capacity 等)。这部分与第 14 章(通信-计算重叠)的通用原理一脉相承,MoE 是它最吃重的应用场景之一。

**另一个正交优化:分层(hierarchical)all-to-all。** 大规模下(几百卡跨多节点),直接全局 all-to-all 的跨节点流量巨大。Tutel 等把它拆成"先节点内 all-to-all(走 NVLink,快)+ 再节点间 all-to-all(走 IB,慢但量被节点内聚合减小)"两级,**用拓扑感知把昂贵的跨节点流量压到最小**——和第 02 章分层 all-reduce、第 06 章拓扑贴图是同一种"拓扑感知通信"思想(待核 Tutel 具体算法)。

## 六、三大工程实现的要点对比

把本章机制对应到三个里程碑实现,帮你读源码时有坐标:

```
                  GShard (Google,2020)      Switch Transformer (Google,2021)   DeepSpeed-MoE (微软,2022)
路由 k            top-2                       top-1(论文核心简化)               可配,常 top-1/2
核心贡献          首次系统化 MoE+专家并行,    "k=1 足够好",极致简化路由,         系统侧工程化:高效 all-to-all、
                  提出辅助损失、capacity      省一半专家计算与通信               PR-MoE/专家分层、推理优化
负载均衡          aux loss + capacity         aux loss(简化) + capacity         同上 + 动态优化
对系统的意义      奠定 dispatch/combine       证明 top-1 可行 → 通信减半          把 MoE 推到可部署的工程基线
                  两次 all-to-all 范式
```

**三个一句话要点:**

- **GShard**:奠定了"专家并行 + dispatch/combine 两次 all-to-all + 辅助损失 + capacity"这套**至今仍是标准的范式**。读它就是读 MoE 系统的"宪法"。
- **Switch Transformer**:核心洞见是 **"top-1 就够"**——把 k 从 2 降到 1,专家计算和 all-to-all 通信都减半,而质量损失很小。**这是性价比的关键简化**,也是为什么很多实现默认 top-1。
- **DeepSpeed-MoE**:贡献在**系统工程**——把 all-to-all 优化、通信重叠、推理时的专家并行+张量并行混合等做到工业可部署,并提出 PR-MoE(部分层用 MoE、靠后的层专家更多)等结构优化。

**和稠密并行的组合(关键工程现实)**:MoE 的专家并行(EP)**不是孤立的**,它和 DP/TP/PP 正交组合。典型布局:**非专家部分(attention、router、LayerNorm)走 DP/TP,专家部分走 EP**;EP 度数通常等于或小于卡总数。一个 token 在一层里:attention 用 TP 算 → router 打分 → EP 的 all-to-all 把它发到专家卡 → 专家算 → all-to-all 收回。**MoE 让原本就复杂的 3D 并行又加了一维(EP),并行度搜索空间更大**——这是第 06 章并行度搜索在 MoE 时代的延伸。

## 七、设计权衡与常见坑

**权衡 1:省算力 vs 暴涨显存与通信。** MoE 的卖点是"算力恒定、参数暴涨",但**参数暴涨直接导致显存(参数+梯度+优化器状态)暴涨 E 倍**,且引入高频 all-to-all。**它不是"免费变大",而是"用显存和通信换算力"**。只在"算力是瓶颈、显存和网络还有余量"时 MoE 才划算;若本来就显存/网络受限,MoE 可能得不偿失。

**权衡 2:k=1 vs k=2。** top-1 省一半专家计算和通信、实现更简单(每 token 恰好一个目的地);top-2 表达力更强(两专家组合)、训练更稳(token 不会因单专家 drop 就完全丢失,还有第二选择),但算力通信翻倍。**没有绝对优劣,Switch 押 top-1,Mixtral 押 top-2**。

**坑 1(认知层最致命):以为 MoE 处理了每个 token。** capacity 截断下,**超额 token 在该层被 drop、只走残差**。调质量问题先看 drop 率和负载分布,别默认每个 token 都被完整计算。

**坑 2:dispatch 用 Python 循环。** 3.4 节那个 for 循环是讲解用的;真实实现必须**向量化**(one-hot dispatch mask + 矩阵乘做置换,或 cumsum 算槽位 + scatter)。照着 for 循环写会慢到不可用——这是 MoE 实现的头号性能陷阱。

**坑 3:忘了反向也有两次 all-to-all。** 和第 03 章 TP 同款坑:新手只数前向 dispatch+combine,**漏掉反向梯度也要 combine+dispatch 一遍**。一个 MoE 层一次迭代是 **4 次 all-to-all**,不是 2 次。

**坑 4:capacity_factor 设太小省资源。** cf 太小 → drop 率飙升 → 模型质量暗中崩坏,而 loss 曲线可能看起来还在降(因为大部分 token 没丢),很隐蔽。**调 cf 一定要监控 drop 率这个指标**,不能只看显存省了多少。

**坑 5:辅助损失系数 α 没调。** α 太小压不住不均衡(专家饿死、dead expert);α 太大干扰主损失伤质量。**dead expert(某专家长期收到 ≈0 token)是 MoE 训练失败的典型症状**,一旦出现要回头查 α 和初始化。

**坑 6:推理负载分布 ≠ 训练。** 训练时辅助损失把负载压均衡了,但**推理时没有辅助损失梯度,且单条/小 batch 推理的 token 分布可能极不均衡**(一句话里的 token 可能都涌向少数专家),导致推理时严重不均衡或高 drop。推理部署要单独评估负载、常用更大 cf。

**坑 7:all-to-all 对网络拓扑极敏感。** `O(D²)` 的链接结构对对分带宽要求高,跨节点慢链路会让 all-to-all 成为绝对瓶颈。**MoE 的扩展性强依赖高速互联**,在弱网络上专家并行度开太大会血亏——务必结合拓扑(节点内 vs 跨节点)规划 EP 度数。

## 八、动手练习

**练习 1(推导题,核心)。** 一个 MoE 层用专家并行,EP 度数 `D`(每卡 1 专家),每卡本地 token 数 `T_local`,隐藏维 `h`,激活 BF16(2 字节),top-k 路由,capacity factor cf。
(a) 写出 capacity C 关于 `T_local, D, cf` 的表达式(注意全局 token `T = D·T_local`,平均每专家 `T/E = T/D`)。
(b) 在用 capacity-padding 的规整 all-to-all 下,推导**单卡单次前向**(dispatch + combine)的实际发送字节数(用 C 表达,而非理想均匀的 T_local)。
(c) 推导**单卡一次完整迭代**(含反向)的总 all-to-all 字节数。
(d) 代入 D=128, T_local=4096, h=6144, k=2, cf=1.25,估算单层单卡每次迭代的 all-to-all 量(MB/GB)。

## 十、小结与承上启下

把这章浓缩成几条**必须刻进肌肉记忆**的结论:

1. **MoE = 用 router + N 个专家替换共享 FFN,每 token 只激活 top-k 个**,从而**参数 ∝ E(可涨几十倍)、单 token 算力 ∝ k(几乎不变)**——容量和算力解耦。但**显存(参数+优化器状态)随 E 暴涨**,这是隐藏代价,也是专家并行的根本动机。
2. **专家并行把 E 个专家分散到 D 张卡**,token 由 router 动态决定去向,因此需要**两次 all-to-all**:dispatch(token 发到专家所在卡)+ combine(结果收回原卡),互为逆置换。前向 2 次、反向 2 次,**一层一迭代共 4 次**。
3. **all-to-all 单卡前向发送量 ≈ 2k·T_local·h·b**(实际由 capacity C 取代 T_local,正比 cf),`O(D²)` 链接结构对网络对分带宽压力大——**MoE 强依赖高速互联**。
4. **负载均衡是 MoE 第一难题**:router 自发"赢者通吃"。三招协同——**辅助损失**(用可导 Pₑ 当梯度把手、不可导 fₑ 当惩罚权重,从源头压均衡)、**capacity factor**(系统侧设硬上限 C = cf·T/E,是显存/通信 vs 质量的旋钮)、**token drop/pad**(C 触发时溢出 token 走残差、空槽补零)。
5. **认知红线:MoE 不保证处理每个 token**——capacity 截断下溢出 token 在该层只走残差。**调 MoE 必看 drop 率和专家负载分布**。
6. **通信与计算重叠**(拆块流水掩盖 all-to-all)和**分层 all-to-all**(拓扑感知,节点内+节点间两级)是让 MoE 吞吐可用的关键工程,收益取决于"计算时间 vs 通信时间"之比。

**这章在整门课里的位置:** 第 02–06 章解决的是**稠密模型**的"放得下、训得动、并行得高效"。MoE 是一条**正交于稠密路线的"稀疏化"路径**——它不优化既有计算,而是**改变"哪些参数参与计算"的契约本身**,用稀疏激活换容量。它引入的**专家并行(EP)是 3D 并行之外的第四个并行维度**,把第 06 章的并行度搜索空间又扩了一维;它的 all-to-all 是第 02、03 章 all-reduce 之外的另一类核心通信原语,而它对**通信-计算重叠**(第 14 章)的依赖比任何稠密结构都更极致。在推理侧,MoE 的专家并行 + capacity 动态调整 + 负载分布问题,会直接接到第 10、11 章的推理优化语境里——一个 MoE 模型的推理服务,既要 KV Cache,又要在每层做 all-to-all 路由,系统复杂度叠加。**理解了 MoE,你就掌握了"参数与算力解耦"这一当代大模型最重要的 scaling 杠杆之一,以及它在系统层索取的全部代价。**
