> 这一章把你从"知道 BPE 怎么合并 token"带到"能为一个多语言模型选对分词方案、读懂 mBERT/XLM-R 为什么能零样本跨语言迁移、解释清楚 curse of multilinguality 这条工程曲线从哪来、并能在低资源/无空格/形态丰富语言上做出有根据的取舍"。子词算法本身的训练细节(BPE/WordPiece/Unigram 的合并与剪枝推导)在算法课 01 已经讲透,本章不重推;本章的重心是**子词作为多语言系统的接口层,它的设计如何决定一个模型能覆盖多少语言、迁移得多好、在哪类语言上必然吃亏**。

## 一、动机:分词器是多语言系统里最被低估的"政治决策"

先把上一层(算法课 01)的结论收住,作为本章的地基:

- **字符/字节级**序列太长(注意力是 `O(n²)`,详见算法课 02),**词级**词表爆炸且有 OOV(未登录词,out-of-vocabulary)。**子词(subword)是被 Zipf 分布逼出来的折中**:高频词整体保留保证序列短,长尾拆成可复用片段保证有限词表表示无限词。
- 三种主流算法:**BPE** 数相邻对的绝对频次自底向上贪心合并;**WordPiece** 按"合并后似然增益"(近似互信息)合并;**Unigram LM** 先放超大候选词表,按"似然贡献"自顶向下剪枝。三者训练判据不同,实践效果都不错。

这些在单语言(尤其英语)场景下是个相对干净的技术选择题。但一旦进入多语言,分词器从"技术选择"升级成**资源分配决策**:

**一个固定大小的词表(比如 25 万)要被几十上百种语言瓜分。给英语多分一些 token,中文、斯瓦希里语就少一些。** token 不是中立的——它是建模预算。一个语言在词表里被分到的有效 token 越多,它的文本被切得越短、语义单元越完整、模型在它上面学得越好。所以"词表怎么训"在多语言里直接决定了**哪种语言是一等公民,哪种是被字节级回退兜底的二等公民**。

这一章要回答四个层层递进的问题:

1. **接口层**:为什么多语言必须用 SentencePiece 这种"语言无关、直接在原始文本上训、把空格也当符号"的方案?字节级 BPE 又解决了什么 SentencePiece 解决不了的问题?
2. **迁移现象**:mBERT/XLM-R 只共享一个子词词表 + 在多语言语料上联合预训练,为什么会**涌现**出"在英语上微调,直接在斯瓦希里语上工作"的能力?机制是什么?
3. **代价曲线**:为什么语言加得越多,每种语言反而越差(curse of multilinguality)?这条曲线背后是什么在拉扯?
4. **评测**:跨语言迁移好不好,怎么量化?对齐(alignment)到底在对齐什么?

## 二、SentencePiece:让分词器不再依赖"语言长什么样"

### 问题:经典子词流水线偷偷假设了"语言像英语"

回顾算法课 01 的 BPE 训练:输入是一个**词频字典**(`"low": 5, "lower": 2, ...`),每个词拆成字符,词尾加 `</w>`。这个流程里藏着两个对英语成立、对很多语言不成立的隐含假设:

**假设一:文本能先被"预分词(pre-tokenization)"成词。** 经典流水线第一步是按空格(和标点)把句子切成词,再对每个词跑 BPE。英语、法语、德语靠空格分词,这一步天经地义。但:

- **中文、日文、泰文没有词间空格**。"我爱自然语言处理"是连续的字符流,"词"的边界本身就是个需要模型(分词器)去学的难题。你没法先验地按空格切。
- 强行用一个语言相关的预分词器(比如中文用 jieba、日文用 MeCab),会把**语言知识硬编码进流水线**,失去通用性,而且引入一个外部依赖——换语言就得换分词器,且这个分词器的错误会传播下去。

**假设二:空格是"分隔符",可以丢掉。** 经典 BPE 把空格当作切词的依据然后扔掉,解码时再按规则(每个词前加空格)拼回去。但这套"加空格规则"本身是语言相关的:英语词间有空格,中文词间没有,法语标点前后空格规则又不同。**丢掉空格 = 把"怎么拼回去"这个语言相关的问题塞给了后处理**,而后处理一旦和训练不一致,解码就出错(算法课 01 提到的"训练推理分词不一致"坑,在多语言里被放大)。

### 机制:把原始文本当字节流,把空格当一个普通符号

**SentencePiece(Kudo & Richardson, EMNLP 2018 demo)的核心设计:不做语言相关的预分词,直接在原始未切分的文本(raw text)上训练和编码;把所有字符——包括空格——一视同仁地当作建模符号。** 它在内部可以用 BPE 或 Unigram LM 作为子词算法(LLaMA、T5 用的是 SentencePiece 包 Unigram;很多模型用 SentencePiece 包 BPE),SentencePiece 本身是**外层的"语言无关化"框架**,不是第四种子词算法。

它做对的三件事:

**1. 空格显式化为元符号 `▁`(U+2581,LOWER ONE EIGHTH BLOCK)。** 训练前,先把每个空格替换成一个可见的特殊字符 `▁`("下划块",通常读作 underscore-block 或直接叫 SentencePiece 的 space marker)。于是:

```
原始文本:  "Hello world"
转义后:    "▁Hello▁world"        # 每个空格(含某些实现里句首的那个)变成 ▁
子词切分:  ["▁Hello", "▁world"]  # ▁ 黏在词上,成为 token 的一部分
```

**关键收益:解码变成纯字符串替换,完全无歧义、语言无关。** 把所有 token 拼接,再把 `▁` 换回空格,就精确还原原文。没有"英语加空格、中文不加"这种规则——`▁` 自己记着哪里有空格。这叫 **lossless / reversible tokenization(可逆分词)**:`decode(encode(x)) == x` 严格成立。对无空格语言(中文),文本里本来就没有 `▁`(除非有空格),切出来的就是连续字符的子词,天然不需要"假装有词边界"。

> 直觉:`▁` 把"空格在哪"这个信息从流水线规则里搬进了词表本身。规则会和语言绑定,词表里的符号不会。

**2. 直接在 raw text 上跑子词算法,不做 word-level 预分词。** 这样"词边界"成了模型自己从数据里学出来的统计模式,而不是外部分词器强加的。对中文,它会把高频的字组合(如"自然""语言")学成子词;对英语,它会把 `▁the`、`▁of` 这种"空格+高频词"学成子词。同一套机制对所有语言生效。

**3. 字符覆盖率(character coverage)旋钮。** 多语言语料里有海量稀有字符(生僻汉字、各种脚本的边角码点)。如果要求 100% 覆盖,每个稀有字符都得占词表一格,浪费预算。SentencePiece 提供 `character_coverage` 参数(英文等小字符集设 1.0,中日韩等大字符集常设 0.9995),意思是"用单字符覆盖语料里出现频率累计达到这个比例的字符,剩下的极稀有字符走字节回退(byte fallback)"。这就把"词表预算"和"字符长尾"解耦了。

### 代码:训练并使用一个多语言 SentencePiece 模型

```python
import sentencepiece as spm

# ---- 训练 ----
# 输入是一个纯文本文件,每行一句,可以是多种语言混合。
# 不需要任何预分词、不需要分词工具、不需要语言标注。
spm.SentencePieceTrainer.train(
    input="multilingual_corpus.txt",
    model_prefix="mlm_spm",
    vocab_size=32000,
    model_type="unigram",        # 也可 "bpe" / "char" / "word"
    character_coverage=0.9995,   # 含中日韩等大字符集时用 <1.0,纯拉丁系可用 1.0
    byte_fallback=True,          # 关键:词表覆盖不到的字符退化成 UTF-8 字节 token,杜绝 OOV
    unk_id=0, bos_id=1, eos_id=2, pad_id=3,
    # 把空格转义成 ▁ 是默认行为(add_dummy_prefix=True 会在句首也加一个)
)

sp = spm.SentencePieceProcessor(model_file="mlm_spm.model")

# ---- 编码 / 解码:对多种语言用同一个模型 ----
for text in ["Hello world", "我爱自然语言处理", "Bonjour le monde"]:
    ids    = sp.encode(text, out_type=int)
    pieces = sp.encode(text, out_type=str)   # 看人话:子词字符串
    back   = sp.decode(ids)
    print(pieces)
    assert back == text, "可逆性应严格成立(byte_fallback 下含极罕见字符也成立)"

# 典型输出(示意,实际取决于语料):
#   ['▁Hello', '▁world']
#   ['▁', '我', '爱', '自然', '语言', '处理']     # 中文:▁是句首dummy,内部无空格
#   ['▁Bon', 'jour', '▁le', '▁monde']
```

注意三个能直接看出机制的点:(1)英语 token 带 `▁` 前缀,中文内部 token 不带——空格信息完全编码在符号里;(2)`byte_fallback=True` 时,任何没进词表的字符(某个生僻 emoji、某个罕见梵文字符)会被拆成它的 UTF-8 字节序列对应的字节 token(SentencePiece 内部记为 `<0xE4>` 这种),**严格零 OOV**;(3)整个过程没有任何 `if language == "zh"` 的分支。

### 分析:SentencePiece 到底解决了什么、没解决什么

**解决的**:语言无关的、可逆的、零 OOV 的、对无空格语言友好的分词接口。这是几乎所有现代多语言模型(mBART、mT5、XLM-R、LLaMA)采用它(或它的等价物)的原因。

**没解决的(本章后面要展开)**:它让"对所有语言用一套机制"成为可能,但**没有让所有语言被公平对待**。词表预算仍然有限,谁在训练语料里占比高,谁就拿走更多 token。SentencePiece 是"统一接口",不是"公平分配"。

## 三、字节级 BPE:用 256 个字节把 OOV 这个概念彻底删掉

### 问题:SentencePiece 的 byte_fallback 是补丁,能不能从根上无 OOV?

SentencePiece 靠 `byte_fallback` 给覆盖不到的字符兜底,但词表主体仍是"字符 + 子词"。GPT-2(Radford et al., 2019)走了另一条更激进的路:**词表的基本单元不是字符,而是字节(byte)。**

为什么字节是个诱人的基本单元?因为 **UTF-8 编码下,任何 Unicode 文本——任何语言、任何 emoji、任何符号——都是一串 0~255 的字节**。如果词表的"原子"是这 256 个字节,那么**任何字符串都能被无损表示,OOV 在定义上不可能发生**。不是"很少发生",是"不可能发生"。

### 机制:在字节上跑 BPE,但要解决两个工程问题

**朴素想法**:把文本看成字节序列,在字节上跑 BPE(初始词表 = 256 个字节,合并高频字节对)。这正是 byte-level BPE 的骨架。但有两个坑:

**坑一:某些字节是控制字符 / 空白,不可打印,会破坏词表文件、日志、可视化。** 比如换行、制表符、空格(0x20)对应的字节直接写进词表文件会乱套。

**GPT-2 的解法:一个 256→可见 Unicode 字符的双射(reversible byte-to-unicode mapping)。** 把可打印 ASCII 范围(以及拉丁补充里可打印的那段)的字节直接映射到它自己;把那些不可打印的字节(控制字符、空格等)一一映射到一段没人用的、可打印的 Unicode 码点上(GPT-2 用了从 U+0100 开始的区段)。这样:

```
字节 0x20 (空格)      -> Ġ  (U+0120, 一个可见字符)
字节 0x0A (换行)      -> Ċ  (U+010A)
字节 0x41 ('A')       -> A  (可打印 ASCII,映射到自己)
```

于是词表里出现的 `Ġworld` 其实代表"空格+world"(对比 SentencePiece 的 `▁world`,异曲同工——都是把空格变成可见符号)。这个映射是**双射且可逆**,编码时字节→可见字符,解码时可见字符→字节→UTF-8 解码回文本。

**坑二:不加约束的话,BPE 可能把"一个字符的几个字节"和"标点/空格的字节"跨边界合并,产生语言学上很怪的 token。** GPT-2 在字节级 BPE 前加了一条基于正则的预分词(用 GPT-2 那个著名的正则,把缩写 `'s`、连续字母、连续数字、连续空格等大致切开),约束合并不要跨过这些边界乱并。这是个工程经验性的护栏,不是算法必需。

### 代码:字节↔可见字符双射(GPT-2 的核心 30 行)

```python
def bytes_to_unicode():
    """构造 256 字节 <-> 可见 Unicode 字符的双射(GPT-2 原版逻辑)。
    思路:可打印的字节映射到自己;不可打印的字节顺次映射到 U+0100 起的可见区段。"""
    # 三段本来就可打印的字节,直接映射到自己对应的 Unicode 码点
    bs = (list(range(ord("!"), ord("~") + 1)) +
          list(range(ord("¡"), ord("¬") + 1)) +
          list(range(ord("®"), ord("ÿ") + 1)))
    cs = bs[:]
    n = 0
    for b in range(256):           # 把剩下的(不可打印)字节补进来
        if b not in bs:
            bs.append(b)
            cs.append(256 + n)     # 映射到 U+0100, U+0101, ... 这段没人占用的可见区
            n += 1
    return {b: chr(c) for b, c in zip(bs, cs)}

byte_encoder = bytes_to_unicode()          # 0..255 -> 可见字符
byte_decoder = {v: k for k, v in byte_encoder.items()}

def to_visible(text: str) -> str:
    # 1) 文本 -> UTF-8 字节;2) 每个字节 -> 可见字符。之后 BPE 在这个可见字符串上跑。
    return "".join(byte_encoder[b] for b in text.encode("utf-8"))

def from_visible(s: str) -> str:
    # 逆过程:可见字符 -> 字节 -> UTF-8 解码。严格还原。
    data = bytes(byte_decoder[ch] for ch in s)
    return data.decode("utf-8")

x = "café 世界 🌍"
v = to_visible(x)
assert from_visible(v) == x        # 任意 Unicode,严格可逆,永不 OOV
print(v)   # 'cafÃ©Ġä¸ĸçķĮĠðŁĮį' —— 一串可见字符,但每个对应原文的一个字节
```

`café`、`世界`、`🌍` 全都先变成 UTF-8 字节,再变成可见字符,BPE 在这串可见字符上合并。任何输入都走得通,**不存在"这个字符不在词表里"这回事**。

### 分析:字节级 BPE 的代价——多语言公平性的隐藏税

字节级"零 OOV"是真香,但对多语言有个**必须算清楚的代价**:

**一个非拉丁字符,在 UTF-8 里要 2~4 个字节;如果分词器主要在英文语料上训练、没给这些字节足够的合并规则,那么一个汉字可能被切成 2~3 个 token,一个 emoji 可能 4 个 token。** 让我们量化这件事(数量级估算,具体随分词器而变):

```
信息等价的一句话,在一个英文为主训练的字节级 BPE 下:
  英语:  "I love natural language processing"  ≈ 6 个 token(词基本各占1个)
  中文:  "我爱自然语言处理"                      ≈ 8~16 个 token
         (8 个汉字 × UTF-8 3 字节 = 24 字节;若几乎无中文合并规则,
          每汉字 2~3 个 token → 16~24;若有部分合并规则 → 8~12)
```

**后果链条**(对所有按 token 计费/计长的系统都成立):

1. **同样信息,中文消耗更多 token** → 在固定上下文窗口(比如 4096 token)里,中文能塞的真实内容更少 → 等效上下文更短。
2. **按 token 计费的 API,中文用户每个字付的钱更多**(这是真实存在的、被批评的"语言税")。
3. **更长的 token 序列 = 更高的注意力开销**(`O(n²)`),低资源语言的推理更慢。
4. **更碎的切分 = 语义单元更不完整**,模型更难学好——一个被切成 3 个无意义字节片段的字,远不如一个整 token 那样携带清晰语义。

> 这条税的根源不是字节级 BPE 这个机制错了,而是**训练语料的语言分布**。字节级保证了"能处理",但"处理得好不好"取决于词表里给这门语言留了多少合并规则。这就把我们引向第四节:多语言模型如何训词表、为什么会"诅咒"。

**SentencePiece vs 字节级 BPE 的选择(实践经验,非绝对)**:

```
                 SentencePiece(常配Unigram)      字节级 BPE(GPT-2系)
基本单元          字符(+byte_fallback兜底)        字节
零 OOV            靠 byte_fallback 实现            天生(定义上不可能OOV)
空格处理          ▁ 元符号                          字节(空格0x20→Ġ),本质相同
对无空格语言      原生友好(直接raw text)          也能工作,但更易把字拆碎
词表可读性        高(token多是真实字符/词)        低(可能是字节片段)
代表              mBART, mT5, XLM-R, LLaMA(spm)    GPT-2/GPT-3/GPT-4, RoBERTa(byte BPE)
多语言公平性      取决于训练语料分布(下一节)      同样取决于训练语料分布
```

两者本质都把空格变成可见符号、都能做到零 OOV、公平性都取决于语料。**真正的差异是默认粒度**:SentencePiece 默认字符级原子,对脚本多样的语言更不容易把字符拆碎;字节级原子在脏数据/任意二进制上更鲁棒。现代很多大模型用的是**字节级 fallback 的 BPE/Unigram**——两者优点的合流。

## 四、多语言模型:为什么共享一个词表就能跨语言迁移

这是本章的理论核心。现象先摆出来,机制再拆。

### 现象:零样本跨语言迁移(zero-shot cross-lingual transfer)

**mBERT(multilingual BERT,Devlin et al. 2018 随 BERT 一起发布)**:就是 BERT 的结构,但在 **104 种语言的 Wikipedia** 上用同一个 WordPiece 词表(约 11 万)联合做掩码语言建模(MLM,详见 NLP 课 06)。没有任何平行语料、没有翻译信号,纯粹把多语言文本倒进同一个模型一起预训练。

**XLM-R(XLM-RoBERTa,Conneau et al. 2020)**:思路相同但更大力——**100 种语言**,语料从 Wikipedia 换成规模大得多的 **CommonCrawl**(过滤后约 2.5TB),用 SentencePiece + Unigram 训了约 25 万的共享词表,用 RoBERTa 式的 MLM(只掩码、去掉 NSP,大 batch、更多步)预训练。

惊人的现象是:**在英语的标注数据上微调一个下游任务(比如 XNLI 自然语言推断、NER 命名实体识别),然后直接拿去跑法语、中文、斯瓦希里语的测试集——不给这些语言一条标注数据——它居然能工作,而且远好于随机。** 这叫**零样本跨语言迁移**:任务知识在英语上学,跨越语言边界迁移到了没标注的语言上。

为什么会这样?这不是被设计出来的,是**涌现**出来的。机制有两层。

### 机制一:共享子词词表制造了"锚点"

如果英语和德语用各自独立的词表,那么"world"的 ID 和"Welt"的 ID 之间没有任何关系,模型从一开始就没法知道它们相关。**但共享词表会让不同语言天然共享一批 token**:

- **完全相同的子串**:数字 `2024`、标点、URL、代码、专有名词(`Obama`、`Paris`、`COVID`)、借词,在很多语言里写法一模一样,共享同一个 token。
- **同源词的共享词根**:`nation`(英)/ `nation`(法)/ `Nation`(德)共享 `▁nation`;拉丁语系语言大量共享词根片段。
- **跨脚本不共享,但有桥梁**:中文和英文几乎不共享 token,但都和"数字、标点、英文专名"这第三方共享——这些成了**跨脚本的锚点(anchor)**。

**这些共享 token 就是"绳子的结"。** 预训练时,模型对每个 token 学一个嵌入向量;一个共享 token(比如 `Paris`)在英语句子和法语句子里都出现,模型被迫给它学一个**同时要在两种语言上下文里都说得通**的向量。一旦有足够多这样的共享锚点,不同语言的嵌入空间就被这些公共点"缝合"在一起,无法各自独立漂移。

> 直觉:想象两块布(英语空间、法语空间),共享 token 是把它们钉在一起的钉子。钉子越多、越分散,两块布就被对齐得越好。

但光有共享 token 还不够——中文和英文几乎没有直接共享的 token,迁移却依然发生。所以还有第二层,更深。

### 机制二:联合 MLM 把表示空间对齐到"语言无关的语义"

这是更本质的一层。**掩码语言建模(MLM)逼模型根据上下文预测被掩盖的词。** 当模型在英语、法语、中文的同一套参数上同时做这件事,它面临一个压力:

**用一套共享的 Transformer 参数,同时解释所有语言的上下文规律。** 参数预算有限,模型没法为每种语言学一套完全独立的"世界模型";最经济的策略是**抽取出跨语言通用的、抽象的语义和句法结构**,把语言特定的部分(具体哪个词形、哪种语序)留给浅层处理。结果是:**中层/高层的隐藏表示,逐渐编码的是"意思",而不是"哪种语言的哪个词"**。

可以这样论证为什么这是最优解。设模型要最小化所有语言的 MLM 损失之和:

```
总损失 ≈ Σ_语言 ℓ E_{x ~ 语言ℓ的语料} [ -log P(被掩token | 上下文; θ) ]
```

θ 是所有语言共享的参数。若模型为每种语言维护互不相干的内部表示,等于用一份参数硬扛 N 套独立任务,参数被稀释;而**当不同语言的"相同意思的句子"被映射到隐藏空间里相近的区域时,一份参数就能复用**——在英语上学到的"否定词翻转句意""主谓一致"这类抽象规律,自动适用于德语、中文的对应结构,因为它们在表示空间里落在同一片区域。**对齐的表示空间是参数共享压力下的最优(最省参数)解**,所以它被优化过程"挤"出来。

把两层合起来,零样本迁移的完整机制是:

```
共享子词词表(机制一)
    └─> 提供跨语言锚点,缝住嵌入层,给"空间对齐"一个起点
联合 MLM + 共享参数(机制二)
    └─> 参数预算压力逼出"语言无关的抽象语义表示"
         └─> 下游任务头在英语表示上学到的决策边界
              └─> 因为法语/中文的语义表示落在同一空间的同一区域
                   └─> 决策边界对它们同样有效 = 零样本迁移
```

**一句话:迁移 = 共享词表把入口对齐 + 联合预训练把中间表示对齐,于是在一种语言上学的任务头能直接用在别的语言上。**

### 机制三(可选的强化):XLM 的 TLM 显式利用平行语料

mBERT/XLM-R 完全没用平行语料(对齐的翻译句对),迁移是自发涌现的。**XLM(Lample & Conneau, 2019)** 提出:如果手上**有**平行语料,可以用一个目标显式地推一把对齐——**TLM(Translation Language Modeling,翻译语言建模)**。

机制极简单也极漂亮:**把一对互为翻译的句子(英语句 + 它的法语译文)拼接成一个序列,一起做掩码语言建模。** 关键在于——**被掩盖的词,模型既能看本句的上下文,也能看另一种语言译文的上下文**:

```
普通 MLM(单语):
  "The cat sat on the [MASK]."         # 只能靠英语上下文猜 "mat"

TLM(拼接平行句,跨语言掩码):
  [EN] The cat [MASK] on the mat . [/s] [FR] Le chat était assis sur le [MASK] .
       ↑掩了 "sat"                              ↑掩了 "tapis"(=mat)
  预测 [MASK]=sat 时,模型可以"偷看"法语的 "était assis"(=sat)拿线索
  预测 [MASK]=tapis 时,可以偷看英语的 "mat"
```

**这就把"翻译对齐"直接灌进了注意力**:模型学会跨语言地从一种语言的词去关注另一种语言的对应词,等于在训练里手把手地把 `sat ↔ était assis`、`mat ↔ tapis` 对齐起来。XLM 的完整训练通常是 **MLM(单语,所有语言)+ TLM(平行语料,语言对)交替**。

**取舍**:TLM 迁移效果更强(尤其在有平行语料的高资源语言对上),但**依赖平行语料**——而平行语料对绝大多数语言对是稀缺的(英-斯瓦希里平行句远少于英-法)。这正是 XLM-R 后来**放弃 TLM、纯靠海量单语 CommonCrawl + MLM** 反而在低资源语言上更强的原因:**数据规模 > 巧妙目标**,当你能拿到 2.5TB 单语文本时,单语 MLM 的涌现对齐已经够好,而平行语料的稀缺会成为瓶颈。这是个非常值得记住的工程判断:**有平行语料且语言对高资源 → TLM 有用;追求覆盖几十上百种长尾语言 → 单语规模化更实在。**

### 代码:零样本跨语言迁移的最小骨架

```python
import torch
from transformers import AutoTokenizer, AutoModelForSequenceClassification

# XLM-R:一个共享 SentencePiece 词表(~25万)+ 100语言联合预训练的 encoder
NAME = "xlm-roberta-base"
tok = AutoTokenizer.from_pretrained(NAME)
model = AutoModelForSequenceClassification.from_pretrained(NAME, num_labels=3)  # 例:NLI

# 看一眼共享词表的"缝合"效应:跨语言的同义/同源词
for w in ["▁Paris", "▁nation", "2024", "▁世界", "▁world"]:
    print(w, "->", tok.convert_tokens_to_ids(w))   # 都映射到同一张共享词表的ID

# --- 训练:只用英语标注数据微调任务头(此处省略训练循环)---
# train_on(english_labeled_data, model, tok)

# --- 推理:直接喂法语 / 中文,不给任何该语言的标注 ---
def predict(text):
    enc = tok(text, return_tensors="pt", truncation=True)
    with torch.no_grad():
        logits = model(**enc).logits
    return logits.argmax(-1).item()

# 同一个在英语上微调的模型,对法语、中文输入直接给出预测 = 零样本跨语言迁移
# predict("This movie is great")     # 英(训练语言)
# predict("Ce film est génial")      # 法(零样本)
# predict("这部电影很棒")             # 中(零样本,跨脚本,靠机制二)
```

要点:`▁Paris`/`2024` 在词表里是跨语言共享 ID(机制一的锚点);中文能零样本工作主要靠机制二(表示空间对齐),因为它和英语几乎无共享 token。**迁移质量从高到低大致是:同语系高资源(法、德)> 异语系高资源(中、阿)> 低资源语言**,这和它们在预训练语料里的占比、与英语的语言学距离强相关。

## 五、Curse of Multilinguality:为什么语言越多,每种越差

### 现象与命名

XLM-R 论文里量化了一条让人不舒服的曲线,被称为 **curse of multilinguality(多语言诅咒)**:

**在模型容量(参数量)固定的前提下,随着预训练覆盖的语言数从少增多,平均每种语言的性能先升后降。** 一开始加语言有好处(更多锚点、更多迁移),但过了某个临界点,**再加语言会拖累已有语言的表现**——蛋糕大小固定,分的人越多,每人越少。

### 机制:三个固定预算被稀释

把"诅咒"拆成三个被同时稀释的固定预算,就不神秘了:

**预算一:词表预算被稀释。** 词表大小固定(比如 25 万)。语言越多,每种语言能分到的有效 token 越少,文本被切得越碎(回到第三节那条税),语义单元越不完整,学得越差。可以粗略写成:

```
每种语言的有效词表份额 ≈ V_total × (该语言语料占比的某种函数)
语言数↑、且分布越不均  →  长尾语言份额↓  →  切分越碎  →  表示越差
```

**预算二:模型容量(参数)被稀释。** 参数 θ 固定。第四节说过,联合训练逼模型抽取跨语言通用表示;但语言之间的差异(语序、形态、脚本)是真实存在的、不可能完全压成同一套规律。语言越多,模型既要"求同"(学通用语义)又要"存异"(记每种语言的特殊性),固定容量被越来越多的"存异"需求挤占,**留给每种语言的有效容量下降**。

**预算三:训练步数 / 数据被高资源语言主导(language imbalance)。** 这是最直接的。CommonCrawl 里英语占了极大比例,斯瓦希里语可能只有英语的万分之一。如果按自然频率采样,**模型几乎只在看英语,低资源语言被淹没**——梯度更新绝大多数来自高资源语言,低资源语言学不动。

### 缓解机制:温度采样(temperature sampling)对抗不平衡

针对预算三,XLM/XLM-R 用了一个标准技巧——**用温度 α 重新加权各语言的采样概率,把分布"压平"**。设语言 ℓ 的自然占比是 `p_ℓ`(它的语料量占总量的比例),采样时不按 `p_ℓ` 采,而按:

```
q_ℓ  =  p_ℓ^α  /  Σ_k p_k^α        # 0 < α < 1,XLM-R 用 α ≈ 0.3(待核具体值)

α = 1   : 按自然频率采样 → 高资源语言完全主导
α → 0   : 趋向各语言等概率 → 极度偏向低资源,但高资源语言被"饿着"
α ≈ 0.3 : 折中,显著抬高低资源语言的采样概率,又不至于饿死高资源语言
```

举个数算清楚直觉(两种语言,英语占 99%、斯瓦希里 1%):

```
自然(α=1):   英 0.99,  斯 0.01            → 斯瓦希里几乎被忽略
α = 0.3:
   英^0.3 ≈ 0.99^0.3 ≈ 0.99700
   斯^0.3 ≈ 0.01^0.3 ≈ 0.25119
   归一化: 英 ≈ 0.997/(0.997+0.251) ≈ 0.799
           斯 ≈ 0.251/1.248        ≈ 0.201   → 斯瓦希里从 1% 抬到 ~20%!
```

**温度采样把低资源语言的"出镜率"提高了一个量级,是对抗 language imbalance 的核心旋钮。** 但它**只缓解预算三,缓解不了预算一和二**——词表和参数还是固定的。所以诅咒不能被消除,只能被推后。

### 怎么真正突破诅咒:加容量

既然根因是"固定预算",**最直接的解法就是把预算做大**:

- **加参数**:XLM-R Large 比 Base 大,在同样语言数下每种语言更好——容量上去了,稀释被对冲。这也是为什么"诅咒"在小模型上明显、在大模型上缓解。
- **加词表**:后续多语言模型(如 XLM-V)直接把词表做到 100 万级,给每种语言更多 token 份额,缓解预算一。
- **模块化 / 语言特定参数**:给每种语言加一点专属参数(adapter、LoRA、语言特定的 FFN,详见算法课 11 PEFT;或 MoE 让不同 token 路由到不同专家,详见算法课 12),让"存异"不再挤占"求同"的容量——这是当前对抗诅咒的活跃方向。

> **关键结论:curse of multilinguality 不是算法 bug,是"固定预算 / 多方瓜分"的资源经济学必然。缓解它的本质只有两条——把预算做大(参数/词表/MoE),或把预算分得更聪明(温度采样/语言特定模块)。**

## 六、低资源、形态丰富、脚本差异:分词层的硬骨头

多语言不只是"语言多",还在于语言之间**结构差异巨大**,而这些差异在分词层就开始咬人。

### 难点一:形态丰富语言(morphologically rich languages),子词切分会爆炸

英语形态简单(`run/runs/running/ran`,变体少)。但很多语言形态极丰富:

- **黏着语(agglutinative)**,如土耳其语、芬兰语、斯瓦希里语:一个"词"由词根 + 一长串后缀粘接而成,一个词能表达英语一整句的意思。土耳其语有个著名的词 `evlerinizden`(≈"from your houses",从你们的房子里)是 `ev`(house)+`ler`(复数)+`iniz`(你们的)+`den`(从…里)粘起来的。
- **后果**:这种语言里"词形"的组合是**爆炸式**的,词级词表彻底没戏;子词切分理论上很合适(应该切成 `ev`/`ler`/`iniz`/`den` 这些有意义的语素)。但**BPE 是按频次而非语素边界切的**,它常常切出语言学上无意义的片段(比如把 `lerin` 切成 `le`+`rin`),既不对齐真实语素,又因为组合多导致每个组合频次低、切得很碎。
- **工程影响**:形态丰富语言在固定词表下被切得更碎 → token 更多 → 第三节那条税 + curse 的预算一,双重受害。这也是为什么这类语言在多语言模型里普遍偏弱。

### 难点二:脚本差异(script diversity)与字节税

世界上的文字系统差异巨大:拉丁、西里尔、阿拉伯、天城文(印地语)、汉字、假名、谚文、泰文……

- **UTF-8 字节数不同**:拉丁字母 1 字节,西里尔/阿拉伯多为 2 字节,汉字/天城文 3 字节,部分 emoji 4 字节。**字节级分词对非拉丁脚本天然不利**(第三节算过)。
- **有的脚本不分词且有复杂连写**:阿拉伯语从右到左、字母随位置变形;泰文不用空格分词且有复杂的元音附标。这些都让"什么是一个 token"更难。
- **同一语言多脚本**:塞尔维亚语可同时用西里尔和拉丁书写,哈萨克语在换字母表。模型若只见过一种脚本,换脚本就崩。

### 难点三:低资源 / 濒危语言,根上是数据不存在

世界有数千种语言,**绝大多数在网上几乎没有数字文本**。这不是算法能解决的——**没有数据,再好的分词器和模型都学不出来**。能做的工程缓解:

- **依附迁移**:靠和高资源语言共享脚本/词根/借词,搭机制一机制二的便车做零样本迁移(对同语系语言有效,对孤立语言效果有限)。
- **温度采样**:把仅有的一点数据的采样率抬高(第五节)。
- **数据增强 / 回译(back-translation)**:用翻译模型把高资源语言的数据翻成低资源语言来扩充(详见 NLP 课 05 翻译);质量受限于翻译模型本身。
- **字符/字节级 + 跨语言迁移**:对极低资源语言,有时字符级反而更鲁棒(不依赖学好的子词词表)。

> 诚实地说:**低资源语言的核心瓶颈是数据公平,不是建模技巧**。分词和迁移能放大已有的一点数据,但变不出无中生有的语言知识。这是 NLP 一个真实的、尚未解决的公平问题。

## 七、跨语言对齐评测:迁移好不好,到底怎么量

光说"能迁移"不够,要能量化。评测分两层。

### 层一:任务级(extrinsic)——下游任务的零样本/少样本表现

最直接:在英语上微调,在多语言测试集上看分数。标准基准(benchmark):

- **XNLI**:自然语言推断(蕴含/中立/矛盾),把英语 NLI 测试集翻译成 15 种语言。是测零样本跨语言迁移的经典指标。
- **XTREME / XTREME-R**(Hu et al. 2020):一个综合多语言基准的合集,覆盖分类、序列标注(NER、POS,详见 NLP 课 04)、问答、检索等多任务多语言,是衡量多语言模型的"主考"。
- **TyDi QA**:覆盖类型学上多样的语言的问答,刻意挑语序/形态差异大的语言。

看这些基准时,有一个关键指标叫 **cross-lingual transfer gap(跨语言迁移差距)**:`(英语上的分数) − (目标语言零样本分数)`。差距越小,说明迁移越彻底。**这个 gap 在低资源、远语系语言上系统性地更大**——是 curse 和数据不平衡在评测上的直接显影。

### 层二:表示级(intrinsic)——嵌入空间到底对齐了没

任务级分数受任务头、微调数据影响,想直接看"表示空间对齐程度",用更内在的探针:

- **双语词典归纳 / 跨语言词检索(BLI, Bilingual Lexicon Induction)**:拿一批已知的翻译词对(`world↔monde`),看在模型的表示空间里,`world` 的最近邻是不是 `monde`。命中率(precision@1/@5)直接度量嵌入对齐质量。
- **跨语言句检索(如 Tatoeba 任务)**:给一句英语,在一堆法语句子里检索它的译文(用句向量的余弦相似度,详见 NLP 课 01 表示)。准确率高 = 句级语义对齐好。
- **表示相似性分析**:用 CKA、SVCCA 等工具比较不同语言同义句的隐藏表示有多接近;或可视化(看不同语言的句子在 t-SNE 里是不是按"意思"而非"语言"聚类)。

```python
# 跨语言句检索:用一个多语言句向量模型直接验证机制二(表示空间对齐)
from sentence_transformers import SentenceTransformer, util
# LaBSE / 多语言 sentence-transformers 是专门为跨语言对齐句向量训练的
m = SentenceTransformer("sentence-transformers/LaBSE")   # 模型名待核以实际可用为准

en = ["I love natural language processing", "The weather is nice today"]
zh = ["我爱自然语言处理", "今天天气不错"]
E = m.encode(en, convert_to_tensor=True, normalize_embeddings=True)
Z = m.encode(zh, convert_to_tensor=True, normalize_embeddings=True)

sim = util.cos_sim(E, Z)            # 2x2 相似度矩阵
print(sim.argmax(dim=1))           # 期望对角线最大:每句英文检索到它的中文译文
# 对角元显著高于非对角 = 跨语言语义对齐成立,定量验证了第四节的机制二
```

**评测纪律(一个常见坑)**:很多跨语言测试集是**从英语翻译来的**(XNLI 就是)。翻译会引入"translationese"(译文腔)——译文的句法/词汇分布偏向源语言,跟该语言的母语文本分布不同。**在翻译测试集上的高分,可能高估了模型在真实母语文本上的表现**。读多语言论文的分数时,要分清测试集是母语原生采集的(如 TyDi QA 部分)还是翻译来的。

## 八、设计权衡与常见坑

- **词表大小的多语言两难**:词表越大,每种语言分到的 token 越多、切分越完整(缓解 curse 预算一),但嵌入矩阵参数 `= V × d` 线性增长(算法课 01 算过),小模型里嵌入层会变成参数大头,且大词表里长尾 token 训练不充分。**单语言 3万~5万,大规模多语言常 25万起,XLM-V 上百万**——没有免费午餐。
- **"零 OOV"≠"处理得好"**:字节级保证任何输入都能编码,但低资源/非拉丁语言可能被切成一堆无意义字节片段。**别把"不报 UNK"当成"支持得好"**——要看实际 token/字符比和下游分数。
- **训练/推理分词必须严格一致(多语言下被放大)**:是否加 BOS/EOS、空格归一化(`▁` 还是 `Ġ`)、Unicode 规范化(NFC/NFKC)、大小写处理——任一不一致都会让推理分布偏离训练分布。多语言里脚本、变音符、全角半角更易出岔子。**永远用同一个 tokenizer 对象 / 同一份配置**。
- **Unicode 规范化的坑**:同一个字符可能有多种 Unicode 表示(组合字符 vs 预组合,如 `é` = `e`+`́` 或单个 `é`)。训练前不统一做 NFC/NFKC 规范化,会让"看起来一样的字"变成不同 token,凭空制造稀疏。
- **温度采样的副作用**:把低资源语言采样率抬太高(α 太小),会**饿着高资源语言**,拖累它们(以及依赖它们做迁移锚点的语言)。α 是个要调的旋钮,不是越平越好。
- **过度依赖共享锚点的脆弱性**:跨脚本迁移(中↔英)严重依赖第三方锚点(数字、专名)和机制二的隐式对齐;若两种语言连专名写法都不共享(不同脚本 + 音译不一致),迁移会显著变弱。
- **"加语言一定更好"的错觉**:curse of multilinguality 说明,容量不变时盲目加语言会**集体变差**。要加语言,得同步加容量(参数/词表)或上模块化(adapter/MoE),否则是负优化。
- **新手坑:以为分词器能事后修**。分词器在预训练前定死,几乎不能改;改了等于换模型重训(算法课 01 强调过)。**多语言里这意味着"开局没给某语言留够 token,这个模型这辈子在这门语言上就吃亏"**,后期只能靠继续预训练/扩词表(代价高)补救。

## 十一、小结与承上启下

这一章把"分词"从单语言的技术选择,升级成了多语言系统的**资源分配与迁移工程**:

- **接口层**:SentencePiece 用 `▁` 元符号 + 直接在 raw text 上训,做到**语言无关、可逆、对无空格语言友好**;字节级 BPE 用 256 字节 + 字节↔可见字符双射,做到**定义上零 OOV**。两者都把空格变成可见符号、都把"公平性"留给了训练语料分布。
- **迁移机制**:mBERT/XLM-R 的零样本跨语言迁移=**机制一(共享子词词表提供锚点,缝住嵌入层)+ 机制二(联合 MLM 在固定参数预算下逼出语言无关的语义表示)**;XLM 的 TLM 在有平行语料时用跨语言掩码显式加强对齐,但数据规模化(XLM-R)在长尾语言上往往更实在。
- **代价**:curse of multilinguality 是"固定预算/多方瓜分"的经济学必然——词表、参数、训练数据三重稀释;温度采样 `q_ℓ ∝ p_ℓ^α` 缓解数据不平衡,但真正突破要靠加容量(参数/词表/MoE)或语言特定模块。
- **硬骨头**:形态丰富语言被子词切碎、脚本差异带来字节税、低资源语言的根因是数据不存在;评测要分任务级(XNLI/XTREME)和表示级(BLI/句检索),且要警惕翻译测试集的高估。

**承上**:本章的"共享表示空间被联合训练挤出来"和算法课的注意力/参数共享、NLP 课 06 的 MLM 是同一套逻辑的延伸。**启下**:这套"多语言一个模型"的范式,正是 NLP 课 09(LLM 时代:经典 NLP 如何被 Prompt 统一)的前奏——当模型大到能在一个词表、一套参数里装下上百种语言和几乎所有任务,"为每个任务/每种语言单独建模"的旧范式就被瓦解了。下一步我们会看到,连"用标注数据微调任务头"这一步,都开始被 in-context learning(算法课 14)取代。分词器仍然是那个在最底层、最沉默、却决定了上限的接口——它定义了这个统一模型究竟能听懂世界上的哪些语言。
