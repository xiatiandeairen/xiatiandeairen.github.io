> 这一章把你从"微调大模型 = 把全部权重都拿梯度更新一遍"带到"能在白板上推清楚 LoRA 为什么只学一个低秩增量就够、为什么它推理时零延迟、QLoRA 为什么能在单卡上微调 65B 模型,并能自己写出 LoRA 线性层、估算它省了多少显存与可训练参数"。读完你应当能回答一个本质问题:**微调到底改变了模型的什么,以至于这个"改变"可以被压进千分之几的参数里。**

## 为什么需要这一章:全量微调的成本是不可承受的

先把问题钉死。你有一个预训练好的大语言模型,参数量记作 N(比如 7B = 7×10⁹,70B = 7×10¹⁰)。现在你要把它适配到一个下游任务——客服问答、代码补全、医疗摘要。最朴素的做法是**全量微调(full fine-tuning)**:把全部 N 个参数都当成可训练变量,在下游数据上跑梯度下降。

这件事在 BERT 时代(参数量 ~10⁸)还能接受,到了大模型时代成本爆炸。我们逐项算账,假设用 Adam 优化器、混合精度训练(这是当前标配,前置见 MLSys 课程《混合精度与数值稳定》):

```
每个可训练参数在训练时占的显存:
  - fp16 权重副本(前向用)         2 字节
  - fp32 权重主副本(master weight)  4 字节
  - fp32 Adam 一阶动量 m            4 字节
  - fp32 Adam 二阶动量 v            4 字节
  - fp16/fp32 梯度                  2~4 字节
  合计约 16 字节 / 参数(经典的 "16 bytes per param" 估算)
```

这个 16 字节里,**真正"模型本体"只有 2 字节(fp16 权重),另外 14 字节全是为了"能更新它"而背的包袱**(主副本 + 优化器状态 + 梯度)。对一个 7B 模型:

```
全量微调显存(仅优化器侧,不含激活):
  7×10⁹ × 16 字节 ≈ 112 GB
```

112 GB 已经超过单张 A100-80G 的容量,必须多卡。70B 模型则需要约 1.1 TB。这还没算前向传播产生的**激活(activation)**——它随 batch size 和序列长度线性增长,在长上下文下往往是显存的另一个大头(前置见《激活重计算》)。

显存只是第一座山。还有两座:

- **存储**:全量微调每个任务都产出一份完整的 N 参数权重。你要服务 100 个客户、每个客户一个定制模型,就是 100 份 7B 权重 ≈ 100 × 14 GB = 1.4 TB 的存储,而且每份都要单独加载、单独占显存。
- **多任务部署**:推理时若要在 100 个微调版本间切换,要么常驻 100 份权重(显存爆炸),要么频繁换入换出(把 14 GB 权重从硬盘搬进显存,延迟以秒计)。

**一个尖锐的问题浮出来了:为了把模型从"通用"调到"会做某个具体任务",我们真的需要动用全部 N 个参数的"自由度"吗?** 直觉上不该需要——下游任务通常只是预训练能力的一个**专门化(specialization)**,不是从头学一种新能力。如果"专门化"本身只占很少的自由度,我们就应该能用远少于 N 的参数来表达它。

这正是 **参数高效微调(Parameter-Efficient Fine-Tuning,PEFT)** 的全部立足点:**冻结绝大部分预训练参数,只引入并训练极少量(通常是原参数的 0.01% ~ 1%)新参数,就能逼近甚至匹配全量微调的效果。** 本章以 LoRA 为主线把这件事讲透,然后铺开 QLoRA、Adapter、Prefix/Prompt-tuning、(IA)³ 这一族方法,讲清何时该用哪个。

### 预备:把"微调改变了什么"形式化

设预训练模型里某个权重矩阵是 W₀ ∈ ℝ^{d×k}(比如注意力里的 W_q、W_v,或 FFN 的某层)。全量微调后它变成 W = W₀ + ΔW,其中

```
ΔW = W_finetuned − W_pretrained
```

是微调**净改变量(weight update / delta)**。全量微调直接学 W(等价于学 ΔW),ΔW 是一个满秩、d×k 个自由度的稠密矩阵。

PEFT 的核心赌注是:**对绝大多数下游任务,这个 ΔW 不需要 d×k 那么多自由度——它有某种低复杂度结构,可以用远少的参数刻画。** 不同 PEFT 方法的区别,本质是对"ΔW 长什么样"下了不同的结构假设。LoRA 下的假设最干净、最成功:**ΔW 是低秩的。** 我们从这里开始。

## 第一节:LoRA 的低秩假设与完整推导

### 问题:能不能用一个低秩矩阵表示 ΔW

LoRA(Low-Rank Adaptation,低秩适配,Hu et al., 2021,微软)的出发点是一个观察 + 一个假设。

**观察(来自更早的工作):** 预训练语言模型有很低的"**内在维度(intrinsic dimension)**"。Aghajanyan et al. (2020) 发现,把微调限制在一个随机选定的低维子空间里(只优化少数几百到几千个坐标,再用一个固定随机投影映射回全参数空间),竟然就能在很多任务上达到全量微调 90% 的效果。换句话说,**"微调需要的有效自由度"远小于参数总数。**

> **直觉:什么叫内在维度低。** 想象 ΔW 是一张 d×k 的大表格,表面上有 d·k 个独立旋钮。但如果微调真正做的事只是"沿着少数几个方向调整"(比如"让模型更礼貌""更偏向法律术语"),那这些旋钮其实是高度联动的——它们被少数几个"宏观方向"驱动。内在维度低,就是说驱动微调的宏观方向数远小于 d·k。

**假设(LoRA 的核心):** 既然有效自由度低,那么 ΔW 本身应当是**低秩(low-rank)**的,或者至少能被一个低秩矩阵很好地近似。一个秩为 r 的矩阵可以分解成两个瘦长矩阵的乘积。于是 LoRA 直接把 ΔW 参数化为:

```
ΔW = B · A         其中  B ∈ ℝ^{d×r},  A ∈ ℝ^{r×k},  r ≪ min(d, k)
```

A 把输入从 k 维"压"到 r 维(降维),B 再从 r 维"升"回 d 维(升维)。乘积 BA 的秩至多是 r(因为秩不超过任何一个因子的秩,而 A、B 的秩都 ≤ r)。这就**从结构上强制** ΔW 的秩不超过 r——低秩假设被硬编码进了参数化里。

> **为什么秩(BA) ≤ r 是显然的。** 矩阵乘积 BA 的列空间被包含在 B 的列空间里(BA 的每一列都是 B 的列的线性组合),而 B 只有 r 列,所以 B 的列空间维数 ≤ r,于是 rank(BA) ≤ rank(B) ≤ r。这是线性代数基本事实,无需额外证明。

### 机制:前向传播怎么改

冻结的原权重 W₀ 不动,新增的可训练参数只有 A 和 B。修改后的线性层前向变成:

```
原始:    h = W₀ · x
LoRA:    h = W₀ · x  +  ΔW · x
            = W₀ · x  +  (α / r) · B · A · x
```

其中:
- x ∈ ℝᵏ 是输入(实际是 batch × seq 的一批向量,这里写单个向量),h ∈ ℝᵈ 是输出。
- W₀ 被**冻结**(requires_grad = False),梯度不流过它;只有 A、B 接收梯度并更新。
- α 是一个**缩放超参(scaling factor)**,α/r 是施加在低秩增量上的缩放系数,下面专门讲它。

**关键的初始化。** LoRA 要求训练开始的那一刻,模型行为和预训练模型**完全一致**(否则相当于在还没学之前就破坏了预训练知识)。这要求初始 ΔW = BA = 0。LoRA 的做法是:

```
A ← Kaiming 均匀初始化(原论文用随机高斯,HF PEFT 实现用 kaiming_uniform_,二者均可)
B ← 全零初始化,B = 0
```

于是初始 BA = B·A = 0·A = 0,初始 h = W₀x,行为不变。**两个都置零行不行?** 不行——若 A、B 都为零,则 ∂L/∂A 和 ∂L/∂B(见下面梯度推导)在初始点都为零,参数永远不更新,训练卡死。让一个随机、一个为零,既保证初始增量为零,又保证梯度能流动,是这个初始化的精髓。

### 推导:参数量从 d×k 降到 r×(d+k)

这是 LoRA 省钱的核心账,必须算清。对一个 d×k 的权重矩阵:

```
全量微调可训练参数:  d × k
LoRA 可训练参数:     |A| + |B| = r·k + d·r = r · (d + k)
```

压缩比:

```
LoRA 参数 / 全量参数 = r·(d + k) / (d·k)
```

代入典型数字感受一下。取 d = k = 4096(LLaMA-7B 隐藏维度量级),r = 8:

```
全量:   4096 × 4096       = 16,777,216 ≈ 1.68×10⁷ 个参数
LoRA:   8 × (4096 + 4096) = 65,536      ≈ 6.55×10⁴ 个参数
压缩比: 65536 / 16777216  ≈ 0.0039  ≈ 0.39%
```

**一个权重矩阵就降到了原来的约 1/256,即千分之四左右。** 这就是标题"千分之一"的来源量级(r 更小、矩阵更大时可到千分之一以下)。

注意一个易被忽略的事实:**压缩比与 d·k 成反比,与 r 成正比**。矩阵越大(d、k 越大),LoRA 越划算;这正好契合大模型——矩阵越大的地方,低秩近似省得越多。当 r·(d+k) ≥ d·k,即 r ≥ d·k/(d+k) 时(d=k 时即 r ≥ d/2),LoRA 反而比全量参数还多,这时低秩参数化失去意义——但实践中 r 通常取 4~64,远小于这个临界值。

### 缩放 α 在数学上到底干什么

很多人把 α 当成玄学超参,其实它的作用可以讲死。LoRA 实际加到前向里的增量是 (α/r)·BA。考虑当你想增大秩 r(比如从 8 调到 16)来给适配更多容量时:

- A、B 是用 N(0, σ²) 与 0 初始化的,初始尺度与 r 无关;训练后 BA 的元素尺度大致随 r 增长(更多秩 = 更多项相加,方差累加)。
- 如果不除以 r,增大 r 会让 (BA)x 的整体幅度跟着变大,相当于隐式地放大了"学习率/步长",你就得重新调学习率。
- **除以 r 起到归一化作用:让"有效更新幅度"对 r 大致不敏感,从而你换 r 时不必重调学习率。** α 则是你能独立旋的那个"增量强度"旋钮。

实践经验(原论文与社区共识):**固定 α(常取 16 或 8),只调 r;α/r 一起决定增量强度。** 一个常见配置是 r=8、α=16,即缩放系数 α/r = 2。也有人主张把 α 设成 2r 让缩放恒为 2。需要强调的是**这些是经验默认值,不是从理论严格推出的最优**,具体任务上最优 (r, α) 仍需扫一扫「待核:不同任务最优 r/α 的系统性结论,社区数据分散」。

> **进阶提醒:rsLoRA。** Kalajdzievski (2023) 指出,在很大的 r 下,α/r 这个缩放会让梯度尺度过小、压制大 r 的收益,提出改用 α/√r(rank-stabilized LoRA)。这说明 α/r 不是唯一正确答案,而是"在常用小 r 区间够用"的工程选择。

### 推理时合并:为什么 LoRA 零额外延迟

这是 LoRA 相对 Adapter 类方法最大的工程优势,务必理解透。训练完成后,A、B 是固定的小矩阵。推理时我们可以把低秩增量**一次性合并(merge)** 回原权重:

```
W_merged = W₀ + (α/r) · B · A          # 一次性算好,d×k 矩阵
推理前向: h = W_merged · x              # 和原始线性层一模一样
```

合并后,推理图里**只有一个 d×k 的矩阵乘**,和没用 LoRA 的原模型在计算上**完全相同**——没有多出来的算子、没有额外的矩阵乘、没有额外访存。**因此 LoRA 在推理期零额外延迟、零额外显存(相比原模型)。** 这是它和"在前向链路里插入额外小网络"的 Adapter(下文)的本质分野:Adapter 的额外层在推理时**无法消去**,永远要多算一遍。

合并是可逆的(W₀ = W_merged − (α/r)BA),所以也能**反合并**,这给多任务部署留了活口:

- **不合并、动态加载**:基座 W₀ 常驻显存,每个任务只存一份极小的 {A, B}(几 MB 级)。来什么任务,就把对应的 (α/r)BA 临时加到前向里——这就是 LoRA 多任务部署的杀手锏。100 个任务 = 1 份基座 + 100 份几 MB 的 adapter,而不是 100 份 14 GB 的全量权重。
- **合并、单任务高吞吐**:若线上只跑一个固定任务,合并掉得到零延迟。

工业级服务(如 vLLM 的 multi-LoRA、S-LoRA)甚至能在一个 batch 里同时服务**不同 LoRA 适配器**的请求——基座共享,各请求挂各自的 (A,B),用分组矩阵乘并行算出各自的增量。

### 该把 LoRA 加到哪些矩阵上

Transformer 的一层里有一堆权重矩阵:注意力的 W_q、W_k、W_v、W_o,FFN 的 W_up、W_down(SwiGLU 还有 W_gate)。LoRA 不必全加。原论文的关键消融结论:

- 在固定可训练参数预算下,**把 LoRA 加到 W_q 和 W_v 上,效果好于只加 W_q**;加到注意力的 query/value 投影性价比最高。
- 后续社区实践趋势是**全注意力 + FFN 都加**(q,k,v,o,up,gate,down),尤其在 QLoRA 设置下,加得越全效果越稳——因为此时基座被量化压缩了表达力,需要 LoRA 在更多位置补偿。

**这背后的取舍**:加的矩阵越多 → 可训练参数越多、容量越大、显存越高,但边际收益递减。预算紧就主攻 q、v;预算松或基座被量化就铺满。

### 代码:LoRA 线性层的 PyTorch 实现

下面是一个能体现全部机制的最小实现——冻结 W₀、随机初始化 A、零初始化 B、前向加低秩增量、支持合并/反合并。

```python
import math
import torch
import torch.nn as nn

class LoRALinear(nn.Module):
    """在一个冻结的 nn.Linear 外面包一层低秩增量。
       h = W0 @ x + (alpha/r) * B @ (A @ x)
    """
    def __init__(self, base_linear: nn.Linear, r: int, alpha: int,
                 dropout: float = 0.0):
        super().__init__()
        self.base = base_linear          # 原始线性层 W0 (含 bias)
        # --- 关键 1:冻结原权重,梯度不流过 W0 ---
        for p in self.base.parameters():
            p.requires_grad = False

        d_out, d_in = base_linear.weight.shape   # W0 形状是 (d_out, d_in)
        self.r = r
        self.scaling = alpha / r                 # 缩放系数 α/r
        self.lora_dropout = nn.Dropout(dropout)

        # --- 关键 2:A 随机、B 全零;保证初始 BA = 0 ---
        # A: (r, d_in)  把输入降到 r 维
        # B: (d_out, r) 把 r 维升回输出维
        self.A = nn.Parameter(torch.empty(r, d_in))
        self.B = nn.Parameter(torch.zeros(d_out, r))
        nn.init.kaiming_uniform_(self.A, a=math.sqrt(5))  # A 用 Kaiming 随机
        # B 保持全零 -> 初始增量为 0 -> 训练起点行为 == 预训练模型
        self.merged = False

    def forward(self, x):
        # 基座输出(冻结,no_grad 不是必须,因为参数已冻结,但能省一点)
        base_out = self.base(x)
        if self.merged:                  # 已合并则增量已在 base 里,直接返回
            return base_out
        # 低秩增量:先 A@x 降维 (..., r),再 B 升维 (..., d_out)
        # 写成 x @ A.T @ B.T 以匹配 (batch, seq, d_in) 的张量布局
        lora_out = self.lora_dropout(x) @ self.A.t() @ self.B.t()
        return base_out + self.scaling * lora_out

    @torch.no_grad()
    def merge(self):
        """把 (alpha/r)·B·A 加进 W0,推理零延迟。可逆。"""
        if self.merged:
            return
        delta = self.scaling * (self.B @ self.A)   # (d_out, d_in)
        self.base.weight.add_(delta)
        self.merged = True

    @torch.no_grad()
    def unmerge(self):
        if not self.merged:
            return
        delta = self.scaling * (self.B @ self.A)
        self.base.weight.sub_(delta)
        self.merged = False
```

几个实现要点,新手容易写错:

1. **形状与转置**:输入张量一般是 (batch, seq, d_in),A 是 (r, d_in)、B 是 (d_out, r)。要算 x·Aᵀ 得到 (..., r),再 ·Bᵀ 得到 (..., d_out)。漏掉转置会直接维度报错。
2. **dropout 加在输入侧**:LoRA 的 dropout 通常作用在进入低秩分支的 x 上(正则化低秩增量),不是加在输出。
3. **合并/反合并要 no_grad 且就地改 base.weight**:合并是部署期操作,不进计算图。
4. **优化器只收集需要梯度的参数**:`optim.AdamW([p for p in model.parameters() if p.requires_grad], ...)`,否则会给冻结的 W₀ 也分配优化器状态,白白吃显存——这是 PEFT 省显存的关键落地点。

### 梯度与显存分析:省的不只是参数

**梯度推导。** 设损失 L,对低秩分支输出 y = (α/r)·BAx,记上游梯度 g = ∂L/∂h(它会流到 base 和 lora 两支)。链式法则:

```
设 s = α/r。前向 lora 分支:  z = A x  (∈ℝʳ),   y = s·B z
∂L/∂B = s · (∂L/∂y) · zᵀ = s · g · (A x)ᵀ      形状 (d×r) ✓
∂L/∂A = s · Bᵀ · (∂L/∂y) · xᵀ = s · Bᵀ g xᵀ    形状 (r×k) ✓
∂L/∂W₀ = 0                                       (冻结,不更新)
∂L/∂x = W₀ᵀ g + s · Aᵀ Bᵀ g                      (梯度仍需穿过本层往下传)
```

注意倒数第二行:**虽然不更新 W₀,但梯度仍要经过 W₀ 算 ∂L/∂x 往更下游传**(否则下游的 LoRA 收不到梯度)。这意味着**反向传播该有的链路一条不少**。那 PEFT 到底省在哪?

**显存账,逐项拆解(这是高频面试与实践误区):**

```
                          全量微调        LoRA
权重 fp16 (前向)           N×2 字节        N×2 字节(基座照样占)  — 不省
优化器状态 m,v (fp32)      N×8 字节        |AB|×8 字节            — 大省
fp32 主权重副本            N×4 字节        |AB|×4 字节            — 大省
权重梯度                   N×(2~4)         |AB|×(2~4)             — 大省
激活(activation)         与 batch/seq 相关,两者量级相近        — 基本不省
```

**结论(必须记住):LoRA 省的是"优化器状态 + 主副本 + 权重梯度"这块,即那"16 字节里的 14 字节"中绝大部分;它不省基座权重本身(基座照样要加载、照样占 fp16 显存),也基本不省激活。** 所以 LoRA 把训练显存从"≈16×N"压到"≈2×N(基座 fp16,被冻结只读) + 16×|AB|(极小) + 激活"。对 7B、r=8 全注意力:|AB| 通常是百万级,16×|AB| 约 100 多 MB(~128 MB),而基座 fp16 是 14 GB——**显存大头从此变成"基座只读权重 + 激活",优化器侧几乎消失**。

这也解释了下一节为什么要做 QLoRA:**既然 LoRA 之后基座的 fp16 权重(14 GB)成了显存大头,那就把它也压缩掉——量化到 4-bit。**

## 第二节:QLoRA——把基座 4-bit 量化,单卡微调大模型

### 问题:LoRA 之后,基座权重成了显存瓶颈

接上节结论:LoRA 把优化器侧显存几乎清零后,7B 训练显存的大头是基座的 14 GB fp16 权重 + 激活;65B 则是 130 GB,仍然单卡装不下。**QLoRA(Dettmers et al., 2023)的目标:把基座权重压到 4-bit,让 65B 模型能在单张 48G(甚至 24G)消费级显卡上微调。** 它是三个技术的组合,逐个讲透。

### 技术一:NF4——给正态分布权重量身定做的 4-bit 类型

4-bit 只有 2⁴ = 16 个可表示的离散值(量化"码本/codebook")。怎么选这 16 个值,直接决定量化误差。普通做法是均匀量化(等间距 16 档),但**神经网络权重经验上近似服从零均值正态分布 N(0, σ²)**——绝大多数权重挤在 0 附近,极少数是大值。均匀量化在 0 附近放太少档(浪费精度)、在尾部放太多档(用不上)。

**NF4(4-bit NormalFloat)的核心思想:信息论上最优的量化,应让每个量化档"装等量的概率质量"(分位数量化,quantile quantization)。** 既然权重服从正态,就用标准正态的分位数来定这 16 个档:把 N(0,1) 的概率质量等分成若干份,取每份的代表点作为码本值。这样**每个码字被使用的频率大致相等**,信息利用率最高。具体地,NF4 取标准正态分位函数(逆 CDF)在等概率分点上的值,再归一化到 [−1, 1],并**保证 0 被精确表示**(因为权重里 0 很常见,且 padding/稀疏需要精确零)。

```
NF4 码本(16 个值,理论构造,归一化到 [-1,1],关于 0 近似对称):
  用 N(0,1) 的逆CDF Φ⁻¹ 在等概率分点上取值,缩放到 [-1,1]
  使得各档承载的概率质量相等(分位数量化)
  并强制包含精确的 0.0
  (具体 16 个数值见 bitsandbytes 源码常量,此处不逐一抄列 —— 待核:精确数值)
```

量化时:对一组权重,先算它们绝对值的最大值 absmax,把权重除以 absmax 缩放到 [−1, 1],再查 NF4 码本找最近的档,存下 4-bit 索引 + 一个 fp16/fp32 的 absmax(缩放因子)。反量化:索引查码本 × absmax。

> **为什么不直接量化成 int4 均匀档?** 因为权重不是均匀分布的。把档位"按概率密度"分配(密的地方档密、疏的地方档疏),在相同 4-bit 预算下量化误差更小。这是 NF4 区别于普通 int4 的全部要点。量化的一般原理(scale、零点、对称/非对称、误差)见 MLSys 课《量化》;NF4 是其中"为正态分布定制码本"的特例。

### 技术二:双重量化(Double Quantization)——把缩放因子也压一道

NF4 是**分块量化(block-wise)**:不是全矩阵共用一个 absmax,而是每 64 个权重一块、各存自己的 absmax(否则一个离群大值会把整块的缩放撑大、毁掉小值精度)。但这又带来开销:每 64 个 4-bit 权重就要存一个 32-bit 的 absmax,**相当于每参数多摊 32/64 = 0.5 bit**。

**双重量化:把这些 absmax 自己再量化一次(8-bit)。** 第一层量化的缩放因子(每块一个 fp32)被收集起来,每 256 个为一组,再做一次 8-bit 量化。这样每参数的额外开销从 0.5 bit 降到约 0.127 bit「待核:精确数值依赖 blocksize 配置」。看着是抠细节,但在 65B 规模上能省下几个 GB,是单卡装下的最后一块拼图。

### 技术三:Paged Optimizer——用统一内存扛显存尖峰

训练时偶发的显存尖峰(比如某个长序列 batch)可能瞬间 OOM。QLoRA 借用 NVIDIA 统一内存(unified memory),让优化器状态在显存不够时**自动分页换出到 CPU 内存(page-out)**、需要时换回,像操作系统的虚拟内存换页一样,避免硬性 OOM 崩溃。这是工程兜底,不影响算法。

### QLoRA 的前向:量化基座 + fp16 LoRA

把三者拼起来,QLoRA 的一层前向是:

```
存储:  W₀ 以 NF4(4-bit)存,absmax 经双重量化压缩
前向:  W₀ 现场反量化回 BF16  ->  W₀_bf16 = dequant_NF4(W₀_4bit)
        h = W₀_bf16 · x  +  (α/r) · B · A · x
反向:  梯度只流到 BF16 的 A、B;W₀ 始终是 4-bit 只读,不更新
```

**关键设计:基座只用来做前向计算,反量化是"即用即弃"的——只在算这一层时临时把 4-bit 解回 BF16,算完即丢,不长期占显存。** 而 LoRA 的 A、B 保持 BF16 全精度(它们是真正要学的、量少,不能再压)。梯度穿过被反量化的基座往下传,但基座本身永不更新。

**为什么这样不掉点?** 一句话:**量化误差被 LoRA 吸收了。** 基座 4-bit 引入的量化噪声,可以被全精度的低秩增量 BA 在训练中部分补偿——LoRA 学到的不仅是任务适配,还隐含修正了量化带来的偏差。QLoRA 论文的核心实证结论是:**4-bit NF4 基座 + LoRA,在多个基准上能匹配 16-bit 全量微调的效果**「待核:具体 benchmark 数值(如 Guanaco/Vicuna 评测)」。

### QLoRA 显存账:65B 单卡的可行性

```
65B 模型基座权重:
  fp16:  65×10⁹ × 2 字节 ≈ 130 GB        (单卡装不下)
  NF4:   65×10⁹ × 0.5 字节 ≈ 32.5 GB      (+ 双重量化的少量缩放因子)
LoRA 优化器侧:百万~千万级参数,几百 MB(可忽略)
激活:  靠梯度检查点(activation checkpointing)压到可控
=> 总显存量级 ~ 35-48 GB,单张 A100-40/48G 可行
```

**4-bit 把基座显存直接砍到约 1/4**,这是 QLoRA 能在单卡微调超大模型的根因。代价是:每次前向多一步反量化(增加少量计算与延迟);量化噪声理论上设了天花板,极限质量可能略低于全精度全量微调(但实测差距很小)。

> **训练快还是慢?** QLoRA 通常比 fp16 全量微调**慢**(反量化开销 + 4-bit 算子未必有 fp16 那么优化的 kernel),但它把"装不下/要多卡"变成了"单卡能跑"——它买的是**可行性和成本**,不是速度。这是关键的取舍定位。

## 第三节:其他 PEFT 方法与选型权衡

LoRA 不是唯一的 PEFT。理解其他方法,本质是理解它们对"ΔW/适配长什么样"下了**不同的结构假设**,以及各自的延迟/质量代价。

### Adapter:在层间插入小瓶颈网络

Adapter(Houlsby et al., 2019,最早的 PEFT 之一)在 Transformer 每个子层(注意力/FFN)**之后**插入一个小模块:

```
Adapter(h) = h + W_up · f( W_down · h )        # 残差 + 瓶颈
  W_down: ℝ^{d×m}  把 d 维压到瓶颈维 m (m ≪ d)
  W_up:   ℝ^{m×d}  再升回 d 维
  f:      非线性(如 ReLU/GeLU)
```

它和 LoRA 形似(都是降维-升维瓶颈),但有**本质区别**:

- **Adapter 是串行插入主干、带非线性的新层**;LoRA 是**并行**旁挂在某个权重矩阵旁、且是线性的。
- **致命点:Adapter 不能合并。** 因为中间有非线性 f,W_up·f(W_down·h) 无法被吸收进任何一个原权重矩阵。**所以 Adapter 在推理时永远多一层前向计算 → 有不可消除的额外延迟。** 在自回归推理(逐 token 生成,本就被访存和小算子拖累)下,这个延迟尤其刺眼。

这正是 LoRA 取代 Adapter 成为主流的核心原因:**同样是低秩瓶颈思想,LoRA 用"线性 + 并行"换来了可合并、零推理延迟。**

### Prefix-tuning / Prompt-tuning:只学一段软提示

这一族不动任何权重,而是**在输入侧或注意力里添加一段可训练的"虚拟 token"**,只学这段的嵌入:

- **Prompt-tuning(Lester et al., 2021)**:在输入 embedding 序列前面拼接 p 个**可训练的连续向量**(soft prompt / 软提示),每个是 ℝᵈ。模型其余全冻结,只学这 p×d 个数。这是参数量最小的 PEFT(p 通常 ~20-100)。
- **Prefix-tuning(Li & Liang, 2021)**:更强——不只在输入层,而是在**每一层的注意力**的 Key、Value 前面都拼接可训练的前缀向量(prefix)。相当于给每层注意力都注入一段可学的"上下文记忆"。参数比 prompt-tuning 多(每层都有),但容量更大。

```
Prompt-tuning:  输入 = [P₁..P_p ; embed(x₁)..embed(xₙ)]   仅 P 可训练
Prefix-tuning:  每层注意力的 K,V 前拼接可训练的 K_pre, V_pre
                Attn(Q, [K_pre;K], [V_pre;V])
```

**取舍**:
- 优点:参数极少;天然支持多任务(每任务一段 prefix,共享主干);prompt-tuning 实现简单。
- 缺点:(1) prefix/prompt 会**占用上下文窗口**(那 p 个虚拟 token 挤占了序列长度预算,且每步推理都要参与注意力,**有持续的延迟与 KV cache 开销**)。(2) 优化较难、对初始化敏感,在小模型上效果常不如 LoRA;**模型越大,prompt-tuning 越接近全量微调**(Lester 论文的核心结论:规模能弥补)。

### (IA)³:学习缩放向量,极致轻量

(IA)³(Liu et al., 2022,"Infused Adapter by Inhibiting and Amplifying Inner Activations")更激进:**不加矩阵,只学几个逐元素的缩放向量(rescaling vectors)**,用它们去逐元素地放缩(⊙ 表示逐元素乘)注意力的 Key、Value 和 FFN 的中间激活:

```
注意力:  K ← l_k ⊙ K,   V ← l_v ⊙ V        l_k, l_v ∈ ℝ^{d_head}
FFN:     中间激活 ← l_ff ⊙ (中间激活)        l_ff ∈ ℝ^{d_ff}
  三个向量 l_k, l_v, l_ff 初始化为全 1(初始恒等),只学它们
```

参数量是**所有 PEFT 里最小的之一**(只有几个向量,数量级 ~ 千分之零点几)。因为是逐元素缩放(线性的对角变换),它**也能合并进权重**(把 l 吸收进对应的投影矩阵),所以推理也可零延迟。代价是容量最小,适配能力有限,适合任务与预训练很接近、只需"微调旋钮"的场景。

### 选型决策表

把上面拧成一张可操作的表(质量为相对经验排序,非绝对):

```
方法            可训练参数量    推理额外延迟   可合并   典型适用场景
全量微调         100%           0(本体)      —       数据充足、算力管够、要榨干天花板
LoRA            ~0.1%-1%       0(合并后)     ✓       默认首选;质量/成本/部署最均衡
QLoRA           ~0.1%-1%       小(反量化)    ✓(慢)  显存极紧、要单卡微调超大模型
Adapter         ~0.5%-5%       有(不可消)    ✗       需要非线性容量、不在意推理延迟
Prefix-tuning   ~0.1%-1%       有(占窗口)    ✗       多任务共享主干、可接受占上下文
Prompt-tuning   ~0.01%         有(占窗口)    ✗       超大模型 + 任务简单,极省参数
(IA)³           ~0.01%         0(合并后)     ✓       任务接近预训练,要极致轻量
```

**实务结论(可直接照用):**
- **默认上 LoRA。** 它在质量、参数量、推理零延迟、多任务部署四个维度都没有明显短板,是当前事实标准。
- **显存装不下基座就上 QLoRA。** 单卡微调 30B+ 模型基本只有这条路。
- 只有在"需要非线性额外容量且完全不在意推理延迟"的特殊场景才考虑 Adapter;在"超大模型 + 极省参数 + 多任务"才考虑 prompt/prefix;在"任务极接近预训练、要最轻"才考虑 (IA)³。

## 设计权衡与常见坑

**1. 秩 r 不是越大越好,但太小会限容量。** r 太小,低秩假设过强,复杂任务(如学一种全新的输出格式/语言)欠拟合;r 太大,参数/显存上去且边际收益递减,甚至接近全量微调失去意义。经验起点:简单适配 r=4~8,复杂任务 r=16~64。**先固定 α、扫 r。**

**2. "LoRA 省显存"的最大误解:以为基座也省了。** 重申第一节结论:**LoRA 省的是优化器状态/主副本/梯度,不省基座 fp16 权重,也基本不省激活。** 想再省基座 → 量化 → QLoRA。想省激活 → 梯度检查点。这三者是正交的,实战常叠加用(QLoRA = 4bit基座 + LoRA + 梯度检查点)。

**3. 该加到哪些层别想当然。** 只加 W_q 往往不够;q+v 是高性价比起点;QLoRA 下建议铺满 q,k,v,o + FFN。漏掉关键矩阵会显著掉点,而掉点原因很隐蔽(看起来 LoRA "没用"其实是没加对地方)。

**4. 合并后再继续训练 = 灾难。** 若 merge 进 W₀ 后忘了 unmerge 就接着训练,梯度会把已合并的增量再算一遍,数值错乱。**训练态保持不合并,只在部署/推理前合并。** 代码里务必用一个 `merged` 标志位守住(见上文实现)。

**5. 学习率要比全量微调大。** LoRA 只动少量参数,且 B 从零起步,通常需要比全量微调**大一个量级**的学习率(全量常用 ~1e-5,LoRA 常用 ~1e-4~3e-4)「待核:最优 LR 依模型与任务,此为社区经验区间」。沿用全量的小学习率会学得极慢。

**6. QLoRA 的反量化精度坑。** 反量化目标精度要用 BF16 而非 FP16(BF16 动态范围大,适配大模型的激活范围);用错精度可能数值溢出/下溢。LoRA 参数本身**不要量化**,保持全精度。

**7. 多 LoRA 服务的正确性。** 一个 batch 混合多个适配器时,要保证每个请求挂对自己的 (A,B),且基座反量化结果可共享。实现错了会出现"串味"(A 请求用了 B 的适配器)。

**8. 别把 PEFT 当万能。** 当下游任务与预训练差异极大(如换一种全新模态、学一门预训练里几乎没有的语言),低秩/软提示假设可能不成立,全量微调或继续预训练仍是更稳的选择。PEFT 的前提是"适配是预训练能力的小幅专门化"。

**练习 1(计算题:省了多少可训练参数)。** 一个 LLaMA-7B 类模型,隐藏维 d=4096,共 32 层,每层在 W_q、W_k、W_v、W_o(均为 4096×4096)上加 LoRA,r=8、α=16。(a) 算总可训练参数量。(b) 与"全量微调这四个投影矩阵"相比的压缩比。(c) 若 |AB| 用 BF16 + AdamW(每参数 16 字节),优化器侧显存约多少 MB?
- *提示*:单矩阵 LoRA 参数 = r·(d+k) = 8·(4096+4096)。每层 4 个矩阵 × 32 层。全量这四个矩阵 = 4·4096²·32。(c) 把总参数 ×16 字节换算成 MB。验证:你应得到可训练参数约 8.4×10⁶ 量级(≈8.4 百万)、压缩比约 0.39%、优化器显存约 128 MB。

**练习 2(分析题:解释"低秩假设"与内在维度)。** 用自己的话回答:(a) "微调的 ΔW 是低秩的"这个假设,和"模型内在维度低"是什么关系?(b) 给一个你认为低秩假设会**失效**的下游任务例子,并说明为什么(从"需要多少独立方向的改变"角度论证)。(c) 为什么 LoRA 把 B 初始化为零、A 初始化为随机,而不是反过来或都为零?从初始增量和梯度流两方面各说一句。
- *提示*:(c) 都为零会让两个参数的梯度在初始点同时为零(看第一节梯度公式 ∂L/∂B ∝ g·(Ax)ᵀ、∂L/∂A ∝ Bᵀg xᵀ,代入 A=B=0 看是否为零);A=0,B 随机会破坏"初始行为不变"。

**练习 3(编码题:验证合并的等价性)。** 基于第一节的 `LoRALinear`,写一段测试:随机构造输入 x,记 `out1 = layer(x)`(未合并),调用 `layer.merge()` 后得 `out2 = layer(x)`,断言 `torch.allclose(out1, out2, atol=1e-5)`。再调用 `unmerge()` 后断言权重恢复到合并前(用合并前 clone 的权重对比)。解释:为什么这个等价性对 LoRA 的"推理零延迟"主张是必要的?
- *提示*:合并就是把 (α/r)BA 加进 W₀,前向数学上完全等价。allclose 的容差是为浮点误差留的。等价 ⇒ 合并后推理 = 一个普通线性层 ⇒ 零额外算子。

- *提示*:抓三个判别点——Adapter 有不可消延迟(扣分);Prefix 占上下文窗口且每步参与注意力(扣分);LoRA 不合并时可几 MB 级切换适配器、合并时零延迟;QLoRA 在显存维度最优但有反量化开销。多任务切换看"每个版本要存/换多大"。

## 小结与承上启下

这一章的全部分量压在一个洞察上:**微调对权重的净改变 ΔW 本质上是低秩的(内在维度低),所以可以用 BA 这样一个秩-r 分解来表达,参数量从 d·k 降到 r·(d+k),通常千分之几。** 由此:

- **LoRA** 用"线性 + 并行旁路 + B 零初始化 + 合并"四件套,做到了**极少可训练参数、训练显存大降(省优化器侧)、推理零延迟、多任务几 MB 级切换**。它的省显存边界要讲清:省优化器侧,不省基座,不省激活。
- **QLoRA** 接着把"LoRA 之后成为大头的基座 fp16 权重"用 NF4(分位数量化)压到 4-bit,加双重量化和分页优化器,**把 65B 模型的微调拉进单卡**;质量不掉的原因是量化误差被全精度 LoRA 吸收。
- **Adapter / Prefix / Prompt / (IA)³** 是对"适配长什么样"的另几种结构假设,各有延迟/质量/参数的取舍;LoRA 之所以成主流,是因为它在可合并(零延迟)这一点上独有优势。

**在整门课里的位置。** 前面的章节里你已经搭好了 Transformer 的骨架(注意力、归一化、位置编码)和把它高效跑起来的训练/推理系统认知;这一章解决的是**已经预训练好之后,如何低成本地把它专门化到你的任务**——它是连接"预训练大模型"和"对齐/部署"的关键工程枢纽。**承上**:LoRA 直接作用在你之前学的那些 W_q、W_v、FFN 矩阵上,是对它们的"低秩补丁"。**启下**:对齐章节里的 RLHF/DPO,在工业实践中**几乎总是叠加 LoRA/QLoRA 来训**(全量跑 PPO/DPO 太贵)——下一步把偏好对齐和 PEFT 合在一起,你就拿到了"用一张卡、千分之一参数,把一个大模型对齐到你想要的行为"的完整能力。

### 练习 1:LoRA 省了多少可训练参数 / 显存

**(a) 总可训练参数量**

单个矩阵的 LoRA 参数量(d=k=4096,r=8):

```
单矩阵 = r·(d + k) = 8·(4096 + 4096) = 8 × 8192 = 65,536
```

每层 4 个矩阵(W_q、W_k、W_v、W_o),共 32 层:

```
总参数 = 65,536 × 4 × 32
       = 65,536 × 128
       = 8,388,608
       ≈ 8.39 × 10⁶(约 838.9 万,即 8.4 百万)✓
```

**总可训练参数 ≈ 8.39 × 10⁶(约 8.4 百万)**,与提示给的量级一致。

**(b) 压缩比(相对全量微调这四个投影矩阵)**

全量微调这四个矩阵(每个 4096×4096,4 个 × 32 层):

```
全量 = d·k × 4 × 32 = 4096² × 128
     = 16,777,216 × 128
     = 2,147,483,648 ≈ 2.15 × 10⁹(约 21.5 亿)
```

压缩比:

```
压缩比 = 8,388,608 / 2,147,483,648
       = 65,536 / 16,777,216        (层数和矩阵数都约掉,等于单矩阵之比)
       = 1 / 256
       ≈ 0.003906
       ≈ 0.39% ✓
```

**压缩比 ≈ 0.39%(约 1/256)。** 注意层数 32 和矩阵数 4 在分子分母同时出现、可约掉,所以总压缩比就等于单个矩阵的压缩比 r·(d+k)/(d·k) = 16/4096 = 1/256,这跟第一节的结论自洽。

**(c) 优化器侧显存**

每参数 16 字节(BF16 权重 2 + fp32 主副本 4 + Adam 一阶动量 m 4 + 二阶动量 v 4 + 梯度 2,合计 16):

```
显存 = 8,388,608 × 16 字节
     = 134,217,728 字节
     = 134,217,728 / 1024 / 1024 MB
     = 128.0 MB(按 1 MiB = 1024² 字节)✓
```

**优化器侧显存约 128 MB(MiB)**,若按十进制 1 MB = 10⁶ 字节则约 134 MB。两种口径都在「约 128 MB」量级,与提示一致。

**对照体感:** 这 ~128 MB 是这 8.4 百万个可训练参数全套训练状态的开销;而 7B 基座本身的 fp16 权重是约 14 GB,后者照样要占显存(LoRA 不省基座)。这正是第一节强调的——**LoRA 把"优化器侧"几乎清零,显存大头从此变成"基座只读权重 + 激活"**。

### 练习 2:低秩假设、内在维度与零初始化

**(a) "ΔW 低秩"与"模型内在维度低"是什么关系**

两者是同一件事在不同对象上的表述,**内在维度低是因,ΔW 低秩是这个因被硬编码进参数化后的果**:

- **内在维度低(Aghajanyan 等的实证)**:微调一个模型真正需要调整的"有效自由度"远小于参数总数 N——把优化限制在一个几百~几千维的随机低维子空间里,就能达到全量微调约 90% 的效果。也就是说,驱动微调的"宏观方向"数量很少。
- **ΔW 低秩(LoRA 的假设)**:既然微调净改变量 ΔW = W − W₀ 只由少数几个独立方向驱动,那么作为一个 d×k 矩阵,它的这些"方向"就是它的列/行空间的基,数目少 ⇒ 秩低。秩 r 恰好就是"ΔW 需要多少个线性独立方向才能张成"的精确度量。
- **逻辑链**:内在维度低 ⇒ 微调只沿少数方向走 ⇒ ΔW 的列空间维数(= 秩)小 ⇒ 可写成 BA(B 提供 r 个列方向,A 给出沿这些方向的系数)。LoRA 把"秩 ≤ r"通过参数化**从结构上强制**,等于把"内在维度 ≤ r"的先验直接焊进了模型。

一句话:**内在维度低是关于"优化过程需要几个自由度"的经验观察,ΔW 低秩是把这个观察落到"权重增量矩阵的秩"上的具体结构假设;LoRA 用 BA 分解把它变成可训练的硬约束。**

**(b) 低秩假设会失效的下游任务例子**

判据:**当适配所需的"独立改变方向数"逼近甚至超过 r(乃至 min(d,k))时,低秩假设失效。** 凡是"不是对已有能力做小幅专门化,而是要灌入大量预训练里几乎不存在的新知识/新结构"的任务都属此类。

举例(任一即可):

- **学一门预训练语料里几乎没出现的语言**(如某种低资源语言或全新文字系统)。模型要重建该语言的词法、句法、语义映射——这不是"沿少数宏观方向微调",而是要在大量不同方向上改写表示,需要的独立方向数极多,ΔW 接近满秩,小 r 会严重欠拟合。
- **学一种全新模态/编码**(例如让纯文本模型理解一种结构化二进制协议、或一种它从没见过的符号体系)。所需的表示重组是大面积的,不是低秩补丁能覆盖的。
- **从"通用聊天"切到需要大量新事实记忆的任务**(把整本新领域知识"记进权重")。记忆型容量需求与秩大致正相关,低秩瓶颈成为信息瓶颈。

从"需要多少独立方向的改变"论证:这些任务要改的方向数 ≫ r,把 ΔW 强行压进秩 r ⇒ 丢失大量必要方向 ⇒ 欠拟合掉点。**此时该选全量微调或继续预训练**(对应第八节坑 8:PEFT 的前提是"适配 = 预训练能力的小幅专门化")。

**(c) 为什么 B 初始化为零、A 随机(而非反过来或都为零)**

回顾第一节梯度公式(s = α/r):

```
∂L/∂B = s · g · (A x)ᵀ      (正比于 Ax)
∂L/∂A = s · Bᵀ · g · xᵀ     (正比于 Bᵀ)
```

- **从初始增量看**:初始必须 ΔW = BA = 0,这样训练第 0 步模型行为和预训练完全一致,不破坏已有知识。只要 A、B 有一个为零,乘积 BA 就为零——B=0、A 随机满足(B·A = 0·A = 0);**反过来 A=0、B 随机也能让 BA=0,但会"破坏初始行为不变"吗?其实 BA 仍为 0、初始行为同样不变**——所以单看初始增量,A=0,B 随机与 B=0,A 随机都行,真正的分水岭在梯度流。
- **从梯度流看(这才是关键)**:
  - 选 **B=0、A 随机**:∂L/∂B ∝ g·(Ax)ᵀ,因 A 随机非零 ⇒ Ax≠0 ⇒ **B 立刻能收到梯度并离开零点**;B 一旦非零,下一步 ∂L/∂A ∝ Bᵀgxᵀ 也非零,A 随之更新。训练正常启动。
  - 选 **A=0、B 随机**:∂L/∂A ∝ Bᵀgxᵀ,B 非零 ⇒ A 能收到梯度;对称地也能启动。所以 A=0,B 随机理论上同样可训(HF 等实现选 B=0,A 随机是惯例,二者对称可行)。
  - 选 **A=B=0(都为零)**:Ax=0 ⇒ ∂L/∂B=0;Bᵀ=0 ⇒ ∂L/∂A=0。**两个梯度在初始点同时为零,参数永远不更新,训练彻底卡死。**(numpy 验证:A=B=0 时 Ax≡0,∂L/∂B 的每一项都乘到 0,故为 0;Bᵀ≡0 同理使 ∂L/∂A=0。)

**两句话总结**:
- 初始增量侧——让 A、B 至少一个为零,保证 BA=0、训练起点行为 == 预训练模型;
- 梯度流侧——绝不能两个都为零,否则 ∂L/∂A、∂L/∂B 在零点同时消失、参数永不更新;"一随机一为零"既满足前者又保证后者(被置零的那个能立即从非零的另一个那里拿到梯度)。

### 练习 3:验证合并的等价性(编码题)

下面给出可直接运行的测试,基于第一节的 `LoRALinear`。CPU、小维度即可跑;依赖 `torch`(`pip install torch`,无需 GPU)。要点:**训练后把 B 设成非零**,否则初始 B=0 会让 ΔW≡0、合并前后平凡相等,测不出真正等价性。

```python
import math
import torch
import torch.nn as nn

# ---- 第一节的 LoRALinear(原样搬入,便于自包含运行)----
class LoRALinear(nn.Module):
    def __init__(self, base_linear: nn.Linear, r: int, alpha: int, dropout: float = 0.0):
        super().__init__()
        self.base = base_linear
        for p in self.base.parameters():
            p.requires_grad = False
        d_out, d_in = base_linear.weight.shape
        self.r = r
        self.scaling = alpha / r
        self.lora_dropout = nn.Dropout(dropout)
        self.A = nn.Parameter(torch.empty(r, d_in))
        self.B = nn.Parameter(torch.zeros(d_out, r))
        nn.init.kaiming_uniform_(self.A, a=math.sqrt(5))
        self.merged = False

    def forward(self, x):
        base_out = self.base(x)
        if self.merged:                       # 已合并:增量已在 base 权重里
            return base_out
        lora_out = self.lora_dropout(x) @ self.A.t() @ self.B.t()
        return base_out + self.scaling * lora_out

    @torch.no_grad()
    def merge(self):
        if self.merged:
            return
        delta = self.scaling * (self.B @ self.A)
        self.base.weight.add_(delta)
        self.merged = True

    @torch.no_grad()
    def unmerge(self):
        if not self.merged:
            return
        delta = self.scaling * (self.B @ self.A)
        self.base.weight.sub_(delta)
        self.merged = False


# ---- 测试 ----
def test_merge_equivalence():
    torch.manual_seed(0)
    base = nn.Linear(16, 8)                    # 小维度,CPU 秒级
    layer = LoRALinear(base, r=4, alpha=8, dropout=0.0)  # 测等价性必须关 dropout!

    # 关键:模拟"训练后" B 非零,否则 ΔW=0 时合并前后平凡相等,测不出东西
    with torch.no_grad():
        layer.B.copy_(torch.randn_like(layer.B) * 0.1)

    x = torch.randn(2, 5, 16)                  # (batch, seq, d_in)

    # 1) 合并前权重 clone 一份,用于稍后验证 unmerge 能复原
    w_before = base.weight.detach().clone()

    # 2) 未合并前向
    layer.eval()                               # 确保 dropout 等推理态(此处已设 0,稳妥起见)
    out1 = layer(x)

    # 3) 合并后前向,断言数值等价(atol 给浮点误差留余量)
    layer.merge()
    out2 = layer(x)
    assert torch.allclose(out1, out2, atol=1e-5), \
        f"merge 改变了输出! max diff = {(out1 - out2).abs().max().item()}"

    # 4) unmerge 后断言权重精确复原到合并前
    layer.unmerge()
    assert torch.allclose(w_before, base.weight, atol=1e-6), \
        "unmerge 未能复原原始权重!"

    print("PASS: 合并前后输出等价,且 unmerge 复原权重。",
          "max(out1-out2) =", (out1 - out2).abs().max().item())


if __name__ == "__main__":
    test_merge_equivalence()
```

运行预期输出类似:`PASS: ... max(out1-out2) = 1e-7 量级`。(本环境无 torch,故用 numpy 复算了同一套合并数学:`x@W0.T+b + s·(x@A.T@B.T)` 与 `x@(W0+s·B@A).T+b` 的最大差为 ~2.2×10⁻¹⁶,即机器精度,等价性成立。)

**为什么这个等价性对"推理零延迟"主张是必要的:**

LoRA 未合并时,前向要算两条支路再相加:`base(x)` 一次矩阵乘,外加 `(α/r)·B·(A·x)` 两次小矩阵乘——比原始线性层多了算子和访存。**只有当 `merge()` 在数学上严格等价于"把 (α/r)BA 一次性加进 W₀",合并后的层才退化成一个普普通通的 d×k 线性层 `h = W_merged·x`**,推理图里和没用过 LoRA 的原模型一模一样:同样的算子数、同样的访存、零额外开销。

换句话说:**等价性是"零延迟"的充要前提**。若合并改变了输出(等价性不成立),要么得在推理时保留两条支路(于是有额外延迟,零延迟主张破产),要么合并就是错的。测试里 `allclose` 的 `atol` 只是为浮点累加误差留的容差,不影响"代数上完全等价"这一结论。这也正是 LoRA 区别于 Adapter 的根本——Adapter 中间夹了非线性 f,`W_up·f(W_down·h)` 无法被吸收进任何原权重矩阵,**不可合并 ⇒ 推理永远多一层 ⇒ 有不可消除的延迟**。

### 练习 4:自回归生成服务的延迟/质量选型(分析题)

**场景判别要点**(逐 token 生成、强延迟敏感、需在 50 个微调版本间切换):自回归推理每生成一个 token 都要重跑一遍前向,本就受访存和小算子拖累,所以**任何"每步都多算的额外算子"或"占用上下文/KV cache 的额外 token"都会被放大 50 步、上百步地累计**;同时 50 个版本要求"切换/共存"的代价必须低。

**(a) 单请求推理延迟排序(从低到高,越低越好)**

```
QLoRA(合并后) ≈ LoRA(合并后) = 0 额外延迟(退化成普通线性层)
  < QLoRA(不合并,需逐层反量化基座) 略有反量化开销
  < Prefix-tuning(每层 K/V 多 p 个前缀 token,每步注意力都要算,且吃 KV cache)
  < Adapter(每个子层后多一层带非线性的前向,不可消,逐 token 放大)
```

- **LoRA**:合并后零额外延迟(练习 3 已证等价);即便不合并,旁路也只是两次小矩阵乘,远轻于 Adapter 的串行新层。
- **QLoRA**:本质是 LoRA + 4-bit 基座。若为单任务合并,可逼近零延迟;但多任务场景一般不合并,且每层前向要把基座 4-bit 现场反量化回 BF16,有**持续的反量化开销**——自回归下每步都付一次,排在 LoRA 之后。
- **Prefix-tuning**:那 p 个虚拟前缀进入每层注意力,**每生成一步都参与 attention 且占 KV cache**,延迟与显存双重持续开销。
- **Adapter**:不可合并,每子层多一遍 `W_up·f(W_down·h)`,在小算子密集的解码阶段尤其刺眼,延迟最差。

**(b) 多任务切换成本排序(从低到高)**

```
LoRA / QLoRA(不合并,动态挂 (A,B))< Prefix-tuning < Adapter
```

- **LoRA/QLoRA(不合并模式)最优**:基座(QLoRA 下还是 4-bit,更省)常驻一份,每个版本只是一组几 MB 级的 {A,B}。50 个版本 = 1 份基座 + 50 份几 MB adapter,切换≈换一组小张量;工业上 vLLM multi-LoRA / S-LoRA 还能在**同一个 batch 里混合不同 adapter**,基座共享、各请求挂各自 (A,B) 用分组矩阵乘并行算——这是为本场景量身定做的能力。
- **Prefix-tuning**:每任务一段 prefix,主干共享,切换成本也低,但每段 prefix 持续占上下文与 KV cache,且容量/效果在中小模型上常不如 LoRA。
- **Adapter**:每个版本是一组带非线性的新层权重,既不能合并也不能像 LoRA 那样在一个 batch 里廉价并行混合,50 版本的共存/切换最重。

**(c) 显存占用排序(从低到高)**

```
QLoRA(4-bit 基座)< LoRA ≈ Adapter ≈ Prefix(三者基座都是 fp16/bf16)
```

- **QLoRA 显存最优**:基座 NF4 压到约 1/4(7B:14GB→~3.5GB;65B:130GB→~32.5GB),这是它唯一也是最大的杀手锏。
- **LoRA / Adapter / Prefix**:基座都按全精度(fp16/bf16)常驻,基座显存相同;三者的"额外参数"都极小,差异主要在激活与额外结构,量级相近。Prefix 因占 KV cache,在长序列/多版本下显存略劣于 LoRA。

**最终选型结论:默认 LoRA(以"不合并、动态挂 adapter"模式部署);若 50 个版本的基座在目标卡上装不下,则上 QLoRA。**

理由综合三维:本场景**强延迟敏感**直接淘汰 Adapter(不可消延迟)和 Prefix(占窗口 + 每步参与注意力);剩下 LoRA 与 QLoRA。

- 若显存够放下一份 fp16 基座 → **选 LoRA 不合并模式**:延迟维度最优(切到的 adapter 几 MB,甚至可在一个 batch 里混合 50 个版本并行服务,multi-LoRA),多任务切换成本最低,延迟接近零。这是延迟敏感 + 多版本切换场景的事实标准答案。
- 若基座本身装不下(如 50 个版本要并发、或基座是 30B/65B 量级单卡放不下)→ **选 QLoRA**:用基座 4-bit 换来"单卡装得下"的可行性,代价是每步多一次反量化(延迟略升,但远小于 Adapter/Prefix 的额外开销),且同样支持几 MB 级多 adapter 切换。

一句话:**延迟敏感把候选锁死在"可合并/可廉价并行"的 LoRA 系;显存是否够放基座决定 LoRA(够)还是 QLoRA(不够)。**
