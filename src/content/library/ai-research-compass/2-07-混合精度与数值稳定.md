## 为什么这一章绕不过去

第 01 章的显存账本里有一行你可能没多想:**混合精度 + Adam 下每个参数约 16 字节 = fp16 参数 2 + fp16 梯度 2 + fp32 参数主副本 4 + Adam 一阶动量 4 + 二阶动量 4**。注意这里同时出现了 fp16 参数和 fp32 参数——同一份权重存了两遍,精度还不一样。这不是冗余,是混合精度训练能不崩的核心。

低精度训练的诱惑非常直接:把激活、权重、梯度从 32 位换成 16 位,显存立刻减半,而 Tensor Core 对 16 位矩阵乘的吞吐通常是 fp32 的数倍(具体倍数与代上 Tensor Core 规格有关,**待核**)。GEMM 是训练里最重的算子(第 01 章 C ≈ 6ND),把它的输入精度砍半,理论上是几乎免费的两倍加速。

但"几乎免费"里藏着一个会让你整夜 debug 的陷阱:**fp16 的可表示数值范围窄到放不下一部分真实的梯度**。梯度一旦掉进表示不了的区间就被归零,反向传播的信息凭空消失,训练发散成 NaN。本章要做的,就是把这个陷阱的每个齿轮拆开:浮点数到底怎么编码、窄在哪、窄了会发生什么、三种补救(loss scaling / fp32 主副本 / 换 bf16)各自在数学上解决了哪一环、以及把精度推到 fp8 后又冒出什么新问题。

**贯穿全章的一个判断标准**:一个数值技巧是不是必需,要看去掉它之后是否有信息在浮点舍入中不可逆地丢失。能丢信息的就是必需品,只是慢一点的不算。

## 一、浮点数的位布局:动态范围与精度是一对此消彼长的资源

### 问题:32 位、16 位、8 位,这些位到底在编码什么

IEEE 754 浮点数把一个有限的位预算分成三段:1 位符号 s、e 位指数(exponent)、m 位尾数(mantissa,也叫尾数域/fraction)。一个正规数(normal number)的值是:

```
value = (-1)^s × (1 + frac) × 2^(E - bias)

其中:
  frac = 尾数域当作小数,范围 [0, 1)        ← m 位,决定"精度"(同一量级内能分多细)
  E    = 指数域当作无符号整数,范围 [1, 2^e - 2]  ← 全 0 和全 1 留作特殊值
  bias = 2^(e-1) - 1                          ← 让指数能表示正负
  (1 + frac) 里的那个隐含 1 = 正规数省下的一位 ← 因为二进制科学计数法首位必为 1
```

两个直觉,记牢:

- **指数位 e 决定动态范围**(dynamic range):能表示的最大数和最小正数之比。每多 1 位指数,可表示的量级范围大致翻倍(指数能取的范围翻倍)。
- **尾数位 m 决定精度**(precision):在任何一个固定量级内,相邻可表示数之间的相对间隔约为 2⁻ᵐ。这个量叫**机器精度 ε(machine epsilon)= 2⁻ᵐ**,意思是"1.0 之后下一个能表示的数是 1 + 2⁻ᵐ"。

这两者在固定位预算下是**零和博弈**:位都用来扩指数,就没位留给尾数;反之亦然。下表是四种格式的拆分(数值已用 Python 验算):

| 格式 | 总位 | 符号 | 指数 e | 尾数 m | bias | 机器精度 ε=2⁻ᵐ | 最小正规数 | 最大数 |
|------|------|------|--------|--------|------|------------------|-------------|--------|
| FP32 | 32 | 1 | 8 | 23 | 127 | 1.19e-7 | 2⁻¹²⁶ ≈ 1.18e-38 | ≈ 3.40e38 |
| FP16 (half) | 16 | 1 | 5 | 10 | 15 | 9.77e-4 | 2⁻¹⁴ ≈ 6.10e-5 | 65504 |
| BF16 (bfloat16) | 16 | 1 | 8 | 7 | 127 | 7.81e-3 | 2⁻¹²⁶ ≈ 1.18e-38 | ≈ 3.39e38 |
| FP8 E4M3 | 8 | 1 | 4 | 3 | 7 | 0.125 | 2⁻⁶ ≈ 1.56e-2 | 448 |
| FP8 E5M2 | 8 | 1 | 5 | 2 | 15 | 0.25 | 2⁻¹⁴ ≈ 6.10e-5 | 57344 |

读这张表的正确姿势,是横向对比 FP16 和 BF16 这一对——它们位数完全相同(都 16 位),却做了相反的取舍:

- **FP16 把位偏向尾数**(10 位尾数、5 位指数):精度好(ε≈9.77e-4),但指数只有 5 位,动态范围窄到最小正规数才 6.1e-5、最大数才 65504。
- **BF16 把位偏向指数**(7 位尾数、8 位指数):指数位与 fp32 完全一样,动态范围和 fp32 一模一样(最小正规数 1.18e-38);代价是尾数只剩 7 位,精度差(ε≈7.81e-3,比 fp16 差约 8 倍)。

**这个取舍差异是本章后半部分所有结论的根源**:bf16 几乎免 loss scaling、fp16 必须 loss scaling,直接来自它们指数位的多寡。

### 推导:fp16 的最小正规数与"为什么是 6.1e-5"

这是必须会手推的基本功。fp16 指数域 e=5,bias = 2⁵⁻¹ - 1 = 15。指数域全 0(E=0)留给次正规数(subnormal),所以最小的正规数对应 E=1:

```
最小正规数 = (1 + 0) × 2^(E - bias)
           = 1 × 2^(1 - 15)
           = 2^-14
           = 6.103515625e-5      ✓ 与 Python 验算一致
```

低于这个值,fp16 进入**次正规数(subnormal)**区间。次正规数放弃了隐含的前导 1(变成 0.frac × 2^(1-bias)),用牺牲精度来表示更小的数,一直到最小的非零值 2⁻²⁴ ≈ 5.96e-8。但次正规数的精度极差(尾数能表示的有效位越来越少),而且很多硬件对次正规数有 flush-to-zero(FTZ)模式直接把它当 0。**所以工程上把 fp16 的"安全下界"看作约 6e-5,而不是 6e-8。**

对照 bf16:指数 8 位、bias=127,最小正规数 = 2^(1-127) = 2⁻¹²⁶ ≈ 1.18e-38。**fp16 的下界比 bf16 高了约 33 个数量级**——这就是"窄"的精确含义。

### 代码:用整数位运算从零解码一个浮点数

把上面的公式写成代码,你就再也不会对浮点格式有神秘感:

```python
import struct

def decode_fp32(x: float):
    """把一个 Python float 按 IEEE754 单精度拆成 (符号, 指数域E, 尾数域bits, 还原值)"""
    bits = struct.unpack('>I', struct.pack('>f', x))[0]   # 取 32 位整数表示
    s    = (bits >> 31) & 0x1
    E    = (bits >> 23) & 0xFF        # 8 位指数域
    frac = bits & 0x7FFFFF            # 23 位尾数域
    bias = 127
    if E == 0:                        # 次正规数:无隐含 1,指数固定为 1-bias
        value = (-1)**s * (frac / 2**23) * 2**(1 - bias)
    elif E == 0xFF:                   # 全 1:inf 或 nan
        value = float('inf') if frac == 0 else float('nan')
    else:                             # 正规数:有隐含 1
        value = (-1)**s * (1 + frac / 2**23) * 2**(E - bias)
    return s, E, frac, value

# 验证机器精度:1.0 的下一个可表示数应是 1 + 2^-23
print(decode_fp32(1.0))                       # E=127, frac=0, value=1.0
print(struct.unpack('>I', struct.pack('>f', 1.0 + 2**-23))[0]
      - struct.unpack('>I', struct.pack('>f', 1.0))[0])   # 输出 1:整数位恰好差 1
```

最后那行揭示了浮点的一个关键性质:**相邻可表示浮点数的整数位表示相差 1**(在同一指数区间内)。1.0 之后的下一个数,其位模式恰好 +1,对应数值 +2⁻²³ = ε。这正是"机器精度"的操作性定义。

## 二、fp16 为什么会崩:小梯度下溢归零导致训练发散

### 问题:训练里有多少梯度落在 fp16 表示不了的区间

把视角从"格式"切到"训练动态"。深度网络反向传播时,梯度的量级跨度极大。靠近损失的层梯度大,深处的层梯度经过多层链式相乘后可能变得很小;不同参数的梯度也分布在好几个数量级上。**经验事实是:训练中后期,相当一部分梯度的绝对值落在 1e-4 ~ 1e-7 这个区间**(具体分布因模型/阶段而异,**待核**,但量级是真实的)。

现在对照 fp16 的安全下界 6.1e-5。一个量级 1e-7 的梯度,在 fp16 里会发生什么?

```python
import numpy as np
g = 1e-7
print(float(np.float16(g)))     # 1.1920928955078125e-07
```

它没有直接归零(因为 numpy 用了次正规数),但已经掉进次正规区——有效精度只剩一两位,而且在开启 FTZ 的硬件上会直接变成 0.0。**一旦梯度变成 0,这条反向路径上的信息就永久消失了**:对应的权重不再更新,如果大量梯度同时下溢,整个反向传播退化成噪声,loss 不降反升,最终 NaN。这就是"fp16 动态范围窄 → 小梯度下溢归零 → 训练发散"的完整因果链。

注意问题出在**下溢(underflow)而非溢出(overflow)**。fp16 上界 65504 通常够用(激活/梯度很少超过几百),真正杀人的是下界。这个观察直接决定了补救方向:**把整条梯度分布往上平移,让小梯度离开下溢区**。

### 推导:loss scaling 的数学——一个常数把梯度抬进可表示区间

补救手段叫 **loss scaling(损失缩放)**,机制简单到优雅。设原始损失为 L,我们在反向传播前把它乘以一个大常数 S(scale factor,如 1024 或 65536):

```
缩放后损失:  L' = S · L

反向传播对任意参数 θ 的梯度(链式法则,S 是常数可提出):
  ∂L'/∂θ = ∂(S·L)/∂θ = S · (∂L/∂θ)

→ 整张计算图里每个梯度都被同比放大了 S 倍
```

关键在于:**这个放大发生在梯度被写进 fp16 之前**。原本量级 1e-7 的梯度会下溢,乘 S=1024 后变成 1e-7 × 1024 ≈ 1e-4,落进 fp16 正规区,精度完好保留:

```python
g = 1e-7
print(g * 1024)                       # 0.0001024
print(float(np.float16(g * 1024)))    # 0.0001024007797241211  ✓ 没有信息丢失
print(float(np.float16(g * 65536)))   # 0.0065536...           更安全
```

放大后梯度照常以 fp16 存储和 all-reduce(数据并行下,见第 02 章)。**最后一步**:在用梯度更新权重之前,必须把这个放大撤销——梯度除以 S 还原回真实量级,否则等价于把学习率偷偷放大了 S 倍,训练同样会炸:

```
真实梯度 = (fp16 缩放梯度) / S      ← 在 fp32 主副本上除回
然后:  θ ← θ - lr · (真实梯度)
```

数学上,loss scaling 是一个**完全可逆的线性变换**:乘 S 再除 S,理论上不改变任何东西。它唯一的作用是改变中间存储 fp16 时的数值落点,把梯度从下溢的危险区平移到精度安全区。这就是为什么它"必需但纯属数值工程"——它不改变算法语义,只改变浮点落点。

### 静态 vs 动态 loss scaling:S 该取多大

S 太小,小梯度仍下溢;S 太大,大梯度乘上去超过 65504 溢出成 inf,反向同样炸。所以 S 的选择本身是个权衡。

- **静态(static)scaling**:全程固定一个 S(如 1024)。简单,但需要人工调:不同模型、不同阶段的梯度分布不同,一个固定值很难全程合适。
- **动态(dynamic)scaling**:自动找当前能用的最大安全 S。机制是一个简单的反馈控制循环:

```
初始化 S 为一个较大值(如 2^16)
每个 step:
  1. 用当前 S 做缩放反向,得到 fp16 梯度
  2. 检查梯度里是否出现 inf / nan(说明 S 太大、溢出了)
  3. 若出现 → 跳过本次参数更新(这一步的梯度作废),并把 S 减半(S ← S/2)
  4. 若连续 N 步(如 2000)都没溢出 → 说明 S 偏保守,把 S 翻倍(S ← S·2)往上试探
```

这个循环的直觉:**S 沿着"溢出边界"自动游走**,尽量取大(让小梯度尽量安全)又不越界(不让大梯度溢出)。代价是偶尔会浪费一步(检测到 inf 时丢弃该 step 的梯度),以及每步要做一次全梯度的 inf/nan 检查(一次 reduce,开销很小)。PyTorch 的 `torch.cuda.amp.GradScaler` 实现的就是这套动态逻辑,后面会贴用法。

## 三、为什么必须保留 fp32 参数主副本:小更新量被大权重"吃掉"

### 问题:即使梯度不下溢,更新也可能凭空消失

loss scaling 解决了"梯度太小存不进 fp16"。但还有第二个独立的崩溃源,它发生在**参数更新这一步**,和梯度下溢无关——就算梯度量级正常,问题照样存在。

考虑用 fp16 直接存权重并更新。典型情况下权重量级在 O(1),而单步的更新量 lr · grad 量级常在 1e-3 ~ 1e-7(学习率小、训练中后期梯度小)。问题是:**把一个很小的数加到一个大很多的数上,在 fp16 里小数会被舍入吃掉**。

直接看实验,触目惊心:

```python
import numpy as np

# fp16 下:1.0 加上 1e-4
w = np.float16(1.0)
print(float(np.float16(w + np.float16(1e-4))))   # 1.0  —— 更新完全消失!

# 累加 1000 次 +1e-4,fp16 vs fp32
w16 = np.float16(1.0)
for _ in range(1000):
    w16 = np.float16(w16 + np.float16(1e-4))
print(float(w16))   # 1.0     —— 1000 次更新一次都没生效

w32 = np.float32(1.0)
for _ in range(1000):
    w32 = np.float32(w32 + np.float32(1e-4))
print(float(w32))   # 1.1000...  —— fp32 正确累积
```

**fp16 里累加 1000 次 1e-4,权重纹丝不动停在 1.0;fp32 正确到达 1.1。** 这就是著名的"**吞没(swamping)**"现象——小加数被大被加数完全吸收。

### 推导:用机器精度算清楚"被吃掉"的边界

为什么是这个结果?用机器精度可以精确预测。在数值 x 附近,fp16 可表示数的间隔(ULP,unit in the last place)是:

```
ULP(x) ≈ |x| · ε = |x| · 2⁻ᵐ        (m=10 for fp16)

在 x = 1.0 处:
  ULP(1.0) = 1.0 × 2⁻¹⁰ ≈ 9.77e-4
```

浮点加法 fl(w + δ) 的结果会被舍入到离真实和最近的可表示数。当更新量 δ 小于半个 ULP 时,真实和 w+δ 离 w 比离 w 的下一个邻居更近,**舍到最近就舍回了 w 本身**:

```
吞没条件:  |δ| < ½ · ULP(w) = ½ · |w| · 2⁻ᵐ

fp16, w=1.0:  阈值 = ½ × 9.77e-4 ≈ 4.88e-4
              更新量 1e-4 < 4.88e-4  →  必然被吃掉   ✓ 与实验一致
```

把它写成一个干净的判据:**当 |更新量 / 权重| < ½ · 2⁻ᵐ 时,这次更新在该精度下不可见。** 对 fp16(m=10),这个相对阈值约 4.9e-4;对 bf16(m=7),约 3.9e-3——bf16 反而更糟(尾数更少、精度更差),所以 **bf16 也救不了主副本问题,这是一个独立于动态范围的问题**。

### 机制:fp32 主副本如何根治吞没

解法是**保留一份 fp32 的参数主副本(master weights)**。更新这一步在 fp32 里做,fp16 只用于前向/反向的算力密集计算。完整的一个 step 数据流:

```
  fp32 主副本 W32  ──(每步开头复制并降精度)──>  fp16 副本 W16
                                                      │
                                                      ▼
                            前向(fp16 GEMM,走 Tensor Core)→ 损失 L
                                                      │ ×S (loss scaling)
                                                      ▼
                            反向(fp16)→ fp16 缩放梯度 g16
                                                      │ ÷S,升精度到 fp32
                                                      ▼
                            优化器在 fp32 里更新:  W32 ← W32 - lr · g32
                                                      │
                          (Adam 的动量 m、v 也存 fp32,见下)
                          下一步回到顶部,把更新后的 W32 再降精度成 W16
```

为什么这样就对了?因为 **W32 的尾数有 23 位,ULP(1.0)=2⁻²³≈1.19e-7,远小于更新量 1e-4**——吞没条件 |δ| < ½·2⁻²³ 几乎永远不成立,小更新被忠实累积。fp16 副本每步从 fp32 重新生成,它承受的舍入误差不会跨步累积(因为下一步又从精确的 fp32 重算),所以只损失单步前向/反向的精度,不损失"权重的历史"。

**这就回答了第 01 章那行账**:每参数 16 字节里,fp16 参数(2B)是给 GEMM 用的快速副本,fp32 参数(4B)是防吞没的精确主副本,两者缺一不可。再加上 Adam 的一阶动量 m 和二阶动量 v——**这两个状态也必须存 fp32**:动量是梯度的指数滑动平均,每步加一个小量进去,同样会被吞没,逻辑和主副本一模一样。所以 16 字节 = 2(fp16 参数) + 2(fp16 梯度) + 4(fp32 主副本) + 4(fp32 动量 m) + 4(fp32 方差 v)。

> **关键澄清(一个常见误解)**:很多人以为混合精度的代价就是"精度低一点"。其实不然——fp32 主副本 + fp32 优化器状态保证了**权重和优化器状态全程是 fp32 精度的**,低精度只发生在前向/反向的 GEMM 输入。混合精度真正省的是激活显存和算力,**不是**优化器状态显存(那部分反而因为多存了 fp16 副本略有增加)。省显存的大头在激活(第 08 章重计算)和分片(第 05 章 ZeRO)。

## 四、BF16:用更宽的指数换尾数,基本免掉 loss scaling

### 推导:bf16 为什么不需要 loss scaling

现在回到第一节那张表里 fp16 vs bf16 的取舍,把它和前两节的崩溃源对上。

loss scaling 存在的唯一理由,是 **fp16 动态范围窄,小梯度会下溢到 6.1e-5 以下**。bf16 的指数位是 8 位,和 fp32 完全一样,最小正规数 2⁻¹²⁶ ≈ 1.18e-38。一个量级 1e-7 甚至 1e-20 的梯度,在 bf16 里都还在正规区,根本不会下溢:

```
fp16 安全下界:  2⁻¹⁴ ≈ 6.10e-5      ← 梯度 1e-7 落进次正规/归零区,需 loss scaling 抬上来
bf16 安全下界:  2⁻¹²⁶ ≈ 1.18e-38    ← 梯度 1e-7 离下界还有 31 个数量级,毫无压力

结论:bf16 的动态范围 = fp32 的动态范围 → 训练中的梯度几乎不可能下溢 → 不需要 loss scaling
```

**这就是"bf16 用更宽指数换尾数 → 基本免 loss scaling"的完整推导**。它不是经验观察,是从指数位数直接推出的:动态范围由指数位决定,bf16 指数位 = fp32 指数位,所以动态范围相同,所以没有 fp16 的下溢问题,所以那个专为对付下溢而生的 loss scaling 也就不需要了。

### bf16 的代价:精度差了 8 倍,谁来承担

天下没有免费的午餐。bf16 只有 7 位尾数,ε≈7.81e-3,比 fp16 的 ε≈9.77e-4 差约 8 倍。这意味着:

- **GEMM 累加更容易丢精度**。解法是硬件层面的:Tensor Core 做 bf16 矩阵乘时,乘积的**累加在 fp32 寄存器里进行**(accumulate in fp32),只有输入是 bf16。所以一长串乘加的累加误差是 fp32 级别的,bf16 的低精度只影响单个输入的表示,不影响累加链。这是 bf16 能用的关键硬件配合(**fp16 的 Tensor Core 也是 fp32 累加**,这点两者一致)。
- **fp32 主副本依然需要**。如上一节所证,主副本针对的是吞没(精度问题),bf16 尾数更少精度更差,吞没只会更严重。所以**换 bf16 免掉的是 loss scaling,免不掉 fp32 主副本**。两个崩溃源(下溢/吞没)是正交的,bf16 只关掉了下溢那个。

### 为什么现代大模型训练几乎全面转向 bf16

权衡下来,bf16 在大规模训练上几乎完胜 fp16:

1. **省心**:不需要调 loss scaling,没有"S 设多大"的玄学,没有动态 scaling 偶尔丢 step 的开销,训练曲线更稳。
2. **更鲁棒**:大模型训练中偶发的激活/梯度尖峰(spike)在 fp16 里可能溢出 65504 直接 NaN,在 bf16 里有 fp32 级的上界(3e38)兜底,不会因为一个尖峰炸掉。
3. **尾数少 3 位的代价在实践中可接受**:因为累加用 fp32、主副本用 fp32,bf16 的低精度只作用在 GEMM 输入这一处,对最终收敛影响有限(具体到不同任务的收敛差异 **待核**,但工业界共识是 bf16 训 LLM 收敛质量与 fp16 相当甚至更稳)。

代价是 bf16 需要硬件支持(Ampere/A100 及之后的 NVIDIA GPU、TPU 原生支持;更老的硬件如 V100 只有 fp16),所以"用 fp16 还是 bf16"有时是被硬件决定的,不全是自由选择。

**一句话记忆**:fp16 = 精度够、范围险(必须 loss scaling);bf16 = 范围稳、精度糙(免 scaling、靠 fp32 累加补精度)。有 bf16 硬件就用 bf16。

## 五、AMP 实战:autocast 与 GradScaler 怎么配合

把前四节落到 PyTorch 的自动混合精度(AMP,Automatic Mixed Precision)。AMP 由两个组件构成,各管一件事:

- **`autocast`**:管"哪个算子用什么精度跑"。它在前向里自动把适合低精度的算子(主要是 GEMM/卷积这类 Tensor Core 友好、对精度不敏感的)切到 fp16/bf16,把对精度敏感的算子(如 softmax、layernorm、loss、各种 reduction/求和)保留在 fp32。这个"白名单/黑名单"是 AMP 内置的,背后逻辑就是第三节的吞没原理——**累加型算子留 fp32,矩阵乘型算子转低精度**。
- **`GradScaler`**:管 loss scaling(第二节的动态 scaling)。**只在 fp16 下需要;bf16 下不需要 GradScaler**(因为 bf16 免 scaling,第四节)。

```python
import torch
from torch.cuda.amp import autocast, GradScaler

model = ...      # 参数本身是 fp32(这就是 fp32 主副本!优化器在 fp32 参数上更新)
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

# ---- fp16 路径:需要 GradScaler 做动态 loss scaling ----
scaler = GradScaler()   # 内部维护动态 scale,初值通常 2^16,溢出则减半
for x, y in loader:
    optimizer.zero_grad()
    with autocast(dtype=torch.float16):       # 前向:GEMM 走 fp16,softmax/loss 留 fp32
        loss = loss_fn(model(x), y)           # loss 是 fp32(autocast 把它留在 fp32)
    scaler.scale(loss).backward()             # ① loss × S 再反向 → 梯度被放大 S 倍存进 fp16
    scaler.step(optimizer)                    # ② 内部先把梯度 ÷S 还原(unscale),
                                              #    检查 inf/nan;有则跳过本 step,无则在 fp32 主副本上更新
    scaler.update()                           # ③ 根据本步是否溢出调整 S(溢出减半 / 连续 N 步无溢出翻倍)

# ---- bf16 路径:动态范围够,不需要 GradScaler,代码更干净 ----
for x, y in loader:
    optimizer.zero_grad()
    with autocast(dtype=torch.bfloat16):      # GEMM 走 bf16,累加在 fp32 寄存器
        loss = loss_fn(model(x), y)
    loss.backward()                            # 直接反向,无需 scale/unscale
    optimizer.step()                           # 优化器照常在 fp32 参数上更新
```

把 `GradScaler` 的三步对回第二节的动态 scaling 循环:`scale(loss).backward()` 是"乘 S 反向",`step` 里隐含"÷S → 查 inf/nan → 决定更不更新",`update` 是"调整 S"。**`step` 里那个"有 inf 就跳过更新"非常重要**——它保证溢出的那一步梯度不会污染 fp32 主副本。如果你手写训练循环忘了用 `scaler.step` 而直接 `optimizer.step`,等于跳过了 unscale,学习率被偷偷放大 S 倍,训练秒炸。这是新手最常踩的坑之一。

> **一个排错心法**:fp16 训练出现 NaN,先看是不是 GradScaler 的 S 已经被一路减半到极小(说明持续溢出,可能是模型本身数值问题或学习率过大),或者你漏了 scaler 的某一步。**bf16 训练出现 NaN,通常不是精度问题**(范围够),要去查数据(脏样本)、学习率、或某个自定义算子里的除零/log(0)。把"是 fp16 还是 bf16"作为分诊的第一个分叉,能省大量时间。

## 六、FP8:把精度推到 8 位,per-tensor / per-block scaling 是生命线

### 问题:8 位还剩多少表示能力

继续往下推到 fp8——H100(Hopper)及之后的硬件原生支持,是当前训练/推理提速的前沿。8 位的位预算极度紧张,只有两种常用布局(见第一节表):

- **E4M3**(4 指数 + 3 尾数):精度相对好(ε=2⁻³=0.125,即相邻数相对间隔高达 12.5%),范围窄(最大 448,最小正规数 1.56e-2)。**用于前向**(激活、权重)——前向的值分布相对集中,要精度。
- **E5M2**(5 指数 + 2 尾数):范围宽(最大 57344,和 fp16 同级),精度更糙(ε=2⁻²=0.25)。**用于反向**(梯度)——梯度动态范围大,要范围。

这个"前向 E4M3、反向 E5M2"的分工,本质是把第一节"指数换尾数"的取舍按用途拆到两种 fp8:前向重精度选尾数多的,反向重范围选指数多的。

关键认知:**fp8 的精度差到不能裸用**。E4M3 的相对间隔 12.5% 意味着,如果一个张量里数值跨度超过它那点动态范围(最大才 448),要么大值溢出、要么小值下溢到 0,二者必居其一。fp8 的可表示窗口太窄,**几乎任何真实张量直接塞进 fp8 都会有大批元素溢出或下溢**。

### 机制:per-tensor / per-block scaling——给每块数据配一个专属的 fp32 缩放因子

解法是 loss scaling 思想的推广,但更精细。核心:**fp8 张量不单独存,而是存成"fp8 数据 + 一个高精度(fp32)缩放因子 scale"的组合**。用的时候真实值 = fp8 值 × scale。

```
存储:   x_fp8 = round_to_fp8( x_real / scale )      其中 scale 是 fp32 标量
读取:   x_real ≈ x_fp8 × scale

scale 的选法:让张量里的最大绝对值 amax 刚好落到 fp8 的可表示上界附近
  scale = amax(x_real) / FP8_MAX        (FP8_MAX = 448 for E4M3)
→ 这样整个张量被"拉伸/压缩"到正好填满 fp8 的窗口,既不溢出也尽量少下溢
```

按 scale 的粒度分几个档次,粒度越细越能贴合局部数值分布、越能容忍异常值,但元数据开销越大:

- **per-tensor scaling**:整个张量共用一个 scale。开销最小(一个张量一个 fp32 标量),但如果张量里有异常值(outlier),amax 被异常值拉大,导致正常值被压到 fp8 窗口底部、下溢精度尽失。这正是第 12 章会讲的"异常值是低精度的死敌"在 fp8 训练里的体现。
- **per-block / per-tile scaling**:把张量切成小块(如每 128 个元素、或 GEMM 的每个 tile),每块一个独立 scale。异常值的影响被局限在它所在的小块里,其他块不受连累,数值稳健性大幅提升。代价是元数据多了(块数 × 一个 fp32),以及 GEMM kernel 要按块应用不同 scale(实现复杂度上升)。DeepSeek-V3 等模型采用的细粒度 fp8 训练走的就是 per-block(如 1×128 / 128×128 分块)路线(具体分块尺寸与做法 **待核**)。

### 推导:per-tensor scaling 为什么能救回精度

直觉是"把数据搬到 fp8 精度最好的区间"。fp8 的相对精度 ε 在整个正规区是恒定的(都是 2⁻ᵐ),但绝对的可表示数密度在大数附近稀、在接近下界处会掉进次正规损失有效位。如果不缩放,一个 amax≈0.01 的小张量塞进 E4M3(下界 1.56e-2),几乎全部元素下溢归零。乘上 scale = amax/448 的倒数把它拉伸到填满 [小, 448] 区间后:

```
未缩放:  张量 ∈ [1e-4, 1e-2],  E4M3 下界 1.56e-2  →  几乎全部下溢为 0  ✗
缩放后:  scale = 1e-2 / 448 ≈ 2.2e-5
         张量/scale ∈ [4.5, 448],全部落在 E4M3 正规区,相对精度 12.5% 保留  ✓
读回:    × scale 还原,误差 ≈ 相对 12.5%(fp8 固有,无法消除,只能靠 fp32 累加摊薄)
```

所以 fp8 训练里,**E4M3/E5M2 只承载"被 scale 归一化后的尾数",真正的量级信息存在 fp32 的 scale 里**。这和 fp32 主副本是同一种思想的不同形态:用一个高精度的"锚"(主副本的 fp32 / fp8 的 scale)托住低精度数据,让低精度只负责它能负责的那部分(局部精度 / 归一化后的相对值)。

### 代码:per-tensor fp8 量化的伪逻辑

```python
import torch

FP8_E4M3_MAX = 448.0     # E4M3 可表示最大值(已验算)

def to_fp8_per_tensor(x: torch.Tensor):
    """把 fp32/bf16 张量量化为 (fp8 数据, fp32 scale)"""
    amax = x.abs().max().clamp(min=1e-12)        # 防 0:全零张量给个下限
    scale = amax / FP8_E4M3_MAX                  # fp32 标量:把 amax 映射到 fp8 上界
    x_scaled = x / scale                         # 拉伸到填满 fp8 窗口
    x_fp8 = x_scaled.to(torch.float8_e4m3fn)     # 实际转 fp8(API 名以框架为准,待核)
    return x_fp8, scale

def from_fp8(x_fp8, scale):
    return x_fp8.to(torch.float32) * scale       # 还原:fp8 值 × scale

# fp8 GEMM 的样子(概念):两个输入各自带 scale,累加在 fp32,输出再按需重新量化
def fp8_matmul(a_fp8, sa, b_fp8, sb):
    # 硬件 Tensor Core 实际在 fp8 输入上做乘、在 fp32 累加器里累加
    out_fp32 = (a_fp8.float() @ b_fp8.float())   # 概念示意;真实走 fp8 Tensor Core
    return out_fp32 * (sa * sb)                  # 两个 scale 相乘还原输出量级
```

注意 `fp8_matmul` 里**输出的 scale 是两个输入 scale 的乘积**(因为 (a/sa·sa)·(b/sb·sb) 展开后量级要乘回 sa·sb)——这是 per-tensor scaling 下链式传播 scale 的基本规则。真实框架(如 NVIDIA Transformer Engine)会自动管理这些 scale 的传播和更新,并用一种叫 **delayed scaling(延迟缩放)** 的技术:不实时算 amax(那要先扫一遍数据,有开销),而是用前几步观测到的 amax 历史来预测当前步的 scale,省掉一次扫描(具体窗口大小与策略 **待核**)。

### fp8 的代价与现状

- **稳定性脆弱**:fp8 训练对 scaling 策略、哪些层用 fp8(通常只对大 GEMM 用 fp8,layernorm/softmax/embedding 等保留 bf16/fp32)、amax 的统计方式都很敏感。一处没调好就发散。所以工业界 fp8 训练目前仍是"高收益、高门槛"。
- **不是全网 fp8**:典型做法是混合——主体 GEMM 用 fp8,数值敏感算子用 bf16,主副本/优化器状态仍 fp32。fp8 是在 bf16 混合精度之上再叠一层激进优化,**不是替代**前面所有机制。
- **推理量化是另一条线**:fp8 也广泛用于推理(权重/激活/KV cache 量化),与第 12 章的 INT8/INT4 量化是相邻话题。训练 fp8 难在反向梯度的范围,推理 fp8 没有反向、相对容易。

## 设计权衡与常见坑

把全章的权衡浓缩成一张决策图和一份避坑清单。

**精度选择决策(从上到下问)**:

```
有 bf16 硬件吗(A100/H100/TPU)?
  是 → 用 bf16:免 loss scaling、最省心、对 spike 鲁棒。默认选这个。
  否(如 V100)→ 用 fp16 + 动态 GradScaler。盯紧 scale 是否一路减半(那是发散前兆)。

要再榨算力、且有 H100 + 能投入调试?
  → 在 bf16 基础上对大 GEMM 叠 fp8(per-block scaling 更稳)。否则不要碰 fp8。

无论哪种:fp32 主副本 + fp32 优化器状态永远保留(防吞没,正交于范围问题)。
```

**新手最常踩的坑**:

1. **以为 bf16 也要 loss scaling**。不要给 bf16 套 GradScaler——多余,还可能因 unscale 引入问题。bf16 范围够,免 scaling。
2. **手写循环漏了 unscale**。fp16 下用了 `scaler.scale(loss).backward()` 却直接 `optimizer.step()`,等于学习率被放大 S 倍。必须走 `scaler.step(optimizer)`。
3. **把权重直接存 fp16/bf16 并原地更新**。这会触发吞没,小更新全部消失,训练像没学习一样停滞。优化器必须在 fp32 主副本上更新(AMP 默认就是,但自定义训练时容易搞错)。
4. **梯度裁剪/范数计算在 scaled 梯度上做**。loss scaling 下梯度被放大了 S 倍,在 unscale 之前算梯度范数会得到错误的(放大 S 倍的)值,裁剪阈值全失效。必须先 `scaler.unscale_(optimizer)` 再裁剪。
5. **用 fp16 累加 reduction(求和、mean、norm、softmax 分母)**。这些是吞没和精度灾难的重灾区(很多小数相加,后面的被前面的吃掉)。AMP 的 autocast 默认把它们留 fp32,但自定义 CUDA/Triton kernel 里要自己保证累加用 fp32(回顾第四节:Tensor Core 的 fp32 累加正是这个道理)。
6. **fp8 用 per-tensor 遇到异常值**。一个 outlier 把 amax 拉爆,正常值全下溢。有 outlier 风险就上 per-block。
7. **混淆"省显存"的来源**。混合精度省的是激活+算力,优化器状态显存反而略增(多了 fp16 副本)。想省优化器状态显存要靠 ZeRO(第 05 章)。

## 动手练习

**练习 1(推导题)** 不查表,从位布局推出 fp16 的最小正规数。已知 fp16 是 1 符号 + 5 指数 + 10 尾数。
*提示*:bias = 2^(e-1) - 1;指数域全 0 留给次正规数,所以最小正规数对应指数域 E=1;最小正规数 = 1 × 2^(E - bias)。算完应得 2⁻¹⁴ ≈ 6.10e-5。再算 fp16 的次正规最小值(放弃隐含 1):2^(1-bias) × 2⁻¹⁰ = 2⁻²⁴ ≈ 5.96e-8。

**练习 2(判断题 + 编码题)** 给定三个梯度值:g₁=8e-5、g₂=3e-5、g₃=2e-8。(a) 哪些在 fp16 正规区(≥6.1e-5)、哪些会掉进次正规/归零?(b) 若用 loss scale S=1024,缩放后哪些进入安全区?(c) 写一段代码,用 numpy 把这三个值分别在"无 scaling"和"S=1024"下转成 fp16,打印还原后的相对误差,验证你的判断。
*提示*:fp16 安全下界 6.1e-5。g₁ 安全;g₂、g₃ 不安全。乘 1024 后:g₂→3.07e-2 安全,g₃→2e-5 仍在下界附近(部分进入次正规,精度仍差)——说明对 g₃ 这种极小梯度,S=1024 还不够大,要更大的 S 或换 bf16。

**练习 3(推导题)** 证明"为什么 bf16 免 loss scaling 而 fp16 不免",并说明"为什么换成 bf16 后仍然需要 fp32 主副本"。
*提示*:loss scaling 针对的是动态范围/下溢问题,由指数位决定;bf16 指数 8 位 = fp32,最小正规数 2⁻¹²⁶,梯度不下溢。fp32 主副本针对的是吞没/精度问题,由尾数位决定;bf16 尾数 7 位比 fp16 的 10 位还少,吞没更严重。两个问题正交,bf16 只解决了前者。

**练习 4(估算题)** 一个 7B 参数模型用 bf16 混合精度 + Adam 训练。(a) 按第 01 章每参数 16 字节估算"参数 + 梯度 + 优化器状态"的显存(不含激活)。(b) 如果改成"权重和优化器状态都用 bf16、不要 fp32 主副本"能省多少,为什么实践中不这么做?(c) 用第三节的吞没判据估算:学习率 1e-3、梯度量级 1e-3,权重量级 1.0,在 bf16(ε=2⁻⁷)下单步更新会不会被吞没?
*提示*:(a) 7e9 × 16 B ≈ 112 GB。(b) 去掉 fp32 主副本(4B)+ 假设动量也 bf16……但会触发吞没导致训练失败,省的显存换来不收敛,不值。(c) 更新量 = lr·grad = 1e-3 × 1e-3 = 1e-6;阈值 = ½ × |w| × 2⁻⁷ = ½ × 1.0 × 7.81e-3 ≈ 3.9e-3;1e-6 ≪ 3.9e-3 → 被吞没。这正是为什么必须 fp32 主副本。

## 小结与承上启下

这一章的内核是**两个正交的崩溃源 + 三件套补救**:

- **下溢**(由指数位/动态范围决定)→ fp16 小梯度归零 → **loss scaling**(静态/动态)抬进可表示区;或者干脆换 **bf16**(宽指数从根上消除下溢,免 scaling)。
- **吞没**(由尾数位/精度决定)→ 小更新加到大权重上被舍掉 → **fp32 主副本 + fp32 优化器状态**,在高精度里累积更新。这个问题 fp16、bf16 都有,bf16 因尾数更少反而更严重。
- 推到 **fp8** 时,可表示窗口太窄,必须用 **per-tensor / per-block scaling**(给每块数据配 fp32 缩放因子)把数据搬进 fp8 的甜区——本质是 loss scaling 思想的精细化推广。
- 一个统一视角:所有这些技巧,都是**用一个高精度的"锚"(fp32 主副本 / loss scale / fp8 的 per-block scale / Tensor Core 的 fp32 累加器)托住低精度数据,让低精度只承担它能承担的那部分**。看懂这一点,fp16/bf16/fp8 的所有设计就不再是零散的 trick,而是同一原理的不同剂量。

**在整门课里的位置**:本章是第 01 章显存账本里"每参数 16 字节"那行的展开证明(为什么要存两份权重),也是后续所有省显存/提速技术的数值地基——第 08 章激活重计算重算的是低精度前向、第 09 章 FlashAttention 的在线 softmax 必须在 fp32 里累加 running max/sum(否则 softmax 数值崩溃,正是本章"累加用 fp32"的直接应用)、第 12 章量化把精度推到 INT8/INT4 时面对的"异常值是死敌"在本章 fp8 的 per-tensor scaling 里已经预演。**精度是贯穿训练和推理的一条暗线:每往下压一个比特,就要多一层 scaling 来托住数值——这章给你的就是看穿这层托举的眼睛。**
