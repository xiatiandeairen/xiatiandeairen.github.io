> 这一章把你从"每个 NLP 任务都要训一个专用模型、设计一个专用损失"的世界,带到"一个通用模型 + 一段提示就能跑大多数任务"的世界——并教你冷静判断:什么时候这是真进步,什么时候是退步。

## 1. 一句话引言

前八章你一路从 n-gram 走到 BERT、GPT、T5,看到的是一个**任务驱动**的 NLP:情感分类有分类头,命名实体识别(NER)有 CRF 解码层,翻译有 seq2seq,摘要有指针生成网络。每个任务有自己的数据格式、自己的损失函数、自己的评测脚本。这一章讲的是一次**坐标系的旋转**:当模型大到一定程度、又被指令微调过之后,所有这些任务在使用层面坍缩成同一个动作——**给模型一段文字,让它续写**。本章不重复 Transformer 内部如何工作(详见算法课 02-05),也不推 in-context learning 与 chain-of-thought 的内部机制(详见算法课 14),更不碰 RLHF 算法(详见强化学习课)。本章只回答一个问题:**这场统一,对"NLP 作为一个任务领域"到底意味着什么,以及作为工程师你该如何决策**。

## 2. 动机:为什么"统一"会发生,而且是必然

### 2.1 经典范式的隐藏成本

先把经典范式的工作流摊开。要上线一个情感分类器,你需要:

```
1. 收集并标注 N 条 (文本, 标签) 数据          ← 人力,数周
2. 选 backbone(BERT-base),加一个分类头        ← 工程
3. 设计损失(交叉熵)、调超参、训练、早停        ← 算力 + 调参
4. 写评测(accuracy / F1)、做错误分析            ← 工程
5. 部署一个专属服务                              ← 运维
```

换一个任务——比如要做 NER——上面**几乎每一步都要重来**:重新标注(标注 schema 完全不同)、换解码层(分类头换成 token 级标注 + CRF)、换损失、换评测(span-level F1)、再起一个服务。十个任务就是十条独立流水线。这里的核心成本不是单条流水线本身,而是**每个新任务的边际成本几乎没有下降**——你做完情感分类,对做 NER 几乎没有复用。

**这才是经典 NLP 真正的痛点:任务之间不共享"能力",只共享"骨架"。** BERT 的预训练让你共享了"语言理解的底座",但任务头、损失、数据、评测仍然是 per-task 的。

### 2.2 统一的技术前提:三件事同时到位

为什么统一在 2020 年后才发生?因为三个条件刚好凑齐:

**前提一:一切任务都能写成"文本→文本"。** 这是 T5(Raffel et al., 2020,你在第八章见过)的核心 thesis:Text-to-Text Transfer Transformer 把分类、翻译、摘要、问答全部塞进同一个 seq2seq 框架,区别只在输入前缀("translate English to German: ...")。这一步把**任务的输出空间统一成了 token 序列**。分类不再是"在 K 个类别上算 softmax",而是"生成类别名这几个 token"。

**前提二:模型大到能做 in-context learning。** GPT-3(Brown et al., 2020)发现:当模型足够大,你不更新任何权重,只在 prompt 里塞几个示例,模型就能模仿着做新任务。这意味着**任务定义从"训练时固化在权重里"迁移到了"推理时写在输入里"**。机制为什么成立(为什么大模型能从上下文里"学")属于算法课 14 的内容,本章只用它的结论。

**前提三:指令微调让模型"听话"。** 光有 in-context learning 还不够——裸 GPT-3 你得精心构造 prompt 它才配合。InstructGPT(Ouyang et al., 2022)用指令微调 + RLHF(详见强化学习课)让模型学会"理解自然语言指令并照做"。从此你可以直接说"把下面这句话分类成正面或负面",而不用拼凑 few-shot 模板。

三者叠加的结果:**任务定义这件事,从"离线训练专用模型"变成了"运行时用一段自然语言指定"。** 这是本章所有讨论的原点。

把这个转变用一句话钉死:**经典 NLP 把"任务"编码进模型参数 θ;LLM 时代把"任务"编码进输入 x。**

## 3. 范式转变:任务版图如何坍缩

### 3.1 形式化:同一个动作

经典监督学习里,任务 t 对应一个独立的条件分布,由专属参数 θ_t 刻画:

```
分类:  P_θ_cls(y | x),  y ∈ {正面, 负面}
NER:   P_θ_ner(tags | x), tags ∈ 标注序列空间
翻译:  P_θ_mt(y | x),    y ∈ 目标语言句子空间
```

每个 θ_t 单独训练。LLM 时代,这些坍缩成**同一个**自回归分布,由**同一组**参数 θ 刻画,任务信息全部移进输入:

```
统一形式:  P_θ(y | prompt_t ⊕ x)

其中 prompt_t 是任务 t 的自然语言描述(可含示例),⊕ 是文本拼接,
y 永远是 token 序列,逐 token 自回归生成:
    P_θ(y | c) = ∏ₜ P_θ(yₜ | c, y_<t)     (c = prompt_t ⊕ x)
```

注意这里发生了两个深刻的变化:

1. **下标 t 从 θ 跑到了输入里**。模型不变,变的是 condition。
2. **输出空间从"任务专属"变成"全词表序列"**。分类的答案"正面"现在是词表里两三个 token,要和"我今天很正面地评价了这部电影"这种自由文本竞争同一个 softmax。这是后面讲"结构化输出"为什么必要的根源。

### 3.2 In-context learning:把训练集塞进 prompt

In-context learning(ICL,上下文学习)是统一的关键武器。形式上,**few-shot prompt 就是把若干 (输入, 输出) 示例写进 condition c**:

```
c = [指令]
    [示例1 输入] → [示例1 输出]
    [示例2 输入] → [示例2 输出]
    ...
    [示例k 输入] → [示例k 输出]
    [真实输入] →                  ← 模型在这里续写,生成答案
```

`k=0` 叫 zero-shot(只有指令),`k≥1` 叫 few-shot。关键直觉:**模型没有更新任何权重**,它是在"识别"这个 prompt 描述的是什么任务、然后"模仿"示例的输入输出映射。为什么大模型能做到这件事(一种主流解释是它在做隐式的梯度下降 / 贝叶斯任务推断),是算法课 14 的核心,这里不展开。

工程上你只需记住 ICL 的**三条经验定律**(后面"常见坑"会量化):
- 示例**选得好**比选得多重要(语义接近测试样本的示例增益最大)。
- 示例**顺序**会显著影响结果——尤其是标签的排列。
- 示例**格式**(分隔符、字段名)必须和真实输入完全一致,否则模型可能"对齐错栏位"。

### 3.3 Chain-of-Thought:让难任务先"想"再答

对于需要多步推理的任务(算术、多跳问答、逻辑题),直接让模型输出答案命中率低。Chain-of-Thought(CoT,思维链,Wei et al., 2022)的做法是**在 prompt 里引导模型先生成中间推理步骤,再给最终答案**:

```
zero-shot CoT 触发词(Kojima et al., 2022):
    在问题后面加一句 "Let's think step by step."
    → 模型先吐出推理过程,再吐答案,难题正确率显著上升
```

从概率角度看,CoT 在做的事可以理解为**把一步难推断分解成若干步易推断的链式条件**:

```
直接:    P(answer | question)              ← 一步,模型要"心算"完
CoT:     P(answer | question, r₁, r₂, ..., r_m)
         其中 rᵢ 是模型自己生成的中间步骤,
         每个 rᵢ ~ P(rᵢ | question, r_<i)
```

每一步 rᵢ 都是模型擅长的"短跨度续写",链起来逼近一个困难的长跨度推断。**为什么这样有效、什么时候失效**(比如小模型用 CoT 反而更差)是算法课 14 的内容。本章的 takeaway 是:CoT 把"推理"也变成了"生成的一部分",进一步坐实了"一切皆生成"。

### 3.4 指令跟随:让用户能"说人话"

ICL + CoT 还需要模型"愿意配合"。指令微调(instruction tuning)和 RLHF 让模型把"自然语言指令"当成第一类任务接口。**这是统一从"研究者玩具"走向"产品"的最后一公里**:普通用户不写 few-shot 模板,他们直接说"帮我把这封邮件改得礼貌一点"。指令跟随的训练机制(SFT + 偏好优化 + PPO/DPO)请看强化学习课;本章把它当成既成事实使用。

## 4. 同一任务的两种活法:微调小模型 vs LLM prompt(代码对照)

下面用**情感分类**和 **NER** 两个任务,把"经典做法"和"LLM 做法"并排放,让你看清坐标系旋转后代码长什么样。

### 4.1 情感分类:微调 BERT(经典)

```python
# 经典:微调一个 BERT 分类器(HuggingFace transformers)
# 机制:[CLS] 向量 → 线性分类头 → 在 2 个类别上算 softmax + 交叉熵
import torch
from transformers import AutoTokenizer, AutoModelForSequenceClassification

tok = AutoTokenizer.from_pretrained("bert-base-uncased")
model = AutoModelForSequenceClassification.from_pretrained(
    "bert-base-uncased", num_labels=2)          # ← 任务专属:加了个 2 类的头

# 训练循环(简化):每个 batch 都在更新全部权重 θ
def train_step(batch_texts, batch_labels):
    enc = tok(batch_texts, padding=True, truncation=True, return_tensors="pt")
    out = model(**enc, labels=torch.tensor(batch_labels))
    out.loss.backward()        # ← 交叉熵损失,任务信息编码进梯度
    optimizer.step(); optimizer.zero_grad()
    return out.loss.item()

# 推理:输出是 logits over {0,1},argmax 得标签
def predict(text):
    enc = tok(text, return_tensors="pt")
    logits = model(**enc).logits
    return logits.argmax(-1).item()             # 0=负面, 1=正面
```

要点:**任务定义在 `num_labels=2` 和那条交叉熵损失里**,是离线、固化在 θ 里的。换任务要换头、换损失、重训。

### 4.2 情感分类:LLM prompt(统一)

```python
# 统一:不训练,运行时用一段提示指定任务
# 机制:任务定义全在 prompt 字符串里,模型权重一动不动
def classify_with_llm(text, llm_generate):
    prompt = (
        "Classify the sentiment of the review as Positive or Negative.\n"
        "Answer with exactly one word.\n\n"
        # —— few-shot 示例:把"训练集"塞进 prompt ——
        "Review: This movie was a complete waste of time.\nSentiment: Negative\n\n"
        "Review: An absolute masterpiece, I cried.\nSentiment: Positive\n\n"
        # —— 真实输入 ——
        f"Review: {text}\nSentiment:"
    )
    out = llm_generate(prompt, max_tokens=3, stop=["\n"])  # 续写
    return out.strip()           # 期望得到 "Positive" / "Negative"
```

要点:**没有训练循环,没有损失函数,没有梯度**。任务定义是那段英文。换任务?改 prompt 字符串即可。`stop=["\n"]` 和 `max_tokens=3` 是在"约束"模型别说废话——这是结构化输出问题的雏形(见 4.4)。

### 4.3 NER:经典 vs LLM

经典 NER 用 token 级标注 + CRF 解码(你在第五章学过维特比):

```python
# 经典 NER:每个 token 一个 BIO 标签,CRF 保证标签序列合法
# 机制:模型对每个 token 出标签分布,CRF 层做全局最优序列解码
#   label 空间如 {O, B-PER, I-PER, B-LOC, I-LOC, ...}
#   维特比解码保证不会出现 "I-PER 前面没有 B-PER" 这种非法转移
model = AutoModelForTokenClassification.from_pretrained(
    "bert-base-cased", num_labels=len(BIO_TAGS))   # ← 任务专属标签集
# 训练用 token 级交叉熵(或 CRF 负对数似然),输出对齐到每个 token
```

LLM NER:让模型直接生成结构化的抽取结果。

```python
# LLM NER:把"抽取"变成"生成一个 JSON"
def ner_with_llm(text, llm_generate):
    prompt = (
        "Extract all named entities. Return a JSON list of "
        '{"text": ..., "type": ...} where type ∈ {PER, LOC, ORG}.\n'
        "If none, return [].\n\n"
        'Text: "Tim Cook visited Beijing for Apple."\n'
        'Entities: [{"text":"Tim Cook","type":"PER"},'
        '{"text":"Beijing","type":"LOC"},{"text":"Apple","type":"ORG"}]\n\n'
        f'Text: "{text}"\n'
        "Entities:"
    )
    raw = llm_generate(prompt, max_tokens=256, stop=["\n\n"])
    import json
    try:
        return json.loads(raw)        # ← 脆弱点:模型可能生成非法 JSON
    except json.JSONDecodeError:
        return None                   # 必须有兜底
```

这里暴露了 LLM 做抽取的**根本矛盾**:经典 CRF **从结构上**保证输出合法(维特比在合法序列空间里搜),而 LLM 是自由生成,**没有任何机制保证输出是合法 JSON**,更别说 entity span 一定在原文里。下一节专门讲怎么治。

### 4.4 结构化输出:把"自由生成"按回笼子里

把 NLP 任务统一成"生成"带来一个新麻烦:**很多任务的输出本该是结构化的**(标签、JSON、函数调用参数),但自回归生成默认是自由文本。三种治理手段,由弱到强:

**手段一:prompt 约定 + 解析兜底。** 就是 4.3 那样,prompt 里写清格式 + few-shot 演示 + `try/except` 解析失败重试。**便宜但不可靠**,失败率随 schema 复杂度上升。

**手段二:约束解码(constrained decoding)。** 在每一步采样时,**屏蔽掉所有会导致违反语法的 token**。比如要求输出合法 JSON,就在解码时维护一个状态机,只允许生成当前语法状态下合法的下一个 token:

```
普通解码:  yₜ ~ softmax(logits)                    over 全词表
约束解码:  yₜ ~ softmax(logits + mask)              over 合法子集
           其中 mask[v] = 0 若 token v 合法,否则 -∞
           合法集由一个语法/正则/JSON-schema 驱动的状态机给出
```

直觉:把概率质量强制重分配到合法 token 上,**从结构上杜绝非法输出**——这是 LLM 端对 CRF "硬约束"思想的复刻。具体的实现(FSM 编译、token 级别的语法约束、与 KV cache 的配合)属于**算法课 13 解码策略**,这里只给思想。开源实现见 `outlines`、`guidance`、以及 vLLM 的 guided decoding(待核:具体 API 名称与版本随项目演进)。

**手段三:函数调用 / tool use。** 主流 API(待核:具体字段名各家不同)让你声明一个函数 schema,模型输出"调用哪个函数、参数是什么"的结构化对象,由框架保证字段合法。本质是"约束解码 + 预定义 schema"的产品化封装。这把 LLM 从"文本生成器"变成"能驱动外部系统的 agent",是下一章 RAG 与工具增强的基础。

**一句话:统一带来了通用性,但也丢掉了经典模型自带的结构保证;约束解码和函数调用是把保证加回来的工程手段。**

## 5. 核心决策:微调 vs prompting,什么时候用哪个

这是本章最有工程价值的部分。**不要默认"有了 LLM 就不微调"——那是新手最大的认知偏差。** 下面给一个可操作的决策框架。

### 5.1 prompting 的真实成本结构

先算账。prompting 看起来"免训练免费",但它的成本藏在**推理**里:

```
单次推理成本 ≈ (prompt_tokens + output_tokens) × 单价
prompt_tokens = 指令 + k 个 few-shot 示例 + 输入

few-shot 的代价:每多一个示例,每次请求都要重付这些 token 的钱和延迟。
若示例总长 1000 tokens、QPS=100、全天候运行:
    每天浪费在"重复读示例"上的 token = 1000 × 100 × 86400 ≈ 8.64×10⁹ tokens/天
```

对比微调:微调是**一次性**训练成本,之后推理时 prompt 极短(甚至 zero-shot),**单次推理便宜得多、快得多**。所以:**高吞吐、长期稳定的任务,微调的总拥有成本(TCO)往往远低于 few-shot prompting。** few-shot 的"省"只在低频、长尾、快速试错阶段成立。

### 5.2 决策矩阵

| 维度 | 倾向 prompting(LLM) | 倾向微调 / 小模型 |
|---|---|---|
| 任务频率 | 低频、长尾、一次性 | 高频、稳定、长期 |
| 吞吐 / 延迟 | 宽松(秒级可接受) | 严苛(毫秒级、高 QPS) |
| 单位成本 | 不敏感 | 极敏感(每条都要省) |
| 输出结构 | 自由文本、弱结构 | 严格 schema、强约束 |
| 数据量 | 标注极少或没有 | 有充足标注数据 |
| 领域 | 通用、常识性 | 专有领域(医疗代码、内部黑话) |
| 隐私 / 合规 | 可外发(或自建) | 数据不能出域 → 本地小模型 |
| 任务稳定性 | 需求经常变 | 定义已冻结 |
| 可解释 / 可审计 | 弱(黑盒生成) | 强(可控、可追责) |

**读法:不是看哪边勾得多,而是看有没有"硬约束"命中。** 隐私不能出域、延迟必须 5ms、成本必须压到极致——任何一条硬约束命中右列,就基本排除纯 LLM 方案。软偏好(数据少、需求变)才在左右之间权衡。

### 5.3 第三条路:用 LLM 造数据去微调小模型

实践中最划算的常是**蒸馏**:用强 LLM 给大量无标注数据打标(或直接生成训练对),再用这些"伪标注"微调一个小模型上线。这样:**研发期享受 LLM 的零样本能力快速冷启动,上线期享受小模型的低成本低延迟。** 这是工业界对"微调 vs prompting"二选一的常见破解。代价是:伪标注有噪声,需要质检和置信度过滤;且小模型能力上限受 teacher 限制。

## 6. NLP 作为独立学科:哪些被吸收,哪些仍独立

回到本章标题的核心问题:**"NLP"这个学科被 LLM 统一到什么程度?**

### 6.1 被高度吸收的(LLM 通常是更好或同样好的选择)

- **开放域问答、摘要、改写、翻译(通用语种)、零样本/少样本分类**:这些任务的"长尾、需要常识、输出可容忍自由文本"特性正好是 LLM 的主场。专用模型在这些任务上正在被淘汰或仅作为成本优化的替代。
- **意图理解、对话**:指令跟随天然覆盖,任务型对话的"填槽"也大量被 prompt + 函数调用取代(详见对话系统章)。

### 6.2 仍然独立存在的(LLM 不是默认答案)

这些场景**不是因为 LLM 做不了,而是因为有硬约束让 LLM 不划算或不合规**:

- **高吞吐 / 低成本 / 低延迟场景**:搜索召回的 query 分类、广告 CTR 的文本特征、内容审核的第一道过滤——QPS 动辄上万,每条都要在毫秒级、成本压到极致。这里小模型(蒸馏 BERT、fastText 甚至逻辑回归)依然是主力。
- **强结构 / 强约束任务**:需要**保证**输出合法的序列标注、句法分析、严格 schema 的信息抽取。CRF / 转移系统的结构保证是 LLM 自由生成给不了的(只能用约束解码逼近,且更贵)。
- **可解释 / 可审计需求**:金融、医疗、法律里需要解释"为什么这么判"。线性模型、CRF 的权重和转移分数可解释、可追责;LLM 是黑盒,出错难归因。
- **数据不能出域**:隐私 / 合规要求数据留在本地,而顶级 LLM 多为闭源 API。自建小模型或本地开源小模型成为必选。
- **极端低资源语言 / 专有领域**:LLM 在训练数据稀薄的语种和高度专业的黑话上仍会胡说,针对性微调的专用模型更可靠。

### 6.3 务实判断口诀

**"先问硬约束,再谈通用性。"** 没有硬约束命中右列时,LLM prompting 是默认起点(开发快、覆盖广);一旦有延迟/成本/隐私/结构/可解释的硬约束,就退回经典方法或走"LLM 蒸馏小模型"的混合路线。**学科没有消失,而是分层了:LLM 吃掉了"长尾通用层",经典方法守住了"高约束的工业层"。**

## 7. 设计权衡与常见坑

**坑 1:prompt 的脆弱性被严重低估。** 同一个任务,改一个分隔符、调换两个示例的顺序、把 "Positive" 写成 "positive",结果可能差很多。实证发现(Lu et al., 2022,"Fantastically Ordered Prompts"):**仅改变 few-shot 示例的排列顺序,同一组示例的准确率可能从接近随机猜到接近 SOTA**(具体跨度待核,但"顺序敏感到能从最差到最好"是确证的现象)。教训:**prompt 必须像代码一样做版本管理和回归测试**,不能改完拍脑袋觉得"差不多"。

**坑 2:把准确率的提升误判为"模型学会了"。** few-shot 加示例涨点,可能只是因为示例**泄露了标签分布**或**对齐了输出格式**,而非模型真的更懂任务。验证方法:用**打乱标签**的示例做对照——如果乱标签也涨点,说明涨的是"格式对齐 / 任务识别",不是"从示例学映射"(Min et al., 2022 的著名发现:很多时候示例的**标签正确与否对结果影响没想象中大**,模型主要从示例学的是"格式和标签空间")。

**坑 3:结构化输出不做约束,线上必炸。** demo 里 `json.loads` 跑通了就上线,结果真实流量里 1%-5% 的输出是非法 JSON(漏引号、多逗号、夹带解释文字)。**任何要被下游程序消费的 LLM 输出,都必须上约束解码或至少做解析 + 重试 + schema 校验**,且要有"解析永久失败"的兜底分支。

**坑 4:用 few-shot 解决本该微调的高频任务,账单爆炸。** 见 5.1 的算账。**长期高频任务塞一堆 few-shot 示例 = 每次请求重复付示例的钱**,几个月的推理成本可能远超一次微调。

**坑 5:CoT 不是万能,小模型用 CoT 可能更差。** CoT 在足够大的模型上才稳定收益;小模型生成的"推理"可能是错的还误导最终答案。不要无脑给所有任务都加 "think step by step"——对简单分类任务,CoT 只是增加延迟和被推理过程带偏的风险。

**坑 6:评测"看起来对"≠ 正确。** LLM 输出流畅,人容易被流畅性骗过(hallucination,幻觉,生成看似合理实则错误的内容)。生成式任务的评测本身就是难题——为什么传统的 BLEU/ROUGE 不够、该怎么评忠实性,是本课后面"文本生成的评测"专章的主题,这里先埋一个钉子:**LLM 时代评测的难度上升了,因为输出空间是开放的。**

**权衡总览:** prompting 用"运行时成本 + 脆弱性 + 黑盒"换"开发速度 + 通用性 + 零标注冷启动";微调用"离线训练成本 + per-task 工程"换"低推理成本 + 强结构 + 可控可解释"。**没有免费午餐,只有约束驱动的选择。**

## 8. 动手练习

**练习 1(决策训练).** 给定三个任务,各自判断该用 prompting 还是微调,并写出决定性的那条"硬约束":
(a) 一个内部工具,产品经理偶尔想把一批用户反馈按主题归类(每天几十条);
(b) 电商搜索的 query 实时分类,峰值 50000 QPS,延迟预算 8ms;
(c) 医院内部把病历自由文本抽成结构化字段,数据严禁出院内网。
*提示:对每个任务过一遍 5.2 的决策矩阵,先找有没有硬约束(延迟/成本/隐私/结构)命中右列;命中就基本定了,没命中再看软偏好。参考答案方向:(a) prompting;(b) 微调小模型;(c) 本地微调 + 约束解码。*

**练习 2(稳健 few-shot prompt 设计).** 为"把一段中文客服对话分类为 {咨询, 投诉, 表扬, 其他}"设计一个稳健的 few-shot prompt。要求:(1) 4 个类各至少一个示例;(2) 输出限制为单个类名;(3) 给"其他"一个明确边界定义防止过度归类。然后做一个小实验:把示例顺序随机打乱 5 次,记录同一批测试样本的预测是否变化。
*提示:用明确的分隔符和固定字段名(如 "对话:" / "类别:");在指令里写死合法类名集合;用 `stop` token 和短 `max_tokens` 防止模型多嘴。对照练习对应坑 1——如果打乱顺序结果大幅波动,说明你的 prompt 还不够稳健,考虑增加示例数或改用约束解码限定输出在 4 个类名上。*

**练习 3(结构化输出加固).** 把 4.3 的 LLM NER 从"prompt + try/except"升级为更可靠的版本。至少实现:(1) JSON schema 校验(字段名、type 取值合法);(2) 校验每个抽取的 entity text 确实是原文子串(防止模型编造 span);(3) 解析或校验失败时的重试 + 最终兜底。
*提示:第 (2) 点是 LLM 抽取相对 CRF 的独有风险——自由生成不保证 span 在原文里。可以用 `entity["text"] in original_text` 做最简校验。想再进一步,去了解约束解码库(`outlines` 的 JSON 模式)如何从解码层面保证 schema 合法,对比"事后校验"和"事中约束"的成本差异。*

**练习 4(蒸馏实验,选做).** 取一个公开情感数据集的无标注部分,用一个 LLM 以 zero-shot 打标,得到伪标注;用伪标注微调一个 `distilbert`,在该数据集的真实测试集上对比"LLM 直接 zero-shot" vs "蒸馏出的小模型"的准确率和单条推理延迟。
*提示:这正是 5.3 的混合路线。观察两点:(1) 小模型准确率能逼近 teacher 到什么程度(通常略低);(2) 延迟和成本差几个数量级。体会"研发用大模型、上线用小模型"的工程逻辑。*

## 9. 源码 / 论文导读

**论文(读这些部分):**
- **T5**(Raffel et al., 2020,"Exploring the Limits of Transfer Learning..."):读引言和第 2 节的 text-to-text 框架定义——理解"一切任务统一成文本→文本"的原始 thesis。
- **GPT-3**(Brown et al., 2020,"Language Models are Few-Shot Learners"):读摘要 + 关于 zero/one/few-shot 设定的那一节——理解任务定义如何从权重迁移到 prompt。机制别在这找,看算法课 14。
- **Chain-of-Thought**(Wei et al., 2022)+ **zero-shot CoT**(Kojima et al., 2022,"Large Language Models are Zero-Shot Reasoners"):读方法节,理解"先推理后回答"的 prompt 构造。
- **InstructGPT**(Ouyang et al., 2022):读 SFT + RLHF 的 pipeline 概述,理解"指令跟随"从哪来——细节去强化学习课。
- **Prompt 脆弱性**:Lu et al., 2022("Fantastically Ordered Prompts and Where to Find Them",示例顺序敏感性)与 Min et al., 2022("Rethinking the Role of Demonstrations",示例标签正确性的作用)——读完你会对 prompt engineering 的"科学性 vs 玄学"有清醒认知。

**开源实现(看这些模块):**
- **HuggingFace `transformers`**:`AutoModelForSequenceClassification` / `AutoModelForTokenClassification`(经典微调路径)与 `pipeline("text-generation")`(生成路径)——把第 4 节两种活法在同一个库里跑通。
- **约束解码**:`outlines`(看它如何把 JSON schema / 正则编译成 token 级 FSM 约束)、`guidance`、vLLM 的 guided decoding(待核:API 名随版本变)。对照算法课 13 的解码理论。
- **蒸馏 / 伪标注**:可用 `transformers` + 任意 LLM API 自己搭(库无强约束),重点在数据流水线设计,不在某个特定框架。

## 10. 小结与承上启下

这一章讲透了一件事:**LLM 把"NLP 任务"的定义位置从模型参数 θ 搬到了输入 x。** 由此,曾经各有专用模型、专用损失、专用评测的分类 / 抽取 / 翻译 / 摘要 / QA,在使用层面坍缩成同一个动作——给 LLM 一段 prompt 让它生成。支撑这次统一的三根柱子是:text-to-text 的任务形式统一、in-context learning 的运行时任务指定、指令跟随的"听话"能力(后两者的内部机制分属算法课 14 与强化学习课)。

但**统一不是消灭**。我们用决策矩阵把"何时 LLM、何时经典"讲清楚了:**先看硬约束(延迟 / 成本 / 隐私 / 结构 / 可解释),命中就退回经典或走蒸馏混合路线;没命中,LLM prompting 是默认起点。** 同时我们看清了统一的代价:prompt 的脆弱性、结构化输出的失保(要靠约束解码 / 函数调用补回)、推理成本的转移、以及评测难度的上升。

**承上**:本章用到的 T5、BERT、GPT 都来自前几章,CRF 的结构保证来自第五章——正是有了那些经典底子,你才能看懂"LLM 用约束解码在复刻 CRF 的硬约束"这类对照。

**启下**:LLM 统一了"用语言指定任务",但它**不知道训练数据之外的事实**,也会一本正经地幻觉。下一章(检索增强生成 RAG)就解决这个缺口——**给大模型接上一个外部知识库,让生成"有据可查"**。而本章埋下的"结构化输出 / 函数调用"正是 RAG 与工具增强的接口基础;埋下的"评测难、幻觉多"则是后面"文本生成评测"与"忠实性"专章要正面攻克的硬骨头。
