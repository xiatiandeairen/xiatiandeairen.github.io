> 这一章把你从"会做序列标注、会调 BERT、会写 LLM prompt"带到"能把一段非结构文本系统地抽成实体、关系、事件、链接到知识库的结构化知识,能从第一性原理讲清 pipeline 与联合抽取的误差传播差异、能推导 TransE 的打分与梯度、能设计一个对 schema 稳健、对噪声鲁棒的工业级抽取方案"。**检索(第 10 章)帮你找到相关文本,但文本本身仍是非结构的;信息抽取是把文本压成机器可推理、可查询、可校验的知识——这是连接"语言"和"知识"的那道工序。**

## 一句话引言

信息抽取(Information Extraction, IE)的根本任务是:给定非结构文本 x,产出一个**结构化对象** y——一组实体、一组关系三元组、一组事件框架,或一个符合预定义 schema 的 JSON。**它的本质是一个"文本 → 结构"的条件生成/标注问题**,核心难点不在于"模型认不认识词",而在于**结构本身有强约束**(实体边界不能交叉、关系两端必须是实体、事件论元必须填进角色槽),以及**标注数据极贵**(标一个关系抽取语料比标分类贵一个数量级)。本章从最经典的 NER 推起,串透 RE/事件/链接/共指五大任务,讲清从 CRF 到 LLM 的方法演进各自的精度-成本-可控权衡,推导知识图谱嵌入(TransE)的机制,最后落到"如何让模型可靠产出 schema"这个 2024 年后最实用的工程问题。

## 一、动机:为什么要把文本变成结构

先回答一个产品经理会问的问题:既然 LLM 已经能读懂文本、能直接回答问题,**为什么还要费力抽成结构**?

三个层层递进的理由,决定了 IE 不会被"直接问大模型"取代:

**理由一:结构可被精确查询和聚合,自由文本不能。** "苹果 2023 财年营收多少"这种问题,如果知识以三元组 `(Apple, revenue_FY2023, 383.3B USD)` 存在图里,是一次 O(1) 的查表;如果只存在 1000 篇财报 PDF 里,你每次都要让模型读一遍,既慢又不稳定。**结构化是把"每次重新理解"摊销成"一次抽取、多次查询"**。当你要做"统计 2023 年所有科技公司里营收同比下滑的"这种聚合分析时,自由文本根本无法 GROUP BY,结构化知识可以。

**理由二:结构可被校验,文本不能。** 三元组 `(阿司匹林, 禁忌, 孕妇)` 可以被领域专家或规则库逐条核对、打标可信度、追溯来源;一段 LLM 生成的医疗建议没法被逐句机器校验。**在高风险领域(医疗、金融、法律),"可校验"是硬约束**,而可校验的前提是结构化。

**理由三:结构是知识图谱、推理、事实校验的底座。** 第 10 章 RAG 用检索补 LLM 的知识,但检索粒度是"文档块";如果知识以图存在,你可以做**多跳推理**(A 的 CEO 是 B,B 又是 C 的董事 → A 和 C 有关联),可以用 KG 给 LLM 的回答**做事实校验**(模型说"X 是 Y 的子公司",去图里查有没有这条边)。这是后面 1.6 节"KG 与 LLM 结合"的核心。

**给做过工程的读者的迁移提示:** 把 IE 理解成"为非结构数据建 ETL 管线"会很顺——抽取(Extract)就是从文本里解析出字段,转换(Transform)是消歧和规范化(把"老马""马斯克""Elon Musk"归一到同一实体),加载(Load)是写进图数据库。你熟悉的"schema 设计""数据质量""脏数据清洗"的全部直觉都能迁移过来,只不过这里的"解析器"是个会犯错的神经网络,所以**误差传播和噪声鲁棒性是第一性的工程考量**。

本章的任务地图,按"抽什么粒度"组织:

```
非结构文本 x
  │
  ├─ 实体(谁/什么)──────── NER 命名实体识别（序列标注，承接第 04 章）
  │                          ↓ 抽出的提及(mention)是后面一切的锚点
  ├─ 关系(实体间)────────── RE 关系抽取（实体对 + 关系类型）
  │                          pipeline（先 NER 再 RE） vs 联合抽取
  ├─ 事件(动态结构)──────── EE 事件抽取（触发词 + 论元角色填充）
  │
  ├─ 把提及挂到世界 ──────── EL 实体链接（mention → KB entity，消歧）
  │                          指代统一 ────── 共指消解（he/它/该公司 → 同一实体）
  ↓
结构化知识 → 知识图谱（h, r, t 三元组）→ KG 嵌入 / KG+LLM 事实校验
```

## 二、NER:从序列标注到 span 分类(承接 CRF 章)

### 2.1 任务与标注体系

**命名实体识别(Named Entity Recognition, NER)**:在文本中找出指称实体的连续片段(span),并给它分类(人名 PER、地名 LOC、机构 ORG、时间、金额……)。它是 IE 的入口——后面 RE/EL/共指全都建立在"先有实体提及"之上。

第 04 章已经把 NER 当成**序列标注**讲透了:用 **BIO 标注体系**(B-PER 表示人名开头,I-PER 表示人名内部,O 表示非实体),把"找 span"转成"给每个 token 打标签"。关键结论是**不能逐位置独立 softmax**,因为标签间有强依赖(I-PER 不能跟在 O 后,也不能跟在 B-LOC 后),所以要用 **CRF 建模标签转移**。这里不重复 CRF 配分函数和维特比的推导(**详见第 04 章**),只承接它往前推:**BERT 时代,NER 的特征怎么变了**。

BIO 之外还有几种变体,各有取舍:
- **BIO**:最常用,3 类标签 × 实体类型数 + 1 个 O。
- **BIOES**(加 E=End、S=Single):显式标出实体结尾和单 token 实体,标签更精细,**对边界的监督更强**,实务中 F1 常比 BIO 高零点几个百分点,代价是标签空间更大、更稀疏。
- **嵌套实体**(nested NER)问题:BIO 无法表达"[北京[大学]]"这种实体套实体——一个 token 只能有一个标签。这是 BIO 体系的**根本局限**,催生了下面的 span 分类法。

### 2.2 BERT+softmax vs BERT+CRF:CRF 层还要不要

把 BERT(预训练章讲透)接到 NER 上,最直接的做法是 **token 分类**:BERT 给每个 token 一个上下文向量 hᵢ ∈ ℝ^d,接一个线性层投到标签空间,每个位置独立 softmax 取 argmax。

那 CRF 层还要不要?这是个有定论但常被误解的工程问题。**结论:用强编码器(BERT)时,CRF 带来的增益显著缩小,但在标签约束强、数据少的场景仍有价值。**

为什么增益缩小?CRF 的价值在于**建模标签间的转移依赖**(防止 O→I-PER 这种非法转移)。但 BERT 的双向自注意力已经让每个位置的表示 hᵢ **看到了整句上下文**,某种程度上隐式编码了邻居标签信息——它"知道"前一个 token 是不是人名的一部分。所以 BERT+softmax 已经能学到大部分约束。**CRF 仍提供的、softmax 给不了的,是"全局归一 + 硬性禁止非法转移路径"**:维特比解码保证输出的标签序列在转移矩阵下合法,而独立 softmax 可能逐位置都"差不多对"却拼出一条非法路径(比如孤立的 I-PER)。

量化地看代价:CRF 层在 BERT 之上只加一个 |L|×|L| 的转移矩阵(|L| 是标签数,通常几十),参数量可忽略;但训练时配分函数 Z(x) 的前向算法是 O(T·|L|²),解码维特比也是 O(T·|L|²),相比 softmax 的 O(T·|L|) 慢一个 |L| 因子。**实务建议:数据充足(>万级标注句)、追求吞吐时用 BERT+softmax;数据少、标签约束强(如 BIOES、嵌套展开)时加 CRF。**

下面是 BERT+token 分类做 NER 的可运行核心(这是本章要求的"对照组"之一):

```python
import torch
import torch.nn as nn
from transformers import AutoModel, AutoTokenizer

class BertNER(nn.Module):
    """BERT + token 分类做 NER。标签空间是 BIO 展开后的标签集，
    例如 {O, B-PER, I-PER, B-ORG, I-ORG, B-LOC, I-LOC}。"""
    def __init__(self, model_name="bert-base-cased", num_labels=7):
        super().__init__()
        self.encoder = AutoModel.from_pretrained(model_name)
        self.dropout = nn.Dropout(0.1)
        self.classifier = nn.Linear(self.encoder.config.hidden_size, num_labels)
        self.num_labels = num_labels

    def forward(self, input_ids, attention_mask, labels=None):
        # h: [B, T, d] 每个 token 的上下文表示
        h = self.encoder(input_ids, attention_mask=attention_mask).last_hidden_state
        logits = self.classifier(self.dropout(h))   # [B, T, num_labels]
        loss = None
        if labels is not None:
            # 关键坑1：subword 对齐——一个词被 BERT 切成多个 subword，
            # 通常只对首个 subword 算 loss，其余位置 label 设为 -100 被 ignore
            loss_fn = nn.CrossEntropyLoss(ignore_index=-100)
            loss = loss_fn(logits.view(-1, self.num_labels), labels.view(-1))
        return logits, loss

# —— subword 与原始词的标签对齐（NER 最容易错的地方）——
def align_labels(text_tokens, word_labels, tokenizer):
    enc = tokenizer(text_tokens, is_split_into_words=True, truncation=True)
    word_ids = enc.word_ids()            # 每个 subword 对应的原词下标，特殊符为 None
    aligned, prev = [], None
    for wid in word_ids:
        if wid is None:
            aligned.append(-100)         # [CLS]/[SEP]/pad 不算 loss
        elif wid != prev:
            aligned.append(word_labels[wid])   # 词的首个 subword 用真标签
        else:
            aligned.append(-100)         # 后续 subword 忽略（也可填 I- 标签，看实现约定）
        prev = wid
    return enc, aligned
```

**这段代码体现的两个核心机制**:(1)NER 在 BERT 上就是**逐 token 的多分类**,与文本分类只差"用每个位置的 hᵢ 而不是 [CLS]";(2)**subword 对齐是工程上最容易出 bug 的点**——BERT 用 subword 分词,一个词可能被切成多个 piece,你必须决定 label 挂在哪个 piece 上,且评估时要把 subword 预测**合并回词级**再算 F1,否则指标虚高/虚低。

### 2.3 span 分类:解决嵌套实体的现代视角

BIO 的根本局限(一 token 一标签,无法嵌套)催生了 **span-based NER**:不再逐 token 标注,而是**枚举所有可能的片段 span (i, j),对每个 span 分类**(是哪类实体,或"不是实体")。

机制:对每个候选 span (i, j),用边界表示拼一个 span 向量,例如 `s_{ij} = [hᵢ ; hⱼ ; (hᵢ−hⱼ) ; width_emb(j−i)]`,再过分类头判类型。这天然支持嵌套(同一 token 可以同时属于多个被分类为实体的 span)。

代价是**组合爆炸**:长度 T 的句子有 O(T²) 个 span,全枚举太贵。工程上限制最大 span 宽度 W(实体很少超过 ~10 词),把候选数降到 O(T·W),并做负采样平衡(绝大多数 span 都是"非实体")。这条线的代表是 SpanBERT 思路与 SpERT(后者把 span 分类和关系抽取联合做,见下节)。**待核:SpERT 具体提出年份与原始 F1 数值以原文为准。**

## 三、关系抽取:pipeline vs 联合抽取的误差传播

### 3.1 任务定义

**关系抽取(Relation Extraction, RE)**:给定文本和其中的实体,判断实体对之间是否存在预定义关系、是哪种。输出是三元组 (head, relation, tail),如句子"马斯克创立了 SpaceX"→ (马斯克, 创立, SpaceX)。这是把 NER 抽出的孤立实体**连成图**的关键一步。

RE 有两种问题设定,要分清:
- **句子级 RE**:关系两端实体在同一句内(本节主讲)。
- **文档级 RE**(DocRED 那类):实体可能跨句,关系要跨句推理,难度高一个量级,常需共指消解辅助。

### 3.2 pipeline:先 NER 再 RE,以及它的致命缺陷

最直观的做法是 **pipeline(流水线)**:第一阶段 NER 抽出所有实体,第二阶段对每个实体对 (eₐ, e_b) 做关系分类。

第二阶段的经典做法叫**实体标记(entity marker)**:在输入里用特殊符号把两个目标实体框出来,让 BERT 知道"判断谁和谁的关系"。例如:

```
原句:   马斯克 创立了 SpaceX 。
标记后: [E1] 马斯克 [/E1] 创立了 [E2] SpaceX [/E2] 。
分类向量: 取 [E1] 和 [E2] 标记位的隐状态拼接 → 分类头 → 关系类型
```

pipeline 的**致命缺陷是误差传播(error propagation)**。这是 IE 里最重要的第一性原理之一,值得定量看清楚:

设 NER 的实体级准确率(一个实体对**两端都抽对**的概率)为 p_ner,RE 在给定正确实体对时的关系分类准确率为 p_re。那么 pipeline 整体抽对一个三元组的概率上界是:

```
P(三元组正确) ≤ P(两端实体都抽对) · P(关系分对 | 实体对)
             = p_ner · p_re
```

举个数:若 p_ner=0.90,p_re=0.85,则 pipeline 端到端上限 ≈ 0.765——**两个 0.9 量级的模块串起来,整体掉到 0.76**。更糟的是,**第一阶段漏掉的实体,第二阶段永远没机会补回来**(召回率被卡死),而第一阶段的假实体会给第二阶段灌进根本不该判关系的实体对(精确率被拉低)。这就是为什么联合抽取被提出来。

### 3.3 联合抽取:让两个任务互相约束

**联合抽取(joint extraction)**:用一个模型同时输出实体和关系,让两个子任务**共享表示、互相约束、误差不串行累积**。核心动机:实体抽取和关系抽取的信息是**双向有用**的——知道"创立"这个关系存在,反过来帮你确认两端是机构和人;知道两个实体的类型,帮你筛掉不合理的关系。

联合抽取有几条主流技术路线,机制各不同:

**(a)标注融合(把关系塞进标签)。** 早期方法(如 Zheng 2017 的 novel tagging)设计一套复合标签,把"实体角色 + 关系类型"编进 BIO 标签里,用一个序列标注模型一次输出。优点是简单;**根本缺陷是无法处理实体重叠**(一个实体参与多个关系时,它只能有一个标签)——这是 SPO 重叠(EntityPairOverlap / SingleEntityOverlap)问题。

**(b)表填充(table filling)。** 把句子的 T 个 token 摆成一个 T×T 的表,表的对角线填实体标签,非对角线 (i,j) 填"token i 所在实体 与 token j 所在实体"的关系。一次预测整张表,天然支持重叠关系。代价是 O(T²) 的输出空间和标注稀疏。

**(c)span + 关系联合(SpERT 式)。** 先用 2.3 的 span 分类抽实体 span,再对所有实体 span 对做关系分类,**两个任务共享同一个 BERT 编码器、联合训练**。这是目前句子级联合抽取的主流形态之一。

**(d)序列生成式(seq2seq / 指针)。** 把三元组抽取转成一个生成问题:用编码器-解码器(seq2seq 框架见第 05 章)直接"生成"出 `(head, relation, tail)(head, relation, tail)...` 这样的序列,或用拷贝/指针机制从原文拷实体。CopyRE、以及后来的生成式 IE(如把抽取统一成文本生成的 TANL、UIE 思路)都属此类。**这条路线和后面 LLM 抽取是连续的**——LLM 抽取本质就是最强的生成式联合抽取。

下面用伪代码对照 pipeline 与联合的结构差异,把"误差是否串行"这件事讲清:

```python
# ===== pipeline:两阶段，误差串行累积 =====
def pipeline_extract(text):
    entities = ner_model(text)              # 阶段1：错了，阶段2无从补救
    triples = []
    for ea, eb in itertools.permutations(entities, 2):
        marked = insert_entity_markers(text, ea, eb)
        rel = re_model(marked)              # 阶段2：只在 entities 正确时才可能对
        if rel != "no_relation":
            triples.append((ea, rel, eb))
    return triples
    # 端到端正确率 ≈ p_ner · p_re，且阶段1漏召回 → 阶段2召回上限被锁死

# ===== 联合（SpERT 式）：共享编码器，联合优化 =====
def joint_extract(text, encoder, span_clf, rel_clf, max_width=10):
    H = encoder(text)                       # 共享上下文表示，一次编码
    # 实体：枚举受限宽度的 span，分类
    spans = enumerate_spans(len(H), max_width)
    span_logits = [span_clf(span_repr(H, s)) for s in spans]
    ents = [s for s, lg in zip(spans, span_logits) if argmax(lg) != "none"]
    # 关系：在抽出的实体 span 对上分类，复用同一个 H
    triples = []
    for ea, eb in itertools.permutations(ents, 2):
        rel = rel_clf(pair_repr(H, ea, eb))
        if argmax(rel) != "no_relation":
            triples.append((ea, argmax(rel), eb))
    return triples
    # 关键：实体损失 + 关系损失对同一个 encoder 反传，梯度互相影响 →
    # 表示被两个任务共同塑形，缓解（不是消除）误差传播

# 联合训练的总损失（机制核心）：
#   L = L_span(实体分类) + λ · L_rel(关系分类)
#   两项梯度都流回共享 encoder，λ 平衡两个任务
```

**这段对照的核心信息增量**:pipeline 的 `ner_model` 和 `re_model` 是**两个独立训练、独立编码**的模型,误差只能串行;联合方法的关键是**共享同一个 `encoder` 并把两个 loss 加起来反传**,使表示被两任务共同塑形。**但要诚实**:联合抽取**缓解**而非**消除**误差传播——它仍然先抽 span 再判关系,漏掉的 span 同样进不了关系分类。真正彻底绕开"先抽实体"的是生成式(直接生成三元组)和下面的 LLM 抽取。

### 3.4 pipeline 不总是输:一个反转

近年有个反直觉的发现值得记住:**精心设计的 pipeline 可以打过联合模型**。Zhong & Chen(2021,PURE)指出,只要(1)实体模型和关系模型用**各自独立**的编码器(而非强行共享),(2)在关系模型输入里**显式注入实体类型标记**,这样的"两个独立模型"反而比当时的联合模型更强。**启示:共享表示不是免费午餐**——强行共享会让两个任务的表示互相"打架"(NER 想要的特征和 RE 想要的特征未必一致)。这告诉你"联合 vs pipeline"不是非此即彼,**关键在于信息如何在两阶段间传递**。**待核:PURE 的具体 F1 提升数值以原文为准。**

## 四、事件抽取与共指、实体链接

### 4.1 事件抽取:触发词 + 论元角色

**事件抽取(Event Extraction, EE)** 比关系更复杂——它抽的是**带角色槽的动态结构**。一个事件由两部分组成:
- **触发词(trigger)**:标志事件发生的词,通常是动词或名词化动词(如"收购""辞职""爆炸"),它决定**事件类型**。
- **论元(argument)**:参与事件的实体,每个填进一个**角色槽(role)**。

例:"2023 年微软以 687 亿美元收购了动视暴雪"
```
事件类型: Acquisition（由触发词"收购"决定）
论元角色填充:
  Acquirer（收购方）= 微软
  Target（被收购方）= 动视暴雪
  Price（金额）     = 687 亿美元
  Time（时间）      = 2023 年
```

**事件抽取本质是"先识别触发词(事件类型分类)→ 再做论元角色填充(把实体填进 schema 的槽)"的两阶段结构预测**,所以它继承了 RE 的全部 pipeline 与联合权衡,还多了一层:**论元角色依赖于事件类型**(同一个实体在"收购"事件里是 Acquirer,在"诉讼"事件里是 Plaintiff)。经典数据集是 ACE 2005(定义了 33 种事件类型和角色 schema)。难点在于:论元可能跨句、一个触发词可能触发多个论元、同一实体可能在多个事件里担任不同角色。

### 4.2 共指消解:把指代统一到同一实体

**共指消解(coreference resolution)**:判断文本中哪些提及(代词"他/它"、名词短语"该公司""这位 CEO"、专名)指向**同一个现实实体**,把它们聚成一个**共指簇(coreference cluster)**。

例:"马斯克收购了推特。**他**随后解雇了 CEO。**这家公司**改名为 X。"
```
簇1 = {马斯克, 他}
簇2 = {推特, 这家公司, X}
```

**为什么 IE 离不开共指**:RE 和事件抽取经常要跨句连关系,而跨句的实体往往是用代词或别称提的。如果不做共指,"他随后解雇了 CEO"里的"他"就接不回"马斯克",这条关系就抽丢了。**共指是文档级抽取的黏合剂**。

主流方法是 **mention-ranking(提及排序)**:对每个提及 mᵢ,在它之前的所有提及(加一个虚拟的"无先行词"哨兵 ε)里,打分选一个最可能的先行词(antecedent)。设候选先行词集合为 {ε, m₁, ..., mᵢ₋₁},模型给每个候选一个分数 s(mᵢ, mⱼ),用 softmax 选:

```
P(先行词 = mⱼ | mᵢ) = exp s(mᵢ, mⱼ) / Σ_{m' ∈ {ε,m₁,...,mᵢ₋₁}} exp s(mᵢ, m')
训练目标：最大化"选中正确簇内任一先行词"的对数似然（簇内任何一个都算对）
```

端到端的代表是 Lee et al.(2017)的 **e2e-coref**:它**联合**地枚举所有候选 span(同样限制最大宽度,O(T²) 用剪枝压到可行)、判断哪些是提及、再做 mention-ranking,**不依赖外部 NER**。这是"联合"思想在共指上的体现。**待核:e2e-coref 的 CoNLL-2012 F1 数值以原文为准。**

### 4.3 实体链接:把提及挂到知识库

**实体链接(Entity Linking, EL)**,又叫实体消歧(entity disambiguation):把文本里的实体**提及(mention)** 链接到**知识库(KB,如 Wikidata)里的唯一实体 ID**。这是 IE 通向"世界知识"的最后一跳——NER 告诉你"这是个机构",EL 告诉你"这个机构就是 Wikidata 里的 Q312"。

为什么难?核心是**歧义(ambiguity)**:同一个 mention 字符串可能对应多个 KB 实体。"苹果"既可以是水果也可以是 Apple Inc.;"Michael Jordan"既可以是篮球运动员也可以是伯克利的机器学习教授。EL 必须**用上下文消歧**。

EL 的标准三阶段流水线:
```
(1) 提及识别(mention detection)：找出要链接的 span（常复用 NER）
(2) 候选生成(candidate generation)：用别名词典/搜索，为 mention 召回一批 KB 候选实体
                                    （如"苹果" → {Apple Inc.(Q312), 苹果(水果)(Q89), ...}）
(3) 候选排序(candidate ranking / 消歧)：用上下文给每个候选打分，选最高
```

第(3)步消歧的核心机制是**计算 mention 上下文与候选实体描述的相似度**。现代做法用**双编码器(bi-encoder,延续第 10 章 DPR 思想)**:用一个编码器把"mention + 句子上下文"编成向量,另一个把"KB 实体的描述(如 Wikipedia 首段)"编成向量,取内积最大者。BLINK(Wu et al., 2020)是这条线的代表:bi-encoder 快速召回候选,再用 cross-encoder(把 mention 上下文和实体描述拼一起,精排)做精确消歧——**这正是第 10 章"双编码器召回 + cross-encoder 重排"在 EL 上的复用**。

```
消歧打分（bi-encoder 形态，复用 DPR/句向量章的内积检索）：
  score(mention m, candidate e) = E_ctx([上下文, m]) · E_ent([e 的描述])
  argmaxₑ score(m, e)，再用 cross-encoder 对 top-k 精排
关键先验：mention 字符串先验 P(e|m)（"苹果"指 Apple Inc. 的基础概率）
         + 上下文一致性（句子里出现"营收""iPhone"→ 偏向 Apple Inc.）
         + 实体间一致性（同篇文章里其他实体也是科技公司 → 强化）
```

**坑:NIL 检测**——mention 可能根本不在 KB 里(新创公司、罕见人名),这时候必须输出 NIL(无链接)而不是硬塞一个最像的。漏掉 NIL 检测会制造大量错误链接,是 EL 上线后的常见事故。

## 五、方法演进:特征工程 → CRF → BiLSTM-CRF → BERT → LLM

把前面四节的任务横切一刀,看**同一个 NER/RE 任务,方法是怎么一代代演进的**,以及每代的**精度-成本-可控**权衡。这是本章的脊柱。

### 5.1 五代方法的机制与权衡

**第一代:特征工程 + CRF / SVM(2003–2014)。** 人手设计特征(词本身、前后缀、是否大写、词性、词典命中、上下文窗口词),喂给 CRF(序列标注)或 SVM(分类)。**精度**取决于特征工程师的功力,领域迁移要重做特征。**成本**:推理极快(线性模型),但**特征工程是巨大的人力成本**,且脆——换个领域全废。**可控**:高,每个特征可解释。

**第二代:BiLSTM-CRF(2015–2018)。** 第 04 章讲透的经典架构:双向 LSTM(第 03 章)自动学上下文特征(替代人手特征),CRF 层(第 04 章)建模标签转移。**精度**:在当时是 SOTA,免去特征工程。**成本**:要标注数据训练,推理比线性模型慢但可接受。**可控**:中,LSTM 是黑箱但 CRF 转移可读。**这一代的意义是"特征自动化"**——不再手工设计特征,但仍需大量标注。

**第三代:BERT 微调(2019–2022)。** 预训练章讲透的范式:BERT 在海量无标注文本上预训练,在 NER/RE 标注数据上微调。**精度**:大幅超越 BiLSTM-CRF,且**预训练让小样本微调也能work**(知识从预训练迁移)。**成本**:推理比 BiLSTM 重(几亿参数),但**标注需求大降**(预训练扛了大部分)。**可控**:中低,更黑箱。**这一代的意义是"特征 + 知识都从预训练来"**,标注成本进一步降到"几百到几千条就能微调出可用模型"。

**第四代:LLM 抽取(2023–)。** 用 GPT-4/Claude 这类生成式大模型,通过 **prompt** 直接让它读文本、按要求输出结构(JSON 三元组)。**精度**:在**零样本/少样本**和**开放/未见过的 schema**上惊人地好(不需要为新关系类型重新训练);但在**有充足标注数据的固定 schema**上,微调的小 BERT 往往**精度更高、更稳定**。**成本**:零训练、零标注即可起步(这是最大优势),但**推理成本高**(API 费用 / 大模型显存)、**延迟高**。**可控**:这是 LLM 抽取最大的痛点——**输出格式不稳定、会幻觉出文本里没有的实体/关系、schema 遵守不可靠**。这就是第七节"结构化输出工程"要解决的。

把权衡总结成一张表(便于决策):

```
方法            标注需求    推理成本   零样本/新schema   精度(数据充足)   输出可控性
特征+CRF         高(+特征)   极低        差               中             高
BiLSTM-CRF       高          低          差               中高           中
BERT 微调        中          中          差(需微调)       高             中
LLM prompt       零/少样本   高          强               中高           低(需约束)
```

**决策规则(承接第 09 章"微调 vs prompting"):**
- schema 固定、有标注、要高吞吐低延迟、要稳定 → **微调 BERT**(精度高、可控、便宜推理)。
- schema 频繁变化 / 关系类型开放 / 没有标注数据 / 需求快速原型 → **LLM prompt**(灵活、零启动成本)。
- 折中方案:**LLM 当标注器**(零样本抽出一批伪标注)→ **蒸馏/微调一个小 BERT** 上线。这是 2024 年很流行的"用大模型造数据、用小模型上线"的范式,兼顾精度成本和可控。

### 5.2 远程监督:用 KB 自动标注,省标注但引入噪声

标注 RE 数据极贵。**远程监督(Distant Supervision, Mintz et al., 2009)** 是一个绕开人工标注的聪明办法,但它有一个**根本性的噪声问题**,必须讲透。

**核心假设(distant supervision assumption):** 如果知识库里已知三元组 (h, r, t)(如 (微软, 收购, 动视暴雪)),那么**任何同时提到 h 和 t 的句子,都被自动标注为表达关系 r**。这样你不用人工标,直接用 KB + 大语料就生成出海量(噪声)训练数据。

**为什么有噪声(第一性原理):** 这个假设**太强了**。同时提到微软和动视暴雪的句子,**未必在说"收购"**——可能是"微软和动视暴雪都将参加 E3 展会"(根本没收购关系),也可能是"在收购动视暴雪之前,微软曾被反垄断调查"(提到两者但句子主语是别的)。于是产生两类噪声:

```
假阳(false positive)：句子同时含 h、t，但并不表达关系 r → 被错标为 r
假阴(false negative)：KB 不完整，某真实关系不在 KB 里 → 含 h,t 的句子被标为"无关系"
```

**怎么缓解(机制要点):**
- **多示例学习(Multi-Instance Learning, MIL)**:不再假设"每个句子都对",而是放松成**"提到 (h,t) 的句子集合里,至少有一句表达 r"**(at-least-one 假设)。把同一实体对的所有句子打成一个 bag,在 bag 级别预测关系,允许 bag 内有噪声句。
- **句子级注意力(Lin et al., 2016, PCNN+ATT)**:在 bag 内对句子做注意力加权,**让模型自己学会给"真正表达关系的句子"高权重、给噪声句低权重**。这是远程监督去噪的标志性工作。
- **现代视角**:LLM 零样本抽取某种程度上**替代了远程监督**——不再需要 KB 对齐造噪声数据,直接让大模型读句子判关系。但远程监督的思想(用已有结构化知识自动生成监督信号)在垂直领域仍有用。

**这就是练习里要你"解释远程监督的噪声"的答案核心**:远程监督用"实体对共现 ⇒ 关系成立"的强假设换标注成本,代价是假阳/假阴噪声,需用 MIL + 注意力在 bag 级别去噪。

## 六、知识图谱:从三元组到嵌入,再到 KG+LLM

### 6.1 三元组与知识图谱

抽取的终点是把实体和关系组织成**知识图谱(Knowledge Graph, KG)**:节点是实体,边是关系,每条边就是一个**三元组 (head, relation, tail)**,简记 (h, r, t)。如 (北京, 首都_of, 中国)、(Apple, founded_by, Steve Jobs)。Wikidata、Freebase(已停)、DBpedia 是著名的通用 KG。

KG 的价值在第一节说过:可查询、可聚合、可推理、可校验。但 KG 有个固有问题——**不完整(incompleteness)**:再大的 KG 也漏掉海量真实关系。于是有了 **KG 补全(KG completion)** 任务:给定已有的图,**预测缺失的边**(链接预测)。这就引出 KG 嵌入。

### 6.2 KG 嵌入与 TransE:h + r ≈ t 的几何直觉

**KG 嵌入(KG embedding)** 的目标:把每个实体和每个关系映射成低维向量,使得**向量运算能反映图的结构**,从而能给任意 (h, r, t) 打一个"成立可信度"分,用于补全。

最经典、最该讲透的是 **TransE(Bordes et al., 2013)**。它的核心思想极其简洁——**把关系建模成实体向量空间里的一个平移(translation)**:

```
若三元组 (h, r, t) 成立，则希望：h + r ≈ t
即头实体向量 + 关系向量 ≈ 尾实体向量
```

几何直觉:从"北京"这个点出发,加上"首都_of"这个**平移向量**,应该落到"中国"附近。同一个关系 r 对所有满足它的实体对,都是**同一个平移**——(北京, 首都_of, 中国) 和 (东京, 首都_of, 日本) 共享同一个 r 向量,所以 `中国−北京 ≈ 日本−东京 ≈ r_首都`。**这正是 word2vec 类比(king−man+woman≈queen,第 02 章)在知识图谱上的翻版**——关系是向量空间里的方向。

**打分函数(score / 能量函数):** 用 h+r 与 t 的距离衡量三元组成立程度,距离越小越可信:

```
f(h, r, t) = ‖ h + r − t ‖   （L1 或 L2 范数）
成立的三元组 f 应小，不成立的应大
```

**训练:margin-based ranking loss(基于间隔的排序损失)。** 我们没有"负三元组"标注,所以**负采样**:把一个真三元组 (h,r,t) 的头或尾随机替换成别的实体,造出"破坏过的"负样本 (h′,r,t′)。目标是让**正样本的分数比负样本至少低一个间隔 γ**:

```
L = Σ_{(h,r,t)∈正} Σ_{(h′,r,t′)∈负} [ γ + f(h,r,t) − f(h′,r,t′) ]₊
其中 [x]₊ = max(0, x) 是 hinge，γ>0 是间隔超参
负样本：(h′,r,t′) = 替换 h 或 t 为随机实体（破坏掉的三元组）
约束：实体向量做 L2 归一化 ‖e‖=1，防止模型靠把向量推到无穷来作弊降 loss
```

**为什么要这个 hinge 和归一化(第一性原理):** hinge 只惩罚"正样本没比负样本低够 γ"的情况,正负已拉开足够间隔时 loss=0,不再无意义地拉大——这给了模型容错空间。归一化是关键的**工程坑**:若不约束实体向量范数,优化器会发现"把所有 t 的范数推大"能让 ‖h+r−t‖ 这类项变化,从而走捷径降低 loss 而非学到结构,所以 TransE 强制 ‖e‖=1。

下面是 TransE 的可运行核心,把上面的机制落成代码:

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class TransE(nn.Module):
    def __init__(self, n_entities, n_relations, dim=50, margin=1.0, p=1):
        super().__init__()
        self.margin, self.p = margin, p
        self.ent = nn.Embedding(n_entities, dim)
        self.rel = nn.Embedding(n_relations, dim)
        nn.init.xavier_uniform_(self.ent.weight)
        nn.init.xavier_uniform_(self.rel.weight)

    def score(self, h, r, t):
        # f(h,r,t) = ||h + r - t||_p ，越小越可信
        return torch.norm(self.ent(h) + self.rel(r) - self.ent(t), p=self.p, dim=-1)

    def forward(self, pos, neg):
        # pos/neg: [B,3] 的 (h,r,t) 三元组索引；neg 是替换头或尾得到的负样本
        self.ent.weight.data = F.normalize(self.ent.weight.data, p=2, dim=-1)  # 约束 ||e||=1
        pos_d = self.score(pos[:,0], pos[:,1], pos[:,2])
        neg_d = self.score(neg[:,0], neg[:,1], neg[:,2])
        # margin ranking：正样本距离 + margin 仍要小于负样本距离
        loss = torch.clamp(self.margin + pos_d - neg_d, min=0).mean()
        return loss

def corrupt(triple, n_entities):
    """负采样：以 0.5 概率替换头或尾为随机实体（造破坏过的三元组）"""
    h, r, t = triple
    if torch.rand(1).item() < 0.5:
        return (torch.randint(0, n_entities, (1,)).item(), r, t)
    return (h, r, torch.randint(0, n_entities, (1,)).item())

# 链接预测推理：补全 (北京, 首都_of, ?) → 对所有候选 t 算 score，取最小
```

**TransE 的局限(讲透才完整):** h+r≈t 的平移假设对 **1-to-1 关系**(首都)很好,但**处理不了 1-to-N / N-to-1 / N-N 关系**。例如关系"导演的作品",一个导演 h 有多部作品 t₁,t₂,...,要求 h+r≈t₁ 且 h+r≈t₂,逼着 t₁≈t₂——把不同作品的向量挤到一起,矛盾。也处理不了**对称关系**(若 r 对称,(h,r,t) 和 (t,r,h) 都成立,要 h+r≈t 且 t+r≈h,推出 r≈0)。这些缺陷催生了后续一长串改进:TransH(把实体投影到关系特定超平面)、TransR(实体和关系在不同空间)、DistMult(双线性打分 hᵀ diag(r) t)、ComplEx(复数空间处理非对称)、RotatE(关系建模成复平面旋转,优雅处理对称/反对称/逆关系)。**待核:各模型在 FB15k/WN18 上的具体 MRR/Hits@10 数值以原文为准。** 记住一条主线:**它们都在"用一个可计算的几何/代数操作表示关系"这个框架下,逐步放宽 TransE 平移假设的限制**。

### 6.3 KG 与 LLM 结合:互补的两种知识

KG 和 LLM 各存一种知识,且**正好互补**——这是 2024 年后最热的方向之一:
- **LLM 的知识**:存在参数里,**覆盖广、会推理、但会幻觉、难更新、难追溯**。
- **KG 的知识**:存在三元组里,**精确、可追溯、可更新、但稀疏、不完整、不会推理**。

结合的三种主流形态:

**(a)KG 增强检索(KG-augmented RAG)。** 第 10 章 RAG 检索文档块,这里改成/补充**检索 KG 子图**。问"A 和 C 有什么关系",在图上做多跳查询,把路径 (A→B→C) 作为结构化证据喂给 LLM。优势是**支持多跳推理**(文档块检索做不到的),且证据是结构化、可解释的。

**(b)KG 做事实校验(fact verification)。** LLM 生成回答后,把其中的事实性断言抽成三元组(又用到 IE!),去 KG 里**核对这条边存不存在**。模型说"X 是 Y 的子公司",查图里有没有 (X, subsidiary_of, Y),没有就标"未经证实"或拒答。这是给 LLM 套**外部事实约束**、抑制幻觉的硬手段,在高风险领域很关键。

**(c)用 LLM 反过来建/补 KG。** 前面整章讲的就是这个——**用 LLM 做 IE,把文本抽成三元组来扩充 KG**。形成闭环:LLM 帮建 KG,KG 帮校验 LLM。

### 6.4 文档智能:版面、表格、OCR 的结构抽取

前面假设输入是纯文本,但真实世界的知识大量躺在 **PDF、扫描件、发票、财报、合同**里——它们有**版面(layout)**,信息靠空间位置组织(表格的行列、票据的字段位置),纯文本流会丢掉这层结构。**文档智能(Document AI)** 处理这类视觉富文本(visually-rich document)。

核心挑战与方法:
- **OCR**:扫描件先要光学字符识别拿到文字 + 每个文字的**坐标框(bounding box)**。坐标是关键——版面信息全在坐标里。
- **版面理解**:把"文本 + 2D 坐标"一起建模。**LayoutLM 系列(Xu et al., 2020 起)** 是代表:在 BERT 的 token embedding 上**加入 2D 位置嵌入**(token 的 x/y 坐标),后续版本(LayoutLMv2/v3)再融入**视觉特征**(把文档当图像过视觉编码器),做到文本+版面+视觉三模态。这样"表格里同一行的两个数"能通过坐标知道它们相关。
- **表格抽取**:识别表格结构(几行几列、单元格合并)再抽内容,常作为独立子任务。
- **下游任务**:表单理解(键值对抽取,如发票号→金额)、文档分类、文档 VQA。

**和本章主线的连接**:文档智能本质是**把 NER/RE 从一维文本流扩展到二维版面**——实体还是要识别,关系(如"字段名↔字段值")还是要抽,只是特征里多了坐标和视觉。**现在多模态 LLM(能直接读文档图像)正在吞并这一块**,但理解"为什么需要 2D 位置嵌入"仍是基本功。

## 七、结构化输出的工程:让模型可靠产出 schema

这是把 LLM 抽取**真正用上生产**的关键一节,也是本章要求的"LLM 约束抽取 prompt"对照组所在。问题很直接:**LLM 生成是自由文本,但 IE 要的是严格符合 schema 的 JSON,怎么保证它不跑偏?**

### 7.1 三个层次的手段

**层次一:prompt 工程(最轻,不保证)。** 在 prompt 里给出 schema 定义、字段说明、few-shot 示例(承接第 09 章 ICL),要求"只输出 JSON"。这是下面的对照 prompt:

```python
EXTRACTION_PROMPT = """你是信息抽取引擎。从文本中抽取公司收购事件，
严格按下面 JSON schema 输出，不要任何额外解释文字。
未出现的字段填 null，不要编造文本里没有的信息。

Schema:
{
  "event_type": "Acquisition",
  "acquirer": "收购方公司名 | null",
  "target":   "被收购方公司名 | null",
  "price":    "金额(含币种) | null",
  "date":     "时间 | null"
}

示例:
文本: "2016年微软以262亿美元收购LinkedIn。"
输出: {"event_type":"Acquisition","acquirer":"微软","target":"LinkedIn","price":"262亿美元","date":"2016年"}

文本: "{input_text}"
输出:"""
# 对照 BertNER：BERT 做 NER 是逐 token 多分类、输出标签序列再解码成 span；
# LLM 抽取是"读文本直接生成结构 JSON"——零训练、能处理开放 schema，
# 但格式与忠实性不被算法保证，必须靠下面的约束 + 校验兜底。
```

prompt 工程的根本缺陷:**它是"请求"不是"保证"**。模型仍可能输出多余的解释、漏字段、JSON 语法错(少引号/逗号)、甚至幻觉出文本里没有的公司。N=1 看着对,放量必出问题。

**层次二:约束/语法解码(硬保证格式)。** 这是从**解码层面**强制输出合法。机制:在每一步解码时,根据目标语法(JSON schema 对应的形式文法),**把所有会导致非法的 token 的概率置零**,只在合法 token 集合里采样。这样**输出在语法上 100% 合法**(一定是可解析的 JSON、字段一定在 schema 里)。实现路径有受限解码(constrained decoding)、用有限状态机/CFG 约束(如 Outlines、llama.cpp 的 GBNF 语法、guidance、jsonformer 等库),以及 OpenAI/部分 API 的 structured output / JSON mode。**这套机制的解码细节详见算法课第 13 章(约束解码那节)**,本章只强调它在 IE 里的价值:**把"格式正确"从概率问题变成确定性问题**。

**层次三:校验与纠错(语义兜底)。** 语法合法 ≠ 内容正确。约束解码保证是合法 JSON,但管不了"幻觉出文本里没有的实体"。所以要加一道**后处理校验闭环**:

```python
import json, jsonschema

def robust_extract(text, llm, schema, max_retry=2):
    for attempt in range(max_retry + 1):
        raw = llm(EXTRACTION_PROMPT.format(input_text=text))
        # 1) 语法校验：能不能解析成 JSON、符不符合 schema
        try:
            obj = json.loads(raw)
            jsonschema.validate(obj, schema)          # 字段/类型校验
        except (json.JSONDecodeError, jsonschema.ValidationError) as e:
            text_feedback = f"上次输出不合法：{e}。请严格按 schema 重新输出。"
            continue                                   # 带错误反馈重试（self-correction）
        # 2) 忠实性校验(grounding)：抽出的实体串必须真出现在原文里，防幻觉
        for field in ("acquirer", "target"):
            v = obj.get(field)
            if v and v not in text:                    # 关键：抗幻觉的硬规则
                obj[field] = None                      # 文本里查无此实体 → 置空，不硬塞
        return obj
    return None   # 多次重试仍失败 → 显式失败，交人工/降级，绝不返回脏数据
```

**这段代码的核心信息增量(也是本章方法论的落点)**:可靠的结构化抽取 = **约束解码保语法 + schema 校验保结构 + 忠实性校验(抽出的内容必须能在原文里找到)保抗幻觉 + 带反馈重试 + 失败显式上报**。**忠实性校验那一行(`v not in text → None`)是抗幻觉最便宜也最有效的护栏**——它把"模型说有"约束成"原文里查得到才算数",直接呼应第 10 章 RAG 的"答案必须由证据支撑"的忠实性思想。

### 7.2 为什么这是 IE 落地的真正瓶颈

回到 5.1 的权衡表:LLM 抽取的"低可控性"是它唯一的硬伤,而 7.1 三层手段就是在补这个硬伤。**一个工业级 LLM 抽取方案的可靠性,几乎不取决于 prompt 写得多花哨,而取决于约束解码 + 校验闭环做得多严**。这正是用户私规则里"实现带语义负载字段要先问用户怎么理解"的精神在 IE 上的体现——**你不能假设模型输出的 JSON 字段语义就等于 schema 定义,必须用校验把这个假设变成保证**。

## 八、设计权衡与常见坑

把全章的权衡和坑收口成一张清单,这些是上线后真正咬人的地方:

- **pipeline vs 联合不是非此即彼。** 联合**缓解**误差传播但不消除,且强行共享表示可能让任务互相打架(4.3 PURE 的教训)。**决策点是"信息如何在子任务间流动",而非"用不用一个模型"**。数据多、schema 固定优先试精心设计的 pipeline + 实体类型注入;数据少、要端到端优先联合。
- **subword 对齐是 NER 的头号 bug 源。** 训练时 label 挂在哪个 subword、评估时怎么合并回词级,搞错指标全错。**评估必须在词级/实体级,不能在 subword 级。**
- **NER 的评估口径要钉死。** 实体级 F1 分**严格匹配**(边界+类型都对)和**宽松匹配**(部分重叠也算),报数前必须说清用哪种——这是论文复现对不上的常见原因。
- **远程监督的噪声是结构性的,不是调参能解决的。** 假阳来自"共现≠关系"的强假设,假阴来自 KB 不完整。必须用 MIL + 注意力在 bag 级去噪,别指望句子级硬标。
- **实体链接必须做 NIL 检测。** KB 不含的 mention 要敢输出"无链接",硬塞最像的会制造大量错误边,污染下游 KG。
- **TransE 别用在 N-N / 对称关系上。** 平移假设在这两类关系上数学上就矛盾,该换 RotatE/ComplEx。**选 KG 嵌入模型前先看你的关系类型分布。**
- **LLM 抽取:格式合法 ≠ 内容忠实。** 约束解码只保语法,**幻觉要靠"抽出的实体必须在原文出现"的忠实性校验单独兜底**。这两件事是正交的,缺一不可。
- **schema 演化是长期成本。** 微调模型每加一个关系/实体类型就要重标重训;LLM prompt 改 schema 近乎零成本——**如果你的 schema 会频繁变,这个长期成本可能压倒 LLM 的高推理单价**,决策时要把 TCO(总拥有成本)算进去,不只看单次推理。
- **别用 LLM 抽取去做有充足标注的固定 schema 高频任务。** 那是微调小模型的主场(更准、更稳、更便宜)。LLM 的主场是低资源、开放 schema、快速原型。**用错场景是常见的成本浪费。**

## 九、动手练习

**练习 1(pipeline vs 联合的误差传播,推导+实验)。** 设 NER 实体对召回率 p_ner=0.9、关系分类准确率 p_re=0.85。(a)推导 pipeline 端到端抽对一个三元组的概率上界,解释为什么"漏召回的实体阶段二补不回来"会进一步压低召回。(b)用 spaCy 或一个微调 BERT 在某个 RE 数据集(如 SemEval-2010 Task 8)上分别跑 pipeline(独立 NER + 独立 RE)和一个联合基线,比较端到端 F1,并定位误差主要来自哪一阶段。**提示:** 联合的关键代码差异在"共享 encoder + 两个 loss 相加反传"(回看 3.3),误差归因看"实体抽错导致的关系错"占比。

**练习 2(远程监督的噪声,分析)。** 给定 KB 三元组 (微软, 收购, 动视暴雪),你用远程监督从语料里抓所有同时含这两个实体的句子标为"收购"。(a)各举两个**假阳**(含两实体但不表达收购)和**假阴**(KB 缺失导致真关系被标为无关系)的具体句子。(b)解释多示例学习(MIL)的 at-least-one 假设如何把句子级噪声降到 bag 级,以及句子级注意力(PCNN+ATT)为什么能进一步去噪。**提示:** 假阳的本质是"共现≠关系",注意力的本质是"让模型自己给真正表达关系的句子高权重"。

**练习 3(TransE 推导与局限)。** (a)写出 TransE 的打分 f(h,r,t)=‖h+r−t‖ 和 margin ranking loss,并解释为什么必须约束 ‖e‖=1(不约束会怎样作弊?)。(b)严格论证:对一个 1-to-N 关系(如"导演→作品",一个导演多部作品),TransE 的平移假设会导致什么矛盾?对对称关系呢?(c)说明 RotatE 用"复平面旋转"建模关系如何能同时处理对称、反对称、逆关系。**提示:** (b)把"h+r≈t₁ 且 h+r≈t₂"两式相减;对称关系把 (h,r,t) 和 (t,r,h) 两个约束联立。

**练习 4(设计一个稳健的结构化抽取方案,综合)。** 需求:从公司财报 PDF 里抽"季度营收"事件(公司、季度、营收金额、同比变化),schema 可能随业务调整。请设计端到端方案并说明每个决策:(a)文档是 PDF,如何处理版面/表格(7.4/6.4)?(b)用微调小模型还是 LLM 抽取?给出决策依据(回看 5.1 决策规则,考虑 schema 会变这一点)。(c)如何保证输出是合法 JSON 且金额、公司名不是幻觉的?画出"约束解码 + schema 校验 + 忠实性校验 + 重试 + 失败上报"的完整闭环。(d)如何把抽出的三元组写进 KG,并用 KG 反过来校验 LLM 后续问答的事实性?**提示:** 答案应当落在 7.1 三层手段 + 6.3 KG+LLM 校验上,且 schema 易变这一条会把(b)推向"LLM 抽取 + 忠实性校验"或"LLM 造数据蒸馏小模型"。

## 十、源码 / 论文导读

- **NER / 序列标注**:HuggingFace `transformers` 的 `AutoModelForTokenClassification` 是 BERT+softmax NER 的标准实现;官方 `token-classification` 示例脚本(`run_ner.py`)演示了 subword 对齐(看 `tokenize_and_align_labels` 函数,这是 2.2 节那个坑的标准答案)。BiLSTM-CRF 与 CRF 推导回看**本课第 04 章**。
- **关系抽取**:读 **PURE**(Zhong & Chen, 2021,"A Frustratingly Easy Approach for Entity and Relation Extraction",NAACL)体会"独立编码 + 实体类型注入 > 强行联合"的反转;联合抽取读 **SpERT**(Eberts & Ulges, 2020,ECAI / arXiv:1909.07755)的 span 分类 + 关系联合架构(代码开源,看它如何在一个 BERT 上挂两个分类头并联合训练)。**待核:作者拼写与年份以原文为准。**
- **远程监督**:Mintz et al.(2009)提出远程监督;**PCNN+ATT**(Lin et al., 2016,"Neural Relation Extraction with Selective Attention over Instances")是 bag 级注意力去噪的必读,精读它的注意力公式如何加权 bag 内句子。
- **共指消解**:Lee et al.(2017)**e2e-coref**(end-to-end neural coreference resolution),读它怎么在不依赖外部 NER 的情况下联合枚举 span + mention-ranking;后续 SpanBERT 在共指上的提升说明"span 级预训练"对结构任务的价值。
- **实体链接**:**BLINK**(Wu et al., 2020,Facebook),开源,bi-encoder 召回 + cross-encoder 精排,是 EL 复用 DPR 思想(第 10 章)的范本。
- **KG 嵌入**:TransE 原论文(Bordes et al., 2013,"Translating Embeddings for Modeling Multi-relational Data");工具库 **OpenKE** / **PyKEEN** 提供 TransE/TransR/DistMult/ComplEx/RotatE 的统一实现,适合对照看不同打分函数怎么落代码。RotatE 读 Sun et al.(2019)。
- **文档智能**:**LayoutLM** 系列(Xu et al., 2020 起,微软),`transformers` 里有 `LayoutLMv3` 实现,精读它的 **2D 位置嵌入**怎么把坐标编进 BERT(6.4 节的核心)。
- **结构化输出 / 约束解码**:库 **Outlines**、**guidance**、**jsonformer**、llama.cpp 的 **GBNF** 语法约束,读它们如何用有限状态机/CFG 在解码时屏蔽非法 token;**约束解码的算法机制详见本库《大模型算法与架构》课第 13 章**。
- **统一 IE**:**UIE**(Lu et al., 2022,"Unified Structure Generation for Universal Information Extraction")把 NER/RE/事件统一成"结构化生成",是从专用模型走向"一个生成模型抽一切"的代表,也是通往 LLM 抽取的桥。

## 十一、小结与承上启下

这一章把"文本 → 结构"这道工序拆透了:

- **任务层**:NER(承接第 04 章序列标注)抽实体 → RE 抽关系三元组 → 事件抽取抽带角色槽的动态结构 → 实体链接把提及挂到 KB → 共指把指代统一。**它们层层依赖,实体提及是一切的锚点。**
- **方法层**:特征+CRF → BiLSTM-CRF → BERT 微调 → LLM 抽取,主线是**"特征自动化 → 知识从预训练来 → 零样本开放 schema"**,代价从"人力标特征"逐步换成"推理算力 + 输出可控性"。**核心权衡表(5.1)和决策规则是本章最实用的产出。**
- **误差传播**是 IE 的第一性原理:pipeline 的 p_ner·p_re 上界、联合抽取的缓解机制、PURE 的反转,共同说明**关键在子任务间的信息流动**。
- **远程监督**用"共现⇒关系"的强假设换标注成本,代价是结构性噪声,靠 MIL + 注意力去噪。
- **知识层**:三元组 → 知识图谱 → **TransE 的 h+r≈t 平移嵌入**(完整推导了打分、margin loss、归一化的必要性、N-N/对称关系的局限)→ KG 与 LLM 互补结合(KG 增强检索、事实校验、LLM 反哺建图)→ 文档智能把抽取扩到二维版面。
- **工程层**:结构化输出的三层护栏——**约束解码保语法(详见算法课第 13 章)+ schema 校验保结构 + 忠实性校验保抗幻觉 + 重试 + 失败上报**,是 LLM 抽取真正落地的瓶颈所在。

**承上**:本章大量复用了前面的积木——CRF(第 04 章)、seq2seq(第 05 章)、句向量/双编码器检索(第 10 章 DPR、BLINK 的召回精排)、微调 vs prompting 决策(第 09 章)、约束解码(算法课第 13 章)。IE 不是孤立技术,是把这些积木**面向"产出结构化知识"重新组装**。

**启下**:抽出的结构化知识(KG)是**事实校验、多跳推理、可信问答**的底座,但"模型生成的内容是否真由知识/证据支撑"这个**忠实性(faithfulness)** 问题,本章只在 7.1 用忠实性校验碰了一下——它将在后续的**评测与忠实性**章被系统讲透:如何度量摘要/问答/抽取的事实一致性、如何评测 RAG 答案是否真被检索支撑、如何用 NLI/QA 自动检测幻觉。同时,IE 产出的"事实"如何服务于**对话与应用**(第 12 章,把抽取/检索/知识接进任务型对话与 Agent),也是下一步的去向。**把文本变成结构,是为了让机器不仅"读得懂",更能"查得到、算得清、校验得了"。**
