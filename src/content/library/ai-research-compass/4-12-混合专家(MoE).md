> 这一章把你从"知道大模型可以做得很大"带到"能从第一性原理推出 MoE 为什么能让总参数涨几十倍而单 token 算力几乎不变、能徒手写出 top-k 路由的前向、能算清它的参数量/FLOPs/显存、能讲透负载均衡损失为什么有效以及它会在哪里坑你"。MoE 是当代大模型(Mixtral、DeepSeek-V3、据公开报道的 GPT-4 等)突破"参数即算力"硬墙的核心架构,这章要求你对它没有任何黑箱。

## 一句话引言

稠密 Transformer 的致命约束是:**每个 token 都要过全部参数,所以参数量一涨,每一步前向的算力就等比例涨**——算力是扩参的硬墙。MoE 的破局点只有一句话:**把一个大 FFN 拆成 N 个小专家,每个 token 只激活其中 top-k 个**。于是总参数 ∝ N(可涨几十倍),单 token 计算 ∝ k(几乎不变)。本章从这个解耦的算术推起,推到路由器 softmax 的形式、负载均衡损失的完整构造与梯度、capacity factor 与 token 丢弃的取舍,再把 Switch Transformer(top-1)和 GShard(top-2)的关键设计讲透。

## 1. 动机:稠密模型的"参数即算力"硬墙

### 1.1 一个 Transformer block 的算力账

先把稠密 Transformer 一层的计算量算清楚,才知道墙在哪。一个标准 decoder block 由两部分组成:多头注意力(MHA)和前馈网络(FFN,也叫 MLP)。设模型隐藏维度 d_model = d,序列长 n,FFN 中间维度 d_ff。

标准 FFN 的形式(两层带激活):

```
FFN(x) = W₂ · act(W₁ x)
  W₁ ∈ ℝ^{d_ff × d}     (升维)
  W₂ ∈ ℝ^{d × d_ff}     (降维)
  通常 d_ff = 4d        (GPT/原始 Transformer 的惯例)
```

单个 token 过一次 FFN 的乘加量:

```
W₁ x:   d_ff × d 次乘加  ≈ 2 · d_ff · d FLOPs
W₂ ·:   d × d_ff 次乘加  ≈ 2 · d · d_ff FLOPs
合计:    ≈ 4 · d_ff · d  = 4 · (4d) · d = 16 d²  FLOPs / token
```

注意力部分(QKV 投影 + 输出投影)单 token 约 8d² FLOPs(四个 d×d 投影,各 2d²),序列内的 `QKᵀ`/`AV` 还有 O(n·d) 一项。在大模型典型配置下(d 上千、d_ff = 4d),**FFN 占了一个 block 里参数量和计算量的大头**——粗算 FFN 参数 ≈ 2·d·d_ff = 8d²,注意力参数 ≈ 4d²,FFN 是注意力的两倍。所以想"把模型做大",最自然的着力点就是 FFN。

### 1.2 硬墙:稠密扩参的代价是等比例算力

稠密模型有一条铁律:**每个 token 都流过全部参数**。把 d_ff 从 4d 加到 8d,参数翻倍,但每个 token 过 FFN 的 FLOPs 也翻倍。把层数加倍同理。形式化地,稠密模型里:

```
单 token 前向 FLOPs  ∝  总参数量 P
```

这个正比关系就是硬墙。它有两个直接后果:

- **训练成本爆炸**。训练一次的总算力 ≈ 6 · P · D(P 是参数量,D 是训练 token 数,这是 Kaplan / Chinchilla 系工作里常用的经验估计,系数 6 = 前向 2 + 反向 4,**待核**:不同文献对该系数的精确推导略有差异,量级是对的)。P 涨 10 倍,同样数据量下训练算力涨 10 倍。
- **推理成本爆炸**。每生成一个 token 的延迟和能耗都 ∝ P。一个 1.7T 参数的稠密模型,每个 token 都要做 1.7T 量级的乘加,服务成本上天。

**核心矛盾**:我们有强烈的理由相信"参数越多,模型记得越多、能力越强"(scaling law 的经验支撑),但稠密架构把"更多参数"和"更贵的每步计算"死死绑在一起。**能不能让参数(知识容量)和算力(每 token 计算)解耦?** 这就是 MoE 要回答的唯一问题。

> 直觉:稠密模型像让每个问题都去问全公司一万名专家,既慢又浪费。理想做法是有一个"前台"(路由器),看一眼问题就把它转给最相关的一两个专家——专家库可以有一万人(参数多),但每个问题只占用一两人的时间(算力省)。

## 2. 机制:用 N 个专家 + 路由器替换 FFN

### 2.1 MoE 层的定义

MoE 的做法极其直接:**把 Transformer block 里的那个 FFN,替换成 N 个结构相同、参数独立的 FFN(称为"专家",expert),外加一个轻量的路由器(router / gating network)**。注意力部分通常保持稠密不变(注意力的参数占比小,且跨 token 交互不好稀疏化)。

记 N 个专家为 E₁, E₂, …, E_N,每个 Eᵢ 就是一个普通 FFN(各自有独立的 W₁⁽ⁱ⁾, W₂⁽ⁱ⁾)。路由器是一个简单的线性层:

```
路由器:  Wᵣ ∈ ℝ^{N × d}      (把 token 表示映射到 N 个专家的打分)
对 token x ∈ ℝ^d:
  logits = Wᵣ x ∈ ℝ^N         (对每个专家的亲和度分数)
  g = softmax(logits) ∈ ℝ^N    (归一化成分布, Σᵢ gᵢ = 1)
```

如果让 token 过**全部** N 个专家再按 g 加权,那是稠密 MoE,计算量是单专家的 N 倍——没省。MoE 的灵魂在于**稀疏激活(sparse activation)**:每个 token 只挑 g 中最大的 k 个(top-k,k 通常是 1 或 2),只让它过这 k 个专家:

```
设 topk(g) 选出分数最大的 k 个专家下标集合 T = {i₁, …, i_k}
输出:  y = Σ_{i ∈ T} ĝᵢ · Eᵢ(x)
```

这里 ĝᵢ 是路由权重。一个关键细节:是直接用 softmax 的 gᵢ,还是只在 top-k 这 k 个上重新归一化?两种都有人用:

```
方案A (Switch, top-1): 直接用 gᵢ (top-1 时就是 max 那个 softmax 值)
方案B (GShard/Mixtral, top-2): 在选中的 k 个上重新 softmax 归一化
  ĝᵢ = exp(logitsᵢ) / Σ_{j∈T} exp(logitsⱼ)    (只在 T 内部归一化, Σ_{i∈T} ĝᵢ = 1)
```

方案 B 让 k 个权重重新和为 1,语义上更干净(输出是 k 个专家输出的凸组合);方案 A 保留了"全局置信度"信息(如果 token 谁都不太像,gᵢ 会偏小)。Mixtral 用的是先 top-2 再在两者上 softmax 归一化。**实现时一定要明确你用哪种,否则输出尺度会差一个因子。**

### 2.2 为什么乘上 gᵢ 而不只是相加:让路由可微

一个新手常问:既然已经用 top-k 选出了专家,为什么还要乘那个权重 gᵢ?直接 `y = Σ Eᵢ(x)` 不行吗?

不行,而且原因深刻。**top-k 这个"选择"操作本身是不可微的**(argmax / 取 top-k 是离散的,梯度无法穿过"选了谁"这个决策)。如果输出只是被选中专家的简单求和,那么路由器的参数 Wᵣ 就**完全收不到梯度**——它决定"选谁",但"选谁"这个离散决策对 loss 的导数处处为 0(除了不可导的跳变点)。路由器永远学不动,就退化成随机路由。

乘上连续的权重 gᵢ 把可微性救了回来:

```
y = Σ_{i∈T} gᵢ · Eᵢ(x)
∂y/∂(logitsⱼ) 对 j∈T 不为零   (gᵢ 是 logits 的可微 softmax 函数)
```

也就是说,**梯度通过权重 gᵢ 的大小这条连续通路回流到路由器**:如果某个被选中的专家 Eᵢ 的输出对降低 loss 有帮助,梯度会推高对应的 gᵢ(从而推高 logitsᵢ),下次这个 token(及相似 token)更可能、更重地路由给 Eᵢ。这是一个"软的、自我强化"的学习信号。**这就是为什么 MoE 的输出必须是加权和而不是简单和——权重承载了路由器全部的梯度。** 离散的"选哪 k 个"不可微,连续的"选中的权重多大"可微,后者替前者把路由器训起来。

(题外:这正是 Shazeer 等人 2017 年那篇 "Outrageously Large Neural Networks" 里用 "noisy top-k gating" 的动机之一——加噪声还能帮助探索,让不那么靠前的专家也偶尔被选中拿到梯度。)

### 2.3 解耦的算术:本章的全部魅力

现在把 MoE 的核心收益用算术钉死。设单个专家(一个 FFN)的参数量是 P_e,单 token 过一个专家的计算量是 C_e。

**总参数量(知识容量)**:

```
P_total ≈ N · P_e  +  (路由器 Wᵣ 的 N·d, 可忽略)
        ∝  N
```

**单 token 计算量(算力)**:每个 token 只过 k 个专家,加上一次路由打分(N·d ≈ 可忽略):

```
C_token ≈ k · C_e  +  (路由 N·d)
        ∝  k        (k 是常数, 通常 1 或 2)
```

**两者相除——这是整章最该背下来的一个比值**:

```
P_total / C_token  ∝  N / k
```

当 k 固定(比如 2),**参数量随 N 线性增长,而单 token 算力几乎不变**。把专家数从 1 加到 64,总参数涨约 64 倍,单 token 算力(k=2)只是从"过 1 个 FFN"变成"过 2 个 FFN"——不到原来的 2 倍。

具体感受一下。Mixtral 8x7B(**以下数值待核**,以官方为准):8 个专家、每 token 选 top-2。总参数约 47B(注意不是 8×7=56B,因为注意力等部分是各层共享/稠密的,只有 FFN 被复制 8 份),但每个 token 实际激活的参数约 13B。**用 13B 的每 token 算力,跑出了接近 47B 容量的模型**——参数与算力在这里被撬开了一道缝。这道缝,就是 MoE 的全部价值。

> 把这件事说绝对一点:稠密模型里"参数量"和"每 token FLOPs"是同一个数的两面;MoE 把它们拆成两个可独立调节的旋钮。N 调容量,k 调算力。这就是"参数与算力解耦"。

## 3. 关键难点一:负载均衡

解耦的代价不是免费的。MoE 引入了一个稠密模型完全没有的新问题:**路由不均衡**。这是 MoE 工程的核心难点,不解决,MoE 跑不起来。

### 3.1 赢者通吃:为什么路由会自发崩塌

考虑训练初期。路由器 Wᵣ 随机初始化,各专家也随机。假设由于初始化的微小随机扰动,专家 3 碰巧比别人稍微"好用"一点点(它的输出让 loss 降得稍多)。会发生什么?

```
专家3 略好用  →  更多 token 被路由给它(g₃ 被梯度推高)
            →  专家3 拿到更多训练数据, 学得更好
            →  专家3 变得更好用
            →  更多 token 被路由给它 ...   (正反馈)
```

这是一个**正反馈失控(rich-get-richer)**。少数几个专家迅速垄断绝大多数 token,其余专家几乎收不到 token、得不到梯度、永远停在随机初始化状态——**死专家(dead experts)**。结局是:你声称有 N 个专家(N 倍参数),实际只有少数几个在干活,巨量参数纯属浪费,容量优势荡然无存。

这个问题在数学上是必然的:softmax 路由 + 梯度上升对"当前最好"的专家,本身就是一个偏向集中的动力系统。**MoE 不会自己学会均匀分工,必须外加一个力把它掰平**。这个力就是负载均衡损失。

### 3.2 负载均衡损失:构造与直觉

我们想要的是:在一个 batch(或一个 micro-batch)里,token 大致均匀地分布到 N 个专家上。怎么把这个愿望写成一个可微的、加到主 loss 上的辅助项?

直接的想法是"惩罚分布不均",比如惩罚每个专家收到 token 比例的方差。但这里有个微妙的不可微问题:"专家 i 收到了多少 token"是一个计数(离散),不可微。Switch Transformer 给出的解法很巧,**把不可微的"分配比例"和可微的"路由概率"乘起来**。

设一个 batch 有 B 个 token(下标 b),N 个专家(下标 i)。定义两个量:

```
fᵢ = 实际分配给专家 i 的 token 比例（fraction dispatched）
   = (1/B) · #{ b : 专家 i 在 token b 的 top-k 中 }          —— 不可微（计数）

Pᵢ = 路由器给专家 i 的平均路由概率（mean routing prob）
   = (1/B) · Σ_b  softmax(Wᵣ x_b)ᵢ                           —— 可微（softmax 的均值）
```

负载均衡损失(Switch Transformer, Fedus et al. 2021,这里取 top-1 形式)定义为:

```
L_balance = α · N · Σᵢ  fᵢ · Pᵢ
```

α 是权重系数(Switch 论文用 α=0.01,**待核**)。乘 N 是为了让这个量的尺度不随专家数变化(见下面的最优值分析)。

**为什么这个形式能促进均衡?完整论证如下。**

第一,看它的最小值在哪。fᵢ 和 Pᵢ 都满足 Σᵢ fᵢ = 1、Σᵢ Pᵢ = 1(都是比例/概率,和为 1)。我们要最小化 `Σᵢ fᵢ Pᵢ`。把它看成两个概率分布 f 和 P 的内积。由约束下的优化:

```
在 Σfᵢ=1, ΣPᵢ=1, fᵢ,Pᵢ≥0 约束下, Σᵢ fᵢ Pᵢ 的最小值
当且仅当 f 和 P 都是均匀分布 (fᵢ = Pᵢ = 1/N) 时取得平衡解,
此时 Σᵢ fᵢPᵢ = N · (1/N)² = 1/N, 乘上前面的 N 得到 L = α · 1。
```

(严格地说 `Σ fᵢPᵢ` 在单纯形上的全局最小是把 f、P 集中到不同的单个分量上得到 0;但在 MoE 训练里 f 由 P 诱导——路由概率高的专家也更容易被分配——两者强相关、同涨同落,在这种正相关约束下,压低 `Σ fᵢPᵢ` 的现实路径是把质量摊平,而非反向集中。均匀分布给出 1/N,远小于"一个专家通吃"的 ≈1。这就是为什么它实践中有效。**这一段是该损失的设计直觉,非闭式证明;严格的博弈式分析见原论文,待核细节**。)

第二,看梯度往哪推。L_balance 对路由器参数的梯度只通过可微的 Pᵢ 这一项(fᵢ 被当作常数,stop-gradient):

```
∂L_balance/∂(Wᵣ)  ∝  α·N · Σᵢ fᵢ · ∂Pᵢ/∂(Wᵣ)
```

直觉:fᵢ 充当**权重**。对一个已经超载(fᵢ 大)的专家,损失会用力压低它的路由概率 Pᵢ;对一个闲置(fᵢ 小)的专家,这个压力很小,它的 Pᵢ 相对被放过、有机会升高。**净效果是把概率质量从拥挤的专家挪向空闲的专家**——正是我们要的均衡力。

**这个损失的精妙之处**:它没有去对不可微的计数 fᵢ 求导(那会失败),而是把 fᵢ 当作一个"测量到的拥挤程度"权重,只让梯度流过可微的 Pᵢ。fᵢ 提供"往哪推"的方向,Pᵢ 提供"可微的推力"。一个不可微的目标,被拆成"不可微的方向 × 可微的力"。这是处理"含离散决策的可微优化"的一个通用范式,值得记住。

### 3.3 另一种正则:router z-loss

除了均衡,还有个数值稳定问题:路由 logits 可能越来越大,softmax 进饱和区(回顾上一章 §3.4,大 logits 让梯度消失),且 exp 容易上溢。Switch / ST-MoE 引入 **router z-loss** 来约束 logits 的量级:

```
L_z = β · (1/B) · Σ_b  ( logsumexp_i( logits_{b,i} ) )²
```

`logsumexp = log Σᵢ exp(logitsᵢ)`,它近似等于 max logit。平方惩罚让 logits 整体不要太大,等价于"鼓励路由 logits 保持小而集中、softmax 不进饱和"。β 很小(ST-MoE 用 1e-3,**待核**)。这一项主要是为 bf16/fp16 训练的数值稳定服务的,实战里和 L_balance 一起加。

## 4. 关键难点二:容量、丢弃与专门化

负载均衡是"软约束"——它鼓励均匀但不保证。实际系统里还需要一个"硬约束"来让计算可批处理(batch),这就引出 capacity。

### 4.1 为什么需要 capacity factor:静态形状的硬约束

理想里每个专家收到的 token 数可以是任意的。但在 GPU/TPU 上做高效批处理,**每个专家的输入张量必须是固定形状**(编译期就要定 shape,动态 shape 极慢甚至不支持)。于是给每个专家设一个固定的**容量(capacity)**——最多接收多少 token:

```
expert_capacity = capacity_factor · (k · B / N)
```

理解这个式子:`k·B` 是一个 batch 里总的"token-专家分配"次数(每个 token 选 k 个专家),`/N` 是平均每个专家应得的份额,`capacity_factor`(容量因子,通常 1.0 ~ 2.0)是给波动留的余量。

- capacity_factor = 1.0:刚好够"完美均衡"时每个专家的量,任何不均衡都会导致溢出。
- capacity_factor = 1.25:给每个专家多 25% 的缓冲,容忍一定不均衡。

### 4.2 token 丢弃(token dropping)

当路由到某个专家的 token 数**超过它的 capacity**,多出来的 token 怎么办?——**被丢弃(dropped)**。被丢弃的 token 不过任何专家,通常靠**残差连接(residual)直接跳过这一层**:

```
正常 token:    y = x + MoE(x)        (MoE 输出 = 加权专家输出)
被丢弃 token:  y = x + 0 = x          (MoE 这一层对它等于没做, 只剩残差)
```

这是一个真实的精度代价:被丢的 token 在这一层失去了专家加工。capacity_factor 的取舍就在这里:

```
capacity_factor 小  →  显存/计算省, 但丢弃多, 精度伤
capacity_factor 大  →  丢弃少, 但每个专家张量更大, 浪费(空槽 padding)、显存涨
```

**这是 MoE 特有的、稠密模型完全没有的 trade-off**:你在用"是否丢 token"来换"批处理的固定形状"。实战里 capacity_factor 常取 1.0~1.25 训练、推理时可调大减少丢弃。注意丢弃率本身是个该监控的指标——丢太多说明负载均衡没做好。

(补充:也有 **无丢弃** 的变体,比如 token-choice 改成 **expert-choice routing**——反过来让每个专家挑它最想要的固定数量 token,天然不溢出、自动均衡,代价是一个 token 可能被 0 个或多个专家选中,语义略不同。这是后续 MoE 工作的一个重要分支,**待核**具体论文细节。)

### 4.3 专家专门化:自发涌现的分工

一个有趣且重要的经验现象:训练好的 MoE,**不同专家会自发分化出不同的"专长"**,虽然我们从没显式告诉它们该专攻什么。这叫**专家专门化(expert specialization)**。

观察到的形态(综合多篇 MoE 工作的报告,**具体归因待核**):

- 在多语言模型里,某些专家偏好特定语言/文字系统的 token。
- 在通用文本里,有专家偏好标点、数字、特定句法角色(如冠词、专有名词)的 token。
- 专门化更多体现在**浅层/token 级特征**上,而非清晰的"数学专家 / 代码专家"这种高层语义分工——后者的证据较弱,别过度浪漫化。

为什么会涌现分工?因为路由器 + 负载均衡共同塑造了一个**软聚类**:负载均衡逼迫 token 摊开到 N 个专家,而梯度又让每个专家去最优拟合"分给它的那部分 token 分布"。两个力一拉,等价于在做一个端到端可微的、带均匀性约束的聚类——专家自然各自占据 token 空间的不同区域。**这不是设计出来的,是优化动力学的副产物。** 理解这点能帮你不对专门化抱不切实际的期待:它是统计分工,不是人类直觉的学科分工。

## 5. 代码:top-k 路由 + 加权专家前向 + 负载均衡损失

下面是能体现全部机制的 PyTorch 实现。为清晰起见用"逐专家 gather"的写法(易读),工业实现会用更高效的 batched/grouped matmul,但数学内核就是这些。

```python
import torch
import torch.nn as nn
import torch.nn.functional as F

class Expert(nn.Module):
    """单个专家 = 一个普通 FFN (SwiGLU 风格可替换, 这里用最简两层)"""
    def __init__(self, d_model, d_ff):
        super().__init__()
        self.w1 = nn.Linear(d_model, d_ff, bias=False)
        self.w2 = nn.Linear(d_ff, d_model, bias=False)
    def forward(self, x):
        return self.w2(F.gelu(self.w1(x)))


class MoELayer(nn.Module):
    def __init__(self, d_model, d_ff, n_experts, top_k=2, capacity_factor=1.25):
        super().__init__()
        self.n_experts = n_experts
        self.top_k = top_k
        self.capacity_factor = capacity_factor
        self.router = nn.Linear(d_model, n_experts, bias=False)   # Wr: 路由打分
        self.experts = nn.ModuleList([Expert(d_model, d_ff) for _ in range(n_experts)])

    def forward(self, x):
        # x: (B, d_model)   这里把 (batch, seq) 已展平成 B 个 token
        B, d = x.shape
        N, k = self.n_experts, self.top_k

        # --- 1. 路由打分 ---
        logits = self.router(x)                      # (B, N)
        probs  = F.softmax(logits, dim=-1)           # (B, N), 每行和为1

        # --- 2. 选 top-k 并在 k 个上重新归一化 (GShard/Mixtral 方案B) ---
        topk_probs, topk_idx = probs.topk(k, dim=-1)         # (B, k), (B, k)
        topk_probs = topk_probs / topk_probs.sum(-1, keepdim=True)  # 归一: Σ=1

        # --- 3. 计算 capacity (每个专家最多收多少 token) ---
        capacity = int(self.capacity_factor * k * B / N)

        # --- 4. 逐专家处理: gather 属于该专家的 token, 过 FFN, 加权 scatter 回去 ---
        y = torch.zeros_like(x)                              # 输出累加器, 被丢弃 token 保持 0
        for i in range(N):
            # 找出在 top-k 里包含专家 i 的 (token, 第几个槽) 位置
            # mask_i: (B, k) 布尔, True 表示该 token 的第 j 个选择是专家 i
            mask_i = (topk_idx == i)                         # (B, k)
            token_ids, slot_ids = mask_i.nonzero(as_tuple=True)   # 1D 索引
            if token_ids.numel() == 0:
                continue
            # capacity 截断: 超出容量的 token 被丢弃 (这里简单取前 capacity 个)
            if token_ids.numel() > capacity:
                token_ids = token_ids[:capacity]
                slot_ids  = slot_ids[:capacity]
            xi = x[token_ids]                                # (n_i, d) 分给专家i的 token
            gi = topk_probs[token_ids, slot_ids].unsqueeze(-1)   # (n_i, 1) 对应路由权重
            ei = self.experts[i](xi)                         # (n_i, d) 专家输出
            # 加权写回: y += g_i * E_i(x)   —— 权重 g_i 承载路由器梯度
            y.index_add_(0, token_ids, gi * ei)

        # --- 5. 负载均衡辅助损失 (Switch 形式) ---
        # f_i: 实际分配比例 (用 one-hot 计数, detach 视为不可微常数)
        with torch.no_grad():
            one_hot = F.one_hot(topk_idx, N).sum(1).float()  # (B, N), 每行被选的专家=1
            f = one_hot.mean(0)                              # (N,) 每专家平均被选次数; 归一前 Σf = k (每 token 选 k 个), 下一行归一成分布
            f = f / f.sum()                                  # 归一成分布
        # P_i: 平均路由概率 (可微)
        P = probs.mean(0)                                    # (N,)
        aux_loss = N * torch.sum(f * P)                      # 乘 N 使尺度稳定

        return y, aux_loss


# --- 自检: 验证解耦 (激活参数 << 总参数) ---
torch.manual_seed(0)
d_model, d_ff, N, k = 64, 256, 8, 2
moe = MoELayer(d_model, d_ff, N, top_k=k)
x = torch.randn(32, d_model)
y, aux = moe(x)

total_params  = sum(p.numel() for p in moe.parameters())
expert_params = sum(p.numel() for p in moe.experts[0].parameters())
print(f"总参数 ≈ {total_params}, 单专家 ≈ {expert_params}, N×单专家 ≈ {N*expert_params}")
print(f"单 token 激活 ≈ {k} 个专家 = {k*expert_params} 参数 (路由器 {moe.router.weight.numel()} 忽略)")
print(f"容量解耦比 P/C ≈ N/k = {N/k}")
print(f"aux_loss = {aux.item():.4f}  (越接近 1 越均衡, 越大越失衡)")
assert y.shape == x.shape
```

实现要点,每个对应前面的推导:

- **权重 `gᵢ * eᵢ` 必须乘上去**(§2.2):这是路由器唯一的梯度通路。漏了它路由器学不动。
- **`index_add_` 累加**:一个 token 可能选了 k 个专家,它们的加权输出要**累加**到同一行,不是覆盖。
- **被丢弃 token 保持 0**(§4.2):`y` 初始化为零,超 capacity 的 token 不写入;外层有残差 `x + y` 时它就只剩 x。
- **`f` 用 `no_grad`/detach**(§3.2):分配比例 fᵢ 是不可微计数,必须 stop-gradient,只让梯度走可微的 Pᵢ。漏掉 detach 会引入错误梯度。
- **softmax 在 N 个专家维做**:`dim=-1`,和上一章注意力沿 key 维同理,搞反就全错。
- **这个逐专家 for 循环在 GPU 上不是最优**:真实实现用 sorting + grouped GEMM(把同专家 token 排到一起做一个大 matmul)或 dispatch/combine 的 einsum,避免 Python 循环。但**正确性语义就是上面这段**。

## 6. 量化分析:参数、FLOPs、显存

### 6.1 参数量(完整推导)

把一层 MoE 和一层稠密 FFN 对比。设 d = d_model,d_ff = 4d(惯例),专家用两层 FFN。

```
单专家参数:  P_e = (d·d_ff) + (d_ff·d) = 2·d·d_ff = 2·d·4d = 8 d²
路由器参数:  N·d                          (通常 << 专家, 可忽略)

MoE 层 FFN 部分总参数:  P_moe = N · P_e + N·d ≈ N · 8d²
稠密 FFN 层参数:        P_dense = 8d²

参数放大倍数:  P_moe / P_dense ≈ N        ← 线性涨于专家数
```

N=64 时,FFN 部分参数涨 64 倍。但注意**整个模型不是涨 64 倍**:注意力、embedding、layernorm 等是稠密共享的,不复制。所以全模型参数放大倍数取决于 FFN 在总参数里的占比(前面算过 FFN 约占 transformer block 参数的 2/3)。

### 6.2 单 token FLOPs(完整推导)

```
路由打分:     logits = Wᵣ x,  N·d 次乘加 ≈ 2Nd  FLOPs
top-k 选择:    O(N) 比较 (排序/堆), 相对乘加可忽略
过 k 个专家:   每个专家 ≈ 4·d·d_ff = 16d² FLOPs (§1.1), 共 k 个
              = k · 16d²  FLOPs

MoE 单 token FLOPs ≈ k·16d² + 2Nd
稠密 FFN 单 token FLOPs ≈ 16d²

计算放大倍数 ≈ (k·16d² + 2Nd) / 16d² = k + N/(8d)
```

关键看那个 `N/(8d)` 修正项——它是路由打分的相对开销。**只要 N ≪ 8d(几乎总成立,d 上千而 N 几十),路由开销可忽略,计算放大倍数 ≈ k**。

把 §6.1 和 §6.2 并排,**解耦被钉死**:

```
参数放大:  ≈ N        (容量旋钮)
算力放大:  ≈ k        (算力旋钮, 与 N 解耦, 因为路由开销 N/(8d) → 0)
```

N=64、k=2、d=4096 时:参数涨 64 倍,算力涨 ≈ 2 + 64/(8·4096) ≈ 2.002 倍。**64 倍的容量,2 倍的算力。** 这就是开头那句"参数与算力解耦"的精确量化。

### 6.3 显存:MoE 真正的代价在哪

这里必须诚实——**MoE 不是免费午餐,它把算力的省,换成了显存(和通信)的涨**。

```
参数显存:  N · P_e —— 全部 N 个专家的权重都要存在显存里!
          虽然单 token 只激活 k 个, 但你不知道下一个 token 路由给谁,
          N 个专家的参数必须全部驻留。  → 参数显存 ∝ N (和稠密放大同步涨)
```

这是 MoE 最大的实际约束:**省的是计算(FLOPs/算力),省的不是参数显存**。一个 47B 参数的 MoE,推理时虽然每 token 只算 13B,但 47B 的权重得全部装进显存(或跨多卡分片)。所以 MoE 的真实定位是:**在"显存够装下大参数、但算力是瓶颈"的场景下,用显存换算力**。

其他显存/系统开销:

```
激活显存:    每个专家的中间激活 (n_i, d_ff), 总量与稠密同阶 (token 总数不变)
通信(分布式): 专家通常分布在不同设备 → token 要 all-to-all 发到对应专家所在卡,
            再 all-to-all 收回。这是 expert parallelism 的核心开销,系统侧难点。
            (专家并行 / all-to-all 的工程细节在 MLSys 课展开, 这里只点名)
负载不均的浪费: capacity padding 的空槽、丢弃 token 的精度损失 (§4.2)
```

一张表收尾:

| 量 | 稠密 FFN | MoE (N 专家, top-k) | 说明 |
|---|---|---|---|
| 总参数 | 8d² | ≈ N·8d² | 容量随 N 线性涨 |
| 单 token FLOPs | 16d² | ≈ k·16d² | 算力随 k(常数)涨,与 N 解耦 |
| 参数显存 | ∝ 1 | ∝ N | **不省**,全专家驻留 |
| 激活显存 | ∝ token 数 | ≈ 同阶 | token 总数不变 |
| 额外系统开销 | 无 | all-to-all 通信、capacity padding、token 丢弃 | MoE 特有 |
| 解耦比 P/C | 固定 | **∝ N/k** | MoE 的灵魂 |

## 7. Switch Transformer 与 GShard:两个标杆设计

### 7.1 GShard(Lepikhin et al., 2020):top-2 + 容量 + 均衡损失

GShard 是把 MoE 推到超大规模(论文做到 6000 亿参数级,**待核**)并把训练跑稳的奠基工作。它确立了至今仍是主流的几个组件:

- **top-2 路由**:每个 token 选 2 个专家。第 2 个专家提供"冗余/对比",经验上比 top-1 更稳、质量更好。
- **expert capacity + 丢弃**:就是 §4 讲的固定容量机制,GShard 系统化地引入。
- **辅助负载均衡损失**:§3.2 那个 `Σ fᵢPᵢ` 形式的雏形。
- **专家分片到设备 + all-to-all**:首次把"专家并行"作为分布式 MoE 的标准范式。

### 7.2 Switch Transformer(Fedus et al., 2021):激进的 top-1

Switch 的核心论点是**反直觉的**:既然 top-2 的第二个专家增加了路由、通信、计算开销,**能不能只用 top-1?** 他们论证 top-1 在仔细设计下不仅可行,而且在固定算力预算下往往更优:

- **top-1(每 token 只选 1 个专家)**:路由最简单,通信量减半(只发到 1 个专家),计算放大倍数从 ≈2 降到 ≈1。
- **简化的负载均衡损失**:就是 §3.2 给出的 `α·N·Σ fᵢPᵢ`。
- **capacity factor + 选择性精度**:用 bf16 训练 + 关键处 fp32,配合 router z-loss(§3.3)稳住数值。
- **结论**:同样的算力(FLOPs)预算下,Switch 可以用 top-1 把专家数堆得更多,反而比 top-2 拿到更好的"质量/算力比"。

### 7.3 top-1 vs top-2:取舍的完整账

这是练习题之一,也是 MoE 设计的核心决策,这里把账算清。设其他都相同,只比 k=1 与 k=2:

```
                          top-1 (k=1)          top-2 (k=2)
单 token 计算放大          ≈ 1                  ≈ 2              (k 倍)
通信量 (all-to-all)        1 份 token            2 份 token        (k 倍)
路由表达力                 token 全押 1 个专家    可混合 2 个专家   top-2 更柔
梯度信号 / 路由学习         较稀疏, 更易失衡       第二专家提供对比, 更稳
专家利用 / 均衡难度         更难均衡(押注更硬)    相对易均衡
质量(同算力预算)          可堆更多专家补回来    单点质量更高
```

核心权衡用一句话概括:

- **top-2 用 2 倍算力和通信,换更柔的路由、更稳的训练、更好的单点质量**。
- **top-1 用极致的算力/通信省,换"省下来的预算可以堆更多专家/更大模型"的自由度**;代价是更难均衡、路由更"硬"。

没有绝对赢家,取决于你的瓶颈是算力还是显存/通信。**Mixtral 选 top-2,Switch 选 top-1**——两条路都通,反映了不同的工程权衡。现代很多模型(如 DeepSeek-MoE 用更细粒度的多专家 + 共享专家 + 更高 k,**待核**)在这个谱系上做更精细的设计。

## 8. 设计权衡与常见坑

- **以为 MoE 省显存**。最致命的误解。MoE 省的是**算力(FLOPs)**,参数显存照样 ∝ N 地涨(§6.3)。全部专家权重必须驻留。把 MoE 当"免费变大"会在显存上撞墙。
- **漏掉路由权重 gᵢ**(§2.2)。`y = Σ Eᵢ(x)` 不乘权重,路由器没梯度,退化成随机路由,N 倍参数白费。这是从零实现最常见的致命 bug。
- **不加负载均衡损失**(§3)。MoE 不会自发均衡,会赢者通吃出死专家。aux loss 是必需品不是可选项。监控每个专家的 token 比例,发现极度倾斜就是它没生效。
- **aux loss 的 fᵢ 忘了 detach**(§3.2)。fᵢ 是不可微计数,必须 stop-gradient;否则给路由器灌错误梯度。
- **α 调得太大或太小**。α 太大,均衡压力盖过主任务,模型为了均匀牺牲质量(路由变得几乎随机);太小,压不住失衡。Switch 用 0.01 量级是个经验起点(**待核**),需按你的 N 和数据调。
- **capacity_factor 与丢弃率失配**(§4.2)。cf 太小丢太多 token 伤精度;太大浪费显存和计算(空槽 padding)。**训练时务必把 token 丢弃率打到日志里监控**——它是 MoE 健康度的核心体检指标。
- **训练/推理 capacity 不一致**。推理时分布可能和训练不同,常把 cf 调大些减少丢弃。忘了调会无谓丢 token 掉精度。
- **softmax 在错误维度**。路由 softmax 沿专家维(N 维),搞反全错,和注意力同坑。
- **专门化的过度浪漫化**(§4.3)。别指望专家自动分成"数学家/程序员";实际多是 token 级浅层统计分工。基于此做强假设(如人工指定专家职责)通常不成立。
- **MoE 微调更难**。专家多、每个专家见的数据少,小数据集微调 MoE 容易过拟合或专家退化,常需要特殊正则或冻结部分专家(**待核** 具体最佳实践)。
- **批大小与 token 数的敏感性**。负载均衡和 capacity 都按 batch 内 token 统计;batch 太小,统计噪声大,均衡难做、丢弃波动大。MoE 偏好较大的有效 batch。

## 9. 动手练习

1. **(推导题)参数量与单 token FLOPs。** 设 d_model=d、d_ff=4d、专家为两层 FFN、N 个专家、top-k。从头推出:(a) MoE 层 FFN 部分总参数 ≈ N·8d²,稠密 FFN 为 8d²,放大 ≈ N;(b) 单 token FLOPs ≈ k·16d² + 2Nd,放大倍数 ≈ k + N/(8d)。**代入 N=64、k=2、d=4096,算出参数放大与算力放大各是多少,并指出为什么算力几乎只取决于 k 而与 N 无关。** *提示*:单专家参数 = 2·d·d_ff;路由开销项 2Nd 要和主项 k·16d² 比大小,看 N/(8d) 是否 ≪ 1。

2. **(分析/论证题)负载均衡损失为何有效。** 写出 `L = α·N·Σᵢ fᵢ·Pᵢ`,(a) 说明 fᵢ(分配比例)和 Pᵢ(平均路由概率)各自可微性,为什么 fᵢ 要 stop-gradient;(b) 论证当 f、P 正相关(拥挤专家路由概率也高)时,压低 `Σ fᵢPᵢ` 的现实路径是把分布摊平到 fᵢ=Pᵢ=1/N,此时 L≈α;(c) 从梯度角度说明为什么 fᵢ 大(超载)的专家其 Pᵢ 被更用力压低。*提示*:把 `Σ fᵢPᵢ` 看成两个单纯形上分布的内积;梯度 ∝ Σ fᵢ·∂Pᵢ/∂Wᵣ,fᵢ 是权重。

3. **(分析题)top-1 vs top-2 取舍。** 列一张表对比 k=1 与 k=2 在:单 token 算力、all-to-all 通信量、路由表达力、训练稳定性、均衡难度、同算力预算下能堆的专家数。**论证 Switch 的核心主张**:"在固定 FLOPs 预算下,top-1 因为省下的算力可以堆更多专家,可能拿到比 top-2 更好的质量"。这个主张在什么前提下成立、什么前提下不成立?*提示*:固定 FLOPs ⇒ k 减半允许 N 翻倍;但通信/显存/均衡难度怎么变?瓶颈在算力 vs 在显存,结论不同。

4. **(编码题)实现并验证解耦 + 均衡。** 基于 §5 代码,(a) 把 N 从 4 改到 64,验证总参数 ≈ N×单专家而单 token 激活参数不变;(b) **故意去掉 aux_loss 训练几百步**(可用一个玩具分类任务),记录每个专家收到的 token 比例,观察是否出现赢者通吃 / 死专家;(c) 加回 aux_loss,看分布是否被掰平。*提示*:统计 `topk_idx` 的直方图;无 aux loss 时大概率看到少数专家占绝大多数 token,加上后趋于均匀。把丢弃率也打出来。

## 10. 源码 / 论文导读

- **Shazeer et al. (2017), "Outrageously Large Neural Networks: The Sparsely-Gated Mixture-of-Experts Layer"**:MoE 用于现代深度网络的开山作。读它的 noisy top-k gating(§2.1 的可微路由动机)和最早的负载均衡损失(importance loss + load loss)。理解"为什么要加噪声"和"为什么必须乘门控权重"。
- **Lepikhin et al. (2020), "GShard"**:读 top-2 路由、expert capacity、auxiliary loss 三件套的定义,以及它怎么把专家分片到设备做 all-to-all。这是分布式 MoE 的范式确立者。
- **Fedus, Zoph, Shazeer (2021), "Switch Transformers"**:精读负载均衡损失那一节(§3.2 的 `α·N·Σ fᵢPᵢ` 出处)和 capacity factor / token dropping 的讨论;以及它对 top-1 优于 top-2(在固定算力下)的论证——这是练习 3 的原始素材。还有 router z-loss(在后续 ST-MoE 论文中更完整)。
- **Mixtral of Experts (Jiang et al., 2024)**:一个开放权重、被广泛复现的 top-2 MoE(8 专家)。看它怎么算"激活参数 vs 总参数"、路由权重的归一化方式(§2.1 方案B),对照本章 §6 的解耦量化。
- **代码:HuggingFace transformers 的 `MixtralSparseMoeBlock`**(或任意 `*MoeBlock`):看真实实现里 router、top-k、归一化、以及 expert 的 grouped 计算怎么写——对照 §5 的逐专家循环,理解工业实现如何用 index/scatter 避免 Python 循环。
- **代码:`megablocks`(Gale et al.)或 DeepSpeed-MoE**:看高效 MoE kernel 怎么把"变长的每专家 token"用 block-sparse / grouped GEMM 处理,避免 capacity padding 的浪费——这是 §5"实现要点"最后一条的工程答案,也是通往 MLSys 课专家并行的桥。
- **DeepSeek-MoE / DeepSeek-V3 技术报告**(**待核** 具体版本与数值):看"细粒度专家 + 共享专家 + 更高 k + 无辅助损失的均衡(aux-loss-free)"等对本章基础范式的演进,理解 MoE 设计的前沿在往哪走。

## 11. 小结与承上启下

这章把 MoE 从**"参数即算力"的硬墙动机**,一路推到 **`y = Σ_{i∈topk} gᵢ·Eᵢ(x)` 的每个符号、解耦比 N/k 的精确量化、路由权重 gᵢ 为何是路由器唯一梯度通路、负载均衡损失 `α·N·Σ fᵢPᵢ` 的构造与有效性、capacity 与 token 丢弃的取舍、以及 Switch(top-1)与 GShard(top-2)的设计权衡**。你现在应该能:在白板上推出 MoE 的参数量与单 token FLOPs、解释为什么算力 ≈ k 而与 N 解耦、讲清负载均衡损失为什么把分布掰平、并诚实地说出 MoE 省的是算力不是显存。

**两条贯穿本章的主线**:其一,**稀疏激活把"参数(容量)"和"算力(每 token 计算)"这两个在稠密模型里同一个数的两面,拆成了两个独立旋钮**——N 调容量,k 调算力,这是 MoE 存在的全部理由。其二,**这个解耦不是免费的**:它把算力的省,换成了参数显存的涨、all-to-all 通信的复杂、负载均衡的工程难题——MoE 的每个收益都有对应的代价,理解代价比理解收益更重要。

**承上启下**:本章只讲了 MoE 的**算法/架构**侧——路由怎么算、损失怎么构造、参数算力怎么解耦。但 MoE 真正的工程难点在**系统侧**:N 个专家分布在几百张卡上,token 要通过 **all-to-all** 通信被发到对应专家所在的设备、算完再收回——这个通信往往是分布式 MoE 训练的真正瓶颈,涉及**专家并行(expert parallelism)**与数据/张量/流水并行的组合。这部分留给 MLSys 课的分布式训练章节。回到算法主线:你已经掌握了 Transformer 里"把 FFN 稀疏化"这一招;它和前几章的注意力、归一化、位置编码一起,构成了你读懂任何一个现代大模型架构图所需的全部模块。下一章我们将进入 [按课程大纲衔接,**待核** 后续章节标题],继续在"如何让模型更大、更快、更强"这条主线上往前推。
