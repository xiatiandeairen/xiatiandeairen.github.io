> 这章把你从"知道 GPU 很快但不知道它为什么慢",带到能用一张 Roofline 图判定任意算子是 compute-bound 还是 memory-bound,并能从实测耗时反推有效带宽/算力、对照理论峰值找出性能损失在哪。后面所有 MLSys 优化(FlashAttention、算子融合、并行、量化)都是在拆这一章定义的物理约束。

## 一、为什么从硬件讲起:优化的下界由物理决定

做系统的人有一个根深蒂固的直觉:**任何性能问题,最终都归结到"计算"和"数据搬运"两件事的竞赛**。CPU 时代你和 cache miss、内存墙、SIMD 打了多年交道;GPU 只是把同一组矛盾换了一套参数重新摆出来——而且摆得更极端。

为什么必须先讲硬件,而不是直接学并行策略?因为**优化的下界是物理量,不是代码技巧**。一个算子要读 X 字节、写 Y 字节、算 Z 次浮点运算,这三个数一旦确定,它在某张卡上能跑多快就有了一个理论上限。你写的所有 CUDA kernel、所有融合、所有调度,本质都是在逼近这个上限。**不知道上限在哪,优化就是盲目试错;知道上限在哪,你能算出"还差多少、值不值得继续优化"。** 这正是 Roofline 模型给系统工程师的礼物——它和你过去用的性能分析方法论是同一个东西,只是坐标轴换了名字。

本章的目标函数很明确:建立一套**能定量预测和归因**的硬件心智模型。读完你应该能回答这类问题——"这个逐元素加法为什么只跑到峰值算力的 1%?"、"把 batch 调大为什么有时根本不影响吞吐?"、"为什么 FlashAttention 不减少 FLOPs 却能快好几倍?"

下面所有具体数字以 **NVIDIA A100(80GB SXM)** 为基准锚点(它是这门课和大量论文的公共参照系),个别处给出 H100 量级做对比。芯片代际数字更新很快,**所有具体带宽/算力数值都应以厂商官方 datasheet 为准,本文标注的数字用于量级推理而非精确引用**「待核」。

## 二、SIMT 执行模型:GPU 的并行不是你以为的那种并行

### 问题:几千个核心怎么被组织起来

A100 有 108 个 **SM(Streaming Multiprocessor,流式多处理器)**,每个 SM 内有大量 CUDA core。但 GPU 的并行单位**既不是单个 core,也不是单个线程**,而是 **warp(线程束)= 32 个线程**。理解这一点是理解后面一切性能现象的钥匙。

NVIDIA 把这套模型叫 **SIMT(Single Instruction, Multiple Threads,单指令多线程)**。它介于 CPU 的 SIMD 和多线程之间:

- **SIMD(单指令多数据,CPU 的 AVX/SSE):** 一条指令操作一个宽向量寄存器(如 8 个 float),程序员要显式写向量化代码,所有 lane 必须做同一件事。
- **SIMT(GPU):** 你写的是**标量**线程代码(看起来每个线程独立),但硬件把 32 个线程**捆成一个 warp,共享同一个取指/译码单元、同一个程序计数器,锁步(lock-step)执行同一条指令**,只是各自作用在不同数据上。

**关键结论:warp 内 32 个线程在硬件上不是 32 个独立执行流,而是一条指令的 32 路数据并行。** 这带来 GPU 性能的第一个铁律——**warp divergence(线程束分歧)**。

### 机制:divergence 为什么让你的 if-else 变慢一倍

假设 warp 里 16 个线程满足 `if (x > 0)`,另 16 个走 `else`。SIMT 没法让一条指令同时执行两个分支,硬件的处理方式是:

```
执行 if 分支时:  16 个满足条件的线程激活,另 16 个被 mask 屏蔽(空转)
执行 else 分支时:刚才被屏蔽的 16 个激活,前 16 个被屏蔽(空转)
→ 两个分支串行执行,这个 warp 的有效吞吐砍半
```

直觉:**一个 warp 走过的"代码路径总长度" = 它内部所有线程走过的不同路径之和**。分支越发散,浪费越多。这就是为什么 GPU 上写 kernel 的金科玉律是"让同一个 warp 内的线程尽量走相同分支"。

在深度学习里这个坑通常不直接咬你(框架的 elementwise kernel 早就规避了),但当你写自定义 CUDA(比如条件采样、稀疏路由 MoE 的 token dispatch)时,divergence 会突然成为头号瓶颈。

### 延迟隐藏:GPU 慢得理直气壮,靠"超额订阅"扛过去

CPU 用大 cache + 乱序执行 + 分支预测,**目标是降低单线程延迟**。GPU 反其道而行:它的单个内存访问延迟其实**非常高**(从 HBM 取数据要几百个时钟周期),它根本不打算降低延迟,而是**用海量并行掩盖延迟(latency hiding)**。

机制是这样的:每个 SM 上可以**同时驻留**多个 warp(A100 每 SM 最多 64 个 warp 待命「待核」)。当 warp A 发出一次访存、要等几百周期时,warp 调度器**立刻零开销切换**到已经就绪的 warp B 继续算——因为每个 warp 的寄存器状态都常驻在 SM 的寄存器堆里,切换不需要保存/恢复上下文(这和 CPU 线程切换有天壤之别)。

```
CPU 思路:  让 1 个线程跑得尽量快(降延迟)→ 大 cache、复杂控制逻辑
GPU 思路:  让 1000 个线程轮流跑(藏延迟)→ 小 cache、海量寄存器、零开销切 warp

核心等式(Little's Law 的 GPU 版,直觉):
   要打满硬件 = 在途的并行操作数 ≥ 延迟 × 吞吐
   → 内存延迟越高、带宽越大,需要"在飞"的访存请求就越多,才能填满流水线
```

**这直接推出"占用率(occupancy)"这个概念**:SM 上实际驻留的 warp 数 ÷ 硬件能驻留的最大 warp 数。占用率太低,意味着没有足够多的 warp 来填补访存延迟的空窗,SM 会空转。后面第六节细讲。

### Tensor Core:为什么 GPU 算力数字一夜涨了一个量级

普通 CUDA core 一个周期做一次 FP32 乘加(FMA)。**Tensor Core** 是 Volta 架构(2017)引入的专用硬件单元,它一条指令就完成一个**小矩阵乘加** `D = A·B + C`(典型如 16×16 或更大的分块,数据类型 FP16/BF16/TF32/INT8/FP8 等低精度,累加到 FP32)。

为什么它能把算力拉高一个量级?因为**矩阵乘法的计算密度极高**:一个 m×k 乘 k×n 的矩阵乘做了 2mkn 次浮点运算,但只读了 mk+kn 个、写了 mn 个元素。Tensor Core 把这种"重计算、相对轻访存"的模式硬连线成单条指令,大幅减少指令发射开销和数据搬运。

**关键数字对比(A100,量级,具体以 datasheet 为准「待核」):**

| 计算类型 | 峰值算力(约) |
|---|---|
| FP32(CUDA core,非 Tensor Core) | ~19.5 TFLOPS |
| TF32(Tensor Core) | ~156 TFLOPS |
| FP16/BF16(Tensor Core) | ~312 TFLOPS |
| FP16(Tensor Core + 稀疏) | ~624 TFLOPS |

**这张表是后面所有 Roofline 推理的算力轴来源,务必记住量级**:**A100 用 Tensor Core 跑 FP16,峰值约 312 TFLOPS,比 FP32 的 19.5 TFLOPS 高约 16 倍。** 这也解释了为什么混合精度训练不是可选项而是必需——不用 Tensor Core,你的卡只发挥了零头算力。

> 系统老兵的迁移:Tensor Core ≈ CPU 的 SIMD/向量单元的极端版,只不过它锁定在"矩阵乘"这一种运算。能不能用上它,取决于你的计算能不能表达成 GEMM(通用矩阵乘)。这就是为什么深度学习框架拼命把各种操作"GEMM 化"。

## 三、内存层级:容量与带宽差几个数量级,以及为什么

GPU 的内存是一个金字塔,**越靠近计算单元越快越小,越远越大越慢**——和 CPU 的寄存器→L1→L2→L3→DRAM 一模一样,只是参数更夸张。下面这张表是本章的物理基石(A100 量级,数值用于推理,精确值以 datasheet 为准「待核」):

| 层级 | 容量量级(每 SM 或全片) | 带宽量级 | 延迟量级 | 类比 CPU |
|---|---|---|---|---|
| 寄存器 Register | 256 KB / SM(65536 个 32-bit) | ~数十 TB/s(私有) | ~1 周期 | 寄存器 |
| L1 / Shared Memory(SRAM) | 192 KB / SM(可配比) | ~十几 TB/s(片上) | ~20-30 周期 | L1 |
| L2 Cache | 40 MB(全片共享) | ~数 TB/s | ~200 周期 | L2/L3 |
| HBM(全局显存,DRAM) | 80 GB | ~2 TB/s | ~400-600 周期 | 主内存 DRAM |
| 主机内存(经 PCIe / NVLink) | 数百 GB+ | PCIe Gen4 x16 ~32 GB/s 单向(双向聚合 ~64 GB/s)「待核」 | ~微秒级 | 跨 NUMA / 磁盘 |

注意 **Shared Memory(共享内存)就是物理上的 L1 SRAM**——在 NVIDIA 架构里它们共用同一块片上 SRAM,程序员可以**手动管理**其中一部分(这是 Shared Memory,被一个线程块 block 内的线程共享),另一部分由硬件当 L1 cache 用。**这块由软件显式管理的 SRAM,是 FlashAttention、所有高性能 GEMM kernel 的舞台**——它们的核心套路就是"把数据从 HBM 搬进 SRAM,在 SRAM 里反复算,算完再写回",最大限度减少对慢速 HBM 的访问。

### 为什么带宽差这么多:物理与经济的双重约束

读者会问:既然寄存器/SRAM 这么快,为什么不把 80GB 全做成 SRAM?

**根本原因是面积、功耗、成本和物理距离。**

```
SRAM(寄存器/L1):每个 bit 用 6 个晶体管(6T cell),速度快但面积大、贵
DRAM(HBM):    每个 bit 用 1 个晶体管 + 1 个电容(1T1C),密度高、便宜但慢

→ 同样的硅片面积,SRAM 存的容量是 DRAM 的几十分之一
→ 把 80GB 做成 SRAM 的芯片面积和成本将完全不可行
```

带宽差异还有第二层物理原因:**距离即延迟,宽度即带宽**。

- 寄存器/SRAM 紧贴计算单元(片上,距离亚毫米级),数据线又宽又短,带宽天然高、延迟天然低。
- HBM 是堆叠在芯片旁边的独立 DRAM 裸片,通过硅中介层(interposer)的超宽总线(数千根线)连接——它已经是当前能做到的最快的大容量内存方案了(HBM = High Bandwidth Memory,高带宽内存),但物理上仍然离计算单元更远、是 DRAM 工艺,带宽和延迟都比片上 SRAM 差一个量级以上。
- PCIe 连接主机内存,要走出芯片、经过主板,带宽再降一个量级(几十 GB/s),这就是为什么"数据从 CPU 搬到 GPU"是训练流水线里要竭力隐藏的开销。

**一句话总结这一节:从寄存器到 HBM,带宽大约逐级降一个数量级,容量逐级升一个数量级。优化的全部艺术,就是让数据尽量待在金字塔上层。**

## 四、Memory Wall:为什么"带宽增速长期落后于算力增速"

这是本章最重要的物理事实,也是整个 MLSys 领域存在的根本原因之一。

### 现象:算力跑得比带宽快

观察过去十年 GPU 代际演进会看到一个稳定的趋势:**峰值算力(FLOPS)的增长速度,长期快于内存带宽(B/s)的增长速度。** 二者的比值——**每字节带宽对应的浮点运算能力**——在持续变大。

```
定义机器平衡点(machine balance):
   B_machine = 峰值算力(FLOP/s) / 峰值带宽(Byte/s)   单位:FLOP/Byte

A100(FP16 Tensor Core)量级估算「待核」:
   B_machine ≈ 312e12 FLOP/s / 2.0e12 Byte/s ≈ 156 FLOP/Byte

直觉解读:这台机器每从 HBM 搬来 1 个字节,就"有能力"算 156 次浮点运算。
   如果你的算法每搬 1 字节算不到 156 次,带宽就成了瓶颈——计算单元在等数据。
```

**这个 156 FLOP/Byte 就是后面 Roofline 模型的"脊点(ridge point)"**,记住它。

### 为什么带宽天生追不上算力:第一性原理

算力的增长可以靠**堆更多计算单元 + 用更激进的低精度(FP32→FP16→FP8)**:精度减半,同样的晶体管面积能塞下约两倍的乘加器,算力近似翻倍——这条路相对好走。

而内存带宽受制于更硬的物理约束:

```
内存带宽 = 总线位宽 × 信号频率

- 总线位宽:受引脚数 / 中介层布线密度限制,不能无限加宽(物理空间、串扰)
- 信号频率:受功耗墙限制(频率越高功耗越大,IO 功耗已是大头),以及信号完整性
- HBM 靠"堆叠 + 超宽总线"硬撑带宽,但每代提升幅度受工艺和封装制约
```

**结论:低精度让算力可以"作弊式"翻倍,而带宽必须靠真实的物理工艺进步,后者慢得多。于是机器平衡点 B_machine 逐代变大,越来越多算子从 compute-bound 滑向 memory-bound。** 这就是 **memory wall(内存墙)**——它不是某个厂商的失误,而是物理规律决定的结构性趋势。

> 系统老兵会心一笑:这就是 CPU 领域 1990s 就提出的 "memory wall" 论文(Wulf & McKee)的 GPU 重演,连机制都一样——处理器变快,内存追不上,差距逐年扩大。你过去为 CPU 做的 cache blocking、数据局部性优化,在 GPU 上以"用 Shared Memory 做 tiling"的形式完全复活。

## 五、Roofline 模型:用一张图判定瓶颈(本章核心)

现在把前面的物理量组装成一个**可操作的判定工具**。这是系统性能分析的"老工具",MLSys 直接拿来用。

### 第一性原理:算术强度(Arithmetic Intensity)

任何 kernel 跑一次,有两个不可约的物理量:

- **W = 计算量**(单位 FLOP,浮点运算次数)
- **Q = 访存量**(单位 Byte,从 HBM 读 + 写回的总字节数)

定义**算术强度(arithmetic intensity,也叫计算密度、operational intensity)**:

```
I = W / Q     单位:FLOP/Byte

含义:每从内存搬运 1 字节,这个算法做了多少次浮点运算。
I 大 → 计算密集,数据来一次能算很久(GEMM)
I 小 → 访存密集,搬来的数据算两下就扔了(逐元素加法)
```

**算术强度是算法的内禀属性,不依赖硬件**(W 和 Q 由算法决定)。而硬件提供两个上限:峰值算力 `π`(FLOP/s)和峰值带宽 `β`(Byte/s)。

### 推导:可达性能的上界

一个 kernel 跑完需要的时间,**至少**是"算完所需时间"和"搬完所需时间"中的较大者(理想情况下二者完全重叠):

```
推导(白板版):
   计算耗时下界:T_compute = W / π
   访存耗时下界:T_memory  = Q / β
   实际耗时:    T ≥ max(T_compute, T_memory)        ……(理想重叠假设)

   有效性能 P = W / T ≤ W / max(W/π, Q/β)
             = min( π ,  W/Q · β )
             = min( π ,  I · β )                      ……(代入 I = W/Q)
```

**这就是 Roofline 公式:** `P ≤ min(π, I·β)`。把它画出来(横轴 I 取对数,纵轴 P 取对数):

```
可达性能 P
  ↑
π ┤            ________________  ← 算力屋顶(compute roof,水平线 P=π)
  │           /
  │          /  ← 带宽屋顶(memory roof,斜线 P = I·β,斜率=β)
  │         /
  │        /
  └───────┴──────────────────→ 算术强度 I (FLOP/Byte)
          I* = π/β
        (脊点 ridge point)
```

**脊点(ridge point)** 在 `I* = π/β` 处——这正是第四节算的**机器平衡点 B_machine**!两个名字,同一个量。

### 判定准则(背下来)

```
若 I < I*(脊点左侧):  P 被斜线 I·β 限制 → memory-bound(访存受限)
                       此时性能 = I·β,只取决于带宽,堆再多算力也没用
若 I > I*(脊点右侧):  P 被水平线 π 限制 → compute-bound(计算受限)
                       此时性能 = π,只有更快的计算单元才能提速

A100 FP16:I* ≈ 156 FLOP/Byte
```

**这个判定决定了你该往哪个方向优化:memory-bound 的算子,优化方向是减少访存(融合、留在 SRAM、提高数据复用);compute-bound 的算子,优化方向是用满 Tensor Core、提高计算效率。搞反方向 = 白费力气。**

### 例子一:稠密 GEMM —— 通常 compute-bound

矩阵乘 `C[M,N] = A[M,K] · B[K,N]`,数据类型 FP16(2 字节)。

```
计算量 W = 2·M·N·K  FLOP
   (输出 M·N 个元素,每个是 K 项乘加,1 次乘加 = 2 FLOP)

访存量 Q(最理想,每个矩阵只读/写一次):
   读 A: M·K·2 字节,读 B: K·N·2 字节,写 C: M·N·2 字节
   Q = 2·(M·K + K·N + M·N) 字节

算术强度:
   I = 2MNK / [2(MK + KN + MN)] = MNK / (MK + KN + MN)

取方阵 M=N=K=4096:
   分子 = 4096³ = 6.87e10
   分母 = 3 × 4096² = 5.03e7
   I = 6.87e10 / 5.03e7 ≈ 1365 FLOP/Byte
```

**1365 ≫ 脊点 156 → 大方阵 GEMM 是 compute-bound。** 这就是为什么 GPU 跑大矩阵乘能逼近峰值算力——它正是 GPU 设计的甜点。

但注意一个**坑**:小矩阵或瘦长矩阵会破坏这个结论。取 M=N=4096、K=64(扁平的 GEMM,常见于某些投影层):

```
I = 4096·4096·64 / (4096·64 + 64·4096 + 4096·4096)
  = 1.07e9 / (2.6e5 + 2.6e5 + 1.68e7)
  ≈ 1.07e9 / 1.73e7 ≈ 62 FLOP/Byte
```

**62 < 156 → 这个扁 GEMM 反而是 memory-bound!** 教训:**GEMM 不是天然 compute-bound,只有当三个维度都足够大时才是。** 推理时的 batch=1 解码、LoRA 的低秩矩阵,都可能让"矩阵乘"掉进 memory-bound 区——这是后面推理优化章节的重要伏笔。

### 例子二:逐元素操作(elementwise)—— 必然 memory-bound

以 GeLU 激活 `y = gelu(x)` 为例,x 是 N 个 FP16 元素。

```
计算量 W ≈ c·N FLOP(c 是每元素的运算次数,GeLU 每元素约 10 次浮点运算,常数级)
访存量 Q = 读 N·2 + 写 N·2 = 4N 字节

算术强度:
   I = c·N / (4N) = c/4 ≈ 10/4 ≈ 2.5 FLOP/Byte（量级,与 N 无关）
```

**关键洞察:I 与数据规模 N 无关,永远是个小常数!** 因为读一个数、算几下、写回去,计算和访存都正比于 N,比值是固定的小数。

```
2.5 ≪ 脊点 156 → 逐元素操作铁定是 memory-bound。
此时可达性能 P = I·β ≈ 2.5 × 2.0e12 = 5e12 FLOP/s = 5 TFLOPS
对比峰值 312 TFLOPS → 只能用上 1.6% 的算力!
```

**这解释了一个让新手震惊的现象:一个 GeLU 跑得到的"算力利用率"只有百分之一点几——不是 GPU 坏了,是这个算子的算术强度决定了它根本碰不到算力屋顶。** 它的真实瓶颈是带宽,它实际"跑满"的是带宽屋顶(接近 2 TB/s)。

**这也是算子融合(operator fusion)的全部理论依据:** 把 `z = gelu(x); w = z + b; ...` 这一串逐元素操作融合成一个 kernel,中间结果 z、w 留在寄存器/SRAM 里不写回 HBM,**访存量 Q 大幅下降,算术强度 I 上升,kernel 从深度 memory-bound 往脊点靠**。融合不减少 FLOP(W 不变),它减少的是 Q——这是 memory-bound 优化的通用范式。

### 例子三:注意力(Attention)—— memory-bound,且是 FlashAttention 的动机

标准注意力(单头,序列长 S,头维度 d,FP16):

```
计算:S = softmax(Q·Kᵀ / √d) · V
   两个矩阵乘:Q·Kᵀ 是 [S,d]·[d,S] = 2·S²·d FLOP
              P·V   是 [S,S]·[S,d] = 2·S²·d FLOP
   W ≈ 4·S²·d FLOP（softmax 的 FLOP 是低阶项,忽略）

访存(朴素实现的致命问题):中间的注意力分数矩阵 P[S,S] 要写回 HBM 再读回来
   读 Q,K,V: 3·S·d·2 字节(小项)
   写 P 矩阵: S²·2 字节
   读 P 矩阵: S²·2 字节(softmax 时)
   主导项 Q ≈ 4·S² 字节(那个 S×S 的大矩阵来回搬)

算术强度(大 S 时,d 远小于 S):
   I ≈ 4S²d / (4S²) = d  FLOP/Byte
```

**惊人结论:朴素注意力的算术强度约等于头维度 d!** 典型 d=64 或 128,而脊点是 156。

```
d = 128 < 156 → memory-bound(且非常接近脊点,但仍在左侧)
d = 64  < 156 → 更深的 memory-bound
```

**瓶颈来源被精确定位了:那个 S×S 的注意力分数矩阵在 HBM 上来回搬运。S=4096 时它有 4096² × 2 字节 = 32 MB,远超 40MB 的 L2、更远超 192KB 的 SRAM,只能落 HBM。** 序列越长,这个 `O(S²)` 的访存越恐怖。

**FlashAttention 的核心思想由此一目了然:绝不把完整的 S×S 矩阵写回 HBM。** 它用分块(tiling)把 Q、K、V 切成能装进 SRAM 的小块,在 SRAM 里增量地算 softmax(online softmax 技巧),只把最终的 [S,d] 输出写回 HBM。**它一点没减少 FLOP(W 不变,甚至因重算略增),却把访存量 Q 从 `O(S²)` 降到 `O(S·d)` 量级,算术强度大幅抬高,把一个 memory-bound 算子拉向 compute-bound——于是快了好几倍。** 这就是"IO 感知算法(IO-aware algorithm)"的精髓,而它的全部理论支撑就是本章的 Roofline。

**到这里你应该有一个统一认知:GEMM 是 GPU 的舒适区(高算术强度),而深度学习里大量的胶水操作(逐元素、归一化、注意力的朴素实现)都卡在带宽上。MLSys 的半壁江山,就是把这些 memory-bound 算子往脊点右边推。**

## 六、三个实战必知概念:kernel 启动、occupancy、coalescing

Roofline 给的是**理论上界**。现实中还有三个工程因素让你跑不到上界,它们是性能调优的日常战场。

### 6.1 Kernel 启动开销:小算子的隐形税

每次从 CPU 启动一个 GPU kernel,有固定的**启动延迟(launch overhead,量级约几微秒「待核」)**——要把 kernel 配置、参数传过去,经过驱动和硬件队列。

```
若一个 kernel 实际计算只需要 2 微秒,而启动开销 5 微秒
→ 70% 的时间花在"启动"上,GPU 大部分时间空着
```

这对**小张量、大量碎片化小算子**是灾难。一个未优化的模型可能有成千上万个微型 kernel(各种 elementwise、reshape、小 matmul),启动开销累积起来主导了整个耗时。

**对策:** (1) 算子融合,把 N 个小 kernel 合成 1 个,N 倍地省启动开销;(2) **CUDA Graph**——把一串 kernel 的启动序列录制成一张图,之后整张图一次性重放,几乎消除逐个启动的 CPU 开销(推理小 batch 解码场景收益巨大);(3) 增大 batch,让每个 kernel 的计算时间远大于启动开销,把固定成本摊薄。

### 6.2 Occupancy(占用率):延迟隐藏够不够

回到第二节的延迟隐藏机制。**occupancy = 实际驻留 warp 数 / 硬件最大可驻留 warp 数**。它衡量"SM 上有没有足够多的 warp 来填补访存延迟的空窗"。

驻留 warp 数受三种**片上资源**竞争限制,谁先耗尽谁封顶:

```
限制因素(三者取最严的那个):
   1. 寄存器:每 SM 寄存器总数 / (每线程用的寄存器数 × 线程数)
      → kernel 用寄存器越多,能塞下的 warp 越少
   2. Shared Memory:每 SM 的 SRAM 总量 / 每个 block 申请的 shared mem
      → block 申请的共享内存越大,能并发的 block 越少
   3. 硬件硬上限:每 SM 最大 warp 数 / 最大 block 数

估算示例(A100,数值量级「待核」):
   设每 SM 寄存器 65536 个,kernel 每线程用 64 个寄存器
   每 SM 可驻留线程 = 65536 / 64 = 1024 线程 = 32 warp
   若硬件上限 64 warp → occupancy = 32/64 = 50%
```

**关键澄清(新手大坑):occupancy 不是越高越好,也不是越高越快。** 它只是"延迟隐藏能力"的代理指标。一旦占用率高到足以掩盖访存延迟(常常 50% 上下就够),再往上提对性能几乎无益;而强行提占用率(比如限制寄存器用量)反而可能逼出寄存器溢出(register spilling,寄存器不够用,变量被踢到慢速 local memory),得不偿失。**高性能 kernel(如 FlashAttention)经常故意用较低占用率换取每线程更多寄存器/更大 tile**——因为它的瓶颈不是延迟隐藏,而是数据复用。**正确姿势:occupancy 是一个"够用即可"的阈值,不是优化目标本身。**

### 6.3 Coalescing(合并访存):为什么访存模式能让带宽差 10 倍

这是 memory-bound 算子里最致命的一个因素,而且最反直觉。

机制:warp 里 32 个线程同时发出访存请求时,硬件以**固定大小的内存事务(transaction,如 32B/64B/128B 一段「待核」)** 为单位从 HBM 取数据。

```
合并访存(coalesced,好):
   warp 内 32 个线程访问连续的 32 个地址(线程 i 访问地址 base + i)
   → 这些地址落在同一段或少数几段连续内存
   → 硬件用 1 次(或极少次)宽事务全取回,每字节都被用上,带宽利用率接近 100%

非合并访存(uncoalesced / strided,坏):
   32 个线程访问分散的地址(如每个相隔 1024 字节)
   → 每个线程触发一次独立的宽事务,但事务里大部分字节没用上就被丢弃
   → 最坏情况:取回 32 段、每段只用 1 个数,有效带宽降到 1/段大小
```

**直觉:HBM 不是按"单个字节"卖货,是按"一整段(cache line 量级)"卖货。** 你买一整段却只用其中一个数,就浪费了整段的带宽。**连续访问 = 整段都用上 = 满带宽;跳跃访问 = 每段只用一点 = 带宽被砍到零头。**

这就是为什么**数据布局(memory layout)在 GPU 上是性能问题而非风格问题**:

- 行优先存储的矩阵,让 warp 沿行方向(连续维度)访问 → 合并;沿列方向访问 → 跨步、非合并。
- 这直接催生了对 **NHWC vs NCHW** 张量布局的选择、对转置操作要用 Shared Memory 做缓冲(先合并读进 SRAM、在 SRAM 里转置、再合并写出)的经典套路。

> 系统老兵的迁移:coalescing 就是 CPU 上的"cache line 利用率 / 空间局部性"。CPU 上你避免 cache 行只用一个字节(false sharing、跨 cache 行跳访问),GPU 上你避免 warp 的访存事务只用一个字节。**同一个物理原理:内存按块搬运,要让块里每个字节都有用。**

## 七、动手:用 PyTorch 实测,反推有效带宽/算力,对照理论峰值

光说不练等于没学。下面这段代码是**性能分析的基本功**:实测一个算子的耗时,反推它实际达到的带宽或算力,和理论峰值对比,从而验证它落在 Roofline 的哪个屋顶下。

**测时的两个铁律(新手必踩的坑):** (1) GPU 是异步的,Python 调用返回不代表算完,必须用 CUDA event 计时并 `synchronize`;(2) 第一次调用包含 cuDNN 算法选择、显存分配等冷启动开销,必须先 **warmup**,再多次取平均。

```python
import torch

def benchmark(fn, *args, warmup=20, iters=100):
    """实测 fn 的平均耗时(毫秒),正确处理 GPU 异步。"""
    # 1) warmup:触发 cuDNN autotune、缓存分配,排除冷启动
    for _ in range(warmup):
        fn(*args)
    torch.cuda.synchronize()          # 等所有 warmup kernel 真正跑完

    # 2) 用 CUDA event 计时(在 GPU 时间线上打点,不受 CPU 调度干扰)
    start = torch.cuda.Event(enable_timing=True)
    end   = torch.cuda.Event(enable_timing=True)
    start.record()
    for _ in range(iters):
        fn(*args)
    end.record()
    torch.cuda.synchronize()          # 必须同步,否则 elapsed_time 读到的是空值
    return start.elapsed_time(end) / iters   # 毫秒/次


# ---------- 实验 A:逐元素加法(预期 memory-bound) ----------
N = 1 << 26                            # 6700 万个元素
x = torch.randn(N, device='cuda', dtype=torch.float16)
y = torch.randn(N, device='cuda', dtype=torch.float16)

t_ms = benchmark(lambda a, b: a + b, x, y)

# 访存量 Q:读 x、读 y、写结果,各 N 个 fp16(2 字节)
bytes_moved = 3 * N * 2
eff_bw = bytes_moved / (t_ms * 1e-3) / 1e12      # TB/s
# 计算量 W:N 次加法
eff_flops = N / (t_ms * 1e-3) / 1e12             # TFLOPS
print(f"[add]  {t_ms:.3f} ms | 有效带宽 {eff_bw:.2f} TB/s | 有效算力 {eff_flops:.2f} TFLOPS")
# 预期:有效带宽逼近 ~2 TB/s(撞带宽屋顶),有效算力远低于 312(碰不到算力屋顶)


# ---------- 实验 B:大方阵 GEMM(预期 compute-bound) ----------
M = K = Nn = 4096
A = torch.randn(M, K, device='cuda', dtype=torch.float16)
B = torch.randn(K, Nn, device='cuda', dtype=torch.float16)

t_ms = benchmark(lambda a, b: a @ b, A, B)

flops = 2 * M * Nn * K                            # 2MNK
eff_tflops = flops / (t_ms * 1e-3) / 1e12
bytes_moved = 2 * (M*K + K*Nn + M*Nn) * 2         # 理想访存(实际有复用,这是下界)
eff_bw = bytes_moved / (t_ms * 1e-3) / 1e12
print(f"[gemm] {t_ms:.3f} ms | 有效算力 {eff_tflops:.1f} TFLOPS | 名义带宽 {eff_bw:.2f} TB/s")
# 预期:有效算力逼近 ~312 TFLOPS(撞算力屋顶)
```

**怎么用这段代码判定瓶颈(方法论):**

1. 算出该算子的算术强度 I,标在 Roofline 横轴上,看它落在脊点左还是右,得到**理论预测**。
2. 实测有效带宽和有效算力。
3. 对照:**memory-bound 算子(实验 A),看有效带宽是否逼近峰值带宽**——逼近了说明带宽用满了,这就是它的天花板,别再想着提算力利用率;**compute-bound 算子(实验 B),看有效算力是否逼近峰值算力**——逼近了说明 Tensor Core 用满了。
4. 若两者都远未逼近,说明卡在第六节的工程因素上(启动开销、占用率、非合并访存),用 Nsight Compute 进一步归因。

**进阶工具(强烈建议上手):** PyTorch 自带 `torch.profiler` 给出逐 kernel 耗时;NVIDIA **Nsight Systems**(nsys,看时间线、找 CPU/GPU 空隙、kernel 启动开销)和 **Nsight Compute**(ncu,单 kernel 深挖,直接给出该 kernel 的算术强度、内存吞吐、占用率、是否合并访存,甚至直接画出 Roofline 图)。**ncu 的 Roofline section 会把你的 kernel 作为一个点画在 Roofline 上,一眼看出离哪个屋顶多远——这是把本章理论变成肌肉记忆的最快路径。**

## 八、设计权衡与常见坑

把本章踩坑清单集中列出,这些是把理论用错的高频姿势:

- **坑 1:看到"算力利用率只有 2%"就以为代码写烂了。** 先算算术强度。如果是 memory-bound 算子,2% 的算力利用率是物理上限,正常。该看的指标是**带宽利用率**而非算力利用率。**用错指标 = 优化方向全错。**

- **坑 2:盲目追求高 occupancy。** 占用率是"够用即可"的阈值,不是目标。强行拉满可能引发寄存器溢出反而变慢。高性能 kernel 常主动用低占用率换大 tile、换数据复用。

- **坑 3:用 `time.time()` 测 GPU 算子。** GPU 异步,Python 返回 ≠ 算完。不 warmup、不 synchronize 测出来的数全是错的(经常测出"快得不真实"或抖动巨大)。必须 CUDA event + synchronize + warmup。

- **坑 4:以为 GEMM 一定 compute-bound。** 只有三个维度都够大才是。瘦长矩阵、batch=1 解码、低秩矩阵会掉进 memory-bound。**判定要算 I,不能凭算子类型拍脑袋。**

- **坑 5:忽视 kernel 启动开销。** 模型里大量碎片小算子时,瓶颈可能根本不是计算或带宽,而是上千次 kernel 启动的累积固定开销。用 nsys 看时间线上的空隙就能发现。对策是融合 + CUDA Graph。

- **坑 6:把 Roofline 当精确预测器。** 它是**上界**和**归因框架**,不是精确性能模型。真实性能受 cache 命中、L2 复用、tensor core 利用效率、bank conflict 等影响,常低于 Roofline 上界。它的价值在于**快速判定瓶颈方向和给出"还差多少"的标尺**,不在于精确到个位数。

- **权衡总览:** GPU 优化本质是在一个三角里腾挪——**减少访存(融合、tiling、低精度)、提高计算效率(用满 Tensor Core)、隐藏延迟(够用的占用率 + 计算通信重叠)**。先用 Roofline 判定瓶颈在哪条边,再针对性发力,别three边乱使劲。

## 九、动手练习

**练习 1(估算题 · 判瓶颈):LayerNorm 的算术强度。**
对一个 [B, S, H] 的张量做 LayerNorm(沿 H 维归一化),FP16。每个元素大致经历"求均值→求方差→归一化→缩放平移",计算量按每元素约 10 FLOP 估算。
(a) 写出计算量 W 和访存量 Q(提示:朴素实现要读入全部元素、写出全部元素,中间统计量是低阶项;设元素总数 n = B·S·H)。
(b) 求算术强度 I,它依赖 n 吗?
(c) 对照 A100 脊点 156,判定 compute-bound 还是 memory-bound,估算它最多能用上峰值算力的百分之几。
> 提示:结论应与第五节的逐元素例子高度一致——I 是个与规模无关的小常数。这解释了为什么 LayerNorm/RMSNorm 是算子融合的头号目标。

**练习 2(估算题 · occupancy):占用率受谁限制。**
某 kernel 每线程使用 96 个寄存器,每个 block 256 线程、申请 16 KB Shared Memory。设 A100 单 SM 有 65536 个寄存器、164 KB Shared Memory、最大驻留 2048 线程「待核」。
(a) 仅看寄存器,单 SM 能驻留多少线程?多少 warp?
(b) 仅看 Shared Memory,单 SM 能驻留多少个 block?多少线程?
(c) 取三者(寄存器、shared mem、硬件上限 2048 线程)的最小值,算出实际占用率。哪个是瓶颈资源?
> 提示:逐个资源算出"能驻留多少线程",取最严的那个。这正是 CUDA Occupancy Calculator 的逻辑。

**练习 3(推导题 · 融合收益):算子融合把算术强度提高多少。**
有一串三个逐元素操作 `b = relu(a); c = b * s; d = c + bias`,a 是 n 个 FP16 元素,s 和 bias 是标量/广播。
(a) **不融合**(三个独立 kernel):写出总访存量 Q_unfused(每个 kernel 都要读输入、写输出回 HBM)。
(b) **融合成一个 kernel**:中间结果 b、c 留在寄存器不落 HBM,写出 Q_fused。
(c) 求 Q_unfused / Q_fused,即访存量降低倍数;在算术强度不变的 W 下,这等价于 I 提高多少倍?
> 提示:不融合时每个中间结果都要"写出去再读回来",这是纯粹的浪费。融合的收益正比于被消除的中间张量数。

**练习 4(编码题 · 实测验证):测一个你怀疑 memory-bound 的算子。**
用第七节的 `benchmark` 函数,实测 `torch.nn.functional.softmax`(对一个 [8192, 8192] 的 FP16 张量沿最后一维)。
(a) 估算它的访存量 Q(softmax 要至少读一遍、写一遍;数值稳定实现可能多读一遍求 max)。
(b) 反推有效带宽,和 A100 峰值带宽(~2 TB/s)对比,看用了百分之几。
(c) 如果有效带宽远低于峰值,用 `torch.profiler` 或 ncu 看看是不是被拆成了多个 kernel(求 max、求 sum、归一化各一遍)导致反复读写 HBM——这正是 FlashAttention 里"online softmax 融合"要解决的问题。
> 提示:数值稳定的 softmax 通常是 `exp(x - max(x)) / sum(...)`,朴素实现对 x 扫多遍,访存量是单遍的好几倍。

## 十、源码 / 论文导读

精读优先级从高到低:

- **Roofline 原始论文:** Williams, Waterman, Patterson, *"Roofline: An Insightful Visual Performance Model for Multicore Architectures"*(CACM 2009)。读它对"算术强度"和"脊点"的定义,以及如何用一张图同时表达计算和带宽屋顶——本章第五节就是它的 GPU 化复述。

- **Memory Wall 的源头:** Wulf & McKee, *"Hitting the Memory Wall: Implications of the Obvious"*(1995)。两页纸的经典,论证处理器与内存速度差距的指数级扩大不可避免——理解第四节"为什么带宽追不上算力"的历史根。

- **FlashAttention:** Dao et al., *"FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness"*(NeurIPS 2022)。重点读它对**标准注意力 HBM 访问量**的分析(论文 Section 2-3),以及 tiling + online softmax 如何把访存从 `O(S²)` 降到 `O(S²d²/M)`(即 `O(N²d²/M)`,M 为 SRAM 大小)。这是本章第五节例子三的完整工程化,**强烈建议把它的 IO 复杂度推导和本章 Roofline 对照着读**。

- **NVIDIA 官方:** *CUDA C++ Programming Guide* 的 "Hardware Implementation"(SIMT、warp、占用率)和 "Performance Guidelines"(coalescing、memory throughput)两章;*Nsight Compute* 文档里的 **Roofline analysis** 一节(它会自动算你的 kernel 算术强度并画在 Roofline 上)。

- **代码层面:** 读 **CUTLASS**(NVIDIA 开源 GEMM 模板库)的 tiling 与 Shared Memory 使用,理解"把数据搬进 SRAM 复用"的工程实现;读 **flash-attention** 仓库的 CUDA kernel,看 tiling 和 online softmax 的真实落地。先不求看懂每一行,先建立"高性能 kernel 的形状长什么样"的直觉。

具体硬件数值(各代 GPU 的峰值算力/带宽/SRAM 大小/最大 warp 数)请以对应 GPU 的官方 **datasheet / Architecture Whitepaper** 为准,本章数字均为量级锚点「待核」。

## 十一、小结与承上启下

这一章给了你一副**物理眼镜**,以后看任何 MLSys 技术都能看穿它在拆哪堵墙:

- **执行模型:** GPU 靠 warp(32 线程锁步)做 SIMT 并行,靠超额订阅 warp 来**隐藏**而非降低延迟;Tensor Core 把 GEMM 硬件化,让低精度算力涨一个量级。
- **内存层级:** 寄存器→SRAM→L2→HBM→主机,带宽逐级降一个数量级、容量逐级升一个数量级;根因是 SRAM/DRAM 工艺与物理距离。优化的母题就是"让数据待在金字塔上层"。
- **memory wall:** 低精度让算力可以作弊式翻倍,带宽只能靠真实工艺进步,所以机器平衡点逐代变大,越来越多算子滑向 memory-bound——这是整个领域的根本张力。
- **Roofline:** `P ≤ min(π, I·β)`,脊点 `I* = π/β`。算术强度 `I = W/Q` 是算法内禀属性,一算就知道瓶颈在带宽还是算力。GEMM(大维度)在算力屋顶下,逐元素/朴素注意力在带宽屋顶下。
- **工程三因素:** kernel 启动开销、occupancy(够用即可)、coalescing(内存按段搬,要让每字节有用)。

**在整门课里的位置:** 这是第 0 章,是所有后续章节的物理坐标系。

- 下一章(01 训练迭代解剖)会把"一步训练"拆成算力账(`C ≈ 6ND`)和显存账(每参数 ~16 字节),那本账回答的是**显存墙和算力墙有多高**;本章回答的是**为什么墙拆起来这么难、拆的方向怎么判定**。
- 再往后的并行章节(02 数据并行起)会引入第三个轴——**通信带宽**(NVLink、PCIe、跨机网络),它和本章的 HBM 带宽是同构的:同样可以为通信定义"算术强度",同样用 Roofline 式思维判定一个并行策略是 compute-bound 还是 communication-bound。**你在本章建立的"计算 vs 数据搬运竞赛"直觉,会原封不动地从片内(HBM)推广到片间(NVLink)、机间(网络)。**
- FlashAttention、算子融合、量化等所有单卡优化,本质都是本章 Roofline 的应用:**减少 Q(访存)或减少 W(计算)或用满 π(算力),把算子在 Roofline 上往右上角推。**

记住这门课最朴素的一句话:**慢,要么是算不过来(compute-bound),要么是搬不过来(memory-bound),而现代深度学习的瓶颈,绝大多数时候是后者。** 把这句话刻进直觉,你已经超过了一半只会调 API 的人。

### 练习 1:LayerNorm 的算术强度

**(a) 计算量 W 与访存量 Q**

设元素总数 n = B·S·H,FP16 每元素 2 字节。

```
计算量 W:每元素约 10 FLOP（均值/方差/归一化/缩放平移都是 O(1) 次运算）
   W = 10·n  FLOP

访存量 Q（朴素实现,主导项)：
   读入全部元素:  n × 2 字节
   写出全部元素:  n × 2 字节
   （中间统计量:每行只产生 1 个 mean、1 个 var,是 O(n/H) 的低阶项,忽略）
   Q = 4·n  字节
```

**(b) 算术强度 I**

```
I = W / Q = 10·n / (4·n) = 10/4 = 2.5  FLOP/Byte
```

n 在分子分母里**完全约掉**。

**结论:I = 2.5 FLOP/Byte,与张量规模 n 无关,是一个固定的小常数。** 原因和第五节逐元素例子一模一样——读一个数、算几下、写回去,计算与访存都正比于 n,比值恒定。

**(c) 判瓶颈 + 算力利用率上界**

```
2.5 ≪ 脊点 156  →  memory-bound（访存受限,且远在脊点左侧,很深）

可达性能 P = I·β = 2.5 × 2.0e12 = 5.0e12 FLOP/s = 5 TFLOPS
占峰值算力比例 = 5 / 312 ≈ 1.6%
```

**结论:LayerNorm 是深度 memory-bound 算子,最多只能用上 A100 峰值算力的约 1.6%。** 该看的指标是**带宽利用率**(应逼近 2 TB/s),而非算力利用率。正因为 I 是与规模无关的小常数,LayerNorm/RMSNorm 这类归一化算子是算子融合的头号目标——把它和相邻的逐元素操作融进同一个 kernel,中间结果不落 HBM,就能把多个 memory-bound 算子的访存合并掉。

### 练习 2:占用率受谁限制

已知:每线程 96 个寄存器,每 block 256 线程,每 block 申请 16 KB Shared Memory;A100 单 SM 有 65536 个寄存器、164 KB Shared Memory、最大驻留 2048 线程。

**说明:真实 CUDA 资源以「block 为粒度」分配(一个 block 要么整个驻留、要么不驻留),所以寄存器/共享内存都要先算「每 block 用多少」再向下取整 block 数。下面按这个真实口径算,并在 (a) 末尾标注与「纯线程粒度」的差别。**

**(a) 仅看寄存器**

```
每 block 用寄存器 = 96 × 256 = 24576 个
单 SM 可驻留 block 数 = ⌊65536 / 24576⌋ = ⌊2.67⌋ = 2 个 block
   → 线程数 = 2 × 256 = 512 线程 = 16 warp
```

(若忽略 block 取整、按纯线程粒度估:65536 / 96 ≈ 682 线程 ≈ 21 warp。但 682 不是 256 的整数倍,无法凑成整 block,真实可驻留只有 512 线程——这正是 block 粒度取整带来的损耗。)

**(b) 仅看 Shared Memory**

```
单 SM 可驻留 block 数 = ⌊164 KB / 16 KB⌋ = ⌊10.25⌋ = 10 个 block
   → 线程数 = 10 × 256 = 2560 线程
```

**(c) 取三者最小值 + 找瓶颈**

```
三种限制各自能驻留的线程数:
   寄存器:        512 线程   ← 最严
   Shared Memory: 2560 线程
   硬件上限:      2048 线程

实际驻留 = min(512, 2560, 2048) = 512 线程

占用率 occupancy = 512 / 2048 = 25%
```

**结论:瓶颈资源是寄存器,实际占用率仅 25%。** 该 kernel 每线程吃掉 96 个寄存器(偏多),把可驻留 warp 数从硬件上限的 64 warp 压到了 16 warp。

提醒(呼应第六节坑 2):25% 占用率**不一定**意味着性能差。若该 kernel 是 compute-bound 或靠大 tile 做数据复用(如 FlashAttention 风格),低占用率换更多寄存器/更大 tile 是**有意为之**的正确选择。只有当它是 memory-bound 且确实因 warp 不足而无法隐藏访存延迟时,才需要考虑通过减少每线程寄存器用量来提占用率(但要警惕逼出 register spilling)。

### 练习 3:算子融合把算术强度提高多少

一串三个逐元素操作 `b = relu(a); c = b * s; d = c + bias`,a 是 n 个 FP16(2 字节)元素,s、bias 为标量/广播(访存可忽略)。

**(a) 不融合(三个独立 kernel)**

每个 kernel 都要从 HBM 读输入、把输出写回 HBM:

```
kernel1 relu:  读 a (2n) + 写 b (2n) = 4n 字节
kernel2 mul:   读 b (2n) + 写 c (2n) = 4n 字节
kernel3 add:   读 c (2n) + 写 d (2n) = 4n 字节

Q_unfused = 4n + 4n + 4n = 12n  字节
```

**(b) 融合成一个 kernel**

中间结果 b、c 算出来后留在寄存器,不写回 HBM:

```
全程只:读 a (2n) + 写最终结果 d (2n)

Q_fused = 2n + 2n = 4n  字节
```

**(c) 访存量降低倍数 = 算术强度提升倍数**

```
Q_unfused / Q_fused = 12n / 4n = 3 倍
```

融合**不改变计算量 W**(relu/mul/add 的 FLOP 数一个没少),只把 Q 降为 1/3。由 I = W/Q:

```
I_fused / I_unfused = Q_unfused / Q_fused = 3 倍
```

**结论:融合把访存量降为 1/3,算术强度提高 3 倍,kernel 在 Roofline 上沿带宽斜线往脊点方向移动 3 倍距离。** 通项规律:这类全逐元素链路,不融合时 Q = 4n·k(k 为 kernel 数),融合后 Q = 4n(只剩首读+尾写),提升倍数 = k = 操作个数。**收益正比于被消除的中间张量数**——每消掉一个落 HBM 的中间结果,就省下「写出去 + 读回来」的 4n 纯浪费访存。这就是逐元素链融合的全部理论依据。

### 练习 4:实测验证 softmax 是否 memory-bound

对 [8192, 8192] 的 FP16 张量沿最后一维做 `F.softmax`。n = 8192 × 8192 = 67108864 ≈ 6.7×10⁷ 元素。

**(a) 估算访存量 Q**

数值稳定实现 `softmax(x) = exp(x − max(x)) / Σexp(x − max(x))`,朴素逐 pass 对 x 扫多遍:

```
pass1 求 max:        读 n
pass2 求 Σexp:        读 n
pass3 归一化并写出:   读 n + 写 n
合计 ≈ 4 读 + 1 写 = 5n 个元素 = 5 × n × 2 = 10n 字节（朴素多 pass 上界）
   = 10 × 6.7e7 ≈ 6.7e8 字节 ≈ 0.67 GB

理论下界（完美融合,仅读一遍写一遍）：
   Q_min = 2n × 2 = 4n 字节 ≈ 2.7e8 字节 ≈ 0.27 GB
```

实际 PyTorch softmax 介于两者之间(取决于是否被拆成多 kernel)。

**(b) 反推有效带宽并对比峰值**

把实测耗时 t(毫秒)代入(注:具体耗时需在 A100 上实跑得到,这里给反推公式与判读标准,**实测数值待核**):

```
有效带宽 BW_eff = Q / (t × 1e-3)   字节/秒,再 / 1e12 转 TB/s
带宽利用率 = BW_eff / 2.0   （A100 峰值 ~2 TB/s）
```

先验判断:softmax 每元素约 5 次浮点运算(减、exp、累加、除),算术强度

```
I ≈ 5n / Q ≈ 5n / 4n ≈ 1.25 FLOP/Byte（用下界 Q=4n)
1.25 ≪ 脊点 156  →  铁定 memory-bound
```

因此**预期有效带宽应逼近 2 TB/s(撞带宽屋顶),而有效算力只有峰值的零点几个百分点——这是物理上限,不是 bug。** 若实测带宽明显低于 2 TB/s(例如只有 30~50%),说明没跑到屋顶,问题出在工程因素(见 c)。

**(c) 用 profiler 定位多 kernel 反复读写 HBM**

```
用 torch.profiler 或 ncu 查看:这一次 softmax 是否被拆成了多个 kernel
（求 max 一个、求 sum 一个、归一化一个),每个 kernel 都把 x 从 HBM 读一遍。
```

若被拆成多 kernel,则实际 Q 接近 10n(而非下界 4n),有效带宽即使打满单 kernel 的读写,整体「有效吞吐 / 数据量」仍偏低——因为同一份 x 被反复从 HBM 拉取。**这正是 FlashAttention 中「online softmax 融合」要消除的浪费:把 max / sum / 归一化在一次扫描(或 tiling 内增量更新)里算完,x 只过一遍 HBM。**

**可运行参考实现(GPU + PyTorch;CPU 仅能验证逻辑,带宽数无意义):**

> 依赖:`torch`(需 CUDA 版)。需在装有 NVIDIA GPU 的机器上跑;无 GPU 时把 `device='cuda'` 改为 `'cpu'`、dtype 改 `float32` 只能验证代码正确性,反推出的「带宽」是 CPU 内存带宽,与本练习结论无关。

```python
import torch
import torch.nn.functional as F

def benchmark(fn, *args, warmup=20, iters=100):
    """实测 fn 平均耗时(毫秒),正确处理 GPU 异步(同第七节)。"""
    for _ in range(warmup):          # 排除 cuDNN autotune / 显存分配冷启动
        fn(*args)
    torch.cuda.synchronize()         # 等 warmup kernel 真正跑完
    start = torch.cuda.Event(enable_timing=True)
    end   = torch.cuda.Event(enable_timing=True)
    start.record()
    for _ in range(iters):
        fn(*args)
    end.record()
    torch.cuda.synchronize()         # 必须同步,否则读到空值
    return start.elapsed_time(end) / iters   # 毫秒/次

# ---- softmax: 怀疑 memory-bound ----
M = N = 8192
x = torch.randn(M, N, device='cuda', dtype=torch.float16)

t_ms = benchmark(lambda a: F.softmax(a, dim=-1), x)

n = M * N
# (a) 访存量:取两个口径,真实值落在区间内
Q_lower = 2 * n * 2          # 完美融合:读1遍+写1遍
Q_naive = 5 * n * 2          # 朴素多 pass:读4遍(max/exp/归一化读)+写1遍

bw_lower = Q_lower / (t_ms * 1e-3) / 1e12   # TB/s(按下界访存算,偏保守)
bw_naive = Q_naive / (t_ms * 1e-3) / 1e12   # TB/s(按多pass访存算,偏乐观)

print(f"[softmax] {t_ms:.3f} ms")
print(f"  有效带宽(Q=2n下界) {bw_lower:.2f} TB/s  -> 占峰值 {bw_lower/2*100:.0f}%")
print(f"  有效带宽(Q=5n上界) {bw_naive:.2f} TB/s  -> 占峰值 {bw_naive/2*100:.0f}%")
print(f"  算术强度 I ≈ {5*n/Q_lower:.2f} FLOP/Byte  << 脊点 156 -> memory-bound")

# (c) 看是否被拆成多个 kernel(反复读 HBM)
from torch.profiler import profile, ProfilerActivity
with profile(activities=[ProfilerActivity.CUDA]) as prof:
    F.softmax(x, dim=-1)
    torch.cuda.synchronize()
print(prof.key_averages().table(sort_by="cuda_time_total", row_limit=10))
# 若看到多个 *softmax* 相关 kernel,即 max/sum/归一化各一遍 -> 反复读写 HBM
```

**判读结论:softmax 算术强度 ≈ 1~1.25 FLOP/Byte,深度 memory-bound;有效带宽应逼近峰值带宽才算「跑到屋顶」。若远低于峰值,十之八九是被拆成多 kernel 反复读 HBM——用融合(online softmax)即可救回。** 具体耗时与带宽百分比需在目标 GPU 上实跑,数值「待核」。
