> 这章把你从"会把一层切到多卡"(张量并行)带到"会把上百层切到多机、并把切分带来的空闲压到可接受",让你能独立设计一套流水线并行的 stage 划分与调度,并算清它的时间开销(气泡)和显存开销(在途激活)。

## 1. 为什么需要流水线并行:张量并行撞墙的地方

第 03 章的张量并行(Tensor Parallelism, TP)解决的是"**单层放不下**":把一个 `[hidden, 4·hidden]` 的 MLP 权重按列/行切到多卡。但张量并行有一条硬约束我们当时就强调过——**它的通信量大且频繁,必须关在单机内的 NVLink 里**。一个 Transformer block 前向就要 2 次 all-reduce(MLP 一次、注意力一次),反向再来 2 次。这些 all-reduce 走的是激活张量,大小是 `batch × seq × hidden`,而且**在关键路径上**(算完才能继续往下)。机间网络(InfiniBand,带宽比 NVLink 低一个量级,「待核:具体倍数随代际变化」)根本喂不饱这种通信密度。

于是 TP 的并行度被卡死在单机卡数(典型 8)。可现在的模型不是"单层放不下",而是"**层数太多、总参数太大**":一个 70B 模型有 80 层,光 fp16 参数就 140 GB,加上第 01 章那本账里的梯度、fp32 主副本、Adam 一阶二阶动量(每参数约 16 字节),训练态要 ~1.1 TB,单卡 80 GB 的 H100「待核」装不下,8 卡 TP 也只有 640 GB,还是不够。

**流水线并行(Pipeline Parallelism, PP)解决的是另一个正交的问题:层数多。** 它不切单层,而是**沿层的维度把模型切成 p 段(stage),每段放一组卡**。stage 0 放第 0~9 层,stage 1 放第 10~19 层,以此类推。一个 token 的前向计算像在流水线上流动:先过 stage 0,把激活传给 stage 1,再过 stage 1……反向梯度反着流回来。

它的好处恰好补上 TP 的短板:**stage 之间只在边界传一次激活**(`batch × seq × hidden` 这么大),而且**点对点(point-to-point)单向发送**,不是 all-reduce。通信次数从"每层 4 次集合通信"降到"每个 stage 边界 1 收 1 发"。这种稀疏、单向的通信,**机间网络扛得住**。所以工程上的标准搭法是:**TP 关在机内吃 NVLink,PP 跨机吃 InfiniBand**(第 06 章 3D 并行会把这个映射讲透)。

这章的核心矛盾只有一个词:**气泡(bubble)**。把模型切成 p 段、像流水线一样跑,天然会有"前面 stage 已经开工、后面 stage 还在等上游"的空闲时间。气泡是 PP 独有的、TP 和 DP 都没有的开销。**整章都在回答:气泡从哪来、有多大、怎么压、压到什么程度就该停手。**

预备知识清单(都来自前三章):
- 一次迭代 = 前向 + 反向 + 优化器更新;**反向 FLOPs ≈ 2× 前向**(第 01 章),所以下文把反向时间记为前向的约 2 倍。
- 反向需要前向时的**激活(activation)**,所以激活必须留到对应反向算完才能释放——这是后面"在途激活显存"的根。
- 显存账本:每参数约 16 字节(混合精度 + Adam),激活显存 ∝ `batch × seq × hidden × layers`。

## 2. 朴素流水线:气泡从哪来

先看最朴素的做法,理解气泡的物理来源。

假设把模型切成 p 个 stage,设备 0 到 p-1。喂进去**一个完整 batch**,让它走完前向再走反向。把"一个 stage 做一次前向"记为 1 个时间单位 t_f,做一次反向记为 t_b(我们刚说过 t_b ≈ 2·t_f,但为了先把气泡讲清,**第一步令 t_f = t_b = 1**,后面再放开)。

时间线(p = 4,横轴是时间,每格 1 单位):

```
         t=0  1   2   3   4   5   6   7
stage0   F                       B
stage1       F                 B
stage2           F           B
stage3               F     B
```

- t=0:只有 stage0 在算前向,stage1/2/3 全空闲(它们在等 stage0 的激活)。
- t=1:激活传到 stage1,stage1 开工;stage0 已经没事干了(整个 batch 只有一份,它算完就闲着)。
- 前向要 p=4 步才"灌满"到最后一级,但灌满那一刻前面的 stage 早就空了。
- 反向同理:从 stage3 倒着往回算,算 stage3 时 stage0/1/2 又在等。

**任何时刻只有 1 个 stage 在干活,其余 p-1 个在发呆。** 设备利用率 ≈ 1/p。p=4 时利用率 25%,p=8 时 12.5%。**把模型切得越多份,反而越浪费——这就是朴素流水线的灾难。** 浪费掉的那 (p-1)/p 的时间,就是**气泡**:设备开着、电费在烧、但没产出。

气泡的物理根源是**数据依赖的串行性**:stage s 的前向必须等 stage s-1 的输出,反向必须等 stage s+1 的梯度。一份数据只能在流水线上占一个位置,必然让其余位置空转。

## 3. 用 micro-batch 把流水填起来:气泡率的精确推导

灾难的根源是"只喂一份数据"。解法很直接:**把一个大 batch 切成 m 个 micro-batch,像往传送带上连续放零件一样依次喂进去**。stage0 处理完 micro-batch 0 的前向,立刻处理 micro-batch 1,同时 micro-batch 0 流到 stage1……这样多个 micro-batch 在不同 stage 上**同时被处理**,把空闲填满。

这就是 **GPipe** 的核心思想(Huang et al., 2019):**先把所有 m 个 micro-batch 的前向全跑完,再把所有反向全跑完。**

### 3.1 画出时间线

p=4,m=4,先只画前向部分,数字是 micro-batch 编号:

```
         t=0  1   2   3   4   5   6
stage0   F0  F1  F2  F3
stage1       F0  F1  F2  F3
stage2           F0  F1  F2  F3
stage3               F0  F1  F2  F3
```

- 灌入阶段(warmup):t=0 时只有 stage0 干活,t=1 时 stage0/1 都干活……要 **p-1 步**才让所有 stage 都开工。这 p-1 步里,流水线没填满,有空闲。
- 稳定阶段(steady):t=3 时 4 个 stage 全在干活(stage0 做 F3、stage1 做 F2、stage2 做 F1、stage3 做 F0),**满负荷**。
- 排空阶段(drain):前向尾部,stage0 先没活,逐级排空。

前向全部做完后,再对称地做反向(从 stage3 往 stage0 倒灌)。前后两段的空闲拼起来,就是总气泡。

### 3.2 推导气泡率公式

设每个 micro-batch 在一个 stage 上前向耗时 t_f、反向耗时 t_b。**先按 t_f = t_b = 1 推**(下一节放开)。

**有用工作(useful work)**:每个设备要处理 m 个 micro-batch 的前向 + m 个的反向 = `m·t_f + m·t_b = 2m` 个单位。这是无论如何都省不掉的真实计算量。

**总时间(makespan)**:看关键路径。最长的一条是 micro-batch m-1(最后一个)走完整条来回:

```
关键路径推导(t_f = t_b = 1):
  · micro-batch 0 的前向走完 stage0:           占 1 步(t=0→1)
  · 它在流水线上往下走到 stage p-1 前向完成:    需要再 p-1 步,共 p 步
  · 但 stage0 要连续喂 m 个 micro-batch 的前向:  stage0 上前向占 m 步
  · 最后一个 micro-batch (m-1) 离开 stage0 后,还要 p-1 步才走到 stage p-1
  · 所以全部前向完成时刻 = m + (p-1)            （stage0 喂完 m 个 + 流水深度 p-1）
  · 反向完全对称,再花 m + (p-1)              （反向同样深度 p、喂 m 个）
  · 但中间衔接处不重叠,总时间:
      makespan = (m + p - 1)·t_f + (m + p - 1)·t_b
               = (m + p - 1)·(t_f + t_b)
      代入 t_f=t_b=1:  makespan = 2(m + p - 1)
```

**气泡时间 = 总时间 − 有用工作**:

```
  bubble = makespan − useful
         = (m + p - 1)·(t_f + t_b) − m·(t_f + t_b)
         = (p - 1)·(t_f + t_b)
```

漂亮的结论:**气泡的绝对量 = (p−1)·(t_f + t_b),只跟 stage 数 p 有关,跟 micro-batch 数 m 无关。** 物理直觉:气泡就是"灌满流水线"(warmup,p-1 步)和"排空流水线"(drain,p-1 步,前向反向各算一半摊到这里)那段填不满的时间,流水线深度 p 定了,这段就定了,喂多少 micro-batch 都改不了它的绝对大小。

**气泡率(bubble fraction)= 气泡 / 总时间**:

```
  bubble_fraction = bubble / makespan
                  = (p-1)·(t_f+t_b) / [(m+p-1)·(t_f+t_b)]
                  = (p - 1) / (m + p - 1)        ★ 核心公式
```

也常写成另一个等价形式,**气泡相对有用工作的比例**:

```
  bubble / useful = (p-1)·(t_f+t_b) / [m·(t_f+t_b)] = (p - 1) / m
```

这个 `(p-1)/m` 更直观:**想把气泡压到有用工作的 x 以下,就要 m ≥ (p-1)/x。**

我用事件级模拟验证过这两个公式,完全吻合(p=8, m=32 时实测气泡率 0.1795,公式 (8-1)/(32+8-1)=0.1795)。

### 3.3 关键工程结论:m 要远大于 p

把 `(p-1)/m` 翻译成工程语言:

| p | m | 气泡率 (p-1)/(m+p-1) | bubble/useful (p-1)/m |
|---|---|---|---|
| 4 | 4 | 42.9% | 75% |
| 4 | 16 | 15.8% | 18.8% |
| 8 | 8 | 46.7% | 87.5% |
| 8 | 32 | 17.9% | 21.9% |
| 8 | 64 | 10% | 11% |

**经验法则:m ≈ 4p 时气泡率落到 ~15%,m ≈ 8p 时落到 ~8%。** 工业界训练通常让 **m 取到 p 的 4~8 倍以上**。比如 p=16,就想要 m ≥ 64 甚至上百。

为什么不能把 m 取到无穷大?三个真实的天花板:

1. **global batch 是有上限的**(收敛性约束,第 02 章讲过大 batch 伤泛化)。`global_batch = micro_batch_size × m × 数据并行度`。m 涨,要么 micro-batch size 缩到很小(GPU 利用率掉,小矩阵喂不饱 Tensor Core),要么 global batch 撑爆收敛允许的上限。
2. **m 越大,在途的激活越多**(GPipe 的致命伤,见第 5 节),显存先爆。
3. **micro-batch 太小,单次前向的矩阵太小,算术强度掉**(回到第 00 章 Roofline:小 GEMM 可能从 compute-bound 滑向 memory-bound,有效算力下降)。

**所以气泡和显存、收敛、算子效率是一组互相拉扯的约束,不能只盯气泡率往下压。** 这是本章后面所有调度优化的出发点。

### 3.4 把 t_b ≈ 2·t_f 放回去

上面为了清晰令 t_f=t_b。真实情况反向约是前向 2 倍(第 01 章)。好消息:**气泡率公式形式不变**。因为 makespan 和 useful 里 (t_f+t_b) 是公因子,直接约掉:

```
  bubble_fraction = (p-1)·(t_f+t_b) / [(m+p-1)·(t_f+t_b)] = (p-1)/(m+p-1)
```

**气泡率与 t_f、t_b 的具体比值无关,只取决于 p 和 m。** 这让公式特别好用——你不需要知道前向反向的精确耗时比,光数 stage 数和 micro-batch 数就能估气泡。(注意:这是"每 stage 计算量均衡"的理想前提下成立;切分不均会另有 imbalance bubble,见第 6 节坑。)

## 4. GPipe vs 1F1B:同样的气泡,差一个数量级的显存

到这儿气泡率讲透了。但有个问题被藏起来了:**反向需要前向的激活**。GPipe"先全部前向、再全部反向",意味着 **stage0 算完 micro-batch 0 的前向后,它的激活要一直留着,直到所有前向做完、反向倒灌回 stage0**——中间隔了整整 m 个 micro-batch 的时间。

### 4.1 GPipe 的显存灾难

**GPipe 下,每个 stage 在反向开始前,要同时存住 m 个 micro-batch 的前向激活。** 因为它先把 m 个 micro-batch 的前向全跑了,一个激活都不能丢(每个都要留给对应反向)。

```
GPipe stage0 时间线(F=前向 B=反向,m=4):
  F0 F1 F2 F3 ...(等所有 stage 前向 + 反向倒灌)... B3 B2 B1 B0
  ↑__________________________________________↑
  这段时间里 F0~F3 的激活全部在显存里 → 峰值 = m 份激活
```

**峰值在途激活 ∝ m。** 而我们第 3 节刚说"m 要取到 4p~8p"才能压气泡——这两个需求**直接打架**:为了降气泡把 m 调大,激活显存就线性爆炸。GPipe 用 activation checkpointing(第 08 章)硬扛,但激活显存正比于 m 这个结构性问题没解决。

### 4.2 1F1B:谁先前向完,谁先反向

**PipeDream(Narayanan et al., 2019)提出的 1F1B(one-forward-one-backward)调度**换了个思路:**不要等所有前向做完,一个 micro-batch 的前向只要走到最后一个 stage,就立刻让它开始反向倒灌**,反向一路释放激活。每个设备进入稳定态后,严格地**做一次前向、紧接做一次反向**(1F1B 的名字由来),交错进行。

直觉:**激活的"诞生—消费"间隔被压到最短**。micro-batch 在 stage s 前向产生的激活,尽快(它走完后续 stage 并反向回来时)被消费掉、释放掉,而不是干等所有人。

1F1B 把每个设备的执行切成三段(这就是写调度器要实现的骨架):

- **warmup(预热)**:设备 i 先只做前向,做 **(p−1−i)** 个 micro-batch 的前向,把流水线灌起来,但还不做反向(因为反向梯度还没倒灌到它)。**越靠前的 stage(i 小),warmup 阶段要囤越多前向**——stage0 要囤 p-1 个。
- **steady(稳定)**:进入 1F1B 节奏,每做一个新 micro-batch 的前向,就做一个老 micro-batch 的反向,一前一后交替,直到所有前向喂完。这段是满负荷、零气泡的主体。
- **cooldown(冷却)**:前向喂完了,把还没做完的反向逐个清空(对称于 warmup)。

### 4.3 关键推导:1F1B 的气泡和 GPipe 一模一样,显存却从 O(m) 降到 O(p)

**气泡:1F1B 的 makespan 和 GPipe 相同,都是 (m+p−1)·(t_f+t_b)。** 我用事件模拟验证过(p=8,m=32 两者都是 78 个单位,气泡都是 14)。直觉:warmup 灌满 + cooldown 排空那段填不满的总时间,由流水线深度 p 决定,和"前向反向是分开做还是交错做"无关。**所以 1F1B 不省气泡——它的气泡率还是 (p-1)/(m+p-1)。**

**那 1F1B 赢在哪?赢在显存。** 关键看每个 stage 同时在途的激活份数:

```
1F1B 峰值在途激活推导(看 stage0,它囤得最多):
  · warmup 阶段 stage0 做 (p-1) 个前向,囤 (p-1) 份激活
  · 进入 steady,stage0 每做 1 个新前向(+1 份激活),
    紧接着第一个反向才倒灌回 stage0(−1 份激活)
  · 所以 stage0 峰值 ≈ (p-1) + 1 = p 份激活
  · 越靠后的 stage 囤得越少:stage i 峰值 ≈ p − i 份,最后一级只囤 1 份
```

**峰值在途激活 ∝ p,不再 ∝ m。** 对比:

| 调度 | 峰值在途激活(最重的 stage0) | 随 m 增大 |
|---|---|---|
| GPipe | m 份 | 线性爆炸 |
| 1F1B | p 份 | **与 m 无关** |

这是质的差别。我们想把 m 取到 4p~8p 来压气泡,GPipe 下激活就是 4p~8p 份;1F1B 下永远是 p 份。**1F1B 让你可以放心大胆地增大 m 压气泡,而不用担心激活显存跟着涨**——这正是 Megatron-LM、DeepSpeed 默认用 1F1B 而非 GPipe 的根本原因。

(注意:1F1B 各 stage 激活份数不同,stage0 最多 p、末级最少 1,平均约 (p+1)/2。这个"前重后轻"的不均衡是 1F1B 的固有特征,做显存规划时要按最重的 stage0 算。)

### 4.4 1F1B 的伪代码(三段式,可照着实现)

下面是单个 stage(设备 i,共 p 个 stage)的 1F1B 调度主循环。这是写流水线引擎的核心骨架,Megatron-LM 的 `forward_backward_pipelining_without_interleaving`「待核:函数名以源码为准」就是这个结构的工业实现。

```python
def pipeline_1f1b(stage_id, p, m, model_stage,
                  recv_act, send_act, recv_grad, send_grad):
    """
    单个 stage 的 1F1B 调度。
    stage_id: 当前设备编号 0..p-1
    p: 总 stage 数; m: micro-batch 数
    recv_act/send_act: 从上游收激活 / 向下游发激活 (P2P, 阻塞或异步)
    recv_grad/send_grad: 从下游收梯度 / 向上游发梯度
    is_first = (stage_id == 0); is_last = (stage_id == p-1)
    """
    is_first = (stage_id == 0)
    is_last  = (stage_id == p - 1)

    # warmup 阶段做多少个纯前向 = p-1-stage_id
    num_warmup = min(p - 1 - stage_id, m)
    num_steady = m - num_warmup           # steady 阶段做多少个 1F1B
    activations = []                       # 缓存待反向消费的激活(FIFO 队列)

    # ---------- warmup: 只前向,囤激活 ----------
    for _ in range(num_warmup):
        x = None if is_first else recv_act()         # 第一级从 dataloader 拿,其余从上游收
        act = model_stage.forward(x)                  # 前向,保留中间激活
        activations.append(act)
        if not is_last:
            send_act(act.output)                      # 把输出发给下游

    # ---------- steady: 严格 1F1B 交错 ----------
    for k in range(num_steady):
        # 1 个 Forward
        x = None if is_first else recv_act()
        act = model_stage.forward(x)
        activations.append(act)
        if not is_last:
            send_act(act.output)

        # 1 个 Backward(消费最老的那份激活,释放显存)
        grad_out = None if is_last else recv_grad()   # 末级用 loss 直接起反向
        act_old  = activations.pop(0)                  # 取最老激活
        grad_in  = model_stage.backward(act_old, grad_out)  # 反向 → 激活在此被释放
        if not is_first:
            send_grad(grad_in)                         # 梯度发回上游

    # ---------- cooldown: 清空剩余反向 ----------
    while activations:
        grad_out = None if is_last else recv_grad()
        act_old  = activations.pop(0)
        grad_in  = model_stage.backward(act_old, grad_out)
        if not is_first:
            send_grad(grad_in)

    # 所有 micro-batch 的梯度已在各层累加完成 → 此时才做一次 optimizer.step()
```

实现时四个最容易踩的点:

1. **P2P 通信和计算要重叠**:`recv_act`/`send_act` 用异步原语(NCCL 的 `isend`/`irecv` 或 PyTorch 的 `batch_isend_irecv`),不能傻等,否则气泡之外又叠一层通信延迟。
2. **梯度累加**:m 个 micro-batch 的反向各产生一份梯度,**累加到同一份参数梯度上**,**所有 micro-batch 反向都做完才 `optimizer.step()` 一次**(一个 global batch 一步)。
3. **死锁**:相邻 stage 的收发顺序若不一致(都先 recv 后 send)会环形死锁,要约定全局收发次序或用 `batch_isend_irecv` 一次性提交。
4. **首/末级特判**:stage0 没有上游(输入来自 dataloader),末级 stage 没有下游(反向从 loss 起步)。

## 5. interleaved 1F1B:每卡放多个非连续 stage,把气泡再砍一刀

1F1B 把显存治好了,但气泡率还是 `(p-1)/(m+p-1)`,在 p 很大(比如跨 32 机、p=32)时依然可观。**interleaved 1F1B(交错式,Megatron-LM 的 virtual pipeline)** 进一步压气泡。

### 5.1 机制:把每个 stage 切成 v 个"模型块"

常规 PP 里,p 个设备对应 p 个连续 stage(设备 i 拿连续的 L/p 层)。interleaved 的做法:**把模型切成 p·v 个更细的块(model chunk),每个设备拿 v 个不连续的块**。

举例 p=4、v=2,模型切成 8 块,层从前到后是块 0~7。映射:

```
设备0: 块0, 块4      设备1: 块1, 块5
设备2: 块2, 块6      设备3: 块3, 块7
（每个设备负责 2 个相隔 p=4 的块,所以叫 "virtual pipeline depth = v"）
```

一个 micro-batch 的前向路径变成:设备0(块0)→设备1(块1)→设备2(块2)→设备3(块3)→设备0(块4)→设备1(块5)→…… **数据在 4 个设备间绕了 v=2 圈。**

### 5.2 为什么气泡变小:推导

气泡的本质是"灌满 + 排空"那段流水线没填满的时间,绝对量 = (p−1)·(单块前向+反向时间)。interleaved 的关键:**把每个块切小到原来的 1/v(每块只有 L/(p·v) 层),单块的前向反向时间也变成 1/v**。

```
interleaved 气泡推导:
  · 流水线"段数"仍是 p 个设备 → warmup/drain 深度仍由 p 决定为 p-1 个"段步"
  · 但每个"段步"现在只处理 1/v 个 stage 的计算量,耗时 = (t_f+t_b)/v
  · 所以气泡绝对量 = (p-1) · (t_f+t_b)/v           （比非交错小 v 倍）
  · 有用工作不变(总计算量没变):useful = m·(t_f+t_b)
  · 气泡率 = (p-1)·(t_f+t_b)/v / makespan
          ≈ (p-1) / [v·(m+p-1)]  ≈ (1/v)·原气泡率   ★
```

**interleaved 1F1B 把气泡率压到非交错版的约 1/v。** 我验证过这个 1/v 因子。直觉版:气泡是"流水线还没站满人的开销",你把每个工位的活拆成 v 份小活,流水线"站满人"需要的总时长不变,但**没站满的那段(灌入+排空)处理的活变成 1/v,占总时间的比例就掉到 1/v**。

### 5.3 代价:通信量涨 v 倍

天下没有免费午餐。数据在设备间绕 v 圈,意味着 **stage 边界的激活/梯度传输次数变成 v 倍**。每个 micro-batch 现在要跨设备边界 `p·v` 次而非 p 次。

```
权衡:
  · 气泡 ↓ 到 1/v
  · stage 间 P2P 通信量 ↑ 到 v 倍
  · 实现复杂度 ↑(每设备要管 v 个块的前向反向交错调度,激活栈变 v 层)
```

所以 v 不是越大越好。**当机间带宽充裕(气泡是主要瓶颈)时,interleaved 划算;当通信已经接近瓶颈时,加 v 反而拖慢。** v 典型取 2~4「待核:最优值依集群拓扑而定」。Megatron-LM 用 `--num-layers-per-virtual-pipeline-stage` 控制 v。

(还有一类更新的 zero-bubble / PP 调度,通过把反向再拆成"算输入梯度"和"算权重梯度"两半、把后者塞进气泡里,把气泡进一步逼近 0,代价是调度更复杂、显存模型更精细。这是 2023 后的前沿方向,机制属于"把已知的空闲填得更满",此处只点到「待核:具体方案细节」。)

## 6. 设计权衡与常见坑

**坑 1:把 p 切太多,以为越多越快。** PP 度 p 一上去,气泡率 (p-1)/(m+p-1) 跟着涨,要靠加大 m 来抵。但 m 受 global batch 和激活显存约束。**结论:PP 度优先取"刚好能把模型放下"的最小值**,而不是越大越好。能用 TP+DP 解决就别盲目加 PP。

**坑 2:stage 切分不均衡,产生 imbalance bubble。** 前面所有气泡推导都假设"每个 stage 计算量相等"。现实里:① embedding 层、最后的 LM head/loss 计算量和中间 transformer block 差很多,容易让首/末 stage 偏重;② 如果按"层数均分"切,但某些层(如带超大 FFN 的层)更重,就不均。**最慢的 stage 决定整条流水线的节拍**,一个 stage 慢 20%,整体就慢 20%。实践要按**计算量(FLOPs)而非层数**均分,首尾 stage 适当少放几层 transformer block 给 embedding/head 让路。

**坑 3:1F1B 的显存是"前重后轻",按平均算会 OOM。** stage0 囤 p 份激活、末级囤 1 份。做显存规划必须**按最重的 stage0(p 份)算**,不能用平均 (p+1)/2。新手常在 stage0 上 OOM 而别的 stage 还很空。

**坑 4:把 PP 和 TP 的通信都丢到机间网络。** 再强调一次第 1 节:TP 通信(all-reduce、频繁、关键路径)必须在机内 NVLink;PP 通信(P2P、稀疏、单向)可以跨机。搞反了(TP 跨机)直接慢到不可用。

**坑 5:P2P 通信没和计算重叠,白白多一层延迟。** 流水线本来就靠"算着这个 micro-batch、传着上个的激活"来藏通信。如果 send/recv 写成阻塞的,通信延迟会叠在每个 stage 边界上,实测吞吐远低于理论。务必用异步 P2P + 双缓冲。

**坑 6:micro-batch size 太小,GPU 喂不饱。** 为压气泡把 m 调大、micro-batch size 调到 1~2,单次前向的矩阵太小,Tensor Core 利用率掉(回到第 00 章 Roofline,小 GEMM 算术强度低)。**气泡率好看了,但每步绝对耗时反而涨。** 要在"气泡率"和"算子效率"间找平衡点,别只优化一个指标。

**与其他技术的组合(承接后续章节):**
- **PP + 激活重计算(第 08 章)**:即便 1F1B 把激活降到 p 份,p 大时还是不小,常叠加 activation checkpointing 进一步省。两者正交。
- **PP + ZeRO/FSDP(第 05 章)**:PP 切层、ZeRO 切优化器状态/梯度/参数,可叠加。但要注意 ZeRO-3 的 all-gather 和 PP 的 P2P 抢带宽,组合时调度要小心。
- **PP + TP + DP = 3D 并行(第 06 章)**:三者正交组合,`global_batch = micro_batch_size × m × DP度`,这个等式是第 06 章并行度搜索的约束核心。

## 7. 动手练习

**练习 1(估算题,基础):** 你要训练一个模型,PP 度 p=16。① 若要把气泡率压到 10% 以下,m 至少要多大?② 用 `bubble/useful = (p-1)/m` 这个更严格的指标压到 10% 以下,m 又要多大?③ 若 global batch 上限是 2048、数据并行度 DP=8,反推每个 micro-batch 的 size 最大能是多少(用 `global_batch = micro_bs × m × DP`),m 取②的值。
> 提示:① 解 (p-1)/(m+p-1) ≤ 0.1。② 解 (p-1)/m ≤ 0.1。③ micro_bs = global_batch /(m × DP),看是否 ≥1,不够说明 m 取太大了,这就是"气泡 vs batch"的硬约束在咬人。

**练习 2(推导题,核心):** 推导 interleaved 1F1B(virtual pipeline 深度 v)的气泡率约为非交错版的 1/v。
> 提示:气泡绝对量 = (灌入+排空的段步数)×(每段步耗时)。段步数仍由 p 决定((p-1) 量级),但每段步只处理 1/v 个 stage 的计算量。写出 makespan 和 useful,作比。注意通信量会涨到 v 倍,说明这个 1/v 不是白拿的。

**练习 3(显存推导题):** 设 hidden=8192,seq=4096,micro-batch size=1,每个 transformer block 的激活(不重计算)约 `s·b·h·c` 字节,c 是和层内结构相关的常数「待核:c 的精确值见第 01/08 章,典型十几倍」。① 在 GPipe(m=64)下,stage0 峰值激活是多少份、约多少字节?② 在 1F1B(p=16,m=64)下呢?③ 两者差几倍?用这个差解释"为什么 1F1B 让你能放心增大 m"。
> 提示:GPipe 峰值 = m 份,1F1B(stage0)峰值 = p 份。把 c 当未知量保留,比值 m/p = 64/16 = 4。

**练习 4(编码题):** 把第 4.4 节的 1F1B 伪代码补成可跑的单机模拟:用一个简单的多层 MLP 切成 p=4 个 stage(可以全放在一张卡上,用 list 模拟 P2P 的收发队列),实现 warmup/steady/cooldown 三段,并**打印每个时刻每个 stage 在做什么(F_i / B_i / idle)**,画出时间线,数一数气泡格子数,验证它等于 (p-1)·(t_f+t_b)。再把它改成 GPipe(全前向再全反向),对比时间线,观察气泡是否相同、但中途缓存的激活份数从 p 变成 m。
> 提示:用一个全局时钟 + 每 stage 一个 micro-batch 队列。验证目标:① 两种调度 makespan 相同;② 1F1B 任意时刻 `len(activations)` 峰值 = p,GPipe 峰值 = m。这一题把本章两个核心结论(气泡相同、显存不同)都跑出来。

## 8. 源码 / 论文导读

- **GPipe**(Huang et al., 2019, *"GPipe: Efficient Training of Giant Neural Networks using Pipeline Parallelism"*):看第 3 节的 micro-batch + re-materialization(就是激活重计算)那段,以及它对气泡(论文叫 "bubble overhead")的分析——它给出的就是 `(p-1)/(m+p-1)` 这个式子(论文里 p 叫 K、m 叫 M)。
- **PipeDream**(Narayanan et al., 2019, *"PipeDream: Generalized Pipeline Parallelism for DNN Training"*)及后续 **PipeDream-2BW**:看 1F1B 调度和"weight stashing"(为保证前向反向用同版本权重而暂存权重)——这是 1F1B 异步变体里的关键正确性细节,同步版 1F1B(Megatron 用的)不需要 weight stashing,理解这个区别很重要。
- **Megatron-LM**(Narayanan et al., 2021, *"Efficient Large-Scale Language Model Training on GPU Clusters Using Megatron-LM"*):**interleaved 1F1B(virtual pipeline)就出自这篇**,看它对 interleaved 气泡 ≈ 1/v 的推导(论文 Section 关于 schedule 的部分「待核:具体节号」),以及 3D 并行映射。
- **开源代码**:读 Megatron-LM 仓库 `megatron/core/pipeline_parallel/schedules.py`「待核:路径随版本变化」,里面 `forward_backward_pipelining_without_interleaving`(1F1B)和 `forward_backward_pipelining_with_interleaving`(interleaved 1F1B)就是第 4、5 节伪代码的工业实现,重点看它的 warmup/steady/cooldown 计数和 `recv_forward`/`send_forward`/`send_backward` 的 P2P 编排、死锁规避。DeepSpeed 的 `PipelineEngine` 是另一份可对照的实现。

## 9. 小结与承上启下

这章把**层间并行**讲透了,一句话串起来:**流水线并行用切 stage 解决"层太多",代价是气泡;micro-batch 把流水填满、气泡率 = (p−1)/(m+p−1)、增大 m 摊薄它;但 GPipe 的激活显存 ∝ m 会跟着爆,1F1B 用"早反向早释放"把激活压到 ∝ p 而气泡不变,interleaved 1F1B 再用每卡多块把气泡砍到 ~1/v(代价是通信涨 v 倍)。**

你现在手里有了三把并行的刀:
- **第 02 章 数据并行**:解决"要更快",复制模型、同步梯度(ring all-reduce 带宽最优)。
- **第 03 章 张量并行**:解决"单层放不下",切层内、关在机内 NVLink。
- **第 04 章(本章)流水线并行**:解决"层太多",切层间、可跨机、当心气泡。

三者**正交**——这正是**第 06 章 3D 并行**的素材:把 TP×PP×DP 按集群拓扑正交组合,`global_batch = micro_bs × m × DP度` 把它们绑在一起。但在那之前,**第 05 章 ZeRO/FSDP** 会先回到数据并行这条线,讲它如何不复制全套状态、用通信换显存,拿到接近模型并行的显存效率——和本章的 1F1B 一样,又是一个"用通信/计算的腾挪换显存"的经典招式。**"省显存"和"压气泡/通信"这对永恒的拉扯,会贯穿整门课剩下的每一章。**
