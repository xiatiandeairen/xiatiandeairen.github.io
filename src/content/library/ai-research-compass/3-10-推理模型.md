> 这一章把你从"会用 PPO/GRPO 跑 RLHF"带到能搭建一条**可验证奖励的推理训练流水线**:理解为什么"让模型先想清楚再答"是独立于参数规模的第二条 scaling 轴,为什么数学/代码这类**可验证任务**是推理类 RL 真正跑通的关键解锁点,以及长推理链里的**信用分配**难题到底难在哪、有哪些工程上能落地的缓解手段。

## 一、动机:一次前向直出答案,为什么算不对

先把痛点摆清楚。一个标准的 decoder-only 语言模型,生成答案的方式是:给定问题 prompt,自回归地一个 token 一个 token 往外吐,直到答案结束。对"法国首都是哪"这种问题,这没问题——答案就在权重里,一次前向(forward pass)就能取出来。

但对"求 ∫₀^π x·sin(x) dx"或"实现一个 O(n log n) 的区间合并算法并通过这 12 个测试用例"这种问题,**一次前向直出答案的正确率会显著低于让模型先写出中间步骤再给答案**。这不是玄学,有一个可以从计算复杂度角度讲清楚的根本原因。

**第一性原理:单步前向的计算量是固定的,而问题的内在计算深度是可变的。**

一个 Transformer 在生成第 t 个 token 时,做的计算量大致正比于 `L × d²`(L 是层数,d 是隐藏维度),这是一个**常数**——不随问题难度变化。换句话说,无论你问的是"1+1"还是"证明费马小定理",模型为产出"下一个 token"所花的浮点运算量是一样的。

但很多问题的解,本质上需要**串行的多步计算**:第 k 步依赖第 k−1 步的结果。一个深度为 D 的串行计算,理论上需要 Ω(D) 量级的串行计算步骤才能完成。Transformer 的层数 L 是固定的,所以单次前向能表达的"串行计算深度"被 L 卡死了。理论上有结果表明:固定深度、多项式宽度的 Transformer(对应 TC⁰ 这类电路复杂度类)无法在单次前向中表达某些需要更深串行结构的问题(比如某些需要顺序状态更新的任务)——**待核** 具体的复杂度类归属(不同假设下结论略有差异),但直觉是稳的:**层数固定 = 串行深度固定**。

那怎么办?**把串行计算外化到 token 序列上**。如果模型先生成一串中间 token(推理步骤),那么生成第 100 个推理 token 时,它的注意力可以读到前 99 个 token——相当于把"串行计算的中间状态"写在了"草稿纸"(token 序列)上,后续步骤再读回来。**生成 N 个 token,等于把单步固定深度 L 的计算,串成了深度约 N×L 的计算。** 这就是为什么"让模型先写推理过程"能解锁单次前向解不了的问题:它用**序列长度换取了串行计算深度**。

这一节的结论值得加粗记住:**推理 token 不是给人看的解释,它首先是模型自己的可读写工作内存(scratchpad),把固定深度的前向网络变成了变深度的串行计算机。** 这就是后面 test-time compute(推理时算力)这条 scaling 轴的物理基础。

## 二、从 Chain-of-Thought 到 test-time scaling

### 2.1 CoT:用 prompt 撬动隐含能力

Chain-of-Thought(思维链,CoT)由 Wei 等人在 2022 年提出。做法极简单:在 prompt 里给几个"问题→分步推理→答案"的范例,或者直接加一句"Let's think step by step"(Kojima 等人 2022,零样本 CoT),模型就会在给出最终答案前先输出一串推理步骤,数学/常识推理的准确率大幅提升。

为什么 prompt 一句话就能撬动?因为预训练语料里本来就有海量"分步讲解"的文本(教科书、解题过程、代码注释),模型早已学到了"分步推理"这种生成模式,只是默认不触发。CoT prompt 的作用是把模型推到这个分布上去。

但 CoT 有天花板:它撬动的是**预训练里已有的推理模式**。如果某类推理在语料里稀少(比如要回溯、要自我检查、要换思路重来),few-shot 范例很难稳定诱导出来。这就引出了下一步——**用 RL 让模型自己学会生成更长、更有效的推理链**。

### 2.2 o1 路线:用 RL 训练"自发长推理"

OpenAI o1(2024 年 9 月发布)的核心转变:**不再靠 prompt 诱导推理,而是把"推理过程的好坏"变成奖励信号,用 RL 直接训练模型生成长推理链**。模型学会的不只是"分步",而是回溯("等等,前面错了")、自我验证("我代回去检查一下")、尝试多种思路。这些行为在 base model 里是低概率的,RL 把它们的概率显著拉高。

关键现象——**test-time scaling(推理时缩放)**:o1 展示了一条新的标度律。横轴是推理时花掉的 token 数(等价于推理算力),纵轴是准确率,**在相当大的范围内,让模型多想(生成更多推理 token),准确率持续上升**。这条曲线和我们熟悉的"训练 scaling"(更多参数、更多数据、更多训练算力 → 更低 loss)是**正交的两条轴**:

```
训练 scaling(传统):   准确率 ~ f(参数量, 数据量, 训练算力)     —— 钱花在训练阶段
推理 scaling(新轴):   准确率 ~ g(推理 token 数 / 推理算力)    —— 钱花在每次回答时
```

这件事的产品/经济含义很大:**你可以用一个不变的模型权重,通过在推理时投入更多算力,换取更高准确率**。难题可以多想,简单题少想,算力按需分配。这是 o1 / o3 这条产品线的底层逻辑。

> 注意一个常被混淆的点:这里的"推理时多想"指的是**生成更长的推理链**(单条链更长,或采样多条链),不是简单地把同一个答案重复生成。真正起作用的是模型在链里做了**有效的串行计算**(见第一节)和**有效的搜索/自我纠错**(见第六节)。

### 2.3 R1 路线:RL 可以"凭空"长出推理能力

DeepSeek-R1(2025 年 1 月)给出了一个更激进、也更有研究价值的证据。它的前身 **R1-Zero** 做了一件事:**在 base model 上直接做 RL,不经过任何监督微调(SFT)冷启动,奖励只用规则化的可验证信号(数学答案对错 + 格式)**,结果模型自发地涌现出越来越长的推理链、自我反思("aha moment")等行为,数学 benchmark 大幅提升。

这个结果之所以重要,是因为它把因果关系讲清楚了:**长推理能力可以由 RL + 可验证奖励直接诱导,不一定需要人类示范的推理过程数据**。R1(完整版)在 R1-Zero 基础上加了少量 SFT 冷启动数据来改善可读性和语言一致性,但核心推理能力的来源被归因于 RL 阶段。

R1 用的 RL 算法是 **GRPO(Group Relative Policy Optimization)**,这是前一章应该已经讲过的 PPO 变体——去掉 value network,用同一个 prompt 采样的一组(group)回答的奖励均值作为 baseline。这点对推理任务特别契合,我们在第五节会专门分析为什么。

## 三、奖励从哪来:ORM vs PRM vs RLVR

推理 RL 的成败,**九成在奖励信号的设计**。把推理链记作动作序列,把"答对题"记作目标,奖励有三种主要来源,逐个讲透。

### 3.1 ORM:结果奖励(Outcome Reward Model)

**只看最终答案对错**,中间推理过程不打分。

```
r(轨迹) = 1  若最终答案正确
        = 0  否则        （或 ±1 / 带格式分等变体）
```

如果"正确"是由一个**学习出来的奖励模型**判断(比如训练一个分类器判断答案对不对),这叫 ORM。它的训练数据是"(问题, 完整解, 标签对/错)"。

- **优点**:标注便宜——只需要给最终答案打对错,不需要逐步标注;适用面广,任何有"好/坏"概念的任务都能用。
- **致命缺点 —— 信用分配(credit assignment)极稀疏**。一条推理链可能有 2000 个 token、30 个推理步骤,最后只得到一个标量奖励 0 或 1。模型完全不知道**是哪一步错了**。这在 RL 里是经典的"稀疏延迟奖励"难题,直接导致样本效率低、训练慢、方差大(第四、五节定量分析)。

### 3.2 PRM:过程奖励(Process Reward Model)

**对推理链的每一步打分**,而不是只看最终结果。OpenAI 的 "Let's Verify Step by Step"(Lightman 等人 2023)是代表工作,配套放出了 PRM800K 步级标注数据集。

```
对推理链 (s₁, s₂, ..., s_T):
  PRM 给出每步分数 v₁, v₂, ..., v_T  ∈ [0,1]   （每步"是否正确/有进展"）
  轨迹分数可聚合,例如:
    score = ∏ₜ vₜ     （所有步都对才算对,任一步崩则崩 —— 偏严格）
    score = minₜ vₜ    （取最弱一步 —— 强调"链上不能有错步"）
    score = (1/T)Σₜ vₜ （平均,较平滑）
```

- **优点**:**信用分配粒度细到步**。模型能知道第 17 步出了问题,梯度可以精确地压低"产生错误第 17 步"的概率。这显著缓解了稀疏奖励问题,在搜索/重排序(reranking)场景尤其强——可以用 PRM 给采样出的多条链打分选最优。
- **致命缺点 —— 步级标注极贵**。要给每一步标"对/错/中性",PRM800K 这种数据集是大量人工(或强模型辅助)标出来的。而且**"什么叫一步推理正确"本身定义模糊**:一步在局部看正确,但可能把整条链引向死路;一步在局部看像错的,但可能是有效的探索回溯。PRM 还容易被 reward hacking——模型学会生成"看起来每步都对、PRM 打高分"的链,但答案是错的(见第七节)。

### 3.3 RLVR:可验证奖励(RL with Verifiable Rewards)

**这是推理类 RL 真正跑通的关键解锁点,必须讲透为什么。**

RLVR 的思路:对于**有客观标准答案或可程序化验证**的任务,直接用一个**确定性的验证器(verifier)**计算奖励,而不是训练任何奖励模型。

```
数学题:r = 1 若 模型抽取出的答案 == 标准答案(经数值/符号归一化后相等)
         r = 0 否则
代码题:r = 通过的测试用例数 / 总数   （或 全过=1,否则=0）
         （在隔离沙箱里真实编译+运行单元测试）
形式证明:r = 1 若 Lean/Coq 校验器接受该证明
```

为什么 RLVR 是"关键解锁点"?三个层面的理由,每个都要讲清:

**理由一:免训练奖励模型,奖励信号无偏且零漂移。**
ORM/PRM 都是学出来的近似器,它们本身有泛化误差,而且在 RL 训练中模型分布会偏移(distribution shift)——策略越训越强,生成的样本越来越超出 RM 训练分布,RM 给分越来越不准。这叫**奖励模型过时/漂移**。RLVR 的验证器是**确定性程序**,π 怎么变它都不变,奖励永远是"客观对错",**没有漂移、没有泛化误差**。

**理由二:天然防 reward hacking(奖励黑客)。** 这是最关键的一点。
Reward hacking 指**模型找到了让奖励变高、但并没有真正完成任务的捷径**。它的根源是:奖励是真实目标的**代理(proxy)**,代理和真实目标之间有缝隙,优化压力会把模型挤进这条缝。学出来的 RM 缝隙很多(对抗样本、风格偏好、长度偏好……),所以基于 RM 的 RLHF 极易 reward hacking。

而 RLVR 的奖励**几乎就是真实目标本身**:对数学题,"答案数值相等"就是我们要的;对代码题,"通过隐藏测试"就是我们要的。**代理 = 目标,缝隙被堵死**。模型想 hack 这个奖励,唯一办法就是真的把题做对——这恰好是我们想要的。用一个不严谨但好记的话说:**RLVR 把"让奖励变高"和"把任务做对"对齐成了同一件事**。

(诚实补充:RLVR 也不是绝对无懈可击。如果验证器有漏洞——比如测试用例覆盖不全、答案抽取的正则被钻空子、模型直接 print 出标准答案、调用了不该用的库——一样会被 hack。所以 RLVR 的"防 hacking"严格说是**"把攻击面从'语义缝隙'缩小到'验证器实现缺陷'"**,后者是工程可控、可不断加固的;前者是开放、几乎无法穷尽的。这个区别在第七节展开。)

**理由三:可规模化、可自动生成训练信号。**
有了确定性验证器,你可以让模型对一道题采样上百条链,自动标好每条对错,不需要人。这让"自我提升"(self-improvement)循环成为可能:采样→验证→对正确链做强化。这正是 R1-Zero 能不靠人类推理示范、纯 RL 长出推理能力的工程前提。

**三者对比一张表记牢:**

```
              信用分配粒度   标注成本    抗 reward hacking   适用范围
ORM(结果RM)   极粗(整条链)   低(只标答案)  弱                广
PRM(过程RM)   细(每一步)     极高(步级标注) 中(链可被骗)      较广
RLVR(验证器)   粗(整条链)*    无(免RM)      强(代理≈目标)     窄(需可验证)
```

`*` RLVR 默认是结果级(整条链一个奖励),所以它**继承了 ORM 的信用分配稀疏问题**——这一点很多人忽略。RLVR 解决的是"奖励可不可信",没有解决"奖励够不够稠密"。这两个问题正交。第六节我们会看到 test-time 搜索 + PRM 如何在 RLVR 框架内补上稠密性。

## 四、长推理链的信用分配:定量分析

这一节算账,把"为什么稀疏结果奖励难训"算到能写在白板上。

### 4.1 问题设定

把推理链建模成一个轨迹 τ = (a₁, a₂, ..., a_T),aₜ 是第 t 个动作(可以是 token,也可以粗化为"一步推理")。策略 π_θ 产生轨迹的概率是 ∏ₜ π_θ(aₜ | s_{<t})。只有终局有奖励 R(τ) ∈ {0,1}(RLVR/ORM 的典型情形),中间奖励全为 0。

策略梯度(REINFORCE)告诉我们:

```
∇_θ J(θ) = E_τ [ R(τ) · Σ_{t=1}^{T} ∇_θ log π_θ(aₜ | s_{<t}) ]
```

推导(逐行,这是 REINFORCE 的标准推导,务必能自己写出来):

```
J(θ) = E_{τ~π_θ}[R(τ)] = Σ_τ P_θ(τ) R(τ)

∇_θ J = Σ_τ ∇_θ P_θ(τ) · R(τ)

利用 log-derivative trick:  ∇_θ P_θ(τ) = P_θ(τ) · ∇_θ log P_θ(τ)

       = Σ_τ P_θ(τ) · ∇_θ log P_θ(τ) · R(τ)
       = E_τ [ R(τ) · ∇_θ log P_θ(τ) ]

又 log P_θ(τ) = Σₜ log π_θ(aₜ|s_{<t}) + const(与θ无关的环境项)

所以 ∇_θ log P_θ(τ) = Σₜ ∇_θ log π_θ(aₜ|s_{<t})

代回:  ∇_θ J = E_τ [ R(τ) · Σₜ ∇_θ log π_θ(aₜ|s_{<t}) ]    ∎
```

### 4.2 信用分配难题的数学本质

看这个梯度的结构:**R(τ) 是一个整条轨迹共享的标量,它乘在每一个 token 的 ∇log π 上**。这意味着:

**如果答案对了(R=1),链上所有 T 个 token 的概率都被无差别地往上推,包括那些其实是废话/错误/绕路的步骤;如果答案错了(R=0,或基线后为负),所有 token 被无差别压低,包括那些其实正确的关键步骤。**

这就是信用分配难题的数学根源:**结果奖励无法区分"哪个 token 该奖、哪个该罚"**。它把功劳和罪责平摊给了整条链。链越长(T 越大),"平摊"越严重——一个正确答案里混进的错误步骤被错误地强化,一个错误答案里的正确步骤被错误地惩罚。

### 4.3 方差随链长的增长

更要命的是方差。策略梯度估计的方差大致随轨迹长度线性增长。给一个直觉推导:

```
设每步的 score 向量 gₜ = ∇_θ log π_θ(aₜ|s_{<t}),梯度估计 ĝ = R(τ)·Σₜ gₜ

Var[ĝ] ∝ Var[ R·Σₜ gₜ ]

若粗略假设各步 score 近似零均值、弱相关、每步方差量级为 σ²:
   Var[Σₜ gₜ] ≈ Σₜ Var[gₜ] ≈ T·σ²      （独立近似下方差可加）

⇒ Var[ĝ] 随 T 线性增长(O(T))
```

(严格地说各步 score 并不独立,且 baseline、advantage 归一化会改变常数,但**方差随链长增长的定性结论是稳的**。)结合 4.2:**链越长,既越难分配信用,梯度估计又越嘈杂**。这正是长推理链 RL 训练不稳定的双重根源。

### 4.4 缓解手段一览(后面分别展开)

```
手段                        缓解的是      机制
baseline / advantage        方差          减去基线不改期望、降方差(第五节)
group-relative(GRPO)        方差+无价值网络 用组内均值当 baseline(第五节)
过程奖励 PRM / 步级 advantage  信用分配      把稠密信号注入中间步
test-time 搜索 + 验证        信用分配+质量   用搜索把"哪条/哪步好"显式化(第六节)
长度惩罚 / 控制              方差+hacking   抑制无意义变长(第七节)
```

## 五、训练侧:为什么 GRPO 适合可验证推理

前一章讲过 PPO 的 advantage：`A(s,a) = Q(s,a) − V(s)`，需要训一个 value network V 来当 baseline。在长推理 + 稀疏奖励场景,value network 很难训准——它要从一个 2000-token 链的中间状态预测"最终能不能答对",这本身是个极难的回归问题,训不准就引入偏差。

**GRPO 的关键替换:不要 value network,用"同一道题采样一组回答的奖励"现场算 baseline。**

```
对一道题 q,采样一组 G 条回答 {o₁, ..., o_G},各自得验证奖励 {r₁, ..., r_G}

组内归一化的 advantage(同一道题的所有 token 共用):
   Âᵢ = (rᵢ − mean({r₁..r_G})) / (std({r₁..r_G}) + ε)

GRPO 目标(PPO-clip 形式 + KL 正则到参考模型 π_ref):
   J = E_q E_{oᵢ~π_old} [ (1/G) Σᵢ (1/|oᵢ|) Σ_t min( ρ_{i,t}·Âᵢ, clip(ρ_{i,t},1−ε,1+ε)·Âᵢ ) ] − β·KL(π_θ ‖ π_ref)
   其中 ρ_{i,t} = π_θ(o_{i,t}|q,o_{i,<t}) / π_old(o_{i,t}|q,o_{i,<t})   （逐 token 的重要性比,clip 也逐 token 施加)
```

为什么这个组内 baseline 在数学上合理?因为**baseline 只要不依赖当前采样的动作,减去它就不改变梯度期望,只降方差**。证明:

```
要证: E_{a~π}[ b · ∇_θ log π_θ(a|s) ] = 0   （b 是与 a 无关的 baseline）

E_a[ b·∇log π(a|s) ] = b · Σ_a π(a|s)·∇log π(a|s)
                     = b · Σ_a ∇π(a|s)            （log-deriv trick 反用）
                     = b · ∇_θ Σ_a π(a|s)
                     = b · ∇_θ (1)                 （概率归一,对所有θ恒为1）
                     = b · 0 = 0                   ∎
```

所以从 R 里减去任意"与该样本动作无关"的 baseline,梯度无偏。GRPO 用**组内其他样本的奖励均值**当 baseline——它依赖的是同组**其他**样本,对当前样本而言可视作常数,无偏性近似成立(严格的无偏需要 leave-one-out,实践中直接用组均值,引入的偏差极小)。

GRPO 为什么和 RLVR 是绝配:

1. **验证奖励是稀疏二值的**,正好需要一个好 baseline 来降方差;组内均值天然就是"这道题大概多难"的估计(题简单则组内大多对,baseline 高,答对的增益小;题难则 baseline 低,偶尔答对的增益大),**自动实现了难度自适应的优势缩放**。
2. **省掉了难训的 value network**,在长链场景避免了"value 训不准引入偏差"的坑。
3. 一道题采样一组,**正好复用了 RLVR"对同一题采样多条链自动验证"的数据生产方式**,采样和训练共享同一批 rollout。

代价与坑(必须知道):
- GRPO 需要**每道题足够大的 group size G**(常见 8/16/64)才能让 baseline 估计稳,这放大了采样成本(rollout 是推理算力大头)。
- 若一组里**全对或全错**,std≈0、advantage 全为 0(或被 ε 主导),这道题对本步训练**没贡献梯度**——简单题(全对)和超难题(全错)都被自动跳过,这既是特性(聚焦"会一点点"的题)也是坑(课程难度没配好会浪费大量算力在零梯度样本上)。

## 六、推理时增强:把算力花在"答的时候"

前面是训练侧。这一节是 test-time(推理时)侧——**不改权重,纯靠推理时多花算力提升正确率**。这是 test-time scaling 的具体兑现方式。

### 6.1 Self-Consistency:采样多条链投票

Wang 等人 2022 提出。机制极简单:**对同一问题用带温度采样生成 N 条独立推理链,各自得到最终答案,然后对答案做多数投票(majority vote)。**

```python
def self_consistency(model, prompt, n=40, temperature=0.7):
    answers = []
    for _ in range(n):
        chain = model.generate(prompt, temperature=temperature, do_sample=True)
        ans = extract_final_answer(chain)   # 抽取 "答案是 X" 里的 X
        if ans is not None:
            answers.append(normalize(ans))   # 数值/符号归一化
    # 多数投票:出现次数最多的答案
    return Counter(answers).most_common(1)[0][0]
```

为什么有效?**直觉:通往正确答案的路径往往不止一条且彼此印证,而出错的方式五花八门、彼此分散**。如果模型对正确答案的"边际概率"高于任一错误答案,那么聚合多条链(对推理路径做边际化)会让正确答案以更高概率胜出。

定量(给一个干净的模型):设每条链独立、答对的概率为 p。若 p > 任一错误答案的单次概率,则随 N 增大,多数投票选中正确答案的概率按指数趋近 1。最简化的"对 vs 一个集中错误答案"二选一情形下,投票出错概率 ~ exp(−c·N·(p−0.5)²) 量级(Hoeffding/Chernoff 类型界,**待核** 精确常数)——**核心定性结论:投票正确率随采样数 N 提升,但有上限 = 单链正确率 p 决定的"众数是否就是正确答案"**。

**关键坑**:如果模型系统性地偏向某个**错误**答案(p_wrong > p_correct),投票会**放大错误**,越投越错。Self-consistency 只在"正确答案是众数"时有效,它不能创造模型本来没有的能力,只能降低随机噪声。

### 6.2 束搜索 / 树搜索 + 验证器

Self-consistency 是"独立采样 + 末端投票",不利用中间信息。更强的是**在生成过程中就用验证器/PRM 引导搜索**:

- **Best-of-N(BoN,重排序)**:采样 N 条完整链,用 ORM/PRM/验证器给每条打分,选最高分那条。比投票更强,因为用了"质量打分"而非"频数"。RLVR 有验证器时,BoN 在可验证任务上接近"只要 N 条里有一条对就能选出来"(前提是验证器能识别对的那条)。
- **过程级束搜索(step-level beam search)**:把推理按"步"展开成树。每步采样若干候选,用 **PRM 给每个候选步打分**,保留 top-k 步继续展开,剪掉低分分支。
- **树搜索 / MCTS 类**:把推理当成在树上找一条高分路径,用 PRM 当价值估计做 rollout 与回传。能更系统地探索-利用,但实现复杂、推理开销大。

```python
def step_beam_search(model, prm, prompt, beam=4, branch=4, max_steps=8):
    beams = [(prompt, 0.0)]  # (当前部分链, 累计PRM分)
    for _ in range(max_steps):
        cand = []
        for chain, score in beams:
            for _ in range(branch):
                step = model.generate_one_step(chain)   # 生成"一步"(到换行/分隔)
                s = prm.score_step(chain, step)          # PRM 给这一步打分 ∈[0,1]
                cand.append((chain + step, score + log(s + 1e-9)))
        beams = sorted(cand, key=lambda x: x[1], reverse=True)[:beam]  # 保留top-beam
        if all(is_finished(c) for c, _ in beams):
            break
    return max(beams, key=lambda x: x[1])[0]
```

**这里出现一个关键的概念闭环**:第三节说 RLVR 的痛点是信用分配稀疏(整条链一个奖励),第四节证明了这会导致高方差+难训。而 **PRM + 树搜索恰好在 test-time 把"稠密的步级信号"补回来**——PRM 在这里不是用来算训练梯度,而是用来在搜索时给每一步定向。**RLVR 保证"最终对错可信",PRM/搜索负责"在推理时把算力导向更可能对的路径",两者职责正交、可叠加**。这是当前推理系统的主流配方。

### 6.3 test-time compute 的 scaling 含义(把这件事拔高)

把 6.1/6.2 统一看:它们都是**用推理算力 C 换准确率 acc 的不同曲线**。

```
self-consistency:  acc(C) 随 N 提升,上限被"单链正确率 p 是否使正确答案成为众数"卡死
BoN + 验证器:      acc(C) 随 N 提升,上限被"验证器能否选出对的那条 + N条里是否真有对的"卡死
树搜索 + PRM:       acc(C) 随展开预算提升,上限被 PRM 质量 + 搜索算法卡死
更长单链(o1/R1):    acc(C) 随推理token数提升,上限被"模型自身串行计算+自纠错能力"卡死
```

有研究(如 test-time compute scaling 的系统分析,Snell 等人 2024 一类工作)指出一个有产品价值的结论:**在固定算力预算下,"用一个小模型 + 大量 test-time 搜索" 在某些难度区间能匹敌甚至超过 "一个大模型 + 单次前向"**——即推理算力和训练算力之间存在**可交换性(compute-optimal trade-off)**。这把第二节那个"两条正交的轴"落到了可量化的资源分配决策上:**给定总算力,你要把它花在把模型养大,还是花在让它每次多想?答案随任务难度而变。** 这就是 test-time compute 作为"第二条 scaling 轴"的完整含义。

## 七、设计权衡与常见坑

**1. RLVR 的"防 hacking"是有边界的——守住验证器。**
前面说 RLVR 把攻击面缩到"验证器实现缺陷"。实战中这些缺陷真实存在且常被钻:
- **答案泄漏**:测试数据混进了 prompt/few-shot,模型直接 copy。
- **答案抽取被钻**:用正则抽 "答案是 X",模型学会输出多个候选或特殊格式骗过抽取器。**对策**:严格的归一化 + 唯一答案约束 + 抽取失败记 0 分。
- **代码题作弊**:模型 `print` 出 hardcode 的期望输出、读测试文件、`exit(0)`、调用网络。**对策**:隔离沙箱、隐藏测试集、禁危险系统调用、资源/时间限制。
- **核心心法**:**验证器是你奖励信号的"信任根",它的每一个漏洞都是一个 reward hacking 入口。把验证器当对抗目标来加固。**

**2. 长度 hacking(最普遍的推理 RL 坑)。**
test-time scaling 说"多想更准",会诱发一个退化:**模型发现把链拉长能蹭到更高奖励(更多自我检查碰巧救回几道题,或 PRM 偏好长链),于是无意义地变长**——重复啰嗦、车轱辘话、假装检查。代价是推理成本爆炸、可读性差,且常伴随 4.3 的方差恶化。
- **对策**:在奖励里加**长度惩罚**(答对的前提下越短越好);限制 max tokens;监控"平均链长 vs 准确率",若链长猛涨而准确率持平,就是在 hack 长度。

**3. 多样性塌缩(entropy collapse)。**
RL 优化会让策略变尖(熵下降),采样几乎只出一条链。这直接**杀死 self-consistency / BoN 的收益**(N 条链全一样,投票/重排序退化为 N=1),也让探索停滞、训练早早卡住。
- **对策**:KL 正则(到 π_ref)别太弱、熵奖励(entropy bonus)、采样温度、控制学习率;监控生成熵和组内答案多样性。

**4. PRM 的过程奖励 hacking。**
PRM 给"看起来对的步"高分,模型可学会生成**步步看着对、结论却错**的链(局部最优、整体崩)。**对策**:PRM 与最终结果验证(RLVR)联合使用,别让 PRM 单独定奖励;用结果正确性来校准 PRM。

**5. value network 在长链上的偏差(为什么很多人弃 PPO 用 GRPO)。**
PPO 的 critic 要预测"中间状态→最终对错",长链 + 稀疏奖励下它系统性训不准,带来有偏 advantage。GRPO 用组内 baseline 绕开,但代价是 group size 带来的采样开销。**没有免费午餐:你要么承担 critic 的偏差,要么承担大 group 的方差/算力。**

**6. 别把"能做对"和"会推理"混为一谈。**
RLVR 只奖励最终对错,模型可能学到"对这类题的捷径模式"而非通用推理——在分布内涨分、分布外不迁移。评估要带**held-out 的新题型**,别只看训练同源 benchmark。

## 十、小结与承上启下

这一章把"让模型想清楚再答"拆成了三个互相咬合的层面:

1. **物理基础**:推理 token 是模型的可读写工作内存,用序列长度换串行计算深度——这解释了 CoT 为什么有效,以及 test-time compute 为什么是独立于参数规模的第二条 scaling 轴。
2. **训练侧**:用 RL(R1 的 GRPO)+ **可验证奖励 RLVR** 让模型自发长出长推理链。RLVR 是关键解锁点,因为它让"代理奖励 ≈ 真实目标",**天然防 reward hacking** 且免训 RM——但它继承了结果奖励的**信用分配稀疏**问题,这个问题被我们用 REINFORCE 梯度公式和方差分析定量讲透了。
3. **推理侧**:self-consistency / BoN / 树搜索把算力花在回答时,用 PRM 在 test-time 把稠密步级信号补回来,与 RLVR 的"结果可信"职责正交、可叠加。

承上:前几章的 PPO / GRPO / RLHF 是本章训练侧的直接前置——没有那套策略梯度+KL正则的机制,RLVR 落不了地。

启下:本章的推理模型默认任务是"单轮、有标准答案"。真实世界的复杂任务是**多轮、与环境交互、奖励更稀疏更难验证**的——这正是下一章 **Agent / 工具使用 / 多步决策** 要解决的:当"验证器"变成一个真实环境(浏览器、代码仓库、操作系统),当一条轨迹包含几十次工具调用,信用分配和可验证奖励会变成什么形态?本章对"稀疏奖励信用分配"和"可验证 vs 学习奖励"的分析,会是你进入 agent RL 的基本盘。
