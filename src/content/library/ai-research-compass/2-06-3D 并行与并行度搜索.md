> 这一章把你从"会单独用数据并行或张量并行"带到"能为一个给定模型 + 给定集群,亲手设计出 (DP, TP, PP) 的最优组合,并说清每个数字背后的显存、通信、气泡代价"。读完你应该能看懂 Megatron-LM 的启动参数表为什么那样填,也能理解 Alpa 为什么要把这件事变成一个搜索问题。

## 1. 为什么需要 3D 并行:三个互相独立的约束

训练一个大模型时,单卡放不下、训不快,但"放不下"其实是**三个不同的物理约束**叠在一起,而每个约束需要一种不同的并行手段去拆。新手最大的误区是把它们混为一谈,以为"并行就是把 batch 切开发给多卡"(那只是数据并行)。

先把三个约束讲清楚,这是后面一切推导的地基:

**约束 A:单层放不下。** 一个 Transformer 层里最大的两块权重是 attention 的 QKV/输出投影和 FFN 的两个矩阵。以 hidden size `h`、FFN 扩展倍数 4 为例,单层权重参数量约 `12·h²`(QKV 各 `h²`、attention 输出 `h²`,FFN 上投影 `4h²`、下投影 `4h²`,合计约 `4h² + 8h² = 12h²`)。当 `h = 12288`(GPT-3 175B 的规模)时,单层就是 `12 × 12288² ≈ 1.81×10⁹` 个参数,fp16 下约 3.6 GB,加上优化器状态会膨胀十几倍。**一张卡的显存装不下哪怕一层的全部状态**,就必须把单层的权重矩阵切开——这是**张量并行(Tensor Parallelism, TP)**要解决的。

**约束 B:层数太多。** 就算每层能放下,几十上百层叠起来,所有层的权重 + 激活 + 优化器状态仍然超出单卡。这时把**不同的层分给不同的卡**,卡 0 算第 1~k 层,卡 1 算第 k+1~2k 层……这是**流水线并行(Pipeline Parallelism, PP)**要解决的。注意它解决的是"深度方向"的放不下,和 TP 正交。

**约束 C:要训得更快。** 假设模型已经能放进若干卡组成的一份"模型副本"里,我们想用更多卡把同样的训练跑得更快——复制多份模型,每份吃不同的数据切片,梯度求平均。这是**数据并行(Data Parallelism, DP)**要解决的。它不解决"放不下",只解决"吞吐"。

**关键洞察(本章的核心):这三个约束彼此独立,所以三种并行可以正交组合。** "正交"在这里有精确含义:你可以独立地选择 TP 度 `t`、PP 度 `p`、DP 度 `d`,它们各自沿着一个独立的维度切分计算与显存,乘起来恰好用掉 `N = t · p · d` 张卡。把这 N 张卡想象成一个三维网格的格点,每个格点是一张 GPU,三个轴分别是 TP、PP、DP——这就是"3D 并行"这个名字的来历。

```
3D 并行 = 三个正交的切分维度

  约束           并行手段        切什么            通信特征
  ----------     ----------      --------------    ----------------
  单层放不下      TP (张量)       切单层的矩阵       每层 forward/backward
                                  (列切/行切)        都要 all-reduce,极重
  层数太多        PP (流水)       切层 (深度方向)     仅相邻 stage 间传激活,轻
  要更快          DP (数据)       切 batch (样本)     每步末尾同步梯度,中等
```

后面我们会反复用到一个量化结论:**TP 通信最重、PP 通信最轻、DP 居中**。这个排序直接决定了"哪种并行该放在带宽更高的链路上",是整个映射方法论的支点。

## 2. 张量并行 TP:切单层,代价是每层两次 all-reduce

### 2.1 机制:Megatron 的列切-行切配对

TP 的核心技巧来自 Megatron-LM:**对一对相邻的矩阵乘法,前一个按列切、后一个按行切,中间的非线性恰好能逐元素并行,只在最后做一次规约。** 我们以 FFN 为例(MLP block),它是 `Y = GeLU(X·A)·B`,其中 `X` 是 `[s, h]`(序列长 s、hidden h),`A` 是 `[h, 4h]`,`B` 是 `[4h, h]`。

把 `A` **按列切**成 `t` 份:`A = [A₁, A₂, ..., Aₜ]`,每份 `[h, 4h/t]`。于是

```
X·A = [X·A₁, X·A₂, ..., X·Aₜ]        ← 每个 GPU 算一块,无需通信
GeLU(X·Aᵢ) 逐元素,天然可并行           ← 关键:列切后 GeLU 不跨分片
```

注意这里为什么必须列切:GeLU 是逐元素非线性,`GeLU(X·A)` 不等于把 `X·A` 行切后分别做 GeLU 再拼(因为行切会把同一个输出元素拆到两张卡上,逐元素函数就算错了)。**列切保证了每个输出列完整地落在一张卡上,非线性可以独立施加。** 这是 Megatron 设计的精髓——选择切分方向去迁就非线性。

接着把 `B` **按行切**成 `t` 份(行数 `4h/t`),第 i 张卡持有 `Bᵢ`:

```
Y = GeLU(X·A)·B = Σᵢ GeLU(X·Aᵢ)·Bᵢ    ← 每卡算一个部分和
```

每张卡用自己那块激活乘自己那块 `Bᵢ`,得到一个**部分和**,最后所有卡做一次 **all-reduce** 把部分和加起来,得到完整的 `Y`。整个 FFN 前向只需 **1 次 all-reduce**。

attention 用同样的套路:把多头**按头切**(每张卡算 `num_heads/t` 个头,这本质也是列切 QKV 投影),输出投影按行切,前向再来 1 次 all-reduce。

反向传播时,梯度的流向决定了通信落在另一侧:前向 all-reduce 的地方,反向是恒等;前向恒等(切分广播)的地方,反向是 all-reduce。所以**一个 Transformer 层前向 2 次 all-reduce(attention + FFN),反向再 2 次,共 4 次**。

```python
# Megatron 风格 FFN 的张量并行(伪代码,体现通信位置)
class ColumnParallelLinear:
    # 权重按输出维(列)切,每张卡持有 A_i: [h, 4h/t]
    def forward(self, x):                  # x 已在所有 TP rank 上复制
        return x @ self.A_i                # 本地算,无通信;输出 [s, 4h/t]

class RowParallelLinear:
    # 权重按输入维(行)切,每张卡持有 B_i: [4h/t, h]
    def forward(self, x_i):                # x_i 是本卡那一片激活 [s, 4h/t]
        y_partial = x_i @ self.B_i         # 部分和 [s, h]
        return all_reduce(y_partial)       # ★唯一通信:跨 TP 组求和

def ffn_tp(x):
    h = ColumnParallelLinear()(x)          # 列切,无通信
    h = gelu(h)                            # 逐元素,天然并行 ← 列切的回报
    y = RowParallelLinear()(h)             # 行切 + all_reduce
    return y                               # 前向共 1 次 all-reduce
```

### 2.2 量化:TP 的通信量为什么"重"

设序列长 `s`、batch 内 micro-batch 的 token 数为 `b·s`(b 是 micro-batch size),hidden `h`,数据类型 fp16(2 字节)。一次 all-reduce 的通信量用 ring all-reduce 模型:对一个大小为 `M` 字节的张量,每张卡收发的数据量约 `2·M·(t-1)/t ≈ 2M`(当 t 大时)。

被规约的张量是激活 `Y`,大小 `M = b·s·h·2` 字节。一个 Transformer 层前向 2 次、反向 2 次 all-reduce,所以**每层每步每卡的 TP 通信字节数**约:

```
C_TP(每层每卡) ≈ 4 次 × 2 × (b·s·h·2)         字节
              = 16 · b·s·h                    字节   (fp16)
```

代入一个常见配置 `b=1, s=2048, h=12288`:

```
C_TP ≈ 16 × 1 × 2048 × 12288 ≈ 4.0×10⁸ 字节 ≈ 0.40 GB / 层 / 步 / 卡
```

一个 96 层模型若整层都做 TP,一步前后向就是 ~38 GB 的 all-reduce 流量,而且这流量**和计算紧密耦合、无法被掩盖**(下一层的输入依赖上一层 all-reduce 的结果)。这就是"TP 通信最重"的来源——它发生在最内层循环,每层都来,且在关键路径上。

**结论(必记):TP 通信量正比于激活大小 `b·s·h`,且每层发生、在关键路径上不可隐藏。所以 TP 必须跑在延迟最低、带宽最高的链路上——也就是机内 NVLink/NVSwitch。一旦 TP 跨机走 InfiniBand/以太网,训练会被通信彻底拖垮。** 这是为什么实践中几乎总有 `t ≤ 单机卡数`(典型 t=8)。

> 量纲自检:`b·s·h` 是激活张量的元素数,×2 是 fp16 字节,×2 是 ring all-reduce 的收发系数,×4 是每层 4 次 all-reduce。每一项都有物理含义,没有凑系数。

### 2.3 序列并行(Sequence Parallelism)补充

Megatron 后续工作发现:层内除了矩阵乘,还有 LayerNorm 和 dropout 这些"沿 hidden 不可切"的算子,它们在标准 TP 下被复制执行、激活也被复制存储,浪费显存。**序列并行**把这些算子沿**序列维 s** 切开,与 TP 配合后,把 all-reduce 拆成 all-gather + reduce-scatter 两半,**总通信量不变但激活显存进一步下降**。理解了 2.1 的列切-行切配对,序列并行只是把"在哪个维度切"的思路推广到了序列维。具体公式建议直接读论文(见导读),这里不展开。

## 3. 流水线并行 PP:切层,代价是流水气泡

### 3.1 朴素流水的问题与 GPipe 的 micro-batch

把 96 层切给 4 个 stage,每个 stage 24 层。最朴素的做法:stage 0 算完整个 batch 的前向,把激活传给 stage 1……这样任意时刻只有一个 stage 在干活,其余三个在等——**利用率只有 1/p**。流水线并行的全部工程都在和这个"等"作斗争。

GPipe 的关键想法:**把一个 batch 切成 m 个 micro-batch,让它们像流水线一样首尾相接地流过各 stage。** stage 0 处理完 micro-batch 1 就立刻传给 stage 1,自己接着处理 micro-batch 2。这样除了流水"灌入"和"排空"的头尾两段,中间各 stage 是满载的。

### 3.2 量化:气泡分数(Bubble Fraction)的推导

这是 PP 最重要的公式,必须能自己推。设 PP 度(stage 数)为 `p`,micro-batch 数为 `m`,每个 micro-batch 在单个 stage 上前向 + 反向的时间为单位 1。

```
推导:GPipe 全前向后全反向(F-then-B)的气泡分数

理想情况(无流水开销):总工作量 = m 个 micro-batch × p 个 stage
  但每个 stage 只需算自己那份,所以每个 stage 的纯计算时间 = m 个单位(前向)
                                                          + m 个单位(反向)
  = 2m 单位。这是"忙碌时间" t_busy = 2m。

灌入 + 排空(气泡)时间:
  前向方向上,最后一个 stage 要等前 (p-1) 个 stage 依次处理完
  第一个 micro-batch 才轮到它 → 灌入气泡 = (p-1) 个单位
  反向方向上,第一个 stage 要等后 (p-1) 个 stage 反向完
  → 排空气泡 = (p-1) 个单位
  气泡总时间 t_bubble = (p-1) + (p-1) = 2(p-1)

气泡分数 = 空闲 / (空闲 + 忙碌)
        = 2(p-1) / [2(p-1) + 2m]
        = (p-1) / (m + p - 1)
```

**结论(必记):气泡分数 = (p − 1) / (m + p − 1)。** 直觉检验:

- `m → ∞` 时气泡 → 0(micro-batch 越多,头尾占比越小);
- `p = 1` 时气泡 = 0(没有流水,无头尾);
- 实践经验法则:要把气泡压到 ~10% 以下,需要 **m ≳ 4p** 到 `8p`。比如 p=8,就要 m≳32 个 micro-batch。

代入 p=4, m=8:气泡 = 3/(8+3) ≈ 27%。这告诉你:**stage 越多,要喂越多 micro-batch 才能摊薄气泡,而 micro-batch 数受 global batch 上限约束**——这就是 PP 和后面 batch 关系的耦合点。

```python
def bubble_fraction(p, m):
    """GPipe (F-then-B) 流水气泡分数;p=stage 数, m=micro-batch 数"""
    return (p - 1) / (m + p - 1)

# 想让气泡 <= 0.1,反解所需 micro-batch 数:
# (p-1)/(m+p-1) <= 0.1  =>  m >= 9(p-1) + ... ≈ 9(p-1)
def min_microbatches_for_bubble(p, target=0.1):
    import math
    return math.ceil((p - 1) * (1 - target) / target)   # m >= (p-1)(1-tgt)/tgt
# min_microbatches_for_bubble(8) -> 63,即 p=8 想压到 10% 气泡需约 63 个 micro-batch
```

### 3.3 1F1B 调度:气泡不变,但峰值激活显存大降

GPipe 的 F-then-B 调度有个隐藏代价:stage 0 在做完所有 m 个 micro-batch 的前向后才开始反向,**它得把这 m 份前向激活全部存着**等反向用。峰值激活显存正比于 m,而我们又想 m 大来压气泡——矛盾。

PipeDream 提出的 **1F1B(one-forward-one-backward)**调度解决了这点:进入稳态后,每个 stage 做一次前向紧接着做一次反向,**反向一完成就能立刻释放那份激活**。结果是**气泡分数和 GPipe 完全相同**(还是 (p−1)/(m+p−1)),但**任意 stage 同时"在飞"的 micro-batch 数被限制在 ≤ p 份**(而非 m 份)。

```
峰值激活显存(以 micro-batch 份数计):
  GPipe (F-then-B):  ∝ m         ← m 大时显存爆
  1F1B (PipeDream):  ∝ p         ← 只跟 stage 数有关,与 m 解耦  ★
```

更精确地说,1F1B 下第 i 个 stage(从 0 计)峰值需要存约 `(p − i)` 份激活,**第一个 stage 最重(p 份)**。这就是为什么 1F1B 几乎是现代框架(Megatron、DeepSpeed)的默认 PP 调度——它让"m 大压气泡"和"激活显存可控"不再打架。

**进阶:interleaved 1F1B(虚拟流水)。** 把每个物理 stage 再细分成 v 个"虚拟 stage"交错排布,可把气泡进一步压到约 `(p−1)/(v·(m+p−1))`——代价是 stage 间传激活的次数变成 v 倍(通信量 ×v)。这是又一个典型的"用通信换气泡"的权衡。

### 3.4 PP 的通信为什么"轻"

PP 只在**相邻 stage 之间传一次激活**(前向)和一次激活梯度(反向),传的是 stage 边界处的张量,大小 `b·s·h·2` 字节,**点对点(P2P)发送,不是 all-reduce**:

```
C_PP(每 micro-batch 每 stage 边界) ≈ 2 × (b·s·h·2)  字节  (前向 + 反向)
```

对比 TP:TP 是**每层 4 次 all-reduce**(规约系数 ~2×、且每层都有),PP 是**每个 stage 边界 2 次 P2P**(无规约系数、且只在为数不多的边界处)。一个 96 层模型若 p=4,只有 3 个内部边界,通信发生次数比 TP 少了一两个数量级。**所以 PP 通信最轻,可以跨机走 InfiniBand,这正是把 PP 放到机间的依据。**

## 4. 数据并行 DP:复制模型,代价是梯度 all-reduce

DP 的机制最简单:把 global batch 切成 `d` 份,每个 DP rank 持有完整模型副本、吃一份数据、独立前向反向算出本地梯度,然后**所有副本对梯度做一次 all-reduce 取平均**,再各自更新(更新后参数仍然一致)。

```
C_DP(每步每卡) ≈ 2 × (P_local · 2)  字节        ← P_local 是本卡持有的参数量(fp16)
```

注意 DP 通信量正比于**参数量**而非激活量,且**每步只发生一次**(在反向结束、更新之前)。它能被**反向计算重叠掩盖**:梯度是反向时一层层算出来的,可以算完一层就异步 all-reduce 那一层的梯度,与后续层的反向计算并行(bucketing + overlap)。所以 DP 的有效通信代价常常远低于其名义字节数。

**DP 通信量介于 TP 和 PP 之间**:比 TP 轻(每步一次 vs 每层多次,且可掩盖),比 PP 重(规约 vs 点对点、全参数 vs 边界激活)。结合带宽,实践排序通常是:**TP 占机内 NVLink → DP 占机间(或机内剩余带宽) → PP 占机间最长链路**。但 DP 和 PP 谁更靠外要看具体拓扑,这点 §6 会细化。

### 4.1 ZeRO:把 DP 的显存冗余切掉

朴素 DP 的致命缺点:**每个副本都存全套优化器状态**。以 Adam + fp16 混合精度为例,单份模型状态的显存峰值约 `16·P` 字节(fp16 参数 2 + fp16 梯度 2 + fp32 参数副本 4 + fp32 momentum 4 + fp32 variance 4 = 16 字节/参数)。d 份副本就是 `16·P·d`,极度浪费。

**ZeRO(Zero Redundancy Optimizer)**沿 DP 维把这些状态切开:

```
ZeRO-1: 切优化器状态(fp32 副本+momentum+variance,12 字节/参数)
        单卡状态 ≈ 4·P + 12·P/d   字节
ZeRO-2: 再切梯度(+2 字节/参数)
        单卡状态 ≈ 2·P + 14·P/d   字节
ZeRO-3: 再切参数本身(+2 字节/参数)
        单卡状态 ≈ 16·P/d         字节   ← 显存随 d 线性下降
```

ZeRO-3 把"复制 d 份"变成"切成 d 片",显存随 d 几乎线性下降,代价是参数在 forward/backward 前要临时 all-gather 回来(额外通信)。**ZeRO 本质上是"用通信换显存"的 DP**,它和 PyTorch FSDP 是同一思想。理解了这点,你会发现"DP vs ZeRO vs TP"在显存维度上是连续谱:都是在不同粒度上切分状态、用不同量的通信赎回。

## 5. 把三者拼起来:global batch 的关系式

现在到了最容易出错、却最该刻进肌肉记忆的地方——**global batch size 的分解**。在 3D 并行 + 梯度累积下:

```
global_batch_size = micro_batch_size × gradient_accumulation_steps × dp_size

记 b = micro_batch_size,  ga = 梯度累积步数,  d = dp_size:
   B_global = b × ga × d
```

逐项解释,每一项的物理含义不能记混:

- **`b`(micro-batch size)**:一次前向喂进流水线的样本数。它决定单 micro-batch 的激活显存,**不参与跨 DP 的样本计数之外的事**。受单卡激活显存上限约束。
- **`ga`(梯度累积步数)**:在一个 DP rank 内,串行跑多少个 micro-batch、累加梯度后才做一次优化器更新与 DP all-reduce。**在 PP 语境下,`m`(micro-batch 数)= `ga`**——你往流水线里灌的 micro-batch 数,正是梯度累积步数。这是 PP 气泡公式里的 m 和 batch 公式的接口。
- **`d`(DP 度)**:几份模型副本并行吃数据。

**注意三件事(新手必踩):**

1. **TP 度 `t` 和 PP 度 `p` 不出现在 global batch 公式里。** 因为 TP 是把"同一份样本的同一层"切到多卡协同算,PP 是把"同一份样本的不同层"分到多卡接力算——它们都不增加"被独立处理的样本数",只是把单份样本的计算摊开。**只有 DP 在复制样本维度上做文章。** 把 t 或 p 误乘进 global batch 是经典错误,会让你以为 batch 比实际大 t·p 倍,学习率全调错。

2. **总卡数 `N = t · p · d`**,而 global batch 只含 d。所以给定 N,你"花"在 t 和 p 上的卡越多,留给 d 的就越少,**能达到的最大 global batch 就越小**(在固定 b、ga 上限下)。这是 3D 并行的根本 tension:并行度预算在"放得下"(t,p)和"批量大/训得快"(d)之间分配。

3. **梯度累积 ga 是"免费"放大 batch 的旋钮**(不增加卡、不增加通信频率,只增加单步时间),所以当 d 受限于卡数、又想要大 batch 时,加 ga 是第一手段;但 ga 也正是 PP 的 m,加 ga 会同时降低 PP 气泡——**这俩是同一个旋钮的两面**,理解了就能一石二鸟。

```python
def global_batch(b, ga, d):
    return b * ga * d          # 注意:不含 t、不含 p

def total_gpus(t, p, d):
    return t * p * d           # 而这里不含 b、不含 ga

# 反例(错误):global_batch = b*ga*d*t*p  ← 把切分误当复制,batch 虚高 t*p 倍
```

## 6. 映射方法论:为给定模型 + 集群选 (t, p, d)

把前面所有量化结论收拢成一套可执行的决策流程。设集群有 `G` 台机、每台 `K` 卡(机内 NVLink,机间 IB),总卡数 `N = G·K`。目标:训练参数量 `P`、`L` 层、hidden `h`、序列 `s` 的模型。

### 步骤 1:先定 TP(把它锁进单机)

由 §2.2,TP 通信在关键路径不可隐藏,**必须走 NVLink**,所以**第一原则:`t ≤ K`(典型 t = K = 8)**。怎么选 t 在 [1, K] 内的具体值?看"单层 + 单 micro-batch 的激活/权重能否放进 t 张卡":

```
单层权重(fp16,含优化器状态约 16 字节/参数,单层 12h² 参数):
   per_layer_state = 16 · 12h² / t   字节(TP 把层内切 t 份)
```

`t` 取到"单层状态能舒服放进单卡剩余显存"的最小值即可——**TP 度宁小勿大**,因为 TP 越大通信越重、且受 NVLink 域大小硬约束。常见做法:`h` 越大 t 取越大(如 h≥12288 取 t=8;h≤4096 往往 t=1 或 2,把 NVLink 留给别的更划算)。

### 步骤 2:再定 PP(用它装下"层数")

剩下要让整个模型(全部 L 层)放进 `t·p` 张卡。给定 t 后,每张卡承担 `L/p` 层的(已被 TP 切过的)状态:

```
每卡模型状态 ≈ 16 · 12h² · (L/p) / t   字节
要求:每卡模型状态 + 激活峰值(1F1B,∝ p 份)+ 框架开销 ≤ 单卡显存
```

由此**反解出满足显存约束的最小 p**。为什么取最小 p?因为 §3.2:**p 越大气泡越大、且需要越多 micro-batch(=ga)去摊薄**,而 ga 上限受 global batch 约束。所以 PP 也是**够用就好**——只把 p 加到"刚好放得下"。

气泡校验:选定 p 后,确认你能喂的 micro-batch 数 `m = ga` 满足 `m ≳ 4p~8p`(气泡 ≤ ~10~20%)。若 ga 不够大(被 global batch 卡住),要么减小 p,要么改用 interleaved 1F1B(§3.3)。

### 步骤 3:DP 吃掉剩下的卡

```
d = N / (t · p)
```

DP 度由总卡数除以 t·p 自然得出。然后校验 global batch:`B_global = b · ga · d` 是否落在算法可接受范围(太大伤收敛、太小浪费并行)。若 B_global 偏小且还想训得更快,可在 d 不变下加 ga(同时降 PP 气泡);若 B_global 已偏大,要么减 ga、要么把一部分 DP 卡的显存压力用 ZeRO 缓解从而允许减小 p/t、把卡腾给别处。

### 步骤 4:决定 t·p·d 三者在物理拓扑上的"贴图"

这是最容易被忽略却影响巨大的一步——**逻辑并行度 ≠ 物理摆放**。原则按通信强度由强到弱贴到链路由快到慢:

```
通信强度排序:  TP  >  DP  >  PP   (近似,DP 因可掩盖常退到 PP 之后)
链路速度排序:  机内 NVLink  >  机间 IB

映射规则:
  - TP 组的 t 张卡 → 必须同机(占满机内一个 NVLink 域)        [硬约束]
  - PP 的相邻 stage → 跨机也无妨(P2P 轻、可被计算掩盖)
  - DP 的 all-reduce → 尽量让同一 DP 组的卡有较好互联;
                       若 DP 可被反向计算充分掩盖,放机间亦可
```

**经典摆法(rank 排布顺序):**通常按 `[PP, DP, TP]` 由外到内编号,即 **TP 在最内层(连续 rank、同机),PP 在最外层(跨机)**,DP 居中。这样 TP 的 t 张卡天然落在同一台机的相邻 GPU 上(吃满 NVSwitch),PP 的 stage 边界落在机间(走 IB,反正轻),DP 组横跨机器(梯度 all-reduce 可被反向掩盖)。**记住这个顺序,它解释了为什么 Megatron 的 rank 计算公式长那样。**

### 一个完整示例:8 机 × 8 卡 = 64 卡,训 GPT-3 175B 级模型

模型:`P ≈ 175B`,`L = 96`,`h = 12288`。集群:`G=8`,`K=8`,`N=64`,机内 NVSwitch、机间 IB。

```
第1步 TP:h=12288 很大,单层状态重,且每机 8 卡 NVLink → 取 t = 8(占满单机)。

第2步 PP:给定 t=8,每卡承担 (96/p) 层、每层 16·12h²/8 字节状态。
   单层 12h² ≈ 1.81e9 参数;×16 字节 ≈ 29 GB/层(全套状态);/t(8) ≈ 3.6 GB/层。
   80GB 卡,留约 50GB 给模型状态 → 可放 ~13 层 → 需 96/13 ≈ 8 个 stage → 取 p = 8。
   (实际还要扣激活+开销,p=8 是稳妥选择,与公开的 175B 训练配置一致)

第3步 DP:d = N/(t·p) = 64/(8·8) = 1。
   → 64 卡恰好放下一份模型副本,DP=1(纯模型并行,无数据并行)。
   要训更快需更多卡(如 1024 卡 → d=16)。

第4步 batch:B_global = b · ga · d = b · ga · 1。
   取 b=1,要喂 m=ga ≳ 4p~8p = 32~64 个 micro-batch 压气泡
   → ga 取 ~64,B_global ≈ 64。气泡 = (8-1)/(64+8-1) ≈ 9.9% ✓

第5步 贴图:每台机的 8 卡 = 1 个 TP 组(NVLink);
   8 台机 = PP 的 8 个 stage(机间 IB,P2P);DP=1 无跨机 all-reduce。
```

```
64 卡 (8 机 × 8 卡) 的 3D 并行映射,t=8, p=8, d=1:

           TP 组(同机 8 卡,NVLink all-reduce,每层都通信)
           ┌───────────────────────────────────┐
 PP stage0 │ 机0: g0 g1 g2 g3 g4 g5 g6 g7 │ 层 1–12
           └──────────────┬────────────────────┘
                          │ P2P 传激活(机间 IB,轻)
 PP stage1 │ 机1: g8 ... g15                 │ 层 13–24
                          │
            ...           ↓ (流水接力)
                          │
 PP stage7 │ 机7: g56 ... g63                │ 层 85–96
           
 纵向(机间)= PP 流水,8 个 stage;横向(机内)= TP,8 路;DP = 1。
```

把 d 调到 16(扩到 1024 卡)只需在外面再复制 16 份这个 64 卡块,DP all-reduce 跨这 16 个块做、可被反向掩盖,global batch 随之 ×16。**这就是 3D 并行的可扩展性:t、p 锁住"放得下",d 线性换吞吐。**

## 7. 自动并行:把映射变成搜索问题(Alpa)

§6 的方法论是人肉启发式。模型/集群一复杂(异构带宽、非对称拓扑、MoE),手调就力不从心。**Alpa**(OSDI 2022)的洞察:**把并行策略分成两个层级,各自变成一个优化问题。**

- **Intra-operator parallelism(算子内,对应 TP/数据切分)**:对每个算子,枚举其可能的切分方式(沿哪个维切、切几份),每种切分有确定的计算 + 通信代价。Alpa 用**整数线性规划(ILP)**在一个 device mesh 内为整张计算图选切分,使总代价最小。
- **Inter-operator parallelism(算子间,对应 PP)**:把计算图切成若干 stage 映射到不同 mesh,用**动态规划(DP)**找最优的 stage 划分与 mesh 分配,使流水气泡 + 通信最小。

```
Alpa 的两层搜索(概念):
  外层 DP:  怎么把计算图切成 stage、每个 stage 配多大的 device mesh
            (这一层决定 PP 结构)
       └─ 内层 ILP: 给定一个 stage + 一块 mesh,
                    为该 stage 内每个算子选切分方式(TP/DP 混合)
                    目标 = min(计算 + 通信代价)
```

它的价值不在于"算法新",而在于把"3D 并行配置"从**专家经验**变成**可被编译器自动求解的代价模型 + 搜索**。理解 §1–§6 的每条权衡(显存 vs 通信 vs 气泡),你才能看懂 Alpa 的代价函数在优化什么——它优化的就是你手算时心里那本账。**自动并行不是取代理解,而是把你的理解形式化成目标函数。**

类似思路也出现在 GSPMD / Mesh-TensorFlow(用 sharding 标注让编译器推导并行)、以及各类 cost-model 驱动的并行搜索里。趋势很清楚:**人定 search space + cost model,机器做组合优化。**

## 8. 设计权衡与常见坑

把全章的权衡浓缩成一张"账本"——三个维度互相挤兑,没有免费午餐:

```
            显存          通信               气泡/利用率        放在哪
 TP↑      层内显存↓     每层 all-reduce↑↑   无气泡             必须机内 NVLink
 PP↑      纵向放更多层   stage 边界 P2P(轻)  气泡↑(需更多 m)   可跨机 IB
 DP↑      不省(ZeRO 才省) 每步 all-reduce(可掩盖) 无气泡         次外层
 ga↑      激活峰值↑(GPipe)无新增通信         气泡↓(=PP 的 m)    单机内串行
 ZeRO↑    显存↓(随 d)    额外 all-gather↑    无气泡             沿 DP 维
```

**新手必踩的坑:**

1. **把 t、p 乘进 global batch。** §5 反复强调:global batch = b·ga·d,不含 t、p。乘错了学习率全错。
2. **TP 跨机。** 把 t=16 配到 2 台 8 卡机上,TP 的 all-reduce 走 IB,训练慢到无法接受。**t 必须 ≤ 单机 NVLink 域大小。**
3. **PP 度大却不给够 micro-batch。** p=8 但 ga 只有 4,气泡 = 7/11 ≈ 64%,一大半算力在空转还以为"用了 8 路并行"。**先用 (p−1)/(m+p−1) 估气泡再拍板。**
4. **GPipe 调度 + 大 m 导致激活 OOM。** 忘了 F-then-B 峰值激活 ∝ m。**默认上 1F1B,峰值 ∝ p 与 m 解耦。**
5. **以为 DP 能省显存。** 朴素 DP 每卡存全套优化器状态,d 再大单卡显存也不降。**要省显存上 ZeRO/FSDP**(沿 DP 维切状态)。
6. **激活检查点(activation checkpointing)漏算进显存预算。** 大模型几乎必开 checkpointing(只存层边界、反向时重算),它把激活显存从 ∝ 层数降到 ∝ √层数 量级,但增加约 33% 的前向重算。做显存预算时要把它和 1F1B 的 p 份激活一起算。**(具体节省比例与实现相关,精确系数待核)**
7. **忽略 rank 贴图。** 逻辑上选对了 (t,p,d),但 rank 编号顺序错了导致 TP 组跨机——配置文件里 `--tensor-model-parallel-size` 等参数的解释顺序要和物理拓扑对上。**(各框架 rank 编号约定细节以其文档为准,待核)**

## 11. 小结与承上启下

这一章的主线只有一句:**三种并行对应三个正交的约束(单层放不下→TP、层数多→PP、要更快→DP),所以能组合成 N = t·p·d 的三维网格;而组合的艺术全在于按"通信强度匹配链路速度"做物理映射(TP 锁机内 NVLink、PP 走机间、DP 居中),并在显存 / 通信 / 气泡这三本账之间求平衡。** 你现在应该能拿起任意"X 机 × Y 卡 + Z 规模模型",照 §6 四步推出一组靠谱的 (t, p, d) 并讲清每个数字的代价。

几个要刻进肌肉记忆的结论:气泡 = (p−1)/(m+p−1);global batch = b·ga·d(不含 t、p);TP 每层 4 次 all-reduce、必须机内;1F1B 让激活显存 ∝ p 而非 m;ZeRO 用通信换显存、显存随 d 降。

**承上**:本章假设你已掌握单种并行的基本原理(数据并行的梯度同步、张量并行的矩阵切分),把它们升维成正交组合。**启下**:我们一直在说"通信可以被计算掩盖""all-reduce 走 NVLink",但**通信本身怎么高效实现**(ring vs tree all-reduce、NCCL 的拓扑感知、计算-通信 overlap 的具体调度)是下一章的主题;而 §4.1 提到的 ZeRO/FSDP 作为"显存优先"的并行范式,以及激活检查点、混合精度等显存优化技术,会在显存与重计算专章里系统展开。3D 并行是大模型分布式训练的"骨架",后面几章给它补上"血肉"。
