> 这一章把你从"会用 DDP、知道 ring all-reduce 带宽最优"带到"能为任意并行策略选对集合通信原语、能手算它在某拓扑上的完成时间、能判断一段计算图哪里该把通信藏进计算、哪里该把小算子融成一个 kernel,并能用 Triton 亲手写出一个融合 kernel"。通信和编译是贯穿全课的两条暗线——前面每一章(数据并行的 all-reduce、张量并行的 all-gather、FlashAttention 的 tiling)都在偷偷用它们。这一章把这两条线提到台面上系统讲透:**通信原语的代价模型 + 计算-通信重叠**,以及**为什么逐个小算子慢 + 算子融合 + Triton 怎么写**。读完你应该能回答:为什么张量并行宁可用两个 `M/N` 字节的 all-gather + reduce-scatter 而不是一个 `M` 字节的 all-reduce?为什么 `x.relu().add(b).mul(2)` 这串看似无害的逐元素操作会让你的 GPU 闲到只跑出峰值带宽的零头?

## 一、动机:两堵墙,两类税

第 00 章给过一个判定准则:一个 kernel 是 compute-bound 还是 memory-bound,看它的**算术强度**(每字节 HBM 流量摊到多少 FLOPs)落在 roofline 的哪一侧。第 02 章给过 **α-β 模型**(传 n 字节耗时 ≈ α + n/β)和 ring all-reduce 的带宽最优性证明。这一章要做的,是把这两套工具从"单卡访存"和"单个原语"推广到**整张计算图 + 整个多卡系统**,并指出现代 MLSys 性能优化里反复出现的两类税:

- **通信税(communication tax):** 多卡协作时,卡与卡之间必须交换数据(梯度、激活、参数分片)。这些字节走的是 NVLink / PCIe / InfiniBand,带宽比卡内 HBM 低一到两个数量级(见 §3 表)。如果让计算干等通信,扩展效率(N 卡吞吐 / N×单卡吞吐)就崩。**对策:计算-通信重叠——把通信延迟藏在计算背后。**
- **编译税(compilation tax):** 一个朴素的深度学习程序是一串独立算子,每个算子启动一个 GPU kernel,每个 kernel 各自从 HBM 把输入读进来、算完把输出写回去。逐元素算子的算术强度极低(每读一个 float 只做几次浮点运算),于是**整段计算撞在带宽墙上**,而且每次 kernel 启动还有固定开销(launch latency)。**对策:算子融合——把多个连续算子合成一个 kernel,中间结果留在寄存器/SRAM 不落 HBM。**

这两类税的共同根源,其实是第 00 章那句话的两个推论:**算力增速长期快于带宽增速,也快于互联带宽增速。** 所以"省 FLOPs"在 2025 年很少是瓶颈,"省字节流量"(无论是 HBM 流量还是网络流量)才是。本章的两个主题——重叠和融合——本质都是在**省字节、藏延迟**。

**给系统老兵的迁移提示:** 通信原语就是 MPI 的 `MPI_Allreduce` / `MPI_Allgather` 那一套,代价模型也沿用 HPC 的 α-β(Hockney)模型,你的直觉是现成的。融合则非常像编译器里的 **loop fusion / kernel fusion**:把两个遍历同一数据的循环合并,省一趟内存遍历。`torch.compile` 干的事情,概念上等价于一个面向张量程序的 JIT 编译器后端。你缺的只是"GPU kernel 启动开销有多大""寄存器/SRAM 能装下多少中间结果"这些硬件常数感。

## 二、集合通信原语:语义与代价模型

### 2.1 七个原语,先建立语义直觉

集合通信(collective communication)指**一组进程(这里是一组 GPU,每个 GPU 一个 rank,共 N 个 rank)共同参与的一次数据交换**。区别于点对点(send/recv),集合操作是"全员参与、语义良定义"的。设每个 rank 持有的数据张量按字节计为参考量,逐个建立直觉:

| 原语 | 语义(一句话) | 输入(每 rank) | 输出(每 rank) |
|---|---|---|---|
| **Broadcast** | root 的数据复制到所有 rank | root 有 M 字节 | 都得到那 M 字节 |
| **Scatter** | root 把数据切 N 份,第 k 份发给 rank k | root 有 M 字节 | 各得 M/N 字节(不同片) |
| **Gather** | 各 rank 的数据收集到 root | 各 M/N 字节 | root 得到 M 字节(拼起来) |
| **Reduce** | 各 rank 的数据按元素归约(sum/max/…)到 root | 各 M 字节 | root 得到 M 字节(归约结果) |
| **All-Reduce** | Reduce 后结果广播给所有 rank | 各 M 字节 | 都得到 M 字节归约结果 |
| **All-Gather** | Gather 后结果广播给所有 rank | 各 M/N 字节 | 都得到 M 字节(全拼) |
| **Reduce-Scatter** | Reduce 后结果切片分发(rank k 拿第 k 片) | 各 M 字节 | 各得 M/N 字节(归约后的片) |
| **All-to-All** | 每 rank 给每个 rank 发一片(矩阵转置) | 各 M 字节(切成 N 片) | 各 M 字节(收来的 N 片) |

两条要刻进肌肉记忆的恒等式(它们是后面所有代价分析的支点):

```
All-Reduce  =  Reduce-Scatter  +  All-Gather       ……(I)
All-Gather   ≈  全员各自 Broadcast 自己的分片         (直觉,非严格等价)
```

恒等式 (I) 是 ring all-reduce 算法的灵魂(第 02 章已用它证明带宽最优):先用 reduce-scatter 让每张卡拿到"全局和的一个分片",再用 all-gather 把这些分片拼回全量。**记住这条,你就能解释为什么 ZeRO 把 all-reduce 拆成 reduce-scatter + all-gather 来省显存——拆开后中间那一刻,每张卡只持有 1/N 的归约结果。**

### 2.2 统一代价模型:总线带宽时间 + 延迟项

第 02 章用 α-β 模型推了 ring all-reduce 的每卡传输量 ≈ 2(N-1)/N·M。这里把它推广成一个**对所有 ring 实现的原语都适用的统一公式**,这样你拿到任何原语都能现场估时。

定义符号:

- **N**:rank 数(GPU 数)。
- **M**:**all-reduce/all-gather 的"全量"字节数**,即每张卡最终要持有的归约/拼接结果的大小。注意各原语对"M 指什么"约定不同,下面逐个标清。
- **β**:每张卡能稳定吞吐的**单向链路带宽**(bytes/s),取节点内 NVLink 或节点间网卡的有效带宽。
- **α**:每步通信的固定延迟(链路建立 + kernel 启动 + 协议头)。

ring 类算法的统一结构是"流水线打 N-1 步,每步每张卡收/发一个 M/N 的分片"。把每张卡**实际通过链路搬运的总字节数**记为 `V_link`,完成时间近似:

```
T ≈ (步数) · α  +  V_link / β
```

各原语的 `V_link`(单向,即"出向"或"入向"字节数,二者在 ring 上对称)与步数,逐个推:

```
Reduce-Scatter:  N-1 步,每步发 M/N 字节  →  V_link = (N-1)/N · M ≈ M
All-Gather:      N-1 步,每步发 M/N 字节  →  V_link = (N-1)/N · M ≈ M
All-Reduce:      = RS + AG               →  V_link = 2(N-1)/N · M ≈ 2M
Broadcast(ring 流水线): V_link ≈ M(root 出 M,其余各转发≈M)
All-to-All:      每 rank 发 M(切 N 片各发 1 片,总和=M)→ V_link ≈ (N-1)/N · M ≈ M
```

**三个让无数人栽跟头的结论,务必背下来:**

1. **All-reduce 的链路流量是 all-gather / reduce-scatter 的两倍**(≈2M vs ≈M)。所以**凡是能拆成 RS+AG 且中间状态本来就需要的场景,拆开不亏;但若只需要其中一半,直接用半个原语省一半流量。**
2. **All-gather 和 reduce-scatter 的"全量 M"含义恰好相反。** All-gather 的输入是每卡 M/N、输出每卡 M;reduce-scatter 的输入是每卡 M、输出每卡 M/N。算时间时 M 都取"全量"(输出拼满 / 输入归约前的全量),`V_link ≈ M`。
3. **All-to-all 的 `V_link ≈ M` 与卡数关系不大(带宽视角),但它对"对分带宽"(bisection bandwidth)极度敏感。** 每张卡同时向所有其他卡发数据,瞬时打满交换网络的对剖面。这是 MoE(专家并行)的核心通信模式,也是为什么 MoE 对网络拓扑挑剔(见 §3.3)。

**一个立刻能用的算时间例子。** 张量并行(第 03 章 Megatron)的一次前向里,行并行 linear 的输出需要一次 all-reduce 把 N 张卡的部分和加起来。设隐藏维 d=8192、序列长 s=2048、batch b=1、fp16(2 字节),则要 all-reduce 的激活 M = b·s·d·2 = 1·2048·8192·2 ≈ 33.5 MB。在 N=8、NVLink 单向有效带宽取 β≈200 GB/s(待核:不同代 NVLink 实测有效带宽差异较大,A100 NVLink3 单向理论 300 GB/s,实测有效常在 200~250 GB/s 区间)下:

```
V_link = 2(N-1)/N · M = 2·(7/8)·33.5MB ≈ 58.6 MB
T ≈ V_link / β = 58.6e6 / 200e9 s ≈ 293 μs   (忽略 α)
```

而这一层的矩阵乘 FLOPs ≈ 2·b·s·d·(4d) ≈ 2·1·2048·8192·32768 ≈ 1.1 TFLOP,A100 fp16 算力取 ~250 TFLOP/s(实测),约 4.4 ms。(这里的 FLOPs 取整个 FFN/线性块 d↔4d 的量级;它和上面被 all-reduce 的 d 维激活不是同一张量的两端——M 的维度是行并行 linear 的 d 维输出,别和 FLOP 里的 4d 隐藏维混为一谈。)**通信 293 μs vs 计算 4.4 ms——所以张量并行在节点内(NVLink)是划算的,通信能被计算盖住;一旦跨节点走 IB(β 掉到 ~25 GB/s),同样的 all-reduce 要 ~2.3 ms,直逼计算时间,扩展效率就崩。这就是"张量并行只在节点内做"的量化依据。**

### 2.3 为什么数据并行用 all-reduce,而 ZeRO / 张量并行偏爱 RS / AG

把恒等式 (I) 和代价模型合起来,就能解释各并行策略的原语选择,这是把前几章串起来的关键:

- **数据并行(DDP):** 每卡有完整梯度副本 M,要的结果是"全局平均梯度"且每卡都要完整一份 → 正是 all-reduce 的语义,`V_link≈2M`。
- **ZeRO-2/3:** 优化器状态/参数分片,每卡只负责 1/N。梯度归约后每卡只需要自己那 1/N 片去更新 → 用 **reduce-scatter**(`V_link≈M`,省一半);更新完参数再 **all-gather** 回完整参数做下一次前向(`V_link≈M`)。一前一后两个半原语,总流量和 all-reduce 一个量级,但**峰值显存**砍到 1/N(因为任一时刻只持有分片)。这就是"通信量不变、显存大降"的 ZeRO 卖点的通信侧解释。
- **张量并行(列并行→行并行):** 列并行 linear 各卡算出激活的不同列分片,行并行 linear 之后要把部分和加起来给所有卡 → all-reduce(前向一次、反向一次)。
- **序列并行(sequence parallelism):** 把张量并行里那次 all-reduce 进一步拆成 reduce-scatter(前向)+ all-gather(反向),配合在序列维切分 LayerNorm/Dropout 的激活,**省激活显存**——又一次 (I) 的应用。
- **专家并行(MoE):** token 要被路由到分散在不同卡上的专家 → **all-to-all**(发过去)+ **all-to-all**(收回来)。

## 三、硬件拓扑:NCCL、NVLink/NVSwitch、InfiniBand

代价模型里的 β 和 α 不是凭空来的,由物理互联决定。不理解拓扑,你算出来的"理论通信时间"会和实测差几倍。

### 3.1 带宽的层级(把数字感刻进脑子)

GPU 之间的数据通路有清晰的层级,带宽逐级跌落(以下为**量级感**,具体型号请核对官方规格,标「待核」处尤甚):

```
卡内 HBM 带宽:        ~1.5~3.3 TB/s   (A100 ~2 TB/s, H100 ~3.3 TB/s)  ← 第 00 章的主战场
节点内 NVLink(卡间): ~300~450 GB/s 单向 (A100 NVLink3 ~300, H100 NVLink4 ~450)「待核」
节点内 PCIe(无NVLink):~32~64 GB/s    (PCIe 4.0 x16 ~32 GB/s 双向)
节点间 InfiniBand:    ~25~50 GB/s 单向 (HDR 200Gb/s≈25GB/s, NDR 400Gb/s≈50GB/s)「待核」
```

**两个数量级的鸿沟在这里:HBM(TB/s)→ NVLink(百 GB/s)→ IB(几十 GB/s)。** 每跨一级,通信代价模型里的 β 掉一个量级,通信时间涨一个量级。所有并行策略的设计,本质都是**把高频、大流量的通信留在高带宽的层级**(张量并行留节点内 NVLink),把低频通信放到低带宽层级(数据并行的梯度 all-reduce 可以跨节点,因为每步只做一次)。

### 3.2 NVLink / NVSwitch:节点内的全连接

- **NVLink** 是 NVIDIA 的点对点 GPU 互联,带宽远高于 PCIe。但裸 NVLink 是点对点的——8 张卡两两直连需要的链路数是 `C(8,2)=28`,布线和端口都不现实。
- **NVSwitch** 是一颗交换芯片,把节点内所有 GPU 接到一个 crossbar 上,实现**任意两卡全带宽、无阻塞的全连接(all-to-all 友好)**。DGX/HGX 这类机器内 8 卡通过 NVSwitch 互联,任意拓扑(ring、tree、all-to-all)都能跑满。**这就是为什么 NCCL 在 NVSwitch 机器上能对小规模灵活选 tree(延迟优)、对大规模选 ring(带宽优)——硬件不限制逻辑拓扑。**

### 3.3 InfiniBand 与对分带宽:跨节点的瓶颈

跨节点走 InfiniBand(或 RoCE)。关键指标除了单链路带宽,还有**对分带宽(bisection bandwidth)**:把集群从中间切成两半,两半之间的总带宽。

- **Ring all-reduce 对对分带宽不敏感**:它只在相邻 rank 间通信,任意时刻跨"中线"的流量是常数。这是 ring 在大规模 fat-tree 网络上仍高效的原因之一。
- **All-to-all(MoE)对对分带宽极度敏感**:每张卡同时往所有卡发,瞬间打满对剖面。一个 over-subscribed(对分带宽不足)的网络会让 MoE 的 all-to-all 成为绝对瓶颈。**这是 MoE 部署对网络拓扑挑剔、常需要 rail-optimized 拓扑的物理原因。**

### 3.4 NCCL 在干什么

**NCCL(NVIDIA Collective Communications Library)** 是上述所有原语在 NVIDIA 硬件上的高性能实现,PyTorch 分布式的默认后端。它替你做了三件你不想自己做的事:

1. **拓扑探测与算法选择**:开机探测 NVLink/NVSwitch/PCIe/IB 拓扑,为每个原语 + 每个消息大小自动选 ring 还是 tree(或 NVLS——NVSwitch 上的硬件加速归约「待核」),并把大张量切成多个 **chunk** 做流水线(pipelining)以重叠"传输"和"归约计算"。
2. **GPU-driven 通信**:整个集合操作作为**一个 CUDA kernel** 在 GPU 上跑,直接读写 HBM、驱动 NVLink/网卡 DMA,不需要 CPU 在每步插手。这意味着 NCCL 集合调用是**异步**入队到某个 CUDA stream 的(关键!这是 §4 重叠的基础)。
3. **GPUDirect RDMA**:让网卡直接 DMA 读写 GPU 显存,数据 GPU→网卡→网卡→GPU,**绕过 CPU 和系统内存**,省掉两次拷贝。

**一个能救命的实践常数:** NCCL 集合操作既然是一个占用 SM 的 kernel,它会**和你的计算 kernel 争抢 SM**。重叠通信和计算时,通信 kernel 抢走的 SM 越多,计算就越慢——所以"完美重叠"在实践中拿不到,通常能藏住 60%~90% 的通信(待核:具体比例强依赖模型、卡、NCCL 版本与 `NCCL_*` 环境变量调参)。

## 四、计算-通信重叠:把通信延迟藏进计算背后

### 4.1 第一性原理:重叠为什么可能

重叠成立的物理前提只有一句话:**通信走的硬件单元(NVLink/网卡 DMA 引擎)和计算走的硬件单元(SM 上的 CUDA/Tensor Core)是物理上独立的,可以同时工作。** 就像 CPU 的内存预取和 ALU 计算能并行一样。于是只要把通信发成**异步**的(发出去不等它完成,立刻回来接着算),理想情况下:

```
不重叠:  T = T_compute + T_comm           (串行,二者相加)
完美重叠: T = max(T_compute, T_comm)        (并行,取大者)
```

**收益上界:** 当 T_comm ≤ T_compute 时,通信被完全吸收,通信时间从总时间里"消失"。这正是 §2.2 那个例子的意义——只要把通信压到比计算短,扩展效率就能逼近 1。**所以系统优化的目标常常不是"消灭通信",而是"让通信短于计算从而被藏住"。**

### 4.2 机制:CUDA stream 与 event

GPU 上一切异步行为的载体是 **CUDA stream**:一个 stream 是一条**有序的操作队列**,同一 stream 内的 kernel 严格按入队顺序执行,**不同 stream 之间默认无序、可并发**。CPU 把 kernel "launch" 到 stream 上其实只是**入队**(非阻塞),GPU 异步地从队列取任务执行。

- **同一 stream**:计算 kernel 和通信 kernel 会串行(谁先入队谁先跑),**无法重叠**。
- **不同 stream**:把通信 kernel 放到一条独立 stream,计算 kernel 放默认 stream,GPU 就能并发执行二者——**这才是重叠的实现方式**。
- **CUDA event**:用来在 stream 之间表达依赖。`event.record(streamA)` 在 A 上插一个时间戳标记,`streamB.wait_event(event)` 让 B 阻塞到 A 跑到那个标记为止。重叠里用它保证"计算用到的通信结果确实已经 ready"。

```python
# 计算-通信重叠的最小骨架(PyTorch 语义,示意,未在本机执行)
import torch

comm_stream = torch.cuda.Stream()      # 专门跑通信的独立 stream
default_stream = torch.cuda.current_stream()

# 1) 在独立 stream 上异步发起通信(NCCL 调用本身就是异步入队)
with torch.cuda.stream(comm_stream):
    handle = torch.distributed.all_reduce(grad_bucket, async_op=True)
    # 注意:async_op=True 立刻返回 handle,通信 kernel 已入队 comm_stream,但未必跑完

# 2) 与此同时,default_stream 上继续做不依赖该通信结果的计算
other_compute(...)   # 例如反向传播的更早层,与这桶梯度的归约并发

# 3) 在真正要用归约结果之前,等通信完成
handle.wait()                          # 等 NCCL kernel 完成
# 或用 event 让某个 stream 等另一个 stream:
# ev = torch.cuda.Event(); ev.record(comm_stream)
# default_stream.wait_event(ev)
optimizer_step_using(grad_bucket)
```

**新手第一坑:在默认 stream 上 `all_reduce(async_op=False)`,然后以为加个 `Stream()` 就重叠了。** 不行——重叠要求(a)通信在独立 stream、(b)与之并发的计算不依赖通信结果、(c)用 event/handle 正确建立 happens-before。三者缺一,要么不重叠,要么读到没归约完的脏数据。

### 4.3 经典模式一:DDP 的反向-通信重叠(bucketing)

这是工业界最成熟的重叠案例,第 02 章已点过,这里讲透机制。关键观察:**反向传播是从最后一层往前一层层算的,最后一层的梯度最先就绪。** 如果等所有层梯度都算完再一次性 all-reduce,通信全暴露在反向之后(串行)。聪明做法:

**梯度桶化(gradient bucketing)+ 边算边通信。** 把参数按反向就绪的逆序分到若干个**桶(bucket)**(每桶约 25 MB,PyTorch 默认「待核」)。反向每填满一个桶,立刻对这个桶异步 `all_reduce`,同时继续算更靠前层的梯度。时间线:

```
反向时间轴 ───────────────────────────────────────────►
计算: [grad层N][grad层N-1][grad层N-2]...[grad层1]
通信:        └AR桶1┘    └AR桶2┘   ...        └AR桶k┘
                ↑ 桶1的all-reduce 与 层N-1/N-2 的梯度计算并发
最终只剩最后一个桶的 all-reduce "尾巴"暴露在反向之后
```

**为什么分桶而不是每个参数一发?** 因为每次 all-reduce 有固定开销 α(launch + 协议 + NCCL 同步),小张量发太多次,α·步数 会反扑(回到 §2.2 的 α 项)。桶太大又减少重叠机会(要等整桶填满才能发)。**桶大小是 α(发太多次的延迟惩罚)和重叠粒度(发太晚的暴露惩罚)之间的 trade-off**——这就是 DDP 有 `bucket_cap_mb` 这个旋钮的根本原因。

### 4.4 经典模式二:流水线并行的 1F1B 与 micro-batch 重叠

第 04 章的流水线并行里,重叠的对象是**点对点通信(激活/梯度在相邻 stage 间传递)**与**计算**。朴素流水线("先全部前向再全部反向")有巨大 **bubble**(气泡,空等时间)。**1F1B(one-forward-one-backward)** 调度让每个 stage 交替做一个 micro-batch 的前向和一个的反向,使得:**stage k 把激活发给 stage k+1 的同时,stage k 已经在算下一个 micro-batch 的前向。** 通信(send/recv 激活)被计算(下一个 micro-batch)盖住。气泡占比近似:

```
bubble 占比 ≈ (p - 1) / (m + p - 1)     p=stage 数, m=micro-batch 数
```

`m` 越大气泡越小——这是"流水线并行要切很多 micro-batch"的量化依据。注意这里藏的是**点对点通信**,机制和集合通信的重叠一样:独立 stream + 异步 send/recv。

### 4.5 经典模式三:计算-通信融合(把通信切片塞进计算切片之间)

更激进的重叠是把一个大算子和它后面的通信**沿某个维度切片、交错执行**:算完第一片就发第一片、同时算第二片……让通信和计算在更细粒度上流水线。例如张量并行里把 GEMM 沿输出维分块,每算完一块就 reduce-scatter 一块。这类"computation-communication overlapping kernel"是 Megatron-LM、TransformerEngine、以及各种 `async-tensor-parallel` 实现的内核,本质是 §4.1 的细粒度流水线版本。**代价:实现复杂、切片引入额外 α 开销和 kernel 复杂度;收益:把本来暴露的通信尾巴也藏掉。**

## 五、编译与算子融合:为什么逐个小算子慢

切到本章下半场。先把"慢"量化,再讲融合怎么救,最后落到 Triton。

### 5.1 第一性原理:逐元素算子必然 memory-bound,且撞两堵墙

考虑一串逐元素操作(elementwise):`y = relu(x * a + b) * 2`,x 是 n 个 fp16 元素的张量。朴素 eager 执行会发**四个独立 kernel**(`mul`、`add`、`relu`、`mul`),每个 kernel 的生命周期是:

```
单个 elementwise kernel 的 HBM 流量(以 *a 这步为例,fp16=2字节):
  读 x:    n · 2 字节
  写 tmp:  n · 2 字节
  → 每个 kernel 4n 字节 HBM 流量(读+写),只做 O(n) 次浮点运算
  算术强度 ≈ (n 次乘) / (4n 字节) = 0.25 FLOP/byte   ← 极低!
```

第 00 章 roofline 判定:算术强度 0.25 FLOP/byte,远低于 A100 的 ridge point(FLOP/s ÷ band ≈ 250e12/2e12 ≈ 125 FLOP/byte)。**结论:逐元素算子被牢牢钉在带宽墙上,Tensor Core 几乎全程闲置。** 四个 kernel 串起来的总 HBM 流量:

```
mul:  读x 写t1      = 2n + 2n = 4n
add:  读t1 读b 写t2 = 2n + 2n + 2n = 6n
relu: 读t2 写t3     = 2n + 2n = 4n
mul:  读t3 写y      = 2n + 2n = 4n
────────────────────────────────
总 HBM 流量 = 18n 字节,4 次 kernel 启动
```

**两堵墙都撞上了:** (a)带宽墙——18n 字节里有 12n 是"中间结果的无谓往返"(t1/t2/t3 写出去又读回来);(b)启动墙——4 次 kernel launch,每次有固定 launch latency(典型 ~5 μs 量级「待核」,小张量时这是绝对主导)。

### 5.2 融合:中间结果留在寄存器,HBM 只读一次写一次

**算子融合(operator fusion)的核心:把多个连续算子合成一个 kernel,中间结果不落 HBM,而是留在寄存器/SRAM。** 上面四步融成一个 kernel 后:

```
fused kernel:  读 x(2n) + 读 b(2n) → 寄存器里算 relu(x*a+b)*2 → 写 y(2n)
总 HBM 流量 = 2n + 2n + 2n = 6n 字节,1 次 kernel 启动
```

**省了多少?** HBM 流量从 18n → 6n,**减少 67%(省 12n)**;kernel 启动从 4 → 1,**减少 75%**。由于这段是 memory-bound,**运行时间近似正比于 HBM 流量,所以融合后约 3× 加速**(在 launch 不主导的中大张量上)。这就是"中间结果留寄存器/SRAM 不落 HBM"这句话的全部威力。

**一般化结论(背下来):** 把 k 个逐元素算子(各自读一个写一个)融成一个,HBM 流量从 ≈`2k·(元素数·字节)`(注:k 个 kernel,相邻共享中间张量,实际是 `读x + k 次写 + (k-1) 次读中间 = (2k)n×... `,简化记忆为"每个未融合算子 ~2 趟 HBM,融合后整段只 ~2 趟")降到 ≈`2·(元素数·字节)`,**省下趟数 = 2k-2 = 2(k-1),即基线 2 趟的 (k-1) 倍,k→∞ 时流量缩减倍数 → k**;kernel 启动从 k 降到 1。这条规律对任何"逐元素链 + 归约"都成立——FlashAttention(第 09 章)就是把 QKᵀ→softmax→×V 这一长串融成一个 kernel、中间矩阵不落 HBM 的极致案例。

### 5.3 什么能融、什么难融

融合不是无脑合并,有清晰的能/不能边界:

- **极易融:** 逐元素链(elementwise→elementwise)。各元素独立,中间值天然只活在一个线程的寄存器里。
- **能融(需技巧):** 逐元素 + 归约(如 `mul` 后 `sum`/`softmax`),归约在 block 内用 SRAM 做(见 §6 fused softmax)。
- **能融(producer-consumer):** GEMM 的 epilogue(矩阵乘后紧跟的 bias+激活),CUTLASS/cuBLASLt 的 epilogue fusion 把激活塞进 GEMM 尾部,省一趟 HBM。
- **难融:** 两个大 GEMM 之间(中间矩阵太大装不下 SRAM,且各自已是 compute-bound,融了不省什么);需要全局同步的操作(如跨 block 的归约,GPU 没有廉价的全局 barrier)。
- **会变慢的反例:** 把一个 memory-bound 算子和一个寄存器压力极大的算子融合,可能导致**寄存器溢出(register spilling)到本地内存**或 **occupancy(同时驻留的线程束数)暴跌**,反而更慢。**融合不是免费的——寄存器和 SRAM 是稀缺资源。**

### 5.4 编译器的角色:torch.compile / XLA / TVM 各自在解决什么

手写融合 kernel 不可扩展(模型几百个算子,组合爆炸)。深度学习编译器自动做这件事:

- **`torch.compile`(PyTorch 2.x):** 三段式——**TorchDynamo** 用 CPython frame eval 钩子把 Python 代码**抓取(capture)**成 FX 计算图(遇到无法处理的动态控制流就 graph break 退回 eager);**AOTAutograd** 把前向图微分出反向图一起优化;后端默认 **TorchInductor**,它把图做**算子融合 + 调度**,并**直接生成 Triton kernel(GPU)/ C++/OpenMP(CPU)**。**记住这条链:Dynamo 抓图 → Inductor 融合 → 生成 Triton。** 所以学 Triton 不只是手写,也是读懂 `torch.compile` 产物的钥匙。
- **XLA(Accelerated Linear Algebra,TF/JAX 的编译器):** 面向**静态图**做整图优化,核心 pass 就是 **fusion**(把逐元素和归约融成 fusion node)+ layout 优化 + 内存规划。JAX 的 `jit` 走 XLA。**特点:静态 shape 假设强、整图优化彻底,但动态性差(shape 一变就重编译)。** 这也是 JAX/XLA 在 TPU 上(同构、静态)如鱼得水的原因。
- **TVM(Apache TVM):** 更偏**推理部署的端到端编译栈**,把模型编译到各种后端(CPU/GPU/边缘/加速器)。核心是 **tensor expression + 自动调度(AutoTVM / Ansor 用机器学习搜最优 schedule)**——把"算什么(compute)"和"怎么算(schedule:循环分块、并行、向量化)"解耦,自动搜 tiling/融合策略。**适合异构后端、追求极致推理延迟的场景。**

**一句话区分:** `torch.compile` = 训练/研究友好、动态性好、生成 Triton;XLA = 静态整图、TPU 本命;TVM = 推理部署、多后端、自动调度搜索。三者的共同底层动作都是**融合 + 调度**,只是抓图方式、动态性假设、目标后端不同。

## 六、Triton 实战:从编程模型到一个融合 kernel

### 6.1 Triton 编程模型:为什么是"block 级"而不是"thread 级"

**Triton 是一种用 Python 写 GPU kernel 的语言/编译器(OpenAI 出品)。** 它和 CUDA 的根本区别在抽象粒度:

- **CUDA 是 thread 级(SIMT):** 你写单个线程的逻辑,手动管 thread/block 索引、shared memory、`__syncthreads`、bank conflict、coalescing。强大但心智负担重。
- **Triton 是 block 级(blocked program):** 你写**一个 program(对应一个 thread block)如何处理一个数据块(block/tile)**。线程在块内怎么分、shared memory 怎么用、访存怎么 coalesce、向量化怎么做——**Triton 编译器自动搞定**。你只需用张量化的 `tl.load/tl.store` 在块这个粒度上思考。**这正好是写融合 kernel 最舒服的粒度:一个 program 把一个 tile 读进 SRAM/寄存器,算完一整条融合链,再写回。**

核心 API(够你写出本章 kernel):

- `tl.program_id(axis)`:当前 program 在网格(grid)里的索引,相当于 CUDA 的 `blockIdx`。决定"我负责哪一块数据"。
- `tl.arange(0, BLOCK)`:生成 `[0,1,...,BLOCK-1]`,用来构造块内偏移。
- `block_start + tl.arange(...)`:得到这个 program 要访问的一组**绝对偏移**(指针算术)。
- `mask = offsets < n`:**边界掩码**。当数据量 n 不是 BLOCK 整数倍时,最后一个块会越界——mask 告诉 `load/store` 哪些 lane 是有效的。
- `tl.load(ptr + offsets, mask=mask, other=0.0)`:从 HBM 把一个块**搬进片上**(寄存器/SRAM),mask 外的位置填 `other`。
- `tl.store(ptr + offsets, value, mask=mask)`:把块写回 HBM,只写 mask 内的。
- `tl.max / tl.sum(x, axis=0)`:**块内归约**(在 SRAM/寄存器里做,这是 fused softmax 的关键)。
- `@triton.jit`:装饰器,标记这是 Triton kernel;`BLOCK_SIZE: tl.constexpr` 表示编译期常量(让编译器能展开、定块大小)。

### 6.2 例一:融合的逐元素 kernel(把 §5.1 的四算子合一)

把 `y = relu(x * a + b) * 2` 融成一个 kernel,直接兑现 §5.2 算出的 6n 字节、1 次启动:

```python
import triton
import triton.language as tl

@triton.jit
def fused_elementwise_kernel(
    x_ptr, b_ptr, y_ptr,          # 输入/输出张量的首地址(GPU 指针)
    a_scalar,                     # 标量 a(逐元素乘的系数)
    n_elements,                   # 张量元素总数
    BLOCK_SIZE: tl.constexpr,     # 每个 program 处理的元素数(编译期常量)
):
    # 1) 我是第几块?算出本块负责的元素绝对偏移
    pid = tl.program_id(axis=0)
    block_start = pid * BLOCK_SIZE
    offsets = block_start + tl.arange(0, BLOCK_SIZE)

    # 2) 边界掩码:最后一块可能越界,mask 外的不读不写
    mask = offsets < n_elements

    # 3) 从 HBM 把这一块读进寄存器(整段融合里 x 只读这一次)
    x = tl.load(x_ptr + offsets, mask=mask, other=0.0)
    b = tl.load(b_ptr + offsets, mask=mask, other=0.0)

    # 4) 在寄存器里一口气算完整条链:mul → add → relu → mul
    #    中间结果 t1/t2/t3 全程不落 HBM!这是省 12n 字节的地方。
    t = x * a_scalar + b          # mul + add
    t = tl.where(t > 0, t, 0.0)   # relu
    y = t * 2.0                   # mul

    # 5) 结果写回 HBM(整段融合里只写这一次)
    tl.store(y_ptr + offsets, y, mask=mask)


def fused_elementwise(x, b, a):
    y = torch.empty_like(x)
    n = x.numel()
    # grid:需要多少个 program 覆盖 n 个元素(向上取整)
    grid = lambda meta: (triton.cdiv(n, meta["BLOCK_SIZE"]),)
    fused_elementwise_kernel[grid](x, b, y, a, n, BLOCK_SIZE=1024)
    return y
```

**逐行对照机制:** `program_id` + `block_start` + `arange` 把一维数据切成 `cdiv(n, BLOCK)` 块,每个 program 认领一块;`mask` 处理非整除的尾块;`tl.load` 把块搬进片上后,第 4 步的所有运算**都在寄存器里发生,t 从不写回 HBM**——这就是融合省下 12n 字节的物理位置;最后一次 `tl.store`。**对照 eager 的四个 kernel(18n 字节、4 次 launch),这个 kernel 是 6n 字节、1 次 launch。**

### 6.3 例二:fused softmax(逐元素 + 归约的融合,且数值安全)

Softmax 是融合的经典考题,因为它既有归约(求 max、求 sum)又有逐元素(exp、除),还有数值稳定性要求。**数值安全的 softmax 必须减去行最大值**(否则 `exp` 上溢):

```
softmax(x)ᵢ = exp(xᵢ - max(x)) / Σⱼ exp(xⱼ - max(x))     ← 减 max,数学等价但不溢出
```

eager 实现要发多个 kernel(max、sub、exp、sum、div),每个都把整行往返 HBM。融合后,**一个 program 处理一整行,行内的 max/exp/sum 全在 SRAM/寄存器做,HBM 只读一次写一次**:

```python
@triton.jit
def fused_softmax_kernel(
    x_ptr, y_ptr,
    x_row_stride, y_row_stride,   # 行间步长(以元素计),用于定位每行起点
    n_cols,                       # 每行列数
    BLOCK_SIZE: tl.constexpr,     # >= n_cols,一整行装进一个块
):
    # 1) 一个 program 负责一整行
    row_idx = tl.program_id(axis=0)
    row_start_ptr = x_ptr + row_idx * x_row_stride
    col_offsets = tl.arange(0, BLOCK_SIZE)
    mask = col_offsets < n_cols

    # 2) 整行读进片上(-inf 填充,保证它不影响 max/exp)
    row = tl.load(row_start_ptr + col_offsets, mask=mask, other=float("-inf"))

    # 3) 数值稳定:减行最大值(块内归约,在 SRAM/寄存器里)
    row_minus_max = row - tl.max(row, axis=0)

    # 4) exp 与归一化(全程不回 HBM)
    numerator = tl.exp(row_minus_max)
    denominator = tl.sum(numerator, axis=0)      # 块内归约求和
    softmax_out = numerator / denominator

    # 5) 写回(只写一次)
    out_row_ptr = y_ptr + row_idx * y_row_stride
    tl.store(out_row_ptr + col_offsets, softmax_out, mask=mask)


def fused_softmax(x):
    n_rows, n_cols = x.shape
    # 块大小取 >= n_cols 的 2 的幂,让一整行装进一个 program
    BLOCK_SIZE = triton.next_power_of_2(n_cols)
    y = torch.empty_like(x)
    fused_softmax_kernel[(n_rows,)](
        x, y, x.stride(0), y.stride(0), n_cols, BLOCK_SIZE=BLOCK_SIZE,
    )
    return y
```

**这个 kernel 的访存账(设行长 d、共 r 行、fp32):**

```
eager softmax(简化为 max/exp/sum/div 共 ~4 趟整张量读写):
  ≈ 多个 kernel,每 kernel 读 4rd 写 4rd ... 量级 ~ 4×(2·r·d·4) 字节 + 多次 launch
fused softmax:
  读 r·d·4 + 写 r·d·4 = 8rd 字节,r 次 launch(每行一个 program)
  → 中间的 exp 结果、max、sum 全在片上,省下所有中间往返
```

**两个新手必栽的坑(务必记住):**

1. **行长 d 超过单个 program 能装的块大小怎么办?** 上面的 kernel 假设 `BLOCK_SIZE >= n_cols`,一整行进一个块。当 d 很大(比如 attention 的长序列)装不下时,必须改成**分块 + online softmax**(running max/running sum,逐块更新),这正是 FlashAttention(第 09 章)的核心机制。**fused softmax 是 FlashAttention 的"婴儿版"——理解了这个,FlashAttention 只是把它推广到 d 装不下 + 还要乘 V。**
2. **`other` 填充值选错会算错。** max 归约的填充必须是 `-inf`(否则 padding 的 0 可能变成 max);若直接对原始值求和的场景,填充要用 0。**填充值要让被 mask 的 lane 对该归约成为"中性元"**(max 的中性元是 -inf,sum 是 0)。

### 6.4 怎么验证融合真的快了:用带宽反推

第 00 章教过用实测反推有效带宽。验证一个融合 kernel 是否到位的判据:**算出它的 HBM 流量,除以实测时间得到有效带宽,看是否逼近硬件峰值带宽。** 对 memory-bound 的融合 kernel:

```
有效带宽 = 总HBM字节 / 实测时间      (例:fused softmax = 8rd / t_measured)
若 有效带宽 ≈ 70%~90% 峰值带宽 → 融合到位、访存接近 coalesced 最优
若 远低于峰值 → 可能 occupancy 不足 / 访存未合并 / 块大小不当
```

**这是判断"该不该继续优化"的硬指标**:已经吃满带宽的 memory-bound kernel,再怎么调也快不了(撞墙了),该去看是不是算法层面少读点字节(如换更激进的融合或换算法)。

## 七、设计权衡与常见坑

**通信侧:**

- **α 反扑(小张量多卡):** §2.2 的代价模型里,卡数 N 一大、单条消息一小,`步数·α` 这一项会主导。表现为"卡加到一定数量后扩展效率断崖"。对策:梯度桶化(攒大再发,§4.3)、用延迟优的 tree/recursive-doubling 替 ring(小消息时)。
- **把张量并行跨节点放:** §2.2 算过,all-reduce 走 IB(β 掉一个量级)会让通信逼近甚至超过计算。**铁律:张量并行留节点内 NVLink;跨节点交给通信频率低的数据并行/流水线并行。**
- **重叠拿不满:** NCCL kernel 抢 SM(§3.4),"完美重叠"是理论上界,实际打折。别按 `max(T_comp, T_comm)` 做承诺,留余量。
- **all-to-all 撞对分带宽:** MoE 的大忌,网络对分带宽不足时 all-to-all 直接成瓶颈(§3.3)。

**编译/融合侧:**

- **过度融合导致寄存器溢出 / occupancy 暴跌:** 融太多、临时变量太多 → 寄存器不够 → spill 到 local memory(其实是 HBM)→ 比不融还慢(§5.3)。**融合有甜点,不是越多越好。**
- **`torch.compile` 的 graph break:** Dynamo 遇到无法抓取的动态控制流(数据依赖的 `if`、`.item()`、打印张量值等)会 graph break,把图切成多段,融合机会被切碎、加速大打折扣。**写能编译的代码:避免在热路径上 `.item()`、Python 层动态分支。**
- **动态 shape 反复重编译:** XLA/Inductor 对每个新 shape 可能重新编译(编译本身耗时几秒到几十秒)。shape 频繁变(变长序列不 padding)会让你陷在编译里。对策:分桶/padding 到有限几个 shape。
- **Triton 块大小选错:** 太小 → occupancy 高但每 program 干活少、launch 开销摊不薄;太大 → 寄存器/SRAM 压力大、occupancy 跌。**块大小要 autotune(`@triton.autotune` 跑几个候选选最快),别拍脑袋。**
- **mask 与填充值不匹配归约语义:** §6.3 的坑,会静默算错(不报错但结果不对),最难查。

## 八、动手练习

1. **(估算题)拆原语省流量。** 一个 7B 模型 fp16 梯度 M=14 GB,N=8 卡。(a)用 all-reduce 同步,每卡链路流量 `V_link` 是多少?(b)若改用 ZeRO-2:reduce-scatter(梯度)+ 各自更新 1/N 参数 + all-gather(参数),两步各自的 `V_link` 之和是多少?二者总流量谁大?(c)ZeRO-2 真正省的是什么,不是通信量而是______?**提示:** 套 §2.2 的 `V_link`:all-reduce≈2(N-1)/N·M,RS≈AG≈(N-1)/N·M;填空想峰值显存。

2. **(估算题)重叠的盈亏平衡点。** 某层计算 4 ms,要 all-reduce 的梯度 M=64 MB。(a)节点内 β=200 GB/s,通信时间多少?能被计算盖住吗?(b)跨节点 β=25 GB/s 呢?(c)若想让跨节点通信仍被盖住,计算至少要多长(给不等式)?**提示:** `T_comm = 2(N-1)/N·M / β`,完美重叠要 `T_comm ≤ T_compute`。

3. **(编码题)手动融合并数账。** 给定 `z = sigmoid(x) * tanh(y) + x`(x、y 各 n 个 fp16)。(a)写出 eager 会发几个 kernel、总 HBM 流量(按"每算子读写各一趟")。(b)写一个 Triton 融合 kernel(参考 §6.2),数清融合后 HBM 流量和 kernel 数。(c)算出 HBM 流量和 launch 各省了百分之几。**提示:** eager 大约 sigmoid/tanh/mul/add 四个算子,注意 x 被用了两次(`sigmoid(x)` 和 `+x`)——融合后 x 只需读一次,这恰好多省一趟。

4. **(推导题)融合的 HBM 流量通式。** 设一条 `k` 个逐元素算子的链,每个算子读 1 个张量写 1 个张量,张量都是 n 元素 c 字节。(a)推导未融合时的总 HBM 流量(注意相邻算子共享中间张量,中间张量既被写又被读)。(b)推导融合成一个 kernel 后的流量。(c)给出"省下的流量倍数"关于 k 的表达式,并说明 k→∞ 时趋近多少倍。**提示:** 未融合 ≈ `(读x:1) + (写中间:k) + (读中间:k-1) + ... ` 仔细数;融合后只有 `读x + 写最终输出 = 2nc`。

## 九、源码 / 论文导读

- **NCCL:** 官方文档的 *Collective Operations* 与 *Environment Variables* 两节,搞清每个原语语义和 `NCCL_ALGO`(Ring/Tree/NVLS)、`NCCL_PROTO`、`NCCL_MIN_NCHANNELS` 的含义。`NCCL_DEBUG=INFO` 打出它实际选了什么算法/拓扑——重叠和带宽调优必看。
- **PyTorch DDP:** 读 `torch/nn/parallel/distributed.py` 的 `Reducer`(C++ 侧 `reducer.cpp`)——梯度桶化、`bucket_cap_mb`、反向 hook 触发异步 all-reduce 的逻辑就是 §4.3 的工业实现。论文 *PyTorch Distributed: Experiences on Accelerating Data Parallel Training*(Li et al., VLDB 2020)系统讲了 bucketing 与重叠。
- **Megatron-LM:** `megatron/core` 里张量并行的 all-reduce/reduce-scatter 插入点,以及 sequence parallelism、`--tp-comm-overlap` 相关的计算-通信重叠 kernel(§4.5)。论文 *Reducing Activation Recomputation in Large Transformer Models*(Korthikanti et al., 2022)讲序列并行如何用 RS+AG 替 all-reduce。
- **Triton:** 官方 tutorials 的 *Fused Softmax* 和 *Matrix Multiplication* 两篇是必做(本章 §6.3 即据 fused softmax 范式)。论文 *Triton: An Intermediate Language and Compiler for Tiled Neural Network Computations*(Tillet et al., MAPL 2019)讲 block 级抽象和编译器为什么能自动管 shared memory / coalescing。
- **torch.compile / Inductor:** PyTorch 文档的 *torch.compile* 与 Inductor 设计文档;`TORCH_COMPILE_DEBUG=1` 或 `TORCH_LOGS="output_code"` 能 dump 出 Inductor 生成的 **Triton 源码**——直接看编译器是怎么融合你的图的,是把本章上下半场打通的最好练习。
- **XLA / TVM:** XLA 的 *Operation Semantics* 与 fusion pass;TVM 的 Ansor 论文 *Ansor: Generating High-Performance Tensor Programs for Deep Learning*(Zheng et al., OSDI 2020)讲自动调度搜索(compute/schedule 解耦)。

## 十、小结与承上启下

这一章把贯穿全课的两条暗线提到台面,讲透了两类系统税及其对策:

- **通信税 → 选对原语 + 重叠。** 七个集合原语的语义和统一代价模型(`T ≈ 步数·α + V_link/β`),恒等式 all-reduce = reduce-scatter + all-gather 是串起 DDP / ZeRO / 张量并行 / 序列并行原语选择的总钥匙;硬件拓扑(HBM→NVLink→IB 两个数量级的带宽落差)决定 β,决定"什么并行放哪一层";计算-通信重叠用独立 CUDA stream + event 把通信延迟藏进计算,把总时间从 `T_comp+T_comm` 压向 `max(T_comp,T_comm)`。
- **编译税 → 融合 + 编译器。** 逐元素算子必然 memory-bound 且撞带宽墙 + 启动墙;融合把 k 个算子的 HBM 流量从 ~2k·趟压到 ~2 趟、launch 从 k 压到 1;`torch.compile`(Dynamo 抓图→Inductor 融合→生成 Triton)、XLA(静态整图)、TVM(多后端自动调度)自动化这件事;Triton 的 block 级抽象让你能亲手写融合 kernel,fused softmax 是 FlashAttention 的婴儿版。

**在整门课里的位置:** 前面第 02~06 章的并行策略,它们的通信成本和重叠机制由本章 §2~§4 兜底;第 09 章 FlashAttention 的 tiling + 中间矩阵不落 HBM,正是本章 §5~§6 融合思想的极致;后续讲推理服务、批处理、量化部署时,torch.compile / Triton 生成的融合 kernel 和集合通信的重叠会反复出现。**至此你应当能拿到一段陌生的分布式训练/推理代码,定位出"哪里通信暴露了、哪里小算子撞带宽了",并给出"加重叠 / 拆原语 / 融合 / 上 torch.compile"的具体改法和量化预期。** 这就是从"会用框架"到"能做系统优化"的分水岭。
