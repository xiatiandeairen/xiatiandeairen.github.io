> 这一章把你从"知道把 FP16 权重压成 INT4 能省显存"带到"能从第一性原理推出量化为什么几乎不掉点、能逐字节算清 W4A16 在 decode 阶段的加速来源、并讲清 GPTQ / AWQ / SmoothQuant 三种代表方法在数学上各自解了什么问题、彼此差在哪——直至你能照着伪代码自己实现一个 group-wise 量化器和 GPTQ 的误差补偿主循环"。

## 引言:一句话和一个反直觉

一句话:**量化不是"把数字粗暴截断",而是"在给定比特预算下,寻找一组离散点,使它对模型输出的扰动最小"——这是一个有目标函数、可以求极值的优化问题。**

反直觉的事实:一个 70 亿参数的模型,把权重从每个数 16 比特压到 4 比特,精度(困惑度 / 下游任务准确率)往往只掉零点几个百分点,甚至在某些任务上看不出差别。**16 比特里有 12 比特是"几乎免费可以扔掉的"——更准确说,4 比特(后文精确算到 b_eff≈4.125)就够表达,省下的约 12 比特对输出几乎无损。** 这件事一开始让人难以相信:训练时我们小心翼翼地用混合精度、怕梯度数值溢出,怎么推理时随手砍掉四分之三的比特反而没事?

这一章就是要把这个反直觉讲到你信服、并且能定量解释。核心分三层:

1. **物理层(为什么值得做):** 上一章(投机解码)反复证明 decode 是 memory-bound——算 1 个 token 的瓶颈是把几百 GB 权重从 HBM 搬一遍。**那么把权重本身做小,搬的字节数直接减少,就是从根上打这个瓶颈。** 这是量化在推理侧最硬的动机,比"省显存"更本质。
2. **数学层(为什么不掉点):** 量化误差是有结构的——它不是随机噪声,而是可以通过"选对 scale、选对粒度、补偿已产生的误差、保护关键通道"来系统性压低。
3. **方法层(三种代表怎么做):** GPTQ 用二阶信息做误差补偿,AWQ 靠激活分布保护 salient 通道,SmoothQuant 把量化难度在激活和权重间做等价迁移。三者解的是同一个问题的不同侧面。

我们从最基础的量化公式推起,不跳步。

## 一、量化基本式:一个仿射映射

### 1.1 问题:把连续区间塞进 2ᵇ 个格子

量化(quantization)做的事,本质是把一段连续浮点值的区间,**映射到有限个整数格点**上。给定 b 比特,我们有 2ᵇ 个可用的整数取值(例如 b=4 时是 16 个,b=8 时是 256 个)。问题是:如何设计这个映射,使还原回来的值尽量接近原值?

最通用的形式是**仿射量化(affine quantization)**,即一个"先缩放、再平移、再取整"的线性映射:

```
量化(quantize):   q = round(x / s) + z
反量化(dequantize): x̂ = s · (q − z)
```

其中:
- `x` 是原始浮点值(比如一个权重);
- `s`(scale,缩放因子)是一个正浮点数,决定"每个整数格子代表多大的真实值跨度",是量化的**分辨率**;
- `z`(zero-point,零点)是一个整数,负责把浮点的零对齐到某个整数格点,处理区间不对称;
- `q` 是量化后的整数(存进显存的就是它);
- `x̂` 是反量化还原值,**一般不等于 x**,二者之差就是量化误差。

`round(·)` 是四舍五入到最近整数。两条式子互为逆运算(忽略 round 的取整损失),所以这是一对编解码器。

### 1.2 推导 scale 和 zero-point

给定要量化的浮点区间 `[x_min, x_max]`,和整数可用区间 `[q_min, q_max]`(b 比特无符号整数时是 `[0, 2ᵇ−1]`),我们要求这个仿射映射把两个区间的端点对齐:

```
x_min ↦ q_min,  x_max ↦ q_max
```

代入 `q = x/s + z`(暂时不取整,先求连续解),两个端点给出两个方程:

```
q_min = x_min / s + z
q_max = x_max / s + z
```

两式相减消掉 z:

```
q_max − q_min = (x_max − x_min) / s
⟹  s = (x_max − x_min) / (q_max − q_min)        ……(scale 公式)
```

再把 s 代回任一式解 z(取整成整数零点):

```
z = round(q_min − x_min / s)                     ……(zero-point 公式)
```

**直觉:** s 就是"真实区间宽度 ÷ 整数区间宽度",即每个整数台阶对应多大的真实跨度;z 是"为了让 x_min 落到 q_min,整数轴要平移多少格"。这两个数,加上量化后的整数 q,就能还原出 x̂。

### 1.3 对称 vs 非对称

上面是**非对称(asymmetric)**量化:`x_min`、`x_max` 各自独立,z 一般非零。它能紧凑地覆盖任意区间(比如 ReLU 后的激活全是非负,区间 `[0, x_max]`,非对称能把 256 个格子全用在正半轴)。

**对称(symmetric)**量化则强制区间关于 0 对称:取 `x_max_abs = max(|x_min|, |x_max|)`,映射区间为 `[−x_max_abs, +x_max_abs]`,并令 `z = 0`。此时公式塌缩为:

```
s = x_max_abs / (2ᵇ⁻¹ − 1)      (用有符号整数 [−(2ᵇ⁻¹−1), 2ᵇ⁻¹−1])
q = round(x / s)                 (无 zero-point)
x̂ = s · q
```

**为什么 weight-only 量化几乎都用对称?** 两个理由,都很实在:

1. **权重分布本身近似零中心、近似对称**(预训练权重通常是均值约 0 的钟形分布),对称量化浪费不大。
2. **计算上省一笔。** 矩阵乘 `Y = X · W`,若 W 用非对称量化 `Ŵ = s_w·(Q_w − z_w)`,展开会多出一项 `−s_w·z_w·X·1`(zero-point 引入的交叉项),需要额外处理;对称量化 `z_w=0`,这一项直接消失,反量化就是一次标量乘,kernel 干净得多。**这是工程上对称量化的硬优势,不是审美选择。**

> ⚠ 新手坑:对称量化里那个分母用 `2ᵇ⁻¹ − 1`(b=8 时是 127)而不是 `2ᵇ⁻¹`(128),是因为有符号整数正负半轴各只有 127 个非零格(INT8 范围 −128…127,但对称量化通常弃用 −128 以保持对称,只用 −127…127)。差一格,但概念上要清楚:对称牺牲了一点表示范围换取 z=0 的计算简洁。

### 1.4 量化误差有多大:一个可算的上界

取整 `round(·)` 引入的误差,在 `[−s/2, +s/2]` 之间(四舍五入最多偏半个台阶)。若假设量化误差在台阶内均匀分布,其方差是经典结论:

```
量化噪声方差:E[(x − x̂)²] = s²/12
       (均匀分布 U[−s/2, s/2] 的方差 = (s)²/12)
```

**这条公式是理解一切的钥匙:量化误差的大小由 s 决定,而 s = 区间宽度 / 格子数。** 要减小误差,只有两条路:

- **减小区间宽度**(分母里的 `x_max − x_min`):这就是 per-channel / per-group 量化和异常值处理要干的事——别让少数极端值把区间撑大;
- **增大格子数**(分母里的 `2ᵇ`):这就是"多给比特",但比特预算是死的,不能随便加。

后面三种方法,本质都是在比特预算固定的前提下,**想方设法压低有效的 s**。记住这一点,GPTQ/AWQ/SmoothQuant 就有了统一的解读视角。

## 二、量化粒度:per-tensor / per-channel / per-group

### 2.1 问题:一个 scale 管整个张量,够不够?

一个权重矩阵 W 形状是 `[out_features, in_features]`(输出维 × 输入维)。最省事的做法是**全张量共享一个 scale**(per-tensor):整个 W 算一个 `x_max_abs`,一个 s。

问题来了:不同输出通道(W 的行)的权重幅度可能差很多。假设第 5 行的权重都在 ±0.01,第 100 行有个别权重到 ±5。per-tensor 下 `x_max_abs ≈ 5`,于是 `s ≈ 5/127 ≈ 0.039`。但第 5 行最大才 0.01,**它整行被量化后全是 0 或 ±1,几乎所有信息丢光**——它的有效比特数远不到 4。少数大值通道"绑架"了 scale,让小值通道陪葬。

### 2.2 解法:细化粒度,各管各的区间

粒度(granularity)就是"多少个权重共享一个 scale"。从粗到细:

```
per-tensor:    整个 W 一个 s          —— 存储开销最小,精度最差
per-channel:   每个输出通道(行)一个 s —— W[i,:] 共享 s_i,行内自适应
per-group:     每行再切成若干组,每 g 个相邻权重一个 s —— 最细,精度最好
```

**per-channel** 让每个输出通道用自己的区间,大值通道和小值通道互不干扰,是 weight-only 量化的常见起点。**per-group**(也叫 group-wise)更进一步:把每行的 `in_features` 个权重按每 g 个一组(典型 g=128)切块,每块一个独立 scale。组越小越贴合局部分布,误差越小,但 scale 的数量(存储开销)随之上升——这是核心权衡,2.4 节定量算。

> **为什么沿 in_features 方向分组?** 因为矩阵乘 `Y[i] = Σ_k X[k]·W[i,k]` 是沿 k(= in_features)做内积累加。把同一段 k 范围的权重归一组、共享 scale,反量化时可以"先按组反量化再累加",和矩阵乘的累加结构对齐,kernel 好写。这是个工程约束驱动的选择,不是随便切的。

### 2.3 激活的量化粒度:为什么更难

权重是静态的(训练完就固定),离线慢慢算 scale 即可。**激活(activation)是动态的**——每次前向、每个输入样本都不同,scale 要么在线算(per-tensor 动态量化,每次前向求一遍 max),要么离线用校准集统计一个固定值(静态量化)。

更要命的是**激活的量化粒度受矩阵乘结构限制**。在 `Y = X · W` 里,X 形状 `[tokens, in_features]`。如果你想对激活做 per-channel(沿 in_features 给每列一个 scale),那么内积 `Σ_k X[t,k]·W[i,k]` 里每个 k 项的 X 端 scale 都不同,无法提到累加外面,**破坏了 INT8 矩阵乘"整数累加 + 最后统一反量化"的高效路径**。所以激活的高效量化粒度通常只能做到 **per-token(每行一个 scale,沿 tokens 方向)** 或 per-tensor——而这恰恰管不住"沿 in_features 通道分布的异常值"。**这个矛盾正是 SmoothQuant 要解的核心问题**(第六节)。

### 2.4 量化:group 量化的额外存储

定量算清 per-group 的代价。设权重矩阵 `[M, K]`(M = out_features,K = in_features),group 大小 g,量化到 b 比特,scale 用 FP16(2 字节)存,对称量化无 zero-point。

```
量化后整数本体:M · K 个数 × b 比特 = M·K·b / 8 字节
scale 数量:每行 K/g 组,共 M·(K/g) 个 scale
scale 存储:M·(K/g) × 2 字节 = 2MK/g 字节

每个权重的"摊薄比特数" b_eff:
  b_eff = (M·K·b/8 + 2MK/g) / (M·K/8)   ……(总字节 ÷ 权重数,再换算成比特)
        = b + 16/g
```

**关键结论:`b_eff = b + 16/g`。** 这条公式极其有用,直接给出 group 量化的"比特税":

```
g = 128, b = 4 → b_eff = 4 + 16/128 = 4.125 比特/权重(开销仅 +3.1%)
g = 64,  b = 4 → b_eff = 4 + 16/64  = 4.25  比特/权重(+6.25%)
g = 32,  b = 4 → b_eff = 4 + 16/32  = 4.5   比特/权重(+12.5%)
per-channel(g=K,设 K=4096)→ b_eff = 4 + 16/4096 ≈ 4.004 比特(几乎免费)
```

**这解释了为什么 g=128 是业界默认甜点:** 它把额外存储压到约 3%,却能让每 128 个权重自适应局部分布,精度收益远大于这 0.125 比特的代价。**若 scale 用 FP32(4 字节)存,公式变成 `b + 32/g`,税翻倍**——所以实践里 scale 多用 FP16/BF16。

(如果还带 zero-point,非对称量化下 zero-point 也要每组存一个,通常用低比特整数,公式再加一项 `b_zp/g`,这里从略。)

## 三、为什么 decode 是 memory-bound:weight-only 量化的加速来源

### 3.1 接上一章:量化打的是同一个瓶颈

上一章的核心结论(roofline):decode 单序列处理 1 个 token 时,算术强度

```
I_decode ≈ 2N FLOP / 2N Byte = 1 FLOP/Byte ≪ 脊点 I*(几十~几百)
```

是重度 memory-bound——**瓶颈是把权重从 HBM 搬过来的字节数,不是算力**。投机解码的办法是"一次搬权重服务 K 个 token"。量化的办法更直接:**把权重本身的字节数做小。**

### 3.2 推导:W4A16 的加速比

**W4A16** 的含义:权重(Weight)用 4 比特存,激活(Activation)保持 16 比特。这是 weight-only 量化的代表配置,专为 decode 设计。来算它的加速。

decode 一个 token,主导耗时是把权重从 HBM 读进来。设权重数 N:

```
FP16 权重:N × 2 Byte = 2N Byte
INT4 权重:N × 0.5 Byte = 0.5N Byte
读权重字节数之比:2N / 0.5N = 4×
```

**因为 decode 是 memory-bound,墙钟时间近似正比于读权重字节数,所以 W4A16 理论上能给 decode 带来约 4× 的加速。** 这是个"几乎白捡"的加速——不改模型结构、不改算法,只是搬的字节少了 4 倍。

但有个关键细节要算清:**计算时仍需把 INT4 反量化回 FP16 再做矩阵乘**(因为激活是 FP16,没有 INT4×FP16 的原生 tensor core 路径)。流程是:

```
INT4 权重(HBM,0.5N 字节)
   → 读进来(这步省了 4× 带宽,是加速来源)
   → 在片上反量化成 FP16(算力开销,但 decode 算力本来闲置,白用)
   → 与 FP16 激活做矩阵乘
```

**反量化的算力开销几乎免费**——因为 decode 本来就 memory-bound,tensor core 在等数据,反量化用的那点 FLOP 填进闲置算力里,不增加墙钟。这正是 weight-only 量化在 decode 上如此划算的根因:**省的是稀缺资源(带宽),花的是过剩资源(算力)。**

### 3.3 为什么不在 decode 用 W8A8

W8A8(权重和激活都 INT8)能用 INT8 tensor core 做整数矩阵乘,理论算力翻倍。但 decode 是 memory-bound,**算力翻倍打不到瓶颈上**——decode 缺的不是算力。而且 W8A8 要量化激活,引入额外的激活异常值难题(第六节),还把权重停在 8 比特(只省 2× 带宽,不如 INT4 的 4×)。

**所以分工很清晰:**

```
decode(memory-bound,延迟敏感):weight-only 低比特(W4A16),最大化带宽节省
prefill / 大 batch 训练后处理(compute-bound):W8A8 整数矩阵乘,吃满 INT8 算力
```

记住这条分工,你就明白为什么 GPTQ/AWQ 主打 W4A16(服务 decode),而 SmoothQuant 主打 W8A8(服务 compute-bound 场景)——**它们瞄准的硬件瓶颈不同**。

### 3.4 PTQ vs QAT

- **PTQ(Post-Training Quantization,训练后量化):** 拿一个训练好的 FP16 模型,**不重新训练**,用少量校准数据(几百条样本即可)直接算出量化参数并量化。成本极低(几分钟到几小时),无需训练基础设施。GPTQ / AWQ / SmoothQuant **全都是 PTQ**——这是它们能被广泛使用的前提。
- **QAT(Quantization-Aware Training,量化感知训练):** 在训练(或微调)过程中就模拟量化的舍入,让模型权重"适应"量化误差,通常精度更高,但需要完整训练流程和数据,成本高。

> **为什么大模型时代 PTQ 成主流?** 因为 70B/100B+ 模型 QAT 的训练成本高到离谱,而 PTQ 配合本章这些巧妙算法,已能把精度损失压到可接受。本章三种方法的全部价值,就在于**"不训练也能高质量量化"**——这是它们的共同擂台。

## 四、异常值是量化的死敌:第一性原理

在讲三种方法前,必须把它们共同要对付的敌人讲透:**异常值(outlier)**。

### 4.1 推导:一个异常值如何毁掉一整组

回到误差公式 `误差 ∝ s`,而 `s = 区间宽度 / 格子数`。对一组要共享 scale 的数 `{x₁,…,x_g}`,对称量化下:

```
s = max_i |xᵢ| / (2ᵇ⁻¹ − 1)
```

scale **完全由组内绝对值最大的那个数决定**。现在设组内 g−1 个数都在 ±1,有 1 个异常值是 100:

```
有异常值:s = 100 / 127 ≈ 0.787
那 g−1 个 ±1 的数,量化后 q = round(±1 / 0.787) = round(±1.27) = ±1
        反量化 x̂ = 0.787 × (±1) = ±0.787,误差约 21%!
        而且它们只用到了 {−1,0,+1} 这 3 个格子,其余 124 个格子全浪费在覆盖 [−100,100] 的空旷区间上

无异常值:s = 1/127 ≈ 0.0079,那些 ±1 的数误差 < 0.4%,127 个格子全用上
```

**一个异常值,把整组的有效比特数从 4 比特打到不足 2 比特。** 这就是"异常值是量化死敌"的精确含义:**它不是自己量化不准,而是它撑大了 scale,让同组所有正常值陪葬。** 量化误差 ∝ s ∝ max|x|,异常值直接顶高了 max|x|。

### 4.2 关键事实:大模型激活里有系统性异常值

研究(LLM.int8、SmoothQuant 等)发现:大模型(约 6.7B 参数以上)的**激活**里,存在**系统性、结构化的异常值**——它们集中出现在**少数固定的特征维度(通道)**上,幅度可达普通激活的几十上百倍,且跨不同输入样本稳定出现在同样的通道。

```
关键观察(三条,后面三种方法都建立在此之上):
1. 异常值主要在激活,不在权重 —— 权重分布相对温和、近似高斯;
   激活(尤其某些层的输入)有极端 outlier 通道。
2. 异常值是"逐通道"的 —— 集中在少数 feature 维度,且跨样本位置稳定。
3. 异常值不能简单丢弃 —— 它们携带重要信息,直接 clip 掉会显著掉点。
```

**这三条事实把量化问题劈成三块,正好对应三种方法:**

```
权重端怎么量化得更准?            → GPTQ(误差补偿)
哪些权重通道因为乘了大激活而格外重要,要保护?  → AWQ(激活感知保护)
激活的逐通道异常值怎么处理才能做 W8A8?        → SmoothQuant(难度迁移)
```

下面逐一推导。

## 五、GPTQ:用二阶信息做误差补偿

### 5.1 问题:逐个量化权重,误差会累积——能不能边量化边补偿?

GPTQ 解决的是**纯权重量化**的精度问题(W4A16,激活不动)。朴素做法是每个权重独立四舍五入到最近格点(round-to-nearest, RTN),但这忽略了一件事:**权重不是孤立的,它们一起决定层的输出。把某个权重量化错了,可以通过微调其他还没量化的权重来补偿这个误差。**

GPTQ 的目标函数是**层级输出重构误差**:对一层,输入校准激活 X,原权重 W,找量化权重 Ŵ 使输出尽量不变:

```
目标:min_Ŵ  ‖ X·Wᵀ − X·Ŵᵀ ‖²_F     (Frobenius 范数,逐输出行可分解)
约束:Ŵ 的每个元素必须落在量化格点上
```

**注意目标不是 `min ‖W − Ŵ‖`(权重本身接近),而是 `min ‖XW − XŴ‖`(输出接近)。** 这是关键区别:我们不在乎权重数值对不对,只在乎它对输出的影响小。两个权重数值偏差相同,但如果一个对应的输入激活大、另一个小,前者对输出的伤害更大——所以补偿要"看激活脸色"。这就引出了 Hessian。

### 5.2 推导:Hessian 从哪来,为什么是 XᵀX

目标函数对一个输出行 w(行向量,长 K)可独立处理。设输入校准矩阵 X 形状 `[n_samples, K]`,该行对输出的贡献是 `X·wᵀ`。重构误差:

```
E(w) = ‖X·wᵀ − X·ŵᵀ‖² = ‖X·(w − ŵ)ᵀ‖²
```

记 δ = ŵ − w(量化引入的权重扰动),展开二阶项(这是关于 δ 的二次型):

```
E = δ · (XᵀX) · δᵀ          (推导:‖Xδᵀ‖² = δ Xᵀ X δᵀ)
```

所以这个二次损失的 **Hessian(二阶导矩阵)就是 `H = 2·XᵀX`**(差个常数因子 2,实现里常直接用 `H = XᵀX` 再配阻尼)。**注意 GPTQ 的量化结果对 H 整体乘一个常数严格不变**:补偿用的 `err = (w−q)/[H⁻¹]_qq` 与补偿方向 `H⁻¹` 第 q 列里的缩放因子相消,量化值 `q` 本身又不依赖 H——所以 H 求和、求平均、带不带因子 2,都不改变最终量化结果,只需配合阻尼数值稳定即可。

```
推导逐行:
  E(δ) = (Xδᵀ)ᵀ(Xδᵀ) = δ (XᵀX) δᵀ
  ∂E/∂δ = 2 (XᵀX) δᵀ
  ∂²E/∂δ² = 2 XᵀX  =: H        ← Hessian 是常数矩阵(二次型),与 δ 无关
```

**直觉:`H = XᵀX` 编码了"各权重通道之间的相关性 + 各通道激活的能量"。** Hᵢᵢ 大,说明第 i 个输入通道激活能量大,动它对输出影响大,量化它要格外小心;Hᵢⱼ 非零,说明通道 i、j 相关,改了 i 可以用 j 来补偿。**这正是误差补偿的数学依据。**

### 5.3 推导:OBS/OBQ——量化一个权重后,如何最优补偿其余权重

GPTQ 的数学内核来自 **OBS(Optimal Brain Surgeon)/ OBQ(Optimal Brain Quantization)**。问题抽象成:已知二次损失 `E(δ) = δ H δᵀ`,现在强制把第 q 个权重 `w_q` 量化(固定为格点值,即 `δ_q` 取定值),问**其余权重该怎么调整,才能把这次量化造成的 E 增量压到最小**?

这是个带等式约束的二次规划。约束是 `δ_q = w_q^quant − w_q`(第 q 维的扰动被钉死),目标 `min δ H δᵀ`。用拉格朗日乘子法解(经典 OBS 结论):

```
最优补偿(把第 q 维误差摊给其余维):
  δ = − (δ_q / [H⁻¹]_qq) · H⁻¹ 的第 q 列        ……(权重更新)

伴随的损失增量(量化第 q 个权重的"代价"):
  ΔE_q = δ_q² / [H⁻¹]_qq = (w_q − w_q^quant)² / [H⁻¹]_qq   ……(误差贡献)
```

```
推导骨架(拉格朗日法):
  min  δ H δᵀ   s.t.  e_qᵀ δᵀ = δ_q     (e_q 是第 q 个单位向量)
  L = δ H δᵀ − λ (e_qᵀ δᵀ − δ_q)
  ∂L/∂δ = 0  ⟹  δᵀ = (λ/2) H⁻¹ e_q
  代入约束 e_qᵀ δᵀ = δ_q  解出 λ/2 = δ_q / [H⁻¹]_qq
  ⟹  δᵀ = (δ_q / [H⁻¹]_qq) · H⁻¹ e_q   (H⁻¹ e_q 即 H⁻¹ 的第 q 列)
  代回 ΔE = δ H δᵀ = δ_q² / [H⁻¹]_qq
```

**两条公式是 GPTQ 的全部数学:**
- **权重更新式**告诉你:量化了 w_q 之后,按 `H⁻¹` 第 q 列的方向去调整所有还没量化的权重,把误差吸收掉;
- **误差贡献式 `ΔE_q = δ_q²/[H⁻¹]_qq`** 告诉你每量化一个权重要付出多大输出代价。

> **直觉:** `[H⁻¹]_qq` 大,意味着 w_q 这个方向"软"(对输出不敏感),量化它代价小;反之代价大。补偿方向 `H⁻¹` 第 q 列,自动把误差导向那些和 w_q 相关、且能低成本吸收误差的通道。**这就是"用二阶信息补偿"的全部含义。**

### 5.4 GPTQ 相对 OBQ 的工程化:任意顺序 + 批量 + Cholesky

OBQ 原版每步选"代价最小的权重"先量化,每次还要更新 H⁻¹,对一层 d×d 权重复杂度高达 `O(d⁴)`,大模型跑不动。GPTQ 做了三个关键工程简化,把它变得可扩展:

```
1. 固定顺序(不再贪心选最小代价):同一层所有输出行,统一按"从左到右"
   逐列量化。理由——所有行共享同一个 H = XᵀX(只取决于输入 X),
   固定顺序让所有行能批量并行处理,H⁻¹ 只需算一次。
   (论文发现:大模型上固定顺序与贪心顺序精度几乎无差,但快几个数量级。)

2. 惰性批量更新(lazy batch update):每量化一列就更新全部权重太碎、
   带宽利用差。改为一次处理一个 block(如 128 列),块内逐列补偿,
   块间一次性批量应用——把内存访问凑成大块,提升吞吐。

3. Cholesky 分解求 H⁻¹:直接对 H⁻¹ 反复行更新数值不稳定(H 接近奇异)。
   先给 H 加阻尼 H ← H + λI(λ 取 H 对角均值的一个小比例,如 1%),
   再做 Cholesky 分解,用分解结果按列顺序取出所需的 H⁻¹ 信息,数值稳定。
```

### 5.5 代码:GPTQ 误差补偿主循环(逐列 + Cholesky)

```python
import torch

def gptq_quantize_layer(W, H, quantizer, blocksize=128, damp_ratio=0.01):
    """
    W: [rows, cols] 待量化权重(rows=out_features, cols=in_features)
    H: [cols, cols] Hessian = XᵀX,由校准激活累加而来(见下)
    quantizer(w_col, ...): 把一列浮点权重四舍五入到量化格点(对称/group-wise)
    返回:量化后权重 Q(已被反量化回浮点,数值落在格点上)
    """
    rows, cols = W.shape
    W = W.clone().float()
    Q = torch.zeros_like(W)

    # --- 1. 阻尼 + Cholesky:得到数值稳定的 H⁻¹ 信息 ---
    damp = damp_ratio * torch.mean(torch.diag(H))
    diag = torch.arange(cols)
    H[diag, diag] += damp                  # H ← H + λI,防奇异
    Hinv = torch.cholesky_inverse(torch.linalg.cholesky(H))  # H⁻¹
    Hinv = torch.linalg.cholesky(Hinv, upper=True)           # 上三角,逐列取系数

    # --- 2. 按 block 逐列量化 + 误差补偿 ---
    for i1 in range(0, cols, blocksize):
        i2 = min(i1 + blocksize, cols)
        W_blk = W[:, i1:i2].clone()
        Q_blk = torch.zeros_like(W_blk)
        Err_blk = torch.zeros_like(W_blk)         # 累积块内补偿误差
        Hinv_blk = Hinv[i1:i2, i1:i2]

        for j in range(i2 - i1):                  # 块内逐列
            w = W_blk[:, j]                        # 第 j 列(所有行)
            d = Hinv_blk[j, j]                     # [H⁻¹]_qq 对应项
            q = quantizer(w)                       # 量化到格点
            Q_blk[:, j] = q
            # 误差贡献:err = (w − q)/d,这是 OBS 更新方向的系数
            err = (w - q) / d
            # 把误差按 H⁻¹ 第 j 列方向,补偿到块内还没量化的右侧列
            W_blk[:, j:] -= err.unsqueeze(1) * Hinv_blk[j, j:].unsqueeze(0)
            Err_blk[:, j] = err

        Q[:, i1:i2] = Q_blk
        # --- 块间一次性批量补偿:把本块累积误差应用到右侧所有未量化列 ---
        W[:, i2:] -= Err_blk @ Hinv[i1:i2, i2:]

    return Q


def accumulate_hessian(layer, calib_loader):
    """用校准数据累加并平均得 H = (2/N)·Σ XᵀX(X 为该层输入激活;因子 2 来自 Hessian=2XᵀX,平均防数值爆掉)"""
    H = None
    n = 0
    for batch in calib_loader:
        X = layer.collect_input(batch)            # [tokens, in_features]
        X = X.float()
        if H is None:
            H = torch.zeros(X.shape[1], X.shape[1], device=X.device)
        # 增量平均,避免数值爆掉(n 累计 token 数)
        H *= n / (n + X.shape[0])
        n += X.shape[0]
        H += (2.0 / n) * (X.T @ X)                # 因子 2 来自 Hessian = 2XᵀX
    return H
```

**复杂度:** 每层 `O(d³)`(Cholesky 分解 `O(d³)` + 误差传播)——相比 OBQ 的 `O(d⁴)` 降了一个数量级,这是 GPTQ 能跑 175B 模型(论文报告单卡数小时量级,具体时长待核)的关键。

**坑:** (1) H 必须用足够的校准 token 累加,样本太少 H 估计不准、补偿反而帮倒忙;(2) damp 太小数值不稳(Cholesky 失败),太大补偿失效(退化成 RTN),1% 对角均值是常见起点;(3) GPTQ **只改权重、不碰激活**,所以它和激活量化(W8A8)正交,可叠加。

## 六、AWQ:激活感知的权重保护

### 6.1 问题:不是所有权重一样重要

AWQ(Activation-aware Weight Quantization)同样做 weight-only(W4A16),但出发点和 GPTQ 不同。它的核心观察:

> **权重的重要性不取决于权重自己的大小,而取决于它要乘的那个激活的大小。**

回到 `Y[i] = Σ_k X[k]·W[i,k]`。对输出贡献大的项,是 `X[k]·W[i,k]` 乘积大的项。如果第 k 个输入通道的激活 `X[k]` 系统性地很大(还记得第四节——激活异常值集中在少数通道),那么**与这个大激活通道相乘的整列权重 `W[:,k]` 就格外关键**——量化它们引入的误差会被大激活放大,直接污染输出。

AWQ 把这些通道叫 **salient(显著)权重通道**。实验发现:**只要保护约 0.1%~1% 的 salient 通道(让它们少受量化损失),就能把量化掉点几乎补回来。** 但"混合精度"(salient 用 FP16、其余 INT4)在硬件上麻烦(不规则、kernel 难写)。AWQ 的巧妙之处:**用一个纯数学等价变换,在全 INT4 的前提下达到"保护 salient 通道"的效果。**

### 6.2 推导:等价缩放——保护通道而不改输出

考虑一个输入通道 j 和它的权重列 `W[:,j]`、激活 `X[:,j]`。引入一个**每通道缩放因子 sⱼ > 0**,做如下等价改写:

```
原始:    Y = Σⱼ X[:,j] · W[:,j]
等价改写:Y = Σⱼ (X[:,j] / sⱼ) · (sⱼ · W[:,j])
              └── 激活缩小 ──┘   └── 权重放大 ──┘
```

这是**严格的代数恒等式**(乘 sⱼ 再除 sⱼ),输出 Y 数学上完全不变。但**被量化的对象变了**:现在量化的是放大后的权重 `sⱼ·W[:,j]`。

**关键机制:对 salient 通道取 sⱼ > 1(放大权重)。** 看它对量化误差的影响——权重列 j 的量化误差 ∝ 它所在 group 的 scale。把 `W[:,j]` 放大 sⱼ 倍,在 per-group 量化里:

```
该权重的相对量化误差 ≈ Δ / |w|,其中 Δ = s_group/2(舍入半台阶)
放大后:w → sⱼ·w,而 group scale s_group 由组内 max 决定
若只放大少数 salient 列、其余不变,组的 max 几乎不动(由非 salient 主导)
⟹ salient 权重的相对误差 ↓ 约 sⱼ 倍(分子 Δ 不变,分母 |w| 放大了 sⱼ 倍)
```

**直觉:** 把重要权重"放大"到 scale 网格里更显眼的位置,它就能被量化得更精确(相对误差变小);代价是激活那边除以 sⱼ,但激活仍是 FP16(W4A16 不量化激活),除法无量化损失,**几乎零代价**。这就是"用很小代价保护 salient 权重"的数学本质——**把权重端的量化精度,从不重要的通道挪一点给重要的通道。**

### 6.3 推导:最优缩放因子怎么定

sⱼ 不能无脑放大:放太大,salient 列自己可能撑大它所在 group 的 max,反而抬高 scale 伤及同组其他权重(回到第四节的异常值陷阱)。AWQ 把它表述为搜索问题:

```
目标:min_s  ‖ X·W − (X·diag(s)⁻¹) · Q(diag(s)·W) ‖²
      Q(·) 是量化-反量化算子
```

直接解析解难求,AWQ 用一个**启发式 + 网格搜索**的实用方案:缩放因子由激活幅度驱动,引入一个标量 α(指数)调节"保护强度":

```
sⱼ = (mean_t |X[t,j]|)^α               (通道 j 的平均激活幅度的 α 次幂)
    或更一般地结合权重幅度:sⱼ ∝ (act_scaleⱼ)^α / (weight_scaleⱼ)^β

α ∈ [0, 1] 用小网格(如 0, 0.1, …, 1)在校准集上搜,
选使上面重构误差最小的 α。
```

```
α = 0   → sⱼ = 1,不缩放,退化成普通 RTN
α = 1   → 完全按激活幅度缩放,salient 通道保护最强,但可能过度
最优 α  → 在两者间,由校准集上的输出重构误差选出
```

**AWQ 只需要激活的"幅度统计"(每通道平均绝对值),不需要 GPTQ 那样的完整 Hessian `XᵀX` 和矩阵求逆。** 所以 AWQ 计算更轻、更快、更稳,且不依赖反向传播。这是它相对 GPTQ 的工程优势。

### 6.4 代码:AWQ 等价缩放(核心片段)

```python
import torch

def awq_search_scale(W, X, quantize_fn, alphas=torch.linspace(0, 1, 11)):
    """
    W: [out, in] 权重;X: [tokens, in] 校准激活
    quantize_fn(w): 对权重做 group-wise 对称量化-反量化,返回 ŵ
    返回:最优逐通道缩放向量 s [in]
    """
    # 每个输入通道的平均激活幅度(salient 信号)
    act_scale = X.abs().mean(dim=0)               # [in]
    ref = X @ W.T                                 # FP16 参考输出
    best_err, best_s = float("inf"), None

    for alpha in alphas:
        s = act_scale.pow(alpha).clamp(min=1e-4)  # [in],salient 通道 s 更大
        s = s / s.mean()                          # 归一化,避免整体偏移
        # 等价变换:权重放大 s,激活缩小 s
        W_scaled = W * s.unsqueeze(0)             # [out, in] 每列乘 sⱼ
        W_hat = quantize_fn(W_scaled)             # 量化放大后的权重
        # 反向把 s 除回去(等价性),用量化后权重重构输出
        Y_hat = (X / s.unsqueeze(0)) @ W_hat.T
        err = (Y_hat - ref).pow(2).mean().item()
        if err < best_err:
            best_err, best_s = err, s

    return best_s   # 部署时:存 Ŵ=Q(W·s) 和 1/s,激活先乘 1/s 再做 W4A16 matmul
```

**部署形态:** 缩放因子 `1/s` 可以**吸收进前一层**(比如把 `1/s` 合并到上一层的 LayerNorm 权重或前一个线性层里),这样推理时**不增加任何额外算子**——激活进来时已经是缩放好的。**这点很重要:AWQ 的等价变换在部署期是"免费"的,不像混合精度要特殊 kernel。**

## 七、SmoothQuant:把量化难度从激活迁移到权重

### 7.1 问题:W8A8 卡在激活异常值上

GPTQ/AWQ 都是 weight-only(激活不量化),服务 decode。但在 **compute-bound 场景**(prefill、大 batch),我们想要 **W8A8**——权重激活都 INT8,用 INT8 tensor core 把算力吃满。问题卡在第四节那个事实上:**激活有逐通道异常值,而激活的高效量化粒度只能 per-token / per-tensor(2.3 节),管不住逐通道异常值。**

```
矛盾的精确表述:
  · 异常值沿"输入通道(feature)"方向分布(per-channel 才能精确量化)
  · 但激活的高效量化只能 per-token(沿 token 方向)或 per-tensor
  · per-token 的 scale 对每个 token 取该行的 max —— 异常值通道仍会撑大这一行的 scale
  ⟹ 激活直接量化到 INT8,异常值导致掉点严重(LLM.int8 的发现)
```

LLM.int8 的解法是把异常值挑出来用 FP16 单算(混合精度),但这破坏了规整的 INT8 GEMM,有性能损耗。**SmoothQuant 想要纯 INT8、规整 GEMM,同时不掉点。**

### 7.2 推导:难度迁移的等价变换

SmoothQuant 用和 AWQ 形式上一样的等价缩放,但**目的相反**。AWQ 是放大 salient 权重(保护权重);SmoothQuant 是**缩小激活异常值通道**(让激活好量化),代价是放大对应权重(让权重难一点):

```
等价:Y = (X · diag(s)⁻¹) · (diag(s) · W)
          └ 激活除以 s ┘   └ 权重乘以 s ┘

对异常值通道 j:取 sⱼ > 1
  ⟹ 激活 X[:,j]/sⱼ 变小(异常值被压平,激活变好量化)
  ⟹ 权重 sⱼ·W[:,j] 变大(权重难一点,但权重本来分布温和、好量化)
```

**核心思想:激活难量化、权重好量化,所以把难度从激活"迁移"一部分到权重,让两边都落在 INT8 能舒服处理的范围。** 这就是名字 "Smooth"(抹平)的由来——把激活那些尖锐的异常值峰抹平,代价分摊到权重上。

### 7.3 推导:迁移强度 α 怎么定

迁移多少?全迁移(把激活异常值完全压平)会让权重那边爆掉。SmoothQuant 用一个**迁移强度超参 α ∈ [0,1]**,让缩放因子在激活幅度和权重幅度之间做几何插值:

```
sⱼ = (max_t |X[t,j]|)^α / (max_i |W[i,j]|)^(1−α)        ……(迁移公式)

α 的含义:
  α = 0   → sⱼ = 1/max|W[:,j]| —— 难度全压在激活(不迁移到权重),即不平滑
  α = 1   → sⱼ = max|X[:,j]|   —— 把激活压到幅度 1,难度几乎全甩给权重
  α ≈ 0.5 → 激活和权重的量化难度大致均衡(典型默认值)
```

**为什么是这个几何平均形式?** 看迁移后两边的幅度:

```
迁移后激活通道 j 的幅度:max|X[:,j]| / sⱼ = max|X[:,j]|^(1−α) · max|W[:,j]|^(1−α)
迁移后权重通道 j 的幅度:max|W[:,j]| · sⱼ = max|X[:,j]|^α · max|W[:,j]|^α
```

```
当 α = 0.5:
  激活幅度 → (max|X| · max|W|)^0.5
  权重幅度 → (max|X| · max|W|)^0.5
  两者相等!—— 量化难度在激活和权重间被均分(几何平均)
```

**这就是 α 的几何意义:它是难度在两端的分配比例。** 异常值越极端的层,可能需要调大 α(多往权重迁),但 α 太大权重也会出现异常值。实践中按层校准选 α(常用 0.5 起步)。

### 7.4 代码:SmoothQuant 平滑(核心片段)

```python
import torch

@torch.no_grad()
def smooth_layer(act_scales, weight, alpha=0.5):
    """
    act_scales: [in] 校准集统计的每输入通道激活最大绝对值 max_t|X[t,j]|
    weight: [out, in] 权重(将被迁移放大)
    返回:逐通道缩放向量 s [in],用于把激活除以 s、权重乘以 s
    """
    w_scales = weight.abs().max(dim=0).values        # [in] 每输入通道权重 max
    # 迁移公式:s = act^α / weight^(1−α)
    s = (act_scales.pow(alpha) / w_scales.pow(1 - alpha)).clamp(min=1e-5)
    return s

@torch.no_grad()
def apply_smooth(prev_ln_or_linear, weight, s):
    """
    把缩放等价地折叠进网络,推理期零额外算子:
      · 激活除以 s  → 折叠进上游(LayerNorm 的 weight/bias 或上一线性层)除以 s
      · 权重乘以 s  → 当前层权重每列乘 s
    """
    prev_ln_or_linear.weight.div_(s)                 # 上游输出 = 激活 ÷ s
    if prev_ln_or_linear.bias is not None:
        prev_ln_or_linear.bias.div_(s)
    weight.mul_(s.unsqueeze(0))                       # 当前权重 × s(每列)
    # 此后:激活、权重都已 smooth,可分别做 per-token / per-channel INT8 量化
```

**部署关键(和 AWQ 同理):** `÷s` 折叠进上游 LayerNorm / 线性层,`×s` 吸收进当前权重——**推理时不增加任何运行时算子,纯离线变换。** 之后激活做 per-token INT8、权重做 per-channel INT8,送进规整 INT8 GEMM。这就是 SmoothQuant 让 W8A8 可行的完整链路。

## 八、三法对比:同一问题的三个侧面

把三种方法钉在统一框架里(都在"固定比特预算下压低有效量化误差"):

```
┌──────────────┬────────────────┬──────────────┬─────────────────┐
│              │ GPTQ           │ AWQ          │ SmoothQuant     │
├──────────────┼────────────────┼──────────────┼─────────────────┤
│ 量化什么      │ 权重(W4A16)   │ 权重(W4A16) │ 权重+激活(W8A8)│
│ 打哪个瓶颈    │ decode 带宽    │ decode 带宽  │ prefill/大batch │
│              │ (memory-bound) │ (同左)       │ 算力(compute)  │
│ 核心数学      │ 二阶 Hessian   │ 激活幅度统计 │ 等价缩放迁移难度│
│              │ 误差补偿(OBS) │ +等价缩放    │                 │
│ 需要的信息    │ 完整 XᵀX +求逆 │ 每通道平均幅 │ 每通道激活/权重 │
│              │                │ 度(轻)      │ max(轻)        │
│ 对付异常值方式│ 间接(补偿吸收)│ 保护salient权│ 主动抹平激活    │
│              │                │ 重通道       │ 异常值          │
│ 部署额外算子  │ 无(只换权重)  │ 无(s折叠)   │ 无(s折叠)      │
│ 计算成本      │ 中(求逆)      │ 低           │ 低              │
└──────────────┴────────────────┴──────────────┴─────────────────┘
```

**关键认知三条:**

1. **GPTQ 和 AWQ 不是竞争,可叠加。** 都做 W4A16,一个靠补偿、一个靠保护,实践中常组合(或在 AWQ 的等价变换后再跑 GPTQ 补偿)。它们瞄准同一瓶颈(decode 带宽)。
2. **SmoothQuant 和它俩瞄准不同瓶颈。** SmoothQuant 是为了让 W8A8 在 compute-bound 场景可行;GPTQ/AWQ 是为了让 W4A16 在 memory-bound 场景精度够。**选哪个,先看你打的是带宽还是算力**(回 3.3 节的分工)。
3. **AWQ 和 SmoothQuant 用的是同一个等价缩放工具,方向相反。** AWQ 放大 salient 权重(保护权重);SmoothQuant 缩小异常激活(代价是放大权重)。**同一把数学锤子,砸不同的钉子。** 想透这一点,你对"等价缩放"这个原语就吃透了。

## 九、KV cache 量化:长上下文的显存救星

前面量化的是权重(静态)。长上下文场景下,**KV cache 才是显存大头**(上一章推过它随序列长度线性增长)。把 KV cache 也量化,是缓解长上下文显存的关键手段。

### 9.1 为什么 KV cache 量化和权重量化不同

```
权重量化:静态、离线、量化一次永久用 —— 可以慢慢优化(GPTQ 求逆都行)
KV cache 量化:动态、在线、每生成一个 token 就新增一份 K/V 要量化/反量化
            —— 必须极轻量,量化/反量化开销不能吃掉省下的带宽
```

所以 KV cache 量化通常用**最简单的对称 per-token(或 per-head)量化**到 INT8 / FP8,在写入 cache 时量化、读出做注意力时反量化。FP8(如 E4M3)在这里很受欢迎,因为它动态范围大、对异常值更宽容,且新硬件有原生支持(具体格式/硬件支持待核于你的目标平台)。

### 9.2 量化:KV cache INT8 量化省多少

复用上一章 KV cache 显存公式骨架。设层数 L、KV 头数 h_kv、头维 d、序列长 S、batch B,每元素字节数 `bytes`:

```
KV cache 字节 = 2(K和V) · L · h_kv · d · S · B · bytes
```

```
FP16 → INT8:bytes 从 2 → 1,KV cache 显存直接减半(2×)
FP16 → INT4:bytes 从 2 → 0.5,减到 1/4(4×),但 INT4 KV 精度风险更高
```

**例:** L=32、h_kv=8(GQA)、d=128、S=8192、B=128、FP16:

```
KV = 2(K和V) × 32 × 8 × 128 × 8192 × 128 × 2 字节
   ≈ 1.37 × 10¹¹ 字节 ≈ 137 GB（FP16;此处 GB 按十进制 10⁹ 计,等于 128 GiB）
量化到 INT8 → ≈ 68 GB；INT4 → ≈ 34 GB
（注:把 B 降到 16,显存按比例降到约 17 GB——KV cache 随 batch 线性增长,
  大并发服务下它就是几十上百 GB 的量级)
```

**长上下文 + 大 batch 下,KV cache 量化能腾出几十上百 GB 显存**,直接决定你能不能在固定显存上服务更长的上下文或更大的并发。这是它的核心价值。

### 9.3 坑

```
1. K 比 V 更难量化:实践观察 Key cache 的异常值通常比 Value 更显著
   (与 RoPE 后的分布有关),有方案对 K 用更细粒度或更高比特。
2. 反量化开销:decode 每步都要反量化整个 KV cache 参与注意力,
   若 cache 巨大,反量化的访存可能反噬——要确认净收益。
3. 量化点选择:per-token 量化要在写入时定 scale,与 PagedAttention
   的分块存储要对齐(每块/每 token 存自己的 scale)。
```

## 十、设计权衡与常见坑(汇总)

```
1. 选 W4A16 还是 W8A8?—— 先问瓶颈。decode/延迟敏感/单序列 → W4A16
   (省带宽);prefill/大batch/吞吐 → W8A8(吃算力)。搞反了优化打不到点上。

2. group size 的权衡:g 越小精度越好但存储税 b_eff = b + 16/g 越高。
   g=128 是甜点(+3%)。盲目调到 g=32 换来的精度可能不值 12.5% 的体积。

3. 对称 vs 非对称:weight-only 几乎都对称(z=0 省交叉项、kernel 干净);
   激活(尤其 ReLU/GELU 后的非负分布)用非对称更紧凑。别一刀切。

4. 异常值是一切的根:没意识到"少数值撑大 scale 毁全组"这条,
   就理解不了三种方法。所有方法本质都在处理"max|x| 被异常值顶高"。

5. 校准集要有代表性:PTQ 全靠校准集统计(GPTQ 的 H、AWQ/SmoothQuant
   的激活幅度)。校准集分布偏离真实负载,量化参数就偏。但也别太多——
   几百条通常够,过多收益递减。

6. 数值稳定坑(GPTQ 专属):H 加阻尼 λI 是必须的,Cholesky 才不炸;
   λ 太小不稳、太大补偿失效。

7. "几乎不掉点"是有边界的:b=4 通常稳;b=3、b=2 会明显掉点,
   需要更激进的方法(group 更细 / 混合精度 / QAT)。别外推过头。

8. 反量化不是免费午餐——只在 memory-bound 时免费:decode 算力闲置,
   反量化白用;但 compute-bound 场景下反量化的 FLOP 要算进成本。

9. 等价缩放的折叠:AWQ/SmoothQuant 的 s 必须能折叠进上游(LayerNorm/
   线性层),否则推理期多一个逐通道乘法算子,白白吃带宽。设计时确认
   上游结构允许折叠(有的归一化层结构会让折叠不成立——要核对)。
```

## 十一、动手练习

**练习 1(推导题,核心)。** 不看正文,独立推导 group 量化的摊薄比特公式 `b_eff = b + 16/g`。设权重矩阵 `[M,K]`、group 大小 g、量化比特 b、scale 用 FP16 存、对称量化。要求写出"整数本体字节数""scale 总字节数""两者相加除以权重数换算成比特"三步。然后回答:若 scale 改用 FP32 存,公式变成什么?若每组还要存一个 INT8 的 zero-point(非对称),再加哪一项?
- 提示:scale 个数 = M·(K/g);FP16 是 2 字节/scale。换算成比特要乘 8 再除以权重数 MK。zero-point 项形如 `8·(M·K/g)/(MK) = 8/g`。

**练习 2(推导题,核心)。** 独立推导 GPTQ/OBS 的两条公式:已知二次损失 `E(δ) = δ H δᵀ`,约束第 q 维 `δ_q` 固定,用拉格朗日乘子法证明最优补偿 `δᵀ = (δ_q/[H⁻¹]_qq)·H⁻¹e_q`,以及损失增量 `ΔE_q = δ_q²/[H⁻¹]_qq`。
- 提示:`L = δHδᵀ − λ(e_qᵀδᵀ − δ_q)`,对 δ 求导置零得 `δᵀ = (λ/2)H⁻¹e_q`,代入约束 `e_qᵀδᵀ=δ_q` 解 λ;`H⁻¹e_q` 就是 H⁻¹ 的第 q 列,`e_qᵀH⁻¹e_q = [H⁻¹]_qq`。

**练习 3(估算题)。** 用第三节方法估算 W4A16 在 decode 上的加速:一个 7B 模型(N≈7×10⁹ 参数),(a) FP16 与 INT4 各读一遍权重的字节数;(b) 假设 decode 纯 memory-bound 且墙钟 ∝ 读权重字节数,理论加速比是多少?(c) 现实中达不到 4×,列举三个吃掉理论加速的因素;(d) 用第二节 `b_eff = b + 16/g`(g=128)修正(a)的 INT4 字节数,加速比变成多少?
- 提示:(a) FP16=2N=1.4×10¹⁰ 字节;INT4=0.5N。(b) 2N/0.5N=4×。(c) 反量化算力(memory-bound 下小)、scale 读取、KV cache 读取占比、激活仍 FP16、kernel 未必打满带宽。(d) INT4 实际 b_eff=4.125 比特 → 字节数 ×(4.125/4),加速 ≈ 4/(4.125/4) ≈ 3.88×。

**练习 4(分析题)。** 解释为什么异常值是量化的死敌,并对比三种方法对付它的方式。要求:(a) 用误差公式 `误差∝s`、`s∝max|x|` 定量说明一个异常值如何毁掉同组的正常值(算一个 g=128、含一个 100 倍异常值的具体例子);(b) 说明 GPTQ(补偿)、AWQ(保护 salient 权重)、SmoothQuant(抹平激活异常)各从哪个角度化解;(c) 论证为什么"激活异常值"比"权重异常值"更棘手(提示:用 2.3 节激活量化粒度受矩阵乘累加结构限制这一点)。
- 提示:(c) 权重可以 per-channel/per-group 自由切分隔离异常通道;激活的高效量化只能 per-token/per-tensor,无法沿 feature 通道隔离异常,所以必须靠 SmoothQuant 这种等价迁移先把异常压平。

## 十二、源码 / 论文导读

- **OBS/OBD 起源(GPTQ 的数学根):** LeCun et al., *Optimal Brain Damage*(1989)和 Hassibi & Stork, *Optimal Brain Surgeon*(1992)。重点看 OBS 用 Hessian 逆做"删一个权重 + 补偿其余"的闭式解——GPTQ 5.3 的两条公式就是它的量化版变体。
- **GPTQ:** Frantar, Ashkboos, Hoefler, Alistarh, *GPTQ: Accurate Post-Training Quantization for Generative Pre-trained Transformers*(2022/2023)。重点读它的算法框(逐列量化 + 块更新 + Cholesky 那一节)和与 OBQ 的对比;`H=XᵀX`、误差补偿更新式与本章 5.2–5.5 一一对应。开源实现看 `AutoGPTQ` / `GPTQModel` 仓库的 `quant` 相关模块(Cholesky、block update 的实现);具体文件名/API 待核于你手头版本。
- **AWQ:** Lin et al., *AWQ: Activation-aware Weight Quantization for LLM Compression and Acceleration*(MIT, 2023)。重点读"salient 权重由激活幅度决定(不是权重幅度)"的实验,以及等价缩放 + α 网格搜索那一节,对应本章 6.2–6.3。开源 `llm-awq` / `AutoAWQ` 仓库看 scale 搜索和折叠进上游的实现。
- **SmoothQuant:** Xiao et al., *SmoothQuant: Accurate and Efficient Post-Training Quantization for Large Language Models*(MIT, 2022/2023)。重点读迁移公式 `s = act^α / weight^(1−α)` 的推导和 α 的几何平均直觉(本章 7.2–7.3),以及它怎么折叠进 LayerNorm。
- **激活异常值的实证源头:** Dettmers et al., *LLM.int8(): 8-bit Matrix Multiplication for Transformers at Scale*(2022)。重点看它对"系统性异常值集中在少数通道、约 6.7B 起出现"的实证——第四节那三条关键事实就来自这条线的工作。
- **工业部署:** vLLM / TensorRT-LLM 对 GPTQ/AWQ 的支持(搜仓库里 `gptq`、`awq`、`marlin` kernel 相关目录——Marlin 是高效的 W4A16 GEMM kernel)。重点看反量化怎么和 GEMM 融合进一个 kernel(对应 3.2 节"读 INT4 → 片上反量化 → matmul"的链路);具体路径/kernel 名待核于版本。

## 十三、小结与承上启下

这一章的三层骨架:**(1) 物理层** —— decode 是 memory-bound(算术强度≈1≪脊点),把权重从 16 比特做小到 4 比特,直接把"读一遍权重"的字节数砍 4 倍,W4A16 理论加速 ≈4×,且反量化的算力开销在闲置算力里白用;**(2) 数学层** —— 量化误差 ∝ scale ∝ max|x|,所以一切优化都是"在固定比特预算下压低有效 scale",而异常值(撑大 max|x|)是共同的死敌;**(3) 方法层** —— GPTQ 用 Hessian `XᵀX` 做 OBS 误差补偿(`ΔE=δ²/[H⁻¹]_qq`)、AWQ 用等价缩放放大 salient 权重通道(代价≈0)、SmoothQuant 用同一个等价缩放反向把激活异常值的难度迁移给权重(`s=act^α/weight^(1−α)`)使 W8A8 可行。把这三层钉死,你就能对任何一个量化方案回答:它打的是带宽还是算力?它怎么对付异常值?它的有效比特和精度代价各是多少?

在整门课里的位置:上两章打的是 decode 这个 memory-bound 瓶颈的不同侧面——KV 分页(第 10 章,省显存)、投机解码(第 11 章,一次搬权重服务多 token、摊薄前向),本章(第 12 章,量化)是第三件套:**直接缩小权重和 KV cache 的字节数,从根上减少要搬的数据。** 这三章合起来,正是上一章结尾预告的"现代低延迟推理引擎三件套":**省显存(分页)、摊薄前向(投机)、缩小数据(量化)。** 三者正交、可叠加——一个真实的推理引擎(vLLM/TensorRT-LLM)会同时开 PagedAttention + 投机解码 + W4A16 量化 + KV cache 量化。学完这三章,你看任何推理系统的优化清单,都能立刻定位:这条优化打的是显存、前向次数、还是搬运字节数——以及它们怎么协同把 decode 的成本压到最低。下一章起,课程会从"单机/单模型推理优化"转向更大尺度的系统议题。
