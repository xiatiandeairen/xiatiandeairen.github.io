## 1. 动机:大模型缺的不是"会说",而是"知道并能交代来源"

把一个大语言模型(LLM)想成一个读了海量书、但**毕业后再没读过新书、答题不许查资料、还不许说"我不知道"**的学生。它的参数 θ 是一次性烧进去的压缩知识,这带来四个结构性缺陷,而且都不是把模型做大就能根治的:

- **幻觉(hallucination)**。语言模型的训练目标是最大化下一个 token 的似然 `P(xₜ | x_{<t})`,优化的是**流畅与统计合理**,不是**真实**。一个流畅且符合语料统计的句子,可以完全是编的。模型没有内建的"我对这条事实有几成把握"的信号,所以会以同样自信的口吻说出真话和假话。(幻觉的成因与类型,在第 13 章评测、第 14 章安全里还会从 NLP 视角细谈。)
- **知识过时**。参数烧录于某个截止日期,之后世界变了它不知道。"现任 X 是谁""某库最新 API"这类时效问题,靠重训整个模型来更新成本极高。
- **长尾知识缺失**。预训练语料里出现几千次的事实模型记得牢,只出现两三次的(你公司内部文档、某篇冷门论文的某个数字)几乎学不进参数——这是压缩的必然代价。
- **无法引用来源**。即使答对了,它也说不清"这句话出自哪份文档第几段"。在医疗、法律、企业问答里,**不可溯源 = 不可用**。

**RAG(Retrieval-Augmented Generation,检索增强生成)的核心主张:不要试图把所有知识塞进参数,而是把知识放在一个可随时更新、可检索、可引用的外部库里;回答问题时先检索出相关片段,再让模型"基于这些片段"生成答案。** 一句话:

```
朴素 LLM:   answer = LLM(question)
RAG:        docs   = Retriever(question, Corpus)        # 先检索 top-k 证据
           answer = LLM(question, docs)                # 再基于证据生成
```

这个简单改动一次性缓解了四个缺陷:幻觉——答案有证据约束;过时——更新文档库即可,不必重训;长尾——库里有就能查到;来源——直接给出引用片段。

**但务必记住一个贯穿全章的真相:RAG 是缓解不是根治。** 检索器可能召回错的或无关的文档;模型即使拿到正确文档,仍可能无视证据、按参数里的旧记忆作答("检索了但仍幻觉")。本章最后的评测一节,正是为了量化这件事。

RAG 由 Lewis 等人在 2020 年正式提出(论文 *Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks*),但它的两半——"信息检索(IR)"和"条件生成"——各自有几十年积累。我们先把检索讲透,再讲生成融合,最后讲评测。

**本章的依赖关系**:检索器的稠密一半(DPR)直接建立在第 06 章的句向量(Sentence-BERT 双塔 + 对比学习)之上,本章会复述其对比损失的推导以保证自洽;生成器内部的注意力、解码细节属于算法课,本章只在"信息怎么流进解码器"这一层展开,不重复 Transformer 推导。

## 2. 检索器之一:稀疏检索 BM25——把"相关"写成一个可解释的打分公式

### 2.1 问题:给定 query,如何从百万文档里排出最相关的几篇?

最朴素的想法:query 和文档共享的词越多越相关。但直接数共享词数有三个明显毛病,BM25 恰好就是为了修这三个毛病而设计的——理解 BM25 的最好方式不是背公式,而是看它如何逐项打补丁。

设 query `q` 含若干词项 `t`,文档 `d`,词项 `t` 在 `d` 中出现次数(term frequency)记 `f(t,d)`。

**毛病一:高频词不该线性加分。** 一个词在文档里出现 1 次 vs 100 次,相关性确实更强,但不该强 100 倍——出现 10 次后基本就"够相关了"。需要一个**随次数增长但饱和(saturating)**的函数。

**毛病二:常见词不该和稀有词同权。** "the""的"几乎每篇都有,匹配上不说明什么;"维特比算法"很稀有,匹配上信息量极大。需要按词的**稀有度**加权,这就是 IDF(inverse document frequency,逆文档频率)。

**毛病三:长文档不该占便宜。** 文档越长,任何词都更可能多次出现,纯靠堆字数就能蹭高分。需要**文档长度归一**,把长文档的 TF 优势压回去。

### 2.2 推导:从三个补丁拼出 BM25

**IDF 项**。一个词项在越少的文档里出现,越有区分力。设文档总数 `N`,含词项 `t` 的文档数 `n(t)`。BM25 用的(Robertson-Spärck Jones)IDF 形式:

```
IDF(t) = ln( (N − n(t) + 0.5) / (n(t) + 0.5) )
```

直觉:`n(t)` 小(稀有词)→ 分子大分母小 → IDF 大;`n(t)` 接近 `N`(几乎到处都有)→ 趋近 0 甚至略小于 0。那个 `+0.5` 是平滑,避免 `n(t)=0` 或 `=N` 时除零/取 ln(0)。它源自一个二元独立性概率模型下"该词出现在相关文档 vs 不相关文档的对数几率比"的估计——你不必记推导,记住**它是稀有度的对数加权**即可。(这和第 01 章 TF-IDF 里的 IDF 是同一个思想的不同变体。)

**TF 饱和项**。要一个从 0 单调增、但有上界、且增速递减的函数。BM25 选了这个形式(其中 `k₁` 是超参,典型取 1.2~2.0):

```
              f(t,d) · (k₁ + 1)
TF_sat(t,d) = ──────────────────
              f(t,d) + k₁
```

验算它确实满足要求:
```
f=0   → 0·(k₁+1)/(0+k₁) = 0                      # 没出现,不加分
f→∞   → f·(k₁+1)/f      = k₁+1                    # 上界 k₁+1,饱和
导数随 f 增大而减小                                 # 边际递减
```
`(k₁+1)` 这个系数只是把上界缩放到 `k₁+1`,不影响排序(对所有词项同比例缩放),写进去是为了让 `f=1` 附近的取值更直观。`k₁` 越大,饱和来得越慢(越接近线性计数);`k₁→0` 则任何 `f≥1` 都几乎记 1 分(变成"出现与否"的布尔检索)。

**文档长度归一**。设文档 `d` 长度(词数)`|d|`,语料平均文档长度 `avgdl`。把分母里的 `k₁` 替换成一个随文档相对长度调节的量,引入超参 `b ∈ [0,1]`:

```
              f(t,d) · (k₁ + 1)
score = ───────────────────────────────────────
        f(t,d) + k₁ · (1 − b + b · |d|/avgdl)
```

看这个调节因子 `(1 − b + b·|d|/avgdl)`:
```
b=0           → 因子 = 1,完全不做长度归一(退化回纯 TF 饱和)
b=1           → 因子 = |d|/avgdl,完全按相对长度归一
文档比平均长   → |d|/avgdl > 1 → 分母变大 → 同样的 f 得分被压低
文档比平均短   → |d|/avgdl < 1 → 分母变小 → 同样的 f 得分被抬高
```
这正好实现"长文档别占便宜"。`b` 典型取 0.75,即做大部分归一但不极端。

**合起来——完整 BM25 打分公式**:

```
                    Σ            f(t,d) · (k₁ + 1)
BM25(q,d) =        t∈q   IDF(t)·─────────────────────────────────
                              f(t,d) + k₁·(1 − b + b·|d|/avgdl)
```

逐项复述一遍它在干什么:对 query 里每个词项,算它的**稀有度权重 IDF**,乘以它在文档里的**饱和后词频**(并按文档长度归一),再把所有词项加起来。三个补丁、三个超参 `(k₁, b, 还有平滑常数)`,缺一不可。

### 2.3 代码:从零实现 BM25 打分

```python
import math
from collections import Counter

class BM25:
    def __init__(self, corpus_tokens, k1=1.5, b=0.75):
        # corpus_tokens: List[List[str]],每篇文档是分好词的 token 列表
        self.k1, self.b = k1, b
        self.docs = corpus_tokens
        self.N = len(corpus_tokens)
        self.doc_len = [len(d) for d in corpus_tokens]
        self.avgdl = sum(self.doc_len) / self.N
        self.tf = [Counter(d) for d in corpus_tokens]      # 每篇文档的词频表

        # 文档频率 n(t):含词项 t 的文档数
        df = Counter()
        for d in corpus_tokens:
            for t in set(d):           # 用 set,一篇文档对一个词只计一次
                df[t] += 1
        # 预计算每个词项的 IDF
        self.idf = {
            t: math.log((self.N - n + 0.5) / (n + 0.5) + 1)   # +1 防 IDF 变负
            for t, n in df.items()
        }

    def score(self, query_tokens, doc_id):
        s = 0.0
        f_d = self.tf[doc_id]
        len_d = self.doc_len[doc_id]
        for t in query_tokens:
            if t not in f_d:
                continue                              # 词项没出现,贡献 0
            freq = f_d[t]
            idf = self.idf.get(t, 0.0)
            denom = freq + self.k1 * (1 - self.b + self.b * len_d / self.avgdl)
            s += idf * freq * (self.k1 + 1) / denom
        return s

    def topk(self, query_tokens, k=5):
        scored = [(i, self.score(query_tokens, i)) for i in range(self.N)]
        return sorted(scored, key=lambda x: -x[1])[:k]
```

注意 `+1` 那个微调:Lucene/Elasticsearch 用的工业版 IDF 是 `ln(1 + (N−n+0.5)/(n+0.5))`,加 1 保证 IDF 恒为正、避免极常见词产生负分干扰排序。这种工程细节在生产系统里很关键。

### 2.4 分析:BM25 的本质优势与天花板

**复杂度**。检索时不必扫全部文档:用**倒排索引(inverted index)**,对 query 里每个词项查"哪些文档含它"的 posting list,只给候选文档打分。若 query 有 `m` 个词项、每个词项平均命中 `L` 篇文档,代价约 `O(m·L)`,与语料总量 `N` 几乎无关——这是 BM25 三十年屹立不倒的工程根基。

**优点**:零训练、可解释(能逐词解释为什么这篇排前面)、对**精确词匹配和稀有词(实体名、错误码、专有名词)极强**、增量更新文档只需改倒排索引。

**天花板:词汇鸿沟(lexical gap)**。BM25 本质是词面匹配。query 说"心脏病发作",文档写"心肌梗死",共享词为零 → BM25 判为不相关,尽管语义完全一致。它不懂同义、不懂改写、不懂跨语言。这正是稠密检索要补的洞。

## 3. 检索器之二:稠密检索 DPR——把语义匹配变成向量内积

### 3.1 问题:如何让"心脏病发作"和"心肌梗死"在表示空间里靠近?

思路:**别在词面匹配,在语义向量空间里匹配**。把 query 和文档各自编码成一个稠密向量(几百维实数),让"语义相关"的 query-文档对在向量空间里内积大、"不相关"的内积小。这就是 DPR(Dense Passage Retrieval,Karpukhin 等,2020)的核心。

它的结构和第 06 章的 Sentence-BERT 一脉相承——**双塔(dual-encoder / two-tower)**:

```
query  q  ──► E_Q(·)  (一个 BERT) ──► 向量 u ∈ ℝ^d
passage p ──► E_P(·)  (另一个 BERT) ──► 向量 v ∈ ℝ^d
相似度  sim(q,p) = u · v          # 内积(或先 L2 归一化再内积 = 余弦)
```

**为什么是双塔而不是单塔?** 单塔(把 query 和 passage 拼起来一起过一个 BERT)叫 cross-encoder,它让二者每一层都互相注意,精度更高;但它**无法预计算**——每来一个 query,都要和每篇文档现拼现算,百万文档就是百万次 BERT 前向,在线检索根本跑不动。双塔的关键工程价值在于:**所有 passage 向量可以离线预先算好、存进向量索引;在线时只需编码 query 一次,然后做向量检索。** 这是"能不能上线"的分水岭。(cross-encoder 我们留到第 5 节做重排序时再请回来——它精度高但只能用在小候选集上。)

### 3.2 推导:对比学习与 in-batch negatives 的 InfoNCE 损失

怎么训练这两个塔,让相关对内积大、不相关对内积小?这是**对比学习(contrastive learning)**:给定一个 query `qᵢ`,我们有一个已知相关的正样本 passage `pᵢ⁺`(比如该问题的标准答案出处),还需要若干**负样本** `p⁻`(不相关的 passage)。目标是让 `sim(qᵢ, pᵢ⁺)` 显著高于所有 `sim(qᵢ, p⁻)`。

把"从一堆候选里挑出正确那个"建模成一个分类问题:正样本是正确类别,负样本是错误类别。用 softmax 归一化相似度,最大化正样本的概率——这就是 **InfoNCE 损失**(Oord 等 2018 提出,DPR 沿用)。对单个 query `qᵢ`,设它的候选集合是 1 个正样本加 n 个负样本:

```
                       exp( sim(qᵢ, pᵢ⁺) / τ )
P(pᵢ⁺ | qᵢ) = ──────────────────────────────────────────────────
              exp( sim(qᵢ,pᵢ⁺)/τ ) + Σⱼ exp( sim(qᵢ,pⱼ⁻)/τ )

L_i = − log P(pᵢ⁺ | qᵢ)
```

`τ`(temperature,温度)是缩放超参:τ 小 → 分布更尖锐,对"正样本必须远超负样本"的要求更狠;τ 大 → 更平滑。最小化 `L_i` 等价于:把分子(正样本相似度)往上推、把分母里那些负样本相似度往下压。这就是我们要的"拉近正对、推远负对"。

**关键技巧:in-batch negatives(批内负样本)。** 训练时手工给每个 query 找负样本既贵又难找到"难负例"。DPR 的妙招:**把同一个 mini-batch 里其他 query 的正样本,直接当成当前 query 的负样本。** 一个 batch 有 B 个 `(qᵢ, pᵢ⁺)` 对,对 query `qᵢ` 来说,`pᵢ⁺` 是正样本,其余 B−1 个 `pⱼ⁺ (j≠i)` 全是负样本——因为别人问题的答案,大概率和我的问题不相关。

这一招的精妙之处全在效率:
```
朴素做法: B 个 query,每个额外采 n 个负样本 → 要编码 B + B·n 个 passage
in-batch: B 个 query 共享同一批 B 个 passage 当负样本池 → 只编码 B 个 passage
          相当于"零额外成本"就得到了 B−1 个负样本/query
```
把整个 batch 的相似度排成一个 `B×B` 矩阵 `S`,`S[i][j] = sim(qᵢ, pⱼ⁺)`。那么**对角线是所有正对,非对角线是所有 in-batch 负对**。损失就变成:在 `S` 的每一行上做 softmax,标签是"对角线那个位置"——这正是一个标准的多分类交叉熵,标签为 `[0,1,2,...,B−1]`(第 i 行的正确类是第 i 列)。

### 3.3 代码:DPR 双塔对比训练(in-batch negatives 的矩阵实现)

```python
import torch
import torch.nn.functional as F
from transformers import AutoModel, AutoTokenizer

class Encoder(torch.nn.Module):
    """一个塔:BERT + 取 [CLS] 作句向量(DPR 原文用 [CLS])。"""
    def __init__(self, name="bert-base-uncased"):
        super().__init__()
        self.bert = AutoModel.from_pretrained(name)
    def forward(self, input_ids, attention_mask):
        out = self.bert(input_ids=input_ids, attention_mask=attention_mask)
        return out.last_hidden_state[:, 0]          # [CLS] 向量,形状 [B, d]

def dpr_loss(q_vecs, p_vecs, temperature=1.0):
    """
    q_vecs: [B, d] 一个 batch 的 query 向量
    p_vecs: [B, d] 对应的正样本 passage 向量
    in-batch negatives:第 i 个 query 的正样本是第 i 个 passage,其余 B-1 个是负样本
    """
    # 相似度矩阵 S[i][j] = q_i · p_j,形状 [B, B]
    scores = q_vecs @ p_vecs.t() / temperature
    # 对角线是正对 → 第 i 行的正确类别就是 i
    labels = torch.arange(q_vecs.size(0), device=q_vecs.device)
    # 在每一行做 softmax 交叉熵 = InfoNCE
    return F.cross_entropy(scores, labels)

# ---- 训练一步 ----
q_enc, p_enc = Encoder(), Encoder()             # 两个独立的塔(也可共享权重)
opt = torch.optim.AdamW(list(q_enc.parameters()) + list(p_enc.parameters()), lr=1e-5)

# 假设 q_batch / p_batch 已 tokenize 成 input_ids/attention_mask
u = q_enc(q_batch.input_ids, q_batch.attention_mask)   # [B, d]
v = p_enc(p_batch.input_ids, p_batch.attention_mask)   # [B, d]
# 若要用余弦相似度,这里先归一化:u = F.normalize(u, dim=-1); v = F.normalize(v, dim=-1)
loss = dpr_loss(u, v, temperature=1.0)
loss.backward(); opt.step(); opt.zero_grad()
```

**实现要点(新手最常踩的坑)**:
- batch size 直接决定负样本数量,B 越大对比信号越强、检索效果越好——这是 DPR/对比学习"吃显存换效果"的根源。工业训练常用很大的 batch,甚至跨多卡用 **gather** 把所有卡的 passage 都收进负样本池。
- 光有 in-batch 负样本(随机负例,大多很"软"、一眼就不相关)往往不够。DPR 原文额外加了 **hard negatives(难负例)**:用 BM25 检索出"词面像但其实答非所问"的 passage 当负样本,逼模型学会更细的语义区分。这一步对最终精度提升很大。

### 3.4 稠密的代价

**优点**:跨越词汇鸿沟,懂同义、改写、跨语言。**代价**:
- 要训练,且严重依赖训练数据的领域。在维基百科上训的 DPR,直接拿去检索法律/医疗文档,可能还不如 BM25——**稠密检索的泛化是它的阿喀琉斯之踵**。
- 对**精确实体、罕见词、数字、代码**反而常不如 BM25:这些东西的语义被压进 768 维向量后容易糊掉,而 BM25 对它们的精确匹配天然无敌。

一句话总结这场对比:**BM25 强在"字对得上",DPR 强在"意思对得上"。** 二者的强弱区恰好互补——这就引出了混合检索。

---

## 4. 稀疏 × 稠密:混合检索

既然 BM25 擅长精确/稀有词、DPR 擅长语义,那就**两个都跑,再融合**。两种主流融合法:

**(a) 分数融合(线性加权)**。两路各自打分,归一化后加权求和。但 BM25 分数(无上界、量纲随语料变)和向量内积(量纲完全不同)不可直接相加,必须先归一化(如 min-max 到 [0,1]):

```
score(q,d) = α · norm(BM25(q,d)) + (1−α) · norm(dense(q,d))
```
`α` 是稀疏/稠密的权重,需在验证集上调。

**(b) 排名融合 RRF(Reciprocal Rank Fusion,倒数排名融合)**。更鲁棒、且不需要归一化分数——它**只用排名不用原始分**,因此天然规避了量纲问题:

```
                Σ            1
RRF(d) =       r∈retrievers ──────────
                          k + rank_r(d)
```
`rank_r(d)` 是文档 `d` 在第 `r` 路检索结果里的排名(1 起),`k` 是平滑常数(原论文用 60)。某文档只要在任一路里排得高,就能拿到高融合分;两路都高则叠加。RRF 因简单、无需调权重、跨异构检索器鲁棒,成为工业 RAG 的默认融合法之一。

**经验**:在多数真实场景,混合检索的召回率**稳定高于**任何单路。代价是要同时维护倒排索引和向量索引两套基础设施。

## 5. 向量检索的工程:为什么精确 KNN 不可行,ANN/HNSW 如何救场

### 5.1 问题:亿级向量里找最近的 k 个,精确算要多久?

DPR 把每篇 passage 编成一个 d 维向量,在线检索 = 给定 query 向量 `u`,在 `N` 个 passage 向量里找内积最大的 top-k。**精确 KNN(k 近邻)** 就是和每个向量都算一遍相似度再排序:

```
精确 KNN 复杂度: O(N · d)   每次查询
```
代入工业规模:`N = 10⁹`(十亿向量),`d = 768`。一次查询要 `≈ 7.68×10¹¹` 次乘加。即便每秒 10¹⁰ 次浮点运算,单次查询也要几十秒——而线上要求毫秒级、每秒成千上万查询。**精确 KNN 在亿级规模线性扫描,彻底不可行。**

出路:放弃"绝对精确",换"极快地找到几乎最近的邻居"。这就是 **ANN(Approximate Nearest Neighbor,近似最近邻)**。我们容忍偶尔漏掉真正的第 1 名、找到第 2/3 名——召回率从 100% 降到 95~99%,换取**百倍千倍提速**。对 RAG 这是极划算的交易:检索器本就不完美,top-k 里多一两个近似邻居无伤大雅。

### 5.2 HNSW 的核心思想:可导航小世界 + 分层跳表

HNSW(Hierarchical Navigable Small World,分层可导航小世界图,Malkov & Yashunin 2016)是目前最主流的 ANN 索引之一(FAISS、hnswlib 都实现了它)。它的设计可以从两个直觉拼出来:

**直觉一:可导航小世界图(NSW)——贪心走图就能逼近最近邻。** 把每个向量当图上一个节点,给它连若干条边到"它的近邻"。检索时不扫全图,而是**从某个入口节点出发,每步走到"当前邻居里离 query 最近的那个",直到无法更近为止(贪心搜索)**。如果这个图建得好(既有连到很近邻居的"短边"保证精度,又有连到远处的"长边"保证能快速跨越空间),那么贪心走几步就能逼近真正最近邻——这就是"小世界"现象(六度分隔):任意两点间存在很短的路径。

**直觉二:分层——像跳表(skip list)一样,先粗后细。** 单层 NSW 的问题:若全用短边,跨越远距离要走很多步;若加长边,又破坏局部精度。HNSW 的解法是**多层结构**:

```
Layer 2 (顶层,稀疏):  o ────────────────── o          少量节点,长边,负责"远距离粗跳"
                       │                    │
Layer 1 (中层):       o ──── o ──── o ──── o
                       │      │      │      │
Layer 0 (底层,稠密):  o─o─o─o─o─o─o─o─o─o─o─o─o        全部节点,短边,负责"局部精搜"
```
检索从**顶层入口**开始贪心走(大步逼近目标区域),走不动了就**降一层**继续(更密的图、更短的边、更细地逼近),一直降到底层(含全部节点)做最后精搜。每个节点被放到哪些层,由一个**指数衰减的随机分布**决定(类似跳表:大部分节点只在底层,少数节点逐层"晋级"到上层),这保证上层稀疏、下层稠密。

**为什么是对数级?** 直觉同跳表:每往上一层,节点数指数减少,一次"降层"就把搜索范围对数级地缩小。每层贪心走的步数有界(因每层的图度数 `M` 是常数上限),总检索复杂度近似:

```
HNSW 检索复杂度 ≈ O(log N)   (经验/理论近似,非严格最坏界)
```
从 `O(N)` 到 `O(log N)`:十亿向量,`log₂(10⁹) ≈ 30`,理论上几十步图遍历就能定位——这就是百万倍的差距来源。

### 5.3 代码:用 FAISS 建 HNSW 索引并检索

```python
import faiss
import numpy as np

d = 768                                  # 向量维度
N = 1_000_000                            # passage 数量
passage_vecs = np.random.randn(N, d).astype('float32')   # 实际是 DPR 编码出来的

# HNSW 索引:M = 每个节点的最大边数(典型 16~64);越大越精越占内存
index = faiss.IndexHNSWFlat(d, 32, faiss.METRIC_INNER_PRODUCT)
index.hnsw.efConstruction = 200          # 建图时的候选队列宽度:越大图质量越高、建得越慢
index.add(passage_vecs)                  # 一次性灌入(HNSW 也支持增量 add)

# 在线检索
index.hnsw.efSearch = 64                 # 检索时候选队列宽度:越大召回越高、越慢(核心调速旋钮)
query_vec = np.random.randn(1, d).astype('float32')
scores, ids = index.search(query_vec, k=5)   # 返回 top-5 的相似度和 passage id
print(ids, scores)
```

**调参直觉**:`efSearch` 是**精度↔速度**的核心旋钮——它控制贪心搜索时维护的候选队列宽度,调大召回率上升、延迟上升。生产中通常先定一个延迟预算,再调 `efSearch` 到刚好满足,实测召回率。`M` 和 `efConstruction` 在建索引时定,影响图质量和内存。

**其他 ANN 路线(知道存在即可)**:IVF(倒排文件,先把向量聚类成桶,查询只搜最近的几个桶)、PQ(乘积量化,把向量压缩成几字节省内存,适合超大规模)。FAISS 把它们做成可组合的模块(如 `IVF1024,PQ64`)。HNSW 召回最高但内存大;PQ 省内存但有精度损失;实践常组合使用。

## 6. 生成融合:检索到的文档怎么喂给模型

检索拿到 top-k 文档后,如何让生成模型用上它们?三种代表性方案,机制差别很大。

### 6.1 朴素 RAG:拼进上下文(retrieve-then-read)

最直接:把 top-k 文档**拼接成一段长文本,连同 question 一起塞进 prompt**,让模型在这段上下文里读、答:

```
prompt = f"""根据以下资料回答问题,只用资料中的信息,并标注引用编号。
[1] {doc1}
[2] {doc2}
...
[k] {dock}

问题:{question}
答案:"""
answer = LLM(prompt)
```

这是今天绝大多数生产 RAG 的形态(配合指令微调过的 LLM,效果好且实现极简)。**局限**:k 个文档全拼进一个上下文,受限于模型的**上下文窗口**;而且把多篇文档塞进一个序列,注意力要在很长的文本里同时处理,长且杂时模型容易"读不过来"(算法课讲的 "lost in the middle"——中间位置的信息容易被忽略)。

### 6.2 FiD:Fusion-in-Decoder——各文档独立编码,解码时融合

FiD(Izacard & Grave, 2020)针对"多文档"做了更精巧的设计。它用 encoder-decoder 模型(如 T5),关键在**编码和解码的分工**:

```
朴素 RAG: 把 [q; doc1; doc2; ...; dock] 拼成一条长序列,一起过编码器
          → 编码器自注意力的代价随总长平方增长,k 大就爆

FiD:     把 question 分别和每篇文档拼成 k 条短序列:
            [q; doc1] ─► Encoder ─► H1   (各自独立编码,可并行)
            [q; doc2] ─► Encoder ─► H2
              ...                   ...
            [q; dock] ─► Encoder ─► Hk
          解码器对 [H1; H2; ...; Hk] 拼起来的全部编码做交叉注意力(cross-attention)
          → 信息在 *解码端* 融合(Fusion-in-Decoder 之名由此而来)
```

**为什么这么设计更好?**
- **编码可并行、复杂度低**。k 篇文档各自独立过编码器,自注意力只在单篇内部(长度 `L`)算,代价 `O(k·L²)`;而朴素拼接是在总长 `k·L` 上算自注意力,代价 `O((k·L)²) = O(k²·L²)`——**FiD 把编码代价从 k 的平方降到线性**,这就是它能吃远多于朴素 RAG 的文档数(几十上百篇)的原因。
- **融合发生在解码端的交叉注意力**:生成每个 token 时,解码器可以同时"看"所有 k 篇文档的编码,从中挑选、综合证据,而不必让 k 篇文档在编码时互相干扰。

**代价**:FiD 需要一个 encoder-decoder 架构并通常要微调,工程比"拼 prompt 调 API"重。在如今超长上下文 decoder-only LLM 普及后,朴素 RAG 因简单常被优先选用,但 FiD 的"独立编码 + 解码融合"思想在需要高效处理大量文档的系统里仍有价值。

### 6.3 端到端可微的 RAG(原始 RAG 论文)

Lewis 2020 的原始 RAG 更进一步:把检索也接进梯度图。生成概率对检索到的文档做**边际化(marginalization)**——

```
                 Σ
P(answer|q) =   z∈top-k   P_retriever(z|q) · P_generator(answer | q, z)
```
即"对每篇检索文档 z,先算它被检索到的概率,再算在该文档下生成答案的概率,加权求和"。这样检索器的打分也能收到生成端的梯度信号、被一起训练(查询编码器可微,文档索引一般定期重建)。它优雅但工程复杂,实践中**朴素 RAG(检索器/生成器各自训、不端到端)因简单和模块化,占了主流**。

## 7. chunking、重排序、引用归因:让 RAG 真正能用的三件事

朴素管线和能上线的 RAG 之间,差的就是这三件工程细节。

### 7.1 Chunking:文档切多大?

文档不能整篇丢进检索——太长则一个向量糊掉所有细节、且超模型窗口。要切成 **chunk(块)**。这是 RAG 里**最被低估、却最影响效果**的一环:

- **切太大**:一个 chunk 混入多个话题,向量语义被稀释,检索到了也夹带大量无关内容,既稀释相关性又浪费上下文窗口。
- **切太小**:把"问题的答案"切碎到跨多个 chunk,任何单个 chunk 都不完整,检索召回不全。
- **重叠(overlap)**:相邻 chunk 留一段重叠(如重叠 10~20%),避免答案恰好被切在边界、两边都不完整。
- **按结构切优于按字数切**:沿段落/标题/句子边界切(尊重文档的自然语义单元),比硬按固定 token 数切效果好——后者经常把一句话拦腰斩断。

**没有万能 chunk 大小**,必须按文档类型(代码、法条、对话、网页)和下游问题用评测来调。这是后面"评测驱动迭代"的典型对象。

### 7.2 重排序(reranking):用 cross-encoder 精排

回想第 3 节:双塔(bi-encoder)能预计算、快,但 query 和文档在编码时**互不见面**,精度有上限;cross-encoder 把 query 和文档**拼起来一起过模型**,每层互相注意,精度高但慢、不能预计算。**两者的优劣恰好可以分工——这就是经典的"召回-精排"两阶段(retrieve-then-rerank)**:

```
阶段一(召回,要快、要广):  bi-encoder/BM25 从百万文档里捞 top-100         # 快,但排序粗
阶段二(精排,要准、量小):  cross-encoder 给这 100 篇逐个精打分,重排出 top-5  # 慢,但只算 100 次
```
cross-encoder 只需对召回阶段给出的少量候选(几十~几百)精排,代价可控;同时把它的高精度用在了最关键的最终排序上。这是工业 RAG 提升答案质量**性价比最高**的一招——一个训好的 reranker(如 cross-encoder)常能显著抬高最终命中率。

```python
# cross-encoder 重排序(概念,用 sentence-transformers 的 CrossEncoder)
from sentence_transformers import CrossEncoder
reranker = CrossEncoder("cross-encoder/ms-marco-MiniLM-L-6-v2")   # 模型名待核

candidates = retrieve(query, top_k=100)                  # 阶段一召回
pairs = [(query, doc.text) for doc in candidates]
scores = reranker.predict(pairs)                         # 阶段二:逐对精打分
reranked = [doc for _, doc in sorted(zip(scores, candidates),
                                     key=lambda x: -x[0])][:5]
```

### 7.3 引用与归因(citation / attribution)

RAG 相对纯 LLM 的杀手锏是**可溯源**。实现上:
- 给每个 chunk 带元数据(来源文档、URL、段落位置)。
- prompt 里给每个 chunk 编号,**要求模型在答案里标注引用编号** `[1][2]`。
- 进一步可做**归因校验**:答案生成后,逐句检查"这句话是否真能由它引用的那篇文档蕴含(entailment)"——这就和第 8 节的忠实性评测接上了。

**坑**:模型即使被要求引用,也可能**乱标引用**(标了 [2] 但内容其实来自参数记忆,文档 [2] 根本没这信息)。引用的存在 ≠ 引用的正确。这必须靠评测来抓。

## 8. 评测:检索质量 + 生成质量 + 忠实性,三者缺一不可

RAG 评测最大的陷阱是**只看最终答案对不对**。这会让你无法定位问题:答案错了,是检索没召回证据(检索的锅),还是召回了但模型没用好(生成的锅)?必须**分层评测**。

### 8.1 检索质量:Recall@k 与 MRR

**Recall@k(前 k 召回率)**:相关文档有没有出现在 top-k 里。设每个 query 有已知的相关文档集合,

```
                 # (相关文档 ∩ top-k 检索结果)
Recall@k(query) = ────────────────────────────────
                      # 相关文档
```
对所有 query 取平均。直觉:**Recall@k 是 RAG 质量的天花板**——如果证据根本没被检索进 top-k,生成模型再强也无米下炊。所以调 RAG 第一件事永远是把 Recall@k 拉上去。

**MRR(Mean Reciprocal Rank,平均倒数排名)**:不只看"在不在 top-k",还看"排多前"。设第一个相关文档出现在排名 `rankᵢ`:

```
        1   Σ      1
MRR =  ──        ──────
        Q   i     rankᵢ
```
相关文档排第 1 → 贡献 1;排第 2 → 贡献 0.5;排第 5 → 贡献 0.2。MRR 高说明相关文档不仅被召回、还排得靠前(对 RAG 重要,因为靠前的文档更可能被模型真正读进去、且在上下文里位置更优)。相关指标还有 nDCG(带分级相关性的折损累积增益,适合"有多个相关文档且相关程度不同"的场景)。

### 8.2 生成质量

答案本身好不好。方法见第 13 章(生成评测),这里只点名:
- **有标准答案时**:Exact Match / F1(抽取式 QA),或语义相似度。
- **开放式答案**:n-gram 重叠类指标(BLEU/ROUGE)信号弱,如今更多用 **LLM-as-judge**(让一个强模型按 rubric 给答案打分)——但要警惕评判模型自身的偏置与不稳定(详见第 13 章)。

### 8.3 忠实性(faithfulness / groundedness):RAG 的命门

这是 RAG 特有、且**最关键**的一维:**答案是否真的由检索到的证据所支撑?** 注意它和"答案正确"是两个不同的轴,可以四象限组合:

```
                  答案正确          答案错误
证据支撑       ✅ 理想(对且有据)   证据本身就错/过时(检索的锅)
证据不支撑     ⚠ 蒙对(危险!)      ❌ 幻觉(检索了仍编)
```
**"蒙对"那一格特别危险**:答案碰巧对,但模型其实没看证据、是按参数记忆答的——这种 RAG 在你部署到新领域、参数记忆失效时会突然崩,而你之前的"答案正确率"评测完全看不出隐患。这正是开篇说的"**检索了但仍幻觉**"的量化体现:检索器把正确文档放进了上下文,生成器却无视它、按旧记忆作答。

如何度量忠实性:
- **NLI/蕴含检验**:把"答案的每个陈述"和"检索到的证据"送进一个自然语言推理(NLI)模型,判断证据是否**蕴含(entail)**该陈述。若有陈述不被任何证据蕴含 → 该陈述未被支撑(疑似幻觉)。
- **LLM-as-judge 判忠实性**:给评判模型"问题 + 证据 + 答案",让它判断答案是否完全由证据支撑、有无证据外的断言(这类思路被 RAGAS 等框架工程化)。

**为什么忠实性必须单独测**:一个 Recall@k 很高、答案正确率也不错的系统,可能忠实性很差(大量"蒙对")。只有把三维分开测,你才能回答"我的 RAG 到底强在哪、换个领域还灵不灵"。这条线索一直延伸到第 13 章(忠实性评测)和第 14 章(幻觉的 NLP 视角)。

### 8.4 端到端评测的代码骨架

```python
def evaluate_rag(queries, gold_docs, gold_answers, rag_system, nli_model):
    recall_at_k, rr, ans_correct, faithful = [], [], [], []
    for q, gdocs, gans in zip(queries, gold_docs, gold_answers):
        retrieved = rag_system.retrieve(q, k=5)               # top-k 文档
        answer    = rag_system.generate(q, retrieved)

        # 1) 检索:Recall@k
        hit = set(d.id for d in retrieved) & set(gdocs)
        recall_at_k.append(len(hit) / max(len(gdocs), 1))
        # 2) 检索:MRR(第一个相关文档的倒数排名)
        rank = next((i+1 for i,d in enumerate(retrieved) if d.id in gdocs), None)
        rr.append(1.0/rank if rank else 0.0)
        # 3) 生成质量:与标准答案比(此处用简化的精确/语义匹配)
        ans_correct.append(answer_match(answer, gans))
        # 4) 忠实性:答案每个陈述是否被检索证据蕴含
        evidence = " ".join(d.text for d in retrieved)
        faithful.append(nli_supports(nli_model, premise=evidence, hypothesis=answer))

    return {
        "Recall@5":   mean(recall_at_k),
        "MRR":        mean(rr),
        "AnswerAcc":  mean(ans_correct),
        "Faithful":   mean(faithful),     # ← 与 AnswerAcc 同时看,才能发现"蒙对"
    }
```
**用法**:`AnswerAcc` 高而 `Faithful` 低 = 你的系统在"蒙对",换领域必崩;`Recall@5` 低 = 先去修检索(chunking/混合检索/reranker),修生成是徒劳。

## 9. 设计权衡与常见坑

- **稀疏 vs 稠密不是二选一,先上 BM25 做基线。** BM25 零训练、强基线、可解释,**永远先把它跑起来当对照**。直接上稠密检索而不和 BM25 比,你根本不知道稠密有没有真带来增益——很多"上了向量库效果还下降"的案例就是没做这个对照。
- **chunking 是头号隐形杀手。** 检索效果差,十有八九先查 chunk 切法,而不是急着换更大的 embedding 模型。
- **embedding 模型的领域匹配 > 模型大小。** 在通用语料训的句向量,放到专业领域(法律/医疗/代码)可能水土不服。优先选领域贴近的模型,必要时在自己的数据上微调,别盲目堆参数。
- **别用未归一化的向量算余弦。** 余弦相似度要求向量先 L2 归一化;直接拿原始向量算内积当余弦,是常见 bug。FAISS 用 `METRIC_INNER_PRODUCT` 时若想要余弦,须自己先归一化向量。
- **top-k 不是越大越好。** k 太大,无关文档涌入上下文,既稀释相关信息("lost in the middle")又烧 token、增延迟。k 要按 Recall@k 曲线和上下文预算一起定。
- **引用 ≠ 忠实。** 模型会乱标引用。务必单独测忠实性,别被"答案里有 [1][2]"骗了。
- **ANN 召回率要实测,不能默认。** HNSW 的 `efSearch` 调小了召回会悄悄掉,而你从延迟上看不出来。上线前用一批 query 对比 ANN 结果和精确 KNN 结果,确认召回率达标。
- **RAG 不解决推理。** RAG 补的是**知识**(查得到的事实),不补**推理**(多步逻辑、计算)。需要多跳推理的问题,光堆检索文档常常无效,得配合多步检索(iterative retrieval)、查询改写、或工具调用——这些是第 12、15 章和 Agent 方向的话题。
- **提示注入风险。** 检索内容来自外部文档,可能被植入"忽略之前指令"之类的恶意文本,劫持生成。RAG/Agent 场景的提示注入是真实安全威胁(详见第 14 章)。

## 12. 小结与承上启下

这一章把 RAG 从"动机"一路讲到"能搭能评":

- **为什么需要 RAG**:LLM 的幻觉、知识过时、长尾缺失、不可溯源,都是参数压缩知识的结构性代价;RAG 用"先检索再生成"把知识外置,缓解(不是根治)这四点。
- **检索器**:BM25 是 IDF(稀有度)× TF 饱和 × 长度归一三个补丁拼出的可解释打分,强在词面精确匹配;DPR 用双塔 + InfoNCE/in-batch negatives 学语义向量,强在跨越词汇鸿沟;二者互补,混合检索(分数融合 / RRF)通常更稳。
- **向量检索工程**:精确 KNN 在亿级 `O(N·d)` 不可行,HNSW 用"分层 + 可导航小世界图 + 贪心搜索"把检索降到近似 `O(log N)`,以可控的召回率损失换百万倍提速。
- **生成融合**:朴素 RAG 拼上下文(主流、简单),FiD 用"独立编码 + 解码融合"高效处理多文档,原始 RAG 还可对文档边际化做端到端可微。
- **工程三件事**:chunking(头号隐形变量)、cross-encoder 重排(性价比最高的提升)、引用归因(杀手锏但需校验)。
- **评测**:必须分层——Recall@k/MRR 量检索(质量天花板)、生成质量量答案、**忠实性**量"答案是否真由证据支撑",三者缺一就无法定位问题、无法判断换领域是否还灵。

**承上**:稠密检索的双塔与对比学习直接继承自第 06 章句向量;query/文档编码用的子词与多语言机制见第 08 章;生成器内部的注意力与长上下文("lost in the middle")属算法课。

**启下**:RAG 的"忠实性"是第 13 章生成评测与第 14 章幻觉视角的核心关卡;检索内容引入的提示注入是第 14 章安全话题;而把本章所有零件——chunking、句向量编码、FAISS 索引、混合检索、reranker、基于证据的生成、三维评测——串成一个真正能跑、能迭代的问答系统,正是第 16 章 Capstone 要你亲手做的事。读完这一章,你已经具备了搭出那套系统并讲清它每个设计为什么这么选的全部基础。

### 练习 1:推导 BM25 各项作用(b=0、k₁→∞、IDF 为何用对数)

**默写完整公式。** 对 query `q` 里每个词项 `t`,设 `f(t,d)` 为 `t` 在文档 `d` 的词频,`|d|` 为文档长度,`avgdl` 为语料平均文档长度,`N` 为文档总数,`n(t)` 为含 `t` 的文档数:

```
                Σ           f(t,d) · (k₁ + 1)
BM25(q,d) =     t∈q  IDF(t)·──────────────────────────────────
                          f(t,d) + k₁·(1 − b + b·|d|/avgdl)

IDF(t) = ln( (N − n(t) + 0.5) / (n(t) + 0.5) )          # 工程版常再 +1 防负
```

它由三个补丁拼成:**IDF(稀有度对数加权)× TF 饱和(随次数增长但有上界)× 长度归一(把长文档优势压回去)**。

---

**(a) 把 `b` 从 0.75 改成 0。**

长度归一因子是 `(1 − b + b·|d|/avgdl)`。代入 `b=0`:

```
1 − 0 + 0·|d|/avgdl = 1     # 恒等于 1,与 |d| 无关
```

于是分母退化为 `f(t,d) + k₁`,**完全关闭文档长度归一**,只剩纯 TF 饱和。

对长文档排名的影响:长文档因为字数多,任何词都更容易反复出现、`f(t,d)` 更大。`b=0.75` 时分母里的 `b·|d|/avgdl` 会随 `|d|` 增大而抬高分母、把长文档的 TF 优势压回去;一旦 `b=0`,这个压制消失,**长文档纯靠堆字数就能蹭高分,排名被系统性抬高**,短小精悍但高度相关的文档反而吃亏。这正是"长文档不该占便宜"那个毛病重新冒出来。

**结论:`b=0` 关闭长度归一,长文档被不公平地抬高排名。**

---

**(b) 当 `k₁→∞` 时 TF 饱和项退化成什么。**

看 TF 饱和项(以平均长度文档 `|d|=avgdl`、归一因子为 1 为例,便于看清,结论与长度无关):

```
        f·(k₁+1)        f·(k₁+1)        f·k₁·(1 + 1/k₁)
TF_sat = ─────────  →   ────────    =   ─────────────────
         f + k₁          f + k₁          k₁·(f/k₁ + 1)
```

当 `k₁→∞`:`1/k₁→0`、`f/k₁→0`,分子分母同除 `k₁`:

```
        f·(1 + 1/k₁)        f·1
TF_sat = ─────────────  →   ──────  = f
          f/k₁ + 1            0 + 1
```

即 `TF_sat → f(t,d)`,饱和上界 `k₁+1` 被推到无穷,曲线变成一条直线,**退化成线性词频(raw TF)**:出现 1 次记 1,出现 100 次记 100,边际不再递减。

此时 `BM25 → Σ IDF(t)·f(t,d)`,**相当于最朴素的 "TF×IDF" 线性打分**(没有饱和补丁的 TF-IDF)。

补充另一极端做对照:`k₁→0` 时分母 `→ f`,`TF_sat → (k₁+1) → 1`(只要 `f≥1`),退化成"出现与否"的**布尔检索**。两端正好框出 BM25 的可调谱:布尔 ← BM25 → 线性 TF-IDF。

**结论:`k₁→∞` 时饱和消失,TF 项退化为线性词频,BM25 退化成朴素的 TF×IDF 线性打分。**

---

**(c) 为什么 IDF 用对数而不是直接用 `1/n(t)`。**

核心:**对数把稀有度的动态范围从"数量级差距"压成"线性差距",避免极稀有词得分爆炸性主导整条 query。**

用 `N=10⁶` 实算对比(已验算):

```
n(t)        ln((N−n+0.5)/(n+0.5)+1)        1/n(t)
1           13.41                          1.0
10          11.46                          0.1
1000         6.91                          1.0e−3
100000       2.30                          1.0e−5
900000       0.11                          1.1e−6
```

- 用 `1/n`:`n=1` 的词权重是 `n=10` 的词的 **10 倍**、是 `n=1000` 的词的 **1000 倍**。一个偶然只出现 1 次的词(常是拼写错误、噪声 token)就能凭 1000 倍权重压垮 query 里所有其他正常词,排序被单个噪声词绑架。
- 用对数:`n=1` 到 `n=1000` 权重只从 13.4 降到 6.9,**差距被压成 2 倍量级而非 1000 倍**,稀有词仍更重要、但不会一家独大;多个词项的贡献能比较均衡地叠加。

另外两点:① 对数源自概率检索模型里"该词出现在相关 vs 不相关文档的对数几率比(log-odds)",有理论出处,不是随手选的;② 对数在 `n` 增大时平滑下降,`1/n` 则在小 `n` 处极陡、大 `n` 处几乎贴地,数值上更不稳。

**结论:对数压缩稀有度的动态范围,防止极稀有词得分爆炸性主导排序,同时保留"越稀有越重要"的单调性,且有概率模型的理论依据。**

### 练习 2:解释 in-batch negatives 并算编码成本账

**为什么"同 batch 里别人的正样本"可以当我的负样本。**

训练目标是让相关的 `(qᵢ, pᵢ⁺)` 内积大、不相关对内积小。对 query `qᵢ` 来说,同一个 mini-batch 里其他 query 的正样本 `pⱼ⁺ (j≠i)`,是**别人问题的答案**——它和 `qᵢ` 在语义上大概率不相关,因此可以**近似当作 `qᵢ` 的负样本**直接用,无需额外采样。

矩阵视角看得最清楚:把整个 batch 的相似度排成 `B×B` 矩阵 `S`,`S[i][j] = sim(qᵢ, pⱼ⁺)`,则:

```
S 的对角线 S[i][i]   = 所有正对(qᵢ 与自己的正样本)
S 的非对角 S[i][j≠i] = 所有 in-batch 负对(qᵢ 与别人的正样本)
```

于是 InfoNCE 损失变成"在 `S` 的每一行做 softmax 交叉熵,标签是对角线位置 `i`"——即标准多分类,标签向量 `[0,1,2,…,B−1]`。一次矩阵乘法 `Q @ Pᵀ` 就同时算出了全部正对和负对。

**这只是近似,会有"假负例"风险**(极小概率别人的正样本恰好也和我相关),但 batch 内这种碰撞概率极低,实践中收益远大于代价。

---

**算一笔编码成本账。** 设 batch 大小 `B`,朴素采样为每个 query 额外采 `n` 个负样本:

```
朴素负采样:  B 个正样本 + B·n 个采来的负样本   → 需编码 B + B·n 个 passage
in-batch:    B 个正样本本身兼当负样本池       → 只需编码 B   个 passage
节省:        每个 query 白送 B−1 个负样本,几乎零额外成本
```

具体数(已验算,取 `n=7`):

```
B=8:    朴素编码   64 个 passage,in-batch 编码   8 个,免费负样本/query = 7
B=128:  朴素编码 1024 个 passage,in-batch 编码 128 个,免费负样本/query = 127
B=512:  朴素编码 4096 个 passage,in-batch 编码 512 个,免费负样本/query = 511
```

可见 in-batch 把 passage 编码量从 `B(1+n)` 降到 `B`,**省下 `B·n` 次 BERT 前向**,且负样本数随 `B−1` 免费增长。这也解释了为什么 DPR/对比学习"吃显存换效果":`B` 越大,负样本越多、对比信号越强、检索越好,工业训练才会用很大 batch 甚至跨卡 `gather` 把所有卡的 passage 都纳入负样本池。

---

**为什么还要额外加 BM25 hard negatives。**

in-batch 负样本是**随机负例**——别人问题的答案和我的问题通常**一眼就不相关**(话题、实体完全不搭),信号太"软":模型轻松就能把它们和正样本分开,学不到细粒度区分能力。

BM25 hard negatives 是**难负例**:用 BM25 检索出"**词面很像但其实答非所问**"的 passage(高词面重叠、低语义相关)。模型必须学会"光词对上不算相关、要看真实语义"才能把正样本从这些难负例里挑出来——这正是 DPR 想要的细粒度语义判别力。DPR 原文的消融显示加 hard negatives 对最终精度提升显著。

**结论:in-batch 负样本免费但太软,只能学粗区分;BM25 难负例提供"词面像却不相关"的硬信号,逼模型学细粒度语义判别,是精度的关键来源。**

### 练习 3:对比稀疏/稠密/混合的可验证实验设计

**实验目标。** 在同一领域数据集上,公平对比 **BM25 / DPR / RRF 混合** 三者的 `Recall@10` 与 `MRR`,并验证"哪类 query 偏向哪种检索器""混合是否稳定不劣于单路"。

**数据集要求。** 一份带标注的领域数据集 `{(query, 相关文档集合)}`(如自建领域 QA 或公开集如 BEIR 中的某个子集)。切分 train/dev/test:DPR 训练用 train,RRF 的权重/`k`、HNSW 的 `efSearch`、chunk 大小等超参在 dev 上调,**最终指标只在 test 上报一次**(避免在 test 上调参导致乐观偏差)。

**三个被测系统(控制变量)。** 三者必须共享**同一套 chunk 切分和同一份语料**,只换检索路径,否则比的不是检索器而是 chunking:

```
A. BM25     : 倒排索引,k1=1.5, b=0.75(先报这条基线!)
B. DPR      : 双塔编码 + HNSW(efSearch 调到召回稳定),向量预编码入库
C. RRF 混合 : 同时跑 A 和 B,各取 top-N(如 100),用 RRF(k=60) 融合后取 top-10
```

**评测指标(对每个 query 算,再对全体取平均):**
- `Recall@10` = |相关文档 ∩ top-10| / |相关文档|(检索质量天花板)
- `MRR` = 第一个相关文档排名的倒数 `1/rank` 的均值(看排得多靠前)

**最小可跑评测代码(numpy/CPU,检索器可替换为真实实现):**

```python
import numpy as np

def recall_at_k(retrieved_ids, gold_ids, k=10):
    """retrieved_ids: 按相关性降序的文档 id 列表;gold_ids: 相关文档 id 集合"""
    topk = set(retrieved_ids[:k])
    gold = set(gold_ids)
    if not gold:
        return 0.0
    return len(topk & gold) / len(gold)

def reciprocal_rank(retrieved_ids, gold_ids):
    gold = set(gold_ids)
    for i, did in enumerate(retrieved_ids):      # i 从 0 起
        if did in gold:
            return 1.0 / (i + 1)                 # 排名从 1 起
    return 0.0

def rrf_fuse(rankings, k=60):
    """rankings: List[List[doc_id]],每路一个按序的 id 列表;返回融合后按分降序的 id"""
    scores = {}
    for ranking in rankings:
        for rank, did in enumerate(ranking):     # rank 从 0 起
            scores[did] = scores.get(did, 0.0) + 1.0 / (k + rank + 1)
    return [did for did, _ in sorted(scores.items(), key=lambda x: -x[1])]

def evaluate(system_retrieve, queries, gold):
    """system_retrieve(q) -> 按相关性降序的 doc_id 列表"""
    recalls, rrs = [], []
    for q in queries:
        ranked = system_retrieve(q)
        recalls.append(recall_at_k(ranked, gold[q], k=10))
        rrs.append(reciprocal_rank(ranked, gold[q]))
    return {"Recall@10": float(np.mean(recalls)), "MRR": float(np.mean(rrs))}

# ---- 自检:用一个玩具例子验证函数正确 ----
if __name__ == "__main__":
    bm25_rank  = ["d3", "d1", "d9", "d2"]     # BM25 把相关文档 d1 排在第 2
    dense_rank = ["d1", "d7", "d2", "d3"]     # DPR  把相关文档 d1 排在第 1
    fused = rrf_fuse([bm25_rank, dense_rank], k=60)
    print("RRF 融合:", fused)                 # d1 两路都靠前 -> 融合后应排第一
    assert fused[0] == "d1"
    assert abs(reciprocal_rank(bm25_rank, {"d1"}) - 0.5) < 1e-9   # 排第2 -> 0.5
    assert abs(reciprocal_rank(dense_rank, {"d1"}) - 1.0) < 1e-9  # 排第1 -> 1.0
    print("自检通过")
```

**预期结果(假设,需用真实数据验证):**

| query 类型 | 预期赢家 | 原因 |
|---|---|---|
| 含罕见实体名 / 错误码 / 精确术语 / 数字 / 代码片段 | **BM25** | 这些被压进 768 维向量易糊掉;BM25 对精确词面匹配天然无敌 |
| 同义改写 / 口语化 / 跨语言 / 概念性问法("心脏病发作"vs"心肌梗死") | **DPR** | 跨越词汇鸿沟,语义相近即可命中,BM25 因共享词为零判不相关 |
| 大多数混合/真实分布 query | **RRF 混合 ≥ 单路** | 任一路排得高就能拿高融合分,召回率稳定不低于单路 |

**判读规则(把实验和本章主张挂钩):**
- **第一件事永远先报 BM25 基线**——不和 BM25 比,你不知道 DPR/混合到底有没有带来真实增益(本章第 9 节明确的头号坑:很多"上了向量库反而下降"就是漏了这个对照)。
- 若混合的 `Recall@10` 在 test 上**没有稳定 ≥** 两条单路,需回查:RRF 的 `k`、两路 top-N 截断、DPR 是否领域失配(可能 DPR 拉低了融合)。
- 若 DPR 在该领域**整体不如 BM25**,大概率是稠密检索的领域泛化问题(在维基训的塔搬到法律/医疗会水土不服),应考虑领域内微调或加 BM25 hard negatives 重训。

**结论:三系统共享同一 chunk/语料只换检索路径;在 dev 调参、test 报一次;先报 BM25 基线;预期精确词类 query BM25 赢、语义改写类 DPR 赢、RRF 混合在多数分布上不劣于单路。**

### 练习 4:RAG 为何"检索到了仍幻觉",并设计检测

**至少三个原因(本质是"证据进了上下文,但没被正确使用"):**

1. **模型偏信参数记忆,不看上下文。** 解码本质仍是最大化 `P(xₜ | x_{<t})`,当检索证据与模型预训练里的"成见"冲突时,模型常按参数里的旧记忆/高频说法作答,把上下文里的正确证据晾在一边。证据越反直觉、越违背模型先验,这种"无视证据"越严重。

2. **证据被淹没在过长上下文中间(lost in the middle)。** top-k 拼成长 prompt 时,注意力对**首尾位置**敏感、对**中间位置**信息容易忽略。正确证据若恰好落在拼接序列的中段,模型实际"读不到",于是退回参数记忆编造。`k` 越大、上下文越长,这个效应越强。

3. **chunk 切碎使证据不完整。** chunking 切太小或切在错误边界,会把"完整答案"劈到多个 chunk;即使其中一块被召回,它**自身不含完整事实**,模型只能拿半截证据脑补补全,补出来的部分就是幻觉。

4. **(补充)被要求引用却乱标。** prompt 要求标 `[1][2]`,模型会照标,但标的来源可能是错的——内容其实来自参数记忆,被引文档根本没这条信息。**"引用存在 ≠ 引用正确"**,这是一种伪装成"有据"的幻觉。

(以上 ① 是最核心、最隐蔽的一类,对应本章 8.3 节四象限里"证据不支撑 + 答案正确 = 蒙对"那一格,以及"证据不支撑 + 答案错误 = 检索了仍编"那一格。)

---

**最小检测实验:专门"抓"蒙对 / 检索了仍幻觉。**

核心思路:**把"答案正确"与"答案是否真由证据支撑"两个轴分开测**,只看答案对错永远抓不到蒙对。给出两个互补的最小实验:

**实验 A — 证据 vs 先验冲突探针(直接抓"无视证据"):**

```
1. 构造一批 query,每条配一篇"证据正确、但与模型先验明显冲突"的文档。
   例:把某事实改成一个反直觉但在该文档语境下成立的值
       (如虚构设定 / 最新变更 / 冷门长尾事实,确保模型先验是"旧值")。
2. 对每条 query 跑两种条件:
      (闭卷)  answer_closed = LLM(question)              # 不给证据,测模型先验
      (RAG)   answer_rag    = LLM(question, 该证据文档)   # 给正确但反先验的证据
3. 判读:
      answer_rag == 证据里的新值   → 模型站证据 ✅(忠实)
      answer_rag == 闭卷的旧值     → 模型站先验、无视上下文 ❌(检索了仍幻觉)
   统计"站先验"的比例 = 该系统无视证据的严重程度。
```

这个设计的关键是**让证据和先验对立**,把"模型到底在用证据还是在用记忆"逼成一个可观测的二选一——若证据和模型先验一致,蒙对会被掩盖,实验就失效。

**实验 B — NLI 蕴含逐句核对(抓"答案不被证据支撑",对应本章 8.4):**

```python
# 用一个 NLI 模型逐句判断:检索证据是否蕴含(entail)答案里的每个陈述
# 依赖:pip install transformers torch;CPU 可跑(模型小,如 MNLI 微调的 base 模型)
from transformers import pipeline

nli = pipeline("text-classification",
               model="MoritzLaurer/DeBERTa-v3-base-mnli",   # 模型名待核,任意 MNLI 模型均可
               top_k=None)

def split_claims(answer):
    # 极简切句;真实系统应做更稳健的子句切分
    import re
    return [s.strip() for s in re.split(r"[。.!?]", answer) if s.strip()]

def faithfulness(evidence, answer):
    """返回 (被证据蕴含的陈述比例, 未被支撑的陈述列表)"""
    claims = split_claims(answer)
    if not claims:
        return 1.0, []
    unsupported = []
    for c in claims:
        # NLI: premise=证据, hypothesis=答案里的一条陈述
        out = nli(f"{evidence} [SEP] {c}")[0]
        label = max(out, key=lambda x: x["score"])["label"].lower()
        if "entail" not in label:          # 非 entailment(neutral/contradiction)= 未被支撑
            unsupported.append(c)
    supported_ratio = 1 - len(unsupported) / len(claims)
    return supported_ratio, unsupported

# ---- 用法 ----
evidence = "维特比算法用动态规划求 HMM 中概率最大的隐状态序列,复杂度 O(T·K^2)。"
answer_good = "维特比算法用动态规划求最优隐状态序列,复杂度是 O(T·K^2)。"
answer_halu = "维特比算法用动态规划求最优隐状态序列,且由 Rabiner 在 1967 年提出。"  # 后半句证据没说

print(faithfulness(evidence, answer_good))   # 接近 1.0,无未支撑陈述
print(faithfulness(evidence, answer_halu))   # <1.0,"Rabiner 1967" 那句被标为未支撑 -> 抓到幻觉
```

**判读(把两实验串起来用,正是本章 8.3 的四象限):**
- 实验 A 里**站先验比例高**,或实验 B 里 `supported_ratio` 低,而**答案正确率(AnswerAcc)却不低** → 系统在"蒙对":此刻答对是靠参数记忆,**一旦换到模型先验失效的新领域就会突然崩**,而只看 AnswerAcc 完全看不出这个隐患。
- 若 `Recall@k` 本身就低 → 先修检索(chunking / 混合检索 / reranker),修生成是徒劳。

**结论:检索了仍幻觉的根因是"证据进了上下文却没被正确使用"(偏信先验、lost in the middle、chunk 切碎、乱标引用);抓它的关键是把"答案正确"与"忠实性"两轴分开测——用证据 vs 先验冲突探针看模型站哪边,用 NLI 逐句蕴含核对答案是否真被证据支撑,二者结合即可量化出"蒙对"这一最危险的隐患。**
