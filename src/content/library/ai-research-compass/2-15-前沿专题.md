> 这一章把你从"会用现成的推理引擎和训练框架"带到"能从第一性原理判断三条最活跃的系统战线各自在打哪堵墙、为什么这么设计、代价在哪":(1) 把 prefill 和 decode 拆到不同 GPU 池的**解耦推理**——因为两阶段的硬件特性正好相反,混在一起必然互相拖累;(2) 上万卡规模下故障成为常态,如何用**异步/增量 checkpoint、自动重调度、SDC 检测**让训练不被频繁打断;(3) 上下文越来越长、KV cache 线性爆炸,如何用**量化、驱逐、稀疏、GQA、prefix caching、chunked prefill** 一层层把它压下去。读完你应该能给一个解耦系统配资源比、能解释为什么 KV 驱逐可能丢关键信息、能讲清增量 checkpoint 比全量快在哪。

## 一、动机:三堵正在被猛攻的墙

前面几章我们把单个优化讲透了:FlashAttention 打访存墙,PagedAttention 打显存碎片,投机解码摊薄读权重成本,量化缩小权重。这些都是**在单个 GPU 或单个算子内部**做文章。这一章退到系统层、集群层,看三个当下(2024–2025)最热的、还没有标准答案的工程战线。它们的共同特点是:**单点优化已经做得差不多了,瓶颈转移到了"怎么把多个阶段、多张卡、多个请求组织起来"。**

我按"问题的根源是什么"把它们串起来,这样你不会把它们当成三个孤立的 trick:

- **解耦推理**的根源是:**prefill 和 decode 的算术强度天差地别**(上一组章节反复用的判据)。第 11 章我们算过,decode 单序列算术强度 ≈ 1 FLOP/Byte,是重度 memory-bound;prefill 一次处理几百上千 token,算术强度高得多,是 compute-bound。两个特性相反的负载塞进同一张卡、同一个批次,会互相伤害——这是解耦的全部动机。
- **万卡容错**的根源是:**故障率随卡数线性叠加**。单卡很可靠不代表万卡可靠;把"几个月不坏一次"乘以一万,平均故障间隔(MTBF)就塌到几十分钟量级。规模本身把"偶发异常"变成了"稳态背景噪声"。
- **长上下文 KV 压缩**的根源是:**KV cache 显存随上下文长度线性增长**(第 10 章的公式),而上下文从 2K 涨到 128K、1M,KV cache 从"配角"变成"绝对主角"。压它有四个正交的方向:压精度、删条目、减头数、跨请求复用。

三条线我都按"问题 → 机制/推导 → 关键数字 → 坑"展开。这一章洞察重于代码(很多系统是闭源或快速演进的),但每个关键结论我都给你能自己验算的量化依据。

## 二、Prefill/Decode 解耦:两个相反的负载不该同居

### 2.1 先把两阶段的"相反"钉死

这是整节的地基,务必从数字上感受到反差,而不是只记住"一个 compute-bound 一个 memory-bound"。沿用第 11 章的记号:稠密模型参数量 N(FP16 下权重 2N 字节),一次前向处理 T 个 token。

```
计算量 ≈ 2 · N · T  FLOP      (T 个位置各过一遍权重)
访存量 ≈ 2 · N      Byte       (权重只读一次,T 个位置共享)
                    + O(T) 的 KV/激活项(相对 2N,prefill 的 T 还没大到压过它时可暂略)
算术强度 I ≈ (2NT) / (2N) = T  FLOP/Byte
```

**结论一行:一次前向的算术强度 ≈ 它处理的 token 数 T。** 于是:

- **decode**:T = 1(自回归每步 1 个 token),I ≈ 1 FLOP/Byte。脊点 I⋆ 在几十到几百量级(第 11 章,具体数值待核于具体卡)。I ≪ I⋆,**memory-bound**,瓶颈是反复把整个权重从 HBM 搬过来。
- **prefill**:T = prompt 长度(512、2K、甚至更长),I ≈ T,轻松越过脊点。**compute-bound**,瓶颈是 tensor core 算矩阵乘的算力。

> **两阶段不只是"快慢不同",而是被两种完全不同的硬件资源卡住:decode 卡在 HBM 带宽,prefill 卡在 FLOPs。** 这就是为什么"用同一种配置同时服务好两者"在物理上是矛盾的——你优化带宽利用就浪费算力,优化算力利用就浪费带宽。

### 2.2 混在一起为什么互相伤害:两种具体的干扰

传统做法是 prefill 和 decode 在同一组 GPU、同一个连续批处理(continuous batching)里跑。问题有两层:

**干扰一:延迟指标互相打架。** 推理服务有两个核心 SLO(服务等级目标):

```
TTFT  = Time To First Token   (首 token 延迟,主要由 prefill 决定)
TPOT  = Time Per Output Token (每个后续 token 延迟,主要由 decode 决定)
       (也常写作 ITL = Inter-Token Latency)
```

当一个长 prompt 的 prefill(可能几十到几百毫秒,因为要算几千个位置)插进来时,同批次里那些正在 decode 的请求会被这个大算力任务**挤占**,它们的 TPOT 瞬间飙高——用户体感是"字突然卡住一下"。反过来,为了不卡 decode 而限制 prefill 批大小,又会让 TTFT 变长。**一套资源、一个调度器,没法同时把 TTFT 和 TPOT 都压到最优,只能折中。**

**干扰二:并行策略的最优解不同。** prefill 算力受限,适合用张量并行(TP)把单次大矩阵乘切到多卡一起算,降低单请求 prefill 延迟;decode 带宽受限、单请求算力需求极小,更适合用**大批次 + 较少 TP**(甚至数据并行式地多副本),把读一遍权重的成本摊给尽可能多的并发序列。两阶段对 TP 度、batch 策略的最优选择**方向相反**,合在一张卡上只能用一套,必然有一方将就。

### 2.3 解法:物理拆开,中间传 KV cache

**核心做法:把集群分成两个池——prefill 池(算力优化)和 decode 池(带宽/批大小优化),各自独立选并行策略、独立调度、独立扩缩容。** 一个请求的生命周期变成:

```
1. 请求进 prefill 池:并行算完整个 prompt,产出
   (a) 第 1 个生成 token
   (b) 这条序列每一层每个位置的 KV cache
2. 把 KV cache 从 prefill 池的 GPU 显存传到 decode 池的 GPU 显存
3. decode 池接手,基于收到的 KV cache 自回归吐后续 token
```

这样 prefill 池可以为 compute-bound 优化到极致,decode 池可以堆大 batch 把带宽利用率拉满,**两个 SLO 解耦,可以分别优化、分别配比资源**。代表系统:**DistServe**(把 prefill/decode 解耦并联合优化资源配比与并行度)和 **Splitwise**(微软,把两阶段拆到不同机器、甚至可以用不同型号的 GPU——prefill 用算力强的,decode 用带宽性价比高的)。

### 2.4 真正的代价:KV cache 的搬运开销,以及它什么时候划算

天下没有免费的解耦。把 prefill 和 decode 拆开,中间多了一步**把 KV cache 跨 GPU(可能跨机器)传输**。这一步的开销必须算清楚,否则解耦可能是负优化。

**先算 KV cache 要传多大。** 用第 10 章的显存公式,一条序列的 KV cache 字节数:

```
KV_bytes = 2 · L · H_kv · d_head · S · P
  2      : K 和 V 两份
  L      : 层数
  H_kv   : KV 头数(MHA 下 = 注意力头数;GQA 下小很多,见第六节)
  d_head : 每头维度
  S      : prompt 长度(prefill 产出的序列长)
  P      : 每个数的字节(FP16 = 2)
注意 L · H_kv · d_head 在 MHA 下 = L · d_model
```

举个量级例子(具体到某模型的精确值待核,这里只为建立量级感):设 L = 80、d_model = 8192(即 MHA 下 H·d_head = 8192)、P = 2、prompt 长 S = 2048:

```
单层单 token 的 K+V = 2 · 8192 · 2 = 32768 Byte ≈ 32 KB
单 token 全层      = 32 KB · 80 = 2.56 MB
整条 prompt(2048) = 2.56 MB · 2048 ≈ 5.2 GB
```

**几个 GB 的 KV cache 要在两个池之间搬一次。** 划不划算,取决于传输时间和它换来的收益的比值:

```
传输时间 T_xfer ≈ KV_bytes / 互联带宽
  · 同机 NVLink:带宽达数百 GB/s 量级 → 5 GB 传输约 10ms 量级(数值待核于具体卡/拓扑)
  · 跨机 InfiniBand/RoCE:带宽低一个量级 → 同样数据可能要几十 ms
```

**权衡的本质:** T_xfer 是被加进 TTFT 的一次性成本(用户等首 token 时多等的)。它要和"解耦带来的 decode 池吞吐提升、prefill 池延迟下降"去比。**收益方:** decode 池不再被 prefill 打断,TPOT 更稳更低,且可堆更大 batch 提吞吐;prefill 池可专注压 TTFT。**当请求够长、QPS 够高、解耦带来的吞吐/延迟收益超过 KV 搬运成本时,解耦净赚;当请求很短(prefill 本来就快、KV 也小)、并发很低时,搬运的固定开销可能盖过收益,不如不拆。**

几个工程上压低 T_xfer 的关键手法,理解它们你才算懂这套系统:

- **layer-wise 流式传输(传输与计算重叠):** 不必等整条序列 prefill 全做完再传。第 l 层的 KV 一算出来就开始往 decode 池传,同时 prefill 池继续算第 l+1 层。**把传输藏在计算后面,T_xfer 的大部分被 overlap 掉**,加进 TTFT 的只剩最后一层的尾巴。这是 DistServe/Splitwise 类系统的关键优化,新手最容易忽略——以为传输是串行附加的,其实绝大部分能重叠。
- **拓扑感知放置:** 尽量让配对的 prefill GPU 和 decode GPU 走 NVLink 而非跨机网络,带宽差一个数量级直接决定 T_xfer。
- **传输时顺手量化:** 传 KV 时降到 FP8 甚至更低,传输字节数减半(decode 端再决定是否反量化),这和第六节的 KV 量化是同一招的复用。

> **解耦推理一句话总结:** 它不是"把一件事拆成两半"这种工程洁癖,而是承认 prefill(compute-bound)和 decode(memory-bound)是被两种硬件资源卡住的、最优配置相反的两个负载,**强行同居必然互相挤占 SLO**;拆开后各自调到最优,代价是中间多一次 KV 搬运——而这次搬运的大部分可以靠 layer-wise overlap 藏掉。判断该不该解耦,就是比"搬运成本"和"两个 SLO 各自优化的收益"。

## 三、万卡训练容错:当故障从"事故"变成"天气"

### 3.1 第一性原理:为什么规模本身制造故障

单张 GPU、单根光缆、单个电源,各自都相当可靠。问题是**可靠性不会随规模保持,而是随规模衰减**。这是个纯概率事实,必须算给你看。

设每个组件(GPU + 其网卡 + 链路等)的平均故障间隔为 MTBF_single(比如假设单卡平均 N 万小时坏一次)。假设故障近似独立,系统有 M 个这样的组件,**只要任意一个坏,整个同步训练就被打断**(因为数据并行 all-reduce、流水线并行都要求全员在场)。则系统级 MTBF:

```
故障是独立泊松过程时,失效率可加:
  系统失效率 λ_sys = M · λ_single = M / MTBF_single
  系统 MTBF_sys   = MTBF_single / M
```

**核心结论:系统平均故障间隔与组件数成反比。** 把单组件的 MTBF 除以一万,无论单组件多可靠,万卡系统的 MTBF 都会塌到很小。**具体数值:** 业界多次公开报告(如大型 LLM 训练日志)显示,万卡级集群的故障间隔在**小时到几十分钟量级**,即一天会发生多次中断(精确数字因集群而异,待核)。**这意味着:在万卡上,"训练几个月不出故障"是不可能事件,故障是稳态背景噪声,必须把"经常坏"当成系统的常态假设来设计,而不是当成需要人工介入的异常。**

这个结论直接否定了小规模时代的两个习惯:(1) "出错了人工重启就行"——在万卡上人工根本跟不上故障频率;(2) "每隔几小时存一次 checkpoint 就够"——故障间隔可能比 checkpoint 间隔还短,会反复丢进度。容错必须自动化、必须高频、必须便宜。

### 3.2 故障的三种形态与对应手段

把容错拆成三个子问题,每个有不同的机制:

| 故障形态 | 表现 | 对应手段 |
|---|---|---|
| **硬故障(fail-stop)** | 卡掉线、进程崩、机器宕 | 高频 checkpoint + 故障检测 + 自动重调度 |
| **慢节点(straggler)** | 卡没坏但变慢(降频、ECC 重试) | 检测 + 隔离/绕开,不一定重启 |
| **静默数据损坏(SDC)** | 结果悄悄算错,无任何报错 | 冗余计算 / 一致性校验,最难 |

下面分别讲透前两类的关键机制(checkpoint 与重调度),SDC 单列一节因为它最反直觉、最致命。

### 3.3 增量 / 异步 checkpoint:为什么不能"停下来存盘"

**朴素 checkpoint 的问题:** 训练状态(模型权重 + 优化器状态 + 梯度动量)在大模型里极大。Adam 优化器要存一阶、二阶动量,**优化器状态通常是权重的 2 倍**(FP32 下),加上权重本身,总 checkpoint 体积可达数 TB。

```
朴素同步 checkpoint:
  1. 所有 GPU 停止计算(barrier)
  2. 把数 TB 状态从 GPU 显存拷到 CPU 内存,再写到分布式存储
  3. 写完才继续训练
  停顿时间 ≈ 状态体积 / 存储写带宽,可能数分钟
```

万卡上故障间隔可能就几十分钟,如果每次存盘停几分钟、还要存得很频繁,**纯停顿开销就能吃掉相当比例的训练时间**。两个关键优化把停顿压到接近零:

**(1) 异步 checkpoint(把"拷出"和"落盘"分离,落盘不阻塞训练):**

```
关键观察:训练继续需要的是 GPU 显存里的状态,而落盘是慢的那一步。
做法:
  阶段 A(快,需短暂阻塞或用副本):把状态从 GPU 显存快照到 CPU 内存(走 PCIe,相对快)
  阶段 B(慢,异步):后台线程把 CPU 内存里的快照写到持久化存储
  → 训练在阶段 A 完成后立即继续,阶段 B 与后续训练计算重叠
  → GPU 的可见停顿 ≈ 只剩阶段 A(显存→内存拷贝),而非整个落盘
```

进一步可以用 GPU 到 GPU 的方式做 **in-memory / 对等 checkpoint**:把自己的状态复制一份到另一台机器的内存(而非慢存储),恢复时从邻居内存拉,比从分布式文件系统读快得多。代价是占用额外内存和一次网络传输。

**(2) 增量 checkpoint(只存变化的部分):**

```
观察:相邻两次 checkpoint 之间,不是所有状态都变了同样多;
      且很多框架按 shard(分片)管理状态。
做法:对每个 shard 维护指纹/版本,只把自上次 checkpoint 后真正改动的 shard 写出。
收益:写出字节数 ≈ 变化比例 · 全量,落盘 IO 显著下降。
注意:这里的"增量"更多是工程实现层面的去重/分片差量;
      Adam 状态每步都在变,纯数学上的"差量很小"不一定成立——
      增量的主要红利来自分片级调度、与异步结合摊平 IO,而非"内容几乎没变"。(待核:不同框架对增量的定义差异较大)
```

> **异步与增量解决的是不同维度:异步压的是"停顿时间"(让落盘不阻塞训练),增量压的是"写出字节数"(让 IO 量下降)。两者正交,通常叠加使用。万卡训练的目标是把 checkpoint 的可见开销摊到接近零,从而可以高频(比如每隔很短就存一次)存盘,把单次故障的进度损失压到最小。**

### 3.4 故障检测与自动重调度

检测到故障、并自动恢复,要解决三个问题:

- **检测(detection):** 靠心跳超时、NCCL/集合通信卡住超时、健康探针(温度、ECC 错误计数、链路状态)。集合通信 hang 是最常见信号——一个 GPU 掉线,all-reduce 会无限等,框架要有超时机制把"卡住"转成"明确的故障事件"而非干等。
- **重调度(rescheduling):** 把故障节点从集群剔除,用热备节点顶上(spare node pool),或在剩余健康节点上重新规划并行布局(可能需要调整并行度)。关键是**让重组后的拓扑仍然合法**——比如流水线级数、TP 组大小要能整除新的节点数。
- **恢复(recovery):** 从最近的 checkpoint 重新加载,**所有节点回滚到同一个一致的步**。这就是为什么 checkpoint 必须高频且便宜——恢复点越新,重算的进度越少。

工业界把这套做成自愈循环:**检测 → 隔离坏节点 → 拉备用节点 → 重组并行拓扑 → 从最近 checkpoint 恢复 → 继续训练**,全程无人工。能把这个循环的端到端时间压到几分钟内,万卡训练才有效率可言。代表性的工程实践见 Meta 的 OPT 训练日志(详细记录了人工时代的痛)和后续 Llama 系列、各大厂的弹性训练框架(具体框架名与机制细节待核)。

### 3.5 静默数据损坏(SDC):最可怕的那一类

**SDC(Silent Data Corruption)** 指硬件算错了但**没有任何报错**:某个乘加单元因制造缺陷或老化,在特定输入下偶尔返回错误结果;ECC 没覆盖到的路径上一个 bit 翻转。它不会让程序崩溃,而是让你的梯度、权重里悄悄混入错误值。

**为什么它最可怕:** 硬故障会立刻暴露(进程死、通信 hang),你知道要恢复;SDC **不暴露**,错误的梯度被 all-reduce 平均后污染所有副本,被优化器累积进权重,可能训练了很久才发现 loss 异常(甚至发现不了,只是模型质量悄悄变差)。**等你察觉时,污染可能已经写进了很多个 checkpoint,无从回滚到干净点。**

检测手段(都不便宜,这是 SDC 的根本难处):

- **冗余计算:** 同一份计算在两处算,比对结果。代价是算力翻倍或部分冗余(只抽样关键计算冗余算)。
- **不变量/一致性校验:** 利用本应成立的数学关系做廉价 check。例如数据并行下,all-reduce 前各副本对**相同输入**应得相同的某些统计量(如梯度范数),不一致就报警;或对关键张量算校验和跨副本比对。
- **确定性重放比对:** 怀疑某步时,在另一硬件上确定性重算同一步,比对输出。

> **SDC 把"正确性"从"假设硬件不会算错"拉回成"必须主动验证硬件算对了"。** 在万卡规模,即使单卡 SDC 概率极低,乘以一万张卡和几个月,撞上的期望次数不可忽略。这是为什么前沿训练系统开始把 SDC 检测当成一等公民——它不是性能问题,是**正确性**问题,而错误的模型可能比慢的训练代价大得多。这一节没有标准解,是当前最开放的战线之一。

## 四、长上下文 KV 压缩:线性爆炸的四种正交打法

### 4.1 问题:KV cache 随上下文线性炸

回到第 10 章的显存公式,固定模型和 batch,KV cache 字节数对序列长度 S 是**线性**的:

```
KV_bytes = 2 · L · H_kv · d_head · S · P  ∝  S
```

当上下文从 2K 涨到 128K(64 倍)甚至 1M(512 倍),KV cache 也涨同样的倍数。**它会迅速超过模型权重本身,成为显存的绝对主角,并直接卡死最大上下文长度和最大并发数。** 长上下文时代,谁能把 KV cache 压下去,谁就能服务更长的上下文、更多的并发。

压它有四个**正交**(可叠加)的方向,对应公式里不同的因子:

```
KV_bytes = 2 · L · H_kv · d_head · S · P
                  ↑                ↑   ↑
          减 H_kv(GQA/MQA)    删 S   降 P
              架构侧          (驱逐/稀疏)(量化)
   外加一个不在公式里的维度:跨请求复用相同前缀(prefix caching)
```

逐个讲透。

### 4.2 打法一:KV 量化(降 P)

最直接:把 KV cache 的精度从 FP16(P=2)降到 INT8(P=1)甚至 INT4(P=0.5),字节数等比缩小。

```
FP16 → INT8 : KV_bytes 减半
FP16 → INT4 : KV_bytes 降到 1/4
```

**机制与坑:** KV 量化和权重量化(第 12 章)是同一套思想——用低位整数表示 + scale/zero-point 反量化,但有一个 KV 特有的关键点:**K 和 V 的数值分布不同,且 K 里常有少数"离群通道"(outlier channels)**(某些维度数值远大于其它,和注意力打分机制有关)。直接 per-tensor 量化会被离群值拉坏整体精度。常见做法是 **per-channel/per-token 分组量化**(对 K 沿 channel 维分组定 scale,对 V 沿 token 维),把离群值的影响局限在它自己的组里。代价是要存额外的 scale,且分组越细额外开销越大——又是一个精度 vs 开销的权衡。

### 4.3 打法二:KV 驱逐 / 稀疏(删 S)

**核心赌注:不是所有历史 token 的 KV 都同等重要;只保留"重要"的,删掉"不重要"的,用近似换显存。** 这是 lossy 的(和量化的轻微 lossy 不同,这是结构性地丢弃信息),所以"怎么判断重要"是全部学问。两条代表路线:

**(1) H2O(Heavy-Hitter Oracle)——按累计注意力分数驱逐。**

```
观察:在注意力矩阵里,少数 token(称 heavy hitters)持续获得很高的注意力权重,
      大量 token 几乎从不被关注。
做法:为每个被缓存的 token 维护一个累计注意力得分(它被后续 query 关注的总和);
      当 KV 预算满了,驱逐累计得分最低的那些 token 的 KV。
直觉:把显存预算花在"真正被反复关注"的 token 上。
```

**(2) StreamingLLM——固定保留"开头几个 + 最近一窗"。**

```
观察(attention sink,注意力汇聚):自回归模型会把大量注意力莫名其妙地分给
      序列最开头的几个 token(即使它们语义上不重要)——它们成了 softmax 的
      "泄压阀"(模型需要一个地方倾倒多余的注意力质量)。
      去掉最开头的 token,模型性能会断崖式崩溃。
做法:永远保留最前面的少数几个 token(sink) + 最近的一个滑动窗口,中间的全丢。
收益:KV cache 大小被钉死成常数(sink + window),可以处理无限长的流式输入
      而显存不增长。
```

**(3) 架构侧的局部/稀疏注意力:** 直接让模型只看局部窗口 + 少数全局 token(如 sliding window attention、Longformer 式稀疏),从训练阶段就限定每个位置只 attend 到 O(window) 个位置,KV 访问量从 O(S) 降到 O(window)。这是把稀疏内建进模型而非推理时事后删。

**为什么驱逐可能丢关键信息(这是练习要你想透的核心):**

> KV 驱逐的根本假设是"重要性可以提前判断且稳定"。但**重要性是相对于未来 query 的,而未来 query 还没来**。一个 token 现在累计注意力分数很低(过去没人关注它),不代表 100 步之后的某个 query 不需要它——典型如长文档里前面提到一次的关键事实("故事开头说主角叫张三"),中间一直没被关注,得分很低被驱逐;结尾问"主角叫什么",这时需要它,但它已经被删了,模型只能瞎编。StreamingLLM 的滑动窗口同理:**超出窗口的中段信息被结构性地永久丢弃**,所以它擅长"流式聊天"(只需近期上下文),但不适合"长文档精确问答"(需要任意位置的细节)。**驱逐用"过去的注意力模式"预测"未来的需求",这个预测在分布外(out-of-distribution)的 query 上必然失效——丢的不是噪声,可能是低频但关键的信号。** 这是 lossy KV 压缩无法回避的本质风险,工程上只能靠"留足预算、保留 sink、对召回敏感场景关掉驱逐"来缓解,无法根除。

### 4.4 打法三:GQA / MQA(减 H_kv)——架构侧治本

这是最干净、被工业界标配采用的一招,因为它不删信息、不降精度,而是**从模型结构上减少 KV 头的数量**。

```
MHA(标准多头):每个 query 头配一个独立的 K 头、V 头。H_kv = H(查询头数)。
MQA(Multi-Query):所有 query 头共享同一个 K 头、V 头。H_kv = 1。
GQA(Grouped-Query):折中,把 query 头分成 G 组,组内共享一对 K/V 头。H_kv = G。
```

KV cache 直接按 H_kv/H 的比例缩小:

```
设 H = 64 个查询头。
  MHA : H_kv = 64,KV cache 基准 = 1×
  GQA : H_kv = 8 (8 组),KV cache = 8/64 = 1/8
  MQA : H_kv = 1,      KV cache = 1/64
```

**为什么能这么干而几乎不掉点:** 多个 query 头共享 K/V,等于强制它们在"看哪些位置"上更一致。实证表明,**适度分组(GQA)在质量几乎无损的前提下把 KV 砍到 1/8 量级**;MQA 更激进、有时略掉点。代价:K/V 表达能力下降(共享后无法让每个头看完全不同的东西),所以 GQA 取了甜点。**现代主流大模型(Llama 2/3 的大尺寸、Mistral 等)普遍用 GQA**,因为它同时减小了 KV 显存(更长上下文、更大 batch)和 decode 时读 KV 的带宽(decode memory-bound,读 KV 也是访存成本)。这一招还和第二节的解耦呼应:**GQA 让要传输的 KV cache 也小了 H/H_kv 倍**,解耦推理的搬运成本随之下降。

### 4.5 打法四:prefix caching(跨请求复用,不在公式里的维度)

前三招都在压"一条序列的 KV 占多少"。prefix caching 换个角度:**很多请求共享相同的前缀**(同一个长 system prompt、同一份被反复提问的文档、few-shot 例子),它们这段前缀的 KV cache **算出来是完全一样的**——因为 KV 只取决于 token 内容和位置(第 10 章的核心观察)。

```
观察:KV(token_i) 只依赖 token_1..i 的内容和位置,与"哪个请求"无关。
做法:把公共前缀的 KV cache 算一次、存起来(按前缀内容做 key 缓存);
      后续命中相同前缀的请求,直接复用缓存的 KV,跳过这段 prefill。
收益:(a) 省 prefill 算力(公共前缀只算一次);
      (b) 省显存(多个请求物理共享同一份前缀 KV,靠 PagedAttention 的
          block 级共享 + 写时复制实现,见第 10 章);
      (c) 降 TTFT(命中前缀的请求几乎跳过 prefill)。
```

这一招的实现直接建立在第 10 章 PagedAttention 的 block 抽象上:KV 按 block 存储,多个请求的 block table 可以指向同一批物理 block(共享),需要写入时再 copy-on-write 分裂。**它和解耦推理是绝配:** prefix caching 大幅缩短甚至消除热门前缀的 prefill,prefill 池负载骤降。代表实现见 vLLM 的 automatic prefix caching 和 SGLang 的 RadixAttention(用基数树组织共享前缀,具体 API 待核)。

### 4.6 打法五(横切):chunked prefill——回头缝合第二节的干扰

第二节说过,长 prompt 的 prefill 会一次占用大量算力、把同批 decode 请求挤卡。**chunked prefill(分块预填充)** 是不彻底解耦时的一种缓解:

```
不把一个长 prompt 的 prefill 一次算完,而是切成若干 chunk(比如每 chunk 512 token),
每个调度步只算一个 chunk,并把这些 prefill chunk 和正在 decode 的请求混进同一批。
效果:把"一个巨大的 prefill 尖峰"摊成"若干个小块",
      每个调度步的算力占用平滑,decode 请求不再被一次性挤爆,TPOT 更稳。
代价:prefill 被切块后,块之间有调度/kernel 启动开销,单请求 prefill 总延迟略增(TTFT 略升)。
```

> **chunked prefill 与解耦是同一问题(prefill 干扰 decode)的两种答案:** 解耦是"物理隔离,根治但要付 KV 搬运";chunked prefill 是"同居但把 prefill 切碎错峰,缓解但不根治、且略抬 TTFT"。生产里常按规模和 SLO 选其一,或在解耦的 prefill 池内部再用 chunking。这正好把本章三节缝成一张网:**两阶段特性相反(节二)→ 长 prompt 让 prefill 尖峰更猛、KV 更大(节四 4.1)→ 要么解耦、要么 chunk、要么用 GQA/驱逐/量化/prefix 把 KV 这条线一起压下去。**

## 五、设计权衡与常见坑(汇总)

**解耦推理:**
- 坑 1:**以为 KV 传输是纯附加成本而否定解耦。** 实际上 layer-wise 流式传输能把大部分传输 overlap 进计算,真正加进 TTFT 的远小于 KV 总大小 / 带宽。不算 overlap 后的有效开销就下结论,会错判。
- 坑 2:**短请求、低并发场景强行解耦。** prefill 本来就快、KV 本来就小,搬运的固定开销盖过收益,负优化。解耦是"长请求 + 高并发"的红利。
- 权衡:prefill 池和 decode 池的配比不是固定的,取决于 workload 的 prompt 长度 / 生成长度比、QPS——这正是练习 1。

**万卡容错:**
- 坑 3:**checkpoint 频率按"存盘很贵"来定(存得稀)。** 一旦用了异步 + 增量把可见开销压到接近零,就应该高频存,把单次故障的进度损失压到最小。频率应由"恢复时能接受丢多少进度"反推,而非由"存盘多贵"决定。
- 坑 4:**只防硬故障,不防 SDC。** 硬故障会暴露,SDC 不会,后者污染权重的代价可能更大。万卡规模 SDC 期望次数不可忽略,必须有主动校验。
- 权衡:SDC 冗余检测越严越费算力,实践中只对关键计算抽样冗余,在"漏检风险"和"算力开销"间取舍。

**长上下文 KV 压缩:**
- 坑 5:**对召回敏感的任务(长文档精确问答、检索、代码库理解)开 KV 驱逐。** 驱逐用过去注意力预测未来需求,在"前面提一次、后面才问"的模式上必然丢关键信息(4.3)。这类任务应保留全量 KV 或只用量化/GQA 这种不删信息的招。
- 坑 6:**StreamingLLM 去掉 attention sink。** 开头几个 sink token 是 softmax 的泄压阀,删了模型性能崩溃——这是反直觉但被反复验证的现象,务必保留。
- 坑 7:**KV 量化用 per-tensor 一刀切。** K 的离群通道会拉坏整体,要 per-channel/per-token 分组。
- 权衡:四招正交可叠加(GQA + 量化 + prefix caching 常一起上),但每招都有自己的精度/开销代价,叠加时代价也叠加;驱逐是其中唯一结构性 lossy 的,最该谨慎。

## 六、动手练习

**练习 1(估算 + 设计题,核心)。** 为一个解耦推理系统设计 prefill 池和 decode 池的 GPU 配比,并说明依据。给定:平均 prompt 长 S_in = 2000、平均生成长 S_out = 500;单卡 prefill 吞吐 ≈ X token/s(compute-bound)、单卡 decode 吞吐 ≈ Y token/s/序列(memory-bound),X、Y 自设合理量级。
- 提示:稳态下两池要"产销平衡"——prefill 池每秒产出的"已 prefill 请求数"要等于 decode 池每秒能消化的请求数。先写出"prefill 池处理一个请求耗时 ≈ S_in / X","decode 池处理一个请求耗时 ≈ S_out / Y"(忽略 KV 传输或把它加进 prefill 侧),再令 N_prefill / (S_in/X) = N_decode / (S_out/Y) 解出 N_prefill : N_decode。讨论:当 S_in/S_out 变大(prompt 越来越长)时配比怎么移动?再补一句 KV 传输开销和 GQA 如何影响这个比。

**练习 2(分析题,核心)。** 讨论 KV 驱逐为什么可能丢关键信息,并构造一个 H2O 和 StreamingLLM 都会失败的具体 query 例子。
- 提示:从"重要性是相对未来 query 的,而未来 query 未知"出发。例子:一篇 100K token 文档,第 50 token 处写"密码是 7391",此后到第 99000 token 一直没被任何内容引用(累计注意力分数极低),最后一个 query 是"文档里的密码是多少"。说明 H2O(低分被驱逐)和 StreamingLLM(超出滑窗被丢)各自如何丢掉第 50 token 的 KV,导致模型无法回答。再说明 GQA 和 KV 量化为什么不会犯这个错(它们不删 token)。

**练习 3(估算题)。** 用第三节的系统 MTBF 公式估算:若单组件 MTBF = 5 年(≈ 43800 小时),分别在 M = 1000、10000、50000 卡规模下,系统 MTBF 是多少?若一次 checkpoint 恢复(含检测+重调度+加载)需 5 分钟,在 M=50000 下,假设故障一发生就触发恢复,纯故障恢复占用的时间比例约是多少?
- 提示:MTBF_sys = 43800 / M 小时。M=50000 时 MTBF_sys ≈ 0.876 小时 ≈ 52.6 分钟。恢复占比 ≈ 恢复时长 / (MTBF_sys + 恢复时长) = 5 / (52.6 + 5)。再思考:若 checkpoint 间隔大于 MTBF_sys 会发生什么(每次故障都要重算超过一个 checkpoint 间隔的进度)。

**练习 4(推导题)。** 用第六节 GQA 公式量化收益。模型 L=80、H=64、d_head=128、P=2、S=128000、batch=1。分别算 MHA(H_kv=64)、GQA(H_kv=8)、MQA(H_kv=1)三种下单条序列的 KV cache 显存。再结合解耦推理的 KV 传输:若该 KV 要跨机以 100 GB/s(待核)传输,三种方案的传输时间各是多少?说明 GQA 同时省了"显存"和"传输/decode 读带宽"两笔账。
- 提示:KV_bytes = 2·L·H_kv·d_head·S·P。MHA:2·80·64·128·128000·2 字节,算出 GB 量级你会被长上下文的 KV 体量震到。GQA 是它的 1/8,MQA 是 1/64。传输时间 = bytes / 带宽。

## 七、源码 / 论文导读

- **解耦推理:** DistServe(Zhong et al.,*DistServe: Disaggregating Prefill and Decoding for Goodput-optimized LLM Serving*)——重点读它怎么把 prefill/decode 解耦后**联合优化两池的并行度和资源配比**,以及对 KV 传输开销与 goodput(满足 SLO 的有效吞吐)的建模。Splitwise(Patel et al., 微软)——重点看它把两阶段拆到**不同型号 GPU** 的论证(prefill 用算力强的、decode 用性价比高的)和 KV 传输优化。两篇对照读,能看清"为什么拆"和"拆了怎么传 KV"。
- **长上下文 KV 压缩:** H2O(Zhang et al.,*H2O: Heavy-Hitter Oracle for Efficient Generative Inference of LLMs*)——读它对 heavy-hitter 现象的统计观察和驱逐策略;StreamingLLM(Xiao et al.,*Efficient Streaming Language Models with Attention Sinks*)——重点读 **attention sink** 的发现(去掉开头 token 性能崩溃的实验),这是全章最反直觉的实证。GQA(Ainslie et al.,*GQA: Training Generalized Multi-Query Transformer Models from Multi-Head Checkpoints*)和 MQA(Shazeer,*Fast Transformer Decoding: One Write-Head is All You Need*)——读 GQA 怎么从已有 MHA checkpoint 通过"平均 K/V 头"低成本转换得到。
- **开源实现(读代码):** vLLM 的 automatic prefix caching、chunked prefill 配置项,和 SGLang 的 RadixAttention(基数树管理共享前缀)是 prefix caching 的工业范本;KV 量化看各推理引擎的 `kv_cache_dtype`(如 fp8)相关实现。具体文件路径、配置项名称、API 因版本快速演进,**务必待核于你手头的版本**,本章给的是机制不是固定 API。
- **万卡容错:** Meta 的 *OPT-175B logbook*(公开训练日志)——读它记录的真实故障频率和人工恢复之痛,理解为什么要自动化;后续各大厂的弹性训练(elastic training)/ 自愈框架的设计文档,以及关于 SDC 的硬件可靠性报告(具体出处待核)。

## 八、小结与承上启下

这一章把三条最活跃的系统战线钉在了同一个第一性原理上:

- **解耦推理** 根植于"prefill compute-bound、decode memory-bound,被两种硬件资源卡住、最优配置相反"——所以拆开各自优化,代价是 KV 搬运(可被 layer-wise overlap 大部分藏掉)。它直接复用了前几章"两阶段算术强度差异"的结论。
- **万卡容错** 根植于"系统 MTBF = 单组件 MTBF / 组件数,规模本身制造故障"——所以故障是常态,必须用异步(压停顿)+ 增量(压 IO)checkpoint 做到高频低开销、自动检测重调度、并把 SDC 这种"悄悄算错"当成一等正确性问题。
- **长上下文 KV 压缩** 根植于"KV cache ∝ 上下文长度,长上下文时代它是显存绝对主角"——四个正交方向(量化降 P、驱逐删 S、GQA 减 H_kv、prefix caching 跨请求复用)各打公式里一个因子,其中只有驱逐是结构性 lossy 的,会因"用过去注意力预测未来需求"而丢关键信息。

**它们彼此咬合:** GQA 同时减小 KV 显存、decode 读带宽、和解耦的传输量;prefix caching 直接削 prefill 池负载、建立在 PagedAttention 的 block 共享上;chunked prefill 和解耦是 prefill 干扰 decode 的两种解法。**理解这张网,你看任何一个 2024–2025 的推理/训练系统(vLLM、TensorRT-LLM、SGLang、各家弹性训练框架),都能一眼拆出它每个设计在打哪堵墙、复用了哪个底层抽象、付了什么代价。**

在整门课里的位置:这是推理/训练系统部分的**集成与前沿**章。前面的章节给了你单点武器(并行、重计算、FlashAttention、PagedAttention、投机解码、量化),这一章告诉你**在集群规模和长上下文时代,这些武器如何被组织成系统、还有哪些墙没被推倒**。它也是承上启下的枢纽:往回,它把前面所有单点优化的第一性原理(算术强度、显存公式、MTBF)收束成系统级判断力;往后(若有 MoE / Agent / 多模态系统等后续专题),解耦、容错、长上下文是它们共同的底座——任何更大的系统都要先解决"两阶段怎么放、故障怎么扛、上下文怎么省"这三件事。**学完这一章,你应该具备的不是记住了几个系统名字,而是拿到一个新系统时,能自己把它拆回这三堵墙、判断它的设计取舍是否成立的能力——这正是工业界专家和调包工程师的分界线。**
