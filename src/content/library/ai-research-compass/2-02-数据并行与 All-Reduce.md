> 这一章把你从"会在单卡上跑一次训练迭代"带到"理解为什么 8 卡、64 卡、上千卡能近乎线性地一起训练同一个模型"。核心是搞透一个被几乎所有分布式训练框架(PyTorch DDP、Horovod、DeepSpeed、Megatron)埋在最底层的通信原语——all-reduce——以及它的带宽最优算法 ring all-reduce 为什么"通信量与卡数近似无关"。读完你能手推任意拓扑下的通信量、能解释 DDP 为什么不会随卡数线性变慢、能讲清梯度累积与大 batch 收敛的关系。

## 一、动机:为什么"加卡"这件事没那么简单

第 01 章我们把一次训练迭代拆到了字节级:前向 + 反向 ≈ `6ND` FLOPs,Adam 混合精度下每参数约 16 字节。现在的问题变了——**单卡能放下模型,但训得太慢**。一个 7B 模型在单张 A100 上训完一遍万亿级 token 需要以"年"计,这显然不能接受。最朴素的想法是:**加卡,让多张卡一起算同一个模型,把吞吐堆上去。**

但"加卡"不是免费的。如果 8 张卡只能换来 3 倍速度,那这笔买卖就很差。**分布式训练的全部难点,可以浓缩成一句话:如何让 N 张卡的有效吞吐尽可能接近 N 倍单卡吞吐?** 衡量这件事的指标叫**扩展效率(scaling efficiency)**:

```
扩展效率 = (N 卡实测吞吐) / (N × 单卡吞吐)
```

理想是 1.0(线性扩展)。掉到 0.5 意味着一半的卡在白烧电。**吞吐被吃掉的地方只有一个根源:卡之间必须交换信息,而通信不是免费的。** 数据并行里,这个"必须交换的信息"就是梯度——每张卡各自算了一份梯度,但参数更新必须用所有卡梯度的平均值,否则各卡的模型会越训越不一样。

所以这一章真正要回答的是两个层层递进的问题:

1. **机制层:** 数据并行到底怎么组织?为什么同步的是梯度而不是参数或激活?
2. **算法层:** "把 N 张卡的梯度求和再发回每张卡"这个操作(就是 all-reduce),最省带宽的做法是什么?省到什么程度?为什么?

第二个问题的答案——ring all-reduce 的 `2(N-1)/N · M` 通信量与卡数近似无关——是过去十年大规模训练能成立的数学基石之一。我们会把它从头推到尾,并证明它在带宽意义上是**最优**的(只差一个常数因子)。

**给系统老兵的迁移提示:** all-reduce 是 MPI 时代就有的经典集合通信原语,ring 算法也不是深度学习发明的(出自 HPC 社区,Baidu 2017 年的工作把它带火到深度学习)。如果你做过 MPI、做过分布式存储的副本同步,这里的直觉是现成的——你缺的只是"为什么是梯度"这层 ML 常识。

## 二、数据并行的机制:每卡一份完整模型

### 2.1 数据并行做了什么

**数据并行(Data Parallelism,DP)的定义:每张卡持有一份完整的模型副本,把一个 batch 的数据切成 N 份(每卡一份子 batch),各卡独立做前向和反向,然后把各卡算出的梯度同步成全局平均,再各自用这个平均梯度更新参数。**

为什么这样就等价于"在一个大 batch 上训练"?这需要一个推导。设全局 batch 有 B 个样本,损失是逐样本损失的平均:

```
L(θ) = (1/B) · Σ_{i=1..B} ℓ(xᵢ; θ)
```

梯度是线性算子,可以拆成各子集之和。把 B 个样本均分到 N 张卡,每卡 b = B/N 个样本,卡 k 持有样本集 Sₖ:

```
∇L(θ) = (1/B) · Σ_{i=1..B} ∇ℓ(xᵢ; θ)
      = (1/B) · Σ_{k=1..N} [ Σ_{i∈Sₖ} ∇ℓ(xᵢ; θ) ]
      = (1/N) · Σ_{k=1..N} [ (1/b) · Σ_{i∈Sₖ} ∇ℓ(xᵢ; θ) ]   ← 因为 B = N·b
      = (1/N) · Σ_{k=1..N} gₖ
```

其中 `gₖ = (1/b) · Σ_{i∈Sₖ} ∇ℓ` 正是**卡 k 自己在子 batch 上算出的局部梯度**。

这就是数据并行能成立的全部数学依据:**全局梯度 = 各卡局部梯度的算术平均。** 每张卡只要把自己的 gₖ 算出来,然后大家做一次"求平均",得到的结果跟单卡在整个 B 样本上算梯度**数值完全一致**(忽略浮点求和顺序带来的微小差异)。因为各卡模型参数 θ 在每步开始时都相同,更新后也相同,所以模型副本始终保持一致——这叫 **DP 的不变量(invariant):所有副本的参数恒等。**

**关键结论:数据并行需要在每个训练步同步的,只有梯度;参数从不需要传输(因为各卡用相同的平均梯度做相同的更新,自然保持一致),激活值也从不跨卡(每卡在自己的子 batch 上前向反向自给自足)。** 这是数据并行通信量小、最容易扩展的根本原因——对比后面几章的张量并行(要传激活)、ZeRO-3(要传参数),数据并行是通信最轻的并行方式。

### 2.2 为什么是"求平均"而不是"求和"

注意上面推导里出现的是 `(1/N) · Σ gₖ`,即平均。但 all-reduce 原语本身做的是**求和**(reduce 的语义是 sum)。差一个 1/N 因子,工程上通常这样处理:要么 all-reduce 求和后各卡自己除以 N,要么在算 gₖ 时就让每卡用 `(1/B)` 而非 `(1/b)` 归一化(等价于预先除好)。PyTorch DDP 的默认行为是**对梯度做求和的 all-reduce,然后框架在内部除以 world_size**(待核:不同框架/不同 reduce op 的归一化时机略有差异,但数学等价)。

为什么不能简单地"求最大"或"取某一卡的"?因为只有求和/求平均才保留了全部 B 个样本的梯度信息,这是上面线性可加性推导决定的,不是约定。

### 2.3 同步 vs 异步:为什么主流是同步

历史上有过**异步数据并行**(各卡算完梯度立刻推给参数服务器更新,不等别人),但它有个致命问题:**梯度过期(stale gradient)**——卡 A 基于 θₜ 算的梯度,推上去时参数已经被卡 B 更新成了 θₜ₊₅,这个梯度方向已经"过时",会损害收敛、引入难以复现的噪声。所以**现代大模型训练几乎清一色用同步数据并行**:每步所有卡用同一份参数,算完梯度做一次 all-reduce,大家拿到完全相同的平均梯度,做完全相同的更新。代价是要等最慢的那张卡(straggler 问题),但换来的是确定性和良好收敛。

## 三、All-Reduce 的语义与朴素实现的代价

### 3.1 All-Reduce 到底是什么

**All-Reduce 的语义:** N 个进程各持有一个等长的数组(在我们这里就是梯度向量),做一次 all-reduce 后,**每个进程都拿到"所有进程数组逐元素求和(或其他归约 op)的结果"**。注意两个"all":一是要把所有进程的数据归约进去,二是结果要分发给所有进程(人人都有)。

把它拆成两个更基本的原语有助于理解(后面 ring 算法正是这么干的):

- **Reduce-Scatter(归约-散射):** 归约后的结果被切成 N 块,**第 k 块只发给进程 k**。做完每个进程各持有最终结果的 1/N。
- **All-Gather(全收集):** 每个进程持有数据的一块,做完每个进程都拿到所有块拼起来的完整数据。

**恒等式:All-Reduce = Reduce-Scatter + All-Gather。** 这个分解是 ring 算法的灵魂。先把"求和"这件重活用 reduce-scatter 分摊到每张卡(每卡只负责把某 1/N 块加全),再用 all-gather 把各卡手里的那 1/N 块结果广播给所有人。

### 3.2 通信代价怎么算:α-β 模型

要比较算法优劣,先要有代价模型。经典的点对点通信代价模型(也叫 **Hockney 模型**)是:

```
传一条 n 字节的消息耗时 = α + n / β
```

- **α(alpha):延迟项(latency)**,发一条消息的固定开销,与消息大小无关(链路建立、协议头、kernel 启动等)。
- **β(beta):带宽(bandwidth)**,单位时间能传多少字节;`n/β` 是传输项(transfer)。

**两个极端:**
- **小消息、多步骤** → α 主导(latency-bound),步数 step count 是关键。
- **大消息、少步骤** → `n/β` 主导(bandwidth-bound),总传输字节数是关键。

梯度同步通常是**大消息**(一个 7B 模型的 fp16 梯度就是 14 GB),所以**带宽项主导**——这就是为什么我们整章都在死磕"每张卡总共要传多少字节"。但要记住,小张量、超多卡的场景里 α 会反扑(见第六节的坑)。

### 3.3 朴素 all-reduce 的劣势

**朴素做法一:reduce-to-root + broadcast(参数服务器思路)。** 选一张卡当 root,所有 N-1 张卡把各自 M 字节梯度发给 root,root 求和后再把 M 字节结果广播回 N-1 张卡。

```
root 收到的字节 = (N-1) · M       ← 入向
root 发出的字节 = (N-1) · M       ← 出向
root 单侧带宽承担 ≈ (N-1) · M = O(N·M)
```

**致命问题:root 是带宽瓶颈,总耗时随 N 线性增长。** 加卡不但不加速,反而让 root 更堵——这是参数服务器架构在大规模下扩展性差的根因。完成时间约 `≈ (N-1)·M/β`(忽略 α 和能否双向并行的细节)。

**朴素做法二:每卡向所有其他卡发自己的梯度(全交换)。** 每卡发 `(N-1)·M`,总通信量 `O(N²·M)`,更糟。

**朴素做法三:递归倍增 all-reduce(recursive doubling)。** 这是个聪明的对数算法:第 1 步与距离 1 的伙伴交换全量并求和,第 2 步与距离 2 的伙伴交换,…,共 `log₂N` 步,每步每卡收发 M 字节。

```
recursive doubling 每卡传输 ≈ log₂N · M   (传输项)
步数 = log₂N            (延迟项 α 极少,只有 log₂N 次)
```

它的**延迟极优**(只有 log₂N 步,小消息时无敌),但**带宽不优**:每卡总共传了 `log₂N · M` 字节,随 N 缓慢增长(8 卡 3M、1024 卡 10M)。对大梯度这就浪费了。

**结论对比表(每卡传输字节量级,带宽视角):**

| 算法 | 每卡传输字节 | 步数(α 项) | 随 N 增长? |
|---|---|---|---|
| reduce-to-root + broadcast | O(N·M)(root) | O(1)~O(N) | 是,线性,且瓶颈集中 |
| 全交换 | O(N·M) | O(1) | 是,线性 |
| recursive doubling | log₂N · M | log₂N | 是,对数 |
| **ring all-reduce** | **2(N-1)/N · M ≈ 2M** | **2(N-1)** | **几乎不随 N 增长** |

最后一行就是我们要证明的主角。**注意:ring 的延迟项最差(2(N-1) 步),但带宽项最优(≈2M,封顶);recursive doubling 反过来。这个 trade-off 是后面"坑"的核心。**

## 四、核心推导:Ring All-Reduce 与它的带宽最优性

### 4.1 拓扑与分块

把 N 张卡排成一个**逻辑环**:0 → 1 → 2 → … → N-1 → 0,每张卡只跟它的**右邻居**(后继)和**左邻居**(前驱)直接通信。**注意这是逻辑环,不要求物理上真的连成环**——只要每张卡能同时进行"向右发"和"从左收"两个方向的传输(全双工链路,如 NVLink、现代网卡都支持),环就能跑满带宽。

把每张卡上的 M 字节梯度向量**等分成 N 块**,每块 M/N 字节。记卡 k 上的第 j 块初始值为 `aₖ,ⱼ`。我们要的最终结果是:每块 j 的全局和 `Sⱼ = Σ_{k} aₖ,ⱼ`,且每张卡都要拿到全部 N 块的 `Sⱼ`。

Ring all-reduce 分两个阶段,各 N-1 步。

### 4.2 阶段一:Reduce-Scatter(N-1 步)

**目标:** 做完之后,卡 k 持有第 `(k+1) mod N` 块(或某个约定块)的**完整全局和** Sⱼ。也就是把"加全 N 块"这件事,让 N 张卡每人负责加全一块。

**机制(逐步):** 每一步里,每张卡把自己"当前负责累加的那一块"发给右邻居,同时从左邻居收到一块,把收到的块**加到自己对应位置**上。关键在于每步发送和接收的块索引**错位滚动**,使得某一块的部分和沿着环一路被累加。

用 N=3 走一遍最直观。初始(行=卡,列=块):

```
        块0      块1      块2
卡0:   a0,0     a0,1     a0,2
卡1:   a1,0     a1,1     a1,2
卡2:   a2,0     a2,1     a2,2
```

约定:第 t 步,卡 k 把它的第 `(k - t) mod N` 块发给右邻居 k+1。

**Reduce-Scatter 第 1 步(t=0):** 卡 k 发第 `k` 块给 k+1。
- 卡0 把 块0 发给卡1 → 卡1 的 块0 变成 a1,0 + a0,0
- 卡1 把 块1 发给卡2 → 卡2 的 块1 变成 a2,1 + a1,1
- 卡2 把 块2 发给卡0 → 卡0 的 块2 变成 a0,2 + a2,2

```
        块0          块1          块2
卡0:   a0,0         a0,1        a0,2+a2,2
卡1:  a1,0+a0,0     a1,1         a1,2
卡2:   a2,0       a2,1+a1,1      a2,2
```

**Reduce-Scatter 第 2 步(t=1):** 卡 k 发第 `(k-1) mod N` 块给 k+1。
- 卡0 发 块2(=a0,2+a2,2)给卡1 → 卡1 的 块2 = a1,2 + a0,2 + a2,2 ✅ 全和 S₂
- 卡1 发 块0(=a1,0+a0,0)给卡2 → 卡2 的 块0 = a2,0 + a1,0 + a0,0 ✅ 全和 S₀
- 卡2 发 块1(=a2,1+a1,1)给卡0 → 卡0 的 块1 = a0,1 + a2,1 + a1,1 ✅ 全和 S₁

```
        块0          块1          块2
卡0:   a0,0     [S₁ 全和]✅     a0,2+a2,2
卡1:  a1,0+a0,0     a1,1       [S₂ 全和]✅
卡2:  [S₀ 全和]✅  a2,1+a1,1      a2,2
```

**N-1=2 步后,reduce-scatter 完成:** 卡0 独占 S₁、卡1 独占 S₂、卡2 独占 S₀。每张卡手里都有且仅有一块的完整全局和。

**每步通信量:** 每张卡发一块(M/N 字节)、收一块(M/N 字节)。**N-1 步后每卡累计发送 = (N-1) · M/N。**

### 4.3 阶段二:All-Gather(N-1 步)

**目标:** 现在每张卡只有一块全和,要让所有卡都拿到全部 N 块全和。这是个纯搬运(不再累加),把每张卡手里那块沿环传一圈。

**机制:** 同样 N-1 步,每步每卡把"当前手里最新的那块全和"发给右邻居,右邻居**直接覆盖写入**(不是相加,因为已是最终值)对应块。错位滚动使得每块全和沿环传播,N-1 步后绕环一周,人人集齐。

接着上面 N=3:

**All-Gather 第 1 步:** 卡 k 把它已算好的全和块发给 k+1。
- 卡2 把 S₀ 发给卡0 → 卡0 现在有 S₁, S₀
- 卡0 把 S₁ 发给卡1 → 卡1 现在有 S₂, S₁
- 卡1 把 S₂ 发给卡2 → 卡2 现在有 S₀, S₂

**All-Gather 第 2 步:**
- 每卡把上一步刚收到的那块继续往右传:卡0 发 S₀ 给卡1、卡1 发 S₁ 给卡2、卡2 发 S₂ 给卡0。

N-1=2 步后,每张卡都集齐 {S₀, S₁, S₂}。**All-reduce 完成,人人持有完整的全局梯度和。**

**每步通信量:** 同样每卡发一块 M/N、收一块 M/N。**N-1 步后每卡累计发送 = (N-1) · M/N。**

### 4.4 通信量总账与"带宽最优"证明

把两阶段加起来。**每张卡在整个 ring all-reduce 中发送的总字节:**

```
总发送字节 = reduce-scatter 阶段 + all-gather 阶段
          = (N-1)·(M/N) + (N-1)·(M/N)
          = 2·(N-1)/N · M
```

接收字节同理 = `2·(N-1)/N · M`(每发一块必收一块)。

**这就是核心结论。当 N → ∞:**

```
lim_{N→∞} 2·(N-1)/N · M = 2M
```

**每张卡的总传输量被牢牢封顶在约 2M,与卡数 N 几乎无关!** N=2 时是 1·M,N=8 时是 1.75M,N=1024 时是 ≈1.998M,封顶 2M。**这就是 ring all-reduce 的"带宽最优性":加再多卡,每张卡的通信带宽负担都不会超过约 2 倍梯度大小。** 对比朴素 reduce-to-root 的 `O(N·M)`,这是质的差别——前者让"加卡"在带宽上几乎免费,后者让"加卡"越加越堵。

**完成时间估计(bandwidth-bound,忽略 α):** 每张卡有 `2(N-1)/N · M` 字节要传,链路带宽 β:

```
T_ring ≈ 2·(N-1)/N · M / β  ≈ 2M/β   (N 大时)
```

**直觉解读:这个时间约等于"把两倍梯度大小的数据从一张卡的网卡推出去"所需的时间,与集群规模无关。** 这正是 ring all-reduce 能支撑大规模训练的原因:同步梯度的墙上时间(wall-clock)不随你扩到多少卡而恶化。

### 4.5 为什么这是最优的(下界论证)

"带宽最优"不是营销词,有信息论意义下的下界。论证如下:

**下界:** 考虑任意一张卡 k。all-reduce 结束时它必须持有全局和,而全局和依赖于其它 N-1 张卡的数据——卡 k 自己无法凭空算出别人的贡献,所以**它至少必须接收"足以确定结果"的信息**。在最优分块下,可以证明每张卡至少要**发送约 M 字节、接收约 M 字节**:它持有的 M 字节本地数据必须"贡献出去"让别人能算进全局和(下界 ~M 发送),而它要得到的全局结果中有约 `(N-1)/N · M` 是别处算出来的,必须收进来(下界 ~M 接收)。

**严格一点的标准结论(出自集合通信复杂度分析,待核精确表述):** 在 reduce-scatter 阶段,每张卡最终只需保有 1/N 的结果,信息论下界是每卡发送 `(N-1)/N · M`、接收 `(N-1)/N · M`;all-gather 对称同理。两阶段相加,**每卡发送/接收的下界都是 `2(N-1)/N · M`。**

**ring all-reduce 恰好达到这个下界(取等),所以它在带宽(传输字节总量)意义上是最优的——没有任何算法能让每张卡传输更少的字节。** 它唯一不优的是延迟:`2(N-1)` 步意味着 α 项随 N 线性增长,小消息时被 recursive doubling 等对数算法吊打。**记住这句话:ring 带宽最优、延迟最差;这决定了它的适用边界。**

### 4.6 现实中的混合算法

NCCL(NVIDIA 的集合通信库,所有 PyTorch 多卡训练的底座)并不只用 ring。它会根据消息大小、卡数、拓扑**自动选择算法**:大消息用 ring 或更先进的 **tree/double-binary-tree**(树形 all-reduce 在带宽接近最优的同时把延迟降到 `O(log N)`),小消息可能走别的。**实践要点:你几乎从不手写 all-reduce,而是理解 NCCL 在替你选什么、为什么——大梯度同步基本踩在"带宽最优"这条线上。**(具体到某版本 NCCL 在什么阈值切换哪种算法,待核。)

## 五、工程:把通信藏进计算里

数学上 `2M/β` 是同步梯度的理论下界时间,但如果你天真地"反向算完 → 调一次 all-reduce → 等它返回 → 更新",这段通信时间是**纯等待**,GPU 在那干瞪眼。DDP 真正的工程精髓,是让这 `2M/β` 几乎**完全消失在反向计算的背后**。这就是**计算-通信重叠(overlap)**。

### 5.1 关键洞察:梯度是"从后往前"一层层产生的

反向传播按计算图逆序进行:**最后一层(靠近 loss)的梯度最先算出来,第一层(靠近输入)的梯度最后算出来。** 这意味着当反向还在算前面的层时,后面层的梯度**早就准备好了,完全可以立即开始同步**,不必等整个反向跑完。

**重叠的核心:一边反向计算前面的层,一边用网卡同步后面已经算好的层的梯度。** 通信发生在另一个 CUDA stream 上,与计算 stream 异步并行。理想情况下,只要反向计算的时间足够长,几乎所有通信都被藏住,梯度同步的"露出时间"趋近于零——这才是 DDP 接近线性扩展的真正原因。

### 5.2 Bucket 分桶:不要为每个参数发一条消息

如果对**每个参数张量**各发一次 all-reduce 会怎样?一个模型有几百上千个参数张量,每个都触发一次集合通信 = 几百上千次 kernel 启动 + 几百上千个 α 延迟项。**α 累加起来会把小张量的通信彻底拖垮**(回忆 α-β 模型:多条小消息 = latency-bound)。

**解法:bucket(分桶)。把多个参数的梯度攒进一个连续的大缓冲区(bucket,如 25 MB 一桶),桶满了就对整桶发一次 all-reduce。** 这样:
- 把几百次小通信合并成几十次大通信,**摊薄 α**(大消息 → bandwidth-bound,回到 ring 的最优区间);
- 大消息也让带宽利用率更高。

**分桶与重叠的配合:** PyTorch DDP 给每个参数注册**梯度 hook**,某个参数的梯度一算完就标记;当一个 bucket 里的所有梯度都就绪,**立即异步触发这个 bucket 的 all-reduce**(不等其它 bucket)。因为梯度从后往前产生,**靠后的层的 bucket 先就绪先开始通信,与前面层的反向计算重叠**。等整个反向结束,通常只剩最后一个(最靠前的)bucket 的通信还没回来——露出的尾巴很短。

**调优坑:** bucket 太大 → 第一个 bucket 要等很久才满,重叠启动晚;bucket 太小 → α 又上来了,且重叠粒度过细。PyTorch 默认 `bucket_cap_mb` 约 25 MB(待核默认值),通常够用,极端拓扑下需调。

### 5.3 DDP 的反向-通信重叠时间线

```
反向计算(单 stream,从后往前):  [层L][层L-1][层L-2] ... [层2][层1]
                                    ↓就绪    ↓就绪          ↓就绪
通信 stream(异步 all-reduce):       [桶3 AR..][桶2 AR..]  ...  [桶1 AR..]
                                                                      └─ 这段尾巴露出 ≈ 通信暴露时间
更新(等所有 AR 完成后):                                                  → optimizer.step()
```

**关键结论:理想重叠下,梯度同步的暴露时间 ≈ max(0, 总通信时间 − 总反向计算时间) + 最后一个 bucket 的通信时间。** 模型越大(反向越久)、bucket 切得越好,暴露时间越接近零,扩展效率越接近 1.0。**这解释了一个常见现象:小模型多卡扩展效率差(反向太短藏不住通信),大模型反而容易接近线性扩展。**

### 5.4 梯度累积:用时间换 batch size

有时你想要一个**很大的全局 batch**(比如训练大模型需要 B=4M token 的 batch 才稳),但显存放不下这么大的子 batch。**梯度累积(gradient accumulation)** 的思路:**做 K 次"前向-反向"但不更新,把 K 次的梯度累加起来,第 K 次后才做一次 all-reduce + 更新。** 这在数学上等价于把子 batch 放大 K 倍(同样靠梯度的线性可加性)。

```
有效全局 batch = 子batch大小 × 数据并行度 N × 累积步数 K
```

**通信优化副作用:梯度累积天然降低通信频率。** 因为只在第 K 步才同步,前 K-1 步的反向完全不通信——**通信量 / 计算量 的比值降到 1/K**,对通信受限的场景(比如跨机带宽差)是免费的优化。但注意:**累积期间不能 all-reduce,所以前 K-1 步失去了"重叠"的机会**——PyTorch 提供 `model.no_sync()` 上下文在累积步里关闭 DDP 的梯度同步,只在最后一步同步。

**坑:累积步内是否对 loss 除以 K?** 要保证累积出的梯度等价于大 batch 的**平均**梯度,通常要把每步 loss 除以 K(或在最后统一缩放),否则梯度被放大 K 倍、等效学习率被放大 K 倍,容易训崩。这是新手最常踩的雷。

## 六、为什么"大 batch"会影响收敛

数据并行(尤其叠加梯度累积)很自然地把全局 batch 撑得很大。但**更大的 batch 不是免费午餐——它会改变优化的动力学,处理不当会损害最终精度或收敛速度。** 这一节讲清机制,因为它是数据并行的核心权衡之一。

### 6.1 大 batch 让梯度"更准"但"信息利用率更低"

mini-batch 梯度是真实梯度的**无偏估计**,其方差随 batch size b 反比下降:

```
Var(梯度估计) ∝ σ² / b      (σ² 为单样本梯度方差)
```

batch 翻倍,梯度估计的标准差降到 1/√2。**直觉上"更大 batch → 梯度更准 → 应该能用更大学习率走更快"。** 这催生了 **linear scaling rule(线性缩放规则,出自 Goyal et al. 2017《Accurate, Large Minibatch SGD》)**:

```
batch 放大 k 倍 → 学习率也放大 k 倍
```

直觉推导:k 个小 batch 各走一步 SGD,累计位移 ≈ `k · η · g`;若合成一个 k 倍大 batch 走一步,要走出同样位移,学习率得放大到 `k·η`(在梯度变化不大的局部近似下成立)。

但这个规则有上限。**问题在于:梯度方差只降到一定程度就饱和了,而学习率不能无限放大。** 当 batch 很大时,梯度已经很准,再大对"准度"帮助甚微,但你为了维持总更新步数,要么把学习率推到不稳定区间(发散),要么接受**总参数更新次数变少**(同样的数据量,batch 大 k 倍 → 更新步数少 k 倍)。

### 6.2 大 batch 的两个真实代价

**代价一:泛化间隙(generalization gap)。** 实证发现(Keskar et al. 2017,待核具体结论表述)**大 batch 训练倾向于收敛到"尖锐(sharp)"的极小值,泛化更差;小 batch 的梯度噪声反而像正则化,帮助逃离尖锐极小值、找到"平坦(flat)"极小值。** 大 batch 把这种有益噪声抹掉了。这是为什么单纯堆 batch 常常掉点。

**代价二:更新步数减少 → 早期收敛慢。** 固定 token 预算下,batch 大 k 倍意味着 optimizer 只走了 1/k 步。优化是迭代过程,步数本身有价值。这就是为什么大 batch 训练几乎都要配 **warmup(学习率预热)**:训练初期参数远离最优,大学习率 + 大 batch 容易在头几步就崩,先用小学习率慢慢热身,等进入较稳区域再拉满。

### 6.3 缓解手段(为什么这些技巧存在)

- **Linear scaling + warmup:** 上面说的标配组合。
- **LARS / LAMB 优化器:** 为超大 batch 设计,**按每层参数的范数自适应缩放学习率(layer-wise adaptive rate scaling)**,让大 batch 下不同层都能用合适步长,把可用 batch 推到几万甚至更大(LAMB 用于 BERT 大 batch 训练,待核)。
- **承认存在临界 batch size:** 实践与理论(OpenAI《An Empirical Model of Large-Batch Training》提出 **gradient noise scale / critical batch size** 概念,待核)都指向**存在一个"临界 batch size":小于它,加 batch 几乎线性减少所需步数(免费加速);大于它,收益急剧递减(再加 batch 步数几乎不再减少,纯浪费算力)。** 这个临界值与任务、模型、训练阶段有关,后期通常更大。

**给数据并行的实践结论:** 你的数据并行度 N 和梯度累积 K 共同决定全局 batch,而**全局 batch 不能无脑往大推**——超过临界 batch size 后,扩展效率在"系统吞吐"上还是线性的(每步更快),但在"达到目标精度所需的总时间"上会变差(步数不再减少甚至需要更多步)。**真正的扩展效率必须算到"收敛"而非"每步吞吐"。这是新手最容易自欺的地方:看着 GPU 利用率满了、每秒 token 数线性涨,以为扩展完美,其实模型收敛变慢了。**

## 七、代码:DDP 使用骨架与 Ring All-Reduce 伪代码

### 7.1 PyTorch DDP 最小骨架

```python
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP
from torch.utils.data import DataLoader, DistributedSampler

def main():
    # torchrun 会自动设好 RANK / WORLD_SIZE / LOCAL_RANK 等环境变量
    dist.init_process_group(backend="nccl")           # 多卡 GPU 用 nccl 后端
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)                  # 每个进程绑一张卡

    model = build_model().cuda(local_rank)
    # DDP 包装:在反向 hook 里挂上 bucket + 异步 all-reduce + 重叠
    model = DDP(model, device_ids=[local_rank],
                bucket_cap_mb=25,                      # 分桶大小,见 5.2(默认值待核)
                gradient_as_bucket_view=True)          # 让 .grad 直接是 bucket 的视图,省一次拷贝

    # 关键:DistributedSampler 保证每张卡拿到 batch 的不重叠子集(2.1 的 Sₖ)
    sampler = DistributedSampler(dataset)              # 自动按 rank 切分
    loader = DataLoader(dataset, batch_size=local_bs, sampler=sampler)

    opt = torch.optim.AdamW(model.parameters(), lr=base_lr)

    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)                       # 每轮重新 shuffle,否则各卡每轮看同样切分
        for step, (x, y) in enumerate(loader):
            x, y = x.cuda(local_rank), y.cuda(local_rank)

            # ---- 梯度累积 K 步:前 K-1 步关同步,省通信(见 5.4)----
            is_sync_step = (step % K == K - 1)
            ctx = model.no_sync() if not is_sync_step else nullcontext()
            with ctx:
                loss = model(x, y) / K                 # 除以 K:累积出"平均"梯度,别忘(5.4 的坑)
                loss.backward()                        # 反向;同步步里 hook 自动触发 all-reduce 并与反向重叠

            if is_sync_step:
                opt.step()                             # 此时 model.parameters().grad 已是全局平均梯度
                opt.zero_grad(set_to_none=True)

    dist.destroy_process_group()
```

**几个机制要点,对应前面的推导:**
- `DistributedSampler` 实现 2.1 的"把 batch 切成不重叠子集 Sₖ",这是数据并行正确性的前提——**如果各卡拿到相同数据,梯度全一样,等于白白浪费 N-1 张卡。**
- `DDP(...)` 包装时就在每个参数上注册了反向 hook,反向时**梯度一就绪就异步 all-reduce 并与后续反向计算重叠**(5.1–5.3),你看不到显式的 all-reduce 调用——它被藏在 `backward()` 里。
- `opt.step()` 时 `.grad` 已经是**全局平均梯度**(NCCL 求和 all-reduce + 框架除以 world_size),所以优化器更新天然满足"所有副本参数恒等"的不变量(2.1)。

### 7.2 Ring All-Reduce 两阶段伪代码

下面用注释化伪代码体现 ring 的两阶段循环机制(假设全双工链路,send 与 recv 可并行)。`chunk[j]` 是本卡梯度的第 j 块,每块 M/N 字节。

```python
def ring_all_reduce(chunk, rank, world_size):
    # chunk: list of N 块,本卡梯度切成 N 份;就地 in-place 更新
    N = world_size
    send_to   = (rank + 1) % N         # 右邻居(后继)
    recv_from = (rank - 1) % N         # 左邻居(前驱)

    # ============ 阶段一:Reduce-Scatter(N-1 步)============
    # 做完后:本卡持有第 (rank+1)%N 块的全局和,其余块是部分和
    for step in range(N - 1):
        send_idx = (rank - step) % N           # 本步要发出去的块索引(错位滚动)
        recv_idx = (rank - step - 1) % N        # 本步要收进来累加的块索引
        # 全双工:同时发 chunk[send_idx] 给右邻、从左邻收一块
        recv_buf = isend_irecv(send=chunk[send_idx], to=send_to,
                               recv_from=recv_from)         # 异步收发,各 M/N 字节
        wait()
        chunk[recv_idx] += recv_buf            # 累加(reduce 的核心:把部分和加上)
    # 此刻:卡 k 的第 (k+1)%N 块 = 该块的全局和 S_{(k+1)%N}

    # ============ 阶段二:All-Gather(N-1 步)============
    # 把每张卡手里那块全局和,沿环传一圈,人人集齐;只覆盖不累加
    for step in range(N - 1):
        send_idx = (rank - step + 1) % N        # 本步要发的(已是最终值的)块
        recv_idx = (rank - step) % N            # 本步要收的块(直接覆盖写入)
        recv_buf = isend_irecv(send=chunk[send_idx], to=send_to,
                               recv_from=recv_from)         # 各 M/N 字节
        wait()
        chunk[recv_idx] = recv_buf             # 覆盖(已是最终全局和,不再相加)
    # 此刻:每张卡的所有 N 块都是全局和 → all-reduce 完成
    return chunk

# 通信量核算(每卡):
#   reduce-scatter:发 (N-1)×(M/N)、收 (N-1)×(M/N)
#   all-gather:   发 (N-1)×(M/N)、收 (N-1)×(M/N)
#   合计每卡发送 = 2(N-1)/N × M ≈ 2M  (与 N 几乎无关 → 带宽最优)
```

**注意伪代码里两阶段唯一的实质差别:reduce-scatter 是 `+=`(累加部分和),all-gather 是 `=`(覆盖最终值)。** 这正是 4.2 与 4.3 的机制落到代码上的体现。索引的"错位滚动"(`-step`)保证了某一块的部分和沿环被逐步加全。

## 八、设计权衡与常见坑

**1. ring 带宽最优,但延迟随 N 线性变差。** `2(N-1)` 步意味着每步都有 α 开销;卡极多(几百上千)且**梯度张量很小**时,α 累加会让 ring 变慢,这时 tree all-reduce(`O(log N)` 步)或 recursive doubling 更优。**这就是为什么 bucket 要把小梯度攒成大消息**(5.2)——把场景从 latency-bound 拉回 bandwidth-bound,让 ring 的带宽最优性真正生效。NCCL 会自动按消息大小选算法,但你得理解它在权衡什么。

**2. straggler(掉队者)问题。** 同步数据并行每步都要等最慢的卡。某张卡因为数据加载慢、被其它进程抢占、或硬件降频,会拖垮整步。**症状:扩展效率忽高忽低、GPU 利用率有规律性塌陷。** 缓解:均衡数据分片、固定 CPU 亲和性、检查是否有卡过热降频。

**3. all-reduce 的浮点求和不保证按位可复现。** ring 的求和顺序与卡数、拓扑有关,浮点加法**不满足结合律**,所以 N=8 和 N=16 跑出的"平均梯度"可能在末位有微小差异 → **多卡训练结果难以逐位复现。** 这不是 bug,是浮点本质。要严格复现需固定卡数、固定 NCCL 算法、必要时用确定性 reduce(代价是慢)。

**4. 梯度累积忘记除以 K。** 5.4 已强调:累积 K 步若不把 loss(或最终梯度)除以 K,等效梯度被放大 K 倍 = 等效学习率被放大 K 倍,**轻则训练抖动重则直接 NaN。** 新手高频雷区。

**5. 忘记 `sampler.set_epoch(epoch)`。** 不调它,DistributedSampler 每个 epoch 给各卡的切分**完全一样**,等于每轮看同样顺序的数据,破坏 shuffle、损害收敛。

**6. 把"每步吞吐线性"当成"扩展成功"。** 第六节的核心警告:大 batch 超过临界点后,每步更快但收敛更慢。**衡量扩展效率必须算到"达到目标 loss 的总墙上时间",不能只看 tokens/sec。** 这是最隐蔽、最容易自我欺骗的坑。

**7. 通信暴露时间被小模型放大。** 5.3 的结论反过来:模型太小、反向太快,通信藏不住,暴露时间占比高,多卡扩展效率差。小模型多卡训练经常事倍功半,先确认瓶颈再加卡。

**8. 单机内 vs 跨机带宽断层。** 机内 NVLink 带宽远高于机间网络(InfiniBand/以太网,带宽量级差几倍到一个数量级,待核具体数字)。纯数据并行跨多机时,机间 ring 的那几跳成为瓶颈;这也是为什么后面要做**分层 all-reduce**(机内先 reduce-scatter、机间再 all-reduce、机内再 all-gather)和 3D 并行(把通信重的张量并行关在机内)。**记住:数据并行是通信最轻的并行,但跨机时它的梯度同步仍可能撞上机间带宽墙。**

## 十一、小结与承上启下

这一章我们把数据并行从动机推到了算法最优性:

- **机制:** 数据并行靠"全局梯度 = 各卡局部梯度的算术平均"这条线性可加性成立,每卡持完整模型、各算一份数据子集,**每步只需同步梯度,不传参数、不传激活**——是通信最轻的并行。
- **算法:** all-reduce = reduce-scatter + all-gather;**ring all-reduce 两阶段各 N-1 步,每卡总传输 `2(N-1)/N · M ≈ 2M`,与卡数近似无关,达到带宽下界——这就是"带宽最优"**;朴素 reduce-to-root 是 `O(N·M)` 且瓶颈集中,差一个数量级。代价是 ring 的延迟项 `2(N-1)` 步随 N 线性变差。
- **工程:** bucket 分桶摊薄 α、把场景拉回带宽最优区间;反向"从后往前出梯度"使通信能与计算重叠,把 `2M/β` 藏进反向背后;梯度累积用时间换大 batch 并顺带降通信频率。
- **权衡:** 大 batch 不是免费的,超过临界 batch size 后收敛收益急剧递减——**扩展效率必须算到收敛,不能只看每步吞吐。**

**承上:** 这章用的 `6ND`、每参数 16 字节都来自第 01 章那本账;ring 的带宽分析建立在第 00 章的带宽墙直觉上。

**启下:** 数据并行有一个它**解决不了**的硬约束——**它要求每张卡放得下一份完整模型**。当单层权重矩阵就超过单卡显存时,数据并行无能为力。下一章(03 张量并行 / Megatron)就来拆这堵墙:**把一层切到多张卡上**,核心是"怎么切才能让卡间通信最少"——你会再次见到 all-reduce,但这次它出现在一个 Transformer block 的前向反向里,频率高得多、也敏感得多。再往后,05 章的 ZeRO/FSDP 会回到数据并行的框架,但用 reduce-scatter + all-gather(正是本章的两个原语!)把"每卡存完整模型"这个浪费也干掉——届时你会发现,本章拆出的 reduce-scatter 与 all-gather,正是整个分片训练体系的两块基石。

### 练习 1:重推 ring all-reduce 通信量并算延迟项

已知:N=8、M=14 GB、单向带宽 β=100 GB/s、单条消息延迟 α=5 μs。

**(a) 每卡总传输字节**

直接代入 ring 的每卡传输公式:

```
每卡总传输 = 2(N-1)/N · M
          = 2 × (8-1)/8 × 14 GB
          = 2 × 0.875 × 14 GB
          = 1.75 × 14 GB
          = 24.5 GB
```

注意 1.75 这个系数:它已经很接近封顶值 2(N→∞ 才取到 2M=28 GB),说明 8 卡时每卡传输量距离理论上限只差 12.5%。

**每卡总发送(=总接收)≈ 24.5 GB。**

**(b) bandwidth-bound 完成时间(忽略 α)**

每卡有 24.5 GB 要从网卡推出,链路带宽 100 GB/s:

```
T_bw ≈ 24.5 GB / 100 GB/s = 0.245 s = 245 ms
```

(这是单卡视角的传输时间;因为 ring 全双工、各卡同时收发且每步只传 M/N,这个值就是整个 all-reduce 的墙上时间量级,与集群规模无关。)

**T_bw ≈ 245 ms。**

**(c) 总步数、α 项总延迟、判定瓶颈类型**

```
总步数 = 2(N-1) = 2 × 7 = 14 步       (reduce-scatter 7 步 + all-gather 7 步)
α 项总延迟 = 步数 × α = 14 × 5 μs = 70 μs = 0.07 ms
```

两项对比:

```
T_bw / T_α = 245 ms / 0.07 ms ≈ 3500 倍
```

传输时间比延迟时间大约 3500 倍,α 项完全可忽略。

**结论:这是典型的 bandwidth-bound(带宽受限)场景,α 几乎不起作用。** 这正是大梯度同步(14 GB 级)使用 ring all-reduce 的原因——ring 的"延迟最差"短板在大消息下被彻底掩盖,而它"带宽最优(每卡封顶约 2M)"的优势充分发挥。反过来,只有当梯度张量极小(M/N 小到接近 α·β 量级)且卡数极多时,那 14 步的 α 累加才会反扑,变成 latency-bound——届时 tree all-reduce(O(log N) 步)才更划算。

### 练习 2:证明 reduce-scatter + all-gather = all-reduce,并解释为何能达到带宽下界

**记号约定。** N 个进程,进程 k 初始持有长度 M 的数组 aₖ。把每个数组等分成 N 块,aₖ 的第 j 块记为 aₖ,ⱼ(各 M/N 字节)。第 j 块的全局和定义为:

```
Sⱼ = Σ_{k=0..N-1} aₖ,ⱼ
```

All-reduce 的目标(逐元素求和语义)是:**做完后每个进程都持有完整的 [S₀, S₁, …, S_{N-1}]**,即每个进程都拿到所有数组逐元素求和的结果。

**第一步:写出 reduce-scatter 后每个进程持有的内容。**

Reduce-scatter 的语义:把全局归约结果切成 N 块,第 j 块只交给进程 j。因此做完后:

```
进程 k 持有且仅持有第 k 块的全局和 Sₖ = Σ_i a_{i,k}
```

(每个进程负责"加全"某一块,最终每进程手里有 1/N 的最终结果。第四节里 ring 的约定是进程 k 持有第 (k+1)%N 块,只是块的编号偏移,不影响下面论证——只需"每进程恰好独占某一块的全局和、N 块被 N 个进程瓜分干净"。)

**第二步:写出 all-gather 后每个进程持有的内容。**

All-gather 的语义:每个进程各持有一块,做完后每个进程都拿到所有块拼起来的完整数据。把第一步的输出(进程 k 持有 Sₖ)喂给 all-gather:

```
all-gather 后,每个进程都持有 {S₀, S₁, …, S_{N-1}}
```

**第三步:验证这正是 all-reduce 的定义。**

每个进程现在持有的完整向量 [S₀, …, S_{N-1}],其中

```
Sⱼ = Σ_{k} aₖ,ⱼ      (第 j 块是所有进程第 j 块之和)
```

把所有块拼起来,就是"所有进程的完整数组逐元素求和"的结果,且每个进程都拥有它。这与 all-reduce 的定义逐字一致。

**∴ Reduce-Scatter ∘ All-Gather = All-Reduce。** 证毕。

**为什么这样拆能达到带宽下界(取等论证)。**

带宽下界来自一个无法回避的事实:all-reduce 结束时,每个进程持有的结果依赖于其它 N-1 个进程的数据,信息无法凭空产生,所以每个进程必须"贡献出自己的数据"且"收进别处算好的结果"。

逐阶段套下界:

- **Reduce-scatter 阶段。** 进程 k 最终只需保有 1/N 的结果(第 k 块),其余 (N-1)/N 的本地数据必须"送出去"参与别的块的求和。信息论下界:每进程至少**发送 (N-1)/N · M、接收 (N-1)/N · M**。

  ring 的 reduce-scatter 用 N-1 步、每步收发 M/N,实际每进程发送/接收 = (N-1)·(M/N) = **(N-1)/N · M —— 恰好取等**。

- **All-gather 阶段。** 进程 k 已有自己那 1/N 块,还缺其余 (N-1)/N 的最终结果,必须收进来;对称地它手里那块也要送给其它进程。下界:每进程至少**发送 (N-1)/N · M、接收 (N-1)/N · M**。

  ring 的 all-gather 同样 N-1 步、每步收发 M/N,实际 = **(N-1)/N · M —— 又取等**。

两阶段相加:

```
ring 每卡总发送 = (N-1)/N·M + (N-1)/N·M = 2(N-1)/N · M
信息论下界    = (N-1)/N·M + (N-1)/N·M = 2(N-1)/N · M
```

**两者完全相等,ring all-reduce 在每阶段都精确踩在下界上,没有任何冗余传输。** 这就是它"带宽最优"的严格含义:不存在任何算法能让每张卡传输更少的字节。

关键洞察:把"求和"这件重活通过 reduce-scatter **分摊**给 N 张卡(每卡只加全 1/N),避免了 reduce-to-root 那种"全部数据涌向一个 root → O(N·M) 瓶颈"的浪费;再用 all-gather **对称地**把结果摊回去。正是这种"分摊 + 对称"结构使每卡负载均衡且恰好等于下界。(代价是 2(N-1) 步的延迟,这是取等付出的对偶代价——下界论证只约束传输字节,不约束步数。)

### 练习 3:解释大 batch 为何影响收敛,并设计验证实验

**一、机制:从三个角度解释**

**角度 1 — 梯度方差视角(为什么"想"用大 batch)。**
mini-batch 梯度是真实梯度的无偏估计,方差随 batch size b 反比下降:

```
Var(梯度估计) ∝ σ² / b      (σ² 为单样本梯度方差)
```

batch 翻倍 → 估计标准差降到 1/√2,梯度"更准"。由此产生 linear scaling rule(Goyal et al. 2017):batch 放大 k 倍 → 学习率放大 k 倍(直觉:k 个小 batch 各走一步累计位移 ≈ k·η·g,合成一个大 batch 一步要走出同样位移就得把 η 放大到 k·η)。这看起来像免费加速。

**角度 2 — 临界 batch size 视角(收益为何饱和)。**
方差只降到一定程度就饱和:batch 已经够大时梯度已经很准,再加 batch 对"准度"几乎无帮助,但学习率不能无限放大(推过头会进入不稳定区直接发散)。OpenAI《An Empirical Model of Large-Batch Training》用 gradient noise scale 刻画出存在一个**临界 batch size**(待核精确表述):
- 小于临界值:加 batch 几乎线性减少达标所需步数(免费加速区);
- 大于临界值:收益急剧递减,再加 batch 步数几乎不再减少 → 纯浪费算力。

且固定 token 预算下,batch 大 k 倍意味着 optimizer 只走了 1/k 步——优化是迭代过程,**步数本身有价值**,步数变少直接拖慢早期收敛。这就是大 batch 几乎都要配 warmup(初期参数远离最优,大 lr+大 batch 头几步易崩)的原因。

**角度 3 — 泛化间隙视角(为何最终精度也可能掉点)。**
实证(Keskar et al. 2017,待核具体结论)指出:大 batch 倾向收敛到**尖锐(sharp)极小值**,泛化更差;小 batch 的梯度噪声反而像隐式正则化,帮助逃离尖锐极小值、找到**平坦(flat)极小值**。大 batch 把这种有益噪声抹掉,于是出现"训练 loss 一样低但验证精度更差"的泛化间隙。

**小结(明确结论):** 大 batch 不是免费午餐——它在"梯度更准"和"信息利用率更低 + 噪声正则消失 + 步数变少"之间权衡。超过临界 batch size 后,系统吞吐(tokens/sec)还在线性涨,但**达到目标精度的总时间反而变差**。衡量扩展效率必须算到"收敛",不能只看每步吞吐。

**二、验证实验设计**

目标:验证"超过临界点后,大 batch 早期收敛更慢、且可能有泛化间隙"。

**控制变量(关键,否则结论无效):**
- 固定**总 token 预算**(不是固定步数,也不是固定 epoch)——保证两组"看过的数据量"一致,差异只来自 batch 大小。
- 固定模型、数据集、优化器、随机种子(种子可多跑几个取平均,见下)。
- 大 batch 组配齐 linear scaling rule(lr 同比放大)+ warmup,否则是在测"忘记调 lr",不是在测 batch 本身。

**实验组:**
| 组 | 全局 batch | 学习率 | warmup |
|---|---|---|---|
| A(基线) | B | η | 标准 |
| B(大 batch) | 8B | 8η(linear scaling) | 加长 warmup |

(若条件允许,再加 batch=2B、4B、16B、32B 几档,用来定位临界 batch size 的拐点。)

**测量指标:**
- (i) **达到相同验证 loss 所需的更新步数 / 墙上时间** —— 检验"步数效率"。在固定 token 预算下,大 batch 步数本就少 8 倍,看它是否仍能在更少步数内达标(临界点内)还是反而需要更多 token(超临界)。
- (ii) **最终验证精度 / 验证 loss** —— 检验泛化间隙。
- (iii)(可选)在最优点附近估 loss 曲面的 sharpness(如 Hessian 最大特征值 / 对参数加小扰动后的 loss 上升幅度),佐证"sharp vs flat"。

**统计严谨性:** 每组用 ≥3 个不同随机种子重复,报告均值±方差——区分"真实差异"与"训练随机噪声",避免被单次 lucky/unlucky run 误导。

**预期结果(可证伪的预测):**
- 大 batch 组**早期收敛更慢**(同样 token 下验证 loss 下降更迟),warmup 后逐渐追上;
- 若 8B 已超过临界 batch size,大 batch 组**达标所需 token 更多**,且**最终验证精度略低**(出现泛化间隙);
- 若 8B 仍在临界点内,两组最终精度相近、步数效率大 batch 更优——这本身也定位了临界值在 8B 以上。

**结论:** 该实验能直接区分"大 batch 在临界点内(免费加速)"还是"已超临界(收敛变慢/掉点)",从而把抽象的"扩展效率必须算到收敛"落成可测量的判据。

### 练习 4:手写 N=4 的 ring all-reduce 并验证正确性

下面是纯 numpy(CPU 可跑,无需 GPU)的实现,严格按第七节 7.2 伪代码的两阶段循环与索引错位滚动机制。用 4 个数组模拟 4 张卡的梯度,各切成 4 块;断言最终每卡结果等于 4 个初始数组逐元素之和,并核对每卡发送字节 = 2(N-1)/N·M。

```python
import numpy as np

def ring_all_reduce(grads):
    """
    grads: list[np.ndarray]，长度 N，第 k 个是卡 k 的完整梯度向量(长度须能被 N 整除)。
    模拟 ring all-reduce，返回 (每张卡 all-reduce 后的结果列表, 每卡发送字节列表)。
    每张卡结果都应等于 sum(grads)。
    """
    N = len(grads)
    L = grads[0].size
    assert L % N == 0, "梯度长度需能被卡数整除以等分成 N 块"
    bs = L // N                      # 每块元素数 = M/N

    # 把每张卡梯度切成 N 块: chunks[k][j] 是卡 k 的第 j 块(独立拷贝，模拟各卡本地存储)
    chunks = [[g[j*bs:(j+1)*bs].copy() for j in range(N)] for g in grads]

    elem_bytes = bs * grads[0].itemsize   # 一块的字节数(M/N 字节)
    bytes_sent = [0] * N                  # 统计每卡累计发送字节

    # ===== 阶段一:Reduce-Scatter(N-1 步)=====
    # 每步：卡 k 发第 (k-step)%N 块给右邻 (k+1)%N，从左邻收一块累加到 (k-step-1)%N 位置
    for step in range(N - 1):
        # 先快照本步所有"要发出去的块"，避免就地 += 污染同一步内别人要收的源数据
        bufs = {}
        for k in range(N):
            send_idx = (k - step) % N
            bufs[k] = chunks[k][send_idx].copy()      # 模拟"发出去的数据"
            bytes_sent[k] += elem_bytes               # 每步每卡发一块
        # 每卡从左邻 recv_from 收到一块，累加(reduce 核心:加上部分和)
        for k in range(N):
            recv_from = (k - 1) % N
            recv_idx  = (k - step - 1) % N
            chunks[k][recv_idx] += bufs[recv_from]    # += 累加部分和

    # 中途验证:reduce-scatter 后卡 k 的第 (k+1)%N 块应已是该块全局和 S_{(k+1)%N}
    full_blocks = [sum(grads)[j*bs:(j+1)*bs] for j in range(N)]
    for k in range(N):
        owned = (k + 1) % N
        assert np.allclose(chunks[k][owned], full_blocks[owned]), \
            f"reduce-scatter 错误:卡{k}的块{owned}不是全局和"

    # ===== 阶段二:All-Gather(N-1 步)=====
    # 每步：卡 k 发第 (k-step+1)%N 块给右邻，从左邻收到的块直接覆盖写入 (k-step)%N
    for step in range(N - 1):
        bufs = {}
        for k in range(N):
            send_idx = (k - step + 1) % N
            bufs[k] = chunks[k][send_idx].copy()
            bytes_sent[k] += elem_bytes
        for k in range(N):
            recv_from = (k - 1) % N
            recv_idx  = (k - step) % N
            chunks[k][recv_idx] = bufs[recv_from]     # = 覆盖(已是最终全局和，不再相加)

    results = [np.concatenate(chunks[k]) for k in range(N)]
    return results, bytes_sent


if __name__ == "__main__":
    np.random.seed(0)
    N = 4
    L = 12                                   # 须能被 4 整除
    grads = [np.random.randn(L) for _ in range(N)]
    expected = sum(grads)                    # 直接逐元素求和作为 ground truth

    results, bytes_sent = ring_all_reduce(grads)

    # 验证 1:人人集齐且等于全局和
    for k in range(N):
        assert np.allclose(results[k], expected), f"卡{k}结果与直接求和不一致"
    print("OK: 4 张卡都拿到逐元素全和，与 sum(grads) 一致")

    # 验证 2:每卡发送字节 == 2(N-1)/N · M
    M = L * grads[0].itemsize                # 每卡完整梯度字节(此处 12×8=96)
    theo = 2 * (N - 1) / N * M               # 2×3/4×96 = 144
    print(f"每卡梯度 M = {M} 字节，理论每卡发送 2(N-1)/N·M = {theo} 字节")
    print(f"实测每卡发送 = {bytes_sent}")
    assert all(b == theo for b in bytes_sent), "发送字节与理论不符"
    print("OK: 每卡发送字节 == 2(N-1)/N·M")
```

**两处验证对应章节机制:**

1. **reduce-scatter 后"每卡恰好一块是全和":** 代码中途断言验证卡 k 的第 (k+1)%N 块等于该块全局和(与 4.2 手算的"卡0独占 S₁、卡1独占 S₂、卡2独占 S₀、…"同构,块编号偏移 +1)。
2. **all-gather 后"人人集齐":** 最终 4 张卡的完整向量都等于 sum(grads)。

**两阶段唯一实质差别**(对应 7.2 末尾的强调):reduce-scatter 用 `+=`(累加部分和),all-gather 用 `=`(覆盖最终值);索引的 `-step` 错位滚动让某块的部分和沿环被逐步加全。

**带宽核对:** N=4 时 M=96 字节,2(N-1)/N·M = 2×3/4×96 = 144 字节,实测每卡发送恰为 144(reduce-scatter 3 步 + all-gather 3 步,每步发一块 M/N=24 字节,共 6×24=144),与公式精确吻合,印证 ring 每阶段都踩在带宽下界上。

**实现注意:** 关键是每步内必须**先快照所有要发送的块再统一收/累加**(代码里的 `bufs`),否则在单进程串行模拟里,先更新的卡会污染后更新的卡本步要读的源数据——真实分布式下各卡并行收发本就用独立缓冲区,不存在此问题,这里只是串行模拟的正确性细节。
