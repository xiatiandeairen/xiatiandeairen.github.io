## 一、方向全景:这是把算法变成现实的工程科学

MLSys(Machine Learning Systems,机器学习系统)的母题只有一句话:**在算力、内存带宽、显存这三堵墙的约束下,把超大规模模型训得动、推得快、跑得省。** 算法研究者画出一个 7000 亿参数的 Transformer,那只是一张图纸;能不能在 1 万张 GPU 上连续训三个月不崩、能不能在线上以每秒几千 token 的速度服务上亿用户、单位 token 成本能不能压到可盈利——这些全是系统问题。MLSys 就是回答这些问题的学科。它是离系统工程师最近的 AI 方向,因为它的底层语言不是损失函数和梯度,而是**带宽、延迟、吞吐、内存层级、并行度、调度**——这些词系统工程师天天打交道。

先把"三堵墙"讲透,因为后面所有技术都是在拆这三堵墙:

- **算力墙(compute wall):** 模型的训练计算量约等于 `6 × 参数量 × 训练 token 数`(前向 2 倍、反向 4 倍,这是业界通用的 FLOPs 估算)。一个百亿到万亿参数模型乘以万亿级 token,就是 10²³~10²⁵ 量级的浮点运算。单卡再快也扛不住,只能堆卡并行——于是有了各种并行策略。

- **显存墙(memory capacity wall):** 一张 A100 只有 40/80GB 显存。训练时显存被三块吃掉:**模型参数、梯度、优化器状态**。用 Adam 优化器、混合精度训练时,每个参数大约要占 **16 字节**(FP16 参数 2 + FP16 梯度 2 + FP32 参数副本 4 + 一阶动量 4 + 二阶动量 4,即著名的"参数×16"),再加上激活值。一个 100 亿参数模型,光这三块就要 160GB,单卡根本放不下。

- **内存带宽墙(memory bandwidth wall):** 这是最隐蔽也最致命的一堵。GPU 的算力(FLOPS)增长远快于显存带宽(HBM 带宽)增长。很多算子(尤其是注意力、逐元素操作)不是"算不过来",而是"数据从 HBM 搬到计算单元的速度跟不上"——这叫 **memory-bound(访存受限)**。这堵墙直接催生了 FlashAttention 这类 IO 感知算法。系统工程师对此应该秒懂:这就是经典的 **CPU 算力 vs 内存墙** 在 GPU 上的翻版,Roofline 模型(横轴算术强度、纵轴可达性能)在这里依然是分析利器。

**研究范式的演变,可以概括为三个阶段:**

第一阶段(~2012–2018)是**"把单机的事做好"**:重心在单卡/单机多卡的算子优化、自动微分框架(TensorFlow、PyTorch)、数据并行扩展。这个阶段模型还不算大(亿级参数),系统的核心矛盾是训练框架的易用性和吞吐。

第二阶段(~2019–2022)是**"破墙"**:GPT-2/GPT-3 把参数推到百亿、千亿,单卡放不下成了硬约束。这一阶段诞生了 MLSys 真正的标志性工作——张量并行(Megatron-LM)、显存分片(ZeRO)、流水线并行(GPipe/PipeDream)、混合精度、梯度重计算。研究焦点从"算得快"转向"放得下且扩得开"。

第三阶段(2022 至今)是**"训练 + 推理双线作战"**:ChatGPT 引爆推理需求,推理成本成了商业生死线。一边是训练侧的万卡集群稳定性与容错;另一边是推理侧的吞吐优化(PagedAttention、连续批处理)、延迟优化(投机解码)、成本优化(低比特量化)。MLSys 从"少数大厂训练几个大模型"的小众领域,变成支撑整个 AI 产业的基础设施学科。

一个关键认知:**MLSys 与传统 HPC/分布式系统/操作系统是同源的。** ZeRO 的分片本质是分布式存储的 sharding;PagedAttention 直接搬运了操作系统虚拟内存的分页思想;连续批处理是调度问题;通信-计算重叠是经典的流水线隐藏延迟。系统出身的人转 MLSys,迁移成本远低于从算法转过来——你缺的只是 Transformer 的结构常识和梯度的数据流,底层直觉是现成的。

## 二、技术脉络地图:为什么走到今天

把整条主线画出来,你会看到一条清晰的"拆墙史"。每一代技术都是为了拆掉前一代撞上的那堵墙:

**训练侧主线:**

```
数据并行(DP,2012~)
  痛点:单卡放得下模型,但要更快/更大 batch
        → 多卡各算一份数据,再 all-reduce 同步梯度
  撞墙:模型本身放不下单卡了

→ 模型/张量并行(Tensor Parallel,Megatron-LM 2019)
  痛点:单层权重矩阵就超过单卡显存
        → 把一个矩阵乘法横/纵切到多卡,卡间用 all-reduce 拼接
  撞墙:通信量巨大,只能在单机 NVLink 内做

→ 流水线并行(Pipeline Parallel,GPipe 2018 / PipeDream 2019)
  痛点:层数太多,按层切到不同卡组,但卡会互相等(气泡)
        → micro-batch 流水起来填满气泡

→ ZeRO 显存分片(ZeRO,2020)
  痛点:数据并行下每张卡都存了一份完整的优化器状态/梯度/参数,极度浪费
        → 把这三者切片分散到所有卡,需要时临时通信凑齐

→ 3D 并行(数据 × 张量 × 流水,Megatron-DeepSpeed 2021)
        → 三种并行正交组合,训千亿/万亿模型的标准配方

→ 序列并行 / ZeRO-Offload / Infinity(2021~)
        → 进一步把激活、优化器状态卸载到 CPU/NVMe,榨干每一字节
```

**贯穿训练全程的"省显存换算力"技术:**

```
混合精度训练(FP16,2017)→ BF16(2019~)→ FP8(2022~,H100 硬件支持)
梯度检查点 / 重计算(activation checkpointing,2016)
```

**推理侧主线(2022 后爆发):**

```
FlashAttention(2022)
  痛点:注意力要算 N×N 矩阵并落显存,长序列下显存 O(N²) 爆炸,且访存受限
        → IO 感知,分块计算,矩阵不落 HBM,显存降到 O(N)

→ 权重量化(GPTQ 2022 / AWQ 2023 / SmoothQuant 2022)
        → 把权重从 FP16 压到 INT4/INT8,显存和带宽双降

→ 连续批处理 + PagedAttention(Orca 2022 / vLLM 2023)
        → 借虚拟内存分页管理 KV cache,消除碎片,请求级动态调度,吞吐数倍

→ 投机解码(Speculative Decoding,2023)/ Medusa(2023)
        → 小模型起草、大模型并行验证,延迟降低,生成质量不变

→ FlashAttention-3(2024)/ MoE 推理系统 / prefill-decode 解耦(2024~)
        → 吃满 H100 新硬件特性;稀疏激活;把两个特性迥异的阶段拆开各自优化
```

读这张图的正确姿势:**每个箭头都是"上一个方案撞到的那堵墙"。** 数据并行撞显存墙催生张量并行和 ZeRO;张量并行撞通信墙被限制在单机内,于是要和流水线并行组合;FlashAttention 撞的是带宽墙;PagedAttention 撞的是显存碎片(本质也是显存墙)。把握住"拆哪堵墙",整个领域就串起来了。

## 三、里程碑成果精讲

这一节是全章的核心。每个成果按"痛点 → 做法 → 突破点 → 真实机制 → 影响"讲透。

**Megatron-LM 张量并行(2019,NVIDIA,Shoeybi 等)** — 痛点:GPT-2 这类模型,**单个 Transformer 层的权重矩阵就大到单卡放不下**,数据并行(每卡存完整模型)彻底失效。做法:把层内的矩阵乘法切到多张卡。对 MLP 的两个线性层,第一个权重矩阵**按列切**(每卡算一部分输出通道),第二个**按行切**,这样中间的非线性激活可以各算各的,只在最后做一次 all-reduce 拼回完整结果;注意力层则**按 head 切**,每张卡负责一部分注意力头。突破点:**这是"层内并行"(intra-layer),把单层的计算和参数都摊开到多卡**,与按层切的流水线并行(层间并行)正交。真实机制:矩阵乘法 `Y = XA` 天然可分块——`A` 按列切成 `[A₁, A₂]`,则 `Y = [XA₁, XA₂]`,两块独立算完拼接即可;关键是巧妙安排切法,使一个 Transformer block 内**只需要 2 次 all-reduce**(前向 1 次、反向 1 次),把通信压到最低。但 all-reduce 通信量大且频繁,**必须依赖单机内 NVLink 的超高带宽**,跨机做张量并行会被网络带宽拖死——这就解释了为什么张量并行通常只在单节点 8 卡内使用。影响:张量并行成为训练巨型模型的基石,"Megatron"几乎是张量并行的代名词,后来与 DeepSpeed 结合成 Megatron-DeepSpeed,是 3D 并行的工业标准实现。

**ZeRO 显存分片(2020,微软 DeepSpeed,Rajbhandari 等)** — 痛点:数据并行虽然能扩展,但**每张卡都冗余存了一份完整的参数、梯度、优化器状态**。用 Adam + 混合精度时优化器状态是大头(每参数 12 字节),N 张卡就浪费 N 份。做法:ZeRO(Zero Redundancy Optimizer)把这三块按卡数切片,每张卡只存 1/N。分三级:**ZeRO-1 切优化器状态、ZeRO-2 再切梯度、ZeRO-3 连参数也切**。突破点:**它保留了数据并行的编程简洁性(逻辑上每卡还是跑完整模型),却拿到了模型并行的显存效率**——这是关键的"鱼和熊掌"。真实机制:以 ZeRO-3 为例,前向计算到某一层时,该层参数当前只有某张卡持有完整副本,于是临时发起一次 **all-gather** 把这层参数广播到所有卡,算完立刻丢弃释放显存;反向同理,梯度算完用 **reduce-scatter** 把每块梯度归约回对应的"属主卡"。代价是增加了通信量,但通过预取(算第 L 层时提前 gather 第 L+1 层)做到通信-计算重叠,实测吞吐损失很小。系统工程师视角:**这就是分布式存储的 sharding + 按需拉取 + 计算后即时回收**,和分布式 KV 存储的思路一脉相承。影响:ZeRO 让"用纯数据并行的框架训千亿模型"成为可能,DeepSpeed 因此成为开源训练框架的事实标准之一;ZeRO-3 的思想后来被 PyTorch 原生吸收为 **FSDP(Fully Sharded Data Parallel)**。

**混合精度训练(2017,百度 + NVIDIA,Micikevicius 等)** — 痛点:FP32 训练占显存、算得慢,而 GPU 的半精度算力是单精度的数倍(Tensor Core 更是几个量级)。直接全用 FP16 又会出问题:梯度数值太小,在 FP16 下直接变成 0(下溢),训练发散。做法:**前向和反向用 FP16 算(快、省显存),但维护一份 FP32 的权重主副本用于参数更新**;同时引入 **loss scaling(损失缩放)**——反向前把 loss 乘一个大常数,把小梯度"放大"到 FP16 能表示的范围,更新前再除回去。突破点:**用混合精度同时拿到了速度、显存、数值稳定性三者**,而不是在精度和性能间二选一。真实机制:FP16 只有约 5 位指数,能表示的最小正规数约 6×10⁻⁵,很多梯度比这还小就归零了;loss scaling 利用反向传播的线性性,梯度会等比例放大,从而落进 FP16 的有效区间。FP32 主副本则保证累积更新时不丢失精度(小的更新量加到大的权重上,在 FP16 下会被吃掉)。后来 **BF16** 出现——它牺牲尾数精度换来和 FP32 一样宽的指数范围,**几乎不需要 loss scaling**,成为大模型训练首选;再后来 H100 支持 **FP8**,把这条路推到极致。影响:混合精度是今天所有大模型训练的默认开关,直接让有效算力翻倍、显存减半。

**梯度检查点 / 激活重计算(2016,Chen 等)** — 痛点:训练时为了反向传播,**前向产生的所有中间激活都得存着**,激活显存随网络深度和 batch/序列长度线性增长,长序列下激活甚至比参数还占显存。做法:前向时**只保存少数几个"检查点"层的激活,其余全部丢弃**;反向需要某层激活时,从最近的检查点**重新前向计算一遍**。突破点:**这是一次经典的"时间换空间"**——用约 1/3 的额外计算量,把激活显存从 O(L) 降到约 O(√L)。真实机制:正常反向需要所有层激活;有了检查点,把网络切成若干段,只在段边界存激活,段内激活反向时临时重算。系统工程师会立刻联想到**日志/快照重放**:不存全量状态,只存检查点 + 重放。影响:这是长序列、大模型训练的标配,和 ZeRO、混合精度叠加使用,共同把"放得下"这件事变成可能。

**FlashAttention(2022,斯坦福,Tri Dao 等)** — 痛点:标准注意力要算 `S = QKᵀ`(N×N 矩阵)、softmax、再乘 V,**中间那个 N×N 的注意力矩阵要写回 HBM 又读出来**,显存 O(N²),序列一长就爆;更关键的是,注意力是**访存受限**的——大量时间花在 HBM 读写而非计算上。做法:**IO 感知(IO-aware)地分块计算**,把 Q、K、V 切成小块,在 GPU 的片上 SRAM(比 HBM 快一两个量级但极小)里完成"乘-softmax-再乘"的全过程,**从不把完整的 N×N 矩阵写回 HBM**。突破点:**把算法瓶颈从"算力"重新定义为"访存",并据此重写算法**——这是 MLSys 思想的最佳范例:不改变数学结果,只改变数据在内存层级间的流动方式。真实机制:难点在 softmax 需要全局归一化(要看到整行),分块后看不到全行。FlashAttention 用 **online softmax(在线 softmax)**:边遍历 K/V 块边维护"当前最大值"和"当前累加和",每来一个新块就按数值稳定的方式重新缩放已有结果,数学上等价于一次性算完整 softmax。这样既避免落显存(显存降到 O(N)),又因为减少 HBM 读写而大幅加速。系统工程师视角:**这就是 cache-aware/blocking 优化**,和矩阵乘法的分块、数据库的 cache-conscious 算法同理——核心是让工作集装进更快的存储层级。影响:FlashAttention 成为长上下文时代的关键基础设施,几乎所有训练/推理框架默认集成;**FlashAttention-2(2023)** 优化并行划分和工作分配进一步提速,**FlashAttention-3(2024)** 针对 H100 的异步特性和 FP8 重写,继续逼近硬件上限。

**PagedAttention / vLLM(2023,UC Berkeley,Kwon 等)** — 痛点:推理时每个请求的 **KV cache(已生成 token 的键值缓存)** 会随生成长度增长,但你**事先不知道会生成多长**。传统做法是按最大长度预留一整块连续显存,导致两类浪费:**内部碎片**(预留多了用不完)和**外部碎片**(空闲块大小不一拼不起来),实测显存利用率常常只有 20%–40%。做法:**把 KV cache 切成固定大小的"块"(block),用一张"块表"(block table)把逻辑上连续的 token 序列映射到物理上分散的显存块**——完全照搬操作系统的虚拟内存分页。突破点:**用分页打破了"KV cache 必须物理连续"的假设**,显存利用率逼近 100%,省下的显存直接用来塞进更多并发请求 → 吞吐倍增。真实机制:每个 token 块按需分配,不连续也无妨,块表负责寻址;更妙的是支持**写时复制(copy-on-write)**——多个请求共享同一前缀(比如同一个 system prompt)时可以共享物理块,只有分叉处才复制,这与操作系统 fork 的 COW 完全一致。配合 **连续批处理(continuous batching)**:不再等整个 batch 都生成完才返回,而是**谁先生成完谁就退出、立刻把腾出的槽位填进新请求**,GPU 利用率拉满。系统工程师视角:这是**虚拟内存 + 进程调度**在 LLM 服务上的精准复刻,读 vLLM 论文你会有强烈的"老朋友"感。影响:vLLM 成为开源 LLM 推理服务的事实标准,PagedAttention 思想被几乎所有推理引擎(TensorRT-LLM、TGI 等)吸收,直接把推理成本打下来一个台阶。

**投机解码(Speculative Decoding,2023,Google / DeepMind,Leviathan 等 & Chen 等)** — 痛点:大模型自回归生成是**逐 token 串行**的,每生成一个 token 都要把整个模型跑一遍,而这一步是**访存受限**的(把几百 GB 权重从 HBM 读一遍只为算一个 token,算力大量闲置)。做法:**用一个小的"草稿模型"快速连续生成 K 个候选 token,再让大模型对这 K 个 token 做一次并行验证**(一次前向就能并行打分 K 个位置),验证通过的就采纳,第一个不通过的位置之后丢弃重来。突破点:**把"串行生成 K 步"变成"并行验证 K 步"**,在不改变输出分布的前提下降低延迟。真实机制:核心是一个巧妙的**拒绝采样**方案——通过特定的接受/拒绝概率设计,**保证最终采样分布严格等于大模型单独解码的分布**(不是近似,是数学上无损)。为什么省时间?因为大模型一次前向算 1 个 token 和算 K 个 token 的耗时几乎一样(访存受限,瓶颈在搬权重不在算),所以"并行验证 K 个"几乎免费,只要草稿命中率够高就净赚。**Medusa(2023)** 走了另一条路:不用单独的草稿模型,而是给大模型加几个"预测头"同时预测未来多个位置,省去维护两个模型的麻烦。影响:投机解码成为低延迟推理的关键技术,被广泛集成进生产推理栈,尤其适合对话这类对首 token 和生成速度敏感的场景。

**低比特权重量化:GPTQ / AWQ / SmoothQuant(2022–2023)** — 痛点:模型权重以 FP16 存储,推理时**既占显存又占带宽**,而前面说过 decode 阶段是访存受限的——把权重压小,等于直接加速。做法:**把权重从 16 位压到 4 位(INT4)或 8 位(INT8),推理时再反量化回来算**。突破点:**在几乎不掉精度的前提下做到 4 bit**,关键在于处理"异常值"。真实机制各有侧重:**GPTQ** 用近似二阶信息(基于 Hessian)逐列量化并补偿误差,把量化误差对输出的影响降到最低,是**训练后量化(PTQ)**的代表;**AWQ(Activation-aware Weight Quantization)** 发现并非所有权重同等重要——**和大激活值相乘的那些权重通道对精度影响最大**,于是按激活分布给重要通道更高精度(等价缩放),用很小的代价保住精度;**SmoothQuant** 解决的是激活里的异常值难量化问题,通过把量化难度从激活"迁移"一部分到权重(数学等价的缩放)来平滑分布,使**权重和激活都能 INT8**。影响:量化是端侧部署和降本的核心手段,让 70B 模型能塞进单张消费级显卡,让推理服务成本进一步下探;**待核**:各方法在不同模型上的精度损失(通常报告为困惑度上升零点几到 1 以内)需以原论文为准。

**3D 并行(数据 × 张量 × 流水,Megatron-DeepSpeed,2021)** — 痛点:单一并行策略都有天花板——张量并行受限于单机 NVLink、流水线并行有气泡、数据并行受显存约束。做法:**三种并行正交组合**:张量并行在节点内 8 卡间切层,流水线并行把层分段跨节点,数据并行复制多份流水线并行梯度。突破点:**把三个维度当成正交坐标轴自由配置**,针对集群拓扑(机内带宽高、机间带宽低)做最优映射——通信量大的张量并行放机内,通信量小的数据/流水放机间。真实机制:这本质是个**通信拓扑感知的并行度搜索问题**,要在气泡率、通信量、显存占用三者间权衡。影响:3D 并行是训练千亿到万亿模型的标准范式,Megatron-DeepSpeed 是其工业级实现,几乎所有公开的超大模型训练都基于它或其变体。

## 四、近十年关键优化技术

这一节把散落在各处的工程优化技术系统梳理一遍。每项按"解决的瓶颈 → 机制"讲。

**3D 并行(数据 + 张量 + 流水)** — 瓶颈:单一并行策略各有天花板。机制:三维正交组合,按集群拓扑分配——张量并行(高通信)塞进机内 NVLink,流水线/数据并行(低通信)走机间网络。配置这三个并行度本身就是一门调参艺术。

**序列并行(Sequence Parallelism)** — 瓶颈:即使用了张量并行,**LayerNorm、Dropout 这些非矩阵操作的激活仍在每卡完整保留**,长序列下激活显存吃不消。机制:沿序列维度把这些激活也切到多卡,与张量并行无缝衔接,进一步压低激活显存。长上下文训练(几十万 token)更进一步用 **Ring Attention / Context Parallel**,让注意力计算沿序列环状流转。

**ZeRO-Offload / ZeRO-Infinity** — 瓶颈:GPU 显存就是不够。机制:**把优化器状态、梯度甚至参数卸载到 CPU 内存(Offload)乃至 NVMe SSD(Infinity)**,用 PCIe/NVMe 带宽换显存容量,让单卡也能训百亿模型。系统视角:这就是**内存层级 + swap**,拿慢存储换容量,关键在预取和重叠把延迟藏住。

**梯度检查点(activation checkpointing)** — 瓶颈:激活显存随深度线性增长。机制:只存检查点层激活,反向时分段重算(见第三节),用约 1/3 计算换激活显存从 O(L) 到 O(√L)。

**混合精度(FP16 / BF16 / FP8)** — 瓶颈:FP32 慢且占显存,Tensor Core 半精度算力数倍于单精度。机制:低精度算、高精度更新;BF16 宽指数范围免去 loss scaling;FP8 在 H100 上把训练/推理算力再翻倍,但需要细粒度缩放(per-tensor/per-block scaling)控制数值。

**FlashAttention 系列** — 瓶颈:注意力访存受限 + N×N 矩阵落显存。机制:分块 + online softmax,在 SRAM 内完成,不落 HBM(见第三节);V2 优化并行划分,V3 吃 H100 异步与 FP8。

**连续批处理(continuous batching)** — 瓶颈:传统静态批处理要等整批生成完,长短不一的请求互相拖累,GPU 大量空闲。机制:**迭代级调度**,以 token 为粒度动态进出 batch,完成的请求立即退出、新请求立即填入,GPU 利用率拉满。源自 Orca(2022),vLLM 发扬光大。

**PagedAttention** — 瓶颈:KV cache 显存碎片化,利用率只有两三成。机制:分页 + 块表 + COW 前缀共享(见第三节),显存利用率逼近满载,腾出空间塞更多并发。

**投机解码 / Medusa** — 瓶颈:自回归 decode 串行且访存受限。机制:小模型起草 + 大模型并行验证 + 无损拒绝采样;Medusa 用多预测头免去独立草稿模型(见第三节)。

**量化(GPTQ / AWQ / SmoothQuant / KV cache 量化)** — 瓶颈:权重/激活/KV cache 占显存占带宽。机制:权重压 INT4/INT8,异常值特殊处理(见第三节);KV cache 也可量化(INT8/FP8),直接缓解长上下文的显存压力。

**MoE 专家并行(Expert Parallelism)** — 瓶颈:MoE(Mixture of Experts,混合专家)模型每次只激活少数专家,但所有专家权重都得存,且 token 要路由到不同专家所在的卡。机制:把不同专家分到不同卡(专家并行),**用 all-to-all 通信把每个 token 发往它该去的专家、再收回结果**。难点在负载均衡(热门专家所在卡过载)和 all-to-all 通信开销,这是当前推理系统的硬骨头之一。

**通信-计算重叠(overlap)** — 瓶颈:并行训练中通信(all-reduce/all-gather/all-to-all)若串行执行会让 GPU 干等。机制:**让通信和计算并发**——算这一层时异步传上一层的梯度,把通信延迟藏在计算背后。这是经典的**流水线隐藏延迟**,HPC 老兵的看家本领。

**算子融合与编译(operator fusion / compilation)** — 瓶颈:逐个小算子各自启动 kernel、各自读写 HBM,启动开销和访存开销叠加(又是带宽墙)。机制:**把多个连续算子融合成一个 kernel**,中间结果留在寄存器/SRAM 不落 HBM,减少 kernel 启动和访存。代表:`torch.compile`、XLA、TVM、Triton(让你用类 Python 写高效 GPU kernel)。FlashAttention 本质也是一种深度融合。

## 五、前沿战线:当下最活跃的研究在攻什么

**万卡集群训练稳定性与容错。** 当训练规模到上万张 GPU、跑数周到数月,**硬件故障从"小概率"变成"必然每天发生"**——某张卡掉了、某根网线抖了、某个节点 ECC 报错,整个训练就可能中断。研究焦点:**异步/增量 checkpoint**(不能每次存盘都停几分钟)、**故障快速检测与自动重调度**(几分钟内剔除坏卡、从最近检查点恢复)、**静默数据损坏(SDC)检测**(最可怕的是计算结果悄悄错了却不报错)。这是纯粹的大规模分布式系统可靠性问题,系统工程师的主场。

**loss spike(损失尖峰)与训练稳定性。** 大模型训练到中途 loss 突然飙升甚至 NaN,可能让几天的训练白费。成因复杂:数值溢出、坏数据、学习率/初始化问题、梯度爆炸。前沿在做:更稳的归一化和初始化方案、梯度裁剪策略、自动检测尖峰并回滚到尖峰前检查点、用 BF16/更稳的优化器减少数值问题。这是工程经验和理论交织的灰色地带,大厂的训练技术报告里这部分往往语焉不详。

**超长上下文推理与 KV cache 压缩。** 上下文从几千 token 涨到几十万甚至百万,**KV cache 直接爆炸**(它随序列长度和层数线性增长)。前沿方向:**KV cache 量化**(INT8/FP8/更低)、**KV cache 稀疏化/驱逐**(只保留重要 token 的 KV,如基于注意力分数淘汰,代表工作 H2O、StreamingLLM)、**多查询/分组查询注意力(MQA/GQA)** 从架构层面共享 KV 头来减少 KV 体积、**prefix caching**(系统级缓存公共前缀的 KV 跨请求复用)。

**prefill / decode 解耦式推理(disaggregation)。** 这是 2024 年的热门方向。关键洞察:LLM 推理的两个阶段**特性截然相反**——**prefill(处理输入 prompt)是算力受限**(一次并行处理所有输入 token,吃算力),**decode(逐 token 生成)是访存受限**(每步只算一个 token,吃带宽)。把它们混在同一批同一组卡上,会互相干扰、谁都没优化好。做法:**把 prefill 和 decode 拆到不同的 GPU 池**,各自用最适合的并行策略和批处理方式,中间把 KV cache 传过去。代表工作 DistServe、Splitwise。系统视角:这是**按工作负载特性做资源隔离与专门化**,和数据库把 OLTP/OLAP 分开是同一套哲学。

**训推一体(unified training-inference)。** RLHF、在线强化学习等场景需要"边训练边推理"(训练循环里要不断采样生成),传统上训练框架和推理框架是两套,切换开销大。前沿在探索让训练引擎高效内嵌推理能力,或让两者共享权重与显存布局,减少来回搬运。

**异构与国产算力适配。** 不再只盯着 NVIDIA。研究/工程焦点:在 AMD GPU、Google TPU、华为昇腾、各类国产加速卡上跑通并优化大模型;跨厂商的统一编译层(把模型 IR 映射到不同硬件);异构集群混合调度。这块工程量巨大且现实需求强烈,尤其在供应受限的背景下。

**MoE 推理系统。** 随着 MoE 架构成为大模型主流,如何高效服务 MoE 成了专门战线:专家负载均衡、all-to-all 通信优化、专家放置与缓存(把热门专家放更快的存储)、稀疏激活下的批处理。

## 六、公认难点 / 开放问题

**通信墙——最根本的瓶颈。** 卡越堆越多,但**卡间(NVLink)、机间(InfiniBand/以太网)的带宽增长远跟不上算力增长**。并行度越高,通信占比越大,最终通信成为天花板:你加再多卡,加速比也上不去(强扩展性失效)。这是物理和工程的双重约束——光速和铜/光纤带宽是硬限制。所有并行策略本质都在"用通信换显存/算力",而通信本身就是最贵的资源。难在:它是个全局优化问题,拓扑、并行策略、通信原语、重叠程度互相耦合,没有放之四海皆准的最优解。

**显存墙——永远不够用。** 模型增长速度长期快于单卡显存增长速度。ZeRO、offload、量化、重计算全是在拆这堵墙,但都有代价(通信、计算、精度)。根本困难在于:**显存容量、显存带宽、算力三者的增长速度不匹配**,且短期内无法靠单一硬件突破解决。

**长上下文的 KV cache 爆炸。** 这是显存墙在推理侧的尖锐体现。KV cache 随上下文长度线性增长,百万 token 上下文的 KV 可能比模型本身还大。压缩、稀疏化、量化都是治标,**如何在不损失长程依赖的前提下大幅压缩 KV,仍是开放问题**——驱逐了"看似不重要"的 token,可能恰好丢了后面要用的关键信息。

**训练稳定性与不可复现性。** loss spike、数值不稳定、大规模下的不确定性(浮点累加顺序不同导致结果有差异),让万卡训练像"走钢丝"。**为什么会 spike、如何提前预测和预防,缺乏完整理论**,很大程度上还靠经验和试错。大规模训练的可复现性也是难题——同样的代码同样的数据,换个集群结果可能不完全一致。

**大规模容错。** 万卡规模下故障是常态而非异常,但**做到"故障无感恢复"(发现坏卡、隔离、从检查点秒级恢复、不丢进度)**在工程上极难,尤其要兼顾 checkpoint 开销不能拖慢正常训练。SDC(静默错误)更是噩梦——结果错了却不报警,可能污染整个训练。

**推理成本——商业生死线。** 即便有了 vLLM、量化、投机解码,**大模型推理的单位成本对很多商业模式仍然偏高**。如何在保证质量和延迟的前提下持续把每 token 成本往下打,是永恒的工程压力。这背后是吞吐(批处理效率)、延迟(用户体验)、成本(硬件利用率)的三角权衡,按下葫芦浮起瓢。

**自动化并行(开放方向)。** 目前 3D 并行的配置很大程度靠专家手工调,**能否自动搜索出给定模型 + 给定集群下的最优并行策略**?Alpa(2022)等工作做了自动并行的探索,但通用、高效、好用的自动并行仍未完全成熟,是个有学术价值的开放问题。

## 七、就业与门槛

MLSys 是**当前 AI 领域最贴合系统工程师、且供需最紧张的方向之一**。岗位类型大致分四类:**训练框架/基础设施工程师**(并行训练、分布式、容错)、**推理引擎工程师**(vLLM/TensorRT-LLM 这类,做吞吐/延迟优化)、**GPU kernel / 性能工程师**(CUDA/Triton 算子优化)、**AI Infra / 平台工程师**(集群调度、资源管理)。

门槛与背景:**这个方向不强制要求 PhD**——它更看重扎实的系统功底(分布式、操作系统、体系结构、CUDA/并行编程、网络),这恰恰是资深系统工程师的强项。算法侧只需理解 Transformer 结构、训练数据流、注意力机制等"够用"的常识,不需要做算法创新。**做前沿研究(发顶会、攻自动并行这类开放问题)PhD 有帮助;但做工业界的训练/推理基础设施,强工程能力 + 对大模型机制的理解就足够,转入门槛对系统老兵相对友好。**

缺口与薪资:大模型基建人才**普遍紧缺**,尤其精通 GPU 性能优化和大规模分布式训练的人;薪资在 AI 领域处于高档(与算法岗相当甚至更高,因为稀缺)。具体数值因地区、公司、级别差异极大,此处不给确定数字——**待核**。
