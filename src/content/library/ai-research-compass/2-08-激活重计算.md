> 这一章把你从"知道反向传播要存激活、显存不够"带到能亲手推导出梯度检查点为什么是 O(√L)、能算出某个 Transformer 配置下重计算省了多少 GB、能看懂并手写 Megatron 的选择性重计算、并知道它和 FSDP / 序列并行怎么组合。读完你应该能在一个真实训练脚本里,把 OOM 调成不 OOM,且清楚自己付出了多少额外算力。

## 一、动机:为什么显存里塞满了"中间结果"

先把账算清楚。训练一个深度网络,显存被四类东西占用:

1. **参数(weights)**:模型本身。
2. **梯度(gradients)**:每个参数一个,反向算出来。
3. **优化器状态(optimizer states)**:Adam 下每个参数还要存一阶动量、二阶动量,加上 FP32 主副本。
4. **激活(activations)**:前向计算时每一层产生的中间张量。

前三类的总量只和**参数量**有关,和你喂多大的 batch、多长的序列**无关**。混合精度 + Adam 下,经典的"参数 × 16 字节"就涵盖了这三类(FP16 参数 2 + FP16 梯度 2 + FP32 主参数 4 + FP32 一阶动量 4 + FP32 二阶动量 4 = 16)。**这部分是固定开销,重计算动不了它。**

第四类——激活——是本章的主角,它的特点完全不同:**激活显存随 batch size、序列长度、网络层数线性增长。** 训练越往大模型、长序列推,激活会从"配角"膨胀成"主角",甚至超过参数本身。

为什么反向传播非要存激活?这是链式法则的硬约束,不是工程偷懒。考虑一层最简单的线性变换 `y = W·x`,反向传播要算两个东西:

```
对输入的梯度:  ∂L/∂x = Wᵀ · (∂L/∂y)
对权重的梯度:  ∂L/∂W = (∂L/∂y) · xᵀ      ← 注意这里需要前向的输入 x
```

**算权重梯度时必须用到前向的输入 `x`。** 同理,经过 ReLU 的地方反向要知道哪些位置被置零了(需要前向激活的符号),经过 Softmax / LayerNorm 的地方反向公式里也带着前向输出。所以框架的自动微分(autograd)在前向时会默默地把这些"反向时还要用"的张量挂在计算图上,留到反向用完才释放。**激活显存的本质,就是计算图为了反向而缓存的前向中间结果。**

这就引出了本章的核心交易:

> **能不能在前向时不存这些中间激活,等反向需要时,临时再前向算一遍把它"重建"出来?**

答案是能,代价是多算一遍前向。这就是 **梯度检查点(gradient checkpointing)**,也叫**激活重计算(activation recomputation)**。它是一笔纯粹的 **time-memory tradeoff(时间换空间)**:省下大量显存,付出约一次额外前向的算力。本章要讲透的,是这笔交易的**最优点在哪、为什么是 √L、以及工业界怎么把它做得更聪明**。

## 二、把激活显存算成一个可推导的量

要谈"省多少",先得有个可计算的激活显存模型。我们以标准 Transformer 层为单位来算,因为这是大模型训练的实际单元。

### 2.1 单层 Transformer 的激活字节数

设:

- `b` = micro-batch size(单次前向的样本数)
- `s` = 序列长度(sequence length)
- `h` = 隐藏维度(hidden size)
- `a` = 注意力头数(number of attention heads)
- 每个激活元素用 2 字节存(FP16/BF16)

一个标准 Transformer 层 = 一个自注意力块 + 一个 MLP 块,中间夹 LayerNorm、残差、Dropout。我们把"反向需要用到、因此要缓存"的张量逐个列出来(这正是 Megatron 2022 那篇论文附录做的事):

```
自注意力块需缓存的激活(字节,× b·s·h 为基准单位 2 字节):
  - LayerNorm 输入                : 2·b·s·h
  - Q, K, V 三个投影的输入(共享)  : 2·b·s·h        (存一份输入即可)
  - QKᵀ 矩阵(注意力分数)         : 2·b·s²·a       ← 注意这里是 s² !
  - Softmax 输出                  : 2·b·s²·a
  - Softmax 后的 Dropout mask     : 1·b·s²·a       (mask 用 1 字节)
  - 对 V 加权后的输出             : 2·b·s²·a  (实际是 attn·V 的输入,量级同上)
  - 输出投影的输入                : 2·b·s·h
  - 注意力块的 Dropout mask       : 1·b·s·h

MLP 块需缓存的激活:
  - LayerNorm 输入                : 2·b·s·h
  - 第一个线性层输入              : 2·b·s·h
  - GeLU 输入(算 GeLU 导数要用)  : 8·b·s·h        (中间维度 4h,故 2·b·s·4h)
  - 第二个线性层输入              : 8·b·s·h        (同样在 4h 维度)
  - MLP 块的 Dropout mask         : 1·b·s·h
```

把同量级的项合并(注:上面的逐项列表是帮助理解的示意,精确系数因张量并行复制项 / 分组方式与论文 Table/Eq.2 略有出入——照列表求和约得线性项 28、二次项 7,而论文公布值为 34、5,以 34/5 为准),Megatron 论文给出的**单层总激活字节数**为:

```
A_layer = s·b·h·(34 + 5·(a·s)/h)   字节                       (式 1)
```

这个公式值得逐项理解,而不是背下来。`34·s·b·h` 是所有"线性增长"项之和(投影输入、LayerNorm 输入、MLP 中间激活、各种 mask),它正比于 `s·b·h`。而 `5·(a·s/h)·s·b·h = 5·a·b·s²` 这一坨,来自注意力里那几个 `b·s²·a` 量级的张量(分数矩阵、Softmax 输出、加权输入、mask)——**它正比于 s²,是长序列下的显存炸点。**

**关键结论(加粗记住):短序列时激活显存 ≈ 34·s·b·h,正比于 s;长序列时 5·a·b·s² 那项主导,正比于 s²。** 这就是为什么"长序列训练"几乎等价于"激活显存危机",也是 FlashAttention(把 s² 的注意力矩阵从不存)和序列并行存在的根本原因。

### 2.2 全模型激活随层数线性增长

设模型有 `L` 层(layers)。若**不做任何重计算**,前向产生的所有层激活都要留到反向,所以:

```
A_total = L · A_layer = L · s·b·h·(34 + 5·a·s/h)   字节         (式 2)
```

**激活总量正比于 L——这就是"激活显存随层数线性增长"的精确含义。** 这是本章一切的出发点:它是线性的 O(L),我们要把它压下来。

### 2.3 一个具体数字,建立量感

拿一个 GPT-3 13B 规模的配置感受一下:`h = 5120`、`a = 40`、`L = 40`、`s = 2048`、`b = 1`(单 micro-batch),FP16。

先算单层(式 1):

```
基准元素数 s·b·h = 2048·1·5120 = 10.49 M 个元素
式 1 的系数:34 + 5·a·s/h = 34 + 5·40·2048/5120 = 34 + 80 = 114

逐项拆(系数已含字节单位，不再乘 2）：
  线性项 34·s·b·h = 34 × 10.49M               ≈ 356.5 MB
  s² 项  80·s·b·h = 80 × 10.49M               ≈ 838.9 MB   (= 5·a·b·s²)
A_layer = 114·s·b·h = 114 × 10.49M            ≈ 1.195 GB / 层
```

40 层:`A_total ≈ 40 × 1.195 ≈ 47.8 GB`。

**对比一下:13B 参数 × 16 字节 ≈ 208 GB 的参数+优化器开销(那是要靠张量并行/ZeRO 切分到多卡的)。激活这 47.8 GB 是每张卡在自己那份计算上都要扛的。** 一张 80GB 的 A100,光激活就吃掉一大半。把 batch 调大、序列加长,立刻 OOM。这就是重计算要解决的现实痛点。

> 注:式 1 假设 attention 用朴素实现(显式存 s² 分数矩阵)。若用了 FlashAttention,`5·a·s²` 那项基本消失,激活模型要改写。这一点在第六节"组合"里展开。

## 三、梯度检查点:机制与 √L 的推导

### 3.1 朴素的两个极端

先看两个极端,理解中间地带的价值。

**极端 A:全存(默认 autograd)。** 前向存下全部 L 层激活,反向直接用。显存 = O(L),额外计算 = 0。这就是式 2,会 OOM。

**极端 B:全不存,全重算。** 前向只保留最初的输入,任何中间激活都不存。反向需要第 `k` 层的激活时,从输入重新前向算到第 `k` 层。显存降到 O(1)(只存一层的工作集),但计算量爆炸:反向到第 `k` 层要重算 `k` 次前向,总额外计算 ≈ `1 + 2 + … + L = O(L²)`。**显存省到极致,算力变成平方级,完全不可接受。**

中间地带:**分段(segmentation)**。这才是工程上真正用的方案。

### 3.2 分段重计算的机制

把 L 层切成若干**段(segment)**,每段连续若干层。机制如下:

```
前向阶段:
  对每个段:
    - 正常前向计算这一段所有层
    - 只把【段的输入激活】存下来(作为 checkpoint)
    - 段内部所有中间激活算完即丢,不缓存

反向阶段(从最后一段往前):
  对每个段:
    - 取出之前存的【段输入激活】
    - 在这个段上【重新前向一遍】,这次保留段内中间激活
    - 用这些重建出的激活,正常做这一段的反向传播
    - 反向完,释放这一段的中间激活
```

直觉:**段边界是 checkpoint(存),段内部是"用算力临时重建"。** 任意时刻显存里只有"所有段边界激活" + "当前正在反向的那一个段的内部激活"。

### 3.3 推导:为什么最优是 √L

这是本章的数学核心,慢推一遍,你会发现 √L 是被一个最简单的"和最小化"逼出来的。

设把 L 层均匀切成 **m 段**,每段 `L/m` 层。我们要算**峰值激活显存** `M(m)`,它由两部分构成:

```
峰值显存 M(m) = (段边界 checkpoint 数) + (单段内部反向时的激活)
             = m              个段边界激活
             + (L/m)          个层的段内激活(反向一个段时要全部重建并保留)
```

把每层激活字节数记为常数 `c`(就是式 1 的 `A_layer`),则:

```
M(m) ≈ c · ( m  +  L/m )                                       (式 3)
```

对 `m` 求导找极小值:

```
dM/dm = c · ( 1 − L/m² )

令 = 0  →  m² = L  →  m* = √L                                  (式 4)
```

二阶导 `d²M/dm² = c·2L/m³ > 0`,确认是极小值。代回式 3:

```
M(m*) = c · ( √L + L/√L ) = c · 2√L = O(√L)                    (式 5)
```

**这就是 O(√L) 的来历:把网络切成 √L 段、每段 √L 层时,峰值激活显存从 O(L) 降到 O(√L),且这是均匀分段下的最优点。** 比如 L = 100,从存 100 份激活降到约 2√100 = 20 份,省 5 倍;L = 10000,从 10000 降到 200,省 50 倍。**L 越大,O(√L) 的收益越夸张——这正是它适配超深模型的原因。**

为什么是"和最小化":`m + L/m` 是一个经典形式,由 AM-GM 不等式(算术-几何均值)立刻知道 `m + L/m ≥ 2√(m · L/m) = 2√L`,等号当且仅当 `m = L/m`,即 `m = √L`。所以 √L 不是巧合,是 `a + b ≥ 2√(ab)` 这条初中不等式的直接产物。

### 3.4 推导:额外计算约等于一次前向

现在算代价。分段重计算的额外计算量是多少?

反向阶段,**每个段都要重新前向一遍**(为了重建段内激活)。所有段加起来,恰好就是把整个网络**多前向了一遍**:

```
额外计算 = ∑(每段重新前向) = 1 次完整前向 forward pass
```

注意这和极端 B(O(L²))的天壤之别:极端 B 的问题是"重算了很多次",而分段方案里**每层只被额外重算一次**(在它所属段反向时),所以总额外计算严格等于 1 次前向,**与分多少段无关**。

把它折算成占总训练算力的比例。标准 Transformer 的前向、反向 FLOPs 比约为 **1 : 2**(反向要算输入梯度和权重梯度两套,约前向两倍)。所以一次"前向+反向"的算力 ≈ `1 + 2 = 3` 个单位。多做一次前向 = `+1` 单位:

```
额外算力占比 ≈ 1 / (1 + 2) = 1/3 ≈ 33%                         (式 6)
```

**这就是"重计算约多花 1/3 算力"这句话的精确出处。** 注意这是"全量重计算每一层"的开销;第五节的选择性重计算能把这 1/3 压到个位数百分比。

### 3.5 把 √L 和 1/3 放在一起看

```
方案            激活显存        额外计算
全存            O(L)            0
全段重计算(√L) O(√L)           ~1 次前向 (~+33% 算力)
全不存(极端B)  O(1)            O(L²)  ← 不可用
```

**√L 分段是显存-算力曲线上的甜点:用固定的 +33% 算力,把显存从线性压到平方根。** 不是所有点都值得;但在"差一点就 OOM"时,这 33% 换来的是"能不能训"的区别,几乎总是划算。

## 四、代码:从 torch.utils.checkpoint 到手写分段

### 4.1 PyTorch 原生 API

PyTorch 的 `torch.utils.checkpoint` 帮你封装了"前向不存、反向重算"的全部脏活。最常用两个入口:

```python
import torch
import torch.nn as nn
from torch.utils.checkpoint import checkpoint, checkpoint_sequential

class TransformerBlock(nn.Module):
    def __init__(self, h, a):
        super().__init__()
        self.attn = nn.MultiheadAttention(h, a, batch_first=True)
        self.ln1, self.ln2 = nn.LayerNorm(h), nn.LayerNorm(h)
        self.mlp = nn.Sequential(nn.Linear(h, 4*h), nn.GELU(), nn.Linear(4*h, h))

    def forward(self, x):
        a, _ = self.attn(self.ln1(x), self.ln1(x), self.ln1(x))
        x = x + a
        x = x + self.mlp(self.ln2(x))
        return x

class Net(nn.Module):
    def __init__(self, L, h, a, use_ckpt=True, n_seg=None):
        super().__init__()
        self.blocks = nn.ModuleList([TransformerBlock(h, a) for _ in range(L)])
        self.use_ckpt = use_ckpt
        # 经验法则:段数取 √L 附近最省显存
        self.n_seg = n_seg or max(1, round(L ** 0.5))

    def forward(self, x):
        if not self.use_ckpt:
            for blk in self.blocks:
                x = blk(x)
            return x
        # checkpoint_sequential 自动把 blocks 均匀切成 n_seg 段
        # 段边界存激活,段内反向时重算 —— 正是第三节的机制
        return checkpoint_sequential(self.blocks, self.n_seg, x,
                                     use_reentrant=False)
```

几个**必须知道的坑**:

- **`use_reentrant`**:新代码一律传 `use_reentrant=False`(新版实现)。老的 `use_reentrant=True` 有一堆限制:不支持无输入张量需要梯度的情况、对 `.grad()` / 多次反向不友好、不能正确处理某些 keyword 参数。这是新手最常踩的兼容性坑。
- **随机性必须可复现**:段内有 Dropout 时,反向重算的那次前向**必须用和第一次前向相同的随机数种子**,否则重算出的 mask 和原来不一致,梯度就错了。`torch.utils.checkpoint` 默认会**保存并恢复 RNG 状态**(`preserve_rng_state=True`)来保证这点。关掉它会得到**静默错误**(梯度悄悄错,不报错),极难 debug。
- **`checkpoint` 包住的函数里不能有 in-place 操作改到反向需要的张量**,否则重算时输入已被改写。

### 4.2 手写一个分段重算,理解 autograd 在干什么

PyTorch 帮你藏掉了机制。为了真正理解,我们用自定义 `autograd.Function` 手写一个"单段重算"。核心就两步:前向时 `torch.no_grad()` 跑(不建图、不存中间激活),只存段输入;反向时重新建图前向一遍,再调 `torch.autograd.grad` 拿梯度。

```python
import torch

class RecomputeSegment(torch.autograd.Function):
    """对一个段做检查点:前向只存输入,反向重算。
    run_fn: 这一段的前向函数(若干层的串联)。"""

    @staticmethod
    def forward(ctx, run_fn, x):
        ctx.run_fn = run_fn
        ctx.save_for_backward(x)          # 只存【段输入】这一个 checkpoint
        with torch.no_grad():             # 关键:前向不建图、不缓存段内中间激活
            y = run_fn(x)
        return y

    @staticmethod
    def backward(ctx, grad_y):
        (x,) = ctx.saved_tensors
        x = x.detach().requires_grad_(True)
        with torch.enable_grad():         # 关键:这次前向【建图】,重建段内激活
            y = ctx.run_fn(x)             # ← 多花的那次前向就发生在这里
        # 用重建出的图,把上游梯度 grad_y 反传到段输入 x
        grad_x = torch.autograd.grad(outputs=y, inputs=x, grad_outputs=grad_y)
        return None, grad_x[0]            # run_fn 不需要梯度,返回 None

def checkpointed_forward(segments, x):
    """segments: List[Callable],每个是一个段的前向。"""
    for seg in segments:
        x = RecomputeSegment.apply(seg, x)
    return x
```

读这段代码时,把第三节的机制对照上:`forward` 里的 `torch.no_grad()` 对应"段内中间激活算完即丢";`save_for_backward(x)` 对应"只存段边界";`backward` 里的 `torch.enable_grad()` + 重跑 `run_fn` 对应"反向时段内重新前向一遍"。**这 20 行就是梯度检查点的全部本质,框架版只是加了 RNG 保存、设备处理、嵌套支持等工程细节。**

### 4.3 怎么切段:把 √L 落到代码里

`checkpoint_sequential(modules, n_seg, x)` 会把 `modules` 均匀切成 `n_seg` 段。把 `n_seg` 设成 `round(√L)` 就拿到了第三节推导的最优点。但**实践中段大小常被显存预算反推**,而不是死守 √L:你有多少富余显存,就允许多大的"单段内部激活",据此定段数(见练习 2)。√L 是"显存预算没特别约束、只想最省"时的默认答案。

## 五、进阶:选择性重计算(Selective Recompute)

全量重计算的 +33% 算力,在大规模训练里是实打实的钱。Megatron 2022 那篇论文(标题就叫 *Reducing Activation Recomputation in Large Transformer Models*)提出了更精细的思路:**不要无脑重算所有层,只重算那些"存起来很占显存、但重算很便宜"的算子。**

### 5.1 核心洞察:激活的"性价比"不一样

回到式 1。注意力里的 `5·a·b·s²` 那几项(分数矩阵、Softmax、Dropout mask)有个特点:

- **占显存极大**:正比于 s²,长序列下是显存第一杀手。
- **重算极便宜**:Softmax、Dropout 这些是**逐元素操作(element-wise)**,FLOPs 微乎其微,几乎不增加计算量。

而矩阵乘法(QKᵀ、attn·V、MLP 的两个 GEMM)的激活:

- **占显存相对小**:正比于 `s·h`(线性项)。
- **重算贵**:GEMM 是 FLOPs 大头,重算它们才是 +33% 的主要来源。

**结论一目了然:重算"便宜又占地"的逐元素/Softmax 部分,保留"贵又不太占地"的 GEMM 输入。** 这就是 selective recompute。

### 5.2 量化:省了多少显存、只花多少算力

Megatron 的 selective 策略具体是:**只对注意力内部那一坨(QKᵀ → Softmax → Dropout → 与 V 相乘)做重计算,其余全部正常缓存。** 用式 1 拆解:

- 被重算掉、从而不必存的激活 ≈ `5·a·b·s²` 那几项(论文里这部分单层约为 `2·b·s²·a`(分数)+ … 合计正是 s² 量级的大头)。
- 仍然缓存的线性项 ≈ `34·s·b·h`(注:论文 selective 下单层缓存约降到 `34·s·b·h`,把 s² 项几乎全砍掉)。

回到第二节那个 13B 例子:`34·s·b·h` 项 ≈ 356.5 MB/层,`5·a·s/h` 贡献的 s² 项 ≈ 838.9 MB/层。**selective recompute 砍掉的正是那 838.9 MB(占单层 70%),保留下来 356.5 MB。**(70% ≈ 838.9 / 1195)

代价呢?重算的只有 Softmax + Dropout + 两个矩阵乘(注意力内部的 BMM)。论文测得在大模型上**额外算力仅约 2%~4%**(远小于全量重计算的 33%),却拿回了大部分显存收益。**这是一个近乎免费的午餐:用 ~3% 算力换掉了 ~70% 的单层激活。**

> 注:上面 "约 2%~4%"、"约 70%" 是论文在其特定配置(如 22B/175B、s=2048)下的实测量级,不同 `s/h/a` 比例下数值会变;此处量级正确,精确数字以论文 Table 为准【待核:具体百分比随配置变化】。

### 5.3 伪代码:selective vs full

```python
# 全量重计算:整个 block 包进 checkpoint
def block_full_recompute(x):
    return checkpoint(transformer_block, x, use_reentrant=False)

# 选择性重计算:只把 attention 内核包进 checkpoint,其余正常算
def block_selective_recompute(x):
    h = layer_norm1(x)
    q, k, v = qkv_proj(h)                      # GEMM 输入正常缓存(贵,但占地小)
    # 只重算这一段:分数矩阵 + softmax + dropout + 与 V 相乘(便宜,但占地大 s²)
    attn_out = checkpoint(attention_core, q, k, v, use_reentrant=False)
    x = x + out_proj(attn_out)
    x = x + mlp(layer_norm2(x))                # MLP 激活正常缓存
    return x
```

**记住这个层次:full recompute(省最多显存、+33% 算力)→ selective recompute(省大部分显存、+几% 算力)→ 不重计算(省 0、+0)。** 工业界训练大模型时,selective 往往是默认起点,只有显存还不够才上 full。

### 5.4 颗粒度谱系

把重计算的"颗粒度"摊开看,你会发现它是连续可调的:

```
颗粒度从粗到细:
  整个 pipeline stage 重算  → 最省显存,算力代价最高
  每个 Transformer 层重算   → torch checkpoint 的典型用法,~+33%
  层内选择性重算(Megatron) → 只重算 attention core,~+几%
  算子级(只重算 dropout)   → 收益最小但几乎零算力代价
```

**这条谱系的统一视角:你在一条 (显存↓, 算力↑) 的帕累托曲线上选点。** 没有"对"的颗粒度,只有"匹配当前显存预算"的颗粒度。

## 六、和其他显存技术的组合

重计算很少单独用,它要和并行/分片技术拼在一起。讲清几个交互点,否则上了大规模就会算错账。

### 6.1 与 FSDP / ZeRO 的组合

**FSDP(Fully Sharded Data Parallel)/ ZeRO 切的是参数、梯度、优化器状态;重计算省的是激活——两者作用在显存的不同板块,正交、可叠加。** 一个典型的大模型训练同时开两者:FSDP 把那 16 字节/参数的固定开销切到 N 张卡(每卡 ≈ 1/N),激活重计算把每卡激活从 O(L) 压到 O(√L) 或 selective。

但有个**真实的交互坑**:FSDP 在前向时会临时 all-gather 出完整参数算这一层,算完释放。如果你**对一个 FSDP 包裹的模块做重计算**,反向重算时会**再 all-gather 一次参数**——于是多了一次通信。所以 FSDP + activation checkpoint 的代价不只是 +33% 算力,**还有额外的参数 all-gather 通信**。PyTorch FSDP 对此有专门处理(在反向重算窗口内复用已 gather 的参数),配置时要留意,否则通信量被低估。【待核:不同 FSDP 版本对"重算窗口内参数复用"的行为有差异】

### 6.2 与序列并行(Sequence Parallel)的组合

第二节说过,激活里 `34·s·b·h` 那部分(LayerNorm、Dropout 的激活)在张量并行下**每张卡都完整保留**,没被切。序列并行(Megatron 2022 同一篇论文提出)沿**序列维 s** 把这些激活也切到张量并行的各卡上,每卡只存 `1/t`(t 为张量并行度)。

**序列并行和选择性重计算是论文里搭配使用的两件套:** 序列并行处理线性项(`34·s·b·h` → 切成 `1/t`),选择性重计算处理 s² 项(`5·a·b·s²` → 重算掉)。两者合起来,把单层激活的两个组成部分各个击破。这是工业级长序列训练的标准配方,不是二选一。

### 6.3 与流水线并行的组合

流水线并行把 L 层切到不同卡组(stage),每个 stage 上还会同时有多个 micro-batch 在"流"(in-flight),**每个 in-flight 的 micro-batch 都要存自己的激活**。所以流水线越深、in-flight micro-batch 越多,激活压力越大——**这正是重计算在流水线并行里几乎是标配的原因。** 注意此时显存峰值要乘上 in-flight micro-batch 数,算账时别漏。

### 6.4 与卸载(offload)的对比

还有一条路是**激活卸载(activation offloading)**:把激活搬到 CPU 内存/NVMe,反向时再搬回来。它和重计算是**互补**而非互斥:

```
                   省显存方式         代价
重计算              丢弃后重算         额外算力(compute-bound 友好)
卸载                搬到 CPU/NVMe      额外 PCIe/NVMe 带宽(IO-bound)
```

**选哪个看你的瓶颈:算力有富余、PCIe 紧张 → 重计算;算力吃紧、PCIe 有富余 → 卸载。** 大规模训练常常两者都用:贵的激活卸载,便宜的激活重算。

## 七、设计权衡与常见坑

把散落各节的"坑"汇成一张清单,这是 review 别人训练脚本时的检查表:

1. **盲目全量重计算。** 很多人一 OOM 就 `checkpoint` 整个模型,白白吃 33% 算力。**先上 selective(只重算 attention core),通常 ~3% 算力就解决问题。** 全量是最后手段。

2. **段数不等于 √L 就慌。** √L 是"无额外约束下显存最省"的解。实践中段数由显存预算反推更合理(练习 2)。而且 `m + L/m` 在 m\* 附近很平坦(二阶导小),**偏离 √L 一点点对峰值显存影响很小**,不必纠结精确值。

3. **关掉 RNG 保存导致静默梯度错误。** Dropout / 随机性算子在重算时若不复现随机状态,梯度悄悄算错、loss 看起来还在降,但模型学不好。**永远让 `preserve_rng_state=True`(默认)。** 这是最隐蔽、最贵的坑。

4. **`use_reentrant=True` 的兼容性陷阱。** 新代码一律 `use_reentrant=False`。reentrant 版对多次反向、`torch.autograd.grad`、无梯度输入等场景有静默限制。

5. **重计算掩盖真实算力账。** 上了重计算后,你的 MFU(model FLOPs utilization)分母要算上额外那次前向,否则会误判训练效率。报告吞吐时说清"含不含重计算开销"。

6. **对已经是 memory-bound 的小算子重计算反而更慢。** 重计算的前提是"算比存便宜"。若某算子本身就是访存受限、重算时又要把输入从 HBM 读一遍,可能得不偿失。**重算的甜点是 compute 富余的场景。**

7. **和 FSDP 组合时漏算 all-gather 通信。** 见 6.1,反向重算会触发额外的参数 gather,通信账要补上。

**一句话权衡总结:重计算把"显存不够"这个硬约束,转化成"算力多花一点"这个软成本——它让原本训不了的配置变得训得了,代价是可控且通常很小的算力,前提是你选对了颗粒度。**

## 八、动手练习

**练习 1(推导题).** 假设把 L 层切成 m 段,但**允许段大小不均匀**。设第 i 段有 `Lᵢ` 层(`∑Lᵢ = L`)。峰值显存 = 段边界数 + 最大段的层数 = `m + max(Lᵢ)`。证明:在 m 固定时,峰值显存被**均匀分段**最小化;并说明为什么"均匀分段 + 再对 m 求最优"会回到 m\* = √L。
*提示:固定 m 时 `max(Lᵢ) ≥ L/m`,等号在均匀时取得;代回 `m + L/m` 再用 AM-GM。*

**练习 2(估算题).** 你有一张 80GB A100,参数+优化器+梯度已用掉 50GB,只剩 30GB 给激活。模型 L = 48 层,单层激活 `A_layer = 1.2 GB`(类似第二节 13B)。问:(a) 不重计算需要多少激活显存?会不会 OOM?(b) 用均匀分段重计算,峰值激活 ≈ `A_layer·(m + L/m)`,要让它 ≤ 30GB,m 至少取多少?(c) 此时相比 √L 最优点,你多存了还是少存了显存?
*提示:(a) 48 × 1.2 = 57.6 GB,远超 30。(b) 解 `1.2·(m + 48/m) ≤ 30` 即 `m + 48/m ≤ 25`,解一元二次。(c) √48 ≈ 6.93,比较你解出的 m 和它。*

**练习 3(编码题).** 把 4.2 节的 `RecomputeSegment` 跑起来:构造一个 4 层 MLP,用它做端到端反向,和"不重计算的普通前向反向"对比**梯度数值**(应当逐元素相等,误差 < 1e-5)和**峰值显存**(`torch.cuda.max_memory_allocated()`,重计算版应明显更低)。再在段内加一个 `nn.Dropout`,验证:不保存 RNG 状态时梯度对不上,保存后对得上。
*提示:对比梯度用 `torch.allclose`;测显存前先 `torch.cuda.reset_peak_memory_stats()`;手动复现 RNG 用 `torch.get_rng_state()` / `set_rng_state()` 把 forward 的状态在 backward 前恢复。*

**练习 4(分析题).** 推导:对一个 Transformer 层,full recompute 和 selective recompute 各自的额外 FLOPs。已知前向 FLOPs 主要由 4 个 GEMM(QKV 投影、注意力 BMM ×2 折算、输出投影、MLP 两个)贡献,逐元素操作(Softmax/Dropout/LayerNorm)FLOPs 可忽略。说明为什么 selective 的额外算力远小于 full,并估算两者比值。
*提示:full 重算所有 GEMM;selective 只重算注意力内部的两个 BMM(QKᵀ 和 attn·V),不重算 QKV 投影、输出投影、MLP。比较被重算的 GEMM FLOPs 占总前向 GEMM 的比例。*

## 十、小结与承上启下

这一章把激活重计算从"一个能省显存的开关"讲成了"一条可推导、可量化、可调颗粒度的工程曲线":

- **机制**:反向需要前向激活(链式法则的硬约束)→ 前向不存、反向临时重算 → 用算力换显存。
- **核心推导**:均匀分 √L 段时峰值激活 `O(L) → O(√L)`(由 `m + L/m ≥ 2√L` 的 AM-GM 逼出),额外算力 ≈ 一次前向 ≈ **总算力的 1/3**(由前向:反向 = 1:2 推出)。
- **进阶**:selective recompute 沿"性价比"切——只重算便宜又占地的 attention 内核,用 ~3% 算力拿回 ~70% 单层激活。
- **组合**:和 FSDP(切固定开销)、序列并行(切线性激活项)、流水线(放大激活压力)、卸载(换 IO 而非算力)各司其职、可叠加。

放到整门课的坐标里:**前面的章节告诉你显存被谁占了(参数/梯度/优化器/激活)、并行怎么切固定开销;本章专攻"激活"这块、且是单卡就能用的省显存手段。它和后面的 ZeRO/FSDP(切固定开销)、FlashAttention(让 s² 激活根本不产生,从源头消灭式 1 里那项)是互补的三把刀。** 理解了本章的 (显存↓, 算力↑) 帕累托视角,你再看 FlashAttention 会发现它走的是另一条路——**不是"重算激活",而是"重排计算让大激活压根不落地"**,从而在省显存的同时还更快。那是下一站要讲透的事。
