> 这一章把你从"模型为什么要预测下一个 token"带到"一段原始文本到底怎么变成一串整数、再变成一串向量送进 Transformer"。读完你能自己实现一个 BPE 分词器,算清楚嵌入矩阵吃掉多少参数,理解为什么输入和输出共享权重(weight tying)既省参数又常常更好,并能判断一个词表设计在中文和代码上会踩什么坑。

## 引言:第 0 步决定了上限

上一章我们说清了语言模型在干什么——给序列建条件概率 `P(xₜ | x_<t)`,用最大似然(等价于最小化交叉熵)去逼近真实分布。但那一章里有个被悄悄跳过的关键词:**token**。我们说"预测下一个 token",可 token 到底是什么?是字?是词?是字节?

这个问题不是细节,而是**整个建模流程的入口,它直接决定序列长度、词表大小、嵌入参数量、稀有词和未登录词(OOV)的处理方式,甚至影响模型能不能做算术、能不能拼出没见过的人名**。Transformer 的输入永远是一串整数 ID,模型本身完全不认识字符;从"字符串"到"整数序列"的这一步映射,叫**分词(tokenization)**;从"整数序列"到"向量序列"的这一步,叫**嵌入(embedding)**。这两步合起来,是把人类语言喂给神经网络的全部接口。

很多人把分词当成预处理脚本里一行 `tokenizer(text)` 就略过了。但实际上:**分词器是模型的一部分,且是冻结的那部分**。它在预训练前就定死了,之后几乎不能改;改了等于换一个模型重训。它的设计错误(比如数字被拆得乱七八糟、某种语言被拆成单字节)会永久限制模型在那类输入上的表现。所以这一章值得认真讲透。

这一章的逻辑链条:
1. **为什么字符级和词级都不行** → 子词(subword)是被逼出来的折中。
2. **三种主流子词算法**(BPE / WordPiece / Unigram)各自怎么构造词表,差别在哪。
3. **词表大小的权衡**:它同时拉扯序列长度、嵌入参数量、稀有词覆盖。
4. **嵌入查表的本质**:它不是"查字典",而是一次 one-hot 矩阵乘法。
5. **weight tying**:为什么输入嵌入和输出投影该共享同一个矩阵。
6. **嵌入空间的几何**:相似的 token 为什么会靠在一起,这是后面一切语义计算的物理基础。

## 一、为什么不用字符,也不用词

### 问题:粒度的两难

我们要选一个"原子单位"来切文本。直觉上有两个极端选择。

**极端一:字符级(character-level)。** 词表就是所有可能的字符。英文大约几十到一百多个(字母、数字、标点);Unicode 全集是十几万个码点。最极端是**字节级**:词表恰好 256,任何文本(任何语言、任何 emoji、任何二进制)都能无损表示,永远不会有未登录词。听起来很美。

代价是**序列变得极长**。一句 100 个英文单词的话,词级是 ~100 个 token,字符级要 ~500 个(英文单词平均 ~5 个字符)。而下一章(第 02 章)会看到,注意力的计算和显存是序列长度 n 的**二次方** `O(n²)`。序列长 5 倍,注意力开销 25 倍。这是字符级的致命伤:**它把建模负担从词表转移到了序列长度,而序列长度的代价是平方级的**。

还有一层更隐蔽的代价:**字符级让模型必须从零学会"字符怎么拼成词"这件低层次的事**,把宝贵的层数和参数浪费在拼写上,而不是语义和推理上。

**极端二:词级(word-level)。** 词表就是语料里出现过的所有词。序列短(一词一 token),语义单位天然对齐。

代价有两个,都很致命:

- **词表爆炸(vocabulary explosion)。** 自然语言的词频服从 **Zipf 定律**(第 k 常见的词,频率大约正比于 1/k)。这意味着词的长尾极长:英语真实语料里几十万、上百万个不同的词形(还没算形态变化 run/runs/running/ran、复合词、拼写变体、专有名词)。词表一旦到几十万,嵌入矩阵参数量爆炸(后面会算),而长尾里每个词只出现几次,根本学不好。
- **未登录词(OOV, out-of-vocabulary)。** 词表是闭集,训练时没见过的词(新词、错拼、罕见专名、代码里的变量名)全都映射到一个特殊的 `<UNK>` token。模型一旦吐出 `<UNK>` 就丢失了全部信息——它既不知道这个词是什么,也无法生成它。对一个要生成任意文本的模型,这是不可接受的。

把两难列成表(数量级是直觉,不是精确值):

```
                字符/字节级        词级
序列长度         很长(×4~5)       短
词表大小 V        小(256~10万)     极大(数十万~百万)
OOV              永不发生           严重
拼写/形态        模型从零学         天然对齐
适合的负担        压在序列长度       压在词表
```

### 机制:子词是被逼出来的折中

**子词(subword)分词的核心思想:常见词作为整体保留,稀有词拆成更小的、可复用的片段。** "tokenization"可能是一个 token,而"antidisestablishmentarianism"会被拆成几个常见词根/片段;一个没见过的新词,也能用已有片段拼出来,从而**彻底消灭 OOV**。

为什么这个折中如此有效?因为它顺应了 Zipf 分布:**把头部高频词用单 token 覆盖(保证序列短),把长尾用可组合的片段覆盖(保证有限词表能表示无限词)**。一个 3 万~10 万的子词词表,既能让英文序列长度接近词级,又能表示任何输入。这正是 GPT、LLaMA、BERT 等几乎所有现代大模型的选择。

子词分词还有几个关键性质:

- **无 OOV**:最坏情况退化到字符/字节,任何字符串都能编码。
- **形态泛化**:`run`、`running`、`runs` 可能共享 `run` 这个子词,模型能在它们之间迁移知识。
- **跨语言**:字节级 BPE(GPT-2 用的)对任何 UTF-8 文本都工作,不需要预先知道语言。

接下来的问题就变成:**给定语料和目标词表大小 V,怎么决定切出哪些子词?** 这就是 BPE / WordPiece / Unigram 三种算法回答的问题。它们的目标一致(找一组好的子词单元),但**构造方式截然不同**:BPE 和 WordPiece 是**自底向上合并**,Unigram 是**自顶向下剪枝**。

## 二、BPE:从字符开始,贪心合并最高频对

**BPE(Byte-Pair Encoding,字节对编码)** 原本是 1994 年的一个数据压缩算法,Sennrich 等人在 2016 年把它引入神经机器翻译做子词分词,从此成为主流。它的训练过程极其简单,但理解透了能帮你看懂一大半分词器行为。

### 机制:训练阶段(构造合并规则)

BPE 训练做的事:**从最小单元(字符)出发,反复找出语料中最高频的相邻单元对,把它合并成一个新单元,记录这条合并规则,直到达到目标词表大小。**

逐步白板推导。假设我们有一个玩具语料(为了清楚,用词频字典表示,词尾加一个特殊结束符 `</w>` 区分词边界——这是经典做法,字节级 BPE 用别的方式处理边界):

```
语料(词 : 频次):
  "low"     : 5
  "lower"   : 2
  "newest"  : 6
  "widest"  : 3

第 0 步:全部拆成字符,初始词表 = 所有字符
  l o w </w>             : 5
  l o w e r </w>         : 2
  n e w e s t </w>       : 6
  w i d e s t </w>       : 3
  初始词表 V = {l,o,w,e,r,n,s,t,i,d,</w>}

第 1 步:统计所有相邻对的频次
  (e,s): 出现在 newest(6) + widest(3) = 9   ← 最高
  (s,t): 6 + 3 = 9                          ← 三路并列最高(取一个,按实现的 tie-break)
  (t,</w>): 6 + 3 = 9                        ← 同为最高
  (l,o): 5 + 2 = 7
  (o,w): 5 + 2 = 7
  ... 
  假设 tie-break 选 (e,s),合并成 "es"
  记录合并规则 #1: (e, s) -> es
  语料更新:
    n e w es t </w>      : 6
    w i d es t </w>      : 3

第 2 步:重新统计
  (es,t): 6 + 3 = 9                          ← 最高
  合并规则 #2: (es, t) -> est
    n e w est </w>       : 6
    w i d est </w>       : 3

第 3 步:
  (est,</w>): 6 + 3 = 9                       ← 最高(est 总在词尾)
  合并规则 #3: (est, </w>) -> est</w>
    n e w est</w>        : 6
    w i d est</w>        : 3

第 4 步:
  (l,o): 5 + 2 = 7                            ← 最高
  合并规则 #4: (l, o) -> lo
    lo w </w>            : 5
    lo w e r </w>        : 2

第 5 步:
  (lo,w): 5 + 2 = 7
  合并规则 #5: (lo, w) -> low
    low </w>             : 5
    low e r </w>         : 2
  ... 继续直到词表达到目标大小 V
```

**关键点**:BPE 训练的产物有两个——(1)**词表**(所有出现过的单元,包括原始字符和所有合并出来的子词),(2)一个**有序的合并规则列表**(merge #1, #2, ...)。顺序至关重要,因为编码时要按这个顺序贪心地应用。

### 机制:编码阶段(用合并规则切新文本)

训练完后,怎么给一个新词(比如 "lowest")分词?**把它拆成字符,然后按训练时记录的合并规则顺序,反复应用能应用的合并,直到没有规则可用。**

```
编码 "lowest":
  初始:  l o w e s t </w>
  扫规则列表,按顺序找第一条能应用的:
  #1 (e,s)->es:     l o w es t </w>
  #2 (es,t)->est:   l o w est </w>
  #3 (est,</w>)->est</w>:  l o w est</w>      (lowest 以 t 结尾,故得到 t</w>,est 与 </w> 相邻,merge #3 适用)
  #4 (l,o)->lo:     lo w est</w>
  #5 (lo,w)->low:   low est</w>
  最终: ["low", "est</w>"]   → 两个 token
```

注意 "lowest" 在训练语料里**从没出现过**,但 BPE 用 "low" + "est" 拼出来了——这就是子词泛化的威力。

### 代码:BPE 训练的核心循环

下面是 BPE 训练最核心的逻辑,去掉了工程细节(并行、缓存、边界处理),保留机制。能体现"统计对频 → 取最高 → 合并 → 记录"这个循环。

```python
from collections import Counter

def get_pair_counts(word_freqs):
    """统计所有相邻 symbol 对的加权频次。
    word_freqs: dict, key 是 symbol 元组如 ('l','o','w','</w>'),value 是该词频次。"""
    pairs = Counter()
    for word, freq in word_freqs.items():
        for i in range(len(word) - 1):
            pairs[(word[i], word[i + 1])] += freq   # 用词频加权,而非词内出现次数
    return pairs

def merge_pair(pair, word_freqs):
    """把语料中所有出现的 pair 合并成一个新 symbol。"""
    a, b = pair
    merged = a + b
    new_word_freqs = {}
    for word, freq in word_freqs.items():
        new_word, i = [], 0
        while i < len(word):
            if i < len(word) - 1 and word[i] == a and word[i + 1] == b:
                new_word.append(merged)   # 合并
                i += 2
            else:
                new_word.append(word[i])
                i += 1
        new_word_freqs[tuple(new_word)] = freq
    return new_word_freqs

def train_bpe(word_freqs, num_merges):
    """word_freqs 初始为字符级拆分;返回有序的合并规则列表。"""
    merges = []
    for _ in range(num_merges):
        pairs = get_pair_counts(word_freqs)
        if not pairs:
            break
        best = max(pairs, key=pairs.get)     # 贪心:取频次最高的相邻对
        word_freqs = merge_pair(best, word_freqs)
        merges.append(best)                  # 记录顺序,编码时按此顺序应用
    return merges
```

编码时反过来,按 `merges` 顺序对一个词反复合并即可(留作练习的扩展)。

### 分析:复杂度与设计要点

- **贪心,非全局最优。** BPE 每步只看当前最高频对,不回头、不全局优化。这意味着它**不保证**找到"在某种目标下最优"的词表——它只是经验上很好用。这点和 Unigram(下面)是本质区别。
- **确定性编码。** 给定合并规则,一个词的切法是**唯一确定**的(按规则顺序贪心)。这既是优点(可复现)也是缺点(没有切分的多样性,而 Unigram 可以采样不同切法做正则化)。
- **训练复杂度。** 朴素实现每次合并要扫一遍全语料统计对频,做 V 次合并就是 O(V × 语料长度),对大语料较慢;实际实现用增量更新(只更新受影响的对)和优先队列加速。这是工程细节,不影响理解。
- **字节级 BPE(GPT-2 起)。** 不在字符上做,而在**字节(0-255)**上做 BPE。好处:词表初始恰好 256,任何 UTF-8 文本都覆盖,真正零 OOV,且不需要 `</w>` 这类语言相关的边界处理(用一套字节级的可见映射处理空格等)。**现代 GPT 系基本都用字节级 BPE。** 代价是某些非拉丁文字(中文、日文)一个字符占多个字节,会被拆得更碎(后面词表权衡会细说)。

## 三、WordPiece:按似然增益合并,而非频次

**WordPiece** 是 BPE 的近亲,最早用于 Google 的语音和后来的 **BERT**(2018)。结构上和 BPE 几乎一样——也是自底向上合并——但**合并的判据不同**。

### 机制:从"最高频"到"最大似然增益"

BPE 选合并对的规则是:**频次最高**。WordPiece 的规则是:**合并后让语料的似然(likelihood)增加最多的那一对**。

具体推导。假设我们有一个一元语言模型(unigram LM),把语料看成各 token 独立出现,token t 的概率 p(t) ∝ 它的频次。一个由 token 序列组成的语料,其(对数)似然就是各 token 对数概率之和。现在考虑把相邻对 (a, b) 合并成新 token "ab":合并前这两个位置贡献 `log p(a) + log p(b)`,合并后贡献 `log p(ab)`。WordPiece 选的是让**整体似然增量最大**的对,可以推出它本质上在最大化一个打分:

```
score(a, b) ≈ log p(ab) − log p(a) − log p(b)
            = log [ count(ab) / (count(a) · count(b)) ] + 常数

(把概率写成 频次/总数,总数项在比较不同 pair 时是公共项,可约去)
```

**对比 BPE:**

- **BPE**:`score = count(a,b)`(只看合并对自己出现多少次)。
- **WordPiece**:`score ≈ count(a,b) / (count(a)·count(b))`(看合并对相对于两个组成部分单独出现的"超出预期"程度)。

这个分母是关键。它惩罚那些"组成部分本身就极高频"的对。举例:`th` 在英语里频次极高,BPE 会很早合并它;但如果 `t` 和 `h` 各自也极高频(它们确实是),WordPiece 的分母很大,分数被压低——**WordPiece 更倾向合并那些"特别爱粘在一起、单独出现少"的对**,这其实就是统计学里的**逐点互信息(pointwise mutual information)**的味道。直觉上,这能切出更有"黏着性"的、更像语言学语素的单元。

### 编码与标记差异

- **编码方式不同。** 经典 BPE 编码按合并规则顺序;WordPiece 编码用**最长匹配**:从词首贪心找词表里最长的能匹配的子词,切下,继续。词内非首子词加 `##` 前缀(BERT 的标志,如 `playing -> play ##ing`),用来区分"词的开头"和"词的中间"。
- **判据需要计数,而非顺序。** WordPiece 训练同样迭代,每步重算各候选对的 score 取最大,合并,直到目标大小。

### 分析

- WordPiece 和 BPE 的产物词表在实践中高度相似,差异主要在长尾切分上。把握住一点就够:**BPE 看绝对频次,WordPiece 看(近似)互信息/似然增益**。
- 「待核」:WordPiece 训练判据的具体打分公式在不同文献/实现里有细节差异(是否取对数、归一化方式),上面给的是其核心思想的等价形式,落地到某个具体实现(如 HF `tokenizers`)前建议核对该实现的精确定义。

## 四、Unigram LM:从大词表自顶向下剪枝

**Unigram language model 分词**(Kudo,2018,实现于 **SentencePiece**)走的是**完全相反**的方向:不是从字符往上合并,而是**先放一个超大的候选词表,再迭代地把"删掉损失最小"的子词剪掉,直到目标大小**。这是当前很多多语言模型(以及 LLaMA 系用的 SentencePiece)的选择。

### 机制:概率模型 + EM + 剪枝

Unigram 把分词显式建模成一个**概率模型**。假设每个子词 x 有一个概率 p(x),一个句子的某种切分 `s = (x₁, ..., x_M)` 的概率是各子词概率的乘积(unigram 假设——子词彼此独立):

```
P(s) = ∏ₘ p(xₘ)
```

一个句子通常有**多种合法切分**(比如 "hello" 可以切成 ["hello"] 或 ["hel","lo"] 或 ["h","e","l","l","o"]……),Unigram 模型给每种切分一个概率。整个语料的似然是:对每个句子,在所有可能切分上求和(marginalize):

```
L = Σ_句子 log ( Σ_{s ∈ 该句所有切分} P(s) )
```

训练目标:**找一组子词概率 {p(x)} 和一个词表,使语料似然 L 最大**。流程:

```
1. 初始化一个超大候选词表(比如用所有高频子串、或先跑一遍 BPE 得到一个大词表)。
2. 重复:
   a. (E 步 + M 步,EM 算法)固定词表,用 EM 估计每个子词的概率 p(x):
      - E 步:对每个句子,用前向-后向(或维特比)算出各切分的后验,
              统计每个子词的期望出现次数。
      - M 步:p(x) ← 该子词期望次数 / 总期望次数。
   b. (剪枝)对词表里每个子词,估算"删掉它会让总似然下降多少"
      (loss_i = 删掉子词 i 后语料似然的减少量)。
   c. 按 loss 从小到大,删掉一批(如最不重要的 10~20%)子词,
      但保留所有单字符(保证任何串都能被切分,即无 OOV)。
3. 直到词表缩到目标大小 V。
```

**核心区别讲清楚:** Unigram 维护的是一个**显式的概率分布 + 一个被剪枝的词表**,它的判据是"这个子词对解释语料的贡献(似然)有多大";BPE/WordPiece 维护的是**合并规则**,判据是"频次/互信息"。

### 编码:用维特比找最优切分

编码一个新句子时,Unigram **不是**贪心,而是用**维特比(Viterbi)算法**在所有合法切分里找概率最大的那个:

```
对句子 c[0..n],定义 best[i] = 切到第 i 个字符为止的最优(最大对数概率)切分得分。
best[0] = 0
for i in 1..n:
    best[i] = max over j<i such that c[j..i] 在词表里 of:
                  best[j] + log p(c[j..i])
回溯得到最优切分。
复杂度 O(n²) 最坏(每个 i 看所有 j),实际用词表里最长子词长度限制 j 的范围,接近 O(n·L)。
```

这带来一个独特能力:**子词正则化(subword regularization)**。因为模型对每种切分都有概率,训练时可以**按概率采样不同切分**(不总用最优切分),相当于数据增强,让模型对分词的不确定性更鲁棒。BPE 的确定性编码做不到这点(BPE-dropout 是事后给 BPE 打的一个类似补丁)。

### 分析:三种算法横向对比

```
              BPE              WordPiece          Unigram LM
方向          自底向上合并       自底向上合并        自顶向下剪枝
判据          相邻对频次最高      似然增益/互信息      子词对语料似然的贡献
是否显式概率   否               否(隐式 unigram)   是(显式 p(x))
编码          按合并规则贪心      最长匹配(##)       维特比找最优切分
切分多样性     无(确定)         无(确定)          有(可采样,正则化)
代表           GPT-2/GPT 系     BERT               LLaMA(SentencePiece)、T5
```

一句话记忆:**BPE 数频次往上拼,WordPiece 看黏着度往上拼,Unigram 从大词表按"似然贡献"往下砍**。三者实践效果都不错,选择往往是历史与生态原因多于纯技术优劣。

## 五、词表大小的权衡:V 同时拉扯三件事

词表大小 V 是分词器最重要的超参数(常见取值 3 万~12 万这个量级,「待核」具体模型的默认值)。它**同时**影响三件互相冲突的事,这是个三方拔河:

**1. 序列长度(V 越大越短)。** V 越大,越多的常见词/词组能被单 token 覆盖,同一段文本的 token 数越少。序列短 → 注意力开销(`O(n²)`)小、推理快、能塞进上下文窗口的内容多。这是 V 大的好处。

**2. 嵌入参数量(V 越大越多)。** 嵌入矩阵是 V × d(下一节细推)。V 翻倍,嵌入和输出投影的参数翻倍,显存和计算都涨。对小模型,这部分可能占总参数的一大块(下节会算到具体比例)。这是 V 大的代价。

**3. 稀有词学习(V 太大反而学不好)。** V 越大,越多长尾词被单独成 token,而长尾 token 在语料里出现次数少,**它们的嵌入向量得到的梯度更新次数少,学不充分**(这叫嵌入的"训练不足")。极端地,V 设到几十万,大量 token 一辈子只在训练里出现几十次,它们的嵌入基本是噪声。反过来 V 太小,常见词被切碎,序列变长且模型得花力气重新拼装语义。所以存在一个**甜点区**。

把三者的权衡写成定性关系:

```
V ↑  →  序列长度 n ↓        (好:注意力 O(n²) 省,上下文利用率高)
V ↑  →  嵌入参数 V·d ↑      (坏:显存/参数膨胀,小模型尤甚)
V ↑  →  每个 token 平均频次 ↓ → 长尾嵌入欠训练  (坏)

最优 V* 是这三者的平衡点,随模型规模 / 语料 / 语言而变。
经验上大模型倾向更大词表(参数占比小、序列省得多更值);
小模型倾向更小词表(嵌入占比太重)。
```

### 中文和代码:词表设计的两个硬骨头

这是 spec 点名要讨论的,也是实战里最容易踩的坑。

**中文。** 字节级 BPE(GPT-2 那套)对中文很不友好:一个常见汉字在 UTF-8 里占 **3 个字节**,如果分词器主要在英文语料上训练、没给中文足够的合并规则,那么一个汉字可能被切成 2~3 个 token(甚至切在字节中间,产生无意义的字节片段)。**后果**:同样信息量的中文文本,token 数可能是英文的数倍——上下文窗口被吃掉、推理变贵、API 按 token 计费时中文用户吃亏。**对策**:在词表训练语料里加入足量中文,让常见汉字/词组各自获得单 token(很多中文/多语言模型这么做);或者用字级 + 子词混合。这也是为什么"在英文数据上训的分词器直接用于中文"是个典型错误。

**代码。** 代码的 token 化有几个独特问题:(1)**缩进和空白**——Python 靠缩进,而很多分词器会把连续空格切碎或合并得不一致,GPT-2 的字节级 BPE 早期对多个空格处理就不理想(后来的模型专门优化了空白 token,比如把常见的缩进量做成单 token)。(2)**标识符**——`getUserById` 这种驼峰命名、`user_id` 这种下划线命名,会被切成多个子词,模型得学会重组。(3)**数字**——见下。这就是为什么代码模型(如 Code Llama 一类)往往用**专门为代码优化过的分词器**,对空白、常见关键字、缩进有特殊处理。

**数字(顺带一提,很重要)。** 数字的分词直接影响模型能不能做算术。如果 "1234" 被切成 ["12","34"] 而 "1235" 被切成 ["123","5"],模型很难学到稳定的进位规律。**有的模型(如部分 LLaMA/相关实现「待核」)强制把数字按单个数字(digit)切分**(每个数字一个 token),让算术结构更规整;这是一个被验证有效的分词设计选择。这个例子完美说明:**分词不是无害的预处理,它会决定模型能力的上限**。

## 六、嵌入:查表的本质是 one-hot 乘矩阵

分词把文本变成了一串整数 ID(每个 ID ∈ {0, 1, ..., V−1})。但神经网络要的是连续向量。**嵌入层(embedding layer)** 就是把每个整数 ID 映射成一个 d 维向量的查找表。

### 机制:嵌入查表 = one-hot × 嵌入矩阵

嵌入矩阵 **E ∈ ℝ^{V×d}**:第 i 行 E[i] 就是 token i 的 d 维嵌入向量。给一个 token ID i,嵌入就是取出第 i 行 E[i]。

这看起来只是数组索引,但**它在数学上严格等价于一次矩阵乘法**,理解这点对后面(尤其 weight tying)很重要。把 token i 表示成一个 **one-hot 向量** eᵢ ∈ ℝ^V(第 i 位是 1,其余是 0),则:

```
eᵢᵀ · E = [0,...,0,1,0,...,0] · E  =  E 的第 i 行  =  E[i]
            (第 i 位为 1)

逐元素看:(eᵢᵀ E)[k] = Σⱼ eᵢ[j] · E[j,k] = E[i,k]   (只有 j=i 项非零)
```

所以**嵌入查表 = one-hot 编码 × 嵌入矩阵**。实现上当然不会真去做这个矩阵乘(那是 V 维 one-hot 乘 V×d,巨大且全是乘 0 的浪费),而是直接索引 `E[i]`——但**梯度的传播完全遵循矩阵乘的规则**:前向只用到了第 i 行,所以反向时也只有第 i 行收到梯度。这解释了上一节说的"长尾 token 嵌入欠训练":**一个 token 的嵌入向量,只有当它在 batch 里出现时才被更新**,出现少就更新少。

整段文本(序列长 n)的嵌入,就是把 n 个 ID 各查一行,堆成一个 n × d 的矩阵,这就是送进第一层 Transformer 的输入(实际还要加位置编码,见第 04 章)。

### 代码:嵌入查表的 PyTorch 片段

```python
import torch
import torch.nn as nn

class TokenEmbedding(nn.Module):
    def __init__(self, vocab_size, d_model):
        super().__init__()
        # 本质就是一个 V×d 的可学习矩阵;nn.Embedding 做的是高效的行索引
        self.embed = nn.Embedding(vocab_size, d_model)

    def forward(self, token_ids):           # token_ids: (batch, seq_len) 的整数张量
        return self.embed(token_ids)        # 返回 (batch, seq_len, d_model)

# 验证"查表 == one-hot 乘矩阵":
V, d = 10, 4
emb = nn.Embedding(V, d)
ids = torch.tensor([3, 7])

direct = emb(ids)                                   # 直接查表
onehot = torch.nn.functional.one_hot(ids, V).float()
matmul = onehot @ emb.weight                        # one-hot 乘嵌入矩阵
assert torch.allclose(direct, matmul)               # 两者完全相等
```

那行 `assert` 不是装饰——它就是这一节的全部论点:**嵌入层不是什么神秘的"语义字典",它就是一个矩阵,查表是它在 one-hot 输入下的高效特例**。

### 嵌入向量初始化与数值范围

嵌入通常用小的随机值初始化(如均值 0、标准差 ~0.02 的正态分布,「待核」具体模型默认值)。为什么不大?因为初始嵌入要和后面的归一化、注意力尺度匹配,太大的初始激活会让训练早期不稳(第 05、07 章会从激活方差和梯度角度细讲)。**一个常见坑**:有的实现会把嵌入输出乘以 √d 再送进网络(原始 Transformer 论文这么做),目的是让嵌入的尺度和位置编码可比;用不用这个缩放、初始化标准差设多少,要和归一化方案配套,不能随便抄。

## 七、嵌入矩阵的参数量:小模型的大头

现在做 spec 要求的核心推导。这是你以后估算任何模型参数构成时反复要用的。

### 推导:嵌入参数量 = V × d

嵌入矩阵 E 是 V × d,所以**输入嵌入的参数量就是 V × d**。

Transformer 的另一端是**输出投影(也叫 LM head / unembedding)**:最后一层输出的每个位置是一个 d 维向量 h,要变成 V 维的 logits(每个 token 一个分数),再过 softmax 得到 `P(xₜ | x_<t)`(回扣上一章)。这个投影是一个 d × V 的矩阵 W_out(以及可选的偏置),把 d 维投到 V 维:

```
logits = h · W_out            h: (..., d),  W_out: (d, V),  logits: (..., V)
P = softmax(logits)
```

W_out 的参数量也是 **d × V**(偏置 V 个,通常省略或忽略不计)。

**所以光是"进口"和"出口"两个矩阵,就有 2 × V × d 个参数。** 我们来感受一下它在总参数里的占比。Transformer 主体(每层的注意力 + FFN)的参数量大约是 `层数 × (大约 12 d²)`(注意力的 Q/K/V/O 约 4d²,FFN 中间放大 4 倍约 8d²,合计 ~12d²;精确推导在第 03 章,这里用数量级)。设层数 L,则:

```
嵌入(双份)≈ 2 V d
主体        ≈ 12 L d²
嵌入占比    ≈ 2Vd / (2Vd + 12 L d²) = V / (V + 6 L d)
```

代入数感受(取 V=5万,这些 L/d 是示例不是真实模型配置):

```
小模型  d=512,  L=6:   6Ld = 6·6·512    = 18432;   占比 ≈ 50000/(50000+18432) ≈ 73%  ← 嵌入主导!
中模型  d=1024, L=24:  6Ld = 6·24·1024  ≈ 147456;  占比 ≈ 50000/197456        ≈ 25%
大模型  d=4096, L=32:  6Ld = 6·32·4096  ≈ 786432;  占比 ≈ 50000/836432        ≈ 6%
大模型  d=8192, L=80:  6Ld = 6·80·8192  ≈ 3.9M;    占比 ≈ 50000/3.98M         ≈ 1.3%
```

**结论(重要):嵌入参数量绝对值固定为 ~2Vd,但占比随模型主体(∝ L·d²)增大而急剧下降。** 在小模型里嵌入可能占大半参数,在百亿级大模型里几乎可以忽略。这直接解释了上一节的词表权衡:**小模型对大词表很敏感(嵌入太重),大模型则可以放心用大词表换序列长度**——大模型用 10 万+词表很常见,正是因为那点嵌入参数在总盘子里不值一提,而省下的序列长度很值钱。

## 八、Weight Tying:输入输出共享一个矩阵

### 机制:为什么能 tie,为什么该 tie

注意上一节两个矩阵:输入嵌入 E 是 V × d,输出投影 W_out 是 d × V。它们形状互为转置。**weight tying(权重绑定)** 就是让它们共享同一份参数:

```
W_out = Eᵀ           （即 logits = h · Eᵀ）
```

**省了多少?** 原本要存 E(V×d)和 W_out(d×V)两份,共 2Vd;tie 之后只存一份 E,省掉 **V × d** 个参数。结合上一节,在小模型里这能省掉总参数的可观比例(那个例子里嵌入占 73%,tie 掉一半就是省 ~36% 总参数)。

**为什么不只是省参数,还常常更好?** 有几个层面的论证:

1. **对称性论证(语义一致)。** 输入嵌入做的事:把 token i 映射到向量 E[i]。输出投影做的事:给定隐状态 h,算它和每个 token 的相似度 `h · E[i]`(点积),相似度高的 token logit 高、概率大。**这两件事用的"token 的向量表示"理应是同一个**——一个 token 在"作为输入"和"作为预测目标"时,它的语义向量没理由不同。tie 把这个先验直接编码进结构,等于免费的归纳偏置(inductive bias)。

2. **正则化论证。** 共享参数减少了自由度,降低过拟合风险,尤其在数据相对参数不富裕时。

3. **梯度论证(更新更充分)。** 不 tie 时,token i 的输入嵌入 E[i] 只在 i 作为**输入**出现时收到梯度,输出向量 W_out[:,i] 只在 i 作为**目标**出现时收到梯度。tie 之后,**两条梯度路径汇聚到同一个向量**——token i 无论作为输入还是作为预测目标出现,都在更新它唯一的表示。这对长尾 token 尤其有利:更新机会翻倍,缓解欠训练。

历史上 Press & Wolf(2017)和 Inan 等(2017)系统论证了 tying 的好处,此后成为标准做法。**现代大模型绝大多数 tie 输入输出嵌入**(也有不 tie 的,通常是超大模型——那点参数无所谓,且解耦可能给一点点表达力,「待核」具体哪些大模型选择不 tie)。

### 代码:weight tying 的实现

```python
class TinyLM(nn.Module):
    def __init__(self, vocab_size, d_model):
        super().__init__()
        self.embed = nn.Embedding(vocab_size, d_model)
        # ... 中间是 N 层 Transformer block(第 03 章)...
        self.lm_head = nn.Linear(d_model, vocab_size, bias=False)
        # 关键一行:把输出投影的权重指向嵌入矩阵(共享同一块内存)
        self.lm_head.weight = self.embed.weight        # tie!

    def forward(self, token_ids):
        x = self.embed(token_ids)        # (B, T, d):查表
        # x = transformer_blocks(x)      # (B, T, d):省略
        logits = self.lm_head(x)         # (B, T, V):x · Eᵀ,因为权重是同一块
        return logits
```

`nn.Linear(d, V)` 的权重形状恰好是 (V, d),和 `nn.Embedding(V, d)` 的权重形状一致,所以可以直接令它们是同一个张量。前向时 `Linear` 算的是 `x @ weight.T = x @ E.T`,正是 `h · Eᵀ`。**这一行 `self.lm_head.weight = self.embed.weight` 就是 weight tying 的全部实现**,机制同 nanoGPT(其实际变量名见导读)。

**一个坑:** tie 之后,初始化和学习率要当心。共享矩阵既要当"嵌入"(尺度宜小)又要当"输出投影"(直接进 softmax,尺度影响 logit 大小),两边的需求略有张力;有的实现会在 logits 上额外乘/除一个缩放因子来调和。盲目 tie 而不管尺度,可能让训练早期 logit 过大、softmax 饱和(具体数值与归一化、初始化的配合见第 05、07 章)。

## 九、嵌入空间的几何:语义为什么变成了距离

最后讲清楚一个常被当作"魔法"的现象:**训练后,语义相似的 token,它们的嵌入向量在 d 维空间里靠得近;不相关的 token 离得远。** 这不是设计出来的,而是训练目标**逼**出来的。

### 机制:相似性从何而来

回看输出投影:下一个 token 的概率 `P(t | h) ∝ exp(h · E[t])`。模型要把概率给对,就得让"在相似上下文里该出现的 token"和那个上下文的隐状态 h 有相似的点积。两个 token,如果它们能出现在高度相似的上下文里(比如 "dog" 和 "cat" 都常跟在 "the" 后、都常做 "feed the ___" 的宾语),那么为了让它们在这些上下文下都拿到合理概率,**它们的嵌入向量必须和相似的 h 都有大点积——这逼着这两个嵌入向量本身变得相似(方向接近)**。这就是著名的分布式语义假设("意义由上下文决定")在嵌入里的体现,也是 word2vec 时代就被验证的现象在 Transformer 里的延续。

**用点积/夹角度量相似度。** 两个嵌入的相似度常用**余弦相似度**(夹角的余弦):

```
cos(E[i], E[j]) = (E[i] · E[j]) / (‖E[i]‖ · ‖E[j]‖)   ∈ [−1, 1]
```

值越接近 1,方向越一致,语义越近。**为什么常用余弦而非欧氏距离?** 因为输出投影里起作用的是**点积** `h·E[t]`,而点积 = ‖h‖·‖E[t]‖·cos;在比较"哪个 token 更可能"时,各 token 嵌入的模长差异会引入偏置(模长大的 token 天然 logit 大),用余弦剥离模长、只看方向,更贴近"语义"这个概念。

### 几何性质与一个著名误解

经典发现(word2vec,Mikolov 2013):嵌入空间里存在**线性结构**,著名的 `king − man + woman ≈ queen`——某些语义关系近似对应空间里平行的位移向量。这说明嵌入不是随机散布,而是组织出了有意义的几何结构。

**但要诚实地泼两盆冷水(这是专家和外行的区别):**

1. **类比关系只是"近似"且脆弱。** `king−man+woman≈queen` 在挑选过的例子上成立,但对很多关系并不成立,且高度依赖词表、训练数据、是否归一化。把它当成嵌入空间的普遍规律是过度解读。
2. **维度不可解释 + 叠加(superposition)。** 你不能指着第 37 维说"这是性别维度"。神经网络会用**远少于概念数的维度,通过方向的线性组合,叠加表示海量概念**(叠加现象),导致单个维度、单个神经元通常对应多个纠缠的概念。这正是机理可解释性(第 15 章)要用稀疏自编码器去"解叠加"的原因。**所以"嵌入每一维有独立语义"是错的,正确的图景是"语义编码在方向和子空间里,且高度叠加"。**

理解到这一层,你才算真懂嵌入:**它把离散符号嵌进连续空间,让"语义相似"变成"几何接近",从而后面所有层都能用连续的、可微的向量运算(注意力的点积、FFN 的线性变换)来处理语言**——这是整个深度学习能处理语言的物理前提。

## 设计权衡与常见坑

把全章的权衡和坑集中收口:

- **分词器一旦定死就是模型的一部分,几乎不可改。** 改词表 = 嵌入矩阵尺寸变 = 等于换模型重训。所以词表设计要在预训练**前**想清楚目标语言/领域分布。这是"第 0 步决定上限"的实操含义。
- **词表大小是三方拔河**(序列长度 ↓ vs 嵌入参数 ↑ vs 长尾欠训练)。**小模型怕嵌入太重(用小词表),大模型放心用大词表换短序列。** 别无脑抄大模型的词表配置到小模型上。
- **数字/缩进/语言覆盖的隐性能力损伤。** 数字不按单数字切会伤算术;空白处理不当伤代码;英文训的分词器直接套中文会让中文 token 数暴涨。这些在 loss 曲线上看不出来,但在下游能力上是硬伤。**评估一个分词器,要看它在你的目标语料上的"压缩率"(平均每字符多少 token)和关键模式(数字、缩进)的切法。**
- **weight tying 几乎总是对的(尤其中小模型),但要管尺度。** tie 后共享矩阵的初始化/缩放要和归一化方案配套,否则早期 softmax 饱和。
- **别把嵌入维度可解释当真。** 叠加现象让"一维一语义"不成立;调试时不要试图人肉读单个维度的含义。
- **一个隐蔽 bug:训练和推理用了不一致的分词器/特殊 token。** 比如加没加 BOS/EOS、空格归一化规则不同,会让推理分布和训练分布错位,模型表现莫名变差。务必让分词器在训练/推理端字节级一致。

## 小结与承上启下

这一章把"文本 → 整数 → 向量"这条入口管线讲透了:

- **分词**:字符级序列太长(注意力 `O(n²)` 受不了),词级词表爆炸又有 OOV;**子词是被 Zipf 分布逼出来的折中**。BPE 数频次自底向上合并、WordPiece 看互信息/似然增益合并、Unigram 用显式概率模型从大词表自顶向下剪枝并支持采样正则化——记住"往上拼 vs 往下砍"和各自判据。
- **词表大小**是序列长度、嵌入参数、长尾学习的三方拔河,小模型怕嵌入重、大模型放心换序列;中文和代码、数字的分词会悄悄决定能力上限。
- **嵌入**就是 V×d 的矩阵,查表 = one-hot 乘矩阵,反向只更新被用到的行;**参数量 2Vd 在小模型是大头、在大模型可忽略**。
- **weight tying** 让输入嵌入和输出投影共享一份矩阵,省 V×d 参数,且因语义对称、正则化、梯度路径合并而常常更好——一行代码的事,几乎总该做。
- **嵌入空间的几何**:训练把"语义相似"逼成"方向接近",这是后面一切连续向量运算处理语言的物理前提;但维度不可解释、类比关系脆弱、存在叠加,别神化。

**承上**:第 00 章告诉你模型在最小化 `−Σ log P(xₜ|x_<t)`,而那个 `P` 的出口正是本章的输出投影 `softmax(h·Eᵀ)`,入口正是本章的嵌入——这一章补上了那个被跳过的"token 到底是什么"。

**启下**:现在我们手里有了一串 n×d 的输入向量。但**注意力对输入顺序是置换不变的**(打乱 token 顺序它察觉不到,第 04 章会证明),且**向量之间还没有任何交互**。下一章(第 02 章)就要回答 Transformer 的核心问题:这些向量之间怎么"互相看"、按需检索信息——也就是**注意力机制**,我们会从信息瓶颈一路推到 `softmax(QKᵀ/√dₖ)V`,并搞清楚那个 √dₖ 到底是怎么来的。

### 练习 1:手跑 BPE 合并

语料按字符拆分后的初始状态(不加 `</w>`):

```
a a a b : 3
a a b   : 2
a b     : 5
初始词表 V = {a, b}
```

#### (a) 前 3 次合并(手算)

**第 1 步——统计相邻对(用词频加权)**

```
(a,a): "aaab" 有 2 个相邻 aa(位置 0-1、1-2)→ 2×3 = 6
       "aab"  有 1 个 aa → 1×2 = 2
       "ab"   无 → 0
       合计 = 8
(a,b): "aaab" 末尾 1 个 ab → 1×3 = 3
       "aab"  末尾 1 个 ab → 1×2 = 2
       "ab"   1 个 ab → 1×5 = 5
       合计 = 10   ← 最高
```

最高频对是 `(a,b)`,合计 **10**。合并规则 #1:**(a, b) → ab**

```
语料更新:
  a a ab : 3
  a ab   : 2
  ab     : 5
```

**第 2 步——重新统计**

```
(a,a):  "a a ab" 有 1 个 aa → 1×3 = 3
(a,ab): "a a ab" 末尾 1 个 (a,ab) → 1×3 = 3
        "a ab"   1 个 (a,ab) → 1×2 = 2
        合计 = 5   ← 最高
```

最高频对是 `(a,ab)`,合计 **5**。合并规则 #2:**(a, ab) → aab**

```
语料更新:
  a aab : 3
  aab   : 2
  ab    : 5
```

**第 3 步——重新统计**

```
(a,aab): "a aab" 1 个 → 1×3 = 3   ← 唯一相邻对,也是最高
```

合并规则 #3:**(a, aab) → aaab**

```
语料更新:
  aaab : 3
  aab  : 2
  ab   : 5
```

**前 3 条合并规则汇总:**

```
#1  (a, b)   -> ab
#2  (a, ab)  -> aab
#3  (a, aab) -> aaab
```

注意这里没有出现 tie(每步最高频对都唯一),所以不依赖 tie-break 规则。这条语料偏巧让 BPE 从右往左把 `b`、`a`、`a` 依次粘到一起,最终把整个 `aaab` 合成了单个 token。

#### (b) 用 `train_bpe` 验证

直接复用本章代码,只把初始 `word_freqs` 设成字符元组:

```python
from collections import Counter

def get_pair_counts(word_freqs):
    pairs = Counter()
    for word, freq in word_freqs.items():
        for i in range(len(word) - 1):
            pairs[(word[i], word[i + 1])] += freq
    return pairs

def merge_pair(pair, word_freqs):
    a, b = pair
    merged = a + b
    new_word_freqs = {}
    for word, freq in word_freqs.items():
        new_word, i = [], 0
        while i < len(word):
            if i < len(word) - 1 and word[i] == a and word[i + 1] == b:
                new_word.append(merged); i += 2
            else:
                new_word.append(word[i]); i += 1
        new_word_freqs[tuple(new_word)] = freq
    return new_word_freqs

def train_bpe(word_freqs, num_merges):
    merges = []
    for _ in range(num_merges):
        pairs = get_pair_counts(word_freqs)
        if not pairs:
            break
        best = max(pairs, key=pairs.get)   # 贪心取最高频对
        word_freqs = merge_pair(best, word_freqs)
        merges.append(best)
    return merges

corpus = {"aaab": 3, "aab": 2, "ab": 5}
word_freqs = {tuple(w): f for w, f in corpus.items()}   # 按字符拆
merges = train_bpe(word_freqs, 3)
print(merges)
# 输出: [('a', 'b'), ('a', 'ab'), ('a', 'aab')]
```

**实际运行输出 `[('a', 'b'), ('a', 'ab'), ('a', 'aab')]`,与手算完全一致。**(已实测)

#### (c) 用合并规则编码 `"aaaab"`

按规则顺序对 `a a a a b` 反复应用:

```
初始:        a a a a b
#1 (a,b)->ab:    a a a ab          （只有末尾的 a,b 相邻,合并一次)
#2 (a,ab)->aab:  a a aab           （末尾 a 与 ab 相邻,合并)
#3 (a,aab)->aaab: a aaab           （末尾 a 与 aab 相邻,合并)
没有更多可应用的规则
```

**最终 token 序列:`["a", "aaab"]`(2 个 token)。**(已用代码验证一致)

`"aaaab"` 没在训练语料里出现过,但 BPE 用一个剩下的单字符 `a` 加上训练里学到的 `aaab`,把它拼了出来——这就是子词分词消灭 OOV 的机制。

### 练习 2:嵌入参数量与占比

记号:V=32000,d=2048,L=16,主体每层 ≈ 12d²,主体总量 = 12Ld²。

先算公共量:

```
Vd      = 32000 × 2048      = 65,536,000      ≈ 65.54 M
12Ld²   = 12 × 16 × 2048²   = 805,306,368     ≈ 805.31 M
```

#### (a) tie(只一份 V×d)

```
嵌入参数  = Vd                 ≈ 65.54 M
主体参数  = 12Ld²              ≈ 805.31 M
总参数    = Vd + 12Ld²         ≈ 870.84 M
嵌入占比  = Vd / (Vd + 12Ld²)
         = 65.54 / 870.84     ≈ 0.0753
```

**tie 时:嵌入 ≈ 65.54 M,主体 ≈ 805.31 M,总计 ≈ 870.84 M,嵌入占比 ≈ 7.53%。**

(等价化简:`Vd/(Vd+12Ld²) = V/(V+12Ld)`,即 `32000/(32000+12·16·2048)=32000/425216≈7.53%`,一致。)

#### (b) 不 tie(双份 2×V×d),以及 tie 省多少

```
嵌入参数(双份) = 2Vd          ≈ 131.07 M
总参数          = 2Vd + 12Ld²  ≈ 936.38 M
嵌入占比        = 2Vd/(2Vd+12Ld²)
               = 131.07/936.38 ≈ 0.1400
```

**不 tie 时嵌入占比 ≈ 14.0%**(化简形式 `V/(V+6Ld)`,即本章正文 line 410 给的就是这个**不 tie** 的占比公式)。

tie 相对不 tie 省掉的就是多出来的那一份输出投影 = **V×d ≈ 65.54 M** 参数。换算成百分比要说清以谁为分母:

```
省下的参数 = Vd ≈ 65.54 M
占「不 tie 总参数」的比例 = 65.54 / 936.38 ≈ 7.0%
占「tie 后总参数」的比例   = 65.54 / 870.84 ≈ 7.5%
```

**tie 省掉 V×d ≈ 65.54 M 个参数,约为不 tie 总参数的 7.0%(或 tie 后总参数的 7.5%)。**

> 提醒:本题 d、L 较大,嵌入本就只占一成左右,所以 tie 省的绝对占比不大。这正印证本章「大模型嵌入占比小」的结论;若换成正文 line 416 那个 d=512/L=6 的小模型,嵌入占比高达 ~73%,tie 一掉就省掉约 36% 总参数,收益才显著。

#### (c) V 从 32000 提到 128000(tie,其他不变)

主体 12Ld² 不变(≈805.31 M);只有嵌入随 V 线性涨 4 倍:

```
Vd(V=128k) = 128000 × 2048    = 262,144,000   ≈ 262.14 M
总参数      = 262.14 + 805.31  ≈ 1067.45 M
嵌入占比    = 262.14 / 1067.45 ≈ 0.2456
```

**V 提到 128000 后,嵌入占比从 7.53% 升到 ≈ 24.6%**(化简 `V/(V+12Ld)=128000/(128000+393216)≈24.6%`,一致)。

**是否划算的讨论(结论:对 d=2048/L=16 这个中小模型,需谨慎,不能无脑加):**

- **代价侧(变重):** 词表翻 4 倍,嵌入从占 ~7.5% 涨到 ~24.6%,多吃约 196 M 参数,且显存/IO 都涨。对中小模型这是肉眼可见的负担。
- **更隐蔽的代价:** V 越大,长尾 token 平均频次越低,梯度更新次数越少,**长尾嵌入欠训练**风险上升——很多新增 token 可能学不充分,白占参数。
- **收益侧:** 大词表能缩短序列、提高多语言(尤其中文/代码)的单 token 覆盖、降低注意力 `O(n²)` 开销和按 token 计费的成本。
- **结论:** 是否值得取决于目标语料。如果这是一个真要服务多语言/中文/代码的模型,序列长度和压缩率的收益可能值这 ~17 个百分点的嵌入膨胀(尤其 tie 已经省了一份);但如果主要是英文、追求小而省,128k 词表对这个规模偏奢侈,更稳妥是先用 32k~64k,把省下的参数留给主体。这正是本章「小模型对大词表敏感、大模型才放心用大词表换序列」的实操含义——d=2048/L=16 还处在「嵌入占比对 V 敏感」的区间,不能照抄百亿级大模型的 128k 配置。

### 练习 3:中文 / 代码的分词代价

#### (a) 英文训练的字节级 BPE 给中文分词,为什么 token 数显著更多

从「UTF-8 字节数」和「合并规则覆盖」两点解释:

1. **UTF-8 字节数(物理下界变高):** 字节级 BPE 的原子单位是字节(0–255)。一个 ASCII 英文字符占 **1 字节**,而一个常见汉字在 UTF-8 里占 **3 字节**。也就是说,中文文本在「还没合并」的起点上,字节数就已经是同字符数英文的约 3 倍——分词的物理下界天然更高。
2. **合并规则覆盖(头部红利吃不到):** BPE 的「序列短」靠的是把高频单元合并成单 token。一个主要在英文上训练的分词器,它的合并规则几乎全花在英文字节序列上,**几乎没有把中文的 3 字节序列合并成「单字 token」或「词组 token」的规则**。结果是中文的字节基本停留在未合并(或只浅合并)状态:一个汉字常被切成 2~3 个字节 token,甚至切在多字节字符的中间,产生无语义的字节碎片。

两者叠加:**物理上中文字节多 3 倍,机制上又享受不到合并红利**,于是同等信息量的中文 token 数可能是英文的数倍。后果是上下文窗口被吃掉、推理变贵、按 token 计费时中文用户吃亏。对策(回扣第五节):在训练语料里加入足量中文,让常见汉字/词组各自拿到单 token。

#### (b) 为什么「数字按单个数字切分」有助于算术,用 "199+1" 说明

**核心:算术(尤其进位)依赖于「位」对齐的稳定表示。** 如果把数字切成单个数字(digit),那么任意数字的 token 序列都是规整的「逐位」表示,同一个数位永远是同一个 token,进位规律可以稳定泛化:

```
单数字切分:
  199 -> ["1","9","9"]
  +1  -> ["+","1"]
  200 -> ["2","0","0"]
模型看到的是规整的逐位结构,"个位 9+1 进位、十位 9+1 进位、百位 1+1"
这种进位模式在所有数字上形式一致,容易学到且能外推到没见过的数。
```

而非单数字切分(按高频子串贪心合并)会破坏这种结构。假设分词器把高频的 "199" 合成了一个 token、把 "200" 也合成一个 token:

```
非单数字切分(示意):
  199 -> ["199"]      # 整块,看不到内部的个/十/百位
  +1  -> ["+","1"]
  200 -> ["200"]      # 又是另一个完全无关的整块 token
```

此时模型要学的不是「逐位进位规律」,而是去**死记** `"199" + "1" = "200"` 这一条具体映射;而 `"299"+"1"`、`"1099"+"1"` 的切法可能完全不同(`["29","9"]`、`["10","99"]`……),进位发生的边界落在了不同的 token 切割上,**同一个进位动作在不同数字上对应到不同的 token 模式**,模型无法把「9 满十进一」这条规律迁移过去。于是算术能力被分词切法直接卡死。这正说明分词不是无害预处理——它决定了模型能力的上限。

#### (c) 中英 + 代码混合模型的 3 条词表设计建议

| # | 建议 | 具体做法 | 缓解哪个坑 |
|---|------|----------|-----------|
| 1 | **语料配比要覆盖三种分布,按目标流量加权** | 训练分词器的语料里同时放入足量中文、英文、代码;按预期使用占比(而非自然语料占比)加权,保证常见汉字、常见词组、常见编程关键字/标识符片段各自拿到**单 token**;用目标语料上的**压缩率**(平均每字符多少 token)做验收指标 | 缓解「英文训的分词器套中文 → 中文 token 数暴涨」:让中文吃到合并红利,压缩率回到合理区间 |
| 2 | **数字强制按单个数字(digit)切分** | 在 pre-tokenization 阶段把连续数字逐位拆开,每个数字一个 token,禁止把数字串合并成块 | 缓解「数字切法破坏进位结构 → 算术能力受损」:逐位表示让进位规律可稳定学习与外推 |
| 3 | **空白/缩进做专门处理** | 把常见缩进量(如 4 空格、8 空格、制表符)做成**单 token**,统一空格归一化规则,保证缩进切法一致、不被切碎 | 缓解「空白处理不当 → 伤代码」:Python 等靠缩进的语言不会因缩进被切碎而丢结构,也省 token |

(三条都回扣第五节的「压缩率 / 单 token 覆盖常见汉字 / digit 切分 / 缩进 token」关键词。)

### 练习 4:实现 Unigram 维特比编码

为保证 `"aba"` 可被切分(无 OOV,需要所有单字符都在词表里),取词表 log 概率为(本章给的 `a/b/ab/aba` 已足够覆盖 `"aba"`):

```
{"a": -1.0, "b": -1.5, "ab": -1.8, "aba": -3.0}
```

维特比实现(CPU 纯 Python,无依赖,可直接跑):

```python
import math

def viterbi_segment(text, logp):
    """Unigram LM 维特比:找对数概率最大的切分。
    logp: dict 子词 -> log 概率(均为负值)。"""
    n = len(text)
    best = [-math.inf] * (n + 1)   # best[i]: 切到第 i 个字符为止的最大对数概率
    back = [None] * (n + 1)        # back[i]: 该最优切分中,最后一个子词的起点 j
    best[0] = 0.0
    maxlen = max(len(w) for w in logp)   # 用最长子词长度限制 j 的范围 -> O(n·L)
    for i in range(1, n + 1):
        for j in range(max(0, i - maxlen), i):
            sub = text[j:i]
            if sub in logp and best[j] > -math.inf:   # 只在 c[j:i] 在词表里时转移
                score = best[j] + logp[sub]
                if score > best[i]:
                    best[i] = score
                    back[i] = j
    if best[n] == -math.inf:
        return None, None          # 不可切分(本题不会发生)
    # 回溯
    seg, i = [], n
    while i > 0:
        j = back[i]
        seg.append(text[j:i])
        i = j
    seg.reverse()
    return seg, best[n]

logp = {"a": -1.0, "b": -1.5, "ab": -1.8, "aba": -3.0}
seg, score = viterbi_segment("aba", logp)
print(seg, score)        # 输出: ['ab', 'a']  -2.8
```

**运行结果:最优切分 = `["ab", "a"]`,对数概率 = −2.8。**(已实测)

**验算——枚举 `"aba"` 的全部合法切分:**

```
["a","b","a"]  : -1.0 + -1.5 + -1.0 = -3.5
["ab","a"]     : -1.8 + -1.0       = -2.8   ← 最大(维特比选中)
["aba"]        : -3.0
```

−2.8 确实是三者中最大的对数概率,维特比正确。

**对比贪心最长匹配(BPE/WordPiece 风格):**

贪心从词首取最长能匹配的子词:`"aba"` 整个在词表里,最长匹配直接吃下 `"aba"`,得到 `["aba"]`,对数概率 = **−3.0**。

```
贪心最长匹配: ["aba"]    => -3.0
维特比(全局): ["ab","a"] => -2.8   （更优）
```

**结论:** 贪心最长匹配被「aba 整块更长」误导,选了概率更低的 `["aba"]`(−3.0);维特比在所有切分上做全局最优,选出 `["ab","a"]`(−2.8)。这正体现两类算法的本质差别——**Unigram 用维特比找全局最优切分,而 BPE/WordPiece 是贪心(按合并顺序或最长匹配),不保证全局最优**。也正因为 Unigram 对每种切分都有显式概率,它才能做子词正则化(按概率采样不同切分)。
