> 这章把你从"知道训练要 forward-backward-step 三步"带到能拿出纸笔,对任意规模的 Transformer,算清一步迭代花多少 FLOPs、占多少字节显存,并据此判断"这模型该上哪种并行 / 省显存技术"。后面所有并行(数据 / 张量 / 流水线)和省显存(ZeRO、重计算、混合精度)技术,本质都是在改写这本账上的某几个数字。

## 0. 为什么先记账

第 0 章我们在物理层把"为什么慢"讲清了:算力(FLOPS)和带宽(GB/s)是两条独立的天花板,Roofline 决定一个 kernel 是 compute-bound 还是 memory-bound。但那是**单个 kernel**的视角。训练一个大模型,你面对的第一个现实问题不是某个 kernel 快不快,而是两个更粗暴的问题:

1. **放不放得下**:这模型连同它的优化器状态、激活,能不能塞进我这张卡的 80GB HBM?塞不下,差多少?
2. **跑多久**:在已知 GPU 峰值算力下,处理完 D 个 token 大概要多少卡·天?

这两个问题的答案,**几乎完全由两个数字决定:参数量 N 和训练 token 数 D**。而把 N、D 翻译成"FLOPs"和"字节"的那本账,就是这一章。

记这本账有个反直觉的价值:**它精确到可以指导架构决策**。当你算出 7B 模型用 Adam 训练需要约 112GB 状态显存、单张 80GB 卡放不下时,你不是去"试试看 OOM 不 OOM",而是立刻知道——要么分片优化器状态(ZeRO-1,第 5 章),要么切模型(张量 / 流水线并行,第 3、4 章)。账本是决策的输入,不是事后的解释。

本章约定:为了让账目干净,主体推导聚焦标准 dense Transformer(GPT 系)。我们先把"理想账本"建立起来(只数矩阵乘),再逐项把现实修正(注意力、归一化、激活函数、嵌入层)加回去,让你既有 60 秒口算的近似,也有能对上 profiler 的精确版。

## 1. 一步迭代的三个阶段:先把骨架立起来

一次训练迭代(一个 mini-batch 走完)严格分三阶段,顺序不可换:

```
阶段              输入                  输出                      显存生命周期
─────────────────────────────────────────────────────────────────────────
1. Forward    输入 x, 权重 W      loss(标量) + 每层激活        激活必须留到反向用完
2. Backward   loss, 激活, W       每个参数的梯度 g = ∂L/∂W     梯度留到 step 用完
3. Optimizer  g, 旧参数, 动量      新参数 W', 新动量            原地更新,瞬时
```

三阶段的算力和显存特征完全不同,必须分开记:

- **Forward** 算力 ≈ 1 个单位,但它**生产**了大量激活(activation),这些激活是反向的输入,**不能丢**,构成显存的一大头。
- **Backward** 算力 ≈ 2 个单位(下一节证明为什么是 2 倍),它**消耗**激活、**生产**梯度。
- **Optimizer step** 算力极小(每参数十几个标量运算,与 N 线性、与 batch 无关),但它**持有**最大的一块常驻显存——优化器状态(Adam 的 m、v 加 fp32 主权重)。

**关键认知:算力大头在 backward,常驻显存大头在 optimizer state,瞬时显存大头在 activation。** 三个"大头"分属三个阶段,这就是为什么省显存技术和省算力技术是不同的技术族——它们在攻击账本的不同行。

下面分两条主线把账记死:**算力账(FLOPs)** 和 **显存账(bytes)**。

## 2. 算力账(一):为什么一次矩阵乘是 2·m·n·k FLOPs

一切从最底层的原子操作开始:矩阵乘法(GEMM)。Transformer 99% 的算力花在 GEMM 上(注意力里的 QKᵀ、AV,以及所有 Linear 层)。

考虑 C = A · B,其中 A 是 m×k,B 是 k×n,C 是 m×n。输出矩阵有 m·n 个元素,每个元素是一个长度 k 的向量点积:

```
C[i][j] = ∑_{l=0}^{k-1} A[i][l] · B[l][j]
```

一个长度 k 的点积包含 **k 次乘法 + (k-1) 次加法 ≈ 2k 次浮点运算**(我们按惯例把"一次乘加 multiply-accumulate, MAC"算作 2 FLOPs,忽略那个 -1)。于是:

```
FLOPs(GEMM) = m·n · (2k) = 2·m·n·k
```

**记住这个 2·m·n·k,它是整本算力账的细胞。** 直觉是:每产出一个输出标量,要做 k 个乘加,每个乘加 2 FLOPs。

> 这里有个常见困惑:有人把乘和加分开数说是 2 FLOPs,有人说现代硬件 FMA(fused multiply-add)一条指令完成所以"算 1 个 op"。**FLOPs 数法和指令数法不是一回事**:论文统计 FLOPs 时,行业惯例统一把一次乘加记为 2 FLOPs(乘 1 + 加 1)。Tensor Core 一条 FMA 指令物理上算 2 FLOPs,所以用 2·m·n·k 去除以 GPU 的标称峰值 FLOPS(那个标称值也是按乘加=2 算出来的),两边的"2"约掉,得到的时间才对得上。**全程只要 2 这个系数前后一致,就不会错。**

### 把一个 Linear 层套进去

Transformer 里一个全连接层:输入 X 形状 (B, d_in)(B 是 batch×seq 展平后的 token 总数,即 tokens),权重 W 形状 (d_in, d_out),输出 (B, d_out)。这是一次 m=B, k=d_in, n=d_out 的 GEMM:

```
Forward FLOPs(Linear) = 2 · B · d_in · d_out
```

注意 **d_in · d_out 正好是这个 Linear 层的参数量**(忽略 bias)。所以可以改写成一个极其有用的形式:

```
Forward FLOPs(一个 Linear 层) ≈ 2 · (token 数) · (该层参数量)
```

**这就是 "6ND" 公式里那个 "2" 的来源**:对一个以 GEMM 为主的模型,前向的 FLOPs ≈ 2 × token 数 × 参数量。把全模型所有 Linear 的参数量加起来就是 N,于是前向 ≈ 2ND。下面要证的是反向为什么是这个的 2 倍。

---

## 3. 算力账(二):反向为什么 ≈ 2× 前向(完整推导)

这是本章第一个"必须讲透"的推导。结论:**反向传播的 FLOPs ≈ 2 倍前向**,因此一次前+反 ≈ 3 倍前向 = 6ND。

为什么是 2 倍?直觉答案:**前向每个 GEMM,在反向要算两个梯度——对输入的梯度和对权重的梯度——每个梯度各是一次和前向同规模的 GEMM。** 下面严格推。

### 设定

取一个 Linear 层(它是模型的代表性算子,整本账几乎全是这种层堆出来的):

```
Y = X · W
  X: (B, d_in)    输入激活
  W: (d_in, d_out) 权重
  Y: (B, d_out)   输出激活
```

前向 FLOPs = 2·B·d_in·d_out。

反向时,上游传来 ∂L/∂Y(记作 dY,形状和 Y 一样 (B, d_out))。我们要算两样东西:

**(a) 对权重的梯度 dW = ∂L/∂W**,用来更新这一层的参数。

**(b) 对输入的梯度 dX = ∂L/∂X**,作为再上一层(更靠近输入)的 dY 传下去。这是链式法则的"接力棒",不算它,梯度就传不到前面的层。

### 逐个求 FLOPs

链式法则。先看 (a)。对单个元素 W[p][q],它影响所有 token 的 Y[:, q]:

```
∂L/∂W[p][q] = ∑_{b} ∂L/∂Y[b][q] · ∂Y[b][q]/∂W[p][q]
            = ∑_{b} dY[b][q] · X[b][p]
```

把所有 (p,q) 写成矩阵形式:

```
dW = Xᵀ · dY        Xᵀ: (d_in, B),  dY: (B, d_out)  →  dW: (d_in, d_out) ✓
```

这是一次 m=d_in, k=B, n=d_out 的 GEMM:**FLOPs = 2·d_in·B·d_out = 2·B·d_in·d_out**,和前向一模一样。

再看 (b):

```
∂L/∂X[b][p] = ∑_{q} dY[b][q] · ∂Y[b][q]/∂X[b][p]
            = ∑_{q} dY[b][q] · W[p][q]
```

矩阵形式:

```
dX = dY · Wᵀ        dY: (B, d_out),  Wᵀ: (d_out, d_in)  →  dX: (B, d_in) ✓
```

这是一次 m=B, k=d_out, n=d_in 的 GEMM:**FLOPs = 2·B·d_out·d_in**,又和前向一样。

### 合账

```
前向:  Y  = X·W       →  2·B·d_in·d_out          (1 个 GEMM)
反向:  dW = Xᵀ·dY      →  2·B·d_in·d_out  ┐
       dX = dY·Wᵀ      →  2·B·d_in·d_out  ┘ 共 4·B·d_in·d_out  (2 个 GEMM)

反向 / 前向 = 4 / 2 = 2  ✓
一次前+反 = 2 + 4 = 6 倍单层"基准 GEMM"
```

**所以反向 ≈ 2× 前向,这不是经验数字,是矩阵微分的必然:每个前向 GEMM 在反向裂变成 dW、dX 两个同规模 GEMM。**

### 一个重要的"省一半"细节

最靠近输入的那一层(以及嵌入层),它的 dX 不需要继续往下传(下面没有可学习的层了),理论上可以省掉那个 dX 的 GEMM。但这只占全模型百层中的一层,**对总账的影响 < 1%**,所以 6ND 里我们不扣它。这告诉你一个估算心法:**抓主干,放过尾数。** 一个 96 层的模型,省掉 1 层的半个反向 = 省掉 0.5/192 ≈ 0.26%,不值得在估算里纠结。

## 4. 算力账(三):训练总算力 C ≈ 6ND 的完整推导

现在把单层结论推广到整个训练过程。

**定义**:N = 模型可训练参数量(主要是所有 Linear/attention 投影的权重),D = 整个训练见过的 token 总数(= 每步 token 数 × 步数,或 batch × seq × steps)。

**推导链**:

```
(1) 单个 Linear 层前向         ≈ 2 · tokens · (该层参数量)        [第 2 节]
(2) 整模型前向(所有层求和)    ≈ 2 · tokens · N                  [对所有层 ∑ 参数量 = N]
(3) 整模型反向 = 2× 前向       ≈ 4 · tokens · N                  [第 3 节]
(4) 一次迭代(前+反)          ≈ 6 · tokens · N
(5) 全训练 = 对所有 step 求和:tokens 累加 = D
    ─────────────────────────────────────────────
    C_train ≈ 6 · N · D    FLOPs
```

```
        ┌─ 前向 2ND ─┐
6ND  =  │            │ ND 为"一遍参数·token"的基准
        ├─ 反向 4ND ─┤   反向 = 2× 前向
        └────────────┘
```

**这就是著名的 C ≈ 6ND。** 它的威力在于:**算力只取决于 N 和 D,与模型形状(多深多宽、注意力怎么设计)几乎无关。** 一个 7B 模型训 1T token,无论你怎么排布层数和宽度,总算力都 ≈ 6 × 7e9 × 1e12 = 4.2e22 FLOPs。

### 三个必须知道的修正

6ND 是"只数 GEMM 主干"的近似。真实账本要加三笔,但三笔都不大:

1. **注意力的 QKᵀ 和 AV(score·value)**:这两个 GEMM 不含可学习参数(参数在 Q/K/V/O 的投影里,已计入 N),但要花算力,且**正比于 seq²**。每层每 token 的注意力额外 FLOPs ≈ `2 · 2 · seq · d_model`(QKᵀ 和 AV 各一次,系数细节见下表)。当 **seq ≪ d_model / (常数)** 时这部分相对小;当 seq 很长(几万)时,attention 的 seq² 项会反超 GEMM,6ND 开始低估。**长上下文场景必须单独算 attention,这也是第 9 章 FlashAttention 存在的理由。**

2. **非 GEMM 算子**:LayerNorm / RMSNorm、softmax、GELU/SwiGLU、残差加法、dropout。这些是逐元素或归约操作,FLOPs 相对 GEMM 小一两个数量级(但它们是 memory-bound,占的是带宽不是算力,见第 0 章 Roofline——它们拖慢的是墙钟时间,不是 FLOPs 账)。

3. **嵌入与输出投影(logits)**:输入嵌入是查表(几乎零 FLOPs);但最后的 vocab 投影 (d_model → vocab) 是个大 GEMM,FLOPs ≈ `2 · tokens · d_model · vocab`,vocab 常达 5万~15万,这一层不可忽略,且它的参数已经在 N 里。

**结论:对 d_model 量级的标准 LLM、中等序列长度,6ND 的误差通常在 ±10% 内**,足够做规模决策。要对上 profiler 的精确值,就把上面三笔加回去(下一节给出含 attention 的精确公式)。

### 精确版前向 FLOPs(每层每 token,供对账)

下表给出一个标准 Transformer decoder 层、单 token、前向的 FLOPs 拆解(B=token 数已提到层外,seq 为序列长,d = d_model,d_ff 为 FFN 中间维,n_h 头数,h_d 每头维度,d = n_h·h_d):

| 子模块 | 计算 | 每 token FLOPs(前向) |
|---|---|---|
| QKV 投影 | x·W_qkv,(d → 3d) | 2 · d · 3d = 6d² |
| QKᵀ(注意力分数) | (seq,h_d)·(h_d,seq) over heads | 2 · seq · d(全头合计) |
| softmax·V | (seq,seq)·(seq,h_d) | 2 · seq · d |
| 输出投影 O | (d → d) | 2 · d · d = 2d² |
| FFN 上行 | (d → d_ff) | 2 · d · d_ff |
| FFN 下行 | (d_ff → d) | 2 · d_ff · d |
| **合计/层/token** | | **8d² + 4d·d_ff + 4·seq·d** |

把"参数相关项"提出来:`8d²+4·d·d_ff = 2·(4d²+2·d·d_ff) = 2·(每层参数量)`,正好印证 2× 参数。**注意力的 `4·seq·d` 项不含参数,这就是 6ND 漏掉的那块,与 seq 成正比(每 token),对整序列即 seq² 增长。** d_ff 通常取 4d(或 SwiGLU 的 ≈ 8d/3 但分 3 个矩阵),代入 d_ff=4d 得每层 ≈ 24d² + 4·seq·d,前向乘 tokens、再乘 3(含反向)即得该层训练算力。

## 5. 算力账(四):把公式变成生产力

有了 C ≈ 6ND,你能直接回答工程问题。两个最常用的:

**(A) 训练要多久 / 多少卡**

```
墙钟时间 T ≈ C / (GPU 数 × 单卡峰值 FLOPS × MFU)
           = 6ND / (P · F_peak · MFU)
```

MFU(Model FLOPs Utilization,模型算力利用率)= **实际有效 FLOPs / 理论峰值 FLOPs**,刻画"你的 6ND 算力里有多少真的跑在硬件峰值上"。大规模训练 MFU 常在 **0.3~0.55**「待核:具体数值随硬件 / 框架差异大,这里给行业常见区间」。把 MFU 放进分母,是因为内存墙、通信、kernel 启动、流水线气泡都会让你拿不到峰值——**第 0 章的 Roofline 和后面所有并行章,本质都在抬高这个 MFU。**

例:N=70e9,D=1e12,P=1024 张某加速卡,单卡 BF16 峰值设为 F_peak(标称值),MFU=0.4:

```
C   = 6 · 70e9 · 1e12 = 4.2e23 FLOPs
T   = 4.2e23 / (1024 · F_peak · 0.4)
    = 4.2e23 / (410 · F_peak)   秒
```

代入你手上卡的真实 F_peak 即得秒数,除以 86400 得卡·天。**注意 6ND 的好处:你不需要知道模型几层、几头,只要 N 和 D。**

**(B) 反推:给定算力预算,N 和 D 怎么分?**
这就是 scaling law(Chinchilla)的入口——固定 C=6ND,在 N 和 D 之间分配以最小化 loss。本章只给账本,scaling law 的取舍属于后续 / 第 15 章范畴。这里点到为止:**C=6ND 是 scaling law 的预算约束方程。**

## 6. 显存账(一):常驻状态 —— "每参数约 16 字节"逐项拆解

现在转到第二条主线,也是新手最容易 OOM 翻车的地方。显存分两类,先讲**常驻状态**(model states:参数 + 梯度 + 优化器状态),它在整个训练期都占着,与 batch 无关、只与 N 有关。

标准配置:**混合精度训练 + Adam/AdamW 优化器**。这是当今 LLM 训练的事实标准(混合精度细节见第 7 章,这里只用结论:计算用 fp16/bf16,优化器主状态用 fp32)。逐项拆:

```
每个参数,常驻显存:
─────────────────────────────────────────────
fp16/bf16 参数副本(给前向/反向算 GEMM 用)     2 字节
fp16/bf16 梯度(反向产出,All-Reduce 通信用)    2 字节
─────────── 以上是"计算精度"那份 ───────────
fp32 参数主副本(master weight,优化器更新用)   4 字节
Adam 一阶动量 m(梯度的滑动平均)               4 字节
Adam 二阶动量 v(梯度平方的滑动平均)            4 字节
─────────── 以上是 fp32 优化器状态那份 ────────
合计                                          16 字节 / 参数
```

**为什么需要 fp32 主副本?** 这是混合精度的核心机制(第 7 章详证):fp16 只有约 10 位有效尾数,当 `参数值 ≫ 学习率×梯度` 时,`W ← W - lr·g` 这一步的更新量会因为浮点对阶被直接舍入吞掉(下溢到 0),参数学不动。解法:**保留一份 fp32 的权重(master copy)做累加更新,前向/反向再把它降精度成 fp16 用。** 所以一个参数同时存在 fp16 和 fp32 两份——这是 "2(fp16参数)+4(fp32主副本)" 看起来"重复存"的原因,它不是冗余,是数值正确性的刚需。bf16 因为指数位多、动态范围大,某些框架可不存 fp32 主副本,但仍普遍保留以稳更新「待核:是否省 fp32 主副本随框架配置而定」。

**Adam 为什么要 m 和 v 两份动量?** Adam 的更新规则:

```
m_t = β₁·m_{t-1} + (1-β₁)·g_t              一阶矩(动量,梯度方向的滑动平均)
v_t = β₂·v_{t-1} + (1-β₂)·g_t²             二阶矩(梯度幅度平方的滑动平均)
m̂  = m_t/(1-β₁ᵗ),  v̂ = v_t/(1-β₂ᵗ)        偏差修正
W_t = W_{t-1} - lr · m̂ / (√v̂ + ε)          自适应步长更新
```

m 提供动量(平滑梯度方向),v 提供每参数自适应学习率(梯度大的方向步子小)。**两者都是 per-parameter 的状态,各占 4 字节 fp32,这就是 Adam 比 SGD 贵的根本原因。** 用纯 SGD(无动量)则没有 m、v,常驻只剩 2+2+4 = 8 字节/参数;SGD+momentum 多一份 m = 12 字节。**优化器选择直接改写这一行账。**

**合账(Adam 混合精度,每参数 16 字节):**

```
模型状态显存 ≈ 16 · N  字节
```

| 模型 N | 16N 模型状态(单卡全量) |
|---|---|
| 1.5B | ≈ 24 GB |
| 7B | ≈ 112 GB |
| 13B | ≈ 208 GB |
| 70B | ≈ 1120 GB ≈ 1.12 TB |

**第一个震撼结论:7B 模型,光是模型状态就 112GB,一张 80GB 卡放不下——还没算激活!** 这就是为什么 7B 以上几乎必须分片(ZeRO,第 5 章)或并行(第 3、4 章)。**16N 是你判断"单卡够不够"的第一道闸门。**

> 常见坑:很多人记成 "7B 模型 14GB(=2×7B 的 fp16 推理显存)就行"——那是**推理**显存,只需要一份 fp16 参数。**训练**显存是推理的约 8 倍(16 字节 vs 2 字节),因为训练要扛梯度和优化器状态。把推理预算套到训练上,是最经典的 OOM 来源。

## 7. 显存账(二):激活显存 —— 与 batch、seq 强相关的那块

模型状态只和 N 有关,与你喂多大 batch 无关。但还有第二大块显存:**激活(activations)** —— 前向过程中产生、必须保留到反向计算梯度时才能释放的所有中间张量。它和 N 无关,而是正比于 **批量 × 序列 × 隐藏维 × 层数**。

### 为什么激活必须留着

回到第 3 节:算 dW = Xᵀ·dY 需要前向时的输入 X;算 GELU 的反向需要它的输入;算 softmax 反向需要 softmax 的输出。**链式法则要求反向时能拿到前向的中间值。** 所以前向每算一层,就得把那层反向需要的中间张量"存起来"挂在计算图上,直到反向用完才释放。**前向是激活的累积过程,显存占用从 0 涨到峰值;反向是释放过程。峰值出现在前向末尾 / 反向初。**

### 量级公式

一层 Transformer 在前向中要存的主要激活(按"每 token 每层"计,乘以 batch×seq 得总量):

```
每层需保留的激活元素数(粗略) ∝ s·b·h × (常数)
  b = batch,  s = seq,  h = d_model
```

把一层里的大块列出来(为说明机制,取代表性项,精确系数随实现/是否用 FlashAttention 而变):

```
注意力块:
  - QKV 投影的输入(用于反向 dW)          : s·b·h
  - Q, K, V 三个张量                       : 3·s·b·h
  - 注意力分数矩阵 (softmax 前/后)          : 2·b·n_h·s²   ← 注意 s² !
  - 注意力输出、投影输入                    : ~2·s·b·h
FFN 块:
  - FFN 中间激活 (h → 4h → h)              : ~8·s·b·h
LayerNorm / dropout mask / 残差等          : 若干 ·s·b·h
```

**关键观察两点:**

1. **绝大多数项 ∝ s·b·h,但注意力分数矩阵 ∝ b·n_h·s²** ——序列一长,s² 项爆炸。这就是为什么"长上下文"是显存噩梦,也是 FlashAttention(第 9 章)用**不存完整 s×s 矩阵**(在线 softmax + 分块)来砍掉这一项的根本动机。**用了 FlashAttention,激活账里那个 s² 大项基本消失。**

2. **总激活 ∝ 层数 L**:每层独立累积一份。把单层乘以 L:

```
总激活显存 ≈ L · s · b · h · (每元素字节数) · (单层系数 A)
```

A 是单层激活系数(把上面那些项加起来的常数,典型值在十几到几十之间,取决于是否存 attention 矩阵、是否 fp16)。有一个被广泛引用的估算(Megatron 的激活内存分析,见导读),给出**不做任何重计算时**,单层激活字节数约:

```
单层激活 ≈ s·b·h · (34 + 5·n_h·s/h)  字节   「待核:34 与 5 两个系数来自特定实现(fp16、含 dropout/attention 矩阵),不同框架略有出入」
```

其中括号里 `34` 是 ∝s·b·h 项的合并系数,`5·n_h·s/h` 来自 ∝s² 的注意力矩阵项。乘 L 即全模型激活。**这个公式的形状(线性项 + s² 项)比具体系数更重要——记住"激活 = 一堆 ∝sbh 的项 + 一个 ∝s² 的注意力项,再 ×L"。**

### 重计算(activation checkpointing)如何改写这一行

激活显存是**唯一可以用算力换的**那块账(模型状态换不掉,它是数值刚需)。机制:前向时只保留每层的输入(边界),中间激活全部丢弃;反向到该层时,**用保留的输入重新跑一遍该层前向**,临时重建中间激活,用完再扔。

```
不重计算:  存 L 层全部中间激活        显存 ∝ L · A · sbh,  算力 = 6ND
全重计算:  只存 L 层的"层边界"激活    显存 ∝ L · sbh(系数大降),算力 ≈ +1 次前向 ≈ 6ND·(1 + 2/6) = 8ND
```

**代价精确量化:多一次前向 = 多 2ND 算力,训练算力从 6ND 变 ≈ 8ND(增加约 33%)**「待核:精确比例取决于哪些层做重计算」。第 8 章会讲更精细的 O(√L) 选择性重计算(只在 √L 个边界存,把显存从 O(L) 降到 O(√L) 而算力只多一遍前向)。**这里你要记住的账本逻辑:激活显存可调,模型状态显存不可调——这决定了你优化时先动哪一行。**

## 8. 显存账(三):完整的一卡显存清单

把所有块拼起来,得到训练时单卡(单设备、无并行)的显存峰值:

```
显存峰值 ≈  16·N           模型状态(参数+梯度+Adam, fp32 主副本)   ← 与 batch 无关
          + 激活            ≈ L·A·sbh (+ s² 注意力项)               ← 与 batch、seq 强相关
          + 临时缓冲        梯度规约缓冲、cudnn workspace、碎片      ← 经验上预留 5~15%
          + CUDA context    驱动/框架固定开销,几百 MB ~ 数 GB「待核」
```

**三块的"可压缩性"完全不同,这张表就是优化决策树的根:**

| 显存块 | 正比于 | 能不能省 | 用什么省(后续章节) |
|---|---|---|---|
| 模型状态 16N | N | 不能砍,只能**分片** | ZeRO-1/2/3、FSDP(第 5 章);张量/流水线并行切 N(第 3、4 章) |
| 激活 | b·s·h·L | 能用**算力换** | 重计算(第 8 章);FlashAttention 砍 s² 项(第 9 章);序列并行 |
| 临时缓冲 | 杂项 | 调框架参数 | 通信/计算重叠、桶大小调优(第 2、14 章) |

**读这张表就是做诊断:**
- 如果模型状态(16N)就超单卡 → **必须**上模型状态分片(ZeRO)或模型并行,这是硬约束,跟 batch 无关。
- 如果模型状态放得下、但加上激活就 OOM → 先调 batch、上重计算 / FlashAttention,实在不行再上序列并行 / 流水线并行。
- 如果两者都放得下、只是想更快 → 那是 MFU / 算力问题,回到第 0 章 Roofline 和后面的通信优化。

**这就是本章承诺的"账本指导并行 / 省显存技术选型"的具体兑现方式。** 后面每一章你都可以回到这张表,看它在攻击哪一行。

## 9. 代码:用一个小 Transformer 把账本验证出来

光推导不够,必须能在真机上把每一项打印出来对账。下面的代码构造一个小 Transformer,用 Adam 训练一步,分别测出**参数 / 梯度 / 优化器状态 / 激活**各占多少字节,验证 16N 和激活量级公式。

```python
import torch
import torch.nn as nn

# ---- 1. 一个最小可对账的 Transformer block 堆叠 ----
class Block(nn.Module):
    def __init__(self, h, n_head, ff_mult=4):
        super().__init__()
        self.ln1 = nn.LayerNorm(h)
        self.attn = nn.MultiheadAttention(h, n_head, batch_first=True)
        self.ln2 = nn.LayerNorm(h)
        self.ff = nn.Sequential(nn.Linear(h, ff_mult*h), nn.GELU(),
                                nn.Linear(ff_mult*h, h))
    def forward(self, x):
        a, _ = self.attn(self.ln1(x), self.ln1(x), self.ln1(x))
        x = x + a
        x = x + self.ff(self.ln2(x))
        return x

class Tiny(nn.Module):
    def __init__(self, L=6, h=512, n_head=8, vocab=32000):
        super().__init__()
        self.emb = nn.Embedding(vocab, h)
        self.blocks = nn.ModuleList([Block(h, n_head) for _ in range(L)])
        self.head = nn.Linear(h, vocab, bias=False)
    def forward(self, idx):
        x = self.emb(idx)
        for blk in self.blocks: x = blk(x)
        return self.head(x)

def bytes_of(tensors):  # 求一组张量的总字节
    return sum(t.numel() * t.element_size() for t in tensors)

dev = 'cuda'
model = Tiny().to(dev).half()          # 用 fp16 计算副本(混合精度的"计算份")
b, s = 8, 256
idx = torch.randint(0, 32000, (b, s), device=dev)

# ---- 2. 参数显存(fp16 计算副本) ----
N = sum(p.numel() for p in model.parameters())
param_bytes = bytes_of(model.parameters())          # 应 ≈ 2·N (fp16)
print(f"参数量 N = {N/1e6:.2f} M")
print(f"参数显存 = {param_bytes/1e6:.1f} MB  (理论 2N = {2*N/1e6:.1f} MB)")

# ---- 3. 跑一步,测梯度 + 激活 + 优化器状态 ----
opt = torch.optim.AdamW(model.parameters(), lr=1e-4)
torch.cuda.reset_peak_memory_stats(dev)
base = torch.cuda.memory_allocated(dev)              # 前向前的常驻基线

out = model(idx)
peak_fwd = torch.cuda.max_memory_allocated(dev)      # 前向峰值(含激活)
act_bytes = peak_fwd - base                          # 激活 ≈ 前向涨幅
print(f"激活显存 ≈ {act_bytes/1e6:.1f} MB   (∝ L·s·b·h)")

loss = out.float().mean()
loss.backward()
grad_bytes = bytes_of([p.grad for p in model.parameters() if p.grad is not None])
print(f"梯度显存 = {grad_bytes/1e6:.1f} MB  (理论 2N = {2*N/1e6:.1f} MB, fp16)")

opt.step()   # 此后 Adam 才创建 m, v
opt_bytes = 0
for st in opt.state.values():
    for k in ('exp_avg', 'exp_avg_sq'):              # m 和 v
        if k in st: opt_bytes += st[k].numel()*st[k].element_size()
print(f"优化器状态(m+v) = {opt_bytes/1e6:.1f} MB  (纯 .half() 模型下 ≈ 4N = {4*N/1e6:.1f} MB, fp16 状态)")
```

**预期对账(数量级,真实数字随 PyTorch 版本/算子实现略有偏差):**

- `参数显存 ≈ 2N`:fp16 每参数 2 字节,直接命中。
- `梯度显存 ≈ 2N`:每参数一个 fp16 梯度。
- `优化器状态 m+v`:vanilla AdamW 的 m、v 继承参数 dtype——纯 `.half()` 模型下是 fp16(每个 2N,合计 ≈ 4N);只有走 `torch.cuda.amp` + `GradScaler` 维护 fp32 master copy 时,优化器在 fp32 主权重上更新,m、v 才是 fp32(每个 4N,合计 8N)。注意纯 `.half()` 模型不会自动维护 fp32 主副本,主副本那 4N 在那时才出现——这正是第 7 章的内容。
- 把 `2N(参数)+2N(梯度)+4N(主副本)+8N(m,v) = 16N` 拼齐,就是账本主行。
- `激活显存`:把 b、s 翻倍,观察它**线性翻倍**(验证 ∝ b·s);把 L 翻倍(改 `Tiny(L=12)`),观察激活**线性翻倍**(验证 ∝ L)。这是验证激活量级公式最直接的实验。

**实验心法:不要只信打印的绝对值,要验证"标度关系"——b 翻倍激活翻倍、L 翻倍激活翻倍、N 翻倍状态翻倍。标度对了,说明你抓住了机制;绝对值的零头是实现细节。**

## 10. 设计权衡与常见坑

**坑 1:把推理显存当训练显存。** 推理 1 份 fp16 参数(2N);训练 16N,差 8 倍。这是最高频 OOM 原因。**判断训练能否单卡放下,起步就乘 16N,再加激活。**

**坑 2:以为"算力省了显存就省"。** 算力(6ND)和显存(16N + 激活)是两本账,优化手段不互通。重计算**省显存但费算力**(6ND→8ND);混合精度**省显存又省算力**(fp16 GEMM 更快且更省);ZeRO **省显存但增通信**(不动算力)。**先问"我是被算力卡住(慢)还是被显存卡住(OOM)",再选对应那本账的工具。**

**坑 3:6ND 在长序列下低估算力。** seq 几万时 attention 的 s² 项反超,6ND 漏算它。**长上下文务必把 attention FLOPs(∝ b·n_h·s²)单独加回去。**

**坑 4:忽略激活的 s² 项导致显存预测严重偏低。** 同上,长序列时 attention 分数矩阵主导激活。用 FlashAttention 后该项消失,你的旧估算会"突然"高估,要重算。

**坑 5:把 batch 加大却忘了激活线性涨。** 模型状态不随 batch 变,激活随 batch 线性变。OOM 时第一反应应是"激活太大了"——降 batch 或开重计算,而不是去动模型状态。

**坑 6:MFU 当成常数。** MFU 受并行策略、序列长度、通信重叠强烈影响,不是固定值。报训练时长时,**MFU 必须基于实测或同档位经验,别拍 0.5 了事**「待核」。

**权衡总览(本章账本视角):**

| 你被什么卡住 | 看哪本账 | 候选武器 | 代价 |
|---|---|---|---|
| OOM:模型状态超卡 | 16N | ZeRO-1/2/3、TP、PP | 增通信 / 增复杂度 |
| OOM:激活超卡 | L·A·sbh | 重计算、FlashAttn、降 batch | 增算力 / 改 kernel |
| 太慢:算力不够 | 6ND / (P·F·MFU) | 混合精度、抬 MFU、加卡 | 数值风险 / 成本 |
| 太慢:通信瓶颈 | (后续章) | 重叠、融合、拓扑感知 | 工程复杂度 |

## 13. 小结与承上启下

这一章你拿到了两本能口算、能对账、能指导决策的账:

**算力账**:一切源于一次 GEMM = 2·m·n·k FLOPs;前向 ≈ 2ND;反向因为每个前向 GEMM 裂成 dW、dX 两个同规模 GEMM 而 = 2× 前向 = 4ND;**训练总算力 C ≈ 6ND**,只取决于参数量 N 和 token 数 D,与模型形状无关(长序列除外,需补 attention 的 s² 项)。算力换算到墙钟要除以 P·F_peak·MFU。

**显存账**:常驻的**模型状态 ≈ 16N**(fp16 参数 2 + fp16 梯度 2 + fp32 主副本 4 + Adam m 4 + v 4),只跟 N 走、不可砍只可分片;**激活** ∝ b·s·h·L(外加 attention 的 s² 项),跟 batch/seq 走、可用算力换(重计算)或用更好的 kernel 砍(FlashAttention)。两块加临时缓冲,就是单卡显存峰值。

**最重要的一件事**:第 8 节那张"哪块显存、正比于什么、能不能省、用什么省"的表,是后面整门课的决策根。你现在能在白板上写出"7B 训练要 112GB 状态、单卡放不下、所以必须分片或并行"——这就是从"试试看 OOM 不 OOM"升级到"先算账再决策"的专家姿势。

**承上**:第 0 章给了单 kernel 的 Roofline(算力 vs 带宽两堵墙)和 MFU 拿不满的物理原因;本章把它放大到整个训练步的总账。**启下**:接下来三章是"分摊这本账"的三种正交切法——

- 第 2 章**数据并行**:复制 16N 到每张卡,用 All-Reduce 同步梯度(攻击的是"算力 / D",不省状态)。
- 第 3 章**张量并行**、第 4 章**流水线并行**:把 N 本身切到多卡(直接砍 16N 那一行)。
- 第 5 章 **ZeRO / FSDP**:把 16N 里的优化器状态 / 梯度 / 参数逐级分片(精准拆第 6 节那 16 字节)。
- 第 7、8 章**混合精度 / 重计算**:改写显存账里"每参数字节数"和"激活"两行。

每学一个新技术,回到这两本账,问一句:**它在改账本的哪个数字,代价记在哪本账上?** 想清这句,你就不会在"该上哪种并行 / 省显存技术"上拍脑袋了。

## 练习参考答案

### 练习 1:7B 模型状态显存 + ZeRO-1 切优化器状态

本题全程用本章表格的十进制约定(GB = 1e9 字节),与正文「7B ≈ 112 GB」一致;若按 GiB(2³⁰)算数值会略小(见末尾备注)。

**(a) 模型状态显存,单卡放得下吗?**

混合精度 + Adam 下,每参数常驻 16 字节,模型状态与 batch、seq 无关,只看 N:

```
模型状态 = 16 · N = 16 · 7e9 = 1.12e11 字节
        = 112 GB
```

80GB 单卡 **放不下**,差:

```
112 − 80 = 32 GB
```

**注意 micro-batch=1、seq=4096 在这一问完全用不上**——模型状态不随 batch/seq 变。这正是本章强调的:16N 是判断"单卡够不够"的第一道闸门,与喂多大数据无关。

**(b) ZeRO-1 把 12N 优化器份切到 8 卡后,每卡多少?**

先拆开 16N 这 16 字节:

```
16N = 2(fp16 参数) + 2(fp16 梯度) + 4(fp32 主副本) + 4(Adam m) + 4(Adam v)
    └─── 4N:计算份 ───┘  └────────── 12N:优化器份 ──────────┘
```

ZeRO-1 只分片那 **12N 优化器份**(fp32 主副本 + m + v),fp16 参数和梯度(共 4N)每卡仍各留整份。每卡:

```
每卡模型状态 = 4N + 12N/8
            = 4N + 1.5N
            = 5.5N
            = 5.5 · 7e9 = 3.85e10 字节
            = 38.5 GB
```

38.5 GB **放得下 80GB 单卡**(暂不计激活),富余约 41.5 GB。

**结论:仅靠 ZeRO-1 分片优化器状态,就把 7B 训练的常驻显存从 112 GB / 卡压到 38.5 GB / 卡,跨过了 80GB 闸门。** 这也解释了为什么 ZeRO-1 是 7B 量级最先尝试的省显存手段——只切最大的那 12N、且不增加额外算力。

> 备注(GiB 约定):若按 1 GiB = 2³⁰ 字节,16N ≈ 104.3 GiB(仍 > 80)、5.5N ≈ 35.9 GiB(仍 < 80),结论不变。

### 练习 2:7B 单卡激活显存,s² 项的统治力与 FlashAttention

参数:h=4096、n_h=32、L=32、s=4096、b=1,fp16。单层激活公式 `s·b·h·(34 + 5·n_h·s/h)` 字节(系数为「待核」的特定实现值,见正文)。

先算基数与两项系数:

```
s·b·h = 4096 · 1 · 4096 = 16 777 216 元素 ≈ 1.678e7
n_h·s/h = 32 · 4096 / 4096 = 32
∝sbh 项系数 = 34
∝s²  项系数 = 5 · 32 = 160        ← 已经是 ∝sbh 项的 4.7 倍
```

**(a) ∝sbh 项 vs ∝s² 项,哪个大?**

单层:

```
∝sbh 项 = 16 777 216 · 34  ≈ 5.70e8 字节  ≈ 0.53 GB
∝s²  项 = 16 777 216 · 160 ≈ 2.68e9 字节  ≈ 2.50 GB
```

乘 L=32 得全模型激活:

```
∝sbh 项总计 = 0.53 · 32 ≈ 17 GB
∝s²  项总计 = 2.50 · 32 ≈ 80 GB
─────────────────────────────────
总激活      ≈ 97 GB
```

**∝s² 的注意力分数矩阵项(≈80 GB)远大于所有 ∝sbh 项之和(≈17 GB),约 4.7 倍。** 这就是提示里要你亲手看到的:s=4096 时,系数 160 vs 34,注意力矩阵单枪匹马统治了激活账。序列再长,这个比值还会按 s 线性继续拉大。

**(b) 用 FlashAttention 砍掉 s² 项后,激活降到多少?**

FlashAttention 不存完整 s×s 分数矩阵(在线 softmax + 分块),∝s² 项基本消失,只剩 ∝sbh 项:

```
激活 ≈ 17 GB         (从 97 GB 降下来,约为原来的 17.5%,砍掉约 82%)
```

**(c) 7B + 这个激活,不分片单卡 80GB 差多少?**

```
模型状态 16N = 112 GB
激活(含 s²) ≈  97 GB
──────────────────────
合计        ≈ 209 GB
```

(若用正文激活数 97 GB,这里和 16N 直接相加。)对 80GB 单卡:

```
209 − 80 ≈ 129 GB   严重超出,差约 130 GB
```

**结论:不分片时 7B + 4K 序列单卡缺口约 130 GB,光模型状态(112 GB)就已超卡,再叠激活(97 GB,其中 80 GB 是 s² 注意力矩阵)更无从谈起。** 这把全章的优化栈一次性串起来了:**16N 那一行只能靠分片(ZeRO/TP/PP,练习 1 已演示),激活那一行可用 FlashAttention 砍 s²(本题 97→17 GB)+ 重计算 + 降 batch。** 两本账分别用不同武器,这正是第 8 节决策表的实战。

> 备注:总额数值随 GB/GiB 约定与激活公式系数(「待核」)浮动;但"两块都远超单卡、s² 项主导激活"的结论稳健。

### 练习 3:证明不重计算 = 6ND,全层重计算 = 8ND

**设定**:N = 可训练参数量,D = 训练总 token 数。本章已证两块基石(第 2、3 节):

```
前向 FLOPs ≈ 2 · N · D        每个 GEMM 产出一份输出,系数 2 来自一次乘加 = 2 FLOPs
反向 FLOPs ≈ 2 × 前向 = 4ND   每个前向 GEMM 在反向裂成 dW=Xᵀ·dY 和 dX=dY·Wᵀ 两个同规模 GEMM
```

**情形一:不做重计算。**

一次迭代 = 一遍前向 + 一遍反向,前向产出的激活全部留在显存里供反向直接取用,反向不需要重算前向:

```
C_iter = 前向 + 反向 = 2ND + 4ND = 6ND
对所有 step 求和(token 累加 = D)得:
    C_train = 6ND  ✓
```

**情形二:全层重计算(activation checkpointing,所有层都重算)。**

机制:前向时只保留每层"层边界"激活、丢弃中间激活;反向走到某层时,**先用保留的边界输入把该层前向重跑一遍**,临时重建中间激活,再做反向。于是反向阶段额外多了一整遍前向:

```
额外前向 = 2ND
C_iter = 原前向 + 额外前向 + 反向
       = 2ND  +  2ND   + 4ND
       = 8ND
对所有 step 求和得:
    C_train = 8ND  ✓
```

**代价量化**:

```
8ND / 6ND = 4/3 ≈ 1.333   →  算力增加约 33%
```

**结论:不重计算 C = 6ND;全层重计算 C = 8ND,算力多约 1/3(多出的恰好是那一遍重跑的前向 2ND)。** 这是"用算力换显存"的精确价签——激活显存大降(O(L)→只存边界),代价是 +2ND 算力。

**补充(上界说明,呼应提示)**:8ND 是"每一层都重算"的**上界**。实际工程常做选择性重计算(第 8 章的 O(√L) 策略:只在 √L 个边界存激活),只对部分层重跑前向,额外算力 < 2ND,因此真实训练算力落在 **6ND ~ 8ND 之间**,越省显存越靠近 8ND。

### 练习 4:放大到 70B,算力 / 墙钟 / 卡数张力

N=70e9,D=2e12 token,P=2048 卡,MFU=0.4。

**(a) 总训练算力 C = 6ND。**

```
C = 6 · 70e9 · 2e12
  = 6 · 1.4e23
  = 8.4e23 FLOPs   (= 840 ZFLOP)
```

**(b) 墙钟约几天?**

```
T = C / (P · F_peak · MFU)
  = 8.4e23 / (2048 · F_peak · 0.4)
  = 8.4e23 / (819.2 · F_peak)
  ≈ 1.025e21 / F_peak   秒
  ≈ 1.187e16 / F_peak   天        (÷86400)
```

代入你手上卡的真实 BF16 标称峰值 F_peak(单位 FLOPS)即得天数。给几个量级参考(F_peak 仅为说明,「待核」按真实硬件代入):

```
F_peak = 1e15 FLOPS(≈1 PFLOPS):T ≈ 1.03e6 s ≈ 11.9 天
F_peak = 2e15 FLOPS           :T ≈ 5.13e5 s ≈  5.9 天
F_peak = 5e14 FLOPS           :T ≈ 2.05e6 s ≈ 23.7 天
```

**(c) 光放下模型状态最少几卡?与算力要的 2048 卡什么关系?**

```
模型状态 16N = 16 · 70e9 = 1.12e12 字节 = 1120 GB（= 1.12 TB）
按每卡 80GB:⌈1120 / 80⌉ = ⌈14⌉ = 14 卡
```

仅从**显存约束**看,14 卡就能把 70B 的模型状态摆下;但**算力约束**(想在合理时间内训完 8.4e23 FLOPs)要 2048 卡。两者给出的卡数差两个数量级。

**结论:显存约束(14 卡)和算力约束(2048 卡)对卡数的要求完全不同,实际取两者的较大者(2048 卡)。** 这正是 3D 并行(第 6 章)要协调的核心张力——14 卡只够"塞下",远不够"跑快";而 2048 卡若只做数据并行,则每卡都要扛整份 16N(1120GB / 卡,远超 80GB),又放不下。于是必须把"切模型放下显存(TP/PP/ZeRO)"和"加卡提吞吐(DP)"两个维度组合,让显存维度切到每卡 ≤ 80GB、数据维度把卡数堆到算力需要的 2048,这就是 3D 并行存在的根本原因。

> 备注:GB 用 1e9 约定时数值如上;按 GiB 算 16N ≈ 1043 GiB,⌈1043/80⌉ 仍 = 14 卡,结论不变。(b) 的天数随 F_peak、MFU 实测值变化,故标「待核」。
