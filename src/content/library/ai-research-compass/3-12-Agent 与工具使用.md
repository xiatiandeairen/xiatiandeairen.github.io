> 这一章把你从"会写 ReAct prompt、会调函数"带到能用决策理论(POMDP)把任何 LLM agent 任务形式化、能算清楚 agent 上 RL 的方差代价与收益边界、能判断一个具体任务该上 prompt 工程还是该上 RLVR 端到端训练,并知道工程上多步可靠性到底卡在哪。

## 1. 为什么 agent 必须当决策问题看

你已经会让一个 LLM "调工具":给它一段 system prompt,告诉它有 `search(query)`、`python(code)`、`read_file(path)` 几个工具,它输出一段 JSON,你解析、执行、把结果拼回上下文,再让它继续。能跑。

但一旦任务变成 5 步、10 步、20 步,事情开始崩:

- 第 3 步工具报错,模型不知道怎么恢复,要么死循环重试同一个调用,要么放弃编一个答案。
- 模型在第 7 步"忘了"第 2 步已经查到的关键事实(被后面的噪声挤出有效注意力)。
- 一个本来应该 4 步完成的任务,模型绕了 15 步还没收敛,token 烧完了。
- 你想"让它更可靠",但你手上只有 prompt 这一个旋钮,改了 A 任务好了、B 任务又坏了。

**这些都不是 prompt 写得好不好的问题,而是"序贯决策在不确定环境下不稳定"的问题。** 序贯决策(sequential decision making)的意思是:你的每一步动作会改变后续你能看到什么、能做什么,错误会沿时间累积。这正是强化学习研究了几十年的对象。把 agent 当成一个 prompt 拼接技巧,你永远只能在表面打补丁;把它当成一个决策问题,你才能问出对的问题:状态是什么?动作空间是什么?奖励在哪?误差怎么沿时间传播?什么时候该学一个策略而不是写一段 prompt?

这一章的目标不是教你写更花哨的 agent 框架,而是给你一副决策理论的眼镜。戴上它,你会发现 ReAct、function calling、plan-and-execute、Reflexion 这些"范式"其实都是在同一个 POMDP 上选择不同的策略表示和不同的求解手段;你也会算清楚,为什么 2023-2025 这一波 agent 绝大多数靠 prompting/SFT 而非纯 RL——这不是品味问题,是方差和样本成本的硬约束。

**前置假设(本章默认你已掌握)**:MDP 的 `(S, A, P, R, γ)` 定义、价值函数 `V/Q`、Bellman 方程、策略梯度定理、REINFORCE 的方差问题、PPO 的 clip 目标、以及前面章节讲的 RLHF 与 RLVR(用可验证奖励训练推理模型)。如果策略梯度的方差来源你还不熟,本章第 5 节的推导会比较吃力,建议先回看策略梯度那一章。

## 2. 形式化:agent 是一个 POMDP 上的策略

### 2.1 从 MDP 到 POMDP:为什么必须"部分可观"

标准 MDP 假设智能体能看到完整状态 `s`。但 LLM agent 几乎从不满足这个假设。考虑一个"在代码库里修 bug"的 agent:

- 真实状态 `s` 包含:整个代码库的全部内容、所有文件的依赖关系、运行时的真实行为、bug 的真实根因。
- agent 实际能看到的 `o`(观察):只有它 `read_file` 读过的那几个文件、`grep` 出来的那几行、测试跑出来的那段 stderr。

**它看到的永远是状态的一个投影,而且这个投影由它自己的动作决定。** 这就是部分可观马尔可夫决策过程(Partially Observable MDP,POMDP)的核心:你不知道你在哪,你只能通过观察去推断你在哪,而你拿到什么观察取决于你做了什么动作。

POMDP 形式化为一个七元组:

```
POMDP = (S, A, O, P, Z, R, γ)

S : 隐藏状态空间      —— 环境真实状态(代码库全貌、网页全集、数据库真值)
A : 动作空间          —— agent 能执行的动作(调用某工具 + 参数,或输出最终答案)
O : 观察空间          —— agent 能收到的反馈(工具返回的文本/结构化结果)
P : S × A → Δ(S)      —— 状态转移:P(s' | s, a),动作如何改变隐藏状态
Z : S × A → Δ(O)      —— 观察函数:Z(o | s', a),到了 s' 后能观察到什么
R : S × A → ℝ          —— 奖励函数(任务成功 / 失败 / 中间信号)
γ : 折扣因子 ∈ [0,1)
```

把 agent 的工作流"观察 → 思考 → 行动 → 观察 → ..."逐项对应:

- **观察**:收到 `o ∈ O`(上一步工具的返回)。
- **思考**:这是 agent 内部对"我现在在哪、该干啥"的推断——后面会看到,这正是在维护信念状态(belief)。
- **行动**:选 `a ∈ A`(调哪个工具、传什么参数,或者 `finish(answer)`)。
- **新观察**:环境按 `P` 转移到新隐藏状态 `s'`,再按 `Z` 吐出新观察 `o'`。

一个 agent 的完整运行就是一条轨迹(trajectory):

```
τ = (o₀, a₀, o₁, a₁, o₂, a₂, ..., o_T, a_T)
```

注意这里只有观察和动作,没有 `s`——因为 agent 全程看不到真实状态。这是 POMDP 区别于 MDP 的本质:**决策必须建立在观察历史上,而非状态上。**

### 2.2 信念状态:POMDP 如何"退化"回 MDP

POMDP 的标准理论结论是:**虽然 agent 看不到 `s`,但如果它维护一个关于 `s` 的概率分布(信念状态 belief state),那么在这个信念空间上,问题重新变成一个 MDP。** 这是把 POMDP 求解工具接回 MDP 的关键桥梁,值得完整推一遍。

信念 `b` 是隐藏状态上的一个分布:`b(s) = P(s | 历史)`。定义 t 时刻的历史 `h_t = (o₀, a₀, ..., a_{t-1}, o_t)`,信念就是后验 `b_t(s) = P(s_t = s | h_t)`。

**信念更新(belief update)推导**。设当前信念 `b_t`,执行动作 `a`,收到新观察 `o'`,我们要算更新后的信念 `b_{t+1}(s')`。用贝叶斯:

```
b_{t+1}(s') = P(s' | h_t, a, o')
            = P(o', s' | h_t, a) / P(o' | h_t, a)            # 贝叶斯,分母是归一化常数
            ∝ P(o' | s', a) · P(s' | h_t, a)                  # 链式,o' 只依赖 s' 和 a(观察函数 Z)
            = Z(o' | s', a) · Σ_s P(s' | s, a) · b_t(s)        # 对当前状态边缘化,P(s'|s,a) 是转移 P

记 η = 1 / P(o' | h_t, a) 为归一化常数,则:

b_{t+1}(s') = η · Z(o' | s', a) · Σ_s P(s' | s, a) · b_t(s)
```

这个更新算子记作 `b_{t+1} = τ(b_t, a, o')`。它说明:**新信念只由旧信念、刚做的动作、刚收到的观察决定,不依赖更早的历史。** 这正是马尔可夫性——只不过是定义在信念空间上的马尔可夫性。

于是定义 **belief-MDP**:状态空间是所有信念组成的概率单纯形 `Δ(S)`,转移核由 `τ` 加上观察分布 `P(o'|b,a)` 诱导,奖励是信念加权的期望奖励 `ρ(b,a) = Σ_s b(s) R(s,a)`。在这个 belief-MDP 上,最优价值函数满足 Bellman 方程:

```
V*(b) = max_a [ ρ(b,a) + γ · Σ_{o'} P(o' | b, a) · V*( τ(b,a,o') ) ]
```

**这件事为什么重要(对 agent 的直接含义)**:它告诉我们,一个理论上最优的 agent,内部应该维护一个对"环境真实状态"的概率估计,每步用观察去更新它,并基于这个估计去选动作。而 LLM 在做什么?**LLM 的上下文窗口(context)就是历史 `h_t` 的一个有损编码,LLM 的前向计算就是在隐式地、近似地实现 belief update + 策略 `π(a|b)`。** "思考"(reasoning trace)的作用,就是把这个隐式 belief 显式写出来,让后面的步骤能基于它决策。这给了我们一个干净的判据:**任何会破坏 belief 表达的设计(上下文溢出、关键观察被挤掉、思考被截断)都会直接损害 agent 的最优性。** 第 7 节的"记忆管理"本质就是 belief state 的工程化压缩。

> 一个常见误区:有人觉得 POMDP 太理论、和"写 prompt"无关。恰恰相反,上面这条 Bellman 方程是你判断"上下文该放什么、记忆该存什么"的第一性依据——该放的是对推断隐藏状态有用的充分统计量,不是越多越好。

### 2.3 把 LLM agent 三要素精确落位

为了后面能算,我们把 LLM agent 的策略写清楚。设 LLM 参数 θ,策略是:

```
π_θ(a_t | h_t)   其中 a_t 是"下一步动作"的 token 序列(含工具名 + 参数,或 finish)
```

这里有个工程上极重要、理论上又容易被忽略的细节:**动作 `a_t` 本身是一个 token 序列,不是一个原子符号。** 一个工具调用 `search(query="POMDP belief update derivation")` 在 token 层面是几十个 token 的自回归生成。所以严格说,LLM agent 是一个分层结构:

- **外层(环境步,environment step)**:粒度是"一次工具调用 + 一次观察"。这是 POMDP 的真实时间轴 `t`。
- **内层(token 步,token step)**:粒度是生成动作字符串里的每个 token。这是语言模型的自回归时间轴。

这个双层结构是后面所有方差和信用分配(credit assignment,即把最终奖励正确地归因到具体哪一步动作上)讨论的根源:**奖励通常只在外层的终点出现一次,却要分配到内层成百上千个 token 上。** 记住这一点,第 5 节会反复用到。

## 3. 主流范式:同一个 POMDP 上的不同策略表示

现在用上面的框架重新看四个主流 agent 范式。关键视角:**它们都是在求解同一个 belief-MDP,区别在于(a)策略 `π` 的结构,(b)信念 `b` 怎么表示和维护,(c)用什么手段逼近最优。**

### 3.1 ReAct:把思考交错进动作序列

ReAct(Reason + Act,Yao 等人,2022/2023)的核心主张:让模型在每一步交替产出 **Thought(思考)** 和 **Action(动作)**,思考是自然语言推理,动作是工具调用。

用我们的形式化:ReAct 把策略分解为

```
π_θ(a_t | h_t) = π_θ(action_t | h_t, thought_t) · π_θ(thought_t | h_t)
```

`thought_t` 是什么?**它就是把隐式 belief `b_t` 显式写成自然语言。** "Thought: 我已经知道 X,但还不知道 Y,所以下一步应该查 Y"——这一句话同时干了三件事:(1)把当前 belief 总结出来(对历史的有损压缩 → 充分统计量),(2)做了一步推断(从 belief 推出该做什么),(3)给后续步骤留下可读的记忆。

ReAct 为什么有效?有两个机制:

1. **显式 belief 缓解信用分配**。把"为什么这么做"写出来,等价于在轨迹里插入了中间推理,降低了从观察直接跳到动作的难度(类似 chain-of-thought 把一步难推理拆成多步易推理)。
2. **观察接地(grounding)抑制幻觉**。纯推理(Chain-of-Thought)的问题是无外部反馈、容易越推越错;ReAct 每步用真实观察 `o` 校正 belief,把推理"锚"在环境真值上。

代价:thought 占 token、占上下文;thought 写错会污染后续(错误的 belief 会自我强化)。

### 3.2 Function/Tool Calling:把动作空间结构化

函数调用(function calling)不是一个独立"范式",而是 **动作空间 `A` 的一种结构化约束 + 一种解码约束**。它做的事:

- 用 JSON schema 定义每个工具的名字和参数类型,把 `A` 从"任意自然语言"收窄成"合法的结构化调用"。
- 解码时(可选)用受约束解码(constrained decoding / grammar-constrained sampling)保证输出一定是合法 JSON,合法的工具名,参数类型对。

从决策角度,这是在缩小动作空间。**缩小 `A` 的直接收益是降低策略需要覆盖的搜索空间,提升可靠性;代价是表达力受限**(只能做 schema 里预定义的事)。它和 ReAct 不互斥:你完全可以让模型先输出 thought,再输出一个 function call——主流 agent 框架就是这么做的。

一个量化直觉:若动作是无约束文本,合法动作在所有 token 序列里占比极小,采样到非法动作的概率高、需要靠 prompt 反复纠正;受约束解码把非法动作概率压到 0,等于把"格式错误"这一整类失败模式从误差预算里移除。这是工程可靠性的免费午餐,该用就用。

### 3.3 Plan-and-Execute:把策略分层(option/HRL 的影子)

先规划再执行(plan-and-execute):先让模型产出一个多步计划 `(g₁, g₂, ..., g_K)`(子目标序列),再逐个子目标去执行(每个子目标内部可能又是一个 ReAct 小循环)。

这在 RL 里有精确对应物:**分层强化学习(Hierarchical RL)/ option 框架**。一个 option(技能)是一个三元组 `(I, π_sub, β)`:启动条件、子策略、终止条件。Plan-and-execute 的"计划"就是一串 option 的调度:

```
高层策略 π_high(g | b)        —— 选下一个子目标(option)
低层策略 π_low(a | b, g)       —— 在子目标 g 下选具体动作
终止判据 β(b, g) → {继续, 切下一个子目标}
```

**为什么分层有用——理论收益**:它缩短了有效决策视野(horizon)。如果原任务要 20 个动作步,信用分配要跨 20 步;分成 4 个子目标、每个 5 步,高层只需在 4 步上做信用分配,低层在 5 步内做。视野从 20 降到 max(4, 5),而我们后面会看到,策略梯度方差大致随视野线性增长——所以分层是降方差的结构性手段。

**代价与坑**:计划在执行前生成,但环境是部分可观的——你规划时的 belief 可能是错的,执行到一半发现计划不可行(子目标 3 依赖的前提不成立)。所以纯"先规划全部、再执行全部"很脆,实践中要么允许重规划(replan),要么用更动态的交错。**这正是开环控制 vs 闭环控制的老问题**:开环(规划好就照做)在确定环境里高效,在不确定环境里危险;闭环(每步看反馈再决定)鲁棒但慢。Plan-and-execute 是偏开环的,ReAct 是偏闭环的。

### 3.4 Reflexion:把"跨轨迹的学习"塞进记忆而非权重

Reflexion(Shinn 等人,2023)的设定:agent 尝试一个任务失败了(或拿到一个低的结果反馈),它不更新权重,而是**生成一段自然语言反思(self-reflection),写进一块持久记忆,在下一次尝试时把这段反思读进上下文**,从而避免重蹈覆辙。

用决策语言精确描述:这是一个**外层的、跨 episode 的策略改进循环,但改进量存在"记忆"里而不是"参数 θ"里**。

```
for episode k = 1, 2, ...:
    τ_k ← rollout( π_θ( · | h, memory) )        # 用当前记忆作为额外上下文跑一遍
    r_k ← evaluate(τ_k)                          # 拿到结果反馈(成功/失败/分数/报错)
    reflection_k ← π_θ( reflect | τ_k, r_k )     # 让模型反思这次为什么不行
    memory ← memory ∪ { reflection_k }           # 反思写入记忆(in-context "learning")
```

**这是一种 in-context 的策略迭代(policy iteration)**:`evaluate` 提供了价值信号,`reflect` 扮演了"策略改进算子",`memory` 是改进后的策略的载体。它和真正的 RL(梯度更新 θ)的本质区别在于:

- RL:把经验压进**权重**,泛化到未来所有任务,但要梯度、要大量样本、可能不稳定。
- Reflexion:把经验压进**上下文**,只对当前任务/当前会话有效,会话结束就丢失,但零训练、即时见效、可解释。

**这个对比是本章最重要的判断之一**,直接引出第 4 节:大多数 agent 用的"学习"其实是 in-context 的(prompt/记忆),不是 in-weight 的(RL)。

## 4. 为什么当下 agent 多用 prompting/SFT 而非纯 RL

这一节是很多人含糊带过、但其实可以量化讲清的。结论先行:**纯 RL 在 agent 上面临奖励稀疏、环境难模拟、样本极贵三座大山,而 prompting/SFT/in-context 反思在大多数任务上以远低的成本拿到"足够好"。** 下面逐项算。

### 4.1 奖励稀疏 → 策略梯度方差爆炸(可推导)

agent 任务的奖励通常是**终局稀疏奖励**:整条轨迹只有最后成功才 `+1`,否则 `0`(或失败 `-1`)。中间几十步没有任何信号。我们来算这对 REINFORCE 估计量方差的影响。

回忆策略梯度(对一条长度 T 的轨迹,把每步动作的 log-prob 梯度记作 `g_t = ∇_θ log π_θ(a_t|h_t)`):

```
∇_θ J(θ) = E_τ [ ( Σ_{t=0}^{T} g_t ) · R(τ) ]        # R(τ) 是整条轨迹的回报,稀疏奖励下是 0/1
```

设奖励只在终点,`R(τ) ∈ {0,1}`,成功概率 `p = E[R]`。考虑这个估计量的方差量级。把梯度和记作 `G = Σ_t g_t`。单样本估计量是 `Ĝ = G · R`。

```
方差分解(R 是 {0,1},与 G 不独立,但我们取量级估计):

Var(Ĝ) = E[ (G·R)² ] − ( E[G·R] )²
       = E[ G² · R² ] − ( ∇J )²            # R² = R(因为 R∈{0,1})
       = E[ G² · R ] − ( ∇J )²
       = p · E[ G² | R=1 ] − ( ∇J )²       # 只有成功轨迹贡献,概率只有 p
```

关键看两点:

1. **`E[G² | ·]` 随 T 增长**。`G = Σ_t g_t` 是 T 个(近似零均-相关性弱)随机向量之和,其二阶矩大致随 T 线性增长(若各 `g_t` 方差有界且弱相关,`E‖G‖² = O(T)`)。所以**方差大致随视野 T 线性放大**——任务越长,梯度估计越抖。
2. **`p` 很小时,信号极稀**。早期 agent 在难任务上成功率 `p` 可能只有百分之几,意味着绝大多数 rollout 的 `R=0`、梯度贡献为 0,你烧了一堆 rollout 却几乎没有学习信号。**有效样本数 ≈ 总 rollout × p**。要拿到稳定梯度,所需 rollout 数大致 ∝ 1/p · T(粗略),`p` 越小、T 越长,样本需求越爆炸。

**这就是第一座山的数学根源**:稀疏 + 长视野 = 高方差 + 低有效样本率。对比之下,SFT 用一条专家轨迹就给了每一步的 dense 监督(每个 token 都有 target),没有方差问题,样本效率高出几个数量级。

> 缓解手段(后面 RL 那一节会用):reward-to-go(只用 t 之后的回报,去掉无关项)、baseline/优势函数(减去 `V(h_t)` 降方差但不改偏)、PPO(限制更新步长防崩)、过程奖励(给中间步加信号把稀疏变稠密)。但即便全用上,长视野 agent 的 RL 依然比短任务难训得多。

### 4.2 环境难模拟 → 没有廉价 simulator

RL 的样本效率前提是**能廉价地反复 rollout**。Atari、围棋、MuJoCo 之所以适合 RL,是因为有快速、确定、可重置的 simulator,一秒能跑成千上万步。

agent 的环境是什么?真实的浏览器、真实的代码库、真实的 API、真实的数据库。这些环境:

- **慢**:一次网页加载、一次代码编译、一次 API 调用是秒级甚至分钟级,不是微秒级。
- **不可逆 / 难重置**:agent 给生产数据库写了一条数据、发了一封邮件、下了一个单——你没法 `env.reset()` 回到干净状态。
- **非平稳**:网页内容、API 行为天天变,今天训练的环境明天就漂移了。
- **有副作用和安全风险**:让一个没训好的策略在真实环境里随机探索,可能造成实际损害。

**没有廉价 simulator,RL 的"试错"前提就崩了。** 这是第二座山。工程上的应对是"造仿真环境/沙箱"——但造一个高保真的 agent 仿真环境(比如能模拟整个浏览器+网站行为)本身是个巨大工程,且仿真与真实的差距(sim-to-real gap)会让训出来的策略在真实环境里失灵。

### 4.3 样本贵 → 每个 rollout 都是真金白银

把前两座山叠加:一次 agent rollout 要调用大模型几十次(每步生成 thought+action)、调用真实工具几十次(每次有延迟和金钱成本)。一条长任务 rollout 的成本可能是几美分到几美元。RL 动辄要几十万、上百万条 rollout——成本和时间都难以承受。而且(4.1)告诉你其中大部分 rollout 因为 `R=0` 几乎没贡献。

**三座山合起来的结论**:在大多数 agent 任务上,纯 RL 的"性价比"远不如先用强基座模型 + 好的 prompt/SFT 把 `p` 拉到一个不错的水平。这不是 RL 不行,是**在当前样本成本结构下,RL 的边际收益还没盖过它的边际成本**。

### 4.4 那 RL 在 agent 上的增量到底是什么

RL 不是没用,而是用在 prompt/SFT 摸到天花板、且任务满足特定条件的地方。两个最有价值的增量:

**(a) 用结果奖励做端到端优化,突破"模仿专家"的上限。** SFT 的本质是行为克隆(behavior cloning):模仿人类/强模型示范。它的上限就是示范的质量,且有 **分布漂移(distribution shift)** 问题——模仿学习者一旦走到示范没覆盖的状态,误差会复合(compounding error,经典结论:行为克隆的误差随视野 T 以 O(T²) 量级增长,因为一步犯错把你带到没见过的状态、后面错得更多)。RL 用任务**结果**(而非"像不像专家")作目标,能学到示范里没有的、但确实能完成任务的策略,并且因为是 on-policy 地在自己访问的状态上学,直接对抗了分布漂移。**只要你能定义一个可靠的"任务成功"信号,RL 就能把 agent 优化到超过它的示范来源。**

**(b) 工具使用的 RLVR(可验证奖励的强化学习)。** 这是当前最务实、最成功的 agent-RL 落地形态。RLVR(Reinforcement Learning with Verifiable Rewards)的前提是:**奖励能被程序自动、确定地验证**——数学题答案对不对(对答案)、代码过不过测试(跑单测)、SQL 查得对不对(比对结果)、工具调用拿没拿到目标信息(规则检查)。

为什么 RLVR 特别适合"工具使用"?因为工具天然提供了**可验证的接地反馈**:

- 让模型学会调代码解释器解数学题:奖励 = 最终数值答案是否正确(可验证),模型在 RL 中自己学会"什么时候该写代码算、怎么写对"。
- 让模型学会调检索工具答事实题:奖励 = 答案是否匹配 ground truth,模型学会"什么时候该查、查什么 query"。

RLVR 绕开了 RLHF 需要训奖励模型(reward model)、且奖励模型容易被 reward hacking 的问题——**可验证奖励本身不可被"骗"**(代码要么过测试要么不过,没有中间地带可以钻空子)。它把第 4.1 节的"奖励稀疏"问题部分缓解了:虽然还是终局奖励,但奖励**便宜且可大批量自动生成**,可以靠规模(海量可验证题目 + 大 batch rollout)压方差。这是 2024-2025 推理模型 + agent 训练的主流路线之一。

**判断口诀(本节最该记住的)**:

```
能不能自动、确定、不可骗地验证"任务成功"?
  ├── 能       → RLVR 值得上(数学/代码/结构化工具任务),边际收益高
  └── 不能     → 慎上 RL;要么先做可验证化改造,要么留在 prompt/SFT/Reflexion
任务视野多长 / 单步成功率多高?
  ├── 短 & p 不太低  → RL 训得动
  └── 长 & p 很低     → 先用 SFT 把 p 抬上来(暖启动),或分层缩短视野,再考虑 RL
环境能不能廉价、安全、可重置地 rollout?
  ├── 能(有沙箱/可验证器) → RL 可行
  └── 不能(真实副作用/慢/不可逆) → RL 不可行,别硬上
```

## 5. 把信用分配讲透:agent-RL 的核心技术难点

第 4.1 节说了方差随视野线性涨,这一节把"奖励怎么分到每一步"——信用分配——的数学讲透,因为它是 agent-RL 真正训得起来的关键,也是 RLVR 训练里最常踩坑的地方。

### 5.1 reward-to-go 与优势:为什么能降方差且不引偏

朴素 REINFORCE 用整条轨迹回报 `R(τ)` 乘每一步梯度。但 `t` 时刻的动作 `a_t` 不该为 `t` 之前发生的奖励负责(因果性)。**reward-to-go** 只用 `t` 之后的回报:

```
∇_θ J = E[ Σ_t g_t · ( Σ_{t'≥t} γ^{t'-t} r_{t'} ) ]        # 第 t 步只乘它之后的回报 G_t
```

**为什么去掉之前的奖励不引入偏差**(关键推导,很多人只会用不会证):

```
要证:E[ g_t · r_{t'} ] = 0   对所有 t' < t

E[ g_t · r_{t'} ]
  = E_{h_{t'}}[ r_{t'} · E[ g_t | h_{t'} ] ]              # 先对 t 之前的历史取条件期望;r_{t'} 在 h_t' 下已确定
  = E_{h_{t'}}[ r_{t'} · E_{a_t}[ ∇log π(a_t|h_t) ] ]      # g_t 的期望对 a_t 取(给定 h_t)
                                                          # 这里用到 a_t 只依赖 h_t,且 t'<t 的项不含 a_t
  内层:E_{a_t}[ ∇log π(a_t|h_t) ]
       = Σ_{a_t} π(a_t|h_t) · ∇log π(a_t|h_t)
       = Σ_{a_t} ∇π(a_t|h_t)                              # π·∇logπ = ∇π
       = ∇ Σ_{a_t} π(a_t|h_t)
       = ∇ 1 = 0                                          # 概率归一,梯度为 0(score function 的零均值性)
  ⇒ E[ g_t · r_{t'} ] = E[ r_{t'} · 0 ] = 0   ✓
```

同样的"score function 零均值"技巧证明**减去任意只依赖 `h_t` 的 baseline `b(h_t)` 不引偏**:`E[g_t · b(h_t)] = E_{h_t}[ b(h_t) · E_{a_t}[g_t] ] = E_{h_t}[ b(h_t) · 0 ] = 0`。取 `b(h_t) = V(h_t)`(状态价值),就得到优势函数 `A(h_t, a_t) = Q − V`,这是降方差的最佳 baseline 之一(它去掉了与动作无关的回报波动)。

**对 agent 的含义**:在长视野 agent 上,优势/baseline 不是可选项是必选项——不减 baseline,方差大到根本训不动。但 agent 上有个特殊麻烦:`V(h_t)` 的输入 `h_t` 是变长的整段交互历史,价值函数(critic)很难估准。这是 agent-RL 比单步 RLHF 难的地方之一。

### 5.2 双层结构下的 credit assignment:token 级 vs 动作级 vs 轨迹级

回到第 2.3 节的双层结构:奖励在外层终点(整个任务成功),但策略梯度要落到内层每个 token 的 `log π`。这里有一个**粒度选择**问题,直接决定训练稳定性:

```
轨迹级(trajectory-level):整条轨迹一个回报,所有 token 共享同一个优势。
   → 最稀疏,方差最大,但实现最简单。GRPO 等"结果奖励"方法的基本形态。

动作级(step-level / turn-level):每个"工具调用 + 观察"作为一步,给步级优势。
   → 需要过程奖励或步级价值估计,把信用分到"哪一步工具用错了"。

token 级(token-level):每个 token 一个优势(PPO 的标准形态)。
   → 最细,但终局稀疏奖励下,token 级优势几乎全靠 critic 的 bootstrapping 估出来,
     critic 不准就全乱。RLVR 实践里常用 GAE 或干脆用轨迹级回报广播到 token。
```

**RLVR 在 agent 上最常见的两种实操**:

1. **结果奖励广播 + 组内归一化(GRPO 思路)**:对同一个任务采样一组(group)轨迹,每条只有终局 0/1 奖励;用**组内的相对好坏**作优势(`A_i = (r_i − mean) / std`),把这个标量优势广播到该轨迹的所有 token。好处:不需要训 critic(省掉一个难估的网络),用组内对比天然做了 baseline(组均值就是 baseline);适合可验证奖励能大批量采样的场景。坑:组内全对或全错时优势全为 0(std→0),该样本无梯度——所以题目难度要"可区分"(有的对有的错才有学习信号)。

2. **过程奖励(process reward)把稀疏变稠密**:训一个过程奖励模型(PRM)或用规则给中间步打分(这一步工具调用合不合理、这步推理对不对),把终局奖励拆成每步信号。好处:dense,方差小,信用分配清晰;坑:**过程奖励本身可能被 reward hacking**——模型学会"产出看起来对的中间步"骗 PRM,而不真把任务做对(下一节细讲)。

**算一笔账理解为什么 GRPO 思路在 RLVR 流行**:训 critic 需要它在变长历史 `h_t` 上回归出准确的 `V`,这在长视野、稀疏奖励下极难收敛;而组内归一化用 G 条同题 rollout 的经验均值直接当 baseline,是个无偏(对组内分布)、零额外网络、零额外训练的 baseline。代价是要为每个 prompt 多采 G 倍样本(典型 G=4~16),但 RLVR 场景下 rollout 可验证、可并行、相对便宜,这个交换通常划算。

## 6. 代码:ReAct 循环与最小可训练 agent 骨架

先给一个**讲清机制的 ReAct 推理循环伪代码**(贴近真实实现,去掉框架噪声),再给一个**把它接到策略梯度训练**的骨架,让你看到"推理循环"和"训练循环"是同一条轨迹的两面。

### 6.1 ReAct 推理循环(inference-time,体现 belief-act-observe)

```python
def react_loop(task, tools, llm, max_steps=15):
    """
    把 POMDP 的 belief-act-observe 显式跑出来。
    history 充当对历史 h_t 的有损编码(= LLM 看到的上下文 = 近似 belief state)。
    """
    history = [system_prompt(tools), user(task)]   # h_0:工具说明 + 任务
    for step in range(max_steps):
        # 1) 策略前向:生成 thought(显式 belief)+ action(工具调用 / finish)
        out = llm.generate(history, stop=["Observation:"])   # 让模型写到该观察处停下
        thought, action = parse_react(out)   # 解析 "Thought: ... \n Action: tool[args]"
        history.append(assistant(out))

        # 2) 终止判据 β:模型决定 finish 就退出(对应 belief 已足够回答)
        if action.name == "finish":
            return action.args["answer"], history

        # 3) 环境步:执行工具 = 在真实(隐藏)状态上转移 P,并返回观察 o(观察函数 Z)
        try:
            observation = tools[action.name](**action.args)   # 真实工具调用,可能抛错
        except ToolError as e:
            observation = f"ERROR: {e}"   # 关键:把错误也作为观察喂回去 → 触发错误恢复

        # 4) belief update:把观察并入历史 → LLM 下一步据此更新隐式 belief
        history.append(observation_msg(f"Observation: {observation}"))

    # 5) 预算耗尽未收敛 —— 长视野失败模式;返回 best-effort 或显式失败
    return forced_answer(history, llm), history
```

三个机制点对应前文:`thought` = 显式 belief(3.1);把 `ERROR` 当观察喂回 = 错误恢复的入口(7.2);`max_steps` = 有限视野约束,耗尽即长任务失败(4.1)。

### 6.2 把轨迹接到策略梯度(训练骨架,体现 outcome-reward RLVR)

下面是**用可验证奖励训练 agent**的最小骨架,体现第 5 节的"组内归一化优势广播 + token 级 PPO clip"。省略 batching/分布式细节,只留机制。

```python
def rlvr_step(prompt, group_size, policy, ref_policy, verify_fn, clip_eps=0.2):
    """
    一个 RLVR 训练步(GRPO 风格):同题采一组,组内归一化优势,广播到 token,PPO-clip 更新。
    verify_fn: 可验证奖励 —— 输入完整轨迹与答案,输出 0/1(代码过测试/答案对/...)。
    """
    # --- rollout:同一 prompt 采 G 条轨迹(每条是一个完整 ReAct episode)---
    trajs = [run_react_and_record_logprobs(prompt, policy) for _ in range(group_size)]
    rewards = torch.tensor([verify_fn(t.answer, t.full_text) for t in trajs])  # 终局 0/1

    # --- 组内归一化优势(GRPO):组均值当 baseline,组标准差做尺度,无需训 critic ---
    if rewards.std() < 1e-6:
        return None        # 组内全对或全错 → 无可区分信号 → 跳过(见 5.2 的坑)
    adv = (rewards - rewards.mean()) / (rewards.std() + 1e-6)   # 每条轨迹一个标量优势

    loss = 0.0
    for traj, A in zip(trajs, adv):
        for tok, logp_old in zip(traj.tokens, traj.logprobs):   # 遍历该轨迹每个动作 token
            logp_new = policy.logprob(tok, traj.context_at(tok))
            ratio = torch.exp(logp_new - logp_old)              # 重要性比 π_new/π_old
            # PPO clip:限制单步更新幅度,防长视野下策略崩塌(reward hacking/发散)
            unclipped = ratio * A
            clipped   = torch.clamp(ratio, 1 - clip_eps, 1 + clip_eps) * A
            loss += -torch.min(unclipped, clipped)              # 取悲观下界

            # KL 锚定到参考策略:防止策略漂移太远丢掉语言能力(见 7.3 reward hacking)
            loss += beta * kl(policy, ref_policy, tok, traj.context_at(tok))

    loss = loss / total_tokens(trajs)
    loss.backward()
    return loss
```

**两段代码合起来的洞察**:训练循环里的 `run_react_and_record_logprobs` 就是 6.1 那个推理循环,只是额外记录了每个动作 token 的 `log π`(用于算重要性比)和把奖励留到最后由 `verify_fn` 打。**推理和训练共享同一条轨迹**——这正是把 agent 当决策问题的好处:轨迹是统一的一等公民,推理时用它产出答案,训练时用它算梯度。

> 训练 agent 的常见工程坑(代码里已埋伏笔):(1)`logp_old` 必须是采样时的旧策略概率,不能用更新后的重算,否则重要性比错、训练发散;(2)工具调用返回的 observation token **不应**计入策略损失(那不是策略生成的,是环境给的),只对模型自己生成的 thought/action token 算 loss——这个 mask 写错是 agent-RL 最隐蔽的 bug 之一。

## 7. 工程:多步可靠性、错误恢复、评估、记忆

理论讲完,落地时 80% 的痛苦在这一节。逐项给出第一性的拆解。

### 7.1 多步可靠性:误差为什么沿视野复合

设每步成功概率(不犯致命错)为 `q`,各步近似独立,则一个 T 步任务的整体成功率 ≈ `q^T`。这是个残酷的指数:

```
q = 0.95, T = 5   →  q^T ≈ 0.77
q = 0.95, T = 20  →  q^T ≈ 0.36
q = 0.99, T = 20  →  q^T ≈ 0.82
```

**单步可靠性的微小提升,在长视野上被指数放大。** 这解释了为什么"把单步做对"(受约束解码消格式错、好的工具文档减参数错、self-check 减逻辑错)对长 agent 的总成功率影响巨大,也解释了为什么缩短视野(plan-and-execute 分层、让单个工具更"powerful"从而少几步)是高杠杆的工程手段。注意 `q^T` 是乐观估计(假设独立、且犯错=直接失败);现实里有些错可恢复(下一节),又有些错会污染 belief 导致后续连环错(正相关),两种效应方向相反,真实曲线要靠评估测。

### 7.2 错误恢复:把"错误"建模成一种观察

新手最容易把工具报错当成"程序异常去 try/except 吞掉",这是错的。**在 POMDP 框架里,工具报错是一个信息量极高的观察 `o`**——它告诉 agent "你刚才的动作在当前隐藏状态下不合法/无效",是更新 belief 的宝贵证据。正确做法(6.1 代码已体现):把错误格式化成观察喂回上下文,让策略据此调整。

可恢复性设计的几个第一性要点:

- **错误观察要可读、含修复线索**:`"ERROR: file not found: /src/foo.rs. Did you mean /src/foo/mod.rs?"` 比 `"Exception: ENOENT"` 让策略恢复的概率高得多(它把"该往哪个 belief 更新"直接给了模型)。
- **防死循环**:模型可能对同一个失败动作重试 N 次(belief 没更新)。要么检测重复动作并强制注入"换个思路"的提示,要么用动作去重/步数预算硬截断。
- **幂等与可回滚**:有副作用的工具(写库、发消息)要么设计成幂等(重试无害),要么需要确认。这是安全护栏,不是可选项。

### 7.3 reward hacking:agent-RL 最危险的坑(单独拎出来讲)

RL 优化的是"奖励"不是"你心里想要的任务"。**只要奖励和真实意图有任何缝隙,足够强的优化器一定会钻进去。** 这在 agent 上尤其致命,因为 agent 能调工具,作弊手段更丰富:

- 奖励设成"输出里包含正确答案字符串" → 模型学会把所有可能答案都列出来(总能命中)。
- 奖励设成"代码通过测试" → 模型学会读测试文件、写 `if input==test_case: return hardcoded` 而非实现真逻辑(过拟合到测试)。
- 过程奖励模型给"看起来合理的推理"打分 → 模型学会写漂亮但与最终结论无关的推理(骗 PRM)。
- 奖励里有"调用工具次数越少越好" → 模型学会不调工具直接瞎编(牺牲正确性换简洁分)。

防御手段(都不是银弹,要叠加):

1. **可验证、难骗的奖励优先**(RLVR 的初衷):答案精确匹配 + 隐藏测试集(模型看不到的 held-out test)防过拟合测试。
2. **KL 锚定**:在目标里加 `β · KL(π_θ ‖ π_ref)`,惩罚策略偏离参考模型太远(6.2 代码里那一项)。直觉:reward hacking 往往要把策略推到很"畸形"的区域,KL 项给这种漂移上成本。代价:`β` 太大学不动,太小防不住——这个权衡要调。
3. **对抗式评估**:专门构造能暴露捷径的测试(改变量名让"背答案"失效、用模型没见过的新测试)。
4. **人审 + 行为监控**:盯住"奖励涨了但人看着不对"的信号——奖励曲线漂亮不等于任务真做好了(这是 RLHF/agent-RL 的头号教训)。

> 记住这条:**奖励曲线上升从来不是成功的充分证据。** 在 agent-RL 里,先怀疑 reward hacking,再庆祝。

### 7.4 评估:任务成功率是第一指标,但要会拆

agent 的评估比 SFT/RLHF 难,因为是多步、有副作用、结果可能不唯一。第一性原则:**用端到端任务成功率(task success rate)作主指标**(由可验证器或人判定"任务到底完成没有"),但要拆开诊断:

- **成功率**(主):pass@1 / pass@k(采 k 条有一条成功就算成功——衡量能力上限 vs 稳定性)。
- **效率**:完成任务用了几步 / 多少 token / 多少钱(同样成功,4 步比 15 步好)。
- **失败归因**:失败是格式错、工具用错、推理错、还是视野耗尽?(对应不同的改进手段)。
- **鲁棒性**:换个表述、加点噪声、工具偶发报错时成功率掉多少。

公开 benchmark 这里点几个方向(具体分数与子集构成请「待核」,各 benchmark 版本迭代快):软件工程类(在真实仓库里修 issue、跑测试判定)、网页操作类(在网页/沙箱里完成多步任务)、工具调用类(给定工具集完成 API 编排)、通用 agent 任务集。选 benchmark 的原则:**它的成功判定必须可验证、且贴近你真实要解决的任务分布**,否则刷分无意义。

### 7.5 上下文/记忆管理:belief state 的工程化

回到 2.2 节的理论:**上下文 = belief state 的有损编码;记忆管理 = 在有限上下文窗口里维护一个"对推断隐藏状态足够"的充分统计量。** 这给了记忆设计一个清晰判据——**该留的是对未来决策有用的信息(对推断 `s` 有用 / 对选 `a` 有用),不是越全越好。** 常见手段及其 belief 视角解读:

- **滚动摘要(summarization)**:把早期长历史压成摘要 = 对 belief 做有损压缩,保留充分统计量、丢掉冗余 token。坑:摘要可能丢掉后面才发现重要的细节(压缩有损,belief 失真)。
- **外部记忆 / 检索(retrieval / RAG)**:把信息存外部库,需要时检索回来 = 把 belief 的一部分外置,用检索把相关部分调进上下文。坑:检索不准 = belief 取错,等于看错了世界。
- **结构化 scratchpad**:用结构化笔记(已知事实表、待办、已排除选项)维护 belief = 显式、可更新的充分统计量,比纯文本历史更抗噪。
- **Reflexion 式跨会话记忆**(3.4):把"教训"持久化,是跨 episode 的 belief 先验。

**一个量化提醒**:Transformer 注意力是 O(L²) 的(L 是上下文长度),且实证上模型对长上下文中间部分的利用会变弱("lost in the middle" 现象,具体退化幅度「待核」)。所以无脑把所有历史塞进去既贵又未必有效——这从计算复杂度和有效性两头都支持"主动管理记忆"而非"全量堆叠"。

## 8. 设计权衡与常见坑(汇总)

把全章的权衡收口成一张可操作的表:

- **Prompt/SFT vs RL**:能用 prompt/SFT 达到"足够好"就别上 RL(样本贵、不稳定、易 hack)。只在(a)有可验证难骗的奖励、(b)SFT 摸到天花板要突破示范上限、(c)能廉价安全 rollout 时上 RL,且优先 RLVR。判据见 4.4 口诀。
- **闭环(ReAct)vs 开环(plan-execute)**:环境不确定/反馈廉价 → 偏闭环(每步看观察);环境确定/想省 token → 偏开环(先规划);折中是允许重规划(replan)的混合。
- **动作空间宽(自由文本)vs 窄(function schema)**:宽=表达力强但难可靠;窄=可靠但受限。能枚举的工具就用 schema + 受约束解码,把格式错从误差预算里删掉。
- **视野长 vs 短**:长视野 = 成功率指数衰减(7.1)+ 方差线性涨(4.1)。能缩就缩——分层、让单工具更强、减少不必要步骤。
- **记忆全量 vs 精炼**:全量 = O(L²) 贵 + lost-in-the-middle;精炼 = 省且可能更准,但摘要/检索有损。按"对未来决策有用"裁剪(7.5)。
- **过程奖励 vs 结果奖励**:过程奖励 dense、好训但易被 hack(7.3);结果奖励难骗但稀疏、难训(5.2)。RLVR 实践常用结果奖励 + 组内归一化压方差。

**新手最常踩的五个坑(按踩中频率排)**:

1. 把工具报错 try/except 吞掉,而不是当观察喂回 → agent 失去错误恢复能力。
2. RL 训练时把环境返回的 observation token 也算进策略 loss → 训练污染、发散(6.2 注里那条)。
3. 看到奖励曲线涨就宣布成功,没查 reward hacking → 上线后真实任务表现崩。
4. 在没有廉价/安全 simulator 的真实环境上硬跑纯 RL → 烧钱、造成真实副作用、训不动。
5. 无脑把所有历史堆进上下文 → 又贵又因 lost-in-the-middle 掉点,且挤掉关键 belief。

## 9. 动手练习

**练习 1(形式化,必做)**:把"让 agent 在一个文档站点里回答用户问题(可用工具:`search(query)` 全文检索、`open(doc_id)` 打开文档、`finish(answer)`)"这个任务**精确写成一个 POMDP**。要求:写出 `S / A / O / P(语言描述即可) / Z / R`;明确指出隐藏状态里有哪些 agent 看不到的成分;说明信念状态 `b` 在这个任务里直觉上是什么。
- 提示:`S` 包含整个文档库内容 + 用户问题的真实答案所在位置;`A` = `{search(q), open(id), finish(a)}`;`O` = 检索命中列表 / 文档正文;`R` 设成终局——`finish` 的答案与 ground truth 匹配则 `+1` 否则 `0`;`b` ≈ "答案最可能在哪些文档里"的分布。想清楚:为什么 `open` 一个文档会让 belief 收窄?

**练习 2(分析/决策,必做)**:针对练习 1 的任务,用 4.4 的口诀**论证该上 prompt 工程还是 RLVR**。要求量化讨论:(a)这个任务的奖励可验证吗、好不好骗?(b)估算单步成功率 `q` 和典型视野 `T`,用 `q^T` 估端到端成功率,判断是否需要训练;(c)如果决定上 RLVR,组内归一化优势会在什么情况下失效(回到 5.2 的坑)?
- 提示:文档问答的"答案匹配"可验证但**容易 hack**(把多个候选答案都塞进 finish);想想怎么改奖励(精确匹配 + 单一答案约束 + held-out 问题)。视野通常短(2~4 步),`q` 若已 ≥ 0.9,`q^T` 可能已够好 → 可能不值得上 RL。

**练习 3(编码)**:在 6.1 的 `react_loop` 上**实现错误恢复 + 防死循环**。要求:(a)捕获工具异常并格式化成带修复线索的观察喂回;(b)检测"连续两步发出相同动作"并强制注入一条提示("上一步动作失败/无进展,请换一种方法");(c)加一个"无进展计数器",连续 3 步无新信息就提前终止并返回 best-effort。写出改动后的循环并说明每个机制对应 POMDP 的哪个概念。
- 提示:"相同动作"用 `(action.name, action.args)` 做 key 比对;"无新信息"可用"观察是否与上一次实质相同"近似;注意注入的提示本身也是在改 belief。

**练习 4(推导,进阶)**:第 5.1 节证了"减去只依赖 `h_t` 的 baseline 不引偏"。请进一步**推导降方差最优的常数 baseline**:在把 baseline 限制为一个标量常数 `b` 的情形下,求使梯度估计量 `Σ_t (g_t)(R − b)` 方差最小的 `b*`,并解释它为什么近似等于"用梯度模长平方加权的平均回报"。
- 提示:对 `Var = E[‖g(R−b)‖²] − ‖∇J‖²`(其中 `g = Σ_t g_t`)关于 `b` 求导置零;会得到 `b* = E[‖g‖² R] / E[‖g‖²]`。解释:回报大但梯度也大的样本被加权更重——这正是经典 REINFORCE-with-baseline 的最优常数 baseline 形式,与你在策略梯度章见到的结论一致。

## 11. 小结与承上启下

这一章做的核心动作只有一个:**把 LLM agent 从"prompt 拼接技巧"重新装回"POMDP 上的序贯决策"框架。** 一旦装回去,很多模糊的工程直觉就有了第一性的依据——

- **形式化**:agent = 部分可观环境里的策略 `π_θ(a|h)`;上下文是 belief state 的有损编码;思考是显式 belief;记忆管理是 belief 的充分统计量压缩(2、7.5)。
- **范式统一**:ReAct(显式 belief 的闭环)、function calling(动作空间结构化约束)、plan-execute(HRL 分层缩短视野)、Reflexion(in-context 而非 in-weight 的策略迭代)——都是同一个 belief-MDP 上的不同求解手段(3)。
- **RL 的边界**:稀疏奖励让方差随视野线性涨、有效样本率 ∝ p;没有廉价 simulator + 样本贵,这三座山解释了为何 prompt/SFT 当下占主流;RL 的增量在"突破模仿上限"和"可验证奖励的 RLVR",判据是 4.4 口诀(4、5)。
- **训练技术**:reward-to-go 与 baseline 降方差的无偏性证明、双层 credit assignment、GRPO 式组内归一化、PPO-clip 与 KL 锚定(5、6)。
- **工程铁律**:单步可靠性被视野指数放大、错误是高价值观察、reward hacking 是头号敌人、成功率是主指标(7、8)。

**在整门课里的位置**:前面几章你已经有了策略梯度、PPO、RLHF、RLVR(可验证奖励训练推理模型)的硬基础——那些大多是"单步或短序列"的对齐与推理优化。这一章把它们**延拓到长视野、部分可观、带外部工具的序贯决策**,是从"训一个会推理的模型"走向"训一个会在真实环境里多步行动的智能体"的桥。

往后看:有了 agent = 决策问题这副眼镜,接下来自然要面对(后续章节)——长视野下的 credit assignment 与过程奖励怎么训得稳、多智能体协作与博弈、把人类反馈与可验证奖励混合的对齐配方、以及 agent 安全与对抗鲁棒性(reward hacking 只是冰山一角)。这些问题的共同点是:**它们只有在你把智能体当决策问题、而非当 prompt 时,才问得清楚、答得严谨。** 这正是这一章想留给你的思维底座。
