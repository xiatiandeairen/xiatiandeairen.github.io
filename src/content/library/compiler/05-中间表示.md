到第 04 章为止,我们手里那棵 AST 已经被类型检查盖过章、确认语义合法了。现在你可能想直接写个 tree-walking 解释器(遍历语法树执行),三百行搞定 fib 和 loopsum——能跑。但只要你打算做任何优化,或者把这门语言编译成字节码(第 07 章),tree-walking 这条路就会在第一个拐弯处撞墙。这一章干的事是:把那棵树**降低(lowering)**——把嵌套结构翻译成扁平的跳转拓扑——成一串三地址指令。降完之后,后端再也不认识 `if` / `while` / `&&` 这些语法概念,它只认识 `jump` / `cond_jump` / `binary`。

本章配套代码全在 `examples/compiler-from-scratch/src/stage05-ir.ts`,它不 import parser(stageNN 文件加载即跑 `main()`,严禁互相 import),而是用 `core/ast.ts` 的节点类型手工建出 fib 体和 loopsum 这两棵树——这恰好让降低的输入完全确定,对拍不受 parser 演进影响。下面所有数字都来自这份代码的真实运行,确定性的(指令条数、基本块数、回边数、副作用计数)我直接引用。

## ✦ 为什么不直接遍历 AST 执行

先回答这个最该被怼回去的问题:既然 AST 能跑,为什么还要多降一层 IR(intermediate representation,中间表示——介于源码和机器码之间的程序表示)?

答案是**优化和代码生成都讨厌树**。树的麻烦在于"一条计算的输入来自哪里"被嵌套结构藏起来了。看 fib 函数体那棵树打印成 S 表达式的样子:

```
((if (< n 2) ((return n))) (return (+ (call fib (- n 1)) (call fib (- n 2)))))
```

`(+ (call fib ...) (call fib ...))` 里,加法的两个操作数是两棵子树,你想知道"第二个 fib 调用的结果喂给了谁",得顺着父指针往上爬。常量折叠、死代码消除、寄存器分配,全都要反复回答"这个值从哪来、被谁用",在树上这是一趟趟递归遍历。

降成线性 IR 后,同一段函数体变成这样(stage 实跑输出,12 条指令):

```
  %t0 = n < 2
  cond_jump %t0 ? L0_then : L2_endif
L0_then:
  return n
  jump L2_endif
L2_endif:
  %t1 = n - 1
  %t2 = call fib(%t1)
  %t3 = n - 2
  %t4 = call fib(%t3)
  %t5 = %t2 + %t4
  return %t5
```

把这两种形态并排看,差别一眼就出来:IR 里每个中间结果都有名字(`%t0`..`%t5`),每条指令的输入要么是字面量、要么是一个具名的临时变量(temp)。`%t5 = %t2 + %t4` 这条加法,它的两个输入是哪条指令产出的?往上扫,`%t2` 来自上面那条 call,`%t4` 来自下面那条 call——线性扫描即可,不用爬树。这就是"三地址码"(three-address code)的本质:每条指令最多碰三个地址(两个操作数 + 一个结果),形如 `t = a op b`,复合表达式靠引入临时变量拆平。第 06 章的常量折叠/DCE 之所以能写得那么短,前提就是输入已经是这种"谁产出、谁消费一目了然"的线性形态。

降低这一步在代码里就是 `Lowerer` 类。它对表达式做的核心动作在 `lowerExpr`:叶子(字面量/变量)直接返回操作数不浪费 temp,复合表达式 emit 一条指令并返回新 temp。

```ts
private lowerExpr(e: Expr): Operand {
  switch (e.kind) {
    case "Literal":
      return { kind: "const", value: e.value };
    case "Var":
      return { kind: "temp", name: e.name }; // 具名变量在 IR 里和 temp 同构(都按名引用)
    case "Binary":
      return this.lowerBinary(e);
    case "Call": {
      const args = e.args.map((a) => this.lowerExpr(a));
      const callee = e.callee.kind === "Var" ? e.callee.name : "<expr-callee>";
      const dst = this.freshTemp();
      this.emit({ kind: "call", dst, callee, args });
      return { kind: "temp", name: dst };
    }
    // ...
  }
}
```

注意 `Var` 那行:**具名变量(sum / i / n)和临时变量在 IR 里同构**,都按名字引用。这是个刻意的简化——这一层不区分"用户写的变量"和"编译器造的中间值",后端统一当成"按名引用的值"处理。代价是这里还没做 SSA(静态单赋值,每个变量只被赋值一次)、没分配寄存器;loopsum 里 `sum` 会被反复 `copy` 覆盖。那些是第 06 章及更后端的活儿,本章只负责把树压平。

### 降低必须是确定性的

有个不显眼但要命的不变量:**同一棵树降两次,必须逐字节降出同样的 IR**。否则对拍(把输出当 golden 文件 diff)就没法做,你也没法判断一次重构有没有改变行为。

这要求 temp/label 计数器封在 `Lowerer` 实例里,而不是全局:

```ts
private freshTemp(): string {
  return `%t${this.tempCounter++}`;
}
private freshLabel(hint: string): string {
  return `L${this.labelCounter++}_${hint}`;
}
```

如果用全局计数器,跑 fib 再跑 loopsum,loopsum 的 temp 编号就会接着 fib 往后排,顺序一变编号就变,对拍立刻挂。stage 的 §1 末尾就跑了这个检查:同一棵树用两个独立 `Lowerer` 实例各降一次,断言 IR 文本逐字相同——输出里那行 `[对拍] 同一棵树降低两次, IR 文本逐字相同 (确定性 OK)` 就是它过了。这是个失败模式的预防:全局可变计数器是新手降低器最容易踩的坑,跑顺序污染编号,bug 还非确定性,极难复现。

## 控制流降低:if / while → label + cond_jump

表达式压平相对直白,真正把"嵌套"翻成"拓扑"的是控制流。`if` 和 `while` 在 AST 里是把子语句整块抱在身上的节点,在 IR 里它们必须摊开成标签 + 跳转。

`while` 的降低看 `lowerWhile`:

```ts
// while c { body }  →
//   Lcond: cond_jump c -> Lbody, Lend
//   Lbody: <body>; jump Lcond     ← 这条 jump Lcond 就是「回边(back-edge)」
//   Lend:
private lowerWhile(cond: Expr, body: readonly Stmt[]): void {
  const lCond = this.freshLabel("cond");
  const lBody = this.freshLabel("body");
  const lEnd = this.freshLabel("endwhile");
  this.emit({ kind: "label", name: lCond });
  const c = this.lowerExpr(cond);
  this.emit({ kind: "cond_jump", cond: c, ifTrue: lBody, ifFalse: lEnd });
  this.emit({ kind: "label", name: lBody });
  for (const s of body) this.lowerStmt(s);
  this.emit({ kind: "jump", target: lCond }); // back-edge
  this.emit({ kind: "label", name: lEnd });
}
```

loopsum 的 `while (i <= 100) { ... }` 降出来是这样(stage 实跑,整段 13 条指令):

```
L0_cond:
  %t0 = i <= 100
  cond_jump %t0 ? L1_body : L2_endwhile
L1_body:
  %t1 = sum + i
  sum = %t1
  %t2 = i + 1
  i = %t2
  jump L0_cond
L2_endwhile:
  print sum
```

关键是块尾那条 `jump L0_cond`——它跳回的目标在自己**前面**。这条边叫**回边(back-edge)**:目标位置在源位置之前的跳转。它是循环在 IR 上的**唯一**标志。源码里 `while` 是个关键字,降完之后关键字没了,后端怎么知道这是循环?就靠这条回边。后端识别循环、做循环不变量外提(把循环里每轮都算同一个值的计算提到循环外)等优化,起点都是先找回边。stage 的 §2 专门扫了这条边,输出 `回边(back-edge): B2(L1_body) → B1(L0_cond)`,并断言 loopsum 恰好 1 条回边(单个 while)。

`if` 同理,降成 `cond_jump c -> Lthen, Lelse` 加上 then 块尾的 `jump Lend`。回到本章开头 fib 的 IR,你能看到 `cond_jump %t0 ? L0_then : L2_endif`——没有 else 分支时,假分支直接跳 endif,省一个标签。

## ⚡ 基本块与 CFG:几乎所有优化的前提

光有线性指令还不够。优化要的是把指令切成**基本块(basic block)**——一段"要么全执行、要么全不执行"的指令:单入口(只能从块首进)、单出口(只能从块尾出),中间不含标签、不含跳转。切完块,块与块之间靠跳转连成**控制流图(CFG,control flow graph)**,节点是基本块,边是跳转关系。

为什么非要切块?因为**数据流分析在基本块粒度上才好做**。块内是直线代码,"这个值在块尾是什么"可以一趟扫出来;跨块的复杂性被收敛到 CFG 的边上。第 06 章的死代码消除(DCE)要回答"哪些块根本到不了"(不可达 = 没有任何前驱路径从入口到达它),这就要 CFG 的边;要回答"这个赋值后面还有人用吗",要顺着后继块走。没有 CFG,这些分析无从下手。

切块用教科书的 leader(块首)法,在 `buildCfg` 里:

```ts
// leader(块首)是: ① 第一条指令; ② 任何 label; ③ 任何跳转的「下一条」指令。
const leaders = new Set<number>();
if (instrs.length > 0) leaders.add(0);
for (let i = 0; i < instrs.length; i++) {
  const ins = instrs[i];
  if (ins.kind === "label") leaders.add(i);
  if (ins.kind === "jump" || ins.kind === "cond_jump" || ins.kind === "return") {
    if (i + 1 < instrs.length) leaders.add(i + 1); // terminator 的下一条是新块起点
  }
}
```

切完块再连边,块尾的指令类型决定后继:`jump L` 唯一后继是 `block(L)`;`cond_jump ?L:M` 两个后继;`return` 无后继(函数出口);其余(fall-through,顺序流入下一块)后继是下一个块。fib 体切出 4 个块(stage 实跑):

```
  B0 (no-label)  指令[0,2)  succ={B1,B3}  pred={—}
  B1 L0_then  指令[2,4)  succ={—}  pred={B0}
  B2 (no-label)  指令[4,5)  succ={B3}  pred={—}
  B3 L2_endif  指令[5,12)  succ={—}  pred={B0,B2}
```

这里有个值得停下来看的细节:**B2 的 pred 是空集 `{—}`**。B2 是 `return n` 后面、`L2_endif` 前面那条孤零零的 `jump L2_endif`(then 块尾的无条件跳转之后剩下的)。没有任何块的 succ 指向 B2——它**不可达**。这正是 DCE 将要清掉的东西:一个 pred 为空(且不是入口块)的基本块,执行流永远到不了,整块可删。stage 在这一章只是把它打印出来给你看见,真正的删除留给第 06 章——但你现在就能理解,为什么 DCE 必须先有 CFG:不可达的判定 = "从入口出发沿 succ 边走不到",这是个图可达性问题,没有 CFG 这张图根本提不出来。

loopsum 的 CFG 则展示了循环在图上长什么样:

```
  B1 L0_cond  指令[2,5)  succ={B2,B3}  pred={B0,B2}
  B2 L1_body  指令[5,11)  succ={B1}  pred={B1}
```

B1(条件块)的 pred 同时含 B0(初次进入)和 B2(每轮循环回来),B2 的 succ 指回 B1——这个"B1↔B2 互相是对方前驱/后继"的环,就是回边在 CFG 上的样子。

CFG 有个必须守住的不变量:**succ 和 pred 互为反向**,即 `b ∈ succ(a)` 当且仅当 `a ∈ pred(b)`。连边时一次性双向维护:

```ts
const link = (from: number, to: number): void => {
  if (!blocks[from].succ.includes(to)) blocks[from].succ.push(to);
  if (!blocks[to].pred.includes(from)) blocks[to].pred.push(from);
};
```

如果 succ/pred 不一致,后续所有图分析(前向 DCE 走 succ、后向活跃变量分析走 pred)就会看到两张矛盾的图,结果全错且难查。stage §3 末尾遍历两个 CFG 逐边校验这个不变量,输出 `[不变量] succ/pred 互为反向, 两个 CFG 均一致 (OK)`。

还有个失败模式埋在 `resolve` 里:跳转目标找不到对应的块(悬空 label)直接抛错。

```ts
const resolve = (label: string): number => {
  const id = labelToBlock.get(label);
  if (id === undefined) throw new Error(`CFG: jump to unknown label ${label}`);
  return id;
};
```

这是给降低器自己兜底——如果哪天 `lowerIf` / `lowerWhile` emit 了一个 `jump` 却忘了 emit 对应的 `label`,构建 CFG 时这里会立刻炸,而不是让一个指向虚空的跳转悄悄溜进第 07 章的字节码里变成运行时灾难。

⚡ **一个开放问题**:本章的 CFG 是"指令切块 + 连边"这种朴素表示,够 fib/loopsum 用。但工业编译器在 IR 表示上至今没有公认的统一答案。LLVM 用 SSA + CFG,这套体系做标量优化非常顺,可它对"循环嵌套结构""数据依赖距离"这类信息的表达并不直接——做激进的循环变换(分块、并行化)时,SSA-CFG 反而是阻碍。于是有了 polyhedral(多面体)模型、有了 MLIR 的多层 dialect、有了 e-graph(等价图,把"所有等价改写"塞进一张图里一次性挑最优,见 egg/Cranelift 的探索)。"什么是适合所有优化的中间表示"——标量优化、循环优化、自动并行、硬件映射各自偏爱的结构互相打架——**目前没有通用解,仍是活跃研究方向**。你这一章手搓的线性 IR + CFG,是这场争论里最古老也最经得起用的那个起点。

## 降低展开比:别被"IR 一定膨胀"的直觉骗了

直觉上总觉得"降低 = 拆细 = 指令变多"。stage §4 实测了 AST 节点数对 IR 指令数的比值,结果反直觉:

```
  fib       AST 18 节点 → IR 12 条指令  (展开比 0.67x)
  loopsum   AST 20 节点 → IR 13 条指令  (展开比 0.65x)
```

两个例子展开比都**小于 1**——IR 指令反而比 AST 节点少。原因在于三地址 IR 把字面量、变量这些叶子收进了"操作数"而非独立指令:AST 里每个 `n` / `1` / `2` 都算一个节点,但在 IR 里它们只是某条 `binary` 的操作数,不占指令。反方向的膨胀来自控制流——一个 `if` / `while` 节点会摊成 `cond_jump` + 多个 `label` + `jump`。所以规律是:**叶子密集的程序展开比趋近 1,分支/循环密集的程序展开比大于 1**。

这里要诚实标注:这两个绝对数字偏 toy,且依赖口径——"AST 节点怎么数""IR 算不算 label 行"换个数法就变。可迁移的不是 `0.67` 这个值,而是相对趋势:叶子越多越接近 1,控制流越多越超过 1。拿这个去估真实程序的 IR 规模会翻车,但理解"IR 不必然膨胀、膨胀来自控制流而非表达式"这个判断是站得住的。

## 失败模式:短路 `&&` 不能当普通二元算符降低

最后一个失败模式最有教益,因为它是"语义"在降低层被悄悄改掉的典型——而且能用一个数字实锤。

考虑 `false && sideEffect()`。短路语义要求:左侧是 `false`,右侧 `sideEffect()` **不该被求值**,副作用次数应为 0。正确的降低必须用 `cond_jump` 把右侧跳过去(`lowerShortCircuit`):

```ts
// a && b  →  t = a; cond_jump t -> Lrhs, Lend(t 已是 false, 不算 b)
//             Lrhs: t = b
//             Lend: (结果在 t)
```

降出来 7 条指令,右侧的 `call sideEffect()` 落在 `L0_rhs:` 块里,只有跳进来才执行:

```
  %t0 = #f
  cond_jump %t0 ? L0_rhs : L1_scend
L0_rhs:
  %t1 = call sideEffect()
  %t0 = %t1
  jump L1_scend
L1_scend:
```

而如果偷懒,把 `&&` 当成普通二元算符降低——"左右都先算成 temp,再做一条 binary 合并"——就成了这样(2 条指令):

```
  %t0 = call sideEffect()
  %t1 = #f && %t0
```

问题一眼可见:右侧的 `call sideEffect()` 被无条件求值了,而且排在最前面。那条 `binary` 只是事后把两个布尔合并,救不回已经发生的副作用。

stage 写了个极简 IR 解释器,用一个带副作用计数的 `sideEffect()` 真跑两种降低,把语义差变成可打印的数字:

```
  正确降低: sideEffect() 实际执行 0 次  (期望 0)
  错误降低: sideEffect() 实际执行 1 次  (短路语义被破坏!)
```

这个 `0 次 vs 1 次` 就是实锤——不是嘴上说"右边会被算",是真跑出来数的。它要传达的判断是:**短路必须在 IR 层用 `cond_jump` 表达,不能留到运算符层**。一旦降到 `%t1 = a && b` 这种单条指令,语义就已经错了,后面任何优化、任何执行器都救不回来。降低层是语义被翻译的最后一道关,这种"控制流伪装成表达式"的结构(短路 `&&` / `||`、三元 `?:`、`a ?? b`)都必须在这一层就拆成跳转,这是降低器的核心职责而非可选项。

到这里,这棵被类型检查认证过的 AST 已经变成一串带基本块和 CFG 的线性指令。它不再关心源语言长什么样——`if`、`while`、`&&` 全成了跳转和标签。第 06 章会在这个线性形态上做常量折叠和死代码消除(你已经在 fib 的 B2 那个不可达块里见过 DCE 的猎物了),第 07 章则把它翻译成可在栈式虚拟机上执行的字节码。"语言长什么样"和"程序怎么执行",从这一章起彻底分家。
