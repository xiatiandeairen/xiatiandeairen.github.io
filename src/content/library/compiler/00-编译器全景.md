先看一个具体的数字，它能把这本书要讲的事一句话说清。同一段斐波那契程序 `fib(25)`，在我们的实现里跑两遍：一遍用「树遍历解释器」（直接在语法树上递归求值），一遍先编译成字节码再用「虚拟机」（一个软件模拟的简单 CPU，循环取指执行）跑。`examples/compiler-from-scratch/src/stage07-vm.ts` 的实测输出是：

```
树遍历访问 AST 节点(操作)数: 2,670,636
VM 执行字节码指令(操作)数:   2,670,639
操作数比(树/VM): 1.00x
⇒ 字节码 VM 加速比 = 2.63x（相对量，可复现断言: >1x）
```

两种模型干的「操作数」几乎一样多（2,670,636 比 2,670,639），可字节码 VM 快了 2.63 倍。赢点不在「做的事更少」，而在「每件事更便宜」——扁平的指令循环比递归下降的树遍历每步开销低。这就是本书的底色：不背术语，每个机制都落到一段能跑的代码、一个能复现的数字、一个会坏的边界上。从「一段文本」到「2.63x 加速」中间隔着七道关，下面就是这张地图。

> 数字诚实声明：这是 toy 程序 + JS 跑 JS 的解释，绝对毫秒偏乐观。本书所有「正文断言」只用**相对量**（加速比 / 指令计数），它们是确定性、可复现的；绝对毫秒只作展示并标注「本机实测」。这条纪律写在 `examples/compiler-from-scratch/src/core/bench.ts` 的文件头注释里，是全书量化输出的唯一出口。

## 一、编译器不是一个黑盒，是一条流水线

初学者脑子里的编译器是个魔法盒子：源码进去，可执行文件出来。这个模型的问题是它没法调试——盒子里出错你无从下手。专业的心智模型是一条**流水线（pipeline）**：每一道关是一个独立的变换，吃上一关的产物，吐下一关的输入。本书把它切成七道关（外加第八关 codegen 落地到机器），逐章实现：

```
源码文本
  │  ① 扫描 / 词法分析 (Lexer)      文本 → Token 流
  ▼
Token 流
  │  ② 解析 / 语法分析 (Parser)     Token → AST(语法树)
  ▼
AST
  │  ③ 名称解析 (Resolver)          绑定变量到作用域 / 算闭包深度
  ▼
带绑定信息的 AST
  │  ④ 类型检查 (Type check)         编译期挡住类型错
  ▼
通过检查的 AST
  │  ⑤ 中间表示 (IR lowering)        AST → 线性三地址码 + 基本块
  ▼
IR
  │  ⑥ 优化 (Optimize)              常量折叠 / 死代码消除
  ▼
优化后 IR
  │  ⑦ 虚拟机 (VM) ─或─ ⑧ 代码生成 (Codegen)
  ▼
执行结果 / 机器码
```

每一关都有清晰的输入输出契约，这带来三个工程上的好处。

**第一，可以单独测、单独坏。** 词法器只管「把 `>=` 切成一个 token 还是两个」，跟「`if` 条件该不该是 bool」毫无关系。`src/stage01-lexer.ts` 的实测产出：合法样例 `fib.lox` 扫出 40 个 token、0 错误；故意写坏的样例一次扫描收集到 2 条错误：

```
bad.lox:1:11: error[lexer]: unexpected character '@'
bad.lox:3:9: error[lexer]: unterminated string literal
```

注意它**一次报全 2 条**，不是「首错即停」。这是专业编译器和玩具编译器的第一个分水岭，下面第三节会讲为什么。

**第二，相邻两关之间的数据结构是稳定契约。** Token 的定义不在词法器里，而在 `src/core/token.ts`——词法器产出它、解析器消费它，两者对同一套 `TokenKind` 枚举达成一致。AST 同理，定义在 `src/core/ast.ts`，是「语法器的产出、也是语义/类型/IR 阶段的共同输入」，全书只有一处定义。这就是为什么本书的 `core/` 目录是「全书共享底座」：把跨关共用的数据结构和工具抽出来，stage 文件之间**禁止互相 import**（每个 stage 文件加载即跑自己的 demo，互相 import 会触发别人的 main）。

**第三，每一关的「变形」可以量化。** AST 降到 IR（intermediate representation，中间表示，介于语法树和机器码之间的线性形式）时，`src/stage05-ir.ts` 实测：`fib` 的 18 个 AST 节点降成 12 条 IR 指令（展开比 0.67x），`loopsum` 的 20 个节点降成 13 条指令（0.65x）。这里指令比节点**少**而非多，原因很具体：三地址 IR 把字面量、变量这些叶子收进「操作数」，不再是独立指令；而控制流反过来会膨胀——一个 `while` 节点摊成 `cond_jump` + 多个 label。能把「为什么这里反直觉地变少了」讲到这个颗粒度，正是本书想替代 survey 稿的地方。

### ⚡ 把「解释器」和「编译器」当对立概念，是最大的误解

教科书常把世界一分为二：编译器把源码翻译成机器码（C、Rust），解释器逐句执行源码（早期 BASIC）。于是学生以为要在「编译」和「解释」之间二选一。这是初学者最深的一个坑。

**现代实现几乎都是混合体。** CPython 把 Python 编译成字节码（`.pyc`）再用一个解释器循环执行字节码；Java 先 `javac` 编译成 JVM 字节码，JVM 再解释 + JIT；V8 跑 JavaScript 也是「解析成 AST → 降到字节码 → 解释 → 热点 JIT」。它们都既「编译」又「解释」。本书走的正是这条主流路线：**解析成 AST → 降到字节码 → VM 执行字节码**（第 02、05、07 章），而不是教科书式的「直接在 AST 上递归求值」。

为什么主流选这条路，上面那个 2.63x 已经给了答案的一半。另一半在 `stage07` 的失败模式 demo 里：树遍历解释器每求一个值都要在对象树上跳指针、查 `kind` 字段分发，缓存不友好；字节码是一段连续数组，VM 一个 `while` 循环顺序取指，分支预测和缓存都更友好。「直接遍历 AST」不是错，它是 walking interpreter，简单、好写、适合教学第一版；但一旦在乎性能，几乎都会再降一层到字节码。这本书两层都写，让你亲眼看到差距，而不是记住一句「字节码更快」。

> ⚡ **开放问题**：解释执行 vs 编译执行的边界正在被 JIT（just-in-time，运行时即时编译）和 AOT（ahead-of-time，提前编译）持续模糊，**目前没有「一种实现一统天下」的通用解**。PyPy 用 meta-tracing JIT 给 Python 提速，CPython 3.13 起也在做实验性 JIT；而 GraalVM 又能把同一份 Java 字节码 AOT 成原生镜像。「该在编译期做多少、运行期做多少」是按工作负载权衡（启动延迟 vs 峰值吞吐 vs 内存），没有放之四海皆准的最优切分点——这是编译器与运行时领域至今活跃的研究方向。

## 二、为什么只做一门小语言（Lox-mini）

很多编译器教程的通病是贪：想覆盖泛型、模式匹配、异常、协程……结果每个特性都讲半页，每个都不到能跑的程度，读者学完一个完整的「假编译器」。本书反过来——**窄到极致，但每一关都是真的、能跑的**。

我们固定一门语言叫 **Lox-mini**（脱胎自 Robert Nystrom《Crafting Interpreters》里的 Lox，再砍一刀）。它只有：整数/浮点/布尔/字符串、`let` 声明与赋值、`if`/`while`、一等函数 + 闭包、`print`。语法关键字总共九个，全表在 `src/core/token.ts::KEYWORDS` 里：

```ts
export const KEYWORDS: ReadonlyMap<string, TokenKind> = new Map([
  ["let", TokenKind.Let],
  ["fn", TokenKind.Fn],
  ["if", TokenKind.If],
  ["else", TokenKind.Else],
  ["while", TokenKind.While],
  ["print", TokenKind.Print],
  ["return", TokenKind.Return],
  ["true", TokenKind.True],
  ["false", TokenKind.False],
]);
```

九个关键字看着寒酸，但它**真能跑出有意思的程序**。全书共用四个样例（在 `examples/compiler-from-scratch/examples/programs/`），每个都精心挑过，覆盖一类机制：

- `fib.lox`——斐波那契递归，考函数调用 + 递归 + 条件。它是全书的性能基准：第 6 章拿它比优化前后指令数，第 7 章数它执行了多少条指令。
- `counter.lox`——闭包计数器，考一等函数 + 词法闭包 + 捕获**可变**变量。这是「小语言」里最难的一块，逼着第 3 章的 resolver 必须把内层 `count` 正确绑到外层作用域。
- `loopsum.lox`——`while` 循环求和 1 到 100，考赋值 + 算术，是第 6 章常量折叠的试验田。
- `bad-type.lox`——**故意写错**的样例，给第 4 章类型检查演示失败模式。

闭包为什么是试金石？看 `counter.lox`：`makeCounter` 返回内层函数 `next`，`next` 捕获并修改外层的 `count`，每次调用 `c()` 输出 1、2、3。`src/stage07-vm.ts` 实测对拍通过：

```
3 次调用 c() 输出: 1, 2, 3
对拍通过: 闭包状态跨调用保持，counter == [1,2,3] ✓
→ upvalue 'count' 被 next 捕获，SET_UPVALUE 每次回写，旧值 +1
```

一门语言只要能正确实现「函数返回后，它捕获的局部变量还活着且可改」，它就触及了作用域、生命周期、堆分配、闭包捕获这一整套真问题。覆盖十个半吊子特性，不如把闭包这一个讲到 upvalue 回写的指令级别——**窄而透，比宽而浅长见识**。这也是本书对「AI 批量生成的 survey 稿」的正面回应：survey 求全，本书求透。

这门语言还有个工程纪律值得说：它**离线、零运行时依赖**，只用 Node 内置的 `fs`/`path`/`url` 读样例文件，不需要任何 LLM、API key 或网络。所有打印的量化数字都是代码真测、真数出来的。你 `git clone` 下来 `npm install && npm run stage07` 就能复现上面每一个数字。

## 三、衡量编译器的四把尺子（钉在 core/bench.ts 里）

「我写了个编译器」是个空话，除非你能回答：它解析多快？优化省了多少？运行时加速多少？类型检查抓住了多少错？本书把这四把尺子钉成代码，量化输出统一走 `src/core/bench.ts`——它是「全书量化输出的唯一出口」，集中一处保证测速口径一致、打印格式一致、对拍工具统一。下面逐把尺子给真实数字。

### 尺子一：解析吞吐（tokens/s, nodes/s）

`src/stage01-lexer.ts` 实测词法吞吐 **23,521,637 tokens/s**（本机实测），`src/stage02-parser.ts` 实测解析吞吐 **32,398,975 nodes/s**（9,600 个 AST 节点，确定性计数）。吞吐衡量「前端能多快把文本嚼成结构」。注意吞吐数（绝对值）是展示性的、依赖机器；节点数 9,600 是**确定性**的，同一输入恒等，可以直接断言。这正是 `bench.ts` 强制的纪律——把「可复现的计数」和「依赖机器的毫秒」分开对待。

### 尺子二：优化前后指令数

这是编译器优化的硬通货。`src/stage06-optimize.ts` 拿一段全是可折叠常量的程序做端到端 demo，实测：

```
静态指令数: 20,001 → 2 (优化 2 轮)
VM 执行条数: 20,001 → 2
指令削减率 = 100.0% (确定性, 可断言)
运行时加速比 = 6638.5x (相对量, 可复现)
```

6638.5x 看着夸张，作者自己在代码里就标了：「这是 toy IR + toy VM 上的合成数据，绝对加速比偏乐观（真实程序不会全是可折叠常量）」。这种**主动标注乐观偏差**的诚实，就是本书想立的规矩。更有教育意义的是它的「不动点」实测：死代码消除（DCE）必须**逐轮跑到不动点**才剥得干净——

```
指令数: 原始 6 → 只跑一遍 5 → 不动点 3
只跑一遍比不动点多留 2 条死指令
(死代码链 d3←d2←d1: 删一层才暴露下一层，需要逐轮 DCE 才剥得干净)
```

「只跑一遍会漏」这件事，背术语永远学不到，跑一遍代码就懂了。

### 尺子三：运行时加速（解释 vs 编译）

第一节那个 2.63x（字节码 VM vs 树遍历，`stage07`）就是这把尺子。第 8 章 codegen 把它推到极致：`src/stage08-codegen.ts` 生成寄存器机的直线机器码，实测 **23,229x 加速比**（直线机器码 vs 树遍历解释，本机）。同样标注了「绝对值偏乐观」。这条尺子量的是「降得越低，跑得越快」的整体趋势。

### 尺子四：类型检查抓错率

前端能挡多少运行时错？`src/stage04-typecheck.ts` 准备了一组坏样例，逐个验证是否被抓到：

```
[字符串 + 数字]  应抓 1 / 实抓 1
[调用非函数]  应抓 1 / 实抓 1
[参数个数不符]  应抓 1 / 实抓 1
[参数类型不符]  应抓 1 / 实抓 1
[if 条件非 bool]  应抓 1 / 实抓 1
[两条 return 类型不一致]  应抓 1 / 实抓 1
  命中率：抓到 6 / 应抓 6 = 100.0%
```

抓错率把「类型系统有没有用」变成一个可测的数。`bad-type.lox` 里那行 `let bad = name + age;`（字符串加数字），第 4 章会精确报到列：

```
bad-type.lox:7:11: error[typecheck]: cannot apply '+' to string and int
  let bad = name + age;   // error: cannot add string and number
```

四把尺子合起来，就是这本书评价每个 stage 的统一标尺。它们不是事后补的 benchmark，而是每章 demo 自带的产出。

## 四、失败模式优先：每个机制先讲它怎么坏

本书有条贯穿全书的写法：讲一个机制，先讲它**怎么坏、为什么这么设计**，再讲它怎么对。happy path 谁都会写，工程价值在边界。举两个本章已经露头的例子。

**例一：词法/语法的「一次报全」vs「首错即停」。** 新手编译器最常见的毛病是到处 `throw new Error("bad token")`——错误散乱、没位置、首错即停，用户改一个错重编一次只看见下一个。专业编译器都是「收集多个错误一次报全」+「每条带源码下划线」。本书把这件事做成全书统一基础设施 `src/core/diagnostics.ts`：各阶段只管 `bag.error(msg, span)`，**不抛异常、不做控制流**，最后用 `report(source)` 渲染成带行列下划线的文本。`stage01` 一次扫出 2 条错误、`stage02` 的 parser 用 panic-mode（恐慌模式，遇错跳到下一个同步点继续）一次报 3 条，靠的都是它。

**例二：栈泄漏导致的栈溢出。** `src/stage07-vm.ts` 专门演示一个「函数返回后忘记 POP 返回值」的坏字节码，逐步加大递归深度观测操作数栈峰值：

```
递归深度 depth | 操作数栈峰值高度 | 结果
            5 |                6 | OK
          500 |              501 | OK
       120000 |     >100,000(越界) | ✗ STACK OVERFLOW
```

栈峰值约等于递归深度、线性增长，最终越界。对比正确实现：`fib(20)` 操作数栈峰值**只有 13 槽**，与调用次数无关。差别就在 VM 是否在每次 `CALL` 返回后 POP 掉用不到的返回值，维持「语句净栈高度 = 0」这个不变量。把「正确」和「坏掉」并排放，机制为什么这么设计就不言自明了。第 3 章 resolver 也用同样手法对比：单遍解析器漏报自引用 `let a = a;`（报 0 个错，应为 1），两阶段解析器才抓到——「为什么需要两遍」用反例讲，比讲定义清楚得多。

## 五、这本书与一份就业的链路

### ✦ 语言/工具链岗的面试真相

如果你冲着「编译器工程师 / 语言运行时 / 工具链」这类岗位，有个反直觉的真相：**面试官不在乎你背了多少文法术语**（LL、LR、first/follow 集这些，需要时查），在乎的是你能不能把「一段文本变成可执行行为」这条链每一环讲到代码级——token 怎么切、AST 怎么建、作用域怎么绑、IR 长什么样、VM 怎么取指、优化删了哪条指令。

这恰好是本书的结构。第 01 到 08 章，每一章是这条流水线上的一个 stage，都给你一个**能跑的实现 + 真实数字 + 一个失败模式**：

- 第 01 章 词法分析——`src/stage01-lexer.ts`，token 切分、最长匹配、错误定位。
- 第 02 章 语法分析——`src/stage02-parser.ts`，递归下降 + Pratt 解析（按运算符优先级解析表达式）、panic-mode 错误恢复。
- 第 03 章 名称解析——`src/stage03-resolver.ts`，两阶段作用域、闭包深度回填、自引用检测。
- 第 04 章 类型检查——`src/stage04-typecheck.ts`，类型推断、抓错率、error 哨兵止级联。
- 第 05 章 中间表示——`src/stage05-ir.ts`，三地址码、基本块、控制流图与回边。
- 第 06 章 优化——`src/stage06-optimize.ts`，常量折叠、死代码消除、不动点迭代。
- 第 07 章 虚拟机——`src/stage07-vm.ts`，栈式字节码 VM、闭包 upvalue、指令计数。
- 第 08 章 代码生成——`src/stage08-codegen.ts`，寄存器分配、spill（寄存器不够时溢出到内存）、就业冲刺。

它们共用 `src/core/` 的 token / ast / diagnostics / bench，在**同一组样例**（`fib` / `counter` / `loopsum` / `bad-type`）上逐章推进。这意味着你可以拿第 2 章产出的 AST、第 3 章 resolve 后的 AST 直接 `astToSExpr` dump 成确定性字符串、`assertEq` 对拍——前后章是咬合的，不是各讲各的。面试时你能把同一个 `fib` 从文本一路讲到 240,805 条 VM 指令（`stage07` 实测 `fib(20)` 执行指令数），这比任何术语堆砌都有说服力。
