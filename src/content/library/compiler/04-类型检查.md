> 流水线第四关。给第 03 章解析+解析名字后的 AST 加一套最小静态类型系统 (int/float/bool/string/fn 签名):自底向上推导每个表达式的类型,检查赋值、调用、算符的类型相容性,校验函数多条 return 同型。通过检查的程序才进入第 05 章 IR 生成——从此后端可以假设类型已知,不必再防 `"a" + 1` 这类运行时崩溃。

先把这章在做什么用一句话钉死:**类型检查的本质是「不运行程序,就证明某一类错误不会在运行期发生」。** `"a" + 1`、给一个 bool 当函数调用、给两参数函数只传一个参数——这些在 Python / JavaScript 里都是要等到那行代码真正执行才炸的运行时错误;在我们这个有静态类型的 Lox-mini 里,它们是编译期的红线,程序根本进不了第 05 章。这不是风格偏好,是把「崩溃发生的时刻」从用户的运行期,前移到你的编译期——前移得越早,挡住的代价越低。

这章的代码全在 `examples/compiler-from-scratch/src/stage04-typecheck.ts`,一个文件自包含。它内置了一个够用就好的 lexer + Pratt parser(只为把样例源码喂成 AST,刻意写薄),真正的主角是后半段的 `TypeChecker`。下面我先讲它怎么算对、再讲它怎么坏,最后给一个目前没有通用解的开放问题。

## 类型检查器在算什么:自底向上,每个表达式一个类型

类型检查的骨架朴素到有点反高潮:**递归遍历 AST,给每个表达式节点算出一个类型,然后在父节点上检查子节点的类型组合是否合法。** 没有魔法,就是一个后序遍历(post-order,先算孩子再算自己)。

类型本身是一个判别联合(discriminated union,用一个 `kind` 字段区分变体的联合类型):

```typescript
type Type =
  | { readonly kind: "int" }
  | { readonly kind: "float" }
  | { readonly kind: "bool" }
  | { readonly kind: "string" }
  | { readonly kind: "nil" }
  | { readonly kind: "error" } // 哨兵:抑制级联误报
  | { readonly kind: "fn"; readonly params: readonly Type[]; readonly ret: Type };
```

注意这里就 7 个变体,`fn` 是唯一的复合类型——它带参数类型列表和返回类型,因为只有它需要在调用点被拆开检查(实参对不对、个数对不对)。这是 toy 语言的诚实:没有泛型、没有 union、没有结构体。但「自底向上推一个类型、在父节点检查组合」这个机制,在 Rust 的 `rustc`、TypeScript 的 `tsc` 里是一模一样的,只是它们的 `Type` 联合有几十上百个变体。

检查二元运算是这个机制最清楚的缩影。看 `checkBinary`:

```typescript
const l = this.checkExpr(expr.left, scope);   // 先算左孩子的类型
const r = this.checkExpr(expr.right, scope);  // 再算右孩子的类型
if (l.kind === "error" || r.kind === "error") return T_ERROR; // 止血(后面讲)
const op = expr.op;

if (op === "+" || op === "-" || op === "*" || op === "/") {
  // '+' 同时是字符串拼接:两侧都是 string 则结果 string。
  if (op === "+" && l.kind === "string" && r.kind === "string") return T_STRING;
  if (isNumeric(l) && isNumeric(r)) return numericResult(l, r);
  this.diags.error(
    `cannot apply '${op}' to ${typeToString(l)} and ${typeToString(r)}`,
    expr.span, "typecheck",
  );
  return T_ERROR;
}
```

先递归算出两个孩子的类型,再用一张「什么操作数组合产出什么结果」的规则表去匹配。`"a" + 1` 走到这里:左是 string、右是 int,既不是「两侧都 string」也不是「两侧都 numeric」,直接落到那条 `diags.error`,精确报到 `expr.span`(这个 span 是第 01 章 lexer 一路带下来的源码位置)。stage 实测里这条规则抓的就是:

```
[字符串 + 数字]  应抓 1 / 实抓 1
  源码: let r = "a" + 1;
  → 1:9 cannot apply '+' to string and int
```

报错带行列(`1:9`),不是一句模糊的「类型错误」。这点很重要——类型检查器的产品质量,一半在于规则对不对,另一半在于诊断能不能让人五秒钟定位。后面会展示它甚至能在源码上画下划线。

## ✦ 真实坏样例:六类典型类型错,命中率 100%

光说能挡没用,得拿坏样例数着抓。stage 内置了一组覆盖六类典型类型错的坏样例,每条标注「应抓几个」,跑完算命中率。这是 toy 但诚实的做法:不是「我相信它能抓」,是数字摆出来。实测输出:

```
③ 坏样例集:精确诊断 + 抓错命中率

[字符串 + 数字]  应抓 1 / 实抓 1
  → 1:9 cannot apply '+' to string and int
[调用非函数]  应抓 1 / 实抓 1
  源码: let n = 3; let r = n(5);
  → 1:20 cannot call non-function of type int
[参数个数不符]  应抓 1 / 实抓 1
  源码: fn f(a, b) { return a + b; } let r = f(1);
  → 1:38 expected 2 argument(s), got 1
[参数类型不符]  应抓 1 / 实抓 1
  源码: fn f(a) { return a + 1; } let r = f("x");
  → 1:37 argument 1: expected int, got string
[if 条件非 bool]  应抓 1 / 实抓 1
  源码: let n = 5; if (n) { print n; }
  → 1:16 if condition must be bool, got int
[两条 return 类型不一致]  应抓 1 / 实抓 1
  源码: fn f(a) { if (a < 1) { return 1; } return "big"; }
  → 1:1 function has inconsistent return types: int vs string

  命中率:抓到 6 / 应抓 6 = 100.0%
```

把这六类挨个对应到三种动态语言里的崩溃:

- **调用非函数** (`n(5)` 其中 `n` 是 int):JS 里这是运行期 `TypeError: n is not a function`。`checkCall` 先算 callee 类型,发现 `callee.kind !== "fn"`,当场报错。
- **参数个数不符** (`f(1)` 给两参函数):JS 里多余参数被忽略、缺的参数变 `undefined`,然后在函数体里某处算出 `NaN` 或 `undefined.foo` 才炸——错误位置离根因十万八千里。这里直接报在调用点:`expected 2 argument(s), got 1`。
- **if 条件非 bool** (`if (n)` 其中 `n` 是 int):这是动态语言「真值性 (truthiness)」的温床。Lox-mini 不接受隐式真值——`expectBoolCond` 要求条件严格是 `bool`。这是个设计取舍:更安全,但也更啰嗦(不能写 `if (count)` 偷懒)。

每一类的共性是:**动态语言里它们都活到运行期、且报错位置往往漂移到离根因很远的地方**;静态类型把它们钉死在编译期、钉死在出错的那个 token 上。

诊断渲染不是裸文本,而是能在源码上画下划线。stage 跑一个真实多错文件 `bad-type.lox`,输出:

```
③' 一条诊断的完整渲染(带源码下划线)
bad-type.lox 共报 2 个错误:
bad-type.lox:7:11: error[typecheck]: cannot apply '+' to string and int
  let bad = name + age;   // error: cannot add string and number
            ^^^^^^^^^^
bad-type.lox:9:5: error[typecheck]: if condition must be bool, got int
  if (age) {              // error: condition must be bool, got number
      ^^^
```

这个下划线靠的是诊断里携带的 `span`(起止偏移量),由第 01 章的 `SourceText.offsetToLineCol` 翻成行列再画 caret。整套诊断基础设施是跨 stage 共享的(`core/diagnostics.ts`),类型检查只负责往里塞条目。

### 为什么要有 `error` 哨兵类型:止住级联误报

上面代码里 `if (l.kind === "error" || r.kind === "error") return T_ERROR;` 这行看着多余,其实是类型检查器能不能用的分水岭。

设想一个未声明的变量 `x`,后面被用了五次:`x + 1`、`x * 2`、`f(x)`……如果 `x` 第一次被查时报「undefined variable」之后返回一个具体类型(比如硬当 int),那后面五处全都正常参与运算、各自又可能报错,一个根因炸出六条诊断,真错被淹没。这叫级联误报(cascading errors)。

设计就是:**任何报过错的位置,类型变成 `error` 哨兵;`error` 与任何类型「相等」,且碰到 `error` 的运算一律不再报新错。** 看 `typeEquals` 的第一行:

```typescript
function typeEquals(a: Type, b: Type): boolean {
  if (a.kind === "error" || b.kind === "error") return true; // 哨兵:已报过错,不再制造新错
  ...
}
```

`error` 像一滴止血药,顺着表达式树往上传,把所有下游误报吸收掉。这是真实编译器都有的机制(GCC/Clang 叫 error recovery)——一个 toy 检查器漏掉它,用户改一个 typo 就被几十条假错刷屏。可迁移的机制就在这:**报错后用哨兵类型续传,而不是中断、也不是硬塞一个真类型。**

## ⚡ 局部类型推断:哪些位置不用标注,哪些位置推不出来必须标

类型系统最招人烦的地方是「到处写类型」。现代语言的解法是类型推断——能从上下文推出来的,就别让人写。这章实现了最简单的一档:局部类型推断(local type inference,只看赋值右边推左边),演示「有标注就检查、无标注就推导」。

看 `let` 语句的处理,核心就两行:

```typescript
case "Let": {
  // 局部类型推断:变量类型 = 初始化表达式的类型,无需标注。
  const t = this.checkExpr(stmt.init, scope);
  scope.define(stmt.name, t);
  return;
}
```

`let x = 1 + 2;` 不用写 `int`,因为右边 `1 + 2` 自己能算出 int,左边照抄。stage 实测这套推断在好样例上全部正确并对拍通过:

```
① 好样例:顶层声明的推导类型(局部推断,无需标注)
fib.lox:
  fib : fn(int) -> int
counter.lox:
  makeCounter : fn() -> fn() -> int
  c : fn() -> int
loopsum.lox:
  sum : int
  i : int
  [对拍] 所有好样例的顶层类型推断 == 期望  ✓
```

`makeCounter` 推成 `fn() -> fn() -> int`——一个返回「返回 int 的函数」的函数,闭包的类型也是从函数体里 return 的东西反推出来的,全程没人写过一个类型标注。

int/float 的区分是局部推断里一个有意思的细节。stage 实测:

```
② 局部类型推断:int / float 区分 + 隐式提升
  let x = 1 + 2;             → int    (期望 int) ✓
  let y = 1.0 * 2;           → float  (期望 float) ✓
  let z = 3.5 + 1.5;         → float  (期望 float) ✓
  let q = 10 / 2 + 100;      → int    (期望 int) ✓
  let mixed = 2 * 3.0 - 1;   → float  (期望 float) ✓
  说明:int 字面量无小数点推 int;任一侧 float 则整式提升为 float。
```

规则是 C 系数值语言的惯例:任一侧是 float,整式提升为 float;全 int 才是 int(`numericResult` 实现)。这里有个跨层的坑值得记:第 01 章 core 的 token 把整数和浮点都存成 JS `number`,丢了「有没有小数点」这个区分。所以 stage04 的内置 lexer 单独记了一个 `numIsFloat` 标志旁路携带——`checkLiteral` 靠它区分 int/float。**这是教训:上游一个看似无害的「都用一种内部表示」决策,会在下游某个 stage 突然需要被区分的地方变成债。**

### ⚡ 推断推不出来的地方:必须强制标注——而这正是 toy 的天花板

局部类型推断的「局部」是有代价的:它只看赋值右边,看不到更远。stage 里有一个被诚实标注出来的局限,藏在 `checkFn` 的注释里:

```typescript
/**
 * 简化(教学诚实标注):本章没有参数类型标注语法,无法对参数做真正的类型推断 ——
 * 参数一律按 int 处理(Lox-mini 主样例 fib 的参数确实是 int)。这是本类型系统的已知
 * 局限:真正的方案需要 Hindley-Milner 全局推断或显式参数标注,放到后续章节。
 */
private checkFn(expr, scope): Type {
  const paramTypes: Type[] = expr.params.map(() => T_INT); // 局限见 doc:参数默认 int
  ...
}
```

这就是「推不出来必须标注」的活样本。函数参数 `fn f(a) {...}` 里的 `a` 是什么类型?局部推断看不出来——`a` 的类型取决于「将来谁来调 `f`、传什么进来」,这是从使用点反推到定义点的信息流,局部推断根本不沿这个方向看。

真实语言在这里只有两条路,没有第三条:

1. **强制标注**:像 Java / Go / 早期 Rust,参数必须写类型 `fn f(a: int)`。简单、错误信息好、IDE 友好,代价是啰嗦。
2. **全局类型推断**:像 Haskell / ML 的 Hindley-Milner 算法,通过「在所有使用点收集约束、统一求解」把参数类型也推出来,连函数签名都能省。强大,但实现复杂、错误信息出了名地难懂(一个 typo 能让 HM 在离根因很远的地方报一个莫名其妙的「无法统一」)。

本章选了最 toy 的第三种:**假装参数都是 int**。这对主样例 `fib`(参数确实是 int)能跑,但它是个明确的天花板——`stage` 把它写进注释而不是假装支持。空函数体的返回类型同理:无 return 推成 `nil`(`resolveReturnType` 里 `returns.length === 0` 返回 `T_NIL`),但如果你想要一个「返回类型由调用方决定」的泛型函数,局部推断给不了。

⚡ **这里有个目前没有通用满意解的开放问题:类型推断的「能力」和「错误信息质量」是一对几乎对立的指标。** 推断越强(越接近全局 HM),你写的标注越少,但一旦出错,编译器越难指出「错在哪、为什么」——因为约束是全局求解的,错误可能在任何一个参与约束的位置爆出来,而不在你写错的那行。Rust 团队为此投入巨大精力做「推断局部化 + 边界标注」的折中,TypeScript 在「上下文类型 (contextual typing)」上反复迭代了十年,至今没有「既几乎不用标注、错误信息又总是精确指向根因」的方案。强标注语言(Go)用啰嗦换来了好懂的错误,强推断语言(Haskell)用难懂的错误换来了简洁——你只能在这条线上选位置,选不到两全。这是语言设计里活的研究问题。

## 失败模式:把 `==` 规则放松一寸,正确性假设漏到运行期

前面讲的都是「检查器抓错」。但类型检查器自己也会坏,而且坏得很隐蔽——不是它崩溃,是它**该报的错没报**,然后这个漏洞一路下移到后端、最终在运行期爆掉。stage 专门 demo 了这个失败模式:一个 `looseEquality` 开关,打开后放行 `1 == "1"`。

正常(严格)模式下,`==` 两侧类型不同就报错。看 `checkBinary` 里 `==` 的分支:

```typescript
if (op === "==" || op === "!=") {
  if (!typeEquals(l, r)) {
    if (this.looseEquality) {
      // 失败模式 demo:放行不同类型相等。这正是「把规则放松」的危险所在。
      return T_BOOL;
    }
    this.diags.error(
      `'${op}' compares different types: ${typeToString(l)} vs ${typeToString(r)}`,
      expr.span, "typecheck",
    );
  }
  return T_BOOL;
}
```

stage 把同一份源码在两种模式下各跑一遍,对照实测:

```
④ 失败模式:放行 1 == "1"(宽松等于)→ 后端假设落空
源码: let same = (1 == "1");
严格模式(正确):报 1 个错 —
  eq:1:13: error[typecheck]: '==' compares different types: int vs string
    let same = (1 == "1");
                ^^^^^^^^
宽松模式(故意放松):报 0 个错 — 程序「通过」类型检查。
  推导 same : bool
```

宽松模式下程序「通过」了类型检查。问题不在类型检查这一层暴露,而在后端:**后端为 `==` 生成的对比指令通常假设「两侧已是同型」**——比如先把两个操作数都当 int 压栈做整数比较(第 07 章的虚拟机会真这么干)。一旦放行 `1 == "1"`,运行期会拿一个字符串的内部指针去和整数 1 做数值比较,结果要么恒 false、要么读到垃圾内存。

这就是类型检查的契约本质:**它通过的每一个程序,都是对后端的一份承诺——「我保证传给你的类型已经对齐了,你不用再防」。** 第 05 章 IR 生成、第 07 章 VM、第 08 章代码生成全都建在这份承诺上,不再做类型校验(那样会重复且拖慢)。所以类型检查器放松一条规则,不是「少报一个警告」那么轻——是**把一个正确性假设从编译期偷偷漏到了运行期**,而下游所有 stage 都已经默认这个假设成立、不设防了。这正是静态类型存在的全部理由,也是为什么生产代码里 `looseEquality` 必须是 false。

可迁移的判断是:**任何「为了方便先放过去」的类型松动,代价不是发生在松动的那一层,而是发生在下游某个假设了严格性的地方——通常是运行期,通常很难定位。** 写类型规则时遇到「这个 corner case 要不要管」,默认答案是管,因为不管的债务会下移、会放大。

## 这一章在流水线里的位置

往回看:第 03 章的 resolver 解决了「每个名字指向哪个声明」(作用域、闭包捕获),给 AST 标注了名字解析信息。类型检查建在它之上——`Scope` 链查变量类型时,信赖的就是名字已经能正确解析。

往前看:通过类型检查的程序进入第 05 章 IR 生成。从这道关之后,**后端可以假设类型已知**:生成 `+` 指令时不用再判断是整数加还是字符串拼接(类型检查已经分流过了),生成调用指令时不用再防参数个数(已经核对过了)。类型检查这道关的价值,就是让后面每一个 stage 都能少写一大堆防御性代码,把「这一类错误不会发生」当成既成事实。

最后照例一句诚实话:这章的类型规则和样例都是教学 toy,绝对结论偏简化(没有泛型、没有子类型、参数硬当 int)。但三个机制是真能带走的:**「编译期挡运行时错」、「error 哨兵止级联误报」、「放松规则等于把漏洞下移到运行期」**。这三条在任何规模的类型检查器里都成立——区别只是 `Type` 联合里有 7 个变体还是 700 个。
