先看一行真实输出。把 `fib.lox`(186 个字符)喂进扫描器,得到 40 个 token、0 个错误,其中一行是:

```
4:9   <           "<"
4:11  Number      "2"  literal=2
```

`4:9` 不是注释,不是 debug 信息,它是这个 `<` token **自带的属性**:第 4 行第 9 列。整本书后面七章——语法分析、名称解析、类型检查、IR 生成——任何一个阶段想报一句"这里错了",指的那个"这里"都是从词法阶段这一刻就钉死的字符区间。这是我想在第一章就讲透的判断:**真正区分玩具 lexer 和工程 lexer 的,不是它能不能切对 token,而是它切错的时候能不能告诉你切错在第几行第几列、并把那段源码下划线给你看。** 能切对是 60 分,能精确报错才是及格线以上。

很多 lexer 教程到"识别出 token 就算赢"为止。那是把最难、最值钱的一半甩给了读者。本章配套代码在 `examples/compiler-from-scratch/src/stage01-lexer.ts`,它跑出的每个数字、每条错误下划线,下面都会引用——不贴想象中的伪代码。

## 词法器为什么必须存在,而不是让 parser 直接读字符

理论上,语法分析器(parser,把 token 拼成语法树的阶段)完全可以直接读字符。实践中没人这么干,原因是**抽象层混叠**:让 parser 一边判断"这几个字符是不是关键字 `if`"一边判断"`if` 后面该跟一个括号表达式",会把两套完全不同的逻辑搅在一起,谁也写不对。

词法器的职责是把一层噪声一次性吃掉,让上层只面对干净的 token 流。看 `stage01-lexer.ts` 文件头注释列的几件事:

- 空白和注释:丢弃,不产 token。
- 数字、字符串:解析成运行期值(`"2"` → `literal=2`,`"hi"` → 去掉引号的 `hi`)。
- 关键字和普通标识符:区分开(`if` 是关键字,`ifx` 是标识符)。
- 多字符运算符:`==` / `>=` / `!=` 合成一个 token,不是两个。

吃掉之后,parser 看到的是 `[fn, Identifier, (, Identifier, ), {, ...]` 这种序列,它再也不用纠结字符级的事。这就是分层的收益:**每一层只解决一个尺度的歧义**。词法解决"字符 → 词"的歧义,语法解决"词 → 句"的歧义。下一章 parser 的全部代码,第一行假设就是"我手上是一串干净 token",它绝不会退回去重新读字符。

### token 是什么:lexeme 与 literal 的刻意分家

token 的结构定义在 `core/token.ts`(被词法器和语法器共享,谁也不许各定义一份):

```ts
// examples/compiler-from-scratch/src/core/token.ts
export interface Token {
  readonly kind: TokenKind;
  /** 源码中的原始切片(诊断回显用)。 */
  readonly lexeme: string;
  readonly span: Span;
  /** 仅字面量 token 有值;非字面量为 undefined。 */
  readonly literal?: LiteralValue;
}
```

这里有个新手常忽略的设计:`lexeme`(源码原文切片)和 `literal`(解析后的值)**故意分成两个字段**。数字 `2` 的 lexeme 是字符串 `"2"`,literal 是 JS 数字 `2`;字符串 `"hi"` 的 lexeme 含引号是 `"\"hi\""`,literal 是去掉引号的 `hi`。

为什么不只存一个?因为它们服务两个不同的消费者,二者诉求冲突:**报错要显原文,求值要用值**。报错时你想下划线显示用户**真正写的**那几个字符(含引号、含 `0x1F` 这种原始写法);求值时你想要解析后的语义值。把它们合并,总有一个消费者拿到的是错的。这是"一个字段服务两个不可调和需求"的典型,分家是对的。

## 单遍扫描:一个指针走到底,不回头

主循环是 `Lexer.scanTokens()`,逻辑朴素到几乎无聊——而无聊正是它的优点:

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts
scanTokens(): Token[] {
  const tokens: Token[] = [];
  while (!this.isAtEnd()) {
    this.start = this.current;
    const tok = this.scanOne();
    // scanOne 返回 undefined 表示「这段是空白/注释,不产 token」。
    if (tok !== undefined) tokens.push(tok);
  }
  // Eof 哨兵:零宽 span 落在文末。parser 靠它停循环。
  tokens.push(makeToken(TokenKind.Eof, "", span(this.current)));
  return tokens;
}
```

两个游标:`start` 是当前正在扫的 token 的起点,`current` 是下一个待读字符。每扫一个 token 前 `start = current` 重置,然后 `scanOne()` 把 `current` 往前推到这个 token 的末尾。**指针只往前走,绝不回退**。

这个"不回退"性质很值钱,它就是任务③吞吐实测的根:把一段代码重复拼到 115,923 个字符、50,007 个 token,本机中位耗时 2.025 ms,折算约 2470 万 token/s。我要诚实标注:**这个绝对值偏乐观**——Lox-mini 是玩具语言,token 种类少,加上 V8 的 JIT 优化,真实生产语言的 lexer 不会这么快。绝对数字不可迁移,但有一个性质可迁移:**单遍扫描对源码长度是线性的、无回溯的**。50k token 也好 500 万 token 也好,每个字符最多被看常数次。一个会回退、会反复重扫的 lexer,最坏情况能退化到平方级,那才是真正该担心的事。

注意循环最后那行 push 的 Eof token:它的 span 是 `span(this.current)`——一个零宽区间,落在 `text.length` 处。这是个哨兵(sentinel,用来标记结束的占位值)。看上面 `fib.lox` 输出最后一行 `11:1 <eof>`,parser 循环靠这个停,不必每处都判"是不是到末尾了"。少一个边界判断,少一类 off-by-one bug。

### scanOne:单字符 switch 打头,需要前瞻的落分支

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts
private scanOne(): Token | undefined {
  const ch = this.advance();
  switch (ch) {
    case "(": return this.make(TokenKind.LParen);
    // ... 其余单字符 ...
    case "+": return this.make(TokenKind.Plus);

    // 多字符运算符:看后一个字符决定。
    case "!": return this.make(this.match("=") ? TokenKind.BangEq : TokenKind.Bang);
    case "=": return this.make(this.match("=") ? TokenKind.EqEq : TokenKind.Eq);
    case "<": return this.make(this.match("=") ? TokenKind.LessEq : TokenKind.Less);
    case ">": return this.make(this.match("=") ? TokenKind.GreaterEq : TokenKind.Greater);
    // ...
    default:
      if (isDigit(ch)) return this.scanNumber();
      if (isAlpha(ch)) return this.scanIdentifier();
      // 非法字符:报错后跳过,不抛。
      this.diags.error(`unexpected character '${ch}'`, span(this.start, this.current), "lexer");
      return undefined;
  }
}
```

确定的单字符直接 `make`;有歧义的(运算符、数字、字符串、标识符)落到对应分支去前瞻。这个 `match("=")` 就是下面要重点讲的"前瞻一位",也是手写 lexer 最经典的坑。

顺带说一个看似无关、实则很关键的取舍:`isDigit` / `isAlpha` 是手写字符比较,不是正则:

```ts
function isDigit(ch: string): boolean {
  return ch >= "0" && ch <= "9";
}
```

热路径上每个字符都要判一次类。正则 `/\d/` 每次匹配有对象创建和潜在回溯开销,在百万字符级别会真切拖慢吞吐。手写 charCode 区间比较是对分支预测友好的整数比较。这是少数几个"看起来啰嗦但有性能理由"该写 inline 注释的地方——代码里确实写了。**不要无脑用正则做 lexer 的字符分类**,那是把热路径让给了对象分配。

## ⚡ 最大吞噬:扫 `>=` 时为什么必须前瞻一位

来到本章的核心机制,也是手写 lexer 第一大坑。

扫到一个 `>` 字符,你不能立刻产出一个 `Greater` token。因为下一个字符如果是 `=`,这俩合起来是 `>=`(`GreaterEq`),是**一个** token,不是两个。规则叫**最大吞噬(maximal munch)**:在当前位置,贪心匹配能匹配的最长 token。`>=` 比 `>` 长,所以优先 `>=`。

实现就是 `match()` 这个"条件前进":

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts
private match(expected: string): boolean {
  if (this.isAtEnd() || this.src[this.current] !== expected) return false;
  this.current++;
  return true;
}
```

看一眼当前字符:是 `=` 就吞掉它、返回 true(于是 `>` 升级成 `>=`);不是就什么都不动、返回 false(`>` 保持单字符)。关键在它**不消费失败的字符**——前瞻是"看一眼",不是"吃下去"。

任务②的对拍证明这事是对的。探针源码 `if (x >= y) { x != 1; }` 扫出来:

```
... if ( Identifier >= Identifier ) { Identifier != Number ; } ...
```

`>=` 是一个 token,`!=` 是一个 token。代码里还钉了断言:`assertEq(byLexeme(">=").kind, TokenKind.GreaterEq, "'>=' is one token")`。

### 不前瞻会怎样:词法"成功"了,token 流却悄悄损坏

这是 ⚡ 标记的开放坑里最阴的一点——**漏前瞻不会让程序崩,它让程序悄悄出错,而且错误爆点离根因很远**。`stage01-lexer.ts` 用 `demoNoLookahead()` 真实演示了这个差异:

```
正确(带前瞻):  "x >= 2"  →  Identifier >= Number
错误(无前瞻):  "x >= 2"  →  Identifier > = Number
```

带前瞻是 3 个 token(`assertEq(good.length, 3)`),不前瞻是 4 个(`assertEq(naive.length, 4)`)。`>=` 被切成了 `[>]` `[=]`。

注意:**词法这一步"成功"了**。没报错,没崩,产出了一串看起来合法的 token。问题埋进了 token 流。它会在下一章 parser 才爆,而且爆得让你摸不着头脑。引用 demo 的输出原话:

> 下一章 parser 解析 `x > = 2` 时:吃掉 `x`、看到 `>` 当二元运算符、期望右侧是表达式 —— 却拿到一个孤立的 `=`(赋值号),报「expected expression」。根因在词法(漏前瞻),报错却出在 parser,排查方向被带偏。

你盯着 parser 报的"expected expression"调一下午,根因却在三百行外的词法器漏了一行 `match("=")`。这就是为什么前瞻必须做对、做在最早的位置:**让歧义在最早、最有上下文的地方被解决**。一旦 token 流污染流到下游,定位成本指数级上升。

### `/` 的双重身份:除号还是注释

`match` 的同一套机制也用来区分"除号 `/`"和"行注释 `//`":

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts
case "/":
  if (this.match("/")) {
    // 行注释:吃到行尾,不产 token。
    while (this.peek() !== "\n" && !this.isAtEnd()) this.advance();
    return undefined;
  }
  return this.make(TokenKind.Slash);
```

`peek()` 是"看当前字符但不前进",越界返回 `"\0"` 哨兵(绝不出现在合法源码里,所以可以安全当结束标记)。注释扫到行尾就停,`return undefined` 表示"这段不产 token"。这又是一次前瞻决策:`/` 后面是不是又一个 `/`,决定它是运算符还是要丢弃的注释。

⚡ 这里值得标一笔"无通用解"的边界:Lox-mini 只需要**前瞻一位**就能消解所有歧义,所以 `match`(看一位)够用。但前瞻一位不是普适的。C++ 的 `>>` 在模板里(`vector<vector<int>>`)到底是右移运算符还是两个闭合尖括号,**靠词法器前瞻多少位都判不出来**——它依赖语法甚至语义上下文。这类"词法和语法纠缠"的歧义,业界没有一个干净的通用解:C++ 早年要求写成 `> >`,后来编译器靠 parser 反向喂信息给 lexer 来 hack;有人主张 scannerless parsing(无独立词法层,语法直接吃字符)来根治,但代价是把两层歧义重新搅回一起。**"词法该独立到什么程度"至今没有让所有人满意的答案**,只有按语言复杂度权衡的工程选择。Lox-mini 选了最干净的那档:文法设计上保证前瞻一位足够。

## 标识符 vs 关键字:也是最大吞噬,只是换了战场

最大吞噬不只管运算符。`if` 是关键字,`ifx` 是标识符——区别也是"匹配最长":

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts
private scanIdentifier(): Token {
  while (isAlphaNum(this.peek())) this.advance();   // 先吞完整个词
  const text = this.src.slice(this.start, this.current);
  const keyword = KEYWORDS.get(text);               // 再查关键字表
  if (keyword !== undefined) {
    if (keyword === TokenKind.True) return this.make(keyword, true);
    if (keyword === TokenKind.False) return this.make(keyword, false);
    return this.make(keyword);
  }
  return this.make(TokenKind.Identifier);
}
```

顺序很关键:**先把整个 `[a-zA-Z_][a-zA-Z0-9_]*` 吞完,再去查关键字表**。如果你扫到 `if` 就急着判定"这是关键字",`ifx` 会被错切成关键字 `if` + 标识符 `x`。先吞到边界(遇到非字母数字才停)再分类,最大吞噬自动成立。任务②的断言钉死了这点:`assertEq(byLexeme("ifx").kind, TokenKind.Identifier, "'ifx' is identifier (longest match)")`。

关键字表 `KEYWORDS` 放在 `core/token.ts`,是全书唯一真相。"`if`/`while` 是关键字还是标识符"是词法器的判断,但这张表词法器和别人共享、绝不各写一份——否则一改枚举两处对不上。表里全是小写:Lox-mini 关键字大小写敏感,`If` 是标识符。

## ✦ 错误定位:把"非法字符"指到第几行第几列,并下划线

回到开篇的判断——精确报错才是工程实现的及格线。现在看它怎么落地。

### 关键架构决策:span 从词法阶段就携带,行列推迟到诊断层算

词法器**不算行列**。它只给每个 token 一个 span——源码里的字符区间 `[start, end)`。算行列是 `SourceText` 的事:

```ts
// examples/compiler-from-scratch/src/core/source.ts
offsetToLineCol(offset: number): LineCol {
  const off = Math.max(0, Math.min(offset, this.text.length));
  // 二分: 找最大的 lineStarts[i] <= off。
  let lo = 0;
  let hi = this.lineStarts.length - 1;
  while (lo < hi) {
    const mid = (lo + hi + 1) >> 1;
    if (this.lineStarts[mid] <= off) lo = mid;
    else hi = mid - 1;
  }
  return { line: lo + 1, col: off - this.lineStarts[lo] + 1 };
}
```

`SourceText` 构造时一次性扫一遍记下每行首字符偏移(`lineStarts` 前缀表),之后任何 offset 反查行列都是 O(log n) 二分。

为什么这么切分?因为**所有阶段——词法、语法、语义、类型——都要把错误指回源码位置**。如果每个阶段各自扫一遍换行算行号,既重复又容易踩坑(off-by-one、CRLF、末行无换行)。集中算一次、共享同一份真相,是单一职责的胜利:**词法只产 span(纯字符偏移,便宜),定位逻辑推迟到真正要打印错误时才算**。这条决策的回报要到第 04 章才完全兑现——那时类型检查器报"类型不匹配",下划线用的还是这套 `span` + `offsetToLineCol`,一行算行列的代码都不用重写。span 是从词法到类型检查全程不变的通用货币。

### 报错不靠 throw:一次扫描收集全部错误

这是工程 lexer 和玩具 lexer 的第二个分水岭。玩具 lexer 遇到第一个非法字符就 `throw`,用户改一个错重编一次只看见下一个。专业编译器**一次报全**。

实现上,词法器遇到错误不抛异常,只往 `DiagnosticBag`(诊断收集袋)push 一条,然后**继续扫**:

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts (scanOne 的 default 分支)
this.diags.error(`unexpected character '${ch}'`, span(this.start, this.current), "lexer");
return undefined;   // 跳过这一个字符,继续扫后面
```

`DiagnosticBag.error()` 的不变量写在 `core/diagnostics.ts`:**只 push,不抛**。

```ts
// examples/compiler-from-scratch/src/core/diagnostics.ts
error(message: string, span: Span, stage?: string): void {
  this.items.push({ severity: Severity.Error, message, span, stage });
}
```

任务④用一个同时含两种错误的坏样例验证:第 1 行有非法字符 `@`,第 3 行字符串未闭合到文件尾。一次扫描产出 16 个 token,**收集到 2 条错误**,各自定位准确:

```
bad.lox:1:11: error[lexer]: unexpected character '@'
  let a = 1 @ 2;
            ^
bad.lox:3:9: error[lexer]: unterminated string literal
  let s = "oops
          ^^^^^
```

下划线是 `SourceText.snippet(span)` 画的——它从 span 起点算出行列,取那一行源码,在下方对齐位置打 `^`。`@` 是单字符错误,一个 `^`;未闭合字符串 span 覆盖 `"oops` 五个字符,五个 `^`。这就是开篇说的"精确指到第 12 行第 5 列并打印下划线片段"。代码里钉了断言:`assertEq(diags.errorCount(), 2, "should collect BOTH lexical errors in one pass")`——证明它确实是收集而非首错即停。

### 错误恢复:坏字符串也产出一个(坏)token

为什么遇到第二个错误(未闭合字符串)时,词法器还能正常报出来?因为第一个错误处理时它**跳过坏字符继续走**,而不是停摆。未闭合字符串的处理更微妙:

```ts
// examples/compiler-from-scratch/src/stage01-lexer.ts
private scanString(): Token {
  while (this.peek() !== '"' && !this.isAtEnd()) this.advance();
  if (this.isAtEnd()) {
    this.diags.error("unterminated string literal", span(this.start, this.current), "lexer");
    // 把已扫部分当一个坏 String token 产出,好让 parser 仍能往下走。
    return this.make(TokenKind.String, this.src.slice(this.start + 1, this.current));
  }
  this.advance(); // 消费闭引号。
  // ...
}
```

它**仍然产出一个 String token**(虽然内容是坏的)。为什么不直接 `return undefined` 把它丢掉?因为下游 parser 期待 `let s = <某个表达式>` 后面有个值。给它一个坏 String,parser 还能继续解析这条语句、报它自己那一层的错;丢掉则可能引发 parser 的连锁误报。这是**错误恢复(error recovery)**的雏形:坏输入下尽量维持结构,让每一层都能各报各的错,而不是第一个错就让整条流水线雪崩。完整的恢复策略(parser 的 panic-mode)是下一章的事,但词法这层已经在为它铺路。

## 边界情况清单:那些 demo 没专门演示但代码处理了的

工程 lexer 的水位线在边界情况。几个值得记的:

- **跨界 offset**:Eof token 的 span 落在 `text.length`(等于而非小于源码长度)。`offsetToLineCol` 对此有定义(`off` 上界 clamp 到 `text.length`),不会越界崩。
- **坏 span 不崩诊断**:`source.ts` 文件头明确写了,越界 offset 一律 clamp 到合法区间而不抛异常——"诊断渲染绝不能因为一个坏 span 把整个编译器打挂,宁可下划线画歪也要把别的错误报出来"。可观测性优先于精确性,这是对的。
- **数字的小数点**:`scanNumber` 只在"点后紧跟数字"时才认小数,避免把 `1.method` 里的点吞进数字。`1.` 和 `.5` 在 Lox-mini 都不合法。
- **CRLF**:`source.ts` 只认 `\n` 作行边界,`\r` 当普通字符留行内。Lox-mini 不在意 `\r` 引起的列位轻微偏差——简单胜过精确,但这是个**显式记录的取舍**,不是疏忽。处理 Windows 换行严格的语言要重新考虑这里。

这些不是"为了完整"硬凑的分支,每一个都对应一类真实会撞上的输入。区分"这个 lexer 我敢用在生产"和"这是个 demo"的,恰恰是这张清单上的项有没有被想过、被显式取舍过。

## 小结:这一章给后面留下了什么

词法分析把字符流切成 token 流,单遍、线性、无回溯。但本章真正想立住的不是"会切 token",而是两条贯穿全书的地基:

1. **span 从这里诞生,全程不变**。每个 token 带精确字符区间,行列推迟到诊断层用 `SourceText` 统一算。第 02 到 08 章任何阶段报错,下划线都靠它——一行重复代码都不用写。
2. **诊断走收集而非抛异常**。`DiagnosticBag` 一次扫描收齐所有错误,坏输入下尽量维持结构让下游继续。这是错误恢复的起点。

两个最该记住的坑:最大吞噬必须前瞻(漏了 `>=` 会切成 `>` `=`,且爆点在 parser 离根因很远);前瞻一位是 Lox-mini 的福气,真实语言的词法/语法歧义(C++ 的 `>>`)至今无通用解。

下一章,语法分析器接过这串 token,开始解决"词 → 句"的歧义。它的第一行假设——"我手上是一串干净的、带 span 的 token"——正是本章交付的契约。它绝不会退回去重新读一个字符。
