先看一个会让人栽跟头的例子:`1 - -2`。它必须解析成「二元减法,右操作数是一元负 2」,也就是 `(- 1 (- 2))`,而不是别的。再看 `-2 * 3`:一元负号只能咬住 `2`,顶层必须是乘法,即 `(* (- 2) 3)`。这两个例子里没有一个括号,可结果的树形天差地别。控制这个差别的,不是几百行嵌套的文法规则,而是一张表里的几个整数——一元负号的绑定力(binding power,你可以先理解成"这个运算符抓操作数抓得多紧")设成 12 还是 8,`-2 * 3` 就会得到两棵语义完全不同的树。本章把这件事讲透:为什么手写工业级 parser 不用经典的 `expr→term→factor` 文法链,而用 Pratt 解析配一张绑定力表;以及一个真实 parser 报一个错时凭什么不当场崩掉,而是跳到下一条语句继续往下报。

第 01 章把源码扫成了一串扁平的 token(词法单元,语言里最小的有意义片段,像 `let`、`+`、`100`)。token 流是线性的,但程序的含义是嵌套的:`while` 的循环体里有赋值,赋值的右边有加法,加法的操作数里又有函数调用。语法分析(parsing)这一关的全部工作,就是把这条扁平的 token 流织成一棵反映嵌套结构的树——抽象语法树(AST)。本章产出的这棵树,会原样交给第 03 章做名称解析(把每个变量引用对应到它的声明)和类型检查。所以这一章的输出质量,直接决定后面所有阶段能不能干活。

本章对应的全部代码在 `examples/compiler-from-scratch/src/stage02-parser.ts`,核心数据结构(AST 节点定义 + S 表达式打印器)在 `examples/compiler-from-scratch/src/core/ast.ts`。下面所有数字都来自这个文件真跑出来的输出,不是讲义里编的。

## 两种解析,各管一摊

手写 parser 的第一个工程判断是:语句和表达式用不用同一套机制?答案是不用,而且这个分工是有道理的。

**语句(statement)用递归下降。** 递归下降(recursive descent)的意思朴素到近乎平庸:每一种文法结构写一个函数,函数内部该解析子结构时就调对应的函数。`if` 语句调 `parseIf`,它内部解析条件表达式、再解析花括号里的语句块,而语句块里可能又有 `while`,于是 `parseBraceBody` 又调回 `parseStmt`。函数调用栈的形状,天然就镜像了语法树的嵌套形状。这就是为什么 Clang、V8 这些工业前端的语句解析骨架都是手写递归下降——它直观、可调试、错误信息好定位,代价只是代码量。

`stage02-parser.ts` 里 `parseStmt` 就是一个纯粹的分发:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
private parseStmt(): Stmt {
  switch (this.peek().kind) {
    case TokenKind.Let:    return this.parseLet();
    case TokenKind.Fn:     return this.parseFnDecl();
    case TokenKind.If:     return this.parseIf();
    case TokenKind.While:  return this.parseWhile();
    case TokenKind.Print:  return this.parsePrint();
    case TokenKind.Return: return this.parseReturn();
    case TokenKind.LBrace: return this.parseBlock();
    default:               return this.parseExprStmt();
  }
}
```

每个 `parseXxx` 长得都一样:`advance()` 吃掉起始关键字,`expect()` 强制要求某个 token(没有就报错),需要表达式的地方调 `parseExpr`,需要语句块的地方调 `parseBraceBody`。这里有个值得停一下的设计决策:函数声明 `fn fib(n) { ... }` 没有单独的 `FnDecl` 节点,而是复用 `Fn` 表达式节点(`name` 可空),包进 `ExprStmt`。

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
private parseFnDecl(): Stmt {
  const fn = this.parseFnExpr();
  return { kind: "ExprStmt", expr: fn, span: fn.span };
}
```

**为什么这么设计?** 因为如果声明和匿名函数是两种节点,后面第 03 章的 resolver 和求值阶段就得为"函数"写两套遍历逻辑。让它们共用一个结构(`Fn`,带名字就是声明,不带名字就是匿名),后续阶段只认一种节点。判别联合 AST(`core/ast.ts` 里用 `kind` 字段区分的那套类型)的全部价值就在这:节点种类越少、越正交,后面每一关的代码就越短。这是一个会反复出现的取舍——AST 的形状是为下游服务的,不是为了忠实复刻语法。

**表达式用 Pratt。** 这是本章的重头戏,下一节单独讲。先说为什么不把表达式也用递归下降写。

## ✦ 为什么不用嵌套文法规则:Pratt 解析与绑定力表

教科书教表达式优先级的经典写法是给每一档优先级写一个文法规则,然后翻译成一个递归函数:

```
expression → equality
equality   → comparison ( ("==" | "!=") comparison )*
comparison → term ( ("<" | "<=" | ">" | ">=") term )*
term       → factor ( ("+" | "-") factor )*
factor     → unary  ( ("*" | "/") unary )*
unary      → ("-" | "!") unary | primary
```

它能跑,Crafting Interpreters 里的 jlox 就是这么写的。但它有两个让人不舒服的地方。第一,**每加一档优先级就加一层函数**——想在比较和加法之间塞一个位运算 `&`,得新写一个函数,还得把上下两层的调用关系改对。第二,**左结合靠 `while` 循环、右结合靠递归,两种结合性用两种代码结构表达**,写错了不容易一眼看出来。优先级和结合性这两个本质上是数值的东西,被编码成了函数的嵌套层数和循环/递归的选择,很别扭。

Pratt 解析(算符优先解析,有的资料叫 precedence climbing)换了个思路:把优先级和结合性都编码成数字,主循环只比较数字。每个二元运算符给一对绑定力 `[leftBp, rightBp]`,整张表是表达式优先级的「唯一真相」:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
const BINARY_BP: ReadonlyMap<TokenKind, BindingPower> = new Map([
  [TokenKind.Eq,       { left: 2,  right: 1  }],  // 赋值,右结合
  [TokenKind.EqEq,     { left: 4,  right: 5  }],  // 相等
  [TokenKind.Less,     { left: 6,  right: 7  }],  // 比较
  [TokenKind.Plus,     { left: 8,  right: 9  }],  // 加减
  [TokenKind.Minus,    { left: 8,  right: 9  }],
  [TokenKind.Star,     { left: 10, right: 11 }],  // 乘除,最高
  [TokenKind.Slash,    { left: 10, right: 11 }],
]);
```

(为节省篇幅省略了 `!=` `<=` `>` `>=` 几行,它们与同档运算符共用 bp。)整个表达式解析就是一个循环加一个递归参数 `minBp`:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
private parseExpr(minBp: number): Expr {
  let left = this.parsePrefix();
  left = this.parsePostfix(left, minBp);   // 调用 f(x) 优先级最高,先处理
  for (;;) {
    const opTok = this.peek();
    const bp = BINARY_BP.get(opTok.kind);
    if (bp === undefined || bp.left <= minBp) break;  // 绑定力不够,交还父层
    this.advance();
    const right = this.parseExpr(bp.right);  // 右递归,门槛 = rightBp 决定结合性
    left = { kind: "Binary", op: opTok.lexeme, left, right, span: spanMerge(left.span, right.span) };
    left = this.parsePostfix(left, minBp);
  }
  return left;
}
```

`minBp` 是这套机制的全部秘密。它是父层传下来的门槛,含义是"我只关心比 X 抓得更紧的运算符,比 X 松的请留给我"。理解了 `minBp`,优先级和结合性就一起解释清了:

- **优先级**:`*` 的 leftBp(10)大于 `+` 的 leftBp(8)。解析 `1 + 2 * 3` 时,吃完 `1 +` 后用 `minBp = 9`(加法的 rightBp)去解析右边,`*` 的 leftBp(10)> 9,于是 `2 * 3` 被吞进加法的右子树,得到 `(+ 1 (* 2 3))`。实测对拍通过。
- **左结合**:`leftBp < rightBp`。解析 `1 - 2 - 3`(最外层 `minBp=0`):吃掉第一个 `-`,右递归用它的 `rightBp=9` 当门槛去解析右操作数;进去先拿到 `2`,再看后面的 `-`,它的 leftBp 是 8,`8 ≤ 9` 门槛不够,于是右递归就此退出,只把 `2` 当右操作数——`1 - 2` 先成型。回到最外层循环,第二个 `-` 的 leftBp(8)> `minBp`(0)被吞,左边整个 `(- 1 2)` 当它的左操作数,得 `(- (- 1 2) 3)`,左结合。关键就在 `rightBp`(9)比 `leftBp`(8)高 1,挡住了同级运算符往右嵌套。实测通过。
- **右结合**:`leftBp > rightBp`。赋值 `=` 用 `{ left: 2, right: 1 }`。解析 `a = b = c`,右递归用 `rightBp=1` 进去,第二个 `=` 的 leftBp 是 2,2 > 1 被吞,于是 `b = c` 嵌进右边,得 `(set! a (set! b c))`。实测通过。

注意所有 bp 都是偶数间隔,留出奇数给结合性微调:左结合写 `[n, n+1]`,右结合写 `[n+1, n]`。

**这个写法的工业级价值是什么?** 加一个新二元运算符——比如想加取模 `%`,优先级同乘除——只需要在 `BINARY_BP` 里加一行 `[TokenKind.Percent, { left: 10, right: 11 }]`,主循环一个字不用动。对比经典文法链需要新增/调整函数层级,Pratt 把"扩展运算符集"这件事变成了纯数据改动。这就是为什么手写 parser(包括很多真实语言的前端)在表达式部分倾向 Pratt:它把语法的可变部分(运算符表)和不变部分(解析循环)彻底分开了。

一元前缀和后缀也纳入同一套数字体系。一元 `-` `!` 用一个比所有二元都高的 `PREFIX_BP = 12`,函数调用 `f(...)` 用更高的 `POSTFIX_BP = 14`:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
const PREFIX_BP = 12;   // 高于所有二元: -a+b = (-a)+b, -a*b = (-a)*b
const POSTFIX_BP = 14;  // 高于一元: -f(x) = -(f(x)), f(x)(y) 左结合链式
```

一元运算符在 `parsePrefix` 里用 `PREFIX_BP` 当门槛递归解析它的操作数:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
case TokenKind.Minus:
case TokenKind.Bang: {
  this.advance();
  const operand = this.parseExpr(PREFIX_BP);  // 关键: 用 PREFIX_BP 而非 0
  return { kind: "Unary", op: t.lexeme, operand, span: spanMerge(t.span, operand.span) };
}
```

这一行 `this.parseExpr(PREFIX_BP)` 是本章下面失败模式那一节的全部隐患所在。先记住它。

把这套机制跑过一组逐条断言,实测全过(`②` 这一段每条都是 `astToSExpr` 对拍固定字符串):

```
✓ 1 + 2 * 3      → (+ 1 (* 2 3))         (乘高于加)
✓ 1 - 2 - 3      → (- (- 1 2) 3)         (减法左结合)
✓ a = b = c      → (set! a (set! b c))   (赋值右结合)
✓ 1 < 2 == true  → (== (< 1 2) #t)       (比较高于相等)
✓ -a + b         → (+ (- a) b)           (一元负号紧于二元加)
✓ !a == b        → (== (! a) b)          (一元 ! 紧于相等)
✓ f(x)(y)        → (call (call f x) y)   (调用左结合链式)
✓ -f(x)          → (- (call f x))        (调用紧于一元: 先调用再取负)
✓ 1 - -2         → (- 1 (- 2))           (二元减 - 一元负)
```

这些 S 表达式不是给人看着舒服的玩具——`astToSExpr`(`core/ast.ts`)是确定性的,同一棵树永远打印同一字符串,所以它能直接 `assertEq` 对拍。这是逐章端到端验证的基础:第 02 章产出的树、第 03 章 resolve 后的树,都能 dump 成一行可 diff 的文本比对,不必肉眼读对象。

顺带提一个容易被忽略的细节:括号在 AST 里**不留痕迹**。`parsePrefix` 遇到 `(` 时把门槛重置为 0 递归解析内部,然后只返回内部表达式,不生成"括号节点":

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
case TokenKind.LParen: {
  this.advance();
  const inner = this.parseExpr(0);   // 括号内重置门槛
  this.expect(TokenKind.RParen, "')'");
  return inner;  // 分组不产生节点 —— 优先级已体现在树的结构里
}
```

为什么不留?因为括号的唯一作用是改变结合方式,而结合方式一旦体现在树的形状里,括号本身就是冗余信息了。`(1 + 2) * 3` 和 `3 * (1 + 2)` 解析完,树里看不到括号,只看到 `*` 的某个子树是 `(+ 1 2)`。下游阶段只关心结构,不关心源码长什么样。

## ⑤ 失败模式:绑定力设错一个数,语义就错

现在回到开头埋的那行 `this.parseExpr(PREFIX_BP)`。把 `PREFIX_BP` 从 12 改成 8——和二元减法的 leftBp 相同——会发生什么?`stage02-parser.ts` 里专门写了一个刻意写坏的迷你解析器 `parseWithBrokenUnaryBp` 来复现,只覆盖演示需要的 number / 一元负 / 二元减三件事,不污染正确实现:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
const BROKEN_PREFIX_BP = 8;  // BUG(教学故意): 应为 12,设成与减法 leftBp 相同
// ...
if (t.kind === TokenKind.Minus) {
  advance();
  const operand = expr(BROKEN_PREFIX_BP);  // ← 错误门槛
  return { kind: "Unary", op: "-", operand, span: spanMerge(t.span, operand.span) };
}
```

拿 `-2 * 3` 喂进去,正确实现和坏实现产出两棵不同的树,实测:

```
输入            : -2 * 3
正确(PREFIX_BP=12) : (* (- 2) 3)   ← 一元负只作用于 2, 顶层是乘法
错误(PREFIX_BP=8)  : (- (* 2 3))   ← 一元负把整个 (* 2 3) 吞进 operand
✓ 两种 binding power 产出不同 AST, 证明它不是可有可无的调参。
```

**根因**:一元运算符解析它的操作数时,用自己的绑定力当门槛。门槛 12 时,后面的 `*`(leftBp 10)抓不动门槛 12,于是一元负只吃到 `2` 就停,把 `*` 留给外层——得到 `(* (- 2) 3)`。门槛降到 8 时,`*` 的 leftBp(10)> 8,于是一元负的操作数解析贪婪地把整个 `2 * 3` 吞了进来——得到 `(- (* 2 3))`。一句话:**一元 operand 的递归门槛若 ≤ 后续二元的 leftBp,一元会贪婪吞掉本应留给二元的运算符。**

这个例子选 `-2 * 3` 是有讲究的:`(-2)*3` 和 `-(2*3)` 算出来都是 -6,数值碰巧相同。但**结构完全不同**——交给下游的语义树不是一回事。换成 `-2 * 3 < x` 这种,结合错位就连最终结果都会错。我特意挑了个数值相同的例子,是想说明一件容易被低估的事:绑定力设错不是"性能调参"那种软问题,它直接改变程序的含义,而且可能藏在某个数值恰好相同的测试里躲过你的眼睛。这就是为什么 `②` 那一组逐条断言里必须有 `-a * b`、`-f(x)` 这种把一元和二元/后缀混在一起的 case——它们专门用来钉死这条边界。

## 解析吞吐:递归下降 + Pratt 是单遍线性的

把 loopsum 那段循环代码复制 400 份拼成一个大程序,在本机实测一遍解析:

```
源码规模      = 35,999 chars, 14,801 tokens
AST 节点数    = 9,600 (确定性计数)
parse 中位数  = 0.288 ms
吞吐          = 33,330,131 nodes/s
```

每秒三千多万节点。但这个绝对数字要诚实地打个折:这是 toy 语言(运算符少、文法简单)、token 已经是内存里的数组(不含磁盘/网络 IO)、解析逻辑全在一个进程的热路径上。真实语言的 parser 要处理几十种运算符、注释、宏、错误恢复路径,绝对值会低不少。

可以迁移的不是这个数字,而是它背后的**复杂度特征**:递归下降 + Pratt 是单遍(single-pass)、线性的——每个 token 基本只被 `peek`/`advance` 看常数次,没有回溯(backtracking)。这就是为什么 parser 通常不是编译器的性能瓶颈:它和输入规模成正比,系数还很小。瓶颈往往在后面的类型检查、优化、代码生成(第 04-08 章)。在 parser 上花心思优化常数,多数时候是优化错了地方。

## ⚡ 错误恢复:报一个错就崩,是玩具;报多个错,才是工具

到这里我们一直在跑能解析成功的代码。但真实 parser 的大部分时间在面对**写错的**代码——少个分号、括号不配对、关键字打错。一个只会报第一个错就停的 parser,逼用户改一行重编译看下一个错,改一轮看一个。专业编译器必须一次尽量多报。

这就是错误恢复(error recovery)。`stage02-parser.ts` 用的是最经典的 panic-mode(恐慌模式)恢复:遇到无法继续的语法错,不当场崩,而是丢弃 token 直到一个「语句边界」——分号之后,或下一条语句的起始关键字之前——然后从那里重新开始解析。这样一处错误不会污染整份文件。

实现上有两个关键决策。第一,**parser 永不把异常当业务控制流**。语法错误一律进 `DiagnosticBag`(诊断收集器),`ParseError` 这个异常只用来做栈展开,快速回到语句边界,catch 在顶层不外泄:

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
parseProgram(): Program {
  const stmts: Stmt[] = [];
  while (!this.atEnd()) {
    const before = this.pos;
    try {
      stmts.push(this.parseStmt());
    } catch (e) {
      if (!(e instanceof ParseError)) throw e;  // 真异常(bug)不吞
      if (!this.opts.recover) break;            // 关恢复: 首错即停
      this.synchronize();                       // 开恢复: 跳到下一语句边界
    }
    // 防御性死循环保护: 某次循环既没前进也没抛错就强制前进一格
    if (this.pos === before && !this.atEnd()) this.advance();
  }
  return stmts;
}
```

注意最后那行防死循环保护。它「永远不会发生」,但 parser 一旦写出某条既不前进也不抛错的路径,就会无限循环挂死,代价太高,所以留一道兜底。这是 `coding.md` 里说的"看似冗余的代码必须带 why 注释"的典型场景。

第二,`synchronize` 怎么定义"语句边界":

```ts
// examples/compiler-from-scratch/src/stage02-parser.ts
private synchronize(): void {
  while (!this.atEnd()) {
    if (this.tokens[this.pos - 1]?.kind === TokenKind.Semicolon) return;  // 刚吃过分号
    switch (this.peek().kind) {
      case TokenKind.Let: case TokenKind.Fn: case TokenKind.If:
      case TokenKind.While: case TokenKind.Print: case TokenKind.Return:
        return;  // 下一个是语句起始关键字
    }
    this.advance();
  }
}
```

跑一段三处独立语法错的坏代码(缺分号、括号不配对、再缺分号),开和关恢复的差异是实打实的:

```
[开 synchronize] 收集到 3 个语法错:
  broken.lox:2:1:  error[parser]: expected ';' after let
  broken.lox:2:15: error[parser]: expected ')'
  broken.lox:3:12: error[parser]: expected ';' after print

[关 synchronize] 只报首错 1 个 (首错即停):
  broken.lox:2:1:  error[parser]: expected ';' after let
```

恢复后一次暴露 3 个错,用户改一轮看全部;不恢复要改 3 轮。代码里关恢复就是 `parse(badSrc, { recover: false })`——同一个 parser,一个 flag 切换两种行为,方便把"恢复的价值"做成可复现的对比而不是口头声称。

**为什么把这一节标成 ⚡(前沿/开放问题)?** 因为 panic-mode 只是"能继续",远不是"恢复得好"。它有两个老大难、至今没有通用解的毛病:

一是**误报级联(cascading errors)**。同步点选得不好,parser 跳过太多或太少,会从一个真错误衍生出一串假错误。我们这个 toy 同步到分号/关键字,够用;但真实语言里(想想 C++ 的模板、Rust 的生命周期),一个错位的 token 能让后面十几个本来正确的 token 全被误报,用户得在一堆噪音里找那个真错。

二是**恢复质量没有客观标准**。"报得好"意味着报告的错误尽量贴近用户的真实意图、尽量少误报、尽量给出能改的建议——但"用户的真实意图"本身就无法从坏掉的 token 流里可靠推断。学术界尝试过各种方向:Burke-Fisher 的错误修复(试探性地插入/删除 token 看哪种修复让解析继续)、基于编辑距离的最小代价修复、近年还有用机器学习预测"程序员大概想写什么"的尝试。但**没有一个方法是通用最优的**——恢复策略本质上要在"少误报"和"多继续"之间权衡,这个权衡依赖具体语言、具体错误模式,目前仍是活跃的研究方向。换句话说,把语法错误恢复做到能用是工程,做到优雅至今没有银弹。

## 交给下一章的东西

本章产出的是 `core/ast.ts` 定义的判别联合 AST:`Expr`(七种节点:Binary / Unary / Literal / Var / Assign / Call / Fn)和 `Stmt`(七种:Let / Print / If / While / Block / Return / ExprStmt)。每个节点带 `span` 指回源码——这是后续阶段报错定位的命脉。

值得留意 `VarExpr` 上有个现在还空着的字段:

```ts
// examples/compiler-from-scratch/src/core/ast.ts
export interface VarExpr {
  readonly kind: "Var";
  readonly name: string;
  readonly span: Span;
  depth?: number;  // resolver(stage03) 回填: 变量到声明跳几层作用域
}
```

语法阶段我们只知道这是个名叫 `name` 的变量引用,不知道它指向哪个声明、在第几层作用域——`depth` 现在是 `undefined`。第 03 章的名称解析(resolver)会遍历这棵树,把每个 `Var` 和 `Assign` 的 `depth` 填上,让运行时能 O(1) 定位变量而不必动态查作用域链。把 `depth` 放在 AST 节点上而非旁路的 map 里,是个刻意的决定:让"未 resolve"和"已 resolve"是同一棵树的两个状态,下游只读一个可空字段,不必同时管理两份数据。

所以这棵树会被改写(in-place 回填),但不会被换掉。第 03 章在它上面做名称解析与类型——下一关见。
