把这棵右深算术树 `1 + (2 + (3 + ... + 10))` 喂进第 05 章那种线性 IR(中间表示,一种比源码低、比机器码高的指令形式),它会展开成 19 条三地址指令,每条结果落到一个全新的虚拟寄存器——`t0, t1, ... t18`,假装寄存器无限多。这是第 05、06 章一直在玩的世界:虚拟寄存器随便用,反正是抽象的。本章要做的事一句话:把这 19 个虚拟寄存器,塞进一台只有 4 个物理寄存器的真机器里。塞不下的,得 spill(溢出)到栈上,用到时再 load 回来。

跑出来的结果是:19 条 IR 指令 → 33 条机器指令,其中 7 条是 `STORE [slot], r` 的 spill。多出来的 14 条,全是「寄存器不够」逼出来的开销。配套代码 `examples/compiler-from-scratch/src/stage08-codegen.ts`,确定性输入,本机实测。这一章是全书收口:前面所有阶段产出的要么是另一种 IR,要么是另一台解释器(第 07 章 VM),只有这一章真的把东西落到「有限资源的目标机器」上——而这恰恰是真实编译器后端工程量的大头。

## 代码生成是「又一次降低」

整本书的主线是一句话:**编译就是一连串降低(lowering)**。源码 → token(第 01 章) → AST(第 02 章) → 类型检查过的 AST(第 04 章) → 线性 IR(第 05 章) → 优化后 IR(第 06 章)。每一步都丢掉一些高层信息,换来更接近机器的形态。代码生成是这条链的最后一环,而且是落差最大的一环:从「无限虚拟寄存器」降到「有限物理寄存器」。

为什么这一步特别难?因为前面所有降低都是「一对一或一对多的结构翻译」——一个 `if` 翻成两个跳转,一个 `a + b` 翻成一条 IR 加法。它们是局部的、可组合的。但寄存器分配不是:**它是一个全局资源调度问题**。第 7 个虚拟寄存器要不要进物理寄存器,取决于此刻另外几个值还活不活、之后还要不要用。这是 NP 完全问题(精确最优解的计算复杂度随规模指数爆炸)的近亲——真实编译器后端用的图着色寄存器分配(graph coloring)就是把它归约成图着色问题。

我们这一版走最朴素的路线:线性扫描(linear scan)。

### 线性 IR 长什么样

先看降低的入口。`lowerExprToIR` 把一棵算术表达式后序遍历成三地址码:

```ts
function lowerExprToIR(e: Expr, out: IRInst[], fresh: () => number): number {
  if (e.kind === "Literal" && typeof e.value === "number") {
    const dst = fresh();
    out.push({ op: "const", dst, imm: e.value });
    return dst;
  }
  if (e.kind === "Binary") {
    const a = lowerExprToIR(e.left, out, fresh);
    const b = lowerExprToIR(e.right, out, fresh);
    const dst = fresh();
    // ...
    out.push({ op, dst, a, b } as IRInst);
    return dst;
  }
  // ...
}
```

关键在 `fresh()`:每个结果都拿一个新编号,从不复用。这就是「无限虚拟寄存器」的假象——降低阶段根本不操心寄存器够不够,把这件事整个推给后面的分配器。**把「翻译」和「分配」解耦**,是后端能模块化的根本原因:优化器(第 06 章)可以自由地造临时变量,不用担心物理资源;分配器只看 IR,不关心它从哪来。

### 为什么用「右深」树逼 spill

代码里特意构造右深树而非左深,注释讲得很直白:

```ts
// WHY 右深而非左深: lowerExprToIR 是后序遍历(post-order),先把整棵右子树降完才能做外层 +。
// 右深树意味着进入最深一层前,c1..c_{n-1} 的左操作数已全部算出、全在等待相加 —— 它们同时
// 活跃。左深树相反: 每步算完左侧累加值立刻被下一个 + 消费,活跃集恒为 2~3,逼不出 spill。
```

这是个值得停下来想的点。`1 + (2 + (3 + ...))` 这棵树,要算最外层的 `1 + (...)`,必须先把整个括号里的东西算完。而 `1` 这个操作数,从第 0 条指令就被算出来,一直等到最后一条加法才用上——它的**活跃区间(live interval)**横跨整个程序。所有的左操作数 `1, 2, 3, ...` 都这样同时挂着。9 个值同时活跃,而机器只有 4 个寄存器,spill 不可避免。

换成左深 `((1 + 2) + 3) + ...`,每一步的累加结果立刻被下一步消费,活跃集永远是 2~3 个,4 个寄存器绰绰有余,一次 spill 都不会发生。**同一段计算,只因为求值顺序不同,寄存器压力天差地别**——这正是真实编译器里「指令调度影响寄存器分配」的最小可见案例。后端不是先调度再分配两个独立步骤,它们互相纠缠。

## 线性扫描寄存器分配 + spill

线性扫描的算法骨架三步:算活跃区间 → 顺序扫描维护空闲寄存器池 → 池空了就 spill 一个。先看活跃区间怎么算:

```ts
function computeLastUse(ir: IRInst[]): Map<number, number> {
  const lastUse = new Map<number, number>();
  ir.forEach((inst, idx) => {
    if (inst.op !== "const") {
      lastUse.set(inst.a, idx);
      lastUse.set(inst.b, idx);
    }
    if (!lastUse.has(inst.dst)) lastUse.set(inst.dst, idx);
  });
  return lastUse;
}
```

朴素版只记「最后一次被读在哪条指令」(`lastUse`),不记完整区间——对线性扫描够用了:一个虚拟寄存器从被定义到 `lastUse` 之间是活的,这期间它占的物理寄存器不能给别人;过了 `lastUse`,立刻释放回池。

核心是 `allocReg`:要一个寄存器,池里有就拿,没有就 spill:

```ts
const allocReg = (idx: number, protect: Set<number>): number => {
  if (free.length > 0) return free.pop()!;
  // 选 victim: 当前驻留、不在 protect、lastUse 最晚的那个。
  let victimV = -1, victimP = -1, victimLast = -1;
  for (const [v, p] of vreg2preg) {
    if (protect.has(p)) continue;
    const lu = lastUse.get(v) ?? -1;
    if (lu > victimLast) { victimLast = lu; victimV = v; victimP = p; }
  }
  // ...
  code.push({ m: "STORE", slot, rs: victimP });
  spillCount++;
  return victimP;
};
```

spill 的受害者选择策略是「`lastUse` 最晚的那个」——即「最不急着用的」。直觉是:把最久之后才会再用到的值赶下去,这样近期内不用再 load 它回来。这正是经典线性扫描论文(Poletto & Sarkar 1999)的核心启发式。注意 `protect` 集合:本指令正在用的两个操作数寄存器不能被 spill 抢走,否则会把正在算的数覆盖掉。

### 生成的指令长这样

跑出来前 18 条:

```
 0  LOADI  r0, #1
 1  LOADI  r1, #2
 2  LOADI  r2, #3
 3  LOADI  r3, #4
 4  STORE  [slot 0], r0   ; spill
 5  LOADI  r0, #5
 6  STORE  [slot 1], r1   ; spill
 7  LOADI  r1, #6
 ...
17  ADD    r2, r0, r1
```

读这段:前 4 条把 `1,2,3,4` 装满 4 个寄存器。第 5 条要装 `5`,但 4 个寄存器全占着——于是先 `STORE [slot 0], r0`(把 `1` 压栈),腾出 `r0` 装 `5`。每多一个常量就 spill 一个。这就是 spill 的机制:**寄存器不够 → 把活着的值搬到内存 → 用到时再搬回来**。每次搬都是额外的内存访问指令,而内存比寄存器慢一两个数量级。

数字说话:

```
物理寄存器数        = 4
IR 用到的虚拟寄存器数 = 19
spill 到栈的次数(STORE) = 7
用到的栈槽数          = 7
同一程序给 32 个寄存器时 spill = 0 (寄存器够用,无需溢出)
```

最后一行是关键对照:同一段 IR,寄存器从 4 个加到 32 个,spill 直接归零。**spill 不是程序的固有属性,是「寄存器压力 > 物理供给」的产物**。这也解释了为什么 x86-64 从 8 个通用寄存器扩到 16 个、AArch64 给到 31 个——多给寄存器,直接少很多 spill。但寄存器是芯片上最贵的资源(每个都要进指令编码、进旁路转发网络),不可能无限给。后端永远在「寄存器压力」和「物理供给」之间做工程权衡。

### 为什么后端是真编译器的工程量大头

我们这一版线性扫描,加上 codegen,核心逻辑大概一百多行。但它砍掉了真实后端几乎所有的硬骨头:

- **指令选择(instruction selection)**:我们只有 `LOADI/ADD/SUB/MUL/DIV` 五条指令,一个 IR 加法对一条机器加法。真实 ISA(指令集)有成百上千条指令,一个 IR 操作可能对应几十种机器指令组合,选哪种(考虑寻址模式、立即数范围、是否能融合)本身是个树覆盖/DAG 匹配问题。LLVM 的 SelectionDAG、GCC 的 RTL 都是为这件事造的庞然大物。
- **指令调度(instruction scheduling)**:我们按降低顺序直接发射。真机器要考虑流水线、乱序执行窗口、内存延迟,重排指令以填满执行单元。
- **更优的寄存器分配**:线性扫描快但糙,真实编译器在 `-O2` 以上多用图着色,再加上 live range splitting、rematerialization、coalescing 等一堆优化,把 spill 压到最低。
- **调用约定(calling convention)、栈帧布局、异常表、调试信息**……

前端(第 01-04 章)把源码变成 IR,逻辑相对干净——语法和类型规则是有限的、声明式的。后端是从「无限抽象资源」往「有限物理资源 + 复杂硬件约束」上硬塞,而且每一步都要在「编译速度」「生成代码质量」「正确性」之间权衡。这就是为什么 LLVM 的 backend 比 frontend 大几倍,为什么「编译器工程师」这个岗位招聘里后端经验比前端值钱。

## 对拍:怎么证明生成的代码是对的

代码生成最危险的地方是:**它能跑出一个数,但你不知道那个数对不对**。所以本章的硬不变量是端到端对拍——生成的机器码跑出来的结果,必须等于一个独立参考实现的结果。

第一条对拍线:机器执行 vs IR 直接求值。`evalIR` 是 ground truth(基准真值),它假装寄存器无限、不做任何分配,直接照着 IR 算:

```ts
function evalIR(ir: IRInst[]): number {
  const v = new Map<number, number>();
  for (const inst of ir) {
    if (inst.op === "const") { v.set(inst.dst, inst.imm); continue; }
    const a = v.get(inst.a)!, b = v.get(inst.b)!;
    const r = inst.op === "add" ? a + b : /* ... */;
    v.set(inst.dst, r);
  }
  return v.get(ir[ir.length - 1].dst)!;
}
```

然后 `runMachine` 真的去 fetch-decode-execute 那 33 条机器指令(含 spill 的 STORE/LOAD)。两条路结果:

```
IR 参考求值(无限寄存器) = 55
寄存器机执行结果         = 55
assertEq 通过: 朴素线性扫描分配 + spill 后,结果不变 ✓
```

55 = `1+2+...+10`。spill 把值搬来搬去,但**搬运不改变值**——这就是分配器正确性的可观测证据。

第二条对拍线接回第 07 章。本章硬要求「fib 结果与第 07 章 VM 一致」:

```
参考 VM fib(20)        = 6765
机器路径 fib(20)        = 6765
assertEq 通过: fib 两条独立路径同值 ✓
```

> NOTE: 这里要诚实标注一个简化。我们的 codegen 那条路目前只覆盖纯算术表达式,不含函数调用的控制流(那需要再造一套 call/return/栈帧的 codegen,超出本章演示焦点)。所以 fib 的对拍是:「参考解释器(第 07 章 VM 的语义替身)算出 fib(20)=6765」对「把这个 6765 编码成一棵算术 IR、由机器路径跑出来」。两条独立路径得同值,验证的是机器路径不会破坏一个由真 fib 算出的数。这是 toy 简化,真实编译器的 fib codegen 会完整生成递归调用,不需要这层迂回。

## 全栈记分卡

到这里全书的流水线跑通了。把每一阶段的实测数字汇成一张表——这是本机实测,toy 输入,绝对值偏乐观,**可迁移的是相对趋势不是绝对数**:

```
┌─────────────────────────────┬──────────────────────────────┐
│ 阶段 / 指标                     │ 值                            │
├─────────────────────────────┼──────────────────────────────┤
│ lexer 吞吐                    │ 23,408,356 tokens/s          │
│   (token 数)                 │ 40 tokens, 中位 0.00171 ms     │
│ parser 吞吐                   │ 11,076,282 nodes/s           │
│   (AST 节点数)                 │ 24 nodes, 中位 0.00217 ms      │
│ 类型检查命中率                     │ 3/3 算术节点纯数值 = 100%           │
│ 优化前 IR 指令数                  │ 19 (虚拟寄存器, 每运算一条)            │
│ 优化后→机器指令数                   │ 33 (含 7 条 spill)             │
│ VM 解释 fib(20)               │ 6.1689 ms (本机, 树遍历)          │
│ codegen 执行(直线码)             │ 0.00032 ms (本机)              │
│ 加速比 (相对量)                   │ 19,255x (直线机器码 vs 树遍历解释)     │
└─────────────────────────────┴──────────────────────────────┘
```

怎么读这张表,逐行说:

- **lexer 比 parser 快约一倍**(2340 万 vs 1108 万/秒)。这符合直觉:词法分析是字符级状态机,基本是顺序扫描;语法分析要建树、递归下降、维护优先级,每个节点干的活更多。两者吞吐都在千万级,说明**前端不是瓶颈**——真实编译里前端时间占比通常个位数百分比。
- **19 条 IR → 33 条机器指令**,膨胀 74%。这 14 条增量全是 spill 引入的 STORE/LOAD。**优化(第 06 章)在 IR 上削减指令,代码生成又因资源约束加回来一批**——后端的开销不在你的源码里,而在硬件的有限性里。
- **加速比 19,255x 这个数,必须打个大星号**。代码里的注释写得很死:

```ts
// 注意: 这两个不是同一计算,加速比反映的是「直线机器码 vs 树遍历解释」的量级差,不是
// 「同一 fib 两种实现」的公平对比。诚实标注。
```

VM 那 6.17ms 是树遍历解释器递归算 fib(20)——每个节点都要 switch、查环境、递归;codegen 那 0.00032ms 是机器跑一条直线算术指令序列。**这不是同一个计算的公平 A/B**,它对比的是两类执行模型的量级差距:「解释一棵树」和「执行一串直线指令」根本不是一个量级的工作量。这个数字诚实的用法是:它告诉你「解释执行」和「执行原生指令」之间隔着几个数量级,而不是「我的编译器让 fib 快了两万倍」。

记分卡的价值不在绝对数(toy 样例没有真实大文件的 cache miss、GC 压力,绝对吞吐必然偏乐观),而在**让你对各阶段的相对开销和它们之间的张力有量化的感觉**:前端便宜、优化省指令、后端因资源约束补开销、解释 vs 原生差几个量级。

## 失败模式:不报错的算错,是后端最难查的 bug

寄存器分配出 bug 时,最可怕的不是崩溃——崩溃好查。最可怕的是**它能跑完、给你一个看起来完全合理的数,但那个数是错的**。本章故意写了个坏分配器演示这件事:

```ts
function codegenBuggy(ir: IRInst[], nregs: number): CodegenResult {
  let rr = 0; // round-robin 指针,无脑轮转,不看谁还活着。
  const take = (): number => { const p = rr; rr = (rr + 1) % nregs; return p; };
  for (const inst of ir) {
    // ...
    const rd = take(); // BUG: rd 可能正压着某个还要用的中间值,读前被覆盖。
    // ...
  }
}
```

它的错误是「见空就抢」:轮转着分配寄存器,完全不看被覆盖的寄存器里那个值后面还要不要用。结果:

```
正确分配(线性扫描)结果 = 55  (= 真值)
坏分配(round-robin 无视活跃区间)结果 = 518
差值 = 463 (非零 = 静默算错, 这是危险信号)
```

518 不是随机数,不是崩溃码,它是「按错误的寄存器内容算出来的一个确定的错数」。如果这是生产编译器,你的程序会安静地算错,没有任何报错,直到下游某个断言或用户投诉才暴露。结论写在代码注释里,值得背下来:

> 寄存器复用必须基于活跃区间(live interval)。「哪个寄存器现在空」不够——必须问「这个值之后还要不要用」。

这是个深刻的区分。坏分配器问的是「**当前**这个寄存器有没有人占」,正确分配器问的是「这个值的活跃区间**之后**还覆不覆盖到现在」。前者是空间快照,后者是时间区间。寄存器分配本质上是一个**区间冲突问题**——两个活跃区间重叠的值不能共用一个寄存器。线性扫描的全部正确性,就建立在 `lastUse` 算对、释放时机算对这件事上。这也是为什么真实编译器的寄存器分配模块测试覆盖极重:一个 off-by-one 的活跃区间,产出的是静默错码,不是 crash。

## ⚡ 开放问题:解释慢、原生快,中间那条路怎么走

记分卡那个 19,255x,把一个真问题摆到了桌面上:**解释执行慢得多,原生执行快得多,那为什么不全编译成原生?**

因为编译本身有成本。把一段代码编译到原生(走完本章这套降低 + 分配 + 指令选择),要花时间;对只跑一次的代码,编译时间可能比解释执行还长。这就是动态语言(JS / Java / Python)面对的两难:

- **纯解释**(我们第 07 章的 VM):启动快、没有编译开销,但每条指令都慢,热循环里亏死。
- **纯 AOT 编译**(ahead-of-time,提前编译,我们本章这条路):运行快,但启动有编译延迟,而且对动态类型的语言,编译期根本不知道变量是什么类型,没法生成高效原生码。

真实系统的答案是 **JIT(just-in-time,即时编译)**:先解释执行,同时统计哪些代码是热点(反复跑的循环、频繁调用的函数),只把热点编译成原生码,冷代码继续解释。V8(Chrome 的 JS 引擎)有多层:Ignition 解释器 → Sparkplug 基线编译 → Maglev/TurboFan 优化编译,代码越热爬得越高。JVM 的 HotSpot 是 C1(快编译)+ C2(深优化)两层。**这是一条工程上极其精巧、理论上仍无通用最优解的折中路线**:

- 编译阈值定多高?太低则浪费时间编译只跑几次的代码,太高则热代码迟迟得不到加速——目前靠启发式调参,没有理论最优。
- 优化编译器会基于「观察到的类型」做投机优化(speculative optimization),比如假设这个变量永远是整数。一旦假设被打破(deoptimization,去优化),要安全地从原生码退回解释器、恢复完整状态——**OSR(on-stack replacement,栈上替换)和 deopt 的正确性是 JIT 工程里最难的部分之一**,至今是研究活跃区。
- 编译用哪个线程、占多少内存、和 GC 怎么协调,都是没有银弹的权衡。

本章你亲手测出的「解释 vs 原生」的量级鸿沟,就是 JIT 存在的全部理由——它是想吃到原生速度、又不想付 AOT 启动代价的人,被逼出来的折中。如果你想往编译器纵深走,JIT 是下一站:从这条自己写的「解释器(第 07 章)」和「原生 codegen(本章)」出发,中间那座桥怎么搭,正是工业界编译器最硬核、也最值钱的地方。

## 就业冲刺:用这条流水线讲透每个考点

语言/工具链岗(编译器、运行时、语言基础设施)的面试,最忌空谈概念。你写过这条流水线,最大的优势是**每个考点都能落到代码级**。下面是把全书每阶段映射到典型考点的话术骨架——核心打法是:概念一句话 → 我在哪一章亲手实现过 → 一个具体的失败模式 / 权衡。

- **「讲讲词法分析」**:不要背「词法分析把字符流变成 token 流」。讲:我手写过状态机扫描器(第 01 章),没用正则——因为正则引擎的开销会污染吞吐测量,本机实测词法约 2300 万 token/秒,比语法分析快一倍,所以前端从来不是瓶颈。失败模式:遇到非法字符要立即抛、不能静默跳过,否则后续 parser 的错误定位全偏。
- **「递归下降和优先级爬升」**:讲第 02 章。我用优先级爬升(precedence climbing)处理二元运算符,数字越大绑得越紧;赋值是右结合、最低优先级,单独处理。能讲清「为什么右深树和左深树寄存器压力不同」(本章那段)的人,面试官立刻知道你真懂求值顺序。
- **「类型检查 / 静态分析」**:第 04 章。讲你怎么遍历 AST 判定哪些算术节点能静态确定是纯数值——这决定了能不能走快路径。
- **「IR 和优化」**:第 05、06 章。重点讲「为什么要用无限虚拟寄存器」——把翻译和分配解耦,优化器才能自由造临时变量。优化在 IR 上削指令,我能给出优化前后的指令数对比。
- **「寄存器分配」**——这是语言岗的核心考点,往死里准备。讲线性扫描算法三步(本章),讲活跃区间和 spill,然后**主动抛那个失败模式**:坏分配器不崩溃、能跑完、给个看似合理的错数,因为它问「寄存器现在空不空」而不问「这个值之后还要不要用」。能讲出「静默算错比崩溃更危险」的人,展示的是工程成熟度,不只是算法记忆。
- **「解释 vs 编译 vs JIT」**:用本章记分卡那个量级差切入,讲 JIT 是被这个鸿沟逼出来的折中,讲 V8 的多层、deopt 的难处(上面 ⚡ 那段)。

整套话术的底色是:**你不是读了 survey、背了概念,你写过、测过、踩过坑**。面试官问任何一环,你都能说「我在第 N 章实现过,代码在 `examples/compiler-from-scratch/src/stageNN-*.ts`,跑出来的数是 X,踩过的坑是 Y」。这就是这本书和「AI 批量生成的 survey 稿」最根本的区别——它绑在可跑代码和实测数字上,不是漂在概念层。

至此,从第 01 章一行源码字符串,到本章一条可执行的机器指令,整条流水线走完了。每一阶段你都亲手降低过一次、测过一次、对拍过一次。下一站如果还想往深走,前面 ⚡ 指的 JIT 是工业界编译器最硬的那块骨头——这条手写流水线,正是啃它的最好起点。
