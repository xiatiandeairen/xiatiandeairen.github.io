先看一个你大概率写错过的程序:`let a = a;`。它在语法上完全合法——第 02 章的 parser 会愉快地给你一棵 `Let { name: "a", init: Var("a") }` 的 AST,没有半点抱怨。但它在语义上是错的:在 `a` 自己的初始化表达式里读 `a`,此刻 `a` 还没有值。JS 早期没拦住这类写法,`var a = a` 静默给你 `undefined`;现代引擎的 `let a = a` 才在运行期抛 TDZ(暂时性死区)错误。本章的 resolver(名称解析器)做的事,是把这个错从运行期提前到编译期——程序还没跑,就告诉你第几行的 `a` 读了一个没就绪的自己。

这就是语义分析这一关的定位:**语法器只证明「这串 token 符合文法」,它对名字一无所知**。`print x` 在语法上永远合法,哪怕 `x` 从未声明。谁声明了谁、哪个引用绑到哪个声明、跨没跨函数边界——这些"意义"问题,parser 不管,得有人专门走一遍 AST 算清楚。算清楚的产物,后面两章直接吃:第 04 章类型检查要知道 `x` 是哪个声明才能查它的类型,第 05 章 IR 生成要知道 `x` 该去栈上第几个槽取值。

本章对应配套仓的 `examples/compiler-from-scratch/src/stage03-resolver.ts`,一个独立可跑的 stage。它手搭一棵对应 `counter.lox` 的 AST(本章不 import parser,各 stage 互不依赖),跑 resolver,把 depth/slot/捕获集合全打印出来给你看死,并用 14 项断言对拍。下面所有数字都来自它的真实运行输出。

## 名称解析在算什么:depth 与 slot

把"`x` 指向哪个声明"推迟到运行期,代价有两条,stage 文件头注释把它说得很直白:

> 1. 每次取变量都要顺着作用域链动态查名字 —— O(深度) 而非 O(1)。
> 2. 拼写错 / 用了未声明的变量,要等真跑到那行才炸,而不是编译期一次报全。

resolver 在执行前一次性回答「每个变量引用绑定到哪个声明」,把答案钉死在 AST 节点上。答案是一对数:

- **depth**(深度):从引用所在的作用域,往外跳几层能到达声明那一层。同层 = 0,父层 = 1,以此类推。
- **slot**(槽号):变量在它所属作用域内的声明序号(0-based),同层内单调递增、不复用。

有了这对 `(depth, slot)`,第 07 章的 VM 取一个局部变量就是"往外跳 depth 层、取第 slot 个槽",O(1) 定位,不用再拿名字去哈希表里一层层查。这是为什么这一步值得单独做一遍:它把运行期每次取变量的 O(深度) 名字查找,前移成编译期一次性的下标计算。

核心数据结构是一摞作用域,每层一个 `Scope`:

```ts
interface Scope {
  readonly declared: Map<string, number>; // name -> slot
  readonly defined: Set<string>; // name 的初始化器已求值完
  next_slot: number; // 下一个声明拿到的 slot,单调递增不复用
}
```

解析一个变量读引用,就是从栈顶往栈底逐层找名字,命中即停——这保证了**就近绑定**(后面讲 shadowing 时会看到它的作用):

```ts
private resolveVar(node: VarExpr): void {
  for (let i = this.scopes.length - 1; i >= 0; i--) {
    const scope = this.scopes[i];
    if (!scope.declared.has(node.name)) continue;
    // ...命中:回填 depth/slot
    const depth = this.scopes.length - 1 - i;
    this.bind(node, depth, scope.declared.get(node.name)!);
    this.recordCaptureIfFree(node.name, i);
    return;
  }
  // 局部各层都没有 → 查全局集
  if (this.globals.has(node.name)) return;
  this.diags.error(`undefined variable '${node.name}'`, node.span, RESOLVER_STAGE);
}
```

### 全局为什么不进作用域栈

`resolveVar` 末尾有个看着多余的分支:局部各层都没找到时,再查一个独立的 `globals` 集合,命中就什么都不做(`depth` 留 `undefined`)。为什么不把顶层也压成一个普通 `Scope`?stage 注释给了理由:

> Lox 语义里全局变量走 VM 的全局表(按名字查),解析结果 depth 必须留 undefined。若给顶层压普通 scope,顶层引用会算出 depth=0,与「全局走名字表」的运行期约定冲突。

这是一条**设计决策,不是疏忽**。Lox 的全局是后期绑定的:函数体可以引用定义在它后面的全局函数(互递归)。所以 `resolve` 先扫一遍顶层 `let` 把名字全注册进 `globals`,再正式解析,否则前向引用会被误报成 undefined。把全局单独成集的好处是:存在性检查(报不报 undefined)和 depth 计算(走不走栈)解耦——全局参与前者、不参与后者。

> ⚡ **开放问题:全局后期绑定到底该多晚?** 这里的 "先扫一遍顶层" 是个偷懒但够用的近似。真实语言里,"一个名字在它被引用时到底绑定到哪个声明" 在有 `eval`、动态 `import()`、运行期改全局对象、REPL 增量定义的语言里**没有通用静态解**——JS 的 `with` 语句和 sloppy-mode 全局对象让引擎至今要保留运行期作用域查找的退路,V8 这类引擎只能靠 inline cache 在运行期赌"这个名字这次还绑到上次那个 slot",赌错就 deopt。静态语言(Rust/Go)靠禁掉这些动态特性才换来 100% 编译期解析。我们这个玩具语言两头都不沾,所以能用"扫一遍"蒙混;但"名称解析在动态语言里能做到多静态"仍是引擎工程的活跃战场,没有银弹。

## 失败模式一:`let a = a` 与两阶段 declare/define

现在回到开篇那个 `let a = a;`。要在编译期抓住它,resolver 用了一个关键技巧:**把"声明一个名字"拆成两阶段**——declare(占位,标记未就绪)和 define(就绪)。看 `Let` 语句的解析:

```ts
case "Let":
  // 两阶段顺序就在这三行:先 declare(占位、未就绪)→ 解析初始化器 → define(就绪)。
  // 顺序不能换:若先 define 再解析初始化器,`let a = a` 就会被当成合法自引用。
  this.declare(stmt.name, stmt.span);
  this.resolveExpr(stmt.init);
  this.define(stmt.name);
  return;
```

顺序是命门。`declare` 把 `a` 放进当前作用域(占了 slot),但**不**标 `defined`。然后解析初始化器 `a` 时,`resolveVar` 在当前层找到了 `a`(declared 命中),但一查就绪位发现没就绪:

```ts
if (!scope.defined.has(node.name)) {
  this.diags.error(
    `cannot read local variable '${node.name}' in its own initializer`,
    node.span,
    RESOLVER_STAGE,
  );
}
```

stage 的 demo ② 跑这个程序(包在块里让 `a` 成为局部,因为全局层不开 scope),输出:

```
── demo ② 两阶段:抓自引用 let a = a; ──
  错误数 = 1
    error[resolver]: cannot read local variable 'a' in its own initializer
```

恰好 1 个错,措辞精确到"在自己的初始化器里读自己"。注意它**不是**报"undefined variable":`a` 确实声明了,只是还没就绪——两阶段让 resolver 能区分这两种语义完全不同的错误。

### 为什么这不是过度设计:单遍解析器的漏报

你可能觉得两阶段太讲究了——新手最自然的写法是"声明一个变量时立刻可见"(declare 即 define,没有未就绪状态)。stage 的 demo ⑤ 专门写了一个这样的朴素单遍解析器当对照组,跑同一个 `let a = a;`:

```ts
const visitStmt = (s: Stmt): void => {
  if (s.kind === "Let") {
    scopes[scopes.length - 1]?.set(s.name, 0); // 声明即可见(致命差异)
    visit(s.init); // 此刻 s.name 已可见 → `let a = a` 解析里的 a 命中自己
  }
  // ...
};
```

声明那行先把 `a` 设成可见,再解析初始化器,于是初始化器里的 `a` 命中自己、静默解析成功,把一个"读未初始化内存"的 bug 放行到运行期。demo ⑤ 的输出把这个漏报钉死:

```
── demo ⑤ 失败模式:单遍(声明即可见)漏报 let a = a; ──
  单遍解析器报告的错误数 = 0  ← 漏!应为 1
  两阶段解析器报告的错误数 = 1  ← 正确抓到
  结论:declare→define 两阶段不是过度设计 —— 没它就放行『读未初始化』。
```

这就是"先讲机制怎么坏、再讲为什么这么设计"的范本:declare/define 两阶段不是教科书摆样子,它是抓 `let a = a` 这类自引用的**必要条件**——少了它,这类 bug 会一路漏到运行期变成 `undefined` 或 TDZ 错。JS 当年正是栽在这里。

## 失败模式二:一轮收齐多类错误,不首错即停

新手编译器的第二个毛病是首错即停——遇到第一个错就 `throw`,用户改一个重编一次只看见下一个错。专业编译器都"收集多个错误一次报全"。本书把这件事做成全书统一基础设施 `DiagnosticBag`(诊断收集器),它的不变量写在 `core/diagnostics.ts`:

> 报错不抛异常(默认): error()/warning() 只是 push 进列表,编译继续往下跑,尽量多收集错误。

resolver 全程不抛异常,所有名字错误都进袋子。stage 的 demo ③ 在一个块里塞了三类不同的错——重复声明、未定义读、给未声明赋值——一轮跑完收齐:

```
── demo ③ 一轮收集多类错误:重复声明 / 未定义 / 给未声明赋值 ──
  本轮收集到 3 个错误(没有首错即停):
    - duplicate declaration of 'x' in this scope
    - undefined variable 'ghost'
    - assignment to undeclared variable 'nope'
```

三类错对应 resolver 里三个不同的检查点。重复声明在 `declare` 抓,且抓到后**不覆盖原 slot**:

```ts
if (scope.declared.has(name)) {
  this.diags.error(`duplicate declaration of '${name}' in this scope`, where, RESOLVER_STAGE);
  return; // 不覆盖原 slot:保留首个声明的绑定,避免后续引用解析到半个状态
}
```

读未定义和写未定义分两个措辞:`resolveVar` 报 `undefined variable`,`resolveAssign` 报 `assignment to undeclared variable`。stage 注释解释为什么要分:

> 读未定义和写未定义都报,但写的诊断措辞不同,便于用户区分意图。

收集完还要渲染成人能读的样子。`DiagnosticBag.report()` 给每条错配上源码行和下划线,**渲染顺序 = 报告顺序**(不重排,便于按发现顺序读):

```
counter.lox:1:1: error[resolver]: duplicate declaration of 'x' in this scope
  // 闭包计数器: 检验「一等函数 + 词法闭包 + 捕获可变变量」。
  ^
```

(下划线落在 offset 0 是因为本章的手搭 AST 用零宽占位 span——本章只测名称解析,不测诊断的行列精度;真实行列由有 parser 的章节提供。)

## shadowing:就近绑定,内层遮蔽外层

`resolveVar` 那个"从栈顶往栈底逐层找、命中即停"的循环,顺手解决了 shadowing(遮蔽:内层同名变量盖住外层)。看 demo ④ 的程序:外层一个 `x`、内层块里又声明一个 `x`,内外各 `print x` 一次。

```
── demo ④ shadowing:内层遮蔽外层,引用绑到最近声明 ──
  内层 print x -> depth=0, slot=0 (绑内层 x)
  外层 print x -> depth=0, slot=0 (绑外层 x)
  [OK] 两个 x 各绑到最近一层,无串扰
```

两个引用 depth 都是 0,但绑的是**不同作用域**的 `x`:内层的引用在内层就找到了 `x`(depth=0,绑内层),外层的引用所在那层就是外层(depth=0,绑外层)。"命中即停" 保证了任何引用都绑到从它出发能最先遇到的那个声明,这正是词法作用域(lexical scope:作用域由代码书写位置决定,不由运行期调用关系决定)的定义。没有这条,内外两个 `x` 就会串扰。

## 闭包不是魔法:自由变量捕获算给你看

现在到本章的重头戏。看这个经典闭包:

```
fn makeCounter() {
  let count = 0;
  fn next() { count = count + 1; return count; }
  return next;
}
let c = makeCounter();
print c(); print c(); print c();
```

`makeCounter` 返回内层函数 `next`,而 `next` 用了 `count`——可 `count` 是 `makeCounter` 的局部变量,`makeCounter` 一返回,它的栈帧就该销毁了。为什么 `c()` 调三次还能记住 `count` 从 1 数到 3?课堂上常说"闭包捕获了环境",听着像魔法。resolver 把这层魔法拆开给你看:它在**编译期**就算出 `next` 引用了哪些不属于自己的变量。

定义很机械:**函数体引用的名字,减去函数自己声明的名字(参数 + 内部 let + 内部 fn 名),剩下的就是自由变量**——必须被闭包捕获的那些。实现靠一个 `FnFrame`:

```ts
interface FnFrame {
  readonly name: string;
  readonly free: Set<string>; // 捕获的自由变量(来自外层的引用)
  readonly scope_floor: number; // 本函数最外层作用域在 scopes 栈中的下标
}
```

`scope_floor` 记下函数定义时的作用域栈高度。解析一个引用时,如果它解析到的声明所在 scope 下标 **小于** 某个函数的 `scope_floor`,说明这个引用跨出了那个函数边界 → 那个函数(及中间所有函数)必须捕获它:

```ts
private recordCaptureIfFree(name: string, scope_index: number): void {
  for (const frame of this.fn_stack) {
    if (scope_index < frame.scope_floor) frame.free.add(name);
  }
}
```

stage 的 demo ① 把 `next` 体里每个 `count` 引用的解析结果和最终捕获集合全打印出来:

```
── demo ① 闭包计数器:解析 (depth, slot) + 捕获自由变量 ──
  resolve 后 next 体 S 表达式 = ((set! count@1 (+ count@1 1)) (return count@1))
    引用 'count' (Assign) -> depth=1, slot=0
    引用 'count' (Var) -> depth=1, slot=0
    引用 'count' (Var) -> depth=1, slot=0
  next 捕获的自由变量集合 = {count}
  makeCounter 捕获的自由变量集合 = {}
```

三个 `count` 引用 depth 全是 1——它们都跨了一层(从 `next` 跳到 `makeCounter`)才找到声明。slot 全是 0——`count` 是 `makeCounter` 作用域里的第一个声明。所以 `next` 捕获 `{count}`,而 `makeCounter` 自己捕获 `{}`(因为 `count` 是它自己的局部,没跨出它的边界)。

"闭包记得住 `count`" 的真相到这里就清楚了:resolver 在编译期算出 `next` 要捕获 `count`,后续阶段据此把 `count` 从栈上"提升"成一个堆上的、`next` 闭包对象持有引用的格子。它捕获的是**引用**(depth=1 指向同一个 `count`),不是值——所以三次调用对同一个 `count` 累加,这正是 `set! count@1` 那个 S 表达式里 `@1` depth 标注的含义。这个捕获集合就是第 05 章生成闭包对象、决定哪些变量要"逃逸"到堆上的直接依据。

注意 `Assign` 节点(`count = count + 1` 里的写)也走捕获登记——`resolveAssign` 同样调 `recordCaptureIfFree`。这很重要:捕获的是可变变量,读和写都得指向同一个被提升的格子,否则计数器加不动。

> ⚡ **开放问题:捕获引用 vs 捕获值,谁对?** 本章默认捕获引用(因为 Lox 闭包要能改 `count`)。但这个选择有著名的坑:循环里建闭包,`for (i...) closures.push(fn(){ return i })`,所有闭包捕获**同一个** `i`,循环结束后全返回最终值——JS 的 `var` 时代和 Go 1.22 之前的循环变量都栽过这个,Go 干脆在 1.22 改了语言语义让每轮迭代新建变量。捕获值能避开这坑,但又丢了"闭包能改外层变量"的能力。**没有放之四海皆准的答案**:Rust 用 `move` 让你显式选,Swift 用 capture list,JS/Python 靠 `let` 块级作用域绕,Go 改了默认。一个语言到底默认捕获引用还是值、要不要给开发者显式控制权,至今是语言设计争论点,resolver 这层只是忠实执行语言定下的语义,定不了这个语义本身。

## 这一遍遍历交出去什么

跑完 14 项断言,stage 退出码 0:

```
=== 全部 14 项断言通过 ===
EXIT=0
```

resolver 这一遍在 AST 上留下两样东西,交给下游:

1. **每个 `Var`/`Assign` 节点的 `(depth, slot)`**——第 04 章类型检查靠它找到声明去查类型,第 05 章 IR 生成靠它决定变量去栈上第几个槽。这是"名称解析"四个字的全部产出。
2. **每个函数的自由变量集合**——第 05 章靠它生成闭包对象,决定哪些局部要逃逸到堆上。

值得一提的是实现上的一个洁癖:slot 和自由变量集合是本 stage 的内部产物,没有进 `core/ast.ts` 的 schema,而是挂在 `Symbol` 键的旁路数据上。stage 注释解释为什么——避免污染对拍(`astToSExpr` 序列化时只读 `depth`,不会把这些内部产物打印进 S 表达式)。这是一种受控的局部不安全转换:旁路数据只在本 stage 内部读写,既不改 core schema,又能在本文件断言。

从第 01 章的 token 流、第 02 章的 AST,到本章给 AST 标上"每个名字指向哪个 slot",这条流水线现在已经知道每个程序"长什么样、说了什么、名字指向谁"。还差最后一道语义关:这些名字指向的东西,**类型对不对**——`count + 1` 里 `count` 真是个数吗?那是第 04 章的事。
