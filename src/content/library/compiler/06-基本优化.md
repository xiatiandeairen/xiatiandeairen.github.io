先看一个谁都写过的表达式:`let secs = 2*60*60`。第 05 章把它降成线性 IR(intermediate representation,中间表示——一串三地址指令),长这样:

```
r0 = const 2
r1 = const 60
r2 = const 60
r3 = r0 * r1
r4 = r3 * r2
print r4
```

这两条乘法,源代码里写死的是常量,运行期却每跑一次就老老实实乘一遍。一个写了 `for (i in 0..1000000) print(2*60*60)` 的循环,VM 要做两百万次乘法去算同一个 7200。这不是用户蠢——人就该写 `2*60*60` 而不是手算成 `7200`(可读性差远了)。把这种「编译期能算的就别留到运行期」补回来,正是优化器的活。本章只做两件最经典的:常量折叠(constant folding)和死代码消除(DCE,dead code elimination),但我要求它们**用指令数说话**——优化效果不是「感觉快了」,是确定性可断言的数字。

本章配套代码在 `examples/compiler-from-scratch/src/stage06-optimize.ts`。它自带一套极简线性 IR(而不是直接优化第 02 章的树形 AST),原因下面第一节就讲。所有「指令数 / 删除条数 / 每轮变更计数 / 执行条数」都是代码真数出来的、确定性的;只有墙钟毫秒是本机实测、仅作展示,不进断言。

## 为什么优化的战场是 IR,不是 AST

常量折叠在树上其实做得了:遍历 AST,看到一个二元节点两个孩子都是字面量,就地换成一个字面量节点。但「死代码消除」「不可达块删除」「活跃性分析(liveness——某个值往后还会不会被用到)」这些都是**以指令为单位、以控制流图(CFG)为骨架**的分析,在嵌套的树上根本没法自然表达——树上没有「下一条指令」「这个块的前驱」这种概念。

所以 `stage06` 把树降成一个最小的指令数组 + 基本块,所有 pass(优化遍)都在它上面跑。这套 toy IR 刻意只够撑起本章的四个机制,核心设计有两个:

`examples/compiler-from-scratch/src/stage06-optimize.ts`:

```ts
type Instr =
  | { readonly op: "Const"; readonly dst: Reg; readonly value: number | boolean }
  | { readonly op: "Bin"; readonly dst: Reg; readonly l: Reg; readonly r: Reg; readonly fn: string }
  | { readonly op: "Un"; readonly dst: Reg; readonly a: Reg; readonly fn: string }
  | { readonly op: "Print"; readonly a: Reg } // 副作用: 写 stdout, 无 dst
  | { readonly op: "Call"; readonly dst: Reg; readonly name: string; readonly args: readonly Reg[] };
```

第一,**「有没有副作用」编进了指令种类本身**:`Print`/`Call` 是副作用指令,`Const`/`Bin`/`Un` 是纯指令。DCE 只要看 `op` 就知道能不能删,不需要另查一张副作用表。这个决定在第五节的失败模式里会显出它的分量——它是把「漏判副作用」这种正确性事故堵在类型层面的设计。

第二,**每个虚拟寄存器恰好被一条指令写一次**(SSA-ish,单赋值)。这让活跃性判断退化成 O(1):一个 `dst` 是死是活,只看它有没有被任何指令读过,不必做逐块倒序的数据流迭代。真实编译器(LLVM 那种)的活跃性是逐基本块倒序求不动点的数据流分析;这里因为单赋值 + 跨块用途少,用一遍全局 live 收集近似就够演示机制,且保持确定性。这是 toy 的取舍,后面会标清楚它的局限。

还有一个贯穿全章的不变量:**每个 pass 是函数式的**——输入一个 IR,返回新 IR + 本轮变更计数,不原地改输入。为什么不图省事原地改?因为本章要干两件事:打印「折叠前 / 折叠后」对照,以及不动点迭代时比较「这一轮还有没有变化」。原地改就拿不到「前」那份做对照,也没法干净地数变更。

## ✦ 常量折叠:把运行期乘法削成编译期常量

折叠 pass 的逻辑是顺着指令流走,维护一个「已知常量」的环境(map:寄存器 → 它的常量值)。遇到 `Bin`/`Un`,如果操作数全在环境里有值,就当场算出来、把这条指令替换成一个 `Const`,并把结果记回环境——让下游运算能链式折叠下去。

`examples/compiler-from-scratch/src/stage06-optimize.ts`:

```ts
case "Bin": {
  const lv = env.get(i.l);
  const rv = env.get(i.r);
  if (lv !== undefined && rv !== undefined) {
    const folded = foldBin(i.fn, lv, rv);
    if (folded !== undefined) {
      changed++;
      env.set(i.dst, folded);
      return { op: "Const", dst: i.dst, value: folded };
    }
  }
  return i;
}
```

「记回环境」这一步是链式折叠能成立的关键。`2*60*60` 降出来是两条乘法:`r3 = r0*r1` 和 `r4 = r3*r2`。第一条折出 `r3 = 120` 并写进环境,第二条才能看见 `r3` 是常量 120、进而折出 `r4 = 7200`。stage 实测:

```
折叠后 IR:
  r3 = const 120
  r4 = const 7200
  print r4
折叠掉的运算指令 = 2 条 (2*60 与 *60 两次乘法 → 一个常量 7200)
assertEq 通过: 折叠结果 = 7200, 折叠条数 = 2
assertEq 通过: 1+2+3 链式折叠 = 6
```

两条乘法被折成零条乘法 + 一个写死的 `7200`。这就是 ✦ 的第一半:优化不是玄学,`2*60*60` 在编译期变成 `7200` 是个能断言的事实(`assertEq` 把它钉死了),不是「我觉得快了」。

### 折叠器知道哪里该收手,这恰恰是它正确的地方

`foldBin` 里有几处「不折叠」,值得专门讲——优化器的正确性常常体现在它**克制**的地方:

`examples/compiler-from-scratch/src/stage06-optimize.ts`:

```ts
case "/":
  // 不折叠除以 0: 折成 Infinity 会把「运行期错误」提前固化成一个错误的常量,
  // 改变了程序在该路径上的行为(本应在运行到这里时才出错)。保守放过。
  return r === 0 ? undefined : l / r;
```

除以零不折。如果硬折,JS 里 `1/0` 是 `Infinity`,你就把一个本该在运行到这一行时才抛的错误,提前固化成了一个错误的常量值,程序行为变了。同理,`foldBin` 只在「双数算术 / 双布尔逻辑」时折,混类型(比如 `number + boolean`)一律放过——那是类型检查器(第 04 章)该报错的事,优化器不越权替它做判断。

记住这条判据,它是整章的主线:**优化的红线是保持「可观察行为」不变**。折出一个会改变程序在某条路径上行为的常量,就越线了。

## ✦ 死代码消除:删掉算了又没人用的指令

折叠把运算变成了常量,但有些常量算出来根本没人用——比如源码里写了 `let x = 1+2+3` 然后 `x` 再没被引用。这种「死值」就该删,这是 DCE 的活。

算法分两步:先扫一遍全程序,把任何指令或 terminator(块的出口跳转)读过的寄存器都收进 `live` 集;再保留满足「(a) 有副作用,或 (b) 它的 `dst` 在 live 集里」的指令,其余删掉。

`examples/compiler-from-scratch/src/stage06-optimize.ts`:

```ts
const body = block.body.filter((i) => {
  const hasEffect = i.op === "Print" || i.op === "Call";
  if (safe && hasEffect) return true; // 保守对待副作用
  const dst = "dst" in i ? i.dst : undefined;
  const keep = dst !== undefined && live.has(dst);
  if (!keep) removed++;
  return keep;
});
```

但删死指令前还有一步:折叠会把 `if (true && false)` 的条件折成常量 `false`,这时分支简化(`passSimplifyBranch`)把这个 `Branch` 改写成无条件 `jump els`,`then` 块就成了**没人跳得到的孤岛**。不可达块消除(`passUnreachableBlocks`)从入口做可达性遍历,把走不到的块整个删掉。这是 DCE 之前的清场:不简化分支,可达性遍历会以为两条分支都能走,死块永远删不掉。

stage 跑一个 `if (true && false)` + 一条死的 `x = 1+2+3` 的样例:

```
不可达块删除 = 1 个 (then 块: cond 恒 false 永远走不到)
死指令删除 = 8 条 (含 x=1+2+3 等无人使用的纯计算)
DCE 后:
  entry:
    jump els
  els:
    p2 = const 2
    print p2
    halt
assertEq 通过: 删除不可达块 = 1
```

一个块、八条死指令,删得干干净净,只剩真正会执行、真正有输出的那条路径。

## ⚡ 不动点:为什么不能只跑一遍

这是本章最该被记住的一点。pass 之间是**互相喂材料**的:常量折叠产出新常量 → 让下游运算这一轮还看不见、下一轮才折得动;折叠又把某些 `Branch` 变成可简化的;分支简化让某些块不可达;删块又让某些寄存器不再被引用,使 DCE 这一轮删不掉、下一轮才删得掉。

一句话:**一遍 pass 暴露出来的优化机会,要再跑一遍才吃得到**。所以正确的做法是跑到不动点(fixed point)——反复跑,直到一整轮下来变更计数归零为止。

`examples/compiler-from-scratch/src/stage06-optimize.ts`:

```ts
for (;;) {
  round++;
  const fold = passConstFold(prog);   prog = fold.prog;
  const branch = passSimplifyBranch(prog); prog = branch.prog;
  const ub = passUnreachableBlocks(prog);  prog = ub.prog;
  const dce = passDce(prog, /*safe*/ true); prog = dce.prog;
  const total = fold.changed + branch.changed + ub.removed + dce.removed;
  rounds.push({ round, /* ... */ total });
  if (total === 0 || round >= MAX_ROUNDS) break;
}
```

(注意那个 `MAX_ROUNDS` 上限:正常程序的变更计数单调递减、几轮内归零,远到不了上限。它纯粹是防御——万一某个 pass 有 bug 反复横跳,不至于死循环。这是该有的兜底。)

最能看清「为什么需要多轮」的是一条**非常量**的死代码链:

```
v  = call read_input();   // 运行期未知, 折不掉
d1 = v + v;               // 只被 d2 用
d2 = d1 + d1;             // 只被 d3 用
d3 = d2 + d2;             // 没人用 → 死
print 99;
```

折叠对它毫无帮助(`v` 是运行期才知道的)。第一轮算 live 时,`d1` 被 `d2` 引用、`d2` 被 `d3` 引用——都「活」着,只有链尾 `d3` 真死,删 1 条。删了 `d3`,`d2` 这一轮才暴露成死的;再删 `d2`,`d1` 才暴露……每轮只剥一层。stage 实测:

```
第 1 轮: fold=0 branch=0 ub=0 dce=1 → 合计 1
第 2 轮: fold=0 branch=0 ub=0 dce=1 → 合计 1
第 3 轮: fold=0 branch=0 ub=0 dce=1 → 合计 1
第 4 轮: fold=0 branch=0 ub=0 dce=0 → 合计 0
共 4 轮, 末轮合计 0 → 到达不动点
指令数: 原始 6 → 只跑一遍 5 → 不动点 3
只跑一遍比不动点多留 2 条死指令
```

「只跑一遍」停在 5 条,留着 2 条死指令(`d1`、`d2`)清不掉;跑到不动点才剥到 3 条。这就是 ⚡:折叠暴露死代码、删死代码暴露可折叠常量,得反复跑到再也不变为止。

这里有个诚实的边界要交代:**「逐轮只剥一层」是本章 toy 的单遍全局 live 算法造成的**。真实编译器的活跃性是逐块倒序求数据流不动点的,一遍就能把这条链整体判死、一次删干净。但「pass 之间互喂、需要 phase ordering(遍序)迭代到收敛」这个现象本身是真实且普遍的——LLVM 的 pass manager 就要反复跑一组 pass 直到稳定。⚡ 的**开放问题**也正在这里:**「pass 该按什么顺序、各跑几遍最优」(phase-ordering problem)目前没有通用最优解**——某个顺序在程序 A 上最好,在 B 上可能更差;跑到不动点能保证「不再有这组 pass 能改进的机会」,但换一组 pass、或一个会增大代码体积的 pass(如内联),不动点甚至不一定存在或不唯一。学术界用机器学习、强化学习搜遍序(Google 的 MLGO 就是一例),仍是活跃研究方向,远没有收敛成教科书定论。

## 端到端:静态指令数 / 执行条数 / 加速比

把折叠和 DCE 串起来,放到一个「重复 N 次同一段可全折叠计算、只 print 最后一个结果」的程序上(其余 N-1 个结果都是死值),三个口径一起测:

```
静态指令数: 20,001 → 2 (优化 2 轮)
VM 执行条数: 20,001 → 2
指令削减率 = 100.0% (确定性, 可断言)
输出一致对拍通过: [7200]
VM 执行耗时(本机实测): 优化前 1.1855 ms, 优化后 0.0002 ms
运行时加速比 = 4974.2x (相对量, 可复现)
```

三个口径分清楚:**静态指令数**(代码里有多少条)和 **VM 执行条数**(运行期真正跑了多少条)都从 20,001 削到 2,削减率 100.0%,这两个是确定性的、能断言的。**加速比 4974.2x** 是墙钟实测的相对量——它能复现,但绝对数字偏乐观。

这里必须诚实:这是 toy IR + toy VM 上的合成数据,程序被刻意构造成「全是可折叠常量」,真实程序绝不会这样。可迁移的是**趋势而非倍数**:常量折叠 + DCE 把「运行期重复计算」削成「编译期一次算好」。真实收益取决于你的程序里有多少计算是编译期可知的——通常远小于 4974x,但「把循环里的不变常量计算提到编译期」这个方向永远成立。(顺带一提:VM 执行条数从 20,001 到 2,意味着第 07 章那个完整 VM 拿到的是优化后的 IR——少跑两万条指令,这是后端各阶段叠加的复利。)

## 失败模式:天真 DCE 会删掉你的副作用

现在回收第一节埋的伏笔。DCE 的核心判据是「`dst` 没人用就删」。但有一类指令,它的价值根本不在返回值,而在**副作用**:

```
tmp = call side_effect();   // 返回值 tmp 没人要, 但调用本身有外部效果
print 42;
```

`tmp` 确实没人用。一个「天真 DCE」——只看 `dst` 是否被引用——会理直气壮把这条 `Call` 删掉。后果是灾难性的:那次外部调用消失了,程序的可观察行为变了。stage 用 `safe` 开关并排跑安全版和天真版,把事故坐实:

```
原始程序输出:        [<effect:side_effect>, 42]
安全 DCE 后输出:     [<effect:side_effect>, 42] (删 0 条, 放过 Call)
天真 DCE 后输出:     [] (删 2 条, 含那条 Call)
天真 DCE 丢失的外部效果: [<effect:side_effect>, 42]
assertEq 通过: 安全 DCE 输出不变; 天真 DCE 确实丢了副作用(失败模式坐实)。
```

天真 DCE 把输出整个删空了——连 `print 42` 都因为 `side_effect` 的返回值链被误判而一起没了。**根因**:天真 DCE 把「值没人要」错当成「指令可删」。但 `Print`/`Call` 的价值在副作用,不在返回值。这正是回到本章主线那条红线——**优化必须保持「可观察行为」不变,而不只是「值」不变**。

这也解释了第一节那个设计决定为什么重要:`stage06` 把「有没有副作用」编进了指令种类,DCE 里那句 `if (safe && hasEffect) return true` 是写在种类判断上的硬保护,而不是依赖某个调用方「记得」去检查一张副作用表。把正确性约束放进类型 / 种类,让「漏判」在结构上无法发生——这比写一行注释提醒「记得别删 Call」可靠得多。真实编译器的 DCE 远比这复杂(要分析纯函数、`memcpy` 这类已知库调用、`volatile` 读写……),但判断的内核完全一样:能删的只有「既无副作用、结果又没人用」的指令。

## 一句话收束

优化器不是黑魔法,是会算账的工程:常量折叠把编译期能算的从运行期捞回来,DCE 把算了没人用的删掉,两者用静态指令数和 VM 执行条数把效果钉成可断言的事实(本章 20,001 → 2)。要吃干净优化机会得跑到不动点——折叠喂出死代码、删死代码喂出新常量,而**遍序怎么排最优仍是开放问题**。贯穿始终的红线只有一条:保持可观察行为,删值可以,删副作用就是 bug。这份优化后的 IR 接下来交给第 07 章的字节码 VM,让你亲眼看见省下的两万条指令——优化省下的不是「感觉」,是 VM 少做的真功。
