第 06 章我们把 AST 降到 IR (中间表示) 又做了优化，但优化完的 IR 还是一棵静态的数据结构——它躺在内存里，不会自己动。本章要回答的是整本书最朴素也最关键的问题：**程序到底是怎么"跑起来"的**。答案有两条路。一条是上一章末尾那个树遍历解释器 (tree-walking interpreter)——直接拿着 AST 递归求值；另一条是把 AST 先编译成一串扁平的字节码 (bytecode，紧凑的指令序列)，再写一个虚拟机循环去执行它。这两条路不是"教学版 vs 生产版"的关系——它们是**真实存在的两种工业选型**，CPython、Lua、JVM、V8 的解释器层、Ruby YARV 全都走了第二条。本章用同一个 `fib(25)` 在两条路上跑，给你一个可复现的数字告诉你**为什么**。

先把心智模型钉死：**字节码 VM 之于源程序，相当于汇编 + CPU 之于 C 代码**。字节码就是这台虚拟 CPU 的"机器码"——只不过这台 CPU 不是硅做的，是一个 `while` 循环加一个 `switch` 模拟出来的。理解了这点，你就理解了为什么第 08 章的"原生代码生成"是本章的自然延伸：到那一章我们只是把"虚拟 CPU"换成真实 CPU，指令从我们自定义的 `Op` 枚举换成 x86/ARM——但"取指令、执行、推进指针"这套循环的灵魂，本章就讲透。

本章配套代码全部在 `examples/compiler-from-scratch/src/stage07-vm.ts`，一个自包含的教学实现。它手工构造 AST 扮演前端 (parser + resolver) 的产物，把注意力聚焦在"树 → 字节码 → VM"这一段后端。下文所有代码片段、所有数字都来自这个文件的真实运行。

## 一、为什么是栈式：从表达式树到指令流

第一个设计决策是 VM 用栈式 (stack-based) 还是寄存器式 (register-based)。这不是随便选的，背后有真实的工程取舍。

**栈式 VM** 的契约极简：绝大多数指令从一个隐式的"操作数栈"顶部取参数、把结果压回栈顶。`ADD` 不带任何操作数——它就是"弹两个、相加、压回去"。**寄存器式 VM** (Lua 5.x、Dalvik 走这条) 的指令带显式寄存器编号：`ADD R3, R1, R2` 意思是 R1+R2 存进 R3。

教学和绝大多数动态语言选栈式，原因不是它快，而是它**和"表达式天然是一棵树"完美对应**。看 `stage07-vm.ts` 编译二元表达式的代码：

```ts
case "Binary": {
  compileExpr(scope, e.left);
  compileExpr(scope, e.right);
  const opMap: Record<string, Op> = { "+": Op.ADD, "-": Op.SUB, "*": Op.MUL, "<": Op.LESS };
  const op = opMap[e.op];
  if (!op) throw new Error(`unsupported binary op '${e.op}' in stage07 subset`);
  emit(scope, op);
  return;
}
```

这就是后序遍历 (post-order)：先 emit 左子树的指令、再 emit 右子树、最后 emit 自己的运算符指令。运行时栈上恰好先有左结果、再有右结果，`ADD` 弹两个一算正好。**整个编译器把"递归求值一棵树"翻译成了"线性 emit 一串指令"**——这就是编译的本质动作：把树的结构信息编码进指令的顺序里。

代价是栈式同一计算需要更多条指令（每个中间值都要 push/pop 一次），但换来编译器极其简单、指令极窄（多数零操作数）。寄存器式指令更少、更接近真机，但寄存器分配是个难题（第 08 章会撞上它的硬核版本）。`fib` 函数体的反汇编印证了"指令窄"：

```
fn fib(arity=1)  locals=[n]  upvalues=[]
  000  GET_LOCAL        0  ; n
  001  CONST            0  ; 2
  002  LESS
  003  JUMP_IF_FALSE    6
  ...
  016  ADD
  017  RETURN
  constants: [2, "fib", 1, 0]
```

注意 `LESS` / `ADD` / `RETURN` 全部零操作数，操作数都在栈上隐式传递。这就是栈式的标志性长相。

### 常量池：字面量不进指令流

留意上面反汇编最后一行 `constants: [2, "fib", 1, 0]`。指令里的 `CONST 0` 不是"压入数字 0"，而是"压入常量池第 0 槽的值"——也就是 `2`。为什么不把字面量直接编进指令？因为指令要定长好取（真机上指令宽度固定才能高效解码），而字面量可以是任意大的数、字符串、甚至是函数。把它们集中放进常量池、指令只存一个小索引，是所有真实字节码格式 (Java `.class` 的 constant pool、Python 的 `co_consts`) 的标准做法。`stage07-vm.ts` 的常量池还做了去重：

```ts
function internConstant(scope: CompileScope, value: LiteralValue): number {
  const idx = scope.constants.findIndex((c) => c === value);
  if (idx >= 0) return idx;
  scope.constants.push(value);
  return scope.constants.length - 1;
}
```

去重保证了**确定性**——同一程序每次编出字节相同的字节码——这对增量编译缓存、对拍测试都是刚需。

## 二、解释循环：VM 的心脏

✦ 现在到了本章的核心。一个字节码 VM 的全部灵魂，是一个 `while(true) switch(op)` 的取指—执行—推进循环 (dispatch loop)。看 `stage07-vm.ts` 的 `runVm`：

```ts
while (frames.length > 0) {
  const frame = frames[frames.length - 1];
  const fn = frame.closure.fn;

  if (frame.ip >= fn.code.length) {
    frames.pop();
    continue;
  }

  const ins = fn.code[frame.ip++];   // 取指 + 推进 ip
  stats.opsExecuted++;

  switch (ins.op) {
    case Op.CONST:
      push(fn.constants[ins.operand!] as RuntimeValue);
      break;
    case Op.ADD: {
      const b = num(pop());
      const a = num(pop());
      push(a + b);
      break;
    }
    // ... 其余 opcode
  }
}
```

`frame.ip` 是指令指针 (instruction pointer)，等价于真机的程序计数器 PC。`fn.code[frame.ip++]` 这一行同时完成"取当前指令"和"指针前移"两件事——这就是冯诺依曼机器最底层的节拍，本章把它写到了你眼前。每一种 `Op` 在 `switch` 里对应一个 case，多数 case 就是几行栈操作。整个执行引擎，去掉注释不到一百行。

为什么这能比树遍历快？把两者并排看就清楚了。树遍历每求值一个节点要：(1) 一次 JS 递归函数调用（压 JS 调用栈帧）、(2) 一次 `switch (e.kind)` 判别、(3) 变量查找走 `Map` 哈希。字节码 VM 每执行一条指令只要：(1) 一次平坦的 `while` 迭代、(2) 一次 `switch (ins.op)`、(3) 变量用**数组下标** O(1) 取（`frame.slots[operand]`）。**没有递归栈帧、没有哈希查找、分支模式对 CPU 分支预测器更友好。** 这三点差异，下一节用数字量化。

这里埋一个关键正确性约定：**每条指令对栈高度的净影响是固定且已知的**。表达式求值净 +1（结果留栈顶），语句求值净 0（不留垃圾）。你能在 `compileStmt` 里看到这个不变量怎么被维护——`ExprStmt` 末尾一定补一条 `POP`：

```ts
case "ExprStmt":
  compileExpr(scope, s.expr);
  emit(scope, Op.POP, undefined, "discard expr-stmt result"); // 维护语句净高度 0。
  return;
```

这条看似多余的 `POP` 是整个 VM 栈平衡的命门。第六节的失败模式就是把它拿掉，演示栈如何崩。

## 三、实测:树遍历 vs 字节码,以及一个反直觉的真相

⚡ 这是本章的开放/前沿钩子，但先给一个**容易被讲错**的事实。很多教程会说"字节码比树遍历快是因为它做的工作更少"——这是错的。`stage07-vm.ts` 用同一个 `fib(25)` 在两条路上跑，特意统计了"操作数"（树遍历访问的 AST 节点数 vs VM 执行的指令数，两者同量纲，都是"一次原子求值动作"）：

```
【④ 实测:AST 树遍历解释 vs 字节码 VM】(同一程序 fib(25))
  两种模型输出一致: fib(25) == 75025 ✓
  树遍历访问 AST 节点(操作)数: 2,670,636
  VM 执行字节码指令(操作)数:   2,670,639
  操作数比(树/VM): 1.00x（≈1:同一算法操作数量相当;VM 的赢点是「每操作更便宜」,非「操作更少」）
```

操作数几乎完全相等——2,670,636 对 2,670,639，比值 1.00x。同一个算法，原子求值动作的数量本来就该相当。**字节码 VM 的胜利不在"做更少的事"，而在"每件事更便宜"。** 这正是上一节那三点差异（无递归帧、无哈希、分支友好）的体现。再看 wall-clock：

```
  树遍历 中位数 = 78.809 ms（本机实测,仅展示）
  字节码VM 中位数 = 29.725 ms（本机实测,仅展示）
  ⇒ 字节码 VM 加速比 = 2.65x（相对量,可复现断言: >1x）
  ⚠ toy 程序 + JS-on-JS 解释,绝对值偏乐观;可迁移的相对趋势是「扁平指令循环 < 递归树遍历的每操作开销」
```

同样的操作数，字节码 VM 快 2.65 倍。这就是 CPython、Lua、JVM 全部选字节码而非直接遍历树的实证理由：**在解释器层，省的不是操作数量，是每操作的固定开销。**

但这里必须诚实标注两条边界，否则就成了它要反对的那种 AI 注水稿：

1. **绝对毫秒数不可迁移。** 这是 toy 程序 + JS 解释 JS（VM 本身用 JS 写、跑在 V8 上），双重解释让绝对值偏乐观。换成 C 写的 VM、换成真实负载，倍数会变。
2. **2.65x 这个具体数字也不该当真，只有"> 1x 且方向稳定"是可复现断言。** 单次 run 的中位数会因机器、负载、V8 JIT 状态浮动。

⚡ 那么前沿在哪？**字节码 VM 的 dispatch 开销本身是个至今没有"通用最优解"的研究问题。** 我们这个 `switch` 派发，在 C 里有个经典优化叫 computed goto（直接跳到下一条指令的 handler，省掉 switch 的边界检查和跳转表回环），GCC/Clang 支持，CPython 3.x 用了它，实测能再快 15%-25%。更激进的有 threaded code、把热路径 JIT 成原生码（V8、PyPy、LuaJIT 走这条）。但**没有一种 dispatch 策略在所有架构、所有负载上都最优**：computed goto 在某些 CPU 的分支预测器上反而更差，JIT 的预热成本对短命脚本是净亏。"解释器 dispatch 怎样最快"到今天仍是 PL 实现领域的活跃课题——这也是为什么各家 VM 的 dispatch loop 长得都不太一样。

### 优化 IR 落到执行层:第 06 章的收益在这里兑现

第 06 章做的常量折叠 (constant folding)，它的价值要到本章才能被"看见"——少 emit 的指令，VM 就真的少跑。`stage07-vm.ts` 把同一函数的优化前/后字节码喂给同一个 VM：

```
【⑤ 优化前/后(常量折叠)字节码在同一 VM 上】
  优化前 work 体: (let work (fn work (n) ((return (+ (- (+ (* 2 3) 4) 1) n)))))
  优化后 work 体: (let work (fn work (n) ((return (+ 9 n)))))   ← 常量子树折成单个字面量
  优化前字节码总指令数: 21 条
  优化后字节码总指令数: 15 条（少 6 条:折叠消除了算术指令）
  ...
  ⇒ 优化后 VM 运行加速比 = 1.22x（少跑指令 → 更快;toy 程序收益小,趋势真实）
```

`(((2*3)+4)-1)` 这团全是字面量的子树，编译期就被折成了单个 `9`，于是字节码从 21 条降到 15 条，执行操作数 19 → 13。这是优化器与执行器之间因果链的闭环：**前端少 emit 一条指令，后端就在每次调用时省一次执行。** toy 程序收益只有 1.22x 不奇怪——折掉的是常数项，真实热循环里这种折叠会被调用次数放大。

## 四、函数调用:call frame 栈

✦ dispatch loop 解决了"一个函数内怎么跑"，函数调用解决"跨函数怎么跑"。机制是 call frame（调用帧）栈。每次调用压一个新帧，每次返回弹一个：

```ts
interface CallFrame {
  readonly closure: Closure;
  ip: number;              // 本帧自己的指令指针
  readonly slots: RuntimeValue[];  // 本帧局部区(参数 + let 声明的局部)
}
```

关键设计：**所有帧共享同一条操作数栈，但各有独立的 `slots`（局部变量区）**。这把"参数传递/返回值"（走共享操作数栈）和"局部变量"（走私有 slots）干净地分开。看 `CALL` 怎么开新帧：

```ts
case Op.CALL: {
  const argc = ins.operand!;
  const calleeIdx = stack.length - argc - 1;   // 栈布局:[..., callee, arg0..arg{argc-1}]
  const callee = stack[calleeIdx];
  if (!isClosure(callee)) throw new Error(`VM: attempt to call a non-function (${typeof callee})`);
  if (callee.fn.arity !== argc) {
    throw new Error(`VM: '${callee.fn.name}' expects ${callee.fn.arity} args, got ${argc}`);
  }
  const slots: RuntimeValue[] = [];
  for (let k = 0; k < argc; k++) slots[k] = stack[calleeIdx + 1 + k];
  stack.length = calleeIdx;       // 把 callee + 实参清出操作数栈,转移进新帧 slots
  frames.push({ closure: callee, ip: 0, slots });
  // ...
}
```

调用约定 (calling convention) 一目了然：编译器先压被调对象、再压各实参，然后 emit `CALL argc`。VM 据 argc 在栈上定位 callee（在 argc 个实参之下），把实参搬进新帧的 slots，再把这些值从操作数栈清掉。`RETURN` 则把返回值留回调用者栈顶、弹掉当前帧：

```ts
case Op.RETURN: {
  const result = pop();   // 弹出返回值
  frames.pop();           // 弹掉当前调用帧
  push(result);           // 返回值交还调用者栈顶
  break;
}
```

注意一个被注释点破的失败模式预兆：**这里只 push 返回值，没做别的栈收缩**——因为局部变量住在独立的 `frame.slots` 里、随帧对象一起被 GC，根本不占操作数栈。操作数栈的平衡靠的是"CALL 已把 callee+args 清掉"+"调用方 emit 的那条 POP"。这条不变量一旦破坏，就是第六节的栈泄漏。

跑通递归 `fib(20)` 是这套机制的验收：

```
【② 栈式 VM 跑通 fib(20)】
  程序输出: 6765
  对拍通过: fib(20) == 6765 ✓
  VM 执行指令数: 240,805 条
  最大调用帧深度: 21(fib 递归深度 + 顶层帧)
  操作数栈峰值高度: 13 槽
```

划重点：跑了 24 万条指令、递归 21 层深，**操作数栈峰值只有 13 槽**。栈高度只跟单个表达式的复杂度有关，跟调用次数无关。`stage07-vm.ts` 还可视化了帧栈随递归的伸缩：

```
【③b 递归调用时帧栈深度变化】fib(6) 调用序列(前 12 次 call):
  depth=2(stack=0)  depth=3(stack=0)  depth=4(stack=0)  depth=5(stack=0)  depth=6(stack=0)
  depth=7(stack=0)  depth=7(stack=1)  depth=6(stack=1)  depth=5(stack=1)  ...
  → 帧深度随递归深入增加、返回时减少;栈高度始终有界(无泄漏)
```

`depth` 随递归深入而涨、返回而落；`stack`（操作数栈高度）始终有界。这就是健康 VM 的指纹。

## 五、闭包:upvalue 怎么捕获状态

最考验设计的是闭包 (closure)：内层函数引用外层的局部变量，而且外层返回后那个变量还得活着。`stage07-vm.ts` 用 upvalue（被捕获的外层变量）数组实现。先看变量解析——编译器在编译期就判定每个变量是 local / upvalue / global：

```ts
function resolveVar(scope, name): { kind: "local" | "upvalue" | "global"; slot: number } {
  const localIdx = scope.locals.indexOf(name);
  if (localIdx >= 0) return { kind: "local", slot: localIdx };

  if (scope.enclosing && variableExistsInEnclosing(scope.enclosing, name)) {
    let upIdx = scope.upvalues.indexOf(name);
    if (upIdx < 0) { scope.upvalues.push(name); upIdx = scope.upvalues.length - 1; }
    return { kind: "upvalue", slot: upIdx };
  }
  return { kind: "global", slot: internConstant(scope, name) };
}
```

这正是第 03 章 resolver 的价值落地：**变量怎么找在编译期就定死，运行时只按槽位索引 O(1) 取**——没有运行时的变量名查找。这是字节码相对树遍历的又一个隐藏优势（树遍历那条路还在用 `Map` 按名字查 env）。

计数器是 upvalue 的最小可观测用例：

```
【③ 闭包计数器:upvalue 捕获 + 跨调用保持状态】
fn next(arity=0)  locals=[]  upvalues=[count]
  000  GET_UPVALUE      0  ; count
  001  CONST            0  ; 1
  002  ADD
  003  SET_UPVALUE      0  ; count
  004  POP             ; discard expr-stmt result
  005  GET_UPVALUE      0  ; count
  006  RETURN
  constants: [1, 0]
  3 次调用 c() 输出: 1, 2, 3
  对拍通过: 闭包状态跨调用保持,counter == [1,2,3] ✓
  → upvalue 'count' 被 next 捕获,SET_UPVALUE 每次回写,旧值 +1
```

打印 `1, 2, 3` 就是铁证：`count` 这个 upvalue 在三次独立调用之间被保持并累加，而不是每次重置为 0。机制在 `SET_UPVALUE`——闭包对象持有一个**可变的** `captured` 数组，回写就改它：

```ts
case Op.SET_UPVALUE:
  // 回写捕获槽:这就是 counter 能累加的机制 —— 闭包持有可变的 captured 数组。
  frame.closure.captured[ins.operand!] = stack[stack.length - 1];
  break;
```

**这里有个必须坦白的简化，也是最容易踩的设计陷阱。** `stage07-vm.ts` 的 upvalue 是**值捕获**（捕获瞬间复制一份值进 `captured` 数组），不是真正的引用捕获。计数器能工作，是因为它把状态就存在闭包自己的那个 `captured` 槽里、自产自销地回写。但值捕获有个真实语言绝不允许的破绽：**如果两个闭包共享同一个外层变量，值捕获会让它们各拿各的快照、互不可见**。Lox/Lua 的正解是"open upvalue"——捕获的是指向栈槽的指针，外层变量还在栈上时多个闭包共享同一指针，外层返回时再"close"（把值搬到堆上、指针重定向）。本章为聚焦 dispatch loop 主线做了值捕获的简化，但你要知道这是个**会改变可观测语义的简化**，不是无害的实现细节——这正是注释里那句"本实现是值捕获，非引用捕获"想提醒的。第 08 章如果要支持完整闭包语义，这里得重写成 open/closed upvalue 机制。

## 六、失败模式:栈泄漏,以及为什么那条 POP 不能省

前面反复强调"每条指令的净栈高度变化是已知的、语句净高度为 0"。现在演示破坏它的后果。`stage07-vm.ts` 手写了一段"递归调用后故意忘记 POP 返回值"的字节码——这是真实编译器里最经典的后端 bug 之一：

```ts
const leakCode: Instr[] = [
  // ...
  { op: Op.CALL, operand: 1 },              // 10  leak(n-1) → 返回值留栈顶
  // 【BUG】此处本应 { op: Op.POP } 丢弃用不到的返回值。故意省略 → 每层泄漏一个槽。
  { op: Op.CONST, operand: 1, note: "0" },  // 11
  { op: Op.RETURN },                        // 12
];
```

为什么用手写字节码而不是加个 flag？注释说得很清楚：**VM 本身永远是正确的，被测的是"错误的字节码"会不会被 VM 如实暴露成栈越界**。这比注入故障开关诚实得多——它证明的是"不变量被破坏后系统会客观地崩"，而非"我手动让它崩"。逐步加大递归深度观测操作数栈峰值：

```
【⑥ 失败模式:函数返回未收缩栈(栈泄漏)→ 栈高度随调用线性增长直到越界】
  递归深度 depth | 操作数栈峰值高度 | 结果
  --------------|------------------|------
              5 |                6 | OK(输出 0)
             50 |               51 | OK(输出 0)
            500 |              501 | OK(输出 0)
           5000 |             5001 | OK(输出 0)
         120000 |     >100,000(越界) | ✗ STACK OVERFLOW: operand stack exceeded 100,000 slots (frames=20002)
```

栈峰值 ≈ depth，**线性增长**。每递归一层就在操作数栈上多留一个没人收的返回值，积累到 STACK_LIMIT (100,000) 就越界。对比第四节那个健康的 `fib(20)`——24 万条指令、递归 21 层，栈峰值死死锁在 13 槽。同一台 VM、同样的递归，差别只在那条丢失的 `POP`。

这个对照给出了一条可操作的诊断法则：**栈峰值若随调用次数线性增长，根因几乎必然是"某条路径上表达式的净栈高度 ≠ 预期"——最常见的就是漏 POP 一个被丢弃的返回值。** 这也解释了 VM 里那个 `STACK_LIMIT` 不是摆设：

```ts
const STACK_LIMIT = 100_000;
const push = (x: RuntimeValue) => {
  stack.push(x);
  if (stack.length > stats.maxStackHeight) stats.maxStackHeight = stack.length;
  if (stack.length > STACK_LIMIT) {
    throw new Error(`STACK OVERFLOW: operand stack exceeded ${fmtNum(STACK_LIMIT)} slots ` +
      `(frames=${frames.length}). 典型根因:函数返回未收缩栈帧(栈泄漏)。`);
  }
};
```

没有这个硬上限，失控的栈泄漏会一路吃内存直到进程被 OOM kill——一个无法诊断的崩溃。有了它，泄漏变成一个**可观测、带根因提示的报错**。这是防御性编程的正确姿势：不是吞掉错误，而是把不可控的失控转成可定位的失败。

## 小结与下一章

本章把"程序怎么跑起来"落到了代码级。核心拿走三样东西：

1. **dispatch loop 是解释器的心脏**——`while + switch` 的取指/执行/推进，加上 call frame 栈和 upvalue 数组，就能跑通递归和闭包。这套循环的灵魂在第 08 章换成真实 CPU 后依然成立。
2. **字节码 VM 快在"每操作更便宜"，不在"操作更少"**——实测同一 `fib(25)` 操作数 1.00x 持平、wall-clock 却快 2.65x，这是 Python/Lua/JVM 全选字节码的实证理由。但 dispatch 怎样最快至今无通用解（computed goto / threaded code / JIT 各有边界）。
3. **栈平衡是一条必须主动维护的不变量**——那条"多余"的 POP 是命门，省掉它栈就随调用线性泄漏直到越界。

第 06 章的优化 IR 在本章兑现成了更少的字节码、更快的执行；本章这个字节码 VM，将作为第 08 章**原生代码生成**的对照基线——到那一章，我们把虚拟 CPU 换成真实硅，指令换成 x86/ARM，但你会发现"取指、执行、栈帧、调用约定"这些本章建立的概念一个都不会丢，只是换了更硬核的实现。
