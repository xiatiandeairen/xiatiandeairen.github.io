> 这章解决的问题:**工程能力是你的长板,但它在 indie 阶段会成为你最大的隐形税**。你越能写漂亮的架构,就越容易掉进过度工程的陷阱——在零用户的产品上消耗本可以验证产品的时间。把这一章读完,你会知道该把工程力气用在哪、该在哪里克制。

---

## 为什么大厂工程范式在 indie 阶段是错的

先把机制讲清楚,这样后面的打法才有根。

大厂的工程严谨——代码审查、分层架构、接口抽象、配置化、灾备容量规划——是为三件事服务的:**规模**、**多人协作**、**长期可维护性**。Google 的基础设施要支撑每秒数十亿次搜索请求，必须解耦、分层、冗余，这是成本合理的工程决策。但这套决策的前提条件是：你已经知道有价值、有用户、有规模需要扛。

一个人做的产品、零用户、产品假设尚未验证——把同一套范式套上来，逻辑前提根本不存在。

**过度工程的代价不是"代码写多了"，是时间被错误分配了。** 你花在可扩展性上的三天，本可以用来上线一个能收到用户反馈的版本。在没有用户数据的前提下做的架构决策，大概率是拍脑袋的——要么过度复杂（为一个永远不会来的规模提前优化），要么方向根本就错了（产品 pivot 了，架构全扔掉）。

工程师转 indie 最常见的路径是这样的：有个想法 → 开始设计 → "这里后面肯定要支持多租户，我现在就设计好" → "数据层最好抽象一下，以后换数据库方便" → "认证系统要通用，先做个完整的 OAuth" → 三个月后，精美的架构图画了一堆，没有一个用户测试过核心假设。

**这是工程师的特有陷阱，因为做架构对你来说很舒服——它的反馈是即时的、可控的、有成就感的。** 而做产品的反馈是模糊的、延迟的、经常令人沮丧的。大脑会选择舒服的那条路。

理解了这个机制，下面的打法才是有根的。

## 打法一：能用 BaaS 不自建，能买不造

**indie 开发的时间不是按小时算，是按"验证一个假设需要多少时间"算的。** 自建基础设施会把这个时间拉长到不可接受的程度。

Supabase、Firebase、Clerk、Stripe、Resend、Cloudflare——这一代 BaaS 产品的出现，意味着你能在一个下午搭完「认证 + 数据库 + 文件存储 + 邮件」，剩下的精力全部押在产品差异化上。2015 年之前，独立开发者得自己搭这些，成本高达数周；现在完全没理由。

**典型误区是"我自己搭更省钱"。** 算一笔账：Supabase 免费档支撑早期产品绰绰有余。你省下的那 $25/月，买不回来被自建耗掉的两周开发时间。等到规模真的大到 BaaS 费用成问题的那一天，恭喜你，那是好问题，届时再迁移。

**在支付上尤其如此**。Stripe 抽成加上固定费用并不便宜，但自建支付（PCI 合规、对账、退款、订阅计费逻辑）轻松耗掉一个月。而且 Stripe 不只是支付，它还是你的订阅管理、发票、税务计算。独立开发者没有任何理由在支付上重新造轮子。

Pieter Levels（Nomad List、Remote OK 的创造者）的整个技术栈常年以极简著称：PHP + SQLite，直接跑在单机上，没有微服务，没有 K8s，页面渲染服务端完成。他对这套"丑"技术栈的解释很直接：够用，而且我能全部掌控，不需要另一个人帮我维护。他的产品在这套栈上跑了多年、收入持续。

## 打法二：能写死不配置化，直到有人付钱倒逼

这条最反工程师直觉，但逻辑很硬。

配置化的代价不只是开发时间，还有**认知负担**。每一个你做出来的配置项，都需要你长期记住它、测试它的组合、处理它可能引入的 bug、在文档里解释它。早期产品没有用户反馈，你做的每一个配置项都是在赌用户真的需要它——而你大概率赌错。

**正确的顺序是反的：先写死一个你认为最有可能对的默认值，上线，等用户真的来抱怨"我想改这个"，再把它变成配置。** 用户抱怨驱动的配置化，每一个都是真实需求；提前做的配置化，多半是自己在脑补。

Notion 早期对「块」的类型是硬编码的——没有开放式的插件系统，类型都是团队内部写死的。用户要什么类型他们就加什么。这让产品在没有搞清楚用户真正需要哪些扩展之前，避免了过早泛化带来的架构复杂度。等用户需求清晰之后，再做开放化的 API 层（待核：时间线）。

这个打法在 AI 产品上尤其重要。System prompt 该写死还是让用户配置？模型选什么写死还是可切换？上下文窗口大小要不要暴露给用户？**默认答案全是写死。** 等用户真的提出要改，再说。

## 打法三：能手动（concierge）不自动化，直到规模倒逼

这条出自 Y Combinator 经典建议：**做那些不可扩展的事（Do things that don't scale）**。

"concierge MVP"的意思是：产品的核心流程先靠人工完成，不写自动化代码。Airbnb 早期创始人亲自去给房东拍照片。DoorDash 最早的配送是创始人自己开车送的。他们没有先写完整的配送调度算法、再验证"有人想要这个服务"——而是先手动跑通一次，确认有真实需求，再考虑自动化。

对 AI 原生产品，concierge 尤其有价值。如果你在做一个 AI 写作助手，第一版未必是跑完整的 prompt pipeline；你可以先让用户提交请求，然后**你自己手动跑 ChatGPT、编辑结果、发回给用户**。这样做你能极快地学到：用户真正想要什么样的输出、哪些 edge case 最多、什么情况下 AI 根本不够用需要人介入。这些信息，在你没有真实用户数据之前，是买不到的。

**自动化的时机是：当手动处理的量让你跟不上的时候。** 那一刻来了，说明有真实需求，自动化的投入才合算。

## 打法四：第一版允许丑、允许 hack，只管能验证就上

这是心态打法，但在实操中最难执行，因为它直接挑战工程师的职业自尊。

**丑的 UI、写死的逻辑、"以后肯定要重构"的代码——在没有付费用户之前，这些都不是问题。唯一的问题是：这个产品有没有人需要？**

Craigslist 的界面从 1990 年代到现在几乎没变，依然是蓝色链接、纯文本列表——这套"丑"的设计在用户看来反而是一种可信的简朴。它解决的问题（本地分类信息）够真实，以至于界面根本不重要。

早期 Facebook 的代码质量被很多后来加入的工程师吐槽为"一团糟"——这不妨碍它成为互联网历史上增长最快的产品之一。Zuckerberg 在哈佛宿舍写的那版，连基础的安全检查都没有（待核：具体 CVE），但它在 24 小时内就覆盖了哈佛全校，验证了核心假设。

对独立开发者：你的第一版代码唯一需要具备的品质是**能运行、能让用户完成核心任务**。注释可以乱、变量名可以随便、错误处理可以简陋、UI 可以默认样式。**把"这代码我拿得出手吗"这个问题推到有付费用户之后再问。**

## 打法五：延迟一切"为未来扩展性"的投入

这是过度工程的核心病灶，单独拎出来讲。

**"未来可能需要"是 indie 阶段最昂贵的五个字。**

每一个"以后可能要支持多语言，所以现在做 i18n 架构"、"未来可能有企业客户，所以现在做多租户"——都是在为一个不确定存在的未来买单。你在用确定的开发时间（现在）换一个不确定的好处（未来的某个也许存在的需求）。

YAGNI——You Aren't Gonna Need It——这条 XP（极限编程）原则在大厂语境里常常被过度套用，但在 indie 阶段是铁律。

一个具体的 anti-pattern：**提前设计数据库 schema 以"支持未来的功能"**。你在不知道产品会往哪走的时候设计了一个"灵活"的 schema，结果产品 pivot 了，这个 schema 不但没用，还成了重构的负担。正确的做法是为当前需要的功能设计最简单的 schema，在需求变化时做迁移。SQLite 的迁移成本远低于你在脑子里做架构推演的成本。

**"扩展性"是你有了规模问题之后才需要解决的问题。** 没有规模问题的产品去做扩展性优化，是为一个不存在的问题预先买单。

## 案例深拆一：Instagram 用 13 人撑起了亿级用户

Instagram 被 Facebook 收购时，团队规模约 13 人（待核：具体人数），却在为当时数量级上亿的用户提供服务（待核：具体 MAU）。

他们用的技术栈：**Python + Django，单体应用**，部署在 AWS 上，图片存储用 S3，数据库用 PostgreSQL，缓存用 Redis/Memcached。没有微服务，没有复杂的服务网格，就是经典的 Web 应用栈，2005 年就成熟了的东西。

这里的关键洞察是：**Instagram 没有发明新的基础设施，它把成熟的技术用到极致，然后把所有精力押在产品上**——滤镜体验、拍照速度、社交图谱。当它在这些维度上跑赢所有竞争对手的时候，架构还是那套"老"架构。

后来他们写过一篇博客详细讲过扩展历程：每次遇到性能问题，他们做的不是重写架构，而是找到最小的瓶颈点做针对性优化。这是"被问题驱动"而不是"被焦虑驱动"的工程文化。

**对一个人的团队意味着什么**：你不需要 Django 的全部，你不需要 PostgreSQL 的全部高级特性，你甚至可以从 SQLite 开始。Instagram 的故事告诉你的是：**用成熟、无聊的技术，全力解决产品问题**，不要分心到基础设施的创新上。

## 案例深拆二：WhatsApp 用 50 人服务近 10 亿用户

WhatsApp 被 Facebook 以约 190 亿美元收购时（待核：金额），团队约 50 人（待核：具体人数），用户量接近 10 亿（待核：具体数字）。这大概是互联网历史上人均服务用户数最高的公司之一。

他们的技术选择：**Erlang**——一门诞生于 1980 年代、为电信系统设计的语言，专门为高并发、高可靠性打造，在互联网创业圈几乎没人用。WhatsApp 选它不是因为时髦，而是因为它天生适合消息系统的并发模型，而且**每个工程师都能深度掌控整个系统**。

他们的工程文化极度克制：没有复杂的监控仪表盘文化、没有大量基础设施工程师、工程师人均负责的服务器数量远超行业均值（待核）。他们的产品功能也刻意简单：早期 WhatsApp 没有图片分享，没有语音消息，只有文本——因为他们的核心 job 是"在任何地方都能收到消息"，其他都是后来加的。

**WhatsApp 的故事里最反直觉的一点是：他们选了一个小众技术，不是为了"时髦"，而是因为它在核心问题上是最合适的工具。** 这是工具选择的正确逻辑：匹配问题，而不是跟风。

另一个关键是他们的商业模式极简：收费应用，后来改为订阅（待核：时间线），没有广告，没有复杂的变现层。这个简单性让整个工程团队可以专注在产品本身，而不是变现基础设施。

## 案例深拆三：Jobs 回归苹果砍产品线

1997 年 Steve Jobs 回归苹果时，苹果有数十个产品线（打印机、相机、Newton PDA、各种 Mac 型号……）。他在白板上画了一个 2×2 矩阵：桌面 vs 便携、消费者 vs 专业——四个格子，四个产品，其余全砍。

**这是极度聚焦的教科书案例**，它说明的不是"少即是多"这个审美观，而是一个组织真理：**资源是有限的，每多一条产品线，就是在稀释每条产品线能得到的专注度。对一个人的团队，这个效应放大十倍。**

你作为独立开发者只有一个人，你的"产品线"不能超过一个。不是说你未来不能做第二个产品，而是当你在做第一个产品的时候，你必须保持对它的全部注意力。每一个"顺手做"的功能、每一个"加了也不难"的新模块，都在稀释核心产品能得到的能量。

苹果 iMac、iBook 的崛起，是砍掉其他产品线之后，所有工程师和设计师的能量都灌进来之后的结果。独立开发者也是这样：**产品一旦聚焦，你对它的理解深度、迭代速度、用户反馈的整合能力都会显著提升。**

## 小结：把工程力用在"反向"

你的工程能力是长板。但在 indie 阶段，这根长板的正确用法是**反向的**：

- 不是用它做出架构完美的东西，而是用它**快速做出能验证假设的东西**
- 不是用它解决"未来可能有"的问题，而是用它**解决现在真实存在的问题**
- 不是用它写让同行赞叹的代码，而是用它**让用户留下来**

**工程能力在 indie 阶段的最大价值，是让你比没有工程能力的人更快地上线、更快地调整、更少地依赖外部帮助。** 不是让你做出更精密的系统。

等到有了付费用户、有了规模问题、有了团队成员，那些大厂的工程实践就会重新有意义——因为前提条件终于满足了。在那之前，克制就是最大的工程判断力。

## 新手 vs 资深

| | 新手 indie（工程背景）| 有 indie 心态的资深 |
|---|---|---|
| 项目启动方式 | 设计数据库 schema + 选框架 + 规划架构 | 最快能跑通核心用户流程的方式是什么 |
| 面对"以后需要扩展"| 现在设计成可扩展的 | YAGNI，有规模问题了再说 |
| 技术栈选择 | 时髦 / 最佳实践 | 最成熟、最无聊、我最熟的 |
| 功能取舍 | 加了也不难，顺手做了 | 这个不加，等用户要了再说 |
| 代码质量底线 | 能拿出手 | 能运行、能让用户完成任务 |
| 基础设施 | 自建，有控制感 | BaaS 优先，省出来的时间做产品 |
| 何时自动化 | 写代码之前先做 | 手动处理到跟不上了再自动化 |

## 反模式（工程师高发）

- **"这里以后肯定要扩展，现在设计好省得返工"**：大概率你为一个永远不会来的未来过度消耗了现在。返工的成本通常远低于过早设计的成本。
- **用 Kubernetes 部署一个日活个位数的产品**：管理复杂度远超价值，一台 $10/月 的 VPS 足够支撑你的早期阶段（待核：当前定价）。
- **把通用性做进第一版**："这个组件以后其他产品也能用"——在你不知道"其他产品"是什么的时候，通用性是一种预消耗。
- **在没有用户之前做完整的监控告警系统**：先上线再说，用户的反馈是最好的监控。
- **代码评审阶段的完美主义**：独立开发阶段你自己评审自己，标准应该是"有没有明显 bug"，而不是"架构是否优雅"。
- **"这个功能不复杂，顺手做了"**：独立开发的时间预算里没有"顺手"，每个功能都是你押在上面的注意力。

## 动手：这章怎么练

1. **做一次"过度工程审计"**：列出你当前（或计划中）产品的所有技术决策，逐一问"这个决策服务的是当前存在的问题，还是假想的未来问题？"把假想未来的部分全部标出来，推后。

2. **设计你的 concierge 版本**：如果你的产品的核心流程可以手动跑通，怎么手动跑？写下来。先手动跑 10 次，再考虑自动化。

3. **给自己定一个"写死清单"**：把你在设计阶段想配置化的东西全部写下来，然后默认都改成写死——只有在有真实用户提出这个需求之后才开放配置。

4. **找一个真实的 indie 开发者的技术选型博客读一遍**：Pieter Levels 的 "Reflect"、levelsio.com 上的访谈，或者 Justin Jackson（Transistor.fm 创始人）分享的早期技术决策。感受一下"够用"的标准在真实产品里长什么样。

## 自测清单

- [ ] 我能说清楚当前产品的哪些技术决策是为"真实现在问题"做的，哪些是为"假想未来"做的
- [ ] 我知道 BaaS 能替代我哪些基础设施，并且默认选了它们
- [ ] 我的第一版技术栈是"最无聊的成熟技术"，而不是最时髦的
- [ ] 我有一个"concierge 版本"的执行方案，能在不写代码的情况下验证核心假设
- [ ] 我没有在第一版里做任何"写死会有点麻烦，但配置化用户目前不需要"的配置项
- [ ] 我能说出自己的核心产品假设，以及最快验证它的路径

## 真实参照

- **Paul Graham《Do Things That Don't Scale》**（2013）—— Y Combinator 创始合伙人写的文章，concierge 模式的理论根源，最好直接读原文
- **Pieter Levels（levelsio）的博客 / 访谈**—— Nomad List / Remote OK 创始人，极简技术栈 + 高产出的典型样本，读他的技术选型分析
- **Justin Jackson / Transistor.fm 的 Build Your SaaS 播客**—— 两个人做 SaaS，里面有大量"该自建还是买"的真实决策案例
- **Instagram Engineering Blog（早期）**—— 他们在高增长阶段如何处理扩展性，可以搜 "Instagram tech stack 2012"
- **WhatsApp Engineering Blog**—— Erlang + 极简团队的工程文化，有公开文章讲他们的服务器/工程师比例
- **《The Lean Startup》Eric Ries**—— MVP 和 concierge 测试的系统方法论背景书
