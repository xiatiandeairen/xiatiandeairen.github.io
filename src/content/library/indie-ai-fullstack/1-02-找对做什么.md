> 独立产品九成死在"做了没人要的东西"。这章教你在动手写代码之前,用 4 条打法把"我想做"转化为"有人付过钱、有人不满意、我能切进去"的真实机会。

---

## 先讲底层:为什么"空白市场"是陷阱

很多工程师出题时有一个根深蒂固的直觉:竞品多 = 红海 = 饱和 = 别进。找一个"还没人做的领域"才是机会。

这个直觉**大概率害死你**。

原因很简单:**真空白市场,大多数不是"没被发现",而是"被人试过、发现没需求"的坟场。** 竞品多,恰恰说明有人验证过真实的支付意愿——用户愿意给钱换取价值这件事,是市场用真金白银证明的。

独立开发的出题逻辑和 VC 资助的大项目不一样。VC 愿意押"开创市场",因为一个就能回本。独立开发没有那个容错空间。**你要找的不是"没人做",而是"有人做、做得烂、我能为一小撮人做得明显更好"。**

红海对独立开发者是朋友,不是敌人。你的任务是在红海里找"做得烂的角"切进去。

第二条底层认知:需求的真实程度,跟"你问用户会不会用"没关系,跟他**过去的真实行为**有关。有人已经在用三个凑合的工具拼接完成某件事——那是最强的需求信号。有人对你说"很有用,你做出来我肯定用"——那是客气话,不是信号。

第三条:分发路径是出题的前置条件,不是发布后才考虑的事。**你没办法持续触达的那群人,产品做得再好也是死的。** 选题的时候就要问清楚:这群人在哪、我有没有办法持续接触他们。

这三条是本章 4 个打法的地基。

## 打法一:在有人付费的红海里找"做得烂"的角

**标准动作**:不找没竞品的市场,找"竞品已经有但用户明显不满意"的细分场景。

入口是竞品,但目光落在竞品的差评和局限上。

**案例深拆:Instagram 从 Burbn 砍到只剩照片**

Burbn 是 Kevin Systrom 最初的产品,定位是签到 App——你可以签到地点、规划行程、和朋友约见面。功能不少,逻辑也说得通,但就是没人真正用起来。

创始人没有放弃、也没有死磕。他们做了一件事:**看数据**。Burbn 里确实有人在活跃,但这批人在用什么功能?看数据发现:几乎所有活跃行为都集中在一个地方——**上传带滤镜的照片**。那是当时手机拍照质量很差的年代,用户想晒图、但手机拍出来很难看,一键滤镜把"烂照片变文艺"这个 job 正好戳中。

Systrom 和 Krieger 砍掉了 Burbn 几乎所有功能,只留下照片、滤镜、分享。这不是"创造"了一个市场——**Flickr 早就证明了人们会在网上分享照片**。Instagram 切的是一个更具体的角:手机拍照 + 即时滤镜 + 移动社交分享,而 Flickr 主要在桌面端、没有滤镜、不是即时分享的产品形态。

一个已经被验证的需求(人们想分享照片),加上一个当时做得烂的角(移动端体验极差、无法快速美化),砍光其他功能,专注做这一件事。**发布后数小时内服务器就被打崩了**——不是因为它做了别人没做的事,而是因为它把一件有人真正想做的事做得足够好。

**给你的动作**:打开你选定领域里前两名竞品的 App Store 或 G2/Product Hunt 的差评区。找出重复出现的那 3 条骂声——"太复杂""客服不行""只能 XX 做不了 YY"。每一条重复的骂声,背后都是一批已经在付费、但痛点没被解决的用户。

## 打法二:看差评区——找"已付费但不满意"的人

这条打法是打法一的信号来源细化,单独拎出来是因为工程师通常不会主动去"刷评论"。

**已付费但不满意的人,是出题最宝贵的信号。** 他们证明了两件事同时成立:一、这件事他们真的在意,值得付钱;二、现有方案没解决好。他们在等一个更好的答案。

**案例深拆:字节跳动的新产品孵化机制**

字节对新 App 的出题逻辑不是"我们想做什么",而是大规模的**数据驱动赛马**。从公开信息来看,字节内部跑过大量实验性 App——被外界看到活下来的有抖音、剪映、豆包;也有大量从未公开或很快关停的产品。

剪映的出题逻辑就很典型。抖音火起来之后,字节手里有一批数据:大量用户想发视频,但制作门槛把很多人拦住了。第三方剪辑工具不是没有——iMovie、VN、必剪都在市场上。但这批用户在差评区和反馈里说的是:**"太复杂"、"不适合竖屏"、"和抖音发布不顺畅"**。字节没有凭空发明视频剪辑需求,它看的是"抖音上发视频的人在用什么工具、在抱怨什么",然后在已有红海里做了一把专门服务抖音生态的剪刀。

剪映后来在国内移动视频剪辑工具里迅速做到头部位置(具体 DAU/市占数字「待核」),核心不是技术领先,而是**用差评区的需求反向定义了功能优先级**:竖屏优先、一键同步抖音、模板比手动剪辑更突出。

这个动作在独立开发里完全可以复制。你不需要字节的团队和资源——你只需要花两个小时把目标市场的差评区滚一遍,把重复出现的骂声记下来。

**给你的动作**:
1. 选定你想做的方向,找到该方向的 Top 3 竞品
2. 去看这三个产品的 App Store 评论(1 星 + 2 星)、Reddit 抱怨帖、Twitter/X 上的骂声
3. 把重复出现 ≥3 次的痛点列成清单
4. 问自己:**有没有一条痛点,是"明显可以做好、但竞品选择不做(或做不到)"的?** 那就是你的切入口。

特别注意"竞品选择不做"和"做不到"的区别。大公司选择不做的,通常是"太细分、不值得投入"——但对独立开发者来说,够细分反而是生存空间。

## 打法三:用过去行为判断真需求(Mom Test)

出题时最常见的自欺欺人,是**用未来承诺当需求信号**。

"你会用这个工具吗?""很有意思,你出了我肯定用!" ——这是客气话,不是信号。人在被问到"你会用吗"的时候,礼貌的默认答案是"会",哪怕他实际上一次也不会打开。

《The Mom Test》(Rob Fitzpatrick)讲的就是这个:连你妈都会骗你——因为她不忍心打击你的热情。所以不能问"会不会用",要问**他实际上怎么做过**。

**过去的行为是唯一诚实的信号。**

**判断框架:三问挖真实行为**

| 不要问(未来承诺) | 要问(过去行为) |
|---|---|
| "你会用一个 XX 工具吗?" | "上次你要处理 XX,你怎么搞的?" |
| "你觉得这个需求重要吗?" | "上个月这个问题你花了多少时间处理?" |
| "如果有这个功能你觉得好吗?" | "现在你用什么工具解决这件事?哪里让你最烦?" |

真需求的标志:**用户已经在用某个烂方法对付这件事**。用 Excel 管的、用三个工具拼起来的、每次都手工重复做的、花大钱雇人做的——这些都是"已经在流血"的信号。

**案例深拆:Notion 的切入时机**

Notion 进的是一个已经挤满竞品的领域:笔记工具(Evernote)、项目管理(Asana、Trello)、文档协作(Google Docs、Confluence)。按"有竞品就别进"的逻辑,Notion 根本不该存在。

Notion 做的是 Mom Test 里的动作:他们不是问用户"你需不需要一个更好的笔记工具",而是观察用户实际上在干什么。他们发现一批工程师和设计师**把 Evernote、Trello、Google Docs、Confluence 四个工具拼在一起用**——笔记放 Evernote、任务放 Trello、文档放 Google Docs、Wiki 放 Confluence。这不是"有了会更方便",这是"现在真的很痛、在硬扛"的行为信号(具体调研细节「待核」,但用户用多工具拼凑这个行为本身是公开讨论的)。

Notion 的出题不是"我们想做一个多合一工具",而是"**有一批人在用 4 个工具拼凑同一件事,这是流血行为**"——然后反过来问:能不能做一个让他们只用一个工具?

这就是 Mom Test 打法的核心:**不是问用户要什么,而是看他们现在怎么凑合。凑合的姿势越难看,需求越真实。**

**给你的动作**:找 3-5 个你认为可能是目标用户的真实的人(不是朋友里的客气鼓励者)。只问过去的事:上次他们怎么解决这件事、花了多少时间、用了什么工具、最烦哪一步。如果你听完之后发现他们根本不觉得这是问题——换题。如果你听完发现他们已经在凑合了很久——这个需求是真实的。

## 打法四:AI 套壳产品的生死问——"用户为什么不直接用 ChatGPT?"

这是 AI 原生产品时代独立开发者必须正视的题。

2023 年以来,AI 套壳产品已经成为独立开发里最常见的一类死法:包一个 API、做个好看的界面、定一个使用场景——产品发布后留存极低,原因大多是同一个:用户试过之后发现直接用 ChatGPT 或者 Claude 就够了,甚至更好用。

这个问题的本质是:**通用大模型已经把"AI 能力"这层做成了商品**。所有人都在调同一批 API,护城河绝不可能来自模型能力本身。

**能活下来的 AI 产品,差异化必须来自"通用模型做不到或做起来很麻烦"的地方**:

- **垂直 workflow**:领域专有的多步骤流程,用户自己用 ChatGPT 得写复杂 prompt、手动做多步操作。你把这些封装成一个按钮。
- **私有/专有数据**:接了用户自己的数据(本地文档、内部知识库、行业语料),ChatGPT 拿不到。
- **嵌入已有工具链**:长在用户本来就在用的地方(IDE、浏览器、设计工具),切换成本趋近零。
- **更可靠的输出**:在某个特定任务上做了大量提示词工程和输出校验,比用户自己问 ChatGPT 准很多。

**案例深拆:GitHub Copilot 活下来的原因**

从第一性原理看,你完全可以把代码粘贴给 ChatGPT 问"帮我补全"——模型能力是一样的。Copilot 的护城河是什么?

**IDE 内嵌入**:你不需要切换窗口、复制粘贴,建议实时出现在光标旁边。这是纯粹的 workflow 摩擦差异,跟模型没关系。

**代码上下文**:Copilot 读的是你当前文件、相关文件的上下文。你把代码粘给 ChatGPT 要手动整理上下文,而且有长度限制。Copilot 自动处理了这件事。

**训练数据专有化**:Copilot 用了大规模的代码语料做特化训练(GitHub 代码库),在代码补全这个具体任务上比通用模型更稳定。

这三条没有一条是"模型更聪明"——都是**把通用能力变成专属 workflow、减少摩擦、接入专属上下文**。这就是活得下去的 AI 产品和死掉的套壳的区别。

**反面案例:AI 写作助手的死亡公式**

市面上死掉的 AI 写作工具大量遵循同一个死亡公式:做一个好看的编辑器、调 GPT-4 API、让用户选几个"润色/改写/续写"的按钮。发布后几周留存趋近零。

原因很简单:用户测试一圈之后发现,**直接在 ChatGPT 里说"帮我改写这段,语气专业一点"结果是一样的**,而且不用多花钱。这类产品的"差异化"只是 UI 皮——换一套皮不是护城河。

**给你的动作**:对你想做的 AI 点子,写下这道题的答案——用一句话填完:"用户用我的产品而不是直接用 ChatGPT,因为\_\_\_\_\_\_\_\_\_\_\_\_\_\_\_。"

如果那个空里填的是"界面更好看"、"更便宜(只是暂时)"、"针对中文优化"(但通用模型中文也很强)——这些都不是护城河,换题。

如果填的是"我接了他们自己的数据/用在他们的 workflow 里/帮他们省掉了 N 步手动操作"——这有活路,继续挖深。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| 找题起点 | "我有个酷点子 / 这个技术有趣" | "这群人现在用什么烂方法在凑合" |
| 对待竞品 | 看到竞品就觉得饱和、放弃 | 竞品 = 需求已验证,找它做得烂的角切进去 |
| 需求验证 | 问"会不会用",听到"会"就开干 | 只听过去行为,凑合越难看越是真需求 |
| AI 产品思路 | 包 API + 好看 UI = 产品 | 先答"为什么不直接用 ChatGPT",答不出换题 |
| 分发时机 | 发布后才想怎么推 | 出题时就锁定"怎么持续够到这群人" |
| 出题节奏 | 闷头做三个月再发布 | 有两三个真实信号才动手,快速验证快速杀 |

## 反模式(工程师高发)

**追蓝海、逃红海**。没竞品不是机会信号,是"没需求或需求太小"的信号。你的第一个产品应该在已经有人付钱的地方找角,不是开荒。

**技术驱动出题**。"我想用 Rust 做一个"/"我想练 RAG"——把工具当目的,找一个题套上去。结果是做出来的东西技术上很有趣,解决的问题没人在乎。需求驱动选题,技术只是实现手段。

**用礼貌反馈当验证**。朋友说"不错啊"、团队内演示时大家鼓掌——这些都不是信号。**信号是:有人掏了钱、有人在用很烂的方式解决这件事、有人在差评区反复骂同一个问题。**

**AI 产品只做皮**。好看的 UI + API 包装 = 最容易被复制、也最容易被通用模型直接替代的组合。皮是可以的,但下面必须有护城河(workflow / 私有数据 / 嵌入渠道)。

**N=1 就宣布可行**。你自己痛 ≠ 一群人痛。从自己痒处出发是好的起点,但必须再找 ≥3 个独立的人确认他们也痛、他们也在为此付出代价——才能从"我的玩具"变成产品。

## 动手:这一章怎么练

1. **差评区挖矿**:找你想进入的领域的 Top 3 竞品,去看 App Store 一二星差评或 G2/Product Hunt 的负面评价。把重复出现 ≥3 次的骂声列出来。看能不能找到一条"竞品选择忽视的细分用户的核心痛点"。

2. **Mom Test 访谈**:对你认为最有感觉的一个痛点,找 3 个真实的潜在用户(不是朋友里会鼓励你的人)。只问一件事:"你上次怎么处理 XX 这件事的?"听他们的做法,不要推销你的方案。听完之后问自己:他们有没有在凑合?凑合的姿势有多难看?

3. **AI 生死问自测**:对你最想做的 AI 点子,写下"用户用我的产品而不是直接用 ChatGPT,因为\_\_\_\_\_\_。"认真填完,找一个不认识你的人问他觉不觉得这条理由成立。

4. **分发路径前置**:在确定出题之前,先问一个问题:"这群目标用户在哪?我现在有没有途径持续接触他们?"如果完全没有答案,要么换题,要么把建立分发渠道列为第一步(写推文、混社群、写专栏),而不是先闷头做产品。

## 自测清单

- [ ] 我能描述出具体是"一小撮**谁**"——有职业/场景特征,不是"所有人"
- [ ] 我知道这群人现在用**什么方式凑合**这件事(越难看越真实)
- [ ] 我在**差评区**找到了重复出现的骂声,证明有人在为这件事付钱但不满意
- [ ] 我问过 **≥3 个真实的人**关于过去行为,而不是"会不会用"
- [ ] (AI 产品)我能用一句话答出"为什么不直接用 ChatGPT",且答案不是"界面更好看"
- [ ] 我知道**怎么持续触达**这群人,在写代码之前就想清楚了
