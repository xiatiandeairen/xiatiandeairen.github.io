> 这章解决一个根本性的产能问题：独立开发者最稀缺的资源是时间和注意力，而 AI 编码工具是目前最划算的杠杆——但只有你真正当好那个"资深工程师"角色，这根杠杆才不会翘歪。

---

## 为什么这是 solo 最大的结构性变化

大厂工程师习惯的是分工体系：你写后端，有人做前端，有人写测试，有人写文档，有人管 CI/CD，还有个 PM 替你想清楚需求。一旦一个人干，这些坑全是你的。

AI 改变了这个等式。它不能替你做判断，但它能做那些"你知道该怎么做但需要时间敲"的活——样板代码、CRUD、测试用例、文档初稿、正则表达式、SQL 查询、错误日志分析……这些事情堆起来，以前要花你 60% 的时间，现在可以大幅压缩（「待核」具体比例因人因任务差异极大）。

**核心机制是：AI 把你的"执行带宽"拉宽了，让你能把更多时间花在只有你能做的事上——判断、架构决策、产品感、用户沟通。**

但这里有个必须直说的陷阱：AI 会非常自信地犯错。它生成的代码能跑、测试能过，但逻辑可能是错的，架构可能是歪的，安全漏洞可能是藏着的。**不是"AI 产出 = 可以信任的产出"，而是"AI 产出 = 需要你审查的初稿"。** 把这个纪律建立起来，才能真正用好它；放弃这个纪律，迟早翻车。

下面是五条具体打法。

---

## 打法一：用 AI 干脏活累活，把你的时间留给判断

独立开发最耗时的不是"难的事"，而是"繁的事"：写一个 CRUD API、配一套 Docker Compose、给每个函数补单测、整理一份 changelog、把 README 翻成英文……这些事每件都不难，但加起来吃掉大量时间。

**把这些交出去。** 你描述清楚意图，让 AI 出初稿，你审查、调整、合进代码库。

具体哪些适合交出去：

- **样板代码**：路由、中间件、数据库 schema、表单校验逻辑。给 AI 一个你已有的类似模块作为参照，让它照格式生成新的，而不是从头发散。
- **测试用例**：给 AI 看函数签名 + 你自己写的一两个 happy path 测试，让它补边界 case——空输入、超长字符串、并发、权限缺失。这类测试你知道该写，但一条条手敲很枯燥。
- **重构**：把一段你知道该收拾但一直拖着的代码扔给它，让它做一遍 rename / extract function / 消除重复，然后你对着 diff 看一遍，合理的接受，不合理的改回来。
- **文档和文案**：函数注释、API 文档、产品介绍页的第一稿、错误提示文案、onboarding 引导词。你写的文案往往太技术腔，AI 出的初稿往往太通用，两者取中正好。

**Cursor 的工作流是个好样本**（Cursor 是目前内嵌 AI 最深的编辑器之一）。它的 Composer 模式允许你用自然语言描述一个跨文件的改动，AI 给出完整 diff，你逐块 review、accept 或 reject。这个"AI 产出 diff，你 review 合并"的节奏，是目前效率最高的人机协作模式之一——它没有绕过你的判断，而是把你的判断成本从"从头写"降到了"看 diff"。

GitHub Copilot 的思路更轻量：在你正常写代码的时候，它持续预测你下一行要写什么，自动补全。最大的价值不是偶尔补出一个神奇的算法，而是把那些"我知道怎么写但还是要花时间敲"的行消掉，让你保持心流而不是因为样板代码打断思路。

**给你的动作**：列出你这周花了超过 2 小时的重复性开发任务，其中有多少是 AI 可以出初稿的？下周试着把其中两件交出去，记录实际节省了多少时间。

## 打法二：把模糊任务拆成 AI 能可靠完成的小块

"帮我实现用户系统"——这种任务扔给 AI，你会得到一堆要么太简陋要么过度设计的代码，然后你花时间改，最后花的时间比自己写还多。

**AI 的可靠性与任务粒度成反比：任务越大、越模糊，产出质量越不稳定；任务越小、越具体，产出质量越高。**

怎么拆：

1. **先自己想清楚接口，再让 AI 实现**。你先设计函数签名、输入输出、边界条件——这个步骤不能外包，因为这就是"架构判断"。然后给 AI 一个清晰的 spec：`实现一个 parse_email_header(raw: &str) -> Result<EmailHeader, ParseError>，其中 EmailHeader 包含 from、to、subject、date 字段，遇到格式错误返回 Err 并附上具体错误位置`。这种粒度，AI 产出的代码质量会高出一个等级。

2. **给上下文，不要让 AI 猜**。把相关的数据结构、已有的类似实现、你的代码风格偏好一起贴给它。Cursor 能直接引用当前 repo 里的文件作为上下文，这比每次都粘贴代码要高效得多。

3. **一次只做一件事**。不要让它同时重构 + 加功能 + 修 bug。每一件事单独一个 session，产出更可控，你 review 的时候也更清楚它改了什么。

4. **用 AI 帮你拆任务本身**。你有一个模糊的功能需求，可以先让 AI 帮你列出实现步骤、列出需要考虑的边界条件、列出可能的技术方案——把这个当"橡皮鸭对话"，帮你理清思路，然后你做决策，再逐步让它实现。

这个"先拆、再给清晰 spec、再让 AI 实现"的纪律，听起来像是多了一道工序，但它实际上是把你从"改烂代码"的沼泽里救出来。前期多花 20 分钟想清楚，后期少花 3 小时修垃圾。

## 打法三：建立"AI 产出 → 你审查把关"的纪律

这是最重要的一条，也是最容易被跳过的一条。

AI 生成的代码看起来很有信心。它格式规整、命名合理、注释详细，通常第一眼很难看出问题。但它有几类高频错误需要你盯：

**逻辑错误**：它按照"大多数情况下合理的逻辑"生成代码，但你的具体业务场景可能是那个例外。比如它生成的分页逻辑，offset 计算可能差 1；它生成的权限检查，可能漏了一个 admin 覆盖 case。这类错误，跑一遍 happy path 是发现不了的。

**安全盲点**：SQL 拼接没有参数化、文件路径没有 canonicalize、用户输入直接传进 shell——这类问题 AI 有时候会犯，有时候不犯，但它不会主动告诉你它犯了。**凡是涉及用户输入、认证、权限、文件操作、外部 API 调用的代码，你必须人工过一遍安全视角。**

**过度设计**：AI 读了很多"最佳实践"，有时候会给你一个 5 层抽象的架构，其实你就需要一个 50 行的直接实现。它不知道你是一个人、你需要几个月内能维护的代码，而不是一个需要支撑 100 人团队的系统。

**过时 API**：训练数据有截止日期，它可能使用了已经 deprecated 的库 API。涉及到版本敏感的依赖，跑之前检查一下。

**建立 review 纪律的具体操作**：不管多小的 AI 产出，在 commit 之前过一遍 diff。不是"扫一眼"，是"逐行理解"——你要能说出每一行它在做什么、为什么这么做。如果你有一行看不懂、说不清，就是信号：要么让 AI 解释，要么自己重写那一块。**"我 merge 了但没完全看懂"是技术债的起点。**

Meta 内部有一条工程文化：code review 是让另一双眼睛找你没看到的问题，而不是走流程。你用 AI 时，你就是那"另一双眼睛"——它生成，你 review，缺了哪一环都不完整。

## 打法四：用 AI 补齐你的弱项，而不是绕过它

对于大多数从大厂来的后端/全栈工程师，弱项集中在三块：**前端样式、设计稿、文案**。AI 在这三块都能有效托底，让你不用为了一个 landing page 去找外包或者等设计师。

**前端样式**：你知道要什么效果，但手写 CSS / Tailwind 很耗时。把你的意图用自然语言描述给 AI（"一个两栏布局，左侧固定宽度 280px 的侧边栏，右侧内容区，移动端折叠侧边栏"），让它出初稿，你再调。v0.dev（Vercel 出的 AI 界面生成工具）可以从文字描述直接生成 React 组件，带样式、带基础交互，质量比你从头写好不少。这不是让你放弃对 UI 的判断，而是让你从"没有初稿"到"有一个可以批改的初稿"。

**设计稿初稿**：Figma 的 AI 功能、Framer AI 都能从文字描述生成 wireframe 级别的设计稿。你不需要完美的像素稿，你需要的是"能跟自己对话、能给测试用户看"的东西。让 AI 帮你出第一版，然后你根据产品逻辑调整。独立产品早期的"设计"，80% 的时间是在做产品决策，20% 是在做视觉——AI 可以帮你把那 20% 的执行成本压低。

**文案和 copy**：产品描述、定价页文案、错误提示、邮件通知、onboarding 引导——这些加起来有几百上千个字符，每一处都要写对、写清楚、写得不像机器翻译。让 AI 出初稿，你在上面改，比从空白开始写快 3-5 倍（「待核」具体比例因写作能力和场景差异较大）。

**边界在哪里**：这三块你需要的是"判断力"，不是"执行力"——你要知道这个布局对不对、这个文案传达了正确意图没有、这个设计是否符合你的产品定位。**AI 补的是执行带宽，不是判断力。判断力没有捷径。**

## 打法五：从"大厂工程思维"切换到"够用就上"

这是一个心态转换，不是工具问题，但 AI 的出现让它变得更重要。

大厂工程师受训的环境是：代码要可扩展、要高可用、要支持下一个量级、要有完善的监控告警、要有 runbook。这些在大厂是对的，因为你写的东西可能跑在 10 亿 QPS 上，一个小错误是事故。

独立产品的环境是：你现在有 0 个用户，你最大的风险不是系统挂了，而是你浪费了 3 周建了一个没人要的东西。

**"Boring tech"是一条真实的建议**：用你最熟的技术栈，不要为了"AI 产品应该用 X 框架"去学新东西，除非新东西直接解决你的核心问题。Rails / FastAPI / Next.js / SQLite——能跑、你懂、生态好，就用它。把 AI 工具嵌进你熟悉的工作流，而不是为了用 AI 工具去重建工作流。

AI 辅助开发在这里的作用是双面的：它能帮你快速出一个"够用的"版本，但它也会诱惑你去做过度工程——"既然 AI 能帮我写，那顺便把这个抽象成可复用模块、顺便加个插件系统……"。这是一个陷阱。**AI 降低了执行成本，但不改变"过度工程比不做更危险"这个事实**，因为你的注意力还是有限的，过度工程的代码还是需要你维护。

Instagram 最初的版本是 13 个人（「待核」团队规模数字）在几周内做出来的，用的是 Python + Django，没有微服务，没有复杂架构，但它在 24 小时内承载了 25000 名用户（「待核」）。那个版本"够用"，足够让他们知道这件事值得继续做。

**给你的标准**：一个功能，如果 AI 帮你出初稿、你审查合并、能在两天内上线测试，就直接上；如果你发现自己在花时间讨论"这个模块未来怎么扩展"，先停下来问——"现在有用户验证这个方向了吗？"没有的话，那个讨论是浪费。

## 一个完整的工作流示意

以"给独立工具产品加一个邮件通知功能"为例，说明 AI 辅助开发的一次完整节奏：

1. **你先想清楚接口**：什么时候触发、发给谁、内容是什么、失败怎么处理——这步不能外包。
2. **让 AI 出样板**：`用 Resend API，实现一个 send_welcome_email(to: str, name: str) 函数，带重试逻辑，失败写入日志`——给清晰 spec，让 AI 生成代码。
3. **你 review diff**：检查 API key 是否从环境变量读取、错误处理是否正确、重试有没有指数退避、有没有把 API key 打进日志——这步不能外包。
4. **让 AI 补测试**：把函数实现给它，让它写 mock Resend API 的单测，覆盖成功、失败、重试三个 case。
5. **你审查测试**：测试是否真的测了你关心的逻辑，而不是在测 mock 本身。
6. **让 AI 写函数注释 + 更新 changelog**。
7. **你合并上线**。

整个流程，你做判断的步骤（1、3、5）是你的时间，其他步骤 AI 承担执行。这个比例，比你全手工做要高效得多；比你完全信任 AI 产出，要安全得多。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| 用 AI 的姿势 | 让 AI 直接解决问题、产出即接受 | 让 AI 出初稿、自己 review、合理接受不合理回 |
| 任务粒度 | 扔大任务（"帮我实现用户系统"） | 拆成有清晰 spec 的小块再交出去 |
| 遇到 AI 犯错 | 反复重试 prompt，期待 AI 自己修好 | 直接自己改，理解这类错误的规律 |
| 弱项补齐 | 靠 AI 生成、自己不学 | 用 AI 出初稿、在审查中补充判断力 |
| 技术选型 | 用 AI 推荐的"最新最好"栈 | 选最熟的、能跑的，AI 嵌进已有工作流 |
| 对 AI 的预期 | "AI 能替我做决策" | "AI 放大我的执行速度，决策还是我的" |

## 反模式（工程师高发）

- **"AI 能写，所以我不用想清楚接口"**：这会让你花大量时间改 AI 生成的不符合需求的代码，反而更慢。spec 不能省。

- **"代码能跑就合进去"**：没有逐行 review 的 AI 代码是定时炸弹，尤其是涉及认证、权限、输入处理的部分。

- **"AI 帮我做前端了，所以我不用建立产品感"**：AI 能帮你把布局敲出来，但"这个 UI 对用户体验是否正确"的判断，是你必须拥有的能力，不能外包。

- **"AI 提速了，所以我能做更多功能"**：AI 给你的是时间，不是正确性——省下的时间应该用来更快地验证，而不是更快地堆功能。

- **过度依赖单个工具**：Cursor 今天很好用，不代表明天还是最优解。核心是"你能用 AI 完成工作"的能力，而不是对某个具体工具的依赖。

## 动手：这章怎么练

1. **找你这周最枯燥的一个开发任务**，用 AI 出初稿，记录：它花了多少时间、你 review 花了多少时间、发现了几处问题。这会校准你对 AI 真实效率提升的预期。

2. **做一次"脏活"批量外包**：把你的项目里积累的 TODO——补测试、补注释、更新 README——集中一个下午用 AI 全部出初稿，你专门做 review 和合并。感受一下"批量执行"的节奏。

3. **找一个你不擅长的弱项**（最可能是前端或文案），用 v0.dev 或 AI 生成第一个版本，然后基于它做调整，而不是从空白开始。记录这个方式和你以往做法的时间差。

4. **建立一条 review checklist**：把你之前发现的 AI 高频错误列出来（安全、逻辑边界、过时 API……），每次 merge AI 产出之前过一遍这个 list，直到它变成肌肉记忆。

## 真实参照（去看这些）

- **Cursor（cursor.com）**：目前内嵌 AI 最深的代码编辑器，Composer 模式是"AI 产出 diff + 你 review"工作流的最佳实践样本；直接用比读介绍有效。
- **GitHub Copilot**：行级补全，嵌进 VS Code / JetBrains，感受"AI 在你写代码时持续在场"是什么体验，与 Cursor 对比，找到适合你的档位。
- **v0.dev（Vercel）**：从文字描述生成 React + Tailwind 组件，前端弱项者的第一个练习场。
- **Gergely Orosz《The Pragmatic Engineer》**：关于大厂工程文化的深度分析，帮你理解"大厂思维"的来源，以便更清醒地知道哪些该抛弃。
- **DHH / Basecamp 的"boring tech"哲学**：Basecamp 几十年用 Ruby on Rails，强调"熟悉的技术 + 专注业务"——这是对"AI 时代应该选最新最酷栈"最好的反驳。
