> **把前 6 模块串成一次真演练** —— 这一章不再讲新机制,而是带你拿一个具体产品,从"一个模糊的点子"一路走到"有人付费",每一步都回扣对应章节的方法、给出可照做的动作、并对照 Apple / 字节 / Google / Meta "如果是我,会怎么决策"。合上文档,你就能照着 checklist 做自己的第一个产品。

---

## 怎么用这一章

前面六个模块,每一章都在练一种**单项能力**:出题、设计、UI、开发、分发、变现。但真做产品时,这些能力不是并列的,是**有先后、有依赖、会互相打架**的一条链。这一章就是把它们焊成一条链,让你看清:**在哪个阶段该用哪一招,以及哪一招做砸了会让后面全塌。**

我会用一个贯穿全程的**示范产品**。先把话说在前面:**这是一个教学用的虚构例子,不是真实存在的公司,也没有真实数据。** 我编它,是为了让每一步都有具体的对象可抓——你完全可以把它换成你自己的点子,流程一模一样。

**示范产品:`PodKit` —— 给独立播客主的"一期音频 → 可发布物料"工具。**

它做一件事:你上传一期播客的录音(或文字稿),它自动产出三样东西——**可直接贴进 Apple Podcasts / 小宇宙的 show notes(节目笔记)、带时间戳的章节标记(chapters)、3 条可发到 X / 小红书的推广短文。**

为什么选它当例子:它够垂直(只服务播客主)、够 AI 原生(核心价值就是 LLM 的总结与改写能力)、痛点真实(独立播客主自己剪完音频已经精疲力尽,写笔记和推广是纯苦力活)、而且小到一个人三周能做出第一版。下面我们就拿它走完七个阶段。

每个阶段我都给一个**时间盒建议**——不是让你卡着秒表,而是逼你"到点就交、不许无限打磨"。独立开发最大的敌人不是难度,是**没有 deadline 的完美主义**。

## 阶段一 · 出题:这题到底值不值得做

**时间盒:3–5 天。产出物:一句话 JTBD + 5 个真实用户的访谈记录 + 一句"为什么不用 ChatGPT"的回答。**

很多工程师跳过这一阶段,直接开写——这是头号死法。**写代码是最贵的验证方式,出题阶段的目标是用最便宜的方式杀掉烂点子。**

### 第一步:给点子打分(回扣《痛点的解剖学》)

用第 1 模块第一章的六把尺,给 `PodKit` 过一遍:

- **痛 = 现状成本 − 新方案成本**:播客主现在怎么写笔记?要么不写(损失 SEO 和转化),要么自己花 40 分钟边听边敲。新方案是 3 分钟出稿、再花 5 分钟改。差值很大——**这是真痛。**
- **频率 × 强度**:每更一期就要做一次,周更的人一年 50 次。属于"中高频 + 中强度",落在好卖区间。
- **止痛药 vs 维生素**:它止的是"我已经累瘫了还得干苦力"的痛,偏止痛药——比"帮你涨粉"这种维生素好卖。
- **感受者 = 付费者吗?**:独立播客主自己干活、自己掏钱,链路短,好收钱。(如果换成"给 MCN 公司用",链路就变长,难卖。)
- **切换成本**:现状是"手动 + ChatGPT 复制粘贴",切换成本几乎为零,容易撬动。
- **头发着火测试**:去搜搜看——有没有人在 Reddit 的 r/podcasting、小宇宙的创作者群里抱怨"写 show notes 太烦"?如果一搜一大把、还有人在用蹩脚方法硬扛,**说明头发已经着火,题成立。**

**六把尺过下来,`PodKit` 在"痛、频率、付费链路"上都站得住。** 如果你的点子有两三把尺过不了(比如低频、或感受者不是付费者),这里就该停,别往下走。

> **如果是字节会怎么做**:字节做任何新方向前,内部有一套近乎冷酷的"留存能不能起来"的数据预判文化(具体机制「待核」)。它不靠创始人拍脑袋,靠的是"先看这个需求的频率和强度撑不撑得起留存"。你一个人没有它的数据中台,但**那六把尺就是你的平民版数据预判**——逼你在写码前先量一遍。

### 第二步:找题 + 验证(回扣《找对做什么》《验证先行》)

打完分只是你自己觉得痛,**得让真实的人证明痛。** 第 1 模块教过:别问"你会用吗"(人人都说会,没用),要问**过去**——"你上一期的 show notes 是怎么写的?花了多久?有没有因为太烦干脆不写?"

具体动作:**去找 5 个真实的独立播客主聊 20 分钟。** 哪里找?小宇宙评论区、Reddit、播客主的 X。问三件事:

1. 你现在怎么做这件事?(挖现状成本)
2. 最烦的是哪一步?(定位真痛点)
3. 你为这事花过钱吗 / 试过什么工具?(头发着火的硬证据)

**5 个里如果有 3 个以上能清晰描述"这步真的很烦、我试过 XX 但不好用",题就验证了。** 如果他们支支吾吾、说"还好吧",说明痛不够,赶紧改题或换题——**这时候掉头,只亏了一周,不亏三个月。**

> **N=1 不算验证**(这是我自己反复栽过的坑):聊一个人说痛,可能只是运气好碰到个 heavy user。**至少要 3 个独立的、不同类型的播客主都说痛**,你才能区分"这是普遍痛点"还是"个例噪声"。验证可行性这件事,样本量是底线。

### 第三步:答好"为什么不用 ChatGPT"(回扣《AI 套壳的护城河》)

这是 AI 原生产品的**生死问题**。用户会想:我直接把文字稿丢给 ChatGPT 让它总结不就行了,凭什么用你、还付费?

第 1 模块《AI 套壳的护城河》给了五类壁垒,`PodKit` 至少要占住其中两类:

- **workflow 壁垒**:用户用 ChatGPT 要自己想 prompt、自己分三次问(笔记、章节、推广文案)、自己调格式、还要手动找时间戳。`PodKit` 把这一整套流程焊死成"上传 → 一键出三样"。**你帮用户省的不是 token,是"怎么用"——这正是这章的核心论点。**
- **UX / 可靠性壁垒**:章节时间戳这件事 ChatGPT 做不好(它拿不到准确的音频时间轴)。`PodKit` 接转录服务拿到带时间戳的逐字稿,再让 LLM 在此基础上分章,**输出的时间戳是准的**——这是通用聊天框给不了的可靠性。

**这两条占住,"为什么不用 ChatGPT"就有了硬答案:因为我把你要手动拼的活儿焊成了一键,而且时间戳这种它做不准的事我做准了。** 答不出这个问题的 AI 点子,不要做。

> **必须警惕的反向假设**(我栽过的另一个坑):别把护城河建在"OpenAI 永远不会做播客工具"上。平台不做某个垂直场景,通常只是"短期顾不上",不是"永远不会"。你的壁垒要建在**平台做不到的地方**——比如你沉淀了用户历史每期的风格偏好(数据壁垒)、你长在播客主的发布工作流里拔不掉(嵌入壁垒)——而不是"它没空做"。

**阶段一交付清单:**
- [ ] 一句话 JTBD:"独立播客主在**更新完一期、精疲力尽时**,想要**几分钟拿到能直接发布的笔记/章节/推广文案**,以便**不漏掉 SEO 和涨粉但又不用再当苦力**。"
- [ ] ≥3 个真实播客主访谈、确认痛点
- [ ] 一句能怼回去的"为什么不用 ChatGPT"

## 阶段二 · 产品设计:砍到"最小可卖"

**时间盒:2–3 天(纯想清楚 + 画草图,不写码)。产出物:一张功能清单(分"做/不做")+ 一份不确定性兜底方案。**

### 第一步:用体验机制反推形态(回扣《产品体验的底层机制》)

第 2 模块第一章那六条机制(JTBD / 认知负荷 / 摩擦 / aha / Hook / 峰终),现在用来反推 `PodKit` 长什么样:

- **认知负荷**:核心界面就一个上传框 + 一个按钮,**不让用户做任何配置选择**。要选语气?要选长度?第一版全部写死默认值,别给选项。
- **摩擦**:能不能不登录就先用一次?可以——**让用户先免费跑一期、看到结果,再要注册**(要保存历史/导出时才注册)。这正是样板章里讲的"先尝到价值再要注册"。
- **aha**:用户的 aha 是**第一次看到生成结果、发现"这质量真能直接用"**的瞬间。所以路径要砍到:上传 → 等 30 秒 → 看到三块成品。中间一个多余步骤都不要。
- **峰终**:结果出来那一下做个小峰值——比如三块物料漂亮地依次淡入、每块右上角一个醒目的"一键复制"。结尾(复制成功)给个轻反馈"已复制,去发布吧"。

> **如果是 Apple 会怎么做**:Apple 的产品哲学是"把复杂留给自己,把简单交给用户"。初代 iPhone 正面只留一个 Home 键,就是把"导航"这件事的认知负荷收敛成一个永远有效的动作。`PodKit` 的对应动作就是:**不管你的播客多复杂,你只需要做一件事——上传。** 其余的复杂(转录、分章、改写、找时间戳)全在我这边消化。

### 第二步:砍到最小可卖(回扣《范围与优先级》)

第 2 模块《范围与优先级》给了一道唯一的筛选题:**"不做它,产品会死吗?"** 拿这道题筛 `PodKit` 想做的一堆功能:

| 想做的功能 | 不做它,产品会死吗? | 决策 |
|---|---|---|
| 上传音频 / 文稿 | 会(这是入口) | **做** |
| 生成 show notes | 会(核心价值) | **做** |
| 生成带时间戳章节 | 会(差异化核心) | **做** |
| 生成 3 条推广文案 | 不会,但很加分、成本低 | **做(它是 aha 的放大器)** |
| 多语言支持 | 不会 | 砍,v2 |
| 团队协作 / 多人账号 | 不会 | 砍,可能永远不做 |
| 自定义语气/长度/风格 | 不会 | 砍,先写死默认值 |
| 直接发布到各平台 | 不会(用户可以复制粘贴) | 砍,v2 |
| 数据分析看板 | 不会 | 砍 |

**砍完只剩"上传 → 出三样 → 复制"。这就是最小可卖版本(不是最小可用,是最小到用户愿意付钱)。**

第 2 模块还有一招特别适合一个人:**第一版允许手动(Concierge)。** 如果你连转录+生成的全自动管线都不确定值不值得搭,可以先**人肉跑**:用户上传,你自己在后台用现成工具拼出结果、发回去。**先用人肉验证"用户愿不愿为这个结果付钱",再决定要不要把它自动化。** 这能帮你省下大量"自动化了却没人要"的工程浪费。

> **如果是 Google 会怎么做**:Google 搜索把"我想找东西"压成一个空白框,是把减法做到了极致——**克制不是没设计,是最难的设计。** 你做 `PodKit` 时最大的诱惑就是"再加个功能",每加一个都觉得"用户会喜欢"。记住:**功能强 ≠ 好用,Google+ 功能比 Facebook 还多,照样死。** 砍,是这阶段的核心纪律。

### 第三步:给 AI 的不确定性兜底(回扣《AI 产品的不确定性体验》)

这是 AI 原生产品比传统产品**多出来的一道必答题**:LLM 会出错、会跑偏、会偶尔胡说。第 2 模块《AI 产品的不确定性体验》讲的就是怎么兜:

- **结果永远可编辑**:生成的笔记/文案,全部放进可直接编辑的文本框,**默认假设"AI 是初稿,人是终稿"**。绝不做成"只读、不可改"。
- **给用户控制权**:每块物料配一个"重新生成"按钮。不满意?再来一次,成本几毛钱,但用户的掌控感是无价的。
- **诚实标注**:章节时间戳如果转录置信度低,标个浅色提示"时间点可能有偏差,建议核对"——**别假装 100% 准确,用户被坑一次就再也不信你。**

> **如果是 Meta 会怎么做**:Meta 在 AI 生成内容上吃过"幻觉翻车"的公关亏(具体事件「待核」)。教训是:**AI 越是不确定,产品越要把"这是 AI 生成、请你确认"的姿态摆明。** 一个人做产品没有公关团队兜底,**更要从第一版就把"可编辑 + 可重生成 + 诚实标注"做进去**——这是 UX 壁垒,也是信任的地基。

**阶段二交付清单:**
- [ ] 一张"做/不做"功能表,核心只剩 3–4 件事
- [ ] 想清楚要不要先 Concierge 人肉跑
- [ ] AI 输出的"可编辑 + 重生成 + 诚实标注"兜底方案

## 阶段三 · UI:用现成系统快速做出"不丑"

**时间盒:2–3 天。产出物:一套能用的视觉(配色+字体+组件)+ 走通的核心界面。**

工程师在这一步最容易两极分化:要么"我不会设计,随便糊一个"(丑到劝退),要么"我要学 Figma 从头设计"(掉进无底洞)。第 3 模块给的解药是:**别原创,用现成的设计系统站在巨人肩上。**

### 第一步:高级感急救(回扣《高级感的视知觉机制》)

第 3 模块第一章那套"高级感急救包",`PodKit` 直接套用:

- **留白**:别把界面塞满。上传框周围大片留白,高级感一半来自"敢空"。
- **统一间距**:所有间距用 4 或 8 的倍数(8px 网格),视觉立刻整齐。
- **克制配色**:一个主色 + 一个中性灰阶,**别用超过两个彩色**。
- **字体层级**:标题、正文、辅助文字三级,拉开字号和字重对比。

**这几条不需要任何审美天赋,是可以照抄的物理规则。** 照做,你的产品就从"开发者自己糊的"变成"看起来像个正经产品"。

### 第二步:直接用设计系统 + 组件库(回扣《设计系统与组件化》《一人速度栈选型》)

第 3 模块《设计系统与组件化》的核心建议:**一个人没有时间也没必要原创设计系统,直接用成熟的。**

具体动作:用 **Tailwind CSS + shadcn/ui**(或类似的现成组件体系)。按钮、输入框、卡片、加载状态全是现成的、互相协调的组件,你只管拼。**你省下的不是几个小时,是"自己调一套协调视觉"这件你根本不擅长、也不该自己干的事。**

> **如果是字节会怎么做**:字节内部有自己的设计语言体系(如 Arco Design 这类开源出来的组件库),目的就是让海量产品**不必每次重造视觉轮子、还能保持一致质感**。你一个人,等于直接白嫖大厂沉淀的设计系统——**这是 AI 时代独立开发最大的杠杆之一:大厂花几百人年沉淀的东西,你 npm install 一下就有了。**

### 第三步:把 AI 交互范式做对(回扣《AI 产品交互范式》《AI 人机协作 UX》)

`PodKit` 是 AI 产品,有几个 AI 特有的 UX 点第 3 模块讲过,必须做对:

- **等待体验**:LLM 生成要几十秒,**绝不能只转个圈让用户干等**。要么流式输出(文字一点点冒出来,像 ChatGPT 那样),要么显示进度阶段("正在转录…正在分章…正在写文案…")。**让等待变得"可感知有进展",焦虑就少一半。**
- **协作姿态**:界面语气是"我帮你起了个草稿,你来定稿",不是"这是标准答案"。生成区和编辑区融为一体,**用户改起来零摩擦。**

**阶段三交付清单:**
- [ ] 套用高级感急救包(留白/8px 网格/克制配色/字体层级)
- [ ] 用现成组件库(Tailwind + shadcn 之类),不原创
- [ ] AI 等待做成流式或分阶段,不让用户干等

## 阶段四 · 开发:别过度工程,用速度栈三周上线

**时间盒:5–10 天(这是整个流程最长的盒子,但也最容易超期)。产出物:能跑通"上传→生成→复制"的线上可访问版本。**

### 第一步:别过度工程(回扣《indie 心态:小团队大杠杆》)

第 4 模块开篇就在反工程师的本能:**你不是在 Google 写服务,你是一个人验证一个假设。** 所以:

- **不要微服务**,一个单体应用搞定。
- **不要 Kubernetes**,一个平台托管(Vercel / Railway 之类)够了。
- **不要自建用户系统**,用现成 Auth(下面说)。
- **不要为"百万用户"做架构**,你现在连 10 个用户都没有。**为不存在的规模做优化,是最经典的工程师陷阱。**

> **如果是 Google 会怎么做**:Google 的工程师以"为规模设计"著称——但那是因为它**真有**十亿级流量。这恰恰是反过来给你的警示:**你不是 Google,套用 Google 的工程标准是东施效颦。** 你的"正确架构"是"能让我这周上线、下周能改"的架构,不是"能扛双十一"的架构。

### 第二步:速度栈选型(回扣《一人速度栈选型》)

第 4 模块《一人速度栈选型》给的原则:**选你最熟的、生态最全的、能让你一个人覆盖前后端的栈。** `PodKit` 的一种典型组合(不唯一,看你熟什么):

- **全栈框架**:Next.js(前后端一把梭,前端工程师上手快)
- **托管**:Vercel(push 即部署,零运维)
- **数据库**:托管 Postgres(Supabase / Neon 之类)
- **转录**:接现成语音转文字 API,**绝不自己训模型**
- **LLM**:接现成大模型 API 做总结/分章/改写

**核心思想:能买的全买,能租的全租,你只写"把它们粘起来 + 你独有的那点 workflow"的胶水代码。** 这是一个人能三周上线的唯一办法。

### 第三步:auth / 支付直接用现成基建(回扣《auth 支付基建与平台税》)

第 4 模块《auth 支付基建与平台税》讲得很清楚:**登录和支付是已经被解决透了的问题,自己造既慢又危险(尤其支付,合规和安全你扛不起)。**

- **登录**:用 Clerk / Auth0 / Supabase Auth 之类,十分钟接好。
- **支付**:用 Stripe(或国内对应方案)。**接受它抽的那几个点"平台税"**——这点钱买的是合规、安全、发票、退款全套,你自己做这些要花几个月还做不好。

### 第四步:把 AI 工程落成产品(回扣《把 AI 工程落成产品》)

这是 `PodKit` 的技术心脏,第 4 模块《把 AI 工程落成产品》专门讲了从"调通 API"到"做成可靠产品"的差距:

- **Prompt 要工程化**:把生成笔记/章节/文案的 prompt 写成版本化的模板,**别散落在代码各处随手拼字符串**。
- **结构化输出**:让 LLM 返回 JSON(三块物料分字段),而不是一坨文本你再去 parse。
- **失败兜底**:API 超时/限流/返回垃圾,要有重试和降级,**不能让用户看到一个白屏或报错堆栈**。
- **成本意识**:每次生成花多少钱,心里要有数(这直接影响阶段六的定价)。

### 第五步:让 AI 帮你写这些代码(回扣《AI 辅助开发》)

第 4 模块《AI 辅助开发》的重点:**你现在是"指挥 AI 写代码"的人,不是埋头敲的人。** 用 Cursor / Claude Code 之类:

- 让它生成样板(表单、API route、组件),你审。
- 让它写你不熟的部分(比如某个 API 的接入),你验。
- **但架构决策、prompt 设计、什么该砍——这些你必须自己拿主意**,AI 是加速器不是决策者。

**这是 AI 时代独立开发的核心杠杆:一个人 + AI 辅助,产出接近过去一个小团队。** 但杠杆放大的是你的判断,判断错了,放大的就是错误。

**阶段四交付清单:**
- [ ] 单体 + 托管平台,不过度工程
- [ ] 转录/LLM/auth/支付全用现成,只写胶水
- [ ] prompt 版本化 + 结构化输出 + 失败兜底
- [ ] 线上能访问、能跑通核心闭环

## 阶段五 · 分发:搞定前 100 个用户

**时间盒:持续动作(从开发后期就启动),首轮发布集中在 1 天。产出物:一篇发布帖 + 前 100 个真实用户 + 一个 build in public 的更新节奏。**

**做完 ≠ 有人用。** 工程师最大的幻觉是"产品好了用户自然来"——不会的。第 5 模块讲的就是:**分发要和开发一样当成核心工作,甚至要更早开始。**

### 第一步:前 100 人靠手动捞(回扣《冷启动与持续获客》)

第 5 模块《冷启动与持续获客》的硬道理:**前 100 个用户不是"涨"来的,是你一个一个"捞"来的。** 对 `PodKit`:

- 回到阶段一访谈过的那 5 个播客主——**他们是你的种子,先请他们用。**
- 去播客主聚集的地方**一个一个发**:小宇宙创作者群、Reddit 的 r/podcasting、X 上的播客主、即刻的相关圈子。
- **不是打广告,是真诚地说"我做了个解决 XX 痛点的小工具,想请你试试给点反馈"。**

**前 100 人是用来打磨产品和收集证言的,不是用来赚钱的。** 这个阶段你要的是"有人真用了、并告诉你哪好哪烂"。

> **如果是字节会怎么做**:字节做新产品时极其擅长**找准目标人群的聚集地集中投放**(它的增长团队是这方面的机器)。你没有它的投放预算和算法,但你有它没有的东西——**你能亲自和前 100 个用户一对一聊。** 这是 solo 的隐藏优势:大厂的 PM 摸不到的用户,你能直接对话。把这个优势用满。

### 第二步:发布日打法(回扣《发布日打法》)

第 5 模块《发布日打法》:别"悄悄上线"。挑一个集中的发布动作(Product Hunt、V2EX、小红书、相关社区),**把发布当成一个事件来做**:

- 提前预热(build in public 攒一波关注,见下条)。
- 发布当天准备好:一句话说清是什么、一个 30 秒 demo 视频/GIF、一个"为什么做它"的故事。
- **发布不是终点是起点**,当天密集回复每一条评论。

### 第三步:Build in public(回扣《Build in public 与个人品牌》)

第 5 模块《Build in public》:**从开发阶段就开始公开记录。** 在 X / 即刻 / 小红书发"我在给播客主做一个自动出 show notes 的工具,今天踩了 XX 坑""第一版长这样,求反馈"。

为什么这对一个人特别重要:

- **边做边攒关注**,发布日就有第一批人接。
- **公开的过程本身就是内容**,人们爱看"一个人从 0 做产品"的真实故事。
- **建立个人品牌**,这是你下一个产品的免费分发渠道——**一个人最值钱的资产之一,是"做事被人看见"的复利。**

**阶段五交付清单:**
- [ ] 手动捞到前 100 个真实用户(从种子访谈对象开始)
- [ ] 做一次集中的发布(社区/Product Hunt/小红书)
- [ ] 开了 build in public 的更新节奏

## 阶段六 · 变现与留存:让人留下来、并掏钱

**时间盒:产品上线后持续 2–4 周观察。产出物:确定 aha 指标 + 一个定价方案 + 一道 AI 成本对冲护栏。**

### 第一步:盯住 aha 指标(回扣《留存与 aha 指标》)

第 5 模块《留存与 aha 指标》的核心:**找到那个"做到就大概率留下"的动作,然后想尽办法把更多新用户推到那个动作。**

对 `PodKit`,aha 指标的候选是:**"新用户在第一周内成功生成并复制了 ≥1 期的物料"。** 逻辑是——只要他真的把生成结果拿去发布了一次、尝到了"省时间又有效果"的甜头,他下一期大概率还来。

具体动作:**埋点量这个动作的完成率。** 如果很多人注册了却没走到"复制"这一步,说明中间有摩擦/有困惑,回去优化激活路径。

> **必须避开的坑——相关 ≠ 因果**(第 5 模块特意警告过):你观察到"复制过的人留存高",**不等于"逼所有人复制就能提留存"**。可能是本来就想用的人才会复制。所以 aha 指标是用来**找激活路径的北极星**,不是拿来生硬地"刷指标"。别把相关当因果去硬推。

### 第二步:定价(回扣《变现与定价》)

第 5 模块《变现与定价》第一句就是:**"你几乎一定定低了。"** 工程师尤其爱按"我花了多少成本"来定价(成本定价),这是错的。**要按"用户获得了多少价值"来定(价值定价)。**

`PodKit` 的价值定价怎么算:它帮一个周更播客主每期省 40 分钟,一个月省下 ~3 小时。这些时间对一个把播客当事业的人值多少?**远不止订阅费那点钱。** 所以定价锚定"我帮你省下的时间/带来的涨粉",而不是"我每次调用花了几毛 token"。

可落地的方案:

- **免费档**:每月免费跑 2 期(让人尝到 aha,这是获客成本)。
- **付费档**:月费解锁不限次数 + 历史保存 + 导出。**定价别羞涩**——参考第 5 模块,低价会**替你筛掉好客户**、还让产品显得廉价。
- **年付**:提供年付折扣,一次性收回现金流、同时把流失摁下去(第 5 模块专门讲了年付的双重好处)。

> **如果是 Apple 会怎么做**:Apple 从不打"便宜"牌,它锚定的是"你获得的体验和身份",所以敢卖高价、且毛利极高。你做 `PodKit` 时的心理障碍——"卖这么贵会不会没人买"——**90% 在你心里,不在用户那里**(第 5 模块原话)。真正愿意付费的专业播客主,不会因为月费贵几十块就走,反而会因为太便宜怀疑你不靠谱。

### 第三步:AI 成本对冲(回扣《变现与定价》成本敏感一节 +《AI 产品的分发与变现》)

这是 AI 原生产品**独有的一道生死护栏**:第 5 模块特别警告——**固定订阅会被重度用户活活拖亏。** 因为你每次生成都要真金白银付转录费 + LLM 费,而订阅是固定的。一个每天上传 5 期的疯狂用户,可能让你每月在他身上亏钱。

对冲动作:

- **设合理用量上限**(比如付费档每月 N 期,超出再叠加),别做"真·无限"。
- **缓存/复用**:同一期重复生成时复用转录结果,别重复付转录费。
- **选型对成本**:简单任务用便宜小模型,复杂的才上贵模型,**别一律用最贵的**。
- **盯单位经济**:心里永远有一笔账——"每个付费用户每月,我的 AI 成本是多少、收入是多少"。**这笔账是负的,规模越大死得越快。**

> **这条是 AI 产品和传统 SaaS 最大的不同**:传统软件边际成本接近零,用户越多越赚;AI 产品**每个用户都在持续烧你的钱**。不把成本对冲做进定价,你会陷入"越成功越亏"的死亡螺旋。

**阶段六交付清单:**
- [ ] 定义并埋点 aha 指标
- [ ] 价值定价(不是成本定价)+ 年付选项
- [ ] AI 成本对冲护栏(用量上限/缓存/模型选型/单位经济)

## 阶段七 · 复盘:迭代,还是杀掉

**时间盒:上线满 30 天做第一次正式复盘。产出物:一个诚实的"继续 / 转向 / 杀掉"决定。**

走到这里,你有了真实数据。**复盘的目的不是自我感动("我做出来了!"),是冷静判断这题到底成不成。** 看三类信号:

**信号一:留存(最硬)。** 用了第一次的人,第二周还回来吗?付费用户续费吗?**留存是产品价值的终极裁判**——拉新可以靠努力,留存骗不了人。如果用户来一次就再不回来,说明价值没真正成立(可能是产品问题,也可能是题本身不够痛)。

**信号二:付费转化。** 免费用户里有多少转付费?如果几乎为零,要么定价/价值主张有问题,要么免费档给太多没人需要升级。

**信号三:获客可持续性。** 前 100 人是你手动捞的,**第 101 到 500 人能不能更省力地来?** 有没有自然的口碑/SEO/转介绍开始转动?如果每个用户都要你手动捞,这门生意规模不起来。

根据信号,做一个诚实的决定:

- **数据起来了(留存稳、有人付费、获客能持续)→ 迭代。** 这时才回头做阶段二砍掉的 v2 功能(多语言、直接发布、自定义风格),**用真实需求驱动,而不是你拍脑袋。**
- **数据平平但有微弱信号(某一小撮人极爱)→ 转向(pivot)。** 也许题没全错,但人群/形态要调。聚焦那一小撮真爱用户,看他们到底为什么爱。
- **数据是死的(没人留、没人付、捞了也不用)→ 杀掉,别恋战。** 这不是失败,是**用三周 + 一笔小钱买到了一个明确答案**。把它当成下一个产品的学费,启动下一题。

> **必须对自己诚实的一条铁律**(我反复栽过):如果你发现自己一次次给产品"重新定位"(改目标人群、改卖点、改叙事),还安慰自己"我们在精进"——**停下来。多次 reframe 往往不是迭代,是这题的核心假设从一开始就没站稳,只是被"我在努力优化"的叙事盖住了。** 该杀掉时杀掉,比硬撑半年体面得多。

> **如果是字节会怎么做**:字节以"赛马 + 快速关停"著称——同时跑多个产品,数据不行就果断砍,不和创始人的面子较劲(具体机制「待核」)。**你一个人更要学这种冷酷:你的时间是最稀缺的资源,沉没成本不是继续的理由。** 一个杀得干脆的人,能比一个舍不得放手的人多试三倍的题。

## 把七个阶段连起来看

回头看这条链,你会发现一个规律:**每个阶段的产出,都是下个阶段的输入;某个阶段偷懒,债会在后面连本带利还。**

- 阶段一出题没验证 → 阶段四辛苦做出来的东西没人要。
- 阶段二没砍范围 → 阶段四永远做不完、上不了线。
- 阶段三不用现成系统 → 你把命耗在你不擅长的视觉上。
- 阶段六不算 AI 成本 → 越成功越亏。
- 阶段七不敢杀 → 你被一个死题拖住,做不了下一个活题。

**这就是为什么"全栈"对 AI 时代的一人公司是必修:不是要你每样精通,是要你每个环节都能判断"这步做对没有、该不该往下走"。** 单点最强但链条断一节,产品照样死——这正是工程师转型最大的盲区:**代码只是七分之一。**

## 新手 vs 资深

| | 新手(工程师本能) | 资深(一人公司视角) |
|---|---|---|
| 起手第一件事 | 打开编辑器开写 | 找 3 个真实用户验证痛点 |
| 对 "为什么不用 ChatGPT" | 没想过 / 觉得功能多就行 | 答得出 workflow + 可靠性壁垒 |
| 功能范围 | 想到的都想做 | 用"不做会死吗"砍到只剩 3 件 |
| UI | 自己糊 / 从头学设计 | 套现成设计系统 + 高级感急救包 |
| 架构 | 为百万用户设计 | 为"这周上线、下周能改"设计 |
| AI 输出 | 当成标准答案直接展示 | 可编辑 + 可重生成 + 诚实标注 |
| 上线后 | 等用户自己来 | 手动捞前 100 + build in public |
| 定价 | 按成本定 / 不好意思要钱 | 按价值定 + 年付 + 成本对冲 |
| 数据不行时 | 舍不得,反复 reframe 硬撑 | 该杀就杀,把答案当学费 |

## 反模式(工程师高发)

- **跳过出题直接开写**:最贵的验证方式,大概率做出没人要的东西。
- **答不出"为什么不用 ChatGPT"还硬做**:AI 原生产品的头号死法。
- **范围不砍、追求"做全"**:一个人永远做不完,上不了线。
- **把命耗在自己不擅长的 UI 原创上**:有现成系统不用,纯属浪费。
- **为不存在的规模过度工程**:你连 10 个用户都没有,先别想百万。
- **把 AI 输出当成标准答案**:不给编辑、不给重生成、不标不确定性,翻车一次失信到底。
- **做完坐等用户**:分发不是产品好了的附属品,是和开发同等的核心工作。
- **固定订阅不设上限**:被重度用户拖到"越成功越亏"。
- **数据死了还反复 reframe**:多次重新定位往往是题没站稳,不是你在精进。

## 动手:这一章怎么练

1. **挑一个你自己的真实小痛点**(最好是你自己天天遇到的),用阶段一的六把尺打分,写出一句话 JTBD 和"为什么不用 ChatGPT"。**今天就做,30 分钟。**
2. **去找 3 个真实的人验证**(不是问会不会用,是问过去怎么做的)。这一步决定后面六步值不值得走。
3. **照着阶段二到阶段四的清单,给自己定一个"21 天上线"的时间盒**,每个阶段卡死交付物,不许无限打磨。
4. **从开发阶段就开一个 build in public 的小号**,每天发一条进展——逼自己暴露在真实反馈下,也提前攒发布日的种子。

## 第一个产品 · 30 / 60 天行动清单

**第 1 周(出题 + 设计)**
- [ ] Day 1–3:六把尺打分 + 写一句话 JTBD + 答"为什么不用 ChatGPT"
- [ ] Day 3–5:找 ≥3 个真实用户验证痛点(过去怎么做、为它花过钱吗)

**第 2–3 周(UI + 开发,目标:上线)**
- [ ] 套现成设计系统 + 高级感急救包,走通核心界面
- [ ] 单体 + 托管平台,转录/LLM/auth/支付全用现成
- [ ] prompt 版本化 + 结构化输出 + 失败兜底
- [ ] AI 辅助开发提速,但架构和 prompt 自己拿主意
- [ ] **Day 21 目标:线上能访问、能跑通核心闭环**

**第 4 周(分发)**
- [ ] 从种子用户开始手动捞前一批真实用户
- [ ] 开 build in public,持续发进展
- [ ] 做一次集中发布(社区 / Product Hunt / 小红书)

**第 5–8 周(变现 + 留存 + 复盘)**
- [ ] 埋点 aha 指标,优化激活路径

- [ ] **Day 30:第一次轻复盘**(留存有没有微弱信号?)

**唯一的铁律:到点就交,别无限打磨。一个上线了被验证"不行"的产品,价值远高于一个永远在打磨、永远没见过用户的"完美"半成品。**
