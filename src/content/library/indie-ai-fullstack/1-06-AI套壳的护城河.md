> **机制先行** —— 模型人人能调,prompt 谁都会写,这一章不教"怎么调 API",教"既然底座是租来的,你的壁垒到底长在哪、为什么它能挡住别人"。看懂这几类护城河,你做每一个产品决策时都能回答那个要命的问题:**用户为什么不直接用 ChatGPT?**
> **这是核心章**:把护城河拆成 5 类,每一类先讲机制(为什么成立),再配 2–3 个大厂真实决策深拆,最后给 solo 一套可操作策略。结尾用一份清单,教你怎么给自己挖护城河。

---

## 先把"套壳"这个词,从骂人话改成中性词

"套壳"在中文社区基本是个贬义词:调个 OpenAI 接口、套个对话框、收点会员费,被骂"没有技术含量、风一吹就倒"。

这个判断对一半。**对的那一半**:模型本身确实不是你的护城河。你今天用 GPT,明天人家也能用 GPT;你能调 Claude,竞品也能调 Claude;模型厂商一发布新功能,可能直接把你这层"壳"的价值清零。底座是租来的,租来的东西护不了城。

**错的那一半**:几乎所有软件都是"套壳"。Notion 是数据库的壳,Figma 是图形渲染的壳,Stripe 是银行清算网络的壳——没人说它们没护城河。**护城河从来不在你"调用了什么",而在你"在调用之外多做了什么、多攒了什么、多嵌进了哪里"。**

所以这一章真正要回答的不是"套壳行不行",而是:**当模型这层底座对所有人开放,差异化只能往哪几个方向找?** 答案有五类。先说一个贯穿全章的判断:**这五类护城河,大厂样样都比你深,你拼正面必输;solo 唯一的活路是往垂直的深处走,挖一条窄而深的沟,深到大厂懒得弯腰、用户离不开。**

下面五类,从最浅、最容易被复制的,讲到最深、最难被撬动的。

---

## 第一类:workflow 壁垒 —— 你帮用户省掉的,不是 token,是"怎么用"

最容易被忽视、也最被低估的一类。

**机制**:通用模型给你的是"原始能力",像一台没装任何软件的裸机。用户要拿它解决一个具体问题,中间隔着一大段苦工:**怎么问(prompt 工程)、问几步(多轮编排)、怎么把上一步结果喂给下一步(流程串联)、怎么把模型的散装输出整理成能用的成品。** 这段苦工,通用产品(ChatGPT)默认丢给用户自己扛。**workflow 壁垒,就是你把这段苦工一次性替用户做完、固化成一键流程。**

**为什么它成立**:大多数用户不是不会用 AI,是**不愿意为了一个具体任务,反复试探"该怎么用 AI"**。一个律师要让 AI 审合同,他得自己琢磨:贴全文还是分段?让它先列风险点还是先逐条标注?输出要表格还是清单?这些他试错三次才摸到门道——而你可以把这三次试错的最优解,焊死成一个按钮。**用户买的不是模型能力,是"已经替我想好怎么用"这件事。** 这正是上一章那把尺的第一个答案:用户不直接用 ChatGPT,因为 ChatGPT 给的是裸能力,你给的是"针对我这个活儿,已经调好的流程"。

**深拆 · 剪映 —— 把"会用 AI 剪辑"焊成一个按钮**
字节的剪映是 workflow 壁垒的范本。底层的能力(字幕识别、智能配音、自动卡点、一键成片)很多是可调用的 AI 能力,理论上一个会写代码的人都能拼出来。但剪映赢在哪?它把"一个普通人想做一条像样的短视频"这条完整 workflow,拆成了傻瓜级的步骤:导入素材 → 自动识别语音生成字幕 → 套模板 → 一键卡点配乐 → 导出。**一个完全不懂剪辑的人,跟着流程走,十分钟出片。** 它省掉的不是某一项技术,是"你得先学会剪辑、再学会怎么指挥 AI"这整段认知成本。这就是为什么剪映能从专业工具的缝里长成国民级应用(具体用户量「待核」)——它把 workflow 做成了产品本体。

**深拆 · 豆包 / Coze —— 把"编排"本身做成产品**
字节的 Coze(扣子)更进一步:它干脆承认"workflow 才是价值",把"怎么把多个模型调用、插件、知识库串成一个 bot"这件事,做成了一个可视化搭建平台。普通人不写一行代码,拖拖拽拽就能把"触发 → 调模型 → 查知识库 → 调外部 API → 整理输出"这条链路搭起来。**Coze 卖的根本不是模型,是"编排能力的平民化"。** 这恰恰证明:在大厂眼里,workflow 都是一门独立生意,何况 solo。

**给 solo 的策略**:
- **盯一个具体到不能再具体的职业动作**。不是"帮律师",是"帮劳动法律师把一份劳动合同里对劳方不利的条款逐条标出来,并给出修改建议"。越窄,你越能把 workflow 调到极致。
- **把你自己试错出来的最优 prompt 链,焊死成默认流程**。用户看到的应该是"上传合同 → 出报告",看不到背后你调了几次、prompt 长什么样。**你的 prompt 工程,就是你的产品,但绝不能暴露成用户的负担。**
- **警惕**:纯 workflow 壁垒是这五类里最浅的,因为竞品看到你的产品就能逆向你的流程。它必须和后面几类(尤其数据、嵌入)叠加,才扛得住。

## 第二类:数据壁垒 —— 平台拿不到的那部分,才是你的

最硬、最被神化、也最容易自欺的一类。

**机制**:模型是公开的,但**喂给模型的私有数据不是**。同样问一个问题,你的产品能拿到用户/行业的专有数据当上下文,通用产品拿不到,你的答案就比它准、比它贴。**数据壁垒的本质是:你掌握了一份别人(尤其平台)拿不到的私有数据,并把它变成模型回答的燃料。**

**为什么它成立(以及它的陷阱)**:这条成立有个**严苛的前提**——数据必须是真正专有、且持续积累的。很多人把"我用了 RAG、我建了知识库"当成数据壁垒,这是自欺:你那份知识库如果是公开资料喂出来的(爬来的网页、公开 PDF),那它对所有人开放,**不构成任何壁垒**。真正的数据壁垒只有两种来源:**(a) 你独家拥有的私有数据**(某个行业内部多年积累、签了独家的数据源);**(b) 用户在使用你产品的过程中,不断产生、且越用越多的数据**(他的历史、他的偏好、他纠正过的结果)。后者尤其关键,因为它会形成**数据飞轮**:用户越用,你越懂他,产品越好用,他越离不开——这才是能自我加固的壁垒。

**深拆 · Gemini —— 数据 + 分发,平台级的双护城河**
Google 的 Gemini 是"数据壁垒"的天花板,也是 solo 最该看清"为什么拼不过"的样本。Gemini 的护城河根本不在模型本身有多强,而在两件 solo 永远摸不到的资产:
- **数据**:Google 手握全球最大的搜索索引、YouTube 的全部视频、Gmail / Docs / Maps 里海量的用户行为(在隐私合规范围内)。当 Gemini 回答一个问题,它背后能调动的私有 + 实时数据,是 ChatGPT 都未必比得了的。
- **分发**:Gemini 被直接焊进 Android、Chrome、Google 搜索框、Workspace。几十亿台设备的默认入口(具体数字「待核」),不需要你"去下载一个 App"。

**这两件事的共同点:都是平台级资产,靠十几年的基础设施和用户关系堆出来的,solo 砸钱砸时间都换不来。** 看清这一点不是让你绝望,是让你别犯傻:**永远不要在"数据广度"和"分发入口"上跟平台拼正面。** 你要找的是平台那份大数据里**照顾不到的垂直死角**——比如某个小众行业的专有数据,大到 Google 不屑于专门做,小到只有深耕该行业的人才拿得到。

**深拆 · 字节的推荐数据飞轮 —— 数据壁垒的"飞轮"形态**
抖音 / 今日头条的推荐引擎,是"用户产生数据 → 反哺模型"这个飞轮最凶的案例。你每一次划走、停留、点赞、看完,都在实时喂养它的推荐模型;模型越懂你,推得越准,你越停不下来,又产生更多数据。**新玩家就算拿到一模一样的算法,也拿不到字节这些年沉淀的行为数据,冷启动时推荐就是不准,留不住人,也就攒不出数据——飞轮转不起来。** 这是数据壁垒最理想的形态:**它不靠一次性的独家数据,靠"用户使用本身"持续加固,时间越久,沟越深。**

**给 solo 的策略**:
- **诚实地问自己:我这份数据,平台真的拿不到吗?** 如果答案是"爬公开网页",那不叫数据壁垒。如果是"我在这个行业有十年人脉,能拿到别人签不到的内部数据",或者"用户用我的产品会不断标注、纠正、积累他的私有库",那才是。
- **优先设计数据飞轮,而不是囤一次性数据**。让用户在使用中产生的数据,沉淀回你的产品、让产品对他越来越好用。**一次性独家数据会过期,飞轮会增值。**(这也呼应了你做产品时反复要回到的那把尺:用户为什么不走?因为他在你这儿攒的东西,带不走、换地方就没了。)
- **警惕一个用户语义陷阱**:你跟用户说"我们用了你的私有数据让结果更准",用户理解的是"安全、专属";但如果你的数据其实是公开的,你这句话就是在透支信任。**你声称的壁垒,和你实际拥有的壁垒,必须一致**——这是诚实问题,也是长期生死问题。

## 第三类:嵌入壁垒 —— 长在用户已经离不开的地方,拔不掉

最稳、最不性感、却最被工程师低估的一类。

**机制**:一个工具如果只是"用完即走"的独立网站,用户换一个竞品的成本几乎为零(换个网址而已)。但如果你的能力**长进了用户每天已经在用的工具或流程里**——他的代码编辑器、他的设计软件、他的工单系统、他团队的协作流——那么"换掉你"就意味着"打断我整个工作流",这个**切换成本**就是你的护城河。**嵌入壁垒的本质:不是把用户拉到你的地盘,而是把你的能力送进用户的地盘,长成他流程的一部分。**

**为什么它成立**:人是路径依赖的动物。**一个嵌进日常工作流的工具,哪怕单点能力不是最强,也比"单点最强但要我跳出来单独用"的工具更难被替换。** 因为替换的代价不是"学一个新工具",是"重新组织我整套已经顺手的流程"。这个代价,远比工程师想象的高。这也是上一章那把尺的又一个答案:用户为什么不直接开 ChatGPT 网页?因为你的能力已经在他的编辑器里、在他选中文字的右键菜单里,**他根本不需要"切出去"——切出去本身就是摩擦。**

**深拆 · Apple Intelligence —— 端侧 + 系统级嵌入,嵌到操作系统里**
Apple 的打法是嵌入壁垒的极致形态。Apple Intelligence 不是一个让你"打开的 App",它被焊进了 iOS / macOS 的系统层:**任何 App 里选中文字,系统级的改写 / 总结就在右键菜单里;Siri、通知摘要、邮件、备忘录,AI 能力无处不在又无需你"专门调用"。** 再加上**端侧推理**(很多任务在你手机/电脑本地跑,不上云),Apple 把两件 solo 摸不到的东西叠在了一起:
- **系统级嵌入**:它不跟你抢"要不要打开 ChatGPT",它直接成了你操作系统的一部分,**你想绕都绕不开**。
- **端侧 + 隐私**:本地处理 = 数据不出设备,这是 Apple 用硬件 + 操作系统 + 芯片(自研芯片的端侧算力)绑在一起的能力,**没有这条全栈链,谁都做不到**。

**对 solo 的意义:Apple 嵌的是"它自己的操作系统",这是平台级资产,你学不了。但嵌入这条思路本身,solo 完全能用——只不过你嵌的不是 iOS,是某个垂直工具。** 比如把你的 AI 能力做成某个设计软件的插件、某个 IDE 的扩展、某个 SaaS 的集成。你嵌进的那个工具就是你的"操作系统"。

**深拆 · GitHub Copilot —— 嵌进编辑器,而不是再开一个网页**
微软/GitHub 的 Copilot 是嵌入壁垒最清晰的产品级案例。它的核心洞察极其简单:**程序员写代码时,最不想做的就是"切出编辑器、打开一个聊天网页、复制代码过去问、再把答案贴回来"。** Copilot 直接长在 VS Code 的光标里——你写着写着,补全就浮现在下一行,Tab 一下就接受。它跟 ChatGPT 调的可能是相近的底层能力,但**体验是天壤之别:一个长在你手指底下,一个要你切窗口**。这个"零切换"的嵌入,就是它在编码场景碾压通用聊天框的根本原因。**注意:Copilot 嵌入的位置(编辑器),恰恰是 solo 也能嵌的位置——做插件,不一定要做平台。**

**给 solo 的策略**:
- **别总想着"把用户拉到我的网站"**。问一句:**用户解决这个问题时,他人已经在哪个工具里?** 然后把你的能力做成那个工具的插件 / 扩展 / 集成,送进去。
- **嵌入会带来真实的切换成本,这是 solo 少有的、能正面对抗"被复制"的壁垒**。竞品就算抄了你的功能,也得重新进用户的工作流,而你已经在里面了。
- **嵌入还顺手解决了分发**(下一类会讲):你嵌进的那个工具,它的用户就是你的潜在用户,你蹭它的生态做冷启动。

## 第四类:UX / 可靠性壁垒 —— 把"能用"做成"好用",兜住模型的不确定性

最考验产品功力、也最容易被工程师当成"锦上添花"而错过的一类。

**机制**:LLM 有个绕不开的特性——**它是概率性的,会胡说、会出错、会时好时坏(幻觉、不稳定)。** 通用产品(ChatGPT)默认把这个不确定性甩给用户:它给你一段可能对也可能错的输出,信不信、怎么核、出错了怎么办,你自己看着办。**UX / 可靠性壁垒,就是你在你的垂直场景里,把这个"不确定性"兜住——用产品设计、校验、兜底、引导,把一个"概率上能用"的模型,做成一个"用户敢托付"的产品。**

**为什么它成立**:这是 LLM 时代特有的、也是最容易被低估的护城河。**模型能力是公开的,但"如何在一个具体场景里把不可靠的模型包装成可靠的体验",是大量场景化的脏活、细活,没法靠调用 API 抄走。** 一个律师用 AI 审合同,他最怕的不是"AI 不够强",是"AI 漏了一条致命条款而我没发现"。谁能在这个场景里把"漏判""幻觉"兜住——比如强制让 AI 标注它的依据、对高风险结论二次校验、对没把握的地方明确说"这条我不确定,请人工复核"——谁就赢得了这个场景里用户的信任。**而信任,是垂直专业场景里最贵的东西。** 这又一次呼应那把尺:用户为什么不用通用 ChatGPT 干这个专业活?因为通用产品不对结果负责、不兜错,而你在你的垂直场景里把可靠性做到了"敢用"。

**深拆 · Apple 的"宁可不做,也不出丑" —— 可靠性即品牌**
Apple Intelligence 在能力上常被吐槽"不如 ChatGPT 聪明",但 Apple 的产品哲学揭示了可靠性壁垒的另一面:**它宁可功能保守、上线慢,也极力避免 AI 在用户面前出丑(给出离谱、尴尬、出错的结果)。** 因为 Apple 押的是"用户对它的信任"——一个会乱说话的 AI,损伤的是整个品牌几十年攒下的"靠谱"形象。**对 Apple,可靠性不是功能,是品牌资产。** 这对 solo 的启示极深:在你的垂直场景里,**一次严重的胡说,可能比十次平庸的回答更致命。** 把"绝不在关键处出错"作为铁律,你就赢得了通用产品给不了的那种信任。

**深拆 · 反面镜子 · Google 的 AI 翻车 —— 不可靠如何反噬**
作为对照,Google 早期 AI 产品有过公开的翻车:无论是搜索的 AI Overviews 给出过荒唐建议,还是更早的演示出错(具体事件细节「待核」),都引发过舆论反噬。**这说明哪怕是 Google 这种数据和分发壁垒拉满的平台,可靠性一旦失守,照样在用户信任上扣大分。** 反过来想:**可靠性是一条平台都未必守得住、但 solo 在窄场景里反而可能守得更好的护城河**——因为你只盯一个场景,你能把这个场景的所有出错模式都摸透、兜死,而平台要面对千万种场景,顾不过来每一个角落。**这恰恰是 solo 以窄打宽的机会。**

**深拆 · 剪映/豆包的"兜底体验" —— 把不确定性藏进顺滑**
回到字节:剪映的"一键成片",背后调用的 AI 也会出错(卡点卡歪、字幕识别错字),但产品的设计是**让你能极轻松地微调、撤销、重来**——AI 给个 80 分的草稿,你顺手调到 95 分。它不假装 AI 完美,而是**用"AI 起草 + 人轻松收尾"的体验,把不确定性兜成了"协作"而非"翻车"。** 这是 UX 兜底的典型:**不跟模型的不确定性硬刚,而是用产品设计把它驯服成用户可控的东西。**

**给 solo 的策略**:
- **在你的垂直场景里,把模型所有的出错模式列出来,逐个设计兜底**:高风险结论二次校验、不确定时明确说"不确定"、给出依据让用户能核、提供轻松的修正/撤销。**这是抄不走的脏活,也是你最该花力气的地方。**
- **把"可靠"做成你的品牌**。在专业垂直场景,"这个工具从不在关键处骗我"是比"这个工具很聪明"强得多的口碑。
- **记住一个反直觉**:**有时"克制"比"全能"更值钱**。一个只做一件事但从不出错的工具,比一个什么都能做但时常翻车的工具,更让人敢托付。

---

## 第五类:分发 / 品牌壁垒 —— 用户先想起谁,谁就赢了一半

最被工程师厌恶("我不想做营销")、却最决定生死的一类。

**机制**:产品再好,用户不知道、想不起、找不到,等于零。**分发壁垒,是你能持续、低成本地把产品送到目标用户面前的能力;品牌壁垒,是用户在某个需求出现时,第一个想起的是你。** 这两件事,决定了你的产品能不能被用到——而被用到,是其他一切护城河的前提。

**为什么它成立**:AI 套壳产品的技术门槛正在快速趋零,**这意味着竞争的胜负手,从"谁做得出来"越来越向"谁触达得到用户、谁占住了用户心智"转移。** 当一百个人都能做出功能相近的产品,赢的那个往往不是技术最好的,是**让目标用户群第一个想起、第一个找到**的那个。对 solo 来说,这恰恰是最反直觉、也最容易被忽视的战场——工程师习惯认为"东西好自然有人用",但事实是:**没有分发,再好的东西也烂在硬盘里。**

**深拆 · Gemini 的分发 —— 平台级的"默认即占领"**
再看一次 Gemini,这次只看分发。Google 不需要"说服你去用 Gemini",它直接把 Gemini 设成 Android 的默认助手、塞进搜索框、嵌进 Chrome 和 Workspace。**几十亿个默认入口(具体数字「待核」),用户甚至不需要"选择"用它,它就在那儿。** 这是分发的终极形态:**用平台的默认位,把分发成本降到零。** 同样,微软把 Copilot 塞进 Windows、Office、Edge,也是一样的逻辑——**用已有的几十亿装机量,做新产品的分发底座。**

**对 solo 的意义:平台靠"默认入口"做分发,这条你彻底学不了——你没有操作系统、没有浏览器、没有几十亿装机量。** 所以 solo 的分发不能靠"占据入口",只能靠**内容 + 社区 + 垂直口碑**:在目标用户聚集的地方(某个专业论坛、某个垂直社群、某个内容平台)持续输出、被人记住。

**深拆 · Meta 的 Llama —— 用"开源"做分发和生态**
Meta 的 Llama 是一个绝妙的、非典型的分发策略案例。Meta 自己不是要靠 Llama 直接赚订阅费,它**把 Llama 开源**,让全世界的开发者免费用、在它基础上做产品。这步棋的逻辑是:
- **用开源换生态**:当全球开发者都基于 Llama 构建,Llama(及其背后的标准、工具链)就成了一种**事实上的基础设施**,Meta 由此占据 AI 生态的话语权,而不被 OpenAI / Google 的闭源模型卡脖子。
- **这是"分发"的另一种形态**:不是把产品分发给最终用户,是**把"底座"分发给开发者,让开发者替你扩散影响力**。

**对 solo 的意义有两层:其一,看清平台的"开源"是商业战略,不是慈善**——它在用免费换生态位。**其二,这反而是 solo 的机会**:既然 Llama 这类强力开源模型免费可用,你的成本结构、你对单一闭源厂商的依赖,都可以被开源模型改善。**平台开源的底座,正好是 solo 可以站上去的肩膀**——你拼不了它的生态影响力,但你能白嫖它的能力做你的垂直产品。

**深拆 · Threads —— 品牌/关系的分发,也有天花板**
Meta 推 Threads 时,几乎是"一键导入你的 Instagram 关注"——用 IG 的现成社交关系,给 Threads 做了一次史上最快的冷启动(早期增长极猛,具体数字「待核」)。**这是用已有产品的用户关系,给新产品做分发,平台级玩法的又一例。** 但 Threads 也暴露了分发的天花板:**导来的用户能不能留下,最终还是回到前四类护城河**——有没有独特价值、有没有形成习惯。**分发能让人来,但留不留得住,要靠产品本身。** 这对 solo 是个提醒:**别把全部赌注押在某次分发爆发上,分发是入口,不是终点。**

**给 solo 的策略**:
- **放弃"占据入口"的幻想,转向"垂直心智"**。你做不到无处不在,但你能做到"在某个细分人群里,提到这个需求第一个想起你"。**窄人群里的强心智,比宽人群里的弱认知值钱得多。**
- **把分发当成产品的一部分来设计,而不是做完产品再想**。你为谁做、他们聚在哪、你怎么持续出现在那儿——这些和功能同等重要。
- **善用开源底座降成本、降依赖**。Llama 这类开源模型,是 solo 对抗"被单一闭源厂商卡脖子"的现实选项。

## 全解剖:把五类护城河叠在一个 solo 案例上想象

假设你做一个"给独立播客主的 AI 制作助手"。单独看任何一类,你都拼不过大厂;但五类叠起来,你就有了一条大厂懒得弯腰、用户离不开的窄沟:

1. **workflow**:把"录音 → 降噪 → 转文字 → 剪掉口水话 → 生成 shownotes 和社媒文案"这条完整流程焊成一键,播客主不用自己琢磨"该怎么指挥 AI";
2. **数据**:用户每期节目的风格、他纠正过的转写、他的常用术语,沉淀成他专属的库,**越用越懂他**(数据飞轮);
3. **嵌入**:做成他已经在用的录音软件 / 发布平台的集成,长进他的流程,不让他切出来;
4. **UX/可靠性**:转写出错能一键修、AI 不确定的地方明确标出来、绝不把人名地名瞎编进 shownotes;
5. **分发/品牌**:在播客创作者社群里持续输出、被口耳相传,成为"独立播客主第一个想起的 AI 工具"。

**看懂这个叠加,你就明白:solo 的护城河不是某一类拉满,是五类各挖一点、叠成复利,且全部死死扣住一个垂直人群。** 大厂在每一类的单项上都比你深,但它不会专门为"独立播客主"把五类都做到位——**这个垂直的深处,就是你的城。**

## 反例:为什么大量套壳产品"风一吹就倒" —— 通用 + 浅

最常见的死法,是同时犯两个错:**做通用,而且每一层都浅。**

一个典型的失败套壳:对话框 + GPT,号称"什么都能问"。它栽在哪?对照那把尺逐条问——**用户为什么不直接用 ChatGPT?** workflow?没有,它就是个裸对话框,没替用户省任何"怎么用"的功夫;数据?没有,它喂模型的全是公开信息;嵌入?没有,它是个孤零零要你专门打开的网站;可靠性?没有,模型怎么说它就怎么显示,出错全甩给用户;分发?没有,做完就指望"东西好自然火"。**五类护城河一条都没挖,它和 ChatGPT 的唯一区别是 logo——这种产品,模型厂商一个版本更新就能把它清零。**

**它的根本错误是定位:它在跟 ChatGPT 拼"通用"。** 而"通用"恰恰是大厂用平台级数据 + 分发筑死的战场,solo 拼通用等于自杀。**活下来的套壳,无一例外是放弃了通用、扎进了某个垂直深处**——窄到大厂不屑做,深到用户离不开。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| 对"套壳"的判断 | "没技术含量,迟早完" | "套壳没错,问题是壳之外挖了几条沟" |
| 护城河长在哪 | "我的 prompt 调得好" | workflow + 数据飞轮 + 嵌入 + 可靠性 + 分发,叠起来 |
| 跟大厂的关系 | 想做得比 ChatGPT 更全 | 绝不拼通用,扎进大厂懒得弯腰的垂直深处 |
| 对数据的理解 | "我建了知识库就是数据壁垒" | "公开数据不算;专有 + 越用越多的飞轮才算" |
| 对模型更新的态度 | 担心被官方功能清零 | 我的壁垒在模型之外,官方更新动不了我的沟 |
| 对分发的态度 | "东西好自然有人用" | 分发是产品的一部分,从第一天就设计 |
| 那把尺 | 没想过 | 每个决策都问:用户为什么不直接用 ChatGPT? |

## 反模式(工程师高发)

- **把"调好 prompt"当护城河**:prompt 是产品的一部分,但它最容易被逆向、被官方能力覆盖,单靠它扛不住。
- **把"用了 RAG / 建了知识库"自封数据壁垒**:如果数据是公开的,它对所有人开放,**零壁垒**。真壁垒只有专有数据或数据飞轮。
- **跟 ChatGPT 拼"通用、全能"**:那是平台用数据 + 分发筑死的战场,solo 拼通用必死。
- **只做独立网站,不做嵌入**:用完即走 = 零切换成本 = 谁都能替换你。
- **把可靠性当锦上添花**:在垂直专业场景,一次关键处的胡说,可能直接毁掉用户信任,比能力弱更致命。
- **做完产品才想分发**:"东西好自然有人用"是工程师最贵的幻觉,没有分发,再好也烂在硬盘里。
- **声称的壁垒和实际拥有的不一致**:对外吹的护城河,自己心里得清楚是真是假——既是诚实问题,也是长期生死问题。

## 动手:这一章怎么练

1. **拿那把尺审判你的产品**:写下"用户为什么不直接用 ChatGPT 干这件事?"——**答案必须落在某个具体的垂直深度上**(某个 workflow、某份专有数据、某个嵌入位、某种可靠性)。答不出,说明你还没有护城河。
2. **给五类护城河逐一打分**:workflow / 数据 / 嵌入 / UX可靠性 / 分发,你的产品各占了几分?**找出最弱的两类,这是你接下来要补的。**
3. **诚实拷问你的"数据壁垒"**:你那份数据,平台真的拿不到吗?是公开爬来的,还是专有/飞轮?**如果是公开的,别再骗自己它是壁垒。**
4. **设计一条数据飞轮**:用户用你的产品时,会持续产生什么数据?这些数据怎么沉淀回产品、让它对这个用户越来越好用、越来越难离开?

## 真实参照(去读这些)

- **Warren Buffett 的"经济护城河(economic moat)"概念** —— "套壳护城河"这个词的思想源头,理解什么样的壁垒能持久
- **Hamilton Helmer《7 Powers》** —— 系统讲解七种持久竞争优势(规模、网络效应、转换成本、品牌等),这一章的五类护城河可与之对照
- **《Crossing the Chasm》(Geoffrey Moore)** —— 垂直切入、占据细分人群心智的经典打法,对应分发/品牌壁垒
- **观察 Perplexity vs Google** —— 一个套壳产品如何靠 UX + 垂直定位,在搜索这个平台筑死的战场里撕开口子(它做对/做错了什么,自己判断)
- **观察 Cursor / GitHub Copilot 的嵌入打法** —— 把 AI 能力嵌进编辑器、长进开发者工作流的活教材
- 注:文中所有营收 / 用户量 / 装机量 / 时间线等具体数字标「待核」,正式版接地核实;产品决策史实(Llama 开源、Copilot 嵌入编辑器、Threads 导入 IG 关系、Apple 端侧策略等)为公开信息
