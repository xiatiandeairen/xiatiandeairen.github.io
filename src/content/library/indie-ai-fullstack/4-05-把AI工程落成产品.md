> 这章不讲 RAG 原理、不讲 prompt 技巧——那些你已经有专门课程了。这章讲的是**你把 AI 接进去之后,产品能不能活下来**。能不能活,99% 取决于产品化这层:单位成本控没控住、质量会不会悄悄漂移、线上出了问题你能不能发现。

---

## 为什么"把 AI 接进去"不等于"做出 AI 产品"

很多工程师第一次做 AI 产品走的路是这样的:接 API → 产品能跑 → 发布 → 用户进来 → 卡住。

卡在哪?不是技术不对,是产品化没做。有三件事在这个阶段最致命:

**第一,单位成本失控。** LLM 的 API 按 token 计费,用得越多,你亏得越多。如果你没有在第一天就建成本看板,你根本不知道每个活跃用户一天烧多少钱。等发现了,可能已经亏出去一笔。这不是理论上的担心——这是几乎每个第一次做 AI 产品的人都会踩的坑。

**第二,质量会漂移。** 模型不是固定的程序,同一个 prompt 在不同版本的模型上、在不同上下文长度下,输出会悄悄变。你改了一处 prompt,某个边缘场景的质量就崩了;供应商悄悄更新模型,你的格式解析就挂了。没有 eval 回归集,你根本不知道什么时候悄悄坏掉的。

**第三,线上失败是黑盒。** 用户说"你的 AI 不好用",你拿什么定位问题?是 prompt 问题?是上下文太长被截断?是某类输入触发了异常?没有输入输出日志、没有失败率监控,你只能靠猜。

这三件事不是锦上添花,是产品能不能存活的地基。工程能力再强,绕不过去。

## 打法一:第一天就埋成本计量,不要等到"烧完了再说"

这条是最容易被跳过的,因为开发阶段用量小、感觉不痛。但你一上线,用量会突然放大,而你根本不知道每个功能点在烧多少。

**机制:AI 产品的成本结构和传统产品不同。** 传统 SaaS,边际成本接近零(多一个用户多一点带宽,便宜到可以忽略)。AI 产品不一样——每次模型调用都在花真钱,而且成本和用户行为强耦合:某个用户如果粘性特别高,他一个人可能就比月费贵。"用得越多越亏"不是悖论,是 AI 产品定价前没建成本视图的必然结果。

**做法:**

在每一个 LLM 调用点,记录三个数:input tokens、output tokens、使用的模型。不用复杂框架,一张数据库表就够:

```
calls(user_id, ts, model, input_tokens, output_tokens, latency_ms, feature)
```

`feature` 字段告诉你是哪个功能点触发的——"summarize"、"chat"、"rewrite",各自单算。

然后做一个最简单的看板(哪怕是一个 SQL 查询脚本也行):

- 昨天总花费多少美元
- 单个活跃用户平均成本
- 成本最高的 feature 是哪个
- 成本 Top 10 的用户是谁

**案例:OpenAI 自己踩过这个坑。** ChatGPT 早期,某些重度用户的对话长度极长,每次续问都要把全部历史带进去,上下文成本以指数级堆叠。后来 ChatGPT 加了对话折叠和摘要机制,本质上是在用工程手段控制 context window 的 token 消耗——因为不控不行,否则单用户成本会爆。对一个 solo 开发者来说,你没有 OpenAI 的资本池来扛,**你必须比大厂更早建成本意识**。

定价要在看板建好、跑了一两周真实数据之后再定。凭直觉定价的结论几乎都是错的。

## 打法二:用 eval 回归集守住质量,改 prompt 和换模型都要回归

大厂有一套"eval 文化":在上线任何 prompt 改动或模型切换之前,先跑一套测试集,把新旧版本的输出都跑一遍,数字好了才合并。这套做法缩到 solo 层面,仍然是你的质量护城河。

**机制:模型不是确定性程序。** 一个在开发时看起来"挺好"的 prompt,在生产环境里可能只有 70% 的 case 表现稳定,另外 30% 是你没测到的边缘情况。你手工测几个例子是不够的——因为你测的例子正好是 prompt 被设计来应对的例子。真正的问题在"你没想到"的输入上。

**怎么建 eval 回归集:**

第一步,收真实失败案例。上线后,每一个用户报告"效果不对"的 case,都记下来:原始输入是什么、你的期望输出是什么、实际输出是什么。这是最有价值的 eval 素材,因为它是真实的边缘案例。

第二步,建黄金集(golden set)。从这些失败 case + 你自己设计的 case 里,积累 20–50 个有代表性的输入-期望输出对。每条 case 至少要有:输入、期望的关键特征(不一定要完全匹配输出,可以是"包含 X"、"不包含 Y"、"格式是 JSON"之类的断言)。

第三步,写一个能批量跑的脚本。输入是黄金集,输出是"通过 N 个,失败 M 个,失败详情"。每次改 prompt 或换模型前跑一次,改完再跑一次,对比结果再决定是否上线。

**案例:Cursor 的质量管理策略。** Cursor(AI 代码编辑器)在公开访谈中提到「待核」,他们维护了一套 benchmark 来评估模型切换对代码补全质量的影响——这让他们能在供应商(Anthropic/OpenAI)更新模型时快速决策"要不要切"。同类工具 GitHub Copilot 背后的 Microsoft,内部有更成熟的 eval pipeline「待核」。solo 做不到这个规模,但思路完全可以照搬:你至少要有 20 个 case 的黄金集。

**一个常见的误区:**"单测不是够了吗?"单测测的是确定性逻辑(API 路由、数据处理),不测模型的输出质量。你不能 `assert output == "the exact sentence"`,你测的是"输出是否满足某些特征"。这是 eval 和单测在本质上的不同——单测永远替代不了 eval。

## 打法三:线上可观测——记录输入输出、监控失败率

AI 产品的"可观测性"和传统服务不同。传统服务你监控 HTTP 状态码就知道对不对;AI 产品,状态码 200 代表"模型回答了",但不代表"回答对了"。你需要两层监控。

**第一层:基础可观测(必做)**

每一次 LLM 调用,记录:

- 输入(prompt 全文或哈希 + 关键变量)
- 输出(模型原始回复)
- 调用时延
- token 消耗
- 模型版本
- 是否有异常(截断?格式解析失败?超时?)

这些日志存在你能查询的地方。出了问题,你能在五分钟内把某个用户某次的完整调用链拉出来——而不是对着用户反馈猜。

**第二层:质量信号(尽力做)**

用户的行为是最诚实的质量信号。记录:

- 用户有没有触发"重试"或"重新生成"(连续触发 = 上次结果不满意)
- 用户有没有手动编辑 AI 的输出(编辑 = 输出有偏差)
- 功能完成率(AI 辅助填写表单,有多少人最终提交了)

这些行为数据比用户主动反馈更客观——用户很少报 bug,但他们的行为会说话。

**案例:Google 的"速度即功能"信条。** Google 内部多年来的原则是把延迟当一等指标。搜索每慢 N 毫秒,用户流失就升一个档「待核」。这个信条被 Google DeepMind 和 Google AI 的产品团队继承进了 AI 产品——Gemini 在 Google Search 里的调用延迟是被单独监控的「待核」。对你来说,**p99 延迟是一个比平均延迟重要得多的数字**——平均延迟好看,但 p99 高意味着有一批用户每次都在等待,而这批人最可能流失。

**最小化可观测实现:** 不要想着第一天就搭 Datadog 或 OpenTelemetry 的完整栈。先用最简单的方式:把每次调用的 JSON 记一条 log 进数据库。一个 Python + SQLite 就够。有了这层,你才能做后续的分析。等规模大了再换 Grafana 或者 Langfuse。

## 打法四:缓存、降级、模型分层控成本

成本控制不只是"省钱",更是"让你能定出合理的价格、做成可持续的生意"。

**缓存:同一输入别付两次钱**

如果你的产品里有大量结构类似的请求(例如"总结这段文本"、"把这段翻译成英文"),相同输入大概率会重复出现。建一个语义缓存:对 prompt 做哈希(或 embedding 近似匹配),命中缓存的请求直接返回历史输出,不调模型。

哪些场景缓存命中率高?模板化程度高的请求——分类、格式转换、固定 few-shot 的任务。哪些场景不适合?强上下文依赖的对话——每次都是新的。

Anthropic 自己的 API 有 **Prompt Caching** 功能(对长 system prompt 做前缀缓存),能把重复调用里 system prompt 部分的 token 成本降到原来的一个零头「待核具体折扣率」。如果你用 Claude 而且 system prompt 长,第一天就开这个。

**模型分层:用对的锤子打对的钉**

不是所有任务都需要最强模型。一个基本的分层策略:

- **简单分类 / 格式转换 / 关键词提取**:用小模型(GPT-4o mini / Claude Haiku 级别),成本低 10-30 倍
- **复杂推理 / 长文生成 / 代码解释**:用主力模型
- **超高质量 / 低频的关键场景**:用旗舰模型

**案例:字节跳动的多模型策略。** 字节在 AI 产品线(豆包、Coze 等)上明确使用了多模型路由——根据任务类型和用户等级分配不同规格的模型「待核」。这不是字节专属策略,任何规模的 AI 产品都应该从第一天就考虑"这个任务到底需要多贵的模型"。

**降级:失败时给用户一个兜底**

LLM 调用的失败率不是零:超时、rate limit、解析异常都会发生。你需要一个降级策略:

- 超时重试一次(但设合理的超时阈值,不要让用户等 30 秒)
- 解析失败时返回一个合理的 fallback(而不是 500 error)
- 高流量时,低优先级功能可以降级到轻量模型甚至关掉,保证核心功能可用

这些不是高级架构,是基本的工程卫生。solo 开发者很容易忽略,因为在开发时你的调用都很快、很稳定——上了生产就不一样了。

## 打法五:把 AI 产品的护城河建在产品化层,不在模型层

这一条是心态层面的,但和前四条同样重要。

**模型层没有护城河。** 你今天用 GPT-4o 写了一个"最强 AI 写作助手",明天 Claude 出了更好的,后天有人把同一个 prompt 接 Gemini,你的模型优势一夜归零。基础模型的能力进步如此之快,建在模型层上的任何护城河都是沙滩城堡。

**产品化层才能建护城河。** 包括:

- **数据飞轮**:你积累的用户行为数据、eval 数据、失败 case 数据,让你比竞争对手更快发现质量问题、更准调 prompt。
- **成本结构优势**:你把单位成本控到比竞争对手低 30%,你就能定出更低的价格或者更好的 free tier。
- **用户体验的打磨**:延迟、稳定性、错误恢复——这些东西大家都懒得做,做了就是差异化。
- **eval 文化积累**:你有 100 个黄金 case,竞争对手没有,你换模型的时候比他快得多,少踩坑。

**案例:Perplexity vs Google Search AI。** Perplexity 的模型能力比 Google 弱得多,但它在 AI 搜索体验的打磨上(来源引用、格式、响应速度、跟进提问)做了大量产品化工作「待核具体数字」。Google 有更强的模型,但产品化迭代速度早期慢于 Perplexity。这说明**产品化执行力可以在一定窗口期内抗衡模型优势**。

这个窗口会关闭——Google 最终会补上。但对 solo 开发者来说,这个规律的意义是:你的机会不在"比 OpenAI 模型更强",而在"在某个垂直场景里把产品化做得比任何人都扎实"。

## 新手 vs 资深

| | 新手做法 | 资深做法 |
|---|---|---|
| 成本管理 | 上线再看,发现亏了再改定价 | 第一天建 token 计量 + 单用户成本看板,数据驱动定价 |
| 质量保证 | 手工测几个例子,感觉不错就上线 | 维护 eval 回归集,改 prompt / 换模型前必跑 |
| 线上问题定位 | 对着用户反馈猜,复现不了 | 有完整的输入输出日志,五分钟内能还原任何调用 |
| 延迟优化 | 平均延迟看起来 OK 就行 | 盯 p99,识别慢 case 的根因 |
| 模型选择 | 所有场景用最强模型,图方便 | 分层:简单任务小模型,复杂任务主力模型 |
| 护城河思维 | "我用了最新的模型" | "我的 eval 集、数据、产品打磨是竞争对手拿不走的" |

## 反模式(工程师高发)

**过度工程化基础设施,忽视产品化细节。** 花三周搭 Kubernetes + Ray Serve 的推理集群,却没建一张 token 成本表。基础设施的天花板远高于你当前需要,但产品化欠债会第一天就咬你。**够用就上,boring tech first。**

**把模型能力当护城河。** "我用了 GPT-4,竞争对手还在用 3.5",这种优势的半衰期以周计。模型能力的收益应该立刻转化到产品体验上,不能停在"我用了更好的模型"这一步。

**没有 eval 就换模型。** 供应商说新模型更好,你就切过去,结果某个核心场景的格式解析全挂了。没有 eval 回归集,每次模型变更都是赌博。

**线上用"查日志"代替"建可观测"。** 出了问题再去翻日志,发现当时根本没记录输入,或者 JSON 格式不对查不了。可观测性必须在上线前建好,出了问题那一刻建为时已晚。

**完美主义 prompt 工程,忽视失败率监控。** 花大量时间把 prompt 调到"在测试集上"表现完美,却不知道线上失败率是 15%——因为生产环境的输入和你测试的完全不同。

## 动手:这一章怎么练

1. **建成本计量表。** 在你的项目里加一个 `llm_calls` 表,记录每次调用的 user_id、model、input_tokens、output_tokens、feature。跑一周真实数据,算出单活跃用户每日成本。

2. **建 20 条 eval 黄金集。** 找出你产品最核心的 3 个 AI 功能,每个写 5-7 个有代表性的输入 + 断言(不要求完全匹配输出,用"包含 / 不包含 / 格式符合"之类的断言)。写一个脚本批量跑,输出通过率。

3. **加输入输出日志。** 在所有 LLM 调用处加 logging,记录 prompt 全文和模型输出。选一个你能查询的地方存起来(SQLite、Supabase 都行)。模拟一个"用户反馈效果不对"的场景,看你能不能在五分钟内找到那次调用的完整记录。

4. **做一次模型分层实验。** 把你的一个简单功能(分类 / 格式转换)从主力模型切到轻量模型,用 eval 集验证质量是否达标,对比成本差异。如果质量没掉、成本降了 80%,那就永久切。

## 自测清单

- [ ] 我知道我的产品**单个活跃用户的每日 token 成本**是多少
- [ ] 我有一个 eval 回归集,改 prompt 或换模型前我会跑它
- [ ] 我能在五分钟内**还原任意一次线上 LLM 调用**的完整输入输出
- [ ] 我监控了**p99 延迟**,不只是平均延迟
- [ ] 我的不同功能用了**不同规格的模型**,而不是全部用最贵的
- [ ] 我有**降级策略**:模型调用失败时用户看到的不是 500 error
