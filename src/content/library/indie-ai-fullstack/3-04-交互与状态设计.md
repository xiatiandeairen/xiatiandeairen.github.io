> 用户不会因为"功能太强"放弃你的产品,但会因为"不知道发生了什么"默默关掉它。这章讲一件事:**任何时刻,用户都不该处于"懵"或"卡住"的状态。**

---

## 为什么状态设计是被忽视的杀手

工程师做产品时,天然关注"功能能不能跑通"——接口返回 200、数据写进库、逻辑正确。但用户感受到的不是逻辑,是**状态**:这个按钮按下去有没有反应?等了五秒什么都没有,是在加载还是卡死了?提交失败了我该怎么办?

**这三个问题对应三种状态:加载中、错误、空数据。** 加上"有数据时"的正常态,任何一个界面都有这四种状态要设计。大多数工程师只设计了正常态,其余三个随便糊——于是用户体验就在这三个角落碎掉了。

状态设计的底层机制很简单:**人脑的焦虑来自不确定性。** 知道"要等一会儿"不会让人烦,"不知道要等多久、等的是什么"才让人焦虑。知道"失败了因为 X"不让人放弃,"500 Internal Server Error"让人不知所措。**把不确定性消除掉,体感就顺滑了。**

下面五条打法,每条一个大厂案例,直接可用。

---

## 打法一:空态要引导,不是白屏

空态(empty state)是最容易被忽略、伤害最大的状态。用户第一次打开你的产品、还没有数据、刚清空了内容——如果这时候是一片白,他会不知道从哪里开始,然后离开。

**空态的正确做法是把它变成"邀请":告诉用户下一步是什么。**

一个好的空态应该包含三个元素:
1. **一句说明**:这里本来会有什么(让用户理解这个区域的用途);
2. **一个行动**:直接给一个 CTA 按钮,把用户引导到第一步;
3. **可选的插图**:让空白不那么让人感到"坏掉了"。

**案例深拆:Notion 的空白页**

Notion 的空白页不是真的空——它的 placeholder 文字是 "Press Enter to continue with an AI block, or '/' for commands."。这句话做了两件事:一是消除焦虑("这里是可以输入的,不是坏掉了"),二是引导下一步("按这个键开始")。对于刚进来的新用户,这一行文字比任何 onboarding 弹窗都有效,因为它在用户最需要它的时候出现——准备开始写的那一刻。

对比一下差做法:很多笔记 App 新建页面后给你一个真空白,光标默默闪烁,用户盯着它不知道该干嘛。

再往深看一层:Notion 为什么选择在 placeholder 里提 AI 功能?因为这是他们的核心卖点,空态就是"第一次深度体验核心价值"的最佳入口。**空态不是设计的尾声,是 aha moment 的起点。**

**给独立开发者的具体动作**:打开你 App 的每个列表、每个主界面,人为清空数据,看看空态长什么样。如果是白屏或者系统默认的"暂无数据",立刻改掉。空态文案可以遵循这个公式:"还没有 [内容名称],[动词] [CTA] 来开始。"

## 打法二:错误态要可恢复,不是甩 error code

错误发生的时候,用户最需要两件事:**知道发生了什么,知道接下来该怎么办。** 一个好的错误提示 = 原因 + 行动。

大多数工程师写错误提示是从程序视角出发的:"Request failed with status code 500"、"NetworkError"、"ERR_CONNECTION_REFUSED"——这些对用户毫无意义,只会让他们觉得产品坏掉了、不信任、走掉。

**错误信息的三个层次,从差到好:**
- 差:技术码(500、ENOENT、null pointer)
- 中:原因("网络连接失败")
- 好:原因 + 出路("网络有点问题,请检查一下连接,然后点这里重试")

**案例深拆:iOS 的错误处理哲学**

Apple HIG(Human Interface Guidelines)对错误提示有明确要求:避免技术术语、用用户能理解的语言描述、告诉用户可以做什么。iOS 自带的 Alert 设计也体现了这一点——即使是系统级错误,也会给出"OK"、"重试"、"取消"这样的明确出口,而不是让用户面对一个没有操作入口的错误信息。

更值得学习的是 iOS 的网络错误处理:App Store 在网络差的时候不是甩一个错误,而是说"您没有连接到互联网。请检查您的网络连接。"——这句话用户能听懂,而且隐含着"我只要去连网就能解决"的路径。

**深一层看**:Apple 为什么这么设计?因为他们把用户当普通人,不当开发者。普通人看到"ERR_SSL_PROTOCOL_ERROR"只会恐慌,看到"这个网站的证书有问题,无法安全连接"才知道是网站的问题不是自己的问题。**恐慌 = 流失。** 好的错误信息直接减少流失。

**给独立开发者的具体动作**:梳理你产品里所有 try-catch 和 error 分支,检查每一处显示给用户的错误信息。用这两个问题测试:"一个不懂技术的人能看懂吗?他知道下一步该干什么吗?" 如果答案是否,改掉。最常见的可恢复错误(网络超时、上传失败、登录过期)都应该有专门设计的提示 + 重试 CTA。

## 打法三:加载用骨架屏和进度反馈,不是空白转圈

加载状态是"存在感最强的等待"——用户知道有东西在发生,但不确定要等多久、等什么。如果只是一个转圈图标,他的焦虑会随时间线性增长;如果有结构化的占位内容,他知道"这里会有内容",心里就稳了。

**骨架屏(Skeleton Screen)的底层逻辑**:与其让用户盯着空白等内容,不如先把内容的"轮廓"渲染出来。用户的眼睛有了落点,大脑预期被填满,感知到的等待时间就会缩短——哪怕实际加载时间没变。

**案例深拆:Facebook 推广骨架屏的背后**

骨架屏由 Luke Wroblewski(后来加入 Google)「待核具体时间线」提出并推广,Facebook 是最早大规模使用它的产品之一,后来写了内部报告讨论它降低用户等待焦虑的效果。

Facebook 为什么要搞骨架屏?因为 News Feed 的图文内容加载顺序不一致——有时候文字先来、图片后来,有时候顺序相反,导致页面"跳动"(layout shift),体验极差。骨架屏先把整体布局占位,内容来了就填进去,消除了跳动感,也消除了"这一块是空的是因为坏掉了还是在加载"的疑惑。

**具体怎么做骨架屏**:用灰色色块(通常 `#E0E0E0` 或 `#F0F0F0`,配合 shimmer 动画)替代文字和图片的位置。关键是**形状要和真实内容接近**:如果真实内容是"头像(圆形)+ 标题(一行长条)+ 两行描述文字",骨架屏就是同样布局的灰色圆和灰色条,而不是随便放几个矩形。用户看骨架屏时已经在预读内容结构,他有预期了,内容来了就是"到了"而不是"突然出现"。

**进度条和确定性**:如果是可以量化进度的操作(文件上传、视频处理、批量导入),给一个进度条比骨架屏更好——进度条传递的信息更多:"我知道完成了多少,大概还要多久"。哪怕进度估算不精确,有进度反馈也比没有强得多。

**给独立开发者的具体动作**:把你 App 里所有网络请求和耗时操作过一遍,每一处问:"这里用户会看到什么?" 如果答案是"空白或转圈超过半秒",考虑加骨架屏;如果是可量化的操作,加进度条。对于首屏(用户打开 App 看到的第一个内容),骨架屏是必须的——首屏是第一印象,不能让它"白一下"。

## 打法四:微交互即时反馈,让操作有质感

用户按下按钮、提交表单、切换开关——如果没有任何即时反馈,他不知道系统有没有接收到操作,只能猜。猜就是焦虑的来源。

**即时反馈的黄金标准:100ms 以内。** 心理学研究表明人感知到"即时"的阈值大约在 100ms「待核具体数字来源」。超过 100ms 才有反馈,用户就会感到"迟钝";超过 1 秒就会觉得"卡了"。

微交互不只是视觉的,还有触觉(Haptics)和声音。在移动端,这三个通道共同构成"这个操作完成了"的感知。

**案例深拆:iOS 转场动画的空间方位感**

iOS 的转场动画不是装饰,是**导航的信息载体**。当你点击一个列表项进入详情页,详情页从右滑入;当你按返回,它从右滑出——这个方向一致性建立了用户对"我在 App 空间里的位置"的认知模型:右边是深处,左边是后退。

为什么这很重要?因为移动端没有物理地图,用户完全靠动画来理解自己在哪、从哪来、能去哪。如果转场方向不一致(有时候从下滑入,有时候从右,随机的),用户就会迷失方位,每次切换都要重新定位——认知负荷上升,体感就变差了。

Apple 在 iOS 7 把这套动画体系化之后,制定了 HIG 里的转场规范,要求第三方 App 遵循同样的方向逻辑。这不是审美强迫,是**让用户在全局 App 生态里保持一套统一的空间认知**。

**微交互的另一个维度:状态变化要有锚点。** 比如一个"收藏"按钮,点击前是空心星星,点击后应该变实心——这个颜色和形状的变化是"我的操作成功了"的视觉证明。如果只有网络请求、没有即时 UI 变化,用户会再点一次(以为第一次没成功),触发双重操作。

**给独立开发者的具体动作**:检查所有"用户操作后会改变状态"的控件(按钮、开关、复选框、收藏/点赞),确保点击后立即有视觉变化(不等网络返回)。然后针对移动端,在成功操作(如"保存成功"、"任务完成")加上轻微的 Haptic 反馈——iOS 用 `UIImpactFeedbackGenerator`,Android 用 `Vibrator` API。这个细节成本极低,体感提升显著。

## 打法五:乐观更新,让操作显得瞬时

乐观更新(Optimistic UI)的思路是:**不等服务器确认,先在 UI 上更新状态;如果请求失败,再回滚。** 对于成功率高的操作(点赞、删除、勾选完成),这让体感从"等待→结果"变成"瞬时完成"。

**为什么能这么做?** 因为大多数常规操作的服务器成功率接近 100%「待核具体比例」。把 99% 的情况优化到瞬时体验,1% 的失败回滚通知一下用户,总账是赚的。

**案例深拆:抖音的点赞和手势交互**

抖音的点赞是乐观更新的典型:双击屏幕,立刻出现一个飞出的红心动画,点赞数立刻跳动——这一切在你手指离开屏幕的瞬间就发生了,不等网络。如果请求失败,点赞数静默回滚(大多数用户根本不会注意到这个 1% 的回滚)。

这个设计的深层逻辑:点赞是情绪性操作——你看到一个笑死了的视频,你的情绪响应在那一刻,你想立刻"表达"。如果点赞要等 200ms 才有反馈,情绪和反馈之间的延迟会让这个动作丧失满足感。**乐观更新让操作和情绪同步,这就是它的价值。**

再看抖音的手势设计:上滑切换视频是全面屏的标配,但抖音做得特别丝滑,核心在于**手势预测**(一旦检测到上滑意图,就开始预加载下一条)和**无缝过渡动画**。用户的感受是"视频被我的手指拖着走",而不是"我请求了下一条"——主动感极强,这是乐观更新 + 预加载 + 动画配合的整体效果。

**下拉刷新的心理契约**:下拉刷新(Pull to Refresh)是 Loren Brichter 在 Tweetie(后被 Twitter 收购「待核」)里发明的交互模式。它为什么好用?因为它给用户**主动控制感**——不是系统自动刷新,是我拉了一下,然后有了新内容。这个控制感带来的满足感,比自动刷新更强。下拉动作也有天然的"期待→释放"的节奏感,释放那一刻等于拉了一把老虎机——正好触发多变奖励的愉悦。**这就是为什么下拉刷新被做成行业标准,靠的不是效率,是心理机制。**

**给独立开发者的具体动作**:整理你产品里所有"用户发起的写操作"(点赞、删除、完成、切换),评估每个操作的预期成功率。成功率 > 95% 的操作,都可以考虑乐观更新:先更新 UI,后发请求,失败再提示。实现成本:前端维护一个乐观状态,请求失败时回滚到原始状态,同时给用户一个 toast 提示("操作失败,已撤销")。

## 独立开发者的三态检查习惯

上面五条打法说的每一处,根源都是同一件事:**每个界面都要设计三种状态——空态、错误态、加载态。**

把这个做成一个固定的 checklist,每做完一个界面就过一遍:

**空态**:
- 这里没有数据时,用户看到什么?
- 有没有说明这里是干什么的?
- 有没有引导用户做第一步的 CTA?

**错误态**:
- 请求失败/操作失败时,用户看到什么?
- 错误信息是人话还是技术码?
- 有没有告诉用户下一步可以怎么办?有没有重试入口?

**加载态**:
- 有网络请求的地方,加载时用户看到什么?
- 是骨架屏还是转圈?转圈超过半秒要换骨架屏。
- 可量化的耗时操作,有没有进度反馈?

这三个问题,把它贴在你的显示器旁边。每次交付一个界面,先过这三问,再给用户看。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| 开发思路 | 跑通正常流程就算完成 | 正常态 + 空态 + 错误态 + 加载态全部设计 |
| 空态处理 | 不处理或写"暂无数据" | 引导语 + CTA,把空态变成 aha 入口 |
| 错误信息 | 直接显示技术错误码或 catch 到的 message | 人话描述原因 + 给出行动出口 |
| 加载反馈 | 全局 loading spinner 到处挂 | 骨架屏首屏、进度条耗时操作、spinner 只用于快速请求 |
| 操作反馈 | 等后端返回再更新 UI | 乐观更新先改 UI,失败再回滚 |
| 动画转场 | 系统默认或不加动画 | 方向一致、有空间感,传递导航位置信息 |

## 反模式(工程师高发)

**只设计正常态,其他三态随缘。** 这是最常见的毛病。正常态是你写测试用例时的 happy path,但用户进来先遇到的往往是空态,失败时遇到的是错误态。这两个状态不设计,体验就在用户最脆弱的时刻崩掉。

**用 toast 代替空态引导。** "没有数据" toast 一闪而过,用户没看清就消失了,还是不知道下一步。空态需要的是持久展示的引导,不是弹一下就走的通知。

**错误信息只对自己有意义。** "ERR_NETWORK_CHANGED"、"NullPointerException"、"HTTP 422 Unprocessable Entity"——这些对用户完全不透明。工程师习惯把自己看到的日志直接扔给用户,根本没想过用户眼里的这段话是什么意思。

**加载状态一律全屏遮罩。** 整个页面被一个半透明蒙层加转圈盖住,用户什么都看不见、做不了——既打断流程,又放大等待焦虑。大多数加载只需要局部 skeleton,不需要全屏封锁。

**乐观更新做一半。** 乐观更新了 UI,但失败时没有回滚和告知——用户以为成功了,实际上没有,数据不一致。要做乐观更新,就必须实现回滚路径。

## 动手:这章怎么练

1. **三态审计**:打开你现有产品的每一个界面,依次模拟空数据、断网、慢网速,用截图记录空态、错误态、加载态各长什么样。标出不合格的,排优先级改。

2. **改三个错误提示**:在你的代码里搜所有 `catch`、`error`、`onError` 的处理分支,找到三处直接给用户看技术信息的地方,改成人话 + 行动建议。

3. **加一处乐观更新**:找一个"成功率高、用户高频操作"的写操作(比如点赞、打标签、勾选完成),把它改成乐观更新:先改本地状态,再发请求。对比改前改后的操作感。

4. **骨架屏替换转圈**:找你 App 里首屏或最重要的列表,把 loading spinner 换成骨架屏。用手机真机测试,在慢网下体感对比改前改后。

## 自测清单

- [ ] 我的每个界面都有明确的**空态设计**(不是白屏,不是"暂无数据"),且有引导下一步的 CTA
- [ ] 我产品的**错误信息**是用户能看懂的人话,且告诉了用户下一步能做什么
- [ ] 首屏和重要列表使用了**骨架屏**,不是 spinner 对着白屏
- [ ] 耗时 > 2 秒的操作有**进度反馈**(进度条或百分比),不是干等
- [ ] 高频写操作(点赞、完成、删除)已实现**乐观更新**,且失败时有回滚和告知
- [ ] 我 App 的**转场动画方向一致**,用户能建立空间方位感
- [ ] 我做了**三态 checklist**,并把它用于每个新界面的交付标准

## 真实参照

- **Luke Wroblewski《Web Form Design》** —— 表单设计经典,含大量"减少错误 / 清晰反馈"的具体做法
- **Apple Human Interface Guidelines(HIG)** —— iOS 的状态设计规范、转场动画规范、Haptics 使用规范,全部免费在线(developer.apple.com)
- **Google Material Design — States** —— Material 对各种状态(enabled/disabled/hover/error/loading)的系统化规范,附代码示例
- **「骨架屏」Luke Wroblewski 原始文章** —— "Mobile Design Details: Avoid The Spinner",2013 年发表「待核具体出处和年份」,骨架屏的理念来源
- **Loren Brichter** —— 下拉刷新的发明者,Tweetie 作者;其设计思路可通过多篇访谈了解
- **《The Design of Everyday Things》(Don Norman)** —— "反馈"和"可见性"的理论底座,状态设计背后的认知科学根源
