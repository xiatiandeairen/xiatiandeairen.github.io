> **这章解决一个工程师的宿命难题**:你有能力做很多,但第一版做多了,反而死得更快。砍范围不是妥协,是独立开发最重要的竞争力——这章给你一套能拿来就用的决策法。

---

## 为什么范围是独立开发的头号杀手

先讲机制,再讲打法。不理解"为什么砍",砍的时候总会心虚,最后越砍越加回来。

**范围不是线性成本,是指数成本。**

功能数量乘以它们之间的交互复杂度。一个功能可以单独存在;两个功能不只是"两倍成本",它们还会互相影响——用户流程、边界状态、错误处理、测试覆盖都要考虑组合。五个功能在一起,潜在的交互路径可以轻松超过二十条。每加一个功能,你不只是多了"一件要做的事",而是给整个系统加了一层复杂度网。

**范围的真实代价,在写代码之前就已经开始累计:**
- 认知负荷:你同时想着五件事,每件都比只想一件想得浅
- 决策摩擦:功能之间要排优先级、要协调 UX,每一次都要开会(即使是和自己开)
- 延期:工程师低估"最后 20%"是铁律,功能越多,低估累积越严重
- Bug 面积:上线的功能越多,出事的概率越高,出了事排查越难
- 维护债:每个发出去的功能,日后都要兼顾向后兼容

独立开发者是一支三人队(开发、产品、运营全自己扛),团队大了能用人力对冲复杂度,你对冲不了。**每多一分范围,就是在把你本来就不够用的注意力再稀释一次。**

最反直觉的地方来了:**约束反而催生好设计。** 这不是安慰话。当你不允许自己用第三个按钮时,你会被逼着把那个按钮背后的东西想清楚,最后做出更聚焦的判断。当你资源充裕时,很容易用"加个选项"来回避真正的产品决策。强制约束让你没有退路,只能对着核心问题死磕。

**工程师在这里有一个特有的危险本能:**过度工程(over-engineering)。你看到一个功能,大脑会自动运行"如果有一天用户要 X 怎么办、这里留个扩展点、配置化会更灵活……"这些直觉在大型系统里是美德,在第一版 MVP 里是毒药。你在为一个还没有几个用户的产品构建"可扩展架构"——你在花真实的时间对抗一个想象中的未来问题。

## 打法一:"不做它,产品会死吗?"——这是唯一的筛选题

每一个想加的功能,只问这一句话:**如果 v1 没有它,产品是否无法完成核心 job?**

不是"用户会不会喜欢",不是"竞品有没有",不是"加了会不会更好",就是这一句——会死吗?

答案只有两种:

- **会死**:这是核心功能,一定在 v1
- **不会死**:放到 v2,写到 backlog,现在不碰

"会不会更好用"是另一个问题。几乎任何东西都能"让产品更好用一点",但这不是 v1 的筛选标准。v1 的任务是用最小的成本验证一件事:这个核心 job,你的核心解法,用户买账吗?不需要更好用,需要能用、能验证。

**大厂案例:初代 iPhone 做了什么减法**

2007 年 1 月,Steve Jobs 发布第一代 iPhone。按当时的技术条件和用户期待,它被批评缺失了很多"应该有"的功能:

- 没有 App Store(第三方应用要等到 2008 年的 iOS 2.0 才上线)
- 没有复制粘贴(这个功能要到 2009 年 iPhone OS 3.0 才加入)
- 没有 3G(只有 EDGE 网络,速度慢很多)
- 没有 MMS(彩信)
- 没有企业级邮件支持

这些都是当时的商务手机(黑莓、Windows Mobile)标配的功能。发布后媒体和业内批评声很大,有人说"这根本不够用"。

Jobs 的逻辑是什么?他要验证一件事:**触摸屏 + 整合 iPod + 互联网,这三件事加在一起,能不能成为一个人们愿意花钱买的产品?** 这是他要回答的根本问题。复制粘贴?不影响回答这个问题。App Store?先看人们愿不愿意为这台设备付钱,再谈生态。3G?当时 3G 耗电严重,会破坏使用体验——加 3G 反而干扰了验证核心假设。

结果:初代 iPhone 卖出「待核」百万台,验证了这个根本假设。然后 App Store、复制粘贴、3G 才一个一个补上。

**每一个"缺失的功能",都是一个主动的决策:我现在不需要它来回答那个最核心的问题。** 这不是资源不够,是优先级判断。

对独立开发者来说,这个问题更简单:你要回答的核心问题是什么?问自己那个问题需要哪几个功能?剩下的全砍。

## 打法二:第一版允许手动——Concierge 优先于自动化

这是工程师最难接受的打法,但也是 ROI 最高的:

**如果一个功能用人工操作可以替代,v1 就用人工操作。**

不要过早自动化。

Concierge MVP 的逻辑是:你在验证用户是否需要这个价值,而不是在验证"自动化这个价值"是否可行。这是两个问题,而且必须先回答第一个。如果用户根本不需要这个价值,你的自动化就是浪费。如果用户需要这个价值,你先用人工提供,等需求验证了、量上来了,再用工程能力替换人工。

这不只是初创公司的穷人打法——它有一个更深的好处:**你在亲手完成每一单时,会看见自动化永远看不见的用户行为和痛点细节。**

**大厂案例:WhatsApp 的极简后台**

WhatsApp 从 2009 年创立到被 Facebook 以「待核」亿美元收购时(2014 年),只有五十多名员工「待核」,却服务着数亿用户「待核」。它的极简不只是产品层的——整个技术和运营都刻意保持最小。

更根本的是它的产品策略:**WhatsApp 多年拒绝在核心功能之外加任何东西**。不加广告、不加游戏、不加公众号、不加"发现"功能、不加积分系统。创始人 Jan Koum 的背景(他小时候家里没钱装电话线,靠邻居走很远才能打电话)让他对通信的核心价值有非常清醒的认识:就是"让我能联系到那个人"。

它的产品功能列表在相当长的时间里就是:发消息、发图片、发语音。群组。已读回执(那个蓝色双勾)。这些都是**不做会死**的东西——没有它们,核心 job"让我能联系到那个人"就无法完成。

"可以更丰富"这件事他们做到了:后来加的语音通话、视频通话、Status——但每次加功能都是在原来已经跑通几亿用户之后。

WhatsApp 用极简赢下了全球通讯,不是因为没有能力做更多,是因为有纪律不做更多。

对独立开发者的实操意义:**你的"后台"可以是一个 Notion 表格 + 你手动发邮件。** 用户付钱买到了价值,不在乎你是怎么送达的。等你手动处理了二十单,你才知道哪个环节值得自动化。

## 打法三:给 MVP 设硬边界,不是软目标

MVP 失控的最常见方式不是一次性决定加五个功能,而是"这一个功能就做一天"、"这个需要不到两小时"、"这个不加显得不完整"——一条一条溜进来,加到最后发现已经不是 MVP 了。

这需要的不是意志力,而是**硬边界**:在动手之前明确写下"v1 包含且仅包含以下功能",列表封口,任何后续想法统一进 v2 backlog,v1 不讨论。

**大厂案例:Instagram 砍掉 Burbn**

2009 年,Kevin Systrom 和 Mike Krieger 做了一个叫 Burbn 的 App。Burbn 是一个地理位置签到应用,功能相当多:签到、计划、打卡、照片、积分、好友动态——基本上是当时 Foursquare 的一个复合体加上照片功能。

用户在用,但没有粘性。两人仔细研究了用户行为数据,发现一个规律:**他们真正频繁使用的功能只有一个——上传照片,加滤镜,分享出去。** 签到、积分、计划这些主体功能几乎没人在用。

他们做了一个极其痛苦的决定:把 Burbn 几乎完全推倒。新产品只保留三件事:
1. 拍照 / 上传照片
2. 加滤镜(让烂照片变好看)
3. 分享到 Facebook / Twitter / 其他平台

新产品叫 Instagram。2010 年 10 月上线,第一天下载量「待核」万次。两年后 Facebook 以 10 亿美元「待核」收购。

砍掉 Burbn 之所以困难,是因为那些功能是真实存在的用户场景,是有一定用户量的,是他们用时间和心血做出来的。**放弃已经存在的功能,比从来没做过更难。** 但正是这次砍范围,才让 Instagram 拿到了清晰的产品身份:这是一个拍照 + 滤镜 + 分享的工具,不是别的。

硬边界的实操建议:用一张纸或一个便签,写下"v1 只做这三件事",贴在你工作的地方。每次有新想法就问:"这是 v1 的三件事之一吗?"不是就写进 v2 文档。不讨论,不商量,直接进 backlog。

## 打法四:写死配置,不要过早泛化

工程师最爱提前泛化:用配置文件替代硬编码、做成可插拔、做成多租户……第一版就把系统设计成"未来支持 100 种情况"。

第一版用户通常只有一种情况——你的第一个用户的情况。

**写死它。**

写死的意思不是写烂代码,是**不要为不存在的需求写可扩展接口**。今天硬编码的东西,当真实需求来了你才知道要扩展的方向。提前泛化的结果往往是:扩展了错误的维度,真实需求来了反而改起来更麻烦。

这里的打法延伸自 Concierge 精神:**先用最简单的实现验证需求,有了需求再做工程化**。

**大厂案例和反例:Google 搜索的硬聚焦**

Google 创立初期(1998-2000 年),门户网站(Yahoo、MSN、Excite)的标准形态是什么?首页必须有新闻、天气、邮件、社区、分类目录、股票行情——"什么都有"被认为是门户的核心竞争力,因为用户每天都需要这些。

Google 首页在当时看起来几乎是"不完整"的:一个搜索框,两个按钮,就完了。Larry Page 和 Sergey Brin 的配置是写死的:这东西只做一件事,搜索。不做门户,不做新闻,不做邮件(Gmail 是后来的事,而且是作为独立产品,而不是首页塞进去)。

对用户的影响:你打开 Google,你知道它能做什么,你知道怎么用——零学习成本。打开雅虎,你要先找到你想用的那个功能在哪。**范围的克制直接转化成了使用体验的清晰度。**

写死配置的实操意义:你的第一版支持"一种定价模式"就行,不要做成可配置的多套定价;你的第一版目标用户是"Rust 工程师",不要做成"可以配置成任何语言的工程师"——先服务好 Rust 工程师,看看用户反馈再决定要不要扩。

## 独立开发者的"砍范围"决策法

把上面四条打法总结成一个可操作的步骤:

**第一步:写出你的核心 job**

用一句话写下:用户在什么情境下,用这个产品完成什么任务?这一句是之后一切决策的锚。

**第二步:列出你想做的所有功能**

不加筛选,把脑子里的都写出来。一般来说这时候会有 20-40 条。

**第三步:对每一条问"会死吗"**

逐条问:如果 v1 没有它,用户能不能完成第一步那个核心 job?不能完成 = 留下;能完成 = 进 v2 backlog。

这一轮筛完,一般会剩下 3-6 条。

**第四步:对剩下的每条问"可以手动替代吗"**

如果可以用 Notion / 邮件 / 人工操作替代,就用替代方案,不写代码。这再砍掉一轮。

**第五步:写死 v1 的边界**

剩下的东西明确写下来,告诉自己"这是 v1 的全部"。文字化、签名、贴出来。任何新想法自动进 v2 backlog,不讨论。

**第六步:v1 上线后看数据再决定 v2**

v2 backlog 里有什么,上线之后看用户实际用了什么、在哪里卡住,再从 backlog 里挑。不是按"当时觉得重要"挑,是按用户行为数据挑。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| v1 功能量 | 尽量多做,免得用户觉得不完整 | 做够回答核心问题那几条,其余全砍 |
| 筛选标准 | "这个用户可能需要" | "没有它产品会死吗" |
| 后台实现 | 先把系统设计好,然后做功能 | 先 Concierge 手动跑通,验证了再工程化 |
| 代码架构 | 提前泛化,为未来扩展留口子 | 写死当下需求,真实需求来了再重构 |
| 范围管理 | 软目标,随时可以"这个也加一下" | 硬边界,v1 列表封口,新想法进 backlog |
| 对"加功能"的态度 | 加功能 = 进步 | 加功能 = 成本,每条要过"会死吗"的门 |

## 反模式(工程师高发)

**"这个功能就两小时"**:两小时的功能不只是两小时。它有设计决策、边界状态、测试、文档、维护、和其他功能的交互。加起来可能是两天。而且这句话你已经说了十次了。

**"竞品有,我们没有显得不专业"**:竞品有是因为竞品有团队、有资源、有时间。你在比赛的是不同维度:你能不能比竞品更快验证核心价值、更快迭代。缺竞品的某个功能不是劣势,是专注的代价——只要你的核心 job 做得比竞品更好。

**"先做完整再发,发半成品丢人"**:你认为的"不完整",用户眼里是"够用"。用户不会看你的代码,他只看这东西能不能帮他完成那个 job。发半成品不丢人,发迟了更危险——你错过了真实反馈的时间窗口。

**"这个功能可扩展性很重要"**:可扩展的维度有无数种,你在扩展哪个维度,取决于用户在哪个维度有需求。在有需求之前先扩展,大概率扩展了错误的方向,等真实需求来了反而是负资产。

**"v1 就先这么凑合,v2 再仔细设计"**:这句话是个陷阱。v2 的前提是 v1 得到了真实用户反馈。如果 v1 因为功能太多、发布太晚,还没发出去就半死不活,v2 根本不会来。**v1 要足够精简,才能足够快,才能有 v2。**

## 动手:这一章怎么练

1. **做一次暴力砍范围**:把你现在手头最近的项目(或最近一个想法),列出你计划做的所有功能。然后对每一条问"没有它产品会死吗",把"不会死"的全挪进"v2 backlog"文档,看看还剩什么。大多数人这一步能砍掉 60% 以上。

2. **找一个 Concierge 替代方案**:从剩下的功能里,找出一个"可以先用手动替代"的功能,写下手动替代的具体操作方式(比如"用 Notion 表格记录 + 每天手动发邮件")。想清楚手动到什么程度,你才能上线 v1。

3. **给 v1 写一张封口清单**:把 v1 里会做的功能明确写成列表,在最后加一行"以上是 v1 的全部,新想法写到 v2-backlog.md,不在 v1 讨论"。把这张清单打出来或截图,放到你能经常看到的地方。

4. **研究一个你喜欢的产品的第一版**:找它最早期的截图或发布记录(Product Hunt、早期博客、Wayback Machine),看看它发布时有什么、没有什么。跟现在的版本对比——哪些功能是上线后才加的?这个功能发布到它上线中间经历了多长时间?

## 自测清单

- [ ] 我能用一句话说清楚 v1 要验证的核心假设
- [ ] 我列出了所有想做的功能,并对每条问过"不做会死吗"
- [ ] 我的 v1 功能列表是封口的,有明确的"不包含"边界
- [ ] 我识别出了至少一个"可以先 Concierge 手动替代"的功能
- [ ] 我没有为"不存在的需求"做可扩展接口或泛化设计
- [ ] v2 backlog 已经建立,新想法有地方放、不会挤进 v1

## 真实参照(去看这些)

- **Eric Ries《The Lean Startup》** —— MVP 概念的系统讲法,Concierge MVP 和 Wizard of Oz MVP 都来自这里
- **Paul Graham《Do Things That Don't Scale》** —— Y Combinator 创始人,直接讲"先手动、别过早自动化"的原始文章,短文,值得每年读一次
- **Marty Cagan《Inspired》** —— 产品经理视角的优先级决策,engineering-driven 和 outcome-driven 的区别讲得很清楚
- **Kevin Systrom 关于 Instagram 早期的访谈** —— 他讲过砍掉 Burbn 的决策过程,可以搜"Kevin Systrom Burbn Instagram pivot"
- **初代 iPhone 发布会(2007 年 Jobs Keynote)** —— 重看一遍,注意他没有提哪些功能——那些都是被刻意放弃的,而不是"还没做到"
