> **这章解决一个高频痛点**:独立开发者做到第三个页面,发现按钮颜色和第一个页面对不上,边距大小全靠感觉,改一次颜色要翻十个文件——产品看起来"不专业"的根子不在审美,在于没有系统。这章教你用程序员最熟悉的方式建一套轻量设计系统,一个人也能把界面做得专业一致。

---

## 为什么"系统"比"审美"更重要

很多工程师盯着自己的界面觉得"哪里不对",然后去找好看的 UI 参考,调颜色、换字体,改一通之后还是感觉差点意思。**问题通常不是颜色选得不好,是同一种蓝出现了三个不一样的值,同样的间距在不同页面用了 12px 和 14px 和 16px。** 不一致本身就会让界面显得"廉价"——用户感觉不到什么叫不一致,但大脑会觉得"这东西不可靠"。

设计系统本质上是**把设计决策写成变量,再到处复用这个变量**。这和写代码把魔法数字提取成常量是一回事。工程师天生擅长这个:你不会在十个地方写 `#3B82F6`,你会写 `COLOR_PRIMARY`。把这个思维搬到样式上,就是设计系统的起点。

**建设计系统还有另一个好处:提速**。有了组件库之后,做新页面是在拼积木,不是每次从零画。独立开发者的时间是最稀缺的,能复用的都该复用。

## 打法一:从 Design Tokens 起步,不要从组件起步

**Design tokens 是设计系统的地基。** 它把颜色、间距、圆角、字号、阴影这些基础决策抽象成有名字的变量,整套界面里只有这些变量,没有散落各处的魔法数字。

**为什么 tokens 先于组件?** 因为组件依赖 tokens。如果你先做组件,每个组件里都硬编码 `#3B82F6`,之后要改品牌色,就要改几十处。先定 token,以后改一个地方,所有组件跟着变。

**典型的 token 分层**:

```css
/* ---- 原始层(Primitive tokens):只是颜色值,没有含义) ---- */
--blue-500: #3B82F6;
--blue-600: #2563EB;
--gray-50:  #F9FAFB;
--gray-900: #111827;
--radius-sm: 4px;
--radius-md: 8px;
--radius-lg: 16px;

/* ---- 语义层(Semantic tokens):表达用途) ---- */
--color-primary:       var(--blue-500);
--color-primary-hover: var(--blue-600);
--color-surface:       var(--gray-50);
--color-text:          var(--gray-900);
--radius-button:       var(--radius-md);
--radius-card:         var(--radius-lg);

/* ---- 间距 scale:始终用 4 的倍数 ---- */
--space-1: 4px;
--space-2: 8px;
--space-3: 12px;
--space-4: 16px;
--space-6: 24px;
--space-8: 32px;
--space-12: 48px;
--space-16: 64px;
```

**为什么是 4px 的倍数?** 大多数屏幕渲染是基于偶数像素对齐的。4 的倍数间距让你在不同屏幕密度下几乎不会出现 1px 的渲染模糊。更重要的是:**有了 scale,你不再需要凭感觉决定"这里用 13px 还是 14px"——你只能选 12 或 16,选择减少了,界面一致了。**

**大厂范本:Google Material Design 的 token 体系**

Google 在 Material Design 3(M3)里把 tokens 做成了公开标准,并把整个体系开放出来。M3 的 color system 分三层:

- **Reference tokens**(原始):所有可能的颜色值(如 `md.ref.palette.primary40 = #6750A4`)
- **System tokens**(语义):把颜色映射到用途(`md.sys.color.primary = md.ref.palette.primary40`)
- **Component tokens**(组件级):组件内部用(`md.comp.filled-button.container.color = md.sys.color.primary`)

这套分层让 Google 能在不动组件代码的情况下,整体换主题(light/dark/动态颜色都通过替换 system tokens 实现)。你不需要照搬这套体系的复杂度,但**"有两层:原始层 + 语义层"这个结构值得复制**——它是你以后能一键换主题、不改组件代码的前提。

M3 还提供了一个工具 **Material Theme Builder**:输入你的品牌色,自动生成符合对比度标准的整套颜色 token。独立开发者直接用这个起步,5 分钟出一套有 accessibility 保障的调色板。

**给你的最小 token 清单**:作为独立开发者,一开始不需要三层体系。把以下 token 定义好就够用了:

| 类别 | Token 数量 | 示例 |
|------|-----------|------|
| 颜色:品牌色 | 2-3 个 | primary、primary-hover、primary-foreground |
| 颜色:中性色 | 5-6 个 | surface、surface-raised、border、muted、text、text-muted |
| 颜色:状态色 | 4 个 | success、warning、error、info |
| 间距 scale | 8-10 个 | 4/8/12/16/24/32/48/64/96/128px |
| 圆角 | 3-4 个 | sm(4px)/ md(8px)/ lg(16px)/ full(9999px) |
| 阴影 | 3 个 | sm / md / lg |
| 字号 scale | 6-8 个 | xs(12px)/sm(14px)/base(16px)/lg(18px)/xl(20px)/2xl(24px)/3xl(30px) |

这大概 35-40 个 token,能覆盖 95% 的界面决策。剩下的全参照这些 token 衍生。

## 打法二:原子组件先做,不要一上来做"大"组件

有了 tokens 之后,**从最小的原子组件开始构建**:Button、Input、Badge、Avatar、Card……而不是一上来就做 "UserProfilePanel" 或 "DashboardSidebar"。

**原子优先的逻辑**是这样的:大组件由小组件拼成,小组件由 token 驱动。如果你一开始做大组件,里面的按钮就会和另一个页面手写的按钮不一致。把 Button 原子化之后,全局所有按钮都是同一个组件,改一处,全局更新。

**实操上,一个"好"的原子组件具备什么**:

```tsx
// 好的 Button 组件该有什么

type ButtonProps = {
  variant: 'primary' | 'secondary' | 'ghost' | 'destructive';
  size: 'sm' | 'md' | 'lg';
  isLoading?: boolean;
  disabled?: boolean;
  leftIcon?: ReactNode;
  children: ReactNode;
  onClick?: () => void;
};

// variant 映射到 token
const variantStyles = {
  primary:     'bg-[var(--color-primary)] text-white hover:bg-[var(--color-primary-hover)]',
  secondary:   'bg-transparent border border-[var(--color-border)] text-[var(--color-text)]',
  ghost:       'bg-transparent text-[var(--color-text)] hover:bg-[var(--color-surface-raised)]',
  destructive: 'bg-[var(--color-error)] text-white',
};

// size 映射到 space token
const sizeStyles = {
  sm: 'h-8  px-[var(--space-3)] text-[var(--text-sm)]',
  md: 'h-10 px-[var(--space-4)] text-[var(--text-base)]',
  lg: 'h-12 px-[var(--space-6)] text-[var(--text-lg)]',
};
```

注意两点:**variant 的语义名(primary/destructive)比颜色名(blue/red)好**——以后改品牌色,组件不用动;**size 的单位锁在 token 上**,不散落成 `px-3` `px-4` 这种 Tailwind 魔法数字。

**从原子到组合的路线图**:

```
Tokens (颜色/间距/字号/圆角)
  ↓
原子组件 (Button, Input, Badge, Avatar, Icon, Text)
  ↓
组合组件 (FormField = Label + Input + ErrorText)
           (Card = 容器 + CardHeader + CardContent)
           (Dialog = 遮罩 + Panel + 关闭按钮)
  ↓
页面级 (DashboardPage = Sidebar + TopNav + main content)
```

**每往上一层,都是在拼下面一层的积木,没有一处写硬编码的像素值或颜色。** 这就是设计系统的一致性来源。

**大厂范本:Apple HIG 的组件哲学**

Apple 的 Human Interface Guidelines 对原子组件有极严格的要求。以 iOS 的 Button 为例,HIG 定义了:

- 三种标准 size:`mini`(26pt)/ `small`(30pt)/ `regular`(34pt)/ `large`(50pt)
- 四种 style:`filled`/ `tinted`/ `gray`/ `plain`
- 强制规定最小点击区域是 44×44pt(基于手指触控精度)
- cornerRadius 跟随 Dynamic Type 自适应(字号变大,圆角也稍增)

这套约束看起来细,但**效果是:你在任何 Apple 原生 App 里看到一个 Button,你立刻知道怎么点它、点了会怎样——这就是一致性建立的信任感**。用户不用重新学"这个 App 的按钮在哪"。

独立开发者的启示:**你不需要 Apple 这种颗粒度,但每个原子组件要有一个"标准形态",全局只用这一个,不要出现"这个页面用的是圆角 6px,那个页面用的是圆角 10px"的情况。**

## 打法三:一个人建轻量设计系统的节奏

**最大的错误是一上来想造大而全的设计系统。** 你不是 Google,没有 20 个人的 design infra 团队,也不需要覆盖几十个平台的 token 层。**一个人的设计系统只有一个目标:让你以后少做重复决策。**

推荐的建设节奏是:

**第一步:先定 tokens,用 CSS 变量或 Tailwind 的 `theme.extend` 写死**

```js
// tailwind.config.js
module.exports = {
  theme: {
    extend: {
      colors: {
        primary:    '#3B82F6',
        'primary-hover': '#2563EB',
        surface:    '#F9FAFB',
        border:     '#E5E7EB',
        muted:      '#6B7280',
      },
      borderRadius: {
        sm: '4px',
        md: '8px',
        lg: '16px',
      },
      spacing: {
        // 覆盖 Tailwind 默认的 4px base,显式命名
        // 直接用 Tailwind 的 p-1=4px p-2=8px p-4=16px 即可
      },
    }
  }
}
```

从这一刻起,你的颜色只有 `primary` 和 `surface`,不会有 `blue-500` 和 `blue-600` 同时出现在代码里。

**第二步:做 5 个最高频的原子组件**

不是做所有可能的组件,是做你在这个项目里出现超过 3 次的东西:Button、Input、Badge、Modal、Card。这 5 个先做好,其他的用到时再加。

**第三步:记一个"组件清单"文档**

不需要 Storybook(对独立开发者来说维护成本太高),维护一个简单的 markdown 文件就够:

```markdown
## 组件清单

| 组件       | 状态 | 位置                    | 用法示例                    |
|-----------|------|-------------------------|-----------------------------|
| Button    | ✅   | components/ui/button    | `<Button variant="primary">` |
| Input     | ✅   | components/ui/input     | `<Input label="邮箱" />`    |
| Badge     | ✅   | components/ui/badge     |                             |
| Modal     | ✅   | components/ui/modal     |                             |
| Skeleton  | 🚧  | 待建                    |                             |
```

这个清单防止你以后忘了已经做了什么,又在另一个地方手写了一个临时 button。

**第四步:定"禁止直出"原则**

凡是在 token 里定义过的值,禁止在组件外部直接写像素值或颜色值。Tailwind 用户:不允许在组件外用 `text-blue-500`,只允许用 `text-primary`。这一条执行下去,一致性自动保障 90%。

## 打法四:用现成组件库省力,不要重复造轮子

**独立开发者最大的时间浪费是从零做 UI**。有好的开源组件库,你完全可以站在巨人的肩膀上:底层组件直接拿,设计决策直接用,只在 tokens 层定制你的品牌风格。

**shadcn/ui:目前最适合独立开发者的组件库**

shadcn/ui(基于 Radix UI + Tailwind CSS)的设计哲学和本章讲的完全一致:

- 每个组件都是**复制到你自己项目的源码**,不是依赖包——你完全控制
- 底层用 **Radix UI** 处理 accessibility(键盘导航、ARIA、焦点管理),不用自己搞
- 样式层全是 **Tailwind CSS**,token 化定制在 `tailwind.config.js` 里一处完成
- 组件颗粒度对:Button、Input、Select、Dialog、Tabs、Toast……刚好是那 5-10 个你最需要的原子

**最值得注意的是:shadcn/ui 的组件默认就不丑**。它的 Button、Card、Input 的字重、间距、圆角是经过设计的,不是工程师拼凑的。你直接用,baseline 就有了专业感。然后在 `globals.css` 里改 7 个 CSS 变量,就变成你的品牌色:

```css
/* globals.css —— 改这里就够了 */
:root {
  --primary: 221.2 83.2% 53.3%;    /* hsl 格式 */
  --primary-foreground: 210 40% 98%;
  --radius: 0.5rem;
}
```

**一次改动,所有组件跟着变**——这就是 token 体系的威力。

**Radix UI:如果你想要更底层的控制**

Radix UI 是 shadcn/ui 的下层,提供的是"功能正确但无样式"的组件:Dialog、DropdownMenu、Select、Tooltip……所有交互逻辑(打开关闭、键盘导航、Esc 关闭、焦点陷阱)都做好了,样式完全自己写。如果你对设计有更高要求、不想受 shadcn/ui 样式约束,可以直接用 Radix。

**Tailwind CSS + shadcn/ui 的对比:差做法 vs 好做法**

| | 差做法 | 好做法 |
|--|--------|--------|
| 按钮颜色 | `bg-blue-500` 直接写在按钮里 | token 化:`bg-primary`,在一处定义 `--primary` |
| 间距 | `p-3` `p-4` `p-5` 混用,凭感觉 | 只用 4 的倍数:p-2/p-3/p-4/p-6,来自 scale |
| 圆角 | 有的 `rounded`,有的 `rounded-lg`,有的 `rounded-xl` | 统一用 `rounded-[var(--radius)]` |
| 字重 | 标题有时 `font-semibold` 有时 `font-bold`,不统一 | 在 tokens 里定义 heading weight,组件复用 |
| Dark mode | 每个组件单独写 `dark:bg-gray-800` | 用 shadcn/ui 的 HSL token 方案,暗色模式一行切换 |

**其他值得了解的选项**:

- **Headless UI**(Tailwind Labs 出品):和 Radix 类似的无样式组件库,Vue 也有
- **Mantine**:React 组件库,比 shadcn/ui 内置更多功能组件(如 DatePicker、RichTextEditor),但自定义余地稍小
- **Ant Design / MUI**:功能最全,但风格偏企业、定制成本高——**做面向 C 端的消费者产品通常不推荐**,显得太"后台系统"

## 大厂怎么做:完整设计系统的参照

理解大厂的设计系统,不是为了照抄,而是为了看懂**背后的决策逻辑**,取你用得上的部分。

**Google Material Design:设计系统的公开教科书**

Material Design(从 M1/2014 到 M3/2021)是目前最完整的开源设计系统之一。Google 把它整个公开,包括:

- 完整的 token 体系(前面讲到的三层结构)
- 所有核心组件的规范文档(交互状态、动效时长、对比度要求)
- Figma 模板、Flutter/Android/Web 实现
- Color system 的科学依据(基于 HCT 色彩空间,比 HSL 在感知均匀度上更好)

**Material Design 为什么值得研究?** 因为 Google 有来自全球几十亿设备、几百个语言环境的产品需求,它的每个设计决策背后都是超大规模的可用性验证。例如:

- M3 的 "dynamic color"(动态颜色):Android 12+ 会从用户壁纸提取颜色生成主题——这要求底层 token 完全解耦,组件层不能硬编码任何颜色。这逼着 Google 把三层 token 体系做得极其严格。你学到的东西是:**"组件不能知道颜色是什么,只能知道'这里用 primary color'"**。

- Material Design 的 elevation(海拔)系统:用 overlay 透明度而不是阴影来表示层级(因为 dark mode 下纯阴影不够明显)。这个细节大多数人不知道,但它解释了为什么 Material 风格的卡片在深色主题里仍然层次清晰。

**Apple HIG:极简和一致的工程化**

Apple 的设计系统没有像 Material 那样全套开放,但 HIG(Human Interface Guidelines)文档本身就是一份详细的"为什么"说明。

Apple 在设计系统上最值得独立开发者学的一点是:**SF 字体(San Francisco)的 optical sizing**。SF 字体在 20pt 以下会自动切换到 Text variant(字母间距稍宽),20pt 以上切到 Display variant(间距紧缩)——**用户从来看不出这个切换,但读起来就是更舒服**。这背后是字体渲染的精细工程:同一套 token(`font-sf-pro`),不同尺寸下自动选最合适的变体。

Apple 把这种"系统做决策,用户和开发者不用操心"的理念贯穿了整个 HIG。这对你的启示:**好的设计系统应该让"用错"变得困难,让"用对"变成默认**。shadcn/ui 的默认值为什么好用?因为它做了大量"你不用想,直接用就是对的"的决定。

## 一致性的视觉对比:高级 vs 廉价

这里给几组具体的对比,帮你看清楚"一致"和"不一致"在视觉上是什么差别。

**对比一:间距不一致 vs 间距系统化**

```css
/* ❌ 廉价:每处都是感觉出来的数字 */
.card { padding: 18px; }
.button { padding: 9px 17px; }
.form-field { margin-bottom: 11px; }

/* ✅ 高级:全来自同一个 scale */
.card { padding: var(--space-4); }          /* 16px */
.button { padding: var(--space-2) var(--space-4); }  /* 8px 16px */
.form-field { margin-bottom: var(--space-3); }       /* 12px */
```

视觉上,前者看起来"怎么哪里都差那么一点点",后者感觉"对齐了"。这不是审美,是节律感——4px 的倍数让元素之间的关系可以被大脑预期。

**对比二:颜色不一致 vs token 化**

```css
/* ❌ 廉价:三处蓝色各不同 */
.nav-link.active { color: #3B82F6; }
.button-primary  { background: #2563EB; }
.badge-info      { background: #60A5FA; }

/* ✅ 高级:全部指向同一个 token,含义清晰 */
.nav-link.active { color: var(--color-primary); }
.button-primary  { background: var(--color-primary); }
.badge-info      { background: var(--color-primary-light); }
```

前者改品牌色要改三处,漏改一处就出 bug;后者改一个 token,全局更新。

**对比三:圆角不统一 vs 统一**

```css
/* ❌ 廉价:到处不一样 */
.card    { border-radius: 12px; }
.button  { border-radius: 6px; }
.input   { border-radius: 8px; }
.badge   { border-radius: 4px; }

/* ✅ 高级:两个 token 覆盖全部 */
.card    { border-radius: var(--radius-lg); }  /* 16px */
.button  { border-radius: var(--radius-md); }  /* 8px */
.input   { border-radius: var(--radius-md); }  /* 8px */
.badge   { border-radius: var(--radius-sm); }  /* 4px */
```

圆角统一之后,界面会有一种"同一个人设计的"整体感。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| 改品牌颜色 | 全局搜 `#3B82F6`,逐个替换,漏掉几处 | 改一个 token,30 秒完成 |
| 加新页面 | 从零开始写样式,感觉哪里对不上 | 拼已有的原子组件,2 小时出一个一致的页面 |
| 间距决策 | 每次都要想"这里用多少 px" | space scale 只有 8 个选项,直接选 |
| 组件库选型 | 觉得 Ant Design 功能最多就用 Ant Design | 根据定制需求选:C 端产品用 shadcn/ui,需要大量数据表格再考虑 Ant |
| Dark mode | 后期再说,到时候再改 | token 化设计从第一天就支持 dark mode |
| 设计系统规模 | 一上来想做"大而全",然后放弃 | 先做 tokens + 5 个组件,用到就加,永远保持可运行 |

## 反模式(工程师高发)

- **"先做完功能,以后再整理样式"**:一旦散乱了再整理的成本是从零开始建 tokens 乘以已有代码量。tokens 应该是第一天就定好的,10 分钟的事。

- **"我自己写一套,不依赖别人的库"**:独立开发者时间最稀缺。shadcn/ui 这类库已经解决了 accessibility、键盘导航、ARIA,你自己重写要花多久?且大概率是写错的。

- **"组件库太重了,我只用 Tailwind 写 class"**:用 Tailwind 写 class 没问题,但如果同一个 button 在 10 处出现,你就应该把它提取成组件——哪怕只是一个接受 `variant` 和 `children` 的简单 React 组件。Tailwind + 组件化才是完整方案。

- **"我的设计系统有 200 个 tokens"**:超过 50 个 token 之后维护成本急剧上升。独立开发者的上限大概是 40-50 个 token,够用了,过了就是过度工程。

- **"每个组件都自己想 size 和 variant"**:Button 有 size,Input 也有 size,要用同一套命名(`sm/md/lg`)和同一套 scale。不同组件之间的 API 不一致,自己用起来都会绕。

## 动手:这章怎么练

1. **在现有项目里统计一次"颜色散点"**:在代码库里搜 `#`,数一数你用了多少个不同的颜色值。超过 10 个就是散了,把它们归并进 tokens。

2. **建一个 `tokens.css`(或 `tailwind.config.js` 的 `theme.extend`)并在 5 个地方替换**:不需要一次全改,先从最高频的颜色和间距开始,感受有 token 和没 token 的差别。

3. **用 shadcn/ui 初始化一个项目,阅读它的 CSS 变量**:看懂它的颜色 token 命名逻辑(为什么用 HSL?为什么要 `--foreground`?),然后改掉 `--primary` 看效果。30 分钟够了。

4. **把你产品里出现 3 次以上的 UI 片段提取成组件**:从 Button 开始。给它 `variant` 和 `size`,保证全项目只有这一个 Button。

## 自测清单

- [ ] 我的项目有一个集中的地方定义颜色、间距、圆角(tokens)
- [ ] 我的代码里没有出现同一个颜色值的两种写法(如 `#3B82F6` 和 `blue-500` 同时存在)
- [ ] 我的间距遵循一个 scale(如 4px 倍数),而不是每次凭感觉
- [ ] 我有至少一个原子组件(Button / Input)接受 `variant` 和 `size` 参数而不是硬编码
- [ ] 我知道 shadcn/ui 是什么,知道怎么安装和修改它的颜色 token
- [ ] 我的设计系统文档(哪怕一个 markdown 文件)记录了已有的组件清单

## 真实参照(去看这些)

- **Material Design 3 官网**(m3.material.io)—— token 体系的完整文档,每个决策都有"为什么"的解释,是目前最好的开源设计系统教材
- **shadcn/ui 官网**(ui.shadcn.com)—— 直接看它的代码和 CSS 变量设计,比看文章理解更快
- **Apple Human Interface Guidelines**(developer.apple.com/design/human-interface-guidelines)—— 看"Foundations > Color / Typography / Layout"三章就够,理解 Apple 为什么这么规定
- **Refactoring UI(Adam Wathan & Steve Schoger)**—— Tailwind 作者写的设计书,工程师友好,大量"差做法 vs 好做法"的具体对比,和本章精神完全一致(「待核」:具体出版年份)
- **Brad Frost《Atomic Design》**—— 原子设计方法论的源头,免费在线阅读;概念清晰但工程落地部分偏早期,结合 shadcn/ui 来看更有用
