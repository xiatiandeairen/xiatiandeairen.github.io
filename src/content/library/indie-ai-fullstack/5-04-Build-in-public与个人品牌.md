> 发布日流量是个幻觉——它不是"发布那天发生的事",而是"发布前几个月一点点攒起来的结果"。这章讲 build in public 的底层逻辑,以及 solo 开发者如何用"时间 + 真诚"替代市场预算。

---

## 为什么发布那天总是"鬼城"

你花了三个月憋出一个产品,发布当天在 Twitter/X 发了一条"我上线了!",结果:37 个点赞,20 个是自己关注的人,3 个是亲戚。

这不是你产品差。这是你在一个没人认识你的地方突然开口说话。

大厂发布新产品,有 PR 稿、有媒体关系、有发布会预热、有广告预算——它的"发布日"是几个月造势的高潮。作为 solo 开发者,你没有这些。但你有一样大厂越来越难做到的东西:**真实的个人**。

**Build in public 的本质,是把大厂发布前的"预热工作"分散到开发全程公开做。** 不是秘密开发、一次性爆发,而是持续公开进展,让人在你发布之前就知道你、等待你、期待你。

发布日有没有人,不取决于发布那天你说了什么,而是发布前几个月你攒了多少关注你的人。

---

## 机制:为什么它有效

有两件事让 build in public 的分发逻辑成立:

**1. 平台算法奖励持续输出,不奖励单次爆发。**

Twitter/X、LinkedIn、YouTube 的算法,都更倾向于推送"活跃、稳定输出"的账号。偶尔的爆款帖会带来流量尖刺,但只有持续输出才能建立算法权重——让平台"认识你"。单次发布一条爆款能带来一批人,但这批人看到的下一条也得是你,才能沉淀成关注者。

**2. 人天然信任"看着长大的东西"。**

你关注了一个人三个月,看他踩过哪些坑、数据从零到有、功能一条条长出来——等他发布时,你的感受不是"又一个不知名的产品",而是"我见证了这个东西"。见证者天然更愿意试用、更愿意传播,因为试用本身也成了他们故事的一部分。

**一句话机制:build in public 是把你的用户获取成本分摊到几个月的"过程内容"里。** 大厂用预算买注意力,你用时间赚注意力。

---

## 打法一:从第一天就公开,不是做好了再说

最常见的误区:**"等我把产品做好了再发"。** 这句话让你把最宝贵的积累期全部沉默了。

Pieter Levels 是这方面最知名的标杆。他在 2014 年前后提出"12 Startups in 12 Months"挑战,几乎是把每个项目的整个开发过程都放在 Twitter 上直播——包括想法确认、技术选型、MVP 进展、上线数字、踩坑复盘。Nomad List 还只是一个手工整理的 Google 表格时,他就在发。正是这种"从第一天就说"的节奏,让他每次发布新项目时都有几万人等在那里(具体粉丝数「待核」)。

**为什么从第一天就说比等做好了再说有效?**

- 失败过程和踩坑帖往往比成功帖更有共鸣。没有人不踩坑,看到有人真诚说"我这步想错了"会自然产生信任。
- 最早关注你的人会变成你最忠实的早期用户。他们参与了你的建造过程,有"我是开创者"的心理认同。
- 过程帖不需要"好消息",只需要真实。"今天 API 报错搞了 6 小时"和"今天做完了注册流程"都是有效内容。

**可操作:**下周一把你手头在做的产品写一条过程帖发出去。格式不用精心设计,说你在做什么、做到哪了、遇到什么问题就够了。这一条帖,价值远大于你三个月后发布当天那条"我上线了"。

## 打法二:发"过程数据",不发"功能清单"

同样是 build in public,有两种截然不同的内容:

**功能清单式(没人看):**"我们上线了功能 A / B / C,欢迎体验!"

**过程数据式(有人转):**"第一个月:日活 12,注册 89,付费 1 人。第二个月改了 onboarding 之后,付费变成 9 人。这是我改了什么。"

过程数据之所以更有传播力,是因为它满足了读者的核心需求:学到可复用的经验。功能清单是广告,过程数据是内容——前者让人想关掉页面,后者让人想转发和收藏。

独立开发者社区里有个非正式惯例:把关键数字定期公开。月活、MRR(月经常性收入)、churn rate、注册转付费比。这些数字本身不一定漂亮,但公开本身传达了一个信号:**这个人说真话,不是表演。**

**案例拆解:Pieter Levels 的公开财务**

Levels 把自己所有产品的收入实时公开在一个页面上,是独立开发者圈子里被引用最多的"benchmark"之一。他的逻辑很清楚:数字公开,既给关注者一个持续回访的理由(每个月来看数字有没有变化),又让"增长"本身变成内容。某个月 MRR 涨了 20%(具体数字「待核」),这个增长本身就是一篇帖子。他不需要另外想内容:进展就是内容。

**可操作:**找出你产品里一个数字——哪怕今天是 0——决定每周或每月公开它,并且说一句你在做什么来改变它。不需要数字好看,需要数字真实。

## 打法三:真诚的踩坑帖 > 精心策划的成功帖

这是很多工程师做 build in public 时最难突破的一关。我们习惯于展示解决方案、不展示迷茫。但在内容传播上,**"我踩了一个大坑"通常比"我做了一个牛功能"传播效果好几倍。**

原因很简单:踩坑是大多数人的日常,成功是少数人的结果。"我对目标用户的假设全错了,重做"这样的帖子,触达的是每个正在做产品的人的共鸣神经。而"我上线了 X 功能"只触达了少数"恰好在找这个功能"的人。

**案例拆解:Indie Hackers 上的失败帖**

Indie Hackers 是独立开发者最大的分享社区之一。这个平台上流量最高的帖子,几乎一半是"失败复盘"类——"我做了 18 个月,最终关掉了,原因是……"这种帖子能上热门,是因为它对还在路上的人极其有用:真实的失败成本、真实的错误路径,是最珍贵的"避坑地图"。

对独立开发者来说,发布踩坑帖还有一个意外收益:**你会因此认识最好的用户和同行**。发布"我在尝试 X 方向,但遇到了 Y 问题"之后,比你见过更多这类问题的人会主动找来。踩坑帖是一种低成本的问题求解机制。

**可操作:**这周找一件你上周想错或做错的事,写一条帖子说清楚:原来以为什么 → 实际是什么 → 现在怎么调整。不用装作你"已经解决了",过程中也可以说。

## 打法四:个人品牌让分发成本随时间下降

Build in public 不只是一种发帖策略,它在积累一样东西:**个人品牌**。

个人品牌在分发上的作用,是让你的"观众"把你当一个值得长期关注的人,而不是一个随机冒出来的产品账号。产品可以失败、可以转型,但"关注这个人"的动作一旦发生,下一个产品的发布就有了一批初始受众。

这就是为什么 Pieter Levels 连续做了多个产品,每个产品发布时都有数万人等待——他的个人账号本身就是渠道。Nomad List、Remote OK、PhotoAI、Interior AI,产品不同,但渠道是同一个:Levels 这个人。

大厂做新产品,靠品牌溢价和渠道资源。独立开发者唯一能建的"品牌资产",就是**"这个人"**。当你以人的身份在做产品,而不是以公司的身份,你有天然的信任优先级——人会信人,不一定信公司。

**个人品牌的复利机制:**

- 第一个产品:0 受众,冷启动,发布日惨淡。
- 持续输出 6 个月:积累几千关注者。
- 第二个产品:发布日有几千人知道,转化部分成用户。
- 继续输出 + 产品成功:受众快速涨。
- 第三个产品:发布日自带初始流量,产品本身可能因此进入增长曲线。

**分发成本是递减的**,前提是你一直在积累"关注这个人"的资产,而不是每次发布都从零开始。

**案例拆解:字节系的"作者 IP"逻辑**

抖音和剪映的增长里,有一个值得 solo 开发者借鉴的逻辑:平台的内容消费是基于"关注这个人"而不仅仅"搜索这个话题"。一个持续输出、有辨识度的作者,能让粉丝每次发布新内容时自动获得分发——算法会先推给关注者,再看关注者反应决定是否更大范围推。

独立开发者的"个人 X 账号"和抖音作者账号是同一个逻辑。平台先给你的关注者看,他们的互动决定你能否扩散到更多人。你积累的关注者,就是你的"保底触达基数"。基数越大,每条帖子的起跑线越高。

**可操作:**给自己的 Twitter/X 或 LinkedIn 主页写一句清晰的自我定位:"我在做 X,每周分享进展和踩坑。"这句话让新来的人知道关注你意味着什么,降低他们关注的决策成本。

## 打法五:节奏 > 质量,但不要无效输出

Build in public 最大的执行陷阱:**要么完美主义——每条帖子都要写得漂亮,所以发不出去;要么无脑刷屏——发了大量低价值内容,反而减损信任。**

节奏比质量重要,指的是:**一周一条真实的进展帖,长期价值远大于偶尔爆发一篇精心打磨的长文。** 持续节奏让算法、让关注者都形成"这个人一直在"的印象。一旦你沉默两个月,这种印象就断了,重启成本很高。

但节奏不等于无效输出。每条发出去的内容,至少得满足一个标准:**读者看完,能拿走一个具体的东西——一个数字、一个判断、一个教训、一个值得关注的视角。**

"今天继续做产品"不是内容。"今天把付费流程改了一步,转化从 2% 涨到 5%(待核),原因是……"才是内容。

**节奏建议(solo 可操作):**

- 每周 1 条:这周做了什么、数字怎么变、一件想分享的事。
- 每月 1 篇:这个月最重要的一个判断或教训。深度够的话,这种帖往往能带来最多高质量关注者。
- 重要里程碑:首个付费用户、月收入第一次过 X(待核)——这种"小高光"帖值得认真写。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| 发布策略 | 做好了再发,发布当天努力造势 | 从第一天就公开,发布日是几个月积累的收获 |
| 内容选择 | 展示功能和"好消息" | 展示过程、数据、踩坑和判断 |
| 沉默时间 | "没进展就不发" | 进展慢时更要发——过程本身就是内容 |
| 对粉丝数的态度 | 焦虑数字涨得慢 | 关注受众质量——愿意试用、愿意反馈的人比僵粉有价值 |
| 产品 vs 个人 | 用产品账号发布 | 用个人账号积累,产品挂在个人名下 |
| 长期资产 | 每个产品重新冷启动 | 个人品牌成为跨产品渠道,分发成本递减 |

## 反模式(工程师高发)

**"做好了再说"——最贵的沉默**

工程师的本能是"完成才发"。但做好的标准是移动靶,很多人等产品"完美"的结果是永远没发出去。更重要的是,开发阶段是积累受众成本最低的时候:这时候你说的话是"进展分享",不是广告,人更愿意看。等你发布了,每条帖子都会被读者过滤为"他在推销"。

**用产品账号代替个人账号**

很多开发者给产品注册单独的 Twitter/X 账号发 build in public 内容。这是路径更窄的选择。产品可能失败,但你的个人积累不会清零。用你自己的账号,哪怕产品后来转型或关掉,关注者还是跟着你而不是跟着产品。

**把 build in public 做成日记**

"今天早上 10 点开始写代码,下午茶喝了杯咖啡"——这不是内容,是日志。内容需要对读者有价值的密度。流水账式的输出不会积累受众,只会消耗关注者的注意力。

**只在顺境发,不在困境发**

很多人顺利时活跃,卡住时沉默。这是最大的失误。困境帖才是最有传播力的帖子类型。"我这周没什么进展,意识到最初的假设可能错了——在考虑 X 和 Y 两条路"这种帖子,比"我上线了新功能"更容易引发真实互动和关注。

**把 build in public 和"营销帖"混用**

Build in public 的信任基础是"这是真实的人在做真实的事"。一旦你开始发"功能上线了,点这里注册"这类广告帖,调性就断了。过程内容和推广内容可以共存,但比例和节奏要控制好。

## 动手:这章怎么练

1. **今天就发第一条过程帖**:说你在做什么产品、做到哪了、遇到什么问题。格式不重要,真实最重要。发在你最活跃的平台上。

2. **设置一个固定的发帖节奏**:每周固定一天写这周的进展摘要发出去。不需要有"好消息",有真实就够了。坚持 8 周,看受众有没有在积累。

3. **找一个你上周做错或想错的事,写成踩坑帖**:按"原来以为 X → 实际是 Y → 现在打算 Z"的格式写。这是最容易引发真实互动的内容类型。

4. **整理你的个人主页**:在你的 X/LinkedIn 主页写清楚"你在做什么 + 你会持续分享什么"。让第一次看到你的人知道关注你能得到什么。

## 自测清单

- [ ] 我已经在做的产品,有没有在任何公开平台上说过"我在做这件事"
- [ ] 我能说清楚自己的发帖节奏(每周一条/每两周一条)并且打算坚持
- [ ] 我发的内容里,有没有真实的数字或真实的踩坑——而不只是功能介绍
- [ ] 我用的是个人账号而不是产品账号在积累受众
- [ ] 如果我明天发布产品,现在已经有多少人知道我在做这件事——这个数字能告诉我还要准备多久

## 真实参照

- **Pieter Levels(@levelsio)** —— build in public 最知名的实践者,12 Startups in 12 Months、Nomad List、Remote OK,几乎全程公开建造。他的 Twitter 和 pieter.levels.io 是第一手案例。
- **Indie Hackers(indiehackers.com)** —— 独立开发者分享收入、增长、踩坑的社区。看任何一个热门案例,都能看到"过程公开"的核心模式。
- **《Show Your Work》—— Austin Kleon** —— 最薄最直接的"为什么要公开你的工作过程"的书,3 小时读完,论点和这章高度呼应。
- **Justin Welsh(@thejustinwelsh)** —— LinkedIn 个人品牌建设的典型案例,从 0 到几十万关注者的路径有详细复盘可查(具体数字「待核」)。
- **Marc Lou(@marc_louvion)** —— 近几年 indie hacker 圈高频出现的案例,持续 build in public + 多产品策略,MRR 成长轨迹有公开记录(具体数字「待核」)。
