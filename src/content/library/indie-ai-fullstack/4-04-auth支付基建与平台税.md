> 这章解决一件事:**哪些基础设施永远别自建、用买的;以及你每笔收入里那一刀"隐形抽成"到底有多深,怎么看清它、绕开它、对冲它。**

---

## 为什么这章要单独讲

大厂工程师最容易犯的错,不是技术选型不当,而是**方向性错误:把精力花在已经有人解决得比你好十倍的问题上**。

auth、支付、基础设施这三件事,独立开发者往往有两种极端:一种是花三个月自建,理由是"我要完全掌控"、"现成方案不够灵活";另一种是稀里糊涂地上了平台(App Store / Google Play),把 30% 的收入交出去,还不知道自己在承担什么风险。

两种都在亏钱,只是亏法不同。

**这章的核心立场只有一句:基建全买现成的,把节省出来的开发时间留给真正的差异化;但要把平台税、依赖风险算明白,能绕开 30% 就绕,能留住自有渠道就留。**

---

## 机制:为什么这些问题"已经解决了"

auth、支付、基础设施都属于**通用工程问题**——全球几百万个产品都需要它们,已经有公司专门把这件事做到了工业级:安全审计、边缘情况处理、合规、自动扩容、多地区支持……这些都是你作为 solo 开发者几乎不可能在合理时间内做到同等质量的。

以 auth 为例,自己实现一个"安全的用户系统"意味着:密码哈希(bcrypt/Argon2)、防暴力破解、session 管理、refresh token 轮换、邮件验证、密码重置、OAuth 2.0 与各平台对接……随便一个细节出问题就是安全漏洞。做一个 B 级功能要三周,还不如 Auth0 / Clerk / Supabase 一个下午接好的安全。

**时间机会成本才是最大的成本。** 你每多花一周在 auth 上,就少一周在产品核心体验上。对于一个人的团队,这个 trade-off 几乎是单边的。

支付更甚:信用卡 PCI DSS 合规本身是一套审计流程,任何直接碰卡号数据的系统都要通过,Stripe 帮你完全屏蔽了这个问题,你的服务器从不接触原始卡号,风险和合规成本全砍掉。

托管基础设施(Vercel / Railway / Fly.io / Render)的逻辑一样:运维服务器是一项全天候的工作——打安全补丁、监控磁盘、处理 DDoS、管理证书……托管方已经把这些做成产品卖给你了,你不需要重造轮子。

**结论:这三件事不自建,不是"偷懒",是战略上的正确分工。**

但有一面硬币的另一面需要讲清楚:**平台税**。

平台税不只是 App Store 的 30%。你每用一个托管 SaaS,每进一个平台分发渠道,就在承受一种依赖——价格上涨、政策变动、下架风险,每一种都是平台手里的筹码。看清这些成本和风险,是这章另一半的任务。

## 打法一:auth 用托管方案,别自己写

### 怎么选

主流选项三类:
- **BaaS 自带 auth**:Supabase、Firebase——你整个后端都在上面,auth 作为套餐一部分。最小化工程量。
- **专门 auth SaaS**:Clerk、Auth0——功能更丰富(magic link、MFA、组织/团队权限),适合需要企业级 auth 的产品。
- **社交登录库**:NextAuth / Auth.js——开源库,自己管理 session,适合想多掌控一点但不想从头写的场景。

### Sign in with Apple——被迫上架的"平台税"变体

**Sign in with Apple 是 Apple 在 2019 年强制要求的**:任何 iOS App 只要支持第三方社交登录(Google、Facebook 等),就必须同时接入 Sign in with Apple。不接——审核拒绝,上不了架。

表面上看这是"给用户更多选择",实际上它的作用是:
1. **把用户留在 Apple 生态**——邮箱隐藏(Apple 生成中继邮箱),你拿不到真实的用户邮件地址。
2. **抬高竞争对手(Google/Facebook)在 iOS 上的入口成本**。
3. **给用户一个"更隐私"的信号**,配合 Apple 的差异化营销。

对 indie 开发者意味着什么:**你得接,但要理解它的限制**——用户用 Sign in with Apple 注册后,你可能拿不到真实邮箱(除非用户允许),这影响你的邮件营销、用户触达能力。解决办法:在注册流程里引导用户额外绑定一个邮件地址(以后可以联系你),或把邮件订阅作为独立步骤。

**这是平台税的软版本**:不是直接抽钱,而是用功能限制卡住你的用户关系。

### 一人团队实操建议

Supabase 是目前综合性价比最高的起点:auth、数据库、存储、边缘函数一体化,免费 tier 够早期项目跑。如果你做 Next.js 全栈,Auth.js + Supabase 是常见搭档。需要 B2B、多租户权限体系,考虑 Clerk。

**一行判断标准:你的 auth 有没有特殊需求(比如自定义 token 格式、SOC2 合规审计日志)?没有就用 Supabase 自带的,一个下午接完,上生产。**

## 打法二:支付用 Stripe,全球收款 + 税务用 MoR

### Stripe 是行业默认答案

Stripe 几乎是独立开发者支付的标准选项。原因直接说:
- PCI DSS 合规由 Stripe 承担,你不碰卡号数据。
- 文档最好、SDK 覆盖最广(几乎所有主流语言/框架都有)。
- 订阅管理(Stripe Billing)、Webhook 通知、退款、争议处理——全套。
- Stripe Checkout 可以直接嵌一个托管的支付页面,不用自己写表单。

费率通常是每笔 2.9% + $0.30(「待核」各地区费率有差异),这是正常的支付处理成本,接受它。

### 面向全球:用 Merchant of Record 处理税务噩梦

如果你的用户分布在多个国家/地区,**增值税(VAT)和销售税(Sales Tax)是一个真正的坑**。欧盟要求数字服务产品向每个成员国的用户按当地税率收 VAT;美国各州销售税规则不一……作为个人开发者,你不可能手动管理几十个国家的税率、报税周期、阈值。

解决方案是用 **Merchant of Record(MoR)**:LemonSqueezy 或 Paddle。

这两个工具的本质是**它们代替你作为"销售方"**——法律上,买家是从 LemonSqueezy / Paddle 买东西,不是从你买。因此所有税务责任、合规问题、退款纠纷,由它们承担。你收到的是扣完税和平台费后的净额。

代价是手续费比 Stripe 略高(大约 5%–8%,「待核」具体费率因产品类型和月营收而异)。但换来的是:你完全不需要懂 VAT,不需要和会计师花时间研究"我的 SaaS 在德国卖是否要注册 VAT"——答案是 LemonSqueezy/Paddle 替你搞定了。

**实际用法对比:**
- 主要用户在美国、产品是 B2B SaaS、你已经注册公司:考虑直接用 Stripe,自己处理或找工具(如 TaxJar、Avalara)帮助税务合规。
- 个人开发者、全球用户、不想碰税务:LemonSqueezy 或 Paddle,交更高的抽成换省心。

**一个真实例子**:很多知名独立产品(Gumroad、LemonSqueezy 自身的用户群中有大量工具产品「待核」)选择 MoR 路线,原因很简单——税务合规一旦出错,罚款可以是几年营收,对 solo 来说是致命的。

## 打法三:基建用托管,别自己运维服务器

### 大厂思维 vs Indie 思维

大厂工程师习惯的世界是:有专门的 SRE 团队、有 on-call rotation、有内部 PaaS(Google 的 Borg/GKE、Meta 的 Tupperware 「待核」)。你从来不需要思考"服务器怎么运维",因为这层对你是透明的。

转做独立开发,**如果你用裸 VPS 自己跑服务**,你就是 SRE、也是 Infra 工程师、也是产品开发、也是客服——这些角色同时叠在你一个人身上。凌晨三点数据库挂了,没有人帮你,你得起来修。

这是一个典型的"大厂工程师转型时没意识到的坑"。

### 现代托管选项覆盖了 99% 的场景

| 场景 | 推荐 |
|------|------|
| 前端 / 全栈 Next.js | Vercel |
| 后端 API、后台任务 | Railway、Render、Fly.io |
| 全功能 BaaS(数据库 + auth + 存储) | Supabase |
| 静态站点 + CDN | Cloudflare Pages、Vercel |
| 文件存储 | Cloudflare R2(比 S3 便宜)、AWS S3 |
| 数据库托管 | Supabase / PlanetScale / Neon |

这些服务处理了:自动扩容、SSL 证书续期、CDN 分发、零停机部署、日志收集。你只需要 push 代码。

**Vercel 是最典型的"为独立开发者设计的基建"**:免费 tier 覆盖早期流量,部署一个 git push 触发,Preview URL 让你秒级看到变更效果,全球 CDN 不需要配置。对比自己管一台 Nginx + 手动部署脚本——完全不是同一个世界。

### 但要看清托管平台的定价陷阱

Vercel 免费 tier 很好,但**超量后费用上涨很快**。有独立开发者分享过:流量突然上来(比如 Hacker News 上了首页),当月账单从 $0 变成几百美元的案例(「待核」具体数字)。解决办法:设置 **Billing Alert**,超过阈值立刻通知你,不要等账单来了才知道。

Railway、Render 也类似——按使用量计费,高峰期可以很贵。早期可以定一个硬性上限(比如每月基建不超过 $X),超了就找更便宜的替代或优化。

**Boring tech 原则在这里适用**:不要追最新的托管服务,选已经稳定运营 3 年以上、有明确付费路线的服务。早期初创跑路或被收购的 BaaS 有过先例(「待核」),你的生产数据不要放在不确定的平台上。

## 打法四:监控和错误上报用现成 SaaS

这是最容易被省掉、但出问题最贵的一类工具。

**你不知道出了什么错,就没法修。** 上线后没有错误监控,等于在黑盒里开车——用户报了 bug 你无法复现,因为你甚至不知道错误栈、用户行为链路、发生频率。

现成工具:
- **错误监控**:Sentry——行业默认,免费 tier 够用,接入 SDK 大约半小时。
- **性能监控 / Uptime**:BetterStack、UptimeRobot——服务挂了自动发邮件或短信。
- **日志**:Axiom、Papertrail——托管日志比自己 grep 服务器日志好一千倍。
- **分析**:Plausible、PostHog——隐私友好的用户行为分析,比 Google Analytics 配置简单。

**一人团队的实操原则:Sentry + UptimeRobot 是最低配置,不装就不要上生产。** 这两个合计费用基本可以免费或极低成本覆盖早期需求。

## 重点:看清平台税,能绕就绕

前面讲的都是"用现成的基建"——这部分要讲另一个现成的东西:**平台分发渠道**背后的真实成本。

### App Store / Google Play 的 30% 是什么

苹果和谷歌对应用内购买收取 **30% 的抽成**——这是"平台税"的标准版。

30% 是个什么概念?如果你的产品定价 $10/月,你实际到手 $7。如果你的毛利率是 70%,平台税把你的实际盈利砍掉了接近一半。这还不算支付处理费(通常已包含在这 30% 里)。

苹果后来对年收入低于 $100 万(「待核」阈值和适用条件)的开发者提供了 15% 的小开发者计划,有所减轻。但核心逻辑没变:**你在平台上卖,平台拿三成。**

### Epic vs Apple:一场把平台税摊在阳光下的战争

2020 年,Epic Games 故意绕过 App Store 支付系统,在 Fortnite 里直接向玩家收款(给玩家更便宜的价格),苹果随即下架 Fortnite。Epic 随后起诉苹果反垄断。

这场诉讼让几件事变得清晰:

1. **平台有单方面下架权**——不需要提前通知,不需要赔偿,在你的产品最赚钱的时候可以直接关掉你的分发渠道。
2. **平台把"支付系统"和"分发渠道"强行捆绑**——你想进 App Store 就必须用它的支付,而用它的支付就交 30%。
3. **平台条款可以随时改变**——苹果历史上多次修改审核规则,调整后开发者必须跟上,否则就是下架。

Epic 诉讼的部分结果是:法院判定苹果必须允许 App 链接到外部支付页面(「待核」此项判决的最终执行状态和地区适用范围仍在变化),但这场战争还在持续。对 indie 开发者的实际意义是:**永远不要把 100% 的分发和收入押注在单一平台上。**

### 怎么算清自己的平台税

一张简单的算法:

```
实际到手 = 定价
  - 平台抽成(App Store/Google Play: 15-30%)
  - 支付处理费(若单独计算)
  - MoR 手续费(若用 LemonSqueezy/Paddle)
  - 退款/争议损耗
```

**在定价时,先算这张表,再定价。** 很多人用"定价 $10"直接算利润率,结果上线后发现实际到手不到 $7,利润被平台税压没了。

### 能绕 30% 就绕:Web 端优先策略

如果你的产品有 Web 版本,**Web 端支付完全绕开平台税**——你用 Stripe 直接收款,手续费大约 3%(「待核」各地区费率),没有 30%。

这就是为什么很多聪明的独立开发者会把主要功能做成 Web App,App 端只做功能子集或者引导用户去 Web 端订阅。苹果的规则不允许你在 App 内主动告诉用户"去网站买更便宜"(「待核」此规则在 Epic 诉讼判决后是否有变化),但你可以做成 Web First,让用户自然在 Web 上完成购买。

**Obsidian 是一个典型**:它主要作为桌面 App 分发,核心功能免费,付费同步通过自己的网站收款。它不在 App Store 上卖订阅,所以没有 30% 的问题。当然,这也意味着放弃了 App Store 的发现流量——trade-off 你自己权衡。

## 平台依赖的系统性风险

平台税不只是钱的问题,更是**控制权**的问题。

**Google 关停 Reader(2013)**——全球最大 RSS 阅读器服务,有数百万用户,Google 突然宣布关停,所有依赖 Reader API 的第三方产品和内容分发方式直接失效。**一个平台的产品决策,可以瞬间清零你在那个平台上建立的所有东西。**

**Apple Sherlocking**——苹果把独立开发者某个热门功能直接做进系统(系统翻译 App、系统截图标注……),把第三方同类应用的市场一夜归零。你做了两年的产品,苹果一个 WWDC 发布会就变成"系统自带功能"的影子。

这不是苹果/谷歌特有的恶意,这是平台的结构性激励:平台的目标是让用户留在平台,任何高价值功能都是平台的合理扩展方向。**对独立开发者来说,这意味着:单点依赖某个平台分发,是在对赌平台不来竞争你。**

**对冲平台依赖的三条原则**:

1. **有自己的网站和独立的 SEO**:这是你不依赖任何平台的分发渠道,你控制的域名和内容,别人拿不走。
2. **有邮件列表**:你和用户之间的直接联系——算法变了、平台挂了、App 被下架,邮件列表还在。用户邮件地址是你最重要的资产之一。
3. **多渠道收入**:有 App Store 收入 + Web 直售收入 + 可能的 API/B2B 渠道,任何单一渠道跌了不致命。

**邮件列表是最被独立开发者低估的资产。** Pieter Levels(Nomad List 创始人,「待核」具体数字)、Justin Jackson 等 indie maker 都反复强调:邮件订阅用户的价值远高于 Twitter/X 关注者,因为平台不能把他们从你身边带走。

## 新手 vs 资深

| | 新手 | 资深 |
|---|---|---|
| auth | 自己实现 session + 密码系统 | 直接用 Supabase / Clerk,半天接完 |
| 支付 | Stripe 接完就觉得搞定了 | Stripe + 算清税务风险,决定是否用 MoR |
| 基建 | 租一台 VPS 自己配 Nginx | Vercel / Railway 托管,设好 Billing Alert |
| App Store | 觉得 30% 是固定成本 | 算清平台税,主动设计 Web 优先收款路线 |
| 平台依赖 | 100% 押单一平台分发 | 同时建网站 SEO + 邮件列表 + 直售渠道 |
| 监控 | 上线后等用户报 bug | Sentry + Uptime 监控是标配,上线前装好 |

## 反模式(工程师高发)

- **"自建 auth 更安全"**:现实是自建更容易出安全漏洞。Clerk / Supabase 的安全工程师专职维护这个,你兼职维护。
- **"我先用 VPS 省钱,以后再迁移"**:迁移成本往往比省下的钱贵。托管服务的 DevOps 成本是隐形的,折算成你的时间可能贵 10 倍。
- **"App Store 流量大,先上架再说"**:没算清平台税就定价,毛利被压垮。也没想到万一被下架的备案。
- **"邮件列表以后再建"**:用户来了走了,你没有一个直接联系他们的方式。后来想建,发现原来的用户已经联系不上了。
- **"增值税、销售税以后再搞"**:税务合规是有回溯期的,欠了两年的 VAT 可以是一笔大钱。早期不用每个国家都注册,但要知道 MoR 帮你解决了这个问题。

## 动手:这一章怎么练

1. **盘一下你的基建账单**:列出所有正在用的 SaaS 和平台,算出每月固定成本 + 每笔交易的平台抽成合计是多少。如果没有产品就假设:定价 $19/月,卖 100 个用户,分别走 App Store 和走 Web 直售,到手差多少?
2. **搭一个最小 auth + 支付原型**:用 Supabase (auth + DB) + Stripe Checkout,从注册到付款走一遍完整流程,统计实际开发时间——这是你的"基准工时",后面每次评估自建成本时对照。
3. **建你的邮件列表落地页**:哪怕产品还没上线,先用 ConvertKit / Resend / Beehiiv 搭一个"留邮件等上线通知"页面。这是对冲平台依赖最低成本的第一步。
4. **查你目标平台的最新条款**:App Store 在 Epic 诉讼后规则有变化,「待核」各地区最新规定。花 30 分钟读一遍最新审核指南里的支付相关条款,知道边界在哪。
