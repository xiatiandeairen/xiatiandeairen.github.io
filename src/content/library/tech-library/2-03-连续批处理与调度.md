## TL;DR

第 1 章讲了一条请求内部 Prefill/Decode 两阶段的形态，第 2 章讲了一条请求的 KV cache 怎么在显存里分页存放。本章往上走一层：**一台 GPU 上同时有几十上百条请求，每条长度不同、Decode 步数不可预测、随时到达随时结束——调度器每一步该把哪些请求、各放多少 token 喂进这一次 forward？**

这是 LLM serving 吞吐的第二个地基（第一个是 PagedAttention）。核心矛盾是：传统的 **request-level（静态）批处理**把一批请求锁死，必须等批内最长那条 Decode 完才能换下一批。生成长度高度倾斜（有的 4 个 token，有的 2000 个），于是**一条 straggler 把整个 batch 的槽位占着空转**，GPU 利用率掉到 20%–40%。

Orca（OSDI 2022，FriendliAI + SNU）给出关键洞察：**iteration-level scheduling**——调度粒度从「一整条请求」降到「一次 iteration（一步 forward）」。每跑完一步就重新决策：结束的请求立刻离开、空出的槽位立刻补进等待队列里的新请求。配套的 **selective batching** 解决了「不等长请求的 attention 没法塞进一个 batched matmul」的工程难题。这套东西在社区里通称 **continuous batching**，Anyscale 实测相对静态批处理吞吐提升 **8×**（叠加 PagedAttention 到 **23×**）。

vLLM 把它工程化，又在 v1（2025-01）做了一次彻底重写：**调度器里不再有「prefill 阶段」「decode 阶段」之分**。每条请求只携带两个数 `num_computed_tokens` vs `num_tokens`，调度器每一步发一个 `{request_id: num_scheduled_tokens}` 的 dict，受一个全局 `token_budget`（`max_num_batched_tokens`，默认 2048）约束。这个统一抽象一口气覆盖了 chunked prefill、prefix caching、speculative decoding。Sarathi-Serve（OSDI 2024）的 **chunked prefill + stall-free batching** 则解决了 prefill（compute-bound）和 decode（memory-bound）混在一个 batch 里互相干扰的问题——把长 prefill 切块，让 decode 永不被一个大 prefill 卡住。

本章按这条史料线推进，每个机制都配**真实 vLLM 源码逐行注解 + 一个纯 Python 可运行 demo** 印证。读完你应该能徒手写一个连续批处理调度器，并讲清 vLLM v1 的 `schedule()` 每一行在防什么坑。

## 1. 设计考古：从静态批处理到 iteration-level scheduling

### 1.1 静态批处理的浪费——掉队者占着茅坑

把多条请求凑一个 batch 一起算，是 GPU 利用率的基本盘：矩阵乘的算术强度随 batch size 上升，把 memory-bound 的 Decode 往 compute-bound 拉。问题在**怎么组批、什么时候换批**。

FasterTransformer、HuggingFace `generate()`、早期 Orca 之前的系统都用 **request-level（静态）批处理**：选 B 条请求组成一个 batch，一路 Decode 到**批内所有请求都生成完**，才释放、才组下一批。FriendliAI 的论文原话点破了它的死穴：

> "Requests that have finished earlier than other requests in a batch cannot return immediately to the client, while newly queued requests must wait to begin until the current batch completely finishes."
> ——FriendliAI, *Iteration Batching* blog（取材见文末）

两个浪费叠在一起：

1. **掉队者浪费（straggler / 早结束浪费）**：一个 batch 里，生成长度由最长那条决定。`gen_len = [4, 64, 6, 8]` 的 batch，那条 64 token 的请求没结束，其余三条早早 emit 完 EOS 却**不能离开**——它们的 batch 槽位还占着，每一步 forward 都在为它们做无意义的 padding 计算。Anyscale 的图说得很直白：「GPU is underutilized as generation lengths of different sequences in a batch differ from the largest generation length of the batch」。
2. **排队浪费（head-of-line blocking）**：新到的请求必须等当前 batch 整个排空才能开始，哪怕 batch 里已经有一半槽位在空转。

生成长度的方差有多大？同一个 chat 服务里，「1+1=?」生成 3 个 token，「写一篇 2000 字的文章」生成 2000 个 token，**差三个数量级**。静态批处理在这种负载下利用率掉到 20%–40%。本章第 3 节的 demo 会把这个数字实测出来。

### 1.2 Orca 的两个发明：iteration-level scheduling + selective batching

Orca（*Orca: A Distributed Serving System for Transformer-Based Generative Models*，Yu, Jeong, Kim, Kim, Chun，OSDI 2022，FriendliAI × Seoul National University）给出两个配套发明。

**发明一：iteration-level scheduling（迭代级调度）**。把调度粒度从「一整条请求」降到「一次模型 iteration」：

> "The scheduler invokes the execution engine to run only a single iteration of the model on the batch ... It continuously schedules a new request when a request in the processing batch completes and slots are available."
> ——Orca / FriendliAI research（取材见文末）

每跑完一步 forward，控制权回到调度器：结束的请求立刻 detokenize 返回给客户端、释放 KV；空出的槽位立刻从等待队列补一条新请求进来。batch 的**成员在每一步都可能变**——所以叫 continuous（连续）batching。掉队者浪费和排队浪费同时消失。这就是 Orca 论文对 NVIDIA FasterTransformer **36.9× 吞吐**（同等 latency 下）提升的主要来源。

**发明二：selective batching（选择性批处理）**。iteration-level scheduling 制造了一个工程难题：一个 batch 里的请求**长度各不相同**（有的在 Prefill 整段 prompt，有的在 Decode 第 500 个 token），它们的 attention 怎么塞进一个 batched matmul？

关键观察是：**Transformer 里大部分算子是 token-independent 的，唯独 attention 不是。**

> "Two input tensors must have a strict shape: `[b, n, m]` and `[b, m, p]` so that output can be a `[b, n, p]` shaped tensor ... non-attention operations like linear layers and GeLU are token-independent operations that can be batched regardless of sequence length."
> ——insujang, *LLM Inference: Continuous Batching and PagedAttention*（取材见文末）

- **QKV projection、MLP、LayerNorm、GeLU**：逐 token 独立。把整个 batch 所有请求的所有 token **flatten 成一个长序列** `[Σtokens, hidden]`，一次大 matmul 算完。请求边界在这里根本不重要。
- **Attention**：每个 token 只能 attend 到**自己那条序列**的历史 KV，不能跨请求。所以 attention 必须**按请求拆开**单独算（或用 PagedAttention kernel 靠 block table 寻址各自的 KV）。

「对非 attention 算子做 token-wise batching、对 attention 算子逐请求处理」——这就是 selective batching。它是 continuous batching 能落地的前提。第 2 章的 PagedAttention kernel 正是「逐请求 attention」的高效实现：靠 block table 让一个 kernel 同时服务多条不等长序列，避免真的写一个 Python for-loop。

## 2. 真实源码精读：vLLM v1 的统一调度循环

> 取材：`https://raw.githubusercontent.com/vllm-project/vllm/main/vllm/v1/core/sched/scheduler.py`，main@`a37b4a9`（latest release v0.23.0）。下面所有标【真实源码 vllm@v1/core/sched/scheduler.py】的代码块均逐字取自该文件，仅对与本章主线无关的分支（KV connector、encoder、mamba、spec decode 的细节）做了**删减并以 `# ...（略）` 标注**，保留的行一字未改。删减处不影响主线逻辑的完整性。

### 2.1 v1 的核心抽象：没有 prefill 阶段，没有 decode 阶段

vLLM v1（2025-01 alpha）相对 v0 最大的调度改动，官方博客原话：

> "It removes the traditional distinction between 'prefill' and 'decode' phases by treating user-given prompt tokens and model-generated output tokens uniformly. Scheduling decisions are represented as a simple dictionary, e.g., `{request_id: num_tokens}`, which specifies the number of tokens to process for each request at each step."
> ——vLLM Blog, *vLLM V1 Alpha*（取材见文末）

这个抽象写在 `schedule()` 方法开头的注释里，逐字如下：

```python
    def schedule(self) -> SchedulerOutput:
        self.current_step += 1
        # NOTE(woosuk) on the scheduling algorithm:
        # There's no "decoding phase" nor "prefill phase" in the scheduler.
        # Each request just has the num_computed_tokens and
        # num_tokens_with_spec. num_tokens_with_spec =
        # len(prompt_token_ids) + len(output_token_ids) + len(spec_token_ids).
        # At each step, the scheduler tries to assign tokens to the requests
        # so that each request's num_computed_tokens can catch up its
        # num_tokens_with_spec. This is general enough to cover
        # chunked prefills, prefix caching, speculative decoding,
        # and the "jump decoding" optimization in the future.
```
【真实源码 vllm@v1/core/sched/scheduler.py:357-368】

逐行拆这个心智模型：

- 每条请求只有两个关键量：`num_computed_tokens`（已经算进 KV cache 的 token 数）和 `num_tokens_with_spec = len(prompt) + len(output) + len(spec_tokens)`（这条序列「应该被算到」的 token 总数）。
- **调度的唯一目标**：每一步给每条请求分配一些 token，让它的 `num_computed_tokens` 去追平 `num_tokens_with_spec`。
- 这个定义下，「Prefill」不过是 `num_computed_tokens` 从 0 一路追到 `len(prompt)`（一次可能追一大段，也可能切块追）；「Decode」不过是每步 `num_computed_tokens` 追 1 个（因为每步只新增 1 个 output token）。**两者在调度器眼里是同一种操作**，只是 `num_new_tokens` 大小不同。
- chunked prefill = 把追平 prompt 的过程切成多步，每步追 `min(剩余, budget)` 个；spec decode = 一步追多个待验证 token。**全是同一个循环的特例。**

这是整章最重要的一句话：**理解了「num_computed_tokens 追平 num_tokens」，就理解了 vLLM v1 调度的全部。** 下面看这个循环怎么写。

### 2.2 schedule() 主循环（一）：先调度 RUNNING 请求

`schedule()` 分两段：先给已经在跑的 `self.running` 请求分配 token（保证在途请求继续前进），再用剩余预算从 `self.waiting` 队列补新请求。先看第一段的骨架：

```python
        scheduled_running_reqs: list[Request] = []
        preempted_reqs: list[Request] = []

        req_to_new_blocks: dict[str, KVCacheBlocks] = {}
        num_scheduled_tokens: dict[str, int] = {}
        token_budget = self.max_num_scheduled_tokens
        # ...（略：pause / encoder / spec decode 的预算初始化）

        # First, schedule the RUNNING requests.
        req_index = 0
        while req_index < len(self.running) and token_budget > 0:
            request = self.running[req_index]
            # ...（略：async scheduling / PP 的提前跳过分支）

            num_new_tokens = (
                request.num_tokens_with_spec
                + request.num_output_placeholders
                - request.num_computed_tokens
            )
            if 0 < self.scheduler_config.long_prefill_token_threshold < num_new_tokens:
                num_new_tokens = self.scheduler_config.long_prefill_token_threshold
            num_new_tokens = min(num_new_tokens, token_budget)
            # ...（略：encoder / mamba 对 num_new_tokens 的进一步收窄）

            if num_new_tokens == 0:
                # ...（略：各种「这步排不进」的原因）
                # NOTE(woosuk): Here, by doing `continue` instead of `break`,
                # we do not strictly follow the FCFS scheduling policy and
                # allow the lower-priority requests to be scheduled.
                req_index += 1
                continue
```

逐行注解关键点：

- `token_budget = self.max_num_scheduled_tokens`：这是**整步所有请求能分到的 token 总数上限**，默认等于 `max_num_batched_tokens=2048`（见 §4 配置）。它是连续批处理「一步喂多少 token 进 GPU」的总闸门。
- `num_new_tokens = num_tokens_with_spec - num_computed_tokens`：正是 §2.1 说的「还差多少才追平」。对一条正在 Decode 的请求，这个值通常是 1；对一条刚进来的长 prompt，这个值是几百几千。
- `if 0 < long_prefill_token_threshold < num_new_tokens: num_new_tokens = threshold`：**chunked prefill 的核心一行**。如果这条请求要追的 token 超过阈值，就只追一个 chunk。长 prompt 被切块，不会一口吃掉整个 budget。
- `num_new_tokens = min(num_new_tokens, token_budget)`：再被全局剩余预算夹一次。预算用光了，后面请求这步就排不进。
- `num_new_tokens == 0` 时用 `continue` 而不是 `break`：注释明说这**故意不严格遵守 FCFS**——这条排不进（比如 encoder budget 耗尽），允许后面优先级低的请求先排。这是个 throughput-over-strict-fairness 的工程取舍。

分到 token 之后要去要 KV block。这一步嵌着 v1 的**抢占逻辑**：

```python
            # Schedule newly needed KV blocks for the request.
            with record_function_or_nullcontext("schedule: allocate_slots"):
                while True:
                    new_blocks = self.kv_cache_manager.allocate_slots(
                        request,
                        num_new_tokens,
                        num_lookahead_tokens=self.num_lookahead_tokens,
                    )

                    if new_blocks is not None:
                        # The request can be scheduled.
                        break

                    # The request cannot be scheduled.
                    # Preempt the lowest-priority request.
                    if self.policy == SchedulingPolicy.PRIORITY:
                        preempted_req = max(
                            self.running,
                            key=lambda r: (r.priority, r.arrival_time),
                        )
                        self.running.remove(preempted_req)
                        # ...（略：priority 模式下回滚已调度状态）
                    else:
                        preempted_req = self.running.pop()

                    self._preempt_request(preempted_req, scheduled_timestamp)
                    preempted_reqs.append(preempted_req)
                    if preempted_req == request:
                        # No more request to preempt. Cannot schedule this request.
                        break

            if new_blocks is None:
                # Cannot schedule this request.
                break

            # Schedule the request.
            scheduled_running_reqs.append(request)
            request_id = request.request_id
            req_to_new_blocks[request_id] = new_blocks
            num_scheduled_tokens[request_id] = num_new_tokens

```

这段是调度与显存管理咬合的地方：

- `allocate_slots(...)` 返回 `None` = KV cache 没 block 了（第 2 章的 block pool 见底）。这时**必须抢占**才能给当前请求腾地方。
- 抢占谁？两种 policy：
  - **FCFS（默认）**：`self.running.pop()` —— 抢**队尾**那条（最晚加入 running 的，通常也是最「年轻」的）。这是个朴素但有效的启发：保护老请求，牺牲新请求。
  - **PRIORITY**：`max(self.running, key=lambda r: (r.priority, r.arrival_time))` —— 抢优先级数值最大（最低优先级）、同优先级里最晚到达的那条。
- `while True` 循环：抢一条不够就接着抢，直到 `allocate_slots` 成功，或者**连当前请求自己都被抢了**（`preempted_req == request`，说明 running 里没有比它更该牺牲的了），那就 `break` 放弃这步调度它。
- 排进去后三件事：记下要给它的 `new_blocks`、记下 `num_scheduled_tokens[id] = num_new_tokens`、`token_budget -= num_new_tokens`。预算被这条请求扣掉。

### 2.3 schedule() 主循环（二）：用剩余预算从 WAITING 补新请求

running 请求都安排完后，如果还有预算、还没抢占发生、还有空槽，就从等待队列补新请求。这段长，抽出主线：

```python
        # Next, schedule the WAITING requests.
        if not preempted_reqs and self._pause_state == PauseState.UNPAUSED:
            # ...（略：skipped_waiting 队列初始化）

                if len(self.running) == self.max_num_running_reqs:
                    break

                request_queue = self._select_waiting_queue_for_scheduling()
                request = request_queue.peek_request()
                request_id = request.request_id
                # ...（略：blocked-status / LoRA / KV-connector 的跳过分支）

                # Get already-cached tokens.
                if request.num_computed_tokens == 0:
                    # ...（略：local + external prefix cache 命中查询）
                    new_computed_blocks, num_new_local_computed_tokens = (
                        self.kv_cache_manager.get_computed_blocks(request)
                    )
                    num_computed_tokens = (
                        num_new_local_computed_tokens + num_external_computed_tokens
                    )
                # ...（略：KVTransfer 的 else 分支）

                # Number of tokens to be scheduled.
                num_new_tokens = request.num_tokens - num_computed_tokens

                # pooling requests to be chunked

                    not self.scheduler_config.enable_chunked_prefill
                    and num_new_tokens > token_budget
                ):
                    # If chunked_prefill is disabled,
                    # we can stop the scheduling here.
                    break

                num_new_tokens = min(num_new_tokens, token_budget)
                assert num_new_tokens > 0
```

注解：

- `if len(self.running) == self.max_num_running_reqs: break`：第二个闸门——**并发请求条数上限**（`max_num_seqs`，默认 128）。token budget 管「一步算多少 token」，max_num_seqs 管「同时有多少条请求在跑」。两个闸门一起决定 batch 形状。
- `get_computed_blocks(request)` → `num_new_local_computed_tokens`：**prefix caching 在调度层的入口**。新请求进来先查它的 prompt 有多少前缀已经在 cache 里（被别的请求算过）。命中的部分 `num_computed_tokens` 直接跳过，不用重算——这是第 2 章 block 共享在调度侧的兑现。
- `num_new_tokens = request.num_tokens - num_computed_tokens`：和 running 分支一模一样的「追平」公式，只是这里是新请求第一次追。
- chunked prefill 的两条岔路：
  - **开启（默认）**：长 prompt 被 `threshold` 切块、被 `token_budget` 夹住，**塞进这步剩余的预算里**，和正在 Decode 的请求共享同一步 forward。
  - **关闭**：`if num_new_tokens > token_budget: break` —— 这条 prompt 一步放不下就**直接停止本步调度**。它得等一个预算够大的空步来独占。这正是 §3.2 demo 里 decode 被 starve 的根源。

排进去后把请求挪进 running，状态机推进：

### 2.5 调度策略：FCFS 与 Priority 两种队列

§2.2 提到 policy 影响抢占谁，也影响从 waiting 队列取谁。两种队列的实现逐字如下，简洁到适合直接读：

```python
class FCFSRequestQueue(deque[Request], RequestQueue):
    """A first-come-first-served queue that supports deque operations."""

    def add_request(self, request: Request) -> None:
        """Add a request to the queue according to FCFS policy."""
        self.append(request)

    def pop_request(self) -> Request:
        """Pop a request from the queue according to FCFS policy."""
        return self.popleft()

    def prepend_request(self, request: Request) -> None:
        """Prepend a request to the front of the queue."""
        self.appendleft(request)
```

```python
class PriorityRequestQueue(RequestQueue):
    """
    A priority queue that supports heap operations.

    Respects the ordering defined in the Request class, where
    requests with a smaller value of `priority` are processed first.
    If multiple requests have the same priority, the one with the earlier
    `arrival_time` is processed first.
    """

    def __init__(self) -> None:
        self._heap: list[Request] = []

    def add_request(self, request: Request) -> None:
        """Add a request to the queue according to priority policy."""
        heapq.heappush(self._heap, request)

    def pop_request(self) -> Request:
        """Pop a request from the queue according to priority policy."""
        if not self._heap:
            raise IndexError("pop from empty heap")
        return heapq.heappop(self._heap)
```
【真实源码 vllm@v1/core/sched/request_queue.py:131-152，节选】

两个设计细节值得点出：

1. **FCFS 用 `deque`**，因为它需要 `prepend`（被抢占的请求 `appendleft` 回队头，下次最优先恢复——见 §2.6）。普通 list 做不到 O(1) 队头插入。
2. **Priority 用 `heapq`**，但它的 `prepend_request` 注释明说「优先级队列里没有 prepend 的概念」，实际还是按 `(priority, arrival_time)` 入堆。所以被抢占的请求在 priority 模式下**不保证优先恢复**，而是按它的优先级重新排队。这是 FCFS 和 Priority 在抢占恢复语义上的一个微妙差异。

vLLM 默认 `policy="fcfs"`。Priority 调度是为「付费用户优先」「交互式请求插队批处理任务」这类 SLO 场景准备的。对比之下 SGLang 走了另一条路（见 §5.3）。

### 2.6 抢占：v1 只剩 recompute，v0 还有 swap

KV cache 不够时怎么抢占，是调度器最体现「LLM 调度特殊性」的地方——因为 KV cache 是**不可压缩、且重算代价高**的资源。看 v1 的抢占实现：

```python
    def _preempt_request(self, request: Request, timestamp: float) -> None:
        """Preempt a request and put it back to the waiting queue.

        NOTE: The request should be popped from the running queue outside of this
        method.
        """
        assert request.status == RequestStatus.RUNNING, (
            "Only running requests can be preempted"
        )
        self.kv_cache_manager.free(request)
        self.encoder_cache_manager.free(request)
        self._inflight_prefills.discard(request)
        request.status = RequestStatus.PREEMPTED
        request.num_computed_tokens = 0
        if request.spec_token_ids:
            request.spec_token_ids = []
        request.num_preemptions += 1
        # ...（略：log）

        # Put the request back to the waiting queue.
        self.waiting.prepend_request(request)
```
【真实源码 vllm@v1/core/sched/scheduler.py:1040-1061，删减处已标注】

逐行：

- `self.kv_cache_manager.free(request)`：把这条请求的所有 KV block **全部还给 pool**（all-or-nothing 抢占，不存在抢一半 block）。
- `request.num_computed_tokens = 0`：**关键一行**。已经算好的 KV 被丢了，所以「已计算 token」清零——下次恢复时这条请求要**从头重算整段 prompt**（RECOMPUTE）。注意如果有 prefix cache，恢复时大概率能命中自己刚被释放的 block（如果还没被别人占用），所以重算不一定是全价。
- `self.waiting.prepend_request(request)`：放回 waiting **队头**（FCFS 下）。下一步调度它最优先恢复——这是抢占公平性的保证：被牺牲的请求不会饿死。
- `num_preemptions += 1`：记账。一条请求被反复抢占是个明确的过载信号（见 §6 生产坑）。

**v1 只有 RECOMPUTE 一种抢占模式。** 但 v0 不是这样。对照 v0.6.3 的 legacy scheduler（取材 `https://raw.githubusercontent.com/vllm-project/vllm/v0.6.3/vllm/core/scheduler.py`）：

两种抢占模式的本质区别：

| | RECOMPUTE | SWAP |
|---|---|---|
| 抢占时做什么 | 释放 GPU 上的 KV block，清空 `num_computed_tokens` | 把 KV block 从 GPU **拷到 CPU 内存**（swap space），保留状态 |
| 恢复时做什么 | 从头重算整段 prompt（一次 Prefill） | 把 KV block 从 CPU **拷回 GPU**，接着 Decode |
| 代价 | 重算 = 一次 prefill 的算力（compute） | 两次 PCIe 拷贝（GPU↔CPU 带宽） |
| 何时更优 | prompt 短、算力空闲 | prompt 长、重算太贵、CPU 内存够 |
| v0 默认 | 单序列请求默认用它（注释说 overhead 更低） | beam search 等多序列请求只能用它（重算不支持多序列） |

**为什么 v1 砍掉了 SWAP？** 这是个值得琢磨的工程决策（官方未在单一文档里逐条说明，以下是基于源码和社区讨论的推断，标【推断】）：

- 【推断】v1 默认开 chunked prefill + prefix caching。RECOMPUTE 时被释放的 block 进入 prefix cache，恢复时若未被覆盖可直接命中，「重算」常常退化成「重新命中缓存 + 算最后没命中的几块」，成本远低于 v0 时代的全量重算。SWAP 的「省下重算」收益被 prefix caching 大幅吃掉。
- 【推断】SWAP 引入 GPU↔CPU 拷贝、swap space 容量管理、`blocks_to_swap_in/out` 在 worker 间的同步，是相当一坨复杂度。v1 重写的主题正是「砍掉 v0 累积的复杂度、统一抽象」，SWAP 是被砍的对象之一。
- 代价是：在「长 prompt + 重算极贵 + 频繁抢占」的极端负载下，v1 的 RECOMPUTE 可能比 v0 的 SWAP 多烧算力。但这种负载下更该做的是扩容或上 P/D 分离，而不是靠 swap 续命。

这个「v0 有两种、v1 砍到一种」的演化，是「统一抽象 > 局部最优」工程哲学的典型样本。

### 3.1 Demo 1：静态批处理 vs 连续批处理

印证 §1.1（掉队者浪费）和 §2.2（iteration-level 调度让结束的请求立刻让位）。完整代码：

```python
#!/usr/bin/env python3
"""
Demo 1: static (request-level) batching vs continuous (iteration-level) batching.

每个 "iteration" = 一步 forward，把当前 running batch 里每条请求推进 1 个 token。
统计 GPU "slot-steps" = Σ_iterations (实际占用的槽位数)，对比静态批处理在哪浪费。
对应 vLLM v1 心智模型：请求只携带 num_computed vs num_total，调度器保持 running 满载。
"""
from collections import deque
from dataclasses import dataclass


@dataclass
class Req:
    rid: int
    prompt_len: int          # prefill tokens（此 demo 简化为 1 iteration）
    gen_len: int             # decode tokens to emit
    arrival: int             # arrival iteration（连续批处理用）
    produced: int = 0
    started_iter: int = -1
    finished_iter: int = -1

    @property
    def done(self) -> bool:
        return self.produced >= self.gen_len


def run_static(reqs, batch_size):
    """Request-level batching：组一个 batch，跑到全部结束，才换下一批。
    这是 FasterTransformer / naive HF generate 模型。假设 t=0 全部就绪（经典离线静态批）。"""
    pool = deque(sorted(reqs, key=lambda r: r.rid))
    it = 0
    slot_steps = 0           # 占用槽位累加
    capacity_steps = 0       # batch_size 累加（你真正付的账单）
    while pool:
        batch = [pool.popleft() for _ in range(min(batch_size, len(pool)))]
        for r in batch:
            r.started_iter = it
        # prefill：整个 batch 1 iteration
        capacity_steps += batch_size
        slot_steps += len(batch)
        it += 1
        # decode 到本批 *最长* 那条结束——其余请求只是占着槽位、不产出。
        max_gen = max(r.gen_len for r in batch)
        for step in range(max_gen):
            capacity_steps += batch_size            # 仍然按 batch_size 付费
            live = sum(1 for r in batch if r.produced < r.gen_len)
            slot_steps += live                      # 但只有 live 个槽位干活
            for r in batch:
                if r.produced < r.gen_len:
                    r.produced += 1
                    if r.done and r.finished_iter < 0:
                        r.finished_iter = it
            it += 1
    return _summarize("STATIC ", reqs, it, slot_steps, capacity_steps)


def run_continuous(reqs, batch_size, max_token_budget):
    """Iteration-level (continuous) batching，vLLM v1 风格。
    每步：1) 调度 running（decode），每条耗 1 token 预算；
          2) 用剩余槽位 + 预算从 waiting 补新请求（admit + 1 prefill step）；
          3) 跑一步 forward；结束的请求离开，下一步释放槽位。
    请求可以随时间到达，中途加入。"""
    by_arrival = sorted(reqs, key=lambda r: (r.arrival, r.rid))
    waiting = deque()
    running = []
    it = 0
    next_idx = 0
    slot_steps = 0
    capacity_steps = 0

    def all_admitted_or_running():
        return next_idx >= len(by_arrival) and not waiting and not running

    while not all_admitted_or_running():
        # 新到达的请求进 waiting 队列（FCFS）
        while next_idx < len(by_arrival) and by_arrival[next_idx].arrival <= it:
            waiting.append(by_arrival[next_idx])
            next_idx += 1

        # ---- 这是 scheduler.schedule() 核心的简化 ----
        token_budget = max_token_budget
        # 1. running 请求（decode）：每条 1 token
        for r in running:
            if token_budget <= 0:
                break
            token_budget -= 1            # decode = 1 scheduled token
        # 2. 回填：有空槽且有预算就从 waiting 拉
        while waiting and len(running) < batch_size and token_budget > 0:
            r = waiting.popleft()
            r.started_iter = it
            running.append(r)
            token_budget -= 1            # (toy) prefill 记为 1 token
        # ------------------------------------------------

        if not running:
            it += 1
            continue

        capacity_steps += batch_size
        slot_steps += len(running)       # 连续批处理把这个值顶到接近满载

        for r in running:
            r.produced += 1
            if r.done and r.finished_iter < 0:
                r.finished_iter = it
        # 结束的请求释放槽位给下一步
        running = [r for r in running if not r.done]
        it += 1

    return _summarize("CONTINUOUS", reqs, it, slot_steps, capacity_steps)


def _summarize(name, reqs, total_iters, slot_steps, capacity_steps):
    util = slot_steps / capacity_steps if capacity_steps else 0.0
    avg_latency = sum(r.finished_iter - r.arrival for r in reqs) / len(reqs)
    return {
        "name": name, "total_iters": total_iters, "slot_steps": slot_steps,
        "capacity_steps": capacity_steps, "utilization": util,
        "avg_latency_iters": avg_latency,
    }


def make_workload():
    # 8 条请求，生成长度高度倾斜——静态批处理最坏情况（一条 straggler 绑架整批）。
    specs = [
        # rid, prompt_len, gen_len, arrival
        (0, 10, 4,  0),
        (1, 10, 64, 0),   # straggler
        (2, 10, 6,  0),
        (3, 10, 8,  0),
        (4, 10, 5,  1),
        (5, 10, 60, 2),   # straggler
        (6, 10, 7,  3),
        (7, 10, 6,  4),
    ]
    return [Req(*s) for s in specs]


def fresh(reqs):
    return [Req(r.rid, r.prompt_len, r.gen_len, r.arrival) for r in reqs]


def main():
    base = make_workload()
    BATCH = 4
    BUDGET = 8  # max_num_batched_tokens-ish (toy)
    s = run_static(fresh(base), BATCH)
    c = run_continuous(fresh(base), BATCH, BUDGET)
    hdr = f"{'scheduler':<12}{'iters':>7}{'slot_steps':>12}{'cap_steps':>11}{'util':>8}{'avg_lat':>9}"
    print(hdr); print("-" * len(hdr))
    for r in (s, c):
        print(f"{r['name']:<12}{r['total_iters']:>7}{r['slot_steps']:>12}"
              f"{r['capacity_steps']:>11}{r['utilization']*100:>7.1f}%"
              f"{r['avg_latency_iters']:>9.1f}")
    print()
    print(f"continuous finishes the same work in {s['total_iters']/c['total_iters']:.2f}x fewer iterations")
    print(f"GPU slot utilization improves {c['utilization']/s['utilization']:.2f}x "
          f"({s['utilization']*100:.0f}% -> {c['utilization']*100:.0f}%)")
    print(f"avg request latency: static {s['avg_latency_iters']:.1f} iters "
          f"-> continuous {c['avg_latency_iters']:.1f} iters")


if __name__ == "__main__":
    main()
```

- 静态批处理跑 126 步：第一批 `[0,1,2,3]` 要等 `rid=1`（gen_len=64）跑完，整整 64 步里有大量步只有 1 条请求在产出；连续批处理 66 步，因为 `rid=1` 在跑的同时，结束的 `rid=0/2/3` 槽位立刻被 `rid=4/6/7` 填上了。**这正是 `running = [r for r in running if not r.done]` + 「下一步从 waiting 回填」两行代码的效果**，对应 §2.2-2.3 的源码。
- latency 从 51.2 降到 21.1 iters（2.4×）：连续批处理下新请求不用等整批排空，对应 §1.1 的「排队浪费」消失。
- 这个 demo 的吞吐增益（1.91×）比 Anyscale 报的 8× 小——因为它只有 8 条请求、batch_size 只有 4，且简化了 prefill。真实负载里请求更多、长度方差更大、batch 更宽，掉队者浪费的占比会高得多，增益随之放大。**demo 的价值是把机制看清，不是复现绝对数字。**

### 3.2 Demo 2：vLLM v1 token-budget 循环 + chunked prefill

印证 §2.1-2.3 的「num_computed 追平 num_tokens」统一循环，和 §2.3 chunked prefill 开/关的差异——即 Sarathi-Serve 的核心。完整代码：

```python
#!/usr/bin/env python3
"""
Demo 2: vLLM v1 统一 token-budget 调度器 + chunked prefill。

复刻 vllm/v1/core/sched/scheduler.py::schedule() 的核心：
  - 没有 prefill/decode 之分。每条请求只知道 num_computed_tokens vs num_total_tokens。
  - 每步发一个 {req_id: num_scheduled_tokens} dict，受全局 token_budget(=max_num_batched_tokens) 约束。
  - 长 prefill 被 *切块* 跨步进行，永不阻塞 decode。

对比：
  (A) chunked prefill ON  -> 1000-token prompt 被切，decode 持续流动
  (B) chunked prefill OFF -> 大 prefill 独占若干步，starve 掉 decode
并报告 decode 被饿死的步数（Sarathi-Serve 攻击的指标）。
"""
from dataclasses import dataclass


@dataclass
class Req:
    rid: str
    num_total: int            # 此 demo 用 prompt 长度建模 prefill 成本
    num_computed: int = 0
    gen_target: int = 8       # prefill 完成后要 emit 的 decode token 数
    generated: int = 0

    @property
    def prefilling(self):
        return self.num_computed < self.num_total

    @property
    def done(self):
        return (not self.prefilling) and self.generated >= self.gen_target


def schedule_step(running, token_budget, long_prefill_threshold, chunked):
    """一步 vLLM 风格循环。返回 {rid: scheduled_tokens}。"""
    out = {}
    budget = token_budget
    for r in running:
        if budget <= 0:
            break
        if r.prefilling:
            num_new = r.num_total - r.num_computed          # 剩余 prompt
            if chunked:
                if long_prefill_threshold and num_new > long_prefill_threshold:
                    num_new = long_prefill_threshold        # 切块上限
                num_new = min(num_new, budget)              # 夹进全局预算
            else:
                # 不切块：放不下也得占满整个预算（建模「prefill 独占一步」的失败模式）
                if num_new > budget:
                    num_new = budget
            if num_new <= 0:
                continue
            out[r.rid] = num_new
            budget -= num_new
        else:
            # decode：恰好 1 个 token
            if budget >= 1:
                out[r.rid] = 1
                budget -= 1
    return out


def apply(running, sched):
    for r in running:
        n = sched.get(r.rid, 0)
        if n == 0:
            continue
        if r.prefilling:
            r.num_computed += n
        else:
            r.generated += 1
    return [r for r in running if not r.done]


def simulate(chunked, token_budget=256, long_prefill_threshold=128):
    # 一个巨大 prompt + 几条已经在跑的短 decode 请求
    running = [
        Req("BIG",  num_total=1000, gen_target=4),
        Req("d1",   num_total=8,    gen_target=12),
        Req("d2",   num_total=8,    gen_target=12),
        Req("d3",   num_total=8,    gen_target=12),
    ]
    for r in running[1:]:        # 预先 prefill 短请求，让它们从 step 0 就在 decode
        r.num_computed = r.num_total
    step = 0
    decode_served_per_step = []
    while running:
        sched = schedule_step(running, token_budget, long_prefill_threshold, chunked)
        if not sched:
            break
        decodes = sum(1 for rid in sched if any(
            r.rid == rid and not r.prefilling for r in running))
        decode_served_per_step.append(decodes)
        running = apply(running, sched)
        step += 1
    return step, decode_served_per_step


def count_starved_steps(decodes_per_step, expected_decodes_early):
    # 前几步 3 条 decode 都活着；健康调度器每步推进全部 3 条。数低于预期的步。
    return sum(1 for d in decodes_per_step[:6] if d < expected_decodes_early)


def main():
    BUDGET = 256
    THRESH = 128  # 切块上限（long_prefill_token_threshold 类比）
    # 注意：切块上限 < 预算是故意的——那段剩余预算正是让 decode 和 prefill chunk
    # 同步在一步里跑的关键。这就是 Sarathi-Serve "stall-free batching" 的全部要点。
    steps_on, dps_on = simulate(chunked=True,  token_budget=BUDGET, long_prefill_threshold=THRESH)
    steps_off, dps_off = simulate(chunked=False, token_budget=BUDGET, long_prefill_threshold=THRESH)
    print(f"token_budget (max_num_batched_tokens) = {BUDGET}, chunk cap = {THRESH}\n")
    print(f"chunked prefill ON : finished in {steps_on} steps")
    print(f"  decode tokens advanced per step (first 8): {dps_on[:8]}")
    print(f"chunked prefill OFF: finished in {steps_off} steps")
    print(f"  decode tokens advanced per step (first 8): {dps_off[:8]}\n")
    print(f"decode-starved steps (of first 6) -- ON: {count_starved_steps(dps_on, 3)}, "
          f"OFF: {count_starved_steps(dps_off, 3)}")
    print(f"interpretation: ON splits the 1000-tok prefill into "
          f"{THRESH}-tok chunks, leaving {BUDGET - THRESH} budget/step for decodes,")
    print("so the 3 short requests advance EVERY step (3,3,3,...). OFF lets the big")
    print(f"prefill grab the full {BUDGET}-tok budget for several steps, starving decodes (0,0,0).")


if __name__ == "__main__":
    main()
```

**运行步骤**：存为 `chunked_demo.py`，执行 `python3 chunked_demo.py`。

**真实输出**（作者机器 CPython 3.12.12）：

```
token_budget (max_num_batched_tokens) = 256, chunk cap = 128

chunked prefill ON : finished in 12 steps
  decode tokens advanced per step (first 8): [3, 3, 3, 3, 3, 3, 3, 3]
chunked prefill OFF: finished in 15 steps
  decode tokens advanced per step (first 8): [0, 0, 0, 3, 4, 4, 4, 4]

decode-starved steps (of first 6) -- ON: 0, OFF: 3
interpretation: ON splits the 1000-tok prefill into 128-tok chunks, leaving 128 budget/step for decodes,
so the 3 short requests advance EVERY step (3,3,3,...). OFF lets the big
prefill grab the full 256-tok budget for several steps, starving decodes (0,0,0).
```

**怎么读，怎么和源码 + Sarathi-Serve 对上**：

- **chunked ON**：`decode tokens per step = [3,3,3,...]`，三条短请求**每一步都在产出**，0 个饿死步。机制：`BIG` 这条每步被 `long_prefill_threshold=128` 切成 128-token 的 chunk（对应 §2.2 源码 `if 0 < long_prefill_token_threshold < num_new_tokens: num_new_tokens = threshold`），吃掉 256 预算里的 128，**剩下的 128 预算正好喂三条 decode 请求**。prefill 和 decode 在同一步 forward 里共存。
- **chunked OFF**：`[0,0,0,3,4,...]`，前三步 decode **颗粒无收**。机制：不切块时 `BIG` 每步把整个 256 预算占满（在真实 vLLM 里是「放不下就 `break`、独占一个空步」），三条 decode 请求被排在 `BIG` 之后、预算已耗尽，只能等。直到第 4 步 `BIG` 终于追平 1000，decode 才开始流动。
- `decode-starved steps: ON 0 / OFF 3` —— 这个 3 就是 Sarathi-Serve 论文里 **generation stall** 的量化。论文原话：「prefill iterations have high latency but saturate GPU compute ... decode iterations have low latency but also low compute utilization」，混在一起时大 prefill 会卡死 decode 的 TBT（time-between-tokens）。chunked prefill 用「切块 + 让 decode 搭便车」消除这个 stall。
- 这正是为什么 vLLM v1 **默认 `enable_chunked_prefill=True`**（§4）。在交互式负载里，decode 的 TBT 直接决定用户体感的「打字速度」，3 步的 stall 就是肉眼可见的卡顿。

两个 demo 合起来，把本章源码的两个核心机制——iteration-level 让位、token-budget 切块——都跑通印证了。

【真实源码 vllm@v1 config/scheduler.py:42-109，节选】

四个最该懂的旋钮：

| 旋钮 | 默认 | 管什么 | 调大的代价 / 调小的代价 |
|---|---|---|---|
| `max_num_batched_tokens` | 2048 | 一步 forward 处理的 token 总预算（§2.2 的 `token_budget`） | 调大：单步吞吐高、prefill 更快，但单步 latency 高、显存峰值高。调小：TBT 稳但吞吐低 |
| `max_num_seqs` | 128 | 同时 running 的最大请求条数（§2.3 的并发闸门） | 调大：吞吐高但每条 KV cache 摊到的显存少、更易触发抢占。调小：抢占少但 GPU 喂不饱 |
| `enable_chunked_prefill` | `True` | 是否切长 prefill（§3.2 印证） | 关掉：长 prompt 独占步、starve decode（TBT 抖动）。开着：TBT 平稳，但 prefill 整体延迟略增 |
| `policy` | `"fcfs"` | 调度/抢占顺序（§2.5） | `"priority"`：支持 SLO 分级，但实现上被抢占请求不保证优先恢复 |

`long_prefill_token_threshold` 默认 0（不额外切，只受 `max_num_batched_tokens` 约束）；只有当 `max_num_partial_prefills > 1`（允许多条 prefill 并发切块）时才会被自动设为 `max_model_len * 0.04`（源码 `config/scheduler.py:252-254`）。

**调参直觉**：`max_num_batched_tokens` 是吞吐/延迟的总开关。离线批处理（追吞吐）往大调（8192、16384);在线交互（追 TBT 稳）保持 2048 左右并依赖 chunked prefill 把长 prefill 摊平。这两个旋钮 + chunked prefill 三者联动，是生产调优的主战场。

## 5. 方案对比

### 5.1 三种批处理范式

| 维度 | 静态批处理（FasterTransformer / HF generate） | 连续批处理（Orca / vLLM v0） | 统一 token-budget（vLLM v1） |
|---|---|---|---|
| 调度粒度 | 整条请求（request-level） | 一次 iteration | 一次 iteration |
| prefill/decode | 分阶段，且批内同阶段 | 分阶段，但批内可混（selective batching） | **无阶段之分**，统一成 token 追平 |
| 掉队者浪费 | 严重（§3.1 量化 33% 利用率） | 消除 | 消除 |
| 长 prefill 阻塞 decode | N/A（不混批） | 存在（除非手动 chunk） | chunked prefill 默认解决 |
| 调度决策表示 | batch 对象 | 多个 `SequenceGroupMetadata` + swap 映射 | 一个 `{req_id: num_tokens}` dict |
| 抢占模式 | 无（batch 跑完才换） | RECOMPUTE + SWAP | RECOMPUTE only |
| 吞吐基准 | 1× | ~8×（Anyscale，naive CB）/ 23×（+PagedAttention） | 1.7× vs v0 |

### 5.2 一个并发负载下跑一遍

设想：稳态 100 条并发请求，prompt 平均 512 token，生成长度服从重尾分布（中位数 50、p99 2000），`max_num_seqs=128`、`max_num_batched_tokens=2048`。

- **静态批处理**：每批最多 128 条，但要等批内 p99（2000 token）那条跑完。那条跑 2000 步期间，中位数请求（50 步就完）早早结束、槽位空转 ~1950 步。利用率 ≈ 中位/最大 ≈ 50/2000，**理论低到 2.5%**（真实因 batch 内有多条长请求会缓和，但仍是个位数到 ~20%）。
- **连续批处理 + chunked prefill（v1）**：每步用 2048 token 预算先喂 ~100 条 running 的 decode（每条 1 token，耗 100 预算），剩 ~1948 预算切给 1–2 条新进来的 512-token prefill（切成 ~4 个 chunk 摊几步）。**running 槽位始终接近满载，decode 的 TBT 不被长 prefill 顶飞**。利用率 80%+。
- **不适用边界**：当 `max_num_batched_tokens` 设得过小（< 单条 prompt 且没开 chunked prefill），v1 会直接报错（§2.3 的 `verify_max_model_len`，源码 `config/scheduler.py:267`）——因为一条 prompt 一步都放不下又不许切，调度器会死锁。这是个真实的配置陷阱。

### 5.3 跨引擎：vLLM vs SGLang 的调度哲学

vLLM v1 默认 **FCFS**，调度顺序与 prefix cache 无关。SGLang 走了不同的路（取材 `https://raw.githubusercontent.com/sgl-project/sglang/main/python/sglang/srt/managers/schedule_policy.py`）——它把调度策略分成两类（以下基于该文件 WebFetch 摘要，标【取材 SGLang schedule_policy.py】）：

- **CacheAwarePolicy**：`LPM`（longest prefix match，按请求与 RadixAttention tree cache 的最长前缀匹配长度**降序**排，让能复用 cache 的请求先跑）、`DFS_WEIGHT`（按 cache tree 的 DFS 权重排）。
- **CacheAgnosticPolicy**：`FCFS`、`LOF`（longest output first）、`RANDOM`、`ROUTING_KEY`。

| | vLLM v1 | SGLang |
|---|---|---|
| 默认策略 | FCFS（cache-agnostic） | 取决于配置，LPM 是其招牌（cache-aware） |
| prefix cache 与调度的关系 | 解耦：先 FCFS 定序，再在 `get_computed_blocks` 查命中 | 耦合：LPM 直接**按 cache 命中长度排序调度顺序**，主动制造命中 |
| 适合 | 通用、公平、可预测 | 大量共享前缀的负载（agent、few-shot、同 system prompt 多请求） |

这是两种合理但不同的取舍：vLLM 把「公平/可预测」放前面，cache 命中是被动收益;SGLang 把「最大化 cache 复用」放前面，主动调度顺序去吃命中。理解这个差异，比记住任何一边的 API 更重要——它决定了你的负载该选哪个引擎。

## 6. 扎根：失败模式、生产坑、底层根因

**坑 1：抢占风暴（preemption thrashing）。**
症状：吞吐断崖、TBT 剧烈抖动，日志刷 `Sequence group ... is preempted`（v0 的 warning，每 50 次抢占打一条）。根因：`max_num_seqs` 设太大，running 请求数 × 各自 KV cache > 显存，于是调度器不停抢占-恢复-再抢占，每次 RECOMPUTE 都重烧一次 prefill 算力。§2.6 的 `num_preemptions += 1` 就是给这个计数的。**根因不在调度器，在显存预算**：要么调小 `max_num_seqs`/`gpu_memory_utilization` 留够 KV cache，要么上 P/D 分离。光改调度策略治标不治本。

**坑 2：长 prefill 顶飞 TBT（关掉 chunked prefill 时）。**
症状：平时 TBT 30ms，偶尔某个请求卡 2s。根因：一条超长 prompt（比如塞了整本文档的 RAG 请求）进来，在没切块时独占好几步 forward，期间所有正在 Decode 的请求**全被冻住**（§3.2 demo 的 `[0,0,0]`）。这是 Sarathi-Serve 论文的核心攻击点。解法就是 §4 的 `enable_chunked_prefill=True`（v1 默认开）。如果你的引擎版本或配置关了它，长 prompt 负载下 TBT p99 会很难看。

**坑 3：`max_num_batched_tokens` 与 `max_model_len` 配置死锁。**
症状：启动直接抛 `max_num_batched_tokens (X) is smaller than max_model_len (Y)`。根因：§5.2 提到的——一条 prompt 一步放不下，又没开 chunked prefill，调度器无法推进。解法：要么开 chunked prefill，要么把 `max_num_batched_tokens` 调到 ≥ `max_model_len`。这是 vLLM 主动用 `verify_max_model_len`（源码 `config/scheduler.py:267-279`）在启动时拦下的，避免你运行时才发现死锁。

**坑 4：FCFS 下的隐性饥饿（`continue` not `break`）。**
§2.2 那段注释 `we do not strictly follow the FCFS scheduling policy` 是把双刃剑。当某条队头请求因为 encoder budget / KV 不足反复排不进（`num_new_tokens == 0` → `continue`），调度器会跳过它去调度后面的请求。好处是吞吐不被一条卡住的请求拖死;隐患是这条请求可能**长时间得不到调度**。生产里如果看到个别请求 TTFT（time-to-first-token）异常长，要怀疑它在队头反复被 `continue` 跳过。

**底层根因（贯穿四个坑）**：LLM 调度的特殊性在于 **KV cache 是不可压缩、长度未知、重算昂贵的资源**。经典 OS 调度可以靠时间片轮转 + 内存换页平滑处理；LLM 里「换页」（SWAP）贵、「时间片」粒度是一整步 forward、「内存需求」随生成长度动态增长且不可预知。所以 LLM 调度器的所有复杂度，本质都是在「不知道请求要跑多久、不知道它要吃多少显存」这个根本不确定性下，做 admission control 和 preemption。记住这一句，前面所有机制都能自洽推出来。

## 7. 面试视角

被问到连续批处理/调度，高频问题与踩分点：

1. **「static batching 为什么慢，continuous batching 快在哪？」** 踩分点：说出掉队者浪费（早结束请求占槽空转）+ 排队浪费（新请求等整批排空）两个，而不只是「利用率高」。能引 Orca 的 iteration-level scheduling 和 Anyscale 23× 加分。
2. **「continuous batching 里不等长请求的 attention 怎么 batch？」** 踩分点：selective batching——非 attention 算子（QKV/MLP）token-wise flatten 成一个大 matmul，attention 逐请求算（PagedAttention kernel 靠 block table 高效实现）。能说清「为什么唯独 attention 不能跨请求」（每 token 只 attend 自己序列的 KV）是关键。
3. **「vLLM v1 的调度器和 v0 最大区别？」** 踩分点：取消 prefill/decode 阶段之分，统一成「num_computed_tokens 追平 num_tokens」+ `{req_id: num_tokens}` dict。能补一句「这个抽象同时覆盖 chunked prefill / prefix caching / spec decode」说明你真读过。
4. **「prefill 和 decode 混在一个 batch 有什么问题，怎么解？」** 踩分点：prefill compute-bound、decode memory-bound，混批时大 prefill 会顶飞 decode 的 TBT（generation stall）。解法 chunked prefill + stall-free batching（Sarathi-Serve），把长 prefill 切块让 decode 搭便车。
5. **「KV cache 不够了调度器怎么办？两种抢占模式区别？」** 踩分点：抢占（preemption）。RECOMPUTE（丢 block、`num_computed=0`、恢复时重算）vs SWAP（拷 CPU、恢复时拷回）。能说出「v1 砍掉了 SWAP 只留 RECOMPUTE，因为 prefix caching 让重算变便宜了」是高阶加分。
6. **「max_num_batched_tokens 和 max_num_seqs 分别管什么？」** 踩分点：前者管一步的 token 总预算（吞吐/延迟开关），后者管并发请求条数（喂饱 GPU vs 触发抢占）。能讲清两者联动 + 调参方向（离线调大 batched_tokens / 在线靠 chunked prefill 稳 TBT）。

---

## 8. 未来与开放问题

- **P/D 分离（disaggregated prefill/decode）**：本章的 chunked prefill 是「在同一 GPU 上调和 P/D 干扰」；Mooncake（*KVCache-centric Disaggregated Architecture*）、DistServe 等走另一条路——把 prefill 和 decode 拆到**不同 GPU 集群**，各自按自己的算力特性优化，靠高速网络传 KV cache。Mooncake 实测真实负载下多处理 **75% 请求**。调度器从此要跨集群协调，是后续章的主题。
- **更聪明的 admission control**：当前 FCFS / priority 都不预测请求的生成长度。若能预测「这条大概生成 50 还是 2000 token」，就能做 SJF（shortest-job-first）类调度降低平均延迟。已有工作用一个小模型预测输出长度来排序，但预测误差和公平性是开放问题。
- **chunk 大小的自适应**：`long_prefill_token_threshold` 现在是个静态配置。理想情况下 chunk 大小应随当前 running 的 decode 负载动态调整（decode 多就把 chunk 切小让出预算，decode 少就切大冲 prefill 吞吐）。Sarathi 后续工作和 vLLM 的 `max_num_partial_prefills` 是这个方向的起步。
- **spec decode 与调度的耦合**：投机解码一步要塞多个待验证 token（`num_tokens_with_spec`），被拒绝的 token 又要回滚 `num_computed_tokens`。这让 token-budget 的会计变复杂（§2.4 已埋了 `spec_token_ids` 的伏笔），是下一章的内容。
