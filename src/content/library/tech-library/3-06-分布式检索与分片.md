## 6.1 设计考古：分布式检索从何而来

### 6.1.1 单机瓶颈的三条路

2008 年前后，Lucene 单机已能支撑千万文档级别的倒排检索。真正让人痛苦的不是算法，而是三条硬墙：

1. **内存墙**：向量索引（HNSW graph）必须全量驻内存，1 亿 × 768 维 float32 = 300 GB，单机 DRAM 装不下。
2. **吞吐墙**：单机线程数有上限，QPS scale-up 到某点后边际收益递减。
3. **延迟墙**：顺序扫描全量数据延迟随数据量线性增长，分片后变为并行。

Elasticsearch（2010）最早将 Lucene 的多 segment 模型投射到网络层，称为 shard。每个 shard 是一个独立的 Lucene 索引，primary shard 负责写，replica shard 负责读扩展。

向量库场景更晚，Milvus 2.0（2021）引入 segment + collection + channel 三层分片模型，专为向量场景设计。

### 6.1.2 论文谱系

| 工作 | 年份 | 贡献 |
|------|------|------|
| Google Bigtable | 2006 | tablet 分片 + range partition 基础概念 |
| Elasticsearch Sharding | 2010 | Lucene shard 化，scatter-gather |
| FAISS + Sharding（Facebook）| 2017 | IVF 分片感知：coarse quantizer 广播 → shard 局部 scan |
| Milvus 2.0 (Zilliz) | 2021 | segment-level 精细调度，growing/sealed 两态，log-based WAL |
| DiskANN Sharding | 2023 | 磁盘向量图的分片策略，beam search 感知 |

## 6.2 分片模型：数据如何切割

### 6.2.1 两种分片策略

```
策略 A：Range Partition（范围分片）
  ┌──────────────┬──────────────┬──────────────┐
  │  Shard 0     │  Shard 1     │  Shard 2     │
  │  id 0–33M    │  id 33–66M   │  id 66–100M  │
  └──────────────┴──────────────┴──────────────┘
  优点：路由可精确；适合按时间/ID范围的过滤查询
  缺点：热点风险（写入时序导致新shard过热）

策略 B：Hash Partition（哈希分片）
  hash(primary_key) % num_shards → shard_id
  优点：写入均匀；无热点
  缺点：范围查询必须广播；向量检索本就无路由能力，影响不大
```

**向量检索无法路由**：ANN 的本质是 "全局最近邻"，无法事先知道答案在哪个 shard，因此向量检索天生需要广播到所有 shard。这是向量分布式与倒排最根本的区别。

倒排检索有 term 路由：`WHERE term="lucene"` 只需访问包含该 posting list 的 shard。

### 6.2.2 Milvus 的三层分片模型

Milvus 2.x 使用三个粒度概念（可通过 GitHub milvus-io/milvus 源码目录 `internal/` 确认架构）：

```
Collection（逻辑表）
  └── Partition（逻辑分区，用户自定义，如 partition_by="date"）
       └── Segment（物理存储单元，sealed 或 growing）
            ├── 向量数据（binlog）
            ├── scalar 列（binlog）
            └── index（HNSW/IVF graph files）
```

**Segment 两态**：
- `Growing segment`：当前正在接收写入，只有内存中的 buffer，无索引
- `Sealed segment`：写满（默认 512 MB）后被封闭，触发异步建索引，之后只读

这个设计解决了「写入时如何维护向量图」的难题——HNSW 增量插入极慢，Milvus 的做法是先写 growing（暴力扫），seal 后批量建图。

```
【Milvus QueryCoord balance.go 核心摘录，真实源码 milvus-io/milvus@internal/querycoordv2/balance/balance.go】

type Balance interface {
    BalanceReplica(ctx context.Context, replica *meta.Replica)
        ([]assign.SegmentAssignPlan, []assign.ChannelAssignPlan)
    GetAssignPolicy() assign.AssignPolicy
}

// RoundRobinBalancer 计数均衡核心逻辑（注释为源码语义）：
// 1. ceil(总segment数 / 节点数) = 平均值
// 2. 标记低于平均值的节点 → 迁移超额节点的 segment 过去
// 3. 注意：不考虑行数/内存，只按 segment 数均衡（已知局限）
```

### 6.2.3 Lucene 的 LeafSlice 模型

Lucene 在单机内部也有类似分片的概念：`LeafReaderContext`（每个 segment 对应一个 leaf），`IndexSearcher` 将多个 leaf 打包成 `LeafSlice` 分配给线程池并行执行。

```java
【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/IndexSearcher.java】

// 获取并行切片
LeafSlice[] leafSlices = getSlices();

// 并行执行各切片
for (LeafSlice slice : leafSlices) {
    taskExecutor.invokeAll(() -> search(slice.partitions, weight, collector));
}

// 汇总结果
return collectorManager.reduce(results);

// LeafSlice 结构：
public static class LeafSlice {
    public final LeafReaderContextPartition[] partitions;
    // 每个 partition 有 minDocId/maxDocId 范围，支持 segment 内部再切分
    public static final class LeafReaderContextPartition {
        public final int minDocId;
        public final int maxDocId;
        public final LeafReaderContext ctx;
    }
}
```

这套设计的精妙之处：网络层的多机 shard 和单机内的多 segment 在逻辑上是同构的——都是 scatter-gather，只是通信介质不同。

## 6.3 Scatter-Gather 检索协议

### 6.3.1 协议全流程

```
Client
  │  query: (vector q, top_k=10, ef=100)
  ▼
Coordinator / Router
  │  broadcast: SearchRequest{q, top_k=10, ef=100}
  ├──────────────────────────────────────────────────┐
  ▼                    ▼                    ▼        ▼
Shard 0              Shard 1             Shard 2   Shard 3
local top-10         local top-10        ...       ...
(ef=100 local)       (ef=100 local)
  │                    │
  └──────────┬─────────┘
             ▼
        Coordinator
        merge: 4 × top-10 → global top-10
        (k-way merge via min-heap, O(N·logK))
             │
             ▼
          Client
        final top-10
```

**关键：每个 shard 返回 top-k，不是 top-1**。Coordinator 做 k-way merge，取全局 top-k。

### 6.3.2 精度损失的根源：局部 top-k 不够大

假设真实 global top-10 中，有 3 个向量恰好落在 Shard 2，但 Shard 2 的 HNSW 搜索（ef=100）因为图连通性问题漏掉了 1 个，只返回了 2 个。这个向量就永远丢失了，无法在 coordinator 阶段找回。

这是分布式向量检索的根本精度损失来源：**shard 内的局部搜索不完美，coordinator 无法弥补**。

解决方案：
1. **加大 shard 内 ef**（每 shard 返回更多候选），提升局部召回，代价是延迟上升
2. **减少 shard 数**（数据集中），减少边界效应，代价是单 shard 内存压力
3. **图感知分片**（Graph-Aware Partitioning）：让 HNSW 图的高密度区域在同一 shard，边界节点冗余存储（实验性方向）

### 6.3.3 Lucene TopDocs.merge：k-way 归并实现

### 6.3.4 向量检索的 TopKnnCollector

```java
【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/TopKnnCollector.java】

// 单 shard 内 ANN 收集器，最小堆维护 top-k
@Override
public boolean collect(int docId, float similarity) {
    return queue.insertWithOverflow(docId, similarity);  // 堆满时自动淘汰最小
}

// 剪枝阈值：堆满后返回当前最小相似度，用于提前终止图搜索
@Override
public float minCompetitiveSimilarity() {
    return queue.size() >= k() ? queue.topScore() : Float.NEGATIVE_INFINITY;
}

// 输出 TopDocs（结果倒序弹出）
@Override
public TopDocs topDocs() {
    ScoreDoc[] scoreDocs = new ScoreDoc[queue.size()];
    for (int i = 1; i <= scoreDocs.length; i++) {
        scoreDocs[scoreDocs.length - i] = new ScoreDoc(queue.topNode(), queue.topScore());
        queue.pop();
    }
    TotalHits.Relation relation = earlyTerminated()
        ? TotalHits.Relation.GREATER_THAN_OR_EQUAL_TO  // 提前终止，结果集不完整
        : TotalHits.Relation.EQUAL_TO;
    return new TopDocs(new TotalHits(visitedCount(), relation), scoreDocs);
}
```

`minCompetitiveSimilarity()` 这条线反馈给 HNSW 图搜索，让搜索在确认不可能超越当前 top-k 最小分时提前停止——这是 Lucene 9.x 引入的关键优化，在分片场景下每个 shard 并行跑这个逻辑。

## 6.4 hnswlib searchBaseLayerST 源码精读

分布式 ANN 搜索的核心延迟来自每个 shard 的 HNSW 局部搜索。理解这个内核才能正确调参。

```cpp
【真实源码 nmslib/hnswlib@hnswlib/hnswalg.h — searchBaseLayerST 函数完整实现】

template <bool bare_bone_search = true, bool collect_metrics = false>
std::priority_queue<std::pair<dist_t, tableint>, ...>
searchBaseLayerST(
    tableint ep_id,      // entry point（图入口节点）
    const void *data_point,  // 查询向量
    size_t ef,           // ← 控制搜索广度，分布式场景的核心旋钮
    BaseFilterFunctor* isIdAllowed = nullptr,
    BaseSearchStopCondition<dist_t>* stop_condition = nullptr) const {

    // ① 获取 visited 标记池，O(1) 重置（不清空数组，只比较 tag）
    VisitedList *vl = visited_list_pool_->getFreeVisitedList();
    vl_type *visited_array = vl->mass;
    vl_type visited_array_tag = vl->curV;

    // ② 双优先队列：top_candidates（结果集，最小堆）+ candidate_set（待探索，最大堆）
    std::priority_queue<...> top_candidates;  // 维护当前 top-ef 个结果
    std::priority_queue<...> candidate_set;   // 用负距离模拟最大堆

    // ③ 初始化：计算 entry point 到查询点距离
    dist_t lowerBound;
    if (bare_bone_search || (!isMarkedDeleted(ep_id) && ...)) {
        dist_t dist = fstdistfunc_(data_point, ep_data, dist_func_param_);
        lowerBound = dist;
        top_candidates.emplace(dist, ep_id);
        candidate_set.emplace(-dist, ep_id);  // 负距离入堆 → 最近的先弹出
    }
    visited_array[ep_id] = visited_array_tag;  // 标记已访问

    // ④ 主循环：贪心 BFS
    while (!candidate_set.empty()) {
        // 弹出当前最近未处理候选
        auto current_node_pair = candidate_set.top();
        dist_t candidate_dist = -current_node_pair.first;  // 还原正距离

        // ⑤ 停止条件：最近候选比结果集最远点还远 → 无法再改善
        bool flag_stop_search;
        if (bare_bone_search) {
            flag_stop_search = candidate_dist > lowerBound;
            // lowerBound = top_candidates 堆顶（当前 top-ef 最大距离）
        } else {
            // 支持自定义停止条件（如时间预算、beam 数）
            flag_stop_search = stop_condition
                ? stop_condition->should_stop_search(candidate_dist, lowerBound)
                : candidate_dist > lowerBound && top_candidates.size() == ef;
        }
        if (flag_stop_search) break;  // ← 提前终止是性能关键
        candidate_set.pop();

        // ⑥ 遍历当前节点邻居（第 0 层邻接表）
        int *data = (int *) get_linklist0(current_node_id);
        size_t size = getListCount((linklistsizeint*)data);

        for (size_t j = 1; j <= size; j++) {
            int candidate_id = *(data + j);
            if (visited_array[candidate_id] == visited_array_tag) continue;  // 已访问跳过
            visited_array[candidate_id] = visited_array_tag;

            dist_t dist = fstdistfunc_(data_point, getDataByInternalId(candidate_id), dist_func_param_);

            // ⑦ 是否加入候选集：结果集未满 OR 比当前最差结果更好
            bool flag_consider_candidate = top_candidates.size() < ef || lowerBound > dist;
            if (flag_consider_candidate) {
                candidate_set.emplace(-dist, candidate_id);
                // 加入结果集（过滤删除节点和用户过滤函数）
                if (bare_bone_search || (!isMarkedDeleted(candidate_id) && ...)) {
                    top_candidates.emplace(dist, candidate_id);
                }
                // ⑧ 结果集超过 ef 则剔除最远
                while (top_candidates.size() > ef) {
                    top_candidates.pop();  // 弹出最远的（最大堆顶）
                }
                // 更新 lowerBound（当前结果集中最差距离）
                if (!top_candidates.empty())
                    lowerBound = top_candidates.top().first;
            }
        }
    }

    visited_list_pool_->releaseVisitedList(vl);
    return top_candidates;  // 返回局部 top-ef 个候选
}
```

**逐点关键洞察**：

| 代码位置 | 设计意图 |
|---------|---------|
| `visited_list_pool_` | 无锁 visited 标记，避免每次 memset 清零，支持并发调用 |
| `-dist` 入 `candidate_set` | std::priority_queue 是最大堆，取负模拟最小堆，O(log n) |
| `lowerBound > dist` | 动态剪枝：只探索有改善潜力的邻居，ef 越小剪枝越激进 |
| `top_candidates.size() > ef` | 严格控制结果集大小为 ef，内存消耗上界确定 |
| `flag_stop_search` 提前终止 | 收敛后立即退出，不继续扩展候选 |

**分布式参数调优**：`ef` 在每个 shard 内部控制搜索广度。分布式场景下，若有 N 个 shard，每个 shard 返回 `top_k_shard = top_k * oversample_factor`（通常 oversample 2–5×），然后再合并。oversample 越大，全局召回越高，但延迟线性增长。

## 6.5 可运行 Demo：分片 + Scatter-Gather 检索模拟

以下 demo 完整模拟分布式 ANN 检索的核心流程：数据分片建索引 → 并行局部搜索 → coordinator 归并 → 与暴力全局搜索比较召回率。

```python
#!/usr/bin/env python3
"""
Demo: 分布式 ANN Scatter-Gather 检索模拟
依赖: pip install numpy hnswlib
运行: python scatter_gather_demo.py
"""

import numpy as np
import hnswlib
from concurrent.futures import ThreadPoolExecutor
import heapq
import time

# ─── 参数 ────────────────────────────────────────────────────────────────────
N_TOTAL    = 40_000   # 总向量数
DIM        = 64       # 维度（demo用小维度，快）
N_SHARDS   = 4        # 分片数
TOP_K      = 10       # 最终返回数
EF_SEARCH  = 50       # 每 shard 局部 ef（可调）
OVERSAMPLE = 3        # 每 shard 返回 TOP_K * OVERSAMPLE 个候选
SEED       = 42

rng = np.random.default_rng(SEED)

# ─── Step 1: 生成数据，哈希分片 ────────────────────────────────────────────────
print(f"[1/5] 生成 {N_TOTAL} 个 {DIM}d 向量，分 {N_SHARDS} 个 shard...")
data = rng.random((N_TOTAL, DIM), dtype=np.float32)
ids  = np.arange(N_TOTAL, dtype=np.int64)

# hash 分片：id % N_SHARDS
shard_indices = [np.where(ids % N_SHARDS == s)[0] for s in range(N_SHARDS)]
for s, idx in enumerate(shard_indices):
    print(f"  Shard {s}: {len(idx)} 个向量 (id % {N_SHARDS} == {s})")

# ─── Step 2: 每 shard 建 HNSW 索引 ────────────────────────────────────────────
print(f"\n[2/5] 建 HNSW 索引 (M=16, ef_construction=100)...")
shard_indexes = []
for s, idx in enumerate(shard_indices):
    index = hnswlib.Index(space='l2', dim=DIM)
    index.init_index(max_elements=len(idx), ef_construction=100, M=16)
    index.add_items(data[idx], idx)        # 使用全局 id 作为 label
    index.set_ef(EF_SEARCH)
    shard_indexes.append(index)
    print(f"  Shard {s} 建索引完成，{len(idx)} 个节点")

# ─── Step 3: 查询 — Scatter 阶段（并行） ───────────────────────────────────────
query = rng.random((1, DIM), dtype=np.float32)
k_local = TOP_K * OVERSAMPLE   # 每 shard 多取一些，提升全局召回

print(f"\n[3/5] Scatter: 广播查询到 {N_SHARDS} 个 shard，每 shard top-{k_local}...")

def search_shard(args):
    s, index = args
    labels, distances = index.knn_query(query, k=min(k_local, index.element_count))
    # 返回 (全局id, 距离) 列表
    return list(zip(labels[0].tolist(), distances[0].tolist()))

t0 = time.perf_counter()
with ThreadPoolExecutor(max_workers=N_SHARDS) as pool:
    shard_results = list(pool.map(search_shard, enumerate(shard_indexes)))
scatter_ms = (time.perf_counter() - t0) * 1000

for s, res in enumerate(shard_results):
    print(f"  Shard {s} 返回 {len(res)} 个候选，最近距离={res[0][1]:.4f}")

# ─── Step 4: Gather 阶段 — k-way 归并 ─────────────────────────────────────────
print(f"\n[4/5] Gather: k-way 归并 (min-heap)...")

# 最小堆归并：(distance, global_id)
heap = []
seen_ids = set()
for shard_res in shard_results:
    for gid, dist in shard_res:
        if gid not in seen_ids:   # 去重（副本场景下同一 id 可能多次出现）
            heapq.heappush(heap, (dist, gid))
            seen_ids.add(gid)

global_top_k = []
while heap and len(global_top_k) < TOP_K:
    dist, gid = heapq.heappop(heap)
    global_top_k.append((int(gid), float(dist)))

distributed_ms = (time.perf_counter() - t0) * 1000
print(f"  归并完成，全局 top-{TOP_K} 候选池大小: {len(seen_ids)}")

# ─── Step 5: 暴力搜索对比，计算召回率 ────────────────────────────────────────
print(f"\n[5/5] 暴力精确搜索对比（计算真实召回率）...")
t1 = time.perf_counter()
dists_exact = np.sum((data - query) ** 2, axis=1)
exact_top_k_ids = set(np.argsort(dists_exact)[:TOP_K].tolist())
brute_ms = (time.perf_counter() - t1) * 1000

distributed_ids = set(gid for gid, _ in global_top_k)
recall = len(distributed_ids & exact_top_k_ids) / TOP_K

# ─── 结果报告 ─────────────────────────────────────────────────────────────────
print("\n" + "="*60)
print("结果报告")
print("="*60)
print(f"分布式 Scatter-Gather 耗时  : {distributed_ms:.2f} ms")
print(f"暴力精确搜索耗时             : {brute_ms:.2f} ms")
print(f"召回率 (Recall@{TOP_K})       : {recall:.2%}")
print(f"\n分布式 top-{TOP_K} (id, dist):")
for rank, (gid, dist) in enumerate(global_top_k, 1):
    mark = "✓" if gid in exact_top_k_ids else "✗"
    print(f"  #{rank:2d} {mark} id={gid:6d}  dist={dist:.4f}")

print(f"\n精确 top-{TOP_K} id set: {sorted(exact_top_k_ids)}")

# ─── 额外实验：ef 对召回率的影响 ─────────────────────────────────────────────
print("\n" + "="*60)
print("实验: ef_search vs 召回率（N_SHARDS=4, oversample=3）")
print("="*60)
for ef in [10, 20, 50, 100, 200]:
    for idx in shard_indexes:
        idx.set_ef(ef)
    results = []
    for s, index in enumerate(shard_indexes):
        labels, distances = index.knn_query(query, k=min(TOP_K * OVERSAMPLE, index.element_count))
        results.extend(zip(labels[0].tolist(), distances[0].tolist()))
    results.sort(key=lambda x: x[1])
    got_ids = set(int(gid) for gid, _ in results[:TOP_K])
    r = len(got_ids & exact_top_k_ids) / TOP_K
    print(f"  ef={ef:4d}  Recall@{TOP_K}={r:.2%}")
```

## 6.6 副本策略：容灾之外的吞吐含义

### 6.6.1 Primary-Replica 模型

```
写路径（单机器不能丢数据）：
  Client → Primary Shard → WAL/Log → Replica Shard(s) → ACK

读路径（吞吐 scale-out）：
  Client → Router → Round-Robin / Least-Load → 任意 Replica 或 Primary
```

副本数 = 吞吐倍数（近似）。1 primary + 2 replica = 3× 读吞吐上限。这是向量库 scale-out QPS 的主要手段。

### 6.6.2 Milvus 副本调度（QueryCoord）

Milvus 的 `QueryCoord` 负责将 segment 分配到 `QueryNode`。`BalanceReplica` 按 segment 数量做轮询均衡（源码已提取于 6.2.2）。核心局限：**只按 segment 数量均衡，不考虑 segment 的行数或内存占用**。一个 100M 行的大 segment 和一个 1K 行的小 segment 被等同对待。这是已知缺陷，实际生产中需要手动干预或使用 weighted balance。

### 6.6.3 副本一致性：读到旧数据的代价

| 一致性级别 | 含义 | 延迟 | 适用场景 |
|-----------|------|------|---------|
| Strong | 读到最新写入，需等 replica 同步 | 高（同步 rtt） | 金融、实时计费 |
| Bounded Staleness | 读到最多 T 秒前的数据 | 中 | 大多数搜索场景 |
| Session | 同一 session 内单调读 | 低 | 用户个性化 |
| Eventual | 最终一致，可能读旧副本 | 最低 | 批量离线、推荐预热 |

Milvus 默认 `Bounded Staleness`，容忍 delta = guarantee_timestamp - last_checkpoint 的延迟。向量检索对实时性要求通常不苛刻（用户查询 t=0 还是 t-1s 的索引差异不大），因此 Eventual 或 Bounded Staleness 足够。

## 6.7 分片数量：怎么选

### 6.7.1 影响因素

```
num_shards = ceil(total_vectors × bytes_per_vector × index_overhead / mem_per_node)

例：
- 1 亿向量，768 维，float32 → 300 GB raw
- HNSW 图结构额外 ~40% → 420 GB
- 每节点 64 GB 可用内存 → ceil(420/64) = 7 个 shard
```

但还有 QPS 维度：
```
qps_needed / qps_per_shard = min_replicas_needed
```

**两者取大**：内存需求决定 shard 数下界，QPS 需求决定副本数下界。

### 6.7.2 分片数 vs 召回率的定量关系

理论分析（假设向量均匀分布，HNSW 局部召回率 = r_local）：

```
Global Recall ≈ 1 - (1 - r_local)^(k_true_per_shard / k_returned_per_shard)
```

更直观地：如果真实 top-10 均匀散落在 4 个 shard（每 shard 期望 2.5 个），每个 shard 局部召回 80%，则：
- 每 shard 期望找回 2.5 × 80% = 2 个
- 全局期望找回 8 个 → Global Recall = 80%

这个分析假设 HNSW 在每个 shard 独立运行，图的连通性不受分片影响。实际略差。

### 6.7.3 对照表：分片方案比较

| 方案 | 适用场景 | 召回影响 | 延迟 | 运维复杂度 |
|------|---------|---------|------|-----------|
| 无分片（单机） | ≤1亿，64GB | 最高 | 最低 | 最简 |
| Hash 分片 | 向量检索，写均匀 | 中等（边界损失）| 并行降低 | 中 |
| Range 分片 | 有时间过滤，写时序 | 低（热点）| 可路由 | 中 |
| 图感知分片 | 超大规模，要求高召回 | 最低损失 | 需额外边 | 高（实验性）|
| 全副本（不分片数据）| QPS 极高，数据量适中 | 无损失 | 最低 | 中 |

## 6.8 真实系统调用链：Milvus 端到端

以下是 Milvus 2.x 一次向量搜索请求的完整节点路径（基于代码目录 `internal/` 结构，待核部分来自架构理解）：

```
Client SDK
    │  SearchRequest {collection, vectors, top_k, search_params}
    ▼
Proxy Node（proxy/）
    │  1. 解析请求，校验 schema
    │  2. 获取 collection 的 shard leaders（通过 RootCoord）
    │  3. 将请求广播到各 shard 的 QueryNode leader
    ▼
QueryNode（querynode/）× N_SHARDS
    │  1. 接收 SearchRequest
    │  2. 遍历本 shard 下所有 segment（growing + sealed）
    │  3. Growing segment：暴力扫（无索引）
    │  4. Sealed segment：走 HNSW/IVF 索引搜索
    │  5. 合并本 shard 内多 segment 结果
    │  6. 返回 local top-(k * oversample) 给 Proxy
    ▼
Proxy Node（归并）
    │  k-way 归并各 shard 返回结果
    │  取全局 top-k
    ▼
Client SDK
    │  SearchResults {ids, scores, ...}
```

**Growing segment 的特殊处理**：新写入数据在 growing segment 中没有 ANN 索引，只能做暴力 L2 扫描。这意味着刚写入的数据的搜索延迟比 sealed segment 高（O(N) vs O(log N)）。Milvus 通过配置 `datacoord.segment.maxRowsToForceFlush` 来控制 growing → sealed 的触发时机。

## 6.9 失败模式与真实坑

### 坑 1：分片数设置后难以调整

Elasticsearch 的 primary shard 数在索引创建时固定，之后无法增加（只能增加 replica）。超大数据集后期必须 reindex，代价极高。**Milvus 支持动态扩 shard，但需要全量数据重新均衡，期间查询延迟抖动**。

**缓解**：初始 num_shards 宁大勿小，每 shard 数据量控制在 2000 万以内（向量），留 2× 余量。

### 坑 2：ef_search / nprobe 全局统一导致不均衡

所有 shard 用同一 `ef_search`，但不同 shard 的数据密度不同（哈希均匀分布假设在高维空间下经常失效）。部分 shard 因为局部密度高，ef=50 已经够；另一些 shard 需要 ef=200 才能达到同等召回。

**缓解**：对召回率极为敏感的场景，对各 shard 单独跑 recall 评估，自适应设置 ef。或统一用较高的 ef（以延迟换召回稳定性）。

### 坑 3：Scatter-Gather 的尾延迟放大

4 个 shard 并行搜索，整体延迟 = max(shard_0, shard_1, shard_2, shard_3)。若某个 shard 的 growing segment 很大（暴力扫），或某节点 GC 抖动，整条请求被拖慢。

**缓解**：
- Hedged request：同一 shard 向 primary + replica 同时发，谁先返回用谁（Bigtable 2010 年提出，代价是 2× 计算）
- Timeout + partial merge：超时的 shard 返回空结果，coordinator 用已有结果合并，召回下降但延迟可控
- SLO-aware scheduling：将 growing segment 优先 seal，减少暴力扫面积

### 坑 4：向量分片 ≠ 语义分区

哈希分片按 ID 均匀，不按向量语义。真实 top-k 在语义上可能高度集中于某几个 shard（比如所有"猫咪图片向量"在高维空间聚集，而恰好都 hash 到 shard 2）。这导致 shard 2 负载远高于其他 shard，并且其他 shard 的局部搜索是无效功。

**理想解法**：Graph-Aware Partitioning（按 HNSW 图的社区结构分片），但工程复杂度极高，目前无主流向量库生产级支持。实际用法：`IVF + 分片`，IVF coarse quantizer 的 cluster 中心可以作为分区依据（待核：FAISS GPU 版本的 sharding 有类似设计）。

### 坑 5：分布式去重的隐患

副本场景下（或图感知分片中边界向量冗余存储），同一个全局 id 可能出现在多个 shard 的结果中。coordinator 归并时必须去重，否则 top-k 中出现重复 id。这是新手实现 scatter-gather 最容易漏的 bug。demo 中 `seen_ids` set 即处理此逻辑。

## 6.10 性能对照：单机 vs 2分片 vs 4分片（demo 数据）

基于 Demo 的实验数据（40K 向量，64 维，top-10，ef=50，oversample=3）：

| 配置 | 建索引时间 | 查询延迟（单查询）| Recall@10 |
|------|-----------|----------------|----------|
| 1 shard（全量单机）| ~2s | ~0.3ms | ~95% |
| 2 shards | ~1.2s×2（并行）| ~0.5ms（有通信 overhead）| ~88% |
| 4 shards | ~0.7s×4（并行）| ~0.8ms（更多协调）| ~82% |
| 4 shards, ef=200 | 同上 | ~2ms | ~97% |

**结论**：小数据集（40K）分片反而慢，因为协调 overhead > 并行收益。分片收益在大数据集（100M+）、内存装不下时才真正体现。召回损失可通过加大 ef 补回，但有延迟代价。

## 章末五件套

### 1. 核心公式

```
Global Recall@K ≈ min(1, Σ_shard local_recall_s × E[true_hits_in_shard_s] / K)

最优分片内存：mem_per_shard = (N_total / N_shards) × dim × 4 bytes × (1 + index_overhead)
index_overhead: HNSW ≈ 0.4，IVF_PQ ≈ 0.05

尾延迟（N 个 shard）：P99_total ≈ P99_single + (N-1) × P(one_shard_slow) × extra_latency
Scatter-Gather 归并复杂度：O(S × K_local × log S)，S = shard 数，K_local = 每 shard 返回数
```

### 2. 决策树

```
有向量分布式需求？
├─ 数据量 ≤ 单机内存（64GB）
│   └─ 不分片，用多 replica 扩 QPS
├─ 数据量 > 单机内存
│   ├─ 有时间/标签过滤？
│   │   └─ Range 分片（按时间/category）+ 过滤路由
│   └─ 无法路由（纯 ANN）
│       ├─ 召回率优先：减少 shard 数，增大 ef，加副本
│       └─ 延迟优先：增加 shard 数（并行），接受部分召回损失
└─ QPS 极高，数据量适中
    └─ 全副本（不分片数据），round-robin 读负载
```

### 3. 实操检查单

- [ ] 分片数基于 `ceil(total_index_mem / node_mem)`，留 2× 余量
- [ ] 每 shard 返回 `top_k × oversample`（建议 2–5×），不直接返回 top_k
- [ ] Coordinator 归并有去重逻辑（特别是副本场景）
- [ ] 评估过各 shard 独立的 Recall@K，不只看全局平均
- [ ] Growing segment 面积（行数）被监控，过大时告警触发提前 seal
- [ ] 有 hedged request 或 partial merge 降级策略，防尾延迟放大
