> **一句话定位**：物理内存管理是内核中"土地局"——所有虚拟地址背后的物理页帧都由它分配与归还。它的两条铁律决定了上层 slab / page cache / 进程映射的命运：**用 2^n 对齐把"能否合并"压成 O(1) 谓词；用迁移类型（migratetype）把"碎片是否可修复"变成一个可治理的属性。**

---

## TL;DR · 本章最硬结论（读完全章应能逐条复述其根因与源码出处）

1. **buddy 的核心不是"分配"，是"合并"**。`buddy_pfn = pfn ^ (1 << order)` 这一行 XOR（`mm/internal.h::__find_buddy_pfn`）让"找邻居"零成本；2^n 对齐保证邻居要么同阶可合并、要么不存在，**消灭了变长分配器里那种"左右都能合一点"的合并歧义**。代价是内碎片（请求 33 页要给 64 页）。
2. **外碎片 buddy 治不了，migratetype + compaction 才治**。free_list 按 `MIGRATE_UNMOVABLE / MOVABLE / RECLAIMABLE` 分桶（`include/linux/mmzone.h::free_area`），compaction（IBM / Mel Gorman，2007，commit 版权头白纸黑字）把 MOVABLE 页搬走腾出连续块。**不可移动页一旦散进 MOVABLE pageblock，会永久卡死 compaction**——这就是 fallback "偷整块"策略的动机。
3. **zone 分层是设备寻址 + 32 位虚拟地址不足的双重历史包袱**。x86-64 上 `ZONE_HIGHMEM` 已从 `enum zone_type` 里被 `#ifdef CONFIG_HIGHMEM` 编译掉，但 `ZONE_DMA`/`ZONE_DMA32` 因 32 位设备总线地址限制永久保留（LWN 91870，2004 年 em64t swiotlb 讨论印证）。
4. **watermark 判定从来不是裸比 `free vs WMARK`**。真正的硬线是 `free_pages <= min + z->lowmem_reserve[highest_zoneidx]`（`mm/page_alloc.c::__zone_watermark_ok`），`lowmem_reserve` 是"高 zone 给低 zone 留的命"。kswapd 在 LOW 唤醒、HIGH 停止；direct reclaim 在 MIN 阻塞调用者；`__GFP_HIGH` 能把 min 砍半再砍 1/4 动用紧急储备。
5. **`struct page` 是内核中最昂贵也最"精神分裂"的 struct**。x86-64 上固定 **64 字节**，开销 **≈ 物理内存的 1.5%（每 1TB 吃 16GB）**。一个 union 同时被 page cache / page_pool / compound tail / ZONE_DEVICE / slab / buddy 复用——`mm_types.h` 注释明写"bit 0 of the first word 必须留给 PageTail()"。folio（5.16+）和正在推进的 **memdesc（目标把 struct page 砍到 8 字节）** 都是为治这个病。

## 8.1 背景：为什么物理内存管理是内核最复杂的子系统之一

### 8.1.1 问题本质：四个互相冲突的约束同时成立

操作系统要把有限、连续的物理 RAM 切成可按需分配/回收的单元，同时满足：

| 约束 | 具体诉求 | 与谁冲突 |
|------|---------|---------|
| **大小多样** | 内核 8 字节 slab 对象 ~ 用户 512MB 大映射 | 与"对齐"和"碎片控制"冲突 |
| **对齐** | DMA 要物理连续、hugepage 要 2MB 对齐 | 与"任意大小高效分配"冲突 |
| **碎片控制** | 跑几天后仍能掏出 order-9（2MB）连续块 | 与"快速分配小块"冲突（小块散落破坏连续性） |
| **NUMA 感知** | 跨 node 有"距离税"，不能乱跨 | 与"内存压力下随便 fallback"冲突 |

关键张力在于**时间尺度跨度极大**：高频路径（per-CPU pageset miss → buddy 摘链）要求 O(1)~O(log n)；极低频整理（compaction 搬页 + 改页表）可以容忍毫秒级停顿。一个数据结构要同时服务这两端，这是物理内存管理复杂度的根。

### 8.1.2 类比锚点（仅用于建立直觉，源码语义以后文为准）

把物理 RAM 想成一栋 1024 套公寓楼：

- **buddy** = 房产规则：只能整层（2^n 套）划出；两个"兄弟楼层"必须大小相同且地址相邻（`pfn ^ size` 关系）才能并成大楼层。
- **migratetype** = 住户类型标签：UNMOVABLE 是焊死的承重墙（内核栈、页表），MOVABLE 是随时能搬的租客（用户页、page cache）。**关键洞见：把承重墙集中在几栋楼，别让它散落到租客楼里，否则整栋楼永远拆不动。**
- **zone** = 按用途分区：地下室（DMA，老设备只能停这）、标准层（Normal）、顶层（HighMem，32 位时代的阁楼）。
- **watermark** = 消防水位线：低于 LOW 喊保安（kswapd）来清理，低于 MIN 住户自己动手清（direct reclaim）。
- **NUMA** = 楼栋间的廊桥：走廊桥（cross-socket interconnect）就是慢。

## 8.2 设计考古：从位图到 buddy + zone + NUMA + compaction（谁、何时、为何）

> 本节是为修复"浮在表面"专设的考古小节。每个机制都回答：**解决了前一代什么缺陷？社区辩论过什么？出处在哪？**

### 8.2.1 早期：位图 + 线性扫描（Linux 0.x）

最早用固定位图管理页，分配扫到第一个空位即止。两个致命缺陷直接催生 buddy：

1. **分配连续多页是 O(n) 扫描**——位图无法 O(1) 回答"这里有没有连续 2^k 页"。
2. **对碎片零认知**——释放时无法廉价判断"能不能和邻居合并成大块"，于是大块请求长期失败。

### 8.2.2 Buddy 系统：把"能否合并"压成 O(1) 谓词（Knuth 1973 → Linux 1.x/2.x）

buddy allocator 出自 Knuth《TAOCP》Vol.1（1973），但**它在 Linux 里的价值不在"分配快"，而在"合并判断快"**。

核心洞见（请记住这一句，它是理解后面所有源码的钥匙）：

> **让"两个块能否合并"成为一个不需要搜索、不需要遍历的 O(1) 算术谓词。**

实现这一点的全部魔法就是一行 XOR（下文 8.3.2 有逐行注解的真实源码）：`buddy_pfn = pfn ^ (1 << order)`。配合两条规则：

- 块大小必须是 2^k 页；
- 2^k 块的起始 PFN 必须是 2^k 的倍数（自然对齐）。

于是"邻居"唯一确定、要么同阶可合并、要么根本不是 buddy——**没有"左边能合 3 页、右边能合 5 页"这种变长分配器的合并歧义**。

> ⚠️ **`MAX_ORDER` 版本陷阱（Staff 面试高频，且是一段真实社区翻车史）**
>
> 这不是教科书八股，是 2023 年真实发生、惊动 Linus 的事故。完整考据见 [LWN《The trouble with MAX_ORDER》(956321)](https://lwn.net/Articles/956321/)：
>
> - **起源（1999, kernel 2.3.27）**：`MAX_ORDER` 引入时是**排他**语义——`MAX_ORDER == 11` 表示维护 11 个 free 链表（order 0~10），实际最大可分配 order 是 **10**。Jonathan Corbet 原话："it is not actually the maximum order"。名字和语义不符，是个潜伏 24 年的 footgun。
> - **2023-03（merged for 6.4, 2023-06）**：Kirill Shutemov 在**九处**发现因误解此语义导致的 bug（"including a longstanding bug in the floppy driver"），于是 commit **`23baf831`** 把语义翻转为**包含**——`MAX_ORDER` 直接表示最大 order，默认值从 11 变 **10**。
> - **2023-09 翻车**：Paolo Bonzini 指出三处 in-tree 改动早被这次语义翻转悄悄改坏却无人察觉；6.5 里 dm-crypt 也炸了。Linus 直言 **"I still think this was all likely a mistake"**，并坚持**改名以制造编译错误**而非静默踩坑。
> - **2023-12-28**：Shutemov 重命名 `MAX_ORDER → MAX_PAGE_ORDER`，并引入 `NR_PAGE_ORDERS`（= 链表个数）。结果：**任何 out-of-tree 代码用旧 `MAX_ORDER` 会编译失败**——这正是 Linus 要的"吵醒比静默错好"。
>
> **结论与读法**：任何写"`MAX_ORDER == 11`"的资料都是 6.4 之前的。读 mm 代码先看版本。本章统一用 v6.12 的 `MAX_PAGE_ORDER`/`NR_PAGE_ORDERS`，事实量级（最大 order 10 / 4MB）不变。

### 8.2.3 Zone：设备寻址 + 32 位虚拟地址不足的双重妥协（Linux 2.2~2.6）

zone 的存在有两个**完全不同**的根因，新手常混为一谈：

1. **设备 DMA 寻址限制（永久性，与字长无关）**：ISA 设备只能寻址低 16MB → `ZONE_DMA`；32 位 PCI 设备只能寻址低 4GB → `ZONE_DMA32`。这个约束**在 64 位上依然存在**，所以 `ZONE_DMA`/`ZONE_DMA32` 在 x86-64 永久保留。[LWN 91870（2004）](https://lwn.net/Articles/91870/) 记录了 em64t 时代的真实痛点：Intel 早期 x86_64 没有 IOMMU（AMD 有），32 位设备访问高地址内存只能靠 **swiotlb bounce buffer**——文章原话称这方案"inelegant"但实用。**这是"为什么 64 位还要 DMA zone"的史料级答案。**
2. **内核虚拟地址不足（32 位特有，已消亡）**：见 8.4 的 HighMem 专题。

### 8.2.4 NUMA：从 SMP 到多 socket 的"距离税"（Linux 2.5~）

SMP 扩到多 socket 后，每 socket 有本地 RAM，跨 socket 走 QPI/UPI/HyperTransport/Infinity Fabric。内核引入 **node → zone → page** 三层层次（`pg_data_t` per node）。NUMA 内存策略（`mbind`/`set_mempolicy` 系统调用）也在此时期成型——详见 8.3.7，那里有 v6.12 内核文档对策略层级的权威定义，包括近年新增的 `MPOL_PREFERRED_MANY` 和 `MPOL_WEIGHTED_INTERLEAVE`。

### 8.2.5 Migration type + Compaction：外碎片的"可治理化"（2007, IBM / Mel Gorman）

这是 buddy 之后物理内存管理最重要的一次架构升级。源码版权头是最硬的考古证据 ——`mm/compaction.c` 开头白纸黑字【真实源码 v6.12 路径】：

```c
/*
 * Memory compaction for the reduction of external fragmentation. Note that
 * this heavily depends upon page migration to do all the real heavy
 * lifting
 *
 * Copyright IBM Corp. 2007-2010 Mel Gorman <mel@csn.ul.ie>
 */
```

**"heavily depends upon page migration"** 这句是理解 compaction 的关键：compaction 自己不搬数据，它依赖 page migration 子系统改页表、搬内容。compaction 只负责"决定搬谁、搬到哪"。

**社区辩论考古（LSFMM 2016, [LWN 684611](https://lwn.net/Articles/684611/)）**：这场讨论暴露了 compaction 的几个真实缺陷，也定下了后续方向：

- **Mel Gorman** 否决了 `ZONE_CMA` 方案，原话 **"simply not acceptable"**——理由是 movable 分配在 I/O 期间被 pin 住、compound page 搬不动会让 CMA 区永久碎片化。
- **Michal Hocko** 提出 compaction 需要优先级 / GFP 风格的 flag，平衡"激进造大页" vs "小分配也要 compaction"。
- **Vlastimil Babka** 提议专门的 **kcompactd** 后台线程——这后来落地了。
- 缺陷点名：compaction 当时"过度聚焦 THP"、"skip bits 阻止了小规模 compaction 成功"。

**proactive compaction（Nitin Gupta，5.6, [LWN 817905](https://lwn.net/Articles/817905/)）**：on-demand compaction 只在分配失败时临时搞，导致 hugepage 分配延迟尖刺。proactive 方案让 kcompactd 持续按 per-node fragmentation score 后台整理。实测 **hugepage 分配延迟降低 70-80 倍**（p95 从 33799µs 降到 429µs），一个 Java 堆分配测试从 ~27 分钟降到 ~4 分钟。落地为 sysctl `vm.compaction_proactiveness`（默认 20，范围 0~100）。

## 8.3 现状：核心数据结构与关键路径（源码精读 + 逐行注解）

### 8.3.1 真实数据结构骨架（v6.12 verbatim）

先把三个核心结构的**真实定义**摆出来【真实源码 v6.12 路径：include/linux/mmzone.h】，后文逐个拆：

```c
struct free_area {
	struct list_head	free_list[MIGRATE_TYPES];  // 按迁移类型分桶的空闲链表
	unsigned long		nr_free;                    // 该 order 总空闲块数
};

enum migratetype {
	MIGRATE_UNMOVABLE,                 // 焊死：内核数据、页表、slab
	MIGRATE_MOVABLE,                   // 可搬：用户匿名页、page cache
	MIGRATE_RECLAIMABLE,               // 可回收但不可搬：如 dentry/inode cache
	MIGRATE_PCPTYPES,	/* the number of types on the pcp lists */
	MIGRATE_HIGHATOMIC = MIGRATE_PCPTYPES,  // 给高阶原子分配预留的紧急块
#ifdef CONFIG_CMA
	MIGRATE_CMA,                       // CMA 连续内存分配器专用
#endif
#ifdef CONFIG_MEMORY_ISOLATION
	MIGRATE_ISOLATE,	/* can't allocate from here */  // hotplug/隔离用，禁分配
#endif
	MIGRATE_TYPES
};

#ifndef CONFIG_ARCH_FORCE_MAX_ORDER
#define MAX_PAGE_ORDER 10            // x86-64 默认：最大 order 10 → 1024 页 → 4MB
#else
#define MAX_PAGE_ORDER CONFIG_ARCH_FORCE_MAX_ORDER
#endif
#define MAX_ORDER_NR_PAGES (1 << MAX_PAGE_ORDER)
#define NR_PAGE_ORDERS (MAX_PAGE_ORDER + 1)   // 链表个数 = 11（order 0..10）
```

**逐行要点**：

- `free_list[MIGRATE_TYPES]`：注意是**二维**的 —— 每个 order 一个 `free_area`，每个 `free_area` 内部又按 migratetype 拆 `MIGRATE_TYPES` 条链表。这就是"同 order 的空闲块还要再按类型隔离"的物理体现，是外碎片治理的地基。
- `MIGRATE_PCPTYPES`：前三种（UNMOVABLE/MOVABLE/RECLAIMABLE）才上 per-CPU pageset 快速缓存；HIGHATOMIC/CMA/ISOLATE 是特殊用途不进 pcp。**这是个隐藏分界线**，后面 `__zone_watermark_ok` 里 `for (mt = 0; mt < MIGRATE_PCPTYPES; mt++)` 就靠它。
- `MIGRATE_HIGHATOMIC = MIGRATE_PCPTYPES`：复用同一个枚举值是有意的——它不上 pcp，但又要排在常规类型之后。给 `GFP_ATOMIC` 的高阶分配留的"应急仓库"。

`enum zone_type` 和 watermark 枚举【真实源码 v6.12 路径：include/linux/mmzone.h】：

```c
enum zone_type {
#ifdef CONFIG_ZONE_DMA
	ZONE_DMA,         // 低 16MB（x86），老 ISA 设备
#endif
#ifdef CONFIG_ZONE_DMA32
	ZONE_DMA32,       // 低 4GB，32 位 PCI 设备
#endif
	ZONE_NORMAL,      // 直接映射区，最常用
#ifdef CONFIG_HIGHMEM
	ZONE_HIGHMEM,     // 仅 32 位：内核虚拟地址不够时 kmap 的高端内存
#endif
	ZONE_MOVABLE,     // 软件定义：只放可移动页，利于 hotplug/compaction
#ifdef CONFIG_ZONE_DEVICE
	ZONE_DEVICE,      // 持久内存(pmem)/设备内存/HMM
#endif
	__MAX_NR_ZONES
};

enum zone_watermarks {
	WMARK_MIN,    // 地板：direct reclaim 触发线
	WMARK_LOW,    // kswapd 唤醒线
	WMARK_HIGH,   // kswapd 停止线
	WMARK_PROMO,  // NUMA 提升专用水位（比 HIGH 更高，给 numa_balancing 升页留余量）
	NR_WMARK
};
```

> 🎯 **两个易错点直接来自这段真实源码**：(1) `ZONE_HIGHMEM` 被 `#ifdef CONFIG_HIGHMEM` 包着——**这个 Kconfig 仅 32 位架构可见**，所以 x86-64 内核里这个枚举值根本不存在，不是"运行时为空"而是"编译期消失"。(2) watermark 其实是**四档**不是三档：v6.12 多了 `WMARK_PROMO`，专给 NUMA balancing 把远端热页"晋升"到本地 node 时做水位判断。老资料说"三档"是对的但不全。

### 8.3.2 buddy 的灵魂：XOR 找 buddy（逐行注解真实源码）

这是全章最该背下来的两个函数【真实源码 v6.12 路径：mm/internal.h】：

```c
static inline unsigned long
__find_buddy_pfn(unsigned long page_pfn, unsigned int order)
{
	return page_pfn ^ (1 << order);
}
```

**逐行根因**：为什么是 XOR 而不是加减？因为对齐保证下，order-k 块的 PFN 低 k 位必为 0，第 k 位是"我是左 buddy 还是右 buddy"的标志位。`^ (1 << order)` 翻转这一位：左 buddy（第 k 位=0）异或后得到右 buddy（第 k 位=1），反之亦然。**一次异或同时处理了"向左找"和"向右找"两个方向**——加减法做不到（你得先判断自己是左是右）。这就是 buddy 算法 O(1) 合并判断的算术内核。

```c
static inline struct page *find_buddy_page_pfn(struct page *page,
		unsigned long pfn, unsigned int order, unsigned long *buddy_pfn)
{
	unsigned long __buddy_pfn = __find_buddy_pfn(pfn, order);   // 1. 算出 buddy 的 PFN
	struct page *buddy;

	buddy = page + (__buddy_pfn - pfn);   // 2. struct page 数组里 buddy 的指针 = 当前指针 + PFN 差
	if (buddy_pfn)
		*buddy_pfn = __buddy_pfn;

	if (page_is_buddy(page, buddy, order))   // 3. 校验：buddy 真的空闲 + 真的同 order 吗？
		return buddy;
	return NULL;   // 不满足 → 不能合并
}
```

**第 2 行的精妙**：因为 `struct page` 是个**连续数组**（mem_map/vmemmap），PFN 差就等于数组下标差，所以 `page + (buddy_pfn - pfn)` 直接 O(1) 拿到 buddy 的 `struct page`，无需任何查表。这也解释了为什么内核坚持 `struct page` 数组的连续性（vmemmap 稀疏映射也要维持"PFN→指针"的线性关系）。

### 8.3.3 分配：`__rmqueue_smallest` + `expand`（拆大块）

分配核心【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static __always_inline
struct page *__rmqueue_smallest(struct zone *zone, unsigned int order,
					int migratetype)
{
	unsigned int current_order;
	struct free_area *area;
	struct page *page;

	/* Find a page of the appropriate size in the preferred list */
	for (current_order = order; current_order < NR_PAGE_ORDERS; ++current_order) {
		area = &(zone->free_area[current_order]);
		page = get_page_from_free_area(area, migratetype);  // 取该 order/类型的一块
		if (!page)
			continue;   // 这个 order 没货，往上找更大的拆

		page_del_and_expand(zone, page, order, current_order,
				    migratetype);   // 摘下 + 把多余的部分拆回低 order
		...
		return page;
	}
	return NULL;   // 一直到 MAX_PAGE_ORDER 都没货 → 失败（上层进慢路径）
}
```

注意 v6.x 把"摘链"和"拆分"合进了 `page_del_and_expand`（旧版本是分开两步），它的实现【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static __always_inline void page_del_and_expand(struct zone *zone,
					struct page *page, int low,
					int high, int migratetype)
{
	int nr_pages = 1 << high;

	__del_page_from_free_list(page, zone, high, migratetype);   // 从 high order 链表摘下整块
	nr_pages -= expand(zone, page, low, high, migratetype);     // 拆：多出来的挂回低 order
	account_freepages(zone, -nr_pages, migratetype);            // 记账：实际占用 = high块 - 拆回的
}
```

而 `expand` 是"拆大块还小块"的精华【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static inline unsigned int expand(struct zone *zone, struct page *page, int low,
				  int high, int migratetype)
{
	unsigned int size = 1 << high;
	unsigned int nr_added = 0;

	while (high > low) {       // 从 high 拆到 low
		high--;
		size >>= 1;        // 块对半分
		VM_BUG_ON_PAGE(bad_range(zone, &page[size]), &page[size]);

		if (set_page_guard(zone, &page[size], high))  // debug: page guard 占位则跳过
			continue;

		__add_to_free_list(&page[size], zone, high, migratetype, false);  // 后半块挂回低 order 链表
		set_buddy_order(&page[size], high);   // 记录后半块的 order（存进 page->private）
		nr_added += size;
	}

	return nr_added;
}
```

**跑一遍**：请求 order-2（4 页），但只有 order-5（32 页）有货。`expand(low=2, high=5)`：
- high=5→4：把 32 页块的后 16 页挂回 order-4，前 16 页留着；
- high=4→3：后 8 页挂回 order-3，前 8 页留着；
- high=3→2：后 4 页挂回 order-2，前 4 页 = 请求的目标，返回。

一次分配顺手在 order-4/3/2 各补了一块空闲。**这就是 buddy"拆一次、链表全档位都得到补给"的优雅之处**，也是它分配/释放都 O(log MAX_PAGE_ORDER) ≤ O(10) ≈ O(1) 的来源。

### 8.3.4 释放：`__free_one_page` 的级联合并（逐行注解，本章最硬源码）

这是 buddy 的"合并引擎"，把上面所有零件串起来【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static inline void __free_one_page(struct page *page,
		unsigned long pfn,
		struct zone *zone, unsigned int order,
		int migratetype, fpi_t fpi_flags)
{
	struct capture_control *capc = task_capc(zone);
	unsigned long buddy_pfn = 0;
	unsigned long combined_pfn;
	struct page *buddy;
	bool to_tail;

	VM_BUG_ON(!zone_is_initialized(zone));
	VM_BUG_ON_PAGE(pfn & ((1 << order) - 1), page);   // 断言：PFN 必须按 order 对齐

	account_freepages(zone, 1 << order, migratetype);

	while (order < MAX_PAGE_ORDER) {           // 尝试逐级向上合并
		int buddy_mt = migratetype;

		if (compaction_capture(capc, page, order, migratetype)) {  // compaction 直接"截胡"这块
			account_freepages(zone, -(1 << order), migratetype);
			return;
		}

		buddy = find_buddy_page_pfn(page, pfn, order, &buddy_pfn);  // ← 8.3.2 的 XOR 找 buddy
		if (!buddy)
			goto done_merging;   // buddy 不空闲/不同阶 → 停止合并

		if (unlikely(order >= pageblock_order)) {   // 大块跨 pageblock：要校验类型可合并性
			buddy_mt = get_pfnblock_migratetype(buddy, buddy_pfn);
			if (migratetype != buddy_mt &&
			    (!migratetype_is_mergeable(migratetype) ||
			     !migratetype_is_mergeable(buddy_mt)))
				goto done_merging;   // 类型不兼容（如一个是 ISOLATE）→ 不合并
		}

		if (page_is_guard(buddy))
			clear_page_guard(zone, buddy, order);
		else
			__del_page_from_free_list(buddy, zone, order, buddy_mt);  // 把 buddy 从空闲链表摘下

		if (unlikely(buddy_mt != migratetype))
			set_pageblock_migratetype(buddy, migratetype);

		combined_pfn = buddy_pfn & pfn;        // 合并后大块的起始 PFN = 两者 AND（取低地址那个）
		page = page + (combined_pfn - pfn);    // 指针挪到合并块头部
		pfn = combined_pfn;
		order++;                                // 升一阶，继续尝试和"更大的 buddy"合并
	}

done_merging:
	set_buddy_order(page, order);

	if (fpi_flags & FPI_TO_TAIL)
		to_tail = true;
	else if (is_shuffle_order(order))
		to_tail = shuffle_pick_tail();         // 安全特性：随机化空闲链表头尾，抗堆喷射
	else
		to_tail = buddy_merge_likely(pfn, buddy_pfn, page, order);  // 启发式：未来可能合并的放尾部

	__add_to_free_list(page, zone, order, migratetype, to_tail);   // 终于挂回空闲链表

	if (!(fpi_flags & FPI_SKIP_REPORT_NOTIFY))
		page_reporting_notify_free(order);     // 通知 free page reporting（虚拟化场景把空闲页还宿主）
}
```

**逐行根因与"为什么不用更简单的设计"**：

- `combined_pfn = buddy_pfn & pfn`：合并后大块头 = 两个 buddy 中地址较小者。因为一左一右、低 k 位都是 0、第 k 位一个 0 一个 1，AND 之后第 k 位归 0 = 取左块 = 自然对齐的大块头。**又是位运算 O(1)，不需要 min() 分支**。
- `while (order < MAX_PAGE_ORDER)`：级联向上，最坏 10 次。**为什么不无限合并？** 因为 4MB（order-10）是 zone 内连续性管理的天花板，再大的连续性由 hugetlb/CMA 等上层机制管。
- `shuffle_pick_tail` / `buddy_merge_likely`：两个看似"多余"的启发式其实各有杀机——shuffle 是 **CVE 级安全加固**（让攻击者无法预测下一个分配地址，挫败 heap-spray）；`buddy_merge_likely` 把"未来很可能和邻居合并"的块放链表尾，减少它被过早分走、破坏潜在大块的概率。**简单实现（永远头插）会显著恶化长期碎片化**，这是生产内核踩过坑后加的。
- `compaction_capture`：释放路径上 compaction 能直接"截胡"刚释放的块（不挂回空闲链表而是直接交给正在跑的 compaction），避免"释放→挂链→compaction 再摘下"的来回。

### 8.3.5 外碎片治理：fallback "偷整块"机制（源码 + 场景推演）

当请求类型的链表空了，要从别的类型"借"。fallback 表【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static int fallbacks[MIGRATE_PCPTYPES][MIGRATE_PCPTYPES - 1] = {
	[MIGRATE_UNMOVABLE]   = { MIGRATE_RECLAIMABLE, MIGRATE_MOVABLE   },
	[MIGRATE_MOVABLE]     = { MIGRATE_RECLAIMABLE, MIGRATE_UNMOVABLE },
	[MIGRATE_RECLAIMABLE] = { MIGRATE_UNMOVABLE,   MIGRATE_MOVABLE   },
};
```

注意**这不是一条全局直线，而是按"请求类型"分别定义**——读这张表要按行读。

真正决定"偷一块还是偷整个 pageblock"的是 `can_steal_fallback` + `steal_suitable_fallback`【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static bool can_steal_fallback(unsigned int order, int start_mt)
{
	if (order >= pageblock_order)
		return true;

	if (order >= pageblock_order / 2 ||
		start_mt == MIGRATE_RECLAIMABLE ||
		start_mt == MIGRATE_UNMOVABLE ||      // ← 关键：UNMOVABLE 请求总是允许偷整块
		page_group_by_mobility_disabled)
		return true;

	return false;
}
```

**这段源码直接证明了"为什么 UNMOVABLE 偷 MOVABLE 比反向更激进"**：只要 `start_mt == MIGRATE_UNMOVABLE`，无论 order 多小都返回 true（允许 steal whole block）。根因在 `steal_suitable_fallback` 的判断逻辑里——它会把整个 pageblock 的剩余空闲页和"同类页"算账，达阈值就 `__move_freepages_block` 把整块改成请求类型：

```c
	if (free_pages + alike_pages >= (1 << (pageblock_order-1)) ||
			page_group_by_mobility_disabled) {
		__move_freepages_block(zone, start_pfn, block_type, start_type);  // 整块易主
		return __rmqueue_smallest(zone, order, start_type);
	}
single_page:
	page_del_and_expand(zone, page, order, current_order, block_type);   // 否则只切一块
	return page;
```

> 💡 **底层根因（这是全章最深的一个"为什么"）**：不可移动页一旦散落进 MOVABLE pageblock，**会永久卡死该 pageblock 的 compaction**——因为 UNMOVABLE 页定义上搬不走，compaction 扫到它就被堵。代价是不对称的：让一个 MOVABLE 页落进 UNMOVABLE 块，无非以后多搬一次；但让一个 UNMOVABLE 页落进 MOVABLE 块，是**不可逆的碎片污染**。所以内核宁可"UNMOVABLE 请求一次偷走整个 MOVABLE pageblock 据为己有"，把不可移动页集中关进同一批 block，也不让它们星罗棋布。**这是用空间局部性换取碎片可治理性的经典 trade-off，照搬到任何"有些对象能搬、有些不能搬"的内存池设计都成立。**

`__rmqueue_fallback` 的完整调度逻辑（含 `ALLOC_NOFRAGMENT` 反碎片提升、`find_smallest` 回退）【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
static __always_inline struct page *
__rmqueue_fallback(struct zone *zone, int order, int start_migratetype,
					unsigned int alloc_flags)
{
	...
	int min_order = order;
	...
	if (order < pageblock_order && alloc_flags & ALLOC_NOFRAGMENT)
		min_order = pageblock_order;   // 反碎片模式：宁可从大块偷，避免污染多个块

	for (current_order = MAX_PAGE_ORDER; current_order >= min_order;
				--current_order) {       // 注意：fallback 从最大 order 往下找！
		area = &(zone->free_area[current_order]);
		fallback_mt = find_suitable_fallback(area, current_order,
				start_migratetype, false, &can_steal);
		if (fallback_mt == -1)
			continue;

		/* MOVABLE 请求且不能整块偷 → 去找最小够用的块（不浪费大块） */
		if (!can_steal && start_migratetype == MIGRATE_MOVABLE
					&& current_order > order)
			goto find_smallest;

		goto do_steal;
	}
	return NULL;
	...
}
```

> 🎯 **面试加分点**：注意 fallback 循环是**从 `MAX_PAGE_ORDER` 往下**（`--current_order`），而正常 `__rmqueue_smallest` 是**从请求 order 往上**（`++current_order`）。方向相反不是笔误：正常分配要"最小够用"省内存；fallback 偷别人的块时要"偷最大的"，因为偷大块改类型一次性解决问题、减少跨类型污染的 block 数量。**这个方向差异是区分"读过源码"和"背过八股"的分水岭。**

### 8.3.6 watermark 判定的真相：`__zone_watermark_ok`（逐行）

水位判定的真实实现，**远比"min/low/high 三个数"复杂**【真实源码 v6.12 路径：mm/page_alloc.c】：

```c
bool __zone_watermark_ok(struct zone *z, unsigned int order,
                         unsigned long mark, int highest_zoneidx,
                         unsigned int alloc_flags, long free_pages)
{
	long min = mark;
	int o;

	/* free_pages may go negative - that's OK */
	free_pages -= __zone_watermark_unusable_free(z, order, alloc_flags);  // 扣掉不可用的（如 HIGHATOMIC 储备、CMA）

	if (unlikely(alloc_flags & ALLOC_RESERVES)) {       // 紧急分配可下探储备
		if (alloc_flags & ALLOC_MIN_RESERVE) {
			min -= min / 2;                     // __GFP_HIGH：水位线砍半
			if (alloc_flags & ALLOC_NON_BLOCK)
				min -= min / 4;             // 不可阻塞的原子分配：再砍 1/4（共砍到 3/8）
		}
		if (alloc_flags & ALLOC_OOM)
			min -= min / 2;                     // OOM 受害者可再下探
	}

	/* Check watermarks for an order-0 allocation request */
	if (free_pages <= min + z->lowmem_reserve[highest_zoneidx])  // ★ 真正的硬线
		return false;

	if (!order)
		return true;

	/* For a high-order request, check at least one suitable page is free */
	for (o = order; o < NR_PAGE_ORDERS; o++) {     // 高阶请求：还得确认真有某个 order 有同类空闲块
		struct free_area *area = &z->free_area[o];
		if (!area->nr_free)
			continue;
		for (mt = 0; mt < MIGRATE_PCPTYPES; mt++) {
			if (!free_area_empty(area, mt))
				return true;
		}
		...
	}
	return false;   // free 够多但没有合适大块 → 仍判失败（防止"水位够但分不出连续块"的假阳性）
}
```

**逐行根因**：

- `free_pages <= min + z->lowmem_reserve[highest_zoneidx]`（带 ★ 那行）：这是全章 watermark 最该背的一行。`lowmem_reserve` 是"高 zone 给低 zone 留的命"——防止一个**能落在任意 zone** 的请求（如普通用户页可落 Normal）把**只能落在低 zone** 的请求（如 DMA 分配只能落 DMA32）所依赖的低 zone 抽干。`highest_zoneidx` 越高（请求越"不挑食"），`lowmem_reserve` 偏置越大，**越难从低 zone 拿到页**。
- `min -= min/2` / `min/4`：`__GFP_HIGH`（即 `ALLOC_MIN_RESERVE`）把水位砍半、`ALLOC_NON_BLOCK`（原子上下文）再砍 1/4。**这就是 `GFP_ATOMIC` "优先级高"的物理含义——不是绕过水位，是把水位线压低、动用 MIN 以下的储备。但储备也会耗尽，所以 `GFP_ATOMIC` 同样会失败返回 NULL。**
- 高阶请求的第二个循环：即使 `free_pages` 数量够，还要确认"真有某个 order≥请求 order 的链表非空"。**这防止了一个经典假阳性：碎成 100 万个 order-0、总量充足，但 order-9 一块都没有时，watermark 也判失败而不是骗调用者"有内存"。**

> 🎯 **面试杀手锏**：不少候选人只记 min/low/high 三个数，会被这三连问打穿——(1) 默认快路径对照的是 **LOW** 不是 MIN（`ALLOC_WMARK_LOW`）；(2) 真正能压到 MIN 以下的是带 `__GFP_HIGH` 的紧急分配，且 `min/2`、`min/4` 是真实砍法；(3) 判定式里那个 `+ lowmem_reserve` 偏置，是只读过文档没读过源码的人完全不会提的。

### 8.3.7 NUMA 内存策略：v6.12 内核文档的权威定义

进程级策略（`set_mempolicy`）和 VMA 级策略（`mbind`）的模式，以 [v6.12 内核文档](https://docs.kernel.org/admin-guide/mm/numa_memory_policy.html) 为准：

| 策略 | 行为 | 适用 |
|------|------|------|
| `MPOL_DEFAULT` | **移除当前策略，回退到上一层 scope** | 不是"本地分配"，是"清空、问上一层" |
| `MPOL_BIND` | 严格绑定 nodeset，选其中最近的有内存的 node | 强本地保证，可接受压力下 OOM |
| `MPOL_PREFERRED` | 优先单个 node，失败按距离递增 fallback | 软偏好 |
| `MPOL_PREFERRED_MANY` | 偏好一组 node，压力下 fallback 到全部 node | **近年新增**，比 PREFERRED 更灵活 |
| `MPOL_INTERLEAVE` | 按页粒度在 nodeset 轮转 | 带宽优先（大数组、流式扫描） |
| `MPOL_WEIGHTED_INTERLEAVE` | 按 `/sys/.../weighted_interleave/` 权重轮转 | **近年新增**，异构内存（CXL）带宽配比 |
| `MPOL_LOCAL` | 显式锁定本地 node（实现为 PREFERRED + `MPOL_F_LOCAL`） | 显式本地 |

**策略层级（这是 `mbind` 能覆写 `set_mempolicy` 的根因）**，内核文档定义三层 scope：

> 1. **System Default Policy**（最宽）→ 2. **Task/Process Policy** → 3. **VMA Policy**（最具体）。

优先级 **VMA > task > system default**。`task_struct` 挂 task 策略，`vm_area_struct->vm_policy` 挂 VMA 策略（均需 `CONFIG_NUMA`）。所以 `mbind`（改 VMA）能在 `set_mempolicy`（改 task）之上做局部覆写。

> 💡 **`MPOL_DEFAULT` vs `MPOL_LOCAL` 的细微但要命的差别**（内核文档原话级）：`MPOL_DEFAULT` 是"**fall back to the next most specific policy scope**"——它是个**移除指令**，清掉当前层策略让上一层接管；`MPOL_LOCAL` 是"**显式锁定本地 node**"。嵌套场景下：如果 task 设了 INTERLEAVE，某 VMA 设 `MPOL_DEFAULT` = 该 VMA 回退用 task 的 INTERLEAVE；设 `MPOL_LOCAL` = 该 VMA 强制本地，无视 task 策略。**面试若把两者画等号，是对 NUMA 策略理解不到位的硬伤。**
>
> 还有一句文档级铁律：**cpuset 优先于 mempolicy**——"when both cpusets and policies are applied to a task, the restrictions of the cpuset takes priority"。容器里 mempolicy 选了 node 但 cpuset 没给该 node，分配照样落空。生产排障常见坑。

### 8.3.8 `struct page`：内核最"精神分裂"的 struct（真实源码 + folio/memdesc 未来）

```c
		atomic_t _mapcount;       // 被页表映射的次数
	};
	atomic_t _refcount;           // 引用计数，DO NOT USE DIRECTLY
	...
} _struct_page_alignment;
```

**逐点解读**：

- `union { lru / buddy_list / pcp_list }`：**同一片 16 字节内存，页在 LRU、在 buddy 空闲链表、在 pcp 缓存时含义完全不同**。这就是"调试 struct page 必须先确认身份"的物理来源。
- `unsigned long private`：注释明写 **"Indicates order in the buddy system if PageBuddy"**——8.3.3 里 `set_buddy_order` 存的 order 就放这。同一字段 page cache 时是 buffer_head、swapcache 时是 swp_entry_t。
- `compound_head` 的 `Bit zero is set`：呼应顶部注释"bit 0 must be kept clear"——尾页用 bit 0=1 标识自己是 tail，所以**其他所有复用者都不能碰 bit 0**，否则 `PageTail()` 假阳性。这是整个 union 设计的"宪法第一条"。
- `union { page_type / _mapcount }`：4 字节也复用——typed folio 用 `page_type` 标用途，普通页用 `_mapcount` 记映射次数（初值 -1，方便 `atomic_inc_and_test`/`atomic_add_negative` 追踪 0↔正 的跃迁）。

**内存开销（已核实数字）**：x86-64 上 `struct page` = **64 字节**，开销 **≈ 物理内存 1.5%（每 1TB 吃 16GB，每 8GB 吃约 128MB）**。一个 vmemmap 物理页（4KB）装 64 个 struct page。

> 💡 **演进考古：folio（5.16+）→ memdesc（进行中）**。
> - **folio（Matthew Wilcox，5.16）**：folio 是"保证指向 head page、永不是 tail page"的 wrapper。动机有二（[LWN 849538](https://lwn.net/Articles/849538/)）：(1) **类型混淆**——Wilcox 原话 "A function which has a struct page argument might be expecting a head or base page and will BUG if given a tail page"；(2) **性能**——消除海量 `compound_head()` 调用，实测基于 THP 的工作负载内核编译 **快 7%**。
> - **社区辩论**：**不是一边倒**。Andrew Morton 嫌它制造 "noise"、API 平行膨胀；Hugh Dickins 也冷淡；但 Dave Chinner 站台 "this abstraction is absolutely necessary"。LWN 当时判断"adoption seems unlikely to happen soon"——结果它还是进了主线，这本身是个"大规模重构如何在分歧中落地"的工程案例。
> - **下一步 memdesc**：终极目标是把 `struct page` 从 **64 字节砍到 8 字节**——每个 page 只存一个 "memory descriptor"，要么把元数据编码进这 8 字节，要么是指向独立结构的指针。1TB 内存的 page 元数据开销将从 16GB 降到 2GB。这是 struct page union "精神分裂"问题的根治方案。

## 8.4 HighMem 的前世今生：一个 32 位时代的工程妥协

### 8.4.1 为什么 32 位需要 HighMem（根因：内核虚拟地址不足，与设备无关）

x86-32 地址空间 4GB，经典 3:1 split：

```
0x00000000 ~ 0xBFFFFFFF  用户空间 (3GB)
0xC0000000 ~ 0xFFFFFFFF  内核空间 (1GB)
```

内核要高效访问物理内存，靠**直接（线性）映射**：`virt = phys + PAGE_OFFSET`，换算 O(1)。但 1GB 内核虚拟地址只够直接映射约 **896MB** 物理 RAM（剩下留给 vmalloc/kmap/固定映射）。物理 RAM 超过 896MB 的部分（当年 2GB/4GB 服务器常见）无法常驻直接映射，只能用 `kmap`/`kmap_atomic` **临时建/拆映射**——这块就是 **HighMem**。

### 8.4.2 HighMem 的开销与真坑

- 每次访问要 `kmap` 建映射、`kunmap` 拆，临时映射槽位有限（竞争点）。
- `kmap_atomic` 禁抢占、不能睡眠——用错地方直接 deadlock/调度异常。
- **DMA 设备根本不能用 HighMem**（没有固定总线地址），数据落在 HighMem 还要 bounce 到低端。

这是 32 位内核最大工程痛点之一，也是推 64 位迁移的硬动力。

### 8.4.3 64 位的解法与"为什么 DMA zone 仍在"

x86-64 内核虚拟地址 128TB（48 位）。4 级页表下 direct-map 基址 `PAGE_OFFSET = 0xffff888000000000`，直接映射窗口 64TB；5 级页表换成 `0xff11000000000000`，窗口扩到 32PB。**无论多大物理 RAM 都能常驻直接映射**，`__va(phys) = phys + PAGE_OFFSET` 对全部 RAM 成立——正是 32 位做不到、才被迫发明 HighMem 的事。

`ZONE_HIGHMEM` / `CONFIG_HIGHMEM` **在 x86-64 编译期不存在**（该 Kconfig 仅 32 位架构可见，见 8.3.1 源码的 `#ifdef CONFIG_HIGHMEM`）。

> **但 `ZONE_DMA`/`ZONE_DMA32` 仍在**——这是两个**正交**的问题，新手最爱混：HighMem 是"内核虚拟地址够不够"（64 位解决了），DMA zone 是"设备总线地址够不够"（64 位没解决，因为 32 位 PCI 设备的物理寻址能力是硬件属性）。[LWN 91870（2004）](https://lwn.net/Articles/91870/) 记录的 em64t swiotlb bounce buffer 方案至今仍是处理"老设备 + 大内存"的兜底。**面试若说"64 位连 DMA zone 都不需要了"是错的。**

## 8.5 内存规整（Compaction）：外碎片的外科手术（双扫描器源码）

### 8.5.1 问题场景（生产真实症状）

数据库/虚拟化宿主跑几天后，`/proc/buddyinfo` 显示 order-0 海量空闲，但 order-9（2MB，= `pageblock_order`）接近 0，THP 或 hugepage 分配失败。**总空闲页充足，但分不出连续块**——这就是 8.3.6 里 `__zone_watermark_ok` 第二个循环要防的假阳性，也是 compaction 存在的理由。

### 8.5.2 双扫描器"会师"算法（真实源码佐证）

compaction 的核心是**两个扫描器相向而行、会师即停**。判定会师的真实代码【真实源码 v6.12 路径：mm/compaction.c】：

```c
/*
 * Test whether the free scanner has reached the same or lower pageblock than
 * the migration scanner, and compaction should thus terminate.
 */
static inline bool compact_scanners_met(struct compact_control *cc)
{
	return (cc->free_pfn >> pageblock_order)
		<= (cc->migrate_pfn >> pageblock_order);   // 两扫描器进到同一/交叉 pageblock 即停
}
```

`__compact_finished` 里的会师终止【真实源码 v6.12 路径：mm/compaction.c】：

```c
	/* Compaction run completes if the migrate and free scanner meet */
	if (compact_scanners_met(cc)) {
		/* Let the next compaction start anew. */
		reset_cached_positions(cc->zone);
		...
```

迁移扫描器隔离可搬页的函数签名与注释【真实源码 v6.12 路径：mm/compaction.c】：

```c
/**
 * isolate_migratepages_block() - isolate all migrate-able pages within
 *				  a single pageblock
 * ...
 * Isolate all pages that can be migrated from the range specified by
 * [low_pfn, end_pfn). ...
 * cc->migrate_pfn will contain the next pfn to scan.
 */
static int
isolate_migratepages_block(struct compact_control *cc, unsigned long low_pfn,
			unsigned long end_pfn, isolate_mode_t mode)
```

**算法图解（与源码字段对应）**：

```
低地址 (migrate_pfn →)                      (← free_pfn) 高地址
[UNMOV][MOV free][MOV][free][UNMOV][MOV]...
  ↑ migrate_scanner                    free_scanner ↑
  从低向高找"被占用的 MOVABLE 页"   从高向低找"空闲块"
  把前者内容搬到后者，更新页表          两者 pageblock 交叉 → compact_scanners_met → 停
```

1. **free 扫描器**（`cc->free_pfn`）从高地址向低找空闲块（迁移目的地）；
2. **migrate 扫描器**（`cc->migrate_pfn`）从低地址向高 `isolate_migratepages_block` 隔离可搬的 MOVABLE 页；
3. page migration 子系统把低地址 MOVABLE 页内容复制到高地址空闲块、**更新所有指向它的页表项**；
4. 低地址腾空 → `__free_one_page` 的级联合并掏出大块；
5. `compact_scanners_met` 为真即停（再走下去就重复劳动）。

**触发方式**：自动（慢路径高阶分配失败 → direct compaction）；后台（kcompactd，按 proactiveness score）；手动（`echo 1 > /proc/sys/vm/compact_memory`）。

### 8.5.3 为什么不用更简单的替代？失败模式与真坑

- **为什么不"一次扫全 zone 整理到底"？** 持锁时间会爆炸——compaction 期间相关操作要序列化，整 zone 一把梭会造成长尾分配延迟尖刺。双扫描器 + skip bits + 分 pageblock 处理，是为了**把停顿切碎、可中断**（`ISOLATE_ABORT`/`COMPACT_CONTENDED` 见 8.5.2 源码路径里的 `compact_zone` 主循环）。
- **失败模式 1：UNMOVABLE 污染**。pageblock 里嵌了不可搬页，migrate 扫描器被堵，该块永远整理不出大块。根治靠 8.3.5 的"偷整块"在源头隔离不可移动页。
- **失败模式 2：compaction 抖动（thrashing）**。proactiveness 调太高 → kcompactd 反复搬页烧 CPU 却收益甚微（[LWN 817905](https://lwn.net/Articles/817905/) 提到要"intelligent back-offs"正是为此）。生产里见过 khugepaged/kcompactd 吃满一个核的案例，多半是碎片化已积重难返 + 主动整理过激的叠加。
- **底层根因（排队论视角）**：compaction 是"用 CPU 时间 + 内存带宽（搬页拷贝）换连续性"。当系统已高度碎片化，搬一页的"边际产出连续块"概率很低，**摊销成本急剧上升**——这就是为什么"内存未碎片化时（启动早期）预留 hugepage"远比"运行几天后再 compaction"划算。时间点选错，同样的算法成本差一个数量级。

### 8.6.3 重要 sysctl

| 参数 | 作用 | 生产建议 |
|------|------|----------|
| `vm.watermark_scale_factor` | 调 min/low/high 间距（单位 0.1%，默认 10） | 内存压力大、回收来不及 → 调到 150~200 |
| `vm.min_free_kbytes` | 强制 min watermark 下限 | 大内存机适当调高，给 kswapd 反应时间 |
| `vm.zone_reclaim_mode` | NUMA zone 回收策略 | 0=关（跨 node 优先，page cache 重的负载）；1=开（本地回收优先） |
| `kernel.numa_balancing` | 自动迁热页到本地 node | 计算密集开；延迟敏感关（迁移本身引短暂 miss 升高） |
| `vm.compaction_proactiveness` | 主动 compaction 强度（0~100，默认 20） | 碎片严重可调高；过高烧 CPU（见 8.5.3 thrashing） |

### 8.6.4 大厂实践

- **数据库（MySQL/PG）**：`mlock` 锁内存防换出；通常关 numa_balancing；`zone_reclaim_mode=0`（page cache 局部性收益 > 跨 node 代价）。
- **JVM**：默认 mmap 分配，堆易全落 node 0 → 大堆务必 `-XX:+UseNUMA`（JVM 自做 interleave）或 `numactl --interleave=all`。
- **Redis**：单线程绑核 + `numactl --localalloc` 标配。
- **内存数据库（SAP HANA 级）**：启动 pre-fault 全页（`MADV_WILLNEED` + 全量读 touch），消除运行时 page fault 延迟抖动。

## 8.7 面试视角：Staff 级高频题（含源码级加分点）

### Q1：buddy 如何处理外碎片？compaction 的本质？

**框架**：(1) buddy **不**解决外碎片，只用 2^n 对齐 + `pfn ^ (1<<order)` 让合并成 O(1) 谓词，消的是**内碎片**（分配不会超过 2^n）；(2) 外碎片根因 = UNMOVABLE 页散落 MOVABLE 区阻止合并；(3) **migratetype 分桶 + "偷整块"才是解题关键**（引 `can_steal_fallback` 里 UNMOVABLE 总返回 true）；(4) compaction = 双扫描器会师（`compact_scanners_met`）+ page migration 搬 MOVABLE 页；(5) 高分：kcompactd / proactiveness / "启动早期预留 hugepage"的排队论理由。
**陷阱**：说 buddy"完全消除外碎片"——错，它只消内碎片。

### Q2：32 位为何要 HighMem，64 位为何不要，但 DMA zone 为何仍在？

**框架**：HighMem 根因 = 内核虚拟地址不足（1GB 只够映 ~896MB）；64 位 128TB 虚拟地址全 RAM 直接映射，`CONFIG_HIGHMEM` 编译期消失；但 DMA zone 是**设备总线寻址**问题（正交于虚拟地址），32 位 PCI 设备硬件限制仍在 → 永久保留（引 LWN 91870 swiotlb）。
**陷阱**："64 位连 DMA zone 都不需要"——错。

### Q3：完整描述 `alloc_pages` 快/慢路径

**框架**：
```
快路径 get_page_from_freelist（默认 ALLOC_WMARK_LOW）：
  选 zone（NUMA 策略+gfp）→ zone_watermark_fast/ok：free > WMARK_LOW + lowmem_reserve？
  → __rmqueue_smallest 从 free_area[order][mt] 摘，必要时 expand 拆高 order
慢路径 __alloc_pages_slowpath：
  放宽到 WMARK_MIN（__GFP_HIGH 再砍 min/2、min/4）+ 唤醒 kswapd
  → 高阶：try_to_compact_pages（direct compaction）
  → direct reclaim（调用者阻塞回收 page cache/匿名页）
  → 重试；__GFP_NORETRY/不可阻塞更早返回 NULL；允许则 OOM
```
**陷阱**：`GFP_ATOMIC` "永不失败"——错，只是压低水位动用储备，储备耗尽照样 NULL。
**⏱ 时效注**：旧 `__GFP_ATOMIC` 标志在 **6.3 左右被删**（Mel Gorman "the end of `__GFP_ATOMIC`"）。现"是否可阻塞"看 `__GFP_DIRECT_RECLAIM`（`gfpflags_allow_blocking()`），"是否动用储备"看 `__GFP_HIGH`。组合宏 `GFP_ATOMIC`（≈ `__GFP_HIGH | __GFP_KSWAPD_RECLAIM`）仍在，日常照用。

### Q4：`zone_reclaim_mode` 设 0 还是 1？

**框架**：0（关）= 跨 node 分配优先于回收本地 page cache，适合 page cache 命中率高的负载（DB、文件服务），跨 node 带宽代价 < cache miss 代价；1（开）= 本地回收优先，适合 NUMA 延迟极敏感、本地压力大。近年默认 0（生产数据显示多数场景 cache 局部性收益胜出）。

### Q5：`struct page` union 的工程权衡？

**框架**：(1) 空间——固定 64 字节 / 1TB→16GB / 1.5%；每用途独立结构体会更大；(2) 代价——调试地狱，同字段（如 `private`：buddy 时存 order、cache 时是 buffer_head）含义随身份变，bit 0 永远留给 PageTail；(3) 演进——folio（5.16，消 `compound_head` 调用、内核编译 +7%，但 Andrew Morton/Hugh Dickins 当初反对）→ memdesc（目标 8 字节）；(4) 高分：`PageBuddy()`/`compound_head()`/`set_buddy_order` 存 `page->private`。

### Q6：生产 NUMA miss 飙升怎么排查？（STAR）

**框架**：观测 `numastat -m` + `perf stat -e node-load-misses` + `/proc/<PID>/numa_maps`；根因常见——fork 后调度到远 node 仍访问本地内存 / numa_balancing 迁页临时 miss 升 / 默认策略与 workload 不匹配 / **cpuset 没给 mempolicy 选的 node**；修复 `numactl --cpunodebind+--membind`、调 `zone_reclaim_mode`、按场景关 numa_balancing。高分：numa_balancing 迁热页**本身**会引短暂 miss 升，是 trade-off 不是 bug。

### Q7：watermark 三档在 kswapd / direct reclaim 的角色？

见 8.3.6。要点：kswapd 在 **LOW 唤醒、HIGH 停止**；direct reclaim 在 **MIN 阻塞调用者**；硬线是 `free ≤ min + lowmem_reserve[highest_zoneidx]`；`__GFP_HIGH` 砍 `min/2`、原子再砍 `min/4`。补充 v6.12 有第四档 `WMARK_PROMO`（NUMA 升页用）。

### Q8（源码深挖题）：为什么 fallback 循环从 MAX_PAGE_ORDER 往下找，而正常分配从请求 order 往上找？

见 8.3.5。正常分配要"最小够用"省内存（`++current_order`）；fallback 偷别人块要"偷最大的"一次性改类型、减少被污染的 pageblock 数量（`--current_order`）。**这是区分读过源码 vs 背过八股的题。**

### 目标

在 NUMA 机器（或 QEMU 模拟）上对比本地/远端**带宽**与**延迟**，并理解两者为何差一个数量级。


### 测试程序（可运行，链接 -lnuma）

```c
// numa_bench.c  —— gcc -O2 -D_GNU_SOURCE -o numa_bench numa_bench.c -lnuma
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <time.h>
#include <sched.h>
#include <numa.h>

#define ARRAY_SIZE (256 * 1024 * 1024)  // 256MB
#define STRIDE 64                        // cacheline stride

static inline long ns_now(void) {
    struct timespec ts;
    clock_gettime(CLOCK_MONOTONIC, &ts);
    return ts.tv_sec * 1000000000L + ts.tv_nsec;
}

double bench_read(char *arr, size_t size) {
    volatile char sink = 0;
    long start = ns_now();
    for (size_t i = 0; i < size; i += STRIDE)
        sink ^= arr[i];
    long end = ns_now();
    return (double)(end - start) / 1e9;  // seconds
}

int main(void) {
    if (numa_available() < 0) { fprintf(stderr, "NUMA not available\n"); return 1; }
    int max_node = numa_max_node();
    printf("NUMA nodes: %d\n", max_node + 1);

    char *arr = numa_alloc_onnode(ARRAY_SIZE, 0);   // 内存固定在 node 0
    if (!arr) { perror("numa_alloc_onnode"); return 1; }
    memset(arr, 0xAB, ARRAY_SIZE);                  // fault in pages

    long accesses = ARRAY_SIZE / STRIDE;
    for (int cpu_node = 0; cpu_node <= max_node; cpu_node++) {
        if (numa_run_on_node(cpu_node) != 0) { perror("numa_run_on_node"); continue; }
        sched_yield();                              // 迁移异步，让出一次确保已在目标 node CPU
        double t  = bench_read(arr, ARRAY_SIZE);
        double bw = (double)ARRAY_SIZE / t / 1e9;   // GB/s
        double ns = (t * 1e9) / (double)accesses;   // ns/cacheline（带宽摊销，非延迟！）
        printf("CPU node %d -> mem node 0: %.2f GB/s (%.1f ns/cacheline)\n",
               cpu_node, bw, ns);
    }
    numa_free(arr, ARRAY_SIZE);
    return 0;
}
```

> ⚠️ **这个 benchmark 量的是"带宽"不是"延迟"**：顺序 + 固定 STRIDE 会被硬件 prefetcher 完全预取，掩盖 load-to-use 延迟。它报的 `ns/cacheline` 是**带宽受限的摊销时间**（满带宽下常 1~3ns），**不是**内存延迟。量真·延迟必须 **pointer-chasing**（随机化 next 指针、依赖链强制串行让 prefetcher 失效）——lmbench `lat_mem_rd`、Intel MLC `--latency` 就这么做。**两类数字差一个数量级，面试混为一谈是常见扣分点。**

**预期量级（参考，硬件相关）**：
- **带宽**（本 benchmark 测的）：本地 ~30-80 GB/s（多 channel 叠加）；跨 node 掉 30-60%。
- **延迟**（需另写 pointer-chasing）：本地 DRAM load-to-use ~70-90ns，跨 1 hop ~120-160ns（约 1.5-2×）；多 hop / 大拓扑可达 2-4×（取决于 socket 数与互联代数 QPI/UPI/IF）。

### 验收标准

1. `numactl --hardware` 显示 ≥2 node 拓扑和距离矩阵；
2. 程序输出跨 node 带宽明显低于本地（≥20%）；
3. `numastat -p <PID>` 观察到 `numa_miss > 0`；
4. 能解释 numastat 各字段（numa_hit/numa_miss/local_node/other_node）；
5. **加分**：另写 pointer-chasing 版，对比本地/远端**延迟**倍数，并解释为何与带宽倍数不同。

## 8.9 设计题：NUMA 感知的内存分配器

**场景**：分布式内存数据库，每 shard 绑定固定 NUMA node，shard 数据结构（hash table + B-tree index）本地分配；shard 间查询跨 node 读。

**要求**：(1) mempolicy 选哪个？mbind vs set_mempolicy？(2) shard 迁移（扩容时移到另一 node）；(3) 监控跨 node 访问升高。

**答题要点**：
- 每 shard 用 **`mbind(MPOL_BIND)`** 把该 shard 的 VMA 强绑本地 node（VMA 级 > task 级，8.3.7）→ 强本地保证；
- 迁移：先 `mbind(MPOL_PREFERRED, new_node)` 改偏好，再逐步 fault-in / `migrate_pages` 搬，期间跨 node 不可避免 → 应用层 backpressure；
- 监控：`/proc/<PID>/numa_maps` + `perf node-load-misses` 接告警；
- **先决检查**：容器场景确认 **cpuset 包含目标 node**（cpuset 优先于 mempolicy，否则绑定失效）；
- 高分：NUMA 感知工作队列——计算线程也绑 node，数据与计算 co-located，避免"内存本地但计算在远 node"的反模式。

## 8.10 系统题：512GB NUMA（4 node × 128GB）跑 Java，GC pause 偶发 1s+

**排查框架**：
```
1. 是否 NUMA 相关：
   numastat -m → 各 node 空闲是否严重不均
   grep -c "N1=" /proc/<PID>/numa_maps → 多少 VMA 在非本地 node
   perf stat -e node-load-misses ...
2. JVM NUMA 配置：是否 -XX:+UseNUMA；默认堆可能全落 node 0
3. 量级估算：128GB/node，单 JVM 堆 100GB → 大概率跨 node；
   跨 node 带宽 ~60% 本地 → GC 全堆扫描时带宽降 → STW pause 拉长
4. 修复：
   A. -XX:+UseNUMA（JVM 自做 interleave，首选）
   B. numactl --interleave=all（均摊跨 node）
   C. 每 node 一个 JVM 实例堆绑本地（水平拆分，复杂度高）
5. 验证：GC 日志 pause 对比 + perf node-load-misses before/after
```
**根因点睛**：GC STW 期间要顺序扫描整个堆，是**带宽受限**操作；堆跨 node 时摊到了较慢的 cross-node 带宽，pause 时间 ≈ 堆大小 / 有效带宽，所以"堆越跨、pause 越长"。

## 8.11 代码题：内核模块读取 buddy 信息

```c
// buddy_info.c — 内核模块示意（API/宏为真实，整体为教学示意）
#include <linux/module.h>
#include <linux/mmzone.h>   // struct zone, free_area, NR_PAGE_ORDERS
#include <linux/mm.h>

static int __init buddy_info_init(void)
{
    struct zone *zone;
    int order, mt;

    /* for_each_populated_zone：只遍历有页的 zone（真实宏，linux/mmzone.h） */
    for_each_populated_zone(zone) {
        pr_info("Zone: %s (node %d)\n", zone->name, zone_to_nid(zone));

        /* 循环上界随版本：
         *   6.4 前：for (order = 0; order < MAX_ORDER; order++)        // 排他
         *   6.4 后：for (order = 0; order < NR_PAGE_ORDERS; order++)   // v6.12 用这个
         */
        for (order = 0; order < NR_PAGE_ORDERS; order++) {
            struct free_area *area = &zone->free_area[order];
            pr_info("  order=%d nr_free=%lu\n", order, area->nr_free);
            for (mt = 0; mt < MIGRATE_TYPES; mt++)
                pr_info("    mt=%d empty=%d\n", mt, list_empty(&area->free_list[mt]));
        }
    }
    return 0;
}
static void __exit buddy_info_exit(void) {}
module_init(buddy_info_init);
module_exit(buddy_info_exit);
MODULE_LICENSE("GPL");
```

**更实用的替代（BPFTrace，可直接运行）**：
```bash
# 跟踪 buddy 分配失败（__alloc_pages 返回 NULL）
bpftrace -e '
kretprobe:__alloc_pages { if (retval == 0) { @fails[comm] = count(); } }
interval:s:5 { print(@fails); clear(@fails); }
'
# 先确认 probe 点名：bpftrace -l "kprobe:*alloc_pages*"
```

**暗坑提示**：
1. `MAX_ORDER` 版本陷阱（8.2.2）：6.4 前排他（默认 11，最大 order 10），6.4 起包含（默认 10）并改名 `MAX_PAGE_ORDER`，链表数用 `NR_PAGE_ORDERS`。某些架构/配置（`CONFIG_ARCH_FORCE_MAX_ORDER`，如大页 ARM64）会调大。**永不硬编码，按版本读宏**。
2. 内核模块直接遍历 zone 需持 `zone->lock`，否则数据不一致；`/proc/buddyinfo` 已替你处理。
3. BPFTrace probe 点名跨版本可能变，先 `-l` 确认。

## 章末心法

> **物理内存管理的三句话本质**：
> 1. **用 2^n 对齐 + 一行 XOR，把"能否合并"压成 O(1) 谓词**（消内碎片，buddy）；
> 2. **用 migratetype 分桶 + "偷整块"隔离不可移动页，把外碎片从"不可治"变成"可治"**（compaction 的地基）；
> 3. **用 watermark + lowmem_reserve，把"何时回收/向谁让命"变成可调的延迟/可靠性 trade-off**。
>
> NUMA 在此之上加一层"距离税"——内核帮你本地优先，但策略（含 cpuset 优先级）调错一样踩坑。而 `struct page` 的 union 是这一切的代价载体，folio/memdesc 正在偿还这笔技术债。
