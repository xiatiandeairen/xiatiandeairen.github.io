> **一句话定位**：中断是硬件与内核之间最原始的异步通信通道；上下半部机制是内核为"又快又全"这个矛盾体开出的工程解法。

---

## TL;DR — 本章最硬结论

1. **硬中断（hardirq）上下文严禁睡眠**，必须以最小代价完成"登记"，把真正的工作推给下半部。这是所有中断设计的第一性原理。
2. **softirq / tasklet / workqueue 是下半部的三档变速**：延迟从纳秒到毫秒，并发从 per-CPU 到线程池，选择依据是能不能睡眠、需不需要并发、允不允许同一 handler 多核并行。
3. **中断亲和配置错误是 NUMA 系统性能的隐形杀手**：网卡中断跑到错误的 NUMA node，每个包都要跨 node 访存，延迟翻倍。
4. **NAPI 不是优化而是救火**：中断风暴到一定程度，轮询反而比"每包中断"省 CPU。这是反直觉但正确的结论。
5. **threaded IRQ 是 PREEMPT_RT 的关键路径**：把 hardirq handler 升级成可调度内核线程，让实时任务能抢占"中断服务"，这是 RT 补丁的核心思路之一。

**前置依赖章**：第 11 章（内核并发模型）、第 12 章（锁与同步原语）。第 19 章（设备驱动）是本章的上游消费者，第 20 章（网络协议栈/NAPI）是本章最重要的真实案例。

## 1. 背景：为什么硬件要打断 CPU

### 1.1 轮询的代价

最简单的 I/O 模型：CPU 不停地问"好了吗？好了吗？"——轮询（polling）。磁盘寻道 10ms，网卡帧间隔 100ns，如果用轮询：
- 磁盘 I/O：CPU 空转 10ms，浪费数千万个时钟周期。
- 高速网卡：100ns 一问，CPU 100% 耗在轮询，没资源做别的事。

中断模型把这个矛盾反转：**让硬件主动通知 CPU**，CPU 做别的事情，硬件就绪时发信号。代价是上下文切换开销（约 1–10μs 量级，依平台），但比轮询的空转划算得多。

### 1.2 "又快又全"的矛盾

中断触发时，CPU 必须**立刻**响应，否则：
- 硬件 FIFO 溢出（网卡 ring buffer 满了丢包）
- 丢失时间戳（音频采样、工业控制）
- 触发 watchdog

但中断处理要做的事往往很多：
- 解析协议头（网络）
- 更新块设备 I/O 状态（存储）
- 通知上层应用

这些工作耗时长，且可能需要睡眠（等待锁、等待内存分配）。**硬件中断上下文严禁睡眠**——睡眠意味着调度，调度意味着换任务，但中断上下文没有对应的 `task_struct` 来保存"继续点"，内核会直接 panic。

**解法**：分治。把紧急的最小操作留在硬中断，把可延迟的工作推到"下半部"。这是所有现代 OS 的共识。

## 2. 中断控制器与中断号

### 2.1 从物理 IRQ 线到向量号

```
  外设                  中断控制器             CPU
  ┌─────┐              ┌──────────────────┐   ┌───────────────────┐
  │ NIC │──IRQ线──────▶│  APIC/GIC/PLIC  │──▶│ vector table      │
  │ SSD │──IRQ线──────▶│  IRQ 路由与优先级 │   │ IDT (x86)         │
  │ USB │──IRQ线──────▶│  亲和性配置       │   │ VBAR (ARM)        │
  └─────┘              └──────────────────┘   └───────────────────┘
```

- **x86**：早期用 8259A PIC（可级联，16 条 IRQ 线），SMP 时代被 **APIC（Advanced PIC）** 替代，含本地 LAPIC（每 CPU）+ I/O APIC（连外设）。向量号 0–255，前 32 个是 CPU 异常，32–255 是可分配 IRQ 向量。
- **ARM**：使用 GIC（Generic Interrupt Controller），ARM GICv3/v4 支持 LPI（Message Signaled 中断）和 ITS（Interrupt Translation Service），是 PCIe MSI-X 在 ARM 平台的实现路径。
- **RISC-V**：PLIC（Platform-Level Interrupt Controller）。

Linux 内核用**虚拟 IRQ 号（virq）**抽象底层差异。驱动看到的 `request_irq(irq, ...)` 里的 `irq` 是 virq，内核通过 irqdomain 映射到硬件 IRQ 号。

### 2.2 MSI 与 MSI-X：从共享 IRQ 线到消息

传统 PCI 设备共享 IRQ 线，存在两个问题：
1. **共享中断**：多个设备共用一条 IRQ，handler 需要轮询谁发的——慢且耦合。
2. **线数量限制**：主板 IRQ 管脚有限。

**MSI（Message Signaled Interrupts）**：设备通过**写一个特殊内存地址**（通常是 APIC 的寄存器地址）来触发中断，这个"消息"携带中断向量号。核心优势：
- 不再需要共享 IRQ 线，消除了共享中断的虚假触发（spurious IRQ）。
- 中断消息写入有天然的写入顺序保证，解决了旧式边沿触发的竞态问题。

**MSI-X**：MSI 的扩展版，一个设备最多支持 2048 个独立 MSI-X 向量。高速网卡利用这一点：
```
  NIC 4 队列
  ├── TX Queue 0 ──▶ MSI-X vector 0 ──▶ CPU 0 的 LAPIC
  ├── TX Queue 1 ──▶ MSI-X vector 1 ──▶ CPU 1 的 LAPIC
  ├── RX Queue 0 ──▶ MSI-X vector 2 ──▶ CPU 0 的 LAPIC
  └── RX Queue 1 ──▶ MSI-X vector 3 ──▶ CPU 1 的 LAPIC
```

每个队列一个向量，每个向量绑定一个 CPU，**消除了 SMP 下的 IRQ 竞争**。这就是为什么高性能网卡配置要做 IRQ affinity + RPS/RFS 配对（见第 20 章）。

💡 **坑点**：虚拟机环境里，vCPU 的 MSI-X 向量由 hypervisor 虚拟，大量 VM 共一个物理 LAPIC 时，中断延迟会显著增加。裸金属和 VM 的中断延迟基线差异可达 5-10x。

### 2.3 内核 IRQ 抽象层

```c
/* 驱动注册中断 handler（待核：具体参数以当前内核 include/linux/interrupt.h 为准）*/
request_irq(unsigned int irq,
            irq_handler_t handler,
            unsigned long flags,
            const char *name,
            void *dev)
```

关键 flags：
| flag | 含义 |
|------|------|
| `IRQF_SHARED` | 允许共享 IRQ 线（旧 PCI 设备需要） |
| `IRQF_TRIGGER_*` | 触发方式（边沿/电平，高/低） |
| `IRQF_DISABLED` | （已彻底删除，约 4.1 / 2015 移除。它早就是 NOOP——现代内核 hardirq handler 一律在关本地中断的状态下运行，无需此 flag。见到老代码用它直接删） |
| `IRQF_THREAD` | 强制线程化（见 threaded IRQ 节） |
| `IRQF_NO_THREAD` | 禁止线程化（即使系统开了 `threadirqs`） |

## 3. 上下半部机制

### 3.1 第一性原理：hardirq 上下文的约束

当 CPU 收到中断信号：
1. 完成当前指令（x86 大多数指令是原子的）
2. 保存现场（CS/RIP/RFLAGS/RSP 压栈，或 ARM 的 LR/SPSR）
3. 切到中断向量对应的 handler
4. Handler 执行完毕，恢复现场，`iret`/`eret` 返回

整个过程中：
- **本地 CPU 中断是禁用的**（同优先级或更低优先级中断不能打断，x86 默认行为，APIC 也支持中断优先级抢占）
- **没有进程上下文**，不能调用任何可能睡眠的函数（`mutex_lock`、`kmalloc(GFP_KERNEL)`、`copy_to_user` 等）
- **运行时间要极短**，否则其他 CPU 的中断延迟跟着升高（在 SMP 下，只有本地 CPU 的中断被禁，其他 CPU 不受影响，但高频 handler 仍是全局 CPU 负担）

下面用一张对照表展示各上下文的约束：

| 上下文 | 可否睡眠 | 可否持有 mutex | 可用 spinlock | 典型操作 |
|--------|---------|----------------|---------------|---------|
| hardirq | ✗ | ✗ | ✓（关中断版） | 读寄存器、队列登记、唤醒 softirq |
| softirq | ✗ | ✗ | ✓ | 协议处理（网络）、定时器回调 |
| tasklet | ✗ | ✗ | ✓ | 轻量延迟，与 softirq 同层 |
| workqueue（普通） | ✓ | ✓ | ✓ | 任意可睡眠工作 |
| threaded IRQ | ✓ | ✓ | ✓ | 可睡眠的"中断处理" |

### 3.2 softirq — 最低延迟的下半部

softirq 是内核内置的、**静态分配**的软件中断机制，数量极少（当前内核确切是 10 个，由 `include/linux/interrupt.h` 里的 `enum` 定义，`NR_SOFTIRQS == 10`）。"静态"是关键：softirq 的种类编译期就定死了，运行时不能新增，这跟下面 tasklet/workqueue 的"动态注册"形成对比。

```
softirq 类型（这是当前内核 enum 的实际顺序，已对照 include/linux/interrupt.h 确认，编号即优先级，越小越先跑）：
  HI_SOFTIRQ        = 0   — 最高优先级 tasklet
  TIMER_SOFTIRQ     = 1   — 定时器轮（timer wheel）到期
  NET_TX_SOFTIRQ    = 2   — 网络发送
  NET_RX_SOFTIRQ    = 3   — 网络接收（NAPI 入口）
  BLOCK_SOFTIRQ     = 4   — 块设备 I/O 完成
  IRQ_POLL_SOFTIRQ  = 5   — blk-mq 轮询（irq_poll 库）
  TASKLET_SOFTIRQ   = 6   — 普通 tasklet
  SCHED_SOFTIRQ     = 7   — 负载均衡
  HRTIMER_SOFTIRQ   = 8   — 高精度定时器（独立 vector，不是 TIMER 的别名；
                            它在 4.16 被重新启用，承载 hrtimer 的 softirq 回调路径）
  RCU_SOFTIRQ       = 9   — RCU 回调处理
  （NR_SOFTIRQS = 10，所以"约 10 个"是确切的 10 个）
```

⚠ 注意区分 `TIMER_SOFTIRQ`（经典 timer wheel）和 `HRTIMER_SOFTIRQ`（高精度定时器），二者是两个独立 vector，**没有合并**。网上不少老资料把它们混在一起，是过时信息。

**执行时机**：
1. hardirq handler 返回时（`irq_exit()` 路径，待核：确切函数名以源码为准）
2. `local_bh_enable()` 重新使能 bottom half 时
3. `ksoftirqd` 内核线程被唤醒时（负载高时 softirq 会推给这个线程）

softirq 的关键性质：
- **per-CPU 执行**：同一个 softirq 可以在多个 CPU 上**同时**执行，handler 必须自己保证并发安全（通常用 per-CPU 数据或 spinlock）。
- **不可被相同 softirq 重入**：同一 CPU 上，正在执行 NET_RX_SOFTIRQ 时，不会再触发 NET_RX_SOFTIRQ（但其他 softirq 类型可以）。

🎯 **面试加分点**：面试官常问"softirq 和 tasklet 的区别"。除了"tasklet 不可多核并行"（同一 tasklet 同时只在一个 CPU 上执行），更核心的是：**softirq 是静态编译进内核的，tasklet 是动态注册的**。驱动写不了 softirq（那是内核子系统的特权），只能用 tasklet 或 workqueue。

### 3.3 tasklet — softirq 的轻量封装

tasklet 建立在 `TASKLET_SOFTIRQ`（普通）和 `HI_SOFTIRQ`（高优先级）之上，提供给驱动用的下半部机制。

**关键约束**：同一个 tasklet 同时只在一个 CPU 上运行。内核用原子位 `TASKLET_STATE_SCHED` 和 `TASKLET_STATE_RUN` 来保证这一点（待核：具体字段名以 `include/linux/interrupt.h` 的 `tasklet_struct` 为准）。

```
tasklet_struct {
    next        // 链表指针（per-CPU 的 tasklet 队列）
    state       // 原子状态位
    count       // 禁用计数（>0 表示被禁用）
    callback    // 你的 handler
    data        // 传给 handler 的指针
}
```

调用路径：
```
hardirq handler
    └── tasklet_schedule(&my_tasklet)
            └── 设置 TASKLET_STATE_SCHED
            └── 加入 per-CPU 的 tasklet_vec 链表
            └── raise_softirq(TASKLET_SOFTIRQ)
                    ↓（hardirq 退出后）
            ksoftirqd 或 irq_exit 中
                └── tasklet_action()
                        └── 遍历 tasklet_vec
                        └── 清 TASKLET_STATE_SCHED，设 TASKLET_STATE_RUN
                        └── 调用 callback()
                        └── 清 TASKLET_STATE_RUN
```

**新手误区**：以为 tasklet 是多线程的。不是——同一个 tasklet 的 callback 严格串行，不需要在 callback 里加锁保护 tasklet 自己的私有数据。

💡 **坑点 / 时效（2026 视角）**：tasklet 自 **5.14（2021）** 起在内核文档里被明确标注"不要在新代码里用"，社区已经讨论彻底删除多年。替代路径有两条，按需求选：
- 需要能睡眠 → **threaded IRQ**（`request_threaded_irq` / `IRQF_THREAD`）或 **workqueue**。
- 仍需要 softirq 那种"低延迟、原子上下文"语义但不想用 tasklet → 近年内核引入了 **`WQ_BH` 类型的 workqueue**（workqueue maintainer Tejun Heo 的工作），让 work 直接跑在 softirq/BH 上下文，作为 tasklet 的平替，社区正在逐个把老 tasklet 用户迁过去。

为什么淘汰它：tasklet 跑在不可抢占的 softirq 上下文，是 PREEMPT_RT 实时性的障碍；而且"同一 tasklet 全局串行"这个语义会让一个慢 tasklet 拖累整条软中断路径。**面试若问"为什么 tasklet 要被淘汰"，答到"实时性障碍 + 全局串行的隐藏序列化 + 有了 threaded IRQ / WQ_BH 更好的替代"就到位了。**

### 3.4 工作队列（workqueue）— 可睡眠的下半部

workqueue 把"要延迟执行的工作"提交给内核工作线程（worker thread）来执行。关键突破：**工作线程是真正的内核线程，有进程上下文，可以睡眠**。

**cmwq（Concurrency Managed Workqueue）**，近年内核（约 2.6.36 后）引入，核心思想：

```
                per-CPU bound workqueue
  ┌─────────────────────────────────────────────────────┐
  │  CPU 0                    CPU 1                      │
  │  worker pool (bound)      worker pool (bound)         │
  │  ┌──────────────────┐    ┌──────────────────┐        │
  │  │ worker1 (运行中)  │    │ worker3 (运行中)  │        │
  │  │ worker2 (阻塞中)  │    │ worker4 (空闲)    │        │
  │  └──────────────────┘    └──────────────────┘        │
  │                                                       │
  │  unbound workqueue（可跨 CPU 迁移）                   │
  │  ┌─────────────────────────────────────┐             │
  │  │  全局 worker pool，按 nice 值分组     │             │
  │  └─────────────────────────────────────┘             │
  └─────────────────────────────────────────────────────┘
```

cmwq 的动态 worker 管理：
- 一个 work 阻塞（睡眠）时，pool 会动态创建新 worker 保证其他 work 不被饿死。
- idle worker 超时后自动销毁，避免线程泄漏。
- 目标：让每个 CPU 的 worker pool 始终有且仅有一个活跃运行（非阻塞）的 worker，避免过度并发。

常用 API（高把握）：
```c
/* 系统默认工作队列，可直接用 */
schedule_work(&my_work);           /* 提交到 system_wq，bound to current CPU */
schedule_delayed_work(&dw, delay); /* 延迟提交 */

/* 自定义工作队列（需要隔离或特殊属性时）*/
wq = alloc_workqueue("my_wq", WQ_UNBOUND | WQ_HIGHPRI, max_active);
queue_work(wq, &my_work);
destroy_workqueue(wq);
```

关键 flags：
| flag | 含义 |
|------|------|
| `WQ_UNBOUND` | 不绑定 CPU，工作可迁移，适合长耗时任务 |
| `WQ_HIGHPRI` | 高优先级 worker pool |
| `WQ_MEM_RECLAIM` | 工作队列参与内存回收，防止死锁（预留应急 worker） |
| `WQ_FREEZABLE` | 系统 suspend 时冻结此队列 |

### 3.5 三者对比速查

```
              softirq           tasklet          workqueue
────────────────────────────────────────────────────────────────
可睡眠         ✗                 ✗                ✓
多核并行       ✓（同类型可并行） ✗（同实例串行）   ✓（多 worker）
注册方式       静态（内核子系统） 动态（驱动）      动态（驱动/内核）
典型用途       网络接收/定时器   旧式驱动下半部    驱动 I/O 完成
延迟量级       ~微秒             ~微秒             ~毫秒（可调）
PREEMPT_RT     ✗（非实时）       ✗（非实时）       ✓（线程可抢占）
新代码推荐     ✗（不开放）       ✗（已弃用）       ✓
```

## 4. 中断风暴与 NAPI

### 4.1 为什么每包一个中断会压垮系统

1Gbps 网卡，64 字节最小帧，线速约 1.488M pps（packets per second，含前导/IFG 的理论上限）。若每包一个中断：
- 1.488M 次/s hardirq，即每个包只有约 670ns 的"预算"。
- 而单次 hardirq 的进入/退出开销（保存恢复寄存器、流水线打断、cache/TLB 抖动）就是 ~1μs 量级，依平台浮动——**单次中断的固定开销已经大于每包预算**。
- 结论：CPU 会 100% 耗在"进中断/出中断"上，几乎没有周期留给真正处理包，网卡 FIFO 随即溢出丢包。

（注意上面两个量级别打架：670ns 是"两个包之间的时间间隔"，~1μs 是"处理一次中断的固定成本"。成本 > 间隔，系统就被中断淹没了——这正是中断风暴的本质。）

这就是**中断风暴（interrupt storm）**。网卡 FIFO 还会因为 CPU 处理不过来而溢出丢包。

### 4.2 NAPI：以轮询代替中断

**NAPI（New API）** 是 Linux 网络子系统的解法（约 2.4 内核引入，已是标配）：

```
正常流量（低负载）：
  包到达 → IRQ → hardirq handler（登记 NAPI） → softirq NET_RX
                                                     ↓
                                              napi_poll()
                                                     ↓
                                              一次处理 N 个包（budget）
                                                     ↓
                                              如果队列空了 → 重新开 IRQ，退出轮询
                                              如果 budget 用完 → yield，下次 softirq 继续


高负载（包速很快）：
  第一个包 → IRQ → 禁用该 NIC 的 IRQ → 进入轮询模式
                                              ↓
                                        持续 napi_poll()
                                              ↓
                                        budget 内处理尽量多的包
                                        没有更多包 → 重开 IRQ
```

核心思路：**用一个中断"启动引擎"，然后轮询直到队列空**，避免每包一个中断的开销。

关键参数（这里有一个高频被讲错的点，单列说清）：
- **单次 poll 的 per-NAPI budget**：由 `net.core.dev_weight` 控制，**默认 64**（高把握）。这是单个 NAPI 实例一次 `->poll()` 最多处理的包数。
- **一次 softirq 调用的全局 budget**：由 `net.core.netdev_budget` 控制，**默认 300**（高把握）。一次 `NET_RX_SOFTIRQ` 里要轮所有挂在本 CPU 上的 NAPI 实例，这 300 是它们处理包数的总预算，跑满就让出 CPU、把剩余工作丢回下一轮 softirq / ksoftirqd。
- 还有 `net.core.netdev_budget_usecs`（默认约 2000μs，依版本）：一次轮询的时间上限，包数没到 300、时间到了也退出，防止 softirq 长期霸占 CPU。
- 三者的关系：**`dev_weight` 管"一个队列一次吃多少"，`netdev_budget`/`netdev_budget_usecs` 管"整轮 softirq 总共干多久"**。把这俩混为一谈是面试常见扣分点。
- NAPI 相关 API（高把握）：`napi_schedule()`、`napi_complete_done()`、`napi_enable()`/`napi_disable()`。驱动的 `->poll()` 返回实际处理的包数；若 < budget 则调 `napi_complete_done()` 退出轮询并重开 IRQ。

🎯 **面试加分点**：NAPI 解决了高负载下的中断风暴，但在低负载时会引入轮询延迟（要等到下次 softirq 才处理包）。这是一个经典的吞吐 vs 延迟 trade-off。现代驱动通常用自适应 NAPI（adaptive coalescing），轻负载时倾向中断，重负载时倾向轮询。

## 5. 中断亲和与负载均衡

### 5.1 IRQ affinity 的重要性

IRQ affinity 控制**哪个 CPU（们）负责服务某个 IRQ**。配置接口：

```bash
# 查看 IRQ 24 的当前 affinity（CPU 位掩码，十六进制）
cat /proc/irq/24/smp_affinity

# 把 IRQ 24 绑定到 CPU 0 和 CPU 1（位掩码 0x3）
echo 3 > /proc/irq/24/smp_affinity

# 查看所有中断分布
cat /proc/interrupts
```

`/proc/interrupts` 格式：
```
           CPU0       CPU1       CPU2       CPU3
  0:          9          0          0          0   IO-APIC    2-edge      timer
 24:      15432       8721          0          0   PCI-MSI 524288-edge   eth0-rx-0
 25:          0          0      12891       9103   PCI-MSI 524289-edge   eth0-rx-1
```

每一行：IRQ 号、每 CPU 的中断次数、中断控制器类型、名称。

### 5.2 NUMA 与中断亲和

NUMA 系统中，中断亲和配置错误会导致**跨 node 内存访问**：

```
  NUMA Node 0                        NUMA Node 1
  ┌───────────────────────────┐      ┌───────────────────────────┐
  │  CPU 0, CPU 1             │      │  CPU 2, CPU 3             │
  │  NIC 物理上接在 PCIe Bus  │      │  内存 Bank                 │
  │  本地 L3 Cache            │ ~~~~ │  远端内存                  │
  └───────────────────────────┘      └───────────────────────────┘
        ↑
  IRQ 错误绑定到 CPU 2/3？
  每个收到的包的 sk_buff 分配在 node 1 的内存
  但 NIC DMA 写入的是 node 0 的 ring buffer
  → 数据搬运跨 NUMA，延迟 +50-100%
```

正确做法：**把网卡的 IRQ 绑到网卡所在 PCIe Bus 关联的 NUMA node 上的 CPU**。

查看设备的 NUMA node：
```bash
cat /sys/bus/pci/devices/<PCI_ADDR>/numa_node
# 或
lspci -v | grep -i numa  # 不是所有版本支持
numactl --hardware        # 查看 NUMA 拓扑
```

工具：`irqbalance` 守护进程自动做 IRQ 分发，但大流量服务器通常**禁用 irqbalance**，改为手动精确绑定（irqbalance 的启发式算法不了解应用层的 CPU 分工）。

### 5.3 中断负载均衡最佳实践（大厂做法）

高性能网络服务器的标准配置：
1. **关闭 irqbalance**
2. 每个 RX 队列绑定一个 CPU（网卡 RSS 队列数 = CPU 核数）
3. 网卡中断 IRQ affinity = 对应 CPU
4. RPS/RFS（软件层面多队列，补充硬件 RSS 的覆盖）
5. 应用层线程同样 pin 到对应 CPU（`sched_setaffinity`）
6. 整条路径 CPU → L3 Cache → 内存全在同一 NUMA node

💡 **坑点**：超线程（Hyper-Threading）下，物理核的两个逻辑 CPU 共享 L1/L2 Cache。把两个 NIC 队列的 IRQ 分别绑到同一物理核的两个 HT 线程，吞吐提升有限且 Cache 竞争激烈。大流量场景建议把 IRQ 绑到不同物理核。

## 6. Threaded IRQ

### 6.1 为什么需要线程化中断

传统 softirq/tasklet 下半部运行在软中断上下文：**不可抢占、不可睡眠、对实时任务不友好**。PREEMPT_RT 补丁集的核心目标之一就是消除不可抢占的内核区域，让实时任务能随时抢占。

**Threaded IRQ（线程化中断）**：把 hardirq handler 的工作分成两部分：
1. **primary handler**（仍在 hardirq 上下文）：只做最紧急的寄存器读写，返回 `IRQ_WAKE_THREAD` 唤醒线程。
2. **thread handler**（在独立内核线程中运行）：真正的中断处理逻辑，运行在进程上下文，**可以被更高优先级任务抢占，可以睡眠**。

```c
/* 驱动注册 threaded IRQ（待核：参数以当前内核 include/linux/interrupt.h 为准）*/
request_threaded_irq(irq,
                     primary_handler,   /* hardirq 上下文，只做最小工作 */
                     thread_fn,         /* 线程上下文，可睡眠 */
                     flags,
                     name,
                     dev_id);
```

内核为每个 threaded IRQ 创建一个实时内核线程（通常名为 `irq/<IRQ_NUM>-<name>`）：
```bash
ps -eLf | grep "irq/"
# 输出示例：
# root       234     2   234  0   1 irq/24-eth0
# root       235     2   235  0   1 irq/25-eth0
```

这些线程默认跑 `SCHED_FIFO`、优先级 **50**（高把握：源码里是 `MAX_RT_PRIO/2`，RT 优先级范围 1–99 的正中间），运行时可用 `chrt -f -p <prio> <tid>` 调整。

⚠ 一个容易混淆的点：`irq/<N>-<name>` 线程在两种情况下都会出现——(1) PREEMPT_RT 内核，(2) 普通内核但启动加了 `threadirqs`，或某个驱动显式用 `IRQF_THREAD` / `request_threaded_irq` 注册。换句话说，**线程化中断不是 RT 内核的专利**，普通内核也能用，只是 RT 内核把它变成了全局默认。优先级 50 这个默认值在两种内核里一致。

### 6.2 全系统 threadirqs

启动参数加 `threadirqs` 可以把**所有**中断强制线程化（标记了 `IRQF_NO_THREAD` 的除外）。这是 PREEMPT_RT 内核的默认模式。

好处：
- 实时任务可以抢占任意中断处理线程
- 中断处理延迟变得可调度、可预测

代价：
- 线程切换开销叠加到每次中断（额外 1-5μs 量级）
- 高中断频率场景（如高速网卡）性能下降

🎯 **面试加分点**：PREEMPT_RT 已逐步合入主线（近年内核，5.x/6.x 逐步推进），threaded IRQ 是其技术核心之一。问"如何保证实时任务不被网络中断影响"，答案就是：threaded IRQ + 降低中断线程优先级 + CPU 隔离（`isolcpus`）。

## 7. 中断上下文的限制总结

```
  hardirq 上下文的绝对禁忌
  ─────────────────────────
  ✗ 任何可能睡眠的操作：
      mutex_lock(), down(), wait_event()
      kmalloc(GFP_KERNEL)  ← 可能触发内存回收
      copy_to/from_user()  ← 可能 page fault → 睡眠
      schedule()           ← 直接触发调度
      msleep() / usleep()  ← 睡眠函数
  
  ✓ 允许的操作：
      spin_lock_irqsave()  ← 关中断版 spinlock
      kmalloc(GFP_ATOMIC)  ← 从应急内存池分配，不睡眠
      tasklet_schedule()   ← 唤醒下半部
      wake_up()            ← 唤醒等待队列中的进程（不是立即调度）
      读写硬件寄存器（ioremap 后的）
      per-CPU 变量访问（天然无竞争）
```

检测误用的工具：
- `might_sleep()` 注解：内核函数在可能睡眠的位置调用这个宏，如果在中断上下文调用会触发 WARN_ON。
- `CONFIG_DEBUG_ATOMIC_SLEEP`：编译选项，开启后在 atomic 上下文（持有 spinlock、禁中断、hardirq）中调用可睡眠函数会立即内核 BUG。
- `preempt_count()`：返回当前的抢占/中断嵌套计数，可用于调试。

## 8. 生产调优与排障

### 8.1 观测工具

```bash
# 查看中断分布（动态刷新）
watch -n 1 cat /proc/interrupts

# 查看软中断统计
cat /proc/softirqs

# 实时中断率（依赖 perf 或 /proc 差值）
perf stat -e irq:irq_handler_entry -a sleep 1

# 查看 ksoftirqd 和 irq 线程的 CPU 使用
top -H  # 或 htop，过滤 ksoftirqd / irq/
```

`/proc/softirqs` 输出示例：
```
                    CPU0       CPU1       CPU2       CPU3
          HI:          1          0          0          0
       TIMER:    1234567    1245678    1198234    1267891
      NET_TX:       1234        456        789       1012
      NET_RX:    4567890    4321098    4412345    4567123
       BLOCK:     234567     245678     198234     267891
    IRQ_POLL:          0          0          0          0
     TASKLET:      12345      11234      13456      12789

         RCU:    5678901    5789012    5456789    5678901
```

重点关注：
- `NET_RX` 严重不均衡 → IRQ affinity 或 RSS 配置有问题
- `SCHED` 异常高 → 负载均衡频繁，可能是调度域配置问题（见第 6 章）
- `RCU` 异常高 → RCU 回调积压（见第 13 章）

### 8.2 中断风暴排障

症状：`/proc/interrupts` 某 IRQ 计数每秒暴增，CPU 使用率高但业务吞吐不增加。

排查步骤：
1. 确认是哪个设备：`cat /proc/interrupts` + IRQ 名称
2. 网卡：检查 `ethtool -S <dev>` 的 `rx_missed_errors`/`rx_fifo_errors`（丢包信号）
3. 检查是否 NAPI 在工作：`ethtool -c <dev>`（coalesce 参数），`rx-usecs`/`rx-frames` 是否合理
4. 调整中断聚合：`ethtool -C <dev> rx-usecs 100 rx-frames 64`（延迟 100μs 或收集 64 帧再中断一次）
5. 极端情况（调试用）：`echo 0 > /proc/irq/<N>/smp_affinity_list` 将中断禁止（危险，仅调试）

### 8.3 P2 项目：观察 /proc/interrupts 与中断分布

**目标**：理解系统中断分布，找到网卡中断的 NUMA 亲和性，观察软中断统计，用 eBPF/bpftrace 追踪单次 hardirq 延迟。

**步骤**：

```bash
# 1. 基线观察：记录 1 秒内中断增量
T1=$(cat /proc/interrupts); sleep 1; T2=$(cat /proc/interrupts)
# 手动或用脚本对比两次快照，找出 delta 最大的 IRQ

# 2. 找出网卡的 IRQ 和当前 affinity
ip link  # 找到网卡名，如 eth0
cat /proc/interrupts | grep eth0
cat /proc/irq/<IRQ_NUM>/smp_affinity_list

# 3. 查看 NUMA 拓扑
numactl --hardware
cat /sys/bus/pci/devices/$(ethtool -i eth0 | grep bus-info | awk '{print $2}')/numa_node

# 4. bpftrace 追踪 hardirq 延迟（示意，需 root + bpftrace 安装）
bpftrace -e '
tracepoint:irq:irq_handler_entry { @start[args->irq] = nsecs; }
tracepoint:irq:irq_handler_exit  {
    if (@start[args->irq]) {
        @latency = hist(nsecs - @start[args->irq]);
        delete(@start[args->irq]);
    }
}
interval:s:5 { print(@latency); clear(@latency); }
'

# 5. 观察软中断统计变化（需要产生流量）
watch -n 1 cat /proc/softirqs
# 同时在另一终端：iperf3 -c <target> -t 30
```

**验收标准**：
- 能说出系统中排名前 5 的 IRQ 是什么设备、触发频率多少
- 能识别网卡中断绑定的 CPU 是否与网卡的 NUMA node 对齐
- bpftrace 脚本能输出 hardirq handler 的延迟直方图（p50/p99）
- 能解释 `/proc/softirqs` 中 NET_RX 数字的含义

## 9. 未来演进

### 9.1 PREEMPT_RT 全面主线化

PREEMPT_RT 补丁集在 Linux 6.x 持续合入主线（进度待核：具体合入版本以 kernel.org 最新状态为准）。其核心变更与中断相关：
- 所有 spinlock 变为 sleeping spinlock（底层是 rtmutex）
- 中断处理全面线程化
- softirq 在特定配置下也线程化

影响：嵌入式实时系统（工控、机器人、汽车 ECU）无需独立维护 RT 内核补丁，主线 + 配置即可。

### 9.2 中断虚拟化与 virt 场景

云环境中，物理中断由 hypervisor 拦截并模拟给 vCPU。新技术路径：
- **Posted Interrupts**（Intel VT-x）：允许 hypervisor 把中断直接"发布"给 vCPU，减少 VM-exit 次数，降低虚拟化中断延迟。
- **SR-IOV + 直接中断分配**：网卡 VF 的 MSI-X 向量直接映射到 VM，hypervisor 不参与数据面中断路径（DPDK/VF passthrough）。

### 9.3 Rust 驱动与中断安全

Rust for Linux 的中断处理框架在设计上用类型系统强制区分中断上下文：
- 在 IRQ handler 上下文中，Rust 的类型系统可以静态阻止分配需要睡眠的对象（通过不同的 Allocator trait 约束）。
- 与 C 的 `might_sleep()` 运行时检查相比，这是编译期保证。

这是 Rust for Linux 在安全性上相对 C 内核代码的一个真实优势。

## 章末五件套

### 高频面试题（Staff 级）

**Q1：为什么 hardirq handler 不能睡眠？说清楚机制，不要只说"规定"。**

答题框架（从"睡眠到底是什么"倒推）：
- 睡眠 = 最终走到 `schedule()` = 把"当前执行状态"保存进某个 `task_struct`、然后切到另一个任务。
- hardirq 上下文是**借用了被中断那个任务的内核栈在跑**，它本身不是一个独立可调度实体，没有自己的 `task_struct` 当容器。真要在这里 `schedule()`：
  - 你保存/恢复的是被中断任务的上下文，等于偷了别人的身份去睡觉；
  - 当前 `preempt_count` 里 hardirq 嵌套计数非零、本地 IF 还是关的，新选中的任务会带着"中断被禁、抢占计数错乱"的状态跑起来——轻则这颗 CPU 此后收不到中断，重则直接死锁/panic。
- 内核怎么拦：可睡眠函数入口普遍调 `might_sleep()`；调度核心检查 `preempt_count()`（含 `in_interrupt()` 这一类标志）；开了 `CONFIG_DEBUG_ATOMIC_SLEEP` 就把警告升级为 BUG，当场崩给你看。

🚫 错误答案陷阱："因为中断优先级最高"——这是 RTOS 语境。Linux 没有"中断永远高于一切"的严格优先级抢占层级（threaded IRQ 之后中断线程甚至能被更高优先级 RT 任务抢占），用这个理由答会被追问到露馅。真正的理由是上面那条**"没有可供睡眠的调度实体 + 会破坏被中断者的上下文"**。

**Q2：softirq、tasklet、workqueue 如何选择？给个决策树。**

```
是否需要睡眠（锁、I/O、内存）？
├── 是 → workqueue（唯一选择）
└── 否 → 是否需要同类型多核并行？
         ├── 是 → softirq（仅内核子系统可用，驱动不能直接用）
         └── 否 → 是否是新代码？
                  ├── 是 → threaded IRQ（优先）或 workqueue
                  └── 否（维护旧代码）→ tasklet（但标注 deprecated）
```

**Q3：中断亲和性配置错误会导致什么问题？如何诊断？**

答题框架：
- NUMA 跨 node 问题：中断在 node 1 的 CPU 处理，但 NIC DMA buffer 在 node 0 → sk_buff 分配在 node 1，数据拷贝跨 QPI/UPI 总线，延迟 +50-100%
- 诊断：`/proc/interrupts` 看分布，`numastat` 看 numa_miss，`perf stat -e cache-misses` 看 LLC miss 率
- 修复：`/proc/irq/<N>/smp_affinity` 配对到 NIC 所在 NUMA node 的 CPU set

---

**Q4：NAPI 的工作原理？为什么高负载下轮询反而比中断高效？**

答题框架：
- 中断开销分析：每次 hardirq 进/出约 1μs 量级（寄存器保存、cache/TLB 抖动、流水线打断），依平台浮动。1Gbps 小包场景约 1.488Mpps，每包预算才 ~670ns，**单次中断成本就已大于每包预算**——光是进出中断就能把一颗甚至多颗核占满，零资源给业务。
- NAPI：用一个中断当"发令枪"开启轮询，一次 `->poll()` 处理 N 个包（N 受 `dev_weight`/`netdev_budget` 约束），把那 ~1μs 的固定成本均摊到 N 个包上，等效每包中断成本降到 ~1μs/N。
- 低负载时恢复中断（轮询发现队列空 → `napi_complete_done()` → 重开 IRQ），保证低延迟。这就是"高负载省 CPU、低负载保延迟"的自适应本质。

---

**Q5：threaded IRQ 如何实现实时性？和普通中断处理有何本质区别？**

答题框架：
- 普通中断：hardirq 上下文 → 不可抢占 → 实时任务无法打断中断处理 → 实时延迟不确定
- threaded IRQ：primary handler（极短，返回 `IRQ_WAKE_THREAD`）→ 唤醒 RT 内核线程 → 线程可被更高优先级任务抢占
- 本质区别：把中断处理从"不可调度的执行流"变成"可调度的内核线程"
- 配合 PREEMPT_RT：实时任务只需要比中断线程优先级高，就能"抢占"中断处理

---

**Q6：`/proc/interrupts` 里 CPU 间分布严重不均，如何系统性地排查？**

答题框架（5 步）：
1. 确认是哪个设备的 IRQ（查名称列）
2. 看 `smp_affinity`：是否被锁定到单 CPU？还是 irqbalance 配置问题？
3. 检查硬件 RSS 队列数：`ethtool -l <dev>`，队列数 < CPU 数导致不均？
4. 检查是否有 `IRQF_SHARED`（共享中断必须由同一 CPU 处理）
5. 根本原因通常三类：affinity 手动配置错误 / 硬件队列不足 / irqbalance 策略与业务不匹配

---

**Q7：内核如何防止 hardirq 里错误调用可睡眠函数？**

答题框架：
- `might_sleep()` 宏：内核在所有"可能睡眠"的函数入口调用，`in_atomic()`（包括 in_interrupt）时触发 WARN
- `CONFIG_DEBUG_ATOMIC_SLEEP`：把 WARN 升级为 BUG（立即崩溃）
- `CONFIG_PROVE_LOCKING`（lockdep）：追踪锁的获取/释放路径，检测在 atomic 上下文获取 sleeping lock
- Rust for Linux：编译期类型系统阻止（未来）

### 设计题

**题目**：你在设计一个支持 100Gbps 线速的用户态包处理框架（类 DPDK），需要决定是否完全绕过内核中断系统。分析内核中断路径 vs 轮询旁路的 trade-off，说明什么场景下每种方案更优，以及混合方案的设计思路。

**关键考点**：
- 内核中断路径的最小开销（softirq + NAPI，约 5-20μs/包量级，高负载下进一步优化）
- 100G 的包速（最小包约 148Mpps），硬件队列深度与 CPU 轮询频率的匹配
- 旁路方案（DPDK/AF_XDP）的代价：CPU 专用（CPU spinning）、内核功能（防火墙/iptables）需额外实现
- 混合方案：XDP 在内核驱动层实现 O(1) 包过滤，快速路径 XDP_DROP/XDP_TX，慢速路径 XDP_PASS 继续走内核栈
- 工程取舍：是否允许牺牲一颗完整的 CPU 核做轮询？是否需要与 kernel networking 互通？

### 系统题

**题目**：生产告警：某 16 核服务器 `top` 显示 CPU 14 的 `si`（softirq）持续 60%+，其他 CPU 的 `si` 接近 0，网络 RX 吞吐正常但 P99 延迟从 1ms 飙到 15ms。定位根因，给出修复方案。

**排查路径**：

1. 确认中断分布：`cat /proc/interrupts | grep eth` → 发现所有 RX 中断都在 CPU 14
2. 检查 affinity：`cat /proc/irq/*/smp_affinity_list` → 网卡所有 IRQ 均绑定到 CPU 14（可能是 `irqbalance` 策略或手工误配）
3. 检查硬件队列数：`ethtool -l eth0` → 只有 1 个 RX 队列（网卡 RSS 配置不当）
4. **根因**：单队列 + 单 CPU 中断处理，CPU 14 被 softirq 打满，处理完所有 softirq 才能响应应用层请求，导致排队延迟
5. 修复：
   - 启用多队列：`ethtool -L eth0 combined 16`（需网卡支持）
   - 分散 IRQ：`for i in ...; do echo $cpu > /proc/irq/$irq/smp_affinity; done`
   - 或：配置 RPS（纯软件多队列）：`echo ffff > /sys/class/net/eth0/queues/rx-0/rps_cpus`
   - 长期：irqbalance 配置 ONESHOT 策略或手动静态绑定

**量级参考**：单 CPU 处理 10G 网卡的 softirq，约能支撑 1-2M pps（64B 包），超过这个范围 CPU 14 的 softirq 必然满负荷。

### 代码题

**题目**：写一个 Linux 内核模块，注册一个 threaded IRQ handler。模拟硬件设备（用软件触发方式），在 primary handler 中记录时间戳并唤醒线程，在 thread handler 中计算两者时间差（即从 hardirq 触发到线程运行的延迟），通过 `/proc` 接口输出统计信息（min/max/avg）。

**骨架代码**（待验证，用于学习骨架，具体 API 以当前内核版本为准）：

```c
// threaded_irq_latency.c
#include <linux/module.h>
#include <linux/interrupt.h>
#include <linux/proc_fs.h>
#include <linux/seq_file.h>
#include <linux/ktime.h>
#include <linux/atomic.h>

#define MODULE_NAME "tirq_lat"

/* TODO: 替换为真实 IRQ 号或用 request_irq 动态分配 */
static int irq_num = -1;
module_param(irq_num, int, 0444);

static ktime_t hardirq_time;
static u64 lat_min = U64_MAX, lat_max = 0, lat_sum = 0;
static atomic_t lat_count = ATOMIC_INIT(0);
static spinlock_t stats_lock;

static irqreturn_t primary_handler(int irq, void *dev_id)
{
    hardirq_time = ktime_get();
    /* 告诉内核：唤醒我的线程 handler */
    return IRQ_WAKE_THREAD;
}

static irqreturn_t thread_handler(int irq, void *dev_id)
{
    ktime_t now = ktime_get();
    u64 lat_ns = ktime_to_ns(ktime_sub(now, hardirq_time));
    unsigned long flags;

    spin_lock_irqsave(&stats_lock, flags);
    if (lat_ns < lat_min) lat_min = lat_ns;
    if (lat_ns > lat_max) lat_max = lat_ns;
    lat_sum += lat_ns;
    spin_unlock_irqrestore(&stats_lock, flags);
    atomic_inc(&lat_count);

    return IRQ_HANDLED;
}

/* TODO: 实现 /proc/tirq_lat 的 seq_file show 函数，输出 min/max/avg */
/* TODO: 实现 proc_create() + request_threaded_irq() 在 module_init */
/* TODO: 实现 free_irq() + remove_proc_entry() 在 module_exit */

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("Threaded IRQ latency measurement");
```

**验证方法**：
```bash

make -C /lib/modules/$(uname -r)/build M=$(pwd) modules

# 加载（需要知道一个可以用的 IRQ 号，或用 softirq 触发方式模拟）

# 触发中断（如向网卡发送数据包）然后观察

# 预期输出：count=1234, min=3456ns, max=89012ns, avg=12345ns

# 卸载
rmmod threaded_irq_latency
dmesg | tail -20  # 确认无 oops/warning
```

**暗坑提示**：
1. `hardirq_time` 被 primary handler（中断上下文）写，被 thread handler（进程上下文）读——如果两者可能并发（同一 IRQ 来了两次，第一个 thread 还没跑完，第二个 primary 已经更新了 `hardirq_time`），需要用 per-IRQ 的 ktime 存储，不能共用一个全局变量。本骨架为简化示意。
2. `spin_lock_irqsave` 在 thread handler（可睡眠上下文）用是安全的，但用 `mutex` 更语义正确——thread handler 是进程上下文，mutex 是首选。
3. `request_threaded_irq` 要求 IRQ 不被其他驱动占用（非 `IRQF_SHARED`），测试时需要找一个空闲 IRQ 或用 `IRQF_SHARED`（需对应修改 handler 逻辑）。
