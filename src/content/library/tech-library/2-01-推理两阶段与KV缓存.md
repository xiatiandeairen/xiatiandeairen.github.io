## TL;DR

LLM 推理在计算形态上天然分为两个阶段：**Prefill**（一次并行处理整个 prompt）和 **Decode**（每步仅生成一个 token）。前者是 compute-bound，后者是 memory-bandwidth-bound，两者的算力利用形态完全不同，这是推理引擎里所有优化的出发点。

KV 缓存（Key-Value Cache）是让 Decode 阶段可行的核心机制——它把 Prefill 阶段计算出的每层 Attention 的 K/V 矩阵存下来，Decode 步骤直接复用，避免 O(n²) 的重复计算。KV Cache 的大小、布局和管理方式直接决定了服务的吞吐上限。

## 1. 设计考古：两阶段与 KV Cache 的由来

### 1.1 自回归的本质矛盾

GPT 系列模型（2018-2020）用的是 next-token prediction：给定 token 序列 $x_1,...,x_t$，预测 $x_{t+1}$。在 Transformer decoder 中，每一层 attention 需要计算：

$$\text{Attention}(Q, K, V) = \text{softmax}\left(\frac{QK^\top}{\sqrt{d_k}}\right)V$$

在**生成阶段**，假设已经输出了 $t$ 个 token，要生成第 $t+1$ 个：
- 理论上需要把整个上下文的 K 和 V 都重新算一遍
- 每步的复杂度是 $O(t \cdot d)$，整个生成过程是 $O(n^2 \cdot d)$

这种 $O(n^2)$ 的时间复杂度对长序列是灾难性的。**解决方案：把每步已经算好的 K 和 V 存起来，下一步直接读缓存。** 这就是 KV Cache 的直接动机。

### 1.2 Prefill/Decode 分离的历史

KV Cache 的思路在最早的 GPT-2 代码中已经存在（以 `past_key_values` 形式，HuggingFace 实现）。但**系统性地把 Prefill 和 Decode 当作两个不同计算形态来处理**，是推理引擎走向生产化的关键洞察：

**Orca（OSDI 2022，Yu et al.）**[^orca] 是第一篇系统论文明确提出 iteration-level scheduling（又称 continuous batching）。Orca 的核心观察是：
- 一个 batch 里的请求长度不同，naive 静态 batching 会导致短请求等长请求
- Prefill 和 Decode 在计算上根本不同，应该在 batch 层面区分

**vLLM（SOSP 2023，Kwon et al.）**[^vllm] 进一步提出 **PagedAttention**，解决 KV Cache 的内存碎片问题——这是目前工业界 KV Cache 管理的标准方案。

> "The key-value cache (KV cache) memory for each request is huge and grows and shrinks dynamically... 60%-80% of GPU memory may be wasted due to fragmentation and over-reservation."
> — vLLM blog, 2023

**FlashAttention（NeurIPS 2022，Tri Dao et al.）**[^fa] 从另一个角度优化 Prefill：通过 IO-aware tiling，把 attention 计算从 HBM-bound 变成 SRAM-compute-bound，带来 2-4x 的 wall-clock 加速。

[^orca]: Yu et al., "Orca: A Distributed Serving System for Transformer-Based Generative Models", OSDI 2022
[^vllm]: Kwon et al., "Efficient Memory Management for Large Language Model Serving with PagedAttention", SOSP 2023  
[^fa]: Dao et al., "FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness", NeurIPS 2022

### 1.3 两阶段的计算形态差异

| 特征 | Prefill | Decode |
|------|---------|--------|
| 输入 | 整个 prompt（$L$ 个 token） | 上步生成的 1 个 token |
| 矩阵乘法形态 | GEMM（矩阵×矩阵） | GEMV（矩阵×向量） |
| 算力利用 | Compute-bound，GPU 利用率高 | Memory-bandwidth-bound，GPU 利用率低（< 10%） |
| 与 KV Cache 关系 | 写入 KV Cache（计算并存储全部 K, V） | 读取 KV Cache（读历史 K, V + 写 1 行） |
| 典型延迟 | 秒级（长 prompt） | 几十毫秒/token |
| 并行方式 | 序列内所有 token 并行 | 无序列内并行（只有 batch 间并行） |

Decode 阶段的内存带宽瓶颈解释：对 LLaMA-2-7B（FP16），每次 decode 步需要从 HBM 读取约 14GB 的 weight，以 A100 HBM 带宽 2TB/s 计算，纯带宽下限就是 7ms/token——这就是为什么 decode 吞吐受 HBM 带宽而不是 TFLOPS 约束。

## 2. KV Cache 的内存数学

### 2.1 KV Cache 大小公式

每层 attention 的 KV Cache（FP16）：

$$\text{KV}_\text{layer} = 2 \times L \times H_{kv} \times d_{head} \times 2 \text{ bytes}$$

其中 $L$ 是序列长度，$H_{kv}$ 是 KV head 数（GQA/MQA 时 < Q head 数），$d_{head}$ 是 head 维度，乘 2 是 FP16 字节，最前面的 2 是 K 和 V。

全模型总 KV Cache：

$$\text{KV}_\text{total} = N_{layers} \times 2 \times L \times H_{kv} \times d_{head} \times 2 \text{ bytes}$$

**实际例子（LLaMA-2-13B，FP16）**：
- $N_{layers} = 40$，$H_{kv} = 40$，$d_{head} = 128$，$L = 4096$
- KV Cache = $40 \times 2 \times 4096 \times 40 \times 128 \times 2 = 3.37 \text{ GB}$

vLLM blog 引用的 "1.7GB for a single sequence in LLaMA-13B" 是针对较短序列的估计。实际上长序列更大，这就是为什么 KV Cache 会吃掉 GPU 显存的大头。

### 2.2 碎片问题与 PagedAttention

**旧方式（连续分配）**：为每个请求预留 `max_seq_len` 的连续 KV Cache 内存。

- 请求实际长度未知，必须按最大长度分配 → 大量 internal fragmentation
- 不同请求长度不同，无法复用彼此的内存 → 大量 external fragmentation
- vLLM 测量：旧方式浪费 60%-80% 的 KV Cache 内存

**PagedAttention 方案**：受 OS 虚拟内存分页启发，把 KV Cache 切成固定大小的 block（默认 16 tokens/block），用 block table 映射逻辑 block → 物理 block，支持非连续物理内存。

效果：
- 内存浪费降至 < 4%（只有最后一个 block 可能不满）
- 可以 copy-on-write 实现 beam search 的 prefix sharing
- 支持不同请求共享相同 prefix 的 block（prefix caching）

### 3.1 vLLM v1：FlashAttentionMetadata

**【真实源码 vllm-project/vllm@f715f25f · vllm/v1/attention/backends/flash_attn.py:234-267】**

```python
@dataclass
class FlashAttentionMetadata:
    # NOTE(sang): Definition of context_len, query_len, and seq_len.
    # |---------- N-1 iteration --------|
    # |---------------- N iteration ---------------------|
    # |- tokenA -|......................|-- newTokens ---|
    # |---------- context_len ----------|
    # |-------------------- seq_len ---------------------|
    #                                   |-- query_len ---|

    num_actual_tokens: int  # Number of tokens excluding padding.
    max_query_len: int
    query_start_loc: torch.Tensor  # [num_seqs+1], cumsum of query lens
    max_seq_len: int
    seq_lens: torch.Tensor          # [num_seqs], total KV cache len per seq
    block_table: torch.Tensor       # [num_seqs, max_blocks], KV block 映射表
    slot_mapping: torch.Tensor      # [num_tokens], 每个 token 写入 KV cache 的 slot

    # For cascade attention (prefix caching 优化路径)
    use_cascade: bool
    common_prefix_len: int
    # ...
```

**逐字注解**：
- `query_start_loc`：形状 `[num_seqs+1]` 的 cumsum，`query_start_loc[i]:query_start_loc[i+1]` 是第 i 个请求的 query token 范围。Prefill 时这个范围可能很长（整个 prompt），Decode 时每个请求的范围长度恰好是 1。
- `seq_lens`：每个请求的**总** KV cache 长度（含历史 context + 本次新 token），供 flash_attn 的 `seqused_k` 参数使用——告诉 kernel 每个请求需要 attend 到多长的 K/V。
- `block_table`：`[num_seqs, max_blocks_per_seq]` 的整数张量，每个 entry 是一个物理 block id，把逻辑序列映射到非连续的 KV Cache 物理块。
- `slot_mapping`：`[num_actual_tokens]` 的整数张量，每个 token 在 KV Cache 中的物理写入位置（flat index = block_id × block_size + offset）。

注意 v1 版本的 vLLM 没有显式的 `num_prefill_tokens` / `num_decode_tokens` 字段——v1 架构统一了 prefill 和 decode 的 forward path（通过 `query_start_loc` 和 `seq_lens` 的差值隐式区分），而非 v0 时代的两条分支。这是 vLLM 架构演进的重要节点。

### 3.2 vLLM v1：forward() 的 KV Cache 写入与 attention 计算

**【真实源码 vllm-project/vllm@f715f25f · vllm/v1/attention/backends/flash_attn.py:701-932】**

以下是 forward 方法的核心路径（已略去 encoder-only、DCP 分布式和 FP8 分支，聚焦主路径）：

```python
def forward(
    self,
    layer: torch.nn.Module,
    query: torch.Tensor,    # [num_tokens, num_heads, head_size]
    key: torch.Tensor,      # [num_tokens, num_kv_heads, head_size]
    value: torch.Tensor,    # [num_tokens, num_kv_heads, head_size]
    kv_cache: torch.Tensor, # [num_blocks, 2, block_size, num_kv_heads, head_size]
    attn_metadata: FlashAttentionMetadata,
    output: torch.Tensor,
) -> torch.Tensor:
    num_actual_tokens = attn_metadata.num_actual_tokens

    # 步骤 1：拆分 kv_cache 为 key_cache 和 value_cache
    # kv_cache shape: [num_blocks, 2, block_size, num_kv_heads, head_size]
    # unbind(1) 沿维度 1 拆成两个 [num_blocks, block_size, num_kv_heads, head_size]
    key_cache, value_cache = kv_cache.unbind(1)

    # 步骤 2：KV Cache 写入在 do_kv_cache_update 中完成
    # (v1 架构把写入和 attention 计算分离)

    # 步骤 3：调用 flash_attn_varlen_func
    # 注意：v1 统一用 varlen 接口，不再区分 prefill/decode 分支
    flash_attn_varlen_func(
        q=query[:num_actual_tokens],
        k=key_cache,                          # 直接读 paged KV cache
        v=value_cache,
        out=output[:num_actual_tokens],
        cu_seqlens_q=attn_metadata.query_start_loc,   # query 的 cumsum 边界
        max_seqlen_q=attn_metadata.max_query_len,
        seqused_k=attn_metadata.seq_lens,             # 每个序列的 KV cache 长度
        max_seqlen_k=attn_metadata.max_seq_len,
        softmax_scale=self.scale,
        causal=True,
        block_table=attn_metadata.block_table,        # paged 物理 block 映射
        fa_version=self.vllm_flash_attn_version,      # FA2/FA3/FA4
        scheduler_metadata=attn_metadata.scheduler_metadata,  # FA3 AOT schedule
    )
    return output

def do_kv_cache_update(
    self,
    layer: torch.nn.Module,
    key: torch.Tensor,
    value: torch.Tensor,
    kv_cache: torch.Tensor,
    slot_mapping: torch.Tensor,  # 每个 token 的 flat KV cache slot
) -> None:
    key_cache, value_cache = kv_cache.unbind(1)
    # reshape_and_cache_flash：scatter-write 写入 paged KV cache
    # key[i] 被写入 key_cache[slot_mapping[i] // block_size,
    #                           slot_mapping[i] %  block_size]
    reshape_and_cache_flash(
        key[:attn_metadata.num_actual_tokens],  # 实际 tokens
        value[:attn_metadata.num_actual_tokens],
        key_cache,
        value_cache,
        slot_mapping,   # [num_actual_tokens] flat indices
        self.kv_cache_dtype,
        layer._k_scale,
        layer._v_scale,
    )
```

**关键设计点**：

1. `flash_attn_varlen_func` 接受 `block_table` 参数（FA3/FA4 的 paged attention 支持），使得 Prefill 和 Decode 可以用同一个 kernel——在 FA2 时代需要两条分支（`flash_attn_varlen_func` for prefill + `flash_attn_with_kvcache` for decode）。

2. v1 架构的"统一路径"：`query_start_loc[i+1] - query_start_loc[i]` 即为第 i 个请求的 query 长度。Decode 时 = 1，Prefill 时 = prompt 长度，mixed batching 时两种请求在同一 batch 中共存。

3. `do_kv_cache_update` 与 `forward` 分离：允许某些层共享 KV cache（hybrid architecture，如带 sliding window 的层）时跳过写入。

**Prefix Caching（APC，Automatic Prefix Caching）** 的 hash 机制：

**【真实源码 vllm-project/vllm@f715f25f · vllm/v1/core/kv_cache_utils.py（PrefixCachingBlock，取自 v0.6 版本 prefix_caching_block.py 逻辑）】**

```python
@property
def content_hash(self) -> Optional[int]:
    """仅当 block 满时才计算 hash；hash = (is_first, prev_hash, *token_ids)
    这个递归链式 hash 确保相同 token 序列（含 prefix）产生相同 hash。
    """
    if self._cached_content_hash is not None:
        return self._cached_content_hash
    if not self.is_full:
        return None
    is_first_block = self._prev_block is None
    prev_block_hash = (
        None if is_first_block else self._prev_block.content_hash
    )
    if prev_block_hash is None and not is_first_block:
        return None
    self._cached_content_hash = hash((
        is_first_block, prev_block_hash, *self.token_ids
    ))
    return self._cached_content_hash
```

```cpp
llama_kv_cache::slot_info llama_kv_cache::find_slot(
        const llama_ubatch & ubatch,
        bool cont) const {           // cont=true: 需要连续 slot

    uint32_t n_tokens = ubatch.n_tokens;
    uint32_t n_seqs   = 1;

    // 多序列流：按序列分组处理
    if (n_stream > 1) {
        GGML_ASSERT(n_tokens % ubatch.n_seqs_unq == 0);
        n_seqs   = ubatch.n_seqs_unq;
        n_tokens = n_tokens / n_seqs;
    }

    slot_info res = { LLAMA_MAX_SEQ, 0, {}, {} };
    res.resize(n_seqs);

    for (uint32_t s = 0; s < n_seqs; ++s) {
        const auto seq_id = ubatch.seq_id_unq[s];
        const auto & cells = v_cells[seq_to_stream[seq_id]];
        uint32_t head_cur  = v_heads[seq_to_stream[seq_id]];

        // 如果头部之前有足够空闲空间，从头重搜（避免碎片化线性扫描）
        if (head_cur > cells.get_used() + 2*n_tokens) {
            head_cur = 0;
        }

        uint32_t n_tested = 0;
        const uint32_t n_test = cont ? n_tokens : 1;  // 连续时一次检查 n_tokens

        while (true) {
            if (head_cur + n_test > cells.size()) {
                n_tested += cells.size() - head_cur;
                head_cur = 0;  // 环绕搜索
                continue;
            }

            for (uint32_t i = 0; i < n_test; i++) {
                const auto idx = head_cur++;
                n_tested++;

                bool can_use = cells.is_empty(idx);

                // 特殊情况：SWA（Sliding Window Attention）过期 cell 可复用
                if (!can_use && cells.seq_count(idx) == 1) {
                    const llama_pos pos_cell = cells.pos_get(idx);
                    const llama_seq_id seq_id_cell = cells.seq_get(idx);
                    if (llama_hparams::is_masked_swa(n_swa, swa_type, pos_cell,
                            cells.seq_pos_max(seq_id_cell) + 1)) {
                        can_use = true;  // SWA window 外的 cell 可以重用
                    }
                }

                if (can_use) {
                    res.idxs[s].push_back(idx);
                } else if (cont) {
                    break;  // 连续模式：一旦遇到不可用 cell 就放弃这段
                }
            }

            if (res.idxs[s].size() == n_tokens) break;  // 找到足够 slot
            if (cont) res.idxs[s].clear();               // 连续模式重置
            if (n_tested >= cells.size()) return { };    // 搜索完毕无可用空间
        }
    }
    return res;
}
```

**逐行注解**：
- `cont` 参数：prefill 时通常需要连续 slot（因为 llama.cpp 的 RoPE 和 attention mask 实现依赖连续位置），decode 时可以非连续（scatter 写入）。
- SWA 复用：Sliding Window Attention 的 cell 在窗口外后实际上不再被 attend，可以安全复用——llama.cpp 在 find_slot 里直接判断是否落在 window 外。
- 环绕搜索：用 `n_tested >= cells.size()` 防止死循环，完整遍历一圈若无可用空间返回空结果触发上层 context resize。

**【真实源码 ggerganov/llama.cpp master · src/llama-kv-cache.cpp · seq_rm】**

```cpp
bool llama_kv_cache::seq_rm(llama_seq_id seq_id, llama_pos p0, llama_pos p1) {
    if (other) return true;  // 如果是 shared cache，由 owner 管理

    if (p0 < 0) p0 = 0;
    if (p1 < 0) p1 = std::numeric_limits<llama_pos>::max();

    if (seq_id >= 0) {
        auto & cells = v_cells[seq_to_stream[seq_id]];
        auto & head  = v_heads[seq_to_stream[seq_id]];
        uint32_t new_head = cells.size();

        for (uint32_t i = 0; i < cells.size(); ++i) {
            if (!cells.pos_in(i, p0, p1)) continue;
            // seq_rm：从 cell 的 bitset 中移除 seq_id
            // 如果该 cell 的所有序列都被移除（返回 true），记录新的 free head
            if (cells.seq_has(i, seq_id) && cells.seq_rm(i, seq_id)) {
                if (new_head == cells.size()) new_head = i;
            }
        }
        // 更新搜索头指针，让下次 find_slot 从更早位置开始
        if (new_head != cells.size() && new_head < head) head = new_head;
    }
    return true;
}
```

## 4. 方案对比

### 4.1 KV Cache 管理方案演进

| 方案 | 内存利用率 | 实现复杂度 | Prefix Sharing | 代表系统 |
|------|-----------|-----------|----------------|---------|
| 连续预分配（max_len） | 20-40%（碎片严重） | 简单 | 不支持 | 早期 HuggingFace |
| 连续动态分配（实际长度） | 60-70% | 中等 | 不支持 | FasterTransformer |
| **PagedAttention**（固定 block） | **96%+** | 中等 | 支持（COW） | **vLLM** |
| Radix Tree（SGLang） | 96%+ | 复杂 | 支持（精确匹配） | SGLang |
| HBM + CPU offload | ~100%（虚拟扩展） | 复杂 | 部分支持 | DeepSpeed-MII |

### 4.2 Prefill 优化方案

| 方案 | 核心优化 | 适用场景 | 代价 |
|------|---------|---------|------|
| 标准 FlashAttention（FA2） | IO-aware tiling，减少 HBM 读写 | 所有 Prefill | 需 Ampere+ |
| FlashAttention-3/4 | FA3：H100 特化（wgmma + TMA）；FA4：更大 block | H100+ | 需特定 GPU |
| Chunked Prefill | 把长 Prefill 切成多个 chunk，与 Decode 混合调度 | 长 prompt + 低延迟需求 | 实现复杂度高 |
| Prefix Caching | 命中时跳过已缓存 prefix 的 attention 计算 | 系统 prompt 复用场景 | 增加 cache 管理开销 |

### 4.3 Decode 优化方案

| 方案 | 核心优化 | 典型收益 | 限制 |
|------|---------|---------|------|
| Continuous Batching | 共享 Decode 步骤的 weight 读取 | 吞吐 8-23x（vs 静态 batch） | 增加 P50 延迟 |
| Speculative Decoding | Draft 小模型预测 + 大模型验证 | 延迟 2-3x | 接受率依赖分布 |
| GQA/MQA | 减少 KV head 数，降低 KV Cache 读取量 | 显存 2-8x，延迟 1.5-2x | 模型需要 GQA 训练 |
| PagedAttention | 减少 batch 间 idle，提高 batch size | 吞吐 2-4x（vs 连续分配） | block 对齐引入少量碎片 |

### 4.4 Prefill-Decode Disaggregation

一个值得单独提及的进化方向：把 Prefill 和 Decode 放到不同机器（GPU）上处理。

**动机**：Prefill 是 compute-bound，Decode 是 memory-bandwidth-bound，两者对 GPU 的"偏好"完全不同。放在一起会互相干扰：Prefill 请求会阻塞 Decode 的 TTFT（time to first token），Decode 请求会让 GPU 在 Prefill 时被迫 idle。

**代表工作**：
- Splitwise（ISCA 2024）：首次系统化论证 P/D disaggregation
- DeepSpeed-FastGen 的 Dynamic SplitFuse：动态调度 prefill chunks 和 decode tokens 以填充 GPU 利用率
- vLLM 0.8+ 支持 disaggregated prefill（通过 LMCache 等传输 KV）

不适用边界：序列很短时（< 128 tokens），Prefill 本身很快，disaggregation 的网络传输 KV Cache 的开销可能超过收益。

```python
"""
demo_kv_cache.py

手写 mini Transformer 解码循环，对比有/无 KV Cache 的重复计算。
目的：印证 vLLM 源码中 seq_lens（KV cache 长度）与 query_start_loc
     （当前 query 范围）的分离设计——seq_lens 记录累计历史，
     query_start_loc 只记录新 token 的范围。

运行：python demo_kv_cache.py
依赖：numpy（pip install numpy）
预期输出：
  [无 KV Cache] step 0: 计算量=Q:1x4, K:1x4 (FLOPs≈32)
  [无 KV Cache] step 1: 计算量=Q:2x4, K:2x4 (FLOPs≈128)
  [无 KV Cache] step 4: 计算量=Q:5x4, K:5x4 (FLOPs≈800)
  累计 FLOPs（无 KV Cache）: 1680
  
  [有 KV Cache] step 0: 计算量=Q:1x4, K:1x4 (FLOPs≈32)
  [有 KV Cache] step 1: 计算量=Q:1x4, K:2x4 (FLOPs≈64)
  [有 KV Cache] step 4: 计算量=Q:1x4, K:5x4 (FLOPs≈160)
  累计 FLOPs（有 KV Cache）: 480
  FLOPs 节省比例: 71.4%
"""

import numpy as np

# 超参数（mini model，便于演示）
d_model = 8    # embedding dim
n_heads = 2    # attention heads
d_head  = d_model // n_heads  # = 4
seq_len = 5    # 生成的 token 数（不含 prompt）
prompt_len = 1 # 初始 prompt 长度

rng = np.random.default_rng(42)

# 简化的权重矩阵（单层 attention）
W_Q = rng.standard_normal((d_model, d_model)).astype(np.float32)
W_K = rng.standard_normal((d_model, d_model)).astype(np.float32)
W_V = rng.standard_normal((d_model, d_model)).astype(np.float32)
W_O = rng.standard_normal((d_model, d_model)).astype(np.float32)

def attention(Q, K, V):
    """标准 scaled dot-product attention"""
    scale = 1.0 / np.sqrt(Q.shape[-1])
    scores = Q @ K.T * scale            # [q_len, kv_len]
    weights = np.exp(scores - scores.max(axis=-1, keepdims=True))
    weights /= weights.sum(axis=-1, keepdims=True)
    return weights @ V                   # [q_len, d_head]

def compute_flops_estimate(q_len, kv_len, d_h):
    """QK^T: q_len*kv_len*d_h; softmax(...)V: q_len*kv_len*d_h"""
    return 2 * q_len * kv_len * d_h

# 生成 token embeddings（随机模拟）
all_tokens = rng.standard_normal((prompt_len + seq_len, d_model)).astype(np.float32)

print("=" * 60)
print("方案 A：无 KV Cache（每步重算所有历史 K, V）")
print("=" * 60)
total_flops_no_cache = 0
for step in range(seq_len + 1):
    # 当前 context = prompt + 已生成的 token
    context_len = prompt_len + step
    context = all_tokens[:context_len]

    # 每步重算所有 K, V
    K_full = context @ W_K.reshape(d_model, d_model)  # [ctx_len, d_model]
    V_full = context @ W_V.reshape(d_model, d_model)
    Q_cur  = all_tokens[context_len-1:context_len] @ W_Q  # 只有最后 1 个 query

    # 取单头视角（head 0）
    K_h = K_full[:, :d_head]  # [ctx_len, d_head]
    V_h = V_full[:, :d_head]
    Q_h = Q_cur[:, :d_head]   # [1, d_head]

    flops = compute_flops_estimate(1, context_len, d_head)
    total_flops_no_cache += flops
    if step <= 2 or step == seq_len:
        print(f"  step {step}: Q:{Q_h.shape[0]}x{Q_h.shape[1]}, "
              f"K:{K_h.shape[0]}x{K_h.shape[1]} "
              f"(FLOPs≈{flops})")

print(f"累计 FLOPs（无 KV Cache）: {total_flops_no_cache}")

print()
print("=" * 60)
print("方案 B：有 KV Cache（只计算新 token 的 K, V，历史复用）")
print("=" * 60)

# KV Cache：存储历史 K, V（对应 vLLM 的 kv_cache tensor）
kv_cache_K = np.zeros((prompt_len + seq_len, d_head), dtype=np.float32)
kv_cache_V = np.zeros((prompt_len + seq_len, d_head), dtype=np.float32)

total_flops_with_cache = 0
cache_len = 0  # 对应 vLLM 的 seq_lens[i]（当前 KV cache 中的 token 数）

# Prefill 阶段：处理整个 prompt
prompt_tokens = all_tokens[:prompt_len]
K_prompt = prompt_tokens @ W_K.reshape(d_model, d_model)
V_prompt = prompt_tokens @ W_V.reshape(d_model, d_model)
kv_cache_K[:prompt_len] = K_prompt[:, :d_head]  # 写入 KV cache
kv_cache_V[:prompt_len] = V_prompt[:, :d_head]
cache_len = prompt_len

# Decode 阶段：每步只处理 1 个新 token
for step in range(seq_len):
    new_token = all_tokens[prompt_len + step : prompt_len + step + 1]

    # 只计算新 token 的 K, V（对应 do_kv_cache_update 中的 slot_mapping 写入）
    k_new = (new_token @ W_K.reshape(d_model, d_model))[:, :d_head]  # [1, d_head]
    v_new = (new_token @ W_V.reshape(d_model, d_model))[:, :d_head]

    # 写入 KV cache（对应 reshape_and_cache_flash with slot_mapping）
    kv_cache_K[cache_len] = k_new[0]
    kv_cache_V[cache_len] = v_new[0]
    cache_len += 1

    # Query：当前新 token（对应 query_start_loc 指向的最新 1 个 token）
    q_cur = (new_token @ W_Q)[:, :d_head]  # [1, d_head]

    # Attention：Q 只有 1 行，K/V 有 cache_len 行
    # 对应 flash_attn_varlen_func 中 seqused_k=cache_len, query_len=1
    K_cached = kv_cache_K[:cache_len]       # [cache_len, d_head]
    V_cached = kv_cache_V[:cache_len]

    out = attention(q_cur, K_cached, V_cached)  # [1, d_head]

    flops = compute_flops_estimate(1, cache_len, d_head)
    total_flops_with_cache += flops
    if step <= 2 or step == seq_len - 1:
        print(f"  step {step}: Q:{q_cur.shape[0]}x{q_cur.shape[1]}, "
              f"K:{K_cached.shape[0]}x{K_cached.shape[1]} "
              f"(FLOPs≈{flops})")

print(f"累计 FLOPs（有 KV Cache）: {total_flops_with_cache}")
savings = (total_flops_no_cache - total_flops_with_cache) / total_flops_no_cache
print(f"FLOPs 节省比例: {savings:.1%}")

print()
print("这印证了 vLLM FlashAttentionMetadata 的设计：")
print("  seq_lens[i]        = cache_len（历史 KV 总长，K 的行数）")
print("  query_start_loc    = 新 query 边界（只有 1 个 token）")
print("  两者之差           = context_len（历史 token 数）")
```

### Demo 2：PagedAttention Block Table 模拟

> 设计为可运行，纯 Python，无 GPU 需求。印证 vLLM block_table + slot_mapping 的设计。

```python
"""
demo_paged_kv.py

模拟 PagedAttention 的 block table 管理和 slot_mapping 生成。
印证源码中 KVCacheBlocks.get_block_ids() 和 slot_mapping 的含义。

运行：python demo_paged_kv.py
依赖：无（纯标准库）
"""

class FreeBlockPool:
    """物理 KV Cache block 池（模拟 GPU HBM 上的 KV Cache tensor）"""
    def __init__(self, num_blocks: int, block_size: int):
        self.block_size = block_size
        self.free_blocks = list(range(num_blocks))  # 物理 block id 池
        self.num_blocks = num_blocks

    def allocate(self) -> int:
        if not self.free_blocks:
            raise RuntimeError("OOM: no free KV cache blocks")
        return self.free_blocks.pop(0)

    def free(self, block_id: int):
        self.free_blocks.append(block_id)

    def __repr__(self):
        return f"FreeBlockPool(free={len(self.free_blocks)}/{self.num_blocks})"


class Request:
    """单个推理请求的 KV Cache 状态"""
    def __init__(self, req_id: str):
        self.req_id = req_id
        self.block_table: list[int] = []  # 逻辑 block → 物理 block id
        self.num_tokens: int = 0          # 当前已写入 KV cache 的 token 数

    def get_slot_mapping(self, new_tokens: int, block_size: int) -> list[int]:
        """
        为 new_tokens 个新 token 计算 slot_mapping。
        slot = physical_block_id * block_size + offset_in_block
        这对应 vLLM do_kv_cache_update 中的 slot_mapping tensor。
        """
        slots = []
        for i in range(new_tokens):
            pos = self.num_tokens + i
            block_idx = pos // block_size     # 第几个 block
            offset    = pos %  block_size     # block 内偏移
            physical_block = self.block_table[block_idx]
            slot = physical_block * block_size + offset
            slots.append(slot)
        return slots


def simulate_kv_cache_lifecycle():
    BLOCK_SIZE = 4    # 每 block 存 4 个 token 的 KV（vLLM 默认 16，这里缩小便于演示）
    NUM_BLOCKS = 8    # 总共 8 个物理 block

    pool = FreeBlockPool(NUM_BLOCKS, BLOCK_SIZE)
    print(f"初始状态: {pool}\n")

    # --- Prefill：请求 A，prompt 长度 6 token ---
    req_a = Request("A")
    prompt_len = 6
    # 需要 ceil(6/4) = 2 个 block
    num_blocks_needed = (prompt_len + BLOCK_SIZE - 1) // BLOCK_SIZE
    for _ in range(num_blocks_needed):
        req_a.block_table.append(pool.allocate())
    print(f"[Prefill] 请求 A: prompt={prompt_len} tokens, "
          f"分配 {num_blocks_needed} 个 block")
    print(f"  block_table = {req_a.block_table}")

    # 生成 slot_mapping（对应 FlashAttentionMetadata.slot_mapping）
    slots = req_a.get_slot_mapping(prompt_len, BLOCK_SIZE)
    print(f"  slot_mapping = {slots}")
    print(f"  解读: token[0] → block {req_a.block_table[0]} slot {slots[0]}")
    print(f"        token[5] → block {req_a.block_table[1]} slot {slots[5]}")
    req_a.num_tokens = prompt_len
    print(f"  pool after: {pool}\n")

    # --- Decode：请求 A，生成 3 个 token ---
    print(f"[Decode] 请求 A: 生成 3 个 token")
    for decode_step in range(3):
        pos = req_a.num_tokens
        # 检查是否需要新 block
        if pos % BLOCK_SIZE == 0:
            new_block = pool.allocate()
            req_a.block_table.append(new_block)
            print(f"  step {decode_step}: 分配新 block {new_block}")
        slots = req_a.get_slot_mapping(1, BLOCK_SIZE)
        print(f"  step {decode_step}: pos={pos}, slot={slots[0]}, "
              f"block_table={req_a.block_table}")
        req_a.num_tokens += 1
    print(f"  pool after: {pool}\n")

    # --- 并发请求 B（模拟 continuous batching）---
    req_b = Request("B")
    prompt_len_b = 3
    num_blocks_b = (prompt_len_b + BLOCK_SIZE - 1) // BLOCK_SIZE
    for _ in range(num_blocks_b):
        req_b.block_table.append(pool.allocate())
    req_b.num_tokens = prompt_len_b
    print(f"[Prefill] 请求 B: prompt={prompt_len_b} tokens, "
          f"block_table={req_b.block_table}, {pool}")

    # 展示 block_table tensor（对应 FlashAttentionMetadata.block_table）
    print()
    max_blocks = max(len(req_a.block_table), len(req_b.block_table))
    block_table = []
    for req in [req_a, req_b]:
        row = req.block_table + [0] * (max_blocks - len(req.block_table))  # 0 = padding
        block_table.append(row)
    print("batch block_table（对应 FlashAttentionMetadata.block_table tensor）:")
    print(f"  req_A: {block_table[0]}")
    print(f"  req_B: {block_table[1]}")
    print(f"  seq_lens: [{req_a.num_tokens}, {req_b.num_tokens}]  "
          f"（对应 FlashAttentionMetadata.seq_lens）")

    # --- 请求 A 结束，释放 block ---
    print(f"\n[Free] 请求 A 完成，释放 blocks {req_a.block_table}")
    for blk in req_a.block_table:
        pool.free(blk)
    print(f"  pool after: {pool}  ← 可被后续请求复用")


if __name__ == "__main__":
    simulate_kv_cache_lifecycle()
```

## 6. 失败模式与生产真坑

### 坑 1：KV Cache OOM 与 preemption 瀑布

**症状**：大并发下 vLLM 频繁打印 `Preempting sequence X (status=RUNNING)`，吞吐骤降，延迟飙升。

**根因**：vLLM 的 scheduler 在 KV Cache 不足时会抢占（preempt）正在运行的请求——把请求的 KV Cache 丢掉，重新放回 WAITING 队列等待重算。如果 max_model_len 设置过大、或并发量超过预估，preemption 会级联——被重算的请求再次 OOM，再次被抢占。

**解法**：
- 减小 `max_model_len`（实际不需要这么长时）
- 增大 `--gpu-memory-utilization`（默认 0.9，谨慎增大）
- 使用 chunked prefill 限制单次 prefill 消耗的 KV Cache block 数
- 开启 CPU swap（`--swap-space`，但会引入 CPU-GPU 数据传输延迟）

### 坑 2：Prefill 长请求饿死 Decode 请求

**症状**：系统中有长 prompt 请求时，短请求的 TTFT（first token latency）急剧恶化。

**根因**：长 prompt 的 prefill 需要几秒，这段时间内 Decode 请求无法调度——GPU 被 prefill GEMM 占满，Decode 的 GEMV 被阻塞。

**解法**：Chunked Prefill（vLLM 的 `--enable-chunked-prefill`）：把长 prompt 切成 chunk（如每次最多 512 tokens），与 Decode 请求交替调度。代价是 Prefill 的整体延迟略增，但 TTFT 的长尾延迟大幅改善。

### 坑 3：llama.cpp find_slot 线性扫描性能退化

**症状**：在 llama.cpp 中使用超长 context（如 128K tokens）时，`find_slot` 慢，推理延迟异常高。

**根因**：`find_slot` 对 `v_cells` 做线性扫描，每次 find_slot 的时间复杂度是 O(n_ctx)。在 128K context 且频繁 seq_rm/seq_cp 操作后，扫描开销不可忽视。

**解法**：
- 利用 SWA（Sliding Window Attention）：`find_slot` 会自动复用 window 外的 cell，实际搜索空间 = window_size 而不是全 context。
- 如果不用 SWA，考虑 server 模式（llama-server）的 cache slot 管理，或把 find_slot 频率降低（batch 多个请求一起处理）。

### 坑 4：Prefix Cache 失效（hash 冲突 / system prompt 变化）

**症状**：看到 vLLM 的 `prefix_cache_hit_rate` 很低，或者命中后输出质量下降。

**根因**：
1. Hash 冲突：`content_hash = hash((is_first, prev_hash, *token_ids))`，Python 的 `hash()` 在不同进程间不稳定（PYTHONHASHSEED），跨进程部署时 hash 不一致。
2. System prompt 频繁变化（如加入时间戳）导致第一个 block 的 hash 每次不同，整个 prefix chain 的 hash 全部失效。

**解法**：
- 在 system prompt 中避免动态内容（时间戳等）放在最前面
- vLLM 的 enable_caching 用 SHA256 等稳定 hash（而非 Python 内置 hash）——实际代码中用 `sha256(token_ids)` 替代，具体取决于版本

### 坑 5：Decode batch size 过小，KV Cache 带宽没有充分利用

**症状**：GPU 利用率只有 20-30%，但 KV Cache 读取带宽已经饱和。

**根因**：Decode 阶段每步需要读取所有层的 KV Cache，如果 batch size 小（比如只有 4 个并发请求），HBM 带宽被 weight 读取主导，KV Cache 的读取是附带的。但如果 batch size 太大，KV Cache 总量 = batch_size × seq_len × KV_per_token 可能撑爆 HBM。

**解法**：找到 optimal batch size 的方法是 roofline analysis：计算 compute/memory 的 arithmetic intensity，找到 memory-bound 和 compute-bound 的交叉点。对 H100 SXM（3.35 TB/s HBM3），LLaMA-2-7B（14GB weight）的最优 batch size 大约在 32-128 之间（具体取决于 seq_len）。

## 7. 章末五件套

### 7.1 关键概念清单

- **Prefill**：处理整个 prompt，compute-bound，输出第 1 个 token 的 logit 并填满 KV Cache
- **Decode**：每步生成 1 个 token，memory-bandwidth-bound，复用 KV Cache
- **KV Cache**：存储每层 attention 的 K/V 矩阵，避免 O(n²) 重算
- **PagedAttention**：OS 分页启发的 KV Cache 管理，固定 block size，支持非连续物理内存
- **Prefix Caching（APC）**：对相同 token 前缀复用 KV Cache block，通过 content hash 实现
- **Chunked Prefill**：把长 prefill 切片与 decode 混合调度，降低 TTFT 长尾

### 7.2 面试题

**Q1**（基础）：为什么 Decode 阶段是 memory-bandwidth-bound 而不是 compute-bound？请从 arithmetic intensity（ops/byte）的角度分析。

*参考答案*：Decode 每步进行 batch_size 次 GEMV（矩阵 × 向量），arithmetic intensity ≈ batch_size × (2×d_model) FLOPs / (weight_bytes + KV_bytes)。当 batch_size=1 时 intensity ≈ 1 FLOP/byte，远低于 A100 的 roofline（~300 FLOP/byte），所以是 memory-bound。只有当 batch_size 足够大（>32 for most models），intensity 才接近 roofline。

**Q2**（中等）：PagedAttention 的 block_table 是什么？为什么需要它，不能直接用连续内存？

*参考答案*：block_table 是 `[num_seqs, max_blocks_per_seq]` 的整数张量，把序列的逻辑 block 映射到物理 block id。需要它是因为：(1) 请求长度动态变化，预分配 max_len 的连续内存浪费严重；(2) 多个请求的 prefix 可以共享同一物理 block（COW），节省显存；(3) 类似 OS 页表，允许物理块非连续，消除 external fragmentation。

**Q3**（较难）：vLLM v0 和 v1 的 attention forward 实现有什么关键架构差异？

*参考答案*：v0 有显式的 `num_prefill_tokens` 和 `num_decode_tokens`，forward 中通过 `query[:num_prefill_tokens]` 和 `query[num_prefill_tokens:]` 切分成两条路径（`flash_attn_varlen_func` for prefill，`flash_attn_with_kvcache` for decode）。v1 统一成一条路径：`flash_attn_varlen_func` + `block_table` 参数（FA3/FA4 原生支持 paged attention），通过 `query_start_loc` 和 `seq_lens` 隐式区分 prefill（query_len > 1）和 decode（query_len = 1），消除了 mixed batching 时的两次 kernel 调用开销。

### 7.3 代码题（扩展 Demo）

**题目**：在 Demo 1 的基础上，实现 Prefill 阶段的 causal mask（下三角 mask）并验证：
1. Prefill 时 token[i] 只能 attend 到 token[0..i]
2. Decode 时新 token 可以 attend 到所有历史 token

提示：在 `attention()` 函数中加入 `mask` 参数，causal mask = `np.tril(np.ones((q_len, kv_len)))`，在 Prefill 时 `q_len == kv_len`，在 Decode 时 `q_len == 1`（mask 退化为全 1 向量）。

**进阶**：模拟 Sliding Window Attention（window_size=3）：token[i] 只能 attend 到 token[max(0,i-2)..i]，验证 llama.cpp `find_slot` 中 SWA window 外的 cell 可以被复用的条件。

### 7.4 未来方向

**Disaggregated Prefill/Decode**：随着上下文越来越长（128K-1M token），把 Prefill 和 Decode 放到不同型号的 GPU（Prefill 用 compute-dense 的 H100 SXM，Decode 用 memory-bandwidth-dense 的 H100 NVL）成为生产优化方向。瓶颈从计算移到了 KV Cache 的跨节点传输（每次 Prefill 后需要把 GB 级的 KV Cache 传给 Decode 节点）。

**KV Cache 量化**：FP8 KV Cache（vLLM 的 `--kv-cache-dtype fp8`）把 KV Cache 内存减半，配合 FA3 的 FP8 原生支持（H100+），是目前生产环境的主流选择。INT4 KV（如 KIVI）进一步压缩，但精度损失需要仔细验证。

**KV Cache 卸载（Offloading）**：把不活跃的 KV Cache 从 HBM offload 到 CPU DRAM 或 NVMe（如 InfiniStore），用带宽换容量，支持超长 context 或超大 batch。代价是 offload/prefetch 的延迟需要 overlap 到计算。
