> **一句话定位**：Linux 内核里没有"线程"这个特权概念——进程和线程共用同一种内核对象 `task_struct`，`fork`/`vfork`/`clone`/`clone3` 四个 syscall 在内核里汇聚到**同一个** `kernel_clone()`，区别只在传进去的 `kernel_clone_args.flags`。整章的主线就是：**"共享什么"是一组 flag 的选择，而这组选择的历史、源码落点、失败模式，才是要吃透的东西。**

---

## TL;DR — 本章最硬结论

1. **内核只有 task，连 syscall 入口都收敛成一个**：`fork`/`vfork`/`clone` 三个老 syscall + 新的 `clone3`，在 `kernel/fork.c` 里全部构造一个 `struct kernel_clone_args` 然后调 `kernel_clone()`。`fork` 不过是 `{.exit_signal = SIGCHLD}`，`vfork` 是 `{.flags = CLONE_VFORK | CLONE_VM, .exit_signal = SIGCHLD}`。这是【真实源码 v6.12 kernel/fork.c】，下文逐行给。
2. **`do_fork` 这个名字本身是历史债**：核心 helper 历史上叫 `do_fork`→`_do_fork`，2020 年 Christian Brauner 在 5.10 把它改名 `kernel_clone()`，理由是它**本质是 clone 的内核侧 helper，叫 fork 名不副实**（commit message 原话下文引）。读旧书看到 `_do_fork` 就是今天的 `kernel_clone`。
3. **clone flag 空间真的用光过**：32 位 flag 在 2019 年只剩最后 1 bit，`CLONE_PIDFD` 占掉它后，`CLONE_NEWTIME`（time namespace）无位可用——这是 `clone3()` 诞生的**直接导火索**，不是为了优雅。源码佐证：`CLONE_NEWTIME = 0x00000080`，是回收了一个 32 位低位，而 `CLONE_CLEAR_SIGHAND = 0x100000000ULL` 已经爬到 64 位区。
4. **D 态不可杀是契约，但 2008 年就有逃生门**：`TASK_KILLABLE`（Matthew Wilcox，2.6.25，2008）让 fatal signal 能打断不可中断睡眠。但能不能被杀**取决于那条等待路径的作者选了哪个原语**——NFS hard mount、很多块设备 I/O 仍是真 D 态。这是 17 年的老特性，别当新东西讲。
5. **状态位是 ABI，且和 procfs 强耦合**：`TASK_UNINTERRUPTIBLE = 0x0002` 这种值是稳定 ABI，内核源码里专门有注释警告"这些位也编码在 `fs/proc/array.c: get_task_state()`"——你 bpftrace 里写 `__state == 2` 能用，正是吃了这个 ABI 稳定性的红利。

## 一、背景：为什么是 `task_struct` 这一种结构

Unix 最初把 process 定义成 "resource isolation + execution context" 的合体。多核时代，单进程内的并发需求让"一个进程一个执行流"不够用：若每个并发单元都是独立进程，IPC 开销和地址空间隔离的代价高得离谱。

**这里有三条互相冲突的工程约束，设计必须同时满足：**

1. 一种结构能同时表达"完全隔离的进程"和"共享地址空间的线程"——否则调度器要维护两套对象。
2. 调度器统一调度，不能给线程开后门——否则负载均衡、优先级、cgroup 计费全要分叉。
3. 创建代价要低——用户态线程库（N:1）切换快但吃不到多核；内核线程（1:1）能吃多核但创建/切换必须够轻。

**Linux 的答案**：一种数据结构（`task_struct`）、一个创建原语（内核里的 `kernel_clone` / 用户态的 `clone`）、用 flags 区分语义。"进程 = 独享一切的 task，线程 = 共享大部分的 task"——这不是教学比喻，是字面实现。

> **为什么不用"进程结构 + 线程结构"两层模型？**（这是 Windows 的路子：`EPROCESS` + `ETHREAD`）
> 对照跑一遍：Windows 模型里线程是进程的子对象，调度器调度 thread，进程是容器。优点是语义清晰、进程级资源（handle table）天然归属明确。代价是：(a) 调度器和进程/线程两层都要打交道；(b) "把一个线程的地址空间换掉"这种操作在模型里是非法的，而 Linux 因为只有 task，`clone(CLONE_VM)` 和 `clone()`（不带 VM）是同一条代码路径的两个 flag，`unshare()` 甚至能让一个运行中的 task 事后剥离出独立 namespace/VM。Linux 的"单一对象"模型把灵活性压到了 flag 层，代价是 `task_struct` 膨胀成内核最大结构体之一（数百字段），且"进程"这个概念在内核里其实是**派生的**（thread group leader），不是一等公民。**不适用边界**：如果你要的是强进程隔离 + 简单心智模型，两层模型其实更不容易写出 bug；Linux 这套的灵活性在容器时代才真正兑现价值（namespace 全靠 clone flag）。

### 2.1 早期 Unix：fork + exec

最初只有 `fork`（复制进程）和 `exec`（替换程序），线程不在视野里。fork 是全量复制：地址空间、fd、信号处理器全复制。早期机器可接受，但地址空间一大，内存拷贝成本就压不住了。

### 2.2 解决方案一：vfork —— 一个"明知危险但有用"的妥协

`vfork` 让父进程**挂起**，子进程**直接借用父进程的地址空间和页表**，直到子进程 `exec` 或 `exit`。它存在的唯一理由：在没有 COW 的年代，"fork 后立刻 exec"这个最常见模式不该付全量拷贝的钱。

它有多脆弱？子进程在 `exec` 前**碰了栈**（哪怕只是函数返回改了返回地址）就会破坏父进程——因为它们是同一份页表。所以 `vfork` 的契约极窄：子进程只能做"设置 fd、调 `_exit` 或 `execve`"，连 `return` 都不许（会破坏父进程的栈帧）。

源码里 vfork 的本质一目了然——它就是带 `CLONE_VFORK | CLONE_VM` 的 clone：

```c
/* 【真实源码 v6.12 kernel/fork.c】SYSCALL_DEFINE0(vfork) 内部 */
struct kernel_clone_args args = {
        .flags          = CLONE_VFORK | CLONE_VM,
        .exit_signal    = SIGCHLD,
};
```

- `CLONE_VM`：共享 `mm_struct`（同一地址空间，同一页表）——这就是"为什么子进程碰栈会污染父进程"。
- `CLONE_VFORK`：让父进程在 `kernel_clone` 里**阻塞在一个 completion 上**，直到子进程释放 mm（exec/exit 时通过 `vfork_done` 唤醒）。这是 vfork "父挂起"语义的实现点。
- `.exit_signal = SIGCHLD`：子进程退出时给父发 `SIGCHLD`（和普通进程一样）——注意这点和线程不同，线程的 `exit_signal` 为 0。

> 💡 **生产真坑**：现代 `vfork` 在 COW + 现代 MMU 下**已无明确性能优势**，新代码不应使用。glibc 的 `posix_spawn` 在某些路径上仍用 `vfork`/`CLONE_VM|CLONE_VFORK` 来避免大内存进程 fork 的页表复制开销（一个 100GB RSS 的 JVM 直接 `fork()` 会复制海量页表，即使 COW 不拷物理页），但这是库作者在受控路径里的精细操作，不是你该手写的东西。

### 2.3 解决方案二：COW —— 把"复制"推迟到"写"

COW 在 Linux 2.x 主流化（概念更早，源自更早的 Unix）。fork 时只复制**页表**（而非物理页），把所有可写页的 PTE 标记为**只读**。任一方写时 MMU 触发 page fault，内核此时才真正分配新物理页并拷贝。

关键洞察：**fork 的大部分真实场景是"fork 后立刻 exec"，根本不触发 COW**——exec 会丢弃整个地址空间，那些精心保护的 COW 页一次都没写就被扔了。所以 COW 让最常见的 fork 路径几乎免费，同时又不像 vfork 那样脆弱（父子地址空间逻辑上独立）。

源码落点（`copy_process` → `copy_mm`）里，"共享还是复制 mm"就是一个 `CLONE_VM` 分支：

```c
/* 【真实源码 v6.12 kernel/fork.c】copy_mm() 核心分支（节选） */
if (clone_flags & CLONE_VM) {
        mmget(oldmm);          /* 线程：直接增引用，共享同一个 mm_struct */
        mm = oldmm;
} else {
        mm = dup_mm(tsk, current->mm);   /* 进程：复制 mm（含逐 VMA 设 COW） */
        if (!mm)
                return -ENOMEM;
}
```

- 走 `CLONE_VM`（线程）：`mmget(oldmm)` 只是 `mm_users` 引用计数 +1，父子线程指向**同一个** `mm_struct`。这就是"同进程线程共享地址空间"的字面含义——所谓共享不是某种魔法，就是 `task->mm` 指针相等。
- 不走 `CLONE_VM`（进程）：`dup_mm` 复制 `mm_struct`，内部 `dup_mmap` 逐 VMA 复制并把可写 PTE 改只读。**这一步才是 fork 的真实开销大头**（见 §4.2）。

#### COW 的精确语义：PTE 只读 vs VMA 可写

fork 后父子各有页表副本，PTE 指向同一批物理页，原本可写的 PTE 被改成**只读**，**但 VMA 仍标记为可写**。这个"PTE 只读 + VMA 可写"的组合，正是内核区分"真只读页"和"COW 写保护页"的依据：

- 真只读映射（如 `mmap(PROT_READ)`）：PTE 只读 **且** VMA 只读 → 写它 = SIGSEGV。
- COW 页：PTE 只读 **但** VMA 可写 → 写它 = 触发 `do_wp_page`（write-protect fault），分配新页。

写时的内核路径大致是：`do_page_fault → handle_mm_fault → handle_pte_fault → do_wp_page()`。注意 fork 后**匿名页**的 COW 走的是 `do_wp_page`，**不是** `do_cow_fault`（后者是文件私有映射首次写的路径，常被初学者混淆）。

`do_wp_page` 的核心决策是**"能不能就地复用这页，省掉一次拷贝"**：检查该页是否被独占（在新内核里通过 folio 的 mapcount/refcount 与 `exclusive` 标志判断）。若对端早已各写各的、该页已是当前 task 独占（引用计数降到 1），直接复用、就地把 PTE 改回可写；只有真共享时才 `wp_page_copy` 分配新页+拷贝。

> ⚠ **源码诚实标注**：v6.12 `mm/memory.c` 里我**确认存在** `wp_page_copy()`、`wp_page_reuse()`、`finish_mkwrite_fault()` 这些函数（WebFetch 取到了它们的名字和注释），其中 `wp_page_copy` 的注释原文是 "Handle the case of a page which we actually need to copy to a new page, either due to COW or unsharing."。但 `do_wp_page()` 本体和 `wp_can_reuse_anon_folio()` 的**精确复用判定条件**位于文件末尾，单次 WebFetch 被长度截断没取到逐字，故此处复用判定的具体谓词标「待核」——我没把它编出来。要逐字请看 [mm/memory.c@v6.12](https://github.com/torvalds/linux/blob/v6.12/mm/memory.c) 的 `do_wp_page`。

### 2.4 解决方案三：clone —— fork 的泛化

`clone` 把"共享什么"做成 flags 参数。用户态原型大致：

```
clone(fn, stack, flags, arg, ...)
      |    |
      |    +-- 新 task 的栈（线程必须自带，因为要和父共享地址空间）
      +------- 入口函数（线程用；裸 clone 当 fork 用时另说）
```

**这是 v6.12 实际的 CLONE_* flag 值，全部逐字取自 UAPI 头**，看 hex 排布能直接读出"flag 空间史"：

```c
/* 【真实源码 v6.12 include/uapi/linux/sched.h】节选 */
#define CLONE_VM        0x00000100   /* 共享地址空间 mm_struct */
#define CLONE_FS        0x00000200   /* 共享 fs_struct（root/cwd/umask） */
#define CLONE_FILES     0x00000400   /* 共享 files_struct（fd 表） */
#define CLONE_SIGHAND   0x00000800   /* 共享信号处理器表 sighand */
#define CLONE_PIDFD     0x00001000   /* 返回一个 pidfd（5.2 引入，占掉倒数第二位） */
#define CLONE_PTRACE    0x00002000
#define CLONE_VFORK     0x00004000   /* 父挂起直到子 exec/exit */
#define CLONE_PARENT    0x00008000
#define CLONE_THREAD    0x00010000   /* 加入父的线程组（同 TGID）——"线程"的核心标记 */
#define CLONE_NEWNS     0x00020000   /* 新 mount namespace */
#define CLONE_SYSVSEM   0x00040000
#define CLONE_SETTLS    0x00080000   /* 设置新 task 的 TLS */
#define CLONE_PARENT_SETTID  0x00100000
#define CLONE_CHILD_CLEARTID 0x00200000   /* futex 退出清零——pthread join 的实现基石 */
#define CLONE_DETACHED       0x00400000   /* 历史遗留，已废弃但不敢回收（见下） */
#define CLONE_UNTRACED       0x00800000
#define CLONE_CHILD_SETTID   0x01000000
#define CLONE_NEWCGROUP      0x02000000
#define CLONE_NEWUTS         0x04000000
#define CLONE_NEWIPC         0x08000000
#define CLONE_NEWUSER        0x10000000
#define CLONE_NEWPID         0x20000000   /* 新 PID namespace */
#define CLONE_NEWNET         0x40000000
#define CLONE_IO             0x80000000   /* 共享 I/O context——32 位最后一位 */
#define CLONE_NEWTIME        0x00000080   /* time ns，被迫回收一个低位（见 §2.6 考古） */
#define CLONE_CLEAR_SIGHAND  0x100000000ULL   /* 已溢出到 64 位区 */
#define CLONE_INTO_CGROUP    0x200000000ULL   /* clone3 专属，fork 进指定 cgroup */
```

**逐行考古要点（这张表本身就是史料）：**
- `CLONE_IO = 0x80000000`：32 位的**最高位**，意味着 0x100~0x80000000 这 24 个 bit 在某个时刻就被用满了。
- `CLONE_PIDFD = 0x00001000` 占掉了一个位（5.2，2019）。
- `CLONE_NEWTIME = 0x00000080`：注意它的值**比 `CLONE_VM` 还小**，落在 0x80——这是因为没有连续的高位可用了，只能去回收一个历史上空着/废弃的低位。**这就是 flag 空间枯竭的源码铁证**，下文 §2.6 接上 clone3 的故事。
- `CLONE_CLEAR_SIGHAND / CLONE_INTO_CGROUP` 用 `0x1xxxxxxxxULL`：已经爬出 32 位，只有 `clone3()`（args.flags 是 `__aligned_u64`）能表达，老 `clone()`（flags 是 `unsigned long`，在传参约定下被当 32 位处理）够不着。

`pthread_create` 在 glibc/NPTL 里底层就是带 `CLONE_VM | CLONE_FS | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD | CLONE_SYSVSEM | CLONE_SETTLS | CLONE_PARENT_SETTID | CLONE_CHILD_CLEARTID` 的 clone（新 glibc 约 2.34+ 走 `clone3`）。其中 `CLONE_CHILD_CLEARTID` 是 `pthread_join` 能工作的关键：线程退出时内核把 TID 内存清零并唤醒等在那个 futex 上的 joiner。

### 2.5 四个 syscall 收敛到一个 helper（源码精读）

这是本章的"啊哈"时刻：`fork`/`vfork`/`clone`/`clone3` 在内核里**不是四套代码**，是四个 `SYSCALL_DEFINE` 各自填一个 `kernel_clone_args` 然后调同一个 `kernel_clone()`。下面四段全是逐字真实源码：

```c
/* 【真实源码 v6.12 kernel/fork.c】SYSCALL_DEFINE0(fork) */
struct kernel_clone_args args = {
        .exit_signal = SIGCHLD,
};
/* 仅设 exit_signal=SIGCHLD：不共享任何东西，退出给父发 SIGCHLD。这就是 fork 的全部。 */
```

```c
/* 【真实源码 v6.12 kernel/fork.c】SYSCALL_DEFINE0(vfork) */
struct kernel_clone_args args = {
        .flags          = CLONE_VFORK | CLONE_VM,
        .exit_signal    = SIGCHLD,
};
```

```c
/* 【真实源码 v6.12 kernel/fork.c】SYSCALL_DEFINE5(clone) */
struct kernel_clone_args args = {
        .flags          = (lower_32_bits(clone_flags) & ~CSIGNAL),
        .pidfd          = parent_tidptr,
        .child_tid      = child_tidptr,
        .parent_tid     = parent_tidptr,
        .exit_signal    = (lower_32_bits(clone_flags) & CSIGNAL),
        .stack          = newsp,
        .tls            = tls,
};
```

**`clone` 这段的 `& CSIGNAL` 是另一处活历史**：老 `clone()` 把"子进程退出时给父发哪个信号"**编码在 flags 的低 8 位**（`CSIGNAL = 0xff`）。所以 `.flags = clone_flags & ~CSIGNAL`（去掉低 8 位拿真正的 CLONE_* 标志），`.exit_signal = clone_flags & CSIGNAL`（低 8 位单独拎出来当退出信号）。这种"把两个语义塞进一个整数"的设计正是 clone3 想摆脱的——

```c
/* clone3 走 clone3_args_valid() 校验后调 kernel_clone(&kargs)。
 * exit_signal 在 clone3 里是 clone_args 结构体的独立字段，不再和 flags 挤在一起，
 * 因此 CSIGNAL 那 8 个低位对 clone3 是“可回收”的。 */
```

最后都汇到这个函数（逐字 + 注解）：

```c
/* 【真实源码 v6.12 kernel/fork.c】kernel_clone() 开头（节选） */
pid_t kernel_clone(struct kernel_clone_args *args)
{
        u64 clone_flags = args->flags;        /* 注意是 u64：才装得下 CLONE_INTO_CGROUP 等高位 */
        struct completion vfork;              /* vfork 的“父挂起”就靠这个 completion */
        struct pid *pid;
        struct task_struct *p;
        int trace = 0;
        pid_t nr;
        /* ... ptrace/seccomp 校验、trace 设置略 ... */
        p = copy_process(NULL, trace, NUMA_NO_NODE, args);  /* 真正干活的在这里 */
```

`copy_process` 的签名也是逐字真实的，注意那个 `__latent_entropy`：

```c
/* 【真实源码 v6.12 kernel/fork.c】copy_process() 签名 */
__latent_entropy struct task_struct *copy_process(
        struct pid *pid,
        int trace,
        int node,
        struct kernel_clone_args *args)
```

- `__latent_entropy`：GCC 插件（CONFIG_GCC_PLUGIN_LATENT_ENTROPY）会在这类被频繁调用的函数里**顺手收集执行路径熵**喂给内核 RNG。fork 是高频路径，是个天然的熵源——这是安全硬化的小心思，不影响功能逻辑，但能解释你在源码里看到这个奇怪注解时的疑惑。

### 2.6 clone3 的诞生：flag 空间真的用光了（设计考古 + 社区辩论）

> 出处：[LWN 790184 "fork: add clone3"，Christian Brauner，2019-06-03](https://lwn.net/Articles/790184/)；[LWN 803209 clone3 文档](https://lwn.net/Articles/803209/)；[LWN 807882 "Cloning into a control group"](https://lwn.net/Articles/807882/)。

**导火索（具体到 bit）**：到 2019 年，`clone()` 的 32 位 flag **只剩最后一位**。`CLONE_PIDFD` 吃掉它之后，正在开发的 time namespace（`CLONE_NEWTIME`）**无位可用**。本可以回收废弃的 `CLONE_DETACHED`，但它在 Mozilla Firefox、musl-libc、BLCR 等用户态被广泛（误）用，**回收会破坏 ABI**，不敢动。死路一条。

**Brauner 的设计哲学（原话引用）**：
> "The idea is to keep clone3() very simple and close to the original clone(), specifically, to keep on supporting old clone()-based workloads."
>
> "do not try to be clever or complex: keep clone3() as dumb as possible"

`struct clone_args` 全部字段（逐字真实 UAPI）：

```c
/* 【真实源码 v6.12 include/uapi/linux/sched.h】struct clone_args */
struct clone_args {
        __aligned_u64 flags;          /* CLONE_* 标志，u64 能装 64 位区的新标志 */
        __aligned_u64 pidfd;          /* 输出：pidfd 写到这个用户地址 */
        __aligned_u64 child_tid;      /* CLONE_CHILD_SETTID 的目标 */
        __aligned_u64 parent_tid;     /* CLONE_PARENT_SETTID 的目标 */
        __aligned_u64 exit_signal;    /* 独立字段！不再挤在 flags 低 8 位 */
        __aligned_u64 stack;
        __aligned_u64 stack_size;     /* 新增：栈大小显式传入（老 clone 没有） */
        __aligned_u64 tls;
        __aligned_u64 set_tid;        /* 指定子进程 PID（容器恢复/CRIU 需要） */
        __aligned_u64 set_tid_size;
        __aligned_u64 cgroup;         /* CLONE_INTO_CGROUP：直接 fork 进指定 cgroup */
};
```

**可扩展性的关键设计——size-based 而非 versioning（这里有真实的社区辩论）**：
`clone3(struct clone_args *, size_t size)` 第二参数是结构体大小。内核拿 `size` 判断用户态用的是哪一代结构：
- 用户结构比内核已知的**小** → 内核把缺的字段当 0。
- 用户结构比内核已知的**大** → 内核检查超出部分**必须全 0**，否则返回 `-E2BIG`（拒绝它不认识的新字段，防止静默忽略）。

**辩论点**：当时有两条路线——(a) 每次扩展出新 syscall（clone4、clone5…），(b) 一个 syscall 靠 size 增长结构体。社区在 Linus 建议下选了 (b)，理由是避免 syscall 数量爆炸。这个"size + 全零校验"的约定后来被沉淀成通用范式，见 [LWN 830666 "Conventions for extensible system calls"](https://lwn.net/Articles/830666/)，`perf_event_open()`、`sched_setattr()`、`openat2()` 同款。**不适用边界**：size-based 扩展要求新字段**语义上"全 0 = 旧行为"**，一旦某个新功能的"关闭态"不是 0，这套就崩了——所以新字段设计时被迫遵守"0 即默认"的纪律，这是隐性约束不是免费午餐。

### 2.7 `_do_fork` → `kernel_clone` 改名（2020, 5.10）

> 出处：[LKML "fork: introduce kernel_clone()"，Christian Brauner，2020-08-18](https://www.mail-archive.com/linux-kernel@vger.kernel.org/msg2272937.html)；patch series 原 tag `kernel-clone-v5.9`，并入 5.10。

commit message 的核心理由（取自 patch 正文）：

> "The old _do_fork() helper doesn't follow naming conventions of in-kernel helpers for syscalls."

迁移策略也很"内核风格"——不搞一次性大重构：先建 `kernel_clone()`，把 `_do_fork()` 留成 `static inline` wrapper，逐个把 caller 切过去，最后删掉 `_do_fork()`。**对你的实际意义**：任何 2020 年前的书/博客/课程里的 `do_fork`/`_do_fork`，对应今天的 `kernel_clone`，代码路径没变，只是名字诚实了——它干的事一直是 clone，从来不只是 fork。

## 三、现状：task_struct 与核心机制（源码精读）

### 3.1 task_struct 全景（前 N 字段逐字）

`task_struct` 定义在 `include/linux/sched.h`，是内核最大结构体之一。先看**真实的前若干字段**（逐字），再看语义分组：

```c
/* 【真实源码 v6.12 include/linux/sched.h】struct task_struct 开头字段 */
struct task_struct {
#ifdef CONFIG_THREAD_INFO_IN_TASK
        struct thread_info              thread_info;   /* 内核栈信息内联进 task（见下注） */
#endif
        unsigned int                    __state;       /* 运行态：R/S/D/T... 的 bitmask */
        unsigned int                    saved_state;   /* RT lock 等场景临时保存原 state */
        randomized_struct_fields_start                 /* 结构体字段随机化起点（安全硬化） */
        void                            *stack;        /* 指向内核栈 */
        refcount_t                      usage;         /* task_struct 引用计数 */
        unsigned int                    flags;         /* PF_* 标志，如 PF_EXITING/PF_KTHREAD */
        unsigned int                    ptrace;
#ifdef CONFIG_SMP
        int                             on_cpu;        /* 是否正在某 CPU 上跑（迁移/唤醒用） */
        struct __call_single_node       wake_entry;
        unsigned int                    wakee_flips;   /* wake-affinity 启发式统计 */
        unsigned long                   wakee_flip_decay_ts;
        struct task_struct              *last_wakee;
        int                             recent_used_cpu;
        int                             wake_cpu;
#endif
        int                             on_rq;         /* 是否在运行队列上 */
        int                             prio;          /* 动态优先级 */
        int                             static_prio;   /* nice 值映射来的静态优先级 */
        int                             normal_prio;
        unsigned int                    rt_priority;
        struct sched_entity             se;            /* CFS/EEVDF 调度实体（见第 5 章） */
```

**逐字段注解里的硬货：**
- `struct thread_info thread_info` 内联在 task 开头（`CONFIG_THREAD_INFO_IN_TASK`）：早年 `thread_info` 放在内核栈底部，栈溢出会直接踩坏它（一个经典攻击面）。把它**挪进 `task_struct`** 是一次安全硬化——栈溢出再也踩不到 `thread_info` 里的 `addr_limit` 等敏感字段。这是"为什么这个字段在结构体最前面"的根因。
- `__state`（不是 `state`）：近年从 `state` **改名 `__state`**，目的是逼所有访问走 `READ_ONCE`/`WRITE_ONCE` 包装宏，防止编译器对状态读写做不安全重排（状态机和唤醒之间有微妙的内存序要求）。你在老内核/老博客看到 `task->state`，在 5.14+ 就是 `task->__state`——bpftrace 脚本要据此改字段名（下文 §4.1）。
- `saved_state`：PREEMPT_RT 里 RT mutex 阻塞时，要临时把 `__state` 改成阻塞态又不能丢掉原来的 `TASK_INTERRUPTIBLE` 等，于是用 `saved_state` 暂存。普通内核也保留这个字段。
- `randomized_struct_fields_start`：CONFIG_RANDSTRUCT 会随机化中间这些字段的布局，让攻击者无法靠硬编码偏移读写 `task_struct`——所以你**不能假设 `task_struct` 字段偏移稳定**，跨内核版本/配置写偏移是自杀。

按语义分组（这部分是导读，非逐字）：

```
task_struct（语义分组导读）
├── 调度    __state / prio 系 / sched_entity se（见第 5 章）
├── 标识    pid（每 task 唯一）/ tgid（getpid 返回的）/ comm[16]（进程名，15+1）
├── 地址空间 mm / active_mm（内核线程 mm=NULL，借用 active_mm = lazy TLB）
├── 文件    fs（fs_struct: cwd/root）/ files（files_struct: fd 表）
├── 信号    sighand（线程组共享处理器）/ signal（组共享信息）/ 私有 pending + 组 pending
├── 亲子    real_parent / parent / children / sibling
├── 命名空间 nsproxy（一组 namespace，见第 22 章）
└── 退出    exit_code / exit_signal / exit_state（EXIT_ZOMBIE/EXIT_DEAD）
```

**三个最常被问到的指针的"共享=指针相等"本质：**
1. `mm`：同进程所有线程指向同一 `mm_struct`。内核线程 `mm == NULL`，但借用前一个用户进程的 `active_mm` 避免无谓 TLB 刷——这叫 **lazy TLB**：内核线程不碰用户地址空间，没必要切页表/刷 TLB，直接蹭上一个的。
2. `files`：`CLONE_FILES` 让两 task 共享同一 `files_struct`，所以一个线程 `close(fd)` 另一个线程立刻看到 fd 没了。
3. `sighand`：线程组共享信号**处理器表**，但每个 task 有自己的**私有 pending 队列**（定向到该线程的信号）和**线程组共享 pending 队列**（定向到进程的信号）。这套双队列是 §3.5 "kill 发给谁"的实现基础。

### 3.2 进程状态机（状态值是 ABI，逐字给）

先看**真实的状态常量定义**（逐字，含确切 hex），再讲状态机——因为这些值是你排障时要硬编码的 ABI：

```c
/* 【真实源码 v6.12 include/linux/sched.h】task state 常量 */
/* Used in tsk->__state: */
#define TASK_RUNNING            0x00000000
#define TASK_INTERRUPTIBLE      0x00000001
#define TASK_UNINTERRUPTIBLE    0x00000002
#define __TASK_STOPPED          0x00000004
#define __TASK_TRACED           0x00000008
/* Used in tsk->exit_state: */
#define EXIT_DEAD               0x00000010
#define EXIT_ZOMBIE             0x00000020
/* Used in tsk->__state again: */
#define TASK_PARKED             0x00000040
#define TASK_DEAD               0x00000080
#define TASK_WAKEKILL           0x00000100   /* 允许 fatal signal 唤醒 */
#define TASK_WAKING             0x00000200
#define TASK_NOLOAD             0x00000400   /* 不计入 loadavg */
#define TASK_NEW                0x00000800
#define TASK_RTLOCK_WAIT        0x00001000
#define TASK_FREEZABLE          0x00002000
#define TASK_FROZEN             0x00008000

/* 便利组合（逐字） */
#define TASK_KILLABLE   (TASK_WAKEKILL | TASK_UNINTERRUPTIBLE)
#define TASK_STOPPED    (TASK_WAKEKILL | __TASK_STOPPED)
#define TASK_IDLE       (TASK_UNINTERRUPTIBLE | TASK_NOLOAD)
```

注意源码里专门有一段注释警告这些位是 ABI、且和 procfs 强耦合（逐字）：

```c
/* 【真实源码 v6.12 include/linux/sched.h】task state 注释块 */
/*
 * Task state bitmask. NOTE! These bits are also
 * encoded in fs/proc/array.c: get_task_state().
 *
 * We have two separate sets of flags: task->__state
 * is about runnability, while task->exit_state are
 * about the task exiting. Confusing, but this way
 * modifying one set can't modify the other one by
 * mistake.
 */
```

**从源码读出的三个非平凡事实：**
1. **`__state` 和 `exit_state` 是两套字段**，故意分开（注释明说 "modifying one set can't modify the other one by mistake"）。`EXIT_ZOMBIE/EXIT_DEAD` 放 `exit_state`，运行态放 `__state`。所以 bpftrace 抓 D 态查的是 `__state == 2`，查僵尸要看 `exit_state`。
2. **`TASK_KILLABLE` 不是独立状态，是组合**：`TASK_WAKEKILL | TASK_UNINTERRUPTIBLE`。它**就是**不可中断睡眠，只是多带了 `TASK_WAKEKILL` 这个"允许致命信号唤醒"的修饰位。这解释了为什么它"像 D 态但能被 kill -9 救"——底层确实是 D 态，区别只在那一个 bit。
3. **`TASK_IDLE = TASK_UNINTERRUPTIBLE | TASK_NOLOAD`**：很多内核线程"睡着等活"时用 `TASK_IDLE`，它是 D 态但带 `TASK_NOLOAD`——**不计入 load average**。这是"为什么一堆 kworker 在睡但 loadavg 不爆"的根因，也是"为什么 Linux loadavg 把 D 态算进去导致它和 CPU 利用率不是一回事"这个经典困惑的另一面。

状态机（导读）：

```
                    wakeup
          +--------------------------+
          |                          |
   [TASK_RUNNING] <---- [TASK_INTERRUPTIBLE]  (S，信号或事件可唤醒)
       |   ^                  ^
       |   +-- 调度回来        | 等可打断资源
       v                      |
   [TASK_UNINTERRUPTIBLE] ----+  (D，完成等待→RUNNING；kill -9 不管用)
       |                          ……除非这条等待路径用了 *_killable（TASK_WAKEKILL 位）
   [__TASK_STOPPED]  -- SIGSTOP / ptrace（T）
       |
   [EXIT_ZOMBIE]  -- exit() 后，等父 wait()（Z，住在 exit_state）
       |
   [EXIT_DEAD]    -- 父 wait() 后彻底释放
```

#### D 态为什么杀不掉——底层根因，不是"内核 bug"

`TASK_UNINTERRUPTIBLE` 是驱动/文件系统向调度器表达的承诺："这段等待期间若被信号打断，内核数据结构可能处于不一致状态，后果不可预料。" 典型：磁盘 I/O 等待、NFS 请求、某些拿内核锁的路径。

**为什么 kill -9 不管用（机制根因）**：`SIGKILL` 的响应点是**进程从内核态返回用户态的边界**（syscall/中断出口处的 `do_signal`）。D 态进程卡在内核里那个 `schedule()` 调用上，**根本走不到返回用户态的边界**，所以信号被设进 pending 队列但永远没机会 delivered。这不是"信号被忽略"，是"信号没到检查点"——区别很重要：进程一旦 I/O 完成被唤醒、跑到返回边界，pending 的 SIGKILL 立刻生效（所以很多 D 态"等存储恢复就自己死了"）。

#### 逃生门：TASK_KILLABLE（2008 就有，别当新东西）

> 出处：[LWN 288056 "TASK_KILLABLE"，Jonathan Corbet，2008-07-01](https://lwn.net/Articles/288056/)，并入 **2.6.25**。

Matthew Wilcox 的洞察（Corbet 转述的原话）：
> "It does not matter if the developer thought about the possibility of an interrupted system call if said system call is doomed to never return to user space."

翻译成工程语言：开发者用 `TASK_UNINTERRUPTIBLE` 是怕"信号打断导致 syscall 半途返回、应用没处理好 EINTR"——但如果这个进程**马上要被杀了**，根本不会返回用户态，那些 EINTR 顾虑统统不成立。所以可以安全地让 **fatal signal**（且仅 fatal signal）打断它。

配套原语（逐字来自 LWN）：`wait_event_killable()`、`schedule_timeout_killable()`、`mutex_lock_killable()`、`wait_for_completion_killable()`、`down_killable()`。各返回 0 成功 / 负错误码（收到 fatal signal）。

**关键的诚实**（这是原章节做对、很多博客做错的点）：能不能被杀**取决于那条等待路径的作者选了哪个原语**。Corbet 在 2008 年就指出采纳很慢——"The NFS client code has been converted... But there are very few other uses of TASK_KILLABLE."。十几年后好转，但 **NFS hard mount 默认仍进真 D 态**，很多本地块设备 I/O 等待也仍不可中断。所以生产上你**照样会遇到杀不掉的 D 态**，KILLABLE 只救回了一部分场景。后续 [LWN 754980 "Killing processes that don't want to die" (2018)](https://lwn.net/Articles/754980/) 讨论了进一步的努力。

> 💡 **生产真坑**：`ps aux` 看到某进程 D 态超过几秒，**几乎肯定是存储链路问题的前兆**——NFS 服务器无响应、SAN 超时、本地 SSD 固件 bug、过载的 dm-crypt/LVM。不是进程 bug，从存储链路排（§4.1 给标准流程）。

### 3.3 fork 的完整路径（对照源码）

```
用户态 fork()
  ↓ syscall（int 0x80 / syscall 指令，见第 3 章）
SYSCALL_DEFINE0(fork)  →  args = {.exit_signal = SIGCHLD}
  ↓
kernel_clone(&args)          【5.10+ 名；旧内核 _do_fork / do_fork】
  ↓
copy_process(NULL, trace, NUMA_NO_NODE, args)   【__latent_entropy 标注的那个】
  ├── dup_task_struct()    分配新 task_struct + 内核栈（含 stack canary）
  ├── copy_creds()         凭证
  ├── copy_sighand()       CLONE_SIGHAND ? 共享 : 复制
  ├── copy_signal()        信号结构
  ├── copy_mm()            ← CLONE_VM 分支：mmget 共享 / dup_mm 复制+设 COW
  │   └── dup_mm → dup_mmap   逐 VMA 复制 PTE 并改只读
  ├── copy_files()         CLONE_FILES ? 共享 : 复制 fd 表
  ├── copy_namespaces()    CLONE_NEW* ? 新建 : 共享
  ├── copy_thread()        架构相关：设好子进程的寄存器/返回值=0
  └── alloc_pid()          分配新 PID（pid 命名空间相关）
  ↓
wake_up_new_task(p)        加入运行队列（child 此时才可被调度）
  ↓
return child_pid          父进程拿到子 PID；子进程 fork() 返回 0
```

`alloc_pid` 的调用是逐字真实的（注意它和 namespace、`set_tid` 的关系）：

```c
/* 【真实源码 v6.12 kernel/fork.c】copy_process() 内 alloc_pid 调用（节选） */
if (pid != &init_struct_pid) {
        pid = alloc_pid(p->nsproxy->pid_ns_for_children,
                        args->set_tid, args->set_tid_size);
```

- `p->nsproxy->pid_ns_for_children`：在**子进程所属的 PID namespace**里分配 PID——这是容器里"PID 1"能存在的根：每个 PID ns 有独立的 PID 分配器。
- `args->set_tid`：`clone3` 的 `set_tid` 能**指定** PID（受限），CRIU 进程恢复/容器迁移要用它把进程恢复成原来的 PID。老 `fork`/`clone` 给不了这能力——这是 clone3 的实战价值之一。

**为什么 fork 后通常父进程先运行**：早年内核曾让子进程先跑（便于立刻 exec、减少 COW 故障）。CFS 起默认改父先跑，由 `sysctl kernel.sched_child_runs_first` 控制，**默认 0**（2009 年的 commit 把默认从 1 翻成 0）。

> ⚠ **时效提醒**：这个 knob 在现代调度器下影响很弱（只影响入队后相对 vruntime/eligibility，不保证严格顺序）。**别写依赖 fork 后谁先跑的代码**——要同步用 pipe/eventfd 显式握手。EEVDF（6.6+）重写 eligibility 后语义进一步弱化，跨版本观察到的行为可能不一致。

### 3.4 exec 的关键语义

`execve` 是真 syscall（其余 `execl`/`execlp`… 是 libc 包装）。它做的：
1. 检查 magic：`#!` shebang 或 ELF magic `\x7fELF`（`binfmt_*` 处理器分派）。
2. **放弃当前 `mm_struct`**（`exec_mmap`/`mm_release`），建全新地址空间——COW 共享的页此刻全断开。
3. 加载 ELF segments，设 `brk`（堆起点）、栈（含 argv/envp/auxv）。
4. **PID/TGID 不变**，`task_struct` 复用；fd 按 `FD_CLOEXEC` 决定关不关；信号处理器 reset 到默认（除被忽略的）。
5. 跳到 ELF entry（或动态链接器 ld.so 入口）。

> 💡 **生产真坑（多线程 exec）**：exec 后，**同线程组的其他线程全部被杀**——exec 是"换掉整个进程"不是"换掉一个线程"。多线程程序里某个非主线程调 `execve`，效果等同从 main 调，其余线程蒸发。内核做法是先把线程组里其他线程 `zap`（de_thread），让调用线程"上位"成新进程的唯一线程。如果其他线程当时持有锁，它们被杀但锁状态由内核地址空间一并丢弃（因为整个 mm 都换了），所以不像 fork 那样留下死锁——但任何依赖"那些线程会跑完"的逻辑都会静默丢失。

### 3.5 PID、TGID 与线程组

```
主线程 task A:  pid=1001, tgid=1001   ← "进程 PID" = TGID = 主线程的内核 PID
  ↓ pthread_create → clone(... | CLONE_THREAD | CLONE_VM | ...)
线程 task B:    pid=1002, tgid=1001
线程 task C:    pid=1003, tgid=1001
```

**用户态 `getpid()` 返回 TGID**，因为 POSIX 要求同进程所有线程看到同一 PID。内核 `getpid` syscall 返回 `task_struct.tgid`；`gettid()` 返回 `task_struct.pid`（线程唯一的内核 PID）。

**从这套映射推出的实战事实：**
1. `/proc/PID` 以 **TGID** 索引：`/proc/1001/` 代表整个进程；`/proc/1001/task/1002/` 才是线程粒度。
2. `kill(pid, sig)` 发给**进程**（TGID），内核从线程组里**挑一个没 block 该信号的线程**送达（用的就是 §3.1 的"组共享 pending 队列"）；`tgkill(tgid, tid, sig)` 精确发给特定线程（pthread 的 `pthread_kill` 底层就是它）。
3. 线程组成员共享 `sighand` 和忽略/默认动作（`signal->action`），但**各自有独立的 `blocked`（signal mask）**——所以"哪个线程收到进程定向信号"取决于谁没 block 它。这是多线程信号处理最反直觉的地方：你以为发给进程的 SIGTERM 会到主线程，实际可能被任意一个没 block 它的线程截获。**最佳实践**：多线程程序用专门一个线程 `sigwait` 同步处理信号，其余线程全 block。

### 3.6 进程树、init、孤儿与 subreaper（源码精读 exit 路径）

Linux 进程树以 PID 1 为根（`/sbin/init` 或 `systemd`）。内核线程由 `kthreadd`（PID 2）创建，`mm == NULL`，`ps` 里显示 `[name]`。

**孤儿过继的真实逻辑**（父先于子退出）——不是简单"甩给 PID 1"，而是沿祖先链找最近的 **child subreaper**。看逐字源码：

```c
/* 【真实源码 v6.12 kernel/exit.c】forget_original_parent() 重定父（节选） */
reaper = find_child_reaper(father, dead);
if (list_empty(&father->children))
        return;

reaper = find_new_reaper(father, reaper);
list_for_each_entry(p, &father->children, sibling) {
        for_each_thread(p, t) {
                RCU_INIT_POINTER(t->real_parent, reaper);
```

- `find_new_reaper` 沿**祖先链向上**找第一个标了 `PR_SET_CHILD_SUBREAPER` 的祖先；没有才一路兜底到**本 PID namespace 的 PID 1**（不是宿主机的）。
- subreaper **不是天生属性**，进程要主动 `prctl(PR_SET_CHILD_SUBREAPER, 1)`。systemd 的 user/service manager 就靠它把一棵服务子树的孤儿留在自己手里 reap，而非甩给真正的 PID 1。
- `for_each_thread` + `RCU_INIT_POINTER`：用 RCU 改 `real_parent` 指针，因为可能有读者（如 `wait`）正在并发遍历，不能直接赋值。

> 💡 **生产真坑（容器僵尸）**：Docker 里 `CMD ["./app"]` 直接起业务进程时，它是 **PID 1**。业务进程通常没 `SIGCHLD` handler、没 wait 逻辑——它派生的子进程退出就成僵尸，**无人收尸**。更糟：本 PID namespace 的 PID 表可能被僵尸耗尽。每个 PID ns 有自己的"1 号进程"，过继兜底是兜到**本 ns 的 PID 1**（就是你那个不会 reap 的业务进程），不是宿主机 init。这就是为什么容器要 `tini`（或 `docker run --init`）当 PID 1——它会 reap 孤儿、转发信号。

### 3.7 僵尸进程的机制（对照 exit.c）

进程 `exit()` 时，内核释放大部分资源（内存、fd、信号等），但**保留 `task_struct` 骨架**（含退出码、CPU 时间统计）。这状态叫 zombie（`exit_state = EXIT_ZOMBIE`）。看逐字源码——状态在 `exit_notify()` 里设置：

```c
/* 【真实源码 v6.12 kernel/exit.c】exit_notify() 设僵尸/自动回收（节选） */
tsk->exit_state = EXIT_ZOMBIE;
/* ... 通知父进程、判断是否需要父 wait ... */
if (autoreap) {
        tsk->exit_state = EXIT_DEAD;          /* 没人需要等 → 直接进 DEAD */
        list_add(&tsk->ptrace_entry, &dead);
}
```

父进程 `wait()`/`waitpid()` 读到退出码后，内核才在 `wait_task_zombie()` 里把它推进到 `EXIT_DEAD` 并 `release_task` 彻底释放。源码里那个"只能有一个线程做这事"的注释很关键（逐字）：

```c
/* 【真实源码 v6.12 kernel/exit.c】wait_task_zombie() 内（节选） */
/*
 * Move the task's state to DEAD/TRACE, only one thread can do this.
 */
state = (ptrace_reparented(p) && thread_group_leader(p)) ?
        EXIT_TRACE : EXIT_DEAD;
```

- **`autoreap`**：若父进程对 `SIGCHLD` 设了 `SIG_IGN` 或带 `SA_NOCLDWAIT`，内核知道"父根本不想 wait"，于是**跳过僵尸态直接 DEAD**——这是"我 ignore SIGCHLD 就不产生僵尸"这条经验法则的源码根因。
- **"only one thread can do this"**：多线程父进程里，可能多个线程同时 `wait` 到同一个僵尸子进程，必须用原子的状态转换保证只有一个真正 `release_task`，否则 double free。

**zombie 不是内存泄漏**：保留的 `task_struct` 是 KB 级。真正的问题是父进程**持续 fork 不 wait**，僵尸堆积最终**耗尽 PID 空间**（`pid_max` 默认约 32768，可调到 4194304）。

排查：
```bash
ps aux | awk '$8 ~ /Z/'                       # 找僵尸
ps -eo pid,ppid,stat,comm | awk '$3 ~ /Z/'    # 找僵尸的父进程
# 然后处理父进程（kill 父 → 僵尸 reparent 到 subreaper/init → 被 reap）
```

> **为什么 `__state == 2` 这个魔数能放心写**：因为 §3.2 的源码注释明说这些位 ABI 稳定且编码进 procfs。你赌的是内核维护者不敢改 `TASK_UNINTERRUPTIBLE` 的值——这个赌注 20 年没输过。但**字段名**（`state` vs `__state`）不是 ABI（是内核内部符号），所以会变，要按版本切换。这就是"值能硬编码、名要适配"的根因。

### 4.2 fork 速度优化——瓶颈不在 COW 拷内存

大量 fork 场景（web prefork）的真实瓶颈：

1. **页表复制（`dup_mmap`）**：父进程 VMA 多、映射大时，逐 VMA 复制 PTE 是真实开销。一个映射了 100GB 的进程，即使 COW 一个物理页都不拷，**光复制页表项**就可能是几十毫秒级。**缓解**：用 hugepage（2MB/1GB）减少 PTE 数量——同样 100GB，4KB 页要 2600 万个 PTE，2MB 页只要 5 万个，页表复制开销差 512 倍。
2. **TLB shootdown**：COW 改 PTE（改只读）后，要发 IPI（处理器间中断）通知**所有跑过这个 mm 的 CPU** 刷 TLB。多核 + 多线程进程里，这是真实的可扩展性瓶颈（IPI 是同步的，要等所有目标 CPU ack）。这也是"为什么大内存多线程进程 fork 慢"的微架构根因——不是拷贝慢，是**核间同步**慢。
3. **NUMA 碎片**：子进程 COW 触发时若本 NUMA 节点内存碎片严重，缺页延迟显著增大。

**Nginx prefork 为什么快**：master 在 `accept` loop 前 fork worker，**此时地址空间还小、页表浅**，fork 快。之后每个 worker 独立运行，COW 物理页分配发生在各自独立时间点，不集中、不互相 IPI 干扰。**对照反例**：一个已经跑了很久、堆了几十 GB 堆的进程再去 fork，就会吃满上面三个开销——所以"要 fork 趁早、趁地址空间小"。

### 4.3 线程 vs 进程 vs 协程——带场景跑一遍

| 维度 | 多进程 | 多线程（1:1） | 协程（N:M） |
|---|---|---|---|
| 隔离性 | 强（崩溃不互染） | 弱（一个线程越界 segfault 整个进程） | 弱（同进程内） |
| 内存开销 | 高（各自页表/地址空间） | 中（共享 mm，但每线程默认 8MB 栈虚拟空间） | 低（KB 级栈，按需增长） |
| 创建成本 | 高（copy_process 全套 + 可能 exec） | 中（copy_process 但共享 mm/files/...） | 极低（用户态，无 syscall） |
| 通信成本 | 高（IPC/共享内存/pipe） | 低（共享堆，但要锁） | 低（共享堆 + 调度器协作） |
| 上下文切换 | 重（内核态，切页表+刷 TLB） | 中（内核态，同进程不刷 TLB） | 轻（用户态，无内核介入） |
| 调度公平/抢占 | 内核保证 | 内核保证 | 取决于 runtime（Go 有抢占，部分协程库没有） |
| 典型选择 | Nginx、PostgreSQL | Java JVM 线程池、传统 C++ 服务 | Go runtime、Erlang、Rust async |

**场景跑一遍：10k 并发连接，每连接需独立上下文。**
- **多进程 prefork**：10k 进程 = 10k 份 `task_struct` + 10k 份页表/地址空间。光内存就吃不消，上下文切换全是重切换（刷 TLB）。**出局**——除非连接数远小于此（如 PostgreSQL 早年几百连接场景）。
- **多线程 1:1**：10k 线程 = 10k 个 `task_struct`（共享 mm，省了地址空间复制）。但**每线程默认 8MB 栈虚拟地址空间 → 10k 线程 = 80GB 虚拟地址**（虚拟，不是物理，但 32 位时代直接爆地址空间；64 位下虚拟够但物理栈 + 内核栈 + 调度开销仍重）。上下文切换 10k 个内核线程，调度器压力和 cache 抖动明显。**勉强可用但不优**。
- **协程 N:M（Go）**：10k goroutine 跑在 GOMAXPROCS 个 OS 线程上。每个 goroutine 初始栈 ~2KB（可增长），10k 个 = ~20MB 量级。切换在用户态、无 syscall、无 TLB 刷。**这才是 10k+ 连接的正解**——这正是 Go/Erlang 在高并发网络服务统治的原因。**不适用边界**：协程吃不到"一个连接一个内核线程"的强抢占语义，若某 goroutine 跑一段不让出的 CPU 密集循环，老版本 Go（无异步抢占前）会饿死同线程其他 goroutine；且协程不提供进程级隔离，一个崩全崩。

> 🎯 **面试加分点（PostgreSQL 为什么多进程）**：历史原因是**安全隔离**（一个查询越界不污染其他连接）+ 早年各平台 pthreads 实现质量参差的**可移植性**。代价是 shared buffer pool 必须用共享内存（`shmget`/mmap），且连接数一大 `fork` 开销 + 每连接进程内存（work_mem 等）顶不住——**这就是 pgbouncer 连接池存在的根本原因**，把几千客户端连接复用到几十个后端进程上。PostgreSQL 16+ 才开始认真讨论线程化，正是因为这套多进程模型在高连接数下的开销在云时代变得刺眼。

## 五、面试视角：Staff 级高频题

### Q1：进程和线程的内核级本质区别？
- 开门见山：内核**没有独立"线程"结构**，都是 `task_struct`，连创建 syscall 都收敛到 `kernel_clone()`。
- 区别在 clone flags：线程带 `CLONE_VM | CLONE_FILES | CLONE_SIGHAND | CLONE_THREAD`，共享地址空间/fd/信号处理器，加入同一线程组（同 TGID）。
- 引申到底层：线程切换和进程切换在内核**几乎一样重**（都要 context switch、切寄存器、切内核栈），**唯一省的是 TLB**——同进程线程共享 `mm_struct`，切换不刷 TLB；进程切换要切页表 + 刷 TLB（除非有 PCID/ASID 硬件支持，能按地址空间标记 TLB 条目避免全刷）。
- **陷阱**：只说"线程共享内存"会被追问到 `mm_struct`、TLB、PCID 层。

### Q2：fork 后父子哪些共享、哪些独立？
- 共享（COW）：**物理内存页**（写时才复制）；fd 指向的 `struct file`（引用计数 +1，**file offset 共享**）。
- 独立副本：虚拟地址空间结构（`mm_struct`、页表）——逻辑独立，物理 COW；PID/TGID；用户栈（COW，写则分裂）。
- **陷阱根源**：父子写**同一个 fd** 会互相影响 offset（因为共享底层 `struct file`）——这是"fork 后忘了关 fd / 父子都写日志文件错乱"的经典 bug。注意区分：`dup` 出的 fd 共享 offset，`open` 同一文件两次的两个 fd **不**共享 offset。

### Q3：D 态 kill -9 为什么无效？怎么处理？
- 信号 delivery 在内核→用户态边界；D 态卡在内核 `schedule()`，到不了那个边界，信号 pending 但不 delivered。
- `TASK_KILLABLE`（**2.6.25 就有，不是新东西**）允许 fatal signal 打断——但前提是**那条等待路径用了 `*_killable` 原语**；NFS hard mount、很多块设备 I/O 仍是真 D 态。
- 处理：`wchan`/`stack` 定位等什么 → NFS 修服务器 / 硬件 I/O 超时查 `iostat`+`dmesg` → 短期重启卡住服务（I/O 一返回 D 态自解）→ 极端可 `hung_task_panic` + kdump 取证（生产慎用）。

### Q4：僵尸怎么产生、怎么消灭？
- 产生：子退出，父没 `wait()`（`exit_state = EXIT_ZOMBIE`，保留 task_struct 等父读退出码）。
- 消灭：(a) 父有 `SIGCHLD` handler 去 wait；(b) kill 父 → 子 reparent 到 subreaper/init，由其 reap；(c) 父对 `SIGCHLD` 设 `SIG_IGN`/`SA_NOCLDWAIT` → 内核 autoreap，**根本不产生僵尸**（源码 `autoreap` 分支）。
- **不能做**：kill 僵尸本身无效（它已不运行）。

### Q5：fork bomb 为什么打死系统？怎么防？
- 指数增长耗尽 `pid_max`，所有创建失败，系统僵死。
- 防：`ulimit -u`（RLIMIT_NPROC，每用户）；systemd 单元 `TasksMax`；**cgroup v2 `pids.max`（最彻底，按层级隔离，容器精确限上限不影响宿主）**。

### Q6：exec 后哪些继承、哪些丢弃？
- 继承：PID/PPID、fd（除非 `O_CLOEXEC`/`FD_CLOEXEC`）、signal mask、进程组/session、umask。
- 丢弃：整个地址空间、信号 pending 队列、信号处理器（reset 默认，被 ignore 的保留）、内存映射、其他线程。
- **加分**：`O_CLOEXEC` 要在 `open()`/`socket()` 时就带（原子），别事后 `fcntl`——否则 fork-exec 之间有窗口泄漏 fd（多线程下尤其，另一个线程可能在窗口内 fork）。

### Q7：getpid() vs gettid()？何时不同？
- `getpid()` → TGID（`task_struct.tgid`），同进程所有线程相同。
- `gettid()` → 内核 PID（`task_struct.pid`），线程唯一。
- 不同：多线程进程里非主线程 `gettid() != getpid()`；主线程 `gettid() == getpid()`（TGID = 主线程内核 PID）。
- 用途：`strace -f`/日志里区分线程；futex 的 TID-based ownership（如 robust mutex）依赖 gettid。

## 六、章末五件套

### 6.1 高频面试题（含陷阱）
1. **进程/线程内核本质区别** — 见 Q1；陷阱：只说"共享内存"，忘 TLB/PCID/线程组。
2. **fork COW 原理 + 真正开销** — COW 延迟物理拷贝，但**页表复制 + TLB shootdown** 是真实开销；陷阱：以为 fork 后内存立刻翻倍。
3. **D 态 kill 无效 + 排查** — 见 Q3；陷阱：答"内核 bug"或"等等就好"。
4. **僵尸原理 + 消灭** — 见 Q4；陷阱：以为 kill 僵尸有效。
5. **多线程 fork 的危险** — fork 后子进程**只有调用线程存活**，其他线程持有的 mutex **不会被释放**，子进程里可能死锁（malloc 内部锁是经典受害者）——这是 `pthread_atfork` 存在的原因。
6. **TGID vs PID，/proc 索引谁** — `/proc` 以 **TGID** 索引；见 Q7。
7. **exec fd 泄漏怎么防** — `O_CLOEXEC`；陷阱：以为 exec 自动关所有 fd。
8. **【深化新增】clone3 为什么出现** — clone flag 32 位用光（`CLONE_PIDFD` 占最后一位，`CLONE_NEWTIME` 无位可用），size-based 可扩展 syscall；陷阱：答"为了更好的 API"——具体导火索是 flag 枯竭。
9. **【深化新增】为什么内核里 fork 的 helper 叫 `kernel_clone` 不叫 do_fork** — 5.10 改名，因为它本质是 clone backend，"fork"名不副实；读旧资料的 `_do_fork` 即此。

### 6.2 实战项目 P0：观察多线程进程的 /proc 呈现

```c
/* thread_demo.c —— gcc -pthread thread_demo.c -o thread_demo */
#define _GNU_SOURCE
#include <pthread.h>
#include <stdio.h>
#include <unistd.h>
#include <sys/syscall.h>   /* SYS_gettid：比硬编码号可移植 */

void *worker(void *arg) {
    pid_t tid = (pid_t)syscall(SYS_gettid);   /* 内核 PID，线程唯一 */
    printf("thread: getpid()=%d (TGID)  gettid()=%d (内核PID)\n", getpid(), tid);
    sleep(60);   /* 保持存活以便 /proc 观察 */
    return NULL;
}
int main(void) {
    pthread_t t1, t2;
    pthread_create(&t1, NULL, worker, NULL);
    pthread_create(&t2, NULL, worker, NULL);
    printf("main:   getpid()=%d  gettid()=%ld\n",
           getpid(), (long)syscall(SYS_gettid));   /* 主线程 gettid==getpid */
    sleep(60);
    return 0;
}
```

### 6.5 系统题
**场景**：生产服务器某进程 D 态线程从 0 突增到 50+，持续 5 分钟以上，业务超时率 30%。排查步骤？
**参考**：
1. `ps aux | awk '$8 ~ /D/'` 确认数量和进程名。
2. `cat /proc/PID/wchan` 或 `sudo cat /proc/PID/stack` 定位等待的内核函数（读不到用 `offcputime-bpfcc -K 5` 按内核栈聚合）。
3. NFS/CIFS：`nfsstat`、`mountstats` 看服务器响应；检查 NFS 服务器可达性。
4. 本地磁盘：`iostat -x 1`、`sar -d 1` 看 await/util；`dmesg | grep -iE 'error|timeout'` 查硬件/驱动报错。
5. 内存压力导致 swap D 态：`vmstat 1`、`free -h`，考虑 OOM 或扩容。
6. 短期：重启卡住服务（I/O 返回后 D 态自解）；根治：修底层存储。

### 6.6 代码题：fork/exec 资源继承观察工具

```c
/* fork_exec_demo.c —— gcc fork_exec_demo.c -o fork_exec_demo
 * ⚠ 暗坑：多线程程序里 fork 只有调用线程存活，其他线程持有的锁不释放。本例单线程，安全。 */
#define _GNU_SOURCE
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/wait.h>
#include <fcntl.h>

int main(void) {
    int fd_leak = open("/dev/null", O_RDONLY);                 /* 无 CLOEXEC，会穿越 exec */
    int fd_safe = open("/dev/null", O_RDONLY | O_CLOEXEC);     /* 带 CLOEXEC，exec 时关闭 */
    printf("[parent] pid=%d ppid=%d fd_leak=%d fd_safe=%d\n",
           getpid(), getppid(), fd_leak, fd_safe);

    pid_t child = fork();
    if (child < 0) { perror("fork"); return 1; }
    if (child == 0) {
        printf("[child]  pid=%d ppid=%d\n", getpid(), getppid());
        printf("[child]  fd_leak 仍开: %s\n", fcntl(fd_leak, F_GETFD) != -1 ? "是" : "否");
        printf("[child]  fd_safe 仍开: %s\n", fcntl(fd_safe, F_GETFD) != -1 ? "是" : "否");
        execlp("ls", "ls", "-la", "/proc/self/fd", NULL);      /* exec 后看哪些 fd 还在 */
        perror("exec failed"); exit(1);
    }
    int status; waitpid(child, &status, 0);
    printf("[parent] child exited status %d\n", WEXITSTATUS(status));
    close(fd_leak); close(fd_safe);
    return 0;
}
```
**验证**：`./fork_exec_demo`，看子进程 exec 后 `/proc/self/fd` 输出——`fd_leak` 对应的 `/dev/null` 仍在，`fd_safe` 已被 exec 关闭、不在列表。

**暗坑提示**：
1. `fork` 在多线程程序里是 async-signal-safety 地雷区：子进程只有 fork 线程存活，libc 内部 malloc lock 若被其他线程持有，子进程里 malloc 永久死锁——`pthread_atfork` 就是为在 fork 前后维护这类锁而生。
2. `exec` 后 fd 泄漏是真实安全漏洞：特权进程 exec 普通程序时泄漏的 fd 可能暴露敏感文件（甚至提权）。现代代码 `open(path, ... | O_CLOEXEC)` 是最佳实践。
3. `vfork` 现代无明确性能优势（COW + 现代 MMU），新代码不用；库里 `posix_spawn` 的 vfork 用法是受控特例。

## 七、未来演进

1. **pidfd：进程的 fd 化**：`pidfd_open`（5.3）以 fd 引用进程，根治经典的 **PID 重用竞态**（你 `kill(pid)` 时目标已死、新进程占了同一 PID → 误杀）。`pidfd_send_signal`、`waitid(P_PIDFD)`、`clone3` 的 `CLONE_PIDFD` 构成更安全的进程管理 API。容器运行时（runc 等）已广泛用 pidfd 避免误杀。
2. **clone3 扩展能力落地**：`set_tid`（指定 PID，CRIU/容器迁移）、`CLONE_INTO_CGROUP`（直接 fork 进目标 cgroup，省掉"fork 后再 move 进 cgroup"的竞态窗口）已被容器运行时采用。size-based 可扩展使后续新能力无需再造 syscall。
3. **io_uring 与任务模型演化**：io_uring 的 `io-wq` async worker 线程池进一步解耦"执行上下文"和"内核任务"的绑定——某种意义上是内核线程模型的下一步（见 io_uring 章）。
4. **Rust for Linux 的 `Task` 抽象**：为 `task_struct` 构建安全 Rust 包装，解决 task 生命周期 + 跨线程引用安全（`task_struct` 的引用计数 `usage`/`get_task_struct`/`put_task_struct` 用错就是 UAF）——内核安全演进重要方向。
5. **EEVDF 调度器（6.6+）替代 CFS**：改变线程 latency 保证机制，对高频 fork/thread 场景的延迟特性有影响；旧的 `sched_child_runs_first` 等 knob 语义进一步弱化（见第 5 章）。
6. **PREEMPT_RT 主线化**：实时抢占合入主线后，更多内核临界区可被抢占，`saved_state`/RT mutex 这类机制从"打补丁"变成一等公民，影响 D 态/睡眠语义的边界。

---

> **本章心法**：Linux 内核只认 `task_struct`——进程是独享一切的 task，线程是共享大部分的 task；四个创建 syscall 收敛到一个 `kernel_clone`，"共享什么"是一组 flag 的选择，而这组 flag 的历史（vfork 的妥协、COW 的延迟、clone 的泛化、flag 空间枯竭逼出 clone3）就是这个子系统的全部设计张力。fork 的廉价靠 COW（但开销在页表 + TLB shootdown，不在拷内存），exec 的安全靠 O_CLOEXEC，D 态的不可杀是内核对驱动契约的信守（TASK_KILLABLE 只在作者选了 `*_killable` 时给逃生门），僵尸不是泄漏而是父进程的失约。
