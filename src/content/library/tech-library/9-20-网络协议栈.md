> **一句话定位**：Linux 网络栈是一台被 67.2 纳秒鞭打着前进的机器——10GbE 线速小包下，每个包只允许你花约 200 个 CPU cycle，而一次 cache miss 就要烧掉 32ns。本章把 sk_buff、NAPI、TCP 快路径、netfilter/conntrack 的**真实源码（v6.12）**逐行拆开，并用 git 考古还原每一代设计是在跟什么失败搏斗。
>
> 本版本为深化版：所有标注【真实源码 v6.12】的代码块均逐字取自 `raw.githubusercontent.com/torvalds/linux/v6.12/`；史料均实际核对了 LWN 文章、USENIX 论文、netdev 邮件列表与 mainline commit message，出处随文标注。

## TL;DR — 本章最硬结论

1. **一切优化的标尺是"每包时间预算"**：10GbE 上 64B 小包线速 14.88 Mpps，意味着每包 **67.2ns ≈ 200 cycles@3GHz**；一次 cache miss ≈ 32ns，一条 x86 `LOCK` 前缀指令 ≈ 8.25ns，一次系统调用 ≥ 42ns（数据：Jesper Brouer，LCA 2015，LWN 629155）。通用内核栈单核转发 1–2 Mpps，bypass 方案宣称 ~15 Mpps——差距不是常数优化能弥补的，是**每包固定成本（skb 分配、原子操作、锁、间接调用）的总和**超出了预算。
2. **NAPI 不是"优化"，是把系统从崩溃边缘拉回来的稳定性修复**。2001 年的实验数据：Pentium II 上 Linux 2.3.99 在约 60 Kpps 输入时进入 congestion collapse——包全进来了，一个都转发不出去（"Beyond Softnet"，Salim/Olsson/Kuznetsov，USENIX ALS'01）。NAPI 的本质是**把"每包一个中断"变成"忙时关中断轮询 + 配额公平调度"**。
3. **budget 是两级的**：每个 NAPI 实例每次 poll 上限 `weight`（几乎总是 64，`NAPI_POLL_WEIGHT`，netdevice.h L2620）；一轮 `NET_RX_SOFTIRQ` 全局上限 `netdev_budget=300` 且不超过 `netdev_budget_usecs`（默认 2 jiffies，HZ=1000 时 2000μs）——两个先到先停（net/core/dev.c `net_rx_action`，net/core/hotdata.c）。
4. **sk_buff 是"元数据与数据分离 + 指针移动代替 memcpy"的教科书**：`skb_push` 的全部实现是两行——`skb->data -= len; skb->len += len;`。整条 TCP→IP→以太网封包路径没有一次数据搬移。
5. **TCP 快路径是一次 32-bit 整数比较同时验证三件事**：`pred_flags` 把"头长无变化 + 仅 ACK 标志 + 窗口无变化"编码进一个字，源于 Van Jacobson 的"30 instruction TCP receive"构想——这段注释在 v6.12 的 tcp_input.c 里仍逐字保留着。
6. **缓存不是免费的——key space 可被攻击者污染的缓存本身就是漏洞**。IPv4 route cache 在 3.6（2012）被整个删除，David Miller 的 commit 原话："The ipv4 routing cache is non-deterministic, performance wise, and is subject to reasonably easy to launch denial of service attacks."（commit 89aef8921bfb）。删除的前提是先把慢路径（FIB trie）做快。
7. **conntrack 经历了三次大手术**：全局自旋锁 → per-bucket 锁（v3.15，commit 93bb0ceb75be，删锁前 perf 显示 ksoftirqd **72.56% 时间花在 `_raw_spin_lock_bh`**）；每条目一个 timer → 惰性回收 + 后台 gc worker（v4.9，commit b87a2f9199ea）；hash 函数换 siphash 抗 hash-flooding。每一刀都对应一个真实生产灾难。
8. **netfilter hook 框架的成本被一路压低但没有归零**：hook 存储从链表改为连续数组（v4.14，commit 960632ece694，commit message 自带 netperf 数据 353.9→364.2）；没有任何规则时由 static key 直接跳过。但只要 conntrack 加载，每包两次 tuple hash 查找躲不掉。
9. **被删除的代码和被合并的代码一样重要**：TCP prequeue（VJ 设想的"在进程上下文做协议处理"）在 4.13 被整体删除，理由是 epoll 时代"进程恰好阻塞在 recv 上"的前提已不成立（commit e7942d0633c4）。读懂为什么删，比记住它曾怎么工作更有价值。
10. **旁路不是背叛而是分层**：XDP（4.8，最初 RFC 讨论于 LWN 682538）在 skb 分配之前给你一次 drop/redirect 的机会（实验数据 20 Mpps drop vs tc+cls_bpf 6.5 Mpps）；MSG_ZEROCOPY（LWN 726917）只对大包划算（microbenchmark +39%，生产负载 +5–8%）。每种旁路都在出卖通用性换确定性。

**前置依赖章**：第 11 章（softirq/workqueue 并发模型）、第 18 章（中断与下半部）。第 17 章（io_uring）、第 21 章（XDP/eBPF）、第 22 章（容器网络）会反复被引用。

**如何读**：20.1–20.2 建立"为什么"（时间预算 + 历史辩论）；20.3–20.6 是源码精读主体；20.8 用三个场景把方案选择推演一遍；赶时间可以先读 20.2.6 的时间线总表和 20.8，再回头补源码。

## 20.1 第一性问题：每包 67.2ns 的时间预算

### 20.1.1 为什么网络栈必须是内核问题

用户程序只想要 `send()`/`recv()` 的字节流抽象，但硬件给的是：以太网帧、IP 分片、TCP 乱序、可能无效的 checksum offload。翻译工作落在内核，因为三个不可外包的职责：

1. **安全边界**：NIC 通过 DMA 直写内存。让用户态直接碰 DMA ring 意味着让它读写任意物理内存（除非配 IOMMU + VFIO，这正是 DPDK 的代价——独占设备、绕过一切内核策略）。
2. **多路复用仲裁**：一块网卡服务全机进程。谁的包是谁的，必须有一个所有人都信任的仲裁者。
3. **协议状态的生存期管理**：TCP 连接状态、重传定时器、拥塞窗口的生存期跟进程生存期解耦（进程崩了，FIN/RST 还得发）。

于是架构被钉死为：**NIC → DMA ring → hardirq → softirq(NAPI) → 协议层 → socket queue → copy_to_user**。本章逐段拆这条路径。

### 20.1.2 时间预算：难优化的量化根因

"网络栈难优化"不是一句感叹，是一张算得出来的账单。以下数据来自 Jesper Brouer 在 LCA 2015 内核 miniconf 的报告（Jonathan Corbet 整理，**LWN 629155**，2015-01-13，已核实原文）：

| 项目 | 数值 | 含义 |
|---|---|---|
| 10GbE，1538B 大包 | 每包间隔 1230ns | 大包场景相对从容 |
| 10GbE，64B 最小帧线速 | **每包 67.2ns**（14.88 Mpps） | 小包场景的死线 |
| 40GbE | 每包 307ns（1538B） | 网卡升级，预算反而紧 |
| 100GbE | ~120ns（约 8.15Mpps，1538B） | 摩尔定律救不了你 |
| 3GHz CPU 在 67.2ns 内 | **约 200 cycles** | 这就是全部家当 |
| 一次 cache miss | **~32ns** | 两次 miss 预算去半 |
| 一条 `LOCK` 前缀原子指令 | ~8.25ns | refcount/原子统计都是它 |
| spinlock lock+unlock 一对 | >16ns | 还没算 contention |
| 一次系统调用 | 42ns（裸）/ >75ns（开 SELinux+audit） | `send()` 的入场费 |
| qdisc 发送路径锁开销 | ~48ns/包（6 次 LOCK 操作） | 即使队列是空的 |
| kmem_cache alloc+free 一对 | ~19ns；真实栈使用模式 ~77ns | skb 元数据的代价 |
| 内核单核转发能力 | 1–2 Mpps | 对照组 |
| bypass 方案（DPDK 等）单核 | ~15 Mpps | 差距 ≈ 10 倍 |

把表里的成本往 67.2ns 里塞一下就明白了：**一次 skb 分配 + 一次 spinlock + 两次 cache miss = 预算耗尽**，协议处理还没开始。所以 bypass 方案的全部秘密就四个词：**批处理（amortize）、预分配、CPU 亲和、无锁**。而内核栈背着通用性（任意协议、任意 socket 数、安全隔离、可观测性）跑同一条赛道。

Brouer 那场报告还有一个刺耳的观察（原文转述）：内核长期强调"多核横向扩展"，恰好**掩盖了单核效率的持续退化**——加核能救吞吐，救不了单流延迟。

### 20.1.3 排队论视角：receive livelock 是服务率坍塌

把收包路径建模成两级串联队列：

```
到达 λ ──▶ [Stage1: hardirq 取包入队] ──▶ backlog queue ──▶ [Stage2: 协议处理] ──▶ 离开
              服务率 μ1                                       服务率 μ2
```

中断驱动模型的致命缺陷：**Stage1 的优先级严格高于 Stage2**（hardirq 抢占一切）。当 λ 升高，每个到达事件都打断 Stage2；λ → 某阈值时，CPU 100% 花在 Stage1（取包入队）+ 中断进出开销上，Stage2 的有效服务率 μ2 → 0。队列满，后续包在入队时被丢弃——**于是 Stage1 做的全部功都是无用功，goodput 归零**。这就是 receive livelock，1996 年由 Mogul & Ramakrishnan 在论文 *Eliminating Receive Livelock in an Interrupt-Driven Kernel* 中命名（"Beyond Softnet" 论文引用其为 [JMKKR96]）。

关键洞见：**这不是吞吐不足问题，是调度优先级倒挂问题**——最高优先级的工作（响应中断）服务于一个已经没有能力消费的下游。解法必然是某种形式的"到达速率高时，降低到达事件的处理优先级"——也就是关中断、改轮询、给消费端让出 CPU。NAPI、io_uring 的 sqpoll、DPDK 的 PMD 全是这一个洞见的变体。

### 20.1.4 五个对立约束（保留并量化）

| 约束 | 与之冲突 | 量化锚点 |
|---|---|---|
| 高吞吐（批处理摊薄）| 低延迟（每包尽快交付）| GRO 攒包 vs `gro_flush_timeout` |
| 多队列并行（RSS 分核）| 同一 flow 必须有序 | 单 TCP 流天然单核上限 |
| 零拷贝 | 安全隔离 | MSG_ZEROCOPY 要 pin 页 + 完成通知 |
| 通用协议栈 | 专用快路径 | conntrack 两次 hash/包逃不掉 |
| 可扩展 hook | 极致性能 | 空 hook 数组化后仍非零成本 |

没有设计能全赢，所以 XDP/DPDK/io_uring 各自在不同维度做了不同的"出卖"。20.8 节用场景推演这些取舍。

## 20.2 设计考古：五场战役

> 本节是"git 历史 + 社区讨论"的专门小节。每场战役给出：谁、何时、为解决什么失败、争论过什么、出处。所有 commit message 引文均通过 GitHub API 核对原文；LWN 文章均实际抓取核对。

### 20.2.1 战役一：softnet 之死与 NAPI 的诞生（1996–2001–2007–2021）

**前史（2.2/2.4 早期，softnet 模型）**：每包一个硬中断，驱动在中断里把包塞进 **per-CPU 共享 backlog 队列**，稍后内核线程（`NET_RX_SOFTIRQ` 前身）从 backlog 取包调 IP 协议处理。这个模型在 "Beyond Softnet" 论文（Jamal Hadi Salim/Znyx Networks、Robert Olsson/Uppsala University、Alexey Kuznetsov/Swsoft-INR，**USENIX 5th Annual Linux Showcase & Conference, Oakland, 2001**；论文 PDF 已取回并全文核对）里被系统性地处刑，论文给出的实测失败模式：

- **崩溃点量化**：Pentium II 上 Linux 2.3.99 转发，输入约 **60 Kpps 即 congestion collapse**——"60K packets per second were coming into the system but none were going out"；最大无损转发率（MLFFR）仅约 **27 Kpps**，且 CPU 100% 被网络占据，用户态全部饿死。
- **公平性失败**：任何一块"贪婪"网卡（收得快）会通过共享 backlog 队列饿死其他网卡。
- **已有补丁救不了**：硬件流控（HFC，2.1 时代就有）靠"backlog 满时关中断"缓解，但论文指出当时只有 DEC 21143（tulip）这类芯片支持中断减速，"Because this is not a general solution, it becomes difficult to widely deploy"；且 NIC 数量增加后无论如何 mitigate 都会塌。

**NAPI 的设计（合入 2.5/2.6 开发线）**：第一个包仍用中断唤醒；随即**关掉该设备的 RX 中断**，把设备挂上轮询工作队列；softirq 轮询各设备，每设备一次最多取 **quota** 个包，quota 用完且还有包 → 排到队尾（公平），ring 清空 → 摘下工作队列、重开中断。论文原文确认了两个常被忽略的细节：

1. quota 概念"is inherited from the max_interrupt_work concept"——继承自老驱动参数，不是凭空发明；
2. 实测不仅吞吐改善（100Mbps/148Kpps 注入下 NAPI 的 MLFFR 达到 65% 输入水平），**系统响应性**也改善：同样负载下 `ls` 列目录，softnet 版本比 NAPI 版本慢 3–4 倍。
3. 设计目标里明确写着 "cure packet reordering that is inherent with SMP support"——老 backlog 模型在 SMP 下同一设备的包可能被不同 CPU 乱序处理，NAPI 把"一个设备（队列）同一时刻只被一个 CPU 轮询"变成不变量。这个不变量今天仍是理解 RSS/RPS 的钥匙。

**第二幕（2.6.24，2007）**：多队列网卡出现，"每个 net_device 一个 poll 函数"的假设崩了。Stephen Hemminger 的 commit **bea3348eef27**（2007-10-03，已核对原文）把 NAPI 从 `struct net_device` 中拆出独立的 `struct napi_struct`："Several devices have multiple independant RX queues per net device... it's easier to support layouts like that if the structure representing the poll is independant from the net device itself."。poll 回调签名从 `foo_poll(struct net_device *dev, int *budget)` 变为 `foo_poll(struct napi_struct *napi, int budget)`。从此"每个 RX 队列一个 NAPI 实例"成为标准形态，RSS 多核并行才有了载体。

**第三幕（5.12，2021）：threaded NAPI**。softirq 有个老毛病：它借用被打断进程的上下文执行，CPU 记账算到无辜进程头上，且调度器无法管理它。Google 的 Wei Wang（与 Paolo Abeni、Hannes Frederic Sowa、Jakub Kicinski 共同署名）的 commit **29863d41bb6e**（2021-02-08，已核对原文）允许每个 NAPI 实例跑在专属内核线程里："Once that threaded mode is enabled and the kthread is started, napi_schedule() will wake-up such thread instead of scheduling the softirq."——把网络轮询变成普通线程，调度器可以给它设优先级、绑核、跟业务隔离。这是 20.8 场景 C 的主角之一。

**贯穿线**：从 1996 论文到 2021 threaded NAPI，二十五年都在回答同一个问题——**网络处理该以什么身份占用 CPU**。中断（霸占）→ softirq（半霸占，借用户进程外衣）→ kthread（被调度管理）→ busy-poll（用户线程自带 CPU 来换延迟）。

### 20.2.2 战役二：IPv4 route cache 之死（1990s–2012）

早期 IPv4 路由查找是 FIB hash + 每 flow 一条 `dst_entry` 的 route cache。设计假设：流量有局部性，缓存命中率高。

两个假设先后崩塌：

1. **攻击者可以控制 key space**：随机伪造 src IP 的 DoS 流量让每个包都 miss 并插入新缓存项——缓存变成攻击放大器（GC 压力、锁、内存）。
2. **真实大站点的命中率也不行**：commit 原文提到 "Even  well behaved sites with great traffic patterns" 命中率也仅约 10% 量级（GitHub commit 页核实转述）。

修复分两步走，顺序很重要：

- **第一步（2.6.39，2011）**：把 LC-trie（FIB trie，Robert Olsson 实现）扶正为唯一实现，删除 fib_hash。David Miller 的 commit **3630b7c050d9**（2011-02-01，已核对原文）："The time has finally come to remove the hash based routing table implementation in ipv4. FIB Trie is mature, well tested, and I've done an audit of it's code..."——**先把无缓存的慢路径做快，才有资格删缓存**。
- **第二步（3.6，2012）**：David Miller 的 commit **89aef8921bfb**（patch 邮件 2012-07-01 发于 netdev，lists.openwall.net 存档已核实；commit 删除 933 行）正式删除 route cache，commit message 第一句即判决书：**"The ipv4 routing cache is non-deterministic, performance wise, and is subject to reasonably easy to launch denial of service attacks."** 后文还有一句值得抄在墙上的："the world was a much friendlier place when the tradeoffs that led to the routing cache's design were considered."

**教训（可迁移）**：当缓存的 key 可被不可信方控制时，缓存就是 DoS 面。修缓存问题的正确姿势经常不是"修缓存"，而是"把 miss 路径做到不需要缓存"。今天每个包都直接查 FIB trie 做最长前缀匹配——确定性的中等成本，优于不确定性的"通常很快偶尔灾难"。

### 20.2.3 战役三：netfilter 三十年——从 ipfwadm 到 nftables，conntrack 三次手术

**朝代更替**（netfilter.org/about 官方史，已核实）：

| 代际 | 内核 | 主导者 | 取代理由 |
|---|---|---|---|
| ipfwadm | 2.0 | （承袭 BSD ipfw 思路） | — |
| ipchains | 2.2 | Rusty Russell | 规则模型太弱 |
| netfilter/iptables | 2.4（开发于 2.3.x） | Rusty Russell；1999-11 与 Marc Boucher 在 Montreal 会议构思 multiple tables 框架 | "re-design and heavily improve the previous ipchains and ipfwadm" |
| nftables | 3.13（2014-01） | Patrick McHardy（2009 公布）；现任项目负责人 Pablo Neira Ayuso（2013 起） | 见下 |

**nftables 为什么必须发生**（LWN **324989**，Corbet，2009-03-24，已核实）：iptables 的扩展模型把协议知识硬编码进内核 ABI，结果是 ip/ip6/arp/ebtables 四套并行实现，"the kernel contains some 10,000 lines of duplicated filtering code"；加新匹配要 copy 整套代码"bang it into a new, protocol-specific shape"；规则更新是整表替换。nftables 的解法激进而优雅：**内核里只放一个极小的字节码虚拟机**（load 包字节到寄存器、比较、跳转），所有协议知识上移到用户态编译器（`nft` 把 `ip daddr 192.168.0.1` 编译成字节码）。这换来：单一引擎覆盖全协议族、规则事务化（原子替换）、用 set/map 把 O(n) 线性链变成哈希/区间查找。LWN 当时就预言了迁移的痛苦："This could be a disruptive and expensive transition"——事实确实如此，iptables-nft 兼容层至今仍在垫着。

**hook 框架自身的性能手术（4.14，2017）**：早期每个 hook 点是 `list_head` 链表，每包遍历 = 指针追逐 = cache miss 链。Aaron Conole 的 commit **960632ece694**（2017-08-23，已核对原文）"netfilter: convert hook list to an array"——hook entries 改为**连续内存数组**，并把注册期才用的 ops 指针挪到数组尾部，热路径只摸函数指针+priv。commit message 自带实验数据：ixgbe 10G、netperf UDP_STREAM 64B、5 个空 hook：**353.9 → 364.2**（Mbps）。配合 per-hook 的 static key（无 hook 时编译期跳过），"空框架税"被压到接近零——但 conntrack 一旦加载就不是空框架了。

**conntrack 三次手术**（这是生产环境真实疼痛的编年史）：

1. **全局锁 → per-bucket 锁（v3.15，2014）**：Jesper Dangaard Brouer 的 commit **93bb0ceb75be**（2014-03-03，已核对原文）删除中心自旋锁 `nf_conntrack_lock`，commit message 直接贴了删锁前的 perf 记录——**ksoftirqd 72.56% 的时间在 `_raw_spin_lock_bh`**，其中 init_conntrack / nf_ct_delete_from_lists / __nf_conntrack_confirm / destroy_conntrack 四个调用方几乎均分。这是"8 核以上服务器一开 conntrack 吞吐腰斩"年代的直接证据。
2. **per-entry timer → 惰性回收 + gc worker（v4.9，2016）**：老设计每条 conntrack 一个 kernel timer，百万连接 = 百万 timer，插删本身成为开销大头。Florian Westphal 的 commit **b87a2f9199ea**（2016-08-25，已核对原文）改为：**过期判定推迟到 lookup 时顺手做**（packet path 自然回收），后台 gc worker 只兜底——"GC happens once every 5 seconds, but we only scan at most 1/64th of the table (and not more than 8k) buckets to avoid hogging cpu... We only need gc to make sure userspace (conntrack event listeners) eventually learn of the timeout, and for resource reclaim in case the system becomes idle."。注意这个设计哲学：**正常情况下回收成本由触碰它的包均摊，后台线程只为"没人触碰"的尸体负责**。（v6.12 的 gc_worker 已再演进为自适应扫描间隔，见 20.6 源码。）
3. **hash 抗污染**：v6.12 的 `hash_conntrack_raw()` 用 **siphash + boot 期随机 key**（源码见 20.6）。又是 route cache 的同款教训：tuple 由对端构造，可预测的 hash = 可制造的 collision 链 = DoS。（jhash→siphash 的具体切换版本，待核。）

### 20.2.4 战役四：TCP 收包路径——VJ 的幽灵与 prequeue 之死

**Van Jacobson 的"30 instruction TCP receive"**：上世纪 90 年代初 VJ 在邮件列表上提出著名构想——常规情况下 TCP 收包可以压缩到 30 条指令：用"header prediction"一次比较排除所有异常分支，数据直接进 socket 队列，checksum+copy 推迟到收包进程的上下文里做。Linux 实现了前半（header prediction 快路径，见 20.4.6 源码——v6.12 注释里仍写着 "The code loosely follows the one in the famous '30 instruction TCP receive' Van Jacobson mail"），并用 **prequeue** 机制尝试了后半。

**prequeue 之死（4.13，2017）是一堂"前提失效"课**：Florian Westphal 的 commit **e7942d0633c4**（2017-07-30，已核对原文）删除整个 prequeue："This only works if the socket being processed belongs to a process that is blocked in recv on that socket. In practice, this doesn't happen anymore that often because nowadays servers tend to use an event driven (epoll) model."——VJ 构想的成立前提是"进程正阻塞在这条 socket 的 recv 上"，epoll 时代一个线程看万条连接，这个前提死了，于是机制随之陪葬（实测只有 netperf 这种朴素 recv 程序受益 ~4%）。**架构启示：每个优化都绑定一个工作负载假设；假设变了要敢删代码。**

**拥塞控制的政权更迭**（默认算法：Reno → BIC（2.6.8 前后）→ CUBIC → BBR 可选）：

- **BIC → CUBIC（2.6.19，2006）**：Stephen Hemminger 的 commit **597811ec167f**（2006-09-25，已核对原文）："Change default congestion control used from BIC to the newer CUBIC which it the successor to BIC but has better properties over long delay links."（注意：常见说法"Linux 默认一直是 Reno 直到 CUBIC"不准确，中间隔了一代 BIC。）
- **BBR（4.9，2016）**：Neal Cardwell 的 commit **0f8782ea1497**（2016-09-20，已核对原文；LWN **701165** 同步报道）是少见的"commit message 像论文"：引用 **Kleinrock 1981** 的最优工作点理论（最大化带宽同时最小化时延），点名 loss-based 拥塞控制两宗罪——深 buffer 末端链路上造成 bufferbloat（"often causing seconds of needless queuing delay"），浅 buffer 高速长链路上对突发丢包过度反应导致吞吐崩塌；BBR 改为持续估计 bottleneck bandwidth 与 min RTT，**只需发送端改动**即可增量部署。LWN 报道同时记录了社区保留意见：BBR 与 loss-based 流共存时的公平性当时未充分验证（"works fine when only one side of the connection is using it"——多 BBR 流挤压 CUBIC 流的问题后来由 BBRv2/v3 持续修补）。

### 20.2.5 战役五：offload 与旁路——LRO→GRO，XDP，MSG_ZEROCOPY

- **LRO → GRO（2.6.29，2009）**：LRO（硬件把同流小包合并成大包）的问题是合并不可逆且丢信息——LWN **358910**（Corbet，2009-10-27，JLS2009 上 Herbert Xu 的报告，已核实）的原话是 LRO "merges everything in sight"，转发/桥接/虚拟化场景会把合并后的"怪包"再发出去，破坏正确性。Herbert Xu 的 GRO 改为**软件合并 + 严格合并判据**（MAC 头必须一致、TCP/IP 头只允许规定差异），保证**可无损还原**（GSO 反向切回去），从而转发路径也能用。GRO 是今天 40G+ 网卡跑满的前提之一，也是延迟敏感场景第一个要审视的旋钮。
- **XDP（4.8，2016）**：Brenden Blanco（PLUMgrid）的 commit **6a773a15a1e8**（2016-07-19，已核对原文）加入 `BPF_PROG_TYPE_XDP`（patch 期名为 PHYS_DEV）：在驱动收包、**skb 分配之前**运行 eBPF，上下文 `struct xdp_md` 只暴露包起止指针。LWN **682538**（2016-04-06，已核实）记录了 RFC 阶段的争论：(1) 早期版本仍构造了最小 skb，被质疑违背初衷；(2) 程序绑定内核数据结构会损害向硬件 offload 的可移植性。性能数据（v2 讨论）：**XDP drop 20 Mpps vs tc+cls_bpf 6.5 Mpps**。XDP 的战略意义在 TL;DR 第 10 条：不再试图让通用栈更快，而是允许"在为时未晚处提前决策"。
- **MSG_ZEROCOPY（4.14，2017）**：Willem de Bruijn 的工作（LWN **726917**，Corbet，2017-07-03，已核实）：`setsockopt(SOCK_ZEROCOPY)` + `send(..., MSG_ZEROCOPY)`，内核 pin 住用户页直接发，完成通知走 socket error queue。关键数字：microbenchmark **+39%**，"more realistic production workload" **+5–8%**——以及内核保留"小包或需变换数据（IPsec、缺硬件 csum）时静默退回拷贝"的权利。零拷贝的会计学：省 copy 的收益必须 > pin 页 + 通知的固定成本，所以只对大块传输划算。

## 20.3 sk_buff 源码精读

### 20.3.1 官方几何注释——内核自己画的图

【真实源码 v6.12 include/linux/skbuff.h L716-L754】

```c
/**
 * DOC: Basic sk_buff geometry
 *
 * struct sk_buff itself is a metadata structure and does not hold any packet
 * data. All the data is held in associated buffers.
 *
 * &sk_buff.head points to the main "head" buffer. The head buffer is divided
 * into two parts:
 *
 *  - data buffer, containing headers and sometimes payload;
 *    this is the part of the skb operated on by the common helpers
 *    such as skb_put() or skb_pull();
 *  - shared info (struct skb_shared_info) which holds an array of pointers
 *    to read-only data in the (page, offset, length) format.
 *
 * Optionally &skb_shared_info.frag_list may point to another skb.
 *
 * Basic diagram may look like this::
 *
 *                                  ---------------
 *                                 | sk_buff       |
 *                                  ---------------
 *     ,---------------------------  + head
 *    /          ,-----------------  + data
 *   /          /      ,-----------  + tail
 *  |          |      |            , + end
 *  |          |      |           |
 *  v          v      v           v
 *   -----------------------------------------------
 *  | headroom | data |  tailroom | skb_shared_info |
 *   -----------------------------------------------
 *                                 + [page frag]
 *                                 + [page frag]
 *                                 + [page frag]
 *                                 + [page frag]       ---------
 *                                 + frag_list    --> | sk_buff |
 *                                                     ---------
 */
```

逐条拆：

- 第一句就是全部设计哲学：**sk_buff 本体是元数据，不持有数据**。元数据在 slab（`skbuff_head_cache`）分配，数据 buffer 单独分配。这个分离让"克隆一个包"（多播、tcpdump tap、TCP 重传队列）只需复制元数据并对数据加引用计数。
- 数据区分两段：**线性区**（head~end，`skb_put/skb_pull` 操作的对象）和 **frags**（`skb_shared_info` 里的 (page, offset, length) 数组，只读）。`sendfile()`/`MSG_ZEROCOPY` 的页直接挂 frags，这就是发送零拷贝的载体。
- `frag_list` 还能链下一个 skb——IP 分片重组、GRO 合并用它。于是一个"包"可能是：线性区 + 至多 `MAX_SKB_FRAGS` 个页片段 + 一串子 skb 的三层复合体。

**逐字段精读（为什么长成这样）**：

1. **开头的四选一 union（next/prev+dev | rbnode | list | ll_node）**：同一个 skb 在不同生命周期阶段属于不同容器——驱动队列用双链表、TCP 乱序重组用红黑树（`rbnode`，注释直说 "used in netem, ip4 defrag, and tcp stack"）、GRO/list 化收包用 `list_head`、socket 延迟释放用 llist。**union 而不是四个字段**，因为任一时刻只属于一个容器——这是用类型不安全换 cache footprint 的典型内核交易。乱序队列用红黑树本身就是一段考古：早年是链表，被"攻击者发送海量乱序小段使内核 O(n) 扫链表"的 DoS 打出来的改造。
2. **`cb[48]`**：48 字节"便签"，注释写明所有权规则——"owned by whoever has the skb queued ATM（at the moment）"。TCP 强转为 `TCP_SKB_CB(skb)`（seq/end_seq/tcp_flags…），IP 用 `IPCB`。**为什么不是各协议往 sk_buff 加字段？** 因为 sk_buff 是全内核最高频分配的结构之一，每加 8 字节，全机内存与 cache miss 率一起涨。48 字节是"够 TCP 用"的历史均衡点；越界即 BUILD_BUG_ON。**坑**：跨层保留 cb 数据必须先 clone——上一层可能随时覆写。
3. **`_nfct`（unsigned long）**：conntrack 指针 + 低 3 位塞 ctinfo 状态——指针低位复用的惯用技巧（`struct nf_conn` 按 8 对齐，低 3 位恒 0）。注意 conntrack 状态**不在 cb[] 里**（cb 会被下游覆写，而 conntrack 状态要活到 POSTROUTING）。
4. **`len` vs `data_len`**：`len` = 总数据长（线性 + frags），`data_len` = 仅 frags 部分。所以**线性区长度 = len - data_len**（`skb_headlen()`）。直接 `tail - data` 读包长是经典 bug——v6.12 里 `__skb_pull` 在 `len < data_len` 时直接 `BUG()`。
5. **`headers` struct_group（L950 起到 L1070）**：把"转发/克隆时需要整体复制的头部元数据字段"框进一个命名组，`__copy_skb_header()` 用**单次 memcpy** 复制整组——字段排布服务于复制路径的指令数，这是结构体布局即性能的实例。
6. **结尾的 `tail/end/head/data/truesize/users` 固定在最后**（注释点名 see alloc_skb()）：分配时元数据与这些"几何指针"一次初始化；`tail/end` 的类型 `sk_buff_data_t` 在 64 位机上是 `unsigned int` 偏移而非指针（`NET_SKBUFF_DATA_USES_OFFSET`，L700-707）——**用 4 字节偏移替代 8 字节指针省 cache**，又一个被预算逼出来的设计。
7. **`truesize`**：本 skb 占用的真实内存（元数据 + buffer 总量，含没用到的 tailroom）。socket 内存记账（`sk_rmem_alloc`/`sk_wmem_alloc`、`tcp_rmem` 限额）全按 truesize 算——**所以收到 64B 小包可能记账 1KB+**（驱动按 2KB 切 buffer 时更夸张）。生产现象：小包流量把 `tcp_rmem` 吃满但实际数据量很小，`ss -m` 看 `rb`/`rmem` 差异即此。
8. **`users`（refcount_t）**：引用计数管"谁还在用这个元数据头"；数据 buffer 的共享另有 `skb_shared_info.dataref` 管（见下节）。两级引用计数分别回答"skb 结构能不能 free"和"数据 buffer 能不能写/free"。

### 20.3.3 skb_shared_info 与 dataref 的上下两半

【真实源码 v6.12 include/linux/skbuff.h L590-L618】

```c

	__u8		flags;
	__u8		meta_len;
	__u8		nr_frags;
	__u8		tx_flags;

	/* Warning: this field is not always filled in (UFO)! */

	 */
```

- 它住在 `skb->end` 之后（数据 buffer 的尾巴上），所以 `skb_shinfo(skb)` 是一次指针运算——但也意味着**摸 shinfo 大概率是一次额外 cache miss**（数据区和元数据不在同一 cache line 邻域）。GRO/GSO 的元数据（gso_size/gso_segs/gso_type）全在这里：**协议栈上层处理 64KB 大段，切分推迟到最后一刻**——TSO 交硬件切，GSO 软件兜底，依据就是这三个字段。
- `dataref` 是数据 buffer 的引用计数，且被劈成两半用——紧随其后的官方注释（L620-L648）值得整段读：

【真实源码 v6.12 include/linux/skbuff.h L620-L648】

```c
/**
 * DOC: dataref and headerless skbs
 *
 * Transport layers send out clones of payload skbs they hold for
 * retransmissions. To allow lower layers of the stack to prepend their headers
 * we split &skb_shared_info.dataref into two halves.
 * The lower 16 bits count the overall number of references.
 * The higher 16 bits indicate how many of the references are payload-only.
 * skb_header_cloned() checks if skb is allowed to add / write the headers.
 *
 * The creator of the skb (e.g. TCP) marks its skb as &sk_buff.nohdr
 * (via __skb_header_release()). Any clone created from marked skb will get
 * &sk_buff.hdr_len populated with the available headroom.
 * If there's the only clone in existence it's able to modify the headroom
 * at will. The sequence of calls inside the transport layer is::
 *
 *  <alloc skb>
 *  skb_reserve()
 *  __skb_header_release()
 *  skb_clone()
 *  // send the clone down the stack
 *
 * This is not a very generic construct and it depends on the transport layers
 * doing the right thing. In practice there's usually only one payload-only skb.
 * Having multiple payload-only skbs with different lengths of hdr_len is not
 * possible. The payload-only skbs should never leave their owner.
 */
#define SKB_DATAREF_SHIFT 16
#define SKB_DATAREF_MASK ((1 << SKB_DATAREF_SHIFT) - 1)
```

这段解决的是一个非常具体的矛盾：**TCP 把发出的段留在重传队列里（持有 payload），下层（IP/以太网）却要在同一块 buffer 的 headroom 写头**。直接共享会互相踩。方案是把 dataref 的 32 位劈开：低 16 位 = 总引用数，高 16 位 = "只引用 payload 的"数量；`skb_header_cloned()` 据此判断"我能不能写 headroom"。注释最后一句的坦诚值得注意——"This is not a very generic construct and it depends on the transport layers doing the right thing"——**内核为了热路径性能接受了一个脆弱但够用的契约**，并把约束写成文档。这种"工程妥协 + 显式契约"是读内核代码时反复出现的味道。

### 20.3.4 指针搬运三兄弟：__skb_push / __skb_pull /（skb_put）

【真实源码 v6.12 include/linux/skbuff.h L2745-L2769】

```c
static inline void *__skb_push(struct sk_buff *skb, unsigned int len)
{
	DEBUG_NET_WARN_ON_ONCE(len > INT_MAX);

	skb->data -= len;
	skb->len  += len;
	return skb->data;
}

void *skb_pull(struct sk_buff *skb, unsigned int len);
static inline void *__skb_pull(struct sk_buff *skb, unsigned int len)
{
	DEBUG_NET_WARN_ON_ONCE(len > INT_MAX);

	skb->len -= len;
	if (unlikely(skb->len < skb->data_len)) {
#if defined(CONFIG_DEBUG_NET)
		skb->len += len;
		pr_err("__skb_pull(len=%u)\n", len);
		skb_dump(KERN_ERR, skb, false);
#endif
		BUG();
	}
	return skb->data += len;
}
```

- **封包 = `data` 指针左移**（push），**解包 = 右移**（pull），**追加数据 = `tail` 右移**（put）。发送方向 TCP 写完 payload，`__tcp_transmit_skb` 调 `skb_push(skb, tcp_header_size)` 在 headroom 写 TCP 头（见 20.5 真实源码），IP 层再 push IP 头，以太网层 push 以太头——**三层封装零 memcpy**，前提是分配时 `skb_reserve()` 预留了足够 headroom（驱动按 `dev->needed_headroom`+`NET_SKB_PAD` 预留）。
- `__skb_pull` 里那个 `BUG()`：pull 不允许越过线性区进入 frags（`len < data_len` 说明你把 frags 的字节当线性区 pull 了）。要安全访问可能在 frags 里的头部，必须 `pskb_may_pull(skb, n)`——**它会在需要时把 frags 数据线性化（搬进线性区）**，所以协议入口（`ip_rcv_core`、`tcp_v4_rcv`）每解析一层前都先 `pskb_may_pull`。忘了这步、直接按指针读头，是驱动/隧道代码里真实出现过的 crash 源。
- 无下划线版本（`skb_push/skb_pull`）是带边界检查的出口函数（在 net/core/skbuff.c），headroom 不足时 `skb_push` 会 panic——所以隧道封装代码必须先 `skb_cow_head()` 确保 headroom，这是写 encap 模块的第一戒律。

### 20.3.5 设计权衡清单（为什么不更简单）

| 设计 | 更简单的替代 | 为什么不 |
|---|---|---|
| 元数据/数据分离 + 两级 refcount | 一块连续内存装包+元数据 | clone（重传/多播/tap）就得整包拷贝；DMA 要求数据区物理连续而元数据不必 |
| headroom 预留 + 指针移动封包 | 每层 alloc 新 buffer 拼接 | 每层一次分配+拷贝，时间预算直接爆 |
| cb[48] 强转 | 各协议加专属字段 | sk_buff 尺寸失控；cache miss 即吞吐 |
| frags + frag_list 复合结构 | 强制线性化 | 零拷贝发送（页挂 frags）与 GRO 合并都依赖非线性表示 |
| tail/end 用 32 位偏移 | 用指针 | 64 位机上每 skb 省 8B×2 + 更密的 cache 排布 |

**失败模式备忘**（都来自真实排障）：
- 读包长用 `skb->len`，读线性区用 `skb_headlen()`，两者混用 = 隧道/DPI 模块隐性截断。
- `skb_clone` 后改 payload = 改了重传队列里的数据（共享 buffer），必须 `skb_copy`/`pskb_copy`。
- truesize 记账：自己造 skb 时乱设 truesize 会骗过 socket 内存限额，OOM 或假"缓冲区满"。
- shinfo 跨 cache line：极端小包场景，驱动选择 build_skb()+页池（page_pool，`pp_recycle` 位）就是为了把 shinfo 初始化成本和 cache 行为控制住。

🎯 **面试钩子**：让候选人解释"为什么 `skb_push` 不检查越界也能安全"（答：检查版入口 + 分配契约 headroom；`__` 前缀 = 调用者保证前置条件，内核命名惯例）。

## 20.4 RX 路径源码全跑：从硬中断到 socket 队列

> 本节按真实调用链排列 v6.12 源码：`____napi_schedule`（中断侧）→ `net_rx_action`（softirq 主循环）→ `__napi_poll`（单实例轮询）→ `napi_complete_done`（关轮询开中断的无锁谈判）→ `ip_rcv`（L3 入口）→ `tcp_v4_rcv`（L4 入口）→ `tcp_rcv_established`（快路径）。

### 20.4.1 中断侧：把 NAPI 挂上轮询队列

【真实源码 v6.12 net/core/dev.c L4511-L4545】

```c
static inline void ____napi_schedule(struct softnet_data *sd,
				     struct napi_struct *napi)
{
	struct task_struct *thread;

	lockdep_assert_irqs_disabled();

	if (test_bit(NAPI_STATE_THREADED, &napi->state)) {
		/* Paired with smp_mb__before_atomic() in
		 * napi_enable()/dev_set_threaded().
		 * Use READ_ONCE() to guarantee a complete
		 * read on napi->thread. Only call
		 * wake_up_process() when it's not NULL.
		 */
		thread = READ_ONCE(napi->thread);
		if (thread) {
			if (use_backlog_threads() && thread == raw_cpu_read(backlog_napi))
				goto use_local_napi;

			set_bit(NAPI_STATE_SCHED_THREADED, &napi->state);
			wake_up_process(thread);
			return;
		}
	}

use_local_napi:
	list_add_tail(&napi->poll_list, &sd->poll_list);
	WRITE_ONCE(napi->list_owner, smp_processor_id());
	/* If not called from net_rx_action()
	 * we have to raise NET_RX_SOFTIRQ.
	 */
	if (!sd->in_net_rx_action)
		__raise_softirq_irqoff(NET_RX_SOFTIRQ);
}
```

逐行：
- 驱动的 hardirq handler 里只做三件事：ack 中断、**写寄存器关掉本队列 RX 中断**、`napi_schedule()`（最终落到这里）。函数全程要求 irq 关闭（`lockdep_assert_irqs_disabled`）。
- **threaded 分支**（20.2.1 第三幕的运行时形态）：设了 `NAPI_STATE_THREADED` 就唤醒专属 kthread 而不是触发 softirq。注释明示了与 `napi_enable()` 的内存屏障配对——`napi->thread` 的发布/读取要防撕裂（READ_ONCE）与重排，这是 lock-free 发布模式的标准写法。
- softirq 分支：挂上 per-CPU `softnet_data.poll_list`，记录 `list_owner`。`in_net_rx_action` 检查是个微优化：如果当前就在 `net_rx_action` 里（轮询中又有新 NAPI 加入），不必重复 raise softirq——主循环退出前会自己检查。

### 20.4.2 net_rx_action：两级 budget 的主循环

【真实源码 v6.12 net/core/dev.c L6931-L6996】

```c
static __latent_entropy void net_rx_action(void)
{
	struct softnet_data *sd = this_cpu_ptr(&softnet_data);
	unsigned long time_limit = jiffies +
		usecs_to_jiffies(READ_ONCE(net_hotdata.netdev_budget_usecs));
	struct bpf_net_context __bpf_net_ctx, *bpf_net_ctx;
	int budget = READ_ONCE(net_hotdata.netdev_budget);
	LIST_HEAD(list);
	LIST_HEAD(repoll);

	bpf_net_ctx = bpf_net_ctx_set(&__bpf_net_ctx);
start:
	sd->in_net_rx_action = true;
	local_irq_disable();
	list_splice_init(&sd->poll_list, &list);
	local_irq_enable();

	for (;;) {
		struct napi_struct *n;

		skb_defer_free_flush(sd);

		if (list_empty(&list)) {
			if (list_empty(&repoll)) {
				sd->in_net_rx_action = false;
				barrier();
				/* We need to check if ____napi_schedule()
				 * had refilled poll_list while
				 * sd->in_net_rx_action was true.
				 */
				if (!list_empty(&sd->poll_list))
					goto start;
				if (!sd_has_rps_ipi_waiting(sd))
					goto end;
			}
			break;
		}

		n = list_first_entry(&list, struct napi_struct, poll_list);
		budget -= napi_poll(n, &repoll);

		/* If softirq window is exhausted then punt.
		 * Allow this to run for 2 jiffies since which will allow
		 * an average latency of 1.5/HZ.
		 */
		if (unlikely(budget <= 0 ||
			     time_after_eq(jiffies, time_limit))) {
			sd->time_squeeze++;
			break;
		}
	}

	local_irq_disable();

	list_splice_tail_init(&sd->poll_list, &list);
	list_splice_tail(&repoll, &list);
	list_splice(&list, &sd->poll_list);
	if (!list_empty(&sd->poll_list))
		__raise_softirq_irqoff(NET_RX_SOFTIRQ);
	else
		sd->in_net_rx_action = false;

	net_rps_action_and_irq_enable(sd);
end:
	bpf_net_ctx_clear(bpf_net_ctx);
}
```

精读：

1. **两级 budget 在此对账**：循环每次取一个 NAPI 实例调 `napi_poll`（其内部上限是该实例的 `weight`，几乎总是 64），返回的实际处理数从**全局 budget（300）**里扣；同时挂着**时间闸门** `netdev_budget_usecs`（默认 `2*USEC_PER_SEC/HZ`，HZ=1000 即 2000μs；注释解释了为什么至少 2 jiffies——保证有 1 jiffy 的真实超时窗口）。**包数或时间任一先到即收工**，未尽事宜重新 raise softirq（高压下由 ksoftirqd 接手）。`sd->time_squeeze++` 就是 `/proc/net/softnet_stat` 第三列——它持续增长 = 这颗 CPU 的 softirq 窗口塞不下流量，该扩队列/调 RSS 了。
2. **为什么 budget 不能调成 30000**：softirq 运行在关 BH 的上下文里，霸占 CPU 会推迟定时器、RCU 静止点和用户进程——"Allow this to run for 2 jiffies... average latency of 1.5/HZ" 这条注释就是公平性合同。1Mpps 单核 × 300 budget = 每轮 ~300μs，调大十倍就是 3ms 的调度黑洞。
3. **无锁交接的小芭蕾**：进入时把 `poll_list` splice 到栈上局部 list（关 irq 仅在 splice 一瞬），退出前再 splice 回去。`in_net_rx_action` + `barrier()` + 重查 `poll_list` 的舞步，防的是 ____napi_schedule 的"不重复 raise softirq"优化在窗口期丢事件——**典型的"优化引入竞态，再用屏障封口"**。
4. `skb_defer_free_flush`：每轮顺手释放其他 CPU 扔回来的延迟释放 skb（`sysctl_skb_defer_max=64`）——**让 free 发生在分配它的 CPU 上**，省跨核 cache 弹跳。这类"顺路干活"在热循环里随处可见。

### 20.4.3 __napi_poll 与 napi_complete_done：关中断/开中断的无锁谈判

【真实源码 v6.12 net/core/dev.c L6765-L6836】

```c
static int __napi_poll(struct napi_struct *n, bool *repoll)
{
	int work, weight;

	weight = n->weight;

	/* This NAPI_STATE_SCHED test is for avoiding a race
	 * with netpoll's poll_napi().  Only the entity which
	 * obtains the lock and sees NAPI_STATE_SCHED set will
	 * actually make the ->poll() call.  Therefore we avoid
	 * accidentally calling ->poll() when NAPI is not scheduled.
	 */
	work = 0;
	if (napi_is_scheduled(n)) {
		work = n->poll(n, weight);
		trace_napi_poll(n, work, weight);

		xdp_do_check_flushed(n);
	}

	if (unlikely(work > weight))
		netdev_err_once(n->dev, "NAPI poll function %pS returned %d, exceeding its budget of %d.\n",
				n->poll, work, weight);

	if (likely(work < weight))
		return work;

	/* Drivers must not modify the NAPI state if they
	 * consume the entire weight.  In such cases this code
	 * still "owns" the NAPI instance and therefore can
	 * move the instance around on the list at-will.
	 */
	if (unlikely(napi_disable_pending(n))) {
		napi_complete(n);
		return work;
	}

	/* The NAPI context has more processing work, but busy-polling
	 * is preferred. Exit early.
	 */
	if (napi_prefer_busy_poll(n)) {
		if (napi_complete_done(n, work)) {
			/* If timeout is not set, we need to make sure
			 * that the NAPI is re-scheduled.
			 */
			napi_schedule(n);
		}
		return work;
	}

	if (n->gro_bitmask) {
		/* flush too old packets
		 * If HZ < 1000, flush all packets.
		 */
		napi_gro_flush(n, HZ >= 1000);
	}

	gro_normal_list(n);

	/* Some drivers may have called napi_schedule
	 * prior to exhausting their budget.
	 */
	if (unlikely(!list_empty(&n->poll_list))) {
		pr_warn_once("%s: Budget exhausted after napi rescheduled\n",
			     n->dev ? n->dev->name : "backlog");
		return work;
	}

	*repoll = true;

	return work;
}
```

- **契约**：驱动 poll 返回实际处理数 `work`。`work < weight` ⇒ ring 已清空，驱动**必须已自行调用 `napi_complete_done` 并重开中断**；`work == weight` ⇒ ring 可能还有货，驱动**不得碰 NAPI 状态**，框架把它挂回 repoll 链表下轮继续。官方 NAPI 文档（docs.kernel.org/networking/napi.html，已核实）特别提醒一个阴险角落："恰好处理完且恰好用满 budget"无法上报，驱动要么不调 complete 等下轮空转一次，要么返回 `budget-1`。无数驱动 bug 死在这个边界上。
- **GRO 的 flush 时机**也在这：实例还要继续轮（用满 weight）时，把"太老的包"flush 上去（`HZ>=1000` 时只 flush 超龄的，否则全 flush）——GRO 攒包的延迟上界由此而来，低延迟调优盯的 `gro_flush_timeout` 在下一个函数登场。

【真实源码 v6.12 net/core/dev.c L6220-L6290】

```c
bool napi_complete_done(struct napi_struct *n, int work_done)
{
	unsigned long flags, val, new, timeout = 0;
	bool ret = true;

	/*
	 * 1) Don't let napi dequeue from the cpu poll list
	 *    just in case its running on a different cpu.
	 * 2) If we are busy polling, do nothing here, we have
	 *    the guarantee we will be called later.
	 */
	if (unlikely(n->state & (NAPIF_STATE_NPSVC |
				 NAPIF_STATE_IN_BUSY_POLL)))
		return false;

	if (work_done) {
		if (n->gro_bitmask)
			timeout = READ_ONCE(n->dev->gro_flush_timeout);
		n->defer_hard_irqs_count = READ_ONCE(n->dev->napi_defer_hard_irqs);
	}
	if (n->defer_hard_irqs_count > 0) {
		n->defer_hard_irqs_count--;
		timeout = READ_ONCE(n->dev->gro_flush_timeout);
		if (timeout)
			ret = false;
	}
	if (n->gro_bitmask) {
		/* When the NAPI instance uses a timeout and keeps postponing
		 * it, we need to bound somehow the time packets are kept in
		 * the GRO layer
		 */
		napi_gro_flush(n, !!timeout);
	}

	gro_normal_list(n);

	if (unlikely(!list_empty(&n->poll_list))) {
		/* If n->poll_list is not empty, we need to mask irqs */
		local_irq_save(flags);
		list_del_init(&n->poll_list);
		local_irq_restore(flags);
	}
	WRITE_ONCE(n->list_owner, -1);

	val = READ_ONCE(n->state);
	do {
		WARN_ON_ONCE(!(val & NAPIF_STATE_SCHED));

		new = val & ~(NAPIF_STATE_MISSED | NAPIF_STATE_SCHED |
			      NAPIF_STATE_SCHED_THREADED |
			      NAPIF_STATE_PREFER_BUSY_POLL);

		/* If STATE_MISSED was set, leave STATE_SCHED set,
		 * because we will call napi->poll() one more time.
		 * This C code was suggested by Alexander Duyck to help gcc.
		 */
		new |= (val & NAPIF_STATE_MISSED) / NAPIF_STATE_MISSED *
						    NAPIF_STATE_SCHED;
	} while (!try_cmpxchg(&n->state, &val, new));

	if (unlikely(val & NAPIF_STATE_MISSED)) {
		__napi_schedule(n);
		return false;
	}

	if (timeout)
		hrtimer_start(&n->timer, ns_to_ktime(timeout),
			      HRTIMER_MODE_REL_PINNED);
	return ret;
}
```

这是全章并发密度最高的 60 行，三件事同时发生：

1. **软件中断合并（IRQ suppression）**：`napi_defer_hard_irqs` + `gro_flush_timeout` 这对旋钮——本轮有活干（`work_done`）就续上 defer 计数；计数未归零时**不重开硬中断**，改为 hrtimer 在 `gro_flush_timeout` ns 后再来轮一次。效果：高 PPS 下中断率被钉在 `1/gro_flush_timeout`，省掉海量中断进出（每次都是上千 cycle）。这就是"用 0 硬件依赖实现自适应 interrupt coalescing"，官方文档推荐的现代默认调优手段。
2. **MISSED 状态机防丢唤醒**：经典竞态——poll 判断 ring 空 → 准备清 SCHED 重开中断；同一瞬间新包到达，hardirq 跑 `napi_schedule_prep` 发现 SCHED 还在，只置 MISSED 位返回（不重复入队）。若此处直接清 SCHED，这个包就睡死了。所以用 `try_cmpxchg` 循环：**看到 MISSED ⇒ 保留 SCHED 并立即 `__napi_schedule` 再轮一次**。那行怪异的 `(val & MISSED) / MISSED * SCHED` 是无分支位运算（注释注明 Alexander Duyck 为帮 gcc 生成更好代码而建议）。这是"状态位 + cmpxchg 重试环"解决 lost-wakeup 的标准范式，与第 11 章的 waitqueue 内存序问题同构。
3. **离场清理**：摘 poll_list（需短暂关 irq）、`list_owner=-1`、必要时启动 hrtimer。

**与官方文档的互证**（docs.kernel.org/networking/napi.html，已核实）：busy polling 状态下本函数直接 return false（"we have the guarantee we will be called later"）——busy poll 的本质是用户线程通过 `SO_BUSY_POLL`/epoll busy poll（`EPIOCSPARAMS` ioctl，参数 busy_poll_usecs/budget/prefer_busy_poll）**自带 CPU 来替 softirq 干活**，所以谁也不许把中断打开。`napi_prefer_busy_poll` 在 `__napi_poll` 里的提前退场同理。

```c
	 *	RFC1122: 3.2.1.2 MUST silently discard any IP frame that fails the checksum.
	 *
	 *	Is the datagram acceptable?
	 *
	 *	1.	Length at least the size of an ip header
	 *	2.	Version of 4
	 *	3.	Checksums correctly. [Speed optimisation for later, skip loopback checksums]
	 *	4.	Doesn't have a bogus length
	 */

	if (iph->ihl < 5 || iph->version != 4)
		goto inhdr_error;
	...
	if (!pskb_may_pull(skb, iph->ihl*4))
		goto inhdr_error;

	iph = ip_hdr(skb);

	if (unlikely(ip_fast_csum((u8 *)iph, iph->ihl)))
		goto csum_error;

	len = iph_totlen(skb, iph);
	if (skb->len < len) {
		drop_reason = SKB_DROP_REASON_PKT_TOO_SMALL;
		__IP_INC_STATS(net, IPSTATS_MIB_INTRUNCATEDPKTS);
		goto drop;
	} else if (len < (iph->ihl*4))
		goto inhdr_error;

	/* Our transport medium may have padded the buffer out. Now we know it
	 * is IP we can trim to the true length of the frame.
	 * Note this now means skb->len holds ntohs(iph->tot_len).
	 */
	if (pskb_trim_rcsum(skb, len)) {
		__IP_INC_STATS(net, IPSTATS_MIB_INDISCARDS);
		goto drop;
	}

	iph = ip_hdr(skb);
	skb->transport_header = skb->network_header + iph->ihl*4;

	/* Remove any debris in the socket control block */
	memset(IPCB(skb), 0, sizeof(struct inet_skb_parm));
	IPCB(skb)->iif = skb->skb_iif;

	/* Must drop socket now because of tproxy. */
	if (!skb_sk_is_prefetched(skb))
		skb_orphan(skb);

	return skb;
	...
}
```

读点：
- **每次 `pskb_may_pull` 之后必须重取 `iph`**——pull 可能触发线性化重分配，旧指针失效。源码里 `iph = ip_hdr(skb)` 出现了三次，不是冗余，是教科书。
- `pskb_trim_rcsum(skb, len)`：以太网最小帧 60B，短 IP 包会被 NIC 填充（padding），必须按 `tot_len` 裁掉，否则 padding 字节会被上层当 payload——历史上真实出过"TCP 流尾部多了垃圾字节"的 bug 类别。
- `skb->transport_header = network_header + ihl*4`：L4 头偏移在这里定格，之后 `tcp_hdr(skb)` 才有意义。**分层在数据上的体现就是这几个偏移字段，不是什么对象间消息传递。**
- `memset(IPCB(skb), 0, ...)`：进入 IP 层先把 cb[] 擦干净——上一层（驱动/桥/隧道）留下的"残渣"必须清除。cb 所有权规则的运行时体现。
- 每条 drop 路径都带 `SKB_DROP_REASON_*`：这是近年（5.17+ 起大规模铺开）的 drop reason 体系，`perf record -e skb:kfree_skb` 能直接看到丢包原因枚举——排障利器（20.9）。
- 出口的 `NF_HOOK(PRE_ROUTING)`：netfilter 第一税卡。无规则时 static key 短路（见 20.6），有 conntrack 时 `nf_conntrack_in` 在此执行。

### 20.4.6 tcp_rcv_established：pred_flags 位戏法

先看内核自述的快路径禁用条件（这段注释本身就是规格说明书）：

【真实源码 v6.12 net/ipv4/tcp_input.c L6069-L6091】

```c
/*
 *	TCP receive function for the ESTABLISHED state.
 *
 *	It is split into a fast path and a slow path. The fast path is
 * 	disabled when:
 *	- A zero window was announced from us - zero window probing
 *        is only handled properly in the slow path.
 *	- Out of order segments arrived.
 *	- Urgent data is expected.
 *	- There is no buffer space left
 *	- Unexpected TCP flags/window values/header lengths are received
 *	  (detected by checking the TCP header against pred_flags)
 *	- Data is sent in both directions. Fast path only supports pure senders

 *	- Unexpected TCP option.

 *	receive procedure patterned after RFC793 to handle all cases.
 *	The first three cases are guaranteed by proper pred_flags setting,
 *	the rest is checked inline. Fast processing is turned on in
 *	tcp_data_queue when everything is OK.
```

**位戏法拆解**（这是面试区分度最高的一段）：

TCP 头偏移 12 起的 32-bit 字 = `doff(4b) | reserved(4b) | flags(8b) | window(16b)`。`tcp_flag_word(th)` 一次读出这 32 位；`TCP_HP_BITS = ~(TCP_RESERVED_BITS|TCP_FLAG_PSH)`（tcp_input.c L112）把保留位和 PSH 掩掉（PSH 不影响处理逻辑）。`tp->pred_flags` 预先算好"期望值"：doff = 当前协商的头长（带 timestamp 选项就是 8 个字）、flags = 仅 ACK、window = 上次见到的窗口。于是**一条 32-bit 比较同时验证了：头长没变（没有奇怪选项）、标志位恰好是纯 ACK（排除 SYN/FIN/RST/URG/ECE/CWR）、窗口没变**。再加两条序列号检查（`seq == rcv_nxt`：按序到达；`ack_seq` 不超前），五个条件三条指令级开销。任何异常（乱序、零窗、紧急数据）发生时，内核把 `pred_flags` 置 0——**让这条比较永远失败**，自然落入慢路径，不需要任何额外分支。

快路径内部（L6134-L6233）再分两支：
- **纯 ACK**（`len <= tcp_header_len`）：`tcp_ack()` 更新发送状态 → 直接 `__kfree_skb` → `tcp_data_snd_check` 看能不能继续发——bulk 发送方每收一个 ACK 走的就是这条，全程不碰接收队列。
- **纯数据**（接收方）：timestamp 顺带校验（PAWS 防回绕）、`tcp_queue_rcv` 进 `sk_receive_queue`、计数器 `LINUX_MIB_TCPHPHITS`（`nstat` 里的 `TcpExtTCPHPHits`，**快路径命中率可观测**）、必要时 `__tcp_ack_snd_check` 安排 delayed ACK、`tcp_data_ready` 唤醒等待进程。
- 中间那句 `if ((int)skb->truesize > sk->sk_forward_alloc) goto step5;`——内存配额不够也踢去慢路径，呼应 20.3.5 的 truesize 记账。

**慢路径**（L6236 起）则是 RFC 793 风格的完整状态处理：`tcp_validate_incoming`（PAWS/seq 窗口/RST 校验）→ `tcp_ack(FLAG_SLOWPATH)` → `tcp_urg` → `tcp_data_queue`（乱序进 OOO 红黑树）→ ACK 决策。**性能含义**：一次乱序不只是"这个包慢了"，是 `pred_flags=0` 让**后续所有包**都走慢路径，直到洞被补齐——这就是为什么轻微丢包率对吞吐的伤害是超线性的（叠加 SACK 处理、可能的 cwnd 缩窗）。

**与 VJ 注释对话**：注释坦言 Linux 没有完全采用 VJ 的"在收包进程上下文做 checksum+copy"方案（"we take the extra cost of the net_bh soft interrupt processing"）。曾经更接近 VJ 构想的 prequeue 已死（20.2.4）；今天的现实主义答案是：softirq 里做协议处理 + GRO 摊薄次数 + （延迟敏感时）busy poll 让用户线程亲自来。

## 20.5 TX 路径：从 send() 到网线

骨架（与原版一致，精度升级）：

```
send() → tcp_sendmsg_locked：copy_from_user 进 skb（或 MSG_ZEROCOPY pin 页挂 frags）
   → sk_write_queue / rtx 队列（红黑树）
   → tcp_write_xmit：受 cwnd、rwnd、Nagle、TSQ、pacing(EDT) 共同闸控
   → __tcp_transmit_skb：skb_push 写 TCP 头（克隆自重传队列持有的 payload skb）
   → ip_queue_xmit → NF_HOOK(OUTPUT) → 路由 → ip_output → NF_HOOK(POSTROUTING)
   → dev_queue_xmit → qdisc（fq/fq_codel/pfifo_fast）→ dev_hard_start_xmit → ndo_start_xmit
   → 填 TX 描述符 → DMA → TX completion 中断 → 释放 skb（tcp_wfree 归还 wmem 配额）
```

【真实源码 v6.12 net/ipv4/tcp_output.c L1368-L1379（__tcp_transmit_skb 节选）】

```c
	skb_push(skb, tcp_header_size);
	skb_reset_transport_header(skb);

	skb_orphan(skb);
	skb->sk = sk;
	skb->destructor = skb_is_tcp_pure_ack(skb) ? __sock_wfree : tcp_wfree;
	refcount_add(skb->truesize, &sk->sk_wmem_alloc);
```

- `skb_push(skb, tcp_header_size)`：20.3.4 承诺的"封包 = 指针左移"在此兑现——TCP 头直接写进预留 headroom。
- `skb->destructor = tcp_wfree` + `refcount_add(truesize, &sk_wmem_alloc)`：发送内存记账的闭环——skb 真正被释放（TX completion 或丢弃）时 destructor 归还配额。**TSQ（TCP Small Queues）就建在这上面**：每流在 qdisc/驱动里未决的字节数超限就不再下发，把队列深度从"网卡 ring + qdisc 任意深"压成每流可控，治的是 TX 侧 bufferbloat。
- `skb_is_tcp_pure_ack` 区分纯 ACK：纯 ACK 不占发送配额（用 `__sock_wfree`），避免 ACK 风暴吃光 wmem。

**工程要点**（每条都是真坑）：

1. **`send()` 返回 ≠ 包已发出**：数据只是进了发送队列；cwnd/rwnd/Nagle/TSQ/pacing 五道闸之后才是 qdisc，再之后是异步 DMA。测"发送延迟"要用 `SO_TIMESTAMPING` 的 SCHED/SND 时间戳，不是 send() 耗时。
2. **Nagle×delayed-ACK 死锁**：Nagle（有未确认小包在途则攒数据）遇上对端 delayed ACK（攒到超时才回，约 40ms 量级）= 凭空 40ms 停顿。RPC/交互协议一律 `TCP_NODELAY`。注意真正根因是**两个"等对方先动"的启发式互锁**——单看任何一个都是对的。
3. **TSO/GSO 分工**：协议栈以 64KB 大段流转（省 N 次协议处理），`dev_queue_xmit` 末端按 `netif_needs_gso` 决定：网卡 advertise TSO ⇒ 整段下发硬件切；否则 GSO 软切。`ethtool -k` 的 tso/gso/gro 开关对应于此。诊断"大包进小包出"先看这三项。
4. **qdisc 锁**：Brouer 数据（20.1.2）qdisc 路径 ~48ns/包锁开销，多核同打一个 TX 队列时更糟——XPS（按 CPU 选 TX 队列）+ 多队列网卡是标准解法；`fq` qdisc 还承担 BBR 需要的 pacing。

### 20.6.2 nf_hook_slow：每包的"过堂"循环

【真实源码 v6.12 net/netfilter/core.c L617-L652】

```c
/* Returns 1 if okfn() needs to be executed by the caller,
 * -EPERM for NF_DROP, 0 otherwise.  Caller must hold rcu_read_lock. */
int nf_hook_slow(struct sk_buff *skb, struct nf_hook_state *state,
		 const struct nf_hook_entries *e, unsigned int s)
{
	unsigned int verdict;
	int ret;

	for (; s < e->num_hook_entries; s++) {
		verdict = nf_hook_entry_hookfn(&e->hooks[s], skb, state);
		switch (verdict & NF_VERDICT_MASK) {
		case NF_ACCEPT:
			break;
		case NF_DROP:
			kfree_skb_reason(skb,
					 SKB_DROP_REASON_NETFILTER_DROP);
			ret = NF_DROP_GETERR(verdict);
			if (ret == 0)
				ret = -EPERM;
			return ret;
		case NF_QUEUE:
			ret = nf_queue(skb, state, s, verdict);
			if (ret == 1)
				continue;
			return ret;
		case NF_STOLEN:
			return NF_DROP_GETERR(verdict);
		default:
			WARN_ON_ONCE(1);
			return 0;
		}
	}

	return 1;
}
```

- `e->hooks[s]` 顺序扫**连续数组**——这就是 20.2.3 那次 v4.14 手术（commit 960632ece694）的成品：无指针追逐，prefetcher 友好。注册期按 priority 排序写入数组，RCU 整体替换。
- 名字叫 `_slow` 是因为外面还有快层：`NF_HOOK` → `nf_hook()` 先查 per-(pf,hooknum) 的 **static key**（`nf_hooks_needed`），一条规则都没有时是一条被 patch 成 NOP 的跳转——**编译期级别的零成本**。所以"空 netfilter 框架税"在现代内核 ≈ 0；税是 conntrack/规则本身收的。
- 判决语义：`NF_ACCEPT` 续审、`NF_DROP` 带 reason 丢弃（注意返回 -EPERM——上层 send() 看到的 EPERM 多半就是 iptables REJECT/DROP 干的，经典排障线索）、`NF_QUEUE` 押送用户态（nfnetlink_queue，IDS/IPS 用）、`NF_STOLEN` 表示 hook 接管了 skb（conntrack 重组分片时用）。

```c
	get_random_once(&nf_conntrack_hash_rnd, sizeof(nf_conntrack_hash_rnd));

	key = nf_conntrack_hash_rnd;

	key.key[0] ^= zoneid;
	key.key[1] ^= net_hash_mix(net);

	return siphash((void *)tuple,
			offsetofend(struct nf_conntrack_tuple, dst.__nfct_hash_offsetend),
			&key);
}
```

- **siphash + boot 随机 key**：tuple 完全由对端构造，若 hash 可预测，攻击者可让百万连接落进同一桶，把 O(1) 查表打成 O(n) 链表扫——hash-flooding DoS。siphash 是为"短输入 + 抗碰撞构造"设计的带密钥 PRF，内核用它替换了可被逆向的 jhash（具体切换版本，待核）。`zoneid`/`net_hash_mix` 混入 key：不同 conntrack zone、不同 netns 的分布互不相关。
- 与 20.2.2 route cache 之死同一条公理的两种回应：route cache 选择"删缓存"，conntrack 没法删（NAT 必须有状态），只能"把 hash 做成攻击者算不出来的"。

设计读解（对照 20.2.3 第二刀的 commit 哲学）：

- **回收主力在 packet path**：lookup 时发现过期顺手杀（`nf_ct_is_expired` 检查随包做）。gc_worker 只兜两个底：通知 conntrack 事件监听者（conntrackd/ulogd 这类用户态需要"及时"知道超时）+ 系统转闲后回收内存。

| 维度 | iptables/xtables | nftables | eBPF 系（XDP/tc） |
|---|---|---|---|
| 内核进入 | 2.4（开发于 2.3.x） | 3.13 | 4.8+（XDP） |
| 匹配模型 | 协议字段硬编码 ABI，链式线性匹配 O(n) | 内核字节码 VM + set/map（哈希/区间），用户态编译 | 任意 C/字节码逻辑，verifier 保证安全 |

## 20.7 net namespace：网络世界的多租户

**考古**（全部已核实）：基础设施由 Eric W. Biederman 合入——commit **5f256becd868**（2007-09-12，→2.6.24）"Basic network namespace infrastructure"，commit message 给出了最初形态：`struct net` 数据结构 + per-ns 初始化注册函数 + 引用计数（get/put 强引用、hold/release 弱引用）。同一周期 Pavel Emelyanov（OpenVZ）的 commit **e314dbdc1c0d**（2007-09-25）合入 **veth** 驱动，commit message 直说动机："Mainly it allows to communicate between network namespaces"。配套教程见 LWN **580893**（Jake Edge，2014-01-22，"Namespaces in operation, part 7"，已核实）。这是 OpenVZ/Virtuozzo 容器经验反哺主线的代表作——Docker（2013）比这套基础设施晚了六年。

**每个 netns 独立持有**：设备列表、FIB 路由表、netfilter/nftables 规则与 conntrack 表（可分 zone）、socket 哈希表、`/proc/net` 视图、大批 per-ns sysctl。

```
  Host netns                          Container netns
  ─────────────                       ─────────────
  eth0(物理) lo                        lo  eth0(=veth 对端,改名)
  FIB / nft 规则 / conntrack           独立 FIB / 规则 / conntrack
       veth0 ◀━━━━ skb 直接跨 ns 转递 ━━━━▶ veth1
```

**工程推论与坑**：

1. veth 的"线"是函数调用：一端 `ndo_start_xmit` 直接把 skb 注入对端的 RX 路径，**不过物理层但要过两遍完整协议栈**——容器间通信的 CPU 成本 ≈ 2×本机栈成本。这正是 kube 网络后来引入 sockmap/IPVS/eBPF host-routing 这类捷径的原因。
2. **per-ns sysctl 是排障暗雷**：容器里 `net.core.somaxconn`、`tcp_tw_reuse` 等与宿主机各自独立——"我明明调了参数"结果调的是宿主机的，是容器化排障第一高频乌龙。`nsenter -t <pid> -n sysctl ...` 验证。
3. netns 创建/销毁不便宜（注册一整套 per-ns 子系统状态；销毁要等 RCU 与 worker 收尾）——海量短命 Pod 场景下 `cleanup_net` 积压曾是真实事故源。复用 netns（pause 容器模型）部分是性能动机。
4. conntrack/nft 规则隔离 ⇒ 容器内 NOTRACK 不影响宿主机；但 veth 包到宿主机侧后**再次**经过宿主机的 conntrack——双重记账，宿主机 `nf_conntrack_max` 要按全机连接总量配。

## 20.8 方案对比：三个场景推演（带不适用边界）

> 方法论：先列硬约束，再沿数据路径算每包预算，最后给反例（什么时候这个结论翻车）。

### 场景 A：1 Mpps 小包 RPC 网关，单核还是旁路？

**设定**：64–256B 请求，目标 1 Mpps 入 + 1 Mpps 出，P99 < 200μs，需要按源 IP 限流，宿主机还跑其他业务。

**预算演算**：1 Mpps = 每包 1000ns。对照 20.1.2：skb 全生命周期 ~77ns + conntrack 两次查找（数十 ns，假设无 miss）+ socket 查找 + 快路径 TCP（数百 ns 含锁与统计）+ 两次 copy（256B 时 copy 本身不大，但 syscall 42–75ns×2）+ qdisc 48ns。**通用栈单核 1–2 Mpps 的上限（Brouer 数据）意味着 1 Mpps 双向已经把一个核吃干**——可行但零余量，P99 会被任何 softirq 干扰打爆。

**选项对比**：

| 方案 | 每包路径 | 预期单核能力 | 代价 |
|---|---|---|---|
| ①通用栈 + 多队列 RSS 摊 4 核 | 全栈 | 1–2 Mpps/核 | 简单；conntrack 税照交；跨核 cache 弹跳 |
| ②XDP 做限流/丢弃 + 正常栈收正经请求 | 恶意/超限流量在 skb 分配前死掉（20 Mpps 级 drop） | 防御面解决，正常面同① | 需要维护 eBPF；XDP 看不到 conntrack 状态 |
| ③AF_XDP/DPDK 全旁路 | 用户态全自管 | ~15 Mpps/核 | 失去内核 TCP——**RPC 是 TCP 的话等于要自带用户态 TCP 栈**，运维/安全/可观测全自建 |
| ④io_uring + 通用栈 | 省 syscall（多次操作一次提交） | syscall 项归零，其余同① | 只削掉 42–75ns×N 项，包处理成本不变 |

**推理结论**：限流诉求 → ② 是确定赢家（这正是 Cloudflare/Katran 的生产形态）；正常流量 → ① RSS 分 4 核 + `napi_defer_hard_irqs/gro_flush_timeout` 控中断率 + 对内部可信平面 NOTRACK 免税；只有当 RPC 协议本身可以 UDP 化/自带可靠性时，③ 才进入讨论区。
**翻车边界**：包大了（1KB+）瓶颈转向带宽和 copy，XDP 收益骤减；连接数百万级时 ① 的 socket 哈希和内存压力反超 per-packet 成本；宿主机混部 → ③ 的独占网卡假设直接不成立。

### 场景 B：拥塞控制选型——CUBIC 还是 BBR？

**跑一遍跨洋链路**：RTT 150ms、瓶颈 10Gbps、深 buffer（运营商常见）。BDP ≈ 187MB。CUBIC：丢包才减窗 ⇒ 稳态把深 buffer 填满 ⇒ 实际 RTT 被队列撑到 300ms+（bufferbloat，BBR commit message 原话 "often causing seconds of needless queuing delay"）；偶发随机丢包（非拥塞）⇒ 窗口腰斩，长肥管道恢复极慢。BBR：持续测 delivery rate + min_rtt，inflight 钉在 ~BDP ⇒ RTT 保持 ~155ms，随机丢包基本无感。**此场景 BBR 碾压**——这正是 Google B4/YouTube 部署它的环境（commit 0f8782ea1497）。

**再跑数据中心**：RTT 50μs、浅 buffer 交换机、incast 流量。丢包=真拥塞信号且反馈极快，CUBIC/DCTCP(+ECN) 工作良好；BBRv1 的带宽探测脉冲反而制造队列毛刺，且与 CUBIC 流共存时**侵占性强**（LWN 701165 当时即记录公平性疑虑，"works fine when only one side of the connection is using it"）。**此场景选 DCTCP（有 ECN）或 CUBIC，不选 BBRv1**；BBRv2/v3 在改善公平性但仍需实测。

**决策规则**：你控制瓶颈两端（内网）→ 按 buffer 深浅与 ECN 可用性选；你只控发送端面向公网（CDN/出海业务）→ BBR 大概率赢；混合流量共存敏感 → 实测公平性再上线。`ss -ti` 看每连接 `bbr` 状态/`cwnd`/`rtt` 是验证工具。

### 场景 C：中断模型四选一

**负载画像 × 模式**：

| 模式 | 机制（源码锚点） | 适用 | 不适用 |
|---|---|---|---|
| 默认 NAPI | 中断↔轮询自切（20.4） | 绝大多数 | 极端尾延迟敏感 |
| defer_hard_irqs + gro_flush_timeout | napi_complete_done 内 hrtimer 续命（L6240-6245） | 高 PPS 降中断率、吞吐型 | 流量稀疏时白等 timeout（μs 级延迟上浮） |
| busy poll（含 epoll busy poll） | 用户线程自带 CPU 轮（NAPIF_STATE_IN_BUSY_POLL） | 交易/存储等 P99.9 敏感且肯烧核 | 多租户混部（烧整核）；连接散布多队列时失效（epoll busy poll 要求 fd 同 NAPI） |
| threaded NAPI | 每 NAPI 一个 kthread（commit 29863d41bb6e；`napi/<if>-<id>`） | 需要把网络处理纳入调度策略：绑核、限优先级、与业务隔离 | 线程唤醒延迟 > softirq 即时性的低延迟场景；官方文档建议配合绑核使用 |

**推演一个具体决策**：容器宿主机，业务抱怨"邻居 Pod 打流量时我的服务长尾飙升"。默认模式下 softirq 记账在受害者进程头上且不可调度隔离 ⇒ **threaded NAPI + 把 `napi/*` 线程圈进专属 cpuset** 是结构性解法；若改用 busy poll 则等于让每个受害者自己烧核，混部密度崩——方向性错误。反过来，单租户低延迟网关用 threaded 反而多一跳唤醒，busy poll 才对。**同一组旋钮，部署形态决定答案。**

### 四个排障场景（修订）

**场景 1：大量 TIME_WAIT** —— `ss -tan state time-wait | wc -l`。Server 侧 TIME_WAIT 基本无害（不耗 listen 端口）；真正受伤的是**主动发起连接的一侧**（client/代理的 upstream 侧）：四元组耗尽 → `connect: EADDRNOTAVAIL`。修复优先级：连接池/长连接 > `tcp_tw_reuse=1`（仅出向，需双端 timestamps）> 扩 `ip_local_port_range` / 多源 IP。**不要**找 tw_recycle（4.12 已删，NAT 下误杀）。

**场景 2：accept queue 满** —— `ss -lnt` 看 LISTEN 行 `Recv-Q/Send-Q`（当前积压/队列容量）；`nstat | grep ListenDrops` 持续涨即坐实。注意语义：队列满时内核默认**丢第三次握手 ACK 而不是回 RST**（除非 `tcp_abort_on_overflow=1`），客户端视角是"连上了然后超时"——这个错位是排障经典迷雾。解法：调大 somaxconn+backlog 只是止血，根治是 accept 线程别被业务逻辑拖住（accept 专线程/SO_REUSEPORT 分片）。

**场景 3：吞吐顶不上去但 CPU 不满** —— 按路径逐站查计数：`ethtool -S eth0 | egrep 'drop|miss|err'`（ring 溢出 → `ethtool -G` 调大 + 检查 IRQ 亲和是否被 irqbalance 搬走）；`/proc/net/softnet_stat` 第 3 列（softirq 预算不够 → RSS 加队列分核）；`nstat` 的 OFO/重传（链路质量/对端）。**先定位丢在哪一站，再调那一站的旋钮**——反着做就是玄学调参。

**场景 4：conntrack 表压力** —— 链路：`nf_conntrack_count` 逼近 max → packet path 同步 early_drop（20.6.3）→ softirq 延迟尖刺 + `nf_conntrack: table full, dropping packet` 日志。处置：调大 max+buckets；缩短 established timeout；可信平面 `iptables -t raw -j NOTRACK`/nft `notrack`；高对抗场景上 XDP 在 conntrack 之前削峰。验证手段：`conntrack -S`（insert_failed/early_drop 计数）、`perf` 看 `nf_conntrack_in` 占比。

- **NAPI 的精细化**：per-NAPI netlink 配置（队列级 defer/timeout/threaded）已在官方文档成文；threaded busy-poll（`threaded: busy-poll` 模式）面向转发与 AF_XDP 负载（docs.kernel.org/networking/napi.html，已核实）。
- **io_uring 网络面**：RECV/SEND/ACCEPT 体系成熟，send zero-copy 与注册缓冲区持续演进（具体到 zerocopy receive 的版本边界，待核）。与 busy poll 组合是"用户态驱动型低延迟"的内核内答案。
- **BIG TCP**：让 GSO/GRO 聚合段突破 64KB（IPv6 先行，IPv4 跟进；具体版本号待核），面向 100G+ 单流；本质仍是 20.2.5 的老配方——加大批处理粒度摊薄每包成本。
- **QUIC**：内核态 QUIC 仍无主线实现，主流在用户态（quiche/ngtcp2/msquic）；内核侧配合是 UDP GSO/GRO 与 socket 层优化。值得玩味：QUIC 把传输层搬回用户态，等于对本章 20.2.4 全部 TCP 内核演进的一次"路线否定"——但它立刻重新发明了 pacing、ACK 聚合和 offload 诉求，时间预算定律不挑协议。
- **eBPF 继续向栈内渗透**：sockmap/sockops 短路本机代理、SK_REUSEPORT 自定义分发、BPF qdisc/拥塞控制（`tcp_congestion_ops` 的 BPF 实现）——"通用栈 + 可编程逃生舱"已是既定路线。
- **Rust**：网络核心路径无 Rust 化计划；驱动与外围工具是潜在入口（保守判断，待核）。

## 章末五件套

### 一、Staff 级高频面试题（含源码级追问）

**Q1：完整描述一个 TCP 包从网卡到 recv() 的路径，并指出三处最贵的成本。**
框架：DMA→hardirq（只 ack+关中断+napi_schedule，dev.c L4511）→ NET_RX_SOFTIRQ `net_rx_action`（300/2ms 两闸）→ driver poll（≤64）→ [XDP] → GRO → `__netif_receive_skb_core`（tap/TC）→ `ip_rcv`+PREROUTING（conntrack）→ FIB trie → LOCAL_IN → `tcp_v4_rcv` socket 哈希 → `tcp_rcv_established` pred_flags 快路径 → `sk_receive_queue` → 唤醒 → `copy_to_user`。
三贵：skb 分配/回收链（~77ns 级）、conntrack+socket 两次哈希查找及其 cache miss、copy_to_user（随包长）。陷阱：epoll 不在收包路径上，它只是唤醒机制。

**Q2：NAPI 的 budget 为什么是两级？分别防什么？**
per-NAPI `weight=64`（NAPI_POLL_WEIGHT）防单设备/单队列垄断一轮 softirq——多 NAPI 间 round-robin 公平；全局 `netdev_budget=300`+`netdev_budget_usecs=2ms` 防**网络整体**垄断这颗 CPU——保护定时器/RCU/进程调度。证据点：`net_rx_action` 源码里 `budget -= napi_poll(...)` 与 `time_after_eq(jiffies, time_limit)` 双判，超限 `time_squeeze++`（softnet_stat 第 3 列）。追问：为什么驱动 poll 用满 weight 时不准自己 napi_complete？——所有权契约，用满即默认还有活，框架保留调度权（`__napi_poll` 注释原文）。

**Q3：`napi_complete_done` 里 NAPIF_STATE_MISSED 解决什么竞态？**
poll 见 ring 空准备清 SCHED 开中断的同时，新包触发 hardirq——`napi_schedule_prep` 见 SCHED 仍在，只置 MISSED。complete 侧 `try_cmpxchg` 环里见 MISSED 则保留 SCHED 并立刻重调度，否则丢唤醒、包睡死。本质是 lost-wakeup 问题的无锁解；加分项：能讲出 `(val & MISSED)/MISSED*SCHED` 是无分支乘法技巧。

**Q4：pred_flags 一次比较验证了哪三件事？乱序为什么伤害超线性？**
doff（头长/选项无变化）+ flags==纯 ACK + window 无变化，三合一 32-bit 比较（TCP_HP_BITS 掩掉 PSH/保留位），再加 seq==rcv_nxt 与 ack 不超前。乱序发生 ⇒ `pred_flags=0` ⇒ 直到洞补齐前**所有**后续段都走慢路径（OOO 红黑树、SACK、可能缩窗）——单点事件全局放大。观测：`TcpExtTCPHPHits` 增速。

**Q5：conntrack 为什么从 per-entry timer 改成 gc worker？表满时真正的延迟杀手是什么？**
百万连接=百万 timer 的插删开销且分布在热路径；v4.9 改为"过期判定 lookup 顺带做 + 后台分片扫描兜底"（commit b87a2f9199ea：每次至多 1/64 表、8k 桶，目的仅是事件通知与闲时回收）。表满杀手不是 gc，而是新建连接 packet path 里的**同步 early_drop**——softirq 内扫桶找牺牲品，造成 P99 尖刺。配套答出 93bb0ceb75be（全局锁删除，删前 ksoftirqd 72.56% 在自旋锁上）可定级。

**Q6：为什么 IPv4 route cache 被删而 conntrack 表没被删？**
同一公理（攻击者可污染 key space 的缓存=DoS 面）下的两种处境：路由查找是纯函数，可以"把 miss 路径做快到不需要缓存"（FIB trie），所以删；conntrack 是 NAT 的**语义必需状态**，无法重算，所以只能加固（siphash 随机 key、per-bucket 锁、early_drop 策略）。引 commit 89aef8921bfb 原文 "non-deterministic, performance wise" 是满分点。

**Q7：listen backlog、SYN queue、accept queue 三者关系与各自溢出表现？**
backlog 参数与 somaxconn 取小 = accept queue 容量；SYN queue 由 tcp_max_syn_backlog 控制，存 request_sock 迷你状态（NEW_SYN_RECV，见 tcp_v4_rcv 源码分支）。SYN 满：syncookies 接管或丢 SYN（客户端重试 SYN）；accept 满：默认**静默丢握手 ACK**（连接在客户端看已建立，随后超时）——`ss -lnt` Recv-Q、`ListenDrops/ListenOverflows` 区分。追问 syncookies 代价：无状态导致部分 TCP 选项（大窗口等）退化。

**Q8：MSG_ZEROCOPY 为什么不默认开启？**
零拷贝省的是 copy 的线性成本，付的是 pin 页+完成通知（error queue）+页生命周期管理的固定成本；小包入不敷出（LWN 726917：microbenchmark +39% 但生产负载仅 +5–8%，且内核保留小包静默回退拷贝的权利）。通用 API 的默认值必须按最差情况设防——这是内核 API 设计的普遍纪律。

### 三、设计题

**设计 10Gbps 入口、P99<100μs、按 IP 限流的 API Gateway。** 考察点（对照 20.8 场景 A/C 推演）：RSS+irq 亲和规划（隔离 NIC 中断核）；限流位置选择 XDP（skb 前，20Mpps 级 drop 能力）vs nftables meter（功能全但每包过栈）并说明为何选前者；GRO 与 P99 的张力（`gro_flush_timeout` 调小或 busy poll）；conntrack 取舍（网关本机连接用 NOTRACK 免税，NAT 需求则接受两次哈希）；io_uring/busy-poll 削 syscall 与唤醒延迟；给出每阶段 ns 级预算表并标明测量手段（`perf`/`bpftrace`/`SO_TIMESTAMPING`）。

### 四、系统题

**现象**：8 核服务器跑 Nginx 短连接，CPU 40%，P99 周期性从 5ms 飙到 200ms，`ss -s` 显示 TIME_WAIT 8 万，dmesg 偶现 `nf_conntrack: table full, dropping packet`。
**期望推理链**：TIME_WAIT 8 万本身对 server 无害（端口不耗在 listen 侧）→ 但每个 TIME_WAIT 在 conntrack 里也占条目（tcp timeout_time_wait 默认 120s）→ `nf_conntrack_count` 逼近 max → packet path 同步 early_drop + 新建连接插入失败 → softirq 尖刺与连接超时重试（重试风暴又加剧表压）→ 周期性来自"流量波动 × 表水位在 95% 阈值附近震荡"。
**验证**：`conntrack -S` 看 early_drop/insert_failed 增量与尖刺时刻对齐；`perf top` 尖刺窗口抓 `nf_conntrack_in`/early_drop 栈；对照 `nf_conntrack_count/max`。
**修复**：调大 max+buckets、established/time_wait timeout 调短、Nginx→upstream 改长连接（根治短连接风暴）、内部可信流量 raw 表 NOTRACK。**反面教材**：见 TIME_WAIT 就调 tw_recycle（已删）或盲调 tw_reuse（对 server 入连接无效）。

### 五、代码题

用 bpftrace 统计 TCP 收包并区分上下文（修订版，含暗坑解释）：

```bash
# 稳定 tracepoint 版（推荐）：统计每进程上下文的收包
bpftrace -e '
tracepoint:net:netif_receive_skb { @rx[comm] = count(); }
interval:s:5 { print(@rx); clear(@rx); }'

# kprobe 版（函数名随版本变化，v6.12 存在 tcp_v4_rcv）：
bpftrace -e 'kprobe:tcp_v4_rcv { @[comm] = count(); } interval:s:10 { print(@); clear(@); }'
```

**暗坑**：(1) softirq 上下文里 `comm` 是被打断者（常见 `ksoftirqd/N` 或任意倒霉进程），不是收包的业务进程——要按业务归账需挂 socket 边界（`syscalls:sys_enter_recvmsg`）或用 `args->skbaddr` 关联；threaded NAPI 下则会看到 `napi/<if>-<id>` 线程名，正好可验证 20.8 场景 C 的配置生效。(2) 先 `bpftrace -l 'tracepoint:net:*'` 确认内核编译了对应 tracepoint。(3) 高 PPS 下 map 打印本身有开销，生产采样加谓词或用 `kfree_skb` 聚合 reason 替代全量计数。

---

> **本章心法**：网络栈的每个设计都是一张欠条——通用性向时间预算赊的账。NAPI 用延迟换稳态，GRO 用延迟换吞吐，conntrack 用每包两次哈希换 NAT 语义，XDP 用放弃抽象换回 67.2ns 里的主动权。读懂欠条上的利率，才知道什么时候该续borrow、什么时候该违约（旁路）。
