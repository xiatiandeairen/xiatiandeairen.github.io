> **一句话定位**：系统调用是用户态程序唯一合法进入内核的通道——理解它，就是理解"用户态→内核态"这道最关键的隔离墙是怎么建的、怎么穿的、怎么守的。

---

## TL;DR（本章最硬结论）

1. **现代 x86-64 用 `syscall` 指令**，不是 `int 0x80`；它不走 IDT，直接通过 MSR（Model Specific Register）跳转，比软中断快约 30-50 ns（平台相关）。
2. **`syscall` 编号是 ABI**，内核承诺"绝不 break userspace"——这是 Linux 最强的工程契约之一，比任何语言 ABI 都更硬。
3. **vDSO 让 `gettimeofday` 等高频调用根本不进内核**——它把内核数据映射进用户空间，完全是用户态读操作。
4. **`strace` 底层是 `ptrace(PTRACE_SYSCALL)`**，每个系统调用两次 trap，生产环境开销不可忽视。
5. **seccomp-bpf 是容器/沙箱的系统调用过滤防线**，是 Docker/Chrome sandbox 的核心机制。

**前置依赖章**：第 1 章（进程地址空间、内核态/用户态基础），后续交叉引用：进程创建见第 4 章；时钟与 vDSO 见第 6 章。

## 2.1 背景：为什么需要这层切换

操作系统的核心资产是"隔离"——内核代码运行在特权级，可以直接操作硬件、管理物理内存；用户程序没有这个权力。但用户程序又必须能请求内核帮忙（打开文件、发送网络包、分配内存）。

这个矛盾的唯一解：**受控入口**。用户态程序触发一次硬件级切换（privilege level change），内核在这个入口处检查、服务、然后返回。系统调用就是这套机制的接口规范。

类比你熟悉的后端架构：用户态是"外网区"，内核是"核心内网区"，系统调用是唯一一个"受控 API Gateway"——有白名单（`seccomp`），有审计（`strace`），有代理缓存（`vDSO`）。

## 2.2 演进：从 `int 0x80` 到 `syscall`

### 早期：软中断 `int 0x80`（x86 32位时代）

早期 Linux（x86 32位，内核 2.x 时代）用 `int 0x80` 触发系统调用：

```
用户态                               内核态
  │                                     │
  │  mov eax, __NR_write (4)            │
  │  mov ebx, fd                        │
  │  mov ecx, buf                       │
  │  mov edx, len                       │
  │  int 0x80  ──────────────────────►  IDT[0x80] → system_call()
  │                                     │  查系统调用表
  │  ◄────────────────────────────────  返回用户态
  │  eax = 返回值                       │
```

`int 0x80` 走的是**中断描述符表（IDT）**：CPU 查 IDT[0x80]，发现这是一个"陷阱门"（trap gate），压栈保存 CS/EIP/EFLAGS，切换到内核栈，跳转到内核注册的 handler。

> 💡 **细节（面试常深挖）**：IDT 里绝大多数门的 DPL 都是 0（只有内核能主动触发），但 `0x80` 这一项被特意设成 **DPL=3**——否则用户态 `int 0x80` 会直接吃 `#GP`。"为什么偏偏 0x80 能从 ring 3 进、别的 `int n` 不行"，答案就是这个 DPL。另外选 trap gate 而非 interrupt gate，意味着进入 handler 时**不自动关中断**（interrupt gate 才会清 IF），中断开关由内核自己控制。

问题：`int` 指令要读 IDT，涉及内存访问，还要走 CPU 的权限检查全流程。Intel 自己测过，这个路径在 Pentium 4 之后越来越慢（流水线预测失败代价变大）。

### `sysenter/sysexit`（Intel，32位快速路径）

Intel 在 Pentium II 引入 `sysenter`，AMD 有对应的 `syscall`（32位版本）。核心思路：**预先把内核入口地址、内核栈指针写到 MSR**，`sysenter` 指令直接从 MSR 加载，绕过 IDT 查表和大量权限检查流程。速度快了，但 32 位版本设计上有一些限制（如不自动保存用户态 ESP）。

### `syscall/sysret`（x86-64 时代，当前主流）

x86-64 引入了干净的 `syscall`/`sysret` 指令对（来自 AMD 64 架构，Intel 后采纳）：

```
MSR 寄存器（系统启动时写好）：
  LSTAR (0xC0000082)  = 内核 syscall 入口地址（近年内核为 entry_SYSCALL_64）
  STAR  (0xC0000081)  = 内核 CS/SS 选择子 + 用户态 CS/SS
  SFMASK(0xC0000084)  = syscall 执行时清除的 RFLAGS 位
```

`syscall` 指令做的事（硬件级，原子）：
1. 将 `RIP`（下一条指令）保存到 `RCX`
2. 将 `RFLAGS` 保存到 `R11`，然后用 `SFMASK` 清掉特定位（如关中断标志）
3. 从 `STAR` 加载新的 CS（内核代码段）
4. 从 `LSTAR` 加载新的 `RIP`，跳进内核

注意：`syscall` **不切换栈**，也不压 IDT 所需的那些东西。内核入口代码（`entry_SYSCALL_64`，近年在 `arch/x86/entry/entry_64.S`；它的地址在启动时由 `arch/x86/kernel/cpu/common.c` 写进 `MSR_LSTAR`）需要自己做：
- `swapgs`：切换 GS 寄存器基址（x86-64 约定用 GS 存 per-CPU 数据，`swapgs` 在用户 GS 和内核 GS 之间交换）
- 切换到内核栈（从 per-CPU 结构取内核 RSP）
- 保存用户寄存器（构造 `pt_regs` 结构）

```
x86-64 系统调用时序图

用户态                                    内核态
  │                                          │
  │ mov rax, __NR_xxx                        │
  │ mov rdi, arg1 (arg2:rsi, arg3:rdx)       │
  │ syscall ─────────────────────────────►  entry_SYSCALL_64
  │          (RCX=返回地址, R11=RFLAGS)        │  swapgs
  │                                          │  切内核栈
  │                                          │  SAVE_REGS → pt_regs
  │                                          │  do_syscall_64        (arch/x86/entry/common.c)
  │                                          │    dispatch by nr →
  │                                          │    调用 __x64_sys_xxx()
  │                                          │  RESTORE_REGS
  │  ◄───────────────────────────────────  sysret
  │ (RIP=RCX, RFLAGS=R11)                    │
  │ rax = 返回值                              │
```

> ⚠️ **时效（dispatch 机制已演进，面试常踩）**：经典心智模型是"`do_syscall_64` 用 `rax` 索引 `sys_call_table[]` 数组，间接调用 `sys_xxx`"。**近年内核已经不再这样 dispatch**——
> - 约 4.17 起，dispatch 从"直接调 `sys_xxx`"改为"调 `__x64_sys_xxx` wrapper"（`SYSCALL_DEFINE` 生成的 thunk，统一从 `pt_regs` 取参数，顺带堵掉 syscall 参数寄存器高位的一类信息泄露，参见 LWN "Rewiring x86 system-call dispatch"）。
> - 约 **6.9** 起，x86-64 把"数组间接调用"整体换成了 **`switch` 语句分发**（生成的分发函数大致是 `x64_sys_call(regs, nr)`，对 `nr` 做 `switch`；待核：具体在 `arch/x86/entry/syscall_64.c` 附近）。动机是 **Spectre v2**：间接调用 `call *table[nr]` 必须套 retpoline，既慢又能被 BTB 投毒；`switch` 编译成直接跳转，绕开间接分支预测器。
> - `sys_call_table[]` **数组本身还在**，但已退化为 tracing 等用途（`kernel/trace/trace_syscalls.c` 仍需拿到调用地址），不再是热路径 dispatch 的核心。
>
> 面试讲"查 `sys_call_table`"不算错（历史与教学模型），但能补一句"现代内核因 Spectre 改成了 switch 直接分发，顺带消灭了一个常被 rootkit hook 的间接调用点"是明显的深度信号。

**工程推论**：
1. `sysret` 比 `iret` 快，因为不需要恢复 CS/SS（用 STAR 固定值）、不读 IDT；代价是有安全限制——这就是著名的 **CVE-2012-0217**（与 Spectre/Meltdown 无关，时间早得多，2012 年）：在 **Intel** CPU 上，如果 `sysret` 要返回的地址（在 `RCX` 里）是 **non-canonical address**，`#GP` 会在**切回用户态特权级之前、仍处于 ring 0 且已切到内核栈的状态下**触发，攻击者借此在内核上下文里跑代码 → 本地提权（当年 Xen / FreeBSD / Windows 都中招）。**AMD** 的 `sysret` 在用户态才 fault，所以不受影响。内核缓解：返回前检查地址是否 canonical，不是就 fall back 到 `iret`——这正是"部分路径仍用 `iret` 兜底"的真实来由。
2. `swapgs` 操作是 Spectre/Meltdown 修复的重点之一（见近年内核引入的 SWAPGS 攻击缓解补丁）。
3. 系统调用进入内核后，中断是否开启由内核控制——这是实时内核（PREEMPT_RT）改造的重要战场。

## 2.3 系统调用表与编号

### 编号即 ABI

Linux x86-64 系统调用表（待核：具体文件近年在 `arch/x86/entry/syscalls/syscall_64.tbl`）将编号 → 处理函数名做映射，例如：

| 编号  | 名称         | 典型参数描述           |
|-------|--------------|------------------------|
| 0     | `read`       | fd, buf, count         |
| 1     | `write`      | fd, buf, count         |
| 2     | `open`       | filename, flags, mode  |
| 60    | `exit`       | error_code             |
| 231   | `exit_group` | error_code             |
| 435   | `clone3`     | ...                    |

编号是硬 ABI——一旦在某内核版本发布，**永远不能改**，即使函数实现完全重写了。所以内核里能看到形如 `sys_newstat` 这种名字（`newstat` 意味着这已经是替换过历史实现的版本）。

💡 **坑点**：x86-64 和 x86-32（compat 模式）的系统调用编号**不同**——32位进程在 64位内核上走 `int 0x80` 用的是 32位表，`write` 编号还是 4。混淆这一点会导致难以理解的 strace 输出。

不同架构的系统调用编号也**不同**（ARM64 有自己的表，RISC-V 有自己的表）——所以系统调用是"不可移植的 ABI"（在 ABI 层可移植，但编号不跨 ISA）。

### "don't break userspace"——内核最强工程契约

Linus Torvalds 有一条出了名的原则：内核升级绝不允许 break 现有用户态程序（只要程序没有依赖 buggy behavior 且行为是 documented 的）。这条规则比很多开源项目都严苛。

工程后果：
- 内核代码里存在大量"legacy 系统调用"（如 `select` 之于 `epoll`），永远不会删
- 新功能用新系统调用（`clone3` 之于 `clone`），旧接口保持不变
- 系统调用接口改动必须经过极严格的审查

🎯 **面试加分点**：如果面试官问"为什么 Linux 内核里有那么多'deprecated'但没删的代码"，这是核心答案。

## 2.4 glibc 封装层与 errno

### 不要直接用 `syscall` 汇编

用户程序几乎从不直接写 `syscall` 汇编，而是调用 glibc（或 musl）的封装函数：

```
应用代码
  │  write(fd, buf, len)
  ▼
glibc libc.so
  │  设置寄存器 (rax=1, rdi=fd, rsi=buf, rdx=len)
  │  syscall 指令
  │  检查 rax 返回值（负数 → 设置 errno，返回 -1）
  ▼
返回结果
```

glibc 做了几件事：
1. **参数准备**：把 C 函数参数按 Linux x86-64 调用约定放到对应寄存器（rdi/rsi/rdx/r10/r8/r9）
2. **执行 `syscall`**
3. **错误翻译**：内核返回负的 errno 值（如 `-ENOENT = -2`），glibc 将其转为 `errno` 全局变量设为 `ENOENT(2)`，函数返回 `-1`

💡 **坑点（errno 线程安全）**：现代 glibc 的 `errno` 是 thread-local（`__thread int errno`），不是真正的全局变量。这也是为什么多线程程序中 errno 是安全的。但有一个经典陷阱：

```c
// 危险！两次系统调用之间不要读 errno
int r1 = write(...);
// 如果这里有信号处理器调用了某系统调用 → errno 被覆盖
int r2 = write(...);
if (r1 < 0) check errno; // errno 可能已是 r2 的错误
```

### 直接用 `syscall()` 函数

glibc 提供了一个通用的 `syscall(number, ...)` 函数，可以直接调用内核（甚至调用内核有但 glibc 没有封装的新系统调用）：

```c
#include <sys/syscall.h>
#include <unistd.h>

long ret = syscall(SYS_write, fd, buf, len);
```

这在调试新内核特性、或者使用较新系统调用（如 `io_uring_setup`）时非常有用。

## 2.5 vDSO：不进内核的"系统调用"

### 问题：高频调用的开销

`gettimeofday` 在某些应用中调用极为频繁（每个日志行、每个 metric 采集）。如果每次都进内核，即使只需几十纳秒，乘以百万次/秒也是显著开销。

### vDSO 方案

**virtual Dynamic Shared Object（vDSO）** 是内核映射到每个进程地址空间的一小段共享内存（通常 4KB 或 8KB），包含：
- 内核维护的时间数据结构（`vvar`，只读映射）
- 用户态可执行的时间查询代码

```
进程地址空间
  ┌─────────────────────────┐ 0xFFFFFFFFFFFFFFFF
  │      内核空间（不可见）   │
  ├─────────────────────────┤ 0xFFFF800000000000
  │                         │
  │      用户栈              │
  │                         │
  │      ...                │
  │                         │
  │  [vdso]  ◄─ 内核映射   │ 随机地址（ASLR）
  │    clock_gettime 代码    │
  │                         │
  │  [vvar]  ◄─ 内核更新   │ 只读，包含时间数据
  │    seqlock + timekeeping │
  │                         │
  └─────────────────────────┘ 0x0
```

当 glibc 的 `clock_gettime(CLOCK_REALTIME, ...)` 被调用时，glibc 的动态链接器在启动时已经找到了 `vDSO`，直接跳转到 vDSO 中的实现——这段代码在用户态运行，读取 `vvar` 里内核定时更新的时间数据（用 seqlock 保证一致性），**完全不触发系统调用**。

### 为什么安全？

`vvar` 是**只读**映射（用户态不能写）。时间数据由内核 tick/hrtimer 中断更新。用户态代码用 seqlock 读：先读版本号，读数据，再确认版本号没变；如果变了（内核正在更新）就重试。这个 seqlock 保证了无锁的读一致性。

### 哪些调用走 vDSO（待核：具体版本和架构可能有差异）

| 调用                     | vDSO 覆盖 |
|--------------------------|-----------|
| `clock_gettime`          | 是（x86-64）|
| `gettimeofday`           | 是（x86-64）|
| `time`                   | 是（x86-64）|
| `clock_getres`           | 是（x86-64 有 `__vdso_clock_getres`）|
| `getcpu`                 | 是（x86-64，底层用 `RDPID`/`RDTSCP`）|

`strace` 看不到 vDSO 中的调用——因为没有真正的系统调用发生。这是新手排查性能问题时的常见困惑。

**（见第 6 章：时钟与 timekeeping 机制，有 vDSO 数据结构的详细展开）**

🎯 **面试加分点**：面试官可能问"为什么 `strace` 里看不到 `gettimeofday`"。答案是 vDSO——这是系统层知识深度的分水岭。

## 2.6 `strace` 与 `ltrace` 原理

### strace 的实现：`ptrace(PTRACE_SYSCALL)`

`strace` 用 `ptrace` 系统调用把目标进程置于被调试状态：

```
strace 进程                              目标进程
  │                                          │
  │ ptrace(PTRACE_ATTACH, pid, ...)          │
  │ 或 fork() + ptrace(PTRACE_TRACEME)      │
  │                                          │
  │ ptrace(PTRACE_SYSCALL, pid, ...)         │
  │ waitpid(pid, ...)  ──────────────────►  syscall 入口停止 (SIGTRAP)
  │ 读 ptrace(PTRACE_GETREGS) 看参数         │  ← 第一次 trap
  │ ptrace(PTRACE_SYSCALL, pid, ...)         │
  │ waitpid(pid, ...)  ──────────────────►  syscall 出口停止 (SIGTRAP)
  │ 读 ptrace(PTRACE_GETREGS) 看返回值       │  ← 第二次 trap
  │ ptrace(PTRACE_SYSCALL, pid, ...)         │
  │ ...                                      │
```

**每个系统调用 = 两次 SIGTRAP + 两次上下文切换到 strace 进程**。这意味着 strace 的开销非常高——在高频系统调用的程序上开 strace，性能可能下降 5-100x 甚至更多。

💡 **生产用 strace 的正确姿势**：
```bash
# 只 attach，不从头开始（不中断进程生命周期）
strace -p <pid>

# 只追踪特定系统调用（减少停止次数）
strace -e trace=network,read,write -p <pid>

# 统计模式（不打印每次调用，只统计，开销小得多）
strace -c -p <pid>

# 限制输出长度（避免 strace 进程成为瓶颈）
strace -s 256 ./program
```

### `strace` 常见生产场景

| 场景                        | 命令                                      |
|-----------------------------|-------------------------------------------|
| 找程序为什么起不来          | `strace ./prog 2>&1 \| head -50`           |
| 追踪文件访问路径            | `strace -e trace=openat,access ./prog`    |
| 看网络调用                  | `strace -e trace=network ./prog`          |
| 定位进程卡在哪个调用        | `strace -p <pid>` 看最后一行              |
| 统计系统调用热点            | `strace -c ./prog`                        |

### ltrace 的原理

`ltrace` 追踪的是**库函数调用**（通过 PLT/GOT hook），不是系统调用。它拦截 glibc 函数（如 `malloc`, `printf`）而不是内核调用。两者互补。

**新手误区**：`ltrace` 在某些较新的 glibc 版本或特定编译选项下可能无法正常工作（因为 symbol binding 变化），而 strace 基于 ptrace 更底层，更可靠。

## 2.7 ABI 稳定性："don't break userspace"深度解析

### 什么是系统调用层面的 ABI

ABI（Application Binary Interface）在系统调用层面包含：
- 系统调用编号（不变）
- 参数个数和类型约定（不变）
- 返回值和 errno 语义（不变）
- 数据结构布局（如 `stat` 结构体字段偏移，不变）

**反例**：`ioctl` 命令码（实际上并没有被很好地约束，不同驱动可以定义自己的码）。

### 内核如何维护 ABI

内核维护了 `linux/abi.h`、`UAPI`（User API）目录（待核：路径大致在 `include/uapi/`），专门存放"暴露给用户态"的头文件。这部分头文件是高度稳定的，任何修改都需要严格审查。

有一个工具 `abigail` 和内核的 `kABI` 机制专门追踪 kernel module ABI 稳定性（这是企业发行版如 RHEL 的强需求）——注意区分：**系统调用 ABI**（绝对不变）vs **内核模块 ABI**（RHEL 保证，upstream 不保证）。

### 实际的 ABI 演进手段

当需要新功能时，内核的策略：
1. **新系统调用**：`openat2` 取代 `open`/`openat`，`clone3` 取代 `clone`（新接口可以修复旧接口的设计缺陷）
2. **flags 扩展**：旧系统调用加新 flags 位（但未知 flags 必须返回 `-EINVAL`，防止旧内核运行新程序时静默忽略）
3. **版本化结构体**：新结构体加 `size` 字段，内核根据 `size` 决定支持到哪个版本（`clone3` 的 `clone_args` 就是这么做的）

## 2.8 seccomp：系统调用的防火墙

### 为什么需要 seccomp

进程的权限等于其能调用的系统调用集合。一个 bug（特别是 CVE）的可利用性，往往取决于攻击者能通过系统调用做什么。如果把进程能调的系统调用限制到最小集（只有 read/write/exit），即使有 RCE 也难以横向移动。

这是容器、沙箱、浏览器 renderer 进程的核心安全机制。

### seccomp 的两种模式

**模式 1：strict（极少用）**

```c
prctl(PR_SET_SECCOMP, SECCOMP_MODE_STRICT);
// 进程此后只能调用 read/write/exit/_exit/sigreturn
```

太严格，几乎没有实用价值。

**模式 2：seccomp-bpf（主流）**

```c
struct sock_fprog prog = {
    .len = ARRAY_SIZE(filter),
    .filter = filter,  // BPF 字节码
};
prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0);  // 必须先设置
prctl(PR_SET_SECCOMP, SECCOMP_MODE_FILTER, &prog);
```

用 BPF（classic BPF，不是 eBPF）写一个过滤器，在每次系统调用时执行：

```
seccomp-bpf 数据流

每次系统调用时，内核传给 BPF 程序：
  struct seccomp_data {
      int nr;           // 系统调用号
      __u32 arch;       // 架构（防止 ABI confusion 攻击）
      __u64 instruction_pointer;
      __u64 args[6];    // 参数
  };

BPF 程序返回动作（取所有 filter 中"最严格"的结果）：
  SECCOMP_RET_KILL_PROCESS → 杀整个进程（4.14+）
  SECCOMP_RET_KILL_THREAD  → 只杀触发的线程（即老的 SECCOMP_RET_KILL！）
  SECCOMP_RET_TRAP    → 发 SIGSYS（可在 handler 里捕获，能拿到被拦的调用号）
  SECCOMP_RET_ERRNO   → 不进内核，直接返回指定 errno（最常用的"软封锁"）
  SECCOMP_RET_TRACE   → 通知 ptrace tracer（无 tracer 时表现为 -ENOSYS）
  SECCOMP_RET_USER_NOTIF → 通知用户态 supervisor（5.0 引入，5.9 起 addfd 才好用）
  SECCOMP_RET_ALLOW   → 放行
```

> 💡 **坑点（KILL 杀的是线程不是进程）**：老接口 `SECCOMP_RET_KILL` 实际等价于 `SECCOMP_RET_KILL_THREAD`——**只杀触发非法调用的那个线程**，多线程进程的其余线程继续跑，往往把进程留在不一致甚至损坏的状态。"杀进程"语义要到 4.14 的 `SECCOMP_RET_KILL_PROCESS` 才有。很多人想当然以为 `RET_KILL` 杀整个进程，这是经典误解。另外，被 `RET_KILL_*` 终止时进程"如同收到 SIGSYS"，但这个信号不可被捕获/忽略（区别于 `RET_TRAP` 那个可捕获的 SIGSYS）。

### 真实案例：Docker 的 seccomp profile

Docker 默认开启 seccomp，有一个白名单 profile（约 300+ 条允许的系统调用），blocklist 包括 `ptrace`、`mount`、`reboot` 等危险调用。

```bash
# 查看容器是否有 seccomp 限制
docker inspect <container> | jq '.[].HostConfig.SecurityOpt'

# 以无 seccomp 运行（不推荐）
docker run --security-opt seccomp=unconfined ...

# 自定义 profile
docker run --security-opt seccomp=/path/to/profile.json ...
```

💡 **坑点（架构混淆攻击）**：64 位进程可以切换到 32 位模式（`int 0x80`）绕过基于编号的 seccomp 过滤——因为 32 位的 `write`(4) 和 64 位的 `stat`(4) 编号相同但语义不同。所以 seccomp filter 必须检查 `arch` 字段，先确认架构再检查编号。Docker 的 profile 有这个检查；自己写 filter 的人经常漏。

### seccomp-notify：用户态 supervisor

近年内核（约 5.0+ 引入，5.9 完善）的 `SECCOMP_RET_USER_NOTIF` 允许将被过滤的系统调用转发到用户态监督进程处理，而不是直接 kill/error：

```
进程 ──── syscall ──► seccomp filter
                              │
               SECCOMP_RET_USER_NOTIF
                              │
                      用户态 supervisor
                       检查/模拟/代理
                              │
                        回写决策
```

这是容器 rootless 运行的关键技术之一（无需真正内核权限，用 supervisor 代理危险调用）。

## 2.9 现代内核的系统调用 fast path

### 内核旁路（Kernel Bypass）

DPDK、RDMA、`io_uring` 都是不同层次的"绕开系统调用热路径"方案：

- **DPDK**：完全不经过内核网络栈，用户态轮询 NIC
- **io_uring**：通过共享内存 ring buffer 批提交，减少每操作一次 syscall 的开销（见近年内核 5.1+ 引入的 `io_uring`）
- **`vDSO`**：高频只读数据不进内核

这些技术都指向同一个工程结论：**系统调用的最大开销不是权限切换本身，而是 cache miss、TLB flush（Meltdown 修复加剧了这一点）和 CPU 流水线代价**。

### Meltdown/Spectre 之后的代价

Meltdown（CVE-2017-5754）修复引入了 KPTI（Kernel Page-Table Isolation）：用户态和内核态用不同的页表，系统调用时需要切换页表（CR3 写）并 flush TLB。这使得系统调用的 overhead 在某些场景增加了 30-100% 以上（具体数字依平台和 workload 而定）。

PCID（Process Context IDentifiers）可以部分缓解 TLB flush 问题（CPU 支持时内核会自动用）。

### 排障场景示范

**场景：进程无响应，但 CPU 不高**
```bash

# 如果只输出一行且不动，说明阻塞在该调用
```

**场景：找进程打开了哪些文件**

**场景：perf 统计系统调用热点**
```bash

# 只追踪特定 pid 的 write 调用

```

## 章末五件套

### 高频面试题（Staff 级）

**Q1：解释 x86-64 上 `syscall` 指令的完整执行流程。**

答题框架：MSR 预写入（LSTAR/STAR/SFMASK）→ `syscall` 指令保存 RIP/RFLAGS → 切换 CS → 跳至内核入口 → `swapgs` → 切换内核栈 → 保存寄存器（`pt_regs`）→ 查 `sys_call_table` → 执行 → `sysret` 回用户态。
陷阱：不要混淆 `int 0x80`（32位 IDT 路径）和 `syscall`（64位 MSR 路径）。

**Q2：为什么 strace 会让程序变慢？`perf trace` 和 strace 有何本质区别？**

答题框架：strace 用 `ptrace(PTRACE_SYSCALL)`，每个调用两次 SIGTRAP + 上下文切换到 strace 进程。`perf trace` 基于 tracepoint（ringbuffer + poll），只在 CPU 上的 perf ring buffer 里记录，不强制切换到 tracer，开销低得多。
加分：提 Spectre 修复后系统调用本身的开销变化（KPTI/TLB flush）。

**Q3：什么是 vDSO，它解决了什么问题，为什么安全？**

答题框架：内核将时间数据映射到只读用户页面（vvar），vDSO 代码用 seqlock 读，完全不进内核。安全性：只读映射 + seqlock 一致性保证。说明 strace 看不到 vDSO 调用的原因。
陷阱：有人说"vDSO 是用户态代码"——是的，但数据由内核更新，用户只读。

**Q4：系统调用号能修改吗？你如何给内核添加一个新的系统调用？**

答题框架：已发布的编号不能改（ABI 契约）。添加新 syscall：在 `arch/x86/entry/syscalls/syscall_64.tbl` 加一行（号码 + 名字 + entry 名）→ 用 `SYSCALL_DEFINEn(name, ...)` 实现函数体 → x86-64 的 `__NR_xxx` 宏和 wrapper 声明由构建系统从 `.tbl` **自动生成**（生成到 `asm/unistd_64.h` 等，不是手改 `unistd.h`）→ 若要支持 32 位进程则还得在 `syscall_32.tbl` 加项（可能需要 compat wrapper）。最后更新 `man-pages` 与 selftests。
进阶：`SYSCALL_DEFINEn` 宏（`n` = 参数个数，0~6）做了什么——它生成内部实现函数，再生成一层 arch 专属 thunk（x86-64 上是 `__x64_sys_xxx`）：thunk 从 `pt_regs` 里按 ABI 顺序取出参数、做必要的符号/截断处理，再调真正实现。这层 wrapper 正是前面"4.17 后 dispatch 改调 `__x64_sys_xxx`"的产物，也顺手堵了参数寄存器高 32 位的一类信息泄露。

**Q5：解释 seccomp-bpf 的架构混淆攻击（architecture confusion）及其缓解。**

答题框架：64位进程通过 `int 0x80` 用 32位调用表，同一编号在两个表的含义不同，可以绕过只检查编号的 filter。缓解：filter 必须先检查 `seccomp_data.arch`，不匹配则 KILL。Docker profile 的实现作为例证。

**Q6：`errno` 的线程安全性和实现。**

答题框架：现代 glibc 中 `errno` 是 thread-local storage（`__thread`），每个线程有独立副本。宏 `errno` 展开为对 TLS 变量的引用。坑：信号处理器中使用系统调用会覆盖 errno，需在 handler 入口保存、出口恢复。

**Q7：`open` 和 `openat` 的区别，以及为什么引入 `openat` 是 ABI 演进的典型案例？**

答题框架：`openat` 引入 `dirfd` 参数，解决了 TOCTOU 竞争问题（`open` 的路径解析基于 cwd，`openat` 可以用固定的目录 fd 作为 anchor）。从 ABI 角度，这是"用新系统调用修复旧接口设计缺陷"的标准做法，`open` 永远不会删。

**Q8：io_uring 如何减少系统调用开销？**

答题框架：SQ/CQ ring buffer 在用户态和内核态共享。用户批量写提交队列后用一次 `io_uring_enter` 提交，甚至 sqpoll 模式下内核线程轮询，0次系统调用。本质：把 N 次 syscall 批化为 1 次甚至 0 次。
（详见近年内核 5.1+ 文档）

### 系统题

### 代码题

**题目**：写一个 Linux C 程序，用 seccomp-bpf 把自己限制为只能调用 `write`、`exit_group`，然后验证其他系统调用被阻断。

```c
// syscall_jail.c
// 注意：需要 Linux，libseccomp 或手写 BPF filter

#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <sys/prctl.h>
#include <linux/seccomp.h>
#include <linux/filter.h>
#include <linux/audit.h>
#include <sys/syscall.h>
#include <errno.h>

/* 最小 BPF filter：只允许 write 和 exit_group */
struct sock_filter filter[] = {
    /* 检查架构（x86-64） */
    BPF_STMT(BPF_LD | BPF_W | BPF_ABS,
             (offsetof(struct seccomp_data, arch))),
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K,
             AUDIT_ARCH_X86_64, 1, 0),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_KILL),

    /* 加载 syscall 号 */
    BPF_STMT(BPF_LD | BPF_W | BPF_ABS,
             (offsetof(struct seccomp_data, nr))),

    /* 允许 write */
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_write, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),

    /* 允许 exit_group */
    BPF_JUMP(BPF_JMP | BPF_JEQ | BPF_K, __NR_exit_group, 0, 1),
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ALLOW),

    /* 其他 → 返回 EPERM */
    BPF_STMT(BPF_RET | BPF_K, SECCOMP_RET_ERRNO | EPERM),
};

struct sock_fprog prog = {
    .len = sizeof(filter) / sizeof(filter[0]),
    .filter = filter,
};

int main(void) {
    /* 必须先设置 NO_NEW_PRIVS */
    if (prctl(PR_SET_NO_NEW_PRIVS, 1, 0, 0, 0) < 0) {
        perror("prctl NO_NEW_PRIVS");
        return 1;
    }

    /* 安装 seccomp filter */
    if (prctl(PR_SET_SECCOMP, SECCOMP_MODE_FILTER, &prog) < 0) {
        perror("prctl SECCOMP");
        return 1;
    }

    /* 此后只有 write/exit_group 可用 */

    /* 这个可以 */
    write(STDOUT_FILENO, "seccomp jail active\n", 20);

    /* TODO: 尝试一个被封锁的调用，验证行为 */
    /* int fd = open("/tmp/test", O_RDONLY); // 应该返回 EPERM */
    /* 取消注释并观察行为 */

    return 0;
}
```

**编译与验证**：
```bash
gcc -o syscall_jail syscall_jail.c
strace ./syscall_jail
# 期望：看到 write 成功，open 返回 -1 EPERM（如果取消注释）

# 如果改 SECCOMP_RET_ERRNO 为 SECCOMP_RET_KILL，open 会触发 SIGSYS 杀进程
```

**暗坑提示**：
1. `offsetof(struct seccomp_data, ...)` 需要包含 `<linux/seccomp.h>`，不同内核版本字段偏移可能有差异——用宏而不是硬编码数字
2. 必须先 `PR_SET_NO_NEW_PRIVS` 再 `PR_SET_SECCOMP`，顺序不能反
3. BPF filter 是以 seccomp_data 为输入的 classic BPF，不是 eBPF——寄存器模型和语法不同
4. 在容器里可能已有 seccomp filter，多层叠加时取"更严格"
5. **你封 `open`，但程序根本不调 `open`(2)**：现代 glibc 的 `open()`/`fopen()` 在 x86-64 上实际发的是 `openat`(257) 而不是 `open`(2)；类似地 `stat()` 走 `newfstatat`/`statx`，`fork()` 走 `clone`。所以"按 libc 函数名去 allowlist syscall"经常对不上号——先用 `strace -f` 确认程序真正发出的是哪个 syscall，再写 filter。这也是为什么真实 profile（如 Docker 的）是按 syscall 号、不是按 libc 函数维护的。

## 本章心法

> 系统调用是隔离墙上唯一的门——`syscall` 指令是钥匙，内核入口的 dispatch（无论是历史的 `sys_call_table` 还是今天的 switch 分发）是锁芯，`vDSO` 是门边挂着的"免进"版本，`seccomp` 是门卫的白名单。不理解这四件事，就不真正理解 Linux 安全模型。
