> **一句话定位**：LKM（Loadable Kernel Module）是在不重启不重编内核的前提下向内核注入代码的机制，是驱动开发、内核扩展与学习内核的最低摩擦入口。

---

## TL;DR — 本章最硬结论

1. **模块就是共享内核地址空间的 `.ko` 文件**，`insmod` 做的是重定位 + 符号解析 + 调用 `module_init`，本质类似动态链接但在特权级 0 执行。
2. **字符设备的核心是 `file_operations` 表**，一切读写 ioctl 都是函数指针分发；`cdev` 把 (major, minor) 与这张表绑定。
3. **`rmmod` 安全的唯一保障是引用计数**，`module_get`/`module_put` + 打开文件的隐式引用是主要来源；漏掉任何一个，rmmod 要么被拒要么 kernel panic。
4. **printk 是调试的底线，动态调试（`pr_debug` + `dyndbg`）是生产可用的版本**；kgdb 用于复杂控制流，ftrace function_graph 是性能分析首选。
5. **从模块到 upstream 的距离是社区流程，不是技术难度**：代码质量/提交格式/patch 系列完整度是真正的门槛，而非算法复杂度。

## 25.1 背景：为什么需要 LKM

### 问题的起点

早期（2.4 之前）驱动必须编进内核镜像：改一个打印机驱动需要重编内核、重启服务器。随着设备种类爆炸，静态内核的二进制体积和维护成本变得不可接受。

LKM 解决三个实际问题：

| 问题 | LKM 的解决方式 |
|------|----------------|
| 驱动迭代周期长 | 不重启加载新代码 |
| 内核镜像臃肿 | 按需加载，不用不占内存 |
| 内核开发门槛高 | 模块可独立编译，开发-测试循环快 |

**工程推论**：
1. LKM 共享内核地址空间，没有任何保护边界——一个有 bug 的模块能 panic 整个系统。这是它与用户态插件（.so）最本质的区别。
2. 模块不是沙箱，是内核的一部分；"可动态加载"不意味着"隔离安全"。

## 25.2 演进：从静态链接到模块化内核

```
Linux 0.x / 1.x   — 全静态，驱动编入 vmlinuz
Linux 2.0          — 实验性模块支持，insmod/rmmod 原型
Linux 2.4          — 模块计数机制初步稳定
Linux 2.6.0        — kbuild 模块构建系统标准化；缺 MODULE_LICENSE 不再被拒，
                     但会给内核打 taint 标记并告警（不是硬性拒绝加载）
Linux 2.6.x 中期   — EXPORT_SYMBOL_GPL 开始区分 GPL 与非 GPL ABI
Linux 3.7          — 模块签名验证（CONFIG_MODULE_SIG）合入主线（注意：是 3.7，不是 4.x）
Linux 4.x / 5.x    — DKMS / kmod 工具链成熟，Secure Boot + 模块签名联动收紧
                     （lockdown LSM 后期把"未签名模块"与 Secure Boot 强绑定）
近年内核           — Rust for Linux：第一批 Rust 内核基础设施合入（6.1，2022 年底）
```

**关键转折**：2.6 的 kbuild 统一了 out-of-tree 模块构建语义（`make -C /lib/modules/$(uname -r)/build M=$(pwd)`），使驱动开发者不再需要维护魔改的 Makefile。这个接口至今基本稳定。

## 25.3 LKM 机制：insmod 到 module_init 的完整路径

### 25.3.1 `.ko` 文件格式

`.ko` 是标准 ELF 可重定位文件（ET_REL），与普通 `.o` 的区别在于额外的几个 section：

```
.modinfo        — MODULE_LICENSE/MODULE_AUTHOR/MODULE_DESCRIPTION 字符串
__ksymtab       — 本模块导出符号表（EXPORT_SYMBOL）
__kcrctab       — CRC 校验，用于符号版本匹配
.gnu.linkonce.this_module — struct module 实例
```

### 25.3.2 insmod 的工作流（系统调用视角）

```
用户态 insmod / modprobe
        │
        ▼ read .ko 到用户态内存
        │
        ▼ syscall: finit_module(fd, params, flags)
              （老接口 init_module 直接传内存）
        │
        ▼ 内核：kernel_read_file → load_module()
              │
              ├─ 1. 为模块各 section 分配内核内存（走 module_alloc/execmem 类分配器，
              │      落在为模块预留的地址区间，不是 kmalloc 的直接映射区）
              ├─ 2. ELF section 拷贝到上一步分配的内核地址空间
              ├─ 3. 符号解析：查 kernel 导出符号表 + 已加载模块符号表
              ├─ 4. 重定位（apply_relocations）
              ├─ 5. 校验：版本/CRC、签名（按策略可拒绝加载）、读取 MODULE_LICENSE
              │      （license 缺失/非 GPL 不拒绝加载，但会给内核打 taint 标记）
              ├─ 6. 把模块加入全局 modules 链表
              └─ 7. 调用 module->init()  ← 就是你的 module_init 函数
```

> 💡 坑点：模块代码段不在内核直接映射区（`kmalloc` 那块线性映射），而是放在专门的模块地址区间（x86-64 上是内核地址空间高端一段为模块预留的 vmalloc 类区域）。所以模块函数地址跟 `vmlinux` 正文不在同一段，但都能被 kallsyms 解析——因为 `.ko` 加载时其符号会被注册进 kallsyms。
>
> ⚠️ staff 级补充：**`kallsyms_lookup_name()` 自 5.7（2020）起不再 `EXPORT_SYMBOL`**，模块已经不能直接调用它了——这是内核团队故意收紧的（防止 out-of-tree 模块靠按名字找未导出符号来"借用"内部 API、绕过 `EXPORT_SYMBOL_GPL` 的边界）。这里拿它当例子是解释 kallsyms 机制，不是建议你在模块里调它。需要 hook 未导出函数的正路是 kprobe/ftrace，不是 `kallsyms_lookup_name` + 强转函数指针。

### 25.3.3 module_init / module_exit 宏展开

```c
// 典型骨架
#include <linux/init.h>
#include <linux/module.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Your Name");
MODULE_DESCRIPTION("A minimal example");

static int __init mymod_init(void)
{
    pr_info("mymod: loaded\n");
    return 0;   /* 非 0 = 加载失败，模块被卸载 */
}

static void __exit mymod_exit(void)
{
    pr_info("mymod: unloaded\n");
}

module_init(mymod_init);
module_exit(mymod_exit);
```

`__init` / `__exit` 是 section 标注：
- `__init` 函数放入 `.init.text`。**内置驱动和可加载模块两种场景，init 段在初始化完成后都会被回收**——内置场景由内核启动后期统一 `free_initmem` 释放（`dmesg` 里那行 `Freeing unused kernel image (initmem) memory` 就是它）；模块场景由 module loader 在 `init` 函数返回后释放 init layout。所以这不是"习惯标注"，是真有内存收益：常见误区是以为模块的 `__init` 段会一直驻留。
- `__exit` 放入 `.exit.text`，若模块内置进内核（不可卸载），该段在链接期就被丢弃（内置模块永远不会卸载，`exit` 代码是死代码）。

> 💡 推论：`__init` 段被释放后地址不可再访问。如果你在某个常驻函数里保存了 `__init` 函数的指针、或让 timer/workqueue 回调指向 `__init` 函数，初始化后调用就是 use-after-free。`__exit` 函数同理不能被 `__init` 之外的常驻路径引用——`modpost` 会用 `section mismatch` 警告抓这类错误，**不要忽略这个警告**。

**工程推论**：
1. `module_init` 返回非零 → 模块不会出现在 `lsmod`，已注册的资源需在 `init` 内部回滚，不要依赖 `exit` 被调用。
2. `exit` 函数绝对不能返回错误——内核不知道如何处理卸载失败。如果资源实在无法安全释放，在 `exit` 里 `BUG()` 比悄悄泄漏强。

### 25.3.4 符号导出与 GPL 边界

```c
EXPORT_SYMBOL(my_func);         // 任何模块可用
EXPORT_SYMBOL_GPL(my_func);     // 仅 MODULE_LICENSE("GPL") 模块可用
EXPORT_SYMBOL_GPL_FUTURE(my_func); // 废弃，勿用
```

`EXPORT_SYMBOL_GPL` 的法律/工程意义：
- 把内核 ABI 中对"GPL 意图"敏感的接口圈起来，非 GPL 模块（闭源驱动）无法调用
- 典型例子：`register_kprobe`、大量内核子系统内部 API
- **不是技术保护，是社区契约**：闭源驱动用 EXPORT_SYMBOL_GPL 接口在法律上是灰色地带

> 🎯 面试加分点：问"为什么 NVIDIA 长期不开源驱动还能运行"——答案是他们用 shim 模块（开源）包装闭源 blob，利用非 GPL 导出符号的空间。这个空间在逐渐收窄。

## 25.4 字符设备完整实现

### 25.4.1 核心数据结构关系

```
应用层 open("/dev/mydev", ...)
        │
        ▼
VFS: file 结构体
  └── file->f_op  ──────────────────────────────────┐
                                                      │
内核设备层：                                          │
  struct cdev {                                       │
      struct kobject kobj;       ← sysfs 节点锚点    │
      struct module *owner;      ← 引用计数          │
      const struct file_operations *ops; ◄────────────┘
      struct list_head list;
      dev_t dev;                 ← (major << 20 | minor)
      unsigned int count;        ← minor range
  };
```

`dev_t` 编码：内核内部的 `dev_t` 是 32 位，**高 12 位 major + 低 20 位 minor**（`MINORBITS == 20`，`MINORMASK == (1<<20)-1`）。这是体系结构无关的内核内部格式，不是 x86-64 专属。注意区分：用户态 glibc 的 `dev_t` 是 64 位、`makedev()` 的位布局与内核内部布局不同，所以不要把用户态看到的设备号位运算直接套到内核侧。`MAJOR()/MINOR()/MKDEV()` 宏定义在 `include/linux/kdev_t.h`。

### 25.4.2 注册流程：三步走

```c
/* 步骤 1：分配设备号 */
dev_t devno;
int major;

// 静态分配（不推荐，易冲突）
// register_chrdev_region(devno, count, "mydev");

// 动态分配（推荐）
alloc_chrdev_region(&devno, 0/*first_minor*/, 1/*count*/, "mydev");
major = MAJOR(devno);

/* 步骤 2：初始化 cdev 并绑定 file_operations */
struct cdev my_cdev;
cdev_init(&my_cdev, &my_fops);
my_cdev.owner = THIS_MODULE;
cdev_add(&my_cdev, devno, 1);  // 返回 0 = 成功，此时设备可被 open

/* 步骤 3：在 /dev 创建节点（依赖 udev/mdev 自动化，或手动 mknod） */
struct class *my_class = class_create(THIS_MODULE, "mydev");
device_create(my_class, NULL, devno, NULL, "mydev%d", 0);
```

> 💡 坑点：`cdev_add` 成功后设备立即可被用户态 open，**在此之前必须确保所有 `file_operations` 内部状态已初始化**。很多初学者在 `cdev_add` 之后才初始化锁/队列，导致并发 open 时竞态。

### 25.4.3 file_operations 完整实现骨架

```c
static struct file_operations my_fops = {
    .owner   = THIS_MODULE,
    .open    = my_open,
    .release = my_release,
    .read    = my_read,
    .write   = my_write,
    .unlocked_ioctl = my_ioctl,  // 注意：旧的 .ioctl（BKL 时代）早已移除，用 .unlocked_ioctl
    .llseek  = no_llseek,        // 见下方版本提示；不可 seek 的设备表达"禁止 seek"
};

static int my_open(struct inode *inode, struct file *filp)
{
    /* inode 是磁盘/设备节点的抽象，file 是本次 open 的实例 */
    /* filp->private_data 是每个 open 实例的私有指针，常用于存设备状态 */
    struct my_dev *dev = container_of(inode->i_cdev, struct my_dev, cdev);
    filp->private_data = dev;
    return 0;
}

static ssize_t my_read(struct file *filp, char __user *buf,
                       size_t count, loff_t *f_pos)
{
    struct my_dev *dev = filp->private_data;
    /* copy_to_user：从内核缓冲区 → 用户态，返回未拷贝字节数 */
    if (copy_to_user(buf, dev->data + *f_pos, count))
        return -EFAULT;
    *f_pos += count;
    return count;
}
```

> ⏱️ 版本提示（`.llseek = no_llseek`）：辅助函数 `no_llseek` 在 **6.12（2024 年底）被移除**。它的语义被收进了 VFS 默认路径：现在表达"这个设备不可 seek"的正规做法是在 `.open` 里调用 `nonseekable_open(inode, filp)`，`.llseek` 留空即可。在 6.12+ 上继续写 `.llseek = no_llseek` 会编译失败（符号不存在）。可 seek 的设备仍用 `default_llseek` 或自定义实现。这是"教程长期抄旧 API"的典型坑——上面骨架保留 `no_llseek` 只为对照旧代码，新写代码请按本提示来。

**`copy_to_user` / `copy_from_user` 的工程意义**：
- 用户态指针可能是 NULL、非法地址、或指向只读映射，直接 dereference 会 oops
- 这两个函数在内部做页表检查，失败返回未拷贝字节数（非 0 即错误）
- **绝不能在持有 spinlock 的情况下调用**（可能睡眠）

### 25.4.4 ioctl 设计

```c
/* _IOW/_IOR/_IOWR 宏生成唯一 ioctl 命令号 */
#define MY_MAGIC  'k'
#define MY_SET_VAL   _IOW(MY_MAGIC, 1, int)
#define MY_GET_VAL   _IOR(MY_MAGIC, 2, int)

static long my_ioctl(struct file *filp, unsigned int cmd, unsigned long arg)
{
    int val;
    switch (cmd) {
    case MY_SET_VAL:
        if (copy_from_user(&val, (int __user *)arg, sizeof(int)))
            return -EFAULT;
        dev->val = val;
        break;
    case MY_GET_VAL:
        if (copy_to_user((int __user *)arg, &dev->val, sizeof(int)))
            return -EFAULT;
        break;
    default:
        return -ENOTTY;   /* 标准：未知 ioctl 返回 ENOTTY，不要返回 EINVAL */
    }
    return 0;
}
```

> 🎯 面试加分点：asm-generic 的 ioctl 命令号是 32 位位域，从低到高依次是 **nr(8) | type(8) | size(14) | dir(2)**——即 `_IOC_NRBITS=8 / _IOC_TYPEBITS=8 / _IOC_SIZEBITS=14 / _IOC_DIRBITS=2`，所以 `(type<<8 | nr) | (size<<16) | (direction<<30)`。`_IO/_IOW/_IOR/_IOWR` 只是按 dir 填这些位的宏。两个工程含义：(1) magic `type` byte 跨子系统防撞，新加 ioctl 要去 `Documentation/userspace-api/ioctl/ioctl-number.rst` 登记;(2) size 段只有 14 位 → 单个 ioctl 的 arg 结构体大小被限制在 `1<<14 == 16KB` 量级，更大的数据要传指针而非内联。注意少数老架构（如历史上的 alpha/mips/powerpc/sparc）有自己的 ioctl 位布局，不走 asm-generic——但你新写的通用驱动按 asm-generic 即可。位域定义见 `include/uapi/asm-generic/ioctl.h`。

## 25.5 proc / sysfs / debugfs 接口

三者定位不同，选错接口是常见设计错误：

```
┌─────────────┬──────────────────┬───────────────────┬──────────────────────┐
│             │   /proc          │   /sys            │   /debugfs           │
├─────────────┼──────────────────┼───────────────────┼──────────────────────┤
│ 历史来源    │ 进程信息（原始） │ 设备模型（统一）  │ 调试用途（2.6.10+） │
│ 语义        │ 每文件自由格式   │ 一文件一值（规范）│ 任意，无稳定性保证  │
│ ABI 稳定性  │ 准稳定（有工具依赖│ 用户态 ABI        │ 明确不稳定           │
│ 使用场景    │ 进程/系统信息    │ 设备参数/驱动属性 │ 内核调试/dump/trace │
│ 挂载点      │ /proc            │ /sys              │ /sys/kernel/debug    │
└─────────────┴──────────────────┴───────────────────┴──────────────────────┘
```

### 25.5.1 procfs

```c
#include <linux/proc_fs.h>

static struct proc_dir_entry *my_proc;

static int my_proc_show(struct seq_file *m, void *v)
{
    seq_printf(m, "counter=%d\n", my_counter);
    return 0;
}

/* single_open 是最简洁的 read-only proc 接口 */
static int my_proc_open(struct inode *inode, struct file *file)
{
    return single_open(file, my_proc_show, NULL);
}

static const struct proc_ops my_proc_fops = {  // 近年内核用 proc_ops 替换 file_operations
    .proc_open    = my_proc_open,
    .proc_read    = seq_read,
    .proc_lseek   = seq_lseek,
    .proc_release = single_release,
};

// 在 init 中：
my_proc = proc_create("mydev", 0444, NULL, &my_proc_fops);
// 在 exit 中：
remove_proc_entry("mydev", NULL);
```

> 💡 坑点：proc 文件删除时若有进程正在读，`remove_proc_entry` 会等待——这是内核的引用计数保护。但如果读路径持有你的设备锁，而 exit 路径也试图获取该锁，就死锁了。

### 25.5.2 sysfs

sysfs 属性绑定在 `kobject` 上（见第 19 章），对驱动开发者通常通过 `device_attribute` 暴露：

```c
#include <linux/sysfs.h>
#include <linux/device.h>

static ssize_t myval_show(struct device *dev, struct device_attribute *attr,
                          char *buf)
{
    return sysfs_emit(buf, "%d\n", my_val);  // sysfs_emit 是 scnprintf 的安全封装
}

static ssize_t myval_store(struct device *dev, struct device_attribute *attr,
                           const char *buf, size_t count)
{
    int val;
    if (kstrtoint(buf, 10, &val))  // kstrtoint: 安全的字符串→整数，替代 atoi
        return -EINVAL;
    my_val = val;
    return count;
}

static DEVICE_ATTR_RW(myval);   // 展开为 struct device_attribute dev_attr_myval

// 在 init 中：
device_create_file(my_device, &dev_attr_myval);
// 在 exit 中：
device_remove_file(my_device, &dev_attr_myval);
```

**sysfs 黄金规则**：一个 sysfs 文件只表达一个值，不要搞 JSON 或多行格式——那是 procfs 风格，违反 sysfs 设计哲学，会被 maintainer 打回来。

### 25.5.3 debugfs

```c
#include <linux/debugfs.h>

static struct dentry *my_debug_dir;

// init:
my_debug_dir = debugfs_create_dir("mydev", NULL);  // 在 /sys/kernel/debug/mydev/
debugfs_create_u32("counter", 0644, my_debug_dir, &my_counter);
debugfs_create_bool("enabled", 0644, my_debug_dir, &my_enabled);

// exit:
debugfs_remove_recursive(my_debug_dir);
```

debugfs 的设计哲学：**快速暴露，不用考虑格式稳定性**。这里有个反直觉但官方明确的约定：**debugfs 接口创建失败时，驱动不应该报错回滚，甚至不应该检查返回值**。`debugfs_create_dir/_file` 失败返回 `ERR_PTR`，而整个 debugfs API 被设计成 ERR_PTR-tolerant——把上一步的错误指针继续传给下一个 debugfs 调用是安全的（内部会识别并跳过）。理由是 debugfs 纯属调试辅助，它挂不上去（比如内核没编 `CONFIG_DEBUG_FS`）也绝不能让驱动主功能失败。所以你常看到正经驱动里 debugfs 调用一律不接返回值——这不是偷懒，是 Greg KH 当年明确表态的推荐写法。

> 对照记忆：cdev/class/device/sysfs 这些是**功能性**注册，失败必须逐级 `goto` 回滚（见 25.15 代码题的错误路径）；debugfs 是**调试性**注册，失败忽略。两类注册的错误处理范式相反，别套混。

## 25.6 内核态调试工具链

### 25.6.1 printk 与日志级别

```c
pr_emerg("...");    // KERN_EMERG   级别 0，系统不可用
pr_alert("...");    // KERN_ALERT   级别 1
pr_crit("...");     // KERN_CRIT    级别 2
pr_err("...");      // KERN_ERR     级别 3，驱动错误用这个
pr_warn("...");     // KERN_WARNING 级别 4
pr_notice("...");   // KERN_NOTICE  级别 5
pr_info("...");     // KERN_INFO    级别 6，正常加载/卸载信息
pr_debug("...");    // KERN_DEBUG   级别 7，默认不打印
```

**动态调试（dyndbg）**：`pr_debug` 在内核启用 `CONFIG_DYNAMIC_DEBUG` 后可运行时开关：

```bash
# 开启某模块所有 debug 打印
echo "module mymod +p" > /sys/kernel/debug/dynamic_debug/control
# 开启特定文件
echo "file mymod.c +p" > /sys/kernel/debug/dynamic_debug/control
# 关闭
echo "module mymod -p" > /sys/kernel/debug/dynamic_debug/control
```

这是生产环境调试的正确姿势——不重编模块，按需开关，开销极低（静态分支）。

### 25.6.2 kgdb — 内核源码级调试

kgdb 通过串口或 kgdboc 网络接口接入 GDB，可在另一台机器上做内核级断点调试：

```bash
# 被调试机：内核启动参数
kgdboc=ttyS0,115200 kgdbwait

# 调试机：
gdb vmlinux
(gdb) set remotebaud 115200
(gdb) target remote /dev/ttyS0
(gdb) break my_ioctl
(gdb) continue
```

> 新手误区：kgdb 调试时**不能**在被调试机上同时做其他操作——调试机发出命令后被调试机完全冻结（NMI watchdog 可能误触发）。生产环境慎用，开发 VM 首选。

### 25.6.3 ftrace — 内核函数追踪

```bash
# 追踪特定函数（function_graph 展示调用栈+耗时）
echo function_graph > /sys/kernel/debug/tracing/current_tracer
echo my_ioctl      > /sys/kernel/debug/tracing/set_graph_function
echo 1             > /sys/kernel/debug/tracing/tracing_on
# ... 执行操作 ...
echo 0             > /sys/kernel/debug/tracing/tracing_on
cat /sys/kernel/debug/tracing/trace
```

输出示例（示意）：
```
 # CPU  DURATION                  FUNCTION CALLS
 # |     |   |                     |   |   |   |
  1) + 12.345 us   |  my_ioctl() {
  1)   1.234 us    |    mutex_lock();
  1)   8.901 us    |    copy_from_user();
  1)   0.567 us    |    mutex_unlock();
  1) + 12.345 us   |  }
```

**ftrace 四个核心使用场景**：

| 场景 | tracer 选择 |
|------|-------------|
| 追踪函数调用图 | `function_graph` |
| 寻找延迟来源 | `irqsoff` / `preemptoff` / `wakeup` |
| 追踪特定函数调用 | `function` + `set_ftrace_filter` |
| 动态 hook（不改源码） | `kprobe_events` |

### 25.6.4 KASAN / UBSAN — 内存错误检测

- **KASAN**（`CONFIG_KASAN`）：检测 use-after-free、out-of-bounds，开发/CI 环境必开。默认是 generic KASAN（编译期插桩，每次内存访问前查 shadow memory）
- **UBSAN**（`CONFIG_UBSAN`）：检测未定义行为（整数溢出、移位越界、对象大小错位等）
- 开销（generic KASAN，量级、依内核配置与负载浮动）：**shadow memory 固定占约 1/8 物理内存**（每 8 字节内核内存配 1 字节 shadow），叠加 redzone + quarantine 后实际 RAM 占用更高；**运行速度约慢 2–3x**。明确不适合生产，专给测试/CI 内核用。
- 补充：还有 **SW_TAGS KASAN**（arm64，基于地址 tag，shadow 仅 1/16，开销更低）和 **HW_TAGS KASAN**（arm64 MTE 硬件，开销低到接近可在线上抽样开启）——名字别记混，generic 是最通用、x86 也能用的那个。

## 25.7 并发与生命周期陷阱

这是模块开发最容易出问题的地方，也是 code review 的重点。

### 25.7.1 模块引用计数

```
内核维护每个模块的引用计数：
  module->refcnt

增加引用的场景：
  1. 用户态进程 open /dev/mydev  → file 关闭时自动递减
  2. 其他模块调用 try_module_get(THIS_MODULE)
  3. 内核内部回调（timer、workqueue）传递了模块函数指针
```

**最常见的生命周期 bug**：

```c
// BUG: workqueue 里的回调用了模块的变量，但 rmmod 时 workqueue 还没执行完
static void my_work_handler(struct work_struct *work)
{
    // 此时模块可能已被 rmmod！
    dev->counter++;
}

// FIX: exit 中必须等 workqueue 完成
static void __exit mymod_exit(void)
{
    cancel_work_sync(&my_work);   // 等待并取消
    // 然后才能释放 dev
}
```

**常见悬挂引用场景**：

| 场景 | 修复方式 |
|------|----------|
| timer 回调中访问模块数据 | `del_timer_sync` 在 exit 中等待 |
| workqueue 任务 | `cancel_work_sync` / `flush_workqueue` |
| 中断处理函数 | `free_irq` 确保 handler 不再被调用 |
| kthread | `kthread_stop` |
| RCU 回调 | `rcu_barrier` 等待 grace period |

### 25.7.2 字符设备并发

```c
struct my_dev {
    struct cdev cdev;
    struct mutex lock;      // 保护 data
    char data[BUF_SIZE];
    int open_count;
    bool busy;              // 单用户语义：同时只允许一个进程 open
};

static int my_open(struct inode *inode, struct file *filp)
{
    struct my_dev *dev = container_of(inode->i_cdev, struct my_dev, cdev);

    mutex_lock(&dev->lock);
    if (dev->busy) {
        mutex_unlock(&dev->lock);
        return -EBUSY;
    }
    dev->busy = true;
    dev->open_count++;
    mutex_unlock(&dev->lock);

    filp->private_data = dev;
    return 0;
}

static int my_release(struct inode *inode, struct file *filp)
{
    struct my_dev *dev = filp->private_data;
    mutex_lock(&dev->lock);
    dev->busy = false;
    mutex_unlock(&dev->lock);
    return 0;
}
```

> 💡 坑点：不要在 `read`/`write` 里对整个设备加 mutex 然后调用 `copy_to_user`/`copy_from_user`——后者可能睡眠（page fault），在 mutex 持有期间睡眠是合法的（mutex 允许睡眠），但持有时间变长，性能差。更精细的做法是先持锁拷贝到内核局部 buffer，解锁，再 `copy_to_user`。

### 25.7.3 常见死锁模式

```
模式 1：双锁顺序不一致
  CPU0: lock(A) → lock(B)
  CPU1: lock(B) → lock(A)  ← 经典死锁

模式 2：在 softirq 上下文中持 mutex
  hardirq 禁止睡眠，softirq 也禁止睡眠，mutex 可睡眠
  → 在 timer 回调/tasklet 中调用 mutex_lock → BUG

模式 3：重入
  read() 持 lock → 调用某函数 → 该函数也 lock() → 死锁
  → 用 mutex_trylock 或改用 rwsem
```

（见第 12 章：锁与同步原语，详细锁层次分析）

## 25.8 Kbuild 构建系统

### 25.8.1 out-of-tree 模块 Makefile

```makefile
# Kbuild 文件（Kbuild 或 Makefile，两者皆可）
obj-m += mydev.o
mydev-objs := mydev_main.o mydev_ioctl.o  # 多文件模块

# 顶层 Makefile（调用内核构建系统）
KDIR ?= /lib/modules/$(shell uname -r)/build

all:
	$(MAKE) -C $(KDIR) M=$(PWD) modules

clean:
	$(MAKE) -C $(KDIR) M=$(PWD) clean

install:
	$(MAKE) -C $(KDIR) M=$(PWD) modules_install
	depmod -a
```

**关键点**：
- `obj-m` 表示构建为模块；`obj-y` 表示编进内核（用于 in-tree 开发）
- `M=$(PWD)` 告诉内核 Kbuild 系统模块源码在哪
- `KDIR` 指向正在运行内核的 build 目录，内含 `Module.symvers`（符号版本表）

### 25.8.2 符号版本与 Module.symvers

```
构建内核时生成：
  /lib/modules/$(uname -r)/build/Module.symvers
  格式：CRC   symbol_name   module_path   export_type

模块构建时，Kbuild 读取该文件，把依赖的内核符号 CRC 嵌入 .ko
insmod 时内核验证 CRC，不匹配 → "disagrees about version of symbol"
```

这是 `modprobe` 比 `insmod` 更智能的地方：modprobe 还解析 `depmod` 生成的依赖图，自动加载前置模块。

### 25.8.3 在内核树内开发（in-tree）

```
linux/
├── drivers/
│   └── mysubsystem/
│       ├── Kconfig      ← 新增 config 选项
│       ├── Makefile     ← obj-$(CONFIG_MY_DRIVER) += mydev.o
│       └── mydev.c
```

`Kconfig` 片段示例：
```kconfig
config MY_DRIVER
    tristate "My example driver"
    depends on X86 && PCI
    help
      This is a sample driver for demonstration.
      If unsure, say N.
```

`tristate`：y（内置）/ m（模块）/ n（不编）。依赖 `depends on` 确保配置合法性。

## 25.11 高频面试题

### 题 1：`insmod` 和 `modprobe` 的区别？

**答题框架**：
- `insmod`：调用 `finit_module` syscall，只加载指定 .ko，不解析依赖
- `modprobe`：先查 `depmod` 生成的模块依赖图（`/lib/modules/$(uname -r)/modules.dep`），自动加载前置模块，支持 `/etc/modprobe.conf` 别名和黑名单

**加分点**：`modprobe -r` 卸载时会递归卸载不再被依赖的模块；`insmod`/`rmmod` 不做依赖管理。

**陷阱**：不要说 `modprobe` 性能更好——两者最终都是 `finit_module` syscall，区别在用户态逻辑，不在内核加载速度。

---

### 题 2：字符设备 major/minor number 的设计哲学是什么？

**答题框架**：
- major：标识驱动类型（哪个驱动处理），VFS 用它找到对应的 `file_operations`
- minor：驱动内部区分具体设备实例（如 `/dev/sda` vs `/dev/sdb`）
- `alloc_chrdev_region` 动态分配 major 是现代做法，避免与历史上静态分配的固定 major 冲突（静态分配清单在内核源码树 `Documentation/admin-guide/devices.txt`，老内核里是 `Documentation/devices.txt`）

**加分点**：mention `MKDEV(major, minor)` / `MAJOR(dev)` / `MINOR(dev)` 宏；mention 内核内部 `dev_t` 是 32 位、高 12 位 major + 低 20 位 minor（`MINORBITS == 20`，体系结构无关）；并点出用户态 glibc 的 `dev_t` 是 64 位、位布局不同，跨内核/用户态边界时别混用位运算。

---

### 题 3：`EXPORT_SYMBOL` vs `EXPORT_SYMBOL_GPL` 的工程意义？

**答题框架**：
- 技术层面：两者都把符号加入 `__ksymtab`，区别在于加载时内核检查模块的 `MODULE_LICENSE`，GPL 限定符号对非 GPL 模块不可见
- 法律/社区层面：内核维护者用它划分 "内核内部 API" 与 "对外稳定接口" 的边界
- 工程实践：subsystem 内部函数优先用 `_GPL`，公开稳定接口用 `EXPORT_SYMBOL`；新增接口被 maintainer 要求用 `_GPL` 是惯例

**陷阱**：这不是技术保护机制，没有 sandbox 隔离；闭源模块用 shim 绕过是历史上存在的做法。

---

### 题 4：模块卸载（rmmod）为什么可能失败？如何保证 rmmod 安全？

**答题框架**：
- `rmmod` 失败通常因为 `module->refcnt > 0`：有进程持有设备文件、有其他模块依赖
- 引用计数来源：open 的文件描述符、`try_module_get`、未完成的 workqueue/timer 任务
- 保证安全：exit 函数必须 1) 停止接受新请求 2) 等待所有异步任务完成（`cancel_work_sync`/`del_timer_sync`）3) 注销所有接口 4) 释放内存

**加分点**：mention `module_put` 的对称调用；mention `THIS_MODULE` 宏在 `file_operations.owner` 的隐式引用计数作用。

---

### 题 5：内核模块中能用哪些内存分配函数？各自适用场景？

**答题框架**（见第 9 章）：

| 函数 | 分配区域 | 适用场景 | 限制 |
|------|----------|----------|------|
| `kmalloc(size, GFP_KERNEL)` | SLUB/SLAB，物理连续 | 小对象（< 几十KB） | 可睡眠，不能在中断/spinlock |
| `kmalloc(size, GFP_ATOMIC)` | 同上但不睡眠 | 中断/softirq 上下文 | 可能失败（不等待回收） |
| `vmalloc(size)` | vmalloc 区，虚拟连续 | 大缓冲区，不需物理连续 | 有 TLB 开销，不能 DMA |
| `get_zeroed_page` | 单页 | 需要页对齐的 4KB 缓冲 | 限于页大小倍数 |
| `dma_alloc_coherent` | DMA 区 | 设备 DMA 缓冲 | 需要 device 结构体 |

**陷阱**：`GFP_KERNEL` 可能睡眠，在 spinlock 持有期间绝不能用。

### 题 6：如何在不重启内核的情况下调试模块 bug？

**答题框架**：按严重程度递进：
1. **printk + dyndbg**：最低成本，运行时开关 `pr_debug`
2. **ftrace kprobe_events**：不改代码，动态插入探针，看参数/返回值
3. **eBPF kprobe**：更灵活，可写聚合逻辑（见第 23 章）
4. **KASAN + 测试内核**：隔离 VM 里开 KASAN，触发场景复现
5. **kgdb**：双机调试，适合复杂控制流

**加分点**：mention `crash` 工具分析 vmcore kdump 文件——生产环境 panic 后分析内核崩溃的标准工具链。

---

### 题 7：sysfs 和 procfs 的设计哲学差异，为什么 maintainer 要求新接口用 sysfs？

**答题框架**：
- procfs 原始设计是进程信息，后被滥用为通用内核-用户接口，格式随意（多值、多行、自由文本）
- sysfs "一文件一值" 规范保证了接口的可脚本化和语义清晰
- 用户态工具（udev、systemd）依赖 sysfs 统一接口做设备管理
- ABI 稳定性：sysfs 属性一旦发布，内核承诺不破坏；procfs 无此承诺

**陷阱**：不要说 procfs 被废弃了——`/proc/cpuinfo`、`/proc/meminfo` 等不会消失，新驱动接口只是不应该滥用 procfs。

---

### 题 8：`copy_to_user` / `copy_from_user` 为什么不能省略？

**答题框架**：
- 用户态指针可以是任意值：NULL、内核地址（试图访问内核数据）、已 unmap 的地址
- 直接 dereference 用户指针 → 可能 oops（无效地址）或安全漏洞（用户传内核地址来读内核数据）
- 这两个函数做 `access_ok` 检查（验证地址在用户空间范围内）+ 处理 page fault（地址合法但未映射时触发 COW/swap in）

**加分点**：mention SMEP/SMAP（x86 硬件特性，阻止内核直接执行/访问用户页），内核里直接解引用用户指针在开启 SMAP 的系统上会立即 fault。

---

### 题 9：（进阶）lockdep 是什么？它如何检测死锁？

**答题框架**：
- lockdep 是内核的运行时锁依赖图验证器（`CONFIG_PROVE_LOCKING`）
- 工作原理：记录所有 "在持有锁 A 时获取锁 B" 的边，构建全局有向图；检测环即潜在死锁
- 输出：触发 `WARNING: possible circular locking dependency detected`，附调用栈
- 开销：显著（~2x），只用于开发/测试内核

**加分点**：lockdep 还检测 "在中断上下文中获取非中断安全锁"（`HARDIRQ-safe` / `HARDIRQ-unsafe` 分类），这是一类比死锁更难用测试覆盖的 bug。

---

### 题 10：（staff 级）Rust for Linux 对内核模块开发意味着什么？

**答题框架**：
- C 内核模块的三类常见 bug：use-after-free（生命周期）、空指针（nullability）、数据竞争（并发）——Rust 类型系统在编译期消除前两类，Rust 的 `Send`/`Sync` 约束在编译期控制第三类
- Rust 内核 API 用 RAII wrapper 封装 C 原语：`Mutex<T>` 强制访问时必须持锁，资源释放在 Drop 中保证
- 当前挑战：Rust 内核 ABI 尚不稳定（近年内核仍在演进），现有 C 代码的 safe wrapper 覆盖率有限，混合语言调试工具链不成熟

## 25.12 实战项目：P1 收口

**目标**：一个带 sysfs 接口、并发安全的字符设备模块

**功能规格**：
- 内核缓冲区读写（环形 buffer 或简单 byte buffer）
- ioctl 接口：清空 buffer / 查询当前大小
- sysfs 属性：`/sys/class/mydev/mydev0/buf_size`（可读写，写时改变 buffer 限制）
- 单用户语义：同时只允许一个进程 open，第二个返回 EBUSY
- `rmmod` 时若有进程持有设备文件，rmmod 被拒（`-EBUSY`），不 oops

**验收步骤**：
```bash
# 1. 构建加载
make && sudo insmod mydev.ko
lsmod | grep mydev       # 应看到 mydev
ls /dev/mydev0           # 设备节点存在
ls /sys/class/mydev/mydev0/buf_size  # sysfs 属性存在

# 2. 基本读写
echo "hello" > /dev/mydev0
cat /dev/mydev0   # 应输出 hello

# 3. ioctl 测试（用自写测试程序或 Python ctypes）
./test_ioctl /dev/mydev0 clear

# 4. 并发测试
for i in $(seq 1 10); do cat /dev/mydev0 & done
# 应看到 EBUSY 错误，不应 oops

# 5. rmmod 安全测试
exec 3</dev/mydev0 &   # 保持一个 fd 打开
sudo rmmod mydev       # 应返回 EBUSY
exec 3>&-              # 关闭 fd
sudo rmmod mydev       # 应成功

# 6. KASAN 测试（在 KASAN 内核上）
dmesg | grep "BUG: KASAN"  # 应无输出
```

## 25.13 设计题

**题目**：设计一个内核模块，为 N 个用户态进程提供共享内存通信信道（生产者-消费者模型），要求：
- 零拷贝（用户态直接读写，不经过 read/write syscall 数据路径）
- 支持多生产者单消费者
- `rmmod` 时等待所有进程关闭连接

**考点与答题要点**：
1. 零拷贝实现：`mmap` 实现（`file_operations.mmap` + `vm_ops`），将内核分配的 page 映射到用户地址空间
2. 共享内存布局：ring buffer header + data，header 里存 head/tail（需用 `smp_store_release`/`smp_load_acquire` 做 memory order，因为多 CPU 场景）
3. 并发控制：多生产者写端用 per-CPU buffer 或 spinlock；消费者读端无锁（单消费者）
4. 生命周期：`mmap` 时 `try_module_get`，`vm_ops->close` 时 `module_put`；`rmmod` 前检查 refcnt

**trade-off 补充**：共享内存方案避免了 syscall overhead，但放弃了内核对数据的隔离保护；消费者 bug 可能损坏 ring buffer header 导致生产者也出问题。这是内核 IPC 设计的经典权衡：performance vs isolation。

## 25.14 系统题（排障场景）

**场景**：生产环境某服务加载了一个字符设备模块后，系统定期出现 "BUG: soft lockup" 警告，怀疑与该模块的 ioctl 处理有关。如何定位？

**排障步骤**：

```
Step 1：确认相关性
  dmesg | grep -i "soft lockup"   # soft lockup 行里直接带 CPU 号和卡住的进程名/PID
  # 内核已经把 backtrace 打在 dmesg 里了，先读那段调用栈——soft lockup 报文自带 stack
  cat /proc/[pid]/wchan           # 进程当前阻塞在哪个内核符号（确认存在；睡眠态才有意义，
                                  #   纯 CPU 自旋的 soft lockup 进程 wchan 往往为 0/空）
  cat /proc/[pid]/stack           # 需 root + CONFIG_STACKTRACE：直接 dump 该任务的内核栈，
                                  #   比 wchan 信息量大得多，定位卡点首选

Step 2：ftrace 定位热点
  echo function > /sys/kernel/debug/tracing/current_tracer
  echo my_ioctl > /sys/kernel/debug/tracing/set_ftrace_filter
  echo 1 > /sys/kernel/debug/tracing/tracing_on
  # 复现，然后看 trace

Step 3：lockdep 验证（若测试内核可用）
  开启 CONFIG_PROVE_LOCKING 的测试内核重现，看是否有 circular locking warning

Step 4：常见 soft lockup 原因（在 ioctl 中）
  - 持 spinlock 进入无限循环（等某个条件）
  - 禁中断后做大量计算
  - 在 spinlock 临界区内调用可睡眠函数（本该用 mutex）

Step 5：代码审查核心点
  审查 ioctl 中所有 spinlock 临界区：临界区时长 > 几十微秒？
  有没有条件等待？应用 wait_event_interruptible 而非 spin 轮询
```

**量级参考**：
- soft lockup 的判定来自 `watchdog_thresh`（`/proc/sys/kernel/watchdog_thresh`，默认 **10 秒**）；soft lockup watchdog 在约 **2 × watchdog_thresh ≈ 20 秒** 没有调度到该 CPU 的 watchdog 线程时报 `BUG: soft lockup`。所以"20 秒"是结果阈值，源头是那个可调的 10 秒。
- 对照 **hard lockup**（CPU 关中断卡死、连 NMI 都打不进 → `BUG: NMI watchdog: hard lockup`）：两者机制不同，soft lockup 是"任务长时间不让出 CPU"，hard lockup 是"中断都进不来"，排障方向不一样，面试常一起问。
- spinlock 临界区量级：通常应 < 1 微秒，几十微秒已偏长，进入百微秒级基本可判定设计有问题（关抢占/可能关中断期间占用过久）。

## 25.15 代码题

**题目**：实现一个最小字符设备模块，包含：
- 内核缓冲区存储最后一次写入的数据
- 读操作返回该数据
- 一个 sysfs 属性显示读写次数
- KASAN 和 lockdep 下无报告

```c
// mydev.c — 带验证注释的可运行骨架
// 构建：make -C /lib/modules/$(uname -r)/build M=$(pwd) modules
// 加载：sudo insmod mydev.ko
// 测试：echo "test" > /dev/mydev0 && cat /dev/mydev0

#include <linux/module.h>
#include <linux/init.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/device.h>
#include <linux/mutex.h>
#include <linux/slab.h>
#include <linux/uaccess.h>
#include <linux/sysfs.h>
#include <linux/atomic.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Example");
MODULE_DESCRIPTION("Minimal char device with sysfs");

#define BUF_SIZE 4096
#define DEV_NAME "mydev"

struct mydev_data {
    struct cdev  cdev;
    struct mutex lock;
    char         *buf;
    size_t        buf_len;
    atomic_t      read_cnt;
    atomic_t      write_cnt;
};

static struct mydev_data mydev;
static struct class  *mydev_class;
static struct device *mydev_device;
static dev_t          mydev_devno;

/* ---- file_operations ---- */

static int mydev_open(struct inode *inode, struct file *filp)
{
    filp->private_data = &mydev;
    return 0;
}

static ssize_t mydev_read(struct file *filp, char __user *ubuf,
                          size_t count, loff_t *fpos)
{
    struct mydev_data *d = filp->private_data;
    ssize_t ret;

    mutex_lock(&d->lock);
    if (*fpos >= d->buf_len) {
        ret = 0;   /* EOF */
        goto out;
    }
    count = min(count, d->buf_len - (size_t)*fpos);
    /* 暗坑：copy_to_user 在 mutex 持有时调用是合法的（mutex 允许睡眠） */
    if (copy_to_user(ubuf, d->buf + *fpos, count)) {
        ret = -EFAULT;
        goto out;
    }
    *fpos += count;
    atomic_inc(&d->read_cnt);
    ret = count;
out:
    mutex_unlock(&d->lock);
    return ret;
}

static ssize_t mydev_write(struct file *filp, const char __user *ubuf,
                           size_t count, loff_t *fpos)
{
    struct mydev_data *d = filp->private_data;
    ssize_t ret;

    if (count > BUF_SIZE)
        return -EINVAL;

    mutex_lock(&d->lock);
    if (copy_from_user(d->buf, ubuf, count)) {
        ret = -EFAULT;
        goto out;
    }
    d->buf_len = count;
    *fpos = count;
    atomic_inc(&d->write_cnt);
    ret = count;
out:
    mutex_unlock(&d->lock);
    return ret;
}

static const struct file_operations mydev_fops = {
    .owner   = THIS_MODULE,
    .open    = mydev_open,
    .read    = mydev_read,
    .write   = mydev_write,
    .llseek  = no_llseek,   /* 6.12+：no_llseek 已删除，改为在 mydev_open 里
                               调用 nonseekable_open() 并删掉这一行（见 25.4.3 版本提示） */
};

/* ---- sysfs ---- */

static ssize_t stats_show(struct device *dev, struct device_attribute *attr,
                          char *buf)
{
    return sysfs_emit(buf, "reads=%d writes=%d\n",
                      atomic_read(&mydev.read_cnt),
                      atomic_read(&mydev.write_cnt));
}
static DEVICE_ATTR_RO(stats);

/* ---- init / exit ---- */

static int __init mydev_init(void)
{
    int ret;

    /* 1. 分配 buffer（在 cdev_add 前，确保所有状态就绪）*/
    mydev.buf = kzalloc(BUF_SIZE, GFP_KERNEL);
    if (!mydev.buf)
        return -ENOMEM;
    mutex_init(&mydev.lock);
    atomic_set(&mydev.read_cnt, 0);
    atomic_set(&mydev.write_cnt, 0);

    /* 2. 分配设备号 */
    ret = alloc_chrdev_region(&mydev_devno, 0, 1, DEV_NAME);
    if (ret)
        goto err_buf;

    /* 3. 初始化并注册 cdev */
    cdev_init(&mydev.cdev, &mydev_fops);
    mydev.cdev.owner = THIS_MODULE;
    ret = cdev_add(&mydev.cdev, mydev_devno, 1);
    if (ret)
        goto err_chrdev;

    /* 4. 创建 /dev 节点（依赖 udev）*/
    mydev_class = class_create(THIS_MODULE, DEV_NAME);
    if (IS_ERR(mydev_class)) {
        ret = PTR_ERR(mydev_class);
        goto err_cdev;
    }
    mydev_device = device_create(mydev_class, NULL, mydev_devno, NULL,
                                 "%s0", DEV_NAME);
    if (IS_ERR(mydev_device)) {
        ret = PTR_ERR(mydev_device);
        goto err_class;
    }

    /* 5. 创建 sysfs 属性 */
    ret = device_create_file(mydev_device, &dev_attr_stats);
    if (ret)
        goto err_device;

    pr_info("mydev: loaded, major=%d\n", MAJOR(mydev_devno));
    return 0;

err_device:
    device_destroy(mydev_class, mydev_devno);
err_class:
    class_destroy(mydev_class);
err_cdev:
    cdev_del(&mydev.cdev);
err_chrdev:
    unregister_chrdev_region(mydev_devno, 1);
err_buf:
    kfree(mydev.buf);
    return ret;
}

static void __exit mydev_exit(void)
{
    device_remove_file(mydev_device, &dev_attr_stats);
    device_destroy(mydev_class, mydev_devno);
    class_destroy(mydev_class);
    cdev_del(&mydev.cdev);
    unregister_chrdev_region(mydev_devno, 1);
    kfree(mydev.buf);
    pr_info("mydev: unloaded\n");
}

module_init(mydev_init);
module_exit(mydev_exit);
```

**暗坑提示**：
1. 错误路径**必须逆序清理**（`goto` 链），否则 KASAN 报 double-free 或 use-after-free。
2. `class_create()` 签名变过：**6.4（2023 年中）起去掉了第一个 `struct module *owner` 参数**，从 `class_create(THIS_MODULE, "name")` 变成 `class_create("name")`（owner 不再需要显式传，内核内部用调用者模块）。本骨架按旧签名写以兼容老内核；在 6.4+ 上编译报"too many arguments"时，删掉 `THIS_MODULE` 那个实参即可。函数原型以当前 `include/linux/device/class.h` 为准。
3. `sysfs_emit` 是 `scnprintf(buf, PAGE_SIZE, ...)` 的安全封装，不会溢出 sysfs page 限制；不要用 `sprintf`。

**验证方法**：
```bash
sudo insmod mydev.ko
echo "hello world" > /dev/mydev0
cat /dev/mydev0          # 应输出 hello world
cat /sys/class/mydev/mydev0/stats  # 应输出 reads=1 writes=1
sudo rmmod mydev
dmesg | tail -5          # 应无 BUG/oops/KASAN 字样
```
