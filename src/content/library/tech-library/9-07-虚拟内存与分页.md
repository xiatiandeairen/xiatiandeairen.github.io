> **一句话定位**：虚拟内存把"物理资源稀缺"问题改写成"地址翻译 + 延迟兑现"问题。本章不停在"页表四级、缺页惰性分配"的概念层，而是把缺页主路径 `handle_mm_fault → __handle_mm_fault → handle_pte_fault` 拆到 v6.12 真实源码、把 COW 的"何时复用何时复制"钉死到 `do_wp_page` 的判断条件、把 TLB shootdown 钉到 `native_flush_tlb_multi` 的 lazy 跳过逻辑、把 VMA 从 rbtree 到 maple tree 再到 per-VMA lock 的演进讲到 commit 和 LWN 文章号。

---

## TL;DR — 本章最硬结论（每条都能在源码 / 史料里落地）

1. **缺页主路径是一棵"按级 alloc"的递归下降，不是查表**：`__handle_mm_fault()` 从 `pgd_offset` 一路 `p4d_alloc / pud_alloc / pmd_alloc`，每一级缺了就地建表，每一级都先问"这一级能不能直接放大页"（`thp_vma_allowable_order`），最后才落到 `handle_pte_fault()`。理解这条"一边走一边可能岔进大页"的路径，是理解 demand paging 和 THP 同源的钥匙。【真实源码 v6.12 `mm/memory.c`，见 §7.5.3】
2. **COW 不是"写就复制"，而是"写时先问能不能独占复用"**：`do_wp_page()` 的核心分支是 `wp_can_reuse_anon_folio()`——若这个匿名 folio 的引用计数已经是 1（只有我在用），直接 `wp_page_reuse` 翻 Writable 位，**不复制一个字节**。只有共享时才走 `wp_page_copy`。fork 后 Redis 内存翻倍的真正变量是"父进程对同一页的写次数 + 该页是否仍被子进程引用"。【真实源码 v6.12 `mm/memory.c`，见 §7.7.2】
3. **VMA 的组织结构换过三次引擎，6.1 起是 maple tree**：rbtree + 双向链表（~6.1 前）→ maple tree（6.1，commit `54a611b6059`，Liam Howlett）→ 在此基础上加 per-VMA lock（6.4，commit `5e31275cc99`，Suren Baghdasaryan）。换 maple tree 的动机白纸黑字写在 commit：rbtree 不支持区间、难做 lockless、遍历低效所以才额外挂链表。【commit + LWN 845507/906852 已核实，见 §7.4】
4. **TLB 必须软件显式 shootdown，因为它不参与 cache 一致性**：`native_flush_tlb_multi()` 真实代码里有一个关键分支——`if (info->freed_tables)` 才给所有核（含 lazy）发 IPI，否则 lazy TLB 的核可以跳过。这条 `freed_tables` 分支是"改权限"和"拆页表"代价差异的根源。【真实源码 v6.12 `arch/x86/mm/tlb.c`，见 §7.6.2】
5. **5 级页表不等于程序立刻拿到 128 PB**：x86 LA57 早在 4.12/4.14 一带由 Kirill Shutemov 合入（LWN 717293），但 `mmap` 默认只在低 47 位分配，**只有应用显式传一个 > 47 位的 hint 地址才会映射到高区**——这是为了不踩"把指针高位塞 tag"的老程序。【LWN 717293 已核实 + 真实源码 `arch/x86/include/asm/pgtable_64_types.h`，见 §7.3.4】
6. **缺页路径在 6.4 起优先走 per-VMA lock，绕开 mmap_lock**：`do_user_addr_fault()` 真实代码里第一步是 `lock_vma_under_rcu(mm, address)`，成功就在 RCU + 单 VMA 读锁下处理缺页，失败才回退全局 `mmap_lock`。多线程大地址空间下的缺页竞争，根因就是这把曾经的全局锁。【真实源码 v6.12 `arch/x86/mm/fault.c` + `mm/memory.c`，见 §7.5.2】

## 7.1 背景：为什么需要虚拟内存（先把问题说准）

### 7.1.1 裸机时代的三个痛点

没有虚拟内存时（早期 MS-DOS、裸片实时系统），程序直接吃物理地址，撞上三堵墙：

```text
【示意，非逐字】
痛点 1 · 隔离缺失
  进程 A 写错指针 → 踩进程 B 的数据 → 系统崩
  根因：物理地址是全局共享命名空间，谁都能写谁

痛点 2 · 物理内存不够
  物理 RAM 4 MB，应用想要 64 MB
  → 只能手工 overlay / swap，是开发地狱

痛点 3 · 重定位
  程序编译期就得知道自己加载到哪个物理地址
  → 写不了位置无关代码，库无法共享
```

虚拟内存用"每进程一套地址翻译表（页表）"一次解决三者：隔离（不同进程的页表互不重叠映射）、过载（地址承诺多于物理 RAM，靠延迟兑现）、重定位（虚拟地址固定，物理地址随便放）。

### 7.1.2 MMU：硬件侧的共谋，以及它的"私有 cache"埋下的全部麻烦

MMU（Memory Management Unit）是 CPU 内部硬件模块，CPU 每次访存它自动做翻译；翻译失败就触发缺页异常交给内核。没有 MMU 的核（部分嵌入式）跑的是 `nommu` Linux，本章不涵盖。

> **底层根因（贯穿全章的那根线）**：MMU 为了快，内部有一个**私有的翻译缓存 TLB**。TLB 是 CPU-local 的、**不参与 MESI 等数据 cache 一致性协议**。这件事看似细节，其实是本章一半"贵"的来源——改页表后必须由软件（内核）显式广播失效（§7.6 TLB shootdown），硬件不帮你。记住这句话，后面所有"为什么这步这么贵"都能往这归。

## 7.2 虚拟地址空间布局（用内核官方文档钉死，不靠记忆画）

### 7.2.1 x86_64 4 级页表的真实内存图

下面这张表**不是凭印象画的**，是 v6.12 内核文档 `Documentation/arch/x86/x86_64/mm.rst` 的真实节选——这是判断"哪段在哪"的权威：

```text
【真实源码 v6.12 Documentation/arch/x86/x86_64/mm.rst 节选】
   Start addr    |   Offset   |     End addr     |  Size   | VM area description
========================================================================================
0000000000000000 |    0       | 00007fffffffffff |  128 TB | user-space virtual memory
0000800000000000 | +128    TB | ffff7fffffffffff | ~16M TB | ... huge non-canonical hole
________________________________________________________________________________________
ffff800000000000 | -128    TB | ffff87ffffffffff |    8 TB | ... guard hole, hypervisor
ffff880000000000 | -120    TB | ffff887fffffffff |  0.5 TB | LDT remap for PTI
ffff888000000000 | -119.5  TB | ffffc87fffffffff |   64 TB | direct mapping of all phys mem
ffffc90000000000 |  -55    TB | ffffe8ffffffffff |   32 TB | vmalloc/ioremap space
ffffea0000000000 |  -22    TB | ffffeaffffffffff |    1 TB | virtual memory map (vmemmap)
ffffec0000000000 |  -20    TB | fffffbffffffffff |   16 TB | KASAN shadow memory
________________________________________________________________________________________
ffffffff80000000 |   -2    GB | ffffffff9fffffff |  512 MB | kernel text mapping
ffffffffa0000000 |-1536    MB | fffffffffeffffff | 1520 MB | module mapping space
ffffffffff600000 |  -10    MB | ffffffffff600fff |    4 kB | legacy vsyscall ABI
```

逐行点几个工程上要命的细节：

- **`direct mapping of all phys mem`（64 TB，`page_offset_base`）**：内核把全部物理内存线性映射在这一段。`phys_to_virt()` / `__va()` 就是在这段上做常量偏移。**这解释了为什么内核访问任意物理页几乎零成本**——不用临时建映射，加个偏移就到。也解释了为什么 32 位内核痛苦（直接映射区只有 ~896 MB，多出来的物理内存得靠 highmem 临时映射，已成历史）。
- **`vmalloc/ioremap space`（32 TB）**：`vmalloc()` 给的是虚拟连续、物理可不连续的内存（见第 9 章）。它和 direct map 是**两套独立的虚拟地址**指向（可能的）同一批物理页——这是排查内核地址时第一个要分清的。
- **canonical hole（中间那条 ~16M TB）**：48 位有效地址下，`[47]` 位必须符号扩展到 `[63:48]`，否则就是 non-canonical 地址，访问触发 **#GP（General Protection Fault），不是 #PF（缺页）**。

> **💡 高频面试陷阱**：被问"内核占多大"答"2 GB / 1 GB"的，暴露知识停在 32 位 x86。64 位是 128 TB 用户 / 128 TB 内核（4 级），且内核空间在所有进程间共享同一套映射——这是为了系统调用进内核后能直接访问内核数据，不必换页表根（KPTI 之后这个"不换"被部分打破，见 §7.10.2）。

### 7.2.2 各段的工程推论（带"为什么"）

| 段 | 特点 | 工程推论 / 根因 |
|---|---|---|
| Text（代码段）| 只读 + 可执行，物理页跨进程共享 | 10 个 nginx 进程，text 物理页只有一份；靠 page cache + 文件映射去重 |
| Data/BSS | 可读写，fork 后 COW | BSS 在 exec 时**不占物理页**，第一次写才分配（demand paging 直接收益） |
| 堆（heap）| `brk`/`mmap` 匿名，向高增长 | glibc malloc 小块走 `brk`，大块（默认 >128 KB，`M_MMAP_THRESHOLD`）走 `mmap` |
| mmap 区 | 文件映射 + 匿名大块，向低增长 | 共享库、`mmap` 文件、大 malloc 都在这；ASLR 随机化的主战场（§7.9） |
| 栈（stack）| 匿名，向低增长，有 guard gap | 默认 8 MB（`ulimit -s`）；guard gap 防止栈与下方 mmap 撞上 |
| vDSO/vvar | 内核映射进用户空间的特殊页 | `clock_gettime` 不陷入内核靠它（见第 2 章） |

> **🎯 面试加分**：为什么堆向高、栈向低？历史遗留 + 让两者在中间尽量晚相撞——早期没有独立 mmap 区时这很实用。现代 Linux 有了 `mmap_base` 居中的独立映射区，两者不再直接相对，但增长方向的惯例保留。

### 7.2.3 VMA：内核眼中的地址空间单元（看真实结构体）

内核用 `struct vm_area_struct`（VMA）描述地址空间里每一段连续映射。所有 VMA 挂在 `mm_struct` 上，`/proc/<pid>/maps` 就是它的文字版。下面是 v6.12 的**真实定义节选**——注意它的第一句注释和老资料完全不同：

```c
/* 【真实源码 v6.12 include/linux/mm_types.h 行 667 起，节选】 */
struct vm_area_struct {
	/* The first cache line has the info for VMA tree walking. */

	union {
		struct {
			/* VMA covers [vm_start; vm_end) addresses within mm */
			unsigned long vm_start;
			unsigned long vm_end;
		};
#ifdef CONFIG_PER_VMA_LOCK
		struct rcu_head vm_rcu;	/* Used for deferred freeing. */
#endif
	};

	struct mm_struct *vm_mm;        /* The address space we belong to. */
	pgprot_t vm_page_prot;          /* Access permissions of this VMA. */

	union {
		const vm_flags_t vm_flags;
		vm_flags_t __private __vm_flags;
	};

#ifdef CONFIG_PER_VMA_LOCK
	bool detached;          /* 从 mm->mm_mt 树上摘下的标记 */
	int vm_lock_seq;        /* 与 mm->mm_lock_seq 比对判断是否被写锁持有 */
	struct vma_lock *vm_lock;
#endif
	/* ... shared.rb（文件映射的 interval tree 链接）... */
	struct list_head anon_vma_chain;
	struct anon_vma *anon_vma;
	const struct vm_operations_struct *vm_ops;  /* 操作函数表 */
	unsigned long vm_pgoff;         /* 文件偏移（页单位） */
	struct file *vm_file;           /* 文件映射时非 NULL */
	/* ... */
} __randomize_layout;
```

逐行读出三个"和老书不一样"的点：

1. **第一行注释 `The first cache line has the info for VMA tree walking`**——内核刻意把 `vm_start/vm_end/vm_mm/vm_flags` 这些"遍历时要碰的字段"塞进第一条 cache line。这是 cache 友好设计，不是随便排的。
2. **`vm_start/vm_end` 和 `vm_rcu` 共用一个 union**——因为开启 `CONFIG_PER_VMA_LOCK` 后，VMA 释放走 RCU 延迟（§7.4.3），释放期的 `rcu_head` 复用了已经用不上的起止地址字段，省内存。
3. **没有 `vm_next`/`vm_prev`，也没有 `vm_rb`**——老资料里那条双向链表和红黑树节点，**6.1 起都没了**。VMA 现在统一挂在 `mm->mm_mt`（maple tree）上。这是 §7.4 设计考古的主角。

> **💡 时效坑点**：面试被问"VMA 怎么组织"，答"红黑树"会暴露知识停在 5.x。正确答法：**看版本——6.1 前 rbtree + 双向链表，6.1 起 maple tree**，换的动机和具体收益见 §7.4。

## 7.3 多级页表与地址翻译

### 7.3.1 为什么不用平坦页表（先算账）

朴素思路：每进程一张平坦表，记录每个虚拟页号 → 物理页号。

x86_64 有效地址 48 位，4 KB 页 → 页数 = 2⁴⁸ / 2¹² = 2³⁶ ≈ 687 亿条目。每条 8 字节 → **单进程页表 512 GB**。根本不可行。

解法：**只记录实际用到的映射**，用树形多级索引按需分配中间节点。地址空间通常极稀疏（几百个 VMA，覆盖率 < 0.01%），树形结构让未用区域的中间表根本不存在。

### 7.3.2 x86_64 四级页表的真实 SHIFT 定义

不靠记忆，看 v6.12 真实头文件——每一级的位移是写死的常量：

```c
/* 【真实源码 v6.12 arch/x86/include/asm/pgtable_64_types.h，节选 + 重排】 */
#ifndef CONFIG_X86_5LEVEL
#define PGDIR_SHIFT		39      /* PGD: 位 [47:39] */
#define PTRS_PER_PGD		512
#define MAX_PTRS_PER_P4D	1       /* 4 级时 P4D 被“折叠”掉 */
#endif

#define PUD_SHIFT	30              /* PUD: 位 [38:30] */
#define PTRS_PER_PUD	512
#define PMD_SHIFT	21              /* PMD: 位 [29:21] */
#define PTRS_PER_PMD	512
#define PTRS_PER_PTE	512          /* PTE: 位 [20:12] */
```

把这些常量翻译成翻译流程（`page offset` = 低 12 位）：

```text
【示意，基于上面真实常量】
虚拟地址 48 位：[47:39][38:30][29:21][20:12][11:0]
                 PGD    PUD    PMD    PTE   offset
                (9bit) (9bit) (9bit) (9bit)(12bit)

CR3 ─→ PGD 基址
   └[47:39]→ PGD entry ─→ PUD 基址
        └[38:30]→ PUD entry ─→ PMD 基址
             └[29:21]→ PMD entry ─→ PTE 基址
                  └[20:12]→ PTE entry ─→ 物理页帧 (PFN)
                       └[11:0]→ + offset = 物理地址
```

每级 512 entry（9 bit），每张表正好 512 × 8B = 4 KB = 一页——**页表自己也是按页管理的，这不是巧合，是按 4 KB 页 + 8 字节 entry 倒推的设计**。

注意上面 `MAX_PTRS_PER_P4D = 1`：4 级模式下 P4D 这一级被"折叠"（folding），`p4d_offset` 直接透传到 PGD。内核源码里 5 级是基准，4 级靠把 P4D 压成 1 项来兼容——这就是为什么 `__handle_mm_fault` 里永远写 `p4d_alloc`（§7.5.3），4 级时它是个 no-op。

### 7.3.3 PTE flag 位（COW / 回收 / 安全的物理开关）

| bit | 名称 | 含义 / 谁来用 |
|---|---|---|
| 0 | Present | 1=页存在；0=触发缺页。demand paging 的总开关 |
| 1 | R/W | 0=只读。**COW 的物理实现就是清这一位**（§7.7） |
| 2 | U/S | 0=只有内核可访问。用户态碰到就 SIGSEGV |
| 5 | Accessed | MMU 自动置位，page reclaim 的 LRU 老化靠它（见第 10 章） |
| 6 | Dirty | 写操作后 MMU 置位，writeback 靠它判断脏页 |
| 63 | NX | 1=不可执行。栈/堆置 NX 挡 shellcode（W^X 防线） |

> **🎯 面试加分**：COW 的硬件实现就是把 PTE 的 R/W 位清零、Present 保持 1——父子都能读，任一方写时 MMU 触发 write-protection fault，内核在 `do_wp_page` 里决定复用还是复制（§7.7.2）。"清 R/W 位"是软件做的，"写时触发 fault"是硬件做的，分工要说清。

### 7.3.4 5 级页表（LA57）：设计考古 + 兼容性取舍

**史料（已核实）**：据 [LWN 717293《Five-level page tables》（2017-03-15，Jonathan Corbet）](https://lwn.net/Articles/717293/)，第五级页表（P4D）由 **Kirill Shutemov** 实现，约在 4.11–4.12 一带合入，把上限"bump 到 128 PiB 虚拟地址空间和 4 PiB 物理地址空间"，并支持 512 GB 大页。

关键设计取舍——**开了 5 级页表 ≠ 程序立刻拿到 128 PB**。LWN 717293 原文：存在一些程序"知道"只有 48 位虚拟地址有效，把别的信息编码在指针高位（典型如某些 JS 引擎的 NaN-boxing、tagged pointer）。为了不踩这些程序：

> "the x86 patches in their current form will not allocate memory in the new address space by default." 应用要拿高地址，必须在 `mmap()` 里**显式传一个 > 47 位的 hint 地址**。

这一取舍在 v6.12 源码里仍然成立，看真实头文件如何用一个运行期变量切换 4/5 级：

```c
/* 【真实源码 v6.12 arch/x86/include/asm/pgtable_64_types.h，节选】 */
#ifdef CONFIG_X86_5LEVEL
#define pgtable_l5_enabled() cpu_feature_enabled(X86_FEATURE_LA57)
#else
#define pgtable_l5_enabled() 0
#endif

#ifdef CONFIG_X86_5LEVEL
#define PGDIR_SHIFT	pgdir_shift     /* 运行期变量！不是常量 */
#define PTRS_PER_PGD	512
#define P4D_SHIFT		39
#define PTRS_PER_P4D		ptrs_per_p4d
#define MAX_POSSIBLE_PHYSMEM_BITS	52
#else
#define PGDIR_SHIFT		39      /* 4 级时是编译期常量 */
#endif
```

逐行读出门道：

- `pgtable_l5_enabled()` 在 5 级编译时退化成一次 CPU feature 检测（`X86_FEATURE_LA57`）——**同一个内核二进制可以同时支持 4 级和 5 级硬件**，靠运行期判断。
- 注意 `PGDIR_SHIFT` 在 5 级模式下从编译期常量 `39` 变成**运行期变量 `pgdir_shift`**。这就是代价：5 级模式下，每次地址拆解多一层间接，编译器没法把 shift 常量折叠进指令。这是"默认不开"的微小性能理由之一。
- 谁需要它？数据中心 scale-up 的 TB 级内存机器（>64 TB 物理）。日常服务器仍是 4 级。

## 7.4 设计考古：VMA 管理的三次引擎更换（rbtree → maple tree → per-VMA lock）

> 这一节是本章的"史料硬货"。每个论断带出处（LWN 文章号 / commit / 邮件线程），讲清"谁、何时、为什么、解决前一代什么缺陷、社区辩论过什么"。

### 7.4.1 第一代：rbtree + 双向链表（远古 ~ 6.1）的"原罪"

5.x 及更早，VMA 同时挂在两个结构上：

```text
【示意，已删除的历史结构】
mm_struct
  ├── mm_rb        红黑树：按地址区间查找“包含地址 X 的 VMA”，O(log n)
  └── (每个 vma) vm_next / vm_prev   双向链表：按地址有序遍历，O(1) 取下一个
```

**为什么要两套？** 红黑树擅长"点查找"（缺页时按 fault 地址找 VMA），但**遍历**（如 `/proc/pid/maps`、fork 时复制所有 VMA、munmap 一段范围）走树要中序遍历，cache 不友好；于是额外挂一条链表专门做有序遍历。两套结构并存 = 每次插入/删除 VMA 要同时维护树和链表，**还得保证两者一致**，又费 cache 又是 bug 温床。

### 7.4.2 第二代：maple tree（6.1，2022 年底）

**史料（已核实）**：据 [LWN 845507《Introducing maple trees》（2021-02-12，Marta Rybczyńska）](https://lwn.net/Articles/845507/)，maple tree 由 **Liam Howlett 和 Matthew Wilcox** 开发，在 2019 年 LSF/MM 峰会作为方案提出。文章直接点了 rbtree 的三宗罪：

> "Rbtrees do not support ranges well, they are difficult to handle in a lockless manner ... and rbtree traversal is inefficient, which is why the additional linked list exists."（rbtree 不擅长区间、难以 lockless、遍历低效——这正是那条额外链表存在的原因。）

合入的 commit 把设计动机写得更干净，这是**真实 commit message**：

```text
【真实 commit 54a611b605901c7d5d05b6b8f5d04a6ceb0962aa】
Liam R. Howlett, 2022-09-06, "Maple Tree: add new data structure"

The maple tree is an RCU-safe range based B-tree designed to use modern
processor cache efficiently. ...
The tree has a branching factor of 10 for non-leaf nodes and 16 for leaf
nodes. With the increased branching factor, it is significantly shorter
than the rbtree so it has fewer cache misses. The removal of the linked
list between subsequent entries also reduces the cache misses and the need
to pull in the previous and next VMA during many tree alterations.
```

逐行读出 maple tree 凭什么赢：

- **`range based B-tree`**：天生支持区间，缺页"找包含地址 X 的 VMA"和 munmap"删 [a,b) 范围"都是原生操作，不用像 rbtree 那样在树上拼区间逻辑。
- **`branching factor 10 / 16`**：B-tree 一个节点放 10~16 个 entry（rbtree 一个节点只有 2 个孩子），树高骤降 → cache miss 少。LWN 845507 补充：节点大小是 **256 字节 = cache line 整数倍**，遍历时一次 cache 拉取覆盖更多 entry。
- **`removal of the linked list`**：链表删了，因为 B-tree 节点内 entry 本身有序，遍历直接走节点内数组，前驱后继就在隔壁，不必"pull in previous and next VMA"造成额外 cache miss。
- **`RCU-safe`**：这是为下一代 per-VMA lock 铺路——LWN 845507 明说 "maple trees were designed to work in a lockless manner, using read-copy-update (RCU)"。

同一个 patch series 里，commit `763ecb035029`（"mm: remove the vma linked list"）**正式删掉那条链表**，把 `free_pgtables / unmap_vmas / zap_page_range` 全改成用 maple tree 遍历（`vma_find()`）。

> **社区辩论的味道**：LWN 845507 提到 maple tree 当时**只解决性能、不动 mmap_lock 本身**——"the patch set preserves the lock for now"。也就是说社区清楚"换数据结构"和"换锁"是两件事，先换数据结构（铺 RCU 底子），再动锁（下一代）。这种分两步走、不一次性梭哈的节奏，是大型子系统改造能合入主线的关键。

### 7.4.3 第三代：per-VMA lock（6.4，2023）——缺页不再抢全局锁

**史料（已核实）**：据 [LWN 906852《Concurrent page-fault handling with per-VMA locks》（2022-09-05，Jonathan Corbet）](https://lwn.net/Articles/906852/)，问题是 `mmap_lock` 这把"保护整个地址空间的读写锁"在多线程大地址空间下成了瓶颈：

> "A process can have a large address space and many threads running (and incurring page faults) concurrently, turning mmap_lock into a significant bottleneck."

**Suren Baghdasaryan** 的方案（在 maple tree 的 RCU 基础上）落地在 commit `5e31275cc99`，机制是**序列号比对 + per-VMA 读写锁**，这是真实 commit message：

```text
【真实 commit 5e31275cc997f8ec5d9e8d65fe9840ebed89db19】
Suren Baghdasaryan, 2023-02-27, "mm: add per-VMA lock and helper functions"

  - read lock - (vma_start_read) requires the vma (vm_lock_seq) and mm
    (mm_lock_seq) sequence counters to differ. If they match then there
    must be a vma exclusive lock held somewhere.
  - write lock - (vma_start_write) requires the mmap_lock to be held
    exclusively and the current mm counter is assigned to the vma counter.
```

逐行解码这套精巧的"序列号即锁"机制：

- **读锁（缺页路径）**：只需检查 `vma->vm_lock_seq != mm->mm_lock_seq`。不相等 = 没人在写这个 VMA，可以安全读。这一步不碰全局 `mmap_lock`，多个 CPU 的缺页可以**真并行**。
- **写锁（改地址空间，如 mmap/munmap/VMA 合并）**：先拿 `mmap_lock` 写锁，再把"当前 mm 的序列号"赋给受影响 VMA 的 `vm_lock_seq`。一次写锁可以"锁住"多个 VMA（VMA 合并场景），因为它们的 seq 都被设成等于 mm 的 seq。
- **为什么这招省**：读侧零原子写、零锁竞争，纯比较两个数；只有真要改地址空间时才有重量级同步。这是典型的"读多写少 → 把代价全压给写侧"。

这就解释了 §7.5.2 里 `do_user_addr_fault` 为什么第一步是 `lock_vma_under_rcu`——缺页热路径走的就是这条 per-VMA 快路。

> **演进史的"前传"**：LWN 906852 提到 per-VMA lock 之前有十几年的 **speculative page fault**（Laurent Dufour 等）尝试——"do the work to resolve a fault is done without taking mmap_lock in the hope that the address space doesn't change"，但"too complex for mainline acceptance"。Baghdasaryan 的版本拿到了约 speculative 方案 **75% 的收益**，但复杂度低到能合入。这是工程史上"次优但能落地" > "最优但合不进主线"的经典案例。

## 7.5 缺页异常处理（从硬件入口到 `handle_pte_fault` 的真实路径）

### 7.5.1 x86 硬件入口：`exc_page_fault`

缺页从 CPU 异常进内核。v6.12 的 x86 入口是 `exc_page_fault`（早期叫 `do_page_fault`，~5.8 idtentry 重构后改名），这是**真实源码**：

```c
/* 【真实源码 v6.12 arch/x86/mm/fault.c 行 1493 起，节选】 */
DEFINE_IDTENTRY_RAW_ERRORCODE(exc_page_fault)
{
	irqentry_state_t state;
	unsigned long address;

	address = cpu_feature_enabled(X86_FEATURE_FRED) ?
			fred_event_data(regs) : read_cr2();   /* ← 出错地址在 CR2 */

	prefetchw(&current->mm->mmap_lock);

	if (kvm_handle_async_pf(regs, (u32)address))      /* KVM 异步缺页旁路 */
		return;

	state = irqentry_enter(regs);
	instrumentation_begin();
	handle_page_fault(regs, error_code, address);     /* ← 进入分流 */
	instrumentation_end();
	irqentry_exit(regs, state);
}
```

逐行读出三个关键事实：

1. **出错地址从 CR2 寄存器读**（`read_cr2()`）——这是 x86 ABI：MMU 翻译失败时把出错的线性地址塞进 CR2。新硬件用 FRED 机制（`fred_event_data`）则从事件数据取，是近年新增的旁路。
2. **`prefetchw(&current->mm->mmap_lock)`**——在真正用到 `mmap_lock` 之前就预取它到 cache（写意图）。这是为缺页这条超高频路径做的微优化，体现内核对缺页延迟的极致在意。
3. **`kvm_handle_async_pf`**——虚拟化场景下，host 把"guest 访问的页被 swap 出去了"伪装成 #PF 注入 guest，guest 在这里旁路处理，不走正常缺页流程。

接着 `handle_page_fault` 按地址空间分流（真实源码）：

```c
/* 【真实源码 v6.12 arch/x86/mm/fault.c 行 1468 起，节选】 */
static __always_inline void
handle_page_fault(struct pt_regs *regs, unsigned long error_code,
			      unsigned long address)
{
	trace_page_fault_entries(regs, error_code, address);
	if (unlikely(kmmio_fault(regs, address)))
		return;

	/* Was the fault on kernel-controlled part of the address space? */
	if (unlikely(fault_in_kernel_space(address))) {
		do_kern_addr_fault(regs, error_code, address);   /* 内核地址：多半是 vmalloc 缺页或 bug */
	} else {
		do_user_addr_fault(regs, error_code, address);   /* 用户地址：主战场 */
		local_irq_disable();
	}
}
```

### 7.5.2 `do_user_addr_fault`：先试 per-VMA 快路，失败回退 mmap_lock

这是把 §7.4.3 的 per-VMA lock 落地的地方。看**真实源码**里"先快后慢"的两段式结构：

```c
/* 【真实源码 v6.12 arch/x86/mm/fault.c 行 1211 起，大幅节选保留主干】 */
void do_user_addr_fault(struct pt_regs *regs, unsigned long error_code,
			unsigned long address)
{
	struct vm_area_struct *vma;
	unsigned int flags = FAULT_FLAG_DEFAULT;
	/* ... 一堆安全检查：内核执行用户内存？SMAP 违例？中断上下文？... */

	local_irq_enable();
	perf_sw_event(PERF_COUNT_SW_PAGE_FAULTS, 1, regs, address);

	/* error_code 的位翻译成 fault flags */
	if (error_code & X86_PF_WRITE)  flags |= FAULT_FLAG_WRITE;
	if (error_code & X86_PF_INSTR)  flags |= FAULT_FLAG_INSTRUCTION;
	if (user_mode(regs))            flags |= FAULT_FLAG_USER;

	if (!(flags & FAULT_FLAG_USER))
		goto lock_mmap;

	/* ===== 快路：per-VMA lock，不抢全局 mmap_lock ===== */
	vma = lock_vma_under_rcu(mm, address);
	if (!vma)
		goto lock_mmap;                          /* 快路失败 → 回退慢路 */
	if (unlikely(access_error(error_code, vma))) {
		bad_area_access_error(regs, error_code, address, NULL, vma);
		count_vm_vma_lock_event(VMA_LOCK_SUCCESS);
		return;
	}
	fault = handle_mm_fault(vma, address, flags | FAULT_FLAG_VMA_LOCK, regs);
	if (!(fault & (VM_FAULT_RETRY | VM_FAULT_COMPLETED)))
		vma_end_read(vma);
	if (!(fault & VM_FAULT_RETRY)) {
		count_vm_vma_lock_event(VMA_LOCK_SUCCESS);
		goto done;
	}
	count_vm_vma_lock_event(VMA_LOCK_RETRY);     /* 快路撞上写者 → 慢路重试 */

lock_mmap:
	/* ===== 慢路：抢 mmap_lock，传统流程 ===== */
retry:
	vma = lock_mm_and_find_vma(mm, address, regs);
	if (unlikely(!vma)) {
		bad_area_nosemaphore(regs, error_code, address);   /* 无 VMA → SIGSEGV */
		return;
	}
	/* ... */
	fault = handle_mm_fault(vma, address, flags, regs);
```

逐层读出工程意图：

- **`error_code` 位 → `FAULT_FLAG_*`**：硬件用 error_code 告诉内核"读还是写、用户还是内核、取指还是访存"。`X86_PF_WRITE` 决定后面要不要走 COW。
- **`lock_vma_under_rcu` 成功就直接 `handle_mm_fault`**，并带上 `FAULT_FLAG_VMA_LOCK` 告诉下层"我只持有 VMA 锁，别碰需要 mmap_lock 的操作"。
- **`VMA_LOCK_SUCCESS / VMA_LOCK_RETRY` 计数**：这俩 vmstat 计数器可以在生产里用 `grep vma_lock /proc/vmstat` 看 per-VMA lock 的命中率——retry 多说明地址空间改动频繁、快路常被打断。
- **慢路 `lock_mm_and_find_vma` 找不到 VMA → SIGSEGV**：这就是"野指针 segfault"的内核实现位置。

`lock_vma_under_rcu` 本体也值得一看（真实源码），它把 §7.4 三代演进全串起来：

```c
/* 【真实源码 v6.12 mm/memory.c 行 6231 起，节选】 */
struct vm_area_struct *lock_vma_under_rcu(struct mm_struct *mm,
					  unsigned long address)
{
	MA_STATE(mas, &mm->mm_mt, address, address);   /* ← maple tree 迭代器 */
	struct vm_area_struct *vma;

	rcu_read_lock();
retry:
	vma = mas_walk(&mas);                          /* ← 在 maple tree 上点查找 */
	if (!vma)
		goto inval;
	if (!vma_start_read(vma))                      /* ← per-VMA 读锁（序列号比对） */
		goto inval;
	if (vma->detached) {                           /* VMA 被并发摘走了 */
		vma_end_read(vma);
		goto retry;
	}
	/* 锁住后再次确认地址仍落在 [vm_start, vm_end) */
	if (unlikely(address < vma->vm_start || address >= vma->vm_end))
		goto inval_end_read;
	rcu_read_unlock();
	return vma;
	/* ... inval 路径返回 NULL，让调用者回退慢路 ... */
}
```

一函数三代史：`MA_STATE`/`mas_walk` 是 **maple tree**（第二代）、`rcu_read_lock` 是 maple tree 的 **RCU 底子**、`vma_start_read` 是 **per-VMA lock**（第三代）。三层叠在一起才有今天缺页热路径的可扩展性。

### 7.5.3 架构无关核心：`__handle_mm_fault` 的递归下降

进入 `handle_mm_fault`（多年稳定的入口名）后，转 `__handle_mm_fault` 做真正的页表遍历。这是本章**最该精读的一段真实源码**——它一边按级建表，一边在每级试探大页：

```c
/* 【真实源码 v6.12 mm/memory.c 行 5816 起，大幅节选保留主干】 */
static vm_fault_t __handle_mm_fault(struct vm_area_struct *vma,
		unsigned long address, unsigned int flags)
{
	struct vm_fault vmf = {
		.vma = vma,
		.address = address & PAGE_MASK,
		.flags = flags,
		.pgoff = linear_page_index(vma, address),
		.gfp_mask = __get_fault_gfp_mask(vma),
	};
	struct mm_struct *mm = vma->vm_mm;
	pgd_t *pgd;
	p4d_t *p4d;

	pgd = pgd_offset(mm, address);          /* CR3 → PGD entry，纯算偏移，不分配 */
	p4d = p4d_alloc(mm, pgd, address);      /* P4D 缺了就建（4 级时是 no-op） */
	if (!p4d)
		return VM_FAULT_OOM;

	vmf.pud = pud_alloc(mm, p4d, address);  /* PUD 缺了就建 */
	if (!vmf.pud)
		return VM_FAULT_OOM;
retry_pud:
	/* 这一级能放 1GB 大页吗？（PUD 级 THP，极少见） */
	if (pud_none(*vmf.pud) &&
	    thp_vma_allowable_order(vma, vm_flags, TVA_IN_PF | TVA_ENFORCE_SYSFS, PUD_ORDER)) {
		ret = create_huge_pud(&vmf);
		if (!(ret & VM_FAULT_FALLBACK))
			return ret;
	}
	/* ... 处理已存在的 PUD 大页（写保护/NUMA）... */

	vmf.pmd = pmd_alloc(mm, vmf.pud, address);   /* PMD 缺了就建 */
	if (!vmf.pmd)
		return VM_FAULT_OOM;

	/* 这一级能放 2MB 大页吗？（PMD 级 THP，常见） */
	if (pmd_none(*vmf.pmd) &&
	    thp_vma_allowable_order(vma, vm_flags, TVA_IN_PF | TVA_ENFORCE_SYSFS, PMD_ORDER)) {
		ret = create_huge_pmd(&vmf);         /* ← THP 在这里诞生 */
		if (!(ret & VM_FAULT_FALLBACK))
			return ret;
	} else {
		vmf.orig_pmd = pmdp_get_lockless(vmf.pmd);
		if (pmd_trans_huge(vmf.orig_pmd) || pmd_devmap(vmf.orig_pmd)) {
			if (pmd_protnone(vmf.orig_pmd) && vma_is_accessible(vma))
				return do_huge_pmd_numa_page(&vmf);    /* 大页 NUMA 迁移 */
			if ((flags & (FAULT_FLAG_WRITE|FAULT_FLAG_UNSHARE)) &&
			    !pmd_write(vmf.orig_pmd)) {
				ret = wp_huge_pmd(&vmf);               /* 大页 COW */
				if (!(ret & VM_FAULT_FALLBACK))
					return ret;
			}
			/* ... */
		}
	}

	return handle_pte_fault(&vmf);          /* 没走大页，落到最后一级 PTE */
}
```

逐层读出这段为什么是全章核心：

- **`pgd_offset` 只算偏移、`*_alloc` 才分配**：PGD 是进程一创建就有的（CR3 指向它），所以只 offset；下面三级"缺了才建"，这正是 demand paging 在**页表层**的体现——连中间页表都是按需分配的，不止数据页。
- **每一级 `*_alloc` 失败都 `return VM_FAULT_OOM`**：缺页路径自己可能因为分不到中间页表而 OOM，这是 OOM 的一个隐蔽来源。
- **每级先试大页（`thp_vma_allowable_order`）再降级**：这把 §7.8 的 THP 和缺页**统一**了——THP 不是独立机制，就是缺页时"这一级如果允许，直接放个 2MB（PMD）或 1GB（PUD）大页"。`create_huge_pmd` 失败返回 `VM_FAULT_FALLBACK`，才继续往下走小页。**这解释了为什么 THP 叫"透明"**：它就嵌在普通缺页路径里，应用根本不知道自己拿的是大页还是小页。
- **最后 `handle_pte_fault`**：所有没走大页的，落到最后一级。

### 7.5.4 `handle_pte_fault`：三类 fault 的真实分流

最后一级是分类决策中枢。这是**真实源码**的核心分流（节选保留判断主干）：

```c
/* 【真实源码 v6.12 mm/memory.c 行 5732 起，节选】 */
static vm_fault_t handle_pte_fault(struct vm_fault *vmf)
{
	pte_t entry;
	/* ... 建立/读取 vmf->pte 和 orig_pte ... */

	if (!vmf->pte)
		return do_pte_missing(vmf);          /* ① PTE 完全没建：匿名页 or 文件页 */

	if (!pte_present(vmf->orig_pte))
		return do_swap_page(vmf);            /* ② 页被换出去了：从 swap 读回（major fault） */

	if (pte_protnone(vmf->orig_pte) && vma_is_accessible(vmf->vma))
		return do_numa_page(vmf);            /* ③ NUMA balancing 的“假缺页” */

	spin_lock(vmf->ptl);
	entry = vmf->orig_pte;
	/* ... pte_same 校验防并发 ... */
	if (vmf->flags & (FAULT_FLAG_WRITE|FAULT_FLAG_UNSHARE)) {
		if (!pte_write(entry))
			return do_wp_page(vmf);          /* ④ 写只读页：COW！（§7.7） */
		else if (likely(vmf->flags & FAULT_FLAG_WRITE))
			entry = pte_mkdirty(entry);      /* 本就可写：只置 dirty 位 */
	}
	entry = pte_mkyoung(entry);              /* 置 accessed 位，喂给 LRU */
	/* ... 写回 PTE，按需 flush spurious TLB ... */
}
```

而 `do_pte_missing` 把"匿名 vs 文件"分开（真实源码）：

```c
/* 【真实源码 v6.12 mm/memory.c 行 3960】 */
static vm_fault_t do_pte_missing(struct vm_fault *vmf)
{
	if (vma_is_anonymous(vmf->vma))
		return do_anonymous_page(vmf);   /* 匿名：堆/栈/匿名 mmap */
	else
		return do_fault(vmf);            /* 文件：走 vm_ops->fault 读文件页 */
}
```

把这几段拼起来，就是教科书里那张"三类 fault 分类图"的**真实代码对应**：

| 类型 | 真实代码路径 | 触发条件 | 代价根因 |
|---|---|---|---|
| minor fault（匿名） | `do_pte_missing → do_anonymous_page` | 匿名页首次访问 | 分配一物理页 + 清零 + 建 PTE，微秒级 |
| minor fault（文件已在 cache） | `do_pte_missing → do_fault` | 文件页已在 page cache | 只建 PTE 指过去，最便宜 |
| major fault（文件 / swap） | `do_fault`（要读盘）/ `do_swap_page` | 文件页不在 cache / 页被换出 | 受 I/O 延迟主导，毫秒级 |
| COW fault | `do_wp_page` | 写只读页（fork 后 / 私有文件映射写） | 可能 memcpy 一页，也可能零拷贝复用（§7.7.2） |
| NUMA hint fault | `do_numa_page` | NUMA balancing 故意设的 protnone | 不是真缺页，是为了触发页迁移统计 |

> **💡 坑点**：`perf stat -e page-faults` 是 minor + major 总和；`/proc/<pid>/stat` 里 `minflt`/`majflt` 分列。大量 major fault = 工作集超过 RAM 或在疯狂换页。但**注意 `do_numa_page` 这类"假缺页"也会被计入**——在开了 NUMA balancing 的机器上看到莫名其妙的高 minor fault，先排查是不是 NUMA hint fault 而非真的内存压力。

### 7.5.5 `do_anonymous_page`：匿名缺页的两条路（零页 vs 真分配）

最常见的 minor fault。看**真实源码**怎么对"读"和"写"区别对待：

```c
/* 【真实源码 v6.12 mm/memory.c 行 4735 起，大幅节选】 */
static vm_fault_t do_anonymous_page(struct vm_fault *vmf)
{
	struct vm_area_struct *vma = vmf->vma;
	struct folio *folio;

	if (vma->vm_flags & VM_SHARED)           /* 共享匿名但无 vm_ops，异常 */
		return VM_FAULT_SIGBUS;
	if (pte_alloc(vma->vm_mm, vmf->pmd))
		return VM_FAULT_OOM;

	/* ===== 读缺页：用全局零页，不分配新内存 ===== */
	if (!(vmf->flags & FAULT_FLAG_WRITE) && !mm_forbids_zeropage(vma->vm_mm)) {
		entry = pte_mkspecial(pfn_pte(my_zero_pfn(vmf->address),
						vma->vm_page_prot));    /* ← 指向全局 zero page，只读 */
		/* ... 装 PTE ... */
		goto setpte;
	}

	/* ===== 写缺页：才真正分配私有页 ===== */
	ret = vmf_anon_prepare(vmf);
	folio = alloc_anon_folio(vmf);           /* 分配（可能是多页 mTHP folio） */
	/* ... */
	nr_pages = folio_nr_pages(folio);
	__folio_mark_uptodate(folio);
	entry = mk_pte(&folio->page, vma->vm_page_prot);
	if (vma->vm_flags & VM_WRITE)
		entry = pte_mkwrite(pte_mkdirty(entry), vma);
	/* ... */
	folio_add_new_anon_rmap(folio, vma, addr, RMAP_EXCLUSIVE);
	folio_add_lru_vma(folio, vma);
setpte:
	set_ptes(vma->vm_mm, addr, vmf->pte, entry, nr_pages);
	/* ... */
}
```

逐行读出两个关键优化：

1. **读缺页用 `my_zero_pfn`（全局零页）**：`calloc` 一大块只读不写、或 BSS 段被读，内核根本不分配新页，所有这些虚拟页**共享同一个全局只读零页**。这是 demand paging 之上的二次优化——延迟到"写"才真正掏物理内存。一旦写，触发 COW（§7.7）从零页复制出私有页。
2. **`alloc_anon_folio` 可能返回多页 folio（mTHP）**：注意 `nr_pages` 不一定是 1。这是 6.8 起的 **multi-size THP**（见 §7.8.4），匿名缺页可以一次给一个 16 KB / 64 KB 的中等 folio，而非死板的 4 KB 或 2 MB。`set_ptes(..., nr_pages)` 一次装多个连续 PTE。

### 7.5.6 Demand Paging 的工程推论

1. **`mmap` 1 GB 文件物理内存消耗接近零**：只建 VMA，文件内容归 page cache 管，访问时才映射。
2. **`malloc(1GB)` 后立即 `free`：物理页从未分配**——因为没写过，`do_anonymous_page` 的"真分配"分支没跑过，free 只删 VMA。
3. **overcommit 成为可能**：Linux 默认允许承诺内存总量 > 物理 RAM，赌"大部分承诺不兑现"。`vm.overcommit_memory` 控制策略，OOM Killer 是赌输时的清算（见第 8 章）。

## 7.6 TLB 与 shootdown：为什么改页表比改内存贵

### 7.6.1 TLB 是什么，代价量级

4 级页表 = 每次访存最坏要 4 次额外内存读做 page walk。TLB（Translation Lookaside Buffer）缓存 VPN→PFN 映射：

```text
【示意，量级为典型值，随微架构浮动】
TLB 命中：             虚拟地址 → TLB → 物理地址          ~1 cycle
TLB miss（page walk）： MMU 走 4 级页表                  ~20–200 cycle（看各级是否命中 cache）
TLB miss + page fault：还要进内核 fault handler          ~微秒到数十微秒
```

### 7.6.2 TLB Shootdown：真实源码里的 lazy 跳过

§7.1.2 埋的线在这里收：**TLB 不参与 cache 一致性**，所以内核改页表后必须显式让所有相关 CPU 的 TLB 失效。多核场景靠 IPI（核间中断）广播，这就是 **TLB shootdown**。看 v6.12 **真实源码**怎么做、以及它最重要的优化：

```c
/* 【真实源码 v6.12 arch/x86/mm/tlb.c 行 903 起，节选】 */
STATIC_NOPV void native_flush_tlb_multi(const struct cpumask *cpumask,
					 const struct flush_tlb_info *info)
{
	count_vm_tlb_event(NR_TLB_REMOTE_FLUSH);
	/* ... trace ... */

	/*
	 * If no page tables were freed, we can skip sending IPIs to
	 * CPUs in lazy TLB mode. They will flush the CPU themselves
	 * at the next context switch.
	 *
	 * However, if page tables are getting freed, we need to send the
	 * IPI everywhere, to prevent CPUs in lazy TLB mode from tripping
	 * up on the new contents of what used to be page tables, while
	 * doing a speculative memory access.
	 */
	if (info->freed_tables)
		on_each_cpu_mask(cpumask, flush_tlb_func, (void *)info, true);
	else
		on_each_cpu_cond_mask(tlb_is_not_lazy, flush_tlb_func,
				(void *)info, 1, cpumask);
}
```

逐行读出这段为什么是"扩展性的隐形战场"：

- **`if (info->freed_tables)` 是整段的灵魂**：
  - **改权限 / 改映射但不释放页表**（`freed_tables == false`）→ 走 `else` 分支 `on_each_cpu_cond_mask(tlb_is_not_lazy, ...)`：**只给"非 lazy"的 CPU 发 IPI**。lazy TLB 的核（跑内核线程 / idle，没在用这个用户 mm）跳过，它下次 context switch 自然会刷。
  - **释放了页表本身**（munmap 大段、进程退出，`freed_tables == true`）→ 走 `on_each_cpu_mask(..., true)`：**给所有核（含 lazy）发 IPI**。为什么？注释写得很清楚——页表内存被释放后可能被复用作别的用途，lazy 核如果**推测执行**（speculative memory access）摸到"曾经是页表、现在是别的数据"的内存，会出灾难性错误。
- **这就是为什么 munmap 大段地址空间特别贵**：不仅 flush 范围大，而且大概率 `freed_tables=true`，必须惊动每一个核，包括那些本来可以躺平的 lazy 核。

入口 `flush_tlb_mm_range` 还有一个范围阈值优化（真实源码）：

```c
/* 【真实源码 v6.12 arch/x86/mm/tlb.c 行 995 起 + 行 951，节选】 */
unsigned long tlb_single_page_flush_ceiling __read_mostly = 33;

void flush_tlb_mm_range(struct mm_struct *mm, unsigned long start,
				unsigned long end, unsigned int stride_shift,
				bool freed_tables)
{
	/* Should we flush just the requested range? */
	if ((end == TLB_FLUSH_ALL) ||
	    ((end - start) >> stride_shift) > tlb_single_page_flush_ceiling) {
		start = 0;
		end = TLB_FLUSH_ALL;        /* ← 范围超过 33 页，干脆整个 mm 全刷 */
	}
	/* ... inc_mm_tlb_gen(mm) 推进 TLB 代号，再 flush_tlb_multi(...) ... */
}
```

逐行读出排队论级别的取舍：

- **`tlb_single_page_flush_ceiling = 33`**：要 flush 的页数 > 33 时，内核**放弃逐页 INVLPG，直接整表全刷**。为什么有这个魔数？逐页 `INVLPG` 每条有固定开销，flush 太多页时"N 条 INVLPG"的总成本超过"一次全刷 + 后续 TLB 重新填充"的成本。33 是经验调出来的拐点（`__read_mostly` 可调）。这是典型的"批量操作选点查 vs 全清"的成本权衡。

> **🎯 面试加分**：被问"munmap 为什么慢"，浅答"要 flush TLB"；深答要点出两层——(1) `freed_tables=true` 逼迫给**所有核**（含 lazy）发 IPI，代价是 IPI 往返 + 等所有核确认的**同步等待**，核越多越贵；(2) 范围超 `tlb_single_page_flush_ceiling`(=33) 直接全刷，导致 munmap 后一段时间该进程所有访存都 TLB miss、要重新 page walk。两个成本叠加。

### 7.6.3 缓解 shootdown 的几条路

- **lazy TLB**：上面已见——内核线程 / idle 不持有用户 mm，可跳过 IPI（仅 `freed_tables=false` 时）。
- **`MADV_FREE`**：把"立即回收 + flush"改成"打标记，有内存压力再真回收"，省掉同步点（与 `MADV_DONTNEED` 的区别就在这——后者立即生效、必须 flush）。
- **批量 flush（mmu_gather）**：`tlb_gather_mmu` 一类机制把多次 unmap 的 flush 攒成一次广播，减少 IPI 次数。

### 7.6.4 PCID/ASID：进程切换不全刷

早期内核进程切换必须全刷 TLB（不同进程映射完全不同），成本极高。x86 PCID（Process Context ID，12 bit）让 TLB entry 带进程标识，切换时不刷，回来直接命中。ARM 是 ASID。

**版本/硬件史（容易答得似是而非）**：PCID 这个 CPUID 特性位 Westmere（2010）就有，但 **Linux 真正用 PCID 省 TLB flush 是 4.14（2017）才进主线**，且高效"按 PCID 精确失效单条翻译"依赖 **INVPCID 指令（Haswell 起）**。它晚到 4.14 不是巧合——**2018 年 Meltdown 的 KPTI 修复让每次系统调用都要换 CR3（页表根），TLB 抖动骤增，PCID 从"锦上添花"变成"刚需"**（KPTI 见 §7.10.2）。两件事被同一波安全事件推着落地。

## 7.7 mmap、匿名映射与 COW（把"何时复制"钉到源码）

### 7.7.1 mmap 两种用法 + SHARED/PRIVATE 差异

```c
/* 【示意，非逐字】 */
// 文件映射：物理页 = page cache 对应文件页
void *p = mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_SHARED, fd, off);
// 匿名映射：纯内存，按需分配，初始零页
void *q = mmap(NULL, len, PROT_READ|PROT_WRITE, MAP_PRIVATE|MAP_ANONYMOUS, -1, 0);
```

| | MAP_SHARED | MAP_PRIVATE |
|---|---|---|
| 写操作 | 直接改底层（文件 / 共享匿名） | 触发 COW，改动只本进程可见 |
| fork 后 | 子写也影响父（共享物理页） | 子写触发 COW，产生私有副本 |
| 典型场景 | IPC、数据库 WAL、共享内存 | malloc、私有栈、私有文件映射 |

### 7.7.2 COW 的真相：先问"能不能独占复用"，再决定复不复制

这是本章纠正"浮于表面"认知的**关键一节**。教科书说"写时复制"，但 v6.12 的 `do_wp_page` 真实逻辑是"**写时先问这页还有没有别人用，没别人就零拷贝复用，有别人才复制**"。看真实源码：

```c
/* 【真实源码 v6.12 mm/memory.c 行 3655 起，大幅节选保留决策主干】 */
static vm_fault_t do_wp_page(struct vm_fault *vmf)
{
	struct vm_area_struct *vma = vmf->vma;
	struct folio *folio = NULL;
	/* ... userfaultfd-wp 处理略 ... */

	vmf->page = vm_normal_page(vma, vmf->address, vmf->orig_pte);
	if (vmf->page)
		folio = page_folio(vmf->page);

	/* 共享映射：本就该让大家看到写，不 COW，直接置可写 */
	if (vma->vm_flags & (VM_SHARED | VM_MAYSHARE)) {
		if (!vmf->page)
			return wp_pfn_shared(vmf);
		return wp_page_shared(vmf, folio);
	}

	/* 私有映射：能独占复用就复用，否则才复制 */
	if (folio && folio_test_anon(folio) &&
	    (PageAnonExclusive(vmf->page) || wp_can_reuse_anon_folio(folio, vma))) {
		if (!PageAnonExclusive(vmf->page))
			SetPageAnonExclusive(vmf->page);
		wp_page_reuse(vmf, folio);       /* ← 零拷贝！只翻 Writable 位 */
		return 0;
	}

	/* 实在不能复用，才复制 */
	if (folio)
		folio_get(folio);
	pte_unmap_unlock(vmf->pte, vmf->ptl);
	return wp_page_copy(vmf);             /* ← 真正 memcpy 一页 */
}
```

而"能不能复用"的判断 `wp_can_reuse_anon_folio` 才是精华（真实源码）：

```c
/* 【真实源码 v6.12 mm/memory.c 行 3585 起，节选】 */
static bool wp_can_reuse_anon_folio(struct folio *folio, struct vm_area_struct *vma)
{
	if (folio_test_large(folio))
		return false;                          /* 大 folio 不复用，整体让回去 */

	if (folio_test_ksm(folio) || folio_ref_count(folio) > 3)
		return false;                          /* 引用多，肯定有别人用 */
	if (!folio_test_lru(folio))
		lru_add_drain();                       /* 排空 LRU 缓存，让 refcount 准确 */
	if (folio_ref_count(folio) > 1 + folio_test_swapcache(folio))
		return false;                          /* 还有别的引用，不能独占 */
	if (!folio_trylock(folio))
		return false;
	if (folio_test_swapcache(folio))
		folio_free_swap(folio);
	if (folio_test_ksm(folio) || folio_ref_count(folio) != 1) {
		folio_unlock(folio);
		return false;                          /* 最终确认：refcount 必须恰好是 1 */
	}
	/* refcount == 1，只有我在用，独占！直接复用 */
	folio_move_anon_rmap(folio, vma);
	folio_unlock(folio);
	return true;
}
```

逐行读出"为什么 Redis BGSAVE 内存翻倍是有条件的"：

- **核心判据是 `folio_ref_count(folio) == 1`**：fork 后父子共享一页，refcount=2。父进程写 → refcount 还是 2 → 不能复用 → `wp_page_copy` 复制。**但是**，如果此时子进程已经退出 / 已经对该页做过自己的 COW，该页 refcount 降回 1 → 父进程再写 → **`wp_page_reuse` 零拷贝**，不再复制。
- **所以 Redis BGSAVE 内存放大的真实变量是**：快照期间父进程**写到多少个不同的页**（每个被写的共享页触发一次复制），以及**子进程是否还在引用这些页**。最坏情况（子进程一直活着、父进程写遍全部数据）才接近 2x。这比"fork 后写就翻倍"的粗糙说法准确得多。
- **`folio_test_large(folio) → return false`**：THP 大页不走复用路径，整体让回去再重新分配。**这正是 §7.8 "THP 放大 COW 代价"的源码根因**——2 MB 大页只要被写一下且不能整体复用，就得复制整个 2 MB，是 4 KB 页的 512 倍。

### 7.7.3 fork 时如何把页设成 COW（真实源码）

COW 的另一半在 fork。看子进程复制父进程页表时怎么把可写页改成只读（真实源码）：

```c
/* 【真实源码 v6.12 mm/memory.c 行 948 起，节选】 */
static __always_inline void __copy_present_ptes(struct vm_area_struct *dst_vma,
		struct vm_area_struct *src_vma, pte_t *dst_pte, pte_t *src_pte,
		pte_t pte, unsigned long addr, int nr)
{
	struct mm_struct *src_mm = src_vma->vm_mm;

	/* If it's a COW mapping, write protect it both processes. */
	if (is_cow_mapping(src_vma->vm_flags) && pte_write(pte)) {
		wrprotect_ptes(src_mm, addr, src_pte, nr);   /* ← 父进程的 PTE 也清 Writable */
		pte = pte_wrprotect(pte);                    /* ← 子进程的 PTE 清 Writable */
	}
	/* ... 共享映射标 clean、清 accessed 位 ... */
	set_ptes(dst_vma->vm_mm, addr, dst_pte, pte, nr);
}
```

逐行读出两个易错点：

- **`wrprotect_ptes(src_mm, ...)` 连父进程一起写保护**——注释 `write protect it both processes` 说得直白。很多人以为只改子进程，其实父进程的 PTE 也被清了 Writable，否则父进程写就直接改了共享页，子进程看到不该看到的写。
- **`is_cow_mapping(...) && pte_write(pte)` 才处理**：本来就只读的页（如代码段）不用动；共享映射（`MAP_SHARED`）不是 COW，走另一分支标 clean。

### 7.7.4 设计考古：Dirty COW（CVE-2016-5195）——一个 11 年的 COW 竞态

**史料（已核实）**：据 [LWN 906852 系列的姊妹文 LWN《Dirty COW and clean commit messages》（2016-10-21，Corbet）](https://lwn.net/Articles/704231/) 及 Linus 的修复 commit `19be0eaffa3a`，Dirty COW 是 COW 处理与 `get_user_pages()` 之间的竞态：通过精心的时序，本地攻击者能把对只读文件映射的写，"漏"成对底层文件的真实写，实现提权。

Linus 的 commit message 是**真实史料**，读起来很有教育意义：

```text
【真实 commit 19be0eaffa3ac7d8eb6784ad9bdbc7d67ed8e619】
Linus Torvalds, 2016-10-13,
"mm: remove gup_flags FOLL_WRITE games from __get_user_pages()"

This is an ancient bug that was actually attempted to be fixed once
(badly) by me eleven years ago in commit 4ceb5db9757a ("Fix
get_user_pages() race for write access") but that was then undone due to
problems on s390 ...

Also, the VM has become more scalable, and what used a purely
theoretical race back then has become easier to trigger.

To fix it, we introduce a new internal FOLL_COW flag to mark the "yes,
we already did a COW" rather than play racy games with FOLL_WRITE ...
```

逐行读出工程史的教训：

- **`an ancient bug ... eleven years ago ... badly`**：Linus 自陈 11 年前（2005）修过一次但修坏了，因 s390 问题被 revert。**竞态 bug 潜伏十余年**。
- **`the VM has become more scalable, ... easier to trigger`**：早年是纯理论竞态，触发不了；随着内核**并发能力增强**（更多核、更细粒度锁），原本几乎不可能的时序窗口变得可触发。**这是"性能优化反而暴露老竞态"的经典案例**——可扩展性提升让蛰伏的 race 现形。
- **`FOLL_COW flag ... rather than play racy games with FOLL_WRITE`**：修法是引入 `FOLL_COW` 显式标记"已经 COW 过了"，用 PTE 的 dirty 位验证，而不是靠 `FOLL_WRITE` 的临时清/置玩时序游戏。

## 7.8 大页：HugePages 与 THP

### 7.8.1 为什么需要大页（算 TLB 账）

x86_64 支持 4 KB / 2 MB / 1 GB 三种页。大量连续访问时（DB buffer pool、KVM 内存），4 KB 页产生海量 TLB miss：

```text
【示意，TLB 容量为典型值】
DB buffer pool: 64 GB
4 KB 页数 = 64Gi / 4Ki = 16,777,216 个 PTE
2 MB 页数 = 64Gi / 2Mi = 32,768 个 PTE        （少 512 倍）

L2 TLB 典型 ~1024–2048 entry
→ 4 KB 页时工作集远超 TLB，几乎每次访问都 miss → page walk
→ 2 MB 页时少 512 倍，命中率大增；且大页可少走一级页表（PMD 直接映射）
```

### 7.8.2 静态 HugePages vs THP

**静态 HugePages（hugetlbfs）**：启动时预留、不参与常规分配、永不 swap；应用经 `MAP_HUGETLB` 或 hugetlbfs 显式用。Oracle / KVM / DPDK 标配。

```bash
cat /proc/meminfo | grep Huge        # HugePages_Total/Free
sysctl -w vm.nr_hugepages=1024        # 运行时预留（碎片化时可能拿不满，最稳是 boot 参数 hugepages=1024）
```

**THP（Transparent Huge Pages）**：内核自动把连续 4 KB 合并为 2 MB，对应用透明。

```bash
cat /sys/kernel/mm/transparent_hugepage/enabled   # [always] madvise never
echo madvise > /sys/kernel/mm/transparent_hugepage/enabled
echo defer+madvise > /sys/kernel/mm/transparent_hugepage/defrag
```

### 7.8.3 设计考古：THP 从哪来，社区怎么辩的

**史料（已核实）**：据 [LWN 423584《Transparent huge pages in 2.6.38》（2011-01-19，Corbet）](https://lwn.net/Articles/423584/)，THP 由 **Andrea Arcangeli** 实现，合入 **2.6.38**。三大组件：

1. **缺页时分配**——"When a page fault occurs, the kernel will attempt to allocate a huge page to satisfy it."（这正是 §7.5.3 `create_huge_pmd` 的来源）
2. **khugepaged 线程**——后台扫描、把小页合并成大页。
3. **拆分机制**——"huge pages must be swappable"，回收时把大页拆回小页。

社区当年的辩论焦点，LWN 423584 原话给了答案：Mel Gorman 的基准显示 THP "improvements of up to 10% or so in some situations"，但"the results were not as good as could be obtained with hugetlbfs, **but THP is much more likely to actually be used**"。

> **这句话是 THP 全部设计哲学的浓缩**：hugetlbfs 性能更好但要管理员手工预留、应用改代码，**实际没人用**；THP 性能次优但零配置、自动生效，**真能跑起来**。又一个"次优但能落地" > "最优但用不上"的案例（和 §7.4.3 per-VMA lock 同一种工程价值观）。

### 7.8.4 设计考古：mTHP（multi-size THP，6.8）——4 KB 和 2 MB 之间的中间档

**史料（已核实）**：commit `19eaf44954df`（Ryan Roberts / Arm，2023-12-07，"mm: thp: support allocation of anonymous multi-size THP"）引入 mTHP——大于基页、小于 PMD 的中间尺寸 folio。这是**真实 commit message**：

```text
【真实 commit 19eaf44954df64f9bc8dec398219e15ad0811497】
We call this new THP extension "multi-size THP" (mTHP).

mTHP continues to be PTE-mapped, but in many cases can still provide
similar benefits to traditional PMD-sized THP: Page faults are
significantly reduced (by a factor of e.g. 4, 8, 16, etc. ...), but
latency spikes are much less prominent because the size of each page
isn't as huge as the PMD-sized variant and there is less memory to clear
in each page fault. ...
Some architectures also employ TLB compression mechanisms ... In this
case, TLB misses will occur less often.
```

逐行读出 mTHP 解决了 THP 的什么痛：

- **`latency spikes are much less prominent`**：传统 2 MB THP 缺页时要清零 2 MB、COW 时要复制 2 MB，造成延迟尖刺。mTHP 用 16 KB / 64 KB 中间档，**缺页/COW 的单次工作量小得多**，延迟平滑。这直接缓解了 §7.8.5 列的"THP 对延迟敏感服务有害"。
- **`continues to be PTE-mapped`**：mTHP 仍用普通 PTE（不像 2 MB THP 用 PMD entry），靠"连续多个 PTE 指向连续物理页"。
- **`TLB compression`**：ARM 的 contiguous PTE（contig bit）能把一组连续对齐的 PTE 在 TLB 里压成一条 entry——这是 §7.5.5 `do_anonymous_page` 里 `alloc_anon_folio` 返回多页 folio 的硬件收益来源。

### 7.8.5 THP 在哪些场景有害（带源码根因）

- **随机小 IO（MongoDB / PostgreSQL）**：2 MB 大页的 COW 代价是 4 KB 的 512 倍——根因见 §7.7.2 `wp_can_reuse_anon_folio` 对 `folio_test_large` 直接 `return false`，大页 COW 必须整体复制。
- **延迟敏感服务**：khugepaged 后台合并 + compaction 在内存争用时引发数百毫秒延迟。mTHP（§7.8.4）部分缓解。
- **Redis**：与 COW 叠加，BGSAVE 期间大页 COW 让内存暴涨更猛（每个被写的大页复制 2 MB）。

**业界共识**：Oracle / MongoDB / Redis 官方文档建议关 THP；这类应用用静态 HugePages。

## 7.9 ASLR（地址空间随机化）

### 7.9.1 设计考古 + 没有 ASLR 的世界

攻击者若知道 `libc` 的 `system()` 固定在 `0x40012345`，overflow 一个返回地址就能 ROP 劫持。ASLR 让这些地址每次启动都变。

**史料（已核实）**：据 [LWN 190139 等及 Wikipedia 交叉印证](https://lwn.net/Articles/190139/)，用户态 ASLR 由 **Arjan van de Ven** 提交，**2.6.12（2005）** 进主线；思想源自 2001 年的 **PaX** 项目和 Ingo Molnar 的 Exec Shield。内核自身的 KASLR 晚得多（3.14，2014）。

### 7.9.2 真实源码：熵从哪来

```bash
cat /proc/sys/kernel/randomize_va_space   # 0=关；1=部分(stack/mmap)；2=完全(默认)
```

v6.12 的 x86 随机化实现（**真实源码**）：

```c
/* 【真实源码 v6.12 arch/x86/mm/mmap.c 行 70 起，节选】 */
static unsigned long arch_rnd(unsigned int rndbits)
{
	if (!(current->flags & PF_RANDOMIZE))
		return 0;
	return (get_random_long() & ((1UL << rndbits) - 1)) << PAGE_SHIFT;
}

unsigned long arch_mmap_rnd(void)
{
	return arch_rnd(mmap_is_ia32() ? mmap32_rnd_bits : mmap64_rnd_bits);
}
```

配套的熵位数写在 Kconfig（**真实源码**）：

```text
/* 【真实源码 v6.12 arch/x86/Kconfig 行 341 起】 */
config ARCH_MMAP_RND_BITS_MIN
	default 28 if 64BIT      /* 64 位下 mmap 区随机化最少 28 bit */
	default 8
config ARCH_MMAP_RND_BITS_MAX
	default 32 if 64BIT      /* 最多 32 bit */
	default 16
```

栈的随机化掩码（**真实源码**）：

```c
/* 【真实源码 v6.12 arch/x86/include/asm/elf.h 行 330】 */
/* 1GB for 64bit, 8MB for 32bit */
#define __STACK_RND_MASK(is32bit) ((is32bit) ? 0x7ff : 0x3fffff)
```

逐行读出"熵到底多少"的精确答案：

- **mmap 区（含共享库基址）**：64 位下 **28–32 bit** 熵（`ARCH_MMAP_RND_BITS`），`<< PAGE_SHIFT` 后随机偏移粒度是页。
- **栈**：`0x3fffff` 掩码 = 22 bit，注释明说 **64 位栈随机化范围 1 GB**（`0x3fffff` 页 × 4 KB ≈ 16 GB 地址范围，实际偏移受其他约束）。
- **`PF_RANDOMIZE` 没置位就 `return 0`**：`setarch -R` 或 `echo 0 > randomize_va_space` 关掉时，这里直接返回零偏移。

### 7.9.3 ASLR 的代价、局限、真坑

**代价**：相同库在不同进程虚拟地址不同 → TLB entry 不能跨进程共享（**物理页能共享，翻译缓存不能**——又是 §7.1.2 那条"TLB 是私有的"）。

**局限**：64 位 mmap 熵 28–32 bit，理论可暴力穷举（代价高）；信息泄漏漏洞（Spectre、格式化字符串）可泄漏基址绕过；需与 CFI + Stack Canary + NX 组合才成完整防线。

> **💡 真坑**：`ulimit -s unlimited` 会影响栈随机化范围（`mmap_base` 计算里 `rlim_stack->rlim_cur` 参与，见 §7.9.2 未列的 `mmap_base()`）；Valgrind 等调试工具会关 ASLR 求确定性地址。生产中别手动关 ASLR，代价不值。

## 7.10 落地：生产排障与调优

### 7.10.1 地址空间分析工具

```bash
cat /proc/<pid>/maps              # VMA 列表（地址范围+权限+映射名）
cat /proc/<pid>/smaps             # 每 VMA 详细物理占用（PSS 等）
cat /proc/<pid>/smaps_rollup      # 全进程内存摘要（算 PSS 最快）
perf stat -e page-faults,minor-faults,major-faults -p <pid>
perf stat -e dTLB-load-misses,dTLB-store-misses -p <pid>
grep -E 'nr_tlb|vma_lock' /proc/vmstat   # TLB 远程 flush 次数 + per-VMA lock 命中/retry
```

> `nr_tlb_remote_flush` 对应 §7.6.2 `count_vm_tlb_event(NR_TLB_REMOTE_FLUSH)`；`vma_lock_success/abort/retry` 对应 §7.5.2 的计数。这俩是判断"TLB shootdown 风暴"和"per-VMA lock 失效率"的一手指标。

### 7.10.2 设计考古：KPTI（Meltdown 修复遗产）

**史料（已核实）**：据 [LWN 741878《The current state of kernel page-table isolation》（2017-12-20，Corbet）](https://lwn.net/Articles/741878/)，KPTI（Kernel Page-Table Isolation）源自 **KAISER 补丁**（Gruss 等，2017-10 公开），为修 Meltdown 而生。机制：

> 每进程两套页表——内核态用完整页表；用户态用一套"sanitized PGD"，内核空间部分"mostly absent"且标 NX。

代价：LWN 741878 估约 **5% 运行时开销**（`nopti` 可关）；每次系统调用多一次 CR3 写 ≈ 部分 TLB flush。**这正是 §7.6.4 PCID 从"锦上添花"变"刚需"的直接原因**——没有 PCID，KPTI 的 CR3 来回切换会把 TLB 抖到崩溃。

### 7.10.3 smaps 关键字段（判断泄漏 / 算公平内存）

```text
/proc/<pid>/smaps 片段：
Size: 4096 kB        VMA 覆盖虚拟范围
Rss: 2048 kB         当前在物理内存的页
Pss: 1024 kB         按共享数分摊（最公平的进程内存度量）
Private_Dirty: 1024  私有已改（swap 候选，泄漏排查重点）
Anonymous: 2048      匿名页（非文件映射）
Swap: 0              已换出量
```

判泄漏：看 `Anonymous + Private_Dirty` 增长趋势；`Pss` 比 `Rss` 公平（共享库不会被每个进程重复计满）。

### 7.10.4 排障：进程内存一直涨

```text
1. pmap -x <pid>            快速总览 RSS/Dirty 分布
2. /proc/<pid>/smaps        按 VMA 的 Private_Dirty 找大头
3. valgrind / ASan          泄漏检测
4. bpftrace 跟 mmap/brk     看是谁在申请（以下为示意）
   bpftrace -e 'tracepoint:syscalls:sys_exit_mmap /pid==TARGET/ {
       printf("mmap ret=%llx\n", args->ret); }'
```

## 7.11 面试视角：Staff 级高频题

### Q1：缺页处理的完整路径，从硬件入口讲到分类。
**框架**：CPU #PF → CR2 存出错地址 → `exc_page_fault`（§7.5.1）→ `handle_page_fault` 按内核/用户地址分流 → `do_user_addr_fault` **先试 per-VMA lock 快路，失败回退 mmap_lock**（§7.5.2，6.4+ 关键演进）→ `handle_mm_fault → __handle_mm_fault` 按级 alloc 页表、每级试大页（§7.5.3）→ `handle_pte_fault` 分流 minor/major/COW/NUMA（§7.5.4）。**加分**：点出 per-VMA lock 是 maple tree RCU 底子上建的，串起 §7.4 演进史。

### Q2：fork 后 COW 完整机制，为什么 Redis BGSAVE 不一定翻倍。
**框架**：fork 时 `__copy_present_ptes` 把**父子两边**PTE 都 wrprotect（§7.7.3）→ 写时 `do_wp_page` 先 `wp_can_reuse_anon_folio` 查 refcount，**==1 就零拷贝复用，>1 才复制**（§7.7.2）→ 所以放大量取决于"父进程写多少不同页 + 子进程是否仍引用"，最坏才 2x。**加分**：THP 下 `folio_test_large` 直接拒绝复用，大页 COW 必复制 2 MB = 512 倍代价。

### Q3：TLB shootdown 是什么，为什么 munmap 大段慢。
**框架**：TLB 不参与 cache 一致性（§7.1.2）→ 改页表必须软件广播 IPI → `native_flush_tlb_multi` 里 `freed_tables` 分支：拆页表时**给所有核（含 lazy）发 IPI**，防 lazy 核推测访问已释放页表内存（§7.6.2）→ 代价在 IPI 往返 + 等所有核确认；且范围 > `tlb_single_page_flush_ceiling`(33) 直接整表全刷。

### Q4：VMA 怎么组织的？（陷阱题）
**框架**：**先反问/声明版本**——6.1 前 rbtree + 双向链表，6.1 起 maple tree（commit `54a611b6059`，§7.4.2）。换的动机：rbtree 不支持区间、难 lockless、遍历低效才额外挂链表；maple tree 是 RCU-safe range B-tree，分支因子 10/16、节点 256B 对齐 cache line。**加分**：maple tree 的 RCU 底子是为 6.4 per-VMA lock（commit `5e31275cc99`）铺路。

### Q5：5 级页表开了就能用 128 PB 吗？
**框架**：不能。LA57 由 Kirill Shutemov 合入（~4.12，LWN 717293）→ 但 `mmap` 默认只在低 47 位分配，**只有显式传 >47 位 hint 才映射高区**（§7.3.4），为兼容把指针高位当 tag 的老程序 → 且 5 级模式下 `PGDIR_SHIFT` 从编译常量变运行期变量，有微小性能代价。

### Q6：minor/major fault 区别与监控，注意什么陷阱。
**框架**：minor = `do_anonymous_page`/cache 命中的 `do_fault`，微秒级；major = 要读盘的 `do_fault`/`do_swap_page`，毫秒级（§7.5.4）→ `/proc/pid/stat` 的 minflt/majflt。**陷阱**：开了 NUMA balancing 时 `do_numa_page` 的"假缺页"也计入 minor fault，别误判成内存压力。

### Q7：demand paging 与 overcommit、OOM 的关系。
**框架**：缺页惰性分配（`do_anonymous_page` 读缺页用全局零页、写才真分配，§7.5.5）→ overcommit 允许承诺 > 物理 RAM → 真耗尽 + swap 用完 → OOM killer 按 oom_score 选杀。**加分**：连中间页表都是 demand 分配的（`__handle_mm_fault` 里 `*_alloc` 失败也 OOM）。

### Q8：MAP_SHARED 和 MAP_PRIVATE 的物理内存视角差异。
**框架**：SHARED 写直接改 page cache 物理页，所有映射者立即可见（`do_wp_page` 走 `wp_page_shared`）；PRIVATE 写触发 COW 分配私有页，与 page cache 脱钩（`wp_page_copy`）。IPC 设计的核心取舍。

## 7.12 实战项目（P0 / P1）

### P0：解剖一个进程的地址空间

**目标**：用 `/proc/<pid>/maps` 和 `smaps` 把一个真实进程（nginx 或 python）的地址空间画全。

```bash
nginx & PID=$!
```

### P1：观测 per-VMA lock 与 TLB shootdown（呼应源码）

**验收**：解释 retry 升高说明地址空间改动频繁打断了缺页快路；解释为什么 munmap 密集型负载会推高 `nr_tlb_remote_flush`。

## 7.13 设计题

**题目**：内存密集 KV 存储（类 Redis），buffer pool 32 GB，部署在 64 GB 机器，定期 fork 子进程做全量快照。设计内存策略（大页选型、NUMA、swap、OOM 防护），重点说明快照期如何控制 COW 放大，给监控指标和阈值。

**考查点**：
- **大页选型**：静态 HugePages（buffer pool 预留）优于 THP——根因是 §7.7.2 大页 COW 必整体复制 2 MB。若用 THP 至少 `madvise` 模式 + `defer+madvise` defrag。
- **COW 放大量化**：放大 ≈ 快照期父进程写到的不同页数 × 页大小（§7.7.2 修正了"必然 2x"的误解）。监控 `Private_Dirty` 增量。
- **NUMA**：绑定减少远端访问；注意 `do_numa_page` 的 hint fault 噪声。
- **swap / OOM**：禁 swap 换确定性延迟，但要 `vm.overcommit_memory=1` 确保 fork 不因 overcommit 检查失败（fork 要复制页表，悲观估计可能误判内存不足）。

## 7.14 系统题

**场景**：线上服务 OOM Kill，但监控显示 container memory 只到限制 60%，怎么排查？

**思路**：
```text
1. cgroup memory.usage 包含 RSS(匿名) + page cache；kmem 某些版本单独计
   → cgroup v2 看 memory.stat 的 anon/file/slab/kernel 分项（v1/v2 字段名差异，待核具体名）
2. dmesg | grep -A20 "Out of memory" → 是 cgroup OOM 还是系统 OOM，看 oom_score_adj
3. slab 爆炸（dentry/inode cache，不算进程 RSS 但算 cgroup）：slabtop / /proc/slabinfo
4. 大页/连续内存分配失败（free 够但拿不到连续大页）：cat /proc/buddyinfo
5. 页表本身的内存：进程地址空间巨大且稀疏时页表可观，见 /proc/pid/status 的 VmPTE
```

**要点**：OOM 不一定"内存用完"，可能是 slab 爆、大页碎片分配失败、cgroup 计算口径与监控不一致、或页表内存（VmPTE）被忽略。

## 7.15 代码题：读取并格式化 /proc/self/maps

```c
/* 【示意代码，非逐字】读自身 /proc/self/maps，统计各类 VMA */
/* 编译: gcc -O2 -Wall -o maps_reader maps_reader.c */
#include <stdio.h>
#include <string.h>

int main(void) {
    FILE *maps = fopen("/proc/self/maps", "r");
    if (!maps) { perror("fopen"); return 1; }
    char line[512];
    long total_vma = 0, anon_kb = 0, file_kb = 0;

    while (fgets(line, sizeof(line), maps)) {
        unsigned long start, end;
        char perms[8], rest[256];
        rest[0] = '\0';   /* 关键：无 pathname 时 %[^\n] 不匹配，必须先清零，否则读栈上垃圾(UB) */
        /* 格式: start-end perms offset dev inode [pathname] */
        if (sscanf(line, "%lx-%lx %7s %*s %*s %*s %255[^\n]",
                   &start, &end, perms, rest) >= 3) {
            long size_kb = (end - start) / 1024;
            char *name = (rest[0] != '\0') ? rest : "[anonymous]";
            printf("%016lx-%016lx %-6s %8ld KB  %s\n",
                   start, end, perms, size_kb, name);
            total_vma++;
            if (rest[0] == '\0' || rest[0] == '[') anon_kb += size_kb;
            else file_kb += size_kb;
        }
    }
    fclose(maps);
    printf("\n总 VMA: %ld\n匿名/特殊: %ld KB\n文件映射: %ld KB\n",
           total_vma, anon_kb, file_kb);
    /* TODO1: 改读 /proc/self/smaps 统计 Private_Dirty */
    /* TODO2: 找出最大 VMA */
    /* TODO3: 检测可执行的匿名映射（潜在 shellcode 注入面） */
    return 0;
}
```

**暗坑**（区分"写过 C"和"写过能上线的 C"）：
- **`%[^\n]` 必须带宽度上限**：写 `%255[^\n]` 不写裸 `%[^\n]`，否则超长 pathname 栈溢出——CWE-787，真实 CVE 常见来源。
- **匹配失败缓冲区是脏的**：匿名行无第五字段，`sscanf` 返回 3 而非 4，没预先 `rest[0]='\0'` 就读到栈残留值——典型 UB，表现为"名字栏偶尔冒乱码"。
- **地址是 16 进制用 `%lx`**，不是 `%ld`。
- **`/proc/self/smaps` 每 VMA 多行**（header + 缩进字段），要状态机解析，不能一行一个 `sscanf` 套到底。

## 7.16 未来演进

### 7.16.1 per-VMA lock 的扩展
6.4 落地后只覆盖匿名页缺页快路（§7.4.3 LWN 906852 说初版"only anonymous pages not in swap"）。后续在持续扩到文件映射、swap-in 等更多路径，目标是把 `mmap_lock` 的接触面压到最小。这是当前 mm 子系统最活跃的可扩展性战场。

### 7.16.2 mTHP 的推进
6.8 的 mTHP（§7.8.4）把"大页"从"4 KB 或 2 MB 二选一"变成连续光谱（16K/32K/64K…）。配合 ARM contiguous PTE，在不引入 2 MB THP 延迟尖刺的前提下削减 TLB miss。预计成为默认匿名内存形态。

### 7.16.3 5 级页表的普及
随 TB 级内存机器普及，LA57（§7.3.4）从"数据中心特例"走向更常见。挑战仍是兼容"指针高位当 tag"的用户态程序。

### 7.16.4 eBPF 内存可观测性
bpftrace / libbpf 可挂钩缺页处理、mmap、TLB flush 路径，做生产级细粒度追踪，无需改应用——正取代传统 perf + /proc 组合（详见第 23/24 章）。

## 本章心法

> **虚拟内存是一张延迟兑现的空头支票，但"延迟"和"兑现"都是有源码、有代价、有竞态的具体动作**：缺页是兑付（`handle_mm_fault` 的递归下降），COW 是"先问能不能赖账再决定要不要复印"（`wp_can_reuse_anon_folio` 的 refcount 判据），TLB shootdown 是"通知所有清算所旧支票作废"（`native_flush_tlb_multi` 的 IPI 广播），而 VMA 管理三代演进（rbtree → maple tree → per-VMA lock）则是清算系统本身的扩容史。
