## 一、设计考古：WebRTC 从哪里来，为什么长这个样子

### 1.1 起源：Google 收购 GIPS + On2

2010 年 5 月，Google 以 6820 万美元收购 Global IP Solutions (GIPS)，获得其核心音视频编解码引擎（NetEQ、iSAC、iLBC、VP8 前身）。同年收购 On2 Technologies，获得 VP8 视频编解码器，并随即将其开源为 libvpx。

2011 年 6 月，Google 将整套实时通信技术开源为 **WebRTC 项目**，同步向 W3C 和 IETF 推进标准化：
- **W3C WebRTC WG**：定义浏览器 JavaScript API（`RTCPeerConnection`、`getUserMedia`）
- **IETF RTCWEB WG**：定义传输协议（ICE、DTLS-SRTP、SCTP over DTLS）

**动机**：打破 Flash/Skype 垄断，让浏览器原生支持音视频通话，消灭插件依赖。

参考文献：
- [RFC 8825] Overview: Real-Time Protocols for Browser-Based Applications (Justin Uberti, Cullen Jennings, 2021)
- [W3C WebRTC 1.0 Recommendation, 2021-01-26]
- libwebrtc 代码库：https://chromium.googlesource.com/external/webrtc

### 1.2 为什么信令协议刻意不标准化

这是 WebRTC 最让初学者困惑的设计决策：**标准里没有信令协议**。

**历史背景**：VoIP 领域曾经历惨烈的协议战争：
- SIP（RFC 3261，2002）：IETF 推动，电信运营商阵营
- H.323：ITU-T 推动，企业视频会议阵营  
- Jingle（XEP-0166）：XMPP 扩展，IM 阵营
- 三套协议互不兼容，网关层极其复杂，开发者苦不堪言

**RTCWEB WG 的选择**（见 RFC 8825 Section 4）：

> "This specification does not define a complete solution for real-time communications. In particular, it does not define signaling protocols... any signaling protocol that can deliver the required information can be used."

翻译成工程语言：**信令就是把 SDP 字符串和 ICE candidate 从 Alice 传到 Bob，传输手段无所谓**。你可以用：
- WebSocket（最常见）
- HTTP long-polling
- 扫二维码手动粘贴（本章 demo 就用这个！）
- 甚至写在纸上让人跑腿

这个选择的代价是：每个应用都要自己实现信令服务器。收益是：与现有 IM 系统、业务逻辑完全解耦。

### 1.3 SDP 的来历：不是 WebRTC 发明的

SDP（Session Description Protocol）来自 RFC 2327（1998），最初是为 MBONE（Multicast Backbone）多播会议设计的**会话公告格式**——一个组播地址上有哪些媒体流，监听者订阅前需要知道编解码参数。

WebRTC 继承的是其 2006 年的修订版 **RFC 4566**，并扩展了大量 WebRTC 专用的 `a=` 属性：
- `a=ice-ufrag` / `a=ice-pwd`：ICE 认证凭据
- `a=fingerprint`：DTLS 证书指纹
- `a=setup`：DTLS 角色（`actpass`/`active`/`passive`）
- `a=mid`：Media stream Identifier，多路 m= 行的索引
- `a=extmap`：RTP 头部扩展映射
- `a=rtcp-fb`：RTCP 反馈能力（NACK、PLI、REMB 等）
- `a=candidate`：ICE candidate（trickle ICE 下可增量添加）

**Offer/Answer 模型**来自 **RFC 3264**（An Offer/Answer Model with SDP，2002，Jonathan Rosenberg & Henry Schulzrinne），是 SIP 信令的核心子协议。WebRTC 直接复用了这个模型。

## 二、WebRTC 协议栈全景

```
┌─────────────────────────────────────────────────────────────┐
│                    应用层 (JS / Native)                       │
│  getUserMedia()  RTCPeerConnection  RTCDataChannel           │
├──────────────┬──────────────────────────────────────────────┤
│  信令层       │             媒体/数据传输层                    │
│  (自定义)     │                                              │
│  WebSocket   │  ┌─────────────┐  ┌────────────────────────┐│
│  HTTP        │  │  SRTP/SRTCP │  │  SCTP over DTLS        ││
│  etc.        │  │  (音视频)    │  │  (DataChannel)         ││
├──────────────┤  ├─────────────┴──┴────────────────────────┤ │
│              │  │           DTLS 1.2 / 1.3                │ │
│              │  ├────────────────────────────────────────┤ │
│              │  │              ICE / STUN / TURN          │ │
│              │  ├────────────────────────────────────────┤ │
│              │  │                UDP (主) / TCP (备)      │ │
└──────────────┴──┴────────────────────────────────────────┘─┘
```

**层次解析**：

| 层 | 协议 | RFC | 职责 |
|---|---|---|---|
| 候选发现 | ICE | RFC 8445 | NAT 穿透、候选地址收集与检测 |
| 辅助穿透 | STUN | RFC 5389 | 获取公网地址映射 |
| 中继 | TURN | RFC 5766 | 对称 NAT 场景的媒体中继 |
| 加密握手 | DTLS 1.2 | RFC 6347 | 密钥协商，为 SRTP/SCTP 提供密钥材料 |
| 媒体加密 | SRTP | RFC 3711 | 音视频加密传输 |
| 数据通道 | SCTP/DTLS | RFC 8261 | 可靠/不可靠 DataChannel |
| 传输反馈 | RTCP | RFC 3550 | 丢包、延迟、带宽估计反馈 |

## 三、SDP 精读：解剖一个真实的 Offer

下面是一个真实的浏览器生成的 WebRTC offer SDP，加了详细注解：

```
v=0
o=- 5551432059371184543 2 IN IP4 127.0.0.1
```
- `v=0`：SDP 版本，永远是 0（RFC 4566 §5.1）
- `o=`：Origin 行。格式：`<username> <sess-id> <sess-version> <nettype> <addrtype> <addr>`
  - `username=-`：WebRTC 约定用 `-` 表示匿名
  - `sess-id`：随机 64-bit 整数，标识一次会话
  - `sess-version`：每次重新协商时递增（renegotiation 的信号）
  - `IN IP4 127.0.0.1`：SDP 原始设计中的地址，WebRTC 场景下无实际意义（真实地址在 ICE candidate 里）

```
s=-
t=0 0
```
- `s=-`：Session Name，WebRTC 用 `-`
- `t=0 0`：时间范围，`0 0` 表示不限时永久会话

```
a=group:BUNDLE audio video data
a=msid-semantic: WMS myStreamId
```
- `a=group:BUNDLE`：**关键！** 声明多个 m= 行复用同一个 UDP 端口（BUNDLE 扩展，RFC 8843）。没有 BUNDLE 每个媒体流要独立打洞，代价极高
- `a=msid-semantic`：WebRTC Media Stream 语义扩展（RFC 8830），关联轨道到 Stream

```
m=audio 9 UDP/TLS/RTP/SAVPF 111 103 104 9 0 8 106 105 13 110 112 113 126
```
- `m=`：媒体描述行。格式：`<type> <port> <proto> <fmt-list>`
  - `9`：BUNDLE 下 port 固定写 9（占位符），真实端口在 ICE candidate
  - `UDP/TLS/RTP/SAVPF`：**解读** = UDP 传输 + TLS（DTLS 实际） + RTP + SAVPF（Secure Audio/Video Profile with Feedback）
  - `111 103 104...`：Payload Type 列表，**顺序即优先级**（高优先在前）

```
c=IN IP4 0.0.0.0
a=rtcp:9 IN IP4 0.0.0.0
a=ice-ufrag:abc123xyz
a=ice-pwd:longrandompassword64charslongatleast
a=ice-options:trickle
a=fingerprint:sha-256 AA:BB:CC:...
a=setup:actpass
a=mid:audio
a=extmap:1 urn:ietf:params:rtp-hdrext:ssrc-audio-level
a=sendrecv
a=msid:myStreamId audioTrackId
a=rtcp-mux
a=rtcp-rsize
a=rtpmap:111 opus/48000/2
a=rtcp-fb:111 transport-cc
a=fmtp:111 minptime=10;useinbandfec=1
a=rtpmap:103 ISAC/16000
a=rtpmap:104 ISAC/32000
a=rtpmap:9 G722/8000
a=ssrc:3948621398 cname:WOrgFmtkReXFmCaG
a=ssrc:3948621398 msid:myStreamId audioTrackId
```

逐行解析关键属性：

| 属性 | 含义 | 重要性 |
|---|---|---|
| `a=ice-ufrag` / `a=ice-pwd` | ICE 连通性检测的凭据，HMAC-SHA1 的密钥 | **安全关键**，每次 ICE 重启必须更新 |
| `a=fingerprint:sha-256` | DTLS 证书的 SHA-256 摘要 | **防 MitM 关键**，与 ICE consent 一起提供端到端安全 |
| `a=setup:actpass` | DTLS 角色声明：offerer 愿意当 client 或 server | Answer 端必须回 `active` 或 `passive` |
| `a=mid:audio` | Media ID，对应 `a=group:BUNDLE audio video data` | BUNDLE 多路复用的索引键 |
| `a=rtcp-mux` | RTP 和 RTCP 复用同一端口（RFC 5761） | 减少一半打洞开销 |
| `a=rtpmap:111 opus/48000/2` | PT 111 对应 Opus 48kHz 双声道 | 解码器映射 |
| `a=fmtp:111 minptime=10;useinbandfec=1` | Opus 参数：最小打包 10ms，启用带内 FEC | 影响延迟与抗丢包 |
| `a=ssrc:xxx cname:yyy` | RTP 同步源的 Canonical Name | RTCP SR/RR 中用于跨流同步（A/V lip sync） |

## 四、Pion 源码精读：信令状态机

### 4.1 PeerConnection 核心结构

【真实源码 pion/webrtc@master/peerconnection.go】

```go
type PeerConnection struct {
    id string
    mu sync.RWMutex

    sdpOrigin sdp.Origin

    // 串行操作队列——这是线程安全的核心机制
    // WebRTC API 调用来自不同 goroutine，必须串行化
    ops *operations

    configuration Configuration

    // JSEP 规范定义的四个描述槽（RFC 8829 §4.1.14）
    // pending = 协商中，current = 已确认生效
    currentLocalDescription  *SessionDescription
    pendingLocalDescription  *SessionDescription
    currentRemoteDescription *SessionDescription
    pendingRemoteDescription *SessionDescription

    signalingState SignalingState  // 信令状态机当前状态

    // 使用 atomic.Value 存储，避免对 iceConnectionState 的读写竞争
    iceConnectionState atomic.Value // ICEConnectionState
    connectionState    atomic.Value // PeerConnectionState

    // 每个媒体轨道对应一个 Transceiver
    rtpTransceivers []*RTPTransceiver

    // WebRTC 传输层栈
    iceGatherer   *ICEGatherer   // 候选收集
    iceTransport  *ICETransport  // ICE 连通性检测与数据转发
    dtlsTransport *DTLSTransport // DTLS 握手与密钥管理
    sctpTransport *SCTPTransport // DataChannel 的 SCTP 传输

    api *API
    log logging.LeveledLogger
}
```

**设计要点**：`ops *operations` 是整个 PeerConnection 线程安全的核心。所有改变状态的操作（`SetLocalDescription`、`SetRemoteDescription`、ICE 候选处理）都通过这个串行队列执行，避免状态机撕裂。

### 4.2 信令状态机

【真实源码 pion/webrtc@master/signalingstate.go】

```go
// SignalingState 对应 JSEP 规范 RFC 8829 §4.3.1
type SignalingState int32

const (
    SignalingStateUnknown          SignalingState = iota
    SignalingStateStable                          // "stable"
    SignalingStateHaveLocalOffer                  // "have-local-offer"
    SignalingStateHaveRemoteOffer                 // "have-remote-offer"
    SignalingStateHaveLocalPranswer               // "have-local-pranswer"
    SignalingStateHaveRemotePranswer              // "have-remote-pranswer"
    SignalingStateClosed                          // "closed"
)
```

状态转换规则（`checkNextSignalingState` 实现）：

```
                    SetLocalDescription(offer)
         ┌─────────────────────────────────────────┐
         │                                         ▼
      Stable ◄──── SetRemoteDescription(answer) ─── HaveLocalOffer
         │                                              │
         │  SetRemoteDescription(offer)                │ SetRemoteDescription(pranswer)
         │                                              ▼
         ▼                                         HaveRemotePranswer
    HaveRemoteOffer                                     │
         │                                              │ SetRemoteDescription(answer)
         │  SetLocalDescription(answer)                 │
         │                                              ▼
         └────────────────────────────────────────► Stable
         │
         │  SetLocalDescription(pranswer)
         ▼
    HaveLocalPranswer
         │
         │  SetLocalDescription(answer)
         ▼
       Stable
```

违反上述转换规则会返回 `InvalidModificationError`，这是实现 JSEP 规范 §4.1.6 的直接体现。

### 4.3 CreateOffer 内部流程

【真实源码 pion/webrtc@master/peerconnection.go，核心逻辑描述】

`CreateOffer` 的核心逻辑（示意，基于 Pion 实际行为）：

```go
func (pc *PeerConnection) CreateOffer(options *OfferOptions) (SessionDescription, error) {
    // 1. 检查信令状态：必须是 Stable 或 HaveLocalOffer
    if pc.signalingState != SignalingStateStable &&
       pc.signalingState != SignalingStateHaveLocalOffer {
        return SessionDescription{}, &InvalidStateError{...}
    }

    // 2. 快照当前 transceivers（带重试，处理并发修改）
    // 最多重试 128 次确保 transceiver 列表稳定
    for i := 0; i < 128; i++ {
        transceivers := pc.GetTransceivers()
        // 生成 SDP...
        if !transceiversMutatedDuringGeneration() {
            break
        }
    }

    // 3. 为每个 transceiver 分配 MID（"0", "1", "2"...）
    // 4. 构建 session-level SDP（v=, o=, s=, t=, a=group:BUNDLE...）
    // 5. 为每个 transceiver 构建 m= 块
    //    - 根据 Direction 设置 a=sendrecv/sendonly/recvonly/inactive
    //    - 填入支持的 codec 列表和对应 rtpmap/fmtp
    //    - 填入 ICE 凭据（此时 gathering 可能尚未完成）
    //    - 填入 DTLS fingerprint
    // 6. 返回 SessionDescription{Type: SDPTypeOffer, SDP: sdpStr}
}
```

### 4.4 SetRemoteDescription 的传输启动链

`SetRemoteDescription` 是最复杂的函数，它触发整条传输栈的启动：

```
SetRemoteDescription(answer)
    │
    ├─ 解析 SDP，更新 MediaEngine（codec 映射）
    ├─ 匹配/创建 RTPTransceiver（按 MID 对应）
    ├─ 处理 ICE 凭据变化（若 ufrag/pwd 变了 → ICE restart）
    └─ 通过 ops 队列串行执行：
           ├─ startICETransport()    ← ICE 开始连通性检测
           ├─ startDTLSTransport()   ← DTLS 握手（取决于 a=setup 角色）
           ├─ startSCTPTransport()   ← DataChannel 协议层启动
           └─ startRTPReceivers()    ← RTP 接收器与 SSRC 绑定
```

**关键细节**：为什么 `SetLocalDescription(answer)` 不触发传输启动？因为 answerer 在收到 offer 后就有了所有信息；offerer 需要等到 answer 到来才能确认 codec、ICE 角色等。

### 4.5 RTPTransceiver：媒体协商的原子单元

【真实源码 pion/webrtc@master/rtptransceiver.go】

```go
type RTPTransceiver struct {
    mid              atomic.Value // string，对应 SDP 中 a=mid:xxx
    sender           atomic.Value // *RTPSender，发送侧
    receiver         atomic.Value // *RTPReceiver，接收侧
    direction        atomic.Value // RTPTransceiverDirection，期望方向
    currentDirection atomic.Value // 已协商生效的方向

    // 用户通过 SetCodecPreferences 覆盖的 codec 优先级
    codecs []RTPCodecParameters

    kind RTPCodecType // RTPCodecTypeAudio 或 RTPCodecTypeVideo

    api *API
    mu  sync.RWMutex
}

// 内部方向设置（atomic.Store 保证可见性）
func (t *RTPTransceiver) setDirection(d RTPTransceiverDirection) {
    t.direction.Store(d)
}
```

四种 Direction 语义：

| Direction | 本端发 | 本端收 | SDP 中 |
|---|---|---|---|
| `SendRecv` | ✓ | ✓ | `a=sendrecv` |
| `SendOnly` | ✓ | ✗ | `a=sendonly` |
| `RecvOnly` | ✗ | ✓ | `a=recvonly` |
| `Inactive` | ✗ | ✗ | `a=inactive` |

**协商规则**（RFC 8829 §5.2.2）：offer 中的 `sendonly` 对应 answer 中的 `recvonly`，即对端视角翻转。

```html
// 等待 ICE gathering 完成的 Promise 辅助函数
// ============================================================
function waitForIceGathering(peerConnection) {
  return new Promise((resolve) => {
    if (peerConnection.iceGatheringState === 'complete') {
      resolve();
      return;
    }
    const handler = () => {
      if (peerConnection.iceGatheringState === 'complete') {
        peerConnection.onicegatheringstatechange = null;
        resolve();
      }
    };
    peerConnection.onicegatheringstatechange = handler;
    // 5 秒超时保护（候选收集卡住时）
    setTimeout(() => {
      peerConnection.onicegatheringstatechange = null;
      resolve();
    }, 5000);
  });
}
```

### Demo 2：Pion Go 后端作为 Answerer（配合 Demo 1 的浏览器端）

此 demo 展示 Pion 作为 server-side WebRTC peer，接收浏览器 offer 并返回 answer。

**依赖**：Go 1.21+，`go get github.com/pion/webrtc/v3`

**文件**：`demo2-pion-answerer/main.go`

```go
//go:build ignore
// demo2: Pion WebRTC DataChannel answerer
// 用法:
//   go run main.go
// 然后将浏览器 offer 粘贴到终端（JSON 格式），回车，
// 程序输出 answer，粘贴回浏览器即可建立 DataChannel。

package main

import (
    "bufio"
    "encoding/base64"
    "encoding/json"
    "fmt"
    "os"
    "strings"

    "github.com/pion/webrtc/v3"
)

// encode 将 SessionDescription 序列化为 base64 JSON
func encode(obj *webrtc.SessionDescription) string {
    b, err := json.Marshal(obj)
    if err != nil {
        panic(err)
    }
    return base64.StdEncoding.EncodeToString(b)
}

// decode 从 base64 JSON 还原 SessionDescription
func decode(in string, obj *webrtc.SessionDescription) {
    b, err := base64.StdEncoding.DecodeString(in)
    if err != nil {
        // 也尝试直接 JSON 解析（兼容 Demo 1 输出的 JSON 格式）
        if err2 := json.Unmarshal([]byte(in), obj); err2 != nil {
            panic(fmt.Sprintf("decode failed: base64 err=%v, json err=%v", err, err2))
        }
        return
    }
    if err := json.Unmarshal(b, obj); err != nil {
        panic(err)
    }
}

func readLine(prompt string) string {
    fmt.Print(prompt)
    scanner := bufio.NewScanner(os.Stdin)
    scanner.Scan()
    return strings.TrimSpace(scanner.Text())
}

func main() {
    // ──────────────────────────────────────────────
    // 1. 配置 PeerConnection（带 STUN server）
    // ──────────────────────────────────────────────
    config := webrtc.Configuration{
        ICEServers: []webrtc.ICEServer{
            {URLs: []string{"stun:stun.l.google.com:19302"}},
        },
    }

    pc, err := webrtc.NewPeerConnection(config)
    if err != nil {
        panic(err)
    }
    defer pc.Close()

    // ──────────────────────────────────────────────
    // 2. 注册事件处理器（在 SetRemoteDescription 前注册！）
    // ──────────────────────────────────────────────

    // 信令状态变化
    pc.OnSignalingStateChange(func(s webrtc.SignalingState) {
        fmt.Printf("[pion] SignalingState → %s\n", s)
    })

    // ICE 连接状态
    pc.OnICEConnectionStateChange(func(s webrtc.ICEConnectionState) {
        fmt.Printf("[pion] ICEConnectionState → %s\n", s)
        if s == webrtc.ICEConnectionStateFailed {
            fmt.Println("[pion] ICE 连接失败，程序退出")
            os.Exit(1)
        }
    })

    // 连接整体状态
    pc.OnConnectionStateChange(func(s webrtc.PeerConnectionState) {
        fmt.Printf("[pion] ConnectionState → %s\n", s)
    })

    // 接收对端创建的 DataChannel
    pc.OnDataChannel(func(d *webrtc.DataChannel) {
        fmt.Printf("[pion] 收到 DataChannel: label=%q id=%d\n", d.Label(), *d.ID())

        d.OnOpen(func() {
            fmt.Printf("[pion] DataChannel %q 已打开\n", d.Label())

            // 主动发送欢迎消息
            if err := d.SendText("Hello from Pion! 🦁"); err != nil {
                fmt.Printf("[pion] 发送失败: %v\n", err)
            }
        })

        d.OnMessage(func(msg webrtc.DataChannelMessage) {
            fmt.Printf("[pion] 收到消息: %s\n", string(msg.Data))
            // Echo 回去
            if err := d.SendText("Echo: " + string(msg.Data)); err != nil {
                fmt.Printf("[pion] echo 失败: %v\n", err)
            }
        })

        d.OnClose(func() {
            fmt.Printf("[pion] DataChannel %q 已关闭\n", d.Label())
        })
    })

    // ──────────────────────────────────────────────
    // 3. 读取 Offer（从 stdin）
    // ──────────────────────────────────────────────
    fmt.Println("=== Pion WebRTC Answerer ===")
    fmt.Println("请粘贴浏览器的 Offer（JSON 格式，即 Demo 1 offerOut 的内容），然后回车：")
    offerStr := readLine("")

    var offer webrtc.SessionDescription
    decode(offerStr, &offer)
    fmt.Printf("[pion] 解析 Offer 成功，type=%s\n", offer.Type)

    // ──────────────────────────────────────────────
    // 4. SetRemoteDescription(offer)
    // ──────────────────────────────────────────────
    if err := pc.SetRemoteDescription(offer); err != nil {
        panic(err)
    }
    fmt.Printf("[pion] SetRemoteDescription 完成，SignalingState: %s\n", pc.SignalingState())

    // ──────────────────────────────────────────────
    // 5. CreateAnswer
    // ──────────────────────────────────────────────
    answer, err := pc.CreateAnswer(nil)
    if err != nil {
        panic(err)
    }
    fmt.Printf("[pion] CreateAnswer 完成\n")

    // ──────────────────────────────────────────────
    // 6. 等待 ICE gathering 完成（Pion 的 GatheringCompletePromise）
    // ──────────────────────────────────────────────
    gatherComplete := webrtc.GatheringCompletePromise(pc)

    if err := pc.SetLocalDescription(answer); err != nil {
        panic(err)
    }
    fmt.Printf("[pion] SetLocalDescription 完成，等待 ICE gathering...\n")

    // 阻塞等待 gathering 完成
    <-gatherComplete
    fmt.Println("[pion] ICE gathering 完成")

    // ──────────────────────────────────────────────
    // 7. 输出 Answer
    // ──────────────────────────────────────────────
    localDesc := pc.LocalDescription()
    answerJSON, _ := json.MarshalIndent(localDesc, "", "  ")
    fmt.Println("\n=== 请将以下 Answer 粘贴回浏览器（Demo 1 的 answerIn）===")
    fmt.Println(string(answerJSON))
    fmt.Println("========================================================\n")

    // ──────────────────────────────────────────────
    // 8. 保持运行，等待连接
    // ──────────────────────────────────────────────
    fmt.Println("等待连接建立... (Ctrl+C 退出)")
    select {} // 永远阻塞，保持 peer connection 活跃
}
```

## 六、方案对比

### 6.1 信令传输方案对比

| 方案 | 延迟 | 实现复杂度 | 可扩展性 | 适用场景 |
|---|---|---|---|---|
| WebSocket | ~10ms | 低 | 中（需 pub/sub） | 通用首选 |
| HTTP Long-polling | 50-500ms | 低 | 低 | 无 WS 支持的受限环境 |
| Server-Sent Events | 单向低延迟 | 低 | 中 | 广播通知型信令 |
| XMPP/Jingle | ~50ms | 高 | 高（联邦） | 企业 IM 集成 |
| SIP | 高 | 极高 | 高（电信） | 电话网关对接 |
| Matrix | 中 | 中 | 极高（去中心化） | 开源社区平台 |

**边界**：WebSocket 在 1 万并发以上需要水平扩展 + sticky session（或 pub/sub 如 Redis）；SIP 互通需要处理 SDP 方言差异（transcode plan-b vs unified-plan）。

### 6.2 SDP 扩展方案对比（Plan B vs Unified Plan）

| | Plan B（已废弃） | Unified Plan（当前标准） |
|---|---|---|
| 多轨表示 | 一个 m= 行含多个 SSRC | 每个轨道一个 m= 行 |
| SSRC 与轨道绑定 | `a=ssrc:<id> msid:...` | `a=msid:...`（轨道级） |
| 规范来源 | Google Chrome 私有扩展 | W3C + JSEP RFC 8829 |
| 兼容性陷阱 | Firefox 从未支持 | Chrome M71+ 默认 |
| 现状 | Chrome 已删除，Pion 不支持 | 唯一正确选择 |

**血泪教训**：2018-2020 年大量 WebRTC 应用因 Plan B/Unified Plan 混用导致 Firefox-Chrome 互通失败。判断方法：看 SDP 里同一 m= 行是否有多个 `a=ssrc` 属性且带不同 `cname`。

### 6.3 ICE candidate 收集策略对比

| 策略 | 描述 | 信令轮次 | 首次连接时间 | 适用场景 |
|---|---|---|---|---|
| Vanilla ICE（等待完成） | 等 gathering complete 后再发 SDP | 1 次 offer + 1 次 answer | 慢（秒级） | Demo 级、网络简单 |
| Trickle ICE（RFC 8840） | SDP 先发，candidate 增量追加 | 1 offer + N candidate + 1 answer | 快（数百毫秒） | **生产首选** |
| Half Trickle | offerer trickle，answerer vanilla | 折中 | 中 | 信令不支持增量推送时 |

Demo 1/2 使用 Vanilla ICE（等 `complete` 再发），生产环境应改为 Trickle ICE。

## 七、失败模式、真坑与根因

### 坑 1：DTLS 角色冲突（`a=setup` 两端都是 `active`）

**现象**：连接在 `ICEConnectionState: connected` 后长时间停在 `connecting`，最终超时。

**根因**：DTLS 握手需要一端做 client（`active`）一端做 server（`passive`）。Offer 端应发 `a=setup:actpass`（愿意任一角色），Answer 端选择 `active` 或 `passive`。若信令逻辑错误导致两端都设 `active`，DTLS 双方都在等对方发 ClientHello，永远死锁。

**排查**：`chrome://webrtc-internals` 中搜索 `dtls`，看 `DTLS transport state` 和 `a=setup` 值。

**修复**：确保 offerer 的 `a=setup:actpass`，answerer 的 `a=setup:active`（Pion 默认行为正确，浏览器也正确，问题多出在自定义 SDP 修改代码）。

### 坑 2：codec 列表不相交（SDP 协商失败）

**现象**：`setRemoteDescription` 成功，`createAnswer` 返回 SDP，但建连后没有媒体流，或 m= 行 port 为 0。

**根因**：`m=` 行 payload type 列表在协商后变为空（port=0 表示该 m= 被 rejected）。常见于：
- 服务端限制了 codec，但客户端只声明了对端没有的 codec
- `SetCodecPreferences` 传入了空列表

**排查**：检查 answer SDP 中各 m= 行的 port 值，以及 `a=rtpmap` 是否与 offer 有交集。

**修复**：确保 codec 白名单包含对端支持的至少一个 codec。

### 坑 3：ICE candidate 时序竞争（trickle ICE 下）

**现象**：偶发性连接失败，重试后又能通。

**根因**：在 `setRemoteDescription` 完成前调用 `addIceCandidate`，candidate 被丢弃（规范要求先有 remote description 才能处理 candidate）。

**正确做法**：

```javascript
// 错误：candidate 可能比 setRemoteDescription 先到
ws.onmessage = async (e) => {
  const msg = JSON.parse(e.data);
  if (msg.candidate) {
    await pc.addIceCandidate(msg.candidate); // ← 若 remote desc 未设置，candidate 丢失
  }
};

// 正确：用队列缓存提前到达的 candidate
const pendingCandidates = [];
let remoteDescSet = false;

ws.onmessage = async (e) => {
  const msg = JSON.parse(e.data);
  if (msg.candidate) {
    if (remoteDescSet) {
      await pc.addIceCandidate(msg.candidate);
    } else {
      pendingCandidates.push(msg.candidate); // 缓存
    }
  }
  if (msg.answer) {
    await pc.setRemoteDescription(msg.answer);
    remoteDescSet = true;
    // 回放缓存的 candidate
    for (const c of pendingCandidates) {
      await pc.addIceCandidate(c);
    }
    pendingCandidates.length = 0;
  }
};
```

### 坑 4：对称 NAT 下纯 P2P 失败（需要 TURN）

**现象**：局域网内测试正常，部署后特定用户（企业内网、运营商级 NAT）连接失败，`ICEConnectionState` 卡在 `failed`。

**根因**：对称 NAT（Symmetric NAT）下，每个目标地址映射不同的外部端口，STUN `srflx` candidate 对另一端无效。必须通过 TURN 中继。

**判断**：`webrtc-internals` 中 `selectedCandidatePair` 如果是 `relay-relay` 则表示走了 TURN，`host-host` 是局域网直连，`srflx-srflx` 是 STUN 穿透成功，`failed` 且只有 host/srflx candidate 则是对称 NAT 问题。

**修复**：部署 TURN 服务器（coturn 是 open source 首选），配置：

```javascript
const config = {
  iceServers: [
    { urls: 'stun:stun.example.com:3478' },
    {
      urls: 'turn:turn.example.com:3478',
      username: 'user',
      credential: 'password'
    }
  ],
  iceTransportPolicy: 'all' // 若要强制走 TURN 测试，改为 'relay'
};
```

### 坑 5：SDP Munging（SDP 字符串手工修改）的脆弱性

**现象**：生产环境偶发奇怪的协商失败，难以复现。

**根因**：有些场景需要修改 SDP（如强制某 codec、修改 bitrate）。通过字符串替换修改 SDP（"SDP munging"）极其脆弱：浏览器版本升级可能改变 SDP 格式，正则失效导致 SDP 损坏。

**正确路径**（优先级顺序）：
1. `SetCodecPreferences()`：控制 codec 优先级，替代手工修改 payload type 顺序
2. `RTCRtpSender.setParameters()`：动态调整 bitrate/codec 参数
3. `addTransceiver()` 的 `sendEncodings`：simulcast 配置
4. 实在需要 munging：解析为 AST，按字段修改，再序列化（用 sdp.js 等库）

|---|---|

| DTLS | Datagram TLS，为 SRTP/SCTP 提供加密和密钥 |
| BUNDLE | 多媒体流复用单一 UDP 端口（RFC 8843） |
| Trickle ICE | ICE candidate 增量传输，加快首次连接（RFC 8840） |
| MID | Media stream Identifier，BUNDLE 下的 m= 行索引 |
| Transceiver | RTPSender + RTPReceiver 的配对，代表一个双向媒体轨道 |
| SSRC | Synchronization Source，RTP 流的 32-bit 唯一标识 |
