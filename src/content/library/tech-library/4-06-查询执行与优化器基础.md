## TL;DR

- SQL 查询经历四步流水线：Parser → Analyzer → Planner → Executor，每步输入输出都是不同的树形 IR。
- 执行器采用 **Volcano/Iterator 模型**（Graefe 1990）：每个算子实现 `open/next/close` 接口，pull-driven 拉取式执行，tuple 从叶子节点流向根节点。
- 优化器分两大部分：**逻辑变换**（谓词下推、子查询展开）+ **基于代价的物理路径搜索**（Selinger 1979 的动态规划 join 排序）。
- 代价模型核心公式：`total_cost = startup_cost + cpu_run_cost + disk_run_cost`，参数来自 `pg_statistic` 和 `pg_class`。
- 三种 join 实现：Nested Loop（O(m·n)，适合小内表）、Hash Join（O(m+n)，最常用）、Merge Join（O(m log m + n log n)，适合有序输入或 range join）。
- 生产中 95% 的慢查询根因：统计信息过期 → 行数估算失误 → 选错 join 算法。

## 背景与设计考古

### 6.1 为什么要有查询优化器

1970 年代，关系数据库刚出现时，工程师普遍假设用户应该自己写出高效的 SQL。System R 项目（IBM 1975-1979）首次证明这个假设是错的：**一个 3 张表的 join 有 6 种连接顺序，不同顺序的代价可能相差 100 倍以上**，用户无法穷举。

1979 年 Selinger 等人发表论文 "Access Path Selection in a Relational Database Management System"，提出基于代价的动态规划优化器，奠定了此后 45 年所有主流数据库优化器的基本架构。PostgreSQL 的 `optimizer/path/joinrels.c` 至今仍直接实现这篇论文的算法。

### 6.2 Volcano 执行模型的起源

1990 年 Goetz Graefe 在 SIGMOD 发表 "Volcano, an Extensible and Parallel Query Evaluation System"，将数据库执行器抽象为**迭代器（Iterator）树**：

- 每个算子（Operator）实现 `open()`、`next()`、`close()` 三个接口。
- 父算子通过调用 `next()` 从子算子拉取一条 tuple。
- 不需要物化中间结果，数据以流水线方式从底层 scan 流向顶层 output。

这个模型被 System R 的继承者 PostgreSQL、Oracle、DB2 全部采纳，称为 **pull-based** 或 **volcano** 执行模型。其优点是代码组合性极强：新增一个算子只需实现三个接口，不需要修改其他算子。

**缺点**：每条 tuple 都有函数调用开销，对 OLAP 场景（扫描亿级行）造成可观的 CPU overhead。这个问题在 2000 年代促使了向量化执行（MonetDB/X100 2005）和代码生成（HyPer 2011）两条分支路线的出现。

## 第一部分：SQL 执行流水线全景

```
SQL 文本
   │
   ▼ Parser (gram.y / scan.l)
原始解析树 (raw parse tree)
   │
   ▼ Analyzer (analyze.c)
查询树 (Query *)  ← 语义分析，名字解析，类型检查
   │
   ▼ Planner (planner.c)
计划树 (PlannedStmt *)  ← 逻辑优化 + 物理路径搜索
   │
   ▼ Executor (execMain.c)
结果行 (TupleTableSlot *)  ← 实际执行，驱动硬件
```

### 核心数据结构

**Query 树**：经过语义分析后的逻辑表示，包含 `targetList`、`jointree`、`havingQual` 等字段。

**PlannedStmt**：执行计划，包含 `planTree`（Plan 节点树）、`subplans`（子计划列表）等。

**PlanState**：执行时状态，与 Plan 一一对应，包含运行时上下文。

## 第二部分：执行器内核 — Volcano 模型实现

### 6.3 PlanState：算子的运行时表示

每个执行算子在运行时表示为一个 `PlanState` 节点（或其子类型）：

```c
/* 【真实源码 postgres@src/include/nodes/execnodes.h】*/
typedef struct PlanState
{
    NodeTag     type;
    Plan       *plan;           /* 关联的静态 Plan 节点 */
    EState     *state;          /* 整个查询的全局执行状态 */

    ExecProcNodeMtd ExecProcNode;  /* 核心：函数指针，调用时返回下一条 tuple */
    ExecProcNodeMtd ExecProcNodeReal; /* 实际实现，上面可能是 wrapper */

    NodeInstrumentation *instrument;  /* EXPLAIN ANALYZE 的统计收集 */

    ExprState  *qual;           /* 过滤条件（qualification）*/
    PlanState  *lefttree;       /* 左子树（outer 输入）*/
    PlanState  *righttree;      /* 右子树（inner 输入）*/

    List       *initPlan;       /* 初始化子计划（非相关子查询）*/
    Bitmapset  *chgParam;       /* 参数变化集合，触发 rescan */

    TupleDesc   ps_ResultTupleDesc;  /* 输出 tuple 的 schema */
    TupleTableSlot *ps_ResultTupleSlot; /* 输出 tuple 的 slot */
    ExprContext *ps_ExprContext;      /* 表达式求值上下文 */
    ProjectionInfo *ps_ProjInfo;      /* 投影信息 */
} PlanState;
```

**关键字段解析**：
- `ExecProcNode`：这个函数指针就是 Volcano 模型的 `next()` 接口。在 `ExecInitSeqScan` 中，根据是否有 qual/projection 会被赋值为不同的特化版本（`ExecSeqScan`、`ExecSeqScanWithQual` 等），是一种编译时特化的 virtual dispatch。
- `chgParam`：嵌套循环 join 用这个机制通知 inner 子树参数变化，触发内层重扫描（rescan）。
- `lefttree/righttree`：构成整个执行树的骨架，对应 join 的 outer/inner 输入。

### 6.4 ExecProcNode：统一的 next() 接口

```c
/* 【真实源码 postgres@src/include/executor/executor.h】*/
static inline TupleTableSlot *
ExecProcNode(PlanState *node)
{
    if (node->chgParam != NULL) /* 参数有变化，先 rescan */
        ExecReScan(node);
    return node->ExecProcNode(node); /* 调用算子实现 */
}
```

这个 inline 函数封装了 Volcano 模型的 `next()` 语义：先检查是否需要 rescan（参数驱动的重扫描，用于 NestLoop join），再调用具体算子的执行函数。

### 6.5 顺序扫描：最简单的叶子算子

**初始化** (`ExecInitSeqScan`)：

```c
/* 【真实源码 postgres@src/backend/executor/nodeSeqscan.c】*/
SeqScanState *
ExecInitSeqScan(SeqScan *node, EState *estate, int eflags)
{
    SeqScanState *scanstate = makeNode(SeqScanState);
    scanstate->ss.ps.plan = (Plan *) node;
    scanstate->ss.ps.state = estate;

    /* 打开 relation，设置 snapshot */
    scanstate->ss.ss_currentRelation =
        ExecOpenScanRelation(estate, node->scan.scanrelid, eflags);

    /* 初始化 tuple slot（保存当前行的容器）*/
    ExecInitScanTupleSlot(estate, &scanstate->ss, ...);

    /* 编译 qual 表达式 */
    scanstate->ss.ps.qual =
        ExecInitQual(node->scan.plan.qual, (PlanState *) scanstate);

    /*
     * 关键优化：根据是否有 qual 和 projection，选择不同的执行函数。
     * 这是避免 branch misprediction 的运行时特化技巧。
     */
    if (scanstate->ss.ps.qual == NULL) {
        if (scanstate->ss.ps.ps_ProjInfo == NULL)
            scanstate->ss.ps.ExecProcNode = ExecSeqScan;        /* 最快路径 */
        else
            scanstate->ss.ps.ExecProcNode = ExecSeqScanWithProject;
    } else {
        if (scanstate->ss.ps.ps_ProjInfo == NULL)
            scanstate->ss.ps.ExecProcNode = ExecSeqScanWithQual;
        else
            scanstate->ss.ps.ExecProcNode = ExecSeqScanWithQualProject;
    }
    return scanstate;
}
```

**扫描核心** (`SeqNext`)：

```c
/* 【真实源码 postgres@src/backend/executor/nodeSeqscan.c】*/
static pg_attribute_always_inline TupleTableSlot *
SeqNext(SeqScanState *node)
{
    TableScanDesc scandesc = node->ss.ss_currentScanDesc;
    EState       *estate   = node->ss.ps.state;
    ScanDirection direction = estate->es_direction;
    TupleTableSlot *slot = node->ss.ss_ScanTupleSlot;

    if (scandesc == NULL)
    {
        /* 第一次调用：打开底层 table scan descriptor */
        uint32 flags = SO_NONE;
        if (ScanRelIsReadOnly(&node->ss))
            flags |= SO_HINT_REL_READ_ONLY; /* 优化：只读提示 */
        if (estate->es_instrument & INSTRUMENT_IO)
            flags |= SO_SCAN_INSTRUMENT;    /* EXPLAIN (ANALYZE, BUFFERS) 需要 */

        scandesc = table_beginscan(node->ss.ss_currentRelation,
                                   estate->es_snapshot, /* MVCC 快照 */
                                   0, NULL, flags);
        node->ss.ss_currentScanDesc = scandesc;
    }

    /* 拉取一条 tuple 到 slot，底层调用 heap_getnextslot */
    if (table_scan_getnextslot(scandesc, direction, slot))
        return slot;   /* 有 tuple：返回 */
    return NULL;       /* 没有了：返回 NULL 表示 EOF */
}
```

**通用扫描循环**（`ExecScan` / `ExecScanExtended` 的语义，注意真实实现在 `execScan.c` 中，ExecScanExtended 是个 header-only 的 `pg_attribute_always_inline` 函数，不在 .c 文件主体中）：

```c
/* 【示意，非逐字 — 展示 ExecScan 的语义核心】
 * 真实入口: postgres@src/backend/executor/execScan.c
 * ExecScan() -> ExecScanExtended() (header inline)
 * 下面等价于 ExecSeqScanWithQualProject 的展开逻辑:
 */
TupleTableSlot *
ExecScan_semantic(ScanState *node,
                  ExecScanAccessMtd accessMtd,
                  ExecScanRecheckMtd recheckMtd)
{
    ExprState       *qual     = node->ps.qual;
    ProjectionInfo  *projInfo = node->ps.ps_ProjInfo;
    ExprContext     *econtext = node->ps.ps_ExprContext;

    for (;;)
    {
        TupleTableSlot *slot = (*accessMtd)(node); /* 调用 SeqNext / IndexNext */
        if (TupIsNull(slot))
            return NULL;            /* EOF */

        /* 设置表达式上下文中的 scan tuple */
        econtext->ecxt_scantuple = slot;
        ResetExprContext(econtext);

        /* 评估 WHERE 条件 */
        if (qual == NULL || ExecQual(qual, econtext))
        {
            /* 通过 qual，做投影（SELECT 列表）*/
            if (projInfo)
                return ExecProject(projInfo);
            else
                return slot; /* 无投影，直接返回 */
        }
        /* 未通过 qual：继续拉取下一条 tuple */
        InstrCountFiltered1(node, 1);
    }
}
```

### 6.6 Hash Join：最复杂的 join 算子

Hash Join 是 OLTP/OLAP 中最常用的 join 实现，分为两阶段：Build（构建哈希表）+ Probe（探测外表）。

**状态机**（`ExecHashJoinImpl`）：

```c
/* 【真实源码 postgres@src/backend/executor/nodeHashjoin.c】*/
#define HJ_BUILD_HASHTABLE          1  /* 阶段1：构建 inner 的哈希表 */
#define HJ_NEED_NEW_OUTER           2  /* 阶段2：拉取下一条 outer tuple */
#define HJ_SCAN_BUCKET              3  /* 阶段3：在哈希桶中查找匹配 */
#define HJ_FILL_OUTER_TUPLE         4  /* 阶段4：LEFT JOIN 填充未匹配 outer */
#define HJ_FILL_INNER_TUPLES        5  /* 阶段5：RIGHT JOIN 填充未匹配 inner */
#define HJ_FILL_OUTER_NULL_TUPLES   6  /* 阶段6：处理 NULL key 的 outer tuples */
#define HJ_FILL_INNER_NULL_TUPLES   7  /* 阶段7：处理 NULL key 的 inner tuples */
#define HJ_NEED_NEW_BATCH           8  /* 阶段8：Grace Hash Join 换批次 */
```

**执行流程核心**（简化）：

```c
/* 【示意，非逐字 — 基于 postgres@src/backend/executor/nodeHashjoin.c 的 ExecHashJoinImpl 逻辑】*/
static TupleTableSlot *
ExecHashJoinImpl(PlanState *pstate, bool parallel)
{
    HashJoinState *node = castNode(HashJoinState, pstate);

    for (;;) {
        switch (node->hj_JoinState) {
        case HJ_BUILD_HASHTABLE:
            /* 如果 outer 为空且非 outer join，直接结束 */
            if (!node->hj_OuterNotEmpty)
                return NULL;
            /* 遍历 inner（build 侧），把所有 tuple 插入哈希表 */
            node->hj_HashTable = ExecHashTableCreate(...);
            // ... inner 全量扫描 + hash insert ...
            node->hj_JoinState = HJ_NEED_NEW_OUTER;
            break;

        case HJ_NEED_NEW_OUTER:
            outerTupleSlot = ExecProcNode(outerPlanState(node));
            if (TupIsNull(outerTupleSlot)) {
                /* outer 扫完了 */
                if (need_fill_inner) {
                    node->hj_JoinState = HJ_FILL_INNER_TUPLES;
                    break;
                }
                return NULL;
            }
            /* 计算 outer tuple 的 hash value，确定它在哪个 batch */
            hashvalue = ExecHashGetHashValue(...);
            if (batchno != curbatch) {
                /* Grace Hash Join：当前 batch 处理不了，写到 batch file */
                ExecHashJoinSaveTuple(outerTupleSlot, ...);
                break; /* 继续拉 outer */
            }
            /* 在哈希表里找对应的 bucket */
            node->hj_CurTuple = ExecHashGetBucketAndBatch(hashtable, hashvalue);
            node->hj_JoinState = HJ_SCAN_BUCKET;
            /* fall through */

        case HJ_SCAN_BUCKET:
            /* 遍历 bucket chain，找 joinqual 匹配的 inner tuple */
            while ((curtuple = node->hj_CurTuple) != NULL) {
                node->hj_CurTuple = HJTUPLE_NEXT(curtuple); /* 链表遍历 */
                if (ExecQual(joinqual, econtext)) {
                    /* 匹配！检查 otherqual，然后投影输出 */
                    if (ExecQual(otherqual, econtext))
                        return ExecProject(node->js.ps.ps_ProjInfo);
                }
            }
            /* bucket 扫完没匹配：LEFT JOIN 需要补 NULL */
            if (node->js.jointype == JOIN_LEFT && !node->hj_MatchedOuter)
                node->hj_JoinState = HJ_FILL_OUTER_TUPLE;
            else
                node->hj_JoinState = HJ_NEED_NEW_OUTER;
            break;

        // ... 其他 case 处理 outer join 补 NULL 和 Grace batch 切换 ...
        }
    }
}
```

**Grace Hash Join（分批处理）** 是 PostgreSQL Hash Join 的内存超限处理机制：当 inner 关系超过 `work_mem` 时，把数据按 hash 分成多个 batch，每次只加载一个 batch 处理。代价分析中对应 `numbatches > 1` 的额外 I/O 惩罚项。

### 6.7 Nested Loop：参数传递机制

```c
/* 【真实源码 postgres@src/backend/executor/nodeNestloop.c】*/
static TupleTableSlot *
ExecNestLoop(PlanState *pstate)
{
    NestLoopState *node = castNode(NestLoopState, pstate);
    NestLoop      *nl   = (NestLoop *) node->js.ps.plan;
    PlanState     *innerPlan = innerPlanState(node);
    PlanState     *outerPlan = outerPlanState(node);
    ExprContext   *econtext  = node->js.ps.ps_ExprContext;
    ListCell      *lc;

    ResetExprContext(econtext);

    for (;;)
    {
        if (node->nl_NeedNewOuter)
        {
            /* 拉取 outer 的下一条 tuple */
            outerTupleSlot = ExecProcNode(outerPlan);
            if (TupIsNull(outerTupleSlot))
                return NULL; /* outer 全扫完了，join 结束 */

            econtext->ecxt_outertuple = outerTupleSlot;
            node->nl_NeedNewOuter  = false;
            node->nl_MatchedOuter  = false;

            /*
             * 关键：把 outer tuple 的字段值写入 inner 的参数槽。
             * 这是 Correlated Subquery / Index NestLoop 的核心机制：
             * inner 侧的 IndexScan 用这些参数做 index lookup，
             * 实现 O(m * log n) 的 index nested loop join。
             */
            foreach(lc, nl->nestParams)
            {
                NestLoopParam *nlp = (NestLoopParam *) lfirst(lc);
                int paramno = nlp->paramno;
                ParamExecData *prm;

                prm = &(econtext->ecxt_param_exec_vals[paramno]);
                /* 从 outer tuple 中取出字段值，放入参数槽 */
                prm->value = slot_getattr(outerTupleSlot,
                                          nlp->paramval->varattno,
                                          &(prm->isnull));
                /* 标记 inner plan 的参数有变化，下次调用会触发 rescan */
                innerPlan->chgParam = bms_add_member(innerPlan->chgParam, paramno);
            }

            /* inner 重新扫描（从头开始，使用新参数） */
            ExecReScan(innerPlan);
        }

        /* 拉取 inner 的下一条 tuple */
        innerTupleSlot = ExecProcNode(innerPlan);
        econtext->ecxt_innertuple = innerTupleSlot;

        if (TupIsNull(innerTupleSlot))
        {
            /* inner 扫完了，标记需要新的 outer tuple */
            node->nl_NeedNewOuter = true;

            /* LEFT/ANTI JOIN：inner 没有匹配时补 NULL */
            if (!node->nl_MatchedOuter &&
                (node->js.jointype == JOIN_LEFT ||
                 node->js.jointype == JOIN_ANTI))
            {
                econtext->ecxt_innertuple = node->nl_NullInnerTupleSlot;
                if (otherqual == NULL || ExecQual(otherqual, econtext))
                    return ExecProject(node->js.ps.ps_ProjInfo);
            }
            continue;
        }

        /* 评估 join condition */
        if (ExecQual(joinqual, econtext))
        {
            node->nl_MatchedOuter = true;
            if (node->js.jointype == JOIN_ANTI) {
                node->nl_NeedNewOuter = true; /* ANTI JOIN 跳过匹配的 outer */
                continue;
            }
            if (node->js.single_match)
                node->nl_NeedNewOuter = true; /* SEMI JOIN / unique inner */

            if (otherqual == NULL || ExecQual(otherqual, econtext))
                return ExecProject(node->js.ps.ps_ProjInfo);
        }
        ResetExprContext(econtext);
    }
}
```

**Index NestLoop Join 的高效性**：当 inner 侧是 IndexScan，且 index 覆盖了 join key 时，inner 每次 rescan 只需做一次 index lookup（O(log n)），整体复杂度降为 O(m log n)，是 OLTP 场景最高效的 join 策略。

## 第三部分：优化器 — 从 SQL 树到执行计划

PostgreSQL 优化器分两层：

```
Query 树（逻辑）
   │
   ▼ subquery_planner()  — 逻辑变换
      - preprocess_expression: 常量折叠, 谓词标准化
      - pull_up_sublinks:      相关子查询展开为 join
      - reduce_outer_joins:    消除不必要的 outer join
      │
      ▼ grouping_planner()  — 物理路径搜索
         - query_planner() → make_one_rel() → 生成所有物理路径
         - 对 GROUP BY / ORDER BY / LIMIT 选择最优上层策略
         │
         ▼ get_cheapest_fractional_path() — 选最优路径
            │
            ▼ create_plan() — Path 树 → Plan 树（可执行）
```

### 6.8 代价估算模型

**基本结构**：每个路径（Path）有两个代价：

```
startup_cost  — 返回第一条 tuple 之前的代价（如 Hash Join 的 build 阶段）
total_cost    — 返回所有 tuple 的总代价
```

**顺序扫描** (`cost_seqscan`, `postgres@src/backend/optimizer/path/costsize.c`):

```c
/* 【真实源码 postgres@src/backend/optimizer/path/costsize.c】*/
void
cost_seqscan(Path *path, PlannerInfo *root,
             RelOptInfo *baserel, ParamPathInfo *param_info)
{
    double spc_seq_page_cost;
    QualCost qpqual_cost;
    Cost   cpu_per_tuple;
    Cost   cpu_run_cost;
    Cost   disk_run_cost;
    Cost   startup_cost = 0;   /* seq scan 没有 startup cost */

    /* 获取表空间的 seq_page_cost（默认 1.0）*/
    get_tablespace_page_costs(baserel->reltablespace,
                              NULL, &spc_seq_page_cost);

    /* 磁盘 I/O 代价：pages * seq_page_cost */
    disk_run_cost = spc_seq_page_cost * baserel->pages;

    /* CPU 代价：每条 tuple 的处理代价 */
    cost_qual_eval(&qpqual_cost, qpquals, root);
    cpu_per_tuple  = cpu_tuple_cost + qpqual_cost.per_tuple;
    cpu_run_cost   = cpu_per_tuple * baserel->tuples;

    path->startup_cost = startup_cost;
    path->total_cost   = startup_cost + cpu_run_cost + disk_run_cost;
}
/*
 * 默认参数：
 *   seq_page_cost    = 1.0
 *   cpu_tuple_cost   = 0.01
 *   cpu_operator_cost = 0.0025
 * 示例：100万行表，1000 pages：
 *   disk_run_cost = 1.0 * 1000 = 1000
 *   cpu_run_cost  = 0.01 * 1e6 = 10000
 *   total         = 11000
 */
```

**Nested Loop Join 初始估算** (`initial_cost_nestloop`):

```c
/* 【真实源码 postgres@src/backend/optimizer/path/costsize.c, L3373-3444】*/
void
initial_cost_nestloop(PlannerInfo *root, JoinCostWorkspace *workspace,
                      JoinType jointype, uint64 enable_mask,
                      Path *outer_path, Path *inner_path,
                      JoinPathExtraData *extra)
{
    Cost startup_cost = 0;
    Cost run_cost = 0;
    double outer_path_rows = outer_path->rows;

    /* startup = outer.startup + inner.startup（两者都必须启动才能开始）*/
    startup_cost += outer_path->startup_cost + inner_path->startup_cost;

    /* outer run cost：完整扫描一次 outer */
    run_cost += outer_path->total_cost - outer_path->startup_cost;

    /* inner rescan startup：outer 每行都需要重启 inner（扣第一次）*/
    if (outer_path_rows > 1)
        run_cost += (outer_path_rows - 1) * inner_rescan_start_cost;

    /* 正常情况（非 SEMI/ANTI）：每条 outer tuple 都完整扫 inner */
    inner_run_cost = inner_path->total_cost - inner_path->startup_cost;
    inner_rescan_run_cost = inner_rescan_total_cost - inner_rescan_start_cost;

    run_cost += inner_run_cost;  /* 第一次扫 inner */
    if (outer_path_rows > 1)
        run_cost += (outer_path_rows - 1) * inner_rescan_run_cost; /* 重扫代价 */

    /* CPU 代价留给 final_cost_nestloop 计算（需要分析 join qual）*/
    workspace->startup_cost = startup_cost;
    workspace->total_cost   = startup_cost + run_cost;
}
/*
 * 核心公式（简化）：
 * total ≈ outer.total + outer.rows * inner.rescan_total + cpu_qual * output_rows
 *
 * 当 inner 是 IndexScan 时，inner.rescan_total = O(log n)，整体 O(m log n)
 * 当 inner 是 SeqScan 时，inner.rescan_total = O(n)，整体 O(m*n)，非常慢
 */
```

**Hash Join 代价估算** (`initial_cost_hashjoin`):

```c
/* 【真实源码 postgres@src/backend/optimizer/path/costsize.c, L4297-4402】*/
void
initial_cost_hashjoin(PlannerInfo *root, JoinCostWorkspace *workspace,
                      JoinType jointype, List *hashclauses,
                      Path *outer_path, Path *inner_path,
                      JoinPathExtraData *extra, bool parallel_hash)
{
    Cost startup_cost = 0;
    Cost run_cost = 0;
    double outer_path_rows = outer_path->rows;
    double inner_path_rows = inner_path->rows;
    int num_hashclauses = list_length(hashclauses);

    /* startup = outer.startup + inner 完整扫描代价（全部插入哈希表）*/
    startup_cost += outer_path->startup_cost;
    run_cost += outer_path->total_cost - outer_path->startup_cost;
    startup_cost += inner_path->total_cost; /* ← inner 全量 build */

    /* 为每条 inner tuple 计算 hash 并插入表的 CPU 代价 */
    startup_cost += (cpu_operator_cost * num_hashclauses + cpu_tuple_cost)
                    * inner_path_rows;
    /* 为每条 outer tuple 计算 hash 的 CPU 代价 */
    run_cost += cpu_operator_cost * num_hashclauses * outer_path_rows;

    /* Grace Hash Join：inner 超出 work_mem 时，额外的 I/O 代价 */
    if (numbatches > 1)
    {
        double innerpages = page_size(inner_path_rows, ...);
        double outerpages = page_size(outer_path_rows, ...);
        startup_cost += seq_page_cost * innerpages;       /* 写 inner batch */
        run_cost += seq_page_cost * (innerpages + 2 * outerpages); /* 读回 */
    }

    workspace->startup_cost = startup_cost;
    workspace->total_cost   = startup_cost + run_cost;
}
/*
 * 核心特点：
 * - startup_cost 高（需要 build 整个哈希表，O(n)）
 * - run_cost 低（outer 每条 tuple 只做 O(1) hash lookup）
 * - 总体 O(m+n)，与 NestLoop 的 O(m*n) 形成鲜明对比
 * - 内存受限时退化为 Grace Hash Join，有磁盘 I/O 代价
 */
```

### 6.9 Join 排序：动态规划

```c
/* 【真实源码 postgres@src/backend/optimizer/path/joinrels.c】*/
void
join_search_one_level(PlannerInfo *root, int level)
{
    List *joinrels;
    ListCell *r;
    int k;

    /*
     * Selinger 1979 动态规划：
     * 已知 level-1 个关系的最优计划，求 level 个关系的最优计划
     *
     * 核心思路：join(level) = join(1, level-1) 的所有组合中最优的
     */

    /* 第一轮：尝试把 base 关系加入 level-1 的 join 结果 */
    foreach(r, root->join_rel_level[level - 1])
    {
        RelOptInfo *old_rel = (RelOptInfo *) lfirst(r);

        /* 只考虑有 join clause 连接的关系（避免无效笛卡儿积）*/
        make_rels_by_clause_joins(root, old_rel, ...);
    }

    /* 第二轮：bushy tree，即 join(k, level-k) 对 2 ≤ k ≤ level-2 */
    for (k = 2; k <= level - 2; k++)
    {
        /* 枚举所有 k 大小和 (level-k) 大小的子集的组合 */
        foreach(r, root->join_rel_level[k])
        {
            RelOptInfo *old_rel = (RelOptInfo *) lfirst(r);
            make_rels_by_clause_joins(root, old_rel,
                                      root->join_rel_level[level - k]);
        }
    }

    /* 兜底：如果还没有找到 join 关系（孤立表），强制笛卡儿积 */
    if (root->join_rel_level[level] == NIL)
        make_rels_by_joins(root, level, false); /* clauseless join */
}

/*
 * 对应关系：
 *   root->join_rel_level[1] = {A, B, C, ...}     基表
 *   root->join_rel_level[2] = {AB, AC, BC, ...}   两表 join
 *   root->join_rel_level[3] = {ABC, ABD, ACD, ...} 三表 join
 *
 * 时间复杂度：O(3^n) 最坏（考虑所有 bushy tree），
 *   但实际因为 join clause 过滤，通常接近 O(n^2)。
 * 当 n > geqo_threshold（默认 12）时，切换到遗传算法（GEQO）。
 */
```

### 6.10 selectivity 估算与统计信息

优化器的代价估算质量完全取决于统计信息的准确性。PostgreSQL 维护三级统计信息：

**Level 1 — pg_class**（表级）:
- `reltuples`：估计行数
- `relpages`：物理页数

**Level 2 — pg_statistic / pg_stats**（列级）:
- `null_frac`：NULL 比例
- `avg_width`：平均列宽（字节）
- `n_distinct`：不同值数量
- `most_common_vals`（MCV）：高频值列表
- `histogram_bounds`：直方图桶边界
- `correlation`：物理存储顺序与逻辑值的相关系数（影响 index scan 代价）

**Level 3 — pg_statistic_ext**（多列）:
- 函数依赖（functional dependencies）：`city => zip_code`
- 多列 n-distinct：GROUP BY 多列时的不同值估计
- 多列 MCV：多列组合的高频值

**selectivity 估算示例**：

```sql
-- 等值条件：col = 'value'
-- 1. 先查 MCV 列表，如果 'value' 在里面，直接用对应频率
-- 2. 不在 MCV 里，用公式估算：(1 - sum(MCV_freqs)) / (n_distinct - MCV_count)
-- 这叫 "most-common-values + histogram" 混合估算模型

-- 范围条件：col BETWEEN 100 AND 200
-- 用 histogram_bounds 插值：
-- selectivity = (200-100) / (histogram_max - histogram_min)
-- 更精确：计算覆盖了几个完整的 bucket + 两端的小数 bucket
```

**相关性（correlation）对 index scan 代价的影响**：

当 `correlation ≈ 1.0` 时，表示数据物理顺序与 index 顺序一致，index scan 几乎是顺序 I/O，代价接近 seq scan。当 `correlation ≈ 0` 时，每次 index lookup 可能都是随机 I/O，代价 = 行数 × `random_page_cost`（默认 4.0）。

```c
/* 【真实源码 postgres@src/backend/optimizer/path/costsize.c — cost_index 中的相关性修正】*/
/* 
 * Table I/O 代价根据 index correlation 在两个极值间插值：
 * max_IO_cost = Selectivity * relPages * random_page_cost  (correlation=0, 全随机)
 * min_IO_cost = Selectivity * relPages * seq_page_cost     (correlation=1, 全顺序)
 * actual_IO   = max_IO_cost + correlation² * (min_IO_cost - max_IO_cost)
 */
```

## 第四部分：三种执行模型对比

| 维度 | Volcano（行模型） | 向量化（列批模型） | 代码编译（HyPer/DuckDB）|
|------|----------|------------|------------|
| 代表系统 | PostgreSQL, MySQL | MonetDB, ClickHouse, DuckDB | HyPer, Umbra |
| 单次处理 | 1 tuple | 1024~4096 tuples（vector）| pipeline 内数据 push 到底 |
| 函数调用开销 | 每 tuple 一次虚调用 | 每 vector 一次 | JIT 后几乎消除 |
| Cache 效率 | 差（每行跨多列）| 好（同列连续）| 依实现 |
| SIMD 利用 | 难 | 天然适合 | 依实现 |
| 实现复杂度 | 低 | 中 | 高 |
| 适合场景 | OLTP（点查、小 join）| OLAP（聚合、大 scan）| 两者皆可 |
| PG 现状 | 主执行器 | JIT 局部优化（不是真正向量化）| LLVM JIT 在 PG 14+ |

### Demo 1：手写 Mini Volcano 执行器（Python）

这个 demo 实现了一个迷你 Volcano 执行引擎，包含 SeqScan、Filter、HashJoin、Project 四个算子，印证 PostgreSQL `ExecSeqScan`、`ExecNestLoop`、`ExecHashJoin` 的设计。

**设计为可运行，请在你环境验证。依赖：Python 3.8+，无需第三方库。**

```python
#!/usr/bin/env python3
"""
Mini Volcano 执行器 demo
对应 PostgreSQL:
  SeqScan    -> nodeSeqscan.c / SeqNext()
  Filter     -> execScan.c / ExecScan() 中的 qual 评估
  HashJoin   -> nodeHashjoin.c / ExecHashJoinImpl()
  Project    -> ExecProject() 中的表达式求值

运行：python3 mini_volcano.py
"""

from __future__ import annotations
from typing import Iterator, Optional, Callable, Any
from dataclasses import dataclass

# ---------- 数据类型 ----------

@dataclass
class Tuple:
    """一条数据行，用 dict 模拟 TupleTableSlot"""
    data: dict

    def __repr__(self):
        return f"Tuple({self.data})"


# ---------- 算子基类（Volcano Iterator 接口）----------

class Operator:
    """
    对应 PlanState 结构体。
    三个方法对应 Volcano 模型的 open/next/close。
    在 PostgreSQL 中：
      open()  -> ExecInitXxx()
      next()  -> ExecProcNode() -> node->ExecProcNode(node)
      close() -> ExecEndXxx()
    """
    def open(self):
        pass

    def next(self) -> Optional[Tuple]:
        raise NotImplementedError

    def close(self):
        pass


# ---------- SeqScan：全表扫描 ----------

class SeqScan(Operator):
    """
    对应 nodeSeqscan.c 中的 SeqNext()。
    Python list 模拟磁盘上的 heap 文件。
    """
    def __init__(self, table: list[dict], alias: str = ""):
        self.table  = table
        self.alias  = alias
        self._iter: Iterator = iter([])

    def open(self):
        # 对应 table_beginscan()：打开扫描描述符
        self._iter = iter(self.table)
        print(f"  [SeqScan '{self.alias}'] open: 开始扫描 {len(self.table)} 行")

    def next(self) -> Optional[Tuple]:
        # 对应 table_scan_getnextslot()
        row = next(self._iter, None)
        if row is None:
            return None  # EOF，对应 TupIsNull()
        # 如果有别名，为所有列加前缀（简化版 range table）
        if self.alias:
            return Tuple({f"{self.alias}.{k}": v for k, v in row.items()})
        return Tuple(dict(row))

    def close(self):
        print(f"  [SeqScan '{self.alias}'] close")


# ---------- Filter：过滤条件（WHERE 子句）----------

class Filter(Operator):
    """
    对应 execScan.c 中 ExecScan 里的 qual 评估逻辑。
    qual: 函数 Tuple -> bool，对应 ExprState + ExecQual()
    """
    def __init__(self, child: Operator, qual: Callable[[Tuple], bool]):
        self.child = child
        self.qual  = qual
        self._filtered = 0

    def open(self):
        self.child.open()
        print(f"  [Filter] open")

    def next(self) -> Optional[Tuple]:
        # 对应 ExecScan 中的 for(;;) 循环：
        # 不断拉 tuple，直到找到一条满足 qual 的，或 EOF
        while True:
            t = self.child.next()
            if t is None:
                return None  # 子算子 EOF
            if self.qual(t):
                return t     # 通过过滤条件
            self._filtered += 1  # 对应 InstrCountFiltered1()

    def close(self):
        self.child.close()
        print(f"  [Filter] close，过滤掉 {self._filtered} 条")


# ---------- HashJoin：Build + Probe 两阶段 ----------

class HashJoin(Operator):
    """
    对应 nodeHashjoin.c 中的 ExecHashJoinImpl() 状态机（简化版）。
    
    简化说明：
    - 省略了 Grace Hash Join（多批次磁盘溢出）
    - 省略了 null key 处理
    - 只实现 INNER JOIN
    
    状态机：
    HJ_BUILD_HASHTABLE -> HJ_NEED_NEW_OUTER -> HJ_SCAN_BUCKET -> ...
    """
    def __init__(self,
                 outer: Operator,     # probe 侧（通常是大表）
                 inner: Operator,     # build 侧（通常是小表）
                 build_key: Callable[[Tuple], Any],   # inner 的 hash key
                 probe_key: Callable[[Tuple], Any],   # outer 的 hash key
                 join_qual: Optional[Callable[[Tuple, Tuple], bool]] = None):
        self.outer      = outer
        self.inner      = inner
        self.build_key  = build_key
        self.probe_key  = probe_key
        self.join_qual  = join_qual or (lambda a, b: True)

        # 运行时状态（对应 HashJoinState）
        self._hashtable: dict[Any, list[Tuple]] = {}   # 模拟 HJ hashtable
        self._phase     = "HJ_BUILD_HASHTABLE"
        self._cur_outer: Optional[Tuple] = None
        self._cur_bucket: list[Tuple] = []
        self._bucket_idx = 0

    def open(self):
        self.outer.open()
        self.inner.open()
        print(f"  [HashJoin] open")

    def _build_phase(self):
        """HJ_BUILD_HASHTABLE：扫描全部 inner，插入哈希表。
        对应 ExecHashTableCreate + inner 全量扫描 + ExecHashTableInsert。"""
        print(f"  [HashJoin] Build phase: 构建哈希表...")
        while True:
            t = self.inner.next()
            if t is None:
                break
            key = self.build_key(t)
            self._hashtable.setdefault(key, []).append(t)
        print(f"  [HashJoin] Build 完成：{len(self._hashtable)} 个不同 key，"
              f"{sum(len(v) for v in self._hashtable.values())} 条记录")
        self._phase = "HJ_NEED_NEW_OUTER"

    def next(self) -> Optional[Tuple]:
        # 进入状态机
        while True:
            if self._phase == "HJ_BUILD_HASHTABLE":
                self._build_phase()

            elif self._phase == "HJ_NEED_NEW_OUTER":
                # 对应 ExecHashJoinImpl 中 HJ_NEED_NEW_OUTER case
                self._cur_outer = self.outer.next()
                if self._cur_outer is None:
                    return None  # outer 耗尽，join 结束
                probe_key = self.probe_key(self._cur_outer)
                # 找到对应的 bucket（可能为空）
                self._cur_bucket  = self._hashtable.get(probe_key, [])
                self._bucket_idx  = 0
                self._phase       = "HJ_SCAN_BUCKET"

            elif self._phase == "HJ_SCAN_BUCKET":
                # 对应 HJ_SCAN_BUCKET case：遍历 bucket chain
                while self._bucket_idx < len(self._cur_bucket):
                    inner_tuple = self._cur_bucket[self._bucket_idx]
                    self._bucket_idx += 1
                    if self.join_qual(self._cur_outer, inner_tuple):
                        # 匹配！合并两个 tuple 的 data，模拟 ExecProject
                        merged = Tuple({**self._cur_outer.data, **inner_tuple.data})
                        return merged
                # bucket 扫完，拉下一条 outer
                self._phase = "HJ_NEED_NEW_OUTER"

    def close(self):
        self.outer.close()
        self.inner.close()
        print(f"  [HashJoin] close，哈希表释放")


# ---------- Project：投影（SELECT 列表）----------

class Project(Operator):
    """
    对应 ExecProject()。
    target_list 是一个 {output_col_name: expr} 的字典，
    expr 是 Tuple -> value 的函数，对应 ExprState + ExecEvalExpr()。
    """
    def __init__(self,
                 child: Operator,
                 target_list: dict[str, Callable[[Tuple], Any]]):
        self.child = child
        self.target_list = target_list

    def open(self):
        self.child.open()

    def next(self) -> Optional[Tuple]:
        t = self.child.next()
        if t is None:
            return None
        # 对每个输出列，调用对应的表达式函数
        projected = {col: expr(t) for col, expr in self.target_list.items()}
        return Tuple(projected)

    def close(self):
        self.child.close()


# ---------- 测试数据 ----------

orders_data = [
    {"id": 1, "customer_id": 101, "amount": 500},

    模拟以下 SQL：
    
    SELECT c.name, c.city, SUM(o.amount) as total_amount
    FROM orders o
    JOIN customers c ON o.customer_id = c.id
    WHERE o.amount >= 400

      Project(name, city, total_amount)
        -> HashAgg(group by name, city)   <-- 这里简化为 Python 外层聚合
          -> HashJoin(o.customer_id = c.id)
              outer: Filter(orders, amount >= 400)
                       SeqScan(orders)
              inner: SeqScan(customers)
    """
    print("=" * 60)
    print("执行查询：orders JOIN customers WHERE amount >= 400")
    print("=" * 60)

        key = (row["name"], row["city"])
        agg[key] = agg.get(key, 0) + row["amount"]

    final = sorted(
        [{"name": k[0], "city": k[1], "total_amount": v} for k, v in agg.items()],
        key=lambda x: -x["total_amount"]
    )

    print("\n--- 查询结果 ---")
    print(f"{'name':<10} {'city':<12} {'total_amount':>12}")
    print("-" * 36)
    for row in final:
        print(f"{row['name']:<10} {row['city']:<12} {row['total_amount']:>12}")
    print(f"\n共 {len(final)} 行")


if __name__ == "__main__":
    run_query()
```

### Demo 2：基于代价的 Join 顺序选择（Python）

这个 demo 模拟了 PostgreSQL 优化器的动态规划 join 排序算法，印证 `joinrels.c` 的设计。

**设计为可运行，请在你环境验证。依赖：Python 3.8+，无需第三方库。**

```python
#!/usr/bin/env python3
"""
Mini 代价优化器：动态规划 Join 排序
对应 PostgreSQL: optimizer/path/joinrels.c 的 join_search_one_level

模拟以下决策过程：给定 3 张表 A、B、C，以及估计的行数、列数，
枚举所有 join 顺序，用简化代价模型选择最优顺序。

运行：python3 mini_optimizer.py
"""

from __future__ import annotations
from dataclasses import dataclass, field
from typing import FrozenSet, Dict, Tuple, Optional
import itertools
import math

# ---------- 代价模型参数（对应 PG GUC）----------

SEQ_PAGE_COST     = 1.0    # seq_page_cost
RANDOM_PAGE_COST  = 4.0    # random_page_cost
CPU_TUPLE_COST    = 0.01   # cpu_tuple_cost
CPU_OPERATOR_COST = 0.0025 # cpu_operator_cost
BYTES_PER_PAGE    = 8192   # 8KB page

# ---------- 关系元数据（对应 RelOptInfo 的关键字段）----------

@dataclass
class Relation:
    name:    str
    rows:    int    # 估计行数（来自 reltuples）
    width:   int    # 平均行宽（字节，来自 pg_stats.avg_width 求和）

    @property
    def pages(self) -> float:
        """估计页数 = rows * width / 8192"""
        return max(1.0, self.rows * self.width / BYTES_PER_PAGE)


@dataclass
class JoinClause:
    """一个 join 条件，例如 A.id = B.a_id"""
    left:        str   # 左表名
    right:       str   # 右表名
    selectivity: float  # join selectivity（1 / max(left.ndistinct, right.ndistinct)）


# ---------- Path：一个候选执行计划 ----------

@dataclass
class Path:
    rels:          FrozenSet[str]  # 该 path 覆盖的关系集合
    rows:          float           # 估计输出行数
    startup_cost:  float
    total_cost:    float
    description:   str             # 人类可读的描述

    def __lt__(self, other: "Path") -> bool:
        return self.total_cost < other.total_cost


# ---------- 代价估算函数（对应 costsize.c）----------

def cost_seqscan(rel: Relation) -> Path:
    """
    对应 cost_seqscan()
    total = seq_page_cost * pages + cpu_tuple_cost * rows
    """
    disk_cost = SEQ_PAGE_COST * rel.pages
    cpu_cost  = CPU_TUPLE_COST * rel.rows
    return Path(
        rels=frozenset([rel.name]),
        rows=float(rel.rows),
        startup_cost=0.0,
        total_cost=disk_cost + cpu_cost,
        description=f"SeqScan({rel.name})"
    )


def cost_hashjoin(outer: Path, inner: Path,
                  join_sel: float, work_mem_pages: int = 4096) -> Path:
    """
    对应 initial_cost_hashjoin() + final_cost_hashjoin()
    
    startup  = inner.total（build 阶段）
             + inner.rows * (cpu_operator_cost + cpu_tuple_cost)  hash + insert
    run      = outer.total - outer.startup（扫 outer）
             + outer.rows * cpu_operator_cost  （probe hash function）
             + cpu_per_tuple * output_rows    （qual + 投影）
    
    如果 inner 超过 work_mem，加 Grace Hash Join 代价。
    """
    inner_pages = inner.rows * 24 / BYTES_PER_PAGE  # 估计 inner 的物理大小

    # Build phase startup
    startup_cost = inner.total_cost
    startup_cost += (CPU_OPERATOR_COST + CPU_TUPLE_COST) * inner.rows

    # Probe phase run cost
    run_cost = (outer.total_cost - outer.startup_cost)
    run_cost += CPU_OPERATOR_COST * outer.rows

    # Grace Hash Join I/O（当 inner 超出 work_mem 时）
    numbatches = max(1, math.ceil(inner_pages / work_mem_pages))
    if numbatches > 1:
        outer_pages = outer.rows * 24 / BYTES_PER_PAGE
        startup_cost += SEQ_PAGE_COST * inner_pages
        run_cost += SEQ_PAGE_COST * (inner_pages + 2 * outer_pages)

    output_rows  = outer.rows * inner.rows * join_sel
    cpu_qual     = (CPU_OPERATOR_COST * 2 + CPU_TUPLE_COST) * output_rows
    run_cost += cpu_qual

    batch_note = f" [Grace {numbatches} batches]" if numbatches > 1 else ""
    return Path(
        rels=outer.rels | inner.rels,
        rows=output_rows,
        startup_cost=startup_cost,
        total_cost=startup_cost + run_cost,
        description=f"HashJoin({outer.description} ⋈ {inner.description}){batch_note}"
    )


def cost_nestloop(outer: Path, inner_rel: Relation,
                  join_sel: float) -> Path:
    """
    对应 initial_cost_nestloop()
    
    适合小 inner，或 inner 有 index（这里模拟无 index 版本）。
    startup  = outer.startup + inner.startup（第一次扫 inner）
    run      = outer.run + outer.rows * inner.total（每行 outer 扫一遍 inner）
    """
    inner_scan  = cost_seqscan(inner_rel)
    startup     = outer.startup_cost + inner_scan.startup_cost
    run_cost    = (outer.total_cost - outer.startup_cost)
    # 第一次扫 inner + (outer.rows - 1) 次 rescan
    run_cost   += inner_scan.total_cost
    run_cost   += max(0, outer.rows - 1) * inner_scan.total_cost

    output_rows = outer.rows * inner_rel.rows * join_sel
    cpu_qual    = (CPU_OPERATOR_COST * 2 + CPU_TUPLE_COST) * output_rows
    run_cost   += cpu_qual

    return Path(
        rels=outer.rels | frozenset([inner_rel.name]),
        rows=output_rows,
        startup_cost=startup,
        total_cost=startup + run_cost,
        description=f"NestLoop({outer.description} ⋈ SeqScan({inner_rel.name}))"
    )


# ---------- 动态规划 join 排序（对应 join_search_one_level）----------

def dp_join_search(
    relations: list[Relation],
    join_clauses: list[JoinClause]
) -> Optional[Path]:
    """
    对应 joinrels.c 的 join_search_one_level()
    
    build join_rel_level[k] 从 k=1 到 k=len(relations)。
    每个 level 保存当前已找到的所有 k 表 join 的最优 Path。
    """
    rel_map = {r.name: r for r in relations}

    # join selectivity 查找表（简化：只有直连 join clause 才有选择性）
    def get_join_sel(rel_names: FrozenSet[str]) -> float:
        for clause in join_clauses:
            if {clause.left, clause.right}.issubset(rel_names):
                return clause.selectivity
        return 0.1  # 无 join clause，笛卡儿积，给个默认值

    # join_rel_level[k] = dict: frozenset(rel_names) -> best Path
    level: Dict[FrozenSet[str], Path] = {}

    # level 1：base relation scan
    for rel in relations:
        path = cost_seqscan(rel)
        level[frozenset([rel.name])] = path
    print("Level 1 paths:")
    for k, p in level.items():
        print(f"  {k}: cost={p.total_cost:.1f}  [{p.description}]")

    current_level = dict(level)  # 当前级别的最优路径

    for join_size in range(2, len(relations) + 1):
        print(f"\nLevel {join_size} paths:")
        next_level: Dict[FrozenSet[str], Path] = {}

        # 枚举所有 join_size 大小的关系子集
        for combo in itertools.combinations(relations, join_size):
            combo_set = frozenset(r.name for r in combo)
            best_path: Optional[Path] = None

            # 枚举分割方式：左边 k 个关系，右边 join_size-k 个关系
            for k in range(1, join_size):
                for left_combo in itertools.combinations(combo, k):
                    left_set   = frozenset(r.name for r in left_combo)
                    right_set  = combo_set - left_set

                    # 查找左右子集的最优路径
                    outer_path = current_level.get(left_set) or level.get(left_set)
                    if outer_path is None:
                        continue
                    # 只有 1 个关系的 inner 尝试 NestLoop，否则只试 HashJoin
                    join_sel = get_join_sel(combo_set)

                    if len(right_set) == 1:
                        inner_name = next(iter(right_set))
                        inner_rel  = rel_map[inner_name]

                        # 方案 1：HashJoin
                        inner_path = cost_seqscan(inner_rel)
                        hj = cost_hashjoin(outer_path, inner_path, join_sel)
                        if best_path is None or hj < best_path:
                            best_path = hj

                        # 方案 2：NestLoop（小表 inner）
                        if inner_rel.rows <= 10000:
                            nl = cost_nestloop(outer_path, inner_rel, join_sel)
                            if best_path is None or nl < best_path:
                                best_path = nl
                    else:
                        inner_path = (current_level.get(right_set) or
                                      level.get(right_set))
                        if inner_path is None:
                            continue
                        hj = cost_hashjoin(outer_path, inner_path, join_sel)
                        if best_path is None or hj < best_path:
                            best_path = hj

            if best_path:
                existing = next_level.get(combo_set)
                if existing is None or best_path < existing:
                    next_level[combo_set] = best_path
                    print(f"  {set(combo_set)}: cost={best_path.total_cost:.1f}"
                          f"  [{best_path.description}]")

        current_level = next_level
        level.update(next_level)

    all_rels = frozenset(r.name for r in relations)
    return level.get(all_rels)


# ---------- 测试 ----------

def main():
    print("=" * 70)
    print("Mini 代价优化器 — Join 顺序选择")
    print("=" * 70)

    # 三张表：orders(大), customers(中), products(小)
    relations = [
        Relation("orders",    rows=1_000_000, width=50),
        Relation("customers", rows=100_000,   width=80),
        Relation("products",  rows=5_000,     width=120),
    ]

    # join clause：orders.customer_id = customers.id
    #              orders.product_id  = products.id
    join_clauses = [
        JoinClause("orders", "customers", selectivity=1/100_000),
        JoinClause("orders", "products",  selectivity=1/5_000),
    ]

    print("\n表信息:")
    for r in relations:
        print(f"  {r.name:<12}: {r.rows:>9,} 行, {r.pages:.0f} 页, {r.width}B/row")

    best = dp_join_search(relations, join_clauses)

    print(f"\n{'='*70}")
    print(f"最优执行计划：")
    print(f"  {best.description}")
    print(f"  估计代价: {best.total_cost:.1f}")
    print(f"  估计输出行数: {best.rows:.0f}")
    print(f"{'='*70}")
    print("\n与 PostgreSQL EXPLAIN 对比方法：")
    print("  psql -c \"EXPLAIN SELECT * FROM orders o")
    print("           JOIN customers c ON o.customer_id = c.id")
    print("           JOIN products p ON o.product_id = p.id;\"")


if __name__ == "__main__":
    main()
```

## 第六部分：三种 Join 算法对比

| 维度 | Nested Loop | Hash Join | Merge Join |
|------|-------------|-----------|------------|
| 时间复杂度 | O(M × N)，有 index 时 O(M log N) | O(M + N) | O(M log M + N log N) |
| 内存需求 | O(1)（流水线）| O(min(M,N))（build 侧）| O(sort buffer) |
| 适合场景 | 小 outer，inner 有 index；OLTP | 大多数 equi-join；work_mem 够 | 输入已有序；range join 无法用 hash |
| startup cost | 低（立刻开始）| 高（需先 build）| 高（需先 sort） |
| 可流水线 | 是（index NL）| 否（需先 build）| 部分 |
| non-equijoin | 支持（theta join）| 不支持 | 部分支持（range）|
| 并行性 | 难（inner rescan 有 lock 问题）| 好（Parallel Hash Join PG11+）| 好（预排序后 merge） |

**什么时候 PostgreSQL 选 NestLoop 而不是 HashJoin**：

1. inner 关系有 index，且 outer 行数少（index NestLoop 代价 < hash build 代价）
2. `enable_hashjoin = off` 强制禁用
3. Outer join 的某些情况（实现细节差异）
4. SEMI JOIN 的内侧有唯一性保证（single_match 优化）

## 第七部分：失败模式与生产真坑

### 坑 1：统计信息陈旧导致行数估算严重偏差

**现象**：EXPLAIN 显示 `rows=1`，实际执行返回 100 万行；优化器选了 NestLoop，实际跑了 20 分钟。

**根因**：`ANALYZE` 没有及时运行，或 autovacuum 被禁用/延迟。优化器的所有代价计算以 `reltuples` 为基础，估算失误直接导致 join 算法选错。

```sql
-- 诊断
SELECT relname, reltuples, relpages,
       now() - pg_stat_user_tables.last_analyze AS since_analyze
FROM pg_class
JOIN pg_stat_user_tables ON relname = relname  -- (简化，实际需 relid 关联)
WHERE relname = 'your_table';

-- 修复
ANALYZE your_table;
-- 或针对关键列提高 statistics target
ALTER TABLE your_table ALTER COLUMN key_col SET STATISTICS 500;
ANALYZE your_table;
```

### 坑 2：相关列的选择性估算偏差（Correlated Columns）

**现象**：WHERE city = 'Shanghai' AND province = 'Shanghai' 的 selectivity 被严重低估，导致优化器认为这个 filter 极少返回行，选了不合适的计划。

**根因**：PostgreSQL 默认假设 WHERE 中多个列相互独立，两个条件的 selectivity 乘积往往远小于真实选择性（两个相关列）。

**修复**：

```sql
-- 创建多列统计信息（PostgreSQL 10+）
CREATE STATISTICS stts (dependencies) ON city, province FROM your_table;
ANALYZE your_table;
-- 之后 city+province 组合的 selectivity 会更准确
```

### 坑 3：Hash Join 内存溢出退化为 Grace Hash Join

**现象**：查询突然变慢 5-10x，`EXPLAIN ANALYZE` 显示 `Batches: 16`。

**根因**：`work_mem` 设置过低（默认 4MB），Hash Join 的 build 侧数据超出内存，触发 Grace Hash Join，需要多次读写磁盘 batch 文件。

```sql
-- 临时调整（当前 session）
SET work_mem = '256MB';
EXPLAIN ANALYZE SELECT ...;

-- 确认 Batches 是否降到 1
-- 如果 Batches: 1 且速度恢复，则是 work_mem 问题
```

### 坑 4：GEQO 对超多表 join 的随机性

**现象**：同一个查询两次执行，EXPLAIN 输出的计划不同，性能差异 5x。

**根因**：超过 `geqo_threshold`（默认 12）张表的 join 会触发遗传算法，每次运行的随机初始种群不同，可能找到局部最优而非全局最优。

**缓解**：
```sql
-- 调高阈值让 DP 处理（但会增加规划时间）
SET geqo_threshold = 20;

-- 或固定随机种子（调试用）
SET geqo_seed = 0.5;

-- 或给 planner 更多时间
SET join_collapse_limit = 20;
```

### 坑 5：Index Scan 在高 random_page_cost 环境下被错误选中

**现象**：云数据库（EBS/COS 高 I/O 延迟），PostgreSQL 默认 `random_page_cost = 4.0`，但实际 SSD 的 random/seq I/O 比值只有 1.1，导致优化器高估 random I/O 代价，放弃用 index 转用 seq scan。

**修复**：
```sql
-- SSD/NVMe 环境
ALTER SYSTEM SET random_page_cost = 1.1;  -- 或 1.5

-- 云 SSD（EBS gp3 等）
ALTER SYSTEM SET random_page_cost = 1.5;

-- 调整后 reload
SELECT pg_reload_conf();
```

## 章末五件套

### 1. 核心概念题

**Q1**: Volcano 模型中，`ExecProcNode` 是通过虚函数表（vtable）实现的吗？不是的话，PostgreSQL 怎么实现多态？

**答**：不是虚函数表，是 **C 函数指针**（`ExecProcNodeMtd`）。在 `ExecInitSeqScan` 等初始化函数中，根据运行时条件（是否有 qual、projection）把不同的函数指针赋给 `PlanState.ExecProcNode`。比 C++ 虚函数表更灵活：一个结构体可以在运行时切换实现，且可以做到无分支（直接调用对应的特化版本），避免 branch misprediction。

**Q2**: PostgreSQL Hash Join 的 `startup_cost` 为什么要包含 inner 关系的全部 total_cost？

**答**：因为 Hash Join 在返回第一条结果之前，必须把 inner 关系全部扫描并插入哈希表（build 阶段）。这意味着 startup_cost = inner.total_cost + build CPU 开销。这也是为什么对于 `LIMIT 1` 这类查询，NestLoop 往往更好：NestLoop 找到第一条匹配就可以返回，startup_cost 很低。

### 2. 源码阅读题

**Q3**: 在 `ExecNestLoop` 中，`nl->nestParams` 是用来做什么的？如果 inner 是 IndexScan，这个机制如何让 index lookup 使用 outer tuple 的字段值？

**答**：`nestParams` 是 NestLoop 节点持有的参数传递列表。每次拉到新的 outer tuple，执行器遍历 `nestParams`，把 outer tuple 对应字段的值写入 `econtext->ecxt_param_exec_vals[paramno]`，然后把 `paramno` 加入 inner plan 的 `chgParam`。当下次调用 `ExecProcNode(innerPlan)` 时，`ExecProcNode` 检测到 `chgParam` 非空，会先调用 `ExecReScan`，inner 的 IndexScan 在 rescan 时用新的参数值重新计算 index scan 的起始/终止 key，实现 correlated index lookup。

### 3. 代码扩展题（基于 Demo 1 扩展）

**Q4**: 给 Demo 1 的 Volcano 执行器添加一个 `SortOperator` 算子，实现对任意列的 ORDER BY。要求：`SortOperator` 必须在 `open` 时完整物化子算子的结果，在 `next` 时逐条返回排好序的 tuple。为什么 Sort 必须物化而不能流水线？

**参考实现要点**：
```python
class Sort(Operator):
    def __init__(self, child, key_fn, reverse=False):
        self.child = child
        self.key_fn = key_fn
        self.reverse = reverse
        self._sorted = []
        self._idx = 0

    def open(self):
        self.child.open()
        # 物化：必须全部读完才能排序
        rows = []
        while True:
            t = self.child.next()
            if t is None:
                break
            rows.append(t)
        self._sorted = sorted(rows, key=self.key_fn, reverse=self.reverse)
        self._idx = 0

    def next(self):
        if self._idx >= len(self._sorted):
            return None
        t = self._sorted[self._idx]
        self._idx += 1
        return t
```

Sort 必须物化的原因：排序需要看到所有输入才能确定第一小值，不可能在看到全部数据之前输出任何结果（这就是为什么 Sort 的 `startup_cost` 高，等于子算子 `total_cost` + 排序开销）。

### 4. 调试/诊断题

**Q5**: 线上查询突然变慢，`EXPLAIN ANALYZE` 显示某节点 `actual rows=1000000` 但 `rows=1`，且类型是 `Hash Join`。你的诊断步骤？

```
1. 确认统计信息是否陈旧
   SELECT relname, last_analyze, n_live_tup, n_dead_tup
   FROM pg_stat_user_tables WHERE relname = 'suspect_table';

2. 检查是否有 correlation 问题（多列 WHERE）
   SELECT attname, correlation FROM pg_stats
   WHERE tablename = 'suspect_table';

3. 手动 ANALYZE 后重新 EXPLAIN 确认 rows 估算是否改善
   ANALYZE suspect_table;
   EXPLAIN (ANALYZE, BUFFERS) SELECT ...;

4. 如果是多列相关性问题，考虑 CREATE STATISTICS
   CREATE STATISTICS stts(dependencies) ON col1, col2 FROM suspect_table;
   ANALYZE suspect_table;
```

## 未来方向

1. **JIT 编译**：PostgreSQL 14+ 的 LLVM JIT 对表达式求值（`ExecEvalExpr`）和 tuple 解包（`slot_deform_heap_tuple`）做代码生成，绕过 interpreter 开销，在 OLAP 查询上有 20-50% 提升。

2. **向量化执行引擎**：DuckDB 完全重写了执行器，采用 vectorized pull model（每次 `next()` 返回 vector<1024 rows>），配合列存，在 OLAP 上比 PG 快 10-50x。

3. **并行查询扩展**：PG 的 Parallel Hash Join（PG11+）通过 Parallel Hash 节点和 DSA（Dynamic Shared Memory Area）实现多 worker 共享哈希表，是现代 HTAP 的基础。

4. **自适应查询执行（AQE）**：Spark 3.0 和商业数据库（如 SQL Server）已支持运行时重新规划（re-optimize at runtime）：在 Hash Join build 完成后根据实际行数决定是否 broadcast，打破了 Selinger 模型静态规划的限制。PostgreSQL 路线图中也有类似讨论（adaptive query execution）。
