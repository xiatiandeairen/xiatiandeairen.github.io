> **一句话定位**：设备驱动是内核的最大代码量领域——不是因为驱动算法复杂，而是因为硬件世界没有统一抽象，内核用"统一设备模型"做了第一层，驱动工程师在第二层按 bus/device/driver 三角关系填空。

---

## TL;DR · 本章最硬结论

1. **kobject 是一切设备对象的基类**：bus、device、driver 都内嵌 kobject，sysfs 的每个目录就是一个 kobject，这是"硬件拓扑 = 文件树"的基础。
2. **probe/remove 是驱动生命周期的核心契约**：当 bus 把 device 和 driver 配对成功，就回调 driver->probe；模块卸载或热拔插时回调 remove。驱动的所有初始化/清理都在这两个函数里。
3. **三类设备（字符/块/网络）注册路径完全不同**：字符设备通过 cdev + file_operations 挂 VFS；块设备通过 gendisk + request_queue 挂 block 层；网络设备通过 net_device 挂协议栈——三条路，互不相交。
4. **设备号是字符/块设备的"坐标"**：主设备号找 driver，次设备号找具体实例，这个映射由内核的 chrdevs/blkdevs 表维护。
5. **用户态驱动（UIO/VFIO）是云计算基础设施的基石**：DPDK、SPDK、虚拟机设备直通都依赖 VFIO 绕开内核协议栈，把 I/O 路径搬进用户态。

## 1. 背景：为什么需要统一设备模型

### 1.1 硬件多样性爆炸

2003 年之前，Linux 没有统一设备模型。PCI 驱动有自己的注册 API，USB 驱动有另一套，platform 设备又是另一套。导致：

- **重复的电源管理代码**：每种 bus 各实现一遍 suspend/resume，逻辑相似但互不复用，BUG 反复出现。
- **没有统一的拓扑视图**：无法回答"这块网卡在哪条 PCI 总线的第几个 slot"，工具无法跨 bus 类型查询。
- **热拔插（hotplug）是噩梦**：USB 热插、PCI-Express 热插、PCMCIA 热插各有各的通知机制，udev 根本没法统一响应。

### 1.2 Greg Kroah-Hartman 的答案：kobject + sysfs

2.6.0 合入统一设备模型（LDD3 时代），核心思路：

```
硬件拓扑   ←→   内存对象图（kobject tree）   ←→   sysfs 文件树
```

这不是"漂亮的抽象"，是**工程必要性**：

- 电源管理（PM）要遍历整棵设备树，按拓扑顺序 suspend/resume——没有统一对象图做不到。
- udevd 要监听设备事件（uevent）——事件源必须标准化到 kobject 层。
- sysfs 让用户态 `lspci`、`lsusb`、`udevadm info` 都成为读文件操作，不需要 ioctl。

**工程推论：**

1. kobject 引用计数（`kref`）决定设备对象何时真正释放——这是驱动最常见 use-after-free 根源。
2. sysfs 属性读写就是 `show`/`store` 回调。kernfs 层保证的是**节点本身的生命周期/重命名/删除安全**（早期是全局 `kernfs_mutex`，约 5.15–5.17 起重构为 `kernfs_rwsem` + hashed lock，减少全局锁竞争）——它**不**替你串行化 `show`/`store` 与驱动其它代码路径之间对设备状态的并发访问。常见误区：以为 sysfs 帮你加了锁。实际上多个读者可并发进 `show`，`store` 也可与中断/工作队列里的驱动代码并发；**驱动暴露的可变状态仍需自己加锁（通常一把 `mutex` 或 `spinlock`）**。
3. uevent 是 udev 的信号源，驱动里显式调 `kobject_uevent` 或通过 `device_add` 隐式触发。

## 2. 演进：从混乱到统一

### 2.1 早期（2.4 及以前）：各自为政

```
PCI layer:  pci_register_driver() → pci_driver.probe()
USB layer:  usb_register()        → usb_driver.probe()
misc:       register_chrdev()     → file_operations
```

三套 API，三套生命周期，三套电源管理。驱动工程师换 bus 类型基本等于重写。

### 2.2 2.6 统一设备模型合入

引入：
- `kobject` / `kset` / `ktype` 三件套作为基类
- `struct bus_type`、`struct device`、`struct device_driver` 统一注册接口
- `sysfs` 作为内核对象的 VFS 投影（`/sys` 挂载点）
- `platform_bus`：给没有物理 bus 的片上设备提供虚拟 bus

### 2.3 设备树（Device Tree）进入主线（2011 年左右）

ARM 架构此前靠 board 文件（`arch/arm/mach-xxx/`）硬编码板级拓扑，2000 多个 board 文件让内核膨胀失控。受 PowerPC OpenFirmware 启发，ARM 引入设备树（DT），把板级硬件描述从 C 代码移到 DTS 文件。

- **DTS → DTB（二进制）**：bootloader（U-Boot）传给内核
- 内核解析 DTB 生成 `of_node`，platform 设备通过 `compatible` 属性匹配驱动
- 主流嵌入式平台（ARM、RISC-V）现在全用 DT；x86 用 ACPI

### 2.4 ACPI（x86 服务器）

x86 服务器的"设备树"叫 ACPI（Advanced Configuration and Power Interface），BIOS/UEFI 提供 ACPI 表，内核解析后生成设备对象，通过 `acpi_driver` 匹配。逻辑与 DT 类似，只是描述语言不同（ASL vs DTS）。

**演进核心教训**：统一设备模型是**应对硬件多样性的代价均摊**——内核维护一套基础设施，驱动工程师只填 probe/remove/ops 三个位置。

## 3. 现状：核心机制与数据结构

### 3.1 kobject 基类

```c
struct kobject {
    const char          *name;
    struct list_head    entry;       /* 在 kset 里的链表节点 */
    struct kobject      *parent;     /* 父节点 → sysfs 父目录 */
    struct kset         *kset;
    const struct kobj_type *ktype;
    struct kernfs_node  *sd;         /* sysfs 目录节点 */
    struct kref         kref;        /* 引用计数 */
    /* ... */
};
```

重要规则：
- **永远不要栈分配 kobject**：kobject 释放由 kref 驱动，必须堆分配（或内嵌在更大的结构里），kref 归零时调 `ktype->release`。
- `kobject_init_and_add()` 同时初始化 + 在 sysfs 创建目录。
- `kobject_put()` 减引用，归零时自动调 release，这是安全释放的唯一路径。

### 3.2 bus / device / driver 三角

```
         bus_type
        /         \
   device ←——→  device_driver
        \         /
         match()
         probe()
```

```c
struct bus_type {
    const char *name;
    int (*match)(struct device *dev, struct device_driver *drv);
    int (*probe)(struct device *dev);   /* 可委托给 drv->probe */
    /* suspend, resume, ... */
};

struct device {
    struct kobject  kobj;
    struct device   *parent;
    struct bus_type *bus;
    struct device_driver *driver;  /* 配对成功后指向 driver */
    void            *driver_data;  /* 驱动私有数据，devm 管理 */
    /* ... */
};

struct device_driver {
    const char      *name;
    struct bus_type *bus;
    int (*probe)(struct device *dev);
    void (*remove)(struct device *dev);
    /* ... */
};
```

**配对流程（以 platform bus 为例）**：

```
device_register(dev)
    → bus_add_device()
    → bus_probe_device()
        → __device_attach()
            → bus->match(dev, drv) → 匹配成功
            → really_probe()
                → drv->probe(dev)   ← 驱动在这里初始化硬件
```

💡 **坑点**：`probe()` 返回 `-EPROBE_DEFER` 表示依赖的资源还没准备好（比如时钟、GPIO），内核会把该 device 放回 deferred 队列，等依赖就绪后重试。新手经常忘记处理这个 error code，导致 probe 在某些启动顺序下静默失败。

### 3.3 sysfs 布局

```
/sys/
├── bus/
│   ├── pci/
│   │   ├── devices/  → 符号链接到 /sys/devices/...
│   │   └── drivers/
│   └── platform/
├── devices/           ← 真实拓扑树
│   ├── platform/
│   │   └── serial8250.0/
│   └── pci0000:00/
│       └── 0000:00:1c.0/  ← PCI 设备目录
│           ├── vendor     ← kobject attribute (只读)
│           └── enable     ← 可写 attribute
├── class/
│   ├── net/           → 符号链接到具体设备
│   └── block/
└── module/
```

每个目录 = 一个 kobject；每个文件 = 一个 `sysfs_attribute`，读写分别对应 `show`/`store` 回调。

### 3.4 三类设备

#### 3.4.1 字符设备（Character Device）

最简单的设备模型，适合顺序 I/O（串口、GPIO、传感器、/dev/random）。

```c
/* 注册路径 */
alloc_chrdev_region(&devno, 0, count, "mydev");
                                    ↑ 动态分配主设备号（推荐）
cdev_init(&cdev, &my_fops);
cdev_add(&cdev, devno, count);

device_create(class, NULL, devno, NULL, "mydev%d", minor);
                                         ↑ 在 /dev/ 创建节点（需 udev 配合）
```

核心数据结构：

| 结构体 | 职责 |
|---|---|
| `struct cdev` | 字符设备内核表示，持有 `file_operations *` |
| `dev_t` | 设备号，高 12 位主设备号，低 20 位次设备号 |
| `struct file_operations` | VFS→驱动的操作向量表（见第 14 章） |
| `struct inode` | VFS inode，字段 `i_cdev` 指向 cdev |

`file_operations` 核心回调：

```c
struct file_operations my_fops = {
    .owner   = THIS_MODULE,
    .open    = my_open,
    .release = my_release,
    .read    = my_read,
    .write   = my_write,
    .unlocked_ioctl = my_ioctl,
    .poll    = my_poll,       /* select/epoll 支持 */
    .mmap    = my_mmap,       /* 零拷贝 mmap 支持 */
};
```

🎯 **面试加分点**：`unlocked_ioctl` vs `compat_ioctl`。前者是 64-bit 内核的标准路径；后者处理 32-bit 用户态程序在 64-bit 内核上的 ioctl（指针大小不同），网卡驱动和存储驱动通常都需要实现 `compat_ioctl`。

#### 3.4.2 块设备（Block Device）

块设备的访问路径经过内核 I/O 调度器和 Page Cache（见第 16 章），适合随机存取（磁盘、SSD、NVMe）。

```
用户 read() → VFS → Page Cache → bio → request_queue → 块驱动 → 硬件
```

核心结构：
- `struct gendisk`：代表一个物理或逻辑块设备
- `struct request_queue`：I/O 请求队列，MQ（multi-queue）架构（近年内核）
- `struct block_device_operations`：类似 file_operations 但更简单

块驱动比字符驱动复杂得多，需要处理 DMA、scatter-gather、队列深度等问题，NVMe 驱动是现代块驱动的标杆实现（见第 16 章）。

#### 3.4.3 网络设备（Network Device）

网络设备不走 VFS，直接挂协议栈，通过 socket API 访问。

核心结构：
- `struct net_device`：网络设备，`alloc_etherdev()` 分配
- `struct net_device_ops`：操作向量表（`ndo_open`、`ndo_start_xmit` 等）
- `struct sk_buff`：网络数据包，驱动收到数据后构造 skb 送给协议栈

网络驱动调用 `register_netdev()` 注册，不经过 cdev 路径。

**三类设备对比：**

| 维度 | 字符设备 | 块设备 | 网络设备 |
|---|---|---|---|
| 访问接口 | /dev/xxx + file_operations | /dev/sdx + block_device_operations | socket API |
| 内核路径 | VFS 直通 | VFS → Page Cache → bio | 协议栈 → sk_buff |
| 典型例子 | 串口、GPIO、/dev/null | 磁盘、SSD | eth0、wlan0 |
| 注册 API | `cdev_add()` | `add_disk()` | `register_netdev()` |
| 用户感知 | open/read/write/ioctl | mount/dd/fio | socket/ping/ip |

### 3.5 设备号管理

```c
/* 主设备号查找驱动，次设备号区分实例 */
dev_t devno = MKDEV(major, minor);
MAJOR(devno);  /* 提取主设备号 */
MINOR(devno);  /* 提取次设备号 */

/* 静态分配（不推荐，可能冲突） */
register_chrdev_region(MKDEV(240, 0), 1, "mydev");

/* 动态分配（推荐） */
alloc_chrdev_region(&devno, 0 /*first_minor*/, 1 /*count*/, "mydev");
```

`/proc/devices` 列出已注册的主设备号和驱动名。内核内部（`fs/char_dev.c`）用一个按 major 取模的哈希数组 `static struct char_device_struct *chrdevs[CHRDEV_MAJOR_HASH_SIZE]`，同一 major 下的多个 minor 区间挂成链表，`__register_chrdev_region` 负责插入。注意这层 `char_device_struct` 表只做"major→注册名+minor 区间"的占位/查重；真正把 open 路由到 `file_operations` 的是另一套 `cdev` + `kobj_map`（见 3.1、§6 Q6）。

### 3.6 probe/remove 生命周期

以 platform driver 为例：

```c
static int my_probe(struct platform_device *pdev)
{
    struct my_priv *priv;
    int ret;

    /* devm_* 系列：资源自动跟随 device 释放，remove 不需要手动释放 */
    priv = devm_kzalloc(&pdev->dev, sizeof(*priv), GFP_KERNEL);
    if (!priv)
        return -ENOMEM;

    priv->base = devm_platform_ioremap_resource(pdev, 0);
    if (IS_ERR(priv->base))
        return PTR_ERR(priv->base);

    priv->irq = platform_get_irq(pdev, 0);
    if (priv->irq < 0)
        return priv->irq;   /* platform_get_irq 失败已返回负 errno，且自带 dev_err 打印 */

    /* devm_request_irq：IRQ 跟随 device 自动释放。返回值必须查 —
     * 共享线申请失败/handler 注册失败都从这里返回错误 */
    ret = devm_request_irq(&pdev->dev, priv->irq, my_irq_handler,
                           IRQF_SHARED, "my-dev", priv);
    if (ret)
        return ret;

    platform_set_drvdata(pdev, priv);

    /* 最后注册对外接口（cdev 或 net_device）：放最后，确保对外可见时
     * 硬件/中断已就绪，避免"接口已开放但 probe 仍在初始化"的竞态 */
    return my_register_device(priv);
}

static int my_remove(struct platform_device *pdev)
{
    struct my_priv *priv = platform_get_drvdata(pdev);
    my_unregister_device(priv);
    /* devm 资源自动释放，不需要手动 free/unmap/free_irq */
    return 0;
}
/*
 * 时效坑（近年内核）：platform_driver.remove 的返回值历史上是 int，
 * 但 remove 的返回值内核其实从不据此中止卸载（返回错误也照样 detach），
 * 是个误导性签名。6.11 把新的 void 版本（曾叫 .remove_new）正式扶正为
 * .remove，老的 int 版本正在被全树转换淘汰。新驱动建议写
 *   static void my_remove(struct platform_device *pdev) { ... }
 * 跨版本写作时以目标内核源码树里同类驱动为准。
 */

static const struct of_device_id my_of_match[] = {
    { .compatible = "vendor,my-device" },
    { }
};

static struct platform_driver my_driver = {
    .probe  = my_probe,
    .remove = my_remove,
    .driver = {
        .name           = "my-device",
        .of_match_table = my_of_match,  /* DT 匹配 */
    },
};
module_platform_driver(my_driver);  /* 展开为 module_init/module_exit */
```

💡 **devm 是现代驱动的标准用法**：`devm_*` 函数把资源加入该 `device` 的 devres 链表。释放时机要分清楚（高频面试坑）：

- **probe 中途失败**：`really_probe` 在 probe 返回错误时立刻调 `devres_release_all()`，已申请的 devm 资源按 LIFO 全部回滚——这正是它消灭 probe 错误路径 goto 迷宫的原因。
- **正常 unbind / remove**：`__device_release_driver` 先调 `drv->remove()`，**remove 返回后**再调 `devres_release_all()` 释放本次绑定申请的 devm 资源。所以 devm 资源是绑在"驱动这次 attach"上、随 **unbind** 释放，**不是**等到 `struct device` 的 `kref` 归零、`device_release()` 真正析构对象时才释放——这两个时机可能相差很远（设备对象可能被 sysfs 句柄长期持有）。

不用 devm 的老代码在 probe 失败路径上极易出 resource leak。

### 3.7 设备树（Device Tree）机制

```dts
/* arch/arm/boot/dts/my-board.dts (示意) */
my_uart: serial@10000000 {
    compatible = "vendor,my-uart", "ns16550a";  /* 从具体到通用，依次尝试匹配 */
    reg = <0x10000000 0x1000>;                  /* 寄存器基地址和大小 */
    interrupts = <GIC_SPI 42 IRQ_TYPE_LEVEL_HIGH>;
    clocks = <&clk 1>;
    status = "okay";
};
```

内核通过 `of_match_table` 中的 `compatible` 字符串找到驱动：

```c
static const struct of_device_id my_uart_of_match[] = {
    { .compatible = "vendor,my-uart", .data = &my_uart_quirks },
    { .compatible = "ns16550a" },
    { }
};
```

`platform_get_resource()` 和 `of_property_read_u32()` 是从 DT 节点读取配置的常用 API。

**ACPI 对照**：x86 路径用 ACPI ID（如 `ACPI0003`）替换 `compatible`，其余 probe/remove 生命周期相同。

### 3.8 用户态驱动：UIO 与 VFIO

#### UIO（Userspace I/O）

最简单的用户态驱动框架：内核只做 interrupt forwarding 和 mmap，驱动逻辑完全在用户态。

```
用户态进程
    ↕  read(/dev/uio0)  ← 等中断
    ↕  mmap(/dev/uio0)  ← 映射设备寄存器
内核 UIO 框架
    ↕  中断 → 写 interrupt count
硬件
```

限制：没有 IOMMU 保护，用户态可以 DMA 到任意物理地址，**不安全，无法用于虚拟化**。

#### VFIO（Virtual Function I/O）

VFIO = IOMMU + 设备直通。核心能力：

1. 把设备 DMA 限制在 IOMMU group 定义的地址空间内（安全）
2. 把整个 IOMMU group 的设备暴露给用户态，QEMU/KVM 用来实现 PCIe passthrough
3. DPDK、SPDK 用 VFIO 把 NIC/NVMe 直接 bypass 内核协议栈

```
QEMU/KVM / DPDK
    ↕  /dev/vfio/0  (VFIO container fd)
VFIO 内核框架
    ↕  IOMMU domain  ← 限制 DMA 范围
    ↕  MSI/MSI-X 中断重映射
硬件 NIC / NVMe
```

🎯 **面试加分点**：VFIO mediated device（mdev）是在一个物理设备内创建多个虚拟设备实例（如 Intel GVT-g GPU 虚拟化），每个 mdev 对应一个 VM，不需要 SR-IOV 硬件支持。（近年内核 mdev 接口有重构，待核具体 API）

## 4. 落地：生产调试与工具

### 4.1 常用命令

```bash
# 查看字符/块设备已注册主设备号
cat /proc/devices

# 查看所有 platform 设备
ls /sys/bus/platform/devices/

# 查看设备绑定的驱动
cat /sys/bus/pci/devices/0000:00:1c.0/driver

# 手动绑定/解绑驱动（不热拔插时用于测试）
echo "0000:00:1c.0" > /sys/bus/pci/drivers/pcieport/unbind
```

### 4.2 驱动调试典型场景

**场景 A：probe 没被调用**

排查流程：
1. `dmesg | grep my_driver`：确认模块加载
2. `cat /sys/bus/platform/devices/my-device/driver`：是否有 driver 符号链接
3. 检查 DT `compatible` 字符串是否与 `of_device_id` 精确匹配（大小写、版本号）
4. 检查 DT 节点 `status = "okay"`（否则内核跳过）
5. `echo "file core/dd.c +p"` 打开 bus_probe_device 的 dynamic debug

**场景 B：probe 中 -EPROBE_DEFER 无限循环**

- 用 `cat /sys/kernel/debug/devices_deferred` 查看谁还卡在 deferred 队列（需 `CONFIG_DEBUG_FS`；该文件随 deferred-probe 调试支持引入，主流发行版内核都有）
- 被依赖的资源（clock、gpio、regulator）是否已注册
- 依赖链是否有环（A 等 B，B 等 A）

**场景 C：字符设备 open 返回 ENODEV**

- `cat /proc/devices` 确认主设备号已注册
- `ls -la /dev/mydev`：确认 mknod 时的主次设备号与注册一致
- `cdev_add` 是否在 `device_create` 之前调用（顺序很重要，device_create 可能触发 udev，udev 立刻 open 设备，但 cdev 还没准备好）

💡 **坑点**：`device_create` 先于 `cdev_add` 是典型 race condition。正确顺序：先 `cdev_add`，再 `device_create`（触发 udev），确保设备 open 时 cdev 已就绪。

**场景 D：驱动卸载时 kernel panic / use-after-free**

- 大概率是中断 handler 还在运行时 `free_irq` 未完成就释放了私有数据
- 全 devm 写法天然安全：unbind 时 `devres_release_all()` 按 LIFO 回滚，`devm_free_irq`（含内部 `free_irq`，会**同步等待在跑的 handler 退出**）发生在 `devm_kfree(priv)` 之前，顺序由 LIFO 保证。出 UAF 的几乎都是裸 `request_irq` + 手动 `free_irq`：手写顺序里先 `kfree(priv)` 再 `free_irq`（或 `free_irq` 漏写），handler 还在访问已释放的 priv
- 用 KASAN 编译内核后复现，报告会精确指出 UAF 点（alloc / free / use 三处 stack）

### 4.3 大厂实践

- **Google/Meta**：大量用 VFIO + DPDK bypass 内核网络栈，单机 NIC 吞吐可达 100Gbps+，内核网络栈成为瓶颈时的标准选项
- **云厂商 virtio**：virtio 是 KVM 虚拟机前后端驱动的标准，virtio-blk、virtio-net 的驱动在 guest 内核里，backend 在 QEMU 或 vhost-kernel 中。理解 virtio 环（vring）是云基础设施驱动工程师必修
- **NVMe 驱动**：现代 NVMe 驱动用 blk-mq（多队列），per-CPU 提交队列，SQ/CQ 深度调优是存储性能工程的核心杠杆（见第 16 章）

## 5. 面试视角：staff 级高频题

**Q1. kobject 引用计数是怎么工作的？release 什么时候被调用？**

答题框架：
- `kobject` 内嵌 `kref`，`kobject_get/put` 增减计数
- `kref` 归零时调 `kobj_type->release`，不是立即析构
- **陷阱**：`device_unregister` 不等于立即释放，只是从 sysfs 删除并减 refcount；如果有其他持有者（如打开的文件描述符），release 延迟到最后一个 put
- 正确答案：`struct device` 通过 `device_release` 最终释放，驱动私有数据由 `devm` 或手动在 release 里清理

**Q2. probe/remove 和 module_init/exit 有什么区别？**

答题框架：
- `module_init`：驱动模块加载时注册到 bus（`driver_register`），不直接初始化硬件
- `probe`：bus 配对成功后针对特定设备实例初始化硬件
- `remove`：热拔插或模块卸载时针对特定实例清理
- `module_exit`：从 bus 注销驱动（`driver_unregister`），触发所有实例的 remove
- **考点**：一个驱动可以同时管理多个设备实例，probe 被多次调用，每次一个实例

**Q3. 为什么推荐 devm_* 而不是手动资源管理？有什么例外？**

答题框架：
- devm 通过 devres 链表跟踪资源，`device_release` 时 LIFO 自动释放，消除 probe 错误路径的 goto 迷宫
- **例外**：资源生命周期超出 device 范围时（如共享资源被多个 device 使用），devm 不适用；性能极敏感路径（devm 有小开销）
- 成熟观点：99% 的情况用 devm，极少数跨设备共享资源用手动管理并加注释

**Q4. UIO 和 VFIO 有什么本质区别？为什么 DPDK 默认推荐 VFIO？**

答题框架：
- UIO：无 IOMMU 保护，用户态可以 DMA 到任意物理地址，在虚拟化环境中安全隐患大
- VFIO：IOMMU domain 限制 DMA 范围，支持中断重映射，可安全用于 VM passthrough
- DPDK 选 VFIO：生产环境多在 VM 里，VFIO 是唯一安全选项；裸金属 IOMMU 也开着，用 VFIO 无额外开销
- **加分**：提到 VFIO noiommu mode（无 IOMMU 硬件时 fallback，等同 UIO 安全级别）

**Q5. 设备树 compatible 属性的匹配顺序是什么？驱动怎么知道是哪个 compatible 命中的？**

答题框架：
- compatible 是字符串列表，内核从左到右逐一与 `of_device_id` 表匹配，第一个命中的获胜
- 设备一般写 `"vendor,specific-v2", "vendor,generic"` 从具体到通用
- 驱动通过 `of_device_get_match_data()` 或检查 `of_match_ptr` 的 `.data` 字段获取命中项携带的私有数据（如 chip quirks）
- **考点**：为什么这样设计——允许驱动支持多个硬件修订版，同时为旧版提供向后兼容

**Q6. 字符设备的 open 系统调用到驱动的 .open 回调经过了哪些层？**

答题框架（函数名以近年内核为准）：
- `sys_openat` → `do_filp_open` → 路径解析 → `do_dentry_open`（旧内核叫 `__dentry_open`）
- 创建字符设备 inode 时 `init_special_inode` 已把 `inode->i_fop` 设成 `def_chr_fops`，其 `.open = chrdev_open`（`fs/char_dev.c`），所以 `do_dentry_open` 调到的"第一个 open"是 `chrdev_open` 而非驱动的
- `chrdev_open` 用 `inode->i_rdev`（dev_t）在全局 `cdev_map` 上做 `kobj_lookup`，拿到注册时 `cdev_add` 进去的 `struct cdev`；首次打开时把 `inode->i_cdev` 缓存上
- 然后 `filp->f_op = fops_get(cdev->ops)`（替换成驱动的 fops），如驱动 fops 自带 `.open` 再调一次——这才是你写的 `my_open`
- 此后该 fd 的所有 read/write/ioctl 直接走 `cdev->ops`，不再经 `chrdev_open`
- **关联**：见第 14 章 VFS 的 inode/file 关系；这正是"VFS 通用 open → 按设备号路由到具体驱动 fops"的接缝

**Q7. 块设备和字符设备为什么有不同的注册路径？**

答题框架：
- 字符设备：数据流，不需要寻址，VFS 直通驱动
- 块设备：随机访问，需要 I/O 调度（CFQ→mq-deadline→none，见第 16 章）、Page Cache（读缓存）、write-back（写延迟聚合），这些是通用基础设施，不应该每个驱动重复实现
- 设计原则：块设备插入了一层通用 block 层，驱动只负责处理 bio/request，上层策略由内核统一管理

## 6. 未来演进

### 6.1 Rust for Linux 驱动

Linux 6.1 合入 Rust 基础设施，6.x 持续完善。Rust 驱动的目标：

- 用类型系统在编译期消灭 use-after-free、double-free（内核驱动最高发 bug 类型）
- `Pin<Box<T>>` + Drop trait 替代 devm 的运行时 devres 链表
- 目前：Apple AGX GPU 驱动（Asahi Linux）是第一个大型 Rust 驱动，树外开发中

**工程判断**：Rust 驱动不会在 5 年内成主流（存量 C 驱动太多），但新设备类型（如 AI 加速器）的驱动可能直接用 Rust 写。

### 6.2 用户态驱动生态成熟

- SPDK：存储协议栈完全在用户态（NVMe over VFIO），消除内核中断和上下文切换
- AF_XDP：网络数据包在内核 XDP 层直接送用户态内存，不需要完整 VFIO passthrough
- io_uring 配合 VFIO（待核：是否已有正式融合方案），理论上可以统一异步 I/O 和设备操作接口

### 6.3 设备树的演进

- YAML DTS schema 验证（近年内核）：DTS 文件通过 JSON Schema 强类型验证，防止 compatible 拼写错误等低级 bug
- Device Tree overlays：运行时动态加载/卸载 DT 片段，用于可配置硬件（FPGA、扩展板）

## 7. P1 项目：写一个完整字符设备驱动

### 目标

写一个内存字符设备 `memdev`：用内核内存模拟一块固定大小缓冲区，支持 open/read/write/lseek/ioctl（获取缓冲区大小）。

### 完整骨架（可直接编译加载）

```c
// memdev.c
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/fs.h>
#include <linux/cdev.h>
#include <linux/uaccess.h>
#include <linux/slab.h>
#include <linux/device.h>

#define DEVICE_NAME  "memdev"
#define BUF_SIZE     (4096)
#define MEMDEV_MAGIC 'm'
#define MEMDEV_GETSIZE _IOR(MEMDEV_MAGIC, 1, int)

struct memdev_priv {
    char  *buf;
    size_t size;
    struct mutex lock;
    struct cdev  cdev;
};

static struct memdev_priv *g_priv;
static dev_t g_devno;
static struct class *g_class;

/* ---- file_operations ---- */

static int memdev_open(struct inode *inode, struct file *filp)
{
    struct memdev_priv *priv =
        container_of(inode->i_cdev, struct memdev_priv, cdev);
    filp->private_data = priv;
    return 0;
}

static int memdev_release(struct inode *inode, struct file *filp)
{
    return 0;
}

static ssize_t memdev_read(struct file *filp, char __user *buf,
                            size_t count, loff_t *ppos)
{
    struct memdev_priv *priv = filp->private_data;
    ssize_t ret;

    mutex_lock(&priv->lock);
    if (*ppos >= priv->size) {
        ret = 0;
        goto out;
    }
    count = min(count, (size_t)(priv->size - *ppos));
    if (copy_to_user(buf, priv->buf + *ppos, count)) {
        ret = -EFAULT;
        goto out;
    }
    *ppos += count;
    ret = count;
out:
    mutex_unlock(&priv->lock);
    return ret;
}

static ssize_t memdev_write(struct file *filp, const char __user *buf,
                             size_t count, loff_t *ppos)
{
    struct memdev_priv *priv = filp->private_data;
    ssize_t ret;

    mutex_lock(&priv->lock);
    if (*ppos >= priv->size) {
        ret = -ENOSPC;
        goto out;
    }
    count = min(count, (size_t)(priv->size - *ppos));
    if (copy_from_user(priv->buf + *ppos, buf, count)) {
        ret = -EFAULT;
        goto out;
    }
    *ppos += count;
    ret = count;
out:
    mutex_unlock(&priv->lock);
    return ret;
}

static loff_t memdev_llseek(struct file *filp, loff_t off, int whence)
{
    struct memdev_priv *priv = filp->private_data;
    loff_t newpos;

    switch (whence) {
    case SEEK_SET: newpos = off; break;
    case SEEK_CUR: newpos = filp->f_pos + off; break;
    case SEEK_END: newpos = priv->size + off; break;
    default: return -EINVAL;
    }
    if (newpos < 0 || newpos > priv->size)
        return -EINVAL;
    filp->f_pos = newpos;
    return newpos;
}

static long memdev_ioctl(struct file *filp, unsigned int cmd,
                          unsigned long arg)
{
    struct memdev_priv *priv = filp->private_data;
    int size = (int)priv->size;

    switch (cmd) {
    case MEMDEV_GETSIZE:
        if (copy_to_user((int __user *)arg, &size, sizeof(int)))
            return -EFAULT;
        return 0;
    default:
        return -ENOTTY;
    }
}

static const struct file_operations memdev_fops = {
    .owner          = THIS_MODULE,
    .open           = memdev_open,
    .release        = memdev_release,
    .read           = memdev_read,
    .write          = memdev_write,
    .llseek         = memdev_llseek,
    .unlocked_ioctl = memdev_ioctl,
};

/* ---- module init/exit ---- */

static int __init memdev_init(void)
{
    int ret;

    /* 1. 分配私有数据 */
    g_priv = kzalloc(sizeof(*g_priv), GFP_KERNEL);
    if (!g_priv)
        return -ENOMEM;

    g_priv->buf = kzalloc(BUF_SIZE, GFP_KERNEL);
    if (!g_priv->buf) {
        ret = -ENOMEM;
        goto err_free_priv;
    }
    g_priv->size = BUF_SIZE;
    mutex_init(&g_priv->lock);

    /* 2. 动态分配主设备号 */
    ret = alloc_chrdev_region(&g_devno, 0, 1, DEVICE_NAME);
    if (ret)
        goto err_free_buf;

    /* 3. 初始化并添加 cdev（先于 device_create！） */
    cdev_init(&g_priv->cdev, &memdev_fops);
    g_priv->cdev.owner = THIS_MODULE;
    ret = cdev_add(&g_priv->cdev, g_devno, 1);
    if (ret)
        goto err_unregister_chrdev;

    /* 4. 创建 class 和 device（触发 udev 在 /dev 创建节点）
     *    注意 class_create 签名在 6.4 改了：
     *      < 6.4 : class_create(THIS_MODULE, DEVICE_NAME)
     *      >= 6.4: class_create(DEVICE_NAME)   // owner 参数被删掉
     *    下面按 6.4+ 写；老内核请加回首个 THIS_MODULE 参数。
     */
    g_class = class_create(DEVICE_NAME);
    if (IS_ERR(g_class)) {
        ret = PTR_ERR(g_class);
        goto err_cdev_del;
    }
    device_create(g_class, NULL, g_devno, NULL, DEVICE_NAME);

    pr_info("memdev: loaded, major=%d\n", MAJOR(g_devno));
    return 0;

err_cdev_del:
    cdev_del(&g_priv->cdev);
err_unregister_chrdev:
    unregister_chrdev_region(g_devno, 1);
err_free_buf:
    kfree(g_priv->buf);
err_free_priv:
    kfree(g_priv);
    return ret;
}

static void __exit memdev_exit(void)
{
    device_destroy(g_class, g_devno);
    class_destroy(g_class);
    cdev_del(&g_priv->cdev);
    unregister_chrdev_region(g_devno, 1);
    kfree(g_priv->buf);
    kfree(g_priv);
    pr_info("memdev: unloaded\n");
}

module_init(memdev_init);
module_exit(memdev_exit);
MODULE_LICENSE("GPL");
MODULE_DESCRIPTION("Simple memory char device demo");
```

### 暗坑提示

1. `class_create` 的签名在 **6.4** 变更：删掉了首个 `owner`（`THIS_MODULE`）参数，变成只收 name（commit 把 `owner` 从 `class_create` 移除）。骨架按 6.4+ 写；在 < 6.4 内核上编译报"too few arguments"时，把首参 `THIS_MODULE` 加回即可。其余如 `device_create`/`cdev_add` 跨版本稳定。判断内核版本可看 `LINUX_VERSION_CODE` 或直接照搬目标内核源码树里同类驱动的调用形式。
2. 用 KASAN 内核测试时，写 probe 错误路径的 goto 链一定要逆序释放资源，否则 KASAN 会报 memory leak。
3. 多进程同时 write 时 mutex 会串行化，生产驱动考虑 per-file 锁或 rwsem。

## 章末五件套

### 高频面试题（5 道 staff 级）

**T1. 从 `open("/dev/ttyS0")` 到串口驱动的 `.open()` 回调，内核走了哪些步骤？**

要点：VFS namei → dentry lookup → inode 类型检查 → `def_chr_fops.open` 即 `chrdev_open`（`fs/char_dev.c`）→ 用 `i_rdev` 在 `cdev_map` 上 `kobj_lookup` 查到 `cdev` → `filp->f_op = fops_get(cdev->ops)` → 调驱动 `.open()`。能画出这条路径的候选人才算真的理解 VFS + 驱动的接缝（见第 14 章）。

错误陷阱：回答"直接调驱动 open"，跳过 VFS 层，说明 VFS 理解不够深。

**T2. 一个设备驱动同时 probe 了 5 个同类设备，它们如何区分各自的私有数据？**

要点：`platform_set_drvdata(pdev, priv)` / `platform_get_drvdata(pdev)` 以 `platform_device` 指针为 key；每个设备实例有自己的 `platform_device`，所以 probe 5 次，每次传入不同 pdev，各自保存独立 priv 指针。

**T3. VFIO 如何保证用户态驱动不能 DMA 到内核的关键内存区域？**

要点：IOMMU 将设备看到的 IOVA（I/O Virtual Address）隔离于物理地址空间，DMA 只能访问 VFIO container 显式映射的 IOVA 范围；中断通过中断重映射（IR）转发，不能注入任意 vector。完整回答需提 SMMU（ARM）/ VT-d（Intel）具体实现。

**T4. devm_request_irq 和 request_irq 的释放时机有何不同？在驱动 remove 中为什么通常不需要 free_irq？**

要点：`devm_request_irq` 把 IRQ 加入 devres 链表，在驱动 **unbind**（`__device_release_driver`：先 `remove()` 返回、再 `devres_release_all()`）时按 LIFO 自动释放——**不是**等 `struct device` 的 `kref` 归零、`device_release()` 析构对象时才释放（设备对象可能被 sysfs 句柄长期持有，两个时机可能相差很远）。`request_irq` 则需要在 remove 中显式调 `free_irq`。关键安全点：`free_irq`/`devm_free_irq` 会**同步等待正在执行的 handler 退出**才返回；UAF 的常见根源是裸写法里 `free_irq` 与 `kfree(priv)` 顺序写反，handler 访问已释放数据。能讲清"unbind 释放 vs 对象析构释放"两个时机的区别才是 staff 水平。

**T5. 设备树 `status = "disabled"` 有什么用？内核如何处理？**

要点：内核在遍历 DT 节点时，`of_device_is_available()` 检查 `status` 属性，`disabled` 的节点不创建 `platform_device`，因此不触发 probe。常用于在单一 DTS 定义多块板的硬件，通过 overlay 按板启用/禁用具体外设。

### 实战项目（P1 主线延伸）

**P1-base（本章）**：如上 memdev 完整实现 + 验收。

**P1-ext-poll**：在 memdev 上加 `poll` 支持——当 buf 非空时 POLLIN 可读，buf 非满时 POLLOUT 可写，用 `wait_queue_head_t` 实现阻塞 read/write。验收：用 Python `select.select` 验证 poll 语义。

**P1-ext-proc**：在 `/proc/memdev_stats` 下暴露读写计数和当前 buf 使用量。要求用 `seq_file` 接口（不要老的 read_proc API）。

**P1-ext-mmap**：实现 `.mmap` 回调，让用户态可以 `mmap` 直接访问内核 buf。验收：mmap 写入后 read 能读到相同内容。

选哪个映射 API（有坑）：两者面向不同来源的内存。`remap_pfn_range` 直接按 pfn 建页表，给的是"裸物理页"，VMA 会被标 `VM_PFNMAP`/`VM_IO`（内核认为这块没有可参与回收/迁移的 `struct page`）。`vm_insert_page`/`vmf_insert_page` 要求传入一个**真正的、可 refcount 的非复合页**（典型来自 `alloc_page`/`alloc_pages(order=0)`），它会把页计入进程 RSS、参与正常的页生命周期。**关键坑**：本章 memdev 的 buf 是 `kzalloc` 的——slab 小块分配既不保证页对齐、底层页也可能是复合页，直接拿去 `vm_insert_page` 行为未定义。要做 mmap，正路是改用 `alloc_page`/`__get_free_pages`（或 `vmalloc` + `vmalloc_to_page` 逐页插）申请整页缓冲，再用 `remap_pfn_range`（`virt_to_pfn(buf)`）或对每页 `vm_insert_page`。这条延伸题的真正考点就是"mmap 要求页粒度、kmalloc 给的不是页"。

### 设计题

**题目**：设计一个多实例字符设备框架，支持动态创建/销毁设备实例（类似 `/dev/loop*` 动态 attach）。每个实例有独立的 4KB 缓冲区，通过 `ioctl(LOOP_CTL_GET_FREE)` 获取空闲次设备号并创建实例。

要求讨论：主设备号注册、次设备号分配策略、并发创建销毁的锁策略（全局 mutex vs per-instance lock）、device 和 cdev 的生命周期关联。

补充辨析（别答错）：分配"空闲 minor 整数"用 **IDA**（`ida_alloc_range`/`ida_free`，本质是稀疏 ID 位图，正是 `/dev/loop`、`/dev/dm-*` 这类的做法）；**xarray**（替代老 `idr`/radix tree）解决的是"整数 ID → 指针"的映射，两者职责不同、不是谁替换谁。典型组合是 IDA 分配 minor、再用 xarray 把 minor 映射到实例结构。

### 系统题

**场景**：生产服务器上一块 PCIe 网卡的驱动在高负载下 3 小时必 panic，panic 信息显示在中断 handler 里访问了一个已释放的 `struct net_device`。

排障路径：
1. 用 KASAN 编译内核（dev/staging 环境），复现 UAF，获取精确 alloc/free stack
2. 检查 `net_device` 引用计数是否配对：传统 API 是 `dev_hold`/`dev_put`，约 5.17 起引入带 `netdevice_tracker` 的 `netdev_hold`/`netdev_put`（`CONFIG_NET_DEV_REFCNT_TRACKER` 下能定位到具体未配对的 hold 点，正是为排这类 UAF 设计的）
3. 检查中断 handler 是否在 `unregister_netdev` 完成前仍在运行（`free_irq` 是否等待中断 handler 退出）
4. 检查是否有 deferred work（tasklet/workqueue）持有 `net_device` 指针但未在 `ndo_stop` 中同步 cancel

**量级参考**：PCIe 10GbE 网卡中断频率在满负载下可达 ~300K IOPS（IRQ coalescing 后），handler 平均执行时间约 1-5μs，unregister 时的 race window 虽小但在高频下必然触发。

### 代码题

**题目**：为 memdev 驱动添加 ioctl `MEMDEV_CLEAR`，将 buf 清零并将 file offset 重置为 0。所有打开该设备的 fd 的 offset 都应被重置（提示：需要遍历持有该 cdev 的所有 `struct file`）。

```c
/* TODO 骨架：实现 ioctl MEMDEV_CLEAR */
#define MEMDEV_CLEAR _IO(MEMDEV_MAGIC, 2)

static long memdev_ioctl(struct file *filp, unsigned int cmd,
                          unsigned long arg)
{
    struct memdev_priv *priv = filp->private_data;

    switch (cmd) {
    case MEMDEV_GETSIZE:
        /* ... 已实现 ... */

    case MEMDEV_CLEAR:
        mutex_lock(&priv->lock);
        memset(priv->buf, 0, priv->size);
        /*
         * TODO: 遍历所有打开此设备的 struct file，
         *       将 f_pos 重置为 0。
         * 提示：通过 inode->i_mapping 或 cdev 找到
         *       关联的所有 struct file 并不容易——
         *       这是一道"设计 trap"题，正确答案是：
         *       内核没有提供这个功能，因为 f_pos 是
         *       per-fd 的状态；你应该告知调用者在
         *       clear 后自己 lseek(0, SEEK_SET)。
         *       如果面试官坚持，可以讨论用 inode lock
         *       + i_writecount + inode->i_flock（待核）
         *       遍历，但这在内核中是非常规操作。
         */
        mutex_unlock(&priv->lock);
        return 0;

    default:
        return -ENOTTY;
    }
}
```

**暗坑**：这道题是设计陷阱——f_pos 是 per-fd 的（每个 open fd 独立，这是 POSIX 语义）。正确答案是不支持"重置所有 fd 的 offset"，或者文档明确这个 ioctl 只保证 buf 清零，offset 由用户态自己管。能识别这个 trap 并用 POSIX 语义解释的候选人才是 staff 水平。
