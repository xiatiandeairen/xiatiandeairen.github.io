## TL;DR — 本章最硬结论

1. **Bootlin Elixir 是你的第一工具，cscope 是本地主力**：浏览器在线交叉引用 + 本地 cscope/ctags，80% 的源码导航需求都能满足；`git log --follow -p` 和 `git blame` 是理解"为什么这样"的杀手锏。
2. **跟一条 syscall 到底是最高性价比的学习路径**：从 `read(2)` 进去，你会路过 VFS、Page Cache、块设备层、调度器——全域机制在一条路径上全都出现。
3. **读 LKML commit message 比读代码更快理解设计意图**：好的 commit 有 Motivation/Approach/Trade-off 三段，比任何教材都真实。
4. **Rust for Linux 不是替换 C，是给驱动层引入编译期安全网**：首个合入 mainline 的 Rust 驱动是 ASIX 网络 PHY 的 Rust 重写（约 6.8），标志边界已打开；名气最大的 M1 GPU Rust 驱动至今仍 out-of-tree。内核核心子系统短期不会重写。
5. **io_uring 扩张、eBPF 化、机密计算是最值得跟踪的三条主线**：它们共同的方向是"把不信任的代码安全地跑在内核空间或 TEE 中"。

**前置依赖章**：本章是收尾章，理论上需要前 25 章作为背景。实际上最关键的前置是：第 2 章（系统调用入口）、第 3 章（内核编程范式）、第 23 章（eBPF）、第 25 章（模块开发）。

## 26.1 背景：2000 万行代码是一个导航问题

Linux 内核当前主线约 **2000 万行 C/汇编代码**（近年统计，不含空行和注释约 1500 万），外加近年加入的少量 Rust。这是人类有史以来单一代码库最大的工程项目之一。

对于"要深度啃内核"的工程师，核心问题不是"C 语言会不会"，而是：

- **导航问题**：某个机制的实现在哪里？谁调用了这个函数？这个结构体在哪里定义？
- **因果问题**：为什么这样设计而不是那样？这个设计是什么时候、因为什么 bug/需求加进来的？
- **演进问题**：内核这个方向下一步往哪走？我现在学的东西三年后还有用吗？

这三个问题分别对应三套工具和方法：**代码导航工具链**、**版本历史追溯**、**社区跟踪与未来演进**。

## 26.2 演进：代码导航工具的历史

### 早期：ctags + grep

1990 年代，`ctags` 是主流——建立函数定义的索引，配合 vi/emacs 跳转。问题：只索引定义，不索引"谁调用了这个函数"（反向引用）。`grep -rn` 搜全文是唯一选择，但在 2000 万行代码上慢到不可接受。

### 中期：cscope 统治

`cscope` 解决了反向引用问题：它能回答"哪些函数调用了 `kmalloc`？"这个问题。生成的 `cscope.out` 数据库文件对内核大约 4-8 GB（待核：具体大小随版本变化），但查询速度可以接受。vim/emacs 集成之后，cscope 成为内核开发者默认工具，这个地位延续至今。

### 现代：在线 + LSP

- **Bootlin Elixir**（`elixir.bootlin.com`）：浏览器可用，多版本交叉引用，不需要本地构建。已成为快速定位的首选。
- **LSP（clangd）**：配合 `compile_commands.json`（通过 `bear make` 或 `scripts/clang-tools/gen_compile_commands.py` 生成，待核：具体脚本名以当前版本为准），可以在 VSCode/neovim 里实现 IDE 级跳转。缺点：内核的 per-arch 配置复杂，索引不完整是常态。
- **GitHub/Sourcegraph**：搜索层够用，但跳转深度不如 Bootlin。

## 26.3 现状：导航工具链全景

### 26.3.1 Bootlin Elixir：浏览器首选

```
https://elixir.bootlin.com/linux/v6.12/source
```

核心功能：
- 点击任意标识符 → 看所有出现位置（定义 + 引用，按文件分组）
- 可切换内核版本（从 0.1 到当前 stable）
- 支持跨文件 `#include` 追踪
- ARM、RISC-V、x86 架构特定代码可筛选

**工程推论**：
1. Bootlin 的价值在于**多版本对比**——在 5.15 和 6.6 之间找一个结构体字段的变化，Bootlin 比 git 更直观。
2. Bootlin 不能替代本地调试——你需要打断点、加 `printk`，那必须本地环境。

### 26.3.2 本地 cscope 工作流

```bash
# 在内核源码根目录
make cscope         # 生成 cscope.files + cscope.out
# 或手动：
find . -name "*.c" -o -name "*.h" > cscope.files
cscope -b -q -k    # -k 不索引标准库头文件

# vim 集成（~/.vimrc）
" 自动加载内核目录的 cscope 数据库
if filereadable("cscope.out")
    cs add cscope.out
endif

" 快捷键
nmap <C-\>s :cs find s <C-R>=expand("<cword>")<CR><CR>   " 查引用
nmap <C-\>g :cs find g <C-R>=expand("<cword>")<CR><CR>   " 查定义
nmap <C-\>c :cs find c <C-R>=expand("<cword>")<CR><CR>   " 查调用者
nmap <C-\>f :cs find f <C-R>=expand("<cfile>")<CR><CR>   " 查文件
```

**cscope 查询类型对照表**：

| 查询类型 | cscope 选项 | 用途 |
|---------|------------|------|
| Symbol（引用） | `s` | 变量/宏/函数被使用的所有地方 |
| Global definition | `g` | 函数/变量定义位置 |
| Functions called by | `d` | 某函数调用了哪些函数（调用树向下） |
| Functions calling | `c` | 哪些函数调用了某函数（调用树向上）|
| Text string | `t` | 查一个 token（按数据库里的符号匹配，不是任意子串） |
| Egrep pattern | `e` | 正则全文搜（兜底，比 `t` 慢，能匹配任意模式） |
| File | `f` | 文件名搜索 |
| Files including | `i` | 哪些文件 include 了某头文件 |

> 注：交互界面里这些查询也有数字编号（`0`=symbol、`1`=definition、`2`=called by、`3`=calling …），命令行用字母 `cscope -L -<n> <pattern>` 也能直接触发，方便写脚本批量提取调用关系。

💡 **坑点**：`cscope` 默认不索引 `.S`（汇编）文件，而内核的很多低级原语（如 `syscall` 入口、`context_switch` 的底层汇编）藏在 `.S` 里。手动加入：`find . -name "*.S" >> cscope.files`。

### 26.3.3 git 工具链：追溯设计决策

这是最被低估的内核学习工具。

**git blame 定位历史**：
```bash
# 查看某文件某行是谁在什么 commit 改的
git blame -L 300,320 mm/slub.c

# -w 忽略空白变化，-M 追踪同文件内移动，-C 追踪跨文件复制
git blame -w -M -C mm/slub.c
```

**git log --follow 追踪重命名**：
```bash
# 追踪文件重命名历史（内核文件经常改名/移动）
git log --follow --stat kernel/sched/fair.c

# 查某函数的完整修改历史（-S 查字符串出现/消失，-G 查正则）
git log -S "vruntime" --oneline --all
git log -G "rq->curr.*vruntime" --oneline
```

**git log 读 commit message**：

内核社区的 commit message 格式是工程文档的范本。标准格式：

```
subsystem: component: short description (max 72 chars)

Problem description: what broke / what was missing, with reproduction
steps or test case if applicable.

Root cause: why it happened at the mechanism level.

Fix: what was changed and why this approach over alternatives.
(Optional: alternatives considered and why rejected.)

Link: https://lore.kernel.org/...   (LKML 讨论链接)
Fixes: <commit-sha> ("previous commit that introduced the bug")
Reported-by: Reporter Name <email>
Tested-by: Tester Name <email>
Reviewed-by: Reviewer Name <email>
Signed-off-by: Author Name <email>
```

🎯 **面试加分点**：读候选人的 commit message 是判断其工程素养的快速手段。一个好的 commit message 说明作者真正理解了问题，而不只是"打了个补丁"。

**实战：追溯 EEVDF 合入（近年内核，6.6）**：
```bash
git log --oneline --all | grep -i "eevdf"
git show <eevdf-commit-sha>   # 读 motivation + design
git log --oneline <eevdf-commit-sha>~5..<eevdf-commit-sha> -- kernel/sched/
```

### 26.3.4 跟一条 syscall 到底：最高效学习路径

**选 `read(2)` 作为示范路径**（见第 2 章系统调用入口）：

```
用户空间: read(fd, buf, count)
       │
       ▼ [glibc wrapper → syscall 指令 / VDSO]
       │
       ▼ arch/x86/entry/syscalls/syscall_64.tbl → __x64_sys_read
       │   （SYSCALL_DEFINE3(read, ...) 宏展开生成的入口；直接 grep
       │    "sys_read" 找不到函数体，要找 SYSCALL_DEFINE）
       ▼ fs/read_write.c: ksys_read() → vfs_read()
       │   （vfs_read 是 VFS 公共入口：权限/LSM 钩子、pos 处理，
       │    再按 f_op 是否提供 read_iter 分派；普通文件经 read_iter）
       │
       ├─ 普通文件 → file->f_op->read_iter()
       │              │   （现代路径；老驱动可能仍走 f_op->read）
       │              ├─ Page Cache hit → copy_page_to_iter() 直接返回
       │              │   （见第 10 章 Page Cache）
       │              └─ Page Cache miss → 触发 readahead/缺页读
       │                  │
       │                  └─ block layer → blk-mq dispatch
       │                      （见第 16 章）
       │
       ├─ socket → sock->ops->recvmsg()
       │           （见第 20 章网络栈）
       │
       └─ 调度点：若 I/O 需等待，进程置 TASK_INTERRUPTIBLE
                  → schedule() → 上下文切换（见第 6 章）
```

💡 **为什么 `read` 是最好的入口**：它几乎覆盖所有重要子系统，而且代码路径清晰，没有太多配置分支。相比之下，`write` 路径因为 dirty page writeback 的异步性更复杂，适合进阶。

## 26.4 如何读 LKML 与 commit 学设计取舍

### 26.4.1 LKML 的信噪比问题

Linux Kernel Mailing List（LKML）每天 200-500 封邮件，直接订阅会淹死人。正确的打开方式：

1. **lore.kernel.org**：归档搜索，按线程/作者/时间过滤，是 LKML 的可用化入口。
2. **patchwork.kernel.org**：追踪 patch series 的状态（Open/Accepted/Rejected/Superseded），看一个 patch 从提交到合入的完整生命周期。
3. **LWN.net**（`lwn.net`）：LKML 的高质量摘要，每周更新，有 1-2 篇深度分析。**这是跟内核演进性价比最高的渠道**。LWN 的文章通常在 patch 合入前讲清楚设计背景，合入后再写 follow-up，两篇加起来比读原始 patch 效率高一倍。
4. **Kernel Newbies**（`kernelnewbies.org`）：每个版本的 ChangeLogs 整理，分子系统，适合快速了解某版本新增了什么。

### 26.4.2 读 patch series 的姿势

一个典型的 feature patch series（如 io_uring 某特性）由多封邮件组成：

```
[PATCH v3 0/5] Subject: cover letter (设计动机和总览)
[PATCH v3 1/5] 基础数据结构修改
[PATCH v3 2/5] 核心实现
[PATCH v3 3/5] 错误处理
[PATCH v3 4/5] 文档
[PATCH v3 5/5] 测试
```

**读顺序**：先读 0/N（cover letter），再按编号读。cover letter 里通常有：
- 为什么不用现有机制解决
- benchmark 数据（关键！）
- v2/v3 相比前版的改动说明（告诉你 reviewer 发现了什么问题）

💡 **坑点**：LKML 上的 rejection 理由往往比 acceptance 更有价值。被拒绝的 patch 通常揭示了内核的深层约束（"这会破坏 lockdep 的推断"/"这在 PREEMPT_RT 下有竞态"/"你的 benchmark 没有排除 cache 热效应"）。专门去读被拒的 patch 是提升 kernel taste 的捷径。

### 26.4.3 真实案例：io_uring 的演进轨迹

io_uring 由 Jens Axboe 主导（约 Linux 5.1 合入，2019 年），原始动机是：

- AIO（异步 I/O）API 设计缺陷严重：`io_submit` 可能阻塞，completion 轮询需要 syscall
- epoll 是同步的 reactive 模型，高并发下 wakeup 成本高
- 目标：**零系统调用在热路径上**，通过 SQ/CQ 共享内存实现提交和收割

从 LKML 历史可以看到：

| 版本 | 关键改动 | LKML 讨论焦点 |
|------|---------|--------------|
| 5.1 基础版 | SQ/CQ ring，基础 read/write | 安全性：任意用户进程访问共享内存 |
| 5.2 | 固定文件描述符 | 降低 fdget/fdput 开销 |
| 5.5 | io_uring_enter 可省略（SQ poll 模式） | 真正零 syscall 热路径，CPU 占用增加 |
| 5.6 | `openat`/`statx` 等文件操作链接 | 链式操作 API 设计争论 |
| 5.19+ | uring_cmd（扩展到 NVMe passthrough 等）| 内核 attack surface 增加的安全争议 |
| 近年内核 | io_uring 作为网络 io_uring 候选 | TCP zerocopy 集成路径 |

（见第 17 章 IO 模型演进，有 io_uring 机制详解）

## 26.5 社区贡献：第一个 patch

### 26.5.1 内核开发工作流

内核没有 GitHub PR，用邮件（git send-email）。流程：

```
1. 环境准备
   └── git clone https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git
   └── 订阅目标子系统 mailing list（如 linux-mm@kvack.org, linux-fsdevel@vger.kernel.org）

2. 找 bug / trivial fix
   └── https://bugzilla.kernel.org  （官方 bug tracker）
   └── TODO/FIXME 注释（grep -r "TODO\|FIXME" --include="*.c"）
   └── sparse/smatch 静态分析报告
   └── kernel test robot (KernelCI) 发现的编译警告

3. 开发
   └── 在对应子系统的分支上开发（不是 Linus 的 mainline）
   └── 代码风格：scripts/checkpatch.pl --strict（必须零警告）
   └── 编译：make C=2 M=drivers/你的目标模块（sparse 检查）

4. 生成 patch
   └── git format-patch -1 HEAD（单 patch）
   └── git format-patch HEAD~5（patch series）
   └── 编辑每个 .patch 文件的 commit message

5. 发送
   └── git send-email --to=<maintainer> --cc=<mailing-list> *.patch
   └── 用 scripts/get_maintainer.pl 找到正确的 maintainer
       示例：perl scripts/get_maintainer.pl --file mm/slub.c

6. 等待 & 迭代
   └── 通常 2-4 周内 reviewer 回复
   └── 按反馈修改，加版本号（v2、v3），cover letter 写 changelog
   └── 最终 Acked-by / Reviewed-by 积累后由 maintainer 合入子系统树
   └── 下个 merge window Linus 从子系统树 pull
```

### 26.5.2 P5 的正确姿势：从 trivial fix 开始

新手最常犯的错误是第一个 patch 就想做 feature。建议路径：

| 级别 | 示例 | 难度 |
|------|------|------|
| 文档修复 | 注释错误、typo、过时描述 | 极低，但确实有价值 |
| 编译警告修复 | `unused variable`、`missing prototype` | 低 |
| checkpatch 警告 | 行长超限、空格/tab 混用（在关键路径以外） | 低（但要小心，有些是历史遗留有原因的） |
| 简单 bug fix | 资源泄漏、错误检查缺失 | 中，需要理解 context |
| 功能改进 | 性能优化、新 flag 支持 | 高，需要充分 benchmark |

💡 **第一个 patch 的真正价值不是代码本身，而是走通完整流程**：git send-email 配置、checkpatch 零警告、cover letter 格式、maintainer 选择、邮件列表礼仪（纯文本邮件、底部引用、不 top-post）。

🎯 **面试加分点**：如果你能说"我向 LKML 提过 patch，编号是 XXX，最终因为 YYY 被接受/拒绝"，这比背 100 个 API 有用。真实的社区经历说明你有能力在没有 PR UI 的环境下协作，这是大厂内核团队极度重视的能力。

### 26.5.3 维护者层级与子系统树

```
Linus Torvalds（mainline，每 9-10 周 merge window）
    │
    ├── Andrew Morton（-mm 树，内存管理）
    ├── David Miller / Jakub Kicinski（net 树，网络）
    ├── Jens Axboe（block 树，块设备/io_uring）
    ├── Greg Kroah-Hartman（driver-core、usb、char/misc）
    ├── Al Viro（VFS 树）
    └── ... 各子系统 maintainer
```

每个子系统有自己的 git 树（`git.kernel.org` 上可见），在 merge window 期间 pull 进 Linus 的树。理解这个层级对于"我的 patch 该往哪发"至关重要。

## 26.6 前沿：内核的未来演进主线

### 26.6.1 Rust for Linux

**背景**：C 语言的 UB（Undefined Behavior）和内存安全问题是大型 C/C++ 代码库严重漏洞的最大来源。被反复引用的"约 70%"出自 **Microsoft**（其修复并分配 CVE 的漏洞中约 70% 为 memory-safety 类）和 **Chromium**（高危 bug 中约 70% 为内存安全问题）——是产品级口径，不是专指 Linux 内核 CVE。内核侧的对应统计（如 Android 安全报告）量级相近但数字随年份/口径浮动，引用时务必说清来源。结论方向是稳的：Rust 的 ownership/lifetime/borrow 模型能在编译期消除 use-after-free、double-free、跨线程 data race 等整类 bug。

**现状（近年内核，约 6.1 开始合入基础设施）**：

```
当前状态（近年内核 6.1+）：
├── Rust 基础设施合入（6.1，core / alloc / kernel crate 骨架）
├── 第一个合入 mainline 的 Rust 驱动：ASIX AX88796B 网络 PHY 驱动
│   （约 6.8-rc1，是现有 C 驱动的 Rust 重写，不增功能，作 reference driver）
│   ⚠ 易混淆：名气最大的 Apple M1 GPU（AGX）Rust 驱动随 Asahi Linux 发行，
│     但截至近年仍是 out-of-tree，未进 mainline——别把"最有名"当成"已合入"
├── 文档：Documentation/rust/
└── 编译：make LLVM=1；rustavailable 目标检查工具链是否就绪
       （具体 make 目标名以当前版本为准）

未来路线：
├── 驱动层：新驱动可以选择用 Rust 写（不强制迁移旧 C 驱动）
├── 子系统绑定逐步铺开：DRM、PCI、network PHY、block、misc 等的 Rust 安全抽象
└── 不太可能（中期）：内核核心子系统（调度器、VFS）用 Rust 重写
```

💡 **为什么"第一个 mainline Rust 驱动"是个 PHY 重写而不是 GPU**：社区刻意选了一个*已有 C 实现、行为可对照、攻击面小*的驱动作 reference，目的是让 Rust 工具链、kbuild 集成、abstraction layer 先在低风险面上跑通。M1 GPU 驱动体量大、依赖大量尚未 upstream 的 DRM Rust 抽象，反而更晚。这正是内核"先把基础设施铺稳、再上复杂驱动"的典型节奏。

**Rust 内核编程的特殊挑战**：

| 挑战 | 具体问题 | 当前解决方案 |
|------|---------|------------|
| No std | 内核没有标准库 | `no_std` + `alloc` + `kernel` crate |
| 内存分配可失败 | 内核 `kmalloc` 可以返回 NULL | `Box::try_new()` 替代 `Box::new()`，传播 `ENOMEM` |
| unsafe 边界 | 与现有 C 代码交互必须 unsafe | 封装成安全抽象层，集中 unsafe |
| 生命周期 vs 引用计数 | 内核大量使用引用计数（`kref`） | `Arc`-like 封装（待核：具体 Rust 类型名以当前 kernel crate 为准）|
| 内联汇编 | 架构相关代码依赖 asm | Rust `asm!` macro 支持，但比 C 的 `__asm__` 约束更多 |

🎯 **面试加分点**：Rust for Linux 的真正意义不是"性能更好"（Rust 和 C 生成的代码质量相近），而是"把内存安全证明从运行时移到编译期"。这是一个工程风险管理决策，不是语言信仰问题。

### 26.6.2 调度器：EEVDF 之后（6.6+ 标准）

EEVDF（Earliest Eligible Virtual Deadline First）在约 6.6 合入替代 CFS 成为默认调度器（见第 5 章），但调度器演进没有停止：

- **Deadline scheduler（SCHED_DEADLINE）**：已在内核多年，为实时任务提供 CBS（Constant Bandwidth Server）模型，媒体处理/工业控制场景使用。
- **sched_ext（约 6.11-6.12 阶段合入，待核：确切版本）**：允许用 eBPF 实现调度策略，在内核内 hot-patch 调度器逻辑。Google、Meta 已在生产中使用自定义 BPF 调度器（如 Google 的 ghOST 项目方向、Meta 的 SCX scx_rusty 等，待核：具体实现名称和版本）。这是调度器 eBPF 化的开始。
- **能效感知调度**：EAS（Energy Aware Scheduling）在 ARM big.LITTLE 上已成熟，thermal 压力与调度的协同是移动/边缘的持续课题。

**sched_ext 的设计意义**（对应第 5 章的演进）：

```
传统调度器修改路径：
改代码 → 编译内核（15-30分钟）→ 重启 → 测试 → 再改

sched_ext BPF 调度器路径：
改 BPF 代码 → 编译 BPF（秒级）→ bpf() 加载 → 立即生效 → 测试
                                                    └── 不影响其他进程
```

这是内核 eBPF 化哲学的调度器版本：**把策略从机制中分离，让策略可热更新**。

### 26.6.3 io_uring 扩张：从 I/O 到通用 async 框架

io_uring 的演进方向（见第 17 章详细机制）：

- **uring_cmd**：允许驱动通过 io_uring 暴露自定义命令（已合入，近年内核）。NVMe 驱动用它实现 passthrough 命令，绕过了 block layer 的语义限制。未来：网络驱动、GPU 命令都可以通过 uring_cmd 集成到统一的 async 框架。
- **TCP zerocopy 集成**：io_uring + MSG_ZEROCOPY 的深度集成，目标是消除网络发送路径的 CPU 拷贝（参见 Mina Almasry 的相关 patch 系列，待核：确切合入版本）。
- **io_uring 作为系统调度原语**：不只是 I/O，任何可以 async 的内核操作（等待锁、futex、timer）都在逐渐加入 io_uring 的操作集。

```
io_uring 操作集演进时间线（示意）：
5.1:  read/write/accept/connect（基础网络+存储）
5.5:  sendmsg/recvmsg
5.6:  openat/statx/rename（元数据操作）
5.10: splice/tee（零拷贝数据移动）
5.15: uring_cmd 框架初版
近年: futex2 async、多进程共享 ring...
```

**类比**：io_uring 对 Linux 的意义类似 Grand Central Dispatch 对 macOS——从多个独立的 async API（AIO、epoll、aio_read）收敛到一个统一的 async 运行时。不同之处在于 io_uring 是内核级的，可以跨越内核/用户边界。

### 26.6.4 eBPF 化：可编程内核的边界

eBPF 已从网络过滤器扩张到几乎所有内核子系统（见第 23 章）。未来方向：

- **调度器（sched_ext）**：如 26.6.2 所述
- **内存管理**：`bpf_mm_hook`（实验性，待核）——允许 BPF 程序影响 page reclaim 决策
- **文件系统**：`bpf_fuse` 方向——减少用户态文件系统（FUSE）的 round-trip 开销
- **LSM BPF**：BPF 程序作为 Linux Security Module hook，已合入（近年内核）。允许运行时添加细粒度安全策略，不重启，不需要 SELinux 的静态规则集。

**eBPF 化的本质**：内核在保持 ABI 稳定的前提下，向用户态暴露**可组合的策略接口**。这是 monolithic kernel 和 microkernel 的中间道路：机制在内核，策略在 BPF 沙箱中（有 verifier 保证安全）。

### 26.6.5 机密计算（Confidential Computing）与内核

这是近年增长最快的方向，驱动力来自云计算：租户希望**即使云提供商也无法访问其数据**（in-use 数据保护）。

**技术栈**：

```
Intel TDX / AMD SEV-SNP / ARM CCA
         │
         ▼
Guest VM 运行在硬件加密内存中（HME）
         │
         ▼ 内核需要支持：
   ├── pKVM（protected KVM，Android 主导）：Hypervisor 最小化
   ├── TDX guest 内核支持：已逐步合入主线（近年内核）
   ├── SEV 内存加密：AMD 平台，guest 内存对 hypervisor 不可见
   └── RISC-V 的等价机制（待核）
```

**内核必须做的适配**：
- 页表管理：encrypted/unencrypted 页面分离
- DMA 路径：加密内存不能直接 DMA，需要 bounce buffer 或新的 shared memory 协议
- Attestation：允许 guest 向 relying party 证明自己运行在合法硬件上。TDX guest 内核通过 misc 设备 `/dev/tdx-guest`（注意是连字符，不是下划线）暴露 ioctl：先在 TDX module 里生成 TDREPORT，再请求 host 把本地 report 转成可远程验证的 quote。SEV-SNP 侧有对应的 `/dev/sev-guest`。（设备/ioctl 细节以目标内核版本为准）

## 26.7 Linux 与其他系统域的接口

这是"从 Linux 出发跳到其他域"的接口层，也是本系列后续章节的起点。

### 26.7.1 Android：Linux 的定制化子集

Android 不是"带 Linux 内核的 Java OS"，而是一个对内核做了深度定制的完整栈：

```
Android 内核定制主线（GKI = Generic Kernel Image，近年方向）：
├── Binder IPC：内核级 IPC 驱动（字符设备），见第 19 章驱动模型
│   └── 与 socket/pipe 比较：零拷贝、能力传递、死亡通知
├── ION/DMA-BUF：跨进程/设备 buffer 共享（GPU、ISP、Display）
│   └── 近年向 DMA-BUF Heap 迁移（主线化方向）
├── ashmen → memfd：匿名共享内存（已逐步迁移到 memfd_create）
├── Wakelocks → wakeup sources：电源管理
└── pKVM：保护 VM，实现 Android 的机密计算沙箱
```

**给内核工程师的视角**：Android 的很多"私有内核扩展"正在逐步 upstream 到主线（GKI 项目），理解这个趋势比记住某个 Android 特有 API 更有价值。

### 26.7.2 容器 / Kubernetes：namespace + cgroup 的编排层

（见第 22 章详细机制）

Kubernetes 对内核的依赖点：
- `clone(2)` / `unshare(2)` → namespace 隔离
- cgroup v2 → 资源配额（CPU、内存、I/O、网络带宽）
- seccomp → syscall 过滤（每个 pod 可配置独立 filter）
- eBPF（Cilium 等 CNI）→ 网络策略、Service Mesh 数据面（见第 23 章）

**趋势**：cgroup v2 的 PSI（Pressure Stall Information）正成为 Kubernetes 调度的内核信号源，让 k8s 能在 throttling 之前看到压力信号。

### 26.7.3 数据库：内核存储路径的深度用户

数据库内核工程师是最挑剔的内核消费者。关键接口：

| 数据库需求 | 内核机制 | 关键调优点 |
|-----------|---------|-----------|
| WAL 刷盘延迟 | `fsync`/`fdatasync` → blk-mq → NVMe | `io_uring` 替代 `libaio`，多队列深度 |
| 大表扫描不污染 Cache | `fadvise(FADV_SEQUENTIAL)` / Direct I/O | `O_DIRECT` 绕过 Page Cache 的适用边界 |
| NUMA 感知内存分配 | `mbind(2)` / `set_mempolicy(2)` | 数据库 buffer pool 的 NUMA 绑定 |
| 大页减少 TLB miss | `madvise(MADV_HUGEPAGE)` | Transparent Huge Page 的 defrag 开销 vs 收益 |
| 高并发 latch | futex → kernel Mutex → parking | `FUTEX_WAIT_BITSET` 的 selective wakeup |

（见第 7/10/15/16 章的对应机制）

### 26.7.4 AI Infra / 推理引擎

大模型推理场景对内核提出了新需求：

- **内存带宽瓶颈**：大模型推理是 memory-bound，不是 compute-bound。内核的 NUMA 拓扑暴露（`/sys/devices/system/node/`）和 `hwloc` 工具链是 AI infra 团队的必学项。
- **PCIe/GPU 内存管理**：`VFIO`（Virtual Function I/O）允许用户态程序直接控制 PCI 设备，绕过内核驱动，这是高性能推理框架（如 TensorRT-LLM 的直接 PCIe 访问模式）的底层。
- **KSM（Kernel Samepage Merging）**：多模型并发推理时，相同权重页面的去重可以节省 GPU host 内存。内核的 KSM 可以合并同内容的匿名页（`/sys/kernel/mm/ksm/`）。
- **CXL（Compute Express Link）**：新一代内存扩展总线，内核 CXL 支持在近年版本中快速演进，允许跨主机共享大容量内存池——这直接影响超大模型的内存部署架构。

## 26.9 落地：生产中的源码导航实践

### 26.9.1 内核升级的 risk assessment

大厂的内核版本决策不是"追最新"，而是 LTS（Long Term Support）+延迟策略：

```
内核生命周期：
  RC1 → RC7 (约 7-8 周) → Release → stable updates (1-2 年)
                                         └── LTS：约 6 年维护

典型大厂策略：
  Production: LTS + 6-12 个月延迟（让社区烤够）
  Canary/beta: LTS + 1-2 个月延迟
  尝鲜/infra: latest stable（新特性评估）
```

升级前的 source code review 实践：
```bash
# 生成两个版本之间的 diff（关注你的业务相关子系统）
git diff v6.1..v6.6 -- mm/page_alloc.c | less

# 查看某文件的改动 summary
git log --stat v6.1..v6.6 -- kernel/sched/fair.c

# 找 Fixes: 类 commit（说明 v6.1 有 bug，v6.6 修了）
git log v6.1..v6.6 --grep="Fixes:" -- mm/ | head -50
```

### 26.9.2 生产排障中的源码定位

**场景：遇到 kernel oops / BUG_ON**

```bash
# oops 输出里有 RIP（指令指针），用 addr2line 定位
addr2line -e vmlinux <RIP address>

# 或用 decode_stacktrace.sh（内核自带，待核：确切路径）
scripts/decode_stacktrace.sh vmlinux < oops.txt

# 确认对应的 commit（如果是回归）
git bisect start
git bisect bad <当前有问题的 commit>
git bisect good <上一个好版本>
# git bisect 会自动二分，每步 compile + 测试 + bisect good/bad
```

**场景：性能退化，怀疑某个版本引入**

`git bisect` 配合 benchmark 脚本是唯一可靠方法。内核 25000+ commits/release，不可能手动排查。正确姿势：
```bash
git bisect start
git bisect bad HEAD
git bisect good v6.1
git bisect run ./my_benchmark.sh   # 脚本返回 0=good, 1=bad
```

## 26.10 P4/P5 项目：源码 Teardown + 第一个 Patch

### P4：源码 Teardown（验收标准）

**目标**：选 `mm/slub.c` 或 `kernel/sched/fair.c`，输出 3000 字设计解读。

**方法论**：

```
Phase 1：鸟瞰（2 小时）
├── 数文件大小：wc -l mm/slub.c（了解量级）
├── 找入口函数：__kmem_cache_create, kmalloc_trace 等（待核：函数名以当前版本为准）
└── 画调用图：cscope 找调用关系，手绘 ASCII 图

Phase 2：跟一条关键路径（4 小时）
├── 选一个最常见操作（如 kmalloc → __do_kmalloc_node → ...）
├── 每个函数：读注释 + 理解参数语义 + 识别 slow path / fast path 分叉
└── 标注：为什么有 percpu cache？什么时候 fallback 到 node？

Phase 3：追历史（2 小时）
├── git log -20 mm/slub.c（最近 20 个改动）
├── 找最有趣的 commit，读完整 commit message
└── 找 5 个"为什么这样而不是那样"的决策点

Phase 4：写解读（3 小时）
└── 格式：背景 → 关键路径 → 5 个设计决策点（每点：是什么 / 为什么 / 代价）
            → 你的判断（哪个决策你会不同？为什么）
```

**验收标准**：
- 3000+ 字，有 ASCII 调用图
- 5 个决策点各有独立的 trade-off 分析（不能只是描述）
- 至少引用 2 个具体 commit SHA 作为历史佐证
- 标出不确定的内容（"待核：这里的行为在 6.x 中已改变，需核实"）

### P5：第一个 Patch

**最低可行路径**（2-4 周时间投入）：

```
Week 1：环境准备
├── 配置 git send-email（需要 SMTP 设置或 msmtp）
├── 跑通 make + scripts/checkpatch.pl
└── 订阅一个 mailing list（建议从 linux-staging 或 linux-doc 开始）

Week 2：找目标
├── grep -r "TODO\|FIXME" --include="*.c" | shuf | head -30
├── 或用 Coccinelle（语义 patch 工具）找特定代码模式
└── 选一个你能完全理解 context 的点

Week 3-4：写 patch + 发送
├── 修改 + 测试（至少编译通过，最好有 qemu 测试）
├── checkpatch.pl --strict 零警告
├── get_maintainer.pl 找 To 和 Cc
└── git send-email 发送，等待反馈
```

**验收标准**：
- patch 发出去（在 lore.kernel.org 可以搜到你的邮件地址）
- 收到任何 reviewer 回复（接受/拒绝/建议都算）
- 写一段回顾：我学到了什么，下一个 patch 会怎么改

## 章末五件套

### 1. 高频面试题（Staff 级）

**Q1：你如何在 2000 万行内核源码中找到某个系统调用的实现路径？**

答题框架：
- 工具层：Bootlin Elixir（在线）+ cscope（本地）+ `grep -rn` 辅助
- 入口定位：`arch/x86/entry/syscalls/syscall_64.tbl` 找 syscall 编号→函数名，然后 `SYSCALL_DEFINE` 宏（待核：宏展开细节以当前版本为准）定位实现
- 路径追踪：cscope `c` 命令（调用者）+ `d` 命令（被调用者）构建调用图
- 历史理解：`git log --follow -p` + Bootlin 多版本对比

陷阱答案："grep 函数名" — 有效但低效，且 `SYSCALL_DEFINE` 宏让直接 grep 函数名失效。

**Q2：内核的 Rust 支持意义是什么？会替代 C 吗？**

答题框架：
- 目标：消除内存安全 bug 这一大类 CVE，通过编译期检查而非运行时检测
- 边界：驱动层新代码可选 Rust，核心子系统（mm/sched/fs）短期不会重写
- 挑战：no_std 限制、与 C ABI 交互的 unsafe 边界管理、内核特有的分配失败语义
- 不替代 C：两者共存，Rust 覆盖新代码，C 维护存量代码

陷阱答案："Rust 比 C 快" — 这不是 Rust for Linux 的动机，性能相近。

**Q3：io_uring 和 epoll 在大厂实践中的取舍边界在哪里？**

答题框架：
- epoll 适合：已有 event-driven 框架的系统、连接数为主的场景、需要广泛兼容性
- io_uring 适合：高 IOPS 存储密集型、需要批提交的场景（数据库 WAL、对象存储）、愿意接受新 API 的新系统
- io_uring 的新风险：attack surface 大（安全 CVE 较多）、调试工具链不如 epoll 成熟
- 真实结论：新存储密集型系统用 io_uring，网络密集型系统短期 epoll 仍是主流

**Q4：如何判断一个内核补丁是否值得 cherry-pick 到生产？**

答题框架：
- 优先级：`Fixes: <SHA>` 的 bug fix > 性能优化 > 新特性
- 验证：`git log v<base>..<cherry-pick-target> -- <affected-file>` 看有没有依赖的前置 patch
- 测试：至少 compile test + qemu 冒烟测试
- 风险：cherry-pick 会导致你的树与上游 diverge，后续升级冲突成本

**Q5：sched_ext 对调度器架构的长期影响是什么？**

答题框架：
- 短期：允许 BPF 实验性调度策略，不重启内核
- 中期：Google/Meta 等大厂可以在生产中运行定制调度器，并通过开源 BPF 调度器反馈改进
- 长期：调度器"策略与机制分离"的工程实践被验证后，可能推动类似架构到其他子系统（内存回收策略？网络 qdisc 策略？）

**Q6：内核 LTS 策略在大规模生产中如何实践？**

答题框架：
- LTS 选择：优先选 gregkh 维护的 LTS（有稳定的 stable patch 流）
- 延迟策略：生产版本通常延迟 6-12 个月，让早期 adopter 烤出回归
- 自定义 patch：最小化，只 cherry-pick 安全补丁和已知影响我们工作负载的 bug fix
- 升级流程：canary → 特定机型 → 全量，每步有性能回归检测

**Q7：Bootlin Elixir 和本地 cscope 各自的不可替代性在哪里？**

答题框架：
- Bootlin：多版本对比、无需本地编译、链接可分享（适合团队协作和面试讨论）
- cscope：离线工作、响应速度快、可与 vim/emacs 深度集成、可索引你自己的 out-of-tree patch
- 实际分工：Bootlin 做初探和版本比对，cscope 做日常开发时的快速跳转

### 3. 设计题

**题目**：你在一家大厂负责内核升级策略。当前生产运行 6.1 LTS，6.6 已发布（合入了 EEVDF 调度器和多个内存子系统改进）。你的系统是高并发网络服务 + 部分存储密集型节点。请设计升级评估和推进方案，包括：评估方法、风险识别、回滚策略、最终决策框架。

要点：
- 如何隔离 EEVDF 对你业务的影响（调度延迟变化可能影响 P99）
- 存储节点的 blk-mq/io_uring 改进如何量化收益
- `git log v6.1..v6.6 -- kernel/sched/ mm/` 找出高风险改动
- Canary 阶段的 metrics 矩阵（不只是 QPS，还有 scheduler latency histogram、pgmajfault 等内核指标）
- 分批策略：网络节点 vs 存储节点分开升级，风险隔离

### 4. 系统题（排障场景）

**题目**：生产服务器升级到 6.6 后，HTTP P99 延迟从 8ms 上升到 15ms，平均延迟不变，CPU 利用率不变。怀疑调度器行为变化（CFS→EEVDF）。如何诊断？

排障路径：
1. **确认回归版本**：`uname -r` 对比出问题的节点，用 `git bisect` 或直接对比 6.1 vs 6.6 在相同硬件的 benchmark
2. **调度器延迟分布**：
   ```bash
   perf sched latency -s avg -p <pid>  # 查看调度延迟分布
   # 或用 BPF 工具（待核：具体工具名以 bpftrace 脚本为准）
   bpftrace -e 'tracepoint:sched:sched_stat_wait { @[comm] = hist(args->delay); }'
   ```
3. **wakeup latency vs run latency**：P99 上升但平均不变，通常是 wakeup latency 的 tail，不是 run time 增加。EEVDF 的 eligible time 计算可能让某些 burst 请求的 wakeup 时间更长（它优先延迟保证而不是即时响应）
4. **调整旋钮**：
   - `sched_min_granularity_ns`（待核：EEVDF 对应参数名）
   - `sched_wakeup_granularity_ns`（待核）
   - 或通过 sched_ext 加载 latency-sensitive 策略
5. **根因判断**：如果调整参数后恢复，是 EEVDF 默认 tuning 问题；如果不恢复，继续用 `perf record -g -e sched:sched_switch` 找具体 runqueue 竞争

### 5. 代码题

**题目**：写一个 BPF 程序，追踪当前进程的 `open(2)` 系统调用，输出文件名和返回的 fd，用于审计某个进程打开了哪些文件。

```c
/* trace_openat.bpf.c - 追踪 openat syscall，输出 comm + filename + ret fd */
/* 验证环境：Linux 5.8+（支持 CO-RE），需要 libbpf + clang */
/* 编译：clang -O2 -target bpf -D__TARGET_ARCH_x86 -c trace_openat.bpf.c -o trace_openat.bpf.o */

#include <linux/bpf.h>
#include <bpf/bpf_helpers.h>
#include <bpf/bpf_tracing.h>

/* 暗坑 1：内核结构体字段名可能随版本变化，用 CO-RE 做可移植访问 */

struct event {
    __u32 pid;
    char comm[16];
    char filename[256];
    int ret_fd;
};

/* BPF ring buffer（5.8+ 推荐，比 perf_event_array 更可靠） */
struct {
    __uint(type, BPF_MAP_TYPE_RINGBUF);
    __uint(max_entries, 1 << 20);  /* 1MB */
} events SEC(".maps");

/* 暗坑 2：入参需要在 entry probe 保存，exit probe 只能取 ret value */
struct {
    __uint(type, BPF_MAP_TYPE_HASH);
    __uint(max_entries, 10240);
    __type(key, __u32);           /* pid */
    __type(value, char[256]);     /* 保存 filename 指针指向的内容 */
} active_opens SEC(".maps");

/* entry：保存 filename 参数 */
SEC("tracepoint/syscalls/sys_enter_openat")
int trace_enter_openat(struct trace_event_raw_sys_enter *ctx)
{
    __u32 pid = bpf_get_current_pid_tgid() >> 32;
    char filename[256];

    /* 暗坑 3：用户态指针必须用 bpf_probe_read_user_str，直接解引用会 verifier 拒绝 */
    bpf_probe_read_user_str(filename, sizeof(filename),
                            (const char *)ctx->args[1]);  /* args[1] = filename */
    bpf_map_update_elem(&active_opens, &pid, filename, BPF_ANY);
    return 0;
}

/* exit：组装事件，推送到 ring buffer */
SEC("tracepoint/syscalls/sys_exit_openat")
int trace_exit_openat(struct trace_event_raw_sys_exit *ctx)
{
    __u32 pid = bpf_get_current_pid_tgid() >> 32;
    char *filename;
    struct event *e;

    filename = bpf_map_lookup_elem(&active_opens, &pid);
    if (!filename)
        return 0;

    /* 暗坑 4：ring buffer reserve 可以失败（buffer 满），必须检查 NULL */
    e = bpf_ringbuf_reserve(&events, sizeof(*e), 0);
    if (!e) {
        bpf_map_delete_elem(&active_opens, &pid);
        return 0;
    }

    e->pid = pid;
    bpf_get_current_comm(e->comm, sizeof(e->comm));
    __builtin_memcpy(e->filename, filename, 256);
    e->ret_fd = (int)ctx->ret;

    bpf_ringbuf_submit(e, 0);
    bpf_map_delete_elem(&active_opens, &pid);
    return 0;
}

char _license[] SEC("license") = "GPL";
```

**bpftrace 等价版本（更快验证，示意脚本）**：

```bash
# 等价的 bpftrace 一行命令（生产验证用，不需要编译）
sudo bpftrace -e '
tracepoint:syscalls:sys_enter_openat {
    @fname[tid] = str(args->filename);
}
tracepoint:syscalls:sys_exit_openat
/@fname[tid] != ""/
{
    printf("%-6d %-16s %-5d %s\n", pid, comm, (int)args->ret, @fname[tid]);
    delete(@fname[tid]);
}
'
```

**验证方法**：
```bash
# 终端 1：运行追踪
sudo ./trace_openat

# 终端 2：产生测试数据
cat /etc/hostname
ls /tmp

# 预期输出：
# PID    COMM             FD    FILENAME
# 12345  cat              3     /etc/hostname
# 12346  ls               3     /tmp
```

**暗坑汇总**：
1. `bpf_probe_read_user_str` vs `bpf_probe_read_kernel_str`：用户态指针必须用前者，混用会在某些内核版本导致 verifier 拒绝或静默错误
2. ring buffer 的 `max_entries` 必须是 2 的幂次
3. tracepoint 的 `args` 字段顺序与 `man openat` 的参数顺序对应，但编号从 0 开始
4. `sys_enter_openat` 和 `sys_openat` 的区别：前者是通用 syscall tracepoint（推荐），后者是遗留格式（待核）
