> **本章定位**:编译器后端的"最后一公里"。前面四章把源码碾成了 SSA 形式的、目标无关的 IR;本章负责把这个 IR 落到一台具体机器上——选指令(instruction selection)、分配寄存器(register allocation)、把抽象的"无限寄存器"塞进真实机器那十几个物理寄存器里。这是整个编译器里**理论最优雅(图着色 = NP 完全)而工程最肮脏(ABI、调用约定、move 解析、寄存器别名)**的一段。
>
> **与 V8 绑定章节的区别**:这里讲的是"编译器域"——从 IR 到目标代码的通用机制,以 Cranelift / regalloc2(Wasmtime 的后端,生产级、SSA-based backtracking)和经典 linear scan 论文为源码锚点。不涉及 V8 的 JS 特定 IC/隐藏类绑定。

---

## TL;DR(先给结论)

1. **代码生成 = 指令选择 + 指令调度 + 寄存器分配**。三者经典上分三趟(pass),现代后端(Cranelift)把指令选择做成 **ISLE 模式匹配 + 反向 maximal-munch 遍历**,寄存器分配交给独立的 **regalloc2** 库。
2. **指令选择的本质是树/DAG 覆盖问题**:用一组带 cost 的"瓦片"(指令模式)去覆盖表达式树,最小化总 cost。两大流派:`maximal munch`(贪心,每步咬最大的瓦片)和 `BURS / 动态规划`(全局最优)。
3. **寄存器分配的本质是图着色**:live range 是节点,同时活跃就连边,寄存器是颜色。Chaitin 1981 证明它 **NP 完全**;但在 SSA 形式上 interference graph 是 **chordal(弦图)**,可线性时间最优着色——这是 SSA 给后端的最大红利。
4. **两条工程路线**:`graph coloring`(质量高、慢,GCC/LLVM 老 RA)vs `linear scan`(快、质量略低,早期 JIT)。现代生产 JIT(SpiderMonkey Ion、Cranelift regalloc2)走的是 **second-chance / backtracking + live range splitting**——介于两者之间,既快又能逼近图着色质量。
5. **spilling(溢出)不是失败,是常态**。寄存器不够时把某个值踢到栈上。**踢谁**是核心启发式:linear scan 踢"结束点最远"的(Belady-like);backtracking 用 **spill weight**(用点权重之和)决定优先级,低权重的被驱逐(evict)。
6. **最容易被低估的脏活:move 解析**。分配完寄存器后,跨基本块边界、live range split 点上会产生一堆"并行 move",其中可能有**环**(swap r1↔r2),必须用 scratch 寄存器拆环。这是 out-of-SSA + RA 之后必做的一步,错了就是静默 miscompile。

## 1. 设计考古:为什么后端长这样

### 1.1 三个子问题的由来

把 IR 变成机器码,逻辑上要回答三个独立的问题:

| 子问题 | 问什么 | 经典名字 |
|---|---|---|
| 指令选择 | 用哪些机器指令实现这段 IR? | instruction selection |
| 指令调度 | 这些指令以什么顺序发射,才能填满流水线、避免 stall? | instruction scheduling |
| 寄存器分配 | 无限多的 IR 临时值,怎么塞进有限的物理寄存器? | register allocation |

这三个问题**互相耦合**(经典的 phase-ordering problem):指令选择会影响寄存器压力(register pressure);寄存器分配插入的 spill 代码又会反过来影响调度。理论上应该一起解,但联合求解是组合爆炸,所以**几乎所有真实编译器都分趟做**,只在边界上做有限的反馈。Cranelift 的做法是:指令选择(lowering)→ 寄存器分配(regalloc2)→ 二进制发射(emit),调度被弱化(它依赖乱序 CPU 自己调度)。

### 1.2 指令选择:从"宏展开"到"树覆盖"

最原始的指令选择是**宏展开**:每个 IR 操作对应一段固定的机器码模板。简单,但生成的代码烂——它看不到上下文,无法把 `a*4+b` 合并成一条 `lea` 指令。

真正的突破是把指令选择**形式化为树覆盖问题**:

- 把表达式建成一棵树(或 DAG);
- 每条机器指令是一个**带 cost 的树模式(tile / pattern)**,比如 x86 的 `lea` 能匹配 `+(reg, *(reg, const))` 这棵子树;
- 目标:用这些 tile **覆盖**整棵树,使总 cost 最小。

经典论文谱系(均经 WebSearch 核实):

- **Aho & Johnson (1976)**,"Optimal Code Generation for Expression Trees":用**动态规划**在表达式树上做最优指令选择,对一大类机器能在**线性时间**得到最优解。这是树覆盖 = DP 的奠基。【来源:WebSearch,Aho-Johnson 经典结论;原文 *JACM*,待核确切页码】
- **BURS(Bottom-Up Rewrite System)**:把 tile 匹配做成 term-rewrite + cost,代表工具 **twig**(Aho et al.)和 **burg / iburg**(Fraser, Hanson, Proebsting,"Engineering a simple, efficient code-generator generator",*ACM LOPLAS* 1992)。burg 在编译期把 cost 折进状态机,运行时一趟 bottom-up labeling 就拿到最优覆盖。【来源:WebSearch 命中 ACM dl.acm.org/doi/abs/10.1145/151640.151642 与 IIT-B retargetability.pdf;BURS 理论由 Pelegrí-Llopart & Graham 提出,待核年份】
- **Maximal Munch**:贪心策略——从根开始,每步咬掉**能匹配的最大 tile**,然后对剩下的子树递归。不保证全局最优,但实现极简(ML/Rust 的模式匹配天然适合)。Cranelift 的 ISLE 本质就是带优先级的 maximal munch。

> **关键洞察**:为什么现代编译器(LLVM SelectionDAG、Cranelift ISLE)在 DAG 而非 tree 上做指令选择?因为 SSA + CSE 之后,公共子表达式被共享,IR 是 **DAG** 不是树。DAG 上的最优覆盖是 **NP 完全**(树上才是 DP 多项式),所以 DAG 上退回到启发式(贪心 + 局部 DP)。

### 1.3 寄存器分配:NP 完全与 SSA 的救赎

寄存器分配的故事更戏剧化。

**Chaitin et al. (1981)**,"Register Allocation via Coloring"(*Computer Languages* 6(1):47–57)把寄存器分配**规约成图着色**:

- 节点 = live range(一个变量从定义到最后使用的活跃区间);
- 边 = 干涉(interference):两个 live range 在某个程序点**同时活跃**;
- 颜色 = 物理寄存器,共 K 个;
- 求 K-着色:相邻节点不同色。

【来源:Wikipedia Register allocation,WebFetch 核实:"foundational work came from Chaitin et al. (1981) ... 'Register allocation via coloring' (*Computer Languages*, 6(1): 47–57)"】

Chaitin 同时证明:**通用图的 K-着色是 NP 完全的,因此最优寄存器分配也是 NP 完全**(任意图都能构造成某段程序的 interference graph)。

【来源:Wikipedia,WebFetch:"Chaitin et al. proved optimal register allocation is NP-complete"】

Chaitin 的工程解法是**经典七阶段 + Kempe 启发式**:

1. Renumber(算 live range)
2. Build(建干涉图)
3. Coalesce(合并 copy 相关的、不干涉的节点——消除 `mov`)
4. Spill cost(估算每个节点 spill 的代价)
5. Simplify(把度数 < K 的节点压栈——Kempe 规则:度 < K 必可着色)
6. Spill Code(度数都 ≥ K 时,选一个 spill 到内存,插 load/store)
7. Select(从栈弹出,逐个染色)

【来源:Wikipedia,WebFetch 逐条核实七阶段名称】

**Briggs et al. (1994)** 的改进("optimistic coloring / conservative coalescing"):Chaitin 在 simplify 阶段碰到度 ≥ K 就立刻 spill,太悲观——度 ≥ K **不代表**不可着色(邻居可能用了重复颜色)。Briggs 的做法是**乐观地把它也压栈**,到 select 阶段真染不上色再 spill。这显著减少了 spill。【来源:Wikipedia,WebFetch:"Conservative Coalescing (Briggs et al., 1994): Only merges variables if it preserves colorability"】

**SSA 的红利**:在 SSA 形式上,每个 value 只定义一次,live range 不会"断开重连",于是 interference graph 变成 **chordal graph(弦图)**。弦图的最优着色是 **多项式时间**(沿 perfect elimination order 贪心即可),chromatic number = 最大团 = 最大寄存器压力。

【来源:Wikipedia,WebFetch:"for SSA-form programs, 'graph coloring portion ... can be solved in linear time.'" 以及 "Using SSA form produces 'shorter live intervals' ... simplifying the allocation problem"】

> ⚠️ **常见误解(待核细节)**:SSA 让"着色"变多项式,**不等于**整个寄存器分配变多项式。spilling(决定踢谁)、coalescing(合并哪些 copy)在 SSA 上**仍然 NP 难**。SSA 只解决了"给定不 spill 的情况下能否染色"这一子问题。把 SSA-based RA 当银弹是错的。

### 1.4 linear scan:为速度牺牲一点质量

图着色慢(建图 O(n²) 空间,着色多轮迭代)。1990 年代 JIT 兴起,编译时间 = 运行时间,等不起。

**Poletto & Sarkar (1999)**,"Linear Scan Register Allocation"(*ACM TOPLAS* 21(5), Sept 1999)提出:**根本不建干涉图**。把每个变量近似成一个**线性区间 `[i, j]`**(live interval),按起点排序,一趟扫过去,用一个 `active` 列表(按结束点排序)模拟"当前占用的寄存器",满了就 spill。复杂度近似 **O(n)**(更准确说 O(V·R) 或 O(V log R))。

【来源:Poletto & Sarkar 1999 原文 PDF,经 pdftotext 提取,见下文 §4 逐行精读】

代价:live interval 是 live range 的**保守过近似**——一个变量在中间有一段"空洞"(hole)不活跃,linear scan 看不见,会浪费一个寄存器。这是 linear scan 质量略逊图着色的根因。后续 **second-chance binpacking(Traub et al. 1998)** 和 **SSA-based linear scan(Wimmer & Franz / Mössenböck)** 通过 **live range splitting**(把区间在空洞处切开)来回收这些空洞——这正是现代 backtracking allocator 的核心思想。

【来源:Wikipedia,WebFetch:"Second-Chance Binpacking (Traub et al., 1998): ... split live ranges"】

### 1.5 现代生产路线:backtracking(以 regalloc2 为例)

Wasmtime/Cranelift 的 **regalloc2** 是当前最值得精读的生产级 RA。它的血统(README 自述):

> "about 50% still, a port of IonMonkey's backtracking register allocator to Rust" with subsequent "generalization, optimization, and improvements."

【**真实源码** bytecodealliance/regalloc2@README.md,WebFetch 核实】

它派生自 SpiderMonkey 的 **BacktrackingAllocator**(Ion JIT 的 RA),核心范式是:

- **live range splitting + backtracking**:不是一趟定死,而是用**优先队列**反复尝试;
- **eviction(驱逐)**:高优先级的 bundle 可以把已分配的低优先级 bundle 踢出去,被踢的重新入队;
- **保证终止**:bundle 越切越小,minimal bundle(只覆盖一条指令)权重无穷大,必定能抢到寄存器或 spill。

这是本章后半段源码精读的主角。它既不是纯图着色也不是纯 linear scan,而是把"区间 + 优先级 + 回溯"揉到一起,实践中**速度接近 linear scan、质量接近图着色**——这就是为什么它能撑起 Wasmtime 这种要求快速编译 + 不错代码质量的 Wasm JIT/AOT。

## 2. 核心机制拆解(数据结构与算法)

### 2.1 IR 端:vreg / preg / operand / constraint

寄存器分配器的输入是一个**带约束的、SSA 形式的函数**。regalloc2 用四个核心类型描述它(全部 **真实源码** bytecodealliance/regalloc2@src/lib.rs,WebFetch 核实):

```rust
// 虚拟寄存器 = SSA value。每类(Int/Float/Vector)最多 2^30 个。
pub struct VReg { bits: u32 }

// 物理寄存器。每类最多 64 个,跨类统一编址。
pub struct PReg { bits: u8 }

// 指令里对寄存器的一次"提及",带约束。
pub struct Operand { bits: u64 }

// 约束:这个 operand 必须落在哪。
pub enum OperandConstraint {
    Any,            // 寄存器或栈都行
    Reg,            // 必须在寄存器里
    Stack,          // 必须在栈上
    FixedReg(PReg), // 必须是这个具体寄存器(如 x86 的 div 要 rax/rdx)
    Reuse(usize),   // 复用第 N 个 operand 的分配(两地址指令:dst=src1)
    Limit(usize),   // (较新的约束类型)
}
```

这套设计的精妙处:**约束系统把目标机器的脏细节(固定寄存器、两地址指令、调用约定)统一编码进 operand**,RA 算法本身保持目标无关。比如:
- x86 `idiv` 隐式用 `rdx:rax` → 用 `FixedReg`;
- x86 `add dst, src`(dst 既读又写)→ 用 `Reuse(src_idx)`,强制 dst 和某个 src 同寄存器;
- 函数调用前,caller-saved 寄存器要保护 → 把 live-across-call 的值约束成 `Reg` 但避开 caller-saved,或干脆 spill。

分配器消费的接口是一个 trait(**真实源码** regalloc2@src/lib.rs):

```rust
pub trait Function {
    fn num_insts(&self) -> usize;
    fn num_blocks(&self) -> usize;
    fn block_succs(&self, block: Block) -> &[Block];   // CFG 后继
    fn inst_operands(&self, insn: Inst) -> &[Operand]; // 一条指令的所有 operand
    fn is_branch(&self, insn: Inst) -> bool;           // 是否块末分支
    fn num_vregs(&self) -> usize;
}
```

> **设计哲学**:regalloc2 把自己做成**纯库**,client(Cranelift)实现 `Function` trait 把 IR "喂"进来,RA 吐出"在哪个程序点插什么 move"的 edit 列表。RA 不碰具体 IR,也不碰具体机器码——只做"分配"这一件事。这是好的关注点分离,值得任何写编译器的人模仿。

### 2.2 输出端:Allocation / Edit / ProgPoint

分配器的产物(**真实源码** regalloc2@src/lib.rs):

```rust
pub struct Output {
    pub num_spillslots: usize,                  // 栈帧要留几个 spill 槽
    pub edits: Vec<(ProgPoint, Edit)>,          // 在哪个点插/删什么,已按程序点排序
    pub allocs: Vec<Allocation>,                // 每个 operand 最终落在哪
    pub inst_alloc_offsets: Vec<u32>,           // allocs 的按指令索引
    // ... debug_locations, stats
}

pub enum Edit {
    // 把一个分配搬到另一个。reg 或 spillslot 都行,
    // 但绝不会生成 stack->stack 的 move(x86 不允许 mem->mem)。
    Move { from: Allocation, to: Allocation },
}

pub enum InstPosition { Before = 0, After = 1 }  // 指令前 or 后
pub struct ProgPoint { bits: u32 }               // (inst, before/after) 打包
```

注意 `Edit` **只有 Move 一种**——所有 spill(reg→stack)、reload(stack→reg)、split 点的搬运、块边界的数据流转,统一抽象成"把数据从 A 搬到 B"。client 拿到这个 edit 列表,在对应程序点插入真实的 `mov`/`ldr`/`str`。

### 2.3 RA 内部:LiveRange / LiveBundle / SpillSet

这是 regalloc2 算法的核心三件套(全部 **真实源码** bytecodealliance/regalloc2@doc/ION.md,WebFetch 核实):

**LiveRange**:
> "A live-range is a contiguous range of program points (half-open, i.e. including `from` and excluding `to`) for which a particular vreg is live with a value."

一个 vreg 可能有**多个** live-range(被 split 之后)。注意是**半开区间** `[from, to)`——和 Poletto-Sarkar 的闭区间 `[i,j]` 不同,工程上半开区间在拼接时更不容易出 off-by-one。

**LiveBundle**:
> "A bundle is a collection of ranges that does not overlap. Each bundle carries: a vector ... of (range, live-range index) tuples, an allocation (starts as 'none'), a 'spillset', and some metadata."

bundle 是**分配的单位**——RA 给 bundle 分配一个寄存器,bundle 里所有 range 共享它。两条关键不变式:
> "(i) no liveranges within a bundle overlap, and (ii) no liveranges within a vreg overlap."

**为什么要 bundle 这一层?** 因为 RA 想把"应该在同一个寄存器里"的多段 range 打包一起分配(减少 move),但分配压力大时又能把 bundle **拆开**(split)分别分配。bundle 是 split / merge 的操作粒度。

**spill weight 和 priority**(决定先分配谁、驱逐谁):
> "A spill weight (sum of ranges' weights), a priority (sum of ranges' lengths), and three property flags: 'minimal', 'contains fixed constraints', 'contains stack constraints'."

- **priority = 各 range 长度之和** → 长 bundle 先分配("when the allocation maps are generally more free"),先占大的、后塞小的。
- **spill weight = 各使用点权重之和** → 决定驱逐:谁权重低谁被踢。

**SpillSet**:
> "Every bundle contains a reference to a spillset. Spillsets are used to assign spillslots near the end of allocation"

共享同一 spillset 的 bundle(通常源自同一 vreg 的不同 split)如果都 spill,可以**共用一个栈槽**——省栈空间。

### 2.4 主分配循环:优先队列 + 驱逐 + 切分

regalloc2 的核心循环(**真实源码** regalloc2@doc/ION.md):

> "The core allocation loop is a recurring iteration of the following: we take the highest-priority bundle from the allocation queue; we compute its requirements; we try to find it a register according to those requirements; if no fit, we either evict some other bundle(s) from their allocations and try again, or we split the bundle and put the parts back on the queue."

拆开看:

1. **取最高优先级 bundle**(优先队列 = 二叉最大堆):
   > "The allocation queue is simply a priority queue (built with a binary max-heap) of (prio, bundle-index) tuples."

2. **算 requirement**(用一个格 lattice 合并 operand 约束):
   > "Where top is 'Unknown' and bottom is 'Conflict'. Between these two, we have: any register ...; any stackslot ...; a particular register. 'Any register' can degrade to 'a particular register', but any other pair of different requirements meets to Conflict."

   如果 requirement 直接冲突(比如同一 bundle 里既要求 FixedReg(rax) 又要求 FixedReg(rcx)),立刻在冲突点 split:
   > "If the requirement indicates a conflict, we immediately split and requeue the split pieces."

3. **找寄存器**;找不到则二选一:**驱逐** 或 **切分**:
   > "We split if (i) the bundle is not already a minimal bundle, and (ii) we've already evicted once in this toplevel iteration without success, or the weight of the current bundle is less than the eviction cost. Otherwise, we evict all conflicting bundles and try again."

4. **驱逐规则(保证不死循环)**:
   > "Eviction only occurs when a bundle is *strictly* higher weight (not higher-or-equal) ... Once a bundle gets down to its 'minimal' size, it has an extremely high weight that is guaranteed to evict any non-minimal bundle."

**终止性证明**(为什么这个回溯不会无限循环)——这是整个设计最漂亮的地方:
> "Termination is guaranteed because (i) bundles always get smaller, (ii) eviction only occurs when a bundle is *strictly* higher weight ..., and (iii) once a bundle gets down to its 'minimal' size, it has an extremely high weight that is guaranteed to evict any non-minimal bundle."

直觉:每次 split bundle 变小、不会变大(单调);驱逐必须严格权重更高(不会 A 踢 B、B 又踢 A 来回拉锯);最坏切到 minimal(单指令)权重爆表,要么抢到寄存器要么 spill,到底了。这把一个看似可能发散的回溯算法**钉死成有限步**。

### 2.5 切分:在第一个冲突点切

切分策略(**真实源码** regalloc2@doc/ION.md):

> "Each iteration of the allocation loop splits at most *once*. This simplifies the splitting code greatly, but also turns out to be a nice heuristic: we split at the point that the bundle first encounters a conflict for a particular preg assignment, then we hint that preg for the first (pre-conflict) piece when we retry."

每轮**最多切一次**,切在"第一次撞到冲突"的点。切完给前半段**提示(hint)**那个寄存器,让它重试时优先拿到。切分后的"loose ends"(冲突点附近、超出实际使用范围的尾巴)被甩进 spill bundle:

> "if the last liverange in the first-half bundle extends beyond its last use, we trim that part off into an empty liverange and place that liverange in the spill bundle."

minimal bundle(切到底):
> "A minimal bundle is one that covers only one instruction."

### 2.6 指令选择端:Cranelift 的反向 lowering

切到指令选择这一侧。Cranelift 的 lowering driver(**真实源码** bytecodealliance/wasmtime@cranelift/codegen/src/machinst/lower.rs)用一个**反向遍历 + maximal munch**的精巧设计:

> "For each non-branch instruction, in reverse order: If side-effecting (load, store, branch/call/return, possible trap), or if used outside of this block, or if demanded by another inst, then lower."

**为什么反向?** 因为指令选择想做"模式合并"(把 `a + b*c` 合成一条乘加),需要知道"一个值是否只被用一次、能否安全地折进使用者"。反向遍历 + 一个 `ValueUseState` 分析就能回答:

> "A value is `Unused` if no references exist; `Once` if only one op refers to it ...; `Multiple` otherwise."
>
> "Multiple is contagious: even if an op's result is directly used once, that value is `Multiple` if the op using it is itself used multiple times."

只有 `Once` 且中间无 side-effect 的值,才能被"看穿"(`get_value_as_source_or_const`)折进 ISLE 模式——这就是 **maximal munch 的安全前提**:不能因为合并而重复执行有副作用的指令,也不能重复算被多次使用的值。

> **设计考古补注**:LLVM 的 SelectionDAG 用的是 DAG 上的 BURS-style 模式匹配(td 文件描述模式 + cost),GlobalISel 转向更命令式的 match。Cranelift 选了第三条路——**ISLE**(一个声明式 DSL,编译成 Rust 模式匹配树),本质是带优先级的 maximal munch over DAG。三家都在解同一个"树/DAG 覆盖"问题,只是 cost model 和匹配引擎不同。【来源:lower.rs WebFetch + 笔者对 LLVM/Cranelift 的背景知识,LLVM 细节待核】

### 2.7 收尾脏活:move 解析(parallel move)

RA 分配完、split 完之后,会在三类地方产生 move:

1. 同一 vreg 的两段 range 落在不同寄存器(split 点)——需要 `mov`;
2. 块边界:前驱块某 vreg 在 r1,后继块期望它在 r2;
3. block param / phi 的数据流转。

ION.md 描述它用 **half-move** 机制把边界 move 配对(**真实源码** regalloc2@doc/ION.md):
> "we generate a vector of 'half-moves', keyed on the edge and from/to vregs, with each holding one of the allocations"

然后按 `(from-block, to-block, to-vreg, kind)` 排序,让 source 和它的所有 dest 相邻,批量配对。

**真正的坑在"并行 move"**:同一个程序点上的一批 move 在语义上是**同时发生**的(并行赋值),但机器只能顺序执行。如果存在环(`r1←r2, r2←r1` 即 swap),顺序执行会破坏数据。解法(**真实源码** regalloc2@doc/ION.md):

> "we perform a depth-first search on the graph induced by the dependencies, which will generate a sequence of sequential moves"

环的处理:
> "the first move writes to a scratch register, and we emit an additional move that moves from the scratch to the first move's dest"

冗余 move 消除:
> "if the destination is already a copy of the source, we elide the move"

这套并行-move 序列化算法的理论出处是 **Boissinot et al., "Revisiting Out-of-SSA Translation"(LIP RR2008-40, 2008)**,以及更早的 Leroy 的算法。核心结论:N 个并行 copy,无环时 N 条顺序 copy 搞定;有环时需 **N+1 条 + 1 个 scratch**。【来源:WebSearch 命中 Boissinot 报告;算法细节见下文 §3 demo 2】

> 这一步**没有"看起来正确就行"**——并行 move 序列化错了是**静默 miscompile**,在 swap、循环回边、调用约定 shuffle 处尤其高发。下面的 demo 2 会把它完整跑一遍并自验证。

### Demo 1:IR → 目标代码 + 线性扫描寄存器分配

**目标**:实现一个最小后端:接收三地址码 IR,算 live interval,跑 Poletto-Sarkar linear scan(只给 2 个寄存器,**故意逼出 spill**),生成类 x86 目标代码。与 §4 的论文源码逐行呼应。

**完整代码**(保存为 `linscan.py`):

```python
#!/usr/bin/env python3
"""Minimal IR -> target lowering + linear-scan register allocation.
Faithful to Poletto & Sarkar (1999) Fig.1. Stdlib only."""
from dataclasses import dataclass

# ---------- tiny IR (三地址码) ----------
#   ('mov', dst, src_imm_or_temp)
#   ('add'/'sub'/'mul', dst, a, b)
#   ('ret', src)
PROG = [
    ('mov', 'a', 1),
    ('mov', 'b', 2),
    ('add', 'c', 'a', 'b'),     # c = a + b
    ('mov', 'd', 10),
    ('mul', 'e', 'c', 'd'),     # e = c * d
    ('add', 'f', 'e', 'c'),     # f = e + c  (c 在这里还活着)
    ('sub', 'g', 'f', 'b'),     # g = f - b  (b 活了很久,跨越大段)
    ('ret', 'g'),
]

def is_temp(x):
    return isinstance(x, str)

# ---------- liveness -> live intervals ----------
@dataclass
class Interval:
    name: str
    start: int
    end: int
    reg: str = None      # 分到的物理寄存器
    slot: int = None     # spill 到的栈槽

def compute_intervals(prog):
    """近似 live interval:[首次提及, 末次提及]。这正是论文的保守近似。"""
    first, last = {}, {}
    for i, ins in enumerate(prog):
        op = ins[0]
        defs, uses = [], []
        if op == 'mov':
            defs = [ins[1]]; uses = [ins[2]] if is_temp(ins[2]) else []
        elif op in ('add', 'sub', 'mul'):
            defs = [ins[1]]; uses = [x for x in ins[2:] if is_temp(x)]
        elif op == 'ret':
            uses = [ins[1]]
        for v in defs + uses:
            if v not in first:
                first[v] = i
            last[v] = i
    ivs = [Interval(v, first[v], last[v]) for v in first]
    ivs.sort(key=lambda iv: iv.start)   # 按起点排序 —— 论文要求
    return ivs

# ---------- linear scan (Poletto & Sarkar Fig.1 逐句对应) ----------
def linear_scan(intervals, regs):
    R = len(regs)
    free = list(regs)
    active = []            # 按 END point 升序维护 —— 论文核心不变式
    next_slot = [0]

    def expire_old(i):
        # ExpireOldIntervals(i): foreach j in active by increasing end
        for j in list(active):
            if j.end >= i.start:        # endpoint[j] >= startpoint[i] -> 停
                return
            active.remove(j)
            free.append(j.reg)          # reg 还池
    def spill_at(i):
        # SpillAtInterval(i)
        spill = active[-1]              # active 末尾 = 结束点最远者
        if spill.end > i.end:
            i.reg = spill.reg           # 抢走它的寄存器
            spill.reg = None
            spill.slot = next_slot[0]; next_slot[0] += 1
            active.remove(spill)
            active.append(i)
            active.sort(key=lambda x: x.end)
        else:
            i.slot = next_slot[0]; next_slot[0] += 1  # spill 新区间自己

    for i in intervals:
        expire_old(i)
        if len(active) == R:
            spill_at(i)
        else:
            i.reg = free.pop(0)
            active.append(i)
            active.sort(key=lambda x: x.end)
    return next_slot[0]

# ---------- codegen,用分配结果 ----------
def loc(iv_map, name):
    iv = iv_map[name]
    return iv.reg if iv.reg else f"[sp+{iv.slot*8}]"

def codegen(prog, iv_map):
    out = []
    for ins in prog:
        op = ins[0]
        if op == 'mov':
            src = ins[2] if not is_temp(ins[2]) else loc(iv_map, ins[2])
            out.append(f"  mov {loc(iv_map, ins[1])}, {src}")
        elif op in ('add', 'sub', 'mul'):
            d, a, b = loc(iv_map, ins[1]), loc(iv_map, ins[2]), loc(iv_map, ins[3])
            mn = {'add':'add','sub':'sub','mul':'imul'}[op]
            out.append(f"  mov {d}, {a}")
            out.append(f"  {mn} {d}, {b}")
        elif op == 'ret':
            out.append(f"  mov rax, {loc(iv_map, ins[1])}")
            out.append("  ret")
    return out

if __name__ == '__main__':
    REGS = ['r1', 'r2']      # 故意只给 2 个,逼出 spill
    ivs = compute_intervals(PROG)
    print("=== live intervals (sorted by start) ===")
    for iv in ivs:
        print(f"  {iv.name}: [{iv.start},{iv.end}]")
    nslots = linear_scan(ivs, REGS)
    iv_map = {iv.name: iv for iv in ivs}
    print(f"\n=== allocation (R={len(REGS)} regs: {REGS}) ===")
    for iv in sorted(ivs, key=lambda x: x.name):
        where = iv.reg if iv.reg else f"SPILLED slot{iv.slot}"
        print(f"  {iv.name}: {where}")
    print(f"  spill slots used: {nslots}")
    print("\n=== generated target code ===")
    for line in codegen(PROG, iv_map):
        print(line)
```

```

**读懂这个输出(与论文呼应)**:

- `b: [1,6]` 活得最久。在 step 3(处理 `d:[3,4]`)时 active 满了(`b,c` 占着 r1/r2 等价情形),`SpillAtInterval` 触发,比较"active 末尾结束点最远者"——`b` 结束点 6 比新区间远,于是 **`b` 被 spill**,寄存器让给后来者。这就是论文 §4 的"spill the interval that ends last"。
- `e` 也被 spill:寄存器压力第二次顶满。
- spill 槽**复用**:虽然 `b` 和 `e` 都 spill,但它们 live interval 不重叠……不过本 demo 为简明给了各自的槽(`slot0`/`slot1`)。生产级 RA(regalloc2 的 SpillSet)会做槽复用——这是 demo 简化点,§5 会点明。

**自验证脚本**(`verify.py`,证明分配无冲突):

即:**没有两个时间上重叠的区间被分到同一个物理寄存器**——这正是寄存器分配的正确性定义(干涉的 live range 不同色)。

> **这个 demo 故意暴露的两个真坑**(教学价值 > 完美):
> 1. 生成的 `imul [sp+8], r1` —— x86 的 `imul` **不能以内存为目的操作数**(只能 reg 目的)。当操作数被 spill 到栈时,naive codegen 会生成非法指令。真实后端必须先把 spilled 操作数 **reload 进一个 scratch 寄存器**再运算。这正是"spill 不只是改地址,还要插 reload + 受指令编码约束"的活教材。
> 2. `mov [sp+0], 2` 之后 `add r2, [sp+0]` —— mem 操作数在 x86 上 OK(add 支持一个 mem 操作数),但 `mov [a], [b]` 这种 **mem→mem 永远非法**,所以 regalloc2 才明确承诺"never generate stack-to-stack moves"(见 §2.2 Edit 注释)。

### Demo 2:并行 move 序列化(拆环 + 自验证)

**目标**:实现 §2.7 的并行-move 序列化。给一批"同时发生"的 move,输出一个**顺序执行也保证语义不变**的序列,环用 scratch 寄存器拆。覆盖 4 种情形:链、2-环(swap)、3-环、fan-out。每种都**模拟执行并断言语义守恒**。

**完整代码**(`parmove.py`):

```python
#!/usr/bin/env python3
"""Parallel-move sequentialization with cycle breaking.
regalloc2 的 'move resolution' 步骤的最小实现。stdlib only."""

SCRATCH = 'scratch'

def sequentialize(par_moves):
    """par_moves: [(src, dst), ...] 语义上同时发生。返回顺序 move 列表。"""
    src_of = {}
    for s, d in par_moves:
        if s != d:                      # 丢弃平凡自 move
            src_of[d] = s               # SSA-out: 每个 dst 至多一个 src
    result = []
    pending = dict(src_of)
    def needed_as_src(loc):
        return any(s == loc for s in pending.values())
    while pending:
        progress = False
        for d in list(pending):
            if not needed_as_src(d):    # d 当前值没人要了 -> 可安全覆盖
                result.append((pending[d], d))
                del pending[d]; progress = True
        if not progress:                # 只剩环 -> 用 scratch 拆一个
            d, s = next(iter(pending.items()))
            result.append((d, SCRATCH))         # 先把 d 的值存进 scratch
            for dd in pending:                  # 读 d 的人改读 scratch
                if pending[dd] == d:
                    pending[dd] = SCRATCH
            result.append((s, d))               # 现在 d 可以接收它的值
            del pending[d]
    return result

def run(label, moves):
    print(f"\n=== {label} ===")
    print("  parallel:", " | ".join(f"{d}<-{s}" for s,d in moves))
    seq = sequentialize(moves)
    print("  sequential:")
    for s, d in seq:
        print(f"    mov {d}, {s}")
    # 模拟执行,证明语义被保留
    initial = {r: f"v_{r}" for s,d in moves for r in (s,d)}
    initial[SCRATCH] = 'v_scratch'
    state = dict(initial)
    for s, d in seq:
        state[d] = state[s]
    expected = {d: initial[s] for s, d in moves if s != d}
    ok = all(state[d] == v for d, v in expected.items())
    print("  semantics preserved:", ok)
    return ok

if __name__ == '__main__':
    ok = True
    ok &= run("chain (no cycle)", [('r1','r2'), ('r2','r3')])
    ok &= run("swap (2-cycle)",   [('r1','r2'), ('r2','r1')])
    ok &= run("3-cycle",          [('r1','r2'), ('r2','r3'), ('r3','r1')])
    ok &= run("fan-out",          [('r1','r2'), ('r1','r3')])
    print("\nALL OK" if ok else "\nFAILURE")
```

**运行**:`python3 parmove.py`

**实际输出**(真实运行,非杜撰):
```
=== chain (no cycle) ===
  parallel: r2<-r1 | r3<-r2
  sequential:
    mov r3, r2
    mov r2, r1
  semantics preserved: True

=== swap (2-cycle) ===
  parallel: r2<-r1 | r1<-r2
  sequential:
    mov scratch, r2
    mov r2, r1
    mov r1, scratch
  semantics preserved: True

=== 3-cycle ===
  parallel: r2<-r1 | r3<-r2 | r1<-r3
  sequential:
    mov scratch, r2
    mov r2, r1
    mov r1, r3
    mov r3, scratch
  semantics preserved: True

=== fan-out ===
  parallel: r2<-r1 | r3<-r1
  sequential:
    mov r2, r1
    mov r3, r1
  semantics preserved: True

ALL OK
```

**读懂关键点**:
- **chain**:`r3<-r2, r2<-r1` 必须**先 r3<-r2 再 r2<-r1**。如果顺序反了,`r2` 先被 r1 覆盖,r3 就拿到错的值。算法靠 `needed_as_src` 自动定序:r2 还被 r3 需要,所以先做 r3。
- **swap**:经典 2-环,无法不借助第三方完成。输出 `scratch←r2; r2←r1; r1←scratch` —— **N+1 = 3 条 move + 1 scratch**,与 Boissinot 的理论结论一致。
- **3-cycle**:N=3 → 4 条 move,验证拆环对任意长度环成立。
- **fan-out**:一个源两个目的,**不是环**,两条独立 move,无需 scratch。

> 这个 demo 直接对应 regalloc2 ION.md 的两句源码注释:"depth-first search on the graph induced by the dependencies"(我的 `needed_as_src` 拓扑定序)+ "first move writes to a scratch register"(我的拆环)。它把"为什么需要 scratch 寄存器""为什么 RA 要预留一个 scratch"这个常被忽视的工程必需品讲清楚了。

## 4. 真实源码精读:Poletto & Sarkar linear scan(逐行)

下面是 Poletto & Sarkar 1999 论文 Fig.1 的**逐字伪代码**,经本地 `pdftotext` 从论文 PDF 提取(原始 PDF 由 WebFetch 取自 UCLA CS132 课程镜像 `web.cs.ucla.edu/~palsberg/course/cs132/linearscan.pdf`)。

> **【真实源码 Poletto & Sarkar 1999, Fig.1,pdftotext 提取】**
>
> ```
> LinearScanRegisterAllocation
>     active ← {}
>     foreach live interval i, in order of increasing start point
>          ExpireOldIntervals(i)
>          if length(active) = R then
>                SpillAtInterval(i)
>          else
>                register[i] ← a register removed from pool of free registers
>                add i to active, sorted by increasing end point
>
> ExpireOldIntervals(i)
>     foreach interval j in active, in order of increasing end point
>          if endpoint[j] ≥ startpoint[i] then
>               return
>          remove j from active
>          add register[j] to pool of free registers
>
> SpillAtInterval(i)
>      spill ← last interval in active
>      if endpoint[spill] > endpoint[i] then
>            register[i] ← register[spill]
>            location[spill] ← new stack location
>            remove spill from active
>            add i to active, sorted by increasing end point
>      else
>            location[i] ← new stack location
> ```

**逐段注解**(每段对应 demo 1 的同名函数):

**主循环 `LinearScanRegisterAllocation`**
- `foreach live interval i, in order of increasing start point` —— **按起点扫**。这是"linear" 的来源:一趟过去。前提是 interval 已按起点排好(demo 里 `compute_intervals` 末尾的 sort)。
- `ExpireOldIntervals(i)` —— 先回收:在 `i` 开始前已经结束的区间,把它们的寄存器还池。
- `if length(active) = R` —— active 列表长度 = 已用寄存器数。等于 R 说明寄存器**用光了**,必须 spill。
- `else register[i] ← a register ...` —— 还有空寄存器,直接分配,把 `i` 插入 active(**保持按结束点排序**,这是后面 spill 能 O(1) 找到"最远者"的关键)。

**`ExpireOldIntervals(i)`**
- `foreach interval j in active, in order of increasing end point` —— active 按结束点升序,所以**最早结束的在前**。
- `if endpoint[j] ≥ startpoint[i] then return` —— 一旦碰到一个"结束点 ≥ 当前起点"的,说明它(和它后面所有结束更晚的)都还活着,**立即返回**。这就是为什么 active 要按结束点排序:能提前 break,均摊 O(1)。
  - ⚠️ 论文用 `≥`(闭区间语义:endpoint 那一刻还活着)。regalloc2 用半开区间 `[from,to)` 所以是 `>`。**这个边界差异是 off-by-one bug 的高发区**,移植时务必对齐区间开闭约定。
- `remove j from active; add register[j] to pool` —— 真正回收。

**`SpillAtInterval(i)`** —— 全章最该背下来的 8 行
- `spill ← last interval in active` —— active 末尾 = **结束点最远者**(因为按结束点升序排)。
- `if endpoint[spill] > endpoint[i]` —— 比较"已在寄存器里、结束最远的那个" vs "新来的 i" 谁结束更晚。
  - **更晚的那个被 spill**。直觉(Belady 思想):一个值越晚才被再次用到,把它踢到内存"亏"得越少,因为它会在寄存器外待最久、最不碍事。
  - 若 `spill` 更晚 → 踢 `spill`,把它的寄存器给 `i`,`spill` 去栈上。
  - `else`(新来的 `i` 反而结束最晚)→ 干脆**直接 spill `i` 自己**,不动已有分配。
- 论文紧跟着说(§4 正文,pdftotext 提取):
  > "Our algorithm spills the interval that ends last, furthest away from the current point. We can find this interval quickly because active is sorted by increasing end point"

  并指出在直线代码、每个区间恰好"一次定义一次使用"时,这个启发式**等价于 Belady 1966 的最优页面替换**,产生最少 spill:
  > "produces code with the minimal possible number of spilled live ranges [Belady 1966; ...]"

**复杂度**(论文正文):active 列表长度 **至多 R**;每个 interval 进出 active 各一次;spill 决策 O(1)(只看末尾)。所以核心循环近似 **O(V)**(V = interval 数),最坏 O(V·R) 或排序的 O(V log V)。对比图着色的多轮迭代 + O(V²) 干涉图——这就是 linear scan 在 JIT 里碾压图着色的本钱。

> **从论文到 regalloc2 的演化**:linear scan 的"踢结束点最远者"是个**纯位置启发式**,看不到"这个值被用了几次"。regalloc2 的 **spill weight = 各使用点权重之和**是它的进化版——不光看位置还看使用频度(循环里的值权重更高,更不该被踢)。而 linear scan 那个"看不见空洞"的缺陷,regalloc2 用 **live range splitting** 在空洞处切开来根治。所以可以把 regalloc2 理解成:**linear scan(区间 + active 思想)+ 使用权重(取代纯位置)+ splitting(回收空洞)+ backtracking/eviction(逼近图着色质量)**。读懂了上面这 8 行,再读 §2 的 regalloc2 就有了坐标系。

## 5. 方案对比

### 5.1 寄存器分配三大路线对比

| 维度 | 图着色 (Chaitin-Briggs) | 线性扫描 (Poletto-Sarkar) | 回溯/二次机会 (regalloc2 / Ion) |
|---|---|---|---|
| 核心数据结构 | 干涉图 (O(V²) 空间) | live interval + active 列表 | LiveRange/Bundle + 优先队列 |
| 时间复杂度 | 多轮迭代,慢 | ~O(V),最快 | 介于两者,接近 linear scan |
| 代码质量 | 高(最优逼近) | 略低(看不见空洞) | 高(splitting 回收空洞) |
| spill 启发式 | spill cost(度数/使用) | 结束点最远 (Belady-like) | spill weight(使用点权重和) |
| live range splitting | 通常不做(或粗粒度) | 不做(原始版) | **核心机制**,细粒度 |
| coalescing(消 move) | 强(conservative/biased) | 弱 | 中(hint + 二次机会) |
| 处理固定约束/ABI | 需额外 pre-color | 笨拙 | 原生(OperandConstraint) |
| 典型使用者 | GCC、LLVM(greedy 本质也是这类) | 早期 HotSpot client、早期 V8 | SpiderMonkey Ion、Wasmtime/Cranelift |
| 适用场景 | AOT、可接受长编译时 | 极致快编译、可容忍质量损失 | JIT/AOT 都要、要快又要质量 |

【来源:Wikipedia register allocation(WebFetch)+ regalloc2 ION.md(WebFetch)+ Poletto-Sarkar 论文;LLVM greedy RA 的"本质是带 splitting 的优先级着色"为笔者判断,LLVM 源码细节待核】

### 5.2 指令选择两大流派对比

| 维度 | Maximal Munch(贪心) | BURS / DP(动态规划) |
|---|---|---|
| 最优性 | 局部最优,非全局 | 树上全局最优(Aho-Johnson) |
| 实现复杂度 | 低(模式匹配即可) | 高(需 cost 折叠 / 生成器) |
| 运行时开销 | 一趟匹配 | bottom-up labeling(burg 也很快) |
| DAG 支持 | 自然(但要防重复副作用) | 树上最优;DAG 退化为启发式 |
| 代表 | Cranelift ISLE、教科书 | twig、burg/iburg、LLVM SelectionDAG(td) |
| 适用场景 | RISC、模式较正交的 ISA | CISC、tile 重叠多、追求最优 |

【来源:WebSearch(Aho-Johnson、BURS、burg)+ Cranelift lower.rs(WebFetch);具体工具归类部分待核】

### 5.3 具体场景:该选哪个

- **写一个 toy 语言 / 教学编译器**:linear scan + maximal munch。两百行能跑,质量够用,正是本章 demo 的路线。
- **写一个 Wasm/JS JIT(要快编译 + 不差的码)**:抄 regalloc2 的 backtracking。别从零写图着色——慢,且固定约束/ABI 处理会让你崩溃。
- **写一个 AOT 优化编译器(C/Rust 级,编译时间不敏感)**:图着色 / LLVM greedy 路线,配激进 coalescing。
- **目标是 CISC(x86)且想要极致码质量**:指令选择上认真做 DAG 覆盖(BURS 风格),因为 x86 的 `lea`、复杂寻址、读-改-写指令让 tile 高度重叠,贪心会漏掉很多合并机会。

### 5.4 不适用边界(什么时候上面的结论失效)

- **linear scan 在控制流复杂、空洞多的代码上质量崩**:大量"区间中间有大段不活跃"的情况,纯 linear scan 浪费寄存器严重——这时必须上 splitting,否则不如直接图着色。
- **图着色在 JIT 里不适用**:编译时间算进运行时,O(V²) 建图 + 多轮迭代等不起。
- **SSA-based 弦图最优着色不能直接降到机器码**:它假设"phi 是并行的、move 免费"。落地必须做 out-of-SSA(把 phi 变成块边界的并行 move)+ move 序列化(demo 2),而**这一步可能引入 spill**(swap 需要 scratch),所以"SSA 上最优"≠"最终最优"。
- **maximal munch 在有副作用的 DAG 上直接用 = bug**:必须先做 use-count 分析(Cranelift 的 `ValueUseState`),只折叠 use-once 且无中间副作用的值,否则会重复执行 load/store。

## 6. 扎根:失败模式、真坑与根因

### 坑 1:并行 move 没拆环 → 静默 miscompile

**现象**:循环回边、phi 落地、调用约定 shuffle 处,某些变量值被悄悄写错,结果偶发错误且极难复现(只在特定寄存器分配下触发)。

**根因**:out-of-SSA 后块边界产生**并行赋值**,顺序化时没检测环(swap)。`mov r1, r2; mov r2, r1` 顺序执行后两个寄存器都变成 r2 的旧值。

**正解**:依赖图拓扑排序 + 环用 scratch 拆(demo 2)。这也是**为什么 RA 必须永久预留一个 scratch 寄存器**——很多人不理解为啥不能把所有寄存器都拿去分配,这就是原因之一。regalloc2 明确处理:"first move writes to a scratch register"。

### 坑 2:spill 了却忘了指令编码约束 → 生成非法指令

**现象**:把某操作数 spill 到栈后,生成出 `imul [mem], reg` 或 `mov [mem], [mem]` 这种 x86 根本不存在的指令,汇编器报错或(更糟)生成错误编码。

**根因**:naive 实现把"spill"等同于"把寄存器名换成 `[sp+offset]`",但**真实 ISA 对 mem 操作数有严格限制**(x86 一条指令至多一个 mem 操作数,某些指令不接受 mem 目的)。demo 1 故意保留了这个 bug 作教材。

**正解**:spill 不是改地址,是**插入显式 reload**:把 spilled 值先 `mov` 进一个 scratch 寄存器,再参与运算。这就是为什么 regalloc2 的 `Edit::Move` 注释强调"stack-to-stack moves will never be generated"——它在 RA 层就保证不产生非法搬运。

### 坑 3:区间开闭约定不一致 → off-by-one,寄存器被过早回收

**现象**:某个值在它"最后一次使用的那条指令"上被覆盖,因为分配器以为它已经死了。

**根因**:Poletto-Sarkar 用闭区间 `[i,j]` + `endpoint[j] ≥ startpoint[i]` 判活;regalloc2 用半开 `[from,to)` + `>`。混用两种约定,边界差一格。**defs 和 uses 在同一条指令上时尤其危险**(读-改-写指令:operand 既是 use 又是 def,它的 live point 怎么算?)。

**正解**:全局统一区间开闭约定;对读-改-写指令用 regalloc2 的 `Reuse` 约束显式表达"dst 复用 src 的寄存器",而不是靠区间巧合。regalloc2 把程序点细分成 Before/After(`InstPosition`)正是为了精确表达"指令前的 use" vs "指令后的 def"。

### 坑 4:固定寄存器约束没 pre-color → div/call 处分配崩溃

**现象**:x86 `idiv` 隐式用 `rax:rdx`,如果分配器不知道,把别的值放在了 rax,执行 idiv 时该值被覆盖。

**根因**:目标机器有大量**隐式固定寄存器**(idiv、shift 用 cl、call 的参数/返回寄存器、caller-saved 在 call 处全失效)。通用 RA 算法看不见这些。

**正解**:用约束系统(`FixedReg(PReg)`)把固定要求编码进 operand;call 处把所有 caller-saved 寄存器建模成"被 call 这条指令 clobber",强制 live-across-call 的值要么在 callee-saved 里要么 spill。这正是 regalloc2 把 ABI 脏活统一塞进 `OperandConstraint` 的价值——**算法目标无关,约束系统吸收所有机器特异性**。

### 坑 5:coalescing 过度 → 制造出无法着色的"超级节点"

**现象**:为了消 `mov`,把两个 copy 相关的 live range 合并,结果合并后的节点度数爆炸,反而逼出更多 spill。

**根因**:激进 coalescing(Chaitin 原始版)合并任意 copy 相关节点,可能把一个本来可着色的图变成不可着色。

**正解**:**conservative coalescing(Briggs 1994)**——只在"合并后仍保持可着色"(合并节点的高度数邻居数 < K)时才合并。regalloc2 走的是另一条路:不显式 coalesce,而是用 **hint**(split 后提示偏好寄存器)+ 二次机会,在分配过程中自然消 move,避开了 coalescing 的 colorability 陷阱。

## 章末五件套

### ① 一句话总结
代码生成 = 把目标无关的 SSA IR,经**指令选择(树/DAG 覆盖,贪心或 DP)**落成机器指令、经**寄存器分配(图着色 NP 完全,SSA 上弦图可线性最优;工程用 linear scan / backtracking 折中)**塞进有限物理寄存器,再用 **parallel-move 序列化**收尾——理论优雅、工程肮脏的"最后一公里"。

### ② 心智模型(一张图记住全章)
```
SSA IR (无限 vreg)
   │
   ▼  指令选择 = 树/DAG 覆盖
      maximal munch(贪心,Cranelift ISLE) | BURS/DP(最优,burg)
      ── 安全前提:只折叠 use-once 且无中间副作用的值
   │
   ▼  寄存器分配 = 图着色(干涉的 live range 不同色)
      ┌─ 图着色:建干涉图,Kempe 简化,质量高/慢   (GCC/LLVM)
      ├─ linear scan:active 列表,踢"结束最远",O(V) (早期 JIT)
      └─ backtracking:Bundle+优先队列+eviction+split  (regalloc2/Ion)
              spill weight = Σ使用点权重;终止性靠"越切越小+严格权重驱逐"
   │
   ▼  out-of-SSA + move 解析(收尾脏活)
      phi → 块边界并行 move → 拓扑序列化,环用 scratch 拆 (N+1 moves)
   │
   ▼  机器码 (有限 preg + spill slots)
```

### ③ 何时用什么(决策清单)
- toy/教学 → linear scan + maximal munch(本章 demo)
- Wasm/JS JIT → backtracking(抄 regalloc2)
- AOT 优化编译器 → 图着色 / LLVM greedy + conservative coalescing
- CISC 极致码质量 → DAG 覆盖(BURS 风格)做指令选择
- 任何情况 → **永远预留一个 scratch 寄存器**(拆 move 环 + reload spill 用)

### ④ 最易翻车点(背下来)
1. 并行 move 不拆环 = 静默 miscompile(swap 必须用 scratch)。
2. spill ≠ 改地址;必须插 reload,且受 ISA mem 操作数约束(x86 不能 mem→mem、imul 不能 mem 目的)。
3. 区间开闭约定要全局统一(`[i,j]`+`≥` vs `[from,to)`+`>`),否则 off-by-one。
4. 固定寄存器(idiv/call/shift)必须 pre-color / 约束建模,否则崩。
5. coalescing 要 conservative,否则制造不可着色超级节点。
