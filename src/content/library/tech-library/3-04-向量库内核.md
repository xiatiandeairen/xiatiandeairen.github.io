## 4.1 设计考古:从「一个索引」到「一个内核」

### 4.1.1 核心区分——算法底座 vs 存储底座

本系列把「数据检索底座」拆成两个相邻但不同的域:

| 维度 | 检索**算法**底座(第 3 章) | 向量库**内核**底座(本章) |
|---|---|---|
| 回答的问题 | 给定 N 个向量,怎么在 sublinear 时间找近邻 | 怎么把算法变成一个能 add / search / delete / filter / 落盘的引擎 |
| 关注点 | 召回-延迟 trade-off、图/桶/量化的数学 | 抽象接口、存储布局、可变性、过滤、并发、持久化 |
| 代表产物 | `getNeighborsByHeuristic2`、ADC 距离表 | `index_factory`、`InvertedLists`、cardinality estimator、segment |
| 失败长相 | 召回崩、延迟爆 | 删不掉、过滤后召回崩、重建卡死、OOM、落盘格式不兼容 |

一句话:**算法决定「单次查询有多快多准」,内核决定「这东西能不能当生产存储用」。** 真实事故里,后者出问题的频率远高于前者——算法是论文验证过的,内核是工程师现攒的。

> 设计考古出处(均 WebFetch / WebSearch 核实,URL 见章末):
> - HNSW 算法:Malkov & Yashunin,*Efficient and robust approximate nearest neighbor search using HNSW graphs*,arXiv:1603.09320(2016,后收入 IEEE TPAMI)。
> - Faiss:facebookresearch/faiss,把 ANN 原语做成可组合 `Index` 体系的工业实现。
> - qdrant:qdrant/qdrant,把「向量索引 + payload 过滤」做成数据库内核,**filterable HNSW + cardinality estimation** 是其核心差异化。

### 4.1.2 为什么内核要「组合」而不是「内置一个算法」

早期向量检索是「选一个算法,写死」。现实很快打脸:

- 高维 + 内存敏感 → 要 PQ 压缩;
- 但 PQ 直接量化原始向量,误差大 → 先 OPQ 旋转,再分段;
- 但全量 PQ 扫描还是 O(N) → 套一层 IVF 粗量化做 non-exhaustive;
- 但 PQ 排序精度不够 → 末端挂一个 Refine 用原向量重排。

于是「一个查询」=`OPQ → IVF → PQ → Refine` 一条**变换链**。Faiss 的关键工程决策:**不把这条链写死,而是定义一个 `Index` 抽象基类,让每段都是可插拔的 `Index`,再用 `index_factory("OPQ32,IVF4096,PQ32,RFlat")` 这种字符串 DSL 把它们拼起来。** 内核的职责从「实现一个算法」变成「编排一组算法 + 管理它们的存储与生命周期」。

这正是为什么本章把「内核」单列:**组合性(composability)本身就是一种工程能力**,它要求每个原语都遵守同一套 add / search / 序列化契约。

## 4.2 索引组合:Faiss 的 `Index` 抽象与 IVF-PQ 内核

### 4.2.1 `IndexIVFPQ` 的类骨架——内核如何「持有」算法

下面是 `IndexIVFPQ` 的核心成员与构造签名。注意它**自己不实现 PQ 数学**,而是**持有**一个 `ProductQuantizer` 成员、继承一个 `IndexIVF` 基类(后者持有 coarse quantizer)——典型的「内核组合算法」结构。

【真实源码 facebookresearch/faiss@faiss/IndexIVFPQ.h】(WebFetch 取得,关键成员与签名逐字)
```cpp
ProductQuantizer pq;            ///< produces the codes
bool do_polysemous_training;
PolysemousTraining* polysemous_training;
size_t scan_table_threshold;
int polysemous_ht;
int use_precomputed_table;
AlignedTable<float> precomputed_table;
```
```cpp
IndexIVFPQ(
        Index* quantizer,         // ← coarse quantizer(IVF 的「桶中心」),本身也是一个 Index
        size_t d,
        size_t nlist,             // ← IVF 桶数
        size_t M,                 // ← PQ 子量化器段数
        size_t nbits_per_idx,     // ← 每段比特数(通常 8 -> 256 centroid)
        MetricType metric = METRIC_L2,
        bool own_invlists = true);
```
```cpp
void add_core(
        idx_t n, const float* x, const idx_t* xids,
        const idx_t* precomputed_idx,
        void* inverted_list_context = nullptr) override;

void encode_vectors(
        idx_t n, const float* x, const idx_t* list_nos,
        uint8_t* codes, bool include_listnos = false) const override;

void precompute_table();          // ← 预计算 query-independent 的距离项
```

逐行解读(内核视角,不重讲 PQ 算法):
- **`Index* quantizer` 作为构造参数**:coarse quantizer 不是写死的 kmeans,而是「任意一个 `Index`」。你可以传 `IndexFlatL2`(精确选桶),也可以传 `IndexHNSWFlat`(连选桶都用 HNSW 加速,这就是 `IVF_HNSW`)。**这就是组合性的回报**:粗量化器本身可被替换升级,内核代码一行不改。
- **`ProductQuantizer pq` 是成员而非继承**:IVF-PQ「is-an IVF index」「has-a PQ」。算法被内核当组件持有。
- **`use_precomputed_table` / `precomputed_table`**:纯算法实现不会有这俩。它们是**内核级优化**——把 IVFADC 距离里 query-independent 的项(centroid 与 PQ centroid 的交叉项)预先算好缓存,查询时省掉一部分计算。这是「把算法装进引擎」时才出现的工程化产物。(算法层的 ADC 推导见第 3 章。)
- **`encode_vectors` 里的 `list_nos`**:编码时**需要知道向量落在哪个桶**,因为 `by_residual` 模式下 PQ 编的是 `x - centroid[list_no]`(残差)。内核负责在 add 流程里先过 coarse quantizer 拿到 `list_no`,再喂给 PQ。**算法(残差量化)和存储(桶归属)在内核这一层被缝合。**

### 4.2.2 `ProductQuantizer`——被内核持有的算法原语长什么样

为了看清「内核持有的算法组件」的边界,这是 PQ 原语的 struct(它继承 `Quantizer`,只管「向量 ↔ code」的双向映射,完全不知道 IVF / 桶 / 存储的存在——干净的单一职责)。

【真实源码 facebookresearch/faiss@faiss/impl/ProductQuantizer.h】(WebFetch 取得,逐字)
```cpp
struct ProductQuantizer : Quantizer {
    size_t M;     ///< number of subquantizers
    size_t nbits; ///< number of bits per quantization index
    size_t dsub;  ///< dimensionality of each subvector
    size_t ksub;  ///< number of centroids for each subquantizer
    bool verbose; ///< verbose during training?

    std::vector<float> centroids;
    std::vector<float> transposed_centroids;
    std::vector<float> centroids_sq_lengths;

    float* get_centroids(size_t m, size_t i) {
        return &centroids[(m * ksub + i) * dsub];   // ← centroid 的扁平布局寻址
    }

    void compute_code(const float* x, uint8_t* code) const;
    void compute_codes(const float* x, uint8_t* codes, size_t n) const override;
    void compute_distance_table(const float* x, float* dis_table) const;
    void search(
            const float* x, size_t nx,
            const uint8_t* codes, const size_t ncodes,
            float_maxheap_array_t* res,
            bool init_finalize_heap = true) const;
};
```

内核视角的三个观察:
1. **`centroids` 是一维 `vector<float>` + `get_centroids(m,i)` 手动寻址**`(m*ksub+i)*dsub`。不是 `vector<vector<vector<float>>>`。**这是内核级内存布局决策**:连续内存 → cache 友好 + 可直接 mmap/序列化。算法书会写「M×ksub×dsub 的三维 centroid 表」,内核里它是一段扁平 buffer。
2. **`transposed_centroids` / `centroids_sq_lengths`**:同一份 centroid 的转置版与预计算平方范数。又是「为了查询时 SIMD / 省算」存的冗余布局——纯算法不需要,引擎需要。
3. **`compute_distance_table` 与 `search` 分离**:距离表(query→256 centroid 的预算)和扫描 code(逐 code 查表累加)是两段。内核可以把「建表」做一次、「扫描」在多个 inverted list 上复用。**算法的「ADC = 建表 + 查表」被拆成内核可调度的两个阶段。**(第 3 章已逐行讲过这两个函数的循环结构,这里只强调它们在内核里是两个可分别调度的 stage。)

### 4.2.3 `InvertedLists`——IVF 的「存储后端」抽象

IVF 把向量按桶归类,「桶里存什么、存哪」就是一个**存储接口**,Faiss 抽象成 `InvertedLists`。这是内核与存储解耦的关键缝。它的 scan 侧抽象是 `InvertedListScanner`:

【真实源码 facebookresearch/faiss@faiss/IndexIVF.h】(WebFetch 取得,签名逐字)
```cpp
size_t scan_codes(
        size_t n,
        const uint8_t* codes,      // ← 一个 inverted list 里的紧凑 code 序列
        const idx_t* ids,          // ← 对应的原始 id
        float* distances,
        idx_t* labels,
        size_t k) const;
```
> 文档注释:此方法 "scan a set of codes, compute distances to current query, and update heap of results if necessary."(WebFetch 取得)

为什么这是内核而非算法:`scan_codes` 不关心 code 是 PQ 编的还是 SQ(标量量化)编的还是 flat 的——它是个**多态扫描器**,由具体 index 类型提供实现。同一套 IVF「选桶 + 遍历 list + 维护 top-k 堆」的骨架(`search_preassigned`),配不同的 `scan_codes`,就得到 IVFFlat / IVFPQ / IVFSQ。**内核 = 稳定的检索骨架 + 可插拔的存储/扫描后端。**

`search_preassigned` 的签名也印证了「内核骨架与粗量化解耦」:

【真实源码 facebookresearch/faiss@faiss/IndexIVF.h】(WebFetch 取得,签名逐字)
```cpp
virtual void search_preassigned(
        idx_t n, const float* x, idx_t k,
        const idx_t* assign,        // ← 桶归属由外部(coarse quantizer)先算好传进来
        const float* centroid_dis,
        float* distances, idx_t* labels,
        bool store_pairs,
        const IVFSearchParameters* params = nullptr,
        IndexIVFStats* stats = nullptr) const = 0;
```
`assign` 是「预分配」——粗量化(选哪些桶)在外面做好传进来。这让「选桶」可以用任意快的方式(甚至 GPU、甚至另一个 HNSW)。`nprobe` 控制选几个桶,定义就在搜索参数里:

【真实源码 facebookresearch/faiss@faiss/IndexIVF.h】(WebFetch 取得,逐字)
```cpp
size_t nprobe = 1;    ///< number of probes at query time
```
> 注意默认 `nprobe = 1`——内核给的默认值极度保守(只扫 1 个桶),这是第 3 章「坑 3:nprobe=1 召回莫名很低」的根因所在。内核默认值 ≠ 生产合理值。

### 4.2.4 小结:一次 IVF-PQ 查询在内核里的数据流

```
query x
  │  ① coarse quantizer.search(x, nprobe)         ← 选 nprobe 个桶(可被替换为 HNSW/GPU)
  ▼
[list_no_1 ... list_no_nprobe]
  │  ② pq.compute_distance_table(x - centroid)    ← 每个桶建一张 query→256 centroid 的表
  ▼
for each list:
  │  ③ scanner.scan_codes(codes, ids, ...)        ← 多态扫描:查表累加 ADC 距离 + 维护堆
  ▼
top-k heap
  │  ④ (可选) Refine: 用原向量精确重排            ← 组合链末端,修 PQ 的排序误差
  ▼
results
```
①②③④ 每一步都是可替换组件。**这就是「内核」相对「算法」的增量:把四个算法阶段缝成一条可配置、可调度、可持久化的流水线。**

## 4.3 可过滤检索:cardinality estimation 驱动的 query planning(向量库 ≠ 向量算法的分水岭)

这是第 3 章完全不碰、却最能体现「内核」的部分。纯 ANN 算法假设「在全集上找近邻」;真实业务永远是「在**满足 payload 条件的子集**上找近邻」——`category = 'shoes' AND price < 100` 的最近邻。这个「向量检索 + 结构化过滤」的缝合,是向量数据库内核的核心难题。

### 4.3.1 三种过滤策略与它们各自的崩法

| 策略 | 做法 | 崩在哪 |
|---|---|---|
| **Post-filter**(先 ANN 后过滤) | HNSW 取 top-k,再丢掉不满足 filter 的 | filter 选择性低时:取了 k 个全被过滤掉,**结果数严重不足甚至为 0** |
| **Pre-filter**(先过滤后暴力) | 先用 payload index 选出满足 filter 的 id 集,再在子集上**暴力**算距离 | 满足 filter 的子集很大时:退化成 O(子集大小) 的暴力,**延迟爆炸** |
| **Filterable / in-graph filter**(图内过滤) | 走 HNSW 图遍历,但**遍历时跳过不满足 filter 的节点** | filter 极度稀疏时:图被过滤得支离破碎,遍历**走不通 / 召回崩**(连通性问题,见 4.3.4) |

关键洞察:**没有一种策略全场景最优**。Post-filter 适合「filter 命中率高」,Pre-filter 适合「filter 命中极少」,Filterable 适合「中间地带」。所以内核必须**先估算 filter 命中多少(cardinality),再选策略**。这就是 query planning。

> 出处(WebSearch / WebFetch 核实):qdrant 官方与 DeepWiki 明确描述其 planner「per-segment 估算 filter cardinality,低于阈值用 payload index 取点(≈pre-filter),高于阈值用 filterable vector index(图内过滤)」。

### 4.3.2 cardinality estimation 的概率数学(qdrant 真实源码)

要在执行前选策略,就得**在不真正执行 filter 的前提下估出它会命中多少点**。qdrant 用 `CardinalityEstimation { min, exp, max, primary_clauses }` 三元区间 + 主子句来表达估计,并对 AND / OR 做概率组合。

【真实源码 qdrant/qdrant@lib/segment/src/index/query_estimator.rs】(WebFetch 取得,关键片段逐字)

AND(`must`,交集)——把各条件当独立事件,概率相乘:
```rust
let exp_estimation_prob: f64 = estimations
    .iter()
    .map(|x| (x.exp as f64) / (total as f64))
    .product();
```
> 即 `P(A∧B) ≈ P(A)·P(B)`,再乘回 `total` 得期望命中数。`min` 用容斥原理累加 `acc + (x as i64) - (total as i64)`(交集下界:全命中要把各集合大小相加再每次减 total 扣重叠);`max` 取各条件的最小值(交集不可能超过最小的那个集合)。

OR(`should`,并集)——用补集法则:
```rust
let element_not_hit_prob: f64 = estimations
    .map(|x| 1.0 - (x as f64 / total as f64))
    .product();
let element_hit_prob = 1.0 - element_not_hit_prob;
```
> 即「命中至少一个」= 1 −「一个都不命中」。`min` 取各条件最大值(并集 ≥ 任一单项),`max` 取各条件之和但封顶 `total`。

内核视角的关键点:
- **为什么是三元区间 `{min, exp, max}` 而非单值**:独立性假设几乎总是错的(`category` 和 `price` 往往相关)。内核**不假装能精确预测**,而是给出区间,让 planner 用 `max`(最坏情况)做保守决策(比如「即使最坏,命中也少于阈值 → 放心用 pre-filter」)。这是**工程上的诚实**——把不确定性显式建模,而不是给个假精确的单值。
- **`primary_clauses`**:能被 payload index 直接加速的子句(如有索引的等值条件)。planner 优先用它们做主过滤,其余子句做 post-check。这是「估计」与「执行计划」一体两面。

### 4.3.3 strategy 路由:cardinality → 选哪条路(qdrant)

估出 cardinality 后,planner 在 4.3.1 三策略间路由。qdrant 把决策拆在几个文件里,核心规则(WebSearch / WebFetch 核实)是:

```
estimate cardinality(filter) -> { min, exp, max }
  │
  ├─ 若 max < full_scan_threshold(以 KB 折算成点数)
  │      → 命中很少,直接 pre-filter:取出这批点暴力算距离
  │        (源码里对应 search_vectors_plain / search_plain_batched)
  │
  └─ 否则(命中较多)
         → filterable HNSW:走图遍历,遍历时跳过不满足 filter 的节点
           (源码里对应 search_with_graph,带 filter context)
```

在「filterable HNSW」内部,qdrant 还会按**选择性**进一步选算法(ACORN 与否):

【真实源码 qdrant/qdrant@lib/segment/src/index/hnsw_index/hnsw/search.rs】(WebFetch 取得,关键判定逐字,变量略简)
```rust
let selectivity = query_cardinality.exp as f64 / available_vector_count as f64;
if selectivity <= acorn_max_selectivity {
    algorithm = SearchAlgorithm::Acorn;
}
```
> `selectivity = 期望命中数 / 可用向量总数`。当选择性足够低(filter 很挑剔),切到 **ACORN** 算法。ACORN 是为「低选择性过滤」设计的图遍历变体(遍历时对邻居做 filter-aware 扩展,缓解图被过滤打散的问题)。这是「内核里套内核」:在 filterable 这条路里再按 cardinality 二次路由。

> ⚠ 待核:`full_scan_threshold` 与 ACORN 选择性阈值的**具体默认数值**及完整 if/else 边界,散落在 `hnsw.rs` / `search.rs` / 配置结构多个文件,WebFetch 单文件只取到了上述判定片段与阈值的存在性(KB→点数的换算 `full_scan_threshold.saturating_mul(BYTES_IN_KB).checked_div(avg_vector_size)`),完整决策树以源码为准。

### 4.3.4 filterable HNSW 的连通性补丁(为什么图内过滤不会散架)

图内过滤有个先天风险:HNSW 图的边是「全集上的近邻关系」建的,一旦按 filter 只保留子集节点,**子集内部可能不连通**——遍历从入口出发,可能到不了某些满足 filter 的孤岛,召回崩。

qdrant 的工程解法(WebSearch 核实,qdrant 官方课程/文章描述):**在建图时,对有 payload index 的字段,按字段值建子图并保证子图连通,再合并回全图**。于是「filter `brand = Apple`」时,Apple 这批点之间已经预建了一张连通子图,遍历在子集内走得通。代价是额外的边和建图开销——**用存储/建图成本换过滤查询的连通性保证**。这是典型的内核级权衡:把「查询时的连通性风险」前移成「建索引时的额外结构」。

> 对比 Faiss:Faiss 主线对「带 filter 的 ANN」支持薄(`IDSelector` 做 id 级过滤,本质偏 pre/post-filter,无 cardinality-driven planner)。**「把过滤当一等公民、用 cardinality 做 query planning」正是向量数据库(qdrant / Weaviate / Milvus)相对纯算法库(Faiss / hnswlib)的内核增量。** 这条线第 3 章完全没有,是本章的核心差异化。

## 4.4 可变性:向量库作为「存储」必须答的题

纯算法实现默认「一次建好、只读查询」。当向量库要当线上存储,**增删改 + 持久化**就成了硬约束。这部分纯算法层几乎不涉及。

### 4.4.1 删除为什么难——以 HNSW 为例

HNSW 的边是「全局近邻关系」织出来的网。删一个节点意味着:它作为别人邻居的那些**反向边**全要修补,否则图出洞、遍历断路。真实工程里几乎没人做「真删 + 实时补图」,而是:
- **Tombstone(软删)**:打删除标记,遍历时跳过,但节点和边仍占内存,只是结果里过滤掉。代价:删多了图里全是「幽灵节点」,遍历效率与召回都退化。
- **定期 rebuild / compaction**:软删积累到阈值,后台重建索引,真正回收。代价:重建期 CPU/内存翻倍,且这是第 3 章「坑 2:efConstruction 先天不足事后补不了」的二次现场——重建是唯一能改图质量的时机。

IVF-PQ 相对友好:删除只需从对应 inverted list 移除 code(list 是可变长结构),不涉及图修补。**这也是选型时常被忽视的维度:写/删频繁的场景,IVF 家族的可变性优于 HNSW。**

### 4.4.2 segment 模型——把「可变」化简为「不可变片段 + 合并」

qdrant / Lucene 系的通用解法:**LSM 式 segment**。新写入进可变 segment(小、可改),后台把多个 segment 合并(merge / optimize)成更大的不可变 segment(建好 HNSW、压缩存储)。查询时**对所有 segment 各自查 + 合并结果**。

这把「在一个大可变索引上原地增删改」这个难题,化简成「小范围可变 + 批量不可变重建」。代价是查询要 fan-out 到多 segment(4.3 的 query planning 因此是 **per-segment** 的——前面引文反复强调「planning per segment」就是这个原因)。**可变性、过滤、查询计划在 segment 模型下被统一组织。**

### 4.4.3 持久化与 mmap——索引落盘的两种姿态

- **全内存 + 序列化快照**:Faiss `write_index` / `read_index` 把整个索引 dump 成一个文件,启动时全量 load 进 RAM。简单,但大索引启动慢、内存即数据集大小。
- **on-disk mmap**:把索引文件 mmap,按需 page-in。qdrant 的 `on_disk_mmap` / `ram_from_mmap`(4.2 提到的 `LoadOption`)就是这条路——**热数据在 page cache,冷数据躺盘上**。代价:首次访问有 page fault 延迟,随机访问模式对 HNSW(本就随机跳点)尤其敏感。这又呼应 4.2.2:**centroid / code 用连续扁平 buffer 存,正是为了 mmap 友好**——内存布局决策在这里收到回报。

## 4.5 列式磁盘原生:Lance 代表的另一条内核路线

Faiss(算法库)和 qdrant(向量 DB)之外,还有第三条内核思路:**把向量当列式数据,索引作为列上的二级结构,磁盘原生而非内存原生**——Lance / LanceDB 代表。

设计动机(WebSearch 核实,Lance 公开资料):向量数据常和大量标量列(元数据、原文、特征)一起存,且数据集超内存。Lance 的取舍:
- **列式存储(类 Parquet 但为随机访问优化)**:向量列与标量列同表存储,过滤(标量列)与向量检索(向量列索引)在同一存储引擎内,天然适合 4.3 的「向量 + 结构化过滤」。
- **磁盘原生 + 零拷贝**:不要求索引全进内存,大数据集友好;代价是延迟比纯内存 HNSW 高。
- **版本化 / 可变**:列式 + manifest 让增删改和数据版本更自然(对比 HNSW 原地删的痛)。

把它和前两者并列,是为了点明**「向量库内核」不是单一形态**:

| 内核形态 | 代表 | 内存模型 | 过滤 | 可变性 | 最适场景 |
|---|---|---|---|---|---|
| 算法库 | Faiss / hnswlib | 内存原生 | 弱(IDSelector) | 弱(偏静态) | 离线/嵌入式、极致单机召回-延迟 |
| 向量 DB | qdrant / Milvus / Weaviate | 内存为主 + mmap | **强(cardinality planner)** | 中(segment) | 在线服务、带复杂过滤的实时检索 |
| 列式磁盘原生 | Lance / LanceDB | **磁盘原生** | 强(同表标量列) | 强(版本化) | 超内存数据集、与数仓/特征表同居 |

> ⚠ 待核:Lance 的具体索引实现细节(其 IVF-PQ / HNSW 变体的磁盘布局、re-rank 策略)未做 WebFetch 源码精读,以上为架构层定性,数值/实现以官方为准。本章把 Faiss / qdrant 做了源码级,Lance 仅作形态对照。

## 4.6 方案对比:同一份数据,内核选型怎么落地

把第 3 章的算法对比上升到「内核组合」层面——实际选型是选**一条组合链 + 一种内核形态**,不是选单个算法。

### 4.6.1 索引组合配方对照表

| 组合(Faiss factory 写法) | 内存/向量 | 召回 | 延迟 | 适用 | 边界/坑 |
|---|---|---|---|---|---|
| `Flat`(IndexFlatL2) | 4·D 字节(原向量) | 100% | 高(O(N)) | N 小(≲10⁵)、要 ground truth | N 大直接 OOM + 慢 |
| `HNSW32,Flat` | 原向量 + 图边(M·8 字节级) | 高 | 低 | 内存够、读多写少、要低延迟 | 内存大;删除难;建图慢 |
| `IVF4096,Flat` | 原向量 + 倒排结构 | 中-高(随 nprobe) | 中 | 可变、删除友好 | nprobe 默认 1 召回崩(坑 3) |
| `IVF4096,PQ32` | **M 字节(≈32B)** | 中 | 低 | **大规模 + 内存敏感** | PQ 排序误差大,常需 Refine |
| `OPQ32,IVF4096,PQ32,RFlat` | M 字节 code + 原向量(Refine) | 中-高 | 中 | 大规模 + 要回召回 | Refine 要存原向量,省内存打折 |
| qdrant filterable HNSW | 原向量 + 图 + payload index | 高 | 低(带过滤仍快) | **在线 + 复杂 filter** | payload index 占额外空间;连通性补丁建图更慢 |

### 4.6.2 按场景定位

- **离线批处理、N≤10⁵、要算召回基线** → `Flat`,别上索引(坑 7:小数据上 ANN 的开销 > 收益)。
- **单机、内存够、读多写少、极致低延迟** → `HNSW,Flat`。接受删除难。
- **十亿级、内存是硬约束** → `OPQ,IVF,PQ,RFlat`。用 4.7 demo 实测压缩比/召回,Refine 修排序。
- **在线服务、查询永远带 `category/price/tenant` 过滤** → 上**向量 DB**(qdrant 等),要的是 4.3 的 cardinality planner,不是更快的 HNSW。**这是「为什么不直接用 Faiss」的标准答案。**
- **数据超内存、向量与大量元数据/原文同居、要版本化** → **列式磁盘原生**(Lance 等)。

### 4.6.3 容易忽视的不适用边界

- **多租户(multi-tenancy)**:把 `tenant_id` 当普通 filter 走 4.3 路由,在租户极多时 cardinality 估计与图连通性都吃力。生产里常**按 tenant 分 collection / 分 segment 物理隔离**,而非纯靠 filter——这是内核选型里最常翻车的隐藏维度。
- **高频更新 + 高召回要求并存**:HNSW 给不了(删除难 + 重建才能保图质量),IVF-PQ 召回不够,往往得 segment 模型 + 定期 compaction 折中,接受「召回随软删漂移、compaction 后回血」的锯齿。
- **filter 与向量分布强相关**:cardinality 的独立性假设(4.3.2)失真严重,planner 可能频繁选错策略。表现为「某些 filter 组合延迟莫名飙高」。

## 4.7 可运行 Demo

> 环境:`python3` + `numpy`。**核心 demo 不需要 faiss**(手写 toy 印证内核机制)。Demo 1 末尾给一段**可选** faiss 对照(缺 faiss 自动跳过)。
> 两个 demo 均已在本机实测跑通(numpy 2.4.4),下方「预期输出」是真实运行结果(kmeans 随机初始化用了固定 seed,但浮点细节可能因 numpy/BLAS 版本有末位差异,趋势稳定)。

### Demo 1:手写 IVF-PQ 内核 toy —— 印证 `IndexIVFPQ` 的两段式 + 压缩比/召回拆解(对应 4.2)

**这个 demo 的内核意义**:不是讲 PQ 算法(第 3 章讲过),而是**复刻 `IndexIVFPQ` 的内核数据流**(4.2.4 那张图):coarse 选桶 → 残差 → PQ 编码成紧凑 code → 查询时建距离表 + 多态扫描 ADC。并**拆解两个误差源**(IVF 剪枝 vs PQ 近似),最后用 Refine(对应 factory 的 `RFlat`)修排序——印证 4.6.1 表里「为什么 IVF-PQ 常要挂 Refine」。

【可运行 demo,纯 numpy。保存为 `ivfpq_toy.py`】
```python
"""手写 IVF-PQ toy:印证 Faiss IndexIVFPQ 的两段式内核结构。
  Coarse quantizer (IVF, nlist 个 cell) + Product Quantizer (M 段子量化器)。
依赖:numpy。"""
import numpy as np

rng = np.random.default_rng(42)

# ---------- 0. 造数据:10 个高斯簇,刻意聚类,便于看出 IVF 召回 ----------
D, N, NQ = 64, 20000, 200
n_clusters = 10
centers = rng.normal(0, 8, size=(n_clusters, D)).astype(np.float32)
labels = rng.integers(0, n_clusters, size=N)
xb = (centers[labels] + rng.normal(0, 1, size=(N, D))).astype(np.float32)
qlabels = rng.integers(0, n_clusters, size=NQ)
xq = (centers[qlabels] + rng.normal(0, 1, size=(NQ, D))).astype(np.float32)

def l2sqr(a, b):  # a:(n,d) b:(m,d) -> (n,m)
    return ((a[:, None, :] - b[None, :, :]) ** 2).sum(-1)

# ---------- 1. brute-force ground truth (用于算 recall) ----------
def topk_bruteforce(q, base, k):
    return np.argsort(l2sqr(q, base), axis=1)[:, :k]
K = 10
gt = topk_bruteforce(xq, xb, K)

# ---------- 2. 一个极简 kmeans (Lloyd),IVF coarse + PQ subquantizer 都用它 ----------
def kmeans(x, n_centroids, iters=15):
    idx = rng.choice(len(x), n_centroids, replace=False)
    cent = x[idx].copy()
    for _ in range(iters):
        assign = np.argmin(l2sqr(x, cent), axis=1)
        for c in range(n_centroids):
            pts = x[assign == c]
            if len(pts):
                cent[c] = pts.mean(0)
    return cent

# ---------- 3. 训练 coarse quantizer:nlist 个 cell (对应 Faiss 的 quantizer) ----------
NLIST = 64
coarse = kmeans(xb, NLIST)                       # (NLIST, D)
list_no = np.argmin(l2sqr(xb, coarse), axis=1)   # 每个 base 向量落在哪个 cell

# ---------- 4. 训练 PQ:对 residual 做 M 段子量化 (Faiss by_residual=True) ----------
M, NBITS = 8, 8           # M 段,每段 256 (=2^8) 个 centroid
KSUB = 1 << NBITS
DSUB = D // M             # 每段子向量维度
residual = xb - coarse[list_no]          # 关键:量化 residual 而非原始向量
pq_centroids = np.zeros((M, KSUB, DSUB), dtype=np.float32)
codes = np.zeros((N, M), dtype=np.uint8) # 每个向量 M 个 uint8 -> code_size = M 字节
for m in range(M):
    sub = residual[:, m*DSUB:(m+1)*DSUB]
    cm = kmeans(sub, KSUB, iters=10)
    pq_centroids[m] = cm
    codes[:, m] = np.argmin(l2sqr(sub, cm), axis=1).astype(np.uint8)

# ---------- 5. 查询:nprobe 个 cell + ADC (Asymmetric Distance Computation) ----------
def search(q, nprobe, k):
    out = np.zeros((len(q), k), dtype=np.int64)
    # 5a. coarse:选 nprobe 个最近 cell —— 这就是 IVF 的 non-exhaustive 关键
    qcell = np.argsort(l2sqr(q, coarse), axis=1)[:, :nprobe]
    for i in range(len(q)):
        cand_ids, cand_dist = [], []
        for cell in qcell[i]:
            members = np.where(list_no == cell)[0]
            if len(members) == 0:
                continue
            # 5b. 对该 cell 建 distance table:query residual 的每段 vs 256 centroid
            qres = q[i] - coarse[cell]
            dt = np.zeros((M, KSUB), dtype=np.float32)
            for m in range(M):
                sub = qres[m*DSUB:(m+1)*DSUB]
                dt[m] = ((pq_centroids[m] - sub) ** 2).sum(1)   # 预算 256 个距离
            # 5c. ADC:用查表累加近似距离,不解码原向量
            c = codes[members]                                   # (n_members, M)
            approx = dt[np.arange(M), c].sum(1)                  # 查表 + 求和
            cand_ids.append(members); cand_dist.append(approx)
        if not cand_ids:
            continue
        cand_ids = np.concatenate(cand_ids); cand_dist = np.concatenate(cand_dist)
        sel = np.argsort(cand_dist)[:k]
        out[i] = cand_ids[sel]
    return out

def recall_at_k(pred, truth, k):
    hit = sum(len(set(pred[i]) & set(truth[i, :k])) for i in range(len(pred)))
    return hit / (len(pred) * k)

print(f"D={D} N={N} | NLIST={NLIST} | PQ M={M} nbits={NBITS} (KSUB={KSUB} DSUB={DSUB})")
raw_bytes = N * D * 4
pq_bytes  = N * M           # M 字节/向量
print(f"compression: raw {raw_bytes/1e6:.1f}MB -> PQ codes {pq_bytes/1e6:.3f}MB "
      f"(+coarse/centroid tables), ratio ~{raw_bytes/pq_bytes:.0f}x on codes")
for nprobe in (1, 4, 8, 16, 32):
    pred = search(xq, nprobe, K)
    print(f"nprobe={nprobe:>2}  recall@{K} = {recall_at_k(pred, gt, K):.3f}")

# ---------- 6. 拆解误差来源:IVF-only(无 PQ 近似)vs IVF-PQ ----------
def search_ivf_exact(q, nprobe, k):
    """只做 IVF 剪枝,cell 内用精确 L2(不走 PQ)。隔离 IVF 的召回损失。"""
    out = np.zeros((len(q), k), dtype=np.int64)
    qcell = np.argsort(l2sqr(q, coarse), axis=1)[:, :nprobe]
    for i in range(len(q)):
        members = np.where(np.isin(list_no, qcell[i]))[0]
        if len(members) == 0:
            continue
        d = l2sqr(q[i:i+1], xb[members])[0]
        sel = np.argsort(d)[:k]
        out[i] = members[sel]
    return out

print("\n-- 误差拆解:IVF-only(cell 内精确)隔离 coarse 剪枝损失 --")
for nprobe in (1, 4, 8, 16):
    pred = search_ivf_exact(xq, nprobe, K)
    print(f"nprobe={nprobe:>2}  IVF-only recall@{K} = {recall_at_k(pred, gt, K):.3f}")

# ---------- 7. 生产常用补丁:PQ 出 shortlist -> 原向量精确 rerank ----------
def search_rerank(q, nprobe, k, rerank=100):
    """IVF-PQ 取 top-`rerank` 候选,再用原始向量精确重排取 top-k。
       对应 Faiss IndexRefineFlat / factory 里的 RFlat。"""
    shortlist = search(q, nprobe, rerank)        # 复用 §5 的 IVF-PQ
    out = np.zeros((len(q), k), dtype=np.int64)
    for i in range(len(q)):
        cand = shortlist[i]
        d = l2sqr(q[i:i+1], xb[cand])[0]         # 精确距离重排
        out[i] = cand[np.argsort(d)[:k]]
    return out

print("\n-- 生产补丁:IVF-PQ shortlist(rerank=100)+ 原向量精确重排 --")
for nprobe in (4, 8, 16):
    pred = search_rerank(xq, nprobe, K, rerank=100)
    print(f"nprobe={nprobe:>2}  IVF-PQ+rerank recall@{K} = {recall_at_k(pred, gt, K):.3f}")

# ---------- 8. (可选) faiss 对照:缺 faiss 自动跳过 ----------
try:
    import faiss
    quantizer = faiss.IndexFlatL2(D)
    index = faiss.IndexIVFPQ(quantizer, D, NLIST, M, NBITS)
    index.train(xb); index.add(xb)
    for nprobe in (4, 8, 16):
        index.nprobe = nprobe
        _, I = index.search(xq, K)
        print(f"[faiss] nprobe={nprobe:>2}  recall@{K} = {recall_at_k(I, gt, K):.3f}")
except ImportError:
    print("\n[faiss 未安装,跳过对照。装: pip install faiss-cpu]")
```

1. **压缩比 ~32×**:5.1MB 原向量 → 0.16MB code(D·4=256B → M=8B per vec)。这是 4.2.2「centroid 扁平 buffer + uint8 code」存储布局的直接收益,也是 IVF-PQ 上十亿规模的根本原因。
2. **误差拆解(本 demo 最有价值的部分)**:IVF-only 在 nprobe=16 时 recall=**1.000**,而 IVF-PQ 死死卡在 **0.29**。结论硬核——**召回瓶颈不在 IVF 剪枝,而在 PQ 近似**(DSUB=8 的子向量量化太糙)。这解释了 4.6.1 表里「IVF-PQ 召回:中」「常需 Refine」的来由:**别一看召回低就加大 nprobe,要先拆清楚误差源**(对应第 3 章「坑」的内核级延伸)。
3. **Refine 修排序**:rerank=100 把 recall 从 0.29 拉到 **0.82**。机理:PQ 作为「粗筛」够格(真近邻大概率落进 top-100 shortlist),作为「精排」不够格(top-100 内部 PQ 距离排序乱)。用原向量重排 shortlist 即修复——这就是 factory 里 `RFlat` / `IndexRefineFlat` 存在的工程理由。**这是「内核组合链末端为什么挂 Refine」的实证。**

> 关于召回绝对值:toy 的 kmeans 是裸 Lloyd(无 OPQ、无多次重启、迭代少),且 recall@10-vs-exact-top10 是严苛指标,故数值低于生产 faiss。**本 demo 的价值在「拆解三段误差结构 + 印证组合链」,不在刷高 recall**(对应第 3 章「坑 7:别拿 toy 性能当真」)。装上 faiss 跑第 8 段可见工程实现召回显著更高。

### Demo 2:filterable-HNSW 内核 toy —— 印证 cardinality → strategy 路由(对应 4.3)

**这个 demo 的内核意义**:这是第 3 章完全没有的东西。它复刻 4.3 的 query planner——**先估 cardinality,再在 pre-filter / filterable-graph / post-filter 间路由**,并实测三种策略各自的崩法(post-filter 结果不足、pre-filter 慢、filterable 在低选择性下召回掉),证明「为什么内核必须按 cardinality 选策略」。

【可运行 demo,纯 numpy。保存为 `filterable_toy.py`】
```python
"""filterable-HNSW toy:印证 qdrant 的 cardinality estimation -> 策略路由。
对比 post-filter / pre-filter / filterable-graph 三策略在不同 filter 选择性下的表现。
依赖:numpy。"""
import numpy as np, time
rng = np.random.default_rng(7)

D, N, NQ = 32, 12000, 100
# 造 8 个高斯簇,让近邻图有结构(纯随机点近邻几乎随机,图搜索学不到东西)
nclu = 8
ctr = rng.normal(0, 6, size=(nclu, D)).astype(np.float32)
clab = rng.integers(0, nclu, size=N)
xb = (ctr[clab] + rng.normal(0, 1, size=(N, D))).astype(np.float32)
xq = (ctr[rng.integers(0, nclu, size=NQ)] + rng.normal(0, 1, size=(NQ, D))).astype(np.float32)
# payload: category in 0..(NCAT-1)。filter = "category == c",选择性 ≈ 1/NCAT
NCAT = 10
category = rng.integers(0, NCAT, size=N)

def l2_1(q, base):           # q:(D,) base:(m,D) -> (m,)  单查询到一批,返回一维距离
    return ((base - q) ** 2).sum(1)
K = 10

# ---------- 1. 建简化 HNSW 单层近邻图(全集),每点 M 条最近边 ----------
M = 16
def build_graph():
    adj = [[] for _ in range(N)]
    for i in range(N):
        d = l2_1(xb[i], xb); d[i] = np.inf
        for nb in np.argsort(d)[:M]:
            adj[i].append(int(nb)); adj[int(nb)].append(i)
    return [list(dict.fromkeys(a)) for a in adj]
adj = build_graph()

# ---------- 2. cardinality estimation(qdrant 思路:命中率 * 总数)----------
def estimate_cardinality(cat):
    return float((category == cat).mean()) * N      # 期望命中数

# ---------- 3. 图搜索:多入口贪心 beam(单入口在单层稀疏图上召回崩)----------
N_ENTRY = 8
entries = rng.choice(N, N_ENTRY, replace=False)
def search_graph(q, k, filt_mask=None, ef=64):
    import heapq
    visited, cand, top = set(), [], []
    for e in entries:                                # 多入口缓解单层图连通性差
        if e in visited: continue
        visited.add(int(e)); de = float(l2_1(q, xb[e:e+1])[0])
        heapq.heappush(cand, (de, int(e)))
        if filt_mask is None or filt_mask[e]:
            heapq.heappush(top, (-de, int(e)))
    while cand:
        cd, c = heapq.heappop(cand)
        if top and -top[0][0] < cd and len(top) >= ef: break
        for nb in adj[c]:
            if nb in visited: continue
            visited.add(nb); dn = float(l2_1(q, xb[nb:nb+1])[0])
            heapq.heappush(cand, (dn, nb))           # 被 filter 掉的点仍可中转扩展(保连通)
            if filt_mask is None or filt_mask[nb]:   # 只有满足 filter 才进结果堆
                if len(top) < ef or dn < -top[0][0]:
                    heapq.heappush(top, (-dn, nb))
                    if len(top) > ef: heapq.heappop(top)
    return [i for _, i in sorted([(-x, i) for x, i in top])[:k]]

# ---------- 4. 三种过滤策略 ----------
def post_filter(q, k, cat, over=1):
    """先无过滤 HNSW 取 k*over,再丢掉不满足 filter 的。低选择性会结果不足。"""
    cand = search_graph(q, k*over, filt_mask=None)
    return [i for i in cand if category[i] == cat][:k]

def pre_filter(q, k, cat):
    """先选出满足 filter 的 id,再暴力算距离。高命中数会很慢。"""
    ids = np.where(category == cat)[0]
    if len(ids) == 0: return []
    d = l2_1(q, xb[ids])                              # 子集内暴力
    return list(ids[np.argsort(d)[:k]])

def filterable(q, k, cat):
    """图内过滤:遍历时跳过不满足 filter 的点(但允许中转)。"""
    return search_graph(q, k, filt_mask=(category == cat))

# ---------- 5. ground truth(子集内精确 top-k)----------
def gt_filtered(qi, cat, k):
    ids = np.where(category == cat)[0]
    d = l2_1(xq[qi], xb[ids])
    return set(ids[np.argsort(d)[:k]])

def eval_strategy(fn):
    rec, miss, t0 = 0.0, 0, time.time()
    for qi in range(NQ):
        cat = qi % NCAT
        truth = gt_filtered(qi, cat, K)
        pred = set(fn(xq[qi], K, cat))
        if len(pred) < K: miss += 1        # 结果数不足的查询数
        rec += len(pred & truth) / K
    return rec/NQ, miss, time.time()-t0

# ---------- 6. 跑对照 ----------
print(f"N={N} categories={NCAT} (每类~{N//NCAT} 点, filter 选择性~{100//NCAT}%)")
print(f"示例 cardinality 估计: category==0 -> exp≈{estimate_cardinality(0):.0f} 点\n")
print(f"{'strategy':20s} {'recall@10':>10s} {'结果不足查询数':>16s} {'耗时s':>8s}")
for name, fn in (("post-filter(x1)", lambda q,k,c: post_filter(q,k,c,over=1)),
                 ("post-filter(x10)", lambda q,k,c: post_filter(q,k,c,over=10)),
                 ("pre-filter(暴力)", pre_filter),
                 ("filterable-graph", filterable)):
    r, miss, t = eval_strategy(fn)
    print(f"{name:20s} {r:>10.3f} {miss:>16d} {t:>8.2f}")
```

**运行步骤**:
```bash
python3 filterable_toy.py
```

**预期输出(本机实测,numpy 2.4.4,约 15s)**:
```
N=12000 categories=10 (每类~1200 点, filter 选择性~10%)
示例 cardinality 估计: category==0 -> exp≈1232 点

strategy              recall@10          结果不足查询数      耗时s
post-filter(x1)           0.074              100     0.26
post-filter(x10)          0.493               94     0.26
pre-filter(暴力)            1.000                0     0.02
filterable-graph          0.870                0     0.42
```
> (kmeans 之外的随机量已固定 seed;浮点末位与耗时随 numpy/BLAS 版本、机器负载有波动,趋势稳定。)**稳定结论**:
> - **post-filter(x1) 灾难性低召回 + 全部「结果不足」**(0.074,100/100 不足):取 10 个里平均只 ~1 个满足 10% 的 filter → 结果数严重不足、recall 崩。这印证 4.3.1 的 post-filter 崩法。
> - **post-filter 加大 over 系数能缓解但治标**:over=10 把召回从 0.074 拉到 0.493,代价是取了 10 倍候选,且仍有 94/100 不足 k 个(选择性更低时要 100×、1000×,迅速退化)。
> - **pre-filter recall=1.000、最快**:子集内暴力必然精确;此处 category==c 仅 ~1200 点,暴力反而最快(0.02s)。**但若 filter 命中 50 万点,这一列耗时会爆**——这正是 planner 要用 cardinality 卡 `full_scan_threshold`(命中过多就别走暴力)的原因。
> - **filterable-graph 召回高(0.870)且不「结果不足」**:图内过滤兼顾召回与延迟,是「中等选择性」的甜点区。

**这个 demo 印证了什么(内核级,第 3 章没有)**:
1. **没有一种策略全场景赢**——post-filter 在低选择性崩(0.074 + 全部结果不足)、pre-filter 精确但命中多时会慢、filterable 居中偏高(0.870)。**这就是 4.3 内核必须先估 cardinality 再路由的根本理由**:这里 `exp≈1232`(10% 选择性)落在「filterable 甜点 / pre-filter 也还能扛」的地带;若 `exp` 极小(如 0.1%,子集只剩几十个点),planner 应切 pre-filter(暴力极快且精确);若 `exp` 极大(filter 几乎不挑),应切近全量 / post-filter(over 系数小也够)。
2. **cardinality 是「决策输入」而非「结果」**:demo 里 `estimate_cardinality` 在执行前就给出 exp,planner 据此选 fn——和 4.3.2 qdrant `CardinalityEstimation` 的用法同构(估计先行,执行在后)。把它接成自动 planner 就是 4.9 的代码题 2。
3. **filterable-graph 的「中转」细节是连通性命门**:代码里被 filter 掉的点**仍能作为中转继续扩展邻居**(被跳过的点照样 `push` 进 `cand` 继续走,只有结果堆 `top` 受 `filt_mask` 限制)。若不这么做,图会被过滤打散、遍历断路——这正是 4.3.4 qdrant「建连通子图」要解决的同一个问题的 toy 版演示。demo 还用了 8 个随机入口(`N_ENTRY`)来缓解单层稀疏图的连通性差,这对应真实 HNSW 用上层做多入口下降的作用。

## 4.8 失败模式与真坑(内核层,区别于第 3 章的算法坑)

> 第 3 章的坑是「算法参数」层(efSearch/nprobe/m 整除等)。这里是「内核工程」层——它们在算法都对的前提下照样让你线上翻车。

### 坑 A:post-filter 在低选择性下静默返回不足 k 个结果
**现象**:加了个 `status='active'` 过滤,搜 10 条结果只回来 2 条,甚至 0 条。
**根因**:用了 post-filter(先 ANN top-k 再过滤),filter 命中率低 → top-k 里大半被滤掉。Demo 2 的 `post-filter(x1)` recall 0.092 就是现场。
**药**:用支持 cardinality-driven planning 的内核(qdrant 等),或手动放大 over-fetch 系数并监控「结果不足率」。**纯算法库(Faiss IDSelector)默认就是这个坑,要自己兜。**

### 坑 B:pre-filter 在高命中 filter 下延迟突刺
**现象**:大部分查询 5ms,带某个「命中 60% 数据」的 filter 时飙到 800ms。
**根因**:planner 选了 pre-filter,但该 filter 命中海量点 → 退化成大规模暴力(Demo 2 pre-filter 列若命中数大就爆)。
**药**:确认内核的 `full_scan_threshold` 配置合理(命中超阈值应转 filterable-graph 而非暴力);警惕 cardinality 估计因独立性假设失真(4.6.3)而误判。

### 坑 C:把 PQ/IVF-PQ 召回低归咎于 nprobe,实则是 PQ 量化误差
**现象**:狂调大 nprobe,召回从 0.29 爬到 0.30 就不动了。
**根因**:瓶颈在 PQ 近似不在 IVF 剪枝(Demo 1 误差拆解:IVF-only=1.000 vs IVF-PQ=0.29)。
**药**:挂 Refine(`RFlat`)做精排(Demo 1:0.29→0.82),或上 OPQ 降量化误差,或加大 M/nbits。**先拆误差源再调参**,别在错误的旋钮上使劲。

### 坑 D:HNSW 当可变存储用,软删积累导致召回与延迟双退化
**现象**:跑几个月后同样 query 召回变差、延迟变高。
**根因**:HNSW 删除是 tombstone,幽灵节点占图、遍历空转(4.4.1)。
**药**:配置 segment compaction / 定期 rebuild 阈值;高频删场景重新评估是否该用 IVF 家族(删除友好)而非 HNSW。

### 坑 E:per-segment query planning 下,单 segment 看着对、整体召回不对
**现象**:小数据测试召回正常,上量后召回掉。
**根因**:segment 模型下 planning 是 per-segment 的(4.4.2),每个 segment 各自选策略、各自 top-k 再合并;若某 segment 选了 post-filter 且选择性低,它贡献的候选就不足,拖累全局。
**药**:理解你的内核是不是 per-segment 决策;监控按 segment 拆分的召回;必要时调 segment 大小 / merge 策略。

### 坑 F:mmap 索引在随机访问负载下首查延迟抖动
**现象**:冷启动或冷数据查询偶发高延迟。
**根因**:on-disk mmap 首次访问 page fault(4.4.3),HNSW 随机跳点放大了 page-in 次数。
**药**:预热(warmup 查询触发 page-in)、给热数据留足 page cache、或对延迟敏感场景用全内存模式。

### 二、三道代码题(扩展 Demo)
1. **给 Demo 1 加 OPQ**:在 PQ 编码前对每个子空间学一个旋转矩阵(可用全局 PCA 近似),对比加 OPQ 前后 IVF-PQ 的 recall。预期:OPQ 让子空间方差更均衡,recall 上升。
2. **给 Demo 2 实现一个真·planner**:写一个 `plan(cat)` 函数,根据 `estimate_cardinality(cat)` 与一个 `full_scan_threshold`,自动在 `pre_filter` / `filterable` / `post_filter` 间选 fn;扫不同选择性(改 category 数从 2 到 1000),画出「planner 自动选择 vs 各固定策略」的 recall/耗时曲线,验证 planner 是各策略的上包络。
3. **Demo 1 加 segment**:把 N 拆成 4 个 segment,各自建 IVF-PQ,查询时各取 top-k 再 merge;对比单 segment vs 4-segment 的召回,观察 fan-out 合并的损耗与 per-segment nprobe 该怎么配。

### 三、一个开放问题
cardinality estimation 的独立性假设(4.3.2)在真实业务里经常失真(filter 字段与向量分布相关)。**能否用「向量空间 + payload 的联合统计」做更准的 cardinality 估计,而不退回全量扫描?** 这关系到下一代向量 DB 能否在「filter 与语义强相关」场景下不翻车——目前主流内核都还停在「假设独立 + min/max 区间兜底」。

### 四、本章踩坑红线(速记)
- 带 filter 的检索,**先想清楚 filter 选择性**——低选择性别用 post-filter,高命中别用 pre-filter。
- 召回低先**拆误差源**(IVF 剪枝 vs PQ 近似),别只会调 nprobe。
- IVF-PQ 召回不够,**挂 Refine** 往往比调任何参数都立竿见影。
- HNSW **不是为高频删除设计的**;写删频繁优先 IVF 家族 / segment compaction。
- 选型选的是**「组合链 + 内核形态」**,不是单个算法;复杂 filter → 向量 DB,超内存 → 列式磁盘原生。
- 内核**默认值通常极保守**(nprobe=1),生产前必须按数据重调。
