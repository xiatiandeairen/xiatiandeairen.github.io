## TL;DR

| 阶段 | 一句话 |
|------|--------|
| Tree-walking | 直接对 AST 递归求值；零编译延迟，但每步都要走指针 |
| 字节码编译 | 一次遍历 AST 发射线性字节码；把"选哪条路"变成数组下标 |
| 栈式 VM | 操作数在显式栈上；指令编码紧凑，dispatch 是性能关键 |
| 寄存器式 VM | Lua 5/Android Dalvik/WebAssembly；减少 push/pop，但指令更宽 |
| Dispatch 优化 | switch→computed goto→threaded code→inline cache(IC) |
| NaN boxing | 把指针/bool/nil 塞进 IEEE 754 NaN payload；省 8 字节/值 |
| Upvalue/Closure | 把栈变量"关闭"到堆，支持 first-class 函数 |

## 0. 设计考古：从直接求值到字节码 VM

### 0.1 最早的解释器：直接对源码求值

1958 年 LISP 1.5 的 `eval` 就是一个 tree-walker——源码读进来即是 S-expression，解释器直接对树递归。John McCarthy 的原始论文 *Recursive Functions of Symbolic Expressions and Their Computation by Machine* (1960) 描述了这个模型。

优点极致简单；缺点是：树节点分散在堆上，每个操作都要解引用指针、做多态 dispatch，cache miss 严重。

### 0.2 字节码 VM 的出现动机

P-code machine（1970s，Pascal），UCSD Pascal，FORTH 的 ITC（Indirect Threaded Code）都在探索同一个问题：**能不能把程序表示成一个紧凑的线性数组，让 CPU 顺序扫描，而不是追指针？**

关键论文：
- Peter Deutsch & Allan Schiffman, *Efficient Implementation of the Smalltalk-80 System* (POPL 1984)——首次系统论述 inline cache。
- Urs Hölzle, Craig Chambers, David Ungar, *Optimizing Dynamically-Typed Object-Oriented Languages With Polymorphic Inline Caches* (ECOOP 1991)——PIC，V8 的 IC 祖先。
- Anton Ertl & David Gregg, *The Structure and Performance of Efficient Interpreters* (2003)——系统对比 switch/threaded/replication dispatch。

### 0.3 栈式 vs 寄存器式

| 维度 | 栈式（JVM、CPython、clox） | 寄存器式（Lua 5、Dalvik、WebAssembly） |
|------|--------------------------|--------------------------------------|
| 指令宽度 | 1–3 字节，操作数隐含在栈顶 | 3–4 字节，显式 dst/src 寄存器 |
| 指令条数 | 多（大量 push/pop） | 少（直接操作虚拟寄存器） |
| 编译器复杂度 | 低（不需要寄存器分配） | 较高（需要虚拟寄存器分配） |
| 实现难度 | 低 | 中 |
| 典型场景 | JVM bytecode、Python .pyc、clox | Lua 5.x、LuaJIT IR、Wasm |

Lua 5 切换到寄存器式（Mike Pall 的贡献）后性能提升约 30%，主要来自减少中间压栈/弹栈。

## 1. Tree-Walking 解释器：结构与代价

### 1.1 核心模式

Tree-walker 的本质：递归函数 `eval(node, env) → value`。

```python
# 【示意，非逐字】基础 tree-walking 解释器结构
class Interpreter:
    def eval(self, node, env):
        match node:
            case NumberLiteral(value=v):
                return v
            case BinaryOp(op='+', left=l, right=r):
                return self.eval(l, env) + self.eval(r, env)
            case VarRef(name=n):
                return env[n]
            case Assign(name=n, value=v):
                result = self.eval(v, env)
                env[n] = result
                return result
```

### 1.2 性能瓶颈剖析

对于 `1 + 2 * 3`，tree-walker 要：

1. 解引用 `BinaryOp` 节点指针 → cache miss  
2. 做 Python/C++ `isinstance`/`match` dispatch → 间接跳转  
3. 递归调用两个子节点 → 栈帧分配  
4. 取回两个值 → 再次解引用  

**每个操作 5–15 条内存访问**。字节码 VM 把同样的计算压缩成 `LOAD_CONST 1 / LOAD_CONST 2 / LOAD_CONST 3 / MULTIPLY / ADD`，CPU 顺序取指，cache 命中率高几个数量级。

## 2. 字节码设计：clox 的 Chunk

### 2.1 Chunk 数据结构

【真实源码 munificent/craftinginterpreters@c/chunk.h】

```c
typedef struct {
  int count;          // 已写入字节数
  int capacity;       // 已分配容量
  uint8_t* code;      // 字节码数组（每条指令 1–3 字节）
  int* lines;         // code[i] 对应源码行号（调试用）
  ValueArray constants; // 常量池
} Chunk;
```

**设计要点**：
- `code` 是 `uint8_t*`，每个 opcode 占 1 字节，操作数跟在后面。
- `lines` 与 `code` 等长，纯调试信息；生产实现常改为 run-length encoding 节省内存（例如 CPython 的 `lnotab`）。
- `constants` 是一个动态数组；`OP_CONSTANT` 后跟 1 字节索引（最多 256 个常量），超过用 `OP_CONSTANT_LONG` + 3 字节索引。

### 2.2 OpCode 集合

【真实源码 munificent/craftinginterpreters@c/chunk.h】

```c
typedef enum {
  OP_CONSTANT,        // 读常量池 [idx] 压栈
  OP_NIL,             // 压 nil
  OP_TRUE, OP_FALSE,
  OP_POP,             // 弹顶（用于表达式语句丢弃返回值）
  OP_GET_LOCAL,       // 读当前 frame 的局部变量槽
  OP_SET_LOCAL,
  OP_GET_GLOBAL,      // 读全局变量哈希表
  OP_DEFINE_GLOBAL,
  OP_SET_GLOBAL,
  OP_GET_UPVALUE,     // 读 closure upvalue
  OP_SET_UPVALUE,
  OP_GET_PROPERTY,    // 字段读（运行时类型检查）
  OP_SET_PROPERTY,
  OP_GET_SUPER,
  OP_EQUAL, OP_GREATER, OP_LESS,
  OP_ADD, OP_SUBTRACT, OP_MULTIPLY, OP_DIVIDE,
  OP_NOT, OP_NEGATE,
  OP_PRINT,
  OP_JUMP,            // 无条件跳转（操作数: 16-bit offset）
  OP_JUMP_IF_FALSE,   // 条件跳转
  OP_LOOP,            // 向后跳（循环）
  OP_CALL,            // 调用（操作数: argc）
  OP_INVOKE,          // 方法调用快速路径
  OP_SUPER_INVOKE,
  OP_CLOSURE,         // 创建 ObjClosure，捕获 upvalue
  OP_CLOSE_UPVALUE,   // 将栈上 upvalue 迁移到堆
  OP_RETURN,
  OP_CLASS,
  OP_INHERIT,
  OP_METHOD
} OpCode;
```

**指令密度对比**：CPython 3.12 opcode 有 ~150 个，JVM 有 ~200 个，clox 教学用 ~37 个——体积越小越易分析，覆盖越全越难维护，实际 VM 常用 "base + specialization" 模式。

## 3. 编译器：单遍 Pratt 解析发射字节码

### 3.1 Compiler 结构

【真实源码 munificent/craftinginterpreters@c/compiler.c】

```c
typedef struct {
  Token current;
  Token previous;
  bool hadError;
  bool panicMode;   // 错误恢复模式：跳过直到同步点
} Parser;

typedef struct Compiler {
  struct Compiler* enclosing;  // 嵌套函数链
  ObjFunction* function;       // 正在编译的函数对象
  FunctionType type;
  Local locals[UINT8_COUNT];   // 局部变量表（256 个上限）
  int localCount;
  Upvalue upvalues[UINT8_COUNT]; // upvalue 捕获表
  int scopeDepth;
} Compiler;
```

**关键点**：每个函数编译时 `Compiler` 压栈（`enclosing` 指针形成链），函数编译完毕 pop 回去——这是**编译时调用栈**，对应运行时的 CallFrame 栈。

### 3.2 Pratt 解析器驱动字节码发射

【真实源码 munificent/craftinginterpreters@c/compiler.c】

```c
static void parsePrecedence(Precedence precedence) {
  advance();
  ParseFn prefixRule = getRule(parser.previous.type)->prefix;
  if (prefixRule == NULL) {
    error("Expect expression.");
    return;
  }
  bool canAssign = precedence <= PREC_ASSIGNMENT;
  prefixRule(canAssign);   // 处理前缀（数字、括号、一元运算）

  while (precedence <= getRule(parser.current.type)->precedence) {
    advance();
    ParseFn infixRule = getRule(parser.previous.type)->infix;
    infixRule(canAssign);  // 处理中缀（二元运算、调用、字段访问）
  }

  if (canAssign && match(TOKEN_EQUAL)) {
    error("Invalid assignment target.");
  }
}

// Pratt 规则表（每个 token 类型 → {prefix_fn, infix_fn, precedence}）
ParseRule rules[] = {
  [TOKEN_LEFT_PAREN]  = {grouping, call,   PREC_CALL},
  [TOKEN_DOT]         = {NULL,     dot,    PREC_CALL},
  [TOKEN_MINUS]       = {unary,    binary, PREC_TERM},
  [TOKEN_PLUS]        = {NULL,     binary, PREC_TERM},
  [TOKEN_BANG]        = {unary,    NULL,   PREC_NONE},
  [TOKEN_IDENTIFIER]  = {variable, NULL,   PREC_NONE},
  [TOKEN_STRING]      = {string,   NULL,   PREC_NONE},
  [TOKEN_NUMBER]      = {number,   NULL,   PREC_NONE},
  // ... 完整表见原文
};
```

**字节码发射例子**：`number()` 函数读到数字字面量时：

```c
static void number(bool canAssign) {
  double value = strtod(parser.previous.start, NULL);
  emitConstant(NUMBER_VAL(value));  // 写入常量池 + 发射 OP_CONSTANT idx
}
```

整个编译器是**无 AST 的单遍编译**——边解析边发射字节码，这是 Pratt 解析器最优雅的特性之一：precedence 控制"发射时机"。

## 4. 栈式 VM：clox run() 精读

### 4.1 VM 结构

【真实源码 munificent/craftinginterpreters@c/vm.h】

```c
#define FRAMES_MAX 64
#define STACK_MAX  (FRAMES_MAX * UINT8_COUNT)  // 64 * 256 = 16384 个值

typedef struct {
  ObjClosure* closure;  // 当前函数的 closure（含字节码）
  uint8_t* ip;          // instruction pointer（字节码数组内的裸指针）
  Value* slots;         // 指向该 frame 在 value stack 上的起点
} CallFrame;

typedef struct {
  CallFrame frames[FRAMES_MAX];  // 调用栈（静态分配）
  int frameCount;

  Value stack[STACK_MAX];        // 操作数栈（静态分配）
  Value* stackTop;               // 栈顶（下一个要写入的位置）

  Table globals;                 // 全局变量哈希表
  Table strings;                 // interned 字符串表
  ObjString* initString;         // cached "init" 字符串

  ObjUpvalue* openUpvalues;      // 链表：仍在栈上的 upvalue
  size_t bytesAllocated;         // GC 当前已分配字节
  size_t nextGC;                 // GC 触发阈值
  Obj* objects;                  // 所有堆对象链表（GC 用）
  // ... gray stack for tri-color GC
} VM;
```

**设计亮点**：
- `frames[]` 和 `stack[]` 都是**静态数组**（非堆分配），避免额外 malloc overhead，也让 GC 扫描根更简单。
- `ip` 是**裸指针**（`uint8_t*`），不是索引——`*ip++` 一条指令即可取操作码，比 `chunk.code[pc++]` 少一次加法。
- `slots` 是指向 `stack[]` 中某处的指针——局部变量访问 = `frame->slots[idx]`，同样是裸指针算术，比哈希表快。

### 4.2 主循环宏定义

【真实源码 munificent/craftinginterpreters@c/vm.c】

```c
#define READ_BYTE()     (*frame->ip++)
#define READ_SHORT()    \
    (frame->ip += 2,    \
     (uint16_t)((frame->ip[-2] << 8) | frame->ip[-1]))
#define READ_CONSTANT() \
    (frame->closure->function->chunk.constants.values[READ_BYTE()])
#define READ_STRING()   AS_STRING(READ_CONSTANT())
#define BINARY_OP(valueType, op)                           \
    do {                                                    \
      if (!IS_NUMBER(peek(0)) || !IS_NUMBER(peek(1))) {    \
        runtimeError("Operands must be numbers.");          \
        return INTERPRET_RUNTIME_ERROR;                     \
      }                                                     \
      double b = AS_NUMBER(pop());                          \
      double a = AS_NUMBER(pop());                          \
      push(valueType(a op b));                              \
    } while (false)
```

**为什么用宏而不是函数？**  
`READ_BYTE()` 会在每条指令都被调用，内联 `*frame->ip++` 让编译器把 `ip` 保持在寄存器里（`register` 语义）——函数调用会强迫 spill。`BINARY_OP` 用 `do { } while(false)` 包裹保证 `return` 在正确上下文里。

### 4.3 Dispatch 循环（switch 版本）

【示意，非逐字——实际结构与此一致，但含更多 case】

```c
static InterpretResult run() {
  CallFrame* frame = &vm.frames[vm.frameCount - 1];

  for (;;) {
    uint8_t instruction;
    switch (instruction = READ_BYTE()) {
      case OP_CONSTANT: {
        Value constant = READ_CONSTANT();
        push(constant);
        break;
      }
      case OP_ADD: {
        if (IS_STRING(peek(0)) && IS_STRING(peek(1))) {
          concatenate();         // 字符串拼接走特殊路径
        } else {
          BINARY_OP(NUMBER_VAL, +);
        }
        break;
      }
      case OP_CALL: {
        int argCount = READ_BYTE();
        if (!callValue(peek(argCount), argCount)) {
          return INTERPRET_RUNTIME_ERROR;
        }
        frame = &vm.frames[vm.frameCount - 1]; // 更新 frame 指针！
        break;
      }
      case OP_RETURN: {
        Value result = pop();
        closeUpvalues(frame->slots);    // 关闭本 frame 的 upvalue
        vm.frameCount--;
        if (vm.frameCount == 0) {
          pop();
          return INTERPRET_OK;
        }
        vm.stackTop = frame->slots;     // 恢复调用者栈顶
        push(result);
        frame = &vm.frames[vm.frameCount - 1];
        break;
      }
      // ... 30+ 更多 case
    }
  }
}
```

**OP_CALL 后立刻更新 `frame`** 是一个常见 bug 点——`callValue` 内部 push 了新 CallFrame，所以必须重新取 `&vm.frames[vm.frameCount - 1]`。

## 5. Dispatch 优化：从 switch 到 computed goto

### 5.1 Switch Dispatch 的代价

```
switch (opcode) {
  case OP_ADD:    ...   // CPU 看到: cmp + jne + cmp + jne + ...
  case OP_SUB:    ...   // 间接跳转表（编译器优化后）
  case OP_RETURN: ...
}
```

编译器通常把 `switch` 编译成**跳转表**（jump table），复杂度 O(1)，但：

1. 每执行完一条指令，都要跳回 `for (;;)` 循环顶部，再做一次跳转表查找。
2. 所有指令共用**同一个间接跳转点**——CPU 分支预测器看到的总是"下一条是 N 种可能之一"，预测准确率低。

### 5.2 Computed Goto（GNU 扩展）

CPython 3.x 及众多生产解释器使用的技术：

【真实源码 python/cpython@Python/opcode_targets.h（节选）】

```c
// GCC/Clang 扩展: &&LABEL 取标签地址
#if !_Py_TAIL_CALL_INTERP
static void *opcode_targets_table[256] = {
    &&TARGET_CACHE,
    &&TARGET_BINARY_SLICE,
    &&TARGET_BUILD_TEMPLATE,
    &&TARGET_BINARY_OP_INPLACE_ADD_UNICODE,
    &&TARGET_CALL_FUNCTION_EX,
    // ... 256 个入口
    &&TARGET_BINARY_OP,
    &&TARGET_BUILD_LIST,
    &&TARGET_BUILD_MAP,
    // ...
};
```

使用方式：

```c
// 初始化
#define DISPATCH() goto *opcode_targets_table[*ip++]

// 每条指令末尾
TARGET_BINARY_OP:
    // ... 执行 BINARY_OP 逻辑 ...
    DISPATCH();  // 展开为: goto *opcode_targets_table[*ip++]

TARGET_LOAD_CONST:
    // ... 
    DISPATCH();
```

**性能提升原理**：
- 每条指令有**自己的间接跳转点**，CPU 分支预测器可以**分别学习**每条指令后"最可能是哪条指令"。
- 消除了循环顶部的额外跳转。
- 实测：CPython 启用 `USE_COMPUTED_GOTOS` 后约快 15–25%（Ertl & Gregg 2003 报告 ~20% 提升）。

**限制**：`&&label` 是 GNU 扩展，MSVC 不支持——CPython 在 MSVC 下回退到 switch。

### 5.3 Threaded Code（更激进）

FORTH 和早期 Smalltalk 用的 **direct/indirect threaded code**：

```
; Direct Threaded Code (DTC) - 每个"指令"就是实现代码的地址
[addr_of_DUP] [addr_of_ADD] [addr_of_PUSH_42] [addr_of_EXIT]
```

执行引擎只需：

```c
#define NEXT() goto **ip++
```

比 computed goto 少一次数组查找（直接是代码地址，不是 opcode 编号）。代价：字节码体积变大（每条指令存地址而非 1 字节 opcode）。

### 5.4 Inline Cache（IC）：从解释到自适应

IC 是解释器和 JIT 之间的桥梁：

```
// 第一次执行 obj.x：
//   1. 慢路径：查 obj 的 class，找 "x" 在 class 布局中的 offset
//   2. 把 offset 写入 IC slot（字节码中的预留空间）

// 第二次执行同一条指令（大概率 obj 同 class）：
//   1. 检查 obj.class == cached_class （一次比较）
//   2. 直接用 cached_offset 读字段（一次内存读）
//   无需再查哈希表
```

V8 的 IC 系统在此基础上演化出 **Polymorphic IC（PIC）**（缓存多个 class→offset 对）和 **Megamorphic IC**（放弃缓存，退回全局 stub）。

## 6. Value 表示：Tagged Union vs NaN Boxing

### 6.1 Tagged Union（默认）

【真实源码 munificent/craftinginterpreters@c/value.h】

```c
typedef enum { VAL_BOOL, VAL_NIL, VAL_NUMBER, VAL_OBJ } ValueType;

typedef struct {
  ValueType type;  // 4 字节 tag
  union {
    bool boolean;
    double number;
    Obj* obj;
  } as;            // 8 字节（最大 member = double）
} Value;           // 总计: 16 字节/值（含 padding）
```

**IS_NUMBER / AS_NUMBER**：

```c
#define IS_NUMBER(value)  ((value).type == VAL_NUMBER)
#define AS_NUMBER(value)  ((value).as.number)
```

简单直观，但 16 字节/值在 stack 上意味着更多 cache 占用。

### 6.2 NaN Boxing（优化变体）

【真实源码 munificent/craftinginterpreters@c/value.h（NAN_BOXING 宏开启时）】

```c
typedef uint64_t Value;  // 整个 value 就是一个 64-bit 整数
```

利用 IEEE 754 double 的 NaN 空间：

```
IEEE 754 double 布局 (64 bit):
[符号 1][指数 11][尾数 52]

安静 NaN (quiet NaN) 条件: 指数全 1 + 尾数最高位 = 1
  = 0_11111111111_1xxx...x (52位)

合法浮点数: 指数不全为 1（或指数全1但尾数=0即±Inf）

NaN boxing 方案:
  浮点数:    直接存，指数不全为 1
  nil:       QNAN | TAG_NIL  (TAG_NIL = 1)
  false:     QNAN | TAG_FALSE (TAG_FALSE = 2)
  true:      QNAN | TAG_TRUE  (TAG_TRUE = 3)
  对象指针:  SIGN_BIT | QNAN | (ptr & 0x0000FFFFFFFFFFFF)
             (现代 64 位系统用户态指针只用低 48 位)
```

```c
#define QNAN      ((uint64_t)0x7ffc000000000000)  // 最小 quiet NaN
#define SIGN_BIT  ((uint64_t)0x8000000000000000)
#define TAG_NIL   1
#define TAG_FALSE 2
#define TAG_TRUE  3

#define IS_NUMBER(v)  (((v) & QNAN) != QNAN)    // 不是 NaN → 是数
#define AS_NUMBER(v)  valueToNum(v)              // memcpy bit cast
#define IS_OBJ(v)     (((v) & (QNAN | SIGN_BIT)) == (QNAN | SIGN_BIT))
#define AS_OBJ(v)     ((Obj*)(uintptr_t)((v) & ~(SIGN_BIT | QNAN)))
```

**效果**：Value 从 16 字节压缩到 8 字节，stack 容量翻倍，cache 命中率显著提升。  
**代价**：指针只能用低 48 位（x86-64 用户态实际 47 位，ARM64 用户态 48 位，足够）；代码可读性下降；需要 `static_assert(sizeof(double) == sizeof(uint64_t))`。

LuaJIT 的 NaN tagging 方案类似但更极致，支持在 NaN payload 里存 32 位整数加速整数运算。

## 7. 闭包与 Upvalue：把栈变量迁移到堆

### 7.1 问题：first-class 函数 + 变量逃逸

```lox
fun makeCounter() {
  var count = 0;
  fun increment() {
    count = count + 1;  // 捕获外层 count
    return count;
  }
  return increment;
}
var c = makeCounter();
c();  // 1 —— makeCounter 已返回，但 count 还活着
```

`count` 在 `makeCounter` 返回后仍需存活。简单方案是每个函数调用分配堆上环境，但 clox 用更精巧的 **upvalue 惰性关闭** 方案。

### 7.2 ObjUpvalue 结构

【真实源码 munificent/craftinginterpreters@c/object.h】

```c
typedef struct ObjUpvalue {
  Obj obj;
  Value* location;  // 【开放状态】指向栈上的 Value
  Value closed;     // 【关闭状态】值本身（当栈帧消失后）
  struct ObjUpvalue* next;  // VM 维护的开放 upvalue 链表
} ObjUpvalue;
```

**状态机**：
```
开放 (open):  location → &stack[i]   (变量还在栈上)
关闭 (closed): location → &self.closed (变量已迁移到堆上的 closed 字段)
```

### 7.3 关闭过程（closeUpvalues）

当函数返回或变量离开 scope 时：

```c
// 【示意，非逐字】closeUpvalues 逻辑
static void closeUpvalues(Value* last) {
  while (vm.openUpvalues != NULL &&
         vm.openUpvalues->location >= last) {
    ObjUpvalue* upvalue = vm.openUpvalues;
    upvalue->closed = *upvalue->location;  // 把栈上的值复制到 closed
    upvalue->location = &upvalue->closed;  // 重定向 location 到 closed
    vm.openUpvalues = upvalue->next;
  }
}
```

关闭后，所有引用同一变量的 closure 都通过 `upvalue->location` 访问，而 `location` 现在指向堆上的 `closed` 字段——**共享语义自动正确**，不需要复制。

### 7.4 编译时 upvalue 捕获

【真实源码 munificent/craftinginterpreters@c/compiler.c】

```c
typedef struct {
  uint8_t index;    // 指向外层 locals[] 或 upvalues[] 的索引
  bool isLocal;     // true = 捕获自直接外层的局部变量
                    // false = 捕获自更外层（upvalue of upvalue）
} Upvalue;
```

编译器遇到 `count` 时：

1. 先在当前函数 `locals[]` 里找 → 找不到
2. 在直接外层 `Compiler.locals[]` 里找 → 找到，在索引 1
3. 在当前 `Compiler.upvalues[]` 注册 `{index=1, isLocal=true}`
4. 发射 `OP_GET_UPVALUE 0`

若嵌套更深（函数 C 捕获函数 B 捕获函数 A 的变量），会递归注册 upvalue chain：`isLocal=false` 表示"我的 upvalue 是外层的 upvalue 而非局部变量"。

## 8. ⭐ 可运行 Demo：从 Tree-Walker 到字节码 VM

下面是一个完整的最小实现，演示同一小语言的两种解释策略。

### Demo 8.1：Tree-Walking 解释器（约 120 行 Python）

```python
#!/usr/bin/env python3
"""
demo_treewalker.py — 最小 tree-walking 解释器
支持: 整数字面量、+/-/*、let 绑定、打印
运行: python3 demo_treewalker.py
"""
from dataclasses import dataclass
from typing import Any

# ── AST 节点 ──────────────────────────────────────────────────
@dataclass
class Num:    val: int
@dataclass
class BinOp: op: str; left: Any; right: Any
@dataclass
class Let:   name: str; val: Any; body: Any
@dataclass
class Var:   name: str
@dataclass
class Print: expr: Any

# ── 解释器 ────────────────────────────────────────────────────
def eval_expr(node, env: dict) -> int:
    match node:
        case Num(val=v):
            return v
        case Var(name=n):
            return env[n]
        case BinOp(op=op, left=l, right=r):
            lv, rv = eval_expr(l, env), eval_expr(r, env)
            return lv + rv if op == '+' else lv - rv if op == '-' else lv * rv
        case Let(name=n, val=v, body=b):
            new_env = env | {n: eval_expr(v, env)}
            return eval_expr(b, new_env)
        case Print(expr=e):
            result = eval_expr(e, env)
            print(f"[tree-walk] {result}")
            return result
        case _:
            raise ValueError(f"Unknown node: {node}")

# ── 测试程序: let x = 10; let y = 20; print(x + y * 2) ──────
# 表示为 AST:
# Let('x', Num(10),
#   Let('y', Num(20),
#     Print(BinOp('+', Var('x'), BinOp('*', Var('y'), Num(2))))))

program = Let('x', Num(10),
           Let('y', Num(20),
             Print(BinOp('+', Var('x'), BinOp('*', Var('y'), Num(2))))))

eval_expr(program, {})
```

**运行**：

```bash
python3 demo_treewalker.py
```

**预期输出**：

```
[tree-walk] 50
```

### Demo 8.2：字节码编译器 + 栈式 VM（约 280 行 Python）

```python
#!/usr/bin/env python3
"""
demo_bytecode_vm.py — 最小字节码编译器 + 栈式 VM
支持: 整数字面量、+/-/*、let 绑定、变量引用、打印、简单 if/else
运行: python3 demo_bytecode_vm.py

架构:
  Compiler: AST → 字节码 (Chunk)
  VM:       执行 Chunk

指令集 (1 字节 opcode + 可选 1 字节操作数):
  PUSH_CONST  idx   — 把常量池[idx] 压栈
  ADD / SUB / MUL   — 弹两个操作数，结果压栈
  LOAD_VAR    idx   — 把变量槽[idx] 压栈
  STORE_VAR   idx   — 弹栈顶存入变量槽[idx]
  PRINT             — 弹栈顶打印
  JUMP_IF_FALSE off — 弹栈顶，若假则 ip += off
  JUMP          off — ip += off（无条件）
  HALT              — 停机
"""
from __future__ import annotations
from dataclasses import dataclass, field
from typing import Any
from enum import IntEnum, auto

# ── Opcode ────────────────────────────────────────────────────
class Op(IntEnum):
    PUSH_CONST    = auto()  # 后跟 1 字节常量索引
    ADD           = auto()
    SUB           = auto()
    MUL           = auto()
    LOAD_VAR      = auto()  # 后跟 1 字节变量槽索引
    STORE_VAR     = auto()  # 后跟 1 字节变量槽索引
    PRINT         = auto()
    JUMP_IF_FALSE = auto()  # 后跟 1 字节（有符号偏移，跳过 N 条字节）
    JUMP          = auto()  # 后跟 1 字节偏移
    HALT          = auto()

# ── Chunk (字节码容器) ─────────────────────────────────────────
@dataclass
class Chunk:
    code: list[int]       = field(default_factory=list)  # 字节码流
    constants: list[int]  = field(default_factory=list)  # 常量池

    def emit(self, *bytes_: int):
        self.code.extend(bytes_)

    def add_constant(self, val: int) -> int:
        idx = len(self.constants)
        self.constants.append(val)
        return idx

    def current_offset(self) -> int:
        return len(self.code)

    def patch(self, offset: int, val: int):
        """回填跳转偏移"""
        self.code[offset] = val

    def disassemble(self):
        """打印字节码（调试用）"""
        ip = 0
        print("=== Bytecode ===")
        while ip < len(self.code):
            op = Op(self.code[ip])
            if op in (Op.PUSH_CONST, Op.LOAD_VAR, Op.STORE_VAR,
                      Op.JUMP_IF_FALSE, Op.JUMP):
                operand = self.code[ip + 1]
                if op == Op.PUSH_CONST:
                    extra = f"  ({self.constants[operand]})"
                else:
                    extra = ""
                print(f"  {ip:04d} {op.name:<18} {operand}{extra}")
                ip += 2
            else:
                print(f"  {ip:04d} {op.name}")
                ip += 1
        print(f"Constants: {self.constants}")
        print()

# ── AST 节点（复用 demo_treewalker 的定义）─────────────────────
@dataclass
class Num:    val: int
@dataclass
class BinOp: op: str; left: Any; right: Any
@dataclass
class Let:   name: str; val: Any; body: Any
@dataclass
class Var:   name: str
@dataclass
class Print: expr: Any
@dataclass
class If:    cond: Any; then: Any; els: Any  # 新增

# ── 编译器 ────────────────────────────────────────────────────
class Compiler:
    def __init__(self):
        self.chunk = Chunk()
        self.vars: dict[str, int] = {}  # name → slot index
        self.next_slot = 0

    def _var_slot(self, name: str) -> int:
        if name not in self.vars:
            self.vars[name] = self.next_slot
            self.next_slot += 1
        return self.vars[name]

    def compile(self, node: Any):
        match node:
            case Num(val=v):
                idx = self.chunk.add_constant(v)
                self.chunk.emit(Op.PUSH_CONST, idx)

            case BinOp(op=op, left=l, right=r):
                self.compile(l)
                self.compile(r)
                match op:
                    case '+': self.chunk.emit(Op.ADD)
                    case '-': self.chunk.emit(Op.SUB)
                    case '*': self.chunk.emit(Op.MUL)

            case Var(name=n):
                slot = self._var_slot(n)
                self.chunk.emit(Op.LOAD_VAR, slot)

            case Let(name=n, val=v, body=b):
                self.compile(v)               # 把 v 的值压栈
                slot = self._var_slot(n)
                self.chunk.emit(Op.STORE_VAR, slot)  # 存入变量槽
                self.compile(b)               # 继续编译 body

            case Print(expr=e):
                self.compile(e)
                self.chunk.emit(Op.PRINT)

            case If(cond=c, then=t, els=e):
                self.compile(c)
                # 发射 JUMP_IF_FALSE，先占位
                self.chunk.emit(Op.JUMP_IF_FALSE, 0)
                jump_if_false_patch = self.chunk.current_offset() - 1

                self.compile(t)
                # 发射 JUMP（跳过 else 分支），先占位
                self.chunk.emit(Op.JUMP, 0)
                jump_patch = self.chunk.current_offset() - 1

                # 回填 JUMP_IF_FALSE 的偏移
                else_start = self.chunk.current_offset()
                # 偏移 = else_start - (jump_if_false_patch + 1)
                # 即：从指令本身之后数，还要跳多少字节
                self.chunk.patch(
                    jump_if_false_patch,
                    else_start - (jump_if_false_patch + 1)
                )
                self.compile(e)
                # 回填 JUMP 的偏移
                after_else = self.chunk.current_offset()
                self.chunk.patch(
                    jump_patch,
                    after_else - (jump_patch + 1)
                )

            case _:
                raise ValueError(f"Unknown node: {node}")

    def finish(self) -> Chunk:
        self.chunk.emit(Op.HALT)
        return self.chunk

# ── 栈式 VM ───────────────────────────────────────────────────
class VM:
    def __init__(self, chunk: Chunk):
        self.chunk = chunk
        self.stack: list[int] = []
        self.vars: list[int] = [0] * 256   # 变量槽（简化：固定大小）
        self.ip: int = 0

    def push(self, v: int): self.stack.append(v)
    def pop(self) -> int:   return self.stack.pop()
    def peek(self) -> int:  return self.stack[-1]

    def run(self):
        code = self.chunk.code
        consts = self.chunk.constants

        while True:
            op = Op(code[self.ip])
            self.ip += 1

            match op:
                case Op.PUSH_CONST:
                    idx = code[self.ip]; self.ip += 1
                    self.push(consts[idx])

                case Op.ADD:
                    b, a = self.pop(), self.pop()
                    self.push(a + b)

                case Op.SUB:
                    b, a = self.pop(), self.pop()
                    self.push(a - b)

                case Op.MUL:
                    b, a = self.pop(), self.pop()
                    self.push(a * b)

                case Op.LOAD_VAR:
                    slot = code[self.ip]; self.ip += 1
                    self.push(self.vars[slot])

                case Op.STORE_VAR:
                    slot = code[self.ip]; self.ip += 1
                    self.vars[slot] = self.peek()  # 注意: 不弹栈
                    # （let binding 后续还需要 body 编译时的栈结构）
                    # 实际上我们在 body 开始前弹掉—— 这里简化为
                    # STORE 不弹，Let 编译后 body 里不再用栈上的这个值

                case Op.PRINT:
                    print(f"[VM] {self.pop()}")

                case Op.JUMP_IF_FALSE:
                    offset = code[self.ip]; self.ip += 1
                    if self.pop() == 0:
                        self.ip += offset

                case Op.JUMP:
                    offset = code[self.ip]; self.ip += 1
                    self.ip += offset

                case Op.HALT:
                    return

# ── 测试 ──────────────────────────────────────────────────────
def test_basic():
    print("=== Test 1: let x=10; let y=20; print(x + y*2) ===")
    prog = Let('x', Num(10),
            Let('y', Num(20),
              Print(BinOp('+', Var('x'), BinOp('*', Var('y'), Num(2))))))
    c = Compiler()
    c.compile(prog)
    chunk = c.finish()
    chunk.disassemble()
    VM(chunk).run()

def test_if():
    print("\n=== Test 2: let x=5; if x then print(x*2) else print(99) ===")
    # x=5 非零 → 走 then 分支 → 打印 10
    prog = Let('x', Num(5),
            If(Var('x'),
               Print(BinOp('*', Var('x'), Num(2))),
               Print(Num(99))))
    c = Compiler()
    c.compile(prog)
    chunk = c.finish()
    chunk.disassemble()
    VM(chunk).run()

def test_if_false():
    print("\n=== Test 3: let x=0; if x then print(1) else print(0) ===")
    prog = Let('x', Num(0),
            If(Var('x'),
               Print(Num(1)),
               Print(Num(0))))
    c = Compiler()
    c.compile(prog)
    chunk = c.finish()
    chunk.disassemble()
    VM(chunk).run()

if __name__ == '__main__':
    test_basic()
    test_if()
    test_if_false()
```

## 9. 方案对比与选型决策

### 9.1 解释策略全对比

| 策略 | 启动延迟 | 峰值吞吐 | 实现复杂度 | 适用场景 |
|------|---------|---------|-----------|---------|
| Tree-walking | 极低 | 低（指针追逐） | 极低 | 脚本原型、DSL、配置语言 |
| 字节码 VM（switch） | 低 | 中 | 低 | Ruby 1.9 前、旧版 PHP |
| 字节码 VM（computed goto） | 低 | 中高 | 中 | CPython、PHP 7+、MRI Ruby |
| 寄存器 VM | 低 | 高 | 中高 | Lua 5、LuaJIT、Android ART |
| 字节码 + JIT | 中（warmup） | 极高 | 极高 | V8、SpiderMonkey、JVM HotSpot |
| AOT 编译 | 高 | 极高 | 极高 | GHC、Rust、Go |

### 9.2 栈式 vs 寄存器式深度对比

| 维度 | 栈式 | 寄存器式 |
|------|------|---------|
| 指令条数（典型程序） | 多 30–50% | 少 |
| 指令字节数 | 少（1 字节 opcode） | 多（含 dst/src 字段） |
| 编译器复杂度 | 低（无寄存器分配） | 需要虚拟寄存器分配 |
| JIT 可优化性 | 较低（隐式数据流） | 较高（显式 def-use） |
| 典型代表 | JVM、CPython、clox | Lua 5、Dalvik、Wasm |

**何时选寄存器式**：准备在上面加 JIT（寄存器式字节码天然接近 SSA IR，JIT 提升更明显）；或者解释执行密集型数值计算（Lua 数值基准常比 Python 快 5–10×）。

### 9.3 Dispatch 策略对比

| 方案 | 性能 | 可移植性 | 实现代价 |
|------|------|---------|---------|
| switch | 基准 | 完全可移植 | 无 |
| computed goto | +15–25% | GCC/Clang only | 低 |
| direct threaded | +20–35% | GCC/Clang only | 中（字节码体积×8） |
| inline cache | +50–200%（热路径） | 需架构支持 | 高 |
| JIT + trace | +5–50× | 需 JIT 后端 | 极高 |

## 10. 失败模式与真坑

### 10.1 Frame 指针失效（OP_CALL 后忘记更新）

```c
case OP_CALL: {
  int argCount = READ_BYTE();
  if (!callValue(peek(argCount), argCount)) { ... }
  // ⚠️ callValue 内部 push 了新 CallFrame！
  // 如果不更新 frame，下一条 READ_BYTE() 还在读旧 frame 的 ip
  frame = &vm.frames[vm.frameCount - 1];  // 必须！
  break;
}
```

**根因**：`frames[]` 是数组，`frame` 是指针——数组没有 realloc（静态），但 `frameCount` 变了，所以指针必须手动更新。Lua VM 里同样的问题触发过多次 bug report。

### 10.2 Upvalue 双重关闭

若同一个 upvalue 对象被两个 closure 持有，且两次 `closeUpvalues` 都处理它：

```
openUpvalues 链: A → B → C
第一次 closeUpvalues(stack[5]): 关闭 C（A.next = B.next = NULL?）
第二次 closeUpvalues(stack[3]): 走 NULL 链，跳过 B —— B 永远不关闭！
```

**正确做法**：`closeUpvalues` 按 `location >= last` 条件遍历，每次只处理满足条件的，链表不断头——clox 正是这样实现的。

### 10.3 NaN Boxing 与 32 位系统

NaN boxing 假设指针 ≤ 48 位有效位。在 32 位系统（指针 32 位）没问题；但在某些嵌入式 64 位平台（CHERI 能力指针 128 位）会完全失效。LuaJIT 为此在 iOS 上（arm64 有额外地址标签位）专门做了 pointer tagging 变体。

### 10.4 computed goto 与 -O0

GCC 在 `-O0` 下不会把 `ip` 保留在寄存器里，computed goto 的性能优势消失甚至反转（多了一次内存读）。CPython 的 `USE_COMPUTED_GOTOS` 在调试构建下自动禁用。

### 10.5 Stack 溢出的静默覆盖

clox 用静态 `stack[STACK_MAX]`，越界写会默默覆盖 `globals` 等后续字段。生产 VM 必须在每次 push 前检查：

```c
static void push(Value value) {
  // 生产实现：
  if (vm.stackTop >= vm.stack + STACK_MAX) runtimeError("Stack overflow");
  *vm.stackTop = value;
  vm.stackTop++;
}
```

clox 教学版本跳过了这个检查。

### 10.6 Upvalue 在 GC 之前被回收

如果 GC 不把 `openUpvalues` 链表作为根扫描，而某个 upvalue 指向的对象没有其他引用，就会被回收——造成悬空指针。clox 的 GC 在 markRoots() 里显式遍历 `vm.openUpvalues` 链。

## 11. 章末五件套

### 11.1 核心概念速查

### 11.4 动手练习

1. **在 Demo 8.2 上加函数调用**：新增 `Fun`/`Call` AST 节点 + `OP_CALL`/`OP_RETURN`；实现多 CallFrame 栈。
2. **把 switch dispatch 改成 computed goto**（Python 无此特性，用 C 重写 VM 核心）：对比 `-O2` 下两者的 `perf stat` 数据。
3. **实现 NaN boxing**：修改 Demo 的 `Value` 类型为 `int64`，用最高位区分整数/对象；观察内存布局变化。
4. **加 upvalue/闭包支持**：在 Demo 8.2 里实现 `Lambda` + `Closure` 节点，复现 clox 的开放/关闭 upvalue 状态机。
5. **写一个 disassembler**：遍历 Chunk，对每条指令打印操作数的**语义**（常量值、变量名），体会调试信息设计（对应 clox `debug.c`）。

### 11.5 思维框架：如何评估一个解释器实现

```
问题1: 它的 Value 表示是什么？（tagged union / NaN boxing / pointer tagging）
       → 决定内存密度和类型检查速度

问题2: 它的 dispatch 策略是什么？（switch / computed goto / threaded / IC）
       → 决定解释器峰值性能上限

问题3: 它怎么处理变量？（哈希表环境 / 栈槽 / 寄存器）
       → 决定局部变量访问速度

问题4: 它怎么处理闭包？（堆分配环境 / upvalue 惰性关闭 / flat closure）
       → 决定函数调用开销

问题5: 它有没有 JIT 路径？（字节码 → IR → 机器码的桥梁在哪里？）
       → 决定长期性能天花板
```

## 附录 A：clox 完整 VM 数据流图

```
源代码 (Lox)
    │
    ▼
[Scanner] ──→ Token 流
    │
    ▼

    │  每个函数编译产生一个 ObjFunction (含 Chunk)
    ▼

    │
    ▼

    │
    ├─ frames[]: CallFrame 栈（64 层）
    │     每层: ip (裸指针) + slots (指向 stack[] 的指针)
    │
    ├─ stack[]: Value 操作数栈（16384 个值）
    │
    ├─ globals: Table（哈希表）
    │
    ├─ openUpvalues: 链表（指向 stack[] 中活跃变量）
    │
    └─ GC: objects 链表 + gray stack（三色标记）
```
