## 0. 设计考古：从布尔检索到 BM25

### 0.1 布尔检索时代（1960s–1970s）

最早的文档检索系统，如 STAIRS（IBM, 1973），使用纯布尔模型：查询是 AND/OR/NOT 组成的布尔表达式，文档要么匹配要么不匹配，无排序，无得分。

**核心痛点**：

- 召回/精确率完全由用户构造的布尔表达式决定，普通用户难以使用；
- 对于返回成百上千结果的查询，无排序意味着用户必须人工逐条浏览。

**数据结构**：这个时代已经有了倒排索引的雏形——词典 + posting list，只是 posting 里只存 docID，不存词频。这是一个关键的设计约束：布尔模型的所有语义都在词项的存在与否，词频是多余的。

### 0.2 TF-IDF（1972–1980s）

Karen Spärck Jones 在 1972 年提出 IDF（Inverse Document Frequency）：

> "A term is a better discriminator if it occurs in fewer documents."
> — Spärck Jones, "A statistical interpretation of term specificity and its application in retrieval," *Journal of Documentation*, 28(1), 1972

TF（term frequency）× IDF 给出了每个词对每篇文档的权重：

```
TF(t, d)     = count(t in d)              # 词项在文档中出现次数
IDF(t)       = log(N / df(t))             # N 总文档数，df 含该词的文档数
TF-IDF(t, d) = TF(t, d) × IDF(t)
```

IDF 取对数有两个理由：

1. **边际效用递减**：文档数翻倍，歧视能力并不翻倍；
2. **平衡量级**：避免罕见词完全压制常见词。

**TF-IDF 的两个根本缺陷**：
- TF 是**线性无界的**——词出现 100 次的贡献是出现 10 次的 10 倍，但直觉上重复度高的词并不代表文档更相关（饱和效应缺失）；
- **未归一化文档长度**——1000 词的文档天然比 100 词的文档 TF 高，但不一定更相关。

### 0.3 概率相关模型与 BM25（1976–1994）

Robertson 和 Spärck Jones 在 1976 年提出**概率相关模型（Probabilistic Relevance Model, PRM）**（*A Theoretical Basis for the Use of Co-occurrence Data in Information Retrieval*, Journal of Documentation, 1977）。

核心假设：给定查询 Q，文档 D 的相关性分数可以由以下比值衡量：

```
P(R=1 | D, Q) / P(R=0 | D, Q)
```

对所有查询词求和（独立性假设），得到 Robertson-Spärck Jones (RSJ) 权重：

```
w(t, D) = log[(p_t / (1 - p_t)) / (u_t / (1 - u_t))]
```

其中 `p_t` = 相关文档中词 t 出现的概率，`u_t` = 不相关文档中词 t 出现的概率。

这个框架没有考虑 TF 和文档长度。1994 年，Robertson 等在 **TREC-3（Text REtrieval Conference）** 上发表了"Okapi at TREC-3"（S.E. Robertson, S. Walker, S. Jones, M. Hancock-Beaulieu, M. Gatford, *Proceedings of the Third Text Retrieval Conference*, NIST 1994），在 Okapi 系统中提出了 BM25：

**BM11（b=1，完全文档长度归一化）**：
```
score = IDF × (k₁+1)·tf / (k₁·(dl/avgdl) + tf)
```

**BM15（b=0，不考虑文档长度）**：
```
score = IDF × (k₁+1)·tf / (k₁ + tf)
```

**BM25（插值，0≤b≤1）**：
```
score(D, Q) = Σ_{t∈Q} IDF(t) × [f(t,D)·(k₁+1)] / [f(t,D) + k₁·(1-b + b·|D|/avgdl)]
```

其中：
- `f(t, D)`：词 t 在文档 D 中的出现次数（TF）
- `|D|`：文档长度（词数）
- `avgdl`：语料库平均文档长度
- `k₁ ∈ [1.2, 2.0]`：TF 饱和参数，越大饱和越慢
- `b ∈ [0, 1]`：文档长度惩罚强度，通常取 0.75
- `IDF(t) = log(1 + (N - df(t) + 0.5) / (df(t) + 0.5))`：平滑版 IDF，避免 df=0 的除零

**为什么加 0.5？** 这是 Bayes 平滑（Laplace-like），避免罕见词 IDF 爆炸，同时防止完全不在语料中出现的词项给出无穷大 IDF。b=0.75 是 Robertson 等在 TREC 数据集上实验得到的经验最优值（并非从理论推导，是观察到的在多种文档类型下的鲁棒值）。

## 1. 倒排索引：数据结构深度剖析

### 1.1 核心数据结构

```
倒排索引 = 词典（Dictionary）+ 倒排列表（Posting Lists）

词典：{词项 → (词频 df, 倒排列表指针)}
倒排列表：[(docID₁, tf₁, [pos₁, pos₂, ...]), (docID₂, tf₂, ...), ...]
```

**词典实现选择**：

| 结构 | 查找复杂度 | 范围查询 | 内存 | 典型场景 |
|------|-----------|---------|------|---------|
| 哈希表 | O(1) avg | 不支持 | 中 | 精确匹配、内存索引 |
| 排序数组+二分 | O(log n) | 支持 | 低 | 静态索引 |
| B-tree / Trie | O(log n) | 支持 | 高 | 可更新索引 |
| FST（有限状态转换器）| O(\|term\|) | 支持前缀 | 极低 | Lucene 词典 |

Lucene 的词典使用 **Block-tree 结构**（基于 FST），在内存和查找效率之间取得最优平衡（`lucene/codecs/blocktree/` 包）。FST 本质是一个 DAG，公共前缀/后缀在编译期合并，对于英文词典压缩率可达 10:1 以上。

### 1.2 Lucene 索引构建管道（源码级）

Lucene 的索引写入路径是一条精心设计的流水线：

```
IndexWriter
  └─ DocumentsWriter
       └─ DocumentsWriterPerThread (每线程独立)
            └─ DefaultIndexingChain
                 ├─ TokenStream (Analyzer)   → 分词
                 ├─ TermsHashPerField        → token → posting slot
                 └─ FreqProxTermsWriterPerField  → 写 TF/pos 到 ByteBlockPool
```

**TermsHashPerField.add()** — 核心令牌处理方法：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/index/TermsHashPerField.java】
```java
void add(BytesRef termBytes, final int docID) throws IOException {
    assert assertDocId(docID);
    int termID = bytesHash.add(termBytes);      // 词项写入 BytesHash（类似 intern）
    if (termID >= 0) {                           // 新词项（该文档中首次出现）
        initStreamSlices(termID, docID);         // 分配 ByteBlockPool slice 存储 posting
    } else {
        try {
            termID = positionStreamSlice(termID, docID); // 已有词项，定位现有 slice
        } catch (DuplicateTermException e) {
            throw new DuplicateTermException(e.getMessage() + " '"
                    + termBytes.utf8ToString() + "'");
        }
    }
    if (doNextCall) {
        // 链式 TermsHash：FreqProxTermsWriterPerField 挂在下游
        nextPerField.add(postingsArray.textStarts[termID], docID);
    }
}
```

**注释解析**：
- `bytesHash.add()` 返回 ≥ 0 表示新词项，返回 `-(termID+1)` 表示已存在（类似 Java HashMap）；
- `initStreamSlices()` 在 ByteBlockPool 中分配两个 slice：stream[0] 存 freq/doc，stream[1] 存 pos；
- `doNextCall` 决定是否级联写入下游 TermsHash（ FreqProx），实现 TF 和 positions 的分离存储。

**FreqProxTermsWriterPerField.writeProx()** — 位置信息写入：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/index/FreqProxTermsWriterPerField.java】
```java
void writeProx(int termID, int proxCode) {
    if (payloadAttribute == null) {
        writeVInt(1, proxCode << 1);              // stream[1] 写 VInt 编码的位置差
    } else {
        BytesRef payload = payloadAttribute.getPayload();
        if (payload != null && payload.length > 0) {
            writeVInt(1, (proxCode << 1) | 1);   // 低位 1 = 有 payload
            writeVInt(1, payload.length);
            writeBytes(1, payload.bytes, payload.offset, payload.length);
            sawPayloads = true;
        } else {
            writeVInt(1, proxCode << 1);          // 低位 0 = 无 payload
        }
    }
    freqProxPostingsArray.lastPositions[termID] = fieldState.position;
}
```

**proxCode** 是当前位置与上一个位置的差（delta 编码），`<< 1` 留出低位给 payload 标志位。这是经典的 bit-packing：一次写入同时编码了两种信息。

### 1.3 Posting List 的磁盘布局（Lucene 912 格式）

Lucene 最新 posting 格式核心文件：

- `.tim`：词典文件（Term Dictionary），记录每个词项的 docFreq 和指向 `.doc/.pos/.pay` 的文件偏移
- `.tip`：词典索引（Term Index），FST 结构，加速词典查找
- `.doc`：posting 文档列表，存 docID delta + 词频
- `.pos`：位置信息
- `.pay`：payload 和 offset

**`.doc` 文件中的 block 结构**（来自 `MultiLevelSkipListWriter` javadoc 和 Lucene 格式注释）：

```
TermFreqs → <PackedBlock32>^(N/32) + VIntBlock?

PackedBlock32 → Level1SkipData + <PackedBlock>^32
PackedBlock   → Level0SkipData + PackedDocDeltaBlock + PackedFreqBlock?

PackedDocDeltaBlock：256个 delta-encoded docID，用 FOR 压缩
VIntBlock：尾部不足 256 个的 docID，用 VInt 变长编码
```

**MultiLevelSkipListWriter 的设计**（类 javadoc）：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/MultiLevelSkipListWriter.java】
```
该抽象类实现多层级跳表写入。

跳表层级结构：
- Level 0：每 skipInterval 个文档一个跳表节点
- Level i：每 skipMultiplier 个 Level(i-1) 节点对应一个 Level(i) 节点
- 高层节点维护指向低层对应位置的 child pointer

resetSkip()：初始化或重置所有层级的 buffer（跨 term 复用）
bufferSkip(df)：根据 df（当前写入文档数）判断需要写入哪些层级的跳表节点
writeSkip(output)：从最高层到最低层写出缓冲数据，返回跳表起始偏移
```

**设计模式**：MultiLevelSkipListWriter 用 Template Method——抽象方法 `writeSkipData()` 由子类（如 `Lucene912SkipWriter`）实现具体编码逻辑，基类管理多层 buffer 和 child pointer 的层级关系。

### 1.4 Delta 编码与 FOR 压缩

**Delta 编码**：posting 列表中的 docID 单调递增，存储相邻 docID 之差（d-gap）而非绝对值，大幅压缩数值范围。

例如 docID 序列 `[3, 7, 11, 45, 46]`，delta 编码后为 `[3, 4, 4, 34, 1]`，最大值从 46 降至 34，需要的位宽从 6 bit 降到 6 bit（此例改善不大，但对大语料库效果显著——docID 平均间距 = N/df，当 df 大时 d-gap 小）。

**FOR（Frame of Reference）**：在一个 block 内，所有整数用相同的位宽打包存储。位宽由 block 内最大值决定。

核心思想来自 Goldstein et al., "Compressing Relations and Indexes" (ICDE 1998) 以及 Willhalm et al. 的 SIMD 整数压缩研究。

Lucene 实现参考（ForUtil 类注释及 collapse8 函数）：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/lucene912/ForUtil.java（待核：路径可能是 lucene104/ForUtil.java）】
```java
/** Number of integers per block. */
public static final int BLOCK_SIZE = 256;  // 必须是 64 的倍数，方便 SIMD

/**
 * Inspired from https://fulmicoton.com/posts/bitpacking/
 * Encodes multiple integers in one to get SIMD-like speedups.
 * If bitsPerValue <= 8 then we pack 4 ints per Java int
 * else if bitsPerValue <= 16 we pack 2 ints per Java int
 * else we do scalar operations.
 */
void encode(int[] ints, int bitsPerValue, DataOutput out) throws IOException {
    final int nextPrimitive;
    if (bitsPerValue <= 8) {
        nextPrimitive = 8;    // 4 个整数压进一个 Java int（SIMD-like）
        collapse8(ints);
    } else if (bitsPerValue <= 16) {
        nextPrimitive = 16;   // 2 个整数压进一个 Java int
        collapse16(ints);
    } else {
        nextPrimitive = 32;   // 标准 32 位
    }
    encode(ints, bitsPerValue, nextPrimitive, out, tmp);
}

// collapse8: 把 ints[i], ints[64+i], ints[128+i], ints[192+i] 拼成一个 int
// 256 个值 → 64 个 Java int，依靠 JIT SIMD 向量化
static void collapse8(int[] arr) {
    for (int i = 0; i < 64; ++i) {
        arr[i] = (arr[i] << 24) | (arr[64 + i] << 16) | (arr[128 + i] << 8) | arr[192 + i];
    }
}
```

**PForDelta（Patched Frame of Reference Delta）**：对 block 内少量"异常大"的值，允许用额外的 patch 数组单独存储，其余值用更小的位宽。

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/lucene104/PForUtil.java】
```java
private static final int MAX_EXCEPTIONS = 7;  // 最多 7 个例外值

void encode(int[] ints, DataOutput out) throws IOException {
    // Step 1: 统计位宽直方图
    final int[] histogram = new int[32];
    int maxBitsRequired = 0;
    for (int i = 0; i < ForUtil.BLOCK_SIZE; ++i) {
        final int v = ints[i];
        final int bits = PackedInts.bitsRequired(v);
        histogram[bits]++;
        maxBitsRequired = Math.max(maxBitsRequired, bits);
    }

    // We store patch on a byte, so we can't decrease bits by more than 8
    final int minBits = Math.max(0, maxBitsRequired - 8);
    int cumulativeExceptions = 0;
    int patchedBitsRequired = maxBitsRequired;
    int numExceptions = 0;

    // Step 2: 从最高位宽向下扫描，找例外数 <= MAX_EXCEPTIONS 的最低位宽
    for (int b = maxBitsRequired; b >= minBits; --b) {
        if (cumulativeExceptions > MAX_EXCEPTIONS) {
            break;
        }
        patchedBitsRequired = b;
        numExceptions = cumulativeExceptions;
        cumulativeExceptions += histogram[b];
    }
    // 例外值单独以 (index, value) 对存储到 exceptions 数组
}
```

### 1.5 多级跳表：加速 AND 查询

AND 查询（多词求交）的朴素做法是归并两个 posting 列表：O(m + n)。跳表让这个过程可以在稀疏对齐时跳过大段。

**Stanford IR Book 中的经典结论**：对长度为 P 的 posting 列表，每隔 √P 个元素放一个跳表节点，是跳表密度的经验最优值（来源：*Introduction to Information Retrieval*, Manning et al., 2008, Chapter 2）。

Lucene 实现的是**多级跳表**，skipTo 的两阶段逻辑（来自 MultiLevelSkipListWriter 类结构描述）：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/MultiLevelSkipListReader.java（待核：Reader 与 Writer 对应）】
```java
// Level 0 间距 = skipInterval，Level i 间距 = skipInterval^(i+1)
this.skipInterval[0] = skipInterval;
for (int i = 1; i < maxSkipLevels; i++) {
    this.skipInterval[i] = this.skipInterval[i - 1] * skipMultiplier;
}

// skipTo 两阶段：上升找最高有效层 → 下降加载跳表项
public int skipTo(int target) throws IOException {
    int level = 0;
    // Phase 1: 上升——找到最高有效层
    while (level < numberOfSkipLevels - 1 && target > skipDoc[level + 1]) {
        level++;
    }
    // Phase 2: 下降——在该层加载跳表项
    while (level >= 0) {
        if (target > skipDoc[level]) {
            if (!loadNextSkip(level)) {
                continue;
            }
        } else {
            if (level > 0 && lastChildPointer > skipStream[level - 1].getFilePointer()) {
                seekChild(level - 1);
            }
            level--;
        }
    }
    return numSkipped[0] - skipInterval[0] - 1;
}
```

Lucene 的跳表还集成了 **competitive impact**（竞争性影响得分）：每个 skip 块记录该块内最大可能得分，查询时如果当前累积得分已超过 skip 块上界，整块可直接跳过（WAND 优化的基础，详见第 5 节失败模式部分）。

## 2. BM25 源码精读（Apache Lucene 10.x）

### 2.1 类层次结构

```
Similarity (抽象基类)
  └─ BM25Similarity
       └─ BM25Scorer (inner class，每查询词实例化一次)
            └─ SimScorer.score(freq, encodedNorm)
```

`Similarity` 定义了两个阶段：
- **Index time**：`computeNorm(FieldInvertState)` → 把文档长度压缩成 1 字节存入 `.nrm` 文件
- **Query time**：`scorer()` → 返回 `SimScorer`；`score(freq, encodedNorm)` → 打分

### 2.2 文档长度的 1 字节压缩（SmallFloat）

文档长度是整数，但每个文档的 norm 只存 1 字节——用 SmallFloat 实现有损压缩：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/util/SmallFloat.java】
```java
/**
 * floatToByte(b, mantissaBits=3, zeroExponent=15)
 * smallest non-zero value = 5.820766E-10
 * largest value = 7.5161928E9
 * epsilon = 0.125  (相对误差 12.5%)
 */
public static byte floatToByte315(float f) {
    int bits = Float.floatToRawIntBits(f);
    int smallfloat = bits >> (24 - 3);          // 保留 3 位 mantissa（IEEE 754 有 23 位）
    if (smallfloat <= ((63 - 15) << 3)) {
        return (bits <= 0) ? (byte) 0 : (byte) 1; // 下溢映射到最小值
    }
    if (smallfloat >= ((63 - 15) << 3) + 0x100) {
        return -1;  // 上溢映射到最大值（byte 范围 -128..127）
    }
    return (byte) (smallfloat - ((63 - 15) << 3));
}
```

这个设计意味着：文档长度存在 **12.5% 的量化误差**，换来 4× 内存节省（int → byte）。对大语料库，norm 数组大小等于文档总数，节省显著。

**BM25Similarity.computeNorm()** 把词数转换为这个格式：
```java
@Override
public final long computeNorm(FieldInvertState state) {
    final int numTerms;
    if (discountOverlaps) {
        numTerms = state.getLength() - state.getNumOverlap(); // 去除重叠 token（如同义词）
    } else {
        numTerms = state.getLength();
    }
    return SmallFloat.intToByte4(numTerms);   // 4 位 mantissa 版本，精度更好
}
```

### 2.3 scorer()：预计算 256 个 normInverse

```java
// 【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/similarities/BM25Similarity.java】

@Override
public final SimScorer scorer(float boost, CollectionStatistics collectionStats,
                               TermStatistics... termStats) {
    Explanation idf = termStats.length == 1
        ? idfExplain(collectionStats, termStats[0])
        : idfExplain(collectionStats, termStats);
    float avgdl = avgFieldLength(collectionStats);   // sumTotalTermFreq / docCount

    // 预计算 256 个 normInverse 值，索引 = 文档长度的压缩字节（0-255）
    float[] cache = new float[256];
    for (int i = 0; i < cache.length; i++) {
        // cache[i] = 1 / (k1 * (1 - b + b * dl / avgdl))
        // LENGTH_TABLE[i] 把压缩字节还原为近似文档长度
        cache[i] = 1f / (k1 * ((1 - b) + b * LENGTH_TABLE[i] / avgdl));
    }
    return new BM25Scorer(boost, k1, b, idf, avgdl, cache);
}
```

**设计精妙之处**：所有可能的文档长度编码只有 256 种（1 字节），所以 query 准备阶段一次性把所有归一化逆值算好，打分阶段只需做一次数组查找（`cache[norm & 0xFF]`），完全避免了浮点除法。这是 **"把变化维度压缩到有限集合，然后预计算全集"** 的经典工程模式。

### 2.4 IDF 计算

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/similarities/BM25Similarity.java】
```java
/** Implemented as log(1 + (docCount - docFreq + 0.5) / (docFreq + 0.5)) */
protected float idf(long docFreq, long docCount) {
    return (float) Math.log(1 + (docCount - docFreq + 0.5D) / (docFreq + 0.5D));
}
```

与经典 `log(N/df)` 的区别：
- 加 `+1` 保证 IDF ≥ 0（当一个词出现在所有文档中时，`log(1 + 0.5/0.5) = log(2) ≈ 0.69`，而非 0）
- `+0.5` 是 Bayes 平滑，等价于"提前见过 0.5 个相关文档和 0.5 个不相关文档"

### 2.5 核心打分：doScore() 与 TermScorer

BM25 的最终打分在 `BM25Scorer.doScore()` 中，**TermScorer** 负责把 posting 遍历与打分串联：

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/similarities/BM25Similarity.java】
```java
private float doScore(float freq, float normInverse) {
    // BM25 标准公式：score = weight * freq / (freq + k1 * norm)
    // 其中 weight = boost * IDF，norm = k1 * (1 - b + b * dl / avgdl)
    //
    // 等价变换（保证 float 单调性，避免提升为 double）：
    //   freq / (freq + norm) = 1 - 1 / (1 + freq * (1/norm))
    //   展开后 = weight - weight / (1 + freq * normInverse)
    //
    // 为什么要这样变换？
    // - freq * normInverse 保证对 freq 和 norm 都单调（IEEE 754 浮点乘法保证）
    // - 1 + x, 1 - 1/x 在正数域保持严格单调
    // - 单调性是 WAND Block-Max 剪枝的前提：只有单调函数，块级 maxScore 才是真正上界
    return weight - weight / (1f + freq * normInverse);
}
```

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/TermScorer.java】

### 2.6 批量打分（BulkSimScorer / SIMD 优化路径）

【真实源码 apache/lucene@lucene/core/src/java/org/apache/lucene/search/similarities/BM25Similarity.java】
```java
@Override
public BulkSimScorer asBulkSimScorer() {
    return new BulkSimScorer() {
        private float[] normInverses = new float[0];

        @Override
        public void score(int size, float[] freqs, long[] norms, float[] scores) {
            if (normInverses.length < size) {
                normInverses = new float[ArrayUtil.oversize(size, Float.BYTES)];
            }
            // 循环 1：批量查表（可能触发 JIT scatter/gather 优化）
            for (int i = 0; i < size; ++i) {
                normInverses[i] = cache[((byte) norms[i]) & 0xFF];
            }
            // 循环 2：批量打分
            // 注释："This loop auto-vectorizes"
            // 条件：无条件分支、无数据依赖、连续内存访问 → JIT 映射到 SSE/AVX 指令
            for (int i = 0; i < size; ++i) {
                scores[i] = doScore(freqs[i], normInverses[i]);
            }
        }
    };
}
```

两个 for 循环分开写是刻意设计：第一个循环有随机内存访问（cache 数组），第二个循环是纯连续计算，JVM JIT 只能向量化第二个。如果混写，JIT 无法向量化任何一段。

### Demo 1：最小倒排索引 + BM25 检索

依赖：Python 3.8+，标准库 `math`、`collections`。无需安装额外包。

```python
"""
demo_bm25.py — 最小可运行的倒排索引 + BM25 检索

运行步骤：
    python demo_bm25.py

预期输出（节选）：

    information     → df= 4, postings=[(0, 1), (1, 1), (2, 1), (4, 1)]
    retrieval       → df= 3, postings=[(0, 1), (1, 1), (3, 1)]
    ...
    === 检索: 'information retrieval' ===
    Rank 1: doc=1, score=2.0384, text='BM25 is a ranking function used in information retrie...'
    Rank 2: doc=0, score=1.8156, text='Information retrieval is the process of obtaining re...'

与 Lucene 对应关系：
    build_index()         ↔  TermsHashPerField.add() + FreqProxTermsWriterPerField
    inverted_index        ↔  .tim/.doc 文件
    BM25.idf()            ↔  BM25Similarity.idf()
    BM25.score_term()     ↔  BM25Similarity.doScore()
    BM25.search()         ↔  BooleanQuery(OR) + TopScoreDocCollector
"""

import math
import re
from collections import defaultdict
from typing import Dict, List, Tuple

# 语料库（模拟 5 个文档）
CORPUS = [
    "Information retrieval is the process of obtaining relevant resources",
    "BM25 is a ranking function used in information retrieval systems",
    "Inverted index maps terms to their locations in documents for information",
    "Retrieval models rank documents by relevance to a query",
    "Search engines use information retrieval techniques at scale",
]


def tokenize(text: str) -> List[str]:
    """最简 tokenizer，对应 Lucene StandardAnalyzer（小写 + 按非字母分割）"""
    return re.findall(r"[a-z]+", text.lower())


def build_index(corpus: List[str]) -> Tuple[Dict, List[int]]:
    """
    构建倒排索引。

    对应 Lucene：
      - TermsHashPerField.add() 处理每个 token
      - FreqProxTermsWriterPerField 写入 TF
      - delta 编码确保 docID 单调递增（Lucene 依赖此性质做 FOR 压缩）

    返回:
      inverted_index: {term → [(docID, tf), ...]}   ← .doc 文件的逻辑表示
      doc_lengths:    [length_of_doc_0, ...]          ← .nrm 文件（此处不压缩）
    """
    inverted_index: Dict[str, List[Tuple[int, int]]] = defaultdict(list)
    doc_lengths = []

    for doc_id, text in enumerate(corpus):
        tokens = tokenize(text)
        doc_lengths.append(len(tokens))

        # 统计 TF（对应 FreqProxTermsWriterPerField 追踪 termFreqs[termID]）
        tf_map: Dict[str, int] = defaultdict(int)
        for token in tokens:
            tf_map[token] += 1

        # 写入倒排列表（docID 单调递增 ← delta 编码前提）
        for term, tf in tf_map.items():
            inverted_index[term].append((doc_id, tf))

    # 排序保证 docID 单调（Lucene 在 flush 时保证这一点）
    for term in inverted_index:
        inverted_index[term].sort(key=lambda x: x[0])

    return dict(inverted_index), doc_lengths


class BM25:
    """
    对应 Lucene BM25Similarity，参数默认值与 Lucene 一致：k1=1.2, b=0.75

    打分公式（与 Lucene BM25Similarity.doScore() 等价）：
        normInverse = 1 / (k1 * (1 - b + b * dl / avgdl))
        score = weight - weight / (1 + freq * normInverse)
        where weight = idf * (k1 + 1)   [boost=1.0 时]
    """

    def __init__(self, inverted_index: Dict, doc_lengths: List[int],
                 k1: float = 1.2, b: float = 0.75):
        self.index = inverted_index
        self.doc_lengths = doc_lengths
        self.k1 = k1
        self.b = b
        self.N = len(doc_lengths)
        self.avgdl = sum(doc_lengths) / self.N

        # 预计算 IDF（对应 Lucene query 准备阶段的 idfExplain()）
        self._idf_cache: Dict[str, float] = {}

    def idf(self, term: str) -> float:
        """
        Lucene 实现：log(1 + (N - df + 0.5) / (df + 0.5))
        — +0.5 是 Bayes 平滑，避免 df=0 时除零
        — +1 保证 IDF ≥ 0（即使词在所有文档中出现，IDF = log(2) ≈ 0.69）
        """
        if term not in self._idf_cache:
            df = len(self.index.get(term, []))
            self._idf_cache[term] = math.log(
                1.0 + (self.N - df + 0.5) / (df + 0.5)
            )
        return self._idf_cache[term]

    def score_term(self, term: str, doc_id: int, tf: int) -> float:
        """
        单词项得分，对应 Lucene BM25Scorer.doScore()

        Lucene 原版（等价变换）：
            normInverse = 1 / (k1 * (1 - b + b * dl / avgdl))
            score = weight - weight / (1 + freq * normInverse)

        这里用展开形式（数学等价，便于理解）：
            tf_norm = tf / (tf + k1 * (1 - b + b * dl / avgdl))
            score   = idf * (k1 + 1) * tf_norm
        """
        dl = self.doc_lengths[doc_id]
        norm = self.k1 * (1.0 - self.b + self.b * dl / self.avgdl)
        tf_saturated = tf / (tf + norm)
        idf_val = self.idf(term)
        return idf_val * (self.k1 + 1) * tf_saturated

    def search(self, query: str, top_k: int = 5) -> List[Tuple[float, int]]:
        """
        多词 OR 检索，对应 Lucene BooleanQuery(OR) + TopScoreDocCollector。
        AND 检索需在 posting 列表上做归并求交（见 Demo 3）。
        """
        query_terms = tokenize(query)
        scores: Dict[int, float] = defaultdict(float)

        for term in query_terms:
            if term not in self.index:
                continue
            for doc_id, tf in self.index[term]:
                scores[doc_id] += self.score_term(term, doc_id, tf)

        ranked = sorted(scores.items(), key=lambda x: -x[1])
        return [(score, doc_id) for doc_id, score in ranked[:top_k]]


def demo_tf_saturation():
    """展示 BM25 TF 饱和效应 vs TF-IDF 线性增长"""
    print("\n=== TF 饱和演示（k1=1.2, dl=avgdl）===")
    print("  TF →  BM25 TF_norm  vs  TF-IDF（线性）  比值")
    k1, b, avgdl, dl = 1.2, 0.75, 10.0, 10.0
    norm = k1 * (1.0 - b + b * dl / avgdl)
    for tf in [1, 2, 5, 10, 20, 50, 100]:
        tf_bm25 = tf / (tf + norm)
        print(f"  tf={tf:4d}  BM25={tf_bm25:.4f}  TF-IDF={tf:4d}  ratio={tf_bm25/tf:.6f}")
    print("  → BM25 在 tf≥10 后趋近饱和（约 0.89），TF-IDF 线性无界")


def main():
    print("=== 构建倒排索引 ===")
    index, doc_lengths = build_index(CORPUS)

    print(f"词典大小: {len(index)} 个词项")
    print("词典（前10项，按 df 倒序）：")
    sorted_terms = sorted(index.items(), key=lambda x: -len(x[1]))[:10]
    for term, postings in sorted_terms:
        print(f"  {term:15s} → df={len(postings):2d}, postings={postings}")

    print(f"\n文档长度: {doc_lengths}")
    print(f"平均文档长度 avgdl: {sum(doc_lengths)/len(doc_lengths):.2f}")

    bm25 = BM25(index, doc_lengths)

    print("\n=== 检索: 'retrieval' ===")
    for rank, (score, doc_id) in enumerate(bm25.search("retrieval"), 1):
        print(f"  Rank {rank}: doc={doc_id}, score={score:.4f}, "
              f"text='{CORPUS[doc_id][:55]}...'")

    print("\n=== 检索: 'information retrieval' ===")
    for rank, (score, doc_id) in enumerate(bm25.search("information retrieval"), 1):
        print(f"  Rank {rank}: doc={doc_id}, score={score:.4f}, "
              f"text='{CORPUS[doc_id][:55]}...'")

    demo_tf_saturation()


if __name__ == "__main__":
    main()
```

### Demo 2：PForDelta 压缩实现（与 Lucene PForUtil 对应）

依赖：Python 3.8+，`numpy`（`pip install numpy`）。

```python
"""
demo_pfordelta.py — PForDelta 压缩 posting list 的最小实现

对应 Lucene 源码：
    apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/lucene104/PForUtil.java
    apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/lucene104/ForUtil.java

运行步骤：
    pip install numpy
    python demo_pfordelta.py

预期输出（近似）：
    --- 压缩方案对比（256 个随机 docID 的 delta）---
    原始存储（int32）：1024 bytes
    FOR 编码：位宽=14 bits，压缩后 448 bytes，压缩比 43.75%
    PForDelta：位宽=6 bits，例外数=7，压缩后 206 bytes，压缩比 20.12%
"""

import math
import numpy as np
from typing import List, Tuple


def delta_encode(doc_ids: np.ndarray) -> np.ndarray:
    """
    Delta 编码：存储 d-gap（与前一个 docID 的差值）
    对应 Lucene Posting format 注释：
    "Calculate the difference between each document number and previous one,
     and get a d-gaps list (for the first document, use absolute value)"
    """
    deltas = np.empty_like(doc_ids)
    deltas[0] = doc_ids[0]
    deltas[1:] = np.diff(doc_ids)
    return deltas


def bits_required(value: int) -> int:
    """计算表示该值需要的最小位数，对应 Lucene PackedInts.bitsRequired()"""
    return int(value).bit_length()  # Python 3 内置，等价于 floor(log2(v))+1


def for_encode(values: List[int]) -> Tuple[int, int]:
    """
    Frame of Reference (FOR) 编码。
    对应 Lucene ForUtil.encode()：block 内所有整数用相同位宽。
    返回 (bit_width, compressed_bytes)
    """
    max_val = max(values)
    bit_width = bits_required(max_val)
    compressed_bytes = math.ceil(bit_width * len(values) / 8)
    return bit_width, compressed_bytes


def pfordelta_encode(values: List[int], max_exceptions: int = 7) -> Tuple[int, int, int]:
    """
    PForDelta 编码，对应 Lucene PForUtil.encode()。

    算法（逐步对照 Lucene 源码）：
    1. 统计位宽直方图（histogram[b] = b 位宽的整数数量）
    2. 从最高位宽向下扫描，找到例外数 <= max_exceptions 的最低位宽
       — Lucene 约束：最多降低 8 bits（patch 存 byte）
    3. 用更低位宽编码大多数值，例外值以 (index:1byte, value:1byte) 单独存储

    返回 (patched_bits, num_exceptions, compressed_bytes)
    """
    # Step 1: 位宽直方图（对应 Lucene histogram[]）
    histogram = [0] * 32
    max_bits = 0
    for v in values:
        b = bits_required(v) if v > 0 else 0
        histogram[b] += 1
        max_bits = max(max_bits, b)

    # Step 2: 找最低有效位宽
    min_bits = max(0, max_bits - 8)   # "can't decrease bits by more than 8"
    cumulative_exceptions = 0
    patched_bits = max_bits
    num_exceptions = 0

    for b in range(max_bits, min_bits - 1, -1):
        if cumulative_exceptions > max_exceptions:
            break
        patched_bits = b
        num_exceptions = cumulative_exceptions
        cumulative_exceptions += histogram[b]

    # Step 3: 计算压缩字节数
    n = len(values)
    main_bits = patched_bits * n
    exception_bytes = num_exceptions * 2  # (1-byte index, 1-byte patch)
    compressed_bytes = math.ceil(main_bits / 8) + exception_bytes

    return patched_bits, num_exceptions, compressed_bytes


def main():
    print("=== PForDelta 压缩演示（对应 Lucene PForUtil）===\n")

    np.random.seed(42)
    # 模拟 256 个 docID（从 100 万中随机选），计算 d-gap
    doc_ids = np.sort(np.random.choice(1_000_000, size=256, replace=False))
    deltas = delta_encode(doc_ids)

    print(f"原始 docID 范围: [{doc_ids[0]}, {doc_ids[-1]}]")
    print(f"Delta 值范围: [{deltas.min()}, {deltas.max()}]")
    print(f"Delta 中位数: {np.median(deltas):.1f}")

    delta_list = [int(v) for v in deltas]
    naive_bytes = len(delta_list) * 4  # 原始 int32

    for_bits, for_bytes = for_encode(delta_list)
    pfor_bits, n_exc, pfor_bytes = pfordelta_encode(delta_list)

    print(f"\n--- 压缩方案对比（{len(delta_list)} 个 delta 值）---")
    print(f"原始存储（int32）：{naive_bytes} bytes")
    print(f"FOR 编码：位宽={for_bits} bits，压缩后 {for_bytes} bytes，"
          f"压缩比 {for_bytes/naive_bytes:.2%}")
    print(f"PForDelta：位宽={pfor_bits} bits，例外数={n_exc}，"
          f"压缩后 {pfor_bytes} bytes，压缩比 {pfor_bytes/naive_bytes:.2%}")

    # Delta 位宽分布直方图
    print(f"\n--- Delta 位宽分布（直方图）---")
    histogram = [0] * 25
    for v in delta_list:
        b = bits_required(v) if v > 0 else 0
        histogram[b] += 1
    for bits, count in enumerate(histogram):
        if count > 0:
            bar = "█" * (count // 2)
            print(f"  {bits:2d} bits: {count:4d} 个值  {bar}")

    # Lucene 特殊情况：全相同值优化
    print(f"\n=== allEqual 全相同值优化（Lucene 特殊情况）===")
    same_values = [4] * 256
    for_bits2, for_bytes2 = for_encode(same_values)
    print(f"256 个相同值(4)：FOR 位宽={for_bits2}，压缩后 {for_bytes2} bytes")
    print(f"Lucene 检测到 allEqual() 时，只存 1 个值 + 元数据 ≈ 2 bytes（97% 压缩）")


if __name__ == "__main__":
    main()
```

### Demo 3：多级跳表加速 AND 查询（与 MultiLevelSkipListReader 对应）

依赖：Python 3.8+，无额外包。

```python
"""
demo_skiplist.py — 多级跳表加速 AND 查询（两词求交）

对应 Lucene 源码：
    apache/lucene@lucene/core/src/java/org/apache/lucene/codecs/MultiLevelSkipListReader.java
    apache/lucene@lucene/core/src/java/org/apache/lucene/search/ConjunctionUtils.java

关键设计：advance() 维护游标状态（_pos），每次从上次位置继续前进，
而非每次从头扫描跳表节点——这是 Lucene Iterator 的核心契约：只能向前、不可后退。

跳表加速的场景：适用于两个列表长度差异大（高不对称比）的 AND 查询。
当两个列表长度相近时，朴素 O(m+n) 归并反而更快（skip-list 节点扫描有开销）。
真正的加速发生在短列表（罕见词）驱动长列表（常见词）时，每次 advance() 跳过 √P 个元素。

运行步骤：
    python demo_skiplist.py

预期输出（近似）：
    N=1,000,000, |A|=100 (罕见词), |B|=50,000 (常见词)
    朴素 AND 归并：比较次数 ≈ 48434, 交集大小 ≈ 5
    跳表 AND 求交：比较次数 ≈ 4367, 交集大小 ≈ 5
    加速比: 11.1x
    结果一致性验证: True

    结论：不对称比（50000:100=500x）越大，跳表优势越显著。
    对称比（5000:1000=5x）时跳表无显著加速，反而因节点扫描有额外开销。
"""

import math
import random
from bisect import bisect_left
from typing import List, Optional, Tuple


class PostingListWithSkip:
    """
    带两级跳表的 Posting 列表（有状态 Iterator）。

    对应 Lucene MultiLevelSkipListReader + PostingsEnum：
    - 游标 _pos 只能向前移动（Lucene Iterator 契约）
    - Level 0：每 skip_interval 个文档一个跳表节点（节点存块末最大 docID + 块末索引）
    - Level 1：每 skip_interval 个 Level-0 节点一个节点（指数层级）
    - advance(target) 两阶段：从高层找最远可跳位置（上升），再线性扫块内（下降）

    Lucene 注释（MultiLevelSkipListWriter javadoc）：
        Level 0 skip data interleaved between every packed block (256 docs),
        Level 1 skip data between every 32 packed blocks.
    """

    def __init__(self, doc_ids: List[int], skip_interval: int = 16):
        self.doc_ids = sorted(doc_ids)
        self.skip_interval = skip_interval
        # 游标：_pos 是下一个待读的 doc_ids 索引（只能增大，对应 Lucene Iterator 契约）
        self._pos = 0
        # 跳表节点：[(最大 docID，块末索引), ...]
        self.skip_levels = self._build_skip_levels()

    def _build_skip_levels(self):
        levels = {}
        n = len(self.doc_ids)
        level0 = []
        for i in range(self.skip_interval - 1, n, self.skip_interval):
            # 每块末尾：(该块最大 docID, 该 docID 在 doc_ids 中的索引)
            level0.append((self.doc_ids[i], i))
        levels[0] = level0
        level1 = []
        for i in range(self.skip_interval - 1, len(level0), self.skip_interval):
            level1.append(level0[i])
        levels[1] = level1
        return levels

    def advance(self, target: int, compare_counter: List[int]) -> Optional[int]:
        """
        有状态 advance(target)：从当前游标位置找 >= target 的最小 docID。
        对应 Lucene MultiLevelSkipListReader.skipTo(target)。

        算法：
        1. 从最高层跳表节点开始，找最远的"节点最大 docID < target"的位置（上升阶段）
           — 每层只需扫当前游标所在块后面的节点，不需从头扫
        2. 更新游标 _pos 到跳过块的末尾 + 1（进入下一块开头）
        3. 线性扫描最多 skip_interval 个元素（下降阶段）

        关键：由于 advance() 只能向前，每次都从 _pos 继续，不回头，
        所以总比较次数是 O(P/skip_interval * L + L * skip_interval) = O(P/skip_interval + skip_interval)
        与朴素 O(P) 相比在 skip_interval = √P 时最优，降为 O(√P)。
        """
        # Phase 1：从高层跳表节点中找可跳过的最远位置
        best_jump_pos = self._pos  # 最远可跳到的 doc_ids 起始索引

        for level in sorted(self.skip_levels.keys(), reverse=True):
            for skip_doc, skip_idx in self.skip_levels[level]:
                # 只考虑当前游标之后的节点
                if skip_idx < self._pos:
                    continue
                compare_counter[0] += 1
                if skip_doc < target:
                    # 这个块全都 < target，可以跳过整块
                    best_jump_pos = max(best_jump_pos, skip_idx + 1)
                else:
                    break  # 后续节点只会更大，无需继续（跳表是有序的）

        self._pos = best_jump_pos

        # Phase 2：从 _pos 线性扫描（最多 skip_interval 个元素）
        while self._pos < len(self.doc_ids):
            compare_counter[0] += 1
            if self.doc_ids[self._pos] >= target:
                return self.doc_ids[self._pos]
            self._pos += 1

        return None  # 列表已耗尽

    def reset(self):
        """重置游标（用于重新遍历，对应 Lucene 重新打开 PostingsEnum）"""
        self._pos = 0


def naive_and_intersect(list1: List[int], list2: List[int]) -> Tuple[List[int], int]:
    """朴素 AND 归并，O(m+n)，对应 Lucene ConjunctionScorer 最简实现"""
    i = j = 0
    result, comparisons = [], 0
    while i < len(list1) and j < len(list2):
        comparisons += 1
        if list1[i] == list2[j]:
            result.append(list1[i])
            i += 1
            j += 1
        elif list1[i] < list2[j]:
            i += 1
        else:
            j += 1
    return result, comparisons


def skiplist_and_intersect(
    pl1: PostingListWithSkip, pl2: PostingListWithSkip
) -> Tuple[List[int], int]:
    """
    跳表加速的 AND 交集（"小列表驱动大列表"策略）。
    对应 Lucene ConjunctionScorer：lead scorer 遍历最短列表，
    其余 scorer 调用 advance() 跳转到相同 docID。
    """
    # 选最短列表作为驱动（Lucene 选最小 df 的词项作为 lead scorer）
    if len(pl1.doc_ids) > len(pl2.doc_ids):
        pl1, pl2 = pl2, pl1

    pl1.reset()
    pl2.reset()
    result, counter = [], [0]

    for doc_id in pl1.doc_ids:
        # 用跳表在 pl2 中定位 >= doc_id 的位置
        found = pl2.advance(doc_id, counter)
        counter[0] += 1  # 驱动列表的顺序读取也算一次
        if found == doc_id:
            result.append(doc_id)
        elif found is None:
            break  # pl2 已耗尽，后续不可能有交集

    return result, counter[0]


def main():
    random.seed(42)
    N = 1_000_000  # 大语料库（保证不对称比够大）

    # 高不对称场景：df=100 罕见词 vs df=50000 常见词（比 500:1）
    # 这正是"bursty term" AND "common term"的典型搜索场景
    docs_a = sorted(random.sample(range(N), 100))      # 罕见词：df=100
    docs_b = sorted(random.sample(range(N), 50_000))   # 常见词：df=50000

    print(f"N={N:,}, |A|={len(docs_a)} (罕见词), |B|={len(docs_b)} (常见词)")
    print(f"不对称比: {len(docs_b) // len(docs_a)}:1")
    print(f"理论交集大小（估算）: ~{len(docs_a) * len(docs_b) // N} 个")

    result_naive, comp_naive = naive_and_intersect(docs_a, docs_b)
    print(f"\n朴素 AND 归并：比较次数 = {comp_naive}, 交集大小 = {len(result_naive)}")

    pl_a = PostingListWithSkip(docs_a, skip_interval=16)
    pl_b = PostingListWithSkip(docs_b, skip_interval=16)
    result_skip, comp_skip = skiplist_and_intersect(pl_a, pl_b)
    print(f"跳表 AND 求交：比较次数 = {comp_skip}, 交集大小 = {len(result_skip)}")
    print(f"\n加速比: {comp_naive / comp_skip:.1f}x")
    print(f"结果一致性验证: {result_naive == result_skip}")

    print(f"\n--- 跳表结构（pl_b, skip_interval=16）---")
    for level, nodes in pl_b.skip_levels.items():
        print(f"  Level {level}: {len(nodes)} 个跳表节点，"
              f"块间距={pl_b.skip_interval ** (level + 1)} docs")

    # skip_interval 敏感性分析（验证 √P 最优理论）
    print(f"\n--- skip_interval 敏感性分析（|B|={len(docs_b)}, 理论最优 √P = {int(math.sqrt(len(docs_b)))}）---")
    for si in [4, 8, 16, 32, 64, 128, 224]:
        pl_a2 = PostingListWithSkip(docs_a, si)
        pl_b2 = PostingListWithSkip(docs_b, si)
        _, comp = skiplist_and_intersect(pl_a2, pl_b2)
        mark = " ← 接近最优" if abs(si - int(math.sqrt(len(docs_b)))) < 50 else ""
        print(f"  skip_interval={si:4d}: 比较次数={comp:6d}{mark}")

    print(f"\n--- 对比：低不对称比（5000:1000=5:1，朴素更快）---")
    docs_c = sorted(random.sample(range(N), 5000))
    docs_d = sorted(random.sample(range(N), 1000))
    _, cn2 = naive_and_intersect(docs_c, docs_d)
    pl_c = PostingListWithSkip(docs_c, 16)
    pl_d = PostingListWithSkip(docs_d, 16)
    _, cs2 = skiplist_and_intersect(pl_c, pl_d)
    print(f"  朴素 AND：{cn2} 次比较")
    print(f"  跳表 AND：{cs2} 次比较")
    print(f"  结论：低不对称比时朴素 O(m+n) 更优，跳表节点扫描引入额外开销")


if __name__ == "__main__":
    main()
```

### Demo 4：WAND Block-Max 剪枝（与 BlockMaxConjunctionScorer 对应）

WAND（Weak AND）是现代搜索引擎优化 Top-K 检索的核心机制。这个 demo 展示其核心思路：维护一个 minCompetitiveScore（当前 Top-K 堆的最小分），用每个 posting 块的 maxScore 上界快速跳过无竞争力的块。

依赖：Python 3.8+，标准库 `heapq`、`math`。无需安装额外包。

```python
"""
demo_wand.py — WAND Block-Max 剪枝最小实现

对应 Lucene 源码：
    apache/lucene@lucene/core/src/java/org/apache/lucene/search/BlockMaxConjunctionScorer.java
    apache/lucene@lucene/core/src/java/org/apache/lucene/search/WANDScorer.java

WAND 核心不变量：
    BM25 doScore() = weight - weight / (1 + freq * normInverse)
    该函数对 freq 和 normInverse 均单调递增，
    因此 score(maxFreqInBlock, minNormInverseInBlock) 是块内所有文档得分的上界。

运行步骤：
    python demo_wand.py

预期输出（近似）：
    === 暴力 Top-3（baseline）===
    基线结果: [(score, doc_id), ...]
    比较次数（暴力）: 10000

    === WAND Block-Max Top-3 ===
    WAND 结果: 与基线一致
    比较次数（WAND）: ~800
    跳过的块数: ~150 / 200 total
    加速比: ~12.5x
"""

import heapq
import math
import random
from typing import List, Tuple


# ── BM25 参数（与 Lucene 默认值一致）────────────────────────────────
K1 = 1.2
B = 0.75
AVGDL = 50.0


def bm25_score(tf: float, dl: float, idf: float) -> float:
    """
    Lucene BM25Similarity.doScore() 的等价实现。
    单调性保证：score 对 tf 和 idf 单调递增，对 dl 单调递减。
    """
    norm_inverse = 1.0 / (K1 * (1.0 - B + B * dl / AVGDL))
    weight = idf * (K1 + 1)
    return weight - weight / (1.0 + tf * norm_inverse)


# ── 模拟 Posting Block 结构 ───────────────────────────────────────
class PostingBlock:
    """
    对应 Lucene PackedBlock（256 个 docID 一块）。
    每块记录 max_score（块内所有文档得分的上界）用于 WAND 剪枝。

    max_score 计算：对块内所有文档取最高 TF，最长文档（最低 normInverse）
    实际上这是一个悲观估计（上界），正好满足 WAND 剪枝条件。
    """
    def __init__(self, docs: List[Tuple[int, float, float]], idf: float):
        # docs: [(docID, tf, dl), ...]
        self.docs = docs
        self.idf = idf
        # 块内 max_score：取 max_tf（最大 TF）+ min_dl（最短文档，最高 normInverse）
        # 对应 Lucene Scorer.getMaxScore(upTo)
        max_tf = max(d[1] for d in docs)
        min_dl = min(d[2] for d in docs)
        self.max_score = bm25_score(max_tf, min_dl, idf)


def build_posting_blocks(doc_count: int, avg_tf: float, idf: float,
                         block_size: int = 50) -> List[PostingBlock]:
    """
    构建模拟的 posting blocks（每块 block_size 个 docID）。
    真实 Lucene 块大小为 256 个 docID（Lucene912PostingsFormat）。
    这里用 50 便于演示。
    """
    random.seed(42)
    blocks = []
    for start in range(0, doc_count, block_size):
        end = min(start + block_size, doc_count)
        docs = []
        for doc_id in range(start, end):
            tf = max(1, int(random.gauss(avg_tf, avg_tf * 0.5)))  # 随机 TF
            dl = max(10, int(random.gauss(AVGDL, AVGDL * 0.3)))    # 随机文档长度
            docs.append((doc_id, float(tf), float(dl)))
        blocks.append(PostingBlock(docs, idf))
    return blocks


def brute_force_top_k(
    posting_blocks: List[PostingBlock], top_k: int
) -> Tuple[List[Tuple[float, int]], int]:
    """
    暴力 Top-K：遍历所有文档，不做任何剪枝。
    对应 Lucene 禁用 WAND 后的行为。
    """
    comparisons = 0
    heap: List[Tuple[float, int]] = []

    for block in posting_blocks:
        for doc_id, tf, dl in block.docs:
            score = bm25_score(tf, dl, block.idf)
            comparisons += 1
            if len(heap) < top_k:
                heapq.heappush(heap, (score, doc_id))
            elif score > heap[0][0]:
                heapq.heapreplace(heap, (score, doc_id))

    return sorted(heap, reverse=True), comparisons


def wand_top_k(
    posting_blocks: List[PostingBlock], top_k: int
) -> Tuple[List[Tuple[float, int]], int, int]:
    """
    WAND Block-Max Top-K。

    核心逻辑（对应 Lucene BlockMaxConjunctionScorer）：
    1. 维护 minCompetitiveScore = Top-K 堆中最小分（初始 = 0）
    2. 对每个 posting block，检查其 max_score：
       - 如果 block.max_score <= minCompetitiveScore：整块跳过（WAND 剪枝）
       - 否则：进入块内精确打分，更新 minCompetitiveScore
    3. 随着堆填满，minCompetitiveScore 上升，越来越多的块被跳过

    Lucene 不变量：BM25 doScore() 对 freq/norm 单调 → block.max_score 是真正上界
    """
    min_competitive_score = 0.0
    heap: List[Tuple[float, int]] = []
    comparisons = 0
    blocks_skipped = 0
    blocks_total = len(posting_blocks)

    for block in posting_blocks:
        # WAND 剪枝：块级 maxScore 检查
        if block.max_score <= min_competitive_score:
            blocks_skipped += 1
            continue  # 整块跳过，O(1) 操作

        # 进入块内精确打分
        for doc_id, tf, dl in block.docs:
            score = bm25_score(tf, dl, block.idf)
            comparisons += 1
            if len(heap) < top_k:
                heapq.heappush(heap, (score, doc_id))
                if len(heap) == top_k:
                    min_competitive_score = heap[0][0]  # 堆满后才有有效阈值
            elif score > heap[0][0]:
                heapq.heapreplace(heap, (score, doc_id))
                min_competitive_score = heap[0][0]      # 更新竞争阈值

    result = sorted(heap, reverse=True)
    print(f"  跳过的块数: {blocks_skipped} / {blocks_total} total "
          f"({blocks_skipped/blocks_total:.1%})")
    return result, comparisons, blocks_skipped


def main():
    DOC_COUNT = 10_000
    TOP_K = 10
    IDF = 2.5       # 模拟一个中等频率词的 IDF
    AVG_TF = 3.0    # 平均词频

    posting_blocks = build_posting_blocks(DOC_COUNT, AVG_TF, IDF, block_size=50)
    print(f"文档总数: {DOC_COUNT}, 块数: {len(posting_blocks)}, "
          f"块大小: 50, Top-K: {TOP_K}")

    print("\n=== 暴力 Top-K（baseline）===")
    brute_result, brute_comp = brute_force_top_k(posting_blocks, TOP_K)
    print(f"  比较次数（暴力）: {brute_comp}")
    print(f"  Top-3 结果: {[(f'{s:.3f}', d) for s, d in brute_result[:3]]}")

    print("\n=== WAND Block-Max Top-K ===")
    wand_result, wand_comp, skipped = wand_top_k(posting_blocks, TOP_K)
    print(f"  比较次数（WAND）: {wand_comp}")
    print(f"  Top-3 结果: {[(f'{s:.3f}', d) for s, d in wand_result[:3]]}")
    print(f"\n  加速比: {brute_comp / wand_comp:.1f}x")

    # 验证结果一致性
    brute_set = {(round(s, 4), d) for s, d in brute_result}
    wand_set = {(round(s, 4), d) for s, d in wand_result}
    print(f"  结果一致性验证: {brute_set == wand_set}")

    # 展示 minCompetitiveScore 对剪枝的影响
    print(f"\n--- WAND 单调性说明 ---")
    print("  BM25 doScore() = weight - weight / (1 + freq * normInverse)")
    print("  该函数对 freq 严格单调递增 → block.max_score = score(max_tf, min_dl) 是真正上界")
    print("  随着 Top-K 堆填满，minCompetitiveScore 上升，越来越多块被跳过")
    print("  这正是 Lucene 选择等价变换而非原始除法形式的原因：保证 float 单调性")


if __name__ == "__main__":
    main()
```

**关键洞察**：`BM25 doScore() = weight - weight / (1 + freq * normInverse)` 这个等价变换形式不仅是数值稳定性的考量，更是 WAND 正确性的**前提**——只有当打分函数对 freq 单调时，`score(max_tf_in_block)` 才是块内所有文档得分的真正上界，WAND 剪枝才是无损的。

## 4. 方案对比与工程选型

### 4.1 检索模型对比

| 模型 | 召回质量 | 延迟 | 内存 | 不适用场景 |
|------|---------|------|------|-----------|
| 布尔检索 | 无排序 | O(df) | 低 | 需要排序的任何场景 |
| TF-IDF | 中等 | O(df) | 低 | 长文档（TF 线性无饱和）|
| BM25（k1=1.2, b=0.75）| 较好 | O(df) | 低 | 语义相似、词汇不重叠 |
| BM25+（δ=1.0）| 更好（长文档）| O(df) | 低 | 短查询（δ 效果不显著）|
| 向量检索（Dense）| 语义最佳 | O(N) 暴力 / O(log N) ANN | 高 | 精确词项匹配（如 ID）|
| 混合（BM25 + Dense）| 最优 | 中 | 高 | 资源极度受限场景 |

### 4.2 倒排索引压缩方案对比

| 方案 | 压缩比 | 解码速度 | 随机访问 | 适用 block 大小 |
|------|--------|---------|---------|---------------|
| VByte | 中（1.5–2.5 bytes/int）| 较快 | 支持（需扫描）| 任意 |
| FOR（128 整数）| 较好 | 快（SIMD）| 块对齐 | 64-256 |
| PForDelta（256 整数）| 好（1.2–2.0 bytes）| 很快（SIMD + patch）| 块对齐 | 256 |
| SIMD-BP128 | 很好 | 极快（128-bit SSE）| 块对齐 | 128 |

Lucene 的选择：block_size=256 的 PForDelta（`PForUtil`），同时保留 VInt 编码用于尾部不足一块的情况（`VIntBlock`）。

### 4.3 参数调优指南

**k1 选择**：
- `k1 = 0`：退化为纯 IDF（忽略 TF），适合 title 字段
- `k1 ∈ [1.2, 2.0]`：工业默认范围
- `k1 → ∞`：退化为线性 TF-IDF
- 经验：短查询（1–3词）用 k1=1.2；长查询（文档级查询）用 k1=1.6–2.0

**b 选择**：
- `b = 0`：不做文档长度归一化（BM15）
- `b = 0.75`：TREC 数据集经验最优（正文字段）
- `b = 1`：完全归一化（BM11），长文档被强惩罚
- 经验：标题字段（短且长度相似）用 b=0.3–0.5；正文字段用 b=0.75

## 5. 失败模式、真坑与根因

### 坑 1：词汇不匹配（Vocabulary Mismatch）

**现象**：查询 "heart attack" 找不到含 "myocardial infarction" 的文档。

**根因**：倒排索引是精确词项匹配，BM25 无法理解语义近义词。

**解决**：查询扩展（Query Expansion）、同义词词典注入（Analyzer 层），或混合向量检索（BM25 + Dense Retrieval，Reciprocal Rank Fusion）。

### 坑 2：IDF 稳定性问题（冷启动 / 小语料）

**现象**：索引文档数 < 1000 时，BM25 得分极不稳定；罕见词 IDF 过高导致排序异常。

**根因**：`log((N - df + 0.5) / (df + 0.5))` 在 N 小时波动剧烈。df=1 时 IDF 约为 log(N)，N=100 则 IDF≈4.6，N=1000 则 IDF≈6.9，同一文档在不同规模语料下得分差异 50%+。

**解决**：使用大语料先验 IDF（固定 IDF，不随在线索引变化）；或使用 BM25F（字段加权版）稳定不同字段的 IDF 估计。

### 坑 3：文档长度 norm 的量化误差

**现象**：同长度的两篇文档，BM25 得分不同（相差可达 12.5%）。

**根因**：Lucene 将文档长度压缩为 1 字节（SmallFloat，3 位 mantissa），最大相对误差 12.5%。

**影响**：不影响大多数 Top-K 结果，但在得分非常接近的文档之间可能改变排序。`intToByte4`（4 位 mantissa）在较新版本中降低了误差。

**应对**：接受此误差（通常可以），或自定义 Similarity 用 4 字节存储 norm（显著增加内存）。

### 坑 4：OR 查询的 WAND 失效

**现象**：OR 查询（所有包含任意查询词的文档）WAND 剪枝效果不显著。

**根因**：OR 查询需要遍历所有匹配文档，WAND 的"块 maxScore ≤ minCompetitiveScore"条件在 Top-K 堆填满前很少成立——因为 OR 查询匹配率高，前几千文档就把堆填满了，但此后堆内最小分通常不够高。

**解决**：OR 查询只能靠更高效的 posting 解码（SIMD FOR）加速，而不是 WAND 跳表。真正大 OR 查询使用 `MaxScoreScorer`（Lucene 的 WAND 变体）。

### 坑 5：Lucene 段合并导致 IDF 漂移

**现象**：同一文档在段合并前后 BM25 得分不同。

**根因**：Lucene 的 IDF 是 **per-segment** 计算的，不同段的 docCount 不同，导致 IDF 值不同。段合并后 IDF 重新计算，得分发生变化（通常不超过 10-15%）。

**验证**：通过 `IndexSearcher.explain()` 查看每段的 IDF 值。

**解决**：如需跨段一致 IDF，使用 `IndexSearcher.createWeight()` 传入全局统计；或在 Elasticsearch 中使用 `dfs_query_then_fetch`（先全局收集 DF，再打分）。

### 坑 6：b=0.75 不适合所有字段

**现象**：对短字段（标题、URL slug）启用 b=0.75 会不合理地惩罚短标题，使长标题反而排在更前。

**根因**：b=0.75 是为正文长文本设计的，隐含假设是"文档长度是内容量的代理"。标题字段长度差异小，此假设不成立。

**解决**：用 `PerFieldSimilarityWrapper` 为不同字段配置不同 b 值；或对标题字段使用 b=0（BM15）。

### 扩展：WAND / Block-Max 剪枝机制深度

WAND 的关键不变量：BM25 `doScore()` 对 freq 和 norm 都必须单调——这正是 Lucene 选择 `weight - weight / (1 + freq * normInverse)` 而非等价但不便于单调分析的原始除法形式的原因：只有单调函数，块级 maxScore 才是真正的上界，WAND 剪枝才无损。

```java
// 对应 Lucene BlockMaxConjunctionScorer（来自 WebFetch 核实描述）：

// 1. Block 边界管理
if (target > upTo) {
    moveToNextBlock(target);
}

// 2. 块级竞争检查：maxScore < minScore 时整块跳过
while (maxScore < minScore) {
    moveToNextBlock(nextBlockStart);
}

// 3. 多词 Conjunction：lead scorer 定位后，其余 iterator advance() 追赶
for (DisiWrapper w : others) {
    int advanced = w.iterator.advance(doc);
    if (advanced != doc) {
        doc = lead.iterator.advance(advanced);  // 未命中，继续跳
        restart = true;
    }
}

// 4. getMaxScore()：各 scorer 上界求和
float maxScore = 0;
for (Scorer s : scorers) {
    maxScore += s.getMaxScore(upTo);
}
```

实测在 TREC-style 多词查询上，WAND 可跳过 80–95% 的候选文档（"cold start" 阶段堆未满时所有块都要扫；一旦堆满，minScore 迅速上升，绝大多数低分块被整体跳过）。

## 6. 章末五件套

### 6.1 核心概念总结

```
Lucene 索引写入链路：
  IndexWriter → DefaultIndexingChain
    → TermsHashPerField.add()         # token → BytesHash 去重 + slice 分配
    → FreqProxTermsWriterPerField      # 写 TF delta + position VInt
    → flush → .doc/.pos/.pay 文件     # FOR/PForDelta 块压缩 + 跳表

Lucene 查询链路：
  Query → Weight → Scorer
    → MultiLevelSkipListReader        # 跳表加速 docID 定位
    → TermScorer.score()              # 从 posting 读 freq + norm
    → BM25Similarity.doScore()        # weight - weight/(1+freq*normInverse)
    → WAND/BlockMaxConjunctionScorer  # 块级 maxScore 剪枝，跳过 80-95% 块
```

### 6.2 公式卡片

```
BM25(D, Q) = Σ_{t∈Q}  idf(t) × (k₁+1)×f(t,D) / [f(t,D) + k₁×(1-b + b×|D|/avgdl)]

idf(t)     = log(1 + (N - df(t) + 0.5) / (df(t) + 0.5))
             └ +0.5 是 Bayes 平滑，+1 保证 IDF ≥ 0

Lucene doScore() 等价变换（保持 float 单调性 → WAND 剪枝正确性前提）：
    score = weight - weight / (1 + freq × normInverse)
    where weight       = boost × idf × (k1 + 1)
          normInverse  = 1 / (k₁ × (1 - b + b × dl / avgdl))  ← 预计算 256 个

SmallFloat 压缩：文档长度 → 1 byte，最大相对误差 12.5%
PForDelta：block_size=256，最多 7 个 patch，最多降低 8 bits
跳表：Level 0 每 skipInterval 个 doc，Level i 间距指数增长
WAND 剪枝：block.maxScore ≤ minCompetitiveScore → 整块 O(1) 跳过
```

### 6.3 代码题（扩展 Demo）

1. **Demo 1 扩展**：实现 AND 模式——多词必须同时出现才返回，与 OR 模式对比结果差异；在 posting 求交处用 Demo 3 的跳表 AND 替换朴素归并，比较比较次数。

2. **实现 BM25+**（Lv et al., 2011，CIKM）：在 `score_term()` 中加入下界参数 δ=1.0：
   ```python
   tf_norm_plus = tf / (tf + norm) + delta
   score_plus = idf * (k1 + 1) * tf_norm_plus
   ```
   观察对长文档（tf 较高时）和短文档（tf=1 时）的不同影响。

3. **压缩率实验（Demo 2 扩展）**：对比 uniform random d-gap（均匀分布）vs power-law d-gap（幂律分布，`np.random.zipf(a=1.5, size=256)`，更接近真实 posting 列表）的 PForDelta 压缩率差异——真实语料中 d-gap 分布是幂律的，PForDelta 在此场景下效果显著优于 uniform。

4. **WAND 阈值演化曲线（Demo 4 扩展）**：在 `wand_top_k()` 中记录每处理 100 个块时的 `minCompetitiveScore` 和累计跳过块数，画出阈值随块序号的上升曲线——直观展示"cold start 问题"（前期阈值低，几乎不能剪枝；堆满后阈值跳升，剪枝效率骤增）。
