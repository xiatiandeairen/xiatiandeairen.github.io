> **一句话定位**：内核 C 不是普通 C——它在极端约束下（无 MMU 保护、无浮点、栈极小、不可随意睡眠）构造出一套独特的编程范式，掌握它是读懂任何内核子系统源码的基础通行证。

---

## TL;DR — 本章最硬结论

1. **侵入式链表（intrusive list）** 是内核的核心组织方式：`list_head` 嵌入对象而非包含对象，`container_of` 反向导航，性能与类型无关性兼得。
2. **内核栈极小**：x86_64 近代默认 16 KB（`THREAD_SIZE = 4 页`），32 位历史上 4/8 KB。这不是保守估计，是写死的物理约束——溢出后果取决于内核版本：现代 x86_64（4.9+ `CONFIG_VMAP_STACK`）栈用独立 vmalloc 区 + guard page，溢出立即 double-fault；老内核栈紧邻 `thread_info`，溢出会静默破坏它再延迟崩溃（注意：4.9+ `thread_info` 已搬进 `task_struct`，不再驻留栈底——见 §1.2 演进注）。
3. **`ERR_PTR`/`PTR_ERR`** 把错误码塞进指针低位：不用额外 out-param，但不可和真实内核地址混用——这是内核源码里随处可见的约定，看不懂会懵。
4. **per-CPU 变量** 是内核规避锁的核心武器：每个 CPU 有独立副本，访问路径上关抢占即可，延迟比 spinlock 低一个数量级。
5. **内核内存不缺页**：内核地址空间直接映射到物理内存（highmem 机制除外），内核代码路径里触发缺页通常是 bug，而非正常的懒分配。

## 1. 背景：内核 C 的约束

### 1.1 内核 C 与用户态 C 的本质差异

```
用户态 C                        内核 C
─────────────────────────────────────────────────────
glibc 兜底，可用 printf/malloc    只有 printk/kmalloc，自己造轮子
缺页 = OS 帮你补物理页            缺页 = 可能是 bug
浮点寄存器随意用                  不能用 FPU（需显式 kernel_fpu_begin）
栈几 MB，随便深递归              栈 16 KB(x86_64)，深递归溢出即 double-fault
睡眠=调用阻塞 syscall 即可       中断上下文/持有 spinlock 时绝对不能睡眠
标准库提供通用容器（STL/glib）    自己实现侵入式链表、红黑树、基数树
```

### 1.2 无浮点约束

内核在绝大多数上下文下不保存/恢复 FPU/SIMD 状态（进出内核时保存全套 XMM/YMM/ZMM 开销太高，且内核路径基本不需要浮点）。少数确需 SIMD 的场景——RAID6 校验、crypto、部分 memcpy 优化——必须用 `kernel_fpu_begin()` / `kernel_fpu_end()` 显式圈出临界区（声明在 `arch/x86/include/asm/fpu/api.h`，这对函数本身禁止抢占/睡眠）。这两个符号高把握确认存在。

**工程推论**：
1. 内核 Makefile 用编译选项禁掉编译器自动发射 SIMD/FPU 指令——x86 上是 `-mno-sse -mno-mmx -mno-sse2 -mno-3dnow` 这一组（部分架构另有 `-msoft-float`）；否则编译器可能为结构体拷贝偷偷用 XMM，破坏"不碰 FPU"的前提。
2. 大量算法要改写为定点数或整数近似（如调度器权重计算、网络拥塞控制的带宽估算）。

> **演进注（thread_info 搬家，约 4.9 / 2016）**：早期 `thread_info` 嵌在内核栈底部，`current` 宏靠"栈指针对齐到 `THREAD_SIZE` 再取栈底"来定位当前任务，所以栈溢出会先砸烂 `thread_info`。x86_64 在 4.9 引入 `CONFIG_THREAD_INFO_IN_TASK`，把 `thread_info` 移进 `task_struct` 开头，`current` 改由 per-CPU 变量（`current_task`，经 `%gs` 访问）给出。配合同期的 `CONFIG_VMAP_STACK`（虚拟映射栈 + guard page），栈溢出从"静默腐蚀邻居"升级为"立即可诊断的 double-fault"。读老资料看到"溢出踩 thread_info"要知道那是旧模型。

### 1.3 不能随意睡眠

内核有两类执行上下文，规则截然不同：

```
进程上下文（task context）        中断上下文（interrupt context）
──────────────────────────────────────────────────────────────
可以睡眠（wait_event 等）         绝对不能睡眠
可以持有 mutex                    只能持有 spinlock
可以访问 current                  current 无意义
```

💡 **坑点**：持有 spinlock 即进入"不可睡眠"状态。机制上，spinlock 在 SMP 内核里是"真自旋 + 关抢占"，在 UP（单核）内核里退化成纯关抢占（自旋体被编译掉）——两种情形持锁期间都关了抢占。一旦睡眠就放弃 CPU、调度走别的任务，而别的任务若来抢同一把锁会原地自旋烧 CPU，持锁者又回不来释放，死锁。`might_sleep()` 宏在 debug（`CONFIG_DEBUG_ATOMIC_SLEEP`）下，会在"原子上下文里却调用了可能睡眠的函数"时打栈告警，生产前调试必开。
>
> 延伸：若这把 spinlock 还会被中断处理程序获取，光关抢占不够——本 CPU 的中断可能在你持锁时打进来、去抢同一把锁，造成本 CPU 自死锁。此时要用 `spin_lock_irqsave()`（持锁同时关本 CPU 中断）。这条规则在 §章末五件套 Q3 的 per-CPU 讨论里同源。

## 2. 侵入式链表：`list_head`

### 2.1 为什么要侵入式

传统链表（C++ STL `std::list`、Java `LinkedList`）：链表节点**包含**数据。
```
[data | next* | prev*] → [data | next* | prev*]
```
问题：每种数据类型需要独立链表实现，或用 `void*` 丧失类型安全。

Linux 内核的侵入式设计：数据对象**嵌入**链表节点。
```c
struct list_head {
    struct list_head *next, *prev;
};

struct task_struct {
    /* ... 大量字段 ... */
    struct list_head tasks;   // 进程链表钩子
    struct list_head children;
    struct list_head sibling;
    /* ... */
};
```

链表只连接 `list_head`，不知道宿主类型。

```
       list_head                list_head
  ┌──────────────┐         ┌──────────────┐
  │ task_struct  │ tasks.next ──▶ task_struct │
  │   ...        │ ◀── tasks.prev           │
  │ [list_head]  │         │ [list_head]  │
  └──────────────┘         └──────────────┘
```

### 2.2 `container_of` 宏：从成员指针反向找宿主

```c
#define container_of(ptr, type, member) ({              \
    const typeof(((type *)0)->member) *__mptr = (ptr);  \
    (type *)((char *)__mptr - offsetof(type, member)); })
```

**解读**：
- `offsetof(type, member)`：编译期计算成员在结构体内的字节偏移。
- 用指针减去偏移，得到结构体起始地址。
- 零开销：完全编译期展开，无运行时计算。

**使用示例**：
```c
struct list_head *pos;
list_for_each(pos, &task_list) {
    struct task_struct *t = list_entry(pos, struct task_struct, tasks);
    // list_entry 即 container_of 的包装
}
```

🎯 **面试加分点**：`container_of` 的类型检查技巧——`typeof(((type *)0)->member)` 利用 GCC 扩展在编译期验证 `ptr` 的类型与 `member` 的类型一致，若不匹配会触发警告，不是纯粹的 `void*` hack。

### 2.3 核心 API 速查

| API | 语义 |
|-----|------|
| `INIT_LIST_HEAD(head)` | 初始化空链表（head→next=head，head→prev=head，循环链表） |
| `list_add(new, head)` | 头插（O(1)，适合栈语义） |
| `list_add_tail(new, head)` | 尾插（O(1)，适合队列语义） |
| `list_del(entry)` | 删除节点（O(1)，但节点指针变为 `LIST_POISON`，防 use-after-free） |
| `list_for_each_entry(pos, head, member)` | 正向遍历，`pos` 直接是宿主类型指针 |
| `list_for_each_entry_safe(pos, n, head, member)` | 遍历时允许删除当前节点（用 `n` 保存 next） |

💡 **坑点**：`list_del` 后节点的 `next`/`prev` 被设为 `LIST_POISON1`/`LIST_POISON2`（非法地址），访问会立刻 oops，这是有意设计用于发现悬空引用 bug。

### 2.4 工程推论

1. 同一个对象可以同时挂在多条链表上（只需嵌入多个 `list_head` 字段）——`task_struct` 同时在进程链表、子进程链表、等待队列上，零额外开销。
2. 链表操作本身不加锁，调用方负责并发保护（spinlock 或 RCU，见第 13 章）。
3. 链表是双向循环链表，空链表 head→next == head，判空 O(1)。

### 2.5 `hlist`：哈希桶专用的省内存变体

`list_head` 的表头要存 `next` 和 `prev` 两个指针。哈希表往往有成千上万个桶，每个桶一个表头——这里多一个指针就是几十 KB 起步的浪费，且绝大多数桶是空的。于是内核为哈希表准备了 `hlist`：

```c
struct hlist_head { struct hlist_node *first; };          // 表头只有 1 个指针
struct hlist_node { struct hlist_node *next, **pprev; };   // 节点的 prev 是"指向前驱 next 字段的指针"
```

两个关键设计（staff 高频追问）：
- **表头单指针**：`hlist_head` 只存 `first`，桶数组体积直接砍半；空桶就是一个 `NULL`。
- **`pprev` 是二级指针**（`**`，指向"前一个节点的 `next` 字段"或"表头的 `first` 字段"）：这样删除一个节点时，无论它是不是头节点，都能用统一的 `*(node->pprev) = node->next` 一句话摘除，不需要为"删头节点"开特例分支。代价是 `hlist` 不是循环链表、不能 O(1) 反向遍历——但哈希桶只需正向扫，无所谓。

一句话记忆：**`list_head` 用于需要双向/循环遍历的有序队列；`hlist` 用于海量、多为空、只正向扫的哈希桶。** 这正是 P1-2 项目里那个问题的答案。

## 3. 红黑树 (`rb_tree`)：有序集合的主力

### 3.1 为什么内核需要红黑树

链表遍历是 O(n)，当集合需要快速按 key 查找时（如按虚拟地址查找 VMA、按超时时间管理 timer）链表不够。

内核选红黑树而非 AVL 树的理由：
- **写操作的再平衡上界更友好**：红黑树插入旋转 ≤2 次、删除旋转 ≤3 次（均为 O(1) 旋转）；AVL 插入也是 O(1) 旋转，但**删除最坏需要 O(log n) 次旋转**一路传播到根。频繁增删的场景（VMA、timer、调度）红黑树的写抖动更可控。
- 代价是查询：AVL 平衡更严格（高度 ≤1.44 lg n），查找略快；红黑树高度上界 2 lg n，查找略慢。内核判断"增删多、查询不极端",故选红黑树。
- 平均与最坏都是 O(log n)，常数因子小，颜色位还能塞进 `__rb_parent_color` 省内存（见 §3.2）。

**典型用途**（稳定）：
- `mm_struct.mm_rb`：进程的 VMA（虚拟内存区域）按地址排序（近年内核已迁移到 maple tree，见下节）
- CFS 调度器的运行队列：按 `vruntime` 排序（6.6 迁移到 EEVDF，但数据结构仍是 rb_tree）
- epoll 的 fd 管理

### 3.2 侵入式红黑树

和 `list_head` 一样，内核红黑树也是侵入式：

```c
struct rb_node {
    unsigned long  __rb_parent_color;  // 父指针 + 颜色位（利用指针对齐的低位）
    struct rb_node *rb_right;
    struct rb_node *rb_left;
};

struct rb_root {
    struct rb_node *rb_node;
};
```

💡 **坑点/精妙设计**：`__rb_parent_color` 把父节点指针和颜色位压缩在同一个字段里——x86_64 指针最低 2 位因对齐保证为 0，用最低位存红/黑色，节省 8 字节。这是内核"每字节都值得花心思"哲学的典型体现。

插入/查找需要调用方自己实现比较逻辑，内核只提供旋转/平衡原语。典型模式：

```c
// 插入：调用方负责找位置，内核负责再平衡
static int my_insert(struct rb_root *root, struct my_node *new)
{
    struct rb_node **link = &root->rb_node, *parent = NULL;
    while (*link) {
        struct my_node *entry = rb_entry(*link, struct my_node, node);
        parent = *link;
        if (new->key < entry->key)
            link = &(*link)->rb_left;
        else if (new->key > entry->key)
            link = &(*link)->rb_right;
        else
            return -EEXIST;
    }
    rb_link_node(&new->node, parent, link);
    rb_insert_color(&new->node, root);
    return 0;
}
```

## 4. 基数树与 Maple Tree

### 4.1 基数树 (`radix_tree` / `xarray`)

**用途**：将整数 key（如页缓存的 page index、inode 号）映射到指针，O(1) 均摊。

内核经历了三代演进：
1. `radix_tree`（早期）：64 叉树，按 6-bit 分组切割索引。
2. `xarray`（Linux 4.20+，Matthew Wilcox 主导）：`radix_tree` 的重新设计，API 更清晰，把一把 `xa_lock`（spinlock）直接嵌进 `struct xarray` 顶层结构、由数据结构自己持有，简化调用方并发管理，逐步替换 `radix_tree`。
3. 当前：page cache（`address_space.i_pages`）已全面使用 `xarray`。

🎯 **面试加分点**：`xarray` 的 `xa_lock` 是 spinlock，但 `xa_lock_irq` / `xa_lock_bh` 变体可以在关中断/关软中断情况下使用，API 层面就把并发上下文编码进去了，减少调用方犯错空间。

### 4.2 Maple Tree（Linux 6.1+）

**背景**：VMA 管理一直是内核性能热点。6.1 之前 `mm_struct` 里同时维护"一棵按地址排序的红黑树（字段 `mm_rb`，高把握）"加"一条按地址串起所有 VMA 的链表（VMA 侧字段历史上是 `vm_next`/`vm_prev`；`mm_struct` 侧的遍历入口字段名待核，常被口语称作 mmap 链表）"。两个结构都得在每次 `mmap`/`munmap` 时更新，插入/删除做两遍、cache 局部性差。

**Maple Tree** 是为这类区间管理设计的 RCU-safe B-tree 变体（Range-based，作者 Liam Howlett / Matthew Wilcox）：
- 叶节点直接存连续地址区间，对 VMA（天然是 `[start, end)` 区间对象）友好，范围查找和"找空洞"是一等操作。
- 节点做得**远大于单条 cache line**（量级约 256 字节，跨多条 cache line），一个内部节点里塞下十几个 key/区间；与红黑树"每个节点一次随机指针跳转"相比，B-tree 的高扇出显著压低树高和随机访存次数（cache 收益来自"少跳几层、每层批量比较"，不是"一个节点正好一条 cache line"）。
- 用单一结构同时替换了红黑树 + 链表，既能 O(log n) 点查、又能有序遍历和区间查询。
- Linux 6.1 合入主线，`vm_area_struct` 的链表字段逐步清理中。

对照表：

| 结构 | 适用场景 | 查找复杂度 | 范围查询 | Cache 友好性 |
|------|---------|------------|----------|-------------|
| `list_head` | 小集合、顺序遍历 | O(n) | 差 | 好（连续遍历） |
| `rb_tree` | 有序集合、点查询 | O(log n) | 一般 | 一般 |
| `xarray` | 整数 key→指针 | O(1) 均摊 | 不支持 | 好 |
| `maple tree` | 地址范围管理 | O(log n) | 好 | 优（B-tree 节点大） |

## 5. `container_of` 深挖与 `offsetof`

（已在 2.2 节详述，此处补充两个延伸点）

### 5.1 `offsetof` 不是运行时计算

```c
#define offsetof(TYPE, MEMBER) ((size_t)&((TYPE *)0)->MEMBER)
```

将 `0` 强制转型为指向 `TYPE` 的指针，取 `MEMBER` 的地址（用 `&` 取址，并不真的解引用，所以运行期不会真去访问 0 地址），编译器把整个表达式折叠为常量，零运行时开销。注意：`&((TYPE *)0)->MEMBER` 在 C 标准里形式上是"对空指针表达式取成员",踩在 UB 边界——成员若是位域或重载了 `operator&`（C++）就更微妙。所以现代内核实际用编译器内建的 `__builtin_offsetof`（GCC/Clang 都有），由编译器直接给出偏移，绕开这段 UB 文本。这与 strict aliasing 无关，纯粹是为了不写出"解引用空指针"的源码形态。

### 5.2 工程推论

1. `container_of` 是内核"类型多态"的实现机制——无需虚函数表，通过嵌入已知偏移的 `list_head`/`rb_node` 实现不同类型挂同一数据结构。
2. 同一套链表/树代码处理所有内核对象，代码复用达到极致，但调试时需要手动计算偏移（或用 `pahole` 工具查看结构布局）。

## 6. 错误处理：`ERR_PTR` / `PTR_ERR` / `IS_ERR`

### 6.1 设计动机

C 语言返回指针的函数，错误路径的传统做法都有缺陷（先澄清：内核里**没有用户态那个全局 `errno`**——libc 的 `errno` 是 syscall 返回用户态后由 C 库设置的；内核内部一律靠"显式返回 `-Exxx` 负错误码"传播错误）：
- 只返回 `NULL`：信息丢失，`NULL` 仅能表达"失败/找不到"，无法区分 `-ENOMEM` 与 `-EINVAL`。
- 返回 `NULL` + 额外 out-param（`int *err`）：调用方负担重，多一个参数、忘记检查就是 bug。

内核方案：复用指针值本身的一小段地址。`ERR_PTR(-E)` 就是把小负数 `-E` 当成指针返回（`(void *)(long)-E`），在 64 位上符号扩展后落在地址空间最顶端那 4 KB（`[-4095, -1]` ↔ `0xFFFF_FFFF_FFFF_F001 ~ 0xFFFF_FFFF_FFFF_FFFF`）。这段最高地址内核刻意不映射，任何合法对象指针都不会落进去，于是"是不是错误码"用一次范围判断就能定。注意：这里没有"塞进指针高位/低位"的位运算——错误码就是指针的全部数值。

### 6.2 实现原理

```
内核虚拟地址空间（x86_64）：
  用户态：0x0000_0000_0000_0000 ~ 0x0000_7FFF_FFFF_FFFF
  内核态：0xFFFF_8000_0000_0000 ~ 0xFFFF_FFFF_FFFF_FFFF

内核合法指针：地址高位全 1，数值上是极大的负数（按有符号解释）
错误码范围：-1 ~ -4095（`MAX_ERRNO == 4095`，确认；定义在 `include/linux/err.h`）
  对应地址：0xFFFF_FFFF_FFFF_FFFF ~ 0xFFFF_FFFF_FFFF_F001
```

`IS_ERR(ptr)` 检查指针是否落在 `[-MAX_ERRNO, -1]` 对应的地址范围内。因为合法内核指针不会落在这个极高地址附近（这段地址没有映射），所以区分是安全的。

```c
// 函数返回 ERR_PTR
struct file *open_file(const char *name) {
    if (!name) return ERR_PTR(-EINVAL);
    // ...
    if (!f) return ERR_PTR(-ENOMEM);
    return f;
}

// 调用方处理
struct file *f = open_file(path);
if (IS_ERR(f)) {
    int err = PTR_ERR(f);  // 还原错误码
    pr_err("failed: %d\n", err);
    return err;
}
```

💡 **坑点**：`IS_ERR(NULL)` 返回 false，因为 `NULL` (0x0) 不在错误范围内。内核有些 API 返回 `NULL` 表示"正常未找到"，返回 `ERR_PTR(err)` 表示"出错了"——语义不同，调用方要同时检查两者：`IS_ERR_OR_NULL(ptr)`（确认存在，同样在 `include/linux/err.h`）。但注意 §章末五件套 Q4 的判断——一个 API 同时返回三态（合法/NULL/ERR_PTR）逼调用方用 `IS_ERR_OR_NULL`，往往是接口语义没设计干净。

**工程推论**：
1. 看内核函数签名，返回 `struct foo *` 不代表只会返回合法指针或 NULL，可能还会返回 `ERR_PTR`——凡是不检查 `IS_ERR` 就 dereference 的代码都是潜在 bug。
2. 这个模式是内核 API 合约的一部分，出现在 VFS、网络、设备驱动大量接口里，不懂它读源码会一脸懵。

## 7. 引用计数：`kref`

### 7.1 为什么需要 `kref`

内核对象（`struct file`、`struct inode`、`struct net_device`……）被多方持有，需要最后一个持有者负责释放。手写引用计数容易犯三类错误：
- 计数归零时忘记释放
- 并发 decrement 导致双重释放
- 对象已释放后仍有人 increment（use-after-free）

`kref` 提供最小化的引用计数原语：

```c
struct kref {
    refcount_t refcount;
};

void kref_init(struct kref *kref);
void kref_get(struct kref *kref);
// return 1 if object released, 0 otherwise
int kref_put(struct kref *kref, void (*release)(struct kref *kref));
```

### 7.2 `refcount_t` vs `atomic_t`

近年内核（4.11+ 左右，待核：确切版本）把 `kref` 底层从 `atomic_t` 改为 `refcount_t`，关键区别：

| | `atomic_t` | `refcount_t` |
|--|-----------|-------------|
| 溢出行为 | 绕回 | 饱和（stuck at max） |
| 下溢到负数 | 允许 | 会触发 WARN |
| 安全性 | 可被精心构造的溢出攻击 | 对 UAF/double-free 有保护 |

🎯 **面试加分点**：这个改动是专门针对 CVE 级别漏洞的防御——攻击者通过多线程竞争使引用计数整数溢出回到正数，从而阻止对象被释放并利用。`refcount_t` 的饱和行为破坏了这类攻击路径。

### 7.3 工程推论

1. `kref_put` 接受一个 `release` 函数指针，而非内联调用 `kfree`——因为不同对象的释放逻辑不同（可能要同时释放子成员、移出链表等）。
2. `kref_get` 在对象已被释放后调用是 UB，调用方必须保证调用时对象仍然存活（通常靠锁或其他同步手段）。
3. **真实工程里更常见的难题是"无锁查找 + 安全提引用"**：你在 RCU 临界区里从一条链表/哈希表查到一个对象指针，但它的 refcount 此刻可能正好被别的 CPU 减到 0、`release` 在路上。直接 `kref_get` 会复活一个濒死对象。标准解法是 `kref_get_unless_zero()`（底层 `refcount_inc_not_zero`）：原子地"仅当当前值非 0 才 +1",返回是否成功；失败说明对象已逻辑死亡，当作没查到。这是 `dget_unless_zero`、`igrab`、众多 RCU-lookup 路径的通用骨架，staff 面试高频。

## 8. Per-CPU 变量

### 8.1 设计动机

多核系统中，全局变量访问需要同步（spinlock 或原子操作）。但对于"每个 CPU 各自统计自己的数据"这类场景，同步纯属浪费——只要保证当前 CPU 访问自己的副本时不被抢占换到另一个 CPU 上即可。

```
CPU 0            CPU 1            CPU 2
  │                │                │
per_cpu_var[0]  per_cpu_var[1]  per_cpu_var[2]
  │                │                │
各自更新，无需跨 CPU 同步
```

### 8.2 声明与访问

```c
// 静态声明（编译期）
DEFINE_PER_CPU(int, my_counter);

// 动态分配（运行期）
int __percpu *dyn_counter = alloc_percpu(int);

// 访问（需关抢占）
int val = get_cpu_var(my_counter);  // 关抢占 + 返回当前 CPU 副本
val++;
put_cpu_var(my_counter);            // 写回 + 开抢占

// 或者手动关抢占（更灵活）
preempt_disable();
int *p = this_cpu_ptr(&my_counter);
(*p)++;
preempt_enable();
```

### 8.3 实现原理：段寄存器魔法（x86_64）

x86_64 使用 `gs` 段寄存器存放当前 CPU 的 per-CPU 基地址（`gs_base`）。

```
per-CPU 变量的内存布局（启动时初始化）：

物理内存：
  ┌──────────────────────────────────────────┐
  │ CPU 0 per-CPU segment                    │ ← gs_base on CPU 0
  │  my_counter offset 0x1234                │
  ├──────────────────────────────────────────┤
  │ CPU 1 per-CPU segment                    │ ← gs_base on CPU 1
  │  my_counter offset 0x1234（同偏移）       │
  └──────────────────────────────────────────┘

访问时：gs:0x1234 → 自动取当前 CPU 的副本，无需任何运算
```

汇编级访问 `%gs:offset(var)` 一条指令完成，比任何锁都快。

🎯 **面试加分点（swapgs）**：x86_64 上 `%gs` 的基址在用户态和内核态指向不同的东西——用户态 `gs_base` 归用户程序（虽然 glibc 的 TLS 实际走 `%fs`），内核态 `gs_base` 才指向本 CPU 的 per-CPU 区。CPU 进出内核（syscall/中断入口）执行 `swapgs` 指令，在两个基址之间翻转。这也是为什么 per-CPU 访问必须在内核态、且 `swapgs` 的配对正确性是内核入口汇编里最敏感的部分之一（Meltdown 之后的 KPTI、以及若干 `swapgs` 错配漏洞都围绕这里）。

### 8.4 工程推论

1. Per-CPU 是内核性能优化的首选工具：网络 `softnet_data`、调度器 `runqueue`、内存分配器的 per-CPU cache（`kmem_cache` 的 `cpu_cache`）都用这个模式。
2. 用 `get_cpu_var` 而非直接 `this_cpu_ptr` 的原因：前者自动关抢占防止中途被调度到别的 CPU，后者需要调用方保证。忘记关抢占是经典 bug，排查时现象是随机数据损坏（某次统计跑到别的 CPU 上了）。
3. Per-CPU 变量**不是** thread-local：如果进程在访问期间被抢占并迁移到另一个 CPU，会访问到不同副本。这就是为什么必须关抢占。

## 9. 内核内存为何不能缺页

### 9.1 内核地址空间的直接映射

x86_64 内核的虚拟地址空间（简化，具体量级依配置）：

```
虚拟地址空间（64 位）：
  0x0000_0000_0000_0000  ┌─────────────────────┐
                         │   用户空间（128 TB）  │  缺页 = 正常（CoW/懒分配）
  0x0000_7FFF_FFFF_FFFF  └─────────────────────┘
  （巨大的规范空洞）
  0xFFFF_8000_0000_0000  ┌─────────────────────┐
                         │  直接映射区（physmap） │ ← kmalloc/vmalloc 在此
                         │  内核代码/数据段      │
                         │  vmalloc 区           │
                         │  vmemmap（struct page）│
  0xFFFF_FFFF_FFFF_FFFF  └─────────────────────┘
```

内核的"直接映射区"（direct map / physmap）在内核启动时就把所有物理内存 1:1 映射上去，页表项提前建好，不会触发缺页中断。

### 9.2 为什么内核代码不能接受缺页

1. **递归死锁**：缺页处理本身是中断服务例程的一部分，如果在持有某个子系统的锁时触发缺页，而缺页处理又需要获取同一把锁，必死锁。
2. **中断上下文无进程**：中断上下文没有对应的进程 mm，无法进行 VMA 查找和页面换入。
3. **实时性保证**：内核代码路径的延迟必须可预测，缺页处理可能触发 I/O（swap in），延迟不可接受。

### 9.3 `vmalloc` 的特殊情况

`vmalloc` 分配的内存历史上是个例外。它在 vmalloc 区建立虚拟地址映射（物理页分配好了，但多个进程页表对内核高半区的同步是懒的），早期 x86 在首次访问该地址、当前进程页表里还没补上对应顶层项时，会走一条内核态缺页处理 `vmalloc_fault()`——把 init_mm 的内核 PGD/P4D 项拷进当前进程页表。这是"确定性、无 swap I/O、不会死锁"的内核态缺页，与"内核不该缺页"的总原则并不矛盾。

> **演进注 / 时效（重要）**：x86-64 在约 5.5（2019 末–2020 初）**删除了 `vmalloc_fault`**——改为在创建 vmalloc 映射时就把内核顶层页表项**预先同步**到所有进程（`arch_sync_kernel_mappings` 一类机制），从根上消除这条缺页路径。动机之一正是这条 fault 路径历史上反复出问题（能在不该缺页的上下文，如 tracing/NMI 里被触发，极难调试）。所以"`vmalloc` 靠缺页补页表"在现代 x86-64 上已是过时描述；32 位等架构可能仍保留类似逻辑。`vmalloc_fault` 这个具体符号：早期内核确实存在，现代 x86-64 已移除——属于"版本相关行为"，引用时务必带版本。

💡 **坑点**：`vmalloc` 地址跨页不一定连续，不能用于需要物理连续的 DMA 操作；`kmalloc`（来自 slab allocator）的物理地址通常连续（小块），适合 DMA。

### 9.4 工程推论

1. 向 `kmalloc`/`kzalloc` 传 `GFP_ATOMIC` 时，内存分配器绝不能睡眠（不能触发 swap），分配可能失败——始终检查返回值。
2. `GFP_KERNEL` 允许调度（在内存压力下睡眠等待内存释放），只能在进程上下文使用。
3. 内核驱动里用 `GFP_ATOMIC` 而非 `GFP_KERNEL` 是常见 cargo-cult：很多场景其实可以用 `GFP_KERNEL`，用 `GFP_ATOMIC` 只是"保险"写法，实际上提高了分配失败率。

## 10. 内核编程杂项范式速查

### 10.1 `BUILD_BUG_ON`：编译期断言

```c
BUILD_BUG_ON(sizeof(struct my_struct) != 64);
```

如果条件为真，编译报错——确保数据结构大小、对齐与预期一致，不需要运行时测试。常见于 ABI 稳定性保证。

### 10.2 `likely`/`unlikely`：分支预测提示

```c
if (unlikely(ptr == NULL))
    return -EINVAL;
```

告诉 GCC/Clang 生成时把"likely"分支放在直通路径上，减少分支预测失败惩罚。内核热路径（中断处理、网络收包、调度器）大量使用。

💡 **坑点**：滥用 `likely`/`unlikely` 反而降低性能——现代 CPU 的分支预测器很强，只在真正可预测（>95% 概率）的分支上用才有收益。

### 10.3 `READ_ONCE`/`WRITE_ONCE`：无锁访问的编译器屏障

```c
// 防止编译器把多次读合并或重排
int val = READ_ONCE(shared_var);
WRITE_ONCE(shared_var, new_val);
```

不提供硬件内存屏障语义（用 `smp_rmb`/`smp_wmb`/`smp_mb` 配套），但防止编译器优化破坏无锁算法的"单次读写"语义。

### 10.4 内存屏障速查

| 原语 | 方向 | 强度 |
|------|------|------|
| `smp_rmb()` | 读屏障 | 读不能向上越过 |
| `smp_wmb()` | 写屏障 | 写不能向下越过 |
| `smp_mb()` | 全屏障 | 读写都不能越过 |
| `barrier()` | 编译器屏障 | 仅防编译器重排 |

（详见第 13 章 RCU 与内存模型）

## 12. 落地：生产排障与工具

### 12.1 `pahole`：查看内核结构布局

```bash
# 查看 task_struct 的字段偏移和 padding
pahole -C task_struct vmlinux

# 找 cache line 跨越（> 64 字节对齐的字段）
pahole --show_cacheline vmlinux | grep task_struct
```

`pahole` 是调优 cache 局部性的必备工具，也是验证 `container_of` 偏移计算是否正确的方法。

### 12.2 `/proc/slabinfo`：查看 slab allocator 使用情况

```bash
cat /proc/slabinfo | sort -k3 -rn | head -20  # 按对象数量排序
# 输出：name / active / total / obj_size / obj_per_slab / ...
```

内存泄漏排查：某个 slab（如 `dentry`、`inode_cache`）数量持续增长而不回落，通常意味着对象没被正确释放（kref 没 put 到零）。

### 12.3 `bpftrace`：追踪引用计数

```bash
# ⚠ 这条多半挂不上：kref_put 是 include/linux/kref.h 里的 static inline，
# 被内联进各调用点，没有独立的内核符号，kprobe 无从附着（示意）。
bpftrace -e 'kprobe:kref_put { @[kstack] = count(); }'   # 通常报 "No probes matched"

# 可行替代：探针 kref_put 内部真正落地的 refcount 函数（非 inline 的有符号实现），
# 或针对你关心的具体对象的 release 回调 / 具体子系统的 put 函数（如 dput）。
# 先用下面这行确认哪些符号真的可探：
grep -E 'refcount_dec_and_test|^dput$' /sys/kernel/tracing/available_filter_functions
```

### 12.4 KASAN：内核地址消毒剂

编译内核时开启 `CONFIG_KASAN`，可以检测：
- `list_del` 后的 use-after-free（访问 `LIST_POISON`）
- slab 越界写
- 栈越界

生产内核一般不开（generic KASAN 量级：CPU ~1.5–3x 慢、内存放大数倍，依工作负载），staging/fuzzing（syzkaller）环境标配。补：ARM64 上还有 SW_TAGS（依赖 top-byte-ignore）和 HW_TAGS（依赖硬件 MTE）两种 KASAN 模式，开销远低于 generic，部分场景甚至接近可生产采样——属较新内核能力。

### 高频面试题（Staff 级）

**Q1：解释 `container_of` 的实现原理，为什么没有运行时开销？它有什么潜在的类型安全问题？**

> 答题框架：`offsetof` 编译期展开 → 指针减偏移 → 零运行时开销。类型安全：GCC `typeof` 检查 `ptr` 类型与 member 类型是否匹配，不匹配给 warning（不是 error）。弱点：如果强制转型绕过 `typeof` 检查，编译不报错但运行时越界访问。
> 
> 陷阱答案："有 typeof 检查所以完全类型安全" — 错，warning 不是 error，也可以被绕过。

**Q2：`list_for_each_entry` 和 `list_for_each_entry_safe` 的区别，什么时候必须用 safe 版本？**

> 答题框架：safe 版本额外保存 next 指针（临时变量 `n`），删除 `pos` 后仍能继续遍历。必须用 safe：遍历过程中要删除当前节点；不能用（也不必要）：只读遍历，或只删除非当前节点。
> 
> 陷阱答案："safe 版本更安全所以一律用" — 错，safe 版本也不支持从不同线程并发删除，safe 只是相对于"删当前节点"安全，不是并发安全。

**Q3：per-CPU 变量为什么需要关抢占而非关中断？二者有什么本质区别？**

> 答题框架：per-CPU 变量的第一类并发威胁是"被抢占→迁移到另一 CPU"，于是去改了别人的副本。关抢占（`preempt_disable`）堵住进程切换，让"读基址→改副本"这段不被打断地落在同一 CPU 上，对**只与其他进程上下文竞争**的 per-CPU 计数器足够。
>
> 但还有第二类威胁：**同一 CPU 上的中断/软中断处理也访问同一个 per-CPU 变量**。即便关了抢占，一个中断照样能打断你这段 RMW（read-modify-write），中断处理器更新了同一副本，回来你再写回就丢更新。这种"进程上下文与本 CPU 中断共享 per-CPU 数据"的场景，必须升级到 `local_irq_disable`（关中断）或 `local_bh_disable`（关软中断），代价是中断/软中断延迟变大。`this_cpu_inc` 这类**单指令 RMW**则因不可被中断打断而无需额外关中断。
>
> 一句话判据：只和"别的 CPU/被迁移的自己"竞争 → 关抢占；还和"本 CPU 的中断/软中断"竞争 → 关中断/软中断；单条原子 per-CPU 指令 → 都不用。

**Q4：`ERR_PTR(-ENOMEM)` 和 `NULL` 返回有什么语义区别？什么情况下 `IS_ERR_OR_NULL` 是坏的设计？**

> 答题框架：`NULL` = "找不到但没出错"（如 `find_task_by_pid` 找不到进程），`ERR_PTR` = "出错了+带错误码"。`IS_ERR_OR_NULL` 把两种情况混在一起处理，说明该 API 语义不清晰——好的 API 应该只返回其中一种 null 语义，不该让调用方猜。用 `IS_ERR_OR_NULL` 通常意味着需要重构 API。

**Q5：`kref` 内部为什么用 `refcount_t` 而非 `atomic_t`？具体防御了什么攻击？**

> 答题框架：`refcount_t` 溢出饱和（不绕回），下溢触发 WARN。防御整数溢出型 UAF 攻击：攻击者使计数从 1 通过多线程竞争 increment 绕回 0，触发 release，再通过 increment 绕回正数阻止释放……`refcount_t` 的饱和打断这个循环。具体 CVE 级别的案例可举 `net/sched` 或 USB 子系统的历史漏洞（具体 CVE 待核）。

**Q6：`vmalloc` 为什么允许缺页而 `kmalloc` 不能？**

> 答题框架：先纠正一个常见口误——`vmalloc` 的物理页并非"访问时才懒分配"，分配时就有了；真正懒的是"内核高半区页表项向各进程页表的同步"。历史上 x86 用 `vmalloc_fault()` 在首次访问时把顶层项拷进当前页表，这是**确定性、无 swap I/O、有界延迟**的缺页，故不违反"内核不该缺页"。`kmalloc` 来自 slab，落在启动期就建好的 direct map 里，根本不缺页。
>
> 加分（时效）：x86-64 约 5.5 起**删除了 `vmalloc_fault`**，改为建映射时预同步内核页表，彻底消灭这条 fault 路径——因为它历史上会在 NMI/tracing 等不该缺页的上下文被触发、极难调试。能讲出这个演进，面试官会认为你读的是近年源码而非二手博客。
>
> 陷阱答案："vmalloc 物理页是访问时才分配的" — 错，混淆了"物理页懒分配"（用户态 CoW 才是）和"页表项懒同步"。

**Q7：画出 `maple tree` 取代 `mm_rb + mmap_list` 的架构演进图，解释 cache 局部性改善的原因。**

> 答题框架：老设计维护两个结构（一个红黑树用于 O(log n) 查找，一个链表用于 O(1) 顺序遍历），每次修改两次写，两次 cache miss。Maple tree 是 B-tree 变体，节点匹配 cache line 大小，单节点内存储多个范围，遍历时 cache miss 次数减少 log factor。6.1+ 的改动直接减少了 `mmap`/`munmap` 的 CPU 时钟数。

### 设计题

**题目**：设计一个内核内存对象池（对象回收池），要求：
- 对象固定大小，分配/释放 O(1)
- 多核并发安全，热路径零锁争用
- 内存回收可推迟（批量还给系统）
- 不允许使用 slab allocator 的现成 API

请给出数据结构设计（per-CPU 空闲链表 + 全局 overflow list 的两级设计），说明每一级的锁策略，以及 fallback 路径。

> 参考答案方向：两级 = per-CPU 空闲 list（无锁，关抢占访问） + 全局 spinlock 保护的 overflow pool。per-CPU 超出 N 个时批量还给全局；全局为空时调 kmalloc。这正是 `kmem_cache` cpu_cache 的简化模型，理解这个就理解了 slab allocator 的核心权衡。

### 系统题

**场景**：生产系统 `dentry` cache（`/proc/slabinfo` 中 `dentry` 一行）在某次大规模 `find / -name foo` 扫描后持续占用 8 GB，`drop_caches` 后恢复正常，但每次类似扫描都复现。你如何排查是正常的 cache 行为还是引用计数泄漏？

**答题要点**：
1. 区分 `dentry` 的两种状态：`in-use`（有进程持有）和 `unused`（可被回收）——`/proc/slabinfo` 的 `active` vs `total` 字段
2. `drop_caches` 能清掉的是 unused，`in-use` 的 dentry 不能清——如果 drop_caches 能恢复就是正常 cache 膨胀
3. 验证引用计数泄漏：`slabtop` 持续监控；要统计引用收支，dentry 的"取引用/放引用"配对是 `dget`/`dput`（不是 `dentry_open`——那是 VFS 里用 dentry 打开文件的函数，语义不同）。现代 dentry 的引用计数实际由 `lockref`（spinlock + count 打包进一个 64 位字、用 cmpxchg 走无锁快路径）承载，所以最稳的观测点是 `dput`（释放侧，`fs/dcache.c`，高把握存在）；`dget` 是 `static inline`、未必有独立 kprobe 符号，必要时改插 `lockref_get` 一类底层函数（具体可探针符号以 `/sys/kernel/tracing/available_filter_functions` 现场确认）。比较取/放调用次数即可判断是否只增不减。
4. 量级估算：现代 x86_64 上 `sizeof(struct dentry)` 量级在 ~192–256 字节（依内核版本/配置，且 slab 实际分配会向对齐/cache line 取整——具体值用 `pahole -C dentry vmlinux` 或 `/proc/slabinfo` 的 `objsize` 列现场核）。取 ~200 字节估算：8 GB / 200 B ≈ 4000 万 dentry，对应约 4000 万次目录/文件访问，大规模 `find /` 的确能产生这个量级——所以"涨到 8 GB"本身不证明泄漏，要看 drop_caches 后能否回落（见第 2 点）

### 代码题

**实现一个内核模块，演示 `list_head`、`container_of`、`kref` 的联合使用。**

```c
// 文件：my_object_pool.c
// 目的：管理一组引用计数对象，按 id 链表查找，支持 get/put 生命周期

#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/init.h>
#include <linux/list.h>
#include <linux/kref.h>
#include <linux/slab.h>
#include <linux/spinlock.h>

MODULE_LICENSE("GPL");
MODULE_AUTHOR("Demo");

struct my_obj {
    int id;
    char name[32];
    struct kref ref;
    struct list_head list;  // 挂在全局链表
};

static LIST_HEAD(obj_list);
static DEFINE_SPINLOCK(obj_lock);

static void obj_release(struct kref *kref)
{
    struct my_obj *obj = container_of(kref, struct my_obj, ref);
    pr_info("releasing object id=%d\n", obj->id);
    // 注意：调用此函数时已是最后一个引用，对象不在链表里（已 list_del）
    kfree(obj);
}

static struct my_obj *obj_create(int id, const char *name)
{
    struct my_obj *obj = kzalloc(sizeof(*obj), GFP_KERNEL);
    if (!obj)
        return ERR_PTR(-ENOMEM);
    obj->id = id;
    strscpy(obj->name, name, sizeof(obj->name));
    kref_init(&obj->ref);   // refcount = 1
    spin_lock(&obj_lock);
    list_add_tail(&obj->list, &obj_list);
    spin_unlock(&obj_lock);
    return obj;
}

// TODO 1: 实现 obj_get(id) — 按 id 查找，若找到则 kref_get，返回指针；未找到返回 ERR_PTR(-ENOENT)
// TODO 2: 实现 obj_put(obj) — 调用 kref_put，在 obj_release 里才真正 kfree
// TODO 3: 模块 exit 时遍历链表，对每个对象调用 obj_put，验证最终全部释放

static int __init pool_init(void)
{
    struct my_obj *a = obj_create(1, "alpha");
    struct my_obj *b = obj_create(2, "beta");
    if (IS_ERR(a) || IS_ERR(b)) {
        pr_err("create failed\n");
        return PTR_ERR(IS_ERR(a) ? a : b);
    }
    pr_info("created %s and %s\n", a->name, b->name);
    // 演示：再 get 一次 a
    kref_get(&a->ref);  // refcount = 2
    // TODO: put 两次 a，验证第二次 put 触发 release
    return 0;
}

static void __exit pool_exit(void)
{
    // TODO 3
    pr_info("pool module exit\n");
}

module_init(pool_init);
module_exit(pool_exit);
```

**验证方法**：
```bash
# 编译（需要内核头文件）
make -C /lib/modules/$(uname -r)/build M=$(pwd) modules

# 加载并观察 dmesg
sudo insmod my_object_pool.ko
sudo dmesg | tail -20

# 卸载
sudo rmmod my_object_pool
sudo dmesg | tail -20  # 验证 release 被正确调用
```

**暗坑提示**：
1. `strscpy` 而非 `strcpy`——内核 6.x+ `strncpy` 已不推荐（存在未终止字符串风险），`strscpy` 总是 NUL 终止（待核：strscpy 的具体行为以内核版本为准）。
2. `kzalloc` 会将内存清零，`kmalloc` 不会——忘记初始化 `list.next/prev` 会导致链表操作访问随机地址。`kref_init` 必须在挂链表之前调用。
3. `obj_release` 回调时不能再持有 `obj_lock`——`kref_put` 可能在 `obj_lock` 持有期间被调用，`release` 里若尝试获取同一把锁则死锁。正确模式：先 `list_del`（持锁），再 `kref_put`（不持锁），`release` 里只做 `kfree`。

## 章末心法

> **内核 C 的精髓是"侵入式"哲学**：不把数据放进容器，而是把容器的钩子嵌进数据——`list_head`、`rb_node`、`kref` 都是这个思想的体现。理解了这一点，你就理解了为什么内核代码总是 `container_of` 满天飞，也理解了为什么同一个对象可以同时挂在十几个不同链表上而零额外开销。
