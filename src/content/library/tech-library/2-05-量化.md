## 5.1 设计考古：为何量化、谁先做

### 5.1.1 瓶颈来源

LLM decode 阶段的计算模式是典型 **memory-bandwidth bound**：对每个 token，每个 Linear 层都要把整个权重矩阵 load 进来乘一次激活向量（batch=1 时 FLOP/byte << GPU 的 arithmetic intensity 阈值）。一张 A100 80GB 的 HBM 带宽约 2 TB/s；一个 70B 参数 FP16 模型每步 decode 要 load ~140 GB 权重——理论下界就要 70ms，严重制约服务吞吐。

量化的第一个动机是**减小权重 size → 减少 memory-bandwidth 消耗**。INT4 对 FP16 理论 4× 压缩，decode latency 可以随之接近 4× 下降。

量化的第二个动机是**让大模型塞进单卡 VRAM**。175B FP16 模型需要 350 GB，4 张 A100 才能放下；INT4 只需约 87 GB，单卡 H100 就可以跑。

### 5.1.2 历史脉络

| 时间 | 事件 |
|------|------|
| 2022-08 | Tim Dettmers 等提出 **LLM.int8()** (arXiv:2208.07339)：发现 emergent outlier feature → mixed-precision INT8 decomposition，首次可无损量化 175B 规模模型 |
| 2022-10 | Frantar 等提出 **GPTQ** (arXiv:2210.17323)：基于二阶 Hessian 信息做逐列校正，4 GPU hours 量化 175B，INT4 几乎无损 |
| 2023-06 | Lin 等提出 **AWQ** (arXiv:2306.00978)：发现 1% salient weights，用 activation 统计导出 per-channel scale，MLSys 2024 Best Paper |
| 2022-09 | NVIDIA 等提出 **FP8** (arXiv:2209.05433)：E4M3/E5M2 两种编码，Hopper 架构原生支持 FP8 GEMM |
| 2023-03 | Georgi Gerganov 在 llama.cpp 引入 **GGUF/GGML quants**：Q4_0/Q4_K_M 等格式，CPU/Metal 友好，纯 C 实现 |
| 2023-Q3 | vLLM 加入 GPTQ / AWQ / bitsandbytes 支持，量化进入生产推理引擎 |
| 2024 | vLLM 加入 FP8 KV Cache、FP8 Linear，Marlin kernel 正式落地，量化技术走向稳定 |

## 5.2 量化基础：数学语义

### 5.2.1 均匀量化的两个变体

**对称量化（Symmetric）**  
```
scale = max(|W|) / (2^(b-1) - 1)
W_q   = round(W / scale)        ∈ [-127, 127] (INT8)
W_deq = W_q * scale
```
存储：仅需一个 scale。零点始终对应 0，硬件友好。

**非对称量化（Asymmetric）**  
```
scale  = (max(W) - min(W)) / (2^b - 1)
zero   = round(-min(W) / scale)   ∈ [0, 255] (UINT8)
W_q    = clamp(round(W / scale) + zero, 0, 255)
W_deq  = (W_q - zero) * scale
```
存储：scale + zero_point，精度更高但存储和计算稍贵。AWQ 默认用此形式。

### 5.2.2 粒度：per-tensor / per-channel / per-group

| 粒度 | scale 数量 | 精度 | 开销 |
|------|-----------|------|------|
| per-tensor | 1 | 最低 | 最小 |
| per-channel (per-output) | out_features 个 | 较好 | 小 |
| per-group (group_size=128) | out_features × in_features/128 | 好 | 中 |
| per-token (动态激活) | 每 token 1 个 | 最高 | 每步计算 |

推理引擎中权重通常 per-group 量化（GPTQ/AWQ 默认 group_size=128），激活可做动态 per-token。

## 5.3 GPTQ：二阶误差最小化

### 5.3.1 核心思想

GPTQ 基于 OBQ（Optimal Brain Quantization），目标函数：

```
min_W_q  ||W·X - W_q·X||_F^2
```

对每一列 weight 独立量化时，用 Hessian H = XX^T 的逆来补偿误差。核心技巧：

1. **Lazy Batch Updates**：一次处理 128 列，减少 Hessian 更新次数
2. **Cholesky Decomposition**：Hessian 求逆改为一次 Cholesky，数值稳定
3. **Column-wise Error Propagation**：量化第 i 列的误差 δ = (w_i - q_i)，通过 H^{-1} 传播到后续列

伪代码（引自论文 Algorithm 1）：

```
H  = 2·X·X^T                      # Hessian
H^{-1} = cholesky(H)^{-T}         # 一次求逆
for i = 0..n_cols-1:
    q_i = quantize(w_i)            # 量化当前列
    err = (w_i - q_i) / H^{-1}_{ii}
    W_{i+1:} -= err · H^{-1}_{i, i+1:}   # 传播误差到右侧列
```

### 5.3.2 实际实现路径（vLLM）

vLLM 中 GPTQ 运行时不再跑校正（离线完成），只做推理侧的 **dequantize + GEMM**。

**【真实源码 vllm@vllm/model_executor/layers/quantization/awq.py】**

```python
# vllm/model_executor/layers/quantization/awq.py  (同为 W4A16 格式参考)
def apply(
    self,
    layer: torch.nn.Module,
    x: torch.Tensor,
    bias: torch.Tensor | None = None,
) -> torch.Tensor:
    qweight = layer.qweight    # INT32 packed 4-bit weights
    scales  = layer.scales     # FP16 scales, shape [K/group, N]
    qzeros  = layer.qzeros     # INT32 packed zero-points
    pack_factor = self.quant_config.pack_factor  # = 32 // 4 = 8

    out_shape   = x.shape[:-1] + (qweight.shape[-1] * pack_factor,)
    reshaped_x  = x.reshape(-1, x.shape[-1])

    # 启发式：token >= 256 时先 dequant 再 matmul（prefill），
    # 否则用 fused awq_gemm kernel（decode 小 batch）
    FP16_MATMUL_HEURISTIC_CONDITION = x.shape[:-1].numel() >= 256

    if FP16_MATMUL_HEURISTIC_CONDITION or envs.VLLM_BATCH_INVARIANT:
        out = ops.awq_dequantize(qweight, scales, qzeros, 0, 0, 0)  # → FP16
        out = torch.matmul(reshaped_x, out)
    else:
        # fused: 内部 kernel 直接做 dequant+GEMM，避免中间 buffer
        out = ops.awq_gemm(reshaped_x, qweight, scales, qzeros, pack_factor)

    if bias is not None:
        out.add_(bias)
    return out.reshape(out_shape)
```

**注解：**
- `qweight` 的 dtype 是 INT32，每个 INT32 打包了 8 个 4-bit weight（pack_factor=8）
- `FP16_MATMUL_HEURISTIC_CONDITION`：prefill 阶段 batch 大，dequant 后做标准 cublas GEMM 更快；decode 阶段 batch 小，fused kernel 更优
- 这一分支逻辑是推理引擎量化的普遍模式

## 5.4 AWQ：Activation-Aware Weight Quantization

### 5.4.1 核心洞察

AWQ 论文（arXiv:2306.00978，MLSys 2024 Best Paper）发现：

> **不是所有权重都一样重要**。只保护 1% 的 salient weights 就能大幅减少量化误差。判断 saliency 的依据是**激活值分布**而非权重本身。

数学：对线性层 y = Wx，引入 per-channel scale s：

```
y = (W · diag(s)^{-1}) · (diag(s) · x)
  = W' · x'
```

W' = W · diag(s)^{-1} 是被 scale 过的权重，量化时误差变小；  
x' = diag(s) · x 是 scale 过的激活，实际推理时对激活做反向 scale 补偿（或等价地把 scale 折叠进 LayerNorm/前一层）。

关键：scale s 从**离线收集的激活统计**（per-channel 的 X 的绝对均值 x_max）推导：

```python
# 真实来源：awq/quantize/auto_scale.py  【真实源码 llm-awq@awq/quantize/auto_scale.py】
# 枚举 20 个候选 ratio in [0, 1]，找 MSE 最小的
scales = x_max.pow(ratio)              # scale ∝ |activation|^ratio
scales = scales / (scales.max() * scales.min()).sqrt()  # 归一化保持幅度

# 对每个 ratio，临时 apply scale 到权重，量化，反量化，算 MSE
loss = (org_out - out).float().pow(2).mean()
best_scales = scales[argmin(loss)]
```

### 5.4.2 AWQ 与 GPTQ 关键差别

| 维度 | GPTQ | AWQ |
|------|------|-----|
| 校正依据 | 二阶 Hessian（权重侧） | 激活统计（activation 侧） |
| 计算量 | 较高，需 Cholesky 求逆 | 轻量，枚举 20 grid points |
| 泛化性 | 倾向 overfit calibration set | 不做 BP，泛化更强 |
| 格式 | INT4，per-group | INT4，per-group，有 zero_point |
| 生产落地 | vLLM GPTQ / GPTQ-Marlin | vLLM AWQ，推荐格式之一 |

## 5.5 FP8：浮点量化与硬件加速

### 5.5.1 格式设计

FP8 论文（arXiv:2209.05433）提出两种编码：

| 格式 | 指数位 | 尾数位 | 动态范围 | 用途 |
|------|--------|--------|----------|------|
| E4M3 | 4 | 3 | 较窄，精度高 | 权重/激活（前向推理） |
| E5M2 | 5 | 2 | 较宽，精度低 | 梯度（训练反向） |

NVIDIA Hopper（H100/H200）原生支持 E4M3fn（不含 NaN/Inf 表示，动态范围更大），GEMM 吞吐是 FP16 的 2×。

### 5.5.3 FP8 量化核心工具函数

**【真实源码 vllm@vllm/model_executor/layers/quantization/utils/quant_utils.py】**

```python
def scaled_quantize(
    x: torch.Tensor,
    group_shape: GroupShape,
    quant_dtype: torch.dtype,       # e.g. torch.float8_e4m3fn
    compute_dtype: torch.dtype | None = None,
) -> tuple[torch.Tensor, torch.Tensor]:
    # 1. 将 (M, N) reshape 为 (BLK_M, GRP_M, BLK_N, GRP_N)
    x_blkd = x_compute.reshape(blk_m, group_shape[0], blk_n, group_shape[1])
    # 2. permute → (BLK_M, BLK_N, GRP_M*GRP_N)，方便沿 group 维 reduce
    x_blkd_permd = x_blkd.permute(0, 2, 1, 3).flatten(start_dim=2)
    # 3. 计算每 block 的 scale = fp8_max / amax
    min_val, max_val = x_blkd_permd.aminmax(dim=-1)
    amax  = torch.maximum(min_val.abs(), max_val.abs()).clamp(min=1e-12)
    scale = fp8_max / amax
    # 4. apply scale，clamp，转 FP8
    x_scl_sat = (
        (x_blkd_permd * scale.unsqueeze(-1))
        .clamp(min=finfo.min, max=finfo.max)
        .reshape(blk_m, blk_n, group_shape[0], group_shape[1])
        .permute(0, 2, 1, 3).reshape(x.shape)
    )
    return x_scl_sat.to(quant_dtype).contiguous(), scale.float().reciprocal()
```

## 5.6 GGUF/GGML 量化：CPU 友好的分块格式

### 5.6.1 格式设计动机

llama.cpp 的目标是在 CPU / Apple Silicon 上跑 LLM。GGML quants 的关键设计约束：

1. **无 GPU 依赖**：格式对 SIMD (AVX2/NEON) 友好
2. **固定块大小**：每 32 或 256 个浮点数一块，一块一个（或多个）scale
3. **整型打包**：4-bit 两个塞一个 uint8，6-bit 用 ql+qh 拆分存储

### 5.6.2 关键 block 结构

**【真实源码 llama.cpp@ggml/src/ggml-common.h】**

```c
// Q4_0：最简单的 4-bit 对称量化，block 大小 32
#define QK4_0 32
typedef struct {
    ggml_half d;           // FP16 delta (scale)，1 个 per block
    uint8_t qs[QK4_0/2];  // 32 个 4-bit quants 打包进 16 个 uint8
} block_q4_0;
// sizeof = 2 + 16 = 18 bytes 存 32 个值，原始 FP32 需 128 bytes → 7.1× 压缩

// Q4_1：非对称，多了 min 字段
#define QK4_1 32
typedef struct {
    union {
        struct { ggml_half d; ggml_half m; };  // delta + min
        ggml_half2 dm;
    };
    uint8_t qs[QK4_1/2];
} block_q4_1;
// sizeof = 4 + 16 = 20 bytes，精度略优

// Q6_K："super-block" 格式，block 256 个值，精度更高
typedef struct {
    uint8_t ql[QK_K/2];      // 256 个值的低 4 bit，128 bytes
    uint8_t qh[QK_K/4];      // 256 个值的高 2 bit，64 bytes
    int8_t  scales[QK_K/16]; // 每 16 个值一个 int8 scale，16 bytes
    ggml_half d;              // super-block 的全局 scale，2 bytes
} block_q6_K;
// 6 bit/weight，QK_K=256 时 sizeof = 128+64+16+2 = 210 bytes

// Q4_K：4-bit K-quants，带二级 scale
typedef struct {
    union {
        struct { ggml_half d; ggml_half dmin; }; // super-block scale 和 min
        ggml_half2 dm;
    };
    uint8_t scales[K_SCALE_SIZE];  // 每 32 个值的 scale，6-bit 量化，12 bytes
    uint8_t qs[QK_K/2];            // 256 个值 4-bit 打包，128 bytes
} block_q4_K;
```

### 5.6.3 Q4_0 量化实现

**【真实源码 llama.cpp@ggml/src/ggml-quants.c】**

```c
void quantize_row_q4_0_ref(const float * GGML_RESTRICT x,
                           block_q4_0 * GGML_RESTRICT y,
                           int64_t k) {
    static const int qk = QK4_0;   // = 32
    const int nb = k / qk;         // 块数

    for (int i = 0; i < nb; i++) {
        // 1. 找 block 内 amax（绝对最大值）和 max（带符号）
        float amax = 0.0f, max = 0.0f;
        for (int j = 0; j < qk; j++) {
            const float v = x[i*qk + j];
            if (amax < fabsf(v)) { amax = fabsf(v); max = v; }
        }

        // 2. delta = max / -8（对称 4-bit 范围 [-8, 7]，-8 对应最大绝对值）
        const float d  = max / -8;
        const float id = d ? 1.0f / d : 0.0f;  // 1/delta，用于量化
        y[i].d = GGML_FP32_TO_FP16(d);         // 存为 FP16

        // 3. 量化并双打包（两个 4-bit 塞一个 uint8）
        for (int j = 0; j < qk/2; ++j) {
            const float x0 = x[i*qk         + j] * id;
            const float x1 = x[i*qk + qk/2  + j] * id;
            // +8.5 偏移后截断到 [0,15]（nibble 范围）
            const uint8_t xi0 = MIN(15, (int8_t)(x0 + 8.5f));
            const uint8_t xi1 = MIN(15, (int8_t)(x1 + 8.5f));
            y[i].qs[j] = xi0 | (xi1 << 4);   // 低 nibble | 高 nibble
        }
    }
}
```

**逐行注解：**
- `max / -8`：Q4_0 使用 offset=8，量化范围 [0,15] → 反量化后 [-8,7]；max 是带符号最大值（最大负数），除以 -8 得正 delta
- `GGML_FP32_TO_FP16`：delta 存 FP16，推理时反量化时乘回去
- 双 nibble 打包：低 4 bit 存 xi0，高 4 bit 存 xi1，节省一半存储

## 5.7 INT4 打包与 Packing 机制

推理引擎需要把量化后的 INT4 打包进 INT32 tensor 才能高效处理。

**【真实源码 vllm@vllm/model_executor/layers/quantization/utils/quant_utils.py】**

```python
def pack_quantized_values_into_int32(
    w_q: torch.Tensor, wtype: ScalarType, packed_dim: int = 0
):
    # 把待打包维度移到最后
    perm    = (*[i for i in range(len(w_q.shape)) if i != packed_dim], packed_dim)
    inv_perm = tuple(perm.index(i) for i in range(len(perm)))
    w_q_perm = w_q.permute(perm)

    pack_factor = 32 // wtype.size_bits   # INT4: 8个值塞一个INT32
    mask        = (1 << wtype.size_bits) - 1  # 0xF for 4-bit

    new_shape_perm        = list(w_q_perm.shape)
    new_shape_perm[-1]  //= pack_factor   # 最后维缩小 8×

    res = torch.zeros(new_shape_perm, dtype=torch.int32, device=w_q.device)
    for i in range(pack_factor):
        # 每次取 stride=pack_factor 间隔的元素，左移对应偏移量后 OR 进去
        res |= (w_q_perm[..., i::pack_factor] & mask) << (wtype.size_bits * i)
    return res.permute(inv_perm)
```

**注解：**
- 4-bit 时 pack_factor=8，一个 INT32 存 8 个 INT4
- `w_q_perm[..., i::pack_factor]`：取第 0, 8, 16... 位置（i=0），第 1, 9, 17...（i=1），等等
- 最终内存布局：`[q0 | (q1<<4) | (q2<<8) | ... | (q7<<28)]`
- kernel 侧反向操作：`(packed >> (4*i)) & 0xF` 取出每个 INT4

## 5.8 bitsandbytes：混合精度 INT8 分解

### 核心思想（arXiv:2208.07339）

LLM 中存在 **emergent outlier features**：某些隐层维度（约 0.1%）的激活值异常大（>6σ），导致整张量统一 INT8 量化时误差极大。

解决方案：**混合精度矩阵乘法分解**

```
Y = X @ W
  = X_outlier @ W_outlier + X_normal @ W_normal
  ≈ FP16(X_outlier @ W_outlier) + INT8(X_normal @ W_normal)
```

- `X_outlier`：包含异常值的极少数列（~0.1%）→ 保持 FP16
- `X_normal`：其余 99.9%+ 的列 → INT8 向量量化（每行独立 scale）

实现代价：两次 GEMM（FP16 + INT8），加上异常列提取的 overhead；但实现了对任意 checkpoint 的**零样本无损 8-bit 量化**，不需离线校正。

## 5.9 方案全景对比

| 方法 | Bit-width | 粒度 | 量化对象 | 需校正集 | 推理 overhead | 适用场景 |
|------|-----------|------|----------|---------|--------------|---------|
| FP8 (static) | 8-bit float | per-tensor / block | W + A | 是（静态scale） | 极低（H100原生） | Hopper 生产服务 |
| FP8 (dynamic) | 8-bit float | per-token | W + A | 否 | 低（实时scale）| H100 灵活部署 |
| GPTQ | INT4 | per-group | W only | 是（128 tok） | 中（decode快，prefill慢） | GPU decode 加速 |
| AWQ | INT4 | per-group | W only | 是（统计离线） | 中 | GPU decode，泛化好 |
| bitsandbytes INT8 | INT8 | per-row | W + A | 否（在线） | 中（两次GEMM） | 无需校正的快速部署 |
| bitsandbytes NF4 | 4-bit nf | per-block | W only | 否（在线）| 高（实时dequant）| LoRA fine-tune，QLoRA |
| GGUF Q4_K_M | 4-bit | per-block-256 | W only | 否（格式转换）| 低（CPU/Metal） | 边缘/本地 CPU 推理 |

**不适用边界：**
- GPTQ/AWQ 在 prefill 大 batch 下收益有限（compute bound 时 memory BW 不是瓶颈）
- FP8 需要 Hopper 及以上，Ampere 无硬件加速
- Q4_0/GGUF 在大 batch GPU 服务场景性能不如 Marlin GPTQ
- bitsandbytes INT8 decode 比 FP16 有时更慢（两次 GEMM 开销）

### Demo 1：手动实现 INT8/INT4 量化并测量误差

```python
#!/usr/bin/env python3
"""
Demo 1: 手动量化 - 对称 INT8 / 非对称 INT4 / 分组量化
依赖: numpy >= 1.21, (可选) torch >= 2.0
设计为可运行，请在你的环境验证（纯 numpy，无 GPU 需求）
"""
import numpy as np

np.random.seed(42)

# 模拟一个 Linear 层权重矩阵，形状 [256, 512]（out_features × in_features）
W = np.random.randn(256, 512).astype(np.float32) * 0.02  # 近似真实权重分布

# ── 1. 对称 INT8 量化（per-tensor） ──────────────────────────────────────────
def symmetric_quantize_int8(W: np.ndarray):
    """对称量化：scale = amax / 127"""
    amax  = np.abs(W).max()
    scale = amax / 127.0
    W_q   = np.clip(np.round(W / scale), -128, 127).astype(np.int8)
    return W_q, scale

def symmetric_dequantize_int8(W_q: np.ndarray, scale: float):
    return W_q.astype(np.float32) * scale

W_q8, scale8 = symmetric_quantize_int8(W)
W_deq8        = symmetric_dequantize_int8(W_q8, scale8)
err8          = np.abs(W - W_deq8)
print(f"[INT8 symmetric per-tensor]")
print(f"  原始 size  : {W.nbytes} bytes")
print(f"  量化后 size: {W_q8.nbytes} bytes (scale: {W_q8.nbytes + 4})")
print(f"  压缩比     : {W.nbytes / (W_q8.nbytes + 4):.2f}x")
print(f"  MAE        : {err8.mean():.6f}")
print(f"  Max error  : {err8.max():.6f}")
print()

# ── 2. 非对称 INT4 量化（per-tensor） ────────────────────────────────────────
def asymmetric_quantize_int4(W: np.ndarray):
    """非对称量化：scale=(max-min)/15, zero=round(-min/scale)"""
    wmax   = W.max()
    wmin   = W.min()
    scale  = (wmax - wmin) / 15.0
    scale  = max(scale, 1e-8)  # 防除零
    zero   = int(np.clip(np.round(-wmin / scale), 0, 15))
    W_q    = np.clip(np.round(W / scale) + zero, 0, 15).astype(np.uint8)
    return W_q, scale, zero

def asymmetric_dequantize_int4(W_q: np.ndarray, scale: float, zero: int):
    return (W_q.astype(np.float32) - zero) * scale

W_q4, scale4, zero4 = asymmetric_quantize_int4(W)
W_deq4               = asymmetric_dequantize_int4(W_q4, scale4, zero4)

# INT4 实际存储：两个值打包进一个 uint8（nibble packing）
def pack_int4(W_q4: np.ndarray):
    """将 INT4 权重矩阵（uint8, 值域[0,15]）按行 nibble packing"""
    assert W_q4.shape[1] % 2 == 0, "列数需为偶数"
    packed = (W_q4[:, 0::2] & 0x0F) | ((W_q4[:, 1::2] & 0x0F) << 4)
    return packed.astype(np.uint8)

W_packed = pack_int4(W_q4)
err4 = np.abs(W - W_deq4)
print(f"[INT4 asymmetric per-tensor]")
print(f"  量化后 packed size: {W_packed.nbytes} bytes")
print(f"  压缩比            : {W.nbytes / (W_packed.nbytes + 4 + 1):.2f}x")  # +scale+zero
print(f"  MAE               : {err4.mean():.6f}")
print(f"  Max error         : {err4.max():.6f}")
print()

# ── 3. 分组量化 INT4（group_size=32，模拟 GPTQ/AWQ 的 per-group） ─────────────
def group_quantize_int4(W: np.ndarray, group_size: int = 32):
    """
    per-group 非对称 INT4 量化
    W shape: [out_features, in_features]
    每 group_size 个 in_features 共享一组 scale+zero
    """
    out_f, in_f = W.shape
    assert in_f % group_size == 0
    n_groups = in_f // group_size
    W_q      = np.zeros_like(W, dtype=np.uint8)
    scales   = np.zeros((out_f, n_groups), dtype=np.float32)
    zeros    = np.zeros((out_f, n_groups), dtype=np.int32)

    for g in range(n_groups):
        chunk   = W[:, g*group_size:(g+1)*group_size]   # [out_f, group_size]
        wmax    = chunk.max(axis=1, keepdims=True)        # per-row max
        wmin    = chunk.min(axis=1, keepdims=True)
        s       = (wmax - wmin) / 15.0
        s       = np.maximum(s, 1e-8)
        z       = np.clip(np.round(-wmin / s), 0, 15).astype(np.int32)
        q       = np.clip(np.round(chunk / s) + z, 0, 15).astype(np.uint8)
        W_q[:, g*group_size:(g+1)*group_size] = q
        scales[:, g] = s[:, 0]
        zeros[:, g]  = z[:, 0]

    return W_q, scales, zeros

def group_dequantize_int4(W_q, scales, zeros, group_size=32):
    out_f, in_f = W_q.shape
    n_groups    = in_f // group_size
    W_deq       = np.zeros_like(W_q, dtype=np.float32)
    for g in range(n_groups):
        s = scales[:, g:g+1]    # [out_f, 1]
        z = zeros[:, g:g+1]
        W_deq[:, g*group_size:(g+1)*group_size] = \
            (W_q[:, g*group_size:(g+1)*group_size].astype(np.float32) - z) * s
    return W_deq

W_qg, scales_g, zeros_g = group_quantize_int4(W, group_size=32)
W_deqg                   = group_dequantize_int4(W_qg, scales_g, zeros_g, 32)

scale_bytes = scales_g.nbytes + zeros_g.nbytes  # scale/zero metadata
quant_bytes = W_qg.nbytes // 2                  # INT4 packing
err_g = np.abs(W - W_deqg)
print(f"[INT4 asymmetric per-group (group_size=32)]")
print(f"  权重 packed size  : {quant_bytes} bytes")
print(f"  scale+zero size   : {scale_bytes} bytes")
print(f"  总计              : {quant_bytes + scale_bytes} bytes")
print(f"  压缩比            : {W.nbytes / (quant_bytes + scale_bytes):.2f}x")
print(f"  MAE               : {err_g.mean():.6f}")
print(f"  Max error         : {err_g.max():.6f}")
print()

# ── 4. 对比汇总 ───────────────────────────────────────────────────────────────
print("=" * 60)
print(f"  原始 FP32 size   : {W.nbytes:>8} bytes   MAE: ---")
print(f"  INT8 per-tensor  : {W_q8.nbytes:>8} bytes   MAE: {err8.mean():.6f}")
print(f"  INT4 per-tensor  : {W_packed.nbytes:>8} bytes   MAE: {err4.mean():.6f}")
print(f"  INT4 per-group32 : {quant_bytes+scale_bytes:>8} bytes   MAE: {err_g.mean():.6f}")
print("=" * 60)
```

### Demo 2：模拟 GGUF Q4_0 格式打包与反量化

```python
#!/usr/bin/env python3
"""
Demo 2: 模拟 llama.cpp GGUF Q4_0 格式
对照真实源码 ggml/src/ggml-quants.c 的 quantize_row_q4_0_ref
依赖: numpy 只 (无 GPU 需求)
设计为可运行，请在你的环境验证
"""
import numpy as np
import struct

QK4_0 = 32  # block size

def quantize_q4_0(weights_flat: np.ndarray):
    """
    对应 llama.cpp ggml/src/ggml-quants.c: quantize_row_q4_0_ref
    输入:  weights_flat: [N] float32，N 必须是 QK4_0 的倍数
    输出:  blocks: list of (d_fp16, qs_packed) per block
    """
    assert len(weights_flat) % QK4_0 == 0
    nb = len(weights_flat) // QK4_0
    blocks = []

    for i in range(nb):
        block = weights_flat[i*QK4_0 : (i+1)*QK4_0]

        # 1. 找 max（带符号，不是 amax）和 amax
        amax = 0.0
        max_val = 0.0
        for v in block:
            if abs(v) > amax:
                amax = abs(v)
                max_val = v

        # 2. delta = max / -8（Q4_0 的量化范围偏移到 [-8,7]，-8 对应 amax 处）
        d   = max_val / -8.0
        id_ = (1.0 / d) if d != 0.0 else 0.0

        # 3. delta 存 FP16
        d_fp16 = np.float16(d)

        # 4. 量化：scale，+8.5 偏移截断，nibble packing
        qs = np.zeros(QK4_0 // 2, dtype=np.uint8)
        for j in range(QK4_0 // 2):
            x0 = block[j]           * id_    # 前半 block
            x1 = block[j + QK4_0//2] * id_   # 后半 block
            xi0 = min(15, int(x0 + 8.5))
            xi1 = min(15, int(x1 + 8.5))
            qs[j] = xi0 | (xi1 << 4)         # 低 nibble | 高 nibble

        blocks.append((d_fp16, qs))

    return blocks

def dequantize_q4_0(blocks):
    """反量化 Q4_0 blocks → float32 array"""
    result = []
    for (d_fp16, qs) in blocks:
        d = float(d_fp16)
        floats = np.zeros(QK4_0, dtype=np.float32)
        for j in range(QK4_0 // 2):
            # 取低 nibble 和高 nibble
            xi0 = qs[j] & 0x0F
            xi1 = (qs[j] >> 4) & 0x0F
            # 反偏移：减去 8，乘以 delta
            floats[j]           = (xi0 - 8) * d
            floats[j + QK4_0//2] = (xi1 - 8) * d
        result.extend(floats)
    return np.array(result, dtype=np.float32)

def block_size_bytes():
    """block_q4_0 的大小：2字节FP16 delta + QK4_0/2 字节 qs = 18 bytes"""
    return 2 + QK4_0 // 2  # = 18

# ── 测试 ──────────────────────────────────────────────────────────────────────
np.random.seed(7)
N = 256  # 8 blocks
weights = np.random.randn(N).astype(np.float32) * 0.02

blocks = quantize_q4_0(weights)
weights_deq = dequantize_q4_0(blocks)

total_bytes_orig  = weights.nbytes
total_bytes_quant = len(blocks) * block_size_bytes()
mae  = np.abs(weights - weights_deq).mean()
mse  = np.square(weights - weights_deq).mean()

print(f"Q4_0 量化结果:")
print(f"  输入 N={N}, 原始 FP32 size : {total_bytes_orig} bytes")
print(f"  blocks 数                 : {len(blocks)}")
print(f"  每 block size             : {block_size_bytes()} bytes (2 FP16 delta + 16 uint8)")
print(f"  量化后 total size         : {total_bytes_quant} bytes")
print(f"  压缩比                    : {total_bytes_orig / total_bytes_quant:.2f}x")
print(f"  MAE                       : {mae:.6f}")
print(f"  MSE                       : {mse:.8f}")
print()
# 打印第一个 block 的详情
d0, qs0 = blocks[0]
print(f"Block 0:")
print(f"  delta (FP16) = {d0}")
print(f"  qs[:4]       = {[hex(x) for x in qs0[:4]]}")
print(f"  原始 [0:4]   = {weights[:4]}")
print(f"  反量化 [0:4] = {weights_deq[:4]}")
```

### Demo 3：FP8 量化的 scale 计算（对照 vLLM scaled_quantize）

```python
#!/usr/bin/env python3
"""
Demo 3: FP8 E4M3 量化 - 模拟 vLLM scaled_quantize 的 block-wise 量化逻辑
依赖: torch >= 2.1（需要 float8_e4m3fn 支持），无 GPU 需求（CPU 运行）
设计为可运行，请在你的环境验证
注：torch.float8_e4m3fn 需 torch >= 2.1；如版本不支持，可用 float16 观察 scale 计算逻辑
"""
import torch
import numpy as np

def fp8_block_quantize(x: torch.Tensor, block_size: int = 128):
    """
    对照 vLLM quant_utils.py: scaled_quantize
    对矩阵 x 做 per-block FP8 E4M3 量化
    block_size: 每个量化块的大小（方形块）
    """
    assert x.ndim == 2
    M, N = x.shape
    assert M % block_size == 0 and N % block_size == 0, \
        f"shape {x.shape} 须能被 block_size={block_size} 整除"

    FP8_MAX = 448.0  # torch.float8_e4m3fn 的最大值 = 2^(2^(4-1)-1) * (1 + (2^3-1)/2^3)
                     # ≈ 448（近似，实际 = 448.0）

    blk_m, blk_n = M // block_size, N // block_size

    # reshape: (M,N) → (BLK_M, BS, BLK_N, BS) → permute → (BLK_M, BLK_N, BS*BS)
    x_blkd       = x.reshape(blk_m, block_size, blk_n, block_size)
    x_blkd_p     = x_blkd.permute(0, 2, 1, 3)             # (BLK_M, BLK_N, BS, BS)
    x_flat       = x_blkd_p.flatten(start_dim=2)           # (BLK_M, BLK_N, BS*BS)

    # 计算每 block 的 amax → scale
    min_val, max_val = x_flat.aminmax(dim=-1)               # (BLK_M, BLK_N)
    amax  = torch.maximum(min_val.abs(), max_val.abs()).clamp(min=1e-12)
    scale = FP8_MAX / amax                                   # (BLK_M, BLK_N)

    # apply scale, clamp, 模拟 FP8 截断（round to FP8 精度）
    x_scaled = (x_flat * scale.unsqueeze(-1)).clamp(-FP8_MAX, FP8_MAX)

    # 尝试真实 FP8 转换（torch >= 2.1）
    try:
        x_fp8    = x_scaled.reshape(blk_m, blk_n, block_size, block_size) \
                            .permute(0, 2, 1, 3).reshape(M, N)
        x_fp8_t  = x_fp8.to(torch.float8_e4m3fn)
        use_real_fp8 = True
    except Exception:
        x_fp8_t  = x_scaled.reshape(blk_m, blk_n, block_size, block_size) \
                            .permute(0, 2, 1, 3).reshape(M, N)
        use_real_fp8 = False

    inv_scale = scale.float().reciprocal()  # 存储的是 1/scale（反量化时乘）
    return x_fp8_t, inv_scale, use_real_fp8

def fp8_block_dequantize(x_fp8, inv_scale: torch.Tensor, block_size: int = 128):
    """反量化：乘以 inv_scale"""
    x_f32 = x_fp8.float()
    M, N  = x_f32.shape
    blk_m, blk_n = M // block_size, N // block_size

    # 把 (BLK_M, BLK_N) 的 inv_scale 广播到每个元素
    x_blkd_p = x_f32.reshape(blk_m, block_size, blk_n, block_size) \
                     .permute(0, 2, 1, 3)                      # (BLK_M, BLK_N, BS, BS)
    x_deq    = x_blkd_p * inv_scale.unsqueeze(-1).unsqueeze(-1)
    return x_deq.permute(0, 2, 1, 3).reshape(M, N)

# ── 测试 ──────────────────────────────────────────────────────────────────────
torch.manual_seed(42)
x = torch.randn(512, 512, dtype=torch.float32)  # 模拟 FFN 激活

x_fp8, inv_scale, real_fp8 = fp8_block_quantize(x, block_size=128)
x_deq                       = fp8_block_dequantize(x_fp8, inv_scale, block_size=128)

mae = (x - x_deq).abs().mean().item()
print(f"FP8 block-wise 量化 (block_size=128):")
print(f"  使用真实 float8_e4m3fn : {real_fp8}")
print(f"  输入 shape             : {x.shape}")
print(f"  scale 张量 shape       : {inv_scale.shape}  (每 block 一个)")
print(f"  scale 张量 dtype       : {inv_scale.dtype}")
print(f"  MAE（FP32 vs 反量化）  : {mae:.6f}")
print(f"  scale 样本值 [0,0]     : {inv_scale[0,0]:.4f}")
print(f"  scale 样本值 [1,2]     : {inv_scale[1,2]:.4f}")
print()
print(f"  对照 vLLM scaled_quantize 注解:")
print(f"  - 存储的是 1/scale（reciprocal），反量化时乘以 inv_scale")
print(f"  - per-block scale shape = (M/BS, N/BS) = ({512//128}, {512//128})")
```

## 5.11 失败模式与生产真坑

### 坑 1：Prefill 阶段 INT4 反而更慢

**现象**：用了 GPTQ INT4 模型，输出 first token latency 反而比 FP16 高。  
**根因**：Prefill 是 compute-bound，INT4 需要 dequant 后才能 GEMM，引入额外 overhead；而 FP16 直接走 cuBLAS GEMM，充分利用 TensorCore。  
**解法**：INT4 的收益在 decode 阶段（memory-bound）；Prefill 用 FP8 或 FP16 + INT4 混合策略（少数 prefill 专用路径保持 FP16）。vLLM AWQ apply 里的 `FP16_MATMUL_HEURISTIC_CONDITION = x.shape[:-1].numel() >= 256` 正是这个判断。

### 坑 2：Outlier 导致 INT8 激活量化精度崩溃

**现象**：INT8 W+A 量化后 perplexity 从 5.3 飙到 20+。  
**根因**：部分隐层维度存在 emergent outlier（>6σ），per-tensor scale 被这些异常值撑大，正常值被压到接近 0。  
**解法**：bitsandbytes 的混合精度分解（异常维度保持 FP16）；或改用 weight-only 量化（激活不量化）；或 SmoothQuant（将激活 outlier 迁移到权重侧）。

### 坑 3：group_size 与 tensor parallelism 不兼容

**现象**：tensor_parallel_size=8 + group_size=128 → RuntimeError（权重切片尺寸不整除）。  
**根因**：TP 把 in_features 切成 P 份，每份 = in_features/P 必须是 group_size 的倍数，否则 scale 对齐错误。  
**解法**：选 group_size 时满足 `in_features / TP % group_size == 0`；vLLM create_weights 里有明确 assert：
```python
if input_size_per_partition % group_size != 0:
    raise ValueError("The input size is not aligned with the quantized weight shape.")
```

### 坑 4：FP8 KV Cache 精度损失累积

**现象**：FP8 KV Cache 开启后长序列（>8k）输出质量明显下降。  
**根因**：KV cache 量化引入的误差在 attention softmax 中非线性放大；E4M3 动态范围窄，异常 key 值被截断。  
**解法**：per-token dynamic scale 比 per-tensor static scale 更稳定；部分场景只量化 value 不量化 key；或降低 KV cache 量化精度（用 INT8 代替 FP8）。

### 坑 5：GGUF 量化模型 GPU Offload 显存计算错误

**现象**：llama.cpp GPU offload 时 VRAM 估算偏小，导致 OOM。  
**根因**：Q4_K_M 的 scale metadata（super-block scales）被忽略未计入显存估算。  
**解法**：正确计算 block_q4_K size = 2+2+12+128 = 144 bytes per 256 weights，不能简单用 "4bit = W*H/2"。

### 坑 6：量化 checkpoint 格式不匹配

**现象**：加载 AWQ 量化模型到 vLLM 报错 shape mismatch。  
**根因**：AutoAWQ 打包格式（column-major interleaving）与 vLLM 内部 kernel 期望的格式不同；版本间格式有变化。  
**解法**：使用 vLLM 自己提供的量化模型列表；或用 `process_weights_after_loading` 做在线格式转换（vLLM fp8.py 中有此逻辑）。

## 5.13 章末五件套

### 1. 核心概念问答

**Q1：为什么 per-group 量化精度优于 per-tensor，代价是什么？**  
A：per-group 在每 group_size 个权重内独立搜索 scale，适应局部权重分布的波动；代价是 scale metadata 体积增加（`out_features × in_features / group_size` 个 scale）和 kernel 侧需要按 group 加载 scale。GPTQ/AWQ 选 group_size=128 是精度/overhead 的工程折衷。

**Q2：FP8 的 E4M3 和 E5M2 分别适合什么？**  
A：E4M3（4 位指数，3 位尾数）尾数精度高，适合权重/前向激活；E5M2（5 位指数，2 位尾数）动态范围宽，适合梯度（训练中梯度跨度大）。推理一般只用 E4M3fn。

**Q3：AWQ 为什么用 activation 统计而不是 weight 统计？**  
A：权重本身各通道方差差异不大；但激活的 per-channel 范围差异可以很大（salient channels 被大量 token 激活，乘以这些 channels 的权重误差会被放大）。AWQ 论文证明：outlier 的 channel 由激活分布决定，与权重大小无关。

### 2. 方案选型决策树

```
需要部署量化模型？
├── 硬件是 H100/H200？
│   ├── Yes → 优先 FP8 W+A（static/dynamic）
│   └── No  → 继续↓
├── 只在 GPU 上跑，显存紧张？
│   ├── 追求最低延迟（decode）→ AWQ INT4（per-group, Marlin kernel）
│   └── 无校正集 / 即插即用 → bitsandbytes INT8 或 NF4
├── CPU / Apple Silicon 推理？
│   └── → GGUF Q4_K_M 或 Q6_K
└── 极限压缩（<4bit）？
    └── → 研究阶段，AQLM/GPTQ 2-bit，生产谨慎
```
