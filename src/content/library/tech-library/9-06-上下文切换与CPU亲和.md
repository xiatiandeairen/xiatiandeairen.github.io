> **一句话定位**：上下文切换是调度器从"决定运行谁"到"让它真的跑起来"的最后一公里。这一公里在 v6.12 内核里只有约 30 行 C（`context_switch()`）加几十条汇编（`__switch_to_asm`），但它牵动 mm 引用计数、membarrier 内存序、lazy TLB、PCID/ASID、FPU/AMX 巨型状态、RSB 填充——把这一公里吃透，等于把"调度落地层"和"微架构状态层"打通。

## TL;DR — 本章最硬结论

1. **裸 `switch_to` 是几十条指令、~几十 ns；`__schedule()` 整条路径 ~数百 ns–1 μs；而 cache/TLB cold start 的"尾效应"能再放大 10x–100x。** 多数人只量到中间那层，把它当成了全部代价。**真正治本的是工作集局部性，不是切换频率。**
2. **`context_switch()` 里 90% 的复杂度不在"换寄存器"，而在"换 mm 时如何不破坏 membarrier 的内存序、如何省掉 lazy TLB 的页表切换、如何安全地 drop 旧 mm"。** 换寄存器只是最后一行 `switch_to(prev, next, prev)`。
3. **voluntary / involuntary 的内核判据比"主动 sleep vs 时间片耗尽"更精确**：`__schedule()` 里 `switch_count` 默认指向 `nivcsw`，**只有当 `!preempt && prev_state` 且任务真的被 `block_task()` 摘下 runqueue** 时才改指向 `nvcsw`。一个"设了睡眠态但在 `schedule()` 前被信号唤醒"或"被抢占"的任务，仍计入 involuntary。
4. **lazy FPU 早已被 eager 取代，驱动力是性能不是安全**：glibc 的 `memcpy`/`memset` 用 SSE2，几乎每个进程一启动就碰 XMM，"等真用到再存"的假设破产；操作 `CR0.TS` 触发 `#NM` 比直接 `XSAVE`/`XRSTOR` 还慢。eager 在 ~4.6 成默认、~4.9 hard-disable lazy；**CVE-2018-3665（LazyFP, 2018-06）是事后追认，先改后爆，因果不能倒置。**
5. **PCID 硬件 12 bit = 4096 个，但 Linux 每 CPU 只用 `TLB_NR_DYN_ASIDS = 6` 个**——源码注释原话是"6 because 6 should be plenty and struct tlb_state will fit in two cache lines"。真实世界"何时被迫刷 TLB"取决于这个 6 槽 LRU 环，不是 4096。
6. **CPU 亲和性是局部性旋钮，不是"绑越多越好"**；`taskset` 只是"优先选这些核"，真正隔离要 `isolcpus`（且它已被官方视为半弃用，cpuset/cgroup v2 `isolated` partition 是现代姿势）。中断亲和与任务亲和正交，低延迟系统必须同拧。

**前置依赖章**：第 5 章（调度器机制：CFS/EEVDF、run-queue、`pick_next_task`、调度时机），第 4 章（进程/线程模型，`task_struct`、`mm_struct`、`active_mm`），第 8 章（NUMA 内存架构，互相引用）。**微架构前置**：TLB / page table walk / 投机执行 / cache 一致性的基本概念（本章会用到，但不展开讲微架构本身）。

> **本章源码基线**：除非另注，所有源码引用基于 **Linux v6.12**（2024-11 发布，PREEMPT_RT 首次进主线的版本）。逐字源码标【真实源码 v6.12 路径】；为讲清而简化的标【示意代码，非逐字】；fetch 不到又没把握的具体符号标「待核」。

## 6.1 背景：上下文切换解决什么问题，引入什么代价

### 6.1.1 为什么需要上下文切换

CPU 是严格串行的执行单元——某一时刻一个逻辑核只执行一段指令流。操作系统要在有限的核上并发运行成千上万个任务，唯一的办法是**时间复用**：把 CPU 从当前任务"剥走"，交给另一个任务，周期性轮换。这个"剥走+交出"就是上下文切换（context switch）。

上下文切换必须做三件事，但**只有第一件是 `switch_to` 的本职**，后两件是 `switch_mm` + 微架构的隐性代价：

- 保存/恢复当前任务的**执行状态**（通用寄存器、栈指针、FPU/向量状态、段基址……）——`switch_to` 负责。
- 如果跨进程：**切换地址空间**（换页表，写 `CR3`）——`switch_mm` 负责。
- 更新 CPU 的"视角"：**TLB、L1/L2/LLC cache、分支预测器/RSB 历史**被新任务逐步污染——没有谁"负责"，它是物理后果。

后两条才是真正的代价来源。本章的组织逻辑就是：**先讲清这三件事在 v6.12 源码里各自落在哪个函数、为什么这么切，再讲它们的微架构代价、生产排障与亲和性调优。**

### 6.1.2 类比：数据库连接池的代价（保留并加深）

你熟悉"连接池复用"——建立 TCP + TLS 握手的固定开销让短连接不可接受。上下文切换同构：

- 直接保存/恢复寄存器 ≈ 握手的 RTT，固定的几十~几百 ns。
- TLB 失效、L1/L2 被污染 ≈ "把连接的预热状态（拥塞窗口、TLS session、应用层缓存）全丢掉"——这才是大头。

但类比有个**关键不对称**，是连接池没有的：上下文切换还要维护**地址空间引用计数**（`mm_count`）和**内存序屏障**（membarrier）。连接池里"换一个连接"不需要全局原子操作；而内核切到 kernel thread 时对 `active_mm` 的 `mmgrab/mmdrop` 在多核上会引发 cache-line bouncing——这是连接池类比照不到的暗坑，6.3.3 会用真实 LWN 史料讲透。

## 6.2 演进：设计考古（谁、何时、为什么、辩论过什么）

> 本节是本次深化的重点之一。每个史料锚点尽量带出处（LWN 文章号 / commit / 邮件线程），并已实际核实。

### 6.2.1 早期内核：硬件 TSS 切换的弃用

Linux 0.x 时代用 x86 的**硬件 TSS（Task State Segment）** 切换——CPU 原生支持把整个任务状态保存到内存里的 TSS。方便，但慢：TSS 切换要刷 TLB、重载一堆描述符、做大量微码工作，且和 SMP/抢占模型格格不入。Linux 很早改为**软件上下文切换**：只手动保存内核真正用到的 callee-saved 寄存器。这套思路一直延续到今天的 `__switch_to_asm`（6.3.2 会读它的真实汇编）。

### 6.2.2 FPU：从 lazy 到 eager 的真实历史与社区辩论

这是本章最容易讲反、也最值得考古的一段。

**lazy FPU 的初衷（~2010 及更早）**：大多数进程不碰浮点，每次切换都 `FXSAVE`/`FXRSTOR` 整套 XMM 像是纯浪费。于是用 **CR0.TS 位 + `#NM`（device-not-available）异常**做惰性：切换时不存 FPU，只置 TS；新任务真正执行第一条 FPU 指令时触发 `#NM`，内核才在异常处理里保存旧任务、恢复新任务的 FPU 状态。LWN 当年有 "Really lazy fpu"（[LWN.net/Articles/391972](https://lwn.net/Articles/391972/), 2010-06）讨论把它做得更激进。

**为什么放弃 lazy（核心驱动力 = 性能，不是安全）**：

- **Ingo Molnar 的大重写（2015-05，208 个 patch）**：[big x86 FPU code rewrite, LWN.net/Articles/643235](https://lwn.net/Articles/643235/)。Molnar 直言旧 FPU 代码"organically grown into somewhat of a spaghetti monster that few (if any) kernel developers understand"。这次重写把 FPU 代码归到 `arch/x86/kernel/fpu/`、把 FPU 状态**直接内嵌进 task struct**、消除复杂的延迟分配。它为"砍掉 lazy 路径"铺平了道路。
- **eager 成默认（~kernel 4.6）**：真正的理由是 **glibc 的 `memcpy`/`memset`/`memcmp` 等热路径普遍用 SSE2**——几乎每个进程一 `exec` 就碰 XMM 寄存器，lazy 的"等真用到再保存"假设根本不成立；既然几乎必触发 `#NM`，那操作 TS 位 + 走异常 *比* 直接 `XSAVE`/`XRSTOR` 一把存 *还慢*。双路径（lazy + eager）只有一条被充分测试，是纯维护负担。
- **lazy 被 hard-disable（~kernel 4.9，commit `ca6938a1cd8a` "x86/fpu: Hard-disable lazy FPU mode"）**，随后 [PATCH] "x86/fpu: Remove 'eagerfpu=' docs" 把参数从文档移除（见 [narkive 邮件存档](https://linux.kernel.narkive.com/54E9kS98/patch-x86-fpu-remove-eagerfpu-docs)）。到 Molnar 的后续 "x86/fpu: Simplify the FPU state machine"（[LWN.net/Articles/712641](https://lwn.net/Articles/712641/), 2017-01）时，提交信息已经写明"**Now that the lazy FPU switching code has been removed** this series capitalizes on the simpler FPU context switching machinery"——lazy 路径事实上从内核删掉了。

**CVE-2018-3665（LazyFP）是事后追认，不是改的原因**：

这个 side-channel（[Cyberus 技术博客](https://cyberus-technology.de/en/articles/intel-lazyfp-vulnerability/)，发现者 Julian Stecklina(Amazon) / Thomas Prescher(Cyberus)，2018-06-06 公开，影响 **Sandy Bridge 到 Skylake**）的机制是：lazy 模式下 CR0.TS=1、FPU 被禁，但**上一个进程的 XMM/YMM 残留在物理 FPU 寄存器里**；攻击者执行一条被禁的 FPU 指令，CPU **投机执行**会在 `#NM` 真正生效前把那个残留值读出来、用它做一次访存地址计算，在 cache 里留下痕迹；随后 `#NM` 把投机结果 squash，但 cache 副作用还在，用 flush-and-reload 就能逐位还原别人的 FPU 数据（与 Meltdown/Spectre 同家族）。

Cyberus 给的内核版本对照表（核实过）：

| 内核版本 | 受影响条件 |
|---------|-----------|
| < 3.7 | 所有 CPU 都受影响（无 eager 选项） |
| < 4.6 | 仅非默认参数 或 Haswell 之前的 CPU |
| < 4.9 且 `eagerfpu=off` | 受影响 |
| **≥ 4.9** | **无条件免疫**（lazy 已删） |

> ⚠️ **三个常见讲反点，逐一钉死**：
> - 不是"AVX 普及后才审视 lazy"——AVX 是 2011 Sandy Bridge，时间错位；真正动因是 glibc 用 **SSE2**（更早、更普遍）。
> - 不是"为了 Spectre/Meltdown 才 eager"——**先改（4.6/4.9, 2016）后爆（2018）**，因果倒置。
> - eager 化"没有性能损失"反而**净赚**（Cyberus 原文 "No performance penalty resulted"），因为省掉了 `#NM` 往返。安全是 bonus。

### 6.2.3 lazy TLB 的设计与一次真实的可扩展性辩论

"FPU lazy"已死，但**"lazy TLB"还活着，且是另一回事**——别混。

**lazy TLB 是什么**：当 CPU 切到一个 **kernel thread（`next->mm == NULL`，如 `kworker`、`ksoftirqd`）** 时，kernel thread 没有自己的用户地址空间，它只跑在内核态——内核态映射在所有进程页表里都一样。所以**没必要换 `CR3`、没必要刷用户 TLB**：内核让 kernel thread "借用"前一个进程的 mm 作为 `active_mm`，页表原封不动。这就是 6.3.1 源码里 `if (!next->mm)` 那个分支的全部动机。

**它引入了什么问题——一次真实社区辩论**：既然借用了别人的 mm，就得防止那个 mm 在 kernel thread 还在用时被释放，于是要 `mmgrab_lazy_tlb()` 增引用、`mmdrop_lazy_tlb()` 减引用。在**几百核**的大机器上，无数 CPU 频繁地切进切出 kernel thread，都去原子地增减**同一个** `mm_struct` 的 `mm_count` 字段——这个 cache line 在 NUMA 各节点间疯狂 bouncing，成为可扩展性瓶颈。

- **Nicholas Piggin 的 "shoot lazy tlbs" 系列**（[LWN.net/Articles/920349](https://lwn.net/Articles/920349/) 与 [922129](https://lwn.net/Articles/922129/)，~2023-01）：思路是**不再对 active_mm 做引用计数**，而是在进程退出做最终 TLB shootdown IPI 时，**顺带**让所有"还借着这个 mm"的 CPU 放掉引用（piggyback 在 flush IPI 上）。把高频原子操作换成低频 IPI，消除 cache-line bouncing。
- 更早还有 Rik van Riel 的 "make lazy TLB mode even lazier"（[lore 邮件](https://lore.kernel.org/lkml/20160831122710.5285c4ca@annuminas.surriel.com/), 2016）和 Andy Lutomirski 的 "Make lazy TLB mode lazier"（[lkml.iu.edu 存档](https://lkml.iu.edu/hypermail/linux/kernel/1810.1/02308.html)），把"何时真的要给 lazy CPU 发 shootdown"收窄到"只有页表页被 unmap 时"。

这条演进线索解释了 v6.12 源码里为什么 `context_switch()` 有 `mmgrab_lazy_tlb` / `mmdrop_lazy_tlb_sched` 这一对、为什么注释里反复强调 membarrier 屏障——它们是十年可扩展性辩论的沉淀，不是凭空写的。

### 6.2.4 PCID/ASID：把"换页表必刷 TLB"撕开一个口子

x86 早期换 `CR3` 必定全刷 TLB（除全局页）。Intel 引入 **PCID（Process-Context ID，12 bit）** 给 TLB 条目打标签，换页表时旧地址空间的条目可带着自己的 PCID 存活、不被刷。**KPTI（Meltdown 缓解，2018, ~kernel 4.14）让 PCID 从"可选优化"变成"刚需"**：KPTI 让内核态/用户态各用一套页表，每次系统调用要换两次 `CR3`；没有 PCID 时这两次换都全刷 TLB，开销 ~30%；有 PCID 时内核/用户各占一个 PCID 标签、TLB 都留住，开销降到 ~5–10%。所以 Linux 4.14 **同时**打开 KPTI 和 PCID（见 6.3.3 的真实源码注释）。

> **史料小结**：本章三大"换地址空间"机制——lazy TLB（省掉切 kernel thread 的页表）、PCID/ASID（省掉切回进程的全刷）、KPTI（被迫多换一次但用 PCID 补偿）——都是**为了少刷 TLB**这同一个目标在不同年代的产物。读 `switch_mm` 一带的代码若不知道这三条线，会觉得到处是莫名其妙的特判。

### 6.2.5 进程 vs 线程切换：把老说法修订到位

"线程切换比进程切换快"在现代多核 NUMA 上需要修订：

- **同进程线程切换**：共享同一 `mm_struct`（同 `CR3`），**走 `context_switch()` 里 `next->mm == prev->mm` 的路径，不调 `switch_mm`、不写 `CR3`、不刷 TLB**——这才是"线程快"的真正机制层根因。
- **跨进程切换**：换 `CR3`。无 PCID 全刷 TLB；有 PCID 靠 6 槽 ASID 环部分保留（6.3.3）。仍比线程切换贵。
- **跨 NUMA node 切换**：内存访问延迟从 ~70ns 涨到 ~140ns，是隐藏大头（6.4.4）。

## 6.3 现状：上下文切换做了什么（源码级精读）

### 6.3.1 总入口：`context_switch()` 真实源码逐行注解

先看调用栈骨架：

```
schedule() → __schedule() → pick_next_task() 选出 next
                          → context_switch(rq, prev, next, rf)
                                ├─ prepare_task_switch()        // 通知/统计/锁准备
                                ├─ arch_start_context_switch()  // paravirt hook
                                ├─ [换 mm 分支：lazy TLB / membarrier / switch_mm_irqs_off]
                                ├─ switch_mm_cid()              // RSEQ concurrency id
                                ├─ switch_to(prev, next, prev)  // ★ 换寄存器+栈，唯一"真切换"
                                └─ finish_task_switch(prev)     // 切回来后清理
```

下面是 `context_switch()` 的**逐字真实源码**与逐行注解。请特别注意：**"换寄存器"只有第 5328 行一行，其余全是"换地址空间 + 维护内存序/引用计数"。**

```c
/*
 * context_switch - switch to the new MM and the new thread's register state.
 */
static __always_inline struct rq *
context_switch(struct rq *rq, struct task_struct *prev,
	       struct task_struct *next, struct rq_flags *rf)
{
	prepare_task_switch(rq, prev, next);

	/*
	 * For paravirt, this is coupled with an exit in switch_to to
	 * combine the page table reload and the switch backend into
	 * one hypercall.
	 */
	arch_start_context_switch(prev);

	/*
	 * kernel -> kernel   lazy + transfer active
	 *   user -> kernel   lazy + mmgrab_lazy_tlb() active
	 *
	 * kernel ->   user   switch + mmdrop_lazy_tlb() active
	 *   user ->   user   switch
	 *
	 * switch_mm_cid() needs to be updated if the barriers provided
	 * by context_switch() are modified.
	 */
	if (!next->mm) {                                // to kernel
		enter_lazy_tlb(prev->active_mm, next);

		next->active_mm = prev->active_mm;
		if (prev->mm)                           // from user
			mmgrab_lazy_tlb(prev->active_mm);
		else
			prev->active_mm = NULL;
	} else {                                        // to user
		membarrier_switch_mm(rq, prev->active_mm, next->mm);
		/*
		 * sys_membarrier() requires an smp_mb() between setting
		 * rq->curr / membarrier_switch_mm() and returning to userspace.
		 *
		 * The below provides this either through switch_mm(), or in
		 * case 'prev->active_mm == next->mm' through
		 * finish_task_switch()'s mmdrop().
		 */
		switch_mm_irqs_off(prev->active_mm, next->mm, next);
		lru_gen_use_mm(next->mm);

		if (!prev->mm) {                        // from kernel
			/* will mmdrop_lazy_tlb() in finish_task_switch(). */
			rq->prev_mm = prev->active_mm;
			prev->active_mm = NULL;
		}
	}

	/* switch_mm_cid() requires the memory barriers above. */
	switch_mm_cid(rq, prev, next);

	prepare_lock_switch(rq, next, rf);

	/* Here we just switch the register state and the stack. */
	switch_to(prev, next, prev);
	barrier();

	return finish_task_switch(prev);
}
```
【真实源码 v6.12 kernel/sched/core.c:5269–5332】

**逐行/逐块注解**：

- `prepare_task_switch(rq, prev, next)`：触发 sched-out preempt notifiers、`perf_event_task_sched_out`、`rseq_preempt`，并为后续锁切换做准备。它与 `finish_task_switch` 严格配对（注释明说 "must be paired"）。
- `arch_start_context_switch(prev)`：**paravirt hook**。在 Xen/KVM-PV 下，把"页表 reload + switch backend"合并成一个 hypercall（注释第一段就在解释这个耦合）。裸机上是空操作。
- **四象限表（注释里那块 ASCII）是整个函数的灵魂**，对应 lazy TLB 设计：
  - `kernel -> kernel`：lazy，`active_mm` 直接转移，不动页表。
  - `user -> kernel`（`!next->mm` 且 `prev->mm`）：lazy，借用 prev 的 `active_mm` 并 `mmgrab_lazy_tlb()` 增引用——**这正是 6.2.3 那场可扩展性辩论的代码现场**。
  - `kernel -> user`（`next->mm` 且 `!prev->mm`）：要真正 `switch_mm`，并把要 drop 的旧 active_mm 暂存到 `rq->prev_mm`，延到 `finish_task_switch()` 里 `mmdrop_lazy_tlb()`（**为什么延后？因为在持 rq->lock 时 mmdrop 可能触发释放路径而死锁**——注释别处明说）。
  - `user -> user`：直接 `switch_mm`。
- `membarrier_switch_mm(...)` + 那段 `sys_membarrier()` 注释：**这是本函数最微妙的部分**。`sys_membarrier()` 系统调用承诺"全系统每个 CPU 在返回用户态前都执行一次完整内存屏障"。`switch_mm_irqs_off()` 内部写 `CR3` 本身就是一个 full barrier，所以正常换 mm 天然满足；但若 `prev->active_mm == next->mm`（换回同一个 mm，没真换页表），那个屏障要靠 `finish_task_switch()` 里的 `mmdrop()` 补。**把这段读懂，是区分"知道有 context_switch"和"懂内核内存序"的分水岭。**
- `switch_mm_irqs_off(prev->active_mm, next->mm, next)`：x86 上最终写 `CR3`、按 PCID/ASID 决定刷不刷 TLB（实现在 `arch/x86/mm/tlb.c`，6.3.3 读它）。
- `switch_mm_cid(rq, prev, next)`：维护 RSEQ 的 per-mm concurrency id（较新机制），注释两次强调它依赖上面的屏障。
- `switch_to(prev, next, prev)`：**唯一真正"换 CPU 寄存器和栈"的一行**，展开成 `__switch_to_asm`（6.3.2）。第三个参数 `prev` 是 `last` 出参——见下文"三参数之谜"。
- `barrier()`：编译器屏障，防止编译器把切换前后的访存重排——因为切换后**栈已经被换掉**，局部变量是从"上次本任务 `schedule()` 时保存的栈"恢复的。
- `return finish_task_switch(prev)`：切回来后的清理（6.3.4）。

> **`switch_to(prev, next, last)` 的三参数之谜**（高频面试题）：为什么要传第三个 `last`？因为切换是"单向跳走、将来从别处跳回"。当 CPU 沿 next 的栈跳走后，**很久以后某个任务又切回 prev 时，prev 在自己栈上"醒来"，但它需要知道"我是从谁那里被切回来的"**（那个 prev 可能已经迁移到别的 CPU、甚至已经死了要 drop 引用）。`__switch_to_asm` 返回的就是这个"真正的前一个任务"，赋给 `last`。源码里调用写成 `switch_to(prev, next, prev)`——第三个 `prev` 是个左值出参，会被汇编返回值覆写。

### 6.3.2 `switch_to` 真实汇编与 C 两段式

`switch_to` 宏本身极简：

```c
struct inactive_task_frame {
#ifdef CONFIG_X86_64
	unsigned long r15;
	unsigned long r14;
	unsigned long r13;
	unsigned long r12;
#else
	unsigned long flags;
	unsigned long si;
	unsigned long di;
#endif
	unsigned long bx;

	/*
	 * These two fields must be together.  They form a stack frame header,
	 * needed by get_frame_pointer().
	 */
	unsigned long bp;
	unsigned long ret_addr;
};

/* ... */

#define switch_to(prev, next, last)					\
do {									\
	((last) = __switch_to_asm((prev), (next)));			\
} while (0)
```
【真实源码 v6.12 arch/x86/include/asm/switch_to.h:23–52】

注意 **x86-64 的 `inactive_task_frame` 是 `r15/r14/r13/r12` 再加 `bx/bp/ret_addr`，没有 `flags`**（`flags` 只在 x86-32 分支）。这个结构体的字段顺序**必须**和下面汇编 push 的顺序逐一对应——这是汇编与 C 之间的硬契约。

干活的是 `__switch_to_asm`：

```asm
/*
 * %rdi: prev task
 * %rsi: next task
 */
.pushsection .text, "ax"
SYM_FUNC_START(__switch_to_asm)
	/*
	 * Save callee-saved registers
	 * This must match the order in inactive_task_frame
	 */
	pushq	%rbp
	pushq	%rbx
	pushq	%r12
	pushq	%r13
	pushq	%r14
	pushq	%r15

	/* switch stack */
	movq	%rsp, TASK_threadsp(%rdi)
	movq	TASK_threadsp(%rsi), %rsp

#ifdef CONFIG_STACKPROTECTOR
	movq	TASK_stack_canary(%rsi), %rbx
	movq	%rbx, PER_CPU_VAR(fixed_percpu_data + FIXED_stack_canary)
#endif

	/*
	 * When switching from a shallower to a deeper call stack
	 * the RSB may either underflow or use entries populated
	 * with userspace addresses. On CPUs where those concerns
	 * exist, overwrite the RSB with entries which capture
	 * speculative execution to prevent attack.
	 */
	FILL_RETURN_BUFFER %r12, RSB_CLEAR_LOOPS, X86_FEATURE_RSB_CTXSW

	/* restore callee-saved registers */
	popq	%r15
	popq	%r14
	popq	%r13
	popq	%r12
	popq	%rbx
	popq	%rbp

	jmp	__switch_to
SYM_FUNC_END(__switch_to_asm)
.popsection
```
【真实源码 v6.12 arch/x86/entry/entry_64.S（`__switch_to_asm`）】

**逐行精读**——这段汇编虽短，但每一行都对应一个"为什么"：

1. `pushq %rbp ... %r15`：保存 **callee-saved 寄存器**（x86-64 ABI 规定 `rbx,rbp,r12-r15` 由被调用者保存）。caller-saved（`rax,rcx,rdx,rsi,rdi,r8-r11`）**不用存**——因为切换发生在 `schedule()` 这个普通 C 调用里，调用约定保证调用者已自行处理 caller-saved。**这是"软件切换比 TSS 快"的根因：只存内核真正承诺保留的十几个寄存器，不存全套。**
2. `movq %rsp, TASK_threadsp(%rdi)` / `movq TASK_threadsp(%rsi), %rsp`：**整个切换的物理核心就这两条**——把当前栈指针存进 prev 的 `task->thread.sp`，再把 next 的 `thread.sp` 装回 `rsp`。**栈一换，"执行流"就换了**：因为函数的局部变量、返回地址全在栈上，换了栈等于换了整个调用链历史。
3. `STACKPROTECTOR` 块：把 next 的 stack canary 装入 per-cpu 的 `fixed_percpu_data`——**栈保护的 canary 是 per-task 的**，切任务必须换，否则栈溢出检测会用错值。
4. `FILL_RETURN_BUFFER`：**投机执行安全**。RSB（Return Stack Buffer，CPU 预测 `ret` 目标的硬件栈）在"从浅调用栈切到深调用栈"时可能 underflow 或残留用户态地址，被 Spectre-类攻击利用。这里用一串 call/ret 把 RSB 填满受控条目。**这一行是 2018 年后投机漏洞的产物，老内核没有**——又一个"源码里看似冗余、实则史料"的例子。
5. `popq` 逆序恢复 next 的 callee-saved 寄存器（next 当年也是这样被存进去的）。
6. `jmp __switch_to`：**尾调用**到 C 函数处理"汇编不方便做的剩余状态"。注意是 `jmp` 不是 `call`——`__switch_to` 返回时直接返回到 `__switch_to_asm` 的调用者（即 `context_switch`），`prev` 指针通过 `%rax` 透传，正好做 `last` 出参。

C 侧的 `__switch_to` 处理段寄存器、TLS、FS/GS base、PKRU、FPU：

```c
__visible __notrace_funcgraph struct task_struct *
__switch_to(struct task_struct *prev_p, struct task_struct *next_p)
{
	struct thread_struct *prev = &prev_p->thread;
	struct thread_struct *next = &next_p->thread;
	int cpu = smp_processor_id();

	WARN_ON_ONCE(IS_ENABLED(CONFIG_DEBUG_ENTRY) &&
		     this_cpu_read(pcpu_hot.hardirq_stack_inuse));

	if (!test_tsk_thread_flag(prev_p, TIF_NEED_FPU_LOAD))
		switch_fpu_prepare(prev_p, cpu);

	/* We must save %fs and %gs before load_TLS() because
	 * %fs and %gs may be cleared by load_TLS().
	 *
	 * (e.g. xen_load_tls())
	 */
	save_fsgs(prev_p);

	/*
	 * Load TLS before restoring any segments so that segment loads
	 * reference the correct GDT entries.
	 */
	load_TLS(next, cpu);

	/* ...DS/ES segment switch... */

	x86_fsgsbase_load(prev, next);
	x86_pkru_load(prev, next);

	raw_cpu_write(pcpu_hot.current_task, next_p);
	raw_cpu_write(pcpu_hot.top_of_stack, task_top_of_stack(next_p));

	switch_fpu_finish(next_p);
	/* ... */
}
```
【真实源码 v6.12 arch/x86/kernel/process_64.c:608–674（节选，省略号处为 DS/ES 段切换与 IO bitmap）】

**关键点**：

- `switch_fpu_prepare` / `switch_fpu_finish`：**这就是 eager FPU 的落地**——直接在切换路径里保存/恢复 FPU 状态（用 `XSAVE`/`XRSTOR` 按 XCR0 掩码），不再依赖 `#NM`。`TIF_NEED_FPU_LOAD` 优化掉了"刚保存又要恢复"的冗余。
- `save_fsgs` / `load_TLS` / `x86_fsgsbase_load`：FS/GS base 是用户态 TLS 的根（glibc 用 `%fs` 寻 TLS），必须 per-task 切。注释解释了顺序约束（先存 fs/gs 再 `load_TLS`，因为 `load_TLS` 可能清掉它们）。
- `x86_pkru_load`：切换 **PKRU（Protection Key Rights）** 寄存器，per-task 的内存保护键权限。
- `raw_cpu_write(pcpu_hot.current_task, next_p)`：更新 per-cpu 的 `current` 指针——**`current` 宏背后就是读这个 per-cpu 变量**，这一行之后 `current` 才指向 next。

> **两段式（asm + C）为什么不全用汇编或全用 C？** 全汇编：换栈这种"脚下抽地毯"的操作必须精确控制寄存器/栈，C 编译器做不到（局部变量在栈上、栈正在被换）。全 C：段寄存器/TLS/FPU 这些逻辑复杂、要调一堆 helper，写成汇编不可维护。所以**把"换栈"这唯一不能用 C 的部分留在 `__switch_to_asm`，其余甩给 C 的 `__switch_to`**——这是工程上"危险区最小化"的经典切法。

### 6.3.3 换地址空间：lazy TLB、PCID/ASID 的真实源码

`switch_mm_irqs_off` 最终落到 `arch/x86/mm/tlb.c`。这个文件顶部的注释把 Linux 的 PCID 哲学讲得很清楚：

> "The x86 feature is called PCID (Process Context IDentifier). It is similar to what is traditionally called ASID on the RISC processors. **We don't use the traditional ASID implementation... Instead we have a small per-cpu array of ASIDs and cache the last few mm's that came by on this CPU**, allowing cheaper switch_mm between processes on this CPU."
> 【真实源码 v6.12 arch/x86/mm/tlb.c 顶部注释（节选）】

而"为什么是 6 个"，答案在头文件里，**注释本身就是最好的设计解释**：

```c
/*
 * 6 because 6 should be plenty and struct tlb_state will fit in two cache
 * lines.
 */
#define TLB_NR_DYN_ASIDS	6
```
【真实源码 v6.12 arch/x86/include/asm/tlbflush.h】

逐点拆解这个"6"的含义，纠正常见误解：

- 硬件 PCID 是 **12 bit = 4096 个**，但 **Linux 每 CPU 只维护一个 6 槽的动态 ASID 环**（`TLB_NR_DYN_ASIDS`），轮换复用。
- "6 should be plenty"：在一个 CPU 上"最近活跃"的进程通常远少于 6 个；够用就行，**槽位多了反而让 `struct tlb_state` 撑出 cache line**——注释直说"will fit in two cache lines"，这是**为了 `tlb_state` 结构体的 cache 友好性**而压低槽数的微架构级权衡。
- 真实世界"何时被迫刷 TLB"由这个 6 槽 **LRU** 决定：进程被踢出环 → 它的 TLB 标签作废 → 下次回来仍要重建 TLB。**所以"有 PCID 就不刷 TLB"是错的**，只是"最近 6 个进程之间来回切不刷"。
- **KPTI 把可用槽再砍**：开了 PTI 后内核态/用户态各占一个 PCID。源码里 `kern_pcid` 的注释解释了这个 ASID→PCID 的映射规则：
  > "If PCID is on, ASID-aware code paths put the ASID+1 into the PCID bits. This serves two purposes. It prevents a nasty situation in which PCID-unaware code saves CR3, loads some other value (with PCID == 0), and then restores CR3, thus corrupting the TLB for ASID 0."
  > 【真实源码 v6.12 arch/x86/mm/tlb.c `kern_pcid()` 注释（节选）】
  （ASID+1 偏移是为了把 PCID==0 留给"PCID-unaware 代码"，避免它误污染 ASID 0 的 TLB——又一个微妙的兼容性设计。）

### 6.3.4 切回来之后：`finish_task_switch()` 与"脚下换栈"的奇妙

`finish_task_switch()` 在新任务的栈上执行，但它清理的是 **prev**。它的注释把"换栈"的诡异讲得很到位：

```c
/*
 * finish_task_switch - clean up after a task-switch
 * @prev: the thread we just switched away from.
 * ...
 * The context switch have flipped the stack from under us and restored the
 * local variables which were saved when this task called schedule() in the
 * past. 'prev == current' is still correct but we need to recalculate this_rq
 * because prev may have moved to another CPU.
 */
static struct rq *finish_task_switch(struct task_struct *prev)
	__releases(rq->lock)
{
	struct rq *rq = this_rq();
	struct mm_struct *mm = rq->prev_mm;
	unsigned int prev_state;
	/* ... preempt_count 校验 ... */
	rq->prev_mm = NULL;
	/* ... */
	prev_state = READ_ONCE(prev->__state);
	vtime_task_switch(prev);
	perf_event_task_sched_in(prev, current);
	finish_task(prev);
	tick_nohz_task_switch();
	finish_lock_switch(rq);
	/* ... */
	if (mm) {
		membarrier_mm_sync_core_before_usermode(mm);
		mmdrop_lazy_tlb_sched(mm);
	}
	if (unlikely(prev_state == TASK_DEAD)) {
		/* ... put_task_struct，回收死亡任务 ... */
	}
	/* ... */
}
```
【真实源码 v6.12 kernel/sched/core.c:5141–5230+（节选）】

**逐点**：

- `struct rq *rq = this_rq();`：**为什么要重新取 `this_rq()`？** 因为这段代码是在 **next 的栈上**跑的，而 prev（此刻 `prev == current`，但物理上我们已经在 next 的执行流里）可能已经被迁移到别的 CPU。注释原话 "prev may have moved to another CPU"。
- `if (mm) { ... mmdrop_lazy_tlb_sched(mm); }`：**这就是 6.3.1 里 `kernel -> user` 分支暂存到 `rq->prev_mm` 的那个旧 active_mm，延到这里、脱离 rq->lock 之后才 drop**——印证了"持锁 mmdrop 会死锁"的设计。同时 `mmdrop_lazy_tlb` 隐含一个 full barrier，正好补上 membarrier 需要的那个屏障（6.3.1 注释里说的"case 'prev->active_mm == next->mm' through finish_task_switch()'s mmdrop()"）。
- `if (prev_state == TASK_DEAD)`：如果 prev 是"最后一次 `schedule()` 后再不回来"的死亡任务，在这里 `put_task_struct` 真正回收它——**一个进程的"葬礼"是在切到它的继任者的栈上办的**。

### 6.3.5 voluntary vs involuntary：内核真实判据（精确版）

原版讲"主动 sleep vs 时间片耗尽"是对的方向，但**内核的实际判据更细**。`__schedule()` 里：

```c
static void __sched notrace __schedule(int sched_mode)
{
	struct task_struct *prev, *next;
	bool preempt = sched_mode > SM_NONE;
	bool block = false;
	unsigned long *switch_count;
	/* ... */
	switch_count = &prev->nivcsw;               // 默认：算非自愿

	preempt = sched_mode == SM_PREEMPT;
	prev_state = READ_ONCE(prev->__state);
	if (sched_mode == SM_IDLE) {
		/* ... */
	} else if (!preempt && prev_state) {        // 非抢占 且 任务设了非RUNNING态
		if (signal_pending_state(prev_state, prev)) {
			WRITE_ONCE(prev->__state, TASK_RUNNING);   // 有信号 → 不真睡，转回RUNNING
		} else {
			/* ... block_task(rq, prev, flags) 把 prev 摘出 runqueue ... */
			block = true;
		}
		switch_count = &prev->nvcsw;        // ★ 只有走到这里才算自愿
	}
	/* ... */
	++*switch_count;                            // 在真正切换处累加
	/* ... */
	rq = context_switch(rq, prev, next, &rf);
}
```
【真实源码 v6.12 kernel/sched/core.c:6548–6693（节选拼接，省略号处为 RCU/屏障/clock 更新）】

**这段代码纠正了三个常见误解**：

1. **默认是 involuntary**：`switch_count` 一开始指向 `nivcsw`。只有满足"非抢占（`!preempt`）+ 任务自己设了非 RUNNING 态（`prev_state` 非零）+ 不是因信号被唤醒"时，才改指向 `nvcsw`。
2. **"设了睡眠态但被信号打断"不算自愿**：`signal_pending_state` 为真时，任务被改回 `TASK_RUNNING`、**`switch_count` 仍是 `nivcsw`**——它没真睡成，只是要去处理信号。
3. **被抢占的任务永远算 involuntary**：`preempt` 为真直接跳过整个自愿分支。所以一个任务即便"正打算睡"，只要在 `schedule()` 前被更高优先级抢占，也计 involuntary。

这两个计数器暴露在 `/proc/<pid>/status` 的 `voluntary_ctxt_switches` / `nonvoluntary_ctxt_switches`，对应 `task_struct` 字段：

```c
/* Context switch counts: */
unsigned long			nvcsw;
unsigned long			nivcsw;
```
【真实源码 v6.12 include/linux/sched.h（`struct task_struct` 节选）】

**工程推论**（保留并强化）：

1. `nivcsw` 持续走高 → CPU 过饱和或被高优先级任务频繁抢占，加核 / 减线程 / 查是否有 RT 任务捣乱。
2. `nvcsw` 极高但吞吐低 → 大量线程在 futex/mutex 上阻塞（futex 等待走主动 `schedule()` + `block_task`，计 voluntary），考虑 lock-free / sharding / 减锁粒度。
3. 两者都低但延迟高 → 不是切换问题，看 NUMA miss / 网络 / 锁的"非阻塞自旋"（自旋不进 `schedule()`，不计任何 ctxsw，但烧 CPU）。

```bash
grep -E 'voluntary|nonvoluntary' /proc/<pid>/status   # 单进程
vmstat 1                                               # cs 列：全系统切换率
pidstat -w -p <pid> 1                                  # cswch/s + nvcswch/s
```

### 6.3.6 开销量级（x86-64 参考，依赖具体硬件）

| 组成项 | 量级 | 说明（对应源码/机制） |
|--------|------|------|
| 裸 `__switch_to_asm` 寄存器拷贝 | ~几十 ns | 6 个 callee-saved push/pop + 换 `rsp`，本身极便宜（6.3.2） |
| `__switch_to` C 侧（段/TLS/FPU/PKRU） | ~几十~上百 ns | eager FPU `XSAVE`/`XRSTOR` 是这里的主要变量，状态越大越贵 |
| `__schedule()` 整条软件路径 | ~数百 ns–1 μs | 含 `pick_next_task`、统计、屏障；同进程线程间、无 TLB 影响时"直接开销"落这里 |
| 跨进程切换（PCID 命中 6 槽环） | ~0.5–1 μs | `switch_mm` 写 CR3，TLB 不全刷但仍部分失效 |
| 跨进程切换（PCID 槽位被挤掉 / 无 PCID） | ~1–5 μs | TLB 全刷，下一段代码 cold start |
| L1/L2/LLC cache miss 积累（"尾效应"） | 数 μs–数十 μs | 取决于工作集，**常是真正大头** |
| 跨 NUMA node 后的内存访问惩罚 | 每次 remote miss 比 local 多 ~50–100ns | 高频访问累积成毫秒级 |
| FPU 状态大小推手 | AVX-512 ~2.5KB+；**AMX TILE 8192B（state component 18）+ TILECFG 64B（17）= 8256B** | 状态越大 `XSAVE`/`XRSTOR` 越慢；见 6.3.7 |

> **辨析**：很多人把"整条 `schedule()` 路径"（数百 ns）当成"裸 `switch_to`"（数十 ns），又把 lmbench `lat_ctx` 报的"进程间 ctxsw 1–5 μs"（含 pipe + syscall 往返）当成裸切换。**三个量级是三回事**，6.10 实验一会用 `taskset` 把它们拆开。

> 💡 **坑点**：`perf stat -e context-switches` 或 `/proc/<pid>/status` 量到的是**次数不是代价**。代价要用 `perf stat -e cache-misses,dTLB-load-misses,cs` 组合，或看 IPC 塌陷（6.12 系统题会用到）。

### 6.3.7 FPU/AMX 巨型状态：lazy 思想在"分配"层的回归

eager FPU 解决了"何时 save/restore"，但 **AMX（Advanced Matrix Extensions）的 TILE 寄存器一把 8KB**，若给每个 task 都预留 8KB context-switch 缓冲，绝大多数不用 AMX 的进程纯浪费内存。内核的解法（[kernel.org xstate 文档](https://docs.kernel.org/arch/x86/xstate.html) + [LWN.net/Articles/867410](https://lwn.net/Articles/867410/)）：

> "Features like AMX TILE_DATA (XSTATE component 18) are enabled by XCR0 as well, but **the first use of related instruction is trapped by the kernel because by default the required large XSTATE buffers are not allocated automatically.**"
> 【kernel.org Documentation/arch/x86/xstate.rst（节选）】

机制：用户态先 `arch_prctl(ARCH_REQ_XCOMP_PERM, XFEATURE_XTILEDATA)` 申请权限；内核用 **XFD（eXtended Feature Disable）** 武装该 feature，任务**首次执行 AMX 指令时触发 `#NM`**，异常处理里**才**分配那块大缓冲并切换进去。

> ⚠️ **别混淆两种 lazy**：这是 **lazy *分配***（buffer 按需分配），不是已废弃的 **lazy *save/restore***（用 `#NM` 决定何时存寄存器）。前者是 2021+ 为 AMX 巨型状态引入的内存优化、健在；后者是 2016 被砍的性能/安全反模式。同样用 `#NM`，目的与年代完全不同——这正是"用同一个硬件机制做截然不同的事"的考古乐趣。

## 6.4 CPU 亲和性（Affinity）

### 6.4.1 亲和性的本质与内核表示

CPU 亲和性是告诉调度器"这个任务只能跑在这几个核上"。v6.12 的 `task_struct` 用一组字段表达：

```c
int				nr_cpus_allowed;
const cpumask_t			*cpus_ptr;
cpumask_t			*user_cpus_ptr;
cpumask_t			cpus_mask;
```
【真实源码 v6.12 include/linux/sched.h（`struct task_struct` 节选）】

字段语义（纠正"只有一个 mask"的旧认知）：

- `cpus_mask`：用户通过 `sched_setaffinity` 设的"允许集"。**历史上叫 `cpus_allowed`，约 kernel 5.3 改名 `cpus_mask`**——写第三方内核模块常见 `#if LINUX_VERSION_CODE >= KERNEL_VERSION(5,3,0)` 兼容判断。
- `cpus_ptr`（`const cpumask_t *`）：**调度器实际读的指针**，平时指向 `&cpus_mask`；但 `migrate_disable()` 等机制会临时把它指向一个单核 mask，实现"短暂强制绑核"而不动用户设置。**这是"用户设的 affinity"和"内核此刻实际允许的 CPU"解耦的设计**——`cpus_mask` 是用户意图，`cpus_ptr` 是当前生效。
- `user_cpus_ptr`：保存用户原始请求，配合 cpuset/`sched_setaffinity` 的"effective affinity = 用户请求 ∩ cpuset 允许"语义（见 6.4.2）。
- `nr_cpus_allowed`：`cpus_ptr` 里置位的核数，热路径缓存避免每次 `cpumask_weight`。

亲和性解决**局部性**：把线程钉在同一核，cache/TLB 状态能跨时间片保留（同核被抢占后再调回来，热度还在），而不必每次 cold start。

### 6.4.2 用户态接口与"silently imposed"语义

```c
#include <sched.h>
cpu_set_t mask;
CPU_ZERO(&mask);
CPU_SET(2, &mask);
CPU_SET(3, &mask);
sched_setaffinity(pid, sizeof(mask), &mask);   // 绑到核 2,3
sched_getaffinity(pid, sizeof(mask), &mask);    // 查询
```

```bash
taskset -c 2,3 ./my_program        # 启动时指定
taskset -pc 2,3 <pid>              # 对已有进程修改
taskset -p <pid>                   # 查当前 mask（16 进制位图）
numactl --cpunodebind=0 --membind=0 ./my_program   # 同时绑 CPU 和内存节点
```

**man 手册里两条容易被忽略、却是排障关键的语义**（[man7 sched_setaffinity(2)](https://man7.org/linux/man-pages/man2/sched_setaffinity.2.html)）：

1. **effective affinity 是交集，且 cpuset 的限制"静默施加"**：
   > "the set of CPUs on which the thread will actually run is the intersection of the set specified in the mask argument and the set of CPUs actually present on the system. The system may further restrict the set ... if the 'cpuset' mechanism ... is being used. **These restrictions ... are silently imposed by the kernel.**"
   
   **生产暗坑**：你 `taskset -c 4-7` 绑了核 4-7，但进程在 cgroup（K8s `cpuset.cpus`）里被限制到 0-3，**`sched_setaffinity` 不报错**，进程实际跑在 0-3。排查时 `taskset -p` 看到的可能是你设的值，而实际生效要看 cpuset。容器环境里"绑核没生效"十有八九是这个交集在作怪。

2. **affinity 跨 fork/exec 继承**：
   > "A child created via fork(2) inherits its parent's CPU affinity mask. The affinity mask is preserved across an execve(2)."
   
   含义：父进程绑了核，`fork` 出的子进程默认也绑——`taskset -c 0 ./launcher` 启动的整棵进程树都会挤在核 0，这是"一绑全家"的常见误用。

`EINVAL`（mask 里没有任何"物理存在且 cpuset 允许"的核）和 `EPERM`（无 `CAP_SYS_NICE` 且 uid 不匹配）是两个高频报错。

### 6.4.3 isolcpus —— 真正的隔离，且它已"半弃用"

`taskset` 只是"优先/限制选这些核"，**通用任务和内核线程仍可能被放上去**。真正把核从调度器候选集移除要 `isolcpus` 启动参数：

```
# /etc/default/grub
GRUB_CMDLINE_LINUX="isolcpus=4-7 nohz_full=4-7 rcu_nocbs=4-7"
```

- `isolcpus=4-7`：把核 4-7 从通用调度器 load balancer 移除，只有显式绑过来的任务才跑。
- `nohz_full=4-7`：关这些核的调度时钟中断（tickless），消除每秒 100/250/1000 次的 timer 抖动。
- `rcu_nocbs=4-7`：把 RCU callback 迁出这些核（RCU 软中断会干扰实时任务）。

> 🕒 **时效/真坑：`isolcpus` 已被官方视为半弃用**（核实自 [codeblueprint 分析](https://www.codeblueprint.co.uk/2019/10/08/isolcpus-is-deprecated-kinda.html) 与内核 cpuset 文档）。两个硬伤：
> 1. **静态**——只能 boot 时定，**运行时不能加减隔离核，只能重启**。
> 2. 早期它**禁掉了被隔离核的 load balancer**，行为容易让人误判。
>
> 现代姿势是 **cpuset / cgroup v2 的 `isolated` partition**（`cpuset.cpus.partition` 写 `isolated`），可运行时动态划分隔离域、可回收。HFT/RT 老系统仍大量用 `isolcpus`（因为简单、确定），但新设计建议优先 cpuset。面试里能说出"isolcpus 已 deprecated-kinda、cgroup v2 isolated partition 是接班人"是加分。

> 💡 **坑点**：`isolcpus` 隔离的核如果没任务显式绑上去，就**完全空转**。别随便隔离大量核，先 `htop`/`mpstat -P ALL` 观察。

### 6.4.4 NUMA 亲和性

NUMA（Non-Uniform Memory Access，见第 8 章）机器上每个 socket 有本地内存控制器，跨 socket 访存走 QPI/UPI 互联，延迟高约 1.5–2x。

```
┌─────────────────────┬─────────────────────┐
│   Socket 0 (Node 0) │   Socket 1 (Node 1) │
│  ┌──────┐ ┌──────┐  │  ┌──────┐ ┌──────┐  │
│  │Core 0│ │Core 1│  │  │Core 4│ │Core 5│  │
│  └──────┘ └──────┘  │  └──────┘ └──────┘  │
│  ┌──────────────┐   │  ┌──────────────┐   │
│  │  Local DRAM  │◄──┼──│  Remote DRAM │   │
│  │   ~70 ns     │   │  │   ~140 ns    │   │
│  └──────────────┘   │  └──────────────┘   │
└─────────────────────┴─────────────────────┘
            QPI/UPI 互联
```

| 场景 | 推荐策略 |
|------|---------|
| 单进程高吞吐（如 Redis） | `numactl --cpunodebind=0 --membind=0`，进程与内存绑同一 node |
| 数据库（内存跨多 node） | `--interleave=all` 均匀分摊，避免单 node 内存控制器成瓶颈 |
| HPC 多线程 | 每个 NUMA node 一个进程组，进程内用 OpenMP 绑核 |
| 微服务容器 | K8s `topologySpreadConstraints` + CPU Manager `static` policy + Topology Manager |

```bash
numactl --hardware                       # 拓扑 + node 间距离矩阵
lscpu | grep -E 'NUMA|Socket|Core'
numastat -p <pid>                        # 进程级 NUMA 命中/miss
cat /sys/class/net/<iface>/device/numa_node   # NIC 在哪个 node
```

> 🎯 **面试加分点（带底层根因）**：Linux 有 **automatic NUMA balancing**（`/proc/sys/kernel/numa_balancing`，近年默认开），通过周期性 unmap 页面、在下次访问时触发 page fault 来"采样"哪个 node 在访问、再迁移任务/页到更近的 node。**但采样本身就是制造 minor fault + 迁移就是数据拷贝**——高频迁移会"NUMA thrashing"，把本该用于计算的带宽烧在搬页上。延迟敏感场景建议 `numa_balancing=0` + 手工绑定。**根因**：自动平衡是"以小步试错逼近最优"的启发式，对稳态负载是净负担，对动态负载才有价值。

### 6.4.5 cache 局部性与绑核决策

**正向收益**：① cache 热度跨时间片保留；② TLB 热度保留（同核 ASID 大概率仍在 6 槽环里）；③ 消除跨 NUMA 访存。

**负向代价**：① 负载不均（一核忙、邻核闲）；② 单点热核 thermal throttling 降频；③ 减少调度器灵活性，突发负载响应差。

**工程推论**：

1. 绑核是 "last resort optimization"——先把软件架构（减锁、减分配、改数据布局）做好再绑。
2. 绑核前 profile：`perf stat -e cache-misses,LLC-load-misses,dTLB-load-misses` 确认 cache/TLB 真是瓶颈。
3. 正确姿势是**绑到 NUMA-local 的核集合**，并把 NIC/磁盘中断也打到同 node。
4. SMT 核要成对处理（见 6.3.6 面试加分点的 SMT 辨析）。

## 6.5 实时调度与低延迟

### 6.5.1 调度类与延迟保证

| 调度类 | 策略 | 典型场景 | 延迟特征 |
|--------|------|---------|---------|
| `SCHED_DEADLINE` | EDF + CBS 带宽控制（**实为最高优先级，先于 RT 类**） | 视频编码、可声明截止时间 | 可声明 deadline，带宽受限不饿死系统 |
| `SCHED_FIFO` | 实时 FIFO，跑起来不让出（直到更高优先级或主动 yield） | 音频服务器、工业控制 | 最低，微秒级 |
| `SCHED_RR` | 实时轮转，有时间片 | 软实时 | 低 |
| `SCHED_OTHER`（CFS/EEVDF） | 普通进程 | 一切通用任务 | 默认，毫秒级 |
| `SCHED_IDLE` | 极低优先级 | 后台计算 | 不保证 |

> **纠一个常见错位**：很多资料把 `SCHED_FIFO` 列为"最高"。实际上 **`SCHED_DEADLINE` 在内核里优先级高于 FIFO/RR**（`dl_sched_class` 排在 `rt_sched_class` 之前）——DEADLINE 任务会抢占 FIFO 任务。

```bash
chrt -f 50 ./my_rtapp          # SCHED_FIFO, prio 50
chrt -r 50 ./my_rtapp          # SCHED_RR
# SCHED_DEADLINE：单位是【纳秒】，不接受 "1ms" 后缀
# runtime=1ms / deadline=5ms / period=10ms：
chrt -d --sched-runtime 1000000 --sched-deadline 5000000 \
        --sched-period 10000000 0 ./encoder
# 末尾的 0 是占位 priority（DEADLINE 不看静态优先级，必须填 0）
chrt -p <pid>                  # 查看
```

> 💡 **坑点**：`SCHED_FIFO` prio 99 进死循环会 **starve 整个系统**（含 watchdog）。保护手段：① `ulimit -r`（RT 优先级上限）；② `/proc/sys/kernel/sched_rt_runtime_us`（默认 950000，即每 1s RT 类最多占 0.95s，留 0.05s 给普通任务"喘气"，这是内核的 RT throttling 安全网）；③ 改用 `SCHED_DEADLINE` 的 CBS 带宽控制。

### 6.5.2 抢占模型与延迟

| 抢占配置 | 适用场景 | 典型最坏调度延迟 |
|---------|---------|------------|
| `PREEMPT_NONE` | 服务器，最高吞吐 | 数毫秒 |
| `PREEMPT_VOLUNTARY` | 桌面，均衡 | <1ms |
| `PREEMPT` | 低延迟桌面/通用，几乎全可抢占 | <100μs |
| `PREEMPT_RT` | 硬实时；自旋锁变可睡眠 mutex，关中断区压到最小 | 最坏延迟可控到几十 μs |
| `PREEMPT_DYNAMIC` | **运行时可切**前三档（`preempt=` 启动参数 / debugfs） | 取决于当前档位 |

```bash
grep PREEMPT /boot/config-$(uname -r)
# 或
zcat /proc/config.gz | grep PREEMPT
# PREEMPT_DYNAMIC 下可运行时查/改：
cat /sys/kernel/debug/sched/preempt
```

> 🕒 **时效（2026 视角 · 重要史实）**：`PREEMPT_RT` 在树外维护了**近二十年**（[LWN.net/Articles/989212](https://lwn.net/Articles/989212/) "The realtime preemption end game — for real this time"，2024-09；溯源约 **2004 年**起步）。它把"sleeping spinlocks（自旋锁改可睡眠 mutex）、threaded IRQ handlers（中断线程化）、高精度定时器"等一块块送进主线长达二十年。**最后一块拦路石是 `printk`**——RT 临界区里不能被 `printk` 阻塞，必须先把 console 输出改成不阻塞的 **nbcon** 架构。Sebastian Andrzej Siewior 把 printk 收口后，**Linux 6.12（2024-11）首次让 `PREEMPT_RT` 成为主线可配置选项**（先支持 x86_64 / arm64 / riscv），不再需要打 `-rt` patch。但**发行版默认内核通常仍是 `PREEMPT`/`PREEMPT_VOLUNTARY`，硬实时部署仍需自己编译开启**。面试里能说出"6.12 mainline / printk(nbcon) 是最后 blocker / 二十年 Gleixner-Rostedt-Siewior 一脉"是硬核加分。

## 6.6 中断亲和

中断亲和（IRQ affinity）与任务亲和**正交**——外设中断打到哪个核处理，直接影响局部性和延迟。详见第 18 章，这里只讲与上下文切换的交叉点。

### 6.6.1 中断对上下文切换的影响

每个中断打断当前任务，触发中断上下文的保存/恢复——一种"半切换"（不完整保存用户态寄存器，因为处理完还回原任务）。但：

- 被中断任务的 cache 被 handler 的工作集污染（**即使没发生任务切换，cache 也被搅了**）。
- 若 handler 触发 `wake_up`（如网络包到来唤醒等待线程），紧接着可能发生一次完整上下文切换——**中断是"隐形的切换放大器"**。

```bash
cat /proc/interrupts                       # 中断分布（按 CPU 列）
cat /proc/irq/<n>/smp_affinity             # 查 IRQ 亲和（位图）
echo f0 > /proc/irq/<n>/smp_affinity        # 把 IRQ n 移到核 4-7
cat /proc/irq/<n>/smp_affinity_list        # 列表形式更易读
```

### 6.6.2 生产最佳实践：核心隔离三件套（+ 中断）

```
┌─────────────────────────────────────────────────┐
│  核 0–3：系统核（OS、中断、后台、RCU callback）  │
│  核 4–7：业务核                                  │
│    isolcpus/cpuset isolated + nohz_full          │
│    + rcu_nocbs + 中断全部迁出 + RT 线程绑定        │
└─────────────────────────────────────────────────┘
```

1. `isolcpus=4-7`（或 cgroup v2 isolated partition）——调度器不主动放任务上业务核。
2. `irqaffinity=0-3`——所有 IRQ 默认只打系统核。
3. 业务线程 `taskset -c 4-7` + `SCHED_FIFO`/`SCHED_DEADLINE`。
4. `nohz_full=4-7`——关时钟中断（注意：**至少留一个 housekeeping 核不能 nohz_full**，否则没核记账）。
5. `rcu_nocbs=4-7`——迁出 RCU 回调。

**为什么"缺一不可"**：`isolcpus` 挡住普通任务但挡不住时钟中断和软中断；`nohz_full` 关掉每 1ms 的 timer 但挡不住 RCU 软中断；`rcu_nocbs` 迁走 RCU 但挡不住外设 IRQ。**四者正交，每个堵一类抖动源，全开才逼近"裸金属独占核"**。

## 6.7 为什么过多线程反而慢

### 6.7.1 三重叠加效应

```
线程数 >> 核数
       │
       ├─► 上下文切换频率 ↑
       │       └─► cache/TLB 污染 ↑ → 有效 IPC 下降
       │
       ├─► 锁竞争 ↑（更多线程在 mutex/futex 上等）
       │       └─► 自愿切换 ↑ → 更多切换 → 回到上面
       │
       └─► 内存带宽竞争 ↑（LLC thrashing + ASID 6 槽环被挤爆 → TLB 也 thrash）
               └─► 实际执行时间 ↓，等待时间 ↑
```

注意第三条比原版多了一层：**线程/进程数远超 6 时，PCID 6 槽 ASID 环被反复挤掉**，每个进程切回来都要重建 TLB——这是"线程多到一定程度，TLB 也开始 thrash"的机制层根因（6.3.3）。

### 6.7.2 量化：C10K 的真实代价

16 核机器跑 10000 线程（**以下数字是量级演示用的假设，换机器即变**）：

- 调度器每次切换 ~1μs，1000 次/秒/核 = 1ms/s = **0.1% CPU 用于切换**——看着很少？
- cache 冷启动：每次切换后 ~1000 个 cache miss × ~100ns = **100μs** = **直接切换开销的 ~100 倍**。
- 10000 线程共享 32MB LLC，每线程工作集 100KB → 总需求 ~1GB ≫ 32MB → 命中率趋 0，几乎每次访存打 DRAM。

> 它建立的直觉：**直接切换开销（零点几个百分点）几乎可忽略，把系统拖垮的是 cache/LLC/TLB cold start 这条尾巴。** "线程数 = 越多越并发"在 CPU-bound 下是错觉。

> 💡 **坑点**：最优线程数经验公式 `N_threads ≈ N_cpu × (1 + wait_ratio)`，`wait_ratio = 阻塞时间/计算时间`。I/O bound 可多几倍；CPU bound 应接近核数。

### 6.7.3 Go 协程 / 事件驱动的优势（精确化）

goroutine 和 Nginx/Node.js 的事件驱动高并发不是"更快"，而是：

1. **用户态调度**：goroutine 切换不进内核、不动 `CR3`、不刷 TLB、不走 `__schedule()`——省掉的正是 6.3 整套机制。
2. **工作集小**：goroutine 初始栈现为 **8KB**、可按需增长的连续栈；传统 pthread 默认**保留** 8MB 虚拟地址栈。**注意 8MB 是 `ulimit -s` 的地址空间预留，不是实际 commit 的 RSS**，但页表项和栈顶 cache line 仍是实打实开销。
3. **M:N 模型**：N 个 goroutine 映射到 M(≈CPU 数) 个 OS 线程；Go runtime 的 GMP 调度器在用户态做 work-stealing。

但这些优势在 **CPU-bound + 共享内存** 下消失，反而多一层 runtime 调度开销——**协程不是银弹，它优化的是"等待型并发"的 per-task 内存与切换成本，不是"计算型并行"的吞吐**。

## 6.8 落地：生产调优与排障

### 6.8.1 排障流程

```
延迟高 / 吞吐低
      │
      ├── vmstat 1 → cs 列异常高 → 上下文切换问题
      │       ├── /proc/<pid>/status → voluntary vs nonvoluntary
      │       │       ├── nonvoluntary 高 → CPU 饱和或被 RT 抢占，加核/减线程/查 chrt
      │       │       └── voluntary 高 → 锁竞争，profile mutex/futex
      │       └── perf stat -e cache-misses,LLC-load-misses → cache 代价
      │
      ├── perf sched record/latency → 调度延迟分布（avg/max/per-task）
      │
      └── numastat / perf stat -e dTLB-load-misses,node-load-misses → NUMA/TLB
                └── numactl 绑 node 测改善幅度
      │
      └── 注意排除"自旋不进 schedule"的烧 CPU：top 看 %sys 高但 cs 不高 → 自旋锁/忙等
```

### 6.8.2 关键命令速查

```bash
# 上下文切换统计
vmstat 1                          # cs 列：全系统切换/秒
pidstat -w -p <pid> 1             # 单进程 cswch/s + nvcswch/s
grep -E 'ctxt_switches' /proc/<pid>/status

# 调度延迟分析
perf sched record -a -- sleep 5
perf sched latency                # 最差/平均延迟 per-task
perf sched timehist | head        # 时间线

# CPU 亲和
taskset -p <pid>                  # 当前 mask（注意：cpuset 可能静默覆盖，见 6.4.2）
taskset -pc 0-3 <pid>
cat /proc/<pid>/status | grep Cpus_allowed_list   # 内核侧实际允许集

# NUMA
numactl --hardware ; numastat -p <pid>

# 中断
watch -n1 cat /proc/interrupts

# 实时调度
chrt -p <pid> ; chrt -f -p 50 <pid>

# cache / TLB miss + IPC（看 stall）
perf stat -e instructions,cycles,L1-dcache-load-misses,LLC-load-misses,dTLB-load-misses -p <pid> sleep 5
```

### 6.8.3 大厂实战案例

**案例 1：电商秒杀延迟抖动**
- 现象：压测时 P99 从 5ms 飙到 50ms。
- 排查：`perf sched latency` 见调度等待延迟高达 30ms；`pidstat -w` 见 `nvcswch/s` 极高。
- 根因：线程池被业务同学为"提高并发"撑到 500 线程，16 核机切换率 20K/s，且大量 futex 等待。
- 解法：线程池缩回 64，P99 恢复到 6ms。**过多线程是反直觉的加减法。**

**案例 2：ML 推理服务 NUMA 陷阱**
- 现象：同样的 GPU 推理，在双路 NUMA 上比单路慢 30%。
- 排查：`numastat -p` 见大量 `numa_miss`，进程跨 node。
- 根因：容器随机打到两 socket，内存 interleave，GPU DMA 走 NUMA remote path。
- 解法：`numactl --cpunodebind=0 --membind=0` 绑 GPU 所在 node（先 `cat /sys/class/.../numa_node` 确认 GPU 在哪个 node），提升 25%。

## 6.9 面试视角

### 6.9.1 高频面试题（Staff 级）

**Q1：`context_switch()` 里真正"换 CPU 寄存器"的是哪部分？其余在干什么？**
框架：唯一换寄存器/栈的是 `switch_to(prev,next,prev)`→`__switch_to_asm`（push/pop callee-saved + 换 `rsp`）；其余约 90% 代码在做 ① lazy TLB 的 mm 借用/`mmgrab_lazy_tlb`；② membarrier 内存序（`switch_mm` 的 `CR3` 写充当 full barrier）；③ `switch_mm_irqs_off` 换 `CR3` 并按 PCID 决定刷不刷 TLB；④ `finish_task_switch` 延后 `mmdrop` 防死锁。能点出"换寄存器只一行、复杂度在换 mm"即 Staff 水准。

**Q2：直接开销 vs 间接开销，哪个大？**
框架：直接——寄存器拷贝 ~几十 ns、`schedule()` 路径 ~数百 ns–1μs、跨进程 `CR3`+TLB ~0.5–5μs；间接——cache cold start ~10–100μs、NUMA remote。结论：**间接常 10x–100x 于直接**，调优瞄准局部性。陷阱答案"切换不贵"——直接代价确实小，但忽视了间接。

**Q3：voluntary / involuntary 的内核精确判据？**
框架：`__schedule()` 里 `switch_count` 默认 `&nivcsw`；仅当 `!preempt && prev_state` 且非信号唤醒、任务被 `block_task` 摘下 runqueue 时才改 `&nvcsw`。所以"被抢占"或"设了睡眠态却被信号打断"都算 involuntary。诊断：nonvol 高→CPU 饱和/RT 抢占；vol 高且吞吐低→futex/锁竞争。

**Q4：为什么 lazy FPU 被废？和 CVE-2018-3665 什么关系？**
框架：废的原因是**性能**——glibc 用 SSE2 让"等用到再存"假设破产，`#NM` 往返比直接 `XSAVE` 还慢；~4.6 eager 默认、~4.9 hard-disable。CVE-2018-3665(LazyFP) 是 **2018 事后追认**（先改后爆），利用 lazy 窗口投机读残留 FPU 寄存器；eager 内核天然免疫。能说"因果不能倒置"是加分。

**Q5：PCID 是 12bit=4096，为什么实际很容易刷 TLB？**
框架：Linux 每 CPU 只用 `TLB_NR_DYN_ASIDS=6` 个 ASID 槽（源码注释："6 should be plenty and tlb_state fits in two cache lines"），6 槽 LRU；进程被挤出环就丢 TLB 标签；KPTI 再让内核/用户各占一个 PCID。所以"有 PCID 不刷 TLB"只对"最近 6 个进程互切"成立。

**Q6：`isolcpus + nohz_full + rcu_nocbs` 各解决什么、缺一不可吗？isolcpus 现在还推荐吗？**
框架：isolcpus 挡普通任务调度但挡不住时钟/软中断；nohz_full 关每 1ms timer 但挡不住 RCU 软中断；rcu_nocbs 迁出 RCU 但挡不住外设 IRQ（要 `irqaffinity`）。四者正交全开才逼近独占核。加分：**isolcpus 已 deprecated-kinda**（静态、不能运行时改），cgroup v2 `isolated` partition 是现代接班人。

**Q7：`SCHED_FIFO` vs `SCHED_DEADLINE`？谁优先级高？**
框架：**DEADLINE 在内核里优先级高于 FIFO/RR**（`dl_sched_class` 先于 `rt_sched_class`）。FIFO 静态优先级、不限带宽、跑死循环 starve 系统；DEADLINE 声明 runtime/deadline/period + CBS 限带宽不饿死系统。保护网：`sched_rt_runtime_us`（默认 RT 类每秒最多 0.95s）。

**Q8：SMT 开启时绑核策略怎么改？为什么 SMT 可能更慢？**
框架：同物理核两逻辑核共享 **L1/L2、执行端口、TLB、store buffer**（不是共享 L3——L3 本就跨核共享）。两个 CPU-bound 线程压一对 sibling 会抢核内资源，单线程吞吐可能不如独占一个逻辑核。`lscpu -p` 找 sibling，要么绑整物理核、要么只用一个并空置兄弟。延迟敏感常 BIOS 关 SMT。

**Q9：`switch_to(prev, next, last)` 第三个参数 `last` 是干嘛的？**
框架：切换是"跳走、将来从别处跳回"。当某任务切回 prev、prev 在自己栈上"醒来"时，需知道"我从谁那被切回"（那个 prev 可能已迁到别的 CPU 或已死要 drop）。`__switch_to_asm` 经 `%rax` 透传返回这个"真正的前任"，赋给 `last`。

## 6.10 实战项目：用 perf 量一次上下文切换的代价

### 目标
量化直接开销 + cache 间接开销，建立"切换不只是几百纳秒"的直觉，并**把三个量级（裸 `switch_to` / `schedule()` 路径 / pipe 往返）拆开**。

### 6.10.1 实验一：直接开销测量（pipe ping-pong）

```c
// ctx_switch_bench.c — 两进程经 pipe 交替 ping-pong，每 round-trip 2 次切换
// 编译：gcc -O2 -o ctx_switch_bench ctx_switch_bench.c
#include <stdio.h>
#include <stdlib.h>
#include <unistd.h>
#include <time.h>

int main(int argc, char *argv[]) {
    if (argc < 2) { fprintf(stderr, "usage: %s <iterations>\n", argv[0]); return 1; }
    int n = atoi(argv[1]);
    int p1[2], p2[2];
    if (pipe(p1) || pipe(p2)) { perror("pipe"); return 1; }
    pid_t child = fork();
    if (child == 0) {
        char buf;
        for (int i = 0; i < n; i++) { read(p1[0], &buf, 1); write(p2[1], &buf, 1); }
        exit(0);
    }
    char buf = 'x';
    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &start);
    for (int i = 0; i < n; i++) { write(p1[1], &buf, 1); read(p2[0], &buf, 1); }
    clock_gettime(CLOCK_MONOTONIC, &end);
    long ns = (end.tv_sec - start.tv_sec) * 1000000000L + (end.tv_nsec - start.tv_nsec);
    printf("Each context switch (incl. pipe+syscall): ~%ld ns\n", ns / (2 * n));
    return 0;
}
```
【示意代码，非逐字（教学 benchmark，非内核源码）】

```bash
gcc -O2 -o ctx_switch_bench ctx_switch_bench.c
taskset -c 0 ./ctx_switch_bench 1000000     # 父子都钉核 0，强制串行 ping-pong
# 确认切换次数 ≈ 2n，反推单次：
taskset -c 0 perf stat -e context-switches ./ctx_switch_bench 1000000
```

**为什么钉同一核**：否则父子在两个核并行各跑各的，测的是 IPC 流水线而非"切出去再切回来"。**验收**：通常 **~1–5 μs**（含两次 syscall + pipe 内核路径 + 两次切换），与 lmbench `lat_ctx` 进程间数字同量级。**这不是裸 `switch_to`（那只数十 ns）**——正是 6.3.6 表格要拆清的点。

### 6.10.2 实验二：cache 代价测量

```bash
# 基准：锁同核（最少 cache miss）
taskset -c 0 perf stat -e cache-misses,LLC-load-misses,context-switches \
    stress-ng --cpu 1 --cache 1 -t 5

# 对照：允许自由迁移
perf stat -e cache-misses,LLC-load-misses,context-switches \
    stress-ng --cpu 1 --cache 1 -t 5
```
**验收**：绑核组 LLC-load-misses 明显低于自由迁移组（常低 2x–10x），能用输出解释哪部分代价更大。

### 6.10.3 实验三：perf sched 调度延迟分析

```bash
sudo perf sched record -a -- sleep 5
sudo perf sched latency        # per-task avg/max lat + switches
sudo perf sched timehist | head -50
# 输出示意：
# Task            | avg lat | max lat | switches
# stress-ng:1234  | 100 us  | 5.3 ms  |  12000
```

### 6.10.4 实验四：NUMA 代价对比（需 NUMA 机器）

```bash
numactl --hardware
# 本地 node
numactl --cpunodebind=0 --membind=0 perf stat \
    -e dTLB-load-misses,node-load-misses ./my_benchmark
# 跨 node（CPU 在 0，内存在 1）
numactl --cpunodebind=0 --membind=1 perf stat \
    -e dTLB-load-misses,node-load-misses ./my_benchmark
# 预期：跨 node 版 latency 高 30–100%，node-load-misses 显著上升
```

## 6.11 设计题

**题目**：设计低延迟 RPC 框架，目标 P99 < 100μs（16 核双路 NUMA、10Gbps NIC）。从 CPU 亲和、中断亲和、线程模型、上下文切换频率四维度给方案并分析 trade-off。

**答题要点**：

1. **线程模型**：不用"一连接一线程"（海量切换 + 8MB 栈/线程）。用 N=核数 的 event loop（`epoll`/`io_uring`），无锁 per-thread handler，连接 hash 到固定 loop（避免跨 loop 迁移破坏 cache）。
2. **CPU 亲和**：隔离 4–8 核给 RPC loop（cgroup v2 isolated partition 或 isolcpus + nohz_full）；每 loop 绑一核；这些核与 NIC 同 NUMA node（查 `/sys/class/net/<iface>/device/numa_node`）。
3. **中断亲和**：NIC 多队列，每队列 IRQ 绑到对应 loop 核（**收包→中断→本核 loop→零跨核切换**）；其余 IRQ `irqaffinity` 迁出 RPC 核；配合 RPS/RFS/XPS（第 18 章）。
4. **上下文切换控制**：loop 线程 `SCHED_FIFO`/`SCHED_DEADLINE`；跨线程调用用无锁队列 + CAS，避免 mutex 触发的 voluntary 切换；busy-poll（`SO_BUSY_POLL` / `io_uring` SQPOLL）在极低延迟档可换掉中断唤醒。
5. **trade-off**：隔离核→降系统灵活性、突发负载差；`SCHED_FIFO`→必须严控 bug 防 starvation（配 `sched_rt_runtime_us` 兜底）；busy-poll→烧 CPU 换延迟，能耗与延迟的取舍；DPDK/内核旁路→进一步消除内核路径但放弃内核网络栈生态。

## 6.12 系统题：真实排障场景（带量级估算 + IPC 根因）

**场景**：32 核服务器跑 Java 微服务（JVM 堆 8GB），高峰 P99 从 10ms 突增 150ms，CPU 使用率约 60%（不饱和），无明显 GC 告警。

**排查（带量级）**：

```
Step 1: vmstat 1
→ cs：80,000/s（正常应 < 20,000）→ 确认切换问题

Step 2: 找来源
pidstat -w 1 | sort -k6 -rn | head -20
→ java 进程 200+ 线程，nvcswch/s（非自愿）极高
→ jstack：70 个线程 RUNNABLE 但都短暂

Step 3: 判断是否 CPU 饱和
→ 60% × 32 ≈ 19 核等效；200 线程/32 核 = 6.25，竞争度不极端
→ 猜 cache thrashing 而非 CPU 不够

Step 4: 验证 cache 假设
perf stat -e instructions,cycles,LLC-load-misses -p <java_pid> sleep 10
→ LLC-load-misses: 50 亿/10s = 5 亿/s
→ 8GB 堆 vs 32MB LLC，工作集 ≫ LLC，确认 thrashing
→ IPC 从 ~2 塌到 <0.5

Step 5: 根因
→ 线程池是 newCachedThreadPool()（无上限），高峰创建 200 线程
→ 200 线程的栈 + 本地变量把工作集撑大，LLC 每次 cold miss

解法：
1. 改 FixedThreadPool(N_cpu × 2) ≈ 64
2. numactl --cpunodebind=0 --membind=0
3. 结果：P99 → 12ms，cs → 8000/s，LLC miss 降 8x，IPC 回到 ~1.8
```

**量级估算（解释"60% 利用率却很慢"的悖论）**：一次打本地 DRAM 的 LLC miss 访存延迟常 ~60–100ns。但 CPU 乱序 + 多 outstanding miss 并行（MLP），单看延迟会高估停顿，取**有效摊薄 ~30ns/miss** 做保守下界：5 亿 miss/s × ~30ns = **15,000ms 停顿/秒 ≈ 15 个核的算力砸在等内存上**。这正解释了悖论——**利用率统计把"core 占着但在 stall 等访存"也算 busy**，大量周期其实没在算。**证实手段：看 IPC**——cache thrashing 时 IPC 从 ~2 塌到 <0.5，是"利用率高但 stall 多"的铁证。

## 6.13 代码题：eBPF/bpftrace 测量上下文切换 off-CPU 延迟

```c
// ctx_switch_bpf.bt — bpftrace
// 运行：sudo bpftrace ctx_switch_bpf.bt <target_pid>
// 注意：示意级；字段名以 `bpftrace -lv tracepoint:sched:sched_switch` 为准
tracepoint:sched:sched_switch
{
    if (args->prev_pid == $1) { @switch_out_ts[$1] = nsecs; }   // 被切出
    if (args->next_pid == $1) {                                  // 切回来
        if (@switch_out_ts[$1]) {
            $delta = nsecs - @switch_out_ts[$1];
            @off_cpu_us = hist($delta / 1000);
            delete(@switch_out_ts[$1]);
        }
    }
}
END {
    printf("\nOff-CPU latency (us):\n"); print(@off_cpu_us);
    clear(@switch_out_ts);
}
```
【示意代码，非逐字（bpftrace 脚本，非内核源码）】

```bash
sudo apt install bpftrace
sudo timeout 10 bpftrace ctx_switch_bpf.bt <your_pid>
# 输出示意：
# @off_cpu_us:
# [0,1)   1234 |#####|
# [1,2)   5678 |####################|
# [1000,2000) 12 | |  <- 被抢占后等很久的尾部异常值
```

**验证**：① 绑核 vs 非绑核各跑一次，对比 P99 尾部；② `stress-ng --cpu 32` 制造繁忙后重跑，看尾延迟变化。

**暗坑**：① `sched_switch` 的参数字段名随内核版本可能变（`prev_pid`/`next_pid` 的确切可用性「待核」，先 `bpftrace -lv` 确认）；② bpftrace 本身在每次 `sched_switch` 触发 BPF，极高切换率下有可观 overhead，会污染测量；③ 现成方案优先用 bcc 的 `runqlat`/`offcputime`/`cpudist`，比手写更稳。

## 章末五件套

### 一、本章心法
> **上下文切换的复杂度 90% 不在"换寄存器"（那只一行 `switch_to`），而在"换地址空间时如何不破坏 membarrier 内存序、如何用 lazy TLB 省掉 kernel thread 的页表切换、如何安全延后 mmdrop"。代价的 90% 不在切换本身，而在切换后的 cache/TLB cold start。优化切换频率是治标，优化工作集局部性才是治本。**

### 二、一页速查
- 切换三段：`switch_mm`（换 CR3 + PCID 决定刷不刷 TLB）/ `switch_to`（换 callee-saved + rsp）/ `finish_task_switch`（延后 mmdrop + 回收死亡任务）。
- vol/invol 判据：默认 invol；仅 `!preempt && prev_state && !signal && 被 block_task` 才 vol。
- PCID：硬件 4096，Linux 每 CPU 6 槽 ASID 环（`TLB_NR_DYN_ASIDS`）。
- FPU：eager（~4.6 默认/~4.9 删 lazy，性能驱动）；AMX 8KB 用 XFD 按需分配（lazy 分配 ≠ lazy save）。
- 亲和：`cpus_mask`(用户意图) vs `cpus_ptr`(当前生效)；effective = 用户 ∩ cpuset（静默覆盖）。
- 隔离：`isolcpus`(deprecated-kinda) → cgroup v2 isolated partition；+ `nohz_full` + `rcu_nocbs` + `irqaffinity` 四件套。
- RT：DEADLINE > FIFO > RR > OTHER；`sched_rt_runtime_us` 兜底；PREEMPT_RT 自 6.12 进主线。
- 命令：`vmstat`(cs) / `pidstat -w` / `perf sched latency` / `numastat -p` / `perf stat -e LLC-load-misses,dTLB-load-misses` + 看 IPC。

### 三、面试 30 秒电梯版
"上下文切换里真正换 CPU 寄存器的只有 `switch_to` 一行（push/pop 6 个 callee-saved + 换 rsp，几十 ns）；`context_switch` 90% 复杂度在换 mm——lazy TLB 让 kernel thread 借用前任 mm 不刷 TLB，membarrier 靠 CR3 写当屏障，mmdrop 延到 finish_task_switch 防死锁。代价大头不是这几百纳秒，是切换后 cache/TLB cold start，10x–100x。所以治本是绑核保局部性、控线程数，不是单纯降切换频率。vol/invol 看内核 `switch_count`，默认 invol，被 block 才 vol。"

### 四、本章之最
- **最容易讲反**：lazy FPU 退场原因（性能 ≠ 安全；先改后爆 ≠ 因 CVE 而改）。
- **最反直觉**：PCID 4096 个但每 CPU 只用 6；线程不一定比进程快的边界（看是否同 mm）。
- **最容易漏**：`sched_setaffinity` 被 cpuset 静默覆盖（容器"绑核没生效"头号坑）。
- **最深的根因**：60% 利用率却慢 = IPC 塌陷 = core 占着但在 stall 等内存。
