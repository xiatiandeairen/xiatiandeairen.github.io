> **前置依赖**：熟悉 Chromium 多进程架构（第 1 章）、SiteInstance/SiteIsolation 概念（第 2 章）、Mojo IPC 基础（第 3 章）。了解 HTTP 请求/响应、重定向、Service Worker 基础。
>
> **TL;DR**：Chromium 导航不是一个函数调用，而是一条横跨 Browser 进程、Network 进程、Renderer 进程的分布式状态机。核心驱动对象是 `NavigationRequest`（Browser 进程）+ `DocumentLoader`（Renderer/Blink）。两者通过 Mojo IPC 握手完成"commit"。整个生命周期分为两大阶段：**Navigation Phase**（URL 确认 → 进程选择 → commit）和 **Loading Phase**（commit 后 → DOMContentLoaded → onload）。理解这条链路是排查白屏/慢加载/跨进程跳转/BFCache 失效/Prerender 激活等生产问题的基础。

## 1. 设计考古：PlzNavigate 重构的来由

### 1.1 旧世界：renderer-initiated 导航的问题

2013 年前，Chromium 的导航模型以 Renderer 进程为控制核心——当用户点击链接或调用 `window.location.href = ...`，Renderer 直接向网络栈发起请求，Browser 进程只负责更新地址栏和历史记录。这个模型在多进程架构确立后逐渐暴露出三个根本性缺陷：

**安全漏洞**：Renderer 可以在 Browser 不知情的情况下发起任意网络请求，包括利用 SSRF 攻击内网服务。更严重的是：Renderer 可以声称"我已经在这个 origin 上了"，伪造导航完成状态，绕过安全检查。

**进程切换时序错乱**：Cross-site 导航需要切换 Renderer 进程（SiteInstance 不同），而旧模型中 Renderer 先拿到响应再通知 Browser，此时 Browser 才能决定用哪个进程——响应体已经送到了错误的进程。

**Policy 无处落脚**：NavigationThrottle（扩展拦截、SafeBrowsing 检查、CSP 等）必须在 Browser 侧执行，但旧模型的请求时序使 Browser 只能事后审计，无法在请求发出前介入。

### 1.2 PlzNavigate：把控制权收回 Browser

2014 年，Chrome 团队在 [issue 320961](https://crbug.com/320961) 提出 **PlzNavigate**（Please let the Browser process navigate），设计目标是：

> "The goal is that the browser is in charge of navigations and the renderer just commits them."

核心架构转变：**所有导航的网络请求由 Browser 进程发起并控制**，Renderer 不再直接碰网络。Browser 拿到完整 response headers 后，决定目标 SiteInstance，创建或复用 Renderer 进程，再通过 Mojo IPC 把"commit 令牌"发给 Renderer，Renderer 只负责消费响应体、构建 Document。

PlzNavigate 于 Chrome 54（2016）在 Desktop 完成全量推送。此后 `NavigationRequest` 成为 Browser 侧导航的核心对象。

### 1.3 NavigationRequest 的设计动机

PlzNavigate 之后的导航架构需要一个对象跟踪从"用户按下回车"到"新 Document commit"的全部状态。`NavigationRequest` 承担这个角色：

- 实现 `NavigationHandle` 接口，暴露给 embedder（Electron、WebView、Chrome extensions）查询/修改导航状态
- 持有 NavigationThrottleRunner，串行执行所有 throttle 检查
- 持有 NavigationURLLoader，管理实际网络请求
- 在 READY_TO_COMMIT 后向目标 RenderFrameHost 发送 commit IPC

`NavigationRequest` 的生命周期覆盖从 `FrameTreeNode::TakeNavigationRequest()` 被创建，到 renderer 调用 `DidCommitProvisionalLoad` IPC 回来之前。commit 后所有权转移到 `RenderFrameHostImpl`。

## 2. 状态机精读：NavigationState

### 2.1 完整状态枚举

【真实源码 content/browser/renderer_host/navigation_request.h】

```cpp
// Chromium src/content/browser/renderer_host/navigation_request.h
// (经 WebFetch 核实，字段顺序来自实际 header)
enum NavigationState {
  NOT_STARTED = 0,
  WAITING_FOR_RENDERER_RESPONSE,  // Browser-initiated: 等待 renderer 响应 BeginNavigation
  WILL_START_NAVIGATION,          // 通知 WebContentsObserver::DidStartNavigation
  WILL_START_REQUEST,             // 运行 WillStartRequest throttles
  WILL_REDIRECT_REQUEST,          // 运行 WillRedirectRequest throttles（每次重定向）
  WILL_PROCESS_RESPONSE,          // 运行 WillProcessResponse throttles
  WILL_COMMIT_WITHOUT_URL_LOADER, // same-document / error page / data URL 等不走 URLLoader 的路径
  READY_TO_COMMIT,                // throttle 全通过；等待 renderer 确认
  DID_COMMIT,                     // renderer 回调 DidCommitNavigation；正常完成
  CANCELING,                      // 中途取消（throttle CANCEL / beforeunload 拒绝 等）
  WILL_FAIL_REQUEST,              // 运行 WillFailRequest throttles
  DID_COMMIT_ERROR_PAGE,          // error page commit 完成
};
```

### 2.2 状态转换图

```
用户/脚本触发 URL 变更
        │
        ▼
  [NOT_STARTED]
        │ Navigator::Navigate() 或 OnBeginNavigation()
        ▼
  [WAITING_FOR_RENDERER_RESPONSE]  ──── (仅 browser-initiated)
        │ BeginNavigation()
        ▼
  [WILL_START_NAVIGATION]
        │ DidStartNavigation 通知 observer
        ▼
  [WILL_START_REQUEST]
        │ throttle runner 串行执行 WillStartRequest
        │ 全部 PROCEED → URLLoader.Start()
        ▼
  [WILL_REDIRECT_REQUEST] ◄──────────────────────┐
        │ 每次 HTTP 3xx 重定向                    │
        │ throttle WillRedirectRequest           │
        └────────────── 循环 ────────────────────┘
        │ 最终非重定向响应到达
        ▼
  [WILL_PROCESS_RESPONSE]
        │ throttle WillProcessResponse
        │ 选择目标 SiteInstance / RenderFrameHost
        ▼
  [READY_TO_COMMIT]
        │ RenderFrameHost::CommitNavigation() IPC 发出
        │ ReadyToCommitNavigation 通知 observer
        ▼
  [DID_COMMIT] 或 [DID_COMMIT_ERROR_PAGE]
        │ renderer DidCommitNavigation IPC 回来
        ▼
      (NavigationRequest 销毁，RFH 接管)

网络失败 → [WILL_FAIL_REQUEST] → [DID_COMMIT_ERROR_PAGE]
任意阶段 throttle CANCEL → [CANCELING]
```

## 3. 核心流程深挖

### 3.1 导航触发：两条入口

**Browser-initiated（地址栏回车、`webContents.loadURL()`）：**

### 3.2 NavigationThrottle 执行机制

`NavigationThrottleRunner` 维护一个 throttle 列表，对每个检查点串行执行：

【真实源码 content/public/browser/navigation_throttle.h，经 WebFetch 核实】

```cpp
// ThrottleAction 定义 throttle 可以做的事
enum ThrottleAction {
  PROCEED,              // 放行，继续下一个 throttle
  DEFER,                // 异步等待，之后 Resume() 恢复
  CANCEL,               // 取消导航（显示 error page）
  CANCEL_AND_IGNORE,    // 取消但不显示 error
  BLOCK_REQUEST,        // 阻止网络请求（WillStartRequest/WillRedirectRequest 可用）
  BLOCK_REQUEST_AND_COLLAPSE,  // 阻止并折叠 iframe
  BLOCK_RESPONSE,       // 阻止响应处理（WillProcessResponse 可用）
};

// 五个可 override 的检查点
virtual ThrottleCheckResult WillStartRequest();
virtual ThrottleCheckResult WillRedirectRequest();
virtual ThrottleCheckResult WillFailRequest();
virtual ThrottleCheckResult WillProcessResponse();
virtual ThrottleCheckResult WillCommitWithoutUrlLoader();
```

**DEFER 的实现**：throttle 返回 DEFER 后，ThrottleRunner 不继续下一个 throttle；throttle 持有 `NavigationHandle*` 引用，在异步操作完成后调用 `Resume()` 或 `CancelDeferredNavigation()`。Runner 还有一个 `ReportStuckThrottle()` 计时器，防止 throttle 无限 hang。

**生产陷阱**：在 `WillProcessResponse` 中 DEFER 是高代价操作——此时 HTTP 连接已建立，对端可能等待 ACK，长时间 defer 会导致连接超时。

### 3.3 进程选择与 RFH 分配

`OnResponseStarted` 触发后，在进入 `WILL_PROCESS_RESPONSE` throttle 前，Browser 需要确定目标 SiteInstance，进而确定目标 RenderFrameHost：

【真实源码 content/browser/renderer_host/render_frame_host_manager.h，经 WebFetch 核实】

```cpp
// 选择或创建目标 RFH
// 可能提前（在 WillStartRequest 时）投机创建 speculative RFH
RenderFrameHostImpl* GetFrameHostForNavigation(
    NavigationRequest* request,
    BrowseInstanceSwapType* swap_type,
    ...);

// 提前准备 renderer process（与 network 请求并行）
void CreateSpeculativeRenderFrame(
    scoped_refptr<SiteInstance> site_instance,
    ...);
```

决策逻辑（简化）：

```
当前 RFH 的 SiteInstance == 目标 SiteInstance ?
    ├── 是：复用当前 RFH（same-site navigation）
    └── 否：需要进程切换
            ├── 目标进程是否已存在（同站点其他 tab）?
            │       ├── 是且允许共享：复用该进程新建 RFH
            │       └── 否：创建新进程，新建 RFH
            └── 创建 speculative RFH（kSpeculative 状态）
```

`LifecycleStateImpl` 枚举记录 RFH 的状态：

【真实源码 content/browser/renderer_host/render_frame_host_impl.h，经 WebFetch 核实】

```cpp
enum class LifecycleStateImpl {
  kSpeculative,           // 投机创建，导航可能被取消
  kPendingCommit,         // CommitNavigation IPC 已发出，等待 renderer 确认
  kPrerendering,          // 预渲染中（不可见）
  kActive,                // 当前活跃可见文档
  kInBackForwardCache,    // 冻结在 BFCache 中
  kRunningUnloadHandlers, // 执行 unload/pagehide 事件
  kReadyToBeDeleted       // 等待析构
};
```

### 3.4 CommitNavigation IPC：Mojo 握手

Browser 侧确认 `READY_TO_COMMIT` 后，调用：

【真实源码 content/browser/renderer_host/render_frame_host_impl.h，经 WebFetch 核实】

```cpp
void CommitNavigation(
    NavigationRequest* navigation_request,
    blink::mojom::CommonNavigationParamsPtr common_params,
    blink::mojom::CommitNavigationParamsPtr commit_params,
    network::mojom::URLResponseHeadPtr response_head,
    mojo::ScopedDataPipeConsumerHandle response_body,       // 响应体 DataPipe
    network::mojom::URLLoaderClientEndpointsPtr url_loader_client_endpoints,
    blink::mojom::ControllerServiceWorkerInfoPtr controller,
    ...);
```

`CommitNavigationParams` 是最复杂的参数包，包含（经 WebFetch 核实）：

```
- origin_to_commit         : 最终确认的 origin（Security 核心）
- storage_key              : Document 的存储隔离 key
- nav_entry_id             : 对应 NavigationEntry
- page_state               : blink::PageState（序列化的滚动/表单状态）
- navigation_token         : 唯一标识此次导航的 UnguessableToken
- not_restored_reasons     : BFCache 未命中原因（通过 CommitParams 送到 renderer）
- view_transition_state    : View Transition API 状态
- lcpp_hint                : LCP Critical Path Predictor 提示
- modified_runtime_features: Blink feature 开关 map
- commit_sent              : Browser 侧 IPC 发出时间（供 Navigation Timing 使用）
```

这个 IPC 通过 Mojo AssociatedRemote（`blink::mojom::LocalFrame`）发送，保序性由 associated channel 保证——不能乱序。

### 3.5 Renderer 侧：DocumentLoader 接管

CommitNavigation IPC 到达 Renderer 后，走 `RenderFrameImpl` → `DocumentLoader::CommitNavigation()`：

【真实源码 third_party/blink/renderer/core/loader/document_loader.h，经 WebFetch 核实】

```cpp
// DocumentLoader 状态机
enum State {
  kNotStarted,     // 刚创建
  kProvisional,    // CommitNavigation 收到但 Document 尚未创建
  kCommitted,      // Document 已创建，Parser 已启动
  kSentDidFinishLoad  // onload 已触发，加载完成
};
```

`CommitNavigation` 做的事：
1. 创建新 `LocalDOMWindow`（如有必要）
2. 创建 `Document` 对象（`Document::Create()`）
3. 调用 `DidInstallNewDocument()`，初始化 CSP、Origin Trial、SecurityOrigin
4. 创建 HTML Parser（`HTMLDocumentParser`）
5. 切换 state → `kCommitted`

之后 `StartLoadingResponse()` 被调用，DataPipe 中的响应体开始流入 Parser。

**注意**：从 Browser 发出 CommitNavigation IPC 到 Renderer 执行完 `CommitNavigation()`，这段时间内 Browser 侧显示的是旧 Document（地址栏已经更新）。这是"nav commit delay"的来源，Production 上偶发白屏/地址栏与页面内容短暂不一致的根因。

### 3.6 DidCommitNavigation：合拢

Renderer 完成 Document commit 后，发回 IPC `DidCommitProvisionalLoad`（旧名；新 Mojo path 是 `mojom::Frame::DidCommitNavigation`）。Browser 在 `RenderFrameHostImpl::DidNavigate()` 中处理：

```cpp
// 示意，非逐字
void RenderFrameHostImpl::DidNavigate(
    const mojom::DidCommitProvisionalLoadParams& params,
    NavigationRequest* navigation_request,
    bool was_within_same_document) {
  // 1. 验证 origin（防止 renderer 欺骗）
  // 2. 更新 NavigationEntry / 历史记录
  // 3. NavigationRequest 所有权从 FrameTreeNode 转移到 RFH
  // 4. LifecycleState → kActive
  // 5. 通知 WebContentsObserver::DidFinishNavigation
  // 6. 调用 CommitPending() 清理旧 RFH
}
```

`CommitPending()` 触发旧 RFH 的 `kRunningUnloadHandlers` → 执行 `pagehide`/`unload` → `kReadyToBeDeleted`（或进入 BFCache → `kInBackForwardCache`）。

## 4. Loading Phase：commit 之后

### 4.1 两阶段分界

| 事件 | Browser 观测点 | Renderer 观测点 |
|------|-------------|--------------|
| `DidStartNavigation` | NavigationRequest 创建 | — |
| `ReadyToCommitNavigation` | READY_TO_COMMIT | — |
| `DidFinishNavigation` | DID_COMMIT | — |
| `DOMContentLoaded` | `WebContentsObserver::DOMContentLoaded(RFH*)` | `document` readyState = interactive |
| `DidFinishLoad` | `WebContentsObserver::DidFinishLoad(RFH*, url)` | `window.onload` 触发后 |
| `DidStopLoading` | 所有 frame 加载完成 | — |

**关键区别**：

- `DidFinishNavigation` 是 **Navigation Phase 结束**，标志 Document 已创建、URL 已确认。此时页面很可能还是空白（Parser 刚开始工作）。
- `DOMContentLoaded` 是 HTML Parser 完成、deferred script 执行完——此时 DOM 树可用，但图片/CSS 可能未完成。
- `DidFinishLoad` 是 `window.onload`——所有同步子资源加载完毕。
- `DidStopLoading` 是 **所有 frame**（包括 iframe）加载完成，是"loading spinner 消失"的时机。

### 4.2 Parser 阻塞点

```
DataPipe → HTMLDocumentParser → Tokenizer → Tree Builder → DOM
                  │
                  ├── <script> (非 async/defer) → JS 引擎执行 → Parser 阻塞
                  ├── <link rel=stylesheet> → CSS Parser → CSSOM → 可能阻塞渲染
                  ├── Preload Scanner（并行扫描 src/href 提前发请求）
                  └── parser_blocked_count_ > 0 → Parser 挂起
```

`DocumentLoader::parser_blocked_count_` 记录当前阻塞 Parser 的脚本数量。每个 blocking `<script>` 执行完后 decrement，到 0 时 Resume。

### 4.3 子资源加载

commit 后，新 Document 持有 `ResourceFetcher`，负责发起所有子资源请求。子资源走正常 URLLoader 路径（Renderer 进程 → Network Service），不经过 `NavigationRequest`。

## 5. 特殊导航类型

### 5.1 Same-Document Navigation

`history.pushState()`、`location.hash` 变更、Navigation API 的 `navigate()` 在 same-document 场景不创建新 Document，走快速路径：

```
NavigationRequest 创建（isSameDocument = true）
    │
    ▼
WILL_COMMIT_WITHOUT_URL_LOADER  ← 跳过 URLLoader 和网络阶段
    │
    ▼
Renderer::CommitSameDocumentNavigation IPC
    │
    ▼
DocumentLoader 更新 URL / history entry，不重建 Document
    │
    ▼
DID_COMMIT
```

### 5.2 Back/Forward Cache（BFCache）激活

BFCache 完全绕过 Navigation Phase 的网络部分：

```
GoBack() → NavigationRequest (isPageActivation = true)
    │
    ▼
NavigationURLLoader type = kNoopForBackForwardCache
    │
    ▼
READY_TO_COMMIT
    │
    ▼
BackForwardCacheImpl::RestoreEntry() → 从内存取出冻结 RFH
    │
    ▼
RFH lifecycle: kInBackForwardCache → kActive
    │
    ▼
Renderer：resume timers / resume JS / 触发 resume + pageshow(persisted=true)
```

BFCache 阻塞因素（真实常见）：

| 阻塞条件 | 原因 |
|---------|------|
| `unload` event listener | 历史遗留；用 `pagehide` 替代 |
| `Cache-Control: no-store` | 响应头指示不缓存 |
| IndexedDB 事务未关闭 | 持有存储锁 |
| WebSocket / WebRTC 开放连接 | 连接状态不可冻结 |
| `window.opener` 未清理（非 noopener） | 关联上下文存在 |
| Sticky feature 使用（Broadcast Channel 等） | 监测到 sticky blocking feature |

`notRestoredReasons` API（Chrome 123+）可在页面加载时读取上次 BFCache miss 原因，已在 `CommitNavigationParams` 中传递。

### 5.3 Prerender 激活

Prerender 是"提前在后台 render 好一个页面，用户真正导航时零延迟激活"：

```
触发条件：<link rel=prerender>、Speculation Rules API
    │
    ▼
PrerenderHost 创建独立 FrameTree（隔离，不可见）
    │
    ▼
独立 NavigationRequest 在 kPrerendering RFH 中完成加载
    │
    ▼
is_ready_for_activation_ = true
    │
    ▼ 用户触发真实导航到该 URL
    ▼
AreInitialPrerenderNavigationParamsCompatibleWithNavigation() 兼容性检查
    │ 通过
    ▼
NavigationRequest (isPrerenderedPageActivation = true)
NavigationURLLoader type = kNoopForPrerender
    │
    ▼
Activate()：将 prerendered RFH 移入 primary FrameTree
    │
    ▼
RFH: kPrerendering → kActive
触发 prerenderingchange / pageshow
```

### 5.4 History 导航

`GoBack()` / `GoForward()` / `GoToIndex()` 的特殊点在于 **pending_entry 管理**：

【真实源码 content/browser/renderer_host/navigation_controller_impl.cc，经 WebFetch 核实】

```cpp
// GoToIndex 设置 pending_entry 并标记 FORWARD_BACK 转换类型
pending_entry_ = entries_[index].get();
pending_entry_index_ = index;
pending_entry_->SetTransitionType(ui::PageTransitionFromInt(
    pending_entry_->GetTransitionType() | ui::PAGE_TRANSITION_FORWARD_BACK));

```

Chrome 还有"History Manipulation Intervention"：对某些利用 `history.pushState` 刷屏的页面，Back 操作会跳过被标记为 `should_skip_on_back_forward_ui()` 的 entry。

## 6. Frame Tree 与多 Frame 导航

### 6.1 FrameTree 结构

每个 Tab 有一棵 primary `FrameTree`，节点是 `FrameTreeNode`，每个 node 对应一个 frame（主 frame 或 iframe）。

【真实源码 content/browser/renderer_host/frame_tree_node.h，经 WebFetch 核实】

```cpp
// FrameTreeNode 在导航期间持有 NavigationRequest
std::unique_ptr<NavigationRequest> navigation_request_;

// 关键方法
TakeNavigationRequest(std::unique_ptr<NavigationRequest>);  // 接管（取消旧的）
ResetNavigationRequest();                                    // 清除并移除 speculative RFH
TransferNavigationRequestOwnership(RenderFrameHostImpl*);   // ReadyToCommit → DidCommit 之间移交
```

**设计约束**：一个 FrameTreeNode **同时只能有一个** active NavigationRequest。`TakeNavigationRequest` 会取消任何正在进行的导航（除非用同一 NavigationRequest 覆盖自己）。

### 6.2 Multiple Page Architecture（MPArch）

Chrome 127+ 引入 MPArch：一个 `WebContents` 可以拥有多棵 FrameTree：

```
WebContents
  ├── primary FrameTree（可见 Tab）
  ├── prerender FrameTree（BG 预渲染，可能有多个）
  └── bfcache FrameTree（冻结历史页面，可能有多个）
```

BFCache 页面、Prerender 页面在激活前各有独立 FrameTree，激活后 primary FrameTree 的 root RFH 被替换。

### 6.3 Cross-Origin Iframe 导航

Iframe 导航有自己独立的 `NavigationRequest`，和主 frame 并行执行：

- iframe 的 `src` 变更 → Renderer 发 `OnBeginNavigation` IPC
- Browser 为该 FrameTreeNode 创建新 NavigationRequest
- 如果 iframe 与主 frame cross-site，且 SiteIsolation 开启，iframe 运行在独立 renderer 进程
- OOPIF（Out-of-Process iframe）通过 `RenderFrameProxyHost` 在不同进程间同步视图

### 7.1 完整回调序列

【真实源码 content/public/browser/web_contents_observer.h，经 WebFetch 核实】

```
Browser-side callbacks（按实际触发顺序）：
1. DidStartNavigation(NavigationHandle*)

     │ HasCommitted()=true 或 failed
     │ NavigationRequest 即将销毁
     │
(Navigation Phase 结束，Loading Phase 开始)
     │
5. DidStartLoading()
     │
6. DOMContentLoaded(RenderFrameHost*)
     │
7. DidFinishLoad(RenderFrameHost*, GURL)  [per-frame]
     │
8. DidStopLoading()
     │ 所有 frame 完成

附加回调：
  DidFailLoad(RFH, GURL, error_code)     — 加载失败
  DocumentOnLoadCompletedInPrimaryMainFrame() — 主 frame onload 完成
```

## 8. 网络层交互：NavigationURLLoader

### 8.1 URLLoader 类型

【真实源码 content/browser/loader/navigation_url_loader.h，经 WebFetch 核实】

```cpp
enum class LoaderType {
  kRegular,                   // 正常导航请求
  kNoopForBackForwardCache,   // BFCache 激活，跳过网络
  kNoopForPrerender,          // Prerender 激活，跳过网络
  kNoopForInitialWebUI,       // 初始 WebUI 页面，跳过网络
};

// 核心控制方法
virtual void Start();
virtual void FollowRedirect(/* 重定向后调用，继续请求 */);
virtual bool SetNavigationTimeout(base::TimeDelta timeout);
virtual void CancelNavigationTimeout();
```

`kRegular` loader 会在 Network Service（独立进程）中建立真实的 HTTP 连接，响应头到达后回调 `NavigationRequest::OnResponseStarted()`。

### 8.2 MojoParams 传递路径

```
Browser (UI thread)
    │ NavigationURLLoader::Start()
    ▼
Network Service (独立进程，IO thread)
    │ URLLoader → 建立 TCP/TLS 连接
    │ 收到 HTTP Response Headers
    │ URLLoaderClient::OnReceiveResponse()
    ▼
Browser (UI thread)
    │ NavigationRequest::OnResponseStarted()
    │ 携带 URLLoaderClientEndpoints (Mojo pipe endpoints)
    │   + URLResponseHead (response headers)
    │   + DataPipe consumer handle (response body)
    ▼
Browser → Renderer: CommitNavigation IPC
    │ 传递 DataPipe consumer handle 给 Renderer
    ▼
Renderer: DocumentLoader 从 DataPipe 读取响应体
```

关键：**DataPipe**（Mojo 的共享内存环形缓冲区）直接连接 Network Service 和 Renderer，响应体不经过 Browser 进程内存拷贝。

### Demo 6：NavigationThrottle 最小 Electron patch 示例（patch 走读）

**目标**：理解如何在 Electron 层注入 NavigationThrottle 等价逻辑（不需要 Chromium full build）。

Electron 通过 `session.webRequest` API 暴露了 `WillStartRequest` 和 `WillProcessResponse` 阶段的 hook，这是对 `NavigationThrottle` 的 embedder-friendly 封装：

```javascript
// Electron 层的 NavigationThrottle 等价
const { session } = require('electron');

// 对应 WillStartRequest throttle
session.defaultSession.webRequest.onBeforeRequest(
  { urls: ['https://malware.example/*'] },
  (details, callback) => {
    console.log(`Blocking navigation to: ${details.url}`);
    callback({ cancel: true });  // 对应 ThrottleAction::CANCEL
  }
);

// 对应 WillProcessResponse throttle（在 headers 到达后但 body 消费前）
session.defaultSession.webRequest.onHeadersReceived(
  (details, callback) => {
    // 检查响应头，决定是否继续
    const headers = details.responseHeaders;
    const csp = headers['content-security-policy'] || [];
    console.log(`Response headers received for: ${details.url}`);
    callback({ responseHeaders: headers });  // 对应 PROCEED
  }
);

// Electron 特有：在导航提交前修改 CSP
session.defaultSession.webRequest.onHeadersReceived(
  (details, callback) => {
    callback({
      responseHeaders: {
        ...details.responseHeaders,
        // 注入/修改安全头
        'X-Frame-Options': ['DENY'],
      }
    });
  }
);
```

若需要真正在 Chromium 层注入 `NavigationThrottle`（用于 Electron 的自定义 browser），需要继承 `ContentBrowserClient` 并重写：

```cpp
// 示意（非逐字），Electron 通过 ElectronBrowserClient 实现
// shell/browser/electron_browser_client.cc
void ElectronBrowserClient::CreateThrottlesForNavigation(
    content::NavigationHandle* handle,
    std::vector<std::unique_ptr<NavigationThrottle>>* throttles) {
  // 添加自定义 throttle
  throttles->push_back(std::make_unique<ElectronNavigationThrottle>(handle));
}
```

`ElectronNavigationThrottle` 的 `WillProcessResponse()` 可以检查响应头并决定是否 BLOCK_RESPONSE（对应 `webRequest.onHeadersReceived` 返回 `cancel: true`）。

## 10. 方案对比：导航类型决策树

### 10.1 何时触发各类导航

| 场景 | 导航类型 | URLLoader | Document 重建 | BFCache |
|------|---------|-----------|-------------|---------|
| 点击链接（cross-origin） | 跨进程导航 | kRegular | 是（新进程） | 旧页面可进 BFCache |
| 点击链接（same-origin） | 普通导航 | kRegular | 是 | 旧页面可进 BFCache |
| `history.pushState()` | Same-document | 无 | 否 | 不触发 |
| `location.hash` 变更 | Same-document | 无 | 否 | 不触发 |
| Back/Forward（BFCache 命中） | BFCache 激活 | kNoopForBackForwardCache | 否（恢复） | — |
| Back/Forward（BFCache miss） | 普通重加载 | kRegular | 是 | — |
| Prerender 激活 | Page activation | kNoopForPrerender | 否（移入） | — |
| `location.reload()` | 重新加载 | kRegular | 是 | 旧页面不进 BFCache |
| Error page | 错误页导航 | 无（内部生成） | 是（error RFH） | 否 |

### 10.2 进程切换判断

```
目标 URL 与当前 SiteInstance 关系？
    ├── same-site（eTLD+1 相同）且 SiteIsolation 允许共享
    │       → 无需进程切换，复用当前 RFH
    ├── cross-site（不同 eTLD+1）或有 COOP/COEP 要求
    │       → 需要新 SiteInstance
    │       └── 目标 site 已有进程且 process limit 未到？
    │               ├── 是（有空闲 renderer）→ 复用进程，新 RFH
    │               └── 否（需要新进程）→ CreateSpeculativeRenderFrame
    └── 同 Document（same-document nav）
            → 完全不涉及进程选择
```

## 11. 失败模式与生产真坑

### 11.1 白屏（Blank Flash）

**症状**：导航完成后短暂白屏，或旧页面消失但新页面未出现。

**根因**：CommitNavigation IPC 从 Browser 发出 → Renderer 执行完 `CommitNavigation()` 有延迟（通常 10-50ms），这段时间 Browser 侧已经 commit（地址栏更新），但 Renderer 尚未完成 Document 创建。旧 RFH 已被设为 `kRunningUnloadHandlers`，新 RFH 还在 `kPendingCommit`。

**缓解**：View Transition API（在 `CommitNavigationParams::view_transition_state` 中传递）允许 Renderer 在切换时保持旧帧的快照，避免白屏。

### 11.2 BFCache 静默失效

**症状**：Back 操作本应 <100ms 但实际触发完整重新加载（1-3s），用户感知为"Back 很慢"。

**根因定位**：

```javascript
// 在页面中监控
window.addEventListener('pageshow', e => {
  if (!e.persisted) {
    // 检查 notRestoredReasons（Chrome 123+）
    const nav = performance.getEntriesByType('navigation')[0];
    console.log('BFCache miss reasons:', nav?.notRestoredReasons);
  }
});
```

常见 root cause：
- `unload` listener（最常见；许多第三方 SDK 引入）
- `Cache-Control: no-store` 响应头（告诉 Chrome 不缓存）
- 页面中 `new SharedArrayBuffer()` 或 `new WebAssembly.Memory({shared:true})`
- 未 close 的 IndexedDB 事务

### 11.3 导航 Throttle 造成的挂起

**症状**：点击链接后 loading spinner 一直转，地址栏不变，也无 error page。

**根因**：某个 `NavigationThrottle` 在 `WillStartRequest`、`WillRedirectRequest` 或 `WillProcessResponse` 返回了 DEFER，但相关异步操作（SafeBrowsing check、Extension 决策等）没有在超时内调用 `Resume()`。

**定位**：chrome://net-internals/#events 过滤 NAVIGATION，或 chrome://tracing 找 `ReportStuckThrottle` 事件。

### 11.4 READY_TO_COMMIT 超时（commit timeout）

**症状**：页面加载进度条在接近完成时卡住，最终出现 ERR_CONNECTION_RESET 或空白页。

**根因**：`NavigationRequest` 有 commit timeout，默认 30 秒（来自 `kDefaultCommitTimeout`）。Browser 发出 CommitNavigation IPC 后，如果 Renderer 在 30s 内未回 `DidCommitNavigation`，Browser 会认为 Renderer 无响应，触发 recovery（kill renderer、显示 error page 或 sad tab）。

常见原因：
1. Renderer 进程 CPU 饱和（巨型 DOM / 大量 blocking script）
2. Renderer 进程被挂起（Debug breakpoint、Android 系统限制）
3. IPC 队列积压（前序大量 IPC 阻塞）

**定位**：chrome://tracing，搜索 `CommitNavigationTimeout`。

### 11.5 Same-Document 和 Cross-Document 混淆

**症状**：`DidFinishNavigation` 被调用但 `DOMContentLoaded` 没有触发，或页面内容没有更新。

**根因**：`navigation_handle->IsSameDocument()` 为 true 的导航不重建 Document，不会触发 `DOMContentLoaded`。

如果业务代码在 `DidFinishNavigation` 里假设"导航完成 = 新 Document 完成解析"，会在 SPA 的 `pushState` 场景下出错。

**正确做法**：区分 `IsSameDocument()`，对 cross-document 导航等 `DOMContentLoaded`。

### 11.6 Prerender 激活的 Origin 不匹配

**症状**：明明触发了 Speculation Rules，但导航时没有走 Prerender 快速路径。

**根因**：`AreInitialPrerenderNavigationParamsCompatibleWithNavigation()` 检查失败，可能原因：
1. 实际导航 URL 与 prerender URL 不完全匹配（query parameter 不同）
2. HTTP 方法不同（POST 不能 prerender）
3. 用户状态变化（Cookie 改变导致响应变化，破坏 No-Vary-Search 缓存）

**定位**：chrome://net-internals/#prerender 查看 prerender 状态和失败原因。

## 12. 章末五件套

### 12.1 TL;DR 重申

Chromium 导航是一条跨 Browser/Network/Renderer 三进程的分布式状态机。`NavigationRequest` 是 Browser 侧的全局状态持有者，12 个 `NavigationState` 覆盖从触发到 commit 的完整路径。`DocumentLoader` 是 Renderer 侧的接管者，负责 commit 后的 Document 构建。两者通过 Mojo IPC（CommitNavigation / DidCommitNavigation）握手。BFCache/Prerender 激活走特殊路径，绕过网络层，用 Noop URLLoader 替代。

### 12.2 前置依赖回顾

- **第 1 章（多进程架构）**：理解 Browser/Renderer/Network 三进程是导航请求分工的基础
- **第 2 章（SiteIsolation）**：NavigationRequest 的进程选择逻辑直接依赖 SiteInstance 判断
- **第 3 章（Mojo IPC）**：CommitNavigation / DidCommitNavigation 的传输机制是 Mojo AssociatedRemote
