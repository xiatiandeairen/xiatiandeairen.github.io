> **一句话定位**：内核可观测体系是系统工程师的"第三只眼"——不侵入生产、不改代码、不重启，就能看清内核每一层在做什么，是性能排障从"猜"到"知"的工程基础设施。

---

## TL;DR — 本章最硬结论

1. **观测工具不是平等的**：`perf stat` 硬件计数器开销 <1%，`strace` 因 ptrace 停止/恢复每次 syscall 开销高达 10-100x；生产环境永远先上低开销工具。
2. **eBPF 统一了可观测层**：kprobe/tracepoint/perf_event 等底层机制并没有消失，eBPF 是在它们之上的安全执行层；理解底层机制才能写出不被编译器或内核版本打翻的 BPF 程序。
3. **火焰图是最高信息密度的 CPU 剖析格式**，但它只看 on-CPU 时间；off-CPU 火焰图（等锁、等 IO）同等重要，两者加在一起才是完整图像。
4. **USE 方法（Utilization/Saturation/Errors）适用于资源型瓶颈，RED 方法（Rate/Errors/Duration）适用于请求型服务**；两者不互斥，生产排障先跑 USE 定资源，再跑 RED 定服务质量。
5. **尾延迟本质上是队列延迟**：p99/p999 毛刺几乎总指向锁争用、NUMA 远访、page cache 抖动、软中断 affinity 等"偶发"问题，必须用分位数分布而非均值定位。

## 24.1 背景：为什么需要内核可观测体系

### 问题根源

系统性能问题有三个核心特征让它难以排查：

1. **跨层性**：一个请求从用户态系统调用，经 VFS、page cache、块层、设备驱动，再到硬件 IOPS，每层都可能是瓶颈，但只有 OS 能看到全局。
2. **瞬时性**：锁争用导致的 p99 毛刺可能只持续几十微秒，传统 metrics（1s 粒度）根本采不到。
3. **干涉效应**：传统的 `strace`、`gdb` 通过 `ptrace` 停止进程，观测行为本身就破坏了被观测系统的性能特征——薛定谔的瓶颈。

### 工程需求

生产环境要求：
- 零重启、零重编译：不能为了加日志重新部署。
- 低开销：观测工具本身不能影响延迟分布。
- 可编程：不同问题需要不同聚合逻辑，固定工具无法覆盖所有场景。
- 安全沙箱：不能让任意代码在内核态崩溃整机。

这四个需求驱动了内核可观测体系从 `printk` 到 `ftrace` 到 `eBPF` 的整个演进史。

## 24.2 演进：从 printk 到 eBPF 的 30 年

```
时代           机制              核心问题
─────────────────────────────────────────────────────────────
1991-2000     printk            需要修改代码重编译重部署
2001-2008     OProfile / gprof  只能采样，粒度粗，无动态探针
2006-2008     kprobe (2.6.9+)   可动态插探针，但编程接口原始
2008-2012     perf_events       硬件 PMU 统一抽象，性能分析革命
2008-2009     ftrace framework  函数级跟踪基础设施，tracefs 挂载
2009-2012     tracepoint        静态探针标准化，ABI 稳定
2014          eBPF 内核基建     3.15 扩展 verifier/JIT（内部，无 syscall）
2014          bpf() syscall     3.18 引入 bpf() 系统调用 + BPF maps
2015          BPF on kprobe     4.1 BPF 可挂 kprobe（观测用途的真正起点）
2016          BPF on tracepoint 4.7 BPF 可挂 tracepoint（字段更稳）
2016-2018     BCC / bpftrace    高层工具链，工程师可用
2019-2021     BTF / CO-RE       一次编译到处运行，版本依赖解决
2020+         BPF ring buffer   5.8 多消费者 ringbuf，取代 perf buffer
2022+         BPF kfunc         内核函数经白名单直调
─────────────────────────────────────────────────────────────
```

**关键演进逻辑**：每一次升级都是为了解决前一代的"观测者效应"或"表达力"问题。eBPF 是终态方向，但 ftrace/tracepoint/kprobe 是 eBPF 的底层基础，不是被取代的旧技术。

⚠️ **版本细节别背串了**（面试高频翻车点）：很多人把"eBPF" 一句话归到 3.15 或 3.18，但**对可观测最关键的两个节点是 4.1（BPF 挂 kprobe）和 4.7（BPF 挂 tracepoint）**。3.18 只是有了 `bpf()` syscall + maps，那时还不能把 BPF 程序挂到探针上做 tracing。"3.18 就统一了所有探针" 是常见误传。（节点版本以 iovisor/bcc 的 kernel-versions.md 为准）

### 24.3.1 tracepoint — 静态探针，ABI 稳定的基石

**定义**：内核开发者在源码中预埋的静态探测点，通过 `TRACE_EVENT()` 宏定义，编译时生成对应的 tracepoint 代码桩。

**工作原理**：
```
未激活时：tracepoint 是一条 nop 指令（x86 5字节 nop）
激活时：  nop 被动态替换为 jmp 到 tracepoint handler
          → handler 调用所有注册的 probe function
          → probe function 可以是 ftrace handler 或 BPF 程序
```

**关键特性**：
- ABI 稳定：`TRACE_EVENT` 宏定义了格式，跨内核版本保持兼容
- 开销极低：未激活时几乎零开销（一条 nop）
- 覆盖面广：调度、内存、网络、块 IO、文件系统均有预埋点

**查看可用 tracepoint**：
```bash
# 列出所有 tracepoint 分类
ls /sys/kernel/debug/tracing/events/

# 查看调度相关 tracepoint
ls /sys/kernel/debug/tracing/events/sched/

# 查看某个 tracepoint 的字段格式
cat /sys/kernel/debug/tracing/events/sched/sched_switch/format
```

💡 **坑点**：tracepoint 的格式定义在 `format` 文件里，写 BPF 程序时应该读这个而不是猜结构体字段名——不同内核版本字段可能不同，但 tracepoint ABI 保证主要字段稳定。

**核心 tracepoint 分类（工程高频）**：

| 子系统 | 关键 tracepoint | 排障用途 |
|--------|----------------|---------|
| sched | `sched_switch`, `sched_wakeup` | 调度延迟、off-CPU 分析 |
| syscalls | `sys_enter_*`, `sys_exit_*` | syscall 延迟分布 |
| block | `block_rq_issue`, `block_rq_complete` | IO 延迟、IO 合并 |
| net | `net_dev_xmit`, `netif_receive_skb` | 网络包路径 |
| kmem | `kmalloc`, `kfree` | 内存分配热点 |
| filemap | `mm_filemap_add_to_page_cache` | page cache 命中/缺失 |
| tcp | `tcp_retransmit_skb` | TCP 重传 |

### 24.3.2 kprobe — 动态探针，任意内核函数

**定义**：不需要预埋，可以在运行时动态挂载到任意内核函数入口（kprobe）或返回（kretprobe）的探针机制。

**工作原理（x86_64）**：
```
挂载时：
  1. 保存目标指令第一字节
  2. 将其替换为 int3（0xCC，断点指令）
  3. 注册 handler

触发时：
  1. CPU 执行 int3，陷入断点 handler
  2. 内核 kprobe handler 调用注册的 pre_handler
  3. 单步执行原指令（或通过 out-of-line 复制执行）
  4. 恢复执行后续指令

（待核：具体是否使用 out-of-line execution buffer，近年内核有优化路径，但 int3 触发机制稳定）
```

**相比 tracepoint 的 trade-off**：

| 维度 | tracepoint | kprobe |
|------|-----------|--------|
| 稳定性 | ABI 稳定，跨版本安全 | 函数签名/位置可变，内核版本敏感 |
| 覆盖范围 | 仅预埋点 | 任意导出/非内联函数 |
| 开销 | nop（未激活）→ 跳转 | int3 触发，稍高 |
| 生产推荐 | 优先 | 有 tracepoint 就不用 kprobe |

**kretprobe 限制**：不能用于递归函数、某些中断处理函数；内核用 `NOKPROBE_SYMBOL()` 宏标记禁止挂载的函数。

🎯 **面试加分点**：被问"kprobe 和 tracepoint 有什么区别"时，不要只说"一个静态一个动态"——要说出 ABI 稳定性、开销差异、以及 eBPF 程序里为什么优先选 tracepoint。

### 24.3.3 perf_events — 硬件 PMU 统一抽象

**定义**：`perf_events` 子系统是 Linux 性能事件的统一接口，既支持硬件 PMU（Performance Monitoring Unit）计数器，也支持软件事件（上下文切换、页错误等）和 tracepoint。

**硬件 PMU 能给什么**：

| PMU 事件 | 含义 | 排障用途 |
|---------|------|---------|
| `cpu-cycles` | CPU 周期数 | CPU 热点定位（采样） |
| `instructions` | 指令数 | IPC（每周期指令数）计算 |
| `cache-misses` | L1-L3 缓存缺失 | 伪共享、NUMA 远访 |
| `cache-references` | 缓存访问总次数 | 配合 cache-misses 算命中率 |
| `branch-misses` | 分支预测失败 | 条件判断密集路径 |
| `page-faults` | 缺页次数 | 内存访问模式 |

**采样原理**：

```
设置采样周期 N（如每 10000 个 cpu-cycles 采样一次）
                    │
           ┌────────▼──────────┐
           │ 硬件 PMU 计数器    │
           │ 每 N 个 cycle     │
           │ 触发 NMI 中断     │
           └────────┬──────────┘
                    │ NMI handler
                    ▼
           记录当前 IP + 调用栈
                    │
                    ▼
           写入 perf ring buffer
                    │
                    ▼
           用户态 perf 工具读取
```

**关键点**：PMU 采样通过 NMI（不可屏蔽中断）实现，所以即使内核在持有自旋锁也能采样——这是 `perf` 能看到锁持有者的原因。

**perf stat vs perf record**：

| 命令 | 模式 | 适用场景 |
|------|------|---------|
| `perf stat` | 计数模式，全程累计 | 快速检查 IPC、缓存命中率 |
| `perf record -g` | 采样模式，生成 perf.data | 热点函数定位、火焰图 |
| `perf top` | 实时采样显示 | 类似 top 的 CPU 热点 |
| `perf trace` | syscall 追踪（底层用 tracepoint）| syscall 延迟分析 |

### 24.3.4 ftrace — 函数级追踪框架

**ftrace** 是内核内置的函数追踪框架，通过 `tracefs`（通常挂载在 `/sys/kernel/debug/tracing/`）暴露控制接口。

**核心机制 — function tracer**：

编译内核时加 `-pg` 选项（gcc 的 mcount 桩），每个函数入口都有一个 `mcount` 调用占位符。ftrace 在运行时将这些占位符替换为对 `ftrace_stub` 或实际 tracer 的调用（待核：具体实现细节，x86 使用 `fentry` 替代 `mcount`，近年内核）。

**ftrace 的多种 tracer**：

```bash
# 查看可用 tracer
cat /sys/kernel/debug/tracing/available_tracers
# 输出类似: blk function_graph function nop ...

# 函数调用图（追踪调用链）
echo function_graph > /sys/kernel/debug/tracing/current_tracer
# 只跟踪某个入口函数的子树（符号名随版本变，见下方坑点）
echo do_sys_openat2 > /sys/kernel/debug/tracing/set_graph_function
cat /sys/kernel/debug/tracing/trace
```

💡 **坑点（符号名会变）**：用 `set_graph_function` / kprobe 锚定函数名时，内核符号会随版本重命名。比如 open 路径的核心函数早年是 `do_sys_open`，较新内核改成了 `do_sys_openat2`；`sys_read` 这类也常被包成 `ksys_read` / `__x64_sys_read`。**永远先 `grep` 一下 `/proc/kallsyms` 或 `available_filter_functions` 确认当前内核到底有哪个名字**，不要照抄博客里的旧符号。

**function_graph 输出示例**（示意，缩进表示调用嵌套、DURATION 是该函数耗时）：
```
# CPU  DURATION                  FUNCTION CALLS
# |     |   |                     |   |   |   |
 0)               |  ksys_read() {
 0)               |    vfs_read() {
 0)   1.234 us    |      __vfs_read();        ← 叶子函数，直接给出耗时
 0)   2.567 us    |    }                      ← vfs_read 整体耗时（含子调用）
 0)   3.100 us    |  }                        ← ksys_read 整体耗时
```
读法：带 `{` 的是进入、带 `}` 的是返回，返回行上的 DURATION 是这一层（含其所有子调用）的总耗时——这正是 function_graph 相比纯 function tracer 的价值：能直接读出每层的"自带+累计"延迟。

**latency tracer**：`irqsoff`、`preemptoff`、`preemptirqsoff` tracer 专门追踪中断/抢占被关闭的最长时间区间，是排查实时性毛刺的利器（见第 18 章中断相关讨论）。

**工程推论**：
1. ftrace 的 `trace-cmd` 是 ftrace 的用户态包装器，比直接写 tracefs 文件友好得多。
2. `perf-ftrace` 子命令把 perf 和 ftrace 结合，共用 perf record 的数据路径。
3. 不需要重编内核，只要内核配置了 `CONFIG_FUNCTION_TRACER` 就能用。

### 24.3.5 eBPF 如何串联以上机制

eBPF 程序不是独立的观测通道——它是附着在 tracepoint/kprobe/perf_event 等**挂载点**上的可编程处理逻辑。

```
观测事件触发（tracepoint/kprobe/perf_event）
                │
                ▼
        eBPF 程序被调用
        （verifier 已保证安全）
                │
        ┌───────┴────────┐
        │                │
   BPF maps 写入     bpf_ringbuf 写入
   （聚合统计）      （流式数据）
        │                │
        ▼                ▼
   用户态读取         用户态消费
   (histograms等)    (实时流)
```

💡 **坑点**：eBPF 程序不能调用任意内核函数，只能调用 `bpf_helper` 函数白名单（近年通过 `kfunc` 扩展了可调用集合，但仍有限制）。这是 BPF verifier 保证安全的核心约束。

## 24.4 perf 采样与火焰图

### 火焰图的信息编码

Brendan Gregg 2011 年发明的火焰图（2013 年 LISA 大会演讲后广为流传，所以常被误记为 2013）是目前 CPU 热点可视化的最高密度格式：

```
┌─────────────────────────────────────────────────────────┐
│                         main                             │  ← 宽 = 出现在采样中的比例
│                         main                             │
├─────────────┬───────────────────────────────────────────┤
│  parse()    │              execute()                      │
├─────┬───────┴──────────┬────────────────────────────────┤
│hash │    lookup()      │           compute()             │
├─────┤                  ├──────────┬─────────────────────┤
│     │                  │  malloc  │    math_ops()        │
└─────┴──────────────────┴──────────┴─────────────────────┘
  x 轴 = 字母序（不是时间序！）
  y 轴 = 调用栈深度（底部是 main，顶部是叶子）
  宽度 = 采样命中该函数的比例
```

**新手误区**：x 轴不是时间轴！相邻的两个框可能是不同时刻的调用，只是字母序放在一起。想看时序必须用 `perf script` 原始数据。

### 完整 on-CPU 火焰图操作流程

```bash
# 1. 采样，带调用栈，采 30 秒
perf record -F 99 -g -- sleep 30
# 或采指定进程
perf record -F 99 -g -p <PID> -- sleep 30

# 2. 导出调用栈数据
perf script > out.perf

# 3. 折叠调用栈（FlameGraph 工具）
./stackcollapse-perf.pl out.perf > out.folded

# 4. 生成 SVG
./flamegraph.pl out.folded > flame.svg
```

**采样频率选择**：`-F 99` Hz 是经典值——取 99 而非 100，是为了**避开与整数频率的周期性活动（定时器、心跳）撞上同步（lockstep / aliasing）**，否则采样会系统性偏向或避开某些代码。99 Hz 对单进程而言开销基本可忽略（远低于 1%）；想要更细的分辨率（采短促突发）可上调到几百 Hz 甚至 997 Hz，开销随频率线性上升。系统全局 `-a` 采样、高频 + 深栈时开销才会明显，需按场景权衡。

### off-CPU 火焰图 — 看"等"在哪里

on-CPU 火焰图只看 CPU 上跑的时间，但高延迟常常是"等"出来的：等锁、等 IO、等调度。

**用 bpftrace 采集 off-CPU**（示意，需 root + 内核支持栈采集）：
```bash
# 思路：线程"离开 CPU"时记下时间戳，"回到 CPU"时算出离开了多久。
# 关键：用同一个 sched_switch 事件同时拿 prev（离开者）和 next（回归者）。
bpftrace -e '
tracepoint:sched:sched_switch {
    // 离开者：记下离开时刻（用 prev_pid 当 key）
    @ts[args->prev_pid] = nsecs;
    // 回归者：若之前记过，算出 off-CPU 时长，并按内核栈聚合
    if (@ts[args->next_pid]) {
        @offcpu[kstack] = hist(nsecs - @ts[args->next_pid]);
        delete(@ts[args->next_pid]);
    }
}'
```

💡 **坑点 1（这是真正容易写错的地方）**：**不要**用 `if (args->prev_state == 0)` 去过滤"只统计被抢占的情况"。`prev_state == 0` 即 `TASK_RUNNING`，意味着线程是被**非自愿抢占**离开 CPU 的；而 off-CPU 分析你通常恰恰想看的是**自愿睡眠**（等锁、等 IO、等 timer）——那些 `prev_state != 0`。加了 `== 0` 过滤反而把你最想看的等待全过滤掉了，这是初学者写 off-CPU 脚本最经典的反向 bug。Brendan Gregg 的 `offcputime` 工具默认**不按 state 过滤**，把所有离开 CPU 的时间都记下来，再用栈区分原因。

💡 **坑点 2（栈聚合 vs PID 聚合）**：按 `args->next_pid` 当 key 只能告诉你"哪个线程等得久"，告诉不了你"等在哪段代码"。生产里要按 `kstack`（内核栈）甚至 `kstack + ustack` 聚合，才能把 off-CPU 时间映射回具体的阻塞点（如 `futex` / `io_schedule` / `rwsem`），进而画 off-CPU 火焰图。

💡 **坑点 3（实现选择）**：`offcputime` 这类成熟工具底层常挂在 `finish_task_switch()`（kprobe）而不是 `sched:sched_switch` tracepoint 上，因为在那里能直接拿到 `prev` 的 `task_struct` 并安全读取上一个任务的栈。tracepoint 版本写起来更直观、字段更稳（`prev_pid`/`next_pid` 等由 `sched_switch/format` 保证），但要小心 in-flight 的负时长（脚本启动前就已睡眠的线程，第一次配对会算出异常值——用 filter 兜住，见 §代码题坑点 2）。

### 内存分配火焰图

```bash

bpftrace -e 'uprobe:/lib/x86_64-linux-gnu/libc.so.6:malloc { @[ustack] = count(); }'
```

## 24.5 USE/RED 方法论

### USE 方法（Brendan Gregg）

专门用于**资源型**（CPU、内存、磁盘、网络接口等）问题。每种资源检查三个维度：

| 维度 | 定义 | 常用指标 |
|------|------|---------|
| **U**tilization | 资源被使用的时间比例 | `mpstat`, `iostat %util` |
| **S**aturation | 资源超载排队的程度 | 运行队列长度, IO 等待队列 |
| **E**rrors | 错误次数 | dmesg 硬件错误, `/proc/net/dev` 丢包 |

**排查顺序**：Errors 先看（硬件故障不可忽视），再看 Saturation（排队代表真正超载），最后看 Utilization（高利用率不一定是问题）。

**实战检查表**：

```bash
# CPU
mpstat -P ALL 1          # 各核 utilization
vmstat 1 | awk '{print $1}' # 运行队列 (r列) = saturation
# 若 r > CPU 数，存在 CPU saturation

# 内存
free -m                   # utilization (used/total)
vmstat 1 | awk '{print $6}' # swap in (si列) > 0 = saturation
cat /proc/vmstat | grep pgmajfault  # major page fault = saturation

# 磁盘
iostat -xz 1              # %util ≈ 设备"忙"的时间比例, await = 每请求延迟
# avgqu-sz / aqu-sz（平均队列深度）> 1 倾向于 saturation
# ⚠ 现代盘的大坑：%util≈100% 在串行 HDD 上=饱和，但在 NVMe/SSD/RAID 上
#   设备能并行处理多个请求，%util=100% 只说明"任何时刻都在干活"，
#   远没到吞吐上限。SSD 上判断饱和要看 await 拐点 + 队列深度 + 实测 IOPS/带宽，
#   不能只看 %util。

# 网络
ip -s link                # RX/TX errors, drops
cat /proc/net/dev         # 丢包 = errors/saturation
```

### RED 方法（Tom Wilkie, Prometheus 生态）

专门用于**请求型服务**（微服务、API、数据库查询）：

| 维度 | 定义 |
|------|------|
| **R**ate | 每秒请求数 |
| **E**rrors | 每秒错误数（或错误率） |
| **D**uration | 请求延迟分布（p50/p95/p99） |

**USE vs RED 对比**：

```
问题类型                    推荐方法
─────────────────────────────────────────────────
"机器/OS 层面哪个资源饱和了"  → USE
"服务质量是否降级"           → RED
生产排障完整流程             → 先 USE 定资源，再 RED 定服务质量
```

**新手误区**：用 CPU utilization 单一指标判断系统是否健康——50% CPU 可能毫无问题，也可能因为锁争用导致 p99 爆炸；必须配合 saturation 和 latency 分布。

## 24.6 CPU/内存/IO/网络分层定位

### 24.6.1 CPU 性能问题定位

**分层分析思路**：

```
第 1 层：应用是否 CPU-bound？
  → perf stat: instructions / cycles → IPC
    （现代 x86 乱序核理论峰值 IPC 约 4~6，但典型整数代码平均才 ~1~2）
    IPC < ~0.5~1: 大量 stall，疑似缓存缺失/内存带宽瓶颈（memory-bound）
    IPC ~1~2:     常见的"健康"区间，多数服务端代码就在这
    IPC > ~3:     很高，通常是向量化/紧凑循环/缓存极友好的计算密集代码
  → 注意：IPC 是相对指标，没有放之四海的"好"阈值，
    要和该 workload 的历史基线比，不要拿绝对数轻易下结论。

第 2 层：哪个函数消耗 CPU？
  → perf record -g + 火焰图
  → 关注宽平台（采样占比高的叶子函数）

第 3 层：为什么慢？
  → perf stat: cache-misses/cache-references
    命中率 < 95%: 怀疑伪共享或内存访问模式差
  → perf stat: branch-misses
    > 5%: 条件密集路径可能受益于 branch hint 或重排

第 4 层：是否有 off-CPU 时间？
  → off-CPU 火焰图 / bpftrace sched_switch 分析
```

**CPU throttling（云环境特有）**：
```bash
# 检查容器 CPU throttle（cgroups v1）
cat /sys/fs/cgroup/cpu/docker/<id>/cpu.stat
# throttled_time 高 = cgroup CPU limit 被打满，不是代码问题
```

### 24.6.2 内存性能问题定位

**内存问题三条主线**：

```
主线 1: 内存带宽饱和
  症状: IPC 低，cache-misses 高
  工具: perf stat -e cache-misses,cache-references
  解法: 改数据访问局部性，NUMA 绑定

主线 2: page fault / 换页
  症状: major page fault 计数上升，IO 读增加
  工具: perf stat -e page-faults; vmstat si/so 列
  解法: 加内存，调 vm.swappiness

主线 3: 内存分配热点
  症状: 高频 malloc/free，slab 分配器压力
  工具: bpftrace 追踪 kmalloc，perf malloc 火焰图
  解法: 对象池化，减少小对象频繁分配
```

**NUMA 远访定位**（见第 8 章物理内存管理）：
```bash
# 检查 NUMA 节点布局
numactl --hardware

# 检查 NUMA 命中/缺失统计
numastat -p <pid>
# numa_miss 高 = 进程分配的内存落在远端 NUMA 节点

# 绑定进程到 NUMA 节点
numactl --cpunodebind=0 --membind=0 -- <command>
```

💡 **坑点**：`numastat` 的 `numa_miss` 不代表所有远访，只代表"尝试在本地分配但失败、退到远端"。通过 PMU 事件 `node-load-misses` 或 `mem-loads` 等可以更精确衡量（具体事件名因 CPU 型号而异，待核）。

### 24.6.3 IO 性能问题定位

**IO 延迟的三层分解**（见第 16 章块 IO 层）：

```
应用层延迟
    │
    ├─ 文件系统层延迟（VFS + FS 实现）
    │  工具: bpftrace ext4_file_operations 相关 kprobe
    │
    ├─ 块层延迟（请求队列、IO 调度器）
    │  工具: blktrace / biolatency (BCC)
    │        iostat -x: await = 提交到完成的平均时长（排队+服务，按请求计）
    │                   r_await / w_await: 读/写分开（新版本）
    │                   svctm: 已在 sysstat 12.0 移除（RHEL 9 起不再输出），
    │                          多队列设备上无法可靠计算，看到也别用
    │
    └─ 设备延迟（SCSI/NVMe 命令到完成）
       工具: nvme smart-log 或 SCSI 统计
```

**biolatency** 是 BCC 工具包中最常用的 IO 延迟直方图：
```bash
# 显示 IO 延迟分布（需要 BCC 安装）
sudo biolatency -D 10 1
# -D: 区分磁盘; 采样 10 秒显示一次
```

**IO 毛刺排查**：
```bash
# blktrace 捕获块层事件（高开销，短时间用）
blktrace -d /dev/nvme0n1 -o - | blkparse -i -
# 看 D→C 时间：D = dispatch to driver, C = completion
```

### 24.6.4 网络性能问题定位

**网络延迟的分层定位**（见第 20 章网络协议栈）：

```
应用层 send() → TCP 协议栈 → qdisc → 网卡驱动 → 网卡 → 链路

分层检查：
┌────────────────────────────────────────────────────────────┐
│ 1. 链路层：丢包/错误                                         │
│    ip -s link; cat /proc/net/dev                           │
├────────────────────────────────────────────────────────────┤
│ 2. TCP 协议栈：重传/拥塞                                     │
│    ss -ti; netstat -s | grep -i retran                     │
│    bpftrace -e 'tracepoint:tcp:tcp_retransmit_skb {...}'   │
├────────────────────────────────────────────────────────────┤
│ 3. 接收端队列：socket buffer                                 │
│    ss -tm (Recv-Q 非零 = 应用消费跟不上)                    │
│    net.core.rmem_max / tcp_rmem                            │
├────────────────────────────────────────────────────────────┤
│ 4. 发送端 qdisc 队列：                                       │
│    tc -s qdisc show dev eth0 (backlog/dropped)             │
├────────────────────────────────────────────────────────────┤
│ 5. 软中断 affinity（见下文）                                 │
└────────────────────────────────────────────────────────────┘
```

**软中断 affinity 问题**：所有网络包在一个核上处理软中断（`%si` 高），其他核闲置——典型的 NUMA 不对齐或 RSS（Receive Side Scaling）未配置。
```bash
# 查看软中断分布
cat /proc/softirqs
# NET_RX 列如果严重不均，需要配置 RSS/RPS

# 配置 RPS（软件 RSS）
echo "f" > /sys/class/net/eth0/queues/rx-0/rps_cpus
```

## 24.7 常见性能反模式

### 24.7.1 伪共享（False Sharing）

**机制**：两个不相关的变量恰好落在同一个 CPU cache line（通常 64 字节），导致一个核写入时让其他核的缓存行失效，强迫远端核重新从内存/LLC 加载。

```
Core 0                    Core 1
────────────────────────────────────────────────────────
struct stats {
  uint64_t counter_a;  ← Core 0 更新  │
  uint64_t counter_b;                 │ ← Core 1 更新
};  // 两者同在一个 64B cache line！

Core 0 写 counter_a → 整个 cache line 被标为"modified"
→ Core 1 的 counter_b 缓存行失效，需重新获取
→ 大量 cache coherence traffic，性能崩溃
```

**检测**：
```bash
perf stat -e cache-misses,LLC-load-misses -p <pid>
# LLC-load-misses 极高 + IPC 极低 → 怀疑伪共享

# perf c2c (cache-to-cache) — 专门检测伪共享（Linux 4.10+ 主线 perf）
# 核心信号是 HITM：load 命中了"另一个核 L1 里处于 modified 状态"的 cache line，
# 这正是多核写同一 cache line 互相 invalidate 的铁证。Remote HITM = 跨 NUMA。
perf c2c record -g -- sleep 10
perf c2c report   # 看 HITM 最高的 cache line 和争用它的代码地址/线程
```

**修复**：用 `__cacheline_aligned` 或 `__attribute__((aligned(64)))` 将热变量对齐到独立 cache line；或使用每 CPU 变量（`DEFINE_PER_CPU`）。

### 24.7.2 锁争用（Lock Contention）

**检测**：
```bash
# 用 bpftrace 追踪 mutex_lock 等待时间（示意）
bpftrace -e '
kprobe:mutex_lock { @start[tid] = nsecs; }
kretprobe:mutex_lock { 
  @wait[comm] = hist(nsecs - @start[tid]); 
  delete(@start[tid]); 
}'
```

**关键区分**：
- 自旋锁争用：CPU 空转（`perf stat` 看到 cycles 高但 instructions 低，IPC 极低）
- 互斥锁争用：进程 sleep（off-CPU 火焰图中看到，on-CPU 看不到）

**工程推论**：单纯看 CPU 利用率高，不代表有效工作——spinlock 争用也会拉高 CPU utilization，但 IPC 会崩溃。

### 24.7.3 NUMA 远访

已在 §24.6.2 详述。

**最常见场景（first-touch 陷阱）**：Linux 默认 NUMA 策略是 **first-touch**——一页物理内存被分配到"第一次写它的那个 CPU"所在的 NUMA 节点，而**不是** `malloc()` 调用时。经典翻车：主线程一次性 `malloc` + `memset` 初始化一大块内存（全落在主线程所在的节点 0），随后把这块内存分给跑在节点 1 的工作线程去算——于是工作线程对每次访问都是跨节点远访。修复思路是"谁用谁初始化"（让最终访问该内存的线程去 first-touch），或显式 `numactl --membind` / `mbind()` 绑定。

**工程推论**：超过 2 NUMA 节点的服务器（常见 4-socket 服务器），`lscpu` 查 NUMA 拓扑，`numastat` 监控，`numactl` 绑定，这三步是标准操作。

### 24.7.4 Page Cache 抖动

**机制**：大量 IO 密集进程的文件读取把 page cache 填满，导致其他进程的热页被逐出，下次访问触发 major page fault，IO 延迟急剧上升。

**检测**：
```bash
# 实时查看 page cache 大小变化
vmstat 1 | awk '{print $4}' # cache 列

# 检查 major page fault
perf stat -e major-faults -p <pid>

# 查看具体哪些文件占用 page cache
# fincore (需要安装) 或 /proc/<pid>/smaps
```

**修复方向**：
- 大文件顺序 IO 使用 `MADV_SEQUENTIAL` 或 O_DIRECT，避免污染 page cache
- 调整 `vm.dirty_ratio` / `vm.dirty_background_ratio` 控制写回节奏
- 对不需要 cache 的冷数据 IO 用 fadvise 提示 `FADV_DONTNEED`

### 24.7.5 调度延迟与软中断风暴

```bash
# 检查软中断执行时间（BCC 工具）
hardirqs    # 硬中断延迟直方图
softirqs    # 软中断执行时间直方图

# 检查调度延迟（从 wakeup 到实际运行）

```

## 24.8 延迟分析与尾延迟

### 为什么均值有毒

```
请求延迟样本: [1ms, 2ms, 1ms, 2ms, 1ms, 500ms]
均值 = 84ms  ← 没有一个用户体验到 84ms
p50 = 1.5ms  ← 中位数用户体验
p99 = 500ms  ← 最差 1% 用户的体验
```

**工程铁律**：服务 SLA 必须用 p99/p999 而不是均值。

**尾延迟放大（tail at scale）——把数算对**：这里有个被讲得稀烂、十有八九讲错的点。正确的版本来自 Dean & Barroso, *The Tail at Scale*（Google, 2013）：

```
设一次用户请求并行扇出 N 个后端，每个后端"变慢"（超过其 p99）的概率独立为 1%。
整个请求只要有一个后端慢就慢（要等最慢那个返回）。
P(整体不慢) = 0.99^N
P(整体慢)   = 1 − 0.99^N

N=1   →  1%      （单个后端，p99 名副其实）
N=100 →  1 − 0.99^100 ≈ 63%   ← 单后端的 1% 尾事件，被放大成 63% 的请求受影响！
N=200 →  ≈ 87%
```

**结论**：扇出度越高，单后端那个"罕见的 1%"就越接近"几乎每次请求都会撞上一次"。所以高扇出服务里，**单后端的 p99 决定整体请求的 p50 量级**——这就是为什么大厂死磕后端尾延迟，并用 hedged request（延迟一点点后补发副本、取先返回者）来砍尾延迟。

⚠️ 别说成"整体最差情况是 10ms"：p99=10ms 只意味着 1% 的请求超过 10ms，最坏情况是**无界**的（取决于分布的尾巴有多长），这正是要看 p999/p9999 和完整直方图、而不是只盯一个分位点的原因。

### 延迟直方图与 heatmap

```bash
# bpftrace 直方图示例（追踪 read 系统调用延迟）
bpftrace -e '
tracepoint:syscalls:sys_enter_read { @ts[tid] = nsecs; }
tracepoint:syscalls:sys_exit_read  {
  if (@ts[tid]) {
    @latency = hist(nsecs - @ts[tid]);
    delete(@ts[tid]);
  }
}'
```

**分位数计算与 HdrHistogram**：生产服务应该使用 HdrHistogram 或类似结构记录延迟——它在 O(1) 空间内保持高精度分位数，比存储所有原始数据实际得多。

### 尾延迟的常见根因地图

```
p99/p999 毛刺
│
├── 锁争用（互斥锁）：off-CPU 火焰图有宽平台
├── 调度延迟：runqlat 直方图有长尾
│   ├── CPU 超卖 / 高负载
│   └── 低优先级任务抢占（设置 SCHED_FIFO / nice 优化）
├── NUMA 远访：numastat miss 高，周期性触发
├── Page cache 逐出：major faults 周期性出现
├── GC 停顿（应用层，但表现为 off-CPU 毛刺）
├── 网络 retransmit：ss -ti 看 retransmits
└── 磁盘 IO 队列满：biolatency 有双峰分布
```

🎯 **面试加分点**：被问"如何定位 p99 延迟毛刺"时，标准答题框架：先区分 on-CPU vs off-CPU（决定看火焰图还是调度延迟），再根据方向用专项工具下钻，最后用 USE 检查资源饱和度。

## 24.9 系统化排障方法论

### 排障五步法

```
1. 量化问题（Problem Quantification）
   → 什么指标超标？超标多少？从什么时候开始？
   → "感觉慢"不是问题，"p99 从 5ms 升到 50ms，从昨天 14:30 开始"才是

2. 圈定资源（Resource Identification）
   → USE 方法扫所有资源层
   → 找到哪个资源 Utilization 或 Saturation 异常

3. 定位热点（Hotspot Localization）
   → on-CPU 火焰图：哪个函数消耗 CPU
   → off-CPU 分析：哪里在等待

4. 根因假设（Root Cause Hypothesis）
   → 基于数据提出可验证假设
   → "怀疑是锁争用，因为 off-CPU 火焰图在 mutex_lock 有 30% 宽度"

5. 验证与修复（Validate & Fix）
   → 可控复现：能复现才能验证修复
   → A/B 测试：修复前后对比 p99 分布
```

### 工具选择决策树

```
问题是什么类型？
│
├─ CPU 高 → 先 perf top 看哪个进程，再 perf record 看哪个函数
│
├─ 延迟高 → 先区分是 CPU-bound 还是 IO/等待型
│           CPU-bound: 火焰图定位函数
│           等待型: off-CPU 分析 / runqlat
│
├─ 内存高 → free/vmstat 看是 cache 还是 anon，再看是否换页
│
├─ IO 高 → iostat -x 看哪个设备，biolatency 看延迟分布
│
├─ 网络问题 → ss -s 看连接统计，perf trace 看 syscall
│
└─ 毛刺/尾延迟 → 先 runqlat 排调度，再 biolatency 排 IO，
                  再 off-CPU 排锁争用
```

### 工具开销速查表

| 工具 | 开销量级 | 生产可用 | 适用场景 |
|------|---------|---------|---------|
| `perf stat` | <1% | 是 | 快速硬件计数器 |
| `perf top/record -F 99` | 1-5% | 是 | CPU 热点采样 |
| `bpftrace` 计数类 | 1-5% | 是 | 自定义聚合统计 |
| `bpftrace` 流式类 | 10-30% | 短时间 | 调试 |
| `ftrace function_graph` | 高 | 谨慎 | 开发环境 |
| `strace` | 10-100x | 否 | 仅调试环境 |
| `blktrace` | 中-高 | 短时间 | IO 详细路径 |

## 24.10 当前实现要点与未来演进

### eBPF 生态的成熟化（近年内核，2019-2024）

- **BTF + CO-RE**：解决了 eBPF 程序的可移植性问题——一次编译，可以在不同内核版本运行，不需要每台机器安装内核头文件。
- **BPF ring buffer**（5.8+）：取代旧的 perf ring buffer 作为 BPF 到用户态的数据通道，支持多消费者、减少数据复制。
- **BPF kfunc**：允许 BPF 程序调用更多内核内部函数（经白名单审查），扩大了可编程范围。
- **Cilium/Falco/Pixie 等**：eBPF 从观测工具扩展到网络数据平面和安全执行层，是基础设施软件的趋势。

### Continuous Profiling（持续剖析）

传统 profiling 是"出了问题再抓"，现代大厂方向是持续低开销采样（通常 `perf record -F 19` 极低频率）并保留历史数据，问题发生后回溯剖析。代表产品：Parca、Pyroscope、Google Wide Profiling。

### DAMON（Data Access Monitor，近年内核）

内核内置的内存访问模式监控框架，可以自动识别冷热内存区域并驱动内存回收/迁移策略，是 NUMA 内存管理的智能化方向。

## 24.11 章末五件套

### 一、高频面试题（Staff 级）

**Q1：解释 perf record 的采样工作原理，为什么它相对低开销？**

答题要点：
- PMU 硬件计数器计到 N 时触发 NMI，NMI handler 记录当前 IP + 调用栈
- 采样是统计近似，不是全量追踪，所以开销正比于采样频率而非被测程序行为
- 错误答案陷阱："perf 用 ptrace 跟踪"——ptrace 是 strace 的机制，会停止进程，和 perf 的 NMI 采样完全不同

**Q2：tracepoint 和 kprobe 的区别，什么场景选哪个？**

答题要点：
- tracepoint：静态预埋，ABI 稳定，开销小（nop），有就优先选
- kprobe：动态挂任意函数，但函数签名/位置可变，内核版本敏感
- 决策准则：先查 `/sys/kernel/debug/tracing/events/` 有没有对应 tracepoint；没有才用 kprobe

**Q3：一个服务的 p99 延迟周期性毛刺，如何系统化定位？**

答题框架：
1. 确认是 on-CPU 毛刺还是 off-CPU（等待）：off-CPU 用 sched_switch + bpftrace 追踪
2. 如果 off-CPU：runqlat 排调度延迟，biolatency 排 IO，mutex/rwlock kprobe 排锁
3. 如果 on-CPU：perf record 取毛刺期间 CPU 采样，火焰图定位
4. 用 USE 检查资源 saturation，排 NUMA 远访（numastat）、page fault（vmstat major）

**Q4：伪共享如何检测和修复？**

答题要点：
- 症状：高 LLC 缓存缺失，低 IPC，多核并发时性能不如单核（加核反而变慢是强信号）
- 检测：`perf c2c record/report`（Linux 4.10+），看 HITM / remote HITM；粗筛可用 `perf stat -e LLC-load-misses`
- 修复：`__cacheline_aligned` 属性 / 显式 padding 到 64B，或改用 per-CPU 变量
- 加分：举例说明结构体 padding 的工程实践；点出 64B 是常见值但非绝对——某些架构/预取器按 128B 配对（如部分 POWER、Apple Silicon 的 128B line），跨平台代码别把 64 写死

**Q5：USE 方法和 RED 方法有什么区别，如何配合用？**

答题要点：
- USE：资源维度（Utilization/Saturation/Errors），查基础设施层
- RED：请求维度（Rate/Errors/Duration），查服务质量层
- 配合：USE 定位哪个资源饱和，RED 定位哪个服务受影响，两者相互佐证
- 错误答案：说 RED 取代了 USE——它们层次不同，不是竞争关系

**Q6：火焰图的 x 轴代表什么，为什么这是常见误区？**

答题要点：
- x 轴是字母排序，不是时间轴
- 相邻两个框可能是完全不同时刻的调用
- 宽度代表该函数在采样中出现的比例（CPU 时间占比）
- 如果想看时序，要用 `perf script` 原始数据或 time-based flamegraph

**Q7：eBPF 程序为什么不会崩溃内核？**

答题要点：
- BPF verifier 在加载时静态验证程序：无循环（或有界循环），无越界内存访问，无调用任意函数（只能调用 helper 白名单）
- JIT 编译执行，不是解释执行，所以运行时性能接近原生代码
- 如果 verifier 拒绝，程序根本加载不进去
- 加分：说明 CO-RE 如何解决跨内核版本内存布局差异

**Q8：为什么 strace 会让被追踪程序慢 10-100 倍？**

答题要点：
- strace 通过 ptrace 追踪，每个 syscall 的 entry/exit 各停一次，**每个 syscall 约 2 次上下文切换**到 tracer 进程，量级约 10–100µs/call
- 对于 syscall 密集的程序（如大量小 IO），这个常数开销被放大到 10–100x（注意：这是 syscall-bound 工作负载的量级，CPU-bound 程序受影响小得多，别一概而论）
- 替代方案：`perf trace`（基于 tracepoint，低开销）或 bpftrace 追踪 syscall 事件
- **加分（说出这个直接区分新手）**：strace 自 v5.3 起有 `--seccomp-bpf`，用 seccomp BPF 过滤器让内核只在"你关心的那几个 syscall"上停下来，其余 syscall 不再陷入 tracer——把"每个 syscall 都停"降为"按需停"，对只关心少数 syscall 的场景能大幅降低开销。这说明真正的瓶颈是**停-切-查-继续的每次往返**，而非 ptrace 本身的存在。

### 二、实战项目 P2：完整性能剖析

**项目目标**：对一个真实负载（可以是本机的 web server、数据库，或刻意构造的压测程序）做完整一轮性能剖析，覆盖 USE → on-CPU 分析 → off-CPU 分析 → 排障假设。

**增量任务**（基于 P0/P1 的基础）：

```
阶段 1：构造负载
  - 用 wrk/ab 对一个本地 HTTP server 施加压力
  - 或用一个带锁争用的多线程 CPU 密集程序

阶段 2：USE 快照
  - 记录 CPU utilization/saturation（mpstat, vmstat）
  - 记录内存 utilization（free, vmstat）
  - 记录 IO utilization（iostat -x）
  - 产出：哪个资源存在 saturation

阶段 3：on-CPU 火焰图
  - perf record -F 99 -g -p <pid> -- sleep 10
  - 生成 SVG 火焰图
  - 产出：CPU 热点函数是什么

阶段 4：off-CPU 分析
  - bpftrace 追踪 sched_switch，统计进程 off-CPU 时间
  - 产出：进程有多少时间在等待，等什么

阶段 5：延迟分布
  - 如果是网络服务：bpftrace 追踪 syscall 延迟直方图
  - 产出：p50/p95/p99 延迟和分布形状

阶段 6：假设与验证
  - 写出 3 个根因假设（至少一个被数据支持）
  - 验证方法：修改代码/配置后对比 p99 前后变化
```

**验收标准**：
- 能输出一张可解释的 on-CPU 火焰图（说明各大平台含义）
- 能输出一张 off-CPU 分析结果（说明主要等待原因）
- USE 检查表有数字，有结论
- 至少一个根因假设有数据支持

### 四、系统题

**排障场景**：一个在线搜索服务，正常情况 p99 = 20ms，但每隔约 5 分钟会出现一个持续 2-3 秒的"毛刺窗口"，期间 p99 飙升到 500ms。服务部署在 2 NUMA 节点的 32 核机器上，使用本地 SSD 存储倒排索引，业务逻辑为纯计算（无外部 IO 等待）。

**排障推导**：
```
步骤 1：确认规律性
  → 5 分钟周期性 → 联想 cron job、GC、checkpoint 等定时操作
  
步骤 2：排查 on-CPU vs off-CPU
  → 毛刺期间 CPU utilization 是否上升？
  → 若 CPU 低但延迟高：off-CPU 等待型
  → 若 CPU 正常但 p99 高：少量请求被阻塞

步骤 3：检查定时任务
  → cat /etc/cron.d/*; systemctl list-timers
  → 是否有每 5 分钟的备份/索引重建/GC 触发

步骤 4：如果 GC（Java/Go 等运行时）
  → 查运行时 GC 日志（-verbose:gc 或 GODEBUG=gctrace=1）
  → off-CPU 火焰图会在 GC stop-the-world 期间显示整个进程 blocked

步骤 5：如果不是 GC
  → biolatency：每 5 分钟是否有 IO 毛刺（checkpoint flush？）
  → runqlat：是否有调度延迟峰值（批处理任务抢 CPU？）
  → numastat：NUMA miss 是否周期性升高（大批量内存访问触发 NUMA 远访？）

量级估算：
  → 2-3 秒毛刺，正常 p99 20ms，毛刺系数 25x
  → 如果是 GC Stop-the-world：典型 Java/Go GC 暂停 10-500ms，25x 延迟增幅符合
  → 如果是 NUMA 远访：单次远访 ~100ns，25x 需要非常密集才能累积到 2-3 秒，更可能是多重因素
```

### 五、代码题

**题目**：写一个 bpftrace 单行程序，统计所有进程 `read` 系统调用的延迟直方图（按进程名分组），运行 10 秒后退出，输出对数直方图。

```bash
#!/usr/bin/env bpftrace
// 运行方式: sudo bpftrace read_latency.bt
// 或单行: sudo bpftrace -e '...' 见下

// 文件版本（read_latency.bt）
tracepoint:syscalls:sys_enter_read
{
    @start[tid] = nsecs;
}

tracepoint:syscalls:sys_exit_read
/@start[tid]/
{
    @latency_ns[comm] = hist(nsecs - @start[tid]);
    delete(@start[tid]);
}

interval:s:10
{
    print(@latency_ns);
    clear(@latency_ns);
    exit();
}
```

**验证方法**：
```bash
# 安装 bpftrace (Ubuntu)
sudo apt-get install bpftrace

# 同时在另一个终端运行 IO 负载
dd if=/dev/zero of=/dev/null bs=1k count=100000

# 运行 bpftrace 脚本
sudo bpftrace read_latency.bt
```

**预期输出**（示意）：
```
@latency_ns[dd]:
[256, 512)             42 |@@@@@                     |
[512, 1k)            187 |@@@@@@@@@@@@@@@@@@@@@@    |
[1k, 2k)             312 |@@@@@@@@@@@@@@@@@@@@@@@@@@|
[2k, 4k)              89 |@@@@@@@@@@                |
...
```

**暗坑提示**：
1. `tid` 是线程 ID，`pid` 是进程 ID，`comm` 是进程名（15 字节限制）——在多线程程序中要用 `tid` 匹配 start/end，不能用 `pid`。
2. `/@start[tid]/` 是过滤器（filter），防止在 bpftrace 开始之前已经 in-flight 的 read 调用产生负延迟（nsecs 溢出）——这是常见 bug。
3. 如果 read 延迟分布呈**双峰**（大量很快 + 少量很慢），通常意味着 page cache 命中/缺失两种情况并存，是进一步排查的线索。
