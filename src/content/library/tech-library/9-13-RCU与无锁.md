> **一句话定位**：RCU（Read-Copy-Update）是 Linux 内核中读侧（几乎）零开销、写时复制、延迟回收的同步机制——它把锁的代价从"每次读"转移到"偶发写 + 时间轴上的宽限期"，是大量读极少写、且旧数据可延迟释放场景的最优解。

> **本版与原版的差别**：原版概念正确但"悬空"。本版给每个核心机制附上 **Linux v6.12 真实源码逐行注解**、补上 **2002→今的设计考古（git/LWN/社区争论）**、用 **具体链表场景把 rwlock / seqlock / RCU 各跑一遍**、并讲透 4 个"扎根问题"：读侧凭什么免费、grace period 凭什么安全、内存序为何必须、失败模式长什么样。所有逐字源码标【真实源码 v6.12 路径】；为讲清而简化的标【示意代码，非逐字】；没把握处标「待核」。

## TL;DR · 本章最硬结论

1. **RCU 读侧"零开销"不是修辞，是源码事实**：在 non-preemptible 内核里，`__rcu_read_lock()` 的函数体**逐字就是一行 `preempt_disable()`**（见 §3.1 真实源码）；在 PREEMPT_RCU 里，它逐字是"对 `current->rcu_read_lock_nesting` 这个**每任务私有**计数器 +1"（§3.2）——两者都**不触碰任何跨 CPU 共享的 cache line**，这是它比 `read_lock()` 快一两个数量级的微架构根因。
2. **宽限期（grace period）是 RCU 的灵魂，本质是"等待"而非"引用计数"**：写者改完指针，必须等所有"改之前就已进入读临界区"的 reader 都退出，旧数据才可回收。McKenney 给的定义是 *"a way of waiting for pre-existing RCU read-side critical sections to completely finish"*（LWN 2007）。代价形态因此和 refcount 完全不同——见 §3.4 / §4。
3. **内存屏障不可省，但 RCU 走的是比 acquire 更弱的"依赖序"**：写者 `rcu_assign_pointer` 是 release store；读者 `rcu_dereference` 靠 **address-dependency ordering**（不是 `smp_load_acquire`）。区分这两者是 staff 与高级的分水岭（§3.5）。
4. **`rcu_dereference` 永远不是 no-op 的真正原因是 `READ_ONCE`，不是硬件 barrier**：v5.9 起内核已删掉 Alpha 的依赖屏障特殊处理，但 `READ_ONCE`（防编译器缓存指针、防 load tearing）在所有架构都留着（§3.5、设计考古 §2.6）。
5. **RCU flavor 在 v4.20 被合并，且起因是一个"可被利用的 bug"**：McKenney 原话——consolidation *"was carried out in response to an exploitable bug resulting from confusion between two flavors"*（LWN 2019）。写新代码只需 `synchronize_rcu` / `call_rcu`（§2.5）。
6. **RCU 不是万能锁**：它给你"安全解引用一个 old-or-new 指针"，不给"读到最新值"、不给"跨对象一致快照"、不给"可变对象的字段同步"（§3.3 反面清单）。

## 1 · 问题：RCU 解决什么

### 1.1 读写锁的隐痛：reader 之间也在抢

设想一个路由表：每秒被查询数百万次，被修改不超过几次。用 `rwlock`：

```
读者：read_lock → 原子修改 reader_count（一个共享 cache line）→ 读数据 → read_unlock
```

问题不在数据，在 **`reader_count` 是所有 CPU 争抢的共享变量**。100 个核同时持读锁，这条 cache line 在所有核之间 ping-pong，MESI 协议把"明明没有数据竞争的并发读"变成一场 cache line 争夺战。

这叫 **reader-reader 竞争**（读者之间在锁元数据上互相拖累）。它是读写锁在高并发读场景的硬天花板：**读越多越慢，线性退化**。

### 1.2 RCU 的洞察：读者根本不需要全局协调

McKenney 在 *"What is RCU, Fundamentally?"*（LWN 2007，下文多次引用）里把 RCU 和 rwlock 的本质差异讲得很干净：

> *"reader-writer locking permits concurrent reads ... RCU supports concurrency between a single updater and multiple readers."*

注意措辞：rwlock 允许"多读并发"，而 RCU 允许"**更新进行中**读仍可继续"——读和写可以真正同时跑，而不是写者一来读者全部被挡。它靠两条不变式做到：

1. **写者用"复制 + 原子换指针"发布新版本**，旧版本**不立即释放**。
2. **所有"在换指针之前就开始读"的读者，读完后系统才回收旧版本**。

读者因此**不修改任何共享状态**，只需保证"我手里的指针在我读的过程中不被释放"。这个保证靠"禁抢占 / 关中断"（让该 CPU 在我读完前不经历调度切换）实现，**完全不需要锁**。

```
没有 RCU：读者 ←→ 写者 ←→ 读者   （三方在 lock metadata 上竞争）
有  RCU： 读者直接读；写者复制后原子换指针；旧版本延迟到宽限期后回收
```

**工程推论**：

1. RCU 把代价从"每次读的互斥"挪到"写后的宽限期等待"。对"读 99.9% / 写 0.1%"的场景，总代价下降一到两个数量级——而且**读越多优势越大**（读侧不退化）。
2. 宽限期的存在意味着 RCU **不适合"写者需要立刻知道旧版本已被释放"** 的场景（如需要立刻复用内存的 allocator 内部结构）。这是 RCU 的固有 trade-off，不是缺陷。

## 2 · 设计考古：RCU 在 Linux 的真实历史与争论

> 这一节是本版相对原版最大的补强：不只给"哪年合入"，而是给 **为什么演进、社区争了什么、史料出处**。所有锚点尽量带 URL。

### 2.1 起源：2.5.43 / 2002 年 10 月

RCU 由 **IBM 的 Dipankar Sarma 与 Paul E. McKenney** 在 **Linux 2.5.43（2002 年 10 月）** 合入主线。McKenney & Walpole 的论文措辞是 *"RCU added to the Linux kernel in October of 2002"*（LWN/Articles/262464，2007-12-17）。

但 RCU 的**思想**更早：McKenney 的工作可追溯到 1990 年代 Sequent 的 DYNIX/ptx（彼时叫 "rclock"），更早的同族机理（Kung & Lehman 1980、VM/XA 的 passive serialization）也常被他在综述里点名。所以 RCU 在 Linux 是"2002 落地"，但作为一类**延迟回收 / epoch-based** 方法，它有十几年的史前史。

**它要替掉 rwlock 的什么**：见 §1。一句话——干掉 reader-reader 在锁元数据上的 cache 争用。

**史料锚点**：
- LWN，*What is RCU, Fundamentally?*（part 1）：https://lwn.net/Articles/262464/
- LWN，*What is RCU? Part 2: Usage*：https://lwn.net/Articles/263130/
- LWN，*RCU part 3: the RCU API*：https://lwn.net/Articles/264090/
- Linux Journal，*Using RCU in the Linux 2.5 Kernel*（McKenney，一手）：https://www.linuxjournal.com/article/6993
- 综述论文，*RCU Usage In the Linux Kernel: Eighteen Years Later*（ACM SIGOPS OSR 2020）：https://dl.acm.org/doi/10.1145/3421473.3421481

### 2.2 发布-订阅与"读到半成品"问题（2007 的系统化阐述）

McKenney 在 2007 的系列里把 RCU 拆成三件套：**Publish-Subscribe**（`rcu_assign_pointer` / `rcu_dereference`）、**Wait-for-pre-existing-readers**（grace period）、**Maintain-multiple-versions**（旧新版本并存）。

发布订阅这一对是为了堵住一个具体竞态：写者"先填字段、再发布指针"，但**CPU/编译器可能重排**，读者可能拿到指针却读到未初始化的字段。`rcu_assign_pointer` 保证 *"the assignment to gp after the assignments to the fields"*；`rcu_dereference` 在订阅侧配对——尤其要对付 **DEC Alpha** 的诡异（§2.6、§3.5）。

### 2.3 经典 RCU 的可扩展性瓶颈 → Tree RCU（2.6.29 / 2008）

经典 RCU 的宽限期检测依赖一个**全局锁/全局状态**，每个 CPU 每个宽限期至少要碰它一次。McKenney 在 *Hierarchical RCU*（LWN，2008-11-04）里点破：

> *"their scalability is limited by a global lock that must be acquired by each CPU at least once during each grace period."*

CPU 数一上去（论文目标是优雅支持 1024、最好 4096 个 CPU），这个全局点就成了热点。解法是把 quiescent-state 上报组织成一棵 **`rcu_node` 树**，每个节点各自带锁，层层聚合：

> 对 4096-CPU、64 个 leaf node 的系统，单锁争用从"全部 CPU 抢 1 把锁"降到"每 node 64 个 CPU"——64x 改善。

这就是把宽限期检测从 **O(NR_CPUS) 竞争降到 O(log NR_CPUS)**。Tree RCU 于 **2.6.29（2009 年初）** 成为主线实现，沿用至今。论文还强调一个常被忽略的动机：**省电**——别在宽限期里无谓唤醒在睡的 CPU（NO_HZ 相关，§2.7）。

**史料锚点**：LWN *Hierarchical RCU*：https://lwn.net/Articles/305782/

### 2.4 可睡眠 SRCU（约 2006 / 2.6.19 前后）

经典 RCU **禁止在读临界区里睡眠**，McKenney 给的理由很直接（LWN，2006-10-09）：

> *"sleeping implies a context switch, which in Classic RCU is a quiescent state."*

也就是说，你一睡，调度器一切换，RCU 就**误判你已退出临界区**——这会导致 use-after-free。更糟的是若允许任意睡眠，宽限期会被**无限拉长**，*"could result in arbitrarily large amounts of memory awaiting the end of a grace period"*。

SRCU 的两个折中：

1. **取消异步 API**（只有 `synchronize_srcu`，没有 `call_srcu` 的早期版本），限制每线程最多一块内存在等。
2. **每子系统独立宽限期**（per-`srcu_struct`），一个坏 reader 拖死的只是它自己那一摊，不连累全系统 RCU 回调。

代价：读侧不再是零开销——每次 `srcu_read_lock` 要**写 per-CPU 计数器**（一对 `c[2]`，按宽限期低位索引）。这是"允许睡眠"的明码标价。

**史料锚点**：LWN *Sleepable RCU*：https://lwn.net/Articles/202847/

### 2.5 RCU flavor 合并（4.20 / 2018，且因一个"可利用 bug"）

一个旧资料常漏的关键变化：**Linux 4.20（2018 年底）把 RCU-bh、RCU-sched、RCU-preempt 三种 flavor 合并成一个**。McKenney 在 *The RCU API, 2019 edition*（LWN，2019-01-23）里点明动因——不是单纯"为优雅"，而是：

> *"The big change is the consolidation of the RCU bh, RCU preempt, and RCU sched flavors, which was carried out in response to an exploitable bug resulting from confusion between two flavors."*

合并后，`CONFIG_PREEMPT=y` 内核里 vanilla 的 `synchronize_rcu()` **一次性覆盖** *"bh-disable, irq-disable, and preempt-disable regions of code in addition to the traditional `rcu_read_lock()`-style ... critical sections."* 老 API（`synchronize_sched`、`synchronize_rcu_bh`、`call_rcu_bh` …）变别名后陆续删除。

这一点也写进了 v6.12 `rcu_read_lock()` 的源码注释里（§3.1 会看到原文）：*"In v5.0 and later kernels, synchronize_rcu() and call_rcu() also wait for regions of code with preemption disabled ..."*——注意注释说的是 **v5.0 起对外语义稳定**，而 flavor 物理合并发生在 4.19/4.20 窗口。

**结论**：理解机制时 sched/bh flavor 仍有价值（它们解释了"为什么关抢占/关下半部也算 quiescent"），但**写新代码只用** `synchronize_rcu` / `call_rcu`。4.20 之后仍独立的 flavor 只有 **SRCU** 和 **Tasks RCU**。

```
普通 RCU      → 读侧禁抢占；4.20 后已吸收 sched/bh 语义（写新代码就用它）
RCU-sched     → [历史] 关抢占即算静止；4.20 并入普通 RCU
RCU-bh        → [历史] 关下半部即算静止；4.20 并入普通 RCU
SRCU          → 可睡眠读临界区，per-srcu_struct 宽限期（仍独立）
Tasks RCU     → 基于"每任务自愿调度过一次"，tracing/ftrace 专用（仍独立）
```

**史料锚点**：LWN *The RCU API, 2019 edition*：https://lwn.net/Articles/777036/

### 2.6 Alpha 依赖屏障的兴亡（4.15 加、5.9 删）

DEC Alpha 是内存序的"奇葩"：因为 split cache，它**可能违反地址依赖**——即"先读到指针 `p`、再用 `p` 去读 `p->field`"，Alpha 居然可能让 `p->field` 的读"看见旧值"。所以历史上 `rcu_dereference` / `READ_ONCE` 在 Alpha 上要插显式 `smp_read_barrier_depends()`。

演进两步：
- **v4.15**：把这个屏障**收进 `READ_ONCE()` 内部**（Alpha 上 `READ_ONCE` 自带一个 `smp_mb()`），使绝大多数代码不必再手写依赖屏障。内核文档原话：*"As of v4.15 ... an smp_mb() was added to READ_ONCE() for DEC Alpha."*
- **v5.9**：Will Deacon 的系列**彻底删除** `smp_read_barrier_depends()` 这个宏，依赖序语义"内嵌"进所有 marked access（`READ_ONCE`/`rcu_dereference`）。内核文档把这一整节标为 **HISTORICAL**。

**这对你的实战含义**：今天写 RCU 读侧，**永远别手写依赖屏障**——`rcu_dereference` 已经把该做的都做了。但要记住"它仍不是 no-op"的根因是 `READ_ONCE`（防编译器，下一节源码可见），不是硬件 fence。

**史料锚点**：
- 内核内存屏障文档（含 HISTORICAL 节）：https://www.kernel.org/doc/Documentation/memory-barriers.txt
- Will Deacon 删除 patch（lore）：https://lore.kernel.org/lkml/20200710165203.31284-6-will@kernel.org/

### 2.7 实时 / NO_HZ 适配（贯穿 2.6.late → 至今）

RCU 的宽限期推进依赖"CPU 周期性经历 quiescent state"。但两类场景打破这个前提：
- **NO_HZ_FULL / nohz_full 的 CPU**：为降低抖动，跑用户态时**关掉调度时钟 tick**——没了 tick，靠谁来报 quiescent？RCU 要专门处理这些 CPU（启动时 `nohz_full=` 指定，回调常配合 NOCB 卸载，§8.1）。
- **PREEMPT_RT 实时内核**：读临界区可被抢占、被抢占的 reader 迟迟不退会拖死宽限期，于是引入 **RCU priority boosting**（临时拉高被抢占 reader 的优先级让它尽快跑完，§4.3）。

社区对 RCU 的一个长期讨论正是**复杂度 vs 可用性**：RCU 的核心算法被反复用 **形式化验证 / 模型检测**（Linux-Kernel Memory Model、CBMC、`tools/memory-model`）盯着，因为它"读侧太聪明所以太容易写错"。这也是为什么 `CONFIG_PROVE_RCU`（lockdep）和 `rcutorture` 是内核 CI 的标配（§6）。

## 3 · 机制源码精读（v6.12 逐行注解）

> 本节所有逐字代码均取自 **Linux v6.12** 真实源码（WebFetch 自 `raw.githubusercontent.com/torvalds/linux/v6.12/...`），标【真实源码 v6.12 路径】。注释为本版所加，标在 `// ←` 处。

### 3.1 读侧（non-preemptible）：`__rcu_read_lock()` 逐字就是 `preempt_disable()`

这是全章最重要的一处源码——它**直接证明**"读侧零开销"不是修辞。

【真实源码 v6.12 `include/linux/rcupdate.h`】（`#else`，即 `!CONFIG_PREEMPT_RCU` 分支）：

```c
#else /* #ifdef CONFIG_PREEMPT_RCU */

#ifdef CONFIG_TINY_RCU
#define rcu_read_unlock_strict() do { } while (0)
#else
void rcu_read_unlock_strict(void);
#endif

static inline void __rcu_read_lock(void)
{
	preempt_disable();          // ← 读侧"加锁"的全部：仅禁抢占，零跨 CPU 写
}

static inline void __rcu_read_unlock(void)
{
	preempt_enable();           // ← "解锁"：恢复抢占
	if (IS_ENABLED(CONFIG_RCU_STRICT_GRACE_PERIOD))
		rcu_read_unlock_strict();   // ← 仅 debug 内核：强一致 GP 自检
}

static inline int rcu_preempt_depth(void)
{
	return 0;
}

#endif /* #else #ifdef CONFIG_PREEMPT_RCU */
```

**逐行要点**：
- `__rcu_read_lock` 体内**没有任何原子指令、没有任何对共享变量的写**。`preempt_disable()` 在非抢占内核里通常只是给 per-CPU 的 `preempt_count` +1（per-CPU = 不跨核竞争），甚至在 `!CONFIG_PREEMPTION` 下可被编译成几乎空操作。**这就是"读侧零开销"的源码级证据**。
- 它如何保证安全？关抢占 ⇒ 本 CPU 在 unlock 前**不会经历上下文切换** ⇒ "该 CPU 经历过一次 schedule" 就**等价于**"该 CPU 上这一段之前的读临界区都退出了"。grace period 正是靠这个等价关系（§4）。

外层用户实际调用的是封装版 `rcu_read_lock()`，它在上面基础上加了 lockdep 记账：

【真实源码 v6.12 `include/linux/rcupdate.h`】：

```c
static __always_inline void rcu_read_lock(void)
{
	__rcu_read_lock();                       // ← 真正的"锁"：见上，仅 preempt_disable
	__acquire(RCU);                          // ← sparse 静态检查标记（非运行时开销）
	rcu_lock_acquire(&rcu_lock_map);         // ← lockdep 记账（仅 PROVE_LOCKING 内核）
	RCU_LOCKDEP_WARN(!rcu_is_watching(),
			 "rcu_read_lock() used illegally while idle");  // ← 在 idle/EQS 里用 RCU 直接报错
}
```

> 注意 `__acquire(RCU)` / `rcu_lock_acquire` / `RCU_LOCKDEP_WARN` 在**生产（非 debug）内核里全部退化为空**。生产内核里 `rcu_read_lock()` 真的就只剩 `preempt_disable()`。

`rcu_read_unlock()` 同理（含一条注释，正好印证 §2.5 的 flavor 合并红利）：

【真实源码 v6.12 `include/linux/rcupdate.h`】：

```c
/**
 * rcu_read_unlock() - marks the end of an RCU read-side critical section.
 *
 * In almost all situations, rcu_read_unlock() is immune from deadlock.
 * In recent kernels that have consolidated synchronize_sched() and
 * synchronize_rcu_bh() into synchronize_rcu(), this deadlock immunity
 * also extends to the scheduler's runqueue and priority-inheritance
 * spinlocks, courtesy of the quiescent-state deferral that is carried
 * out when rcu_read_unlock() is invoked with interrupts disabled.
 * ...
 */
static inline void rcu_read_unlock(void)
{
	RCU_LOCKDEP_WARN(!rcu_is_watching(),
			 "rcu_read_unlock() used illegally while idle");
	rcu_lock_release(&rcu_lock_map); /* Keep acq info for rls diags. */
	__release(RCU);
	__rcu_read_unlock();             // ← non-preempt 下即 preempt_enable()
}
```

### 3.2 读侧（PREEMPT_RCU）：逐字是"对每任务私有计数器 +1"

可抢占内核里 reader 可以被抢占，所以不能只靠 `preempt_disable`。但它**仍然不碰跨 CPU 共享状态**——它操作的是 `current->rcu_read_lock_nesting`，一个**每任务私有**字段：

【真实源码 v6.12 `kernel/rcu/tree_plugin.h`】：

```c
/*
 * Preemptible RCU implementation for rcu_read_lock().
 * Just increment ->rcu_read_lock_nesting, shared state will be updated
 * if we block.
 */
void __rcu_read_lock(void)
{
	rcu_preempt_read_enter();                 // ← 见下：对 current->rcu_read_lock_nesting +1
	if (IS_ENABLED(CONFIG_PROVE_LOCKING))
		WARN_ON_ONCE(rcu_preempt_depth() > RCU_NEST_PMAX);
	if (IS_ENABLED(CONFIG_RCU_STRICT_GRACE_PERIOD) && rcu_state.gp_kthread)
		WRITE_ONCE(current->rcu_read_unlock_special.b.need_qs, true);
	barrier();  /* critical section after entry code. */   // ← 编译器屏障：临界区不上移
}
EXPORT_SYMBOL_GPL(__rcu_read_lock);
```

那个 +1 的 helper（一目了然——没有原子操作，只有 `WRITE_ONCE`/`READ_ONCE`）：

【真实源码 v6.12 `kernel/rcu/tree_plugin.h`】：

```c
static void rcu_preempt_read_enter(void)
{
	WRITE_ONCE(current->rcu_read_lock_nesting,
		   READ_ONCE(current->rcu_read_lock_nesting) + 1);
}

static int rcu_preempt_read_exit(void)
{
	int ret = READ_ONCE(current->rcu_read_lock_nesting) - 1;

	WRITE_ONCE(current->rcu_read_lock_nesting, ret);
	return ret;
}
```

退出侧才是"有戏"的地方——只有当嵌套归零、且本任务在临界区里**被抢占过**（`rcu_read_unlock_special.s` 非零）时，才走慢路径补报 quiescent state：

【真实源码 v6.12 `kernel/rcu/tree_plugin.h`】：

```c
/*
 * Preemptible RCU implementation for rcu_read_unlock().
 * Decrement ->rcu_read_lock_nesting.  If the result is zero (outermost
 * rcu_read_unlock()) and ->rcu_read_unlock_special is non-zero, then
 * invoke rcu_read_unlock_special() to clean up after a context switch
 * in an RCU read-side critical section and other special cases.
 */
void __rcu_read_unlock(void)
{
	struct task_struct *t = current;

	barrier();  // critical section before exit code.
	if (rcu_preempt_read_exit() == 0) {              // ← 最外层退出
		barrier();  // critical-section exit before .s check.
		if (unlikely(READ_ONCE(t->rcu_read_unlock_special.s)))
			rcu_read_unlock_special(t);          // ← 仅"被抢占过"等特例走慢路径
	}
	if (IS_ENABLED(CONFIG_PROVE_LOCKING)) {
		int rrln = rcu_preempt_depth();
		WARN_ON_ONCE(rrln < 0 || rrln > RCU_NEST_PMAX);
	}
}
EXPORT_SYMBOL_GPL(__rcu_read_unlock);
```

**两种实现的统一结论**：non-preempt 改 per-CPU 的 `preempt_count`，preempt 改 per-task 的 `rcu_read_lock_nesting`——**都不是跨 CPU 共享的 cache line**。这就是 §1 那句"读者不修改任何全局协调状态"的源码兑现。

### 3.3 不变式 + 反面清单（RCU 不保证什么）

**核心不变式**：宽限期开始后，任何**新发起**的 `rcu_read_lock()` 都看不到旧指针（写者已换）；宽限期结束时，所有"换指针之前进入临界区"的读者已退出。此后旧内存安全回收。

> ⚠️ **RCU 不保证什么（最高频的 staff 级误区）**：RCU 给你的是 **"安全解引用一个 old-or-new 指针"**，不是"读到最新值"，更不是"多对象一致快照"。
> 1. **读者可能整段都在用旧版本**：与更新并发的读者，即使写者已 `synchronize_rcu` 返回，仍可能整段读临界区都在用旧对象——**设计如此**（旧对象活到宽限期后）。要"读必须看到最新写"，用锁 / seqlock，别用 RCU。
> 2. **不跨对象排序**：RCU 只保证单个被保护指针的发布顺序。一次逻辑更新若改两个独立 RCU 指针，读者可能看到"指针 1 已更新、指针 2 还旧"的撕裂态。要原子换一组数据 → **打包进一个结构体整体替换**。
> 3. **不冻结对象内容**：RCU 保证对象在你读期间不被释放，但不保证字段不变。约定是"发布后不可变"；可变字段要另加同步（如内嵌 seqcount，dcache 就是这么干的，§7 例）。

### 3.4 写侧发布：`rcu_assign_pointer` 的真实实现（含 NULL 快路径）

【真实源码 v6.12 `include/linux/rcupdate.h`】：

```c
#define rcu_assign_pointer(p, v)					      \
do {									      \
	uintptr_t _r_a_p__v = (uintptr_t)(v);				      \
	rcu_check_sparse(p, __rcu);					      \
									      \
	if (__builtin_constant_p(v) && (_r_a_p__v) == (uintptr_t)NULL)	      \
		WRITE_ONCE((p), (typeof(p))(_r_a_p__v));		      \
	else								      \
		smp_store_release(&p, RCU_INITIALIZER((typeof(p))_r_a_p__v)); \
} while (0)
```

**逐行要点**：
- 一般情况走 `smp_store_release(&p, ...)`——**release store**：保证"所有对 `*v` 字段的写" happen-before "把指针发布出去"。这正是堵住"读者拿到指针却读到半成品"的那道屏障。
- **NULL 快路径**：当 `v` 是编译期常量 NULL 时，退化成 `WRITE_ONCE`——因为"发布一个 NULL"没有"先填字段"可排序，省掉 release 屏障是安全且更快的。这是个常被忽略但很能体现"内核抠性能"的细节。
- 源码注释自己说明了它的三重作用：插内存屏障（在需要的架构上）、**禁止编译器把"初始化结构体的代码"重排到指针赋值之后**、并**文档化"这个指针会被 RCU 读侧解引用"**。最后一点是工程价值——它让 review 者一眼看出哪些指针受 RCU 保护。

配套还有一个"无屏障"的初始化宏，理解它能帮你判断"什么时候可以省屏障"：

【真实源码 v6.12 `include/linux/rcupdate.h`】（节选注释 + 定义）：

```c
/**
 * RCU_INIT_POINTER() - initialize an RCU protected pointer
 * ...
 * Note that unlike rcu_assign_pointer(), RCU_INIT_POINTER() provides no
 * ordering guarantees for either the CPU or the compiler.
 */
#define RCU_INIT_POINTER(p, v) \
	do { \
		rcu_check_sparse(p, __rcu); \
		WRITE_ONCE(p, RCU_INITIALIZER(v)); \
	} while (0)
```

> 用错 `RCU_INIT_POINTER`（该用 `rcu_assign_pointer` 的地方用了它）的后果，注释原文：*"results in impossible-to-diagnose memory corruption"*——结构在 crash dump 里看着没问题，但并发 reader 可能看到未初始化值。只在"赋 NULL / 尚无 reader / 改的是无需排序的统计字段"时才用它。

### 3.5 读侧订阅：`rcu_dereference` 的真实实现（为什么不是 no-op）

【真实源码 v6.12 `include/linux/rcupdate.h`】：

```c
#define rcu_dereference(p) rcu_dereference_check(p, 0)

#define rcu_dereference_check(p, c) \
	__rcu_dereference_check((p), __UNIQUE_ID(rcu), \
				(c) || rcu_read_lock_held(), __rcu)

#define __rcu_dereference_check(p, local, c, space) \
({ \
	/* Dependency order vs. p above. */ \
	typeof(*p) *local = (typeof(*p) *__force)READ_ONCE(p); \   // ← 关键：READ_ONCE，不是 acquire
	RCU_LOCKDEP_WARN(!(c), "suspicious rcu_dereference_check() usage"); \  // ← lockdep：临界区外用就报警
	rcu_check_sparse(p, space); \
	((typeof(*p) __force __kernel *)(local)); \
})
```

**这段源码同时回答了两个高频问题**：

1. **为什么 `rcu_dereference` 永远不是 no-op？** 因为它内核是 `READ_ONCE(p)`。`READ_ONCE` 强制每次都从内存读这个指针、并防止 load tearing/合并——**否则编译器可能把上次读到的 `p` 缓存进寄存器，绕过写者的更新**。这是纯编译器层面的必要性，**和硬件 barrier 无关**，所以即便 v5.9 删了 Alpha 依赖屏障，这个 `READ_ONCE` 也一个字都不能省。

2. **它是 acquire load 吗？不是。** 注释写得很清楚——`/* Dependency order vs. p above. */`，它依赖的是 **address-dependency ordering**：后续 `local->field` 的访问地址**来自 `local` 本身**，CPU 天然保证"先读到指针、再用它访存"。这比 `smp_load_acquire` **弱也更便宜**：acquire 会对该 load **之后所有**访存排序，依赖序只对"依赖于这个指针的访问"排序。

🎯 **面试加分点（精确区分 acquire vs address-dependency）**：很多人把 `rcu_dereference` 说成 "acquire load"，**不够精确**。
- 除 **DEC Alpha**（历史问题，§2.6 已删特殊处理）外，所有主流架构（含 ARM/POWER 这些弱内存序机器）都尊重地址依赖，依赖序在它们上**免费**——不生成任何 fence 指令。
- 反观 `smp_load_acquire`，在 ARM 上会生成 `ldar` 这类带序指令，**更贵**。
- 所以 RCU 读侧"零开销"的微架构根因，正是它走**依赖序**而非 acquire——弱内存序机器上一条 fence 都不需要，只要保证编译器不乱来（`READ_ONCE`）。

**`rcu_dereference` vs `rcu_assign_pointer` 屏障对照表**：

| 操作 | 源码内核 | 提供的序 | 防住什么 |
|------|----------|----------|----------|
| `rcu_assign_pointer(p, v)` | `smp_store_release`（NULL 走 `WRITE_ONCE`） | release：所有对 `*v` 的写 happen-before 指针发布 | 读者拿到指针却读到半成品 |
| `rcu_dereference(p)` | `READ_ONCE(p)` + address-dependency | 依赖序：依赖该指针的 load 不被提前到指针读之前 | 编译器缓存旧指针 / load tearing / Alpha split-cache（历史） |
| `synchronize_rcu()` | 全宽限期等待（内部多次 `smp_mb`，§4） | 返回时两侧 load/store 均已可见 | reader 还在用旧对象时就回收 |

**配对模型**：写者 release ↔ 读者 dependency。两者合力的 happens-before 是"release 之前的 store" → "通过该指针发出的、依赖于它的 load"。读者不需要额外屏障，是因为依赖序内嵌在 `rcu_dereference` 语义里。把"acquire/release 配对"和"release/dependency 配对"分开讲，是 staff 与高级工程师的分水岭。

### 3.6 完整发布-订阅范式：内核文档的 canonical 例子

下面这段是 `Documentation/RCU/whatisRCU.rst` 里的**官方范例**，把读写两侧拼全：

【真实源码 v6.12 `Documentation/RCU/whatisRCU.rst`】：

```c
void foo_update_a(int new_a)
{
	struct foo *new_fp;
	struct foo *old_fp;

	new_fp = kmalloc(sizeof(*new_fp), GFP_KERNEL);
	spin_lock(&foo_mutex);                                          // ← 写者之间仍要互斥（RCU 不管写写）
	old_fp = rcu_dereference_protected(gbl_foo,
				lockdep_is_held(&foo_mutex));           // ← 持锁读旧指针：用 _protected 变体
	*new_fp = *old_fp;          // copy                              // ← "Copy"：复制旧版本
	new_fp->a = new_a;          // update                            // ← "Update"：在副本上改
	rcu_assign_pointer(gbl_foo, new_fp);                            // ← 发布新版本（release）
	spin_unlock(&foo_mutex);
	synchronize_rcu();                                              // ← 等所有 pre-existing reader 退出
	kfree(old_fp);                                                  // ← 此刻回收旧版本才安全
}

int foo_get_a(void)
{
	int retval;

	rcu_read_lock();                                                // ← 读侧"锁"：仅 preempt_disable
	retval = rcu_dereference(gbl_foo)->a;                           // ← 安全解引用 old-or-new
	rcu_read_unlock();
	return retval;
}
```

**这段把 "Read-Copy-Update" 三个字逐字对上了**：`*new_fp = *old_fp`（Copy）、`new_fp->a = new_a`（Update）、`rcu_read_lock/dereference`（Read）。文档对 `synchronize_rcu` 的定义就一句：*"synchronize_rcu() does this by blocking until all pre-existing RCU read-side critical sections on all CPUs have completed."*

文档同时给出 RCU 的"核心五件套 API"：

【真实源码 v6.12 `Documentation/RCU/whatisRCU.rst`】（原文列举）：
```
a. rcu_read_lock()
b. rcu_read_unlock()
c. synchronize_rcu() / call_rcu()
d. rcu_assign_pointer()
e. rcu_dereference()
```

### 3.7 `synchronize_rcu` vs `call_rcu`

```
synchronize_rcu()：同步阻塞，调用者一直 block 到宽限期结束
    适用：写者可以等（模块卸载、低频配置更新）
    代价：可能等几十毫秒（取决于 CPU 负载与调度）；【会睡眠】→ 绝不可在持 spinlock / 中断上下文里调

call_rcu(head, func)：异步注册回调，宽限期结束后由 RCU 内核线程调用 func
    适用：写者不能 block（中断上下文、高频写路径）
    代价：回调堆积时有内存压力（rcu_barrier 可等回调排空）
```

`kfree_rcu(ptr, rcu_head_field)` 是最常见封装。McKenney 原话（LWN 2019）：*"The kfree_rcu() primitive serves as a shortcut for an RCU callback that does nothing but free the structure ... can both simplify code and reduce the need for rcu_barrier()."* 一行搞定延迟释放，且省掉手写回调和 `rcu_barrier` 的麻烦。

## 4 · 宽限期物理学：凭什么能安全回收

> 这是"扎根"的核心——把 §3 的不变式落到"quiescent state 到底是什么、谁推进、卡住怎么办"。

### 4.1 quiescent state 的物理定义

在 non-preemptible 内核里，**quiescent state（静止态）** 的最小事件是：某 CPU 经历一次 **上下文切换 / 进入 idle 循环 / 从中断返回到用户空间** 中的**任意一个**。

为什么这三件事等价于"读临界区已退出"？因为 `rcu_read_lock()` = `preempt_disable()`（§3.1 源码），**持读锁期间这三件事一件都不会发生**（关抢占 ⇒ 不切换；不会在临界区里去 idle；从中断返回用户态意味着不在内核临界区）。所以：

```
CPU 0: [rcu_read_lock ... rcu_read_unlock] [schedule] [rcu_read_lock ...]
                                             ^
                                       quiescent state 在此
                                       ⇒ CPU 0 上"换指针之前"的读临界区已全部结束
```

**宽限期 = 等到每个 CPU 都至少报告过一次 quiescent state**。一旦所有 CPU 都报过，说明"换指针之前进入的所有读临界区"在所有 CPU 上都退出了——旧对象安全。

> 💡 **坑点**：在 `rcu_read_lock()` 期间调用任何可能 sleep 的函数（`kmalloc(GFP_KERNEL)`、`mutex_lock`、拷贝用户内存可能缺页…）：non-preempt 下直接死锁/BUG（关抢占里睡眠）；PREEMPT_RCU 下会触发 lockdep 警告或更隐蔽的宽限期错乱。**读临界区里只能做"不睡眠"的事**——要睡眠请用 SRCU。

### 4.2 Tree RCU 怎么聚合（O(log N)）

```
                      rcu_state（全局 RCU 状态）
                           |
              ┌────────────┴─────────────┐
           rcu_node[0]               rcu_node[1]      ← 内部节点（聚合）
           /    \                    /    \
       rcu_node rcu_node         rcu_node rcu_node    ← leaf node（每 16~64 CPU 一个）
        CPU0-3   CPU4-7          CPU8-11  CPU12-15

每个 CPU 在 quiescent state 时向其 leaf node 报告（只竞争本 node 的锁）
leaf node 收齐辖下 CPU 后向上层报告
顶层确认所有分支已报 → 宽限期结束
```

结构体名以源码为准：`rcu_state` / `rcu_node` 定义在 `kernel/rcu/tree.h`（高把握）；每 leaf 管多少 CPU 由 `RCU_FANOUT_LEAF` 配置（典型 16）。把单点全局锁拆成树，正是 §2.3 那个 64x 改善的来源。

### 4.3 读者卡住怎么办：强制推进机制（toy RCU 没有的部分）

如果某个 reader（或某 CPU）迟迟不报 quiescent state，宽限期会不会永远不结束？真实内核有**多重强制推进**，这正是它和"用户态 toy epoch 实现"的关键差异：

- **scheduler tick 周期性补报**：时钟中断里检查，若本 CPU 在内核态待太久也能推动 quiescent state。
- **force quiescent state（FQS）**：宽限期拖太久，GP kthread 主动去扫 leaf node，对长时间不报的 CPU 施压。v6.12 `kernel/rcu/tree.c` 里 `force_qs_rnp()` 上方注释原文：*"Scan the leaf rcu_node structures. For each structure on which all CPUs have reported a quiescent state and on which there are tasks blocking the current grace period, initiate RCU priority boosting."*【真实源码 v6.12 `kernel/rcu/tree.c` 注释】
- **RCU priority boosting**（PREEMPT_RCU）：被抢占又迟迟不退的 reader，临时拉高其优先级让它尽快跑完退出（§2.7）。
- **expedited 宽限期**：`synchronize_rcu_expedited()` / 启动参数 `rcu_expedited`，用 IPI 把所有 CPU 踢一脚强制过一次 quiescent state——快但伤实时性。
- **RCU CPU stall detector** 兜底：实在推不动就打印 stall 告警（§6.1）。

**所以"宽限期实践中几乎总会前进"**——而 §章末代码题里的 toy 不会（卡一个 reader 就 epoch 不动），这是 toy "非生产级"的核心原因之一。

## 5 · 方案对比实战：一个链表，三种锁各跑一遍

> 这是本版的"思考感"环节。**别只看结论表，跟着场景推一遍**。

**场景**：一个全局单链表 `routes`，被 100 核每秒查询数百万次（`lookup(key)` 遍历找匹配项），偶尔（每秒个位数）插入/删除一项。要求读侧 P99 < 1μs，更新无停服，内存安全。

### 5.1 用 rwlock 跑一遍

```c
read_lock(&routes_lock);            // ← 原子改 reader_count（共享 cache line）
for (n = head; n; n = n->next)
        if (n->key == key) { ... }
read_unlock(&routes_lock);          // ← 再原子改 reader_count
```

- **读侧开销**：每次读两次原子操作改 `reader_count`。100 核并发读 ⇒ 这条 cache line 在 100 个核间 ping-pong，**读越多越慢，线性退化**。P99 在高并发下会被 cache 争用顶穿。
- **写侧代价**：`write_lock` 要等所有 reader 放锁，写期间**所有读被挡**（停服感）。
- **回收时机**：写者持写锁时直接改链表、直接 `kfree` 旧节点——**回收即时**，无延迟。
- **判决**：数据可原地改、回收即时，但**读侧 cache 争用是死穴**，过不了"百万 QPS、P99<1μs"。❌

### 5.2 用 seqlock 跑一遍

```c
do {
        seq = read_seqbegin(&routes_seq);     // ← 读序号（不写共享状态，便宜）
        // ...遍历链表... 但遍历途中节点可能被写者 kfree！
} while (read_seqretry(&routes_seq, seq));     // ← 序号变了就整段重读
```

- **读侧开销**：读两次 seqcount + 一个 `smp_rmb`，**不写共享状态**，比 rwlock 便宜得多，读不退化。
- **致命问题**：seqlock 的模型是"读到撕裂就重试"，它**默认数据本身在原地、只是值可能撕裂**。但链表遍历途中，写者一旦 `kfree` 了某节点，reader 的 `n->next` 就是 **use-after-free**——seqlock **拦不住悬空指针**。
- **结论**：seqlock 适合**小的、定长、原地更新**的值（如 `jiffies`、几个标量打包），**不适合"遍历会跟随指针走"的链表 + 释放节点**。除非配合 RCU 保证"遍历期间节点不被释放"（这正是 dcache RCU-walk 的做法，§7）。单用 seqlock：❌（会 UAF）。

### 5.3 用 RCU 跑一遍

```c
// 读者
rcu_read_lock();                                       // ← 仅 preempt_disable，零跨核写
list_for_each_entry_rcu(n, &routes, list)              // ← 内含 rcu_dereference
        if (n->key == key) { ...复制出需要的字段... }
rcu_read_unlock();

// 写者（删一项）
spin_lock(&routes_wlock);                              // ← 写写之间仍互斥
list_del_rcu(&victim->list);                           // ← 把节点摘出链表（reader 可能还在用）
spin_unlock(&routes_wlock);
synchronize_rcu();   // 或 kfree_rcu(victim, rcu)      // ← 等所有 pre-existing reader 退出
kfree(victim);                                          // ← 此刻回收才安全
```

- **读侧开销**：`rcu_read_lock` = `preempt_disable`（§3.1 源码），`list_for_each_entry_rcu` 每步一个 `rcu_dereference`（= `READ_ONCE` + 依赖序，弱内存序机器零 fence）。**读不写任何共享状态，读越多不退化**。P99<1μs 轻松达标。
- **写侧代价**：写写互斥（RCU 不管写写）+ `list_del_rcu`（O(1)）+ **等一个宽限期**才能回收。写侧实际比 rwlock **更重**——优势全在读侧。
- **回收时机**：**延迟到宽限期后**。期间旧节点和新链表并存（"maintain multiple versions"）。这也意味着内存峰值略高（旧节点多活一会儿）。
- **判决**：读侧零退化 + 内存安全（延迟回收堵住 UAF）。✅ 这就是内核路由表选 RCU 的原因。

### 5.4 对照表（先推理后看表）

| 维度 | rwlock | seqlock | RCU |
|------|--------|---------|-----|
| 读侧开销 | 2× 原子改 reader_count（cache 争用，**线性退化**） | 2× 读 seqcount + rmb（不退化） | `preempt_disable`（弱序机器零 fence，**不退化**） |
| 读侧能否遍历会释放的链表 | 能（持锁防释放） | **不能**（拦不住 UAF） | 能（延迟回收防 UAF） |
| 写侧代价 | 等所有 reader 放锁，写时挡读 | 写时 reader 重试 | 写写互斥 + **等宽限期**（最重） |
| 数据须可复制 | 否 | 否（原地小值） | **是**（写时复制） |
| 回收时机 | 即时 | 即时 | **延迟到宽限期后**（内存峰值略高） |
| 读者可睡眠 | 否 | 否 | 否（**SRCU 才可以**） |
| 最佳场景 | 读写均频繁、数据不可复制、要即时回收 | 小定长值、原地更新、读极多 | **读极多写极少、可复制、可延迟释放** |

### 5.5 RCU 的不适用边界（反面）

不要无脑上 RCU。下列场景 RCU 是错的工具：

1. **读必须看到最新写**（如读后立刻据此做强一致决策）→ RCU 读者可能整段用旧值。用锁 / seqlock。
2. **需要即时回收 / 内存极紧**（如 allocator 内部、要立刻复用的对象池）→ 宽限期会让旧对象多活几十毫秒。
3. **写多读少 / 读写各半**→ 写侧"复制 + 等宽限期"比锁还重，得不偿失。
4. **读临界区内必须睡眠**（IO 等待、可能缺页的 copy_to_user、调用可能阻塞的回调）→ 用 **SRCU**，但要接受读侧不再零开销。
5. **要原子地换一组互相关联的对象**→ 单 `rcu_assign_pointer` 只发布一个指针，会撕裂。打包进一个结构体整体替换。

## 6 · 落地：生产调优与排障

### 6.1 RCU stalls（宽限期卡死）

**现象**：日志出现 `rcu_sched detected stalls` 或 `INFO: rcu_preempt self-detected stall on CPU`，系统卡顿但常不 panic。

**根因**：某 CPU 长时间不报 quiescent state（即 §4.3 的强制推进也救不回来的极端）：
- 内核线程关抢占后陷入长循环（NAPI poll 死循环、memory compaction bug…）。
- 中断风暴：某 CPU 始终在中断/softirq 上下文。
- **虚拟化 steal time**：VM 被 hypervisor 调度走，guest 看不到时钟中断，quiescent state 不前进。

**排查命令**：
```bash
# 最直接：stall 日志本身带 CPU 号和调用栈
dmesg | grep -i rcu

# RCU 运行时状态在 debugfs（需挂 debugfs + CONFIG_RCU_TRACE）
# 4.20 flavor 合并后通常只剩一个 flavor 目录，常见名 rcu_preempt
# 目录/文件名随版本变，以实际 ls 为准，不要硬记
ls /sys/kernel/debug/rcu/
cat /sys/kernel/debug/rcu/rcu_preempt/rcudata   # 每 CPU 的 RCU 状态（字段含义随版本，待核）
```

**调优参数**（参数名高把握，挂载点随发行版略异）：
```bash
# RCU CPU stall 超时（默认 21 秒）—— module 参数
cat  /sys/module/rcupdate/parameters/rcu_cpu_stall_timeout
echo 60 > /sys/module/rcupdate/parameters/rcu_cpu_stall_timeout

# 强制 expedited 宽限期（IPI 踢所有 CPU，加速但伤实时）
#   启动参数：rcupdate.rcu_expedited=1
#   运行时：  sysctl kernel.rcu_expedited
```

> 💡 **坑点（云环境）**：VM 被暂停时 quiescent state 不推进，**云上 RCU stall 告警常比物理机多**——优先怀疑 steal time，不一定是内核 bug。

### 6.2 lockdep / `CONFIG_PROVE_RCU`

开 `CONFIG_PROVE_RCU=y` 后，lockdep 会抓两类错：
- 在 `rcu_read_lock()` **之外**调用 `rcu_dereference()`——正是 §3.5 源码里那条 `RCU_LOCKDEP_WARN(!(c), "suspicious rcu_dereference_check() usage")`。
- 在 RCU 读临界区内调用 sleep 函数。

开发必开，生产只在 debug 内核开。这是把"用错 RCU"从线上 UAF 提前到开发期告警的关键护栏（§8.2 Rust 想把它进一步前移到编译期）。

### 6.3 `rcutorture`：内核自带 RCU 压测

`CONFIG_RCU_TORTURE_TEST=m` 加载后，以可配 reader/writer 数、持续时间压测 RCU 实现正确性（含人为制造的边界）。大厂内核 CI 在合并任何 RCU 相关改动时必跑——呼应 §2.7 社区对"RCU 太容易写错"的长期警惕。

### 6.4 eBPF 观察 RCU 行为（示意，函数名随内核调整）

```bash
# 统计 rcu_read_lock 临界区长度分布
bpftrace -e '
kprobe:rcu_read_lock  { @start[tid] = nsecs; }
kprobe:rcu_read_unlock /@start[tid]/ {
  @us = hist((nsecs - @start[tid]) / 1000);
  delete(@start[tid]);
}'

# 观察 call_rcu 回调来源栈（找谁在狂注册回调）
bpftrace -e 'kprobe:call_rcu { @[kstack] = count(); }'
```
> 注意 §3.1 显示 `rcu_read_lock` 是 `static inline`，可能被内联/优化掉而 kprobe 挂不上——这种情况用 tracepoint（如 `trace_rcu_*`，需 `CONFIG_RCU_TRACE`）更稳。

## 7 · 无锁补充：CAS / ABA / hazard pointer / epoch（RCU 的"同族"）

> RCU 本质是 **epoch-based reclamation** 的内核特化。把它放进无锁回收的谱系里，能看清它解决的是哪一类问题。

### 7.1 原子操作与 CAS

无锁的硬件基石：
- **x86**：`LOCK XADD`（原子加，返回旧值）、`LOCK CMPXCHG`（CAS 硬件实现）。
- **Linux 封装**（高把握）：`atomic_read/set/inc`、`atomic_dec_and_test`、`atomic_cmpxchg(v, old, new)`（**无论成败都返回操作前实际值**，调用者拿返回值和 `old` 比）、`try_cmpxchg(v, &old, new)`（返回 bool，失败把当前值写回 `old`，更适合重试循环，少一次比较）。

CAS 语义：
```
if (*ptr == expected) { *ptr = new; return true; }   // 成功
else                  { return false; }                // 失败，调用者重试（用返回值作新 expected）
```

### 7.2 ABA 问题：无锁最大陷阱

```
时刻0：ptr → Node_A(value=10)
T1：读 ptr 得 Node_A；准备 CAS(ptr, Node_A, Node_B)
T2：删 Node_A，分配 Node_C（恰好复用 Node_A 的地址），ptr → Node_C
T3：删 Node_C，ptr → Node_A（地址又换回来）
T1：执行 CAS，发现 ptr == Node_A → 成功！但 Node_A 早已是不同对象
```

内存复用让"地址相同 ≠ 对象相同"。三种对策：

| 方法 | 原理 | 代价 |
|------|------|------|
| Tagged pointer | 高位带版本号，每次 CAS 版本+1 | 需 128-bit CAS 或牺牲地址位 |
| Hazard pointer | 读者发布"我正在访问此指针"，回收前检查 | per-thread 状态，读侧有 store 开销 |
| Epoch-based reclamation | 全局 epoch，所有读者离开旧 epoch 才回收 | **RCU 本质就是它** |

> 💡 用户态 C++11 的 `compare_exchange_weak/strong`、`std::atomic<T*>` 都**不自带 ABA 防护**——这正是 epoch/RCU 类机制存在的理由之一。

### 7.3 RCU 在内核的广泛应用

```
子系统                       用法
────────────────────────────────────────────────────────
网络路由表 (fib_table)        查询走 RCU 读，更新走写路径（§5 那个例子的真身）
task_struct credentials       进程 cred 用 RCU 保护（下详）
dentry/inode (RCU-walk)       路径查找 LOOKUP_RCU 下无锁 dcache 查找
模块系统                       module 列表遍历
netfilter hook                hook 链表读
设备/驱动注册表               bus/driver 匹配查找
```

**为什么 `task_struct` 的 credentials 用 RCU**：进程改自身凭证（`setuid` 走 `prepare_creds` → `commit_creds`）时，分配新 `cred`，填好后原子替换 `task->cred`，旧 cred 等宽限期后释放。**关键是"读侧常是别的任务"**：当 A 通过 `/proc/<B>/status` 读 B 的 uid/gid 时，用 `rcu_read_lock()` + `__task_cred()`（内部 `rcu_dereference`）**直接解引用，不持有 B 的任何锁**；B 同时改自己 cred 也安全，因为旧 cred 活到宽限期后。

> 💡 **dcache RCU-walk 是 RCU + seqlock 的经典组合**（正好印证 §5.2 的判决）：路径查找先试无锁 RCU 模式（`__d_lookup_rcu`，配合 dentry 的 **seqcount** 检测并发改名/失效），失败再 "dropback" 到带引用计数 + spinlock 的 ref-walk。RCU 负责"遍历期间 dentry 不被释放"，seqcount 负责"内容没被并发改坏"——两者各补对方的短板。（`__d_lookup_rcu` 定义在 `fs/dcache.c`，主遍历循环在 `fs/namei.c`，函数名随版本变动，以源码为准。）

## 8 · 未来演进

### 8.1 NOCB RCU（No-Callback）

把 RCU 回调卸载到专用的 `rcuop/N` 内核线程，减少 quiescent-state 处理对实时延迟的干扰。`PREEMPT_RT` 和 `nohz_full` 严重依赖此特性（呼应 §2.7）。启动参数 `rcu_nocbs=`。

### 8.2 Rust for Linux 的 RCU 抽象

Rust for Linux（驱动部分已进主线）在设计类型安全的 RCU 封装：用 lifetime + ownership 在**编译期**保证 `rcu_dereference` 只在持有"RCU 读侧 guard"时调用；用 `Guard` 型 RAII 封装 `rcu_read_lock/unlock`。目标是把 §6.2 那条 **lockdep 运行时检查前移到编译期**——"在读临界区外解引用"这类 bug 直接编译不过。这是 Rust 类型系统在内核同步安全上的标志性应用。（具体 API 名随项目演进，待核。）

### 8.3 用户态 RCU（liburcu / folly::rcu）

Facebook、Google 的高性能用户态服务广泛使用 `liburcu`（Mathieu Desnoyers 维护），在 userspace 实现 epoch-based reclamation，无需进内核。`folly::rcu`（C++）是工业级实现；Rust 生态有 `crossbeam-epoch`。它们和内核 RCU 同源不同身：内核靠"关抢占 + 调度点"定义宽限期，用户态靠"线程周期性进入 quiescent 调用"——所以用户态 RCU 必须显式调 `rcu_quiescent_state()` 之类。

### 8.4 io_uring 与无锁 I/O

`io_uring`（见第 11 章）内部大量用 lock-free ring（SQ/CQ），以 `smp_store_release` / `smp_load_acquire` 配对实现生产者-消费者，不需 syscall 级锁。这是"无锁思想在 I/O 路径"的当代实践。

## 章末五件套

### 高频面试题（含答题框架）

**T1 · RCU 读临界区为什么不需要锁？（要求答到源码级）**
- 读侧不修改任何**跨 CPU 共享**状态：non-preempt 下 `rcu_read_lock` 逐字是 `preempt_disable()`（改 per-CPU `preempt_count`）；PREEMPT_RCU 下逐字是对 `current->rcu_read_lock_nesting`（per-task）`WRITE_ONCE` +1。
- 关抢占/记账让 quiescent state 可被追踪（"经历调度" ≡ "临界区退出"）。
- 写者原子换指针 + 延迟回收，保证读者要么看旧、要么看新、且旧对象活到宽限期后。
- **错误答案陷阱**：说"因为用了原子操作"——读侧恰恰**没有**原子操作；那是 CAS（写侧无锁结构）的事。

**T2 · 内存屏障在 RCU 里插在哪？为什么 `rcu_dereference` 不是 no-op？**
- 写侧 `rcu_assign_pointer` = `smp_store_release`（NULL 走 `WRITE_ONCE`）；读侧 `rcu_dereference` = `READ_ONCE` + **address dependency**（不是 acquire）。
- 不是 no-op 的根因是 `READ_ONCE`：防编译器把指针缓存进寄存器、防 load tearing——纯编译器层面，和硬件 barrier 无关；这就是 v5.9 删了 Alpha 依赖屏障后它仍存在的原因。
- 加分：区分"release/dependency 配对"和"acquire/release 配对"——RCU 走前者，弱内存序机器零 fence。

**T3 · 为什么大型路由表查找优先 RCU 而非 rwlock？高流量下吞吐差异的根因？**
- rwlock 的 `reader_count` 是 cache 争用热点，读越多越退化；RCU 读侧只 `preempt_disable`、无跨核写、不退化。流量越大差异越大（rwlock 线性退化，RCU 几乎平）。

**T4 · `synchronize_rcu` 在持 spinlock 时调用会怎样？**
- 它会**睡眠**（等宽限期），持 spinlock 睡眠 = 死锁/内核 BUG。正确做法：先放 spinlock，或改 `call_rcu` / `kfree_rcu` 异步注册回调。

**T5 · ABA 最小复现时序？内核 RCU 如何绕过 ABA？**
- A→B→A 的地址复用三步（见 §7.2）。RCU 通过 epoch（宽限期）保证"旧地址在所有 pre-existing reader 退出前不被复用"——延迟回收（不是 refcount）从根上消除 ABA。

**T6 · 何时该用 SRCU 而非经典 RCU？代价？**
- 读临界区内需要 sleep（IO 等待、可能缺页的用户拷贝、可能阻塞的回调）时用 SRCU。代价：每次 `srcu_read_lock` 写 per-CPU 计数器（有 cache 开销），宽限期 per-`srcu_struct` 而非全局。理由源头：经典 RCU 里"睡眠 = 上下文切换 = quiescent state"，会误判 reader 已退出（§2.4 McKenney 原话）。

**T7 · Hazard pointer 和 epoch-based 在进度性质上的区别？**
- hazard pointer 更偏 lock-free（读者发布 hazard 不依赖别的线程进度）；epoch-based 要求所有线程周期性推进 epoch，有线程卡住 epoch 就不前进（obstruction-free 级）。RCU 是 epoch-based 变体，但内核版有多重**强制推进**（FQS / boosting / expedited / stall detector，§4.3）兜底，所以实践中宽限期几乎总会前进——这正是它和用户态 toy 的关键差异。

### 实战项目：P3 · 用户态 RCU-lite 模拟

**目标**：用 C 实现一个最小 epoch-based reclamation，亲手感受宽限期本质，并**亲眼看到它和真实内核 RCU 的差距**（卡一个 reader 就回收停摆）。

**规格**：
```
状态：
  - 全局 epoch（atomic）
  - per-thread 本地 epoch（thread_local，临界区内=当前 epoch，临界区外=RETIRED）
  - 回收队列：per-thread 链表，存待回收指针 + retire_epoch
API：
  rculite_read_lock()   → local_epoch = global_epoch
  rculite_read_unlock() → local_epoch = RETIRED
  rculite_assign_ptr/deref → store_release / load_acquire
  rculite_retire(ptr, free_fn) → 挂到当前 epoch 的回收队列
  rculite_quiesce()     → 推进 global_epoch；遍历回收队列，若所有线程 local_epoch > retire_epoch 则 free
验收：
  1. 多线程读写 shared linked list，无 use-after-free（ASan）
  2. 写者 quiesce 后旧节点被正确释放（ASan/valgrind）
  3. 全程不用锁（只用 atomic load/store/CAS）
代码骨架见下方"代码题"。
```

### 设计题：高并发配置热更新系统

**题目**：配置每 30 秒热更一次，但每秒 100 万次读取配置项。要求读侧 P99 < 1μs、更新无停服、内存安全。

**答题框架**：

1. **第一反应 `std::shared_ptr` + `atomic_store`？不行**——`shared_ptr` 的 refcount 是 cache 争用热点，100 万 QPS 下原子加减就是瓶颈（和 rwlock 同病，呼应 §1）。
2. **RCU 思路**：Config 不可变（immutable after construction）；维护 `atomic<Config*>`；读者 `read_lock → deref → ... → read_unlock`；更新者分配新 Config、填好、`assign_ptr`、旧 Config 经 `retire` 延迟释放。
3. **量级估算**：100 万 QPS / 10 核 ≈ 每核 10 万 QPS；`preempt_disable`/epoch +1 开销 < 10ns，P99 < 100ns 绝对可达；宽限期即便 100ms 也远小于 30s 更新间隔；内存最多 2 份 Config 并存（旧+新），通常 <1MB，可接受。
4. **与 COW 的区别**：COW 是"写时才复制（被触发）"，RCU 是"主动复制到新对象再发布"——方向相反，RCU 的写代价在"创建新副本"而非"发现有读者才复制"。

### 系统题：线上 RCU Stall 排障场景

**场景**：8 核 Linux，`dmesg` 出现下日志，系统极慢但未 panic：
```
rcu_sched detected stalls on CPUs/tasks: { 3 } (detected by 0, t=21002 jiffies)
```

**排查步骤**：

1. **先排 steal time**（云环境最常见，§6.1）：
   ```bash
   vmstat 1 5     # 看 st 列
   top            # 看 %st
   ```
   `st > 10%` ⇒ 优先找 IaaS 排 noisy neighbor，而非内核 bug。
2. **确认 CPU 3 在干嘛**：
   ```bash
   echo l > /proc/sysrq-trigger     # 打印所有 CPU backtrace（需 CONFIG_MAGIC_SYSRQ）
   dmesg | grep -A 20 "CPU.*3"
   ```
3. **常见根因栈**：网络驱动 NAPI poll 死循环；softirq 处理过长致 CPU 3 长期在中断上下文；内核 bug 无限重试（如 memory compaction）。
4. **缓解**：已知驱动 bug → 临时下线 CPU 3（`echo 0 > /sys/devices/system/cpu/cpu3/online`）；IRQ affinity 问题 → 重分发中断（`/proc/irq/N/smp_affinity`）。
5. **根治**：升级驱动/内核；steal time → 迁 VM。

### 代码题：RCU-lite 骨架（可运行，带 TODO）

```c
// rculite.h — 最小用户态 epoch-based RCU（仅供理解原理，非生产级）
// 编译（ASan/TSan 互斥，分两次跑）：
//   查 UAF ：gcc -O2 -fsanitize=address -pthread rculite_demo.c -o demo_asan
//   查竞争 ：gcc -O2 -fsanitize=thread  -pthread rculite_demo.c -o demo_tsan
#include <stdatomic.h>
#include <stdint.h>
#include <stdlib.h>
#include <pthread.h>

#define MAX_THREADS   64
#define RETIRED_EPOCH UINT64_MAX

static atomic_uint_fast64_t g_epoch = 0;             // 全局 epoch（写者推进）

// per-thread 本地 epoch。必须 _Atomic：回收者会跨线程 atomic_load 它。
static _Thread_local _Atomic uint_fast64_t tl_epoch = RETIRED_EPOCH;

typedef struct { atomic_uint_fast64_t *epoch_ptr; } ThreadEntry;
static ThreadEntry g_threads[MAX_THREADS];
static atomic_int  g_thread_count = 0;

typedef struct RetireNode {
    void *ptr; void (*free_fn)(void *); uint64_t retire_epoch;
    struct RetireNode *next;
} RetireNode;
static _Thread_local RetireNode *retire_list = NULL;

void rculite_thread_init(void) {                     // 每线程调一次
    int idx = atomic_fetch_add(&g_thread_count, 1);
    g_threads[idx].epoch_ptr = &tl_epoch;
}

static inline void rculite_read_lock(void) {
    // 写本线程 tl_epoch = 当前 global epoch。本线程自己写自己读 → relaxed 即可；
    // 真正可见性由回收者侧 acquire-load + quiesce 全局序保证。
    atomic_store_explicit(&tl_epoch,
        atomic_load_explicit(&g_epoch, memory_order_acquire),
        memory_order_relaxed);
}
static inline void rculite_read_unlock(void) {
    atomic_store_explicit(&tl_epoch, RETIRED_EPOCH, memory_order_release);
}

#define rculite_assign_ptr(p, v) atomic_store_explicit((p), (v), memory_order_release)
#define rculite_deref(p)         atomic_load_explicit((p), memory_order_acquire)

void rculite_retire(void *ptr, void (*free_fn)(void *)) {
    RetireNode *n = malloc(sizeof(*n));
    n->ptr = ptr; n->free_fn = free_fn;
    n->retire_epoch = atomic_load_explicit(&g_epoch, memory_order_relaxed);
    n->next = retire_list; retire_list = n;
}

// 所有线程都已离开 epoch <= retire_epoch 的临界区？
static int all_readers_past(uint64_t retire_epoch) {
    int count = atomic_load(&g_thread_count);
    for (int i = 0; i < count; i++) {
        uint64_t e = atomic_load_explicit(g_threads[i].epoch_ptr, memory_order_acquire);
        if (e <= retire_epoch) return 0;             // 还有读者在临界区
    }
    return 1;
}

void rculite_quiesce(void) {
    atomic_fetch_add_explicit(&g_epoch, 1, memory_order_acq_rel);   // 推进 epoch
    RetireNode **prev = &retire_list, *cur = retire_list;
    while (cur) {
        if (all_readers_past(cur->retire_epoch)) {
            cur->free_fn(cur->ptr);
            *prev = cur->next; RetireNode *tmp = cur; cur = cur->next; free(tmp);
        } else { prev = &cur->next; cur = cur->next; }
    }
}
```

**暗坑提示（务必读，这些正是 toy 与内核 RCU 的差距）**：
1. `_Thread_local` 变量地址在线程退出后失效——`g_threads[i].epoch_ptr` 会悬空。生产需线程退出时 `rculite_thread_fini` 注销。
2. `all_readers_past` 的扫描和写者 `retire_epoch` 赋值之间有 TOCTOU 窗口，TSan 会报（这里有意留作 TODO）。
3. **"某线程卡在临界区 ⇒ epoch 永不推进 ⇒ 回收永远停摆"**——这恰是 toy 和**真实内核 RCU 的关键差异**：内核有 scheduler tick 补报、force quiescent state、PREEMPT_RCU 的 priority boosting、expedited、stall detector 多重强制推进（§4.3），所以内核宽限期"几乎总会前进"，而这个 toy 不会。别把两者当一回事。

## 章末心法

> **RCU 的本质是"把锁的代价从读侧移到时间轴上"——读者（几乎）零开销，写者用宽限期换安全。理解了"`rcu_read_lock` 逐字就是 `preempt_disable`"和"宽限期 = 等所有 pre-existing reader 退出"这两句，就理解了所有 epoch-based 无锁回收方案的共同灵魂。**
