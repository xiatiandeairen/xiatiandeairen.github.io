## 1. 设计考古：GC 的历史与动机

### 1.1 起源：McCarthy 1960

GC 的概念由 John McCarthy 在 1960 年 *Recursive Functions of Symbolic Expressions and Their Computation by Machine* 中首次描述，作为 LISP 实现的一部分。那篇论文提出了经典的两阶段 mark-sweep 思路：

> "We wish to determine those list structures which are no longer accessible, and to return their words to the free-storage list."

论文背景：LISP 1.5 在 IBM 704 上运行，内存极度紧张（只有 4K 个 36-bit word）。手动 `free` 在 LISP 的 S-expression 图结构中实际上不可行——对象图可能被多个 list 共享，没有明确的 owner。

**核心动机**：

| 场景 | 手动管理的困难 | GC 解决方式 |
|------|-------------|-----------|
| 函数式语言（共享不可变值）| 无单一 owner，引用计数 overhead 高 | 追踪式 GC 一次性识别整批垃圾 |
| 循环数据结构 | 引用计数永远 > 0，无法回收 | 可达性分析，循环整体为垃圾 |
| 动态类型 VM | 类型在运行时才知，指针识别难 | GC 持有完整类型信息 |

### 1.2 引用计数的前驱：Collins 1960

与 McCarthy 同年，George Collins 独立提出引用计数（*A Method for Overlapping and Erasure of Lists*，CACM 1960）。两条路线从那时起并行发展 50 余年，直到现代语言把它们混合使用。

### 1.3 论文里程碑（考古列表）

| 年份 | 论文 / 事件 | 贡献 |
|------|-----------|------|
| 1960 | McCarthy，CACM | 首个 mark-sweep GC |
| 1960 | Collins，CACM | 引用计数 |
| 1963 | Minsky | Compact/moving GC，首次提出压缩思路 |
| 1969 | Dijkstra et al. | Baker & Cheney copying GC 理论基础 |
| 1970 | Cheney | Semispace copying（BFS order） |
| 1984 | Lieberman & Hewitt | 代际 GC（generational GC） |
| 1992 | Dijkstra, Lamport, et al. | Tri-color abstraction + 并发写屏障（`On-the-Fly Garbage Collection`，CACM 1978 原版，1992 整理） |
| 1994 | Boehm & Weiser | Conservative GC（不需要类型信息的 mark-sweep） |
| 1996 | Wilson survey | *Uniprocessor Garbage Collection Techniques*，史上最全综述 |
| 2011 | G1 GC 发布 | Garbage-First region-based 并发 GC |
| 2018 | ZGC / Shenandoah | 亚毫秒 STW 并发 GC |

## 2. 引用计数：最古老的管理策略

### 2.1 基本原理

每个堆对象维护一个整数计数器，追踪有多少引用指向它。引用建立时 +1，引用销毁时 -1，计数归零时立即释放。

```
object.refcount = 0        # 创建时
object.refcount += 1       # 赋值给变量
object.refcount -= 1       # 变量离开作用域
if object.refcount == 0:
    free(object)           # 立即回收
```

**优点**：
- 回收是局部的、即时的，无 stop-the-world pause
- 内存使用上限接近程序实际需求（不会积压大批垃圾）
- 实现简单，调试友好

**致命缺陷：循环引用**

```python
# Python 中制造一个循环
a = []
b = []
a.append(b)   # a.refcount stays 1, b.refcount = 2
b.append(a)   # b.refcount stays 2, a.refcount = 2

del a
del b
# 此时 a 和 b 均 refcount = 1，永远不会归零！
```

这正是 CPython 必须在引用计数之外叠加一个 cycle detector（追踪式 GC）的根本原因。

### 2.2 延迟引用计数（Deferred Reference Counting）

每次函数调用时对局部变量做 inc/dec 操作，成本极高（一次赋值可能触发两次原子操作）。DRC（Deutsch & Bobrow 1976）的思路：对栈上短命引用不做计数，只对堆-to-堆引用计数；栈引用通过定期扫描栈来补偿。

### 2.3 CPython 的 Py_INCREF / Py_DECREF 实现

CPython 中每个 Python 对象的头部（`PyObject`）包含：

```c
// cpython/Include/object.h (示意，非逐字)
typedef struct _object {
    Py_ssize_t ob_refcnt;   // 引用计数
    PyTypeObject *ob_type;  // 类型指针
} PyObject;
```

`Py_INCREF` 和 `Py_DECREF` 在非 GIL-disabled 模式下是非原子操作（GIL 保证了单线程执行），在 Python 3.13+ 的 no-GIL 模式下变成原子 fetch-add。

### 2.4 Swift/Rust 中的引用计数变种

- **Swift ARC**（Automatic Reference Counting）：编译器在 IR 层插入 `retain`/`release` 调用，优化掉不必要的 retain-release pairs（成对消除）。
- **Rust**：`Rc<T>` 是单线程非原子引用计数，`Arc<T>` 是原子引用计数，两者均无 GC 追踪——用所有权系统在编译期排除大多数循环场景。

## 3. 追踪式 GC：Mark-Sweep 全解剖

### 3.1 算法骨架

**三阶段流程**（McCarthy 原始设计已具备这一结构）：

```
Phase 1: Stop the world
Phase 2: Mark  — 从 GC roots 出发，递归标记所有可达对象
Phase 3: Sweep — 遍历堆，回收未标记对象
```

**GC Roots** 是可达性分析的起点，通常包括：
- 寄存器和栈上的指针
- 全局变量 / 静态变量
- JNI/FFI 引用（从外部持有的对象）

### 3.2 Mark 阶段：Tri-Color Invariant

Dijkstra 1978 提出的 tri-color 抽象是理解并发 GC 的基础：

| 颜色 | 含义 |
|------|------|
| **White** | 尚未访问，或本轮已确认不可达（候选垃圾） |
| **Gray** | 已发现可达，但其出边尚未全部处理 |
| **Black** | 已发现可达，且所有出边已处理完毕 |

**不变式（Strong Tri-Color Invariant）**：不存在从 Black 对象直接引用 White 对象的边。  
满足此不变式时，White 集合即为垃圾。

Mark 算法（BFS 版）：

```python
# 【示意，非逐字】
gray_queue = deque(gc_roots)  # 所有 roots 染灰
mark_root(gray_queue)

while gray_queue:
    obj = gray_queue.popleft()
    for child in obj.references():
        if child.color == WHITE:
            child.color = GRAY
            gray_queue.append(child)
    obj.color = BLACK          # 出边处理完毕，染黑
```

DFS 版（递归实现更简单但有栈溢出风险，clox 用迭代灰色工作队列解决了这个问题）。

### 3.3 Sweep 阶段

遍历整个堆（或对象链表），回收仍为 White 的对象：

```python
for obj in all_objects:
    if obj.color == WHITE:
        free(obj)
    else:
        obj.color = WHITE      # 重置为白，为下一轮准备
```

### 3.4 Mark-Compact（压缩）

Minsky 1963 提出：在 sweep 之后把存活对象紧缩到堆的一端，消除碎片，使分配只需 bump-pointer：

```
Before compact: [A][_][B][_][_][C][_]
After compact:  [A][B][C][_][_][_][_]
```

代价：需要更新所有指向被移动对象的指针（two-pass：先计算新地址，再批量更新），通常需要额外的 forwarding pointer 空间。

### 3.5 Copying GC（Cheney 1970）

Semispace 设计：把堆分为 from-space 和 to-space，每次 GC 把所有存活对象从 from 复制到 to，交换角色：

```
From-space: [A]->[B]->[C]  (A,C reachable; B is garbage)
  copy A → To-space[A']
  copy C → To-space[C']
  update A'.ref_to_C = C'
Swap spaces.
From-space (new to-space) ready for next allocation.
```

**优点**：
- 分配只是 bump-pointer，O(1)
- 自动压缩，无碎片
- 只与存活对象数量成比例，垃圾越多越快

**缺点**：需要 2x 内存（一半 semi-space 始终空着）。

## 4. 代际 GC：利用弱代际假设

### 4.1 弱代际假设（Weak Generational Hypothesis）

Lieberman & Hewitt 1984（*A Real-Time Garbage Collector Based on the Lifetimes of Objects*，CACM 1983）通过 Smalltalk 程序的 profiling 发现：

> **Most objects die young.**（大多数对象在创建后很快就死去。）

即：程序的绝大多数临时分配（循环变量、中间结果、字符串拼接）在第一次 GC 之前就失去了引用。

基于这一假设，代际 GC 的设计原则是：**频繁扫描年轻代，偶尔扫描老年代**。

### 4.2 CPython 的三代结构

【真实源码 python/cpython@Include/internal/pycore_interp_structs.h】

```c
typedef struct {
    // Tagged pointer to next object in the list.
    // 0 means the object is not tracked
    _Py_ALIGNED_DEF(_PyObject_MIN_ALIGNMENT, uintptr_t) _gc_next;

    // Tagged pointer to previous object in the list.
    // Lowest two bits are used for flags documented later.
    // Those bits are made available by the struct's minimum alignment.
    uintptr_t _gc_prev;
} PyGC_Head;

struct gc_generation {
    PyGC_Head head;      // 双向链表头节点（sentinel）
    int threshold;       // 触发收集的阈值
    int count;           // 当前分配数/子代收集次数
};
```

**三代设计**：
- **Generation 0**（年轻代）：新分配的 container 对象（list/dict/set/class instance 等）进入这里。`threshold=700`，count 超过时触发 gen0 收集。
- **Generation 1**（中间代）：在 gen0 收集中存活下来的对象晋升至此。gen0 收集 `threshold1=10` 次后触发 gen1 收集。
- **Generation 2**（老年代）：长期存活对象。gen1 收集 `threshold2=10` 次后触发 gen2 收集（full collection）。

默认阈值：`gc.get_threshold()` 返回 `(700, 10, 10)`（CPython 3.12）。

### 4.3 跨代引用问题：Write Barrier

代际 GC 的一个核心挑战：如果老年代对象持有年轻代对象的引用，只扫描年轻代会漏掉这条根。解决方案：

**Card Table**（Java HotSpot）：把老年代内存分成 N 个 512-byte card，任何对 card 的写操作会标记这个 card 为 dirty。Young GC 时扫描 dirty cards 找出跨代引用。

**CPython 的简化方案**：CPython 没有 moving GC，不需要 card table。它用另一个维度解决：gc_generation 维护 `count` 字段，当 gen0 count 超阈值时，同时也收集 gen0 的父代（向上包含），这样跨代引用天然在同一次收集中处理。

## 5. CPython 真实源码精读

CPython 的 GC 是引用计数 + 循环检测器的双层结构：

```
Layer 1: 引用计数（每次 INCREF/DECREF）→ 处理 DAG 对象（无循环）
Layer 2: 循环检测器（定期触发）→ 处理循环引用的对象图
```

### 5.1 gc_collect 入口

【真实源码 python/cpython@Modules/gcmodule.c】

```c
static Py_ssize_t
gc_collect_impl(PyObject *module, int generation)
{
    PyThreadState *tstate = _PyThreadState_GET();
    if (generation < 0 || generation >= NUM_GENERATIONS) {
        _PyErr_SetString(tstate, PyExc_ValueError, "invalid generation");
        return -1;
    }
    return _PyGC_Collect(tstate, generation, _Py_GC_REASON_MANUAL);
}
```

### 5.2 三步核心算法

算法的精华在 `Python/gc.c` 中的三个函数，它们共同实现了"通过引用计数差分检测循环垃圾"的思路：

**Step 1: update_refs — 拷贝引用计数到 gc_refs 字段**

【真实源码 python/cpython@Python/gc.c】

```c
/* Set all gc_refs = ob_refcnt.  After this, gc_refs is > 0 and
 * PREV_MASK_COLLECTING bit is set for all objects in containers.
 */
static Py_ssize_t
update_refs(PyGC_Head *containers)
{
    PyGC_Head *next;
    PyGC_Head *gc = GC_NEXT(containers);
    Py_ssize_t candidates = 0;

    while (gc != containers) {
        next = GC_NEXT(gc);
        PyObject *op = FROM_GC(gc);
        if (_Py_IsImmortal(op)) {
            // immortal 对象（Python 3.12+ 的常量对象）跳过，从链表移除
            assert(!_Py_IsStaticImmortal(op));
            _PyObject_GC_UNTRACK(op);
            gc = next;
            continue;
        }
        gc_reset_refs(gc, Py_REFCNT(op));  // gc_refs = ob_refcnt
        _PyObject_ASSERT(op, gc_get_refs(gc) != 0);
        gc = next;
        candidates++;
    }
    return candidates;
}
```

关键点：`gc_refs` 暂存在 `_gc_prev` 的高位 bits（tagged pointer 技术利用了指针对齐的低 2 位存 flags）。

**Step 2: subtract_refs — 减去内部引用**

```c
/* Subtract internal references from gc_refs.  After this, gc_refs is >= 0
 * for all objects in containers. The ones with gc_refs > 0 are directly
 * reachable from outside containers, and so can't be collected.
 */
static void
subtract_refs(PyGC_Head *containers)
{
    traverseproc traverse;
    PyGC_Head *gc = GC_NEXT(containers);
    for (; gc != containers; gc = GC_NEXT(gc)) {
        PyObject *op = FROM_GC(gc);
        traverse = Py_TYPE(op)->tp_traverse;
        (void) traverse(op,
                        visit_decref,   // 对每个子对象执行 gc_refs--
                        op);
    }
}
```

`tp_traverse` 是每种 container 类型实现的回调，枚举其所有子对象引用。`visit_decref` 把子对象的 `gc_refs` 减 1。

**为什么这样能找到循环垃圾？**

设有循环：`A → B → A`，两者均只被彼此引用：
- `update_refs` 后：`A.gc_refs = 1, B.gc_refs = 1`（来自 ob_refcnt）
- `subtract_refs` 后：
  - 遍历 A 的出边 → B.gc_refs -= 1 → B.gc_refs = 0
  - 遍历 B 的出边 → A.gc_refs -= 1 → A.gc_refs = 0
- 结论：两者 gc_refs 都为 0 → 不可达，是垃圾！

**Step 3: move_unreachable — 分离可达/不可达**

```c
static void
move_unreachable(PyGC_Head *young, PyGC_Head *unreachable)
{
    PyGC_Head *prev = young;
    PyGC_Head *gc = GC_NEXT(young);

    /* Invariants: all objects "to the left" of us in young are reachable
     * (directly or indirectly) from outside the young list as it was at entry.
     */

    while (gc != young) {
        if (gc_get_refs(gc)) {
            /* gc is definitely reachable from outside the original 'young'.
             * Mark it as such, and traverse its pointers to propagate
             * reachability to any objects it refers to that are still
             * tentatively unreachable.
             */
            PyObject *op = FROM_GC(gc);
            traverseproc traverse = Py_TYPE(op)->tp_traverse;
            (void) traverse(op, visit_reachable, (void *)young);
            _PyGCHead_SET_PREV(gc, prev);
            gc_clear_collecting(gc);  // 清除 COLLECTING 标志
            prev = gc;
        }
        else {
            /* This *may* be unreachable.  To make progress,
             * assume it is.  visit_reachable will move gc back
             * into young if that's so.
             */
            prev->_gc_next = gc->_gc_next;
            // 移入 unreachable 链表（NEXT_MASK_UNREACHABLE 标记链表类型）
            PyGC_Head *last = GC_PREV(unreachable);
            last->_gc_next = (NEXT_MASK_UNREACHABLE | (uintptr_t)gc);
            _PyGCHead_SET_PREV(gc, last);
            gc->_gc_next = (NEXT_MASK_UNREACHABLE | (uintptr_t)unreachable);
            unreachable->_gc_prev = (uintptr_t)gc;
        }
        gc = (PyGC_Head*)prev->_gc_next;
    }
    young->_gc_prev = (uintptr_t)prev;
    unreachable->_gc_next &= ~NEXT_MASK_UNREACHABLE;
}
```

注意 `visit_reachable`：当一个可达对象（gc_refs > 0）的出边指向当前"候选垃圾"列表中的对象时，说明该对象实际上是可达的，需要把它移回 young 列表——这是一个不动点扩散（fixpoint propagation），保证不漏标。

### 5.3 Tagged Pointer 技术

CPython 在 `_gc_prev` 中通过低 2 位 bits 编码状态标志（`PREV_MASK_COLLECTING`, `PREV_MASK_UNREACHABLE`），高位存真实指针。这依赖 `PyGC_Head` 的内存对齐保证低位为零。这是一种非常典型的空间节省技术，在 V8、LLVM 等中广泛使用。

## 6. 并发与增量 GC：写屏障与 Tri-Color Invariant

### 6.1 STW 的代价

Stop-The-World 暂停时间与存活对象数量线性相关。对于大堆（数 GB），单次 full GC 可以导致秒级 pause，在低延迟服务中完全不可接受。

### 6.2 增量 GC（Incremental GC）

把 GC 工作切割成小片段，与 mutator（应用线程）交替执行。核心问题：**mutator 执行期间可能修改对象图**，破坏 tri-color invariant。

**危险场景**：

```
GC 状态：A（black）→ B（gray），C（white）
Mutator 执行：
  1. A.ref = C        # black 对象 A 新增了对 white 对象 C 的引用
  2. B.ref = null     # 删除了 B 对 C 的引用
GC 继续：
  扫描完 B（已 gray），发现 C 无被 gray 引用 → C 判定为垃圾
  但 C 实际上通过 A 可达！→ 错误回收！
```

### 6.3 写屏障（Write Barrier）

写屏障是在 mutator 执行对象图写操作时插入的一段 GC 协作代码，目的是维护 tri-color invariant。

**Dijkstra（Incremental）写屏障**：

```python
def write(src, field, new_value):
    if gc_phase == MARKING:
        if new_value.color == WHITE:
            new_value.color = GRAY    # 新引用对象染灰，防止漏标
            gray_queue.append(new_value)
    src.field = new_value
```

维护 **Weak Tri-Color Invariant**：black 对象可以引用 white 对象，但 white 对象必须被 gray 对象直接或间接保护。

**Yuasa（Snapshot-At-The-Beginning, SATB）删除屏障**：

```python
def write(src, field, new_value):
    old_value = src.field
    if gc_phase == MARKING:
        if old_value != null and old_value.color == WHITE:
            old_value.color = GRAY    # 被删除的引用对象染灰（SATB 快照语义）
            gray_queue.append(old_value)
    src.field = new_value
```

G1 GC 使用 SATB 写屏障。SATB 的语义是：标记以 GC 开始时的对象图快照为准，快照后被删除的引用也视为可达（保守但正确）。

### 6.4 并发 GC 的代价

现代并发 GC（G1, ZGC, Shenandoah）的写屏障已变得非常轻量（单条 store + 条件分支），但仍然有成本：

- **ZGC**：使用 Load Barrier（读屏障）+ colored pointers，每次读取指针时检查是否需要 relocation。
- **Shenandoah**：使用 Brooks Forwarding Pointer，每次读/写都通过一层间接（indirection），支持并发 evacuation。

## 7. GC Safety 与编译器的协作

### 7.1 GC Safety 是什么

当 GC 运行时，它需要准确知道当前所有指向堆对象的指针（GC roots）。如果编译器把一个堆指针优化到寄存器里，GC 必须能找到它；如果 GC 移动了对象（moving GC），编译器生成的代码必须能被通知到更新指针。

这种"GC 与编译器互知"的机制称为 **GC Safety**（或 GC Safepoints）。

### 7.2 Safepoint

Safepoint 是程序执行中某些特定位置，在这些位置 GC 可以安全地暂停 mutator：

- 所有活跃对象指针（live pointers）对 GC 可见
- 编译器在这些位置生成 **stack maps**（也称 GC maps），描述哪些寄存器/栈槽包含 GC 引用

JVM HotSpot 使用 poll-based safepoints：

```asm
# 生成的代码中，每次循环回边（back-edge）或方法调用前插入：
mov  rax, [safepoint_page]   # 读取 safepoint page
# 如果 GC 想暂停，把 safepoint_page 标记为不可读
# 触发 SIGSEGV → JVM 信号处理器接管 → 线程进入 safepoint
```

### 7.3 编译器如何生成 Stack Maps

以 LLVM statepoints 为例（用于 GHC/Julia 等使用 LLVM 后端且有 GC 的语言）：

```llvm
; LLVM Statepoint IR（示意，非逐字）
; 把 GC_call 包装成 statepoint，描述哪些 SSA 值是 GC refs
%sp = call token (i64 0, i32 0) @llvm.experimental.gc.statepoint.p0f_isVoidf(
    void ()* @gc_func,
    i32 0,
    i32 0,
    ...
    i8 addrspace(1)* %live_obj_1,    ; 这些是 GC roots
    i8 addrspace(1)* %live_obj_2
)
; 在 statepoint 之后通过 gc.relocate 重新获取（可能已被移动的）指针
%obj1_relocated = call i8 addrspace(1)* @llvm.experimental.gc.relocate(
    token %sp,
    i32 4, i32 4      ; 基指针和衍生指针的索引
)
```

`gc.relocate` 告诉 LLVM：如果 GC 移动了对象，必须用新地址替换这个 SSA 值。这要求编译器在 statepoint 之后不能复用 statepoint 之前的指针值。

### 7.4 Conservative vs Precise GC

**Conservative GC**（Boehm-Demers-Weiser GC）：
- 不需要类型信息，把所有看起来像指针的值都当 GC root
- 优点：可用于 C/C++ 等没有类型标注的语言
- 缺点：可能有 false positives（整数碰巧像指针），无法 compact（因为无法确定是否真的是指针，不敢更新它）

**Precise GC**：
- 编译器提供精确的类型信息 / stack maps
- 可以 compact，可以并发移动

CPython 是 conservative 的（它不移动对象），Lua 5.x 是 precise 的（有类型标注）。

## 8. Finalizer、弱引用与对象复活

### 8.1 Finalizer 的问题

Finalizer（Python 的 `__del__`，Java 的 `finalize()`）允许对象在被回收前执行清理逻辑。这引入了一系列复杂问题：

**问题 1：回收顺序不确定**  
如果循环中的多个对象都有 finalizer，调用顺序无法预期。Python 2.x 把这类对象扔到 `gc.garbage` 列表而不回收。

**问题 2：对象复活（Resurrection）**  
Finalizer 可以把 `self` 赋给全局变量，导致对象"死而复生"：

```python
class Zombie:
    def __del__(self):
        Zombie.instance = self   # 我复活了！
```

CPython 的解决方案（来自 garbage_collector.md，六步流程）：
1. 处理 weakref 回调（可达 weakref 的回调先执行）
2. 把有 legacy `tp_del` 的对象移到 `gc.garbage`（不回收）
3. 调用 `tp_finalize`（`__del__`）；标记为已 finalize
4. **重新检测复活**：重跑可达性分析
5. 清理指向不可达对象的 weakref
6. 对剩余不可达对象调用 `tp_clear`（打断循环引用）

### 8.2 弱引用（Weak Reference）

弱引用不阻止 GC 回收对象，当对象被回收时弱引用变为 null（Python 中抛 `ReferenceError`）。

实现：GC 维护一个 weakref 列表。在回收对象之前，先 nullify 所有指向该对象的弱引用。

```python
import weakref

class Foo:
    pass

obj = Foo()
ref = weakref.ref(obj)
print(ref())   # <Foo object>
del obj
print(ref())   # None
```

### 8.3 Java 的四层引用强度

Java 把引用强度分为四级（`java.lang.ref`）：

| 类型 | 回收时机 | 典型用途 |
|------|---------|---------|
| Strong Reference | 永不自动回收 | 普通变量 |
| SoftReference | 内存不足时 | Cache（内存压力触发清理）|
| WeakReference | 下次 GC | 规范化 map（WeakHashMap）|
| PhantomReference | Finalization 后 | 替代 finalizer，做清理工作 |

## 9. 方案对比：全景对照表

### 9.1 GC 算法对比

| 算法 | 时间复杂度 | 空间开销 | 碎片 | 最大 Pause | 适用场景 |
|------|-----------|---------|------|-----------|---------|
| 引用计数 | O(1) per update | 1 word/object | 有 | 无 STW（但析构链可能长）| Swift ARC、CPython（非循环部分）|
| Mark-Sweep | O(live + heap) | O(1) 额外 | 有 | Full heap 扫描 | 简单实现、小堆 |
| Mark-Compact | O(live + heap) | O(1) 额外 | 无 | Full heap 扫描 | 追求无碎片 |
| Copying | O(live) | 2x 内存 | 无 | 与存活对象成比例 | 年轻代（Eden → Survivor）|
| 代际 | O(young) 常见 | 多代结构 | 可能 | 取决于代际配置 | 大多数现代 JVM/Python |
| 增量/并发 | O(live) 增量 | 写屏障开销 | 取决于是否 compact | <10ms（G1）/ <1ms（ZGC）| 低延迟服务 |
| Conservative | O(live + heap) | 无需 type info | 有 | Full heap 扫描 | C/C++ 配 Boehm GC |

### 9.2 语言 GC 策略对比

| 语言/运行时 | 主策略 | 代际 | 并发 | 移动 | 备注 |
|-----------|-------|------|------|------|------|
| CPython | 引用计数 + 循环追踪 | 是（3代）| 否（GIL）| 否 | Python 3.13 开始实验 no-GIL |
| Java HotSpot | 分代 + G1/ZGC | 是 | 是 | 是 | ZGC 目标 <1ms pause |
| Go | 并发三色标记 | 否（单代）| 是 | 否 | 牺牲吞吐量换延迟 |
| V8（JS）| 分代（Scavenger + MC）| 是 | 是（Oilpan）| 是 | Oilpan 用于 DOM 对象 |
| Lua 5.4 | 增量 Mark-Sweep | 否 | 否 | 否 | 简洁，适合嵌入 |
| .NET CLR | 分代 Server GC | 是（4代）| 是 | 是 | Gen 3 = LOH 大对象堆 |
| Rust | 无 GC（所有权）| N/A | N/A | N/A | 编译期排除 |
| Swift | ARC + 弱引用 | N/A | N/A | N/A | 循环需手动打断 |

### 9.3 何时不用 GC

- **实时系统**（硬实时，如飞控、医疗设备）：GC pause 不可预测，禁用。
- **内存极度受限**（IoT, MCU）：GC 本身的 metadata 开销不可接受。
- **Rust 的场景**：生命周期在编译期可证明时，根本不需要运行时 GC。

### 10.1 Demo 1：基础 Mark-Sweep GC（30 行核心）

**文件**：`demo_gc_markswep.py`

```python
#!/usr/bin/env python3
"""
Demo: Minimal Mark-Sweep GC in Python

模拟一个简单运行时：
- Object 持有引用列表（children）
- GCHeap 持有全部分配的对象
- mark_sweep() 执行一次完整的 GC 周期

运行: python3 demo_gc_marksweep.py
预期输出:
  Allocated: A B C D E
  === GC Cycle 1 ===
  Roots: ['A', 'B']
  Marking from root: A
    Marking: A (white -> gray)
    Marking: C (white -> gray)
    Marking: E (white -> gray)
  Marking from root: B
    Marking: B (white -> gray)
    Marking: D (white -> gray)
  Sweep phase:
    [KEEP] A (reachable)
    [FREE] B is reachable -- wait, B ref'd by root
    [FREE] Collecting: unreachable_obj
  Live objects after GC: ['A', 'B', 'C', 'D', 'E']
"""

from __future__ import annotations
from typing import Optional

# ── 颜色常量（Tri-Color Mark） ──────────────────────────────────────────────
WHITE = 0   # 未访问 / 候选垃圾
GRAY  = 1   # 已发现，子对象未完全处理
BLACK = 2   # 已完全处理

COLOR_NAME = {WHITE: "white", GRAY: "gray", BLACK: "black"}


class GCObject:
    """堆上的托管对象"""

    def __init__(self, name: str):
        self.name = name
        self.color: int = WHITE
        self.children: list[GCObject] = []   # 对象图的出边

    def add_ref(self, child: "GCObject") -> None:
        self.children.append(child)

    def __repr__(self) -> str:
        return f"Obj({self.name}, {COLOR_NAME[self.color]})"


class GCHeap:
    """
    极简堆：维护所有已分配对象的链表（对应 CPython 的 gc_generation 链表）。
    """

    def __init__(self):
        self.objects: list[GCObject] = []   # 所有已分配对象
        self.roots:   list[GCObject] = []   # GC roots（模拟栈/全局变量）
        self.collected_total = 0

    def alloc(self, name: str) -> GCObject:
        obj = GCObject(name)
        self.objects.append(obj)
        print(f"  [alloc] {obj.name}")
        return obj

    # ── Mark Phase ─────────────────────────────────────────────────────────

    def _mark_one(self, obj: GCObject, gray_queue: list) -> None:
        """把对象染灰并入队（对应 CPython visit_reachable）"""
        if obj.color == WHITE:
            print(f"    Marking: {obj.name} (white -> gray)")
            obj.color = GRAY
            gray_queue.append(obj)

    def _trace(self, gray_queue: list) -> None:
        """BFS: 处理灰色队列直到为空（对应 clox traceReferences）"""
        while gray_queue:
            obj = gray_queue.pop(0)   # dequeue
            # 处理所有出边，把白色子对象染灰
            for child in obj.children:
                self._mark_one(child, gray_queue)
            obj.color = BLACK         # 出边处理完毕，染黑
            print(f"    Blackened: {obj.name}")

    def mark(self) -> None:
        """Mark 阶段：从所有 roots 出发"""
        for root in self.roots:
            print(f"  Marking from root: {root.name}")
            gray_queue: list[GCObject] = []
            self._mark_one(root, gray_queue)
            self._trace(gray_queue)

    # ── Sweep Phase ────────────────────────────────────────────────────────

    def sweep(self) -> int:
        """Sweep 阶段：回收 white 对象，重置 black → white（对应 clox sweep）"""
        survived = []
        freed = 0
        for obj in self.objects:
            if obj.color == BLACK:
                obj.color = WHITE     # 重置为白，准备下一轮
                survived.append(obj)
                print(f"    [KEEP] {obj.name}")
            else:
                print(f"    [FREE] Collecting: {obj.name}")
                freed += 1
        self.objects = survived
        self.collected_total += freed
        return freed

    # ── Full GC Cycle ──────────────────────────────────────────────────────

    def collect(self, label: str = "GC") -> int:
        print(f"\n=== {label} ===")
        print(f"  Roots: {[r.name for r in self.roots]}")
        print(f"  Heap size before: {len(self.objects)}")

        # Phase 1: reset all to white（CPython 的 update_refs 简化版）
        for obj in self.objects:
            obj.color = WHITE

        # Phase 2: mark from roots
        print("  [Mark phase]")
        self.mark()

        # Phase 3: sweep
        print("  [Sweep phase]")
        freed = self.sweep()

        print(f"  Freed: {freed}, Live: {len(self.objects)}")
        print(f"  Live objects: {[o.name for o in self.objects]}")
        return freed


# ── Demo 场景 ───────────────────────────────────────────────────────────────

def demo_basic():
    """场景 1：基础可达性——孤立对象被回收"""
    print("=" * 60)
    print("DEMO 1: Basic reachability")
    heap = GCHeap()

    # 分配对象
    a = heap.alloc("A")
    b = heap.alloc("B")
    c = heap.alloc("C")
    orphan = heap.alloc("orphan")  # 无任何引用

    # 建立引用图：A → B → C
    a.add_ref(b)
    b.add_ref(c)

    # 只有 A 是 root
    heap.roots = [a]

    heap.collect("GC Cycle 1")
    assert [o.name for o in heap.objects] == ["A", "B", "C"], "错误：应保留 A B C"
    print("  [PASS] orphan was collected")


def demo_cycle():
    """场景 2：循环引用——mark-sweep 正确回收，引用计数无法处理"""
    print("\n" + "=" * 60)
    print("DEMO 2: Cycle detection (引用计数做不到，mark-sweep 可以)")
    heap = GCHeap()

    # 根可达对象
    root_obj = heap.alloc("root")

    # 循环：X → Y → X（且无外部引用）
    x = heap.alloc("X")
    y = heap.alloc("Y")
    x.add_ref(y)
    y.add_ref(x)

    heap.roots = [root_obj]  # X, Y 不在 roots 中

    heap.collect("GC Cycle 2")
    names = [o.name for o in heap.objects]
    assert "X" not in names and "Y" not in names, "错误：X Y 应被回收"
    assert "root" in names, "错误：root 应保留"
    print("  [PASS] cycle X->Y->X was collected")


def demo_resurrection_guard():
    """
    场景 3：演示 finalizer 语义（简化）
    真实的对象复活（resurrection）需要 finalizer 把 self 存入全局，
    这里只演示"复活后重新加回 roots 需重跑 GC"的逻辑。
    """
    print("\n" + "=" * 60)
    print("DEMO 3: Resurrection-style re-rooting")
    heap = GCHeap()

    survivor = heap.alloc("survivor")
    doomed   = heap.alloc("doomed")
    heap.roots = [survivor]

    heap.collect("Before resurrection")
    # doomed 被回收

    # 模拟 finalizer "复活" doomed（重新 alloc）
    resurrected = heap.alloc("resurrected")
    heap.roots.append(resurrected)   # finalizer 把它加回了 roots

    heap.collect("After resurrection")
    names = [o.name for o in heap.objects]
    assert "resurrected" in names
    print("  [PASS] resurrected object survived second GC")


if __name__ == "__main__":
    demo_basic()
    demo_cycle()
    demo_resurrection_guard()
    print("\n所有 Demo 通过。")
```

### 10.2 Demo 2：模拟 CPython Cycle Detector（引用计数差分）

这个 demo 直接模拟 CPython 的三步算法：update_refs → subtract_refs → move_unreachable。

**文件**：`demo_cpython_cycle_detector.py`

```python
#!/usr/bin/env python3
"""
Demo: CPython-style Cycle Detector
直接模拟 Python/gc.c 的三步算法：
  1. update_refs: 把 ob_refcnt 拷贝到 gc_refs
  2. subtract_refs: 通过 tp_traverse 减去内部引用
  3. move_unreachable: gc_refs == 0 的移入垃圾候选

运行: python3 demo_cpython_cycle_detector.py
"""

from __future__ import annotations


class TrackedObject:
    """模拟 CPython PyGC_Head 管理的容器对象"""

    def __init__(self, name: str):
        self.name = name
        self.ob_refcnt: int = 0          # 模拟真实引用计数
        self.gc_refs:   int = 0          # 临时 GC 引用计数（update_refs 后使用）
        self.children: list[TrackedObject] = []

    def add_ref(self, child: "TrackedObject") -> None:
        self.children.append(child)
        child.ob_refcnt += 1             # 模拟 Py_INCREF

    def __repr__(self) -> str:
        return f"Obj({self.name}, ob_refcnt={self.ob_refcnt}, gc_refs={self.gc_refs})"


class CPythonCycleDetector:
    """
    模拟 CPython gc.c 的循环检测核心逻辑
    （不包含代际结构，只演示单代算法）
    """

    def __init__(self):
        self.generation: list[TrackedObject] = []    # 追踪的 container 对象
        self.external_roots: list[TrackedObject] = [] # 外部根（模拟栈/全局）

    def track(self, obj: TrackedObject) -> TrackedObject:
        """模拟 _PyObject_GC_TRACK"""
        self.generation.append(obj)
        obj.ob_refcnt += 1              # 被 generation 链表持有
        return obj

    def add_external_root(self, obj: TrackedObject) -> None:
        """外部根引用（模拟栈变量）"""
        self.external_roots.append(obj)
        obj.ob_refcnt += 1

    # ── Step 1 ──────────────────────────────────────────────────────────────

    def update_refs(self) -> None:
        """
        对应 Python/gc.c: update_refs()
        gc_refs = ob_refcnt（减去 generation 链表自身持有的 1）
        """
        print("\n[Step 1] update_refs")
        for obj in self.generation:
            # 减去 generation 链表持有的那 1 个引用，得到"外部真实引用计数"
            obj.gc_refs = obj.ob_refcnt - 1
            print(f"  {obj.name}: ob_refcnt={obj.ob_refcnt} → gc_refs={obj.gc_refs}")

    # ── Step 2 ──────────────────────────────────────────────────────────────

    def subtract_refs(self) -> None:
        """
        对应 Python/gc.c: subtract_refs()
        对每个 container 遍历其子对象，把子对象的 gc_refs -= 1
        （这样循环内部的互相引用被消去）
        """
        print("\n[Step 2] subtract_refs")
        for obj in self.generation:
            for child in obj.children:
                if child in self.generation:   # 只处理被追踪的对象
                    child.gc_refs -= 1
                    print(f"  {obj.name} → {child.name}: {child.name}.gc_refs = {child.gc_refs}")

    # ── Step 3 ──────────────────────────────────────────────────────────────

    def move_unreachable(self) -> tuple[list, list]:
        """
        对应 Python/gc.c: move_unreachable()
        gc_refs == 0 → 候选垃圾；gc_refs > 0 → 可达
        然后从可达对象扩散，把间接可达的对象也标为可达
        """
        print("\n[Step 3] move_unreachable")

        reachable   = []
        unreachable = []

        # 第一轮：按 gc_refs 分类
        for obj in self.generation:
            if obj.gc_refs > 0:
                reachable.append(obj)
            else:
                unreachable.append(obj)

        # 扩散：从 reachable 出发，把间接可达的 unreachable 对象拉回来
        # （对应 CPython 的 visit_reachable 回调）
        worklist = list(reachable)
        rescued = set()

        while worklist:
            obj = worklist.pop()
            for child in obj.children:
                if child in unreachable and child not in rescued:
                    print(f"  Rescue: {child.name} (reachable via {obj.name})")
                    rescued.add(child)
                    worklist.append(child)

        # 从 unreachable 移除被救的对象
        final_unreachable = [o for o in unreachable if o not in rescued]
        final_reachable   = reachable + list(rescued)

        print(f"\n  Reachable: {[o.name for o in final_reachable]}")
        print(f"  Garbage:   {[o.name for o in final_unreachable]}")
        return final_reachable, final_unreachable

    def collect(self) -> list[TrackedObject]:
        """完整的一次循环检测周期"""
        print("=" * 50)
        print("CPython Cycle Detector — collect()")
        print(f"Generation: {[o.name for o in self.generation]}")
        print(f"External roots: {[o.name for o in self.external_roots]}")

        # 外部根加入 reachable（不在 subtract_refs 减去的范围）
        for root in self.external_roots:
            root.gc_refs = root.ob_refcnt  # roots 的引用来自外部，不减

        self.update_refs()
        self.subtract_refs()
        reachable, garbage = self.move_unreachable()

        print(f"\nCollecting {len(garbage)} objects:")
        for obj in garbage:
            print(f"  [FREE] {obj.name}")
            self.generation.remove(obj)

        return garbage
```

### 10.3 Demo 3：模拟 clox tri-color Mark（Crafting Interpreters 风格）

clox 的 GC 和上面的 mark-sweep 基本思路一致，但有几个关键工程细节：

**真实 clox 源码（crafting-interpreters/c/memory.c）**：

【真实源码 munificent/craftinginterpreters@c/memory.c】

```c
void collectGarbage() {
#ifdef DEBUG_LOG_GC
  printf("-- gc begin\n");
  size_t before = vm.bytesAllocated;
#endif

  markRoots();          // 把所有根染灰
  traceReferences();    // BFS 处理灰色队列直到为空
  tableRemoveWhite(&vm.strings); // 从 string intern table 移除白色对象
  sweep();              // 回收白色对象

  vm.nextGC = vm.bytesAllocated * GC_HEAP_GROW_FACTOR;   // 更新下次 GC 阈值
}

// 标记所有根
static void markRoots() {
  for (Value* slot = vm.stack; slot < vm.stackTop; slot++) {
    markValue(*slot);   // 扫描值栈上的所有对象
  }
  for (int i = 0; i < vm.frameCount; i++) {
    markObject((Obj*)vm.frames[i].closure);  // 每个调用帧的 closure
  }
  for (ObjUpvalue* upvalue = vm.openUpvalues;
       upvalue != NULL;
       upvalue = upvalue->next) {
    markObject((Obj*)upvalue);  // 开放的 upvalue（闭包捕获的变量）
  }
  markTable(&vm.globals);       // 全局变量表
  markCompilerRoots();          // 编译器自身持有的对象（编译期 GC）
  markObject((Obj*)vm.initString);
}

// BFS 追踪灰色对象
static void traceReferences() {
  while (vm.grayCount > 0) {
    Obj* object = vm.grayStack[--vm.grayCount];
    blackenObject(object);  // 处理出边，子对象染灰；自身染黑
  }
}

// Sweep：遍历对象链表，回收未标记（isMarked == false）的对象
static void sweep() {
  Obj* previous = NULL;
  Obj* object = vm.objects;     // vm.objects 是所有堆对象的单链表头
  while (object != NULL) {
    if (object->isMarked) {
      object->isMarked = false;  // 重置标记
      previous = object;
      object = object->next;
    } else {
      Obj* unreached = object;
      object = object->next;
      if (previous != NULL) {
        previous->next = object;
      } else {
        vm.objects = object;
      }
      freeObject(unreached);     // 回收
    }
  }
}
```

**关键设计点**：

1. **灰色工作队列（vm.grayStack）**：clox 用一个动态数组作为灰色对象队列，避免 DFS 递归的栈溢出风险。这和 CPython 用 `tp_traverse` 的扩散逻辑异曲同工。

2. **编译器也是 GC 根**（`markCompilerRoots()`）：编译期间，parser 持有的函数对象也是 GC root——否则在编译期触发 GC 会回收正在编译的函数。这是一个常见的工程坑。

3. **String Intern Table 的弱引用语义**（`tableRemoveWhite`）：clox 的 string interning 表持有 string 的"弱引用"，在 sweep 之前先清理掉即将被回收的 string 条目，避免 dangling pointer。这是弱引用语义在实际 VM 中的典型应用。

4. **GC 触发策略**（`vm.nextGC = vm.bytesAllocated * GC_HEAP_GROW_FACTOR`）：每次 GC 后把下次触发阈值设为当前存活大小的 N 倍（默认 2）。这保证了 GC 频率与内存压力成比例，而非固定周期。

## 11. 真实工程坑与失败模式

### 11.1 Finalizer 导致的 GC 暂停尖峰

**现象**：某服务 full GC 后出现不规律的长尾 pause（比预期 GC 时间多 10-100ms）。

**根因**：大量对象带有 `__del__` / `finalize()`，GC 在 sweep 阶段串行执行 finalizer，且 finalizer 可能持锁、做 I/O、甚至触发新的对象分配。

**解决方案**：
- Java：使用 `Cleaner`（JDK 9+）替代 `finalize()`，`Cleaner` 在单独线程执行清理逻辑。
- Python：使用 `weakref.finalize()` 或 context manager (`with`) 替代 `__del__`。
- 通用：资源清理要显式（RAII/`using`/`with`），不要依赖 GC 时机。

### 11.2 Boehm GC 的 False Pointer 问题

**场景**：C 程序用 Boehm GC，某个 `int64_t` 的值恰好等于堆上某个对象的地址。

**后果**：Boehm 会把这个整数当做指针，认为那个对象可达，永远不会回收它（内存泄漏，而非崩溃）。

**定位**：用 `GC_dump()` 或 `GC_find_leak` 模式运行，找出哪些对象从未被回收但也没有明确持有者。

### 11.3 Write Barrier 漏打导致 GC 误回收

**场景**：在实现并发 GC 的 VM 中，某处直接修改了对象的引用字段（`obj->field = new_val`），没有走写屏障路径。

**后果**：新插入的引用可能是 white 对象，且没被染灰，GC 误以为它不可达而回收。导致悬空指针（dangling pointer）崩溃或数据损坏。

**预防**：
- 在 IR 层强制所有堆写操作经过 write barrier intrinsic（参见 LLVM `@llvm.experimental.gc.statepoint`）。
- 用 sanitizer（如 `-fsanitize=address`）配合 GC 测试套件，加大 GC 频率（设小 GC 阈值）放大问题窗口。

### 11.4 Safepoint 位置不足导致 GC 长等待

**场景**：一段紧密的数值循环（无函数调用、无分支检查），GC 想触发 stop-the-world，但 mutator 线程一直在循环体内运行，迟迟到不了 safepoint。

**后果**：GC 等待时间超长（"time to safepoint" spike），所有其他线程已暂停但在等一个慢线程。

**解决**：在循环回边（back-edge）强制插入 safepoint poll；或使用 signal-based safepoints（ZGC 风格）。

### 11.5 代际 GC 的 Remembered Set 膨胀

**场景**：大量老年代对象持有年轻代对象的引用（如：全局 cache 指向临时结果）。

**后果**：Remembered Set / Card Table 几乎全标为 dirty，Young GC 退化为扫描整个老年代，失去代际优化效益。

**解决**：检查业务逻辑，避免将大量短命对象的引用存入长期存在的老年代容器（典型场景：长生命周期 Map 的 value 是临时分配的对象）。

### 11.6 CPython GC 与 `__del__` 的历史包袱

CPython 3.4 之前，如果循环引用中有任何对象定义了 `__del__`，整个循环都不会被回收（放入 `gc.garbage` 列表）。CPython 3.4（PEP 442）改进了 finalizer 的安全调用顺序，解决了这个长期 gotcha，但旧代码中仍可能踩到。

## 12. 章末五件套

### 12.1 核心心智模型

GC 的本质是**可达性问题**：一个对象是否有从 GC roots 出发的引用路径。引用计数是对可达性的局部近似，追踪式 GC 是对可达性的全局精确计算。两者的混合（CPython 的双层结构）是工程权衡：用引用计数处理大多数无环对象（快速、局部），用追踪式 GC 处理少数有环对象（彻底、批量）。

### 12.2 关键数据 / 阈值

- CPython 默认 GC 阈值：`(700, 10, 10)`，即 700 次分配触发 gen0，gen0 收集 10 次触发 gen1，gen1 收集 10 次触发 full GC。
- clox GC 增长因子：`GC_HEAP_GROW_FACTOR = 2`，每次 GC 后下次阈值 = 当前存活 × 2。
- G1 GC 默认 max pause time goal：200ms；ZGC 目标：< 1ms。
- JVM Safepoint 轮询点：方法调用、循环回边；ZGC 使用 load barrier 而非 safepoint。

### 12.4 常见面试 / 设计问题

1. **为什么引用计数无法处理循环引用？用一个 3 节点例子证明。**  
   答：循环中每个节点的 refcnt 保持 > 0，即使外部引用为 0，永远不归零。

2. **CPython 的 cycle detector 如何在不移动对象的情况下找到循环垃圾？**  
   答：update_refs 拷贝 refcnt → subtract_refs 消去内部引用 → gc_refs == 0 的即为循环垃圾。

3. **什么是 Tri-Color Invariant？为什么并发 GC 需要写屏障维护它？**  
   答：不变式 = 不存在 black → white 边。mutator 可能创建 black → white 边（新引用赋值），写屏障捕获这一修改，把 white 对象染灰。

4. **G1 和 ZGC 的核心区别是什么？**  
   答：G1 分 region，增量 mixed GC，STW pause 与存活对象量成比例；ZGC 使用 colored pointers + load barrier，evacuation 并发进行，STW 仅用于扫描根（亚毫秒）。

5. **Safepoint 是什么？为什么 JIT 编译的代码需要特别处理 Safepoint？**  
   答：Safepoint 是 GC 可以安全暂停 mutator 的位置，需要 stack map 精确描述活跃 GC 引用。JIT 代码没有解释器的隐式循环检查，需要主动插入 safepoint poll。
