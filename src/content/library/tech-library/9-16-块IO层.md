> **一句话定位**：块 IO 层是文件系统 IO 请求到硬件驱动之间的"交通管制系统"——它决定了谁先走、能不能拼车、路上能不能合并，是磁盘性能调优的命门。这一版不再停留在"它做什么"，而是带你走进 `block/` 目录的真实源码、回到 2001/2013/2018/2019 的几次关键设计辩论现场，把每个机制的"为什么是这样、为什么不是那样、什么时候会崩"讲透。

---

## TL;DR — 本章最硬结论

1. **bio 是内核块 IO 的最小单位**，一个 bio 表达"对哪些（可离散的）内存段，做用于某段磁盘 LBA 的什么操作"；request 是合并 bio 后等待调度/dispatch 的队列元素。两者不是同一层，bio 由 FS 构造，request 由 blk-mq 分配并绑定 tag。
2. **单队列架构（legacy request_fn）在 NVMe 下成为瓶颈**，根本原因不是"线程不够"，而是**一把全局锁保护的一条链表**在多核上 cache line bounce。这句话有原始出处：LWN 2013 年原文说该请求队列"is protected by a single lock which, on a large system, will bounce frequently between the processors. It is a linked list, a notably cache-unfriendly data structure."（[LWN 552904](https://lwn.net/Articles/552904/)）blk-mq 用 per-CPU software staging queue + per-hardware dispatch queue + per-CPU tag cache 消除它。
3. **IO 调度器不是万能药**：NVMe 上多数场景用 `none`，mq-deadline 只在需要"读不被写饿死 + 软 deadline"时上；机械盘混合负载用 BFQ。盲目换调度器不如先量化瓶颈——而且**换调度器不是做 IO QoS 的唯一手段**（见结论 5）。
4. **plugging 是被低估的优化**：进程级 plug 在短窗口内批量积攒 request 一次性提交，减少锁争用和调度器震荡。但它有一条铁律——**进程一旦要睡（`schedule()`）就必须 flush plug**，否则 IO 会被睡眠中的进程无限压住。
5. **cgroup IO 控制有三套正交机制，且大多与调度器无关**：`io.max`（硬限流，blk-throttle）、`io.latency`（延迟保护，blk-iolatency，4.19，rq-qos 框架，**不挑调度器**）、`io.cost`/`io.weight`（按成本比例，blk-iocost，5.4，**不挑调度器**）。只有 v1 的 `blkio.weight` 死绑 CFQ、在 5.0 随 CFQ 被删而失效。"NVMe + none 就做不了 IO QoS" 是错的。

## 16.1 背景：为什么需要一个独立的块 IO 层

### 问题域

磁盘（无论 HDD / SSD / NVMe）和上层文件系统之间有一条语义鸿沟：

- **文件系统**看到的是"逻辑文件偏移 → 文件块号"，它不关心是哪个磁盘、磁头在哪。
- **设备驱动**看到的是"第 N 个 LBA 执行读/写"，它不关心文件是什么。
- 两者之间还有 RAID、LVM、device mapper 等 stacked device，可以无限叠加（甚至递归 submit）。

块 IO 层要在这条鸿沟上同时解决五件事：

| 问题                          | 解法                              | 本章小节 |
|-------------------------------|-----------------------------------|---------|
| FS 层多个小写能不能合并？      | bio merge + request merge         | 16.3.3  |
| 机械盘寻道要不要排序？         | IO scheduler（elevator）          | 16.3.4  |
| 多个进程公平竞争磁盘？         | cgroup io 限流/权重/延迟保护      | 16.5    |
| 多核 NVMe 吞吐被锁限制？       | blk-mq 多队列 + tag               | 16.2 / 16.3.6 |
| 异步 IO 如何和驱动对接？       | 提交/完成路径分离（tag 回调）     | 16.4    |

### 类比（但要小心类比的边界）

把块 IO 层类比成**城市交通枢纽**：bio 是乘客、request 是车厢（合并几个同向乘客）、IO 调度器是发车策略（谁先走）、blk-mq 的 hardware queue 是多条轨道。

⚠️ **类比的失效点**（这正是浮于表面 vs 扎根的分界）：交通枢纽的"轨道"是物理独占的，而 blk-mq 的 software queue 到 hardware queue 是**多对一或一对一的映射**，且真正的并发瓶颈不在"轨道数"而在"售票窗口共享一个账本（全局锁 + 共享 cache line）"。下面整章我们都会回到这个底层根因：**blk-mq 的胜利是 cache locality 的胜利，不是并行度的胜利。**

### 16.2.1 2001：bio 取代 buffer_head——第一次"解耦内存与磁盘"

`block/bio.c` 文件头至今仍写着：

```c
// SPDX-License-Identifier: GPL-2.0
/*
 * Copyright (C) 2001 Jens Axboe <axboe@kernel.dk>
 */
```
【真实源码 v6.12 `block/bio.c` 第 1-3 行】

为什么 2.4 的 `buffer_head` 不够用？`buffer_head` 把"一块内存缓冲"和"一个磁盘块"**一对一绑死**，一个 IO 请求只能描述一块连续内存 ↔ 一块连续磁盘。这在两件事上撞墙：

1. **scatter-gather 做不了**。一次 DMA 想"把 8 个分散在不同物理页的缓冲，写到一段连续磁盘 LBA"，`buffer_head` 表达不了——只能拆成 8 个独立请求，8 倍的提交开销。
2. **高端内存（highmem）撞墙**。32 位时代物理页可能不在内核直接映射区，`buffer_head` 的设计假设缓冲可直接寻址，处理 highmem 要 bounce buffer，丑陋且慢。

`struct bio` 的破局点：它把"内存侧"抽象成一个 **`bio_vec` 数组**（页 + 偏移 + 长度），"磁盘侧"抽象成一个**起始扇区 + 迭代器**。内存可以是 N 段离散页，磁盘是一段连续 LBA——**多对一**。这就是块层第一次真正的"解耦"。

> **扎根点**：这不是"换了个结构体名字"。bio 的 `bio_vec[]` 直接对应硬件 DMA 的 scatter-gather list。从 bio 到驱动的 SG 表几乎是一一映射，这是为什么 `max_segments`（硬件 SG 表能装几段）会成为 merge 的硬约束（见 16.3.3）——**bio 的数据结构形状是被 DMA 引擎的物理约束倒逼出来的**。

### 16.2.2 2013：单队列瓶颈被"系统性"证明——SYSTOR'13 论文与 LWN 552904

2.6 的可插拔调度器架构（CFQ / deadline / anticipatory / noop）在机械盘上跑了将近十年没问题。原因是 HDD 单盘 IOPS < 200，每个 IO 寻道就要 5-10ms，软件侧那把全局锁的几百纳秒开销完全淹没在物理延迟里。

NVMe 把这个平衡彻底打破。2013 年 Axboe 团队（Matias Bjørling、Jens Axboe、David Nellans、Philippe Bonnet）在 **SYSTOR'13** 发表 *"Linux block IO: introducing multi-queue SSD access on multi-core systems"*（[全文 PDF 在 kernel.dk](https://kernel.dk/systor13-final18.pdf)，已核实可下载），论文的目标自述是：设计一个新一代块层，能在单设备多核系统上处理**数千万级 IOPS**，并随核数近线性扩展、在多 socket NUMA 上也成立。

LWN 同年 6 月 5 日由 Jonathan Corbet 写了第一篇系统介绍（[LWN 552904](https://lwn.net/Articles/552904/)，报道 Axboe 与 Shaohua Li 的 patch set）。它对瓶颈的刻画是本章必背的一句原话：

> "[the request queue] is protected by a single lock which, on a large system, will bounce frequently between the processors. It is a linked list, a notably cache-unfriendly data structure."
> ——[LWN 552904, Jonathan Corbet, 2013-06-05]

把这句话拆成两个**底层根因**：

1. **single lock → cache line bounce**。一把 spinlock 物理上是一个 cache line。CPU0 拿锁要把这条 line 拉到自己的 L1（MESI 协议下置为 Modified/Exclusive），CPU7 想拿就得让 CPU0 的 line 失效再拉过来——这就是 cache line bouncing。锁本身没"忙"，但 cache 一致性流量随核数**超线性**增长。这是排队论之外、纯微架构层面的开销。
2. **linked list → cache-unfriendly**。请求队列是链表，遍历/插入要追指针，每跳一个节点很可能是一次 cache miss。多核同时改这条链表，等于让多核共享并反复弄脏同一组 cache line。

> **数字辨析（必背，常考）**：NVMe spec 的三个"上限"不能混：
> - **IO 队列数量**：spec 允许多达 65535 个 IO submission/completion queue 对；实际由驱动按 CPU 数创建，常见"每 CPU 一条"。
> - **单队列深度**：spec 允许每条 IO queue 最多 65536 个 entry；但**真实设备暴露的 queue depth 远小于此**，常见 128 / 256 / 1024，看 controller。Linux 侧查 `cat /sys/block/nvme0n1/queue/nr_requests`。
> - 网上常说的"NVMe 支持 64K 并发"指**队列数 × 深度的理论上界**，不是任何单一维度，更不是某块盘实际能 inflight 的命令数。
>
> 所以单队列块层的尴尬是：**硬件给了你几万条潜在并发通道，软件侧却只有一把锁、一条链表**。论文与 LWN 的整个论证就是要把这道软件墙拆掉。

### 16.2.3 2013-12：blk-mq 落地——读原始 commit message

blk-mq 于 **Linux 3.13** 合入，原始 commit 是 `320ae51feed5c2f13664aa05a76bec198967e04d`，作者 Jens Axboe（[GitHub commit](https://github.com/torvalds/linux/commit/320ae51feed5c2f13664aa05a76bec198967e04d)，已核实）。commit message 里把设计意图说得很直白——它定位自己是块层的**第三种排队模型**，目标是最终取代前两种（经典 `request_fn` 模型与 stacked/make_request 模型）。核心特性（commit 自述，转述）：

- "Per-cpu queues for queueing IO, which then funnel down into x number of hardware submission queues" —— 这就是 software staging queue → hardware dispatch queue 的两层结构。
- **Scalable tagging with per-cpu caches** —— tag 分配自带 per-CPU 缓存，避免单一原子计数成热点（后来演化为 `sbitmap`）。
- **不靠 per-device request 跟踪的 timeout 处理**。
- 支持**非 1:1 的 sw→hw 队列映射**、per-request payload、merge 与 plug。

致谢名单（commit 末尾）：Shaohua Li、Alexander Gordeev、Christoph Hellwig、Mike Christie、Matias Bjørling、Jeff Moyer——基本是当年块层的核心圈子。

> **社区辩论的影子**：注意 commit 强调"a potential 3rd queueing model"——blk-mq **不是一夜之间替换**所有驱动的。它先作为可选路径并存，scsi、nvme、virtio-blk、各驱动**逐个迁移**，单队列代码又活了 5 年才删。这种"新旧并存、逐步迁移、最后大清理"是内核重构的典型节奏，下一段就是清理。
>
> （scsi-mq 在 3.17 引入、默认开启 mq 的精确版本节点「待核」；这类驱动 enablement 时间线散在各自子系统，不影响主线判断。）

### 16.2.4 2019：5.0 大清理——CFQ 为什么"port 不过去"被直接删

到 **Linux 5.0**（2019 初），legacy 单队列块层连同 CFQ / legacy-deadline / noop 一起被删。这不是"待核"，是确定版本，且有确定 commit：`f382fb0bcef4c37dc049e9f6963e3baf204d815c`，subject **"block: remove legacy IO schedulers"**，作者 Jens Axboe（[GitHub commit](https://github.com/torvalds/linux/commit/f382fb0bcef4c37dc049e9f6963e3baf204d815c)，已核实，删了约 6025 行 / 7 个文件，含 `IOSCHED_NOOP`/`IOSCHED_DEADLINE`/`IOSCHED_CFQ` 的配置项）。commit message 短到只留一句技术备注：

> "Retain the deadline documentation, as that carries over to mq-deadline as well." —— [commit f382fb0b]

**为什么 CFQ 不能像 deadline 那样 port 成 mq 版？** 这是 staff 面试的好问题，根因在 CFQ 的核心机制与 mq 语义不兼容：

- CFQ 的公平性建立在**单一时间轴上的时间片轮转 + anticipation（空转等待同进程下一个 IO）**。它假设"我有一条队列、一个全局视角，能精确控制此刻让哪个进程的 IO 出去、并故意停顿几毫秒等它"。
- blk-mq 是**多条 hardware queue 并行 dispatch**，没有"单一时间轴"。CFQ 的 anticipation idle 在多队列下既难协调（idle 哪条队列？）又与"NVMe 要尽量喂满硬件"的目标直接冲突。
- 结果：与其勉强翻译一个语义错位的 CFQ，社区选择用 **BFQ**（4.12 已合入，budget 而非时间片，原生支持 mq + cgroup）承接"机械盘公平调度 + 比例分配"的需求，CFQ 直接删。

这就是"为什么不用看似更简单的替代"——把 CFQ 强行 mq 化，得到的是一个又慢又不公平、还要长期维护的怪物；删掉它、让 BFQ 接班，是更干净的取舍。

## 16.3 现状：blk-mq 核心机制（源码级）

### 16.3.1 IO 栈分层概览

```
用户态 write() / read()
        │
        ▼
   VFS（文件描述符→inode）
        │
        ▼
   文件系统（ext4/xfs/btrfs）——决定哪些块需要读写
        │  submit_bio()
        ▼
   通用块层 Generic Block Layer
   ├── bio 构造与合并
   ├── device mapper / md RAID（可选，透明 stacked，可能递归 submit）
   └── blk-mq 队列管理
        │  submit_bio() → submit_bio_noacct() → blk_mq_submit_bio()
        │  （历史：~5.9 之前经 make_request_fn 间接调用 blk_mq_make_request；
        │   5.9 移除了 make_request_fn 间接层，现在直接是 blk_mq_submit_bio）
        ▼
   IO 调度器（mq-deadline / BFQ / kyber / none）
        │
        ▼
   blk-mq hardware queue dispatch（tag 分配）
        │
        ▼
   设备驱动 queue_rq()（NVMe / SCSI / virtio-blk）
        │
        ▼
   硬件（SSD / NVMe / HDD）
```

**关键观察**：`submit_bio()` 是文件系统→块层的边界；驱动拿到 request 后向硬件提交，通过中断或 polling 回来 `blk_mq_complete_request()`。

### 16.3.2 struct bio — 块 IO 的原子请求（逐行注解真实源码）

这是 v6.12 的真实定义，逐字摘录后逐行注解：

```c
struct bio {
	struct bio		*bi_next;	/* request queue link */
	struct block_device	*bi_bdev;
	blk_opf_t		bi_opf;		/* bottom bits REQ_OP, top bits
						 * req_flags.
						 */
	unsigned short		bi_flags;	/* BIO_* below */
	unsigned short		bi_ioprio;
	enum rw_hint		bi_write_hint;
	blk_status_t		bi_status;
	atomic_t		__bi_remaining;
	struct bvec_iter	bi_iter;
	union {
		blk_qc_t		bi_cookie;
		unsigned int		__bi_nr_segments;
	};
	bio_end_io_t		*bi_end_io;
	void			*bi_private;
	/* ... CONFIG_BLK_CGROUP / INLINE_ENCRYPTION / DEV_INTEGRITY 条件字段省略 ... */
	unsigned short		bi_vcnt;
	unsigned short		bi_max_vecs;
	atomic_t		__bi_cnt;
	struct bio_vec		*bi_io_vec;
	struct bio_set		*bi_pool;
	struct bio_vec		bi_inline_vecs[];
};
```
【真实源码 v6.12 `include/linux/blk_types.h`】

逐字段读出设计意图：

- `bi_next` —— bio 可以串成链（一个 request 里可能挂多个 bio）。
- `bi_bdev` —— 目标块设备。stacked device（dm/md）转发 bio 时会改写它。
- `bi_opf` —— **一个字段塞两样东西**：低 8 bit 是操作（`REQ_OP_READ`/`WRITE`/`FLUSH`/`DISCARD`/zone ops…），高 24 bit 是 flag（`REQ_SYNC`/`REQ_FUA`/`REQ_PREFLUSH`/`REQ_NOWAIT`/`REQ_POLLED`…）。`blk_opf_t` 实为 `__u32 __bitwise`。这种打包是为了让 bio 更紧凑、一次原子读就拿到操作+语义。
- `bi_status` —— `blk_status_t` 是 `u8 __bitwise`，取值 `BLK_STS_OK`(0) 到 `BLK_STS_INVAL`(19) 等。注意它**不是** errno，是块层自己的状态码，到上层才翻译成 `-EIO` 之类。
- `__bi_remaining` / `__bi_cnt` —— 两个原子计数，分别管"还有多少未完成的子部分"（用于 bio 链/split 后的 endio 收敛）和引用计数。带 `__` 前缀是约定：不要直接碰，用 helper。
- `bi_iter`（`struct bvec_iter`）—— **当前迭代位置**：起始扇区 `bi_sector`、剩余字节、`bi_io_vec` 下标。这是本结构最精妙处：bio 被 `__bio_split_to_limits()` 切分时，**不复制数据、不复制 bio_vec 数组**，只是推进/克隆这个 iter。split 是 O(1) 的指针游戏。
- `union { bi_cookie; __bi_nr_segments; }` —— 同一块空间在不同生命周期阶段复用：提交前后语义不同（polling cookie vs 段数缓存），省内存。
- `bi_io_vec` —— **指向段数组的指针，遍历用它**。
- `bi_inline_vecs[]` —— flexible array，**仅是内联存储**。段数少时 `bi_io_vec` 指向 `bi_inline_vecs`，省一次单独分配；段数多时另外从 `bi_pool` 分配，`bi_io_vec` 指向那块。

💡 **坑点 1（源码级，必考）**：别把 `bi_io_vec`（遍历用的指针）和 `bi_inline_vecs[]`（内联存储）搞混。看到代码里 `bio->bi_io_vec == bio->bi_inline_vecs` 是"小 bio 走了内联优化"的判据，不是 bug。
💡 **坑点 2**：`bi_sector` 这个老字段早已移进 `bi_iter.bi_sector`，读老代码注意。bio 不必然对应物理连续磁盘区域（虽然单个 bio 的 LBA 是连续的，但 split / merge 后语义在 request 层组合），也不必然连续内存（scatter-gather 的全部意义）。
💡 **坑点 3（时效）**：~5.x 起 page cache 与块层 folio 化，bio 描述的内存单位从 page 过渡到 folio，会看到 `bio_for_each_folio_*` 迭代器，语义仍是"若干段内存"。

> **扎根点**：`bio_alloc_bioset` 从 `bio_set` 的 **mempool** 分配，不是普通 `kmalloc`。为什么？因为 IO 完成路径（writeback、内存回收下刷脏页）本身可能在**内存紧张时**运行——如果分配 bio 又要等内存回收、而内存回收又要靠这个 bio 写出脏页，就死锁了。mempool 预留一小撮对象保证"刷脏页方向"的 IO 永远能拿到 bio。这是"为什么不用 kmalloc"的经典内核内存序/前向进展（forward progress）论证。

### 16.3.3 bio 合并与 request 合并

提交 bio 到队列后，内核尝试**两级合并**：

**bio merge（plug merge / scheduler merge）**：新 bio 的 LBA 范围恰好接在某个 pending request 的头部（front merge）或尾部（back merge），直接 append 进那个 request，不生成新 request。

**plug 批量**：进程在 plug 窗口内积攒多个 request，unplug 时一次性入队（见 16.3.5）。

**合并条件**（主要）：

1. **LBA 相邻**（front：新 bio end == 旧 req start；back：新 bio start == 旧 req end）。
2. **操作类型一致**（不能把 READ merge 进 WRITE request；FLUSH/FUA 等带屏障语义的一般不合并）。
3. **不超 `max_sectors_kb`**（单 request 字节上限，设备/队列限制）。
4. **不超 `max_segments`**（硬件 SG 表段数上限——回到 16.2.1：bio_vec 段直接映射 DMA SG 表，硬件能装几段是物理约束）。

🎯 **面试加分点**：merge lookup **不是 O(N) 遍历**。blk-mq 在 software queue 上维护一个按 LBA 哈希的结构（merge hash），新 bio 来了直接哈希查"有没有 request 的端点正好等于我的端点"。这是 O(1) 期望。把"合并靠遍历队列"说出来直接掉分。

> **失败模式 / 真坑**：**plug 里只攒了一个 request 时 merge 收益为零，但开销仍在**（哈希查找、plug list 维护）。这就是为什么纯随机小 IO（LBA 永不相邻）下，merge 几乎不触发，调优重点要转向"减少 IO 数量 / 增大 IO 尺寸 / 用 io_uring 批量"，而不是指望 merge。排障口诀：**合并率低不一定是 bug，先问"这负载本来就能合并吗"**。

### 16.3.4 IO 调度器演进对照（带源码佐证）

| 调度器       | 引入版本   | 退出单队列 | 适用场景              | 核心算法               |
|--------------|-----------|-----------|----------------------|----------------------|
| noop         | 早期       | 5.0 移除  | （被 none 替代）      | 简单 FIFO            |
| deadline(legacy)| 2.5.x  | 5.0 移除  | 机械盘 real-time      | FIFO + sorted，超时提升 |
| CFQ          | 2.6.x     | **5.0 移除**| 机械盘混合负载       | 进程公平，时间片 + anticipation |

#### mq-deadline 源码精读：读为什么不被写饿死

mq-deadline 是 Jens Axboe 2016 年把 legacy deadline "适配到 blk-mq 框架"的产物——文件头自己这么写：

```c

/*
 *  MQ Deadline i/o scheduler - adaptation of the legacy deadline scheduler,
 *  for the blk-mq scheduling framework
 *
 *  Copyright (C) 2016 Jens Axboe <axboe@kernel.dk>
 */
```
【真实源码 v6.12 `block/mq-deadline.c` 第 1-6 行】

默认 tunable（这些数字是面试高频，现在有逐字出处）：

```c
static const int read_expire = HZ / 2;  /* max time before a read is submitted. */
static const int write_expire = 5 * HZ; /* ditto for writes, these limits are SOFT! */
/*
 * Time after which to dispatch lower priority requests even if higher
 * priority requests are pending.
 */
static const int prio_aging_expire = 10 * HZ;
static const int writes_starved = 2;    /* max times reads can starve a write */
static const int fifo_batch = 16;       /* # of sequential requests treated as one
				     by the above parameters. For throughput. */
```
【真实源码 v6.12 `block/mq-deadline.c`】

逐行读出语义：

- `read_expire = HZ/2` → **读的软 deadline 500ms**；`write_expire = 5*HZ` → **写 5000ms**。注释明说"these limits are SOFT"——它们不是硬实时保证，是"超过这个时间就提升该方向优先级"的触发阈值。**读 deadline 比写短 10 倍，这就是"读优先、读不被写饿死"的来源**。
- `writes_starved = 2` → 连续放行 2 批读之后，**强制放一批写**，防止写被读饿死（反方向公平）。
- `fifo_batch = 16` → 一旦决定按某方向 dispatch，连发 16 个**扇区相邻**的请求当作一批，摊薄"切换方向 + 重新排序"的开销，换吞吐。
- `prio_aging_expire = 10*HZ` → ioprio 优先级老化：低优先级请求等够 10s 也会被放出来，防优先级反转饿死。

调度状态记在 `struct deadline_data`（逐字摘）：

```c
struct deadline_data {
	struct dd_per_prio per_prio[DD_PRIO_COUNT];
	/* Data direction of latest dispatched request. */
	enum dd_data_dir last_dir;
	unsigned int batching;		/* number of sequential requests made */
	unsigned int starved;		/* times reads have starved writes */
	int fifo_expire[DD_DIR_COUNT];
	int fifo_batch;
	int writes_starved;
	int front_merges;
	u32 async_depth;
	int prio_aging_expire;
	spinlock_t lock;
};
```
【真实源码 v6.12 `block/mq-deadline.c`】

注意它**仍有一把 `spinlock_t lock`**——mq-deadline 的调度决策是串行化的。这正是"为什么 NVMe 上 none 更快"的一个直接证据：上调度器就意味着重新引入一个串行点。dispatch 主函数：

```c
static struct request *dd_dispatch_request(struct blk_mq_hw_ctx *hctx)
{
	struct deadline_data *dd = hctx->queue->elevator->elevator_data;
	const unsigned long now = jiffies;
	struct request *rq;
	enum dd_prio prio;

	spin_lock(&dd->lock);
	rq = dd_dispatch_prio_aged_requests(dd, now);
	if (rq)
		goto unlock;

	/*
	 * Next, dispatch requests in priority order. Ignore lower priority
	 * requests if any higher priority requests are pending.
	 */
	for (prio = 0; prio <= DD_PRIO_MAX; prio++) {
		rq = __dd_dispatch_request(dd, &dd->per_prio[prio], now);
		if (rq || dd_queued(dd, prio))
			break;
	}

unlock:
	spin_unlock(&dd->lock);
	return rq;
}
```
【真实源码 v6.12 `block/mq-deadline.c`】

读出 dispatch 顺序：**先处理优先级老化的（防饿死）→ 再按 ioprio 从高到低，高优先级有 pending 就忽略低优先级**。真正的"读 vs 写 + 饿死翻转"逻辑在 `__dd_dispatch_request` 里，关键判据（逐字摘）：

```c
if (deadline_fifo_request(dd, per_prio, DD_WRITE) &&
    (dd->starved++ >= dd->writes_starved))
	goto dispatch_writes;
```
【真实源码 v6.12 `block/mq-deadline.c` `__dd_dispatch_request` 内】

一行读懂："如果有写请求在等、且读已经连续把写饿死 `>= writes_starved`(=2) 次了，这次强制 dispatch 写。"——`writes_starved` 这个 sysfs 参数的全部含义就压在这一行里。

#### BFQ 的核心理念（4.12，Paolo Valente）

BFQ 在 **Linux 4.12** 合入（[内核文档](https://docs.kernel.org/block/bfq-iosched.html)）。核心：给每个进程分配一个 I/O **budget（字节数）**，budget 用完才切换，而不是 CFQ 那种固定时间片。好处：

- 大 IO 不会因时间片到期被过早打断（减少机械盘无效磁头跳跃）。
- **分配的是带宽，不是时间**——慢进程不会因为"占了时间片但没传多少数据"而吃亏。
- 原生支持 cgroup（这是 5.0 删 CFQ 后 `io.weight` 比例语义的承接者之一）。

Valente 自己 2019 年还在持续优化 BFQ（[LWN 784267, 2019-03-29](https://lwn.net/Articles/784267/)，"Improving the performance of the BFQ I/O scheduler"，进 5.2），文中给出可量化的收益：DBENCH 上约 **+150% 吞吐**、负载下应用启动**最高快 80%**——这些数字说明 BFQ 在它的目标场景（交互性 + 混合负载）确有硬价值，不是摆设。

💡 **坑点（必考）**：BFQ 在高 IOPS 的 NVMe 上有额外 CPU 开销——**每个 IO 都要 budget 记账**。NVMe 单盘几十万 IOPS 时，这点 per-IO 开销会变成可见的 CPU 占用甚至吞吐损失。生产中 NVMe 建议 `none`，**别照搬机械盘经验**。

### 16.3.5 plugging 机制详解（含真实源码佐证的不变量）

Plugging 是**进程级**的 IO 聚合，不是设备级。典型调用：

```c
struct blk_plug plug;
blk_start_plug(&plug);     // 启动 plug
// ... 多次 submit_bio()
blk_finish_plug(&plug);    // flush，批量提交
```
【示意代码，非逐字】（这是标准用法骨架，真实文件系统路径里 `blk_start_plug`/`blk_finish_plug` 通常包在 writeback/`__filemap_fdatawrite_range` 等更外层）

内部：`blk_start_plug` 把栈上 `plug` 挂到 `current->plug`（`task_struct` 里确有 `struct blk_plug *plug`）。`blk_mq_submit_bio` 时若发现有 active plug，就把 request 挂到 plug 的本地 list、**不立刻入调度器、不抢 hw queue**——这一点在下面 16.3.7 的真实 `blk_mq_submit_bio` 源码里能直接看到那两行 `if (plug) { blk_add_rq_to_plug(plug, rq); return; }`。

> ⚠️ **关键不变量（近年内核很考究的点）**：plug **不会无限攒**，否则睡眠中的 IO 会被永远压住。它在以下任一条件提前 flush：
> 1. plug list 中 request 数超过阈值（量级约 16，`BLK_MAX_REQUEST_COUNT` 一类常量）；
> 2. **进程被调度出去——`schedule()` 会主动 flush 当前 plug**；
> 3. 显式 `blk_finish_plug`。
>
> 第 2 条是设计铁律："进程一旦要睡就 flush plug"。面试被追问"plug 会不会把 IO 永远压住"——标准答案就是这条：**睡眠即 flush，保证前向进展**。底层根因是死锁避免：若进程 plug 着一批写然后去睡（比如等内存），而唤醒它的事件又依赖那批写落盘，不 flush 就死锁。

**工程推论**：

1. 文件系统 writeback 路径几乎都包 plug——这是批量 `fsync` 远快于逐条写的原因之一（不只是 page cache 的功劳）。
2. `O_SYNC`/`fdatasync` 触发 plug flush + flush/FUA 屏障，代价明显高于异步 buffered write。
3. `blktrace` 里看到大量 `P`（plug）/`U`（unplug）是文件系统在正常聚合 IO，不是问题。

### 16.3.6 blk-mq 队列映射（源码级数据结构）

核心数据结构，逐字读真实定义的关键部分：

**`blk_mq_tag_set`——描述硬件能力**：

```c
struct blk_mq_tag_set {
	const struct blk_mq_ops	*ops;
	struct blk_mq_queue_map	map[HCTX_MAX_TYPES];
	unsigned int		nr_maps;
	unsigned int		nr_hw_queues;
	unsigned int		queue_depth;
	unsigned int		reserved_tags;
	unsigned int		cmd_size;
	/* ... */
	struct blk_mq_tags	**tags;
	struct blk_mq_tags	*shared_tags;
	/* ... */
};
```
【真实源码 v6.12 `include/linux/blk-mq.h`，省略号处为本节不展开字段】

- `nr_hw_queues` —— **硬件 dispatch 队列数**，NVMe 驱动按 CPU 数填它。
- `queue_depth` —— **每队列 tag 数**（队列深度）。
- `map[HCTX_MAX_TYPES]` + `nr_maps` —— **按 IO 类型分多套映射**：`HCTX_TYPE_DEFAULT`/`READ`/`POLL`。这是为什么 polling IO 能走独立 hw queue 不被普通 IO 干扰。
- `cmd_size` —— 每 request 尾部给驱动预留的 per-command 私有区（如 NVMe 把 command 结构放这）。

**`blk_mq_hw_ctx`——硬件队列上下文**（逐字摘注释，最能体现设计）：

```c
/**
 * struct blk_mq_hw_ctx - State for a hardware queue facing the hardware
 * block device
 */
struct blk_mq_hw_ctx {
	struct {
		/** @lock: Protects the dispatch list. */
		spinlock_t		lock;
		/**
		 * @dispatch: Used for requests that are ready to be
		 * dispatched to the hardware but for some reason (e.g. lack of
		 * resources) could not be sent to the hardware. As soon as the
		 * driver can send new requests, requests at this list will
		 * be sent first for a fairer dispatch.
		 */
		struct list_head	dispatch;
		 /**
		  * @state: BLK_MQ_S_* flags. Defines the state of the hw
		  * queue (active, scheduled to restart, stopped).
		  */
		unsigned long		state;
	} ____cacheline_aligned_in_smp;
	/* ... */
	struct blk_mq_tags	*tags;
	struct blk_mq_tags	*sched_tags;
	/* ... */
};
```
【真实源码 v6.12 `include/linux/blk-mq.h`】

逐行扎根：

- `lock` / `dispatch` / `state` 三个被包进一个匿名 struct 并 `____cacheline_aligned_in_smp` —— **故意把最热的字段对齐到独立 cache line**，避免和别的字段假共享（false sharing）。这是 blk-mq "胜在 cache locality"的微观体现：连一个 hctx 内部都在和 cache line bounce 较劲。
- `dispatch` 注释点出一个常被忽视的语义：它装的是"**本来想发给硬件、但因资源不足（tag 满、驱动 busy）发不出去**"的请求；驱动一旦能收，**这条 list 优先发**，保证公平、不让这些"受阻请求"插队插到队尾被饿死。
- `tags` vs `sched_tags` —— 两套 tag。`tags` 是真正对应硬件 inflight 槽位的；`sched_tags` 是有调度器时调度层用的"虚拟更深"的 tag 池，让调度器有更多请求可排序而不被硬件深度卡死。**这是"为什么上调度器后 `nr_requests` 可以大于硬件队列深度"的原因。**

**`queue_rq` 回调——块层与驱动的边界**：

```c
struct blk_mq_ops {
	/**
	 * @queue_rq: Queue a new request from block IO.
	 */
	blk_status_t (*queue_rq)(struct blk_mq_hw_ctx *,
				 const struct blk_mq_queue_data *);
	/* ... */
};
```
【真实源码 v6.12 `include/linux/blk-mq.h`】

NVMe 驱动实现的就是这个 `queue_rq`（即 `nvme_queue_rq`），把 request 翻译成 NVMe command 写进 submission queue、敲 doorbell。返回 `BLK_STS_OK` / `BLK_STS_RESOURCE`（资源不足，请稍后重试，request 回到上面那条 `dispatch` list）等。

**Tag 机制**：每 hw_queue 一个 tag bitmap（`struct sbitmap_queue`，scalable bitmap，分段 + per-CPU hint，避免单一原子计数成热点）。发 request 就 alloc tag，完成按 tag 回调 free。Tag 满（深度耗尽）→ back-pressure 到上层（阻塞或返回资源忙）。NVMe 真实 per-queue depth 常见 128~1024（见 16.2.2 数字辨析），生产很少真跑满。

**sw → hw 映射策略**：

| 策略                | 使用场景                        |
|--------------------|-------------------------------|
| 1 hw queue         | 驱动只支持 1 条硬件队列（如 virtio-blk 旧版） |
| N hw = N NUMA node | NUMA 系统减少跨节点访问          |
| N hw = N CPU       | NVMe，充分利用硬件并发          |

### 16.3.7 提交路径主函数 `blk_mq_submit_bio`（逐行注解真实源码）

这是 v6.12 提交路径的真实主函数（逐字摘录，省略 zone write / crypto 等正交分支后逐行注解）。**这是本章把"提交路径"从口述变成源码的关键**：

```c
void blk_mq_submit_bio(struct bio *bio)
{
	struct request_queue *q = bdev_get_queue(bio->bi_bdev);
	struct blk_plug *plug = current->plug;          // ① 取当前进程的 plug
	const int is_sync = op_is_sync(bio->bi_opf);
	struct blk_mq_hw_ctx *hctx;
	unsigned int nr_segs;
	struct request *rq;
	blk_status_t ret;

	/*
	 * If the plug has a cached request for this queue, try to use it.
	 */
	rq = blk_mq_peek_cached_request(plug, q, bio->bi_opf);   // ② plug 里有缓存 request 先复用
	/* ... zone write plug 分支省略 ... */

	bio = blk_queue_bounce(bio, q);                  // ③ 需要时做 bounce buffer（highmem/限制）

	if (!rq) {
		if (unlikely(bio_queue_enter(bio)))      // ④ 拿 queue usage 引用（防 queue 被冻结/拔盘）
			return;
	}

	/*
	 * Device reconfiguration may change logical block size, so alignment
	 * check has to be done with queue usage counter held
	 */
	if (unlikely(bio_unaligned(bio, q))) {           // ⑤ 对齐检查（必须持引用做，防热重配）
		bio_io_error(bio);
		goto queue_exit;
	}

	bio = __bio_split_to_limits(bio, &q->limits, &nr_segs);  // ⑥ 按设备限制切分（O(1) 改 iter）
	if (!bio)
		goto queue_exit;

	if (!bio_integrity_prep(bio))                    // ⑦ DIF/DIX 数据完整性准备（如启用）
		goto queue_exit;

	if (blk_mq_attempt_bio_merge(q, bio, nr_segs))   // ⑧ 尝试合并到已有 request；成功则本 bio 消化掉
		goto queue_exit;
	/* ... zone plug 分支省略 ... */

new_request:
	if (!rq) {
		rq = blk_mq_get_new_requests(q, plug, bio, nr_segs);  // ⑨ 分配新 request + tag
		if (unlikely(!rq))
			goto queue_exit;
	} else {
		blk_mq_use_cached_rq(rq, plug, bio);         // ⑨' 复用缓存 request
	}

	trace_block_getrq(bio);                          // ⑩ tracepoint（blktrace 的 'G' 等靠它）

	rq_qos_track(q, rq, bio);                         // ⑪ rq-qos 钩子：iolatency/iocost/wbt 在这挂

	blk_mq_bio_to_request(rq, bio, nr_segs);          // ⑫ 把 bio 装进 request

	ret = blk_crypto_rq_get_keyslot(rq);              // ⑬ inline 加密 keyslot（如启用），省略错误分支

	if (op_is_flush(bio->bi_opf) && blk_insert_flush(rq))  // ⑭ FLUSH/FUA 走专门的 flush 状态机
		return;

	if (plug) {                                       // ⑮ ★关键：有 plug 就挂进 plug、直接返回
		blk_add_rq_to_plug(plug, rq);                 //    不入调度器、不抢 hw queue
		return;
	}

	hctx = rq->mq_hctx;
	if ((rq->rq_flags & RQF_USE_SCHED) ||             // ⑯ 有调度器 → 入调度器再 run
	    (hctx->dispatch_busy && (q->nr_hw_queues == 1 || !is_sync))) {
		blk_mq_insert_request(rq, 0);
		blk_mq_run_hw_queue(hctx, true);
	} else {
		blk_mq_run_dispatch_ops(q,                    // ⑰ 否则尝试"直发"，绕过排队最快路径
			blk_mq_try_issue_directly(hctx, rq));
	}
	return;

queue_exit:
	/*
	 * Don't drop the queue reference if we were trying to use a cached
	 * request and thus didn't acquire one.
	 */
	if (!rq)
		blk_queue_exit(q);
}
```
【真实源码 v6.12 `block/blk-mq.c`（约 2790 行起）；为讲清省略了 zone write plug、crypto 错误处理等正交分支，省略处已用注释标注，**保留分支均逐字**】

把这段读透，等于读懂了"一个 bio 进块层后到底发生了什么"，几个最值得品的点：

- **⑮ vs ⑯/⑰ 三条出口**：有 plug → 攒着（最常见，writeback 路径）；无 plug 但有调度器或硬件忙 → 入调度器排队；无 plug 无调度器且不忙 → **`blk_mq_try_issue_directly` 直发硬件**，这是 NVMe + none 的极速路径，连 software queue 排队都省了。**这就是"none 为什么快"的源码级答案**：它让大量请求走 ⑰ 这条最短路。
- **⑪ `rq_qos_track`**：cgroup 的 `io.latency`/`io.cost` 和自动的 writeback throttling（wbt）全挂在 rq-qos 框架这一个钩子上。**注意它在调度器选择（⑯）之前**——这从源码层证明了 16.5 的结论：**iolatency/iocost 是调度器无关的**，它们在请求"还没碰到调度器"时就介入了。
- **④ `bio_queue_enter` / queue_exit**：每个 bio 进出都要在 queue 的 usage counter 上加减引用。为什么？为了支持**热拔盘 / 队列冻结**（`blk_mq_freeze_queue`）——拔盘时要等所有 inflight bio 退出引用归零才安全拆队列。这是"为什么不能简单地 free 队列"的并发正确性根因。

## 16.4 IO 路径关键环节追踪

### 写路径全貌（以 ext4 buffered write 为例）

```
1. write() 系统调用
2. VFS → ext4_file_write_iter()
3. generic_perform_write() → 写入 page cache（buffered 只写脏页就返回，见第 10 章）
4.（异步）后续由 writeback（per-bdi flusher 线程，pdflush 继任者）或 fsync 触发真正落盘
5. ext4 writeback 路径构造 bio：bio_alloc() + bio_add_page()
   （入口函数名随版本/folio 化漂移：近年走 ext4_writepages → iomap/mpage、
    并已 folio 化为 *_write_folio，「待核」以当前源码为准——不要背死函数名）
6. blk_start_plug()                       [plug 窗口开始]
7. submit_bio(bio) → submit_bio_noacct()
   ├── device mapper / md 透明转发（可选，stacked device 递归 submit）
   └── blk_mq_submit_bio()                [≈5.9 起的名字；旧版经 make_request_fn]
8. 插入 plug list（if plug active）        [对应 16.3.7 步骤 ⑮]
9. blk_finish_plug() → flush → 合并尝试 → 入调度器队列
10. 调度器选 request → blk_mq_dispatch_rq_list()
11. 驱动 queue_rq()（如 nvme_queue_rq）→ 写 NVMe submission queue + doorbell
12. NVMe 硬件处理 → completion interrupt 或 poll
13. blk_mq_complete_request() → bio_endio() → page writeback 完成
```

### 同步 vs 异步 IO 对比

| 维度             | Synchronous (sync read)     | Buffered async write       | io_uring（见第 17 章）       |
|------------------|----------------------------|-----------------------------|------------------------------|
| 进程是否阻塞      | 是（等 bio 完成）           | 否（写完 page cache 就返回） | 可选（sq polling/async）      |
| page cache 参与   | 读时先查 cache              | 写先进 cache                | 可绕过（O_DIRECT）            |
| completion 通知   | schedule() 等待唤醒        | writeback callback          | CQ ring                      |
| 适用场景          | 小文件随机读               | 日志、流式写                 | 高 IOPS 低延迟              |

### Direct IO（O_DIRECT）路径

O_DIRECT 绕过 page cache，直接对齐磁盘扇区 read/write。近年 ext4 的 O_DIRECT 已改走 **iomap** 框架（`iomap_dio_rw`，具体函数名以当前源码为准）：

```
write(fd, buf, len) with O_DIRECT
→ ext4 → iomap_dio_rw()（旧资料里的 blockdev_direct_IO 已被 iomap 取代，「待核」当前符号）
→ 锁定用户页（get_user_pages） + 构造 bio 直接指向用户页
→ submit_bio() → 调度器/直发 → 驱动
→ 等待 bio 完成（sync）或经 kiocb 回调（aio/io_uring）
```

**工程推论 / 真坑**：

1. O_DIRECT 写必须按**设备逻辑/物理扇区对齐**（512B 或 4K），**缓冲区地址、文件偏移、长度三者都要对齐**，否则 `EINVAL`——新手最常踩。
2. 数据库（InnoDB、PostgreSQL）自管 buffer pool，需 O_DIRECT 避免双缓冲（page cache + buffer pool）浪费内存、且 crash 语义清晰。
3. O_DIRECT 不一定更快：机械盘随机小 IO 下 buffered + 预读可能更好；NVMe 大顺序 IO 下 O_DIRECT 避免 page cache 压力（避免大量脏页 + writeback 抖动）更优。

## 16.5 cgroup IO 控制：三套正交机制（带 LWN 出处）

这一节是另一个"浮于表面 vs 扎根"的重灾区——很多资料把"IO QoS"简化成"上 BFQ"。真实内核里有**三套正交机制**，且其中两套**与调度器无关**（挂在 16.3.7 ⑪ 的 rq-qos 框架上）。

### v1 vs v2 架构差异

| 对比项          | cgroup v1 blkio                    | cgroup v2 io                           |
|----------------|------------------------------------|----------------------------------------|
| 权重调度        | blkio.weight（**死绑 CFQ，5.0 删 CFQ 后失效**） | io.weight（BFQ 或 iocost 支持，work-conserving）|
| 绝对限流        | blkio.throttle.read_bps_device 等  | io.max（bps + iops 上限）              |
| 延迟控制        | 无                                  | io.latency（目标延迟，work-conserving） |
| 成本比例        | 无                                  | io.cost（blk-iocost 成本模型）          |
| 统计           | blkio.throttle.io_service_bytes 等 | io.stat                                |

### 三套机制逐一说清（含史料）

**① io.max / blk-throttle —— 硬限流，最古老、最直白**
提交路径上插 throttle check（入口 `blk_throtl_bio` 一类），超限的 bio 被 hold 到下个时间窗再放，**不是 drop**，对应用透明（只是变慢）。调度器无关。

**② io.latency / blk-iolatency —— 延迟保护，4.19，调度器无关**
[LWN 758963（Jonathan Corbet，2018-07-05，报道 Josef Bacik）](https://lwn.net/Articles/758963/) 把它讲透了。配置格式：`major:minor target=<目标延迟ms>`。机制（原文转述 + 引用）：

- 用一个**约 100ms 的短窗口**跟踪每个 cgroup 实际看到的延迟。
- 某 group 开始 miss 目标时，"all other peer groups with larger targets are throttled to free up some bandwidth; the group with the tightest latency target is thus given the highest priority"——**延迟目标最紧的优先**。
- "If all groups are meeting their targets, no throttling is done"——**全达标就不限**，work-conserving，不浪费带宽。

**为什么它是"调度器无关"的关键史料**：同一篇 LWN 明说它要替代的**旧方案**（cgroup v1 weight 路径）的毛病是——

> "depends heavily on the CFQ I/O scheduler and loses functionality in its absence. It doesn't work at all with multiqueue block devices."
> ——[LWN 758963]

blk-iolatency 正是为"在 multiqueue / 无 CFQ 世界里也能做 IO QoS"而生，它活在 rq-qos 框架（16.3.7 ⑪），**与 none/mq-deadline 共存**。这就是"NVMe + none 也能做 IO QoS"的史料级证据。

**③ io.cost / blk-iocost —— 成本比例分配，5.4，调度器无关**
[LWN 793460（Tejun Heo，2019-07）](https://lwn.net/Articles/793460/)。它要解决一个 IO 特有的难题：**IO 不像 CPU（wallclock）或内存（字节）那样有"trivially observable"的成本度量**——同一块盘，顺序 vs 随机、大 vs 小，吞吐能差好几个数量级。iocost 的做法：

- **给每个 IO 估一个 cost**：先按顺序/随机定 base cost，再叠加 size-proportional cost。
- 按 cgroup 的**层级权重**发放 IO 额度（work-conserving 比例分配）。
- **planning / execution 两条路径分离**：planning 每隔几毫秒跑一次复杂计算定预算，execution 每个请求跑、轻量执行——**用"慢路径算策略、快路径执行"避免在 SSD 高 IOPS 下让控制逻辑成为瓶颈**。这是个很 staff 的架构取舍。
- iocost "is fully deployed at Meta, on almost all our machines"——这是它经过超大规模生产验证的背书。
- 同样挂 rq-qos，**不挑调度器**。

> ⚠️ **要纠正的过度简化**：很多资料说"cgroup IO 控制必须靠 BFQ"。**只有"`io.weight` 经 BFQ 的比例分配"这一条路径成立**。`io.max`（限流）、`io.latency`（iolatency）、`io.cost`（iocost）**都不依赖 BFQ**，在 `none` 上照常工作。

💡 **生产真坑**：cgroup **v1** 的 `blkio.weight` 死绑 5.0 已删的 CFQ，5.0 之后事实失效（v1 的 `blkio.throttle.*` 限流仍可用，失效的只是 weight）。很多线上 K8s 节点还在 v1，"以为限了权重实际没限"——排障必查。解法：迁 v2，按需求选 `io.latency`/`io.cost`/`io.max`/(BFQ+`io.weight`)。

### throttle 限流工作原理

```
cgroup A: io.max 限 100MB/s
进程 A submit_bio → 提交路径 throttle 检查（入口 ≈ blk_throtl_bio）
  ├── 检查当前 window 内已消耗 bytes/iops
  ├── 未超限 → 直接 submit
  └── 超限 → bio 入 throttle 队列 → 等下一个时间窗 → timer 到期 → 重新 submit
```
限流是把超限 bio **hold 住等下个窗口**，不是 drop，所以表现为"延迟抖动"而非报错——排障时这是判据。

### 典型排障场景

**场景 1：iostat %util=100% 但吞吐不高**
根因：%util = "有 IO 在飞的时间百分比"，对深队列/多队列 NVMe 几乎无意义——任意时刻队列里有 1 个小 IO 就能让 %util≈100%。
正确信号：看 `await` 是否异常高于该盘基线；看 `aqu-sz`（旧版 `avgqu-sz`）是否持续逼近 `nr_requests`。**底层根因是 Little's Law**：`aqu-sz ≈ 到达率 × 平均逗留时间`，队列深度持续走高才是真饱和，%util 只回答"队列空不空"。

**场景 2：机械盘写延迟抖动大**
1. 看合并率：`iostat -x` 的 `wrqm/s`、`%wrqm`；或 blktrace 里 Q→M 比例。合并率低 = 小 IO 风暴没聚合（但先确认这负载本来能不能合并，见 16.3.3 真坑）。
2. 看 plug：blktrace 有没有 P/U。
3. `cat /sys/block/sdX/queue/rotational`（=1 机械盘）且调度器**不应是 none**（机械盘要靠调度器排序减寻道）。
4. 看是否写风暴把读饿死（mq-deadline 的 `writes_starved`/`read_expire` 就是干这个的，见 16.3.4 源码；极端写仍可能抬高读尾延迟）。
5. 看 cgroup 限流是否在 hold bio（表现为延迟抖动而非报错）。

**场景 3：容器内 IO 被限速但 `blkio.weight` 设了值**
大概率是 cgroup v1 + CFQ 已删的坑：
```bash
cat /proc/cgroups | grep blkio        # 确认 v1 还是 v2
cat /sys/block/sda/queue/scheduler    # 确认调度器
```
调度器是 mq-deadline/none 时 v1 `blkio.weight` 无效。迁 v2：要延迟保护用 `io.latency`（不挑调度器）；要比例分配用 `io.cost`（不挑调度器）或 BFQ+`io.weight`（HDD/混合负载更合适，NVMe 慎用 BFQ）。

### 大厂生产做法

| 场景              | 推荐配置                          | 原因                               |
|-------------------|---------------------------------|-----------------------------------|
| NVMe OLTP 数据库  | scheduler=none，O_DIRECT，io.latency 保护 | 数据库自管 buffer，要最低延迟；none 走直发路径(16.3.7 ⑰) |
| 机械盘日志归档    | scheduler=mq-deadline，顺序写    | deadline 防超时，顺序写避免随机寻道 |
| 混部宿主机（HDD/混合盘）| scheduler=BFQ，cgroup v2 io.weight | BFQ budget 公平，机械盘上按比例隔离好 |
| 混部宿主机（NVMe）| scheduler=none，cgroup v2 io.latency（+ io.cost 按需） | NVMe 上 BFQ 记账开销大；SLA 用 io.latency/iocost，调度器无关 |
| 容器/K8s 节点     | cgroup v2，io.max 限 iops+bps（关键服务叠 io.latency） | v1 weight 在 CFQ 删后失效（5.0）|

## 16.7 面试视角

### Staff 级高频题

**Q1：blk-mq 相比单队列解决了什么根本问题？为什么单队列在 NVMe 上是瓶颈？**
- 单队列：全局 `request_queue` + spinlock。多核并发时 **cache line bounce + lock contention**——原始出处 LWN 552904："a single lock which ... will bounce frequently between the processors. It is a linked list, a notably cache-unfriendly data structure."
- 量化直觉：HDD IOPS<200、锁开销 <1μs 无所谓；NVMe 500k IOPS、每 IO 仅 ~2μs 预算，锁/cache 一致性流量占比飙升。
- blk-mq 解法：per-CPU sw queue（提交路径基本无跨核锁）、per-hw queue 对应硬件 SQ、per-CPU 缓存的 tag（sbitmap）。
- ⚠️ 陷阱："blk-mq 是多线程调度"——错。**核心是减少锁竞争和 cache bouncing，是 cache locality 的胜利，不是并行度的胜利。**

**Q2：bio 和 request 的区别？什么时候合并？**
- bio：FS 构造的原始 IO 单位（哪些内存段 ↔ 哪段 LBA，什么操作）。
- request：blk-mq 分配、**绑定 tag** 的调度/dispatch 单元，可由多个 bio 合并而成。
- 合并：plug flush 时（多 bio→一 request）、调度器/提交路径 merge lookup 时（新 bio append 进已有 request 头/尾）。
- 条件：LBA 相邻 + 操作一致 + 不超 `max_sectors_kb`/`max_segments`。merge 走 **哈希 O(1)**，不是遍历。

**Q3：BFQ / mq-deadline / none 各自场景？**
- BFQ：budget-based 公平，**机械盘/慢盘**混合负载（前台交互 + 后台 writeback），减少无效磁头移动；可承接 `io.weight`。NVMe 慎用（per-IO 记账开销）。
- mq-deadline：deadline + sorted，**读写不对称 deadline（read 500ms / write 5000ms，源码 `HZ/2` vs `5*HZ`）** 保证读不饿死、读优先，适合对尾延迟有要求的 SSD/NVMe（如 DB）。
- none：无排序直发，NVMe 首选，CPU 开销最低（16.3.7 ⑰ 直发路径）。
- ⚠️ 陷阱 1：别说"SSD 用 noop"——noop 是 legacy 单队列的、5.0 已删，mq 时代是 none。别推荐 CFQ（5.0 删）。
- ⚠️ 陷阱 2：别说"做 cgroup IO 控制就得上 BFQ"——`io.latency`/`io.cost` 调度器无关。

**Q4：plugging 的目的？不 plug 会怎样？plug 会不会把 IO 永远压住？**
- 目的：进程短窗口攒 request 批量 flush，减少 per-request 开销（tag、调度器插入、队列锁）。
- 不 plug：每个小 bio 立刻入调度器，调度器频繁扰动、合并机会丢失、吞吐降、延迟抖。
- **不会永远压住**：`schedule()`（进程要睡）会主动 flush plug，list 超阈值也 flush——这是死锁避免的不变量。

**Q5：cgroup v1 blkio.weight 为什么 5.0 后失效？**
- `blkio.weight` 权重路径依赖 CFQ 时间片实现比例分配。
- CFQ + legacy 单队列块层在 **Linux 5.0（commit f382fb0b "block: remove legacy IO schedulers"，Jens Axboe）** 一并删除。
- mq-deadline/none 不实现权重语义，于是 weight 被忽略。
- ⚠️ 失效的只是 **v1 weight**；v1 `blkio.throttle.*` 仍可用。
- ⚠️ 别答死"只有 BFQ 能做 IO 控制"：v2 的 `io.latency`(blk-iolatency, 4.19) 和 `io.cost`(blk-iocost, 5.4) **调度器无关**，none 上照常工作。

**Q6：iostat %util=100% 说明磁盘饱和了吗？**
- %util = "有 IO 在飞的时间 / 窗口"，对深队列多队列 NVMe **几乎无意义**——任意时刻队列有 1 个 IO 就≈100%。
- 真饱和信号：`await` 持续高于该盘基线；`aqu-sz`（旧 `avgqu-sz`）持续逼近 `nr_requests`（Little's Law）。
- ⚠️ 时效（sysstat 12+，2018）：`svctm` 已移除（并发下无意义）；`avgqu-sz`→`aqu-sz`；`%util` 官方手册标注"对 NVMe 等不再表示饱和度"。

**Q7：O_DIRECT 对齐要求？违反会怎样？**
- 缓冲区地址、文件偏移、IO 长度都要对齐到设备逻辑块大小（512B 或 4K native）。
- 违反 → `EINVAL`。
- 查：`blockdev --getss`（逻辑扇区）/ `--getpbsz`（物理扇区）。
- ⚠️ 4K native（Advanced Format）设备必须 4K 对齐，否则 EINVAL 或 RMW 性能惩罚。

**Q8（新增，区分候选人深度）：为什么 CFQ 没被 port 成 mq 版，而 deadline 可以？**
- deadline 的逻辑是"基于 LBA 排序 + 软超时翻转方向"，**无状态依赖单一时间轴**，干净映射到每条 hw queue → mq-deadline。
- CFQ 依赖**单时间轴的时间片轮转 + anticipation idle（故意空转等同进程下一个 IO）**，在多 hardware queue 并行 dispatch 下既难协调、又与"喂满 NVMe"目标冲突。
- 社区选择：删 CFQ，用 4.12 已有的 BFQ（budget 而非时间片，原生 mq + cgroup）承接其需求。这是"宁可换实现也不强行翻译错位语义"的取舍。

## 16.8 未来演进

### io_uring 对块层的影响（见第 17 章）
io_uring 通过 submission ring 批量提交、减少 per-IO syscall 与态切换。块层视角：io_uring **最终仍走 blk-mq 提交 bio**，不是完全绕过；但 polling 模式（`IORING_SETUP_IOPOLL`）+ NVMe polling 可做到无中断 IO，延迟 <10μs 量级。它和 plug 是同一哲学（批量 > 单个）在更上层的体现。

### NVMe Passthrough 与字符设备直通
部分超低延迟场景用 NVMe passthrough（`/dev/ng*` character device，约 Linux 5.14+，「待核」精确版本）让用户态直接下发 NVMe Admin/IO command，**完全绕过块层调度器**。代价：失去 cgroup 限流与 page cache。这是块层"被绕过"的一种方向。

### Rust for Linux 与驱动层
Rust for Linux 已合入（Linux 6.1+），未来块驱动可用 Rust 编写。社区有 NVMe 驱动的 Rust 重写实验、blk-mq API 的 Rust 抽象层在做（「待核」具体进展）。

### 持久内存（PMEM）与 DAX
PMEM 通过 DAX（Direct Access）完全绕过块层，文件系统直接用 `load/store` 访问持久内存——没有 bio、没有调度器、没有 DMA。这是块层"被绕过"的极端案例，代表存储层的分叉演进。

### 三、设计题
**题目**：为多租户块存储设计 IO QoS。20 个租户（容器），优先级高/中/低，高优 p99<5ms，低优保证 throughput≥10MB/s。底层 NVMe，Linux 5.15+。

**参考要点**：
1. **选 cgroup v2**：v1 weight 随 CFQ 删失效；v2 才有 io.latency/io.cost/io.max。
2. **调度器选 none（或 mq-deadline），不选 BFQ**——本题最易错。NVMe 上 BFQ per-IO budget 记账 CPU 开销明显，而延迟 SLA **不靠 BFQ，靠 io.latency（blk-iolatency，调度器无关，rq-qos 框架，源码见 16.3.7 ⑪）**。能用机制满足 SLA 就别动调度器，这是更 staff 的取舍。
3. **高优延迟保护**：给高优 cgroup 设 `io.latency` target（如 5ms 余量），blk-iolatency 在其 miss 时 throttle 其他 cgroup（[LWN 758963] 的"tightest target 优先"机制）。
4. **低优下限/上限**：`io.max` 设 iops/bps 上限防其拖垮系统；"保证 ≥10MB/s"是 **reservation**，单纯 io.max（上限）不直接给下限，实践靠"给高优限上限 + work-conserving 让低优捡剩余"或 `io.cost` 权重保证最小份额——**点出"下限 vs 上限语义不同"是加分**。
5. **失效场景**：硬件已物理饱和（io.latency 无法超越硬件极限）、层级配置错、buffered write 延迟记在 writeback 而非业务 cgroup（**cgroup writeback 需正确开启，否则归属错**）。
6. **监控**：`cat /sys/fs/cgroup/<cg>/io.stat`；biolatency 配合 cgroup 维度。

### 四、系统题（含量级估算）
**题目**：线上 MySQL 突发 IO 延迟 1ms→50ms，只在每天 02:00-03:00 出现。给完整排查思路。

```
1. 时间窗定位：systemctl list-timers; crontab -l; ls /etc/cron.*  （找 02:00 的 backup/logrotate/checkpoint）
2. 进程级：iotop -ao 在 02:00 看谁在写。InnoDB innodb_io_capacity 默认 200，checkpoint 可 burst 到数千 IOPS
3. 块层队列：iostat -x 1 看 aqu-sz/await。await 远超该盘基线 + aqu-sz 逼近 nr_requests = 队列被占满
   （阈值对比"这块盘平时正常值"，不存在跨设备通用硬阈值）
4. cgroup 限流：cat /sys/fs/cgroup/.../mysql.service/io.stat，看 02:00 是否被 throttle（延迟抖动表征）
5. 调度器：NVMe+mq-deadline 时大量写可能 starve 读；read_expire/write_expire/writes_starved 在
   /sys/block/<dev>/queue/iosched/（源码默认 HZ/2、5*HZ、2，见 16.3.4）；调小 read_expire 压读尾延迟但更频繁打断写

量级参考：
- NVMe 顺序读 2-5 GB/s（100GB 约 20-50s burst）
- HDD 顺序读 100-200 MB/s（100GB 约 500-1000s 持续压力）
- MySQL p99 IO 延迟正常：NVMe <2ms，HDD <20ms
```

### 五、代码题
**题目**：bpftrace 统计每进程 block IO 延迟分布，按进程名聚合直方图。

先把"测哪段延迟"说清（面试常追问，答错掉分）：

| 配对 tracepoint | 测的延迟段 | 含义 |
|---|---|---|
| `block_rq_insert` → `block_rq_complete` | 入调度器 → 完成 | 含调度器排队 + 设备服务（≈ I→C） |
| `block_rq_issue` → `block_rq_complete` | dispatch 给驱动 → 完成 | **纯设备服务时间**（D→C），bcc `biolatency` 默认就是这个 |

下面用 `issue→complete` 量**设备服务时间**（语义最干净）：

```bash
#!/usr/bin/env bpftrace
// bio_latency_by_comm.bt — 测设备服务时间(issue→complete)，按进程名聚合
// 运行：sudo bpftrace bio_latency_by_comm.bt
// ⚠️ 字段以实际内核为准：cat /sys/kernel/debug/tracing/events/block/block_rq_issue/format

tracepoint:block:block_rq_issue
{
    // ⚠️ 用 (dev,sector) 作 key 是"示意级"：高并发同扇区可能短暂重入碰撞。
    //    严格做法应 hook 内核函数拿 struct request* 当 key（参考 bcc biolatency）。
    @start[args->dev, args->sector] = nsecs;
    // comm 在 issue 点是触发 dispatch 的上下文进程；writeback 多显示 kworker，
    // O_DIRECT/同步路径才显示业务进程名——这正是区分"谁在真正打盘"的依据。
    @comm_map[args->dev, args->sector] = comm;
}

tracepoint:block:block_rq_complete
{
    // complete 跑在中断/softirq 上下文，此处 comm 不可信，
    // 故归属用 issue 点暂存的 @comm_map。
    if (@start[args->dev, args->sector]) {
        $latency_us = (nsecs - @start[args->dev, args->sector]) / 1000;
        @latency_hist[@comm_map[args->dev, args->sector]] = hist($latency_us);
        @io_count[@comm_map[args->dev, args->sector]] = count();
        delete(@start[args->dev, args->sector]);
        delete(@comm_map[args->dev, args->sector]);
    }
}

END
{
    printf("\n=== IO service-time distribution by process (us) ===\n");
    print(@latency_hist);
    printf("\n=== IO count by process ===\n");
    print(@io_count);
    clear(@start);
    clear(@comm_map);
}
```
【示意代码，非逐字】（这是教学脚本，不是内核源码；真实生产建议直接用 bcc `biolatency`/`biosnoop`）

**暗坑提示**：
1. `block_rq_insert` 与 `block_rq_issue` 是**两个不同 tracepoint**，不是别名：insert=进调度器（16.3.7 ⑯），issue=dispatch 给驱动。选哪个取决于测哪段延迟。
2. dev+sector 作 key 在高并发下有碰撞风险（示意级）；严格做法 kprobe 拿 `struct request *`，参考 bcc biolatency。
3. **归属语义坑**：buffered write 实际打盘在 writeback，归属 `kworker`；只有 O_DIRECT/同步直发才归属业务进程。排障"谁在打盘"务必先想清，否则冤枉 kworker。
4. 高 IOPS（>100k/s）长时间追踪可能撑满 BPF map / 丢事件；`BPFTRACE_MAP_KEYS_MAX` 等环境变量可调上限（变量名以 `man bpftrace`/当前版本为准）。

## 心法

> **块 IO 层的本质有两层**：
> 1. **"批量 > 单个"**——合并 bio、plug 积攒、batch dispatch、io_uring 批量提交，凡能聚合再打的，聚合后吞吐一定更高（延迟未必更低）。调优永远在问"你的 IO 为什么没被合并"。
> 2. **"locality > 并行"**——blk-mq 的全部胜利不是"开了更多线程/队列"，而是把"一把全局锁 + 一条共享链表"拆成 per-CPU 结构，**消灭 cache line bounce**。从 `____cacheline_aligned_in_smp` 到 per-CPU tag cache 到 sw/hw 两层队列，处处都是 cache locality 的工程。理解这两层，就理解了块层四十年演进的主线。
