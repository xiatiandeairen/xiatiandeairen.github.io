> **一句话定位**：VFS 是内核里一套"文件系统适配器层"——用四个核心对象（`super_block`/`inode`/`dentry`/`file`）加几张 ops 指针表，让 `open/read/write` 这组系统调用对 ext4、NFS、tmpfs、procfs、socket、epoll fd 说同一套话。但真正把它从"美学口号"变成"工程系统"的，是 **dcache + RCU-walk 的多核可扩展性**、**mount namespace + shared subtrees 的传播语义**、以及 **`i_rwsem` 并行 lookup** 这三场具体的、有日期有 commit 有社区辩论的战役。

---

## TL;DR — 本章最硬结论

1. **四大对象的字段布局本身就是设计**：`struct dentry` 把字段**按 cacheline 显式分组**——"RCU lookup touched fields" 挤在头 64 字节，`d_lockref`（锁+引用计数）被特意推到第二条 cacheline 之后，注释原话 *"keep separate from RCU lookup area if possible!"*。这不是随手排的，是为了让无锁查找永远不碰会被原子写脏的缓存行。【真实源码 v6.12 include/linux/dcache.h】
2. **RCU-walk 是一场可扩展性战役，不是优化技巧**：2010 年 Nick Piggin 的 store-free path walking（merged 2.6.38）解决的根因是 *"the constant incrementing and decrementing of dentry reference counts creates a great deal of cacheline bouncing"*（LWN 419811）。慢路径 `ref-walk` 每过一个 `dentry` 都 `atomic_inc`，多核下 `/`、`cwd` 这种公共 dentry 的引用计数缓存行在核间疯狂弹跳，缓存命中也救不了你。
3. **mount namespace ≠ 容器隔离（时间错觉）**：mount namespace 是 *Linux 第一个 namespace*，2002 年随 2.4.19 落地，名字 `CLONE_NEWNS` 因此成了历史包袱（Kerrisk, LWN 689856）。但它早期是"全有或全无"——光驱换碟在别的 namespace 里看不见，只能逐个 namespace 重挂。工业级可用要到 **2.6.15（2006）的 shared subtrees**（Ram Pai 主笔、Al Viro 迭代，LWN 159077）。
4. **"一切皆文件"的机械核心是 `file->f_op` 分发，绑在 `struct file` 上而非 inode 上**：`read(fd)` 走 `fdget → file->f_op->read_iter`，VFS 对"背后是 ext4 还是 socket 还是 epoll"一无所知。v6.12 的 `struct file_operations` 已经有 `read_iter`/`write_iter`/`uring_cmd`/`splice_read` 等 **30+ 个 hook**。【真实源码 v6.12 include/linux/fs.h】
5. **目录锁从 `i_mutex` 进化到 `i_rwsem` 是 2016 年的并行 lookup 战役**：Al Viro 的洞见是 *"lookup is non-destructive, so there is no real conceptual reason to stop it from happening in parallel"*（LWN 685108，landed 4.7）。代价是要发明 in-lookup dentry 机制防止两个并行 lookup 给同名建出两个 dentry。

**前置依赖章**：第 4 章（进程/`task_struct`/`files_struct`）、第 7 章（虚拟内存/mmap）、第 9 章（slab/shrinker）、第 10 章（Page Cache/`address_space`）。RCU 与 seqlock 的内存序基础建议先翻第 11 章（并发原语）。

## 一、背景：为什么需要 VFS（以及为什么"if-else 分发"这条更简单的路走不通）

### 1.1 朴素方案：`read()` 里写一坨 `if (fs_type == ...)`

```c
// 【示意代码，非逐字】—— 这是"反面教材"，内核从不这么写
ssize_t sys_read(int fd, ...) {
    struct file *f = current->files->fd[fd];
    switch (f->fs_type) {
        case EXT4: return ext4_read(f, ...);
        case NFS:  return nfs_read(f, ...);
        case FAT:  return fat_read(f, ...);
        // 每加一个文件系统就改这里 → 核心 syscall 与所有 fs 强耦合
    }
}
```

这条路有三个**根因级**问题，不止是"难维护"：

1. **闭合集合假设**：`switch` 要求编译期枚举所有文件系统。但 NFS、FUSE、可加载模块（`insmod`）的文件系统是**运行期**才注册的，编译期根本不知道有谁。`switch` 无法表达"运行期可扩展的分发"。
2. **跨模块符号依赖**：`sys_read` 引用 `ext4_read` 就意味着 vmlinux 核心要链接每一个文件系统的实现，模块化（把 ext4 编成 `.ko` 按需加载）直接破产。
3. **抽象泄漏到上层**：应用 `read(fd)` 时被迫感知 fd 背后是什么。Unix 的契约恰恰相反——**应用只认 fd，不认文件系统**。

### 1.2 解法：运行期可重绑的函数指针表（不是 C++ vtable，是更显式的版本）

VFS 的手段是把分发**数据化**：每个对象挂一张 ops 表（`f_op`/`i_op`/`s_op`/`d_op`），表里是函数指针，`open` 时由具体子系统把自己的实现**塞进对象**。`read()` 只做 `file->f_op->read_iter(...)`，一次间接跳转。

为什么不直接用 C++ 虚函数？因为内核要的语义比 vtable 更灵活：

- **per-object 而非 per-class**：C++ 的 vptr 绑在类型上，一个类型一张 vtable。VFS 允许**同一个 inode 被 open 两次得到两个 `struct file`，各自挂不同的 `f_op`**（实践罕见但机制允许，例如 `/proc` 下不同视图）。这是 per-instance 多态，vtable 给不了。
- **运行期重绑**：`d_op`、`i_fop` 可以在对象生命周期里被替换（如挂载点 dentry 被打上 `DCACHE_MANAGED_DENTRY` 改变穿越行为）。
- **零成本布局控制**：函数指针表是普通 `struct`，内核能精确控制它放在哪条 cacheline、是否 `__ro_after_init`、`____cacheline_aligned`——`struct inode_operations` 结尾就有 `____cacheline_aligned`【真实源码 v6.12 include/linux/fs.h】。vtable 的布局是编译器的，你管不着。

这是后端工程师熟悉的**策略模式**，但落在内核态、C 语言、可热插拔模块、且对缓存行布局锱铢必较的语境里。理解这点，下面整章都是它的展开。

## 二、设计考古：VFS 这套抽象是怎么被"逼"出来的

> 这一节是本章的"史料"硬货。不讲"VFS 很优雅"，讲**谁、何时、为了解决前一代什么缺陷、社区辩论过什么**，尽量带出处。

### 2.1 史前：Unix v7 的"单一文件系统"死锁（约 1979）

最早的 Unix 没有 VFS。整个内核只认一种文件系统（s5fs）：inode 存磁盘固定偏移，目录是一种特殊文件（存 `name → inode number`）。`namei()`（"name to inode"，今天 `fs/namei.c` 的名字就是它的化石）直接写死了 s5fs 的磁盘布局。这个设计极简，但**把"文件系统类型"这个维度焊死在了内核里**——想同时挂 MS-DOS 软盘和本地盘？做不到。

### 2.2 转折：Sun 的 vnode 架构，被 NFS 逼出来的（SunOS 2.0，1985；论文 USENIX 1986）

VFS 这套抽象**不是 BSD 发明的，是 Sun Microsystems 为了把 NFS 塞进 SunOS 而设计的**。奠基文献是 Steve Kleiman 的 *"Vnodes: An Architecture for Multiple File System Types in Sun UNIX"*（USENIX Summer Conference, Atlanta, June 1986, pp. 238–247）。

论文白纸黑字列了**四条设计目标**（这是面试"VFS 为什么这样设计"的标准答案出处）：

1. 文件系统实现必须**清晰切成"文件系统无关层"和"文件系统相关层"**，中间是良定义的接口；
2. 要同时支持 **4.2BSD FFS（本地盘）、非 Unix 文件系统（MS-DOS）、stateless 的 NFS、stateful 的 RFS**；
3. 要支持远程文件系统的**服务端**；
4. 文件系统操作必须**原子**。

**关键洞见——为什么是 NFS 逼出了 VFS**：本地 FS 可以假设"操作同步完成、inode 常驻内存"。但 NFS 操作有网络延迟、有分区故障、服务端 inode 不在本地、还要支持 stateless 重传。这迫使接口必须把 `vnode`（virtual node，Linux `inode` 的概念前身）的**生命周期管理（缓存、失效、引用计数）显式抽象出来**——因为你不能再假设"拿到 inode 指针就一直有效"。VFS 的抽象边界从第一天起就是被**异构文件系统的差异**撑开的，不是凭空的美学。目标 2 里 stateless（NFS）和 stateful（RFS）并列，正是这个张力的来源。

1987 年后由 Sun 和 AT&T 合作的 **System V Release 4（SVR4）** 把 vnode 收编为标准，成为后续几乎所有 Unix 衍生系统的范本。

> **面试防张冠李戴**：BSD 走的是**自己的一套 vnode 变体**，不是 Sun 设计的直接继承。"BSD 引入 VFS"是常见的说反——奠基者是 Sun/Kleiman。

### 2.3 Linux 入场：0.95 的 VFS（Rémy Card，1992）

Linux 0.1（1991）只支持 Minix FS。到 0.95（1992），Rémy Card（后来 ext/ext2 的作者）引入了 Linux 自己的 VFS 层，确立 `super_block`/`inode`/`file` 三大对象。这一版奠定了后续三十年的骨架——但**还没有 `dentry`**，路径查找靠 inode cache + 字符串比较，这是后面要还的债。

### 2.4 dcache 与 `dentry` 的引入（Linux 2.1，1997）：把"路径"做成一等公民

随负载增大，"每次路径查找都字符串比较 + 查 inode cache"成了瓶颈。Linux 2.1 引入 `dentry cache`（dcache），把路径**每个分量**缓存成 `dentry` 对象，用哈希表加速。`fs/dcache.c` 顶部至今写着：

```c
/*
 * (C) 1997 Thomas Schoebel-Theuer,
 * with heavy changes by Linus Torvalds
 */
```
【真实源码 v6.12 fs/dcache.c 第 1–8 行（节选）】

这条注释本身就是史料：dcache 是 1997 年 Thomas Schoebel-Theuer 写的、Linus 重度改过。同一段注释还点出了 dcache 与 icache 的核心契约（下文 §5.3 详解）：

```c
/*
 * Notes on the allocation strategy:
 *
 * The dcache is a master of the icache - whenever a dcache entry
 * exists, the inode will always exist. "iput()" is done either when
 * the dcache entry is deleted or garbage collected.
 */
```
【真实源码 v6.12 fs/dcache.c 顶部注释】

"dcache is a master of the icache"——dentry 是 inode 的"主人"，dentry 在 inode 就一定在。这一句决定了后面回收顺序"先 dentry 后 inode"的全部逻辑。

### 2.5 RCU-walk：2010 年的多核可扩展性战役（Nick Piggin；merged 2.6.38）

这是 Linux VFS **最重要的一次性能演进**，也是史料最丰富的一战。出处：Jonathan Corbet, *"Dcache scalability and RCU-walk"*, LWN.net, 2010-12-14（Article 419811）。

**前一代缺陷（ref-walk 的根因）**：传统路径查找每经过一个 `dentry` 都要 `dget()`（`atomic_inc` 引用计数）、用完 `dput()`。LWN 原话：

> *"the constant incrementing and decrementing of dentry reference counts creates a great deal of cacheline bouncing."*

底层根因是**缓存一致性协议（MESI）下的 false sharing 式弹跳**：`/`、当前进程 `cwd` 这些**人人都要经过**的公共 dentry，它们的引用计数字段所在的缓存行，在每个做路径查找的 CPU 上被反复 `atomic_inc`/`dec`。每次原子写都要求该缓存行进入 Exclusive/Modified 态、invalidate 其他核的副本。于是**即便 dentry 100% 命中 dcache、一次磁盘都不碰**，多核高并发下光是"维护引用计数"就把核间总线打满。这是排队论意义上的伪共享串行化——并行度被一个本可无锁的计数器锁死。

**解法（store-free path walking）**：RCU-walk 全程**只持 RCU 读锁，不写任何共享缓存行**——不 `atomic_inc` 引用计数、不取 `d_lock`。正确性靠每个 dentry 的 **seqlock（`d_seq`）**：读时记下序号，走完校验序号没变，就证明这一步没被并发 `d_move`/rename 改过。因为查找路径上**零写入**，公共 dentry 的缓存行可以在所有核上以 Shared 态共存、永不 invalidate，核间竞争趋近于零。这才是它"快"的根因——不是少了几条指令，是**消灭了缓存行弹跳**。

**社区辩论的真实温度**：LWN 同篇指出这套补丁 *"has had nowhere near enough review"*、复杂度高、VFS reviewer 稀缺，是"set to be merged for 2.6.38"时还带着风险硬上的。这说明 RCU-walk 不是干净利落的设计，是一次**用复杂度换可扩展性、且社区自己都心里没底**的豪赌。理解这点，你才明白为什么它的回退路径（`try_to_unlazy`）写得那么绕——那是为正确性付的复杂度税。

### 2.6 mount namespace 与 shared subtrees：容器隔离的两步走

**第一步（2002，Linux 2.4.19，Al Viro）**：引入 Linux **史上第一个 namespace**——mount namespace。flag 叫 `CLONE_NEWNS`（"New NS"），因为当时它是唯一一个 namespace，没人想到以后还有 PID/net/user ns，名字成了**至今没改的历史包袱**（Kerrisk, *"Mount namespaces and shared subtrees"*, LWN 689856）。

**但 2002 年的版本"全有或全无"**，不实用。Kerrisk 给的经典反例（原文）：

> *"suppose, for example, that a new disk is loaded into an optical disk drive. In the original implementation, the only way to make that disk visible in all mount namespaces was to mount the disk separately in each namespace."*

光驱换张碟，要在每个 namespace 里各挂一遍——运维灾难。

**第二步（2006，Linux 2.6.15，Ram Pai 主笔 / Al Viro 迭代）**：shared subtrees（挂载传播），LWN 159077（Corbet, 2005-11-08）原文 *"written primarily by Ram Pai"*、*"Al Viro has worked with a few iterations"*。它引入四种传播类型：

| 传播类型 | flag | 一句话语义 |
|---|---|---|
| shared | `MS_SHARED` | 挂载/卸载事件**双向**在 peer group 内传播 |
| slave | `MS_SLAVE` | 从 master 单向**接收**事件，自己的事件**不**回传 |
| private | `MS_PRIVATE` | 谁也不传谁（2.6.14 之前的老行为） |
| unbindable | `MS_UNBINDABLE` | private + 禁止被 bind mount（防递归 bind 爆炸） |

今天 systemd 默认把 `/` 设成 `shared`、容器运行时（runc/containerd）再按需把容器内挂载点改成 `slave`/`private`，全建立在这套机制上。

> **面试时间错觉纠正**：说"mount namespace = 2002 就有容器隔离"是错的。2002 只是有了隔离原语，**工业级可用要到 2006 的 shared subtrees**（隔了约三年，Kerrisk 原文 "around three years after"）。

**第三步补全（2013，Linux 3.8，user namespace）**：user ns + mount ns 让**非 root** 也能建隔离挂载树，rootless Docker/podman 由此成立。

### 2.7 并行 lookup：2016 年把目录锁从互斥升级成读写锁（Al Viro；landed 4.7）

出处：Jake Edge, *"VFS parallel lookups"*, LWN 685108（报道 Al Viro 在 LSFMM 2016 的工作）。

**前一代缺陷**：所有目录操作都持 `i_mutex`（目录 inode 的互斥锁），**包括最常见的 `lookup`**。于是同一目录下两个进程同时 `open` 不同文件，也要排队。Viro 的洞见（原文）：

> *"the most common operation, lookup, is non-destructive, so there is no real conceptual reason to stop it from happening in parallel."*

lookup 是只读的，凭什么互斥？

**解法**：把 `i_mutex` 换成 `i_rwsem`（读写信号量，这就是为什么 v6.12 的 `struct inode` 里是 `struct rw_semaphore i_rwsem`【真实源码】，老资料里的 `i_mutex` 已不存在）。lookup 以**共享（读）模式**持锁，create/unlink/rename 等破坏性操作仍持**独占（写）模式**。

**新引入的复杂度——in-lookup dentry**：放开并行 lookup 后冒出新竞态：两个并行 lookup 查同一个**还不在 dcache 里**的名字，可能各自建一个 dentry，同名两个 dentry 是灾难。Viro 的解法（LWN 原文转述）是造一个"标记 lookup 进行中"的对象、*"tracked in a hash on the parent"*；后到的 lookup 发现同名 in-progress，*"simply waits until the earlier lookup is done"*。这就是 `struct dentry` 里那个 union 字段 `d_in_lookup_hash` 和 `d_wait` 的来历——它们和 `d_lru`/`d_alias` 共用内存，专给 in-lookup 状态用【真实源码 v6.12 include/linux/dcache.h，见 §3.2】。

> **这是一个"放开并发必然引入新竞态、再用新机制兜底"的范式样本**。从 `i_mutex` 到 `i_rwsem` 一行改动，背后是 in-lookup dentry 一整套机制。面试被问"i_rwsem 比 i_mutex 强在哪"，答得出 in-lookup 才是 staff 级。

## 三、现状：四大对象的**真实**字段布局（源码精读）

上一版章节给的 struct 是"概念版"（字段名对、但简化了）。这一版给**逐字源码**，因为**字段的排列顺序、union、cacheline 注释本身就是设计**，简化掉就把最硬的东西丢了。

### 3.1 `struct file`：打开状态 + `f_op` 分发表的载体

```c
/**
 * struct file - Represents a file
 * @f_count: reference count
 * ...
 * @f_freeptr: Pointer used by SLAB_TYPESAFE_BY_RCU file cache (don't touch.)
 */
struct file {
	atomic_long_t			f_count;
	spinlock_t			f_lock;
	fmode_t				f_mode;
	const struct file_operations	*f_op;          // ← "一切皆文件"的分发表，就这一个指针
	struct address_space		*f_mapping;     // ← 指向 Page Cache（inode->i_mapping 的缓存）
	void				*private_data;
	struct inode			*f_inode;       // ← cached inode（= f_path.dentry->d_inode）
	unsigned int			f_flags;
	unsigned int			f_iocb_flags;
	const struct cred		*f_cred;
	/* --- cacheline 1 boundary (64 bytes) --- */
	struct path			f_path;         // ← { vfsmount, dentry }，路径查找的产物
	union {
		/* regular files (with FMODE_ATOMIC_POS) and directories */
		struct mutex		f_pos_lock;
		/* pipes */
		u64			f_pipe;
	};
	loff_t				f_pos;          // ← 当前读写偏移（per-open，不在 inode 上）
#ifdef CONFIG_SECURITY
	void				*f_security;
#endif

	struct fown_struct		*f_owner;
	errseq_t			f_wb_err;
	errseq_t			f_sb_err;
#ifdef CONFIG_EPOLL
	struct hlist_head		*f_ep;          // ← 这个 file 被哪些 epoll 监听
#endif
	union {
		struct callback_head	f_task_work;
		struct llist_node	f_llist;
		struct file_ra_state	f_ra;           // ← readahead 状态
		freeptr_t		f_freeptr;      // ← SLAB_TYPESAFE_BY_RCU 释放链
	};
	/* --- cacheline 3 boundary (192 bytes) --- */
} __randomize_layout
  __attribute__((aligned(4)));
```

【真实源码 v6.12 include/linux/fs.h，约 1540–1580 行】

**逐行/逐设计注解**：

- `f_count` 是 `atomic_long_t` 引用计数。`rm` 一个被打开的文件时，`unlink` 只把 dentry 摘掉，`f_count` 仍 > 0，inode 与 Page Cache 不释放——这是 `lsof | grep deleted` 那道经典题的源码根据。
- **`f_op` 是整个"一切皆文件"的支点**。ext4、socket、pipe、epoll 各填一张同形状的 `file_operations`，VFS 只认这一个指针。
- **`f_pos` 在 `struct file` 上，不在 inode 上**：同一 inode 被 `open` 两次得到两个 `struct file`，各有独立偏移。这是为什么两个进程读同一文件互不干扰偏移。
- `f_pos_lock` 与 `f_pipe` 共用一个 **union**：普通文件/目录需要 `f_pos_lock` 保护原子偏移（`FMODE_ATOMIC_POS`），管道根本没有"偏移"概念，那块内存复用成 `f_pipe`。**这是内核"同一对象服务异构语义、靠 union 省内存"的典型手法**——`struct file` 是热点对象、量大（每个 open 一个），每省 8 字节都乘以百万计。
- `f_freeptr` + 注释里的 **`SLAB_TYPESAFE_BY_RCU`**：`struct file` 的 slab 缓存用了 type-safe-by-RCU。含义是——一个 `file` 被释放后，它的内存可能立即被复用成**另一个 `file`**（类型不变），但不会在 RCU 宽限期内变成别的类型。这让 `fdget` 的无锁快路径成立：拿到指针后即使它刚被释放重分配，至少还是个合法的 `struct file`，再用 seq/count 校验真伪。**这是"为什么 fdget 能无锁"的底层根因**，不是凭空的。
- `__randomize_layout`：开了 `CONFIG_RANDSTRUCT` 时编译器会打乱字段顺序，作为利用缓解（attacker 不能假设字段偏移）。所以**生产内核里这些字段的实际偏移可能和源码顺序不同**——排障时别硬背偏移。

### 3.2 `struct dentry`：字段按 cacheline 分组是核心设计

```c
struct dentry {
	/* RCU lookup touched fields */
	unsigned int d_flags;		/* protected by d_lock */
	seqcount_spinlock_t d_seq;	/* per dentry seqlock */
	struct hlist_bl_node d_hash;	/* lookup hash list */
	struct dentry *d_parent;	/* parent directory */
	struct qstr d_name;
	struct inode *d_inode;		/* Where the name belongs to - NULL is
					 * negative */
	unsigned char d_iname[DNAME_INLINE_LEN];	/* small names */
	/* --- cacheline 1 boundary (64 bytes) was 32 bytes ago --- */

	/* Ref lookup also touches following */
	const struct dentry_operations *d_op;
	struct super_block *d_sb;	/* The root of the dentry tree */
	unsigned long d_time;		/* used by d_revalidate */
	void *d_fsdata;			/* fs-specific data */
	/* --- cacheline 2 boundary (128 bytes) --- */
	struct lockref d_lockref;	/* per-dentry lock and refcount
					 * keep separate from RCU lookup area if
					 * possible!
					 */
	union {
		struct list_head d_lru;		/* LRU list */
		wait_queue_head_t *d_wait;	/* in-lookup ones only */
	};
	struct hlist_node d_sib;	/* child of parent list */
	struct hlist_head d_children;	/* our children */
	/*
	 * d_alias and d_rcu can share memory
	 */
	union {
		struct hlist_node d_alias;	/* inode alias list */
		struct hlist_bl_node d_in_lookup_hash;	/* only for in-lookup ones */
	 	struct rcu_head d_rcu;
	} d_u;
};
```
【真实源码 v6.12 include/linux/dcache.h】

**这段是本章源码精读的高潮，逐点拆**：

1. **字段被显式分成两组，第一组就叫 `/* RCU lookup touched fields */`**：`d_flags`/`d_seq`/`d_hash`/`d_parent`/`d_name`/`d_inode`/`d_iname` 全挤在头 64 字节。为什么？因为 RCU-walk（§3.5）只读这几个字段，把它们塞进**同一/相邻 cacheline**，一次 lookup 步骤只需 fetch 一两条缓存行。
2. **`d_lockref`（锁+引用计数）被特意推到 cacheline 2 之后，注释原话 *"keep separate from RCU lookup area if possible!"`**：这是 §2.5 那场可扩展性战役的**物理落地**。`d_lockref` 是会被 `atomic` 写脏的字段（ref-walk 时 `dget`/`dput`）。把它和 RCU-walk 只读区**隔到不同缓存行**，意味着：ref-walk 在核 A 上猛写 `d_lockref` 弄脏它那条缓存行时，**完全不会 invalidate** 核 B 上 RCU-walk 正在读的"只读区"那条缓存行。**结构体布局直接编码了"读写分离、避免伪共享"的并发策略**——这是把排队论结论写进了字段顺序里。面试问"dentry 为什么这么排字段"，答这条注释满分。
3. `d_hash` 是 `hlist_bl_node`（bl = bit-locked，哈希桶头自带 1-bit 自旋锁），不是普通 `hlist_node`。**老资料和上一版章节写 `struct hlist_node d_hash` 是过时的**——dcache 哈希桶用 bit spinlock 省内存（锁状态藏在指针最低位）。
4. **引用计数叫 `d_lockref`，不是 `d_count`**：`lockref` 是"spinlock + count 打包进一个 8 字节、能用单条 cmpxchg 原子更新"的结构。这本身又是一次微优化——常见路径下连 spinlock 都不真正自旋，直接 cmpxchg 改 count。
5. `d_iname[DNAME_INLINE_LEN]`：短名字**内联存**在 dentry 里，不另分配。绝大多数文件名很短（< 40 字节），内联省掉一次 kmalloc + 一次指针解引用 + 一次缓存 miss。`d_name.name` 短名时就指向这个内联缓冲区。
6. **三处 union，每处都是"互斥状态复用内存"**：
   - `d_lru` ←→ `d_wait`：普通 dentry 在 LRU 上用 `d_lru`；**in-lookup 状态**（§2.7 并行 lookup）用 `d_wait` 挂等待队列。一个 dentry 不可能同时"在 LRU 上"和"正在被 lookup"，所以共用内存。
   - `d_u` 里 `d_alias` ←→ `d_in_lookup_hash` ←→ `d_rcu`：`d_alias` 把同 inode 的多个 dentry（硬链接！）串成 alias list；`d_in_lookup_hash` 只给 in-lookup 用；`d_rcu` 是释放时挂 RCU 回调用。三者生命周期互斥。
   - **`d_in_lookup_hash` 和 `d_wait` 这两个字段，就是 §2.7 那场 2016 并行 lookup 战役在数据结构里留下的疤**。没有那场战役，这两个 union 成员根本不存在。
7. `d_children`/`d_sib`：v6.12 用 `hlist`（`d_children` 头 + 各孩子 `d_sib` 兄弟节点）组织父子。**老资料里的 `d_subdirs` + `d_child` 已被重构成 `d_children` + `d_sib`**——又一个"老书过时"的点。

### 3.3 `struct inode`：元数据本体，注意"哪些字段 path walking 不碰"

【真实源码 v6.12 include/linux/fs.h，约 1002–1150 行（节选并标注）】

**逐点注解**：

- **`/* Stat data, not accessed from path walking */`** 这行注释把 inode 切成两半：path walking（lookup 热路径）只碰前半的 `i_op`/`i_sb`/`i_mapping`/`i_flags`，`i_ino`/`i_size`/时间戳这些只有 `stat(2)` 才读。**和 dentry 一样按访问热度分组布局**——热路径字段靠前、聚在少数 cacheline。
- **`i_nlink` 是 const/非 const 双视图 union**：对外暴露 `const unsigned int i_nlink`（防止文件系统代码手贱直接改链接数），内核内部走 `set_nlink()`/`inc_nlink()`/`drop_nlink()` 改 `__i_nlink`。**用 union + const 强制"链接数只能通过受控 API 修改"**，是 C 里模拟"私有字段"的手法。
- **`i_link` 揭穿 symlink 的本质**：符号链接的 inode 里，`i_link` 直接指向目标路径字符串。这就是 §3.5 "symlink 处理"的源码依据——`get_link` 多数情况直接返回 `i_link`，根本不用读"文件内容"。`i_pipe`/`i_cdev`/`i_link` 共用一个 union，因为一个 inode 要么是管道、要么是字符设备、要么是符号链接，互斥。
- **`i_fop`（默认模板）vs `file->f_op`（实例）**：inode 存一份默认 `i_fop`，`open` 时拷给新 `file` 的 `f_op`。这解释了 §3.1 那句"理论上同一 inode 可挂不同 f_op"——因为 `f_op` 是 open 时**拷贝**的，不是共享 inode 的。
- `i_rwsem`：§2.7 战役的遗产，曾是 `i_mutex`。
- `i_dentry`（hlist）：一个 inode 指向**它的所有 dentry**。硬链接 = 同一 inode 上挂多个 dentry alias，全串在这条 `i_dentry` 上（每个 dentry 用 `d_u.d_alias` 接入）。**硬链接的源码本质在这里一目了然**。

### 3.5 路径查找（path lookup）：RCU-walk 真实源码精读

这是 VFS 最核心的操作，每次 `open/stat/execve` 都过它。上一版章节描述了 rcu-walk/ref-walk，这一版**给真实源码并逐行注解**。

**先看那段定调的注释**（解释两种模式与回退）：

```c
/*
 * Path walking has 2 modes, rcu-walk and ref-walk (see
 * Documentation/filesystems/path-lookup.txt).  In situations when we can't
 * continue in RCU mode, we attempt to drop out of rcu-walk mode and grab
 * normal reference counts on dentries and vfsmounts to transition to ref-walk
 * mode.  Refcounts are grabbed at the last known good point before rcu-walk
 * got stuck, so ref-walk may continue from there. If this is not successful
 * (eg. a seqcount has changed), then failure is returned and it's up to caller
 * to restart the path walk from the beginning in ref-walk mode.
 */
```
【真实源码 v6.12 fs/namei.c 约 759 行】

这段注释把回退策略说死了：rcu-walk 卡住时，**在"最后一个已知正确的点"补上引用计数**转 ref-walk 接着走（不是从头）；连补引用都失败（seqcount 变了，说明并发改动），才**整条从头用 ref-walk 重来**。

**核心快路径函数 `lookup_fast()`**：

```c
static struct dentry *lookup_fast(struct nameidata *nd)
{
	struct dentry *dentry, *parent = nd->path.dentry;
	int status = 1;

	/*
	 * Rename seqlock is not required here because in the off chance
	 * of a false negative due to a concurrent rename, the caller is
	 * going to fall back to non-racy lookup.
	 */
	if (nd->flags & LOOKUP_RCU) {
		dentry = __d_lookup_rcu(parent, &nd->last, &nd->next_seq);
		if (unlikely(!dentry)) {
			if (!try_to_unlazy(nd))
				return ERR_PTR(-ECHILD);
			return NULL;
		}

		/*
		 * This sequence count validates that the parent had no
		 * changes while we did the lookup of the dentry above.
		 */
		if (read_seqcount_retry(&parent->d_seq, nd->seq))
			return ERR_PTR(-ECHILD);

		status = d_revalidate(dentry, nd->flags);
		if (likely(status > 0))
			return dentry;
		if (!try_to_unlazy_next(nd, dentry))
			return ERR_PTR(-ECHILD);
		if (status == -ECHILD)
			/* we'd been told to redo it in non-rcu mode */
			status = d_revalidate(dentry, nd->flags);
	} else {
		dentry = __d_lookup(parent, &nd->last);
		if (unlikely(!dentry))
			return NULL;
		status = d_revalidate(dentry, nd->flags);
	}
	if (unlikely(status <= 0)) {
		if (!status)
			d_invalidate(dentry);
		dput(dentry);
		return ERR_PTR(status);
	}
	return dentry;
}
```
【真实源码 v6.12 fs/namei.c 约 1656 行】

**逐行注解**（这是本章最该读懂的一段）：

- `if (nd->flags & LOOKUP_RCU)`：分叉。带 `LOOKUP_RCU` 走无锁快路径；否则走 ref-walk 的 `__d_lookup`。
- `__d_lookup_rcu(parent, &nd->last, &nd->next_seq)`：**无锁查 dcache**（详见下面 `__d_lookup_rcu` 源码），把找到时的 `d_seq` 序号写回 `nd->next_seq`。
- 找不到 → `try_to_unlazy(nd)`：尝试把"目前为止 RCU-walk 走到的位置"锁定（补引用计数）转 ref-walk。补成功返回 `NULL`（让上层用 ref-walk 重查这一步）；**补失败返回 `-ECHILD`**（`ECHILD` 在 namei 里是"RCU 失败，请用 ref-walk 从头重来"的内部信号）。
- **`read_seqcount_retry(&parent->d_seq, nd->seq)`**：这是 RCU-walk 正确性的命门。校验"我查 child 这段时间里，parent 没被并发改过"（比如 parent 被 `d_move` rename 了）。序号变了就 `-ECHILD` 退出。**这就是 seqlock 兜底的那一行**——无锁读靠"读完回头看号没变"保证一致。
- `d_revalidate(dentry, nd->flags)`：调文件系统的 `->d_revalidate`。**对本地 FS 这步基本是 no-op（直接 status=1）；对 NFS，这步要回服务端校验、可能睡眠**。`status > 0` 直接返回成功；否则 `try_to_unlazy_next` 尝试转 ref-walk。
- 注释 `/* we'd been told to redo it in non-rcu mode */`：`d_revalidate` 返回 `-ECHILD` 表示"我在 RCU 模式下没法校验（要睡眠），请转 ref-walk 再调我一次"。

> **面试高频追问"为什么 NFS 不能走 rcu-walk 快路径"**——源码级答案：NFS 的 `d_revalidate` 要回服务端校验（可能睡眠），而 RCU-walk 临界区持 RCU 读锁**绝对不能睡眠**。所以 `lookup_fast` 里 `d_revalidate` 一旦返回 `-ECHILD`，必然 `try_to_unlazy_next` 降级到 ref-walk。这是 NFS 路径查找天然比本地 FS 慢的源码根因。

**回退函数 `try_to_unlazy()`**（注意：**老资料/上一版章节里的 `unlazy_walk()` 已重命名为此**）：

```c
static bool try_to_unlazy(struct nameidata *nd)
{
	struct dentry *parent = nd->path.dentry;

	BUG_ON(!(nd->flags & LOOKUP_RCU));

	if (unlikely(!legitimize_links(nd)))
		goto out1;
	if (unlikely(!legitimize_path(nd, &nd->path, nd->seq)))
		goto out;
	if (unlikely(!legitimize_root(nd)))
		goto out;
	leave_rcu(nd);
	BUG_ON(nd->inode != parent->d_inode);
	return true;
out1:
	nd->path.mnt = NULL;
	nd->path.dentry = NULL;
out:
	leave_rcu(nd);
	return false;
}
```
【真实源码 v6.12 fs/namei.c 约 780 行】

- `legitimize_links/legitimize_path/legitimize_root`：把 RCU-walk 期间"只看了一眼没持引用"的 symlink 栈、当前 path、root 都补上正式引用计数。每一步都内含 seqcount 校验——**只要发现序号变了就 `goto out` 返回 false**（让上层从头 ref-walk）。
- `leave_rcu(nd)`：退出 RCU 读临界区。**成功路径和失败路径都调它**——RCU 读锁必须释放。
- 函数名 **`unlazy`**：把"lazy（懒，不持引用）"的 RCU-walk 状态转成"eager（持引用）"的 ref-walk 状态。

**无锁 dcache 查找 `__d_lookup_rcu()`**（带它那段教科书级注释）：

```c
/**
 * __d_lookup_rcu - search for a dentry (racy, store-free)
 * @parent: parent dentry
 * @name: qstr of name we wish to find
 * @seqp: returns d_seq value at the point where the dentry was found
 * Returns: dentry, or NULL
 *
 * __d_lookup_rcu is the dcache lookup function for rcu-walk name
 * resolution (store-free path walking) design described in
 * Documentation/filesystems/path-lookup.txt.
 * ...
 * NOTE! The caller *has* to check the resulting dentry against the sequence
 * number we've returned before using any of the resulting dentry state!
 */
struct dentry *__d_lookup_rcu(const struct dentry *parent,
				const struct qstr *name,
				unsigned *seqp)
{
	u64 hashlen = name->hash_len;
	const unsigned char *str = name->name;
	struct hlist_bl_head *b = d_hash(hashlen);
	struct hlist_bl_node *node;
	struct dentry *dentry;
	...
	hlist_bl_for_each_entry_rcu(dentry, node, b, d_hash) {
		unsigned seq;
		/* ... 大段注释解释 raw_seqcount_begin 为何不等序号稳定 ... */
		seq = raw_seqcount_begin(&dentry->d_seq);
		if (dentry->d_parent != parent)
			continue;
		if (d_unhashed(dentry))
			continue;
		if (dentry->d_name.hash_len != hashlen)
			continue;
		if (dentry_cmp(dentry, str, hashlen_len(hashlen)) != 0)
			continue;
		*seqp = seq;
		return dentry;
	}
	return NULL;
}
```
【真实源码 v6.12 fs/dcache.c 约 2159 行（节选）】

- `hashlen = name->hash_len`：一把取出打包的 hash+len（§3.2 的 qstr 优化）。
- `d_hash(hashlen)`：算哈希桶。桶头是 `hlist_bl_head`（bit-locked）。
- `hlist_bl_for_each_entry_rcu`：**RCU 读侧遍历桶链，不加桶锁**。
- `seq = raw_seqcount_begin(&dentry->d_seq)`：记下候选 dentry 的序号。**`raw_` 版不等序号稳定**（原文注释解释：反正后面慢比较会 seqretry，命中就要退出 RCU 了，这里不值得等）。注意 `raw_seqcount_begin` 仍含 `smp_rmb()`，保证读到的 `d_name.name` 是 NUL 结尾的（内存序在这里是load-bearing 的）。
- 四个 `continue` 过滤：parent 不对 / 已 unhash / 长度+哈希不符 / 逐字节不等。
- `*seqp = seq; return dentry`：**返回 dentry 和它的序号，但绝不替调用者校验**——注释吼了 *"The caller has to check the resulting dentry against the sequence number"*。校验在上面 `lookup_fast` 那行 `read_seqcount_retry`。

**完整路径查找流程（`/home/user/app/config.yaml`）**：

```
1. 从 task_struct->fs->root 或 cwd 出发（nd->path.dentry）
2. 对每个分量调 lookup_fast(nd)：
   "home"        → __d_lookup_rcu 命中 → d_seq 校验 OK → 取 d_inode → 权限检查(exec bit)
   "user"        → __d_lookup_rcu 命中 → ...
   "app"         → __d_lookup_rcu 未命中(返回 NULL)
                    → try_to_unlazy 补引用转 ref-walk
                    → 慢路径 walk_component → 调 dir->i_op->lookup()
                      (ext4_lookup 读磁盘目录块 / nfs_lookup 回服务端)
                    → 新 dentry 经 d_splice_alias 插入 dcache
   "config.yaml" → ...
3. 产物：struct path { vfsmount, dentry }，存进 file->f_path
```

### 3.6 "一切皆文件"的机械核心：`fd → file → f_op` 分发到底在哪一层

"一切皆文件"能成立，关键在于 **`read(fd)` 的分发绑在 `struct file` 上、而不是 inode 上**。一次 `read(2)` 的内核路径：

```
read(fd) → fdget(fd)
            └─ current->files->fdt->fd[fd]  → struct file*   (按 fd 索引进程 fdtable)
                                               └─ file->f_op->read_iter(...)   ← 分发点
```

- 分发依据是 `file->f_op`，而 `f_op` 是 `open` 时由具体子系统（ext4 / sockfs / procfs / anon_inode）**塞进这个 file 对象**的（源头是 inode 的 `i_fop`，§3.3）。VFS 对"背后是什么"一无所知，只认 `f_op` 这张表。这就是 socket、pipe、epoll 能和普通文件共用 `read/write/poll/close` 语义的根因——它们填了**同形状的 `file_operations`**。
- **分发单位 per `struct file`，不是 per inode**：同 inode `open` 两次得两个 `file`，各有独立 `f_pos`/`f_flags`，理论上甚至可挂不同 `f_op`。
- **`fdget` 快路径**：靠 §3.1 那个 `SLAB_TYPESAFE_BY_RCU` + 引用计数，单线程持 fd 时连原子操作都能省（用 `FDPUT_FPUT` 标志记是否需要后续 `fput`）。

`struct file_operations` 在 v6.12 已经膨胀到 **30+ 个 hook**（这是"一切皆文件"覆盖面的证据）：

```c

	struct module *owner;
	fop_flags_t fop_flags;
	loff_t (*llseek) (struct file *, loff_t, int);
	ssize_t (*read) (struct file *, char __user *, size_t, loff_t *);
	ssize_t (*write) (struct file *, const char __user *, size_t, loff_t *);
	ssize_t (*read_iter) (struct kiocb *, struct iov_iter *);    // ← 现代异步读，io_uring 走它
	ssize_t (*write_iter) (struct kiocb *, struct iov_iter *);
	int (*iopoll)(struct kiocb *kiocb, struct io_comp_batch *, unsigned int flags);
	int (*iterate_shared) (struct file *, struct dir_context *); // ← 读目录（并行版）
	__poll_t (*poll) (struct file *, struct poll_table_struct *);// ← select/poll/epoll 靠它
	long (*unlocked_ioctl) (struct file *, unsigned int, unsigned long);
	...
	int (*mmap) (struct file *, struct vm_area_struct *);
	int (*open) (struct inode *, struct file *);
	int (*release) (struct inode *, struct file *);
	int (*fsync) (struct file *, loff_t, loff_t, int datasync);

```
【真实源码 v6.12 include/linux/fs.h，约 2068 行起（节选）】

- **`read_iter`/`write_iter`（基于 `iov_iter`）才是现代主路径**，老的 `read`/`write` 多数文件系统已不实现。io_uring 调的就是 `read_iter`（带 `IOCB_NOWAIT`）。
- **`uring_cmd` 是 io_uring 时代的扩展点**（§6.2）：让设备/文件系统暴露自定义 io_uring 操作（如 NVMe passthrough、ublk）。**这是"一切皆文件"在 io_uring 时代的延伸**——连"自定义命令"都被收编进 `file_operations`。
- `splice_read`/`copy_file_range`/`remap_file_range`：零拷贝家族，让数据在内核内 fd 间搬不经用户态。

> **细节修正**：上一版章节/部分老资料提到的 `anon_inode_getfile_secure` 这个函数名，在 **v6.12 里已不存在**——现在统一走 `__anon_inode_getfile(..., context_inode, secure)` 内部分发，对外接口是 `anon_inode_create_getfd`。所以"整个系统只有一个 anon inode"已从"绝对真理"退化为"默认成立、安全敏感路径有独立 inode"——面试当绝对真理说会被抓。【函数名以 v6.12 fs/anon_inodes.c 实际为准】

> 一句话记牢：**fd 是进程级索引，`struct file` 是"打开状态 + f_op 分发表"的载体，inode 是文件实体。"一切皆文件"统一的是 `f_op` 这层接口，不是底层实现。**

### 3.7 procfs / sysfs / tmpfs：伪文件系统与 seq_file

**对照表：伪文件系统速查**

| 文件系统 | 挂载点 | 本质 | 典型用途 |
|----------|--------|------|---------|
| `procfs` | `/proc` | 进程/内核信息动态视图 | `/proc/PID/`、`/proc/meminfo` |
| `sysfs` | `/sys` | 设备模型（kobject）投影 | 驱动参数、热插拔 |
| `tmpfs` | `/tmp`,`/run`,`shm` | 内存文件系统（inode + Page Cache） | 共享内存、临时文件 |
| `devtmpfs` | `/dev` | 设备节点自动创建 | 字符/块设备访问 |
| `cgroup2fs` | `/sys/fs/cgroup` | cgroup 控制接口 | 资源限制 |
| `overlayfs` | 容器根 | 多层叠加 | 容器镜像分层 |
| `debugfs` | `/sys/kernel/debug` | 调试接口 | tracing、kprobes |

**procfs 读取机制**：`cat /proc/meminfo` 不读磁盘，而是调内核函数 `meminfo_proc_show()`（`fs/proc/meminfo.c`），读内核统计量（`si_meminfo()` 汇总、各 zone 页计数等），用 `seq_printf` 格式化返回。**每次 `cat` 都是一次内核函数调用，没有缓存——高频读 `/proc` 有真实 CPU 成本，不是"读内存白嫖"**。（`totalram_pages` 在 5.0 后从全局变量变成读 atomic 的访问函数 `totalram_pages()`，又一个版本演进点。）

`seq_file` 是 procfs/sysfs 输出大量数据的通用框架，提供 `start/next/show/stop` 迭代器回调，解决"内核要输出的内容 > 用户缓冲区"的分块问题——内核不必手动管理偏移和"这次读到哪了"。

**sysfs 与 PAGE_SIZE 约束**：sysfs 是设备模型（kobject/ktype/kset）的文件系统投影，每个 kobject 一个目录、每个 attribute 一个文件，读写触发驱动 `show`/`store`。**关键约束：每个 sysfs attribute 读写必须一次操作内完成、不超过 PAGE_SIZE（通常 4KB）**——为保证原子语义、让驱动不必处理分块。这是 sysfs 区别于 procfs 的硬约束（procfs 用 seq_file 支持分块）。

## 四、落地：生产调优与排障（每条都带"底层根因"）

### 4.1 路径查找慢：dcache miss / negative dentry 排查

```bash
# dcache 与 inode cache 用量
slabtop -s c | head -20
grep -E "^dentry|^inode_cache" /proc/slabinfo

# dcache 总体状态：dentry数 未使用数 年龄阈值 ...
cat /proc/sys/fs/dentry-state

# 内核侧 d_lookup 热点（需 root + 内核符号）
sudo bpftrace -e 'kprobe:d_lookup { @[comm] = count(); }'
# RCU-walk 命中 vs 降级 ref-walk 的比例（间接看 try_to_unlazy 调用频率）
sudo bpftrace -e 'kprobe:try_to_unlazy { @unlazy = count(); }'
```

**💡 negative dentry 的坑（底层根因）**：`open("/tmp/notexist")` 返回 ENOENT 时，内核仍可能缓存一个 `d_inode == NULL` 的 **negative dentry**。好处是下次再查同名直接 dcache 命中返回 ENOENT、不碰磁盘。**根因级危害**：高并发频繁 `stat` 不存在的文件（典型：每次请求按时间戳拼一个不存在的缓存文件名去 `stat`），negative dentry 暴涨，挤占 dcache 把**有效** dentry 顶出 LRU，导致真实路径查找 miss 率飙升。这是"缓存被污染数据淹没"的经典反模式——和应用层缓存被冷数据击穿同构。

**应急**（生产慎用）：`echo 2 > /proc/sys/vm/drop_caches` 清 dcache+icache（连带清 reclaimable slab），会让随后一段时间所有路径查找重新 miss、I/O 抖动。

### 4.2 `/proc/sys/fs` 关键参数（区分"旋钮"与"只读观测"）

| 参数 | 含义 | 性质 |
|------|------|------|
| `file-max` | 系统最大 fd 数 | **可调**，高并发服务调大（如 1000000） |
| `nr_open` | 单进程 fd 上限硬顶 | **可调**，`ulimit -n` 不能超它 |
| `dentry-state` | dcache 统计 | **只读观测** |
| `inode-nr`/`inode-state` | inode cache 用量 | **只读观测**——inode 缓存早由 slab+shrinker 动态管理，老资料里能写的 `inode-max` 已失效 |
| `pipe-max-size` | 单管道最大缓冲区 | **可调**，高吞吐日志管道调大 |

> **know 戳一下**：把"只读观测"当"调优旋钮"是常见误区。`inode-state` 只能看不能调；想限制 inode 缓存只能靠内存压力 + shrinker，没有直接上限旋钮。

### 4.3 overlayfs 与容器排障

```bash
cat /proc/self/mountinfo                       # 当前进程看到的挂载视图（含传播类型）
nsenter --target PID --mount -- cat /proc/mounts   # 进特定容器的 mount namespace 看
# overlay 行示例：overlay / overlay rw,lowerdir=...,upperdir=...,workdir=...
```

**💡 overlayfs 的 inode 号坑（根因）**：upper 层和 lower 层合并后，两层 inode 号可能冲突（各自底层 fs 分配）。导致 `find -inum`、`tar` 的硬链接检测等**依赖 inode 号唯一性**的工具在容器里行为异常。根因是 inode number 只在**单个文件系统内**唯一，overlayfs 是合成视图、跨了两个底层 fs。生产准则：**容器内不要跨层依赖 inode 号唯一性**。（overlayfs 有 `xino` 特性可在高位 bit 编码层号缓解，但有地址空间代价。）

### 4.4 strace 追踪 open 的 VFS 路径

```bash
strace -e trace=openat,read,close cat /proc/meminfo
# openat(AT_FDCWD, "/proc/meminfo", O_RDONLY) = 3
#   → path lookup（lookup_fast 逐分量）→ procfs inode → file->f_op->open → 装 f_op → fd_install → 返回 3
# read(3, "MemTotal: ...", 65536) = N
#   → file->f_op->read_iter → procfs seq_file → meminfo_proc_show() 实时生成
# close(3) = 0 → fput → f_count 归零 → file->f_op->release
```

想看**内核侧** path lookup 的栈（注意 `strace -k` 打的是**用户态**栈，不是内核栈）：

```bash
sudo bpftrace -e 'kprobe:vfs_open { printf("%s\n", kstack); exit(); }'
```

### 4.5 inotify / fanotify：基于 VFS hook 的文件监控

`inotify` 原理：VFS 在 `vfs_create`/`vfs_unlink`/`vfs_rename` 等关键点埋了 `fsnotify_*` hook（`fsnotify_create`/`fsnotify_unlink`/`fsnotify_move`，统一走 `fsnotify` 框架）。watch 注册时把 mark 挂到目标 inode 的 `i_fsnotify_marks`（§3.3 那个 `struct fsnotify_mark_connector __rcu *i_fsnotify_marks` 字段就是它）。文件操作触发 hook → 事件投递到监听 fd 的队列 → 用户态 `read(inotify_fd)` 取走。inotify 和 fanotify 共用这套 `fsnotify` 基础设施。

**🎯面试加分**：inotify 是 **inode 级**监控（不跟文件名）——文件 `rename` 后原 watch 继续有效，但对新路径名的访问不触发老 watch。fanotify 提供更强语义（含路径级监控、权限事件可拦截 open），但需 `CAP_SYS_ADMIN`。

## 五、方案对比：把推理过程走一遍（不只列表，跑场景）

> 硬性要求第 3 条：对照表 + 具体场景跑一遍，带读者沿推理走，点出不适用边界。

### 5.1 路径查找：RCU-walk vs ref-walk —— 用一个 8 核场景跑

| 维度 | rcu-walk（快路径） | ref-walk（稳妥路径） |
|---|---|---|
| 持锁 | 仅 RCU 读锁 | `dget` 引用 + 必要时 `d_lock`/`rename_lock` |
| 对共享缓存行 | **零写入** | 每过一个 dentry `atomic_inc/dec` |
| 多核竞争 | 趋近零（缓存行可 Shared 共存） | 公共 dentry 缓存行弹跳 |
| 能否睡眠 | **绝对不能**（持 RCU 读锁） | 可以 |
| 适用 | 本地 FS、dcache 命中、无需 revalidate | NFS、要读 symlink 内容、managed 挂载点、并发改动 |

**场景跑一遍：8 核机器，8 个线程同时 `stat("/var/log/app/x.log")`，全部 dcache 命中。**

- **若强制 ref-walk**：每个线程对 `/`、`var`、`log`、`app` 四个公共 dentry 各做一次 `dget`+`dput`（8 次 `atomic_inc`/`dec`）。`/` 这个 dentry 的 `d_lockref` 缓存行被 8 个核轮流抢 Exclusive 态——**每次原子操作要等缓存行在核间迁移（几十到上百 ns）**。8 核 × 4 层 × 2 次 = 64 次跨核缓存行争用。吞吐被这个串行点卡死，**加核不加速、甚至更慢**（争用更激烈）。这就是 LWN 419811 说的 cacheline bouncing。
- **走 rcu-walk**：8 个线程对同样四个 dentry **全部只读**（`__d_lookup_rcu` + `read_seqcount_retry`）。这四个 dentry 的"RCU lookup touched"缓存行在 8 个核上以 **Shared 态并存**，**没有任何核 invalidate 它**。8 个线程几乎完全并行，吞吐随核数线性扩展。**这就是为什么 §3.2 要把 `d_lockref` 和只读区隔开缓存行**——隔开后写 `d_lockref` 的核不会踩到读只读区的核。

**不适用边界**：把同一场景换成 **NFS** 上的路径。`d_revalidate` 要回服务端校验、可能睡眠 → rcu-walk 临界区不能睡 → 每个线程都在 `lookup_fast` 里 `d_revalidate` 返回 `-ECHILD` → `try_to_unlazy_next` 降级 ref-walk → 又退回 cacheline bouncing + 网络往返。**结论：rcu-walk 的红利只在"本地 + 命中 + 无需 revalidate"时兑现；NFS 路径查找慢是双重的（既降级又走网络）。**

### 5.2 容器文件隔离：mount namespace+overlay vs chroot vs 虚拟机

| 方案 | 隔离强度 | 镜像共享 | 开销 | 致命短板 |
|---|---|---|---|---|
| `chroot` | **弱**（只换根，不隔挂载树） | 无 | ~0 | 有 `CAP_SYS_ADMIN` 可逃逸；不隔 mount |
| mount ns + overlay + bind | 中（文件系统视图隔离） | **强**（共享 lowerdir） | 低 | overlay rename/inode 号语义坑 |
| 虚拟机（独立 kernel + 虚拟块设备） | 强 | 弱（各自镜像） | 高（内存/启动） | 重，密度低 |

**场景跑一遍：要跑 500 个共享同一基础镜像、各带自己 volume 的容器。**

- **chroot 直接出局**：`chroot` 只限制路径前缀，不隔离挂载树——容器 A `mount` 一个设备，容器 B 和宿主都看得见；且有能力的进程能 `chroot` 逃出（经典 `chroot` 逃逸）。**"容器隔离靠 chroot"是错的**，这是面试陷阱。
- **虚拟机**：500 个 VM = 500 份内核 + 500 份镜像内存，密度太低、启动太慢，否决。
- **mount ns + overlay 胜出，推理链**：① 每容器一个 mount namespace（`unshare(CLONE_NEWNS)` + `pivot_root` 切根），挂载树互不可见；② 500 个容器**共享同一只读 lowerdir**（基础镜像），各自独立 upperdir（写层，CoW），磁盘只多存"改动量"——这是密度的关键；③ volume 通过 bind mount 进各 namespace；④ 宿主 `/` 设 `shared`、容器内挂载点设 `slave`/`private`（§2.6 shared subtrees），既能让宿主新挂的设备按需传进去，又不让容器的挂载泄回宿主。
- **不适用边界**：overlay 的 **rename 跨层语义**——把 lower 层的大目录 rename 到 upper，需要把整个目录树从 lower **复制**到 upper（copy-up），大目录 rename 可能慢到秒级。**若工作负载有频繁大目录 rename（如某些构建系统），overlay 会成瓶颈**，此时考虑 btrfs 子卷快照（CoW 原生、rename 不 copy-up，但要求宿主 FS 是 btrfs）。

### 5.3 inode/dcache 回收顺序：为什么必须"先 dentry 后 inode"

这不是"约定"，是 §2.4 那句 *"dcache is a master of the icache"* 的强制后果。**是 dentry 通过 `d_inode` 把 inode "钉"在内存里**（dentry 持 inode 引用），不是反过来。所以：

```
内存压力 → shrinker 扫 superblock 上的两条 LRU：
   s_dentry_lru（先）：释放冷 dentry → dentry 的 d_inode 引用消失
                                       → inode 的 i_count 才可能归零
   s_inode_lru（后）：i_count == 0 的 inode 才能被回收
```

**场景跑一遍：一个 inode 上挂着 3 个 dentry（硬链接）+ 1 个进程持有的 `file`。** 想回收这个 inode？必须先让 3 个 dentry 全部释放（`i_dentry` 链清空），且那个 `file` 也 `close`（`f_count` 归零、`f_inode` 引用消失），`i_count` 才到 0。**只要还有任一 dentry 指着它、或任一进程 `file` 引用着它，inode 就回收不掉**——这正是 `rm` 一个被打开文件、数据不释放的机制（章末 T2）。

**生产排障**：`/proc/slabinfo`（或 `slabtop`）里 `dentry` 和 `inode_cache` 异常大，通常是路径查找压力大、negative dentry 污染、或 fd/inode 泄漏。`lsof | grep deleted` 抓"已删但被持有"的 inode。

## 六、未来演进

### 6.1 Rust for Linux 与 VFS：引用计数模型 vs ownership 的阻抗

Rust for Linux（主线合入始于 6.1，持续演进）正为 VFS 接口提供 Rust 抽象，目标是新文件系统驱动可用 Rust 写、与 C VFS 互操作。难点是**阻抗匹配**：VFS 对象生命周期靠引用计数（`i_count`/`d_lockref`/`kref`），而 Rust 倾向 ownership/borrow。Rust 侧要精确表达 `file_operations`/`inode_operations`/`super_operations` 的生命周期语义——借用检查器在"避免 use-after-free inode"上有天然优势，但要把"一个 inode 被多个 dentry 共享、引用计数管理"映射成 Rust 类型（`Arc`/`ARef`）仍在演进。

### 6.2 io_uring 与 VFS：不是绕过，是批量化 + `*_iter` 异步接口

io_uring（5.1+）的核心增益**不是"绕过 VFS"**，而是用一对和用户态共享的环（SQ/CQ）把多次 I/O 的提交和收割批量化，省掉"每操作一个 syscall"的进出内核开销。它访问文件**仍走 VFS**：优先调底层异步 `->read_iter`/`->write_iter`（带 `IOCB_NOWAIT`），**只有当某操作会阻塞时才甩给内核工作线程池 `io-wq` 兜底**——这才是"offload 到工作线程"的真实触发条件，不是常态。`file_operations` 的 `uring_cmd`（§3.6）让设备/文件系统暴露自定义 io_uring 操作（NVMe passthrough、ublk）。**理解 VFS 的 `*_iter` 异步接口，是理解 io_uring 为何既快又通用的基础。**

### 6.3 bcachefs：在 VFS 框架内做企业特性的工程案例

bcachefs（主线合入 6.7）在 VFS 框架下实现快照、压缩、加密、多设备 RAID，同时保持 VFS 接口兼容。它历经多年 RFC 的合入过程，本身就是"如何在 VFS 既有契约内塞进复杂语义"的案例——也伴随关于"维护者协作模式"的社区争论。

### 6.4 FUSE：用户态文件系统与 passthrough

FUSE 让用户态程序实现文件系统，经 `/dev/fuse` 字符设备与 VFS 通信：每次 VFS 调用通过 FUSE 协议发请求到用户态守护进程、等响应。代价是额外上下文切换 + 数据拷贝，换来实现的安全隔离。生产案例：GCS FUSE、S3FS、SSHFS。**FUSE passthrough（近年）**试图对已打开的底层文件做直通，绕过用户态守护、减少拷贝，把"读写直接落到底层 fd"——是 FUSE 性能短板的针对性补丁。

## 章末五件套

### 高频面试题（含答题要点）

**T1**：hardlink 和 softlink 在 VFS 层面的区别？为什么目录不能 hardlink？
- hardlink = 同一 inode、不同 dentry（多个 dentry 串在 inode 的 `i_dentry` 链上，各用 `d_u.d_alias` 接入）；只能同 fs 内（inode number 仅 fs 内唯一）。
- softlink = 新 inode，数据是目标路径串（存在 inode 的 `i_link` 字段，`get_link` 返回它让 namei 继续查）。
- 目录默认不能 hardlink：防目录树成环 → `find`/`fsck` 死循环（且会破坏 `..` 唯一父的假设）。

**T2**：`rm` 一个正被打开的文件，数据何时真正释放？
- `unlink` 只摘 dentry；只要进程持 fd（`file->f_count > 0`，`file` 经 `f_inode` 持 inode 引用），inode 与 Page Cache 不释放；进程 `close`/`exit` 后 `i_count` 归零才真正释放（§5.3 的回收链）。
- 加分：`lsof +L1` / `lsof | grep deleted` 排查"已删但磁盘不释放"。源码根据：§3.1 的 `f_count` + §3.3 的 `i_count`。

**T3**：为什么 `cat /proc/PID/maps` 每次输出可能不同？
- procfs 输出是实时内核函数调用结果、无缓存（§3.7）；进程内存映射随 mmap/munmap 变化；多线程下有竞态。seq_file 在迭代输出期间也不锁死整个地址空间。

**T4**：容器里 `df` 看到的是宿主机磁盘还是容器的？overlayfs 如何影响？
- `df` 调 `statfs`，VFS 转发给对应 fs 的 `statfs` 实现；overlayfs 的 `statfs` 通常汇报 **upperdir 所在 fs** 的空间；容器常看到宿主机磁盘总量（用户困惑点）。

**T5**：`open()` 的 `O_CREAT|O_EXCL` 原子性谁保证？
- 本地：VFS 的 `vfs_create` 调 `i_op->create` 时持**父目录 inode 的写锁**（`inode_lock`，即 §2.7 后的 `i_rwsem` 写模式；老资料的 `i_mutex`）。单机内由此保证原子。
- 跨网络：NFS 不能靠本地锁。早期 NFSv2/v3 EXCLUSIVE create 有重传伪失败。NFSv4 OPEN 引入 **exclusive-create verifier**（客户端生成一次性 cookie，服务端记在文件时间戳字段）：重传到达时服务端认出"同一次创建的重试"返回成功而非 EEXIST——把非幂等创建变成可安全重传的幂等操作（细节以 RFC 7530/8881 为准）。

**T6**：mmap 和 read 走 Page Cache 的路径区别？（见第 10 章）
- mmap 经 `vm_area_struct` 建立虚拟地址→Page Cache 页映射，缺页时填充；read 经 `file->f_op->read_iter` 把 Page Cache 拷到用户缓冲；两者都过 inode 的 `address_space`（`i_mapping`/`i_data`，§3.3）。

**T7**：什么是 superblock？一个文件系统类型能有多个 superblock 实例吗？
- 内核里的 `struct super_block` 是 in-memory **挂载实例**（每次挂载一个，§3.4），同一类型（如 tmpfs）可多次挂载 = 多个 superblock。
- 别和**磁盘上的 on-disk superblock** 混：磁盘位置是 **fs 私有约定**——ext2/3/4 主 superblock 在偏移 1024（前 1024 留 boot block）、多 block group 留备份；XFS/btrfs 布局完全不同。**"superblock 在偏移 1024"只对 ext 家族成立**。

**T8（新增，考 v6.12 演进）**：目录锁为什么从 `i_mutex` 变成 `i_rwsem`？引入了什么新机制？
- 因为 lookup 是只读的，没理由互斥（Al Viro, LWN 685108）。换成 `i_rwsem` 后 lookup 持读锁可并行，create/unlink/rename 持写锁。
- 新机制：**in-lookup dentry**——两个并行 lookup 查同名未缓存项可能各建一个 dentry，灾难。解法是给"lookup 进行中"造个标记对象、挂在 parent 的哈希里，后到者等先到者完成（`struct dentry` 里 `d_wait` + `d_u.d_in_lookup_hash` 就是为此存在，§3.2）。

### 实战项目：P0 — 追踪 open/read 的完整 VFS 路径 + 看 RCU-walk 降级

**目标**：建立"系统调用 → VFS → 文件系统"的源码级感知，并**亲眼看到 rcu-walk 降级 ref-walk**。

```bash
# Step 1: 基础观察
strace -e trace=openat,read,write,close,stat,lstat cat /etc/hostname

# Step 2: 用户态栈（注意是用户栈不是内核栈）
strace -k -e openat cat /etc/hostname
# 内核侧 path lookup 栈（需 root + 符号）：
sudo bpftrace -e 'kprobe:vfs_open { printf("%s\n", kstack); exit(); }'

# Step 3: 看 RCU-walk vs 降级（本地 FS 几乎全命中 rcu，NFS 路径会狂降级）
sudo bpftrace -e 'kprobe:try_to_unlazy { @[comm]=count(); }'   # 跑点本地+NFS 负载对比

# Step 4: hardlink 的 inode 共享
echo test > /tmp/orig.txt; ln /tmp/orig.txt /tmp/hardlink.txt
stat /tmp/orig.txt /tmp/hardlink.txt    # inode number 相同、Links: 2

# Step 5: 删文件但持有 fd（§5.3 回收链）
exec 3< /tmp/orig.txt; rm /tmp/orig.txt
ls -la /proc/self/fd/3                   # 仍可访问，标 (deleted)
lsof +L1 | grep orig                     # 找到已删但被持有
exec 3>&-
```

**验收**：能说出 `openat("/etc/hostname")` 至少 5 个内核步骤（path lookup → dentry → inode → `f_op->open` → `fd_install`）；能解释 `Links: 2`；能用 `bpftrace` 观察到 NFS 路径比本地路径 `try_to_unlazy` 调用多得多（**亲手验证 §5.1 的推理**）。

### 设计题

**题目**：设计支持"多租户文件系统隔离"的容器平台，使每租户容器：① 看不到其他租户文件；② 可挂自己 volume；③ 共享一套基础镜像。从 VFS 层描述实现并分析 trade-off。

**答题要点**（推理见 §5.2）：
1. 每容器独立 mount namespace（`unshare(CLONE_NEWNS)` + `pivot_root` 切根，§2.6）。
2. overlay：共享只读 lowerdir（镜像层，省磁盘）+ 独立 upperdir（租户写层，CoW）。
3. volume 经 bind mount 进 namespace。
4. shared subtrees：宿主 `/` 设 shared、容器内设 slave/private，控制传播方向（§2.6）。
5. **Trade-off**：overlay rename 跨层 copy-up（大目录极慢）；CoW 写放大；多层 overlay 深度性能下降「具体限制值待核」；替代方案 btrfs 子卷快照（CoW 原生、rename 不 copy-up，但要求宿主 FS 是 btrfs）。

### 系统题

**场景**：线上 Java 服务，`/proc/PID/fd` 下几万个 fd，`lsof` 显示大量 fd 指向已删除文件（`deleted`），磁盘持续增长。描述排查、根因、解决。

**框架**：
1. **确认**：`lsof -p PID | grep deleted | wc -l`。估算占用磁盘**别数 lsof 默认输出的列**（列宽随路径变、不稳）；用机器可读：`lsof -p PID -F sn`（`-F s` 出大小、`-F n` 出名字，每条以字段标志开头，脚本好解析）；或直接看 `/proc/PID/fd/` 里指向 `(deleted)` 的目标大小。
2. **根因分类**：log4j/logback 滚动日志建新文件但旧 fd 没关（线程本地 `FileOutputStream`）；`createTempFile`+`deleteOnExit` 但进程不退；finally 块没关 fd。
3. **定位**：`strace -e trace=close,open -p PID -c` 看 open >> close 即泄漏；`jstack` 看持 `FileInputStream` 的线程。
4. **应急**：可经 `/proc/PID/fd/N` 在进程外读内容，但**无法强制关闭**（只能杀进程）；**磁盘不释放直到那些 fd 关闭**（§5.3）。
5. **根治**：代码修复 + 监控 `process_open_fds`（由 `/proc/PID/fd` 计），设阈值告警。

### 代码题

**题目**：写一个内核模块，在 `/proc/` 下建 `mymodule`，读时返回提示（教学示意 dentry 统计），写时无操作。重点演示 **5.6 后 `proc_ops` 替代 `file_operations`** 的 procfs 注册 + seq_file 框架。

```c
// proc_mymodule.c —— 【示意代码，非逐字】教学用，结构对、按版本可能要调
// 验证环境：Linux >= 5.6（proc_ops）；更低版本用 file_operations + proc_create
#include <linux/module.h>
#include <linux/proc_fs.h>
#include <linux/seq_file.h>

static int mymodule_show(struct seq_file *m, void *v)
{
	// 教学示意：真要读 dentry 统计应读已有接口 /proc/sys/fs/dentry-state，
	// 直接访问内核内部计数符号不稳定、且多版本变化（待核具体符号可见性）
	seq_printf(m, "dentry-state: (see /proc/sys/fs/dentry-state)\n");
	seq_printf(m, "module loaded: %s\n", THIS_MODULE->name);
	return 0;
}

static int mymodule_open(struct inode *inode, struct file *file)
{
	// single_open：seq_file 单页模式，适合输出 < PAGE_SIZE
	return single_open(file, mymodule_show, NULL);
}

// Linux 5.6（commit by Alexey Dobriyan）起 proc 条目用 struct proc_ops，
// 不再复用庞大的 struct file_operations —— 这是一次"给 proc 瘦身"的演进：
// proc 条目根本用不到 file_operations 的几十个 hook，单独一张小表更省内存、语义更清。
static const struct proc_ops mymodule_ops = {
	.proc_open    = mymodule_open,
	.proc_read    = seq_read,
	.proc_lseek   = seq_lseek,
	.proc_release = single_release,
};

static struct proc_dir_entry *proc_entry;

static int __init mymodule_init(void)
{
	proc_entry = proc_create("mymodule", 0444, NULL, &mymodule_ops);
	if (!proc_entry)
		return -ENOMEM;
	pr_info("mymodule: /proc/mymodule created\n");
	return 0;
}

static void __exit mymodule_exit(void)
{
	proc_remove(proc_entry);
	pr_info("mymodule: /proc/mymodule removed\n");
}

module_init(mymodule_init);
module_exit(mymodule_exit);
MODULE_LICENSE("GPL");
MODULE_AUTHOR("learning");
MODULE_DESCRIPTION("VFS procfs demo");
```

**验证**：
```bash
make && sudo insmod proc_mymodule.ko
cat /proc/mymodule
cat /proc/sys/fs/dentry-state    # 对比真实数据
sudo rmmod proc_mymodule && dmesg | tail -5
```

**暗坑**：
1. `proc_ops`（5.6 引入）替代 `file_operations`，低版本内核必须用 `file_operations`——**这本身就是 §3.6 "proc 条目其实不需要完整 f_op" 的演进证据**。
2. `single_open` 把整个输出缓冲在内存，输出大时改用 seq_file 迭代器（`start/next/stop/show`）。
3. `rmmod` 必须正确 `proc_remove`，否则访问已卸载模块的 proc 文件会 kernel panic（模块代码段已释放，`->proc_read` 指向野地址）。

---

> **本章心法**：VFS 不是文件系统，是一张**接口协议 + 一套为多核可扩展性而生的缓存机制**。四大对象的**字段布局**（按 cacheline 读写分离）、RCU-walk 的**零写入无锁查找**、mount namespace+shared subtrees 的**传播语义**、`i_rwsem` 的**并行 lookup**——这四样才是把"一切皆文件"从哲学口号变成能扛百万 QPS 的工程系统的真正硬核。读懂 `struct dentry` 那句 `keep separate from RCU lookup area if possible!`，你就读懂了内核工程师怎么把排队论结论写进字段顺序里。
