## TL;DR（先给结论）

- **PPO（Proximal Policy Optimization, Schulman et al. 2017）是把 TRPO 的「trust region」思想用一阶方法重做一遍**。第 3 章的朴素 policy gradient 有个致命问题：on-policy 采来的一批数据，只能走一步梯度就作废，**样本效率极低**；而你一旦想多走几步 / 步子迈大，策略就可能崩（policy collapse）。TRPO 用一个硬性 KL 约束 + 二阶优化（conjugate gradient + Hessian-vector product）解决了「步子别迈太大」，但实现复杂、不好和 dropout/参数共享配合。PPO 的全部贡献就是：**用一个 clip 过的 surrogate objective，把「别让新旧策略差太远」这件事压进一阶 SGD 里**，既能对同一批数据走多个 epoch（样本效率↑），又稳。
- **核心就一个式子——clipped surrogate objective**：记概率比 `r_t(θ) = π_θ(a_t|s_t) / π_θ_old(a_t|s_t)`，则
  `L^CLIP(θ) = E_t[ min( r_t·A_t , clip(r_t, 1−ε, 1+ε)·A_t ) ]`，推荐 `ε=0.2`。这个 `min`+`clip` 的组合是一个**悲观下界（pessimistic bound）**：当某个动作的优势 `A>0`、你想加大它的概率时，比值涨到 `1+ε` 就**截断收益**，不再奖励你继续偏离；当 `A<0` 时对称地在 `1−ε` 处截断。它的净效果是「在信任区域内放心更新，越界就没有梯度好处」。
- **PPO 不是只有 clip。完整 loss 是三项**：`L = L^CLIP − c1·L^VF + c2·S`，即 policy loss + value function loss（critic 的 MSE）+ entropy bonus（鼓励探索）。critic 给的 `V(s)` 通过 **GAE（第 3 章已铺垫）** 算出 advantage `A_t`，`λ` 在 bias-variance 轴上滑动，典型 `γ=0.99, λ=0.95`。
- **PPO 是「实现细节驱动」的算法**：论文给的式子很短，但能不能跑出论文里的分数，取决于一大堆"工程暗知识"——advantage normalization、value loss clipping、orthogonal init、Adam 的 `eps=1e-5`、learning rate annealing、reward/observation 归一化……CleanRL 和 ICLR blog《The 37 Implementation Details of PPO》专门把这些挖出来。**本章源码精读直接逐行拆 CleanRL 的 `ppo.py`，把每个细节标出来。**
- **⭐可运行 demo（本章重中之重）**：从零写一个最小 PPO（clip + GAE，纯 PyTorch + gymnasium），跑通 CartPole-v1，**预期看到 episodic return 从 ~20 涨到 500（封顶）**。代码与 CleanRL 源码逐段呼应，去掉了 wandb/tensorboard/CLI 等噪音，**~150 行单文件，设计为可运行，请在你的环境验证**。另配一个 GAE 单元测试 demo，验证 `λ=0`/`λ=1` 退化成 one-step TD / Monte Carlo。
- **PPO 是 RLHF 的事实标准算法（第 5 章主角）**：InstructGPT / ChatGPT 的 RLHF 第三阶段就是 PPO。把 RL 框架映射到语言模型：**state = prompt + 已生成的 token，action = 下一个 token，reward = reward model 打分 − β·KL(policy‖SFT)**。这里出现了**两种不同的 KL**：(1) 作为 reward shaping 项的「policy vs reference」per-token KL（防止跑偏 SFT），(2) PPO 自身用于监控/early-stop 的「新策略 vs 旧策略」approx KL。两者极易混淆，§6 专门拆开讲，并标 fact / 待核。

## §1 设计考古：从 policy gradient 的痛点到 PPO

### 1.1 动机：on-policy PG 的两个结构性痛点

第 3 章我们推出了 Actor-Critic 的更新：用 advantage `A_t` 加权 `∇ ln π_θ(a_t|s_t)` 做梯度上升。理论很漂亮，但真拿去训练，会撞上两堵墙：

**痛点一：样本效率极低（数据用一次就废）。**
policy gradient 是 **on-policy** 的——梯度公式 `E_{τ~π_θ}[...]` 要求数据由**当前**策略 `π_θ` 采集。你跑一批 rollout，走**一步**梯度，`θ` 就变了，刚才那批数据严格来说已经不再来自当前策略，再用就是有偏的。所以朴素 PG 是「采一批、用一次、扔掉」。在仿真便宜的玩具任务上还能忍，在 RLHF 这种「采样 = 大模型 generate，贵到肉疼」的场景里，这是灾难。

**痛点二：步长敏感，容易崩（policy collapse）。**
PG 的目标曲面对参数高度非线性。学习率稍大、或某个 batch 的 advantage 估计有偏，一步更新就可能把策略推到一个**糟糕且难以恢复**的区域——因为新策略一旦变差，它采的数据也跟着变差，形成恶性循环。监督学习里「一步走坏了下个 batch 还能拉回来」的容错，在 RL 里不成立（数据分布是策略自己造的）。

这两个痛点其实指向同一个解法：**如果我能限制「新策略别离旧策略太远」，那么 (a) 在这个信任范围内，用旧策略采的数据估新策略的目标就近似有效，于是可以对同一批数据多走几步（样本效率↑）；(b) 步子被限制住，也就不会一步崩（稳定性↑）。** 这就是 trust region 思想。

### 1.2 前身：TRPO 用硬约束 + 二阶优化实现 trust region

**TRPO（Trust Region Policy Optimization, Schulman et al. 2015）** 是第一个把 trust region 落地的算法。它的优化问题是：

```
maximize_θ   E_t[ (π_θ(a_t|s_t) / π_θ_old(a_t|s_t)) · A_t ]      ← surrogate objective（importance-weighted advantage）
subject to   E_t[ KL( π_θ_old(·|s_t) ‖ π_θ(·|s_t) ) ] ≤ δ        ← 硬性 trust region 约束
```

直觉：**在「新旧策略平均 KL 散度不超过 δ」的信任区域里，最大化 importance-weighted 的 advantage**。这个 surrogate 是真实目标的一个一阶近似，理论上能保证（在约束内）单调改进。

TRPO 怎么解这个带约束优化？用**二阶方法**：对目标做线性近似、对 KL 约束做二阶（Fisher 信息矩阵）近似，再用 conjugate gradient + Hessian-vector product 求解，最后 line search 确保满足约束。

> 【设计考古，WebFetch 核实：Wikipedia "Proximal policy optimization"】 原文："TRPO uses the Hessian matrix (a matrix of second derivatives) to enforce the trust region, but the Hessian is inefficient for large-scale problems."

TRPO 的问题不在理论，在工程：
1. **二阶优化复杂**。要算 Fisher-vector product，要 conjugate gradient，实现门槛高、容易出 bug。
2. **和现代深度学习组件不兼容**。参数共享（actor/critic 共享 backbone）、dropout、batch norm 这些会破坏 KL 的 Fisher 近似假设。
3. **不能简单地"多走几个 epoch"**：约束优化每次解一个子问题，不像 SGD 那样自然支持 minibatch 多 epoch。

### 1.3 PPO 的发明：把 trust region 压进一阶 clip（核心思想）

PPO 的全部洞察是：**与其用一个硬约束逼着优化器待在信任区域里，不如改造目标函数本身，让"离旧策略太远"这件事在目标里直接不划算。** 这样就退回到普通的一阶 SGD，什么二阶、什么约束优化，全不要了。

> 【设计考古，WebFetch 核实：arxiv.org/abs/1707.06347 摘要】作者 John Schulman, Filip Wolski, Prafulla Dhariwal, Alec Radford, Oleg Klimov（OpenAI），2017。摘要原文：PPO 方法 "alternate between sampling data through interaction with the environment, and optimizing a 'surrogate' objective function using stochastic gradient ascent"，并称其拥有 "some of the benefits of trust region policy optimization (TRPO), but they are much simpler to implement, more general, and have better sample complexity (empirically)."

记概率比（probability ratio，本质是 importance weight）：

```
r_t(θ) = π_θ(a_t | s_t) / π_θ_old(a_t | s_t)
```

注意 `r_t(θ_old) = 1`。如果直接最大化 TRPO 的 surrogate `E_t[r_t·A_t]`，没有约束，优化器会把 `r_t` 推到天上去（对 `A>0` 的动作无限加大概率）——这正是会崩的那个目标。PPO 的 **clipped surrogate objective** 给它加了个"天花板"：

```
L^CLIP(θ) = E_t[ min( r_t(θ)·A_t ,  clip(r_t(θ), 1−ε, 1+ε)·A_t ) ]
```

> 【真实公式，WebFetch 核实：PPO 论文 Eq.(7) + OpenAI Spinning Up PPO 文档 + Wikipedia 三处一致】推荐 `ε=0.2`。

**逐项拆解这个式子的几何（关键，务必看懂）：**

把 `clip(r, 1−ε, 1+ε)` 理解成「把 `r` 夹在 `[1−ε, 1+ε]` 区间内」。Spinning Up 给了一个等价的分情况写法，最直观：

```
当 A_t ≥ 0（这个动作是好动作，我们想加大它的概率）：
    L = min( r·A ,  (1+ε)·A )
    → r 在 [0, 1+ε] 内时，L = r·A，正常奖励你加大概率；
    → r 一旦超过 1+ε，L 被钉死在 (1+ε)·A，再加大 r 没有任何额外收益（梯度归零）。
    直觉：好动作可以加强，但加强到 1+ε 就够了，别贪。

当 A_t < 0（这个动作是坏动作，我们想减小它的概率）：
    L = min( r·A ,  (1−ε)·A )      ← 注意 A<0，min 选的是更负（更小）的那个
    → r 在 [1−ε, +∞) 内时…… 实际起作用的是：r 降到 1−ε 以下时，L 被钉死在 (1−ε)·A；
    → 即坏动作可以削弱，但削弱到 1−ε 就够了，别把它一脚踩到地底（防止过度更新）。
```

> 【设计考古，WebFetch 核实：OpenAI Spinning Up PPO 文档】原文给出 `g(ε,A) = (1+ε)A` (当 A≥0) 与 `g(ε,A) = (1−ε)A` (当 A<0)，并指出 clipping 的作用是 "removing incentives for the policy to change dramatically"，`ε` 表示 "how far away the new policy can go from the old while still profiting."

**为什么是 `min`（取较小者）而不是直接 clip？** 这是 PPO 最精妙的一点。如果只写 `clip(r,1−ε,1+ε)·A`，会丢掉一个重要的安全性质。`min` 让这个目标成为真实 surrogate 的一个**悲观下界（lower bound / pessimistic estimate）**：
- 当更新让目标变好（在信任区内）时，clip 限制其上界，防止单步过猛；
- 当更新让目标变坏（`r` 往错误方向跑很远）时，`min` 选择了那个**没有被 clip 保护的、更差的项**，于是这个糟糕的更新会被如实计入 loss 并被惩罚——也就是说，clip 只在"对我们有利的方向"上设天花板，绝不在"对我们不利的方向"上给保护伞。

> 一句话记忆：**clip 给收益封顶，min 让损失穿透**。这保证了 PPO 既不会因贪心而过度更新，也不会因 clip 而对真正变坏的更新视而不见。

### 1.4 PPO 的另一个变体：adaptive KL penalty（次要，了解即可）

论文其实给了两个 PPO 变体。除了上面的 clip 版（PPO-Clip，实践中的绝对主流），还有 **PPO-Penalty（adaptive KL penalty）**：

```
L^KLPEN(θ) = E_t[ r_t·A_t − β · KL(π_θ_old ‖ π_θ) ]
```

把 KL 从 TRPO 的硬约束改成软惩罚项 `−β·KL`，并**自适应调 β**：每次更新后测一下实际 KL `d`，若 `d > 1.5·d_targ` 就 `β←2β`（惩罚加重），若 `d < d_targ/1.5` 就 `β←β/2`（放松）。

> 【设计考古，WebFetch 核实：PPO 论文 §4 + Wikipedia】论文实验结论：**clip 版总体优于 penalty 版**，所以工业界默认 PPO-Clip。但请记住这个 `−β·KL` 软惩罚的形式——**RLHF 里的 KL penalty（§6）用的正是这个思想，只不过那里的 KL 是 policy vs reference（SFT），不是 vs old policy。** 这是两件事，§6 会反复强调。

### 1.5 完整目标：clip + value + entropy

PPO 在实践中通常 actor 和 critic 共享网络（或至少同时训练），所以 loss 是三项加权：

```
L_t(θ) = E_t[ L^CLIP_t(θ) − c1·L^VF_t(θ) + c2·S[π_θ](s_t) ]
```

- `L^CLIP`：上面的 clipped policy objective（要**最大化**，所以在代码里取负做最小化）。
- `L^VF = (V_θ(s_t) − V_target_t)²`：critic 的 value function 回归损失（要**最小化**，所以前面是 `−c1`，因为整体是最大化形式）。`V_target` 一般用 `returns = advantages + V_old`。
- `S[π_θ]`：policy 的 entropy，**最大化**它鼓励探索、防止过早收敛到确定性策略。

> 【真实公式，WebFetch 核实：PPO 论文 Eq.(9)】论文给的典型系数 `c2=0.01`（entropy）；`c1` 论文写 1，但在 Atari 等共享网络设置里常用，CleanRL 的独立网络实现用 `vf_coef=0.5`（见 §2）。这个系数差异本身就是"实现细节"的一个例子。

### 1.6 PPO-Clip 算法骨架（Algorithm 1）

> 【设计考古，WebFetch 核实：PPO 论文 Algorithm 1 + Spinning Up 伪代码】

```
for iteration = 1, 2, ... do
    for actor = 1, 2, ..., N do                          # N 个并行环境
        run policy π_θ_old in environment for T timesteps  # 采集 rollout
        compute advantage estimates A_1, ..., A_T          # 用 GAE（需要 critic V）
    end for
    # 现在手上有 N·T 个 (s,a,r,A,V_old,logp_old) 样本
    optimize surrogate L wrt θ, with K epochs and minibatch size M ≤ N·T
    θ_old ← θ                                            # 更新 old policy 为最新
end for
```

注意三个关键设计，全部对应 §1.1 的两个痛点：
1. **N 个并行 actor**：一次采 `N·T` 个样本，降低梯度方差（第 3 章方差问题），也提高吞吐。
2. **K 个 epoch 重复利用同一批数据**：这就是 PPO 比朴素 PG 样本效率高的来源——因为有 clip 兜底，同一批数据可以安全地走 K 步（典型 K=4 ~ 10）。
3. **`θ_old ← θ` 在每个 iteration 末更新**：在 K 个 epoch 内 `π_θ_old` 固定（用来算 `r_t` 的分母和采数据），iteration 之间才同步。

### 2.1 超参数（Args dataclass）—— 数字本身就是知识

派生量（运行时算）：

```python

args.batch_size = int(args.num_envs * args.num_steps)        # 4*128 = 512
```

### 2.2 网络结构：orthogonal init 是实现细节

```python
# 【真实源码 vwxyzjn/cleanrl@cleanrl/ppo.py】
def layer_init(layer, std=np.sqrt(2), bias_const=0.0):
    torch.nn.init.orthogonal_(layer.weight, std)        # 正交初始化
    torch.nn.init.constant_(layer.bias, bias_const)
    return layer

class Agent(nn.Module):
    def __init__(self, envs):
        super().__init__()
        self.critic = nn.Sequential(
            layer_init(nn.Linear(np.array(envs.single_observation_space.shape).prod(), 64)),
            nn.Tanh(),
            layer_init(nn.Linear(64, 64)),
            nn.Tanh(),
            layer_init(nn.Linear(64, 1), std=1.0),        # critic 输出层 std=1.0
        )
        self.actor = nn.Sequential(
            layer_init(nn.Linear(np.array(envs.single_observation_space.shape).prod(), 64)),
            nn.Tanh(),
            layer_init(nn.Linear(64, 64)),
            nn.Tanh(),
            layer_init(nn.Linear(64, envs.single_action_space.n), std=0.01),  # actor 输出层 std=0.01
        )

    def get_value(self, x):
        return self.critic(x)

    def get_action_and_value(self, x, action=None):
        logits = self.actor(x)
        probs = Categorical(logits=logits)                # 离散动作：类别分布
        if action is None:
            action = probs.sample()                       # rollout 时采样
        return action, probs.log_prob(action), probs.entropy(), self.critic(x)
```

**逐行注解里的实现细节：**
- **actor / critic 是两个独立 MLP**（不共享 backbone）。这是 CleanRL 在连续/离散控制上的选择；RLHF 里因为模型巨大，actor 和 critic 往往共享 transformer 主干、各挂一个 head（§6）。
- **orthogonal init + 特定 std**：hidden 层用 `std=√2`（配 ReLU/Tanh 的经验值），**critic 输出层 `std=1.0`，actor 输出层 `std=0.01`**。actor 输出层用极小的 0.01，是为了让初始策略接近均匀分布（logits 接近 0 → 各动作概率接近均等），避免一开始就过度自信某个动作。这是被反复验证过的实现细节，去掉它会明显伤性能。
- `get_action_and_value` 一次返回 `(action, log_prob, entropy, value)`——这正是 PPO 每一步需要的全部量。`action=None` 时采样（rollout），传入 `action` 时复算它的 log_prob（update 阶段算新策略下旧动作的概率，即 `r_t` 的分子）。

### 2.3 Rollout + GAE：advantage 怎么算出来

**第一步，采集 rollout**（把 `T×N` 个样本存进 buffer）：

```python
# 【真实源码 vwxyzjn/cleanrl@cleanrl/ppo.py】（storage 与 rollout 主体）
obs = torch.zeros((args.num_steps, args.num_envs) + envs.single_observation_space.shape).to(device)
actions = torch.zeros((args.num_steps, args.num_envs) + envs.single_action_space.shape).to(device)
logprobs = torch.zeros((args.num_steps, args.num_envs)).to(device)
rewards = torch.zeros((args.num_steps, args.num_envs)).to(device)
dones = torch.zeros((args.num_steps, args.num_envs)).to(device)
values = torch.zeros((args.num_steps, args.num_envs)).to(device)
...
for step in range(0, args.num_steps):
    global_step += args.num_envs
    obs[step] = next_obs
    dones[step] = next_done
    with torch.no_grad():                                  # rollout 不需要梯度
        action, logprob, _, value = agent.get_action_and_value(next_obs)
        values[step] = value.flatten()
    actions[step] = action
    logprobs[step] = logprob                               # 存下 logp_old（关键：算 r_t 用）
    next_obs, reward, terminations, truncations, infos = envs.step(action.cpu().numpy())
    next_done = np.logical_or(terminations, truncations)
    rewards[step] = torch.tensor(reward).to(device).view(-1)
    next_obs, next_done = torch.Tensor(next_obs).to(device), torch.Tensor(next_done).to(device)
```

注意 `with torch.no_grad()` 包住整个采样——rollout 阶段只是和环境交互、记录数据，**不建图、不回传**。`logprobs[step]` 存的就是 `log π_θ_old(a|s)`，是后面计算概率比 `r_t` 的分母（取 exp 前）。

**第二步，GAE 计算 advantage**（第 3 章铺垫的 `λ` 加权 TD，这里是工程实现）：

```python
# 【真实源码 vwxyzjn/cleanrl@cleanrl/ppo.py】（bootstrap + GAE 反向递推）
with torch.no_grad():
    next_value = agent.get_value(next_obs).reshape(1, -1)   # 末状态 bootstrap value
    advantages = torch.zeros_like(rewards).to(device)
    lastgaelam = 0
    for t in reversed(range(args.num_steps)):               # 从后往前递推
        if t == args.num_steps - 1:
            nextnonterminal = 1.0 - next_done
            nextvalues = next_value
        else:
            nextnonterminal = 1.0 - dones[t + 1]
            nextvalues = values[t + 1]
        delta = rewards[t] + args.gamma * nextvalues * nextnonterminal - values[t]
        advantages[t] = lastgaelam = delta + args.gamma * args.gae_lambda * nextnonterminal * lastgaelam
    returns = advantages + values                          # value target = A + V_old
```

**这八行是 GAE 的全部，逐行拆：**
- `delta = r_t + γ·V(s_{t+1})·nonterminal − V(s_t)`：这就是 TD residual `δ_t`（第 2 章）。`nextnonterminal = 1 − done` 是关键——**episode 结束时 `V(s_{t+1})` 不该被 bootstrap**（终止状态后没有未来），所以乘 0 切断。
- `advantages[t] = δ_t + (γλ)·nonterminal·lastgaelam`：这是 GAE 的递推形式 `A_t = δ_t + γλ·A_{t+1}`，等价于第 3 章的 `A_t = Σ_l (γλ)^l δ_{t+l}`。从后往前算，`lastgaelam` 累积着 `A_{t+1}`。
- `returns = advantages + values`：value 网络的回归目标。等价于 `λ-return`，比纯 Monte Carlo return 方差小。

> **实现细节警示**：`nextnonterminal` 在每个 episode 边界切断 bootstrap，是 GAE 正确性的命门。漏掉它，advantage 会把跨 episode 的奖励错误地连起来，训练直接发散。这是新手最常踩的坑之一。

### 2.4 PPO update：clip loss 的真实代码（本章最核心代码）

先 flatten，再多 epoch + minibatch 循环：

```python
# 【真实源码 vwxyzjn/cleanrl@cleanrl/ppo.py】（flatten batch）
b_obs = obs.reshape((-1,) + envs.single_observation_space.shape)
b_logprobs = logprobs.reshape(-1)         # logp_old
b_actions = actions.reshape((-1,) + envs.single_action_space.shape)
b_advantages = advantages.reshape(-1)
b_returns = returns.reshape(-1)
b_values = values.reshape(-1)             # V_old
```

```python
# 【真实源码 vwxyzjn/cleanrl@cleanrl/ppo.py】（K-epoch minibatch update，核心）
b_inds = np.arange(args.batch_size)
clipfracs = []
for epoch in range(args.update_epochs):           # K 个 epoch 复用同一批数据
    np.random.shuffle(b_inds)
    for start in range(0, args.batch_size, args.minibatch_size):
        end = start + args.minibatch_size
        mb_inds = b_inds[start:end]

        # 用当前 θ 复算「旧动作」的新 log_prob / entropy / value
        _, newlogprob, entropy, newvalue = agent.get_action_and_value(b_obs[mb_inds], b_actions.long()[mb_inds])
        logratio = newlogprob - b_logprobs[mb_inds]   # log r_t = log π_θ - log π_old
        ratio = logratio.exp()                        # r_t = π_θ / π_old

        with torch.no_grad():
            # approx KL，仅监控/early-stop 用，不回传梯度
            old_approx_kl = (-logratio).mean()                 # k1 估计：-log r
            approx_kl = ((ratio - 1) - logratio).mean()        # k3 估计：(r-1)-log r
            clipfracs += [((ratio - 1.0).abs() > args.clip_coef).float().mean().item()]

        mb_advantages = b_advantages[mb_inds]
        if args.norm_adv:                              # advantage 归一化（实现细节）
            mb_advantages = (mb_advantages - mb_advantages.mean()) / (mb_advantages.std() + 1e-8)

        # === Policy loss：clipped surrogate（§1.3 的式子落地）===
        pg_loss1 = -mb_advantages * ratio
        pg_loss2 = -mb_advantages * torch.clamp(ratio, 1 - args.clip_coef, 1 + args.clip_coef)
        pg_loss = torch.max(pg_loss1, pg_loss2).mean()

        # === Value loss：可选 clip ===
        newvalue = newvalue.view(-1)
        if args.clip_vloss:
            v_loss_unclipped = (newvalue - b_returns[mb_inds]) ** 2
            v_clipped = b_values[mb_inds] + torch.clamp(
                newvalue - b_values[mb_inds], -args.clip_coef, args.clip_coef,
            )
            v_loss_clipped = (v_clipped - b_returns[mb_inds]) ** 2
            v_loss_max = torch.max(v_loss_unclipped, v_loss_clipped)
            v_loss = 0.5 * v_loss_max.mean()
        else:
            v_loss = 0.5 * ((newvalue - b_returns[mb_inds]) ** 2).mean()

        # === 三项合并 ===
        entropy_loss = entropy.mean()
        loss = pg_loss - args.ent_coef * entropy_loss + v_loss * args.vf_coef

        optimizer.zero_grad()
        loss.backward()
        nn.utils.clip_grad_norm_(agent.parameters(), args.max_grad_norm)  # 梯度裁剪
        optimizer.step()

    if args.target_kl is not None and approx_kl > args.target_kl:
        break                                          # KL 超阈值就 early-stop
```

**最关键的 policy loss 三行，对照 §1.3 的式子逐行解码：**
```python
pg_loss1 = -mb_advantages * ratio                                          # -(r·A)
pg_loss2 = -mb_advantages * torch.clamp(ratio, 1-clip, 1+clip)             # -(clip(r)·A)
pg_loss = torch.max(pg_loss1, pg_loss2).mean()                             # -min(...) = max(-...)
```
注意符号：论文是**最大化** `min(r·A, clip(r)·A)`，代码里要**最小化 loss**，所以两项都加负号，`min` 就变成 `max`。`torch.max(pg_loss1, pg_loss2)` 完全等价于 `−min(r·A, clip·A)`。**这就是 §1.3 那个"clip 封顶、min 穿透"目标的逐字落地。**

**几个必须标出来的实现细节（论文不写、但决定成败）：**

1. **approx KL 用 k3 估计**：`approx_kl = ((ratio-1) - logratio).mean()`。这不是教科书的 `KL = Σ p·log(p/q)`，而是 John Schulman 的低方差无偏估计 k3。

   > 【设计考古，WebFetch 核实：joschu.net/blog/kl-approx.html + 代码注释 `http://joschu.net/blog/kl-approx.html`】三个估计：`k1 = -log r`（无偏但高方差，因为对一半样本为负，而真 KL 恒正）；`k2 = ½(log r)²`（低方差但有偏）；`k3 = (r−1) − log r`（无偏 **且** 低方差，恒非负，用了控制变量技巧：在无偏的 k1 上加一个期望为 0 的项 `(r−1)`，`E_q[r−1] = E_q[p/q] − 1 = 0`）。代码里 `old_approx_kl` 是 k1、`approx_kl` 是 k3，前者仅作对照日志。**记住 k3，第 5/6 章 RLHF 的 KL penalty 实现会再用到。**

2. **advantage normalization**（`norm_adv`）：每个 minibatch 内把 advantage 标准化成均值 0、方差 1。这显著稳定训练（让 policy loss 的尺度与 lr 解耦），是 PPO 最重要的实现细节之一。注意是 **per-minibatch** 归一化。

3. **value loss clipping**（`clip_vloss`）：critic 的更新也被 clip 在 `V_old ± ε` 内，防止 value 一步跳太远。`torch.max(unclipped, clipped)` 同样取悲观（更大）的那个。**有意思的是**，《37 details》的实验显示这个细节其实**对性能帮助不明显甚至有时有害**，但论文用了、CleanRL 默认开着——这是一个"沿袭下来但收益存疑"的细节，知道即可。

4. **梯度裁剪** `clip_grad_norm_(..., 0.5)`：把所有参数的梯度总范数裁到 ≤ 0.5，防止偶发大梯度炸掉训练。

5. **early stopping by KL**（`target_kl`）：如果新旧策略的 approx KL 超过阈值，提前结束这次 iteration 的剩余 epoch——这是把 trust region 思想"再补一刀"的保险。CleanRL 默认 `target_kl=None`（不启用），但 RLHF/很多实践会开。

### 2.5 收尾：explained variance + lr annealing

```python
# 【真实源码 vwxyzjn/cleanrl@cleanrl/ppo.py】（诊断指标 + lr 退火）
y_pred, y_true = b_values.cpu().numpy(), b_returns.cpu().numpy()
var_y = np.var(y_true)
explained_var = np.nan if var_y == 0 else 1 - np.var(y_true - y_pred) / var_y
...
# iteration 开头：
if args.anneal_lr:
    frac = 1.0 - (iteration - 1.0) / args.num_iterations
    lrnow = frac * args.learning_rate
    optimizer.param_groups[0]["lr"] = lrnow
```

- **`explained_variance`**：`1 − Var(returns − values)/Var(returns)`。这是诊断 critic 好坏的**头号指标**。接近 1 说明 critic 拟合得好；若长期接近 0 或为负，说明 value 网络根本没学到东西、advantage 估计不可信——PPO 排障第一个该看的就是它。
- **learning rate annealing**：lr 从初值线性退火到 0。又一个能涨分的实现细节。

> 小结这一节：PPO 论文的 loss 式子三行就能写完，但 CleanRL 里和它绑在一起的 ~7 个实现细节（orthogonal init、adv norm、value clip、grad clip、lr anneal、k3 approx kl、bootstrap mask）才是"能跑出论文分数"的真正门槛。下一节的 demo 会保留其中最关键的几个，去掉次要的，让你看清主干。

### 3.2 完整代码（ppo_cartpole_min.py）

```python
# ppo_cartpole_min.py
# 最小 PPO（clip + GAE）解 CartPole-v1。【示意，非逐字】，设计为可运行，请在你的环境验证。
# 与 vwxyzjn/cleanrl@cleanrl/ppo.py 逐段呼应；为教学做了单环境简化。
import numpy as np
import torch
import torch.nn as nn
import torch.optim as optim
import gymnasium as gym
from torch.distributions.categorical import Categorical

# ---------- 超参数（对应 CleanRL 的 Args）----------
ENV_ID        = "CartPole-v1"
TOTAL_STEPS   = 80_000     # 总交互步数（demo 用，约几分钟 CPU）
NUM_STEPS     = 512        # 每次 rollout 的步数 T（单环境）
GAMMA         = 0.99
GAE_LAMBDA    = 0.95
CLIP_COEF     = 0.2        # ε
UPDATE_EPOCHS = 4          # K
NUM_MINIBATCH = 4
ENT_COEF      = 0.01
VF_COEF       = 0.5
MAX_GRAD_NORM = 0.5
LR            = 2.5e-4
SEED          = 1

torch.manual_seed(SEED); np.random.seed(SEED)
device = torch.device("cpu")

def layer_init(layer, std=np.sqrt(2), bias_const=0.0):
    nn.init.orthogonal_(layer.weight, std)        # 正交初始化（实现细节，§2.2）
    nn.init.constant_(layer.bias, bias_const)
    return layer

class Agent(nn.Module):
    def __init__(self, obs_dim, act_n):
        super().__init__()
        self.critic = nn.Sequential(
            layer_init(nn.Linear(obs_dim, 64)), nn.Tanh(),
            layer_init(nn.Linear(64, 64)),      nn.Tanh(),
            layer_init(nn.Linear(64, 1), std=1.0),
        )
        self.actor = nn.Sequential(
            layer_init(nn.Linear(obs_dim, 64)), nn.Tanh(),
            layer_init(nn.Linear(64, 64)),      nn.Tanh(),
            layer_init(nn.Linear(64, act_n), std=0.01),   # actor 输出层小 std
        )
    def get_value(self, x):
        return self.critic(x)
    def get_action_and_value(self, x, action=None):
        logits = self.actor(x)
        probs = Categorical(logits=logits)
        if action is None:
            action = probs.sample()
        return action, probs.log_prob(action), probs.entropy(), self.critic(x)

env = gym.make(ENV_ID)
obs_dim = int(np.prod(env.observation_space.shape))
act_n   = env.action_space.n
agent = Agent(obs_dim, act_n).to(device)
optimizer = optim.Adam(agent.parameters(), lr=LR, eps=1e-5)   # eps=1e-5 是实现细节

batch_size    = NUM_STEPS
minibatch_size = batch_size // NUM_MINIBATCH
num_iterations = TOTAL_STEPS // batch_size

# rollout buffer（单环境，形状 [T] 或 [T, obs_dim]）
obs_buf      = torch.zeros((NUM_STEPS, obs_dim))
actions_buf  = torch.zeros(NUM_STEPS)
logprobs_buf = torch.zeros(NUM_STEPS)
rewards_buf  = torch.zeros(NUM_STEPS)
dones_buf    = torch.zeros(NUM_STEPS)
values_buf   = torch.zeros(NUM_STEPS)

global_step = 0
next_obs, _ = env.reset(seed=SEED)
next_obs = torch.tensor(next_obs, dtype=torch.float32)
next_done = torch.zeros(1)
ep_returns = []   # 记录每个 episode 的 return，用来看收敛

for iteration in range(1, num_iterations + 1):
    # lr 线性退火（实现细节，§2.5）
    frac = 1.0 - (iteration - 1.0) / num_iterations
    optimizer.param_groups[0]["lr"] = frac * LR

    # ---------- rollout：采 T 步 ----------
    ep_ret = 0.0
    for step in range(NUM_STEPS):
        global_step += 1
        obs_buf[step] = next_obs
        dones_buf[step] = next_done
        with torch.no_grad():
            action, logprob, _, value = agent.get_action_and_value(next_obs.unsqueeze(0))
            values_buf[step] = value.flatten()
        actions_buf[step] = action
        logprobs_buf[step] = logprob
        next_obs_np, reward, term, trunc, _ = env.step(int(action.item()))
        done = term or trunc
        rewards_buf[step] = float(reward)
        ep_ret += float(reward)
        next_obs = torch.tensor(next_obs_np, dtype=torch.float32)
        next_done = torch.tensor([float(done)])
        if done:
            ep_returns.append(ep_ret); ep_ret = 0.0
            next_obs_np, _ = env.reset()
            next_obs = torch.tensor(next_obs_np, dtype=torch.float32)

    # ---------- GAE：反向递推算 advantage（§2.3）----------
    with torch.no_grad():
        next_value = agent.get_value(next_obs.unsqueeze(0)).reshape(-1)
        advantages = torch.zeros_like(rewards_buf)
        lastgaelam = 0.0
        for t in reversed(range(NUM_STEPS)):
            if t == NUM_STEPS - 1:
                nextnonterminal = 1.0 - next_done
                nextvalues = next_value
            else:
                nextnonterminal = 1.0 - dones_buf[t + 1]
                nextvalues = values_buf[t + 1]
            delta = rewards_buf[t] + GAMMA * nextvalues * nextnonterminal - values_buf[t]
            advantages[t] = lastgaelam = delta + GAMMA * GAE_LAMBDA * nextnonterminal * lastgaelam
        returns = advantages + values_buf

    # ---------- flatten ----------
    b_obs       = obs_buf
    b_logprobs  = logprobs_buf
    b_actions   = actions_buf.long()
    b_advantages = advantages
    b_returns   = returns
    b_values    = values_buf

    # ---------- PPO update：K epoch × minibatch（§2.4）----------
    b_inds = np.arange(batch_size)
    for epoch in range(UPDATE_EPOCHS):
        np.random.shuffle(b_inds)
        for start in range(0, batch_size, minibatch_size):
            mb_inds = b_inds[start:start + minibatch_size]
            _, newlogprob, entropy, newvalue = agent.get_action_and_value(
                b_obs[mb_inds], b_actions[mb_inds])
            logratio = newlogprob - b_logprobs[mb_inds]
            ratio = logratio.exp()                          # r_t

            mb_adv = b_advantages[mb_inds]
            mb_adv = (mb_adv - mb_adv.mean()) / (mb_adv.std() + 1e-8)   # adv 归一化

            # policy loss：clipped surrogate
            pg_loss1 = -mb_adv * ratio
            pg_loss2 = -mb_adv * torch.clamp(ratio, 1 - CLIP_COEF, 1 + CLIP_COEF)
            pg_loss = torch.max(pg_loss1, pg_loss2).mean()

            # value loss：带 clip
            newvalue = newvalue.view(-1)
            v_unclipped = (newvalue - b_returns[mb_inds]) ** 2
            v_clipped = b_values[mb_inds] + torch.clamp(
                newvalue - b_values[mb_inds], -CLIP_COEF, CLIP_COEF)
            v_loss = 0.5 * torch.max(v_unclipped, (v_clipped - b_returns[mb_inds]) ** 2).mean()

            entropy_loss = entropy.mean()
            loss = pg_loss - ENT_COEF * entropy_loss + VF_COEF * v_loss

            optimizer.zero_grad()
            loss.backward()
            nn.utils.clip_grad_norm_(agent.parameters(), MAX_GRAD_NORM)
            optimizer.step()

    # ---------- 日志：最近 episode 平均 return ----------
    if len(ep_returns) > 0:
        recent = np.mean(ep_returns[-20:])
        with torch.no_grad():
            approx_kl = ((ratio - 1) - logratio).mean().item()   # k3
        print(f"iter {iteration:3d} | step {global_step:6d} | "
              f"avg_return(last20) {recent:6.1f} | approx_kl {approx_kl:+.4f}")

env.close()
print("done. CartPole 解决标准：连续多回合 return 接近 500（封顶）。")
```

### 3.5 配套 demo：GAE 单元测试（验证 λ 两个极端）

> 验证 §2.3 的 GAE 在 `λ=0` 退化成 one-step TD、`λ=1` 退化成 Monte Carlo。只需 `numpy`。**设计为可运行，请在你的环境验证。**

```python
# gae_unit_test.py  —— 验证 GAE 的两个边界。只需 numpy。
import numpy as np

def gae(rewards, values, next_value, dones, gamma, lam):
    T = len(rewards)
    adv = np.zeros(T); last = 0.0
    for t in reversed(range(T)):
        nextnonterminal = 1.0 - (dones[t+1] if t+1 < T else 0.0)
        nextv = next_value if t == T-1 else values[t+1]
        delta = rewards[t] + gamma * nextv * nextnonterminal - values[t]
        adv[t] = last = delta + gamma * lam * nextnonterminal * last
    return adv

rewards = np.array([1.0, 1.0, 1.0, 1.0])
values  = np.array([0.5, 0.6, 0.7, 0.8])
next_value = 0.0
dones   = np.array([0.0, 0.0, 0.0, 0.0])   # 无中途终止
gamma   = 0.99

# λ=0：A_t 应等于 one-step TD residual δ_t = r_t + γV(s_{t+1}) - V(s_t)
adv0 = gae(rewards, values, next_value, dones, gamma, lam=0.0)
delta = np.array([rewards[t] + gamma*(values[t+1] if t+1<4 else next_value) - values[t]
                  for t in range(4)])
assert np.allclose(adv0, delta), (adv0, delta)
print("λ=0  OK: A_t == one-step TD δ_t =", np.round(adv0, 4))

# λ=1：A_t 应等于 Monte Carlo return - V(s_t) = Σ γ^k r - V(s_t)
adv1 = gae(rewards, values, next_value, dones, gamma, lam=1.0)
mc = np.zeros(4)
for t in range(4):
    g = 0.0
    for k, tk in enumerate(range(t, 4)):
        g += (gamma**k) * rewards[tk]
    mc[t] = g - values[t]      # 末尾 next_value=0，无 bootstrap
assert np.allclose(adv1, mc), (adv1, mc)
print("λ=1  OK: A_t == MC return - V(s_t) =", np.round(adv1, 4))
print("两个极端都通过 —— GAE 的 λ 确实在 TD(0) 与 MC 之间插值。")
```

预期输出：

```
λ=0  OK: A_t == one-step TD δ_t = [1.094 1.093 1.092 0.2  ]
λ=1  OK: A_t == MC return - V(s_t) = [3.4404 2.3701 1.29   0.2   ]
两个极端都通过 —— GAE 的 λ 确实在 TD(0) 与 MC 之间插值。
```

> 上面的 λ=1 数字是本文档实际运行 `gae_unit_test.py` 抓取的真实输出（assert 通过）；λ=0 同理。这是本章唯一一段「真实运行抓取」的输出。

> 这个测试把第 3 章的 GAE 理论"钉死"成可执行断言：`λ=0` 只看一步（低方差高 bias），`λ=1` 看完整条轨迹（高方差无 bias），中间 `λ=0.95` 是 PPO 的默认折中。

## §4 方案对比

### 4.1 横向：PPO vs 同期/前后算法

| 算法 | 类别 | trust region 怎么做 | 优化阶数 | 样本复用 | 实现难度 | 典型场景 |
|---|---|---|---|---|---|---|
| **REINFORCE** | on-policy PG | 无 | 一阶 | 用一次 | 极低 | 教学；GRPO 等回归本源（第 3 章） |
| **A2C / A3C** | on-policy AC | 无（靠小 lr） | 一阶 | 用一次 | 低 | 简单控制、并行采样 |
| **TRPO** | on-policy AC | 硬 KL 约束 | **二阶**（Fisher+CG） | 单步约束优化 | **高** | 对稳定性要求极高、能接受复杂实现 |
| **PPO-Clip** | on-policy AC | **clip surrogate** | 一阶 | **K epoch 复用** | 中 | **绝大多数连续/离散控制 + RLHF（主力）** |
| **PPO-Penalty** | on-policy AC | 自适应 KL 软惩罚 | 一阶 | K epoch 复用 | 中 | 论文给的变体，实践少用（clip 更优） |
| **DDPG / TD3 / SAC** | **off-policy** AC | 不适用 | 一阶 | **replay buffer 大量复用** | 中-高 | 连续控制、样本极贵、追求样本效率 |

**几个对比要点（带具体场景）：**

- **PPO vs TRPO**：同样是 trust region，PPO 用一阶 clip 换掉二阶约束。代价是 PPO 的"信任区域"是**启发式**的（clip 不严格等价于 KL 约束），偶尔会有单步 KL 偏大；但实现简单到一个文件、能和任意网络结构配合、能多 epoch 复用数据。**结论：除非你有非用 TRPO 不可的理由（如论文复现），新项目直接 PPO。** 这也是为什么"Since 2018, PPO was the default RL algorithm at OpenAI"【WebFetch 核实：Wikipedia】。

- **PPO（on-policy）vs SAC/TD3（off-policy）**：这是**最重要的边界**。PPO 是 on-policy，数据"采一批用一批"（虽然能 K epoch 复用，但一个 iteration 后还是要扔），样本效率天然不如 off-policy。SAC 用 replay buffer，一条经验能被反复采样训练很多次，**样本效率高一个数量级**。
  - 场景 A：**仿真便宜、想要稳定 + 好调**（如大多数 MuJoCo 控制、游戏）→ PPO。它对超参不敏感、`ε=0.2` 几乎万用、几乎不会因为一个坏 batch 就崩。
  - 场景 B：**真实机器人 / 采样极贵 / 追求样本效率** → SAC 或 TD3。
  - 场景 C：**离散动作 + 大规模并行仿真**（Atari、Isaac Gym）→ PPO（并行 actor 把样本效率的劣势用吞吐补回来）。

- **PPO 用于 RLHF 的特殊性**：LLM 场景采样 = 大模型 generate，**极贵**，按理该用 off-policy。但 RLHF 选 PPO，是因为 (1) reward 是序列级稀疏信号，off-policy 的 Q 学习在超长动作序列（每个 token 是一个 action）上极难训稳；(2) PPO 的 clip + KL penalty 天然提供"别跑偏 SFT"的稳定性，这对对齐至关重要。代价就是贵——所以 2024 年后才有 GRPO 等"砍掉 critic、省一半显存"的简化（第 3 章 §6 / 本章 §6.3）。

### 4.2 纵向：PPO 内部的设计选择

| 选择 | 选项 A | 选项 B | 何时选 |
|---|---|---|---|
| trust region 实现 | clip | KL penalty | 默认 clip；KL 不稳时可叠加 early-stop |
| advantage 估计 | GAE(λ=0.95) | MC(λ=1) / TD(λ=0) | 默认 GAE 0.95；reward 极稀疏可调高 λ |
| actor/critic 网络 | 独立 | 共享 backbone | 小任务独立；大模型（RLHF）共享省显存 |
| value loss | clip | 不 clip | 论文用 clip，但《37 details》显示收益存疑，两者皆可 |
| adv 归一化 | per-minibatch | 关闭 | 默认开（强烈推荐） |

## §5 失败模式与真坑（PPO 为什么训练不稳）

PPO 比朴素 PG 稳得多，但绝非"开箱即用永不崩"。下面是真实工程里高频踩的坑，**每个都给现象 → 根因 → 排查**。

### 5.1 GAE 的 done mask 漏掉 → 训练直接发散

- **现象**：return 完全不涨，或 loss 出现 NaN/巨大值；多 episode 的环境尤其明显。
- **根因**：GAE 递推里 `nextnonterminal = 1 − done` 忘了乘，于是 advantage 把**上一个 episode 终止后**的 value 错误 bootstrap 进**下一个 episode** 的开头，advantage 估计彻底错乱。
- **排查**：单步打印 GAE：在 done=1 的那一步，`delta` 应只含 `rewards[t] − values[t]`（`nextvalues` 被乘 0）。用 §3.5 的单元测试加一个"中途 done"的 case 验证。

### 5.2 ratio 爆炸（val/ratio 跑到 2.0 / 1000 / 0.1）

- **现象**：`approx_kl` 突然飙升，`clipfrac` 接近 1，return 断崖下跌。
- **根因**：新旧策略一步差太远，`r_t = π_θ/π_old` 失控。常见诱因：lr 太大、`K`（update_epochs）太多导致同一批数据被过度拟合偏离 old policy、advantage 没归一化导致 loss 尺度过大、或某个 minibatch 的 advantage 有离群值。
- **排查**：
  - 监控 `val/ratio` 应**浮动在 1.0 附近**。TRL 文档明确说：`ratio` 太高（2.0/1000）或太低（0.1）意味着"updates between consecutive policies are too drastic"【WebFetch 核实：TRL PPO docs Cookbook】。
  - 先降 lr、再减小 `update_epochs`（4→2）、确认 adv 归一化开着。
  - 开启 `target_kl` early-stop（如 0.01~0.05）做硬保险。

### 5.3 critic 学不动（explained_variance ≈ 0 或负）

- **现象**：`explained_variance` 长期接近 0 甚至为负，policy 跟着学不好。
- **根因**：value 网络拟合失败 → advantage 估计是噪声 → policy gradient 方向不可信。可能是 `vf_coef` 太小、value target（returns）尺度异常（reward 没归一化）、或 critic 容量不足。
- **排查**：把 `explained_variance` 作为**头号诊断指标**盯着。若为负，先查 reward scale（过大的原始 reward 会让 value 回归极难），考虑 reward/return 归一化；再调 `vf_coef`。

### 5.4 entropy 过早塌缩 → 探索不足、卡局部最优

- **现象**：return 涨到一半就停滞，policy entropy 迅速趋近 0（策略过早变确定性）。
- **根因**：`ent_coef` 太小，policy 太快"自信"，停止探索。
- **排查**：监控 entropy 曲线，若掉得太快，调大 `ent_coef`（0.01→0.02+）。注意：actor 输出层的 `std=0.01` 初始化本身就是为了让初始 entropy 高一点（§2.2）。

### 5.5 "改了论文式子却跑不出论文分数" → 实现细节缺失

- **现象**：自己照着论文公式实现，能跑、不崩，但分数明显低于论文/CleanRL。
- **根因**：缺了那一堆实现细节——adv normalization、orthogonal init、lr annealing、Adam `eps=1e-5`、observation/reward 归一化、value clip……单个看都不起眼，**累加起来决定成败**。这正是《The 37 Implementation Details of PPO》整篇在讲的事。
- **排查**：拿 CleanRL `ppo.py` 当 ground truth 逐项 diff。**不要从论文伪代码起手写 PPO，要从 CleanRL 起手改。**

### 5.6 两种 KL 混淆（RLHF 专属，§6 详述）

- **现象**：RLHF 里 KL penalty 调不对，模型要么跑飞（胡说八道刷 reward），要么完全不动（被 reference 锁死）。
- **根因**：把"PPO 自身的 new-vs-old approx KL（监控/early-stop 用）"和"RLHF reward shaping 里的 policy-vs-reference KL（防跑偏 SFT 用）"搞混，调错了系数。
- **排查**：明确区分两者（§6.2），前者是算法内部稳定性、后者是对齐约束。

## §6 与 LLM 对齐的连接（第 5 章前导）

> 这是把 PPO 接到 RLHF 的桥。RLHF 的完整流程（SFT → reward model → PPO）是第 5 章主角，这里只把**「PPO 怎么映射到语言模型」**讲清，并把最容易混的两种 KL 拆开。涉及具体工业配置处标 fact / 待核。

### 6.1 RL 框架到语言模型的映射

把自回归生成看成一个 MDP【fact，业界标准建模】：

| RL 概念 | LLM 中的对应 |
|---|---|
| state `s_t` | prompt + 到目前为止已生成的 token 序列 |
| action `a_t` | 下一个 token（动作空间 = 整个词表，几万维离散） |
| policy `π_θ` | 被训练的语言模型（actor），给出 next-token 分布 |
| reward | **整条 response 生成完**后，reward model 给一个标量分；中间 token reward = 0（稀疏，序列级） |
| episode | 一次完整的 prompt→response 生成 |
| value `V(s)` | critic head 估"从当前 token 位置起，这条 response 最终能拿多少 reward" |

于是 PPO 的每个组件都有了对应：rollout = 用当前模型 generate 一批 response；advantage = 用 GAE 把序列级 reward 分摊到每个 token；clip surrogate = 限制更新后的模型 next-token 分布别离更新前差太远。

> **【真实源码佐证，WebFetch 核实：HuggingFace TRL PPOTrainer 文档】** TRL 的 PPO 涉及**四个模型**：`model`（policy/actor，被训）、`ref_model`（reference，通常是 SFT 模型的冻结副本，算 KL 用）、`reward_model`（打分）、`value_model`（critic，估 value）。这正是上表的工程落地。

### 6.2 两种 KL，必须分清（本节核心）

RLHF-PPO 里出现两个不同的 KL，混淆它们是 RLHF 调参翻车的头号原因：

**KL ①：policy vs reference，作为 reward shaping 项（防跑偏 SFT）。**
RLHF 不能让模型为了刷高 reward 而胡说八道（reward hacking），所以在 reward 里减去一个"离 SFT 模型多远"的惩罚：

```
reward_total(token) = (序列末尾的 reward model 打分)  −  β · KL( π_θ(·|s) ‖ π_ref(·|s) )
```

这是 **per-token** 的：每个 token 都减一点"偏离 reference 的代价"，把模型"拴"在 SFT 附近，保证它还说人话。这正是 §1.4 的 **PPO-Penalty 软惩罚思想**——但这里 KL 的对象是 **reference（SFT）**，不是 old policy。

> **【真实源码佐证，WebFetch 核实：TRL PPOTrainer 文档 logged metrics】** TRL 把它记为：`objective/non_score_reward = β · kl.sum(1)`（`β`=`kl_coef`，默认 0.05），`objective/rlhf_reward = score − non_score_reward`。`objective/kl` = "mean KL between current policy and reference policy"。**Debug TIP 原文**："`objective/rlhf_reward`: this is the ultimate objective of the RLHF training. If training works as intended, this metric should keep going up."

**KL ②：new policy vs old policy，PPO 自身的算法内部量（监控 + early-stop）。**
这就是 §2.4 那个 `approx_kl = ((ratio-1) − logratio).mean()`（k3 估计），衡量"这次 PPO 更新让策略动了多少"，用来监控 ratio 是否爆炸、是否该 early-stop。它和 reference 无关。

> **【真实源码佐证，WebFetch 核实：TRL docs】** TRL 明确区分：`objective/kl`（policy vs reference，KL ①）vs `policy/approxkl_avg`（"average approximate KL between **consecutive PPO policies**"，KL ②），文档特地注明"Note that this is not the same as objective/kl."

**一句话钉死区别**：
- **KL ①（vs reference, β·KL 进 reward）**：对齐约束，防模型跑偏 SFT 去 reward hacking。调它控制"模型能离 SFT 多远"。
- **KL ②（vs old policy, approx KL 监控）**：算法稳定性，防单步 PPO 更新太猛。调它（target_kl）控制"单次更新的步长"。

### 6.3 KL 估计器的选择：k1 vs k3（呼应 §2.4）

RLHF 实现里 KL ① 的估计也用 John Schulman 的 k1/k3：

> **【真实源码佐证，WebFetch 核实：TRL PPOConfig】** `kl_estimator: Literal["k1","k3"] = "k1"`。文档原文："Defaults to 'k1', a straightforward, unbiased estimator. Can be set to 'k3', an unbiased estimator with lower variance which 'appears to be a strictly better estimator'. Cannot be set to 'k2', as it is used for logging purposes." 即 §2.4 讲的三个估计器，在 RLHF 工业实现里被原样复用。这就是为什么 §2.4 让你"记住 k3"。

### 6.4 RLHF-PPO 的典型配置（fact + 待核）

> **【真实源码佐证，WebFetch 核实：TRL PPOConfig 默认值】** `learning_rate=3e-6`（比控制任务的 2.5e-4 小约百倍——大模型微调要很小的 lr）、`num_ppo_epochs=4`、`cliprange=0.2`（ε 沿用经典 PPO）、`vf_coef=0.1`、`cliprange_value=0.2`、`gamma=1.0`（RLHF 常用不折扣，因为是 episodic 序列任务）、`lam=0.95`（GAE λ 沿用）、`kl_coef=0.05`、`temperature=0.7`。

> **【验证信号，WebFetch 核实：TRL 文档 benchmark】** TRL 在 1B 模型 TL;DR 摘要任务上跑 PPO：PPO checkpoint 拿到 64.7% 偏好率，vs SFT 的 33.0%（GPT-4o mini 当 judge）。这是"PPO 确实在把模型往人类偏好推"的实证。

> **【待核】** InstructGPT / ChatGPT 的具体 PPO 超参数（β、lr、epoch、batch）OpenAI 未完整公开；业界数值多来自复现工作（如 TRL、DeepSpeed-Chat、《N+ Implementation Details of RLHF with PPO》），不同实现差异不小，**不要把某一家的数字当作"标准答案"**。第 5 章会展开 RLHF 全流程时再细化，并标注每个数字的出处。

> **【方向性提示，非本章重点】** 2024 年后，因为 PPO 在 RLHF 里要同时载 4 个模型（policy/ref/reward/value）、显存和工程开销大，社区出现了**砍掉 critic** 的简化路线：**GRPO**（用一组 response 的奖励均值当 baseline，回归 REINFORCE 本源，见第 3 章 §6）、**RLOO**、**REINFORCE++** 等。它们本质是"在 RLHF 场景下，序列级稀疏 reward + KL penalty 已经够稳，未必需要 PPO 那么重的 critic + clip 机制"。**但 PPO 仍是理解这一切的基准**——你必须先吃透 PPO，才能判断什么时候可以砍 critic、砍了会损失什么（value baseline 的方差缩减）。

## 五件套（章末固定）

### 一、概念题（检验是否真懂）

1. PPO 的 clipped objective 为什么用 `min(r·A, clip(r)·A)` 而不是直接 `clip(r)·A`？请用"对我们有利的方向 vs 不利的方向"解释 `min` 的作用。
2. 当 `A > 0` 且 `r = 1.5`（ε=0.2）时，policy loss 对这个样本的梯度是多少？为什么？`A < 0` 且 `r = 0.5` 时呢？

### 二、推导题

1. 从 importance sampling 出发，推出 PPO 的 surrogate `E_{s,a~π_old}[ r_t(θ)·A_t ]` 为什么是 `E_{π_θ}[A_t]` 的（一阶）近似，以及为什么"新旧策略差太远"会让这个近似失效。
2. 推导 k3 估计 `(r−1) − log r` 是 `KL(q‖p)` 的无偏估计（提示：`E_{x~q}[r] = 1`，再用 `log r = log(p/q)`），并说明它为什么恒非负（提示：`x − 1 ≥ log x`）。

### 三、代码题（扩展 demo）

### 四、排查题（给现象找根因）

### 五、思辨题（专家级判断）

1. PPO 的 clip 是 trust region 的"启发式近似"，不严格等价于 KL 约束。在什么情况下这个近似会明显失真（提示：考虑高维动作 / 单个 minibatch 内 ratio 分布很宽的情形）？这是否解释了为什么有些工作要在 clip 之上再加 `target_kl` early-stop？
