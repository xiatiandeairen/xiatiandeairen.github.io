> 一句话定位：内核并发不是"用锁保护共享变量"那么简单——它是**多核（SMP）、抢占（preemption）、中断（IRQ/softirq/NMI）三个独立异步源**同时轰炸同一地址空间的战场。搞清楚"谁会打断谁、在哪个上下文、能否睡眠"才是写出正确内核代码的前提；而这三件事，内核**全部压缩进一个叫 `preempt_count` 的 per-CPU 整数里**——读懂它，半个内核并发模型就通了。

---

## TL;DR — 本章最硬结论

1. **内核并发有三个独立来源**：SMP 并行、内核抢占、中断/软中断/NMI——它们各自对应不同的保护原语，混用必出 bug。三者的"原子性状态"被统一编码进 `preempt_count`：低 8 bit 抢占、次 8 bit softirq、再 4 bit hardirq、再 4 bit NMI（v6.12 `include/linux/preempt.h` 逐字可查）。
2. **原子上下文（atomic context）不能睡眠**，这不是惯例而是物理约束。真正的检查点在 `schedule_debug()` → `in_atomic_preempt_off()`，命中后走 `__schedule_bug()` 打印 `BUG: scheduling while atomic`（v6.12 `kernel/sched/core.c:5856/5903` 逐字可查）。是否 panic 取决于 `panic_on_oops`/`panic_on_warn`。
3. **PREEMPT_RT 不是魔法**，它把几乎所有 spinlock 换成可睡眠的 RT mutex，吞吐换延迟。它**在内核外维护了约 20 年（2004→2024）**，分批进主线，最后一块拼图是 printk/NBCON 重写，6.12（2024-09）合入后 x86/arm64/riscv 才第一次能用纯主线编出完整 RT 内核。
4. **2024 年内核抢占模型正在被推倒重来**：Gleixner 主张干掉 `PREEMPT_NONE`/`PREEMPT_VOLUNTARY` 这两个**编译期**选项和散落全内核的 `cond_resched()`，用 **lazy preemption（`TIF_NEED_RESCHED_LAZY`，PREEMPT_LAZY，6.13 合入）** 把三种非 RT 模型统一成**运行时可切**。这是 staff 级别必须知道的"现在进行时"。
5. **编译器和 CPU 都会重排内存访问**，`barrier()`/`smp_rmb()`/`READ_ONCE()`/`smp_load_acquire()` 不是风格建议，是正确性硬需求。x86 TSO 也挡不住 Store Buffering——SB litmus 在 x86 上照样要 `smp_mb()`。

**前置依赖章**：第 4 章（进程/线程，`task_struct`）、第 5 章（调度器，EEVDF）、第 6 章（上下文切换）。锁机制详见第 12 章，RCU 详见第 13 章，中断子系统详见第 18 章。

**本章相对原版的增量**：① 把 `preempt_count`、`scheduling while atomic`、`migrate_disable`、softirq offset 从"概念描述"换成 **v6.12 逐字源码 + 逐行注解**；② 新增 **§2.6 设计考古**——BKL 的死亡、x86 per-CPU preempt_count 的迁移、PREEMPT_RT 20 年主线化、2024 lazy preemption 的争论，全部带 commit / LWN 出处；③ 每个机制补"为什么不用更简单的替代 + 失败模式 + 底层根因"。

## 1. 背景：为什么内核并发比用户态难得多

用户态并发的经典心智模型是：多个线程共享堆，用 mutex 保护，一把锁一个临界区，出了临界区就安全。这个模型在内核里**根本不够用**。

### 1.1 三个独立的"异步打断源"

```
CPU 0                          CPU 1
│                              │
│  执行内核代码（进程上下文）    │  执行内核代码（进程上下文）
│                              │
│  ← 硬件中断（来自 NIC）       │  ← 硬件中断（来自 timer）
│    hardirq 抢占 CPU 0        │    hardirq 抢占 CPU 1
│                              │
│    ↓ 软中断/tasklet          │    ↓ 软中断/tasklet
│      处理积压的包收发         │      处理定时器到期
│                              │
│  ← 内核抢占（另一个高优任务） │
│    __schedule(SM_PREEMPT)    │
```

- **SMP 并行**：两个 CPU 同时跑不同进程的内核代码，访问同一数据结构。最像用户态多线程，用 spinlock 解决。
- **内核抢占**：进程在内核态执行时被调度器抢占，切到另一个进程继续跑内核代码。**单核也存在**。
- **中断/软中断/NMI**：硬件中断随时打断当前 CPU 上的任何代码（不管你在进程上下文还是另一个中断里），软中断（softirq）在中断返回时或 `ksoftirqd` 线程里执行，NMI（不可屏蔽中断）连关中断都挡不住。

💡 **坑点**：只用 mutex 保护的代码在单核 UP 内核里可能没问题，但一打开 SMP 或中断就错——mutex 不禁中断，中断处理程序可以在持有 mutex 的临界区中途打断进来；更糟的是，如果中断处理程序也去 `mutex_lock()` 同一把锁，它在原子上下文里试图睡眠，直接 `scheduling while atomic`。

### 1.2 错误代价不对等

用户态线程崩了，内核收拾残局；内核代码崩了，整个系统 panic（或更阴险地"带病前行"，把损坏的状态扩散到别处再以诡异方式崩）。这要求内核并发的正确性标准远高于用户态——"99% 不触发"在内核里等于"埋了颗雷"。

### 1.3 可重入（Reentrancy）与竞态（Race）的区别

**可重入**：函数被"重新进入"（同 CPU 的中断打断，或多 CPU 并发调用），但只要没有共享可变状态，就没有竞态。纯函数天然可重入。

**竞态（Race Condition）**：两条执行路径，对同一可变状态做了 check-then-act 或 read-modify-write，且执行顺序不确定，导致结果不正确。

区分这两个概念是写内核代码的基本功。中断安全（interrupt-safe）≠ 无竞态；无竞态的前提是找到所有并发路径，并证明不存在不一致交叉。

## 2. 演进：从 Big Kernel Lock 到 lazy preemption

### 2.1 UP 时代：根本没有 SMP 并发

早期 Linux（0.x ~ 1.x）跑在单核 x86 上，中断处理程序与进程代码之间的唯一并发源就是中断。解法是"关中断（`cli`）进临界区，开中断（`sti`）出临界区"，简单粗暴、完全正确，但**不可扩展**——一旦上多核就崩。

### 2.2 2.0 — BKL（Big Kernel Lock）

Linux 2.0 引入 SMP 支持，解法是一把全局锁 `kernel_flag`，史称 Big Kernel Lock（BKL）。所有进入内核的路径都先拿 BKL，退出内核再放。

```
进程A进内核 → lock_kernel() → 执行内核代码 → unlock_kernel() → 回用户态
进程B进内核 → lock_kernel() → 阻塞等待 → ...
```

BKL 正确性容易保证，缺点是把 SMP 变成伪并行——多核等于单核，扩展性为零。BKL 还有个怪异语义：它**会随睡眠自动释放、唤醒时自动重拿**（递归可重入），这让它比普通锁更难推理，也是后来必须铲除的原因之一。

### 2.3 2.2 ~ 2.6 — 细粒度锁替换 BKL

从 2.2 开始，开发者逐步把 BKL 拆开，给每个子系统配自己的锁（per-subsystem → per-structure → per-field）。这个过程持续了十年。

> ⚠️ **时效与版本对照（高频记错点）**：BKL 的实现代码在 **Linux 2.6.39（2011）** 被彻底清空，收尾 commit 是 Arnd Bergmann 的 `4ba8216c`，标题 **"BKL: That's all, folks"**【真实 commit，下文 §2.6.1 逐字引用】。很多资料说"3.0 移除 BKL"——那是因为 2.6.39 的下一个版本被 Linus 改名成 3.0，时间紧挨着。**面试说"2.6.39"是稳的。**

### 2.4 抢占内核的演进

早期 Linux 内核**不可抢占**：进程一进内核态，调度器就不会主动拿走它的 CPU，直到它主动 `yield`、阻塞或返回用户态。这保证"内核代码串行（单核视角）"，但延迟高、实时性差。

| 配置选项 | 何时出现 | 抢占点 | 典型用途 |
|---|---|---|---|
| `PREEMPT_NONE` | 最早期 | 只有显式 `schedule()`/阻塞点 | 服务器、吞吐优先 |
| `PREEMPT_VOLUNTARY` | 2.6 早期（~2004） | 额外的 `might_resched()`/`cond_resched()` 检查点 | 桌面（轻度） |
| `PREEMPT`（=`PREEMPT_FULL`） | 2.6 中期 | 几乎所有内核路径（spinlock 临界区外） | 桌面/嵌入式 |
| `PREEMPT_RT` | 主线 6.12（2024-09） | 几乎全部，spinlock → sleeping lock | 硬实时 |
| `PREEMPT_LAZY`（新） | 6.13（2024-12） | 见 §2.6.4：运行时可调的"懒抢占" | 统一前三者 |

🎯 **面试加分点**：`PREEMPT_FULL` 和 `PREEMPT_RT` 的根本区别在 spinlock 语义。Full 的 spinlock 持有期间不能调度（spin 等待）；RT 把大多数 spinlock 换成 RT mutex（可睡眠），意味着即使"持锁"期间也可被抢占——概念上革命性，代价是高频锁竞争场景上下文切换开销累积。

### 2.5 现状一句话

到 v6.12，内核并发模型 = **三源（SMP/抢占/中断）× 一个 `preempt_count` × 一套内存序原语（`READ_ONCE`/acquire-release/barrier）× 一组上下文相关锁（spinlock / mutex / RCU / local_lock）**。下面 §2.6 先讲这套东西怎么演化来的（设计考古），§3 再逐个拆源码。

### 2.6 设计考古：四场真实的设计争论（带出处）

> 这一节是本章的"扎根"核心。每个机制都不是凭空设计的，是为了修前一代的具体缺陷、在邮件列表/LWN 上吵出来的。下面四段全部来自实际 WebFetch 的 commit / LWN / Phoronix。

#### 2.6.1 BKL 之死：`4ba8216c` "That's all, folks"

【真实 commit：github.com/torvalds/linux commit `4ba8216cd90560bc402f52076f64d8546e8aefcb`，作者 Arnd Bergmann】

commit message 开头逐字（实际取得）：

> "This removes the implementation of the big kernel lock, at last. A lot of people have worked on this in the past, so the credit for this patch should be with everyone who participated in the hunt."

它删掉了 `lib/kernel_lock.c`（136 行）和 `include/linux/smp_lock.h`（65 行），清掉 `init/Kconfig`/`lib/Kconfig.debug` 里的 BKL 选项。

- **谁/何时**：Arnd Bergmann 收尾，2011，进 2.6.39。但 commit 本身强调这是**集体十年长跑**的终点（Alan Cox、Ingo Molnar、Thomas Gleixner 等都在 hunt 里）。
- **解决了什么缺陷**：BKL 的"睡眠自动释放/递归"语义让它无法做细粒度推理，且是 SMP 扩展性的天花板。
- **底层根因**：BKL 把"正确性"建立在"一次只有一个 CPU 在内核里"上——这与 SMP 的存在意义直接矛盾。拆 BKL 的十年，本质是把"一把大锁"重写成"成百上千把语义清晰的小锁 + RCU + per-CPU"。

#### 2.6.2 x86 把 preempt_count 搬进 per-CPU：`c2daa3b`

【真实 commit：github.com/torvalds/linux commit `c2daa3bed53a81171cf8c1a36db798e82b91afe8`，作者 Peter Zijlstra，committer Ingo Molnar，进 v3.13（2013）】

commit message 逐字（实际取得）：

> "Convert x86 to use a per-cpu preemption count. The reason for doing so is that accessing per-cpu variables is a lot cheaper than accessing thread_info variables."

它还把 `thread_info::preempt_count` 改名成 `thread_info::saved_preempt_count`——故意制造编译错误，逼所有人改用新的 per-CPU 接口，防止有人偷偷直接访问旧字段。

- **解决了什么缺陷**：`preempt_count` 是内核里**被检查最频繁的值之一**（每次 `preempt_enable`、每次 `spin_unlock`、每次 `schedule` 都读）。旧实现要先从 `current` 解引用 `thread_info` 才能拿到它，多一次内存解引用。
- **底层根因（微架构）**：per-CPU 变量在 x86 上是 `%gs:offset` 一条访存，且能和 `current_task`、`__preempt_count` 等热数据塞进同一 cache line，`__switch_to()` 切换时一并 save/restore，减少 cache miss。
- **staff 级结论**：所以"preempt_count 在 thread_info 里"在 **x86 上是过时说法**。准确表述：**逻辑上 per-task，x86 物理实现是 per-CPU（`DECLARE_PER_CPU(int, __preempt_count)`），切换时在 per-CPU 槽和 `saved_preempt_count` 之间倒腾。**

#### 2.6.3 PREEMPT_RT 的 20 年主线化长征

PREEMPT_RT（旧称 "the RT patch" / RTL）的目标：把 Linux 改造成可做**确定性低延迟**的硬实时内核。它是内核史上**最久的 out-of-tree 项目之一**。

时间线（综合 LWN 实际取得的两篇）：

- **设计哲学（2005）**【LWN 146861，Paul McKenney，"A realtime preemption overview"，2005-08-10】：核心思路是把"抢占当成多加一个虚拟 CPU"——既然内核为 SMP 已经有了完整的 locking 基础设施，那就复用它来支持"几乎无限的抢占"，而不是重写内核。文章里 Ingo Molnar 大量出现在评论中解释设计取舍。这一篇定义了今天 RT 的根本套路：**preemptible spinlock + preemptible IRQ handler（跑在进程上下文）+ priority inheritance**。
- **分批进主线（5.x 中后期，~2019）**：locking 框架重构、`local_lock`、threaded IRQ、可睡眠 spinlock 底层的 RT mutex 陆续进主线；`CONFIG_PREEMPT_RT` 这个 Kconfig 选项约 5.3（2019）就出现了，但当时还不能完整编译。
- **终局（2024）**【LWN 989212，Jonathan Corbet，"The realtime preemption end game — for real this time"，2024-09-06】：文章指出该项目"始于约 20 年前（~2004）"，Sebastian Andrzej Siewior 宣布最后里程碑。**最后的卡点是 printk 子系统重写（NBCON consoles）**——老 printk 会阻塞、破坏 RT 延迟保证。
- **printk/NBCON 合入 6.12**【Phoronix，2024-09-17，"The Printk Blocker For Upstreaming Real-Time PREEMPT_RT Has Been Merged"】：逐字结论"With the printk bits merged, PREEMPT_RT could be enabled on X86, ARM64 and Risc-V." NBCON 把控制台打印 offload 到专用内核线程（`CON_NBCON`），消除阻塞。

> ⚠️ **措辞精度（staff 级细节）**：说"PREEMPT_RT 在 6.12 进主线"是对的（指**完整可用、三大架构可编**）；但若有人说"6.12 才开始合 RT"则**错**——核心机制 5.x 就在了，6.12 只是补上最后一块。一句话："RT 的肉早就进来了，6.12 补的是 printk 这层皮。"
>
> 另一个常被忽略的现实【989212 评论】：进主线 ≠ 发行版默认开。它仍是 `CONFIG_PREEMPT_RT` 编译选项，distro 默认不带。对终端用户"是否 mainline 差别不大"——主线化的真正价值是**维护成本下降、不再 rebase 巨型 out-of-tree patch**。

#### 2.6.4 2024 进行时：lazy preemption 要干掉 `cond_resched()`

这是**当前最该知道的"现在进行时"**，也是区分"读过 2020 年前博客"和"跟得上 2024 主线"的分水岭。

**问题的起点**【LWN 944686，Corbet，"Revisiting the kernel's preemption models (part 1)"，2023-09-21】：Ankur Arora 想用长跑指令加速 x86 清内存，但这些指令会拉高延迟。他提了个 `TIF_ALLOW_RESCHED` 标志，结果发现行不通——标了"可抢占"的代码仍可能通过中断/陷入进入不可抢占上下文。讨论由此发散到一个更根本的问题：**现在四种抢占模型 + 散落全内核的 `cond_resched()` 调用本身就是个烂设计。**

Linus 的原话（实际取得）：

> "I really hate that, because I was hoping we'd be able to use this to not have so many of those annoying and random 'cond_resched()' calls."

**根因**：`PREEMPT_NONE`/`PREEMPT_VOLUNTARY` 是**编译期**选项。为了让 `PREEMPT_NONE` 内核也不至于在长循环里饿死别人，开发者被迫在内核各处**手塞几百个 `cond_resched()`**——"如果该让出 CPU 了就让一下"。这些调用是 (1) 散弹式的、维护噩梦；(2) 在 `PREEMPT_FULL` 下纯属冗余；(3) 永远塞不全（总有漏的长路径）。

**解法：lazy preemption（PREEMPT_LAZY）**【LWN 994322，Corbet，"The long road to lazy preemption"，2024-10-18；Peter Zijlstra 提交 patch series；6.13 合入，6.15 继续完善 per LWN 1011770】：

引入**第二个标志** `TIF_NEED_RESCHED_LAZY`，与既有 `TIF_NEED_RESCHED` 配对：

```
事件发生（如更高优任务唤醒）
        │
        ├─ PREEMPT_FULL 语义：置 TIF_NEED_RESCHED  → 下一个安全点立刻抢占
        │
        └─ "懒"语义：     置 TIF_NEED_RESCHED_LAZY → 先不抢，只打个标
                          ├─ 返回用户态时：两个标志任一置位都抢占
                          └─ 下一个 timer tick：把 LAZY 升级成"紧急"再抢
```

- **效果**：用一个运行时旋钮，把 `PREEMPT_NONE`（吞吐）和 `PREEMPT_FULL`（低延迟）的行为**统一成可在 `/sys` 或启动参数里切换**，不再需要编译期三选一，也不再需要满地 `cond_resched()`。Gleixner 预期最终**只剩不到 5%** 的 `cond_resched()` 需要保留（实际取得）。
- **性能数据**【994322，Mike Galbraith 测试，实际取得】："throughput with lazy preemption falls just short of that with PREEMPT_VOLUNTARY"——懒抢占吞吐略低于 `PREEMPT_VOLUNTARY`，但换来了不用手塞抢占点、且延迟可控。
- **和 RT 的关系**：lazy 的双标志机制其实**借鉴自 PREEMPT_RT**（RT 早就有"lazy"概念区分普通任务和 RT 任务的抢占紧迫度）。RT 主线化之后，这套机制下放给非 RT 内核复用——这是 §2.6.3 那 20 年长征的一个红利。

🎯 **面试王炸**：被问"内核抢占有几种模型"时，答完四种之后追一句：**"不过 2024 年起方向是用 lazy preemption（`TIF_NEED_RESCHED_LAZY`，6.13 进主线）把 NONE/VOLUNTARY/FULL 三种编译期模型统一成运行时可切，并逐步删掉散落的 `cond_resched()`——这套双标志机制是从 PREEMPT_RT 借来的。"** 这句话直接把面试官从"你背过书"切到"你跟主线"。

## 3. 现状：内核并发模型源码精读

### 3.1 执行上下文分类

```
┌──────────────────────────────────────────────────────────┐
│                    内核执行上下文                          │
│  ┌─────────────────────────┐  ┌────────────────────────┐ │
│  │   进程上下文             │  │    中断上下文            │ │
│  │  (Process Context)      │  │   (Interrupt Context)  │ │
│  │  · 有关联的 task_struct  │  │  · 无 task_struct 关联  │ │
│  │  · 可以睡眠              │  │  · 不能睡眠             │ │
│  │  · 可以访问用户空间       │  │  · 不能访问用户空间      │ │
│  │  系统调用路径             │  │  硬中断 (hardirq)       │ │
│  │  内核线程 (kthread)       │  │  软中断 (softirq)       │ │
│  │  工作队列 (workqueue)     │  │  tasklet / NMI         │ │
│  └─────────────────────────┘  └────────────────────────┘ │
└──────────────────────────────────────────────────────────┘
```

**原子上下文（atomic context）** 是更精确的说法：代码处于以下任一状态时都不能睡眠：在中断处理程序里（hardirq/softirq/NMI）、持有 spinlock、中断被禁用（`local_irq_disable()`）、抢占被禁用（`preempt_disable()`）、RCU 读锁持有期间。

判断上下文的宏族——**全部就是 `preempt_count` 的掩码运算**。这是 v6.12 `include/linux/preempt.h` 逐字源码：

```c
/* 【真实源码 v6.12 include/linux/preempt.h】 */
#define in_irq()		(hardirq_count())     /* 在硬中断里（已弃用别名） */
#define in_hardirq()		(hardirq_count())     /* hardirq 计数非零 */
#define in_softirq()		(softirq_count())     /* 在软中断里 或 BH 被禁 */
#define in_interrupt()		(irq_count())         /* hardirq ∪ softirq ∪ NMI 任一非零 */
#define in_task()		(!(preempt_count() & (NMI_MASK | HARDIRQ_MASK | SOFTIRQ_OFFSET)))
#define in_atomic()	(preempt_count() != 0)
#define in_atomic_preempt_off() (preempt_count() != PREEMPT_DISABLE_OFFSET)
```

逐行注解：
- `in_hardirq()`/`in_softirq()` 直接看对应掩码位是否非零，**零分支、一次访存**——这正是 §2.6.2 把 preempt_count 搬进 per-CPU 的回报：这些宏在中断热路径上被狂调。
- `in_task()` 的实现很微妙：它检查的是 `NMI | HARDIRQ | SOFTIRQ_OFFSET`，**注意是 `SOFTIRQ_OFFSET` 而不是 `SOFTIRQ_MASK`**。原因见 §3.3.1：`SOFTIRQ_OFFSET`（1×）表示"正在处理软中断"，`SOFTIRQ_DISABLE_OFFSET`（2×）表示"只是 `local_bh_disable` 关了下半部"。`in_task()` 只想排除"正在跑软中断"，不想把"进程上下文里临时关了 BH"也算成中断上下文。这种位编码的精巧是 staff 级考点。
- `in_atomic()` 就是 `preempt_count() != 0`——但**有个致命陷阱**，见下。
- `in_atomic_preempt_off()` = `preempt_count() != PREEMPT_DISABLE_OFFSET`，意思是"除去一层正常的 `preempt_disable` 之外还有别的原子层"。这正是 `schedule()` 里判定非法睡眠的依据（§3.2）。

> ⚠️ **`in_atomic()` 的著名陷阱**——这段注释 v6.12 逐字保留，建议背下来：
> ```c
> /* 【真实源码 v6.12 include/linux/preempt.h】 */
> /*
>  * Are we running in atomic context?  WARNING: this macro cannot
>  * always detect atomic context; in particular, it cannot know about
>  * held spinlocks in non-preemptible kernels.  Thus it should not be
>  * used in the general case to determine whether sleeping is possible.
>  * Do not use in_atomic() in driver code.
>  */
> #define in_atomic()	(preempt_count() != 0)
> ```
> **根因**：在 `PREEMPT_NONE` 内核上，`spin_lock()` 内部的 `preempt_disable()` 被编译期优化成空——既然根本不抢占，就没必要维护抢占计数。于是"持有 spinlock"**不会**抬高 `preempt_count`，`in_atomic()` 在这种配置下**检测不到持锁**。所以官方明令"Do not use in_atomic() in driver code"，它只配做调试启发，不能拿来做正确性判断。
>
> 这个 bug 不是孤例，它是 2020 年一场大讨论的核心【LWN 831678，Corbet，"Four short stories about preempt_count()"，2020-09-18】：Thomas Gleixner 发了 patch 集，Linus 在里面放话——**"根据 `preempt_count()` 探测上下文来改变行为，是 fundamentally wrong 的"**，主张上下文应该当参数显式传，而不是运行时反推；Paul McKenney 则反驳，说 RCU 这类 API 不靠上下文感知就会 API 爆炸。这场争论至今没有完全收口，但它解释了为什么内核对 `in_atomic()` 这么忌惮。

### 3.2 为什么中断里不能睡眠：源码级真相

经典问题，答案不是"规范不允许"，而是物理 + 源码双重决定的。我们直接看**真正的检查点**。当任何路径走到 `__schedule()`，第一件事是 `schedule_debug()`：

```c
/* 【真实源码 v6.12 kernel/sched/core.c:6548】 */
static void __sched notrace __schedule(int sched_mode)
{
	struct task_struct *prev, *next;
	bool preempt = sched_mode > SM_NONE;
	...
	cpu = smp_processor_id();
	rq = cpu_rq(cpu);
	prev = rq->curr;

	schedule_debug(prev, preempt);   /* ← 非法睡眠的检查就在这 */
	...
}
```

`schedule_debug()` 的核心判定（逐字 + 逐行）：

```c
/* 【真实源码 v6.12 kernel/sched/core.c:5884】 */
static inline void schedule_debug(struct task_struct *prev, bool preempt)
{
	...
	if (unlikely(in_atomic_preempt_off())) {   /* preempt_count 超过"一层正常 preempt_disable" */
		__schedule_bug(prev);                  /* → 打印 scheduling while atomic */
		preempt_count_set(PREEMPT_DISABLED);   /* 强行把计数复位，试图让系统苟住 */
	}
	rcu_sleep_check();                          /* 顺带查 RCU 读锁里是否睡眠 */
	SCHED_WARN_ON(ct_state() == CT_STATE_USER); /* 不该从用户态上下文进这里 */
}
```

`__schedule_bug()` 才是那条人人见过的报错的源头：

```c
/* 【真实源码 v6.12 kernel/sched/core.c:5856】 */
static noinline void __schedule_bug(struct task_struct *prev)
{
	/* Save this before calling printk(), since that will clobber it */
	unsigned long preempt_disable_ip = get_preempt_disable_ip(current);

	if (oops_in_progress)
		return;

	printk(KERN_ERR "BUG: scheduling while atomic: %s/%d/0x%08x\n",
		prev->comm, prev->pid, preempt_count());   /* ← 注意：打印的是 raw preempt_count！ */

	debug_show_held_locks(prev);     /* 把当前持有的锁全列出来——排障关键信息 */
	print_modules();
	if (irqs_disabled())
		print_irqtrace_events(prev); /* 中断从哪关的 */
	if (IS_ENABLED(CONFIG_DEBUG_PREEMPT)) {
		pr_err("Preemption disabled at:");
		print_ip_sym(KERN_ERR, preempt_disable_ip);  /* 抢占在哪行被禁的——直接指向罪犯 */
	}
	check_panic_on_warn("scheduling while atomic"); /* panic_on_warn 开了就在这停机 */

	dump_stack();
	add_taint(TAINT_WARN, LOCKDEP_STILL_OK);  /* 给内核打"已被污染"标记 */
}
```

逐行价值点（这就是排障实战）：
1. `printk(... "0x%08x", preempt_count())`——**报错里那个 `0x00000002` 之类的十六进制就是整个 `preempt_count`**。排障第一步是按 §3.6 的位域把它拆开：哪一层非零（hardirq? softirq? preempt?）。
2. `debug_show_held_locks` + `print_irqtrace_events` + `Preemption disabled at:`——内核把"你持了哪些锁、中断在哪关的、抢占在哪行禁的"全给你打出来了。**很多人忽略 `Preemption disabled at:` 那一行，它直接指向罪犯源码位置**（需 `CONFIG_DEBUG_PREEMPT`）。
3. `check_panic_on_warn`——**这才是"会不会 panic"的真相**：默认只是打印 + taint 继续跑；`panic_on_warn=1` 才当场停机。
4. `preempt_count_set(PREEMPT_DISABLED)`——内核试图"复位计数让系统苟着"，但**调度状态已经被破坏**，后续随时可能以更诡异的方式崩。所以 `scheduling while atomic` 是"已经太晚"的信号。

> ⚠️ **精确措辞**：`scheduling while atomic` 走 `printk` + `check_panic_on_warn` + `dump_stack` 路径，**是否停机取决于配置**。面试说"触发 BUG/oops，配置激进（`panic_on_warn`/`panic_on_oops`）时当场 panic，否则带病前行"比一口咬定"必然 panic"更准确。

**更深一层（语义根因）**：`schedule()` 假设"调用者是一个能被放进 runqueue/wait queue、之后再被唤醒恢复的 task"。中断上下文里没有这样的语义单元——它借用的是被打断进程的栈，没有自己的 `task_struct`。`schedule()` 真要切走，**返回路径是未定义的**（切回来时该恢复谁？）。`in_atomic_preempt_off()` 检查只是把这个未定义行为提前拦下。

💡 **更早的防线**：真正可靠的不是等 `schedule_debug` 拦截，而是 `CONFIG_DEBUG_ATOMIC_SLEEP`——它让 `might_sleep()` 在你**刚调用可睡眠函数那一刻**就报警（而非等到真走进 `schedule()`）。典型罪犯：`kmalloc(GFP_KERNEL)`——`GFP_KERNEL` 在内存紧张时触发 direct reclaim，可能睡眠；中断里必须用 `GFP_ATOMIC`（从原子内存池分配，失败直接返回 NULL）。

### 3.3 竞态来源分析框架

识别竞态的正确步骤：

**Step 1：枚举并发执行路径** —— 对一段访问共享状态 S 的代码，问"哪些执行流能同时到达这里"：
- 同一 CPU 上：hardirq 可打断进程代码；softirq 可打断进程代码；抢占可切到另一进程。
- 不同 CPU 上：任何时刻另一 CPU 可能在跑同一段内核代码（如网卡收包路径）。

**Step 2：找到共享可变状态** —— 只读状态不需保护；只被单 CPU 访问的 per-CPU 变量不需跨 CPU 锁（但需禁抢占/禁中断防同 CPU 中断打断）。

**Step 3：构造不一致交叉序列** ——

```c
/* 非原子的 check-then-act（示意代码，非逐字，勿直接用） */
if (ptr != NULL) {
    /* 另一个 CPU 在这里把 ptr 设 NULL 并 kfree */
    ptr->field = 1;  /* use-after-free */
}

/* 非原子的 read-modify-write（示意代码，非逐字） */
counter++;  /* 实际是 load → add → store，中间可被打断 → lost update */
```

**Step 4：选对保护原语** ——

| 竞态类型 | 推荐原语 |
|---|---|
| SMP 两 CPU 并发（不涉中断） | `spinlock_t`（或 mutex，如可睡眠） |
| 进程上下文 vs 中断处理程序 | `spin_lock_irqsave()` |
| 进程上下文 vs 软中断 | `spin_lock_bh()` |
| 单纯原子整数 | `atomic_t` / `atomic64_t` |
| 读多写少、读路径极热 | RCU（见第 13 章） |
| 位操作（flags 字段） | `set_bit()` / `clear_bit()` 等原子位操作 |
| 保护 per-CPU 数据（RT 友好） | `local_lock`（见 §4.3.1） |

#### 3.3.1 softirq 为什么用 "2× offset"：源码里的精巧编码

这是个隐藏的 staff 级考点。看 v6.12 `kernel/softirq.c` 的注释（逐字）：

```c
/* 【真实源码 v6.12 kernel/softirq.c:91】 */
/*
 * SOFTIRQ_OFFSET usage:
 *
 * On !RT kernels 'count' is the preempt counter, on RT kernels this applies
 * to a per CPU counter and to task::softirqs_disabled_cnt.
 *
 * - count is changed by SOFTIRQ_OFFSET on entering or leaving softirq
 *   processing.
 *
 * - count is changed by SOFTIRQ_DISABLE_OFFSET (= 2 * SOFTIRQ_OFFSET)
 *   on local_bh_disable or local_bh_enable.
 *
 * This lets us distinguish between whether we are currently processing
 * softirq and whether we just have bh disabled.
 */
```

逐行拆解这个设计：
- softirq 区有 8 bit。内核**故意**让"进入软中断处理"加 `1×SOFTIRQ_OFFSET`，而"`local_bh_disable()` 关下半部"加 `2×SOFTIRQ_OFFSET`。
- **为什么要区分**：`local_bh_disable()` 是在**进程上下文**里临时挡住软中断（比如网络栈持有某状态时不想被收包软中断打断），此时你**仍在进程上下文、仍可睡眠**（在非 RT 内核上）；而"正在跑某个软中断处理函数"是真·中断上下文、不可睡眠。两者都会抬高 softirq 计数，但语义天差地别。
- 用奇偶/倍数编码区分：`in_serving_softirqs()` 类查询能据此判断"我是真在跑 softirq，还是只是有人关了 BH"。这就是 §3.1 里 `in_task()` 为什么用 `SOFTIRQ_OFFSET`（1×）而非 `SOFTIRQ_MASK` 的原因——它要排除"真在跑软中断"，但不把"进程上下文关了 BH"误判成中断上下文。

> 💡 **底层根因**：这是典型的"用一个整数的位编码省掉一个分支/一个额外字段"的内核手法。代价是可读性——不看注释根本想不到 1× 和 2× 有语义差别。这也是为什么内核源码必须连注释一起读。

### 3.4 原子操作与内存模型

内核并发不只"加锁"，还涉及编译器和 CPU 的重排序。

**编译器重排**：编译器为优化可能互换两条独立指令顺序。单线程语义等价，多核可能破坏假设。`barrier()` 告诉编译器"别把这条前后的读写越过这个点"。注意 §3.1 看到的 `preempt_disable()` 定义里就带了 `barrier()`：

```c
/* 【真实源码 v6.12 include/linux/preempt.h】 */
#define preempt_disable() \
do { \
	preempt_count_inc(); \
	barrier(); \
} while (0)

#define preempt_enable() \
do { \
	barrier(); \
	if (unlikely(preempt_count_dec_and_test())) \
		__preempt_schedule(); \
} while (0)
```

逐行注解：`preempt_disable` 先抬计数**再 `barrier()`**——保证临界区代码不会被编译器挪到"抬计数"之前；`preempt_enable` 先 `barrier()` **再降计数**——保证临界区代码不会被挪到"降计数"之后。两个 `barrier()` 把临界区"夹住"。`preempt_enable` 末尾的 `preempt_count_dec_and_test()`：降到 0 就立刻 `__preempt_schedule()`——这就是"出临界区时如果攒了抢占请求，马上响应"的实现。**注意它只是编译器屏障，不是 CPU 内存屏障**——禁抢占解决的是同 CPU 调度，不解决跨 CPU 内存可见性，后者要 §3.4 下面的 `smp_*`。

**CPU 乱序与 Store Buffer**：现代 CPU 不保证程序顺序。x86 内存模型相对强（TSO），但 ARM/RISC-V 弱序，允许更激进重排。看经典 **Store Buffering（SB）litmus**——反直觉之处：**即使 x86 TSO，两个 CPU 也可能都读到 0**：

```
初始 A = B = 0
CPU 0                      CPU 1
store A = 1               store B = 1
load  r1 = B   ← 可能 0    load  r2 = A   ← 可能 0
结果 r1 == 0 && r2 == 0 是允许的
```

根因：每个 CPU 把 store 先放进自己的 **store buffer**，还没刷到全局可见就先执行了后面的 load——于是各自读到对方"尚未公布"的旧值。x86 TSO 唯一放松的恰是 **store→（不同地址的）load** 可乱序；它**不允许** store→store、load→load、load→store 乱序（这点比 ARM/RISC-V 强）。

> 💡 **教学要点**：很多人以为"x86 强序 = 不用屏障"，错。SB 这个 case 在 x86 上就需要 `smp_mb()`（编译出 `mfence` 或 `lock` 前缀）才能禁止。x86 省掉的是 acquire/release（编成普通 mov），但 **full barrier `smp_mb()` 在 x86 上有真实指令成本**。弱序架构连 acquire/release 都要真屏障——这是为什么内核坚持 `smp_load_acquire`/`smp_store_release` 而非裸读写：让架构层自己决定要不要插屏障，x86 上是 no-op，ARM 上才付费。

**屏障必须配对**——这是 `Documentation/memory-barriers.txt` 反复强调的纪律（实际取得）：

```
/* 【真实源码 v6.12 Documentation/memory-barriers.txt，SMP BARRIER PAIRING】 */
CPU 1                    CPU 2
WRITE_ONCE(a, 1);        x = READ_ONCE(b);
<write barrier>          <read barrier>
WRITE_ONCE(b, 2);        y = READ_ONCE(a);
```

文档原话："When dealing with CPU-CPU interactions, certain types of memory barrier should always be paired. A lack of appropriate pairing is almost certainly an error." 一个 CPU 的写屏障**必须**配对另一个 CPU 的读屏障，否则可见性顺序不成立。**孤立的屏障几乎总是 bug。**

**内核的解法**：

| 原语 | 作用 |
|---|---|
| `READ_ONCE(x)` | 防编译器合并/消除/撕裂这次读 |
| `WRITE_ONCE(x, v)` | 防编译器合并/消除/撕裂这次写 |
| `smp_rmb()` | SMP 读屏障：屏障前 load 对屏障后 load 可见 |
| `smp_wmb()` | SMP 写屏障：屏障前 store 在屏障后 store 之前提交 |
| `smp_mb()` | 全屏障（双向） |
| `smp_store_release(p, v)` / `smp_load_acquire(p)` | C11 acquire/release 语义，写新内核代码首选 |

💡 **坑点**：`volatile` 在 C 里只"告诉编译器不要优化这个变量的访问"，**不提供原子性，也不插内存屏障**。内核里 `volatile` 几乎永远是错的选择，用 `READ_ONCE`/`WRITE_ONCE`。

🎯 **面试加分点**：`READ_ONCE` 实质是 `*(volatile typeof(x) *)&x`，生成机器码和 `volatile` 类似，但语义是"这次读是有意为之的并发读，不要优化掉"，配合 `WRITE_ONCE` 构成 plain data race 的最小防护；真正的内存序保证要 `smp_load_acquire`/`smp_store_release`。

### 3.5 PREEMPT_RT 的深层机制

```
PREEMPT_FULL 的 spinlock 持锁期间：
  spin_lock() → preempt_disable()      // 抬 preempt_count
  临界区（不可被调度抢占）
  spin_unlock() → preempt_enable()     // 降，可能触发 __preempt_schedule
  → 持锁 CPU 一直 spin 等另一个 CPU 释放，期间不调度

PREEMPT_RT 的 spinlock（底层 RT mutex）：
  spin_lock() → 可能 sleep（block）
  临界区（可以被调度抢占！）
  spin_unlock() → 唤醒等待者
  → 锁竞争时线程睡眠，调度器可运行其他任务
```

PREEMPT_RT 引入**优先级继承（Priority Inheritance）**解决优先级反转：低优任务持锁时，临时继承等它的高优任务的优先级，防止中优任务抢占低优持锁者导致高优饥饿。这个机制在 §2.6.3 那篇 2005 LWN 里就被 McKenney 列为 RT 三大支柱之一。

> ⚠️ RT 化的代价（底层根因）：Full 的 spinlock 竞争是"忙等"（占 CPU 但不切换，cache 热）；RT 的 spinlock 竞争是"睡眠+唤醒"（释放 CPU 但付出两次上下文切换 + cache 冷）。在**短临界区高频竞争**场景，RT 反而更慢——这就是"吞吐换延迟"。RT 适合"宁可平均慢一点，也要最坏延迟可控（jitter 小）"的硬实时负载。

### 3.6 内核抢占计数器（preempt_count）位域：逐字源码

这是 v6.12 `include/linux/preempt.h` 的**逐字注释**（背下来，面试直接默写）：

```c
/* 【真实源码 v6.12 include/linux/preempt.h】 */
/*
 * We put the hardirq and softirq counter into the preemption
 * counter. The bitmask has the following meaning:
 *
 * - bits 0-7 are the preemption count (max preemption depth: 256)
 * - bits 8-15 are the softirq count (max # of softirqs: 256)
 *
 * The hardirq count could in theory be the same as the number of
 * interrupts in the system, but we run all interrupt handlers with
 * interrupts disabled, so we cannot have nesting interrupts. Though
 * there are a few palaeontologic drivers which reenable interrupts in
 * the handler, so we need more than one bit here.
 *
 *         PREEMPT_MASK:	0x000000ff
 *         SOFTIRQ_MASK:	0x0000ff00
 *         HARDIRQ_MASK:	0x000f0000
 *             NMI_MASK:	0x00f00000
 * PREEMPT_NEED_RESCHED:	0x80000000
 */
```

配套的 bit 宽度定义（逐字）：

```c
/* 【真实源码 v6.12 include/linux/preempt.h】 */
#define PREEMPT_BITS	8
#define SOFTIRQ_BITS	8
#define HARDIRQ_BITS	4
#define NMI_BITS	4

#define PREEMPT_SHIFT	0
#define SOFTIRQ_SHIFT	(PREEMPT_SHIFT + PREEMPT_BITS)
#define HARDIRQ_SHIFT	(SOFTIRQ_SHIFT + SOFTIRQ_BITS)
#define NMI_SHIFT	(HARDIRQ_SHIFT + HARDIRQ_BITS)
```

```
preempt_count 位域（bit 从低到高）：

  bit:  31      23..20    19..16    15..8     7..0
       ┌──────┬────────┬─────────┬─────────┬─────────┐
       │ NEED │  NMI   │ hardirq │ softirq │ preempt │
       │RESCHED│(4bit) │ (4bit)  │ (8bit)  │ (8bit)  │
       └──────┴────────┴─────────┴─────────┴─────────┘
        bit31  高位 ←──────────────────────────→ 低位

  preempt_count == 0 → 可以调度
  preempt_count  > 0 → 不能调度（在某种原子上下文中）
```

逐行解读那段注释里的设计决策（这就是"为什么这样设计"）：
- **"max preemption depth: 256"（8 bit）**：preempt_disable 的嵌套可能很深——库代码 A 调 B 调 C，每层各自 `preempt_disable`/`enable`，层层叠加。8 bit（256 层）给足余量。
- **softirq 8 bit**：同理，加上 §3.3.1 的 1×/2× 编码，需要的位更多。
- **hardirq 只 4 bit**：注释自己解释了——"we run all interrupt handlers with interrupts disabled, so we cannot have nesting interrupts"，硬中断默认关中断执行、不嵌套，理论上 1 bit 都够；**但注释又自嘲**"there are a few **palaeontologic**（古生物级的）drivers which reenable interrupts in the handler"——有些老掉牙的驱动会在处理程序里重开中断，所以留 4 bit 防它们。这句注释是内核幽默感的经典样本，也是"代码考古"的活化石。
- **`PREEMPT_NEED_RESCHED`（bit 31）**：x86 的小技巧——把"需要重新调度"取反编进 preempt_count 最高位，于是 `preempt_count == 0`（含该位）就能一条比较同时判断"可抢占 且 有抢占请求"，省一次内存读。这是 §2.6.2 per-CPU 化之后顺手做的微优化。

`preempt_count` 的增减由这两个函数实际执行（带 DEBUG 校验，逐字）：

```c
/* 【真实源码 v6.12 kernel/sched/core.c:5785】 */
void preempt_count_add(int val)
{
#ifdef CONFIG_DEBUG_PREEMPT
	if (DEBUG_LOCKS_WARN_ON((preempt_count() < 0)))   /* 下溢检测 */
		return;
#endif
	__preempt_count_add(val);
#ifdef CONFIG_DEBUG_PREEMPT
	DEBUG_LOCKS_WARN_ON((preempt_count() & PREEMPT_MASK) >=
				PREEMPT_MASK - 10);                /* 抢占层快溢出预警（剩 10 层报） */
#endif
	preempt_latency_start(val);                       /* 抢占延迟统计起点 */
}
EXPORT_SYMBOL(preempt_count_add);
NOKPROBE_SYMBOL(preempt_count_add);   /* 禁 kprobe：自己被探测会无限递归 */
```

逐行价值：`PREEMPT_MASK - 10` 那行——内核在抢占嵌套快用满 256 层时**提前 10 层报警**，因为一旦溢出会污染相邻的 softirq 位，制造灾难性的上下文误判。`NOKPROBE_SYMBOL` 是因为 kprobe 自己会动 preempt_count，探测这个函数会递归爆栈。

补一句源码：`preempt_disable()` 增加计数，`spin_lock()` 内部也调 `preempt_disable()`，而 `local_irq_disable()` **不增加** preempt_count——但禁中断本身阻止调度（调度依赖 timer 中断；timer 都关了，调度器没机会跑）。所以"关中断"和"禁抢占"是**两个正交的维度**，前者不抬 preempt_count 却同样达到"当前 CPU 不会被切走"的效果。

## 4. 落地：生产环境中的并发排障

### 4.1 工具链

**lockdep（Lock Dependency Validator）**（`CONFIG_LOCKDEP`）：跟踪每把锁的获取顺序，构建有向图，检测潜在死锁（循环依赖）。

```
[  123.456789] WARNING: possible circular locking dependency detected
[  123.456790] task/1234 is trying to acquire lock: ... (lockA) ...
[  123.456791] but task holds lock: ... (lockB) ...
[  123.456792] which lock already depends on the new lock.
```

lockdep 不只报实际死锁，还报"潜在死锁路径"——即使本次运行没触发，只要依赖图有环就报。是 code review 重要辅助。

**KCSAN（Kernel Concurrency Sanitizer）**（`CONFIG_KCSAN`，合入 5.8，2020）：专检 data race，编译器插桩记录每次内存访问，检测同一地址的并发 non-atomic 读写。

```
BUG: KCSAN: data-race in function_a / function_b
write to 0xffff... of 8 bytes by task 123: function_a+0x10/0x40
read  to 0xffff... of 8 bytes by task 456: function_b+0x20/0x50
```

KCSAN 是目前内核里发现 data race 最有效的工具，已挖出大量历史遗留的"technically racy but never triggered"代码。

**KASAN（Kernel Address Sanitizer）**：检测越界和 use-after-free，不直接检测竞态，但并发 bug 常以 UAF 形式暴露。

### 4.2 `scheduling while atomic` 排障（接 §3.2 源码）

```
BUG: scheduling while atomic: process/1234/0x00000002
```

排障步骤（现在你知道每一步对应 §3.2 哪行源码）：
1. **解码那个十六进制** `0x00000002`——那是 `__schedule_bug` 里 `printk("...0x%08x", preempt_count())` 打的 raw preempt_count。按 §3.6 位域拆：`0x02` = preempt 层 = 2，说明嵌了两层 `preempt_disable`/spinlock，没有 hardirq/softirq 位 → 大概率是"持 spinlock 时调了可睡眠函数"。
2. **看 `Preemption disabled at:` 那行**（需 `CONFIG_DEBUG_PREEMPT`）——`__schedule_bug` 专门打了它，直接指向抢占在哪行被禁的，即罪犯源码位置。
3. **看 `debug_show_held_locks` 列出的锁** + 调用栈，找在原子上下文里调了哪个可睡眠函数。常见罪犯：`kmalloc(GFP_KERNEL)` 在中断里、持 spinlock 时 `mutex_lock()`、用错工作队列（需 sleepable 却用普通）。

### 4.3 per-CPU 变量的正确用法

per-CPU 变量（`DEFINE_PER_CPU`）每 CPU 一份副本，消除 SMP 并发，但同一 CPU 上的中断/抢占仍可打断。官方 `Documentation/locking/preempt-locking.rst`（Robert Love 著）的 **RULE #1** 逐字精神（实际取得）："Per-CPU data structures need explicit protection"——抢占后重新调度时"the previous value of smp_processor_id may not equal the current. You must protect these situations by disabling preemption around them."

```c
/* ✗ 错误：进程上下文里裸用 this_cpu_ptr 又不禁抢占
 *   取到指针后、用之前可能被抢占并迁移到另一 CPU，
 *   后续操作打在"另一个 CPU 的副本"上 —— 数据归错人（示意代码，非逐字） */
ptr = this_cpu_ptr(&my_per_cpu);   /* 仅在已禁抢占/中断上下文才安全 */
do_stuff(ptr);                     /* ← 这中间若被抢占迁移就错了 */

/* ✓ 正确：get_cpu_var / put_cpu_var 成对，临界期间钉在本 CPU */
my_var = get_cpu_var(my_per_cpu);  /* 内部 preempt_disable，绑定当前 CPU */
/* ... 使用 my_var ... */
put_cpu_var(my_per_cpu);           /* 内部 preempt_enable */

/* 单次原子读写：this_cpu_inc 自身保证"针对当前 CPU 副本的这一次操作"
 *   不被同 CPU 抢占割裂，是最便宜的用法 */
this_cpu_inc(my_counter);
```

要点：`get_cpu_var()` 禁抢占保证"整段临界区拿到的副本都属于同一（当前）CPU"。`this_cpu_inc()` 这类**单条**操作自带"对本 CPU 副本的 RMW 不被同 CPU 抢占打断"保证，不需手动禁抢占——但只覆盖那一条，跨多步访问仍要 `get_cpu_var` 或显式 `preempt_disable`。

#### 4.3.1 `preempt_disable()` vs `migrate_disable()`：PREEMPT_RT 带来的分裂（源码级）

这是近年（5.x 后）staff 级高频考点，根因正是 §2.6.3 的 PREEMPT_RT。传统上保护 per-CPU 数据用 `preempt_disable()`，它把两个副作用捆在一起：(1) 当前任务不会被抢占；(2) 因此也不会被迁移到别的 CPU。**第二点才是 per-CPU 正确性真正需要的**——你只想"钉在这个 CPU 上"，不一定需要"完全不让别人跑"。

在 RT 上，`preempt_disable()` 这种"长时间禁抢占"直接破坏实时性。于是引入 `migrate_disable()`——看 v6.12 源码（逐字）：

```c
/* 【真实源码 v6.12 kernel/sched/core.c:2313】 */
void migrate_disable(void)
{
	struct task_struct *p = current;

	if (p->migration_disabled) {            /* 已禁过 → 纯计数嵌套，秒返回 */
#ifdef CONFIG_DEBUG_PREEMPT
		WARN_ON_ONCE((s16)p->migration_disabled < 0);
#endif
		p->migration_disabled++;
		return;
	}

	guard(preempt)();                       /* 仅在"设置标志"这一瞬间禁抢占 */
	this_rq()->nr_pinned++;                 /* 告诉负载均衡器：本 rq 多了个钉住的任务 */
	p->migration_disabled = 1;
}
EXPORT_SYMBOL_GPL(migrate_disable);
```

逐行注解，对比 `preempt_disable` 的本质差别：
- `migrate_disable` **只在 `guard(preempt)()` 那一瞬间禁抢占**（设置标志期间），设完就退出禁抢占域——**之后任务仍可被抢占**，只是调度器看到 `migration_disabled` 就不会把它迁到别的 CPU。
- `this_rq()->nr_pinned++` 是关键代价：它给**负载均衡器**引入一个"钉在这个 CPU、但仍可抢占"的约束。负载均衡想搬走这个 CPU 上的任务时，得绕开被钉住的——这正是它"不免费"的根因。
- 对比 `preempt_disable()`（§3.4 那段源码）：后者直接 `preempt_count_inc()` 把整个 CPU 的抢占都关了。`migrate_disable` 不动 preempt_count，只设 per-task 标志。

配套的是 **`local_lock`** 这一族原语：非 RT 内核上 `local_lock` 退化成 `preempt_disable`/`local_irq_save`；RT 上变成一把 per-CPU 睡眠锁 + `migrate_disable`，既保护 per-CPU 数据，又不长时间堵死抢占。§3.3.1 看到的 RT softirq 实现里那个 `struct softirq_ctrl { local_lock_t lock; int cnt; }` 就是活例子。

> 💡 **面试一句话**：`preempt_disable` = 不让出 CPU（顺带不迁移）；`migrate_disable` = 只是不迁移、仍可被抢占。RT 内核大量把"为保护 per-CPU 而禁抢占"重写成 `local_lock`/`migrate_disable`，是 RT 化的核心套路之一。
>
> ⚠️ `migrate_disable()` 不免费（上面 `nr_pinned++` 就是证据）：它给调度器和负载均衡引入"钉住但可抢占"的额外约束，早期曾有关于它影响调度延迟、被滥用的争论。不是 `preempt_disable` 的无脑替代，只在确需时用。

### 4.4 大厂实践：并发 bug 的 code review 清单

- [ ] 函数在中断上下文可能被调用吗？如果是，有没有用 `GFP_ATOMIC`？有没有调会 `might_sleep()` 的函数？
- [ ] spinlock 保护了所有访问该字段的路径（**包括读路径**）吗？
- [ ] 持 spinlock 期间有没有调 mutex 或其他可睡眠原语？
- [ ] 对共享指针/标量的读写用了 `READ_ONCE`/`WRITE_ONCE` 或更强的 acquire/release 吗？
- [ ] 位操作用的是原子版本（`set_bit` 而非 `|=`）吗？
- [ ] per-CPU 变量访问期间禁了抢占（或用了 `local_lock`）吗？RT 友好吗？
- [ ] 屏障**成对**了吗（写屏障配读屏障）？孤立屏障几乎总是 bug。
- [ ] RCU 保护的数据，写路径等了 grace period 再 free 吗？（见第 13 章）

## 5. 面试视角

### 5.1 Staff 级高频题

**Q1：内核并发有哪几个来源？分别对应什么保护原语？**
- SMP 并行 → spinlock（不可睡眠路径）/ mutex（可睡眠路径）
- 内核抢占 → `preempt_disable()` / spinlock（持锁自动禁抢占）
- 硬中断 → `spin_lock_irqsave()` / `local_irq_disable()`
- 软中断 → `spin_lock_bh()`
- 陷阱：只说"用锁"不区分来源 → 直接失分。加分：指出三者状态全编码进 `preempt_count`。

**Q2：为什么中断处理程序不能睡眠？（源码级答法）**
1. 中断处理程序借被打断进程的栈、没有自己的 `task_struct`，不是能放回 runqueue 再唤醒的执行单元。
2. `schedule()` 假设调用者是 task；中断上下文这个假设失效，返回路径未定义。
3. 真正的拦截点：`__schedule()` → `schedule_debug()` → `in_atomic_preempt_off()` 命中 → `__schedule_bug()` 打 `BUG: scheduling while atomic`（`kernel/sched/core.c:5903/5856`）。是否 panic 取决于 `panic_on_warn`/`panic_on_oops`。
- 加分1：`__schedule_bug` 会打 `Preemption disabled at:` 直指罪犯行（需 `CONFIG_DEBUG_PREEMPT`）。
- 加分2：更早防线是 `CONFIG_DEBUG_ATOMIC_SLEEP`，让 `might_sleep()` 在调用点就报警。
- 反向陷阱：能否在中断里"调一个会 schedule 的函数但传 nonblocking 标志"？能——`kmalloc(GFP_ATOMIC)`、`spin_trylock`、`mutex_trylock`，关键是这些路径**保证不进 `schedule()`**。区分"函数有时会睡"和"这次调用一定不睡"是核心功力。

**Q3：`spin_lock()` 和 `spin_lock_irqsave()` 的区别？何时用哪个？**
- `spin_lock()` 禁抢占不禁中断。若持锁路径和中断处理程序都访问同一数据，中断可打断你（同 CPU）→ 中断里再次拿同一锁 → 死锁。
- `spin_lock_irqsave()` 同时禁本 CPU 中断防上述死锁；`irqsave` 保存 flags 是为了正确恢复（调用者可能本来就禁了中断）。
- 原则：数据只被进程上下文访问 → `spin_lock()`；也被中断处理程序访问 → `spin_lock_irqsave()`。

**Q4：KCSAN vs lockdep？**
- lockdep：检测锁获取顺序依赖图，找潜在死锁。
- KCSAN：检测同一内存地址的并发非原子访问（data race），与锁无关。
- 区别：lockdep 干净的代码仍可能有 data race（如用了 `READ_ONCE` 但漏了内存序）。

**Q5：`READ_ONCE` vs `volatile`？**
- `volatile`：只防编译器优化掉读写；无原子性、无屏障。
- `READ_ONCE`：实质 `*(volatile typeof(x) *)&x`，语义是"有意的并发读"；配 `WRITE_ONCE` 构成 plain 并发访问的最小防护。
- 更强需求用 `smp_load_acquire`/`smp_store_release`。

**Q6：内核抢占的几个级别 + PREEMPT_RT 的核心代价？外加 2024 新方向？**
- 参见 §2.4 表 + §2.6.4。RT 代价 = 锁竞争时上下文切换开销 + 优先级继承复杂性；换确定性延迟（jitter 小）。
- **王炸加分**：2024 起 lazy preemption（`TIF_NEED_RESCHED_LAZY`，6.13）正把 NONE/VOLUNTARY/FULL 统一成运行时可切并删 `cond_resched()`，机制借自 RT。

**Q7：per-CPU 变量为什么不能消除所有并发？外加 RT 视角？**
- 消除了跨 CPU 竞争；但同 CPU 上中断可打断（需禁中断/原子操作），抢占后可迁移（需 `get_cpu_var` 禁抢占）。
- 加分：RT 上不用 `preempt_disable` 而用 `local_lock`/`migrate_disable`——只钉 CPU 不堵抢占。

**Q8：preempt_count 为什么把 preempt/softirq/hardirq 编码在同一字段？**
- 统一判"是否原子上下文"：`preempt_count != 0` 一个检查。
- 嵌套安全：各层独立计数，正确恢复。
- 性能：单整数读、零分支（§2.6.2 还把它 per-CPU 化 + bit31 编 NEED_RESCHED 进一步省访存）。
- 深水区：hardirq 只 4 bit 是因为中断默认不嵌套，留 4 bit 防"古生物级"重开中断的老驱动（注释原文 palaeontologic）。

**Q9：data race 一定是 bug 吗？内核里有"故意的 data race"吗？**
- C11/C++11 认为 data race 是 UB。
- 内核有少量"故意 racy read"（不加锁读、仅用于 heuristic/debug），用 `data_race()` 宏标注，告诉 KCSAN"已知且故意"。
- 加分（撕裂）：64-bit 值在 32-bit 架构上可能拆成两次 store/load → 读者看到"一半旧一半新"。`READ_ONCE`/`WRITE_ONCE` 能**阻止编译器主动制造撕裂**（拆分/合并/重发），但**不能把超过机器字长的访问变成单条原子指令**——32-bit 上对 64-bit 用 `WRITE_ONCE` 仍可能撕裂。"单次访问原子"前提：类型 ≤ 机器字长且自然对齐。x86-64 上对齐 8 字节访问本身原子（这是大量"无锁读指针/long"在 x86 能跑的原因），但**对齐保证原子性不保证内存序**，跨变量可见性顺序仍要 acquire/release。

**Q10：如何分析一段内核代码是否有竞态？（方法论）**
1. 枚举所有到达该代码的执行流（进程/kthread/中断/软中断）。
2. 标出共享可变状态。
3. 对每对执行流构造并发序列，看能否产生不一致。
4. 工具：KCSAN 运行时检测、lockdep 依赖图。

### 6.2 实验环境（示意，按本机调整）

### 6.3 分析练习：一段带竞态的伪代码（示意代码，非逐字，不可直接用）

```c
static struct my_obj *global_obj = NULL;
static int obj_refcount = 0;

/* 路径 A：进程上下文 */
void use_object(void)
{
    if (global_obj != NULL) {         /* ← 竞态点 1 */
        obj_refcount++;               /* ← 竞态点 2：非原子 RMW */
        do_work(global_obj);
        obj_refcount--;               /* ← 竞态点 3 */
    }
}

/* 路径 B：另一个进程上下文 */
void destroy_object(void)
{
    if (obj_refcount == 0) {          /* ← 竞态点 4：check-then-act */
        kfree(global_obj);            /* ← 与竞态点 1 交叉则 UAF */
        global_obj = NULL;
    }
}
```

**手工分析**：
1. 竞态点 1+4：A 读 `global_obj != NULL` 为真后、调 `do_work` 前，B 可能 `kfree` 它 → use-after-free。
2. 竞态点 2：`obj_refcount++` 非原子（load→add→store），两 CPU 同跑 → lost update，导致"明明有人在用却被判 refcount==0 释放"。
3. 竞态点 1 的指针读没有 `READ_ONCE`，编译器可能多次读：第一次非 NULL 进 if，体内某次读可能已是 NULL → NULL deref。

**正确修法（思路）**：`kref` + 原子 refcount API；指针读用 RCU（见第 13 章）或 spinlock；`obj_refcount` 用 `atomic_t`。

## 7. 设计题

**题目**：设计一个内核对象缓存（Object Cache），对象被进程上下文高频读取、中断处理程序偶尔更新。要求：(a) 零死锁；(b) 读路径不阻塞中断；(c) 写路径更新对读路径完整可见（不能读到半更新）。

**考点**：
1. 是否识别出"读路径在进程 + 中断上下文都可能发生"这一关键约束。
2. 是否自然引向 RCU（读路径 `rcu_read_lock`，写路径 `synchronize_rcu` 后 `kfree`）。
3. 能否分析 RCU grace period 在中断上下文的含义：中断不能 sleep、不能调 `synchronize_rcu`；可用 `call_rcu` 异步回调。
4. 加分：SRCU（Sleepable RCU）vs 普通 RCU 的适用场景；以及"半更新可见"如何用 `rcu_assign_pointer`（内含 release 屏障）+ `rcu_dereference`（内含 dependency-ordering）保证发布原子性——回扣 §3.4 的 acquire/release。

## 8. 系统题：排障场景

**场景**：生产内核 (x86_64, SMP, PREEMPT_FULL) 偶发 `BUG: scheduling while atomic`，调用栈显示在网卡 RX 软中断里，深处调了某三方厂商驱动的 API，该 API 内部用 `GFP_KERNEL` 分配内存。

**问题与参考答案框架**：
1. **为何触发**：软中断是中断上下文（preempt_count 的 softirq 位非零），`GFP_KERNEL` 在内存压力下触发 direct reclaim → 调 `schedule()` 等内存释放 → `schedule_debug()` 的 `in_atomic_preempt_off()` 命中 → `__schedule_bug`。
2. **临时规避（不改三方驱动）**：把三方 API 调用从软中断路径移出，丢进 workqueue；软中断只做入队，workqueue 在进程上下文执行、允许 sleep。
3. **根本修法**：三方驱动改用 `GFP_ATOMIC`（从原子池分配、失败返回 NULL）；或重新设计，在非原子上下文预分配。
4. **复现与验证**：staging 内核开 `CONFIG_LOCKDEP` + `CONFIG_DEBUG_ATOMIC_SLEEP`（后者在调用点立刻报警，不必等真 schedule）；压测制造内存压力触发 reclaim 路径；从 `printk` 里那个 raw `preempt_count` 十六进制确认是 softirq 位非零（§3.2/§4.2）。

**量级参考**：软中断处理 RX 包的上下文切换开销 < 1μs（现代 x86），而 `GFP_KERNEL` 在内存压力下可能等待数毫秒 → 数量级差异，不能在软中断里用。

## 9. 代码题：内核模块验证 preempt_count

**可运行**骨架，在不同上下文打印 `preempt_count`，直观理解原子层次。**需在 QEMU 或 test kernel 运行，勿在生产机加载未验证模块。**

```c
// preempt_demo.c — 示意内核模块，演示 preempt_count 层次（示意代码，非逐字）
#include <linux/module.h>
#include <linux/kernel.h>
#include <linux/init.h>
#include <linux/spinlock.h>
#include <linux/preempt.h>   /* preempt_count() */

MODULE_LICENSE("GPL");
MODULE_DESCRIPTION("preempt_count visualization");

static DEFINE_SPINLOCK(demo_lock);

static int __init preempt_demo_init(void)
{
    pr_info("[demo] normal: pc=0x%08x (preempt层=%lu)\n",
            preempt_count(), preempt_count() & PREEMPT_MASK);   // 期望 0

    preempt_disable();
    pr_info("[demo] after preempt_disable: pc=0x%08x\n", preempt_count()); // 期望 preempt层=1
    preempt_enable();

    spin_lock(&demo_lock);
    pr_info("[demo] holding spinlock: pc=0x%08x\n", preempt_count()); // 期望 preempt层=1
    spin_unlock(&demo_lock);

    // TODO（练习）：
    // 1. local_irq_disable() 前后打印 preempt_count，观察是否变化
    //    （答：不变——禁中断不抬 preempt_count，但禁中断本身阻止调度）
    // 2. 嵌套两层 preempt_disable，观察计数变成 2
    // 3. 用 in_atomic() 检查；并思考为何 PREEMPT_NONE 下持锁时它可能仍是 0（§3.1 陷阱）
    return 0;
}
static void __exit preempt_demo_exit(void) { pr_info("[demo] unloaded\n"); }
module_init(preempt_demo_init);
module_exit(preempt_demo_exit);
```

**Makefile（示意）**：
```makefile
obj-m += preempt_demo.o
KDIR ?= /lib/modules/$(shell uname -r)/build
all:
	$(MAKE) -C $(KDIR) M=$(PWD) modules
clean:
	$(MAKE) -C $(KDIR) M=$(PWD) clean
```

**验证**：`make && sudo insmod preempt_demo.ko && dmesg | grep '\[demo\]' && sudo rmmod preempt_demo`

**暗坑**：
- `preempt_count()` 返回值**编码了多个字段**（§3.6），直接打印不是"层数"。提取纯 preempt 层用 `preempt_count() & PREEMPT_MASK`（`PREEMPT_MASK = 0x000000ff`，逐字见 §3.6）。本 demo 只动 preempt 层，所以巧合等于层数——别据此以为 `preempt_count()` 总是层数。
- `PREEMPT_NONE` 内核上 `spin_lock()` 的 `preempt_disable` 被优化掉，持锁时 preempt_count **不变**（§3.1 `in_atomic()` 陷阱的实验复现）。
- UP（单处理器）内核上 spinlock 是 no-op，行为又不同。
- `PREEMPT_RT` 内核上 spinlock 是 RT mutex，`spin_lock()` 时 preempt_count 行为与 §3.5 描述不同（RT 下可被抢占）。

## 章末心法

> **在内核里，正确的第一步不是"加把锁"，而是先问：这段代码可以被谁、在什么上下文、以什么顺序打断？答案决定用什么原语，甚至决定要不要重新设计数据路径。而"谁能打断、在什么上下文"这两个问题的答案，内核全部压进了一个叫 `preempt_count` 的整数——读懂它，你就读懂了内核并发模型的骨架。**
