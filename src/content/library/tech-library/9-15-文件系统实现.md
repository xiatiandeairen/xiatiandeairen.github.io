> **一句话定位**：磁盘文件系统是持久化存储的软件契约——它把"可能随时断电的块设备"包装成"崩溃后仍然自洽的命名空间"。ext4 是理解 Linux FS 实现的最佳教科书：它身上叠了 25 年的演进地层（ext2→3→4），每一层都对应一场真实的社区争论和一组可在 git 历史里追到的 commit。本章不只讲"它怎么工作"，更要带你走一遍"它为什么长成这样、当年争论过什么、源码里到底写了什么"。

---

## TL;DR（本章最硬结论）

1. **崩溃一致性是文件系统的根本难题，不是性能优化的附属品**。fsync 的开销、journal 的设计、COW FS 的出现，全部围绕"断电后磁盘上的数据结构不能是半截状态"这一核心展开。读懂这一条，后面所有机制都是它的推论。
2. **extent 替代 indirect block 是 ext4 最大的工程飞跃，而且它是被"文件越来越大"逼出来的**。ext3 的 15 指针结构（12 direct + 3 间接）在 4TB 文件上彻底崩溃；Alex Tomas 在 2003–2006 年（为 Lustre）做的 extent + 48-bit 块号补丁，最终成了 ext3 必须 fork 成 ext4 的直接导火索（[LWN 187321](https://lwn.net/Articles/187321/)、[LWN 194869](https://lwn.net/Articles/194869/)）。
3. **fsync 慢不是 bug，是 journal commit + 设备 cache flush 的代价**。`ext4_sync_file()` 的真实源码（v6.12）清楚地把它拆成"刷脏页 → commit transaction → `blkdev_issue_flush`"三步；fast commit（Harshad Shirwadkar，5.10）就是为了绕开第二步的"全局连坐"而生。
4. **2009 年的"ext4 丢配置文件"事件是一堂活的工程伦理课**。延迟分配把 ext3 时代"意外获得"的崩溃安全窗口从 5 秒拉到 30–60 秒，引爆 KDE/Ubuntu 社区论战；Ted Ts'o 坚持"POSIX 从没保证过崩溃安全"，但还是加了 `auto_da_alloc` 给"懒程序"兜底（[LWN 322823](https://lwn.net/Articles/322823/)）。
5. **btrfs 的 COW 和 ext4 的 journal 是两条不同的一致性路线，不是谁更好**。各有 trade-off。而 bcachefs 在 2024–2025 年"合入主线又被踢出"（6.7 进、6.18 出），则是"代码能 merge ≠ 能长期留在主线"的经典治理案例（[LWN 1040120](https://lwn.net/Articles/1040120/)）。

## 15.0 阅读地图：本章的"地层"结构

文件系统这门学问最容易"浮在表面"，是因为大多数材料把 ext4 当成一个静态的、设计好的系统来讲。但 ext4 真实的样子是**地质沉积**——

```
            真实的 ext4 = 25 年沉积的工程地层

  ┌───────────────────────────────────────────────────────┐
  │  v6.x 现代层：fast commit / metadata_csum / bigalloc   │  ← 2020s
  ├───────────────────────────────────────────────────────┤
  │  ext4 层：extent / delayed alloc / mballoc / 48-bit    │  ← 2006–2008
  ├───────────────────────────────────────────────────────┤
  │  ext3 层：jbd journal / data=ordered / htree           │  ← 1999–2001
  ├───────────────────────────────────────────────────────┤
  │  ext2 层：block group / inode / bitmap / 间接块         │  ← 1993
  └───────────────────────────────────────────────────────┘
   你今天 mount 的 ext4，每一层都还在，靠 feature flag 决定哪些激活
```

本章按"先讲每层是被什么缺陷逼出来的（考古），再精读它在 v6.12 源码里长什么样（源码），最后对比它和替代方案在具体场景下的差异（推演）"来组织。**所有源码块都标注了来源等级**：

## 15.1 背景：磁盘文件系统要解决什么

### 15.1.1 三大本质挑战

磁盘文件系统不是"把文件存到块设备上"这么简单，它真正要解决的是三个互相拉扯的工程问题：

```
┌─────────────────────────────────────────────────────────┐
│              磁盘文件系统的三角张力                       │
│                                                          │
│   持久化（Durability）                                    │
│        ╱                                                 │
│       ╱   崩溃一致性（Crash Consistency）                 │
│      ╱           ╲                                       │
│  性能（Performance）─────────────────                    │
│                                                          │
│  三者不可兼得，所有 FS 设计本质上是在这个三角里选位置      │
└─────────────────────────────────────────────────────────┘
```

**持久化**：`write()` 返回后，数据什么时候真正落盘？这里有三层"已写入"——用户态进程的 libc buffer、内核 page cache、块设备内部的 write cache（DRAM/SLC cache）。每一层都可能在断电时丢数据。`write()` 返回只保证数据到了 page cache，**离持久介质还隔着两层**。

**崩溃一致性**：在任意时刻断电，磁盘上的文件系统元数据（inode、目录项、bitmap）不能处于半写入状态。这是最硬的约束——因为元数据是"描述其它数据在哪"的数据，元数据坏一个块可能让整盘失联。

**性能**：顺序写可以达到磁盘带宽上限，随机写在 HDD 上会烂掉（每次都要寻道 + 旋转延迟），在 SSD 上也比顺序写差（FTL 的写放大、GC）。FS 的分配策略、预读策略、日志策略全部影响性能。

> **底层根因（为什么这三者天然冲突）**：崩溃一致性要求"写有顺序"（先 A 后 B，否则崩在中间不一致），而性能要求"写无顺序"（让调度器自由重排、合并、并行）。**强制顺序 = 牺牲并行 = 损失带宽**。整个文件系统史就是在"用最少的顺序约束换取足够的一致性"。journal 是一种答案，COW 是另一种，下面会逐个拆。

### 15.1.2 类比：数据库 WAL 与文件系统 Journal——同构不是巧合

后端工程师最熟悉的类比是数据库的 WAL（Write-Ahead Log）：先写日志，再写数据页，crash 后 replay 日志恢复一致性。**ext3/ext4 的 journal 和数据库 WAL 几乎同构，这不是巧合**——两者解决的是同一个数学问题：如何用一个 append-only 的、能原子提交的副本，把"对原地数据的多步修改"变成"要么全发生要么全不发生"。

这个等价不是泛泛而谈。Stephen Tweedie 在 2000 年渥太华 Linux 研讨会（OLS 2000）介绍 ext3 设计时，明确把它讲成"对文件系统元数据做事务日志"（演讲转录：[olstrans.sourceforge.net/release/OLS2000-ext3](https://olstrans.sourceforge.net/release/OLS2000-ext3/OLS2000-ext3.html)）。他选择"只记元数据"而不是"记所有数据"，本质上和数据库选择"只 WAL 索引/系统表、用 doublewrite buffer 保数据页"是同一类工程权衡。

关键区别：
- 数据库 WAL 知道"事务边界"（`BEGIN`/`COMMIT`），所以能做应用语义的原子性；文件系统 journal 不知道应用的事务边界，只能保证**单个文件系统操作**（一次 `rename`、一次 `create`）的原子性。这条区别是后面"为什么 POSIX 不保证跨文件原子"的根源。
- 数据库 WAL 是应用层协议，可以自定义 recovery 语义；文件系统 journal 是 FS 层协议，recovery 必须在 mount 时无人值守地跑完。

## 15.2 演进：ext2 → ext3 → ext4（设计考古）

这一节是本次深化的重点之一。每一步演进都对应**具体的人、具体的年份、具体的前代缺陷、具体的社区争论**。

### 15.2.1 ext2（1993）：经典 block-based FS，没有 journal

ext2 是理解所有后继者的基础。磁盘布局：

```
磁盘布局（ext2/3/4 共同祖先，feature flag 决定上层激活什么）
┌─────────────────────────────────────────────────────────┐
│ Boot  │ Block Group 0 │ Block Group 1 │ ... │ BG N        │
└─────────────────────────────────────────────────────────┘

Block Group 内部：
┌──────────┬──────────┬──────────┬──────────┬────────────┐
│Superblock│GDT       │Block     │Inode     │Data Blocks │
│(备份)     │备份       │Bitmap    │Table     │            │
└──────────┴──────────┴──────────┴──────────┴────────────┘
  GDT = Group Descriptor Table
  把盘切成 group 的根因：让 inode 和它的数据块物理上靠近，减少 HDD 寻道
```

**ext2 inode 的数据块索引——15 指针结构**。这个结构是后面一切的"原罪"，必须看清楚它在 v6.12 源码里到底怎么定义（ext4 为了兼容仍保留这套常量）：

```c
/*
 * Constants relative to the data blocks
 */
#define	EXT4_NDIR_BLOCKS		12
#define	EXT4_IND_BLOCK			EXT4_NDIR_BLOCKS
#define	EXT4_DIND_BLOCK			(EXT4_IND_BLOCK + 1)
#define	EXT4_TIND_BLOCK			(EXT4_DIND_BLOCK + 1)
#define	EXT4_N_BLOCKS			(EXT4_TIND_BLOCK + 1)
```
【真实源码 v6.12 `fs/ext4/ext4.h`】

逐行读：
- `EXT4_NDIR_BLOCKS = 12`：前 12 个指针直接指向数据块（direct）。4K block 下覆盖前 48KB——**绝大多数小文件就靠这 12 个指针，零间接块开销**。这是个精心选的常数：小到不浪费 inode 空间，大到覆盖了系统里海量的小文件（配置、源码、日志行）。
- `EXT4_IND_BLOCK = 12`：第 12 个指针指向一个"间接块"（indirect），里面装满块号。4K block / 4 字节一个块号 = 1024 个块号 = 再覆盖 4MB。
- `EXT4_DIND_BLOCK = 13`：双重间接，1024×1024 个块号 = 4GB。
- `EXT4_TIND_BLOCK = 14`：三重间接，1024³ 个块号 ≈ 4TB。
- `EXT4_N_BLOCKS = 15`：总共 15 个指针，存在 inode 的 `i_block[]` 数组里。

**这就是 ext3 的 4TB 单文件上限的来源**——三重间接也只能寻址到 4TB（[LWN 187321](https://lwn.net/Articles/187321/) 明确把"inode 的 15 指针结构对大文件越来越低效"列为 ext3 必须扩展的两大原因之一）。

**ext2 的根本缺陷：没有 journal**。创建一个文件需要改 4 处磁盘结构：inode bitmap（标记 inode 已用）+ block bitmap（标记数据块已用）+ 目录 entry（新增文件名→inode 映射）+ inode 本身（写入元数据）。任何一步之后断电就是不一致。`fsck` 能修，但 `fsck` 要扫整盘——TB 级盘上要扫几十分钟到几小时，这在生产上不可接受。**正是这个"fsck 扫全盘"的痛，催生了 ext3。**

### 15.2.2 ext3（1999 原型，2001 进主线）：Stephen Tweedie 加 journal

**史料锚点**（来自 [Wikipedia ext3](https://en.wikipedia.org/wiki/Ext3) 与 OLS 2000 转录交叉印证）：
- **谁**：Stephen Tweedie（Red Hat），主导设计；移植到 2.4 的有 Peter Braam、Andreas Dilger、Andrew Morton、Alexander Viro、Ted Ts'o。
- **何时**：1999 年开始原型（2.2 内核分支），**2001-11-23 随 Linux 2.4.15 首次稳定发布**。Red Hat 7.2（2001-10）把它设为默认 FS。
- **为什么**：消除 ext2 的"崩溃后 fsck 扫全盘"。journal 让 recovery 从"扫整盘"降到"replay 一小段日志"——几秒钟搞定。

ext3 在 ext2 上**原地**加了一个 journal inode（通常 inode 号 8），磁盘布局完全兼容（这是关键卖点：ext2 可以不 dump-restore 直接升级 ext3）。引入三种 journaling 模式：

| 模式 | 日志内容 | 性能 | 一致性强度 | 真实风险 |
|------|---------|------|-----------|---------|
| `data=writeback` | 只记元数据，数据落盘无序 | 最快 | 元数据一致；数据不保证 | crash 后文件尾部可能是**旧的垃圾数据**（size 已更新，数据块还没写）——可能泄露其他文件的内容，有安全隐患 |
| `data=ordered`（默认）| 只记元数据，但强制数据块先于元数据 commit 落盘 | 中 | 数据一致性有保证 | 见 15.2.3 的延迟分配窗口问题 |
| `data=journal` | 元数据 + 数据都写 journal | 最慢（数据写两遍，写放大 2x） | 最强 | 罕用；但对 fsync 密集的小写入，有时反而更快（journal 是顺序写） |

> **`data=ordered` 的精妙之处（也是它存在的全部理由）**：它不把数据写进 journal（避免 2x 写放大），但在 transaction commit **之前**强制把这次事务涉及的数据块刷到它们的正式位置。这样保证了一个不变式："元数据一旦 commit，它指向的数据块一定已经是新内容"。代价是引入了一个隐式的"data-before-metadata"排序约束——而正是这个**意外获得**的约束，让 ext3 时代大量"不 fsync 就 rename"的程序碰巧崩溃安全。ext4 的延迟分配打破了这个意外，于是有了 2009 年的大论战（下节）。

**htree——顺带解决的目录扩展性问题**。ext2/3 早期目录是线性数组，几千个文件就退化成 O(n) 查找。htree（Hash Tree）的考古（[HTree - Wikipedia](https://en.wikipedia.org/wiki/HTree) 与 [ext2.sourceforge.net OLS 2005 paper](http://ext2.sourceforge.net/2005-ols/paper-html/node3.html)）：
- **Daniel Phillips** 2000 年设计 htree 数据结构，2001-02 在 ext2 上实现（2.4 内核）。
- **Ted Ts'o** 在 2.5 开发周期清理并合入主线，还实现了 e2fsck 对 htree 的支持。
- **Christopher Li + Andrew Morton** 2002 年把它移植到 ext3，加上 journal 的崩溃一致性。
- 效果：把单目录可用文件数从"几千"拉到"几千万"。htree 是**定深 1–2 层、高扇出、按文件名 hash 索引、不需要 rebalance** 的树——之所以不用标准 B-tree，是因为定深结构在崩溃恢复时简单得多（不会有树旋转导致的中间不一致）。这个"为了崩溃恢复简单而牺牲理论最优"的取舍，是 FS 设计的典型思路。

### 15.2.3 ext4（2006 fork）：extent + 延迟分配——被大文件逼出来的分家

这是整章最重要的一段考古。**ext4 不是"计划中的下一代"，而是一场关于"要不要动稳定的 ext3"的争论的结果。**

**导火索：Alex Tomas 的 extent 补丁**。Alex Tomas（在 Cluster File Systems 公司，为 Lustre 分布式文件系统工作）在 2003–2006 年间开发了一组 ext3 扩展：extent、48-bit 块号、多块分配器（[ext4 - Wikipedia](https://en.wikipedia.org/wiki/Ext4) 确认这批改动"originally developed by Cluster File Systems for the Lustre file system between 2003 and 2006"）。

**关键史料**（[LWN 187321](https://lwn.net/Articles/187321/)，2006-06-12，Jon Corbet 写，标题"Ext3 extensions for large filesystems"）：
- 作者明确列出 ext3 的两个硬限制：① **8TB 文件系统上限**（有符号 32-bit 块号只能数 2 gigablocks）；② **4TB 单文件上限**（inode 的 15 指针结构对大文件越来越低效）。
- 解决方案：extent（把"连续逻辑块 + 连续物理块"压成一个结构，而非一个个指针）+ 48-bit 块寻址（块号拆成 `ee_start_hi` 16 位 + `ee_start_lo` 32 位 = 48 位，支持 1024PB FS）。
- **文章原话**（崩溃安全地引入新格式）："An ext3 filesystem mounted with the extents option enabled will handle files stored in the old way, using block pointers, as always. New files will be created using extents, however."
- **争论的种子**：文章结尾留了一句——"Whether that better filesystem should still be called ext3 is controversial, but that is a subject for another article."（要不要还叫 ext3 是有争议的。）

**分家的决定**（[LWN 194869](https://lwn.net/Articles/194869/)，2006-08，"extents for ext4"；补丁本身 Mingming Cao + Alex Tomas，dated 2006-08-09）：社区的结论是——**这些改动太大、风险太高，不能直接动正被全世界生产系统依赖的 ext3；fork 一个 ext4，让 ext3 保持稳定，激进特性在 ext4 里演进**。这就是为什么今天 ext3 和 ext4 是两个独立的内核模块（虽然 ext4 模块也能挂载 ext2/ext3）。

> **这个决定的工程智慧**：当一个组件被海量生产系统依赖时，"在原地大改" 的爆炸半径不可接受。fork 出新名字 = 给用户一个**显式的选择开关**（你要稳定选 ext3，要新特性选 ext4），而不是偷偷改变所有人脚下的地基。对比 15.2.4 的 2009 事件——那次正是因为 ext4 的延迟分配偷偷改变了一个"用户依赖但从未承诺"的行为，才引爆论战。**显式分家 vs 隐式变更，是这两段历史的核心对比。**

#### extent 替换 block map——核心机制 + 真实源码

ext3 的间接块树 vs ext4 的 extent，先看直觉对比：

```
ext3 的 block map（间接块树）：
  inode.i_block[0..11] → 12 个 direct blocks
  inode.i_block[12]    → indirect（1024 个块号）
  inode.i_block[13]    → double indirect
  inode.i_block[14]    → triple indirect
  问题：1GB 连续文件 = 25.6 万个块号，分散在多级间接块里。
        随机写时每改一个块就要 journal 一条记录；读文件尾要 3 次间接块 IO。

ext4 的 extent tree：
  inode.i_block[60 字节] 复用为：ext4_extent_header + 最多 4 条内联 extent
  overflow 时变成 extent B-tree（index 节点指向更多 extent block）
  单条 extent = [起始逻辑块, 起始物理块(48bit), 长度(≤32768 块=128MB)]
  1GB 连续文件 = 8 条 extent（每条 128MB）。journal 记录从 25 万 → 8 条。
```

**这是 ext4 最大的工程飞跃**，源码里 extent 的三个结构体定义（逐字取得，注意它们的版权头还写着 Alex Tomas 和 Cluster File Systems）：

```c
struct ext4_extent {
	__le32	ee_block;	/* first logical block extent covers */
	__le16	ee_len;		/* number of blocks covered by extent */
	__le16	ee_start_hi;	/* high 16 bits of physical block */
	__le32	ee_start_lo;	/* low 32 bits of physical block */
};

struct ext4_extent_idx {
	__le32	ei_block;	/* index covers logical blocks from 'block' */
	__le32	ei_leaf_lo;	/* pointer to the physical block of the next *
				 * level. leaf or next index could be there */
	__le16	ei_leaf_hi;	/* high 16 bits of physical block */
	__u16	ei_unused;
};

struct ext4_extent_header {
	__le16	eh_magic;	/* probably will support different formats */
	__le16	eh_entries;	/* number of valid entries */
	__le16	eh_max;		/* capacity of store in entries */
	__le16	eh_depth;	/* has tree real underlying blocks? */
	__le32	eh_generation;	/* generation of the tree */
};
```
【真实源码 v6.12 `fs/ext4/ext4_extents.h`，SPDX: GPL-2.0，Copyright 2003-2006 Cluster File Systems, Inc，Author: Alex Tomas】

逐字段精读 `ext4_extent`（12 字节，刚好让 60 字节的 `i_block[]` 装下 1 个 header + 4 条 extent）：
- `ee_block`（逻辑块）：这条 extent 覆盖的文件内**逻辑**起始块。
- `ee_len`（长度）：覆盖多少个块。**最高位被借去当 "unwritten" 标志**（见下）。
- `ee_start_hi` + `ee_start_lo`：**物理**起始块号，48 位（16+32），这就是 [LWN 187321](https://lwn.net/Articles/187321/) 说的"split across `ee_start_hi` and `ee_start`"——48 位块号 × 4KB block = 1024PB 寻址。

`ext4_extent_header.eh_depth` 是树形结构的关键：`0` 表示这一块是 leaf（直接装 `ext4_extent`），`>0` 表示是 index 节点（装 `ext4_extent_idx`，指向下一层）。inode 内联的那个 header 如果 `eh_depth==0`，整个文件就 ≤4 条 extent，零额外 IO——这覆盖了系统里绝大多数文件。

**`ee_len` 的 15-bit 玄机（高频考点，源码实锤）**：`ee_len` 是 16 位，但内核用最高位标记 "unwritten"（`fallocate` 预分配但未写入的空洞）。两个上限宏：

```c
#define EXT4_EXT_MAGIC		cpu_to_le16(0xf30a)
#define EXT_INIT_MAX_LEN	(1UL << 15)
#define EXT_UNWRITTEN_MAX_LEN	(EXT_INIT_MAX_LEN - 1)
```
【真实源码 v6.12 `fs/ext4/ext4_extents.h`】，配套注释原文："If ee_len is <= 0x8000, it is an initialized extent. Otherwise, it is an unwritten one."

- 已写入 extent 最大 `EXT_INIT_MAX_LEN = 1<<15 = 32768` 块（4K block = 128MB）。
- unwritten extent 最大 `32767` 块（少 1 块，因为最高位被借走当 flag）。
- **读到 unwritten extent，内核直接返回零页，不发实际读 IO**——这就是 `fallocate` 预分配后读到全零却几乎不耗磁盘的原因。也是数据库/虚拟机镜像用 `fallocate` 预占空间却不立即写盘的底层机制。

> **微架构/根因视角：为什么 extent 是对的，而 indirect block 在现代硬件上特别糟**。HDD 时代，间接块的额外 IO 是"多一次寻道"（~10ms）。SSD 时代更隐蔽但更致命：间接块让大文件的元数据**分散**，导致 ① 顺序读大文件时元数据访问打断预读流水线；② 删除大文件要遍历海量间接块、产生大量随机读和 journal 写——`rm` 一个几百 GB 的 ext3 文件可能卡几十秒。extent 把"O(块数) 的元数据"压成"O(extent 数)"，删除/截断/seek 全部受益。这是个**算法复杂度层面**的改进，不是常数优化。

#### extent 树的查找——真实源码 + 逐行注解

extent 不是数组就完事，文件大了要变成 B-tree。查找时在每一层做二分。这是真实的 leaf 层二分查找：

```c
/*
 * ext4_ext_binsearch:
 * binary search for closest extent of the given block
 * the header must be checked before calling this
 */
static void
ext4_ext_binsearch(struct inode *inode,
	struct ext4_ext_path *path, ext4_lblk_t block)
{
	struct ext4_extent_header *eh = path->p_hdr;
	struct ext4_extent *r, *l, *m;

	if (eh->eh_entries == 0) {
		/*
		 * this leaf is empty:
		 * we get such a leaf in split/add case
		 */
		return;
	}

	ext_debug(inode, "binsearch for %u:  ", block);

	l = EXT_FIRST_EXTENT(eh) + 1;
	r = EXT_LAST_EXTENT(eh);

	while (l <= r) {
		m = l + (r - l) / 2;
		ext_debug(inode, "%p(%u):%p(%u):%p(%u) ", l,
			  le32_to_cpu(l->ee_block), m, le32_to_cpu(m->ee_block),
			  r, le32_to_cpu(r->ee_block));

		if (block < le32_to_cpu(m->ee_block))
			r = m - 1;
		else
			l = m + 1;
	}

	path->p_ext = l - 1;
	ext_debug(inode, "  -> %d:%llu:[%d]%d ",
		le32_to_cpu(path->p_ext->ee_block),
		ext4_ext_pblock(path->p_ext),
		ext4_ext_is_unwritten(path->p_ext),
		ext4_ext_get_actual_len(path->p_ext));
}
```
【真实源码 v6.12 `fs/ext4/extents.c`，约 2053–2095 行】

逐行注解（这段值得细看，它揭示了几个"教科书不会讲但源码里就这么写"的细节）：
- `if (eh->eh_entries == 0) return;`：**空 leaf 直接返回**。注释解释这发生在 split/add 过程中——也就是说查找函数要容忍"正在被修改的中间态树"，这是并发/原子性设计渗透到每个函数的体现。
- `l = EXT_FIRST_EXTENT(eh) + 1;`：注意 `+1`！二分从**第二个** extent 开始。因为 extent 数组是按 `ee_block` 升序排的，要找"覆盖目标块的那条 extent"，等价于找"最后一个 `ee_block <= block` 的 extent"。从 `first+1` 开始 + 最后 `path->p_ext = l - 1`，是这个"找前驱"语义的标准实现技巧。
- `m = l + (r - l) / 2;`：用 `l + (r-l)/2` 而非 `(l+r)/2`——**避免指针加法溢出**。这是 C 里二分的经典防御写法（这里 l/r 是指针，`l+r` 是非法的指针运算，所以必须这么写）。
- `if (block < m->ee_block) r = m - 1; else l = m + 1;`：标准二分。注意 `else` 分支（相等也往右）保证最终 `l-1` 落在"最后一个 `<= block`"上。
- `path->p_ext = l - 1;`：循环结束 `l` 指向"第一个 `> block`"的位置，减一就是目标 extent。
- `ext_debug(...)` 里调用 `ext4_ext_is_unwritten()` 和 `ext4_ext_get_actual_len()`——后者就是处理上面那个"最高位是 unwritten 标志"的逻辑，把 16 位里的真实长度抠出来。

对应的 index 层二分几乎同构（`ext4_ext_binsearch_idx`，约 2001–2050 行），区别只是用 `ei_block`/`EXT_FIRST_INDEX`，最后 `path->p_idx = l - 1` 指向要下钻的子树。**从 root 到 leaf，每层一次 `binsearch_idx`，最后一次 `binsearch`**，整条查找路径就是 O(深度 × log 扇出)。对绝大多数文件深度=0（全内联），一次都不用下钻。

> **失败模式/真坑：extent 碎片化**。如果一个文件被大量随机写、空洞、`fallocate` 切割，它的 extent 数会暴涨，extent tree 长高，每次 map 块都要走多层二分 + 读多个 extent block。极端情况下 extent tree 本身的元数据比小文件还大。`filefrag -v <file>` 能看到一个文件被切成几千条 extent——这时 `e4defrag` 能把它们重新分配成连续区域、塌缩 extent 数。数据库的数据文件如果不预分配（`fallocate`）而是边写边长，最容易吃这个亏。

#### 延迟分配（delayed allocation）——以及它引爆的 2009 大论战

**机制**：

```
write()   → 只占 page cache、在内存里记"这段还没分配物理块"（da = delayed alloc），
            不立刻分配磁盘块
writeback → 真正要刷盘时才一次性分配物理块（此时能看到完整的脏页范围，
            做 extent 批量分配，更容易找到连续区域）
好处：① 减碎片（攒够了再分配）② 减小写的分配开销 ③ 和 mmap 场景兼容
```

源码入口是 `ext4_da_write_begin`（da = delayed alloc）。它的签名是逐字的：

```c
static int ext4_da_write_begin(struct file *file, struct address_space *mapping,
			       loff_t pos, unsigned len,
			       struct folio **foliop, void **fsdata)
```
【真实源码 v6.12 `fs/ext4/inode.c`，约 2733 行（签名逐字；函数体在本次取材中以转述形式返回，故下面只描述行为，不冒充逐字源码）】

它的关键行为（描述，非逐字源码）：
- 先查 FS 是否已被强制 shutdown；
- 判断要不要退回非延迟路径（**空间紧张或正在做 fsverity 时退回** `ext4_write_begin`）——这是个重要细节：低空间时延迟分配会**自动关闭**，因为延迟分配让"还有多少空间"变得不确定，逼近满盘时必须立即确定分配以免 over-commit；
- 用 `__filemap_get_folio()` 拿/建 folio；
- 调 `ext4_block_write_begin()` 时**传 NULL handle**（还没开 journal transaction）——因为延迟分配此刻不碰磁盘块、不改元数据，自然不需要 journal。这正是"读源码别找错入口"的点：默认 ext4 走的是 `ext4_da_write_begin`/`ext4_da_write_end`，而**不是** `ext4_write_begin`/`ext4_write_end`（后者是关延迟分配或特殊路径才走）。

**2009 年大论战——延迟分配如何变成"丢配置文件"事件**（[LWN 322823](https://lwn.net/Articles/322823/)，2009-03-11，"That massive filesystem thread"）：

事件经过（源自 LWN 与 Ubuntu bug tracker 交叉印证）：
1. 大量桌面应用（KDE 重灾区）用 "open → write → close → rename" 模式替换配置文件，**且不 fsync**。它们这么写在 ext3 上"碰巧"安全——因为 `data=ordered` 的隐式 data-before-metadata 约束 + 5 秒提交窗口，rename 提交时数据基本已落盘。
2. ext4 的延迟分配把这个窗口拉到 **30–60 秒**，而且取消了"为安全提前分配/落盘"的行为。LWN 原文："If blocks have not been allocated for a file, there is no need to write them quickly as a security measure... Those blocks will, instead, wait until the kernel decides to flush them out."
3. 结果：升级到 ext4 后掉电，**rename 已提交（文件名指向新 inode）但数据块还没落盘 → 正式文件变成 0 字节**。KDE 桌面用户大面积丢配置，Ubuntu bug 爆炸，社区论战引爆。

**这是一场关于"契约 vs 习惯"的经典争论：**

| 立场 | 谁 | 核心论点（LWN 原话） |
|------|----|----|
| "这不是 bug" | **Ted Ts'o**（ext4 维护者） | "Since ext3 became the dominant filesystem for Linux, application writers and users have started depending on this, and so they become shocked and angry when their system locks up and they lose data --- even though POSIX never really made any such guarantee." 正确做法是显式 `fsync()`/`fdatasync()`；FS 不该提供未承诺的保证。 |
| "这是事实标准，不能说改就改" | 应用开发者 | ext3 的行为已成 de facto 标准；要求所有人到处加 fsync 会损害性能、违背 Unix 抽象原则。 |

**Ts'o 的最终处理（务实派的胜利，2.6.30，2009）**——他没有等应用改完，而是给"懒程序"加了兜底（LWN 列出三个 patch）：
1. `EXT4_IOC_ALLOC_DA_BLKS` ioctl：允许强制给某文件分配延迟块。
2. 截断标志：被 truncate 的文件 close 时强制分配。
3. **rename 屏障**：当一个文件被 rename 覆盖到另一个文件上时，强制先分配落盘。

这三个合起来就是 `auto_da_alloc` 挂载选项（默认开）。它检测 "replace-via-rename" / "replace-via-truncate" 这两种常见模式，在 rename/truncate 提交前把新文件的延迟分配块落盘。

> **这件事的深层教训（面试常考的"工程哲学"题）**：
> 1. **"被依赖的未承诺行为"是隐形契约**。POSIX 没承诺 ≠ 可以随便改——一旦海量软件依赖了某个实现细节，它就成了事实接口。这和 [Hyrum's Law]（"接口有足够多用户后，你承诺的每个行为都会被某人依赖"）完全吻合。
> 2. **但兜底不是契约**。`auto_da_alloc` 只覆盖 rename/truncate 两种模式，不覆盖所有不 fsync 的情况。**可移植的正确做法仍是 `write → fsync → rename`**。把 `auto_da_alloc` 当成"可以不 fsync"是新一代误区。
> 3. **对比 15.2.3 分家决定**：fork ext4 是"显式给用户选择"，2009 事件是"隐式改变所有人行为"——后者的代价远高于前者。这组对比是 ext 演进史最值得记住的一条。

【真实源码 v6.12 `fs/ext4/ext4.h`（节选关键字段；中间 OS 相关 union 与尾部扩展字段省略）】

**"一区两用"的关键**：`i_block[EXT4_N_BLOCKS]` 这 60 字节——
- 旧解释（无 `EXT4_EXTENTS_FL` flag）：12 direct + indirect + dind + tind 共 15 个块指针。
- 新解释（有 `EXT4_EXTENTS_FL` flag）：前 12 字节是 `ext4_extent_header`，后 48 字节内联最多 4 条 `ext4_extent`。

**用哪种解释由 `i_flags` 里的 `EXT4_EXTENTS_FL` 决定**。这是教科书级的"同一块存储、靠 flag 多态解释"的设计，省下了改 inode 大小的兼容性灾难。

⚠ 两个长得像的字段别混（高频面试坑）：
- `i_block[]`（数组）= 数据块指针 / extent 内联区。
- `i_blocks_lo`（标量）= 该文件占用的**扇区**数（512 字节单位），给 `du`/`stat` 用。注意单位是扇区不是块，这是历史包袱，常被问到。

### 15.3.3 fsync() 的关键路径——真实源码精读

这是本章源码精读的核心之一。fsync 到底慢在哪？看 `ext4_sync_file` 的真实实现：

```c
int ext4_sync_file(struct file *file, loff_t start, loff_t end, int datasync)
{
	int ret = 0, err;
	bool needs_barrier = false;
	struct inode *inode = file->f_mapping->host;

	if (unlikely(ext4_forced_shutdown(inode->i_sb)))
		return -EIO;

	ASSERT(ext4_journal_current_handle() == NULL);

	trace_ext4_sync_file_enter(file, datasync);

	if (sb_rdonly(inode->i_sb)) {
		smp_rmb();
		if (ext4_forced_shutdown(inode->i_sb))
			ret = -EROFS;
		goto out;
	}

	if (!EXT4_SB(inode->i_sb)->s_journal) {
		ret = ext4_fsync_nojournal(file, start, end, datasync,
					   &needs_barrier);
		if (needs_barrier)
			goto issue_flush;
		goto out;
	}

	ret = file_write_and_wait_range(file, start, end);
	if (ret)
		goto out;

	ret = ext4_fsync_journal(inode, datasync, &needs_barrier);

issue_flush:
	if (needs_barrier) {
		err = blkdev_issue_flush(inode->i_sb->s_bdev);
		if (!ret)
			ret = err;
	}
out:
	err = file_check_and_advance_wb_err(file);
	if (ret == 0)
		ret = err;
	trace_ext4_sync_file_exit(inode, ret);
	return ret;
}
```
【真实源码 v6.12 `fs/ext4/fsync.c`】

逐段精读，把"fsync 为什么慢"落到每一行：

1. `if (unlikely(ext4_forced_shutdown(...))) return -EIO;`：FS 已被检测到错误而 shutdown（见 15.8 "变只读"排障），fsync 直接报错。
2. `ASSERT(ext4_journal_current_handle() == NULL);`：**进 fsync 时不能持有 journal handle**。这是死锁防御——fsync 内部要等 transaction commit，如果调用者还攥着一个 handle 没 stop，commit 永远等不到它，死锁。这行 assert 把这个不变式写进了代码。
3. `file_write_and_wait_range(file, start, end);`：**第一笔开销**——把这个文件的脏页刷到块层并**等它完成**。延迟分配的块就是在这里真正分配 + 发 IO 的。fsync 慢的第一来源：如果你刚 `write()` 了几百 MB 还没回写，这一步要等全部落盘。
4. `ext4_fsync_journal(inode, datasync, &needs_barrier);`：**第二笔开销**——提交相关的 journal transaction（内部会调 `jbd2_complete_transaction` 之类，等 commit 完成），并通过 `needs_barrier` 出参告诉外层"要不要补一个设备屏障"。`datasync` 参数就是 `fdatasync` 与 `fsync` 的分叉点：datasync 时如果只有 mtime 变了（不影响读数据），可以不等那个纯时间戳的 transaction。
5. `if (needs_barrier) blkdev_issue_flush(inode->i_sb->s_bdev);`：**第三笔开销**——给块设备发 FLUSH，逼它把内部 write cache 写进持久介质。这一步把"数据/journal 到了设备"升级成"数据/journal 进了非易失介质"。

> **纠正一个常见误解（源码实锤）**：很多人以为"有 journal 就不用发设备 flush"。错。看 `ext4_fsync_journal` 会把 `needs_barrier` 置 true，外层照样 `blkdev_issue_flush`。journal 保证的是"崩溃后能 replay 到一致"，但 replay 的前提是 journal 本身已落到持久介质——所以 commit 之后仍要 flush。`barrier=0`（或某些云盘默认关 flush）就是拿掉这道保险：journal 看似 commit，实则可能还在设备 cache 里，掉电一样丢。

**fsync 延迟的数量级**（结合上面三步）：
- NVMe SSD：几百微秒 ~ 几毫秒（flush 命令本身 + 少量 journal 写）。
- SATA SSD：~1–5ms。
- HDD：几毫秒 ~ 几十毫秒（旋转延迟 + 磁头寻道，commit block 和 flush 各要一次落盘）。

### 15.3.4 Journal 内部机制（jbd2）——API 序列（文档实锤）+ commit 屏障（源码实锤）

**jbd2 = Journal Block Device 2**，ext4 的 journal 实现。它的源码顶部版权头逐字写着："Written by Stephen C. Tweedie <sct@redhat.com>, 1998. Copyright 1998 Red Hat corp"——也就是说 jbd2 的血脉直接来自 ext3 时代 Tweedie 的 jbd（ext4 改进为 jbd2，API 统一加 `jbd2_` 前缀）。

**文件系统修改元数据的标准 API 序列**（来自内核官方文档，逐字引用）：

```
1. jbd2_journal_start()         → 开 transaction，返回一个 handle
2. jbd2_journal_get_write_access() / _get_create_access() / _get_undo_access()
                                → 修改 buffer 前调，让 journal 层有机会拷贝原始数据
3. （在内存里修改 buffer_head）
4. jbd2_journal_dirty_metadata()→ 标记这个 buffer 被改了，挂到 transaction
5. jbd2_journal_stop()          → 结束 handle
```

文档原话（[v6.12 `Documentation/filesystems/journalling.rst`](https://raw.githubusercontent.com/torvalds/linux/v6.12/Documentation/filesystems/journalling.rst)）：
- 开事务："To do this use jbd2_journal_start() which returns a transaction handle."
- 可嵌套："jbd2_journal_start() and its counterpart jbd2_journal_stop() ... are nestable calls, so you can reenter a transaction if necessary, but remember you must call jbd2_journal_stop() the same number of times as jbd2_journal_start() before the transaction is completed."
- 为什么先 get_access："Before you start to modify a buffer you need to call jbd2_journal_get_create_access() / jbd2_journal_get_write_access() / jbd2_journal_get_undo_access() as appropriate, this allows the journalling layer to copy the unmodified data if it needs to."
- 改完要标记："once you are have done so you need to call jbd2_journal_dirty_metadata()."

逐条解读这套 API 为什么是这个形状：
- **start/stop 必须配对、可嵌套**：一个高层 FS 操作（比如 rename）会调多个会改元数据的子函数，每个都想"开个事务"。jbd2 用嵌套计数把它们**合并进同一个外层 transaction**——保证 rename 涉及的所有修改要么一起 commit 要么一起丢。`start` 几次就得 `stop` 几次，最外层 stop 才真正结束。这也是 15.3.3 里"进 fsync 不能持 handle"的原因：handle 没 stop = transaction 没封口。
- **改 buffer 前必须 get_write_access**：journal 要在你动手前把原始 buffer 拷一份（undo 用），并把这个 buffer 登记到当前 handle。**先登记后修改**——顺序反了，journal 就拷到已经被改脏的内容，undo 失效。
- **get_create_access vs get_write_access vs get_undo_access**：create 用于"新分配的元数据块"（不需要保留旧内容）、write 用于"原地改"、undo 用于需要保留旧值做回滚（如 bitmap 释放）。三选一对应三种语义，体现 journal 对"旧数据要不要留"的精细控制。

**异步 commit 与屏障——源码实锤**。transaction 攒够或定时，由 commit 线程刷盘。最关键的"如何保证 commit block 的持久性"，看真实的 commit record 提交函数：

```c
static int journal_submit_commit_record(journal_t *journal,
				transaction_t *commit_transaction,
				struct buffer_head **cbh,
				__u32 crc32_sum)
{
	struct commit_header *tmp;
	struct buffer_head *bh;
	struct timespec64 now;
	blk_opf_t write_flags = REQ_OP_WRITE | JBD2_JOURNAL_REQ_FLAGS;

	*cbh = NULL;

	if (is_journal_aborted(journal))
		return 0;

	bh = jbd2_journal_get_descriptor_buffer(commit_transaction,
					JBD2_COMMIT_BLOCK);
	if (!bh)
		return 1;

	tmp = (struct commit_header *)bh->b_data;
	ktime_get_coarse_real_ts64(&now);
	tmp->h_commit_sec = cpu_to_be64(now.tv_sec);
	tmp->h_commit_nsec = cpu_to_be32(now.tv_nsec);

	if (jbd2_has_feature_checksum(journal)) {
		tmp->h_chksum_type = JBD2_CRC32_CHKSUM;
		tmp->h_chksum_size = JBD2_CRC32_CHKSUM_SIZE;
		tmp->h_chksum[0] = cpu_to_be32(crc32_sum);
	}
	jbd2_commit_block_csum_set(journal, bh);

	BUFFER_TRACE(bh, "submit commit block");
	lock_buffer(bh);
	clear_buffer_dirty(bh);
	set_buffer_uptodate(bh);
	bh->b_end_io = journal_end_buffer_io_sync;

	if (journal->j_flags & JBD2_BARRIER &&
	    !jbd2_has_feature_async_commit(journal))
		write_flags |= REQ_PREFLUSH | REQ_FUA;

	submit_bh(write_flags, bh);
	*cbh = bh;
	return 0;
}
```
【真实源码 v6.12 `fs/jbd2/commit.c`】

逐行精读这段——它把"journal 如何用块层屏障保证原子性"讲透了：
- `tmp->h_commit_sec/nsec`：commit block 里写当前时间戳。recovery 时用它判断 transaction 的时序。
- `if (jbd2_has_feature_checksum(...))`：给 commit block 算 CRC32。**这是检测"半截写入"的关键**——如果 commit block 只写了一半就掉电，checksum 对不上，recovery 就知道这个 transaction 不完整、丢弃它。
- `clear_buffer_dirty(bh); set_buffer_uptodate(bh);`：标记这个 buffer 内容是最新的、即将写出。
- **核心两行**：
  ```c
  if (journal->j_flags & JBD2_BARRIER &&
      !jbd2_has_feature_async_commit(journal))
  	write_flags |= REQ_PREFLUSH | REQ_FUA;
  ```
  当启用 barrier（`JBD2_BARRIER`，默认）且没开 async_commit 时，给 commit block 的写**带上两个屏障标志**：
  - `REQ_PREFLUSH`：**写这个 block 之前，先 flush 设备 cache**——保证 commit block 之前的所有 journal 数据（descriptor + data block）已经落到持久介质。这就是"commit block 必须最后到、且它到的时候前面都已到"的硬件级实现。
  - `REQ_FUA`（Force Unit Access）：**这个 block 本身必须直接落到持久介质**，不许停在设备 cache。
  - 两个合起来 = "前面的都进盘了，我（commit block）也进盘了"——commit block 一旦持久，整个 transaction 就原子地"算数"了。
- `submit_bh(write_flags, bh);`：把带屏障的写发给块层。

> **这就是 journal 原子性的物理基础**。"原子提交"不是软件魔法，是靠 `REQ_PREFLUSH`（提交前栅栏）+ commit block 的 checksum（半写检测）+ `REQ_FUA`（提交本身落盘）三者合力。关掉 barrier（`barrier=0`）= 去掉 `REQ_PREFLUSH|REQ_FUA` = 这两行不执行 = commit block 可能比它"保护"的数据先落盘，掉电后 recovery 看到"完整的 commit block 指向不完整的数据"——比没 journal 还糟。**这是 `barrier=0` 在没有掉电保护（UPS/带电容的 RAID 卡）时绝不能用的根本原因。**

**Recovery 流程**（mount 时自动跑）：扫 journal → 找所有有合法 commit block（checksum 对）的完整 transaction → replay 到正式位置 → 没有合法 commit block 的尾部 transaction 直接丢弃。**这保证了原子性：要么整个 transaction 生效，要么完全没发生。**

🎯 **面试加分点**：jbd2 提供的是 **journal 层的原子性**，不是 FS 操作的原子性，更不是应用操作的原子性。一个 FS 操作（rename）通过 handle 嵌套合并进一个 transaction 获得原子性；但跨多个文件的应用级操作，jbd2 无能为力——因为它看不到应用的事务边界（呼应 15.1.2）。

## 15.4 崩溃一致性与 fsync 语义

### 15.4.1 正式定义

- **原子性（Atomicity）**：单个 FS 操作（rename/create/delete）要么全完成要么完全没发生，无中间态。由 journal transaction 保证。
- **Durability（持久性）**：`fsync()` 成功返回后，立即断电也不丢。由 fsync 路径的三步（刷脏页 + commit + flush）保证。
- 注意这俩是**正交**的：原子但不持久（操作要么全要么不要，但可能整个回滚丢失）、持久但要靠你调 fsync。

### 15.4.2 rename 的原子性 + 标准安全写入模式

`rename(tmp, final)` 是原子性的最典型需求。它需要的磁盘操作：① 父目录删 tmp 项 ② 父目录加 final 项 ③ 若 final 已存在，旧 inode link count 减一（可能 unlink）④ 目录 inode 的 mtime/ctime。

没 journal：操作①后掉电 → tmp 没了 final 也没 → 文件丢失。有 journal：①②③④ 打进一个 transaction，commit 前掉电整个丢弃（tmp 还在），commit 后掉电 replay（final 生效）——**原子**。

💡 **新手误区**：`rename()` 本身原子（POSIX 保证），但 rename 之前的 write/fsync **不会自动发生**。标准安全写入：

```c
// 标准安全写入模式（POSIX，可移植，不依赖 auto_da_alloc 兜底）
int fd = open("file.tmp", O_WRONLY|O_CREAT|O_TRUNC, 0644);
write(fd, data, len);
fsync(fd);              // ← 关键！否则 rename 后 crash，数据还在 page cache 里
close(fd);
rename("file.tmp", "file.final");   // POSIX 原子替换
// 还需 fsync 父目录，保证"file.final 这个目录项"本身落盘
int dir_fd = open(".", O_RDONLY);
fsync(dir_fd);
close(dir_fd);
```
【示意代码，非逐字源码——这是应用层正确用法范式】

为什么最后要 fsync 父目录：rename 改的是**目录**这个 inode 的内容（目录项）。`fsync(fd)` 只保证文件数据 + 文件 inode 落盘，不保证"父目录里新增的那条目录项"落盘。掉电后可能：文件 inode 和数据都在，但目录里找不到它（目录项丢了）。fsync 父目录补这一刀。

### 15.4.3 fsync vs fdatasync vs O_SYNC

| 接口 | 刷数据 | 刷元数据 | 典型开销 | 场景 |
|------|--------|---------|---------|------|
| `fsync(fd)` | 是 | 是（含 mtime/size） | 最高 | 需要完整一致性 |
| `fdatasync(fd)` | 是 | 仅影响读数据的元数据（如 size，**不含纯 mtime**） | 稍低 | 数据库 WAL 首选 |
| `O_SYNC` | 每次 write 同步 | 是 | 极高 | 极少用 |
| `O_DSYNC` | 每次 write 同步 | 仅数据相关 | 高 | 比 O_SYNC 稍好 |
| `sync()` / `syncfs()` | 全局 | 全局 | 最高（全盘/全 FS） | 系统维护 |

**fdatasync 快于 fsync 的源码级原因**：回看 15.3.3 的 `ext4_sync_file(... int datasync)`，`datasync` 参数传进 `ext4_fsync_journal`。当只有 mtime 变了（追加写不改 size 时常如此？——不，追加改 size；但**原地覆盖写**不改 size，只改 mtime），fdatasync 可以不等那个"纯时间戳"的 transaction commit。对数据库这种高频原地覆盖 WAL/数据页的场景，省掉这个 commit 显著降低写 latency——这就是几乎所有数据库（PostgreSQL、MySQL）默认用 `fdatasync` 而非 `fsync` 的原因。

### 15.4.4 O_DIRECT：绕开 page cache（但不等于持久化）

```
普通 buffered write：用户态 → page cache → （异步 writeback）块层 → 设备
O_DIRECT write：      用户态 → （同步等 IO）块层 → 设备（绕开 page cache）
```

**适用场景**：数据库自管 buffer pool（不想被内核 page cache 双重缓存浪费内存）、需要精确控制 IO 对齐/buffer 生命周期、避免大顺序扫描污染 page cache 把工作集挤出去。

**硬约束**：buffer 地址 / offset / size 必须对齐到 logical block size（512B 或 4096B）。违反通常 `EINVAL`，但**某些情况内核会静默回退到 buffered IO**（如写入需要分配新块、写空洞、写文件尾），这时 O_DIRECT 退化成 page cache 写——别假设"开了 O_DIRECT 就一定绕开 page cache"。

🎯 **最容易被问倒的点：O_DIRECT ≠ 持久化**。O_DIRECT 只绕 page cache，让 write 直接进块层；它**不保证数据进了持久介质**——设备内部 write cache 还没 flush（没发 `REQ_FUA`/FLUSH），元数据（如 size 变化）也可能还在 journal 没 commit。要真持久，O_DIRECT 仍需配 `fsync()`，或用 `O_DIRECT|O_DSYNC`（后者让内核在每次 O_DIRECT 写后补 FLUSH/FUA）。把 O_DIRECT 当"同步落盘"是经典误区——它绕过的是**内核缓存**，不是**设备缓存**。

## 15.5 写时复制 FS（btrfs）与日志结构思路——方案对比的"思考感"

### 15.5.1 两条一致性路线的本质

- **ext4 journal 的本质**：先写日志，再改原地。原地数据会被覆盖。
- **btrfs COW 的本质**：永不改原地，新数据写到新位置，然后原子地切换指针。

```
ext4 journal 写一个块：
  1. 写 journal（新数据 + 描述符进 journal 区）
  2. commit（写带屏障的 commit block，见 15.3.4 源码）
  3. checkpoint（把 journal 里的数据写回正式位置，覆盖旧数据）

btrfs COW 写一个块：
  1. 分配新块 B'，写入新数据
  2. 父节点也 COW（指向 B'），递归到 root
  3. 原子地更新 root 指针（super block 里的 root 指针一翻，新树生效）
  4. 旧块 B 变为可回收（若无快照引用）

┌─────────────────────────────────────────────────────────┐
│                  btrfs B-tree COW                       │
│  写前：root → [A] → [B] → [C(data)]                      │
│  写 C 的一个块：                                          │
│    1. 写 C'(新内容)  2. 写 B'(指向 C')                    │
│    3. 写 A'(指向 B') 4. 原子把 root 从 A 翻到 A'          │
│  写后：root → [A'] → [B'] → [C'(new)]                    │
│         旧的 A→B→C 还在 ← 保留引用就是一个快照！          │
└─────────────────────────────────────────────────────────┘
```

**COW 天然支持快照**：旧根保持引用即成低开销快照，不需拷贝数据。这是 btrfs/ZFS 相对 ext4 的杀手级特性。

### 15.5.2 跟着一个具体场景走一遍对比

**场景：一个 8KB 文件，原地改其中 4KB（数据库页更新这种）**

- **ext4（data=ordered）**：① 改的是已分配的块，writeback 时原地覆盖那 4KB ② 元数据基本没变（size 不变），journal 几乎不写 ③ fsync 时刷这 4KB + 一次 flush。**写放大 ≈ 1x，碎片 0**。
- **btrfs COW**：① 那 4KB 不能原地改，必须分配新块写 ② 父节点 → 上层 → root 全部 COW（一串元数据块都得重写）③ 新块和旧块物理位置不同 → **文件碎片化** ④ fsync 走 btrfs 的 log tree。**写放大 > 1x（数据 + 一串元数据），且每次改都制造碎片**。

**结论推演**：高频原地小写（数据库）→ ext4 明显优。这就是为什么"生产数据库极少跑在 btrfs 上"。

**场景：每天给一个 500GB 目录做一致性快照备份**

- **ext4**：没有原生快照。只能 ① LVM 快照（块级 COW，性能差、写满会爆）② `cp --reflink`（ext4 不支持 reflink，xfs/btrfs 才支持）③ 上层应用自己做。**别扭且重**。
- **btrfs**：`btrfs subvolume snapshot` 一条命令，毫秒级，零数据拷贝，之后 `btrfs send` 还能做增量备份。**碾压性优势**。

**结论推演**：需要频繁快照/增量备份 → btrfs/ZFS 明显优。systemd-homed 用 btrfs subvolume 做用户目录、容器 storage driver 用 btrfs，都是吃这个红利。

### 15.5.3 全维度对照表（带不适用边界）

| 维度 | ext4 journal | btrfs COW | 不适用边界 |
|------|-------------|-----------|----------|
| 随机小写 | 好（原地） | 差（写放大 + 碎片） | btrfs 不适合 DB workload |
| 顺序大写 | 好 | 好（有树更新开销） | — |
| 快照 | 无原生 | 原生低开销 | 需快照别选 ext4 |
| 碎片化 | 延迟分配缓解 | 长期严重 | btrfs 重写盘需定期 balance |
| crash 一致性 | journal | COW 原子切换 | 都过关 |
| checksum 数据 | 仅元数据 | 数据+元数据全 checksum | 怕静默损坏选 btrfs |
| SSD 友好 | 有 trim | 不原地覆盖，对 SSD GC 友好 | — |
| 成熟度（2026） | 极成熟 | 成熟，但 RAID5/6 不可用 | 见下 |

💡 **btrfs RAID5/6 的 write hole（生产红线）**：条带的数据块和 parity 块的写入**不是原子的**，掉电时无法判断哪些已落盘，可能用错误的 parity 重建出坏数据。`btrfs-progs` 创建 RAID5/6 时直接打印 "strongly discouraged"。**截至 2026 年这条仍成立**——要 parity RAID 就用 mdraid + 上层 FS，或 ZFS raidz（raidz 通过变长条带 + COW 规避了 write hole）。这是个"COW 也没能自动解决的硬件级原子性问题"，值得记住：COW 解决了**单设备**的原子切换，但**多设备条带的跨盘原子性**是另一个量级的难题。

### 15.5.4 日志结构文件系统（LFS）思路

LFS 更激进：**把整盘当成一个 append-only 日志**，所有写都追加到日志末尾，定期 segment cleaning（GC）回收。

- 优势：所有写都是顺序写（HDD/SSD 都友好）；崩溃恢复快（日志本身就是数据）。
- 劣势：GC 写放大；读性能差（文件块分散在日志各处）。
- 现实落点：**F2FS**（Flash-Friendly FS，Android 广泛用）借鉴 LFS；**SSD/NVMe 的 FTL 本质就是 LFS**（闪存不能原地改写，必须擦除块，FTL 内部就是 log + GC）；纯 LFS 在桌面/服务器没大规模采用。**ZNS SSD 把这个思路推到硬件接口层**（见 15.10）。

> **一个深刻的视角**：你今天写 ext4 到一块 SSD 上，是"原地覆盖语义的 FS"跑在"LFS 语义的 FTL"上。ext4 以为自己在原地改块 B，FTL 其实把它写到了新的闪存页、把旧页标记为可回收。**两层 GC/写放大叠加**——这就是为什么 SSD 上 ext4 也建议开 `discard`/定期 `fstrim`：让 FS 告诉 FTL"这些块我不要了"，FTL 的 GC 才能少搬数据。理解这层叠加，是理解现代存储栈写放大的关键。

## 15.6 XFS 对比——为什么 RHEL 默认换了它

```
                ext4                    XFS
─────────────────────────────────────────────────────────
起源         Linux 2.6 (2006)        SGI IRIX 1994, Linux 2001
最大文件      16TB（4K block）        8EB
最大 FS       1EB                     8EB
Journal       jbd2（元数据/数据可选）  只记元数据，无 data journal
分配器        mballoc                 AG（Allocation Group）并行分配
目录          htree                   B+tree
Online resize 支持                    只扩容不缩容
Defrag        e4defrag                xfs_fsr
适合          通用、小文件多           大文件、高并发写、NFS
生产          最广（Ubuntu 默认）      RHEL 7+ 默认，大文件存储，AWS EFS 底层
```

**XFS 的核心哲学：Allocation Group 并行**。XFS 把盘切成多个 AG，每个 AG 独立管理自己的 inode/block 分配，**AG 之间可并行分配，几乎无全局锁**。

🎯 **为什么 RHEL 7+ 把默认 FS 从 ext4 换成 XFS？**（面试高频）
- **多核扩展性**：ext4 的块/inode 分配有较多全局锁竞争（虽然 mballoc 有 per-CPU 优化），XFS 的 AG 设计天然分区并行，在几十核服务器上高并发写差距明显。
- **大文件/大 FS**：XFS 上限高一个量级，元数据结构（B+tree）对超大目录/超大文件更稳。
- **但 ext4 仍有优势**：小文件密集、元数据密集场景 ext4 不输甚至更好；ext4 工具生态（e2fsprogs）更成熟；ext4 支持 data journal（XFS 不支持），某些需要数据日志的场景只能 ext4。
- **取舍本质**：XFS 赌"大、并发、流式"，ext4 赌"通用、小文件、成熟"。RHEL 面向企业大数据/虚拟化，选 XFS 合理；桌面/通用发行版很多仍默认 ext4。

## 15.7 元数据 vs 数据一致性——ext4 的保证边界

| 场景 | ext4 的保证 |
|------|-----------|
| 任意 crash 后 mount | 可 mount（journal 保元数据一致，replay 后自洽） |
| write() 后 crash，没 fsync | 数据可能丢（还在 page cache） |
| fsync() 后 crash | 数据不丢（已 commit + flush） |
| rename(tmp, final) | 原子（POSIX 保证 + journal 实现） |
| rename() 前没 fsync tmp | tmp 内容可能不完整（除非 auto_da_alloc 兜底，但别依赖） |
| append write 后 crash | 旧状态或新状态二选一（取决于是否 fsync） |
| `data=writeback` 下 crash | 文件尾可能是**旧垃圾数据**（size 更新了数据没写） |

**核心边界**：ext4 保证**元数据一致**（FS 结构不矛盾、能 mount），**不自动保证数据持久**（数据持久要应用调 fsync）。这条边界是所有"crash 后丢数据"问题的判定标尺。

**新手误区**：`close(fd)` ≠ `fsync(fd)`。close 只释放 fd，不触发落盘。进程正常退出也不保证（内核会 flush 脏页，但断电前可能来不及）。

## 15.8 落地：生产调优与排障

### 15.8.1 挂载选项调优

```bash
mount | grep "^/dev"; cat /proc/mounts          # 看当前挂载选项

# /etc/fstab 常用调优（按需）
# noatime    : 不更新 atime，省大量元数据写（几乎所有场景推荐）

# commit=60  : journal commit 间隔（默认 5s），调大减 fsync 频率但加大丢失窗口
/dev/sda1 /data ext4 defaults,noatime 0 2
```

### 15.8.2 性能/状态工具

```bash
iostat -x 1                         # IO 统计：%util、r_await/w_await、aqu-sz
biolatency                          # 块 IO 延迟直方图（BCC/bpftrace）
ext4slower 10                       # ext4 操作 >10ms 的请求（BCC 工具）

cat /proc/sys/vm/dirty_background_ratio
```

### 15.8.3 真实排障场景

**场景 1：fsync 延迟从 1ms 飙到 50ms**
1. `iostat -x 1` 看 `%util` 和 `w_await`——磁盘打满了吗？写队列深吗？
2. 看是否有大批脏页积压触发**同步** writeback（`vmstat 1` 的 `bo`/`wa` 列）——某个批处理产生海量脏页，超过 `dirty_ratio` 触发同步回写，**阻塞了前台 fsync**（这是最常见根因）。
3. 检查 cgroup IO throttle（`cat /sys/fs/cgroup/.../io.max`）。
4. 检查 barrier/flush 是否被云盘/虚拟化层放大（`dmesg`）。
5. SSD 查 SMART：是否 wear leveling/GC 导致延迟尖刺（NVMe `smartctl`）。
- 解决：cgroup IO 隔离批处理；调低 `dirty_background_ratio` 让后台 writeback 提前介入，避免攒到同步阈值。

**场景 2："No space left on device" 但 df 显示有空间**
- 根因：**inode 耗尽**（每个文件吃一个 inode，mkfs 时 inode 数就定死了）。海量小文件（邮件、缓存、session）目录最容易中招。
```bash
df -i                                                    # 看 IUse%
find /path -xdev -printf '%h\n' | sort | uniq -c | sort -rn | head   # 找 inode 密集目录
```
- 注意：ext4 的 inode 数 mkfs 后**不能在线扩**（不像 XFS 动态分配 inode）。这也是 XFS 在小文件极多场景的一个优势。

**场景 3：文件系统突然变 read-only**
- 根因：内核检测到 FS 错误（metadata checksum 不符、结构矛盾），按 `errors=remount-ro`（默认）降级只读防止进一步损坏——对应 15.3.3 源码里那个 `ext4_forced_shutdown` 检查。
```bash
dmesg | grep -iE "ext4|EXT4-fs error|remount"            # 找触发错误
# 修复（必须先 umount）：
fsck.ext4 -y /dev/sda1
```
- 深层：频繁无故变只读往往是**硬件信号**（盘要坏、线缆/控制器问题、内存 bit flip 写坏元数据）。先查 SMART/dmesg 硬件错误，别只反复 fsck。

### 15.8.4 关键内核参数（writeback，直接影响 fsync 行为）

```bash
/proc/sys/vm/dirty_ratio               # 脏页超内存 % 触发"同步"回写（默认 20）—— 飙到这里会阻塞写者
/proc/sys/vm/dirty_background_ratio    # 脏页超 % 触发"后台"回写（默认 10）—— 提前泄洪
/proc/sys/vm/dirty_expire_centisecs    # 脏页超 N×10ms 被回写（默认 3000=30s）
/proc/sys/vm/dirty_writeback_centisecs # 回写线程唤醒间隔（默认 500=5s）
```
**调优心法**：前台延迟尖刺多半是脏页攒到 `dirty_ratio` 触发同步回写。把 `dirty_background_ratio` 调小（如 5）让后台线程更早开始泄洪，能把"突然的同步阻塞"摊成"平稳的后台流量"。

### 15.9.2 Teardown 路线：追一次 write() → fsync → journal commit

```
1. vfs_write()（fs/read_write.c）→ f_op->write_iter
2. ext4_file_write_iter()（fs/ext4/file.c）→ buffered 分支
3. generic_perform_write()（mm/filemap.c）
   → a_ops->write_begin = ext4_da_write_begin  ← 延迟分配，NULL handle，不开 journal
   → copy_from_user
   → a_ops->write_end   = ext4_da_write_end    ← 标 folio dirty，记 da 预留
   ⚠ 默认 delayed alloc，入口是 _da_ 版本，别找成 ext4_write_begin
4. ── 此时数据只在 page cache，物理块未分配 ──
5. 用户调 fsync → ext4_sync_file()（fs/ext4/fsync.c，本章已精读）
   → file_write_and_wait_range()  ← 这里 writeback 才真正分配块、发 IO（走 mballoc/extents）
   → ext4_fsync_journal()         ← 提交相关 transaction
   → blkdev_issue_flush()         ← 设备屏障
6. journal commit：jbd2_journal_commit_transaction()（fs/jbd2/commit.c）
   → 7 个 phase，最后 journal_submit_commit_record() 带 REQ_PREFLUSH|REQ_FUA（本章已精读）
7. checkpoint：commit 后把 journal 数据写回正式位置，回收 journal 空间
```

**验收标准（P4）**：
- 能描述一次 buffered write 从"进 page cache"到"延迟分配的块在 fsync 里才落盘"的完整时序。
- 能解释 `ext4_da_write_begin` 为什么传 NULL handle（此刻不改元数据）。
- 能用源码说明 fsync 的三笔开销分别对应哪几行。
- 能解释 `journal_submit_commit_record` 里 `REQ_PREFLUSH|REQ_FUA` 两行如何构成原子提交的物理基础。
- 能说清 journal commit 和 checkpoint 的区别（commit = 写进 journal 区并标记有效；checkpoint = 写回正式位置并回收 journal 空间）。

## 15.10 未来演进

### 15.10.1 ext4 的持续改进

- **metadata_csum**（近年 mke2fs 默认开）：给 superblock/GDT/inode/目录块/extent 块加 CRC32C，检测磁盘 bit flip 和错位写。对应 15.8.3 场景 3 的"checksum 不符 → 变只读"。
- **fast commit**（**Harshad Shirwadkar**，**Linux 5.10**，2021-01，[LWN 842385](https://lwn.net/Articles/842385/)）：解决普通 jbd2 commit 的"连坐"问题——LWN 原话 "fsync() causes a lot of unrelated I/O"，因为标准 fsync 会刷**所有** fd 的 pending 元数据（transaction 是全局的）。fast commit 维护一条**并行的、文件级的轻量 redo log**，只记当前 fd 相关的少量变更，且"信息能重建的就不记"（如 inode table、block bitmap 能从 extent 信息推出来，就不写进 log）。
  - **实测收益**（Shirwadkar 的 benchmark）："20-200% performance improvements ... for local filesystems, and 30-75% improvement for NFS workloads."
  - **限制**：仅 `data=ordered` 可用；只覆盖 unlink/link/inode create/块增删/inode replay 等少数操作，其它（如改扩展属性）**回退**到标准 commit；首次挂载后第一个 commit 仍走 full commit；它是对 jbd2 的**补充而非替代**。
- **bigalloc**：以 cluster（多块）为分配单位，减 bitmap/元数据开销，适合大文件为主的盘（代价：小文件浪费空间）。

### 15.10.2 btrfs 的成熟化与红线

- **RAID5/6 仍不可用于生产**（write hole，见 15.5.3）——2026 年依然成立。
- btrfs send/receive 在容器（podman/docker 的 btrfs storage driver）、增量备份场景有真实应用。
- systemd-homed 用 btrfs subvolume 做加密用户目录。
- Meta 等大厂用 btrfs 做**快照**，但核心存储仍多用 XFS——再次印证 15.5.2 的场景结论。

### 15.10.3 bcachefs：合入又被踢出主线的活教材（时效：2026-06）

bcachefs 由 bcache 作者 **Kent Overstreet** 开发，试图结合 ext4 的性能和 btrfs 的 COW 特性（COW + checksums + compression + encryption + replication/tiering 原生支持）。但比技术特性更值得讲的，是它**"代码能 merge，却没能留在主线"**的完整治理故事。

**精确时间线**（[LWN 1040120](https://lwn.net/Articles/1040120/) + [Wikipedia Bcachefs](https://en.wikipedia.org/wiki/Bcachefs) + [Phoronix](https://www.phoronix.com/news/Bcachefs-Removed-Linux-6.18) 交叉印证）：
- **2023-10 合入，随 Linux 6.7（2024-01 发布）首次进正式内核**，标 experimental。
- **2024-04-07**：Linus 公开吐槽——"if you thought bcachefs was stable already, [I have a bridge to sell you]"。
- **2024-08**：Linus 再评 "nobody sane uses bcachefs and expects it to be stable"；同期 Debian 维护者 orphan 了 bcachefs-tools。
- **2024-11**：因"written abuse of another community member"，Overstreet 在 6.13 周期被**限制提交**（6.14 周期恢复）。
- **2025（merge window）**：决定性裂痕——Overstreet 在 merge window **晚期**提交一个大改动（`journal_rewind`，是**新增功能而非 bug fix**），违反内核"merge window 后只收 fix"的铁律，与 Linus 激烈往复。
- **2025-06-26**：Linus 宣布移除，原话 "I don't really feel comfortable being involved at all"。
- **2025-08**：状态从 Supported 改为 "Externally maintained"，对应 **Linux 6.17**。
- **2025-09（late）**：代码从主线**完全移除**，自 **Linux 6.18** 起内核树不再含 bcachefs（约 11.7 万行），改以 **DKMS / 外部模块**存在。Linus 的收尾理由："It's now a DKMS module, making the in-kernel code stale, so remove it to avoid any version confusion."

**面试/认知价值**：
1. **主线准入门槛不只是代码质量**，还有对 release 流程（merge window 纪律）和协作规则的遵守。技术再好，破坏 maintainer 之间的信任协议一样被踢出。
2. **对比 ext4 的演进**：ext3→ext4 的分家是"在尊重流程下，用新名字给用户显式选择"；bcachefs 是"绕过流程强推改动"。同样是大改动，前者被社区接纳并成为默认 FS，后者被移除。**流程不是官僚，是大型协作的信任基础设施**——这是本章贯穿始终的一条线（呼应 15.2.3 和 2009 事件）。
3. 被移出主线后，生产更不应依赖；它本就不及 ext4/XFS 成熟。

### 15.10.4 NVMe / ZNS 把 LFS 推到硬件层

Zoned Namespace（ZNS）SSD 把 LFS 思想带到硬件接口：
- SSD 暴露成若干 zone，每个 zone **只能顺序写**（像 HDD 轨道 / SMR）。
- 消除 SSD 内部 FTL 的 GC 写放大（GC 责任上移给 FS）。
- f2fs、btrfs 已有 ZNS 支持；**ext4 对 ZNS 支持有限**——ext4 的原地覆盖语义与 ZNS 的顺序写约束有根本冲突（呼应 15.5.4：ext4 是"原地语义"，ZNS 要求"LFS 语义"，两者不兼容）。这也解释了为什么 ZNS 时代 f2fs/btrfs 更受青睐。

## 15.11 面试视角与高频题

**Q1：ext4 的 journal 如何保证 rename() 的原子性？**
- rename 改哪些结构（源/目标目录项、可能的旧 inode link count）；
- jbd2 用 handle 嵌套把这些修改并进一个 transaction；
- crash 在 commit 前（整体丢弃，tmp 还在）vs commit 后（replay 生效）；
- 加分：commit block 靠 `REQ_PREFLUSH|REQ_FUA` + checksum 实现原子（能引源码）；还需 fsync 父目录才保证目录项落盘。

**Q2：为什么 fsync() 这么慢？如何优化？**
- 三笔开销（引 `ext4_sync_file` 源码）：刷脏页等回写 + commit transaction + `blkdev_issue_flush`；
- 优化：`fdatasync`（跳过纯 mtime 的 commit）、批量化（攒一批写一次 fsync，throughput vs latency）、fast commit（5.10，绕开全局 transaction 连坐）、`io_uring` 的 async fsync（不阻塞调用线程）；
- 错误答案陷阱：用 O_DIRECT 替代 fsync——O_DIRECT 绕 page cache 但不 flush 设备 cache，不保证持久。

**Q3：ext4 和 btrfs 的一致性机制本质区别？各自适合什么场景？**
- journal（原地改 + WAL 回滚/重放）vs COW（永不覆盖 + 原子切根）；
- 用 15.5.2 的两个场景跑一遍（8KB 原地小写 → ext4 优；500GB 快照 → btrfs 优）；
- 加分：btrfs 数据库 workload 碎片化的根因（COW 让原地改变成异地写）；btrfs RAID5/6 的 write hole。

**Q4：data=ordered / writeback / journal 的区别与 trade-off？**
- 见 15.2.2 表；重点 data=ordered 如何用"data-before-metadata"防止"size 更新了数据没写"；
- 陷阱：data=writeback crash 后可能暴露旧垃圾数据（有安全隐患），不是一定丢数据但可能读到错误内容。

**Q5：2009 年 ext4 延迟分配丢数据事件讲一下，它的工程教训是什么？**（考工程哲学）
- 延迟分配把崩溃安全窗口从 5s 拉到 30–60s，打破 ext3 时代"不 fsync 也碰巧安全"的隐式约束（引 LWN 322823、Ts'o 原话）；
- Ts'o 立场 "POSIX never made such guarantee" vs 应用方 "de facto standard"；
- 解决：`auto_da_alloc`（2.6.30）兜底 rename/truncate 模式，但不是契约；正确做法仍是 write→fsync→rename；
- 教训：被海量软件依赖的"未承诺行为"是隐形契约（Hyrum's Law）；显式变更（fork ext4）远优于隐式变更。

**Q6：O_DIRECT 的对齐要求从哪来？为什么 FS 不自动处理非对齐？**
- 块设备按扇区（512/4096B）读写，O_DIRECT 直接操作 DMA buffer 需物理对齐；
- 非对齐要 read-modify-write，引入额外 IO + 并发窗口；
- FS 选择直接拒绝（EINVAL）而非自动处理：保持语义简单、不引入隐式开销；
- 加分：O_DIRECT ≠ 持久，仍需 fsync 或 O_DIRECT|O_DSYNC。

**Q7：extent 相比 indirect block 的性能差异具体在哪？**
- 元数据读：indirect 访问大文件尾要 3 次间接块 IO，extent 最多 B-tree 深度次（通常 0–2）；
- journal 写放大：追加写大文件，indirect 每个新块都 journal 一个块号，extent 可扩展现有 extent（journal 一次）；
- 删除/截断：indirect 要遍历海量间接块（rm 大文件卡顿），extent O(extent 数)；
- 根因：extent 是算法复杂度改进（O(块数)→O(extent 数)），不是常数优化。

**Q8：bcachefs 为什么被踢出主线？说明了什么？**
- 6.7 进、6.18 出；导火索是 merge window 晚期提交非 fix 的大改动（journal_rewind）+ 与社区的信任破裂；
- 说明：主线准入 = 代码质量 + 流程纪律 + 协作信任，三者缺一不可；对比 ext4 分家（尊重流程的大改动）。

**Q9：为什么说 close(fd) ≠ fsync(fd)？什么时候可省 fsync？**
- close 只释放 fd 不落盘；进程退出也不保证断电前刷完；
- 可省 fsync：允许丢失的日志追加、临时文件、可重建的缓存。

## 15.12 章末五件套

### 高频面试题（补充 2 道）

**Q10：fast commit 解决了什么问题？它的限制是什么？为什么不能完全替代 jbd2？**
- 解决标准 fsync 的"全局 transaction 连坐"（一次 fsync 刷所有 fd 的 pending 元数据）；走文件级轻量 redo log，能重建的信息不记（LWN 842385，5.10，Shirwadkar）；
- 限制：仅 data=ordered；只覆盖少数操作类型，其余回退 full commit；首个 commit 仍 full；是补充非替代。

**Q11：metadata_csum 能检测什么、不能检测什么？为什么 ext4 不默认校验数据块？**
- 能检测元数据（sb/GDT/inode/目录/extent 块）的 bit flip 和错位写，触发只读保护；
- 不校验**数据块**内容（那是 btrfs/ZFS 的领域）——因为数据校验需要每次读都算 checksum + 每次写都更新，开销大，且 ext4 的设计哲学是"元数据一致归我，数据完整归应用"；
- 这正是"怕静默数据损坏就选 btrfs/ZFS"的分界线。

### 设计题

**题目**：设计一个"安全文件写入库"，封装 write/fsync/rename，保证 ① 任意时刻 crash 不出现半写入文件 ② 吞吐不低于直接 write() 的 80%。

要点：
- 核心链路：write → tmp → fsync → rename → fsync(dir)；不依赖 `auto_da_alloc` 兜底（要可移植）。
- 吞吐优化：内部 buffer + 批量 fsync（不是每次 write 都 sync）；trade-off 是 buffer 越大、可能丢失的数据越多。
- 进阶：`O_TMPFILE`（3.11+）创建无名临时文件，避免 tmp 被其它进程看到 + 避免崩溃残留 tmp 文件——但注意 `O_TMPFILE` inode 用 `linkat` 只能"创建新链接"不能覆盖，要原子替换已存在文件仍需 link 到临时名再 `rename`。

### 系统题

**排障场景**：PostgreSQL（data 在 ext4）每天凌晨 3 点写入延迟从 2ms 突增到 200ms，持续约 10 分钟后恢复。

量级估算 + 排查：
1. 凌晨 3 点的定时任务？`crontab -l` / `journalctl` / 备份脚本日志。
2. 是否脏页批量 writeback？`iostat -x 1` 看 `%util` 曲线、`vmstat 1` 看 `bo`/`wa`。
3. 估算：PG `shared_buffers` 8GB，若备份/批处理把脏页推到 `dirty_ratio`=20% → 1.6GB 脏页，按 100MB/s 落盘 ≈ 16s 同步阻塞窗口；持续 10 分钟说明是**持续产生脏页**的任务（如全库备份边读边脏），不断顶在同步回写阈值上。
4. 解决：① `dirty_background_ratio` 调到 5%（提前后台泄洪，避免顶到同步阈值阻塞前台）② cgroup IO 隔离备份任务 ③ 备份改用 `nice`/`ionice` + 限速，或走只读副本。

### 代码题

```c
/*
 * 安全写入工具（可运行，POSIX C，Linux >= 3.11）
 * 保证：写成功后 crash，文件要么旧内容、要么新内容，不会中间态
 * 编译：gcc -o safe_write safe_write.c
 * 验证：kill -9 中断后检查文件内容
 *
 * 语义澄清（重要）：
 *  - linkat(O_TMPFILE inode) 是"创建新链接"，不是 rename，无法覆盖已存在目标（EEXIST）。
 *    本函数仅用于"目标不存在"的首次创建。
 *  - 要原子替换已存在文件：O_TMPFILE 写好 → linkat 到临时名 → rename(临时名, 正式名) → fsync(父目录)。
 *    rename 才有覆盖语义。
 */
#define _GNU_SOURCE
#include <fcntl.h>
#include <stdio.h>
#include <string.h>
#include <unistd.h>
#include <errno.h>

int safe_write_file(const char *dir, const char *path, const char *data, size_t len) {
    /* 打开 dir 下的无名临时文件（数据写完前没有名字，崩溃也不残留 tmp） */
    int fd = open(dir, O_WRONLY | O_TMPFILE, 0644);
    if (fd < 0) {
        perror("open O_TMPFILE");
        /* TODO: 回退到 open(tmp_path, O_WRONLY|O_CREAT|O_TRUNC, 0644) + rename */
        return -1;
    }
    /* 写入（生产代码需循环处理 partial write） */
    ssize_t w = write(fd, data, len);
    if (w != (ssize_t)len) { perror("write"); close(fd); return -1; }
    /* 关键：fsync 确保数据真正落盘，之后 link/rename 才有意义 */
    if (fsync(fd) != 0) { perror("fsync"); close(fd); return -1; }
    /* 通过 /proc/self/fd/N 给无名 inode 起名（避开 AT_EMPTY_PATH 的权限要求） */
    char proc_path[64];
    snprintf(proc_path, sizeof(proc_path), "/proc/self/fd/%d", fd);
    /* TODO: 若 path 已存在需 link 到临时名再 rename 覆盖；此处仅 path 不存在场景 */
    if (linkat(AT_FDCWD, proc_path, AT_FDCWD, path, AT_SYMLINK_FOLLOW) != 0) {
        perror("linkat"); close(fd); return -1;
    }
    close(fd);
    return 0;   /* 注：完整实现还应 fsync 父目录，保证新目录项落盘 */
}

int main(void) {
    const char *content = "Hello, safe write!\n";
    if (safe_write_file("/tmp", "/tmp/test_safe.txt", content, strlen(content)) == 0)
        printf("Safe write succeeded.\n");
    return 0;
}
```
【示意/可运行示例代码，非内核源码——演示应用层正确用法，含 TODO 标注待补的回退/覆盖/目录 fsync】

**暗坑提示**：① `O_TMPFILE` 非所有内核/FS 支持，生产需 fallback；② `fs.protected_hardlinks=1`（现代默认）下 linkat 要用 `/proc/self/fd/N`；③ `path` 已存在时 linkat 返回 EEXIST，需先 link 临时名再 rename；④ 别忘 fsync 父目录。

## 心法

**文件系统的本质契约**：把"可能随时断电的块设备"包装成"崩溃后仍然自洽的命名空间"。

把本章压成三句话带走：
1. **一致性靠"原子提交"，原子提交靠硬件屏障**——journal 的 `REQ_PREFLUSH|REQ_FUA` + commit block checksum，或 COW 的原子切根，本质都在解决同一个问题：让多步磁盘修改对掉电"要么全发生要么全不发生"。
2. **持久化是应用的责任，不是 FS 的默认**——ext4 只保证元数据一致和"你调了 fsync 的数据"持久；`write()` 返回、`close()`、O_DIRECT 都不等于落盘。2009 年那场论战就是整个生态学这一课的学费。
3. **演进史就是一部权衡史，也是一部协作史**——extent 替间接块（算法复杂度）、journal vs COW（写放大 vs 碎片）、ext4 分家 vs bcachefs 被踢（流程与信任）。读 FS 不要只读"它怎么工作"，要读"它为什么是现在这样，以及当年差点变成什么样"。
