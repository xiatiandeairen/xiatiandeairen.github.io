> **一句话定位**：Linux 锁机制是内核多核扩展性的脊梁。本章不满足于"会选锁"——我们沿着 30 年的 git 历史与邮件列表辩论，把 qspinlock / mutex / seqlock 的 v6.12 真实源码逐行拆开，搞清每一个字节、每一条 barrier 为什么在那里，以及当年社区为什么吵成那样。

## TL;DR（本章最硬结论）

1. **sleep 是分界线，但 v6.12 之后这条线本身成了配置项**：PREEMPT_RT 在 6.12 完成主线化（2024-09，20 年长跑），`spinlock_t` 在 RT 内核下就是 `rt_mutex`——"spinlock 不可睡眠"严格说只对 `raw_spinlock_t` 永真。
2. **x86 spinlock 三代演进的驱动力全是公平性与 cache 流量**：TAS 锁有 100 万次量级的不公平获取（Nick Piggin 实测）→ ticket lock（2.6.25）解决公平性但留下 O(N) cacheline 广播 → qspinlock（4.2）用"pending bit + 压缩 MCS 队列"同时解决两者，还保住了 4 字节 ABI。
3. **mutex 是一场政治斗争的产物**：2005 年 12 月 Linus 骂初版 mutex 提案 "INCREDIBLY BROKEN"，Ingo Molnar 用"semaphore 隐藏了多年的 wake-two bug"反杀，2.6.16 合入。今天的 mutex = fastpath CAS + 乐观自旋（从 -rt 树移植，2.6.30）+ handoff 防饿死（v4.10）。
4. **seqlock 的全部正确性挂在两对 barrier 上**：writer 的 `++/smp_wmb`、reader 的 `smp_rmb/再读`。少任何一个都会出现"偶数序号 + 撕裂数据"。含指针数据结构禁用不是建议，是源码注释里的硬规定。
5. **lockdep 证明的是"已观测依赖图无环"，不是"无死锁"**；data race 归 KCSAN，内存序归 LKMM（herd + litmus tests，4.17 进树）。三件工具职责不重叠。
6. **锁的争用代价主要是 cache 一致性协议流量，不是指令数**：`perf top` 里 `native_queued_spin_lock_slowpath` 高居前列时，瓶颈在互连带宽与 NUMA 跨节点延迟，优化方向是减少共享而非换更快的锁。

**前置依赖章**：第 5 章（进程调度与抢占）、第 11 章（并发与 SMP 基础、cache 一致性）、第 13 章（RCU，本章多处对照）。

**本章源码基线**：Linux **v6.12**（选它有讲究：这是 PREEMPT_RT 完成主线化的版本，也是 LTS）。所有标注【真实源码】的片段均直接取自 `https://raw.githubusercontent.com/torvalds/linux/v6.12/<path>`，史料均经 LWN / GitHub commit 实地核实，未核实处明确标「待核」。

## 12.1 背景：锁的代价为什么是微架构问题

### 从"禁中断"到"cache 一致性协议"

单核时代内核靠**禁中断**实现互斥：没有第二个执行流，关掉抢占源即可。SMP 把问题换了量级——16 个 core 同时跑内核代码时，互斥的本质变成了**让一个 cache line 的修改权在核间安全流转**。

三个正交的问题（混在一起谈必出错）：

1. **原子性**：read-modify-write 不能被打断。硬件给的工具是带 `LOCK` 前缀的指令（x86）或 LL/SC（ARM/RISC-V）。
2. **可见性**：CPU0 的写什么时候对 CPU1 可见。由 cache 一致性协议（MESI 族）保证——但只保证"最终"与"单变量"，不保证多变量之间的次序。
3. **顺序性**：编译器和 CPU 都会乱序。store buffer 让"我写完了 A 再写 B"在别的核眼里可能变成"先看到 B"。这是 barrier 要解决的。

### ACQUIRE / RELEASE：锁与内存序的正式契约

内核对锁的内存序承诺写在 `Documentation/memory-barriers.txt` 里。这是逐字原文：

【真实源码 v6.12 `Documentation/memory-barriers.txt`（节选）】

```text
 (5) ACQUIRE operations.

     This acts as a one-way permeable barrier.  It guarantees that all memory
     operations after the ACQUIRE operation will appear to happen after the
     ACQUIRE operation with respect to the other components of the system.
     ACQUIRE operations include LOCK operations and both smp_load_acquire()
     and smp_cond_load_acquire() operations.

     Memory operations that occur before an ACQUIRE operation may appear to
     happen after it completes.

 (6) RELEASE operations.

     This also acts as a one-way permeable barrier.  It guarantees that all
     memory operations before the RELEASE operation will appear to happen
     before the RELEASE operation with respect to the other components of the
     system. RELEASE operations include UNLOCK operations and
     smp_store_release() operations.
```

逐句拆解，这两段话是全章的钥匙：

- **"one-way permeable"（单向可渗透）**：ACQUIRE 是一面只许"向后渗透"的墙——临界区内的访存不能往前漂出锁外，但锁前面的访存可以往后漂进临界区。RELEASE 反之。这意味着 lock/unlock 不是 full barrier，是两面方向相反的半透膜；两个相邻临界区之间的代码可以被"夹"进任何一边。
- **推论**：`UNLOCK A; LOCK B` 的组合不构成 full barrier（原文后文明确说 "a RELEASE+ACQUIRE pair is -not- guaranteed to act as a full memory barrier"）。在依赖跨锁顺序的代码里，这是真实坑源。
- **工程意义**：本章后面读 qspinlock / mutex / seqlock 源码时，每个 `_acquire` / `_release` / `_relaxed` 后缀都是作者对"这里最少需要什么序"的精确声明。`_relaxed` 不是偷懒，是证明过"此处不需要序"之后省下的一次 barrier 开销——在 ARM64 这种弱内存序架构上每省一次都是真金白银。

**工程推论①**：锁原语 = 原子指令 + 内存序契约 + 等待策略。三者独立设计：原子指令决定无争用开销，内存序契约决定弱序架构上的正确性与开销，等待策略（自旋/排队/睡眠）决定争用行为。后面每种锁都按这三轴分析。

## 12.2 设计考古 I：BKL 的兴衰（1996–2011）

> 演进主线一句话：**Linux 的锁史 = 不断把"一把锁"打碎成"对的锁"的历史**。起点就是那把著名的大锁。

### BKL 是什么、为什么难拆

**BKL（Big Kernel Lock）**：`lock_kernel()/unlock_kernel()`，一把全局锁保护整个内核，Linux 2.0 的 SMP 支持靠它起家。它的特殊语义是后来一切麻烦的根源：

- **可递归**：持有时可再次获取（计数）。
- **睡眠自动放锁**：持 BKL 的任务被调度走时自动释放、切回时自动重取。

这两条"宽松"语义让代码可以在不理解锁依赖的情况下工作——也让每一处 BKL 的移除都得先回答"这里到底在保护什么"。这就是拆除耗费十几年的根本原因。

### 2008 年的"BKL 反击战"：Linus 故意让它烂

2.6.26-rc 周期，BKL 底层换成新的 generic semaphore 实现后，Intel 的 Yanmin Zhang 报告 AIM benchmark **掉了 40%**。新 semaphore 追求公平（锁直接移交给等待者），但被指定的接收线程可能还没在跑、cache 是冷的，造成"死区时间"。Linus 的处理方式成了内核史上的名场面：他把 BKL 改回 spinlock 实现，并且**拒绝继续优化它**：

> "Quite frankly, maybe we _need_ to have a bad BKL for those to ever get fixed... why not just say 'Hell no - if you have issues with it, you need to work'"
> —— Linus Torvalds，2008-05（LWN: *The Big Kernel Lock strikes again*，2008-05-13，Jonathan Corbet）

**故意保留一个性能差的全局锁，作为逼迫各子系统迁移的工程杠杆**——这是"机制设计"思维在内核社区的真实应用。

### 终局：2.6.37 可关闭，2.6.39 删除

收尾人是 Arnd Bergmann。LWN *The real BKL end game*（2011-01-26）记录了最后一批钉子户的处置：**i830、autofs3、smbfs 直接删除**，**appletalk、hpfs 移入 staging**，其余修复。时间线两步走（面试高频，别记成一步）：

- **2.6.37**（2010 末）：`CONFIG_BKL` 开关引入，第一次可以编译出完全无 BKL 的内核。
- **2.6.39**（2011-05）：BKL 本体删除。

最终 commit 值得记住它的标题：

> commit `4ba8216cd905`（**"BKL: That's all, folks"**，Arnd Bergmann）
> 删除 `lib/kernel_lock.c`（-136 行）与 `include/linux/smp_lock.h`（-65 行），7 个文件 +2/-232。
> commit message："A lot of people have worked on this in the past, so the credit for this patch should be with everyone who participated in the hunt."
> Acked-by: Alan Cox, Ingo Molnar, Thomas Gleixner。

**工程推论②**：锁粒度细化的正确路径是"先有正确的粗粒度锁，再做 profiling 驱动的细化"。但 BKL 教训补充了下半句：**宽松语义（递归、自动放锁）是技术债的放大器**——它让粗粒度锁可以"先能跑"，代价是细化时每一处都要重新考古真实依赖。这也是后文 mutex 坚持严格所有权语义（不可递归、谁锁谁解）的历史动机。

## 12.3 设计考古 II：spinlock 三代（TAS → ticket → qspinlock）

### 第一代：TAS 锁的公平性灾难

2.6.25 之前的 x86 spinlock 是 test-and-set 风格（锁值减到负数表示忙，释放即写 1，谁抢到算谁的）。问题是**完全不公平**：释放后谁先把 cache line 抢到手谁赢，而刚释放锁的核（及其邻居）在 cache 上有先天优势。Nick Piggin 的实测数据（LWN *Ticket spinlocks*，Jonathan Corbet，2008-02-06）：

> "On an 8 core (2 socket) Opteron, spinlock unfairness is extremely noticable, with a userspace test having a difference of up to 2x runtime per thread, and some threads are starved or 'unfairly' granted the lock up to 1 000 000 (!) times."

同一把锁，有的线程跑 2 倍时长，有的被"插队"**一百万次**。在 NUMA 上这不是统计噪声，是结构性饥饿：锁在一个 socket 内部传来传去，远端 socket 几乎拿不到。

### 第二代：ticket spinlock（2.6.25，commit `314cdbefd1fd`）

Nick Piggin 的方案（"x86: FIFO ticket spinlocks"，2008-01-30，合入 2.6.25）：16 bit 拆成两个 8 bit 字段——`next`（取票机）与 `owner`（叫号屏）。获取 = 原子取票 + 等叫号；释放 = `owner++`。FIFO 严格公平，延迟可预期。

但它埋了一颗雷：**所有等待者自旋在同一个 cache line 上**。每次释放，`owner++` 这一写操作会 invalidate 所有等待者的 cache 副本，N 个等待者全部跨核重新拉取——O(N) 的一致性流量，锁越热越糟，NUMA 上跨 socket 流量被放大一个量级。

### 第三代：qspinlock（v4.2，commit `a33fda35e3a7`）

学界答案早就有：**MCS lock**（Mellor-Crummey & Scott，1991）——每个等待者排进链表，自旋在**自己的节点**上，释放者只写后继一个节点。O(N) 广播变 O(1) 点对点。qspinlock 源码开头就引用了这篇论文（见下节源码）。

但 MCS 直接进内核有两个 ABI 级障碍（这是面试深水区，能讲清楚的人很少）：

1. **尺寸**：MCS 需要一个 tail 指针（8 字节）。内核 `spinlock_t` 必须保持 **4 字节**——它被嵌在 `struct page` 等海量结构体里，变大 = 全内核内存膨胀。
2. **API**：经典 MCS 的 unlock 需要传入 lock 时用的 node，而内核 `spin_unlock(lock)` 只有一个参数。

Waiman Long（HP，后 Red Hat）与 Peter Zijlstra 的解法（LWN *MCS locks and qspinlocks*，Corbet，2014-03-11）：

> "by storing a CPU number rather than a pointer, the qspinlock patch set allows all of the relevant information to be crammed into a single 32-bit word."

外加一个关键发明——**pending bit**：第一个等待者根本不进 MCS 队列，只置一个 bit 原地自旋。Corbet 的解释：两 CPU 争用的场景占绝大多数，pending bit 让这种场景完全绕开 per-CPU 节点的 touch，省掉 cache miss。

**考古细节（社区时间感）**：LWN 2014-03 的文章预测 "should be enough to get this code merged as soon as the 3.15 development cycle"——实际上 qspinlock 直到 **v4.2**（2015 年中）才合入（已用 GitHub compare API 核实：不在 v4.1，在 v4.2），中间一年多耗在 paravirt 场景的拉锯上：虚拟化环境里 vCPU 可能在持锁/排队时被 hypervisor 切走，排队锁会让所有后继干等，必须配套 pv_wait/pv_kick 机制（这就是源码里 `CONFIG_PARAVIRT_SPINLOCKS` 分支的来历）。commit message 里 Waiman Long 对性能的定位很克制，照录：

> "about the same speed in single-thread"；"much faster in high contention situations especially when the spinlock is embedded within the data structure"；轻中度争用下可能略慢（slowpath 开销更高）；特别适合 "NUMA machines with a large number of cores"。

**工程推论③**：qspinlock 不是"更快的锁"，是**高争用下更稳的锁**。三代演进的每一步都是用单线程路径的少量复杂度，换争用路径的数量级改善——这个 trade-off 方向（优化尾部而非均值）是基础设施组件演进的普遍规律。

## 12.4 源码精读 I：qspinlock（v6.12）

### 12.4.1 数据结构：32 bit 里塞下一个队列

【真实源码 v6.12 `include/asm-generic/qspinlock_types.h`（节选：省略 big-endian 分支与 NR_CPUS≥16K 布局）】

```c
typedef struct qspinlock {
	union {
		atomic_t val;

		/*
		 * By using the whole 2nd least significant byte for the
		 * pending bit, we can allow better optimization of the lock
		 * acquisition for the pending bit holder.
		 */
#ifdef __LITTLE_ENDIAN
		struct {
			u8	locked;
			u8	pending;
		};
		struct {
			u16	locked_pending;
			u16	tail;
		};
#endif
	};
} arch_spinlock_t;
```

```c
 * Bitfields in the atomic value:
 *
 * When NR_CPUS < 16K
 *  0- 7: locked byte
 *     8: pending
 *  9-15: not used
 * 16-17: tail index
 * 18-31: tail cpu (+1)
```

逐行解剖：

- **`union` 三视图**：同一个 32 bit 字，可以当一个 `atomic_t` 整体 CAS；也可以拆成 `locked`/`pending` 两个 u8 单独写；还可以拆成 `locked_pending`(u16)/`tail`(u16) 两半。**这是性能设计，不是花活**——pending 占满一整个字节（注释明说），使"清 pending + 置 locked"可以用一次 16 bit 普通写完成（`WRITE_ONCE(lock->locked_pending, _Q_LOCKED_VAL)`），不需要原子指令；unlock 只写 `locked` 一个字节。前提是架构支持 8/16 bit 原子性写——源码注释明确把这列为适用条件。
- **`tail = tail_cpu + tail_idx`**：MCS 的 8 字节 tail 指针被压成 14 bit CPU 编号（+1，0 表示空）+ 2 bit 索引。指针之所以可以变编号，是因为 MCS 节点是 **per-CPU 静态数组**，"CPU 号 + 数组下标"即可重建指针。
- **2 bit idx 哪来的**：一个 CPU 最多同时在 4 个上下文里各等一把 spinlock——task、softirq、hardirq、NMI（spinlock 不可重入自身上下文，所以每层最多一个）。4 = 2 bit。这是把"内核执行模型的不变量"直接编码进了位布局。
- **`NR_CPUS >= 16K` 分支**：CPU 数超过 16383 时 pending 缩成 1 bit、挪 tail 位宽，代价是 pending/tail 操作退化为整字 CAS（源码里 `_Q_PENDING_BITS == 8` 的两套实现就是为此）。

### 12.4.2 fastpath：一条 CAS

【真实源码 v6.12 `include/asm-generic/qspinlock.h`（节选：两函数间的预处理守卫已省略）】

```c
static __always_inline void queued_spin_lock(struct qspinlock *lock)
{
	int val = 0;

	if (likely(atomic_try_cmpxchg_acquire(&lock->val, &val, _Q_LOCKED_VAL)))
		return;

	queued_spin_lock_slowpath(lock, val);
}

static __always_inline void queued_spin_unlock(struct qspinlock *lock)
{
	/*
	 * unlock() needs release semantics:
	 */
	smp_store_release(&lock->locked, 0);
}
```

- lock fastpath：期望整个字为 0（无锁、无 pending、无队列），CAS 成 `_Q_LOCKED_VAL`(=1)。`_acquire` 后缀正是 12.1 节契约的落地。失败时把读到的 `val` 直接传给 slowpath——省一次重读。
- **unlock 永远是这一行**：对 `locked` 字节做 release store。不需要知道有没有队列、队列多长——这是 qspinlock 设计的精髓之一：**把"通知下一个等待者"的复杂度全部留在等待侧**，释放侧 O(1) 且无原子指令。对比：ticket 的 `owner++` 指令开销同级，但会触发全体等待者重读；MCS 原版 unlock 还要寻找 next 节点、API 上要传 node。qspinlock 把释放侧做到了"单条 store + 至多 2 个核重读"。

### 12.4.3 slowpath 总览：源码自带的状态机图

【真实源码 v6.12 `kernel/locking/qspinlock.c`（函数头注释，节选：省略参数说明行）】

```c
/**
 * queued_spin_lock_slowpath - acquire the queued spinlock
 * (queue tail, pending bit, lock value)
 *
 *              fast     :    slow                                  :    unlock
 *                       :                                          :
 * uncontended  (0,0,0) -:--> (0,0,1) ------------------------------:--> (*,*,0)
 *                       :       | ^--------.------.             /  :
 *                       :       v           \      \            |  :
 * pending               :    (0,1,1) +--> (0,1,0)   \           |  :
 *                       :       | ^--'              |           |  :
 *                       :       v                   |           |  :
 * uncontended           :    (n,x,y) +--> (n,0,0) --'           |  :
 *   queue               :       | ^--'                          |  :
 *                       :       v                               |  :
 * contended             :    (*,x,y) +--> (*,0,0) ---> (*,0,1) -'  :
 *   queue               :         ^--'                             :
 */
```

三元组 `(tail, pending, locked)`。两条慢路：**pending 路线**（第二个到达者）和 **queue 路线**（第三个及以后）。下面按真实代码走一遍。

### 12.4.4 pending 路线：两 CPU 争用不碰队列

【真实源码 v6.12 `kernel/locking/qspinlock.c`（slowpath 前半，节选）】

```c
	if (val == _Q_PENDING_VAL) {
		int cnt = _Q_PENDING_LOOPS;
		val = atomic_cond_read_relaxed(&lock->val,
					       (VAL != _Q_PENDING_VAL) || !cnt--);
	}

	/*
	 * If we observe any contention; queue.
	 */
	if (val & ~_Q_LOCKED_MASK)
		goto queue;

	/*
	 * trylock || pending
	 *
	 * 0,0,* -> 0,1,* -> 0,0,1 pending, trylock
	 */
	val = queued_fetch_set_pending_acquire(lock);

	if (unlikely(val & ~_Q_LOCKED_MASK)) {

		/* Undo PENDING if we set it. */
		if (!(val & _Q_PENDING_MASK))
			clear_pending(lock);

		goto queue;
	}

	/*
	 * We're pending, wait for the owner to go away.
	 *
	 * 0,1,1 -> *,1,0
	 *
	 * this wait loop must be a load-acquire such that we match the
	 * store-release that clears the locked bit and create lock
	 * sequentiality; ...
	 */
	if (val & _Q_LOCKED_MASK)
		smp_cond_load_acquire(&lock->locked, !VAL);

	/*
	 * take ownership and clear the pending bit.
	 *
	 * 0,1,0 -> 0,0,1
	 */
	clear_pending_set_locked(lock);
	lockevent_inc(lock_pending);
	return;
```

逐段精读：

1. **`val == _Q_PENDING_VAL`（(0,1,0)）的特判**：撞上"前任 owner 已释放、pending 持有者还没完成接力"的瞬间窗口。等它一下，但**有界**（`_Q_PENDING_LOOPS`：generic 默认 1，架构可覆盖调大）——源码注释直说原因：无界等待无法保证 forward progress。一个量级在纳秒的过渡态都要考虑活锁，这是内核级严谨。
2. **`val & ~_Q_LOCKED_MASK` → queue**：除 locked 外还有任何位非零（已有 pending 或队列），说明自己至少是第三个，直接排队。pending 位只留给第二名——这保证 pending 路线最多一人，否则多个 pending 自旋者会重演 ticket lock 的广播问题。
3. **`queued_fetch_set_pending_acquire`**：原子地 fetch-or 置 pending 并返回旧值。注意接下来的 **undo 逻辑**：置完才发现已有人排队/pending（race 窗口），若 pending 是自己置的要清掉再去排队——否则一个永远不会被接力的 pending 位会卡死 `n,0,0 -> 0,0,0` 转换，后继全部死等。**先乐观操作、发现冲突再撤销**，undo 路径的正确性比正常路径难想得多，这种代码每一行都对应一个具体的 race。
4. **`smp_cond_load_acquire(&lock->locked, !VAL)`**：只盯 `locked` **一个字节**自旋（不是整个 val），等 owner 的 release store。acquire/release 配对：owner 临界区内的所有写，保证对我可见。在 ARM64 上 `smp_cond_load_acquire` 还会编译成 WFE 类等待（见 12.4.7 mcs_spinlock.h 注释），自旋不烧功耗。
5. **`clear_pending_set_locked`**：`WRITE_ONCE(lock->locked_pending, _Q_LOCKED_VAL)`——就是 12.4.1 说的那次 16 bit 普通写，(0,1,0)→(0,0,1) 一步完成，无原子指令。为什么敢用普通写？因为此刻 pending 持有者是**唯一**有权改这两个字节的人（locked 已空、pending 是自己、队列动不了这两字节）——不变量给了免锁授权。

### 12.4.5 queue 路线：压缩版 MCS

【真实源码 v6.12 `kernel/locking/qspinlock.c`（slowpath 后半，节选）】

```c
queue:
	lockevent_inc(lock_slowpath);
pv_queue:
	node = this_cpu_ptr(&qnodes[0].mcs);
	idx = node->count++;
	tail = encode_tail(smp_processor_id(), idx);

	/*
	 * 4 nodes are allocated based on the assumption that there will
	 * not be nested NMIs taking spinlocks. ...
	 */
	if (unlikely(idx >= MAX_NODES)) {
		lockevent_inc(lock_no_node);
		while (!queued_spin_trylock(lock))
			cpu_relax();
		goto release;
	}

	node = grab_mcs_node(node, idx);

	/*
	 * Ensure that we increment the head node->count before initialising
	 * the actual node. If the compiler is kind enough to reorder these
	 * stores, then an IRQ could overwrite our assignments.
	 */
	barrier();

	node->locked = 0;
	node->next = NULL;
	pv_init_node(node);

	if (queued_spin_trylock(lock))
		goto release;

	/*
	 * Ensure that the initialisation of @node is complete before we
	 * publish the updated tail via xchg_tail() ...
	 */
	smp_wmb();

	old = xchg_tail(lock, tail);
	next = NULL;

	if (old & _Q_TAIL_MASK) {
		prev = decode_tail(old);

		/* Link @node into the waitqueue. */
		WRITE_ONCE(prev->next, node);

		pv_wait_node(node, prev);
		arch_mcs_spin_lock_contended(&node->locked);

		next = READ_ONCE(node->next);
		if (next)
			prefetchw(next);
	}

	val = atomic_cond_read_acquire(&lock->val, !(VAL & _Q_LOCKED_PENDING_MASK));

locked:
	if ((val & _Q_TAIL_MASK) == tail) {
		if (atomic_try_cmpxchg_relaxed(&lock->val, &val, _Q_LOCKED_VAL))
			goto release; /* No contention */
	}

	set_locked(lock);

	if (!next)
		next = smp_cond_load_relaxed(&node->next, (VAL));

	arch_mcs_spin_unlock_contended(&next->locked);
	pv_kick_node(lock, next);

release:
	__this_cpu_dec(qnodes[0].mcs.count);
}
```

逐段精读（这一段是本章最硬的部分，值得慢读）：

1. **`idx = node->count++` 与 `barrier()`**：per-CPU 的 4 个 qnode 用 `count` 做"嵌套层数游标"。`barrier()`（编译器屏障）保证 count 自增先于节点初始化落地——否则中断在两者之间到来、拿同一个 idx，会覆写我们的节点。**这是单核上的编译器问题，不是 SMP 问题**，所以用 `barrier()` 而非 `smp_*`。区分这两类屏障是源码精读的基本功。
2. **`idx >= MAX_NODES` 的退化路径**：注释直白承认——理论上 NMI 嵌套可能耗尽 4 个节点，此时直接放弃排队、裸自旋 trylock。"This is not the most elegant solution, but is simple enough."——**为极端罕见场景保留一条正确但不优雅的逃生路，比为它复杂化主路径更工程**。
3. **入队前再 trylock 一次**：刚 touch 过冷的 per-CPU cacheline，期间锁可能已经空了。一次几乎免费的投机，避免整套排队开销。
4. **`smp_wmb()` + `xchg_tail` 用 `_relaxed`**：先用写屏障保证节点初始化全局可见，再发布 tail。正因为屏障已显式给出，`xchg_tail` 才敢用 relaxed 语义（源码注释："We can use relaxed semantics since the caller ensures that the MCS node is properly initialized before updating the tail"）。**屏障打包思维**：把一次 wmb 的开销摊给后面所有 relaxed 操作。
5. **`old & _Q_TAIL_MASK` 分支**：xchg 返回的旧 tail 非空 → 有前驱 → `decode_tail` 重建前驱指针 → `WRITE_ONCE(prev->next, node)` 挂链 → `arch_mcs_spin_lock_contended(&node->locked)` 在**自己节点**上自旋。这就是 MCS 的 O(1) cache 流量本体。`prefetchw(next)`：还在等的时候就把后继节点的 cacheline 以写意图预取——为将来的 unlock 提前付 cache miss，把延迟藏进等待时间。锱铢必较到这个程度。
6. **队头等待 `!(VAL & _Q_LOCKED_PENDING_MASK)`**：队头不只等 locked 清零，还要等 pending 清零——**pending 持有者优先级高于队列**（它先来）。acquire 语义与 owner/pending 者的 release store 配对。
7. **出队的两种收尾**：若 `val` 的 tail 还是自己（我是唯一排队者）→ 一次 CAS 把 `(n,0,0)` 直接换成 `(0,0,1)`，队列清空；CAS 失败说明有新人正在入队 → 走 `set_locked()`（只写 locked 字节）+ 等 `node->next` 非空 + `arch_mcs_spin_unlock_contended(&next->locked)`（release store 接力给后继）。注意 CAS 用 `_relaxed`：源码在等待循环处注释过，序已由前面的 acquire 等待建立，这里不需要重复付费。

### 12.4.6 场景跑一遍：4 个 CPU 抢一把锁

把上面的代码变成时序。`(tail, pending, locked)`，tail 记为持有者 CPU 编号（实际是 encode 后的值）：

| 时刻 | 事件 | 锁字状态 | 各 CPU 在干什么 |
|---|---|---|---|
| t0 | CPU0 fastpath CAS 成功 | (0,0,1) | CPU0 进临界区 |
| t1 | CPU1 fastpath 失败 → slowpath：无争用位 → fetch_set_pending | (0,1,1) | CPU1 自旋在 `locked` 字节 |
| t2 | CPU2 到达：`val & ~LOCKED` 非零（pending 在）→ queue：取 qnodes[0]、xchg_tail，旧 tail 空 → 自己即队头 | (C2,1,1) | CPU2 自旋在整字 `val`，等 locked+pending 双清 |
| t3 | CPU3 到达：同样 queue，xchg_tail 返回 C2 → 挂到 CPU2 节点后 | (C3,1,1) | CPU3 自旋在**自己节点的 locked 字段**（本地！） |
| t4 | CPU0 unlock：release store `locked=0` | (C3,1,0) | 重读锁字的只有 CPU1 与队头 CPU2（≤2 核）；CPU3 在自己的 per-CPU line 上不受影响 |
| t5 | CPU1 `clear_pending_set_locked` | (C3,0,1) | CPU1 进临界区；CPU2 看到 pending 清了，继续等 locked |
| t6 | CPU1 unlock | (C3,0,0) | CPU2 的等待条件满足 |
| t7 | CPU2：tail≠自己（是 C3）→ `set_locked` | (C3,0,1) | CPU2 进临界区；写 `next->locked=1` 把 CPU3 提升为队头 |
| t8 | CPU3 成为队头，改盯全局 val | (C3,0,1) | CPU3 等 CPU2 释放 |
| t9 | CPU2 unlock；CPU3 的 CAS：tail==自己 → (0,0,1)，队列消失 | (0,0,1) | CPU3 进临界区 |

对照 ticket lock 跑同一场景：t4 时刻 ticket 的 `owner++` 会同时 invalidate CPU1/2/3 三个核的 cache line，三个核同时跨核重读；qspinlock 下盯锁字的**永远至多两个核**（pending 持有者 + 队头），其余排队者全在自己的 per-CPU line 上等。**N 个等待者，每次释放的一致性流量：ticket O(N)，qspinlock O(1)（上界 2）**——这就是 commit message 里 "much faster in high contention" 的微架构本体。

**不适用边界**（对照表之外的思考）：单线程/低争用下 qspinlock 的 slowpath 比 ticket 复杂（commit message 自己承认轻中度争用可能略慢）；2 CPU 系统上 pending 路线几乎总能命中、MCS 机器从不启动，此时它约等于一个带额外分支的 TAS 锁。qspinlock 赢在的是"几十核都在抢"的尾部场景——大多数锁一辈子遇不到，但遇到的那一刻决定系统是否雪崩。

### 12.4.7 MCS 节点与 arch 钩子

【真实源码 v6.12 `kernel/locking/mcs_spinlock.h`（节选）】

```c
struct mcs_spinlock {
	struct mcs_spinlock *next;
	int locked; /* 1 if lock acquired */
	int count;  /* nesting count, see qspinlock.c */
};

/*
 * Using smp_cond_load_acquire() provides the acquire semantics
 * required so that subsequent operations happen after the
 * lock is acquired. Additionally, some architectures such as
 * ARM64 would like to do spin-waiting instead of purely
 * spinning, and smp_cond_load_acquire() provides that behavior.
 */
#define arch_mcs_spin_lock_contended(l)					\
do {									\
	smp_cond_load_acquire(l, VAL);					\
} while (0)

/*
 * smp_store_release() provides a memory barrier to ensure all
 * operations in the critical section has been completed before
 * unlocking.
 */
#define arch_mcs_spin_unlock_contended(l)				\
	smp_store_release((l), 1)
```

- 节点 16 字节，4 个正好一条 64B cacheline（qspinlock.c 注释明说，PV 模式下扩到 32 字节、一线两个）。
- "ARM64 would like to do spin-waiting"：`smp_cond_load_acquire` 在 ARM64 上用 LDXR/WFE 实现——核进低功耗等待，cache line 被改写时由硬件事件唤醒。**同一份通用代码，在不同微架构上有不同的等待物理形态**，这是 arch 钩子层存在的意义。

## 12.5 设计考古 III：mutex 的诞生（2005 年 12 月的火药味）

### 一场关于"要不要新原语"的辩论

2005 年底之前，内核可睡眠互斥靠 semaphore（`down()/up()`）。David Howells 提出把 semaphore 收窄成 mutex 的初版 patch，Linus 的回应（LWN *Semaphores and mutexes*，2005-12-22）毫不留情：

> "A patch that creates a non-counting mutex .. that is SLOWER than the current counting one .. is simply INCREDIBLY BROKEN." —— Linus Torvalds

Andrew Morton 站在同一边：

> "down() and up() work just fine and they're small, efficient, well-debugged and well-understood."

几天后 Ingo Molnar 带着从 **realtime preemption 树**抽出来的 mutex 实现入场，策略完全不同：**独立数据类型、独立 API（`mutex_lock/mutex_unlock`）、不动 semaphore、不搞 flag day**，子系统自愿逐个迁移。性能数据：8 路系统上某文件系统 benchmark **快了一倍以上**。

转折点是一个考古级发现：调查性能差异时发现 x86 semaphore 的唤醒路径**每次唤醒两个进程**，制造无谓争用——一个藏了多年的 bug。Ingo 的总结一锤定音：

> "If this really is a bug that hid for years, it shows that the semaphore code is too complex to be properly reviewed."

2.6.16 合入：commit `6053ee3b32e3437e8c1e72687850f436e779bd49`（"[PATCH] mutex subsystem, core"，Ingo Molnar，2006-01-09，Signed-off-by Ingo Molnar + Arjan van de Ven，message 只有一句："mutex implementation, core files: just the basic subsystem, no users of it."——增量合入策略的教科书示范）。

**mutex 从 semaphore 身上换来的语义收紧**：严格所有权（谁锁谁解）、不可递归、不可在中断上下文用。每条限制都是换取性能与可验证性的筹码——所有权确定才能做 owner 字段优化（后述）、才能做优先级继承、lockdep 才能给出强检查。**语义越窄，实现越快，工具越强**。

### 两次大升级（都已核实）

- **乐观自旋（adaptive spinning）**：commit `0d66bf6d3514`（Peter Zijlstra，2009-01-12，"mutex: implement adaptive spinning"），合入 **v2.6.30**（compare API 核实：不在 .29，在 .30）。v6.12 的 mutex.c 文件头至今保留这段血统记录：

## 12.6 源码精读 II：mutex（v6.12）

### 12.6.1 数据结构：一个 long 装下 owner + 三个标志位

【真实源码 v6.12 `include/linux/mutex_types.h`】

```c

	atomic_long_t		owner;
	raw_spinlock_t		wait_lock;
#ifdef CONFIG_MUTEX_SPIN_ON_OWNER
	struct optimistic_spin_queue osq; /* Spinner MCS lock */
#endif
```

【真实源码 v6.12 `kernel/locking/mutex.c`】

```c
/*
 * @owner: contains: 'struct task_struct *' to the current lock owner,
 * NULL means not owned. Since task_struct pointers are aligned at
 * at least L1_CACHE_BYTES, we have low bits to store extra state.
 *
 * Bit0 indicates a non-empty waiter list; unlock must issue a wakeup.
 * Bit1 indicates unlock needs to hand the lock to the top-waiter
 * Bit2 indicates handoff has been done and we're waiting for pickup.
 */
#define MUTEX_FLAG_WAITERS	0x01
#define MUTEX_FLAG_HANDOFF	0x02
#define MUTEX_FLAG_PICKUP	0x04
```

- **`owner` 是整个设计的支点**：一个 `atomic_long_t` 同时编码"谁持有"（task_struct 指针）+ 三个状态位（指针按 L1 cacheline 对齐，低位天然空闲）。一次原子读就能同时回答"锁空吗、谁拿着、有没有人等、要不要移交"——fastpath/乐观自旋的全部判断依据压在一个字里，**一条 cacheline 的事实来源**。
- 三个 flag 合起来是一个微型协议：`WAITERS`（unlock 时得去唤醒）→ `HANDOFF`（队首受够了，unlock 时必须指名移交）→ `PICKUP`（已移交，只有指定人能拿）。后面 slowpath 会看到三态如何流转。
- `wait_lock` 是 `raw_spinlock_t`（注意不是 `spinlock_t`——mutex 自己是 RT 下的可睡眠体，但保护 wait_list 的内层锁必须永远是真自旋）。
- 同文件里还有 PREEMPT_RT 形态：`struct mutex { struct rt_mutex_base rtmutex; ... }`——RT 内核下 mutex 整体换芯为支持优先级继承的 rt_mutex。

### 12.6.2 fastpath：无争用 = 一条 CAS

【真实源码 v6.12 `kernel/locking/mutex.c`】

```c
static __always_inline bool __mutex_trylock_fast(struct mutex *lock)
{
	unsigned long curr = (unsigned long)current;
	unsigned long zero = 0UL;

	if (atomic_long_try_cmpxchg_acquire(&lock->owner, &zero, curr))
		return true;

	return false;
}

static __always_inline bool __mutex_unlock_fast(struct mutex *lock)
{
	unsigned long curr = (unsigned long)current;

	return atomic_long_try_cmpxchg_release(&lock->owner, &curr, 0UL);
}
```

- lock：期望 owner 整字为 0（没人持有**且三个 flag 全空**），CAS 成 `current`。任何一个 flag 非零都会让 fastpath 失败落入慢路——比如 `WAITERS` 在，就必须走慢路处理唤醒义务。把"有没有特殊情况"的判断免费搭载在同一条 CAS 上，零额外分支。
- unlock：期望 owner 正好是自己且无 flag，CAS 回 0。`_acquire`/`_release` 与 12.1 契约精确对应。
- 对比 spinlock fastpath：同样一条 CAS。**无争用时 mutex 并不比 spinlock 贵**——"mutex 慢"是一个流传甚广的过时直觉，慢的是争用后的路径形态（睡眠 vs 自旋），不是无争用开销。

### 12.6.3 midpath：乐观自旋与 OSQ

【真实源码 v6.12 `kernel/locking/mutex.c`（节选）】

```c
/*
 * Look out! "owner" is an entirely speculative pointer access and not
 * reliable.
 *
 * "noinline" so that this function shows up on perf profiles.
 */
static noinline
bool mutex_spin_on_owner(struct mutex *lock, struct task_struct *owner,
			 struct ww_acquire_ctx *ww_ctx, struct mutex_waiter *waiter)
{
	bool ret = true;

	lockdep_assert_preemption_disabled();

	while (__mutex_owner(lock) == owner) {
		/*
		 * Ensure we emit the owner->on_cpu, dereference _after_
		 * checking lock->owner still matches owner. And we already
		 * disabled preemption which is equal to the RCU read-side
		 * crital section in optimistic spinning code. Thus the
		 * task_strcut structure won't go away during the spinning
		 * period
		 */
		barrier();

		/*
		 * Use vcpu_is_preempted to detect lock holder preemption issue.
		 */
		if (!owner_on_cpu(owner) || need_resched()) {
			ret = false;
			break;
		}

		cpu_relax();
	}

	return ret;
}
```

（节选去掉了 ww_mutex 分支；完整逻辑见原文件 352–387 行。）

- **自旋的赌注**："owner 正在某个 CPU 上跑 → 它大概率马上释放 → 自旋比睡眠+唤醒（μs 级调度代价）划算"。三个止损条件：owner 换人（重试 trylock）、owner 不在跑（`owner_on_cpu`，赌输了，去睡）、自己该让出 CPU（`need_resched`，别为了省 μs 把调度延迟搞炸）。
- **`vcpu_is_preempted`**（`owner_on_cpu` 内部）：虚拟化场景的同款问题——owner 的 vCPU 可能被 hypervisor 切走，物理上根本没在跑，`on_cpu` 还是 1。没有这个检查，云上 guest 会集体空转。和 qspinlock 的 paravirt 分支呼应：**虚拟化把"运行中"变成了不可信状态，每个自旋决策都要打折**。
- **`barrier()` + "speculative pointer"注释**：owner 指针随时可能失效，安全性靠"关抢占 ≈ RCU 读临界区"保证 task_struct 不被释放（与 task_struct 的 RCU 释放规则联动）。**指针解引用的安全性论证写进注释**——这种代码评审文化值得照搬。
- **`noinline` 的理由是 perf 可观测性**：故意阻止内联，让这个函数在火焰图上有名有姓。性能工具友好度被当成 API 设计约束。
- **OSQ（`osq_lock`）为什么存在**：多个 CPU 同时想乐观自旋时，如果都直接盯 `lock->owner`，又是一场 cacheline 乒乓。于是自旋者先排进一条 MCS 队列（`kernel/locking/osq_lock.c`，per-CPU 节点 `optimistic_spin_node`），**只有队头真正自旋 owner 字段**。OSQ 与 qspinlock 的 MCS 同源但更复杂：自旋者可能中途 `need_resched` 要**从队列中间退出**（unqueue），所以节点有 `prev` 指针、有三步取消协议（osq_lock.c 里 Step A/B/C 注释）。qspinlock 等待者不可取消，单链即可。**"可取消"这一个需求差异，让数据结构从单链变双链、代码量翻几倍**——需求对复杂度的杠杆作用。
- 另一个差异：OSQ 节点每 CPU 只有 **1 个**（qspinlock 是 4 个）——osq_lock.c 注释解释：sleeping lock 不会在中断上下文获取，自旋期间关抢占，所以不存在嵌套。**上下文规则直接决定内存布局**。

### 12.6.4 slowpath 与 handoff：防饿死协议

【真实源码 v6.12 `kernel/locking/mutex.c` `__mutex_lock_common`（核心循环，节选）】

```c
	raw_spin_lock(&lock->wait_lock);
	/*
	 * After waiting to acquire the wait_lock, try again.
	 */
	if (__mutex_trylock(lock)) {
		...
		goto skip_wait;
	}

	...
	/* add waiting tasks to the end of the waitqueue (FIFO): */
	__mutex_add_waiter(lock, &waiter, &lock->wait_list);

	set_current_state(state);
	for (;;) {
		if (__mutex_trylock(lock))
			goto acquired;

		if (signal_pending_state(state, current)) {
			ret = -EINTR;
			goto err;
		}

		raw_spin_unlock(&lock->wait_lock);
		schedule_preempt_disabled();

		first = __mutex_waiter_is_first(lock, &waiter);

		set_current_state(state);
		/*
		 * Here we order against unlock; we must either see it change
		 * state back to RUNNING and fall through the next schedule(),
		 * or we must see its unlock and acquire.
		 */
		if (__mutex_trylock_or_handoff(lock, first))
			break;

		if (first) {
			trace_contention_begin(lock, LCB_F_MUTEX | LCB_F_SPIN);
			if (mutex_optimistic_spin(lock, ww_ctx, &waiter))
				break;
			trace_contention_begin(lock, LCB_F_MUTEX);
		}

		raw_spin_lock(&lock->wait_lock);
	}
```

协议逐步还原：

1. **三层递降**：`__mutex_trylock`（fast）→ `mutex_optimistic_spin`（mid，函数开头处，节选未含）→ 挂 `wait_list` 睡眠（slow）。`Documentation/locking/mutex-design.rst`（v6.12）对此的定性值得整句记住："While formally kernel mutexes are sleepable locks, it is path (ii) that makes them more practically a hybrid type."
2. **每次醒来先 `trylock_or_handoff(lock, first)`**：如果我是队首（`first`），这个调用会在 trylock 失败时顺手置上 `MUTEX_FLAG_HANDOFF`。下一次 unlock 看到 HANDOFF 位，就不再公平竞争，而是**指名移交**：把 owner 直接写成队首任务并置 `PICKUP` 位——从此任何 spinner 的 trylock 都会失败（`__mutex_trylock_common` 里 `PICKUP` 在且不是本人则 break）。
3. **饿死是怎么被精确堵住的**：乐观 spinner 物理上比睡眠者醒得快，没有 handoff 时队首可能被无限插队（v4.10 之前的真实 bug 域）。handoff 的设计聪明在**不是一上来就公平**——第一次排队照旧允许窃取（吞吐优先），只有"醒来发现又被偷了"的队首才升级为强制移交（公平兜底）。吞吐与公平不是二选一，是**分层触发**。
4. **队首专属自旋**：成为队首后还能以 waiter 身份再做乐观自旋（`mutex_optimistic_spin(..., &waiter)`，不进 OSQ）——醒着的队首与 OSQ 队头双线竞争，谁先逮到算谁的（但 handoff 一旦置位，胜者必是队首）。
5. **`schedule_preempt_disabled()` 前后的状态机舞步**：`set_current_state(state)` 与 unlock 侧的唤醒之间靠 wait_lock + 内存序闭环，注释里"we must either see it change state back to RUNNING ... or we must see its unlock"是教科书级的 lost-wakeup 论证。

### 12.6.5 一个被注释钉死的 UAF 坑

【真实源码 v6.12 `kernel/locking/mutex.c`（`mutex_unlock` 文档注释，节选）】

```c
 * The caller must ensure that the mutex stays alive until this function has
 * returned - mutex_unlock() can NOT directly be used to release an object such
 * that another concurrent task can free it.
 * Mutexes are different from spinlocks & refcounts in this aspect.
```

场景：用 mutex 保护对象，约定"最后一个使用者 unlock 后 free"。错。`mutex_unlock` 的慢路在改完 owner 之后还要碰 `wait_lock/wait_list`——另一个任务可能在 owner 清零的瞬间拿到锁并 free 对象，此时 unlock 还没走完，**unlock 自己 UAF**。spinlock 的 unlock 是单条 store 所以没这个窗口。生产中这类"锁保护自己所在对象的生命周期"的 bug 隐蔽且致命，正确做法是引用计数（`refcount_t`）或 RCU 管生命周期、锁只管互斥。

### 12.6.6 场景跑一遍：三条路径的真实决策

设 CPU0 持有 mutex 正在跑，CPU1/CPU2 先后 `mutex_lock`：

| 阶段 | CPU1 | CPU2 | owner 字 |
|---|---|---|---|
| 到达 | fastpath CAS 失败（owner=CPU0task） | 同失败 | `task0` |
| 决策 | `mutex_can_spin_on_owner`：owner on_cpu → 进 OSQ，成为 OSQ 队头，自旋 owner | 进 OSQ 排在 CPU1 后，自旋**自己的 OSQ 节点** | `task0` |
| CPU0 释放 | 看到无 flag → fast unlock CAS 成功；CPU1 的 trylock 命中 | 还在 OSQ 里等 | `task1` |
| CPU1 持锁但被调度走 | —— | `mutex_spin_on_owner` 发现 `!owner_on_cpu` → 退出 OSQ → 挂 wait_list 睡眠，置 `WAITERS` | `task1\|WAITERS` |
| CPU1 释放 | 慢路 unlock：见 WAITERS → 唤醒队首 CPU2 | 醒来 trylock；若期间被第三方 spinner 偷锁，则下轮置 HANDOFF | `task2` |

三条路径的延迟量级：fast ≈ 单 CAS（几十 ns 内）；mid ≈ 自旋至 owner 释放（赌对时远小于 1μs）；slow ≈ 两次上下文切换（数 μs 起步）。**乐观自旋的全部价值 = 把大量本该走 slow 的争用降级到 mid**，mutex-design.rst 原话："the performance of this lock has been seen to significantly improve a number of workloads."

## 12.7 源码精读 III：seqlock 家族（v6.12）

### 12.7.1 一族三层：seqcount_t → seqcount_LOCKNAME_t → seqlock_t

【真实源码 v6.12 `include/linux/seqlock_types.h`（节选）】

```c
/*
 * Sequence counters (seqcount_t)
 *
 * This is the raw counting mechanism, without any writer protection.
 *
 * Write side critical sections must be serialized and non-preemptible.
 *
 * This mechanism can't be used if the protected data contains pointers,
 * as the writer can invalidate a pointer that a reader is following.
 */
typedef struct seqcount {
	unsigned sequence;
#ifdef CONFIG_DEBUG_LOCK_ALLOC
	struct lockdep_map dep_map;
#endif
} seqcount_t;

#define SEQCOUNT_LOCKNAME(lockname, locktype, preemptible, lockbase)	\
typedef struct seqcount_##lockname {					\
	seqcount_t		seqcount;				\
	__SEQ_LOCK(locktype	*lock);					\
} seqcount_##lockname##_t;

SEQCOUNT_LOCKNAME(raw_spinlock, raw_spinlock_t,  false,    raw_spin)
SEQCOUNT_LOCKNAME(spinlock,     spinlock_t,      __SEQ_RT, spin)
SEQCOUNT_LOCKNAME(rwlock,       rwlock_t,        __SEQ_RT, read)
SEQCOUNT_LOCKNAME(mutex,        struct mutex,    true,     mutex)

typedef struct {
	seqcount_spinlock_t seqcount;
	spinlock_t lock;
} seqlock_t;
```

- **三层分工**：`seqcount_t` 是裸序号（写者互斥自己保证）；`seqcount_LOCKNAME_t` 把"哪把锁在保护写者"编码进**类型**；`seqlock_t` = seqcount + 内置 spinlock 的全家桶。
- **"can't be used if the protected data contains pointers"**——指针禁令不是博客经验，是类型定义处的官方注释。原因：读者无锁追指针时，写者可能已释放指针指向的内存（UAF）。
- `SEQCOUNT_LOCKNAME` 宏家族是为 PREEMPT_RT 铺的路（见 12.7.4）；这套关联锁 API 大约在 5.8–5.10 周期完成现代化改造（具体系列与作者「待核」，但 v6.12 形态如上，确凿）。

### 12.7.2 读写协议源码：每个 barrier 都有名有姓

【真实源码 v6.12 `include/linux/seqlock.h`（节选）】

```c
#define __read_seqcount_begin(s)					\
({									\
	unsigned __seq;							\
									\
	while ((__seq = seqprop_sequence(s)) & 1)			\
		cpu_relax();						\
									\
	kcsan_atomic_next(KCSAN_SEQLOCK_REGION_MAX);			\
	__seq;								\
})

static inline int do_read_seqcount_retry(const seqcount_t *s, unsigned start)
{
	smp_rmb();
	return do___read_seqcount_retry(s, start);
}

static inline void do_raw_write_seqcount_begin(seqcount_t *s)
{
	kcsan_nestable_atomic_begin();
	s->sequence++;
	smp_wmb();
}

static inline void do_raw_write_seqcount_end(seqcount_t *s)
{
	smp_wmb();
	s->sequence++;
	kcsan_nestable_atomic_end();
}
```

（reader 起点的 `seqprop_sequence` 对 `seqcount_t` 退化为 `smp_load_acquire(&s->sequence)`——seqlock.h 211 行。）

**协议全貌**（写者序号奇=写中、偶=空闲）：

```
writer:  seq++ (odd)  → smp_wmb → 写数据 → smp_wmb → seq++ (even)
reader:  seq1 = load_acquire(seq)（奇数则自旋）→ 读数据 → smp_rmb → seq2 = READ_ONCE(seq)
         seq1 != seq2 → 全部作废重来
```

**四个屏障逐个证明必要性**（推演掉任何一个会发生什么）：

1. **writer 第一个 `smp_wmb`**（`begin`：`++` 之后）：没有它，数据写可能重排到 `seq++` 之前可见 → 读者在序号还是偶数时就看到半新数据，且 retry 检测不到（首尾序号一致）。
2. **writer 第二个 `smp_wmb`**（`end`：`++` 之前）：没有它，`seq++`（变偶）可能先于数据写可见 → 读者看到"已写完"的序号 + 没写完的数据。
3. **reader 的 `load_acquire`**：保证数据读不会被乱序到读 seq1 **之前**（否则读了旧数据配新序号，假阴性逃逸）。
4. **reader 的 `smp_rmb`**（retry 里）：保证数据读全部完成后才读 seq2。没有它，seq2 的读可能被提前，撕裂写发生在"seq2 已读、数据还在读"的窗口里——retry 通过但数据是坏的。

**读侧代码的纪律**（从协议推出，不是风格建议）：临界区内读到的值可能是任意撕裂的中间态，所以**只能搬进局部变量，不能解引用、不能除、不能 `BUG_ON`、不能做任何不可回滚的事**，一切使用必须等 retry 通过。

还有一处细节：`kcsan_atomic_next(KCSAN_SEQLOCK_REGION_MAX)`——seqlock 读者按定义就是在和写者 data race（良性、有协议兜底），KCSAN（data race 检测器）会报警，所以源码内嵌了"接下来这段是协议内 race，别报"的标注。**新工具（KCSAN）逼着老原语（seqlock）把自己的非常规行为显式声明**，工具链和原语是共同演化的。

### 12.7.3 真实用例：timekeeping

【真实源码 v6.12 `kernel/time/timekeeping.c`（节选）】

```c
static struct {
	seqcount_raw_spinlock_t	seq;
	struct timekeeper	timekeeper;
} tk_core ____cacheline_aligned = {
```

### 12.7.4 PREEMPT_RT 下的活锁问题与"读者助推"

经典 seqlock 在 RT 内核下有个致命组合：写者可被抢占（RT 里 spinlock 临界区可抢占）→ 低优先级写者持有奇数序号时被高优先级读者抢占 → 读者在 `while (seq & 1)` 里自旋 → 写者永远没机会跑完 → **livelock**。v6.12 的解法直接写在 `seqprop_sequence` 的生成宏里：

【真实源码 v6.12 `include/linux/seqlock.h`（`SEQCOUNT_LOCKNAME` 宏内，节选）】

```c
	unsigned seq = smp_load_acquire(&s->seqcount.sequence);		\
									\
	if (!IS_ENABLED(CONFIG_PREEMPT_RT))				\
		return seq;						\
									\
	if (preemptible && unlikely(seq & 1)) {				\
		__SEQ_LOCK(lockbase##_lock(s->lock));			\
		__SEQ_LOCK(lockbase##_unlock(s->lock));			\
									\
		/*							\
		 * Re-read the sequence counter since the (possibly	\
		 * preempted) writer made progress.			\
		 */							\
		seq = smp_load_acquire(&s->seqcount.sequence);		\
	}								\
	return seq;
```

读者发现序号是奇数（写者在写）时，**去 lock 再立刻 unlock 写者的那把锁**。这一对操作看似无意义，实际是优先级继承的触发器：RT 下锁是 rt_mutex，读者去 lock 会把自己的高优先级**继承给持锁的写者**，写者被助推跑完，unlock 返回时写已结束。这就是 `seqcount_LOCKNAME_t` 必须把关联锁编码进类型的根本原因——**RT 读者需要知道"该去推谁"**。一个为非 RT 世界发明的无锁读协议，靠类型系统改造在 RT 世界活了下来。

## 12.8 源码精读 IV：rwsem 的现代形态（v6.12，简读）

rwsem 的文件头是一部浓缩演进史，credit roll 直接列在源码里：

【真实源码 v6.12 `kernel/locking/rwsem.c`（节选）】

```c
/* kernel/rwsem.c: R/W semaphores, public implementation
 *
 * Written by David Howells (dhowells@redhat.com).
 * Derived from asm-i386/semaphore.h
 *
 * Writer lock-stealing by Alex Shi <alex.shi@intel.com>
 * and Michel Lespinasse <walken@google.com>
 *
 * Optimistic spinning by Tim Chen <tim.c.chen@intel.com>
 * and Davidlohr Bueso <davidlohr@hp.com>. Based on mutexes.
 *
 * Rwsem count bit fields re-definition and rwsem rearchitecture by
 * Waiman Long <longman@redhat.com> and
 * Peter Zijlstra <peterz@infradead.org>.
 */
```

```c
/*
 * On 64-bit architectures, the bit definitions of the count are:
 *
 * Bit  0    - writer locked bit
 * Bit  1    - waiters present bit
 * Bit  2    - lock handoff bit
 * Bits 3-7  - reserved
 * Bits 8-62 - 55-bit reader count
 * Bit  63   - read fail bit
 */
```

要点速读：

- **count 一个字编码全部状态**：bit0 写锁、bit1 有等待者、bit2 handoff（和 mutex 同款防饿死协议）、bit8-62 读者计数。读者获取 = `atomic_long_add(READER_BIAS)` 一条指令，然后看结果判断有没有撞上写者。
- **owner 字段的"hint"语义**：写者持有时 owner 精确；读者持有时只记"最后一个进来的读者 + READER_OWNED 位"，源码注释坦白："Ideally we would like to track all the readers that own a rwsem, but the overhead is simply too big."——**精确性向开销低头的显式决策**，调试 reader-owned rwsem 挂死时要知道 owner 只是线索不是真相。
- **乐观自旋的不对称**：写者等待时可对 owner 自旋（从 mutex 抄来）；owner 是读者群时没有单一 owner 可盯，自旋会盲赌，于是有 `RWSEM_NONSPINNABLE` 位——写者自旋超时就把锁标记为"别再自旋"。读多场景里 rwsem 的写者注定走慢路，这是它和 RCU 的根本差距。
- 5.3 周期的 rearchitecture（Waiman Long）重定义了上述位布局并统一了 spinlock 路线的实现（此前 32/64 位两套汇编）。

**写者饥饿**仍是 rwsem 的固有倾向（读者源源不断时写者等不到全空窗口），handoff 位缓解但不消除——读写比例失衡到一定程度，正确答案是换 RCU 而不是调 rwsem。

## 12.9 RT 语义改写：spinlock_t 的两幅面孔（v6.12 主线化完成）

2024-09-20，PREEMPT_RT 在 **6.12 合并窗口完成主线化**（最后的 blocker 是 printk 重构；首批覆盖 x86/x86_64、ARM64、RISC-V）——从 2004 年 -rt 树算起整整 20 年。这让"spinlock 不可睡眠"从公理降级为配置项：

【示意代码，非逐字：结构忠实于 v6.12 `include/linux/spinlock_types.h` + `spinlock_types_raw.h`，注释与省略为本书所加】

```c
typedef struct raw_spinlock {
	arch_spinlock_t raw_lock;        /* x86 上即 qspinlock（12.4 节） */
	...
} raw_spinlock_t;

#ifndef CONFIG_PREEMPT_RT
typedef struct spinlock {
	union {
		struct raw_spinlock rlock;   /* 非 RT：spinlock_t 就是 raw_spinlock */
		...
	};
} spinlock_t;
#else
/* PREEMPT_RT kernels map spinlock to rt_mutex */
typedef struct spinlock {
	struct rt_mutex_base	lock;    /* RT：换芯为可睡眠、带优先级继承的 rt_mutex */
	...
} spinlock_t;
#endif
```

- **非 RT**：`spinlock_t` ≈ `raw_spinlock_t` ≈ qspinlock，三层只是封装。
- **RT**：`spinlock_t` 整体换成 `rt_mutex_base`——临界区变得可抢占、可睡眠（所以 RT 下持"spinlock"再分配内存不再是 bug），优先级继承内建。而 `raw_spinlock_t` 保持真自旋 + 关抢占，留给调度器、中断入口、timekeeping 这些**物理上不能睡**的路径。
- 工程后果：**选 `spinlock_t` 还是 `raw_spinlock_t` 从"风格问题"变成"语义声明"**——你在回答"我的临界区能否容忍被抢占"。lockdep 的 `wait_type`（`LD_WAIT_SPIN` vs `LD_WAIT_CONFIG`，见 spinlock_types_raw.h 的 DEP_MAP 初始化）会静态检查嵌套合法性：raw 锁内不许拿 RT 下会睡的锁。
- 优先级反转的教科书案例仍是 **Mars Pathfinder（1997，VxWorks）**：低优先级气象线程持 mutex、中优先级通信线程抢占它、高优先级总线线程超时触发 watchdog 整机重启；地面复现后远程把 mutex 参数改为开启优先级继承修复。Linux 的 `rt_mutex` 用同款 PI 机制，并经由 futex(PI) 暴露给用户态。

**中断变体速查**（语义在 RT/非 RT 下均成立，实现不同）：

```c
spin_lock(lock);                 // 不动中断
spin_lock_irq(lock);             // 关本地中断；unlock 无条件开 → 嵌套场景错误之源
spin_lock_irqsave(lock, flags);  // 保存再关；unlock 恢复原状 → 默认选这个
spin_lock_bh(lock);              // 关 softirq
```

同一把锁同时被进程上下文与中断上下文使用而不用 `_irqsave`，等于在本 CPU 上制造"自旋等自己"死锁——这正是 lockdep irq-safety 检查（12.10）盯的模式。

## 12.10 正确性工具链：lockdep、KCSAN、LKMM

### 设计考古 IV：lockdep（2006）

Ingo Molnar 2006-05 以 **61-part patch series** 发布 lock validator（LWN *The kernel lock validator*，Corbet，2006-05-31；先进 -mm（2.6.17-rc5-mm1），随后于 2.6.18 窗口进入主线）。两个奠基性设计：

1. **lock class 而非 lock instance**：静态锁以地址为 key，动态锁以初始化点的静态 key 归类——"所有 inode 里的那把锁"是**一个**类。万级实例坍缩成数百个类，依赖图才可维护。这是 lockdep 能跑在每次 `make` 出的开发内核里的前提。
2. **检查的是"可能性"而非"事实"**：LWN 原文："the validator can extrapolate a much larger set of possible sequences"——只要 `A→B` 与 `B→A` 两条边在**任意时间、任意任务**上各出现过一次，环就成立，不需要真死锁发生。一次单线程跑过的代码路径就能宣判未来的并发死锁。

加上 irq-safety 规则（LWN 原文："Any spinlock which is acquired by a hardware interrupt handler can never be held when interrupts are enabled"），以及 chain hash 缓存（同样的持锁序列第二次出现直接查表跳过检查，控制 overhead）。

**局限与误报处理**（staff 必备）：

- 它仍是**动态**工具：没执行到的锁序看不见，覆盖率即天花板。
- lock class 粒度造成经典误报：父子目录 inode 按地址序加锁是合法的，但同类两实例在 lockdep 眼里是自环。解法是 subclass 注解：`mutex_lock_nested(lock, subclass)`、`lockdep_set_subclass()`、`mutex_lock_nest_lock()`（声明外层锁已串行化这组获取）。**注解用错方向会把真死锁一起静音**，每个 `_nested` 都该对照子系统的锁层级文档。
- 生产不开：内存与 CPU 开销大，标准做法是 CI/压测内核全开（`CONFIG_PROVE_LOCKING`），线上关闭。
- 职责边界：lockdep 不查"没加锁的裸竞争"——那是 **KCSAN**（`CONFIG_KCSAN`，~5.8+）的领域；"持 spinlock 睡眠"由 `CONFIG_DEBUG_ATOMIC_SLEEP` 的 `might_sleep()` 网兜住。三件工具三类 bug，混为一谈是面试常见扣分点。

### 设计考古 V：LKMM——把 memory-barriers.txt 形式化（4.17）

`memory-barriers.txt` 长约 3000 行（v6.12 实测 3014 行），人脑推内存序极易出错。2017 年 Jade Alglave、Luc Maranget、Paul McKenney、Andrea Parri、Alan Stern 发布形式化的 **Linux Kernel Memory Model**（LWN *A formal kernel memory-ordering model (part 1)*，2017-04-14）：

> "the first realistic automated representation of Linux-kernel memory ordering, and is also the first to incorporate read-copy-update (RCU) ordering properties."

工作方式：写 **litmus test**（十几行的并发片段 + 一个问题"这个结果可能吗"），用 **herd** 工具对模型穷举所有合法执行。2018-01 McKenney 的 commit `1c27b644c0fd`（"Automate memory-barriers.txt; provide Linux-kernel memory model"）把模型放进内核树 `tools/memory-model/`，合入 **v4.17**（compare API 核实）。从此"这两个 CPU 上这四行代码能不能看到 r1=0 && r2=0"有了机器答案——review 弱内存序 patch 时，litmus test 是比嘴仗强一个文明等级的论据。

## 12.11 方案对比与场景推演（选型的思考过程）

### 总对照表

| 原语 | 读者开销 | 写者开销 | 可睡眠 | 含指针数据 | 公平性 | 致命弱点 |
|---|---|---|---|---|---|---|
| qspinlock | —（互斥无读写之分） | 无争用 1 CAS | 否（RT 下例外见 12.9） | OK | FIFO+pending 微插队 | 临界区必须极短 |
| mutex | — | 无争用 1 CAS | 是 | OK | FIFO+窃取+handoff 兜底 | 中断上下文禁用 |
| rwsem | 1 原子加 | 等读者清空 | 是 | OK | handoff 缓解饿死 | 写者饥饿倾向 |
| seqlock | ~2 次普通读+屏障 | seq±2+屏障+互斥 | 读侧否 | **禁止** | 写者绝对优先 | 读侧不能有副作用；写频高则读饿 |
| RCU（13 章） | ≈0 | copy+update+grace period | 读侧否（经典） | **OK** | 无饿死 | 写延迟大；内存翻倍窗口 |
| percpu + 汇总 | 0（本地） | 0（本地） | — | — | — | 读全局值贵且非瞬时一致 |

表只能排除明显错误答案；真正的选型靠下面这样的推演。

### 场景 A：64 核上的全局统计计数器（每核每秒百万次自增，每分钟读一次总值）

- 候选 1：`spinlock + u64`。每次自增都要把锁的 cacheline 抢到本核。64 核 × 1M/s 次跨核转移——互连直接饱和，**错误答案**。
- 候选 2：`atomic64_t`。省了锁逻辑，但 cacheline 乒乓一点没少（LOCK XADD 同样要独占 line）。比候选 1 好一个常数，**仍是错误答案**。
- 候选 3：`DEFINE_PER_CPU(u64, cnt)`，自增走本地 line（0 跨核流量），读端 `for_each_possible_cpu` 累加 64 条 line。读端代价 O(NR_CPUS) 且结果是"扫描期间的模糊和"——对每分钟读一次的监控指标完全可接受。**正确答案**，内核 percpu_counter 即此模式的封装（外加 batch 阈值控制误差）。
- **边界反转**：如果需求变成"每次自增后立刻要精确总值"（比如配额扣减），percpu 模糊和不能用，正确答案反而退回 atomic + 分片（percpu_counter 的 `_sum` 慢路），或者重新设计需求本身。**读写两端的精度/频率需求共同决定数据布局**，没有万能答案。

### 场景 B：连接跟踪表（100K 查询/s，10 更新/s，value 是结构体指针）

1. 第一反应 rwsem？读者每次 `down_read` 都要原子加 count——100K/s 的原子写在多核上又是 cacheline 争用，读路径根本不是"无干扰"的。排除。
2. seqlock？读写比完美匹配（10000:1），但 value 是**指针**：读者拿到指针、还没解引用，写者已 free——12.7.1 的源码注释明令禁止。**这条边界与频率无关，是内存安全问题**。排除。
3. RCU：读侧 `rcu_read_lock`（≈关抢占，零原子操作）+ `rcu_dereference` 追指针；写侧 copy-update + grace period 后 free。指针安全恰好是 RCU 的本职（grace period 保证没有读者还看着旧指针）。**正确答案**——这正是内核路由 cache、conntrack 的真实选择。
4. 反向边界：如果 value 是 16 字节的纯值（无指针），seqlock 反而优于 RCU——不需要为每次更新分配新对象、没有 grace period 延迟、代码简单一个量级。**"含不含指针"是 seqlock vs RCU 的第一分界线，比读写比更优先**。

### 场景 C：跨 NUMA 高争用锁——要不要 NUMA-aware？

现状：qspinlock 的 FIFO 队列对 NUMA 盲目——队列里 socket0/socket1 的等待者交错，锁每次移交都可能跨 socket（cacheline 跨互连转移，量级上比同 socket 贵数倍）。

Oracle 的 Alex Kogan 与 Dave Dice 提出 **CNA（Compact NUMA-Aware lock）**（LWN *NUMA-aware qspinlocks*，Corbet，2021-04-12）：把等待队列劈成主/副两条，队头发现 next 在别的 node 就把它移去副队列——锁尽量在本 node 内传递。收益与代价都很直白：

> "As the number of contending threads grows, though, the speedup does as well, approaching a factor of two for ridiculously heavily contended loads."
> 风险："the primary queue may never empty out and the other nodes in the system will be starved for the lock."
> 缓解：主队列 10ms（默认）不空，整条副队列提升到主队列头。

**结局（考古的反面教材价值）**：系列从 2019-01 起改了 **14+ 版，始终未合入主线**。原因值得咀嚼：它动的是"内核最基础同步原语"的公平性语义，10ms 的饥饿窗口对某些负载就是延迟尖刺；而收益集中在"荒谬级争用"——那本身往往是该修数据结构而不是修锁的信号。**社区对基础原语的语义改动极端保守，哪怕 2x 收益在桌上**。这条线索同时告诉你：你的服务若真撞上跨 NUMA 锁争用，主线不会救你，出路是 per-NUMA-node 数据分片（应用层 CNA）。

## 12.12 落地：失败模式与生产排障

### 失败模式清单（每条都对应前文机制）

| 失败模式 | 机制根因 | 信号 |
|---|---|---|
| CPU 99% 但 QPS 低 | spinlock 争用空转 | `perf top` 里 `native_queued_spin_lock_slowpath` 居前（qspinlock slowpath 的导出符号，12.4 节那个函数） |
| 偶发 ms 级长尾 | 持锁路径里有 I/O / 直接回收 / 锁链 | `perf lock report` wait time 分布双峰 |
| RT 任务超时 | 优先级反转（无 PI 的锁） | trace 里高优先级任务 blocked on 低优先级 owner |
| 整机卡死无 oops | ABBA 死锁（lockdep 没覆盖到的路径） | NMI watchdog / `hung_task` 打印持锁栈 |
| 偶发数据错乱 | 裸 data race（没加锁）或 seqlock 读侧带副作用 | KCSAN 报告；或"重试协议里出现不可回滚操作"的代码审查 |
| unlock 崩溃 UAF | mutex_unlock 后对象被并发 free（12.6.5） | KASAN: use-after-free in `__mutex_unlock_slowpath` |

### 排障 playbook（自上而下）

```bash
# 1. 定性：是不是锁
perf top -a -g            # 看 native_queued_spin_lock_slowpath / __mutex_lock 慢路 / rwsem_down_* 占比
# 2. 定量：哪把锁、等多久
perf lock record -a -g -- sleep 10 && perf lock report
perf lock contention -a -b sleep 10   # 较新内核：基于 BPF，开销低，直接给调用栈聚合
# 3. 锁内幕：争用计数（需 CONFIG_LOCK_STAT，生产慎开）
cat /proc/lock_stat
# 4. 自定义测量：bpftrace 抓 mutex 等待分布（probe 名以 bpftrace -l 实测为准）
bpftrace -e 'kprobe:__mutex_lock_slowpath { @s[tid]=nsecs; }
             kretprobe:__mutex_lock_slowpath /@s[tid]/ { @w=hist(nsecs-@s[tid]); delete(@s[tid]); }'
```

判读要点：

- **量级心智模型**（依微架构浮动，记量级不记数字）：无争用 lock ≈ 一条 LOCK 前缀指令 ≈ 10–30ns（x86 上自带 full barrier，无须额外 mfence）；cacheline 跨核拉取 ≈ 几十 ns；跨 socket ≈ 100ns 上下；mutex 真睡眠 ≈ 数 μs（两次切换）。看到 **40ms** 这种数字，绝不是"锁本身慢"，是持锁路径里嵌了 I/O / 直接回收 / 另一把锁——去看持锁代码而不是换锁。
- 优化的优先级排序（屡试不爽）：**①持锁路径瘦身**（把分配、I/O、可延迟工作挪出临界区）→ **②减少共享**（percpu、分片、hash bucket 锁）→ **③改读写形态**（seqlock/RCU）→ **④最后才是换锁实现**。大部分"锁性能问题"在 ① ② 就终结了；直接跳 ④ 的人通常会带着同样的争用换一个符号名继续燃烧 CPU。
- `trace_contention_begin/end`（12.4/12.6 的 v6.12 源码里均已内建；首次引入约在 5.19 周期「待核」）：`perf lock contention` 的数据源，对 off-CPU 的 mutex/rwsem 争用同样有效——这是近年排障体验最大的升级，老的 `perf lock` 依赖 lockdep tracepoint 路径。

### 锁顺序的组织学

ABBA 死锁的工程级预防不是"小心点"，是**把锁层级写成文档并让 lockdep 兜底**：内核里 `mm/filemap.c`、`fs/inode.c` 等文件头都有大段 lock ordering 注释块（`->i_rwsem` before `->invalidate_lock` 之类）。你的子系统也该有这张表——评审新代码时先问"这次加锁在层级表的哪一行"。

## 12.13 未来演进

1. **PREEMPT_RT 后时代**（已发生，6.12）：RT 主线化不是终点而是起点——`spinlock_t` 的双语义将持续暴露历史代码里"其实需要 raw"的误用；`wait_type` 静态检查会越收越紧。新代码的默认姿势：除非有"物理不能睡"的证明，一律 `spinlock_t`/`mutex`。
2. **CNA/NUMA-aware 锁悬而未决**：14+ 版未合入（12.11 场景 C）。主线立场实质上是"高争用锁该被消灭而不是被优化"。预期：NUMA 感知继续以**数据结构分片**而非锁实现的形式落地。
3. **Rust for Linux 的锁封装**：`Mutex<T>` 把"数据被锁保护"从注释升级为类型——不持 guard 拿不到 `&mut T`，"忘了加锁"成为编译错误；guard 离开作用域自动 unlock。但 Rust 救不了锁顺序（ABBA 是运行时拓扑问题），lockdep 在 Rust 驱动时代依然是底线。
4. **形式化持续渗透**：LKMM 已覆盖 RCU 序语义，litmus test 正在成为弱序 patch 的 review 标配；KCSAN 在 CI 的普及让"裸读写共享变量"的旧习惯加速出清（连 seqlock 都被迫自我声明，12.7.2）。

## 章末五件套

### 高频面试题（Staff 级，每题给推理链不给背诵点）

**Q1：spinlock 和 mutex 怎么选？"mutex 慢"对不对？**

推理链：第一刀切上下文——中断/softirq/持 raw 锁内只能 spinlock。第二刀切持锁时长与争用形态。"mutex 慢"是双重过时：无争用两者都是一条 CAS（12.6.2 源码）；有争用时 mutex 先乐观自旋（owner on_cpu 时行为≈spinlock），赌输才睡。反而高争用下 spinlock 空转烧的 CPU 比 mutex 睡眠贵。加分：说出 mutex 三层路径名 + handoff 防饿死位；终结句：RT 内核下 spinlock_t 本身就是 rt_mutex，这道题的前提在 6.12 后需要先问"哪种内核"。

**Q2：qspinlock 为什么需要 pending bit？去掉它会怎样？**

推理链：MCS 队列解决 O(N) 广播，但入队要 touch per-CPU 节点的冷 cacheline + xchg tail——对"只有两个 CPU 短暂争用"这一统治性场景是纯开销。pending bit 让第二名原地等 locked 字节，完全不碰队列机器。去掉后：每次轻微争用都付 MCS 全价，低争用延迟显著回退（这正是 commit message 承认"轻中度争用可能略慢"后用 pending 挽回的部分）。加分：pending 只许一人（第三名见 pending 即入队），否则重演广播风暴；undo 路径处理置位 race（12.4.4）。

**Q3：为什么 qspinlock 必须塞进 4 字节？挤进去靠哪三招？**

推理链：`spinlock_t` 嵌在 `struct page` 等海量结构体里，变大即全内核内存+cache 足迹膨胀，ABI 级硬约束。三招：①tail 指针→"CPU 号+1"+2bit 嵌套 idx（per-CPU 静态节点数组可由编号重建指针）；②嵌套层数封顶 4（task/softirq/hardirq/NMI——执行模型不变量直接进位布局）；③队头不在自己节点上等而是盯锁字本身，于是 unlock 无需知道队列（一条 release store，12.4.2）。

**Q4：seqlock 读者协议里两个 barrier 各防什么？为什么禁指针？**

推理链：按 12.7.2 四屏障推演讲两个读侧的——load_acquire 防"数据读漂到 seq1 之前"（旧数据配新序号）；retry 的 smp_rmb 防"seq2 读漂到数据读之前"（撕裂逃过检测）。禁指针：retry 只能告诉你"读到的值作废"，但解引用一个已 free 的指针在 retry 之前就崩了——是内存安全问题不是一致性问题，所以 v6.12 类型定义注释直接写死禁令；要指针就上 RCU（grace period 保证指针在读者眼里活着）。

**Q5：讲讲 BKL 的死亡过程，它留下的最大遗产是什么？**

推理链：语义（递归+睡眠自动放锁）→ 为什么难拆（语义掩盖真实依赖）→ 2008 semaphore 风波（40% AIM 回归，Linus 改回 spinlock 并故意不优化："maybe we _need_ to have a bad BKL"）→ 2.6.37 CONFIG_BKL → 2.6.39 commit "BKL: That's all, folks"（Arnd Bergmann）。遗产：①细粒度锁与 per-subsystem 并发重构被倒逼完成；②lockdep（2006）恰逢锁关系复杂化而成为刚需；③"严格语义优于宽松语义"成为后续 mutex 设计的指导思想。

**Q6：mutex 的 handoff 解决什么问题？为什么不直接全程 FIFO？**

推理链：乐观自旋引入锁窃取——spinner 物理上快于被唤醒的睡眠者，队首可能被无限插队（饿死，v4.10 前真实存在）。全程 FIFO 等于禁止窃取，把"owner 刚释放、spinner 正热"的最佳交接场景也强制走唤醒慢路，吞吐大跌（2008 年 BKL semaphore 的 40% 回归就是公平优先的代价实证）。handoff 是分层触发：默认允许窃取，队首"醒来发现又被偷"才置 HANDOFF 位，下次 unlock 指名移交（PICKUP 位拒绝一切 spinner）。吞吐为常态，公平为兜底。

**Q7：lockdep、KCSAN、DEBUG_ATOMIC_SLEEP 三者职责怎么划？lockdep 误报怎么处理？**

推理链：lockdep=锁**顺序**类死锁（依赖图找环，含 irq-safety）；KCSAN=没加锁的裸 data race；DEBUG_ATOMIC_SLEEP=原子上下文里睡眠（might_sleep+preempt_count）。lockdep 按 lock class 归类（同一初始化点=同类），父子 inode 这类同类多实例的合法定序会误报自环 → `mutex_lock_nested`/`lockdep_set_subclass`/`mutex_lock_nest_lock`，且注解前必须对照锁层级文档（注错=静音真死锁）。强调：lockdep 证明的是"已观测图无环"，覆盖率是天花板。

**Q8：什么是优先级反转？Linux 与 RT 的解法路径？**

推理链：H 等 L 持的锁，M 抢占 L → H 被 M 无限期间接阻塞。案例 Mars Pathfinder（1997，VxWorks，PI 开关被关，地面复现后远程开启修复）。Linux：普通 mutex/spinlock 不做 PI；`rt_mutex` 做优先级继承（L 临时继承 H 的优先级）；PREEMPT_RT 把 spinlock_t/mutex 整体换芯 rt_mutex（12.9 源码），且 seqlock 读者在 RT 下靠"lock/unlock 写者的锁"触发 PI 助推（12.7.4——能讲出这个的基本是真读过源码）。

**Q9：unlock 一把 qspinlock 为什么只写一个字节就够？**

推理链：分工设计——等待者各自盯自己该盯的位置（pending 者盯 locked 字节、队头盯整字、队列其余盯自己节点），release store 清 locked 后，"接力"由等待侧的 acquire 等待自行完成；owner 无需知道队列存在。字节可独立写依赖 union 布局（locked 独占 bit0-7）+ 架构支持子字原子写。对比：ticket 的 owner++ 也便宜但唤醒 O(N)；原版 MCS unlock 要传 node 还要找 next。

**Q10：线上 CPU 高 QPS 低，给出完整定位路径。**

推理链：`perf top -a -g` 看 `native_queued_spin_lock_slowpath`/`__mutex_lock` 慢路符号占比 → `perf lock contention -a -b`（基于 trace_contention_* tracepoint，5.19+）拿到争用锁的调用栈聚合 → 区分两类：自旋热（spinlock，烧 CPU）vs 睡眠热（mutex/rwsem，QPS 低但 CPU 不一定高）→ 看持锁路径里有没有 I/O/分配/嵌套锁 → 优化顺序：临界区瘦身 > 减共享（percpu/分片）> 改读写形态（RCU/seqlock）> 换锁。给量级锚点：无争用几十 ns / 跨 socket 百 ns / 睡眠 μs 级 / ms 级必有别的东西。

### 实战项目（动手验证三件工具的职责边界）

写一个内核模块 `dual_lock_module.c`：两个 kthread 各自增共享计数器 10000 次。

1. 基线：spinlock 保护，验证结果 20000。
2. 故意去锁 → 开 `CONFIG_KCSAN` 跑，收 data race 报告（lockdep 对此沉默——亲手验证 Q7 的职责划分）。
3. spinlock 临界区里加 `msleep(1)` → 开 `CONFIG_DEBUG_ATOMIC_SLEEP`，收 "BUG: sleeping function called from invalid context"（机制：msleep 入口的 `might_sleep()` 检查 `preempt_count`，持 spinlock 时非零）。
4. 构造 ABBA：两线程以相反顺序拿两把 mutex → 开 `CONFIG_PROVE_LOCKING`，注意 lockdep **在死锁发生前**（仅凭两条依赖边）就报 "possible circular locking dependency"。
5. 进阶：用 `mutex_lock_nested` 错误地注解掉第 4 步的报告，体会"注解静音真死锁"的危险。

暗坑：`kthread_run` 起线程；`kthread_stop` 需要线程轮询 `kthread_should_stop()` 否则 rmmod 卡死；KCSAN/lockdep 内核要自己编，发行版内核默认全关。

### 设计题

**设计一个 per-namespace 的内核连接表**：1M 条目，查询 500K/s（8 成命中热点 1% 条目），插入/删除 1K/s，条目含指向 `struct conn_info` 的指针，要求 p99 查询 < 2μs，运行在 4-socket NUMA 机器。

要点（按 12.11 推演风格答）：含指针 + 读写比 500:1 → 读路径 RCU（seqlock 因指针出局，rwsem 因读侧原子加在 500K/s 下乒乓出局）；写路径 1K/s 用 per-bucket spinlock（mutex 也可，但删除要和 RCU 回收协作，spinlock 临界区足够短）；hash bucket 数 ≥ 2^17 摊薄写争用；NUMA：热点 1% 条目的 cacheline 会跨 socket 共享——条目内把读热字段（key、指针）与写热字段（统计计数）分 cacheline，统计用 percpu；resize 用 RCU + 双表渐进迁移；给出 grace period 的内存翻倍窗口估算（1K/s 删除 × grace period 时长 × 条目大小）。

### 系统题

**复盘 2008 BKL-semaphore 事故**：给你 2.6.26-rc1 的现场——AIM7 掉 40%，怀疑新 generic semaphore。要求：①设计实验区分"semaphore 实现变慢"vs"公平性策略改变行为"（提示：对比直接移交 vs 竞争获取的唤醒-运行间隔，用 sched trace 测"锁已移交但接收者未上 CPU"的死区时长）；②解释为什么"公平的锁"会比"不公平的锁"吞吐差（cache 热度、唤醒延迟窗口期锁不可用）；③给出两个修复方向并预测 Linus 会选哪个、为什么（修 semaphore vs 改回 spinlock 并冻结 BKL 优化——答案见 12.2，机制设计压倒局部最优）。

### 代码题

用 bpftrace 写"rwsem 写者等待时间分布 + 超 10ms 打印持锁栈"，并解释为什么 kretprobe 方案对 rwsem 读者路径会系统性低估（提示：reader fastpath 是内联的原子加，没有可挂的函数边界；正确做法是用 `tracepoint:lock:contention_begin/end` 过滤 `flags & LCB_F_WRITE`）。验证负载：多进程对同一文件 mmap 写（争 `mm->mmap_lock`——内核里最著名的 rwsem，没有之一）。

## 本章心法

> **锁的三十年演进只在反复回答一个问题：让谁等、在哪等、等多久换多少吞吐。**
> TAS 让所有人挤在门口（谁壮谁进）；ticket 发号但全场盯同一块叫号屏；qspinlock 让每人在自己椅子上等、只有下一位站到门口；mutex 先赌"里面的人马上出来"再去睡，睡过头有 handoff 兜底；seqlock 干脆不拦读者、只在出口验票；RCU 连票都不验，让旧世界多活一个 grace period。
> ——选锁时别问"哪个快"，问"我的等待者是谁、它能不能睡、它等不到会发生什么"。三问答完，锁自己会浮出来。
