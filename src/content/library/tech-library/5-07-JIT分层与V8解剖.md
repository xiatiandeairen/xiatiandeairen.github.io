## TL;DR(本章最硬结论)

1. **JIT 的本质是"用编译时间换运行时间",而分层 JIT 是"按热度分摊这笔交易"**:冷代码用解释器(零编译成本)、温代码用快速基线编译器、烫代码才用昂贵的优化编译器。V8 今天有 4 档:**Ignition(字节码解释器)→ Sparkplug(基线)→ Maglev(中层优化)→ TurboFan(顶层优化)**。
2. **动态语言 JIT 的核心武器是"推测优化 + 去优化(deopt)"**:JS 没有静态类型,优化编译器靠**运行时反馈**(feedback vector)猜类型,猜对了生成飞快的特化代码,猜错了**退回解释器(deopt)**重来。整个性能模型建立在"假设成立"上。
3. **内联缓存(Inline Cache, IC)是这套反馈的载体**:每个属性访问点缓存它见过的对象**形状(hidden class / Map)**。单态(monomorphic,只见过 1 种形状)最快;多态(polymorphic,几种)还行;**巨态(megamorphic,太多种)退化成字典查找**,优化器基本放弃。
4. **写出快 JS = 让对象形状稳定**:同一构造顺序、不中途加删属性、数组别变成"洞数组(holey)"——本质是配合 IC 维持单态。这是"JIT 物理"决定的,不是玄学。
5. **deopt 风暴是头号生产性能坑**:某段热代码被优化后反复因假设失效被 deopt、再优化、再 deopt,在解释器和编译器之间打摆子,比一直解释还慢。

**前置依赖**:第 6 章(解释器与字节码 VM——Ignition 就是个字节码解释器);与 Chromium 域第 11 章(V8↔Blink 绑定)呼应(那章讲 JS 怎么碰 DOM,本章讲 JS 本身怎么被执行)。

## 1. 设计考古:V8 为什么从"两档"走到"四档"

JIT 不是新东西(Smalltalk、Self、HotSpot JVM 早有分层),但 V8 的演进史是理解"为什么需要这么多档"的最佳教材。

| 阶段 | 大致时间 | 架构 | 解决/暴露的问题 |
|---|---|---|---|
| Full-codegen + Crankshaft | ~2010–2017 | 一个"快但笨"的基线编译器(Full-codegen 直接由 AST 生成机器码)+ 一个优化编译器(Crankshaft) | 没有解释器→**冷代码也要编译成机器码**,内存大、启动慢;Crankshaft 直接吃 AST,**难维护、难支持新语言特性** |
| **Ignition + TurboFan** | 2017 起 | 字节码解释器 Ignition + 优化编译器 TurboFan(sea-of-nodes IR) | 用字节码替代基线机器码→**内存大降**(字节码比机器码紧凑);TurboFan 吃字节码+反馈而非 AST,**更通用更可优化**。代价:**纯解释比旧基线机器码慢** |
| **+ Sparkplug** | 2021 | 在 Ignition 和 TurboFan 之间插一个**基线编译器** | 解决"Ignition 太慢、TurboFan 太贵"的中间真空:Sparkplug **不建 IR、不做优化**,几乎线性地把字节码翻成机器码,**编译极快**,填补启动后峰值前的空窗 |
| **+ Maglev** | 2023 | 再插一个**中层优化编译器**(SSA-based,比 TurboFan 轻) | Sparkplug 不优化、TurboFan 太慢启动;Maglev 做**轻量优化**(用 IC 反馈但不做 TurboFan 那么激进的分析),**编译速度和代码质量都居中** |

> 演进的主线:**档位越来越多,本质是"编译成本"和"代码质量"的曲线被切得越来越细**——让每段代码都落在性价比最优的那一档,并能随热度平滑升档。这条"分层 + 按热度升级"的思路,JVM(解释器→C1→C2)、JSC(LLInt→Baseline→DFG→FTL,四档)都是同构的。

各档的具体类名/文件路径(如 Ignition 的 `BytecodeGenerator`、TurboFan 的 sea-of-nodes 节点定义、Maglev 的目录)以 `github.com/v8/v8` 为准——「待核」。

## 2. 机制精读

### 2.1 Ignition:字节码解释器(承接第 6 章)

V8 先把 JS 源码经 parser → AST → **字节码**(由 Ignition 的字节码生成器产出),然后用一个**寄存器式字节码解释器**执行。要点:

- **寄存器式而非纯栈式**:Ignition 用一组虚拟寄存器(accumulator 累加器 + 局部寄存器),比纯栈式少很多 push/pop。(对照第 6 章 clox 的栈式 VM。)
- **dispatch**:解释器主循环按字节码 opcode 跳到对应 handler。V8 的 handler 是用 TurboFan 的底层子集(CodeStubAssembler / Torque)预先编译好的机器码片段——**解释器本身也是"编译产物"**,这是 V8 的精巧处。(Torque/CSA 的具体写法「待核」。)
- **字节码紧凑**:相比直接生成机器码,字节码内存占用小一个量级——这是 2017 年换 Ignition 的主要动机(尤其手机上)。

### 2.2 反馈与内联缓存(IC):动态优化的燃料

JS 是动态类型:`a.x` 里 `a` 可能是任何形状的对象,`a + b` 可能是数加、字符串拼接……要优化,必须知道**运行时实际遇到了什么**。V8 用两样东西记录:

**(1) Hidden Class / Map(对象形状)**
V8 不为每个对象存"属性名→值的字典"(太慢),而是给**形状相同的对象共享一个 Map**(隐藏类),Map 记录"属性 x 在偏移 0、y 在偏移 8……"。这样属性访问变成**定偏移读取**,和静态语言的 struct 一样快。

关键:**加属性会"换 Map"**。`o={}; o.x=1`(Map A)`; o.y=2`(Map B)。两个对象只要构造路径一致,就走到同一个 Map,从而共享。路径不一致(一个先加 x 一个先加 y)→ 不同 Map → 形状分裂。

**(2) Feedback Vector + Inline Cache**
每个"会变化"的操作点(属性读写、函数调用、算术)在 feedback vector 里有一个槽,记录它**见过的 Map**。这就是内联缓存(IC),它有三个状态:

| IC 状态 | 含义 | 性能 |
|---|---|---|
| **monomorphic** 单态 | 该访问点只见过 1 种 Map | 最快:校验 Map == 缓存值,命中即按缓存偏移直接读 |
| **polymorphic** 多态 | 见过 2–N 种 Map(N 通常 4 左右) | 较快:在小列表里线性找匹配的 Map |
| **megamorphic** 巨态 | 见过太多 Map | 退化:放弃缓存,走通用的字典查找(慢),优化器也基本不再特化 |

(单态/多态的精确阈值、megamorphic 的处理细节「待核」。)

> **这是全章最该刻进脑子的因果链**:对象形状稳定 → IC 保持单态 → 优化编译器敢于特化 → 快。形状混乱 → IC 巨态 → 优化器放弃 → 慢。**JS 性能优化的绝大多数"玄学规则",根都在这。**

### 2.3 分层升级:热度计数与 OSR

- 函数被调用/循环被执行时,V8 累加**热度计数**;过阈值就把这个函数**升档**(Ignition→Sparkplug→Maglev→TurboFan)。
- **OSR(On-Stack Replacement,栈上替换)**:一个**正在执行的长循环**没法等它返回再升级,V8 会在循环中途把当前栈帧"换"成优化后的代码继续跑。(阈值与触发细节「待核」。)
- 升档不是单向的:优化代码 deopt 后退回低档,攒够热度可再升。

### 2.4 推测优化与去优化(deopt):整套体系的命脉

TurboFan/Maglev 拿到字节码 + feedback,做**推测优化**:既然这个加法点一直是"数 + 数",就**假设**它永远是数,生成纯整数/浮点加法的机器码,省掉所有类型检查。

但假设可能破。所以优化代码里埋了**轻量校验**(guard):一旦发现实际类型不符(突然来了个字符串),触发 **deopt**——丢弃这段优化机器码,**把执行状态精确还原**,退回 Ignition 从对应字节码继续。

- deopt 要求优化器维护"机器码状态 ↔ 字节码状态"的映射(deopt 元数据),这是优化编译器最复杂的部分之一。
- **eager deopt**(执行到 guard 失败立刻退)vs **lazy deopt**(因别处假设失效,标记此代码下次进入时退)。(精确分类「待核」。)

### 2.5 TurboFan 的 sea-of-nodes(顶层优化)

TurboFan 的 IR 是 **sea-of-nodes**:控制流和数据流统一成一张图,节点只受真实依赖约束、不被无谓的语句顺序绑死,给调度器更大自由做优化(类似但比第 3/4 章的线性 SSA 更激进)。它做内联、类型特化、范围分析、逃逸分析等重型优化——**质量最高、编译最贵**,所以只给最烫的代码用。Maglev 则是更轻的 SSA 优化器,介于 Sparkplug 和 TurboFan 之间。(节点种类、调度算法「待核」。)

## 3. ⭐ 配套可运行 demo:用 Python 模拟内联缓存

下面这个 demo **不依赖任何库,`python3 ic_demo.py` 直接跑**,印证 §2.2 的核心因果:形状稳定→单态→快;形状发散→巨态→退化。

```python
# ic_demo.py —— 模拟 hidden class(Map)+ inline cache 的单态/多态/巨态
class Map:
    """隐藏类:记录属性名->偏移,以及"加某属性后转移到哪个 Map"(转移树)"""
    _next_id = 0
    def __init__(self):
        self.offsets = {}        # name -> slot
        self.transitions = {}    # name -> Map(加该属性后的形状)
        self.id = Map._next_id
        Map._next_id += 1

ROOT = Map()

class Obj:
    """对象 = 一个 Map(形状)+ 一个紧凑的 slots 数组(定偏移存值)"""
    def __init__(self):
        self.map = ROOT
        self.slots = []
    def set(self, name, value):
        m = self.map
        if name in m.offsets:                 # 已有该属性:原地写
            self.slots[m.offsets[name]] = value; return
        if name not in m.transitions:         # 形状转移:新建/复用下一个 Map
            nm = Map(); nm.offsets = dict(m.offsets); nm.offsets[name] = len(m.offsets)
            m.transitions[name] = nm
        self.map = m.transitions[name]
        self.slots.append(value)

class InlineCache:
    """一个属性读取点的 IC:缓存见过的 (Map.id -> 偏移)"""
    LIMIT = 4   # 超过即 megamorphic
    def __init__(self): self.cache = {}; self.state = "uninitialized"; self.slow = 0; self.fast = 0
    def load(self, obj, name):
        mid = obj.map.id
        if mid in self.cache:                 # IC 命中:直接按缓存偏移读(快路径)
            self.fast += 1; return obj.slots[self.cache[mid]]
        # IC 未命中:慢路径查形状,并记录
        off = obj.map.offsets[name]; self.slow += 1
        if self.state != "megamorphic":
            self.cache[mid] = off
            n = len(self.cache)
            self.state = "monomorphic" if n == 1 else ("polymorphic" if n <= self.LIMIT else "megamorphic")
            if self.state == "megamorphic": self.cache.clear()   # 巨态:放弃缓存
        return obj.slots[off]

def make(order):
    o = Obj()
    for k, v in order: o.set(k, v)
    return o

# 场景 A:所有对象同样的构造顺序 -> 同一个 Map -> 单态
print("== 场景 A:形状稳定(构造顺序一致)==")
icA = InlineCache()
for _ in range(1000):
    o = make([("x",1),("y",2),("z",3)])
    icA.load(o, "x")
print(f"IC 状态={icA.state}  快路径={icA.fast}  慢路径={icA.slow}")

# 场景 B:每个对象构造顺序都不同 -> 形状发散 -> 巨态
print("== 场景 B:形状发散(顺序乱)==")
icB = InlineCache()
import itertools
perms = list(itertools.permutations([("x",1),("y",2),("z",3),("w",4)]))
for i in range(1000):
    o = make(list(perms[i % len(perms)]))
    icB.load(o, "x")
print(f"IC 状态={icB.state}  快路径={icB.fast}  慢路径={icB.slow}")
```

**预期输出**(逻辑确定,无随机):

```
== 场景 A:形状稳定(构造顺序一致)==
IC 状态=monomorphic  快路径=999  慢路径=1
== 场景 B:形状发散(顺序乱)==
IC 状态=megamorphic  快路径=0  慢路径=1000
```

**读这个 demo 你就摸到了 V8 的脉**:场景 A 第一次未命中后全走快路径(单态),场景 B 因为形状一直在变,IC 崩成巨态、永远走慢路径。真实 V8 里,巨态还会让 TurboFan 放弃特化这个点——慢上加慢。

## 4. 方案对比:四档各自的位置

同一个函数,随热度可能经历全部四档。它们的取舍:

| 档 | 编译成本 | 代码质量 | 用在 | 类比(JVM/JSC) |
|---|---|---|---|---|
| Ignition(解释) | 零(无编译) | 最低 | 只跑几次的冷代码 | JVM 解释器 / JSC LLInt |
| Sparkplug(基线) | 极低(线性翻译,无 IR) | 低(无优化,但省了解释开销) | 温代码、启动期 | JSC Baseline |
| Maglev(中层) | 中(轻量 SSA) | 中 | 较热但不值得 TurboFan 的代码 | JSC DFG / JVM C1 |
| TurboFan(顶层) | 高(sea-of-nodes 重优化) | 最高 | 最烫的核心热点 | JSC FTL / JVM C2 |

**用一个场景串起来**:一个网页加载——

1. 大量只执行一两次的初始化代码:留在 **Ignition**,编译它们纯属浪费。
2. 事件回调、被调用几十次的函数:升 **Sparkplug**,去掉解释开销但不值得深优化。
3. 一个被频繁调用的工具函数:升 **Maglev**,轻量优化即可。
4. 渲染/计算热循环(每帧跑千万次):升 **TurboFan**,值得花大代价榨干。

**为什么不直接全上 TurboFan?** 因为编译本身要时间和内存。对只跑两次的代码,TurboFan 的编译开销远超它省下的执行时间——**净亏**。分层就是为了不做这种亏本买卖。

## 5. 扎根:失败模式与生产真坑

1. **deopt 风暴**:热函数被优化后反复因假设失效被 deopt→再优化→再 deopt,在档位间打摆子,比一直解释还慢。常见诱因:一个本来是 int 的变量偶尔被赋成 `undefined`/字符串;隐藏类不稳。诊断:`--trace-deopt`(Node 加 `--trace-deopt`)看哪个函数反复 deopt。
2. **megamorphic 访问点**:一个泛型函数处理几十种不同形状的对象(如通用序列化器),其属性访问点变巨态,永远慢路径。缓解:对热路径按形状分流,或减少形状种类。
3. **形状不稳定**:构造函数里属性赋值顺序不一致、对象创建后又 `delete`/动态加属性、把对象当字典用(`obj[dynamicKey]=...`)——都会制造大量 Map。建议:构造时一次性定好所有属性(哪怕先赋 `null`),别 `delete`(用 `obj.x = undefined`),真当字典就用 `Map`。
4. **洞数组(holey array)**:`a=[1,2,3]` 是 packed,`a[100]=x` 制造空洞→V8 降级数组的元素种类(elements kind),访问变慢。别在数组里挖洞。
5. **过早微优化的陷阱**:绝大多数代码根本不会被升到 TurboFan,为它们手抠 IC 是浪费。**先用 profiler 找到真正进了优化档的热点**(`--prof` / DevTools Performance),再针对性优化。

**心法**:在 V8 上写快 JS,不是写"更聪明的算法技巧",而是**给优化器喂稳定、可预测的形状与类型**——让它的推测一直猜对、永不 deopt。你是在和一个赌徒合作,你的工作是让它的赌注总能赢。

## 6. 未来演进

- **Maglev 的成熟与普及**:中层优化把"启动后到峰值前"这段体验持续拉平,移动端尤其受益。
- **WebAssembly 的协同**:Wasm 是静态类型、无需推测,V8 的 Wasm 引擎(Liftoff 基线 + TurboFan 优化)是另一条分层线;JS↔Wasm 边界优化是热点。
- **指针压缩 / 内存优化**:V8 持续在 Map、feedback vector 的内存上做文章(海量对象时这些元数据不小)。
- **对工程师的含义**:档位会更多更平滑,但"形状/类型稳定 = 快"这条底层物理不会变——它源于动态语言 JIT 的本质,值得吃透到肌肉记忆。

## 章末五件套

**① 高频面试题(带要点)**
1. *V8 为什么要分层(4 档)?* → 编译成本 vs 代码质量按热度分摊;冷代码编译纯亏;给每段代码落到性价比最优档 + 平滑升级。
2. *什么是内联缓存?单态/多态/巨态?* → 记录访问点见过的对象形状(Map);单态最快(1 种)、多态可接受(几种)、巨态退化成字典查找且优化器放弃。
3. *deopt 是什么?为什么需要?* → JS 动态类型,优化基于运行时反馈做推测;假设破时退回解释器(精确还原状态)。是推测优化能成立的安全网。
4. *为什么"对象形状要稳定"能让 JS 变快?* → 稳定形状→IC 单态→优化器敢特化成定偏移访问;形状乱→巨态→慢路径。把这条因果链讲全是加分。
5. *Ignition 为什么是寄存器式字节码而不是直接机器码?* → 字节码紧凑省内存(2017 换它的主因);解释器 handler 本身预编译成机器码。
6. *(进阶)hidden class 是怎么共享和转移的?* → 同构造路径的对象共享 Map;加属性沿转移树到新 Map;路径不一致→形状分裂。
