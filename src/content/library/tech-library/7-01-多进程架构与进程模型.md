> **前置依赖**：熟悉操作系统进程/线程基础、IPC 概念、浏览器渲染基本流程（HTML → DOM → Layout → Paint）。
>
> **TL;DR**：Chromium 用「多进程 + 沙箱」替代单进程模型，核心驱动力是稳定性（renderer crash 不崩 browser）、安全性（沙箱限权）、Spectre 后的内存隔离。进程类型包括 Browser / Renderer / GPU / Network / Utility。进程分配以 SiteInstance → SiteInfo → ProcessLock 为决策链，现代 Desktop 默认走 site-per-process 强隔离；Android 因内存压力只隔离高风险站点。理解这一层是读懂 OOPIF、RenderFrameProxy、Mojo IPC 以及 Electron 进程模型的基础。

## 1. 设计考古：为什么必须多进程

### 1.1 单进程浏览器的根本缺陷

2006 年前，所有主流浏览器都是单进程。这带来三个不可回避的问题：

**稳定性**：任意 tab 的 renderer 崩溃 → 整个浏览器崩溃。网页内容（HTML/CSS/JS）高度不可信，parsing 漏洞、JS 无限循环、内存越界写都能把整个应用拖垮，用户体验等同于早期 Windows 的「协作式多任务」——任意程序挂起 = 系统卡死。

**安全性**：渲染引擎（Blink/WebKit）需要解析来自任意来源的内容，攻击面极大。单进程意味着一旦 renderer 被 exploit，攻击者获得的是完整的用户权限——读取文件、执行程序、访问 OS API，无任何沙箱隔壁。

**内存管理**：长时间运行的 tab 积累内存碎片，GC 压力无法通过操作系统正常回收。只有进程退出才能彻底归还内存。

Chromium 设计文档的开篇直接点明这一逻辑（来源：chromium.org/developers/design-documents/multi-process-architecture，本章已实际 Fetch 核实）：

> "It's nearly impossible to build a rendering engine that never crashes or hangs. It's also nearly impossible to build a rendering engine that is perfectly secure."

类比操作系统进化：MS-DOS（协作式多任务）→ Windows NT（抢占式多进程）；单进程浏览器 → 多进程浏览器，是同一逻辑的再演。

### 1.2 Chromium 多进程架构的诞生（2008）

2008 年 Chrome 1.0 发布时，多进程模型就是核心设计之一，相关 design doc 早于代码提交：

- **进程崩溃隔离**：renderer 进程崩溃显示 sad tab，不影响 browser UI 和其他 tab。
- **沙箱（sandbox）**：renderer 进程运行在 OS 级受限沙箱中，不能直接访问文件系统、网络、显示，必须通过 browser process 代理。
- **内存回收**：每个 tab 是独立进程，OS 可在内存压力时优先 swap 不可见 tab，精确控制。

### 1.3 Spectre/Meltdown 后的强化（2018）

2018 年 Spectre/Meltdown 漏洞公开，使多进程架构的安全意义从「沙箱围栏」升级到「内存物理隔离的唯一可靠手段」。Chromium 安全文档（经 Fetch 核实，URL：chromium.googlesource.com/chromium/src/+/main/docs/security/side-channel-threat-model.md）明确：

> "We have to assume that user code can reliably gain access to all data within a renderer process through speculation."

即：**同进程内的数据机密性已不可依赖**。Spectre 使 JS/Wasm 代码可以通过 timing side-channel 读取同进程任意内存。唯一的防御是把不同站点的数据放到不同进程——这直接驱动了 **Site Isolation** 在 Chrome 67（2018）的全面开启。

内存开销（经 chromium.org/Home/chromium-security/site-isolation Fetch 核实）：
- Desktop：全站点隔离约增加 10–13% 内存（多 tab 场景）
- Android：只隔离登录站点，约增加 3–5%

### 1.4 架构演进时间线

| 年份 | 里程碑 |
|------|--------|
| 2008 | Chrome 1.0：Browser + Renderer + GPU 三进程基础架构 |
| 2011 | GPU 进程独立（GPU Acceleration），Renderer 与 GPU 分离 |
| 2013 | Plugin 进程（PPAPI）；老式 NPAPI plugin 逐步废弃 |
| 2017 | Network Service 提案，网络栈沙箱化启动 |
| 2018 | Chrome 67：Site Isolation 默认开启（Desktop），Spectre 防御 |
| 2019 | Network Service 在 Desktop 移出 browser process |
| 2020 | Origin-Keyed Process Isolation（OOPIF + OAC header） |
| 2021 | Android 高风险站点隔离（2GB+ RAM 设备） |
| 2023+ | Fenced Frames 进程隔离；Storage Partition 隔离 |

## 2. 进程类型全景

### 2.1 官方进程类型枚举

【真实源码 content/public/common/process_type.h】

```cpp
// Copyright 2012 The Chromium Authors
// Do not remove or reorder the elements in this enum, and only add new items
// at the end, right before PROCESS_TYPE_MAX. We depend on these specific
// values in histograms.
enum ProcessType {
  PROCESS_TYPE_UNKNOWN = 1,
  PROCESS_TYPE_BROWSER,       // 2：主进程
  PROCESS_TYPE_RENDERER,      // 3：渲染进程
  PROCESS_TYPE_PLUGIN_DEPRECATED,   // 4：已废弃
  PROCESS_TYPE_WORKER_DEPRECATED,   // 5：已废弃
  PROCESS_TYPE_UTILITY,       // 6：工具进程
  PROCESS_TYPE_ZYGOTE,        // 7：Linux/ChromeOS 孵化器
  PROCESS_TYPE_SANDBOX_HELPER,// 8：沙箱辅助
  PROCESS_TYPE_GPU,           // 9：GPU 进程
  PROCESS_TYPE_PPAPI_PLUGIN_DEPRECATED,  // 10
  PROCESS_TYPE_PPAPI_BROKER_DEPRECATED,  // 11

```

注释强调"不能重排、只能末尾追加"，因为这些值直接写入 UMA histogram。

### 2.2 各进程类型职责对照

| 进程类型 | 关键职责 | 沙箱级别 | 每个 Chrome 实例数量 |
|---------|---------|---------|---------------------|
| **Browser** | UI、导航、进程管理、权限决策 | 无沙箱（完整权限） | 1 |
| **Renderer** | HTML/CSS/JS 执行（Blink + V8） | 严格沙箱 | 1 per site（Desktop）；多 tab 可能复用 |
| **GPU** | 硬件加速绘制、WebGL、视频解码 | 有限沙箱 | 1（崩溃恢复后可重启） |
| **Network** | HTTP/S、WebSocket、DNS、Cache | 严格沙箱 | 1（自 Chrome 80+ 独立进程） |
| **Utility** | PDF 解析、音视频解码、拼写检查等 | 依任务而定 | 按需多个 |
| **Zygote** | (Linux/ChromeOS) fork 模板进程，加速子进程启动 | 预沙箱 | 1 |

### 2.3 Browser Process（主进程）核心角色

Browser process 是唯一拥有完整系统权限的进程。其核心职责：

1. **UI 线程（BrowserThread::UI）**：窗口管理、导航决策、tab lifecycle。
2. **IO 线程（BrowserThread::IO）**：IPC 消息路由，所有子进程的消息均从 IO 线程接收再路由到目标线程。
3. **RenderProcessHost 管理**：每个 renderer 进程在 browser side 有一个对应的 `RenderProcessHost` 对象。
4. **安全策略执行**：`ChildProcessSecurityPolicy` 检查 renderer 是否有权 commit 某 URL。

### 2.4 Renderer Process 内部结构

每个 renderer 进程内部结构（【真实源码 chromium.org/developers/design-documents 核实】）：

```
Renderer Process
├── RenderProcess（全局单例）
│   ├── Mojo Channel → Browser Process
│   └── IPC Channel 建立
├── RenderThread（主渲染线程）
│   ├── Blink Engine（layout / paint / style）
│   └── V8 Isolate（JS 执行）
└── 每个文档
    └── RenderFrame（↔ RenderFrameHost in Browser）
```

来自官方 design doc（已 Fetch 核实）：
> "There will be exactly one `RenderProcess` object per renderer process."
> "Each renderer contains one or more `RenderFrame` objects."

### 2.5 GPU Process 特殊性

GPU process 是全局单例（【真实源码 content/browser/gpu/gpu_process_host.cc】）：

```cpp
// 强制单例：static 数组只允许一个 host per kind
g_gpu_process_hosts[kind] = this;
// DCHECK 防止重复创建
```

- 每隔 `forgive_minutes` 减一次崩溃计数（时间窗口宽恕）
- 超过 fallback 阈值（约 3–6 次，平台相关）→ 降级到软件渲染

## 3. 进程分配决策链：SiteInstance → SiteInfo → ProcessLock

这是整个进程模型最复杂也最核心的部分。

### 3.1 概念层次

```
BrowsingInstance          ← HTML 5 "related browsing contexts" 单元
  └── SiteInstance(s)     ← 同一 site 的文档集合（每 BrowsingInstance 唯一）
        └── SiteInfo      ← 安全主体（security principal），决定进程归属
              └── ProcessLock ← 进程被锁定到的 site/origin
```

**BrowsingInstance**（【真实源码 content/browser/browsing_instance.h，已 Fetch】）：

代表 HTML 5 规范中"有脚本连通性"的 browsing context 集合（即相互持有引用的 tab/frame 集合）。关键字段：
- `site_instance_map_`：站点到 SiteInstance 的映射，保证同 BrowsingInstance 内同一站点只有一个 SiteInstance
- `default_site_instance_`：非隔离站点共享的默认实例
- `web_exposed_isolation_info_`：COOP/COEP 隔离信息
- `token_`：`UnguessableToken`，不可猜测的唯一标识

**SiteInfo**（【真实源码 content/browser/site_info.h，已 Fetch】）：

安全主体，代表"一个进程被允许渲染什么内容"。核心字段：

```cpp
// 站点 URL（通常是 scheme + eTLD+1）
GURL site_url_;
// AgentCluster key（决定 SharedArrayBuffer 等 API 可用性）
AgentClusterKey agent_cluster_key_;
// COOP/COEP 隔离信息
WebExposedIsolationInfo web_exposed_isolation_info_;
// 是否要求独立进程（COOP header 触发）
bool does_site_request_dedicated_process_for_coop_;
// JIT/V8 优化禁用标志
bool is_jit_disabled_;
bool are_v8_optimizations_disabled_;
// 沙箱 iframe 标志
bool is_sandboxed_;
// fenced frame 内容标志
bool is_fenced_;
```

**"site" 的精确定义**（来自 site_info.h 注释，已 Fetch 核实）：

> "A site is often the scheme plus the eTLD+1 of a URL, allowing same-site URLs to share a process even when document.domain changes. However, some site URLs can be finer grained (e.g., origins) or coarser grained (e.g., file://)."

**ProcessLock**（【真实源码 content/browser/process_lock.h，已 Fetch】）：

进程锁，三种状态：
- **Invalid**：尚未关联任何 SiteInstance，拒绝所有访问
- **AllowAnySite**：已关联 SiteInstance 但未锁定到特定 site；要求 StoragePartition / COOP/COEP 匹配
- **LockedToSite**：进程严格锁定到单一 site，拒绝其他 site 的内容 commit

状态转移：Invalid → AllowAnySite（第一个 SiteInstance 关联时）→ LockedToSite（site 确定且进程未被使用时可升级）

### 3.2 进程分配决策流程

```
导航请求到达
      ↓
SiteInstance::GetOrCreateProcess()           [content/browser/site_instance_impl.cc]
      ↓
已有进程?
  ├── Yes → 检查 ProcessLock 是否兼容
  └── No  → GetOrCreateProcessForSite()
              ├── process-per-site 模式? → SiteProcessMap 查找
              ├── 有 spare renderer?    → SpareRenderProcessHostManager::MaybeTakeSpare()
              └── 新建 RenderProcessHostImpl
                      ↓
              LockProcessIfNeeded()           [site_instance_impl.cc]
                      ↓
              ChildProcessSecurityPolicy::LockProcess()  [security policy]
```

**SiteInstance::SetSite() 的惰性设计**（来自 site_instance_impl.h 注释，已 Fetch）：

> "The site is not set at SiteInstance creation time but rather when a navigation response is received, to enable better process sharing during redirects."

这是一个重要设计决策：SiteInstance 在创建时不立即绑定 site，等到导航响应返回、确认最终 URL 后才 SetSite + LockProcess。这样跨域重定向链可以在确认最终目的地前保持进程复用的灵活性。

### 3.3 进程模型选项历史与现状

（来源：chromium.googlesource.com/chromium/src/+/main/docs/process_model_and_site_isolation.md，已 Fetch）

| 模型 | 描述 | 当前状态 |
|------|------|---------|
| **site-per-process** | 每个 site 独立进程；OOPIF 跨进程 iframe | Desktop 默认（Chrome 67+） |
| **process-per-site-instance** | 跨 site 导航创建新进程；同 site 多 tab 可共享 | 历史选项，已被 site-per-process 替代 |
| **process-per-site** | 同 site 的所有 tab 合并到一个进程 | 仅用于 NTP 和 Extensions |
| **process-per-tab** | 每 browsing context group 一个进程，跨 site 不换 | 历史选项 |
| **single-process** | 调试用 `--single-process` flag | 仅测试/调试，不支持 |
| **Partial isolation** | 只隔离高风险站点 | Android 2GB+ RAM 设备 |

### 3.4 Spare Renderer（预热进程）

SpareRenderProcessHostManager 维护一个"预热但空闲"的 renderer 进程（【真实源码描述来自 render_process_host_impl.cc，已 Fetch】）：

```
SpareRenderProcessHostManagerImpl
├── PrepareForFutureRequests()  ← 预判即将到来的导航，提前 fork + init renderer
└── MaybeTakeSpare()           ← 新导航到来时，尝试接管 spare 进程
```

SpareProcessMaybeTakeAction（已 Fetch 核实）：
```cpp
enum SpareProcessMaybeTakeAction {
  kNoSparePresent,          // 没有 spare 进程
  kMismatchedBrowserContext,// BrowserContext 不匹配
  kRefusedByEmbedder,       // embedder 拒绝复用
  kSpareTaken               // 成功接管
};
```

意义：renderer 进程启动（fork + sandbox + V8 init）耗时 >100ms，spare 机制可把用户感知的导航延迟削减。

### 3.5 进程复用资源阈值

进程复用受内存约束（【真实源码 render_process_host_impl.cc，已 Fetch】）：

`IsBelowReuseResourceThresholds()` 检查：
- 主 frame 数量是否超过配置阈值
- 进程私有内存占用是否超过 512MB（subframe 默认）
- 是否附加了 DevTools（附加 DevTools 时禁止复用，避免调试污染）

```cpp
// 接口注释（原文）：
// Interface that represents the browser side of the browser <-> renderer
// communication channel. There will generally be one RenderProcessHost
// per renderer process.
class CONTENT_EXPORT RenderProcessHost
    : public IPC::Listener,
      public base::SupportsUserData {
 public:
  // Initialize the new renderer process, returning true on success.
  // This must be called once before the object can be used.
  virtual bool Init() = 0;

  // Gracefully shuts down the renderer process. Does not wait for unload
  // handlers to run.
  virtual bool Shutdown(int exit_code) = 0;

  // Shutdown without unload, only if no active documents.
  virtual bool FastShutdownIfPossible() = 0;

  // Obtain routing ID for IPC message dispatch.
  virtual int GetNextRoutingID() = 0;

  // --- Reference counting for process lifetime ---
  // Each holder must call AddRoute / RemoveRoute.
  virtual void AddRoute(int32_t routing_id, IPC::Listener* listener) = 0;
  virtual void RemoveRoute(int32_t routing_id) = 0;

  // Worker ref counts keep process alive while workers run.
  virtual void IncrementWorkerRefCount() = 0;
  virtual void DecrementWorkerRefCount() = 0;

  // Spare renderer detection.
  virtual bool IsSpare() const = 0;

  // Process lock: restricts which documents can render here.
  virtual void SetProcessLock(const IsolationContext&,
                               const ProcessLock&) = 0;

  // --- Static helpers ---
  static RenderProcessHost* FromID(ChildProcessId child_process_id);
  static AllHostsIterator AllHostsIterator();
  static bool IsProcessLimitReached();
```

### 4.2 RenderProcessHostImpl::Init() 核心流程

【真实源码描述 content/browser/renderer_host/render_process_host_impl.cc，已 Fetch，以下为结构性注解，非逐字源码】

```cpp
bool RenderProcessHostImpl::Init() {
  // 1. 幂等保护：已初始化则直接返回
  if (is_initialized_) return true;

  // 2. 初始化 Mojo Channel Proxy
  //    注意：进程死亡通知期间也可能调用 Init()，此时 Channel 为 null，
  //    需要重新初始化
  InitializeChannelProxy();
  // InitializeChannelProxy 内部：
  //   - 创建 MojoInvitation（引导 IPC 连接）
  //   - 建立 IPC::ChannelFactory（UI/IO 线程 task runner）
  //   - 获取 renderer_interface_ 关联代理
  //   - 暂停 channel：等 OnProcessLaunched 信号后再 flush + unpause

  // 3. GPU Channel 提前建立（可选）
  if (ShouldSendGpuChannelEarly()) {
    EstablishGpuChannel();
  }

  // 4. 进程启动
  if (run_renderer_in_process()) {
    // 测试/单进程模式：在当前进程内启动 renderer 线程
    g_renderer_main_thread_factory(startup_info);
  } else {
    // 正常路径：异步启动子进程（避免阻塞 UI 线程，启动耗时 >100ms）
    child_process_launcher_ = std::make_unique<ChildProcessLauncher>(
        std::make_unique<RendererSandboxedProcessLauncherDelegate>(),
        cmd_line,
        GetID(),
        this,                          // Client* 回调通知
        std::move(mojo_invitation),    // Mojo IPC 引导
        process_error_callback,
        file_data,
        histogram_memory,
        tracing_memory
    );
    // ChildProcessLauncher 异步完成后调用 OnProcessLaunched()
    // OnProcessLaunched() 中执行 channel->Unpause() + Flush()
  }

  is_initialized_ = true;
  return true;
}
```

关键设计点：
- **Channel 先暂停后解封**：在 OnProcessLaunched 前暂停 channel，确保消息在进程就绪后才发送，避免竞态。
- **异步启动**：`ChildProcessLauncher` 不阻塞 UI 线程，注释明确说明 "often it can take > 100ms to create the process"。

## 5. OOPIF：跨进程 iframe

Out-of-Process iFrame（OOPIF）是 site-per-process 落地后的必然产物：当 `https://parent.com` 的页面包含 `<iframe src="https://other.com">` 时，两者必须在不同进程渲染。

### 5.1 RenderFrameProxy 机制

【真实源码 content/browser/renderer_host/render_frame_proxy_host.h，已 Fetch】

```
Browser Process
├── RenderFrameHost A  ←→  Renderer Process A（parent.com）
│     └── RenderFrameProxyHost B  ←→  Renderer Process A 中的 RemoteFrame B
└── RenderFrameHost B  ←→  Renderer Process B（other.com）

Renderer Process A（parent.com）
├── RenderFrame（主文档）
└── RemoteFrame（other.com iframe 的占位符）
    ↑ 对应 Browser 侧的 RenderFrameProxyHost

Renderer Process B（other.com）
└── RenderFrame（iframe 真实渲染）
```

关键注释（已 Fetch 核实）：
> "When a page's frames are rendered by multiple processes, each renderer has a full copy of the frame tree."

每个 renderer 维护完整 frame 树，自己渲染的 frame 是真实 `RenderFrame`，其他进程渲染的是 `RemoteFrame`（占位符）。

### 5.2 跨进程通信路径

- **postMessage**：经 Mojo IPC，`RenderFrameProxyHost` 路由 → 目标 `RenderFrameHost` → 目标 renderer
- **focus/visibility/viewport 同步**：Browser 侧通过 `RenderFrameProxyHost` 同步属性变更
- **frame resize 异步性**（已 Fetch 核实的已知限制）：
  > "Full-page layout became asynchronous, affecting pages that change the size of a frame and then send a postMessage to it, since the receiving frame may not yet know its new size."

## 6. 进程模型在 Electron 中的映射

Electron 在 Chromium 多进程基础上叠加 Node.js，产生独特的进程模型。

### 6.1 Electron 进程类型

（来源：electronjs.org/docs/latest/tutorial/process-model，已 Fetch）

| Electron 进程 | 对应 Chromium 进程 | 额外能力 |
|--------------|-----------------|---------|
| Main Process | Browser Process | 完整 Node.js + native API |
| Renderer Process | Renderer Process | 可选 Node.js（contextIsolation + preload） |
| Utility Process | Utility Process | Node.js（`UtilityProcess` API，可与 renderer 通 MessagePort） |

`process.type` 值（来自 electron docs，已 Fetch）：
- `"browser"` - main process
- `"renderer"` - renderer process
- `"service-worker"` - service worker context
- `"worker"` - web worker context
- `"utility"` - UtilityProcess

### 6.2 ElectronRendererClient：Node.js 植入 Renderer

【真实源码 shell/renderer/electron_renderer_client.cc，已 Fetch】

```cpp
ElectronRendererClient::ElectronRendererClient()
    : node_bindings_{NodeBindings::Create(
          NodeBindings::BrowserEnvironment::kRenderer)},
      electron_bindings_{
          std::make_unique<ElectronBindings>(node_bindings_->uv_loop())} {}
```

Node.js 初始化发生在 `DidCreateScriptContext`：

1. 推迟文档加载，等待 preload 脚本异步执行
2. **保存 Blink 的 fetch API**（关键：Node.js 初始化会覆盖 `globalThis.fetch`，必须先保存）
3. 建立每 window 的 Node environment
4. 启动 UV 轮询 loop

### 6.3 libuv ↔ Chromium 消息循环集成

【真实源码 shell/common/node_bindings.cc，已 Fetch】

```
Chromium MessageLoop（UI 线程）
      │
      ├─ PostTask → UvRunOnce()    ← 触发 libuv 事件循环一次迭代
      │
libuv Event Loop（embed thread）
      ├─ PollEvents()              ← 等待 IO 事件
      └─ uv_sem_post()            → 通知 Chromium 线程 libuv 有工作
```

设计关键：**不替换 Chromium message loop**，而是通过 semaphore 协调两个 loop 交替运行，避免死锁。

```cpp
// 示意：PrepareEmbedThread + UvRunOnce 交替
void NodeBindings::PrepareEmbedThread() {
  // 创建独立 embed thread，专门 poll libuv events
  uv_thread_create(&embed_thread_, EmbedThreadRunner, this);
}
// EmbedThreadRunner:
//   while (true) {
//     uv_sem_wait(&embed_sem_);   // 等 Chromium 通知
//     PollEvents();               // poll libuv IO
//     uv_sem_post(&uv_sem_);      // 通知 Chromium libuv 有工作
//   }
```

### 6.4 Electron 与 Chromium 进程模型差异对比

| 维度 | Chromium（纯浏览器） | Electron |
|------|-------------------|---------|
| Renderer 有 Node.js | 否 | 可选（webPreferences.nodeIntegration） |
| contextIsolation 默认 | N/A（网页直接访问 window） | 默认开启（preload 通过 contextBridge） |
| sandbox | 默认开启（Desktop） | 历史上默认关闭，新版（≥v25）默认开启 |
| 跨进程通信 | Mojo IPC | Mojo IPC + ipcRenderer/ipcMain |
| 进程数量控制 | 由 BrowsingInstance 决定 | 可通过 webPreferences 控制是否复用 |
| GPU 进程 | 独立 | 同 Chromium |
| Network 进程 | 独立（Chrome 80+） | 同 Chromium（Electron 内嵌版本依赖） |

## 8. 失败模式、生产真坑与根因

### 8.1 Renderer 崩溃与 Sad Tab

**现象**：tab 显示"Aw, Snap!"，reload 可恢复。
**根因**：Renderer 进程崩溃。Browser process 通过 Mojo/IPC 连接的进程句柄监控检测（来源：chromium.org multi-process-architecture doc，已 Fetch）：
> "Each Mojo or IPC connection watches process handles. If these handles are signaled, the renderer process has crashed."

**产线影响**：
- 单 tab 崩溃不影响其他 tab（多进程隔离的核心收益）
- 崩溃的 renderer 若持有 SharedWorker，会影响所有使用该 worker 的 tab
- BrowserContext 层级的 session storage 可能丢失（renderer 内存）

**诊断**：`chrome://crashes`，或 `--enable-crash-reporter` + Breakpad/Crashpad minidump 分析。

### 8.2 GPU 进程反复崩溃导致软件渲染回退

**现象**：页面动画卡顿，硬件加速失效，WebGL 报错。
**根因**：GPU 进程崩溃计数超阈值，触发 fallback 到软件渲染（SwiftShader）。GPU driver bug 是常见触发源。

**代码证据**（已 Fetch，gpu_process_host.cc）：
- `recent_crash_count_` 超过 `GetFallbackCrashLimit()`（约 3–6 次）
- 时间窗口内崩溃不被宽恕 → GPU fallback

**排查**：`chrome://gpu`，查看 "Graphics Feature Status"，看是否有 `Hardware accelerated: Disabled`。

**修复路径**：更新/回滚 GPU driver；在 Electron 应用中通过 `app.disableHardwareAcceleration()` 主动 fallback 以避免用户遇到崩溃。

### 8.3 进程数量超限导致新 tab 打开慢

**现象**：大量 tab 打开后，新 tab 导航明显变慢。
**根因**：`RenderProcessHost::IsProcessLimitReached()` 触发，强制复用已有 renderer 进程（即使跨 site），退化为 process-per-site 模式，增加进程查找开销；同时 `IsBelowReuseResourceThresholds()` 的内存检查 (>512MB) 可能强制新建进程，增加延迟。

**排查**：
```bash
# 查看当前 Chrome renderer 进程数量
ps aux | grep -i "chrome.*renderer" | wc -l
# 或 chrome://process-internals 统计 SiteInstance 数量
```

**缓解**：关闭不用的 tab（每个 tab ~120–150MB renderer 内存）；生产 Electron 应用可通过 `webPreferences` 控制 BrowsingInstance 策略。

### 8.4 OOPIF postMessage size 陷阱

**现象**：`iframe.contentWindow.postMessage(largeData)` 延迟比预期高。
**根因**：跨 site OOPIF 的 postMessage 走 Mojo IPC（序列化 → browser process 路由 → 目标 renderer 反序列化），而同进程 postMessage 是直接内存拷贝。大消息（>1MB）会产生明显延迟。

**测量**：
```javascript
// 测量同进程 vs 跨进程 postMessage 延迟差异
const start = performance.now();
iframe.contentWindow.postMessage({ data: new ArrayBuffer(1024 * 1024) }, '*');
// 在 iframe 内监听并 postMessage 回来
window.addEventListener('message', (e) => {
  console.log('RTT:', performance.now() - start, 'ms');
});
// 预期：同进程 <1ms；跨进程 OOPIF 可能 >10ms
```

**规避**：使用 `SharedArrayBuffer` + `Atomics`（需要 COOP/COEP header，即在同一 Agent Cluster）；或将大数据通信设计为流式而非单次传输。

### 8.5 Electron contextIsolation 误配导致安全漏洞

**现象**：renderer 中的 `window` 对象可直接访问 Node.js API。
**根因**：`contextIsolation: false`（Electron 旧默认值）或 `nodeIntegration: true`，攻击者注入 XSS 后可 require('child_process').exec() 执行任意命令。

**根因链**（结合源码）：
1. `contextIsolation: false` → preload 脚本和网页共享同一 V8 context
2. ElectronRendererClient 在 kRenderer 环境中初始化 Node.js bindings
3. Node.js 的 `require` 函数泄漏到 `window` → 完整 Node.js 访问

**修复**：始终开启 `contextIsolation: true`（Electron v12+ 默认），通过 `contextBridge.exposeInMainWorld` 暴露最小必要 API。

### 8.6 Network Service 进程崩溃与请求中断

**现象**（Chrome 80+）：页面部分资源加载失败，控制台显示 `net::ERR_FAILED`，重试后恢复。
**根因**：Network Service 进程崩溃。Chrome 会自动重启 Network Service 并恢复 in-flight 请求（尽力而为），但重启期间的新请求失败。

**排查**：`chrome://histograms/NetworkServiceCrash`，或 `--enable-logging --v=1` 查看 network service 重启日志。

## 9. 方案对比：进程模型选项的适用边界

### 9.1 进程模型对比表

| 模型 | 隔离级别 | 内存开销 | 启动速度 | Spectre 防护 | 适用场景 |
|------|---------|---------|---------|-------------|---------|
| **site-per-process（默认）** | 最高（eTLD+1 级） | 最高（+10–13%） | 略慢（更多进程） | 最强 | Desktop Chrome；安全要求高 |
| **process-per-site** | 中（同 site 合并） | 中 | 中 | 中（同 site 共享内存） | NTP、Extensions |
| **Partial isolation** | 中（高风险站点独立） | 低（+3–5%） | 中 | 中 | Android 2GB+ |
| **single-process** | 无 | 最低 | 最快 | 无 | 单元测试，自动化测试 |
| **Electron（默认）** | 取决于 sandbox/contextIsolation | 同 Chromium | 略慢 | 依配置 | 桌面应用 |

### 9.2 "site" vs "origin" 隔离粒度对比

| 粒度 | 范围 | 示例 | 触发条件 |
|------|------|------|---------|
| **Site（eTLD+1）** | scheme + eTLD+1 | `https://github.com`（含 `gist.github.com`） | 默认 site-per-process |
| **Origin** | scheme + host + port | `https://api.github.com:443` | `Origin-Agent-Cluster: ?1` header |
| **Process isolation for COOP** | 独立进程 | `cross-origin-opener-policy: same-origin` 触发 | COOP header |
| **Sandboxed iframe** | 完全独立 | `<iframe sandbox>` | SiteInfo.is_sandboxed_ |

**不适用边界**：
- Origin-level 隔离需要页面主动发送 `Origin-Agent-Cluster: ?1` header，默认不开启。
- COOP 进程隔离有 `SharedArrayBuffer` 限制（需配合 `COEP`）。
- Android 低内存设备无法开启完整 site isolation，高价值 app 需额外评估安全风险。

## 章末五件套

### 1. 核心概念一句话

多进程架构把"不可信内容执行"（renderer）与"权限操作"（browser）分离到不同 OS 进程，用 OS 级隔离弥补应用层安全的先天不足。

### 2. 最易混淆的三对概念

| 概念 A | 概念 B | 区别 |
|--------|--------|------|
| **SiteInstance** | **SiteInfo** | SiteInstance 是运行时对象（含进程引用）；SiteInfo 是安全主体（纯数据，决定进程归属） |
| **BrowsingInstance** | **BrowserContext** | BrowsingInstance 对应 HTML5 related browsing contexts（可跨 tab，持有脚本引用）；BrowserContext 是 profile 级容器（如 Incognito） |
| **ProcessLock** | **SandboxPolicy** | ProcessLock 是"进程被允许渲染什么 site"的白名单（软件层 enforced by browser process）；SandboxPolicy 是"进程被 OS 允许做什么系统调用"（OS 层 enforced by kernel） |

### 3. 下一步深入路径

- **第 2 章**（建议）：Mojo IPC 消息管道——进程间通信的管道建立、接口绑定、消息序列化。
- **第 3 章**（建议）：导航流程——FrameTreeNode、RenderFrameHostManager、cross-BrowsingInstance 导航与 BackForwardCache。
- **OOPIF 深挖**：RenderFrameProxy / RemoteFrame / 跨进程 hit-testing。
- **Electron 安全**：contextIsolation 实现细节、preload 脚本安全边界、contextBridge 序列化规则。
