> **TL;DR**
> 投机解码是一种**无损加速**技术：用一个廉价的 draft 模型提前猜 k 个 token，再让 target 模型一次并行验证。验证轮次只消耗 1 次 target 前向，却平均输出 >1 个 token，突破了 memory-bandwidth 瓶颈。接受率越高、batch size 越小，加速比越显著；反之批量推理场景收益递减。

## 6.1 设计考古：为什么需要投机解码

### 根本瓶颈：decode 是 memory-bandwidth-bound

大模型 decode 阶段每步只生成 1 个 token，但 GPU 计算单元的峰值算力远超 HBM 带宽所能喂饱的量：

- H100 SXM：FP16 峰值 1979 TFLOPS，HBM3 带宽 3.35 TB/s
- 一个 70B FP16 模型参数量 ~140 GB；每 decode 步读全部参数一次（KV cache 已在片上）
- 每步计算量约 2 × 70B ≈ 140 GFLOPs（batch=1），带宽消耗 140 GB
- 算术强度 ≈ 1 FLOP/Byte，远低于设备 ridge point（~600 FLOP/Byte）

**结论**：decode 期间 GPU 大部分时间在等数据，计算单元空转。把 1 次 target forward 的计算量用来验证 k 个 token，边际成本极低。

### 论文发端

**两篇 2022-2023 年的奠基论文** 几乎同时独立提出了该思路：

1. **Leviathan et al., "Fast Inference from Transformers via Speculative Decoding"**  
   ICML 2023，Google Research。Yaniv Leviathan, Matan Kalman, Yossi Matias。  
   arXiv:2211.17192（2022 年 11 月上传）。  
   核心贡献：提出 draft+verify 框架，证明 modified rejection sampling 可以**无损**保证输出分布等价于 target 模型。在 T5-XXL 上实现 **2-3x 加速**。

2. **Chen et al., "Accelerating Large Language Model Decoding with Speculative Sampling"**  
   DeepMind，2023 年 2 月。Charlie Chen, Sebastian Borgeaud, Geoffrey Irving 等。  
   arXiv:2302.01318。  
   在 Chinchilla 70B 上实现 **2-2.5x 加速**，分布式 setup，同样采用 modified rejection sampling。

两篇论文均独立证明了相同的核心定理：modified rejection sampling 后，期望输出分布严格等于 target 分布。

### 更早的相关思路

- **Blockwise Parallel Decoding**（Stern et al., 2018, NeurIPS）：使用多个独立预测头并行生成，是最早的多 token 验证思路，但没有正式证明分布等价性。
- **Speculative Execution** 来自 CPU 微架构（branch prediction），论文作者明确借用了这一类比。

## 6.2 核心算法：modified rejection sampling

### 6.2.1 直觉理解

设 draft 分布为 $p(x)$，target 分布为 $q(x)$。对 draft 猜出的 token $\hat{x}$：

$$
\text{accept with prob} = \min\!\left(1,\; \frac{q(\hat{x})}{p(\hat{x})}\right)
$$

- 如果 target 认为该 token 比 draft 更可能（$q > p$），100% 接受
- 如果 target 认为该 token 不够好（$q < p$），按比例 $q/p$ 接受
- 拒绝后，从"残差分布" $\text{norm}(\max(0,\, q(x) - p(x)))$ 重采样一个 bonus token

**关键定理**：该过程输出的边际分布恰好等于 $q(x)$（target 分布），与 draft 质量无关。

### 6.2.2 算法步骤（Algorithm 1，来自 Leviathan et al.）

```
设 draft 模型 M_p，target 模型 M_q，猜测长度 k

1. 用 M_p 自回归生成 k 个草稿 token: x̃₁, x̃₂, ..., x̃ₖ
   同时记录每步分布 p(·|x₁...xₙ, x̃₁...x̃_{i-1})

2. 将 [x₁...xₙ, x̃₁...x̃ₖ] 整体输入 M_q，一次 forward 得到
   k+1 个位置的 logits（对应验证 x̃₁...x̃ₖ 及生成 bonus token）

3. for i = 1 to k:
     u ~ Uniform(0,1)
     if u < min(1, q(x̃ᵢ) / p(x̃ᵢ)):
         accept x̃ᵢ
     else:
         从 norm(max(0, q(x) - p(x))) 采样替换 token
         break（终止验证）

4. 若所有 k 个均接受，从 M_q 的第 k+1 位 logits 额外采样 1 个 bonus token

输出：至少 1 个，最多 k+1 个新 token
```

### 6.2.3 期望加速比推导

设每步平均接受长度为 $\bar{\alpha}$（acceptance rate），每次投机 k 步：

- 平均每轮输出 token 数：$E[\text{accepted}] = \frac{1 - \alpha^{k+1}}{1-\alpha}$（等比数列），其中 $\alpha$ 是单步接受率
- 每轮代价 ≈ 1 次 draft forward × k + 1 次 target forward
- 若 draft 模型速度是 target 的 $\gamma$ 倍（$\gamma \gg 1$），近似加速比：

$$
\text{speedup} \approx \frac{1 - \alpha^{k+1}}{1-\alpha} \cdot \frac{1}{1 + k/\gamma}
$$

实践中 $\alpha \approx 0.7$-$0.9$（任务依赖），$\gamma \approx 5$-$10$（小模型 vs 大模型），$k = 3$-$5$，典型加速 **2-3x**。

### 6.3.1 整体流程（v0.5.x 版本，概念仍适用）

获取自：WebFetch `https://raw.githubusercontent.com/vllm-project/vllm/v0.5.5/vllm/spec_decode/spec_decode_worker.py`

```python
# 【真实源码 vllm-project/vllm@v0.5.5/vllm/spec_decode/spec_decode_worker.py】

@nvtx_range("spec_decode_worker._run_speculative_decoding_step")
def _run_speculative_decoding_step(
        self, execute_model_req: ExecuteModelRequest,
        num_lookahead_slots: int) -> List[SamplerOutput]:
    """Execute a single step of speculative decoding.
    
    This invokes the proposer worker to get k speculative tokens for each
    sequence, then scores each speculative token using the scoring worker.
    """
    # ① 将 target 上一步的 hidden states 传给 draft 模型（EAGLE 需要）
    execute_model_req.previous_hidden_states = self.previous_hidden_states
    self.previous_hidden_states = None

    with Timer() as proposal_timer:
        # ② Draft 阶段：proposer 自回归生成 k 个草稿 token
        proposals = self.proposer_worker.get_spec_proposals(
            execute_model_req, self._seq_with_bonus_token_in_last_step)

    execute_model_req.previous_hidden_states = None

    with Timer() as scoring_timer:
        # ③ Verify 阶段：scorer（target 模型）一次 forward 打分
        proposal_scores = self.scorer.score_proposals(
            execute_model_req,
            proposals,
        )

    with Timer() as verification_timer:
        # ④ Rejection Sampling：决定哪些 token 被接受
        accepted_token_ids, target_logprobs = self._verify_tokens(
            execute_model_req.seq_group_metadata_list, proposal_scores,
            proposals, execute_model_req.num_lookahead_slots)

    stage_times = (proposal_timer.elapsed_time_ms / num_lookahead_slots,
                   scoring_timer.elapsed_time_ms,
                   verification_timer.elapsed_time_ms)

    # ⑤ 拼装输出，每个被接受的 token 包装成独立 SamplerOutput
    return self._create_output_sampler_list(
        execute_model_req.seq_group_metadata_list,
        accepted_token_ids,
        target_logprobs=target_logprobs,
        k=execute_model_req.num_lookahead_slots,
        stage_times=stage_times)
```

注解：
- `proposer_worker` 可以是 MultiStepWorker（小模型）、NGramWorker（n-gram 匹配）或 MedusaWorker
- `scorer` 是 target 大模型的包装
- `previous_hidden_states`：EAGLE 框架中，draft 模型需要 target 上一步的倒数第二层 hidden states 作为条件输入

### 6.3.2 _verify_tokens：rejection sampling 调用入口

```python
# 【真实源码 vllm-project/vllm@v0.5.5/vllm/spec_decode/spec_decode_worker.py】

@nvtx_range("spec_decode_worker._verify_tokens")
def _verify_tokens(self, seq_group_metadata_list, proposal_scores,
                   proposals, max_proposal_len):
    """Determine which speculative tokens are accepted using the
    probabilities of each token according to the proposer and scorer models.
    """
    # 分离：有草稿的序列 vs 无草稿的序列（fallback to normal decode）
    _, spec_indices = split_batch_by_proposal_len(
        seq_group_metadata_list, proposal_lens_list, select_proposal_len_zero=False)
    _, non_spec_indices = split_batch_by_proposal_len(
        seq_group_metadata_list, proposal_lens_list, select_proposal_len_zero=True)

    # target 模型对前 k 个位置的概率（去掉 bonus token 位置）
    proposal_verifier_probs = proposal_scores.probs[spec_indices, :-1]
    # target 模型的 bonus token（k+1 位置）
    bonus_token_ids = proposal_scores.token_ids[spec_indices, -1:]
    # draft 模型分配的概率
    proposal_probs = proposals.proposal_probs[spec_indices]
    proposal_token_ids = proposals.proposal_token_ids[spec_indices]

    # 调用 RejectionSampler 或 TypicalAcceptanceSampler
    accepted_token_ids = self.spec_decode_sampler(
        target_probs=proposal_verifier_probs,   # q(x̃ᵢ)
        bonus_token_ids=bonus_token_ids,
        draft_probs=proposal_probs,             # p(x̃ᵢ)
        draft_token_ids=proposal_token_ids,
        **sampler_extra_kwargs,
    )
    # ... 拼合无草稿序列的输出，重排为原始顺序
```

### 6.3.3 Triton Kernel：rejection sampling 实现（v1 架构）

vLLM v1 将 rejection sampling 用 Triton kernel 实现，追求极致并行。  
获取自 GitHub API，**commit `f1e13f7`**，路径 `vllm/v1/worker/gpu/spec_decode/rejection_sampler_utils.py`

```python
# 【真实源码 vllm-project/vllm@f1e13f7/vllm/v1/worker/gpu/spec_decode/rejection_sampler_utils.py】
# 核心 Triton kernel：逐 token 执行 modified rejection sampling

@triton.jit
def _rejection_kernel(
    sampled_ptr, sampled_stride,
    rejected_steps_ptr,
    target_rejected_logsumexp_ptr,
    draft_rejected_logsumexp_ptr,
    target_logits_ptr, target_logits_stride,
    target_local_argmax_ptr, target_local_argmax_stride,
    target_local_max_ptr, target_local_max_stride,
    target_local_sumexp_ptr, target_local_sumexp_stride,
    draft_sampled_ptr,
    draft_logits_ptr, draft_logits_stride_0, draft_logits_stride_1,
    draft_local_max_ptr, draft_local_max_stride,
    draft_local_sumexp_ptr, draft_local_sumexp_stride,
    cu_num_logits_ptr,
    idx_mapping_ptr,
    temp_ptr,           # 每个 request 的 temperature
    seed_ptr,           # 每个 request 的随机种子
    pos_ptr,
    synthetic_conditional_rates_ptr,
    vocab_num_blocks,
    PADDED_VOCAB_NUM_BLOCKS: tl.constexpr,
    HAS_DRAFT_LOGITS: tl.constexpr,  # draft 是否提供完整 logits
    SYNTHETIC_MODE: tl.constexpr,
):
    req_idx = tl.program_id(0)   # 每个 request 一个 Triton program
    # ...
    rejected_step = 0
    accepted = True

    for i in range(num_tokens - 1):   # 遍历 k 个草稿位置
        if accepted:
            draft_sampled = tl.load(draft_sampled_ptr + logit_idx + 1)

            if temp == 0.0:
                # Greedy 模式：draft 必须与 target argmax 完全匹配
                target_argmax = ...   # 从预计算的 block-level stats 中取
                accepted &= target_argmax == draft_sampled
                # 无论接受/拒绝，先存草稿 token，拒绝时会被 argmax 覆盖
                tl.store(sampled_ptr + req_idx * sampled_stride + i,
                         draft_sampled if accepted else target_argmax)
            else:
                # Stochastic 模式：核心 rejection sampling 公式
                target_logit = tl.load(target_logits_ptr + ...)
                target_lse = _compute_global_lse(...)   # log-sum-exp
                target_log_prob = target_logit - target_lse   # log q(x̃ᵢ)

                u = tl_rand64(seed, pos, includes_zero=False)  # u ~ Uniform(0,1)

                if HAS_DRAFT_LOGITS:
                    draft_logit = tl.load(draft_logits_ptr + ...)
                    draft_log_prob = draft_logit - draft_lse  # log p(x̃ᵢ)
                else:
                    draft_log_prob = 0  # one-hot draft: p(x̃ᵢ) = 1

                # ★ 核心判断：log p(x) > log(u) + log q(x)
                # 等价于 p(x) > u * q(x)
                # 等价于 u < p(x)/q(x) = min(1, q/p) 接受准则的 log 形式
                # 注意：这里 target=p，draft=q，与论文记号相反！
                # vLLM 中 target_log_prob 是我们要等价的分布 (论文里的 q)
                #         draft_log_prob 是 proposal 分布 (论文里的 p)
                accepted &= target_log_prob > tl.log(u) + draft_log_prob

                tl.store(sampled_ptr + req_idx * sampled_stride + i, draft_sampled)

            rejected_step += accepted   # 记录接受了多少步（用于后续 bonus token 索引）

    tl.store(rejected_steps_ptr + req_idx, rejected_step)
```

**逐行关键注解**：
- `target_log_prob > tl.log(u) + draft_log_prob`：这是 `u < exp(target_log_prob - draft_log_prob) = q(x)/p(x)` 的等价 log 形式，精度更好，避免 exp 溢出
- `HAS_DRAFT_LOGITS=False`：当 draft 是 one-hot（如 n-gram），`p(x)=1`，`log p = 0`，退化为"只要 target 有正概率就接受"
- `tl.program_id(0)` 对应一个 request，batch 内所有 request 并行执行

### 6.3.4 残差分布重采样（bonus token / rejected token）

```python
# 【真实源码 vllm-project/vllm@f1e13f7/vllm/v1/worker/gpu/spec_decode/rejection_sampler_utils.py】
# _resample_kernel 中的残差逻辑

if is_bonus:
    # 全部接受后的 bonus token：直接用 target logits 采样
    residual_logits = target_logits

elif HAS_DRAFT_LOGITS:
    # 拒绝时：从 max(q(x) - p(x), 0) 归一化后采样
    # log 形式：target_log_probs + log(max(1 - exp(draft_log_probs - target_log_probs), 0))
    ratio = tl.exp(draft_log_probs - target_log_probs)   # p(x) / q(x)
    residual_logits = tl.where(
        ratio < 1.0,
        target_log_probs + tl.log(1 - ratio),   # log(q(x) - p(x)) 当 q > p
        float("-inf"),                            # q < p 时置为 -inf（概率 0）
    )

else:
    # One-hot draft（n-gram）：残差 = target 分布中排除该 token
    rejected_draft_token = tl.load(draft_sampled_ptr + ...)
    residual_logits = tl.where(
        block != rejected_draft_token,
        target_logits,
        float("-inf"),   # 被拒绝的那个 token 概率清零
    )
```

这段代码完整实现了论文中残差分布的三种情形，数值稳定性优于直接做 softmax 后相减。

### 6.3.6 MultiStepWorker：小模型草稿的多步执行

```python
# 【真实源码 vllm-project/vllm@v0.5.5/vllm/spec_decode/multi_step_worker.py】

@torch.inference_mode()
def sampler_output(self, execute_model_req, sample_len,
                   seq_ids_with_bonus_token_in_last_step):
    # 路径 ①：GPU multi-step（TP1 draft，单卡可在 GPU 上循环 k 步）
    if isinstance(self.model_runner, TP1DraftModelRunner) \
            and self.model_runner.supports_gpu_multi_step(expanded_request):
        expanded_request.num_steps = sample_len
        model_outputs = self.execute_model(execute_model_req=expanded_request)
    else:
        # 路径 ②：CPU 串行多步（每步追加新 token，再次 forward）
        for _ in range(sample_len):
            model_output = super().execute_model(
                execute_model_req=expanded_request)
            self._append_new_tokens(
                model_output, expanded_request.seq_group_metadata_list)
            model_outputs.append(model_output)
```

`TP1DraftModelRunner` 会在 GPU 上融合 k 步 draft 为一次调用（CUDA Graph 捕获），避免 Python 循环开销。

## 6.4 各类 Draft 方案对比

| 方案 | 来源 | Draft 方式 | 需要训练？ | 接受率典型值 | 适用场景 |
|------|------|-----------|----------|------------|---------|
| **小模型草稿** | Leviathan'23, Chen'23 | 独立小 LM（参数量 1/10-1/50） | 否（直接用） | 0.7-0.85 | 通用；有合适配对模型 |
| **N-gram / Prompt Lookup** | Fu et al. ICML'24 | 在 context 内查 n-gram 匹配 | 否 | 0.4-0.7 | 长 prompt 摘要、代码、重复对话 |
| **Medusa** | Cai et al. 2024 | k 个额外 MLP 头，附在主模型上 | 是（微调） | 0.65-0.8 | 固定主干、资源受限场景 |
| **EAGLE** | Li et al. 2024 | draft 头用 target hidden states 作条件 | 是（轻量训练） | 0.8-0.92 | 最高接受率；需 hidden states 传递 |
| **EAGLE-2** | Li et al. 2024 | 动态 draft tree（context-aware） | 是（EAGLE 基础上） | 0.85-0.95 | 比 EAGLE-1 快 20-40% |
| **MTP（Multi-Token Pred）** | Gloeckle et al. 2024 | 训练时多头联合预测 | 是（改训练目标） | - | DeepSeek-V3 等原生支持 |

### 方案选择矩阵

```
高接受率需求     → EAGLE-2
零额外模型      → N-gram（prompt 含大量复用段）
通用开箱即用    → 小模型草稿（如 Llama-68M 配 Llama-70B）
多硬件/量化限制 → Medusa（heads 很轻量）
原生多 token    → MTP（如 DeepSeek-V3 MTP 头）
```

## 6.5 Medusa：树形注意力与并行验证

### Medusa 架构

**arXiv:2401.10774** | Tianle Cai, Yuhui Li, Zhengyang Geng, Hongwu Peng, Jason D. Lee, Deming Chen, Tri Dao

Medusa 不使用独立 draft 模型，而在主模型 top 层接 k 个独立 MLP head：

```python
# 【真实源码 FasterDecoding/Medusa@main/medusa/model/medusa_model.py】

self.medusa_head = nn.ModuleList([
    nn.Sequential(
        *([ResBlock(self.hidden_size)] * medusa_num_layers),  # 1-2 层 MLP
        nn.Linear(self.hidden_size, self.vocab_size, bias=False),
    )
    for _ in range(medusa_num_heads)   # 通常 k=5
])
# head[0] 预测 t+1，head[1] 预测 t+2，...，head[k-1] 预测 t+k
# 所有 head 共享同一份 hidden_state（主模型最后一层输出）
```

### Tree Attention Mask

Medusa 的 k 个头各产生 top-N 候选，笛卡尔积形成一棵候选树。验证时用一次 forward，通过定制 attention mask 让每个候选路径只看到其祖先 token：

```python
# 【真实源码 FasterDecoding/Medusa@main/medusa/model/utils.py】

def generate_medusa_buffers(medusa_choices, device="cuda"):
    """Generate buffers for the Medusa structure."""
    sorted_medusa_choices = sorted(medusa_choices, key=lambda x: (len(x), x))
    medusa_len = len(sorted_medusa_choices) + 1

    # 构建树形 attention mask：每个节点只 attend 到自己的祖先
    medusa_attn_mask = torch.eye(medusa_len, medusa_len)
    medusa_attn_mask[:, 0] = 1   # 所有节点都可以看到根节点（当前已生成 token）
    # ... 递归填充父子关系
    
    medusa_buffers = {
        "medusa_attn_mask": medusa_attn_mask.unsqueeze(0).unsqueeze(0),
        "tree_indices": medusa_tree_indices,      # 候选树的 token 序列
        "medusa_position_ids": medusa_position_ids,
        "retrieve_indices": retrieve_indices,     # 从验证结果中取回最优路径
    }
    return medusa_buffers
```

### Medusa 候选生成与后验评估

```python
# 【真实源码 FasterDecoding/Medusa@main/medusa/model/utils.py】

def generate_candidates(medusa_logits, logits, tree_indices, retrieve_indices, ...):
    """Generate candidates from Medusa heads."""
    # 主模型最后一个 token 的 greedy/sampling 结果
    candidates_logit = torch.argmax(logits[:, -1]).unsqueeze(0)
    # 每个 Medusa head 取 top-K 候选
    candidates_medusa_logits = torch.topk(medusa_logits[:, 0, -1], TOPK, dim=-1).indices
    # 拼合，通过 tree_indices 重排为树形输入
    candidates = torch.cat([candidates_logit, candidates_medusa_logits.view(-1)], dim=-1)
    tree_candidates = candidates[tree_indices]
    return cart_candidates, tree_candidates.unsqueeze(0)

def evaluate_posterior(logits, candidates, temperature, ...):
    """Select best verified candidate path."""
    if temperature == 0:
        # Greedy：候选路径与 target argmax 逐位对比
        posterior_mask = (candidates[:, 1:] == torch.argmax(logits[:, :-1], dim=-1)).int()
        candidates_accept_length = (torch.cumprod(posterior_mask, dim=1)).sum(dim=1)
        accept_length = candidates_accept_length.max()
        best_candidate = torch.argmax(candidates_accept_length).to(torch.long)
        return best_candidate, accept_length
```

`torch.cumprod`：一旦某位不匹配，后续都变 0，精确实现"连续前缀接受"语义。

## 6.6 EAGLE：借 Hidden States 提升接受率

### 设计动机

**EAGLE（arXiv:2401.15077）**：Yuhui Li, Fangyun Wei, Chao Zhang, Hongyang Zhang  
EAGLE-2（arXiv:2406.16858）：同作者

小模型草稿仅看 token 序列，信息量不足；EAGLE 观察到 target 模型的 **penultimate layer hidden states** 包含丰富的语义预测信息。将这份 hidden states 传给轻量 draft head，draft 的预测更准确。

### EAGLE 架构（vLLM v1 实现）

```python
# 【真实源码 vllm-project/vllm@f1e13f7/vllm/v1/spec_decode/eagle.py】

class EagleProposer(SpecDecodeBaseProposer):
    def __init__(self, vllm_config, device, runner=None):
        super().__init__(
            vllm_config,
            device,
            pass_hidden_states_to_model=True,   # ← 关键标志
            runner=runner,
        )
# SpecDecodeBaseProposer 初始化时，pass_hidden_states_to_model=True
# 告知 draft 模型需要 target 的 hidden states 作为额外输入
```

EAGLE 的 draft head 结构：
1. Target 每次 decode 产生 `hidden_state_t`（penultimate layer output）
2. Draft head 以 `[token_embedding(x̃_{t-1}), hidden_state_t]` 为输入，自回归生成 k 步

EAGLE-2 在此基础上引入**动态 draft tree**：根据 draft model 的 confidence scores 估计每个分支的 acceptance rate，只展开高置信度的分支，避免浪费计算在注定被拒绝的低概率路径。

## 6.7 配套可运行 Demo

### Demo 1：Toy Speculative Decoding（纯 numpy，展示 draft+verify 核心逻辑）

**设计为可运行，无 GPU 依赖，仅需 `numpy`。**  
此 demo 与 6.2 节算法和 6.3.3 节 Triton kernel 的逻辑一一对应。

```python
#!/usr/bin/env python3
"""
Toy Speculative Decoding Demo
演示 modified rejection sampling 算法的核心逻辑。
用 numpy 模拟 draft/target 分布，展示接受率与加速比。

依赖: numpy (pip install numpy)
运行: python spec_decode_toy.py
"""
import numpy as np
rng = np.random.default_rng(42)


# ──────────────────────────────────────────────
# 模拟 target 和 draft 的分布（vocab_size=10）
# ──────────────────────────────────────────────
VOCAB_SIZE = 10

def make_target_probs():
    """target 分布：集中在 vocab 前半"""
    raw = np.array([3.0, 2.5, 2.0, 1.5, 1.0, 0.5, 0.3, 0.2, 0.1, 0.1])
    return raw / raw.sum()

def make_draft_probs(target_q, noise=0.3):
    """draft 分布：target 加噪，模拟小模型的不完美预测"""
    noise_vec = rng.dirichlet(np.ones(VOCAB_SIZE) * 0.5) * noise
    noisy = target_q * (1 - noise) + noise_vec
    return noisy / noisy.sum()


# ──────────────────────────────────────────────
# 核心算法：modified rejection sampling
# 对应 vLLM rejection_sampler_utils.py _rejection_kernel
# ──────────────────────────────────────────────
def rejection_sampling_accept(
    draft_token: int,
    q: np.ndarray,   # target 分布（论文中的 q）
    p: np.ndarray,   # draft  分布（论文中的 p）
    u: float,        # Uniform(0,1) 随机数
) -> tuple[bool, int]:
    """
    判断 draft_token 是否被接受，若拒绝则从残差分布采样替代 token。
    
    接受准则：u < q(x̃) / p(x̃)，等价于 u < min(1, q/p)
    对应 Triton kernel 中：target_log_prob > log(u) + draft_log_prob
    """
    accept_prob = min(1.0, q[draft_token] / (p[draft_token] + 1e-9))
    
    if u < accept_prob:
        return True, draft_token
    else:
        # 残差分布：norm(max(q - p, 0))
        residual = np.maximum(q - p, 0)
        residual_sum = residual.sum()
        if residual_sum < 1e-9:
            # 极端情况：残差为零，从 q 直接采样
            residual = q
            residual_sum = 1.0
        residual /= residual_sum
        replacement = rng.choice(VOCAB_SIZE, p=residual)
        return False, replacement


def speculative_decode_step(
    q: np.ndarray,      # target 分布（固定，模拟 1 次 target forward）
    p_list: list,       # 每步的 draft 分布（k 步 draft forward）
    draft_tokens: list, # k 个草稿 token（从各自 p_list[i] 采样）
) -> list[int]:
    """
    执行一轮投机解码，返回接受的 token 序列（长度 1 到 k+1）。
    
    对应 _run_speculative_decoding_step 中 proposal → score → verify 三阶段。
    """
    accepted = []
    
    for i, (draft_tok, p_i) in enumerate(zip(draft_tokens, p_list)):
        u = rng.uniform(0, 1)
        is_accepted, out_token = rejection_sampling_accept(draft_tok, q, p_i, u)
        
        if is_accepted:
            accepted.append(out_token)
        else:
            # 拒绝：用残差分布的替代 token 终止
            accepted.append(out_token)
            return accepted   # 早停，不看后续草稿
    
    # 全部接受：从 target 采样 bonus token（第 k+1 个）
    bonus_token = rng.choice(VOCAB_SIZE, p=q)
    accepted.append(bonus_token)
    return accepted


# ──────────────────────────────────────────────
# 实验：不同 draft 质量下的接受率与加速比
# ──────────────────────────────────────────────
def run_experiment(k: int = 4, num_steps: int = 10000, noise: float = 0.3):
    q = make_target_probs()
    
    total_tokens_output = 0
    total_target_forwards = 0   # 每轮 1 次 target forward
    
    for _ in range(num_steps):
        # Draft 阶段：k 次 draft forward，生成 k 个草稿
        p_list = [make_draft_probs(q, noise=noise) for _ in range(k)]
        draft_tokens = [rng.choice(VOCAB_SIZE, p=p_i) for p_i in p_list]
        
        # Verify 阶段：1 次 target forward（此处 q 固定，真实场景每步不同）
        accepted = speculative_decode_step(q, p_list, draft_tokens)
        
        total_tokens_output += len(accepted)
        total_target_forwards += 1
    
    tokens_per_step = total_tokens_output / total_target_forwards
    # 理论上 naive decode 每步 1 个 token（1 次 target forward）
    # 投机解码：平均每次 target forward 输出 tokens_per_step 个
    speedup_estimate = tokens_per_step  # 假设 draft 速度远快于 target
    
    return tokens_per_step, speedup_estimate


# ──────────────────────────────────────────────
# 验证：rejection sampling 后输出分布等于 target
# ──────────────────────────────────────────────
def verify_output_distribution(num_samples: int = 100000):
    """验证 rejection sampling 的分布等价性定理。"""
    q = make_target_probs()
    p = make_draft_probs(q, noise=0.4)
    
    # 方法 1：直接从 target 采样
    direct_samples = rng.choice(VOCAB_SIZE, size=num_samples, p=q)
    
    # 方法 2：rejection sampling
    spec_samples = []
    for _ in range(num_samples):
        draft_tok = rng.choice(VOCAB_SIZE, p=p)
        u = rng.uniform(0, 1)
        _, out_tok = rejection_sampling_accept(draft_tok, q, p, u)
        spec_samples.append(out_tok)
    spec_samples = np.array(spec_samples)
    
    # 对比两种方法的经验分布
    print("\n=== 分布等价性验证 ===")
    print(f"{'token':>6} | {'target_q':>10} | {'direct':>10} | {'spec_decode':>12} | {'diff':>8}")
    print("-" * 55)
    for t in range(VOCAB_SIZE):
        direct_freq = (direct_samples == t).mean()
        spec_freq = (spec_samples == t).mean()
        print(f"{t:>6} | {q[t]:>10.4f} | {direct_freq:>10.4f} | {spec_freq:>12.4f} | {abs(spec_freq-q[t]):>8.4f}")
    
    max_diff = max(abs((spec_samples == t).mean() - q[t]) for t in range(VOCAB_SIZE))
    print(f"\nMax L∞ diff (spec vs target): {max_diff:.4f}  (期望 < 0.01 for N={num_samples})")


if __name__ == "__main__":
    print("=" * 60)
    print("Toy Speculative Decoding Demo")
    print("=" * 60)
    
    # 实验 1：不同 draft 质量对接受率的影响
    print("\n=== 实验 1：draft noise 对平均接受 token 数的影响 (k=4) ===")
    print(f"{'noise':>8} | {'tokens/step':>12} | {'接受率估计':>12}")
    print("-" * 40)
    for noise in [0.05, 0.15, 0.30, 0.50, 0.80]:
        tps, speedup = run_experiment(k=4, num_steps=20000, noise=noise)
        alpha_est = (tps - 1) / 4  # 从 tokens/step 反推单步接受率（近似）
        print(f"{noise:>8.2f} | {tps:>12.3f} | {alpha_est:>12.3f}")
    
    # 实验 2：k 值对加速的影响（高质量 draft）
    print("\n=== 实验 2：猜测长度 k 对加速比的影响 (noise=0.15) ===")
    print(f"{'k':>4} | {'tokens/step':>12} | {'理论上限(α=0.85)':>18}")
    print("-" * 40)
    alpha_theory = 0.85
    for k in [1, 2, 3, 4, 5, 8]:
        tps, _ = run_experiment(k=k, num_steps=20000, noise=0.15)
        # 理论值：(1 - α^(k+1)) / (1 - α)
        theory = (1 - alpha_theory**(k+1)) / (1 - alpha_theory)
        print(f"{k:>4} | {tps:>12.3f} | {theory:>18.3f}")
    
    # 实验 3：验证输出分布等价
    verify_output_distribution(num_samples=50000)
```

### Demo 2：树形验证（Medusa style，纯 Python）

```python
#!/usr/bin/env python3
"""
Tree Verification Demo — 模拟 Medusa 的树形候选验证
说明：用嵌套 dict 模拟候选树，展示为什么树形验证能比线性验证多接受 token。

依赖: numpy
运行: python tree_verify_toy.py
"""
import numpy as np
rng = np.random.default_rng(123)

VOCAB_SIZE = 20

def sample_from(probs):
    return rng.choice(len(probs), p=probs / probs.sum())

def topk_tokens(probs, k=2):
    return np.argsort(probs)[-k:][::-1]

def run_linear_verify(target_q, draft_seqs, k=3):
    """线性策略：取第一条路径，逐位验证"""
    tokens = draft_seqs[0]   # shape (k,)
    accepted = []
    for i in range(k):
        accept_prob = min(1.0, target_q[i, tokens[i]] / (1.0/VOCAB_SIZE + 1e-9))
        if rng.uniform() < accept_prob:
            accepted.append(tokens[i])
        else:
            # 残差采样
            replacement = sample_from(target_q[i])
            accepted.append(replacement)
            return accepted
    bonus = sample_from(target_q[k])
    accepted.append(bonus)
    return accepted

def run_tree_verify(target_q, draft_seqs, k=3):
    """树形策略：多条候选路径，选最优通过路径"""
    best_accept = []
    
    for path in draft_seqs:   # 尝试每条路径
        accepted = []
        early_reject = False
        for i in range(k):
            accept_prob = min(1.0, target_q[i, path[i]] / (1.0/VOCAB_SIZE + 1e-9))
            if rng.uniform() < accept_prob:
                accepted.append(path[i])
            else:
                replacement = sample_from(target_q[i])
                accepted.append(replacement)
                early_reject = True
                break
        if not early_reject:
            bonus = sample_from(target_q[k])
            accepted.append(bonus)
        if len(accepted) > len(best_accept):
            best_accept = accepted
    
    return best_accept

# 实验
print("=== 线性 vs 树形验证 ===")
print(f"{'n_paths':>10} | {'linear_avg':>12} | {'tree_avg':>12} | {'提升':>8}")
print("-" * 45)

k = 3
n_trials = 5000

for n_paths in [1, 2, 4, 8]:
    linear_total, tree_total = 0, 0
    
    for _ in range(n_trials):
        # 模拟 target 的位置级分布（k+1 步）
        target_q = rng.dirichlet(np.ones(VOCAB_SIZE) * 2.0, size=k+1)
        
        # 生成 n_paths 条草稿路径
        draft_seqs = [
            np.array([topk_tokens(target_q[i] + rng.normal(0, 0.05, VOCAB_SIZE), 1)[0]
                      for i in range(k)])
            for _ in range(n_paths)
        ]
        
        linear_total += len(run_linear_verify(target_q, draft_seqs, k))
        tree_total += len(run_tree_verify(target_q, draft_seqs, k))
    
    lin_avg = linear_total / n_trials
    tree_avg = tree_total / n_trials
    print(f"{n_paths:>10} | {lin_avg:>12.3f} | {tree_avg:>12.3f} | {(tree_avg/lin_avg - 1)*100:>7.1f}%")
```

## 6.8 生产真坑与失败模式

### 坑 1：Batch size 增大后加速消失

**现象**：单请求 3x 加速，但 batch=32 时降回 1.1x 甚至更慢。  
**根因**：
1. draft 模型也需要计算 batch 内所有序列，其 latency 从 O(1) 变 O(batch)
2. target 模型的计算量随 batch 增加，变为 compute-bound，此时带宽节省优势消失
3. batch 内序列长度不齐，短序列会被 padding 浪费

**解法**：
- vLLM 的 `disable_by_batch_size`：超过阈值自动 fallback 到 normal decode
- 研究方向：batch-level speculative decoding（序列间 draft 共享）

### 坑 2：KV Cache 分裂导致 OOM

**现象**：开启投机解码后 GPU 显存用量明显增加，甚至 OOM。  
**根因**：draft 模型有自己的 KV cache，且 draft 验证阶段需要暂存 k 个位置的 KV，相当于每个序列额外占用 k × layer × 2 × d_model × sizeof(dtype) 字节。

**vLLM 的处理**：`split_num_cache_blocks_evenly` 函数按 block size 比例拆分可用 GPU blocks 给 draft/target 各自的 KV cache manager。

### 坑 3：分布式场景的同步顺序

**现象**：多卡 TP/PP 场景下，draft 和 target 的 rank 同步出错，导致死锁或结果错误。  
**根因**：投机解码需要 target → draft 的 hidden state 传递，打破了正常 decode 的单向数据流。vLLM 注释：  
> "non-driver ranks: scorer forward first, then proposer forward for prefill; but proposer forward first for decode"  

不同阶段的执行顺序不同，必须严格对齐所有 rank 的执行序列，否则 NCCL 操作错序导致挂起。

### 坑 4：接受率统计不准导致误判加速

**现象**：测报的 acceptance rate 很高（0.9），但端到端延迟未改善。  
**根因**：acceptance rate 统计的是 token 级接受比例，但实际加速比取决于：
- draft 模型 latency（draft 越快越好）
- k 值设置（k 太小收益小，k 太大被拒绝后浪费 draft 计算）
- 实际 TTFT 和 ITL

**正确监控指标**：`tokens_per_target_forward`（每次 target forward 平均输出 token 数），而不是 acceptance rate。

### 坑 5：n-gram 在创意生成场景失效

**现象**：摘要任务 2x 加速，但创意写作/开放生成场景退化到 0.9x（比 normal decode 慢）。  
**根因**：n-gram 在 context 中找匹配，创意生成时输出与 prompt 差异大，几乎找不到匹配，draft 质量极差（acceptance rate < 0.2）；但 n-gram 查找本身有 CPU 开销，净效果为负。

**解法**：对任务类型感知地开关投机解码（vLLM 支持 per-request 的 `speculative_config`）。

### 坑 6：draft/target tokenizer 必须完全一致

HuggingFace assisted generation 文档明确：  
> "The assistant must have the exact same tokenizer as your model"

如果 draft 用不同版本的 tokenizer（如 sentencepiece vs tiktoken），token id 映射不同，draft 提出的 token id 在 target 词表中是另一个词，rejection 率 100%，且难以发现（acceptance rate 不为 0，因为有 bonus token）。

## 6.9 方案对比实测（概念推演）

### 内存带宽节省分析

| 场景 | target forward | 输出 token 数 | 带宽节省 |
|------|---------------|-------------|---------|
| naive decode | 1 | 1 | 基准 |
| 投机 k=4，α=0.8 | 1 | ~3.5 | 3.5x token/forward |
| 投机 k=4，α=0.5 | 1 | ~2.1 | 2.1x token/forward |
| 投机 k=4，α=0.3 | 1 | ~1.5 | 1.5x token/forward |

注意：以上是 target forward 级别的 token 吞吐；加上 draft 模型的开销，实际端到端加速比更低，通常 draft/target 速度比决定上限：

$$\text{实际加速} = \frac{\bar{\alpha}_{\text{tokens/step}}}{{1 + k \cdot (t_{\text{draft}}/t_{\text{target}})}}$$

### 不同方案场景适用性

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 长代码补全 | N-gram / EAGLE | 代码高度重复，n-gram 高命中；EAGLE 精度高 |
| 多轮对话（短回复） | 小模型草稿 | 短序列 draft 开销低，小模型效果好 |
| 文档摘要 | N-gram | 摘要大量复用原文，命中率极高 |
| 大批量推理 | 通常不开 | batch 越大收益越小，compute-bound 后无收益 |
| 移动端/边缘（单核） | Medusa | 无需独立 draft 模型，单 GPU 内完成 |

## 6.11 未来方向

1. **Universal Speculative Decoding**（异构 tokenizer）：HuggingFace `UniversalSpeculativeDecodingGenerator` 已开始探索，但 token 对齐问题仍棘手。
2. **Online Draft Model Adaptation**：根据运行时 acceptance rate 动态调整 draft 模型或 k 值，避免场景切换时的性能抖动。
3. **Batch-Level 投机**：多个请求共享同一组 draft token（如相同 prompt prefix），减少 draft 模型的批量开销。
4. **与 MTP 训练融合**：DeepSeek-V3 等模型原生 MTP 头，推理时 MTP 即是免费的 draft 层，消除独立 draft 模型的额外开销。
5. **投机解码 + 推测 KV Cache**（Lookahead KV）：draft 阶段预填充部分 KV，减少 verify forward 的 cache miss。

## 6.12 面试题 / 扩展挑战

### 基础理解

**Q1**：为什么 modified rejection sampling 能保证输出分布等于 target 分布？能否用一句话直觉解释？

> 直觉：对每个 token 位置，接受率 $\min(1, q/p)$ 使得"草稿正确的概率"和"草稿错误但补救"合在一起，恰好构成一个接受 $q$ 分布的采样器——多余的 $p$ 被残差分布"消化"掉了。

**Q2**：为什么 batch size 增大后投机解码效果变差？从带宽/算术强度角度分析。

**Q3**：n-gram draft 的 `HAS_DRAFT_LOGITS=False` 对应什么数学假设？这在 rejection sampling 公式里如何体现？

> One-hot draft：$p(\hat{x}) = 1$，$\log p = 0$。Rejection kernel 里 `draft_log_prob = 0`，即 accept 准则退化为 `target_log_prob > log(u)`，即只要 target 有正概率就以该概率接受。

### 代码挑战（扩展 Demo 1）

**C1**：修改 Demo 1，实现 **greedy verification**（temperature=0 时 draft 必须与 target argmax 完全匹配）。比较 stochastic 和 greedy 模式下的 acceptance rate 差异。

**C2**：为 Demo 1 增加 **adaptive k**：如果上一轮 acceptance rate < 0.5，下一轮减小 k；如果 > 0.8，增大 k。观察自适应策略下平均 tokens/step 的变化。

**C3**（深度）：用 numpy 实现 Demo 2 的树形 rejection sampling（而不是逐路径独立采样），确保整棵树的输出分布严格等于 target——提示：树中每个节点的接受/拒绝决策需要考虑父节点已经接受的条件概率。

**C4**（实现挑战）：阅读 vLLM `rejection_sampler_utils.py` 的 `_rejection_kernel`，解释为何日志概率比较（`target_log_prob > tl.log(u) + draft_log_prob`）在数值上优于概率比较（`u < q/p`），并写出两者的 overflow/underflow 边界条件。
