## TL;DR

Chromium 的调度器是**一棵三层树**：最底层是 `base::MessagePump`（OS 事件桥），中间层是 `base::SequenceManager`（多优先级 TaskQueue 仲裁器），顶层是 `blink::MainThreadSchedulerImpl`（语义调度决策者，基于 UseCase/RAIL 模式动态改变 TaskQueue 优先级）。一个"事件循环轮次"= `MessagePump::Run` → `Delegate::DoWork` → `SequenceManager::SelectNextTask` → `MainThreadSchedulerImpl::ComputePriority` → 执行任务 → microtask checkpoint → (可能) 渲染帧。理解这棵树是读懂 Jank、INP、Back/Forward Cache 冻结等所有调度问题的前提。

## 一、设计考古：从 MessageLoop 到 SequenceManager

### 1.1 最初的 MessageLoop（2009 之前）

Chromium 最早（~2009）只有一个 `base::MessageLoop` 类，它直接持有一个优先队列和一个 OS 级等待句柄，在 `Run()` 里循环：投递的 `Task`（`base::Closure`）进单一队列，FIFO 执行。这个模型简单但有三个致命限制：

1. 无优先级——IPC 回调和低优先级定时器平等竞争；
2. 无跨线程安全的"只推一次"语义——多线程投递可能让线程意外唤醒；
3. 不可测试——虚拟时间、fake clock 无法注入。

### 1.2 SequenceManager 重构（2017–2019）

Chromium 团队在 2017 年立项，将 `MessageLoop` 拆解为：

- `base::MessagePump`：纯粹的 OS 事件等待抽象，不知道任务；
- `base::SequenceManager`：多优先级 TaskQueue 调度器，持有 `MessagePump` 并实现 `Delegate`；
- `base::TaskQueue`/`TaskRunner`：任务的投递句柄。

设计意图（见 chromium/src `base/task/sequence_manager/README.md`【真实源码 chromium@base/task/sequence_manager/README.md】）：

> "a set of prioritized FIFO task queues, which allows funneling multiple sequences of immediate and delayed tasks on a single underlying sequence."

核心数据结构变化：每个 `TaskQueueImpl` 维护四条内部队列——
- `immediate_incoming_queue`：任意线程投递即时任务的 inbox（有锁）；
- `immediate_work_queue`：主线程专属工作队列，由 `SequenceManager` 批量从 inbox swap 过来（lock-free 读）；
- `delayed_incoming_queue`：延时任务 inbox；
- `delayed_work_queue`：到期后从 `delayed_incoming_queue` 迁移过来。

**为什么 swap 而不是 copy？** 减少跨线程锁争用——只在 swap 时短暂加锁，避免每次 `SelectNextTask` 都持锁。【真实源码 chromium@base/task/sequence_manager/task_queue_impl.h】

### 1.3 Blink Scheduler 叠加层（2016–至今）

与 `SequenceManager` 并行，Blink 团队构建了 `MainThreadSchedulerImpl`，目的是**把浏览器的高层语义**（用户正在触摸屏幕、页面正在加载、渲染帧饥饿）转换成 `SequenceManager` 能理解的优先级整数。

关键设计文献：
- Chromium design doc "Renderer Scheduler" (2015, internal)，公开摘要可见 [`scheduler/README.md`](https://chromium.googlesource.com/chromium/src/+/main/third_party/blink/renderer/platform/scheduler/README.md)
- RAIL 模型博客（2015, Ilya Grigorik）：Response/Animation/Idle/Load 四阶段分别对应不同调度策略

## 三、核心机制源码精读

### 3.1 MessagePump：OS 与任务的桥接

【真实源码 chromium@base/message_loop/message_pump.h】

`MessagePump::Delegate` 接口核心：

```cpp
// 【真实源码 chromium@base/message_loop/message_pump.h】
class Delegate {
 public:
  // 执行一批即时/到期任务。返回 NextWorkInfo 告知 pump：
  //   .delayed_run_time = 下次需要唤醒的时刻（TimeTicks::Max() 表示无延时任务）
  //   .can_run_immediate_work = 是否还有即时任务
  virtual NextWorkInfo DoWork() = 0;

  // 在无任务时执行（idle 期）
  virtual bool DoIdleWork() = 0;

  // Pump 进入 sleep 前的 hook
  virtual void BeforeWait() = 0;

  // 每个工作单元的 RAII 跟踪器（用于 tracing + 嵌套检测）
  virtual ScopedDoWorkItem BeginWorkItem() = 0;
};
```

`MessagePumpDefault::Run()` 核心循环（简化，【真实源码 chromium@base/message_loop/message_pump_default.cc】）：

```cpp
// 【真实源码 chromium@base/message_loop/message_pump_default.cc】
void MessagePumpDefault::Run(Delegate* delegate) {
  for (;;) {
    // 1. 执行 native 平台事件（macOS: NSApp, Linux: epoll, Win: GetMessage）
    bool did_native_work = DoNativeWork();

    // 2. 执行 Chromium 任务（委托给 SequenceManager）
    Delegate::NextWorkInfo next_work_info = delegate->DoWork();
    if (next_work_info.is_immediate())
      continue;  // 还有即时任务，不等待

    // 3. Idle 工作（RequestIdleCallback 任务在此执行）
    if (delegate->DoIdleWork())
      continue;

    // 4. 等待，直到下次任务到期或 ScheduleWork() 信号
    delegate->BeforeWait();
    event_.TimedWait(next_work_info.delayed_run_time);
    // ScheduleWork() 通过 event_.Signal() 打断等待
  }
}
```

**注意**：macOS 有专门的 `MessagePumpCFRunLoop`，嵌套 CFRunLoop；Windows 有 `MessagePumpForUI` 用 `GetMessage` 抽走 Win32 消息。

### 3.2 SequenceManager：任务选择与优先级

【真实源码 chromium@base/task/sequence_manager/sequence_manager_impl.cc】

`SelectNextTaskImpl()` 伪代码（基于源码注解重构，非逐字）：

```cpp
// 【示意，非逐字，基于 chromium@base/task/sequence_manager/sequence_manager_impl.cc】
Optional<SelectedTask> SequenceManagerImpl::SelectNextTaskImpl() {
  // Step 1: 将各 TaskQueue 的 immediate_incoming_queue swap 到 immediate_work_queue
  ReloadEmptyWorkQueues();

  // Step 2: 把到期延时任务从 delayed_incoming_queue 迁移到 delayed_work_queue
  MoveReadyDelayedTasksToWorkQueues(LazyNow(&main_thread_clock_));

  // Step 3: 优先级选择
  while (true) {
    WorkQueue* work_queue = nullptr;
    bool should_run = selector_.SelectWorkQueueToService(&work_queue);
    if (!should_run) return std::nullopt;  // 没有可运行任务

    // Step 4: 已取消任务跳过
    const Task& task = work_queue->GetFrontTask();
    if (task.task.IsCancelled()) {
      work_queue->PopTaskFromFront();
      continue;
    }

    // Step 5: 不可嵌套任务在嵌套 RunLoop 中延迟
    if (nesting_depth_ > 0 && !task.nestable) {
      non_nestable_task_queue_.push_back(work_queue->TakeTaskFromFront());
      continue;
    }

    // Step 6: 返回选中的任务
    return SelectedTask{task, work_queue->task_queue()->GetTaskType(),
                        work_queue->task_queue()->GetQueuePriority()};
  }
}
```

**优先级选择算法**（【真实源码 chromium@base/task/sequence_manager/task_queue_selector.cc】）：

```cpp
// 【真实源码 chromium@base/task/sequence_manager/task_queue_selector.cc】
// 用 bitfield 追踪哪些优先级有可运行任务：O(1) 找最高优先级
TaskQueue::QueuePriority ActivePriorityTracker::HighestActivePriority() const {
  return static_cast<TaskQueue::QueuePriority>(
      std::countr_zero(active_priorities_));
  // active_priorities_ 是一个 uint64_t bitset，低 bit = 高优先级
  // countr_zero 找最低有效位，即优先级数字最小（最高）的那个
}
```

每个优先级对应一个 `WorkQueueSets`（min-heap，按 `EnqueueOrder` 排序）。`WorkQueueSets::GetOldestQueueAndTaskOrderInSet()` 在同优先级内挑 FIFO 最老的任务，O(1)。【真实源码 chromium@base/task/sequence_manager/work_queue_sets.h】

**饥饿防护**：`immediate_starvation_count_` 记录"延时任务连续被选中但同优先级有即时任务"的次数；上层策略可据此强制选即时任务。【真实源码 chromium@base/task/sequence_manager/task_queue_selector.cc】

### 3.3 MainThreadSchedulerImpl：语义调度决策

这是理解 Blink 调度行为的核心层。

#### UseCase 状态机

```cpp
// 【真实源码 chromium@third_party/blink/renderer/platform/scheduler/main_thread/use_case.h】
enum class UseCase {
  kNone,                      // 默认空闲状态
  kEarlyLoading,              // FCP 之前的加载期（更激进的 loading 优先）
  kLoading,                   // FCP 之后的加载期

  kCompositorGesture,         // 合成器线程驱动的 scroll/pinch

  kMainThreadGesture,         // 主线程驱动的 scroll

  kDiscreteInputResponse,     // 点击/按键后等待第一帧

```

#### UpdatePolicy / UpdatePolicyLocked

【真实源码 chromium@third_party/blink/renderer/platform/scheduler/main_thread/main_thread_scheduler_impl.cc】

```cpp
// 【示意，非逐字，基于 chromium@...main_thread_scheduler_impl.cc】
void MainThreadSchedulerImpl::UpdatePolicyLocked(UpdateType update_type) {
  // 1. 计算当前 UseCase（综合 gesture 历史、touch 状态、加载状态）
  auto [use_case, expected_duration] = ComputeCurrentUseCase();
  main_thread_only().current_use_case = use_case;

  // 2. RAIL 模式映射
  RAILMode rail_mode = ComputeCurrentRAILMode(use_case);

  // 3. 根据 UseCase 计算 Compositor 队列优先级
  //    kTouchstart → compositor 升为 kVeryHighPriority（阻断其他任务）
  //    kCompositorGesture → compositor 升为 kHighPriority
  //    kDiscreteInputResponse → compositor 升为 kHighPriority
  TaskQueue::QueuePriority compositor_priority =
      ComputeCompositorPriority(use_case);

  // 4. 检查渲染饥饿（RenderingPrioritizationState）
  //    若距上次帧 > 100ms → kRenderingStarved → 强制渲染队列到 kHighPriority
  //    若 render-blocking 任务 > 500ms → kRenderingStarvedByRenderBlocking
  UpdateRenderingPrioritizationState();

  // 5. 更新所有 TaskQueue 的优先级
  UpdateTaskQueueState(compositor_queue_.get(), compositor_priority);
  UpdateTaskQueueState(v8_task_queue_.get(), ...);
  // ...
}
```

`ComputeCurrentUseCase()` 关键逻辑（【真实源码 chromium@...main_thread_scheduler_impl.cc, ~line 1320】）：

```cpp
// 【示意，非逐字，基于 chromium@...main_thread_scheduler_impl.cc】
UseCase MainThreadSchedulerImpl::ComputeCurrentUseCase() const {
  // touchstart 期间：只允许 input 处理，最高优先级
  if (any_thread_.awaiting_touch_start_response)
    return UseCase::kTouchstart;

  // 连续手势识别
  if (any_thread_.have_seen_a_gesture) {
    if (any_thread_.gesture_is_compositor_driven)
      return UseCase::kCompositorGesture;  // 合成器主导 scroll
    // ... 其他手势类型
  }

  // 离散输入（click/key）后等渲染
  if (any_thread_.waiting_for_discrete_input_response)
    return UseCase::kDiscreteInputResponse;

  // 加载阶段
  if (!main_thread_only().have_seen_a_beginmainframe)
    return UseCase::kEarlyLoading;
  if (main_thread_only().is_loading)
    return UseCase::kLoading;

  return UseCase::kNone;
}
```

#### RAIL 模式

RAIL = Response/Animation/Idle/Load，每种模式对应不同的调度策略：

| RAIL 模式 | 对应 UseCase | 关键策略 |
|-----------|-------------|---------|
| kResponse | kTouchstart, kDiscreteInputResponse | Compositor 最高优先，阻断长 JS |
| kAnimation | kCompositorGesture, kSynchronizedGesture | 保帧，限制主线程任务时长 |
| kIdle | kNone（无动画无输入） | 开放 IdleCallback 窗口 |
| kLoad | kEarlyLoading, kLoading | 提升 Loading 队列优先级 |

### 3.5 FrameSchedulerImpl：帧级节流与冻结

每个 `LocalFrame` 拥有一个 `FrameSchedulerImpl`，负责：

1. **Wake-up 节流**：后台 tab 帧级任务（`kFrameThrottleable`）由 `TaskQueueThrottler` 控制——插入 fence，`GetNextAllowedWakeUp()` 把下次唤醒时间推移到预算允许的最早时刻。【真实源码 chromium@third_party/blink/renderer/platform/scheduler/common/throttling/task_queue_throttler.h】

2. **CPU 预算池**：`CPUTimeBudgetPool` 实现"后台 tab 只能用 X% CPU"。任务执行后调 `RecordTaskRunTime()` 扣除预算；预算耗尽时 `GetNextAllowedRunTime()` 返回未来某时刻，让节流器推迟唤醒。【真实源码 chromium@third_party/blink/renderer/platform/scheduler/common/throttling/cpu_time_budget_pool.h】

3. **BFCache 冻结**：当页面进入 Back/Forward Cache 时，`can_be_frozen=true` 的队列被加 fence，直到页面恢复（`can_run_in_bfcache=true` 的特殊队列仍可执行）。

4. **任务时间上报**：对 Browser 进程的任务计时，超过阈值时通过 Mojo 上报（跨进程 UMA）。

### 3.6 rAF 与渲染管线接入

`requestAnimationFrame` 的执行位置在主线程事件循环中是**明确夹在 task 结束与下一帧 commit 之间**，不是一个普通 TaskQueue 的任务。

```
[Task 执行完毕]
    ↓
microtask checkpoint（V8 microtask queue 清空）
    ↓
[cc::BeginMainFrame 到达]  ← 由合成器线程通过 IPC 触发
    ↓
ScriptedAnimationController::ServiceScriptedAnimations()
    ├── ExecuteFrameCallbacks()    ← rAF 回调
    ├── ExecuteVideoFrameCallbacks()
    └── DispatchEvents()           ← resize/scroll 等
    ↓
Layout / Style / Paint
    ↓
Compositor Commit
```

`ScriptedAnimationController::ScheduleAnimationIfNeeded()`【真实源码 chromium@third_party/blink/renderer/core/dom/scripted_animation_controller.cc】：

```cpp
// 【真实源码 chromium@...scripted_animation_controller.cc】
void ScriptedAnimationController::ScheduleAnimationIfNeeded(
    cc::BeginMainFrameReason reason) {
  if (!GetExecutionContext() || GetExecutionContext()->IsContextPaused())
    return;
  auto* frame = GetWindow()->GetFrame();
  if (!frame) return;
  if (HasScheduledFrameTasks()) {
    // 通知合成器：主线程需要下一帧
    frame->View()->ScheduleAnimation(reason);
  }
}
```

关键：rAF **不在** `kCompositor` TaskQueue，而是在 `BeginMainFrame` 合成器 IPC 到达后，在主线程的 commit 阶段同步执行。这就是为什么 rAF 回调中做 DOM 操作不会 Layout Thrashing（一次 rAF 内只有一次 layout），而在 rAF 外反复读写 offsetWidth 就会（每次读强制 layout）。

### 3.7 IdleScheduler：空闲任务机制

`IdleHelper` 管理 `requestIdleCallback` 的执行窗口：

1. `DidCommitFrameToCompositor()` 触发 → 启动一个短 idle 期（估算距下帧的剩余时间）；
2. `BeginMainFrameNotExpectedUntil(t)` 触发 → 启动长 idle 期（到 t 为止）；
3. `IdleTimeEstimator` 用滚动历史估算下帧 compositor 任务耗时，给出 `GetExpectedIdleDuration()`。【真实源码 chromium@third_party/blink/renderer/platform/scheduler/main_thread/idle_time_estimator.h】

IdleCallback 被放入 `kIdle` TaskQueue（BestEffort 优先级），只有在 idle 窗口打开且没有更高优先级任务时才执行，且 deadline 由 `IdleHelper` 动态计算后注入回调。

### 3.8 Prioritized Task Scheduling（scheduler.postTask()）

W3C 标准 API，Chromium 实现在 `DOMScheduler`/`DOMTaskSignal`。

```javascript
// 三个固定优先级
await scheduler.postTask(fn, { priority: 'user-blocking' }); // input 响应
await scheduler.postTask(fn, { priority: 'user-visible' });  // 默认，可见更新
await scheduler.postTask(fn, { priority: 'background' });    // 不阻塞渲染

// 动态优先级（TaskController）
const controller = new TaskController({ priority: 'user-visible' });
scheduler.postTask(fn, { signal: controller.signal });
controller.setPriority('background');  // 运行时降级
```

内部映射【真实源码 chromium@third_party/blink/renderer/core/scheduler/dom_scheduler.h】：
- 每个优先级对应一个 `kWebScheduling` 类型的 `TaskQueue`；
- `signal_to_task_queue_map_` 用 WeakRef 跟踪动态优先级信号 → TaskQueue 映射；
- `continuation queue`：Promise.then 续体走独立的 continuation 队列（比普通 postTask 高优先级）。

## 四、Electron 的事件循环整合

Electron = Chromium content layer + Node.js + Electron API。最关键的挑战是：**Chromium 的 `base::RunLoop` 和 Node.js 的 libuv 是两个独立的事件循环**，必须共存在同一个线程（Browser 进程主线程）。

### 4.1 整合策略：嵌入式 uv 轮询

【真实源码 electron@shell/common/node_bindings.cc】

```
Chromium MessagePump::Run() 主循环（持续运行）
         │
         │ 每次 Chromium 任务执行完，有闲置时间时
         ▼
   NodeBindings::UvRunOnce()
         │
         ├── uv_run(loop, UV_RUN_NOWAIT)  ← 非阻塞跑一次 libuv
         │     处理 fs/net/timer 等 Node 事件
         │
         └── 发信号给 embed_thread
                   │
                   ▼
          EmbedThreadRunner（独立线程）
                   │
                   ├── PollEvents()   ← 阻塞等 libuv fd 就绪
                   │
                   └── 就绪时 task_runner_->PostTask(&UvRunOnce)
                               │
                               ▼
                       重新回到 Chromium 主线程执行
```

**关键代码路径**【真实源码 electron@shell/common/node_bindings.cc】：

```cpp
// 【真实源码 electron@shell/common/node_bindings.cc】
void NodeBindings::StartPolling() {
  // 捕获 Chromium 主线程的 TaskRunner
  task_runner_ = base::SingleThreadTaskRunner::GetCurrentDefault();
  // 启动 embed 线程（持续 poll libuv）
  embed_thread_ = base::Thread("node_uv_embed");
  embed_thread_.Start();
  embed_thread_.task_runner()->PostTask(
      FROM_HERE, base::BindRepeating(&NodeBindings::EmbedThreadRunner, this));
}

// embed 线程：等 libuv 有事件 → 唤醒主线程
void NodeBindings::EmbedThreadRunner() {
  while (true) {
    uv_sem_wait(&embed_sem_);   // 等主线程释放信号量
    PollEvents();               // 阻塞等 libuv I/O 就绪
    // libuv 有事件了，投递任务到 Chromium 主线程
    task_runner_->PostTask(FROM_HERE,
        base::BindOnce(&NodeBindings::UvRunOnce, base::Unretained(this)));
  }
}

// 主线程：非阻塞跑 libuv，然后释放信号量
void NodeBindings::UvRunOnce() {
  node::Environment* env = uv_env();
  {
    v8::HandleScope handle_scope(env->isolate());
    uv_run(env->event_loop(), UV_RUN_NOWAIT);  // 非阻塞跑一轮
  }
  uv_sem_post(&embed_sem_);  // 允许 embed 线程继续 poll
}
```

**为什么是信号量而不是条件变量？** 防止 ABA 问题：embed 线程 poll 时主线程可能多次调用 `UvRunOnce`，信号量保证每次 poll 周期只对应一次 Chromium 任务投递。

### 4.2 Electron 对 Chromium 调度器的定制

Electron 基本**不修改** `MainThreadSchedulerImpl` 或 `SequenceManager`，而是利用 Chromium 的标准 TaskRunner 接口：

- Browser 进程主线程 = Chromium UI 线程 = `base::RunLoop`
- `PreMainMessageLoopRun()` 中调用 `node_bindings_->StartPolling()` 挂入 Node 轮询
- 微任务：`js_env_->CreateMicrotasksRunner()` 注册 V8 微任务执行器，在 Chromium 任务完成后触发 microtask checkpoint

**Electron 主进程的优先级**：Node.js 回调（I/O 事件、timer）通过 `UvRunOnce → PostTask` 投递到 Chromium 主线程的**默认 TaskQueue**（`kDefault`，`kRegular` 优先级）。这意味着 Node.js I/O 回调的优先级低于 `kInput`/`kCompositor`，但高于 `kIdle`/`kBestEffort`——符合预期。

## 五、方案对比

### 5.1 任务投递方式对比

| 方式 | TaskQueue 类型 | 优先级 | 典型延迟 | 适用场景 |
|------|--------------|--------|---------|---------|
| `setTimeout(0)` | kFrameThrottleable | kRegular | 1–4ms（前台）/1min（后台激进节流） | 低优先级延迟工作 |
| `queueMicrotask` | 不走 TaskQueue | 最高（task 末尾） | 0（当前 task 结束即执行） | Promise 链、轻量续体 |
| `requestAnimationFrame` | BeginMainFrame IPC | 帧同步 | 下一帧（~16ms@60fps） | 动画、DOM 读写批处理 |
| `requestIdleCallback` | kIdle | kBestEffort | 不确定（帧间空隙） | 分析、预取，允许 deadline 内放弃 |
| `scheduler.postTask({priority:'user-blocking'})` | kWebScheduling | kInput 级 | 极低 | 输入响应类更新 |
| `scheduler.postTask({priority:'background'})` | kWebScheduling | kBestEffort | 较高 | 不阻塞渲染的统计/日志 |
| `MessageChannel.postMessage` | kDefault | kRegular | ~0（微任务后） | 比 setTimeout(0) 更快的"宏任务" |

### 5.2 后台节流级别对比

| 场景 | 节流类型 | 效果 |
|------|---------|------|
| 前台 tab | 无节流 | 正常执行 |
| 后台 tab（标准） | wake-up 节流 | setTimeout 最小间隔 1s |
| 后台 tab（激进，30s 后） | 激进 wake-up 节流 | 每分钟才允许一次 wake-up |
| 隐藏页 + Intensive Throttling | CPU 预算池 | 最多 1% CPU 时间 |
| BFCache | 完全冻结 | 所有 freezable 队列停止 |
| WebView（Android） | 可 Pause | `can_be_paused_for_android_webview` |

### 5.3 Electron vs 浏览器渲染进程调度差异

| 维度 | Chromium Renderer | Electron 主进程 |
|------|-----------------|----------------|
| 主线程类型 | Blink 渲染主线程 | Chromium Browser UI 线程 |
| 核心调度器 | MainThreadSchedulerImpl | 标准 base::SequenceManager |
| 额外事件源 | CC BeginMainFrame | Node.js libuv（通过 embed thread） |
| 微任务执行器 | V8 MicrotaskQueue（Blink 管理） | V8 MicrotaskQueue（Electron `MicrotasksRunner`） |
| rAF 支持 | 完整（ScriptedAnimationController） | 不支持（无渲染帧概念） |
| 节流/BFCache | 完整 FrameScheduler 体系 | 无（无 Frame 概念） |

## 六、失败模式、生产真坑与根因分析

### 坑 1：Long Task 导致 INP 劣化

**现象**：页面 click 响应延迟 > 200ms，INP 指标差。  
**根因**：`kDiscreteInputResponse` UseCase 触发后，主线程有长于 16ms 的 JS 任务（数组排序、JSON 序列化等），`kCompositor` 队列虽已被提升优先级，但**任务一旦开始执行就不会被抢占**（Chromium 无内核级抢占，只有协作式让出）。  
**处方**：
1. 用 `scheduler.postTask({priority:'user-blocking'})` 把真正需要即时响应的逻辑单独投递，其余降为 `background`；
2. 用 `isInputPending()`（Chrome 87+）在长循环中主动检测 input，提前 yield：
   ```javascript
   while (tasks.length > 0) {
     if (navigator.scheduling?.isInputPending()) {
       await scheduler.yield(); // 让出给 input 处理
     }
     process(tasks.shift());
   }
   ```

### 坑 2：Layout Thrashing（强制同步 Layout）

**现象**：滚动帧耗时 > 16ms，Performance 面板显示大量紫色 Layout。  
**根因**：在一个 JS task（非 rAF）中交替读 `offsetWidth`（触发 layout）和写 `style`（dirty layout），每次读都强制同步 layout，O(n²) 开销。  
**根本原因（调度层）**：读操作不在 BeginMainFrame 之后、写操作不在 rAF 回调内，无法享受 Blink 的布局批处理。  
**处方**：所有 DOM 写在 rAF 回调开头批量执行，读操作收到 rAF 回调之前完成（或用 ResizeObserver/IntersectionObserver 异步观察）。

### 坑 3：setTimeout 后台节流"失联"

**现象**：后台 tab 中用 `setTimeout(ping, 30000)` 实现心跳，30s 后发现实际延迟 60s~120s。  
**根因**：`kFrameThrottleable` 队列在 tab 后台 30s 后进入激进节流，实际 wake-up 间隔变成 60s（Intensive Throttling），节流由 `CPUTimeBudgetPool` + `TaskQueueThrottler` 合力执行。  
**处方**：改用 Service Worker + Push API；或如果必须用定时器，改用 `kFrameUnpausable` 队列（即不设 `can_be_intensively_throttled=true`）——但不能在渲染进程随意创建，需要 Blink 内部 API。

### 坑 4：Electron 中 Node.js 回调阻塞 Chromium UI

**现象**：Electron 主进程中 `fs.readFileSync` 或大量同步 crypto 操作导致窗口冻结。  
**根因**：`UvRunOnce` 在 Chromium 主线程执行 `uv_run(UV_RUN_NOWAIT)`，如果 libuv 的 callback 里有阻塞调用，直接阻塞整个 `MessagePump::Run()` 的当前 task，后续 Chromium 渲染/IPC 任务无法被选中。  
**处方**：Node.js 所有 I/O 用异步 API；CPU 密集任务用 `worker_threads`（独立线程，有自己的 libuv loop，不占主线程）。

### 坑 5：BFCache 进出时 Timer 漂移

**现象**：页面从 BFCache 恢复后，`setInterval` 回调提前或延后几百毫秒。  
**根因**：BFCache 冻结时 `kFrameThrottleable` 队列全部加 fence（停止），恢复时 fence 解除，但 `DelayedWakeUpQueue` 的 wake-up time 是基于冻结前的 wall clock 计算，恢复后所有积压的到期任务会**集中 burst 执行**。  
**处方**：监听 `pageshow` event（`persisted=true`），在 BFCache 恢复时重置 timer；或使用 `requestAnimationFrame` 代替 `setInterval` 做动画（rAF 天然帧同步）。

### Demo 3：用 CDP 编程观察 Task 优先级变化（可跑代码）

**完整代码**：`demo-cdp-scheduler.mjs`

```javascript
// demo-cdp-scheduler.mjs
// 依赖：Node.js 18+，chrome 用 --remote-debugging-port=9222 启动
// 运行：node demo-cdp-scheduler.mjs

import WebSocket from 'ws';  // npm install ws

const CDP_URL = 'http://localhost:9222/json';

async function main() {
  // 1. 获取可用的 target
  const targets = await fetch(CDP_URL).then(r => r.json());
  const page = targets.find(t => t.type === 'page');
  if (!page) throw new Error('No page found. Open a page in Chrome first.');

  const ws = new WebSocket(page.webSocketDebuggerUrl);
  let msgId = 0;

  const send = (method, params = {}) => new Promise((resolve) => {
    const id = ++msgId;
    ws.send(JSON.stringify({ id, method, params }));
    const handler = (data) => {
      const msg = JSON.parse(data);
      if (msg.id === id) {
        ws.off('message', handler);
        resolve(msg.result);
      }
    };
    ws.on('message', handler);
  });

  await new Promise(r => ws.once('open', r));

  // 2. 启用 Performance 域，监听 Long Task
  await send('Performance.enable', { timeDomain: 'timeTicks' });
  await send('Target.setAutoAttach', { autoAttach: true, waitForDebuggerOnStart: false });

  // 3. 注入一段 JS：制造 Long Task + rAF + scheduler.postTask
  await send('Runtime.evaluate', {
    expression: `
      // Long Task（阻塞主线程 80ms）
      const start = Date.now();
      while (Date.now() - start < 80) {}
      console.log('[LongTask] done at', performance.now().toFixed(2));

      // rAF：下一帧执行
      requestAnimationFrame(() => {
        console.log('[rAF] fired at', performance.now().toFixed(2));
      });

      // scheduler.postTask：user-blocking 级别
      if (window.scheduler) {
        scheduler.postTask(() => {
          console.log('[postTask user-blocking] at', performance.now().toFixed(2));
        }, { priority: 'user-blocking' });

        scheduler.postTask(() => {
          console.log('[postTask background] at', performance.now().toFixed(2));
        }, { priority: 'background' });
      }

      // microtask
      queueMicrotask(() => {
        console.log('[microtask] at', performance.now().toFixed(2));
      });

      // setTimeout(0)
      setTimeout(() => {
        console.log('[setTimeout(0)] at', performance.now().toFixed(2));
      }, 0);
    `,
    awaitPromise: false
  });

  // 4. 收集 console 输出（观察执行顺序）
  await send('Runtime.enable');
  ws.on('message', (data) => {
    const msg = JSON.parse(data);
    if (msg.method === 'Runtime.consoleAPICalled') {
      const args = msg.params.args.map(a => a.value).join(' ');
      console.log('[CDP LOG]', args);
    }
  });

  await new Promise(r => setTimeout(r, 2000));
  ws.close();
}

main().catch(console.error);
```

**预期主进程输出**（Node.js 侧）：

```
[main] app ready
[main] before blocking
[main] after non-blocking setup, elapsed: 0 ms
[main] setImmediate fired at ...     ← libuv check phase，比 timer 早
[main] Node timer fired at ...       ← libuv timers phase

```

**预期渲染进程输出**（在 DevTools Console）：

```
[renderer] script start
[renderer] script end
[renderer] microtask         ← task 末尾 microtask checkpoint
[renderer] user-blocking     ← kWebScheduling 高优先级
[renderer] setTimeout(0)     ← 普通宏任务
[renderer] rAF               ← 下一帧
```

**与源码呼应**：
- 主进程无 `MainThreadSchedulerImpl`，无 rAF 概念，`setImmediate` 走 libuv check phase → `UvRunOnce` → Chromium 主线程 task；
- 渲染进程有完整 Blink 调度器，优先级分层明确。
