## 进程会死，这不是 edge case

前面每一个 stage 都默默假设了一件事：进程能活到把循环跑完。真实的 agent 没这福气。它对着会抖的外部工具跑几分钟到几小时，跑到一半 box 被 OOM kill（内存超限被系统杀），一次部署把 pod 滚掉，或者有人按了 Ctrl-C。如果"agent 到目前为止做了什么"这份状态只活在进程内存里，那么 6 步里第 4 步崩溃，意味着从第 0 步重来——每个工具重跑一遍，每个 token 重付一遍，更糟的是，每个**副作用**（side effect，比如扣款、发邮件、插一行数据库）重新触发一遍。

你现在用的 Claude Code 就是个活体参考：会话能 `/compact`（把长对话压缩成摘要继续）之后接着干，workflow 能 resume，我这次给你写这一章本身就是上一轮被压缩后续作的。这些能力背后是同一个机制——**进程的状态必须在进程外有一份可恢复的拷贝**。这一章我们把它从头写出来：`examples/agent-from-scratch/src/stage15-durable.ts`，全程离线、确定性，状态 checkpoint 到真实的临时文件，"崩溃"是在最危险的那个点注入的一个 throw——真实的 SIGKILL/OOM 落在同一个位置，下一个进程读 checkpoint 接着跑。

先说结论，免得你以为这章只讲 checkpoint：**给对话做 checkpoint 是不够的**。让一次重试变安全的那份状态（幂等账本）本身也是持久化状态，漏掉它，at-least-once（至少执行一次）执行会悄无声息地变成双重副作用。整章就在讲这一个 trade-off。

## 状态就是可序列化的 messages

第 1 章已经把这条钉死了：循环的全部状态必须是可序列化的显式数据，不能藏在调用栈或闭包里，`runAgent` 的状态其实只有一样东西——`messages` 数组。可恢复性就是这条约束的直接红利。如果状态是显式数据，把它写到磁盘、再从磁盘读回来重建一个进程，是平凡的；如果状态藏在调用栈深处的局部变量和半成品 Promise 里，没有任何持久化方案救得了你。

stage15 的 checkpoint 结构就两个字段（`examples/agent-from-scratch/src/stage15-durable.ts`）：

```ts
interface AgentState {
  // 至今为止的对话。这就是 resume 便宜的原因：我们把它重发给模型，
  // 而不是重跑产生它的那些回合。
  messages: Message[];
  // 多少个循环 step 已经持久化完成。resume 游标：新进程信它来定位续点。
  // 故意写在 step 工作"之后"，所以它会比真实副作用滞后一个崩溃窗口。
  completedSteps: number;
}
```

`messages` 是续跑便宜的根本原因——resume 时我们把整段对话重发给模型，而不是重跑产生它的那些回合（LLM 是无状态的，每轮 harness 都要重发整个 transcript，这是第 1 章和第 5 章的老朋友了）。`completedSteps` 是续跑游标。`saveCheckpoint` 每步把整个 blob 写一遍，cold start 还是 resume 完全由"checkpoint 文件存不存在"决定：

```ts
const restored = loadCheckpoint(checkpointPath);
const state: AgentState =
  restored ?? { messages: [{ role: 'user', content: userInput }], completedSteps: 0 };
// 循环计数器从磁盘恢复，所以 resume 的进程从中途接上。
for (let step = state.completedSteps; step < maxSteps; step++) { ... }
```

注意 `loadCheckpoint` 在没有文件时返回 `null`，那个 `null` 就是"cold start"信号，循环据此分叉。这是个有意的设计：用文件存在与否做唯一判据，而不是塞一个 `isFirstRun` 布尔到别处——状态机里多一个独立布尔就多一组非法组合（参第 12 章对状态收敛的讨论）。

### checkpoint 的原子性缺口

代码里有一句诚实的注释值得拎出来——`saveCheckpoint` 每步 `writeFileSync` 整个 blob，但生产级 checkpoint 应该是 write-to-temp + rename（POSIX 上 rename 原子），这样崩在写一半时不会留下半个文件。stage15 没做这步，但它选择**fail loud**：半个 JSON 在 resume 时会让 `JSON.parse` 直接抛错，而不是 resume-from-garbage。这是个可以接受的取舍——恢复时炸掉比恢复出一个静默错误的状态强得多。这条对应运行时规范里"失败行为显式三选"中的 fail-fast：数据正确性优先时，不能假装成功。

## 危险窗口：副作用先于 checkpoint

现在到了这章的核心。看这个循环里 step 的执行顺序：先调工具（副作用 fire）→ 把 tool_result 推进 messages → **然后才**推进 `completedSteps` 并写 checkpoint。

```ts
state.messages.push({ role: 'user', content: results });

// 崩溃窗口：副作用已经触发（且 ledger durable 时记录已落盘），
// 但 step checkpoint 还没写。这个 gap 存在于每一个真实系统里。
if (step === crashAfterEffectAtStep) {
  throw new SimulatedCrash(state.completedSteps);
}

// step checkpoint：到这里 completedSteps 才推进并持久化。
// 在这一行之前崩 = resume 会重跑 step——当且仅当 ledger 去重了它才安全。
state.completedSteps = step + 1;
saveCheckpoint(checkpointPath, state);
```

这里有个根本性的物理事实：**你无法把"做副作用"和"记录副作用"跨进程边界变成一个原子动作**。扣款发生在外部支付系统，checkpoint 写在你的磁盘，这两个写永远存在一个时间差。崩在这个差里——副作用已经 fire，记录它的 checkpoint 还没落地——resume 就会重跑这个 step。

这正是 stage15 注入崩溃的位置（`crashAfterEffectAtStep: 1`，在第 2 笔订单扣款 fire 之后、它的 checkpoint 写入之前）。这个崩溃窗口不是 demo 编出来的戏剧性，它是每个真实系统都有的。

所以 checkpoint-after-step 给你的保证是：**in-flight step 的 at-least-once 执行**。你拿不到免费的 exactly-once（恰好一次）。对一个只读操作，重跑无害；对"扣信用卡"，重跑就是双重扣款。怎么让 at-least-once 变安全？这是下一节的事——但先记住：checkpoint 本身解决不了这个问题，它就是制造这个问题的东西。

## 幂等：让重跑安全的那份状态

幂等键（idempotency key，让重复请求结果一致的稳定标识）是这里的解法。工具在副作用 fire 的**那一刻**就记下"键 K 已完成 → 这是上次的结果"，于是重跑返回旧结果、**不重复**副作用。stage15 的 `chargeOnce` 是全章最该读懂的函数：

```ts
function chargeOnce(store: LedgerStore, key: string, amountUsd: number): string {
  const prior = store.ledger[key];
  if (prior !== undefined) {
    // 重试路径：副作用已经发生过。返回与上次相同的结果——
    // 调用方分不出这是重试还是原始调用，这正是目的。不二次扣款。
    return `${prior} (idempotent replay, no new charge)`;
  }
  chargeFireCount += 1;
  const result = `charged $${amountUsd} (charge #${chargeFireCount})`;
  store.ledger[key] = result;
  if (store.ledgerPath) {
    writeFileSync(store.ledgerPath, JSON.stringify(store.ledger, null, 2), 'utf8');
  }
  return result;
}
```

两个设计点决定它对不对：

第一，**键是操作的身份，不是参数**。stage15 的 mock policy 给每笔订单派生一个稳定键 `charge:${order.id}`（如 `charge:A-1002`），所以"重跑 step 1"和"原本的 step 1"用同一个键，账本认出它们是"同一笔扣款被重试"，哪怕跨进程重启。如果键是随机生成或从参数 hash 来的，重跑就成了新键，去重失效。

第二——也是这章最不直觉的设计——**账本在 fire-time 就独立持久化，不等 step checkpoint**。看 `chargeOnce` 里 `writeFileSync` 的位置：在 `chargeFireCount += 1` 之后立刻写。`AgentState` 里**故意不包含**这个账本。为什么要把两份持久化状态拆成两条不同的写入时机？因为我们怕的崩溃窗口恰好是"副作用与 step checkpoint 之间"。如果账本等到 step checkpoint 才一起写，它就会在最需要它的那一刻丢失。

把它跑起来验证幂等的隔离行为，输出是（玩具数据，单进程内存账本）：

```
IDEMPOTENCY CHECK (same key twice, fresh in-memory ledger):
  1st call: charged $30 (charge #1)
  2nd call: charged $30 (charge #1) (idempotent replay, no new charge)
  real charges fired: 1 (expected 1: retry deduped)
```

同一个键调两次，`chargeFireCount` 只涨到 1。这是 §EFFECT 里那个进程全局计数器的用途：它数的是**真实 fire 次数**，不是调用次数，所以一次重复扣款会表现为一个错误的数字——这是断言"恰好一次"唯一诚实的方式。注意工具 spec 还做了一道防御：`charge_card` 没传 `idempotency_key` 时直接抛错，而不是无键裸扣——无法去重就拒绝执行，比扣错强。

## 完整 demo：崩溃后续跑（Scenario A）

把 checkpoint 和幂等合起来跑一遍崩溃-续跑。process #1 跑到 step 1 扣款 fire 后崩，process #2 读 checkpoint 续跑。Scenario A 的账本是 durable 的：

```
SCENARIO A (ledger durable) — process #1 (crashes mid-run):
  [cold]   no checkpoint; starting fresh
  [ckpt]   step 1 durable: tool_use charge_card | charges fired so far=1
  [crash]  process died; simulated crash; last durable step = 1
SCENARIO A (ledger durable) — process #2 (resumes from checkpoint):
  [resume] checkpoint at step 1; ledger has 2 prior charge(s) (durable)
  [ckpt]   step 2 durable: tool_use charge_card | charges fired so far=2
  [ckpt]   step 3 durable: tool_use charge_card | charges fired so far=3
  [ckpt]   step 4 durable: text answer | charges fired so far=3
  RESULT A:
    answer            : Done. 3 orders charged.
    resumed from step : 1 (step 1 re-ran; earlier steps did not)
    steps in proc #2  : 3 (of 4 total)
    real charges fired: 3 (expected 3: re-run of step 1 was deduped)
```

读这串数字的关键在最后一行：`real charges fired: 3`。step 1 在崩溃前 fire 过一次扣款（process #1 的 `charges fired so far=1`），它的 checkpoint 没落地，所以 process #2 从 step 1 重跑——但 `[resume]` 那行显示账本里已有 2 笔 durable 记录，重跑的扣款命中账本被去重。3 笔订单各扣一次，step 1 跑了两次但只 fire 一次。这就是 at-least-once 执行 + 幂等去重合起来给出的、最接近 exactly-once 的东西。

（一个诚实标注：这些是玩具数据。三笔订单、四个 step、固定脚本的 mock policy 替代了真实 LLM——`generateAtStep` 用循环 `step` 而非 MockLLM 的回合计数器来定 policy，因为 resume 会 new 一个全新的 MockLLM、它的计数器归零了，能信的游标只有从磁盘恢复的 `step`。`charge` 是个进程内计数器在替身一个不可撤销的外部动作。token/金额都不是真实测量。）

## 失败模式：checkpoint 全了，但漂移了（Scenario B）

现在删掉一块状态看会怎样。Scenario B 一切照旧——同样的崩溃点、同样的 resume、同样的对话 checkpoint——只有一处不同：账本只活在内存里（`ledgerPath=null`，代码里这是 §DRIFT 注入点；生产代码没有这个开关，永远持久化）。

```
SCENARIO B (ledger NOT durable) — process #2 (resumes from checkpoint):
  [resume] checkpoint at step 1; ledger has 0 prior charge(s) (NOT durable — drift)
  [ckpt]   step 2 durable: tool_use charge_card | charges fired so far=3
  [ckpt]   step 3 durable: tool_use charge_card | charges fired so far=4
  [ckpt]   step 4 durable: text answer | charges fired so far=4
  RESULT B:
    answer            : Done. 3 orders charged. (identical to A — the bug hides here)
    real charges fired: 4 (expected 3, got MORE: order A-1002 charged twice)
  COMPARISON (real side effects fired):
    ledger durable     : 3 charges  (each order once)
    ledger drift       : 4 charges  (+1 duplicate from the re-run step)
```

process #2 启动时账本是空的（`ledger has 0 prior charge(s)`）。step 1 重跑时，那笔 `charge:A-1002` 在空账本里看起来像第一次扣款，于是**又 fire 一次**。结果：`real charges fired: 4`，订单 A-1002 被扣了两次。

这个 bug 最毒的地方在那行 `answer : Done. 3 orders charged. (identical to A)`：**agent 给出的回答和 Scenario A 一模一样**。对话 transcript 完整、checkpoint 完整、最终答案完整——一切看起来都对。bug 只在真实副作用计数里现形，从 3 漂移到 4。如果你只看 agent 的输出、不看外部系统的真实账，你永远发现不了它。

这就是 §DRIFT 的教训，也是整章想钉进你脑子的一句话：**对话的 checkpoint 全不代表状态全。让一次重试变安全的那份状态（幂等账本）本身就是持久化状态，漏掉它，at-least-once 执行会静默地变成双重副作用**。漂移不是因为 checkpoint 写错了，而是因为有一份关键状态根本没进 checkpoint 体系。这跟状态规范里"写两处必须同事务/同原子，否则一处必须是另一处的派生"是同一个道理——这里账本和 step checkpoint 既不同事务、也不互为派生，两条独立的持久化时机各自必须正确。

## ⚡ 开放问题：跑几小时/几天的长程 agent

stage15 的模型干净得不真实：4 个 step、一个崩溃点、一份账本、一个稳定键。真实的长程 agent（long-horizon，跑几小时到几天）把每一处都放大成未解的难题，目前**没有通用解，仍在研究**：

- **状态会无限膨胀**。`messages` 是续跑游标，但跑几天它会涨到几十万 token，每次 resume 重发整段 transcript 既贵又会超 context window。于是你得压缩历史（`/compact` 就是干这个），但压缩是有损的——丢掉的细节可能正是 resume 后正确决策需要的。"该压什么、该留什么、压缩后 resume 会不会漂"，目前是工程手艺加经验法则，没有理论保证。这直接接第 5 章上下文工程和第 6 章记忆系统的悬而未决处。

- **幂等键的边界很难划对**。stage15 里"一笔扣款"的身份显然是订单号。真实任务里，一个 step 可能调五个工具、写三个外部系统、其中两个你没有控制权（第三方 API 可能根本不接受幂等键，stage15 的 `charge_card` 没键就抛错算是奢侈）。"什么粒度算同一个操作"在组合操作下没有干净答案。

- **半成品操作的恢复**。崩在"调了外部 API 但还没收到响应"时，你既不知道副作用 fire 没 fire，账本里也没记录。stage15 假设 fire 和记账之间只隔几行同步代码；真实的网络调用让这个窗口大到必须靠对账（reconciliation，事后比对你的账和外部系统的账）来补，而对账逻辑本身又是一套要维护的系统。

- **多 agent 共享状态的一致性**。第 10、11 章的子 agent 各自 checkpoint，但它们碰同一份外部状态时，谁的 resume 先跑、账本怎么跨 agent 共享、补偿（compensate）怎么协调，落到分布式事务和 saga 的老问题上——而这些在"agent 临场决定下一步做什么"的非确定性下，比传统 workflow 还难，因为你连"接下来会写哪些状态"都预测不了。

务实的现状是：长程 agent 的可靠性靠把"不可恢复的危险操作"尽量挡在确定性 workflow 里（第 12 章），让 agent 只在低副作用的区域自由发挥，再用幂等 + 对账兜底。这不是解法，是绕路。能不能给"非确定性、长程、多副作用"的 agent 一套像数据库事务那样的通用保证，是这个领域还没攻下的山头。
