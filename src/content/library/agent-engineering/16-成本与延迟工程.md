一个 8 步的 agent workload，全用贵模型跑要 `$3.54030`；改成「先用便宜模型试，不行再升级」的 cascade（级联）后，同样 8 个任务只花 `$1.71758`——省了 51.5%。但这 51.5% 不是白捡的：cascade 的正确率从 8/8 掉到 7/8，有 1 个任务便宜模型自信地答错了。这就是本章的全部张力：**省下的每一分钱和丢掉的每一分正确率，是同一场赌注的两面**。

这些数字全部来自配套仓 `examples/agent-from-scratch/src/stage16-cost.ts` 的真实运行输出。说清楚两件事免得你误读：(1) 每个 token 数都来自 `estimateTokens`（约 4 字符/token，见 `core/llm.ts`），是个稳定的启发式，不是真 tokenizer，对中文偏差明显；(2) 单价是我编的整数（便宜档输入 `$0.25`/1K、贵档 `$3`/1K），不对应任何厂商真实价目表。所以**绝对美元数是玩具，但比值（cascade vs baseline、缓存折扣、升级率）是这个 stage 真实、可复现的产出**。这章讲比值。

如果你前面的章节是顺着读的：第 01 章拆了 agent loop，第 02 章接了工具和 MCP，第 03/04 章做了沙箱和权限，第 05 章讲了上下文工程和 KV cache，第 06 章做了记忆。到这章，agent 已经能干活了——现在要让它**算得起账、跑得够快**。这两件事是同一个工程问题：钱和延迟都正比于「你调了多大的模型、调了几次、每次重发了多少 token」。

## 成本的三个杠杆

agent 的账单几乎全是 LLM token 费。压它只有三个杠杆，按 ROI 排序：

1. **模型分级（routing / cascade）**——简单步用便宜模型，只在真需要时付贵模型的钱。这是最大杠杆，因为贵档和便宜档的单价差是一个数量级（本章设为 12-15 倍）。
2. **prefix 缓存**——system prompt + 工具 spec 这段固定前缀每次都重发，让它字节级不变就能按折扣价计费。接第 05 章的 KV cache。
3. **并行与流式**——这俩主要砍延迟（wall-clock time），顺带通过缩短上下文窗口的累积间接省点钱。

注意杠杆 1 和 2 是相乘的：路由决定**调哪个模型**，缓存决定**每次调用多便宜**。先看路由，因为它最大、也最危险。

新手第一误区：以为压成本靠「prompt golf」——把 prompt 改短点。错。prompt 字数省下的是边角料；真正的钱在「你为一个 `2+2` 的任务付了 GPT-4 级模型的价」。砍这个才是数量级的差异。

## 杠杆一：模型分级与 cascade

### baseline：全用贵模型

最朴素的 agent 每一步都调最强的模型。正确，但你为每个琐碎步骤都付了溢价。`runBaseline` 就是这条对照线（`stage16-cost.ts:218`）：

```ts
async function runBaseline(llm: LLM, system: string, tasks: Task[], cache: PrefixCache): Promise<CallRecord[]> {
  const records: CallRecord[] = [];
  for (const task of tasks) {
    const opts: GenerateOptions = { system, messages: [{ role: 'user', content: task.prompt }] };
    const record = await callModel(llm, EXPENSIVE, opts, cache);
    // Expensive tier is competent for everything in this toy → always correct.
    records.push({ ...record, taskId: task.id, escalated: false, correct: true });
  }
  return records;
}
```

跑出来：`always-expensive cost=$3.54030 correct=8/8 escalations=0 (0%) cache_hits=7/8`。8 个任务全对，但全价。

### cascade：便宜档先筛，低信心才升级

cascade 的赌注很简单：**真实 agent workload 偏简单**——一堆琐碎查询/编辑，少数几个真需要推理的硬步骤。所以先用便宜模型试，它给出的答案带一个自评信心；信心过线就留下便宜答案，过不了线才升级到贵模型。`runCascade` 的核心（`stage16-cost.ts:235`）：

```ts
const cheapTurn = await cheap.generate(opts);
// ... 算便宜调用的成本（含缓存）...
const confidence = confidenceOf(textOf(cheapTurn.content));
if (confidence >= ESCALATE_BELOW) {
  // 留下便宜答案。它对，当且仅当任务确实简单——
  // 便宜档对硬任务过度自信就会在这里答错（under-escalation）。
  records.push({ ..., tier: 'cheap', escalated: false, correct: !task.groundTruthHard, ... });
  continue;
}
// 升级：便宜那次调用是沉没成本，现在再付一次贵的。
const expRec = await callModel(expensive, EXPENSIVE, opts, expCache);
records.push({ ..., tier: 'expensive', escalated: true, costUsd: cheapCost + expRec.costUsd, correct: true, ... });
```

这里有两个工程细节，是 happy-path 教程不会讲的：

**升级是双重计费，不是替换。** 升级的任务你付了便宜调用 + 贵调用两笔钱（`costUsd: cheapCost + expRec.costUsd`）。便宜那次是沉没成本。所以 cascade 只在「大多数任务都过线」时才划算——升级率一旦太高，双重计费会反噬，cascade 可能比 baseline 还贵。`ESCALATE_BELOW = 0.6`（`stage16-cost.ts:132`）是这个阈值，它直接决定你站在「省钱」还是「赔钱」哪一边。

**便宜档对硬任务的诚实是 cascade 能 work 的前提。** 代码里便宜档对硬任务返回 `[conf=0.40]`（低信心 + 错答案），对简单任务返回 `[conf=0.95]`（高信心 + 对答案）。cascade 全靠这个信心信号来决定升不升级。**一个隐藏不确定性、对硬任务也自信满满的便宜模型，比一个会示弱的便宜模型危险得多**——它会把错答案高信心地塞给你，cascade 根本没机会拦截。这点在选便宜档模型时是硬指标:你要的是 calibration（校准——信心和实际正确率对得上），不只是 accuracy。

### 跑出来的账

```
always-expensive   cost=$3.54030  correct=8/8  escalations=0 (0%)  cache_hits=7/8
cascade            cost=$1.71758  correct=7/8  escalations=3 (38%)  cache_hits=6/8
cascade saves $1.82272 (51.5%) vs always-expensive, at an escalation rate of 38%.
```

51.5% 的省钱，代价是 38% 的升级率和 1 个答错。3 个任务触发了升级（付双份），其余 5 个留在便宜档。这个比值——**省一半钱、漏一个答案**——就是 routing 的本质交易，不是 bug，是设计选择。你的业务能不能接受这个交易，取决于那 1 个错答案值多少钱。

把这套机制对照你正在用的 Claude Code：它的 sub-agent（子 agent）机制本质就是手动分级——你让一个轻量 agent 跑探索性的 grep/read，把贵的主模型留给需要全局判断的合成步骤。`/compact`（压缩对话历史）则是延迟侧的杠杆,缩短重发的上下文。区别在于 Claude Code 的分级是你（人）显式编排的，而 cascade 想把这个判断自动化——这正是下面失败模式的来源。

## 失败模式：路由这场赌注往两个方向输

路由是赌注，赌输有两个方向。stage16 用一个隐藏的 `groundTruthHard` 标签（真相：这任务到底需不需要贵模型）来给路由的猜测打分。生产里你没有这个 oracle——这恰恰是路由有风险的根本原因。`demoRoutingFailures` 把两种输法并排打出来（`stage16-cost.ts:327`）：

```
  t7: guess=easy truth=hard → UNDER-ESCALATION (kept hard task on cheap model → wrong answer)
  t8: guess=hard truth=easy → OVER-ESCALATION (paid premium on an easy task → wasted money)
  static-router mistakes: 1 under-escalation (correctness bug), 1 over-escalation (cost bug).
```

这两个任务是**故意设计**来骗过路由的（`stage16-cost.ts:360` 的 `TASKS`）：

- **t7（under-escalation，欠升级）**：`Solve the halting subset for f.`——一个短、无关键词的硬任务。`routeIsHard` 的启发式靠「长度 > 80 字符 或 含 prove/derive/design 等关键词」判断（`stage16-cost.ts:114`），这句两者都不沾，于是猜「简单」，把硬任务留在便宜模型上 → **答案错了**。这是正确性 bug。
- **t8（over-escalation，过度升级）**：`Please kindly go ahead and simply copy the text from file A into file B, thank you so much.`——一个啰嗦但琐碎的任务。它超过 80 字符，路由猜「难」，升到贵模型 → **白花钱**。这是成本 bug。

两个方向的代价不对称：欠升级你拿到**错答案**（可能进了下游、酿成更大代价），过度升级你只是**多花钱**。对大多数业务，欠升级远比过度升级痛——这意味着你该把路由阈值调保守（宁可多升级），用钱买正确率。

关键洞察在代码注释里说死了（`stage16-cost.ts:347`）:

```
  the cascade dodges under-escalation by re-checking confidence and escalating,
  but it CANNOT dodge over-escalation it never triggers — that risk lives in the
  confidence signal's calibration, not the router heuristic.
```

**cascade 能躲欠升级**：因为它不信一次性的静态猜测，而是真的去问便宜档「你有多确定」，信心低就升级——t7 这种被关键词路由漏掉的硬任务，cascade 靠 `[conf=0.40]` 还是能救回来。**但 cascade 躲不掉它从不触发的过度升级**:如果便宜档对一个简单任务莫名其妙地示弱（返回低信心），cascade 就会无谓升级。这个风险不在路由启发式里,而在**信心信号的校准质量**里。换句话说，cascade 把「路由准不准」的问题，转译成了「便宜档的信心可不可信」的问题——后者更难，但至少它是 self-correcting 的方向。

这也解释了为什么 stage 的 cascade 结果是 7/8 而不是 6/8：静态路由会在 t7 和 t8 各错一次，cascade 靠信心信号把 t7（欠升级）救了回来（升级到贵模型答对），只剩下「便宜档过度自信」那一类它拦不住的错。

## 杠杆二：prefix 缓存经济学

路由决定调哪个模型，缓存决定每次调用多便宜，两者相乘。这里直接复用第 05 章的 KV cache 机制——system prompt + 工具 spec 这段固定前缀每次调用都重发，如果它字节级不变，厂商就从注意力缓存里按折扣价（本章设为 0.1x，即缓存输入便宜 10 倍）供给它。

stage16 用一个 FNV-1a 哈希 + 一个 `Set<string>` 模拟缓存命中（`stage16-cost.ts:179`）：第一次见到某前缀 → MISS（全价，缓存被填充）；之后任何相同前缀 → HIT（前缀部分按折扣计费）。所以冷启动的第一次调用 miss，其余命中；整个热重跑则从第一次就命中。

```
prefix cache: a second identical run costs $1.65030 (8/8 calls hit the cached prefix),
saving a further $0.06727 on resent system+tool tokens.
```

注意这里两件事：

**缓存折扣是在路由之上叠加的。** 第二遍跑（warm cache）路由决策完全不变，只是重发的 system+tool token 走了折扣价，又省了 `$0.06727`。这是把缓存效果从路由效果里隔离出来的对照——同样的路由，更低的账单。

**这个省得不多，因为玩具 prompt 短。** `$0.06727` 在 `$1.71` 上只占约 4%。但生产里 system prompt + 工具 spec 常常是输入 token 的大头（几千 token 的工具定义很常见），缓存折扣的绝对值会大得多。stage 的相对数字偏小，是因为它的前缀只是一句 `'You are an agent...'`——这是玩具数据的如实标注，别拿这个 4% 当生产预期。

第 05 章讲过的坑这里仍然成立：**前缀改一个字节，从那个字节往后的缓存全 miss**。所以工程上要把易变内容（时间戳、用户名、动态上下文）放到前缀**之后**，把 system prompt 和工具 spec 这些稳定部分钉死在最前面。Claude Code 的 `ToolSearch`（按需加载工具 schema）和这个直接相关——它不把全部工具 spec 一次性塞进前缀，而是延迟加载，既缩短前缀又保住前缀的稳定性。

## 杠杆三：并行与流式（延迟侧）

前两个杠杆砍的是钱，第三个砍的是 wall-clock 延迟。stage16 没有专门 demo 这个（它的重点是成本的可测性），所以这里讲机制和取舍，不编数字。

**并行**:agent loop 里若一步要调多个互不依赖的工具（比如同时 read 三个文件），别串行 await，用 `Promise.all` 一把发出去。延迟从「三次往返之和」降到「最慢那次」。但有两个约束:(1) 受 `arch-runtime` 的资源上限规矩管——别 `Promise.all` 几百个并发，要用 `pLimit(N)` 限流，否则打爆 rate limit 或连接池;(2) 只有**无依赖**的步骤能并行,有数据依赖的（A 的输出喂给 B）必须串。这正是第 14 章 workflow 编排要解决的问题——确定性地把 DAG（有向无环图）里能并行的支路并行掉。

**流式（streaming）**:模型边生成边返回 token，而不是等整个 response 完成。它不省总 token、不省钱，但大幅降**首字延迟**（time-to-first-token），对交互式 agent（如 Claude Code 的实时输出）是体验关键。代价是:流式下你拿不到完整 response 才能做的事（比如等 tool_use block 完整才能执行），所以工具调用和流式的交互要小心处理边界。

工程取舍很直接:并行和流式都是**延迟优化**，不碰成本。如果你的瓶颈是账单，先上杠杆 1、2;如果瓶颈是用户等得不耐烦，才上杠杆 3。别把它们混为一谈——我见过团队为了「优化成本」上流式，结果账单一分没动。

## ⚡ 开放问题：「何时升级到大模型」仍然没有通用解

把这章收口在最痛的地方:**自动判定一个任务该不该升级到大模型,目前还没有通用解,全靠启发式。** 这不是 stage16 偷懒,是这个领域当前的真实状态。

stage16 用了两种启发式,都不完美:

1. **静态路由**（`routeIsHard`,看长度 + 关键词）——廉价、零额外调用,但 t7/t8 那两个对抗样本直接戳穿它。这是大多数团队第一版会 ship 的东西,也是它的失败模式 ARE 失败模式。
2. **cascade 的信心阈值**（`confidence >= 0.6`）——能 self-correct 欠升级,但把问题转译成了「信心信号可不可信」。而**信心校准本身是个未解的研究问题**:LLM 的自评信心常常 miscalibrated,简单任务过度自信、难任务也可能盲目自信。logprobs、self-consistency（多次采样看一致性）、verifier model（专门的裁判模型）都是当前在试的信号,但没有一个是 drop-in、跨任务通用的解。

更深一层:理想的路由器需要**在不真正解题的前提下,预判这题有多难**——这本身近乎和解题一样难(你怎么知道 `Solve the halting subset for f.` 是硬任务,除非你已经理解了它?)。这是个鸡生蛋问题,学术界叫 routing/cascading under uncertainty,正在研究,没有银弹。

实践上的诚实建议:别追求完美路由器。**先量化你的两种错误的代价**(欠升级丢的正确率值多少钱、过度升级浪费的钱有多少),然后把阈值调到让总代价最小的那一侧——对多数业务是保守(宁可多花钱也少答错)。然后用真实流量持续校准这个阈值。这是工程,不是求最优解;在 calibration 这个上游问题被解决之前,路由永远是一场需要你下注的赌博,而这章教你的是怎么把这场赌博的两面都量化成钱和正确率,而不是凭感觉。

## 小结

- **成本三杠杆按 ROI 排序:模型分级 > prefix 缓存 > 并行/流式。** 前两个砍钱(相乘关系),第三个砍延迟。别用 prompt golf 当主力,那是边角料。
- **cascade 的省钱和丢正确率是同一场赌注**:stage16 实测省 51.5%（`$3.54` → `$1.72`),代价是 7/8 而非 8/8,升级率 38%。能不能接受取决于那个错答案值多少钱。
- **路由两个失败方向**:欠升级(硬任务留便宜档 → 错答案,正确性 bug)、过度升级(简单任务付溢价 → 浪费钱,成本 bug)。cascade 靠信心信号能救欠升级,救不了它从不触发的过度升级。
- **缓存折扣叠在路由之上**,但 stage 的 4% 是玩具 prompt 太短所致,生产里工具 spec 占大头时折扣绝对值大得多。前缀改一字节缓存全 miss——把易变内容放前缀之后。
- **⚡「何时升级」无通用解**:静态路由被对抗样本戳穿,信心阈值把问题转译成 calibration(本身未解)。务实做法是量化两种错误的代价、把阈值调到总代价最小那侧、用真实流量持续校准。
