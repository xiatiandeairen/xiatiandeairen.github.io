> 阶段三 · 规模与持久 · 第 5 章
>
> 第 1 章讲控制循环时埋了一条曲线：每轮把整个对话历史重发给模型，所以一个跑 T 轮的 Agent 的总成本不是 O(T)，是 **O(T²)**。当时只是个 printout，这一章把它当问题来解。

第 1 章的循环代码里有这么三行注释，是这整章的起点（`examples/agent-from-scratch/src/stage01-loop.ts:37-39`）：

```ts
// inputTokens for each model call, in order. The whole transcript is resent
// every turn, so this grows turn over turn — the cost of a T-turn agent is the
// SUM of these, i.e. O(T^2) in transcript length, not O(T). See printout.
```

模型 API 是无状态的——它不记得上一轮。要让 Agent「记得」三轮前查了什么，唯一的办法是把那三轮原样塞进这一轮的请求里。于是第 10 轮的请求包含前 9 轮的全文，第 50 轮包含前 49 轮。每个 token 在它进入历史后的**每一轮**都要重新付费。这就是为什么一个长会话的 Agent，越往后每轮越贵、越慢——不是模型变笨了，是你在反复付同一笔钱。

所以真正约束一个 Agent 会话的资源，从来不是「模型有多聪明」，是 **token 预算**。这件事大到需要从「prompt engineering」改名叫 **context engineering（上下文工程）**：prompt engineering 是在调一句话的措辞，context engineering 是在管理一个有限的、会被反复计费的预算池——什么进窗口、什么留在窗口外、什么放在窗口的哪个位置。

你现在就泡在一个 context engineering 的活体里。这个会话里发生过 `/compact`（把前面的对话压成摘要继续）、subagent 把一大坨探索结果消化完只回传结论、provider 在背后命中 prompt cache——这一章的四个机制，就是 Claude Code 在你看不见的地方一直在做的事。下面每个机制都配 stage05 的真实数字，跑 `cd examples/agent-from-scratch && npm run stage05` 当场复现。

**先把一个诚实标注放在最前面**：stage05 全程离线、不连网、不调真模型。所有 token 数来自 `estimateTokens`（`core/llm.ts` 里一个「约 4 字符 = 1 token」的启发式），**不是真的 BPE tokenizer**，对中文尤其不准。压缩/卸载/cache 那三组数字是在这个估算上做的「同口径前后对比」（before 和 after 用同一把尺，比例可信）；第四个机制 context rot 的曲线是**手写的示意玩具，不是任何模型的实测**——这点到那一节我会再喊一遍。

## 机制一：压缩（compaction）——把旧轮折叠成摘要

最直接的省钱办法：当历史快撑满预算时，把一堆旧对话折叠成一段摘要，腾出空间。stage05 造了一个 16 条消息的排查对话（一个「checkout 页面间歇性 500」的 debug 过程），设预算 200 token，实测：

```
transcript    : 16 messages, 269 tokens (budget 200)
over budget?  : yes — compact before next turn
folded turns  : 13 (kept first task + last 2 turns verbatim)
tokens        : 269 → 137  (51% of original, 2.0x smaller)
invariant     : decision preserved=true, open TODO preserved=true (else: agent amnesia)
```

`269 → 137 token`，折叠了 13 个旧轮，压到原来的 51%、2 倍小。策略很朴素（`src/stage05-context.ts` 的 `compactConversation`）：保留**第一条**（最初的任务描述）+ **最近 2 轮**原文，中间全部折成一条摘要。

这就是 Claude Code 的 `/compact` 在干的事——当你这个会话的历史快到窗口上限，它把前面几十轮压成一段「之前我们做了 X、决定了 Y、还差 Z」的摘要，然后带着这段摘要继续。区别只是 stage05 为了离线、可复现，用机械规则（扫 `DECISION:` / `TODO:` 标签）抽取摘要，而真 Agent 会让模型自己写这段摘要。但**契约完全一样**。

### 关键不变量：压缩字节容易，压缩时保住「活状态」才是本事

压缩 transcript 的字节是 trivial 的——`.slice()` 一下就行。难的是在扔字节的同时**不扔掉 Agent 还需要用的活状态**：没做完的任务、已经拍板的决策。丢了这些，Agent 会失忆——重新问你已经答过的问题、重做已经做过的事、甚至推翻自己 20 轮前定下的方案。

stage05 把这条不变量写进了注释（`src/stage05-context.ts:39-47`）：

```
// Invariant that makes compaction safe vs lossy: the summary MUST preserve
// "live" state — unfinished tasks and key decisions the agent still needs to
// act on. ... Compressing *transcript bytes* is easy; the skill is
// compressing bytes WITHOUT compressing live state.
```

代码里这条不变量长这样：中间那段被折叠的对话，凡是带 `DECISION:` / `TODO:` 标签的句子，**逐字捞进摘要**；其余纯 chatter（「我去看下日志」「好的继续」）才折成一句计数。实测输出的 `decision preserved=true, open TODO preserved=true` 就是在证明：那条「根因是连接池耗尽（max=10），TODO 把池子调到 30」的关键状态，活着穿过了压缩。

### 一个真实的失败：我自己第一版把决策弄丢了

这个机制有个我亲手踩过的坑，值得讲。第一版的策略是「保留第一条 + 最近 N 轮原文，中间折叠」——听起来没问题。但我构造测试对话时,把那条 `DECISION` 放在了**会话中段**，然后又追加了两轮收尾 chatter（「听起来对，去试吧」「正在改配置跑 replay」）。结果最近 2 轮的窗口里装的全是这两句噪音,而那条决策正好落在被折叠区的中段——`preserved=false`，Agent 失忆。

这个 bug 的根因不是「保留窗口太小」,而是**「最近」不等于「重要」**。一条决策可能在第 8 轮拍板,然后被 40 轮无关 chatter 推到「旧」区。如果你的压缩策略只信「位置上靠后」，它会系统性地丢掉早拍板、长期有效的决策。stage05 的测试对话是**故意这么排的**（注释 `src/stage05-context.ts:78-92` 明说「raised EARLY (mid-session)... they sit inside the region that gets folded, so the summary must rescue them」）——它把 decision 放在折叠区中段,就是为了逼出「摘要必须主动救援活状态」这个契约,而不是「保留最近 N 轮就完事」。

工程结论：**压缩的正确性指标不是压缩比,是「活状态零丢失」**。先定义什么是活状态（开放任务 + 未撤销的决策),再让摘要保证它们逐条幸存,最后才谈压到多小。反过来——先追求压缩比——必然在某个会话上丢决策,而且丢得静默,你只会看到 Agent 突然「变笨」。

## 机制二：卸载（offloading）——大观测留个句柄,别塞进窗口

压缩处理的是「历史太长」。卸载处理的是另一个问题：**单次工具结果太大**。

一个工具可能一次返回远超「每轮都值得带着」的量——一个 4000 行的目录树、一个巨型文件、一个 API 全量 dump。如果你把它原样塞进 transcript,它不只这一轮贵——因为 transcript 每轮重发,这坨东西会**在后续每一轮**都被重新计费。一次 43000 token 的 `ls -R`,在一个还要跑 30 轮的会话里,就是 43000 × 30 的账单。

卸载的做法:把完整结果写到**窗口外**的存储(stage05 用一个内存 Map 模拟磁盘),窗口里只留一个**句柄 + 一句摘要**。Agent 需要细节时,用句柄按需重新读取,但在那之前不付钱。stage05 实测(`offloadLargeObservation`,4000 行):

```
tool output   : 4000-line listing = 43729 tokens if inlined
in context    : handle + summary  = 61 tokens
saved / turn  : 43668 tokens (and this saving recurs EVERY turn, since transcript is re-sent)
recoverable?  : read_store("store://ls/call_ls_1") → 4000 lines (full fidelity on demand)
```

`43729 token`(内联)→ `61 token`(句柄+摘要),**每轮省 43668 token**,而且这笔节省**每一轮都复发**。窗口里留的那条是:`[offloaded] 4000 entries (43729 tokens) stored at store://ls/...  Re-read with read_store(...) if you need specifics.`——足够 Agent 判断「我现在需不需要展开它」,需要时全保真捞回来。

这正是 subagent 隔离 context 的核心机制。当 Claude Code 派一个 subagent 去「读遍整个 codebase 找出所有调用方」,subagent 自己的窗口里塞满了几十个文件的全文——但它回传给主 Agent 的,只有一句结论(「调用方在这 3 处:A/B/C」)。那几十个文件的全文从来没进过主 Agent 的窗口,所以主 Agent 不会在后续每轮为它们付费。subagent = 一个会自己消化、只回传摘要的「带脑子的句柄」。本系统提示词最前面那段「The agent's final message is returned to you as the tool result」说的就是这件事。

### 失败模式:摘要太瘦,Agent 反复回捞

卸载不是免费的。它把成本从「每轮都付全量」换成了「需要时付一次全量 + 每轮付摘要」。如果摘要写得太瘦——瘦到 Agent 没法判断「我要的东西在不在里面」——它会**反复 `read_store` 回捞**,每次回捞都把那 43729 token 重新拉进窗口。捞个三四次,你卸载省下的就全吐回去了,还多搭上几轮往返延迟。

所以摘要的设计目标不是「最短」,是「**让 Agent 不用展开就能做对去/留决策**」。stage05 的摘要里塞了 entry 数、token 数、顶层目录结构——这些恰好是「我要找的文件在不在这棵树里」的判据。一个反面例子:摘要只写「[offloaded] a directory listing」,Agent 完全无法判断,只能每次都捞。**卸载的隐性契约是摘要的信息密度**,不是存储本身。

## 机制三:KV cache 前缀稳定性——稳定的前缀才是便宜的前缀

前两个机制减少 token 数量。这第三个不减数量,而是让**同样的 token 更便宜**。

模型在生成时会缓存它已经处理过的 token 的注意力 key/value 张量(就叫 KV cache)。下一次调用,provider 可以对**最长的完全相同前缀**复用这份缓存,只重新处理后面分叉的部分。命中缓存的 input token 计费便宜得多(数量级差异)、几乎不加延迟。

但有个硬约束:缓存按**逐字节精确前缀匹配**。你调换两个工具的顺序、改系统提示里一个词、在最前面塞一个时间戳——从那个改动点起,前缀就变了,缓存从那里开始全部失效,后面的全部重新按原价付。

stage05 用 FNV-1a(一个简单的滚动哈希)对 `[system, ...tools]` 分段做哈希来模拟「provider 拿什么当缓存 key」(`hashSegments`)。实测三个 case:

```
base prefix   : system + 3 tools = 56 tokens, hash=16eb63f9
turn 2 (same) : hash=16eb63f9 → HIT — all 56 prefix tokens served from cache (cheap, ~0 added latency)
edit system   : hash=fae227f0 → MISS — diverges at segment 0/4; re-pay 56 tokens (whole prefix invalid)
swap 2 tools  : hash=e4b7f56b → MISS — diverges at segment 1/4; re-pay 45 tokens after the swap
```

三个 case 把规律讲透了:

- **前缀一字不改** → `hash=16eb63f9` 和上一轮相同 → **HIT**,56 个前缀 token 全部走缓存,便宜、零额外延迟。
- **改 system 一个词**(把 `precisely` 改成 `carefully`)→ `hash=fae227f0` → **MISS**,而且因为 system 是第 0 段、排在最前,从段 0 就分叉 → 整个 56 token 前缀全作废,全部重付。
- **只调换前两个工具的顺序**(工具内容一字没改,只换位置)→ `hash=e4b7f56b` → **MISS**,从段 1 分叉,重付那之后的 45 token。

注意第三个 case 的反直觉:工具的**内容**完全没变,你只是换了个排列顺序,缓存就从换位点开始全部失效。模型不在乎「这两个工具我都见过」,它只认「字节序列从这里开始和缓存里的不一样了」。

结论就是 stage05 自己打印的那两句(`src/stage05-context.ts:318-319`):

```
lesson        : the longer the prefix you keep byte-identical, the more you cache.
                put volatile content (timestamps, retrieved docs) LAST, not in the prefix.
```

**前缀保持逐字节一致的部分越长,你缓存得越多。** 推论是一组很具体的工程纪律:

- 系统提示、工具定义这种每轮都一样的东西,放最前面,并且**别动它**——哪怕「优化一个用词」也会让整段前缀作废。
- 工具列表的顺序要**固定**,别按「相关性」动态重排——重排 = MISS。
- 真正易变的东西——当前时间戳、本轮检索到的文档片段、用户最新输入——放在**最后**,绝不进前缀。一个常见的坑:有人为了「让模型知道现在几点」把 `当前时间: 2026-06-21 14:32:07` 塞进 system prompt 开头,结果每一轮这个秒级时间戳都不同,前缀**永远 MISS**,整个会话的 cache 全废。

这是 Claude Code 这类工具在背后吃到的便宜:它的系统提示和工具定义在整个会话里逐字节稳定,所以每轮的这一大段前缀都命中 provider 的 prompt cache。你写自己的 Agent 时,把这条当硬约束:**前缀是资产,改前缀是花钱**。

## ⚡ 机制四:context rot——长上下文会在「中段」烂掉(前沿/开放问题)

前三个机制都有确定的工程答案。第四个没有——它是个**还没解决的开放问题**,我把它单独标出来。

现象叫 context rot(上下文腐烂):随着窗口里塞的东西变多,模型**使用**其中某个事实的能力不是均匀的。经验上,靠近开头和结尾的事实召回率高,**中段会塌陷**——这就是有名的「lost in the middle(迷失在中间)」。同一个事实,放窗口边缘模型能用,埋窗口中段就用不出来。

stage05 画了一条 U 形曲线来让这个形状具体化:

```
depth   recall  bar
  0%     95%   █████████████████████████████
 ...
 50%     45%   ██████████████
 ...
100%     95%   █████████████████████████████

shape         : edge ~95% vs middle ~45% — same fact, worse recall mid-context
```

边缘约 95%、中段约 45%——同一个事实,埋中段召回率掉一半。

**但这条曲线必须诚实标注**——stage05 自己在打印它之前先喊了一句(我原样转述):

```
    *** ILLUSTRATIVE TOY: hand-written U-curve, NOT a real model / benchmark ***
```

代码注释里说得更死(`src/stage05-context.ts:330-335`):这条 U 形召回率是一个**手写的、关于位置的函数**(`illustrativeRecallAt`,就是 `0.45 + 0.5 * 边缘距离` 算出来的),它**不调用任何模型、没对任何 benchmark 校准**,只是为了把问题的**形状**画出来。`95% vs 45%` 这两个数字是教学卡通,**不是任何模型上的实测证据**。

我在这里也照样标注:**上面这条曲线不能当数据引用**。它说明的是「中段塌陷」这个现象的**形状**值得你警惕,但具体塌多深、在哪个上下文长度开始塌、不同模型差多少——**必须用真模型在真的 needle-in-a-haystack(大海捞针:在长文档里塞一个事实,问模型能否捞出来)benchmark 上实测**才算数。stage05 离线,做不到这件事,所以它老实承认自己只画了个卡通。这正是这一章唯一一个我请你**别信数字、只信形状**的地方。

即便如此,工程含义是清楚的,而且和前三个机制咬合:**一个必须被用到的事实,放窗口结尾,或者在临用前重新注入一遍;别把它埋在中段。** 把它和机制一连起来——压缩时,那条「活状态」摘要应该放在靠近最新轮的位置,而不是塞在折叠区中间;和机制三连起来——结尾本来就是放易变内容的地方,正好也是召回率高的地方。三个机制指向同一个布局原则:**窗口是有位置语义的,首尾贵重,中段廉价。**

stage05 的收尾两句把这个开放性也标了出来:

```
takeaway      : if a fact must be used, place it near the END (or re-inject it); do not bury it mid-context.
               but VERIFY with a real model — this curve is a teaching cartoon, not data.
```

「但要用真模型验证」——这就是这个前沿问题的诚实状态。缓解手段(重排序、把 needle 放最后、缩短上下文)都还是活跃研究,没有一个「照着做就对」的定论。

## 收束:把窗口当预算来花

回到开头那条 O(T²) 曲线。这一章的四个机制,本质上是四种「往预算池里省钱 / 让钱更值」的手段,各管一段:

| 机制 | 管的问题 | stage05 实测 | 一句话 |
|------|---------|-------------|--------|
| 压缩 | 历史太长 | 269 → 137 token(2.0x),活状态零丢失 | 折叠旧轮,但保住决策/TODO |
| 卸载 | 单次结果太大 | 43729 → 61 token,每轮省 43668 | 大观测留句柄,需要时再捞 |
| KV cache | 同样 token 太贵 | 同前缀 HIT(56 全命中);改一词 MISS 重付 56;调顺序 MISS 重付 45 | 前缀逐字节稳定 = 资产,易变内容放最后 |
| ⚡ context rot | 长窗口中段烂掉 | 边缘 ~95% vs 中段 ~45%(**手写示意,非实测**) | 要被用的事实放结尾或重注入——但用真模型验证 |

四个机制有个共同的世界观:**context window 不是无限的草稿纸,是一个有限的、会被反复计费的、有位置语义的预算池。** 谁进窗口(卸载)、什么时候腾空(压缩)、字节怎么排布(KV cache)、重要的东西放哪个位置(context rot)——这四件事合起来就是 context engineering。

它和下一章直接接续:第 6 章讲长期记忆,把状态从「每轮重发的窗口」搬到「跨会话持久的磁盘」。卸载是这件事的最小形态(搬到会话内的窗口外存储);长期记忆是它的完全体(搬到会话之间的持久存储)。两章是同一个动作在不同时间尺度上的展开——**把不该一直占着预算的东西,挪出预算去。**
