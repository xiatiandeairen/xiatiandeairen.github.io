市面上讲 Agent 的材料有个通病：篇幅很长，但读完你还是不会写。它们罗列概念（ReAct、Memory、Multi-Agent、RAG……），却很少回答工程师真正要问的三个问题——**这东西到底怎么转、什么时候会坏、坏一次要付多少钱**。这本书反着来：每一章只讲一个能力，每个能力都对应一段能跑的代码，每段代码都附带它的失败模式和真实数字。

先把最容易被神秘化的词拆掉。

## 一句话祛魅：Agent 不是魔法，是一个循环

一个 Agent，去掉所有包装，就是这个东西：

```
Agent = LLM + 循环(loop) + 工具(tools) + 终止条件(stop condition)
```

模型负责「想下一步做什么」，循环负责「反复问模型直到事情做完」，工具是模型能伸出去改变世界的手，终止条件决定什么时候停。没了。所谓「自主」「智能体」，工程上就是这四件东西拼起来。本书第 1 章会用 40 行 TypeScript 把这个循环写出来，你会发现它朴素到有点失望——这是好事，失望意味着你看清了它。

那 Agent 和普通的「AI 工作流（workflow）」差在哪？差在**谁决定下一步**。

- **Workflow**：控制流是你写死的。第一步抽取、第二步检索、第三步总结——步骤和顺序在代码里，模型只在每一格里填内容。可预测、可测试、可复现。
- **Agent**：控制流交给了模型。下一步做什么、做几步、什么时候停，是模型在运行时决定的。灵活，但也因此不可预测、难测试、会跑飞。

这不是「谁更高级」，是一把选型的刀：**任务路径确定就用 workflow，路径需要模型临场判断才用 agent**。你正在用的这个 Claude Code 环境同时提供两者——一个模型驱动的主循环（agent），和一个让你把多个子任务用确定性脚本编排起来的 Workflow 工具。它自己就示范了：成熟的系统是 agent 和 workflow 的混合，不是非此即彼。

## 为什么从「裸写」开始，而且不用框架

LangChain、LlamaIndex 这类框架的价值是帮你少写样板，但对**学习**是负价值：它们把上面那个循环、把工具调用的协议、把上下文怎么管理，全藏进了抽象后面。你调 `agent.run()` 得到结果，却不知道中间发生了什么——而「中间发生了什么」恰恰是这本书要教的。

所以配套代码仓 [`examples/agent-from-scratch`](#配套代码) 只用一个外部依赖：`@anthropic-ai/sdk`，而且它是可选的。**没有 API key 也能跑**：每个 stage 都带一个确定性的离线 mock，机制是真的，只有模型被脚本化。这样你读代码、改代码、看输出，不花钱、不联网、不等网络。等你想看真模型驱动同一个循环，填一个 key，代码一行不改。

## 解剖图：一个 Agent 的器官

把 Agent 当成一个有机体，它有这些器官。每个器官可以单独失效，单独攻击，也单独优化——所以可以单独讲。

| 器官 | 它负责 | 不管它会怎样 | 本书位置 |
|---|---|---|---|
| **控制循环 harness** | 反复问模型、跑工具、决定何时停 | 无限空转 / 提前退出 / 烧钱 | 第 1 章 |
| **工具 + MCP** | 让模型能读写世界、接外部能力 | 模型有嘴无手，只能聊天 | 第 2 章 |
| **执行沙箱** | 隔离模型生成的命令，挡「能做什么」 | 一条注入命令删光你的盘 | 第 3 章 |
| **权限授权** | 在工具执行前判定「准不准做」 | 私有数据被一句注入外泄 | 第 4 章 |
| **上下文工程** | 把有限的 context window 当预算经营 | 成本随轮数平方膨胀、模型失忆 | 第 5 章 |
| **记忆系统** | context 之外的长期记忆与召回 | 每次对话都从零开始 | 第 6 章 |

这六个器官有依赖顺序，所以章节是有序的，建议顺读：

- **第 1 章（循环）** 是地基。它会算出一个关键的坏消息：因为每一轮都要把整段对话重发给模型，一个 T 轮的 agent，成本是各轮输入 token 的**求和**，随轮数近似按 **O(T²)** 增长——不是你直觉以为的线性。这个伏笔，第 5 章来还。
- **第 2 章（工具 + MCP）** 给循环装上手。先讲 `tool_use`/`tool_result` 的报文长什么样、并行调用的硬约束、工具报错怎么让模型自己恢复；再从零手写一个 MCP（Model Context Protocol，模型接外部工具的统一协议）的最小实现，并演示它打开的新攻击面。
- **第 3 章（沙箱）** 和 **第 4 章（权限）** 是一对纵深防御：沙箱挡「能不能做」（内核层），权限挡「准不准做」（授权层）。第 4 章会完整复现 Simon Willison 的 **lethal trifecta（致命三要素）**——私有数据 + 不可信内容 + 外发通道——让你亲眼看见一句藏在数据里的注入怎么把密钥发到攻击者服务器，以及哪一道防线真能拦住它（剧透：不是 system prompt 里那句「请不要外泄」）。
- **第 5 章（上下文）** 还第 1 章埋的 O(T²) 债：压缩、卸载、KV 缓存前缀稳定性，三招都给出压缩前后的真实 token 数。
- **第 6 章（记忆）** 处理 context 之外的事：分层记忆、向量召回、遗忘，以及记忆投毒——第 4 章的不可信内容如果渗进了长期记忆，会沿着时间轴反复污染你的 agent。

## 这本书的质量契约

为了不滑回「长而浅」，每一章都受同一套硬约束。读的时候你可以拿它当验收清单：

1. **可跑代码**：每个机制都指向配套仓里一段真实、能运行的代码，不是想象的伪代码。
2. **实测数字**：所有量化都来自代码的真实输出。token 用 `~4 字符/token` 估算（代码里标注了它不是真 tokenizer），有 key 时换成 API 实测用量；示意性的玩具数据（比如第 5 章那条 context rot 曲线）会被明确标成「示意，不是实测」。**不编造数字**是底线。
3. **失败模式优先**：每个机制先讲它怎么坏、为什么这么设计，再讲它怎么用。只会 happy path 的人上不了生产。
4. **一个开放问题**：每章留一个用 ⚡ 标注的前沿/未解问题——长程循环的稳定性、code-mode 工具调用、沙箱逃逸与冷启动的矛盾、lethal trifecta、context rot、记忆投毒。这些今天都没有干净的通用解。诚实地告诉你「这里还没有答案」，比假装一切都已解决更有用。

## 这本书（暂时）没讲什么

诚实地划边界。规划与推理、多 Agent 编排、评测与可观测性、长程可靠性、以及把所有开放问题汇总成一张「未解问题地图」——这些是这套书的后续，不在这一卷里。这一卷聚焦六个**一等公民器官**，把它们做到能写、能跑、能讲透；其余等这六章的工艺被验证之后再展开。窄而透，胜过宽而浅。

## 配套代码

代码仓在本站仓库的 `examples/agent-from-scratch/` 目录。结构与本书章节一一对应：

```
src/
  core/          # 所有 stage 共享的最小类型 + LLM 抽象（真 SDK 适配器 + 离线 mock）
  stage01-loop.ts          # 第 1 章
  stage02-tools.ts         # 第 2 章 · function calling
  stage02-mcp.ts           # 第 2 章 · 手写 MCP（+ mcp-server.ts）
  stage03-sandbox.ts       # 第 3 章
  stage04-permissions.ts   # 第 4 章
  stage05-context.ts       # 第 5 章
  stage06-memory.ts        # 第 6 章
```

跑起来：

```bash
cd examples/agent-from-scratch
npm install
npm run stage01      # 然后 stage02:tools / stage02:mcp / stage03 / stage04 / stage05 / stage06
```

读法建议：先读这一章建立地图，然后**一章正文配一次 `npm run`**——先跑出输出，再回正文看每个数字是怎么来的、每个机制为什么这么设计。看着代码真的拦下一次注入、真的把 43000 token 压成 61 个，比读十段「Agent 具备记忆能力」有用得多。
