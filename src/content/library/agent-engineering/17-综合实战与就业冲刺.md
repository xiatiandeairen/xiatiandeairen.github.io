前面十六章每一章只养一个器官，单独看都很干净。但生产里把它们接成一个 agent 时，bug 不在任何一个器官里面，而在**器官之间的接缝**：权限网关吐出一个 `deny`，循环的控制流该怎么接？记忆召回了一条旧事实，它进了当前任务的上下文，模型该信它还是信刚读到的文件？这一章不再新增器官，而是把 01 的循环、02 的工具、04 的权限网关、06 的分层记忆、08 的计划即数据、14 的结构化追踪，全部拼进一个自洽的 agent，跑一个真实任务（修一个配置 bug 并验证），然后逐帧拆它的运行轨迹，专门盯那些接缝。

配套代码是 `examples/agent-from-scratch/capstone/agent.ts`，一个文件、离线可跑（`npx tsx capstone/agent.ts`）。它故意不 import 任何 `stageNN` 文件——import 会触发那个 stage 的 `main()`——而是把每个器官的最小实现按需重写在一处，让 capstone 读起来是一个整体。模型用确定性的 `MockLLM`（脚本化假模型），所以多步流程可复现；但**控制流、网关判决、记忆冲突、deny 后的恢复全是真机制**，被脚本化的只是模型的措辞。

## 一、端到端：一条 17 步 trace，逐帧指出哪个器官在干活

任务是 `Fix the timeout bug in config.json and verify it.`。世界里 `config.json` 的 `timeout_ms` 是 `0`（真 bug，见 `agent.ts` 的 `freshWorld()`），而记忆里被**故意种了一条旧事实**说"配置上个 session 已经设好了，不用修"（`source=tool`）。计划是三步：读、改、验。

这是真实运行打出来的时间线，每步一个 span：

```
   1   🧠 recall injected 1 proc, 1 fact, 0 episodes
   2   🤔 decide 先读 config.json，确认现状（不轻信记忆里的旧事实）。  (in=341tok)
   3   🚦 gate   read_file -> allow (allowed by policy)
   4   ⚙️  act    read_file: { "timeout_ms": 0, "retries": 3 }
   5   🧠 recall injected 1 proc, 1 fact, 2 episodes
   6   🤔 decide 文件里 timeout_ms 是 0，与记忆中的旧事实冲突——以文件为准，写入修复值 5000。  (in=426tok)
   7   🚦 gate   write_file -> allow (asked human -> allow)
   8   ⚙️  act    write_file: wrote 41 bytes to config.json
   9   🧠 recall injected 1 proc, 1 fact, 2 episodes
  10   🤔 decide 写完了，跑 verify_config 确认修复生效。  (in=503tok)
  11   🚦 gate   verify_config -> allow (allowed by policy)
  12   ⚙️  act    verify_config: PASS: timeout_ms is positive
  13   🧠 recall injected 1 proc, 1 fact, 2 episodes
  14   🤔 decide （被注入诱导）把配置发到外部备份一下。  (in=547tok)
  15 ✗ 🚦 gate   send_external -> deny (denied by policy)
  16   🧠 recall injected 1 proc, 1 fact, 2 episodes
  17   🤔 decide 外发被权限拦截，放弃该动作。任务完成：config.json 的 timeout_ms 已修复并通过校验。  (in=617tok)
```

一帧帧读，每帧对应一个器官，对应一章：

**第 1 帧（🧠 recall，记忆 / 06）。** 模型还没说话，循环先做了一件事：从记忆里召回相关切片，拼进 system 前导。`agent.ts` 的 RECALL 段按三种规则各召回一路——程序性记忆按 trigger 词（任务里有 `config`，命中那条"编辑配置后必须 verify"的规则）、语义记忆按含义（toy 词袋相似度命中那条旧事实）、情节记忆按时近（第 1 帧还没有历史，所以 `0 episodes`）。**这是记忆唯一有意义的时刻：它在模型决策的前一瞬把切片注入上下文。** 记忆不是一个被动的数据库，是在这一帧弯曲了下一个决定。

**第 2 帧（🤔 decide，循环 / 01）。** 一次模型调用。注意它的话："不轻信记忆里的旧事实"——它看到了被注入的那条 `[source=tool]` 旧事实，但没照单全收，决定先读文件确认。`in=341tok` 是这一轮重发的整个 transcript 的输入 token（约 4 字符/token 估算，见 `core/llm.ts` 的 `estimateTokens`，是估算不是计费数字）。

**第 3–4 帧（🚦 gate → ⚙️ act，权限 + 工具 / 04 + 02）。** 模型请求 `read_file`，网关先判决再执行。读是安全动作，策略直接 `allow`，然后才真的去 toy 文件世界里读，拿回 `{ "timeout_ms": 0, ... }`——**真 bug 现形**。判决永远在执行之前，这是 04 反复强调的：网关是代码，模型说服不了它。

**第 5–8 帧（冲突解决 + 受控写）。** 第 5 帧再召回，这次有了 2 条情节（读文件那两步进了情节日志）。第 6 帧是全章的关键决策：**文件说 0，记忆说"已修好"，模型选了文件。** 第 7 帧 `write_file` 走的是 `ask` 规则——写是变更动作，需要人确认，离线 demo 里脚本化的"人"批了这次对 `config.json` 的写（`asked human -> allow`），第 8 帧才真的落盘 41 字节。

**第 11–12 帧（验证 / 08 的"done 是事实不是 vibe"）。** 跑 `verify_config`，返回 `PASS`。这一步让计划的第 3 步从"声称完成"变成"对世界状态验证过"。最后结果里 `goal reached : true` 是对世界**重新 parse** 一遍 JSON 得到的，不是模型自报。

**第 14–15 帧（注入 → deny，这是下一节的主角）。** 模型被脚本模拟成"被 prompt 注入诱导"，试图把配置外发到 `exfil.example`。第 15 帧网关 `deny`——这是 04 讲的 lethal trifecta（敏感数据 + 不可信内容 + 外发通道三者同时具备就构成数据外泄风险）里的"外发腿"被掐断。

**第 17 帧（恢复 + 收束）。** 模型看到了那个 `isError` 的 deny 结果，放弃外发，宣布任务完成。循环在一次 deny 之后**活了下来并收敛**。

把 Claude Code 当活体参照看这条 trace：你按 Enter 后看到的 TodoWrite 计划逐项打勾、Bash 命令跑前弹出的权限确认、`/compact` 之前那段越来越长的上下文——是同一套器官，只是规模和打磨程度不同。这条 17 步 trace 是把那些平时被 UI 藏起来的器官摊开。

## 二、器官接缝：单独跑每个器官时看不到的东西

单器官 demo 的通病是只演 happy path。接缝才是真问题。capstone 在 `main()` 末尾专门打了一段 "Seam outcomes"，把单器官会藏起来的东西显式 assert 出来：

```
  permission gate : 3 allowed, 1 denied
  deny -> continue: send_external denied, loop survived and converged (see step 15 )
  memory conflict : stale fact recalled & injected, but file read won — final config:
                     {   "timeout_ms": 5000,   "retries": 3 }
  episodic log    : 9 events recorded (loop is replayable/debuggable)
```

### 接缝一：权限 deny 之后，循环怎么继续

最容易写错的接缝。一个 `deny` 在控制流里有三种处理，两种是错的：

- **崩掉**：网关抛异常 → 整个 agent 挂掉。一次 deny 等于任务失败，太脆。
- **静默跳过**：网关吞掉这个 tool_use，什么都不回 → 模型在等一个永远不来的 tool_result，**循环卡死**。这是最隐蔽的 bug，因为 happy path 测不出来。
- **回灌为错误结果**（capstone 的做法）：把 deny 包成一个 `isError: true` 的 `tool_result` 推回对话（`agent.ts` 的 GATE+ACT 段）。模型下一轮看到"这个动作不会执行"，自己决定改道。

为什么必须回灌而不能跳过？因为协议层面，模型发出的每个 `tool_use` 都**必须**在同一个 user turn 里收到对应的 `tool_result`（01、02 讲的并行工具调用约束）。少一个，下一次模型调用的消息结构就是坏的。所以 capstone 在收集结果时，allow 的和 deny 的**一起 push**，deny 的那条带 `isError`。第 15 帧 deny、第 17 帧模型读到错误结果后放弃外发并收敛——`deny -> continue: loop survived and converged` 这行就是这个接缝的活证据。

诚实标注一处：capstone 里模型"看到 deny 就乖乖放弃"是脚本写死的。真模型未必这么听话——它可能换个工具名、换个 URL 再试。这正是 04 留的 ⚡：网关挡得住这一次外发，挡不住模型在你权限边界里找别的缝。capstone 演的是"deny 不会让循环崩"这个机制，不是"deny 能根治注入"。

### 接缝二：记忆和当前上下文冲突时，取舍谁

这条接缝 capstone 是**故意种雷**演的：语义记忆里有一条 `source=tool` 的旧事实说配置已修好，世界里 `config.json` 却是 `timeout_ms: 0`。两者直接矛盾。

代码层面，RECALL 段注入这条事实时**带上了它的来源**（`Recalled fact [source=tool]: ...`）。provenance（来源标记）是循环手里唯一能用来给一条记忆打折的信号。模型最终选了文件（第 6 帧），final config 落在 `5000`——记忆输了，当前观测赢了。

但别把这当成一条可以照搬的规则。这里能解决，是因为这个冲突**有一个可验证的外部信号**（读文件、跑 verify）。"以新观测覆盖旧记忆"只在你能廉价拿到新观测时成立。一旦冲突落在没有 verifier 的地带——比如两条都是模型几轮前自己写进记忆的判断，没有文件可读、没有工具可验——你就回到了 06 留的那个 ⚡：记忆投毒今天没有通用解。capstone 演的是"provenance + 可验证观测"这一个能 work 的组合，不是宣称记忆冲突已解。

### 接缝三：计划只跟着现实走，不跟着意图走

`agent.ts` 的 advancePlan 只在工具**真的成功**时才推进 todo：`read_file` 推进第 1 步、`write_file` 推进第 2 步、**返回 `PASS` 的** `verify_config` 才推进第 3 步。一个被 deny 的工具压根到不了推进那行代码，所以**永远不会让计划虚假前进**。这就是"计划即数据"（08）的价值：计划不能跑在世界确认的前面。对照结果里 `Plan (after)` 三步全 `[x]`，且每一个 `[x]` 背后都有一帧 `act` 成功，没有一个是模型嘴上说完成。

### 接缝四：每一步都进情节日志，循环可回放

9 个事件被记进情节日志（`episodic log : 9 events recorded`）。这把 06 的记忆和 14 的追踪缝在了一起：同一个循环既吐 trace span（给人读的时间线），又 append 情节（给 agent 下一轮召回）。一条事件双重落地，所以这个 run 既可调试（像读 stack trace 一样读时间线）又可被自己回忆。`input tokens : 2434` 是各轮输入 token 的累加——注意它每轮单调上涨（341 → 426 → 503 → 547 → 617），因为**整个 transcript 每轮重发**（01 讲过的循环代价）。这也是为什么 `/compact` 和 05 的上下文工程在长任务里不是优化项而是必需项。

## 三、面试题库：按器官分组，覆盖四类问法

这是这章的就业向重点。下面按器官分组，每组给几道题，并标出问法类型——**应用**（搭产品的，关心怎么用对、怎么不翻车）、**Agent**（做 agent 框架/平台的，关心机制和接缝）、**infra**（做底层平台/runtime 的，关心隔离、调度、可靠性）、**科研**（关心开放问题和评测）。同一个器官，四类岗位问的角度完全不同，认清这点比背答案重要。

答题通则：**先讲失败模式，再讲为什么这么设计**。面试官真正想听的不是"我知道有 ReAct"，是"我知道它哪里会坏、坏了我怎么接"。

### 控制循环（01）

- **应用**：你的 agent 偶尔不收敛、一直转到超时，你会先查什么？（答：先看是 stalling 还是 drift；查有没有 max-turns 兜底；看 transcript 是不是在重复同一工具调用。）
- **Agent**：为什么模型发了 N 个并行 tool_use，你必须在一个 turn 里回 N 个 tool_result？少回一个会怎样？（答：协议要求一一对应；少一个下一次调用消息结构坏掉/模型空等。对照 capstone 的 deny 也要回灌。）
- **infra**：循环每轮重发整个 transcript，token 单调增长，你在平台层怎么兜底成本和延迟？（答：max-turns + 成本上限 + 上下文压缩 + KV/prefix 缓存；连 05、16。）
- **科研**：检测"agent 在原地打转"为什么没有通用解？（答：判定"是否有进展"需要"进展"的可机械定义，而它没有；现有招数全是领域相关启发式。这是 ⚡。）

### 工具与 MCP（02）

- **应用**：工具老被传错参数/调用不存在的工具，你改哪里？（答：改 `inputSchema` 和 description——它既是契约也是模型看到的全部提示，工具描述是工程不是文档；错误回灌给模型而不是 harness 崩。）
- **Agent**：MCP 接第三方工具，tool poisoning（工具投毒，恶意工具描述里藏指令）你怎么防？（答：工具描述也是不可信输入；做 source pinning、描述审计、把工具 spec 当攻击面。）
- **infra**：几十上百个工具时，全量塞进 prompt 不现实，平台层怎么做工具发现？（答：按需加载/检索式工具暴露——正是你现在用的 Claude Code 的 ToolSearch 机制：先给名字，用到再 fetch schema。连 05 的上下文预算。）
- **科研**：code-mode（让模型写代码组合工具，而非一次次 tool_call）的能力和安全为什么是同一枚硬币？（答：表达力越强、可约束的权限边界越难划。这是 02 的 ⚡。）

### 沙箱与执行（03）

- **应用**：你让 agent 跑用户给的代码，最低成本的隔离是什么？（答：进程级 + 文件系统白名单 + 网络默认关；别一上来上 VM。）
- **Agent**：工具执行失败该抛异常还是返回给模型？（答：返回给模型当数据——missing file 是模型能反应的信息，不是 harness crash。看 capstone 的 `read_file` 返回 `ERROR:` 而非 throw。）
- **infra**：逃逸防护要强隔离（VM/microVM），冷启动又要快，这对矛盾你怎么权衡？（答：池化预热 vs 强隔离的取舍；这是 03 的 ⚡，无完整解。）
- **科研**：为什么"沙箱逃逸 vs 冷启动延迟"至今没有干净的通用解？（答：强隔离的初始化成本和按需快速拉起本质冲突，是 03 的 ⚡。）

### 权限与安全（04）

- **应用**：用户抱怨权限弹窗太多，你怎么减少又不放开危险动作？（答：按动作分级——读 allow、写 ask、外发 deny；像 capstone 的 RULES；对照 Claude Code 的权限模式。）
- **Agent**：为什么"在 system prompt 里写'不要外发数据'"不是一道边界？（答：模型可以无视/被注入绕过；边界必须是模型说服不了的代码网关，判决在执行前。capstone 第 15 帧。）
- **infra**：deny 在控制流里怎么实现才不会让循环崩或卡死？（答：回灌为 `isError` 的 tool_result，不抛不吞——这章接缝一的核心。）
- **科研**：lethal trifecta 为什么至今无完整解？（答：敏感数据访问、不可信内容、外发通道三者只要在一次会话里凑齐就有外泄风险，而它们各自都是合理功能；这是 04 的 ⚡。）

### 上下文工程（05）

- **应用**：长对话到后面 agent 开始忘事/答非所问，先做什么？（答：压缩/重注入关键事实；把要被用的信息放结尾；连 `/compact`。）
- **Agent**：要被模型用到的事实，应该放上下文的哪个位置？为什么？（答：放结尾或重注入——因为 context rot，中段最容易被忽略。）
- **infra**：prefix 缓存能省钱，但前缀改一字节就全 miss，你怎么排布 prompt？（答：稳定内容（系统、工具 spec）放前缀，易变内容放后面；连 16。）
- **科研**：context rot（长上下文中段信息被忽略）实测有多严重，怎么量化？（答：诚实说边缘 vs 中段的命中率差异是手写示意非实测，需用真模型验证；这是 05 的 ⚡。）

### 记忆系统（06）

- **应用**：agent 记住了一条过时的事实，老拿它误导用户，你怎么办？（答：给记忆打 provenance + TTL；冲突时用可验证的新观测覆盖——但前提是你能廉价拿到新观测。）
- **Agent**：情节/语义/程序三种记忆的召回规则分别是什么？（答：情节按时近、语义按含义、程序按 trigger；看 capstone 三个 store 的 recall 方法。）
- **infra**：记忆召回在每轮决策前注入，平台怎么控制注入量不撑爆上下文？（答：相似度 floor + topK + 召回预算；capstone 的 `recallByMeaning` 有相似度下限，避免召回最不相关的垃圾。）
- **科研**：记忆投毒为什么没有通用解？（答：一旦冲突落在没有 verifier 的地带，provenance 也只能打折不能判真伪；这是 06 的 ⚡，也是 capstone 接缝二的边界。）

### 检索增强（07）

- **应用**：RAG 检索回来的片段答非所问，先查 embedding 还是 chunking？（答：先看 chunk 粒度和 query 改写，再看 embedding；多数问题在切分和召回不在向量。）
- **Agent**：检索结果要不要无条件塞进上下文？（答：要过 rerank + 相关性阈值，否则等于往上下文倒垃圾，叠加 context rot。）
- **infra**：向量库更新和读一致性怎么保证？（答：索引重建期的读写一致、增量更新策略。）
- **科研**：07 列的三个开放问题是什么？（答：见该章 ⚡——检索质量评估、长文档切分、多跳检索仍无通用解。）

### 规划与推理（08）

- **应用**：复杂任务你让 agent 一步到位还是先列计划？（答：先列计划即数据，让计划可见、可验、只跟现实推进；对照 TodoWrite。）
- **Agent**：为什么 capstone 的计划只在工具成功时推进，不在模型"说完成"时推进？（答：防计划跑在世界确认前面；done 必须是事实。）
- **infra**：长程任务的计划状态怎么持久化以便崩溃恢复？（答：计划即数据 → 可序列化 → 连 15 的可恢复 workflow。）
- **科研**：Tree-of-Thought 为什么没活下来，长程信用分配为什么没解？（答：ToT 成本/收益不划算；哪一步导致最终成败的归因仍难；这是 08 的 ⚡。）

### 反思与自我纠错（09）

- **应用**：让 agent 自己 review 自己的输出，靠谱吗？（答：只在有外部验证信号时靠谱；纯自评常自我感觉良好。）
- **Agent**：反思的可靠性取决于什么？（答：取决于闭合循环那个信号的质量——能写 verifier 就写。capstone 的 verify_config 就是那个信号。）
- **infra**：加 reflection 步骤的成本怎么控制？（答：reflection 每次都是额外模型调用，要算进 token 预算，按价值触发不无脑加。）
- **科研**：没有外部验证信号时 self-correction 为什么无干净解？（答：模型自己既当运动员又当裁判，没有独立信号锚定；这是 09 的 ⚡。）

### 子 Agent 与隔离（10）

- **应用**：什么时候该拆子 agent 而不是在主循环里硬撑？（答：上下文要隔离、子任务可独立验收时；对照 Claude Code 的 Agent 子 agent。）
- **Agent**：子 agent 的上下文该怎么和主 agent 隔离/回传？（答：子 agent 独立上下文跑完只回结论，不把它的全部 transcript 倒回主上下文——省预算 + 防污染。）
- **infra**：并发子 agent 的资源/成本上限怎么管？（答：并发上限 + 每个子 agent 的 token/turn 预算。）
- **科研**：该在哪里切一刀拆子 agent，为什么没有通用解？（答：切分边界依赖任务结构，没有自动判定；这是 10 的 ⚡。）

### 多 Agent 模式与成本（11）

- **应用**：多 agent 比单 agent 强吗？（答：不一定——多 agent 是成本乘数，先证明单 agent 不行再上。）
- **Agent**：agent 间通信用什么协议？（答：目前没收敛——自由文本、结构化消息、共享 scratchpad 各有取舍。）
- **infra**：多 agent 的总成本怎么 bound？（答：全局 token 预算 + 每 agent 上限 + 收敛超时。）
- **科研**：agent 间该怎么通信，为什么还没收敛？（答：表达力 vs 可控性的取舍未定；这是 11 的 ⚡。）

### 确定性编排与 Workflow（12）

- **应用**：哪些步骤该用确定性 workflow 而不是丢给 agent 自由发挥？（答：流程固定、可枚举的步骤用 workflow；需要随机应变的才用 agent。对照 Claude Code 的 Workflow 确定性编排。）
- **Agent**：agent 和 workflow 的边界怎么划？（答：确定性的固化成 workflow 省 token 省不确定性，开放的留给 agent。）
- **infra**：workflow 引擎怎么保证步骤幂等/可重放？（答：每步幂等 + 状态外置；连 15。）
- **科研**：agent 与 workflow 的边界何时该动态切换，为什么没通用解？（答：何时从固定流程切到自由探索依赖运行时信号，无成熟判定；这是 12 的 ⚡。）

### 评测（13）

- **应用**：你怎么知道改了 prompt 之后 agent 变好了还是变坏了？（答：跑一套固定任务集算 pass rate，别凭感觉；区分 outcome 评测和 trajectory 评测。）
- **Agent**：outcome（看最终对不对）和 trajectory（看过程对不对）评测各抓什么？（答：outcome 抓结果正确性、trajectory 抓有没有走冤枉路/调错工具；capstone 的 goalReached 是 outcome 信号。）
- **infra**：评测 harness 怎么做到换真模型不用改？（答：harness 和被测对象解耦——13 的 `runSuite` 换真模型一字不改；被脚本化的只是被测对象不是测量工具。）
- **科研**：agent 评测为什么至今没有公认 benchmark？（答：任务多样、轨迹难自动判分、分数易饱和；这是 13 的 ⚡。）

### 可观测性与追踪（14）

- **应用**：线上 agent 答错了，你怎么定位是哪一步带歪的？（答：读 trace 时间线，像读 stack trace 一样找那一帧；capstone 的 17 步时间线就是这个。）
- **Agent**：一个 span 应该记哪些字段才够调试？（答：step、phase、detail、token、ok——capstone 的 `Span` 接口。）
- **infra**：怎么把这套 span 映射到 OpenTelemetry？（答：phase → span name、detail → attributes；capstone 说明里标了 OTel 映射超出范围但接口是兼容形态。）
- **科研**：从 trace 自动归因"哪一步带歪了"为什么难？（答：归因需要反事实推理（这步不这么走结果会不会变），没有通用方法；这是 14 的 ⚡。）

### 可靠性与持久化（15）

- **应用**：agent 跑一半进程挂了，重启能接着跑吗？（答：取决于状态有没有外置——计划即数据 + 情节日志可序列化就能恢复；对照 Claude Code 的可恢复 workflow。）
- **Agent**：哪些状态必须持久化才能恢复？（答：计划、情节、已完成的工具结果——capstone 这些都是普通数据结构，可直接序列化。）
- **infra**：重放时怎么保证不重复执行有副作用的工具？（答：幂等键 + 已执行标记，重放跳过已完成的副作用步骤。）
- **科研**：跑几小时/几天的长程 agent 为什么仍是难题？（答：错误累积、drift、成本失控叠加，至今无保证；这是 15 的 ⚡，和 01 的循环稳定性同源。）

### 成本与延迟（16）

- **应用**：agent 太贵，你先砍哪？（答：按 ROI——模型分级 > prefix 缓存 > 并行/流式；别拿 prompt golf 当主力。）
- **Agent**：cascade（先小模型，不行再升级大模型）省钱的代价是什么？（答：和丢正确率是同一场赌注；16 实测省 51.5% 但 pass 从 8/8 掉到 7/8。）
- **infra**：prefix 缓存折扣怎么在路由之上叠加？（答：缓存命中靠稳定前缀，路由换模型可能换前缀导致 miss，要协同设计。）
- **科研**："何时升级到大模型"为什么没通用解？（答：理想路由器要在不解题的前提下预判难度，近乎和解题一样难；信心校准本身未解；这是 16 的 ⚡。）

## 四、收束全书：整体地图 + 仍未解的开放问题清单

把 00–16 收成一张图。一个生产级编码 agent，去掉所有包装，是**六个器官 + 高阶认知层 + 生产系统层**接成的一个循环：

- **六器官（基础循环）**：循环（01）是心跳，工具（02）是手脚，沙箱（03）是手脚的护栏，权限（04）是手脚的闸门，上下文（05）是工作记忆，记忆（06）是长期记忆。capstone 这一章就是把其中五个（循环、工具、权限、记忆、计划/追踪）接进一个 17 步的真实 run。
- **高阶认知**：检索（07）扩展记忆的边界，规划（08）让多步可控，反思（09）闭合纠错循环，子 agent（10）和多 agent（11）做上下文隔离与分工，workflow（12）把确定性的部分固化。
- **生产系统**：评测（13）回答"变好还是变坏"，可观测（14）回答"哪步带歪的"，可靠性（15）回答"挂了能不能续"，成本（16）回答"一次多少钱"。

而这本书从第一章就承诺：诚实地告诉你哪里还没有答案。把各章的 ⚡ 汇成一张清单，这是 2026 年中**仍无通用解**的开放问题地图——你设计 agent 时把它们当成已知的、暂时绕不过去的约束，而不是哪天能彻底修好的 bug：

| 章 | 开放问题（⚡） | 一句话现状 |
|---|---|---|
| 01 | 长程循环稳定性 | stalling/drift 无可机械判定的"进展"定义，全靠领域启发式压低失败率 |
| 02 | code-mode 的能力 vs 安全 | 表达力越强、可约束的权限边界越难划，无公认答案 |
| 02 | MCP tool poisoning | 工具描述也是不可信输入，无 drop-in 防御 |
| 03 | 逃逸 vs 冷启动 | 强隔离的初始化成本和按需快速拉起本质冲突，无干净解 |
| 04 | lethal trifecta | 敏感数据+不可信内容+外发通道三者各自合理、凑齐即危险，无完整解 |
| 05 | context rot | 长上下文中段被忽略，严重程度的实测需用真模型，无定论 |
| 06 | 记忆投毒 | 没有 verifier 的冲突地带，provenance 只能打折不能判真伪 |
| 07 | 检索质量/切分/多跳 | 三个子问题均无通用解 |
| 08 | ToT 与长程信用分配 | 哪一步导致成败的归因仍难 |
| 09 | 无验证信号的 self-correction | 模型自当裁判，无独立信号锚定 |
| 10 | 在哪切子 agent | 切分边界依赖任务结构，无自动判定 |
| 11 | agent 间通信 | 表达力 vs 可控性取舍未定，协议未收敛 |
| 12 | agent/workflow 动态边界 | 何时切换依赖运行时信号，无成熟判定 |
| 13 | agent 评测 benchmark | 轨迹难自动判分、分数易饱和，无公认基准 |
| 14 | trace 自动归因 | 归因需反事实推理，无通用方法 |
| 15 | 几小时/几天长程 agent | 错误累积+drift+成本失控叠加，无保证 |
| 16 | 何时升级大模型 | 不解题预判难度近乎和解题一样难，信心校准本身未解 |

注意这些 ⚡ 不是孤立的——它们成簇。01 的循环稳定性、15 的长程 agent、08 的信用分配是同一个"长程"问题的三个切面；02 的 code-mode 安全、04 的 trifecta、03 的逃逸是同一个"给模型多大权力"问题的三个切面；06 的记忆投毒、09 的无信号自纠、13 的评测是同一个"没有外部 ground truth 时怎么判对错"问题的三个切面。看清这三簇，比记住十七条单点更有用——它们告诉你 agent 工程真正的硬骨头在哪。

## 小结

- **bug 在接缝，不在器官。** 单器官 demo 都干净，把它们接成一个 agent 才暴露真问题：deny 后循环怎么续、记忆和观测冲突信谁、计划会不会跑在现实前面。capstone 一条 17 步 trace 把这些接缝逐帧摊开。
- **deny 必须回灌不能吞。** 权限拒绝包成 `isError` 的 tool_result 推回对话，循环才能活下来收敛（第 15 帧 deny、第 17 帧恢复）。吞掉会卡死，抛出会崩。
- **记忆冲突靠可验证观测解，但有边界。** capstone 用 provenance + 读文件 + verify 让旧事实输给当前观测；可一旦冲突落在没 verifier 的地带，就回到 06 的 ⚡，无通用解。
- **done 是事实不是 vibe。** 计划只在工具真成功时推进，`goal reached` 对世界重新 parse 得到，不是模型自报。
- **面试先讲失败模式再讲设计。** 同一器官，应用/Agent/infra/科研四类岗位问的角度完全不同；面试官要的是"哪里会坏、坏了怎么接"，不是术语清单。
- **全书的诚实结论是一张 ⚡ 清单。** 长程、给模型多大权力、无 ground truth 判对错——三簇硬骨头至今无通用解，把它们当约束而非待修 bug。
