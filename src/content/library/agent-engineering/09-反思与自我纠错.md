> 阶段四 · 自我改进 · 第 9 章
>
> 上一章把任务拆对了。这一章处理拆对之后**第一次还是做错**。

一次 LLM 调用是一次性猜测。让它变好最便宜的办法，不是换更大的模型、也不是把 prompt 写得更长，而是套一个**循环**：生成 → 评判 → 如果不过，把失败原因喂回去重试。这就是 Reflexion（Shinn et al. 2023，「让 agent 反思自己失败的尝试再重试」）。

但这个循环有一个致命的前提，绝大多数「self-correction」教程都把它糊过去了：**循环靠什么信号闭合**。换个信号，同一个循环代码，行为天差地别——一种会稳稳收敛，另一种会原地打转，还有一种会信心十足地把错误答案发出去。本章配套代码 `examples/agent-from-scratch/src/stage09-reflection.ts` 把这三种行为做成三个 demo，全部离线、确定性、可复现，跑 `npx tsx src/stage09-reflection.ts` 当场看输出。下面引用的每个「第几轮通过」「跑了几轮」都是代码算出来的，不是我narrate的。

诚实标注一句免得你拿 toy 当生产用：stage09 的模型是 `MockLLM`——脚本化的状态机，每一轮回什么 slug 是写死的（`src/stage09-reflection.ts:211-251`）。**机制是真的，模型是假的**。但本章要证的恰恰是机制层面的结论，跟模型聪不聪明无关——把 mock 换成真模型，循环和校验器的代码一字不动（文件头 `:32-35` 已说明这是故意的）。运行时打印的 `[llm] ANTHROPIC_API_KEY not set — running offline mock` 就是这个意思。

## 两种评判者：verifier 和 critic，类型一样，可信度天差地别

整章的判断挂在一个极易被糊掉的区分上（`src/stage09-reflection.ts:9-14`）：

- **verifier**（验证器）：用一个**存在于模型之外**的信号评判——单元测试、编译器、正则、校验和。这是 ground truth（地面真值，客观对错），跟它没法argue。
- **critic**（评论者）：用模型**自己的意见**评判——「这看起来对吗？」。没有 ground truth，可以被argue，**包括被它自己argue**。

stage09 选的任务是「给标题生成 URL slug，满足四条格式规则」（`:54-59`）：

```
rule 1: only lowercase letters, digits, and single hyphens
rule 2: no leading or trailing hyphen
rule 3: no two hyphens in a row
rule 4: total length between 8 and 40 characters
```

选它不是随便选的。这四条规则可以用一个正则 + 长度检查**零判断**地裁出对错（`verifySlug`，`:73-80`），这正是它配当 verifier 而非 critic 的原因——没有任何「看起来对」的空间。规则故意做得琐碎（模型第一次大概率对个九成），循环才有真东西可改。`verifySlug` 是纯代码、里面没有模型——这就是全部的点：它能跟模型唱反调，而模型说服不了它。

关键的坑藏在类型里。看循环的签名（`:108-112`）：

```ts
type Judge = (candidate: string) => VerifierResult;
```

verifier 和 critic **都是 `Judge`**。类型系统**区分不了** ground truth 和 opinion——只有运行时行为能。这就是为什么一个调用方可以把错的那个 judge 接进去而毫无察觉：编译器不会拦你。循环里有个 `trustworthy` 布尔（`:114-118`），但它**不改控制流**——循环对两种 judge 一视同仁，它只用来给结果打标签（「verified」还是看起来一样的「accepted」）。这映射的是现实：运行时没有任何东西强迫 critic 的「looks good」为真，只有**你对这个 judge 是什么的认知**能。

循环本身就一个函数（`reflect`，`:119-168`），靠参数 `judge` 参数化。整个实验就是换 judge：传真 verifier 它收敛，传 same-model critic 它不收敛。循环代码三个 demo 完全一样——**问题永远不在循环，在信号**。还有一个细节值得点出：失败的尝试连同反馈一直留在 `messages` 里（`:127-130`），所以第 N+1 轮看得见前面每一次错。没有这个累积，Reflexion 就退化成「闭眼重新抽样」——这跟第 5 章上下文工程是一条线：反思的记忆就是那段越滚越长的 transcript。

## ✦ Demo 1：verifier 闭环——有 ground truth 就收敛

第一个 demo 用真 verifier 驱动循环（`:282-289`）。mock 第一轮故意违反规则 1 和 3（大写 + 双连字符），第二轮读了反馈把两个都改对（`makeConvergingMock`，`:211-219`）。这是反思的 happy path：差一点点的第一次猜测 + 精确反馈 + 一次修复。运行输出：

```
[1] VERIFIER-driven loop (ground-truth signal) — expect convergence:
    (llm=mock)
  round 1: [fail] "My--First-Post"  → rule 1: ...; rule 3: no two hyphens in a row
  round 2: [PASS] "my-first-post-2026"
  outcome  : verified (on round 2 of 2)
  final    : "my-first-post-2026"
    cross-check: verifySlug("my-first-post-2026").ok = true
```

**第 2 轮通过**，而且最后那行 cross-check 是独立复核：用真 verifier 再验一遍 final，证明「verified」不是循环自己说了算（`:288-289`）。这里有两个设计决策值得讲为什么：

第一，`verifySlug` **一次返回所有违反的规则**（`:73-79` 把 failures 全 push 进数组），而不是只报第一条。如果一轮只报一个错，四条全错的 slug 要四轮才修完，每轮修一条像剥洋葱。一次全报，一次重试就能全修——这是把「反馈的信息密度」直接做进 verifier 的返回值。

第二，反馈是**逐规则点名**的（`:154-160` 把每条 failure 原样喂回去），不是「make it better」。文件头那句话点破了机理（`:51-53`）：模糊反馈（「再改改」）是 critic 循环震荡的根因；精确反馈（「rule 3: 不能两个连字符连一起」）是 verifier 循环收敛的根因。同一个循环，反馈精度决定收不收敛。

把这套对照你正在用的 Claude Code 就清楚了：当它跑 Bash 在沙箱里执行测试、读编译器报错、跑 lint，那是 verifier 闭环——失败信息是客观的、可执行的，下一轮能精确照着改。本项目 `npm run build`（zod schema 校验）和 `npm run lint:unused`（死代码扫描）就是这种 ground-truth verifier：它们能跟生成的代码唱反调，而代码说服不了它们。

## ✦ Demo 2：震荡——没有可用信号，循环原地打转

第二个 demo 演没有可用信号时会发生什么（`:291-300`）。mock 在两个**各自都失败**的候选之间来回横跳：`my-post`（太短，违反规则 4）和 `my-first-post-`（尾部连字符，违反规则 2），永远落不到一个合法值上（`makeOscillatingMock`，`:229-237`）。运行输出：

```
[2] OSCILLATION (generator cannot act on feedback) — expect NO convergence:
  round 1: [fail] "my-post"  → rule 4: total length between 8 and 40 characters
  round 2: [fail] "my-first-post-"  → rule 2: no leading or trailing hyphen
  round 3: [fail] "my-post"  → rule 4: ...
  round 4: [fail] "my-first-post-"  → rule 2: ...
  round 5: [fail] "my-post"  → rule 4: ...
  outcome  : exhausted (5 rounds, none passed)
  final    : "[exhausted: no candidate passed]"
    diagnosis: 5 rounds cycled through only 2 distinct candidates → the loop is rewriting in circles, not improving.
```

**5 轮，0 通过，只在 2 个 distinct 候选之间打转**。这里的诊断数字是代码算的（`:296`，用 `Set` 数不重复候选），不是我看出来的——这正是「震荡」可机械检测的判据：rewrite 了 N 轮但 distinct 候选数远小于 N，就是在画圈而不是在改进。

这个 demo 是用真 verifier 跑的，目的只是**把不收敛清楚地显示出来**（verdict 每轮都 fail）。但它要演示的成因是**生成器没法 act on feedback**——这恰恰是模糊 critic 制造出来的局面：critic 一会儿说「太短了加长点」，加长后又说「太长了/不好看」，generator 跟着翻来覆去。

那么**什么时候反思反而更差**？震荡就是答案之一。一次性猜测至少给你一个确定的（错误的）答案;开了反思循环但信号不可用，你花了 5 倍 token，最后什么都没拿到——`final` 是 `[exhausted]`。这里有个值得点出的诚实设计：循环耗尽时返回的是**老实的失败标记**，而不是把最后那个仍然破的候选包装成答案交出去（`:163-167`）。一个把不收敛藏在自信 final string 后面的反思循环，正是这整章要反对的反模式。

收敛的工程对策也在这个 demo 的反面：**给重试设硬上限**（这里是 5，`:285` `:294` 的 `maxRounds`），并且**检测画圈**（distinct 候选数停滞就提前停，别真跑满）。延续第 1 章控制循环里那个「循环必须有刹车」的原则——反思循环是 agent loop 的一个特例，同样需要 budget 和终止条件，否则就是烧钱的永动机。

## ✦ Demo 3：自我说服——critic 是同一个模型，看着像成功，其实不是

这是三个里最危险的一个，因为它**长得像成功**。第三个 demo 把一个第一次就自信地错的 generator（`makeConfidentlyWrongMock`，输出 `My-Great-Title`，违反规则 1 大写，`:246-251`）配上一个 same-model critic（`makeSelfCritic`，`:189-198`）。这个 critic 被脚本化成**永远批准**——因为这就是 same-model critic 对自己自信而错误的输出的经验行为：它给自己背书。运行输出：

```
[3] SELF-PERSUASION (critic = same model, no ground truth) — looks like success, isn't:
  round 1: [PASS] "My-Great-Title"
  outcome  : accepted (on round 1 of 1)
  final    : "My-Great-Title"
    cross-check: critic said PASS, but verifySlug("My-Great-Title").ok = false
    reality    : the shipped answer actually breaks 1 rule(s): rule 1: ...
```

**第 1 轮就「通过」**，干净利落，零重试。但最后两行 cross-check 是致命一刀（`:308-313`）：critic 说 PASS，真 verifier 说 `ok = false`，发出去的答案**实际违反 1 条规则**。注意 outcome 是 `accepted` 不是 `verified`——这个词是故意选的，**它不暗示正确**（`:101-105` 的注释专门说明）。

为什么 same-model critic 会自我说服？因为**生成答案的分布和评判答案的分布是同一个**。模型对 `My-Great-Title` 输出高概率，是因为它的内部表示认为这是好 slug;你再问同一个模型「这个 slug 好吗」，它的内部表示**还是那个**，自然说好。让一个人检查自己刚写的代码有没有 bug，他大概率看不出来——不是因为他蠢，是因为让他犯错的那个盲区，同样会让他漏掉这个错。Critic = generator 时，反思不是在「检查」，是在**给同一次猜测盖个自信的章**。

这就引出本章的**核心判断**，也是我希望你只记住这一句的话：**有客观验证信号时反思才可靠**（编译器/测试/校验器/正则/类型检查器——任何模型之外、能跟模型唱反调的东西）；**没有时，critic 和 generator 是同一个模型，反思循环退化成一台把猜测洗成自信的机器**。stage09 结尾的 lesson 把三个 demo 的数字摆在一起说的就是这件事（`:316-321`）：

```
lesson:
  [1] verifier had ground truth → converged in 2 round(s), cross-check holds.
  [2] no usable signal → 5 rounds, 0 passes, cycled 2 candidates (never converges).
  [3] same-model critic → 'accepted' in 1 round, but verifier proves it ships 1 broken rule(s).
  reflection is only as good as the signal closing the loop. Wire a real verifier.
```

工程结论非常具体：**别把 self-critique 当验证**。如果你能写出 verifier（测试能跑、schema 能校、API 能调真实端点返回 4xx），就用它闭环，self-critique 至多当个提前过滤的廉价启发式。回到第 4 章的取舍——你愿意为正确性付多少代价：跑一遍测试套件的几秒和 CI 成本，换的是「accepted ≠ correct」这个隐性 bug 不被发出去。这笔账几乎总是划算的。

## ⚡ 开放问题：没有外部验证信号时，self-correction 至今无干净解

stage09 能把结论演得这么干净，是因为 slug 任务**有** verifier。但 agent 干的很多活根本没有：

- 「这封邮件语气得体吗」「这段总结忠于原文吗」「这个产品命名好不好」——**没有编译器**。
- 「这个架构方案合理吗」——能编译不代表设计对，真正的判据要等几个月后才显现。
- 「这个对外宣称的事实准确吗」——除非你接一个外部检索 + 核查管线（那本身就是在**造一个 verifier**），模型自己核对自己，就是 Demo 3。

学界目前的探索方向（debate / self-consistency 多次采样投票 / LLM-as-judge / 多模型互评）本质都在试图**逼近一个外部信号**：让另一个模型、另一次采样、或一群模型来当那个「模型之外的东西」。它们能在某些任务上把准确率往上抬，但都没有摆脱根问题——评判者和被评判者**共享同一类盲区与同一批训练偏差**。换个模型评，只是把「自我说服」摊薄成「同类相互说服」，没有变成 ground truth。本项目 MEMORY 里记着一条同源教训（`feedback_mechanical_vs_generative.md`）：机械可检测的维度清零之后，剩下的质量差距落在不可机械化的区域（Idea / Voice）——那正是没有 verifier 的地带，再扩检查清单也够不到。

**我的明确判断：无外部验证信号下的 self-correction，今天没有通用的干净解，仍在研究。** 不是工程没接好，是问题本身的结构——你要么找到/构造一个真正外部的信号（哪怕昂贵：人工标注、真实环境执行、第三方核查），要么诚实接受这部分输出的「accepted」永远只是 accepted，不是 verified。把它当 verified 上线，就是 Demo 3 在生产里复现。下一章（10 子 agent）会看到，多 agent 协作里这个问题会被放大——多个 critic 子 agent 不会凭空造出 ground truth，搭不好只是让自我说服变成集体说服。

## 这一章的最小内核

- 反思 = 生成 → 评判 → 把失败精确喂回 → 重试，失败的尝试要留在 context 里累积（`reflect`，`src/stage09-reflection.ts:119-168`）。
- verifier 和 critic 在类型上不可区分（都是 `Judge`），可信度只由运行时决定——**接错信号编译器不拦你**（`:108-118`）。
- 有 ground truth → 第 2 轮收敛，cross-check 成立（Demo 1）。
- 无可用信号 → 5 轮 0 通过、只在 2 个候选间画圈，必须靠硬上限 + 画圈检测刹车（Demo 2）。
- same-model critic → 第 1 轮「accepted」，但真 verifier 证明它发出去了 1 条破规则（Demo 3，最危险，因为像成功）。
- 一句话：**反思的可靠性 = 闭合循环那个信号的质量**。能写 verifier 就写；写不出的地带，self-correction 无干净解（⚡）。
