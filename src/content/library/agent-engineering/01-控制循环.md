把"AI agent"这个词从你脑子里删掉一分钟。它不是一种新模型，不是某个具备"自主性"的智能体，也不需要任何框架。它是这段代码：

```ts
const messages = [{ role: 'user', content: userInput }];
for (let turn = 0; turn < maxTurns; turn++) {
  const assistant = await llm.generate({ system, messages, tools });
  messages.push({ role: 'assistant', content: assistant.content });
  if (assistant.stopReason !== 'tool_use') return assistant;  // 模型说完了
  // 否则：跑模型要的工具，把结果塞回去，再循环
  messages.push({ role: 'user', content: await runTools(assistant.content) });
}
```

一个 `while` 循环，循环体是一次 LLM 调用，调用结果里如果有"我想调工具"的信号就执行工具、把结果喂回去，否则退出。这就是 agent 的全部骨架。本章配套代码 `examples/agent-from-scratch/src/stage01-loop.ts` 的 `runAgent` 函数就是它的完整、可跑版本——60 行,没有依赖,`npm run stage01` 直接出结果。本书后面所有章节（工具、沙箱、权限、上下文压缩、记忆）都是往这个循环里加东西,所以这一章是地基:你得先彻底看清这个循环长什么样、在哪儿会坏。

下面所有讨论都绑定这份代码。它故意只实现了 Anthropic Messages API 的一个极小子集（见 `examples/agent-from-scratch/src/core/types.ts`），既能接真实模型也能跑离线 mock——没有 API key 时跑一个脚本化的假模型,机制全是真的,只有"模型说什么"是写死的。这样你能在零成本、确定性的环境里反复观察循环行为。

## Agent vs Workflow:控制流之刀

这是整章最该先想清楚的一刀,也是业界最被滥用的一对词。判据只有一句话:**下一步由谁决定?**

- **控制流写死在代码里** = workflow(工作流)。"先调 A,拿结果做判断,走 B 或 C,再汇总"——这套分支是工程师写的 `if/else`,模型只在每个节点里干一件被规定好的事。
- **下一步由模型决定** = agent。代码不知道接下来会调哪个工具、调几次、什么时候停;模型每一轮看着当前对话自己决定,harness(宿主程序)只负责忠实执行模型的决定并把结果还回去。

你现在正在用的 Claude Code 就是活体对照。它的 **Workflow 工具**是确定性编排:你把若干 stage 串成固定的 DAG(有向无环图),哪步先跑、哪步依赖哪步是写死的,可恢复、可预测——那是 workflow。而 Claude Code 主循环本身是 agent:它读你的话,自己决定先 Grep 还是先 Read、要不要起 subagent、什么时候收手回话。同一个产品里两种范式并存,不是谁取代谁。

为什么这刀重要?因为它直接决定**可靠性的来源**。Workflow 的可靠性来自结构——路径是你画的,坏了能在图上指出哪条边错了。Agent 的可靠性来自模型的判断力,而判断力是概率性的、会漂移的。**能用 workflow 解决的别上 agent**:你把决策权交给模型,就同时交出了可预测性。Agent 只在"分支空间太大、写不出穷举的控制流"时才划算(比如"修复这个 bug"——你没法预先写出所有可能的排查路径)。这是贯穿全书的取舍,不是技术细节。

在 `runAgent` 里,这把刀体现为一件事:循环体里**没有任何业务分支**。代码不关心模型调的是 `word_count` 还是别的工具,不关心调用顺序,唯一的控制逻辑是"`stopReason` 是不是 `tool_use`"。决策权 100% 在模型手里,harness 只是个忠实的执行器。这正是它是 agent 而非 workflow 的代码层证据。

## ✦ ReAct 不是 prompt 技巧,是状态机

你大概见过 ReAct(Reasoning + Acting,推理-行动)被当成一种"提示词写法"介绍:让模型先写 Thought、再写 Action、再写 Observation。那是表象。ReAct 的本质是一个**状态机**,而 `runAgent` 的循环结构就是这台状态机的实现。

看 mock 模型脚本化的两轮(`stage01-loop.ts` 末尾),它精确对应 ReAct 的一个回合:

```ts
// turnIndex === 0: Thought + Action
return { content: [
  { type: 'text', text: '我先数一下词数。' },                    // Thought
  { type: 'tool_use', id: 'call_1', name: 'word_count', input: {...} }, // Action
]};
// turnIndex >= 1: 读上一轮的 tool_result(Observation),给出答案
```

状态机的三个状态:

1. **Thinking/Acting**——模型那一轮的输出。可能纯思考(text)、可能带工具请求(tool_use),可能两者都有(一句推理 + 几个并行工具调用)。这一步由 `llm.generate` 产生。
2. **Observing**——harness 执行工具,把结果作为 `tool_result` 塞回 `messages`。这是 `runAgent` 里那段 `for (const block of assistant.content)` 循环干的事:对每个 `tool_use` 块找到对应实现、执行、收集结果。
3. **状态转移**——由 `stopReason` 驱动。`tool_use` → 回到状态 1 再转一圈;其它 → 终止。

关键洞察:**ReAct 不是你"提示"模型去做的,是这个循环结构强制出来的**。模型每一轮只能产出 text 或 tool_use;harness 看到 tool_use 就必然进入 Observe 再回到模型。Thought-Action-Observation 的节奏是数据结构和循环逼出来的,不是 prompt 措辞求来的。这就是为什么把 ReAct 当 prompt 技巧会踩坑——你以为靠措辞能让模型"更会用工具",实际决定它行为的是 harness 的状态转移规则。

注意一个工程细节:并行工具。模型一轮里可以发多个 `tool_use` 块,代码把它们**全部执行后作为一个 user 轮一次性塞回**(`messages.push({ role: 'user', content: results })`),而不是一个工具一个来回。这是 Messages API 的硬约束——一轮里所有 tool_use 必须在下一轮一起被回答。漏掉这点,接真实模型时会直接报错。

## ✦ 停止条件:三种设计,三种坑

agent 是循环,循环就得有出口。这是 harness 设计里最容易被忽视、也最容易出事的部分。`runAgent` 实现了两类出口,加上一类隐式的,一共三种停止条件,每种都有自己坏的方式。

### 1. max-turns(硬上限)——防无限循环

```ts
for (let turn = 0; turn < config.maxTurns; turn++) { ... }
// 循环跑完没返回 = 撞上限
return { answer: '[stopped: max turns reached]', turns: config.maxTurns, stopReason: 'max_turns' };
```

这是**整个循环里最重要的一个可靠性旋钮**,代码注释里写得很直白:"防止一个永不停止调工具的模型无限计费"。失败模式叫**无限工具调用**:模型陷进"调工具→看结果→再调工具"的怪圈,永远不发 `end_turn`。没有 max-turns,这就是一张无上限的账单 + 一个挂死的进程。

设计上有两个讲究:

- **默认值要低**(配套里默认给的小)。宁可早停返回一个"我没搞定"的标记,也别让一个跑飞的 agent 烧钱。上限是给坏情况兜底的,不是给正常情况用的。
- **撞上限要返回标记而不是抛异常**。代码返回 `stopReason: 'max_turns'` 而不是 `throw`,把"降级还是失败"的决定权留给调用方——也许它想拿已有的部分结果做个体面的回复,也许它想重试。harness 的职责只有一条:保证循环永不真的跑无限。

坑:max-turns 设太低会**误杀**正常的长任务(一个需要 8 步的合理任务被 6 步上限截断);设太高又失去兜底意义。这个值没有放之四海的默认,得按任务复杂度调——而你常常事先不知道任务要几步,这本身就是麻烦的开始(见文末开放问题)。

### 2. 模型 end_turn(正常退出)——防提前终止

```ts
if (assistant.stopReason !== 'tool_use') {
  return { answer: textOf(assistant.content), ..., stopReason: 'answered' };
}
```

这是**正常出口**:模型觉得自己说完了,发出 `end_turn`(或 `max_tokens` 等任何非 `tool_use` 的信号),循环退出。注意代码写的是 `!== 'tool_use'`,而不是 `=== 'end_turn'`——任何"不是工具请求"的停止原因都当正常结束处理。这是个有意的稳健选择:你不想因为漏判某个停止原因而把循环卡在那儿。

坑:**提前终止(premature stopping)**。模型可能在任务没真正完成时就发 `end_turn`——它"觉得"答完了,但其实漏了一步。这是 agent 最隐蔽的失败之一,因为循环正常退出、没有任何报错,你只是悄悄拿到了一个不完整的答案。harness 层对此无能为力:循环只认 `stopReason`,它无法判断模型"是真完成了还是自以为完成了"。这个判断属于后面章节(任务验证、reflection)要补的能力,但要先在这儿认清:**正常退出 ≠ 正确退出**。

### 3. 无 tool_use(隐式空转)——防原地打转

第三种情况藏在前两种之间:模型既不调工具、也没真把活干完,只是反复输出文字、原地打转。代码层面它会走 end_turn 分支正常退出(所以不会无限循环),但语义上是**空转**——花了 token,没产出。

更恶劣的变体是模型反复调**同一个工具、同样的参数**,或者在两三个工具间来回**震荡**(oscillation),每轮都"有进展"的样子,实则在原地绕圈,直到撞上 max-turns。`runAgent` 这种最小循环对此**完全没有防御**——它忠实执行模型的每一个决定,哪怕这个决定是第 5 次重复第 1 轮的动作。检测空转/震荡需要额外的状态(记住历史动作、查重复),那是循环之外的事。这里要建立的认知是:**最小 harness 只保证"不会真的跑无限"(靠 max-turns 兜底),不保证"每一轮都有效"**。

## ✦ 中断与恢复:为什么循环要可暂停可续

到现在 `runAgent` 是一口气跑完的:进去,转几圈,出来。真实世界的 agent 几乎都不能这样。原因有四个,每个都要求循环可以**在两轮之间停下、之后从原状态继续**:

- **要人确认**。模型决定删文件、发邮件、花钱——harness 得在执行工具前停下来问人(第 4 章权限)。这意味着循环要能在"模型已决定、工具未执行"这个中间点冻结。
- **跑得太久**。长任务可能要几十轮、几分钟甚至更久,进程不能一直占着。
- **会崩**。机器重启、连接断了,你不想从第一轮重来。
- **要省 token**。对话太长了得压缩历史再续(下一节的成本曲线会让你明白为什么这是刚需)。

你正在用的 Claude Code 把这件事做成了产品能力:它的 **Workflow 有 resume**——一个编排跑到一半中断,能从断点续上;而你和 Claude Code 的这次会话本身,在上下文快满时会 **/compact**(压缩历史),压缩后继续对话,本质就是"在循环的两轮之间,换掉 messages 的内容再续跑"。

为什么这件事在架构上要趁早想?因为它对 harness 提了一个硬要求:**循环的全部状态必须是可序列化的显式数据,不能藏在调用栈或闭包里**。看 `runAgent` 的状态其实只有一样东西——`messages` 数组。这不是巧合,是设计:只要你能把 `messages` 存下来,就能在任何两轮之间停掉进程、之后用这个数组重建循环继续跑。这是"agent 的状态 = 对话本身"这一深层事实的第一次露面,后面记忆、上下文工程都建在它上面。一个把状态散落在局部变量里的循环,是没法暂停恢复的——这是很多自研 agent 在要加"人工确认"或"断点续跑"时推倒重来的根因。

> 配套的 `stage01` 还没实现 resume(它一口气跑完),但它已经把状态收敛成了单一的 `messages` 数组——把 resume 加上去只是"存数组 / 读数组"的事,不需要改循环结构。这种"现在不做、但结构上留好口子"的克制,本身是 harness 设计的一部分。

## 量化:为什么成本是 O(T²),不是 O(T)

把 `npm run stage01` 跑出来,看这两行真实输出:

```
input tok : 98 → 150 (grows every turn — full transcript resent)
total in  : 248 input tokens billed across the run
turns     : 2 (stop: answered)
```

(没有 API key 时模型用离线 mock,token 数来自一个稳定的字符估算器——不是真 tokenizer,但增长规律是真的;接真实模型时这些数字换成 API 返回的精确 usage。)

读这三个数:

- 第 1 轮输入 **98** token(系统提示 + 工具定义 + 用户问题)。
- 第 2 轮输入 **150** token——比第 1 轮多了 52,因为第 2 轮要把第 1 轮的助手回复 + 工具结果**全部一起重发**。
- 整轮跑下来计费的输入 token 是两轮之和:**248**。

这里藏着 agent 成本的根本真相,也是很多人对 agent 账单失算的地方:**LLM 是无状态的,每一轮 harness 都要把整个对话历史(transcript)重新发一遍**。模型不"记得"上一轮——它每次只看你这次发的 messages。`runAgent` 里 `messages` 只增不减,每轮 `llm.generate` 都把当前的完整 `messages` 传过去。

所以一个 T 轮 agent 的总输入成本不是各轮成本里的最大值,也不是 O(T)——它是**各轮输入 token 的求和**。每多一轮,被重发的历史就长一截,于是:

```
总输入 ≈ tok₁ + tok₂ + ... + tok_T
     ≈ (base) + (base + Δ) + (base + 2Δ) + ...   ← 每轮历史线性变长
     ≈ O(T²)                                       ← 求和后是平方级
```

第 T 轮单轮的输入是 O(T)(历史线性增长),T 轮求和就是 **O(T²)**。在我们这个 2 轮的小例子里这只是 98 + 150;但想象一个跑 30 轮、每轮工具结果还很大的真实 agent——重发的历史会把账单推成平方级膨胀,而且越往后每一轮越贵。

这正是第 5 章**上下文工程**要解决的根问题,这里先把伏笔点明:既然成本来自"每轮重发整个 transcript",省钱的方向就只有一个——**别每轮都重发全部**。压缩历史(/compact)、prompt caching(缓存不变的前缀,这也是为什么把系统提示和工具定义放在最前面)、把旧轮次摘要化、把可重算的信息移出对话……所有上下文工程技巧,本质都是在跟这条 O(T²) 曲线作斗争。你现在理解了曲线从哪来,后面那些技巧才不会是一堆零散的"优化建议",而是对同一个根因的不同攻法。

## ⚡ 开放问题:长程 agent 的循环稳定性

上面讲停止条件时埋了个雷:最小 harness 靠 max-turns 兜底,保证不跑无限,但不保证每轮有效。把任务拉长——20 轮、50 轮、上百轮——这个雷就炸了,而**它至今没有通用解**。

具体表现成两个互相纠缠的失败模式:

- **空转/打转(stalling)**:agent 反复调同一工具、重复同样的尝试,或在几个动作间**震荡**(oscillation),每轮看着像在推进,实则原地绕圈,直到撞 max-turns 才停。
- **目标漂移(drift)**:轮数一多,早期的目标在不断重发又被新内容稀释的长 transcript 里逐渐失焦,agent 慢慢跑偏到一个相关但不对的任务上。

为什么难?因为这不是 harness 能在循环里 `if` 一下解决的。检测"在打转"需要判断"这一步相对历史是不是真有进展"——而"进展"本身没有通用的、可机械判定的定义。各种实践里的招数都是**启发式、领域相关、各有盲区**:检测连续重复的工具调用(挡不住语义重复但参数不同的震荡)、给 agent 加 reflection/self-critique 步骤(增加 token 成本,且模型自评常常自我感觉良好)、引入独立的 critic/verifier 模型(谁来保证 critic 不漂移)、限制每个工具的调用次数(可能误杀合理的重复)。每一招都能挡住一部分,没有一招能根治,组合起来也只是把失败率往下压,压不到零。

说白了:**"agent 能不能稳定地跑完一个长程任务"目前是经验问题,不是有保证的工程属性**。这是当前 agent 研究最活跃的前沿之一(规划、世界模型、可验证奖励、过程监督等方向都在攻它),也是为什么生产里跑长程 agent 的人都会铺一层 max-turns + 成本上限 + 人工检查点——不是因为这套组合优雅,而是因为还没有更可靠的东西可用。你设计自己的 agent 时,把它当成一条已知的、暂时绕不过去的约束,而不是一个你哪天能彻底修好的 bug。
