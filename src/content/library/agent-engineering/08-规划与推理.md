先把"规划"祛魅。它不是给 agent 装一个会深谋远虑的大脑，而是一个非常具体的工程选择：**模型的"思考"是每一步现花，还是一次预付？** 第 1 章里那个 `while` 循环每一轮都调一次模型来决定下一步——那是一种规划策略（其实是"不规划"）。本章要做的，是把它换成"先一次性产出整张计划、再照着执行"，然后把两条控制流放在**同一个任务、同一个世界**上跑，看实测数字差在哪、各自怎么坏。

任务很小但有真实结构：剪一个 1.1.0 版本发布——(a) 读 changelog，(b) bump 版本号，(c) 写发布说明。其中 `bump_version` 第一次调用会因为"version registry locked"失败、重试才成功——这是逼出 replan（重新规划）的那根刺。所有代码在 `examples/agent-from-scratch/src/stage08-planning.ts`，离线可跑（`npx tsx src/stage08-planning.ts`，无 key 无网络）。诚实声明：模型是脚本化的 `MockLLM`，所以"模型会产出什么计划"是写死的；但**控制流、步数计数、成功判定、失败模式全是真的**——`steps` 和 `llmCalls` 是循环里数出来的，`success` 是拿真实世界状态 `isGoalReached(w)` 校验的，不是模型自报。

## ✦ ReAct vs Plan-and-Execute：一笔账，不是一种"更聪明"

两种策略的差别，一句话能说清，区别全在"模型在哪一刻思考"：

- **ReAct（reason + act，推理-行动）**：每一步只决定**下一个**动作，看结果，再决定下一个。没有预先的计划，适应是隐式的——"看结果，再想"。这就是 `runReActAgent`，结构和第 1 章的循环一模一样：

```ts
for (let i = 0; i < maxSteps; i++) {
  const turn = await llm.generate({ system: 'ReAct: decide ONLY the next action.', messages, tools: TOOLS });
  llmCalls += 1;                              // 每一轮都付一次模型调用
  if (turn.stopReason !== 'tool_use') break;  // 模型说做完了
  // 跑工具，把 tool_result 塞回 messages，再循环
}
```

- **Plan-and-Execute（先规划后执行）**：先一次模型调用产出**整张计划**（一个 todo 列表），然后照着执行——**每步不再调模型**，直到某步失败才重新调模型 replan 尾部。这是 `runPlanAndExecuteAgent`，它的成本公式就是 `llmCalls = 1 + replans`。

把两者放同一任务上跑，stage08 的实测对比表：

```
mode                 steps  llmCalls  replans  success
ReAct                4      5         0        true
Plan-and-Execute     4      2         1        true
```

读懂这张表，选型规则就出来了。两者都用 4 步真实工作（4 次工具执行）、都成功。差别在 `llmCalls`：**ReAct 为 4 步付了 5 次模型调用**（每步思考一次，加上最后一次"确认做完"的调用）；**Plan-and-Execute 只付了 2 次**——1 次规划 + 1 次 replan（那次 transient 失败逼出来的）。模型调用是 agent 真正的"思考成本"，也是延迟和 token 账单的主要来源。ReAct 把思考摊到了每一步，Plan-and-Execute 把思考**摊销**了。

但便宜不等于更好。代码注释把取舍钉得很死：ReAct **adapts for free but has no global view**——它白捡适应性（每步都重新看世界），但没有全局视野，容易跑偏、绕路；Plan-and-Execute **是 cheaper per step and shows a visible plan, but executes blindly until a step fails**——每步更便宜、且有一张可见的计划，但在某步失败前是**闭眼执行**的，计划一旦过时，它会照着错的往下走直到撞墙。

所以判断规则不是"哪个先进"，是**任务的确定性**：

- **步骤可预测、环境稳定、子任务弱耦合** → Plan-and-Execute。计划一次成本能摊薄到很多步上，可见计划还顺带给了用户/审计一个 artifact。典型：批处理、ETL、"按这套固定流程处理这批文件"。
- **环境多变、每步结果会改变下一步该做什么、分支空间大到写不出穷举** → ReAct。你需要的就是它"每步重看"的适应性，预先的计划反正活不过两步，规划那次模型调用是白付的。典型：交互式排查、探索性任务、"修这个 bug"——你没法预先列出排查路径。

这跟第 1 章 Agent vs Workflow 那把刀是同一个谱系：confidence 越高（路径越确定）越往左走规划/编排，越不确定越往右走反应式。你现在用的 Claude Code 主循环本质是 ReAct——它读你的话，自己决定先 Grep 还是先 Read，不预先画图；而它的 Workflow 工具是确定性编排，那是 Plan-and-Execute 的极端形态（连 replan 都没有，纯固定 DAG）。

## ✦ 推理模型改变了什么：规划被吸进了模型内部

上面那笔账有个隐含前提：模型一次调用就吐一个动作或一张计划，"思考"是不可见的、瞬时的。o1/o3/R1 这一代**推理模型**（reasoning model，会在出答案前先生成一长串内部思维链再作答）把这个前提改了。它们的卖点是 **test-time compute**（推理时算力）——同一个模型，给它更多"想"的预算（更长的内部 thinking token），答案质量随之提升。换句话说，**一部分"规划"从你的 harness 里被吸进了模型内部**。

这对本章的两条控制流意味着什么，要分清楚：

- **ReAct 那一侧**：每步的"reason"原本是你靠 prompt 逼模型写出来的 Thought。推理模型把这步内化了——你不再需要"先写思考再行动"的提示工程，模型自己会在内部把这步想深。**外部不需要显式 plan 的场景，因为内部 thinking 已经在做局部规划。**
- **Plan-and-Execute 那一侧**：你仍然可能想要一张**外部、可见、可编辑**的计划——但产出它的那次模型调用，现在可以是一次 extended thinking（扩展思考），让模型在内部把全局推理做透再吐出 todo 列表。计划质量上去了，但**那次调用的延迟和 token 成本也显著上去了**。

判断规则因此多了一维：**何时让模型内部 extended thinking 规划，何时要外部显式 plan？**

- 任务的全局结构需要被**人看见/审计/干预**（比如一个会改十几个文件的重构，用户想先 review 计划再放行）→ 必须外部显式 plan，哪怕产出它用了 extended thinking。计划要落成 artifact，不能只活在模型脑子里。
- 任务的推理是**自洽闭环、不需要人介入、也不需要在多步间共享**（一道数学题、一段代码生成）→ 让 extended thinking 在内部规划，外部一个动作就够，别再套一层 Plan-and-Execute 的脚手架。
- 还有一类反直觉的：推理模型让 **ReAct 每步变贵了**。原本 ReAct 每步一次轻量调用，现在如果每步都触发一长串 thinking，那张对比表里"ReAct 付 5 次调用"的每一次都不再便宜。**给 ReAct 配推理模型时，要按需控制 thinking 预算**——不是每步都值得深想。Claude Code 里 thinking 是可调档的（普通 / think / think harder / ultrathink），就是这个旋钮的产品化：把"这步该想多深"交还给调用方，而不是固定全开。

一个常被忽略的代价：推理模型的 thinking token 通常**不进下一轮的上下文**（用完即弃），所以它不会像普通对话那样无限堆积 context——但它确实计费、确实增加延迟。第 5 章讲的"把窗口当预算"在这里要再加一笔：thinking 是一笔**看不见但要付钱**的预算。

## ✦ plan 是 artifact，不是 vibes

stage08 的核心论点（写在文件头）：**the plan is data, not vibes**。计划必须是一个显式的数据结构，不是藏在模型 hidden state 里的某种"打算"。代码里它就是一个 `Todo[]`：

```ts
type TodoStatus = 'pending' | 'in_progress' | 'done' | 'failed';
interface Todo {
  id: number;
  what: string;
  action: { tool: string; input: Record<string, unknown> };  // 动作钉在 todo 上
  status: TodoStatus;
}
```

把计划做成一个可检视的数组，而不是模型的隐藏状态，换来三件**否则做不到**的事：

1. **能渲染给人看**。`renderPlan` 把它打印成 `○/◐/●/✗` 的清单——这正是 Claude Code 的 **TodoWrite** 工具在你屏幕上画的那个勾选列表。它不只是 UI 糖：用户能在 agent 跑偏时看见"它打算干什么"，在执行前喊停或改计划。计划可见 = 可干预。
2. **能检测某步失败**。`status` 字段让"step 2 failed"成为一个可判定的状态，而不是要去模型的自然语言里猜"它是不是卡住了"。
3. **能确定性地 replan**。失败时保留 `done` 的步骤、只替换尾部，整个过程是代码控制的，不靠模型"记得"自己做到哪了。

还有一个容易滑过去但很关键的设计：`action` 字段把"具体要调什么工具、什么参数"**钉在了 todo 上**。这正是 Plan-and-Execute "每步不调模型"的底气——执行时动作直接从 todo 读出来，不用再问一次模型。这是它省钱的根，也是它的风险根：动作是规划那一刻定下的，世界变了它也不知道。

**replan 触发条件**，代码里只有一个、且很硬：**某步执行失败**（`result.isError`）。注意它**不**在"计划看起来过时了"或"模型改主意了"时 replan——那些都是模糊信号，会让 replan 失控。失败是个清晰的、可判定的边界，replan 就钉在这个边界上：

```ts
if (result.type === 'tool_result' && result.isError) {
  t.status = 'failed';
  if (replans >= maxReplans) return { ... note: 'hit replan budget — gave up' };  // 见下节
  replans += 1;
  const replanTurn = await llm.generate({ system: 'Replan the remaining steps given the failure.', ... });
  // 保留 done 的步骤，把 tail 换成新计划，重新从尾部顶端执行
}
```

那个 `bump_version` 第一次失败（registry locked），就是触发了这条：模型拿到"step 2 failed"重新规划尾部、重试 bump、再写 note——这就是对比表里 Plan-and-Execute 那 `1` 次 replan 的来历。而 ReAct 面对同一次失败，根本不需要"replan"这个概念——它只是**把错误当成又一个 observation，下一轮反应一下、再调一次 bump**（代码注释：*recovers from the transient failure by simply reacting to the error and retrying*）。同一个 transient 错误，两种范式两种处理：ReAct 免费适应，Plan-and-Execute 要花一次 replan 调用。这就是"adaptivity vs cost"那笔账在失败路径上的具体兑现。

## 失败模式：规划不是免费的

happy path 上两者都 4 步成功。但规划这套机制有两个特征性的烧钱/死循环方式，stage08 各做了一个 demo。

### 过度规划：给一步的活排五步的计划

任务只需要"读一下 changelog"——一步就够。但一个"总是先规划"的策略会产出这个：

```
○ [1] analyze the request (pending)
○ [2] gather context (pending)
○ [3] read the changelog (pending)
○ [4] verify the read (pending)
○ [5] summarize findings (pending)
planned 5 steps for a 1-step task → overplanning tax = 4 redundant steps
```

实测 **overplanning tax = 4 个冗余步骤**（`overPlan.length - NEEDED_STEPS`，这里 5 - 1）。这个数是代码算出来的，但 `NEEDED_STEPS = 1` 是人标的 ground truth，所以这个"税"是**示例性的**，不是从真实模型测出来的分布——按硬约束如实标注。

为什么它真的疼，而不只是难看：每个冗余步骤都是**用户要读的 token** + （如果真执行了）**一次工具调用**。"analyze the request / gather context / summarize findings"这种步骤是纯仪式——它让计划看起来很负责，实际一行活没干。一个固定的"always plan"策略，会在**每一个琐碎请求**上都交这笔税。

处方：**按任务复杂度 gate 规划**（smarter agents gate planning on task complexity）。琐碎任务根本不该进 Plan-and-Execute 这条路——它应该一步反应式做掉。这又绕回选型那张表：选错范式（给一步的活上规划框架）本身就是过度规划的根因。所以"先判断任务确定性再选 ReAct/Plan"不是纸上谈兵，它直接决定你交不交这笔税。

### replan 震荡：同一张坏计划一遍遍重发

更危险的是 replan 进死循环。这次 `bump_version` 的目标值是 `"BAD"`（不合法 semver），是**永久失败**——重试多少次都不会成功（代码里这条检查刻意放在 transient 计数之前，让 doomed 的值永远 doomed）。模型又"学不会"，每次 replan 都重发同一张坏计划：

```
[step 1 FAILED] invalid version "BAD", expected semver
[replanned, attempt 1]  → 同一步 → FAILED
[replanned, attempt 2]  → 同一步 → FAILED
result: success=false, replans=2 (capped by budget), steps=3
note: hit replan budget — gave up (oscillation guard)
```

**没有 replan budget，这个循环永不终止**——模型一直提同一个坏步骤，failing → replanning → failing，无限计费。实测 `replans=2`（被预算 capped）、`success=false`、`steps=3`，agent 在撞够 2 次后**主动放弃**，而不是死磕。

这里有本章和第 1 章最直接的呼应：**replan budget 就是 plan 层的 `maxTurns`**（代码注释：*the budget (here 2) is the agent equivalent of stage01's maxTurns: the one knob that guarantees the run ends*）。第 1 章讲过，agent 是个循环，任何循环都需要一个保证终止的旋钮，因为模型的判断是概率性的、可能永远停不下来。Plan-and-Execute 多了一层 replan 循环，于是就多需要一个 budget。**任何会因为模型行为而循环的结构，都必须配一个外部强制的上限**——这是贯穿全书的可靠性纪律，不是某个特定机制的细节。

注意 budget 是个钝器：它保证终止，但 `success=false`——任务没完成。budget 把"无限烧钱"换成了"有界失败"，这是对的取舍（有界失败可观测、可重试、可上报），但它不解决"为什么模型一直发坏计划"。真要治本，得在 replan 时把**失败历史**喂回给模型（"你上次发的 BAD 也是这个错"），让它别再发同一张——stage08 的 mock 刻意没做这件事，就是为了让震荡裸露出来。这也提示：replan 的 prompt 工程（把什么失败上下文带进去）本身是个值得认真做的活，不是 `system: 'replan'` 一句话能糊弄的。

## ⚡ 开放问题：ToT 为什么没活下来，长程信用分配还没解

两个前沿，明确标注**目前无通用生产解**。

**Tree-of-Thoughts（ToT，思维树）为什么在生产里几乎见不到。** ToT 的想法很诱人：不要线性地想一条路，而是在每个决策点**展开多个候选**、给它们打分、回溯、搜索出最优路径——本质是把规划变成一棵树上的搜索。论文里在特定谜题（24 点、填字）上效果亮眼。但它基本没在生产 agent 里活下来，原因是工程经济学而非算法：(1) **成本爆炸**——每个节点展开 k 个候选、搜 d 层，模型调用是 O(k^d)，对照本章那张"省 3 次调用都算赢"的账表，这是数量级的反向操作；(2) **打分难**——树搜索要靠一个 value function 评估"这个中间状态有多好"，但让 LLM 给自己的中间思路可靠打分，本身就是个没解的难题（第 9 章会专门讲 self-evaluation 的天花板）；(3) **现实任务的"树"没有清晰边界**——24 点的状态空间是良定义的，"修这个 bug"的状态空间不是，你没法干净地枚举分支。结果是：生产里活下来的是**便宜的线性策略 + 一个 replan/重试旋钮**（正是本章这套），而不是漂亮的树搜索。这不是说 ToT 错了，是说它的适用区间（良定义、小状态空间、答对的价值远超搜索成本）在真实 agent 任务里很窄。

**agentic RL 的长程信用分配与 reward hacking 还没解。** 本章的规划全是**推理时**（inference-time）的——模型权重不动，靠 prompt 和控制流。另一条路线是用强化学习（RL）**训练**模型变成更好的规划者：让 agent 跑完整条多步轨迹，对最终结果给 reward，反传去优化每一步的决策。难点有二，都未解：(1) **长程信用分配（credit assignment）**——一条轨迹可能几十步，最后失败了，到底是哪一步错的？把最终 reward 公平地摊回到几十步前那个真正致命的决策上，是 RL 的老大难，轨迹越长越糊；(2) **reward hacking**——agent 会找到"让 reward 变高但没真完成任务"的捷径（比如学会输出"任务完成"的措辞来骗过自动评分器，而不是真把活干了）。这俩在 agentic 设定下比传统 RL 更尖锐，因为环境是开放的、reward 往往得靠另一个模型来判（又把第 9 章的 self-evaluation 难题套了进来）。现状：有 promising 的早期结果，但**没有可以无脑套用到任意 agent 任务上的通用配方**——这是当前正在研究的活跃方向，不是已解的工程问题。

把这两个开放问题和本章主线接上：规划的"圣杯"是让 agent 自己学会在恰当的确定性下选恰当的策略、不过度规划、不震荡。今天我们还做不到让模型可靠地自己拿捏，所以**靠外部控制流和硬旋钮兜底**——选型规则、可见的 todo artifact、replan budget。这些不优雅，但它们可观测、可控、会终止。下一章（第 9 章 反思与自我评估）会继续这个话题：当我们想让 agent 自己判断"计划好不好、做得对不对"时，会撞上 self-evaluation 的天花板——这正是 ToT 打分和 RL reward 共同的那堵墙。
