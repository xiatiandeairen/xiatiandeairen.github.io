> 阶段四 · 运维与规模 · 第 14 章
>
> 第 1 章把循环跑通了，但它吐出一个错答案时只给你一行字。这一章让你看见循环内部。

你调过一个 agent 没有？不是看它最终答得对不对——是它答错了，你得找出**为什么**。第 1 章的 `runAgent` 是个干净的 while 循环：模型决策、调工具、把结果塞回 context、再决策，直到模型说「我答完了」。跑顺的时候它优雅得不像话。跑歪的时候，你拿到的全部线索是一行 `answer: 抱歉，我没能查到光速。`——然后呢?这答案是模型瞎编的吗？是某个工具抛了异常吗？是它压根没调工具就放弃了吗？是前面某一步拿了个错值、后面每一步都忠实地在错值上接着算吗？

光看最终输出，这四种你一种都分不出来。一行答案告诉你循环坏了，但坏在**哪一步**——它一个字都不说。这一章就是给循环装上一双眼睛：把每一步都记成一条结构化记录，凑成一条时间线，让「在哪坏的」从猜变成读。

配套代码 `examples/agent-from-scratch/src/stage14-tracing.ts`，全离线、不要 key、不联网，跑 `npx tsx src/stage14-tracing.ts` 当场出两条对照的 timeline。诚实标注一句免得你拿 toy 当生产：里头的 token 数来自 MockLLM 的 `estimateTokens`（约 4 字符/token 的估算，不是真 BPE 分词器，见 `src/core/llm.ts:16`），耗时是**注入的延迟模型**不是真实测量（文件头的 HONESTY NOTE 和每处打印都标了）。但 step 数、失败步、被 blame 的那一步是**精确的**——从真实记录的 span 里算出来的，不是叙事编的。

## span：循环每一步发一条记录

先把名词钉死。一条 **span**（跨度，一段被计时的工作单元）就是循环里**一步**的结构化快照。stage14 的 span 长这样（`src/stage14-tracing.ts:44-52`）：

```ts
interface Span {
  step: number;       // 单调递增的步号，时间线的 x 轴
  type: SpanType;     // 'decision' | 'tool'
  tool?: string;      // 仅 type==='tool' 时有：哪个工具跑了
  tokens: number;     // decision: 本轮计费的 input token；tool: 回灌 context 的 payload token
  durationMs: number; // 本步墙钟耗时（mock 下是注入值，见 HONESTY NOTE）
  ok: boolean;        // false = 这步是根因候选；见 findBlame()
  detail?: string;    // 给人读的一行说明（错误文本 / 工具名 / 答案头部）
}
```

注意 `type` 只有两个值：`decision` 和 `tool`。这不是随便分的——它对应循环每轮干的两件**本质不同**的事：模型在**想**（把整段对话历史变成下一个动作），和 harness 在**做**（执行模型点名的那个工具）。OTel GenAI 约定把这两件事分成两种 span（inference span 和 tool execution span），stage14 照搬。为什么不合成一个「step」span？因为一条 run 慢，可能是**模型**慢（推理 token 多）也可能是**工具**慢（一次 DB 查询），合并的 span 没法告诉你是哪个（`src/stage14-tracing.ts:36-39`）。这是可观测性里最常见的自毁动作：把两个不同来源的耗时揉成一个数，然后发现自己优化不了。

### Tracer 不是全局单例，是被传进去的

你可能下意识想把 tracer 做成全局的——到处 `import { tracer }` 然后 `tracer.log(...)`。stage14 故意不这么做（`src/stage14-tracing.ts:54-85`）：

```ts
class Tracer {
  private readonly spans: Span[] = [];
  constructor(private readonly now: () => number = () => performance.now()) {}

  async span(step, type, run): Promise<Span> {
    const t0 = this.now();
    const partial = await run();      // 真正干活的回调
    const durationMs = this.now() - t0;
    const span: Span = { step, type, durationMs, ...partial };
    this.spans.push(span);
    return span;
  }
}
```

两个设计决定值得停一下。一是 tracer 当**参数传进循环**，不是全局状态——这样 happy run 和 buggy run 各拿一个 tracer，两条时间线干净分开、可对照，而不是糊成一坨。二是**时钟注入**（`now` 是构造参数）：循环本身从不碰时钟，计时是 tracer 的活，每一步用同一套计时逻辑（decision span 和 tool span 不会因为各自手写计时而产生口径漂移）。注入时钟还顺手解决了 demo 的一个真问题——离线 mock 跑在微秒级，真实耗时全是调度噪声，所以 demo 喂一个**注入的单调时钟**让时间线读得出来（`src/stage14-tracing.ts:388-403`）。机制（per-step span、求和、blame）和生产一模一样，只有时钟源不同。这是 `functional core / imperative shell`（纯核 + 命令壳）的一个具体落地：把不确定性（时钟）赶到边界，核心逻辑可复现。

### 追踪是副产品，绝不能改变行为

traced 循环（`src/stage14-tracing.ts:100-171`）和第 1 章的 `runAgent` 形状一样，只是每次模型调用、每次工具调用外面包了一层 `tracer.span(...)`。关键不变量写在 `src/stage14-tracing.ts:90-91`：**把 tracer 拿掉，循环照常工作**。可观测性只能**记录**行为，不能**改变**行为——这条听起来废话，但违反它是真实事故来源：你加了个 trace 钩子，钩子里 `await` 了一次远程上报，于是每一步多了 50ms 网络延迟、或者上报失败把整个循环带崩了。stage14 的 span 是纯本地 append，不阻塞、不抛、不联网，正是为了守住这条线。

## 读 trace：summary、timeline、blame

记下来只是第一半，能**读**才有用。先看 happy run 的实测输出：

```
── Run 1: HAPPY ──────────────────────────────────────────
answer: 光速：[1] 299792458 m/s in vacuum (stop: answered)
timeline:
  ✓ step  0 | decision           |   119 tok |  120.0 ms | → call look_up
  ✓ step  1 | tool:look_up       |     6 tok |    8.0 ms | 299792458 m/s in vacuum
  ✓ step  2 | decision           |   165 tok |  140.0 ms | → call cite
  ✓ step  3 | tool:cite          |     7 tok |    6.0 ms | [1] 299792458 m/s in vacuum
  ✓ step  4 | decision           |   207 tok |   95.0 ms | → answer: 光速：[1] 299792458 m/s in vacuum
summary: 5 steps | 2 tool calls | 504 tokens | 369.0 ms | 0 failed
blame : none — clean run
```

这条时间线读起来像一份给时间用的 stack trace：自上而下五步，决策和工具交替，每步带 token 和耗时。光这一屏你能看出几件单看答案永远看不到的事。

第一，**token 在逐步涨**：decision 步的 token 是 119 → 165 → 207。这正是第 5 章讲的 context 膨胀——每轮把整段对话历史重发一遍，所以越往后越贵。trace 把这条成本曲线**逐步**画出来了，而不是只给你一个 504 的总数（`src/stage14-tracing.ts:119-121`）。第 16 章讲成本时，这条逐步的 token 曲线就是入口。第二，**耗时分布在哪**：decision 步 120/140/95ms，tool 步只有 8/6ms——时间花在模型推理上，不在工具上。如果哪天 tool 步飙到几百毫秒，你一眼就知道该去查那个工具而不是怪模型。第三，`summary` 那行的每个数字都是从记录的 span 里**求和**出来的，不是叙事编的（`src/stage14-tracing.ts:204-211`）：`totalTokens` 是所有 span 的 tokens 之和，`failedSteps` 是 `!ok` 的计数。可观测性的纪律就是这条——展示的数必须能追到原始记录。

### 结构化字段的命名为什么是死活攸关的

这条 timeline 之所以能机器读，全靠 span 是**结构化**的——`tokens` 是个 number 字段，不是塞在一句话里的 `"花了 119 个 token"`。如果你把这些信息写成自由文本日志（`console.log("step 0 decided to call look_up, used 119 tokens")`），那么求和、过滤失败步、按耗时排序——每一个都得先写正则去文本里抠数字，而正则会在你改一次日志措辞时悄悄失效。这就是 `summarize` 能五行写完、`findBlame` 能一行写完的前提：字段有名字、有类型、位置固定。

命名还得**对**。stage14 不自己造字段名，照 OpenTelemetry GenAI 语义约定来（`src/stage14-tracing.ts:13-17`、`246-277`），这样 span 丢进真实追踪后端（Jaeger / Tempo / Honeycomb）不用改名就能用。看 buggy run 导出的前两条 OTLP span：

```json
{"name":"chat mock","gen_ai.operation.name":"chat","gen_ai.usage.input_tokens":119,"gen_ai.system":"mock","duration_ms":120}
{"name":"execute_tool look_up","gen_ai.operation.name":"execute_tool","gen_ai.tool.name":"look_up","gen_ai.usage.output_tokens":0,"gen_ai.system":"mock","duration_ms":8,"error.type":"step_failed"}
```

`gen_ai.operation.name=chat` 标决策、`=execute_tool` 标工具调用，`gen_ai.tool.name` 记哪个工具，`error.type` 是 OTel 标记 span 失败的约定字段（`src/stage14-tracing.ts:253-260`）。为什么这层一致性值钱：约定就是命名稳定性——你今天叫 `gen_ai.usage.input_tokens`，后端的 dashboard、告警规则、解析器全都按这个 key 建。哪天你心血来潮改成 `inputTokens`，下游全静默坏掉（这跟 log schema 一旦发布就不该改名是同一条规矩）。所以 stage14 的做法是：内部用人体工学友好的 `Span` 形状，到**边界**才用 `toOtelSpan` 转成点分键（`src/stage14-tracing.ts:256-277`），不让满循环都是 `gen_ai.xxx` 这种字符串。纯核 + 命令壳，又一次。

**对标你正在用的 Claude Code**：它的每个工具调用——Bash 跑命令、Read 读文件、ToolSearch 按需加载工具 schema、Agent 派子 agent——本质上就是一条 tool span（`gen_ai.operation.name=execute_tool`），每次模型回合是一条 decision span。第 12 章的 Workflow（确定性编排）和这章是互补关系：workflow 决定**该走哪一步**，trace 记录**实际走了哪一步、花了多少**。TodoWrite 那个计划清单其实是 trace 的一个亲戚——它把「打算做的步」显式化，trace 把「实际做的步」记下来，两者一对就知道 agent 有没有跑偏计划。

## 失败模式：长 trace 里定位「带歪的那一步」

happy path 谁都会看。trace 的全部价值在出错时才兑现。看 buggy run 的实测输出——这条 run 里模型在 step 0 把 key 写错了（要查 `speed-of-light` 它却查了 `lightspeed`，KB 里没有）：

```
── Run 2: BUGGY (wrong key on step 0 cascades) ───────────
answer: 抱歉，我没能查到光速。 (stop: answered)
timeline:
  ✓ step  0 | decision           |   119 tok |  120.0 ms | → call look_up
  ✗ step  1 | tool:look_up       |     0 tok |    8.0 ms | threw: no fact for key "lightspeed"  ⟵ first failure (root-cause candidate)
  ✓ step  2 | decision           |   173 tok |  140.0 ms | → call cite
  ✗ step  3 | tool:cite          |     0 tok |    6.0 ms | threw: cite called with empty fact (was look_up skipped?)
  ✓ step  4 | decision           |   221 tok |   95.0 ms | → answer: 抱歉，我没能查到光速。
summary: 5 steps | 2 tool calls | 513 tokens | 369.0 ms | 2 failed
blame : step 1 (tool:look_up) — "threw: no fact for key "lightspeed""
lesson: 2 steps failed, but the root cause is the FIRST (step 1); steps after it are downstream symptoms.
```

`2 failed`——两步带 `✗`。新手会去看**最后**那个失败（step 3 的 `cite called with empty fact`），因为它离最终的失败答案最近。这正是错的方向。step 3 是**症状**：模型在 step 1 拿到「no fact for key」的错误后，没停下来，反而硬着头皮去 cite 一个空事实——所以 step 3 的失败是 step 1 的下游连锁反应。真正的根因是**第一个**失败：step 1 的错 key。后面每一步都忠实地在这个错值上往下接。

stage14 把这条教训机械化成 `findBlame`（`src/stage14-tracing.ts:222-224`）：

```ts
function findBlame(spans: readonly Span[]): Span | null {
  return spans.find((s) => !s.ok) ?? null;   // 第一个 ok:false
}
```

一行，取**第一个**失败 span。为什么是 first-failing 不是 last-failing：失败会级联。模型调了个坏工具（失败），拿到错误，然后在错误上又扑腾五步。最后那个失败是症状，第一个才是根因。timeline 里那个 `⟵ first failure (root-cause candidate)` 箭头就指在 step 1（`src/stage14-tracing.ts:230-233`）。这就是这一章的核心可操作结论：**长 trace 里，从第一个 `✗` 开始查，不是从最后一个**。trace 短的时候这没什么；等你 agent 跑了 40 步、错答案前面挂了 12 个 `✗` 的时候，「先看第一个」能省你半小时。

### 三种失败模式都被 trace 记下来，不靠崩溃

值得指出：traced 循环对三种失败都不是 crash，而是**记一条 `ok:false` 的 span 再把错误喂回模型**——这跟第 1、2 章「工具抛错不崩循环、把错误回灌让模型自救」是一脉的（`src/stage14-tracing.ts:142-163`）：

1. **幻觉工具名**——模型点了个不存在的工具。记 `ok:false`，detail 写 `unknown tool: X`，回灌 isError 结果（`src/stage14-tracing.ts:144-148`）。
2. **工具自己抛异常**——比如上面的错 key。记 `ok:false`，错误文本进 detail 也回灌给模型（`src/stage14-tracing.ts:156-162`）。
3. **永不收敛**——跑满 `maxTurns` 都没出终止答案。trace 的形状就是「一串 decision span，没有终结的 answer span」，一眼认出失控循环（`src/stage14-tracing.ts:168-170`）。

第三种尤其是 trace 才看得见的——光看答案你只会拿到一个 `[stopped: max turns reached]`，但 timeline 会把那 N 步原地打转的样子画出来，你能看清它是在哪两个状态间来回横跳。

### findBlame 是启发式，不是真相

得把 `findBlame` 的天花板讲清楚，免得你过度信它。它是个**启发式**（heuristic，经验规则），不是 ground truth（`src/stage14-tracing.ts:216-221`）。它只抓**机械失败**——工具抛了、工具名不存在这类 `ok:false`。但有一类 bug 它**完全看不见**：一个 `ok:true` 的工具调用，返回了一个**格式正确但内容是错的**结果。比如 `look_up` 成功返回了，但返回的是错误年份的数据；或者模型在 decision 步选了个语法合法、语义却错的工具。这些步全是 `✓`，`failedSteps` 数到 0，`findBlame` 返回 `none — clean run`——可答案就是错的。这是**模型质量 bug**，不是**机械 bug**。trace 缩小了搜索范围，它**不替代判断**。下一句就是本章的开放问题。

## ⚡ 开放问题：从 trace 自动归因「哪一步带歪了」

`findBlame` 的「取第一个 `✗`」在 stage14 里精准，是因为这个 demo 的失败恰好是机械的、会抛异常的。真实 agent 的大多数「跑歪」**不抛异常**：模型在第 7 步做了个看似合理、实则把整个任务带偏的决策，工具乖乖执行、`ok:true`、span 全绿，然后第 8 步往后全在这个错前提上认真干活，最后交出一个自洽但错误的结果。**从这样一条全绿的长 trace 里自动指出「就是第 7 步那个决策带歪的」——这件事今天没有通用解，正在研究。**

难在哪：这不是找异常，是**反事实归因**（counterfactual attribution）——「如果第 7 步换个决策，结局会不会不同」。要回答它，理论上你得能 replay（回放）那条 trace、在第 k 步注入一个不同的决策、重跑下游、看结局变没变。但 agent 的下游充满不确定性（模型采样、工具返回的外部状态），重跑一次结果可能就不一样——你分不清结局变了是因为你的干预，还是因为 LLM 本来就 nondeterministic（非确定性，同输入不同输出）。当前的几条路子各有硬伤：让一个更强的模型当裁判读整条 trace 指出问题步（LLM-as-judge）——它自己也会幻觉、也会被长 trace 带偏，且没有 ground truth 校准时它的判断本身不可信；用 step-level reward model 给每步打分——得先有大量标注好「这步是不是错」的 trace，而标注「哪步带歪了」恰恰就是我们想自动化的那件事，鸡生蛋。所以今天能可靠自动化的，仍只到「第一个机械失败」这一层；再往上的「第一个**语义**失败」，trace 给你证据，归因还得靠人。这也是为什么本书第 13 章讲 eval（评估）和这一章是搭配的——你没法自动归因单条 trace 的根因，但你可以在一批 trace 上跑评估、看失败率分布，把「人去读哪条 trace」这件事变得有的放矢。

把这章收一句：**没有 trace 就没法调 agent**。一行答案是个谜语，一条结构化时间线是份证词。stage14 的 trace 能精确告诉你第一个机械失败在哪、token 怎么涨的、时间花在模型还是工具上；它**不能**告诉你那个全绿 trace 里哪个决策悄悄把任务带歪了——那是下一个十年要解的问题，今天先用「看第一个 `✗`」和一双肯读 timeline 的眼睛顶着。
