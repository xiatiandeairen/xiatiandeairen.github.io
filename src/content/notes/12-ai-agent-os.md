# 我给 AI Agent 写了一套操作系统

## AI 能写代码，但不能交付

上个月我让 Claude 帮我重构一个搜索模块。需求很明确：把客户端搜索改成基于预生成索引的方案。

Claude 写得很快。20 秒，一套完整的实现就出来了。代码能跑，测试能过。我合了代码，部署上线。

第二天早上发现搜索页白屏了。

排查了半小时才发现：Claude 在重构搜索模块时，顺手改了一个被三个其他页面引用的工具函数的签名。它改的那个函数确实在搜索模块里用到了，但 Claude 没有意识到这个函数还有其他调用方。测试？测试只覆盖了搜索模块本身。

这不是个例。日常使用 AI 编程助手，三类问题反复出现：

**AI 做了但不对。** 你说"重构搜索"，它确实重构了，但按照自己的理解重构的——你要的是换数据源，它给你改了 UI 组件结构。需求没有对齐，代码写了等于白写。

**改了一处坏了三处。** AI 没有全局视野。它看到的是你指向的文件，不是整个系统的依赖关系。改动范围溢出是常态，不是例外。

**上次的教训这次全忘。** 你花 15 分钟告诉它"这个项目不用 lodash，用原生方法"，它照做了。第二天新对话，第一行代码就 `import _ from 'lodash'`。

这些问题的共同点是什么？**不是 AI 不够聪明，而是它没有工作流程。** 它收到指令就立刻执行，不评估、不规划、不验证、不记忆。

> **关键洞察：** AI Agent 的可靠性瓶颈不是模型能力，而是流程缺失。一个没有流程的天才程序员和一个没有流程的 AI，犯的错误是相似的——只是 AI 犯得更快。

## 缺的不是能力，是操作系统

一个刚毕业的工程师进入团队，你不会指望他第一天就能独立交付一个跨模块重构。不是因为他不会写代码——他的算法可能比你还好——而是因为他还没有建立起**工作方法论**：什么时候该先问清楚需求，什么时候该画个设计图，什么时候该跑一遍回归测试，什么时候该请人 review。

这些方法论不是天赋，是后天训练出来的。是公司的流程、团队的习惯、自己踩坑之后的经验。

AI Agent 面临完全相同的问题。

Claude、GPT-4、Gemini——模型能力已经足够强了。它们能理解复杂的代码库，能写出符合惯例的实现，能解释自己的设计决策。但它们缺少一样东西：**一个外部的流程系统**来约束"什么时候该做什么"。

我把这个流程系统叫做 AI Agent 的"操作系统"。操作系统不负责计算——那是 CPU（模型）的事。操作系统负责调度：哪个任务先执行、需要什么资源、执行完怎么验证、出了问题怎么恢复。

过去一年，我构建了这样一套系统：5 个独立的 Skill（技能），每个解决 AI 工作流中的一个具体缺陷。

## 五个 Skill，一个闭环

这 5 个 Skill 不是随意拼凑的。它们构成一个完整的工程闭环：

```
  Decay（发现问题）
    ↓
  Diagnose（定位根因）
    ↓
  Know（积累教训）
    ↓
  Sprint（可靠执行）
    ↓
  Refactor（保持精炼）→ 回到 Decay
```

每个 Skill 解决一个精确的问题，不多不少。

**Sprint — 任务执行引擎。** AI 接到任务不再直接写代码。Sprint 用 3 个 yes/no 问题评估任务复杂度，自动裁剪出合适长度的流水线。简单 bug 修复走 3 个阶段，跨模块重构走 7 个阶段。每个阶段有明确的输入输出，阶段之间通过 handoff 文档传递上下文。执行结束后，结构断言（Anchor）验证 AI 没有修改不该修改的文件。

**Know — 知识编译器。** 在项目中建立结构化知识库，每条知识带有触发模式。关键机制是 auto-recall：AI 修改某文件前，自动检查相关约束。"这个模块不能引入外部依赖"这类规则，不需要每次对话重复——Know 在执行前主动拦截。知识有衰减机制，长期不触发的条目自动降级，防止知识库膨胀成噪声库。

**Diagnose — 结构化调试。** AI 默认的调试方式是"改一处看一下"，本质上是随机搜索。Diagnose 强制执行 3 层收敛模型：先定位区域（Zone），再确定机制（Mechanism），最后找根因（Root Cause）。每层有对应的验证方法，每轮最多 3 个假设，3 轮未收敛强制上报人类。把调试从碰运气变成有方向的搜索。

**Decay — 项目健康监测。** 用 Rust 实现的 CLI 工具，扫描文件结构和 Git 历史，从结构度、复杂度、脆弱度三个维度评分。用 SQLite 存储快照追踪趋势。作用不是发现 bug，而是发现**趋势**——项目是在变健康还是在变脆弱，重构之后是改善了还是恶化了。

**Refactor — 文档压缩器。** Skill 文档越写越长，但 AI 的 context window 是有限的。Refactor 用 6 条规则（语义守恒、信息密度最大化、术语统一等）压缩 SKILL.md 文件，用更短的文本表达相同的语义。这是一个自举系统——它用来优化其他 Skill 的定义文件。

> **关键洞察：** 这 5 个 Skill 的顺序不是随意的。Decay 发现问题，Diagnose 定位问题，Know 记住教训，Sprint 避免再犯，Refactor 保持系统可维护。它们形成的是一个**学习循环**，不只是一组工具。

## 设计原则：什么样的指令 AI 才能执行

构建这些 Skill 的过程中，我踩过很多坑。最初的版本写得像人类的工作手册——"适当评估风险""合理分配时间"——这些话人类能理解，AI 完全无法执行。

经过反复迭代，沉淀出 4 条设计原则：

**原则一：精确到可机器执行。**

"评估任务复杂度"是不可执行的。"回答 3 个 yes/no 问题，根据组合选择对应的流水线模板"是可执行的。

每一条规则都必须能被拆解成明确的判断条件和执行动作。没有"大概""适当""酌情"——这些词对 AI 来说等于没说。

Sprint 里的一个具体例子：流水线裁剪不是让 AI "判断是否需要 design 阶段"，而是问一个客观问题——"实现方案是否唯一且明确？是/否"。是，跳过 design。否，进入 design。

**原则二：最少的人类干预点。**

如果每一步都需要人确认，那 AI 的价值就没了。但完全不确认又会失控。

解决方案是在 Skill 中用 `[STOP:confirm]` 和 `[STOP:choose]` 标记精确定义干预点。日常执行 AI 自主完成，只在**不可逆决策**（架构选型、删除文件、修改公共接口）时暂停等待确认。

Sprint 的 7 个阶段中，通常只有 2-3 个需要人类确认：brainstorm 的需求锁定、design 的方案选择、review 的最终审批。其余阶段自动流转。

**原则三：所有状态可追溯。**

AI 对话是临时的，但工程决策需要持久化。

每个 Skill 都有对应的持久化机制：Sprint 在 `.sprint/{id}/` 目录保存状态和 handoff 文档；Know 用 JSONL 索引和事件日志；Diagnose 写会话文档记录整个调试过程；Decay 用 SQLite 存快照。

这不只是为了复盘——更重要的是**恢复**。对话中断了、context window 满了、需要换一个会话继续，持久化的状态让 AI 能接上之前的进度，而不是从头开始。

**原则四：单独可用，组合更好。**

5 个 Skill 之间没有硬依赖。Sprint 不需要 Know 也能工作，Diagnose 不需要 Decay 也能调试。

这是刻意的设计。如果 Skill 之间耦合太紧，任何一个出问题都会拖垮整个系统。独立性保证了**优雅降级**——某个 Skill 不适用于当前场景，直接跳过就好，不影响其他 Skill 的运行。

组合使用确实更好：Sprint 执行时 Know 自动 recall 相关约束，Diagnose 定位的问题可以直接输入 Know 积累，Decay 发现的退化趋势可以触发 Sprint 任务。但这些是"加分项"，不是"必需项"。

> **常见误区：** 很多人尝试写 AI 工作流时，会试图在一个巨大的 prompt 里塞进所有规则。结果要么超出 context window，要么 AI 只执行了前半部分就"忘了"后面的。正确的做法是拆分成独立 Skill，每个 Skill 有自己的文件，按需加载。

## 一个真实任务的全流程

说了这么多原则，看一个实际的例子。

**任务：** 给博客网站添加 RSS 订阅功能。

**没有 Skill 时：**

我告诉 AI "给网站加 RSS"。它立刻生成了一个 `rss.xml` 路由，写了 50 行代码，包含文章标题和链接。看起来能用。

但它漏了：内容摘要没有转义 HTML 实体（导致某些阅读器解析失败）、没有设置正确的 Content-Type header、hardcode 了网站 URL 而不是读配置、没有在 `<head>` 里添加 RSS 自动发现标签。

我花了 40 分钟逐个修复这些问题。

**有 Skill 时：**

Sprint 先问 3 个问题：需求清晰吗？（是——RSS 规范很明确）设计方案唯一吗？（是——Astro 有标准的 RSS 集成方式）高风险吗？（否——新增功能，不影响现有代码）

裁剪出短流水线：`plan → execute → quality → insight`。

**Plan 阶段** 列出具体任务和 Anchor：

```
任务 1: 创建 /rss.xml 端点
任务 2: 在 BaseLayout 的 <head> 中添加 RSS 自动发现标签
任务 3: 使用 SITE_CONFIG 中的 URL，不 hardcode

MUST_EXIST: src/pages/rss.xml.ts
MUST_IMPORT: src/pages/rss.xml.ts -> @utils/constants
FILE_NOT_MODIFIED: src/pages/index.astro
```

注意最后一行——`FILE_NOT_MODIFIED: src/pages/index.astro`。这是一个约束：添加 RSS 功能不应该需要修改首页。如果 AI 在执行时修改了首页，Anchor 检查会失败并拦截。

**Execute 阶段** 逐个完成任务。

**Quality 阶段** 运行 Anchor 检查 + 构建验证。这一轮发现 AI 确实在 `rss.xml.ts` 里 hardcode 了 URL，Anchor（`MUST_IMPORT constants`）报错，AI 自行修正。

**Insight 阶段** 记录本次 Sprint 的经验——"RSS 端点需要显式设置 Content-Type"，这条经验通过 Know 的 `/know learn` 进入知识库，下次遇到类似任务时会自动 recall。

整个过程人工干预 0 次。最终产出比"直接让 AI 写"多花了约 30 秒（流水线评估和 Anchor 检查的时间），但省了 40 分钟的后续修复。

> **实践建议：** 不要试图一次搭建完整的 Skill 系统。从 Sprint 开始——它的投入产出比最高。最小版本只需要 plan + execute + quality 三个阶段，加上 2-3 个 Anchor。在实际使用中感受到"AI 忘记约束"的痛点后，再引入 Know。感受到"调试低效"的痛点后，再引入 Diagnose。

## 你也可以构建自己的 Skill 系统

我的 5 个 Skill 是针对自己工作流优化的结果，不一定完全适合你。但底层思路是通用的：**识别 AI 的行为缺陷 → 设计对应的流程约束 → 用精确的指令实现**。

如果你想从零开始，建议的路径：

**第一步：记录 AI 出错的模式。** 用两周时间，每次 AI 犯错就记一笔——是需求理解错了、还是改动范围溢出了、还是忘了之前的约定。你会发现错误是高度集中的，通常 3-4 种模式覆盖了 80% 的问题。

**第二步：针对最频繁的模式写第一个 Skill。** 不要追求完整，写一个能解决你最痛的那个问题的最小 Skill。50 行 Markdown 就够了。

**第三步：在实际使用中迭代。** Skill 不是写完就不动了。每次 AI 执行偏离预期，就是一个改进 Skill 的机会。我的 Sprint 经历了 14 次重写，Know 经历了 8 次。初始版本和现在的版本几乎没有相似之处。

有一个判断标准可以检验你的 Skill 是否有效：**使用 Skill 后，你是否更少需要手动修改 AI 的输出？** 如果答案是"没感觉到区别"，那 Skill 可能定义得不够精确，需要回到设计原则第一条——找到那些模糊的措辞，替换成可机器执行的判断条件。

所有代码开源在 GitHub。但比起直接复制我的 Skill，我更建议你从自己的痛点出发构建自己的。最好的 Skill 系统不是最复杂的那个，而是最贴合你工作流的那个。
