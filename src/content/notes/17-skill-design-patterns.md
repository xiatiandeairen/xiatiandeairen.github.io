# Skill 设计模式：如何写出 AI 真正能执行的指令

## 为什么你的 prompt 无法成为 Skill

很多人尝试给 AI 写工作流程时，写的是这样的东西：

```
在开始编码之前，先充分理解需求，确保方案合理。
编码时注意代码质量，做好错误处理。
完成后进行充分测试，确保没有引入 bug。
```

这段话对人类完全有效——一个有经验的工程师看到这些提醒会自然地执行相应的行为。但对 AI 来说，这三行等于什么都没说。

"充分理解需求"——什么算充分？理解到什么程度可以开始编码？

"确保方案合理"——合理的标准是什么？和谁确认？

"充分测试"——跑单元测试？集成测试？手动测试？覆盖率多少算充分？

这些措辞对人类管用是因为人类有**隐性知识**——多年经验告诉我们"充分"大概意味着什么。AI 没有这些隐性知识，它需要**显式的判断条件和执行步骤**。

从 prompt 到 Skill 的跨越，核心就是把隐性知识变成显式规则。以下是我从 5 个 Skill 中提炼出的 6 个设计模式。

## 模式一：用判断条件替代形容词

**反模式：** "如果任务比较复杂，先做设计。"

**Skill 模式：** "回答以下问题：实现方案是否唯一且明确？是 → 跳过设计。否 → 进入设计阶段。"

区别在于："比较复杂"是主观判断，不同时刻的 AI 会给出不同答案。"实现方案是否唯一且明确"是客观问题，答案是确定的。

Sprint 的 3 问裁剪机制就是这个模式的典型应用。它不让 AI 判断"这个任务需要几个阶段"，而是用 3 个 yes/no 问题推导出答案。AI 的自由度被限制在回答客观问题上，而不是做主观评估。

适用场景：任何需要 AI "判断"的环节。检查你的 Skill 中是否有以下词汇：适当、合理、充分、足够、必要时。每一个都应该被替换为具体的判断条件。

## 模式二：有限选项而非开放生成

**反模式：** "选择合适的调试策略。"

**Skill 模式：** "从以下 5 种验证方法中选择：read-code / run-test / add-log / isolate / git-bisect。每种的使用条件见下表。"

AI 在开放生成模式下会"创造性"地发明策略——有时候创造出的方法是合理的，有时候是离谱的。给它一个有限选项列表（枚举），它的行为变得可预测。

Diagnose 的验证方法库就是这个模式。它不让 AI 自己想"怎么验证假设"，而是给了 5 种方法，每种有明确的适用条件和限制。AI 的工作变成了"选择"而不是"发明"。

关键细节：**限制条件和选项同样重要**。"run-test：只跑相关测试，不跑全量"——没有这条限制，AI 可能会跑整个测试套件等 15 分钟。

> **关键洞察：** 给 AI 的自由度应该和它的可靠性成正比。在 AI 判断力还不够稳定的领域（调试策略、风险评估），用有限选项。在 AI 判断力足够可靠的领域（代码实现、文本生成），给予自由度。Skill 设计的艺术在于找到这个边界。

## 模式三：Gate 控制——谁决定、什么时候决定

Skill 中的每一步都应该标记谁来决定是否继续：

| Gate 类型 | 含义 | 标记 |
|----------|------|------|
| **user** | 人类确认后才能继续 | `[STOP:confirm]` |
| **auto** | AI 根据条件自动判断 | （无标记，默认） |
| **always** | 始终执行，不检查 | （无标记） |

Sprint 中 Design 阶段的方案选择是 `[STOP:choose]`（user gate）——技术方案必须人类选，AI 不能自己决定。Execute 阶段的任务执行是 auto gate——AI 自行完成，不需要逐个确认。

Gate 的设计原则：**不可逆的决策用 user gate，可逆的操作用 auto gate**。

什么是不可逆的？架构选型（选了 A 方案就要推翻已写的 B 代码）、数据迁移（执行后回滚很困难）、公共接口变更（下游依赖者会受影响）。

什么是可逆的？写一个新文件（不喜欢就删掉）、修改内部实现（不影响外部接口）、添加测试用例（多了不嫌多）。

过多的 user gate 会让 AI 变成一个需要逐步审批的"实习生"——每做一步都要等人确认，效率归零。过少的 user gate 会让 AI 在关键决策上自作主张。

## 模式四：Handoff 文档——阶段之间的信息压缩

多阶段 Skill 的一个核心问题：后续阶段怎么获取前序阶段的信息？

**反模式：** 把整个对话历史丢给下一个阶段。

问题：对话历史里有大量的讨论、尝试、回退、重复——真正有价值的信息可能只占 10%。AI 的 context window 被噪声占满。

**Skill 模式：** 每个阶段写一份结构化的 handoff 文档，下一个阶段只读这份文档。

Sprint 的 Brainstorm 阶段可能经历了 3 轮讨论才锁定需求，但 handoff 文档只包含最终结论：目标、成功标准、排除项。Design 阶段不需要知道讨论过程。

Handoff 的结构因阶段而异，但都遵循一个原则：**只包含下游需要的最小信息集**。

这有一个反直觉的好处：handoff 文档本身就是一种验证。如果你无法把一个阶段的产出压缩成一份结构化文档，通常意味着这个阶段的工作还没有真正完成——结论还不够清晰、共识还没有达成。

## 模式五：数字化约束——消除"差不多"

**反模式：** "不要生成太多假设。" / "文件不要太大。"

**Skill 模式：** "每轮最多 3 个假设。" / "单文件超过 300 行标记为'大文件'。"

AI 对"太多""太大""太长"没有一致的理解。不同的对话、不同的上下文，它的判断会漂移。给一个确切的数字，行为就稳定了。

Diagnose 中所有的限制都是数字化的：每轮 3 个假设、每个假设 2 次验证、3 轮上限、read-code 最多 3 个文件、add-log 最多 3 处。这些数字不是随意的——它们是在实际使用中调优的结果——但具体是 3 还是 5 不是关键，关键是**有一个明确的数字**。

> **常见误区：** "把数字定大一点，给 AI 更多空间。"这个直觉是错的。AI 在没有约束时不会做出"更好"的判断，而是做出"更多"的动作——更多的假设、更多的修改、更多的输出。约束的目的不是限制 AI 的能力，而是强制它聚焦。3 个假设意味着 AI 必须选最可能的 3 个，而不是列出 10 个"都有可能"的猜测。

## 模式六：失败路径和终止条件

Skill 不只要定义"怎么做"，还要定义"做不成怎么办"。

**反模式：** 只定义正常流程，假设每一步都能成功。

**Skill 模式：** 每个可能失败的步骤都有对应的处理策略。

Sprint 的 Anchor 检查失败时：AI 在 Quality 阶段内修复，重新检查。两次修复仍然失败？上报用户，不继续。

Diagnose 的假设验证失败时：排除该假设，生成下一个。所有假设都排除？后退一层或回到分类。3 轮都没收敛？上报用户，附上排除清单。

Long Sprint 的 Direction Lock 被违反时：回退到违反 Lock 的那个子 Sprint 之前的状态，重新执行。

失败路径是 Skill 中最容易遗漏、也最重要的部分。因为 AI 在遇到意外情况时的默认行为是**继续做下去**——忽略错误、绕过检查、生成一个"看起来对"的结果。没有明确的失败路径，AI 的错误会被掩盖，直到很晚才暴露。

定义失败路径的检查清单：
- 每个 `[STOP:confirm]` 点，如果用户说"不行"怎么办？
- 每个验证步骤，如果失败了怎么办？
- 整个流程走了多久没有进展算"卡住了"？卡住了做什么？
- 最终的"逃生出口"在哪里？

> **实践建议：** 写完一个 Skill 之后，做一次"红队测试"——假设 AI 在每一步都做了最差的选择，看你的 Skill 能不能把它拉回来。如果有任何一步的答案是"不能"，那里就需要加一个失败路径。Skill 的健壮性不取决于正常流程多完善，而取决于异常流程能兜住多少。
