# Decay：用 Rust 给代码做体检

## 技术债务是慢性病

没有人会在某一天早上醒来发现项目突然"变烂了"。技术债务是渐进的——每天多一个 workaround，每周多一个"临时"方案，每月多一层不必要的抽象。单独看每一次妥协都是合理的，但累积起来就是结构性退化。

更隐蔽的是，退化的初期几乎没有症状。代码能跑，测试能过，功能能交付。直到某一天你想加一个简单功能，发现需要改 8 个文件；或者一个新人入职，花了两周还搞不清楚模块之间的依赖关系。

这时候再回头清理，成本已经是当初的十倍。

Decay 解决的不是"怎么修技术债务"——那是具体的工程决策。Decay 解决的是**怎么在退化的早期发现它**。就像体检不治病，但能在症状出现之前发现异常指标。

## 三个维度，三个数字

Decay 从三个维度评估项目健康度，每个维度 0-100 分：

**结构度（Structural）** 衡量项目的组织是否清晰。

扫描指标：文件总数、目录最大深度、顶级目录数量、文件命名一致性。

一个健康的项目，文件按职责分布在合理的目录结构中，顶级目录不超过 10 个（太多说明缺乏组织），目录深度不超过 5 层（太深说明过度嵌套）。

当结构度下降时，通常意味着：新功能被随意放在了不合理的位置、目录结构失去了一致性、出现了"杂物抽屉"式的 `utils/` 或 `misc/` 目录。

**复杂度（Complexity）** 衡量单个文件的体量是否合理。

扫描指标：大文件比例（>300 行）、平均文件行数、最大文件行数。

文件越大越难理解、越难测试、越难安全地修改。一个 2000 行的组件几乎不可能被 AI 正确地局部修改——它会遗漏上下文、修改错误的位置、或者引入与文件其他部分不一致的代码。

复杂度上升的信号：某几个"核心"文件越来越大（因为所有新功能都往里加）、大文件比例从 5% 涨到 15%。

**脆弱度（Fragility）** 衡量项目对变更的抵抗力。

扫描指标：Git 提交热点集中度（前 10% 的文件占了多少比例的提交）、单文件最大代码行数、变更频率最高的文件列表。

如果 80% 的提交都集中在 5% 的文件上，这些文件就是脆弱点——每次改它们都有引入 bug 的风险，但又不得不频繁修改。

> **关键洞察：** 这三个维度不是独立的。结构度下降往往导致复杂度上升（文件放错位置 → 功能堆积 → 文件变大），复杂度上升导致脆弱度上升（大文件难以安全修改 → 修改频繁出 bug → 变更集中在几个文件上）。Decay 把它们分开评分，是为了让你知道退化的**起点在哪里**。如果结构度下降而复杂度和脆弱度还正常，说明问题刚开始——现在调整目录结构的成本最低。

## 趋势比绝对值重要

Decay 用 SQLite 存储每次扫描的快照。这不是为了查看"当前分数是多少"——绝对值的参考意义有限，因为不同类型的项目健康标准不同（一个 SDK 和一个全栈应用的合理结构差异很大）。

真正有价值的是**趋势**。

```
2026-01  structural: 82  complexity: 75  fragility: 70
2026-02  structural: 80↓ complexity: 74  fragility: 71↑
2026-03  structural: 76↓ complexity: 78↑ fragility: 75↑
2026-04  structural: 74↓ complexity: 82↑ fragility: 80↑
```

看绝对值：4 月的分数还算可以，74/82/80 都在"可接受"范围。

看趋势：结构度连续 4 个月下降，复杂度和脆弱度持续上升——这是一条退化曲线。如果不干预，两三个月后就会降到"痛苦"区间。

趋势信息让你可以**在症状出现之前采取行动**。

另一个常见场景：你做了一次大重构，想知道效果如何。

```
重构前  structural: 68  complexity: 85  fragility: 82
重构后  structural: 78↑ complexity: 72↓ fragility: 70↓
```

三个维度都改善了——说明重构有效。但如果：

```
重构前  structural: 68  complexity: 85  fragility: 82
重构后  structural: 75↑ complexity: 70↓ fragility: 85↑
```

结构度和复杂度改善了，但脆弱度反而上升了——说明重构虽然优化了代码结构，但把变更集中到了少数几个新文件上。这些文件会成为新的脆弱点。这种洞察单看代码 diff 是看不出来的。

## 为什么用 Rust

Decay 用 Rust 实现，这不是技术炫耀，而是出于三个实际需求：

**扫描速度。** Decay 需要遍历整个项目的文件树和 Git 历史。一个中等规模的项目（几千个文件、几万条提交）在 Python 里可能需要 30 秒，在 Rust 里 2 秒。这个差距在你想要把 Decay 集成到 CI/CD（每次提交自动扫描）时非常关键。

**单二进制分发。** `cargo build --release` 产出一个没有运行时依赖的二进制文件。不需要 Node.js、不需要 Python、不需要 Docker。`cp` 到任何机器就能跑。对于一个开发工具来说，安装成本越低越好。

**Git 集成。** `git2` 库提供了直接读取 Git 仓库的能力，不需要 shell 出去调 `git log`。这意味着更快的 Git 历史分析、更好的错误处理、以及在没有 `git` CLI 的环境中也能工作。

存储层用 SQLite 也是同样的理由——零配置、单文件、随项目走。快照数据库就在项目目录下（遵循 XDG 标准），不需要额外的数据库服务。

> **常见误区：** "我的项目很小，不需要健康监测。"恰恰相反——小项目的退化速度更快，因为没有团队流程来约束代码质量。一个人的项目尤其容易出现"所有功能堆在一个文件里"的问题。Decay 的三分钟扫描可以在退化变严重之前给你一个预警。

## 和 AI 工作流的集成

Decay 本身不依赖 AI，但它和 AI 工作流配合得很好。

**Sprint 触发。** 当 Decay 发现某个维度持续下降时，可以自动创建一个 Sprint 任务——"结构度连续 3 个月下降，需要调整 src/utils/ 的目录组织"。这个 Sprint 走正常的流水线：plan 阶段分析哪些文件应该移动、execute 阶段执行重组、quality 阶段用 Anchor 验证没有破坏导入关系。

**重构验证。** AI 完成一次重构后，跑一次 Decay 扫描，对比前后分数。如果某个维度不升反降，说明重构方向可能有问题——在合入代码之前就能发现。

**Know 联动。** Decay 标记出的高脆弱文件可以输入 Know——"src/utils/notes.ts 是提交热点，修改时需要格外谨慎"。这条知识用 directive + defensive 模式，AI 修改该文件时会收到提醒。

> **实践建议：** 最简单的起步方式是在两个时机跑 Decay：一是每次大重构之后（验证效果），二是每月初（追踪趋势）。不需要每次提交都跑——那会产生太多噪声。当某个维度连续两次下降时，才值得花时间去排查原因。
