# Diagnose：AI 调试不应该是碰运气

## AI 调试的默认模式：随机搜索

你告诉 AI："部署后搜索页白屏了，帮我排查。"

AI 的第一反应是看搜索页的代码。它发现一个 `fetch` 调用，猜测可能是 API 地址错了，改了地址。不行。又猜可能是 CORS 问题，加了一个 header。还是不行。然后猜可能是 JSON 解析错误，加了 try-catch。仍然白屏。

三轮下来，AI 已经改了三个地方，没有一个命中。更糟糕的是，前两次修改引入了新的变更，现在你不仅要修原来的 bug，还要回退 AI 引入的无效改动。

这就是 AI 调试的默认模式：**看到症状 → 猜一个原因 → 试一个修复 → 不行就猜下一个**。本质上是随机搜索——没有方向、没有收敛、没有终止条件。

人类经验丰富的工程师不会这样调试。他们会先分类：这是构建错误还是运行时错误？是代码逻辑问题还是环境配置问题？能稳定复现吗？分类之后，搜索空间一下子缩小了 80%。

Diagnose 就是把这套方法论交给 AI。

## 四维分类：先缩小搜索空间

遇到 bug 时，Diagnose 要求 AI 先回答 4 个分类问题，而不是直接开始猜：

| 维度 | 选项 | 意义 |
|------|------|------|
| **问题类型** | crash / build / logic / flaky | 决定用什么工具排查 |
| **变更范围** | single-file / cross-module / dependency | 决定排查的广度 |
| **可重现性** | always / sometimes / no-repro | 决定排查策略 |
| **环境** | pure-code / config / infra | 决定排查方向 |

回到搜索页白屏的例子：

- 类型：crash（页面白屏 = 运行时崩溃）
- 范围：cross-module（部署后才出现，可能涉及构建配置）
- 可重现：always（每次打开都白屏）
- 环境：config（本地正常，部署后出问题 → 大概率是环境/配置差异）

四个维度一填，方向就清楚了：**不用看代码逻辑，先看构建配置和部署环境的差异**。搜索空间从"整个代码库"缩小到了"构建配置 + 部署环境"。

如果 AI 不做这个分类，它会从代码逻辑开始找——这正是前面那三轮无效修改的根因。

> **关键洞察：** 调试的效率不取决于你多快能修复，而取决于你多快能定位。四维分类的目的不是为了"走流程"，而是用 30 秒的分类工作换来 30 分钟的搜索时间。

## 三层收敛：Zone → Mechanism → Root Cause

分类之后，Diagnose 用三层收敛模型定位根因：

**第一层：Zone（哪里）。** 问题出在系统的哪个区域？是前端渲染、是 API 调用、是数据库查询、还是构建流程？

在搜索页白屏的例子中，四维分类已经提示了方向：config + cross-module。Diagnose 的第一层假设是"构建产物和本地开发不一致"。验证方法：对比本地 `dev` 模式和 `build` 模式的输出。

发现 `build` 模式下缺少一个 JSON 文件——搜索索引没有被生成到 `dist/` 目录。Zone 定位完成：**问题在构建流程**。

**第二层：Mechanism（怎么）。** 通过什么机制导致了问题？索引文件为什么没出现在 `dist/` 里？

验证方法：检查生成搜索索引的集成代码，对比 dev 和 build 的钩子。发现集成代码只注册了 `astro:server:setup`（dev 模式）的钩子，没有注册 `astro:build:done`（build 模式）的钩子。

Mechanism 定位完成：**dev 钩子有，build 钩子缺失**。

**第三层：Root Cause（为什么）。** 为什么 build 钩子缺失？回到 git log 看——上一次重构搜索功能时，AI 重写了集成代码，把 build 钩子弄丢了。

根因找到了。修复方案明确：补上 `astro:build:done` 钩子。

三层收敛的关键是**每一层都有验证**。不是猜到第三层才验证，而是每一层的假设都要用证据确认，再进入下一层。这确保了方向不会跑偏——如果第一层的 Zone 定位就错了，后面的 Mechanism 和 Root Cause 全部白做。

## 验证方法库：不是只有"看代码"

Diagnose 为每一层假设提供了 5 种验证方法，每种有明确的使用条件和限制：

| 方法 | 什么时候用 | 限制 |
|------|-----------|------|
| **read-code** | 确认代码逻辑是否符合预期 | 最多同时看 3 个文件 |
| **run-test** | 利用已有测试验证假设 | 只跑相关测试，不跑全量 |
| **add-log** | 观察运行时状态 | 最多 3 处，事后必须清理 |
| **isolate** | 构造最小复现 | 独立于主代码库 |
| **git-bisect** | 找到引入问题的提交 | 仅限回归类 bug |

限制条件是刻意设计的。

"最多看 3 个文件"——防止 AI 在 read-code 阶段陷入"读完所有相关代码"的深渊。3 个文件够了吗？如果第一层 Zone 定位准确，3 个文件足够验证 Mechanism。如果不够，说明 Zone 定位有误，应该回退重新分类，而不是继续读更多文件。

"add-log 最多 3 处"——防止 AI 在代码里撒满 `console.log` 然后从日志海洋里捞信息。3 处意味着 AI 必须有假设——"我认为数据在这个点之后变了"——然后精确地在那个点加日志验证。

> **常见误区：** 很多人（和 AI）调试时的第一反应是加日志。但没有假设的日志是无效的——你只会看到一堆数据，不知道要找什么。正确的顺序是：先有假设 → 再选验证方法 → 如果假设需要运行时数据才能验证 → 才用 add-log。read-code 和 run-test 能解决的问题，不需要动运行时。

## 3 轮上限：强制收敛或上报

Diagnose 最重要的约束是**3 轮上限**：

- 每轮最多 3 个假设
- 每个假设最多验证 2 次
- 3 轮还没收敛 → 强制停止，上报用户

为什么需要这个上限？

因为 AI 不会主动放弃。人类工程师调试半小时没进展，会主动停下来重新审视方向，或者去问同事。AI 不会——它会继续生成假设、继续验证、继续失败、继续循环。我见过 AI 在一个配置问题上调试了 12 轮，最终改了十几个文件，没有一个是对的。

3 轮上限的设计逻辑：

**第 1 轮**通常基于症状的直接推断，命中率最高。如果分类准确，大约 60% 的 bug 在第 1 轮就能定位。

**第 2 轮**基于第 1 轮的排除结果调整方向。比如第 1 轮排除了"代码逻辑错误"，第 2 轮聚焦到"配置差异"。

**第 3 轮**是最后机会。如果前两轮的假设都被否定，说明对问题的理解可能有根本性偏差——可能分类本身就错了。这时候继续在错误的方向上搜索只会浪费时间。

到第 3 轮上限时，Diagnose 要求 AI 输出：

1. 已排除的假设及其证据
2. 当前最可能的方向（即使置信度不高）
3. 建议人类检查的具体位置

这比"调试了 12 轮什么都没修好"有用得多。人类拿到排除清单和方向建议，往往能很快找到根因——因为人类有 AI 缺少的全局上下文。

> **关键洞察：** 3 轮上限不是对 AI 能力的不信任，而是对调试过程的理性约束。即使是人类专家，一个 bug 调试超过 3 轮方向性探索还没有头绪时，继续单干的效率也会急剧下降。正确的做法不是更努力地找，而是引入新的视角——对人类来说是找同事讨论，对 AI 来说是上报用户。

## 方向调整策略

3 层收敛不是线性的"走到底再回头"。Diagnose 定义了 4 种方向调整策略，在收敛过程中随时可以触发：

**后退一层。** 当前层的所有假设都被否定了，但上一层的定位仍然可信。比如 Mechanism 层的 3 个假设全部排除，但 Zone 没问题——回到 Mechanism 层换一批假设。

**回到分类。** 当前层的假设不仅被否定，而且否定的证据暗示四维分类可能有误。比如你分类为 "pure-code"，但排查中发现本地和线上行为不一致——说明可能是 "config" 而不是 "pure-code"。回到分类重新判断。

**扩展信息。** 现有信息不足以生成有效假设。需要更多上下文：检查 git log 看最近的变更、检查部署日志看有没有告警、检查依赖版本是否有更新。

**上报用户。** 到达 3 轮上限，或者出现 AI 无法获取的信息（需要访问生产环境、需要联系第三方服务商、需要理解业务上下文）。上报时必须提供排除清单和方向建议。

这 4 种策略的优先级是有序的：优先后退一层（最小调整），其次回到分类（中等调整），再次扩展信息（获取更多上下文），最后上报用户（引入人类判断）。

> **实践建议：** 如果你想在现有的 AI 工作流中引入 Diagnose 的思路，不需要实现完整的系统。最小版本只需要两样东西：第一，在让 AI 调试之前，要求它先完成四维分类并告诉你；第二，设置一个明确的轮次上限（建议 3 轮），到达上限后 AI 必须停下来汇报而不是继续尝试。这两个约束就能消除 90% 的"AI 无限循环调试"问题。
