## 1. 产品愿景

### 产品核心

- **定位**: 报纸/杂志排版风格的个人技术内容站，Astro + Tailwind CSS 构建，部署于 GitHub Pages
- **动机**: 个人技术写作、阅读笔记和生活随想散落各平台，缺少统一发布和归档入口；现有博客模板千篇一律，无法满足对阅读体验的审美要求
- **长远愿景**: 在报纸美学中提供极致阅读体验的个人品牌载体，让读者感受到「这不是普通博客」的设计差异

### 价值体系

| 层级 | 价值 | 衡量指标 |
|------|------|---------|
| **即时价值** | 作者发布文章后，读者在任何设备上获得报纸级排版的阅读体验 | Lighthouse Performance ≥ 90（目标值），375/768/1440px 三断点无布局问题 |
| **累积价值** | 内容持续积累形成知识网络，标签/主题/系列构建多维发现路径，读者回访率上升 | 文章总数（当前 13 篇真实内容），系列完整度，标签覆盖率 |
| **战略价值** | 从「又一个技术博客」到有辨识度的个人品牌载体，内容完全自有可控 | 搜索引擎收录量（目标值，待验证），社交分享点击率（目标值，待验证） |

### 核心问题

| 问题 | 发生频率 | 单次成本 | 影响面 | 现有应对 |
|------|---------|---------|--------|---------|
| 个人技术内容散落各平台，缺少统一发布和归档入口 | 每次写文章（估算: 周均 1-2 篇） | 多平台分发 + 读者无法从一个入口找到所有内容 | 作者本人 + 所有潜在读者 | 依赖第三方平台（掘金/Medium），内容不可控，排版受限 |
| 现有静态博客模板同质化严重，阅读体验无差异化 | 每次搭建/选择博客框架 | 选择模板后仍需大幅定制，最终效果与其他博客雷同 | 所有想要差异化呈现的博客作者 | 在现有模板上深度定制，改动量接近从零开发 |

### 目标用户

| 角色 | 典型场景 | Before | After | 预估提效 |
|------|---------|--------|-------|---------|
| 作者本人（写作 + 发布） | 完成一篇技术文章后发布并归档 | 内容散落各平台，格式不统一，无法统一归档管理 | 统一发布入口，报纸排版自动应用，自动归档/分类/系列串联 | 发布流程从多平台分发简化为单点 Markdown 发布 |
| 对 AI 工程化感兴趣的开发者（阅读） | 通过搜索引擎或社交分享链接进入，阅读技术文章 | 打开一个长得和其他博客一样的页面，读完一篇不知道还有什么 | 报纸风格阅读体验 + TOC 导航 + 系列串联 + 相关推荐引导继续阅读 | 单次访问浏览页数从 1 提升到 2+（目标值，待验证） |

### 竞品对比

| 方案 | 定位 | 目标用户 | 核心功能 | 优势 | 局限 |
|------|------|---------|---------|------|------|
| **夏天的爱人** | 报纸风格个人技术内容站 | 作者 + AI/技术方向读者 | 报纸排版、TOC、系列文章、标签体系、暗色模式、双语 | 设计差异化，数据完全自有，零运行成本（GitHub Pages） | 无评论系统（v2 计划），无内置统计，需手动 Markdown 发布 |
| Hexo/Hugo 博客 | 通用静态博客框架 | 个人博主 | 主题模板库、Markdown 写作、插件生态、多部署方式 | 生态成熟，主题丰富，社区活跃 | 模板同质化，深度定制成本高于收益，切换主题丢失定制 |
| Medium/掘金 | 内容发布平台 | 写作者 + 读者 | 所见即所得编辑器、推荐算法、社交互动、评论系统 | 自带流量和读者，零搭建成本 | 内容不可控（平台下架风险），广告干扰，排版自由度受限 |

## 2. 版本规划

### 汇总

| 版本 | 核心方向 | 核心指标变化 | 状态 | 周期 | 里程碑 |
|------|---------|------------|------|------|--------|
| v1 — 打磨至成熟 | 全站基础可用性 + 阅读体验完善 + 细节收尾 | ↑ 功能完整度 0% → 100%；↓ 布局问题 3 断点全通过 | 开发中 | 2026.04 — 待定 | M1-M3 |
| v2 — 扩大触达 | 搜索引擎可发现、社交可预览、读者可订阅互动 | ↑ SEO 结构化数据 0 → 100%；↑ 读者互动渠道 0 → 2 种 | 规划中 | 待定 | M4-M7 |
| v3 — 差异化体验 | 报纸风格独特交互，区别于普通技术博客 | ↑ 主题模式 2 → 3；↑ 导航方式 +命令面板 | 规划中 | 待定 | M8-M10 |

### 版本详情

#### v1 — 打磨至成熟

- **战略意图**: 确保全站基础体验无缺陷，验证核心假设——报纸风格能在所有设备上提供良好阅读体验
- **投入产出**: 3 个里程碑逐步交付 → 全站可用（M1）、阅读体验完整（M2）、细节无瑕疵（M3）
- **优先级依据**: 基础可用性是后续版本前提；移动端布局破损等于放弃一半以上读者，修复紧迫性高
- **风险与依赖**: 示例数据删除可能影响标签/主题路由生成；OG 图片自动生成需评估额外 npm 依赖
- **成功指标**: Lighthouse Performance ≥ 90, Accessibility ≥ 90；三断点无布局问题；暗色模式无硬编码颜色残留
- **核心价值**: 1. 移动端可正常阅读 2. 长文有 TOC + 系列导航 + 相关推荐 3. 代码块有复制按钮 4. 搜索有高亮 + OG 分享图
- **用户覆盖**: 作者 dogfood + 分享链接触达的技术读者
- **核心指标**（基线 → v1）:

| 指标 | 基线 | v1 | 变化 |
|------|------|-----|------|
| 响应式断点覆盖 | 仅桌面端可用 | 375/768/1440px 全通过 | ↑ 3 断点 |
| 暗色模式覆盖 | 部分页面有残留 | 全站无硬编码颜色 | ↑ 100% 覆盖 |
| 文章导航手段 | 0 种 | TOC + 系列导航 + 相关推荐 | ↑ 3 种 |
| 代码块功能 | 仅显示 | 语言标签 + 复制按钮 | ↑ 2 项增强 |

#### v2 — 扩大触达

- **战略意图**: 让内容被搜索引擎发现、社交平台预览、读者可订阅和互动，解决 v1 "能用但没人知道" 的问题
- **投入产出**: 4 个里程碑 → SEO 基建 + 阅读引导增强 + 读者连接渠道 + 知识可视化
- **优先级依据**: v1 解决"能用"，v2 解决"被发现"和"留下来"；SEO 基建越早做，积累效应越强
- **风险与依赖**: 评论系统依赖 GitHub Discussions 稳定性（giscus）；知识图谱需额外可视化库（如 D3/vis.js）
- **成功指标**: Google Search Console 有收录；文章社交分享有预览卡片；至少 1 条读者评论
- **核心价值**: 1. 搜索引擎可发现 2. 社交分享有预览卡片 3. 读者可评论互动 4. 内容关联可视化
- **用户覆盖**: 搜索引擎自然流量 + 社交平台分享触达
- **核心指标**（v1 → v2）:

| 指标 | v1 | v2 | 变化 |
|------|-----|-----|------|
| SEO 结构化数据 | 无 | Article schema 全覆盖 | ↑ 100% |
| 社交预览 | 无 OG 图片 | 自动生成预览卡片 | ↑ 全覆盖 |
| 读者互动渠道 | 0 种 | RSS 引导页 + giscus 评论 | ↑ 2 种 |
| 阅读引导信号 | 0 种 | 阅读时间 + 进度条 + 头版轮换 | ↑ 3 种 |

#### v3 — 差异化体验

- **战略意图**: 发挥报纸风格的独特优势，打造区别于普通技术博客的交互体验，建立设计辨识度
- **投入产出**: 3 个里程碑 → 多主题模式 + 极客导航 + AI 增强内容
- **优先级依据**: v1/v2 是功能补齐和触达扩展，v3 是差异化锦上添花；需要基础完善后才有意义
- **风险与依赖**: AI 摘要依赖 frontmatter analysis 数据质量；阅读数据看板需要引入轻量统计方案
- **成功指标**: 3 种主题可切换；⌘K 命令面板可用；@media print 输出排版精良的 PDF
- **核心价值**: 1. 报纸风格的独特视觉体验（sepia 模式） 2. 极客友好的快捷交互（⌘K） 3. AI 辅助内容理解
- **用户覆盖**: 高频回访读者 + 对独特体验敏感的开发者
- **核心指标**（v2 → v3）:

| 指标 | v2 | v3 | 变化 |
|------|-----|-----|------|
| 主题模式 | 2（light/dark） | 3（+sepia） | ↑ 1 种 |
| 导航方式 | 传统导航 | +命令面板 ⌘K | ↑ 1 种 |
| 内容理解辅助 | 无 | AI 摘要 | ↑ 1 种 |
| 打印适配 | 无 | @media print 排版优化 | ↑ 全新 |
