# 设计迭代反模式与约束协议

源自 2026-04-19 首页 hero + sidebar 布局迭代（18 轮改动未收敛），总结为什么迭代失败，以及如何用"约束协议"替代"局部打补丁"。

## 1. 反模式：局部打补丁迭代

### 症状

同一块 UI 改了 18 轮仍未收敛。每轮满足用户当前一条反馈，打破上一轮建立的约束：

| 轮次 | 改动 | 打破了什么 |
|------|------|-----------|
| v3 | 给 excerpt 加 `flex: 1` 撑开 | 文字不够长 → box 底部大面积空白 |
| v4 | 评论区加 `flex: 1 + space-between` 均匀分布 | 评论间距被拉开，视觉不紧凑 |
| v6 | 增加摘要到 500 字填满 box | hero 高度超出侧列，整体撑高 |
| v7 | line-clamp 5 + flex:1 强制撑满 | 文字不够长时 excerpt box 内部留白 |
| v8 | sidebar article flex:1 均分 | 侧列 natural 撑高，反向驱动 row |
| v9 | chips nowrap + shrink:0 防溢出 | `min-width: auto` 传导，撑爆 grid 列宽 → 左右比例反了 |

每一轮都是"只看刚改的那一处视觉验证"，没有全局回归。

### 根因

1. **约束散在对话中，从未汇总**。用户多轮追加需求（评论 ≤3、摘要撑满、评论吸底、左右等高、高度由侧列决定、不破坏 secondary headline 可见），AI 每次只满足最新一条。
2. **约束之间互相冲突但未识别**。"评论吸底 + 摘要撑满 + 固定高度 + 变长文字"物理上不可同时成立，第 10 轮才列出矛盾表。
3. **edge case 从未枚举**。评论 0 条 / 摘要 <80 字 / 标签 3 行这些情况直到第 14 轮才被用户提醒考虑。
4. **验证只看当前 viewport，不看测量数值**。早期没跑 playwright，靠肉眼"看起来对"，直到宽度被撑反了才发现 main=321 side=497。

## 2. 正确做法：约束协议（Constraint Protocol）

设计类任务（布局/间距/视觉层级）改动前必做 5 步。

### Step 1 · 约束清单

所有约束列成表格，不只是当前反馈：

| 约束 ID | 描述 | 类型 | 来源 |
|---------|------|------|------|
| C1 | 左右两列等高 | 布局 | 用户 R4 |
| C2 | 摘要填到评论顶部无 gap | 视觉 | 用户 R5 |
| C3 | 评论最多 3 条，可能 0 | 内容 | 用户 R7 |
| C4 | 总高度 = sidebar natural 内容和 | 布局 | 用户 R11 |
| C5 | 标签自然换行 | 视觉 | 用户 R11 |
| C6 | secondary headline 必须首屏可见 | 布局 | CLAUDE.md invariant |

### Step 2 · 矛盾检查

约束两两配对检查：

| 约束 A | 约束 B | 冲突? | 说明 |
|--------|--------|-------|------|
| C1 左右等高 | C4 高度由侧列决定 | ✅ 成立 | 共立 |
| C2 摘要撑满 | C3 评论可 0 条 | ⚠️ 冲突 | 评论 0 时摘要需撑所有空间，文字不够会留白 |
| C2 摘要撑满 | C4 高度可变 | ⚠️ 冲突 | 摘要 flex 只在 box 有固定 target 时能撑 |
| C1 等高 | secondary 首屏 | ⚠️ 约束 | 高度上限受 viewport-header 限制 |

冲突必须和用户对齐取舍，不能默默选一个。

### Step 3 · 变量枚举 Edge Cases

列出所有动态变量及其范围：

| 变量 | 最小值 | 正常 | 最大值 |
|------|--------|------|--------|
| 摘要字数 | 0 | 200 | 1000+ |
| 评论数 | 0 | 3 | 3（硬上限） |
| 标签行数 | 1 | 2 | 3 |
| sidebar 文章数 | 0 | 3 | 3 |
| 侧栏文章标题行数 | 1 | 2 | 3+ |

对 edge case 组合画出布局表现：

| 场景 | 方案 A 表现 | 方案 B 表现 |
|------|------------|------------|
| 标准 | ✅ | ✅ |
| 评论 0 | ⚠️ gap 200px | ✅ |
| 摘要短 | ⚠️ box 空 | ✅ |

### Step 4 · 锁定变量

识别出波动最大的变量，通过 UI 约束把它钉死：

- 评论数波动大 → 占位补齐强制 3 条
- 标签行数波动大 → `max-height` + `overflow: hidden` 限 2 行
- sidebar 文章数 → 不足补占位卡
- 标题行数 → `-webkit-line-clamp` 限 2 行

钉死后的变量变成常量，布局可预测。

### Step 5 · 数值验证

用 playwright / devtools 测量实际数值，不靠肉眼：

```js
document.querySelector('.layout-featured-main').getBoundingClientRect()
// { width, height, top, bottom }
```

关键断言：
- 两列 width 比例 ≈ 期望 fr 比例
- 两列 height 相等（误差 <1px）
- 关键元素 bottom 落在 viewport 内

## 3. 设计决策框架

布局类任务有 4 种"谁驱动高度"的范式，没有哪个绝对好，但必须显式选：

| 范式 | 高度来源 | 适用 | 代价 |
|------|---------|------|------|
| 固定值 | `grid-template-rows: 480px` | 可预测最强 | 不适配内容变化 |
| 主列驱动 | main 内容自然和 | 主列内容稳定 | 侧列需 filler 吸收差值 |
| 侧列驱动 | side 内容自然和 | 侧列内容稳定 | 主列内容需适配或 gap |
| 两侧都固定 | 两侧内容都 clamp/限行 | 最稳定 | 失去灵活度 |

**选范式时必须考虑**：哪一列的变量更可控？不可控的那列不能当 anchor。

## 4. Sprint 是否能避免？

### Sprint 的适用性

Sprint 的 brainstorm / design stage 理论上能把约束梳理前置。但：

- **Sprint 太重**：18 轮微调每轮触发 sprint 不现实
- **Sprint 的 design 阶段在代码前做，但视觉迭代本质是"看到效果再判断"**
- **用户也不知道自己所有约束**，要通过多轮迭代暴露。这是设计本质，不是 sprint 能消除的

### Sprint 的真正用处

- **首次建立布局时**用 sprint：brainstorm 明确约束清单，design 画出 edge case 表
- **后续调整不用 sprint**，但遵循本文档的 Constraint Protocol

### 轻量替代

对于设计微调（非首次建立），用**30 秒前置**替代完整 sprint：

1. 列当前约束（不超过 5 条）
2. 写一句话本次改动打破了哪条约束
3. 不打破任何约束 → 直接改
4. 打破约束 → 先和用户确认取舍

## 6. 锚点提醒

**遇到以下信号，先停下跑 Constraint Protocol，不要直接改代码**：

- 用户反馈"又不对了" / "变形了" 两次以上
- 同一区域连续改动 >3 轮未收敛
- 本次改动会影响 >1 个视觉维度（宽/高/间距/字号）
- 布局用了 flex/grid，并存在 3 条以上约束
