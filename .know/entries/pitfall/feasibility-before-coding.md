## 技术可行性评估必须在编码之前

### 规则

提出技术方案时，先评估底层能力边界（CSS 规范限制、API 限制、性能约束），确认可行后再写代码。不要写完代码再发现根本走不通。

### Why

本次为封面图"左上角环绕"布局反复修改 5 轮（float → flex → 渐隐 → 长文本 → 放弃），才去调研 CSS float+flex 兼容性。调研结论是"规范层面不可能"。如果第一轮就调研，可以节省全部来回。

### How to apply

- 涉及 CSS 布局组合（float+flex、float+grid、position+flex）时，先确认规范兼容性
- 涉及浏览器 API 时，先查 Can I Use
- 涉及性能敏感操作时，先估算量级
- 不确定时说"我需要先调研可行性"，而不是直接开写
